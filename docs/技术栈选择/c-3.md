## 二、核心框架层 — RPC与HTTP

### 2.1 Kitex — RPC 框架 【已有复用】

| 属性 | 详情 |
|------|------|
| **库** | `github.com/cloudwego/kitex v0.16.0` |
| **协议** | Thrift IDL + TTHeader 传输协议 |
| **序列化** | FastPB + Frugal（高性能） |
| **所在位置** | `common/rpc/init.go` — 统一初始化 |
| **钱包模块用途** | ser-wallet 服务的全部 RPC 接口暴露（E1~E6 核心资金操作 + 币种配置 + 钱包业务） |

**在钱包模块中的使用场景**：

```
1. ser-wallet 作为 Kitex Server
   → 暴露 WalletService 全部 RPC 方法（约40+个接口）
   → 供 gate-font/gate-back 网关转发调用
   → 供 ser-finance 跨服务调用 E1~E6

2. ser-wallet 作为 Kitex Client
   → 调用 ser-finance 的 F4(SubmitWithdrawOrder)（提现提交财务）
   → 调用 ser-user 的 D9(KYC查询) / D10(用户信息)
```

**为什么使用它**：
- 工程已统一选型，所有微服务间通信均走 Kitex RPC
- CloudWeGo 生态的字节跳动级高性能 RPC 框架
- IDL 先行开发模式已在工程中验证成熟
- 多路复用（MuxTransport）减少连接数，降低资源占用
- TTHeader 协议支持元数据透传（trace_id、user_id 等）

> 依据：工程认知c-3.md 3.1节 RPC基础设施

**工程配置参照**：

```go
// common/rpc/init.go 已有的统一配置
opts := []client.Option{
    client.WithRPCTimeout(3 * time.Second),
    client.WithMuxConnection(1),
    client.WithMetaHandler(transmeta.ClientTTHeaderHandler),
}
```

## 三、数据存储层 — 数据库与ORM

### 3.1 TiDB — 分布式数据库 【已有复用】

| 属性 | 详情 |
|------|------|
| **类型** | 分布式 NewSQL（MySQL 5.7 协议兼容） |
| **驱动** | `gorm.io/driver/mysql v1.6.0`（MySQL 协议连接） |
| **ETCD配置路径** | `/slg/conf/tidb/addr|port|username|password` |
| **所在位置** | `common/pkg/db/mysql/mysql.go` — 统一初始化 |
| **钱包模块用途** | 全部 12 张表的持久化存储 |

**在钱包模块中的使用场景**：

```
核心场景（资金安全最高优先级）：
  ① wallet_balance — 余额行级锁（SELECT FOR UPDATE / 原子SQL更新）
  ② wallet_transaction — 流水表唯一索引保证幂等
  ③ 余额变更+流水写入同事务（GORM Transaction）

业务场景：
  ④ currency_config / base_currency / currency_rate — 币种与汇率CRUD
  ⑤ recharge_order / withdraw_order / exchange_record — 订单读写
  ⑥ withdraw_account / user_currency_preference — 用户数据读写
  ⑦ rate_log / operation_log — 审计日志追加写入
```

**为什么使用它**：
- 工程统一选型，所有微服务共用 TiDB 集群（每个服务独立数据库）
- MySQL 兼容：支持行级锁（SELECT FOR UPDATE）、事务隔离级别、DECIMAL 精确运算
- 分布式能力：数据量增长时自动 Region 分裂，无需手动分库分表
- 支持 `DECIMAL(24,8)` 精确十进制存储，满足金额计算的精度要求
- 水平扩展能力为未来流水表膨胀预留了空间

> 依据：工程认知c-3.md 3.4节数据库、架构设计c-3.md 5.1节数据库选型、技术挑战c-3.md 第九节流水表膨胀

**TiDB 特性在钱包模块中的具体应用**：

| TiDB 特性 | 钱包应用 |
|-----------|---------|
| 行级锁 (Pessimistic Lock) | wallet_balance 并发余额更新 |
| DECIMAL 精确运算 | 所有金额字段存储和 SQL 层面计算 |
| 唯一索引 | wallet_transaction (order_no, operation_type) 幂等保障 |
| 自动 Region Split | wallet_transaction 日增量大时自动分散存储 |
| MySQL协议兼容 | GORM 无感知接入，无需特殊适配 |

### 3.2 GORM — ORM 框架 【已有复用】

| 属性 | 详情 |
|------|------|
| **库** | `gorm.io/gorm v1.31.1` |
| **所在位置** | `common/pkg/db/mysql/` — 全局 DB 实例 |
| **钱包模块用途** | 全部数据库操作的 ORM 层 |

**在钱包模块中的使用场景**：

```
1. 事务管理（最核心）
   query.Q.Transaction(func(tx *query.Query) error {
       // 余额变更 + 流水写入 在同一事务中
   })

2. 条件查询
   query.Q.WalletBalance.Where(
       query.Q.WalletBalance.UserID.Eq(userId),
       query.Q.WalletBalance.CurrencyCode.Eq(code),
       query.Q.WalletBalance.DeleteAt.Eq(0),
   ).First()

3. 原子更新（不经过 Go 读取，直接在 SQL 层面运算）
   query.Q.WalletBalance.Where(...).
       UpdateSimple(
           query.Q.WalletBalance.AvailableBalance.Add(amount),
       )

4. 悲观锁
   tx.WalletBalance.Where(...).Clauses(clause.Locking{Strength: "UPDATE"}).First()
```

**为什么使用它**：
- 工程统一 ORM，所有服务都使用 GORM
- 完善的事务支持，满足资金操作的 ACID 需求
- 支持 `Clauses(clause.Locking{})` 实现 SELECT FOR UPDATE
- 审计插件 UserPlugin 自动填充 create_by/update_by/create_at/update_at
- PrepareStmt 缓存 + SkipDefaultTransaction 提升性能

> 依据：工程认知c-3.md 3.4节数据库、7.4节 GORM Callback

**连接池配置**（沿用工程默认值）：

| 参数 | 值 | 说明 |
|------|---|------|
| MaxIdleConns | 10 | 空闲连接数 |
| MaxOpenConns | 20 | 最大打开连接数 |
| ConnMaxLifetime | 1h | 连接最大存活时间 |
| PrepareStmt | true | 预编译语句缓存 |
| SkipDefaultTransaction | true | 非显式事务时跳过包裹 |

### 3.3 GORM Gen — 代码生成器 【已有复用】

| 属性 | 详情 |
|------|------|
| **库** | `gorm.io/gen v0.3.27` |
| **所在位置** | 各服务的 `internal/gen/gorm_gen.go` — 生成入口 |
| **钱包模块用途** | 从 12 张表自动生成 Model + 类型安全 Query |

**在钱包模块中的使用场景**：

```
1. 代码生成
   // internal/gen/gorm_gen.go
   g := gen.NewGenerator(gen.Config{...})
   g.ApplyBasic(
       g.GenerateModel("currency_config"),
       g.GenerateModel("wallet_balance"),
       g.GenerateModel("wallet_transaction"),
       // ... 12张表
   )
   g.Execute()

2. 生成产物
   internal/gorm_gen/
   ├── model/     # Go 结构体（对应表字段）
   ├── query/     # 类型安全查询构建器（Q.WalletBalance.UserID.Eq()）
   └── repo/      # 预置 CRUD（仅生成参考，实际使用 rep/ 层手写）

3. 加密字段映射（提现账户敏感字段）

       gen.FieldType("account_no", "utils.EncryptedString"),
       gen.FieldType("holder_name", "utils.EncryptedString"),
   )
```

**为什么使用它**：
- 工程统一的代码生成方案，所有服务都用 GORM Gen
- 类型安全查询，编译期即可发现字段名拼写错误
- 减少手写 SQL/查询代码，降低出错概率
- 支持自定义字段类型映射（如 `EncryptedString` 自动加解密）

> 依据：工程认知c-3.md 6.6节 ~ 6.7节代码生成

## 四、缓存层 — Redis

### 4.1 Redis — 缓存与辅助功能 【已有复用】

| 属性 | 详情 |
|------|------|
| **库** | `github.com/redis/go-redis/v9 v9.17.2` |
| **客户端模式** | `redis.NewUniversalClient`（支持单机/集群/Sentinel） |
| **ETCD配置路径** | `/slg/conf/redis/addr|password` |
| **所在位置** | `common/pkg/rds/redis_init.go` — 初始化 / `redis_util.go` — 操作封装 |
| **钱包模块用途** | 配置缓存、防重提交、Token验证辅助 |

**在钱包模块中的使用场景**：

```
场景1：币种配置缓存（读多写少）
  Key: wallet:currency:list          → 已启用币种列表
  Key: wallet:currency:{code}        → 单币种配置详情
  Key: wallet:rate:{code}            → 当前汇率
  Key: wallet:base_currency          → 基准币种配置
  TTL: 与定时刷新间隔对齐 / B端修改时主动失效
  策略: Cache-Aside（缓存旁路）— 读请求先查缓存，未命中查DB回填

场景2：防重复提交（C端用户操作）
  Key: wallet:submit:{user_id}:{action}:{hash}
  TTL: 5秒
  操作: SetNX — 成功则放行，失败则拦截
  场景: 用户快速双击"兑换"/"提现"按钮

场景3：分布式锁（备用，当前阶段不使用）
  Key: wallet:lock:{user_id}:{currency}:{wallet_type}
  操作: SetNX + TTL
  场景: 如果未来数据库行锁不够用时的升级方案

场景4：幂等辅助快速拦截（可选优化）
  Key: wallet:idempotent:{order_no}:{op_type}
  TTL: 1小时
  操作: 入账成功后写入，下次相同请求先查Redis快速返回
  注意: 这是性能优化，不是幂等的核心保障（核心靠DB唯一索引）
```

**为什么使用它**：
- 工程已统一部署和使用 Redis，连接配置从 ETCD 获取
- `redis_util.go` 已封装 Set/Get/Del/SetNX/HSet/Eval 等全部常用操作
- 币种配置数据读多写少（读写比约 100:1），非常适合缓存
- 防重提交的 SetNX 原子操作简洁高效
- UniversalClient 模式支持后续从单机平滑切换到集群

> 依据：工程认知c-3.md 3.5节 Redis封装、架构设计c-3.md 第九节缓存架构

**已封装可直接使用的操作**：

| 操作 | 函数 | 钱包中的用途 |
|------|------|------------|
| 字符串读写 | `rds.Set()` / `rds.Get()` | 缓存币种配置/汇率 |
| 删除 | `rds.Del()` | 缓存主动失效 |
| 原子写入 | `rds.SetNX()` | 防重提交 / 分布式锁 |
| 过期设置 | `rds.Expire()` | 缓存 TTL 管理 |
| 批量读取 | `rds.MGet()` | 批量获取多币种配置 |
| Hash操作 | `rds.HSet()` / `rds.HGetAll()` | 缓存结构化的币种配置 |
| Lua脚本 | `rds.Eval()` | 复杂原子操作（如带条件的缓存更新） |

**钱包模块不需要使用 Redis 的场景**：

| 场景 | 不用 Redis 的原因 |
|------|-------------------|
| 余额缓存 | 余额是资金数据，必须实时从 DB 读取，不可缓存（架构设计ADR-004） |
| 流水查询 | 流水表数据量大且写密集，缓存命中率低，直接查 DB |
| 核心幂等保障 | DB 唯一索引是幂等的最终保障，Redis 只做辅助快速拦截 |

## 五、消息通信层 — 消息队列

### 5.1 NATS JetStream — 轻量级消息队列 【已有复用】

| 属性 | 详情 |
|------|------|
| **库** | `github.com/nats-io/nats.go v1.48.0` |
| **ETCD配置路径** | `/slg/conf/nats/addr` |
| **所在位置** | `common/pkg/nats/` — 完整封装（连接/发布/订阅/JetStream） |
| **钱包模块用途** | 延迟消息、异步通知 |

**在钱包模块中的使用场景**：

```
场景1：提现超时查询（延迟消息）
  提现提交财务 RPC 调用超时时：
  → 标记订单状态为 SUBMIT_UNCERTAIN
  → 发送 NATS 延迟消息（JSPublishDelay, 延迟30秒）
  → 消费者收到后查询财务侧订单状态
  → 根据结果更新本地状态或执行解冻补偿

  Subject: wallet.withdraw.timeout.check
  延迟: 30秒 → 1分钟 → 5分钟（递增重试）

场景2：充值入账异步通知（可选）
  充值入账成功后：
  → 发布消息通知其他服务（如推送通知、更新用户等级等）
  → ser-wallet 只负责发布，不关心消费方

  Subject: wallet.recharge.credited

场景3：汇率更新广播（可选）
  定时任务刷新汇率后：
  → 广播汇率变更事件
  → 有需要的服务自行订阅消费

  Subject: wallet.rate.updated
```

**为什么使用它**：
- 工程已封装完善的 NATS 客户端，支持 Core NATS + JetStream + 延迟消息
- 延迟消息功能（`JSPublishDelay`）天然适合超时查询场景
- JetStream 持久化保证消息不丢失
- 消费者组（QueueSubscribe）支持多实例部署时的负载均衡
- 相比 Kafka 更轻量，适合低频异步场景

> 依据：架构设计c-3.md 第八节通信架构、技术挑战c-3.md 第四节跨服务一致性

```
  update_at  ← 当前毫秒时间戳

钱包模块的 12 张表全部包含这 4 个审计字段
→ 注册插件后自动填充，业务代码无需手动设置
```

## 十一、需新增引入的技术

> 以下技术在现有工程中尚未使用，但钱包模块实现过程中需要引入。

### 11.1 shopspring/decimal — 高精度十进制运算 【需新增】

| 属性 | 详情 |
|------|------|
| **库** | `github.com/shopspring/decimal` |
| **最新稳定版** | v1.4.0 |
| **引入位置** | ser-wallet 的 go.mod |
| **用途** | 所有金额计算的精确运算 |

**为什么必须引入**：

```
问题：Go 原生 float64 存在精度丢失
  0.1 + 0.2 = 0.30000000000000004  ← 不可接受
  汇率换算涉及多步乘除，精度误差会累积

现状：工程中目前没有引入此库（各服务不涉及金额计算）

钱包模块中需要精确计算的场景：
  ① 汇率换算：from_amount × rate = to_amount
  ② 浮动汇率：platform_rate × (1 + float_pct) = deposit_rate
  ③ 赠送计算：to_amount × bonus_ratio = bonus_amount
  ④ 手续费计算：withdraw_amount × fee_rate = fee
  ⑤ 余额校验：before_balance + amount = after_balance
  ⑥ 偏差阈值：|real_rate - platform_rate| / platform_rate × 100 = deviation_pct
```

**使用方式**：

```go
import "github.com/shopspring/decimal"

// 汇率换算
fromAmount := decimal.NewFromString("100.00")
rate := decimal.NewFromString("0.00038168")
toAmount := fromAmount.Mul(rate)  // 精确结果

// 保留指定小数位
result := toAmount.Round(8)  // 加密货币保留8位
result := toAmount.Round(2)  // 法币保留2位

// 与 GORM DECIMAL(24,8) 的交互
// decimal.Decimal 可以直接转 string 存入 DB
dbValue := result.String()        // "0.03816800"
restored, _ := decimal.NewFromString(dbValue)  // 从DB读取恢复
```

**选择理由**：
- Go 生态中最成熟的高精度十进制库（GitHub 6k+ Stars）
- 支持任意精度，不会溢出
- 提供 Round/Floor/Ceil/Truncate 等完整的舍入方法
- 支持 `RoundHalfUp`（四舍五入）— 符合需求中的舍入规则
- 实现了 `json.Marshaler/Unmarshaler`，JSON 序列化无问题
- 实现了 `database/sql` 的 `Scanner/Valuer`，可直接与 GORM 配合
- 业界金融/支付系统（Stripe Go SDK、各种交易所）的标准选择

> 依据：技术挑战c-3.md 第五节金额精度安全、架构设计c-3.md ADR-006

**全链路精度规范**：

| 层 | 技术 | 类型 | 不允许 |
|----|------|------|--------|
| DB 存储 | TiDB | `DECIMAL(24,8)` | FLOAT/DOUBLE |
| Go 运算 | shopspring/decimal | `decimal.Decimal` | float64 |
| IDL 传输 | Thrift | `string` | double/i64 |
| 前端展示 | 按币种精度截断 | string | — |

### 11.2 三方汇率 API 调用 — HTTP Client 【需新增】

| 属性 | 详情 |
|------|------|
| **技术** | Go 标准库 `net/http` 或工程已有的 HTTP 工具 |
| **引入位置** | ser-wallet 的 `internal/ser/rate_service.go` |
| **用途** | 调用 CoinGecko / ExchangeRate-API / Fixer 等三方汇率 API |

**为什么需要**：

```
汇率定时刷新流程需要调用外部 HTTP API 获取实时汇率：
  ① 并发请求 3 家汇率提供商
  ② 设置 2 秒超时
  ③ 解析 JSON 响应提取汇率数据
  ④ 异常过滤（偏差 > 50% 的数据丢弃）
  ⑤ 取均值作为实时汇率
```

**实现方式**：

```go
// 使用标准库 net/http + sonic JSON 解析
// 不需要引入额外依赖

client := &http.Client{Timeout: 2 * time.Second}

// 并发请求3家API
type rateResult struct {
    Provider string
    Rate     decimal.Decimal
    Err      error
}
results := make(chan rateResult, 3)

go fetchCoinGecko(client, currencyCode, results)
go fetchExchangeRateAPI(client, currencyCode, results)
go fetchFixer(client, currencyCode, results)

// 收集结果，过滤异常，取均值
```

**为什么不引入第三方 HTTP 库**：
- Go 标准库 `net/http` 已足够成熟
- 工程中已有 `common/pkg/utils/http_util.go` HTTP 工具函数
- 调用场景简单（GET 请求 + JSON 解析），不需要 resty/go-resty 等
- sonic 已作为 JSON 引擎，解析性能足够

> 依据：实现思路c-3.md 第七节汇率定时刷新、技术挑战c-3.md 第六节汇率实时性

## 十二、不需要引入的技术

> 以下技术经过分析后明确排除，说明排除理由。

### 12.1 分布式事务框架（Seata / DTM）— 不引入

| 考量 | 分析 |
|------|------|
| **场景** | 提现冻结-提交的跨服务一致性 |
| **排除理由** | ① 提现场景频率不高（非秒级高并发）<br>② 只有两方参与（钱包+财务），补偿逻辑简单<br>③ 引入框架的运维成本 >> 手写补偿<br>④ 架构设计 ADR-002 已决策不引入 |
| **替代方案** | 补偿模式/Saga（手写正向+补偿逻辑） |

> 依据：架构设计c-3.md ADR-002、技术挑战c-3.md 第四节

### 12.2 事件溯源（Event Sourcing）— 不引入

| 考量 | 分析 |
|------|------|
| **场景** | 资金流水记录 |
| **排除理由** | ① wallet_transaction 流水表已经是 INSERT ONLY，本质上已实现"事件记录"<br>② 完整的 Event Sourcing 需要 Event Store + Projection，架构复杂度大幅增加<br>③ 当前业务不需要"从事件流重建状态"的能力 |
| **替代方案** | 流水表 INSERT ONLY + before/after_balance 链条 |

### 12.3 独立限流组件（Sentinel / rate-limiter）— 不引入

| 考量 | 分析 |
|------|------|
| **场景** | API 限流防刷 |
| **排除理由** | ① 网关层的 Redis SetNX 防重提交已能防用户级重复请求<br>② 当前阶段无公网直接暴露（都经过网关），系统级限流需求不强<br>③ 如需全局限流，可在网关层统一加，不需要在服务层引入 |
| **替代方案** | Redis SetNX 防重提交 + 网关层防刷 |

### 12.4 Prometheus + Grafana — 不引入（非本模块职责）

| 考量 | 分析 |
|------|------|
| **场景** | 系统监控和指标采集 |
| **排除理由** | ① 整个工程目前都未集成 Prometheus<br>② 这是基础设施级别的决策，不是单个服务模块的职责<br>③ 可以通过日志 + XXL-JOB 告警覆盖核心监控需求 |
| **替代方案** | 结构化日志 + 自定义告警 |

### 12.5 WebSocket — 不引入

| 考量 | 分析 |
|------|------|
| **场景** | 汇率实时推送、余额变动实时通知 |
| **排除理由** | ① 当前需求只要求"手动刷新页面展示新数据"<br>② WebSocket 基础设施增加系统复杂度<br>③ 后续如有需求，是单独的基础设施项目 |
| **替代方案** | 前端轮询 / 手动刷新 |

### 12.6 Redlock（Redis 分布式锁高级方案）— 不引入

| 考量 | 分析 |
|------|------|
| **场景** | 余额并发控制 |
| **排除理由** | ① 数据库行级锁（原子SQL / SELECT FOR UPDATE）已满足需求<br>② 架构设计 ADR-003 已决策当前阶段用数据库行锁<br>③ Redlock 需要多个 Redis 节点，增加基础设施依赖 |
| **替代方案** | 数据库原子SQL更新 + SELECT FOR UPDATE |

### 12.7 数据库迁移工具（golang-migrate / atlas）— 不引入

| 考量 | 分析 |
|------|------|
| **场景** | 数据库 Schema 管理 |
| **排除理由** | ① 工程现有模式是手动执行 DDL + GORM Gen 生成代码<br>② 团队规模小，手动管理 DDL 可控<br>③ 引入迁移工具需要改变整个工程的数据库管理流程 |
| **替代方案** | DDL SQL 文件 + GORM Gen 代码生成 |

## 十五、总结

### 15.1 核心结论

1. **92% 复用率**：钱包模块使用的 24 项技术中，22 项在现有工程中已有成熟封装，只需新增 `shopspring/decimal` 和三方汇率 HTTP 调用
2. **零基础设施变更**：不需要部署新的中间件或服务，所有基础设施（TiDB、Redis、ETCD、NATS、Kafka、XXL-JOB）均已就绪
3. **最小依赖原则**：明确排除了 6 项不需要引入的技术，避免过度设计
4. **渐进式复杂度**：Kafka 异步日志、Redis 分布式锁等技术标记为"可选/后续"，需要时再启用

### 15.2 新增依赖清单

仅需在 ser-wallet 的 `go.mod` 中新增：

```
require (
    github.com/shopspring/decimal v1.4.0
)
```

其他所有依赖均通过 `common/pkg` 或 `common/rpc` 间接引入，无需在 ser-wallet 中直接声明。

### 15.3 开发阶段技术启用路径

| 阶段 | 启用的技术 | 说明 |
|------|-----------|------|
| **Phase 0 基础搭建** | Kitex, GORM, GORM Gen, ETCD | 服务骨架 + DB 初始化 + IDL 生成 |
| **Phase 1 币种配置** | + Redis (缓存), Zap (日志) | 币种CRUD + 汇率管理 |
| **Phase 2 钱包核心** | + shopspring/decimal, AES, Mask | 余额操作 + 充值/兑换/提现 |
| **Phase 3 定时任务** | + XXL-JOB, net/http, cron | 汇率刷新 + 超时扫描 |
| **Phase 4 跨服务** | + NATS (延迟消息), Kitex Client | 财务联调 + 补偿机制 |
| **Phase 5 优化** | + Kafka (可选) | 审计日志异步化(如需要) |

### 15.4 最终原则

> **"能复用的绝不新造，能简单的绝不复杂，能推迟的绝不提前引入。"**

这与架构设计的六条原则完全一致：
- 遵循现有工程模式（复用 22/24 项技术）
- 不过度设计（排除 6 项不必要技术）
- 最小依赖原则（仅新增 1 个 Go 库）
- 渐进式复杂度（按阶段逐步启用）
