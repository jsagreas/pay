## 1. 分析方法论

### 1.1 分析思路

```
需求场景 → 需要什么能力 → 工程中有没有 → 有则复用/没有则评估引入
```

三步走：
1. **从需求出发**：根据钱包模块的业务场景，明确需要哪些技术能力
2. **从工程出发**：扫描 `common/pkg/` 和各 `ser-*` 模块，盘点已有的全部中间件和组件
3. **差异分析**：需求能力 vs 已有能力，缺什么补什么，不多引入

### 1.2 分类标准

| 分类 | 定义 | 标记 |
|------|------|------|
| **已有·复用** | 工程 `common/pkg/` 已封装，直接引用即可 | 🟢 |
| **已有·扩展** | 工程已有基础组件，钱包场景需要在此基础上做业务封装 | 🟡 |
| **新增·引入** | 工程中不存在，钱包模块确实需要新引入的依赖 | 🔵 |
| **不需要** | 市面上常见但当前阶段不需要的技术 | ⚪ |

### 1.3 信息来源

- 工程代码：`go.work`（12模块）、`common/pkg/`（全部公共组件源码）、各 `ser-*/go.mod`
- 已有分析文档：需求理解、架构设计、数据模型、表结构设计、技术挑战、实现思路、链路关系（共11份）

### 3.1 CloudWeGo Hertz — HTTP网关框架

| 属性 | 详情 |
|------|------|
| **库** | `github.com/cloudwego/hertz v0.10.3` |
| **位置** | `gate-back/`（B端网关）、`gate-font/`（C端网关） |
| **钱包用途** | 接收前端HTTP请求，路由转发到 ser-wallet RPC服务 |

| 网关 | 路由前缀 | 鉴权方式 | 钱包接口举例 |

| gate-font | `/api/wallet/*` | BlackCheck + Token验证 | 余额查询、充值、提现、兑换（C端用户操作） |
| gate-back | `/api/wallet/*` | WhiteCheck + RBAC权限 | 币种管理、风控审核、财务出款（B端后台操作） |

**已有中间件能力**（无需额外开发）：
- 链路追踪注入（TraceID自动生成）
- 访问日志（`middleware.HertzAccLogStr()`）
- CORS处理（tracer中间件内置）
- 上下文传递（UserID、Token、DeviceType等通过metainfo传播）

## 4. 数据存储层（已有·复用）

### 4.1 MySQL — 关系型数据库

| 属性 | 详情 |
|------|------|
| **驱动** | `gorm.io/driver/mysql v1.6.0` + `go-sql-driver/mysql v1.9.3` |
| **位置** | `common/pkg/db/mysql/mysql.go` |
| **钱包用途** | 全部业务数据的持久化存储 |
| **为什么用** | 工程统一数据库选型；钱包的ACID事务需求（余额操作）与MySQL强事务能力天然匹配 |

**已有初始化配置**：

| 参数 | 值 | 说明 |
|------|---|------|
| MaxIdleConnections | 10 | 最小空闲连接 |
| MaxOpenConnections | 20 | 最大连接数（钱包高并发场景可能需要评估是否调大） |
| MaxConnectionLifeTime | 1小时 | 连接最大存活时间 |
| PrepareStmt | true | 预编译语句缓存，提升重复SQL性能 |
| SkipDefaultTransaction | true | 跳过GORM默认包事务，提升单条操作性能 |

**钱包存储的核心表**（10张，详见表结构设计文档）：
- 余额表（wallet）、交易流水（transaction_log）、充值订单（deposit_order）
- 提现订单（withdrawal_order）、提现账户（withdrawal_account）、币种配置（currency_config）
- 兑换记录（exchange_record）、奖励记录（reward_record）、汇率变更日志（rate_change_log）、系统配置（system_config）

**钱包核心依赖的MySQL能力**：

| 能力 | 用法 | 对应场景 |
|------|------|---------|
| DECIMAL类型 | `DECIMAL(20,8)` | 金额精确存储，零浮点误差 |
| 唯一索引 | `UNIQUE KEY (order_id, log_type)` | 幂等控制，防止重复记账 |
| 条件UPDATE | `WHERE available_balance >= amount` | CAS乐观锁，余额并发安全 |
| 本地事务 | `BEGIN...COMMIT` | 扣款+记流水原子性 |
| 联合索引 | `(user_id, currency_code, wallet_type)` | 多币种钱包快速定位 |

### 4.2 GORM + gorm/gen — ORM与代码生成

| 属性 | 详情 |
|------|------|
| **库** | `gorm.io/gorm v1.31.1` + `gorm.io/gen v0.3.27` |
| **位置** | 各服务 `internal/gen/gorm_gen.go`（生成入口）→ `internal/gorm_gen/`（生成产物） |
| **钱包用途** | 数据库操作的唯一入口，禁止裸SQL |
| **为什么用** | 工程统一ORM规范，gen提供类型安全+编译期检查，减少运行时错误 |

**gorm/gen 工作流程**：

```
DDL建表 → gorm_gen.go配置表名 → go generate → 自动生成 Model/Query/Repo
```

**生成产物层级**：
- `internal/gorm_gen/model/` — 表对应的Go结构体（自动映射字段类型）
- `internal/gorm_gen/query/` — 类型安全的查询构造器（编译期校验字段名）
- `internal/gorm_gen/repo/` — 仓库层封装（CRUD基础方法）

**钱包业务层调用模式**（遵循工程现有模式）：

```
handler.go（接口入口）
    ↓ 调用
internal/ser/xxx_ser.go（业务逻辑层）
    ↓ 调用
internal/rep/xxx_rep.go（数据访问层，组合gorm_gen）
    ↓ 使用
internal/gorm_gen/query/（gen生成的类型安全查询）
```

**钱包场景关键用法**：

| 场景 | gorm/gen 用法 |
|------|-------------|
| 余额扣减(CAS) | `query.Wallet.Where(...).UpdateColumn("available_balance", gorm.Expr("available_balance - ?", amount))` |
| 事务操作 | `query.Q.Transaction(func(tx *query.Query) error { ... })` |
| 幂等插入 | `query.TransactionLog.Create(&log)` + 唯一索引捕获duplicate error |
| 分页查询 | `query.TransactionLog.Where(...).Offset(offset).Limit(limit).Find()` |
| 状态机更新 | `query.WithdrawalOrder.Where(q.Status.Eq(old)).Update(q.Status, new)` |

## 5. 缓存层（已有·复用）

### 5.1 Redis — 分布式缓存与原子操作

| 属性 | 详情 |
|------|------|
| **库** | `github.com/redis/go-redis/v9 v9.17.2` |
| **位置** | `common/pkg/rds/redis_init.go`（初始化）、`common/pkg/rds/redis_util.go`（工具方法） |
| **钱包用途** | 多场景：缓存 + 分布式锁 + 防重复 + 计数器 |
| **为什么用** | 工程统一缓存组件；Redis的原子操作能力满足钱包防重、限流需求 |

**钱包模块使用Redis的具体场景**：

#### 场景1：汇率缓存（Cache-Aside模式）

| 属性 | 说明 |
|------|------|
| **Key格式** | `wallet:rate:{from}_{to}`，如 `wallet:rate:USD_BSB` |
| **Value** | JSON字符串：`{"rate": "10.00", "update_time": 1709020800000}` |
| **TTL** | 30-60秒（短TTL保证实时性） |
| **使用API** | `rds.Set()` / `rds.Get()` |
| **为什么缓存** | 汇率API调用有频率限制和网络延迟，缓存减少外部依赖 |
| **一致性策略** | 定时任务刷新 + 写入时更新缓存 |

#### 场景2：币种配置缓存

| 属性 | 说明 |
|------|------|
| **Key格式** | `wallet:currency:config` 或 `wallet:currency:{code}` |
| **Value** | JSON字符串（币种配置信息） |
| **TTL** | 5-10分钟（配置变更频率低） |
| **使用API** | `rds.Set()` / `rds.Get()` / `rds.Del()`（配置变更时主动清除） |
| **为什么缓存** | 每笔交易都要查币种配置（精度、状态、限额），高频读场景 |

#### 场景3：防重复提交（分布式锁）

| 属性 | 说明 |
|------|------|
| **Key格式** | `wallet:lock:deposit:{user_id}` 或 `wallet:lock:withdraw:{user_id}` |
| **实现方式** | `SETNX` + TTL（5秒） |
| **使用API** | `rds.UniversalClient().SetNX()` |
| **为什么** | 用户快速重复点击充值/提现，防止产生重复订单 |

#### 场景4：幂等Token（可选增强）

| 属性 | 说明 |
|------|------|
| **Key格式** | `wallet:idem:{order_id}` |
| **TTL** | 24小时 |
| **使用API** | `rds.Set()` / `rds.Get()` |
| **说明** | 数据库唯一索引已是主要幂等手段，Redis做前置快速拦截，减少DB压力 |

**已有Redis API能力满足度**：

| 钱包需求 | Redis命令 | rds包支持 |
|---------|----------|----------|
| 缓存读写 | SET/GET | ✅ `rds.Set()` / `rds.Get()` |
| 缓存删除 | DEL | ✅ `rds.Del()` |
| 防重复锁 | SETNX | ✅ `rds.UniversalClient().SetNX()` |
| 原子递增 | INCR/INCRBY | ✅ `rds.Incr()` / `rds.IncrBy()` |
| 批量读取 | MGET | ✅ `rds.MGet()` |
| Pipeline | PIPELINE | ✅ `rds.Pipelined()` |
| Lua脚本 | EVAL | ✅ `rds.Eval()` |
| 设置过期 | EXPIRE | ✅ `rds.Expire()` |

> 结论：`rds` 包已有的API完全覆盖钱包所有Redis使用场景，无需额外封装。

**ETCD能力在钱包的使用**：

| 能力 | 用法 | 调用方式 |
|------|------|---------|

**ETCD高性能读取机制**（已有，钱包直接受益）：
- `atomic.Value` 实现无锁内存缓存
- Copy-On-Write策略更新
- 热路径读取零网络开销
- 支持Hook回调，配置变更时触发业务逻辑（如刷新汇率缓存）

| 充值成功通知 | Producer | `wallet_balance_changed` | 通知其他服务用户余额已变化 |
| 提现状态变更 | Producer | `wallet_withdrawal_status` | 通知前端/消息推送服务 |
| 财务出款结果 | Consumer | `finance_payment_result` | 接收财务模块的出款完成回调 |
| 对账数据 | Producer | `wallet_reconciliation` | 定时对账结果推送 |

**已有能力**：
- **幂等生产者**：`Idempotent = true`（exactly-once语义，防消息重复）
- **Hash分区**：按UserID分区，保证同一用户的消息顺序
- **异步发送**：不阻塞主流程
- **重试机制**：最多10次，间隔1秒
- **消费者组**：支持多实例消费，自动rebalance
- **手动/自动提交**：可按场景选择offset提交策略

**调用模式**（链式API）：
```
producer.NewSend().
    WithKey(userId).           // 按用户分区
    WithJsonContent(event).    // JSON序列化
    Send("wallet_balance_changed")
```

### 10.1 shopspring/decimal — 高精度十进制计算 🔵

| 属性 | 详情 |
|------|------|
| **库** | `github.com/shopspring/decimal` |
| **最新版** | v1.4.0 |
| **工程现状** | **未引入**，各 `go.mod` 中均无此依赖 |
| **钱包用途** | 所有金额计算的唯一方式 |
| **引入原因** | 金融系统硬性要求，Go原生 `float64` 存在精度丢失 |

**为什么必须引入**：

```
float64 精度问题：
  0.1 + 0.2 = 0.30000000000000004  ← 不可接受

decimal 精度保证：
  decimal.NewFromString("0.1").Add(decimal.NewFromString("0.2"))
  = 0.3  ← 精确
```

**钱包中使用decimal的场景**：

| 场景 | 操作 | 说明 |
|------|------|------|
| 余额扣减 | `balance.Sub(amount)` | 减法精确 |
| 汇率换算 | `amount.Mul(rate)` | 乘法精确 |
| 手续费计算 | `amount.Mul(feeRate)` | 百分比计算 |
| 限额校验 | `amount.GreaterThan(maxLimit)` | 比较精确 |
| 多API汇率取均值 | `sum.Div(decimal.NewFromInt(3))` | 除法精确 |
| 前端展示格式化 | `amount.StringFixed(2)` | 指定小数位 |

**核心API一览**：

| 方法 | 用途 |
|------|------|
| `decimal.NewFromString("100.50")` | 从字符串创建（推荐，与Thrift string字段对接） |
| `decimal.NewFromFloat(100.50)` | 从浮点创建（不推荐） |
| `a.Add(b)` / `a.Sub(b)` | 加 / 减 |
| `a.Mul(b)` / `a.Div(b)` | 乘 / 除 |
| `a.GreaterThan(b)` / `a.LessThan(b)` | 大于 / 小于 |
| `a.Equal(b)` | 等于比较 |
| `a.IsZero()` / `a.IsNegative()` | 零值 / 负值判断 |
| `a.StringFixed(8)` | 格式化为固定8位小数 |
| `a.Round(2)` | 四舍五入到2位 |
| `a.Truncate(2)` | 截断到2位（向零取整） |

**引入方式**：
```
cd ser-wallet && go get github.com/shopspring/decimal
```

**数据流中的decimal位置**：
```
前端(string) → Thrift IDL(string) → Go服务(decimal.Decimal) → GORM(decimal→string→DB DECIMAL)
                                                              ↑ 所有计算在这一层完成
```

**与GORM集成**：
- 数据库字段类型为 `DECIMAL(20,8)`
- GORM读取时得到 `string`，用 `decimal.NewFromString()` 解析
- 写入时用 `decimal.String()` 转回字符串

## 11. 技术栈与业务场景映射

### 11.1 充值流程技术栈

```
[用户发起充值]
  │
  ├─ Hertz网关 → 路由转发 + 鉴权

  ├─ Snowflake → 生成充值订单号

  ├─ Redis SETNX → 防重复提交
  ├─ MySQL事务 → 创建订单 + 记流水
  ├─ HTTP工具 → 调用三方支付
  ├─ Zap日志 → 记录操作日志
  └─ TraceID → 全链路追踪

[三方支付回调]
  │
  ├─ MySQL事务 → 更新订单状态 + 增加余额 + 记流水
  ├─ 唯一索引 → 幂等校验（防重复回调）
  ├─ Kafka → 发送余额变更事件
```

### 11.2 提现流程技术栈

```
  │
```

## 12. 不需要引入的技术（排除清单）

### 12.1 分布式事务框架（TCC/Saga/Seata）

| 说明 | 详情 |
|------|------|
| **为什么不用** | 钱包核心操作（扣款+记流水）在同一个数据库内完成，本地事务足够 |
| **跨服务场景** | 通过"重试+幂等+对账"三板斧解决，不需要引入分布式事务框架 |
| **依据** | 技术挑战文档P0-4：分布式一致性方案选择"最终一致性"而非"强一致" |

### 12.2 独立分布式锁库（Redisson/Redsync）

| 说明 | 详情 |
|------|------|
| **为什么不用** | 钱包的并发控制主要靠数据库CAS（乐观锁），不依赖分布式锁做余额操作 |
| **防重复提交** | 直接用 `rds.UniversalClient().SetNX()` 即可，不需要引入完整分布式锁库 |
| **依据** | 技术挑战文档P0-1：并发安全方案选择"CAS乐观锁" |

### 12.3 本地内存缓存库（BigCache/FreeCache/Ristretto）

| 说明 | 详情 |
|------|------|
| **为什么不用** | ETCD已有基于 `atomic.Value` 的内存缓存机制；汇率等数据走Redis缓存已满足性能需求 |
| **如果后续需要** | 可以考虑在Redis前加一层本地缓存，但当前阶段属于过度设计 |

### 12.4 限流组件（uber/ratelimit/golang.org/x/time/rate）

| 说明 | 详情 |
|------|------|
| **为什么不用** | 限流职责在网关层（gate-font/gate-back），不在业务服务层 |
| **已有防护** | Redis SETNX防重复 + 数据库CAS防超额，已形成双层防护 |

### 12.5 ORM增强（dbresolver读写分离）

| 说明 | 详情 |
|------|------|
| **工程现状** | `go.mod` 中有 `gorm.io/plugin/dbresolver v1.6.2`，但当前无使用证据 |
| **为什么暂不用** | 钱包第一阶段数据量有限，单库足够；读写分离是后期优化手段 |
| **预留** | 如果后续读压力大，可启用dbresolver做读写分离，无需改业务代码 |

| **事务保证** | 通过"本地事务提交后再发Kafka"+ "对账兜底"实现最终一致性 |

## 14. 总结

### 14.1 核心结论

**工程基础设施成熟度极高**。钱包模块需要的几乎所有技术组件在 `common/pkg/` 中都已有现成封装，只需引入 `shopspring/decimal` 一个新依赖。

### 14.2 技术栈清单（按层级）

| 层级 | 组件 | 数量 | 状态 |
|------|------|------|------|
| 核心框架 | Hertz + Kitex + Thrift | 3 | 全部已有 |
| 数据存储 | MySQL + GORM/gen | 2 | 全部已有 |
| 缓存 | Redis (go-redis/v9) | 1 | 已有 |
| 配置注册 | ETCD v3 | 1 | 已有 |
| 消息事件 | Kafka + NATS | 2 | 全部已有 |
| 定时任务 | Cron + XXL-JOB | 2 | 全部已有 |
| 基础工具 | Snowflake + AES + HTTP + 日志 + 追踪 + Copier + UUID | 7 | 全部已有 |
| **新增** | **shopspring/decimal** | **1** | **需引入** |
| **合计** | | **20** | **19已有 + 1新增** |

### 14.3 为什么只需要新增1个依赖

| 分析维度 | 结论 |
|---------|------|
| **框架层** | CloudWeGo全家桶覆盖HTTP/RPC/IDL，不需要其他框架 |
| **数据层** | MySQL+GORM/gen覆盖存储+ORM，事务/索引/CAS都是MySQL原生能力 |
| **缓存层** | Redis覆盖缓存+锁+防重复，go-redis/v9功能完整 |
| **消息层** | Kafka（持久化）+NATS（实时）互补，无需第三种MQ |
| **任务层** | Cron（本地）+XXL-JOB（分布式）覆盖全部定时需求 |
| **工具层** | 雪花ID、加密、HTTP、日志、追踪全部已有封装 |
| **唯一缺口** | Go没有原生decimal类型，金融计算必须引入shopspring/decimal |

### 14.4 钱包模块的技术侧重点

技术栈已经就绪，钱包模块的核心挑战不在"选什么技术"，而在"怎么用好已有技术"：

| 重点 | 说明 |
|------|------|
| **CAS乐观锁用对** | `WHERE available_balance >= amount` 写对条件，处理好竞争失败的返回 |
| **事务边界画对** | 扣款+记流水在同一个事务内，不多不少 |
| **decimal用对** | 全链路string → decimal → string，中间不经过float64 |
| **缓存策略用对** | 汇率短TTL、币种配置长TTL、余额不缓存 |
| **幂等做对** | orderId+唯一索引是主线，Redis防重是辅助 |
| **状态机做对** | 提现7步状态严格按WHERE条件流转，不能跳步 |

### 14.5 后续可关注（不急）

以下技术在业务量增长后可以考虑引入，但当前阶段不需要：

| 技术 | 触发条件 | 说明 |
|------|---------|------|
| 读写分离(dbresolver) | 读QPS > 单库承受能力 | GORM已引入插件，启用即可 |
| 本地缓存(BigCache等) | Redis成为瓶颈 | 在Redis前加一层L1缓存 |
| 分库分表 | 单表数据量 > 5000万 | transaction_log表可能最先触发 |
| Elasticsearch | 需要复杂检索/聚合 | 流水数据的多维分析 |

---

> **一句话总结**：工程基础设施成熟，钱包模块"拎包入住"，只需 `go get github.com/shopspring/decimal`。核心挑战在业务逻辑的正确性（CAS、事务、幂等、状态机），而非技术选型。
