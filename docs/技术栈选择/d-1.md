## 3. 第二类：数据存储层（已有 · 复用）

### 3.1 TiDB（分布式数据库）

| 属性 | 说明 |
|------|------|
| **类型** | 分布式 NewSQL，MySQL 兼容协议 |
| **驱动** | go-sql-driver/mysql v1.9.3 |
| **连接配置** | ETCD 动态读取（`/slg/conf/tidb/`） |
| **工程位置** | `common/pkg/db/mysql/`（初始化 + 连接池） |
| **用途** | 主数据库，存储所有业务数据 |
| **为什么用它** | MySQL 兼容（零迁移成本）；分布式水平扩展；强一致事务（悲观锁支持）；工程已部署使用 |
| **钱包场景** | 存储钱包账户、交易流水、充值/提现订单、币种配置、汇率日志等全部核心表。资金操作依赖其 **悲观行锁（SELECT FOR UPDATE）** 保障并发安全 |

**连接池配置（已有默认值）：**
```
MaxOpenConns:    20    -- 最大连接数
MaxIdleConns:    10    -- 最大空闲连接数
ConnMaxLifetime: 1h    -- 连接最大存活时间
```

**钱包模块考量：** 资金操作涉及短事务 + 行锁，当前默认连接池配置对初期流量足够。如后续并发上升，可通过 ETCD 动态调整连接池参数。

### 3.2 GORM + gorm.io/gen（ORM + 代码生成）

| 属性 | 说明 |
|------|------|
| **版本** | GORM v1.31.1 + gen v0.3.27 |
| **工程位置** | `common/pkg/db/mysql/`（初始化）、各 `ser-*/internal/gorm_gen/`（生成代码） |
| **代码生成脚本** | `common/script/gen_mysql.sh` |
| **用途** | ORM 操作数据库 + 类型安全的查询 DSL 自动生成 |
| **为什么用它** | Go 生态最成熟的 ORM；gen 插件提供类型安全查询（编译期检查）；自定义类型支持（EncryptedString）；工程已统一使用 |
| **钱包场景** | 钱包所有数据库操作通过 GORM Gen 生成的 query.Q 执行；事务用 `query.Q.Transaction()`；行锁用 `.ForUpdate()` |

**GORM 关键配置（已有）：**
```go
SkipDefaultTransaction: true  // 普通操作不自动包事务（性能优化）
// 需要事务时显式使用 query.Q.Transaction()
```

**代码生成流程：**
```
1. 建表 SQL → 执行到 TiDB
2. 编写 internal/gen/gorm_gen.go（指定要生成的表）
3. 运行 gen_mysql.sh → 自动生成 model/ + query/
4. 业务代码使用 query.Q.{Table}.WithContext(ctx).Where(...).Find/Create/Update
```

### 3.3 database/sql 标准事务（Go 标准库）

| 属性 | 说明 |
|------|------|
| **用途** | GORM 底层依赖 database/sql 管理连接池和事务 |
| **为什么提及** | 钱包的资金操作（扣款、加款、冻结、流水写入）必须在**同一个数据库事务**内完成，这是 GORM Transaction 的底层保障 |
| **钱包场景** | `query.Q.Transaction(func(tx *query.Query) error { ... })` 确保余额变更 + 流水写入的原子性 |

### 4.1 Redis（缓存 + 分布式锁）

| 属性 | 说明 |
|------|------|
| **版本** | go-redis/v9 v9.17.2 |
| **连接配置** | ETCD 动态读取（`/slg/conf/redis/`） |
| **工程位置** | `common/pkg/rds/redis_util.go`（工具库，支持全数据结构） |
| **用途** | 缓存、分布式锁、Token 存储、计数器 |
| **为什么用它** | 内存级读写性能（微秒级）；丰富数据结构；原子操作（SetNX 分布式锁）；工程已深度使用 |

**钱包模块具体用途：**

| 用途 | Redis 操作 | 说明 |
|------|-----------|------|
| **币种配置缓存** | Hash（HSet/HGet/HGetAll） | 币种列表查询频繁、变更少，缓存后避免每次查 DB |
| **汇率缓存** | String（Set/Get） + TTL | 汇率有有效期，Cache-Aside + 主动写入双模式 |
| **提现频率限制** | String（Incr） + TTL | 限制单用户提现频率（如每日 N 次） |
| **充值订单防重** | SetNX + TTL | 短窗口内防止重复提交同一笔充值 |
| **余额快照缓存** | String（Set/Get） | 非资金操作的余额展示（读多写少场景） |
| **分布式锁** | SetNX + TTL | 对账任务、批量操作等需要全局互斥的场景 |

**已有的操作封装（直接复用，禁止自行封装 go-redis）：**
```
rds.Set / rds.Get / rds.Del / rds.Expire        — String 操作
rds.HSet / rds.HGet / rds.HGetAll / rds.HDel    — Hash 操作
rds.Incr / rds.IncrBy / rds.Decr                — 计数器
rds.SetNX                                        — 分布式锁（带 TTL）
rds.Pipeline                                      — 批量操作
rds.ZAdd / rds.ZRange / rds.ZRangeByScore        — 有序集合
```

### 4.2 Cache-Aside 模式（工程已有实践）

| 属性 | 说明 |
|------|------|
| **用途** | 读缓存模式：先查 Redis，缺失则查 DB 并回写 Redis |
| **为什么用它** | 业界最广泛的缓存策略，简单可靠，一致性可控 |
| **钱包场景** | 币种列表、汇率查询、提现配置等读多写少的数据 |

**模式示意（工程已有，直接复用）：**
```
读取：Redis GET → 命中返回 → 未命中查 DB → 写入 Redis（设 TTL）
写入：更新 DB → 删除 Redis 缓存（下次读时重建）
```

## 6. 第五类：安全与加密（已有 · 复用）

### 6.1 AES-256-CBC 加密（敏感数据）

| 属性 | 说明 |
|------|------|
| **算法** | AES-256-CBC + Base64 编码 |
| **密钥来源** | ETCD（`/slg/conf/security/sensitive_data_key` + `sensitive_data_iv`） |
| **工程位置** | `common/pkg/utils/crypto_util.go` |
| **用途** | 敏感字段自动加解密 |
| **为什么用它** | AES-256 是业界标准对称加密算法（NIST 认证）；GORM 自定义类型自动触发，业务层无感知；工程已有成熟组件 |
| **钱包场景** | 提现账户（银行卡号、账户名）等敏感字段使用 `EncryptedString` 类型，GORM 写入时自动加密、读取时自动解密 |

**使用方式（零成本复用）：**
```go
// 在 GORM model 中声明字段类型为 EncryptedString
type WithdrawAccount struct {
    AccountNo   utils.EncryptedString  // 自动加解密
    AccountName utils.EncryptedString  // 自动加解密
}
```

## 11. 第十类：需要新引入的组件评估

### 11.1 shopspring/decimal（高精度十进制运算）— 建议引入

| 属性 | 说明 |
|------|------|
| **库地址** | github.com/shopspring/decimal |
| **最新版本** | v1.4.0 |
| **用途** | 高精度十进制运算，避免浮点数精度丢失 |
| **为什么建议引入** | 虽然数据库层面用 bigint 存储最小单位整数（分/聪），但**汇率计算的中间过程**涉及乘除运算，存在溢出和精度丢失风险。decimal 库可作为计算层的安全网 |

**具体场景分析：**

| 场景 | 当前方案（纯整数） | decimal 增强 |
|------|-------------------|-------------|
| 余额加减 | bigint 直接运算，无精度问题 | 不需要 |
| 汇率换算 | 先乘后除（`amount * rate / 10^8`），可能中间值溢出 | decimal 做中间运算，最终截断为整数 |
| 费率计算 | 手续费 = 金额 × 费率，需要确定舍入规则 | decimal 精确控制舍入（Round/Floor/Ceil） |
| 对账差异计算 | 两端金额比对 | 不需要 |

**评估结论：**

| 观点 | 说明 |
|------|------|
| **引入** | 汇率换算和费率计算需要精确的舍入控制，decimal 库是 Go 金融领域事实标准（Stripe Go SDK 使用），防御性编程 |
| **不引入** | 纯整数先乘后除 + int64 范围检查也能覆盖大多数场景，额外依赖增加维护成本 |
| **建议** | **引入但限定使用范围** — 仅在汇率换算和费率计算的中间过程使用，存储和接口层仍然用 bigint 整数 |

**使用示例：**
```go
import "github.com/shopspring/decimal"

// 汇率换算：100 USD → CNY，汇率 7.24（存储为 724000000，即 ×10^8）
amount := decimal.NewFromInt(10000)        // 100.00 USD（最小单位分）
rate := decimal.NewFromInt(724000000)       // 7.24 × 10^8
precision := decimal.NewFromInt(100000000)  // 10^8

result := amount.Mul(rate).Div(precision)   // 精确运算
cnyAmount := result.IntPart()               // 截断为整数：72400（分）= 724.00 CNY
```

|------|---------|------|
| 日终对账 | robfig/cron + GORM Gen | 定时扫描当日流水，与账户余额比对 |
| 异常告警 | Zap 日志（初期）/ NATS 通知（可选） | 对账差异记录到日志，人工巡检 |

## 14. 不引入项及理由

| 序号 | 技术/框架 | 不引入理由 |
|------|----------|-----------|
| 1 | **Seata / DTM（分布式事务框架）** | 引入重、运维复杂。钱包的跨服务一致性通过"幂等 RPC + 重试 + 定时补偿"三板斧已可覆盖，这也是支付宝/微信支付的主流做法。Seata 适合更复杂的多库多表场景，当前不需要 |
| 2 | **Event Sourcing（事件溯源）** | 架构复杂度高，学习和维护成本大。transaction_log 流水表已满足所有审计和溯源需求，本质上是简化版的事件记录，够用 |
| 3 | **CQRS（读写分离架构）** | 钱包的读写量级相当（不是读远大于写或写远大于读的极端场景），引入 CQRS 增加架构复杂度但收益有限。TiDB 本身具备分布式读写能力 |
| 4 | **数据库读写分离中间件** | TiDB 是分布式数据库，自身已具备水平扩展和读写分离能力（TiKV 多副本），无需额外引入 ProxySQL 或 ShardingSphere 等中间件 |
| 5 | **独立限流组件（Sentinel / go-rate）** | 网关层已有请求频率控制（IP 黑名单 + 认证中间件）；钱包层面的提现频率限制用 Redis Incr + TTL 即可实现，不需要专门的限流框架 |
| 6 | **ElasticSearch（全文搜索）** | 钱包业务没有全文搜索需求，交易流水查询通过 DB 索引覆盖。工程中 `common/pkg/es/` 虽已有 ES 封装，但钱包场景不需要 |

## 15. 总结

### 15.1 核心原则

**三个原则贯穿技术栈选择：**

1. **复用优先** — 工程已有的成熟组件优先复用，不重复造轮子（18 项直接复用）
2. **够用就好** — 不引入比问题更复杂的方案（6 项明确不引入）
3. **有据可依** — 每项选型都有工程现状或行业实践作为依据

### 15.2 一句话总结

> ser-wallet 的技术栈核心是 **Kitex RPC + GORM/TiDB + Redis + ETCD + Cron** 的组合，全部来自工程已有基础设施。唯一建议新引入的是 **shopspring/decimal**（汇率计算精度保障）。资金安全靠 **DB 悲观行锁 + 事务 + 唯一索引幂等**，不需要引入分布式事务框架等重量级组件。
