# 二、复用现有 — 框架层（4 项）

## 2.1 Kitex — RPC 框架

```
来源: common/pkg/go.mod → cloudwego/kitex v0.16.0
代码: common/rpc/init.go、common/rpc/rpc_client.go

在钱包模块中的用途:
  ① ser-wallet 作为 Kitex 服务端，接收来自网关和其他服务的 RPC 调用
  ② ser-wallet 作为 Kitex 客户端，调用财务模块、ser-kyc 等外部服务
  ③ 提供 Credit/Debit/Freeze/Unfreeze 等 RPC 接口给财务模块调用

为什么用它:
  · 项目统一的 RPC 框架，所有 12 个服务都用 Kitex（无第二选择）
  · CloudWeGo 生态，字节跳动开源，高性能、生产验证
  · 内建 TTHeader 元数据透传，支持链路追踪上下文传递
  · 支持 Thrift IDL，与项目的 IDL-first 开发模式匹配

钱包模块具体使用方式:
  · 服务端: rpc.InitRpcServerParams("wallet_service", port) → 启动 Kitex Server
  · 客户端: 在 common/rpc/rpc_client.go 新增 WalletClient() 惰性单例
  · 超时: 默认 3 秒（InitRpcClientParams 中已配置），满足余额操作通常 <50ms 的需求
  · 多路复用: WithMuxConnection(1) + WithMuxTransport()（已配置）
```

## 2.2 Hertz — HTTP 网关框架

```
来源: gate-back/go.mod、gate-font/go.mod → cloudwego/hertz v0.10.3
代码: gate-back/biz/、gate-font/biz/

在钱包模块中的用途:
  ① gate-font 新增钱包 C 端路由（/api/wallet/*）— 查余额、充值、提现、兑换、记录
  ② gate-back 新增钱包 B 端路由（/admin/api/wallet/*）— 币种管理、汇率配置、管理员查询

为什么用它:
  · 项目统一的 HTTP 网关框架，两个网关都是 Hertz（无第二选择）
  · 与 Kitex 同属 CloudWeGo 生态，无缝集成
  · 内建参数绑定（BindAndValidate）、路由分组、中间件链

钱包模块具体使用方式:
  · C 端: 在 gate-font/biz/router/ 下新增 wallet 路由注册文件
  · B 端: 在 gate-back/biz/router/ 下新增 wallet/currency 路由注册文件
  · Handler 编写: 复用 rpc.Handler[REQ, RESP](ctx, c, rpc.WalletClient().Method) 一行委托模式
  · 路由规范: POST 统一、/{module}/{entity}/{action} 命名
```

## 2.3 GORM + GORM Gen — ORM 框架

```
来源: common/pkg/go.mod → gorm.io/gorm v1.31.1 + gorm.io/gen v0.3.27
代码: common/pkg/db/mysql/mysql.go、common/pkg/db/cmd/generate.go

在钱包模块中的用途:
  ① user_wallet / wallet_flow / deposit_order 等 12 张表的模型自动生成
  ② 19 个标准 CRUD 方法自动生成（Create/Update/Delete/Get/List/Page/Count/Exists）
  ③ 事务管理（query.Q.Transaction）用于 ledger 层的原子操作
  ④ UserPlugin 自动填充审计字段（create_by/create_at/update_by/update_at）

为什么用它:
  · 项目统一的 ORM（所有服务的数据访问层都通过 GORM Gen 生成）
  · 类型安全: Gen 生成的 query 层避免手写 SQL 字符串
  · SkipDefaultTransaction: true — 不自动开事务，ledger 层手动控制
  · UserPlugin 自动审计 — 从 context 中取 UserID，自动填充到审计字段

钱包模块具体使用方式:
  · 代码生成: 运行 common/pkg/db/cmd/generate.go → 生成 ser-wallet/internal/gorm_gen/
  · 模型目录: ser-wallet/internal/gorm_gen/model/（*.gen.go）
  · 查询目录: ser-wallet/internal/gorm_gen/query/（类型安全查询 API）
  · 仓储目录: ser-wallet/internal/gorm_gen/repo/（19 个标准 CRUD 方法/表）
  · 扩展查询: ser-wallet/internal/rep/（手写复杂查询，如 FOR UPDATE 带锁查询）
  · 事务模式: query.Q.Transaction(func(tx *query.Query) error { ... })
  · 加密字段: gen.FieldType("receiver_info", "utils.EncryptedString")

关键配置:
  FieldNullable: true        → 可空字段映射为 Go 指针类型（*string、*int64）
  FieldSignable: false       → 不区分有符号/无符号
  SkipDefaultTransaction: true → 不自动包事务（手动控制，性能优化）
```

# 三、复用现有 — 存储层（3 项）

## 3.1 TiDB — 分布式数据库

```
来源: 运行环境（MySQL 协议兼容），ETCD 配置 /slg/conf/tidb/*
代码: common/pkg/db/mysql/mysql.go（通过 MySQL 驱动连接）

在钱包模块中的用途:
  ① 存储所有钱包业务数据（user_wallet、wallet_flow、订单表、币种配置等 12 张表）
  ② SELECT FOR UPDATE 行锁 — 并发余额操作的一致性保障
  ③ 唯一索引 — wallet_flow 表幂等控制
  ④ 本地事务 — ledger 层原子操作（余额变更 + 流水记录同一事务）

为什么用它:
  · 项目统一的数据库（所有服务都用 TiDB，无第二选择）
  · 分布式架构: 自动数据分片（不需要手动分库分表）
  · MySQL 兼容: 完全支持 SELECT FOR UPDATE、UNIQUE KEY、事务等
  · 水平扩展: 千万级流水表对 TiDB 来说是小数据量

钱包模块为什么不需要分库分表:
  · TiDB 本身就是分布式数据库，数据按 Region（~96MB）自动分片
  · 与 MySQL 分库分表方案（ShardingSphere 等）不同，TiDB 对应用层完全透明
  · wallet_flow 表即使增长到亿级，TiDB 原生处理能力足够

关键数据库参数（已在 common/pkg/db/mysql/ 中配置）:
  MaxIdleConns: 10
  MaxOpenConns: 20
  MaxLifetime: 1h
  PreparedStmt: true
  SlowThreshold: 200ms
```

  · go-redis v9 封装完善: String/Hash/List/Set/SortedSet/Pipeline/Lua 全覆盖
  · 已有 SetNX 分布式锁基础能力（rds.SetNX）

钱包模块使用的 Redis 能力清单:

  已有，直接用:
  ┌──────────────────┬───────────────────────────────────────────────┐
  │ Redis API        │ 钱包中的用途                                    │
  ├──────────────────┼───────────────────────────────────────────────┤
  │ Set/Get/Del      │ 余额缓存、币种配置缓存、汇率缓存的读写和失效    │
  │ Incr + Expire    │ 接口频控（充值 3次/min，提现 5次/hour）          │
  │ SetNX            │ 定时任务分布式防重（汇率拉取、对账脚本）          │
  │ HSet/HGet/HGetAll│ 结构化缓存（用户钱包信息多字段缓存）             │
  │ Eval (Lua)       │ 复杂原子操作（如频控+过期的原子脚本）            │
  │ Pipeline         │ 批量操作（缓存预热、批量失效）                   │
  │ Exists/Ttl       │ 缓存状态检查                                   │
  └──────────────────┴───────────────────────────────────────────────┘

## 4.1 Snowflake — 分布式 ID 生成

```
来源: common/pkg/utils/snowflake.go（自研实现 + bwmarrin/snowflake 库）
代码: utils.GenerateSnowflakeID()、utils.GenerateUniqueID()

在钱包模块中的用途:
  ① 所有表的主键 ID（user_wallet.id、wallet_flow.id、deposit_order.id 等）
  ② 订单号前缀: "C" + snowflake（充值），"T" + snowflake（提现），"E" + snowflake（兑换）

为什么用它:
  · 项目统一的 ID 生成策略（所有表主键都是雪花 ID）
  · 全局唯一 + 时间有序 + 不依赖 DB 自增（TiDB 分布式场景下自增有性能问题）
  · 41 位时间戳 + 5 位数据中心 + 5 位机器 + 12 位序列 = 69 年不重复

特别说明:
  · 不使用数据库自增（autoIncrement:true 在 TiDB 分布式场景下有热点问题）
  · 订单号用雪花 ID 的字符串形式 + 业务前缀，兼顾唯一性和可读性
```

# 十、新增引入（1 项）

## 10.1 shopspring/decimal — 高精度十进制计算

```
库地址: github.com/shopspring/decimal
版本: v1.4.0（最新稳定版）
GitHub Stars: 6,400+（Go 生态最流行的精确计算库）

当前工程状态:
  · 确认不存在 — common/pkg/go.mod 中没有 shopspring/decimal
  · 确认不存在 — 全工程无任何 decimal 相关引用
  · 现有工程中没有任何高精度金额计算的需求（之前的服务不涉及金融计算）

为什么必须引入:

  钱包模块的汇率计算需求:
    · 实时汇率: 1 VND = 0.00004312 USDT（高精度小数）
    · 兑换计算: 100 VND × 0.00004312 × 10 = 0.04312 BSB
    · 浮动比例: 充值浮动 ±0.5%，提现浮动 ±0.3%
    · 这些计算用 float64 会产生精度丢失（金融场景不可接受）

  用在哪里:
    ① 汇率存储: 数据库用 VARCHAR(32) 存高精度汇率字符串
    ② 汇率计算: 从 DB 读取汇率字符串 → decimal.NewFromString() → 精确乘除
    ③ 金额转换: 展示值和存储值之间的转换（÷ 10^decimals）
    ④ 浮动比例计算: 实时汇率 × (1 ± 浮动比例) = 充值/提现汇率

  为什么不用其他方案:
    · math/big: Go 标准库，但 API 过于底层（需手动管理精度/四舍五入）
    · float64: 精度丢失问题，金融场景绝对不可用
    · BIGINT 整数: 余额存储可以用整数，但汇率计算的中间步骤必须用 decimal

  核心 API:
    rate, _ := decimal.NewFromString("0.00004312")    // 从字符串创建
    amount := decimal.NewFromInt(100)                  // 从整数创建
    result := amount.Mul(rate)                         // 精确乘法
    result = result.Round(8)                           // 四舍五入到指定位数
    intVal := result.Shift(8).IntPart()                // 转换为最小单位整数

引入方式:
  在 ser-wallet/go.mod 中添加:
    require github.com/shopspring/decimal v1.4.0

  不需要加到 common/pkg/go.mod（只有钱包模块需要，不污染公共包）

行业参考:
  · Stripe: 内部使用精确计算库处理多币种汇率
  · 支付宝: Java 端使用 BigDecimal，Go 端使用 shopspring/decimal
  · 数据库金额用 BIGINT 整数，中间计算用 decimal，这是业界标准组合
```

# 十一、明确不引入（6 项）

> 这些技术我们评估过但确认不需要，说明理由，避免后续反复讨论

## 11.1 分布式事务框架（TCC/SAGA/Seata）

```
不引入原因:
  · 钱包所有表在同一个 TiDB 库中 → 本地事务（BEGIN/COMMIT）完全覆盖
  · 兑换的 3 个钱包行都在 user_wallet 表中 → 单表多行操作，本地事务足够
  · 跨服务调用（钱包 ↔ 财务）通过幂等 + 重试解决，不需要分布式事务
  · 分布式事务增加巨大复杂度（TCC 需要实现 Try/Confirm/Cancel），收益为零
```

## 11.2 消息队列（Kafka / NATS）

```
不引入原因（钱包模块不使用，尽管工程中已有）:
  · 现有工程的所有服务间通信都是同步 RPC（Kitex）
  · 充值回调: 财务模块 → RPC 调用 ser-wallet.Credit → 同步返回（可靠且简单）
  · 异步消息引入额外复杂度: 消息丢失/重复/顺序保证、消费者管理
  · 钱包场景下同步 RPC + 幂等 = 满足所有需求
  · 如果未来需要异步通知（如充值成功推送），可以由通知服务消费 wallet 事件，但 wallet 本身不需要引入 MQ
```

## 11.3 Elasticsearch

```
不引入原因:
  · 钱包的查询模式是按用户 + 币种 + 时间范围查询 → 数据库索引完全覆盖
  · wallet_flow 表有 idx_user_currency_time 索引 → 精准匹配 + 时间范围排序
  · 不存在全文检索需求（不需要搜索"金额包含某个关键字"）
  · TiDB 的查询性能对于钱包场景绰绰有余
```

## 11.4 Redlock 分布式锁库

```
不引入原因:
  · 并发控制用数据库行锁（SELECT FOR UPDATE）→ 不需要分布式锁
  · 定时任务防重用 XXL-JOB（框架自带分布式调度）→ 不需要分布式锁
  · 少数需要防重的场景（如汇率定时拉取）用 rds.SetNX 就够了（已有）
  · Redlock 增加复杂度且有争议（Martin Kleppmann vs Antirez 的经典争论）
```

## 11.5 独立限流中间件库（golang.org/x/time/rate 等）

```
不引入原因:
  · 钱包的限流需求简单: 按用户+操作类型的频控
  · 用 Redis INCR + EXPIRE 即可实现（rds 包已有这两个操作）
  · 不需要令牌桶/漏桶等复杂算法
  · 如果未来需要更精细的限流，再引入也不迟
```

## 11.6 复式记账框架

```
不引入原因:
  · 复式记账（Double Entry）适用于有多方账户互转的场景（如平台账户 ↔ 用户账户）
  · 本项目的钱包是单边账户模型（只跟踪用户余额）
  · 充值: 外部资金 → 用户钱包（不需要记录"平台收入"账户）
  · 提现: 用户钱包 → 外部（不需要记录"平台支出"账户）
  · 余额账户制（balance + flow）完全满足需求，复式记账是过度设计
```

# 十四、总结

```
技术栈选择的核心原则:

  ① 项目已有的，全部复用，一个不少
     · 18 项现有技术直接使用，不重复造轮子
     · 包括框架、存储、工具、安全、运维等各个层面

  ② 确实没有且必须有的，精准引入
     · 只引入 1 项新依赖: shopspring/decimal（高精度计算）
     · 原因: 汇率计算是钱包核心需求，float64 精度不可接受，工程中无替代品

  ③ 可以不要的，坚决不引入
     · 6 项评估后明确不引入
     · 每一项都有清晰的"为什么不需要"的理由
     · 不因为"大厂在用"或"行业最佳实践"就盲目引入

  钱包模块的技术栈本质上是:
    "基于项目已有的全套基础设施 + 补一个 decimal 库"

  这说明项目的基础设施建设是扎实的:
    Kitex + Hertz + GORM Gen + Redis + ETCD + Snowflake + AES + TTHeader + ...
    已经覆盖了微服务开发所需的各个层面。

  钱包模块的技术复杂度不在"用了什么新技术"，
  而在"怎么用好已有技术来保障资金安全":
    · GORM 事务 + 行锁 → 并发安全
    · wallet_flow 唯一索引 → 幂等
    · BIGINT + decimal → 金额精度
    · AES + 脱敏 → 数据安全
    · Cache-Aside + 短 TTL → 缓存一致性
    · XXL-JOB + 对账脚本 → 运维保障
```
