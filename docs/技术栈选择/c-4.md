# ser-wallet 技术栈选择

> 来源：11 份 c-4 分析文档 + 工程全量代码扫描（12 模块 go.mod + common/pkg 全部组件 + 中间件初始化代码）
> 定位：明确钱包模块实现需要用到的所有技术/中间件/组件，区分"工程已有可直接复用"与"需新引入"

## 一、核心框架层

### 1.1 Kitex — RPC 框架

| 维度 | 详情 |
|------|------|
| **版本** | v0.16.0 |
| **工程位置** | 所有 `ser-*` 服务 + `common/rpc/` |
| **钱包用途** | ser-wallet 全部 12 个 RPC 接口（8 个钱包域 + 4 个币种域）的服务端实现，以及调用 ser-finance、ser-kyc、ser-user 等外部服务的客户端调用 |
| **为什么用它** | 工程统一选型，字节跳动开源，高性能（Netpoll 网络库），原生支持 Thrift IDL，与 ETCD 注册中心深度集成（kitex-contrib/registry-etcd） |

**钱包模块具体使用场景**：

```
服务端（接收请求）：
  - WalletService.GetBalance          ← gate-font / 其他 ser-* 调用
  - WalletService.Deposit             ← ser-finance 充值回调
  - WalletService.Withdraw            ← gate-font 用户发起提现
  - WalletService.Exchange            ← gate-font 用户发起兑换
  - WalletService.ManualCredit        ← gate-back 运营加款
  - WalletService.ManualDebit         ← gate-back 运营减款
  - WalletService.GetCurrencyConfig   ← ser-finance / gate-font 查询币种
  - WalletService.UpdateRate          ← gate-back / 定时任务更新汇率
  等 12 个接口

客户端（调用他人）：
  - ser-finance → 查询支付通道、提现审核状态
  - ser-kyc     → 查询用户 KYC 等级（提现前置校验）
  - ser-user    → 查询用户基础信息
```

**复用说明**：`common/rpc/` 已有 21 个 RPC 客户端工厂方法，钱包模块只需在 `common/rpc/` 注册 `ser-wallet` 客户端（如果 ser-finance 需要调我），以及使用已有的客户端工厂方法调用外部服务。零额外配置。

## 二、数据存储层

### 2.1 TiDB — 关系数据库

| 维度 | 详情 |
|------|------|
| **版本** | 兼容 MySQL 协议（通过 GORM MySQL 驱动 v1.6.0 连接） |
| **工程位置** | `common/pkg/db/mysql/mysql.go` 统一初始化 |
| **钱包用途** | 全部 10 张表的主存储（currency_config / wallet / wallet_transaction / deposit_order / withdrawal_order / withdrawal_account / exchange_order / rate_change_log / wager_requirement / wallet_daily_snapshot） |
| **为什么用它** | 工程统一选型，所有服务都使用 TiDB。水平扩展能力强，兼容 MySQL 语法，天然支持在线 DDL。对于钱包这种写入量大的场景（流水表高增长），TiDB 的分布式存储比单机 MySQL 更有优势 |

**钱包模块具体关注点**：

```
1. 连接配置：从 ETCD 读取 /slg/conf/tidb/addr, port, database
2. 连接池：复用默认 MaxIdle=10, MaxOpen=20, MaxLifetime=1h
   （首版够用，流水表增长后可按需调大）
3. 事务模式：SkipDefaultTransaction=true（GORM 默认不开事务，需要时手动开）
   对于余额操作，必须手动开事务 + SELECT FOR UPDATE
4. 慢查询：200ms 阈值，自动记录
5. 字符集：utf8mb4 + utf8mb4_bin（区分大小写排序）
```

**复用说明**：`common/pkg/db/mysql/mysql.go` + `grom_log.go` + `user_plugin.go` 三件套直接复用。ser-wallet 只需 `internal/cfg/mysql.go` 初始化连接。

### 2.2 GORM — ORM 框架

| 维度 | 详情 |
|------|------|
| **版本** | v1.31.1 |
| **工程位置** | 全部 `ser-*` 服务 |
| **钱包用途** | 数据库 CRUD、事务管理、乐观锁校验、审计字段自动填充 |

**钱包模块核心使用模式**：

```go

   Where("user_id = ? AND currency_code = ? AND wallet_type = ?", uid, code, wType).
   First(&wallet)

// 2. 乐观锁（防并发更新）
result := tx.Model(&wallet).
   Where("version = ?", oldVersion).
   Updates(map[string]interface{}{
       "balance": newBalance,
       "version": oldVersion + 1,
   })
if result.RowsAffected == 0 { /* 冲突重试 */ }

// 3. 事务（余额变更 = 更新余额 + 写流水，同一事务）
db.Transaction(func(tx *gorm.DB) error {
    // step1: SELECT FOR UPDATE 锁定钱包行
    // step2: 更新余额 + version
    // step3: 写入 wallet_transaction（带 before/after）
    return nil
})

// 4. 审计字段自动填充（UserPlugin 已有）
// create_by / update_by / create_at / update_at 自动写入
```

### 3.1 Redis — 业务缓存 + 分布式锁

| 维度 | 详情 |
|------|------|
| **版本** | go-redis v9.17.2 |
| **工程位置** | `common/pkg/rds/redis_init.go` + `redis_util.go` |
| **钱包用途** | 6 大用途（下面详细展开） |
| **为什么用它** | 工程统一选型，已有完整的 Redis 操作封装（字符串/Hash/List/Set/ZSet/管道/Lua），连接池和初始化全部就绪 |

**钱包模块 6 大 Redis 使用场景**：

```
场景 1：币种配置缓存
  Key:    wallet:currency:config:{currency_code}
  结构:   JSON (CurrencyConfig 完整对象)
  TTL:    1h（ETCD Watch 或 B端更新时主动刷新）
  用途:   C端查询可用币种列表、余额展示时获取精度/符号
  操作:   Set / Get / Del（更新时删除）

场景 2：汇率缓存
  Key:    wallet:rate:{from}:{to}
  结构:   JSON (汇率值 + 更新时间 + 快照版本)
  TTL:    5min（频率较高的币对）~ 30min（稳定币对）
  用途:   兑换时获取汇率，避免每次查库
  操作:   Set / Get

场景 3：幂等键去重
  Key:    wallet:idempotent:{idempotency_key}
  结构:   String (订单ID 或 "processing")
  TTL:    24h
  用途:   充值回调/兑换请求幂等校验，防止重复入账
  操作:   SetNX（原子检查并设置）

场景 4：分布式锁
  Key:    wallet:lock:balance:{user_id}:{currency_code}:{wallet_type}
  结构:   String (唯一标识，UUID)
  TTL:    10s（含续期机制）
  用途:   极端高并发场景下的余额操作互斥（行锁之上的二级保护）
  操作:   SetNX + Lua 原子解锁（详见第四节加强方案）

场景 5：提现频率限制
  Key:    wallet:withdraw:limit:{user_id}:{date}
  结构:   String (当日提现次数)
  TTL:    到当日 23:59:59
  操作:   Incr + Expire（原子递增 + 过期）

场景 6：用户钱包摘要缓存（可选）
  Key:    wallet:summary:{user_id}
  结构:   JSON (所有币种余额摘要)
  TTL:    30s（短 TTL，余额变更时主动删除）
  用途:   首页/钱包页快速展示，减少多表联查
  操作:   Set / Get / Del
```

**Redis Key 规范**：遵循 `common/pkg/consts/redis_key/redis_key.go` 的统一命名规范，新增钱包相关 Key 常量。

## 四、分布式锁（需加强）

### 4.1 现状分析

| 维度 | 详情 |
|------|------|
| **当前实现** | `common/pkg/rds/redis_util.go` 中 `SetNX(key, value, expiration)` 返回 bool |
| **问题** | ① 无安全解锁（任何人都能 Del 掉别人的锁）② 无续期（业务执行超过 TTL 锁自动失效）③ 无唯一标识绑定 |

**这不是"缺少分布式锁"，而是"现有的不够完整"。**

### 4.2 钱包场景为什么需要加强

```
场景：用户同时在两个设备发起提现
  设备A → gate-font-实例1 → ser-wallet-实例1 → SELECT FOR UPDATE（行锁 OK）
  设备B → gate-font-实例2 → ser-wallet-实例2 → SELECT FOR UPDATE（行锁 OK）

行锁本身就能保证安全，那分布式锁的价值是什么？
  → 快速失败。行锁需要等待（另一个事务提交后才释放），分布式锁可以立即返回"操作中，请稍后"
  → 防止长事务。如果业务涉及外部 RPC 调用（如查 KYC），事务时间不可控，行锁期间占连接
  → 用户体验。秒级返回"操作中"比等待 3-5 秒好

结论：行锁是安全底线，分布式锁是体验优化。两层防线，不是二选一。
```

### 4.3 加强方案

在 `common/pkg/rds/` 中新增完整的分布式锁封装（不引入新依赖，基于现有 Redis Eval/Lua 能力）：

```
新增文件：common/pkg/rds/redis_lock.go

核心能力：
  1. Lock(key, uniqueID, ttl) → 基于 SetNX + 唯一标识
  2. Unlock(key, uniqueID)    → Lua 脚本原子校验+删除（只有持有者能解锁）
  3. TryLock(key, uniqueID, ttl) → 非阻塞尝试
  4. 可选：Renew(key, uniqueID, ttl) → 续期（如果业务执行时间可能超 TTL）

Lua 解锁脚本（已有 Eval 封装支持）：
  if redis.call("get", KEYS[1]) == ARGV[1] then
      return redis.call("del", KEYS[1])
  else
      return 0
  end

技术依赖：
  - redis_util.go 的 SetNX（已有）
  - redis_util.go 的 Eval（已有，支持 Lua 脚本执行）
  - 零新依赖
```

**为什么不用 Redlock**：单 Redis 实例/集群场景下，SET NX + Lua 解锁已经足够。Redlock 适用于多个独立 Redis 实例的场景，我们用的是 Redis 集群（`NewUniversalClient` 支持），单次 SetNX 即可保证。

**为什么不引入 redsync 等第三方库**：改动量太小（一个文件 50 行代码），不值得引入新依赖。现有的 `SetNX` + `Eval` 组合已经能实现。

### 5.1 Kafka — 资金事件流（重量级）

| 维度 | 详情 |
|------|------|
| **版本** | sarama v1.46.3 |
| **工程位置** | `common/pkg/kafka/producer.go` + `consumer.go` |
| **钱包用途** | 发布资金变动事件，供下游消费（对账系统、审计系统、数据分析） |
| **为什么用它** | ① 工程已有 ② Kafka 的有序性和持久化能力适合资金事件——不能丢、不能乱序 ③ AsyncProducer + Hash 分区保证同一用户的事件有序 |

**钱包模块 Kafka 使用场景**：

```
Topic 1: wallet.transaction.created
  触发: 每笔交易流水写入成功后
  消息体: {user_id, transaction_type, currency_code, amount, balance_after, created_at}
  分区键: user_id（保证同一用户事件有序）
  消费者: 对账系统（异步校验流水与余额一致性）、数据分析

Topic 2: wallet.order.status.changed
  触发: 充值/提现/兑换订单状态变更时
  消息体: {order_type, order_no, old_status, new_status, updated_at}
  分区键: order_no
  消费者: 财务系统（订单状态同步）、运营看板

Topic 3: wallet.risk.alert（可选）
  触发: 异常操作检测（大额提现、频繁兑换、余额异常等）
  消息体: {user_id, alert_type, detail, timestamp}
  消费者: 风控系统
```

**复用说明**：`producer.go` 的 `Send(topic, key, value)` + `consumer.go` 的 `ConsumerGroup` 直接复用。Producer 已支持幂等、Hash 分区、自动重试。

### 5.2 NATS JetStream — 实时通知（轻量级）

| 维度 | 详情 |
|------|------|
| **版本** | v1.48.0 |
| **工程位置** | `common/pkg/nats/nats.go` + `consumer.go` |
| **钱包用途** | 轻量级实时通知，对时效性要求高但不需要持久化保证的场景 |
| **为什么用它** | ① 工程已有 ② NATS 延迟极低（微秒级），适合实时推送 ③ JetStream 支持持久化但比 Kafka 轻量 ④ 支持延迟发布（`JSPublishDelay`）|

**钱包模块 NATS 使用场景**：

```
Subject 1: wallet.balance.updated
  触发: 用户余额变更后
  消息体: {user_id, currency_code, wallet_type, new_balance}
  用途: 推送到 WebSocket（ser-socket）→ 用户端实时刷新余额显示

Subject 2: wallet.deposit.completed
  触发: 充值到账后
  消息体: {user_id, amount, currency_code}
  用途: 触发站内通知、App Push

Subject 3: wallet.withdraw.status
  触发: 提现状态变更（审核通过/拒绝/到账）
  消息体: {user_id, order_no, new_status, reason}
  用途: 推送到 WebSocket → 用户端实时状态更新
```

**Kafka vs NATS 分工**：

```
Kafka（记账/对账/审计场景）：
  - 不能丢：资金事件必须持久化
  - 要有序：同一用户的事件必须按顺序消费
  - 多消费者：对账、审计、分析各自消费

NATS（实时通知/用户体验场景）：
  - 要快：余额变了立刻推给前端
  - 可容忍丢失：偶尔一条通知没到，前端下次拉取就行
  - 轻量：不需要 Kafka 那么重的基础设施
```

```
任务 1: 汇率同步（必须）

  频率: 每日凌晨 2:00
  逻辑: 遍历所有钱包，校验 balance = SUM(交易流水变动)，不一致的标记告警
  为什么用 XXL-JOB: 执行时间长，需要日志追踪 + 失败重试

任务 3: 日快照生成（可选）
  频率: 每日凌晨 3:00
  逻辑: 生成 wallet_daily_snapshot，保留历史余额快照
  为什么用 XXL-JOB: 批量操作，需要集中调度

任务 4: 过期订单清理（应该）
  频率: 每 10 分钟
  逻辑: 将超时未支付的充值订单标记为过期（pending → expired）
  为什么用 XXL-JOB: 多实例只执行一次
```

## 十八、为什么不需要引入新技术

以下是一些可能会被"想当然"提出但实际不需要的技术：

| 你可能想到的 | 为什么不需要 |
|-------------|-------------|
| **Redlock（分布式锁库）** | 我们用的是 Redis 集群（UniversalClient），单次 SetNX 已保证一致。Redlock 是为多个独立 Redis 实例设计的 |
| **redsync（Go Redlock 库）** | 同上。且改动量太小（50 行），不值得引入新依赖 |
| **decimal 库（shopspring/decimal）** | 我们用 bigint 最小单位存储（分/聪），整数运算无精度问题。decimal 库是给"必须用浮点"的场景准备的 |
| **TCC/Saga 框架（seata-go 等）** | 跨服务一致性用"本地事务 + 幂等重试"够了。TCC 框架引入复杂度远大于收益 |
| **独立对账服务** | 首版用 XXL-JOB 定时任务跑对账脚本即可，独立服务是后期流量大了才考虑的 |
| **MongoDB** | 所有数据都是结构化的（余额、订单、流水），TiDB 完全胜任 |
| **Prometheus + Grafana** | 工程已引入 OpenTelemetry 依赖但未对接后端，这是基础设施团队的事，钱包模块不需要自己搞 |
| **goose/Flyway（数据库迁移）** | 工程统一用手动 SQL 文件管理，不改变现有约定 |
| **本地缓存（bigcache/ristretto）** | Redis 缓存已足够，钱包数据一致性要求高，本地缓存增加失效复杂度 |

## 十九、技术选型决策总结

| 决策维度 | 结论 |
|---------|------|
| **新依赖引入数** | 0 个 |
| **需封装增强数** | 1 个（分布式锁） |
| **直接复用数** | 22 个 |
| **总技术栈覆盖** | 100%（所有需求均有对应技术方案） |

**核心结论**：

1. **工程基础设施完备**。CloudWeGo 全家桶（Hertz + Kitex + Thrift）+ GORM + TiDB + Redis + ETCD + Kafka + NATS 的组合，覆盖了钱包模块从接入到存储到异步到安全的全部需求
2. **不造轮子**。所有中间件封装都在 `common/pkg/` 中，经过其他服务验证，直接复用
3. **唯一需要增强的是分布式锁**。现有 `SetNX` 缺少安全解锁和续期，钱包场景需要在此基础上加 50 行 Lua 封装
4. **不引入不需要的复杂度**。TCC/Saga、Redlock、decimal、独立对账服务等"听起来高大上"的东西，在当前阶段要么不需要，要么用已有方案就能解决
