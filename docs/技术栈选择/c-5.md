```

读操作：GetOneById, GetOneByConds, GetListByIds, GetListByConds,
        GetPageByConds, GetValuesByConds[T], Count, Exists
```

**手写 Rep 层扩展的场景**：

```
余额操作（乐观锁UPDATE）—— 自动生成的 Update 方法不支持条件更新

  → 手写聚合查询（如每日提现累计额）
```

**为什么用 GORM + Gen**：

```
1. 工程统一，7个已有服务全部使用
2. Gen 的类型安全查询：编译期检查字段名和类型，不会出现拼错列名的SQL
3. 代码生成减少样板代码：18+标准 CRUD 方法自动生成
4. UserPlugin 自动填充审计字段：create_by/update_by/create_at/update_at
5. 批量操作支持：BatchCreate/BatchUpdate 内置分批+事务
```

## 3.3 UserPlugin — 审计字段自动填充

| 维度 | 说明 |
|------|------|
| **状态** | 已有，mysql.Init() 时自动注册 |
| **所在位置** | common/pkg/db/mysql/user_plugin.go |
| **钱包用途** | 自动填充所有表的 create_by/update_by/create_at/update_at |

**工作机制**：

```
GORM Callback 插件，注册在 Before Create 和 Before Update 回调上：

Create 回调：
  → 通过反射找到结构体的 CreateBy/CreateAt/UpdateBy/UpdateAt 字段
  → CreateBy = tracer.GetUserId(ctx)     ← 从 RPC 上下文获取当前操作人
  → CreateAt = time.Now().UnixMilli()    ← 毫秒级时间戳
  → 同时设置 UpdateBy 和 UpdateAt

Update 回调：
  → UpdateBy = tracer.GetUserId(ctx)
  → UpdateAt = time.Now().UnixMilli()

支持 *int64（指针类型）和 int64（值类型）两种字段声明
```

**钱包中的注意点**：

```
1. 余额表的 create_by 场景：
   C端用户自己的操作 → UserPlugin 自动从 context 获取 user_id
   RPC 被调用（财务入账）→ context 中的 user_id 是财务服务的操作人
   需确认：context 中传递的是哪个用户ID

2. 流水表只有 create_at，没有 update（Append-Only）
   → 结构体只定义 CreateAt 字段，不定义 UpdateAt
   → UserPlugin 找不到字段就不会填充

3. 审计字段命名必须严格匹配：CreateBy, CreateAt, UpdateBy, UpdateAt
   → 不是 CreatedBy 或 Created_At，是 CreateBy 和 CreateAt
```

## 4.1 Redis (go-redis v9) — 缓存 + 分布式原语

| 维度 | 说明 |
|------|------|
| **状态** | 已有，全部服务共用 Redis |
| **所在位置** | common/pkg/rds/ |
| **钱包用途** | 币种配置缓存、汇率缓存 |

**初始化方式**：

```go
// ser-wallet/internal/cfg/redis.go
func InitRedis() {
    once.Do(func() {
        rds.Init(rds.InitOption{
            Addr:     etcd.DirectGet(ctx, env.RedisAddr),
            Password: etcd.DirectGet(ctx, env.RedisPassword),
        })
    })
}
```

**钱包中的具体使用场景**：

```
场景1：币种列表缓存
  Key:  wallet:currency:list
  Value: 可用币种列表 JSON
  TTL:  5 分钟
  操作: rds.Set() / rds.Get() / rds.Del()

场景2：币种详情缓存
  Key:  wallet:currency:detail:{code}
  Value: 单个币种完整配置 JSON
  TTL:  5 分钟

场景3：汇率缓存
  Key:  wallet:rate:{code}
  Value: 汇率值字符串
  TTL:  1-3 分钟

缓存策略: Cache-Aside
  读: 先 rds.Get() → 命中返回 → 未命中查DB → rds.Set() 写入
  写: 更新DB → rds.Del() 删缓存 → 下次读自动重建
```

**rds 包提供的能力清单（钱包直接用到的）**：

```go
// 基础读写（缓存核心操作）
rds.Set(ctx, key, value, ttl)   → 设置缓存
rds.Get(key)                    → 读取缓存
rds.Del(keys...)                → 删除缓存（写操作后主动失效）

// 计数器（可选：限频/统计）
rds.Incr(key)                   → 自增（如接口调用计数）

// 分布式锁原语（可选：防并发操作）
rds.SetNX(key, value, ttl)      → 如不存在则设置（锁获取）

// 管道批量操作（可选：批量缓存预热）
rds.Pipelined(func(pipe) error) → 批量执行减少RTT

// 底层客户端（需要特殊操作时的逃生通道）
rds.UniversalClient()           → 获取原生 go-redis 客户端
```

**钱包不使用 Redis 的场景**：

```
✗ 用户余额 → 不缓存，必须实时读DB
✗ 订单状态 → 不缓存，状态流转中必须实时
✗ 流水记录 → 不缓存，Append-Only 直接查DB
✗ 幂等控制 → 不用Redis做幂等，用DB唯一键（更可靠）
✗ 分布式锁 → 不需要，余额操作用DB行锁（乐观锁）
```

**为什么不用 Redis 做余额缓存**：

```
1. 余额是高频写数据，缓存几乎永远与DB不一致
2. 用户看到过时余额（充值后余额没变）→ 投诉
3. 余额查询走唯一索引，TiDB < 1ms
4. 缓存带来的性能收益 << 一致性风险
```

```

  SetHeaderInfo:   提取 ClientIP/Language/Token 等写入 metainfo
  WhiteCheck:      IP白名单校验（rpc.IpClient() → ser-ip）
  BackAuthMiddleware:
```

**C端请求中间件链（gate-font）**：

```
SetHeaderInfo → BlackCheck → FontAuthMiddleware → Handler

  BlackCheck:      IP黑名单校验
  FontAuthMiddleware:
    → Redis GET user:token:{token} → 反序列化 CUserTokenPayload
    → 写入 metainfo: UserID
```

**钱包不需要做的**：

```
✗ 不需要自己实现认证 → 网关已经做了
✗ 不需要自己解析Token → tracer.GetUserId(ctx) 直接拿到
✗ 不需要自己做IP校验 → 中间件已经做了
✗ 不需要自己做RBAC → BackAuthMiddleware + CheckUrlAuth 已做

钱包只需要做的：
  C端接口中确保 user_id 来自 context（不信任入参中的 user_id）
  B端写接口调 rpc.BLogClient().AddActLog() 记审计日志
```

## 14.1 Kafka / NATS — 消息队列

| 维度 | 说明 |
|------|------|
| **状态** | 已有封装（common/pkg/kafka/ + common/pkg/nats/）|
| **钱包是否需要** | **不需要** |

```

  1. 钱包与财务的交互是同步 RPC（请求-响应模式）

     不需要通过消息队列通知缓存刷新

什么时候才需要：
  如果未来充值入账改为消息驱动（财务发消息→钱包消费入账）

  当前不需要
```

```
  钱包只存图标的 S3 相对路径（字符串字段），不直接操作 S3
```

# 十六、总结

## 16.1 核心结论

```
1. 不引入任何新组件
   现有 common/pkg 提供的能力完全覆盖钱包模块需求
   Go + Kitex + Hertz + GORM + TiDB + Redis + ETCD + Snowflake + AES
   这套组合就是钱包模块的全部技术栈

2. 新增代码量（common层）非常小
   namesp.go 新增3个常量
   rpc_client.go 新增1个客户端函数
   go.work 新增1行 use
   两个网关各新增2个文件（handler + router）

3. 核心工作在 ser-wallet 内部
   IDL定义 → 桩代码生成 → 建表 → GORM Gen → 业务逻辑
   全部使用已有的工程范式和工具

4. 后续可按需启用的组件
   Cron/XXL-JOB: 定时对账（规模增长后）
   Kafka/NATS: 异步入账（吞吐量瓶颈后）
   这些组件 common/pkg 中已有封装，启用时零引入成本
```

## 16.2 技术栈决策清单

| 决策项 | 选择 | 理由 |
|--------|------|------|
| RPC框架 | Kitex（已有） | 工程统一，21服务全用 |
| HTTP网关 | Hertz（已有） | 工程统一，中间件链完整 |

| ID生成 | Snowflake（已有） | 全局唯一、趋势递增 |
| 加密 | AES-256-CBC（已有） | 工具已封装、密钥ETCD管理 |
| 日志 | Zap（已有） | Kitex侧统一日志 |
