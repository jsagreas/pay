```

  财务 ──RPC──▶ ser-wallet :: CreditWallet / DebitWallet / FreezeBalance ...

作为 RPC 客户端（调用别人）：
  ser-wallet ──RPC──▶ 财务服务 :: GetRechargeMethod / MatchChannel
```

- RPC 中间件：详细日志记录（含 trace_id）

**钱包需做的事**：
- 在 `common/rpc/rpc_client.go` 添加 `WalletClient()` 函数
- ser-wallet 的 main.go 中调用 `InitRpcServerParams("wallet_service", port)`

### 4.1 Redis — 分布式缓存

| 项目 | 详情 |
|------|------|
| **客户端** | github.com/redis/go-redis/v9 v9.17.2 |
| **代码路径** | `common/pkg/rds/redis_init.go`（初始化）、`common/pkg/rds/redis_util.go`（53个函数） |
| **连接模式** | 支持单机和集群（UniversalClient，基于延迟路由） |
| **钱包用途** | 多场景缓存 + 分布式锁 + 防重提交 |
| **为什么用它** | 项目统一缓存组件，高性能（微秒级读写），功能丰富 |

**钱包中的具体用途**：

| 用途 | 使用的 Redis 能力 | 对应函数 | 说明 |
|------|-----------------|---------|------|
| **余额缓存** | GET / SET / DEL | `rds.Get()` `rds.Set()` `rds.Del()` | Cache-Aside 模式，TTL 5min |
| **币种列表缓存** | GET / SET / DEL | 同上 | TTL 1h |
| **汇率缓存** | GET / SET | 同上 | Write-Through，定时任务直接写 |
| **用户当前币种** | GET / SET | 同上 | TTL 7d |
| **分布式锁** | SETNX | `rds.SetNX()` | 防重提交（充值/提现/兑换） |
| **Hash 存储** | HSET / HGET | `rds.HSet()` `rds.HGet()` | i18n 错误消息（已有模式） |
| **Pipeline** | 批量操作 | `rds.Pipelined()` | 批量查询多币种余额 |
| **Lua 脚本** | EVAL | `rds.Eval()` | 复杂原子操作（如需要） |

**Redis Key 规范**（遵循已有模式 `common/pkg/consts/redis_key/`）：

```go
// 钱包模块 Redis Key 规划
const (

    WalletLockRecharge   = "wallet:lock:recharge:" // + {user_id}
```

**钱包需做的事**：

### 5.1 ETCD v3 — 配置中心 + 服务注册发现

| 项目 | 详情 |
|------|------|
| **版本** | v3.6.7 |
| **代码路径** | `common/pkg/etcd/etcd_init.go`（初始化）、`etcd_cache.go`（带缓存的 Watch） |
| **钱包用途** | 三大能力全部使用 |
| **为什么用它** | 项目统一配置中心和服务注册 |

**钱包中的三大能力**：

```
能力1：配置中心
  ├── 数据库连接配置：/slg/serv/wallet/db
  ├── 服务端口：/slg/serv/wallet/port
  ├── Redis 连接：/slg/serv/wallet/redis（或复用全局）
  ├── 汇率拉取频率：/slg/conf/wallet/rate_refresh_interval
  ├── 三方汇率API地址：/slg/conf/wallet/rate_api_urls
  └── 业务开关（如有）：/slg/conf/wallet/feature_flags

能力2：服务注册
  └── ser-wallet 启动时注册到 ETCD，其他服务通过 ETCD 发现

能力3：Watch 动态配置
  └── 使用 Watcher 监听配置变更（如汇率偏差阈值动态调整）
```

**现有基础设施**：
- 初始化：`etcd.InitEtcd()` 读取 `ETCD_ADDR` 环境变量
- 缓存读取：`etcd.Get(ctx, key)` — 带 atomic.Value 本地缓存，无锁并发读
- 动态监听：`etcd.Hook(key, callback)` — 配置变更时执行回调
- 直接操作：`etcd.DirectGet/Put/Delete()` — 绕过缓存直接读写

**钱包需做的事**：
- 在 `common/pkg/consts/namesp/namesp.go` 添加 ser-wallet (022) 的常量：
  ```go
  // EtcdWalletService 022
  EtcdWalletService = "wallet_service"
  EtcdWalletPort    = "/slg/serv/wallet/port"
  EtcdWalletDb      = "/slg/serv/wallet/db"
  ```
- 在 ETCD 中写入对应配置值

### 6.3 钱包不用消息队列的理由

```
1. 资金操作需要同步确认结果，异步消息增加不确定性
2. 充值/提现/兑换都是同步 RPC 模式
3. 审计日志用 Kitex oneway（异步 RPC）已足够
4. 当前没有"高吞吐异步处理"的场景
5. 架构设计文档已明确：当前阶段不引入消息队列
```

## 12. 无需新增 — 钱包模块不需要额外引入的技术

以下技术在钱包场景中可能会被"想当然"地引入，但实际上**不需要**：

| 技术 | 为什么不需要 | 替代方案 |
|------|------------|---------|
| shopspring/decimal（精确计算库） | 金额用 int64 最小单位整数，纯整数运算无精度问题 | Go 原生 int64 |
| 分布式事务框架（Seata/DTM） | 跨服务一致性用幂等+重试解决 | 同步 RPC + 幂等键 |
| 消息队列（NATS/Kafka） | 当前无高吞吐异步场景，资金操作需同步确认 | Kitex 同步 RPC |
| 熔断器（Hystrix/Sentinel） | 调用链不深，超时+降级已足够 | 3s RPC 超时 + 分级降级 |
| 分库分表中间件 | TiDB 天然分布式，无需应用层分片 | TiDB 水平扩展 |
| CQRS 读写分离框架 | 单服务读写量不大 | 标准读写同库 |
| Event Sourcing 框架 | 余额+流水模型已满足审计 | wallet_flow 表 |
| 独立的幂等服务 | 数据库 UNIQUE 约束 + 状态机检查已足够 | wallet_flow.idempotent_key |
| 独立的限流组件 | 网关层已有 IP 管控，Redis SetNX 做业务级防重 | rds.SetNX() |
| 独立的配置管理框架 | ETCD Watch 已满足动态配置 | etcd.Hook() |

### 13.2 sync.Map — 进程内本地缓存

| 项目 | 详情 |
|------|------|
| **场景** | 币种列表极高读极低写，可在进程内缓存减少 Redis 调用 |
| **现有情况** | Go 标准库自带 sync.Map，无需引入任何依赖 |
| **是否必须引入** | 否，是优化手段，不是必需 |

```go
// 进程内缓存币种列表
var currencyCache sync.Map  // key: "all", value: []*CurrencyConfig
var cacheTime     atomic.Int64

func GetCurrencyListCached() []*CurrencyConfig {
    if time.Now().UnixMilli() - cacheTime.Load() < 10*60*1000 {  // 10分钟
        if val, ok := currencyCache.Load("all"); ok {
            return val.([]*CurrencyConfig)
        }
    }
    // 从 Redis 或 DB 加载
    list := loadFromRedisOrDB()
    currencyCache.Store("all", list)
    cacheTime.Store(time.Now().UnixMilli())
    return list
}
```
