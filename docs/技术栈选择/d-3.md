## 1. 技术栈全景总览

### 1.1 核心结论

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│   钱包模块需要的绝大多数核心技术，现有工程已全部具备               │
│   仅需新增 1 个库：shopspring/decimal（金额精确计算）            │
│                                                                 │
│   已有组件：28+（全部复用）                                      │
│   必须新增：1（shopspring/decimal — 源于 DECIMAL(20,8) 架构决策） │
│   可选新增：2（均为增强型，非必需）                                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**为什么 d-3 多了一个必须新增？**

```
d-3 架构决策 (ADR-06)：金额使用 DECIMAL(20,8) + decimal 库
  原因：多币种系统中不同币种最小单位不同
        VND 无小数、BRL 2位小数、USDT 8位小数
        int64 最小单位方案在跨币种换算时极易出错
        decimal 语义直观：1.00 就是 1，0.00000001 就是 0.00000001

  → 数据库：DECIMAL(20,8) 存储
  → Go 侧：需要 shopspring/decimal 库做精确计算
  → 当前 go.mod 中没有此依赖 → 必须新增
```

### 3.1 TiDB（MySQL 兼容）— 关系型数据库

| 项目 | 详情 |
|------|------|
| **驱动** | gorm.io/driver/mysql v1.6.0 |
| **代码路径** | `common/pkg/db/mysql/mysql.go` |
| **连接配置** | ETCD 读取（如 `/slg/serv/wallet/db`） |
| **钱包用途** | 钱包全部数据的持久化存储（余额、流水、订单、配置等），**唯一数据源** |
| **为什么用它** | 项目统一数据库，MySQL 兼容（所有 SQL 写法通用），天然分布式支持水平扩展 |

**钱包中的关键能力**：

| 能力 | TiDB 支持情况 | 钱包用途 |
|------|-------------|---------|
| DECIMAL(20,8) | ✅ 原生支持 | 金额存储（8 位小数统一精度，ADR-06） |
| SELECT FOR UPDATE | ✅ 悲观事务模式（4.0+ 默认） | 余额并发控制 |
| UNIQUE 约束 | ✅ | 幂等键防重 (biz_type, order_no, wallet_type) |
| 事务 ACID | ✅ | 余额+流水同事务原子提交 |
| CHECK 约束 | ✅（5.0+） | 余额非负校验 CHECK(balance >= 0) |
| SUM/比较/排序 | ✅ DECIMAL 原生支持 | 对账 SQL 可直接写 SUM(amount) |
| 水平扩展 | ✅ 天然支持 | 未来数据量增长 |

**d-3 特有 — DECIMAL(20,8) 存储策略**：

```sql
-- DECIMAL(20,8) 容量分析
-- 整数部分 12 位：最大 999,999,999,999（约 1 万亿）
-- 小数部分 8 位：最小精度 0.00000001（USDT 8 位满足）
-- 覆盖所有币种：VND（整数）、BRL（2位）、USD（2位）、USDT（8位）

balance         DECIMAL(20,8) NOT NULL DEFAULT 0.00000000
frozen_amount   DECIMAL(20,8) NOT NULL DEFAULT 0.00000000
amount          DECIMAL(20,8) NOT NULL DEFAULT 0.00000000
```

**d-3 特有 — 余额不缓存（ADR-02）**：

```
核心决策：余额只存 TiDB，不缓存到 Redis
理由：
  1. 资金数据一致性优先级远高于读性能
  2. TiDB 行级查询（主键/唯一索引）耗时 <1ms
  3. 缓存更新策略在并发扣款场景下都有一致性缺陷
  4. C 端用户级 QPS 完全可承受

影响：Redis 仅用于 币种/汇率缓存 + 防重锁，不用于余额缓存
```

**现有初始化模式**（代码实证）：

```go

// 1. 事务 + FOR UPDATE（资金操作的核心模式）
query.Q.Transaction(func(tx *query.Query) error {
    // 悲观锁锁定钱包行
    wallet, _ := tx.Wallet.WithContext(ctx).
        Where(tx.Wallet.UserID.Eq(userID)).
        Where(tx.Wallet.CurrencyCode.Eq(code)).
        Where(tx.Wallet.WalletType.Eq(wType)).
        ForUpdate().First()

    // shopspring/decimal 精确计算
    newBalance := wallet.Balance.Add(amount)

    // 更新余额 + 版本号
    tx.Wallet.WithContext(ctx).
        Where(tx.Wallet.ID.Eq(wallet.ID)).
        Where(tx.Wallet.Version.Eq(wallet.Version)).
        Updates(map[string]interface{}{
            "balance": newBalance,
            "version": wallet.Version + 1,
        })

    // 写流水（同一事务）
    tx.WalletTransaction.WithContext(ctx).Create(&transaction)

    return nil
})
```

### 4.1 Redis — 分布式缓存

| 项目 | 详情 |
|------|------|
| **客户端** | github.com/redis/go-redis/v9 v9.17.2 |
| **代码路径** | `common/pkg/rds/redis_init.go`（初始化）、`common/pkg/rds/redis_util.go`（60+ 方法） |
| **连接模式** | UniversalClient（根据配置自动切换单机/哨兵/集群，基于延迟路由） |
| **钱包用途** | 配置缓存 + 分布式锁 + 防重提交（**不缓存余额** — ADR-02） |
| **为什么用它** | 项目统一缓存组件，高性能（微秒级读写），功能丰富 |

**钱包中的具体用途**（与 d-2 的关键差异：**去掉了余额缓存**）：

| 用途 | 使用的 Redis 能力 | 对应函数 | 说明 |
|------|-----------------|---------|------|
| **币种列表缓存** | GET / SET / DEL | `rds.Get()` `rds.Set()` `rds.Del()` | Cache-Aside，TTL 1h |
| **汇率缓存** | GET / SET | 同上 | Write-Through，定时任务直接写 |
| **用户当前币种** | GET / SET | 同上 | TTL 7d |
| **分布式锁** | SETNX | `rds.SetNX()` | 防重提交（充值/提现/兑换） |

| ~~余额缓存~~ | — | — | **d-3 不缓存余额（ADR-02）** |

**d-3 与 d-2 的差异 — NATS 从"备用"变为"使用"**：

```
d-2 决策：钱包不使用消息队列
d-3 决策（ADR-05）：
  · 资金操作 → 同步 RPC + 幂等（保安全）
  · 事件通知 → NATS JetStream（做扩展）
  · 两者职责清晰分离，互不干扰
```

**钱包中 NATS 的具体用途**（架构设计/d-3.md 第五部分）：

```
ser-wallet 发布:
  Subject: wallet.balance.changed
  Payload: {userId, currencyCode, walletType, balanceAfter, bizType}
  触发时机: 任何 Credit/Debit 成功后

订阅方:
  · 活动服务 → 判断是否触发充值奖金
  · 推送服务 → 余额变动通知推送给用户
  · 统计服务 → 实时资金看板

注意: 这是辅助通道，不是资金操作的主通道
      资金操作必须走同步 RPC，不走消息队列
      即使 NATS 完全宕机，资金操作不受任何影响
```

| 对比维度 | NATS JetStream | Kafka | 钱包选择 |

| 工程封装 | ✅ 完善 | ✅ 完善 | 两者都有 |
| **结论** | **钱包使用** | 备用 | NATS 够用且更轻量 |

## 12. 需要新增 — shopspring/decimal 精确计算库

> 这是 d-3 唯一需要新增的依赖，源于架构决策 ADR-06

### 12.1 组件信息

| 项目 | 详情 |
|------|------|
| **库名** | github.com/shopspring/decimal |
| **最新版本** | v1.4.0（2024年发布） |
| **GitHub Stars** | 6.2k+（Go 社区最成熟的精确 decimal 库） |
| **Go Doc** | https://pkg.go.dev/github.com/shopspring/decimal |
| **License** | MIT |
| **当前工程状态** | ❌ 未引入（common/pkg/go.mod 中不存在） |

### 12.2 为什么必须引入

```
根本原因：ADR-06 决策选择 DECIMAL(20,8) 存储金额

场景：多币种系统中不同币种最小单位不同
  · VND（越南盾）：无小数位，金额都是整数
  · BRL（巴西雷亚尔）：2 位小数
  · USDT（泰达币）：8 位小数
  · BSB（平台币）：可能 2-8 位小数

如果用 int64 表示最小单位：
  VND 1 块 = 1，USDT 1 块 = 100000000
  跨币种换算时极易出错，语义不直观

如果用 float64：
  0.1 + 0.2 = 0.30000000000000004  ← 浮点精度灾难
  资金场景绝对不允许

选择 DECIMAL：
  数据库：DECIMAL(20,8) 原生精确存储
  Go 侧：需要一个能精确计算的 decimal 库 → shopspring/decimal
  二者配合：DB 精确存 → Go 精确算 → Thrift string 精确传 → 前端 string 精确展示
```

### 12.3 shopspring/decimal 核心 API（钱包场景常用）

```go
import "github.com/shopspring/decimal"

// ---- 创建 ----
d := decimal.NewFromString("1500.50000000")    // 从 string 创建（最常用）
d := decimal.NewFromFloat(1500.5)               // 从 float（注意精度）
d := decimal.NewFromInt(1500)                    // 从整数
d := decimal.Zero                                // 零值

// ---- 四则运算（不可变，返回新值）----
result := a.Add(b)                              // a + b（入账）
result := a.Sub(b)                              // a - b（扣款）
result := a.Mul(b)                              // a * b（汇率换算）
result := a.Div(b)                              // a / b（汇率换算）
result := a.DivRound(b, 8)                      // a / b 保留 8 位小数

// ---- 比较 ----
a.LessThan(b)                                   // a < b（余额不足判断）
a.LessThanOrEqual(b)                            // a <= b
a.GreaterThan(decimal.Zero)                     // a > 0（金额正数校验）
a.Equal(b)                                      // a == b
a.IsZero()                                      // a == 0
a.IsNegative()                                  // a < 0（防负数校验）

// ---- 格式化 ----
d.String()                                      // "1500.50000000"
d.StringFixed(2)                                // "1500.50"（VND 展示用）
d.StringFixed(8)                                // "1500.50000000"（USDT 展示用）
d.Truncate(2)                                   // 截断到 2 位小数（不四舍五入）
d.Round(8)                                      // 四舍五入到 8 位小数

// ---- GORM 集成 ----
// decimal.Decimal 已实现 sql.Scanner 和 driver.Valuer 接口
// GORM 自动处理 DECIMAL(20,8) ↔ decimal.Decimal 转换
// GORM Gen 中通过 gen.FieldType("balance", "decimal.Decimal") 配置

// ---- JSON 集成 ----
// decimal.Decimal 已实现 json.Marshaler 和 json.Unmarshaler
// JSON 序列化为字符串 "1500.50000000"（不是 float）
// 与 Sonic JSON、标准库 JSON 均兼容
```

### 12.4 钱包中的典型使用场景

```go
// 场景1：入账（Credit）
func (s *AccountSer) Credit(ctx context.Context, walletID int64, amount decimal.Decimal, ...) error {
    return query.Q.Transaction(func(tx *query.Query) error {
        wallet, _ := tx.Wallet.WithContext(ctx).Where(...).ForUpdate().First()

        // 精确计算新余额
        newBalance := wallet.Balance.Add(amount)

        // 写入流水
        flow := &model.WalletTransaction{
            Amount:        amount,
            BalanceBefore: wallet.Balance,
            BalanceAfter:  newBalance,
        }
        tx.WalletTransaction.WithContext(ctx).Create(flow)

        // 更新余额
        tx.Wallet.WithContext(ctx).Where(...).Updates(...)
        return nil
    })
}

// 场景2：汇率换算
func ConvertCurrency(amount decimal.Decimal, fromRate, toRate decimal.Decimal) decimal.Decimal {
    // amount * fromRate / toRate，保留 8 位小数
    return amount.Mul(fromRate).DivRound(toRate, 8)
}

// 场景3：防超扣
if wallet.Balance.Sub(amount).IsNegative() {
    return ret.BizErr(ctx, errs.BalanceInsufficient)
}

// 场景4：Thrift IDL 金额传输（string ↔ decimal 转换）
amount, err := decimal.NewFromString(req.Amount)  // string → decimal
resp.Balance = wallet.Balance.String()             // decimal → string
```

| 分库分表中间件 | TiDB 天然分布式，无需应用层分片 | TiDB 水平扩展 |
| CQRS 读写分离框架 | 余额不缓存，单服务读写量不大 | 标准读写同库 |
| Event Sourcing 框架 | 余额+流水模型已满足审计要求 | wallet_transaction 表 |
| 独立的幂等服务 | 数据库 UNIQUE 约束 + 状态机检查已足够 | (biz_type, order_no, wallet_type) UNIQUE |
| 独立的限流组件 | 网关层已有 IP 管控，Redis SetNX 做业务级防重 | rds.SetNX() |
| 独立的配置管理框架 | ETCD Watch 已满足动态配置 | etcd.Hook() |
| API Gateway（Kong/Envoy） | Hertz 网关已具备限流/鉴权/路由能力 | gate-font/gate-back |
| gRPC/Protobuf | 项目统一 Thrift IDL + Kitex RPC | 不混用两套 IDL |
