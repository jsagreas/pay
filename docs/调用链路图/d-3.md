# ser-wallet 全链路调用图 — 上帝视角的请求生命周期（d-3 版）

> **文档定位**：屏蔽代码细节，从外部观测者视角，完整描绘钱包服务**每一类请求从出生到消亡的全过程**——开始在哪、中间发生了什么、跑到哪里去了、最终变成什么样。
>
> **文档目的**：作为开发指导方针，让你在编码任何一个接口之前都能准确回答三个问题：
> 1. 这个请求经过了哪些节点？
> 2. 哪个节点最复杂/最关键？
> 3. 哪个节点可能出问题？
>
> **核心认知**：处理一件事无非三种模式——**流程**（步骤 A→B→C）、**判断**（if-else 分支）、**循环**（重试/轮询/定时）。本文档用这三种模式拆解钱包服务的所有链路。

## 1. 系统全景：一个请求的完整旅程

### 1.1 三句话总结

**任何请求都遵循同一个路径**：

```
外部世界 → 网关（鉴权+路由）→ 钱包服务（业务逻辑）→ 数据库/缓存/其他服务 → 原路返回
```

**钱包服务内部都遵循同一个分层**：

```
handler（接收+转发）→ service（编排+判断）→ repository（事务+持久化）
```

**所有资金操作都遵循同一个安全模型**：

```
参数校验 → 幂等查重 → 行锁保护 → 余额变更+流水写入（原子事务）→ 唯一索引兜底
```

## 2. 七层漏斗模型：请求从外到内穿透的每一层

> 一个请求从用户指尖到数据库磁盘，要穿过 7 层。每一层都可能"拦截"请求，不让它继续。

```
Layer 7  用户设备层     │ App/Web 发起 HTTPS POST
         ─────────── │ ↓ 可能被拦：无网络/DNS 失败
Layer 6  TLS/网络层    │ 域名解析 → TCP 建连 → TLS 握手
         ─────────── │ ↓ 可能被拦：证书过期/网络不通
Layer 5  网关中间件层   │ SetHeaderInfo → IP 校验 → Token 鉴权 → RBAC
         ─────────── │ ↓ 可能被拦：黑名单/Token 过期/无权限
Layer 4  RPC 传输层    │ Thrift 序列化 → TTHeader 注入 → ETCD 发现 → 连接池
         ─────────── │ ↓ 可能被拦：服务未注册/超时 3s
Layer 3  Service 层    │ 参数校验 → 业务规则 → 幂等查重 → 跨服务 RPC
         ─────────── │ ↓ 可能被拦：参数非法/余额不足/重复请求/依赖服务挂了
Layer 2  Repository 层 │ 开事务 → FOR UPDATE 行锁 → 数据操作 → COMMIT
         ─────────── │ ↓ 可能被拦：锁等待超时/唯一索引冲突/事务回滚
Layer 1  存储层        │ TiDB 执行 SQL → 写入 Raft Log → 复制 → 持久化
         ─────────── │ 可能被拦：磁盘满/TiDB 集群故障
```

**每层的「守门员」角色**：

| 层 | 守门员 | 拦截条件 | 拦截后果 |
|----|--------|---------|---------|
| L5 | 黑名单中间件 | IP 在黑名单 | 403 Forbidden |
| L5 | 鉴权中间件 | Token 无效/过期 | 401 Unauthorized |
| L5 | RBAC 中间件 | B 端无此 URL 权限 | 403 No Permission |
| L4 | ETCD Resolver | 目标服务无实例 | RPC Discovery Failed |
| L4 | 超时控制 | RPC 超过 3s | Timeout Error |
| L3 | 参数校验 | 金额 ≤ 0 / 币种不存在 | 6022002 ParamInvalid |
| L3 | 余额校验 | 可用余额 < 请求金额 | 6022201 BalanceInsufficient |
| L3 | 幂等校验 | 相同 orderNo 已处理 | 返回首次结果（不重复执行） |
| L2 | FOR UPDATE | 行锁等待超时 | 事务回滚 + 重试 |
| L2 | UNIQUE 索引 | (biz_type, order_no, wallet_type) 冲突 | 幂等返回成功 |

### 5.2 F11 发起提现 — 整个系统最复杂的单接口

```

│  │     user_id, currency_code, wallet_type=1,                               │

│  ├─ 【流程】计算                                                           │
│  │   BSB兑换量 = VND金额 / rate                                           │
│  │   赠送10% = BSB兑换量 × 0.10                                           │
│  │   合计到账 = BSB兑换量 + 赠送量                                         │
│  └─ 返回预览结果（纯计算，不修改任何数据）                                  │
│                                                                            │
│  F16 CreateExchange（确认兑换）★★★ 双钱包原子事务                          │
│  ├─ ① 【判断】参数校验 + 汇率有效性检查                                    │
│  │   └─ 汇率最后更新时间 < 5分钟前？→ 过期则重新获取                       │
│  │                                                                         │
│  ├─ ② 【流程】★ 进入 DB 事务（最关键）                                    │
│  │   BEGIN TRANSACTION                                                     │
│  │   │                                                                     │
│  │   │ a. SELECT * FROM wallet                                             │
│  │   │    WHERE user_id=? AND currency_code='VND' AND wallet_type=1        │
│  │   │    FOR UPDATE;  ← 锁住法币钱包                                     │
│  │   │                                                                     │
│  │   │ b. 【判断】VND余额 ≥ 兑换金额？                                    │
│  │   │    └─ 否 → ROLLBACK → 余额不足                                     │
│  │   │                                                                     │
│  │   │ c. SELECT * FROM wallet                                             │
│  │   │    WHERE user_id=? AND currency_code='BSB' AND wallet_type=1        │
│  │   │    FOR UPDATE;  ← 锁住 BSB 中心钱包                                │
│  │   │    ⚠️ 加锁顺序：按 currency_code 字母序（BSB < VND），防死锁        │
│  │   │    → 实际执行顺序应先 BSB 再 VND！                                  │
│  │   │                                                                     │
│  │   │ d. SELECT * FROM wallet                                             │
│  │   │    WHERE user_id=? AND currency_code='BSB' AND wallet_type=2        │
│  │   │    FOR UPDATE;  ← 锁住 BSB 奖励钱包（赠送部分）                    │
│  │   │                                                                     │
│  │   │ e. UPDATE wallet SET balance -= VND兑换量 (VND中心)                  │
│  │   │ f. UPDATE wallet SET balance += BSB兑换量 (BSB中心)                  │
│  │   │ g. UPDATE wallet SET balance += BSB赠送量 (BSB奖励)                  │
│  │   │                                                                     │
│  │   │ h. INSERT wallet_transaction × 3条                                  │
│  │   │    (VND扣款流水 + BSB中心入账流水 + BSB奖励入账流水)                │
│  │   │                                                                     │
│  │   │ i. INSERT exchange_order × 1条                                      │
│  │   │    (记录汇率快照、兑换前后金额)                                     │
│  │   │                                                                     │
│  │   │ j. INSERT reward_record × 1条（若有赠送）                           │
│  │   │    (记录稽核倍数要求)                                               │
│  │   │                                                                     │
│  │   COMMIT                                                                │
│  │                                                                         │
│  └─ ③ 返回: {exchange_order_no, bsb_amount, bonus_amount, total}           │
│                                                                            │
│  复杂度：★★★☆☆ (双钱包事务有一定复杂度，但无跨服务依赖)                    │
│  挑战：① 多行 FOR UPDATE 的加锁顺序（防死锁）                              │
│        ② 汇率快照一致性（预览到确认之间汇率可能变化）                       │
│        ③ 赠送比例10%是否可配（P0-05待确认）                                │
│                                                                            │
│  最大优势：★★★ 不依赖任何外部 RPC，可完全独立开发和测试                     │
│  → 建议最先开发，用来验证钱包核心能力                                      │
└────────────────────────────────────────────────────────────────────────────┘
```

## 12. 三种模式拆解：流程、判断、循环

### 12.1 流程模式（步骤 A→B→C）

> 占比最高，约 70% 的代码都是顺序流程

```
典型代表：充值入账链路

  参数校验 → 幂等查重 → 开事务 → 行锁 → 计算 → 更新余额 → 写流水 → 提交事务

  特点：每一步都有明确的输入和输出
  编码心法：确保每一步的错误都被捕获并向上传播
```

### 12.2 判断模式（if-else 分支）

> 占比约 25%，是业务复杂度的主要来源

```
典型代表：提现七步校验

  KYC状态？
    ├─ 已认证 → 继续
    └─ 未认证 → 方式是USDT？
                 ├─ 是 → 放行（有限额差异）
                 └─ 否 → 拒绝

  充值来源？
    ├─ 纯法币   → 只允许法币提现
    ├─ 纯USDT   → 只允许USDT提现
    └─ 混合     → 只允许法币提现

  余额充足？
    ├─ 是 → 冻结
    └─ 否 → 拒绝

  编码心法：
    ① 先处理拒绝路径（early return），保持主路径清晰
    ② 每个判断分支都要有明确的错误码和用户提示
    ③ 判断条件尽量从配置/DB中读取，不硬编码
```

### 12.3 循环模式（重试/轮询/定时）

> 占比约 5%，但都是关键场景

```
场景 1：前端轮询充值状态
  while (attempts < 20 && !terminal_status) {
    status = call GetDepositStatus()
    if status in [成功, 失败, 失效] → break
    sleep(3~10秒，递增)
  }

场景 2：财务 RPC 重试（财务侧实现）
  for i := 0; i < 3; i++ {
    resp = wallet.WalletCredit(req)
    if resp.code == 0 → break
    sleep(1s × 2^i)  // 指数退避
  }
  钱包侧保证：重复调用幂等返回成功

场景 3：汇率定时同步（cron）
  每 3 分钟 → 调三方API → 计算 → 更新
  每 5 分钟 → 扫描异常冻结 → 自动解冻

  编码心法：
    ① 循环必须有终止条件（最大次数/超时）
    ② 幂等保证让重试安全
    ③ 定时任务要有监控和告警
```
