### 1.2 一个请求穿越的 7 层

| 层序 | 名称 | 在哪 | 做什么 |
|------|------|------|--------|
| L1 | 网关入口 | gate-font / gate-back | 鉴权、限流、路由转发 |
| L2 | handler | ser-wallet/handler.go | 一行代码委托（纯透传） |
| L3 | ser 业务层 | internal/ser/*.go | 参数校验、业务规则、编排调度 |
| L4 | ledger 记账层 | internal/ledger/ | 余额变更的唯一入口（事务+行锁+幂等+流水） |
| L5 | repo 数据层 | internal/rep/ + gorm_gen/ | SQL 执行 |
| L6 | 存储层 | TiDB + Redis | 持久化 + 缓存 |
| L7 | 外部服务 | ser-finance / ser-kyc / ... | 跨服务 RPC 调用 |

> 普通查询请求走 L1→L2→L3→L5→L6（跳过 ledger），余额变更请求走 L1→L2→L3→L4→L5→L6

## 二、请求入口：一个请求如何进入系统

### 2.1 C端请求生命周期（以"查看钱包余额"为例）

```
时间线    位置              发生了什么
──────    ────              ──────────
T+0ms     用户手机          点击"钱包"页面
T+10ms    gate-font         收到 POST /api/wallet/overview
T+11ms    ├ IGlTracer       生成 UUID trace-id，注入上下文
T+12ms    ├ SetHeaderInfo   提取 IP/Language/Device/Token 到 metainfo
T+13ms    ├ BlackCheck      调 ser-ip 检查 IP 黑名单（异步宽松，失败放行）
T+15ms    ├ FontAuthMiddleware  Redis GET user:token:{token}
          │                     → 解析出 userId=12345
          │                     → 注入 X-User-ID 到 metainfo
T+16ms    └ Handler         rpc.WalletClient().GetWalletOverview(ctx, req)
                             ↓ Kitex RPC 调用（TTHeader 自动携带 8 个上下文头）
T+18ms    ser-wallet         handler.go: GetWalletOverview(ctx, req)
T+19ms    ├ ser/wallet_ser   从 ctx 取 userId，查 Redis 缓存
T+20ms    │                  缓存未命中 → 查 TiDB user_wallet 表
T+22ms    │                  按子钱包类型聚合：中心+奖励=资金钱包，主播钱包，代理钱包
T+23ms    │                  写 Redis 缓存（TTL 60s）
T+24ms    └ 返回             {centerBalance: "50000", rewardBalance: "3000", ...}
                             ↓ RPC 响应
T+26ms    gate-font          ret.HttpOne() 包装为 JSON
T+28ms    用户手机           收到余额数据，页面渲染
```

### 2.2 B端请求生命周期（以"创建币种"为例）

```
时间线    位置              发生了什么
──────    ────              ──────────
T+0ms     B端后台           运营点击"新增币种"→ 提交表单
T+10ms    gate-back         收到 POST /admin/api/wallet/currency/create
T+11ms    ├ IGlTracer       生成 trace-id
T+12ms    ├ WhiteCheck      调 ser-ip 检查 IP 白名单（严格，失败拒绝） ⚠️
T+14ms    ├ BackAuthMiddleware  Redis GET {token} → 解析管理员身份
          │                     调 ser-buser.CheckUrlAuth 验证 URL 权限 ⚠️
T+16ms    └ Handler         RPC → ser-wallet.CreateCurrency(ctx, req)
T+18ms    ser-wallet         ser/currency_ser.go: 校验币种代码唯一性
T+20ms    │                  调 ser-s3.UploadFile 上传图标 → 获取 icon_url
T+22ms    │                  写入 currency_config 表
T+24ms    │                  调 ser-blog.AddActLog 记录审计日志（oneway 异步，不阻塞）
T+25ms    │                  清除 Redis 币种缓存
T+26ms    └ 返回             创建成功
```

### 2.3 TTHeader 跨服务透传（8 个上下文头）

```
每一次 RPC 调用，以下信息自动透传，不需要手动传递：

gate → ser-wallet → ser-finance → ...

携带：
  X-Trace-Id       全链路追踪ID（UUID）
  X-User-ID        用户ID（鉴权中间件注入）
  X-User-Name      用户名
  X-User-Token     Token
  X-Client-IP      客户端真实IP
  X-User-Language  语言偏好
  X-User-Device-ID 设备ID
  X-User-Device-Type 设备类型

意义：任何一个服务在任何一层，都能拿到请求的完整上下文，实现全链路追踪和审计。
```

## 三、服务内部：请求在 ser-wallet 内如何流转

### 3.1 两种请求模式

**模式 A：查询类（不涉及余额变更）**

```
handler → ser → repo/gorm_gen → TiDB/Redis → 返回
                                   │
                              跳过 ledger 层
```

适用于：GetWalletOverview、QueryRecordPage、GetCurrencyDetail、ListActiveCurrencies 等

**模式 B：写入类（涉及余额变更）**

```
handler → ser → ⭐ledger → repo → TiDB → 返回
                  │
            开事务 → 幂等检查 → 行锁 → 改余额 → 写流水 → 提交 → 清缓存
```

适用于：所有 WalletCredit/Debit/Freeze/Unfreeze、CreateExchangeOrder 等

### 3.2 Ledger 层：5 个原子操作的统一模板

```
每一个 ledger 操作都遵循完全相同的 6 步模板：

┌────────────────────────────────────────────────────────────┐
│  BEGIN TRANSACTION                                          │
│                                                            │
│  Step 1: 幂等检查                                           │
│    INSERT INTO wallet_flow (order_no, wallet_type, flow_type, ...)│
│    → 唯一索引 (order_no, wallet_type, flow_type)             │
│    → 重复？返回 "already_processed"（视为成功）              │
│    → 新记录？继续下一步                                      │
│                                                            │
│  Step 2: 行锁                                               │
│    SELECT * FROM user_wallet                                │
│    WHERE user_id=? AND currency_id=? AND wallet_type=?      │
│    FOR UPDATE                                               │
│    → 锁住这一行，其他事务等待                                │
│                                                            │
│  Step 3: 校验                                               │
│    Credit:   无额外校验（加钱永远可以）                       │
│    Debit:    balance >= amount ?                             │
│    Freeze:   balance - frozen >= amount ?                    │
│    Unfreeze: frozen >= amount ?                              │
│    Exchange: 源币种余额 >= amount ?                           │
│                                                            │
│  Step 4: 变更余额                                           │
│    UPDATE user_wallet SET balance = balance ± amount         │
│                                                            │
│  Step 5: 补全流水                                           │
│    UPDATE wallet_flow SET balance_before=?, balance_after=?  │
│                                                            │
│  COMMIT                                                     │
│                                                            │
│  Step 6: 清缓存                                             │
│    Redis DEL wallet:bal:{userId}:{currencyId}               │
└────────────────────────────────────────────────────────────┘
```

## 四、链路一：充值

> 最复杂的链路：跨 4 个系统（用户 → 钱包 → 支付 → 三方），含异步回调
> 全程耗时：秒级~分钟级（取决于用户支付速度）

```

      │               │              │                   │                │
  ①  │──打开充值页──→│              │                   │                │
      │               │──RPC────────→│                   │                │
      │               │              │                   │                │
  ②  │               │              │──GetDepositMethods→│               │

      │               │              │                   │                │
```

### 4.2 每步详解

| 步骤 | 负责方 | 做什么 | 复杂度 | 挑战 |
|------|--------|--------|--------|------|
| ① 打开充值页 | gate→wallet | 鉴权 + 路由 | 低 | — |
| ② 获取充值方式 | wallet→finance | RPC 获取方式列表，钱包侧缓存 5min | 低 | 支付模块未就绪时需 Mock |
| ③ 选方式+金额 | 用户操作 | 前端交互 | — | — |
| ④ 前置校验 | wallet 内部 | 重复转账检测(查DB) + 金额范围校验 | 中 | 重复检测的判定规则需和产品确认 |
| ⭐⑤ 通道匹配 | wallet→finance | 核心瓶颈——如果支付侧无可用通道，整个充值链断裂 | **高** | ⚠️通道全忙/限额/维护时的降级策略 |
| ⑥ 创建订单 | wallet 内部 | 写 deposit_order，生成 C+16位订单号 | 低 | 订单号全局唯一性保证 |
| ⑦ 用户支付 | 用户→三方 | 在三方页面完成支付，**此步不可控** | — | ⚠️用户可能不付款、支付超时 |
| ⑧ 回调接收 | 三方→finance | 支付成功回调，支付模块验签 | 中 | ⚠️回调可能丢失、延迟、重复 |
| ⭐⑨ 入账通知 | finance→wallet | RPC 调 WalletCredit，**资金安全红线** | **最高** | ⚠️幂等必须100%可靠，超时必须正确重试 |
| ⑩ 余额入账 | wallet/ledger | 事务内：幂等检查→行锁→加余额→写流水→提交 | **高** | ⚠️并发安全、流水完整性 |
| ⑪ 奖金入账 | finance→wallet | 可选步骤，入奖励钱包 | 中 | 流水稽核标记（倍数要求） |

### 4.3 充值链路的数据状态变化

```
时间    deposit_order.status    user_wallet.balance    wallet_flow
────    ───────────────────    ──────────────────     ───────────
T6      待支付                  不变                    无
T10     成功                    +amount                 +1条(充值入账)
T11     成功                    +bonus(奖励钱包)        +1条(奖金入账)
```

## 五、链路二：提现

> 最长的链路：从用户发起到最终到账可能跨越数小时
> 涉及冻结-确认模型，是资金安全的核心

### 5.1 完整时序图

```

      │               │              │                 │            │              │
```

### 5.2 每步详解

| 步骤 | 负责方 | 做什么 | 复杂度 | 挑战 |
|------|--------|--------|--------|------|
| ① KYC 查询 | wallet→kyc | 查用户认证状态，决定可用提现方式 | 低 | ⚠️KYC 服务不可用时的降级（仅允许USDT） |
| ② 获取提现方式 | wallet→finance | RPC 获取方式列表 | 低 | 需 Mock |
| ③ 规则过滤 | wallet 内部 | KYC + 资金来源规则双重过滤 | **中高** | ⚠️资金来源追踪——混合充值场景的判定规则复杂 |
| ④ 用户操作 | 前端 | 选方式、填账户、输金额 | — | — |
| ⑤ 五项校验 | wallet 内部 | KYC + 限额 × 2 + 账户 + 姓名一致性 | **高** | ⚠️持有人姓名匹配（忽略大小写/空格） |
| ⭐⑥ 冻结余额 | wallet/ledger | 行锁内冻结，确保不超扣 | **最高** | ⚠️并发多笔提现时的串行化 |
| ⑦ 创建订单 | wallet 内部 | 写 withdraw_order | 低 | — |
| ⭐⑧ 提交审核 | wallet→finance | 冻结后才提交；**如果此步失败必须立即解冻** | **最高** | ⚠️提交失败后解冻的可靠性 |
| ⑨⑩ 三级审核 | finance 内部 | 风控→财务→出款审核（钱包不参与） | — | 钱包只等结果 |
| ⭐⑪a 出款成功 | finance→wallet | WalletDeduct：从冻结金额中扣除（不是从可用余额扣） | **最高** | ⚠️必须幂等，重复调用不可重复扣 |
| ⑪b 驳回 | finance→wallet | WalletUnfreeze：解冻归还用户 | **高** | ⚠️解冻失败 = 用户资金被锁死 |
| ⑪c 出款失败 | finance→wallet | 同 ⑪b | **高** | 同上 |

### 5.3 提现链路的数据状态变化

```
时间    withdraw_order.status    balance    frozen    available(=balance-frozen)
────    ────────────────────    ───────    ──────    ─────────────────────────
T6      待风控审核               1000       200       800     ← 冻结200
T8      待风控审核(已提交)        同上       同上      同上
T11a    成功                     800        0         800     ← 扣除冻结200，总余额减少
T11b    驳回                     1000       0         1000    ← 解冻，总余额不变
T11c    出款失败                 1000       0         1000    ← 同驳回
```

### 5.4 关键设计：冻结-确认模型

```
为什么需要冻结？
  提现从用户发起到出款成功可能经过数小时。
  如果不冻结，用户在审核期间把钱花了，出款时余额不足。

三步模型：
  Freeze  → 锁定金额（用户看到余额减少，但钱还在账户里）
  Deduct  → 真正扣除（出款成功，钱离开账户）
  Unfreeze → 释放锁定（出款失败/驳回，钱归还可用）

每一步都是独立的幂等 RPC，任何一步超时重试都不会出错。
```

## 六、链路三：兑换

> 最纯粹的链路：完全在 ser-wallet 内部闭环，不调任何外部服务
> 但事务最复杂：单个操作涉及 3 次余额变更 + 3 条流水，必须原子

### 6.1 完整时序图

```
     用户          gate-font      ser-wallet (全部内部)
      │               │              │
  ①  │──兑换页面────→│──RPC────────→│
      │               │              │ 查 currency_rate 表
      │               │              │ 计算：1 法币 = X BSB
      │←─展示汇率────│←─────────────│
      │               │              │
  ②  │──输入金额────→│              │ PreviewExchange
      │               │──RPC────────→│
      │               │              │ bsb = amount × rate × 10
      │               │              │ bonus = bsb × 10%
      │←─预览结果────│←─────────────│ (获得XX BSB + 赠送XX BSB)
      │               │              │
  ③  │──确认兑换────→│──RPC────────→│ CreateExchangeOrder
      │               │              │
  ⭐④│               │              │ ┌─── BEGIN TRANSACTION ───┐
      │               │              │ │                         │
      │               │              │ │ 幂等检查(orderNo)      │
      │               │              │ │                         │
      │               │              │ │ 锁行: 法币中心钱包      │
      │               │              │ │ 校验: 法币余额 ≥ amount │
      │               │              │ │ 扣款: 法币余额 -= amount│
      │               │              │ │ 流水: 法币扣减记录      │
      │               │              │ │                         │
      │               │              │ │ 锁行: BSB中心钱包       │
      │               │              │ │ 加款: BSB余额 += bsb   │
      │               │              │ │ 流水: BSB入账记录       │
      │               │              │ │                         │
      │               │              │ │ 锁行: BSB奖励钱包       │
      │               │              │ │ 加款: BSB奖励 += bonus  │
      │               │              │ │ 流水: BSB赠送记录       │
      │               │              │ │                         │
      │               │              │ │ 写入 exchange_order     │
      │               │              │ │                         │
      │               │              │ └─── COMMIT ──────────────┘
      │               │              │
      │               │              │ 清缓存(法币+BSB)
      │←─兑换成功────│←─────────────│
```

### 6.2 每步详解

| 步骤 | 做什么 | 复杂度 | 挑战 |
|------|--------|--------|------|
| ① 查汇率 | 读 currency_rate 表（可走缓存） | 低 | — |
| ② 预览试算 | 纯计算：shopspring/decimal 高精度 | 中 | ⚠️精度——必须用 decimal 库，不能用 float |
| ③ 用户确认 | 前端交互 | — | — |
| ⭐④ 原子兑换 | 单事务 3 次余额变更 + 3 条流水 + 1 条订单 | **最高** | ⚠️死锁防范（多行锁必须统一排序） |

### 6.3 兑换链路的数据状态变化

```
操作前:
  VND 中心钱包: 1,000,000
  BSB 中心钱包: 100
  BSB 奖励钱包: 20

兑换 500,000 VND → BSB (汇率: 1 VND = 0.0004 BSB):
  bsb = 500000 × 0.0004 = 200 BSB
  bonus = 200 × 10% = 20 BSB

操作后（一个事务内完成）:
  VND 中心钱包: 500,000   (-500,000)
  BSB 中心钱包: 300        (+200)
  BSB 奖励钱包: 40         (+20, 需完成X倍流水)

流水记录: 3条
  flow-1: VND中心 -500000 (兑换扣除)
  flow-2: BSB中心 +200    (兑换获得)
  flow-3: BSB奖励 +20     (兑换赠送)
```

### 6.4 死锁防范

```
问题：兑换需要同时锁 3 行（VND中心 + BSB中心 + BSB奖励）

用户A兑换 VND→BSB: 先锁VND，再锁BSB
用户B兑换 BSB→VND: 先锁BSB，再锁VND
→ 死锁！

解决：统一锁排序规则
  按 (currency_id ASC, wallet_type ASC) 排序后依次加锁
  所有兑换操作都按相同顺序锁定，消除环形等待
```

## 十、链路七：汇率定时任务

> 后台自动运行，无用户触发
> 分钟级执行，影响全局汇率展示

### 10.1 汇率更新链路

```
ser-cron / 内置 cron → ser-wallet

流程:
  ① 拉取 3 家三方汇率 API（HTTP 请求，非 RPC）
  ② 取平均值 → 实时汇率
  ③ 判断：|实时汇率 - 平台汇率| / 平台汇率 ≥ 阈值？
     ├─ 否 → 不更新，结束
     └─ 是 → 继续
  ④ 更新平台汇率 = 实时汇率均值
  ⑤ 计算入款汇率 = 平台汇率 × (1 + 入款浮动%)
  ⑥ 计算出款汇率 = 平台汇率 × (1 - 出款浮动%)
  ⑦ UPDATE currency_rate 表
  ⑧ INSERT rate_change_log（审计日志）
  ⑨ DEL Redis 汇率缓存

复杂度: 中
挑战:
  ⚠️ 三方 API 部分不可用时的降级（2 家可用取均值，1 家可用直接用，0 家不更新）
  ⚠️ 汇率突变检测（如 API 返回异常值，需要做合理性校验）
```

## 十一、所有链路的共性模式

### 11.1 三种基本处理模式

正如你所说，处理一件事情无非是**流程、判断、循环**三种模式：

| 模式 | 在钱包中的体现 | 示例 |
|------|---------------|------|
| **流程（顺序）** | 链路中每一步按顺序执行 | 充值：获取方式→匹配通道→创建订单→等回调→入账 |
| **判断（分支）** | 条件决定走哪条路 | 提现：KYC 状态决定方式过滤；审核结果决定 Deduct 还是 Unfreeze |
| **循环（重复）** | 重试机制 + 定时任务 | 回调失败后的指数退避重试；分钟级汇率拉取 |

### 11.2 所有写操作的统一守则

```
规则 1: 余额变更必须经过 ledger 层（唯一入口）
规则 2: ledger 内部必须在事务中完成（锁+改+记 原子化）
规则 3: 每笔余额变更必须有对应的流水记录（审计命脉）
规则 4: 跨服务调用必须幂等（orderNo 唯一索引）
规则 5: 重复调用返回"已处理"不返回"错误"（调用方区分对待）
规则 6: 冻结后的提交如果失败，必须立即解冻（不留孤儿冻结）
规则 7: 先本地操作再远程调用（先冻结再提交审核，而非反过来）
```

### 11.3 所有链路的共性基础设施

```
                    所有链路共用
                    ┌──────────┐
                    │ Trace ID │ ← 全链路追踪（UUID，跨服务透传）
                    │ TTHeader │ ← 8 个上下文头自动跨服务传递
                    │ 幂等索引 │ ← wallet_flow (orderNo, walletType, flowType) 唯一
                    │ 行级锁   │ ← SELECT FOR UPDATE 串行化并发
                    │ 错误码体系│ ← 6023xxx，已处理=6023001（不是错误）
                    │ 审计日志 │ ← wallet_flow（细粒度）+ ser-blog（粗粒度）
                    │ Redis缓存│ ← Cache-Aside，写后清缓存
                    └──────────┘
```

## 十二、复杂度热力图

### 12.1 按链路排序

```
复杂度从高到低：

████████████  提现（冻结-确认模型，三级审核，最长耗时）

██            余额查询（Redis + DB，最简单）
```

### 12.2 按步骤排序（哪些步骤最关键）

| 关键度 | 步骤 | 链路 | 原因 |
|--------|------|------|------|
| ⭐⭐⭐⭐⭐ | ledger 原子操作 | 全部写入链路 | 余额变更的唯一入口，出错 = 资金事故 |
| ⭐⭐⭐⭐⭐ | WalletCredit (入账) | 充值 | 幂等失败 = 重复入账 = 资金损失 |
| ⭐⭐⭐⭐⭐ | Freeze + SubmitWithdraw | 提现 | 冻结后提交失败必须解冻，否则资金锁死 |
| ⭐⭐⭐⭐ | WalletDeduct / Unfreeze | 提现 | 出款结果的最终处理，幂等必须可靠 |
| ⭐⭐⭐⭐ | Exchange 原子事务 | 兑换 | 3 次变更必须全成功或全失败，死锁防范 |
| ⭐⭐⭐ | 通道匹配 | 充值 | 支付模块的核心能力，钱包依赖 |
| ⭐⭐⭐ | KYC+资金来源过滤 | 提现 | 合规要求，规则复杂 |
| ⭐⭐ | 重复转账检测 | 充值 | 用户体验保护 |
| ⭐⭐ | 汇率偏差检测 | 定时任务 | 三方 API 异常时的降级 |
| ⭐ | CRUD 操作 | 币种配置 | 常规操作 |

## 十三、每步挑战速查表

> 一张表查清所有挑战点，供开发时逐一攻破

| 挑战 | 所在链路 | 所在步骤 | 严重程度 | 解决方案 |
|------|---------|---------|---------|---------|
| 并发余额操作互相覆盖 | 所有写入 | ledger 层 | 致命 | 悲观锁 SELECT FOR UPDATE |
| 重复 RPC 导致重复入账 | 充值/补单 | WalletCredit | 致命 | DB 唯一索引 (orderNo, walletType, flowType) |
| 冻结后提交失败，资金锁死 | 提现 | Freeze→Submit | 致命 | Submit 失败立即 Unfreeze |
| 兑换多行锁死锁 | 兑换 | Exchange 事务 | 严重 | 统一锁排序 (currency_id ASC, wallet_type ASC) |
| 出款结果回调丢失 | 提现 | 三方→支付 | 严重 | 支付侧重试 + 日对账 |
| 充值回调丢失 | 充值 | 三方→支付 | 严重 | 支付侧补查 + 日对账 + 补单 |
| 浮点精度导致金额偏差 | 兑换/汇率 | 金额计算 | 严重 | BIGINT 存储 + shopspring/decimal 计算 |
| 汇率 API 异常值 | 定时任务 | 汇率拉取 | 中等 | 3 源交叉验证 + 合理性校验 + 降级策略 |
| KYC 服务不可用 | 提现 | 资格校验 | 中等 | 降级策略：仅允许 USDT 提现 |
| 资金来源规则判定 | 提现 | 方式过滤 | 中等 | 记录首次充值类型，混合后标记不回退 |
| 大表分页性能 | 记录查询 | QueryRecordPage | 中等 | (user_id, currency_id) 索引 + TiDB 分布式能力 |
| 缓存与 DB 不一致 | 余额查询 | Redis 缓存 | 低 | Cache-Aside + 写后清缓存 + TTL 60s |
| 财务模块未就绪 | 充值/提现 | 所有支付 RPC | 阻塞 | Mock 实现，70% 工作量可独立推进 |
