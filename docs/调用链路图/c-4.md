## 一、系统拓扑：请求经过的所有节点

### 1.1 节点清单

一个请求从用户手机出发，可能经过以下节点：

```
 ┌──────────────────────────────────────────────────────────────────────────┐
 │                                                                          │
 │  【第一层：用户终端】                                                     │
 │   App / H5                                                               │
 │   → 发出 HTTPS 请求                                                      │
 │                                                                          │
 │  【第二层：网关】                                                         │
 │   gate-font（C端网关）  或  gate-back（B端网关）                          │
 │   → HTTP 路由 + 鉴权 + 参数解析 + 黑/白名单                              │
 │   → 转化为 Kitex RPC 调用                                                │
 │                                                                          │
 │  【第三层：业务服务】                                                     │
 │   ser-wallet（我负责）                                                    │
 │     ├─ handler层：接收RPC请求，参数校验，调用ser层                         │
 │     ├─ ser层：业务编排，事务管理，调用rep层和外部RPC                       │
 │     ├─ rep层：数据库操作，GORM查询，事务执行                              │
 │     └─ domain层：BalanceEngine（余额变更引擎）                            │
 │                                                                          │
 │   ser-finance（别人负责）                                                 │
 │     → 通道管理、三方支付对接、审核流程                                     │
 │                                                                          │
 │   ser-kyc（别人负责）                                                     │
 │     → KYC实名认证状态查询                                                │
 │                                                                          │
 │   ser-blog（公共服务）                                                    │
 │     → 审计日志记录（oneway，发射即忘）                                    │
 │                                                                          │
 │  【第四层：基础设施】                                                     │
 │   TiDB（数据库）                                                         │
 │   Redis（缓存）                                                          │
 │   ETCD（配置中心 + 服务发现）                                            │
 │   Kafka / NATS（消息队列，暂无直接使用，预留）                            │
 │                                                                          │
 │  【第五层：外部系统】                                                     │
 │   三方支付渠道（代收/代付）                                               │
 │   三方汇率API（3家）                                                     │
 │   区块链网络（USDT TRC20/ERC20/BEP20）                                   │
 │                                                                          │
 └──────────────────────────────────────────────────────────────────────────┘
```

### 1.2 节点间的连接方式

```
连接方式一览：

  App ──HTTPS──→ gate（网关）               同步，用户等待
  gate ──Kitex RPC──→ ser-wallet            同步，TTHeader协议，ETCD服务发现
  ser-wallet ──函数调用──→ 币种配置域        同步，零网络开销（同服务内部）
  ser-wallet ──Kitex RPC──→ ser-finance     同步，跨服务
  ser-wallet ──Kitex RPC──→ ser-kyc         同步，跨服务
  ser-wallet ──Kitex oneway──→ ser-blog     异步，发射即忘，不等返回
  ser-wallet ──GORM──→ TiDB                 同步，数据库读写
  ser-wallet ──go-redis──→ Redis            同步，缓存读写
  ser-finance ──HTTP──→ 三方支付             同步/异步（回调）
  ser-wallet（定时任务）──HTTP──→ 三方汇率API 同步

  关键认知：
  用户能感知到延迟的 = 同步链路上所有节点的响应时间之和
  用户感知不到的 = oneway调用（审计日志）、异步回调（充值到账）
```

## 二、请求的通用骨架：每个请求都经历的6层

不管是充值、提现还是查余额，**每一个请求都穿过相同的6层结构**：

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│  Layer 1：网关层（gate-font / gate-back）                        │
│  ────────────────────────────────────────                       │
│  做什么：路由匹配 → 鉴权（JWT Token / Session）→ 参数反序列化    │
│  怎么做：Hertz HTTP框架自动处理                                  │
│  输出：一个合法的、已鉴权的 RPC 请求对象                         │
│  耗时：~1-3ms                                                   │
│                                                                 │
│  Layer 2：Handler层（ser-wallet/internal/handler/）              │
│  ────────────────────────────────────────                       │
│  做什么：接收 RPC 请求 → 参数校验（必填/格式/范围）→ 调用 Ser    │
│  怎么做：自动生成的 handler 方法，手写参数校验                    │
│  输出：一个经过校验的业务请求                                    │
│  耗时：<1ms                                                     │
│                                                                 │
│  Layer 3：Ser层（ser-wallet/internal/ser/）                      │
│  ────────────────────────────────────────                       │
│  做什么：业务逻辑编排——判断+流程+循环都在这里                     │
│  怎么做：组合调用 rep 层、domain 层、外部 RPC                    │
│  关键：事务管理在这一层发起                                      │
│  耗时：看具体链路，1ms ~ 100ms                                  │
│                                                                 │
│  Layer 4：Domain层（ser-wallet/internal/domain/）                │
│  ────────────────────────────────────────                       │
│  做什么：BalanceEngine——余额变更的唯一入口                       │
│  怎么做：行锁+乐观锁+写流水，全在一个事务里                      │
│  关键：所有涉及"改余额"的操作必须经过这里                        │
│  耗时：10-30ms（含数据库事务）                                   │
│                                                                 │
│  Layer 5：Rep层（ser-wallet/internal/rep/）                      │
│  ────────────────────────────────────────                       │
│  做什么：数据库 CRUD，GORM 查询，事务执行                        │
│  怎么做：gorm_gen 自动生成的查询 + 手写复杂查询                  │
│  输出：数据库读写结果                                            │
│  耗时：1-10ms（取决于查询复杂度和数据量）                        │
│                                                                 │
│  Layer 6：基础设施层（TiDB + Redis）                             │
│  ────────────────────────────────────────                       │
│  做什么：持久化数据、提供缓存                                    │
│  怎么做：SQL执行、Redis命令                                      │
│  输出：数据                                                     │
│  耗时：TiDB ~1-5ms/query，Redis ~0.1-1ms/command               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

一个请求的生命周期：
  用户App → Layer1 → Layer2 → Layer3 → Layer4/5/6 → 原路返回 → 用户App

注意：Layer4（BalanceEngine）只有"写余额"的操作才经过。
      纯读操作（查余额、查记录）走 Layer3 → Layer5 → Layer6，跳过 Layer4。
```

### 3.2 充值链路分步拆解

```
═══════════════════════════════════════════════════════════════════════
阶段A：加载充值页面（C-3 GetDepositMethods）
═══════════════════════════════════════════════════════════════════════

  第1跳：App → gate-font
  ─────────────────────
  做什么：HTTP POST /api/wallet/deposit/methods
  发生了：鉴权（解析JWT Token提取userId）、路由匹配、参数解析
  产物：合法的RPC请求（含userId、currency_code）
  模式：【流程】纯顺序

  第2跳：gate-font → ser-wallet（handler）
  ────────────────────────────────────────
  做什么：Kitex RPC 调用
  发生了：服务发现（ETCD获取ser-wallet地址）、负载均衡、序列化
  产物：handler接收到请求对象
  耗时：~1ms
  模式：【流程】

  第3跳：handler → ser层
  ─────────────────────
  做什么：参数校验（currency_code非空、是已启用币种）
  发生了：校验失败 → 直接返回错误
  产物：合法的业务请求
  模式：【判断】校验分支

  第4跳：ser层 → Redis
  ──────────────────
  做什么：读取已启用币种列表缓存
  发生了：cache hit → 直接返回；cache miss → 查TiDB后写缓存
  产物：币种列表（验证请求的currency_code有效）
  模式：【判断】缓存命中/未命中分支

  第5跳：ser层 → ser-finance（RPC）     ★★★ 跨服务调用 ★★★
  ──────────────────────────────────
  做什么：GetAvailableDepositMethods（获取该币种下的充值方式列表）
  发生了：RPC请求到财务服务，财务查自己的支付配置表
  返回了：[{方式编码, 名称, 图标, 金额范围, 金额档位, 手续费率}]
  模式：【流程】

  ▲ 挑战点：这是阻塞级依赖
    → 如果ser-finance不可用或响应慢，充值页面无法渲染
    → 降级策略：超时3s后返回空列表 + 提示"暂无可用充值方式"

  第6跳：原路返回
  ──────────────
  ser层 → handler → gate-font → App
  App渲染充值页面：展示充值方式列表、金额档位、当前汇率

  链路总跳数：6跳
  链路总耗时：~50-200ms（主要瓶颈在第5跳跨服务调用）
  涉及服务：ser-wallet + ser-finance + Redis
  数据库操作：0-1次（缓存未命中时查币种）
  复杂度：★★☆☆☆（低，纯查询链路）


═══════════════════════════════════════════════════════════════════════
阶段B：创建充值订单（C-4 CreateDepositOrder）
═══════════════════════════════════════════════════════════════════════

  第1跳：App → gate-font → ser-wallet（同上，省略）

  第2跳：handler → ser层（参数校验）
  ──────────────────────────────────
  做什么：
    - currency_code 非空、已启用
    - amount > 0、在允许范围内
    - deposit_method 是有效的充值方式
    - bonus_activity_id 有效（如果有）
  模式：【判断】多重校验，任何一项不过就返回错误

  第3跳：ser层 → TiDB（重复转账检测）     ★★ 关键判断 ★★
  ──────────────────────────────────────
  做什么：查 deposit_order 表
    WHERE user_id=? AND currency_code=? AND amount=? AND status IN (待支付,进行中)
  判断：
    count = 0  → 正常，继续
    count = 1-2 → 返回警告标记（前端弹窗，用户可选择继续）
    count >= 3 → 拦截，返回错误（USDT不受此限制）
  模式：【判断】三级分支

  第4跳：ser层 → TiDB（创建订单）
  ─────────────────────────────
  做什么：
    生成订单号 C+16位
    INSERT deposit_order（状态=待分配通道）
  产物：数据库中多了一条充值订单记录
  模式：【流程】

  第5跳：ser层 → ser-finance（RPC）     ★★★ 关键跨服务调用 ★★★
  ──────────────────────────────────
  做什么：传订单号+用户信息+充值方式+金额 → 财务分配支付通道
  发生了：
    财务内部：根据轮询规则匹配代收通道 → 对接三方API → 获取支付链接
    财务侧：创建自己的通道订单记录
  返回了：
    银行/电子钱包 → 三方支付跳转URL
    USDT → 收款地址信息（或跳过此步，由C-5单独获取）
  模式：【流程】但内部包含财务的复杂逻辑

  ▲ 挑战点：
    → 这步是整个充值链路中最慢的（可能需要500ms-2s，因为要请求三方API）
    → 如果三方API超时，整个创建订单流程失败
    → 需要考虑：钱包已建订单但财务分配通道失败 → 订单状态标记为"分配失败"

  第6跳：ser层 → TiDB（更新订单状态）
  ─────────────────────────────────
  做什么：UPDATE deposit_order SET status=待支付, pay_url=?
  模式：【流程】

  第7跳：ser层 → ser-blog（审计日志，oneway）
  ─────────────────────────────────────────

  链路总耗时：~200ms-3s（第5跳是瓶颈）
  涉及服务：ser-wallet + ser-finance + ser-blog + TiDB
  数据库操作：3次（查重复+写订单+更新状态）
  复杂度：★★★★☆（高，涉及跨服务+状态管理+异常处理）

═══════════════════════════════════════════════════════════════════════

  触发方：不是用户，是ser-finance（收到三方支付回调后内部调用）
  时机：用户完成支付后，三方推送回调给财务，财务验签后RPC调钱包

  ─────────────────────────────────────────
  做什么：CreditDeposit(order_no, user_id, currency_code, amount, bonus_amount)
  注意：这不经过gate网关，是服务间直接RPC调用

  第2跳：handler → ser层（幂等检查）     ★★★ 核心防线 ★★★
  ──────────────────────────────────────
  做什么：
    查 wallet_transaction 表：
    WHERE order_no=? AND wallet_type=1 AND tx_type=1
  判断：
    已存在 → 直接返回成功（这笔已经入过账了，幂等保护）

    三方可能重复推送回调（网络重试、补偿机制）

  │                                                                      │

  │         → 写入流水，唯一键 uk_order_wallet_tx 兜底幂等                │

  │  【如果有奖金】                                                       │

  链路总耗时：~20-50ms
  涉及服务：ser-finance → ser-wallet → TiDB
  数据库操作：6-8次SQL在一个事务中
  复杂度：★★★★★（最高，涉及并发安全+幂等+事务+多表写入）

═══════════════════════════════════════════════════════════════════════
```

```
═══════════════════════════════════════════════════════════════════════
阶段B：执行兑换（C-7 ExecuteExchange）    ★★★★ 事务最复杂 ★★★★
═══════════════════════════════════════════════════════════════════════

  跳1：App → gate-font → ser-wallet

  跳2：ser层 → Redis（读快照）
       → 【判断】快照不存在或过期 → 返回错误，要求重新预览

       → 【判断】法币中心钱包可用余额 < 兑换金额 → 返回余额不足

  跳4：ser层 → domain层 → TiDB（事务操作）

  ┌─ BEGIN TRANSACTION ──────────────────────────────────────────────┐
  │                                                                  │
  │  步骤1：Deduct 法币中心钱包                                      │
  │    SELECT wallet FOR UPDATE (user_id, 法币, type=1)              │
  │    UPDATE balance = balance - 法币金额, version+1                │

  │                                                                  │
  │  步骤3：Add BSB奖励钱包（赠送部分）                               │
  │    SELECT wallet FOR UPDATE (user_id, BSB, type=2)               │
  │    → 如果不存在 → INSERT创建                                     │
  │    UPDATE balance = balance + BSB赠送金额, version+1              │
  │    INSERT wallet_transaction (BSB赠送流水)                        │
  │                                                                  │
  │  步骤4：创建兑换订单                                              │
  │    INSERT exchange_order                                         │
  │                                                                  │
  └─ COMMIT ─────────────────────────────────────────────────────────┘

  事务中操作了3个不同的钱包行，涉及约9-12条SQL

  跳5：返回成功

  复杂度：★★★★☆
  跨服务调用：0
  数据库操作：9-12次SQL在一个事务中
  挑战：
    → 一个事务内操作3个钱包行的锁顺序必须一致（避免死锁）
    → 推荐锁顺序：按 (user_id, currency_code, wallet_type) 排序
    → 首次兑换时钱包行可能不存在，需要INSERT OR SELECT
    → 快照过期的用户体验（提示"汇率已变化，请重新确认"）
```

## 十三、三种基本模式在各链路中的分布

你说得对——细节层面处理任何事情无非是**流程、判断、循环**三种模式。
以下分析每条链路中这三种模式的出现位置和密度。

### 13.1 流程（顺序执行）的出现位置

```
每条链路都有流程——请求从一个节点到下一个节点就是流程。
但"纯流程"（无判断无循环的直线段）集中在：

  ① 网关到handler：每个请求都是纯流程（鉴权→路由→RPC调用）
  ② BalanceEngine内部：锁→校验→更新→写流水→提交，严格顺序
  ③ 审计日志：所有写操作结尾都append一个oneway日志调用
  ④ 缓存清除：B端写操作后清除对应Redis key

  流程密度最高的链路：余额查询（几乎全是流程，无判断无循环）
  流程密度最低的链路：提现创建（判断密度极高）
```

### 13.2 判断（条件分支）的出现位置和类型

```
判断是整个系统复杂度的主要来源。按类型分：

  ═══ 安全类判断（必须有，缺少=系统漏洞）═══

  幂等判断：出现在所有RPC写操作
    位置：R-6/R-7/R-8/R-9/R-10/R-11/R-12 的入口
    模式：查wallet_transaction → 已处理则返回成功
    频率：每次RPC调用都执行

  余额判断：出现在所有扣款/冻结操作
    位置：C-7(兑换)、C-9(提现)、R-11(减款)
    模式：available >= amount ? 继续 : 返回6022001
    频率：每次扣款操作

  并发判断：出现在所有BalanceEngine操作
    位置：domain层
    模式：version校验 → 不匹配则重试或失败
    频率：每次余额变更

  ═══ 业务类判断（需求驱动，缺少=功能不完整）═══

  资金来源判断：只出现在提现创建
    位置：C-9 内部
    模式：fund_source → 决定允许的提现方式
    频率：每次提现

  KYC判断：只出现在提现页面加载
    位置：C-8 内部
    模式：KYC状态 → 过滤提现方式
    频率：每次打开提现页

  限额判断：只出现在提现创建
    位置：C-9 内部
    模式：单笔上下限 + 每日累计限额
    频率：每次提现

  重复转账判断：只出现在充值创建
    位置：C-4 内部
    模式：同条件订单数 → 0/1-2/≥3 三级
    频率：每次充值

  快照过期判断：只出现在兑换执行
    位置：C-7 入口
    模式：Redis TTL → 过期则拒绝
    频率：每次兑换

  ═══ 降级类判断（容错，缺少=体验差但不致命）═══

  缓存命中/未命中：币种列表、汇率
    位置：查询链路中
    模式：Redis有→用缓存，Redis无→查DB+写缓存

  外部RPC超时降级：
    位置：C-3(充值方式)、C-8(提现方式)
    模式：超时→返回空列表+提示

  汇率API降级：
    位置：T-1(汇率同步)
    模式：3→2→1→0家成功的四级降级
```

### 13.3 循环（遍历/重复）的出现位置

```
循环在钱包系统中出现较少（不是批量处理系统），但存在于：

  ① 汇率同步任务：遍历所有启用币种
     for each currency in enabledCurrencies {
       请求3家API → 计算 → 判断 → 更新
     }
     循环次数：4-6次（币种数）

  ② 余额查询：遍历用户的所有子钱包
     for each wallet in userWallets {
       累加余额
     }
     循环次数：2-5次（子钱包数）

  ③ 兑换事务：步骤1→2→3，实际上是对3个钱包做循环操作
     循环次数：3次（扣法币+加BSB中心+加BSB奖励）

  ④ 重试循环（代码层面）：乐观锁冲突时重试
     for retry := 0; retry < maxRetry; retry++ {
       执行操作
       if success { break }
     }
     循环次数：通常1次（冲突率极低时）

  循环密度最高的链路：汇率同步（对每个币种都跑一遍完整流程）
  循环密度最低的链路：大多数C端操作（一个用户一个币种一个操作，无循环）
```

## 十四、关键认知：你必须记住的8条链路规律

### 规律1：所有改余额的路，最后都经过同一个门

```
不管是充值/兑换/提现/加款/减款/补单，
不管请求从C端还是B端进来，
不管是同步还是异步，
最终都要经过 BalanceEngine 的5个方法之一。

这就是为什么 BalanceEngine 是最先开发、最需测试的组件。
它是唯一的口——管住这一个口，全系统的资金操作就安全了。
```

### 规律2：写余额的链路都是"先查后改同事务"

```
每一个写余额操作的底层模式都是：

  BEGIN TX
    SELECT ... FOR UPDATE   ← 锁住这一行
    校验（余额够不够/版本对不对）
    UPDATE wallet ...       ← 改余额
    INSERT transaction ...  ← 写流水
  COMMIT

无一例外。掌握了这个模式就掌握了所有写操作的骨架。
```

### 规律3：充值和提现是"两段式"链路

```
充值 = 创建订单（同步，用户等） + 入账（异步，回调触发）
提现 = 创建订单+冻结（同步，用户等） + 确认/解冻（异步，审核后触发）

两段之间可能隔分钟到天。这意味着：
  → 需要持久化状态（订单表的status字段）
  → 需要幂等（两段可能因重试重复触发）
  → 需要超时处理（第二段一直不来怎么办）
```

### 规律4：跨服务调用只有两种模式

```
模式A：钱包调别人（查询性质，同步等结果）
  → 调ser-finance查充值/提现方式
  → 调ser-kyc查KYC状态
  → 调ser-blog记日志（oneway，不等）

模式B：别人调钱包（操作性质，钱包被动执行）
  → ser-finance调钱包充值入账
  → ser-finance调钱包冻结/解冻/确认扣款
  → ser-finance调钱包人工加减款/补单

模式A失败 = 页面不完整但不影响资金安全
模式B失败 = 资金操作不完整，需要重试/补偿
```

### 规律5：兑换是唯一的"纯内部"资金操作

```
充值 = 外部钱进来（跨系统）
提现 = 内部钱出去（跨系统）
人工加减款 = B端发起（跨服务）
兑换 = 纯钱包内部流转（零依赖）

这意味着：
  → 兑换可以最先开发、最先测试、最先上线
  → 兑换是验证 BalanceEngine 正确性的最佳场景
  → 兑换出Bug不会影响外部系统（影响面可控）
```

### 规律6：判断密度和链路风险成正比

```
判断越多的链路，出Bug的概率越高：

  提现创建(C-9)：5个判断 → 风险最高
    资金来源+限额+余额+KYC+并发

  充值创建(C-4)：3个判断 → 风险中等
    重复拦截+奖金门槛+通道分配

  兑换执行(C-7)：2个判断 → 风险较低
    快照过期+余额充足

  余额查询(C-1)：0个判断 → 风险最低
    纯查询

  开发策略：判断密度高的链路需要更多的单元测试覆盖边界条件。
```

### 规律7：事务内SQL数量和链路可靠性成反比

```
事务中SQL越多，耗时越长，锁持有时间越长，失败概率越高：

  兑换执行：9-12条SQL/事务 → 最长事务，最高死锁风险
  充值入账：6-8条SQL/事务  → 较长事务
  提现冻结：4条SQL/事务    → 中等
  确认扣款：4条SQL/事务    → 中等
  人工加款：3条SQL/事务    → 最短

  应对策略：
  → 事务内不做外部调用（不RPC/不Redis/不HTTP）
  → 锁顺序统一（按 user_id+currency_code+wallet_type 排序）
  → 事务超时设置合理（防止长事务占资源）
```

### 规律8：每条链路都有一个"确定性骨架"

```
不管需求怎么变，以下骨架不会变：

  充值骨架：创建订单 → 等支付 → 回调入账
  提现骨架：校验+冻结 → 等审核 → 确认/退回
  兑换骨架：预览汇率 → 确认 → 三步事务（扣A+加B+加C）
  查询骨架：读DB → 格式化 → 返回

  变的是细节（字段多几个少几个、规则松一点严一点），
  不变的是骨架。先写骨架代码，再填充细节，是最安全的开发策略。
```
