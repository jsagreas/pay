### 1.2 调用方向总表

```
双向强依赖（充值/提现核心链路互调）：
  ser-wallet ←→ ser-finance

单向依赖（钱包调别人）：
  ser-wallet → ser-kyc        提现前KYC校验
  ser-wallet → ser-blog       B端操作审计（oneway异步）
  ser-wallet → ser-user       B端列表展示用户信息
  ser-wallet → ser-buser      B端展示操作人名称
  ser-wallet → ser-i18n       多语言翻译（大部分由网关自动处理）

单向依赖（别人调钱包）：
  ser-user    → ser-wallet    用户注册时 InitWallet
  ser-finance → ser-wallet    充值入账 CreditWallet
  ser-finance → ser-wallet    提现终态 UnfreezeBalance / ConfirmDebit
  ser-finance → ser-wallet    人工加减款 CreditWallet / DebitWallet
  ser-live    → ser-wallet    打赏扣款 DebitWallet（未来）
  ser-activity→ ser-wallet    奖金发放 CreditWallet（未来）
```

### 1.3 数据存储分布

```
db_wallet（ser-wallet 负责）：
  wallet / wallet_flow / freeze_detail     ← 钱包核心
  recharge_order / withdraw_order          ← 业务订单
  exchange_record / withdraw_account       ← 业务记录
  currency_config / rate_log / base_currency_config  ← 币种配置

db_finance（ser-finance 负责）：
  deposit_channel / payout_channel         ← 通道配置
  recharge_method / withdraw_method        ← 方式配置
  manual_patch/credit/debit_order          ← 人工操作
  channel_polling_rule / channel_daily_stat ← 通道管理

Redis（ser-wallet 使用）：
  wallet:currency:list      ← 启用币种列表（TTL 1h）
  wallet:currency:{code}    ← 单币种配置（TTL 1h）
  wallet:base_currency      ← 基准币种（TTL 24h）
  wallet:user_currency:{uid}← 用户当前币种（TTL 7d）
  wallet:rate:{from}:{to}   ← 汇率缓存（TTL 5min）
```

### 2.2 逐步拆解（9步）

| 步骤 | 执行者 | 做什么 | 类型 | 复杂度 |
|:----:|--------|--------|:----:|:------:|
| ① | ser-wallet | 收到getRechargeMethodList请求 → RPC调ser-finance获取当前币种可用充值方式 → 返回前端展示 | 流程 | ★ |
| ② | ser-wallet | 收到createRechargeOrder → 创建本地recharge_order(status=待支付) | 流程 | ★★ |
| ③ | ser-wallet→ser-finance | RPC调MatchChannelAndCreateOrder → ser-finance内部匹配通道 → 调三方代收API → 返回payUrl/地址 | 流程 | ★★★ |
| ④ | 用户 | 在三方页面完成支付 | 外部 | — |
| ⑤ | ser-finance | 接收三方回调 → 验签 → 解析支付结果 | 流程+判断 | ★★★ |
| ⑥ | ser-finance→ser-wallet | RPC调CreditWallet通知充值成功 | 流程 | ★ |
| ⑦ | **ser-wallet** | **CreditWallet核心事务**（见下方详解） | **流程+判断** | **★★★★★** |
| ⑧ | ser-wallet | 如有奖金 → 第二次CreditWallet → 奖励钱包 | 判断+流程 | ★★ |
| ⑨ | ser-wallet | 更新recharge_order status=成功 | 流程 | ★ |

### 2.3 核心步骤深度展开 — 步骤⑦ CreditWallet事务

这是整个充值链路最关键的一步。拆开来看，内部是 **流程+判断** 的组合：

```
CreditWallet 内部流程（一个DB事务内完成）：
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  1. 开启数据库事务 BEGIN                                       │
│     │                                                        │
│  2. 悲观锁定钱包记录                                            │
│     SELECT * FROM wallet                                     │
│     WHERE user_id=? AND currency_code=? AND wallet_type=?    │
│     FOR UPDATE                                               │
│     │                                                        │
│  3. 构造幂等键 key = "{bizType}:{refOrderNo}:{walletType}"     │
│     │                                                        │
│  4. 尝试插入流水记录                                            │
│     INSERT INTO wallet_flow (idempotent_key, before_balance,  │
│     after_balance, amount, ...) VALUES (...)                  │
│     │                                                        │
│  5. ─── 判断 ───────────────────────────────────              │
│     │                                                        │
│     ├── 唯一约束冲突（重复请求）                                  │
│     │   └── 返回"已处理"，不更新余额（幂等保护）                   │
│     │                                                        │
│     └── 插入成功（首次请求）                                     │
│         │                                                    │
│         6. 更新余额                                           │
│            UPDATE wallet SET balance = balance + amount       │
│            WHERE id = ?                                      │
│            │                                                 │
│         7. 提交事务 COMMIT                                    │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 2.4 关键节点与挑战

| 节点 | 挑战 | 应对方案 |
|------|------|---------|
| **步骤③ RPC调ser-finance** | ser-finance模块不存在，是最大阻塞风险 | Phase 3用Mock桩接口替代 |
| **步骤⑤ 三方回调** | 回调归谁接收？（灰色地带，待支付组确认） | 倾向方案A：回调到ser-finance，验签后调我们CreditWallet |
| **步骤⑦ CreditWallet** | 重复回调导致重复上账 | 幂等三重保障：业务键+DB唯一约束+事务内先INSERT后UPDATE |
| **步骤⑦ 并发** | 同一用户同时多笔充值回调 | 悲观锁 SELECT...FOR UPDATE 串行化 |
| **步骤③ 超时** | MatchChannelAndCreateOrder调三方可能慢 | 设10秒超时，超时后标记订单失败 |
| **步骤⑧ 奖金** | 奖金入账必须和主金额入账原子关联 | 两次CreditWallet用不同幂等键，但业务上绑定同一充值订单 |

### 2.5 充值链路的状态流转

```
recharge_order.status 状态机：

  [创建] ──→ 1.待支付 ──→ 2.成功（终态，余额已增加）
                     ├──→ 3.失败（终态，三方返回失败）
                     └──→ 4.取消/过期（终态，超时未支付）

wallet.balance 变化：
  步骤⑦前：balance = X
  步骤⑦后：balance = X + amount（中心钱包）
  步骤⑧后：奖励钱包 balance += bonus_amount（如有）
```

| ② | ser-wallet | 二次KYC验证 + 查限额配置(RPC) + 校验金额/次数/限额 | 流程+判断 | ★★ |

### 3.4 提现终态处理 — UnfreezeBalance vs ConfirmDebit

```
UnfreezeBalance（解冻 — 驳回/失败场景）：
┌──────────────────────────────────────────────┐
│ 1. 按 refOrderNo 查 freeze_detail            │
│ 2. 遍历冻结明细（1-2条）                       │
│    对每条：                                    │
│    ├── FOR UPDATE 锁对应钱包                   │
│    ├── frozen_amount -= freeze_amount         │
│    └── 更新 freeze_detail.status = 3(已解冻)   │
│ 3. 写 wallet_flow（解冻流水）                   │
│ 4. 更新 withdraw_order.status = 对应终态       │
└──────────────────────────────────────────────┘

ConfirmDebit（确认扣款 — 出款成功场景）：
┌──────────────────────────────────────────────┐
│ 1. 按 refOrderNo 查 freeze_detail            │
│ 2. 遍历冻结明细（1-2条）                       │
│    对每条：                                    │
│    ├── FOR UPDATE 锁对应钱包                   │
│    ├── balance -= freeze_amount              │
│    ├── frozen_amount -= freeze_amount         │
│    └── 更新 freeze_detail.status = 2(已确认扣) │
│ 3. 写 wallet_flow（扣款流水）                   │
│ 4. 更新 withdraw_order.status = 7(出款成功)    │
└──────────────────────────────────────────────┘
```

### 3.5 提现链路的状态流转

```
withdraw_order.status 状态机（7种状态，4种终态）：

  [创建] → 1.待风控 ──→ 2.风控驳回 ★终态（解冻）
                   └──→ 3.待财务 ──→ 4.财务驳回 ★终态（解冻）
                                └──→ 5.待出款 ──→ 6.出款失败 ★终态（解冻）
                                             └──→ 7.出款成功 ★终态（确认扣款）

wallet.balance / frozen_amount 变化：
  步骤③后：balance不变, frozen_amount += X（资金被冻结，可用余额减少）
  驳回后：  balance不变, frozen_amount -= X（冻结解除，可用余额恢复）
  成功后：  balance -= X, frozen_amount -= X（余额真正减少）
```

### 3.6 关键节点与挑战

| 节点 | 挑战 | 应对方案 |
|------|------|---------|
| **步骤③ 冻结跨钱包** | 中心不够需冻奖励，两个钱包的冻结金额分配必须精确记录 | freeze_detail 表逐钱包记录，解冻时按明细还原 |
| **步骤④ 通知方式** | RPC还是MQ？（灰色地带） | 倾向MQ（解耦），待支付组确认 |
| **步骤⑤⑥ 状态驱动** | 谁来更新withdraw_order的状态？ | ser-finance审核后通知ser-wallet，ser-wallet自己更新（不跨库写） |
| **步骤⑦ 终态互斥** | UnfreezeBalance和ConfirmDebit对同一单只能执行一个 | freeze_detail.status做状态校验，已解冻的不可再确认 |
| **全链路 时间跨度** | 从冻结到终态可能跨数小时甚至数天 | 冻结三阶段设计天然支持长周期 |

### 4.2 逐步拆解

| 步骤 | 做什么 | 类型 | 关键点 |
|:----:|--------|:----:|--------|
| ① | 读Redis汇率缓存 + 查用户法币余额 + 计算可兑金额 | 流程 | 汇率从缓存读（5min TTL） |
| ② | 锁定汇率 | 流程 | 防止事务执行期间汇率变动 |
| ③ | 悲观锁法币钱包 → 校验余额 → 扣减 | 流程+判断 | 余额不足直接拒绝 |
| ④ | 整数运算换算BSB金额 | 流程 | amount × rate / 10^8，注意取整方向 |
| ⑤ | 悲观锁BSB中心钱包 → 加款 | 流程 | — |
| ⑥ | 如有赠送 → 悲观锁BSB奖励钱包 → 加赠送 | 判断+流程 | 赠送比例从配置读取 |
| ⑦ | 写exchange_record + 3条wallet_flow | 流程 | 事务内一次性提交 |

### 4.3 为什么兑换不需要分布式事务

```
关键理解：兑换涉及的所有表（wallet × 3个记录、wallet_flow × 3条、exchange_record × 1条）
           全部在 db_wallet 一个数据库内。

           所以用本地事务（BEGIN...COMMIT）就能保证原子性。
           不需要 Seata、DTM 等分布式事务框架。

           这是兑换链路相比充值/提现的最大简化点。
```

### 4.4 关键挑战

| 挑战 | 说明 | 应对 |
|------|------|------|
| 事务内锁多行 | 一个事务要锁2-3行wallet记录 | 按固定顺序加锁（先法币后BSB，先中心后奖励），避免死锁 |
| 汇率时效 | 用户看到的汇率和实际执行的可能有差 | 锁定时验证汇率未过期（5min窗口） |
| 取整方向 | 整数除法的精度损失 | 统一向下取整（用户不会多得，平台兜底） |

### 5.2 要点


> 复杂度：★★☆☆☆（标准CRUD，但缓存一致性要注意）
> 涉及模块：gate-back → ser-wallet → Redis → ser-blog
> 核心挑战：缓存更新策略

### 7.1 典型操作时序（以编辑币种为例）

```
 运营        gate-back      ser-wallet            Redis          db_wallet      ser-blog
  │              │              │                    │              │              │
  │  编辑币种配置 │              │                    │              │              │
  │─────────────▶│ updateCurrency                   │              │              │
  │              │─────────────▶│                    │              │              │
  │              │              │  1.参数校验         │              │              │
  │              │              │  (汇率浮动比例范围等)│              │              │
  │              │              │                    │              │              │
  │              │              │  2.DB更新           │              │              │
  │              │              │────────────────────────────────▶│              │
  │              │              │  UPDATE currency_config          │              │
  │              │              │◀────────────────────────────────│              │
  │              │              │                    │              │              │
  │              │              │  3.重算汇率          │              │              │
  │              │              │  deposit_rate = real_rate × (1 + float_pct)    │
  │              │              │  withdraw_rate = real_rate × (1 - float_pct)   │
  │              │              │                    │              │              │
  │              │              │  4.删除缓存         │              │              │
  │              │              │───────────────────▶│              │              │
  │              │              │  DEL wallet:currency:list         │              │
  │              │              │  DEL wallet:currency:{code}       │              │
  │              │              │◀───────────────────│              │              │
  │              │              │                    │              │              │
  │              │              │  5.写操作日志（异步） │              │              │
  │              │              │──────────────────────────────────────────────▶│
  │              │              │  oneway: AddActLog  │              │              │
  │              │              │                    │              │              │
  │              │  编辑成功     │                    │              │              │
  │◀─────────────│◀─────────────│                    │              │              │
```

### 7.2 缓存一致性策略

```
写操作（create/update/status/setBaseCurrency/refreshRate）：
  1. 先写DB
  2. 再删Redis缓存（wallet:currency:list + wallet:currency:{code}）
  3. 下次读操作时自动回源DB并重建缓存

为什么是"删缓存"而不是"写缓存"？
  - 删缓存更安全：即使删除后瞬间有读请求，最坏情况是回源DB（正确数据）
  - 写缓存有风险：并发场景下可能把旧数据覆盖新数据
  - 这是经典的 Cache-Aside Pattern
```

## 八、链路七：汇率定时拉取 — 无人值守后台任务

> 复杂度：★★★☆☆（并发调用+容错+偏差判断）
> 涉及模块：ser-cron/内置定时器 → 三方汇率API → ser-wallet → Redis
> 核心挑战：三方API容错、异常值过滤、偏差阈值触发

### 8.1 完整时序图

```
 定时器(3min)     ser-wallet        汇率API-A      汇率API-B      汇率API-C
    │                │                 │              │              │
    │  触发           │                 │              │              │
    │───────────────▶│                 │              │              │
    │                │                 │              │              │
    │                │  ①并发调用3个API（goroutine × 3，各自5秒超时）    │
    │                │────────────────▶│              │              │
    │                │───────────────────────────────▶│              │
    │                │──────────────────────────────────────────────▶│
    │                │                 │              │              │
    │                │  各自返回汇率     │              │              │
    │                │◀────────────────│              │              │
    │                │◀───────────────────────────────│              │
    │                │◀──────────────────────────────────────────────│
    │                │                 │              │              │
    │                │  ②收集结果                                     │
    │                │  ┌──────────────────────────────────┐         │
    │                │  │ 成功3个？2个？1个？0个？              │         │
    │                │  │                                  │         │
    │                │  │ 过滤异常值：                       │         │
    │                │  │   计算均值 avg                     │         │
    │                │  │   偏离均值 > 30% 的排除             │         │
    │                │  │                                  │         │
    │                │  │ 有效结果 >= 2个？                  │         │
    │                │  │ ├── YES → 取平均值 = new_real_rate │         │
    │                │  │ ├── = 1个 → 保守：只更新real_rate   │         │
    │                │  │ │         不更新platform_rate      │         │
    │                │  │ └── = 0个 → 不做任何更新，告警      │         │
    │                │  └──────────────────────────────────┘         │
    │                │                                               │
    │                │  ③偏差判断                                     │
    │                │  ┌──────────────────────────────────┐         │
    │                │  │ deviation = |new_real - platform| │         │
    │                │  │            / platform             │         │
    │                │  │                                  │         │
    │                │  │ deviation >= deviation_pct？       │         │
    │                │  │ ├── YES → 更新全套汇率：           │         │
    │                │  │ │   real_rate = new_real           │         │
    │                │  │ │   platform_rate = new_real       │         │
    │                │  │ │   deposit_rate = new × (1+浮动%) │         │
    │                │  │ │   withdraw_rate = new × (1-浮动%)│         │
    │                │  │ │   写 rate_log（trigger=偏差触发）  │         │
    │                │  │ │                                 │         │
    │                │  │ └── NO  → 只更新 real_rate         │         │
    │                │  │           platform_rate 不变       │         │
    │                │  └──────────────────────────────────┘         │
    │                │                                               │
    │                │  ④更新Redis缓存                                │
    │                │  SET wallet:rate:{from}:{to} TTL=5min         │
    │                │                                               │
```

### 8.2 容错矩阵

| 场景 | API-A | API-B | API-C | 处理策略 |
|------|:-----:|:-----:|:-----:|---------|
| 全部成功 | ✅ | ✅ | ✅ | 过滤异常值，取平均 |
| 2个成功 | ✅ | ✅ | ❌ | 取2个平均，正常更新 |
| 1个成功 | ✅ | ❌ | ❌ | 保守策略：只更real_rate |
| 全部失败 | ❌ | ❌ | ❌ | 不更新任何汇率，日志告警 |
| 1个异常值 | ✅ | ✅ | ⚠️偏离>30% | 排除异常值，取2个平均 |

## 九、辅助链路：余额查询 / 记录查询 / KYC预查

这些链路相对简单，是纯读操作，不涉及资金变动。

### 9.1 余额查询（getWalletInfo）

```
用户 → gate-font → ser-wallet
                       │
                       ├── 1. 读Redis获取用户当前币种偏好
                       │     wallet:user_currency:{uid}
                       │
                       ├── 2. 查wallet表（当前币种的中心+奖励钱包）
                       │     SELECT * FROM wallet WHERE user_id=? AND currency_code=?
                       │
                       ├── 3. 读Redis获取启用币种列表
                       │     wallet:currency:list
                       │
                       └── 4. 组装返回
                             中心余额、奖励余额、资金余额(中心+奖励)、冻结金额、
                             可用余额、币种列表、当前币种配置

无外部RPC依赖，纯本地DB+Redis读取。响应应在 50ms 以内。
```

### 9.2 记录查询（getRecordList / getRecordDetail）

```
用户 → gate-font → ser-wallet
                       │
                       ├── 按类型路由查询不同的表：
                       │   ├── type=充值  → SELECT FROM recharge_order
                       │   ├── type=提现  → SELECT FROM withdraw_order
                       │   ├── type=兑换  → SELECT FROM exchange_record
                       │   └── type=奖励  → SELECT FROM wallet_flow WHERE biz_type=4
                       │
                       └── 分页 + 按时间倒序 + 格式化金额展示

无外部RPC依赖。币种格式化信息（千分位、小数位等）从Redis币种缓存读取。
```

### 9.3 切换币种（switchCurrency）

```
用户 → gate-font → ser-wallet
                       │
                       ├── 1. 校验目标币种是否启用
                       │     读Redis wallet:currency:{code}
                       │
                       ├── 2. 更新Redis用户币种偏好
                       │     SET wallet:user_currency:{uid} = newCode TTL=7d
                       │
                       ├── 3.（可选）如有user_currency_preference表，同步写DB
                       │
                       └── 4. 返回成功

纯缓存操作，响应极快。后续所有接口基于新币种上下文工作。
```

## 十一、编码优先级指导

### 11.1 从链路复杂度推导编码顺序

```
Phase 1 — 先搭最简单但被依赖最多的：
════════════════════════════════════════════
  1. InitWallet（★★，被ser-user依赖）
  2. CreditWallet/DebitWallet（★★★★，被所有资金操作依赖）
  3. GetBalance（★，被所有展示依赖）
  4. 币种配置CRUD（★★，被所有币种相关功能依赖）
  5. GetCurrencyList/GetExchangeRate（★，被所有需要币种的场景依赖）

Phase 2 — 独立闭环链路：
════════════════════════════════════════════
  6. 汇率定时拉取（★★★，独立无依赖）
  7. 兑换（★★★，纯本地事务闭环）
  8. 余额查询/记录查询（★，纯读操作）
  9. 切换币种（★，纯缓存操作）
  10.提现账户CRUD（★★，标准CRUD）

Phase 3 — 跨模块复杂链路（依赖ser-finance）：
════════════════════════════════════════════
  11. FreezeBalance/UnfreezeBalance/ConfirmDebit（★★★★★，冻结三阶段）
  12. 充值全链路（★★★★★，含回调幂等）
  13. 提现全链路（★★★★★，含冻结+审核+终态）
```

### 11.2 每个Phase完成后能跑通的链路

```
Phase 1 完成后：
  ✅ 用户注册 → 钱包初始化 → 余额为0但钱包存在
  ✅ B端币种配置 → 启用/禁用/编辑
  ✅ 人工加款 → CreditWallet → 余额增加
  ✅ 人工减款 → DebitWallet → 余额减少
  ✅ 其他模块可调 GetBalance / GetCurrencyList

Phase 2 完成后：
  ✅ Phase 1 全部 +
  ✅ C端查看余额 → getWalletInfo
  ✅ C端切换币种 → switchCurrency
  ✅ C端法币→BSB兑换 → 完整闭环
  ✅ C端查看记录 → 充值/提现/兑换/奖励记录
  ✅ 汇率自动更新 → 定时拉取+偏差触发

Phase 3 完成后：
  ✅ Phase 1 + Phase 2 全部 +
  ✅ C端充值全链路 → 选方式 → 创建订单 → 跳转支付 → 回调上账
  ✅ C端提现全链路 → KYC校验 → 冻结 → 审核 → 出款 → 终态
```

### 11.3 编码时每一步的检查清单

对照上面的链路图，写每个接口时问自己：

```
□ 这个接口在哪条链路的第几步？
□ 它的上游是谁（谁调它）？入参从哪来？
□ 它的下游是谁（它调谁）？出参给谁用？
□ 内部是流程、判断、还是循环？有几个分支？
□ 有没有需要幂等的场景？幂等键是什么？
□ 有没有并发风险？需不需要加锁？
□ 出错了怎么办？上游怎么感知？用户看到什么？
□ 这个步骤在热力图中是几颗星？如果 ≥ ★★★，需要额外写单测
```
