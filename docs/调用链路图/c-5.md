# 一、请求入口分类与通用流转模型

## 1.1 三种请求入口

ser-wallet 的 46 个方法，请求从**三个入口**进入：

```
入口一：C端用户请求（21个）
  用户App/H5 → gate-font(Hertz HTTP) → ser-wallet(Kitex RPC)
  特征：userId 从 JWT → metainfo → tracer.GetUserId(ctx) 隐式获取
  认证：gateway 层 my_router.Api 做 JWT 校验

入口二：B端后台请求（12个）
  运营后台 → gate-back(Hertz HTTP) → ser-wallet(Kitex RPC)
  特征：operatorId 从 B端JWT 获取，userId 显式传参
  认证：gateway 层 my_router.Api 做 B端JWT 校验 + RBAC 权限

入口三：模块间RPC直调（12个）
  ser-finance → ser-wallet(Kitex RPC)  // 不经过任何网关
  特征：userId 必须显式传参（无 JWT），调用方身份靠 ETCD 服务注册保证
  认证：无（内网服务间信任）
```

## 1.2 通用请求流转模型

**不管从哪个入口进来，所有请求在 ser-wallet 内部的流转模式完全一致：**

```
请求进入
  │
  ▼
┌──────────────────────────────────────────────────┐
│  Handler 层（handler.go）                          │
│  ● 零逻辑，一行委托                                │
│  ● 46个方法 = 46行代码                             │
│  ● 唯一职责：把 req 传给 Service，把 resp 返回      │
└────────────────────┬─────────────────────────────┘
                     │
                     ▼
┌──────────────────────────────────────────────────┐
│  Service 层（internal/ser/*.go，8个文件）            │
│  ● 业务编排中心 — 这是所有逻辑的"大脑"              │
│  ● 职责：参数校验 → 调外部RPC → 业务计算 → 发起事务  │
│  ● 可以调其他 Service（如 DepositService 调         │
│    BalanceService）                                │
│  ● 可以调 domain 层的封装方法                       │
│  ★ 关键规则：RPC 在事务之外做                       │
└────────────────────┬─────────────────────────────┘
                     │
         ┌───────────┼───────────┐
         │           │           │
         ▼           ▼           ▼
  ┌────────────┐ ┌────────┐ ┌──────────────┐
  │ domain 层  │ │ Rep 层 │ │ 外部 RPC    │
  │ (封装层)   │ │ (数据) │ │ (远程调用)  │
  │            │ │        │ │             │
  │ balance_op │ │ 事务内 │ │ Finance     │
  │ cache      │ │ GORM   │ │ KYC         │
  │ mapper     │ │ 操作   │ │ Blog        │
  │ amount     │ │        │ │ User        │
  └────────────┘ └───┬────┘ └─────────────┘
                     │
                     ▼
              ┌────────────┐
              │  TiDB 数据库 │
              │  11张表      │
              └────────────┘
```

## 1.3 两种本质不同的操作

从数据变更的角度，46 个方法可以分为两大类：

```
类型一：纯读操作（17个）
  ● 不涉及事务
  ● 不涉及幂等
  ● 可以走缓存（币种配置走 Redis，余额不走）
  ● 链路短、复杂度低

  包括：币种/汇率/日志查询(5个) + 余额查询(2个) + 记录查询(5个) +
        充值方式/状态/地址/奖金查询(4个) + 提现方式/账户/限额/KYC(4个) +
        RPC余额查询(1个) + 枚举(1个)
        实际约 22 个读接口

类型二：写操作（24个，含事务）
  ● 必须在数据库事务内完成
  ● 写操作必须幂等（RPC暴露的写接口）或有重复校验
  ● 涉及余额变动的必须带乐观锁
  ● 链路长、复杂度高

  包括：币种CRUD(4个) + 汇率设置(1个) + 充值创建(1个) + 兑换创建(1个) +
        提现创建(1个) + 提现账户保存(1个) + 充值入账(1个) +
        提现冻结/确认/解冻(3个) + 人工加/减/补(3个) + 钱包冻结(1个)
        重复检查(1个) 等
```

# 二、四层架构的请求穿透路径

## 2.1 完整的一次请求穿透（以"兑换"为例）

```
时间线 ──────────────────────────────────────────────────────────────→

[用户App]                    [gate-font]              [ser-wallet]                           [TiDB]
    │                            │                        │                                    │
    │  POST /wallet/exchange/    │                        │                                    │
    │  create {currency, amount} │                        │                                    │
    │  + JWT Token (Header)      │                        │                                    │
    │ ──────────────────────────→│                        │                                    │
    │                            │ ① JWT校验              │                                    │
    │                            │ ② 提取userId→metainfo  │                                    │
    │                            │ ③ 透传到RPC            │                                    │
    │                            │ ──RPC调用──────────────→│                                    │
    │                            │                        │ handler.go:                        │
    │                            │                        │   return ser.CreateExchangeOrder(req)│
    │                            │                        │                                    │
    │                            │                        │ exchange_service.go:                │
    │                            │                        │ ④ userId := tracer.GetUserId(ctx)  │
    │                            │                        │ ⑤ 参数校验(currency/amount)        │
    │                            │                        │ ⑥ 获取最新汇率(Redis/DB)           │
    │                            │                        │ ⑦ 金额计算(整数运算)               │
    │                            │                        │ ⑧ 余额预检                        │
    │                            │                        │                                    │
    │                            │                        │ ===== 事务开始 =====                │
    │                            │                        │ ⑨ VND available -= 230000 ─────────→│ UPDATE WHERE available>=230000
    │                            │                        │    (乐观锁，rows affected=1?)      │←── rows=1 ✓
    │                            │                        │ ⑩ BSB available += 1000 ───────────→│ UPDATE
    │                            │                        │ ⑪ BSB奖励 available += 50 ─────────→│ UPDATE
    │                            │                        │ ⑫ 写兑换订单 ────────────────────→│ INSERT exchange_order
    │                            │                        │ ⑬ 写流水x3 ──────────────────────→│ INSERT balance_flow x3
    │                            │                        │ ===== 事务提交 =====                │
    │                            │                        │                                    │
    │                            │                        │ ⑭ 组装 resp 返回                   │
    │                            │ ←──────────────────────│                                    │
    │ ←──────────────────────────│ JSON 响应               │                                    │
    │  {code:0, data:{...}}      │                        │                                    │
```

**关键观察**：
- 网关层只做认证+转发，零业务逻辑
- Handler 层只做委托，零业务逻辑
- 所有业务逻辑集中在 Service 层
- 事务内只有 DB 操作（步骤⑨-⑬），没有 RPC
- 乐观锁在步骤⑨，一条 SQL 完成"检查+扣减"

## 2.2 跨模块 RPC 的穿透路径（以"充值入账"为例）

```
[支付渠道]        [ser-finance]                [ser-wallet]                    [TiDB]
    │                 │                            │                             │
    │ 渠道回调        │                            │                             │
    │ (支付成功)      │                            │                             │
    │ ───────────────→│                            │                             │
    │                 │ ① 验签/防重放              │                             │
    │                 │ ② 更新财务订单状态          │                             │
    │                 │                            │                             │
    │                 │ ── Kitex RPC ─────────────→│                             │
    │                 │    DepositCredit(req)       │                             │
    │                 │                            │ handler.go → rpc_service.go  │
    │                 │                            │ ③ 幂等校验(refOrderNo) ──────→│ SELECT deposit_order
    │                 │                            │    已入账? → 直接返回true     │←── status=success → return
    │                 │                            │                             │←── status=pending → 继续
    │                 │                            │                             │
    │                 │                            │ ===== 事务开始 =====         │
    │                 │                            │ ④ 订单状态→success ─────────→│ UPDATE deposit_order
    │                 │                            │ ⑤ available += amount ──────→│ UPDATE user_balance
    │                 │                            │ ⑥ 写流水 ──────────────────→│ INSERT balance_flow
    │                 │                            │    (唯一键: refOrderNo+type) │   (唯一键冲突→幂等兜底)
    │                 │                            │ ===== 事务提交 =====         │
    │                 │                            │                             │
    │                 │ ←────────────── true ──────│                             │
    │                 │ ③ 更新财务状态为已通知       │                             │
```

**关键观察**：
- 不经过任何网关，Kitex 服务间直连（ETCD 发现）
- userId 由调用方显式传入（无 JWT）
- 两层幂等：应用层(步骤③)查状态 + DB层(步骤⑥)唯一键
- 事务内纯 DB 操作，无 RPC

## 3.2 读操作链路（5个）— 最简链路

```
QueryCurrencyPage / QueryCurrencyDetail / QueryExchangeRatePage / QueryRateLogPage / QueryOperationLogPage

用户请求 → gate-back → handler → Service → Rep → DB → 返回

具体步骤：
  ① 参数解析（分页参数、筛选条件）
  ② 构造查询条件（GORM Where 链式）
  ③ 执行分页查询（SELECT + COUNT）
  ④ DTO 转换（model → resp）
  ⑤ 返回

复杂度：流程(5步)，无判断分支，无循环
挑战：无
耗时关键：DB 查询（单表查询，毫秒级）
```

## 3.3 写操作链路（4个）— 标准CRUD + 审计日志

```
CreateCurrency / UpdateCurrency / ToggleCurrencyStatus / SaveExchangeRate

用户请求 → gate-back → handler → Service → [事务{DB写}] → AddActLog(RPC) → 返回
```

以 **SaveExchangeRate** 为例（最完整的写操作）：

```
步骤 ① 参数校验
        │
        │  判断：currency_code 是否存在于 currency_config 表？
        │  ├─ 否 → 返回错误"币种不存在"
        │  └─ 是 → 继续
        │
步骤 ② 查旧汇率（用于写日志）
        │  SELECT rate FROM exchange_rate WHERE currency_code = ?
        │
步骤 ③ ===== 事务开始 =====
        │  UPSERT exchange_rate（存在则更新，不存在则插入）
        │  INSERT rate_log（旧值/新值/操作人/时间）
        ===== 事务提交 =====
        │
步骤 ④ 删除 Redis 缓存
        │  DEL wallet:rate:{currency}
        │  （下次查询时 Cache-Aside 自动重建）
        │
步骤 ⑤ 调 ser-blog RPC（异步，oneway void）
        │  rpc.BLogClient().AddActLog(ctx, &ActAddReq{
        │      Backend: "wallet",
        │      Model:   "rate",
        │      Page:    "汇率配置",
        │      Action:  "设置汇率 VND/BSB = 23000",
        │  })
        │  ★ oneway = fire-and-forget，不等返回，不影响业务
        │
步骤 ⑥ 返回成功

复杂度：流程(6步) + 判断(1个币种存在性)
挑战：缓存一致性 — 先写DB再删缓存，极端并发下有短暂不一致窗口（可接受）
```

## 3.4 RPC暴露链路（4个）— 被其他模块调用

```
GetAvailableCurrencyList / GetCurrencyInfo / GetExchangeRate / GetCurrencyConfig

调用方(ser-finance等) → ser-wallet handler → Service → Redis? → DB → 返回

具体路径（带缓存）：
  ① 收到 RPC 请求
  ② 查 Redis 缓存
     ├─ 命中 → 直接返回（毫秒级）
     └─ 未命中 → 查 DB → 写入 Redis（设 TTL） → 返回
  ③ DTO 转换 → 返回 resp

复杂度：流程(3步) + 判断(1个缓存命中)
挑战：无（纯读取，Cache-Aside 模式是工程标准做法）
```

# 四、链路二：钱包首页与余额查询

> 入口：C端用户（gate-font）
> 接口数：2个
> 开发阶段：第三期
> 外部RPC依赖：0个
> 复杂度：★☆☆☆☆

## 4.1 QueryWalletBalance — 余额查询

```
[C端App] 打开钱包页面

请求路径：用户 → gate-font → handler → wallet_service.go → Rep → DB → 返回

步骤 ① gateway 从 JWT 提取 userId → metainfo 透传
步骤 ② Service 从 ctx 获取 userId（tracer.GetUserId）
步骤 ③ 参数中取 currency（当前选中币种）
步骤 ④ 查 user_balance 表：
        SELECT * FROM user_balance
        WHERE user_id = ? AND currency = ?
        → 返回最多 4 行（4 个子钱包）

        判断：如果没有记录（新用户首次进入）
        ├─ 返回 4 个子钱包余额全为 0
        └─ 有记录 → 按 wallet_type 组装返回

步骤 ⑤ 组装 resp：
        {
          centerBalance:  {available: 1050, frozen: 0},
          rewardBalance:  {available: 50, frozen: 0},
          agentBalance:   {available: 0, frozen: 0},
          anchorBalance:  {available: 0, frozen: 0},
          currency: "BSB",
          precision: 2
        }

★ 余额不走缓存，直接查 DB（金融数据必须实时准确）
★ 这是钱包最高频的读接口（每次打开钱包都调）
```

## 4.2 QueryUserCurrencyList — 可用币种列表

```
请求路径：用户 → gate-font → handler → wallet_service.go → [Redis + DB] → 返回

步骤 ① userId 从 ctx 获取
步骤 ② 查可用币种列表（走 Redis 缓存）：
        → 返回 [BSB, VND, IDR, THB, USDT]（5 个启用的币种）

步骤 ③ 对每个币种，查该用户的余额合计：
        SELECT currency, SUM(available) as total
        FROM user_balance
        WHERE user_id = ?
        GROUP BY currency

步骤 ④ 合并：币种信息 + 余额合计 → 列表返回
        [
          {code: "BSB", name: "基准币", icon: "...", total: 1100, precision: 2},
          {code: "VND", name: "越南盾", icon: "...", total: 2300000, precision: 0},
          ...
        ]

★ 币种列表走缓存，余额查 DB
★ C端用此列表做币种切换弹窗
```

## 6.2 CreateExchangeOrder 详细链路 ★★★

这是 ser-wallet **单事务复杂度最高**的接口（一个事务内 5 个写操作）：

```
步骤 ① 重新获取最新汇率
        │  rate := domain.GetExchangeRate(ctx, "VND")  // Redis → DB
        │  ★ 不用前端传的汇率值！防止页面停留过久汇率已变
        │
步骤 ② 金额计算（全整数运算，无浮点）
        │  sourceAmount := req.Amount     // 230000（最小单位整数）
        │  targetAmount := sourceAmount / rate  // 230000 / 23000 = 10
        │  giftAmount := targetAmount * giftRate / 100  // 10 * 5 / 100 = 0
        │
        │  ★ 注意精度：如果 BSB precision=2，则：
        │     targetAmount = 1000（10.00 BSB 的最小单位）
        │     giftAmount = 50（0.50 BSB 的最小单位）
        │     具体计算公式取决于精度定义（待确认问题）
        │
步骤 ③ 余额预检
        │  判断：VND 中心钱包 available >= sourceAmount?
        │  ├─ 否 → 返回"余额不足"
        │  └─ 是 → 继续（注意：预检是参考，真正的保障在乐观锁）
        │
步骤 ④ 事务操作（core of core）
        │
        │  rep.ExchangeInTx(ctx, func(tx *gorm.DB) error {
        │
        │    // 操作一：扣法币（乐观锁 ★★★）
        │    rows := tx.Model(&UserBalance{}).
        │        Where("user_id=? AND currency=? AND wallet_type=1", userId, "VND").
        │        Where("available >= ?", sourceAmount).
        │        Update("available", gorm.Expr("available - ?", sourceAmount)).
        │        RowsAffected
        │    if rows == 0 { return ErrInsufficientBalance }
        │    // ★ 这一行 SQL 同时完成"余额检查"和"余额扣减"
        │    //   并发场景下，只有一个请求能成功，其余返回余额不足
        │
        │    // 操作二：加 BSB 中心钱包
        │    tx.Model(&UserBalance{}).
        │        Where("user_id=? AND currency=? AND wallet_type=1", userId, "BSB").
        │        Update("available", gorm.Expr("available + ?", targetAmount))
        │    // ★ 如果 BSB 余额行不存在，需先 INSERT OR UPDATE（懒创建）
        │
        │    // 操作三：加 BSB 奖励钱包（赠送）
        │    tx.Model(&UserBalance{}).
        │        Where("user_id=? AND currency=? AND wallet_type=2", userId, "BSB").
        │        Update("available", gorm.Expr("available + ?", giftAmount))
        │
        │    // 操作四：写兑换订单
        │    tx.Create(&ExchangeOrder{
        │        OrderNo: snowflakeID,
        │        UserId: userId,
        │        SourceCurrency: "VND", SourceAmount: sourceAmount,
        │        TargetCurrency: "BSB", TargetAmount: targetAmount,
        │        GiftAmount: giftAmount, Rate: rate,
        │        Status: SUCCESS,
        │    })
        │
        │    // 操作五：写流水（3 条）
        │    tx.Create(&BalanceFlow{userId, "VND", 1, EXCHANGE_DEDUCT, -sourceAmount, ...})
        │    tx.Create(&BalanceFlow{userId, "BSB", 1, EXCHANGE_CREDIT, +targetAmount, ...})
        │    tx.Create(&BalanceFlow{userId, "BSB", 2, EXCHANGE_GIFT,   +giftAmount, ...})
        │
        │    return nil
        │  })
        │
步骤 ⑤ 返回兑换结果

复杂度：流程(5步) + 判断(2个) + 事务内5个写操作
挑战等级：★★★
  挑战一：事务内多表写操作的原子性保证（全成功或全回滚）
  挑战二：乐观锁的正确实现（WHERE available >= ?）
  挑战三：懒创建 — BSB余额行可能不存在（新用户首次兑换）
  挑战四：流水的 before_balance / after_balance 准确记录
  挑战五：整数精度计算 — 除法取整方向（向上/向下）影响用户实际收到的金额

★★★ 为什么兑换是"烟囱测试"：
  它覆盖了余额操作的所有核心逻辑：
    ✓ 乐观锁（扣VND）
    ✓ 多币种变动（VND减、BSB加）
    ✓ 多子钱包变动（中心+奖励）
    ✓ 事务内多表操作（balance + order + flow）
    ✓ 流水记录（3条）
  但完全不依赖任何外部模块！
  兑换跑通 = 充值入账/提现扣款/人工加减款都可以复用同一套余额操作逻辑
```

## 7.2 CreateWithdrawOrder 详细链路 ★★★★★

**全模块最复杂的单一接口**：

```

  校验A：限额校验（本地）
    │  读 currency_config → 拿到 min_withdraw, max_withdraw, daily_withdraw_limit
    │  查今日已提现金额：
    │    SELECT SUM(amount) FROM withdraw_order
    │    WHERE user_id=? AND currency=? AND status IN ('reviewing','success')
    │    AND DATE(created_at) = CURDATE()
    │
    │  判断：amount < min_withdraw?  → 错误"低于最低限额"
    │  判断：amount > max_withdraw?  → 错误"超过单笔上限"
    │  判断：今日已用 + amount > daily_withdraw_limit? → 错误"超过每日限额"

  校验B：KYC校验（2个RPC，事务外）
    │
    │  判断：提现方式是 USDT?
    │  ├─ 是 USDT → 跳过KYC（链上匿名转账）
    │  └─ 非 USDT → 必须KYC
    │     │
    │     │  status := rpc.KycClient().KycQuery(ctx, req)
    │     │  判断：status != KYC_PASSED?
    │     │  ├─ 未通过 → 错误"请先完成实名认证"
    │     │  └─ 已通过 → 继续
    │     │
    │     │  detail := rpc.KycClient().KycDetail(ctx, req)
    │     │  判断：detail.RealName != withdrawAccount.HolderName?
    │     │  ├─ 不一致 → 错误"提现账户户名与实名认证不一致"
    │     │  └─ 一致 → 继续

  校验C：余额校验（本地）
    │  判断：available >= amount（+ fee，取决于手续费方向）?
    │  ├─ 不够 → 错误"可用余额不足"
    │  └─ 够 → 继续

步骤 ② 冻结余额（事务内，乐观锁 ★★★）

    ===== 事务开始 =====
    │

    │
    │  INSERT withdraw_order (order_no=雪花ID, status='reviewing', ...)
    │  INSERT balance_flow (change_type=WITHDRAW_FREEZE, ...)
    │
    ===== 事务提交 =====
    │
    │  ★★★ 到这里，用户的钱已经被冻结了！
    │  ★★★ 下面调 RPC 如果失败，必须补偿！

步骤 ③ 调支付提交审核（事务外，RPC ★★★★★）

    │  resp, err := rpc.FinanceClient().SubmitWithdrawReview(ctx, &Req{
    │    UserId: userId, Currency: "VND", Amount: amount,

    │     │

    │     │    UPDATE withdraw_order SET status='failed' WHERE order_no=?
    │     │
    │     │  补偿步骤三：写回退流水
    │     │    INSERT balance_flow (change_type=WITHDRAW_UNFREEZE_COMPENSATE, ...)
    │     │
    │     │  判断：补偿是否成功？

    │     │  └─ 失败 → ★ 极端情况 ★

复杂度：流程(6步) + 判断(8个) + 事务(1个，含冻结) + 外部RPC(3个) + 补偿(1个分支)
挑战等级：★★★★★
  挑战一：冻结后 RPC 失败的补偿 — 这是全模块最危险的代码路径
  挑战二：并行校验的编排 — 3个校验条件中2个涉及 RPC
```

# 八、链路六：财务调钱包 RPC

> 入口：ser-finance 通过 Kitex RPC 直调（不经网关）
> 接口数：8个
> 开发阶段：第三期实现，第四期联调
> 核心特征：所有写接口遵循 **完全相同的四步统一模式**

## 8.1 统一处理模式（8个接口的公共骨架）

```
任何一个 RPC 写接口进来后：

步骤 ① 幂等校验（快速返回路径）
        │  用 refOrderNo/refAdjustNo 查状态
        │  判断：已处理完成？
        │  ├─ 是 → 直接返回 true（不做任何操作）
        │  └─ 否 → 继续
        │
步骤 ② 参数校验
        │  userId > 0? currency 合法? amount > 0? walletType ∈ [1,2,3,4]?
        │  任何不合法 → Kitex BizErr 返回错误码
        │
步骤 ③ 事务操作
        │  ===== 事务开始 =====
        │  3a. 余额变动（具体操作因接口而异，见下方差异表）
        │  3b. 订单状态更新（如有关联订单）
        │  3c. 写 balance_flow（唯一键：ref_order_no + change_type → 幂等兜底）
        │  ===== 事务提交 =====
        │
步骤 ④ 返回 true

★ 所有 8 个接口的步骤①②④完全相同
★ 唯一差异在步骤③的余额操作方式
★ 这就是"统一余额变动方法"的价值 — 步骤③抽象为一个通用函数
```

## 8.2 8个接口的差异对照表

```
接口名              │ 余额操作                        │ 乐观锁 │ 关联订单
─────────────────────┼──────────────────────────────────┼────────┼──────────
DepositCredit       │ available += amount               │  否    │ deposit_order → success
WithdrawFreeze      │ available -= X, frozen += X       │  是    │ withdraw_order → reviewing
WithdrawConfirm     │ frozen -= amount                  │  否    │ withdraw_order → success
WithdrawUnfreeze    │ available += X, frozen -= X       │  否    │ withdraw_order → failed
ManualCredit        │ available += amount               │  否    │ （修正记录）
ManualDebit         │ available -= amount               │  是    │ （修正记录）
ManualPatch         │ available += amount               │  否    │ deposit_order → patched
QueryBalanceRpc     │ （纯读取，无事务）                │  无    │ 无
```

**需要乐观锁的只有 2 个**：WithdrawFreeze（冻结扣余额）和 ManualDebit（人工减款）。因为只有"扣余额"才有超扣风险。

## 8.3 以 DepositCredit 为例的完整链路

```
ser-finance ──Kitex RPC──→ ser-wallet

步骤 ① 幂等校验
        │  SELECT status FROM deposit_order WHERE order_no = req.RefOrderNo
        │  判断：
        │  ├─ status = 'success' → return true（已入过账）
        │  ├─ status = 'pending' → 继续（首次入账）
        │  └─ 记录不存在 → return error（订单号无效）

步骤 ② 参数校验
        │  userId > 0, currency 合法, amount > 0, walletType ∈ [1,2,3,4]

步骤 ③ 事务操作
        │  ===== 事务开始 =====
        │
        │  3a. UPDATE deposit_order SET status='success' WHERE order_no=? AND status='pending'
        │      判断：rows=0 → 并发重复（另一个请求先完成），回滚，return true
        │
        │  3b. UPDATE user_balance SET available = available + ?
        │      WHERE user_id=? AND currency=? AND wallet_type=?
        │      （如果行不存在 → INSERT 懒创建）
        │
        │  3c. INSERT balance_flow (ref_order_no=?, change_type=DEPOSIT_CREDIT, amount=+?, ...)
        │      唯一键约束：(ref_order_no, change_type)
        │      判断：唯一键冲突 → 幂等兜底，return true
        │
        │  [可选] 如果有奖金活动：
        │    3d. UPDATE user_balance SET available = available + giftAmount
        │        WHERE user_id=? AND currency='BSB' AND wallet_type=2
        │    3e. INSERT balance_flow (change_type=DEPOSIT_GIFT, ...)
        │
        │  ===== 事务提交 =====

步骤 ④ return true

复杂度：流程(4步) + 判断(3个幂等检查) + 事务(3-5个写操作)
挑战等级：★★★
  挑战一：双层幂等的正确实现 — 应用层+DB层都要处理
  挑战二：奖金活动的条件逻辑 — 有奖金时事务内多出2个操作
  挑战三：懒创建余额行 — 新用户首次充值时 user_balance 行不存在
```

# 一句话总结

**ser-wallet 的 46 个接口，从请求链路视角看可以归结为 7 条主链路。其中 22 个是简单的读操作（查库返回），13 个是标准写操作（遵循统一余额变动方法），10 个涉及跨模块 RPC 调用，1 个（CreateWithdrawOrder）是全模块唯一需要"事务+多RPC+补偿"的极高复杂度链路。编码时的攻关重心应集中在：第三期的"统一余额变动方法"（决定所有写操作的正确性）和第四期的"提现创建+补偿"（决定资金安全的底线）。**
