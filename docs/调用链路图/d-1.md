### 1.2 节点间通信协议

| 链路 | 协议 | 超时 | 传播内容 |
|------|------|------|---------|
| 用户 → 网关 | HTTP/HTTPS POST | N/A | JSON Body + Headers |
| 网关 → 微服务 | Kitex RPC (TTHeader) | 3s | TraceID + UserID + IP + Lang + DeviceID |
| 微服务 → 微服务 | Kitex RPC (TTHeader) | 3s | 同上，自动透传 |
| 微服务 → TiDB | GORM (TCP) | 根据连接池配置 | SQL |
| 微服务 → Redis | go-redis (TCP) | 默认 | 命令 |
| 微服务 → 三方API | HTTP GET | 5s（建议） | API Key |
| 三方支付 → ser-finance | HTTP POST 回调 | N/A | 签名 + 订单数据 |

### 1.3 一句话理解每个节点的角色

```
用户手机        = 请求的起点和终点，只关心"操作结果"
gate-font      = 门卫 — 验身份、记日志、转发请求，不做业务
gate-back      = 管理员门卫 — 验身份+验权限，更严格
ser-wallet     = 钱包管家 — 管余额、管币种、管订单生命周期
ser-finance    = 出纳 — 对接银行(渠道)、处理收付款、审核资金
ser-kyc        = 身份核验员 — 告诉钱包"这个人是否实名认证过"
ser-user       = 户籍处 — 提供用户基础资料
ser-blog       = 档案员 — 记录B端操作日志(oneway，不等回复)
TiDB           = 金库 — 余额、流水、订单的最终真实来源
Redis          = 便签纸 — 幂等标记、配置缓存、频率计数
ETCD           = 通讯录 — 服务发现 + 配置中心
第三方支付渠道   = 银行 — 实际的收款和付款
三方汇率API     = 路透社 — 提供实时汇率数据
```

## 第二章：请求的九层通道——一个请求从头到尾经历什么

### 2.1 C端请求的标准生命周期（九层）

以"用户查询钱包余额"这个最简单的请求为例，完整走一遍九层：

```
┌─────────────────────────────────────────────────────────────────────────┐
│ 第1层：用户设备                                                          │
│   用户点击"钱包"tab → App/H5 发出 HTTP POST /api/wallet/overview        │
│   Headers: X-User-Token, X-User-Language, X-Device-ID, X-Device-Type   │
└───────────────────────────────────┬─────────────────────────────────────┘
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│ 第2层：网关入口 (gate-font, Hertz)                                      │
│   IGlTracer.Start() → 生成 UUID TraceID → 注入 ctx → 设置 CORS         │
│   HertzAccLogStr()  → 记录: <traceId> 200 - 12ms POST /api/wallet/... │
└───────────────────────────────────┬─────────────────────────────────────┘
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│ 第3层：中间件链 (按顺序执行)                                              │
│   SetHeaderInfo()    → 从HTTP头提取Token/IP/Lang/Device → 写入metainfo  │
│   BlackCheck()       → RPC调ser-ip检查IP黑名单 → 不在黑名单则放行         │
│   FontAuthMiddleware → Redis查Token → 解析出UserID → 注入metainfo       │
└───────────────────────────────────┬─────────────────────────────────────┘
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│ 第4层：路由分发                                                          │
│   匹配 POST /api/wallet/overview → wallet.GetWalletOverview handler     │
└───────────────────────────────────┬─────────────────────────────────────┘
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│ 第5层：HTTP→RPC桥接 (rpc.Handler[REQ, RESP])                            │
│   BindAndValidate → JSON Body 反序列化为 Kitex生成的 Thrift 结构体        │
│   rpc.WalletClient() → sync.Once单例 → ETCD发现ser-wallet地址            │
│   WalletClient().GetWalletOverview(ctx, req) → Kitex RPC调用             │
│   (metainfo自动透传：TraceID, UserID, IP, Lang, DeviceID)                │
└───────────────────────────────────┬─────────────────────────────────────┘
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│ 第6层：RPC服务入口 (ser-wallet, Kitex Server)                            │
│   RPC中间件 → access log: [traceId] GetWalletOverview cost=5ms          │
│   handler.go → WalletServiceImpl.GetWalletOverview(ctx, req)            │
│   纯委派 → walletService.GetWalletOverview(ctx, req)                    │
└───────────────────────────────────┬─────────────────────────────────────┘
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│ 第7层：业务逻辑层 (internal/ser/wallet.go)                               │
│   从 ctx 取 UserID → 校验 currencyCode 有效 → 查 currency_config        │
│   查 wallet_account WHERE user_id=X AND currency_code=Y                 │
│   → 返回4种钱包(中心/奖励/主播/代理)各自的 available + frozen             │
│   格式化金额（按币种精度）→ 组装响应                                       │
└───────────────────────────────────┬─────────────────────────────────────┘
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│ 第8层：数据层 (internal/rep/ → GORM Gen → TiDB)                         │
│   SELECT available_balance, frozen_balance                               │
│   FROM wallet_account                                                    │
│   WHERE user_id=? AND currency_code=? AND wallet_type IN (1,2,3,4)      │
│   → 返回 0~4 行数据                                                     │
└───────────────────────────────────┬─────────────────────────────────────┘
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│ 第9层：响应回流 (原路返回)                                                │
│   ser-wallet 返回 resp → Kitex 序列化 → 网关 rpc.Handler 接收            │
│   ret.HttpOne(ctx, resp, I18nClient(), nil)                              │
│   → HttpOk() 包装为 R{traceId, code:0, msg:"", data:{...}}             │
│   → c.JSON(200, R) → HTTP响应 → 用户屏幕显示余额                        │
│                                                                          │
│   如果出错：                                                              │
│   → ret.HttpErr() → 解析 BizStatusError code                            │
│   → RPC调 ser-i18n 翻译错误消息 → R{traceId, code:60240XX, msg:"..."}   │
└─────────────────────────────────────────────────────────────────────────┘
```

### 2.2 九层通道的关键认知

```
第1-4层（网关层）：所有请求都走，你不需要写代码，框架已搞定
第5层（桥接层）：所有请求都走，你只需要写一行 handler 函数
第6层（RPC入口）：所有请求都走，handler.go 是纯委派，不放业务逻辑
第7层（业务层）：★ 你的主战场 ★ 所有业务判断和编排都在这里
第8层（数据层）：GORM Gen 已生成基础 CRUD，你写复杂查询和事务
第9层（响应层）：框架统一处理，你只需要 return resp, err
```

**你日常编码 95% 的时间花在第7层和第8层。** 第5层偶尔涉及（注册新路由），其余都是框架级别的基础设施。

## 第三章：每条业务链路的完整请求旅程

### 3.0 链路阅读指南

每条链路用统一格式描述：
- **→** 同步调用（调用方等待返回）
- **⇢** 异步触发（不等待，后续独立执行）
- **[判断]** 分支点（if/else）
- **{循环}** 重复执行点
- **★** 关键步骤（复杂/高风险）
- **⚡** 挑战点（需要特别注意）

---

### 3.1 充值——查看充值方式

```
复杂度：★★☆☆☆  外部依赖：1个RPC  数据变更：无

用户打开充值页面
│
├→ gate-font [身份验证]
│
├→ ser-wallet.GetDepositMethods(userID, currencyCode)
│    │
│    ├→ [内部] currency_config 确认币种启用
│    │
│    ├→ [RPC] ser-finance.GetPaymentMethods(currencyCode, "deposit")  ★
│    │    └→ 返回：方式列表(名称/图标/快捷金额/最小最大/排序/推荐)
│    │
│    ├→ [内部] 组装响应（合并币种显示配置 + 支付方式列表）
│    │
│    └→ 返回充值方式页面数据
│
└→ 用户看到充值页面

链路特征：流程型（线性，无分支无循环）
唯一风险：ser-finance不可用 → 页面空白 → 需降级策略
```

### 3.2 充值——创建充值订单 ⚡⚡⚡

```
复杂度：★★★★★  外部依赖：1个RPC(关键路径)  数据变更：INSERT订单

用户点击"确认充值"(金额=500000 VND, 方式=银行转账)
│
├→ gate-font [身份验证]
│
├→ ser-wallet.CreateDepositOrder(userID, currencyCode, methodID, amount, activityID?)
│    │
│    ├→ [判断1] 金额校验：amount >= 方式最小值？
│    │    └→ 否：返回"低于最小充值金额"
│    │
│    ├→ [判断2] ★ 重复充值检测：                                         ⚡
│    │    │   查 deposit_order: 同userID + 同currency + 同amount
│    │    │   + status IN (pending_payment, processing)
│    │    │
│    │    ├→ count >= 3：硬拦截，返回"重复充值过多"
│    │    ├→ count 1-2：软提醒（前端已确认则带 confirmOverride=true 跳过）
│    │    ├→ count 0：放行
│    │    └→ [判断] USDT链上充值：跳过检测（链上确认慢，pending正常）
│    │
│    ├→ [内部] 锁定汇率快照：读 currency_config 当前汇率，写入订单
│    │
│    ├→ [判断3] 有活动奖励？
│    │    ├→ 是：算奖励金额 = amount × bonus% ，算稽核流水
│    │    └→ 否：跳过
│    │
│    ├→ [RPC] ★★ ser-finance.MatchAndCreateChannelOrder              ⚡⚡
│    │    │   入参：(methodID, currencyCode, amount, platformOrderNo=C+16)
│    │    │
│    │    │   finance内部：
│    │    │   ├→ 查支付方式 → 关联收款渠道列表
│    │    │   ├→ 轮询算法选择最优渠道(成功率×时间×预警×权重)
│    │    │   ├→ 调第三方渠道API创建订单
│    │    │   └→ 返回：渠道订单号 + 支付信息(跳转URL/二维码/链上地址)
│    │    │
│    │    ├→ [判断] 渠道匹配失败？
│    │    │    └→ 返回"当前充值方式暂不可用"（★ 不创建平台订单）
│    │    │
│    │    └→ 成功：拿到 channelOrderNo + paymentInfo
│    │
│    ├→ [内部] 创建平台充值订单（仅在渠道匹配成功后才创建）             ⚡
│    │    │   INSERT deposit_order:
│    │    │     order_no = C+16位随机
│    │    │     status = pending_payment
│    │    │     amount, currency, method, channel, rate_snapshot, activity_info
│    │    │
│    │    └→ 注意：先调渠道成功 → 再创建订单。不是先创建再调渠道。
│    │         理由：避免渠道失败后产生脏订单。
│    │
│    └→ 返回 (orderNo, paymentType, paymentUrl/qrCode/address, expireTime)
│
└→ 用户被引导到第三方支付页面 / 展示二维码 / 展示链上地址

链路特征：流程 + 判断混合型（3个分支判断 + 1个关键外部调用）
三大挑战：
  ① 重复充值检测的边界条件（USDT豁免、confirmOverride）
  ② MatchAndCreateChannelOrder 是关键路径——它失败整个订单就失败
  ③ 订单创建时机——必须在渠道成功后才创建（先渠道后订单）
```

### 3.3 充值——支付成功回调入账 ⚡⚡⚡

```
复杂度：★★★★☆  外部依赖：被动接收  数据变更：UPDATE余额 + INSERT流水

★ 这是一条异步链路——从第三方回调触发，不是用户直接发起

第三方支付渠道 → HTTP回调 → ser-finance
│
├→ finance: 验证签名（渠道密钥）
├→ finance: 幂等检查（防重复回调）
├→ finance: 更新渠道订单状态 = 支付成功
├→ finance: 更新平台订单状态 = 支付成功
│
├→ [RPC] ★ ser-wallet.CreditBalance (入账主体)                         ⚡
│    │   入参：userId, currencyCode, walletType=1(中心),
│    │         amount=订单金额, refOrderNo=C+16, txType=deposit
│    │
│    │   ┌─ Balance Engine 核心写路径（每次余额变更都走这条路）──────┐
│    │   │                                                          │
│    │   │  Step 1: 幂等检查                                        │
│    │   │    Redis: SetNX wal:idem:{C+16}:{deposit}:{1} TTL=24h   │
│    │   │    → 已存在：直接返回成功（重复回调安全）                   │
│    │   │    → 不存在：继续                                        │
│    │   │                                                          │
│    │   │  Step 2: BEGIN Transaction                               │
│    │   │                                                          │
│    │   │  Step 3: SELECT ... FOR UPDATE                      ⚡   │
│    │   │    锁定 wallet_account 精确一行                           │
│    │   │    WHERE user_id=X AND currency_code='VND'               │
│    │   │          AND wallet_type=1                                │
│    │   │    锁持有时间 < 10ms                                     │
│    │   │                                                          │
│    │   │  Step 4: 校验（入账无条件限制，直接通过）                  │
│    │   │                                                          │
│    │   │  Step 5: UPDATE wallet_account                           │
│    │   │    SET available_balance = available_balance + amount     │
│    │   │                                                          │
│    │   │  Step 6: INSERT transaction_log                          │
│    │   │    (before_amt, change_amt, after_amt, ref_order_no,     │
│    │   │     tx_type, wallet_type)                                │
│    │   │    唯一索引: (ref_order_no, tx_type, wallet_type)        │
│    │   │                                                          │
│    │   │  Step 7: COMMIT                                         │
│    │   └──────────────────────────────────────────────────────────┘
│    │
│    └→ 返回成功
│
├→ [判断] 有活动奖励？
│    ├→ 是：[RPC] ser-wallet.CreditBalance (入账奖励)
│    │    │  入参：walletType=2(奖励), amount=奖励金额,
│    │    │        refOrderNo=同一个C+16, txType=deposit_bonus,
│    │    │        auditMultiplier=N, auditTurnover=M
│    │    │
│    │    │  走同样的 Balance Engine 写路径
│    │    │  幂等键不同：(C+16, deposit_bonus, 2) ≠ (C+16, deposit, 1)
│    │    │  所以不会被幂等拦截，两次入账互不影响
│    │    │
│    │    └→ 返回成功
│    └→ 否：跳过
│
├→ finance: 向第三方回复 success（第三方停止重发回调）
│
└→ 链路结束。用户余额已增加。

链路特征：流程 + 判断型（幂等判断 + 奖励判断）
三大挑战：
```

### 3.7 提现——审核与终态（完整状态机）⚡⚡⚡

```

这是一条跨越两个模块的长链路，由finance驱动状态流转：

                        ┌──────────────────────────────────┐
                        │     提现订单状态机 (7个状态)       │
                        └──────────────────────────────────┘

创建提现订单                                     用户余额状态
    │                                          avail=700, frozen=300
    ▼
┌─────────────────┐
│ 1.待风控审核      │
│ pending_risk     │
└────┬────────┬───┘
     │        │
  [通过]    [拒绝]
     │        │
     │        ▼
     │   ┌─────────────┐
     │   │ 2.风控拒绝    │  → finance调 wallet.UnfreezeBalance
     │   │ risk_rejected│    frozen→available, avail=1000, frozen=0
     │   │ 【终态】      │    用户钱回来了
     │   └─────────────┘
     │
     ▼
┌─────────────────┐
│ 3.待财务审核      │
│ pending_finance  │
└────┬────────┬───┘
     │        │
  [通过]    [拒绝]
     │        │
     │        ▼
     │   ┌─────────────┐
     │   │ 4.财务拒绝    │  → finance调 wallet.UnfreezeBalance
     │   │ fin_rejected │    frozen→available, avail=1000, frozen=0
     │   │ 【终态】      │    用户钱回来了
     │   └─────────────┘
     │
     ▼
┌─────────────────┐
│ 5.待打款          │  → finance发起第三方代付
│ pending_payout   │
└────┬────────┬───┘
     │        │
  [成功]    [失败]
     │        │
     ▼        ▼
┌──────────┐ ┌────────────┐
│ 7.打款成功│ │ 6.打款失败   │
│ payout_ok│ │ payout_fail│
│ 【终态】  │ │ 保持冻结    │  ← avail=700, frozen=300（不动）
└──────────┘ │ 等人工处理  │  ← 不自动解冻！防止渠道实际已付出
  │          └────────────┘
  │            │
  │         [人工确认]
  │            ├→ 重试打款（换渠道）

  │            └→ 确认已付：手动扣冻结
  │
  ▼
finance调 wallet.DebitBalance
(txType=withdraw_confirm, 从frozen扣除)
frozen -= 300000
avail=700, frozen=0
用户钱真正离开了系统

★ 三种终态对钱包的影响：
  拒绝(状态2/4)：UnfreezeBalance → 钱回到可用 → 用户可再次使用
  成功(状态7)：DebitBalance → 钱从冻结消失 → 用户真正提走了
  失败(状态6)：什么都不做 → 钱保持冻结 → 等人工 → 最终走2或7

★ TCC 映射：
  Try    = 创建订单时的 FreezeBalance（3.6节）
  Confirm = 打款成功时的 DebitBalance
```

```
无论请求从哪条链路进来——充值回调、提现冻结、兑换、手动调账、投注扣款——
最终都必须经过同一个入口：Balance Engine 的 ModifyBalance 方法。

这是钱包模块的"心脏"。心脏只做一件事：安全地修改余额并记录流水。
```

### 4.2 ModifyBalance 完整流程图

```
ModifyBalance(userId, currencyCode, walletType, opType, amount, refOrderNo, txType, remark)
    │
    │  ┌─────────────────────────────────────────────────┐
    │  │ 入口校验层                                       │
    │  │                                                  │
    │  │  amount > 0?        → 否：返回 INVALID_AMOUNT    │
    │  │  walletType ∈ 1~5?  → 否：返回 INVALID_PARAM    │
    │  │  currencyCode有效?   → 否：返回 CURRENCY_DISABLED │
    │  └──────────────────────────┬──────────────────────┘
    │                             ▼
    │  ┌─────────────────────────────────────────────────┐
    │  │ 幂等检查层（双层保护）                             │
    │  │                                                  │
    │  │  Redis: GET wal:idem:{refOrderNo}:{txType}:{wt}  │
    │  │  → 存在：直接返回"成功"（幂等命中）                 │
    │  │  → 不存在：SetNX 设置（TTL=24h）→ 继续            │
    │  │                                                  │
    │  │  ★ 即使Redis挂了，Step 6的DB唯一索引也会兜底       │
    │  └──────────────────────────┬──────────────────────┘
    │                             ▼
    │  ┌─────────────────────────────────────────────────┐
    │  │ 事务执行层                                       │
    │  │                                                  │
    │  │  BEGIN Transaction                               │
    │  │                                                  │
    │  │  SELECT available_balance, frozen_balance         │
    │  │  FROM wallet_account                             │
    │  │  WHERE user_id=? AND currency_code=?             │
    │  │        AND wallet_type=?                         │
    │  │  FOR UPDATE;           ← 悲观行锁，精确一行       │
    │  │                        ← 锁持有 < 10ms           │
    │  │                                                  │
    │  │  [判断] 根据 opType 执行不同操作：                 │
    │  │                                                  │
    │  │  ┌─ opType=1(入账) ─────────────────────────┐   │
    │  │  │ 无前置条件                                │   │
    │  │  │ available += amount                       │   │
    │  │  └──────────────────────────────────────────┘   │
    │  │                                                  │
    │  │  ┌─ opType=2(扣减) ─────────────────────────┐   │
    │  │  │ [判断] available >= amount?                │   │
    │  │  │   否 → ROLLBACK → 返回 BALANCE_NOT_ENOUGH │   │
    │  │  │   是 → available -= amount                │   │
    │  │  └──────────────────────────────────────────┘   │
    │  │                                                  │
    │  │  ┌─ opType=3(冻结) ─────────────────────────┐   │
    │  │  │ [判断] available >= amount?                │   │
    │  │  │   否 → ROLLBACK → 返回 BALANCE_NOT_ENOUGH │   │
    │  │  │   是 → available -= amount                │   │
    │  │  │        frozen += amount                   │   │
    │  │  └──────────────────────────────────────────┘   │
    │  │                                                  │
    │  │  ┌─ opType=4(解冻) ─────────────────────────┐   │
    │  │  │ [判断] frozen >= amount?                   │   │
    │  │  │   否 → ROLLBACK → 返回 FROZEN_NOT_ENOUGH  │   │
    │  │  │   是 → frozen -= amount                   │   │
    │  │  │        available += amount                │   │
    │  │  └──────────────────────────────────────────┘   │
    │  │                                                  │
    │  │  ┌─ opType=5(扣冻结) ───────────────────────┐   │
    │  │  │ [判断] frozen >= amount?                   │   │
    │  │  │   否 → ROLLBACK → 返回 FROZEN_NOT_ENOUGH  │   │
    │  │  │   是 → frozen -= amount                   │   │
    │  │  │   ★ 资金真正离开系统（total减少）           │   │
    │  │  └──────────────────────────────────────────┘   │
    │  │                                                  │
    │  │  INSERT INTO transaction_log (                   │
    │  │    user_id, currency_code, wallet_type,          │
    │  │    tx_type, ref_order_no,                        │
    │  │    before_available, before_frozen,               │
    │  │    change_amount, direction,                     │
    │  │    after_available, after_frozen,                │
    │  │    remark, created_at                            │
    │  │  )                                               │
    │  │  ← 唯一索引 (ref_order_no, tx_type, wallet_type) │
    │  │  ← 如果唯一冲突 = 幂等兜底 → ROLLBACK → 返回成功  │
    │  │                                                  │
    │  │  COMMIT                                          │
    │  └──────────────────────────┬──────────────────────┘
    │                             ▼
    │  返回操作结果（成功 + 操作后余额）
    │
    └→ 完成

★★★ 铁律 ★★★
1. 余额变动 和 流水记录 在同一事务 → 不可能出现"余额变了但没有流水"
2. 对账公式永远成立：最新流水的 after_available == wallet_account.available_balance
3. 任何绕过 Balance Engine 直接改数据库的行为都是严重Bug
```

### 7.2 每个步骤的"通过标准"

| 步骤 | 通过标准 |
|------|---------|
| Balance Engine | 100个并发入账 → 余额精确；重复orderNo → 幂等拒绝；余额不足 → 正确拒绝 |
| 币种配置 | CRUD正常；禁用币种后相关操作拒绝；基准币种设后不可改 |
| 汇率引擎 | 3个API全活→均值更新；1个挂→2个均值；全挂→保旧值+告警 |
| 兑换执行 | VND→BSB正确；余额不足拒绝；并发兑换无死锁；流水after=account余额 |
| 充值创建 | Mock渠道成功→创建订单；Mock渠道失败→不创建订单；重复检测生效 |
| 提现创建 | 未KYC拒法币提现；姓名不匹配拒绝；超日限拒绝；冻结+建单原子 |
| 提现终态 | 审核拒绝→解冻正确；打款成功→扣冻结正确；打款失败→保持冻结 |
