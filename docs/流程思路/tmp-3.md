## 一、全局视角：ser-wallet 在整个系统中的定位与流转角色

### 1.1 ser-wallet 承担三个角色

```
角色一：C端用户操作的入口处理者
  用户通过 App → gate-font → ser-wallet 发起请求
  ser-wallet 负责：参数校验、业务逻辑判断、组合内部子模块数据、按需调用外部模块、返回结果
  场景：查余额、充值、兑换、提现、查记录

角色二：B端管理操作的入口处理者
  管理员通过 后台 → gate-back → ser-wallet 发起请求
  ser-wallet 负责：配置管理（币种/汇率）、数据查询（用户钱包）、审计日志
  场景：币种配置管理、汇率日志查询、用户钱包查询

角色三：其他模块的能力提供者（被调用方）
  财务/投注/直播/活动等模块通过 RPC 调用 ser-wallet 的接口
  ser-wallet 负责：余额变更（上账/扣款/冻结/解冻/确认）、余额查询
  场景：充值回调上账、投注扣款/返奖、提现确认/退回、人工修正
```

### 1.2 三个角色之间的关系

```
                   ┌──────────────────────────────┐
                   │          C端用户操作          │
                   │  查余额/充值/兑换/提现/记录    │
                   └──────────────┬───────────────┘
                                  │ 产生订单、变更余额
                   ┌──────────────▼───────────────┐
                   │    ser-wallet 内部数据层       │
                   │  用户钱包余额 + 订单记录 +     │
                   │  提现账户 + 币种配置 + 流水     │
                   └──────────────┬───────────────┘
                    ┌─────────────┴─────────────┐
    ┌───────────────▼──────────┐    ┌───────────▼────────────────┐
    │     B端管理操作          │    │    外部模块 RPC 调用       │
    │  配置影响C端行为          │    │  回调/审核结果变更余额      │
    │  （币种启禁用→充值不可用）│    │  （充值成功→上账到中心钱包）│
    └──────────────────────────┘    └────────────────────────────┘

关键认知：三个角色操作的是同一份数据。B端配置变更会实时影响C端行为，
外部模块的回调会变更C端用户看到的余额。这就是为什么所有写操作都需要
事务保证和幂等控制。
```

## 二、模块内部：ser-wallet 的四大子模块及其内部流转

ser-wallet 内部按业务拆分为四个子模块，它们之间有明确的依赖层次。

### 2.1 四大子模块定义

```
子模块A：币种配置（基础数据层）
  职责：管理币种基础数据（币种列表、汇率、展示规则、启禁用状态）
  数据表：币种表、汇率日志表
  特点：纯基础数据供给，不依赖其他子模块，被所有子模块依赖

子模块B：钱包核心（资金操作层）
  职责：用户钱包的余额管理（查询、上账、扣款、冻结、解冻、确认扣除）
  数据表：用户钱包表、钱包流水表
  特点：所有涉及"钱"的操作最终都要经过这一层

子模块C：交易业务（业务逻辑层）
  职责：充值、兑换、提现的完整业务流程处理
  数据表：充值订单表、提现订单表、兑换记录表、提现账户表
  特点：组合调用子模块A和B，同时对接外部模块（财务/KYC/活动）

子模块D：记录与查询（展示查询层）
  职责：交易记录的查询和展示
  数据表：读上述所有表
  特点：纯读操作，不涉及余额变更
```

### 2.2 子模块之间的依赖层次

```
                 ┌───────────────────────┐
                 │  子模块A：币种配置     │  ← 最底层，被所有模块依赖
                 │  币种列表 / 汇率 /     │
                 │  展示规则 / 状态       │
                 └───────────┬───────────┘
                             │ 向上供给
                 ┌───────────▼───────────┐
                 │  子模块B：钱包核心     │  ← 核心层，提供余额操作原子能力
                 │  余额查询 / 上账 /     │
                 │  扣款 / 冻结 / 解冻   │
                 └──┬────────────────┬───┘
                    │ 被业务层调用   │ 被外部RPC直接调用
         ┌──────────▼──────────┐    │
         │ 子模块C：交易业务    │    │
         │ 充值/兑换/提现流程   │    │
         └──────────┬──────────┘    │
                    │               │
         ┌──────────▼──────────┐    │
         │ 子模块D：记录查询    │    │
         │ 读取所有交易记录     │    │
         └─────────────────────┘    │
                                    │
                       ┌────────────▼────────────┐
                       │ 外部模块 RPC 调用        │
                       │ 财务/投注/直播/活动      │
                       │ 直接调用子模块B的原子能力 │
                       └─────────────────────────┘

关键认知：外部模块调用钱包时，跳过了子模块C（交易业务层），
直接调用子模块B（钱包核心层）的原子操作。
这意味着钱包核心层的接口设计必须足够通用和原子化。
```

### 2.3 子模块A（币种配置）内部流转思路

币种配置是整个钱包系统的"地基"，它的数据变更会影响所有上层操作。

```
【B端管理员操作流程】

管理员首次使用 → 设置基准币种（SetBaseCurrency）
  思路：只能设一次，不可更改。平台启动前必须完成。
  内部流转：
    校验是否已设置 → 确认标记校验（防误操作）→ 写入全局配置
    → 初始化该币种平台汇率（首次取实时汇率）→ 记审计日志

管理员日常操作 → 编辑币种（EditCurrency）
  思路：修改图标、汇率浮动%、阈值%。修改后实时影响C端展示和计算。
  内部流转：
    校验币种存在 → 如有图标上传则调ser-s3获取预签名URL
    → 如修改了浮动%/阈值%则重算入款/出款汇率
    → 更新数据库 → 刷新缓存（Redis）→ 记审计日志

管理员日常操作 → 启用/禁用币种（StatusCurrency）
  思路：禁用币种会导致C端该币种下所有功能不可用。
  内部流转：
    校验非基准币种（基准不可禁用）
    → 如果是禁用：查询是否有用户在该币种下有余额 → 有则附带警告
    → 更新状态 → 刷新缓存 → 记审计日志

【定时任务流程】

汇率定时刷新 → RefreshExchangeRate（定时触发，如每分钟一次）
  思路：维持汇率数据的时效性，是所有涉及汇率操作（充值BSB/兑换/提现BSB）的数据基础。
  内部流转：
    查所有启用的非基准币种 → 对每个币种并行调3家三方汇率API
    → 取平均值得实时汇率 → 计算偏差值（与当前平台汇率的偏差）
    → 偏差 >= 阈值：更新平台汇率 + 重算入款/出款汇率 + 写汇率日志
    → 偏差 < 阈值：只更新实时汇率展示值
    → 所有结果写入Redis缓存
  异常处理思路：
    某一家API超时 → 用另外两家的平均值
    两家以上超时 → 跳过本轮更新，保持上一轮数据
    全部超时 → 告警，不更新，等下一轮
```

### 2.4 子模块B（钱包核心）内部流转思路

钱包核心提供的是**原子操作**，每个方法只做一件事，由上层组合调用。

```
【余额查询】getWalletBalance(userID, currencyCode, walletType?)
  思路：最高频操作，每次打开钱包首页都会调。
  内部流转：
    查Redis缓存（命中则直接返回）
    → 未命中：查数据库（user_wallet表）
    → 组装结果：可用余额 + 冻结余额（中心钱包特有）
    → 回填缓存
  性能考虑：缓存有效期短（如30s），余额变更时主动失效缓存

【上账（加钱）】creditWalletCore(userID, currencyCode, amount, walletType, orderNo)
  思路：所有"给用户加钱"的操作（充值、返奖、加款、补单）最终都走这个方法。
  内部流转：
    幂等校验（orderNo是否已处理，查wallet_flow表或独立幂等表）
    → 校验币种启用 → 如果用户没有该钱包则自动创建
    → 事务内：指定钱包余额 += amount → 写入钱包流水（类型+关联订单号）
    → 失效Redis缓存
  幂等实现思路：
    用orderNo作为唯一键，写入幂等表（或在流水表中用唯一索引）
    → 插入成功：首次处理，继续执行
    → 唯一键冲突：重复请求，直接返回上次结果

【扣款（减钱）】debitWalletCore(userID, currencyCode, amount, orderNo)
  思路：消费/投注扣款，需遵循"优先扣中心，再扣奖励"规则。
  内部流转：
    幂等校验 → 查中心钱包余额 + 奖励钱包余额
    → 校验总余额 >= amount（不足则拒绝）
    → 计算分配：centerDebit = min(centerBalance, amount)；rewardDebit = amount - centerDebit
    → 事务内：中心余额 -= centerDebit；奖励余额 -= rewardDebit
    → 写流水（记录各钱包实际扣了多少，投注场景后续返奖需要这个比例）
    → 失效Redis缓存

【冻结】freezeCore(userID, currencyCode, amount, orderNo)
  思路：提现申请时，先冻结金额（从可用余额移到冻结余额），确保审核期间用户不能把这笔钱花掉。
  内部流转：
    幂等校验 → 校验可用余额 >= amount
    → 事务内：可用余额 -= amount；冻结余额 += amount
    → 写冻结流水（状态：冻结中）
    → 失效缓存

【解冻】unfreezeCore(userID, currencyCode, amount, orderNo)
  思路：提现被驳回时，把冻结金额退回可用余额。与冻结互为逆操作。
  内部流转：
    校验该orderNo有冻结记录且状态为"冻结中"
    → 事务内：冻结余额 -= amount；可用余额 += amount
    → 更新冻结流水状态：冻结中 → 已解冻
    → 失效缓存

【确认扣除】confirmDebitCore(userID, currencyCode, amount, orderNo)
  思路：提现出款成功时，正式扣掉冻结金额。资金永久离开用户钱包。
  内部流转：
    校验该orderNo有冻结记录且状态为"冻结中"
    → 事务内：冻结余额 -= amount（不退回可用，直接减掉）
    → 更新冻结流水状态：冻结中 → 已确认扣除
    → 写扣款流水
    → 失效缓存

【状态机约束】
  冻结 → 确认扣除（出款成功时走这条路）
  冻结 → 解冻（驳回/失败时走这条路）
  确认扣除和解冻针对同一笔冻结**互斥**，只能走其中一条。
  代码层面：用冻结流水的状态字段控制，状态不是"冻结中"则拒绝操作。
```

### 2.5 子模块C（交易业务）内部流转思路

交易业务层是最复杂的，因为它要组合子模块A（币种）+ 子模块B（钱包核心）+ 外部模块。

**充值流程的流转思路见第三节，兑换流程见第四节，提现流程见第五节。**

### 2.6 子模块D（记录查询）内部流转思路

```
【记录列表查询】
  思路：四个Tab（充值/提现/兑换/奖励）读不同的数据表，按当前币种隔离。
  内部流转：
    根据recordType分发 →
      充值记录：查充值订单表（WHERE currency_code = ? AND user_id = ?）
      提现记录：查提现订单表
      兑换记录：查兑换流水表
      奖励记录：查奖励流水表
    → 币种配置：获取格式化规则
    → 按规则格式化金额（VND/IDR整数，THB/BSB两位小数）
    → 分页返回

【记录详情查询】
  思路：在列表数据的基础上扩展字段，附带失败原因等。
  内部流转：
    根据recordType + recordID 查详情 → 组装全量字段
    → 失败状态附带原因枚举
```

## 三、充值流程全景思路（最长链路，涉及方最多）

### 3.1 充值流程的参与方

```
参与方            角色              参与时机
────────         ────────          ────────
ser-wallet       流程主导者         全程参与
币种配置(内部)    基础数据供给       页面加载时（币种、汇率）
财务模块(外部)    支付配置+通道能力   充值方式获取 + 创建通道订单 + 回调上账
活动模块(外部)    奖金活动数据       充值页加载时（可选奖金列表）
三方支付通道       实际收款方         创建通道订单后 → 回调支付结果
```

### 3.2 充值流程六个阶段

```
阶段一：进入充值页面 ——————————————————————————————
  触发：用户从钱包首页点击"充值"

  ser-wallet 处理思路：
    1. 校验当前币种是否启用（子模块A）
    2. 调用财务模块.GetPaymentConfig(currencyCode, "deposit")获取充值方式列表
       ← 返回：方式名称/图标/档位金额/金额范围/关联通道/排序
    3. 如果当前币种是BSB：获取入款汇率（子模块A），用于展示"1BSB=X法币"
    4. 并行请求活动模块获取可用奖金活动（可降级）
    5. 组装返回：充值方式列表 + 档位 + 汇率(BSB时) + 奖金活动列表

  关键认知：
    - 充值方式和档位完全由财务模块的"支付配置"决定，钱包不维护这些数据
    - 如果财务模块接口不可用，充值页无法展示任何充值方式
    - 如果活动模块不可用，降级为"无奖金可选"，不阻断充值流程

阶段二：用户选择充值方式和金额 ——————————————————————
  触发：用户选择支付方式（如银行转账）、选择或输入金额

  ser-wallet 处理思路（前端逻辑为主，后端参与校验）：
    1. 前端按选中方式展示对应档位和金额范围
    2. BSB币种：前端根据汇率实时计算"充值X法币 ≈ BSB Y"
    3. 如果用户选了奖金活动：前端展示"额外+XXX"
    4. 手动输入金额时：校验不小于最低档位

阶段三：用户点击"去充值" → 创建订单 ——————————————
  触发：用户确认充值金额和方式，点击"去充值"按钮

  ser-wallet 处理思路（核心链路）：
    1. 校验基础参数（币种启用、金额范围合法）
    2. 重复转账检测（CheckDuplicateDeposit）：
       查自己的充值订单表 → 同用户+同币种+同金额+待支付/进行中
       → 0笔：通过
       → 1-2笔：返回提示信息（前端弹窗，用户可选继续，如果继续则重新调本接口带confirm标记）
       → ≥3笔：强制拦截，返回错误
       注意：USDT链上充值不适用此规则
    3. BSB充值时：锁定当时的入款汇率（快照），计算实际支付金额
    4. 如果选了奖金活动：调活动模块.ValidateBonus校验有效性
       ← 返回奖金金额、流水倍数
    5. 创建本地充值订单（状态：待支付），记录所有快照数据
    6. 调财务模块.CreatePayOrder(订单信息)
       → 财务内部：轮询匹配通道（三系数评分排序）→ 创建通道订单
       ← 返回：通道订单号 + 支付信息
       - 银行转账/电子钱包：三方承载页URL
       - USDT：收款地址 + 二维码
    7. 更新本地订单，关联通道订单号
    8. 返回给C端：支付跳转信息

  关键认知：
    - 这是钱包模块中同步调用链路最长的接口
    - 重复转账检测是纯内部逻辑，不依赖外部
    - 创建通道订单是对财务模块的强依赖，失败则整个充值失败
    - 本阶段还没有余额变更（用户还没付钱）

阶段四：用户在三方页面完成支付 ———————————————————
  触发：用户在三方承载页完成付款（银行转账/扫码支付/链上转账）

  ser-wallet 不直接参与。
  此阶段由三方通道 → 财务模块处理。
  用户端可以通过轮询 GetDepositOrderStatus 查看订单状态。

阶段五：支付结果回调 → 上账 ————————————————————
  触发：三方通道回调财务模块，告知支付成功/失败

  流转思路：
    三方通道 → 财务模块（接收回调）
    → 财务内部更新充值订单状态
    → 支付成功时：财务模块 → [RPC] ser-wallet.CreditWallet(userID, currencyCode, amount, orderNo)

  ser-wallet 处理思路（被调用方）：
    1. 幂等校验：orderNo是否已处理（防重复回调导致重复入账）
    2. 校验币种存在且启用
    3. 如果用户没有该币种钱包：自动创建
    4. 中心钱包余额 += 充值金额
    5. 如果有奖金活动：奖励钱包余额 += 奖金金额
    6. 写入钱包流水
    7. 更新本地充值订单状态为"已完成"

  关键认知：
    - 幂等性是第一要务，三方通道可能重复回调
    - 此步骤是在"被调用"角色下执行，财务模块是调用方
    - BSB充值的汇率用的是阶段三锁定的快照汇率，不是当前实时汇率
    - 从用户点充值到这一步，可能经过数秒（扫码）到数小时（链上确认）

阶段六：充值完成后的状态同步 ——————————————————————
  触发：上账完成后

  ser-wallet 处理思路：
    - C端用户下次查余额（GetWalletHome）会看到新余额
    - 充值记录（GetRecordList）会展示该笔订单
    - Redis缓存已在上账时失效，查询会拿到最新数据
```

### 3.3 充值流程中的异常处理思路

```
异常场景                       处理思路
──────────                   ──────────
财务模块不可用                  充值页无法展示方式 → 提示"暂不可用"
创建通道订单超时                 本地订单标记"通道创建失败"，用户可重试
三方通道回调延迟                 本地订单保持"待支付"，用户可查询进度
回调重复到达                    幂等校验拦截，直接返回上次结果
回调金额与订单金额不一致         记录异常日志，不上账，进入人工处理
用户在充值页等待过程中汇率变动   以阶段三锁定的快照汇率为准，不受影响
币种在充值过程中被禁用           已创建的订单继续处理完成（到账后上账），但不允许新建订单
```

## 四、兑换流程全景思路（最短链路，内部闭环）

### 4.1 兑换流程的参与方

```
参与方            角色              参与时机
────────         ────────          ────────
ser-wallet       流程全权处理       全程参与
币种配置(内部)    平台汇率供给       汇率查询时
```

**兑换是唯一不依赖任何外部模块的C端写操作。** 这使它成为最先开发联调的最佳选择。

### 4.2 兑换流程三个阶段

```
阶段一：进入兑换页面 ——————————————————————————————
  触发：用户在法币充值中心点击"兑换"icon（BSB充值页无兑换入口）

  ser-wallet 处理思路：
    1. 校验当前币种是法币（VND/IDR/THB），BSB下不展示兑换入口
    2. 从币种配置获取该法币对BSB的平台汇率
    3. 统一展示口径：1 BSB = X 法币
    4. 获取赠送比例配置（如10%）
    5. 返回：汇率 + 赠送比例 + 用户法币钱包余额

阶段二：用户输入兑换金额 ——————————————————————————
  触发：用户在输入框输入法币金额

  ser-wallet 处理思路（前端+后端协同）：
    前端实时计算：
      兑换获得BSB = 输入金额 / 汇率
      赠送BSB = 兑换获得 × 赠送比例
      实际到账BSB = 兑换获得 + 赠送
    后端校验：
      输入金额 <= 法币中心钱包可用余额（超出则提示"已达当前可兑换上限"）

阶段三：用户确认兑换 → 执行 ——————————————————————
  触发：用户点击"兑换"按钮

  ser-wallet 处理思路：
    1. 校验币种启用状态
    2. 获取平台汇率并锁定（防止事务执行过程中汇率变动）
    3. 校验法币中心钱包余额 >= 兑换金额
    4. 计算：
       兑换获得BSB = amount / 锁定汇率
       赠送BSB = 兑换获得 × 赠送比例
    5. 同一数据库事务内执行（三步操作必须原子性）：
       法币中心钱包余额 -= 兑换金额
       BSB中心钱包余额 += 兑换获得BSB
       BSB奖励钱包余额 += 赠送BSB（附带流水倍数要求）
    6. 写入兑换流水记录（含汇率快照、赠送信息、流水倍数）
    7. 失效Redis缓存（法币和BSB两个币种的缓存都要失效）
    8. 返回兑换结果（3秒内完成，体验要求）

  关键认知：
    - 这是跨两个币种的余额变更（法币减少 + BSB增加），必须在同一事务内
    - 赠送到奖励钱包的金额，需要完成X倍流水后才能提现
    - 事务失败则全部回滚，不能出现"法币扣了但BSB没加"
    - 没有任何外部RPC调用，事务边界完全在ser-wallet数据库内
```

## 五、提现流程全景思路（前置条件最多，异步链路最长）

### 5.1 提现流程的参与方

```
参与方            角色              参与时机
────────         ────────          ────────
ser-wallet       流程前半段主导     KYC检查 → 方式获取 → 创建订单+冻结
ser-kyc(外部)    KYC认证状态供给    进入提现页时校验
ser-user(外部)   用户信息供给       保存提现账户时获取KYC姓名
财务模块(外部)    提现方式配置       提现方式/费率/限额 + 创建审核订单
                 审核+出款          风控审核 → 财务审核 → 匹配通道出款
三方通道          实际出款方         出款执行
币种配置(内部)    汇率/格式规则      BSB提现时出款汇率
```

### 5.2 提现流程八个阶段

```
阶段一：进入提现页面 → KYC门槛检查 ———————————————
  触发：用户从充值中心点击"提现"

  ser-wallet 处理思路：
    1. 调 ser-kyc.KycQuery(userID) 查询KYC认证状态
    2. 根据状态分流：
       → 未认证：返回"未认证"标记，前端弹窗提供"去认证"和"继续提现"
         - "去认证"：跳转KYC认证页面（ser-kyc负责）
         - "继续提现"：跳过KYC，仅限USDT提现
       → 已认证：直接进入提现方式选择页

  关键认知：
    - KYC是"可选但解锁高阶渠道的前提"，不是强制
    - 未认证用户不是完全不能提现，只是受限于USDT渠道

阶段二：获取可用提现方式 ———————————————————————
  触发：通过KYC门槛后，展示提现方式列表

  ser-wallet 处理思路（最复杂的过滤逻辑）：
    1. 查用户充值历史（内部数据），确定提现规则：
       法币充值 → 可提法币
       USDT充值 → 可提USDT
       法币+USDT混合充值 → 只可提法币
       法币 → 兑换BSB → 可提法币
    2. 调财务模块.GetPaymentConfig(currencyCode, "withdraw")
       ← 返回：提现方式列表（方式名/图标/费率/限额/排序）
    3. 综合过滤：
       KYC状态过滤（未认证只保留USDT）
       × 充值历史过滤（只保留符合规则的渠道）
       × 财务配置过滤（只保留已启用的方式）
       = 最终可用的提现方式列表
    4. BSB币种：附加出款汇率信息
    5. 返回：提现方式列表 + 费率/限额 + KYC状态

  关键认知：
    - 提现方式的确定需要三方数据：KYC状态 + 充值历史 + 财务配置
    - 这三个数据源的组合过滤是提现流程中最复杂的业务逻辑

阶段三：获取/填写提现账户信息 ———————————————————
  触发：用户选择某种提现方式（如银行卡）

  ser-wallet 处理思路：
    查询该用户该提现方式下已保存的账户信息
    → 有记录（非首次）：返回已保存信息（只读模式），用户直接进入金额输入
    → 无记录（首次）：标记为首次提现，引导用户填写：
      - 银行卡：银行名称/银行账号/账户持有人/开户网点
      - 电子钱包：钱包类型/手机号/账户持有人

  保存账户信息时：
    1. 调 ser-user.GetUserInfoExpend(userID) 获取KYC认证姓名
    2. 强制校验：用户填写的"账户持有人" == KYC姓名
       （前端已置灰不可编辑并自动填充，后端做二次校验防绕过）
    3. 校验通过 → 保存到提现账户表

  关键认知：
    - "账户持有人必须=KYC姓名"是后端强校验，不能只依赖前端
    - 提现账户信息存在ser-wallet库，属于钱包模块的数据
    - 后续提现复用已保存信息，减少操作步骤

阶段四：用户输入提现金额 → 校验 ————————————————
  触发：用户输入提现金额

  ser-wallet 处理思路（前端+后端协同校验）：
    前端实时校验：
      金额 >= 单笔最低限额（来自阶段二获取的财务配置）
      金额 <= 单笔最高限额
    后端提交时校验：
      单笔限额校验（上述重做一遍，防前端绕过）
      单日限额校验：今日已提现金额 + 本次金额 <= 单日上限
      BSB提现时：用出款汇率计算实际出款金额
      余额校验：中心钱包可用余额 >= 提现金额

阶段五：用户确认提现 → 冻结 + 创建订单 —————————
  触发：用户点击"确认提现"按钮

  ser-wallet 处理思路（核心链路）：
    1. 再次调ser-kyc校验KYC状态（防止KYC状态在此期间变更）
    2. 再次校验限额（防并发）
    3. BSB提现：锁定出款汇率快照
    4. 校验中心钱包可用余额 >= 提现金额
    5. FreezeBalance（冻结）：
       可用余额 -= 提现金额
       冻结余额 += 提现金额
       写入冻结流水
    6. 创建本地提现订单（状态：待风控审核）
    7. 调财务模块.CreateWithdrawOrder(订单信息)
       → 财务内部：创建提现审核订单
       ← 返回：财务订单号
    8. 更新本地订单，关联财务订单号
    9. 返回："提现进行中"状态

  关键认知：
    - 冻结操作在调用财务模块之前执行，确保即使财务调用失败也不会出现"余额可用但已提交"
    - 如果步骤7（调财务模块）失败：需要回滚冻结（解冻退回），但这个回滚要注意处理
    - 冻结后用户看到的"可用余额"会减少，但"总余额"不变（可用+冻结=总额）

阶段六：后台审核流程 ——————————————————————————
  触发：提现订单进入财务模块的审核队列

  流转思路（财务模块主导，钱包被调用）：
    风控审核 →
      驳回：财务模块 → ser-wallet.UnfreezeBalance → 解冻退回 → 订单状态"风控驳回"
      通过：进入财务审核
    财务审核 →
      驳回：财务模块 → ser-wallet.UnfreezeBalance → 解冻退回 → 订单状态"财务驳回"
      通过：进入出款

  ser-wallet 被调用时的处理思路：
    UnfreezeBalance(userID, currencyCode, amount, orderNo)：
      校验冻结记录存在且为"冻结中"
      → 冻结余额 -= amount → 可用余额 += amount
      → 更新冻结流水状态 → 更新本地提现订单状态
      → 用户下次查余额会看到恢复

阶段七：出款执行 ————————————————————————————
  触发：财务审核通过，进入出款阶段

  流转思路：
    财务模块匹配代付通道 → 创建代付订单 → 三方通道执行出款
    → 出款结果回传：
      成功：财务模块 → ser-wallet.ConfirmDebit(userID, currencyCode, amount, orderNo)
      失败：财务模块 → ser-wallet.UnfreezeBalance(userID, currencyCode, amount, orderNo)

  ser-wallet 被调用时的处理思路：
    ConfirmDebit(出款成功)：
      校验冻结记录为"冻结中"
      → 冻结余额 -= amount（正式扣除，资金永久离开）
      → 更新冻结流水状态为"已确认扣除"
      → 写入扣款流水 → 更新本地订单状态为"出款成功"

    UnfreezeBalance(出款失败)：
      同阶段六驳回时的处理 → 退回到可用余额

阶段八：提现完成后的状态同步 ——————————————————————
  触发：出款结果处理完成

  ser-wallet 处理思路：
    - 更新本地提现订单的最终状态
    - 用户查询提现记录可看到最终结果
    - 出款成功：用户总余额减少
    - 驳回/出款失败：用户余额恢复原状
```

### 5.3 提现流程时间线认知

```
用户点击提现               ——→ 即时（阶段一~五，秒级）
风控审核                  ——→ 分钟~小时（取决于人工审核速度）
财务审核                  ——→ 分钟~小时
出款执行                  ——→ 分钟~天（取决于通道和T+N规则）
整体从申请到到账           ——→ 可能数小时到数天

在这整个时间跨度内，用户的提现金额一直处于"冻结"状态：
  - 不能被花掉（可用余额已减）
  - 不会凭空消失（冻结余额在）
  - 最终要么确认扣除（出款成功），要么解冻退回（驳回/失败）
```

## 六、外部模块调用钱包的流程思路（钱包作为被调用方）

### 6.1 财务模块 → 钱包（6类调用）

```
调用场景一：充值成功上账（CreditWallet）
  时机：三方通道回调支付成功后
  钱包处理：幂等校验 → 中心钱包 += 金额 → 写流水
  频率：中（每笔充值一次）
  关键：幂等性

调用场景二：提现确认扣除（ConfirmDebit）
  时机：出款成功后
  钱包处理：校验冻结状态 → 冻结余额减少 → 写流水
  频率：低（每笔提现最终一次）
  关键：状态机一致性

调用场景三：提现解冻退回（UnfreezeBalance）
  时机：审核驳回或出款失败后
  钱包处理：校验冻结状态 → 冻结转可用 → 写流水
  频率：低
  关键：与ConfirmDebit互斥

调用场景四：充值补单（SupplementCredit）
  时机：B端补单审核通过后
  钱包处理：幂等校验 → 中心钱包 += 补单金额 → 奖励钱包 += 赠送金额 → 写流水(B+前缀)
  频率：极低
  关键：幂等性 + 双钱包操作

调用场景五：人工加款（ManualCredit）
  时机：B端加款审核通过后
  钱包处理：幂等校验 → 自动创建钱包(如不存在) → 指定钱包类型 += 金额 → 写流水(A+前缀)
  频率：极低
  关键：支持四种钱包类型(中心/奖励/主播/代理) + 自动创建

调用场景六：人工减款（ManualDebit）
  时机：B端减款审核通过后
  钱包处理：幂等校验 → 查所有钱包余额 → 从最高余额的钱包开始级联扣减 → 写流水(M+前缀)
  频率：极低
  关键：跨钱包类型级联扣减 + 实际减款可能<设定金额
  级联扣减示例：
    设定减1000，中心余额800，奖励余额150，主播余额100
    → 扣中心800 → 扣奖励150 → 扣主播50 → 实际减1000
    如果总余额只有900 → 实际减900，记录差异
```

### 6.2 投注模块 → 钱包（3类调用）

```
调用场景一：投注扣款（DebitWallet / BetDebit）
  时机：用户下注时
  钱包处理：幂等校验 → 优先扣中心再扣奖励 → 记录各钱包扣款比例 → 写流水
  频率：高（用户每次下注）
  关键：扣款比例要记录清楚，后续返奖需要按比例分配
  特殊逻辑：
    投注金额800，中心钱包余额600，奖励钱包余额300
    → 扣中心600(75%) + 扣奖励200(25%)
    → 流水记录：centerRatio=75%, rewardRatio=25%
    → 后续返奖时按此比例分配到对应钱包

调用场景二：投注返奖（CreditWallet / BetReturn）
  时机：投注结算时
  钱包处理：
    普通返奖 → 全部进中心钱包（需求4.2资金流向：投注返奖→中心钱包）
    混合投注返奖 → 按投注时的扣款比例分配到中心和奖励钱包
  频率：高
  关键：混合返奖的比例计算

调用场景三：余额查询（GetBalance）
  时机：下注前校验余额
  钱包处理：查缓存或查库，返回可用余额
  频率：极高
```

### 6.3 直播/社交模块 → 钱包（2类调用）

```
调用场景一：主播收入入账（AnchorIncome）
  时机：用户送礼后，分成结算时
  钱包处理：幂等校验 → 主播钱包余额 += 收入金额 → 写流水
  频率：中
  关键：入账到主播钱包，不是中心钱包

调用场景二：余额查询（GetBalance）
  时机：用户送礼前校验余额
  钱包处理：同上
```

### 6.4 活动/任务模块 → 钱包（1类调用）

```
调用场景：活动返奖入账（RewardCredit）
  时机：用户完成活动/任务，获得奖励时
  钱包处理：幂等校验 → 奖励钱包余额 += 奖励金额 → 写流水（含活动信息、流水倍数）
  频率：中
  关键：入账到奖励钱包，不是中心钱包。需记录流水倍数要求。
```

## 七、钱包调用外部模块的流程思路（钱包作为调用方）

### 7.1 调用财务模块（3个核心调用）

```
调用一：获取支付配置 → GetPaymentConfig(currencyCode, type)
  使用场景：C3充值方式列表 / C9提现方式列表
  调用时机：C端用户进入充值或提现页面
  返回数据：方式列表+档位+费率+限额+排序
  超时处理：充值/提现页显示"暂不可用"
  缓存策略：支付配置变更频率低，可以短期缓存（如5分钟）

调用二：创建充值通道订单 → CreatePayOrder(orderInfo)
  使用场景：C4创建充值订单
  调用时机：用户点击"去充值"
  传入数据：用户ID/币种/金额/充值方式
  返回数据：通道订单号+支付信息（承载页URL或收款地址）
  超时处理：本地订单标记"通道创建失败"，用户可重试

调用三：创建提现审核订单 → CreateWithdrawOrder(orderInfo)
  使用场景：C12创建提现订单
  调用时机：用户确认提现（已完成冻结后）
  传入数据：用户ID/币种/金额/账户信息/冻结流水号
  返回数据：财务订单号
  失败处理：需要回滚冻结（调自己的UnfreezeBalance）
```

### 7.2 调用 ser-kyc（1个核心调用）

```
调用：KycQuery(userID)
  使用场景：C9提现方式列表 / C12创建提现订单
  调用时机：用户进入提现页 + 确认提现时
  返回数据：认证状态（未认证/审核中/已认证）
  影响：决定用户可以使用哪些提现渠道
  超时处理：视为"未认证"（保守策略），仅开放USDT提现
```

### 7.3 调用 ser-user（1-2个调用）

```
调用一：GetUserInfoExpend(userID)
  使用场景：C11保存提现账户
  调用时机：用户首次保存提现账户信息时
  返回数据：KYC认证姓名
  用途：强制校验"账户持有人"==KYC姓名

调用二：BatchGetUserInfo(userIDs) — 可选
  使用场景：B6用户钱包列表
  调用时机：B端查询用户钱包时
  返回数据：用户昵称等基础信息
  用途：B端展示
```

### 7.4 调用 ser-blog（审计日志）

```
调用：CreateOperationLog(操作信息)
  使用场景：所有B端写操作（B2编辑币种/B3设置基准币种/B4启禁用币种）
  调用时机：每次B端写操作完成后
  这是工程规范要求，所有B端操作都需要审计留痕
  失败处理：日志写入失败不应阻断业务（降级为本地日志）
```

### 7.5 调用 ser-s3（图标上传）

```
调用：GetPresignUrl(fileType, fileSize)
  使用场景：B2编辑币种时上传图标
  调用时机：管理员上传图标时
  约束：SVG/PNG格式，≤5KB
```

### 7.6 调用活动模块（奖金活动）

```
调用一：GetAvailableBonuses(userID, currencyCode, scene="deposit")
  使用场景：C6获取可用奖金活动列表
  调用时机：充值页加载时
  降级策略：活动模块不可用 → 展示"无奖金可选"，不阻断充值

调用二：ValidateBonus(activityID, amount)

  调用时机：创建充值订单时
  返回数据：奖金金额、流水倍数
  降级策略：校验失败 → 去掉奖金，以无奖金方式继续充值
```

## 八、B端管理操作对C端的影响传导

B端的管理操作不是孤立的，它们会实时影响C端用户的体验。这个传导关系需要清楚认知。

### 8.1 币种管理 → C端影响

```
B端操作：禁用某币种（如禁用VND）
传导路径：
  StatusCurrency(VND, disabled) → 更新数据库 → 刷新Redis缓存
  → C端用户下次请求任何VND操作 → 币种校验不通过 → 返回"该币种暂不可用"
影响范围：
  - VND充值不可用
  - VND提现不可用
  - VND兑换不可用
  - VND钱包余额仍可查看（不清零，只是不能操作）
  - 已创建的VND充值/提现订单继续处理完成（幂等保证）

B端操作：修改汇率浮动%（如VND浮动从1%改为2%）
传导路径：
  EditCurrency(VND, floatRate=2%) → 重算入款/出款汇率 → 更新数据库 → 刷新Redis
  → C端用户下次进入BSB充值选越南银行 → 汇率展示变化
  → 用户下次VND提现BSB → 出款汇率变化
影响范围：
  - BSB充值的入款汇率变化（用户看到的"1BSB=X法币"会变）
  - BSB提现的出款汇率变化
  - 已锁定汇率快照的进行中订单不受影响
```

### 8.2 财务支付配置 → C端影响（间接传导）

```
财务B端操作：禁用某充值方式（如禁用VND电子钱包扫码）
传导路径：
  财务模块更新支付配置 → C端调钱包获取充值方式 → 钱包调财务GetPaymentConfig
  → 返回结果中不再包含该方式 → C端不展示

财务B端操作：修改提现费率
传导路径：
  财务模块更新提现配置 → C端调钱包获取提现方式 → 钱包调财务GetPaymentConfig
  → 返回新费率 → C端展示新的手续费

关键认知：这些配置数据不在ser-wallet库内，每次C端请求都是实时从财务模块获取。
这保证了配置变更的及时性，但也带来了对财务模块接口的强依赖。
```

## 十、关键流程节点的处理原则

### 10.1 幂等性原则

```
所有写操作都通过 orderNo 做幂等控制。

实现思路：
  方案A：在钱包流水表用 orderNo 做唯一索引
    首次：INSERT 成功，继续执行
    重复：INSERT 冲突，直接返回上次结果
  方案B：独立幂等表（idempotent_key + result_snapshot）
    首次：INSERT 成功，执行业务逻辑，写入结果快照
    重复：查到已有记录，直接返回快照

需要幂等的接口：
  CreditWallet / DebitWallet / FreezeBalance / UnfreezeBalance / ConfirmDebit
  ManualCredit / ManualDebit / SupplementCredit / RewardCredit / AnchorIncome
  DoExchange / CreateDepositOrder / CreateWithdrawOrder
```

### 10.2 事务边界原则

```
原则：只在ser-wallet自己的数据库内开事务，不跨服务做分布式事务。

兑换场景（同一事务内）：
  法币中心余额 -= X → BSB中心余额 += Y → BSB奖励余额 += Z → 写流水
  三步操作在同一个MySQL事务内，任一步失败全部回滚。

充值上账（单步事务）：
  中心钱包余额 += X → 写流水
  两步操作在同一事务内。

冻结（单步事务）：
  可用余额 -= X → 冻结余额 += X → 写流水
  三步操作在同一事务内。

不做分布式事务的场景：
  创建充值订单（本地创建订单 + 调财务创建通道订单）
  → 本地先创建，调财务成功后更新关联
  → 调财务失败：本地订单标记失败，不需要复杂的补偿

创建提现订单（本地冻结 + 调财务创建审核订单）
  → 先冻结，再调财务
  → 调财务失败：需要解冻（调自己的unfreezeCore），这不是分布式事务
```

### 10.3 余额安全原则

```
原则一：先校验后操作
  所有余额变更前，先查询当前余额并校验充足性。
  校验和变更在同一事务内（行级锁或乐观锁防并发）。

原则二：钱只能在预期方向流动
  上账（加钱）：只有经过幂等校验的合法请求才能加钱
  扣款（减钱）：只有余额充足时才能扣钱
  冻结（转移）：只从可用转到冻结，不能反向（反向用解冻接口）

原则三：冻结-确认/解冻状态机严格互斥
  冻结记录一旦被确认扣除或解冻，状态不可逆转。
  同一笔冻结的ConfirmDebit和UnfreezeBalance互斥。

原则四：流水可追溯
  每一次余额变更都写入钱包流水表，记录：
    变更前余额 / 变更金额 / 变更后余额 / 变更类型 / 关联订单号 / 操作时间
  这是对账一致性的基础（需求9非功能性要求）。
```

### 10.4 缓存策略原则

```
高频读接口（钱包首页/余额查询）→ Redis缓存
  缓存key：wallet:balance:{userID}:{currencyCode}
  缓存有效期：30-60秒
  失效策略：写操作后主动失效（余额变更时 DEL 对应key）

币种配置数据 → Redis缓存
  缓存key：wallet:currency:{currencyCode}
  缓存有效期：5-10分钟
  失效策略：B端编辑/启禁用后主动失效

汇率数据 → Redis缓存
  缓存key：wallet:rate:{currencyCode}
  缓存有效期：由定时任务刷新频率决定
  更新策略：定时任务每次执行后写入新值

财务支付配置 → 可选短缓存
  缓存有效期：1-5分钟
  考虑：财务配置变更频率低，短期缓存可减少RPC调用
```

### 核心认知

```
1. 钱包的本质是一个"余额管理中枢"，对上层提供原子化的余额操作能力
2. 币种配置是地基，被所有操作依赖；没有币种数据，钱包无法运行
3. 财务模块是最紧密的外部协作方，双向依赖，需要最早对齐接口契约
4. 充值和提现的"完整链路"跨越多个模块+异步回调，但钱包只需要做好自己负责的那段
5. 兑换是唯一不依赖外部的C端写操作，最适合作为首个端到端验证场景
6. 所有余额变更遵循：幂等校验 → 余额校验 → 事务内变更 → 写流水 → 失效缓存
7. 冻结-确认/解冻构成严格状态机，是提现流程正确性的基石
8. 开发顺序：币种配置 → 钱包核心 → 兑换验证 → RPC对外 → 充值(联调) → 提现(联调) → 记录
```
