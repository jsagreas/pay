## 一、全局流程思维框架

### 1.1 我负责什么、怎么想

我负责的是 **币种配置 + 多币种钱包** 两个模块。从流程角度看，我的工作处在整个系统的**中间枢纽位置**：

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│   上游：谁会调用我                                        │
│   ├── C端用户（通过 gate-font → HTTP）                    │
│   │     充值、兑换、提现、查余额、查记录                    │
│   ├── B端运营（通过 gate-back → HTTP）                    │
│   │     币种配置、汇率管理                                 │
│   └── 财务模块 ser-finance（通过 RPC）                    │
│         入账、冻结、解冻、扣款、加减款、查余额               │
│                                                         │
│   我负责的处理核心：                                       │
│   ├── 币种配置模块：管理货币基础数据和汇率                   │
│   └── 钱包模块：管理用户资金（五种子钱包 × 多币种）          │
│                                                         │
│   下游：我会调用谁                                        │
│   ├── 币种配置模块（同进程内调用，不走网络RPC）              │
│   │     汇率查询、换算计算、精度格式化                       │
│   ├── ser-kyc（RPC）：查KYC状态和实名信息                  │
│   ├── ser-blog（RPC oneway）：所有写操作的审计日志           │
│   ├── ser-s3（RPC）：币种图标上传                          │
│   ├── ser-buser（RPC）：B端操作人名称解析                  │
│   └── ser-finance（RPC，待确认方向）：支付通道/配置读取       │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 1.2 三层架构的流程方向

```
Layer 0  币种配置（基础设施层）
           │
           │ 提供：汇率、精度、币种状态、换算能力
           │ 方向：单向向上输出，不依赖任何业务模块
           ▼
Layer 1  多币种钱包（核心业务层）
           │
           │ 提供：余额管理、入账、冻结/解冻/扣款、加减款
           │ 消费：币种配置的汇率和精度
           │ 方向：对下依赖币种配置，对上被财务调用
           ▼
Layer 2  财务管理（运营管理层，他人负责）
           │
           │ 消费：钱包的资金操作能力
           │ 方向：通过RPC调用钱包接口完成资金变动
```

**核心思路**：**币种配置是地基，钱包是枢纽，财务是上层消费者**。我的实现顺序必须自底向上：先币种配置 → 再钱包核心（余额体系）→ 再钱包业务（充值/兑换/提现）→ 最后对接财务的RPC接口。

### 1.3 两条主线：C端用户操作 vs B端/RPC被动响应

我需要处理的流程分为两大类：

**主线一：C端用户主动操作**（我发起，走HTTP）
- 用户充值 → 我创建订单 → 通知财务选通道 → 等三方回调
- 用户兑换 → 我内部处理 → 法币减+BSB加 → 完成
- 用户提现 → 我创建订单+冻结 → 通知财务审核 → 等审核+出款结果
- 用户查余额/记录 → 我查DB返回

**主线二：B端/外部模块被动触发**（别人调我，走RPC）
- 财务通知我入账（WalletCredit）→ 我给中心钱包加钱
- 财务通知我冻结/解冻/扣款 → 我操作冻结余额
- 财务通知我人工加减款 → 我按子钱包分别操作
- 区块链监听通知USDT到账 → 我自行入账

## 二、币种配置模块内部流程

> 依据：需求理解 第三章（币种配置模块）+ 链路关系 第一章（币种配置各接口链路）

### 2.1 币种配置的整体处理思路

币种配置模块的流程最简单直接——它是**纯CRUD + 定时任务**，不依赖任何业务模块，是整个系统的地基。

```
处理思路：

1. 币种数据管理（B端CRUD）
   → 运营创建/编辑币种 → 校验+存DB → 审计日志
   → 关键约束：币种代码唯一、类型不可改、基准币种一次性

2. 汇率自动维护（定时任务）
   → 定时拉取三方API → 计算偏差 → 超阈值更新 → 生成日志
   → 这是所有汇率数据的唯一来源

3. 对外输出配置能力（RPC/HTTP查询）
   → 其他模块通过RPC读取币种配置、汇率、换算结果
   → 高频调用，需要进程内缓存
```

### 2.2 币种CRUD流程

```
┌─ 新增币种 CreateCurrency ─────────────────────────────┐
│                                                        │
│  B端前端 → gate-back → CreateCurrency                  │
│    1. 校验币种代码唯一性（查DB）                          │
│    2. 校验币种类型合法（法币/加密/平台币）                  │
│    3. 如有图标 → RPC调 ser-s3.UploadFile（≤5KB）        │
│    4. 写入币种表（20个字段）                              │
│    5. RPC调 ser-blog.AddActLog（oneway异步）             │
│    6. 返回结果                                          │
│                                                        │
│  要点：这是起点接口，不依赖任何业务接口                     │
│  注意：需求说"币种初始由开发导入"，但接口层仍需提供           │
└────────────────────────────────────────────────────────┘

┌─ 编辑币种 UpdateCurrency ─────────────────────────────┐
│                                                        │
│  流程同 CreateCurrency，区别在于：                       │
│    - 不可编辑项：币种代码、币种类型（前端灰显+后端校验）     │
│    - 可编辑项：浮动%、阈值%、限额、图标、排序、精度、符号等  │
│                                                        │
└────────────────────────────────────────────────────────┘

┌─ 启禁用币种 ToggleCurrencyStatus ─────────────────────┐
│                                                        │
│  流程：查币种存在 → 校验基准币种不可禁用 → 更新状态 → 审计日志│
│  影响：禁用后C端 ListActiveCurrencies 不再返回该币种       │
│  注意："已有余额需特殊提示"——禁用不清零，只是C端不可见       │
│                                                        │
└────────────────────────────────────────────────────────┘

┌─ 设置基准币种 SetBaseCurrency ─────────────────────────┐
│                                                        │
│  流程：查是否已设置 → 已设置则拒绝 → 确认币种存在 → 写标记  │
│  硬约束：一次性操作，设置后永远不可修改                      │
│  影响：设置USDT为基准后，BSB锚定关系 10BSB=1USDT 生效      │
│  前端配合：二次确认弹窗提醒用户                             │
│                                                        │
└────────────────────────────────────────────────────────┘
```

### 2.3 汇率自动更新流程（定时任务 T1）

这是币种配置模块**最核心的自动化流程**——它是所有汇率数据准确性的根基。

```
定时任务触发（cron，例如每分钟一次）
    │
    ├─ Step 1: 拉取实时汇率
    │    HTTP请求3家三方汇率API
    │    取平均值 = 实时汇率
    │    （如果某家API不可用，用剩余的取平均，需容错）
    │
    ├─ Step 2: 逐币种检测偏差
    │    遍历所有启用状态的币种：
    │    │
    │    ├─ 计算入款方向偏差值：
    │    │    偏差% = |实时汇率 - 当前平台汇率| / 实时汇率 × 100%
    │    │
    │    ├─ 计算出款方向偏差值（同公式）
    │    │
    │    └─ 判断：入款偏差 ≥ 阈值% 或 出款偏差 ≥ 阈值%
    │         │
    │         ├─ 是 → 进入 Step 3 更新
    │         └─ 否 → 跳过该币种
    │
    ├─ Step 3: 更新平台汇率
    │    ├─ 更新该币种的平台汇率 = 实时汇率快照
    │    ├─ 重新计算入款汇率 = 新平台汇率 × (1 + 浮动%)
    │    ├─ 重新计算出款汇率 = 新平台汇率 × (1 - 浮动%)
    │    ├─ 写入汇率日志表（日志编号/时间/币种/实时/平台/入款/出款）
    │    └─ RPC调 ser-blog.AddActLog（审计日志）
    │
    └─ 结束本轮检测
```

**关键思考点**：
- 这个定时任务如果不运行，平台汇率就是静止的，所有充值/提现/兑换的汇率都不更新
- 入款偏差和出款偏差是**分别计算**的，任一方超阈值都触发更新
- 更新后入款汇率和出款汇率是用**新的平台汇率**重新计算，不是各自独立更新
- 汇率浮动%和阈值%是每个币种独立配置的，所以每个币种的敏感度不同

### 2.4 汇率查询与换算流程

```
┌─ GetExchangeRate（C端查汇率）──────────────────────────┐
│                                                        │
│  入参：源币种、目标币种、方向（入款/出款）                  │
│  处理：                                                 │
│    1. 查币种表获取平台汇率和浮动%                          │
│    2. 如果方向=入款：返回汇率 = 平台汇率 × (1 + 浮动%)    │
│       如果方向=出款：返回汇率 = 平台汇率 × (1 - 浮动%)    │
│  返回：使用汇率 + 展示口径                                │
│                                                        │
│  被谁调用：                                              │
│    - 钱包.GetDepositConfig（充值页展示汇率）               │
│    - 钱包.PreviewExchange（兑换试算）                     │
│    - 钱包.GetWithdrawConfig（提现页展示换算）              │
│    - C端前端直接调用                                      │
│                                                        │
│  思路：纯计算，无状态修改。数据来源于定时任务维护的平台汇率    │
│  优化：可缓存，定时任务更新后刷新缓存                       │
└────────────────────────────────────────────────────────┘

┌─ CalcExchangeAmount（RPC换算计算）─────────────────────┐
│                                                        │
│  入参：源币种、目标币种、金额、方向                        │
│  处理：                                                 │
│    1. 获取对应方向的汇率（复用GetExchangeRate逻辑）        │
│    2. 计算换算金额（金额 × 汇率 或 金额 / 汇率）          │
│    3. 按目标币种精度四舍五入（法币2位，加密币8位）           │
│  返回：换算金额 + 使用汇率 + 精度                         │
│                                                        │
│  核心定位：统一换算入口，避免各模块自行计算导致精度不一致     │
│  被谁调用（高频！）：                                     │
│    - 钱包.CreateDepositOrder（USDT充值换算）              │
│    - 钱包.PreviewExchange / CreateExchangeOrder（兑换）   │
│    - 钱包.CreateWithdrawOrder（BSB提现换算）              │
│    - 钱包.DepositNotify（USDT到账换算为用户币种）          │
│    - 财务模块（统计报表金额转换）                          │
│                                                        │
│  因为同进程调用，无网络开销，但仍建议缓存汇率数据            │
└────────────────────────────────────────────────────────┘

┌─ GetCurrencyConfig（RPC获取币种配置）──────────────────┐
│                                                        │
│  入参：币种代码（单个或批量）                              │
│  返回：精度/小数位数/千分位符号/小数点符号/货币符号/类型/状态 │
│                                                        │
│  核心定位：所有涉及金额展示的接口都需要调用它来格式化金额     │
│  被调用频率最高——钱包模块5个接口 + 财务模块都需要           │
│  强烈建议进程内缓存，币种配置变化频率极低                    │
└────────────────────────────────────────────────────────┘
```

### 2.5 币种配置模块的流程依赖总结

```
流程依赖图：

CreateCurrency / SetBaseCurrency（人工操作，系统初始化阶段）
         │
         ▼
汇率定时任务 T1（自动运行，持续维护汇率数据）
         │
         ▼
ListActiveCurrencies / GetExchangeRate / GetCurrencyConfig / CalcExchangeAmount
（被钱包和财务模块消费）

结论：币种配置模块是"供给侧"，不依赖任何下游反馈。
     只要币种创建了、汇率定时任务跑起来了，配置能力就可用了。
```

## 三、钱包模块内部流程 — 充值子模块

> 依据：需求理解 4.3-4.5（充值模块）+ 链路关系 2.3-2.7（充值相关接口链路）

### 3.1 充值的整体处理思路

充值是**用户往钱包里放钱**的流程。从我的角度看，充值流程有两个关键分支：

```
分支A: USDT充值（平台承载页）
  - 我自己处理，不需要跳三方
  - 我生成收款地址和二维码
  - 区块链监听服务回调我确认到账
  - 我自己入账（或经finance入账，待确认）

分支B: 银行转账/电子钱包（三方托管页）
  - 我创建订单，通知finance选通道
  - 用户跳到三方页面支付
  - 三方回调给finance
  - finance再调我的WalletCredit入账
```

### 3.2 进入充值页面的流程

用户进入充值页面时，前端需要一次性获取所有渲染数据：

```
C端 → GetDepositConfig(用户ID, 当前币种)
    │
    ├─ [1] 确认币种有效
    │    调用币种.ListActiveCurrencies → 当前币种在启用列表中？
    │    不在 → 返回错误提示
    │
    ├─ [2] 获取汇率（仅BSB币种需要）
    │    调用币种.GetExchangeRate(方向=入款)
    │    → 展示"1 BSB = X USDT"
    │    其他法币币种不需要汇率展示
    │
    ├─ [3] 获取可用充值方式列表
    │    待确认方向：
    │    方案A: RPC调 finance.QueryDepositMethod → 读财务的支付配置
    │    方案B: 查本地缓存的支付配置表
    │    → 返回各方式的配置（限额、手续费、快捷金额档位）
    │
    ├─ [4] 获取额外奖金活动列表
    │    查询当前有效的额外奖金活动
    │    → 活动ID、名称、+N%标签、最低充值门槛、流水倍数
    │
    └─ 返回：充值方式列表 + 各方式配置 + 汇率(BSB) + 奖金活动列表

思路：这是一个聚合接口，把多个数据源拼装成一个响应，减少前端请求次数。
```

### 3.3 创建充值订单的核心流程

```
C端 → CreateDepositOrder(用户ID, 币种, 充值方式, 金额, 奖金活动ID可选)
    │
    ├─ [1] 金额校验
    │    金额 ≥ 该充值方式的最低限额？
    │    不满足 → 返回提示"未达最低充值金额"
    │
    ├─ [2] 重复转账拦截（仅银行转账和电子钱包，USDT不拦截）
    │    ┌───────────────────────────────────────────┐
    │    │ 查DB：同UserID + 同币种 + 同金额              │
    │    │ 状态=待支付/进行中 的订单有多少笔？             │
    │    │                                             │
    │    │ 0笔    → 正常继续                            │
    │    │ 1~2笔  → 返回"软拦截"标记                    │
    │    │          前端弹窗警告，但允许用户点"继续充值"    │
    │    │ ≥3笔   → 返回"硬拦截"标记                    │
    │    │          前端弹窗禁止，只能"查看详情"或关闭      │
    │    └───────────────────────────────────────────┘
    │    为什么USDT不拦截？因为链上充值以实际到账为准，
    │    用户可以分多笔转到同一地址，不存在"重复下单"问题
    │
    ├─ [3] 额外奖金校验（如果用户选了奖金活动）
    │    ├─ 金额 ≥ 该活动的最低充值门槛？
    │    │    不满足 → 气泡提示"需最低充值XXX"
    │    ├─ 该笔订单是否已关联其他活动？（单笔互斥）
    │    └─ 记录奖金活动ID和比例到订单
    │
    ├─ [4] USDT充值的汇率换算
    │    如果是USDT充值：
    │    调用币种.CalcExchangeAmount(用户币种→USDT, 方向=入款)
    │    → 记录：原始金额 + 换算后USDT金额 + 使用汇率
    │
    ├─ [5] 创建订单
    │    生成订单号：C + 16位随机数
    │    写入充值订单表
    │
    ├─ [6] 按充值方式分发
    │    ┌─ USDT ──────────────────────────────────────┐
    │    │ 不需要调财务模块                                │
    │    │ 返回"进入USDT地址页"标记                        │
    │    │ 后续由 GetDepositAddress 生成收款地址            │
    │    └─────────────────────────────────────────────┘
    │    ┌─ 银行转账 / 电子钱包 ────────────────────────┐
    │    │ 需要通知财务模块选通道创建支付                    │
    │    │ 方案A: RPC调 finance.CreatePayment            │
    │    │        → 返回三方跳转URL                       │
    │    │ 方案B: 写消息队列，finance消费处理               │
    │    │ → 返回订单号 + 三方跳转URL                      │
    │    └─────────────────────────────────────────────┘
    │
    ├─ [7] 审计日志
    │    RPC调 ser-blog.AddActLog（oneway异步）
    │
    └─ 返回：订单号 + 状态 + 跳转信息
```

### 3.4 USDT充值后续流程（平台承载页）

```
用户进入收款地址页面：

C端 → GetDepositAddress(用户ID, 订单号, 网络类型TRC20/ERC20/BEP20)
    │
    ├─ 校验订单存在且状态正确
    ├─ 生成/获取该用户该网络的收款地址
    │   （预分配地址池 or 链上动态生成，具体方案待技术评审）
    ├─ 生成二维码
    └─ 返回：收款地址 + 二维码 + 网络类型 + 预计到账时间

用户链上转账后...等待区块链确认...

区块链监听服务 → DepositNotify(链上交易ID, 收款地址, USDT金额, 确认数)
    │
    ├─ [1] 根据收款地址匹配用户和订单
    ├─ [2] 确认数 ≥ 安全阈值（防双花攻击）
    ├─ [3] 幂等检查（同一链上交易ID不重复处理）
    ├─ [4] 汇率换算
    │    调用币种.CalcExchangeAmount(USDT→用户币种, 方向=入款)
    ├─ [5] 入账处理（复用WalletCredit逻辑）
    │    中心钱包余额 += 换算后金额
    │    更新订单状态 → 已完成
    │    写资金流水记录
    ├─ [6] 奖金入账（如有，复用WalletCreditReward逻辑）
    │    奖励钱包余额 += 奖金金额
    │    标记流水稽核要求
    └─ [7] 审计日志

C端轮询：
C端 → QueryDepositStatus(订单号) → 返回当前状态
```

### 3.5 银行/电子钱包充值后续流程（三方托管页）

```
    │                               （这步在财务模块内部处理）
    │

    ├─ finance 更新订单状态
    │
    ├─ finance → RPC调 wallet.WalletCredit(订单号, 金额)
    │    │
    │    ├─ 幂等检查（订单号已入账？→直接返回成功）
    │    ├─ 中心钱包余额 += 金额
    │    ├─ 更新订单状态 → 已完成
    │    ├─ 写资金流水
    │    └─ 审计日志

         │
         ├─ 幂等检查

         └─ 写资金流水
```

### 3.6 充值子模块的关键流程思考

**为什么需要重复转账拦截？**
- 银行转账和电子钱包的订单在三方侧创建后，用户可能忘记已下过单又重新下。如果同一金额同一用户多笔待支付订单，用户实际只付一笔，但系统可能匹配错。软拦截(1-2笔)是提醒，硬拦截(≥3笔)是禁止。
- USDT不拦截是因为链上充值以收款地址+到账金额为准，不存在"匹配错"的问题。

**为什么额外奖金要单笔互斥？**
- 一笔充值只能参加一个奖金活动，因为奖金计算基于充值金额，多个活动叠加会导致奖金计算混乱和薅羊毛风险。

**充值的两种入账路径**
- USDT路径：DepositNotify → 我自行入账（模块内闭环）
- 三方路径：finance.CollectNotify → finance调我的WalletCredit（跨模块RPC）
- 两条路径最终都走到 WalletCredit 的相同逻辑，保证入账操作一致

## 四、钱包模块内部流程 — 兑换子模块

> 依据：需求理解 4.6（兑换模块）+ 链路关系 2.8-2.9（兑换接口链路）

### 4.1 兑换的整体处理思路

兑换是**最简单、最独立**的子模块：
- 仅支持法币→BSB **单向**兑换
- 全程在钱包模块内部闭环
- 不涉及财务模块、不涉及三方支付、不涉及审核
- 只依赖币种配置的汇率数据

```
思维模型：
用户的法币子钱包 ──(减法币)──→ 汇率换算 ──(加BSB)──→ 用户的BSB子钱包
                                   ↑
                             币种配置.CalcExchangeAmount
```

### 4.2 兑换预览流程

```
C端 → PreviewExchange(用户ID, 源币种(法币), 金额)
    │
    ├─ [1] 校验：源币种必须是法币（VND/IDR/THB），目标固定BSB
    │    BSB页面不展示兑换入口（BSB不能换回法币）
    │
    ├─ [2] 校验：金额 ≤ 当前法币中心钱包余额
    │    超出 → 返回提示"已达当前可兑换上限"
    │
    ├─ [3] 获取汇率
    │    调用币种.GetExchangeRate(源=法币, 目标=BSB, 方向=出款)
    │    展示口径统一为：1 BSB = X 法币
    │
    ├─ [4] 计算
    │    调用币种.CalcExchangeAmount(法币→BSB)
    │    兑换获得 BSB = 法币金额 / 汇率
    │    赠送 BSB = 兑换获得 × 赠送比例%
    │    实际到账 BSB = 兑换获得 + 赠送
    │
    ├─ [5] 精度处理
    │    调用币种.GetCurrencyConfig(BSB) → BSB保留2位小数
    │
    └─ 返回：兑换获得BSB + 赠送BSB + 实际到账BSB + 使用汇率 + 赠送比例

注意：PreviewExchange 是"试算"，不产生任何资金变动，不写DB。
```

### 4.3 执行兑换流程

```
C端 → CreateExchangeOrder(用户ID, 源币种, 金额, 前端传入的预览汇率)
    │
    ├─ [1] 校验：源币种 = 法币，目标 = BSB
    │
    ├─ [2] 校验：金额 ≤ 当前法币余额
    │
    ├─ [3] 重新计算最终金额
    │    ┌─────────────────────────────────────────────┐
    │    │ 必须重新调用 CalcExchangeAmount 计算            │
    │    │ 不能直接用 PreviewExchange 的结果               │
    │    │ 原因：两次调用之间汇率可能因定时任务更新而变化      │
    │    │                                               │
    │    │ 可选：比对前端传入的预览汇率与当前实际汇率          │
    │    │ 偏差过大则提示用户"汇率已变化，请重新确认"         │
    │    └─────────────────────────────────────────────┘
    │
    ├─ [4] DB事务操作（必须原子性）
    │    ┌─── 在同一个事务中 ──────────────────────────┐
    │    │ ① 法币中心钱包余额 -= 法币金额                 │
    │    │ ② BSB中心钱包余额 += 兑换获得BSB               │
    │    │ ③ BSB奖励钱包余额 += 赠送BSB（如有赠送）        │
    │    │    → 标记流水稽核要求（赠送金额 × 流水倍数）     │
    │    │ ④ 写入兑换订单记录                             │
    │    │ ⑤ 写入资金流水记录（法币扣减 + BSB入账）         │
    │    └──────────────────────────────────────────┘
    │
    ├─ [5] 审计日志：ser-blog.AddActLog（oneway）
    │
    └─ 返回：兑换成功 + 实际到账BSB（要求3秒内返回）
```

### 4.4 兑换子模块的关键流程思考

**为什么只支持法币→BSB单向？**
- BSB是平台内部币，业务场景是"用法币买BSB去投注/消费"。反向换回没有业务场景需求。

**为什么要重新计算而不用预览结果？**
- 预览和执行之间可能间隔几秒到几分钟。这期间汇率定时任务可能更新了平台汇率。如果用过期汇率执行，会产生汇率差损失。

**事务原子性至关重要**
- 兑换涉及两个不同币种的子钱包余额变动（法币减、BSB加），如果只完成一半就失败了（法币扣了但BSB没加），用户资金就凭空消失了。必须在同一个DB事务中完成所有操作。

**赠送部分进奖励钱包而非中心钱包**
- 赠送的BSB有流水要求，需要追踪。放入奖励钱包可以天然隔离，不影响中心钱包的可用余额计算。

## 五、钱包模块内部流程 — 提现子模块

> 依据：需求理解 4.7（提现模块）+ 链路关系 2.10-2.13（提现接口链路）+ 依赖关系 2.2（提现链路）

### 5.1 提现的整体处理思路

提现是**最复杂的子模块**，涉及 KYC、限额校验、冻结/解冻/扣款状态机、跨模块双审核、三方代付。

```
提现状态机：

  用户发起 → [创建订单+冻结] → 审核中 → [风控审核] → [财务审核] → [出款]
                                   │           │              │
                                   │           ├→ 驳回 → 解冻归还
                                   │           │
                                   │           └→ 通过
                                   │                    │
                                   │                    ├→ 驳回 → 解冻归还
                                   │                    │
                                   │                    └→ 通过 → 发起代付
                                   │                              │
                                   │                              ├→ 出款成功 → 最终扣款
                                   │                              │
                                   │                              └→ 出款失败 → 解冻归还

  冻结金额的三个出路：
  ① 审核驳回 → WalletUnfreeze → 解冻归还到可用余额
  ② 出款失败 → WalletUnfreeze → 解冻归还到可用余额
  ③ 出款成功 → WalletDeduct → 最终从账户扣除（冻结余额清零）
```

### 5.2 进入提现页面的流程

```
C端 → GetWithdrawConfig(用户ID, 当前币种)
    │
    ├─ [1] 查KYC状态
    │    RPC调 ser-kyc.GetUserKycStatus(用户ID)
    │    ┌──────────────────────────────────────────┐
    │    │ 未认证 → 可用提现方式 = 仅USDT              │
    │    │          (前端弹窗：可选"去认证"或"继续提现") │
    │    │ 已认证 → 可用提现方式 = 全渠道               │
    │    │          (银行卡/电子钱包/USDT)              │
    │    └──────────────────────────────────────────┘
    │
    ├─ [2] 确定提现币种规则
    │    查DB用户充值历史，确定"这笔钱是怎么来的"：
    │    ┌──────────────────────────────────────────┐
    │    │ 仅法币充值     → 允许法币提现               │
    │    │ 仅USDT充值     → 允许USDT提现              │
    │    │ 法币+USDT混合  → 走法币提现                │
    │    │ 法币→兑换BSB   → 追溯原始，走法币提现        │
    │    │                                          │
    │    │ 核心原则："从哪来回哪去"                     │
    │    └──────────────────────────────────────────┘
    │
    ├─ [3] 获取提现方式配置
    │    各方式的：单笔最低/最高限额、手续费率、T+N到账时间
    │    （来源待确认：finance的支付配置 or 本地配置）
    │
    ├─ [4] 查当日已提额度
    │    查DB当日提现记录 → 计算已提总额
    │    → 用于判断是否超出单日限额
    │
    ├─ [5] 汇率（BSB提现场景）
    │    调用币种.GetExchangeRate(方向=出款)
    │    → 展示"提现 X BSB ≈ Y 法币/USDT"
    │
    └─ 返回：KYC状态 + 可用方式 + 各方式配置 + 当日已提额度 + 汇率
```

### 5.4 创建提现订单的核心流程

```
C端 → CreateWithdrawOrder(用户ID, 币种, 提现方式, 金额, 账户ID)
    │
    ├─ [1] 账户校验
    │    提现账户存在且归属该用户
    │
    ├─ [2] 限额校验
    │    ├─ 金额 ≥ 单笔最低限额？ 不满足 → "未达单笔最低金额"
    │    ├─ 金额 ≤ 单笔最高限额？ 超出 → "超出单笔最高限额"
    │    └─ 当日已提总额 + 本次金额 ≤ 单日限额？ 超出 → "超出当日限额"
    │
    ├─ [3] 费用计算
    │    手续费 = 金额 × 手续费率%
    │    实际到账 = 金额 - 手续费
    │
    ├─ [4] BSB提现汇率换算（如果是BSB币种）
    │    调用币种.CalcExchangeAmount(BSB→目标法币/USDT, 方向=出款)
    │    记录换算金额和使用汇率
    │
    ├─ [5] 核心事务操作（必须原子，这是最关键的步骤）
    │    ┌─── 在同一个DB事务中 ────────────────────────┐
    │    │ ① 生成订单号 T+16位 → 插入提现订单表           │
    │    │ ② 执行 WalletFreeze 冻结逻辑：                │
    │    │    子钱包可用余额 -= 冻结金额                    │
    │    │    子钱包冻结余额 += 冻结金额                    │
    │    │    插入冻结记录（关联提现单号）                   │
    │    │ ③ 写入资金流水记录                              │
    │    └──────────────────────────────────────────┘
    │
    │    为什么必须事务？
    │    如果创建了订单但冻结失败 → 订单进入审核但钱没冻住
    │    → 用户可以把钱花掉 → 审核通过出款时余额不足
    │    如果冻结了但订单创建失败 → 用户的钱被冻住但没有订单
    │    → 无法解冻，资金被永久锁定
    │
    ├─ [6] 通知财务开始审核
    │    方案A: RPC调 finance.SubmitWithdrawReview(订单号)
    │    方案B: 写消息队列，finance 消费
    │    （具体机制待与财务模块负责人对齐）
    │
    ├─ [7] 审计日志
    │
    └─ 返回：订单号 + 状态(审核中) + 预计到账时间
```

### 5.5 提现后续的被动响应流程

提现订单创建后，后续流程由财务模块驱动，我的角色是**被动响应**：

```
场景1: 风控审核驳回
  finance → RPC调 wallet.WalletUnfreeze(冻结记录ID, 原因=风控驳回)
      ├─ 校验冻结记录存在且状态=冻结中
      ├─ 事务：冻结余额 -= 金额，可用余额 += 金额
      ├─ 更新冻结记录状态 → 已解冻
      ├─ 写资金流水（类型=解冻归还）
      └─ 审计日志

场景2: 财务审核驳回
  同场景1，finance调 wallet.WalletUnfreeze，原因=财务驳回

场景3: 出款成功
  finance → RPC调 wallet.WalletDeduct(冻结记录ID)
      ├─ 校验冻结记录存在且状态=冻结中
      ├─ 事务：冻结余额 -= 金额（最终扣除）
      ├─ 更新冻结记录状态 → 已扣款
      ├─ 更新提现订单状态 → 已完成
      ├─ 写资金流水（类型=提现扣款）
      └─ 审计日志

场景4: 出款失败
  finance → RPC调 wallet.WalletUnfreeze(冻结记录ID, 原因=出款失败)
  同场景1

关键点：
  WalletUnfreeze 和 WalletDeduct 是互斥的——
  同一笔冻结，要么被解冻（钱还给用户），要么被扣款（钱离开用户），
  不能两个都发生。需要在冻结记录上做状态检查防止重复操作。
```

### 5.6 提现子模块的关键流程思考

**KYC与提现渠道的关系**
- 未认证用户也可以提现，但只能走USDT。这是合规需求——法币提现需要实名认证才能确认银行账户归属。USDT提现不需要因为链上地址是匿名的。

**"从哪来回哪去"的提现币种规则**
- 这是反洗钱合规要求。防止用户用A币种充值，通过平台换成B币种后用B币种提现到不同渠道。
- 追溯逻辑：法币充值→兑换BSB→提现BSB时，追溯到原始是法币充值，所以走法币提现通道。

**冻结机制的本质**
- 冻结不是扣款，是"预留"。冻结的金额从可用余额移到冻结余额，用户看到的可用余额减少了，但总余额没变。
- 审核通过出款成功后才真正扣除（WalletDeduct），审核驳回或出款失败则解冻归还（WalletUnfreeze）。
- 这个机制保证了：提现期间用户不能把待提现的钱花掉。

**提现账户不可修改**
- 这是风控要求。防止盗号者修改提现账户后提走资金。如果用户需要修改提现账户，只能联系客服人工处理。

## 七、钱包模块内部流程 — 子钱包余额体系

### 7.1 子钱包结构与余额流程

这是钱包模块最底层的数据模型，所有充值/兑换/提现/加减款都在这个结构上操作。

```
用户在每个币种下，拥有一套完整的子钱包结构：

┌───────────────────────────────────────────────────────┐
│ 用户 UserID + 币种 CurrencyCode                         │
│                                                        │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐             │
│  │ 中心钱包  │  │ 奖励钱包  │  │ 主播钱包  │             │
│  │ 可用余额  │  │ 可用余额  │  │ 可用余额  │             │
│  │ 冻结余额  │  │ 冻结余额  │  │ 冻结余额  │             │
│  └──────────┘  └──────────┘  └──────────┘             │
│       │              │              │                   │
│       └──────────────┘              │                   │
│       资金钱包余额                    │                   │
│       = 中心 + 奖励                  │                   │
│       (C端用户看到的"余额")          主播钱包独立          │
│                                                        │
│  ┌──────────┐  ┌──────────┐                            │
│  │ 代理钱包  │  │ 场馆钱包  │  ← 预留，本期暂不实现        │
│  └──────────┘  └──────────┘                            │
│                                                        │
│  每个币种一套完整结构，币种间互不干扰                       │
└───────────────────────────────────────────────────────┘
```

### 7.2 各类操作对子钱包的影响流程

```
┌─ 充值入账 ────────────────────────────────────────────┐
│ WalletCredit      → 中心钱包.可用余额 += 金额           │
│ WalletCreditReward → 奖励钱包.可用余额 += 奖金金额       │
│                      + 创建流水稽核要求记录               │
└────────────────────────────────────────────────────────┘

┌─ 兑换 ────────────────────────────────────────────────┐
│ CreateExchangeOrder:                                   │
│   法币.中心钱包.可用余额 -= 法币金额                      │
│   BSB.中心钱包.可用余额 += 兑换获得BSB                    │
│   BSB.奖励钱包.可用余额 += 赠送BSB（如有，含流水要求）      │
└────────────────────────────────────────────────────────┘

┌─ 提现冻结 ────────────────────────────────────────────┐
│ WalletFreeze:                                          │
│   中心钱包.可用余额 -= 冻结金额                           │
│   中心钱包.冻结余额 += 冻结金额                           │
│   + 插入冻结记录                                        │
│                                                        │
│ WalletUnfreeze（驳回/失败）:                             │
│   中心钱包.冻结余额 -= 冻结金额                           │
│   中心钱包.可用余额 += 冻结金额  (归还)                   │
│                                                        │
│ WalletDeduct（出款成功）:                                │
│   中心钱包.冻结余额 -= 冻结金额  (最终扣除，钱离开系统)     │
└────────────────────────────────────────────────────────┘

┌─ 人工加减款 ──────────────────────────────────────────┐
│ WalletManualAdd:                                       │
│   中心钱包.可用余额 += X                                 │
│   奖励钱包.可用余额 += Y                                 │
│   主播钱包.可用余额 += Z                                 │
│   代理钱包.可用余额 += W                                 │
│   (各子钱包独立操作)                                     │
│                                                        │
│ WalletManualSub:                                       │
│   各子钱包：实际扣款 = min(期望扣款, 当前余额)             │
│   保证不出现负余额                                       │
└────────────────────────────────────────────────────────┘

┌─ 消费扣款（未来：投注/打赏）─────────────────────────────┐
│ WalletConsume:                                         │
│   消费优先级：先扣中心钱包，再扣奖励钱包                    │
│   如果混合扣款（两个钱包都用到了）→ 记录各自比例            │
│   例：投注100，中心出70+奖励出30 → 比例 7:3              │
│                                                        │
│ WalletReward（返奖）:                                   │
│   按投注时的中心/奖励比例，分别归入对应钱包                  │
│   例：返奖200 → 中心140 + 奖励60                        │
│                                                        │
│ （WalletConsume/WalletReward是预留接口，本期视评审决定）    │
└────────────────────────────────────────────────────────┘
```

### 7.3 余额查询的两种场景

```
场景1: C端用户查余额 → GetWalletOverview
  ├─ 返回：资金钱包余额(=中心+奖励) + 各子钱包可用余额
  ├─ 首次访问时：自动初始化该币种下的子钱包账户记录
  └─ 调用币种.GetCurrencyConfig 格式化金额

场景2: 内部RPC查余额 → GetUserBalance
  ├─ 返回：各子钱包的 可用余额 + 冻结余额 + 总余额
  ├─ 场景：财务人工加减款弹窗需要展示用户各子钱包余额
  └─ 与C端版区别：返回冻结金额，面向内部，字段更全
```

## 八、模块间联动流程 — 币种配置 ↔ 钱包

> 依据：链路关系 第三章（模块内部调用关系）

### 8.1 调用方向

**单向：钱包 → 币种配置**（钱包调用币种配置的能力，币种配置不调用钱包）

### 8.2 调用关系全景

```
钱包模块的哪些接口需要调用币种配置：

[格式化金额类] —— 调用 GetCurrencyConfig
│  GetWalletOverview    → 格式化余额展示
│  GetWalletDetail      → 格式化子钱包余额
│  QueryRecordPage      → 格式化记录金额
│  GetRecordDetail      → 格式化详情金额
│  PreviewExchange      → BSB精度处理
│
[汇率展示类] —— 调用 GetExchangeRate
│  GetDepositConfig     → 充值页展示入款汇率（BSB充值）
│  PreviewExchange      → 兑换页展示汇率
│  GetWithdrawConfig    → 提现页展示出款汇率（BSB提现）
│
[金额换算类] —— 调用 CalcExchangeAmount
│  CreateDepositOrder   → USDT充值金额换算
│  PreviewExchange      → 兑换试算
│  CreateExchangeOrder  → 兑换执行
│  CreateWithdrawOrder  → BSB提现换算
│  DepositNotify        → USDT到账换算为用户币种
│
[币种有效性] —— 调用 ListActiveCurrencies
│  GetDepositConfig     → 确认当前币种有效
```

### 8.3 因为在同一服务进程内，调用方式不走网络RPC

```
实际代码层面的处理思路：

币种配置模块和钱包模块虽然逻辑上是两个独立模块，
但在 ser-wallet 服务中是同一个进程。
所以这些"调用"实际上是 Go 包内方法调用，不经过网络。

优化策略：
├─ 币种配置数据（ListActiveCurrencies、GetCurrencyConfig）
│    → 进程内缓存，变化频率极低
│    → 币种CRUD操作后刷新缓存
│
├─ 汇率数据（GetExchangeRate、CalcExchangeAmount）
│    → 进程内缓存，但需要在定时任务更新后刷新
│    → 缓存key：币种+方向
│
└─ CalcExchangeAmount 被调用频率最高（5次），缓存收益最大
```

### 9.3 跨模块调用的安全保障流程思路

**幂等性**
```
为什么需要：
  finance收到三方回调后调 wallet.WalletCredit，
  如果网络超时，finance不确定wallet是否成功，会重试。
  如果wallet没有幂等检查，就会重复入账。

实现思路：
  每个入账类接口（WalletCredit/CreditReward/ManualAdd/ManualSub）
  都用"关联订单号/修正单号"做幂等key。
  ├─ 收到请求 → 查DB：这个订单号是否已处理？
  ├─ 已处理 → 直接返回成功（不执行任何操作）
  └─ 未处理 → 正常执行
```

**事务边界与补偿**
```
跨服务调用无法做分布式事务，需要补偿机制：

场景：finance调wallet.WalletCredit成功，但finance自身后续逻辑失败
  → wallet已经入账了，无法自动回滚
  → 需要人工处理：finance调wallet.WalletManualSub冲正

场景：finance调wallet.WalletFreeze超时
  → finance不知道wallet是否冻结了
  → wallet需要支持"查询冻结状态"（按提现单号查）
  → finance超时后先查状态再决定重试
```

## 十一、端到端场景流程 — 六大核心业务场景

> 把上面所有拆散的模块内流程和联动流程串起来，展示完整的端到端视角

### 场景一：用户充值 USDT（全流程）

```
时间线：
───────────────────────────────────────────────────────────────

① 用户打开充值页面
   C端 → GetDepositConfig
         → [币种] ListActiveCurrencies ← 确认币种有效
         → [币种] GetExchangeRate(入款) ← BSB场景获取汇率
         ← 返回：充值方式+配置+汇率+奖金活动

② 用户选择额外奖金（可选）
   C端 → ListAvailableBonus
         ← 返回：奖金活动列表

③ 用户输入金额，点击充值
   C端 → CreateDepositOrder(USDT, 金额, 奖金活动ID)
         → [币种] CalcExchangeAmount(用户币种→USDT) ← 换算
         → [DB] 创建订单 C+16位
         → [审计] ser-blog.AddActLog
         ← 返回：订单号

④ 用户查看收款地址
   C端 → GetDepositAddress(订单号, TRC20)
         → 生成收款地址+二维码
         ← 返回：地址+二维码

⑤ 用户完成链上转账，等待确认...

⑥ 区块链监听回调
   监听服务 → DepositNotify(链上交易ID, 地址, USDT金额)
         → [校验] 匹配用户和订单
         → [校验] 确认数 ≥ 安全阈值
         → [校验] 幂等检查（防重复）
         → [币种] CalcExchangeAmount(USDT→用户币种)
         → [入账] 中心钱包 += 换算后金额（WalletCredit逻辑）
         → [入账] 奖励钱包 += 奖金（如有，WalletCreditReward逻辑）
         → [审计] ser-blog.AddActLog

⑦ 用户查看结果
   C端 → QueryDepositStatus(订单号) ← 已完成
```

## 十二、资金安全保障流程思路

> 依据：需求理解 4.9（非功能性需求）+ 依赖关系 4.3（跨模块注意事项）

### 12.1 幂等性保障

```
哪些接口必须幂等：
├── WalletCredit       ← finance回调可能重试
├── WalletCreditReward ← 同上
├── WalletManualAdd    ← 审核通过操作可能重试
├── WalletManualSub    ← 同上
├── DepositNotify      ← 区块链监听可能重复通知

幂等实现思路：
  以"关联订单号/修正单号/链上交易ID"为幂等key
  ├─ 请求到达 → 查DB是否已处理
  ├─ 已处理 → 直接返回上次结果（不执行操作）
  └─ 未处理 → 正常执行 → 记录已处理

  可用DB唯一索引 或 Redis分布式锁实现
```

### 12.2 事务一致性保障

```
需要事务保证的操作：
├── CreateExchangeOrder  法币减+BSB加 必须原子
├── CreateWithdrawOrder  创建订单+冻结 必须原子
├── WalletCredit         入账+写流水 必须原子
├── WalletFreeze/Unfreeze/Deduct  余额变动+状态变更 必须原子
├── WalletManualAdd/Sub  多子钱包操作+写流水 必须原子

实现思路：
  所有涉及余额变动的操作，使用 GORM 事务（gorm.DB.Transaction）
  在同一个事务中完成 余额变动 + 流水写入 + 状态更新
  任一步失败则整体回滚
```

### 12.3 流水与余额一致性

```
核心原则：任何时刻，子钱包余额 = 该子钱包所有流水的净值之和

实现思路：
  每次余额变动都写一条流水记录：
  ├─ 流水类型：充值入账/奖金入账/兑换扣减/兑换入账/提现冻结/
  │           提现解冻/提现扣款/人工加款/人工减款/消费扣款/返奖入账
  ├─ 变动金额：正数=增加，负数=减少
  ├─ 关联订单号：追溯到具体操作
  ├─ 变动后余额：记录变动后的快照值
  └─ 定期对账：流水净值 ≠ 余额时告警
```

### 12.4 防双花（提现场景）

```
问题：用户提现100，如果不冻结，在审核期间用户又消费了这100
解决：WalletFreeze 机制
  提现发起 → 立即冻结 → 可用余额减少 → 用户无法消费已冻结的钱
  审核通过出款成功 → WalletDeduct 最终扣除
  审核驳回/出款失败 → WalletUnfreeze 解冻归还
```

### 12.5 防重复入账（充值场景）

```
问题：三方回调可能因网络问题发送多次，或finance因超时重试
解决：所有入账接口做幂等检查（见12.1）
  额外保险：充值订单表加状态字段
  ├─ 待支付 → 进行中 → 已完成（单向流转，不可逆）
  └─ 已完成状态的订单不再接受入账操作
```

## 十三、实现顺序的流程依赖分析

> 依据：依赖关系 第六章（实现顺序建议）+ 链路关系 第三章（依赖层级）

### 13.1 四层依赖决定实现顺序

```
必须严格按依赖层级从底向上实现：

Phase 1: 地基层（币种配置核心）
├── 数据库表：币种表、汇率日志表
├── IDL + gen_kitex.sh 生成RPC代码
├── gorm_gen.go 生成查询代码
├── 接口：CreateCurrency, UpdateCurrency, ToggleCurrencyStatus, SetBaseCurrency
├── 接口：QueryCurrencyPage, GetCurrencyDetail, QueryRateLogPage
├── 定时任务 T1：汇率偏差检测
└── 验证：B端可以配置币种，汇率自动更新

Phase 2: 配置能力层（对外输出）
├── 接口：ListActiveCurrencies, GetExchangeRate
├── 接口：GetCurrencyConfig, CalcExchangeAmount, GetPlatformRate
├── 进程内缓存机制
└── 验证：C端可以获取币种列表和汇率，RPC调用方可获取配置

Phase 3: 钱包核心层（子钱包余额体系）
├── 数据库表：子钱包余额表、资金流水表、冻结记录表
├── 接口：GetWalletOverview, GetWalletDetail, GetUserBalance
├── 首次访问初始化逻辑
└── 验证：C端可以查余额，内部RPC可以查余额

Phase 4: 兑换（最简单的业务，先做来验证整个流转）
├── 数据库表：兑换订单表
├── 接口：PreviewExchange, CreateExchangeOrder
└── 验证：法币↔BSB兑换全流程

Phase 5: 充值
├── 数据库表：充值订单表
├── 接口：GetDepositConfig, CreateDepositOrder, GetDepositAddress
├── 接口：QueryDepositStatus, ListAvailableBonus
├── 回调：DepositNotify
├── RPC对外：WalletCredit, WalletCreditReward
└── 验证：USDT充值+三方充值全流程

Phase 6: 提现
├── 数据库表：提现订单表、提现账户表
├── 接口：GetWithdrawConfig, GetWithdrawAccount, SaveWithdrawAccount
├── 接口：CreateWithdrawOrder
├── RPC对外：WalletFreeze, WalletUnfreeze, WalletDeduct
└── 验证：提现+冻结/解冻/扣款全流程

Phase 7: 财务联动 + 记录
├── RPC对外：WalletManualAdd, WalletManualSub
├── 接口：QueryRecordPage, GetRecordDetail
└── 验证：人工加减款+记录查询
```

### 13.2 为什么是这个顺序

```
Phase 1→2：币种配置是地基，所有业务操作都依赖它的汇率和精度数据
Phase 3：子钱包余额是所有资金操作的载体，必须先建好
Phase 4：兑换最简单（内部闭环，不涉及三方和审核），用它来验证余额体系是否正确
Phase 5：充值需要对接三方和区块链，比兑换复杂，但不需要冻结机制
Phase 6：提现最复杂（KYC+冻结+审核+代付），放在最后
Phase 7：人工加减款和记录查询依赖Phase 5和6产生的数据
```

### 13.3 可以并行的部分

```
在某些Phase内部，有些工作可以并行：

Phase 1 内部：
  ├─ 并行：所有币种CRUD接口（彼此独立）
  └─ 串行：定时任务T1 依赖币种表已有数据

Phase 5 内部：
  ├─ 并行：GetDepositConfig + ListAvailableBonus（独立查询）
  └─ 串行：WalletCredit/CreditReward 依赖 CreateDepositOrder

Phase 6 内部：
  ├─ 并行：GetWithdrawConfig + GetWithdrawAccount（独立查询）
  └─ 串行：WalletFreeze/Unfreeze/Deduct 依赖 CreateWithdrawOrder
```

## 总结

### 我的核心流程认知

1. **币种配置是纯供给侧**——它只产出数据（币种配置、汇率、换算能力），不消费任何业务数据。是所有流程的起点。

2. **钱包是中间枢纽**——对下依赖币种配置的数据，对上向财务模块暴露资金操作能力。C端用户通过HTTP操作钱包，财务模块通过RPC操作钱包。

3. **子钱包余额体系是钱包的心脏**——中心/奖励/主播三种子钱包，每种有可用余额和冻结余额，每个币种一套独立结构。所有充值、兑换、提现、加减款最终都是在这个结构上做加减法。

4. **兑换是内部闭环**——不涉及任何外部模块，是验证整个钱包架构的最佳起点。

5. **充值有两条入账路径**——USDT（DepositNotify自行入账）和三方（finance→WalletCredit），最终汇聚到相同的入账逻辑。

6. **提现是冻结状态机**——冻结→(审核通过+出款成功→扣款) 或 (驳回/失败→解冻)。创建订单和冻结必须原子操作。

7. **finance是上层消费者**——它通过9条RPC调我的接口完成资金变动，我被动执行。所有入账类接口必须幂等。

8. **实现顺序自底向上**——币种配置 → 子钱包余额 → 兑换 → 充值 → 提现 → 财务联动。
