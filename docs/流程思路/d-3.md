# 第一部分：总体流程全景

## 1.1 业务场景全景图

钱包模块承载的所有业务场景，按用户视角的操作路径排列：

```
用户打开APP
  │
  ├─① 个人中心 → 查看余额 → 币种切换（联动刷新全局数据）
  │
  ├─② 充值 → 选币种 → 选方式 → 选奖金(可选) → 输入金额 → 提交
  │          → 跳转支付 → 支付完成 → 回调查询 → 入账
  │
  ├─③ 提现 → KYC校验 → 选方式 → 选/建账户 → 输入金额 → 提交
  │          → 三级审核 → 通道出款 → 到账
  │
  ├─④ 兑换 → 选法币 → 输入金额 → 预览汇率 → 确认兑换 → 即时到账
  │
  ├─⑤ 记录 → 充值/提现/兑换/奖励 四Tab → 列表 → 详情
  │
  └─⑥ 被动入账/扣款（用户无感知的后台操作）
     ├─ 投注扣款/返奖（游戏模块调钱包RPC）
     ├─ 礼物消费/主播入账（直播模块调钱包RPC）
     ├─ 活动奖励入账（活动模块调钱包RPC）
     └─ 人工加减款（财务后台审核通过后调钱包RPC）
```

## 1.3 数据流向维度

钱包系统存在两种核心数据流：

**信息流**（查询方向）：
```
币种配置 ──提供──→ 钱包/财务/游戏/统计
  │ 币种信息、汇率数据、格式规则
  │
钱包余额 ──提供──→ 财务（人工修正关联）、游戏（投注前校验）
  │
财务方式 ──提供──→ 钱包C端（充值方式列表、提现方式列表）
```

**资金流**（钱的方向）：
```
充值：外部资金 → 三方通道 → 财务确认 → 钱包入账（中心钱包）
提现：钱包冻结 → 财务审核 → 通道出款 → 钱包扣除（中心钱包）
兑换：法币钱包扣除 → BSB钱包增加（进程内事务）
投注：中心钱包(优先) + 奖励钱包(补充) → 游戏扣款
返奖：游戏结算 → 按原始比例分回中心/奖励钱包
礼物：中心钱包扣除 → 主播钱包增加
奖励：活动/任务 → 奖励钱包增加
```

# 第二部分：币种配置模块处理流程

> 定位：L0层基础底座，是所有资金业务的前置依赖
> 职责：提供币种管理、汇率管理、汇率定时同步能力
> 进程位置：ser-wallet 服务内部子模块（与钱包共享进程，不走RPC网络调用）

## 2.1 币种管理流程

### 2.1.1 币种生命周期

```
新增币种 ──→ 默认禁用 ──→ 启用 ──→ 正常使用 ──→ 禁用（下架）
   │                       │           │
   │                       │           ▼
   │                       │     被C端引用（余额/充值/提现）
   │                       │           │
   │                       │    禁用前需评估：
   │                       │    是否有用户持有该币种余额？
   │                       │    是否有进行中的订单？
   │                       │
   │                       ▼
   │              用于：F2币种切换、F3充值方式、
   │                    F4发起充值、F8提现方式、
   │                    F15/F16兑换、所有记录筛选
   │
   └──→ 配置信息：
        编码（不可改）、名称、类型（法币/加密/平台）、
        图标、汇率浮动%、阈值%、小数位数、千分位符号
```

### 2.1.2 新增币种处理流程

```
B端管理员发起（gate-back → /admin/api/wallet/currency/add）
  │
  ▼
① 参数校验
   ├─ 币种编码：非空、格式合法（大写字母3-5位）、全局唯一
   ├─ 币种名称：非空
   ├─ 币种类型：法币(1) / 加密货币(2) / 平台货币(3)
   ├─ 汇率浮动%：0~100，合理范围
   └─ 阈值%：0~100，合理范围
  │
  ▼
② 唯一性查重
   └─ DB查询：相同编码是否已存在（含软删除的）
      ├─ 已存在 → 返回错误"币种编码已存在"
      └─ 不存在 → 继续
  │
  ▼
③ 图标处理（如有）
   └─ 调用 ser-s3 上传图标文件（SVG/PNG≤5k）→ 获得图标URL
  │
  ▼
④ 持久化
   └─ DB插入币种记录
      · 审计字段由 UserPlugin 自动填充（create_by, create_at）
      · 默认状态：禁用（需管理员手动启用）
  │
  ▼
⑤ 操作审计
   └─ 调用 ser-blog.AddActLog（操作类型：新增币种，变更内容：完整配置信息）
  │
  ▼
⑥ 返回成功
```

### 2.1.3 币种状态切换处理流程

```
B端管理员发起（启用/禁用）
  │
  ▼
① 查询当前币种状态
  │
  ▼
② 状态变更影响评估
   ├─ 启用 → 无风险，直接允许
   └─ 禁用 → 需评估影响：
      ├─ C端币种切换将不再返回该币种
      ├─ 已持有该币种余额的用户如何处理？（需评审确认）
      └─ 是否有进行中的充值/提现订单？
  │
  ▼
③ 执行状态变更
   └─ DB更新状态 + 操作审计日志
  │
  ▼
④ 联动影响
   └─ C端 F2 获取币种列表时，禁用的币种将被过滤
   └─ 所有依赖该币种的操作（充值/提现/兑换）将不可用
```

### 2.1.4 基准币种设置流程

```
B端管理员发起（一次性操作）
  │
  ▼
① 查询是否已设置过
   ├─ 已设置 → 返回错误"基准币种设置后不可更改"（需求文档3-2明确规定）
   └─ 未设置 → 继续
  │
  ▼
② 校验目标币种
   └─ 该币种编码是否存在且启用（推荐USDT）
  │
  ▼
③ 写入配置
   └─ DB写入基准币种配置（全局唯一配置项）
  │
  ▼
④ 操作审计
  │
  ▼
⑤ 影响范围
   └─ 所有统计报表、对账审计将以此币种为换算基准
   └─ 财务服务通过 R4 GetBaseCurrency 获取此配置
```

## 2.2 汇率管理流程

### 2.2.1 汇率体系关系图

```
                    ┌─────────────┐
                    │ 三方汇率API  │ (3家取均值)
                    │ A / B / C   │
                    └──────┬──────┘
                           │ 定时拉取（1~5分钟）
                           ▼
                    ┌─────────────┐
                    │  实时汇率    │ ← 三方均值，每次拉取更新
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
              ▼            ▼            ▼
     ┌─────────────┐ ┌──────────┐ ┌──────────┐
     │  入款汇率    │ │ 出款汇率  │ │ 平台汇率  │
     │ =实时×(1+F%) │ │=实时×(1-F%)│ │=条件更新  │
     └──────┬──────┘ └─────┬────┘ └─────┬────┘
            │              │            │
            │              │            │
     用于：充值、        用于：提现、   用于：统计、
     兑换、进场馆        退出场馆      报表、对账
```

### 2.2.2 定时汇率同步处理流程

```
定时任务触发（XXL-JOB / cron，建议频率：1~5分钟）
  │
  ▼
① 查询所有需同步的币种
   └─ DB查询：启用状态 + 非平台币种（排除BSB，BSB锚定USDT固定汇率）
  │
  ▼
② 对每个币种并行/逐个处理：
  │
  ├─ ③ 拉取三方汇率
  │    ├─ HTTP调用 API-A → 获取汇率A
  │    ├─ HTTP调用 API-B → 获取汇率B
  │    └─ HTTP调用 API-C → 获取汇率C
  │    └─ 如果某家API失败 → 降级为可用API的均值（至少1家）
  │    └─ 全部失败 → 该币种本轮跳过，记录告警日志
  │
  ├─ ④ 计算实时汇率
  │    └─ 实时汇率 = (汇率A + 汇率B + 汇率C) / 有效个数
  │
  ├─ ⑤ 更新实时汇率
  │    └─ DB更新该币种的实时汇率记录
  │
  ├─ ⑥ 判断是否需要更新平台汇率
  │    ├─ 偏差值 = |实时汇率 - 当前平台汇率| / 实时汇率 × 100
  │    ├─ 偏差值 < 阈值% → 不更新，跳过
  │    └─ 偏差值 >= 阈值% → 需要更新
  │         ├─ DB更新平台汇率 = 当前实时汇率
  │         └─ DB写入汇率日志（变更时间、币种、旧值、新值、偏差值、阈值）
  │
  └─ ⑦ 完成该币种处理

任务完成，等待下次触发
```

### 2.2.3 汇率被调用的场景流程

```
场景1：充值时获取汇率（F4 发起充值）
  └─ 如果用户充值BSB（跨币种） → 获取入款汇率
     └─ 入款汇率 = 实时汇率 × (1 + 该币种汇率浮动%)
     └─ 用于计算：用户支付法币金额 → 到账BSB金额

场景2：提现时获取汇率（F11 发起提现）
  └─ 如果BSB提现为法币/USDT → 获取出款汇率
     └─ 出款汇率 = 实时汇率 × (1 - 该币种汇率浮动%)

场景3：兑换预览/执行（F15/F16）
  └─ 法币 → BSB → 获取入款汇率
     └─ 到账BSB = 法币金额 / 入款汇率

场景4：统计报表（财务服务调用）
  └─ 所有币种换算为基准币种 → 获取平台汇率

场景5：进入游戏场馆（预留）
  └─ 获取入款汇率 + 记录汇率快照
  └─ 退出时用进入时的快照汇率转回
```

## 2.3 币种配置为钱包提供的内部服务

由于币种配置和钱包在同一个 ser-wallet 进程中，以下能力通过进程内函数调用（非RPC网络）提供：

```
钱包核心模块 ──进程内调用──→ 币种配置模块
  │
  ├─ GetCurrencyInfo(code)       → 获取单个币种完整信息
  ├─ GetEnabledCurrencyList()    → 获取所有启用币种列表
  ├─ GetExchangeRate(code, type) → 获取指定类型汇率（实时/入款/出款/平台）
  ├─ GetBaseCurrency()           → 获取基准币种
  ├─ ValidateCurrency(code)      → 校验币种编码有效性
  └─ GetCurrencyFormat(code)     → 获取金额格式化规则（小数位/千分位）
```

**同时通过RPC暴露给外部服务**：
```
外部服务 ──Kitex RPC──→ ser-wallet
  │
  ├─ 财务服务 → GetCurrencyInfo / GetExchangeRate / GetBaseCurrency
  ├─ 游戏服务 → GetExchangeRate（进出场馆）
  └─ 统计服务 → GetBaseCurrency / BatchGetExchangeRates
```

# 第三部分：钱包核心模块处理流程

> 定位：L1层核心业务，C端用户的资金操作中枢
> 职责：余额管理、充值、提现、兑换、记录、入账/扣款/冻结/解冻能力暴露

## 3.1 钱包初始化与余额查询流程

### 3.1.1 用户钱包初始化

```
触发时机：用户首次注册 / 首次打开钱包页面 / 首次被入账
  │
  ▼
① 检查该用户是否已有钱包记录
   └─ DB查询：user_id + currency_code 维度
  │
  ▼
② 如果不存在 → 自动创建钱包记录
   └─ 为每个启用币种创建：
      ├─ 中心钱包记录（balance=0, frozen=0）
      ├─ 奖励钱包记录（balance=0, frozen=0）
      └─ 主播钱包记录（balance=0, frozen=0）← 是否所有用户都有，待评审确认
   └─ 代理钱包、场馆钱包 → 预留，首期不创建
  │
  ▼
③ 如果已存在 → 直接返回
```

### 3.1.2 余额查询处理流程（F1）

```
C端用户请求（个人中心页加载 / 币种切换后）
  │
  ▼
① 认证鉴权
   └─ FontAuthMiddleware → 从Redis获取CUserTokenPayload → 注入userId到ctx
  │
  ▼
② 提取参数
   ├─ userId ← tracer.GetUserId(ctx)（中间件已注入）
   └─ currencyCode ← 请求参数（当前选中的币种）
  │
  ▼
③ 查询钱包数据
   └─ DB查询该用户+该币种的所有钱包类型余额：
      ├─ 中心钱包余额（centerBalance）
      ├─ 奖励钱包余额（bonusBalance）
      ├─ 主播钱包余额（anchorBalance）← 如果是主播身份
      └─ 冻结金额（frozenAmount）
  │
  ▼
④ 计算展示值
   ├─ 资金钱包余额 = centerBalance + bonusBalance（需求文档2-1"对用户展示的总余额"）
   ├─ 可提现余额 = centerBalance - frozenAmount（需求文档7-5）
   └─ 按币种格式化金额（VND/IDR整数用.千分位，THB/BSB两位小数用,千分位）
  │
  ▼
⑤ 返回
   └─ {totalBalance, centerBalance, bonusBalance, frozenAmount, withdrawableBalance, currency}
```

## 3.2 充值完整流程

### 3.2.1 充值全链路（用户视角 → 系统视角）

```
用户操作                     系统处理                           跨模块交互
────────                    ────────                          ──────────
打开充值页                   F3: 获取充值方式列表               → 财务.GetDepositMethods
  │                           └─ 按币种过滤
  │                           └─ 返回方式列表(名称/图标/金额范围/手续费)
  ▼
选择充值方式                 展示方式详情页
  │
  ▼
(可选)选择奖金活动            F6: 获取奖金活动列表               → 活动服务.GetAvailableBonusActivities
  │                           └─ 按用户+币种过滤
  │                           └─ 返回活动列表(奖金比例/最低充值额/流水倍数)
  ▼
输入充值金额                 F7: 重复转账检测（前端预检或内置在F4中）
  │                           └─ 查DB: 同用户+同币种+同金额近期记录数
  │                           └─ 0笔:安全 / 1-2笔:警告 / ≥3笔:拦截
  ▼
确认提交                     F4: 发起充值（核心接口）
  │                           ├─ 币种校验（币种配置，进程内）
  │                           ├─ 重复转账检测
  │                           ├─ 奖金活动校验（条件）               → 活动服务.CheckBonusEligibility
  │                           ├─ BSB汇率获取（条件，币种配置，进程内）
  │                           ├─ 创建充值订单                       → 财务.CreateDepositOrder
  │                           │    └─ 财务通道轮询分配
  │                           │    └─ 财务创建平台订单(待支付)
  │                           │    └─ 财务对接三方通道 → 三方API
  │                           │    └─ 返回支付URL/USDT地址
  │                           ├─ 记录本地充值订单（关联财务订单号）
  │                           └─ 如有奖金活动，记录关联关系
  │
  ▼
跳转支付 / 展示USDT地址      F5: (前端轮询)查询订单状态          → 财务.GetOrderStatus
  │                           └─ 返回: 支付中/成功/失败/已失效
  ▼
支付完成                     三方通道回调 → 财务服务接收
  │                           └─ 财务校验回调签名+金额
  │                           └─ 财务更新订单状态 → 充值成功
  │                           └─ 财务调钱包入账                    ← 财务.→ ser-wallet.WalletCredit
  │                                ├─ 幂等校验（同orderNo不重复入账）
  │                                ├─ 事务：中心钱包余额 += 到账金额
  │                                ├─ 事务：插入资金流水
  │                                └─ 返回成功
  │                           └─ 如有奖金活动
  │                              └─ 财务调钱包奖励入账              ← 财务.→ ser-wallet.BonusCredit
  │                                   ├─ 事务：奖励钱包余额 += 奖金金额
  │                                   ├─ 事务：记录稽核流水要求
  │                                   └─ 返回成功
  ▼
用户看到余额更新
```

### 3.2.2 充值关键决策点

| 决策点 | 判断逻辑 | 结果 |
|--------|----------|------|
| 重复转账 | 同用户+同币种+同金额 近期次数 | 0→通过, 1-2→警告, ≥3→拦截 |
| 奖金活动 | 充值金额 ≥ 活动最低充值额？ | 满足→记录奖金, 不满足→不参与 |
| BSB充值 | 币种=BSB时需跨币种换算 | 获取入款汇率 → 计算法币支付金额 |
| 通道分配 | 财务按配比轮询+半小时切换 | 自动选择代收通道 |
| 入账幂等 | 同orderNo是否已入账 | 已入账→直接返回成功(不重复加钱) |

### 3.3.2 提现冻结/解冻/扣除的余额变化

```

  中心钱包余额 = 4900, 冻结 = 0, 可提现 = 4900    ← 余额正式减少
```

| KYC校验 | 已认证/未认证 × 提现方式 | 未认证+法币→拒绝, 未认证+USDT→允许 |

## 3.4 兑换完整流程

### 3.4.1 兑换全链路（最自治的流程）

```
用户操作                     系统处理                           跨模块交互
────────                    ────────                          ──────────
打开兑换页                   展示当前法币余额
选择法币币种                 获取该法币对BSB的汇率               进程内：币种配置.GetExchangeRate
输入兑换金额                 F15: 兑换预览（实时计算）            无跨模块调用
  │                           ├─ 校验：币种必须是法币（VND/IDR/THB）
  │                           ├─ 获取入款汇率 = 实时汇率 × (1 + 浮动%)
  │                           ├─ 兑换获得BSB = 法币金额 / 入款汇率
  │                           ├─ 赠送BSB = 兑换获得BSB × 10%
  │                           ├─ 合计到账 = 兑换获得 + 赠送
  │                           └─ 校验法币余额是否充足
  ▼
确认兑换                     F16: 发起兑换                       无跨模块调用
  │                           ├─ 重新获取入款汇率（防止预览后汇率变化）
  │                           ├─ 重新计算兑换结果
  │                           ├─ 余额校验
  │                           ├─ 【DB事务开始】
  │                           │    ├─ 法币中心钱包 -= 兑换金额
  │                           │    ├─ 法币扣款流水（类型：兑换扣款）
  │                           │    ├─ BSB中心钱包 += 兑换获得BSB
  │                           │    ├─ BSB入账流水（类型：兑换入账-中心钱包）
  │                           │    ├─ BSB奖励钱包 += 赠送BSB
  │                           │    ├─ BSB入账流水（类型：兑换赠送-奖励钱包）
  │                           │    ├─ 创建兑换记录（关联汇率快照）
  │                           │    └─ 记录赠送的稽核流水要求（X倍流水后可提现）
  │                           ├─ 【DB事务提交】
  │                           └─ 返回兑换结果
  ▼
用户看到BSB余额增加
```

**关键特征**：
- 兑换是**最独立、最自治**的流程 — 完全不涉及任何跨模块RPC调用
- 不调用财务服务（无需通道、无需审核）
- 不调用KYC服务（兑换不需要实名认证）
- 唯一依赖：币种配置（同进程内调用，零网络开销）
- 因此**建议最先开发兑换功能**，可独立测试

### 3.4.2 兑换事务的原子性要求

```
兑换事务涉及的DB操作（必须在同一事务中）：
  ① 法币中心钱包扣款
  ② 法币流水记录
  ③ BSB中心钱包入账
  ④ BSB入账流水记录
  ⑤ BSB奖励钱包入账（赠送部分）
  ⑥ BSB赠送流水记录
  ⑦ 兑换记录
  ⑧ 稽核流水要求记录

任何一步失败 → 全部回滚 → 用户余额不变
```

# 第四部分：RPC暴露接口处理流程（被其他模块调用）

> 钱包作为资金中枢，需要暴露RPC接口给财务、游戏、直播、活动等模块调用
> 所有RPC接口的通用准则：幂等性、事务性、审计留痕

## 4.1 通用入账流程（R1 WalletCredit）

```
调用方（财务充值回调 / 补单 / 其他入账场景）
  │
  ▼
① 参数校验
   ├─ userId > 0
   ├─ amount > 0
   ├─ currencyCode 有效
   ├─ walletType 有效（中心/奖励/主播/代理）
   └─ orderNo 非空
  │
  ▼
② 幂等校验（核心！防重复入账）
   └─ DB查询：流水表中是否已存在 bizType + orderNo 的记录
      ├─ 已存在 → 直接返回成功（幂等，不重复操作）
      └─ 不存在 → 继续
  │
  ▼
③ 事务操作
   └─ 【DB事务 + SELECT FOR UPDATE】
      ├─ 锁定该用户+该币种+该钱包类型的记录
      ├─ 余额 += amount
      └─ 插入流水记录（方向:收入, bizType, orderNo, 时间戳）
  │
  ▼
④ 返回
   └─ 成功 + 入账后余额
```

**被调用场景**：
- 财务充值成功 → bizType="deposit", walletType=中心钱包
- 财务补单通过 → bizType="supplement", walletType=中心钱包
- 活动奖励 → 通过R16 BonusCredit（底层复用WalletCredit逻辑, walletType=奖励钱包）

## 4.2 投注扣款流程（R10 BetDebit — 优先级扣款）

```
调用方（游戏/投注服务，用户下注时）
  │
  ▼
① 参数校验 + 幂等校验
  │
  ▼
② 优先级扣款逻辑（需求文档2-1"先扣中心钱包，不足部分再扣奖励钱包"）
   └─ 【DB事务 + SELECT FOR UPDATE】
      │
      ├─ 锁定中心钱包 + 奖励钱包
      │
      ├─ 计算扣款分配：
      │    centerDebit = min(centerBalance, totalAmount)
      │    remaining = totalAmount - centerDebit
      │    bonusDebit = min(bonusBalance, remaining)
      │
      ├─ 校验总额：centerDebit + bonusDebit ≥ totalAmount ?
      │    ├─ 不足 → 回滚，返回"余额不足"
      │    └─ 充足 → 继续
      │
      ├─ 中心钱包余额 -= centerDebit
      ├─ 奖励钱包余额 -= bonusDebit
      │
      ├─ 插入流水记录（记录中心/奖励各扣了多少）
      │
      └─ 计算扣款比例：
           centerRatio = centerDebit / totalAmount
           bonusRatio = bonusDebit / totalAmount
  │
  ▼
③ 返回
   └─ 成功 + {centerRatio, bonusRatio}
   └─ 调用方需保存此比例，用于后续返奖时按比例分配
```

## 4.3 投注返奖流程（R11 BetCredit — 按比例返奖）

```
调用方（游戏/投注服务，赛事结果确认后）
  │
  ▼
① 参数校验 + 幂等校验
  │
  ▼
② 按比例计算返奖分配（需求文档2-1"混合投注按占比返奖"）
   ├─ centerCredit = winAmount × originalBetRatio.centerRatio
   └─ bonusCredit = winAmount × originalBetRatio.bonusRatio
  │
  ▼
③ 事务操作
   └─ 【DB事务】
      ├─ 中心钱包余额 += centerCredit
      ├─ 奖励钱包余额 += bonusCredit
      └─ 插入两条流水（分别标注返奖来源）
  │
  ▼
④ 返回成功
```

## 4.4 冻结/解冻/扣除流程（R3/R4/R2 — 提现生命周期）

这三个接口构成提现资金的完整生命周期：

```
提现发起                    审核驳回                   出款成功
R3 WalletFreeze     →     R4 WalletUnfreeze    or   R2 WalletDebit
  │                          │                         │
  ▼                          ▼                         ▼
冻结 amount               解冻 amount                扣除 amount
frozen += amount          frozen -= amount            balance -= amount
可用余额减少              可用余额恢复                frozen -= amount
                                                     余额正式减少

状态变化：
  available ──freeze──→ frozen ──unfreeze──→ available（驳回路径）
  available ──freeze──→ frozen ──debit────→ deducted（出款路径）
```

## 4.5 人工加减款流程（R5/R6）

```
财务审核通过后调用
  │
  ▼
R5 ManualCredit / R6 ManualDebit
  │
  ├─ 参数校验：walletType 必须是中心/奖励/主播/代理之一
  ├─ 幂等校验：同orderNo不重复执行
  ├─ 事务操作：
  │    R5: 指定钱包余额 += amount, 插入流水(类型:人工加款)
  │    R6: 指定钱包余额 -= amount, 插入流水(类型:人工减款)
  └─ 返回成功
```

# 第七部分：流程设计关键原则

## 7.1 资金安全原则

| 原则 | 措施 | 应用场景 |
|------|------|---------|
| 幂等性 | 同一orderNo不重复执行 | R1/R2/R3/R4/R5/R6/R10/R11/R14-R18所有资金操作 |
| 事务性 | 余额变更+流水记录必须在同一DB事务 | 所有资金操作 |
| 行级锁 | SELECT FOR UPDATE 防并发 | 余额变更前锁定钱包记录 |
| 流水完整 | 每笔资金变动必有流水记录 | 需求文档9"任何资金变动都要有流水记录" |
| 余额非负 | 扣款前校验余额充足 | R2/R10/R14/F11/F16所有扣款场景 |
| 审计追踪 | 关键操作写操作日志 | 所有B端写操作、C端提现 |

## 7.2 跨模块调用容错原则

| 场景 | 容错策略 |
|------|---------|
| 财务服务不可用 | F3/F4/F5/F8/F11返回友好错误提示，不崩溃 |
| KYC服务不可用 | F11提现按"未认证"处理（保守策略） |
| 活动服务不可用 | F6返回空列表，F4跳过奖金校验 |
| 三方汇率API失败 | 降级为可用API均值，全失败则沿用上次汇率 |
| ser-blog不可用 | 日志写入失败不影响主流程（非阻塞式） |

## 7.3 开发实施顺序建议

基于依赖分析和流程复杂度，建议的开发顺序：

```
Phase 1：基础底座（无外部依赖）
  ├─ 币种配置 B端9个接口（纯DB CRUD）
  ├─ 币种配置 RPC暴露6个接口
  └─ 汇率同步定时任务

Phase 2：钱包核心（依赖Phase1）
  ├─ 钱包初始化 + F1余额查询 + F2币种列表
  ├─ R1/R2 通用入账/扣款（所有RPC的基础）
  ├─ R3/R4 冻结/解冻
  └─ R7/R8 余额查询RPC

Phase 3：兑换模块（最独立，可单独测试）
  ├─ F15 兑换预览
  └─ F16 发起兑换

Phase 4：充值模块（依赖财务服务）
  ├─ F3/F4/F5/F6/F7 充值全链路
  └─ 需与财务开发者对齐RPC契约

Phase 5：提现模块（依赖财务+KYC+用户服务）
  ├─ F8/F9/F10/F11/F12 提现全链路
  ├─ R5/R6 人工加减款
  └─ 需与财务开发者对齐冻结时序

Phase 6：扩展能力
  ├─ R10/R11 投注扣款/返奖
  ├─ R14/R15/R16/R17 礼物/主播/奖励/代理
  ├─ R18 钱包间转账
  ├─ F17-F21 记录查询
  └─ B1/B2 后台查询接口
```

## 7.4 待与财务开发者对齐的关键问题

| 编号 | 问题 | 影响 |
|------|------|------|
| Q1 | 提现冻结时序：钱包先冻结再调财务建单，还是财务建单时回调钱包冻结？ | 决定F11和R3的调用顺序 |

## 8.2 Service层方法调用关系

```
wallet_service.go（核心，被其他service引用）
  ├─ Credit(ctx, userId, currency, walletType, amount, bizType, orderNo)
  │    ← deposit_service.go 充值入账时调用
  │    ← bet_service.go 返奖时调用
  │    ← 直接暴露为RPC R1/R5/R15/R16
  │
  ├─ Debit(ctx, userId, currency, walletType, amount, bizType, orderNo)
  │    ← withdraw_service.go 提现扣款时调用
  │    ← bet_service.go 投注扣款时调用
  │    ← 直接暴露为RPC R2/R6/R14
  │
  ├─ Freeze(ctx, userId, currency, amount, orderNo)
  │    ← withdraw_service.go 提现冻结时调用
  │    ← 直接暴露为RPC R3
  │
  ├─ Unfreeze(ctx, userId, currency, amount, orderNo)
  │    ← 直接暴露为RPC R4
  │
  └─ GetBalance(ctx, userId, currency)
       ← 被所有service引用
       ← 直接暴露为RPC R7/R8

currency_service.go（被钱包核心引用）
  ├─ GetCurrencyInfo(ctx, code)
  ├─ GetEnabledList(ctx)
  ├─ GetExchangeRate(ctx, code, rateType)
  └─ GetBaseCurrency(ctx)
       ← exchange_service.go 获取入款汇率
       ← deposit_service.go 获取入款汇率（BSB充值）
       ← withdraw_service.go 获取出款汇率（BSB提现）

exchange_service.go（最独立）
  ├─ Preview(ctx, fromCurrency, amount) → 调用 currency_service
  └─ Execute(ctx, fromCurrency, amount) → 调用 currency_service + wallet_service

deposit_service.go（依赖财务RPC）
  ├─ GetMethods(ctx, currency) → 调用 rpc.FinanceClient().GetDepositMethods
  ├─ Create(ctx, req) → 调用 rpc.FinanceClient().CreateDepositOrder
  └─ GetStatus(ctx, orderNo) → 调用 rpc.FinanceClient().GetOrderStatus

withdraw_service.go（依赖最多）
  ├─ GetMethods(ctx, currency) → 调用 rpc.FinanceClient() + 本地充值来源过滤
  ├─ Create(ctx, req) → 调用 ser-kyc + wallet_service.Freeze + rpc.FinanceClient()
  └─ AccountCRUD → 调用 ser-user（KYC姓名校验）
```
