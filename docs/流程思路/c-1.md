## 一、全局流程架构

### 1.1 三大模块的角色定位

在整个钱包系统中，三大模块扮演的角色截然不同：

```
┌─────────────────────────────────────────────────────────────────────────┐
│                          用户视角（C端）                                  │
│                                                                         │
│     "我要充值"    "我要提现"    "我要兑换"    "查看记录"                    │
│         │              │            │             │                      │
│         └──────────────┼────────────┼─────────────┘                      │
│                        ▼                                                 │
│                ┌──────────────┐                                          │
│                │   钱包模块    │  ← 唯一面向C端的资金操作入口               │
│                │   (枢纽层)    │  ← 承上（接用户）启下（调其他模块）          │
│                └──────┬───────┘                                          │
│                       │                                                  │
│         ┌─────────────┼──────────────┐                                   │
│         ▼             ▼              ▼                                   │
│  ┌────────────┐ ┌──────────┐ ┌────────────┐                             │
│  │  财务管理   │ │ 币种配置  │ │  KYC服务   │                              │
│  │ (通道+审核) │ │ (汇率+币) │ │ (认证)     │                              │
│  └────────────┘ └──────────┘ └────────────┘                             │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────┐
│                          管理视角（B端）                                  │
│                                                                         │
│    运营/财务管理员                                                        │
│         │                                                                │
│    ┌────┴─────────────────────────────┐                                  │
│    ▼                                  ▼                                  │
│  ┌────────────┐                ┌────────────┐                            │
│  │  财务管理   │   审核通过后    │  币种配置   │                             │
│  │ (审核+通道  │───RPC调用───►  │ (汇率+币种  │                             │
│  │  +统计)    │               │  管理)       │                             │
│  └─────┬──────┘               └────────────┘                             │
│        │                                                                 │
│        │ 审核通过后RPC调用                                                 │
│        ▼                                                                 │
│  ┌────────────┐                                                          │
│  │  钱包模块   │  ← 被动接受余额变更指令（入账/扣款/冻结/解冻/加款/减款）      │
│  │ (余额执行)  │                                                          │
│  └────────────┘                                                          │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**一句话概括各模块角色**：

| 模块 | 角色 | 一句话 |
|------|------|--------|
| **钱包模块** | C端门面 + 余额管家 | 面向用户承接所有资金操作，面向财务暴露余额变更RPC |
| **币种配置** | 基础设施 | 提供币种和汇率数据，被所有模块依赖，不主动调用任何业务模块 |
| **财务管理** | B端管家 + 通道代理 | 管理通道/支付配置/审核流程，审核通过后调钱包RPC执行余额变更 |

### 1.2 数据流转的总体规律

从需求分析中总结出三条规律：

**规律一：C端操作 → 钱包发起 → 外部配合 → 钱包落地**

所有C端用户操作的起点和终点都在钱包模块。钱包负责创建订单、校验规则、操作余额，外部模块（财务/币种配置/KYC）提供配置数据或执行特定步骤（如通道匹配、回调验签），最终结果（余额变更）回到钱包落地。

**规律二：B端操作 → 财务发起 → 钱包执行**

所有B端管理员的余额修正操作（审核通过/驳回、补单、加款、减款）由财务模块承接前端页面和审核流程，审核通过后通过RPC调用钱包模块完成实际的余额变更。钱包模块不直接暴露B端HTTP接口。

**规律三：币种配置是"被动地基"**

币种配置从不主动调用钱包或财务的任何接口。它只被查询——钱包查汇率/币种列表，财务查平台汇率做统计换算。币种配置的唯一主动动作是定时从三方API拉取汇率。

## 二、钱包模块内部子模块流程

### 2.1 子模块划分思路

根据链路关系分析中的内部共享方法梳理（链路关系/c-1.md 第5节），钱包模块内部应按业务域划分为以下子模块：

```
ser-wallet/
├── handler.go                  ← IDL方法到service的一对一委派
└── internal/
    ├── ser/
    │   ├── wallet_ser.go       ← 余额操作核心（被所有业务线共享）
    │   ├── deposit_ser.go      ← 充值业务线
    │   ├── withdrawal_ser.go   ← 提现业务线
    │   ├── exchange_ser.go     ← 兑换业务线
    │   ├── record_ser.go       ← 记录查询（纯读取）
    │   ├── rate_ser.go         ← 汇率服务（封装RPC调用+缓存）
    │   └── rpc_ser.go          ← RPC对外入口（供财务等外部调用）
    └── rep/
        ├── wallet_rep.go       ← 余额表操作
        ├── deposit_rep.go      ← 充值订单表操作
        ├── withdrawal_rep.go   ← 提现订单表/提现账户表操作
        ├── exchange_rep.go     ← 兑换记录表操作
        ├── transaction_rep.go  ← 流水日志表操作
        └── currency_rep.go     ← 币种配置表操作（如币种配置为子模块）
```

### 2.2 余额核心子模块（wallet_ser）— 所有业务线的根基

**这是整个钱包模块最核心的部分**。充值、提现、兑换、人工修正等所有涉及余额变更的操作，最终都归结为调用 wallet_ser 提供的原子方法。

#### 核心方法及其调用者

```
wallet_ser 提供的原子方法             被哪些场景调用
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
getBalance(userId, currency)       C1币种列表余额、C2余额查询、C7提现校验、
                                   C11兑换校验、C14概览、R1对外余额查询、R7减款校验

addBalance(tx, userId, currency,   R2充值入账、R6人工加款、R8补单入账、
    walletType, amount)            C11兑换(加BSB)

deductBalance(tx, userId,          C11兑换(扣法币)、R7人工减款
    currency, walletType, amount)

freezeBalance(tx, userId,          C7创建提现订单(冻结)
    currency, walletType, amount)

unfreezeBalance(tx, userId,        R4提现驳回解冻
    currency, walletType, amount)

confirmDeduct(tx, userId,          R5提现确认扣款(冻结→消失)
    currency, walletType, amount)

createTransactionLog(tx, ...)      所有余额变更操作(C7,C11,R2-R8)

idempotencyCheck(orderId)          所有RPC对外余额变更方法(R2-R8)
```

#### wallet_ser 内部处理流程

```
任何余额变更请求进入 wallet_ser
    │
    ├── Step 1: 幂等检查（RPC对外方法才需要）
    │       查询 idempotency 表/字段
    │       ├── 已处理过 → 直接返回成功（不重复操作）
    │       └── 未处理过 → 继续
    │
    ├── Step 2: 余额校验（扣减/冻结操作才需要）
    │       SELECT available_balance FROM wallet
    │       WHERE user_id=? AND currency=? AND wallet_type=?
    │       ├── available < amount → 返回余额不足错误
    │       └── available >= amount → 继续
    │
    ├── Step 3: 数据库事务内执行余额变更
    │       BEGIN TRANSACTION
    │       │
    │       │ 根据操作类型执行不同SQL：
    │       │ ├── addBalance:      available += amount
    │       │ ├── deductBalance:   available -= amount (WHERE available >= amount)
    │       │ ├── freezeBalance:   available -= amount, frozen += amount
    │       │ ├── unfreezeBalance: frozen -= amount, available += amount
    │       │ └── confirmDeduct:   frozen -= amount (钱真正"出去了")
    │       │
    │       │ 写入流水日志：
    │       │ INSERT INTO transaction_log
    │       │ (user_id, currency, wallet_type, type, amount, balance_before, balance_after,
    │       │  order_id, remark, created_at)
    │       │
    │       │ 写入幂等记录（如适用）：
    │       │ INSERT INTO idempotency (order_id, processed_at)
    │       │
    │       COMMIT
    │
    ├── Step 4: 异步审计日志
    │       rpc.BLogClient().AddActLog(...)  ← 异步，失败不阻塞
    │
    └── 返回操作结果
```

**关键设计约束**（来源：依赖关系/c-1.md 第8节）：
- 所有余额变更必须在**数据库事务**内完成（余额更新 + 流水日志 + 幂等记录）
- 扣减/冻结使用**乐观锁**（WHERE available >= amount，检查 affected_rows）
- RPC对外的余额变更方法必须**幂等**（通道可能重复回调，财务可能重试）
- 每次余额变更必须**记录流水**（事前余额、事后余额、操作类型、关联订单号）

### 2.3 充值子模块（deposit_ser）流程

充值是C端三大业务线中**跨模块最复杂**的——需要协调钱包、财务、币种配置三个模块。

#### 充值总体流程

```
用户视角                     钱包模块内部处理                           外部模块
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[1] 打开充值页     ─────►  deposit_ser.GetDepositMethods()
                             │
                             ├── rate_ser → RPC → 币种配置.GetExchangeRate
                             │                    获取入款汇率（BSB充值时展示用）
                             │
                             ├── RPC → 财务.GetDepositMethods(currency)
                             │         获取当前币种可用充值方式+快捷金额档位
                             │
                             └── 合并返回：方式列表 + 汇率信息

[2] 选择奖金活动   ─────►  deposit_ser.GetBonusActivities()
    (可选)                   │
                             └── 查询可用活动列表（数据源待评审确认）

[3] 输入金额       ─────►  deposit_ser.CheckDuplicateDeposit()
    点击"去充值"              │
                             ├── 查本地：同用户+同币种+同金额的待支付订单
                             │   ├── 0笔 → 通过
                             │   ├── 1-2笔 → 返回提醒（前端弹窗但允许继续）
                             │   └── >=3笔 → 拦截，返回错误
                             │
                             │  （此步可合并到创建订单接口内部）
                             │
                             ▼
                           deposit_ser.CreateDepositOrder()
                             │
                             ├── 1. 参数校验
                             │   ├── 金额 >= 最低限额
                             │   ├── 币种是否启用
                             │   └── 奖金活动门槛校验（如选了活动）
                             │
                             ├── 2. rate_ser → RPC → 币种配置.GetExchangeRate
                             │   获取入款汇率，计算支付金额（BSB充值时需换算）
                             │
                             ├── 3. 本地创建充值订单
                             │   订单号：C + 16位随机数
                             │   状态：待支付
                             │   保存：汇率快照、奖金活动信息
                             │
                             ├── 4. RPC → 财务.MatchChannelAndCreatePayment ──►  财务模块
                             │   财务执行：                                         │
                             │   ├── 加权轮询选择代收通道                           │
                             │   ├── 用通道密钥向三方发起支付请求                    │
                             │   └── 返回：支付页URL/收款信息/通道订单号             │
                             │                                                    │
                             ├── 5. 更新本地订单（通道订单号+支付信息）                │
                             │                                                    │
                             └── 6. 返回支付页信息给C端                              │

[4] 用户支付       ─────►  (用户在三方页面或USDT页面完成支付)

[5] 支付成功                                                            财务模块
    三方回调        ◄──────────────────────────────────────────────  三方通道回调
                                                                        │
                                                         财务.充值回调处理
                                                         ├── 验签
                                                         ├── 校验金额匹配
                                                         ├── 更新财务记录
                                                         └── RPC → 钱包.CreditDeposit
                                                                        │
                           rpc_ser.CreditDeposit()  ◄───────────────────┘
                             │
                             ├── wallet_ser.idempotencyCheck(orderId)
                             │   已入账 → 直接返回成功
                             │
                             ├── wallet_ser.addBalance(中心钱包, amount)
                             │   + 更新订单状态 → 支付成功
                             │   + 写流水日志
                             │   (同一事务内)
                             │
                             └── 返回入账成功

[6] 查看记录       ─────►  record_ser.GetDepositRecords()
                             │
                             └── 分页查询本地充值订单表
```

#### 充值流程的关键节点

| 节点 | 风险 | 处理策略 |
|------|------|---------|
| 创建订单后匹配通道失败 | 产生"死单"（订单已创建但无支付渠道） | 匹配失败则将订单标记为失败或删除 |
| 汇率在用户操作期间变动 | 用户看到的金额和实际计算的金额不一致 | 创建订单时实时取汇率并保存快照，不依赖展示阶段的缓存值 |
| 通道回调重复 | 重复入账（用户余额多加了） | 幂等处理：按orderId去重，重复调用直接返回成功 |
| 通道回调丢失 | 用户付了钱但没到账（资损） | 需要可靠消息/重试机制，或定时对账补单 |

### 2.4 提现子模块（withdrawal_ser）流程

提现是**依赖链最长、涉及模块最多**的业务线——钱包 + 财务 + KYC + 币种配置四方协作。

#### 提现总体流程

```
用户视角                     钱包模块内部处理                           外部模块
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[1] 点击"提现"     ─────►  withdrawal_ser.GetWithdrawalMethods()
                             │
                             ├── RPC → KYC.GetKycStatus(userId)
                             │   ├── 未认证 → 过滤：只保留USDT提现方式
                             │   └── 已认证 → 不过滤：全渠道可用
                             │
                             ├── RPC → 财务.GetWithdrawalMethods(currency)
                             │         获取提现方式列表+限额规则+手续费率
                             │
                             ├── rate_ser → RPC → 币种配置.GetExchangeRate
                             │         BSB提现时需要出款汇率展示
                             │
                             └── 按KYC状态过滤方式列表 + 附加汇率 → 返回

[2] 选择提现方式   ─────►  withdrawal_ser.GetWithdrawalAccount()
    填写/确认账户             │
                             ├── 查本地：该用户+币种+方式类型的已保存账户
                             │   ├── 有 → 返回账户信息（只读回显）
                             │   └── 无 → 返回空，前端展示填写表单
                             │
                             ▼ （如首次提现）
                           withdrawal_ser.SaveWithdrawalAccount()
                             │
                             ├── RPC → KYC.GetKycInfo(userId)
                             │   获取KYC认证姓名
                             │
                             ├── 姓名一致性校验
                             │   用户提交的"持有人姓名" vs KYC姓名
                             │   规则：忽略大小写 + 忽略多余空格
                             │   不一致 → 拒绝保存
                             │
                             └── 保存到本地 withdrawal_account 表

[3] 输入金额       ─────►  withdrawal_ser.CreateWithdrawalOrder()
    确认提现                  │
                             ├── 1. RPC → KYC.GetKycStatus  (二次确认)
                             │   防止C6到C7之间用户KYC状态变化
                             │
                             ├── 2. 校验提现账户存在且完整
                             │
                             ├── 3. 校验提现限额
                             │   ├── 单笔限额：amount >= 最低 且 <= 最高
                             │   ├── 单日限额：今日已提现总额 + 本次 <= 上限
                             │   └── 超限 → 返回具体提示
                             │
                             ├── 4. 校验余额
                             │   wallet_ser.getBalance → available >= amount
                             │
                             ├── 5. rate_ser → RPC → 币种配置.GetExchangeRate
                             │   BSB提现：用出款汇率计算实际出款金额
                             │
                             ├── 6. 计算手续费和到账金额
                             │   手续费 = amount × 手续费率
                             │   到账金额 = amount - 手续费
                             │
                             ├── 7. 【核心事务】冻结 + 创建订单
                             │   BEGIN TRANSACTION
                             │   ├── wallet_ser.freezeBalance(中心钱包, amount)
                             │   │   available -= amount, frozen += amount
                             │   │   WHERE available >= amount (乐观锁)
                             │   ├── INSERT withdrawal_order
                             │   │   订单号: T + 16位随机数
                             │   │   状态: 待风控审核
                             │   └── wallet_ser.createTransactionLog(冻结)
                             │   COMMIT
                             │
                             └── 返回：订单信息 + "提现进行中"

                    ──── 至此，C端操作结束，进入B端审核流程 ────

                                                                    财务B端
[4] 风控审核                                                    ┌───────────┐
                                                                │ 审核弹窗   │
                                                                │ 通过/驳回  │
                                                                └─────┬─────┘
                                                                      │
                    ┌─────────────────────────────────────────────────┤
                    │                                                 │
              [驳回分支]                                        [通过分支]
                    │                                                 │
                    ▼                                                 ▼
    rpc_ser.UnfreezeBalance()  ◄── RPC ──          (进入财务审核环节)
      │                                                               │
      ├── 幂等检查                                              [财务审核通过]
      ├── wallet_ser.unfreezeBalance                                  │
      │   frozen -= amount                                            ▼
      │   available += amount                          财务发起代付(调三方通道)
      │   (钱退回用户)                                                │
      ├── 更新订单状态 → 驳回                              ┌─────────┤
      └── 写流水日志                                       │         │
                                                     [出款成功]  [出款失败]
                                                           │         │
                                                           ▼         ▼
                    rpc_ser.ConfirmWithdrawal() ◄── RPC    rpc_ser.UnfreezeBalance()
                      │                                      (同上驳回流程)
                      ├── 幂等检查
                      ├── wallet_ser.confirmDeduct
                      │   frozen -= amount
                      │   (available不变，钱真正"出去了")
                      ├── 更新订单状态 → 出款成功
                      └── 写流水日志
```

#### 提现流程的关键约束

| 约束 | 原因 | 后果 |
|------|------|------|
| 冻结 + 创建订单必须**同一事务** | 先创建后冻结 → 订单已存在但余额未冻结的窗口期，并发操作可能超额 | 使用数据库事务保证原子性 |
| KYC二次确认 | 用户从选择提现方式到确认提现期间，KYC状态可能变化 | 在创建订单时再查一次KYC状态 |
| 提现审核是**多级串行** | 风控 → 财务 → 出款，中间任何环节驳回都要解冻 | 每个审核节点都是财务→钱包RPC的调用点 |
| 审核通过/驳回是**互斥分支** | 同一订单不会同时被通过和驳回 | ConfirmWithdrawal 和 UnfreezeBalance 不会对同一订单同时执行 |

### 2.5 兑换子模块（exchange_ser）流程

兑换是三大业务线中**最简洁**的——全程只依赖币种配置的汇率，不涉及财务模块。

#### 兑换总体流程

```
用户视角                     钱包模块内部处理                    外部模块
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[1] 打开兑换页     ─────►  exchange_ser.GetExchangeRateForSwap()
    (从法币充值页             │
     进入，BSB页              ├── rate_ser → RPC → 币种配置.GetExchangeRate
     不显示兑换)              │   获取当前法币→BSB的汇率
                             │
                             ├── 获取赠送比例（数据源待评审确认）
                             │
                             └── 返回：汇率(1 BSB = X 法币) + 赠送比例

[2] 输入法币金额   ─────►  前端基于汇率自行计算预览：
                             兑换获得BSB = 法币金额 ÷ 汇率
                             赠送BSB = 兑换获得BSB × 赠送比例
                             实际到账BSB = 兑换获得 + 赠送

[3] 点击"兑换"    ─────►  exchange_ser.ExecuteExchange()
                             │
                             ├── 1. rate_ser → RPC → 币种配置.GetExchangeRate
                             │   【再次取最新汇率】不用展示时的缓存值
                             │   防止汇率变化导致计算偏差
                             │
                             ├── 2. 校验余额
                             │   wallet_ser.getBalance(当前法币)
                             │   available < amount → "已达当前可兑换上限"
                             │
                             ├── 3. 计算兑换结果
                             │   兑换获得BSB = 法币金额 ÷ 汇率
                             │   赠送BSB = 兑换获得BSB × 赠送比例
                             │   实际到账BSB = 兑换获得 + 赠送
                             │
                             ├── 4. 【核心事务】三步余额变更
                             │   BEGIN TRANSACTION
                             │   │
                             │   ├── wallet_ser.deductBalance
                             │   │   (法币, 中心钱包, 法币金额)
                             │   │   ← 扣减法币余额
                             │   │
                             │   ├── wallet_ser.addBalance
                             │   │   (BSB, 中心钱包, 兑换获得BSB)
                             │   │   ← 兑换所得进中心钱包
                             │   │
                             │   ├── wallet_ser.addBalance
                             │   │   (BSB, 奖励钱包, 赠送BSB)
                             │   │   ← 赠送部分进奖励钱包（需完成流水才能提现）
                             │   │
                             │   ├── INSERT exchange_record
                             │   │   (法币金额, 汇率快照, 兑换获得, 赠送, 实际到账)
                             │   │
                             │   └── wallet_ser.createTransactionLog × 3
                             │       (法币扣除 + BSB中心入账 + BSB奖励入账)
                             │
                             │   COMMIT
                             │
                             └── 返回：兑换成功 + 兑换获得/赠送/实际到账

[4] 兑换成功                 3秒内显示成功提示
                             左上角返回充值中心
                             右上角跳转记录
```

#### 兑换流程的核心特点

1. **完全在钱包模块内闭环**：不需要财务模块参与，没有通道、没有三方支付、没有审核
2. **唯一外部依赖是汇率**：如果币种配置的汇率接口不可用，兑换瘫痪
3. **赠送部分进奖励钱包**：这是需求明确的规则——赠送的BSB需完成流水要求后才能提现
4. **三步余额变更必须同事务**：扣法币 + 加BSB到中心 + 加赠送BSB到奖励，三者原子执行

### 2.6 记录查询子模块（record_ser）流程

记录查询是**最简单**的子模块——纯本地数据库查询，无任何外部依赖。

```
record_ser 处理流程（适用于所有4种记录）

C端请求(分页参数 + 当前币种) → record_ser.GetXxxRecords()
    │
    ├── 分页查询本地表
    │   SELECT * FROM xxx_table
    │   WHERE user_id = ? AND currency = 当前币种
    │   ORDER BY create_time DESC
    │   LIMIT ? OFFSET ?
    │
    └── 返回列表（金额/方式/时间/状态）

C端请求(订单号) → record_ser.GetXxxOrderDetail()
    │
    ├── 单笔查询
    │   SELECT * FROM xxx_table
    │   WHERE order_id = ? AND user_id = ?
    │
    └── 返回详情
```

**核心规则**：记录跟随币种切换——切换币种 = 切换记录数据，不存在跨币种混合展示。

### 2.7 汇率服务子模块（rate_ser）流程

rate_ser 是一个**共享基础服务层**，封装对币种配置RPC的调用，被 deposit、withdrawal、exchange 三条业务线共同依赖。

```
rate_ser 处理流程

调用方(deposit/withdrawal/exchange) → rate_ser.GetExchangeRate(currency)
    │
    ├── 检查本地缓存（Redis或进程内缓存）
    │   ├── 缓存命中且未过期 → 直接返回
    │   └── 缓存未命中或已过期 ↓
    │
    ├── RPC → 币种配置.GetExchangeRate(currency)
    │   返回：平台汇率 + 入款汇率 + 出款汇率
    │
    ├── 写入缓存（设置合理TTL，如30秒-1分钟）
    │
    └── 返回汇率数据
```

**设计考量**：
- 汇率是**高频读取**接口（充值/提现/兑换都需要），适合加缓存层
- 缓存TTL不宜过长——汇率可能因阈值机制更新
- 对于**执行操作**（创建订单、执行兑换），应该实时取汇率而非用缓存值——因为金额计算必须精确

### 2.8 RPC对外子模块（rpc_ser）流程

rpc_ser 是钱包模块面向外部服务（主要是财务模块）的**入口层**，所有方法都委托给 wallet_ser 执行实际的余额变更。

```
rpc_ser 提供的方法与内部委托关系
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

R1: GetUserBalance
    └── wallet_ser.getBalance()

R2: CreditDeposit (充值入账)
    ├── wallet_ser.idempotencyCheck(orderId)
    ├── wallet_ser.addBalance(中心钱包, amount)  ← 事务内
    ├── 更新 deposit_order.status = 成功          ← 事务内
    └── wallet_ser.createTransactionLog           ← 事务内

R3: FreezeBalance (冻结余额)
    └── wallet_ser.freezeBalance()

R4: UnfreezeBalance (解冻退回)
    ├── wallet_ser.idempotencyCheck(orderId)
    ├── wallet_ser.unfreezeBalance()              ← 事务内
    ├── 更新 withdrawal_order.status = 驳回       ← 事务内
    └── wallet_ser.createTransactionLog           ← 事务内

R5: ConfirmWithdrawal (确认提现扣款)
    ├── wallet_ser.idempotencyCheck(orderId)
    ├── wallet_ser.confirmDeduct()                ← 事务内
    ├── 更新 withdrawal_order.status = 出款成功    ← 事务内
    └── wallet_ser.createTransactionLog           ← 事务内

R6: ManualCredit (人工加款)
    ├── wallet_ser.idempotencyCheck(orderId)
    ├── wallet_ser.addBalance(指定钱包类型, amount) ← 事务内
    └── wallet_ser.createTransactionLog            ← 事务内

R7: ManualDebit (人工减款)
    ├── wallet_ser.idempotencyCheck(orderId)
    ├── wallet_ser.getBalance → 校验余额 >= amount
    ├── wallet_ser.deductBalance(指定钱包类型, amount) ← 事务内
    └── wallet_ser.createTransactionLog               ← 事务内

R8: SupplementCredit (补单入账)
    ├── wallet_ser.idempotencyCheck(supplementOrderId)
    ├── wallet_ser.addBalance(中心钱包, amount)       ← 事务内
    ├── wallet_ser.addBalance(奖励钱包, bonusAmount)  ← 事务内（如有赠送）
    └── wallet_ser.createTransactionLog × N           ← 事务内
```

**rpc_ser 的统一处理模式**：每个方法都遵循 `幂等检查 → 事务内余额变更 → 异步审计` 的三段式。

## 三、币种配置模块内部流程

### 3.1 币种配置的功能简洁性

币种配置是三个模块中**最独立、最简洁**的，它的所有功能都在自身内部闭环，不主动调用钱包或财务的任何接口。

```
币种配置模块功能全景

B端接口（gate-back路由）
├── B1: 币种列表查询      → 查本地DB + 计算入款/出款汇率
├── B2: 币种编辑          → 更新本地DB + (图标上传→S3)
├── B3: 基准币种设置      → 更新本地DB（一次性操作）
├── B4: 汇率日志列表      → 查本地DB
└── B5: 实时汇率手动刷新  → HTTP→三方API(3家) → 更新本地DB

RPC对外接口（供钱包/财务调用）
├── CR1: GetExchangeRate   → 查本地DB + 计算
└── CR2: GetCurrencyList   → 查本地DB

定时任务
└── 汇率定时更新           → HTTP→三方API → 阈值判断 → 更新DB + 写日志
```

### 3.2 汇率更新机制——阈值驱动流程

这是币种配置模块中**最核心**的处理流程：

```
汇率定时更新任务触发（cron定时）
    │
    ├── 遍历所有启用的币种（BSB除外，BSB汇率写死10:1）
    │       │
    │       ├── Step 1: 获取实时汇率
    │       │   HTTP → 三方API_1 → rate_1
    │       │   HTTP → 三方API_2 → rate_2
    │       │   HTTP → 三方API_3 → rate_3
    │       │   实时汇率 = (rate_1 + rate_2 + rate_3) / 3
    │       │
    │       ├── Step 2: 计算偏差值
    │       │   当前入款汇率 = 平台汇率 × (1 + 汇率浮动%)
    │       │   当前出款汇率 = 平台汇率 × (1 - 汇率浮动%)
    │       │   入款偏差 = |(实时汇率 - 当前入款汇率) / 实时汇率| × 100%
    │       │   出款偏差 = |(实时汇率 - 当前出款汇率) / 实时汇率| × 100%
    │       │
    │       ├── Step 3: 阈值判断
    │       │   ├── 入款偏差 < 阈值% 且 出款偏差 < 阈值%
    │       │   │   → 不更新，跳过该币种
    │       │   │
    │       │   └── 入款偏差 >= 阈值% 或 出款偏差 >= 阈值%
    │       │       → 触发更新
    │       │
    │       ├── Step 4: 更新平台汇率（如触发）
    │       │   UPDATE currency_config
    │       │   SET platform_rate = 实时汇率,
    │       │       realtime_rate = 实时汇率
    │       │   WHERE code = ?
    │       │   (新的入款/出款汇率随平台汇率自动变化)
    │       │
    │       └── Step 5: 写入汇率日志（如触发）
    │           INSERT INTO rate_change_log
    │           (log_no, update_time, currency_code,
    │            realtime_rate, platform_rate,
    │            deposit_rate, withdrawal_rate)
    │
    └── 异步审计日志（记录本次定时任务执行）
```

**关键细节**：
- 入款和出款的偏差值**分别判断**，任一方达到阈值就更新
- 更新的是**平台汇率**（设为实时汇率值），入款/出款汇率通过 `平台汇率 × (1 ± 浮动%)` 自动派生
- 平台汇率初始为空时，直接取实时汇率值（首次填充）
- BSB汇率写死（10 BSB = 1 USDT），不走定时更新

### 3.3 汇率的四层结构

```
数据源                     存储/计算                      使用场景
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

三方API(3家平均)  ──►  实时汇率(展示用)              B端页面展示，
                       │                            手动刷新更新
                       │ 阈值判断后
                       ▼
                    平台汇率(DB存储)  ──────────►    报表统计换算
                       │
                       ├── × (1 + 浮动%) ──►  入款汇率    充值时金额换算
                       │
                       └── × (1 - 浮动%) ──►  出款汇率    提现时金额换算
```

## 四、跨模块联动流程详解

### 4.1 钱包 → 外部：主动调用的场景

钱包模块主动调用外部共有 **3 个目标模块、6 条调用链**：

#### 4.1.1 钱包 → 币种配置（2条）

```
调用1: 查询币种列表及状态
触发: C1 获取币种列表及余额（用户切换币种时）
链路: ser-wallet → RPC → 币种配置.GetCurrencyList
用途: 获取哪些币种启用、各自展示信息（代码/名称/图标/状态）
频率: 低（用户切换币种时才触发）
失败影响: 币种选择页无法展示 → 可用缓存降级

调用2: 查询汇率
触发: C3充值方式、C4创建充值订单、C6提现方式、C7创建提现订单、C10兑换汇率、C11执行兑换
链路: ser-wallet → rate_ser → RPC → 币种配置.GetExchangeRate
用途: 获取入款/出款/平台汇率，用于页面展示和金额计算
频率: 高（充值/提现/兑换都需要）
失败影响: 充值/提现/兑换全部瘫痪 → 必须考虑缓存+降级
```

#### 4.1.2 钱包 → 财务（3条）

```
调用3: 查询充值方式列表
触发: C3 获取充值方式列表（用户进入充值页）
链路: ser-wallet → RPC → 财务.GetDepositMethods(currency)
用途: 获取当前币种可用的充值方式、快捷金额档位、支付类型
频率: 中（用户每次进入充值页）
失败影响: 充值页无法展示充值方式 → C端充值不可用

调用4: 查询提现方式列表
触发: C6 获取提现方式列表（用户进入提现页）
链路: ser-wallet → RPC → 财务.GetWithdrawalMethods(currency)
用途: 获取当前币种可用的提现方式、限额规则、手续费率
频率: 中（用户每次进入提现页）
失败影响: 提现页无法展示提现方式 → C端提现不可用

调用5: 匹配通道并创建支付
触发: C4 创建充值订单（用户点击"去充值"）
链路: ser-wallet → RPC → 财务.MatchChannelAndCreatePayment(...)
用途: 财务执行通道轮询匹配+向三方发起支付请求+返回支付页信息
频率: 中（每笔充值一次）
失败影响: 充值订单创建失败 → 需要回滚本地订单
特殊性: 这是唯一一条"钱包调财务做写操作"的链路（其他两条都是读取配置）
```

#### 4.1.3 钱包 → KYC（1条）

```
调用6: 查询KYC认证状态及姓名
触发: C6提现方式列表（过滤）、C7创建提现订单（二次确认）、C8保存提现账户（姓名校验）
链路: ser-wallet → RPC → KYC.GetKycStatus(userId)
用途: 判断用户能用哪些提现方式、校验提现账户持有人姓名
频率: 低（仅提现流程触发）
失败影响: 无法判断KYC状态 → 提现流程中断
```

### 4.2 外部 → 钱包：被动接受调用的场景

钱包模块被外部调用共有 **7 条RPC链**，全部由财务模块发起，全部涉及余额变更：

```
┌──────────────────────────────────────────────────────────────────────────┐

│  └─────────────────────┘                                                 │
│                                                                          │
│  ┌─────────────────────┐                                                 │
│  │ 提现审核通过后        │──► R5: ConfirmWithdrawal ──► frozen-N (钱出去)   │
│  └─────────────────────┘                                                 │
│                                                                          │
│  ┌─────────────────────┐                                                 │
│  │ 提现审核驳回后        │──► R4: UnfreezeBalance ──► frozen→available     │
│  └─────────────────────┘                                                 │
│                                                                          │
│  ┌─────────────────────┐                                                 │
│  │ 补单审核通过后        │──► R8: SupplementCredit ──► 中心+奖励 available+N│
│  └─────────────────────┘                                                 │
│                                                                          │
│  ┌─────────────────────┐                                                 │
│  │ 加款审核通过后        │──► R6: ManualCredit ──► 指定钱包 available+N     │
│  └─────────────────────┘                                                 │
│                                                                          │
│  ┌─────────────────────┐                                                 │
│  │ 减款审核通过后        │──► R7: ManualDebit ──► 指定钱包 available-N      │
│  └─────────────────────┘                                                 │
│                                                                          │
│  ┌─────────────────────┐                                                 │
│  │ 审核/修正前查余额     │──► R1: GetUserBalance ──► 纯读取，无副作用       │
│  └─────────────────────┘                                                 │
│                                                                          │
│  共同特点：                                                               │
│  ├── 除R1外，全部涉及余额变更（写操作）                                     │
│  ├── 全部需要幂等处理（财务可能重试）                                       │
│  ├── 全部在数据库事务内执行                                                │
│  └── 全部记录流水日志                                                     │
└──────────────────────────────────────────────────────────────────────────┘
```

### 4.3 模块间不对称依赖的处理策略

依赖关系/c-1.md 第9.5节指出的不对称性：

| 方向 | 数量 | 性质 | 可靠性要求 |
|------|------|------|-----------|
| 钱包→财务 | 3条 | 读取配置 | 中（可缓存降级） |
| 财务→钱包 | 7条 | 写入余额 | 极高（不能失败/重复/丢失） |
| 钱包→币种配置 | 2条 | 读取汇率 | 高（可缓存短暂降级） |

**处理策略**：

```
钱包→外部（主动调用）的容错策略：
├── 汇率查询：rate_ser 加 Redis 缓存层，缓存未命中才调RPC
│   缓存TTL建议30s-1min，平衡实时性和可用性
│   执行操作（创建订单/兑换）时强制取最新，不用缓存
│
├── 充值/提现方式查询：可考虑本地缓存（方式配置不常变）
│   但首版建议直接RPC调用，不过度优化
│
└── 通道匹配：无法缓存（每次通道状态不同），失败则订单创建失败
    需要在创建订单前做好补偿（失败→回滚本地订单）

外部→钱包（被调用）的可靠性保障：
├── 幂等设计：所有余额变更RPC方法基于orderId去重
│   重复调用返回成功而不重复操作
│
├── 事务保障：余额变更+流水日志+订单状态更新在同一DB事务内
│   要么全部成功，要么全部回滚
│
└── 接口高可用：钱包的RPC对外接口是全系统被依赖最多的
    需要保证服务稳定性（健康检查、超时控制、限流保护）
```

### 5.1 一笔充值的完整生命周期

```

T0  【前置配置阶段（运营操作，一次性）】

    │
    ├── C端 → gate-font → ser-wallet.GetCurrencyListWithBalance
    │   钱包 → 币种配置(RPC): 获取启用币种列表
    │   钱包 → 本地DB: 查询各币种余额
    │   → 返回币种选择页数据

    └── C端 → gate-font → ser-wallet.GetDepositMethods
        钱包 → 财务(RPC): 获取充值方式
        钱包 → 币种配置(RPC): 获取汇率
        → 返回充值页数据（方式列表 + 汇率）

T2  【用户选择金额和方式，点击充值】
    │
    └── C端 → gate-font → ser-wallet.CreateDepositOrder
        钱包: 参数校验 + 重复充值检查
        钱包 → 币种配置(RPC): 取汇率快照
        钱包: 创建本地充值订单(C+16位, 状态:待支付)
        钱包 → 财务(RPC): 匹配通道+创建支付
        钱包: 更新订单(通道信息)
        → 返回支付页信息

T3  【用户完成支付】
    │
    └── 用户在三方页面/USDT页面完成付款操作
        (此阶段后端无感知，等待回调)

T4  【通道回调→入账】
    │
    ├── 三方通道 → 财务.充值回调Webhook
    │   财务: 验签 + 校验金额 + 更新财务记录
    │
    └── 财务 → 钱包(RPC): CreditDeposit
        钱包: 幂等检查(orderId)
        钱包: [事务] 中心钱包 available += 充值金额
                      订单状态 → 支付成功
                      写入流水日志
        → 入账完成

T5  【用户查看结果】
    │
    └── C端 → gate-font → ser-wallet.GetDepositRecords
        钱包: 查本地充值订单表
        → 显示充值成功

    数据流: C端→钱包→财务(通道)→三方→财务(回调)→钱包(入账)
    形态: 环形链路（去程走财务，回程也走财务，最终落地钱包）
```

```
时间线
──────────────────────────────────────────────────────────────────────────

T0  【前置条件】
    │
    ├── 用户已有钱包余额（充值/兑换过）
    ├── 用户已完成KYC认证（或选择仅USDT提现）
    ├── B端-财务：代付通道已配置
    └── B端-财务：提现方式已配置

T1  【用户点击提现→选择方式】
    │
    ├── C端 → gate-font → ser-wallet.GetWithdrawalMethods
    │   钱包 → KYC(RPC): 查认证状态 → 决定过滤规则
    │   钱包 → 财务(RPC): 查提现方式列表
    │   钱包 → 币种配置(RPC): 查汇率(BSB时)
    │   → 返回可用提现方式
    │
    └── C端 → gate-font → ser-wallet.GetWithdrawalAccount
        └── 返回已保存账户（首次为空→填写）

T2  【用户填写账户(首次) + 输入金额 + 确认】
    │
    ├── (如首次) ser-wallet.SaveWithdrawalAccount
    │   钱包 → KYC(RPC): 获取认证姓名做校验
    │   钱包: 保存到本地
    │
    └── ser-wallet.CreateWithdrawalOrder
        钱包 → KYC(RPC): 二次确认认证状态
        钱包: 校验限额(单笔+单日) + 校验余额
        钱包 → 币种配置(RPC): 取汇率(BSB时)
        钱包: 计算手续费和到账金额
        钱包: [事务] 冻结余额 + 创建订单(T+16位, 状态:待风控审核)
                      写入流水日志
        → 返回"提现进行中"

    ── C端操作结束 ── 进入B端审核 ──

T3  【风控审核】
    │
    ├── B端管理员 → gate-back → 财务.提现记录列表
    │   财务: 展示订单信息
    │   财务 → 钱包(RPC): GetUserBalance 查用户余额(辅助判断)
    │
    ├── 管理员锁定订单 → 审核
    │   ├── [驳回] → 财务 → 钱包(RPC): UnfreezeBalance
    │   │   钱包: [事务] frozen→available + 订单状态→驳回 + 流水日志
    │   │   → 钱退回用户 ──────── 流程结束
    │   │
    │   └── [通过] → 进入财务审核

T4  【财务审核】
    │
    ├── [驳回] → 同T3驳回流程
    │
    └── [通过] → 财务发起代付(调三方出款通道)

T5  【出款结果】
    │
    ├── [出款成功] → 财务 → 钱包(RPC): ConfirmWithdrawal
    │   钱包: [事务] frozen -= amount (钱真正出去)
    │                订单状态→出款成功 + 流水日志
    │   → 提现完成 ──────── 流程结束
    │
    └── [出款失败] → 财务 → 钱包(RPC): UnfreezeBalance
        钱包: [事务] frozen→available + 订单状态→出款失败 + 流水日志
        → 钱退回用户 ──────── 流程结束

    数据流: C端→钱包(冻结+建单)→财务(审核)→钱包(扣款/解冻)
    形态: 三次钱包↔财务交互（是依赖链最长的业务流程）
```

### 5.3 一笔兑换的完整生命周期

```
时间线
──────────────────────────────────────────────────────────────────────────

T0  【前置条件】
    │
    ├── 用户当前选中法币币种(VND/IDR/THB)，且有法币余额

T4  【兑换成功提示(3秒内)】
    │
    └── 用户可查看兑换记录

    数据流: C端→钱包→币种配置(汇率)→钱包(事务完成)
    形态: 最简洁链路（全程不涉及财务模块）
```

## 七、实现阶段的流程推进思路

### 7.1 阶段划分原则

基于依赖关系分析的 Layer 0-6 分层模型，实现阶段按**地基→骨架→业务→收尾**四步走：

```
Phase 1: 地基层 ── IDL + 表结构 + 基础设施
    │
Phase 2: 骨架层 ── 币种配置(全量) + 钱包余额核心 + RPC对外接口
    │
Phase 3: 业务层 ── 充值 / 提现 / 兑换 三线并行
    │
Phase 4: 收尾层 ── 记录查询 + 汇率定时任务 + 对接联调
```

## 八、关键流程决策点

以下是实现过程中需要在评审中明确的**流程层面**决策：

### 8.1 充值回调归属

```
方案A: 三方通道 → 财务(验签+更新) → 钱包(RPC入账)
方案B: 三方通道 → 钱包(验签+更新+入账)

推荐A的理由：
├── 通道密钥在财务模块管理，验签逻辑应在财务
├── 财务需要同步更新自己的充值记录
└── 符合"财务管通道+钱包管余额"的职责分工
```

### 8.2 币种配置部署形态

```
方案A: 独立服务(ser-currency)，独立IDL，独立命名空间
方案B: 作为ser-wallet的内部子模块，共享IDL和数据库

两者对流程的影响：
├── 方案A：钱包调币种配置走真正的RPC（跨进程通信）
│          IDL分两套文件
│          优点：解耦，可独立扩展
│          缺点：多一个服务维护
│
└── 方案B：钱包调币种配置走本地方法调用（同进程）
           IDL合并为一套
           优点：简单，减少RPC开销
           缺点：耦合度高，无法独立扩展
```

### 8.3 消费扣款的优先级处理

```
需求规则：消费时先扣中心钱包，不够再扣奖励钱包

实现位置：R9 ConsumeDebit（待投注模块同步开发）

流程复杂点：
├── 如果中心余额够 → 只扣中心
├── 如果中心不够 → 扣全部中心 + 扣差额从奖励
└── 返奖时 → 按各钱包实际出资比例分配

这个流程首版可能不做（投注模块未开始），但IDL应预留方法定义
```

### 8.4 奖金活动数据源

```
待评审确认：充值时可选的奖金活动，数据从哪来？

方案A: 钱包自管（自建活动配置表+CRUD接口）
方案B: 独立活动模块提供（钱包通过RPC获取）
方案C: 前端直接调用活动模块（钱包不经手）

影响流程：
├── 方案A → deposit_ser 需要查本地活动表
├── 方案B → deposit_ser 需要新增一个RPC调用
└── 方案C → deposit_ser 不涉及活动数据
```

## 九、流程安全红线

### 9.1 事务一致性红线

| 操作 | 必须在同一事务内完成 | 违反后果 |
|------|-------------------|---------|
| 提现冻结 + 创建订单 | 余额冻结 + 订单INSERT | 冻结成功但订单未创建 → 资金锁死 |
| 兑换三步变更 | 扣法币 + 加BSB中心 + 加BSB奖励 | 扣了法币没加BSB → 资损 |
| 余额变更 + 流水日志 | UPDATE wallet + INSERT log | 余额变了但无流水 → 对账无法追溯 |

### 9.2 幂等性红线

| RPC方法 | 去重依据 | 违反后果 |
|---------|---------|---------|
| R2 CreditDeposit | orderId | 重复入账 → 用户余额多了 |
| R4 UnfreezeBalance | orderId | 重复解冻 → 余额异常 |
| R5 ConfirmWithdrawal | orderId | 重复扣款 → 用户余额少了 |
| R6 ManualCredit | orderId | 重复加款 → 用户余额多了 |
| R7 ManualDebit | orderId | 重复减款 → 用户余额少了 |
| R8 SupplementCredit | supplementOrderId | 重复补单入账 → 余额多了 |

### 9.3 数据一致性红线

```
核心不变量：用户的可用余额 + 冻结余额 = 该钱包的真实余额
          即 available_balance + frozen_balance = total_balance

任何操作都不能打破这个等式：
├── 冻结: available -= N, frozen += N → 总额不变 ✓
├── 解冻: frozen -= N, available += N → 总额不变 ✓
├── 确认扣款: frozen -= N → 总额减少(钱真出去了) ✓
├── 入账: available += N → 总额增加(钱进来了) ✓
└── 异常: available 和 frozen 分别变动但不配对 → 等式被打破 ✗
```

## 十、总结

### 10.1 核心流程思维

1. **余额是核心中的核心**：wallet_ser 是所有业务线的基石，其设计质量决定整个系统的可靠性
2. **事务保原子性**：任何涉及多步余额变更的操作，必须在同一数据库事务内完成
3. **幂等保安全性**：所有RPC对外余额变更方法，必须基于唯一标识去重
4. **分层保可维护性**：业务层(deposit/withdrawal/exchange) → 核心层(wallet) → 数据层(rep)，上层依赖下层，同层互相独立

### 10.2 三大业务线的复杂度排序

```
兑换(最简单)  <<<  充值(中等)  <<<  提现(最复杂)

兑换: 1个外部依赖(汇率) + 1个事务
充值: 2个外部依赖(财务+汇率) + 1个事务 + 异步回调
提现: 3个外部依赖(财务+汇率+KYC) + 1个事务 + 多级审核 + 互斥分支

建议先做兑换（验证余额核心），再做充值，最后做提现
```

### 10.3 与其他模块的协作思路

```
与财务模块的协作：
├── 优先对齐RPC接口契约（钱包的R1-R8 + 财务的FR1-FR3）
├── 各自先用mock数据独立开发
├── 接口稳定后进行联调
└── 重点联调：充值回调入账链路 + 提现审核扣款/解冻链路

与币种配置的协作：
├── 如果是同一个服务（方案B），直接本地调用，无需联调
└── 如果是独立服务（方案A），需要先稳定CR1/CR2两个RPC接口

与KYC服务的协作：
├── 确认KYC服务已有 GetKycStatus / GetKycInfo 接口
└── 钱包侧只需调用，不需要KYC做任何修改
```
