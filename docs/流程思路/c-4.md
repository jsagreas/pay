> 来源：136张需求/原型解析 + 接口评估 + 依赖关系 + 链路关系 + 工程认知
> 范围：我负责的钱包(C端) + 币种配置(B端)，以及与财务管理(别人负责)的联动
> 目的：把"要做什么"和"怎么串起来"说清楚，作为后续编码的思路蓝图

## 一、全局流程总览：系统是怎么运转的

### 1.1 三个模块的角色定位

```
币种配置 = 数据基座（提供币种、汇率等基础数据，所有模块都要读它的数据）
钱包     = 余额中枢（管理用户的钱，所有资金变动最终落到钱包的余额上）
财务管理 = 通道与审核枢纽（管理充值/提现的支付通道和审核流程，不直接管余额）
```

一句话概括运转方式：**币种配置提供"尺子"（汇率+币种规则），财务管理负责"搬运"（通道进出款+审核），钱包负责"记账"（余额增减+流水）。**

### 1.2 资金流动的三条主干链路

整个系统的资金只有三种流动方向：

```
外部 → 用户钱包     （充值、奖励、补单、人工加款）
用户钱包 → 外部     （提现、人工减款）
用户钱包内部流转     （法币兑换BSB、子钱包间调拨）
```

不管业务场景多复杂，所有操作最终都归结为这三个方向。

### 1.3 模块协作的整体时序

从系统上线到日常运营，按时间线来看协作关系：

```
第一阶段：系统初始化（运营操作，一次性）
  ① 运营在B端设置基准币种（USDT）→ 币种配置
  ② 运营在B端创建各币种（BSB/VND/IDR/THB）→ 币种配置
  ③ 汇率定时任务开始运行 → 币种配置
  ④ 运营在财务B端配置充值/提现通道 → 财务管理
  ⑤ 运营在财务B端配置各币种的充值方式和提现方式 → 财务管理

第二阶段：用户日常使用（C端操作，高频）
  → 充值 / 兑换 / 提现 / 查记录
  → 钱包为主，财务为辅，币种配置提供数据

第三阶段：运营日常管理（B端操作）
  → 人工加款 / 人工减款 / 充值补单 → 财务创建+审核，钱包执行余额变更
  → 汇率自动更新 + 查看日志 → 币种配置
  → 充值/提现记录查看+导出 → 财务管理
```

## 二、币种配置模块：流程思路

### 2.1 币种配置在整个系统中的位置

币种配置是**纯粹的数据提供者**——它不参与任何资金操作，但所有涉及"钱"的操作都必须先读取它的数据。

```
币种配置的数据消费方：
  ├→ 钱包（内部函数调用，同服务）
  │    - 充值：读取入款汇率
  │    - 兑换：读取平台汇率
  │    - 提现：读取出款汇率
  │    - 余额展示：读取币种展示规则（千分位、小数位、货币符号）
  │    - 币种切换：读取已启用币种列表
  │
  └→ 财务管理（RPC调用，跨服务）
       - 所有页面的币种下拉筛选：读取币种列表
       - 金额换算列/统计报表：读取汇率+基准币种
```

**关键结论**：币种配置和钱包在同一个 ser-wallet 服务内，币种配置的数据被钱包通过**内部函数调用**消费，不走 RPC，零网络开销。但被财务管理消费时走 RPC。

### 2.2 基准币种设置流程

```
触发：运营在B端（gate-back → /admin/api/wallet/currency/setBase）操作
频率：系统生命周期内仅一次
前置条件：无
后置影响：后续所有币种创建、汇率换算、对账统计都基于此

运营操作
  │
  ▼
SetBaseCurrency
  │
  ├→ 校验：是否已设置过基准币种
  │    已设置 → 拒绝（不可更改）
  │    未设置 → 继续
  │
  ├→ 写入基准币种记录（currency_code="USDT"，is_base=true）
  │
  ├→ 记录操作日志 → ser-blog.AddActLog（oneway）
  │
  └→ 返回成功

影响范围：
  → 后续 CreateCurrency 校验基准是否已设置
  → 后续 SyncRealtimeRate 以基准币种为换算锚点
  → 财务统计报表的基准换算依赖此配置
```

> 依据：币种配置需求文档 B3——基准币种推荐USDT，配置后不可更改

### 2.3 币种创建/管理流程

```
触发：运营在B端操作
频率：上线初期频繁，后期偶尔

运营创建币种
  │
  ▼
CreateCurrency
  │
  ├→ 校验：基准币种是否已设置（未设置 → 拒绝）
  ├→ 校验：币种代码是否重复
  ├→ 写入币种记录（18个字段：代码/名称/类型/国家/图标/符号/千分位规则/小数位/汇率浮动%/偏差阈值%/排序/状态等）
  ├→ 操作日志 → ser-blog
  └→ 返回成功

运营更新币种
  │
  ▼
UpdateCurrency
  │
  ├→ 校验：币种是否存在
  ├→ 更新可修改字段（汇率浮动%、偏差阈值%、排序值、名称等）
  ├→ 操作日志 → ser-blog
  └→ 返回成功

运营启用/禁用币种
  │
  ▼
ToggleCurrencyStatus
  │
  ├→ 翻转状态（启用↔禁用）
  ├→ 操作日志 → ser-blog
  └→ 返回成功

  禁用影响：
    → C端币种切换列表不再显示该币种
    → 已有该币种余额的用户仍可查看但无法新增充值/兑换
    → 财务管理筛选下拉中仍显示（历史记录需要）
```

> 依据：币种配置需求文档 B3（18个配置字段）+ 产品原型 C1/C2

### 2.4 汇率自动同步流程

```
触发：定时任务（建议每5-10分钟一次）
频率：高频自动执行

SyncRealtimeRate 定时触发
  │
  ▼
  对每个已启用币种执行：
  │
  ├→ 1. 并发请求3家三方汇率API
  │      API-A → 返回汇率 rate_a
  │      API-B → 返回汇率 rate_b
  │      API-C → 返回汇率 rate_c
  │
  ├→ 2. 计算实时汇率 = (rate_a + rate_b + rate_c) / 3
  │
  ├→ 3. 计算偏差值 = |实时汇率 - 当前平台汇率| / 实时汇率 × 100%
  │
  ├→ 4. 判断是否触发更新
  │      偏差 < 阈值 → 跳过，不更新
  │      偏差 ≥ 阈值 → 执行更新：
  │          平台汇率 = 实时汇率
  │          入款汇率 = 实时汇率 × (1 + 汇率浮动%)
  │          出款汇率 = 实时汇率 × (1 - 汇率浮动%)
  │
  ├→ 5. 如果发生更新，写入汇率变更日志
  │      日志字段：时间、币种、实时/平台/入款/出款汇率
  │
  └→ 循环处理下一个币种

  后续影响：
    → C端充值看到的汇率换算会用最新入款汇率
    → C端兑换看到的汇率会用最新平台汇率
    → C端提现看到的汇率换算会用最新出款汇率
    → B端汇率日志页可以查到这条更新记录
```

**思路要点**：
- 三方API可能超时或失败，需要有降级策略（如某家失败则取剩余2家平均值，全部失败则本轮跳过不更新）
- 精度规则：法币保留2位小数，加密货币保留8位小数
- 不是每次轮询都更新，只有偏差超阈值才更新，避免频繁写入

> 依据：币种配置需求文档 B4——实时汇率3家取均值，偏差≥阈值触发更新

## 三、钱包模块 C端：四大功能的流程思路

### 3.1 充值流程：用户把钱充进来

#### 3.1.1 完整流程总览

```
用户打开钱包
  │
  ▼
展示余额 + 币种切换
  │  W-1 GetWalletBalance（余额）
  │  W-2 GetAvailableCurrencies（币种列表）
  │
  ▼
用户点击"充值"
  │
  ▼
加载充值页面
  │  W-3 GetDepositMethods → 调用财务 EXT-1 获取充值方式列表
  │  根据当前币种展示对应充值方式和金额档位
  │
  ▼
（可选）用户点击奖金选择
  │  W-6 GetBonusActivities → 获取当前可用奖金活动列表
  │  用户选择某个活动（或选"无奖金"）
  │
  ▼
用户选择充值方式 + 输入/选择金额 + 确认提交
  │
  ▼
W-4 CreateDepositOrder 处理流程
  │
  ├→ 重复转账拦截检查
  │    查同用户+同币种+同金额+状态=待支付/进行中的订单数
  │    0笔 → 正常
  │    1-2笔 → 返回警告标记（客户端弹窗提示，用户可继续）
  │    ≥3笔 → 拦截，返回错误（USDT链上充值不受此限制）
  │
  ├→ 奖金门槛校验（如果选了活动）
  │    充值金额 < 活动最低门槛 → 拒绝
  │
  ├→ 生成订单号 C+16位
  │
  ├→ 创建钱包侧充值记录（状态=待支付，含奖金活动关联）
  │
  ├→ 通知财务分配通道
  │    财务侧：根据轮询规则匹配代收通道 → 创建财务侧订单
  │
  ├→ 操作日志 → ser-blog
  │
  └→ 返回订单信息
       USDT → 调用 W-5 获取收款地址 + 二维码
       银行/电子钱包 → 返回三方承载页URL（用户跳转三方支付）

  ▼
用户完成支付
  │  USDT：链上转账到平台地址
  │  银行/电子钱包：在三方页面完成支付
  │
  ▼
客户端轮询等待
  │  W-7 GetDepositOrderStatus（定时查状态）
  │  直到状态变为 已完成/已失败/已关闭
  │
  ▼
三方支付成功回调到达系统
  │  接收方：财务管理（回调地址在财务侧配置）
  │  财务验签 → 更新财务侧订单状态
  │
  ▼
财务调用钱包上账
  │  W-R2 CreditDeposit（RPC）
  │
  ├→ 幂等校验（同一订单只能上账一次）
  │    已上账 → 直接返回成功（防重复回调）
  │
  ├→ 充值金额 → 中心钱包余额增加
  │
  ├→ 如果有额外奖金
  │    奖金金额 = 充值金额 × 奖金比例
  │    奖金 → 奖励钱包余额增加
  │    记录稽核要求（X倍流水才能提现）
  │
  ├→ 写入交易流水（充值类型，状态=已完成）
  │
  ├→ 更新订单状态 → 已完成
  │
  ├→ 操作日志 → ser-blog
  │
  └→ 返回成功

  ▼
用户看到余额更新
  │  W-1 余额已变化
  │  W-14 可以查到这笔充值记录
```

#### 3.1.2 充值流程的核心思路

**三个关键节点**：

1. **创建订单时**（W-4）：钱包创建记录，但**不改余额**。此时钱还没到。
2. **财务回调上账时**（W-R2）：钱到了，**此时才改余额**。充值金额→中心钱包，奖金→奖励钱包。
3. **余额变更和流水写入在同一事务中**：保证对账一致性。

**两个关键防线**：

1. **重复转账拦截**（W-4 内部）：防用户手滑，3笔同条件订单就拦截
2. **幂等性**（W-R2 内部）：防三方重复回调，同订单只入账一次

**分币种差异**：

| 币种 | 推荐充值方式 | 承载方 | 回调路径 |
|------|------------|--------|---------|
| BSB | USDT | 平台自己（W-5展示地址） | 链上确认（需链上监听或人工确认） |
| VND | 银行转账 | 三方页面 | 三方→财务→钱包W-R2 |
| IDR | 电子钱包扫码 | 三方页面 | 三方→财务→钱包W-R2 |
| THB | 电子钱包扫码 | 三方页面 | 三方→财务→钱包W-R2 |

> 依据：需求文档 A3 充值模块 + 产品原型 D3/D4/D5/D6

### 3.2 兑换流程：法币换成平台币BSB

#### 3.2.1 完整流程

```
用户在法币币种下（VND/IDR/THB）
  │
  ▼
点击"兑换BSB"入口
  │  注意：只有法币下才有此入口，BSB下没有（已经是BSB了）
  │
  ▼
W-1 展示当前法币余额
  │
  ▼
用户输入兑换金额（法币金额）
  │
  ▼
W-8 GetExchangePreview 实时预览
  │
  ├→ 读取平台汇率（内部函数调用币种配置）
  │    获取：1 BSB = X 法币
  │
  ├→ 计算：
  │    兑换获得BSB = 法币金额 / 汇率
  │    赠送BSB = 兑换获得 × 赠送比例（如10%）
  │    实际到账 = 兑换获得 + 赠送
  │
  ├→ 生成汇率快照ID（绑定当前汇率，防预览后汇率变动）
  │
  └→ 返回：汇率 + 兑换获得 + 赠送 + 实际到账 + 快照ID

  ▼
用户确认兑换
  │
  ▼
W-9 ExecuteExchange
  │
  ├→ 校验汇率快照是否过期
  │    过期（如超过5分钟）→ 拒绝，要求重新预览
  │
  ├→ 校验法币余额 ≥ 兑换金额
  │    不足 → 拒绝
  │
  ├→ 事务操作（原子性，全成功或全失败）：
  │    ① 扣减 法币中心钱包余额（减法币金额）
  │    ② 增加 BSB中心钱包余额（加兑换获得部分）
  │    ③ 增加 BSB奖励钱包余额（加赠送部分）
  │    ④ 记录赠送部分的稽核要求（X倍流水）
  │    ⑤ 写入交易流水（兑换类型）
  │
  ├→ 操作日志 → ser-blog
  │
  └→ 返回：兑换成功 + 新余额

  ▼
用户看到两个币种的余额都变了
  │  法币余额减少，BSB余额增加
  │  W-14 可以查到兑换记录
```

#### 3.2.2 兑换流程的核心思路

**最大特点：零跨模块依赖**。兑换是钱包内部闭环操作，不调用任何外部 RPC。

**关键设计点**：

1. **汇率快照机制**：W-8 预览时锁定汇率生成快照ID，W-9 执行时校验快照是否过期。防止"用户看到的汇率"和"实际执行的汇率"不一致。
2. **跨币种事务**：法币扣款和BSB加款必须在同一事务中。先扣后加，扣失败不加，加失败回滚扣。
3. **赠送部分进奖励钱包**：不是进中心钱包，且有流水要求。防止用户兑换后直接提现套利。
4. **单向兑换**：只能法币→BSB，不能反过来。BSB提现时按出款汇率换算成法币。

> 依据：需求文档 A4 兑换模块 + 产品原型 D7

### 3.3 提现流程：用户把钱拿出去

#### 3.3.1 完整流程

```
用户点击"提现"
  │
  ▼
W-10 GetWithdrawalInfo 聚合信息
  │
  ├→ RPC调用 ser-kyc.KycQuery
  │    返回：KYC状态（已认证/未认证）+ 认证姓名
  │
  ├→ RPC调用 财务.GetAvailableWithdrawalMethods
  │    返回：当前币种可用提现方式 + 手续费 + 限额规则
  │
  ├→ 根据KYC状态过滤提现方式
  │    未认证 → 只保留USDT提现
  │    已认证 → 保留全部方式
  │
  ├→ 读取出款汇率（内部，用于BSB换算展示）
  │
  └→ 返回：KYC状态 + 可用方式 + 限额 + 汇率

  ▼
【如果未认证KYC】
  │  客户端弹窗：
  │  A. "去认证" → 跳转KYC流程（ser-kyc负责，非我模块）→ 认证完成回来重新调 W-10
  │  B. "继续提现" → 只能选USDT提现方式

  ▼
用户选择提现方式
  │
  ▼
W-13 GetWithdrawalAccounts 查询已保存的提现账户
  │
  ├→ 有已保存账户 → 自动填充到表单
  │
  └→ 无已保存账户 → 引导用户填写

  ▼
【首次提现或新增账户】
  │
  ▼
W-12 SaveWithdrawalAccount 保存账户信息
  │
  ├→ 账户持有人 = KYC认证姓名（自动填充，不可修改）
  ├→ 写入账户信息（银行名/卡号 或 电子钱包类型/钱包号）
  └→ 返回账户ID

  ▼
用户输入提现金额
  │
  ▼
W-11 CreateWithdrawalOrder 创建提现订单
  │
  ├→ 提现规则校验（资金来源决定提现方式）
  │    法币充的钱 → 只能法币提
  │    USDT充的钱 → 只能USDT提
  │    法币+USDT混合充的 → 法币提
  │    法币→BSB→提 → 还是按法币提
  │    核心：钱从哪来就往哪去
  │
  ├→ 限额校验
  │    金额 < 单笔最低 → 拒绝（弹窗提示最低限额）
  │    金额 > 单笔最高 → 拒绝（弹窗提示最高限额）
  │    今日已提现 + 本笔 > 每日限额 → 拒绝
  │
  ├→ 余额校验
  │    可用余额 < 提现金额 → 拒绝
  │
  ├→ 事务操作（原子性）：
  │    ① 生成订单号 T+16位
  │    ② 冻结对应金额（可用余额减少 → 冻结余额增加）
  │    ③ 创建提现订单记录
  │    ④ 写入交易流水（提现类型，状态=处理中）
  │
  ├→ 通知财务管理（提现订单进入审核队列）
  │
  ├→ 操作日志 → ser-blog
  │
  └→ 返回：订单号 + 预计到账时间

  ▼
财务管理接管三级审核
  │
  ├→ 第一级：风控审核
  │    通过 → 进入第二级
  │    拒绝 → 财务调用 W-R4 解冻返还
  │
  ├→ 第二级：财务审核
  │    通过 → 进入第三级
  │    拒绝 → 财务调用 W-R4 解冻返还
  │
  └→ 第三级：出款审核 → 分配出款通道 → 通道出款
       │
       ├→ 出款成功 → 财务调用 W-R5 ConfirmWithdrawalDeduct
       │    ├→ 幂等校验
       │    ├→ 冻结余额减少（最终扣除，资金永久移出）
       │    ├→ 更新流水状态 → 已完成
       │    └→ 操作日志 → ser-blog
       │
       └→ 出款失败 → 财务调用 W-R4 UnfreezeForWithdrawal
            ├→ 幂等校验
            ├→ 冻结余额减少 → 可用余额恢复（资金回到用户手中）
            ├→ 更新流水状态 → 已失败
            └→ 操作日志 → ser-blog

  ▼
用户查看结果
  │  W-14 提现记录Tab
  │  W-15 详情（含失败原因 + "换方式"按钮）
```

#### 3.3.2 提现流程的核心思路

**最复杂的流程，因为链路最长、参与模块最多、涉及合规最严**。

**五个关键节点**：

1. **W-10 信息聚合**（进入提现页）：同时调 KYC + 财务两个外部服务，是钱包中外部依赖最重的接口
2. **W-11 创建+冻结**（提交提现）：冻结和订单创建**必须在同一事务中**，先冻结后创建。冻结失败不创建订单。
3. **等待审核**（财务内部）：这段时间钱包不参与，资金处于冻结状态
4. **W-R4 解冻**（审核拒绝/出款失败）：冻结金额恢复为可用，用户可以重新操作
5. **W-R5 扣款**（出款成功）：冻结金额永久扣除，提现完成

**W-R4 和 W-R5 互斥**：同一笔提现订单只能走其中一条路。W-R4 执行后 W-R5 不能再执行，反之亦然。通过订单状态机保证。

**提现资金状态流转**：

```
可用余额 ──(W-11冻结)──→ 冻结余额 ──(W-R5扣款)──→ 移出系统（提现成功）
                              │
                              └──(W-R4解冻)──→ 可用余额（提现失败/拒绝）
```

> 依据：需求文档 A5 提现模块 + 需求文档 E4 提现审核流程

## 四、钱包模块 RPC供他：被财务调用的流程思路

### 4.1 总体思路：事件驱动模式

财务管理调用钱包的 8 个 RPC 接口，全部是**事件驱动**——财务侧发生了某件事，然后通知钱包做对应操作。

```
财务事件                         钱包RPC                    钱包执行
──────                          ──────                     ──────
充值成功回调        →  W-R2 CreditDeposit          →  中心钱包+余额，奖励钱包+奖金
创建提现工单        →  W-R3 FreezeForWithdrawal     →  可用→冻结
审核拒绝/出款失败    →  W-R4 UnfreezeForWithdrawal   →  冻结→可用（返还）
出款成功            →  W-R5 ConfirmWithdrawalDeduct  →  冻结→移除（扣款）
人工加款审核通过     →  W-R6 ManualCredit            →  指定子钱包+金额
人工减款审核通过     →  W-R7 ManualDebit             →  指定子钱包-金额
补单审核通过        →  W-R8 CreditCorrection        →  中心钱包+补单额，奖励钱包+赠送
查用户余额          →  W-R1 GetUserWalletBalance     →  返回各子钱包余额（只读）
```

### 4.2 所有 RPC 写操作的通用处理模式

```
财务 → RPC → 钱包
  │
  ├→ 1. 幂等校验（用订单号做唯一键）
  │      已处理过 → 直接返回成功（不重复操作）
  │      未处理 → 继续
  │
  ├→ 2. 状态校验（对提现相关接口）
  │      W-R4/W-R5：订单必须处于"已冻结"状态
  │      W-R4 和 W-R5 互斥
  │
  ├→ 3. 事务操作
  │      余额变更 + 流水写入在同一事务中
  │      失败则整体回滚
  │
  ├→ 4. 操作日志
  │      → ser-blog.AddActLog（oneway，不影响主流程）
  │
  └→ 5. 返回结果
```

**幂等性是所有 RPC 写接口的硬性要求**。原因：
- 三方回调可能重复推送（充值场景）
- 财务侧可能因网络问题重试请求
- 必须保证同一订单号只执行一次余额变更

### 4.3 人工修正流程详解

#### 人工加款

```
财务人员在B端打开加款弹窗
  │
  ▼
W-R1 GetUserWalletBalance → 展示目标用户各子钱包当前余额
  │
  ▼
财务人员填写加款信息
  │  指定用户ID、币种
  │  分别输入各子钱包的加款金额：
  │    中心钱包: 100
  │    奖励钱包: 50
  │    主播钱包: 0
  │    代理钱包: 0
  │  选择加款类型（运营补偿/运营奖励/财务修正/测试加款）
  │  填写备注
  │
  ▼
财务人员提交 → 生成订单号 A+16位
  │  创建人 ≠ 审核人（双人四眼原则）
  │
  ▼
另一位财务审核通过
  │
  ▼
W-R6 ManualCredit
  │
  ├→ 幂等校验（订单号）
  ├→ 按子钱包类型分别增加余额
  ├→ 为每个有变动的子钱包写入交易流水
  ├→ 操作日志 → ser-blog
  └→ 返回加款结果
```

#### 人工减款

```
流程同上，区别在于：
  │
  ▼
W-R7 ManualDebit
  │
  ├→ 幂等校验
  ├→ 余额校验（每个子钱包）：
  │    减款金额 ≤ 当前余额 → 扣减款金额
  │    减款金额 > 当前余额 → 扣当前余额（不报错，扣能扣的）
  ├→ 按子钱包分别扣减
  ├→ 写入流水（记录实际扣减金额）
  └→ 返回结果（含每个钱包实际扣减金额）
```

**减款特殊逻辑**：余额不足时不报错，而是扣实际余额。这是需求明确要求的——防止因金额不精确匹配导致减款失败。

#### 充值补单

```
财务人员关联原充值订单创建补单
  │  生成订单号 B+16位
  │  填写：补单金额、赠送金额、稽核倍数
  │
  ▼
审核通过
  │
  ▼
W-R8 CreditCorrection
  │
  ├→ 幂等校验
  ├→ 补单金额 → 中心钱包
  ├→ 赠送金额 → 奖励钱包（含稽核要求记录）
  ├→ 写入流水
  └→ 返回结果
```

> 依据：需求文档 E5 人工修正（加款/减款/补单）

### 5.2 每个操作对应的余额变动

| 操作 | 中心钱包 | 奖励钱包 | 主播钱包 | 备注 |
|------|---------|---------|---------|------|
| 充值上账 | +充值额 | +奖金额 | — | 奖金有流水要求 |
| 兑换 | 扣法币/加BSB | +赠送BSB | — | 赠送有流水要求 |
| 提现冻结 | 冻结X | — | — | 可用→冻结 |
| 提现确认 | -冻结X | — | — | 冻结→移出 |
| 提现解冻 | 解冻X | — | — | 冻结→可用 |
| 人工加款 | +A | +B | +C | 按指定金额分别加 |
| 人工减款 | -A | -B | -C | 余额不足扣实际余额 |
| 补单 | +补单额 | +赠送额 | — | 赠送有流水要求 |

### 5.3 统一余额变更入口的思路

所有余额变更必须经过统一的内部函数，不允许直接操作数据库：

```
统一入口函数：
  addBalance(userId, currencyCode, walletType, amount, txType, orderNo)
  deductBalance(userId, currencyCode, walletType, amount, txType, orderNo)
  freezeBalance(userId, currencyCode, walletType, amount, orderNo)
  unfreezeBalance(userId, currencyCode, walletType, amount, orderNo)
  confirmDeduct(userId, currencyCode, walletType, amount, orderNo)

每个函数内部：
  ├→ 校验参数
  ├→ 事务：更新余额 + 写流水
  ├→ 返回结果
```

**为什么要统一入口**：
1. 对账一致性：每次余额变更都必须有对应流水，统一入口保证不遗漏
2. 并发安全：统一入口可以加行锁或乐观锁
3. 审计追溯：所有余额变动都能追溯到操作来源

### 6.1 我调用别人的场景

### 7.2 余额操作的并发安全思路

```
方案：数据库行级锁（SELECT ... FOR UPDATE）

每次余额变更：
  BEGIN
    SELECT balance, frozen_amount FROM wallet
      WHERE user_id=? AND currency_code=? AND wallet_type=?
      FOR UPDATE;                    ← 锁定这一行

    -- 校验余额是否满足操作要求
    -- 更新余额
    UPDATE wallet SET balance=? WHERE id=?;

    -- 写入流水
    INSERT INTO wallet_transaction (...) VALUES (...);
  COMMIT

为什么用行锁而不是分布式锁：
  - 余额操作是数据库事务内的操作，行锁天然保证事务隔离
  - 锁粒度精确到「用户+币种+钱包类型」，不影响其他用户
  - 事务结束自动释放，不存在锁泄漏问题
```

## 八、订单状态机

### 8.1 充值订单状态流转

```
待支付 ──→ 进行中 ──→ 已完成
  │            │
  │            └──→ 已失败
  │
  └──→ 已关闭（超时未支付）

状态变更触发：
  待支付 → 进行中：用户跳转到三方页/开始链上转账
  进行中 → 已完成：财务回调 → W-R2 CreditDeposit 上账成功
  进行中 → 已失败：三方回调通知失败
  待支付 → 已关闭：超时未支付（定时任务或延时队列关闭）
```

### 8.2 提现订单状态流转

```
处理中 ──→ 审核中 ──→ 出款中 ──→ 已完成
  │           │          │
  │           │          └──→ 已失败（出款失败）
  │           │
  │           └──→ 已拒绝（审核拒绝）
  │
  └── 提现订单创建时就是"处理中"，金额已冻结

状态变更触发：
  处理中 → 审核中：订单进入财务审核队列
  审核中 → 已拒绝：W-R4 解冻（风控/财务拒绝）
  审核中 → 出款中：三级审核通过，分配通道出款
  出款中 → 已完成：W-R5 确认扣款（通道出款成功）
  出款中 → 已失败：W-R4 解冻（通道出款失败）

  已拒绝/已失败 = 资金解冻返还用户
  已完成 = 资金永久移出
```

### 8.3 兑换——无需状态机

兑换是**同步操作**，没有中间状态。执行成功就是成功，失败就回滚。不存在"待处理"或"审核中"的状态。

## 九、按实现优先级的流程分组

### 第一批：基座（必须最先做）

```
1. 币种配置 CRUD（CC-1~CC-6）
   → 没有币种数据，后面什么都做不了
   → 纯B端操作，不依赖其他模块

2. 汇率定时同步（CC-T1）
   → 汇率不更新，充值/兑换/提现的金额换算无法工作

3. 币种配置 RPC 供他（CC-R1~CC-R4）
   → 财务管理等模块要用，需要尽早提供
```

### 第二批：钱包核心（余额+流水+查询）

```
4. 钱包余额查询（W-1, W-R1）
   → 底层能力：getBalance 函数

5. 币种列表（W-2）
   → 用户进钱包的第一步

6. 交易记录查询（W-14, W-15）
   → 需要先有流水表结构
```

### 第三批：兑换（内部闭环，不依赖外部）

```
7. 兑换预览 + 执行（W-8, W-9）
   → 零外部依赖，可以独立验证
   → 验证点：跨币种事务、汇率快照、赠送进奖励钱包
```

### 第四批：RPC供财务（余额变更能力）

```
8. 充值上账（W-R2）
9. 冻结/解冻/扣款（W-R3, W-R4, W-R5）
10. 人工加减款（W-R6, W-R7）
11. 补单（W-R8）
   → 全部是余额变更操作，复用底层函数
   → 验证点：幂等性、事务原子性
```

### 第五批：充值链路（依赖财务RPC）

```
12. 充值方式查询（W-3）→ 依赖财务 EXT-1
13. 创建充值订单（W-4）→ 含重复拦截逻辑
14. USDT收款信息（W-5）
15. 奖金活动（W-6）→ 数据源待确认
16. 订单状态轮询（W-7）
```

### 第六批：提现链路（依赖财务+KYC）

```
17. 提现信息聚合（W-10）→ 依赖财务 EXT-2 + KYC EXT-3
18. 提现账户管理（W-12, W-13）
19. 创建提现订单（W-11）→ 含冻结逻辑
```

## 十、关键设计决策点

以下是流程中需要在编码前明确的设计决策：

### 10.1 已确定的设计原则

| 原则 | 说明 | 来源 |
|------|------|------|
| 余额变更+流水写入在同一事务 | 保证对账一致性 | 需求A7 |
| 所有RPC写操作幂等 | 用订单号做唯一键 | 需求A7 |
| 提现冻结+订单创建在同一事务 | 防止冻结成功但订单创建失败 | 需求A5 |
| W-R4和W-R5互斥 | 同一订单不能既解冻又扣款 | 状态机约束 |
| 汇率快照机制 | 预览汇率和执行汇率一致 | 需求A4 |
| 先扣中心后扣奖励 | 消费时的扣款优先级 | 需求B2 |
| 账户持有人=KYC姓名 | 防冒名提现 | 需求A5 |
| 钱从哪来往哪去 | 法币充→法币提 | 需求A5 合规 |

### 10.2 待确认的决策点

| 序号 | 决策点 | 影响范围 | 当前倾向 |
|------|--------|---------|---------|
| 1 | 充值/提现订单数据归属 | W-4/W-11 创建的订单存在钱包还是财务 | 钱包存自己的，财务存自己的，通过订单号关联 |
| 2 | 三方支付回调接收方 | 回调到财务还是钱包 | 财务接收，验签后 RPC 通知钱包上账 |
| 3 | 奖金活动数据源 | W-6 从哪里获取活动列表 | 当前21个服务无活动服务，可能需新建或临时由钱包自管 |
| 4 | 稽核流水追踪 | X倍流水达标后谁通知钱包解除限制 | 可能是投注服务追踪，达标后通知钱包 |
| 5 | USDT收款地址管理 | W-5 返回的地址从哪来 | 可能是固定地址配置，也可能是区块链服务动态生成 |
| 6 | 预留钱包接口 | 场馆/代理钱包首版是否留骨架 | 表结构预留wallet_type字段，接口暂不实现 |
