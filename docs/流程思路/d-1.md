> 基于：需求理解(d-1) + 接口评估(d-1) + 链路关系(d-1) + 依赖关系(d-1) + 工程认知(d-1)

## 一、总体流程思路概述

### 1.1 我负责的模块全景

我负责的 ser-wallet 服务内部包含**两个子模块**，一个对外联动：

```
ser-wallet 服务
├── 子模块A：币种配置（基础设施层）
│   ├── 币种CRUD（B端）
│   ├── 汇率管理（三方拉取 + 阈值自动更新）
│   ├── 充值档位/提现限额配置（B端）
│   └── RPC对外供给（给钱包业务层 + 给财务模块读取）
│
├── 子模块B：钱包业务（核心业务层）
│   ├── 余额管理（查询/冻结/解冻/入账/扣减）
│   ├── 充值流程（C端发起 → 匹配通道 → 建单 → 等待回调入账）
│   ├── 兑换流程（法币→BSB，钱包内部自闭环）
│   ├── 提现流程（C端发起 → 冻结 → 提交财务审核 → 等待回调解冻/扣款）
│   ├── 记录查询（充值/提现/兑换/奖励四类记录）
│   ├── 提现账户管理（绑定银行卡/电子钱包/USDT地址）
│   └── RPC对外供给（给财务回调入账/冻结/解冻/扣款/调整）
│
└── 对外联动
    ├── 调用财务：获取充值方式列表、匹配支付通道（2条出口）
    ├── 被财务调用：入账/冻结/解冻/扣款/调整（5条入口）
    ├── 调用KYC：提现前校验认证状态（1条出口）
    └── 调用Blog：审计日志（oneway，不阻塞）
```

### 1.2 一句话流程主线

**币种配置先行（地基）→ 钱包核心功能搭建（余额/兑换/记录/提现账户）→ 充值提现链路与财务联调 → 增强功能补充**

### 1.3 核心处理思维

- **子模块A（币种配置）是纯数据供给方**，不调用任何业务模块，自身可独立运作
- **子模块B（钱包业务）的所有资金操作都必须满足三个硬约束**：事务原子性、幂等性、流水可追溯
- **与财务的联动是双向的**：充值时我调财务（获取方式+匹配通道），充值完成时财务调我（入账）；提现时我冻结后推给财务审核，审核后财务调我（解冻/扣款）

## 二、子模块A：币种配置处理流程

### 2.1 币种配置总体流程思路

币种配置是整个钱包体系的**数据地基**。它管理的是"什么币种可用、汇率多少、充值档位怎么设、提现限额多少"等基础数据。这些数据一旦就绪，钱包业务层和财务模块都能正常工作。

**核心流程思路**：

```
                    ┌─────────────────────────────────────────┐
                    │         币种配置子模块处理流程              │
                    │                                          │
  B端操作员 ──────► │  ① 导入/新增币种基础数据                   │
                    │     ↓                                    │
                    │  ② 配置汇率浮动%和阈值%                   │
                    │     ↓                                    │
  三方汇率API ────► │  ③ 定时拉取实时汇率（3家取平均）            │
                    │     ↓                                    │
                    │  ④ 计算偏差值，判断是否超阈值               │
                    │     ├─ 超阈值 → 更新平台汇率 → 计算入款/出款汇率 → 写汇率日志  │
                    │     └─ 未超阈值 → 保持不变                │
                    │     ↓                                    │
                    │  ⑤ 配置充值档位（快捷金额、最低金额）       │
                    │     ↓                                    │
                    │  ⑥ 配置提现限额（单笔/单日/手续费/T+N）    │
                    │     ↓                                    │
                    │  ⑦ 启用币种（状态开关）                    │
                    │     ↓                                    │
                    │  ⑧ RPC接口对外供给：                      │
                    │     → 钱包业务层读取币种/汇率/档位/限额      │
                    │     → 财务模块读取币种/汇率用于统计           │
                    └─────────────────────────────────────────┘
```

### 2.2 汇率自动更新流程（核心子流程）

这是币种配置中最有逻辑的一个子流程：

```
定时触发（频率可配，测试期1天1次，上线后按运营需求调）
  │
  ├─ 步骤1：从3家三方API拉取目标币种的实时汇率
  │   → 取3家平均值作为"实时汇率"
  │   → 某家API不可用时降级为可用的那几家平均
  │
  ├─ 步骤2：判断当前是否存在平台汇率
  │   ├─ 不存在（首次）→ 直接用实时汇率作为平台汇率 → 跳到步骤4
  │   └─ 存在 → 进入步骤3
  │
  ├─ 步骤3：计算偏差值
  │   → 偏差值 = |实时汇率 - 平台汇率| / 实时汇率 × 100
  │   → 偏差值 ≥ 阈值% → 更新平台汇率 = 实时汇率 → 跳到步骤4
  │   → 偏差值 < 阈值% → 保持不变，结束
  │
  ├─ 步骤4：基于新平台汇率计算入款/出款汇率
  │   → 入款汇率 = 平台汇率 × (1 + 汇率浮动%)    ← 用户充值时用，平台赚差价
  │   → 出款汇率 = 平台汇率 × (1 - 汇率浮动%)    ← 用户提现时用，平台赚差价
  │
  └─ 步骤5：写入汇率变更日志
      → 记录：币种代码、实时汇率、更新后的平台汇率、入款汇率、出款汇率、时间
```

**依据**：币种配置需求文档"4-2汇率规则说明"明确了偏差值公式和阈值判断逻辑。

### 2.3 币种配置对外供给的完整场景

| 调用方 | 什么时候调 | 调什么 | 返回什么 |
|-------|---------|-------|---------|
| 钱包 — 余额概览 | 用户打开钱包首页 | GetActiveCurrencyList + BatchGetExchangeRates | 所有启用币种列表 + 各币种汇率 |
| 钱包 — 充值页面 | 用户进入充值界面 | GetDepositWithdrawConfig + GetExchangeRate | 充值档位 + BSB入款汇率 |
| 钱包 — 兑换 | 用户兑换预览/执行 | GetExchangeRate | 法币→BSB的入款汇率 |
| 钱包 — 提现 | 用户申请提现 | GetDepositWithdrawConfig + GetExchangeRate | 提现限额规则 + BSB出款汇率 |
| 钱包 — 币种校验 | 每次操作前 | GetActiveCurrencyList 或 GetCurrencyDetail | 币种是否启用 |
| 财务 — 统计报表 | 生成充值/提现统计 | GetActiveCurrencyList + GetExchangeRate | 币种列表 + 平台汇率（非入出款汇率） |
| 财务 — 配置页面 | 通道配置/支付配置B端 | GetActiveCurrencyList | 币种下拉列表 |

**处理思路**：因为币种配置和钱包在同一个 ser-wallet 服务内，上表中"钱包调用币种配置"实际是**内部方法调用**，不走RPC。而"财务调用币种配置"则走**RPC**（rpc.WalletClient().GetActiveCurrencyList）。

## 三、子模块B：钱包核心——余额管理处理流程

### 3.1 余额管理的核心数据模型思路

在展开每个业务流程之前，先明确余额管理的核心模型思路：

```
每个用户 × 每个币种 × 每种钱包类型 = 一个钱包账户

一个钱包账户有两个关键数字：
├── 可用余额 — 用户可以自由支配的金额
└── 冻结余额 — 因提现审核等原因暂时锁定的金额

"资金钱包余额" = 中心钱包可用余额 + 奖励钱包可用余额（C端展示的总额）
```

**所有余额变动只通过两个底层方法实现**：

```
getBalance(userId, currencyId, walletType) → 读取可用余额和冻结余额
modifyBalance(userId, currencyId, walletType, operationType, amount, orderId) → 修改余额

operationType 包括：
├── credit    — 增加可用余额（充值入账、人工加款）
├── debit     — 减少可用余额（人工减款、兑换扣减源币种）
├── freeze    — 可用余额 → 冻结余额（提现申请）
├── unfreeze  — 冻结余额 → 可用余额（提现驳回/出款失败）
└── debit_frozen — 减少冻结余额（提现出款成功）
```

每次 modifyBalance 必须同时生成一条 transaction_log 流水记录。这是**不可协商的硬约束**。

**依据**：需求文档"9非功能性需求"明确"账务流水与账户余额的强一致性校验机制"。

### 3.2 钱包账户懒创建流程

用户首次使用某个币种时，系统中还没有对应的钱包账户记录，需要自动初始化：

```
用户请求某币种的余额/充值/提现/兑换
  │
  ├─ 查询该用户 + 该币种下的钱包账户记录
  │
  ├─ 存在 → 直接返回/继续操作
  │
  └─ 不存在 → 自动创建4种钱包类型的账户记录
      ├─ 中心钱包（可用=0，冻结=0）
      ├─ 奖励钱包（可用=0，冻结=0）
      ├─ 主播钱包（可用=0，冻结=0）
      └─ 代理钱包（可用=0，冻结=0）— 预留
      注意：场馆钱包在当前阶段不创建（需求文档说当前游戏均为自研，不经过场馆钱包中转）
```

**依据**：需求文档"2-1资金钱包体系"描述了5类钱包，其中场馆钱包的使用场景为"进入igaming场馆时"，而"当前游戏均为自研游戏，进入游戏后钱包余额=资金钱包余额（不经过场馆钱包中转）"。

### 3.3 消费扣款优先级流程（跨钱包扣减）

需求文档明确了一条关键规则：**消费扣款优先级：先扣中心钱包余额，再扣奖励钱包余额**。

这条规则影响到投注扣款（未来由投注系统调用）和兑换扣款（自己实现）：

```
请求扣减金额 X
  │
  ├─ 读取中心钱包可用余额 A
  ├─ 读取奖励钱包可用余额 B
  │
  ├─ A + B < X → 余额不足，拒绝
  │
  ├─ A >= X → 只扣中心钱包
  │   → modifyBalance(中心, debit, X)
  │
  └─ A < X → 先扣光中心钱包，差额扣奖励钱包
      → modifyBalance(中心, debit, A)
      → modifyBalance(奖励, debit, X - A)
      → 如果是投注场景，还需记录两个钱包的扣减比例（后续返奖时按比例分配）
```

**投注返奖的额外规则**：中心钱包余额投注返奖计入中心钱包，奖励钱包余额投注返奖计入奖励钱包。混合投注时按投注金额占比分开计返奖。

**依据**：需求文档"2-1资金钱包体系7.png"明确规定了扣款优先级和混合投注返奖比例规则。

## 四、充值全流程处理思路

### 4.1 充值总体流程（从用户操作到余额变更）

充值是整个钱包系统中**最长的一条链路**，横跨用户→钱包→财务→三方→财务→钱包六个节点。

**我的模块在充值链路中的职责**：
- 展示充值方式和档位（调用财务+币种配置）
- 重复转账拦截（本地校验）
- 创建充值订单（本地写入）
- 接收充值回调并入账（被财务调用）

**我不负责的部分**：
- 通道轮询匹配逻辑（财务负责）
- 三方支付交互细节（财务+三方）
- 三方回调接收和解析（财务负责，转发给我）

### 4.2 银行/扫码充值详细处理流程

```
                       用户操作                      我的模块处理                    财务模块处理

                         │                              │   │  (如有关联奖金活动)         │
                         │                              │   ├─ 生成入账流水记录           │
                         │                              │   └─ 更新订单状态为"已完成"     │
                         │                              │                             │

                         │                              │── 查本地DB：订单最新状态       │
                         │◄──── 返回：成功/待支付/失败 ────│                             │
```

**关键处理思路**：

1. **拦截在前、建单在后**：先做重复转账检测和金额校验，通过后才匹配通道和建单。这样避免无效订单
2. **汇率快照锁定**：BSB充值时，创建订单时锁定当前入款汇率，不管之后汇率怎么变，这笔订单按锁定时的汇率计算
3. **入账必须幂等**：同一订单号多次回调只入账一次，基于订单号+状态机双重保障
4. **充值金额和奖金分开入不同钱包**：充值金额进中心钱包，额外奖金进奖励钱包，两者的提现规则不同（奖金需完成流水倍数）

### 4.3 USDT充值处理流程（与银行/扫码的区别）

```
用户选择USDT充值 + 选择网络(TRC20/ERC20)
  │
  ├─ 与银行/扫码的不同点：
  │   ├─ 不需要匹配通道（不调财务.MatchDepositChannel）
  │   ├─ 不跳转三方页面（展示收款地址让用户自行转账）
  │   ├─ 重复转账检测豁免（链上以实际交易为准，需求文档明确"不适用于USDT链上充值"）
  │   └─ 到账时间取决于区块链确认（2~10分钟），非三方通道实时回调
  │
  ├─ 步骤1：获取/分配收款地址
  │   → 查本地DB：该用户该网络是否已分配地址
  │   → 未分配 → 调用链上服务分配新地址（三方API或内部服务，待评审确认）
  │   → 已分配 → 复用
  │
  ├─ 步骤2：创建充值订单 [DB写]
  │   → 类型=USDT链上，状态=待支付，锁定入款汇率
  │
  ├─ 步骤3：返回收款地址+二维码+风险提示
  │
  └─ 后续：链上监听服务确认到账 → 回调（同银行/扫码的步骤④之后的入账逻辑）
```

**依据**：需求文档"5-2USDT充值"和"5-5重复转账拦截规则"明确USDT不适用重复转账检测。

### 4.4 充值奖金处理思路

```
用户在充值页面选择奖金活动
  │
  ├─ 展示阶段：
  │   → 列出可用奖金活动（"获得额外50%奖金，需最低充值BSB 1000"）
  │   → 用户可选"无奖金"
  │   → 选择活动后，充值档位联动变化（显示奖金比例角标如"+50%"）
  │   → 输入框右侧展示"额外+XXX"
  │
  ├─ 校验阶段：
  │   → 手动输入金额 < 活动最低门槛 → 提示"该活动最低充值额需BSB 1000"
  │   → 同一笔订单只关联一个活动（互斥）
  │
  └─ 入账阶段：
      → 充值金额 → 中心钱包
      → 额外奖金(充值金额 × 奖金比例) → 奖励钱包
      → 奖励钱包的这笔奖金需完成x倍流水才可提现
      → 订单记录中关联活动ID、奖金比例、奖金金额、流水要求
```

## 五、兑换全流程处理思路

### 5.1 兑换流程特点

兑换是钱包内部**完全自闭环**的流程，**不涉及财务模块**，**不涉及三方交互**。这意味着：
- 不需要等财务联调，可以独立开发测试
- 链路短，只有钱包内部处理
- 唯一的外部依赖是汇率数据（本服务内部的币种配置）

### 5.2 兑换详细处理流程

```
用户在兑换页面（仅VND/IDR/THB法币钱包有此入口，BSB没有）
  │
  ├─ 预览阶段 [纯计算，不写DB]
  │   │
  │   ├─ 用户输入法币金额 X
  │   │
  │   ├─ 内部调用：获取该法币→BSB的入款汇率 R
  │   │
  │   ├─ 计算：
  │   │   兑换获得BSB = X ÷ R
  │   │   赠送BSB = 兑换获得BSB × 赠送比例%（赠送比例来源待评审确认）
  │   │   实际到账BSB = 兑换获得 + 赠送
  │   │
  │   ├─ 校验：X > 当前法币可用余额 → 提示"已达当前可兑换上限"
  │   │
  │   └─ 返回：汇率、兑换获得、赠送比例、赠送金额、实际到账、流水倍数要求
  │
  └─ 执行阶段 [DB写，事务内]
      │
      ├─ 步骤1：再次获取汇率并锁定
      │   → 防止预览到执行之间汇率变动
      │
      ├─ 步骤2：校验源币种余额
      │   → 查中心钱包可用余额 ≥ 兑换金额
      │   → 不足 → 拒绝
      │
      ├─ 步骤3：执行余额变更 [必须在同一个DB事务内]
      │   → modifyBalance(源币种·中心, debit, X)     ← 扣减源法币
      │   → modifyBalance(BSB·中心, credit, 兑换获得)  ← 增加BSB中心钱包
      │   → modifyBalance(BSB·奖励, credit, 赠送金额)  ← 增加BSB奖励钱包（赠送部分）
      │   → 写入 exchange_record（记录汇率、金额、赠送比例、流水要求）
      │   → 生成3条 transaction_log（源币种扣减 + BSB入账 + 赠送入账）
      │
      ├─ 步骤4：审计日志 (oneway)
      │
      └─ 返回：兑换成功 + 明细
```

**关键处理思路**：

1. **预览和执行分两步**：预览是纯计算（不写DB、不扣余额），执行时才真正操作
2. **执行时重新获取汇率**：不信任预览时的汇率，执行时以最新汇率为准并锁定
3. **事务原子性**：扣减源币种和增加BSB必须在同一事务内，不能出现"扣了法币但BSB没加上"的情况
4. **赠送进奖励钱包**：赠送部分和兑换主体分开入不同钱包，赠送需完成流水后才可提现

**依据**：需求文档"6兑换模块"明确兑换范围、汇率展示口径、赠送规则。

## 六、提现全流程处理思路

### 6.1 提现流程特点

提现是整个钱包系统中**流程最复杂**的链路，因为：
- 涉及KYC认证校验（合规要求）
- 涉及余额冻结/解冻（资金安全）
- 涉及财务三级审核（风控/财务/出款）
- 涉及三方代付通道出款
- 存在多种终态（成功/驳回/出款失败），每种终态对余额的处理不同

### 6.2 提现详细处理流程

```
用户在提现页面
  │
  ├─ 阶段一：前置校验
  │   │
  │   ├─ ①点击"提现"按钮
  │   │   → RPC调出：ser-kyc.KycQuery(userId) → 获取KYC认证状态
  │   │   → 内部调用：获取提现限额配置（单笔/单日限额）
  │   │   → 查本地DB：当前币种可用余额、今日已提现金额
  │   │   → 计算：剩余可提额度 = 单日限额 - 今日已提现
  │   │
  │   ├─ 根据KYC状态分流：
  │   │   ├─ 未认证 → 弹窗：
  │   │   │   ├─ "去认证" → 跳转KYC页面
  │   │   │   └─ "继续提现" → 仅可选择USDT提现
  │   │   └─ 已认证 → 直接进入提现选择页，全渠道可用
  │   │
  │   └─ 返回：KYC状态、可用余额、单笔限额、单日限额、今日已用、剩余可提
  │
  ├─ 阶段二：绑定提现账户（首次提现时）
  │   │
  │   ├─ ②判断是否已有提现账户
  │   │   → 查本地DB：该用户该币种是否有已绑定的提现账户
  │   │   → 有 → 跳过绑定，展示已有账户信息（只读）
  │   │   → 没有 → 进入绑定流程
  │   │
  │   ├─ 绑定流程：
  │   │   → RPC调出：ser-kyc.KycQuery(userId) → 获取KYC认证姓名
  │   │   → "账户持有人"字段自动填充KYC姓名，**不可编辑**
  │   │   → 用户填写：银行名称(下拉)/账号/开户网点 或 电子钱包类型/手机号 或 USDT地址
  │   │   → 写入提现账户表 [DB写]
  │   │
  │   └─ 后续提现：自动带出已绑定账户信息，用户只需输入金额
  │
  ├─ 阶段三：创建提现申请
  │   │
  │   ├─ ③用户输入提现金额，点击"确认提现"
  │   │
  │   ├─ A. 金额校验
  │   │   → 金额 < 单笔最低 → 提示"未达到单笔提现最低金额"
  │   │   → 金额 > 单笔最高 → 拒绝
  │   │   → 今日已用 + 本次 > 单日限额 → 提示"已超出当日提现限额"
  │   │
  │   ├─ B. 余额校验
  │   │   → 查中心钱包可用余额 ≥ 提现金额 + 手续费
  │   │   → 不足 → 提示余额不足
  │   │
  │   ├─ C. 冻结余额 [DB写，事务内]
  │   │   → modifyBalance(中心, freeze, 提现金额 + 手续费)
  │   │   → 可用余额 -= 冻结金额
  │   │   → 冻结余额 += 冻结金额
  │   │   → 写入冻结流水记录
  │   │
  │   ├─ D. 创建提现订单 [DB写，与C同一事务]
  │   │   → 状态=待审核
  │   │   → BSB提现时锁定出款汇率快照
  │   │   → 关联冻结记录、提现账户ID
  │   │
  │   ├─ E. 提交给财务审核
  │   │   → 将提现订单信息推送给财务模块（具体对接方式待确认）
  │   │
  │   ├─ F. 审计日志 (oneway)
  │   │
  │   └─ 返回：提现申请已提交 + 订单号
  │
  ├─ 阶段四：财务三级审核（不在我的模块内，但我需要清楚流程）
  │   │
  │   │  ┌──────────────────────────────────────────┐
  │   │  │  风控审核 → 通过? ──否──→ 驳回            │
  │   │  │      ↓是                                  │
  │   │  │  财务审核 → 通过? ──否──→ 驳回            │
  │   │  │      ↓是                                  │
  │   │  │  出款审核 → 通过? ──否──→ 驳回            │
  │   │  │      ↓是                                  │
  │   │  │  向三方代付通道发起出款                      │
  │   │  │      ↓                                    │
  │   │  │  三方回传出款结果                           │
  │   │  └──────────────────────────────────────────┘
  │   │
  │   │  审核结果分三种终态：
  │
  ├─ 终态A：审核驳回（任一级不通过）
  │   │
  │   │  财务 → 钱包 RPC：UnfreezeBalance(userId, currencyId, 冻结金额, 关联单号)
  │   │   → 幂等校验：按关联单号查冻结记录，已解冻则直接返回
  │   │   → 执行解冻 [DB写，事务内]：
  │   │     → 冻结余额 -= 冻结金额
  │   │     → 可用余额 += 冻结金额（**钱归还给用户**）
  │   │     → 更新冻结记录状态=已解冻
  │   │     → 更新提现订单状态=失败
  │   │     → 写入解冻流水记录
  │   │
  ├─ 终态B：出款成功
  │   │
  │   │  财务 → 钱包 RPC：DebitBalance(userId, currencyId, 冻结金额, 关联单号)
  │   │   → 幂等校验：按关联单号查，已扣款则直接返回
  │   │   → 执行扣款 [DB写，事务内]：
  │   │     → 冻结余额 -= 冻结金额（**从冻结中扣减，非可用余额**）
  │   │     → 更新冻结记录状态=已扣款
  │   │     → 更新提现订单状态=已完成
  │   │     → 写入扣款流水记录
  │   │
  └─ 终态C：出款失败（审核通过但三方代付失败）
      │
      │  处理方式同终态A → 调用UnfreezeBalance → 解冻归还用户
```

**关键处理思路**：

1. **冻结与建单必须原子**：阶段三的步骤C和D必须在同一个DB事务内。冻结失败则不建单，建单失败则回滚冻结
2. **冻结→扣款是两步走**：先冻结（可用→冻结），审核通过后再扣款（冻结→扣减）。中间状态（冻结中）用户看到的可用余额已减少，但钱还在系统中
3. **三种终态清晰对应**：驳回/出款失败→解冻归还；出款成功→从冻结中扣减。两条路径互斥
4. **KYC是提现的前置门槛**：未认证只能USDT提现，已认证才能全渠道。这是合规硬约束
5. **提现账户信息不可自改**：首次绑定后后续复用，持有人姓名必须等于KYC认证姓名

**依据**：需求文档"7-1提现模块"的KYC关系表、"7-2~7-4"提现方式详细规则、财务需求文档"2-4提现记录"的三级审核流程图。

## 七、记录查询处理流程

### 7.1 记录模块流程思路

记录模块是**纯读取**模块，不产生任何数据写入，不调用任何外部服务。它只是把充值/提现/兑换/奖励四类操作的历史记录展示给用户。

```
用户在记录页面（4个Tab：充值、提现、兑换、奖励）
  │
  ├─ 核心规则：当前选中币种 = 记录数据的币种（不支持跨币种混合记录）
  │   → 切换币种时，四个Tab的数据全部刷新

  │
  ├─ 提现记录
  │   → 查 withdraw_order 表
  │   → 展示同上，状态多"审核中"

  │
  ├─ 兑换记录

  │   → 展示：兑换金额、赠送比例%、实际到账BSB、时间
  │
  └─ 奖励记录

      → 展示：奖励金额、奖励类别、时间
```

**处理思路**：记录模块完全无外部依赖，可以在表结构设计好之后**最先开发完成**，用来做端到端测试验证的"可观测窗口"——充值/兑换/提现完成后立即去记录页验证数据是否正确。

**依据**：需求文档"8-1~8-5"记录模块规则、产品原型"11-1~11-2记录"。

## 十、模块间联动——别人调用我的处理流程

### 10.1 被调用接口总览

财务模块在5个场景下会调用我的RPC接口：

```
财务场景                    调用我的接口              余额操作
──────────────────────────────────────────────────────────────
充值回调成功              → CreditBalance           → 可用余额 +
充值补单审核通过          → CreditBalance           → 可用余额 +
提现审核驳回/出款失败     → UnfreezeBalance         → 冻结 → 可用（归还）
提现出款成功              → DebitBalance            → 冻结余额 -
人工加款审核通过          → ManualAdjust(加)        → 可用余额 +
人工减款审核通过          → ManualAdjust(减)        → 可用余额 -
人工减款前校验            → GetUserBalance          → 只读
```

### 10.2 所有被调用接口的统一处理模式

这5个资金操作接口（CreditBalance、FreezeBalance、UnfreezeBalance、DebitBalance、ManualAdjust）必须遵循统一的处理模式：

```
任何一个资金操作RPC请求进来
  │
  ├─ 第1步：幂等校验
  │   → 按"关联单号"查本地DB
  │   → 如果该单号已处理过 → 直接返回成功（不重复执行）
  │   → 这是所有资金操作的**第一道防线**
  │
  ├─ 第2步：前置校验（按操作类型不同）
  │   ├─ CreditBalance：查订单状态是否为"待入账"
  │   ├─ FreezeBalance：查可用余额 ≥ 冻结金额
  │   ├─ UnfreezeBalance：查冻结记录状态是否为"冻结中"
  │   ├─ DebitBalance：查冻结记录状态是否为"冻结中"
  │   └─ ManualAdjust(减)：查可用余额 ≥ 减款金额
  │
  ├─ 第3步：执行余额变更 [DB写，事务内]
  │   → 调用 modifyBalance 底层方法
  │   → 同时生成 transaction_log 流水记录
  │   → 同时更新关联订单状态（如有）
  │
  ├─ 第4步：审计日志 (oneway)
  │
  └─ 返回结果
```

### 10.3 ManualAdjust的特殊处理

人工加款/减款有一个特殊点：可能同时对**多个钱包类型**操作。

```
财务 → ManualAdjust(userId, currencyId, adjustments=[
  {walletType=中心, amount=1000},
  {walletType=奖励, amount=500},
  {walletType=主播, amount=200},
])
  │
  └─ 处理：在同一个DB事务中，对3个钱包账户分别执行余额变更
      → modifyBalance(中心, credit/debit, 1000)
      → modifyBalance(奖励, credit/debit, 500)
      → modifyBalance(主播, credit/debit, 200)
      → 3条 transaction_log
      → 事务提交（全部成功或全部回滚）
```

**依据**：财务需求文档"2-5人工修正"明确"选择币种→对4类钱包分别设置加款金额"。

## 十二、跨场景状态流转全景

### 12.1 充值订单状态机

```
创建 → 待支付 → 进行中 → 成功(入账) → 已完成
                    ↓
                  失败/超时

状态流转触发者：
  待支付 → 进行中：三方回调支付中
  进行中 → 成功：三方回调支付成功 + 钱包入账
  进行中 → 失败：三方回调支付失败
  待支付 → 超时：超过支付有效期
```

### 12.2 提现订单状态机

```
创建 → 待审核 → 风控审核中 → 财务审核中 → 出款审核中 → 出款中 → 已完成
                    ↓              ↓              ↓         ↓
                  驳回(解冻)     驳回(解冻)     驳回(解冻)   出款失败(解冻)

状态流转触发者：
  待审核 → 审核中：财务B端操作
  审核中 → 驳回：财务审核 → 调钱包UnfreezeBalance
  出款中 → 已完成：三方回调 → 调钱包DebitBalance
  出款中 → 出款失败：三方回调 → 调钱包UnfreezeBalance
```

### 12.3 余额字段的状态流转

```
可用余额的增减：
  + ：充值入账(CreditBalance)、解冻归还(UnfreezeBalance)、人工加款(ManualAdjust)
  - ：冻结(FreezeBalance)、兑换扣减(ExecuteExchange)、人工减款(ManualAdjust)

冻结余额的增减：
  + ：冻结(FreezeBalance)
  - ：解冻(UnfreezeBalance)、扣款确认(DebitBalance)

约束：可用余额 + 冻结余额 = 账户总余额 ≥ 0
```

## 十三、数据流生命周期全景

### 13.1 一笔充值的完整数据流

```
时刻T1 — 用户点击充值：
  写入 deposit_order（状态=待支付, 金额=1000 VND, 汇率快照=71, 关联通道=xxx）

时刻T2 — 三方回调成功：
  更新 deposit_order（状态=已完成）
  更新 wallet_account（中心钱包可用余额 += 1000 VND）
  写入 transaction_log（类型=充值入账, 金额=+1000, 关联订单号=xxx）
  [如有奖金]：
  更新 wallet_account（奖励钱包可用余额 += 500 VND）
  写入 transaction_log（类型=充值奖金, 金额=+500, 关联活动=yyy, 流水要求=3倍）

时刻T3 — 用户查看记录：
  读取 deposit_order → 展示充值记录
  读取 transaction_log → 展示流水明细
```

### 13.2 一笔提现的完整数据流

```
时刻T1 — 用户申请提现（钱包处理）：
  更新 wallet_account（中心钱包可用余额 -= 500, 冻结余额 += 500）
  写入 freeze_record（状态=冻结中, 金额=500, 关联提现单=xxx）
  写入 withdraw_order（状态=待审核, 金额=500, 手续费=10, 汇率快照=69）
  写入 transaction_log（类型=冻结, 金额=-500(可用), +500(冻结)）

时刻T2 — 财务三级审核中：
  无钱包侧数据变动（审核在财务模块内部流转）

时刻T3a — 审核驳回（财务调钱包UnfreezeBalance）：
  更新 wallet_account（冻结余额 -= 500, 可用余额 += 500）← 钱归还
  更新 freeze_record（状态=已解冻）
  更新 withdraw_order（状态=失败）
  写入 transaction_log（类型=解冻, 金额=+500(可用), -500(冻结)）

时刻T3b — 出款成功（财务调钱包DebitBalance）：
  更新 wallet_account（冻结余额 -= 500）← 从冻结中扣减，钱真正离开系统
  更新 freeze_record（状态=已扣款）
  更新 withdraw_order（状态=已完成）
  写入 transaction_log（类型=提现扣款, 金额=-500(冻结)）
```

### 13.3 一笔兑换的完整数据流

```
时刻T1 — 用户执行兑换（1000 VND → BSB，汇率71，赠送10%）：
  更新 wallet_account（VND中心钱包可用余额 -= 1000）
  更新 wallet_account（BSB中心钱包可用余额 += 14.08）  ← 1000 ÷ 71
  更新 wallet_account（BSB奖励钱包可用余额 += 1.41）   ← 14.08 × 10%
  写入 exchange_record（源=1000 VND, 获得=14.08 BSB, 赠送=1.41 BSB, 汇率=71）
  写入 transaction_log × 3（VND扣减 + BSB入账 + BSB赠送入账）
```

## 十五、核心处理要点总结

### 15.1 资金安全三原则

| 原则 | 含义 | 落地方式 |
|------|------|---------|
| **事务原子性** | 涉及多表的余额变更必须在同一事务内 | GORM Transaction + 同一事务内写余额表+流水表+订单表 |
| **幂等性** | 同一请求重复调用不产生重复效果 | 每个资金操作基于"关联单号"去重，先查后写 |
| **流水可追溯** | 每笔余额变动都有对应流水记录 | modifyBalance 方法内部强制同时写 transaction_log |

### 15.2 依赖阻塞分析

| 依赖项 | 阻塞哪些接口 | 是否可控 |
|-------|------------|---------|
| 币种配置（自己） | 几乎所有钱包接口 | **完全可控**，优先开发 |
| 财务GetPaymentMethods | C3、C11（充值/提现方式展示） | 需等财务，但不阻塞钱包核心 |
| 财务MatchDepositChannel | C5（充值建单） | 需等财务，Phase 3 联调 |
| ser-kyc | C12、C14（提现校验+绑定账户） | 已有服务，不阻塞 |
| ser-blog | 所有写操作的审计日志 | 已有服务，oneway不阻塞 |

**结论**：77%的钱包接口（27/35）可以完全独立开发测试，只有充值方式展示和充值建单两个场景需要等财务模块联调。

### 15.3 模块内子流程的复用关系

```
getBalance ← 被8个接口复用（C1, C2, C10, C12, C15, R2, R5, R6, R7）
modifyBalance ← 被8个接口复用（R1~R5, C10, C15, R8）
checkIdempotent ← 被6个接口复用（R1~R5, R8）
checkWithdrawLimit ← 被3个接口复用（C12, C15, C16）

→ 这4个底层方法是钱包模块的"基础设施"，一旦实现好，上层接口开发速度会非常快
```

### 15.4 一句话总结每个子模块的流程核心

- **币种配置**：静态数据管理 + 汇率自动更新 → 是地基，先做
- **余额管理**：getBalance + modifyBalance + 事务 + 幂等 → 是核心引擎
- **充值**：展示方式 → 拦截检测 → 匹配通道 → 建单 → 等回调入账 → 链路最长，双向依赖财务
- **兑换**：预览计算 → 执行扣减+入账 → 自闭环，最简单
- **提现**：KYC校验 → 限额校验 → 冻结 → 推审核 → 等回调解冻/扣款 → 流程最复杂，三种终态
- **记录**：纯查询，零依赖 → 最早可用
- **提现账户**：绑定+复用 → 只依赖KYC
