## 1. 模块全景与子模块拆解

### 1.1 我方负责的两大模块

```
ser-wallet（钱包服务）
├── 子模块A：钱包核心（余额管理 + 资金操作原语）
│     CreditWallet / DebitWallet / FreezeBalance / UnfreezeBalance / ConfirmDebit / GetBalance / InitWallet
│     → 这是整个资金体系的"原子操作层"，所有资金变动都经过这里
│
├── 子模块B：充值（C端）
│     充值方式展示 → 创建充值订单 → 接收回调上账
│     → 涉及联动：财务（通道匹配）、币种配置（汇率）
│
├── 子模块C：提现（C端）
│     提现方式展示 → 创建提现订单 → 冻结余额 → 等待审核结果
│     → 涉及联动：财务（提现方式配置、出款）、KYC（认证校验）、币种配置（汇率）
│
├── 子模块D：兑换（C端）
│     兑换信息查询 → 执行兑换（法币→BSB）
│     → 涉及联动：币种配置（汇率）
│
├── 子模块E：记录（C端）
│     充值记录 / 提现记录 / 兑换记录 / 奖励记录
│     → 纯读取，基本独立
│
├── 子模块F：提现账户（C端）
│     账户保存 / 列表 / 删除
│     → 完全独立
│
└── 子模块G：币种配置（B端管理）
      币种CRUD / 汇率管理 / 汇率日志 / 定时任务
      → 涉及联动：S3（图标上传）、三方汇率API、审计日志
```

### 1.2 子模块间的调用关系

```
            ┌──────────────────────────────┐
            │   子模块A：钱包核心（原语层）   │
            │   Credit / Debit / Freeze... │
            └──┬────┬────┬────┬────┬───────┘
               │    │    │    │    │
    ┌──────────┘    │    │    │    └──────────┐
    ▼               ▼    ▼    ▼               ▼
 子模块B         子模块C  子模块D         外部模块
 (充值)          (提现)   (兑换)         (财务RPC调用)
    │               │       │
    │               │       │
    ▼               ▼       ▼
 子模块E：记录（读取充值/提现/兑换产生的订单和流水数据）

 子模块G：币种配置 ← 被子模块B/C/D调用获取汇率和币种信息
```

**核心认知**：子模块A（钱包核心）是所有资金操作的统一入口。无论是C端用户操作（充值/提现/兑换），还是外部模块RPC调用（财务上账/扣款/冻结），最终都汇聚到子模块A的原子操作函数中执行。

## 2. 核心数据流向总览

### 2.1 资金进入平台（入金方向）

```
用户充值请求
  │
  ├── 法币充值 ─────→ 财务匹配通道 → 三方支付 → 回调 → CreditWallet(中心钱包) + 流水
  │
  ├── USDT充值BSB ──→ 财务匹配通道 → 链上转账 → 回调 → 汇率换算 → CreditWallet(中心钱包) + 流水
  │
  └── 充值 + 选择活动奖金 → 同上 + 额外 CreditWallet(奖励钱包) + 流水

人工加款（B端审核通过）
  └── 财务 RPC → CreditWallet(指定钱包类型 × 最多4个) + 流水

充值补单（B端审核通过）
  └── 财务 RPC → CreditWallet(中心钱包) + 流水
```

### 2.2 资金离开平台（出金方向）

```
用户提现请求
  │
  ├── 创建提现订单 → FreezeBalance(冻结可用余额)
  │
  ├── 审核通过 + 出款成功 → ConfirmDebit(正式扣减冻结金额) + 流水
  │
  └── 审核驳回 / 出款失败 → UnfreezeBalance(解冻恢复余额)

人工减款（B端审核通过）
  └── 财务 RPC → DebitWallet(指定钱包类型 × 最多4个) + 流水
```

### 2.3 资金内部流转（不涉及出入金）

```
法币兑换BSB
  │
  ├── DebitWallet(法币-中心钱包, 兑换金额)
  │
  ├── CreditWallet(BSB-中心钱包, 兑换后BSB金额)
  │
  └── CreditWallet(BSB-奖励钱包, 赠送BSB金额)  ← 有赠送时

游戏投注扣款（未来扩展）
  └── DebitWallet(当前币种, 投注金额)  优先中心钱包 → 不足扣奖励钱包

游戏返奖（未来扩展）
  └── CreditWallet(当前币种, 奖金进奖励钱包 / 本金返中心钱包)
```

### 3.2 充值流程的关键处理思路

**第一步：展示可用充值方式**
- 思路：充值方式的配置（名称/图标/档位/范围/关联通道）由财务模块管理
- 处理：钱包服务 RPC 调用财务获取配置 → 根据用户状态（游客/登录）过滤 → 返回前端
- 依据：需求 5-1 充值方式总览 — "游客仅可用USDT充值"

**第二步：防重复提交**
- 思路：同一用户短时间内相同币种相同金额的充值需要拦截
- 处理：查询近期订单（Redis 或 DB），1-2笔返回警告标记让前端弹窗，≥3笔直接拦截
- 依据：需求 5-5 重复转账拦截规则

**第三步：汇率换算（仅BSB充值）**
- 思路：USDT充值BSB时需要按入款汇率换算实际到账BSB数量
- 处理：调币种配置 GetExchangeRate 获取入款汇率
- 依据：需求 5-2 USDT充值 — "仅BSB充值界面显示汇率换算"
- 法币充值法币钱包不需要换算（1:1直充）

**第四步：匹配通道创建订单**
- 思路：这是充值流程最核心的跨模块交互点
- 处理：调财务 MatchChannel → 匹配成功得到通道信息和三方URL → 创建平台订单
- 失败处理：无可用通道 → 返回"系统繁忙，请更换充值方式"
- 依据：财务需求 2-3 充值流程 + 2-1 通道轮询规则

**第五步：回调上账（核心资金操作）**
- 思路：这是资金安全的关键节点，必须幂等 + 事务原子性
- 处理：验签 → 幂等检查（订单状态非待支付就返回成功不重复上账）→ 验金额 → 事务内执行 CreditWallet + 写流水 + 更新状态
- 关键约束：CreditWallet 在此场景是**内部函数调用**（同进程同事务），不是 RPC
- 依据：需求 9 非功能性需求 — "充值回调必须幂等"、"账务流水与余额强一致"

### 3.3 充值回调的归属边界问题

> **待评审确认**：充值回调是由钱包服务直接接收三方回调，还是由财务服务接收后 RPC 调钱包上账？
>
> 方案A：钱包直接收回调 → 验签密钥需要从财务同步 → 钱包可在同一事务中完成上账
> 方案B：财务收回调 → 财务验签后 RPC 调 CreditWallet → 跨服务调用，需保证最终一致性
>
> 两种方案各有利弊，需要与财务开发者协商确认

### 4.2 提现流程的关键处理思路

**第一阶段：展示提现方式（我方处理）**
- 并行调用：ser-kyc 查认证状态 + 财务获取提现方式列表
- 根据 KYC 状态过滤：未认证 → 仅 USDT；已认证 → 全部
- BSB 币种时额外调币种配置获取出款汇率用于展示
- 依据：需求 7-1 提现模块

**第二阶段：创建提现订单（我方处理，核心操作）**
- 多重校验链：参数范围 → KYC二次校验 → 账户归属 → 姓名一致性 → 余额充足
- 冻结余额：FreezeBalance 是**内部函数调用**，在创建订单的同一事务中执行
- 扣款优先级：先冻结中心钱包的余额，中心不足再冻结奖励钱包（需求 2-1 资金钱包体系）
- 创建订单后状态为"待风控审核"，后续审核流程由财务模块接管
- 依据：需求 7-5 提现限制、7-2 银行卡提现

**第三阶段：审核与出款（财务处理，通过 RPC 回调我方）**
- 三种结果：
  - 出款成功 → 财务 RPC 调 ConfirmDebit → 冻结金额正式扣减 + 写资金流水
  - 审核驳回 → 财务 RPC 调 UnfreezeBalance → 解冻恢复可用余额
  - 出款失败 → 财务 RPC 调 UnfreezeBalance → 同上
- 关键约束：ConfirmDebit 和 UnfreezeBalance 是 **RPC 接口**（由财务跨服务调用），需保证幂等

**提现流程的职责边界**：
- 我方职责：展示 → 校验 → 冻结 → 创建订单（到"待风控审核"状态为止）
- 财务职责：风控审核 → 财务审核 → 提交出款 → 接收出款结果 → RPC 回调我方
- 清晰的分界点：提现订单创建完成（状态=待风控审核）后，控制权转移给财务

```
 │◀── 展示兑换信息 ────────┤                            │
 │    汇率/可用余额/赠送   │                            │
 │                        │                            │
 ├── 输入兑换金额          │                            │
 ├── 确认兑换 ────────────▶│                            │
 │                        │                            │
 │                        ├── 再次获取汇率 ────────────▶│
 │                        │   (防页面停留期间汇率变动)    │
 │                        │◀── 返回最新汇率 ───────────┤
 │                        │                            │
 │                        ├── [计算兑换结果]             │
 │                        │   BSB主金额=法币÷汇率       │
 │                        │   BSB赠送=BSB主×赠送比例     │
 │                        │                            │
 │                        ├── [事务:跨币种原子操作]      │
 │                        │   ① DebitWallet(法币-中心, 兑换额)
 │                        │   ② CreditWallet(BSB-中心, BSB主额)
 │                        │   ③ CreditWallet(BSB-奖励, BSB赠送) ← 有赠送时
 │                        │   ④ 写3条资金流水
 │                        │   ⑤ 创建兑换记录
 │                        │                            │
 │                        ├── [审计日志] → ser-blog     │
 │                        │                            │
 │◀── 兑换成功 ────────────┤                            │
 │    扣除法币/获得BSB/赠送 │                            │
```

### 5.2 兑换流程的关键处理思路

**核心特征：完全自主，不依赖财务**
- 兑换是钱包模块内部完成的流程，不涉及三方通道，不需要财务模块参与
- 唯一外部依赖是币种配置的汇率查询

**事务设计要点**：
- 涉及两个币种的钱包表（法币钱包 + BSB钱包），必须在同一数据库事务中完成
- 操作顺序：先扣法币 → 再加BSB → 再加赠送 → 写流水 → 创建记录
- 任何一步失败整体回滚
- 依据：需求 9 非功能性需求 — "账务流水与余额强一致"

**兑换方向限制**：
- 仅支持 法币 → BSB（单向）
- BSB 不能兑换回法币
- 依据：需求 6 兑换模块 — "仅法币钱包有兑换入口"

**赠送奖金处理**：
- 赠送部分进入 BSB 奖励钱包（非中心钱包）
- 赠送部分有稽核流水要求（X倍流水后方可提现）
- 依据：需求 6 兑换模块 — "赠送部分需完成X倍流水后才可提现"

## 8. 币种配置管理流程

### 8.1 币种 CRUD 流程

```
B端管理员操作
  │
  ├── 新增币种:
  │     ① 校验币种代码唯一性
  │     ② 上传图标 → ser-s3.UploadIcon → 返回URL
  │     ③ 写入币种配置表(默认禁用)
  │     ④ 审计日志 → ser-blog
  │
  ├── 编辑币种:
  │     ① 查当前配置(记录变更前值)
  │     ② [有图标变更] 上传新图标 → ser-s3
  │     ③ 更新配置(汇率浮动/偏值/图标/千分位/小数位)
  │     ④ 重算入款/出款汇率 = 实时汇率 × (1±浮动%)
  │     ⑤ 审计日志(记录变更前后差异)
  │
  ├── 启用/禁用:
  │     ① 更新状态
  │     ② 清除该币种缓存(其他服务下次请求时刷新)
  │     ③ 审计日志
  │     影响：禁用后C端不可见，充值/提现/兑换均不可用
  │
  └── 设置基准币种:
        ① 检查是否已设置(已设置→拒绝，不可更改)
        ② 写入基准币种配置
        ③ 审计日志
```

**处理思路**：
- 币种的基础信息（代码/名称/类型）创建后不可修改，只能修改运营参数
- 编辑汇率浮动时要联动重算入款/出款汇率
- 禁用操作影响范围广，需要清除缓存让变更即时生效
- 基准币种是一次性设置，设置后锚定为全局计算基准
- 依据：需求 3-3 后台配置页面说明

## 9. 汇率管理全流程

### 9.1 汇率数据流全景

```
三方汇率API-1 ──┐
三方汇率API-2 ──┼── 并发获取 → 取均值 → 实时汇率
三方汇率API-3 ──┘
                                    │
                                    ▼
                            计算偏差值
                  |新实时汇率 - 当前平台汇率|
                  ─────────────────────── × 100%
                        新实时汇率
                                    │
                        ┌───────────┴───────────┐
                        ▼                       ▼
                  偏差 < 阈值               偏差 ≥ 阈值
                        │                       │
                  仅更新实时汇率         更新平台汇率 + 入款/出款汇率
                        │                       │
                        │              入款 = 实时 × (1 + 浮动%)
                        │              出款 = 实时 × (1 - 浮动%)
                        │                       │
                        └───────────┬───────────┘
                                    ▼
                             写入汇率日志
                        (触发原因/变动前后/时间/操作人)
```

### 9.2 触发方式

| 触发方式 | 频率 | 操作人 | 说明 |
|----------|------|--------|------|
| 定时任务 | 分钟级 | 系统 | 自动拉取并检测偏差 |
| 手动刷新 | B端管理员触发 | 管理员 | 即时更新指定或全部币种汇率 |
| 编辑币种浮动 | B端编辑触发 | 管理员 | 修改浮动%后立即重算入款/出款 |

### 9.3 定时任务处理思路

```
Cron 触发(分钟级)
  │
  ├── 查所有启用状态的币种(需要汇率更新的)
  │
  ├── 对每个币种:
  │     ├── 并发调3家三方汇率API
  │     ├── 取有效值均值(排除异常值)
  │     ├── 偏差检测 → 决定是否更新平台汇率
  │     ├── 更新币种配置表
  │     └── 写汇率日志
  │
  └── 任务结束
```

**思路**：
- 定时拉取和手动刷新**复用同一函数**，仅触发方式和操作人不同
- 三方API调用应有超时和降级（某家不可用时用剩余的取均值）
- 偏差阈值是每个币种独立配置的（由B端编辑的"偏值"字段决定）
- 依据：需求 4-2 汇率规则

## 10. 被调用方流程 — RPC 资金操作原语

> 以下是钱包作为**被调用方**，接收其他模块（主要是财务）RPC 请求时的处理流程
> 这些是子模块A（钱包核心）的原语操作，是整个资金体系的基石

### 10.1 CreditWallet（上账/加款）

```
RPC请求(用户ID / 币种 / 钱包类型 / 金额 / 业务类型 / 关联订单号)
  │
  ├── [校验] 参数合法性 + 金额>0 + 币种启用
  │
  ├── [查询] 目标钱包是否存在
  │     不存在 → 返回错误(钱包未初始化)
  │
  ├── [事务]
  │     ├── 余额 += 金额
  │     ├── 写资金流水(方向=入, 业务类型, 变动前→变动后, 关联订单号)
  │     └── (可选)删除余额缓存
  │
  └── 返回: 操作后余额
```

**调用方与场景**：
- 财务 → 充值成功上账（中心钱包）
- 财务 → 活动奖金发放（奖励钱包）
- 财务 → 充值补单审核通过
- 财务 → 人工加款审核通过（可能同时对4个钱包类型调用4次）

### 10.2 DebitWallet（扣款/减款）

```
RPC请求(用户ID / 币种 / 钱包类型 / 金额 / 业务类型 / 关联订单号)
  │
  ├── [校验] 参数合法性 + 金额>0
  │
  ├── [查询] 目标钱包余额
  │     余额不足 → 返回错误(余额不足)
  │
  ├── [事务]
  │     ├── 余额 -= 金额
  │     ├── 写资金流水(方向=出, 业务类型, 变动前→变动后)
  │     └── 删除余额缓存
  │
  └── 返回: 操作后余额
```

**调用方与场景**：
- 财务 → 人工减款审核通过（可能对4个钱包类型分别扣款）
- 未来：游戏 → 投注扣款

### 10.3 FreezeBalance（冻结）

```
RPC请求(用户ID / 币种 / 金额 / 关联订单号)
  │
  ├── [查询] 可用余额（中心+奖励 - 已冻结）
  │     可用 < 金额 → 返回错误(余额不足)
  │
  ├── [事务]
  │     ├── 按优先级冻结: 先中心钱包 → 不足部分冻结奖励钱包
  │     ├── 可用余额 -= 金额
  │     ├── 冻结金额 += 金额
  │     ├── 写资金流水(方向=冻结, 关联订单号)
  │     └── 记录冻结明细(哪个钱包冻了多少，解冻时需要)
  │
  └── 返回: 冻结成功 + 冻结明细
```

**调用方与场景**：
- 提现流程 → 创建提现订单时（此时可能是内部调用也可能是RPC，取决于架构设计）

### 10.4 UnfreezeBalance（解冻）

```
RPC请求(用户ID / 币种 / 金额 / 关联订单号)
  │
  ├── [查询] 冻结明细(按关联订单号查)
  │     无冻结记录 → 返回错误
  │
  ├── [幂等检查] 是否已解冻 → 已解冻直接返回成功
  │
  ├── [事务]
  │     ├── 按冻结明细反向解冻(中心冻了多少还多少，奖励同理)
  │     ├── 冻结金额 -= 金额
  │     ├── 可用余额 += 金额
  │     └── 写资金流水(方向=解冻, 关联订单号)
  │
  └── 返回: 解冻成功
```

**调用方与场景**：
- 财务 → 提现审核驳回
- 财务 → 出款失败

### 10.5 ConfirmDebit（确认扣款）

```
RPC请求(用户ID / 币种 / 金额 / 关联订单号)
  │
  ├── [查询] 冻结明细(按关联订单号查)
  │     无冻结记录 → 返回错误
  │
  ├── [幂等检查] 是否已确认扣款 → 已扣款直接返回成功
  │
  ├── [事务]
  │     ├── 冻结金额 -= 金额(正式扣减)
  │     ├── 写资金流水(方向=出, 业务类型=提现, 变动前→变动后)
  │     └── 更新冻结明细状态=已确认
  │
  └── 返回: 扣款成功
```

**调用方与场景**：
- 财务 → 三方出款成功回调后

### 10.6 GetBalance（余额查询）

```
RPC请求(用户ID / 币种)
  │
  ├── [查询] 各子钱包余额
  │     优先走Redis缓存 → miss时查DB → 写缓存
  │
  └── 返回:
        ├── 中心钱包余额
        ├── 奖励钱包余额
        ├── 冻结金额
        └── 资金余额(中心+奖励)
```

**调用方与场景**：
- 财务 → 人工加减款页面展示用户当前余额

### 10.7 InitWallet（钱包初始化）

```
RPC请求(用户ID)
  │
  ├── [查询] 是否已初始化 → 已初始化直接返回成功(幂等)
  │
  ├── [内部] 为该用户创建所有启用币种的钱包记录
  │     对每个启用币种:
  │       创建中心钱包记录(余额=0)
  │       创建奖励钱包记录(余额=0)
  │       (预留)创建其他钱包类型记录
  │
  └── 返回: 初始化成功
```

**调用方与场景**：
- ser-user → 用户注册成功后调用

### 13.2 数据表归属关系

| 子模块 | 拥有的数据表 | 读取其他子模块的表 |
|--------|------------|------------------|
| A-钱包核心 | 钱包表(wallet)、资金流水表(wallet_flow)、冻结明细表(freeze_detail) | 无 |
| B-充值 | 充值订单表(recharge_order) | 钱包表(余额变更通过子模块A) |
| C-提现 | 提现订单表(withdraw_order) | 钱包表(冻结通过子模块A)、提现账户表(子模块F) |
| D-兑换 | 兑换记录表(exchange_record) | 钱包表(扣款加款通过子模块A) |
| E-记录 | 无（纯读取） | 充值订单/提现订单/兑换记录/资金流水（只读） |
| F-提现账户 | 提现账户表(withdraw_account) | 无 |
| G-币种配置 | 币种配置表(currency_config)、汇率日志表(rate_log) | 无 |

### 13.3 共享的公共能力

所有子模块共享以下公共能力：
- **审计日志**：所有写操作 → ser-blog.AddActLog(oneway)
- **统一响应**：ret.BizErr / ret.HttpOne
- **链路追踪**：tracer.GetUserId / tracer.GetTraceID
- **错误码**：6022xxx 格式
- **缓存**：Redis 缓存余额、币种列表、汇率

## 14. 跨模块联动全景

### 14.1 我方调用他方（出向调用）

```
ser-wallet ──→ 财务服务
  ├── GetRechargeMethodList(币种)        ← 充值时获取充值方式配置
  ├── GetWithdrawMethodList(币种)        ← 提现时获取提现方式配置
  ├── MatchChannel(币种/方式/金额)       ← 充值时匹配代收通道
  └── SubmitPayout(订单/金额/账户)       ← 提现审核通过后提交出款(待确认边界)

ser-wallet ──→ 币种配置（本服务内部或RPC）
  ├── GetCurrencyList()                  ← 钱包页面币种列表
  ├── GetExchangeRate(币种对/类型)       ← 充值/提现/兑换时获取汇率
  ├── GetCurrencyDetail(币种)            ← 金额格式化展示规则
  └── GetBaseCurrency()                  ← 基准币种信息

ser-wallet ──→ ser-kyc
  └── GetKycStatus(用户ID)              ← 提现前校验KYC状态

ser-wallet ──→ ser-blog
  └── AddActLog(oneway, 异步)           ← 所有写操作的审计日志

币种配置 ──→ ser-s3
  └── UploadIcon(图标文件)              ← 新增/编辑币种时上传图标

币种配置 ──→ 三方汇率API × 3
  └── HTTP GET 汇率数据                 ← 定时任务 + 手动刷新
```

### 14.2 他方调用我方（入向调用）

```
财务服务 ──→ ser-wallet
  ├── CreditWallet(上账)                ← 充值成功/补单/人工加款
  ├── DebitWallet(扣款)                 ← 人工减款
  ├── FreezeBalance(冻结)               ← 提现创建(待确认边界)
  ├── UnfreezeBalance(解冻)             ← 提现驳回/出款失败
  ├── ConfirmDebit(确认扣款)            ← 出款成功
  └── GetBalance(余额查询)              ← 人工修正页面展示余额

财务服务 ──→ 币种配置
  ├── GetCurrencyList()                 ← 币种筛选下拉
  ├── GetExchangeRate()                 ← 报表汇率换算
  └── BatchGetExchangeRate()            ← 批量报表换算

ser-user ──→ ser-wallet
  └── InitWallet(用户ID)               ← 用户注册时初始化钱包
```

### 14.3 联动频率与关键性

| 调用关系 | 频率 | 关键性 | 失败影响 |
|----------|------|--------|---------|
| 钱包 → 财务(充值方式) | 中频(C端每次进充值页) | 高 | 用户无法看到充值方式 |
| 钱包 → 财务(匹配通道) | 中频(每次充值提交) | **最高** | 用户无法完成充值 |
| 钱包 → 币种配置(汇率) | 高频(充值/提现/兑换) | 高 | 无法展示汇率/无法计算金额 |
| 钱包 → ser-kyc | 低频(提现时) | 中 | 无法校验KYC（可降级为允许） |
| 财务 → 钱包(CreditWallet) | 中频(充值成功时) | **最高** | 用户充值不到账 |
| 财务 → 钱包(Freeze/Unfreeze) | 低频(提现审核时) | 高 | 提现金额无法冻结/解冻 |
| 币种配置 → 三方API | 分钟级 | 中 | 汇率不更新（用旧值） |
| 全部 → ser-blog | 高频 | 低 | 审计日志缺失（不影响业务） |

## 15. 流程优先级与开发顺序建议

### 15.1 基于依赖关系的开发优先级

```
第一优先级（无外部依赖，独立开发）
├── 子模块A：钱包核心（CreditWallet/DebitWallet/Freeze/Unfreeze/ConfirmDebit/GetBalance/InitWallet）
│     理由：所有其他子模块和外部调用都依赖这一层
│     占比：约 7 个 RPC 接口
│
├── 子模块G：币种配置核心（CRUD + RPC查询）
│     理由：汇率和币种数据是充值/提现/兑换的前置依赖
│     占比：约 7 个 B 端接口 + 4 个 RPC 接口
│
└── 子模块F：提现账户管理
      理由：完全独立，无任何外部依赖
      占比：约 3 个 C 端接口

第二优先级（依赖第一优先级 + 币种配置）
├── 子模块D：兑换
│     理由：仅依赖子模块A（钱包核心）+ 子模块G（汇率），不依赖财务
│     占比：约 2 个 C 端接口
│
├── 子模块G：汇率定时任务
│     理由：依赖三方 API，但核心逻辑在内部
│     占比：1-2 个定时任务
│
└── 子模块E：记录查询
      理由：依赖子模块B/C/D的订单表，但表结构可以先定义
      占比：约 2 个 C 端接口

第三优先级（依赖财务模块）
├── 子模块B：充值
│     理由：强依赖财务的充值方式配置和通道匹配
│     需要：与财务协商接口契约
│     占比：约 5 个 C 端接口
│
└── 子模块C：提现
      理由：强依赖财务的提现方式配置 + KYC + 审核流程
      需要：与财务和 KYC 协商接口契约
      占比：约 4 个 C 端接口
```

## 16. 关键设计要点总结

### 16.1 资金安全设计要点

| 要点 | 处理思路 | 依据 |
|------|---------|------|
| 余额与流水强一致 | 余额变更和流水写入必须在同一数据库事务中 | 需求 9 非功能性需求 |
| 充值回调幂等 | 以订单状态为判据：非"待支付"状态直接返回成功 | 需求 9 非功能性需求 |
| 提现冻结→确认/解冻 | 三阶段操作（冻结→等待→确认/解冻），每阶段都幂等 | 提现流程需求 |
| 消费扣款优先级 | 先扣中心钱包，不足再扣奖励钱包 | 需求 2-1 资金钱包体系 |
| 跨币种事务 | 兑换操作涉及两个币种的钱包，必须原子性 | 兑换模块需求 |
| RPC 调用幂等 | CreditWallet/DebitWallet/Freeze/Unfreeze/ConfirmDebit 全部幂等 | 分布式系统基本要求 |

### 16.2 架构设计要点

| 要点 | 处理思路 |
|------|---------|
| 子模块A的双重身份 | CreditWallet 等既是 RPC 接口（被财务调用），也是内部函数（被充值/提现/兑换在同事务中调用） |
| 币种配置的部署方式 | 可以与钱包在同一服务内（减少 RPC 开销），也可以独立部署（更好的隔离），待评审确认 |
| 充值回调归属 | 回调是钱包直接收还是财务收后调钱包？影响事务边界，待协商 |
| 缓存策略 | 余额用短TTL缓存（5min），币种列表用中TTL（1h），汇率用短TTL（1-5min） |
| 审计日志异步 | oneway 调用 ser-blog，不阻塞主流程，不影响事务 |

### 16.3 跨模块协作要点

| 要点 | 我方处理 | 对方处理 |
|------|---------|---------|
| 充值上账 | 提供 CreditWallet RPC，保证幂等和事务性 | 财务在回调验证通过后调用 |
| 提现冻结 | 提供 FreezeBalance/UnfreezeBalance/ConfirmDebit | 财务在审核各阶段调用 |
| 人工修正 | 提供 CreditWallet/DebitWallet/GetBalance | 财务在审核通过后调用 |
| 充值方式 | 调财务 GetRechargeMethodList + 过滤 | 财务维护配置数据 |
| 提现方式 | 调财务 GetWithdrawMethodList + KYC过滤 | 财务维护配置数据 |
| 汇率查询 | 币种配置提供 GetExchangeRate | 财务统计时也调同一接口 |
