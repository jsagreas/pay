# 一、整体流程全景

## 1.1 你要实现的两个模块定位

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  币种配置模块（B端）                                      │
│  ─────────────                                          │
│  纯管理 + 纯数据提供方                                    │
│  你配好数据，别的模块来读                                   │
│  不依赖任何其他模块                                       │
│                                                         │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  钱包模块（C端 + RPC服务提供方）                            │
│  ────────────────────────                                │
│  ① C端：用户直接操作（充值/兑换/提现/记录/余额）              │
│  ② RPC服务方：财务模块回调你（入账/扣款/解冻/加减款）          │
│  你既是主动调用方，也是被动服务方                             │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 1.2 你的模块与外部的交互全景

你的代码与外部有以下交互点，按方向分：

**你主动调用别人（出方向）**：
- 钱包 → 币种配置：拿币种列表、汇率、配置参数（自己模块内部的RPC，也可能走内部方法调用）
- 钱包 → 财务：获取充值/提现方式、创建渠道支付订单、提交提现审核
- 钱包 → 用户服务：KYC状态查询、KYC信息查询

**别人调用你（入方向）**：
- 财务 → 钱包：充值入账、提现确认扣款、提现失败解冻、人工加款、人工减款、人工补单、余额查询

**你独立闭环（不涉及外部）**：
- 兑换（只需汇率，内部完成）
- 记录查询（读本地表）
- 余额查询（读本地表）
- 币种配置全部B端操作（读写本地表）

## 1.3 六条核心业务流程线

你要实现的所有功能，归纳为6条主线：

```
流程线1：币种配置管理 → B端运营配置币种和汇率（纯B端，最基础）
流程线2：钱包首页+余额 → C端用户查看余额、切换币种（纯读取）
流程线3：充值 → C端用户充值法币/USDT到钱包（跨模块：钱包↔财务↔渠道）
流程线4：兑换 → C端用户法币/USDT兑换BSB（内部闭环，最简单的交易链路）
流程线5：提现 → C端用户从钱包提现到银行卡/电子钱包/USDT（跨模块：钱包↔财务↔用户服务↔渠道）
流程线6：被调用 → 财务模块回调/人工修正操作你的余额（RPC服务提供方）
```

下面逐条展开。

# 二、流程线1：币种配置管理（B端）

## 2.1 总体思路

币种配置是整个体系的地基。它的流程最简单：**B端运营人员在后台管理界面做增删改查，数据存到DB，其他模块通过RPC读取**。

没有复杂的状态流转，没有跨模块写操作，没有异步回调。你要做的就是把CRUD做好，把RPC查询接口提供好。

## 2.2 币种管理流程

```
B端运营人员
    │
    ├─[新增币种]
    │   │
    │   ├── 前端提交: 币种名称、代码、图标、精度（小数位数）、
    │   │            充值限额（最低/最高）、提现限额（单笔/每日）、
    │   │            手续费规则（固定/百分比）、T+N到账天数、排序权重
    │   │
    │   ├── 后端处理:
    │   │   ├── 1. 参数校验（必填项、格式、精度范围合理性）
    │   │   ├── 2. 唯一性校验（币种代码不能重复）
    │   │   ├── 3. 写入币种配置表（默认状态=禁用，需手动启用）
    │   │   └── 4. 写操作日志（操作人、操作类型=新增、时间、完整新值）
    │   │
    │   └── 返回: 成功/失败
    │
    ├─[编辑币种]
    │   │
    │   ├── 前端提交: 币种ID + 修改的字段
    │   │
    │   ├── 后端处理:
    │   │   ├── 1. 查询旧值（日志需要记录改前改后）
    │   │   ├── 2. 参数校验
    │   │   ├── 3. 更新币种配置表
    │   │   └── 4. 写操作日志（操作人、操作类型=编辑、改前值、改后值）
    │   │
    │   └── 注意: 修改精度、限额等参数，C端下次查询时立即生效
    │         不需要主动通知钱包，钱包每次用都会查最新值
    │
    ├─[启用/禁用币种]
    │   │
    │   ├── 后端处理:
    │   │   ├── 1. 更新状态字段
    │   │   └── 2. 写操作日志
    │   │
    │   └── 业务影响: 禁用后，C端获取可用币种列表时拿不到这个币种
    │         已有该币种余额的用户不受影响（余额还在），只是不能操作了
    │
    └─[币种列表查询]
        │
        └── 后端处理: 分页查询币种配置表，返回全部字段 + 状态
```

## 2.3 汇率管理流程

```
B端运营人员
    │
    ├─[设置/修改汇率]
    │   │
    │   ├── 前端提交: 币种代码 + 新汇率值（格式："1 BSB = X 法币"）
    │   │
    │   ├── 后端处理:
    │   │   ├── 1. 参数校验（汇率值 > 0，币种存在）
    │   │   ├── 2. 查询旧汇率（日志用）
    │   │   ├── 3. 更新汇率配置表
    │   │   └── 4. 写汇率变更日志（币种、旧值、新值、操作人、时间）
    │   │
    │   └── 业务影响: 下次有人调汇率查询接口就拿到新值
    │         正在进行中的兑换/充值不受影响（它们在创建时已经锁定了当时的汇率）
    │
    ├─[汇率查询（B端页面）]
    │   └── 后端处理: 查询所有币种的当前汇率，展示在管理页面
    │
    └─[汇率变更日志查询]
        └── 后端处理: 分页查询汇率日志表，支持按币种、时间范围筛选
```

## 2.4 RPC暴露流程（被其他模块调用）

你暴露的4个RPC读接口，处理流程都很简单：

```
其他模块 ──RPC──→ 币种配置服务
    │
    ├── 获取可用币种列表: 查DB where status=启用，按排序字段排序，返回列表
    ├── 获取币种详情: 查DB where 币种ID/代码，返回全字段
    ├── 获取汇率: 查DB where 币种代码，返回汇率值
    └── 获取配置参数: 查DB where 币种代码，返回精度/限额/手续费/T+N
```

**设计考量**：这4个接口的调用频率会很高（钱包每次操作都可能调），需考虑**缓存策略**：
- 币种列表和详情变更频率极低（运营偶尔改一次），适合较长缓存（如Redis 5-10分钟）
- 汇率变更频率稍高但也不频繁，适合中等缓存（如Redis 1-3分钟）
- 缓存更新时机：B端修改操作成功后主动刷新缓存

# 三、流程线2：钱包首页+余额（C端）

## 3.1 总体思路

用户打开钱包页面，看到的是**当前选中币种下的余额信息**。这是纯读操作，不涉及任何写操作和外部依赖。

## 3.2 钱包首页加载流程

```
用户打开钱包/个人中心页面
    │
    ├── C端发起两个并行请求:
    │
    │   ┌──[请求1] 可用币种列表查询
    │   │
    │   │   后端处理:
    │   │   ├── 1. [RPC/内部] → 币种配置.获取可用币种列表（status=启用的币种）
    │   │   ├── 2. [本地DB] 查询该用户在每个币种下的余额概览
    │   │   │      （4个子钱包的总余额合计，按币种分组）
    │   │   └── 3. 拼装返回: [{币种代码, 币种名称, 图标, 余额合计}, ...]
    │   │
    │   │   用途: C端展示币种切换弹窗，让用户看到"VND余额xxx / BSB余额xxx / ..."
    │   │
    │   └──[请求2] 当前币种余额详情
    │
    │       后端处理:
    │       ├── 1. 从请求上下文获取当前选中币种（C端会传）
    │       ├── 2. [本地DB] 查询该用户在该币种下的4个子钱包余额:
    │       │      中心钱包.可用余额 / 中心钱包.冻结余额
    │       │      奖励钱包.余额
    │       │      代理钱包.余额
    │       │      主播钱包.余额
    │       └── 3. 返回4个子钱包的余额信息
    │
    │       用途: C端主页面展示各子钱包余额
    │
    └── C端渲染页面:
        ├── 顶部: 当前币种名称+图标，币种切换入口
        ├── 余额区域: 4个子钱包余额
        └── 功能按钮区域:
            ├── BSB钱包: "充值" "提现"
            └── 法币钱包: "充值" "兑换" "提现"
```

## 3.3 币种切换流程

```
用户点击币种切换 → 弹出币种列表 → 选择新币种
    │
    ├── C端行为: 切换当前币种上下文（本地状态），重新请求余额详情
    │
    ├── 后端处理: 同请求2（余额详情），只是币种参数变了
    │
    └── C端行为: 整体刷新页面
        ├── 余额变为新币种的余额
        ├── 后续进入充值/提现页面，方式列表也是新币种的
        └── 记录页面也切换为新币种的记录
```

**关键规则（需求文档4-2）**：币种切换 = 整体上下文切换。不存在"VND钱包里看BSB记录"这种场景。C端传什么币种，后端就返回什么币种的数据。

# 四、流程线3：充值（C端，跨模块）

## 4.1 总体思路

充值是**钱包发起、财务执行、渠道完成、财务回调、钱包入账**的跨模块流程。你的代码要处理"发起"和"入账"两头，中间的渠道对接交给财务。

充值流程分5个阶段：进入页面 → 预校验 → 创建订单 → 等待支付 → 入账。

## 4.2 阶段一：进入充值页面

```
用户点击"充值"按钮
    │
    ├── C端并行发起:
    │
    │   ┌──[请求1] 获取可用充值方式列表
    │   │
    │   │   后端处理:
    │   │   ├── 1. [RPC] → 财务.支付配置查询
    │   │   │      入参: 当前币种 + 类型=充值 + 端=App/H5/Web
    │   │   │      返回: [{方式名称, 图标, 金额范围, 快捷金额, 排序}, ...]
    │   │   │
    │   │   └── 2. 直接透传返回C端
    │   │
    │   │   如果财务服务不可用: 返回空列表或降级提示"充值暂不可用"
    │   │   这是一个强依赖——没有财务的支付配置数据，充值页面展示不出任何充值方式
    │   │
    │   └──[请求2] 可选奖金活动列表（如果页面有奖金入口）
    │
    │       后端处理:
    │       ├── 1. 查询当前可参与的充值奖金活动配置
    │       │      (活动模块或本地配置，取决于架构决策)
    │       └── 2. 返回: [{活动名称, 奖金比例, 最低充值门槛, 流水倍数}, ...]
    │
    └── C端渲染:
        ├── 充值方式列表（USDT / 银行转账 / 电子钱包 等）
        ├── 快捷金额标签（如 100, 500, 1000, 5000）
        └── 奖金选择入口
```

## 4.3 阶段二：用户选择方式、输入金额、点击确认

这里是充值的**关键决策点和预校验阶段**。

```
用户选择了充值方式、输入了金额、可能选了奖金活动
    │
    ├── C端发起: 创建充值订单
    │
    └── 后端处理（核心流程）:
        │
        ├── Step 1: 基础参数校验
        │   ├── 币种存在且启用？
        │   ├── 金额格式合法？（正数、精度符合币种配置）
        │   ├── 金额在该充值方式的允许范围内？
        │   └── 奖金活动ID有效？充值金额满足活动最低门槛？
        │
        ├── Step 2: 重复转账拦截（需求文档5-5）
        │   │
        │   │   查询条件: 本用户 + 本币种 + 同金额 + 状态=处理中 的充值订单数
        │   │
        │   ├── 0笔: 通过，继续
        │   ├── 1笔: 返回 code=DUPLICATE_WARN_1
        │   │        C端弹窗："您有一笔相同金额订单正在处理，是否继续？"
        │   │        用户点"继续" → C端带 force=true 重新请求
        │   ├── 2笔: 返回 code=DUPLICATE_WARN_2
        │   │        C端弹窗警告，建议修改金额
        │   │        用户仍可选择继续 → force=true
        │   └── ≥3笔: 返回 code=DUPLICATE_BLOCK
        │            C端强制拦截，用户必须改金额或换方式
        │
        │   注: 重复转账校验可作为独立接口，也可合并在创建订单接口内
        │       取决于前端交互设计——如果需要先校验再确认，就独立
        │       如果直接创建时校验并返回警告码，就合并
        │
        ├── Step 3: 币种配置参数校验
        │   ├── [RPC/内部] → 币种配置.获取配置参数
        │   └── 校验: 金额 ≥ 单笔最低, 金额 ≤ 单笔最高
        │
        │   注: Step 2 和 Step 3 可并行执行，互不依赖
        │
        ├── Step 4: 生成本地充值订单
        │   ├── 生成平台订单号（雪花ID）
        │   ├── 写入充值订单表:
        │   │   用户ID / 币种 / 金额 / 充值方式 / 状态=待支付
        │   │   平台订单号 / 创建时间
        │   │   奖金活动ID（如有）/ 奖金比例（如有）
        │   └── 此时用户余额不变（充值是先支付后入账，不是先扣后退）
        │
        ├── Step 5: 调用财务创建渠道订单
        │   ├── [RPC] → 财务.创建渠道支付订单
        │   │   入参: 用户ID, 币种, 金额, 支付方式ID, 平台订单号
        │   │
        │   │   财务内部处理（你不用管）:
        │   │   ├── 通道轮询匹配（成功率/时间/告警系数加权选择最优通道）
        │   │   ├── 调用第三方渠道API创建支付单
        │   │   └── 返回: 渠道订单号 + 支付信息
        │   │
        │   │   返回给你:
        │   │   ├── 渠道订单号（关联存储到你的订单表）
        │   │   └── 支付信息:
        │   │       ├── USDT充值: 收款地址 + 二维码URL + 链网络
        │   │       └── 银行/电子钱包: 第三方支付页面跳转URL
        │   │
        │   └── 如果RPC调用失败:
        │       ├── 本地订单标记状态=失败
        │       └── 返回C端错误提示"充值暂不可用，请稍后重试"
        │
        ├── Step 6: 更新本地订单
        │   └── 写入渠道订单号，状态保持=待支付
        │
        └── 返回C端: 支付信息
            ├── USDT: 展示收款地址 + 二维码，用户在外部钱包转账
            └── 其他: 跳转到第三方支付页面
```

## 4.4 阶段三：等待支付 + 充值入账

用户在第三方完成支付后，**你不会直接收到通知**。流程是：第三方 → 财务（回调）→ 钱包（RPC入账）。

```
[用户在第三方完成支付]
    │
    ▼
第三方渠道回调 → 财务模块（回调处理，你不管这段）
    │
    ▼
财务确认支付成功 → [RPC调用] → 你的钱包.充值入账接口
    │
    └── 你的后端处理（充值入账 — 你暴露的RPC接口）:
        │
        ├── Step 1: 幂等校验
        │   ├── 用平台订单号查询: 该订单是否已经入过账？
        │   ├── 已入账: 直接返回"成功"（幂等返回，不重复操作）
        │   └── 未入账: 继续
        │
        ├── Step 2: 订单状态校验
        │   ├── 查本地充值订单，确认状态=待支付
        │   └── 如果状态不对（已失败/已取消）: 返回错误
        │
        ├── Step 3: 事务操作（必须同一个DB事务）
        │   │
        │   │ ── BEGIN TRANSACTION ──
        │   │
        │   ├── 3a. 更新充值订单状态 → 成功
        │   │       补充: 渠道订单号、入账时间
        │   │
        │   ├── 3b. 中心钱包余额 + 充值金额
        │   │       UPDATE balance SET available = available + amount
        │   │       WHERE user_id=X AND currency=Y AND wallet_type=中心
        │   │
        │   ├── 3c. 如有奖金活动: 奖励钱包余额 + 奖金金额
        │   │       奖金 = 充值金额 × 奖金比例
        │   │       UPDATE balance SET available = available + bonus
        │   │       WHERE user_id=X AND currency=BSB AND wallet_type=奖励
        │   │       同时记录: 该笔奖金需完成X倍流水才能提现
        │   │
        │   ├── 3d. 写余额变动流水
        │   │       充值入账流水: 用户ID/币种/金额/变动类型=充值/订单号/变动前余额/变动后余额
        │   │       如有奖金: 再写一条奖金入账流水
        │   │
        │   │ ── COMMIT TRANSACTION ──
        │   │
        │   └── 事务保证: 订单状态更新 + 余额增加 + 流水记录 三者一致
        │       不能出现"余额加了但流水没写"或"流水写了但余额没加"
        │
        └── 返回财务: 成功/失败
```

## 4.5 充值流程中的异常处理

```
异常场景                              处理方式
─────────────────────────────────────────────────────────
财务创建渠道订单RPC失败               本地订单标记失败，C端提示重试
用户支付超时（扫码5分钟未支付）         本地订单标记超时/过期
财务回调多次（渠道重发）               幂等校验，订单号去重，已入账直接返回成功
财务传入的订单号本地找不到             返回错误，财务侧记录异常
入账事务失败（DB异常）                返回失败，财务侧记录异常等待人工补单
```

# 五、流程线4：兑换（C端，内部闭环）

## 5.1 总体思路

兑换是**最简单、最独立**的交易链路。法币/USDT → BSB 单向兑换，完全在钱包模块内部完成，只依赖币种配置的汇率，不涉及财务模块、不涉及外部渠道。

如果说充值和提现是"出门办事"（需要跟财务和渠道打交道），兑换就是"关起门在家操作"。

**这也是你开发时应该最先验证的交易链路——因为它依赖最少，可以最早跑通。**

## 5.2 进入兑换页面

```
用户点击"兑换"按钮（只有法币钱包才有，BSB没有兑换按钮）
    │
    └── C端发起: 获取兑换汇率及预览
        │
        └── 后端处理:
            ├── 1. [RPC/内部] → 币种配置.获取汇率（当前法币 对 BSB）
            │      返回: "1 BSB = 23000 VND"
            ├── 2. 获取赠送比例配置（如有兑换赠送活动）
            │      返回: 赠送5%
            └── 3. 返回C端: 汇率值 + 赠送比例
```

## 5.3 用户输入金额、确认兑换

```
用户输入兑换金额（如 230000 VND）→ C端实时展示预览 → 点击确认
    │
    └── 后端处理（创建兑换订单 — 核心流程）:
        │
        ├── Step 1: 重新获取最新汇率
        │   ├── [RPC/内部] → 币种配置.获取汇率
        │   └── 为什么重新取？因为用户在页面停留期间汇率可能已变
        │       用提交时的最新汇率计算，而非页面展示时的旧汇率
        │
        ├── Step 2: 金额计算
        │   ├── 兑换BSB数量 = 输入法币金额 ÷ 汇率
        │   │   例: 230000 VND ÷ 23000 = 10 BSB
        │   ├── 赠送BSB数量 = 兑换BSB × 赠送比例
        │   │   例: 10 BSB × 5% = 0.5 BSB
        │   └── 注意精度处理: 按BSB和法币各自的精度配置做截断/四舍五入
        │
        ├── Step 3: 余额校验
        │   └── 用户当前币种的中心钱包可用余额 ≥ 输入法币金额？
        │       不够 → 返回错误"余额不足"
        │
        ├── Step 4: 事务操作（必须同一个DB事务）
        │   │
        │   │ ── BEGIN TRANSACTION ──
        │   │
        │   ├── 4a. 扣减源币种中心钱包余额
        │   │       UPDATE balance SET available = available - 230000
        │   │       WHERE user_id=X AND currency=VND AND wallet_type=中心
        │   │
        │   ├── 4b. 增加BSB中心钱包余额
        │   │       UPDATE balance SET available = available + 10
        │   │       WHERE user_id=X AND currency=BSB AND wallet_type=中心
        │   │
        │   ├── 4c. 增加BSB奖励钱包余额（赠送部分）
        │   │       UPDATE balance SET available = available + 0.5
        │   │       WHERE user_id=X AND currency=BSB AND wallet_type=奖励
        │   │       同时记录: 该笔赠送需完成X倍流水才能提现
        │   │
        │   ├── 4d. 写兑换订单记录
        │   │       用户ID / 源币种=VND / 目标币种=BSB / 源金额=230000
        │   │       目标金额=10 / 赠送金额=0.5 / 使用汇率=23000
        │   │       状态=成功 / 订单号 / 创建时间
        │   │
        │   ├── 4e. 写余额变动流水（3条）
        │   │       ├── VND 中心钱包 -230000（类型=兑换扣减）
        │   │       ├── BSB 中心钱包 +10（类型=兑换增加）
        │   │       └── BSB 奖励钱包 +0.5（类型=兑换赠送）
        │   │
        │   │ ── COMMIT TRANSACTION ──
        │   │
        │   └── 事务保证: 扣减+增加+赠送+订单+流水 五者一致
        │
        └── 返回C端: 兑换成功 + 兑换明细
            ├── 扣减: 230000 VND
            ├── 获得: 10 BSB（中心钱包）
            └── 赠送: 0.5 BSB（奖励钱包，需X倍流水）
```

**关键设计点**：
- 汇率在事务之前获取（RPC调用不放在事务内，避免拉长事务时间）
- 计算结果进入事务后直接执行，不再二次查汇率
- 扣减和增加在同一事务里，不存在"扣了VND但BSB没加上"的中间状态

# 六、流程线5：提现（C端，跨模块最复杂）

## 6.1 总体思路

提现是所有流程中**跨度最长、涉及模块最多、状态流转最复杂**的流程。

从用户提交到最终到账，经过：**钱包（发起+冻结）→ 用户服务（KYC校验）→ 财务（三级审核+渠道出款）→ 钱包（最终扣款或退回）**。

你的代码在提现流程中承担**头和尾**：开始时做校验+冻结+提交审核，结束时接收审核结果做扣款或解冻。中间的审核和出款你不参与。

## 6.2 阶段一：进入提现页面

```
用户点击"提现"按钮
    │
    ├── C端并行发起4个请求:
    │
    │   ┌──[请求1] KYC状态查询
    │   │
    │   │   后端处理:
    │   │   ├── 判断当前币种的提现是否需要KYC
    │   │   │   ├── USDT提现: 不需要KYC（加密货币匿名特性）
    │   │   │   └── 法币提现: 必须KYC
    │   │   │
    │   │   ├── 需要KYC的场景:
    │   │   │   ├── [RPC] → 用户服务.KYC状态查询（入参: 用户ID）
    │   │   │   ├── 已认证: 返回 kyc_status=VERIFIED
    │   │   │   └── 未认证: 返回 kyc_status=NOT_VERIFIED
    │   │   │       C端引导用户去KYC认证页面，阻断提现流程
    │   │   │
    │   │   └── 注意: 这是后端强制校验，不能只靠C端拦截
    │   │       即使C端被绕过，创建提现订单时后端也会再校验一次
    │   │
    │   ├──[请求2] 获取可用提现方式列表
    │   │
    │   │   后端处理（两步串行）:
    │   │   ├── Step A: [RPC] → 财务.支付配置查询
    │   │   │   入参: 当前币种 + 类型=提现
    │   │   │   返回: 平台支持的全部提现方式候选集
    │   │   │
    │   │   └── Step B: 根据用户充值历史过滤
    │   │       ├── [本地DB] 查询该用户的充值来源记录
    │   │       │   （有没有充过法币？有没有充过USDT？）
    │   │       │
    │   │       ├── 过滤规则（需求文档7-1，合规要求）:
    │   │       │   ├── 只充过法币 → 只保留法币提现方式（银行卡/电子钱包）
    │   │       │   ├── 只充过USDT → 只保留USDT提现方式
    │   │       │   ├── 法币+USDT都充过 → 只保留法币方式（混合以法币为准）
    │   │       │   └── 从没充值过 → 可能无法提现（需确认业务规则）
    │   │       │
    │   │       └── 返回过滤后的提现方式列表
    │   │
    │   │   核心逻辑: 资金从哪个通道进来，就从哪个通道出去
    │   │
    │   ├──[请求3] 提现限额查询
    │   │
    │   │   后端处理:
    │   │   ├── [RPC/内部] → 币种配置.获取配置参数
    │   │   └── 返回: 单笔最低/最高、每日限额、手续费规则、T+N天数
    │   │       C端用于: 输入框限制 + 费用展示 + 到账时间提示
    │   │
    │   └──[请求4] 提现账户信息查询
    │
    │       后端处理:
    │       ├── [本地DB] 查询该用户已保存的提现账户信息
    │       │   ├── 有记录: 返回银行名称/卡号/户名 或 电子钱包ID
    │       │   │          C端只读展示（首次之后自动回显）
    │       │   └── 无记录: 返回空，C端展示填写表单
    │       └── 返回: 账户信息 或 空
    │
    └── C端渲染:
        ├── KYC未通过: 展示"请先完成实名认证"引导页，阻断
        ├── KYC已通过: 正常展示
        │   ├── 可用提现方式列表
        │   ├── 金额输入框（限额提示）
        │   ├── 手续费展示
        │   ├── 到账时间提示（T+N）
        │   └── 提现账户信息（回显或填写）
        └── BSB提现额外展示: 汇率换算"提现100 BSB ≈ 2,300,000 VND"
```

## 6.3 阶段二：用户填写信息、确认提现

```
用户选择提现方式、输入金额、填写/确认账户信息 → 点击"确认提现"
    │
    └── 后端处理（创建提现订单 — 最复杂的订单创建流程）:
        │
        │ ══ 第一段：并行校验 ══
        │
        ├── [并行] Step 1: 限额校验
        │   ├── [RPC/内部] → 币种配置.获取配置参数
        │   ├── 单笔金额 ≥ 最低限额？
        │   ├── 单笔金额 ≤ 最高限额？
        │   └── [本地DB] 今日已提现累计 + 本笔 ≤ 每日限额？
        │
        ├── [并行] Step 2: 户名一致性校验
        │   ├── [RPC] → 用户服务.KYC信息查询（拿到KYC认证的真实姓名）
        │   ├── 提现账户户名 vs KYC姓名
        │   │   比对规则: 忽略大小写，忽略首尾空格，不要求完全精确匹配
        │   └── 不一致 → 返回错误"提现账户户名与实名信息不符"
        │
        ├── [并行] Step 3: 余额校验
        │   ├── [本地DB] 中心钱包.可用余额 ≥ 提现金额 + 手续费？
        │   └── 不够 → 返回错误"余额不足"
        │
        │   注: Step 1/2/3 三者互不依赖，全部并行执行
        │       全部通过才继续，任一失败则返回对应错误
        │
        │ ══ 第二段：串行执行 ══
        │
        ├── Step 4: 冻结余额
        │   │
        │   │   这是提现流程最关键的一步:
        │   │   不是直接扣钱，而是"冻结"——把钱从可用余额转到冻结余额
        │   │   用户看到可用余额减少了，但钱并没有真的出去
        │   │   等审核通过出款成功后才真正扣除
        │   │
        │   ├── [本地DB] 事务操作:
        │   │   UPDATE balance SET
        │   │       available = available - (amount + fee),
        │   │       frozen = frozen + (amount + fee)
        │   │   WHERE user_id=X AND currency=Y AND wallet_type=中心
        │   │       AND available >= (amount + fee)  -- 乐观锁条件
        │   │
        │   └── 如果更新行数=0: 说明余额不足（并发竞争），返回错误
        │
        ├── Step 5: 生成本地提现订单
        │   ├── 生成平台订单号（雪花ID）
        │   └── 写入提现订单表:
        │       用户ID / 币种 / 金额 / 手续费 / 提现方式
        │       平台订单号 / 状态=审核中 / 创建时间
        │       提现账户信息 / 冻结流水号
        │
        ├── Step 6: 保存/更新提现账户信息
        │   └── 如果是首次填写: 写入用户提现账户表
        │       后续提现会自动回显这些信息
        │
        ├── Step 7: 提交财务审核
        │   ├── [RPC] → 财务.提交提现审核
        │   │   入参: 用户ID, 币种, 金额, 手续费, 提现方式,
        │   │         平台订单号, 提现账户信息
        │   │   返回: 审核单号
        │   │
        │   └── ★ 如果RPC调用失败（关键异常场景）:
        │       │
        │       │ 此时余额已经冻结了，但财务没收到审核请求
        │       │ 用户的钱"卡住了"——冻结了但没人审核
        │       │
        │       └── 补偿处理:
        │           ├── 解冻余额（冻结→退回可用）
        │           ├── 更新本地订单状态=失败
        │           ├── 写流水记录（冻结回退）
        │           └── 返回C端错误"提现暂不可用，请稍后重试"
        │
        └── 返回C端: 提现申请已提交，等待审核
```

[财务内部审核流程 — 你不管这段]

    ┌──────────────────────────────────────────────┐

    │                                              │

    │ │   │ ── BEGIN TRANSACTION ──                │

    │ │   │   UPDATE balance SET                   │
    │ │   │       frozen = frozen - (amount+fee)   │

    │ │   ├── 更新订单状态=成功                      │
    │ │   ├── 写余额变动流水（类型=提现扣款）          │
    │ │   │ ── COMMIT TRANSACTION ──               │
    │ └── 4. 返回: 成功                             │
    │                                              │
    └──────────────────────────────────────────────┘

    ┌──────────────────────────────────────────────┐
    │ 场景B: 任一级审核拒绝 或 渠道出款失败            │
    │                                              │
    │ 财务 → [RPC] → 你的钱包.提现失败解冻接口        │
    │                                              │
    │ 你的处理:                                     │
    │ ├── 1. 幂等校验                               │
    │ ├── 2. 订单状态校验                            │
    │ ├── 3. 事务操作:                              │
    │ │   │ ── BEGIN TRANSACTION ──                │

# 七、流程线6：被调用 — 财务回调/人工修正（RPC服务提供方）

## 7.1 总体思路

除了C端主动发起的操作外，你的钱包模块还要**作为RPC服务提供方**，被财务模块调用。这些调用主要发生在两个场景：

1. **渠道回调**：充值/提现的最终结果通知（上面流程线3和5已覆盖）
2. **人工修正**：B端操作人员手动加款/减款/补单

人工修正是你还没详细看到的流程，下面展开。

## 7.2 人工加款流程

```
场景: 运营需要给用户补偿（活动奖励、客诉赔付、系统误差等）

财务B端操作人创建加款单 → 审核人审核通过
    │
    ▼
财务 → [RPC] → 你的钱包.人工加款接口
    │
    └── 你的处理:
        │
        ├── Step 1: 幂等校验
        │   └── 用修正单号查: 是否已执行过？已执行 → 直接返回成功
        │
        ├── Step 2: 参数校验
        │   ├── 用户ID有效？
        │   ├── 币种存在且启用？
        │   ├── 金额 > 0？
        │   └── 子钱包类型合法？（中心/奖励/代理/主播 四选一）
        │
        ├── Step 3: 事务操作
        │   │ ── BEGIN TRANSACTION ──
        │   ├── 指定子钱包余额 + 金额
        │   │   UPDATE balance SET available = available + amount
        │   │   WHERE user_id=X AND currency=Y AND wallet_type=指定类型
        │   ├── 写余额变动流水
        │   │   类型=人工加款 / 关联修正单号 / 原因 / 操作人
        │   │ ── COMMIT TRANSACTION ──
        │
        └── 返回: 成功/失败
```

## 7.3 人工减款流程

```
场景: 用户违规、误操作退回、财务调整等

财务B端操作人创建减款单（会先查用户余额确认够扣）→ 审核通过
    │
    ▼
财务 → [RPC] → 你的钱包.人工减款接口
    │
    └── 你的处理:
        │
        ├── Step 1: 幂等校验（修正单号去重）
        │
        ├── Step 2: 参数校验（同加款）
        │
        ├── Step 3: 余额校验
        │   └── 指定子钱包的可用余额 ≥ 减款金额？
        │       不够 → 返回"余额不足，减款失败"
        │       财务侧会标记该修正单=执行失败
        │
        ├── Step 4: 事务操作
        │   │ ── BEGIN TRANSACTION ──
        │   ├── 指定子钱包余额 - 金额
        │   │   UPDATE balance SET available = available - amount
        │   │   WHERE user_id=X AND currency=Y AND wallet_type=指定类型
        │   │       AND available >= amount  -- 乐观锁，再次确认
        │   ├── 写余额变动流水
        │   │   类型=人工减款 / 关联修正单号 / 原因
        │   │ ── COMMIT TRANSACTION ──
        │
        └── 返回: 成功/失败
```

```
场景: 用户实际付款了，但第三方渠道回调丢失，系统没自动入账

    │
    ▼
财务 → [RPC] → 你的钱包.人工补单入账接口
    │
    └── 你的处理:
        │
        ├── Step 1: 幂等校验（补单号去重）
        │
        ├── Step 2: 查找原充值订单
        │   ├── [本地DB] 用原充值订单号查
        │   ├── 订单存在 且 状态=失败/超时 → 继续
        │   ├── 订单不存在 → 返回错误
        │   └── 订单状态=成功 → 已入过账，不需要补单，返回错误
        │
        ├── Step 3: 事务操作
        │   │ ── BEGIN TRANSACTION ──
        │   ├── 中心钱包余额 + 补单金额
        │   ├── 更新原充值订单状态 → 补单成功
        │   ├── 写余额变动流水
        │   │   类型=人工补单 / 关联原订单号+补单号
        │   │ ── COMMIT TRANSACTION ──
        │
        └── 返回: 成功/失败
```

# 十、流程中的关键设计要点

## 10.1 事务边界原则

```
原则: RPC调用不放在DB事务内

正确做法:
  获取汇率(RPC) → 计算 → [开事务] → 扣减 → 增加 → 写流水 → [提交事务]

错误做法:
  [开事务] → 获取汇率(RPC) → 计算 → 扣减 → 增加 → 写流水 → [提交事务]
  ✕ RPC在事务内会拉长事务持有时间，如果RPC超时，事务长时间不提交，影响DB连接池

规则:
  - 所有RPC调用在事务之前完成，拿到计算结果
  - 事务内只做本地DB操作（余额变动+订单更新+流水记录）
  - 事务尽可能短，减少锁持有时间
```

## 10.2 幂等设计原则

```
所有RPC暴露的写接口必须幂等:

充值入账:    平台订单号 → 查已入账? → 是则直接返回成功
提现确认:    订单号 → 查订单状态=已完成? → 是则直接返回成功
提现解冻:    订单号 → 查订单状态=已失败? → 是则直接返回成功
人工加减款:  修正单号 → 查已执行? → 是则直接返回成功
人工补单:    补单号 → 查已执行? → 是则直接返回成功

幂等实现方式:
  方式1: 先查状态，已完成就返回（简单直接）
  方式2: 唯一键约束，插入流水时如重复则catch唯一键冲突（更可靠）
  推荐: 两者结合，先查快速返回，写入时再加唯一键兜底
```

## 10.3 补偿机制

```
跨模块调用失败时的补偿:

场景1: 创建充值订单 → 调财务.创建渠道订单RPC失败
  补偿: 本地订单标记失败，返回C端错误（无余额影响，充值是先支付后入账）

场景2: 创建提现订单 → 冻结余额成功 → 调财务.提交审核RPC失败
  补偿: 解冻余额 + 本地订单标记失败 + 写回退流水（有余额影响，必须补偿）

场景3: 财务调钱包.充值入账 → 钱包返回失败（DB异常等）
  补偿: 财务侧记录异常，等待人工补单（财务已确认渠道到账，钱包入账失败需人工介入）
```

## 10.4 余额操作安全

```
所有余额扣减操作使用乐观锁:

UPDATE balance SET available = available - amount
WHERE user_id=X AND currency=Y AND wallet_type=Z
    AND available >= amount  -- 关键: 乐观锁条件

如果受影响行数 = 0 → 说明余额已不足（可能并发被其他操作先扣了）

绝对不能:
  先 SELECT available → 应用层判断 ≥ amount → 再 UPDATE SET available = X
  这种方式在并发场景下会出现负余额（两个请求同时读到足够余额，各扣一次）
```

## 10.5 流程优先级（开发建议）

```
建议开发顺序（从最独立到最依赖）:

第一步: 币种配置B端CRUD + RPC暴露（无任何外部依赖）
第二步: 钱包余额查询 + 记录查询（纯读，验证数据模型）
第三步: 兑换功能（只依赖汇率，最简单的交易链路，验证事务操作）
第四步: 钱包RPC暴露接口（入账/扣款/解冻/加减款，供财务联调）
第五步: 充值功能（依赖财务支付配置+渠道订单）
第六步: 提现功能（依赖最多：财务+用户服务+KYC）

理由:
  - 第一到三步可以完全独立开发，不等任何人
  - 第四步提供RPC接口给财务，越早越好（IDL先行，stub先给）
  - 第五六步需要财务接口就绪才能联调，先把钱包侧逻辑写好，mock财务返回
```
