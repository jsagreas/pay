# 流程思路：ser-wallet 模块内与模块间的完整处理流程

> 分析依据：ser-wallet/tmp/ 下 152 张产品原型和需求文档图片的分析结果
> 核心视角：我方负责（钱包+币种配置）的内部处理流程 + 与他方（财务管理）的联动流程
> 目的：清楚每个业务场景从触发到结束的完整流转路径

---

## 第一部分：总体流程架构

### 1.1 三层流程关系概览

### 1.2 核心处理原则（贯穿所有流程）

| 原则 | 说明 | 依据 |
|------|------|------|
| 币种隔离 | 所有操作在当前选中币种下进行，不跨币种混合 | 需求文档3.3币种切换规则 |
| 余额一致 | 账务流水记录与账户余额必须强一致性校验 | 需求文档3.10非功能性需求 |
| 幂等处理 | 所有资金变动操作必须幂等，防止重复执行 | 需求文档3.10非功能性需求 |
| 消费优先 | 先扣中心钱包，再扣奖励钱包 | 需求文档4.1.2消费优先级 |
| 冻结模式 | 提现采用"冻结→审核→扣除/解冻"模式，不直接扣款 | 需求文档5.5提现流程 |
| 稽核约束 | 奖金/赠送金额受稽核倍数约束，未完成流水不可提现 | 需求文档3.6奖金机制、5.6.1补单 |

## 第二部分：C端用户操作流程

### 2.1 钱包首页加载流程

**触发**：用户进入钱包页面

```
用户打开钱包页面
    │
    ├── 1. 读取用户当前选中币种（优先本地缓存，否则取默认币种）
    │
    ├── 2. 请求钱包余额
    │   └── GetBalance(userId, currencyCode)
    │       ├── 查询用户在该币种下的中心钱包余额
    │       ├── 查询用户在该币种下的奖励钱包余额
    │       └── 返回：资金余额 = 中心 + 奖励（分别显示）
    │
    ├── 3. 请求币种列表（用于切换）
    │   └── GetCurrencyList()
    │       └── 查询所有启用状态的币种配置
    │           ├── 币种代码、名称、图标、排序
    │           └── 按排序值返回
    │
    └── 4. 页面渲染
        ├── 顶部：当前币种 + 资金余额
        ├── 中部：充值 / 提现 / 兑换 入口按钮
        └── 底部：交易记录入口
```

**币种切换流程**：

```
用户点击币种切换
    │
    ├── 1. 弹出币种选择列表
    │
    ├── 2. 用户选择目标币种
    │
    ├── 3. 本地切换当前币种标识
    │
    └── 4. 重新请求数据
        ├── GetBalance(userId, newCurrencyCode)  → 新余额
        ├── 充值方式列表跟随切换
        ├── 提现方式列表跟随切换
        └── 交易记录跟随切换
```

**依据**：原型 2-1~2-3个人中心、需求文档3.3币种切换规则

### 2.2 充值完整流程

#### 2.2.1 充值页面初始化

```
用户进入充值页面
    │
    ├── 1. 获取当前币种的充值方式列表
    │   └── GetDepositMethods(currencyCode)
    │       ├── 我方查询币种配置（确认币种有效且启用）
    │       ├── 调用财务模块获取：该币种下的充值方式列表
    │       │   └── RPC调他方：ser-finance.GetPaymentMethods(currencyCode, "deposit")
    │       │       返回：[{方式名称, 图标, 档位列表, 金额范围, 默认方式标记}]
    │       └── 组装返回给前端
    │
    ├── 2. 获取当前可用的奖金活动列表（可选）
    │   └── GetBonusActivities(currencyCode)
    │       └── 查询匹配当前币种的奖金活动
    │
    └── 3. 页面渲染
        ├── 余额展示区
        ├── 奖金选择组件（默认"无奖金"或最佳匹配）
        ├── 充值方式Tab（默认选中推荐方式）
        ├── 快捷金额档位按钮
        └── 手动金额输入框
```

#### 2.2.2 用户提交充值

```
用户输入金额 → 点击"去充值"
    │
    ├── 1. 前端校验
    │   ├── 金额 >= 最低档位
    │   ├── 金额在允许范围内
    │   └── 如选了奖金活动：金额 >= 活动最低门槛
    │
    ├── 2. 重复转账拦截检查（仅银行/电子钱包，USDT不适用）
    │   └── CheckDuplicateDeposit(userId, currencyCode, amount)
    │       ├── 查询：同userId + 同币种 + 同金额 + 状态为(待支付/处理中)的订单数
    │       ├── 0笔 → 继续
    │       ├── 1-2笔 → 返回警告标记（前端弹窗，用户可选择继续）
    │       └── 3笔 → 返回强拦截标记（前端弹窗，不允许继续）
    │
    ├── 3. 创建充值订单
    │   └── CreateDeposit(userId, currencyCode, amount, method, bonusActivityId?)
    │       │
    │       ├── a. 生成订单号：C + 16位随机数
    │       ├── b. 查询币种的入款汇率（如果涉及跨币种）
    │       ├── c. 计算奖金（如有选择活动）
    │       │   ├── 活动赠送金额 = 充值金额 × 奖金百分比
    │       │   ├── 稽核流水 = (充值金额 + 赠送金额) × 稽核倍数
    │       │   └── 记录稽核信息
    │       ├── d. 创建平台订单（状态=待支付）
    │       │   └── 记录：用户ID、币种、金额、方式、汇率、奖金信息、订单号
    │       ├── e. 调用财务模块创建通道订单
    │       │   └── RPC调他方：ser-finance.CreateChannelOrder(orderInfo)
    │       │       返回：{通道订单号, 支付页面URL/QR码/收款信息}
    │       └── f. 返回支付信息给前端
    │
    └── 4. 前端跳转支付
        ├── USDT：显示收款地址 + QR码
        ├── 银行转账：跳转三方承载页
        └── 电子钱包：显示QR码 + 倒计时
```

**依据**：原型 3-1~3-4充值币种切换、4-1虚拟币充值、5-1~5-4切换充值方式、6-1~6-2奖金选择、7-1~7-4通用回调；需求文档3.4充值模块、3.5重复转账拦截、3.6奖金机制

#### 2.2.3 充值成功回调处理（被调方）

```
财务模块收到三方支付回调 → 支付成功
    │
    └── 调用我方RPC：DepositCallback(orderNo, actualAmount, channelOrderNo)
        │
        ├── 1. 幂等校验
        │   ├── Redis SetNX：deposit:callback:{orderNo}
        │   └── 已处理过 → 直接返回成功（幂等）
        │
        ├── 2. 查询订单
        │   ├── 按 orderNo 查询充值订单
        │   └── 校验订单状态 == 待支付（否则拒绝）
        │
        ├── 3. 更新订单状态
        │   └── 状态 → 支付成功，记录完成时间、通道订单号
        │
        ├── 4. 钱包加款（核心余额操作）
        │   ├── 充值金额 → 加入中心钱包
        │   ├── 活动赠送（如有）→ 加入奖励钱包
        │   └── 到账总额 = 充值金额 + 活动赠送
        │
        ├── 5. 写入账变流水
        │   ├── 中心钱包流水：类型=充值，金额=充值金额
        │   └── 奖励钱包流水（如有）：类型=活动赠送，金额=赠送金额
        │
        ├── 6. 记录稽核信息（如有奖金）
        │   └── 创建稽核记录：稽核倍数、需完成流水
        │
        └── 7. 返回成功
```

**依据**：需求文档5.4充值流程（步骤6："触发充值额度到钱包用户上"）、6.1财务管理→钱包接口（CreditWallet）

#### 2.2.4 充值失败/超时处理

```
两种场景：

场景A：三方回调支付失败
    └── 财务模块调用：DepositFailed(orderNo, failReason)
        ├── 幂等校验
        ├── 更新订单状态 → 支付失败
        ├── 记录失败原因
        └── 不操作余额

场景B：订单超时未支付（定时任务）
    └── 定时扫描：状态=待支付 且 创建时间超过有效期
        ├── 批量更新状态 → 支付超时（视为失败）
        └── 不操作余额
```

### 2.3 兑换完整流程

**兑换方向**：法币(VND/IDR/THB) → BSB（单向，BSB页面无兑换入口）

```
用户在法币钱包页面点击"兑换"
    │
    ├── 1. 兑换页面初始化
    │   ├── 获取当前币种余额
    │   ├── 获取当前币种对BSB的汇率
    │   │   └── GetExchangeRate(fromCurrency, "BSB")
    │   │       ├── 查询币种配置：当前币种的平台汇率
    │   │       └── 计算：1 BSB = X 法币（统一展示格式）
    │   └── 获取兑换赠送比例（如有活动）
    │
    ├── 2. 用户输入兑换金额
    │   ├── 校验：金额 <= 当前币种可用余额
    │   │   └── 超过 → 提示"已达当前可兑换上限"
    │   ├── 实时计算：
    │   │   ├── 兑换所得BSB = 输入金额 / 汇率
    │   │   ├── 赠送奖金BSB = 兑换所得 × 赠送比例（如10%）
    │   │   └── 实际到账BSB = 兑换所得 + 赠送奖金
    │   └── 展示：兑换所得、赠送、实际到账、稽核要求
    │
    └── 3. 用户确认兑换
        └── Exchange(userId, fromCurrency, amount)
            │
            ├── a. 参数校验
            │   ├── 币种有效且启用
            │   ├── 金额 > 0 且 <= 可用余额
            │   └── 汇率有效（未过期）
            │
            ├── b. 锁定汇率（防止并发时汇率变动导致差异）
            │   └── 取当前平台汇率作为本次兑换汇率
            │
            ├── c. 扣减法币钱包余额（中心钱包）
            │   └── 法币中心钱包余额 -= 兑换金额
            │
            ├── d. 增加BSB钱包余额
            │   ├── BSB中心钱包 += 兑换所得BSB
            │   └── BSB奖励钱包 += 赠送BSB（如有）
            │
            ├── e. 写入账变流水
            │   ├── 法币流水：类型=兑换支出，金额=-兑换金额
            │   ├── BSB流水（中心）：类型=兑换收入，金额=+兑换所得
            │   └── BSB流水（奖励，如有）：类型=兑换赠送，金额=+赠送金额
            │
            ├── f. 记录稽核信息（赠送部分）
            │   └── 赠送BSB对应的稽核倍数和流水要求
            │
            └── g. 返回兑换结果
```

**依据**：原型 8兑换BSB；需求文档3.7兑换模块

**关键设计点**：
- 兑换是**模块内操作**，不涉及财务模块，我方独立完成
- 操作两个币种的余额（法币减、BSB加），但属于同一用户，需事务保证
- 赠送部分进奖励钱包，带稽核约束

### 2.4 提现完整流程

提现是最复杂的流程，涉及 **KYC校验 → 订单创建 → 余额冻结 → 多级审核 → 出款/解冻**。

#### 2.4.1 提现发起

```
用户点击"提现"
    │
    ├── 1. KYC状态检查
    │   └── 调用ser-kyc：CheckKycStatus(userId)
    │       ├── 未认证 → 弹窗提示
    │       │   ├── "去认证" → 跳转KYC流程
    │       │   └── "继续提现" → 仅显示USDT提现方式
    │       └── 已认证 → 显示全部提现方式
    │
    ├── 2. 获取提现方式列表
    │   └── GetWithdrawMethods(currencyCode, kycStatus)
    │       ├── 查询该币种下可用的提现方式
    │       ├── 未KYC → 仅返回USDT方式
    │       └── 已KYC → 返回全部（银行卡/电子钱包/USDT）
    │
    ├── 3. 获取用户提现账户信息（如有历史记录）
    │   └── GetWithdrawAccount(userId, methodType)
    │       ├── 有记录 → 自动填充（只读，不可修改）
    │       └── 无记录 → 空表单，需首次填写
    │
    └── 4. 页面渲染
        ├── 提现方式选择
        ├── 账户信息（首次填写 / 自动填充）
        ├── 金额输入
        ├── 手续费 + 到账金额 实时计算
        └── 每日限额提示
```

#### 2.4.2 提现提交

```
用户填写信息 → 点击"确认提现"
    │
    ├── 1. 校验
    │   ├── 金额 >= 单笔最低
    │   ├── 金额 <= 每日限额（已提现 + 本次 <= 限额）
    │   ├── 金额 <= 可用余额（可用 = 总额 - 已冻结）
    │   ├── 账户持有人 == KYC姓名（忽略大小写和空格）
    │   └── 稽核检查：奖励钱包中未完成稽核的金额不可提现
    │
    ├── 2. 提现路由校验
    │   ├── 法币充值记录 → 允许法币提现
    │   ├── USDT充值记录 → 允许USDT提现
    │   ├── 法币+USDT充值 → 允许法币提现
    │   └── 法币→BSB兑换 → 允许法币提现
    │
    ├── 3. 创建提现订单
    │   └── CreateWithdraw(userId, currencyCode, amount, method, accountInfo)
    │       │
    │       ├── a. 生成订单号：T + 16位随机数
    │       │
    │       ├── b. 保存/更新提现账户信息（首次提现时）
    │       │
    │       ├── c. 查询出款汇率（如涉及跨币种）
    │       │
    │       ├── d. 计算手续费
    │       │   └── 手续费 = 提现金额 × 手续费率
    │       │
    │       ├── e. ★ 冻结用户余额 ★
    │       │   └── FreezeBalance(userId, currencyCode, walletType, amount)
    │       │       ├── 可用余额 -= 冻结金额
    │       │       ├── 冻结余额 += 冻结金额
    │       │       └── 总余额不变
    │       │
    │       ├── f. 创建订单（状态=待风控审核）
    │       │   └── 记录：所有订单信息 + 冻结金额 + 汇率快照
    │       │
    │       ├── g. 写入账变流水
    │       │   └── 类型=提现冻结
    │       │
    │       └── h. 返回：订单创建成功
    │
    └── 至此，用户端流程结束，后续由B端处理
```

**依据**：原型 9-1~9-4提现触发认证、10-1~10-6提现；需求文档3.8提现模块、5.5提现记录

#### 2.4.3 提现审核流程（B端 + 跨模块联动）

```
提现订单创建（状态=待风控审核）
    │
    ├── 阶段1: 风控审核（B端操作员）
    │   ├── 通过 → 状态变更为"待财务审核"
    │   │
    │   └── 驳回 → 状态变更为"风控驳回"（终态）
    │       └── ★ 解冻用户余额 ★
    │           └── 财务模块调我方RPC：UnfreezeBalance(userId, currencyCode, walletType, orderNo, amount)
    │               ├── 冻结余额 -= 解冻金额
    │               ├── 可用余额 += 解冻金额
    │               └── 写入账变流水：类型=提现解冻（风控驳回）
    │
    ├── 阶段2: 财务审核（B端操作员）
    │   ├── 通过 → 状态变更为"待出款"
    │   │   └── 计算手续费，准备发送到支付通道
    │   │
    │   └── 驳回 → 状态变更为"财务驳回"（终态）
    │       └── ★ 解冻用户余额 ★（同上逻辑）
    │
    ├── 阶段3: 出款执行
    │   └── 财务模块通过通道发起代付
    │       │
    │       ├── 通道返回成功
    │       │   └── 状态 → "出款成功"（终态）
    │       │       └── ★ 扣除冻结金额 ★
    │       │           └── 财务模块调我方RPC：DeductFrozenAmount(userId, currencyCode, walletType, orderNo, amount)
    │       │               ├── 冻结余额 -= 扣除金额
    │       │               ├── 总余额 -= 扣除金额（资金真正离开钱包）
    │       │               └── 写入账变流水：类型=提现成功
    │       │
    │       └── 通道返回失败
    │           └── 状态 → "出款失败"
    │               └── ★ 保持冻结，不自动解冻 ★
    │                   └── 需人工处理（重试出款 或 手动解冻）
```

**关键设计思路**：
- 提现全程采用"冻结-解冻-扣除"三步模式，不直接扣余额
- 创建订单时我方冻结，审核驳回时财务模块调我方解冻，出款成功时财务模块调我方扣除
- 出款失败时**不自动解冻**，防止资金风险，需人工介入

#### 2.4.4 提现余额状态变化一览

```
初始状态：可用余额 = 1000，冻结余额 = 0

① 创建提现订单（提现500）
   → 可用余额 = 500，冻结余额 = 500

② 情况A：审核驳回 → 解冻
   → 可用余额 = 1000，冻结余额 = 0（恢复原状）

② 情况B：出款成功 → 扣除冻结
   → 可用余额 = 500，冻结余额 = 0（资金离开）

② 情况C：出款失败 → 保持冻结
   → 可用余额 = 500，冻结余额 = 500（等待人工处理）
```

### 3.2 汇率更新流程（定时任务）

```
定时任务触发（可配置频率）
    │
    └── UpdateExchangeRates()
        │
        ├── 1. 获取实时汇率
        │   ├── 调用三方API-A → 汇率A
        │   ├── 调用三方API-B → 汇率B
        │   └── 调用三方API-C → 汇率C
        │   └── 实时汇率 = (汇率A + 汇率B + 汇率C) / 3
        │       （某个API失败时取可用API的平均值）
        │
        ├── 2. 遍历每个需要更新的币种
        │   └── 对每个币种：
        │       │
        │       ├── a. 更新实时汇率字段
        │       │
        │       ├── b. 检查平台汇率是否需要更新
        │       │   ├── 平台汇率为空（初始状态）
        │       │   │   └── 直接以实时汇率计算平台汇率
        │       │   │       ├── 平台汇率 = 实时汇率
        │       │   │       ├── 入款汇率 = 实时汇率 × (1 + 浮动%)
        │       │   │       └── 出款汇率 = 实时汇率 × (1 - 浮动%)
        │       │   │
        │       │   └── 平台汇率已存在
        │       │       ├── 计算偏差 = |实时 - 平台| / 实时 × 100%
        │       │       ├── 偏差 < 阈值 → 不更新平台汇率
        │       │       └── 偏差 >= 阈值 → 更新平台汇率
        │       │           ├── 新平台汇率 = 实时汇率
        │       │           ├── 新入款汇率 = 实时汇率 × (1 + 浮动%)
        │       │           ├── 新出款汇率 = 实时汇率 × (1 - 浮动%)
        │       │           └── ★ 生成汇率日志 ★
        │       │
        │       └── c. 更新Redis缓存中的汇率数据
        │
        └── 3. 完成，等待下次触发
```

**依据**：需求文档4.5.2汇率获取与计算、4.5.3阈值更新机制

**关键设计思路**：
- 汇率更新是**我方独立完成的定时任务**，不依赖财务模块
- 三方API取平均值增加容错
- 平台汇率不是每次都更新，只有偏差达到阈值才触发
- 每次平台汇率更新必须生成日志（审计要求）

```
│  ├── 新可用余额 = 当前可用 - amount                          │
│  ├── 更新余额                                               │
│  └── 写入账变流水                                            │
│                                                            │
│  FreezeBalance(userId, currency, walletType, amount)        │
│  ├── 锁定用户钱包行                                         │
│  ├── 校验：可用余额 >= amount                                │
│  ├── 可用余额 -= amount                                     │
│  ├── 冻结余额 += amount                                     │
│  ├── 更新（总余额不变）                                      │
│  └── 写入账变流水                                            │
│                                                            │
│  UnfreezeBalance(userId, currency, walletType, amount)      │
│  ├── 锁定用户钱包行                                         │
│  ├── 校验：冻结余额 >= amount                                │
│  ├── 冻结余额 -= amount                                     │
│  ├── 可用余额 += amount                                     │
│  ├── 更新（总余额不变）                                      │
│  └── 写入账变流水                                            │
│                                                            │
│  DeductFrozenBalance(userId, currency, walletType, amount)  │
│  ├── 锁定用户钱包行                                         │
│  ├── 校验：冻结余额 >= amount                                │
│  ├── 冻结余额 -= amount                                     │
│  ├── 总余额 -= amount（资金真正离开）                          │
│  ├── 更新                                                   │
│  └── 写入账变流水                                            │
└────────────────────────────────────────────────────────────┘

关键约束：
- 所有方法必须在事务中执行
- 必须先加锁再读余额（防止并发覆盖）
- 每次操作必须写账变流水（审计需求）
- 每次操作前必须做幂等校验（orderNo唯一）
```

### 6.2 账变流水记录结构

```
每笔余额变动产生一条流水记录：

┌──────────────────────────────────────────┐
│              账变流水字段                   │
│                                          │
│  流水ID      — Snowflake生成              │
│  用户ID      — 关联用户                    │
│  币种代码    — 所属币种                     │
│  钱包类型    — 中心/奖励/主播/代理/场馆      │
│  变动类型    — 充值/提现/兑换/奖励/冻结/...  │
│  关联订单号  — C/T/B/A/M + 16位            │
│  变动前余额  — 操作前的余额快照              │
│  变动金额    — 正数=收入，负数=支出          │
│  变动后余额  — 操作后的余额                  │
│  冻结变动    — 冻结/解冻金额（如有）          │
│  备注        — 操作说明                     │
│  创建时间    — 毫秒时间戳                    │
└──────────────────────────────────────────┘

变动类型枚举：
  deposit        — 充值到账
  deposit_bonus  — 充值活动赠送
  withdraw_freeze — 提现冻结
  withdraw_unfreeze — 提现解冻（驳回）
  withdraw_deduct — 提现扣除（出款成功）
  exchange_out   — 兑换支出（法币减少）
  exchange_in    — 兑换收入（BSB增加）
  exchange_bonus — 兑换赠送
  manual_credit  — 人工加款
  manual_debit   — 人工减款
  supplement     — 充值补单
  bet_deduct     — 投注扣款（场馆消费）
  bet_reward     — 投注返奖
  gift_purchase  — 礼物消费
  activity_reward — 活动奖励
```

### 6.3 币种配置读取流程

```
所有涉及币种/汇率的操作都先经过以下流程：

获取币种信息（所有子模块共用）
    │
    ├── 1. 先查Redis缓存
    │   └── key: "wallet:currency:{currencyCode}"
    │       ├── 命中 → 直接返回（高性能路径）
    │       └── 未命中 → 继续
    │
    ├── 2. 查数据库
    │   └── SELECT * FROM currency_config WHERE code = ? AND status = 1
    │       ├── 找到 → 写入Redis缓存 + 返回
    │       └── 未找到 → 返回错误（币种不存在或已禁用）
    │
    └── 3. 返回币种完整信息
        ├── 币种代码/名称/类型/国家
        ├── 货币符号/千分位/小数点
        ├── 实时汇率/平台汇率/入款汇率/出款汇率
        └── 浮动率/阈值/排序/状态

缓存刷新时机：
  - 汇率定时任务更新后 → 刷新缓存
  - B端修改币种配置后 → 刷新缓存
  - 缓存设置合理TTL → 过期自动失效
```

### 6.4 幂等校验流程

```
所有资金操作都必须做幂等校验：

请求到达（含 orderNo）
    │
    ├── 方式一：Redis SetNX（推荐用于高并发场景）
    │   └── SetNX("wallet:idempotent:{orderNo}:{opType}", "1", TTL=24h)
    │       ├── 设置成功 → 首次请求，继续处理
    │       └── 设置失败 → 重复请求，直接返回成功（幂等）
    │
    ├── 方式二：数据库唯一约束（作为兜底保障）
    │   └── 账变流水表对 (order_no, op_type) 设唯一索引
    │       ├── INSERT成功 → 首次请求
    │       └── 唯一冲突 → 重复请求
    │
    └── 双保险：Redis快速拦截 + 数据库兜底
```

## 第七部分：消费优先级处理流程

### 7.1 单币种消费扣款流程

```
场景：用户在某币种下消费N金额

消费扣款请求（userId, currency, amount）
    │
    ├── 1. 查询该币种下的钱包余额
    │   ├── 中心钱包可用余额 = A
    │   └── 奖励钱包可用余额 = B
    │
    ├── 2. 校验总余额
    │   └── A + B >= amount ？
    │       ├── 否 → 返回余额不足
    │       └── 是 → 继续
    │
    ├── 3. 计算扣款分配（先中心后奖励）
    │   ├── 中心扣款 = min(A, amount)
    │   ├── 剩余 = amount - 中心扣款
    │   └── 奖励扣款 = 剩余（此时 剩余 <= B，已校验）
    │
    ├── 4. 执行扣款（事务内）
    │   ├── 中心钱包可用余额 -= 中心扣款
    │   └── 奖励钱包可用余额 -= 奖励扣款
    │
    ├── 5. 写入账变流水
    │   ├── 中心钱包流水（如果中心扣款 > 0）
    │   └── 奖励钱包流水（如果奖励扣款 > 0）
    │
    └── 6. 返回扣款明细
        └── {centerDeducted: 中心扣款, rewardDeducted: 奖励扣款}
```

### 7.2 混合投注返奖流程

```
场景：用户投注100（中心扣80 + 奖励扣20），中奖返200

返奖请求（userId, currency, amount=200, betOrderNo）
    │
    ├── 1. 查询原投注扣款比例
    │   └── 根据 betOrderNo 查账变流水
    │       ├── 中心扣款 80 → 占比 80%
    │       └── 奖励扣款 20 → 占比 20%
    │
    ├── 2. 按比例拆分返奖
    │   ├── 中心返奖 = 200 × 80% = 160
    │   └── 奖励返奖 = 200 × 20% = 40
    │
    ├── 3. 执行加款
    │   ├── 中心钱包 += 160
    │   └── 奖励钱包 += 40
    │
    ├── 4. 写入账变流水
    │   ├── 中心：类型=投注返奖，+160
    │   └── 奖励：类型=投注返奖，+40
    │
    └── 5. 累计稽核流水（奖励钱包部分）
```

**依据**：需求文档4.1.2消费优先级"混合投注时，返奖按投注金额比例拆分"

## 第八部分：异常处理流程

### 8.1 充值回调重复到达

```
三方多次回调同一订单
    │
    ├── 第1次：幂等键不存在 → 正常处理 → 设置幂等键
    │
    ├── 第2次：幂等键存在 → 跳过处理 → 返回成功
    │
    └── 关键：第2次不能报错，必须返回成功（否则三方会持续重试）
```

### 8.2 提现出款失败

```
通道返回出款失败
    │
    ├── 不自动解冻（防止用户趁机转走资金后再申诉）
    │
    ├── 标记订单为"出款失败"
    │
    └── 等待运营人员处理
        ├── 选项A：重新发起出款（换通道）
        ├── 选项B：手动解冻（确认不出款）
        └── 选项C：联系三方确认实际状态
```

### 8.3 余额对账异常

```
定时对账任务
    │
    ├── 对每个用户+币种+钱包类型
    │   ├── 计算：初始余额 + SUM(所有入账流水) - SUM(所有出账流水)
    │   ├── 对比：数据库中的当前余额
    │   └── 不一致 → 记录差异，告警通知
    │
    └── 人工介入排查差异原因
```

### 8.4 汇率API全部不可用

```
定时任务获取汇率失败
    │
    ├── API-A失败 + API-B失败 + API-C失败
    │
    ├── 不更新实时汇率（保持上次值）
    │
    ├── 记录告警日志
    │
    └── 平台汇率保持不变（业务不受影响，因为平台汇率已缓存）
```
