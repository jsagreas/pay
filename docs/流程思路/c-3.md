# 多币种钱包系统 — 流程思路（完整版）

> 基于需求理解、链路关系、接口评估、依赖关系四份文档的综合提炼
> 目标：梳理模块内部处理流程 + 模块之间联动流程 + 完整业务场景的端到端流程
> 三个模块：币种配置（自己负责）、多币种钱包（自己负责）、财务管理（他人负责，理解联动）
> 当前需求状态：预评审阶段，大方向已定型，细节可能随跨部门评审变动

## 一、全局流程总览

### 1.1 三模块角色定位

```
币种配置 ── 基础设施层 ── "提供数据的人"
    │ 提供：币种列表、汇率、精度、图标
    │ 特征：纯被调用，不调用别人，零内部依赖
    │
钱包模块 ── 核心业务层 ── "执行操作的人"
    │ 提供：余额管理、充值、兑换、提现、记录
    │ 特征：中间枢纽，调用币种配置和财务，同时被财务调用
    │
财务模块 ── 运营管理层 ── "审核决策的人"
    │ 提供：通道配置、支付方式、审核流程、统计报表
    │ 特征：与钱包双向依赖（钱包查配置，财务调钱包操作余额）
```

### 1.2 总体调用方向

```
                ┌───────────────┐
                │   C端用户      │
                │  (App / H5)   │
                └───────┬───────┘
                        │ HTTP请求
                        ▼
              ┌─────────────────────┐
              │   gate-font 网关     │
              │  (Hertz HTTP → RPC)  │
              └─────────┬───────────┘
                        │ RPC调用
        ┌───────────────┼────────────────┐
        ▼               ▼                ▼
┌──────────────┐ ┌──────────────┐ ┌──────────────┐
│  ser-coincfg  │ │  ser-wallet   │ │  ser-user    │
│  币种配置     │ │  多币种钱包   │ │  用户(已有)   │
│              │ │              │ │              │
│  R1~R7 RPC   │ │  E1~E6 RPC   │ │  KYC/Info    │
└──────┬───────┘ └──────┬───────┘ └──────────────┘
       ▲                │ ▲
       │ 查汇率/币种     │ │ 查配置/提交审核
       │                ▼ │ 操作余额(入账/冻结/解冻)
       │         ┌──────────────┐
       └─────────│  ser-finance  │
                 │  财务(他人)    │
                 └──────────────┘
```

### 1.3 数据流方向概括

```
配置数据流（从上到下）：
  B端管理员 → 币种配置B端 → 币种/汇率数据写入DB → RPC暴露给钱包/财务读取
  B端管理员 → 财务B端 → 通道/支付配置写入DB → RPC暴露给钱包读取

业务数据流（从左到右）：
  C端用户操作 → 钱包创建订单/执行操作 → 写入本地DB
  三方通道回调 → 财务确认 → 调用钱包RPC入账/扣款 → 更新钱包DB

资金操作流（双向）：
  钱包 → 财务：提交提现审核（同步，用户操作触发）
  财务 → 钱包：操作余额（异步，回调/审核触发）
```

## 二、币种配置模块内部流程

> 依据：需求理解c-3.md 第二节、链路关系c-3.md 第二节

### 2.1 模块内部子流程总览

```
币种配置模块内部有 4 条独立流程线：

① B端配置管理流程 ── 管理员维护币种数据
② 汇率自动更新流程 ── 定时任务拉取三方汇率
③ C端数据读取流程 ── 前端页面获取币种/汇率展示
④ RPC数据供给流程 ── 其他模块获取币种/汇率/精度信息
```

### 2.2 B端配置管理流程

**流程思路**：管理员通过B端后台维护币种信息，所有操作都是本地DB读写，不涉及任何外部模块调用。

```
1. 基准币种设置（一次性，不可逆）
   管理员登录B端 → 进入币种配置页面 → 顶部区域下拉选择基准币种(USDT)
   → 点击提交 → 系统弹出二级确认弹窗 "您确认要将平台基准币种设置为USDT吗?"
   → 管理员确认 → 写入基准币种表 → 提交按钮隐藏，后续不可修改
   → BSB价值(10BSB=1USDT)和锚定汇率由代码写入，不走配置

2. 币种编辑
   管理员点击列表行的"编辑" → 弹出编辑弹窗（币种代码+名称只读）
   → 可修改：汇率浮动%、偏差阈值%、币种图标(SVG/PNG≤5KB)、启用/禁用状态
   → 保存 → 更新币种配置表 → 产生操作日志记录
   → 影响：汇率浮动%变化会影响入款/出款汇率计算结果

3. 启用/禁用币种
   管理员操作启禁用开关 → 前置校验：基准币种(USDT)不可禁用
   → 禁用时：需检查是否有关联业务在用（若用户该币种有余额需提示）
   → 更新状态 → 产生操作日志记录
   → 影响：禁用后C端该币种不可使用、不可充值、不可提现

4. 操作日志
   每次编辑/启用/禁用操作自动写入操作日志表
   日志记录：操作类型(编辑/启用/禁用) + 操作对象(币种编号+代码) + 操作人 + 时间
```

**关键认知**：币种数据由开发代码入库（初始化脚本），后台仅做配置调整（图标/开关/汇率参数），不提供新增币种的UI入口。

### 2.3 汇率自动更新流程

**流程思路**：定时任务按配置频率自动拉取三方汇率，通过阈值机制判断是否更新平台汇率，确保汇率在可控范围内波动。

```
定时任务触发（频率可配，开发期间约1天1次，上线后按运营需求）
    │
    ▼
调用 3 家三方汇率API
    │
    ▼
3家数据取平均值 → 得到实时汇率
    │
    ▼
遍历每个已启用币种：
    │
    ├─ 该币种平台汇率是否已存在？
    │   │
    │   ├─ 不存在（首次）→ 直接以实时汇率作为平台汇率
    │   │   → 计算入款汇率 = 实时汇率 × (1 + 汇率浮动%)
    │   │   → 计算出款汇率 = 实时汇率 × (1 - 汇率浮动%)
    │   │   → 写入汇率日志
    │   │
    │   └─ 已存在 → 计算偏差值
    │       │
    │       │  入款偏差值 = |实时汇率 - 平台汇率| / 实时汇率 × 100%
    │       │  出款偏差值 = |实时汇率 - 平台汇率| / 实时汇率 × 100%
    │       │
    │       ├─ 入款偏差值 ≥ 阈值% 或 出款偏差值 ≥ 阈值%（任一超阈值即更新）
    │       │   → 更新该币种平台汇率为实时汇率
    │       │   → 重新计算入款汇率和出款汇率
    │       │   → 写入汇率日志记录
    │       │
    │       └─ 两个偏差值都 < 阈值%
    │           → 平台汇率保持不变，不产生汇率日志
    │
    └─ 处理下一个币种（各币种独立判断，互不影响）
```

**关键认知**：
- 手动刷新(B7)和自动刷新(T1)调用同一套核心逻辑
- 某个币种超阈值只更新该币种，其他币种不受影响
- 汇率精度：法币2位小数、加密货币8位小数

### 2.4 C端数据读取流程

```
C端请求 → gate-font网关路由 → ser-coincfg HTTP Handler
    │
    ├── C1 获取已启用币种列表
    │   → 读取币种配置表 WHERE status=启用
    │   → 返回币种代码/名称/类型/图标列表
    │
    ├── C2 获取指定币种汇率
    │   → 读取该币种的平台汇率 + 汇率浮动%
    │   → 实时计算入款汇率和出款汇率返回
    │
    └── C3 获取基准币种信息
        → 读取基准币种表
        → 返回基准币种代码(USDT) + BSB锚定比例信息
```

### 2.5 RPC数据供给流程

```
其他模块通过RPC调用 → ser-coincfg Kitex Handler
    │
    ├── R1 GetPlatformRate(币种A, 币种B)
    │   → 读取平台汇率 + 汇率浮动%
    │   → 返回入款汇率和出款汇率
    │   → 调用方：钱包兑换预览(W12)、充值页汇率展示
    │
    ├── R3 GetCurrencyConfig(币种代码)
    │   → 读取该币种完整配置（精度/千分位符号/小数点符号/图标）
    │   → 调用方：钱包查询余额(W1) 用于金额格式化
    │
    ├── R4 GetAllEnabledCurrencies()
    │   → 读取所有启用币种列表
    │   → 调用方：钱包获取可用币种(W2)
    │
    ├── R5 GetTransactionRate(币种A, 币种B, 方向)
    │   → 锁定当前平台汇率返回，用于交易期间汇率固定
    │   → 调用方：钱包创建充值/兑换/提现订单(W5/W7/W8/W13/W15)
    │   → 调用频率最高，每次创建订单都会调用
    │
    └── R7 IsCurrencyEnabled(币种代码)
        → 轻量级校验，返回布尔值
        → 调用方：钱包切换币种(W3)
```

### 2.6 币种配置模块流程小结

```
                     ┌──────────────┐
    B端管理员配置 ──► │   币种配置DB  │ ◄── 定时任务刷新汇率
                     │              │
                     │ 币种表       │
                     │ 汇率表       │
                     │ 基准币种表    │
                     │ 汇率日志表    │
                     │ 操作日志表    │
                     └──────┬───────┘
                            │
                    ┌───────┴───────┐
                    ▼               ▼
              C端HTTP读取      RPC暴露读取
              (C1~C3)         (R1~R7)
                                │
                    ┌───────────┼───────────┐
                    ▼           ▼           ▼
                 钱包模块     财务模块     竞彩域等

数据流向：配置写入 → DB → 数据读出
模块特征：单向数据流，只写只读，内部无复杂互调
```

## 三、钱包模块内部流程

> 依据：需求理解c-3.md 第三节、链路关系c-3.md 第三节、接口评估c-3.md 第二节

### 3.1 模块内部架构分层

```
┌─────────────────────────────────────────────────────────┐
│                   C端HTTP接口层                           │
│                                                         │
│  钱包首页组(W1~W3)  ── 余额/币种切换                      │
│  充值组(W4~W11)     ── 充值方式/创建订单/奖金/状态         │
│  兑换组(W12~W13)    ── 兑换预览/创建兑换                   │
│  提现组(W14~W22)    ── 提现方式/创建订单/账户管理           │
│  记录组(W23~W27)    ── 充值/提现/兑换/奖励4Tab+详情        │
│                                                         │
│  所有业务接口都向下调用"钱包核心服务层"                     │
├─────────────────────────────────────────────────────────┤
│                   钱包核心服务层（WalletCoreService）      │
│                                                         │
│  E1 GetBalance         ── 查询余额（读操作基础）           │
│  E2 CreditWallet       ── 加款（最高频的写操作）           │
│  E3 DebitWallet        ── 减款                            │
│  E4 FreezeBalance      ── 冻结余额                        │
│  E5 UnfreezeBalance    ── 解冻归还                        │
│  E6 ConfirmFrozenDeduction ── 确认扣除冻结资金            │
│                                                         │
│  这6个操作既通过RPC暴露给财务模块调用                      │
│  也被上层C端接口在内部复用（方法调用，非RPC）               │
├─────────────────────────────────────────────────────────┤
│                   数据持久层                              │
│                                                         │
│  钱包余额表 / 资金流水表 / 充值订单表 / 提现订单表          │
│  兑换流水表 / 提现账户表 / 奖励流水表                      │
└─────────────────────────────────────────────────────────┘
```

### 3.2 钱包首页流程（W1~W3）

**流程思路**：用户进入钱包首页，看到当前币种的余额和可操作入口，可以切换币种。

```
用户打开钱包首页
    │
    ├─ 第1步：W1 查询钱包余额
    │   → 读取钱包余额表（当前币种下各子钱包余额）
    │   → 外部调用：币种配置.R3(GetCurrencyConfig) → 获取精度/千分位规则
    │   → 格式化余额金额（如 VND: 9.000.000 / BSB: 1,000,000.00）
    │   → 计算资金钱包余额 = 中心钱包余额 + 奖励钱包余额
    │   → 返回：资金钱包总余额 + 各子钱包明细
    │
    ├─ 第2步：用户点击"切换"进入币种选择页
    │   → W2 获取用户可用币种列表
    │   → 外部调用：币种配置.R4(GetAllEnabledCurrencies) → 获取所有启用币种
    │   → 查询本地用户币种偏好表 → 标记当前选中币种
    │   → 展示币种列表（含各币种余额+选中标记）
    │
    └─ 第3步：用户点击目标币种
        → W3 切换当前操作币种
        → 外部调用：币种配置.R7(IsCurrencyEnabled) → 校验目标币种仍在启用
        → 更新本地用户当前币种设置
        → 切换后影响：所有后续操作（余额/充值/提现/兑换/记录）基于新币种
```

**关键认知**：
- W1~W3 不存在内部互调
- 币种切换是全局性操作，决定了后续所有业务的数据维度
- 不支持跨币种混合展示与操作

### 3.3 充值流程（W4~W11 + CB1/CB2）

**流程思路**：用户选择充值方式 → 输入金额 → 可选奖金 → 创建订单 → 支付 → 回调入账。充值有3种类型（USDT/银行/扫码），核心逻辑一致，差异在通道交互方式。

#### 3.3.1 充值总体流程

```
用户在当前币种下点击"充值"
    │
    ▼
① 获取充值方式列表 ── W4
    │ 外部调用：财务.F1(GetRechargeMethodList) 按币种+地区过滤
    │ 返回可用充值方式列表（如BSB: USDT推荐/银行/扫码）
    │ 各方式含：名称、图标、快捷金额档位、金额范围
    │
    │ 📌 BSB充值时页面需额外展示汇率信息
    │    → 调用币种配置.R1(GetPlatformRate) 获取入款汇率
    │    → 展示"汇率：1BSB = 0.1USDT"
    │
    ▼
② 用户选择奖金（可选）── W10
    │ 获取当前可用奖金活动列表
    │ 奖金互斥：同一笔充值仅限参与一个奖金活动
    │ 选择后联动：快捷金额标签带角标(+50%)，输入框右侧实时显示"额外+XXX"
    │ 门槛校验：金额 < 最低充值门槛时提示"该活动最低充值额为BSB 1000"
    │
    ▼
③ 用户选择充值方式+输入金额+点击"去充值"
    │
    ├── USDT充值 ── W5 ── 走 3.3.2 流程
    ├── 银行转账 ── W7 ── 走 3.3.3 流程
    └── 扫码充值 ── W8 ── 走 3.3.4 流程
```

#### 3.3.2 USDT充值子流程（W5 → CB2）

```
用户选择USDT充值，输入金额，点击"去充值"
    │
    ▼
W5 创建USDT充值订单
    │
    ├─ 第1步：重复转账检查 ── 内部调用W9
    │   → 查本地充值订单表：同UserID+同币种+同金额+状态(待支付/进行中)
    │   → 0笔 → 放行继续
    │   → 1-2笔 → 返回软拦截提示（用户可选"继续充值"或"查看详情"）
    │   → ≥3笔 → 返回硬拦截，阻止创建
    │   📌 注意：USDT链上充值不存在重复下单问题，W5可跳过W9或直接返回放行
    │
    ├─ 第2步：锁定汇率
    │   → 外部调用：币种配置.R5(GetTransactionRate) → 锁定当前汇率
    │
    ├─ 第3步：生成订单
    │   → 生成平台充值订单号(C+16位随机数)
    │   → 写入本地充值订单表（订单状态：待支付）
    │   → 订单包含：用户ID、币种、金额、汇率、奖金信息、订单号
    │
    ├─ 第4步：获取收款地址
    │   → 复用W6逻辑获取USDT地址（TRC20默认/ERC20/BEP20）
    │   → 返回：网络类型 + 收款地址(可复制) + 二维码数据
    │
    └─ 返回C端：平台承载页展示收款地址+二维码+风险提示
        │
        │ （用户在链上转账…等待区块确认…通常2-10分钟）
        │
        ▼
    CB2 USDT链上确认回调
        │ 区块链监控服务 → 确认到账 → 通知钱包模块
        │
        ├─ 幂等校验：同一订单号不重复处理
        ├─ 更新本地充值订单状态 → 支付成功
        ├─ 内部调用E2(CreditWallet)：订单金额 → 中心钱包
        ├─ 如有奖金：再次调用E2(CreditWallet)：奖金金额 → 奖励钱包
        └─ 写入资金流水记录
```

#### 3.3.3 银行转账充值子流程（W7 → CB1）

```
用户选择银行转账充值，输入金额，点击"去充值"
    │
    ▼
W7 创建银行转账充值订单
    │
    ├─ 第1步：重复转账检查 ── 内部调用W9（银行转账必须检查）
    │   → 1-2笔 → 软拦截弹窗"您有N笔同等金额交易正在进行中"
    │   → ≥3笔 → 硬拦截"为避免重复扣款，建议更换其它方式"
    │
    ├─ 第2步：锁定汇率 ── 同W5
    │
    ├─ 第3步：生成订单(C+16位) ── 同W5
    │
    ├─ 第4步：请求三方通道创建订单
    │   → 通道选择可能通过财务模块的轮询算法匹配
    │   → 无可用通道 → 提示"通道维护中"，订单状态:支付失败
    │   → 匹配成功 → 获取第三方跳转URL
    │
    └─ 返回C端：跳转三方承载页（展示银行信息+备注UserID+时间戳）
        │
        │ （用户在三方页面完成支付…等待回调…）
        │
        ▼
    CB1 充值通道回调通知（归属待评审：可能到财务模块）
        │
        ├─ 如果回调到财务模块（大概率）：
        │   → 财务验证金额一致性
        │   → 验证通过 → 财务调用钱包RPC E2(CreditWallet) 入账
        │   → 验证不通过 → 订单状态:支付失败
        │
        └─ 如果回调到钱包模块：
            → 钱包验证回调签名 + 幂等校验
            → 内部调用E2(CreditWallet) 入账
```

#### 3.3.4 扫码充值子流程（W8 → CB1）

```
W8 创建扫码充值订单
    │
    ├─ 流程与 W7 银行转账完全一致
    ├─ 唯一差异：第4步获取的不是跳转URL，而是二维码数据+倒计时
    └─ 三方承载页展示：二维码 + 金额(可复制) + 倒计时(到期订单失效) + 备注
```

#### 3.3.5 充值流程关键节点总结

```
各充值方式的共同处理节点：

  ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
  │ 重复转账  │──►│ 锁定汇率  │──►│ 生成订单  │──►│ 对接通道  │──►│ 回调入账  │
  │ 检查(W9) │   │ (R5)     │   │ (C+16位) │   │ (三方)   │   │ (E2)    │
  └──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘
       │                                              │               │
   软/硬拦截                                     平台承载/三方       幂等处理
  (银行/扫码必检)                                                  (不重复入账)
  (USDT可跳过)
```

### 3.4 兑换流程（W12~W13）

**流程思路**：兑换是最简洁的业务链路，仅依赖币种配置模块的汇率接口，所有资金操作在一个事务内本地完成。

```
用户在法币充值页面点击"兑换" → 进入兑换页面
    │
    ▼
① W12 兑换预览/汇率试算
    │ 外部调用：币种配置.R1(GetPlatformRate) → 获取入款汇率
    │ 内部复用：E1(GetBalance)逻辑 → 读取用户当前币种余额
    │ 本地计算并展示：
    │   "1 BSB = ₫2,620"
    │   兑换金额：₫X,XXX（用户输入法币金额）
    │   兑换获得：BSB xxx.xx
    │   赠送奖励(如10%)：BSB xx.xx
    │   实际到账：BSB xxx.xx
    │
    │ 输入校验：金额超出余额 → 提示"已达当前可兑换上限"
    │ 仅支持方向：法币(VND/IDR/THB) → BSB 单向，不可反向
    │
    ▼
② W13 创建兑换订单（用户点击"兑换"按钮）
    │
    ├─ 第1步：锁定兑换汇率
    │   → 外部调用：币种配置.R5(GetTransactionRate) → 锁定汇率
    │
    ├─ 第2步：再次校验余额
    │   → 复用E1逻辑或直接读表 → 确认余额 ≥ 兑换金额
    │
    ├─ 第3步：在同一个DB事务中执行三个操作 ⚠️ 关键
    │   ┌───────────────────────────────────────────────┐
    │   │ BEGIN TRANSACTION                              │
    │   │                                               │
    │   │ ① 扣减源币种中心钱包余额 ── 复用E3(DebitWallet)逻辑  │
    │   │    如：THB中心钱包 -฿1,000                     │
    │   │                                               │
    │   │ ② 增加BSB中心钱包余额 ── 复用E2(CreditWallet)逻辑   │
    │   │    如：BSB中心钱包 +285.71                     │
    │   │                                               │
    │   │ ③ 赠送部分增加BSB奖励钱包余额 ── 复用E2逻辑         │
    │   │    如：BSB奖励钱包 +28.57（赠送10%，需完成X倍流水）│
    │   │                                               │
    │   │ ④ 写入兑换流水记录                              │
    │   │                                               │
    │   │ COMMIT                                         │
    │   └───────────────────────────────────────────────┘
    │
    └─ 返回："兑换成功，获得BSB 314.28"（3秒内完成）
```

**关键认知**：
- W13内部复用E2/E3的核心逻辑（同一个Service内方法调用，不是RPC）
- 兑换操作全部在本地DB事务内完成，不涉及财务模块
- 赠送部分需关联稽核流水要求，完成X倍流水后才可提现
- 这是可以最早跑通的完整业务链路（仅依赖币种配置）

### 3.5 提现流程（W14~W22）

**流程思路**：提现是整个系统中最复杂的业务链路，涉及4个模块（钱包+币种配置+财务+用户），且钱包与财务存在双向异步调用。

#### 3.5.1 提现入口与KYC判断

```
用户在当前币种下点击"提现"
    │
    ▼
W14 获取提现方式列表
    │
    ├─ 第1步：查询KYC状态
    │   → 外部调用：用户.D9(GetKYCStatus)
    │   → 降级策略：KYC接口不可用 → 默认"未认证" → 仅允许USDT提现
    │   │
    │   ├─ 未认证 → 弹出温馨提示弹窗
    │   │   │
    │   │   ├── "去认证" → 跳转KYC认证页面
    │   │   │    （姓名/国家/手机号/证件类型/证件号/上传正反面）
    │   │   │    （认证成功后解锁全渠道提现）
    │   │   │
    │   │   └── "继续提现" → 仅展示USDT提现方式
    │   │
    │   └─ 已认证 → 无弹窗，直接进入提现方式选择
    │
    ├─ 第2步：获取提现方式
    │   → 外部调用：财务.F2(GetWithdrawMethodList) → 按币种获取全部提现方式
    │   → 与KYC状态取交集：
    │      已认证 → 返回全部方式（银行卡+电子钱包+USDT）
    │      未认证 → 仅保留USDT方式
    │
    └─ 返回C端：提现方式列表（含名称+到账时间标签）
```

#### 3.5.2 提现账户信息填写

```
用户选择某种提现方式（如银行卡）
    │
    ├─ W17/W19/W21 获取账户信息（本地读取）
    │   → 读取本地提现账户表
    │   → 首次：为空 → 进入填写流程
    │   → 非首次：返回已保存的账户信息（只读，不可自行修改）
    │
    ├─ 首次填写 → W18/W20/W22 保存账户信息
    │   │
    │   │ 银行卡(W18)：
    │   │   *银行名称(下拉) + *银行账号 + *持有人(KYC姓名自动填充,置灰)
    │   │   + 开户网点(选填)
    │   │
    │   │ 电子钱包(W20)：
    │   │   *钱包类型 + *手机号码 + *持有人(KYC姓名自动填充,置灰)
    │   │
    │   │ USDT(W22)：
    │   │   *网络类型 + *USDT地址（不涉及KYC姓名）
    │   │
    │   ├─ W18/W20需获取KYC姓名：
    │   │   → 外部调用：用户.D10(GetUserInfo) → 获取KYC认证姓名
    │   │   → 自动填充"账户持有人"字段（置灰不可编辑）
    │   │   → 校验：持有人姓名必须与KYC姓名一致（忽略大小写和多余空格）
    │   │
    │   └─ 保存到本地提现账户表（首次填写后复用，不可自行修改）
    │
    └─ 进入金额输入页
```

#### 3.5.3 提现金额输入与校验

```
W16 展示提现须知
    │ 外部调用：财务.F3(GetWithdrawConfig)
    │ 返回并展示：单笔限额(最低~最高)、单日限额、手续费率(%)、T+N到账时间
    │
    ▼
用户输入提现金额
    │
    ├─ 金额输入框支持"最大金额"快捷链接
    ├─ BSB提现时额外展示汇率："汇率：1 BSB = ฿3.50"
    ├─ 底部实时计算：提现金额 / 手续费(=金额×费率%) / 到账金额(=金额-手续费)
    │
    └─ 校验规则：
        ├─ 账户信息未填写完成 → 确认提现按钮置灰
        ├─ 超出每日限额 → 弹窗"您已超出当日提现限额"
        └─ 低于单笔最低 → 弹窗"您未达到单笔提现最低金额"
```

#### 3.5.4 提现订单创建与提交（最复杂的内部调用链）

```
用户点击"确认提现"
    │
    ▼
W15 创建提现订单 ⚠️ 这是钱包模块中最复杂的接口
    │
    ├─ 第1步：校验提现限额
    │   → 外部调用：财务.F3(GetWithdrawConfig) → 获取限额配置
    │   → 本地查询今日已提现总额 → 判断是否超出限额
    │   → 不通过 → 返回限额错误
    │
    ├─ 第2步：锁定出款汇率（仅BSB提现需要）
    │   → 外部调用：币种配置.R5(GetTransactionRate) → 锁定出款汇率
    │   → 法币直提不需要跨币种换算
    │
    ├─ 第3步：计算手续费
    │   → 手续费 = 提现金额 × 手续费率(来自F3配置)
    │   → 到账金额 = 提现金额 - 手续费
    │
    ├─ 第4步：冻结用户余额 ⚠️ 关键操作
    │   → 内部调用E4(FreezeBalance)的核心逻辑（方法调用，非RPC）
    │   → 校验：可用余额 ≥ 提现金额
    │   → 执行：可用余额减少，冻结余额增加
    │   → 写入钱包余额表
    │
    ├─ 第5步：生成订单
    │   → 生成平台提现订单号(T+16位随机数)
    │   → 写入本地提现订单表（订单状态：待风控审核）
    │
    ├─ 第6步：提交财务审核 ⚠️ 关键操作
    │   → 外部调用：财务.F4(SubmitWithdrawOrder)
    │   → 传入：订单号T+16位, 用户ID, 币种, 金额, 手续费,
    │          到账金额, 提现方式, 账户信息, 锁定汇率
    │   │
    │   ├─ 提交成功 → 进入财务审核流程
    │   │
    │   └─ 提交失败 → ⚠️ 必须回滚冻结！
    │       → 内部调用E5(UnfreezeBalance)逻辑 → 解冻归还
    │       → 返回用户"提现提交失败，请稍后重试"
    │
    │   ⚠️ 超时处理：
    │   如果F4调用超时，无法确定是否成功
    │   → 不能直接解冻（可能财务已接收）
    │   → 需要查询财务侧订单状态后决定
    │
    └─ 第7步：返回结果
        → 弹出"提现进行中"半屏弹窗
        → 显示：提现方式、订单号、金额、时间

⚠️ 第4步(冻结)和第6步(提交财务)之间的一致性设计：
    成功路径：冻结成功 → 提交财务成功 → 返回成功
    失败路径：冻结成功 → 提交财务失败 → 回滚解冻 → 返回失败
```

#### 3.5.5 提现后续流程（财务侧异步处理）

```
提现订单已提交到财务模块（状态：待风控审核）
    │
    │ ▼▼▼ 以下是财务模块异步处理，钱包模块被动接收RPC调用 ▼▼▼
    │
    ├─ 风控审核
    │   ├─ 驳回 → 财务调用钱包.E5(UnfreezeBalance) → 解冻归还冻结金额
    │   │         → 提现订单状态更新为"风控驳回"
    │   └─ 通过 → 进入财务审核（订单状态：待财务审核）
    │
    ├─ 财务审核
    │   ├─ 驳回 → 财务调用钱包.E5(UnfreezeBalance) → 解冻归还
    │   │         → 提现订单状态更新为"财务驳回"
    │   └─ 通过 → 进入出款环节（订单状态：待出款）
    │
    └─ 代付通道出款
        │ 财务轮询匹配可用代付通道 → 向通道发起出款请求
        │ 通道回传出款结果
        │
        ├─ 出款失败
        │   → 财务调用钱包.E5(UnfreezeBalance) → 解冻归还冻结金额
        │   → 订单状态：出款失败
        │
        └─ 出款成功
            → 财务调用钱包.E6(ConfirmFrozenDeduction) → 最终扣除冻结资金
            → 订单状态：出款成功
            → 资金真正离开用户钱包

📌 提现订单一生只会被调用一次 E5 或 E6（互斥）：
   驳回/出款失败 → E5(解冻归还)
   出款成功 → E6(确认扣除)
   幂等要求：同一订单号的E5/E6必须幂等，防止重复解冻或重复扣除
```

**流程思路**：记录模块是纯查询操作，4个Tab独立查表，不存在外部依赖和内部互调。

```
用户进入"记录"页面

    │
    ├─ 充值Tab(W23)
    │   → 读取本地充值订单表 WHERE currency=当前币种
    │   → 分页返回：金额/方式/时间/状态(已完成/充值中/失败-红)/详情
    │   → 数据来源：W5/W7/W8创建的充值订单 + CB1/CB2更新的状态
    │
    ├─ 兑换Tab(W25)
    │   → 读取本地兑换流水表 WHERE currency=当前币种
    │   → 分页返回：兑换金额/赠送X%/实际到账/时间

    │   → 分页返回：奖励金额/奖励类别/时间

    │
    └─ 记录详情(W27)
        → 根据订单号+类型读取对应表的详情
        → 提现详情：订单号/金额/手续费/到账金额/时间/状态/方式/持卡人/银行/卡号
        → 充值详情：订单号/订单金额/入账金额/状态/时间/方式/备注
        → 失败时显示原因 + "更换其他方式"链接
```

### 3.7 RPC暴露层流程（E1~E6）

**流程思路**：这6个接口是钱包模块对外暴露的原子能力，纯本地DB操作。被财务模块通过RPC调用，也被钱包内部C端接口复用。

```
E1 GetBalance（查询余额）
    │ 读取钱包余额表 → 返回各子钱包余额明细
    │ 调用方：财务人工加减款弹窗 + 钱包内部W12/W13/W15
    │ 纯读操作

E2 CreditWallet（加款）—— 被调用频率最高
    │ 往指定用户的指定子钱包增加余额
    │ 写入：钱包余额表 + 资金流水表
    │ 流水字段：操作类型/金额/订单号/时间/操作前余额/操作后余额
    │ 幂等设计：同一订单号不重复入账
    │ 调用方：财务(充值入账/补单/加款) + 钱包内部(W13兑换/CB1/CB2回调)

E3 DebitWallet（减款）
    │ 从指定用户的指定子钱包扣减余额
    │ 特殊规则：余额不足时实际扣款=当前余额（不报错）
    │ 写入：钱包余额表 + 资金流水表
    │ 调用方：财务(人工减款) + 钱包内部(W13兑换)

E4 FreezeBalance（冻结余额）
    │ 可用余额减少，冻结余额增加
    │ 校验：可用余额 ≥ 冻结金额
    │ 写入：钱包余额表（可用余额 + 冻结余额两字段变更）
    │ 调用方：钱包内部W15(创建提现)

E5 UnfreezeBalance（解冻归还）
    │ 冻结余额减少，可用余额恢复
    │ 写入：钱包余额表 + 资金流水表
    │ 幂等设计：同一订单号不重复解冻
    │ 调用方：财务(审核驳回/出款失败) + 钱包内部W15(提交失败回滚)

E6 ConfirmFrozenDeduction（确认扣除冻结资金）
    │ 冻结余额减少，资金真正离开用户钱包（确认出账）
    │ 写入：钱包余额表 + 资金流水表
    │ 幂等设计：同一订单号不重复确认
    │ 调用方：财务(出款成功)
    │ 仅通过RPC被财务调用，钱包内部不使用
```

**核心服务层复用关系图**：

```
WalletCoreService（对应E1~E6，共6个核心方法）
    │
    ├── GetBalance()      ← ExchangeService.Preview()(W12) 调用
    │
    ├── Credit()          ← ExchangeService.Execute()(W13) 调用
    │                     ← RechargeCallbackService.Handle()(CB1/CB2) 调用
    │                     ← 财务模块通过RPC调用（充值入账/补单/加款）
    │
    ├── Debit()           ← ExchangeService.Execute()(W13) 调用
    │                     ← 财务模块通过RPC调用（人工减款）
    │
    ├── Freeze()          ← WithdrawService.Create()(W15) 调用
    │
    ├── Unfreeze()        ← WithdrawService.Create()(W15) 失败回滚调用
    │                     ← 财务模块通过RPC调用（审核驳回/出款失败）
    │
    └── ConfirmDeduction() ← 仅财务模块通过RPC调用（出款成功）

这种设计确保核心资金操作（加/减/冻结/解冻/确认扣除）走同一套代码，
无论C端操作还是RPC调用，逻辑统一，保证一致性。
```

## 四、跨模块联动流程

### 4.1 钱包 → 币种配置 联动流程

> 依据：链路关系c-3.md 第四节

**联动性质**：单向同步调用。钱包是调用方，币种配置是提供方。

```
联动场景一：展示类（每次页面加载时调用）

  W1(余额查询) ──调用──→ R3(GetCurrencyConfig)
    时机：每次加载钱包首页
    获取：千分位符号/小数点符号/小数精度/图标
    用途：格式化金额展示（VND: 9.000.000 / BSB: 1,000,000.00）

  W2(币种列表) ──调用──→ R4(GetAllEnabledCurrencies)
    时机：加载钱包首页、进入币种切换页面
    获取：所有已启用币种（代码/名称/类型/图标）

  W3(切换币种) ──调用──→ R7(IsCurrencyEnabled)
    时机：用户点击切换币种
    获取：布尔值（防止切换到被B端禁用的币种）

  W12(兑换预览) ──调用──→ R1(GetPlatformRate)
    时机：进入兑换页面
    获取：平台入款汇率（展示"1 BSB = ₫2,620"）


联动场景二：交易类（创建订单时必须调用）

  W5/W7/W8(充值订单) ──调用──→ R5(GetTransactionRate)
    时机：用户点击"去充值"后
    获取：锁定的交易汇率（订单期间汇率不变）

  W13(兑换订单) ──调用──→ R5(GetTransactionRate)
    时机：用户点击"兑换"后
    获取：锁定的兑换汇率

  W15(提现订单) ──调用──→ R5(GetTransactionRate)
    时机：用户点击"确认提现"后
    获取：锁定的出款汇率（仅BSB提现需要）


调用频率排序：
  R5(GetTransactionRate) >> R3/R4(页面加载) >> R1(兑换预览) >> R7(切换)
```

### 4.2 钱包 → 财务模块 联动流程

> 依据：链路关系c-3.md 第五节

**联动性质**：同步调用。钱包是调用方，财务提供配置数据和审核入口。

```
联动场景一：获取配置数据

  W4(充值方式列表) ──调用──→ F1(GetRechargeMethodList)
    时机：C端进入充值页面
    传入：当前币种代码 + 用户地区
    获取：可用充值方式（名称/图标/支付类型/快捷金额/金额范围/排序）
    影响：F1不可用 → 用户看不到充值方式 → 无法充值

  W14(提现方式列表) ──调用──→ F2(GetWithdrawMethodList)
    时机：C端进入提现页面
    传入：当前币种代码
    获取：可用提现方式（名称/到账时间标签/排序）
    后处理：与KYC状态取交集后返回C端

  W16(提现限额费率) ──调用──→ F3(GetWithdrawConfig)
    时机：C端进入提现方式详情页
    传入：币种代码 + 提现方式
    获取：单笔限额(最低~最高)、单日限额、手续费率(%)、T+N

  W15(创建提现-校验) ──调用──→ F3(GetWithdrawConfig)
    时机：确认提现时的校验阶段
    用途：校验金额是否超出限额，计算手续费


联动场景二：提交业务数据

  W15(创建提现-提交) ──调用──→ F4(SubmitWithdrawOrder)
    时机：冻结余额成功后
    传入：订单信息（T+16位, 用户ID, 币种, 金额, 手续费, 到账金额,
           提现方式, 账户信息, 锁定汇率）
    获取：提交结果（成功/失败）
    失败处理：必须回滚冻结 → 调用本地UnfreezeBalance逻辑
```

### 4.3 财务 → 钱包 联动流程

> 依据：链路关系c-3.md 第七节

**联动性质**：异步RPC调用。财务是调用方，钱包是执行方。由通道回调或B端审核操作触发。

```
联动场景一：充值入账

  三方通道回调充值成功
    │
    ▼
  财务模块验证金额一致性
    │
    ├─ 通过 → 调用钱包.E2(CreditWallet)
    │   第1次：订单金额 → 中心钱包
    │     传入：user_id, wallet_type=CENTER, amount=订单金额,
    │            order_no=C+16位, source=RECHARGE
    │   如有奖金 → 第2次：奖金金额 → 奖励钱包
    │     传入：user_id, wallet_type=REWARD, amount=奖金额,
    │            order_no=C+16位, source=RECHARGE_BONUS
    │
    └─ 不通过 → 订单标记支付失败，不调用钱包


联动场景二：提现审核各节点

  提现订单生命周期中财务调用钱包的完整链路：

  ┌──────────┐
  │ 创建提现  │ → 钱包侧已冻结(E4) → 订单提交到财务(F4)
  └────┬─────┘
       ▼
  风控审核驳回 → 财务调用 → 钱包.E5(UnfreezeBalance) → 解冻归还
  财务审核驳回 → 财务调用 → 钱包.E5(UnfreezeBalance) → 解冻归还
  代付出款失败 → 财务调用 → 钱包.E5(UnfreezeBalance) → 解冻归还
  代付出款成功 → 财务调用 → 钱包.E6(ConfirmFrozenDeduction) → 确认扣除


联动场景三：人工修正

  充值补单审核通过 → 财务调用 → 钱包.E2(CreditWallet)
    → 补单金额→中心钱包 + 补单赠送→奖励钱包

  人工加款审核通过 → 财务调用 → 钱包.E2(CreditWallet) × N次
    → 按4种子钱包分别加款（中心/奖励/主播/代理，仅有金额的调用）

  人工减款审核通过 → 财务调用 → 钱包.E3(DebitWallet) × N次
    → 按4种子钱包分别扣减（余额不足时实际扣款=当前余额）

  人工加/减款创建时 → 财务调用 → 钱包.E1(GetBalance)
    → 获取各子钱包余额展示在弹窗中
```

## 八、关键处理要点汇总

### 8.1 资金操作一致性要点

```
① 兑换事务一致性
   → 扣源币种+加BSB+加赠送 必须在同一个DB事务内
   → 任一步失败全部回滚

② 提现冻结-提交一致性
   → 冻结余额(E4)成功 + 提交财务(F4)失败 → 必须回滚解冻(E5)
   → F4超时 → 不能直接解冻（可能财务已接收） → 需查询财务侧状态

③ 幂等处理
   → E2(CreditWallet)：同一订单号不重复入账
   → E5(UnfreezeBalance)：同一订单号不重复解冻
   → E6(ConfirmFrozenDeduction)：同一订单号不重复确认
   → CB1/CB2(充值回调)：同一回调不重复处理
```

### 8.2 消费优先级处理

```
资金钱包余额 = 中心钱包余额 + 奖励钱包余额

消费扣款顺序：
  ① 先扣中心钱包余额
  ② 中心不足部分再扣奖励钱包余额

混合投注返奖：
  如果单次投注使用了两个钱包的余额混合投注
  → 需根据投注金额占比分别计返奖
  → 中心钱包部分的返奖 → 中心钱包
  → 奖励钱包部分的返奖 → 奖励钱包
```

### 8.3 提现币种规则

```
充值方式 → 可提现方式：
  法币充值     → 法币提现
  USDT充值    → USDT提现
  法币+USDT混合 → 法币提现
  法币→BSB兑换  → 法币提现
```

### 8.4 订单号格式

```
C + 16位随机数 ── 充值订单
T + 16位随机数 ── 提现订单
B + 16位随机数 ── 充值补单
A + 16位随机数 ── 人工加款
M + 16位随机数 ── 人工减款
```

### 8.5 外部RPC不可用时的影响范围

```
币种配置全部不可用 → 几乎全部C端功能瘫痪（W1~W8, W12~W15）
  不受影响：W9,W10,W11,W17~W27,E1~E6

财务F1不可用 → 充值方式列表无法获取 → 无法充值
财务F2不可用 → 提现方式列表无法获取 → 无法提现
财务F3不可用 → 提现限额无法获取 → 无法提现
财务F4不可用 → 提现订单无法提交 → 需回滚冻结

用户D9不可用 → 降级为"未认证" → 仅USDT提现
用户D10不可用 → 无法自动填充KYC姓名 → 影响首次提现账户填写
```

## 九、流程思路总结

### 9.1 一句话概括每个子模块的流程特征

| 子模块 | 流程特征 | 核心复杂度来源 |
|--------|---------|-------------|
| 币种配置 | 配置写入→DB→数据读出，单向数据流 | 汇率阈值更新逻辑 |
| 钱包核心 | 6个原子操作，纯本地DB | 幂等设计、并发安全 |
| 充值 | 创建订单→对接通道→异步回调入账 | 3种通道差异、重复拦截、奖金联动 |
| 兑换 | 最简链路，一个事务搞定 | 事务内多子钱包操作一致性 |
| 提现 | 最复杂链路，4模块异步双向协作 | 冻结-提交一致性、超时处理、状态机 |
| 记录 | 纯查表展示，无复杂逻辑 | 数据来源依赖其他子模块产生订单 |

### 9.2 三条最关键的处理链路

```
第一条（依赖链最长）── 提现
  W14→(D9+F2) → W16→(F3) → W18→(D10) → W15→(R5+E4+F4) → 等待 → 审核→(E5/E6)
  涉及4个模块，异步双向调用，是系统联调最大风险点

第二条（调用频率最高）── 充值
  W4→(F1) → W9 → W5/W7/W8→(R5) → 等待 → CB1/CB2→(E2)
  涉及3个模块，回调入账需保证幂等

第三条（最简洁自治）── 兑换
  W12→(R1) → W13→(R5+E3+E2)
  仅涉及2个模块，本地事务完成，可最早跑通验证
```

### 9.3 总体实现思路

```
第一步：打好地基 ── 币种配置 + 钱包核心RPC
  → 为所有业务流程提供基础数据和原子操作能力

第二步：从简到难 ── 兑换 → 充值 → 提现
  → 兑换最简单先跑通，验证核心Service层设计
  → 充值需要财务配合，中等复杂度
  → 提现最复杂最后做，全链路联调

第三步：协调优先 ── IDL定义尽早锁定
  → 钱包RPC(E1~E6) + 财务RPC(F1~F4) 的Thrift IDL
  → 锁定后各自Mock对方独立开发
  → 最后真实联调重点验证：充值回调入账 + 提现审核全流程

核心原则：
  ① 钱包核心Service层(E1~E6)是共享基础 → 设计好了上层业务自然顺畅
  ② 对外RPC调用做好降级 → 币种配置加缓存 / KYC接口降级为未认证
  ③ 所有资金操作保证幂等 → 同一订单号不重复处理
  ④ 提现冻结-提交保证一致 → 提交失败必须回滚冻结
```
