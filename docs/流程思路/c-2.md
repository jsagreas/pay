# ser-wallet 整体流程思路

> 来源：需求理解、接口评估、依赖关系、链路关系四份分析文件的综合提炼
> 视角：**我负责实现钱包多币种模块（含币种配置子模块）**，需要清楚模块内、模块间的处理流程
> 目的：形成可落地的开发思路——知道先做什么、后做什么、每一步怎么处理、和谁联动

---

# 第一部分：全局流程架构

## 1. 三模块协作的本质

整个钱包体系由三个模块协作完成，它们的角色定位和协作关系如下：

```
┌──────────────────────────────────────────────────────────────────┐
│                        角色定位                                    │
│                                                                  │
│  币种配置 = "规则制定者"  →  定义有哪些币种、汇率怎么算、浮动多少    │
│  钱包模块 = "用户代言人"  →  面向用户提供充值/提现/兑换/余额管理     │
│  财务管理 = "执行管家"    →  管理支付通道、处理资金流转、审核把关     │
│                                                                  │
│  自己负责：币种配置 + 钱包模块（同属 ser-wallet 服务）               │
│  他人负责：财务管理（独立服务，通过 RPC 交互）                       │
└──────────────────────────────────────────────────────────────────┘
```

## 2. 数据流全景

所有业务场景的数据流方向可以归纳为五条线：

```
① 配置数据流（单向下沉）
   币种配置 ──→ 钱包模块 + 财务管理
   方向：底层往上流，币种和汇率数据是所有功能的基础

② 充值资金流（双向闭环）
   用户 → 钱包 → 财务 → 三方通道 → 财务 → 钱包
   方向：钱包发起3次RPC调财务，财务回调1~2次给钱包上分

③ 提现资金流（双向 + 异步审核）
   用户 → 钱包 → 财务 → [三级审核] → 三方通道 → 财务 → 钱包
   方向：钱包发起2次RPC调财务 + 1次调KYC，财务审核完异步回调钱包

④ 兑换资金流（完全内闭环）
   用户 → 钱包内部完成（调币种配置获取汇率，3次余额操作在同一事务中）
   方向：不跨模块，零RPC

⑤ 人工修正流（单向）
   管理员 → 财务 → 钱包
   方向：始终是财务主动调钱包（QueryBalance + Credit/Debit）
```

## 3. 初始化依赖层级

系统不是一上来就能用，有严格的配置前置关系，理解这个是制定开发顺序的基础：

```
Layer 0 (地基)         Layer 1 (通道)        Layer 2 (方式)        Layer 3 (业务可用)
━━━━━━━━━━━━━          ━━━━━━━━━━━━          ━━━━━━━━━━━━          ━━━━━━━━━━━━━━
自己负责               他人负责(财务)        他人负责(财务)         自己负责(钱包)

币种配置 ─────────→ 代收/代付通道 ────→ 充值/提现方式 ────→ 用户可充值/提现
(创建币种,配汇率)    (创建通道,配轮询)   (关联通道)

币种配置 ────────────────────────────────────────────────→ 用户可兑换
(汇率配置就绪)                                              (法币→BSB)
```

**核心结论：币种配置是整个系统的最底层依赖，必须最先开发完成。**

# 第二部分：币种配置模块的处理流程

> 币种配置是系统地基，100%独立无外部依赖，应最先开发

## 4. 币种配置整体流程思路

### 4.1 模块职责边界

币种配置只做三件事：
1. 管理平台支持哪些币种（CRUD + 启用禁用）
2. 管理每个币种的汇率体系（四层汇率 + 浮动比例 + 阈值）
3. 定时从第三方拉取实时汇率并判断是否更新平台汇率

### 4.2 币种管理流程

```
管理员操作流程:

新增币种:
  管理员 → 填写(名称,简称,icon,充值浮动%,提现浮动%,阈值%) → 提交
    │
    ├──→ 校验简称是否重复（不允许重复）
    ├──→ icon 上传到 S3（唯一的外部RPC依赖：ser-s3）
    ├──→ 写入币种配置表（初始状态：禁用）
    └──→ 返回成功

编辑币种:
  管理员 → 修改字段 → 提交
    │
    ├──→ 查当前配置
    ├──→ [若修改icon] 重新上传 S3
    ├──→ 更新配置表
    └──→ [若修改了浮动比例/阈值] 触发 RateService.RecalculateRates
              │
              ├── 新充值汇率 = 实时汇率 × (1 + 新充值浮动%)
              └── 新提现汇率 = 实时汇率 × (1 - 新提现浮动%)

启用/禁用币种:
  管理员 → 切换状态
    │
    ├──→ [禁用前] 可选安全校验：检查是否有活跃关联的通道/支付方式
    │         └── 有活跃关联 → 提示风险（禁用后关联通道的交易会受影响）
    └──→ 更新状态
```

### 4.3 汇率体系处理流程

这是币种配置模块最核心的逻辑，涉及四层汇率的计算和联动：

```
四层汇率关系:

  实时汇率（每分钟从3个第三方API取平均值）
       │
       │  偏差 = |实时 - 平台| / 实时 × 100%
       │  偏差 > 阈值 ？
       │
       ├── 是 → 更新平台汇率 = 实时汇率，记录变更日志
       │          │
       │          ├── 充值汇率 = 新平台汇率 × (1 + 充值浮动%)
       │          └── 提现汇率 = 新平台汇率 × (1 - 提现浮动%)
       │
       └── 否 → 平台汇率不变，充值/提现汇率也不变
```

### 4.4 定时任务处理流程

```
SyncThirdPartyRates（分钟级定时触发）
  │
  ├──→ 获取所有启用状态的法币币种
  │
  ├──→ 对每个法币:
  │      │
  │      ├──→ 调API_1获取汇率(currency→USDT)
  │      ├──→ 调API_2获取汇率(currency→USDT)
  │      ├──→ 调API_3获取汇率(currency→USDT)
  │      │
  │      ├──→ 实时汇率 = (rate1 + rate2 + rate3) / 3
  │      │      [如果某个API不可用，取剩余API的平均值]
  │      │      [如果全部不可用，使用最近一次缓存的汇率]
  │      │
  │      └──→ 更新实时汇率到数据库
  │
  └──→ 串行触发 CheckRateThreshold
         │
         ├──→ 对每个法币:
         │      │
         │      ├── 读取当前实时汇率和平台汇率
         │      ├── 计算偏差百分比
         │      │
         │      ├── [偏差 > 阈值]
         │      │      ├── 更新平台汇率
         │      │      ├── 联动更新充值/提现汇率（RecalculateRates）
         │      │      └── 写入汇率变更日志（记录时间、旧值、新值、触发原因）
         │      │
         │      └── [偏差 ≤ 阈值] → 跳过
         │
         └──→ 完成
```

### 4.5 币种配置对外提供的能力

币种配置通过三种方式向外提供数据：

```
方式一：同服务内部调用（钱包模块 → 币种配置 Service 层）
  · GetUserCurrentCurrency()   → 钱包几乎每个接口都调
  · GetPlatformRate()          → 兑换功能调用
  · GetCurrencyInfo()          → 币种切换时校验
  · GetAvailableCurrencies()   → 用户可用币种列表

方式二：RPC接口（财务模块通过RPC调用）
  · GetAvailableCurrencies     → 财务配通道/方式时选择币种
  · GetExchangeRate            → 财务做统计时基准币种换算
  · GetCurrencyInfo            → 财务展示币种信息

方式三：管理员B端操作
  · 8个B端接口提供币种CRUD和汇率管理
```

### 4.6 币种配置的开发思路

```
开发顺序（全部可独立开发，不等任何人）:

第一步: 数据模型 + 基础CRUD
  → currency_config 表（id, name, symbol, icon, status, 充值浮动%, 提现浮动%, 阈值%）
  → rate 表（currency_id, realtime_rate, platform_rate, deposit_rate, withdraw_rate）
  → rate_change_log 表（currency_id, old_rate, new_rate, reason, created_at）
  → 8个B端接口（全是CRUD，直接用 GORM Gen 生成 repo）

第二步: 汇率计算逻辑
  → RateService.RecalculateRates（被多处调用的核心方法）
  → 四层汇率的联动计算

第三步: 定时任务
  → SyncThirdPartyRates（调3个外部API）
  → CheckRateThreshold（阈值判断+联动更新）

第四步: RPC接口
  → 3个RPC接口（GetAvailableCurrencies, GetExchangeRate, GetCurrencyInfo）
  → 供钱包内部和财务模块调用

关键注意:
  · RecalculateRates 是被3处调用的核心方法（UpdateCurrency, UpdateRateConfig, CheckRateThreshold）
  · 所有汇率变更必须记录日志（审计需要）
  · 基准币种 USDT 配置后不可变更
  · BSB与USDT的锚定关系 10 BSB = 1 USDT 是硬编码常量
```

# 第三部分：钱包模块的处理流程

> 钱包模块是用户直接打交道的模块，包含余额管理、充值、提现、兑换、记录五大子模块

## 5. 钱包余额与币种切换子模块

### 5.1 余额查询流程

```
用户打开个人中心/钱包页
  │
  ▼
GetWalletInfo
  │
  ├──→ CurrencyService.GetUserCurrentCurrency()   // 获取当前选中币种
  │
  ├──→ WalletService.GetBalance(userId, currencyId, centerType)   // 中心钱包余额
  │
  ├──→ WalletService.GetBalance(userId, currencyId, rewardType)   // 奖励钱包余额
  │
  └──→ 组装返回:
         · 资金钱包余额 = 中心 + 奖励（用户前端展示的数字）
         · 中心钱包余额（详情展示）
         · 奖励钱包余额（详情展示）

特点: 纯内部操作，高频接口，考虑做缓存
```

### 5.2 币种切换流程

```
用户在个人中心选择切换币种
  │
  ▼
SwitchCurrency(targetCurrencyId)
  │
  ├──→ CurrencyService.GetCurrencyInfo(targetCurrencyId)
  │         │
  │         └── 币种不存在或已禁用 → 返回错误
  │
  ├──→ WalletService.EnsureWalletExists(userId, targetCurrencyId)
  │         │
  │         └── 该币种下用户钱包不存在?
  │               → 懒初始化：自动创建中心钱包(余额0) + 奖励钱包(余额0)
  │
  └──→ UserService.UpdateCurrentCurrency(userId, targetCurrencyId)
         │
         └── 更新成功后，用户后续所有操作（充值/提现/兑换/记录）
             都基于新币种

关键理解:
  · 币种切换是"全局开关"——切换后所有页面数据跟随变化
  · 钱包是懒初始化的——用户第一次切到某个币种才创建钱包
  · 每个币种的钱包体系完全独立（不同币种之间余额不互通）
```

### 5.3 钱包数据模型的处理思路

```
每个用户 × 每个币种 × 每种钱包类型 = 一行记录

用户A + VND + 中心钱包 → 余额: 100000 VND, 冻结: 0
用户A + VND + 奖励钱包 → 余额: 5000 VND
用户A + BSB + 中心钱包 → 余额: 500 BSB, 冻结: 0
用户A + BSB + 奖励钱包 → 余额: 50 BSB
用户A + USDT + 中心钱包 → 余额: 0（还没往USDT充过钱）

消费扣款顺序（重要规则）:
  1. 先扣中心钱包
  2. 中心钱包不够，再扣奖励钱包
  3. 两个加起来还不够 → 余额不足

冻结逻辑（提现时使用）:
  · 可用余额 = 余额 - 冻结金额
  · 用户提现时: 可用余额减少，冻结金额增加（总余额不变）
  · 提现成功: 冻结金额扣除（总余额减少）
  · 提现拒绝: 冻结金额返回到可用余额
```

## 6. 充值子模块处理流程

充值是跨模块耦合最重的流程，涉及4~5次跨模块RPC调用。

### 6.1 充值整体流程（三阶段）

```
阶段1: 页面加载
━━━━━━━━━━━━━━
  用户进入充值页
    │
    ├──→ [C端] GetDepositMethods(currencyId)
    │         │
    │         ├── [内部] 获取当前币种
    │         ├── [RPC→财务] 财务.GetDepositMethods(currencyId)
    │         │         │
    │         │         └── 财务查支付配置表 → 返回启用的充值方式列表
    │         │              [{名称, 类型(银行转账/QRIS/链上), icon, 最小/最大金额, 排序}]
    │         │
    │         └── 返回给用户: 可用充值方式卡片列表
    │
    └──→ [C端] GetDepositBonusList(currencyId)  // 可选，用户点"奖金"面板时
              │
              └── 返回: [{活动名, 奖金比例, 最低充值门槛, 稽核倍数}]

阶段2: 用户提交充值
━━━━━━━━━━━━━━━━━
  用户选择方式 + 输入金额 + (可选)选奖金活动 → 点击充值
    │
    ▼
  [C端] CreateDepositOrder(currencyId, methodId, amount, bonusActivityId?)
    │
    │  步骤①: 重复转账拦截（内部）
    ├──→ 查询: 同一用户 + 同一币种 + 同一金额的待处理订单数量
    │         ├── 0笔 → 正常通过
    │         ├── 1~2笔 → 返回警告，前端弹窗让用户确认后重新提交(带确认标记)
    │         └── ≥3笔 → 直接阻断，不允许提交
    │
    │  步骤②: 匹配支付通道（RPC→财务）
    ├──→ 财务.MatchDepositChannel(currencyId, methodId, amount)
    │         │
    │         │  财务内部处理:
    │         ├── 查该币种+方式类型对应的轮询规则
    │         ├── 筛选: 金额在范围内 + 未超日交易上限 + 通道启用
    │         ├── 按权重轮询选择一个通道
    │         │     ├── 选中 → 返回 {channelId, channelCode}
    │         │     └── 全部不可用 → 返回错误
    │         │
    │         └── 匹配失败 → 钱包返回"通道繁忙，请换一种方式"
    │
    │  步骤③: 创建通道订单（RPC→财务）
    ├──→ 财务.CreateChannelOrder(channelId, userId, amount, orderNo)
    │         │
    │         │  财务内部处理:
    │         ├── 生成财务侧订单记录
    │         ├── 调三方通道API创建支付订单
    │         │     ├── 成功 → 获取 payUrl/qrCode + expireTime
    │         │     └── 失败 → 返回通道错误
    │         │
    │         └── 返回: {thirdPartyOrderNo, payUrl, expireTime}
    │
    │  步骤④: 创建钱包侧充值订单（内部）
    ├──→ 写入订单表: 状态=待支付, 订单号=C+16位随机数
    │
    └──→ 返回给用户: {orderNo, payUrl/qrCode, expireTime}
         用户被引导到三方支付页面

阶段3: 支付回调（异步，由财务触发）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  用户在三方支付页面完成支付
    │
    ▼
  三方通道 ──回调──→ 财务管理模块
    │
    │  财务内部: 验签 + 校验订单 + 更新财务侧订单状态
    │
    │  跨模块回调:
    ├──→ [RPC] ser-wallet.Credit(userId, currencyId, centerType, actualAmount, orderNo)
    │         │
    │         │  钱包内部处理:
    │         ├── 幂等校验: 该orderNo是否已处理过? 是→直接返回成功
    │         ├── 中心钱包余额 += actualAmount
    │         ├── 更新订单状态 → "已完成"
    │         └── 创建流水记录
    │
    │  [如果选了奖金活动]
    ├──→ [RPC] ser-wallet.CreditReward(userId, BSB_ID, rewardType, bonusAmount, orderNo)
    │         │
    │         ├── 奖励钱包余额 += bonusAmount
    │         └── 创建奖励记录(来源=充值奖金, 稽核倍数=X)
    │
    └── 充值流程完成
```

### 6.2 充值流程的关键处理要点

```
① 幂等性保证:
   · Credit接口必须基于orderNo做幂等——同一个orderNo只能上分一次
   · 通道可能重复回调，必须能安全处理

② 金额一致性:
   · 实际到账金额可能与用户填的不完全一致（通道手续费）
   · 以通道回调的实际金额为准，不以用户提交的金额为准

③ 重复转账拦截:
   · 是前置检查，不是绝对防重——最终防重靠幂等校验
   · 检查维度: 同一用户 + 同一币种 + 同一金额 + 待处理状态

④ 奖金处理:
   · 充值本金 → 中心钱包
   · 奖金部分 → 奖励钱包（需完成稽核才能转到中心钱包）
   · 一笔充值只能选一个活动

⑤ 钱包方对充值的控制范围:
   · 钱包控制: 重复拦截、订单创建、余额增加
   · 财务控制: 通道匹配、通道订单、回调处理
   · 钱包无法控制: 通道是否可用、支付是否成功、回调是否及时
```

## 7. 提现子模块处理流程

提现流程比充值多了KYC校验、资金来源规则、冻结机制和三级审核。

### 7.1 提现整体流程（四阶段）

```
阶段1: 资格校验 + 页面加载
━━━━━━━━━━━━━━━━━━━━━━━━
  用户进入提现页
    │
    ├──→ [C端] CheckWithdrawEligibility(currencyId)
    │         │
    │         │  ① KYC校验（RPC→外部服务）
    │         ├──→ ser-kyc.GetKycStatus(userId)
    │         │         └── 返回: {kycLevel, isVerified}
    │         │
    │         │  ② 资金来源规则判断（内部）
    │         ├──→ 分析该用户所有充值记录的币种类型:
    │         │      · 只充过法币 → allowFiat=true, allowUSDT=false
    │         │      · 只充过USDT → allowFiat=false, allowUSDT=true
    │         │      · 都充过 → allowFiat=true, allowUSDT=false（混合→只能法币提）
    │         │
    │         │  ③ 综合判断
    │         ├──→ 法币提现 + 未KYC → "需要先完成实名认证"
    │         │    USDT提现 → KYC可选
    │         │
    │         └── 返回: {eligible, allowedTypes, kycRequired}
    │
    └──→ [C端] GetWithdrawMethods(currencyId)
              │
              ├──→ [RPC→财务] 财务.GetWithdrawMethods(currencyId)
              │         └── 返回: 该币种下启用的提现方式列表
              │
              ├──→ 根据资格校验结果过滤掉不允许的方式类型
              │
              └── 返回给用户: 过滤后的可用提现方式

阶段2: 用户提交提现
━━━━━━━━━━━━━━━━━
  用户选方式 + 输入金额 + 填收款信息 → 点击提现
    │
    ▼
  [C端] CreateWithdrawOrder(currencyId, methodId, amount, receiverInfo)
    │
    │  ① 余额校验
    ├──→ 查可用余额（余额 - 已冻结金额）
    │         └── 可用余额 < amount → "余额不足"
    │
    │  ② 限额校验
    ├──→ 查今日已提现累计金额
    │         ├── 单笔超限 → "超过单笔提现限额"
    │         └── 今日累计超限 → "超过每日提现限额"
    │
    │  ③ 资金来源规则校验（二次校验）
    ├──→ 确认选择的提现方式类型符合资金来源规则
    │
    │  ④ 冻结金额（关键步骤）
    ├──→ WalletService.Freeze(userId, currencyId, amount)
    │         │
    │         └── 可用余额 -= amount, 冻结金额 += amount
    │             （总余额不变，但用户不能再用这笔钱充值/兑换/消费）
    │
    │  ⑤ 创建钱包侧提现订单
    ├──→ 写入订单表: 状态=审核中, 订单号=T+16位随机数
    │
    │  ⑥ 提交到审核流程（RPC→财务）
    ├──→ 财务.SubmitWithdrawRequest(orderNo, userId, amount, methodId, receiverInfo)
    │         │
    │         └── 财务创建审核记录，状态=风控审核中
    │
    └──→ 返回给用户: {orderNo, status="审核中"}

阶段3: 三级审核（财务内部，异步进行）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  ┌─────────────────────────────────────────────┐
  │  风控审核 ──→ 通过? ──→ 财务审核 ──→ 通过?    │
  │              └ 拒绝→⑧    │          └ 拒绝→⑧ │
  │                          ▼                   │
  │              出款审核 ──→ 通过? ──→ 出款       │
  │                          └ 拒绝→⑧            │
  └─────────────────────────────────────────────┘

  钱包在此阶段无需参与，等待财务回调即可

阶段4: 审核结果回调（财务→钱包，异步）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  审核通过 + 出款成功:
    │
    ├──→ [RPC] ser-wallet.Debit(userId, currencyId, amount, orderNo)
    │         │
    │         ├── 幂等校验
    │         ├── 从冻结金额中扣除（不是从可用余额扣）
    │         ├── 更新订单状态 → "已完成"
    │         └── 钱已到用户外部账户
    │
  审核拒绝 / 出款失败:
    │
    └──→ [RPC] ser-wallet.Unfreeze(userId, currencyId, amount, orderNo)
              │
              ├── 冻结金额 -= amount, 可用余额 += amount
              └── 更新订单状态 → "已拒绝" / "出款失败"
```

### 7.2 提现流程的关键处理要点

```
① 冻结机制的必要性:
   · 用户提交提现后到审核完成可能经过数小时甚至数天
   · 如果不冻结，用户可以在等待期间把要提的钱花掉
   · 冻结后: 可用余额减少，用户无法超额使用
   · 冻结不影响总余额的展示

② 资金来源规则:
   · 纯法币充值 → 只能法币提现
   · 纯USDT充值 → 只能USDT提现
   · 混合充值 → 只能法币提现（风控要求）
   · 这个规则的判断需要查历史充值记录

③ 提现扣款的特殊处理:
   · 提现成功时的Debit不是扣"可用余额"，而是扣"冻结金额"
   · 实现上: DebitFromFrozen(减少冻结金额和总余额)
   · 与普通消费的Debit(减少可用余额)逻辑不同

④ KYC校验策略:
   · 法币/电子钱包提现 → KYC必须完成
   · USDT提现 → KYC可选
   · 如果KYC服务不可用 → 只影响提现，不影响其他功能
   · 考虑降级策略: KYC服务超时时，允许USDT提现但禁止法币提现

⑤ 钱包方对提现的控制范围:
   · 钱包控制: 资格校验、余额/限额校验、冻结/解冻、订单管理
   · 财务控制: 三级审核、通道出款
   · 钱包在审核阶段是被动的: 只等财务回调Debit或Unfreeze
```

## 8. 兑换子模块处理流程

兑换是唯一完全自闭环的业务，零跨模块调用。

### 8.1 兑换整体流程

```
阶段1: 汇率查询
━━━━━━━━━━━━━━
  用户进入兑换页
    │
    ▼
  [C端] GetExchangeRate(currencyId)
    │
    ├──→ CurrencyService.GetPlatformRate(currencyId)  // 内部Service调用
    │
    └──→ 计算并返回:
           · 当前平台汇率（例: 1 USDT = 25000 VND）
           · 1单位法币 = X个BSB（基于平台汇率 + 10BSB=1USDT锚定）
           · 兑换奖金比例 = 10%
           · 用户输入金额后实时计算预估到账BSB

阶段2: 确认兑换
━━━━━━━━━━━━━━
  用户输入法币金额 → 确认兑换
    │
    ▼
  [C端] CreateExchangeOrder(currencyId, amount)
    │
    │  ① 获取下单时刻的汇率（不是查询时的，防止汇率变动导致差异）
    ├──→ CurrencyService.GetPlatformRate(currencyId)
    │
    │  ② 计算
    ├──→ bsbAmount = amount × platformRate / BSB_USDT_RATE
    │    bonusAmount = bsbAmount × 10%（兑换奖金）
    │
    │  ③ 校验法币余额
    ├──→ WalletService.GetBalance(userId, currencyId, centerType)
    │         └── 余额 < amount → "余额不足"
    │
    │  ④ 执行兑换（必须在同一个数据库事务中）
    ├──→ 【事务开始】
    │      ├── WalletService.Debit(userId, currencyId, centerType, amount)    // 扣法币
    │      ├── WalletService.Credit(userId, BSB_ID, centerType, bsbAmount)    // 加BSB
    │      ├── WalletService.Credit(userId, BSB_ID, rewardType, bonusAmount)  // 奖金加BSB奖励钱包
    │      └── OrderService.CreateExchangeRecord(...)                         // 记录
    │    【事务结束】
    │
    └──→ 返回: {success, exchangedBSB, bonus}
```

### 8.2 兑换流程的关键处理要点

```
① 事务原子性（最重要）:
   · 扣法币中心钱包 + 加BSB中心钱包 + 加BSB奖励钱包 = 3个余额操作
   · 这3个操作跨越不同币种的钱包记录
   · 必须在同一个数据库事务中完成，任何一步失败全部回滚
   · 这是整个系统中对事务要求最严格的操作

② 并发安全:
   · 用户可能同时发起多次兑换（快速点击）
   · 需要在扣款时使用乐观锁或行锁，防止超扣
   · 建议: SELECT ... FOR UPDATE 锁定法币钱包行 → 检查余额 → 执行操作

③ 汇率时点:
   · 使用下单时刻的汇率，而不是页面展示时的汇率
   · 页面展示的汇率仅供参考，实际兑换以提交时查询的平台汇率为准
   · 如果汇率变动大，可以提示用户"汇率已变化，请确认"

④ 兑换方向限制:
   · 只支持 法币 → BSB（单向）
   · BSB 不能换回法币（如需变现只能走提现流程）
   · 当前支持: VND→BSB, IDR→BSB, THB→BSB

⑤ 兑换奖金:
   · 10%额外BSB奖金，进入奖励钱包
   · 奖金需完成X倍流水的稽核才能转入中心钱包
   · 稽核逻辑（TransferToCenter）由稽核系统触发
```

## 10. 钱包核心RPC子模块处理流程

> 这些接口不面向用户，是被财务管理模块调用的"被动能力"

### 10.1 五个核心RPC的处理流程

```
Credit（加款/上分）—— 被调用场景: 充值成功、补单通过、人工加款通过
━━━━━━━━━━━━━━━━
  输入: userId, currencyId, walletType(center/reward), amount, orderNo, reason
    │
    ├──→ 幂等校验: orderNo是否已处理过?
    │         ├── 是 → 直接返回成功（幂等保证）
    │         └── 否 → 继续
    │
    ├──→ WalletService.Credit → 指定钱包余额 += amount
    │         └── 乐观锁/行锁保证并发安全
    │
    └──→ 创建流水记录

Debit（扣款/下分）—— 被调用场景: 提现完成、人工减款通过、消费扣款
━━━━━━━━━━━━━━━━
  输入: userId, currencyId, walletType, amount, orderNo, reason
    │
    ├──→ 幂等校验
    │
    ├──→ 查余额: 余额 >= amount ?
    │         └── 否 → 返回"余额不足"错误
    │
    ├──→ WalletService.Debit → 指定钱包余额 -= amount
    │
    └──→ 创建流水记录

Freeze（冻结）—— 被调用场景: 提现创建时
━━━━━━━━━━━━━━
  输入: userId, currencyId, amount, orderNo
    │
    ├──→ 查可用余额: (余额 - 已冻结) >= amount ?
    │         └── 否 → 返回"可用余额不足"
    │
    ├──→ 可用余额 -= amount, 冻结金额 += amount
    │
    └──→ 创建冻结记录

Unfreeze（解冻）—— 被调用场景: 提现拒绝、出款失败
━━━━━━━━━━━━━━━━
  输入: userId, currencyId, amount, orderNo
    │
    ├──→ 冻结金额 -= amount, 可用余额 += amount
    │
    └──→ 创建解冻记录

QueryBalance（查询余额）—— 被调用场景: 人工加减款前查余额
━━━━━━━━━━━━━━━━━━━━━
  输入: userId, currencyId
    │
    └──→ 返回: {中心钱包余额, 奖励钱包余额, 冻结金额, 可用余额}
```

### 10.2 RPC接口的关键设计原则

```
① 幂等性:
   · Credit/Debit 必须基于 orderNo 幂等
   · 同一个 orderNo 无论调多少次，只执行一次
   · 财务回调可能重试，不能重复加/扣款

② 原子性:
   · 余额操作 + 流水记录 在同一个事务中
   · 不能出现"钱加了但没记录"或"记录了但钱没加"

③ 并发安全:
   · Credit/Debit/Freeze/Unfreeze 可能被并发调用
   · 使用行锁（SELECT FOR UPDATE）或乐观锁（version字段）
   · 防止: 超扣、重复加、冻结金额与实际不一致

④ 这些接口只被调入，不调出:
   · 它们是纯粹的"余额操作原子能力"
   · 不需要知道调用方的业务逻辑（不关心是充值还是补单）
   · 通过 reason 参数记录操作原因，仅用于日志/审计
```

# 第四部分：跨模块联动流程

## 11. 财务管理模块联动钱包的完整流程

### 11.1 人工修正流程（财务→钱包，单向）

```
充值补单流程:
━━━━━━━━━━━
  管理员A: 发现用户充值了但系统没到账（通道回调丢失）
    │
    ▼
  [B端-财务] 创建补单
    │
    ├──→ 输入原充值订单号 → 查原订单确认存在且状态为失败/超时
    ├──→ 检查: 该订单是否已补过（同一订单只能补一次）
    ├──→ [RPC] ser-wallet.QueryBalance(userId, currencyId) → 展示当前余额
    ├──→ 创建补单记录（订单号: B+16位, 状态: 待审核）
    │
    ▼
  管理员B(≠管理员A): 审核通过
    │
    ├──→ [RPC] ser-wallet.Credit(userId, currencyId, centerType, amount, "B...")
    │         └── 钱包: 中心钱包余额增加
    └── 补单完成

人工加款流程:
━━━━━━━━━━━
  管理员A: 给用户加钱（赠送奖金/活动补偿/客诉赔付）
    │
    ├──→ [RPC] ser-wallet.QueryBalance → 查当前余额
    ├──→ 创建加款记录（订单号: A+16位, 必须填备注/原因）
    │
    ▼
  管理员B审核通过
    │
    ├──→ [RPC] ser-wallet.Credit(userId, currencyId, centerType, amount, "A...")
    └── 加款完成

人工减款流程:
━━━━━━━━━━━
  管理员A: 给用户扣钱（纠正错误加款/违规处罚）
    │
    ├──→ [RPC] ser-wallet.QueryBalance → 查余额，余额不足则提示
    ├──→ 创建减款记录（订单号: M+16位, 必须填备注/原因）
    │
    ▼
  管理员B审核通过
    │
    ├──→ [RPC] ser-wallet.QueryBalance → 再次确认余额（可能审核期间用户花了）
    ├──→ [RPC] ser-wallet.Debit(userId, currencyId, centerType, amount, "M...")
    │         └── 余额不足 → 审核通过但执行失败
    └── 减款完成

共同规则:
  · 创建方 ≠ 审核方（双人复核）
  · 金额只能是正数
  · 必须填写备注/操作原因
```

### 11.2 跨模块调用方向全景汇总

```
钱包 ──调出──→ 财务（6个RPC接口）:
  1. GetDepositMethods      ← 充值页加载时
  2. MatchDepositChannel    ← 充值提交时匹配通道
  3. CreateChannelOrder     ← 匹配到通道后创建订单
  4. GetWithdrawMethods     ← 提现页加载时
  5. SubmitWithdrawRequest  ← 提现提交时送入审核
  6. QueryChannelOrderStatus← 查通道订单状态

财务 ──调入──→ 钱包（7个RPC接口）:
  1. QueryBalance           ← 人工修正查余额
  2. Credit                 ← 充值成功/补单/人工加款
  3. Debit                  ← 提现完成/人工减款
  4. Freeze                 ← 提现创建时冻结
  5. Unfreeze               ← 提现拒绝/失败时解冻
  6. QueryWalletDetail      ← B端查看用户钱包
  7. CreditReward           ← 奖金发放

钱包 ──调出──→ 其他服务:
  1. ser-kyc.GetKycStatus   ← 提现资格校验
  2. ser-buser.GetUserInfo  ← B端查看用户信息
  3. ser-s3.Upload          ← 币种icon上传
  4. 三方汇率API × 3        ← 定时拉取汇率
```

# 第五部分：开发阶段流程思路

## 12. 总体开发策略

基于依赖关系分析，约70%的接口可以独立开发，30%需要联调。开发策略分三个阶段：

### 12.1 阶段一：独立开发（不等任何人）

```
目标: 把所有不依赖外部模块的接口全部开发完毕

组A: 币种配置全模块（~13个接口/任务）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  开发顺序:
  1. 数据模型设计 + GORM Gen 生成 model/query/repo
  2. CurrencyConfigService（币种CRUD+状态管理）
  3. RateService（汇率计算+联动更新+变更日志）
  4. 8个B端接口（全是CRUD + 配置管理）
  5. 3个RPC接口（对外暴露币种和汇率数据）
  6. 2个定时任务（汇率同步+阈值检查）

  验证标准:
  · 能在B端完成币种创建、编辑、启用/禁用
  · 汇率配置生效，四层汇率计算正确
  · 定时任务能拉取外部汇率并判断是否更新平台汇率
```

# 第六部分：关键技术决策点

## 13. 每个子模块的技术关键点总结

### 13.1 必须保证的原子性操作

```
场景一: 兑换操作（3次余额变更）
  扣法币中心钱包 + 加BSB中心钱包 + 加BSB奖励钱包
  → 必须在同一个MySQL事务中
  → 涉及不同币种的钱包行，需要统一加锁顺序防死锁

场景二: Credit/Debit（余额变更 + 流水记录）
  余额变更 + 写流水记录表
  → 同一事务

场景三: Freeze（可用余额减少 + 冻结金额增加）
  两个字段的变更
  → 同一条UPDATE语句即可
```

### 13.2 必须保证的幂等操作

```
Credit: 基于orderNo幂等（充值回调/补单/加款可能重试）
Debit: 基于orderNo幂等（提现扣款可能重试）
Freeze: 基于orderNo幂等（提现冻结不能重复冻结）
Unfreeze: 基于orderNo幂等（解冻不能重复解冻）

实现方式:
  方案A: 独立去重表（orderNo唯一索引，处理前先INSERT，失败说明已处理）
  方案B: 订单状态机（查订单状态，已完成的直接返回成功）
```

### 13.3 必须考虑的并发场景

```
场景一: 用户同时发起兑换和充值
  兑换在扣法币余额时 + 充值回调在加法币余额时
  → 需要行锁保证余额一致

场景二: 用户快速重复点击兑换
  两次兑换请求并发到达
  → 第一次扣款后余额不足，第二次应该返回"余额不足"

场景三: 财务并发回调
  充值成功回调 + 人工加款回调 同时到达
  → 两次Credit都应该成功，余额正确累加

建议:
  · 所有余额变更使用 SELECT ... FOR UPDATE 行锁
  · 加锁顺序: 按 (userId, currencyId, walletType) 排序，防止死锁
```

# 第七部分：流程检查清单

## 15. 每个业务场景的处理要点速查

### 15.1 充值流程检查清单

```
□ 重复转账拦截: 同用户+同币种+同金额，≥3笔待处理则阻断
□ 通道匹配失败: 返回"通道繁忙"，引导用户换方式
□ 通道订单创建失败: 不创建钱包侧订单，返回错误
□ 充值回调幂等: 同一orderNo只上分一次
□ 奖金活动: 充值本金→中心钱包，奖金→奖励钱包
□ 一笔充值只能选一个活动
□ 实际到账金额以通道回调为准
□ 订单号格式: C+16位随机数
```

### 15.2 提现流程检查清单

```
□ KYC校验: 法币/电子钱包提现必须KYC，USDT可选
□ 资金来源规则: 纯法币充→法币提，纯USDT充→USDT提，混合→法币提
□ 余额校验: 用可用余额(余额-冻结)而不是总余额
□ 限额校验: 单笔限额 + 每日累计限额
□ 冻结机制: 提交后立即冻结，审核期间用户无法使用这笔钱
□ 审核通过: Debit扣的是冻结金额，不是可用余额
□ 审核拒绝: Unfreeze恢复冻结金额到可用余额
□ 审核回调幂等: Debit/Unfreeze基于orderNo幂等
□ 订单号格式: T+16位随机数
```

### 15.3 兑换流程检查清单

```
□ 方向限制: 只支持法币→BSB，不支持反向
□ 汇率时点: 使用下单时刻的汇率，不用查询时的
□ 事务原子性: 扣法币+加BSB+加奖金 三个操作在同一事务
□ 并发安全: 行锁防止超扣
□ 奖金计算: 兑换金额的10%额外BSB，进奖励钱包
□ BSB锚定: 10 BSB = 1 USDT（硬编码常量）
```

### 15.4 RPC接口检查清单

```
□ Credit/Debit: 基于orderNo幂等
□ 余额变更: 行锁保证并发安全
□ 余额变更 + 流水记录: 同一事务
□ Debit: 余额不足时返回明确错误（不是默默失败）
□ Freeze: 可用余额不足时返回错误
□ 这些接口不关心调用方的业务逻辑，只负责可靠地操作余额
```

### 15.5 币种配置检查清单

```
□ 基准币种USDT: 配置后不可变更
□ 币种简称: 不允许重复
□ 禁用币种: 需提示关联通道/方式的影响
□ 修改浮动比例: 联动更新充值/提现汇率
□ 汇率变更: 必须记录变更日志
□ 定时任务: 第三方API全挂时使用缓存汇率
□ 阈值机制: 偏差超阈值才更新平台汇率（避免频繁波动）
```
