## 元认知核心理解（行为准则）

### 一、严格约束（红线规则）

**第一约束：默认只读**
- 所有操作默认只读，未经用户口头明确同意，严禁任何增删改操作
- 分析阶段严禁产生编码行为，分析结束后不得擅自进入编码阶段

**行为红线：**
- 禁止猜测/臆想 → 必须基于分析或问询获得明确信息
- 禁止片面分析 → 必须全面完整深入，不可依赖局部信息做推理
- 禁止擅自编码 → 必须得到用户明确同意后才能编码
- 禁止无中生有 → 反馈问题必须有依据，不能把没问题的说成有问题
- 禁止无脑咨询 → 能通过分析工程代码得到的信息，不得抛给用户
- 禁止想当然 → 凡事要有依据、凭证、理由
- 要有主见 → 基于充分分析后据理力争，有则改之无则加勉

**思考准则：**
- 宏观：清楚架构、模块功能、文件作用
- 微观：清楚代码职责、函数作用、变量含义
- 闭环：模块间关系、文件间依赖、完整调用链路
- 深入：穷极思想追代码逻辑，不浅尝辄止
- 全面：架构 + 调用链路 + 流程 + 设计思路 + 标准规范 + 接口细节 + 技术栈

**互动节奏：**
- 分步骤、分阶段、小步慢跑迭代推进
- 先给思路步骤 → 拆解需求 → 确认 → 执行 → 验证 → 推进
- 编码前必须确认，编码后必须验收

**执行路径（双轨并行）：**
- 轨道1：深度分析项目脚手架（技术栈/模块/接口/链路/流程/规范/标准）
- 轨道2：深度分析需求文档和原型（明确后端职责/前端边界/前后端协作点）
- 两者缺一不可，只有脚手架无法知道做什么，只有需求无法做得合规

### 二、元认知框架（9大类对话质量控制）

文件结构：上半部分=用户应提供什么信息；下半部分=AI必须怎么做

1. **问题定义与目标对齐** — 先搞清背景/意图/目标/任务性质/边界/紧急度/可分解性；信息不足必须先问不得假设
2. **协作方式与交互控制** — 按指定角色参与；未授权不替代决策；信息不足优先澄清关键缺口（最多3个高影响问题）；不擅自切换协作模式
3. **用户认知与表达控制** — 适配用户水平；默认结构化输出；结论→理由→步骤顺序；控制信息密度
4. **事实输入与证据基础** — 只基于用户提供的事实推理；区分确定结论与推测；标注假设前提；证据冲突必须显式指出
5. **方案设计与实施落地** — 给可执行步骤而非方向建议；每步说明输入/动作/预期输出；先评估必要性与可行性；锁定对象边界防止漂移
6. **约束、环境与非功能控制** — 严格遵守硬约束；环境不明必须暂停；默认优先安全与稳定；优先保守路径
7. **风险控制与稳定性** — 未明确风险等级默认高风险处理；禁止越权；信息严重不足必须暂停；结论标注时效窗口
8. **验证闭环与成功判定** — 给出可观察的通过/不通过信号；支持多轮迭代；每轮说明改了什么/为什么改
9. **全局策略控制** — 围绕用户声明的优化目标；防止策略漂移；历史冲突必须显式指出；记忆写入需用户确认；不隐性取舍

### 三、中间件层深度解析（核心骨架）

**文件位置：** `common/pkg/middleware/` 共5个文件

#### 3.1 SetHeaderInfo() — 请求信息提取（info.go）
从HTTP请求提取信息注入context（通过metainfo跨RPC自动传播）：
- `X-Client-IP` ← c.ClientIP()
- `X-User-Language` ← Header
- `X-User-Device-ID` ← Header
- `X-User-Device-Type` ← Header
- `X-User-Token` ← Header优先，Cookie("session_token")备选

#### 3.2 BlackCheck() — C端IP黑名单（gate-auth.go）
- 调 `rpc.IpClient().OnBlackList(ctx)` → IP在黑名单=拒绝
- RPC失败时**默认放行**（宽松策略）
- 错误码：`errs.IpInBlackTable`

#### 3.3 WhiteCheck() — B端IP白名单（gate-auth.go）
- 调 `rpc.IpClient().OnWhiteList(ctx)` → IP不在白名单=拒绝
- RPC失败时**默认禁行**（严格策略）
- 错误码：`errs.IpNotInWhiteTable` / `errs.IpAnalysisErr`

#### 3.4 FontAuthMiddleware() — C端Token鉴权（gate-auth.go）
1. `tracer.GetUserToken(ctx)` 取token，空=拒绝
2. `rds.Get(redis_key.RedisTokenKey + token)` → Redis key格式 `user:token:{token}`
3. JSON反序列化为 `session.CUserTokenPayload{ID, UID, LastLoginIp, LastLoginTime, Device, DeviceType, NickName}`
4. 将UserID和NickName注入context（metainfo）
5. ID=0则拒绝

#### 3.5 BackAuthMiddleware() — B端Token+URL权限（gate-auth.go）
1. 同上Token校验，但Redis key直接用token（**无前缀**，与C端不同）
2. 反序列化为 `session.BUserTokenPayload{ID, UserName}`（注意UserName的json tag是"uid"）
3. **额外步骤：** 调 `rpc.BUserClient().CheckUrlAuth(ctx, c.FullPath())` 校验URL级RBAC权限
4. 权限不通过=拒绝，错误码 `errs.CheckUrlAuthFailed`

#### 3.6 RedirectS3() — S3文件重定向（s3.go）
- 路由：`/s3/*path`
- `s3/public/` 前缀 → CDN域名直接拼接URL → 302重定向
- 其他路径 → 从Redis Hash `s3:hs:config:` 读过期配置 → AWS PresignGetObject生成签名URL → no-cache头 → 302重定向

#### 3.7 InitRedis() — Redis初始化（redis.go）
- sync.Once单例
- 从ETCD读 `/slg/conf/redis/addr` 和 `/slg/conf/redis/password`
- 创建 UniversalClient（支持集群，逗号分隔多地址）

#### 3.8 HertzAccLogStr() — 访问日志（hertz_acclog.go）
- 格式：`<trace_id=xxx> ${status} - ${latency} ${method} ${path}`
- trace_id从响应头 `X-Trace-Id` 获取

### 四、链路追踪系统（tracer.go）

**常量定义（HTTP Header名）：**
```
TraceIDHeader="X-Trace-Id", ClientIPHeader="X-Client-IP", UserTokenHeader="X-User-Token"
UserIdHeader="X-User-ID", UserNameHeader="X-User-Name", UserLangHeader="X-User-Language"
UserDeviceIDHeader="X-User-Device-ID", UserDeviceTypeHeader="X-User-Device-Type"
```

**IGlTracer.Start()：** 生成UUID作为trace_id → 存入context → 设置CORS头 → 设置到响应头
**Get函数族：** GetTraceID/GetUserId/GetUserToken/GetClientIp/GetUserLang/GetDeviceID/GetDeviceType/GetUserName
**传播机制：** 通过 `bytedance/gopkg/cloud/metainfo.WithPersistentValue()` 存入context，Kitex TTHeader自动传播到下游RPC服务

### 五、统一响应与错误码（ret/ + errs/）

**R结构体（ret/project_ret.go）：** `{TraceId, Code(errs.ErrorType), Msg, Data}`
**错误码类型：** `type ErrorType int32`，实现error接口
**标准错误码起始：** 6000000（SystemBaseError），含OperateErr/ParamErr/IpInBlackTable/TokenNotExist/CheckUrlAuthFailed等
**业务错误码规范：** 6-8位，前2位=服务编号+模块编号，后几位=序列号
**核心函数：**
- `ret.BizErr(ctx, code, msg...)` → 包装为kerrors.BizStatusError
- `ret.HttpOne(ctx, data, i18nClient, err)` → 统一HTTP响应，自动翻译错误信息
- `ret.HttpErr(ctx, err, i18nClient)` → HTTP错误响应
- 所有响应自动附带TraceId

### 六、ETCD三层架构（etcd/）

**初始化（etcd_init.go）：** 环境变量`ETCD_ADDR`读集群地址 → sync.Once创建client → Ping验证
**直接操作（etcd_direct.go）：** DirectGet/DirectPut/DirectDelete/DirectPutWithLease（带TTL租约）
**缓存+热更新（etcd_cache.go）：**
- `LoadAndWatch(prefix)` → 全量拉取+启动watchLoop协程
- 内部用 `atomic.Value` 存 `map[string]string`，Copy-On-Write更新
- `Get(ctx, key)` 从内存缓存读（极速，无网络I/O）
- `Hook(key, func)` 注册配置变更回调，支持热更新

### 七、数据库层（db/）

**MySQL初始化（db/mysql/mysql.go）：** GORM + PrepareStmt + SkipDefaultTransaction（性能优化）
**用户审计插件（user_plugin.go）：** GORM回调，创建/更新前自动填充 CreateBy/UpdateBy/CreateAt/UpdateAt（毫秒时间戳，从tracer.GetUserId获取）
**GORM日志（grom_log.go）：** 慢查询阈值200ms，输出SQL耗时/行数/语句
**分页DTO（db/dto/page_result.go）：** `PageResult[T]{Data, Total, PageTotal, PageNo, PageSize}`
**GORM Gen代码生成（db/cmd/generate.go + repo_template.go）：**
- GenCodeWithAll → 连DB扫全表 → 生成model+query+repo
- Repo模板含29个CRUD方法（Create/BatchCreate/Update/BatchUpdate/DeleteById/GetOneById/GetPageByConds等）
- 支持加密字段映射（EncryptedString类型，GORM自动加解密）
- 分页默认按ID倒序，pageSize最大500

**init.go — Server参数：** ReusePort + MuxTransport + ETCD Registry + TTHeader MetaHandler + 访问日志中间件
**init.go — Client参数：** ETCD Resolver + RPCTimeout(1200s) + MuxConnection(1) + TTHeader + TTHeader Transport
**rpc_client.go — 20个单例客户端（sync.Once）：**
编号001-021，CUser已注释禁用。每个客户端模式相同：`xxxOnce.Do → xxxservice.NewClient(namesp.EtcdXxxService, InitRpcClientParams()...) → 存指针`
**handler.go — 3个泛型Handler：**
- `Handler[REQ,RESP]` — 反射处理指针/非指针REQ → BindAndValidate → rpcMethod(ctx,req) → ret.HttpOne(ctx,resp,I18nClient(),err)
- `HandlerNoParam[RESP]` — 直接rpcMethod(ctx) → ret.HttpOne
- `HandlerWithCSV[REQ,RESP]` — 同Handler但检查resp是否实现CSVResponse接口 → 设置Content-Disposition+octet-stream → c.Write(data)
- CSVResponse接口：`GetFilename() string` + `GetData() []byte`
