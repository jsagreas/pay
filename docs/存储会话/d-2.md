### 二、严格约束红线（不可违反）

#### 第一约束：默认只读
- 所有操作默认只读，未经用户明确口头允许和指令，**禁止任何增删改操作**
- 分析阶段严禁产生增删改；分析结束不得自行进入编码阶段

#### 行为约束
- **禁止猜测臆想**：必须基于分析工程或主动问询获得明确信息
- **禁止片面分析**：局部信息得出的推理是级联错误，必须全面完整深入
- **禁止想当然**：凡事要有依据、凭证、理由
- **禁止无中生有**：不能把没问题的说成有问题

- **能分析获取的信息必须自主分析**：禁止无脑咨询用户
- **无法自行获取的前置信息必须主动问询**：不得想当然操作

#### 思考约束
- **宏观+微观**：清楚架构全局 + 每个函数变量细节
- **闭环分析**：模块间关系、依赖、职责全部搞清楚
- **穷极追踪**：追代码逻辑要彻底，禁止浅尝辄止蜻蜓点水
- **必须清楚**：架构、调用链路、业务流程、设计思路、标准规范、接口细节、技术栈

#### 执行约束
- 先深度分析项目脚手架（技术栈/模块/接口/调用链路/流程/开发风格/规范）
- 再深度分析需求文档和原型（区分前后端职责，做到宏观+细节）
- 两者缺一不可：只看项目不知道做什么，只看需求做不出符合规范的实现

#### 互动约束
- **分步骤分阶段**：拆解→确认→执行→验证→迭代
- **小步慢跑**：先给思路步骤，得到确认才编码
- **编码需明确授权**：未经同意禁止编码，编码后需验收才能继续

### 三、代码生成流程（关键，已实证验证）

#### 3.1 IDL → Kitex 代码生成

- **IDL 位置**：`common/idl/ser-xxx/`，每个服务一个目录
- **IDL 组织方式**：`service.thrift` 为主入口文件，通过 `include` 引入业务 thrift 文件（如 `banner.thrift`, `home_category.thrift`），还可引入公共定义（如 `../common/enum.thrift`）
- **脚本**：`common/script/gen_kitex.sh`
- **脚本行为**：
  - 扫描 `common/idl/` 下每个子目录，匹配 `*service.thrift` 文件
  - 工作目录切到 `common/pkg/`
  - 执行 `kitex -module common/pkg -I <idl_dir> -service <service_name> -gen-path ./kitex_gen/ <thrift_file>`
  - **生成产物统一落在 `common/pkg/kitex_gen/ser_xxx/`**（注意：不是在各 ser-xxx 内部）
  - 如果 `ser-xxx/` 已存在 handler.go + main.go，脚本会先拷到 pkg 下处理，生成后拷回（保留已有 handler 和 main）
  - 自动清理临时产物（bootstrap.sh, kitex_info.yaml, build.sh, scripts/）
- **实证**：`common/pkg/kitex_gen/` 下有 20 个服务子目录 + 1 个 common 目录

#### 3.2 数据库 → GORM Gen 代码生成

- **入口文件**：`ser-xxx/internal/gen/gorm_gen.go`（每个涉及 DB 的模块各一个）
- **核心调用**：`mysql_gen.GenCodeWithAll(dbname, "ser-xxx", "")`
  - `dbname` 从 ETCD 获取（如 `etcd.DirectGet(ctx, namesp.EtcdAppDb)` → `/slg/serv/app/db`）
  - 连接数据库，遍历所有表，自动生成代码
- **生成产物位置**：`ser-xxx/internal/gorm_gen/`
  - `model/` — 每张表的 Go 结构体（如 `banner.gen.go`）
  - `query/` — GORM Gen 类型安全查询接口（如 `banner.gen.go`, `gen.go`）
  - `repo/<表名小写>r/` — 每张表一个 repo 包，模板自动生成 18 个标准 CRUD 方法（Create, BatchCreate, Update, BatchUpdate, BatchSave, UpdateByIds, UpdateByConds, UpdateFields, UpdateFieldsByConds, DeleteById, DeleteByConds, GetOneById, GetOneByConds, GetListByIds, GetListByConds, GetPageByConds, Count, Exists 等）
- **实证**：`ser-app/internal/gorm_gen/repo/` 下有 `bannerr/`, `homecategoryr/`，内含模板生成的标准 CRUD 代码

### 四、模块内部三层架构

```
ser-xxx/
├── handler.go              # Kitex RPC 入口（实现 IDL 定义的 Service 接口）
├── main.go                 # 服务启动（ETCD→日志→DB→Redis→Kitex Server→注册）
├── internal/
│   ├── gen/gorm_gen.go     # GORM Gen 代码生成入口
│   ├── gorm_gen/           # 自动生成（model/ + query/ + repo/）
│   └── service/            # 业务逻辑层
└── go.mod
```

- **Handler**：接收 RPC 请求 → 提取 metainfo（TraceID、操作人等）→ 调用 Service → 组装响应
- **Service**：核心业务逻辑 → 调用 Repo 做持久化 → 按需跨服务 RPC 调用
- **Repository**：自动生成的标准 CRUD + 手写的业务查询扩展

### 五、网关层模式

- **核心模式**：`rpc.Handler[REQ, RESP](ctx, rpc.XxxClient().Method)` — 泛型一行代理，Handler 不含业务逻辑
- **gate-back（B 端）**：
  - 路由前缀：`/admin/open`（免认证）、`/admin/api`（需认证）
  - 中间件链：`SetHeaderInfo → WhiteCheck → BackAuthMiddleware`（Token + URL 权限 CheckUrlAuth）
- **gate-font（C 端）**：
  - 路由前缀：`/open`（免认证）、`/api`（需认证）
  - 中间件链：`SetHeaderInfo → BlackCheck → FontAuthMiddleware`
  - 差异：BlackCheck（黑名单拦截）vs WhiteCheck（白名单放行）；无 URL 级权限校验

### 六、关键约定规范

| 约定 | 规范 | 依据 |
|---|---|---|
| 错误码 | 6 位数字，按服务分段（6000=common, 6002=buser, 6003=bcom, 6004=i18n, 6005=ip, 6007=s3, 6010=item, 6011=app） | 各模块 errs 定义 |

### 七、新模块 0-1 构建流程（综合验证）

> 以下流程基于工程代码实证 + 用户反馈综合梳理

1. **编写 IDL 文件**：在 `common/idl/ser-xxx/` 下创建 thrift 文件（主入口 `service.thrift` + 业务模块 thrift），定义 Service 接口、请求/响应结构体
2. **执行 gen_kitex.sh**：运行 `common/script/gen_kitex.sh`，自动扫描 IDL 生成 Kitex 代码到 `common/pkg/kitex_gen/ser_xxx/`
3. **编写 GO 文件**：在 `ser-xxx/` 下按三层架构组织代码（handler.go, main.go, internal/service/）
4. **如涉及数据库**：编写 `ser-xxx/internal/gen/gorm_gen.go`，执行后自动生成 model/query/repo 到 `ser-xxx/internal/gorm_gen/`
5. **模块间调用**：在 `common/rpc/rpc_client.go` 注册新服务 Client 单例；网关侧（gate-back/gate-font）注册路由和 Handler
6. **相关中间件引入使用**：按 B/C 端需求接入对应认证、鉴权、日志等中间件
7. **业务闭环验证**：完整链路联调（HTTP → Gateway → RPC → Handler → Service → Repo → DB）
