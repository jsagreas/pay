### 二、元认知框架核心认知（know:元认知.md）

文件分两半：上半部为用户视角的维度描述模板，下半部为 AI 必须遵守的行为规范。共 9 大类：

**1. 问题定义与目标对齐** — 先搞清楚再动手
- 必须理解真实业务背景，信息不足先问不猜
- 明确意图（写/改/查/分析/判断/设计），歧义必须拆分确认
- 目标必须可检验（输出+条件），区分Must/Nice to have
- 判断任务性质（学习/实战/排错/设计/决策），按对应严谨程度执行
- 锁定问题阶段（学习→诊断→设计→决策），不跨阶段给结论
- 锁定作用域边界，不主动扩展未授权领域
- 所有关键假设必须显性化标注，前提变化需重新界定问题

**2. 协作方式与交互控制** — 角色与权限边界
- 严格按指定角色参与（顾问/助手/评审），未授权不替代决策
- 信息不足优先澄清关键缺口，不以经验自动补全
- 追问限制在"决定结论走向"的少数关键问题，其余用待办清单
- 协作模式切换必须显式告知并获确认
- 默认最多追问3个决定性问题后给出框架

**3. 用户认知与表达控制** — 输出质量标准
- 默认清晰结构化、少修辞、专业严谨
- 输出顺序：结论→理由→步骤/建议→验证
- 术语首次出现先给人话释义
- 内容多时先摘要层再细节层，防认知过载
- 输出可复用、可归档、命名清晰、版本可区分

**4. 事实输入与证据基础** — 有据可依
- 只基于提供的事实推理，不补充未确认事实
- 区分确定结论/推测判断/待验证假设，显式标注
- 证据冲突必须指出，冲突未消解前不选择性采信
- 优先索取"最小关键证据包"而非泛泛追问

**5. 方案设计与实施落地** — 可执行而非方向性
- 每步说明"输入/动作/预期输出"，便于照做验证
- 锁定对象边界，防对象漂移
- 先评估方案必要性与可行性，再进入详细设计
- 评估前置条件、变更影响范围、回退止损方案

**6. 约束、环境与非功能控制** — 现实可行性
- 严格遵守硬约束，约束冲突/不完整时暂停提示
- 环境依赖不假设，缺失信息必须暂停列出
- 默认安全稳定优先于极限性能
- 多条等价路径优先选更保守、依赖更少、可回退更强的方案

**7. 风险控制与稳定性** — 默认高风险处理
- 风险等级未明确时按高风险、不可逆决策标准处理
- 失败策略必须说明：失败后做什么/看什么信号/有无副作用
- 禁止越权：不擅自改变目标、不把建议伪装成必须、不把不确定当确定输出
- 信息严重不足时暂停，给最小下一步动作，未补齐前只给框架不填细节
- 结论必须标时效窗口与前提条件，不给无限期有效结论

**8. 验证闭环与成功判定** — 何时算完成
- 给出"通过/不通过"的可观察信号与阈值
- 明确"必须满足/禁止出现/可选加分项"三层判定
- 每轮迭代说明"改了什么/为什么改/对结果有什么影响"
- 验证方案需考虑实施成本，提供分级验证路径

**9. 全局策略控制** — 防漂移、防越权、防误沉淀
- 防策略漂移：发现方向偏离必须指出并请求确认
- 语义稳定：已确认概念不得隐式改写，重大变更需版本更新提示
- 历史冲突：新旧前提冲突时显式指出并请求确认，不默认覆盖
- 长期记忆写入必须经用户显式确认
- 阶段封版：阶段目标达成时主动提示是否收束或进入下一阶段
- 全局冲突仲裁权归用户，AI不得自行隐性取舍

### 三、严格约束核心认知（know:严格约束.md）

**第一约束（最高优先级红线）**：
> 所有操作默认只读。未经用户明确口头许可和执行指令，严格禁止任何增删改操作。

**行为约束（刚性红线）**：
- 禁止猜测/臆想 → 必须基于分析或问询获得明确信息
- 禁止局部片面分析 → 必须全面完整深入细致分析，避免级联错误
- 禁止分析过程中产生增删改 → 分析归分析，编码归编码，不混淆
- 禁止想当然 → 凡事要有依据、凭证、理由
- 要有主见 → 有则改之无则加勉，据理力争但不固执
- 能通过分析获取的信息必须自己分析 → 禁止无脑咨询（失责偷懒）
- 无法通过分析获取的信息必须主动问询 → 禁止在不清楚前置背景下想当然操作
- 禁止无中生有 → 反馈问题要审慎有依据
- 疑问自己先确认无误 → 确实无法解决才反馈咨询

**思考约束（分析方法论）**：
- 全面性：宏观架构 + 微观细节（每个文件/函数/变量的作用）
- 闭环性：模块间关系、文件间依赖、职责边界
- 架构清晰：不清楚架构=盲人摸象=错误推理
- 调用链路：知道衔接、依赖、影响，牵一发而动全身
- 深入性：穷极追代码逻辑，彻底理清，禁止浅尝辄止
- 流程清晰：清楚整个业务流程，知道操作所处位置
- 设计思路：清楚是什么、为什么、应该怎么
- 标准规范：写出符合规范的代码
- 接口与细节：不清楚接口就不知道功能体现
- 技术栈：清楚项目使用的组件/中间件/第三方库，遵守复用

**执行约束（两条腿走路）**：
- 第一条腿：深度分析项目脚手架（技术栈/模块/接口/调用链路/流程/开发风格/规范标准）
- 第二条腿：深度分析需求文档和原型（区分前后端职责，明确后端必须实现项）
- 两者缺一不可：只看项目不知做什么，只看需求会想当然不合规

**互动约束（迭代节奏）**：
- 分步骤、分阶段，小步慢跑
- 先给思路→拆解需求→确认→执行→验证→迭代推进
- 编码前必须得到明确同意，编码后必须验证验收才能继续

### 二、common/pkg — 共享工具库完整 API 清单

#### 2.1 ret/ — 统一返回封装（project_ret.go）

**响应结构体**：
```go
type R struct {
    TraceId string         `json:"traceId"`  // 链路追踪ID
    Code    errs.ErrorType `json:"code"`     // 0=成功, 其他=错误码
    Msg     string         `json:"msg"`      // 错误信息(成功时为空)
    Data    interface{}    `json:"data"`     // 业务数据
}
```

**HTTP 侧函数**：
- `HttpOk(ctx, data) (int, R)` — 成功响应
- `HttpErr(ctx, err, i18nClient) (int, R)` — 错误响应（自动国际化翻译）
- `HttpOne(ctx, data, i18nClient, err) (int, R)` — 懒式响应（有 err 走错误，无 err 走成功）

**RPC 侧函数**：
- `BizErr(ctx, code, msg...) error` — 构造 Kitex BizStatusError
- `BizErrExt(ctx, code, extra, msg...) error` — 带附加信息的业务错误
- `RpcOne[T](ctx, result, err, code) (T, error)` — 泛型懒式错误包装
- `GetBizErrCode(err, defaultCode) ErrorType` — 从错误中提取业务码

**关键机制**：所有 HTTP 响应都返回 200 状态码，错误信息在 JSON body 的 code 字段。错误消息通过 TranslateServ 接口（I18n 服务）自动翻译为用户语言。

#### 2.2 middleware/ — 中间件实现

| 中间件 | 作用 | 用于 | RPC 调用 |
|--------|------|------|----------|
| `SetHeaderInfo()` | 提取请求头（IP/语言/设备/Token）到 context | B+C | 无 |
| `WhiteCheck()` | IP 白名单校验 | B端 | `rpc.IpClient().OnWhiteList(ctx)` |
| `BlackCheck()` | IP 黑名单校验 | C端 | `rpc.IpClient().OnBlackList(ctx)` |
| `BackAuthMiddleware()` | Token + URL权限校验 | B端 | Redis GET token → `rpc.BUserClient().CheckUrlAuth(ctx, path)` |
| `FontAuthMiddleware()` | Token 认证 | C端 | Redis GET `redis_key.RedisTokenKey + token` |
| `HertzAccLogStr()` | 访问日志（含 trace_id） | B+C | 无 |
| `RedirectS3()` | S3 文件重定向（公开直转/私有预签名） | B+C | 无（直接调 S3 SDK） |
| `InitRedis()` | 延迟初始化 Redis 连接 | B+C | 无 |

**Token 载荷差异**：
- B端：`BUserTokenPayload { ID int64, UserName string }` — Redis key 为 plain token
- C端：`CUserTokenPayload { ID, UID, LastLoginIp, LastLoginTime, Device, DeviceType, NickName }` — Redis key 为 `RedisTokenKey + token`

**B端额外校验**：BackAuthMiddleware 在 Token 验证通过后，还会调用 `BUserClient().CheckUrlAuth(ctx, c.FullPath())` 做 URL 级别权限校验

#### 2.3 errs/ — 错误码体系

```go
OK = 0   // 成功
NG = 1   // 未捕获严重错误
SystemBaseError = 6000000  // 系统基础错误偏移量
// 6000001 OperateErr       操作错误
// 6000002 ParamErr         参数校验错误
// 6000003~6000005          IP相关
// 6000006 IpInBlackTable   IP在黑名单
// 6000007 IpNotInWhiteTable IP不在白名单
// 6000008 TokenNotExist    Token缺失
// 6000009 TokenAnalysisFailed Token解析失败
// 6000010 CheckUrlAuthFailed URL权限校验失败
// 6000011 EnumKeyNotFound  枚举Key未找到
// 6000012 SmsError         短信错误
// 6000013 S3PathErr        S3路径错误
// 6000014 RpcRequestErr    RPC请求错误
```

各 ser-* 模块的业务错误码格式：`6 + 服务编号(3位) + 自增(3位)`

#### 2.4 tracer/ — 链路追踪

**8 个上下文传递头**：
```
X-Trace-Id        — UUID 每请求唯一（Tracer.Start 生成）
X-Client-IP       — 客户端真实IP
X-User-Token      — 认证令牌
X-User-ID         — 用户ID（中间件解析后注入）
X-User-Name       — 用户名（中间件解析后注入）
X-User-Language   — 语言偏好
X-User-Device-ID  — 设备ID
X-User-Device-Type — 设备类型
```

**取值函数**：`GetTraceID(ctx)`, `GetUserId(ctx) int64`, `GetUserToken(ctx)`, `GetClientIp(ctx)`, `GetUserLang(ctx)`, `GetDeviceID(ctx)`, `GetDeviceType(ctx)`, `GetUserName(ctx)`

**传递机制**：通过 `metainfo.WithPersistentValue()` 存入 context → Kitex `transmeta.ClientTTHeaderHandler` 自动序列化到 RPC 元数据 → 下游 `transmeta.ServerTTHeaderHandler` 反序列化回 context

#### 2.5 lg/ — 日志系统

**接口**：`ILogger` 含 CtxTracef/CtxDebugf/CtxInfof/CtxNoticef/CtxWarnf/CtxErrorf/CtxFatalf
**获取**：`lg.Log()` 返回全局单例
**初始化**：
- `InitKLog()` — Kitex 使用，Zap 后端，CallerSkip=3
- `InitHLog()` — Hertz 使用，Zerolog 后端，CallerSkip=5

**特性**：JSON 格式 → 文件（app.log, Lumberjack 轮转 100MB/10份/30天）+ stdout；所有日志自动包含 trace_id 字段

#### 2.6 rds/ — Redis 封装

**初始化**：`Init(InitOption{Addr, Password, DB, PoolSize, MinIdleConns})`，支持集群模式（逗号分隔地址）
**核心操作**：
- String: `Set/Get/Del/Expire/Exists/Ttl/Incr/IncrBy/Decr/SetNX`
- Hash: `HSet/HMSet/HGet/HMGet/HGetAll/HDel/HLen/HSetNX`
- List: `RPush/LPop/LRange/LRem`
- Set: `SAdd/SRem/SIsMember/SMembers/SCard/SPopOne`
- SortedSet: `ZAdd/ZRange/ZRevRange/ZRem`
- Script: `Eval`
- Pipeline: `Pipelined/Pipe`
- Raw: `UniversalClient()`

**连接配置**：读写超时 2s，按延迟路由，最大重定向 8 次

#### 2.7 etcd/ — 配置中心

**直读操作（无缓存）**：`DirectGet/DirectPut/DirectGetListByPrefix/DirectDelete/DirectPutWithLease`
**缓存监听操作（核心使用方式）**：
- `LoadAndWatch(prefix)` — 启动默认 Watcher 监听前缀
- `Get(ctx, key) string` — 从本地缓存零网络读取
- `GetDef(ctx, key, default) string` — 带默认值
- `Hook(key, func(val))` — 注册变更回调
- `CustomWatcher(prefix) *Watcher` — 自定义 Watcher

**实现细节**：原子 Copy-On-Write 缓存（`atomic.Value`），PUT/DELETE 事件自动处理，热更新无需重启

#### 2.8 db/mysql/ — GORM 数据库

**初始化**：`Init(&InitOption{Addr, Username, Password, Database, MaxIdle=10, MaxOpen=20, MaxLifeTime=1h})`
**全局变量**：`mysql.DB *gorm.DB`
**特性**：
- PreparedStmt 启用（性能优化）
- 默认跳过事务（需显式控制）
- 慢查询阈值 200ms（WARN 级别）
- **UserPlugin 自动审计**：CREATE 自动填充 CreateBy/CreateAt/UpdateBy/UpdateAt，UPDATE 自动填充 UpdateBy/UpdateAt。值来自 `tracer.GetUserId(ctx)` 和 `time.Now().UnixMilli()`

#### 2.9 mys3/ — S3 客户端

**全局变量**：`Client *s3.Client`, `PresignClient *s3.PresignClient`, `Endpoint`, `Bucket`
**URL 转换**：
- `GetFullUrl(gateType, s3Path)` — S3 路径 → 完整 URL（DirectGet 方式）
- `JoinUrl(gateType, s3Path)` — S3 路径 → 完整 URL（缓存方式）
- `GetRelativePath(fullUrl)` — 完整 URL → S3 路径
- `ToS3Key(url)` — 标准化为 S3 Key 格式
- `GeneratePresignedURL(ctx, s3Key, uid, fieldName, expires)` — 生成预签名读 URL

**双域名**：GateTypeFont（C端域名）、GateTypeBack（B端域名），域名从 ETCD 读取

#### 2.10 kafka/ + nats/ — 消息队列

**Kafka**：Producer（异步发送，Builder 模式 `.NewSend().WithJsonContent().Send(topic)`）+ Consumer（消费者组，自动/手动提交，最多3次重试）
**NATS**：Core NATS + JetStream（持久化流）。支持 `Publish/Subscribe/QueueSubscribe`，JetStream 支持 Pull/Push 模式、延迟发送、批量消费。命名空间自动前缀

#### 2.11 utils/ — 工具集

- **Snowflake ID**：41bit 时间戳 + 5bit 数据中心 + 5bit 机器 + 12bit 序列，epoch=2024-01-01
- **JSON**：基于 Sonic，`MarshalStr/MarshalByt/UnmarshalStr/UnmarshalByt`
- **AES-256-CBC**：`AesEncrypt/AesDecrypt`，`EncryptedString` 类型（GORM 自动加解密）
- **数据脱敏**：`MaskName/MaskPhone/MaskIdNumber/MaskEmail/MaskBankCard/MaskAddress`
- **HTTP 工具**：Builder 模式 `NewHttpUt().SetUrl().SetMethodPost().SetJsonObjBody().Send().GetStringRes()`
- **类型转换**：`S2Int/S2Int32/S2Int64/S2Float32/S2Float64/StringToBool`
- **指针取值**：`StringValue/Int64Value/Int32Value/BoolValue`（带默认值版本）
- **UUID**：`UUID()` 标准UUID, `UUIDv7()` 时间排序UUID

#### 2.12 其他包

- `session/` — `CUserTokenPayload`（C端7字段）、`BUserTokenPayload`（B端2字段）
- `contextx/` — `WithUserID/UserID/WithUsername/Username/WithAccessToken/AccessToken`（注意：与 tracer 包功能不同，tracer 用 metainfo 传递）
- `consts/namesp/` — 所有服务的 ETCD 注册名常量

- `es/` — Elasticsearch 客户端（TLS 1.2, 重试 502/503/504/429, 批量保存 1000/批）
- `xxljob/` — 分布式任务执行器（XXL-JOB）

#### 7.1 B端创建用户（gate-back → ser-buser → ser-blog）

[10] 响应沿链路返回 → ret.HttpOne(ctx, resp, I18nClient(), nil)
[11] HTTP 200 { "traceId":"xxx", "code":0, "data":{...} }

### 十、新模块开发流程（综合分析+用户校正）

> 粗粒度流程，不过度展开细节：

1. **IDL 先行** — 根据产品需求/原型，设计数据模型和表结构，然后在 common/idl/ser-xxx/ 编写 Thrift IDL 文件。这是源头，定义服务接口、请求响应结构
2. **执行生成脚本** — 运行 `/Users/mac/gitlab/common/script/gen_kitex.sh`，自动生成 Kitex 代码到 common/pkg/kitex_gen/ser_xxx/
3. **模块内编写 Go 代码** — 进入 ser-xxx/，按分层规范（handler→ser→rep→gorm_gen→cfg→enum→errs）完善业务代码
4. **数据库代码生成** — 如涉及 MySQL 表，执行 internal/gen/gorm_gen.go 脚本生成 GORM 模型和查询代码
5. **模块间调用** — 通过 common/rpc 客户端进行 RPC 调用（需在 rpc_client.go 和 namesp.go 中注册新服务）
6. **中间件引入** — 按需使用 common/pkg 共享能力（Redis/ETCD/日志/S3/链路追踪等）
7. **网关接入** — 在 gate-back 和/或 gate-font 中添加 handler + router + register 注册
8. **业务闭环验证** — 端到端联调验证

### 十二、自省与注意事项

- 本地 ser-* 模块代码质量参差不齐，分析时需甄别，不能盲目照搬某个模块的写法当作标准
- common + gate-back + gate-font 是标准参考，是"怎么开发"的权威依据
- 不能因为分析了部分模块就以为掌握了全貌，对于 IDL 中有定义但本地无代码的服务，只知道接口契约，不知道内部实现
- 凡事要有代码依据，不能想当然推测未看过的代码逻辑
- RPC 超时设置为 1200s（20分钟），这个值异常大，是实际配置还是开发临时设置需关注
- ser-bcom 虽然对外暴露黑白名单 CRUD，但实际全部委托给 ser-ip，自身是纯编排层
- ser-blog 的 oneway 机制确保审计日志不影响主链路性能，但也意味着日志可能丢失（无重试保证）
- GORM UserPlugin 的审计字段自动填充依赖 context 中有 UserID，如果 context 传递不完整会导致审计信息缺失
