### 2.2 ETCD配置系统（核心）

**两种读取模式：**

| 模式 | 函数 | 场景 | 性能 |
|------|------|------|------|
| Watch+Cache | `etcd.Get(ctx, key)` | 高频访问 | 内存读取，纳秒级 |
| Direct | `etcd.DirectGet(ctx, key)` | 单次初始化 | 网络请求，毫秒级 |

**Watch机制实现细节：**
- `LoadAndWatch(prefix)` 启动后台goroutine
- 全量拉取所有Key → 存入 `atomic.Value`（Copy-On-Write）
- 监听PUT/DELETE事件 → 更新缓存 → 触发Hook回调
- 并发读取无锁，线程安全

**Hook机制：** `etcd.Hook(key, func(val string))` 配置变化时自动触发

### 2.7 中间件详细逻辑（middleware包）

**SetHeaderInfo()：**
```
提取请求头 → ClientIP/UserLang/DeviceID/DeviceType/Token
Token优先从Header(X-User-Token)，降级从Cookie(session_token)
全部通过 metainfo.WithPersistentValue() 注入ctx
```

**BlackCheck()（C端）：**
```
调用 rpc.IpClient().OnBlackList(ctx)
  → ser-ip 解析客户端IP → 查ISO国家代码 → 匹配c_black_list表
返回true → c.Abort + 返回IpInBlackTable错误
返回false → c.Next继续
调用失败 → 默认放行（容错设计）
```

**WhiteCheck()（B端）：**
```
调用 rpc.IpClient().OnWhiteList(ctx)
  → ser-ip 检查客户端IP → 匹配sys_white_list表
返回true → c.Next继续
返回false → c.Abort + 返回IpNotInWhiteTable错误
调用失败 → 默认禁行（安全优先）
```

**FontAuthMiddleware()（C端Token认证）：**
```
1. 从ctx获取Token（SetHeaderInfo已注入）
2. Token为空 → 返回TokenNotExist
3. Redis查询: rds.Get("user:token:" + token)
4. 空结果 → 返回TokenAnalysisFailed
5. JSON反序列化为 CUserTokenPayload{ID, UID, NickName, LastLoginIp, ...}
6. ID=0 → 返回TokenAnalysisFailed
7. 注入UserID和UserName到ctx → c.Next
```

**BackAuthMiddleware()（B端Token+URL权限认证）：**
```
1. 从ctx获取Token
2. Token为空 → 返回TokenNotExist
3. Redis查询: rds.Get(token)  // 注意：B端key就是token本身，无前缀
4. JSON反序列化为 BUserTokenPayload{ID, UserName}
5. 注入UserID和UserName到ctx
6. 调用 rpc.BUserClient().CheckUrlAuth(ctx, c.FullPath())
7. 无权限 → 返回CheckUrlAuthFailed
8. 有权限 → c.Next
```

**RedirectS3()（S3文件代理）：**
```
1. 获取路径 c.Param("path")
2. 拼接 objKey = "s3/" + path
3. 公开文件(s3/public/*) → CDN域名直接302重定向
4. 私有文件 → 生成预签名URL
   a. 从Redis Hash获取业务类型配置(过期时间)
   b. mys3.PresignClient.PresignGetObject()
   c. 设置Cache-Control: no-store防缓存
   d. 302重定向到预签名URL
```

- WithReusePort(true) — 端口复用
- WithMuxTransport() — 多路复用
- WithRegistry(etcdRegistry) — ETCD服务注册
- WithMetaHandler(ServerTTHeaderHandler) — TTHeader元数据解析
- WithMiddleware — 日志中间件（打印TraceID+包名+服务名+方法名+耗时+调用方信息）

**客户端 InitRpcClientParams()：**
- WithResolver(etcdResolver) — ETCD服务发现
- WithRPCTimeout(1200s) — 超时20分钟（注意：非常长）
- WithMuxConnection(1) — 单连接多路复用
- WithMetaHandler(ClientTTHeaderHandler) — TTHeader元数据注入
- WithTransportProtocol(TTHeader) — 二进制协议

### 3.4 Handler适配器三种模式

**Handler[REQ, RESP]（标准模式，最常用）：**
```
1. 反射初始化REQ（支持指针/值类型）
2. c.BindAndValidate() 绑定+校验HTTP参数
3. 调用rpcMethod(ctx, reqData)
4. c.JSON(ret.HttpOne(ctx, resp, I18nClient(), err))
```

**HandlerNoParam[RESP]（无参数模式）：**
```
直接调用rpcMethod(ctx) → c.JSON(ret.HttpOne(...))
```

**HandlerWithCSV[REQ, RESP]（CSV导出模式）：**
```
1. 同Handler绑定参数+调用RPC
2. 检查resp是否实现CSVResponse接口
3. 是 → 设置Content-Disposition+Content-Type头 → c.Write(data)
4. 否 → 返回JSON
```

## 五、各服务深度分析

### 5.1 ser-buser（002）— 后台用户权限管理

**数据库表：** sys_user, sys_role, sys_menu, sys_user_role, sys_role_menu

**RBAC模型：**
```
sys_user (1) ←→ (N) sys_user_role (N) ←→ (1) sys_role
sys_role (1) ←→ (N) sys_role_menu (N) ←→ (1) sys_menu
```

**菜单树：** menu_type: M=目录, C=菜单, F=按钮; parent_id=0为顶级

**核心逻辑：**
- 密码加盐加密（每用户独立salt）
- Google Authenticator双因素认证（bind_google标记）
- 超管super_admin_flag=2不可删除
- 角色已绑定用户不可删除
- 菜单有子记录不可删除
- CheckUrlAuth：通过用户角色→角色菜单→菜单perms/path 校验URL权限

**错误码段：** 6002000+（6002003=UserNotFound, 6002043=UserNameExist等）

### 5.2 ser-bcom（003）— 业务通用配置

**数据库表：** query_enum

**核心功能：**
- 开关配置：从ETCD读写（/slg/switch/bcom/...），非数据库存储
- 枚举查询：从query_enum表读，调i18n翻译标签
- 黑白名单：代理ser-ip的CRUD操作
- 邮箱验证：半成品

**特殊设计：** 开关配置直接读写ETCD，不走数据库

**错误码段：** 6003000+

### 5.3 ser-i18n（004）— 国际化多语言

**数据库表：** langs

**6种语言：** zh_cn(简中), zh_tw(繁中), en(英), id_id(印尼), vi_vn(越南), th_th(泰)

**Key编码规则：** int64，前缀表示Category
- 1xxxx=客户端业务, 2xxxx=客户端报错
- 3xxxx=后台业务, 4xxxx=后台报错
- 5xxxx=服务器业务, 6xxxx=服务器报错（与错误码对应）

**缓存策略：** Redis缓存，LoadLangs预加载，LangTransOne/List优先查缓存

**错误码段：** 6004000+（6004107=KeyNoMatchCategory关键校验）

### 5.4 ser-app（011）— App内容管理

**数据库表：** banner, home_category

**Banner核心逻辑：**
- 名称在同一page_position下唯一
- 仅禁用状态可编辑
- 启用状态不可删除
- 排序冲突自动后移
- 图片存为JSON多语言格式: [{lang:"zh-CN", url:"s3Key"}, ...]
- C端ListBanner：Redis缓存(5分钟) + 预上线时间过滤 + 语言匹配
- 图片URL：DB存s3Key相对路径，返回时拼接完整URL

**分类关联类型：** 1=常规直播, 2=语聊房, 3=体育赛事, 4=真人视讯, 5=短视频

**错误码段：** 6011000+（6011001=BannerNameAlreadyExist）

### 5.5 ser-item（010）— 道具管理

**数据库表：** item, item_publish

**道具编码：** 自动生成 D3001递增

**多场景支持：** item_type存逗号分隔ID（"1,2,3"）
- 1=直播间, 2=语聊房, 3=专属活动

**item vs item_publish分离：** item表存配置（名称/图标/价格），item_publish表存上架信息（标签/排序/有效期/上线状态/前端显示）

**文件类型校验：**
- 静态图标：jpg/png，≤100kb
- 动态图标：gif/json，≤100kb
- 特效：webp，≤5mb

**错误码段：** 6010000+

### 5.6 ser-ip（005）— IP黑白名单管理

**数据库表：** c_black_list（国家ISO黑名单）, sys_white_list（IP白名单）

**黑名单逻辑：** IP→解析国家ISO代码→匹配c_black_list
**白名单逻辑：** 直接匹配IP地址

**开关控制：** 从ETCD读取开关状态，关闭则不做校验

**错误码段：** 6005000+

### 5.7 ser-s3（007）— 文件存储

**数据库表：** s3_config（上传配置）, s3_documents（文件记录）

**配置驱动上传：** 按bs_type从s3_config表读校验规则
- max_size, max_duration, allowed_ext, is_public, expires, path_template

**S3Key生成：** 按path_template生成，支持{owner_id}/{year}/{month}/{day}变量

**两种上传模式：**
1. 小文件直传：UploadFile，文件内容通过RPC传递
2. 大文件预签名：GetUploadPreSign → 客户端直传S3 → UploadComplete通知

**业务类型(bs_type)枚举：** 1=头像原图, 2=缩略图, 3=背景图, 20=Banner图, 21=证件照, ...

**错误码段：** 6007000+（6007113=BigFileNeedSign）

## 九、新模块开发流程（验证总结）

### 粗粒度：编写IDL → 编写Go文件 → 执行相关脚本 → 模块间调用 → 中间件引入 → 业务闭环验证

### 详细步骤：

1. **需求分析** → 数据模型设计 → 表结构设计
2. **编写Thrift IDL**（common/idl/ser-xxx/）定义service/struct/method
3. **执行gen_kitex.sh**（common/script/gen_kitex.sh）生成kitex_gen代码
4. **创建ser-xxx目录**，编写main.go/handler.go
5. **执行gorm_gen.go**（ser-xxx/internal/gen/gorm_gen.go）生成model/query/repo
6. **编写业务代码**：internal/ser/（业务层）、internal/errs/（错误码）、internal/enum/（枚举）、internal/cache/（缓存）
7. **在common/rpc/rpc_client.go中**注册新的客户端单例
8. **在common/pkg/consts/namesp/中**注册服务名和端口Key
9. **在网关biz/router/和biz/handler/中**添加路由和handler
10. **在biz/router/register.go中**注册路由

**gen_kitex.sh核心命令：**
```bash
kitex -module common/pkg -I {idl_dir} -service {service_name} -gen-path ./kitex_gen/ {thrift_file}
```

**gen_mysql.sh核心逻辑：** 遍历所有ser-*目录，执行 `go run internal/gen/gorm_gen.go`

## 十、设计模式和开发规范

### 10.1 设计模式

| 模式 | 使用位置 |
|------|---------|
| 单例(sync.Once) | 所有RPC客户端, ETCD, Redis, DB, S3 |
| 适配器 | lg包(KLogAdapter/HLogAdapter) |
| 建造者 | utils.HttpUt链式API |
| 观察者 | etcd.Hook配置变化回调 |
| Copy-On-Write | etcd Watcher缓存更新 |
| 模板方法 | db/cmd/repo_template.go代码生成 |
| 策略 | ret包错误处理(BizErr/SystemErr) |
| 代理 | 网关Handler代理到RPC服务 |

### 10.2 通用开发规范

| 规范项 | 说明 |
|--------|------|
| 软删除 | delete_at时间戳（0=未删除），查询必须排除 |
| 审计字段 | create_by/at/update_by/at + 操作人姓名冗余 |
| 分页 | pageNo(从1开始)/pageSize, offset=(pageNo-1)*pageSize |
| 缓存 | Cache Aside模式，变更清缓存，防穿透写空 |
| 操作日志 | 增删改通过 rpc.BLogClient().AddActLog() oneway审计 |
| 多语言 | 6种(zh-CN/zh-TW/en/id/vi/th)，ser-i18n翻译 |
| C端Token | Redis key="user:token:"+token, TTL由业务控制 |
| B端Token | Redis key=token本身 + CheckUrlAuth权限 |
| 操作人 | 从tracer context获取（非前端传入） |
| 初始化顺序 | InitKLog → LoadAndWatch → InitRedis → InitMySQL → 解析端口 → 实例化Service → 启动Kitex/Hertz |
| 全部POST | 网关所有业务路由均为POST方法 |
| 错误码段 | 60{服务编号}xxx，如ser-buser=6002, ser-app=6011 |

### 10.3 经验教训

- 回答必须基于实际代码分析，不能凭经验推测
- 不同ser-*模块质量参差不齐，标准答案在 common + 两个网关中
- 不人云亦云，有依据时坚持自己的分析结论
- gen_kitex.sh（中心式）vs make gen（各目录式）是两种不同策略
- B端Token Redis key无前缀（key=token），C端有前缀（"user:token:"+token）
- BlackCheck失败默认放行（C端容错），WhiteCheck失败默认禁行（B端安全优先）
