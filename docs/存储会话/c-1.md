### 二、元认知框架 — 9大维度控制体系（know:元认知.md）

文件包含两个视角（用户描述版 + AI必须遵守版），核心行为要求：

**1. 问题定义与目标对齐**
- 信息不足时先指出缺失项并请求补充，不得自行假设
- 区分意图（写/改/查/分析/判断/设计）、锁定目标（可检验的"输出+条件"）
- 明确任务性质（学习/实战/排错/设计/决策），按对应严谨程度回答
- 锁定作用域边界，不主动扩展至未授权领域

**2. 协作方式与交互控制**
- 按用户指定角色参与，未授权不替代决策
- 信息不足优先澄清关键缺口（最多追问3个决定性问题），不以经验自动补全
- 不擅自拆分阶段，不频繁无效追问

**3. 用户认知与表达控制**
- 默认"结论→理由→步骤/建议→验证"的输出顺序
- 结构化、可复用、可归档的输出形式
- 先摘要层再细节层，避免认知过载

**4. 事实输入与证据基础**
- 只基于用户提供的事实材料推理，不补充未确认事实
- 区分确定结论与推测判断，标注假设前提与待验证点
- 证据冲突时显式指出，不选择性采信

**5. 方案设计与实施落地**
- 给可执行步骤（输入/动作/预期输出），不只给方向性建议
- 锁定方案对象与边界，防止对象漂移
- 评估前置条件、变更影响范围、回退止损方案

**6. 约束、环境与非功能控制**
- 严格遵守硬性限制，约束冲突时暂停提示
- 环境不明不假设，数据不全不构造
- 默认优先安全与稳定

**7. 风险控制与稳定性**
- 默认按高风险、不可逆决策标准处理
- 不得擅自改变目标、补充不存在的事实、替用户做未授权决策
- 信息严重不足时暂停，给出最小下一步动作，不盲目生成

**8. 验证闭环与成功判定**
- 给出通过/不通过的可观察信号与阈值
- 支持多轮迭代，每轮说明改了什么、为什么改
- 明确验收主体与判定权归属

**9. 全局策略控制**
- 防止语义漂移与策略漂移，历史结论不擅自覆盖
- 长期记忆写入/更新必须经用户显式确认
- 全局冲突由用户仲裁，AI不得自行隐性取舍

### 七、新模块开发流程（综合分析+用户确认）

**实际流程（基于代码事实）：**

1. **IDL先行** — 根据需求原型，设计数据模型和表结构，在 `common/idl/ser-xxx/` 下编写 thrift 文件定义 RPC 接口，这是源头
2. **执行 gen_kitex.sh** — 运行 `/Users/mac/gitlab/common/script/gen_kitex.sh`
   - 脚本从 `common/pkg` 目录执行 kitex
   - 如果 ser-xxx 已有 handler.go/main.go，会先拷贝到 common/pkg 再生成后拷回（增量更新方法存根）
   - 生成代码输出到 `common/pkg/kitex_gen/ser_xxx/`
   - 注：Makefile 的 `make gen` 是另一条路径（从各服务目录执行），两者都输出到同一位置
3. **编写GO文件** — 去 ser-xxx 模块完善内部代码逻辑（handler/service/repo/cfg/errs/enum等）
4. **GORM代码生成** — 涉及MySQL时，执行 `go run internal/gen/gorm_gen.go`
   - 实际逻辑：连接ETCD获取DB配置 → 调用 `mysql_gen.GenCodeWithAll()` → 输出 model/query 代码
5. **模块间调用** — 通过 `rpc.XxxClient()` 实现服务间RPC协作
6. **中间件引入使用** — 按需引入 common/pkg 提供的基础设施（日志/缓存/S3/消息队列等）
7. **业务闭环验证** — 端到端验证功能完整

**粗粒度表述：编写IDL → 编写GO → 执行脚本 → 模块间调用 → 中间件引入使用 → 业务闭环验证**

### 十、完整请求链路（HTTP入口 → 数据库 → 返回）

#### A. gate-back 完整链路示例（以创建Banner为例）

```
[客户端] POST /admin/api/app/banner/create
    ↓
[Hertz Server] 收到请求
    ↓
[tracer.IGlTracer.Start()]
    - 生成 UUID trace_id
    - 设置 CORS 响应头（Allow-Origin: *）
    - ctx = metainfo.WithPersistentValue(ctx, "X-Trace-Id", uuid)
    ↓
[middleware.HertzAccLogStr()]
    - 注册 trace_id tag 到访问日志
    - 日志格式: <trace_id=xxx> 200 - 45ms POST /admin/api/app/banner/create
    ↓
[middleware.SetHeaderInfo()]
    - 从 HTTP Header / Cookie 提取并存入 metainfo:
      X-Client-IP, X-User-Language, X-User-Device-ID,
      X-User-Device-Type, X-User-Token（优先Header，降级Cookie session_token）
    ↓
[middleware.WhiteCheck()]
    - rpc.IpClient().OnWhiteList(ctx) → ser-ip 判断 IP 是否在白名单
    - 失败默认拒绝（fail-closed）
    ↓
[middleware.BackAuthMiddleware()]
    - 从 metainfo 取 token → Redis 直接以 token 为 key 查询
    - 反序列化为 BUserTokenPayload{ID, UserName}
    - 设置 X-User-ID, X-User-Name 到 metainfo
    - rpc.BUserClient().CheckUrlAuth(ctx, "/admin/api/app/banner/create") → RBAC 权限
    ↓
[app_router: Api.POST("/app/banner/create", app.CreateBanner)]
    ↓
[handler: app.CreateBanner(ctx, c)]
    → rpc.Handler(ctx, c, rpc.AppClient().CreateBanner)
    ↓
[rpc.Handler[REQ, RESP] — common/rpc/handler.go]
    1. reflect 判断 REQ 是否指针类型，相应初始化
    2. c.BindAndValidate(bindTarget) — 解析JSON请求体+校验
    3. rpcMethod(ctx, reqData) — 携带 metainfo 的 ctx 发起 RPC 调用
       （metainfo 通过 TTHeader 协议自动传递到下游服务）
    4. c.JSON(ret.HttpOne(ctx, resp, I18nClient(), err)) — 统一返回
    ↓
[RPC 传输 → ser-app 接收]
    - Kitex Server 端中间件打印访问日志（init.go 中的 WithMiddleware）
    - 格式: <trace_id> packageName.serviceName.methodName | cost=xx | err=xx
    ↓
[ser-app handler.go: AppServiceImpl.CreateBanner()]
    → bannerService.CreateBanner(ctx, reqData)
    ↓
[ser-app internal/ser/BannerService.CreateBanner()]
    1. 去重图片（按语言）
    2. 校验参数
    3. bannerRepo.ExistsByName() — 检查名称唯一性
    4. 图片URL转相对路径（mys3.GetRelativePaths）
    5. bannerRepo.ShiftAndCreate() — 事务：移位排序值 + 插入
    6. bannerCache.DelAll(ctx) — 清除Redis缓存
    7. rpc.BLogClient().AddActLog() — 异步审计日志
    ↓
[ser-app internal/rep/BannerRepo.ShiftAndCreate()]
    query.Q.Transaction(func(tx *query.Query) error {
        tx.Banner.Where(pagePosition, sortValue>=req).UpdateSimple(SortValue.Add(1))
        tx.Banner.Create(banner)
    })
    ↓
[GORM → TiDB/MySQL 执行SQL]
    ↓
[返回链路] resp → handler.go → RPC → gateway → ret.HttpOne → JSON
```

#### B. 统一响应格式

```json
// 成功
{"traceId": "uuid", "code": 0, "msg": "", "data": {...}}
// 失败
{"traceId": "uuid", "code": 6011001, "msg": "Banner名称已存在", "data": null}
```

- `ret.HttpOk(ctx, data)` → code=0
- `ret.HttpErr(ctx, err, i18nClient)` → 从 BizStatusError 提取 code，调用 i18n 翻译 msg
- `ret.HttpOne(ctx, data, i18nClient, err)` → 自动判断成功/失败

#### C. RPC错误传递机制

```
服务层 → ret.BizErr(ctx, errCode, "msg")
       → kerrors.NewBizStatusError(code, msg) — Kitex BizStatusError
网关层 → ret.HttpErr 从 BizStatusError 提取 code
       → I18nClient().LangTransOne(ctx, code) 翻译
       → 返回 R{Code: code, Msg: 翻译后消息}
```

### 十一、common/pkg 全量包详解

#### 1. ret/ — 响应标准化

| 函数 | 用途 | 场景 |
|------|------|------|
| `HttpOk(ctx, data)` | 成功返回 | 网关→客户端 |
| `HttpErr(ctx, err, trans)` | 错误返回（自动翻译） | 网关→客户端 |
| `HttpOne(ctx, data, trans, err)` | 懒人版（自动判断） | 最常用 |
| `BizErr(ctx, code, msg)` | 创建RPC业务错误 | 服务层→网关 |
| `BizErrExt(ctx, code, extra, msg)` | 带扩展字段的RPC错误 | 特殊场景 |
| `RpcOne(ctx, result, err, code)` | RPC层懒人版 | 服务间调用 |
| `GetBizErrCode(err, default)` | 提取错误码 | 错误处理 |

#### 4. etcd/ — 配置管理（两种模式）

| 模式 | 函数 | 特点 | 适用 |
|------|------|------|------|
| 直接读取 | `DirectGet(ctx, key)` | 每次网络调用，强一致 | 初始化/低频配置 |
| 缓存监听 | `LoadAndWatch(prefix)` + `Get(ctx, key)` | atomic.Value 零分配读，后台 Watch 更新 | 高频读取/运行时配置 |

**Watcher 机制：**
- 首次 `LoadAndWatch("/slg/conf/")` 加载前缀下所有 KV 到内存 map
- 启动 watchLoop goroutine 监听变更（PUT/DELETE）
- PUT → Copy-On-Write 更新 atomic.Value + 触发 Hook 回调
- DELETE → 从 map 移除

**env_tool/：** 本地开发环境工具，将远程ETCD配置拷贝到本地ETCD，支持本地替换服务

### 十九、S3文件系统设计

**两种访问路径：**
1. **公开文件** (`s3/public/...`) → 直接302到S3域名URL（无签名）
2. **私有文件** (`s3/其他/...`) → 生成预签名URL（默认15分钟过期）

**上传流程：**
- 客户端 → 网关 → `ser-s3.UploadFile()` → 校验(大小/格式/业务类型) → `mys3.Client.PutObject()` → 返回URL
- 或预签名上传：客户端 → `GetUploadPreSign()` → 获取预签名POST URL → 客户端直传S3

**路径模板变量：** `{owner_id}`, `{year}`, `{month}`, `{day}` — 按用户和日期组织文件

**配置驱动：** s3_config表定义每种business_type的限制（大小/格式/公私/过期时间）
