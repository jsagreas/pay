### 二、元认知框架（know:元认知.md）
文件分上下两部分，上半部分定义用户应提供的信息维度，下半部分定义AI必须遵守的行为准则。九大类：

1. **问题定义与目标对齐**：先理解背景/意图/目标/任务性质/问题边界/紧急程度/可分解性；信息不足必须追问不得假设；区分阶段性目标与终局目标

3. **用户认知与表达控制**：根据用户水平调整深度；默认"结论→理由→步骤→验证"顺序；结构化输出、控制信息密度；术语首次出现时给人话释义

5. **方案设计与实施落地**：给可执行步骤（输入/动作/预期输出）；锁定讨论对象避免漂移；先评估必要性与可行性；考虑前置条件/变更影响/回退止损
6. **约束、环境与非功能控制**：严格遵守硬性限制；目标冲突时请求用户排序；环境/依赖不明确必须暂停；默认优先安全与稳定
7. **风险控制与稳定性**：未明确风险等级按高风险处理；必须说明失败处理策略；不得越权替代决策；结论标注时效窗口与前提条件；输出前自检
8. **验证闭环与成功判定**：给出可观察的通过/不通过信号；区分"逻辑正确"与"用户认可"；支持多轮迭代每轮说明变更点；验证方案考虑成本可执行性
9. **全局策略控制**：围绕用户声明的优化目标；历史结论冲突必须显式指出；不得自行建立/更新长期记忆；监控策略漂移；阶段封版机制

### 三、脚手架核心机制（基于 common 代码分析）

#### 3.1 IDL 与代码生成
- **IDL 位置**：common/idl/ser-xxx/service.thrift，这是服务契约的源头
- **生成脚本**：`/Users/mac/gitlab/common/script/gen_kitex.sh` — 扫描 idl 目录，Kitex 编译 thrift 生成桩代码到 common/pkg/kitex_gen/
- **GORM 生成**：各服务内部 `internal/gen/gorm_gen.go`（如 ser-app/internal/gen/gorm_gen.go），执行后输出 model/ 和 query/ 到 gorm_gen 目录

#### 3.2 RPC 客户端工厂（common/rpc/rpc_client.go）
- 19 个懒加载单例客户端（sync.Once 模式）
- InitRpcServerParams: ETCD 注册、端口复用、多路复用传输、中间件日志
- InitRpcClientParams: ETCD 解析器、1200s 超时、多路复用连接
- 通用 Handler 封装：Handler[REQ,RESP]、HandlerNoParam[RESP]、HandlerWithCSV[REQ,RESP]

#### 3.3 网关路由模式
- **gate-back (B端)**：`/admin/open` (SetHeaderInfo → WhiteCheck) + `/admin/api` (+ BackAuthMiddleware)
- **gate-font (C端)**：`/open` (SetHeaderInfo → BlackCheck) + `/api` (+ FontAuthMiddleware)
- Handler 统一委托：`rpc.Handler(ctx, c, rpc.XxxClient().Method)`
- 路由注册：每个模块 biz/handler/xxx/ + biz/router/xxx/ → register.go 汇总

#### 3.4 统一响应与错误体系
- 响应封装：`ret.R{TraceId, Code, Msg, Data}`，通过 ret.HttpOne/HttpErr 输出
- 错误码：6 位数 `6{服务编号}{序号}`，如 6011001 = ser-app Banner名重复
- 业务错误：`ret.BizErr(ctx, errCode)` 包装，支持 i18n 翻译

#### 3.5 上下文传递链
- 网关层 SetHeaderInfo 提取 → Kitex metainfo 传播 → 服务层 tracer.GetXxx(ctx) 获取
- 传递字段：UserID、Token、TraceID、Language、DeviceID、DeviceType、ClientIP

#### 3.6 服务内部分层（从工程代码归纳的通用模式）
```
handler.go        → 薄层委托，不含业务逻辑
internal/ser/     → 业务逻辑（校验、编排、RPC调用、缓存管理）
internal/rep/     → 数据访问（GORM 查询，软删除过滤）
internal/domain/  → 领域辅助（部分服务有：校验器/映射器/缓存/日志构建器）
internal/cfg/     → 配置初始化（MySQL/Redis，sync.Once 单例，ETCD 取参）
internal/enum/    → 常量定义（审计日志页面/动作、业务枚举）
internal/errs/    → 错误码定义
internal/gorm_gen/→ ORM 自动生成代码（model/ query/ repo/）
```

#### 3.7 数据库设计约定（从现有表结构归纳）
- 主键：bigint AUTO_INCREMENT
- 软删除：delete_at bigint (0=存活) 或 delete_by bigint (0=未删)

## 深度分析补充（第二轮逐行级分析结果）

> 以下内容基于对 common/rpc、common/pkg/middleware、common/pkg/consts、两个网关全部handler/router、所有IDL文件、7个服务main.go+handler.go+内部结构的逐行阅读。

### 七、RPC Handler 处理流程（common/rpc/handler.go）

#### 7.1 三种Handler泛型函数

| 函数 | 签名 | 用途 |
|------|------|------|
| `Handler[REQ,RESP]` | `(ctx, c, rpcMethod func(ctx, REQ, ...callopt) (RESP, error))` | 标准请求：反射初始化REQ → BindAndValidate → rpcMethod → ret.HttpOne |
| `HandlerNoParam[RESP]` | `(ctx, c, rpcMethod func(ctx, ...callopt) (RESP, error))` | 无参请求：直接调用 → ret.HttpOne |
| `HandlerWithCSV[REQ,RESP]` | 同Handler但多CSV检查 | 支持CSV导出：同Handler流程 + 检查resp是否实现CSVResponse接口 → 二进制下载 |

**CSVResponse接口**：`GetFilename() string` + `GetData() []byte`

#### 7.2 Handler内部反射逻辑
```
1. var reqData REQ → reflect.ValueOf(&reqData).Elem()
2. 指针类型？→ reflect.New(val.Type().Elem()) 创建实例，bindTarget = reqData
   值类型？→ bindTarget = &reqData
3. c.BindAndValidate(bindTarget) → 解析JSON/Form + 验证
4. rpcMethod(ctx, reqData) → (resp, err)
5. c.JSON(ret.HttpOne(ctx, resp, I18nClient(), err))
```

#### 7.3 RPC Server 中间件
- 日志格式：`<TraceID> PackageName.ServiceName.MethodName | cost=耗时 | err=错误 | from:调用方地址:服务名.方法名`

### 八、认证中间件完整链路

#### 8.1 前台认证 FontAuthMiddleware（gate-font /api 路由）
```
Token提取(tracer.GetUserToken) → 空检查(6000008)
→ Redis查询(rds.Get("user:token:" + token)) → 空检查(6000009)
→ JSON反序列化(CUserTokenPayload{ID,UID,LastLoginIp,LastLoginTime,Device,DeviceType,NickName})
→ ID=0检查(6000009) → 设置ctx(UserID,NickName) → c.Next
```

#### 8.2 后台认证 BackAuthMiddleware（gate-back /admin/api 路由）
```
Token提取(tracer.GetUserToken) → 空检查(6000008)
→ Redis查询(rds.Get(token)) ← 注意：无前缀！直接用token作key
→ JSON反序列化(BUserTokenPayload{ID,UserName})
→ ID=0检查(6000009) → 设置ctx(UserID,UserName)
→ ★URL权限检查: rpc.BUserClient().CheckUrlAuth(ctx, c.FullPath()) → 失败(6000010)
→ c.Next
```

#### 8.3 前后台认证差异对比

| 维度 | 前台(Font) | 后台(Back) |
|------|-----------|-----------|
| Redis Key | `"user:token:" + token` | `token`（无前缀） |
| Payload类型 | CUserTokenPayload(7字段) | BUserTokenPayload(2字段) |
| 用户名字段 | NickName | UserName |
| URL权限检查 | 无 | 有(CheckUrlAuth) |
| IP检查 | BlackCheck(Fail-Open) | WhiteCheck(Fail-Closed) |

#### 8.4 BlackCheck vs WhiteCheck

| | BlackCheck(前台) | WhiteCheck(后台) |
|---|---|---|
| RPC调用 | IpClient().OnBlackList() | IpClient().OnWhiteList() |
| 错误时 | **放行**(log warning) | **禁止**(log error + abort) |
| 命中时 | check=true→阻止(6000005) | check=false→阻止(6000006) |

#### 8.5 完整HTTP请求处理链
```
tracer.Start(生成TraceID+CORS) → SetHeaderInfo(提取IP/Lang/DeviceID/DeviceType/Token到ctx)
→ [前台]BlackCheck / [后台]WhiteCheck
→ [/api路由]FontAuth或BackAuth
→ Handler(rpc.Handler委托RPC调用)
→ ret.HttpOne(成功/失败统一响应)
```

### A. 多币种钱包需求文档（42张）

#### A1. 需求背景与目标
- 建立清晰、安全、可扩展的多币种钱包系统
- 支撑：多币种充值支付、游戏投注结算、礼物打赏主播收益、提现合规风控(KYC)、资金流水对账
- 适用端：iOS / Android / H5，用户角色：普通用户 + 主播

#### A2. 钱包总体设计
- **币种维度隔离**：每个币种独立余额与账本
- 当前支持：BSB(平台币,1BSB=0.1USDT)、VND、IDR、THB
- 金额展示规则：

| 币种 | 千分位 | 小数位 |
|------|--------|--------|
| VND  | .      | 整数   |
| IDR  | .      | 整数   |
| THB  | ,      | 2位    |
| BSB  | ,      | 2位    |

- 当前选中币种 = 所有操作的唯一数据源（充值/提现/兑换/记录）
- 不支持跨币种混合展示与操作

#### A3. 充值模块

**充值方式总览：**

| 类型 | 方式 | 平台承载页 | 回调方式 |
|------|------|-----------|---------|
| 虚拟币 | USDT(TRC20/ERC20) | 是 | 区块链监听 |
| 银行 | 银行转账 | 否(三方) | 三方回调 |
| 电子钱包 | QRIS/PromptPay | 否(三方) | 三方回调 |

**USDT充值：** 游客也可充值，网络选TRC20(默认)/ERC20/BEP20，展示收款地址+二维码+复制按钮

**银行转账充值：** 用户选金额→创建订单→三方承载页展示转账信息→回调更新状态

**扫码充值(QRIS/PromptPay)：** 印尼/泰国法币推荐方式，三方二维码+倒计时

**重复转账拦截规则：**
- 维度：同UserID + 同币种 + 同金额，状态=待支付/进行中
- 1-2笔：弹窗警告，可继续
- 3笔(上限)：强制拦截，不允许创建新订单
- 不适用于USDT链上充值

**额外奖金选择规则：**
- 位于余额下方、支付方式上方
- 默认"无奖金"或最优匹配活动
- 选择后金额标签显示+比例角标（如+50%）
- 额外奖金=充值金额×奖金比例
- 门槛校验：低于最低充值门槛弹提示
- 单笔仅限一个奖金活动
- 奖金入账奖励钱包，需完成X倍流水

#### A4. 兑换模块（法币→BSB）
- 支持VND/IDR/THB→BSB（BSB币种下无兑换入口）
- 汇率口径：1BSB = X法币
- 赠送奖励（如10%），赠送部分需完成X倍流水后可提现
- 实际到账 = 兑换获得 + 赠送

#### A5. 提现模块

**KYC与提现关系：**
- 未认证：弹窗→"去认证"(解锁全渠道) / "继续提现"(仅USDT)
- 已认证：直接进入提现方式选择

**提现规则：**
- 法币充值→法币提现
- USDT充值→USDT提现
- 法币+USDT混合充值→法币提现
- 法币→BSB→法币提现

**提现方式：**
- 银行卡提现：首次填写银行信息，后续自动填充
- 电子钱包(GoPay/OVO/DANA/PromptPay等)：首次填写，后续复用
- BSB提现：支持全渠道，显示汇率转换
- 账户持有人 = KYC认证姓名（自动填充不可编辑）

**提现限制（后台配置）：** 单笔限额、单日限额、手续费规则、T+N到账时间

**校验提示：** 超出每日限额/低于单笔最低→弹窗提示

#### A6. 记录模块
- 4类：提现、充值、兑换、奖励
- 钱包当前币种 = 记录当前币种，不支持跨币种混合
- 详情页：订单号、金额明细、状态、时间、方式、失败原因

#### A7. 非功能性需求
- 对账一致性：账务流水与账户余额强一致性校验
- 幂等性：充值回调防重复入账
- KYC强制校验、提现账户白名单、姓名强一致校验
- 大额/异常频率触发人工审核

### B. 币种配置需求文档（24张）

#### B1. 文档背景
- 钱包体系承载用户在平台内资金流转与消费（娱乐消费+igaming投注）
- 通过钱包类型划分与资金流向规则确保用户体验一致、资金安全可控

#### B2. 资金钱包体系

**钱包定义：** 资金钱包承载真实资金，支持平台币+多法币+多虚拟币

**钱包结构（5种）：**

| 钱包类型 | 资金来源 | 资金去向 | 说明 |
|---------|---------|---------|------|
| 中心钱包 | 充值/投注返奖/其他钱包转入 | 投注/消费/提现(需完成稽核) | 用户主资金钱包 |
| 奖励钱包 | 活动/任务返奖/VIP/分销奖励 | 稽核完成→中心钱包 | 奖金暂存，需区分各活动 |
| 主播钱包 | 礼物打赏/主播活动奖励 | 提现/转入中心钱包 | 主播收益专用 |
| 代理钱包(预留) | 代理分成 | 提现/转入中心钱包 | 代理佣金专用 |
| 场馆钱包(预留) | 中心+奖励钱包转入 | 退出场馆→回中心+奖励 | igaming场馆临时钱包 |

**余额规则：**
- 资金钱包余额 = 中心钱包余额 + 奖励钱包余额
- 消费优先扣中心钱包，再扣奖励钱包
- 混合投注返奖按金额占比分别计入对应钱包

#### B3. 币种配置

**币种类型：**

| 类型 | 示例 |
|------|------|
| 法定货币 | IDR、VND、THB |
| 加密货币 | USDT、BTC、ETH |
| 平台货币 | BSB |

**基准币种：**
- 推荐USDT，配置后不可更改
- BSB价值：10BSB = 1USDT，汇率锚定USDT
- 用途：活动奖金计算、多币种投注统计、钱包资产统计、财务对账

**后台配置页面字段（18个）：**
序号、币种代码(唯一)、币种名称、币种类型、币种国家、币种图标(SVG/PNG≤5K)、货币符号、千分位符号、小数点符号、实时汇率、平台汇率、汇率浮动(%)、入款汇率、出款汇率、偏值(阈值%)、排序、状态(启用/禁用)、操作人、操作时间

**编辑弹窗：** 可上传币种图标、配置汇率浮动(2位小数%)、配置阈值(2位小数%)、启用/禁用状态

#### B4. 汇率规则

**汇率类型：**
- **实时汇率**：3家三方API取平均值
- **平台汇率**：初始=实时汇率，偏差值≥阈值时更新
- **入款汇率** = 实时汇率 × (1 + 汇率浮动%)
- **出款汇率** = 实时汇率 × (1 - 汇率浮动%)
- **偏差值** = |实时汇率 - 平台汇率| / 实时汇率 × 100
- **阈值**：可配置，偏差值>阈值时更新平台汇率+入出款汇率+生成日志

**汇率展示规则：**

| 货币类型 | C端展示 | 系统侧 |
|---------|--------|-------|
| 法定货币 | 小数点后2位 | 小数点后2位(不足补0,超出四舍五入) |
| 加密货币 | 小数点后8位 | 小数点后8位(不足补0,超出四舍五入) |

**汇率日志：** 在操作日志页增加"汇率日志"页签，每次平台汇率变更生成一条

### D. 多币种钱包产品原型（32张）

主要页面流：
1. **三方充值页(QRIS)**：二维码+倒计时+订单失效两态
2. **个人中心**：钱包余额+币种切换(BSB/VND/IDR/THB)+我的权益+功能菜单
3. **充值中心**：按币种展示不同支付方式和金额档位
   - BSB：USDT(推荐)/银行/电子钱包，100-100000档位，有汇率换算
   - VND：银行转账，200K-200M档位
   - IDR：电子钱包扫码(推荐)/银行，150K-150M档位
   - THB：电子钱包扫码(推荐)/银行，300-350K档位
4. **USDT支付页**：网络选择(TRC20/ERC20/BEP20)+收款地址+二维码
5. **奖金选择**：半屏浮窗列出可用活动，选后显示+比例角标和额外金额
6. **通用回调**：充值进行中弹窗(查看详情/继续转账)，重复充值弹窗(1-2笔可继续/3笔强制拦截)
7. **兑换BSB**：法币余额→输入金额→兑换获得+赠送10%→实际到账
8. **提现触发认证**：KYC弹窗→认证表单(姓名/国家/手机/证件/上传正反面)
9. **提现方式**：
   - 越南：银行卡(T+1)
   - 印尼：GoPay/OVO/DANA/ShopeePay(即时)+银行卡(T+1)
   - 泰国：PromptPay(即时)+银行卡(T+1)
   - BSB：全渠道+汇率展示
10. **记录页**：提现/充值/兑换/奖励4个Tab，详情弹窗(含失败原因和换方式按钮)

### E. 财务管理需求文档（25张）— 参考模块

#### E1. 通道配置
- **代收通道**：币种/通道名称/支付类型(唤醒/原生/混合/扫码)/单笔限额/代收费率/代收成本/入账方式/排序/状态
- **代付通道**：币种/通道名称/单笔限额/代付费率/代付成本/排序/状态
- **通道轮询规则**：三种模式(随机/轮询/权重)，按币种分别配置，可勾选启用通道和排序

#### E2. 支付配置
- **充值方式**：币种/充值方式/图标(PNG≤512K)/充值档位/金额范围/关联代收通道/充值费率/大额通道额度/排序/状态
- **提现方式**：币种/提现方式/图标/关联代付通道/提现手续费/排序/状态

#### E3. 充值记录
- 搜索：订单号/三方订单号/时间/币种/用户编号/UID/状态(充值中/已完成/已失败/已关闭)
- 字段(22个)：含充值金额/手续费/实际到账/代收费率/三方费用/平台利润等
- **充值流程**：用户选币种→选方式→选金额→提交→轮询匹配通道→创建订单(待支付)→通道创建编号→用户支付→回调→验签→上分(已完成)
- 导出需密码+谷歌验证码

#### E4. 提现记录
- 搜索：订单号/三方订单号/时间/币种/用户ID/UID/手机号/订单状态/审核状态/出款状态/锁定状态
- **订单状态**：提现中/已打款/已拒绝/已关闭
- **审核状态**：待审核/风控通过/风控拒绝/财务通过/财务拒绝
- **出款状态**：待出款/出款中/出款成功/出款失败
- **操作**：风控审核→财务审核→出款审核→锁定/解锁/强制解锁
- **提现流程**：用户选币种→实名认证检查→自动风控审核(不过→人工风控)→提交订单→金额检验→人工财务审核→人工出款审核→通道出款→回调

#### E5. 人工修正
- **充值补单(B前缀)**：关联原充值订单，含补单金额/补单赠送/稽核倍数/稽核流水，审核通过→上账
- **人工加款(A前缀)**：选用户→选币种→按钱包类型(中心/奖励/代理/主播)分别输入金额，加款类型(运营补偿/运营奖励/财务修正/测试加款)
- **人工减款(M前缀)**：选用户→选币种→按钱包类型分别输入减款金额(校验余额≥0)，减款类型(异常追回/风控处罚/财务冲正/费用调整)
- 自己提交不允许自己审核

#### E6. 数据统计
- **通道统计**：代收/代付维度，支持日期+时间段+通道+币种筛选，卡片汇总+表格明细
- **充值统计**：按日统计，含充值人数/首充人数/订单数/成功率/总额/入账/手续费/成本/利润
- **提现统计**：按日统计，含提现人数/首提人数/订单数/成功率/总额/入账/手续费/成本/利润
- UTC时区转换：后台站点时区(如UTC+7)→服务器UTC+0存储
