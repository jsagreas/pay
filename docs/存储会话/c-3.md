### 一、严格约束核心摘要（know:严格约束.md）

**第一约束 — 只读默认**
- 所有操作默认只读，未经用户口头明确同意，严禁增删改任何文件
- 分析过程中严禁产生写操作，分析结束不得自动进入编码阶段

**行为红线**
- 禁止猜测、臆想 — 必须有依据、有凭证、有理由
- 禁止局部片面分析 — 必须全面完整深入，不能以偏概全导致级联错误
- 禁止擅自编码 — 编码前必须得到明确同意
- 禁止无中生有 — 不能把没问题的说成有问题
- 禁止无脑咨询 — 能通过分析代码获取的信息必须自行获取，不能推给用户
- 必须主动问询 — 无法通过分析获取的关键前置信息，必须主动询问
- 不能人云亦云 — 要有自己主见，基于充分分析据理力争，有则改之无则加勉

**思考标准**
- 宏观：清楚架构、模块、流程、调用链路、技术栈、规范标准
- 微观：清楚每个文件的职责、每个函数的作用、每个变量的含义

**执行方式**
- 两条腿走路：深度分析项目脚手架（架构/规范/链路）+ 深度分析需求文档（功能/细节/前后端分工）

**互动节奏**

### 二、元认知体系核心摘要（know:元认知.md）

该文件分为两部分：上半部分是维度定义框架，下半部分（分隔线后）是 **AI 必须遵守的行为准则版本**，九大类共覆盖 70+ 个维度。

**九大类及核心行为要求：**

1. **问题定义与目标对齐** — 搞清问什么/要什么；背景不足必须追问不得假设；目标必须可检验（输出+条件）；区分阶段性目标与终局目标
2. **协作方式与交互控制** — 按指定角色参与；未授权不替代决策；信息不足优先澄清关键缺口（最多3个高影响问题）；不擅自拆分阶段
3. **用户认知与表达控制** — 默认结构化中等深度；输出顺序：结论→理由→步骤→验证；术语首次出现给人话释义
4. **事实输入与证据基础** — 只能基于用户提供的事实推理；区分确定结论/推测/假设并显式标注；证据冲突必须指出不得选择性采信
5. **方案设计与实施落地** — 给可执行步骤（输入/动作/预期输出）；锁定对象边界防漂移；先评估方案必要性与可行性再设计
6. **约束、环境与非功能控制** — 严格遵守硬限制；环境/依赖不明必须暂停；冲突时列出冲突点请求排序；默认保守路径
7. **风险控制与稳定性** — 未明确风险等级默认按高风险处理；禁止越权；信息严重不足时暂停而非猜测继续；结论必须标注前提和时效
8. **验证闭环与成功判定** — 给出通过/不通过的可观察信号；区分逻辑正确与用户认可；支持多轮迭代，每轮说明改了什么/为什么
9. **全局策略控制** — 防止策略漂移；语义锁定后不得隐式改写；历史结论与新输入冲突必须显式指出；长期记忆写入需用户确认

- `db/mysql` — GORM + TiDB 初始化
- `db/cmd` — **GORM Gen 代码生成器**（generate.go + repo_template.go）
- `mys3` — AWS S3 封装
- `middleware` — 中间件（SetHeaderInfo/BlackCheck/WhiteCheck/FontAuth/BackAuth/RedirectS3）

#### 9.2 common/rpc — RPC 客户端工厂

**文件**：`rpc_client.go`、`init.go`、`handler.go`

**rpc_client.go**：
- 每个服务一个 `sync.Once` 惰性单例客户端
- 模式：`var xxxService *xxxclient.Client` + `var xxxOnce sync.Once` + `func XxxClient() xxxclient.Client`
- 服务发现通过 ETCD Resolver 自动完成（`InitRpcClientParams` 中配置）

**init.go**：
- `InitRpcServerParams(servName string, servPort int) []server.Option` — 服务端配置（端口复用/多路复用/ETCD注册/TTHeader链路追踪/访问日志中间件）
- `InitRpcClientParams() []client.Option` — 客户端配置（ETCD解析/超时/多路复用/TTHeader）

**handler.go** — 三个泛型处理器：
- `Handler[REQ, RESP]` — 标准流程：反射初始化REQ → BindAndValidate → 调RPC → ret.HttpOne 统一返回
- `HandlerNoParam[RESP]` — 无参数版，跳过绑定直接调RPC
- `HandlerWithCSV[REQ, RESP]` — 支持CSV下载，如果resp实现CSVResponse接口则返回文件流

#### 9.3 common/idl — Thrift IDL 定义

- 目录结构：`common/idl/{ser-xxx}/` 每个服务一个子目录
- 每个服务至少包含 `service.thrift`（服务接口定义）+ 若干类型定义文件
- 命名空间规范：`namespace go ser_xxx`（下划线风格）
- 服务接口命名：`service XxxService { ... }`（大驼峰）
- 请求/响应命名：`XxxQueryReq`/`XxxQueryResp`（大驼峰 + Req/Resp 后缀）
- 字段注解：`(api.body = "fieldName")` 供 Hertz 绑定使用
- 当前共约 72 个 .thrift 文件，覆盖 20+ 个服务

#### 9.4 gate-back — B端管理网关

- 112 个 HTTP 端点，覆盖 14 个业务模块
- 三个路由组：
  - `/admin/open` — 无需鉴权（中间件：SetHeaderInfo → WhiteCheck）
  - `/admin/api` — 需要鉴权（中间件：SetHeaderInfo → WhiteCheck → BackAuthMiddleware）
  - `/s3/*path` — S3 文件代理转发
- 路由注册入口：`biz/router/register.go` → `GeneratedRegister()` 中逐个调用各模块 Router
- Handler 极薄：每个方法一行 `rpc.Handler(ctx, c, rpc.XxxClient().Method)`
- 启动流程：ETCD初始化 → Redis初始化 → S3初始化 → Hertz服务启动（ETCD注册）

#### 9.5 gate-font — C端客户网关

- 65 个 HTTP 端点
- 两个路由组：
  - `/open` — 公开接口（中间件：SetHeaderInfo → BlackCheck）
  - `/api` — 需Token认证（中间件：SetHeaderInfo → BlackCheck → FontAuthMiddleware）
  - `/s3/*path` — S3 文件代理转发
- 与 gate-back 架构完全一致，区别：路径前缀不同、安全策略不同（白名单 vs 黑名单）、认证方式不同

**两个网关的关键区别**：
| 维度 | gate-back（B端） | gate-font（C端） |
|------|-----------------|-----------------|
| 路径前缀 | /admin/open、/admin/api | /open、/api |
| 访问控制 | WhiteCheck（白名单准入） | BlackCheck（黑名单拦截） |
| 认证 | BackAuthMiddleware | FontAuthMiddleware |
| 面向 | 后台管理人员 | 终端用户（App/H5/Web） |

### 十、开发工作流（代码实证 + 用户校正）

新模块从0到1的实际流程（粗粒度）：

**1. 编写 IDL 文件（源头）**
- 基于需求文档和原型，先设计数据模型和表结构
- 在 `common/idl/ser-xxx/` 下定义 .thrift 文件（类型 + 接口）
- 这是一切代码生成的起点，接口契约先行

**2. 执行 gen_kitex.sh 脚本**
- 路径：`/Users/mac/gitlab/common/script/gen_kitex.sh`
- 行为：自动扫描 `common/idl/` 下所有子目录中的 `*service.thrift` 文件
- 生成目标：`common/pkg/kitex_gen/` 下对应的 RPC 桩代码
- 关键机制：如果服务目录已存在 `handler.go` + `main.go`，脚本先拷贝出来保护，生成完再拷贝回去；全新模块则直接生成骨架代码
- 执行后清理：删除 `bootstrap.sh`、`kitex_info.yaml`、`build.sh`、`scripts/` 等 Kitex 副产物

**3. 编写 GO 文件（模块内业务）**
- 在 `ser-xxx/` 下完善 `internal/` 的业务代码
- 涉及 MySQL 时，编写一个 `internal/gen/gorm_gen.go`（参考 `ser-app/internal/gen/gorm_gen.go`）
  - 调用 `mysql_gen.GenCodeWithAll(dbname, serviceName, prefix)` 或 `GenCode(dbname, serviceName, prefix, tables...)`
  - 生成产物（输出到 `ser-xxx/internal/gorm_gen/` 下）：
    - `model/` — 表结构体（每表一个 `xxx.gen.go`）
    - `query/` — 类型安全查询方法（每表一个 `xxx.gen.go` + `gen.go` 入口）
    - `repo/{modelname}r/{modelname}r.go` — **自动生成的标准 CRUD 方法集**（模板来自 `common/pkg/db/cmd/repo_template.go`）
  - repo 模板提供的方法：Create、BatchCreate、Update、BatchUpdate、BatchSave、UpdateByIds、UpdateByConds、UpdateFields、UpdateFieldsByConds、DeleteById、DeleteByConds、GetOneById、GetOneByConds、GetListByIds、GetListByConds、GetPageByConds、GetValuesByConds、Count、Exists

**4. 模块间调用**
- 通过 `common/rpc/` 的客户端工厂调用其他服务：`rpc.XxxClient().Method(ctx, req)`

**5. 相关中间件引入使用**
- 复用 `common/pkg/` 中已封装的 Redis、Kafka、ES、ETCD、S3 等组件

**6. 业务闭环验证**
- 网关路由接入、端到端链路打通、功能验证

说明：`domain/` 层并非每个服务都有，`rep/` 层是在自动生成的 `gorm_gen/repo/` 基础上做业务扩展查询。各服务因不同开发者风格存在差异，以 common + 两个网关的模式为准。

### 十二、关键设计规范（代码实证）

| 规范项 | 标准 | 依据 |
|-------|------|------|
| ID生成 | 雪花算法 int64 | utils 包、model 中 ID 类型 |
| 时间字段 | int64 毫秒时间戳 | Thrift IDL 中用 i64、model 中用 *int64 |
| 审计字段 | create_by、update_by、create_at、update_at | 所有 model 结构体 |
| 软删除 | delete_at=0 未删除，非零=删除时间戳；delete_by 记录操作人 | model + 查询条件 .Where(t.DeleteAt.Eq(0)) |

| 统一响应 | `{traceId, code, msg, data}` | common/pkg/ret |
| 链路追踪 | X-Trace-Id 贯穿全链路，通过 TTHeader 在 RPC 间传递 | contextx、middleware、init.go |

### 十三、端到端请求生命周期（完整链路）

以 **B端管理后台** 一个典型的已认证POST请求为例（如 `/admin/api/buser/user/page`）：

```
客户端请求 → Hertz HTTP Server
  │
  ├─1. IGlTracer.Start()
  │    生成 UUID → 设置 X-Trace-Id 响应头 → 写入 metainfo（贯穿全链路）
  │    设置 CORS 响应头（Access-Control-Allow-Origin: *）
  │
  ├─2. HertzAccLogStr()
  │    记录请求日志（含 trace_id、method、path）
  │
  ├─3. SetHeaderInfo() 中间件
  │    从请求头提取：ClientIP、Language、DeviceID、DeviceType
  │    Token获取：先查 X-User-Token header → 不存在则查 session_token cookie
  │    全部写入 metainfo.WithPersistentValue（通过 TTHeader 跨 RPC 传递）
  │
  ├─4. WhiteCheck() 中间件【B端独有】
  │    调用 rpc.IpClient().OnWhiteList(ctx) → ser-ip 服务
  │    ├─ ser-ip 内部：读取 ETCD 开关 etcd_key.BackendWhite
  │    │   如果开关关闭(Status=0)→ 直接放行（返回false+nil）
  │    │   如果开关开启 → 获取客户端IP → 查 sys_white_list 表
  │    ├─ RPC错误 → 默认**禁行**（Abort）← 安全策略：fail-closed
  │    └─ 不在白名单 → Abort 返回 IpNotInWhiteTable 错误
  │
  ├─5. BackAuthMiddleware() 中间件【B端独有】
  │    5a. 从 metainfo 获取 token（tracer.GetUserToken）
  │        token 为空 → Abort（TokenNotExist）
  │    5b. Redis 查询：rds.Get(token) ← 注意：B端直接用 token 做 key，无前缀！
  │        查不到 → Abort（TokenAnalysisFailed）
  │    5c. 反序列化为 BUserTokenPayload{ID, UserName}
  │        ID == 0 → Abort（TokenAnalysisFailed）
  │    5d. 将 UserID、UserName 写入 metainfo（后续 RPC 可读取）
  │    5e. 调用 rpc.BUserClient().CheckUrlAuth(ctx, c.FullPath())
  │        ├─ ser-buser 内部：
  │        │   从 ctx 获取 userId → 查 sys_user_role 获取角色
  │        │   角色不存在或 status=0 → 返回 false
  │        │   super_admin_flag == "1" 或 "2" → 直接返回 true（跳过菜单检查）
  │        │   普通角色 → LEFT JOIN sys_role_menu + sys_menu
  │        │   WHERE sys_menu.path = 请求URL AND status=1 → COUNT>0 则通过
  │        ├─ RPC错误 → Abort（CheckUrlAuthFailed）
  │        └─ 鉴权失败 → Abort（CheckUrlAuthFailed）
  │
  ├─6. 路由Handler（极薄层）
  │    典型一行代码：rpc.Handler(ctx, c, rpc.BUserClient().QueryUserPage)
  │
  │    Handler[REQ,RESP] 内部流程：
  │    6a. reflect 初始化 REQ（处理指针/非指针类型）
  │    6b. c.BindAndValidate(req) — Hertz 绑定请求参数并校验
  │    6c. rpcMethod(ctx, req) — 发起 Kitex RPC 调用
  │        ├─ 客户端配置（InitRpcClientParams）：
  │        │   ETCD Resolver 服务发现 → 多路复用 → TTHeader 元数据传递
  │        │   超时：1200s（代码实际值，注释写3s ← 已知差异）
  │        ├─ 目标服务处理：
  │        │   InitRpcServerParams 中间件记录访问日志（注意：当前实现日志在 next() 之前打印）
  │        │   → handler.go 入口 → internal/ser/ 业务逻辑层
  │        │   → internal/gorm_gen/repo/ 自动CRUD 或 internal/rep/ 手写扩展
  │        │   → GORM → TiDB（UserPlugin 自动填充 CreateBy/UpdateBy/CreateAt/UpdateAt）
  │        └─ 返回结果或 BizStatusError
  │    6d. ret.HttpOne(ctx, resp, I18nClient(), err) — 统一响应
  │        ├─ 成功 → HttpOk → {traceId, code:0, data:resp}
  │        └─ 失败 → HttpErr：
  │            提取 BizStatusError 的 code
  │            → 调用 I18nClient().LangTransOne(ctx, code) 翻译错误消息
  │            → 翻译失败兜底：使用原始 BizMessage
  │            → {traceId, code:业务错误码, msg:多语言消息}
  │
  └─ HTTP 200 响应（注：所有业务异常也返回200，靠 code 字段区分）
```

**C端请求与B端的差异**：
| 环节 | B端（gate-back） | C端（gate-font） |
|------|-----------------|-----------------|
| 路径前缀 | /admin/open、/admin/api | /open、/api |
| IP安全 | WhiteCheck — **白名单准入，fail-closed** | BlackCheck — **黑名单拦截，fail-open** |
| Token认证 | BackAuthMiddleware | FontAuthMiddleware |
| Redis Token Key | `{token}`（无前缀） | `user:token:{token}`（有前缀） |
| Token Payload | BUserTokenPayload{ID, UserName} | CUserTokenPayload{ID, UID, LastLoginIp, LastLoginTime, Device, DeviceType, NickName} |
| URL鉴权 | CheckUrlAuth（RBAC菜单权限） | 无URL鉴权 |
| 登录设置 | 返回 token + roleId + roleName | 设置 session_token Cookie（SameSite=None, Secure, HttpOnly） |

### 十四、B端登录完整流程（代码实证：ser-buser/internal/ser/user_service.go）

```
1. IP白名单校验 → rpc.IpClient().OnWhiteList(ctx)
   ├─ 校验失败/RPC错误 → 默认禁行
   └─ 不在白名单 → 返回 IpNotInWhiteTable

2. 查询用户 → repo.QueryUserByUserName(ctx, req.UserName)
   └─ 用户不存在 → 返回 UserNotFound

3. 状态检查 → userNormal.Status == enum.Fail(0)
   └─ 已禁用 → 返回 ThisAccountHasBeenDisabled

4. 密码验证 → utils.CheckPassword(req.Password, user.Password, user.Salt)
   └─ 密码错误 → 返回 PasswordError
   说明：salt + hash + bcrypt 机制

5. Google 2FA（可选）
   ├─ req.GoogleCode 不为空且不为"0"时触发
   ├─ 用户已绑定 GoogleAuthKey → VerifyAuthenticator(secret, code) — TOTP 校验
   │   └─ 校验失败 → 返回 WrongGoogleCode
   └─ 用户未绑定 → 返回 NotBindGoogleCode

6. 更新 Google 绑定状态
   └─ 有 GoogleAuthKey 但 BindGoogle=0 → 更新为已绑定

7. 生成Token → utils.GenToken()

8. 缓存Token到Redis
   ├─ Key: {token}（无前缀，即 token 本身）
   ├─ Value: JSON{ "id": userId, "uid": userName }
   │   注意：BUserTokenPayload.UserName 的 JSON tag 是 "uid"
   └─ TTL: 30天

9. 设置上下文 → metainfo 写入 UserID、Token

10. 审计日志 → rpc.BLogClient().AddActLog（记录登录行为）

11. 查询角色 → repo.GetRoleByUserId(ctx, userId)

12. 返回 → {Token, Id, BindGoogle, RoleId, RoleName}
```

### 十五、RBAC 权限体系详解（代码实证：ser-buser）

**表关系**：
```
sys_user ──1:N──> sys_user_role ──N:1──> sys_role ──1:N──> sys_role_menu ──N:1──> sys_menu
用户              用户角色关联            角色               角色菜单关联            菜单
```

**super_admin_flag 三级权限**：
| 值 | 含义 | CheckUrlAuth行为 | 菜单可见性 | 是否可删除 |
|----|------|-----------------|-----------|-----------|
| "0" | 普通角色 | 需要查 sys_role_menu + sys_menu 匹配URL | 仅关联的启用菜单 | 可删除 |
| "1" | 系统管理 | **直接跳过**，返回 true | 所有启用菜单(status=1) | 不可删除 |
| "2" | 超级管理 | **直接跳过**，返回 true | 所有菜单（含禁用） | 不可删除，且**隐藏**：分页查询 WHERE super_admin_flag IN (0,1) |

**菜单级联逻辑**：
- **禁用级联向下**：禁用菜单A → 递归查找所有子菜单 → 全部禁用（getAllChildIds 递归）
- **启用级联向上**：启用菜单C → 递归查找所有父菜单 → 全部启用（getAllParentIds 递归）
- 菜单类型：M=目录、C=菜单、F=按钮

### 二十、已发现的问题和差异（代码实证）

**确认的代码问题**：

1. **RPC服务端日志时序问题**（common/rpc/init.go:49-64）
   - 当前实现：先记录 `time.Since(start)` 和 `err`，然后才调用 `next(ctx, req, resp)`
   - 后果：cost 永远 ≈ 0，err 永远 = nil
   - 应在 `next()` 之后记录

2. **RPC客户端超时注释与实际不符**（common/rpc/init.go:81）
   - 注释写"超时3秒"，实际 `1200 * time.Second` = 20分钟
   - 可能是调试遗留

3. **gate-back UpdateUser 使用了错误的Handler**（gate-back handler）
   - 使用了 `HandlerWithCSV` 而非 `Handler`
   - 更新用户操作应返回JSON，不是CSV

4. **ser-ip DeleteBlackList 审计日志标签错误**（ser-ip handler）
   - 删除黑名单时，审计日志写的是 "whitelist" 而非 "blacklist"

5. **sys_user_role 表字段注释互换**（ser-buser model）
   - user_id 字段注释写"角色ID"，role_id 字段注释写"菜单ID"
   - 实际含义正确，仅注释错误

**已知的设计差异/值得关注点**：

6. **B端/C端 Token Key 格式不一致**
   - B端：直接用 token 做 Redis Key（无前缀）
   - C端：`user:token:` + token（有前缀）
   - 不影响功能，但风格不统一

7. **BUserTokenPayload.UserName 的 JSON tag 是 "uid"**（session/login.go:14）
   - `UserName string \`json:"uid"\`` — JSON序列化后字段名为"uid"而非"userName"
   - 与 CUserTokenPayload 的字段命名风格不同

8. **ser-bcom 初始化了 Redis 但未使用**
   - 枚举查询和开关配置均无缓存层
   - 开关直接走 ETCD DirectGet

9. **服务编号标注不一致**
   - item 在 namesp.go 注释写 009 实际应为 010
   - content_center 编号 013 在 namesp 与 rpc_client 中位置不同

10. **菜单级联递归无深度限制**
    - getAllChildIds/getAllParentIds 理论上无限递归
    - 依赖数据正确性（无环），如果存在循环父子关系则会栈溢出
