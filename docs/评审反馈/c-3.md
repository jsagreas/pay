## 二、必须澄清的问题

> 以下每条不澄清，后端将无法确定实现方案。

### Q1：充值回调到底走谁？

**现状**：文档中充值流程图（2-3充值记录-1充值流程）显示回调到达服务端后直接入账。但通道配置和支付方式都在财务模块管理。

**问题**：三方通道的回调地址配在财务侧还是钱包侧？

```
方案A：回调 → 财务模块 → 验签+更新订单 → 调钱包RPC CreditWallet 入账
方案B：回调 → 钱包模块 → 验签+更新订单+入账
```

**影响**：决定了钱包模块是否需要处理通道验签、通道配置的读取，以及充值订单的归属问题（充值订单表建在哪个服务）。

**建议方案A**（回调到财务模块）：
- 通道密钥/签名配置在财务侧，验签也应该在财务侧
- 钱包只暴露 CreditWallet RPC，职责更单一
- 但需要确认：**充值订单表建在钱包还是财务？**

> 依据：接口评估c-3.md 2.2节"关键架构决策点"、需求理解c-3.md 4.3.2节充值流程

### Q3：冻结-提交的职责边界

**现状**：提现流程中，文档说"用户提交提现 → 冻结余额 → 进入风控审核"。但提现订单的"风控审核"和"财务审核"都在财务模块。

**问题**：提现创建的执行链路到底是什么？

```
链路A：C端提交 → 钱包冻结余额 → 钱包创建提现订单(待风控) → 钱包调财务RPC提交审核
链路B：C端提交 → 钱包冻结余额 → 财务创建提现订单(待风控)
链路C：C端提交 → 财务创建提现订单 → 财务调钱包RPC冻结余额
```

**关键区别**：提现订单表建在钱包还是财务？是否需要两边都有订单记录？

**建议链路A**：
- 钱包负责冻结余额+本地记录（状态对账用）
- 同步调用财务 RPC 提交审核
- 财务侧维护完整审核流程
- 后续审核通过/驳回由财务调钱包 RPC（UnfreezeBalance / ConfirmFrozenDeduction）

> 依据：需求理解c-3.md 4.4.2节提现流程、架构设计c-3.md 6.4节

### Q11：汇率公式中"实时汇率"和"平台汇率"的对象

**现状**：文档的汇率公式有一处表述不一致。

```
文档原文（需求理解 2.4.2 汇率规则）：
  入款汇率 = 实时汇率 × (1 + 汇率浮动%)
  出款汇率 = 实时汇率 × (1 - 汇率浮动%)

但后台列表字段说明（需求理解 2.5.2 第14字段）：
  出款汇率 = 实时汇率 × (1 - 汇率浮动%)

而汇率规则说明图（4-2）中又提到：
  偏差值 = (实时汇率 - 平台汇率) / 实时汇率 × 100%
  偏差值 ≥ 阈值时"更新平台汇率"
```

**问题**：入款/出款汇率的计算基数到底是"实时汇率"还是"平台汇率"？

```
理解A：入款/出款汇率 = 实时汇率 × (1 ± 浮动%)
  → 平台汇率是什么？等于实时汇率？那偏差值永远为0

理解B：平台汇率 = 实时汇率（当偏差≥阈值时同步）
        入款汇率 = 平台汇率 × (1 + 浮动%)
        出款汇率 = 平台汇率 × (1 - 浮动%)
  → 更合理。平台汇率不是每次都更新，只有偏差≥阈值才更新
  → 入款/出款是基于平台汇率而非实时汇率计算
```

**需要产品确认是理解B**。否则"平台汇率"和"偏差阈值"的设计就没有意义了。

> 依据：需求理解c-3.md 2.4.2节 vs 2.5.2节

**建议**：按"发起操作时的币种"归属。兑换记录按"from_currency"（法币侧）展示，BSB 提现按"BSB"展示。

> 依据：需求理解c-3.md 3.5.2节核心规则

## 三、建议优化的设计

> 以下不是阻塞性问题，但建议产品侧考虑优化。

### S1：建议将"提现币种规则"简化

**现状**：文档规定——法币充→法币提，USDT充→USDT提，混合充→法币提。

**问题**：这需要钱包记录每个用户的充值来源组成，逻辑非常复杂：
- 用户先充 100 VND，再充 50 USDT，现在是"混合"，只能法币提
- 用户法币提走了一部分，剩余的全是 USDT 充值的金额，能切回 USDT 提现吗？
- 中心钱包余额来自充值+投注返奖+活动奖励，来源追溯极其困难

**建议**：
- 方案一（推荐）：BSB 钱包统一允许所有提现方式，不做来源追溯
- 方案二：改为"只要有过 USDT 充值记录，就允许 USDT 提现；只要有过法币充值记录，就允许法币提现"

> 依据：需求理解c-3.md 3.4.2节提现币种规则

---

### S2：建议明确"资金钱包余额"的展示定义

**现状**："资金钱包余额 = 中心钱包 + 奖励钱包"。但主播钱包和代理钱包的余额是否也在总余额中？

**建议**：页面上明确标注"资金余额"包含哪些子钱包。主播/代理钱包如果不算在内，需要在 UI 上有独立入口展示。

> 依据：需求理解c-3.md 2.1.3节核心余额规则

---

### S3：建议重复转账拦截增加时间窗口

**现状**：重复转账判定条件是"同UserID + 同币种 + 同金额"的待支付订单。

**问题**：如果用户一个月前有一笔同金额的待支付订单（超时未支付但未关闭），新充值还会被拦截。

**建议**：增加时间窗口限制，如"24小时内"同UserID+同币种+同金额的待支付订单数。或者增加**订单过期自动关闭**机制（如 30 分钟未支付自动标记失败）。

> 依据：需求理解c-3.md 3.2.5节重复转账拦截规则

---

### S4：建议明确订单过期/超时机制

**现状**：扫码充值有倒计时，银行转账文档未提到超时。USDT 充值无倒计时。

**建议**：统一定义各类型订单的超时时间：
- 银行转账待支付：建议 30 分钟后自动失效
- 扫码支付：使用通道返回的倒计时
- USDT 链上：建议 24 小时后自动失效

**理由**：如果没有过期机制，待支付订单会无限累积，影响重复转账判定和数据统计准确性。

---

### S5：建议明确"订单号"的生成规范

**现状**：文档提到各种订单号格式（C+16位、T+16位、B+16位、A+16位、M+16位），但说是"随机数"。

**问题**："随机数"无法保证全局唯一。

**建议**：后端将使用 **前缀 + 时间戳(毫秒) + 序列号** 的方式生成，而非纯随机数。长度保持 1+16 位，全局唯一，且支持按时间排序。产品侧是否接受？

> 依据：实现思路c-3.md 第八节订单号生成

---

### S6：建议优化出款汇率公式

**现状**：文档第 14 字段描述"出款汇率 = 实时汇率 × (1 - 汇率浮动%)"。

**潜在风险**：如果汇率浮动设置为 100% 或更大（虽然不太可能），出款汇率会变为 0 或负数。

**建议**：后端将对汇率浮动值做上限校验（如不超过 50%），产品侧是否需要在后台编辑时也加前端校验？

---

### S7：建议增加"兑换金额上限"

**现状**：兑换只检查余额上限（"已达当前可兑换上限"），没有单笔上限和单日上限。

**问题**：如果没有上限，理论上用户可以一笔兑换全部余额（可能是巨额），如果汇率有偏差会造成较大损失。

**建议**：考虑增加单笔兑换上限（可配），与充值/提现保持规格一致。

---

### S8：建议明确"游客"的定义和权限边界

**现状**：文档说"游客也可用USDT充值"。

**问题**：
1. "游客"是未登录用户？还是已登录但未完成注册的临时用户？
2. 游客充值后的资金归属——没有 user_id 怎么建钱包余额行？
3. 游客充值后注册/登录，资金怎么关联到正式账户？

**如果"游客"有 user_id（已登录的临时用户）**，则技术上没有问题。如果是完全未登录的匿名用户，需要完全不同的实现方案。

> 依据：需求理解c-3.md 3.1.4节页面交互

---

### S9：建议确认提现"到账时间 T+N"的起算点

**现状**：银行卡提现"T+1个工作日"。

**问题**：
1. T 是提交申请日？审核通过日？出款日？
2. "工作日"是谁的工作日？中国？用户所在国家？
3. 这个 T+N 是展示给用户的参考值还是系统自动判定的到账截止时间？

**建议**：明确为展示值（仅供用户参考），实际到账取决于通道和银行处理速度，系统不做到账时间保证。

---

### S10：建议确认汇率日志中"平台汇率 = 实时汇率"的含义

**现状**：汇率日志字段中写"平台汇率 = 该币种更新时的三方实时汇率"。

**问题**：如果平台汇率 = 实时汇率，那这两列的值永远一样？那为什么要记两列？

**实际含义**：应该是"更新后的新平台汇率 = 触发更新时的实时汇率"，也就是日志记录的是**更新后的快照**。

**建议**：字段描述修改为"更新后的平台汇率"，避免开发理解歧义。

> 依据：需求理解c-3.md 2.6节汇率日志

---

## 四、风险预警

> 以下是产品侧需要知晓的技术限制和潜在问题。

### R1：三方汇率 API 不可控

**风险**：3 家三方汇率 API 可能同时不可用（网络故障、API 限流、服务商倒闭）。

**应对**：
- 3 家全部不可用时，保持当前平台汇率不变（不更新）
- 记录告警日志，需要运营关注
- 产品需接受：极端情况下平台汇率可能滞后于市场

**需要产品确认**："全部不可用时保持不变"是否可接受？还是需要有手动设置汇率的后门？

---

### R2：提现超时场景的用户体验

**风险**：提现提交到财务审核的 RPC 调用可能超时（网络抖动），此时资金已冻结但审核状态不确定。

**应对**：
- 订单状态标记为"处理中"
- 后台异步查询确认
- 最长 30 分钟内自动解决（查到则更新，查不到则解冻回滚）

**需要产品确认**：用户在这 30 分钟内看到的状态是什么？"处理中"？是否需要提示"请稍后查看"？

---

### R3：USDT 充值的确认时间不可控

**风险**：区块链确认时间取决于网络拥堵状况，可能几分钟到几小时不等。

**应对**：
- 页面提示"通常 2-10 分钟"已包含在需求中
- 但如果超过 1 小时仍未确认，是否需要给用户发通知？
- 是否需要"超时未确认"的订单自动处理逻辑？

---

### R4：并发场景下的用户体验

**风险**：充值回调和用户操作（提现/兑换）同时到达时，数据库行锁会导致短暂等待（毫秒级）。

**应对**：正常情况下用户无感知。但在极端高并发时（如活动发奖），个别请求可能响应变慢。

**需要产品知晓**：这不是 Bug，是资金安全机制的正常表现。不会丢失数据，只是个别请求延迟几十毫秒。

---

### R5：汇率更新的"窗口期"

**风险**：汇率定时更新时，正好有用户在操作兑换/充值。

**应对**：我们设计了"订单锁汇"机制——用户下单时锁定当时的汇率，即使之后汇率更新也不影响已创建的订单。

**需要产品知晓**：这意味着同一时刻不同用户的订单可能使用不同的汇率（一个在更新前下单，一个在更新后下单）。这是正常的，不是 Bug。

---

### R6：金额精度的展示层约束

**风险**：后端使用 DECIMAL(24,8) 存储，但不同币种的展示精度不同（法币 2 位、加密 8 位）。

**应对**：后端返回的金额始终是完整精度的 string，由前端根据币种配置的精度进行截断展示。

**需要产品确认**：截断规则是四舍五入还是截断（向下取整）？文档说"四舍五入"。

---

### R7：数据统计的时区问题

**风险**：文档提到"服务器以 UTC 基准时间记录"、"站点时区展示"。时区转换在跨天统计时可能导致数据不一致。

**例子**：UTC+7 的越南用户在 UTC 17:00（当地 00:00）充值，这笔属于今天还是明天的统计？

**应对**：统一按站点时区的 0 点作为日切点，查询时转换为 UTC 查数据库。

**需要产品确认**：站点时区是全局唯一的还是每个国家不同？

---

### R8：初始数据依赖

**风险**：钱包模块依赖币种配置先就绪、财务模块的通道/支付方式配置先就绪。如果这些基础数据没准备好，钱包的大部分功能无法联调测试。

**建议**：
- 开发阶段用 Mock 数据跑通流程
- 但联调前必须确认：币种初始数据、至少一个测试通道、至少一个充值/提现方式配置

---

## 五、跨团队协作需确认的事项

> 以下需要与财务模块团队当面对齐。

### C1：E1~E6 接口参数契约

需要与财务团队达成一致的 RPC 接口参数：

| 接口 | 需确认的关键参数 |
|------|----------------|
| CreditWallet | walletType（哪个子钱包）、orderNo（调用方提供还是钱包生成）、source（来源标识格式） |
| DebitWallet | 余额不足时的行为（返错？还是扣可用余额？）、是否需要传子钱包类型 |
| FreezeBalance | orderNo 格式、冻结失败时的错误码 |
| UnfreezeBalance | 用 orderNo 匹配还是用 freezeId 匹配 |
| ConfirmFrozenDeduction | 同 UnfreezeBalance |
| GetBalance | 返回所有子钱包还是指定子钱包 |

**建议**：先锁定 IDL 定义，双方签字确认后再各自开发。

### C2：充值入账的触发时机

需要与财务团队确认：

```
场景：用户充值 100 VND，选择了额外 50% 奖金活动

财务侧调钱包入账时：
  调一次 CreditWallet？还是调两次？

方案A（一次调用，传两个金额）：
  CreditWallet(amount=100, walletType=CENTER, bonusAmount=50, bonusWalletType=REWARD)

方案B（两次调用）：
  CreditWallet(amount=100, walletType=CENTER, orderNo=C001, source=RECHARGE)
  CreditWallet(amount=50, walletType=REWARD, orderNo=C001, source=RECHARGE_BONUS)
```

方案B更灵活但需要两次 RPC 调用。方案A接口更复杂但一次搞定。

### C3：提现状态回调机制

需要确认财务审核/出款各节点如何通知钱包：

```
方案A（同步RPC回调）：
  财务审核驳回 → 财务调钱包 UnfreezeBalance RPC

方案B（异步消息通知）：
  财务审核驳回 → 发送 Kafka/NATS 消息 → 钱包消费消息后解冻

方案C（钱包主动查询）：
  钱包定时扫描"待审核"订单 → 调财务查询状态 → 自行更新
```

**建议方案A**（同步RPC）：最简单直接，财务完成审核动作后直接调钱包 RPC。

---

### C4：人工加款/减款的钱包操作粒度

需要确认：人工加款可以同时给 4 个子钱包加款。

```
问：这 4 个子钱包的加款是一个 RPC 调用还是 4 个？

如果一个 RPC：
  CreditWallet(userId, currencyCode, amounts=[{CENTER:100}, {REWARD:50}, ...])
  → 接口复杂，但一次事务保证原子性

如果四个 RPC：
  CreditWallet(userId, currencyCode, walletType=CENTER, amount=100)
  CreditWallet(userId, currencyCode, walletType=REWARD, amount=50)
  ...
  → 接口简单，但不在同一事务中（跨 RPC 无事务）
```

**建议**：4 个独立 RPC 调用。理由：
- 每个子钱包的入账是独立操作，有各自的流水记录
- 部分失败（如中心钱包加款成功但奖励钱包加款失败）需要财务侧处理
- 保持 CreditWallet 接口简单

## 六、文档自身存在的矛盾点

### M1：出款汇率公式不一致

**矛盾**：
- 汇率规则说明（4-2）："出款汇率 = 实时汇率 × (1 - 汇率浮动%)"
- 偏差值公式："偏差值 = (实时汇率 - 平台汇率) / 实时汇率 × 100%"

如果出款汇率基于实时汇率计算，那每次实时汇率更新出款汇率就变，"平台汇率"和"阈值"机制就没有意义。

**推测正确含义**：出款汇率 = **平台汇率** × (1 - 浮动%)，需要产品确认。

---

### M2：偏差值公式中的分母

**矛盾**：
- 文档："偏差值 = (实时汇率 - 平台汇率) / **实时汇率** × 100%"
- 编辑弹窗说明："偏差值 = (实时汇率 - 出入款汇率) / **实时汇率** × 100%"

偏差值到底是"实时 vs 平台汇率"还是"实时 vs 出入款汇率"？这两者不同（出入款汇率还乘了浮动比例）。

**推测**：应该是"实时 vs 平台汇率"，因为阈值的目的是判断平台汇率是否需要更新。

---

### M3：充值流程中"奖金"入账时机

**矛盾**：
- 充值流程说"支付成功 → 打款到对应用户上分"
- 奖金选择规则说"选择奖金后，充值金额与奖金将入账充值钱包"

**问题**："充值钱包"是什么？是指中心钱包？奖金不应该入奖励钱包吗（因为有稽核要求）？

**推测**：
- 充值金额 → 中心钱包
- 奖金金额 → 奖励钱包
- "充值钱包"是文档笔误，应为"对应子钱包"

---

### M4：BSB 锚定的方向

**矛盾**：
- 基准币种说明："10 BSB = 1 USDT"，即 1 BSB = 0.1 USDT
- 钱包总体设计："1BSB = 0.1USDT"
- 但兑换展示："1 BSB = ₫2,620"

**问题**：BSB 与法币的汇率是通过"BSB→USDT→法币"两级换算？还是直接配置"BSB→法币"汇率？

**推测**：BSB 锚定 USDT，法币汇率基于 USDT 汇率换算。即 1 BSB = 0.1 USDT = 0.1 × VND/USDT汇率。需确认。

---

### M5：主播钱包/代理钱包的提现流程

**矛盾**：
- 主播钱包"可直接提现"
- 提现流程只描述了从中心钱包的提现

**问题**：主播钱包的提现是走相同的提现流程（风控→财务→出款）？还是有独立的提现通道？主播钱包提现时冻结的是主播钱包的余额还是先转入中心钱包再提现？

---
