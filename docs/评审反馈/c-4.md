## P0 必须澄清（12 个）

### P0-1. 充值/提现订单数据到底归谁

**问题**：充值订单和提现订单的数据，是钱包建表存储，还是财务建表存储？还是两边各存一份？

**为什么要问**：
- 产品需求中，C端有"充值记录"和"提现记录"页面（属于钱包模块），B端也有"充值记录"和"提现记录"管理页面（属于财务模块）
- 两个页面展示的是同一份数据，但数据到底存在哪个服务的数据库中？

**如果不澄清会怎样**：
- 如果两边各存一份：数据一致性是噩梦，订单状态变更需要双写或同步，任何一方漏更新就数据不一致
- 如果只在一方存：另一方的查询需要跨服务 RPC 调用，影响接口设计和性能

**我的建议**：
```
方案A（推荐）：充值/提现订单只在钱包侧建表存储（单一数据源）
  - C端记录页：钱包直接查自己的表
  - B端记录页：财务 RPC 调钱包查订单数据
  - 好处：数据一致性有保证，订单状态只在一个地方维护
  - 代价：财务B端查记录列表需要跨服务调用

方案B：订单在财务侧建表（财务管理通道和审核，订单天然归财务）
  - C端记录页：钱包 RPC 调财务查订单数据
  - B端记录页：财务直接查自己的表
  - 好处：财务管审核流程方便
  - 代价：C端每次查记录都要跨服务

方案C：两边各存一份（最差，不推荐）
  - 双写 + 状态同步 = 复杂度翻倍
```

**需要产品/架构拍板，这个决定影响 4-5 张表的归属和 10+ 个接口的调用链路。**

### P0-2. 三方支付回调谁来接收

**问题**：用户在三方支付页面付完钱后，三方的支付结果回调，是打到财务服务还是钱包服务？

**为什么要问**：
- 需求文档中提到"充值回调"和"提现回调"，但没有明确说回调由哪个服务接收
- 回调涉及验签（三方签名校验）、通道对账，这些逻辑理论上应该在管理通道的服务（财务）侧

**如果不澄清会怎样**：
- 钱包不知道要不要暴露 HTTP 回调端点
- 如果钱包接回调，钱包就需要知道三方通道的签名密钥和协议，职责边界模糊
- 如果财务接回调，钱包只需要提供一个 RPC 接口 `CreditDeposit` 供财务在验签成功后调用

**我的理解**：
```
回调应该由财务侧接收（因为通道和密钥归财务管理），流程应该是：
  三方支付 → 回调 → 财务服务（验签 + 更新通道记录）→ RPC 调钱包 CreditDeposit（上账）

如果是这样，钱包模块不需要实现任何 HTTP 回调接口，只需要提供 RPC 供财务调用。
请确认这个理解是否正确。
```

### P0-3. 充值订单是谁创建的

**问题**：C端用户点击"充值"提交后，充值订单（C 开头的订单）是由钱包服务创建，还是由财务服务创建？

**为什么要问**：
- 需求文档描述了创建充值订单的流程，但没说是哪个服务执行 INSERT
- 创建订单的服务需要：① 执行重复转账拦截 ② 关联奖金活动 ③ 分配支付通道 ④ 对接三方支付获取支付链接

**分歧点**：
- ①② 是钱包侧的业务逻辑（重复转账拦截需要查钱包的历史订单，奖金关联是钱包功能）
- ③④ 是财务侧的业务逻辑（通道分配和三方对接归财务管理）

**我的建议**：
```
拆成两步：
  1. 钱包执行业务校验（重复拦截 + 奖金关联）→ 创建钱包侧的预订单（状态=待分配通道）
  2. 钱包 RPC 调财务"分配通道并创建支付" → 财务返回支付链接/收款信息
  3. 钱包更新订单状态为"待支付"，返回给前端

这样职责清晰：钱包管业务校验和订单记录，财务管通道和支付。
如果不是这个流程，请产品/财务开发明确说明创建订单的完整流程。
```

### P0-4. 钱包和财务之间的 RPC 接口契约需要双方对齐

**问题**：钱包提供给财务调用的 8 个 RPC 接口（上账/冻结/解冻/扣款/加款/减款/补单/查余额），以及钱包调用财务的 2 个接口（查充值方式/查提现方式），双方的接口参数和返回值需要在评审会上达成一致。

**为什么要问**：
- 这 10 个跨服务接口是整个系统最核心的耦合点
- 双方各自理解的参数可能不一样（比如财务认为传 userId 就够了，钱包认为还需要传 walletType 和 idempotencyKey）
- 联调阶段再发现接口不匹配，改起来代价很大

**需要确认的关键参数**：

| 接口 | 需确认的点 |
|------|----------|
| CreditDeposit（充值上账） | 是否需要区分充值金额和奖金金额？奖金进哪个钱包？ |
| FreezeForWithdrawal（冻结） | 冻结是在哪个步骤触发？用户提交提现时？还是风控审核通过后？ |
| ManualCredit/Debit（人工加减款） | 需要传哪些参数？是否需要指定子钱包类型？ |
| 查充值/提现方式 | 财务返回的数据结构是什么？包含金额档位吗？ |

**建议**：评审会上，钱包和财务双方拿出各自的接口设计，当面对齐参数格式和调用时机。

### P0-5. 奖金活动的数据从哪来

**问题**：充值页面有"额外奖金选择"功能，用户可以选择一个活动获得额外奖金。这个活动数据从哪个服务提供？

**为什么要问**：
- 当前工程的 21 个服务中，没有明确的"活动管理"服务
- 需求文档描述了奖金活动的规则（最低充值门槛、奖金比例、流水倍数要求），但没说这些数据在哪配置、哪个服务管理
- 钱包需要调用某个接口获取"当前可用的奖金活动列表"

**可能的情况**：
```
A. 活动数据在财务模块管理（充值配置的一部分）→ 钱包 RPC 调财务获取
B. 有独立的活动/营销服务管理 → 钱包 RPC 调活动服务获取
C. 活动数据临时放在钱包/币种配置中管理（不合理但可能是临时方案）
D. 首版不做奖金活动功能，延后实现
```

**如果不澄清**：这个接口我不知道调谁，也不知道返回什么字段，充值流程中的奖金逻辑就无法实现。

### P0-6. 流水稽核（X倍流水）谁来追踪

**问题**：奖励钱包的钱需要"完成 X 倍流水才能提现/转入中心钱包"。这个"流水达标"的追踪和判定，由谁来做？

**为什么要问**：
- 流水稽核涉及用户每一笔投注/消费的金额累计
- 投注数据在投注服务，消费数据可能分散在多个服务
- 钱包本身不产生投注数据，如果由钱包来追踪流水，钱包就需要监听投注事件

**可能的方案**：
```
A. 投注/消费服务追踪流水进度，达标后通知钱包"解锁奖励" → 钱包执行转账
B. 钱包自己建 wager_requirement 表，监听 Kafka 投注事件累计流水
C. 独立的稽核服务专门负责此事
```

**需要确认**：
1. 首版是否需要实现流水稽核？
2. 如果需要，追踪逻辑归哪个服务？
3. 钱包需要提供什么接口配合？（比如"解锁奖励转入中心"的接口）

**如果不澄清**：奖励钱包的钱什么时候可以提现这个核心规则就没法落地。

### P0-7. USDT 收款地址谁管理

**问题**：USDT 充值需要展示收款地址 + 二维码 + 网络类型（TRC20/ERC20/BEP20）。这个收款地址是哪个服务生成和管理的？

**为什么要问**：
- USDT 收款地址涉及区块链交互（地址生成、链上交易监听、到账确认）
- 这不是钱包模块应该直接处理的事情（钱包管余额，不管链上交互）
- 但需求文档中只说"展示收款地址"，没说地址从哪来

**可能的情况**：
```
A. 有独立的区块链/加密支付服务 → 钱包调该服务获取地址
B. 收款地址通过三方支付通道获取（类似银行充值跳三方页面）→ 归财务管理
C. 平台有固定的 USDT 收款地址（人工管理）→ 写在配置中心
D. 首版 USDT 充值走第三方加密支付平台（如 NOWPayments）→ 跟银行充值一样跳三方
```

**如果是 D**：那 USDT 充值和银行充值的流程是一样的（都跳三方），GetUSDTDepositInfo 这个接口的设计就需要调整。

### P0-8. 提现冻结的触发时机

**问题**：用户提交提现后，冻结余额这个操作，是用户提交的那一刻就冻结，还是等风控审核通过后再冻结？

**为什么要问**：
- 需求文档说"提交订单 → 钱包冻结金额"，暗示提交即冻结
- 但三级审核（风控→财务→出款）可能会拒绝，拒绝后要解冻
- 如果提交即冻结：用户提交后余额立即减少，审核期间钱被锁住（可能几小时甚至几天）
- 如果审核通过后才冻结：审核期间用户可以把钱花掉，审核通过时余额不足怎么办

**我的理解**：
```
应该是提交即冻结（保护资金）：
  1. 用户提交提现 → 立即冻结金额（balance 不变，frozen 增加，available = balance - frozen）
  2. 审核拒绝 → 解冻（frozen 减少）
  3. 出款成功 → 确认扣款（balance 减少 + frozen 减少）

这样能保证审核通过时一定有足够的钱。
请确认是否是这个时机。
```

### P0-9. "钱从哪来往哪去"的具体规则

**问题**：需求文档说"法币充的钱只能法币提，USDT充的钱只能USDT提"。这个规则在技术上如何判断？

**为什么要问**：
- 用户可能先法币充值 1000 VND，再 USDT 充值 100 USDT，余额混在一起了
- 提现时怎么判断这笔提现是"法币充的钱"还是"USDT充的钱"？
- 如果用户法币充了 1000 VND，花掉 500，再 USDT 充了 100 USDT，提现时规则怎么算？

**具体疑问**：
1. 是按"第一笔充值方式"判断还是按"余额来源比例"判断？
2. 如果用户只充了法币，把法币换成 BSB，再从 BSB 提现，算法币提还是 BSB 提？
3. "混合充值→法币提"这个规则，是指只要有任何一笔法币充值记录，就必须法币提？
4. fund_source（资金来源）字段应该怎么维护？是跟着充值订单走还是跟着钱包余额走？

**需要产品给出明确的规则算法，不能靠"理解"来实现，否则上线后用户投诉"为什么我不能 USDT 提现"。**

### P0-11. 禁用币种后已有余额用户怎么处理

**问题**：运营禁用一个币种后，已经持有该币种余额的用户怎么办？

**为什么要问**：
- 需求文档只说"禁用后 C 端不再显示"，但没说已有余额的处理方案
- 实际场景：越南区域下架，运营禁用 VND，但有 1000 个用户还有 VND 余额

**具体疑问**：
1. 禁用后用户能否继续提现（把钱取出去）？
2. 禁用后用户能否看到自己的 VND 余额？
3. 是否需要强制把 VND 兑换成其他币种？
4. 禁用前是否有前置检查（如：还有 N 个用户有余额，是否确认禁用？）

**如果不澄清**：禁用逻辑只是简单改个状态字段，但用户资金处理没有方案，上线后一定出问题。

### P0-12. 多币种余额展示逻辑：用户看到的"总余额"是什么

**问题**：需求文档说"用户看到的余额 = 中心钱包 + 奖励钱包（资金钱包余额）"。但用户有多个币种的钱包，首页展示的"总余额"是哪个？

**为什么要问**：
- 用户同时有 VND 余额 1,000,000 和 BSB 余额 50，首页显示什么？
- 是显示当前选中币种的余额？还是换算成基准币种的总余额？
- 产品原型中的"钱包余额"是指单币种余额还是汇总余额？

**需要确认**：
1. 首页/个人中心展示的是当前选中币种的余额（币种隔离展示）
2. 还是所有币种折算成 USDT 的总余额
3. 如果是方案 1，用户切换币种时余额数字变化是前端切换还是重新请求接口

## P1 强烈建议澄清（9 个）

### P1-1. 场馆钱包和代理钱包首版是否需要预留

**问题**：需求标注"预留，暂不实现"，但从后端角度需要确认：
1. 首版数据库表中是否要预留 wallet_type=4（代理）和 wallet_type=5（场馆）？
2. 首版接口中是否要预留对这两种钱包类型的参数支持？

**建议**：数据库层面预留（枚举值写好但接口中暂不支持），接口层面暂不处理。这样后续加入只需解除校验限制，不需要改表。

## 不是疑问但需要同步的信息（开发视角的声明）

以下不需要产品回答，但需要在评审会上同步给所有参与方知道：

### 声明 1：金额存储方案

```
所有金额字段在数据库中存储为整数（bigint），单位为最小货币单位：
  - 法币（VND/IDR/THB）：最小单位 = 分（×100）
  - 加密货币（USDT/BTC）：最小单位 = 聪（×10^8）
  - 平台币（BSB）：最小单位 = 分（×100）
  - 汇率：×10^8 存储

前端展示时需要按币种的 decimal_digits 进行格式化。
这是金融系统的行业标准，避免浮点精度问题。
所有金额相关的接口参数和返回值都是整数，前端需要配合。
```

### 声明 2：余额变更的安全机制

```
所有余额变更操作采用三层防护：
  1. 数据库行级锁（SELECT FOR UPDATE）：保证同一用户同一钱包同一时间只有一个事务操作
  2. 乐观锁（version 字段）：防止并发更新丢失
  3. 幂等键（idempotency_key）：防止重复入账/扣款

财务调钱包的 RPC 接口，idempotency_key 是必传参数。
如果传了重复的 key，钱包返回之前的操作结果（幂等），不会重复执行。
```

### 声明 3：对账机制

```
每一笔余额变更，wallet_transaction 记录同时写入 balance_before 和 balance_after。
定时对账任务（每日凌晨）会校验：
  当前余额 = 初始余额 + SUM(所有交易流水的变动额)
不一致时自动告警。

这个机制要求：
  - 任何余额变更必须在同一个数据库事务中更新 wallet 表和写入 wallet_transaction
  - 不允许有"改了余额但没写流水"或"写了流水但没改余额"的情况
```

### 声明 4：联调时间预留

```
钱包和财务模块之间有 10 个跨服务 RPC 接口。
建议在排期中预留至少 3-5 个工作日的联调时间。
联调前需要：
  1. 双方 IDL 定义完成并 code review
  2. 双方各自的单元测试通过
  3. 准备好联调环境（TiDB + Redis + ETCD + 两个服务同时运行）
```
