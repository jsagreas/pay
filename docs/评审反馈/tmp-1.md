## 二、第一类：必须拍板的问题

> 这些问题不定下来，代码骨架就定不了。评审会上**必须讨论并形成结论**。

### 2.1 充值流程的调用方向到底怎么走？

**现状**：需求文档描述了充值的业务流程，但没有明确 ser-wallet 和 ser-finance 之间"谁调谁"。

**两种可能的理解**：

```
理解A（wallet主导）：
  用户发起充值 → wallet创建订单 → wallet调finance"帮我选通道、创建支付"
  → finance返回支付URL → wallet给前端
  → 三方回调finance → finance调wallet入账

理解B（finance主导）：
  用户发起充值 → wallet创建订单 → wallet通过MQ通知finance"有新充值"
  → finance选通道、创建支付 → 把支付URL回传给wallet或直接给前端
  → 三方回调finance → finance调wallet入账
```

**为什么必须现在定**：
- 这决定了 wallet 的 IDL 是否需要一个 "RequestPayment" 的 RPC 方法调用 finance
- 这决定了 finance 是否需要暴露"选通道"的 RPC 给 wallet
- 这决定了前端拿到支付URL的链路是"C端→wallet→finance→C端"还是其他路径
- 两种方案的代码结构完全不同，评审后改方向=大规模返工

**我的建议**：
```
倾向理解A（wallet主导），理由：
  1. 充值是用户在钱包页面发起的操作，wallet作为入口方更自然
  2. wallet创建订单+返回支付URL是一个同步流程，用户在等结果
  3. finance负责"通道选择+三方对接"的能力，但由wallet来驱动调用
  4. 回调方向已明确：三方→finance→wallet入账，这不变

请产品和财务确认这个方向。如果选B也行，但需要明确MQ消息的格式和前端获取支付URL的路径。
```

### 2.2 充值/提现方式配置数据存在哪个库？

**现状**：需求文档5.4描述了"按币种配置可用充值方式和提现方式"，包含单笔限额、手续费、状态等。但没有说这些配置数据是 finance 管还是 wallet 管。

**为什么重要**：
- wallet 的 `GetDepositConfig` 和 `GetWithdrawConfig` 接口需要读取这些数据
- 如果在 finance 库 → wallet 每次都要 RPC 调 finance 查配置（多一次网络开销）
- 如果在 wallet 库 → B端管理页面要放在 wallet 端，但需求文档把"支付配置"归在财务管理模块
- 这影响建表位置、接口设计、缓存策略

**我的建议**：
```
方案一（推荐）：配置数据存在 finance 库，wallet 通过 RPC 读取 + 本地缓存
  理由：需求文档明确把"支付配置"放在财务管理模块，职责归属应一致
  实现：wallet 调 finance.GetDepositMethods(currencyCode)，结果缓存5分钟

方案二：配置数据在 wallet 库，finance 的 B端管理页面通过 RPC 写入 wallet
  理由：减少 wallet 对 finance 的运行时依赖
  缺点：B端"支付配置"页面的数据不在自己的库里，管理割裂

请讨论确认。
```

### 2.3 提现创建后怎么通知 finance 开始审核？

**现状**：需求文档描述了提现的双审核流程，但没有说"wallet 冻结完成后，finance 怎么知道有一笔新提现要审核"。

**三种选项**：

| 方案 | 实现方式 | 优点 | 缺点 |
|------|---------|------|------|
| A 同步RPC | wallet冻结后直接RPC调finance.StartReview() | 简单直接 | wallet和finance强耦合，finance挂了wallet也报错 |
| B 异步MQ | wallet冻结后发消息到NATS/Kafka，finance消费 | 解耦，wallet不关心finance状态 | 审核有延迟（通常秒级） |
| C 轮询 | finance定时查wallet的"待审核"订单 | 最简单 | 实时性差，定时任务有时间窗口 |

**我的建议**：
```
倾向方案B（异步MQ），理由：
  1. wallet冻结成功后，它的工作已经完成了，不需要等finance审核结果
  2. MQ的重试机制保证消息不丢失（NATS JetStream有NakWithDelay）
  3. finance可以按自己的节奏消费，不会因为提现量突增导致wallet超时
  4. 工程中NATS JetStream已封装好，现成可用

需要和finance负责人确认这个方案。
```

### 2.4 USDT链上充值的回调谁来接？

**现状**：需求文档4.3.1描述USDT充值需要"区块链监听服务确认到账"，但没说回调打到哪里。

**两种选项**：

```
方案A：回调打到 wallet
  wallet 直接处理链上确认 → 自己入账
  优点：链路短
  缺点：wallet需要处理链上验签、确认数逻辑

方案B：回调打到 finance
  finance 统一处理所有支付回调（包括三方和链上）→ 入账时 RPC 调 wallet
  优点：回调处理逻辑集中，wallet只管"被通知入账"
  缺点：多一跳

推荐方案B：与银行转账/电子钱包的回调路径保持一致，wallet不直接对接链上服务。
```

### 2.5 订单表放在哪个库？

**现状**：充值订单和提现订单的数据，wallet和finance都需要读写。但微服务的原则是"数据库私有，不共享"。

**我的理解和建议**：
```
充值订单（t_deposit_order）→ 放在 db_wallet
  理由：充值订单的创建者是wallet（用户在钱包页面发起），
       wallet负责订单的全生命周期状态维护

提现订单（t_withdraw_order）→ 放在 db_wallet
  理由：提现订单的创建者也是wallet（用户在钱包发起），
       虽然finance负责审核和出款，但它通过RPC读取wallet的订单信息

finance 不直接读 db_wallet：
  finance 需要订单信息时，通过 RPC 调 wallet 的查询接口
  如：wallet.GetWithdrawOrderDetail(orderNo) → 返回订单+审核状态

finance 自己的表（db_finance）：
  通道配置、支付配置、人工修正订单 → 这些归 finance

请确认这个划分。
```

```

  - 已有余额通过"人工减款"手动处理

策略C：分阶段
  - 先软禁（仅允许提现），设定截止时间
  - 截止后硬禁
```

## 四、第三类：后端视角的设计建议

> 这些不是"问题"，而是基于技术实现经验提出的优化建议。

### 4.1 重复转账拦截——建议调整拦截维度

**当前设计**：同一UserID + 同一币种 + 同一金额，状态为待支付/进行中。

**我的补充建议**：
```
建议增加"同一充值方式"维度：
  同一用户 + 同一币种 + 同一金额 + 同一充值方式

理由：
  用户先用银行转账充100，订单待支付中
  然后想换成电子钱包充100
  如果只看金额一样就拦截，用户被误拦了——换个方式充一样的金额完全合理

当然，如果产品有意拦截"同金额不同方式"的场景，也请明确说明。
```

### 4.2 提现账户"不可修改"——需要有应急出口

**当前设计**：提现账户首次绑定后不可修改，这是风控要求。

**我的建议**：
```
完全理解风控需求，但建议保留后台人工修改通道：

场景：用户换了银行卡、注销了电子钱包、原银行账户被冻结
如果没有修改通道，用户的钱永远取不出来——这是比风控更严重的用户权益问题。

建议方案：
  - C端用户不可修改（保持不变）
  - B端运营可通过"重新绑定"操作为用户修改提现账户
  - 操作需要审核（类似人工修正）
  - 每次修改记录日志（原账户→新账户），方便审计

请产品确认是否需要这个后台能力。如果不做，请明确当用户账户信息变更时的处理方案。
```

### 4.3 订单号生成——建议统一规范

**当前设计**：前缀 + 16位随机数（C/T/B/A/M + 16位）。

**我的技术建议**：
```
"16位随机数"的具体生成方式需要统一：

方案A（推荐）：前缀 + 雪花ID
  C1234567890123456（C + 雪花ID的16位）
  优点：全局唯一有保障、有时序性、可追溯生成时间
  与工程中已有的 myid.GenID() 一致

方案B：前缀 + 时间戳 + 随机数
  C20260227143012345（时间+随机5位）
  优点：人肉可读
  缺点：高并发下可能重复，需要额外去重

方案C：前缀 + UUID截断
  不建议——UUID无序，对数据库索引不友好

建议统一用方案A，与工程现有的雪花ID生成器一致。
兑换订单目前没有定义前缀，建议用 E+16位。
```

### 4.4 汇率更新——建议增加手动触发能力

**当前设计**：定时任务自动更新，偏差达阈值才触发。

**我的建议**：
```
建议B端增加"手动刷新汇率"按钮：

场景：
  - 汇率API抽风导致长时间没更新，运营需要手动介入
  - 突发事件导致汇率剧烈波动，运营要紧急更新
  - 定时任务异常停止，运营发现汇率好久没变了

实现很简单——和定时任务调用同一个 Service 方法即可。
在B端币种配置页面加一个按钮就行。

汇率日志表已预留 trigger_type 字段（1自动/2手动），支持区分。
```

### 4.5 充值状态查询——建议增加主动推送能力

**当前设计**：C端轮询 GetDepositStatus 接口查充值是否到账。

**我的建议**：
```
轮询是可以的，但建议同时支持WebSocket推送：

理由：
  用户在充值页面等待到账，如果纯轮询：
  - 轮询间隔太长 → 用户觉得"好慢"
  - 轮询间隔太短 → 服务器压力大

  WebSocket推送的体验更好：充值到账那一刻立刻通知用户

但这不是本期必须做的——轮询完全能用。
列出来是为了让产品知道有这个技术能力（工程中ser-socket已建好），
后续优化体验时可以加。

## 六、第五类：前后端协同确认

### 6.1 金额传输格式

```
所有金额字段在接口中用 string 类型传输（不是 number/float）。

示例：
  请求："amount": "100.50"
  响应："balance": "50000.00"
  响应："usdtAmount": "0.00012345"

前端需要注意：
  - 不能用 JavaScript 的 number 类型做金额运算（有精度问题）
  - 展示时按币种配置的精度和分隔符格式化
  - 输入时发送原始字符串，不做 parseFloat

这是金融系统的行业标准做法，请前端确认是否清楚。
```

### 6.2 币种切换的全局影响

```
产品需求要求"切换币种后，余额、充值方式、记录列表全部跟着切换"。

从接口角度：
  - 每次切换币种，前端需要重新调用：
    GetWalletOverview(currencyCode)
    GetDepositConfig(currencyCode)
    GetWithdrawConfig(currencyCode)
    PageRecord(currencyCode, recordType)

  - 不会有一个"全币种"接口一次返回所有币种的数据
  - 这是性能考虑——每个币种的数据独立，一次全返回数据量大且大部分浪费

请前端确认接口调用频率是否可接受（切换一次4个接口）。
```

### 6.3 提现状态的前端展示映射

```
提现有8种后端状态，但C端用户不需要看到全部：

后端状态           → C端展示       → 色标
1 审核中           → "审核中"       → 橙色
2 风控通过         → "审核中"       → 橙色（用户不需要知道是哪一步）

接口中会返回 status（后端值）和 statusText（展示文本）两个字段，

## 八、第七类：风险预警

> 不是阻碍开发，但各方需要提前知晓，做好准备。

### 8.1 并发安全的"最后一道防线"在数据库

```
钱包系统用了乐观锁+幂等索引+事务来保证资金安全。
但需要知道的是：如果出现极端并发（同一用户短时间内大量操作），
乐观锁重试次数会增加，用户可能偶尔看到"操作繁忙请重试"的提示。

这不是bug，是保护机制在工作。
如果产品/运维看到这类投诉，不需要恐慌，查一下是否同时有多笔操作即可。

需要运维支持：上线后需要监控乐观锁重试率指标，持续高于5%时评估是否调优。
```

### 8.2 三方汇率API的不可控风险

```
汇率更新依赖3家三方API。需要知道的风险：
  - 3家全挂 → 汇率停止更新，使用最后一次有效值
  - API返回异常数据 → 需要做合理性校验（如汇率突变超100%拒绝采纳）
  - API计费/限频 → 需要运营提前采购API服务

建议运营/产品在评审后尽快确认3家API的服务商，我这边需要知道：
  - API地址和认证方式
  - 返回数据格式
  - 限频策略
  - 是否有测试环境
```

### 8.3 USDT链上充值的延迟和不确定性

```
USDT链上充值不像三方支付有即时回调，区块链确认时间：
  TRC20: 通常1-5分钟
  ERC20: 通常3-15分钟（取决于gas费和网络拥堵）
  BEP20: 通常1-3分钟

这意味着用户充值后可能要等比较长时间才到账。
建议产品在C端页面有明确的等待时间提示，避免用户焦虑投诉。
同时需要确认：是否需要"充值超时未到账"的自动关单逻辑？超时时间设为多少？
```

### 8.4 跨模块联调的时间风险

```
wallet 的充值和提现功能依赖 finance 的接口。
如果 finance 的开发进度延迟，wallet 的这两块功能只能先做mock联调，真实联调要等。

建议：
  1. 评审后立刻确认 wallet⇄finance 的接口契约（IDL先定）
  2. 两边各自按接口开发，不等对方
  3. 约定联调时间点，双方同时联调
  4. 如果某一方延迟，另一方先用mock数据跑通自己的链路
```
