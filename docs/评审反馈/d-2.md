## 2. P0 — 必须当场定论（阻塞开发）

### P0-1 充值流程的职责边界到底怎么切？

**现状**：需求文档描述了完整的充值流程（选方式→选金额→提交→三方支付→回调→上账），但没有明确说"哪一步由哪个服务执行"。

**具体问题**：

| 步骤 | 谁做？ | 需要明确 |
|------|--------|---------|
| ① 获取充值方式列表 | 钱包调财务？还是钱包自己存？ | 充值方式配置数据归属 |
| ② 创建充值订单 | 钱包创建？财务创建？ | 订单归属哪个库 |
| ③ 调用三方支付 | 钱包直接调三方？还是经财务中转？ | 三方对接谁做 |
| ④ 三方支付回调 | 回调钱包？回调财务？ | 回调地址注册在谁那里 |
| ⑤ 上账（余额变动） | 钱包执行，但谁触发？ | 触发时机和调用方 |

**为什么 P0**：这直接决定了钱包服务的接口设计——是需要 `CreateRechargeOrder` 接口还是只需要 `CreditWallet` RPC 接口。两种切法完全不同的代码量和接口数量。

**我的判断（需确认）**：
- 方案A：**钱包只管上账** — 财务负责创建订单、对接三方、接收回调，最后调钱包 RPC 上账。钱包最"瘦"，但 C 端充值页面的数据（方式列表、订单详情）需要从财务获取。
- 方案B：**钱包管订单 + 上账** — 钱包创建订单、展示详情，但调三方支付走财务通道。回调由财务接收，再通知钱包上账。钱包"厚"一些，但 C 端页面数据自给自足。

**评审诉求**：产品和财务端开发一起定论，画出明确的泳道图。

### P0-2 提现流程的职责边界到底怎么切？

**现状**：提现涉及 6 个状态（待风控→风控驳回/待财务→财务驳回/待出款→出款失败/出款成功），产品文档只画了状态流转，没说哪个状态在哪个服务。

**具体问题**：

| 步骤 | 谁做？ | 影响点 |
|------|--------|--------|
| ① 提交提现申请 + 冻结余额 | 钱包做（确定） | — |
| ② 风控审核 | 谁的系统？独立风控服务？还是财务模块附带？ | 决定是否需要对接风控 RPC |
| ③ 财务审核 | 财务做（确定） | — |
| ④ 发起出款（调三方代付） | 财务做（确定） | — |
| ⑤ 出款回调 | 回调谁？ | 财务接收，然后通知钱包？ |
| ⑥ 确认扣款 / 解冻 | 钱包做（确定） | 但由谁触发？ |

**特别关注**：
- 提现订单是存在钱包库还是财务库？
- 如果钱包只负责"冻结→确认/解冻"，提现订单的完整生命周期管理归谁？
- C 端的"提现记录"页面数据从哪取？

**评审诉求**：请产品出提现流程的完整泳道图，标注每一步的执行服务。

### P0-3 USDT 到底是"币种"还是"充提方式"？

**现状**：需求文档存在两种表述混用的情况——
- 充值模块说"USDT 是一种充值方式"（与银行、电子钱包并列）
- 提现模块说"游客只能 USDT 提现"，暗示 USDT 是提现的"币种"
- 币种配置模块的表格里列出了 USDT 作为"加密货币"类型

**矛盾点**：
- 如果 USDT 是充值方式 → 用户充 USDT 后，钱到 BSB 钱包（或法币钱包），USDT 只是入金通道
- 如果 USDT 是独立币种 → 用户有 USDT 钱包，USDT 余额独立存在

**这决定了**：
- 是否需要建 `currency_code = 'USDT'` 的钱包行？
- wallet 表的 currency_code 是 4 种（BSB/VND/IDR/THB）还是 5 种（+USDT）？
- USDT 提现是从"USDT 钱包"出金？还是从 BSB 钱包按汇率换算成 USDT 出金？

**我的分析**：从产品原型看，USDT 充值时页面显示的是 "USDT→BSB 汇率换算"，说明 USDT 充值的到账币种是 BSB。但提现时又说"USDT 提现"，逻辑上应该是"BSB→USDT 出金"。如果是这样，USDT 只是入金/出金通道，不是独立钱包。但需要产品明确确认。

**评审诉求**：明确 USDT 的定位——独立币种钱包 or 仅作为充提通道？

---

### P0-4 用户钱包何时创建？

**现状**：需求文档只描述了"用户有多个币种的钱包"，没有说钱包何时创建。

**两种方案**：

| 方案 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| A. 注册时全量创建 | 用户注册成功后，为所有启用币种创建中心+奖励钱包 | 简单直接，查询不用判空 | 多建了一堆空钱包行（5币种×2类型=10行/用户） |
| B. 首次使用时懒创建 | 用户首次操作某币种时再创建该币种钱包 | 节省存储 | 每次查询和操作都要判断钱包是否存在，代码复杂度高 |

**我的倾向**：方案 A（注册时全量创建）。10 行数据的存储成本远低于每次操作都加"不存在则创建"的逻辑复杂度。

**但需要确认**：
1. "注册"是指 C 端注册？还是游客转正式用户？
2. 新增币种后，已有用户的钱包要不要补建？（比如后续上线 PHP 菲律宾比索）
3. 如果方案 A，ser-user 注册成功后需要调 ser-wallet 的 `InitWallet` RPC，这个联动怎么保证？

### P0-5 游客充值到底怎么落地？

**现状**：需求写"游客可用 USDT 充值"。

**问题**：
1. 游客没有注册，没有 user_id。钱包表的 user_id 关联谁？
2. 游客的 USDT 充值订单挂在谁名下？
3. 游客充值后，余额存在哪？转正式用户后如何迁移？
4. 如果游客不注册就走了，那笔充值怎么处理？

**可能的方案**：
- A. 游客充值前强制注册（简化方案，但与"游客可充值"的需求不符）
- B. 游客有临时 ID（如 device_id），充值挂临时 ID，注册后合并
- C. 游客充值只是展示充值地址，实际到账在注册/登录后才发生

**评审诉求**：这个场景到底是真游客还是"未完成 KYC 的注册用户"？如果是真游客（无 user_id），技术实现成本高且风险大。建议确认是否可以改为"注册但未 KYC 的用户可 USDT 充值"。

### P0-6 提现方式的路由规则需要精确定义

**现状**：需求给了以下提现规则：
- 法币充值 → 法币提现
- USDT 充值 → USDT 提现
- 法币 + USDT 混充 → 法币提现
- 法币 → BSB → 法币提现

**问题**：这个规则过于笼统，实际执行时存在大量歧义——

1. **"法币充值→法币提现"是指必须原币种提现吗？** 用户充了 VND，能不能提 THB？
2. **混充场景的细节**：用户先充了 100 VND，又充了 50 USDT（到 BSB），现在提现 80 BSB → 走法币提现？但是 BSB 里的钱来自两个来源，怎么判断？
3. **提现方式选项怎么生成？** 用户进入提现页面时，系统根据什么逻辑展示可选的提现方式列表？是根据"历史充值记录"推算？还是根据"当前钱包币种"直接映射？
4. **跨币种提现**：用户在 VND 钱包页面点提现，只能用 VND 银行卡提现？还是也可以选 USDT 提现？

**我的建议**：
这个规则用"标签"方式追踪资金来源太复杂（需要每笔钱记来源），建议简化为：
- **法币钱包提现 → 该币种对应国家的提现方式（银行/电子钱包）**
- **BSB 钱包提现 → USDT 提现 + 各国法币提现（按汇率换算）**
- 不追踪资金来源，以"当前操作的钱包币种"决定提现方式

**评审诉求**：请产品给出完整的判断逻辑，最好是决策树或决策表。

### P0-7 与财务模块的 RPC 接口契约

**现状**：钱包和财务之间存在双向 RPC 调用，但目前没有任何接口契约文档。

**钱包需要调用财务的接口**（我方需求）：

| 接口 | 用途 | 何时调用 |
|------|------|---------|
| 获取充值方式列表 | C 端充值页面展示 | 用户进入充值页 |
| 获取提现方式列表 | C 端提现页面展示 | 用户进入提现页 |
| 创建支付订单（待定） | 提交充值后调三方 | 取决于 P0-1 定论 |

**财务需要调用钱包的接口**（我方提供）：

| 接口 | 用途 | 何时调用 |
|------|------|---------|
| CreditWallet（上账） | 充值成功后加钱 | 充值回调后 |
| FreezeBalance（冻结） | 提现申请冻结 | 待定（取决于 P0-2） |
| UnfreezeBalance（解冻） | 审核驳回/出款失败 | 驳回/失败时 |
| ConfirmDebit（确认扣款） | 出款成功 | 出款成功回调后 |
| DebitWallet（直接扣款） | 人工减款 | 后台操作 |
| GetBalance（查余额） | 财务后台查看 | 后台查看时 |

**评审诉求**：
1. 双方接口的入参/出参需要在评审后 **一周内** 定稿（IDL 级别）
2. 特别是回调/通知机制——是同步 RPC 还是消息队列？
3. 谁先出 IDL？建议钱包先出"我方提供"的接口 IDL，财务同步出"财务提供"的接口 IDL

### P0-8 提现限额参数由谁维护、存在哪里？

**现状**：需求提到"每日提现次数上限、每日金额上限、单笔最低/最高、大额触发人工审核"，但没说这些参数从哪来。

**问题**：
1. 这些限额是全局统一？还是按币种分别设？还是按用户等级分别设？
2. 参数由谁配？产品运营后台？财务后台？币种配置后台？
3. 参数存在哪个服务？币种配置表？还是独立的限额配置表？
4. "大额"的阈值是多少？谁定？

**我的建议**：
- 如果限额跟币种强相关 → 放 currency_config 表，由币种配置模块管理

**评审诉求**：明确参数归属。这决定了钱包提现时的校验逻辑——是自己查本地配置，还是 RPC 调其他服务。

## 3. P1 — 强烈建议当场明确（影响架构设计）

### P1-1 汇率三方 API 是哪 3 家？异常降级策略是什么？

**文档描述**："实时汇率由 3 家三方 API 取均值"

**需要明确**：
1. 具体是哪 3 家服务商？（影响 API 对接工作量和费用）
2. 1 家宕机 → 取 2 家均值？
3. 2 家宕机 → 用最后一次有效汇率？还是直接停止交易？
4. 3 家全挂 → 系统降级策略是什么？
5. 3 家返回的汇率差异过大（如偏差超 5%）怎么处理？取中位数？报警？

**为什么 P1**：汇率是资金系统的定价基础，降级策略直接影响资金安全。

---

### P1-2 汇率更新频率和触发机制

**文档描述**："偏差值 ≥ 阈值时更新平台汇率"

**需要明确**：
1. 定时拉取频率是多少？每分钟？每 5 分钟？每 30 分钟？
2. 偏差阈值是多少？1%？2%？5%？
3. 阈值是全局统一还是每个币种单独配？
4. 更新平台汇率是自动更新还是需要人工审核确认？
5. 汇率更新后，正在进行中的充值/提现/兑换订单是否受影响？（应该不受影响，下单时已快照）

### P1-3 兑换功能的精确规则

**文档描述**："法币→BSB 单向兑换，有赠送奖励"

**需要明确**：
1. **汇率用哪个？** 入款汇率？出款汇率？还是单独的"兑换汇率"？（入款/出款汇率是给充值/提现用的，兑换应该用哪个？）
2. **赠送比例谁配？** 固定比例（如全场 10%）？还是分档（兑换 100 送 5%，兑换 1000 送 10%）？
3. **赠送配置在哪管理？** 币种配置后台？活动系统？还是独立配置？
4. **有没有兑换限额？** 单次最低/最高？每日上限？
5. **兑换是否收手续费？** 文档没提，默认理解为不收，请确认。
6. **BSB→法币反向兑换是否需要？** 文档只提法币→BSB，但 BSB 提现到法币本质上是反向兑换 + 出金。这个"反向"是在兑换模块做，还是在提现模块做？

### P1-4 奖励钱包的稽核倍数（流水要求）规则

**文档描述**："赠送部分需完成 X 倍流水后可提现"

**问题**：
1. **"流水"怎么定义？** 投注流水？消费流水？充值 + 投注总流水？
2. **计算维度**：是奖金金额 × X 倍 = 需完成的流水？还是充值总额 × X 倍？
3. **流水数据从哪来？** 钱包本身不知道用户投注了多少，需要投注服务提供。那是钱包主动去查？还是投注服务实时推送？
4. **校验时机**：提现时校验？还是实时更新一个"已完成流水"字段？
5. **多笔奖金叠加怎么算？** 第一次充值送 100 BSB（需 5 倍流水），第二次充值送 200 BSB（需 3 倍流水），总流水要求是 100×5 + 200×3 = 1100？还是按最新的倍数统一算？

**为什么 P1**：这个规则直接影响提现接口的校验逻辑和跨服务调用。如果当期不做稽核校验，也需要明确说"一期不做"。

### P1-5 风控审核是独立服务还是财务模块的功能？

**文档描述**：提现流程有"待风控审核→风控驳回/通过"环节

**需要明确**：
1. 风控审核是一个独立的风控服务？还是财务模块内的功能？
2. 风控规则有哪些？（大额、高频、异常 IP、黑名单...）
3. 如果是独立服务，钱包需要对接风控 RPC 吗？还是财务去对接？
4. 风控审核是自动还是人工？有没有自动放行的阈值？

### P1-6 人工加减款 / 充值补单的完整流程

**文档描述**："双人审核，操作人 ≠ 审核人"

**需要明确**：
1. 双人审核逻辑在财务系统完成？钱包只接收最终的"已审核通过"的扣款/加款指令？
2. 还是钱包也要实现审核功能？（这会大大增加钱包的复杂度）
3. 人工加款是直接加到中心钱包？还是奖励钱包？还是可选？
4. 人工减款如果余额不足怎么办？部分扣除？还是直接失败？
5. 补单（充值订单漏上账）的入口在哪？财务后台？

### P1-7 充值订单的超时与取消机制

## 5. P3 — 建议讨论（优化体验或降低风险）

### P3-1 充值/提现/兑换操作的用户通知

产品文档没有提及通知机制：
- 充值成功 → 需要推送通知吗？App 推送？站内消息？

- 兑换成功 → 需要通知吗？
- 大额到账 → 需要特殊通知吗？

**建议**：至少提现驳回需要通知，否则用户不知道为什么钱没到。

### P3-2 退款场景是否需要考虑？

产品文档未提及以下场景：
- 充值后发生退款（如信用卡 chargeback）
- 投注取消后退款
- 活动奖金发放错误需要回收

**建议**：即使一期不做，也需要确认这些场景的处理策略。因为如果需要做，钱包的 `DebitWallet` 接口需要支持"强制扣款（允许扣到负余额）"模式。

### P3-3 对账机制

1. 日终对账的规则？是否需要？

3. 钱包内部的 balance 与 flow 一致性校验，是否需要定时自检？

**建议**：至少需要一个定时任务做 balance = SUM(flow) 的一致性校验，发现不一致及时报警。

### P3-4 操作安全确认

提现是高风险操作，是否需要额外安全验证？
- 支付密码？

产品文档只提了 KYC 校验，没提操作级别的安全验证。

### P3-5 并发提现的处理

用户快速连续点击提现按钮：
- 前端做防抖就够了？
- 后端是否需要分布式锁防止同一用户同一时间提交多笔提现？
- 如果 A 笔提现冻结了余额，B 笔提现还能提吗？（应该可以，只要可用余额足够）

1. 余额是否需要实时刷新？WebSocket 推送？还是手动刷新？
2. 充值成功后余额变动的感知延迟是多少？（回调→上账→缓存更新→前端刷新）
3. 多端同时在线，一端操作后另一端的余额展示如何同步？

## 8. 后端对产品方案的技术可行性意见

### 8.1 可行且无异议的设计

以下产品设计从技术角度看合理，后端无异议：

| 设计 | 技术评价 |

| 钱包按币种隔离 | 合理，标准做法 |
| 中心钱包 + 奖励钱包分离 | 合理，消费优先级可控 |
| int64 最小单位存储金额 | 合理，避免浮点精度问题 |
| 充值/提现/兑换独立状态机 | 合理，职责清晰 |
| 提现冻结三阶段 | 合理，资金安全标准模式 |

| 币种配置后台管理 | 合理，可配置化 |
| BSB 锚定 USDT | 合理，平台币常见做法 |

### 8.2 技术上有顾虑的设计

| 设计 | 顾虑 | 建议 |
|------|------|------|
| 游客可 USDT 充值 | 无 user_id 无法关联钱包，实现复杂且有资金风险 | 改为"注册但未 KYC 用户可 USDT 充值" |
| 提现路由规则追踪资金来源 | 需要记录每笔钱的"来源标签"，数据模型和计算逻辑都极其复杂 | 简化为"按当前钱包币种决定提现方式" |
| 3 家 API 取均值 | 三方 API 可能不稳定，均值可能被异常值拉偏 | 改为"3 家取中位数"，异常值自动排除 |
| 偏差值触发自动更新平台汇率 | 自动更新如果频率过高可能导致用户体验不稳定 | 建议加"最小更新间隔"（如至少 5 分钟才能更新一次） |
| 重复转账"3笔拦截" | 无时间窗口定义可能误伤正常用户 | 建议限定"24 小时内" |

### 8.3 建议产品重新考虑的设计

| 设计 | 问题 | 替代方案 |
|------|------|---------|
| 提现方式按充值来源路由 | 实现复杂度极高（需要追踪每分钱的来源），且混合场景规则模糊 | 按钱包币种直接映射提现方式（法币钱包→法币提现，BSB钱包→USDT或法币提现） |

## 10. 后端建议的方案优化点

### 10.1 关于充值流程的建议

**建议方案**：采用"职责分离"模式——

```
用户 → gate-font → ser-wallet（创建充值订单）
                  → ser-wallet → ser-finance（请求支付通道）
                  → ser-finance → 三方支付
三方回调 → ser-finance（接收回调、验签）
         → ser-finance → ser-wallet（CreditWallet RPC 上账）
```

**理由**：
- 钱包拥有充值订单，C 端页面数据自给自足，不需要跨服务查询
- 三方对接和通道管理归财务，钱包不感知三方细节
- 上账由财务触发、钱包执行，职责清晰
- 钱包的 CreditWallet 接口天然幂等，回调重试安全

### 10.2 关于提现流程的建议

**建议方案**：

```
用户 → gate-font → ser-wallet（创建提现订单 + 冻结余额）
管理员 → gate-back → ser-finance（风控/财务审核）
       → ser-finance → 三方代付
       → ser-finance 收到出款结果
       → ser-finance → ser-wallet（ConfirmDebit 或 UnfreezeBalance RPC）
```

**理由**：
- 提现订单归钱包（因为冻结/解冻/确认扣款都在钱包侧）
- 审核和出款归财务（因为对接三方和资金调度是财务职责）
- C 端提现记录查钱包库（数据自给自足）
- B 端审核列表查钱包库（status 筛选），审核操作调财务

### 10.3 关于汇率机制的建议

1. 三方汇率取**中位数**而非均值（抗异常值）
2. 加入"汇率熔断"机制：如果实时汇率相比上次变化超过 10%，暂停自动更新，通知运营人工确认
3. 平台汇率更新加**最小间隔限制**（如 5 分钟），避免频繁波动影响用户体验
4. 汇率缓存使用 ETCD Watch 机制（工程中已有 `etcd_cache.go`），更新后所有实例秒级生效

### 10.4 关于提现方式路由的建议

**放弃"追踪资金来源"方案**，改为简单映射：

| 当前操作钱包 | 可选提现方式 |
|-------------|-------------|
| VND 钱包 | 越南银行卡 |
| IDR 钱包 | GoPay/OVO/DANA/ShopeePay/银行卡 |
| THB 钱包 | PromptPay/银行卡 |
| BSB 钱包 | USDT + 所有国家法币提现（按出款汇率换算） |

**理由**：
- 追踪资金来源需要为每笔流水打"来源标签"，数据模型复杂度翻倍
- 混合充值场景下的规则极其模糊，需要太多边界判断
- 简单映射方案覆盖 95% 的业务场景，且用户理解成本低
- 如果产品坚持追踪来源，建议作为二期功能
