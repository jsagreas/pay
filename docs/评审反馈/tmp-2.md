## 一、必须当场确认的关键决策（阻塞开发，无法绕过）

> 以下每一项如果不确认，对应的代码就没法写。建议评审会上优先讨论。

### 1.1 金额的传输和存储单位

**现状**：需求文档没有明确定义金额的技术表示方式。

**问题**：系统中所有金额字段（充值金额、提现金额、余额、汇率、手续费）到底用什么单位？

| 方案 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| 元级 Decimal | 存 100.50 表示一百元五角 | 直观，前端不用换算 | 浮点精度问题（虽然 Decimal 可避免） |
| 最小单位整数 | 存 10050 表示一百元五角（分） | 绝对无精度丢失 | 不同币种最小单位不同，VND 没有分 |

**关键矛盾**：我们的币种精度天然不一致 — VND/IDR 没有小数、THB 2位小数、USDT 最多8位小数、BSB 2位小数。如果用"最小单位整数"，不同币种的换算基数不同，极易出错。

**我的建议**：统一用 DECIMAL(20,8) 存储和传输，由数据库层保证精度。业务代码用 shopspring/decimal 库运算，杜绝浮点。显示精度由 currency_config 表中的 decimal_digits 字段控制。

**需要确认**：
1. 产品是否认可这个方案？
2. RPC 接口中金额字段用 string 传输（最安全）还是 i64（最小单位整数）？
3. 财务模块是否同意统一方案？（**必须两边一致，否则联调时金额会对不上**）

---

### 1.2 基准币种：确认是 USDT 还是需要选择？

**现状**：需求文档说"基准币种只能设置一次，设完不可改"，推荐 USDT。

**问题**：
1. 基准币种是系统上线前由运营在后台设置，还是开发时代码写死？
2. 如果是运营设置，上线初期还没设置基准币种时，涉及基准币种的功能（如跨币种统计）怎么处理？是阻断还是跳过？
3. 一旦设错了（比如手误设成了 VND），真的没有任何回滚手段？是否需要一个超管级别的重置能力作为后门？

**我的建议**：在数据库级别做硬约束（base_currency 表只允许存一条记录），但建议产品考虑是否需要一个紧急修正通道（比如需要 2 个以上超管同时确认才能变更）。

---

### 1.3 钱包账户的创建时机

**现状**：需求文档未说明用户的钱包账户何时创建。

**问题**：
- **方案 A — 注册时创建**：用户注册时，为每个启用币种 × 每个钱包类型批量创建账户记录（假设 5 币种 × 5 类型 = 25 条记录）
- **方案 B — 首次操作时懒创建**：用户第一次充值/被加款时，按需创建对应的单条账户记录

| 对比 | 方案 A（注册时） | 方案 B（懒创建） |
|------|---------------|---------------|
| 用户体验 | 注册即可查余额（全0） | 首次操作前查余额返回空，需要特殊处理 |
| 数据量 | 大量空账户记录 | 只有活跃用户有记录 |
| 开发复杂度 | 简单 | 需要处理"账户不存在→自动创建→再操作"的并发安全 |
| 跨模块影响 | 用户注册流程需要调钱包 RPC | 无影响 |

**我的建议**：方案 B（懒创建），但需要产品确认 — 用户在从未充值过的情况下进入钱包页面，余额展示是"0.00"还是不显示该币种？

### 1.4 充值订单的归属与回调链路

**现状**：充值涉及"平台订单"和"通道订单"两层，需求文档对两者归属描述模糊。

**问题**：
1. **平台充值订单表**（C+16位）由谁建、存在哪个服务的数据库中？
2. **充值回调链路**：三方支付通道回调到哪个服务？

```
链路 A：三方 → 财务模块（验签+处理）→ RPC 调钱包（CreditWallet 入账）
链路 B：三方 → 钱包模块（验签+处理+入账）
```

**我的理解**：链路 A 更合理 — 验签需要通道密钥，密钥归财务管理，所以回调应该先到财务。但需要明确确认，因为这直接决定了代码写在哪个服务、表建在哪个库。

**需要确认**：
1. 充值的"平台订单"由钱包建还是财务建？
2. 充值回调是到财务还是到钱包？
3. 充值状态的更新由谁负责？（支付成功 → 谁更新平台订单状态 → 谁触发入账）

---

### 1.5 提现冻结由谁发起

**现状**：需求文档描述"用户提交提现 → 冻结余额 → 进入审核"，但没有明确冻结操作由哪个服务执行。

**两种理解**：
- **理解 A**：用户在C端提交提现请求 → **钱包内部直接冻结** → 创建提现订单 → 通知财务进入审核流程
- **理解 B**：用户在C端提交提现请求 → 钱包创建订单（不冻结）→ 财务模块接到订单后 → **调钱包 RPC FreezeBalance 冻结**

**我的建议**：理解 A。理由：冻结越早越安全 — 如果用户提交后到财务冻结之间有时间差，用户可能在这个窗口期发起消费把余额花掉，导致冻结时余额不足。

**需要确认**：产品和财务方是否认同这个理解？

### 1.6 钱包类型枚举值定义

**现状**：需求定义了 5 种子钱包（中心/奖励/主播/代理/场馆），但没有定义它们在系统中的编号。

**需要确认的枚举值**：

| 钱包类型 | 建议编号 | 当前状态 |
|---------|---------|---------|
| 中心钱包 | 1 | 已实现（核心） |
| 奖励钱包 | 2 | 已实现（核心） |
| 主播钱包 | 3 | 已实现 |
| 代理钱包 | 4 | 预留 |
| 场馆钱包 | 5 | 预留 |

**关键**：这个枚举值会写入 Thrift IDL，作为钱包和财务模块之间的 RPC 合同。两边必须用同一套编号。如果评审后变更，所有已写的代码和已联调的接口全部需要调整。

## 二、需要产品澄清的业务规则（理解偏差风险）

> 以下是我们读需求文档后的理解，可能和产品意图不一致，需要当场对齐。

### 2.1 提现渠道路由规则 — 边界情况

需求说"怎么充的就怎么提"。我们理解的规则矩阵：

| 充值方式 | 允许的提现方式 |
|---------|-------------|
| 只充过法币 | 只能法币提现 |
| 只充过 USDT | 只能 USDT 提现 |
| 法币 + USDT 都充过 | 按法币提现 |
| 法币充值 → 兑换成 BSB | 法币提现（追溯原始充值方式） |

**需要澄清的边界情况**：

1. **用户从未充值过，余额来自人工加款/活动奖金/投注返奖**：此时没有充值记录，那提现渠道路由规则怎么判断？允许全部渠道？还是默认 USDT？还是不允许提现？
2. **用户曾经充过法币但很久之前全部提光了，后来只通过 USDT 充值**：此时"法币充值记录"存在但已完结，路由规则是看"有过充值记录"还是"当前存在未提完的充值金额"？
3. **用户在 BSB 钱包中通过兑换获得了余额，现在想提现 BSB 钱包**：BSB 本身不能直接提现（需要先换回法币？），还是 BSB 钱包根本没有提现入口？
4. **新增币种后，用户在新币种下没有任何充值记录**：新币种的提现路由规则是什么？

### 2.2 重复转账拦截 — 规则细节

需求说"同一用户同一币种同一金额"，1-2 笔警告，3 笔强拦截。

**需要澄清**：
1. **时间窗口**：拦截判断的"同金额待处理订单"是查最近多久的？24小时？还是所有"待支付"状态的订单？
2. **金额精确匹配**：100.00 和 100 是"同一金额"吗？还是需要精确到最后一位小数？
3. **跨充值方式**：用户用银行转账充了 100，再用电子钱包充 100 — 这算重复吗？还是只在同一充值方式内检测？
4. **拦截解除条件**：之前的待处理订单超时失效后，计数是否自动清零？

### 2.3 奖金/优惠活动 — 数据来源和规则

需求说充值时可选奖金活动，但活动数据从哪来不清晰。

**需要澄清**：
1. 奖金活动由**谁配置**？在**哪个模块**的后台配置？是财务模块的支付配置中，还是单独的运营活动模块？
2. 奖金百分比、门槛金额、稽核倍数 — 这三个参数是**绑定在充值方式上**还是**独立的活动实体**？
3. "一笔充值只能参与一个活动" — 这个约束由前端控制还是后端校验？
4. 如果第一期没有活动模块，充值页面的奖金选择组件怎么处理？**后端接口是否需要先预留这个字段但不实现逻辑？**

### 2.4 稽核/流水要求 — 计算和判断规则

需求提到"奖金需完成 N 倍流水才能提现"，但细节不全。

**需要澄清**：
1. **稽核倍数由谁决定**？每笔操作各自的稽核倍数是固定的还是可配置的？配在哪里？
2. **流水计算口径**：什么样的消费算"流水"？投注金额？打赏金额？购买道具？还是所有消费都算？
3. **稽核是按单笔还是全局**？如果用户充值两次都有奖金，稽核流水是各算各的，还是合并？
4. **提现时的稽核校验**：是"未完成稽核的金额不能提现"，还是"有未完成稽核记录就完全不能提现"？
   - 例：中心钱包 1000 + 奖励钱包 500（稽核未完成），提现中心钱包的 1000 是否允许？
5. **稽核完成的判定时机**：实时计算还是定时更新？实时计算性能开销大，定时更新有延迟。

### 2.5 BSB 的特殊性

BSB 在需求中被定义为"平台货币，锚定 USDT（10 BSB = 1 USDT）"。

**需要澄清**：
1. **BSB 的汇率是绝对固定的还是可调整的**？如果 10:1 是写死的，那 currency_config 表中 BSB 的汇率字段怎么处理？不走定时更新？
2. **BSB 钱包能否直接提现**？还是必须先兑换回法币再提现？需求说 BSB 页面没有兑换入口，那 BSB 页面有提现入口吗？
3. **BSB 能否直接充值**？还是只能通过法币兑换获得？
4. **BSB 的 currency_type 值**：法定货币=1、加密货币=2、平台货币=3 — BSB 属于类型 3，后台配置页上 BSB 那行是只读不可编辑对吧？

### 2.6 币种禁用时的存量处理

需求说币种可以启用/禁用，基准币种不可禁用。

**需要澄清**：
1. **禁用一个币种时，该币种下有余额的用户怎么办**？余额冻结？强制兑换？还是只是新用户不可选择该币种，老用户可以继续用？
2. **禁用币种是否影响正在进行的充值/提现订单**？
3. 能不能明确一个规则：**禁用 = 不允许新操作，但已有余额可查看和提现**？

### 2.7 USDT 充值 — 游客场景

需求说游客（未登录）也能看 USDT 充值信息。

**需要澄清**：
1. 游客看到收款地址和 QR 码后，如果真的往这个地址转了 USDT，**到账怎么关联到用户**？游客没有 userId。
2. 是否每个用户有**独立的 USDT 收款地址**？还是全平台共用一个地址，用转账金额/备注来区分用户？
3. 如果是共用地址 + 金额区分：两个用户恰好转了相同金额怎么区分？
4. USDT 链上充值的到账确认是**由我们自己监控区块链**还是**通过三方支付通道**？

### 2.8 提现账户修改限制

需求明确"用户不能自行修改提现账户"。

**需要澄清**：
1. **运营在 B 端能不能修改用户的提现账户**？如果用户的银行卡注销了、账号变了怎么办？
2. 如果运营可以修改，在哪个模块的 B 端操作？需要审核流程吗？
3. 不同币种/不同提现方式的账户是**独立的**还是**共用的**？（例：用户设置了 VND 的银行卡，切换到 THB 后需要重新设置吗？）

### 2.9 充值订单超时时间

需求说"订单过期后 QR 码失效"。

**需要确认**：充值订单从创建到超时的有效期是多少？**5 分钟？10 分钟？30 分钟？** 不同充值方式的超时时间是否不同（如银行转账可能需要更长时间）？这个值是写死还是可配置？

### 2.10 提现手续费规则

需求提到提现有手续费，但细节不全。

**需要确认**：
1. 手续费是**固定金额**还是**按百分比**？还是**两者取较大值**？
2. 手续费从哪里扣？从提现金额中扣（用户收到的少）？还是额外从余额中扣（冻结金额 = 提现金额 + 手续费）？
3. 手续费率由谁配置？在哪配？不同币种/不同提现方式的费率不同吗？

## 三、需求文档存在的缺失/矛盾（文档本身的问题）

> 以下是我们在分析过程中发现的需求文档/原型中的缺失项或自相矛盾的地方。

### 3.1 提现订单状态机未完整定义

需求提到提现有多种状态（待风控审核、待财务审核、待出款、出款成功、出款失败、风控驳回、财务驳回），但没有给出完整的**状态流转图**。

**缺失的内容**：
1. 从每个状态可以转移到哪些状态？（例：出款失败后可以转回"待出款"吗？还是只能人工解冻？）
2. 状态流转由谁触发？（锁定/解锁/强制解锁 这几个操作分别影响什么状态？）
3. 出款失败后的处理路径是什么？需求说"保持冻结等人工处理"，但人工处理有哪些选项？

**建议**：请产品补充一张**完整的提现状态流转图**，标明每个状态之间的转移条件和触发角色。

### 3.2 "锁定/解锁/强制解锁" 的定义模糊

提现记录管理中提到了锁定、解锁、强制解锁操作。

**缺失的内容**：
1. "锁定"是什么意思？锁定提现订单不被其他人操作？锁定用户不能发起新提现？
2. 和"冻结"是什么关系？冻结是余额级别的操作，锁定是订单级别的操作？
3. "强制解锁"和"解锁"的区别是什么？需要什么权限？

### 3.3 人工加款支持多钱包同时操作 — 但原子性未说明

需求说人工加款时可以同时选择多个钱包类型，各自填写金额。

**缺失的内容**：
1. 一次审核通过后，多个钱包的加款是**一个原子操作**（全成功或全失败）还是**逐个独立操作**（中心加成功但奖励加失败）？
2. 如果是逐个独立操作，部分成功时怎么处理？

**技术影响**：如果要求原子操作，我们需要把多个 CreditWallet 放在一个事务中。但因为是 RPC 调用（财务模块调钱包），单个 RPC 调用只能操作一个钱包。要实现原子性，要么新增一个"批量加款"RPC 接口，要么在财务模块侧做补偿。

### 3.4 兑换赠送比例的配置缺失

需求说兑换时有赠送奖金（如 10%），但没有说明：
1. 这个赠送比例在哪里配置？币种配置里？独立的兑换配置？活动系统？
2. 不同币种的赠送比例一样吗？
3. 赠送活动是长期的还是限时的？有没有开关？

### 3.5 消费优先级"返奖按比例拆分"的精度问题

需求说"混合投注时，返奖按投注金额比例拆分"。

**存在的精度问题**：
- 投注 100（中心 33.33 + 奖励 66.67），返奖 200
- 中心返奖 = 200 × 33.33% = 66.66，奖励返奖 = 200 × 66.67% = 133.34
- 66.66 + 133.34 = 200.00（正好）
- 但如果不是这么整：投注 100（中心 33.33 + 奖励 66.67），返奖 99
- 中心 = 99 × 33.33% = 33.00，奖励 = 99 × 66.67% = 66.00
- 33.00 + 66.00 = 99.00（正好？），但 33.0067 + 65.9933 如果四舍五入不对就差一分

**需要确认**：精度差异（分币级别）怎么处理？归到哪个钱包？

### 3.6 预留钱包（代理/场馆）的影响范围

需求提到代理钱包和场馆钱包是"预留"功能。

**需要确认**：
1. 预留意味着当前不开发？那代码中是否需要预留枚举值和数据结构？
2. 代理钱包/场馆钱包和中心/奖励钱包在消费优先级中是什么位置？消费时会从代理/场馆钱包扣款吗？
3. 场馆钱包的"进场馆时转入、离场馆时转出"逻辑是否在当前版本实现？如果不实现，C端 UI 上是否显示这两种钱包？

## 四、跨模块职责边界确认（与财务模块开发方必须对齐）

> 以下是钱包和财务两个模块之间的职责划分，必须在评审会上与财务方当面确认。

### 4.1 充值链路职责划分

```
用户发起充值
    │
    ├── [钱包] 创建平台订单（C+16位）
    ├── [钱包] 记录订单信息（金额/币种/方式/奖金）
    ├── [??? ] 匹配支付通道 → 这一步由谁执行？
    │         选项A: 钱包调财务 RPC 匹配
    │         选项B: 钱包只创建订单，财务异步拉取匹配
    ├── [财务] 创建通道订单
    ├── [财务] 返回支付信息（URL/QR码）
    │
    ├── ... 用户支付 ...
    │
    ├── [财务] 接收三方回调 → 验签
    ├── [财务] 更新通道订单状态
    ├── [财务] 调钱包 RPC CreditWallet → 入账
    └── [??? ] 更新平台订单状态 → 这一步由谁执行？
              选项A: 财务回调钱包时一起更新
              选项B: 钱包入账后自己更新
```

**需要确认的 3 个"???"**：
1. 通道匹配是钱包调财务同步 RPC，还是异步消息？
2. 平台订单状态更新是钱包自己维护，还是财务来维护？
3. 如果充值失败/超时，平台订单状态由谁更新？

### 4.2 提现链路职责划分

```
用户发起提现
    │
    ├── [钱包] 校验（KYC/限额/姓名/路由规则）
    ├── [钱包] 冻结余额
    ├── [钱包] 创建提现订单（T+16位，状态=待风控审核）
    │
    ├── [财务] 风控审核
    │   ├── 通过 → 待财务审核
    │   └── 驳回 → [财务] 调 [钱包] UnfreezeBalance 解冻
    │
    ├── [财务] 财务审核
    │   ├── 通过 → 待出款
    │   └── 驳回 → [财务] 调 [钱包] UnfreezeBalance 解冻
    │
    ├── [财务] 发起通道代付
    │   ├── 成功 → [财务] 调 [钱包] DeductFrozen 扣除冻结
    │   └── 失败 → 保持冻结，人工处理
    │
    └── [??? ] 提现订单状态由谁更新？
              每个审核阶段的状态变更是钱包更新还是财务更新？
```

**需要确认**：
1. 提现订单表在**钱包的数据库**中，但审核操作在**财务的 B 端**执行。财务审核后怎么更新钱包的订单状态？
   - 选项 A：财务每次审核都调钱包 RPC 更新状态
   - 选项 B：财务有自己的提现记录表，两边各维护各的
2. "锁定"功能是锁财务那边的审核操作，还是锁钱包这边的提现订单？

### 4.3 充值/提现方式配置数据的获取方式

C端展示的充值方式列表和提现方式列表数据**由财务模块在后台配置**。

**需要确认**：
1. 钱包的 C 端接口 `GetDepositMethods` / `GetWithdrawMethods` 是**实时 RPC 调财务获取**还是**定期同步到本地缓存**？
2. 如果是实时 RPC：财务模块对这个查询接口的 RT 和 QPS 有保障吗？C端每次进充值页面都要调。
3. 充值/提现方式的数据结构（名称/图标/档位/限额/手续费率）的 IDL 由谁定义？需要两边提前对齐。

### 4.4 人工修正（补单/加款/减款）的完整交互

**需要确认**：
1. 财务模块审核通过后调钱包 RPC 入账/扣款，这个 RPC 调用如果**超时**了（网络抖动，钱包其实已经入账了），财务模块会重试吗？如果重试了，钱包的幂等机制能保证不重复入账吗？
2. 幂等键的 orderNo 由谁生成？财务模块生成还是钱包生成？
3. 加款的 orderNo 格式：一次加款操作涉及多个钱包，每个钱包用同一个 orderNo 还是不同的？

**我的建议**：
- orderNo 由**发起方（财务模块）**生成
- 一次多钱包加款，使用同一个 orderNo + 不同的 walletType 组合做幂等键
- 这样财务模块可以安全重试，钱包不会重复入账

### 4.5 核心 RPC 接口入参出参需要在评审中对齐

以下 6 个接口是双方联调的"合同"，建议在评审中至少达成**入参/出参的共识**：

| 接口 | 关键入参 | 关键出参 | 需要对齐 |
|------|---------|---------|---------|
| CreditWallet | userId, currency, walletType, amount, orderNo, opType | afterBalance | opType 的枚举值 |
| DebitWallet | userId, currency, walletType, amount, orderNo, opType | afterBalance | 余额不足时的错误码 |
| FreezeBalance | userId, currency, walletType, amount, orderNo | frozenAmount | 可用余额不足时的错误码 |
| UnfreezeBalance | userId, currency, walletType, amount, orderNo | afterBalance | orderNo 对应的关联 |
| DeductFrozen | userId, currency, walletType, amount, orderNo | afterBalance | 冻结不足时的错误码 |
| QueryBalance | userId, currency? | list\<walletInfo\> | 返回哪些字段、是否包含冻结 |

## 五、设计合理性探讨与建议（站在技术角度的优化建议）

> 以下不是质疑产品设计，而是站在后端实现角度提出的改进建议。

### 5.1 建议：基准币种设置增加确认保护

**现状**：设置基准币种只有二次确认弹窗。

**建议**：
- 增加"输入基准币种代码再次确认"的交互（类似 GitHub 删除仓库需要输入仓库名）
- 因为这是**全系统唯一且不可逆的操作**，仅靠"确认/取消"弹窗防误操作力度不够

### 5.2 建议：提现账户"不可修改"策略需要有运营后门

**现状**：用户不能自行修改提现账户。

**风险**：
- 用户银行卡注销后永远无法提现
- 用户银行合并（如银行被收购）后账号变更
- KYC 信息变更后姓名不匹配

**建议**：增加 B 端"提现账户修正"功能，需要：
1. 二级审核（两个不同角色的管理员确认）
2. 记录完整的修改日志（谁改的、改之前是什么、改之后是什么）
3. 修改后的姓名仍需与最新 KYC 姓名一致

### 5.3 建议：汇率 API 源应可配置

**现状**：定时从 3 个三方汇率 API 取平均值。

**问题**：
- 具体用哪 3 个 API？需要产品/运营确认选哪些服务商
- 如果某个 API 长期不稳定需要替换，是改代码还是改配置？

**建议**：汇率 API 的地址、权重、超时时间放在 ETCD 配置中心，支持热更新。不建议代码中写死。

### 5.4 建议：币种配置应有"初始数据导入"方案

**问题**：系统上线时 currency_config 表是空的。5个币种（BSB/VND/IDR/THB/USDT）的初始数据谁来录入？需要逐个在 B 端手动添加？还是提供一个初始化脚本？

**建议**：
- 提供初始化 SQL 脚本，预置 5 个币种的基础配置
- B 端"新增币种"功能是否需要？还是只能编辑已有币种？需求文档只提到了"编辑"，没有"新增"

### 5.5 提问：实时汇率刷新倒计时的技术含义

需求原型中币种列表页有"实时汇率刷新倒计时"。

**需要澄清**：
- 这个倒计时是前端定时轮询后端？还是只是 UI 展示元素？
- 刷新频率是多少？和后端汇率定时更新频率一致吗？
- 如果前端需要实时刷新，对后端接口的 QPS 压力如何评估？

## 六、风险预警与排期影响

> 以下是可能影响开发周期的风险项，需要提前让产品和项目管理知晓。

### 6.1 财务模块的 RPC 接口是我方的阻塞依赖

**风险**：充值和提现的 C 端功能都依赖财务模块提供 RPC 接口（获取支付方式、通道匹配、发起代付）。如果财务模块 RPC 延迟交付，我方的充值和提现功能无法联调。

**应对方案**：
- 我方可以先用 Mock 数据开发，但 Mock 和真实联调之间可能有差异
- **建议在评审中确认财务模块的交付节奏，至少给出 RPC 接口 IDL 的定义时间点**
- 我方可以先独立完成的部分（币种配置、钱包核心、兑换、记录查询）不受影响

### 6.2 KYC 模块的接口确认

**风险**：提现需要调 ser-kyc 的接口检查 KYC 状态和获取 KYC 姓名。需要确认：
1. ser-kyc 现有的 RPC 接口是否满足我们的需求？（从 IDL 看有 KycQueryReq/KycQueryResp，返回 UserKycInfo 包含 name 和 status）
2. KYC 的 status 值有哪些？已认证/未认证/审核中/已驳回 — 审核中和已驳回算"未认证"吗？

### 6.3 活动模块的不确定性

**风险**：充值奖金活动数据如果来自独立的活动模块，而该模块可能当前不存在或尚未开发。

**影响**：
- 充值页面的"奖金选择"功能可能第一期无法实现
- 兑换的"赠送比例"配置也有类似问题

**建议**：评审中明确——第一期是否实现奖金/赠送功能？如果实现，活动数据从哪取？

### 6.4 USDT 链上充值的技术深度

**风险**：USDT 链上充值涉及区块链监控，技术复杂度远高于普通银行转账/电子钱包。

**需要确认**：
- USDT 充值是通过**三方支付通道**（封装好了链上监控）还是我们**自建链上监控**？
- 如果是三方通道，哪家？已经接入了吗？
- 如果是自建，这个复杂度可能需要单独评估排期

### 6.5 多币种同时操作的并发复杂度

**风险**：兑换操作需要在一个事务内操作两个币种的余额（法币减、BSB 加）。如果两个币种的钱包记录不在同一行（它们在不同的 wallet_account 行），需要在事务内锁定多行。

**技术方案已有**：固定锁序（按 currency_code 排序后依次加锁，避免死锁）。但需要确认产品上是否存在更多"跨币种原子操作"的场景。
