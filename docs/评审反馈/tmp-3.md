# ser-wallet 需求评审反馈（后端开发视角）

> 准备时间：2026-02-27
> 评审性质：跨部门需求评审大会——预评审版本定稿前最后的沟通窗口
> 我的角色：后端开发，主责钱包+币种配置模块，配合财务模块联调
> 文档目标：整理所有需要在评审会上提出的疑问、意见、建议、风险提示，确保评审通过后按文档执行不留隐患
> 核心态度：评审前多问一个问题 = 开发中少踩一个坑。评审通过后板上钉钉，现在不问以后就没机会了。

## 使用说明

本文档分为五个部分：

| 部分 | 内容 | 用途 |
|------|------|------|
| 一、必须澄清的问题 | 需求文档中明确存在歧义或缺失的点 | 评审会上必须逐一确认，拿到明确答案 |
| 二、存在风险的设计 | 当前方案可以实现但存在隐患的点 | 评审会上提出风险，建议产品重新考虑 |
| 三、跨模块边界问题 | 钱包与财务模块的职责划分模糊地带 | 需要财务模块开发者一起参与确认 |
| 四、建议优化的设计 | 从后端实现角度提出的改进建议 | 非阻塞性，但采纳会让开发更顺畅 |
| 五、确认无歧义的设计 | 已理解且无异议的核心规则 | 评审会上快速过，不浪费时间 |

每个问题标注优先级：
- **🔴 必答** — 不回答这个问题，开发无法启动或会方向性错误
- **🟡 重要** — 不回答不影响启动，但会在联调阶段暴露问题
- **🟢 建议** — 有答案更好，没有也能用合理默认值先推进

## 一、必须澄清的问题（16 个）

### 1.1 钱包体系核心规则

#### Q1 🔴 奖励钱包 → 中心钱包的"稽核流水"具体规则是什么？

**现状：** 需求文档 2.1 写到"奖励钱包需完成稽核流水后可转入中心钱包"，但**全文没有定义什么叫"完成稽核流水"**。

**需要明确的：**
- "稽核流水"是指用户需要消费/投注达到 X 倍金额？X 是谁配置的？配在哪？
- 奖励钱包的"稽核流水倍数"是跟着每一笔奖励记录走，还是全局统一？
- 举例：用户奖励钱包有 200，稽核倍数 3 倍，是指用户需要用这 200 去投注产生 600 流水才能转出？还是需要用中心+奖励的总余额产生 600 流水？
- 转入中心钱包是自动的还是用户手动触发？
- 稽核进度是否需要在 C 端展示？

**影响：** 这个规则直接影响数据库表结构设计（需要记录每笔奖励的稽核要求和进度）、余额展示逻辑（奖励钱包余额中有多少已解锁可转出）、以及消费时的流水计算逻辑。不清楚这个规则，钱包的核心数据模型就定不下来。

#### Q2 🔴 消费扣款"优先中心再奖励"后，混合投注返奖的具体计算规则？

**现状：** 需求文档 2.1 写到"消费时优先扣除中心钱包余额，再扣奖励钱包余额"，"混合投注时需根据投注金额占比分开计返奖"。

**需要明确的：**
- 假设用户中心钱包 800，奖励钱包 200，投注 1000 全扣完。中心占比 80%，奖励占比 20%
- 若返奖 500，是否返奖 400 到中心 + 100 到奖励？
- 还是返奖全部到中心钱包（因为返奖是"投注返奖 → 中心钱包"）？
- 如果是前者，需求文档的"投注返奖 → 中心钱包"资金流向就和"按比例分返"矛盾了
- 如果是后者，那"按比例分开计返奖"到底在哪里体现？是给投注模块看的统计数据还是实际要拆分入账？

**影响：** 扣款的 RPC 接口设计——DebitWallet 需不需要返回"中心实际扣了多少、奖励实际扣了多少"？如果投注模块需要这个比例来计算返奖，那这个返回值是必须的。

#### Q3 🔴 主播钱包"可直接提现"走什么流程？和中心钱包提现一样吗？

**现状：** 需求文档 2.1 写到"主播钱包支持提现和转入中心钱包"，但提现模块（第 7 章）的全部流程描述和原型都是针对"中心钱包提现"设计的，**没有任何一处提到主播钱包的提现入口和流程**。

**需要明确的：**
- 主播钱包提现的入口在哪？是同一个提现页面选择钱包类型？还是在直播模块有独立入口？
- 主播钱包提现是否需要 KYC？
- 主播钱包提现是否走同样的"冻结→风控审核→财务审核→出款"流程？
- 主播钱包提现是否也适用"充什么提什么"的规则？主播收入的"充值来源"怎么判定？
- "转入中心钱包"是否需要完成稽核流水（需求写"转入后需稽核"但代理钱包标注了，主播钱包好像没标）？

**影响：** 这直接决定提现接口是否需要支持"钱包类型"参数、主播提现是单独流程还是复用现有流程。当前版本如果主播钱包提现不做，需要在评审会上明确说"V1 不含主播钱包提现"。

#### Q4 🟡 场馆钱包和代理钱包的"预留"到什么程度？

**现状：** 需求标注"预留"，但在余额规则、消费规则、资金流向中都出现了这两种钱包。

**需要明确的：**
- V1 版本数据库要不要建这两种钱包的表结构？
- V1 的"资金钱包余额 = 中心 + 奖励"这个公式是否不变？（即场馆和代理不参与余额计算）
- 消费扣款顺序是否只涉及中心和奖励？场馆和代理即使有余额也不参与消费扣款？
- 预留到什么时候启用？有没有大致的时间线？（这决定我现在的代码是直接硬编码"中心+奖励"还是做成可配置）

**影响：** 表结构和枚举设计。如果确定 V1 不做，WalletType 枚举可以只定义中心(1)和奖励(2)两种，主播(3)是否也不做需要一并确认。

### 1.2 充值模块

#### Q5 🔴 充值订单到底归谁创建和管理？

**现状：** 充值流程涉及两个服务（钱包和财务），但需求文档没有明确说"充值订单"这个实体归谁所有。

目前可以有两种理解：
- **理解A**：钱包创建本地充值订单 → 调财务创建通道订单 → 两边各存一份
- **理解B**：钱包不存充值订单，只做入口 → 全部丢给财务 → 财务管理充值全流程 → 成功后回调钱包上账

**需要明确的：**
- 充值记录页（C 端的"充值"Tab）数据从谁的表里查？
- 充值订单状态的权威来源是谁？钱包还是财务？
- 如果钱包存一份、财务存一份，两边数据不一致怎么办？
- B 端"充值记录"管理页面的数据源是哪个服务？

**我的建议：** 钱包不存充值订单，充值订单完全由财务管理。钱包只负责：① 提供充值入口（查方式、查档位），② 接收充值成功的回调上账，③ 记录钱包流水。C 端充值记录查询转发给财务接口。

**影响：** 表结构设计（钱包是否需要 deposit_order 表）、接口设计（C 端充值记录接口是自己查还是转发）、数据一致性方案。

### 1.3 提现模块

#### Q9 🔴 提现规则"充什么提什么"的具体判定逻辑

**现状：** 需求 7.1 第 2 点给出四条规则，但没说"充值来源"怎么追踪。

**需要明确的：**
- "法币充值"是指用户**曾经**充过法币，还是**当前余额中有法币来源的资金**？
- 场景：用户充了 1000 VND，全部兑换成 BSB，VND 余额为 0。此时用户在 BSB 钱包申请提现，规则判定为"法币充值→法币提现"——但用户 VND 已经没钱了，怎么提法币？
- 用户充了 USDT，又充了法币。后来 USDT 部分全花完了（投注输了），法币部分还在。此时提现规则是"混合→法币提现"还是"法币→法币提现"（因为 USDT 部分已经没了）？
- 规则判定的维度是整个账户级别还是单一币种级别？
- 是否需要记录每一笔余额的"资金来源标签"？（这是一个重要的数据模型决策）

**影响：** 这个问题直接影响数据模型设计——是否需要对用户余额做"来源标记"。如果需要，复杂度会显著增加（每笔资金都要标记来源，消费时要按来源扣减）。如果只是简单判断"历史上是否充过法币/USDT"，则只需要一个标志位。

#### Q15 🟡 入款汇率和出款汇率的计算公式有两种写法

**现状：** 需求文档中有两处写了入款/出款汇率的公式，但写法不一致：

- 需求 4.2 写：`入款汇率 = 实时汇率 × (1 + 浮动%)`、`出款汇率 = 实时汇率 × (1 - 浮动%)`
- 后台页面字段说明 3.3 写：`入款汇率 = 平台汇率 + 平台汇率×浮动%`、`出款汇率 = 平台汇率 - 平台汇率×浮动%`

**差异：** 一个用"实时汇率"作为基数，一个用"平台汇率"作为基数。两者不同——当偏差值未超过阈值时，平台汇率保持不变，但实时汇率在变。

**需要明确的：**
- 入款/出款汇率到底基于"实时汇率"还是"平台汇率"计算？
- 如果基于实时汇率，入款/出款汇率会随每次刷新波动
- 如果基于平台汇率，入款/出款汇率只在平台汇率更新时才变

**影响：** 汇率计算逻辑的正确性，写错了就是每一笔充值/提现金额都算错。

## 二、存在风险的设计（7 个）

### R1 🔴 提现"充什么提什么"的规则可能导致用户无法提现

**风险场景：**
```
用户充了 1000 VND → 全部兑换成 BSB → VND 余额为 0 → BSB 余额 100
用户想提现 BSB
→ 规则判定："法币充值→法币提现"
→ 但用户现在 VND 余额为 0
→ 用户无法提现任何钱
```

**问题：** 用户合法操作（充值→兑换→提现）的路径被规则堵死了。

**建议：** 评审时讨论是否需要补充规则——"兑换后的 BSB 按原始充值来源判定提现方式"或"BSB 钱包的提现来源追溯到兑换前的币种"。或者允许 BSB 钱包用户自主选择提现币种，不受"充什么提什么"约束。

---

### R2 🟡 人工减款"从最高金额钱包依次扣减"可能导致扣错钱包

**风险场景：**
```
用户中心钱包 800，奖励钱包 200，主播钱包 500
财务要减款 600
→ 按"最高金额钱包优先"：先扣中心 600（中心剩 200）
→ 但如果这次减款是"异常追回主播收入"，应该从主播钱包扣才对
```

**问题：** "从最高金额钱包开始扣"是一个无差别的通用规则，不区分减款原因。

**建议：** 人工减款是否应该允许指定目标钱包类型？（类似人工加款可以指定加到哪个钱包）。如果一定要"自动从最高扣"，是否需要在审核环节展示"预计从哪些钱包扣多少"供审核人确认？

---

### R3 🟡 同一用户的提现账户在不同币种下是否共享？

**风险场景：**
```
用户在 VND 下绑定了越南银行卡 A
用户切换到 IDR，进入提现页面
→ 是否展示之前在 VND 下绑定的银行卡 A？
→ 但银行卡 A 是越南银行，IDR 是印尼盾，明显不匹配
```

**需求文档没说：** 提现账户是按币种隔离的还是全局共享的？

**合理推断：** 应该按"提现方式+币种"隔离。越南银行的账户只在 VND 下出现，印尼银行的只在 IDR 下出现，USDT 地址在所有支持 USDT 的币种下共享。

**建议：** 评审时确认提现账户的隔离维度。

---

### R4 🟡 汇率刷新期间的并发问题

**风险场景：**
```
汇率定时刷新 → 正在更新 VND 的平台汇率（旧值 15800，新值 16100）
同一时刻用户A 正在创建充值订单（读到了旧汇率 15800）
汇率更新完成
充值成功后上账 → 用的是哪个汇率？提交时的 15800 还是上账时的 16100？
```

**问题：** 需求没有明确汇率的"快照时机"。

**建议：** 评审时确认：充值/提现/兑换在创建订单时是否锁定当时的汇率快照？如果是，需要在订单表中记录快照汇率。这关系到用户看到的金额和最终结算金额是否一致——不一致会引发大量客诉。

---

### R5 🟡 币种禁用后的在途订单处理

**风险场景：**
```
用户创建了 VND 充值订单，状态"待支付"
B端管理员此时禁用了 VND
用户在三方页面完成了支付
三方回调到来 → VND 已禁用 → 是否上账？
```

**问题：** 禁用币种时，是否需要检查有没有"在途订单"？如果有在途订单，是禁止禁用还是允许禁用但在途订单继续完成？

**类似场景：** 提现在途中禁用币种、兑换进行中禁用币种。

**建议：** 评审时明确禁用币种的"生效时机"和"在途处理策略"。

---

### R6 🟢 USDT 充值的"平台承载页"安全风险

**观察：** 需求 5.2 写到 USDT 充值是"平台承载页"，展示收款地址和二维码。

**风险：** 收款地址的管理和展示涉及严重安全问题——如果收款地址被篡改，所有 USDT 充值都会打到攻击者地址。

**需要明确的：**
- 收款地址从哪里获取和管理？是我们自己钱包系统生成还是三方支付通道提供？
- 如果是三方提供，是否每笔充值一个新地址还是全局共用？
- 收款地址的安全存储和展示方案？

**影响：** 这可能需要和安全团队或区块链团队对齐。

---

### R7 🟢 充值档位和充值方式完全来自财务模块，钱包无法降级

**观察：** C 端充值页面（GetDepositMethods）的数据100%来自财务模块的支付配置接口。

**风险：** 如果财务模块接口不可用（宕机/超时），钱包的充值页面会完全空白——没有充值方式，没有档位，用户无法充值。

**建议：** 评审时讨论是否需要钱包本地缓存一份充值方式列表作为降级方案？或者至少展示一个"当前充值通道维护中"的友好提示？

## 三、跨模块边界问题（6 个）

> 以下问题需要**财务模块开发者**和**钱包模块开发者**在评审会上一起确认。

### B1 🔴 钱包和财务的接口契约什么时候定？

**现状：** 钱包依赖财务的 3 个接口（获取充值方式、获取提现方式、创建通道订单），财务依赖钱包的 10 个 RPC 接口（上账、扣款、冻结等）。**但财务模块还没有 IDL**。

**需要确认的：**
- 财务模块预计什么时候出 IDL？
- 在财务 IDL 出来之前，钱包能否先 Mock 财务接口推进开发？
- 双方是否约定一个时间点完成接口对齐？（建议在评审通过后 1 周内）

**影响：** 如果不及时对齐，两边各做各的，联调时会出大问题。

---

### B2 🔴 充值/提现订单的数据归属

**核心问题：** C 端的充值记录和提现记录，用户查询时调谁的接口？

| 方案 | 钱包存不存充值/提现订单 | C端记录从哪查 | 优劣 |
|------|----------------------|-------------|------|
| A. 财务全权管理 | 不存，只存钱包流水 | 转发到财务接口 | 简单不冗余，但钱包依赖财务 |
| B. 两边各存一份 | 存一份简化版订单 | 钱包自己查 | 独立性好，但要保证两边一致 |
| C. 钱包管订单、财务管通道 | 钱包存完整订单 | 钱包自己查 | 钱包自洽，但和财务 B 端管理页冲突 |

**建议：** 评审时两边开发者一起定方案。我倾向方案 A，但需要财务保证提供 C 端查询接口。

---

### B3 🟡 冻结操作谁来发起？

**现状：** 提现流程中，用户点击"确认提现"后需要冻结余额。但需求没说冻结是钱包自己做还是财务回调来做。

两种理解：
- **理解A**：钱包自己冻结 → 创建提现申请给财务 → 财务审核 → 成功/驳回时回调钱包确认/解冻
- **理解B**：钱包创建提现申请给财务 → 财务回调钱包冻结 → 财务审核 → 成功/驳回时回调确认/解冻

**我的建议：** 应该是理解 A——钱包自己冻结后再通知财务。因为冻结必须在用户提交的瞬间完成（防止冻结前余额被其他操作消耗），如果等财务回调来冻结会有时间窗口。

**需要确认：** 财务模块开发者是否同意这个流程设计？

---

### B4 🟡 充值成功回调上账时，钱包需要收到哪些字段？

**需要明确的：** 财务回调 CreditWallet 时，需要传哪些信息？

| 字段 | 是否必须 | 说明 |
|------|---------|------|
| userID | 必须 | 哪个用户 |
| currencyCode | 必须 | 哪个币种 |
| amount | 必须 | 上账金额（实际到账，已扣手续费） |
| orderNo | 必须 | 充值订单号（幂等键） |
| walletType | 必须 | 入哪个钱包（一般是中心钱包） |
| bonusAmount | 看情况 | 奖金金额（如果有关联活动） |
| bonusActivityID | 看情况 | 活动ID |
| flowType | 必须 | 流水类型（充值=具体类型编码） |
| remark | 可选 | 备注 |

**关键问题：** 奖金金额是钱包在创建充值订单时已经记录了，还是需要财务回调时传回来？如果钱包不存充值订单（方案A），那奖金信息就必须由财务回调时带回。

---

### B5 🟡 人工修正的审核通过后，谁调谁？

**现状：** 人工加款/减款/补单的审核在财务 B 端操作。审核通过后需要变更钱包余额。

**需要确认的：**
- 审核通过后，是财务直接 RPC 调钱包的 ManualCredit/ManualDebit/SupplementCredit？
- 还是财务更新自己的订单状态后，由定时任务异步调钱包？
- 如果是同步 RPC 调用，调用失败怎么处理？（审核通过了但钱没加上）
- 是否需要对账机制来兜底？

---

### B6 🟢 财务模块 B 端管理页面查用户钱包余额，调谁的接口？

**场景：** 财务 B 端在"提现审核"页面需要查看用户的钱包余额（审核参考）。在"人工加款"页面需要查用户当前各钱包余额。

**选项：**
- A. 财务直接 RPC 调 ser-wallet 的 GetBalance
- B. 财务自己存一份用户余额缓存

**建议：** A 方案更合理——余额的权威来源是钱包。但需要确认 GetBalance 这个 RPC 接口是否需要返回"按钱包类型展开的余额明细"（财务人工加款需要知道每个钱包的余额）。

## 四、建议优化的设计（5 个）

### S1 🟢 建议"基准币种设置"增加保护机制

**现状：** 基准币种"只可设置 1 次，提交时需二级确认"。

**建议增加：**
- 设置前必须确认所有币种数据已导入完毕
- 设置时弹窗展示"设置为 USDT 后将无法更改，所有统计和对账将以 USDT 为基准"
- 考虑是否增加"超级管理员+OTP"双重确认（防止误操作）

---

### S2 🟢 建议充值/提现的"状态"枚举统一定义

**现状：** 需求中提到的状态：
- 充值状态：待支付、支付成功、支付失败（需求 2.3）
- 提现状态：待风控审核、风控驳回、待财务审核、财务驳回、待出款、出款失败、出款成功（需求 2.4）
- C 端记录展示状态：已完成、审核中、失败（原型 11-1）

**问题：** B 端状态有 7 种，C 端展示只有 3 种。C 端的"审核中"映射到 B 端的哪些状态？

**建议：** 评审时明确 B 端内部状态 → C 端展示状态的映射关系，写入文档。

```
C端"审核中" = B端的 [待风控审核 + 待财务审核 + 待出款]？
C端"失败"   = B端的 [风控驳回 + 财务驳回 + 出款失败]？
C端"已完成" = B端的 [出款成功]
```

---

### S3 🟢 建议明确金额的最小单位和精度规范

**现状：** 需求说"法币小数后 2 位，加密货币小数后 8 位"，但**没说内部存储精度**。

**建议文档补充：**
- 内部存储统一使用整数（最小单位），如 VND 以"盾"为单位（整数）、THB 以"萨当"为单位（×100）、USDT 以"聪"为单位（×10^8）
- 明确每个币种的最小单位倍数
- 明确四舍五入的规则：充值时向上取整（对平台有利）还是四舍五入（公平）？提现时向下取整（对平台有利）？

---

### S4 🟢 建议明确"汇率日志"和"操作日志"的关系

**现状：** 汇率日志放在"操作日志页新增 Tab"。

**疑问：**
- 操作日志是 ser-blog 管的，汇率日志是 ser-wallet 产生的。数据在两个不同服务的库里。
- 前端在"操作日志"页面切换到"汇率日志"Tab 时，是调 ser-blog 的接口还是调 ser-wallet 的接口？
- 还是说汇率日志也写入 ser-blog？

**建议：** 评审时确认汇率日志的存储位置和查询入口。

---

### S5 🟢 建议考虑多时区问题

**现状：** 财务数据统计需求 3.1 提到"服务器以 UTC 基准时间记录，管理后台可设置站点时区"。

**问题延伸：**
- C 端用户看到的时间是 UTC 还是本地时区？
- "单日提现限额"的"一天"是 UTC 的一天还是用户所在时区的一天？
- 汇率日志的"更新时间"是 UTC 还是站点时区？

**建议：** 统一约定：存储层全部 UTC，展示层前端按用户时区转换，"单日限额"按 UTC 0 点切割。评审时确认。

### 评审核心诉求

> 作为钱包模块的后端开发，我需要在评审会上拿到以下关键答案才能顺利启动开发：
>
> 1. **稽核流水的具体规则** — 否则奖励钱包的数据模型无法设计
