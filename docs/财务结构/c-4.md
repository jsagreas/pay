## 二、财务模块的六大功能域

在深入文件细节之前，先建立全局认知——财务模块管什么：

```
┌─────────────────────────────────────────────────────────────────┐
│                     ser-finance 六大功能域                        │
├──────────────┬──────────────┬───────────────┬──────────────────┤
│              │              │               │                  │
│  通道配置     │  支付配置     │  订单管理      │  人工修正         │
│  (代收+代付)  │  (充值+提现   │  (充值+提现    │  (补单+加款      │
│  管"水管"    │   方式)       │   记录+审核)   │   +减款)         │
│              │  管"水龙头"   │  管"水流账单"  │  管"人工补水"    │
│              │              │               │                  │
├──────────────┼──────────────┴───────────────┴──────────────────┤
│              │                                                 │
│  数据统计     │  通道统计 + 充值统计 + 提现统计                    │
│  管"水表"    │  按日维度汇总                                     │
│              │                                                 │
└──────────────┴─────────────────────────────────────────────────┘
```

**一句话类比**：如果钱包是"用户的钱包"，那财务就是"公司的收银台"。
- 通道 = 收银台背后连的银行/支付公司
- 支付方式 = 收银台前面给用户看的付款选项
- 订单管理 = 每笔收款/付款的记账和审核
- 人工修正 = 出了问题时的手工调账
- 数据统计 = 每日对账报表

**③ deposit_service.go — 充值订单管理**

| 属性 | 说明 |
|------|------|
| 结构体 | `DepositService` |
| 依赖注入 | `rep.DepositOrderRepo` + `domain.WalletRPC` + `domain.CurrencyRPC` + `domain.PaymentGateway` + `domain.ChannelRouter` + `domain.BlogRPC` |
| 服务对象 | B 端财务人员 + 三方支付回调 |

| 方法 | 端 | 职责 |
|------|-----|------|
| PageDepositOrder | B端 | 充值记录列表（22 字段，多维筛选） |
| DetailDepositOrder | B端 | 充值记录详情 |
| ExportDepositOrder | B端 | 导出充值记录（需密码 + 谷歌验证码） |
| **HandleDepositCallback** | **回调** | **★ 三方充值回调：验签 → 更新订单状态 → 调钱包 CreditDeposit 上账** |

**HandleDepositCallback 内部流程（这是钱包开发者必须理解的）**：

```
三方支付成功回调到达
    │
    ├── ① 验签（domain.PaymentGateway.VerifyCallback）
    ├── ② 查询充值订单（rep.DepositOrderRepo.GetByOrderNo）
    ├── ③ 幂等校验（订单已成功则直接返回）
    ├── ④ 更新订单状态 → 充值成功
    ├── ⑤ ★ 调钱包上账（domain.WalletRPC.CreditDeposit）
    │      传入：orderId, userId, currencyCode, amount, bonusAmount, idempotencyKey
    ├── ⑥ 写审计日志
    └── ⑦ 返回成功（通知三方不再重复回调）
```

**提现状态机（这是最复杂的部分）**：

```
创建订单 ──→ 待风控审核
                │
        ┌───────┴───────┐
        ▼               ▼
    风控通过          风控驳回
        │               │
        ▼               ▼
  待财务审核      ★ 调钱包解冻返还
        │
  ┌─────┴─────┐
  ▼           ▼
财务通过    财务驳回
  │           │
  ▼           ▼
待出款    ★ 调钱包解冻返还
  │
  ▼
出款审核通过 → 请求代付通道
  │
  ├── 代付成功回调 → ★ 调钱包 ConfirmWithdrawalDeduct（确认扣款）
  │
  └── 代付失败回调 → ★ 调钱包 UnfreezeForWithdrawal（解冻返还）
```

**每次审核拒绝或出款失败，财务都必须调钱包 RPC 解冻**——这是钱包开发者必须清楚的。

## 六、★ 与钱包模块的联调接口全景（最重要的部分）

### 6.1 双向调用矩阵

```
                   ser-wallet                          ser-finance
                 ┌──────────┐                        ┌──────────┐
                 │          │   ★ 8 个 RPC 调用       │          │
                 │ balance  │ ◄─────────────────────  │ wallet   │
                 │ _service │                         │ _rpc     │
                 │ .go      │                         │ .go      │
                 │          │   ★ 3 个 RPC 调用       │          │
                 │ currency │ ◄─────────────────────  │ currency │
                 │ _service │                         │ _rpc     │
                 │ .go      │                         │ .go      │
                 │          │                         │          │
                 │ deposit  │                         │ payment  │
                 │ _service │ ──────────────────────► │ _service │
                 │ .go      │   ★ 2 个 RPC 调用       │ .go      │
                 └──────────┘                        └──────────┘

                    共 13 个双向 RPC 交互接口
```

### 6.2 财务 → 钱包：8 个 RPC（财务触发，钱包执行）

| 序号 | 财务侧触发场景 | 调用钱包接口 | 钱包侧影响 | 何时触发 |
|------|-------------|-----------|-----------|---------|
| 1 | 人工加减款弹窗打开 | GetUserWalletBalance | 只读：展示各子钱包余额 | 运营点击"加款"/"减款"按钮 |
| 2 | 充值回调验签通过 | **CreditDeposit** | **写入：充值金额→中心钱包 + 奖金→奖励钱包** | 三方支付成功回调到达 |
| 3 | 提现订单创建成功 | **FreezeForWithdrawal** | **写入：用户余额→冻结额** | 用户提交提现订单 |
| 4 | 风控/财务审核拒绝 | **UnfreezeForWithdrawal** | **写入：冻结额→返还余额** | 审核人点击"拒绝" |
| 5 | 代付出款失败 | **UnfreezeForWithdrawal** | **写入：冻结额→返还余额** | 代付通道回调失败 |
| 6 | 代付出款成功 | **ConfirmWithdrawalDeduct** | **写入：扣除冻结额（钱真正离开）** | 代付通道回调成功 |
| 7 | 人工加款审核通过 | **ManualCredit** | **写入：指定子钱包加款** | 审核人点击"通过" |
| 8 | 人工减款审核通过 | **ManualDebit** | **写入：指定子钱包扣款** | 审核人点击"通过" |
| 9 | 补单审核通过 | **CreditCorrection** | **写入：补单金额+赠送入账** | 审核人点击"通过" |

> 注：序号 4 和 5 调用同一个接口 UnfreezeForWithdrawal，所以是 8 个接口、9 个场景。

### 6.3 钱包 → 财务：2 个 RPC（钱包触发，财务提供数据）

| 1 | C 端用户打开充值页面 | **GetAvailableDepositMethods** | 当前币种可用充值方式列表 + 每种方式的金额档位 + 限额 |
| 2 | C 端用户打开提现页面 | **GetAvailableWithdrawalMethods** | 当前币种可用提现方式列表 + 限额 |

### 6.4 财务 → 币种配置：3 个 RPC（读取公共数据）

| 接口 | 使用场景 |
|------|---------|
| GetCurrencyList | 所有 B 端页面的币种筛选下拉框 |
| GetExchangeRate | 充值/提现记录中"基准币种换算"列的计算 |
| GetBaseCurrency | 统计报表的汇率基准 |

```
                     │              ┌──────────┐
                     └──────────────│ 财务      │
                      失败:Unfreeze │ 回调处理   │──→ 成功: 钱包 ConfirmDeduct
                                   └──────────┘
```

## 七、数据归属边界（谁的数据归谁管）

| 数据 | 归属 | 另一方的关系 |
|------|------|------------|
| 代收/代付通道配置 | **财务** | 钱包不直接接触通道 |
| 充值/提现方式配置 | **财务** | 钱包通过 RPC 查询可用方式 |
| 充值订单生命周期 | **财务** | 钱包只在"上账"时被调用一次 |
| 提现订单生命周期 | **财务** | 钱包在"冻结/解冻/扣款"时被调用 |
| 人工修正审核 | **财务** | 钱包只在"审核通过"时被调用 |
| 三方回调验签 | **财务** | 钱包不接收三方回调 |
| 通道统计/充值统计/提现统计 | **财务** | 钱包不参与统计 |
| **用户钱包余额** | **钱包** | 财务必须通过 RPC 修改，不能直连钱包数据库 |
| **交易流水** | **钱包** | 钱包自己记，财务不操作 |
| **币种和汇率** | **钱包（币种配置域）** | 财务通过 RPC 读取 |

**核心原则**：**财务管通道和订单，钱包管余额和流水，币种配置管汇率。三者通过 RPC 交互，绝不直连对方数据库。**

## 十、与 ser-wallet 的规范对照

| 维度 | ser-wallet | ser-finance | 一致性 |
|------|-----------|-------------|--------|
| 顶层文件 | main.go + handler.go + go.mod | 同左 | 完全一致 |
| handler 委托模式 | 7 个 Service | 6 个 Service | 模式一致，数量不同 |
| ser/ 命名 | xxx_service.go | xxx_service.go | 完全一致 |
| rep/ 命名 | 表名.go | 表名.go | 完全一致 |
| domain/ | 5 个文件（RPC + 汇率API） | 7 个文件（RPC + 三方支付 + 路由） | 模式一致，domain/ 更重 |
| enum/ | 3 个文件 | 3 个文件 | 完全一致 |
| errs/ | 6022xxx | 6023xxx（推测） | 段号不同 |
| IDL 文件数 | 6 个 | 9 个 | 财务接口更多，需要更多文件 |
| 表数量 | 8+1+1 = 10 | 9+3 = 12 | 财务稍多 |
| 方法总数 | ~33 | ~46 | 财务 B 端接口更多 |

**核心差异**：
- 财务模块是**纯 B 端服务**（无 C 端接口），只有 2 个 RPC 供钱包 C 端使用
- 财务的 domain/ 层更重——需要对接三方支付通道（签名/验签/请求/回调）
- 财务有导出功能（需要密码+谷歌验证码的安全校验）
- 财务有最复杂的状态机（提现 7 种状态 + 锁定机制 + 三级审核）
