# ser-finance 工程目录结构设计

> 定位：在不写具体代码的前提下，完整呈现 ser-finance 模块"应该长什么样"
> 设计依据：扫描 ser-app（25 文件）、ser-buser（35 文件）、ser-bcom（21 文件）等现有服务的真实目录结构
> 工程规范：遵循现有分层架构（handler → ser → rep → gorm_gen）、命名惯例、代码生成流程
> 覆盖范围：ser-finance 目录本体 + 需要新增/修改的关联模块文件
> 职责说明：ser-finance 由同事负责开发，本文档用于了解对方模块结构，便于钱包模块联调协作
> 核心特征：**纯 B 端服务**（0 个 C 端接口），所有接口通过 gate-back 暴露；**三方通道集成层**（pay/）是该服务独有架构层

## 2. 根目录文件详解

ser-finance 根目录 **3 个 Go 文件**，与所有现有服务完全一致：

### 2.1 文件清单

| 文件 | 职责 | 代码规模 |
|------|------|---------|
| `main.go` | 服务启动入口 | ~60-80 行 |
| `handler.go` | Kitex Handler 委托层 | ~200-250 行 |
| `go.mod` | Go 模块定义 | ~10-15 行 |

### 2.2 main.go

**职责**：按固定顺序初始化基础设施，创建 Service 实例，启动 Kitex RPC Server。

**启动序列**（遵循全工程统一模式）：

```
1. lg.InitKLog()                          → 初始化日志
2. etcd.InitEtcd()                        → 初始化 ETCD 连接
3. cfg.InitMysql()                        → 初始化 TiDB 连接 + gorm_gen 绑定
4. cfg.InitRedis()                        → 初始化 Redis 连接
5. 创建 Service 实例                       → ser.NewChannelService() 等
6. flag + ETCD 获取端口                    → 监听端口
7. financeservice.NewServer(handler, opts) → 启动 Kitex Server
8. srv.Run()                              → 阻塞运行
```

**与 ser-wallet 的差异**：ser-wallet 启动序列中有"启动汇率定时任务"，ser-finance 没有定时任务（除非后续需要定时统计聚合）。

**工程参照**：ser-app/main.go、ser-item/main.go（完全相同的结构，替换服务名即可）

### 2.3 handler.go

**职责**：实现 Kitex 生成的 `FinanceService` 接口。每个方法是一行委托调用，**不包含任何业务逻辑**。

**结构**：

```
type FinanceServiceImpl struct {
    channelSer  *ser.ChannelService      // 通道配置
    paymentSer  *ser.PaymentService      // 支付配置
    rechargeSer *ser.RechargeService     // 充值记录
    withdrawSer *ser.WithdrawService     // 提现记录
    manualSer   *ser.ManualService       // 人工修正
    statSer     *ser.StatService         // 数据统计
    rpcSer      *ser.RpcService          // 对外 RPC
}
```

每个接口方法格式统一：

```
func (s *FinanceServiceImpl) GetDepositChannelList(ctx, req) (resp, err) {
    return s.channelSer.GetDepositChannelList(ctx, req)
}
```

**约 37 个 B 端方法 + 约 8 个 RPC 方法 = ~45 个方法**，每个方法 3 行代码。

**与 ser-wallet handler 的差异**：
- ser-wallet：持有 9 个 Service 指针（含 domain 层上游的 rpcSer）
- ser-finance：持有 7 个 Service 指针（含对外 rpcSer）
- ser-wallet 有 C 端 + B 端 + RPC 三类接口
- ser-finance **只有 B 端 + RPC 两类接口**（0 个 C 端）

### 2.4 go.mod

**职责**：声明模块路径和 Go 版本。

```
module ser-finance

go 1.25.5
```

依赖解析由 go.work 统一管理，go.mod 保持极简。

```
├── enum/       枚举常量定义（1 文件）

├── pay/        三方通道集成层（5 文件）          ← ser-finance 特有
├── ser/        Service 业务逻辑层（7 文件）

```

**手写文件数**：23 个
**自动生成文件数**：~34 个
**合计**：~57 个

### 3.2 cfg/ — 基础设施初始化

```
cfg/
├── mysql.go    # TiDB 连接初始化 + gorm_gen query 绑定
└── redis.go    # Redis 连接初始化
```

| 文件 | 职责 | 核心逻辑 |
|------|------|---------|
| mysql.go | 初始化数据库连接 | `sync.Once` → 从 ETCD 读 db_finance 连接串 → `mysql.Init(opt)` → `query.SetDefault(db)` |

**数据库**：独立的 `db_finance`，与钱包的 `db_wallet` 物理隔离。

### 3.4 errs/ — 错误码定义

```
errs/
└── code.go     # 6023xxx 系列错误码
```

**错误码格式**：`6{NNN}{MMM}` — 其中 NNN=023（服务编号），MMM=子模块序号。

| 段号 | 范围 | 子模块 | 示例 |
|------|------|--------|------|
| 0 | 6023001-6023099 | 通道配置 | 通道不存在、密钥格式无效 |
| 1 | 6023100-6023199 | 支付配置 | 方式名重复、通道ID无效 |
| 2 | 6023200-6023299 | 充值记录 | 订单不存在、导出权限不足 |
| 3 | 6023300-6023399 | 提现记录 | 审核权限不足、订单已锁定、状态不可流转 |
| 4 | 6023400-6023499 | 人工修正 | 申请人=审核人禁止、原订单不存在、金额异常 |
| 5 | 6023500-6023599 | 数据统计 | 日期范围无效、导出超限 |
| 6 | 6023600-6023699 | 三方通道 | 签名失败、通道请求超时、回调验签失败 |
| 7 | 6023700-6023799 | RPC 接口 | 参数校验失败、钱包 RPC 调用失败 |

### 3.5 pay/ — 三方通道集成层

```
pay/
├── dispatcher.go    # 通道轮询调度算法
├── collector.go     # 代收请求提交
├── payer.go         # 代付请求提交
├── callback.go      # 三方回调处理
└── signer.go        # 签名/验签工具
```

**为什么 ser-finance 需要 pay/ 层（其他服务大多没有）**：

财务模块的核心差异化在于**对接多家三方支付通道**。每家通道的 API 格式、签名方式、回调结构都不同。如果把通道集成逻辑散落在 Service 里，会导致 Service 层代码膨胀、难以维护。抽到 pay/ 层：
- Service 层只关心"充值要选个通道发出去"
- pay/ 层封装"怎么跟具体通道交互"
- 新增通道只改 pay/ 层，Service 不动

**与 ser-wallet domain/ 层的类比**：
- ser-wallet 的 domain/ 是**资金原子操作**的抽象（被多个 Service 复用）
- ser-finance 的 pay/ 是**三方通道交互**的抽象（被充值/提现 Service 复用）
- 两者都是"从 Service 层下沉一层，解决被多处调用的公共能力"

| 文件 | 职责 | 核心逻辑 |
|------|------|---------|
| dispatcher.go | 通道选择算法 | 输入：币种+金额+通道类型 → 查可用通道 → 计算综合分数（权重×成功率系数×时间系数×预警系数） → 返回最优通道 |
| collector.go | 代收提交 | 输入：通道配置+订单信息 → 拼装三方请求体 → 签名 → 发起 HTTP POST → 解析返回（支付 URL / 充值地址） |
| payer.go | 代付提交 | 输入：通道配置+提现信息+收款账户 → 拼装三方请求体 → 签名 → 发起 HTTP POST → 解析返回（通道订单号） |
| callback.go | 回调处理 | 输入：三方 HTTP 回调 → 识别通道来源 → 验签 → 解析金额/状态 → 分发到充值上账或提现确认 |
| signer.go | 签名工具 | 按通道配置的签名方式（RSA/MD5/HMAC-SHA256），统一的 Sign() / Verify() 接口 |

**注意**：dispatcher.go 内部依赖 channel_polling_config 表的轮询规则参数。

### 3.6 ser/ — Service 业务逻辑层

```
ser/
├── channel_ser.go     # 通道配置管理
├── payment_ser.go     # 支付配置管理
├── recharge_ser.go    # 充值记录管理
├── withdraw_ser.go    # 提现记录管理
├── manual_ser.go      # 人工修正管理
├── stat_ser.go        # 数据统计
└── rpc_ser.go         # 对外 RPC 接口逻辑
```

每个文件对应一个 Service struct，由 handler.go 持有并委托调用。

| 文件 | Service 结构体 | 包含的接口方法 | 依赖关系 |
|------|---------------|--------------|---------|
| channel_ser.go | ChannelService | DepositChannelList, DepositChannelEdit, PayoutChannelList, PayoutChannelEdit, DepositPollingConfig, PayoutPollingConfig | gorm_gen/repo/、cache/ |
| payment_ser.go | PaymentService | RechargeMethodList, RechargeMethodCreate, RechargeMethodEdit, WithdrawMethodList, WithdrawMethodCreate, WithdrawMethodEdit | gorm_gen/repo/、钱包 RPC（GetCurrencyList） |
| recharge_ser.go | RechargeService | RechargeRecordList, RechargeRecordExport, RechargeRecordDetail | gorm_gen/repo/（只读查询）、rep/ |
| withdraw_ser.go | WithdrawService | WithdrawRecordList, WithdrawRecordExport, RiskReview, FinanceReview, PayoutReview, LockOrder, UnlockOrder, ForceUnlock | pay/（出款审核触发代付）、钱包 RPC（审核驳回→解冻/出款成功→确认扣款） |
| manual_ser.go | ManualService | PatchOrderList, CreatePatch, ReviewPatch, CreditOrderList, CreateCredit, ReviewCredit, DebitOrderList, CreateDebit, ReviewDebit | 钱包 RPC（CreditWallet/DebitWallet/GetBalance）、双人审核校验 |
| stat_ser.go | StatService | ChannelStat, RechargeStat, WithdrawStat, StatExport, AggregateDaily（内部定时） | rep/stat_rep（聚合查询）、币种 RPC（GetExchangeRate 换算） |
| rpc_ser.go | RpcService | GetRechargeMethodList, GetWithdrawMethodList, MatchChannel, SubmitPayout, RechargeCallback, GetChannelFeeRate | pay/ 层、gorm_gen/repo/ |

**文件拆分原则**：一个子模块 = 一个 Service 文件。rpc_ser.go 单独拆出是因为 RPC 接口的调用方是外部服务（钱包等），与 B 端管理接口的调用方不同。

**与 ser-wallet ser/ 层的差异**：

| 对比项 | ser-wallet | ser-finance |
|--------|-----------|-------------|
| Service 文件数 | 9 个 | 7 个 |
| 调用 domain/pay 层 | 充值/提现/兑换/RPC → domain/ | 提现审核/RPC → pay/ |
| C 端接口分布 | wallet_ser/recharge_ser/withdraw_ser 等 | **无**（全 B 端） |
| 最复杂的 Service | withdraw_ser（冻结 + KYC + 通道） | withdraw_ser（三级审核 + 锁定 + 出款） |
| 外部 RPC 调用量 | 调财务 3-4 个 + KYC 1 个 | 调钱包 6 个 + 币种配置 3 个 |

### 3.7 rep/ — 手写 Repository 层

```
rep/
├── channel_rep.go     # 通道自定义查询
├── recharge_rep.go    # 充值记录自定义查询
├── withdraw_rep.go    # 提现记录自定义查询
├── manual_rep.go      # 人工修正单自定义查询
└── stat_rep.go        # 统计聚合查询
```

**为什么需要手写 rep/ 层？**

gorm_gen 生成的 repo/ 提供标准 CRUD（Create、GetOneById、GetPageByConds 等），但以下场景需要手写：

| 文件 | 包含的自定义查询 | 为什么 gorm_gen 不够 |
|------|----------------|-------------------|
| channel_rep.go | `GetAvailableChannels(currencyCode, channelType, amount)` — 按币种+金额范围+启用状态筛选 | 需要组合条件 + 金额范围 min/max |
| channel_rep.go | `GetChannelStats(channelId, days)` — 最近 N 天成功率/平均时间 | 需要时间窗口 + 聚合计算 |
| recharge_rep.go | `GetRechargeRecordPage(conds, cursor)` — 多条件动态筛选 + 游标分页 | B 端 21 字段列表，动态组合条件 |
| withdraw_rep.go | `GetWithdrawRecordPage(conds, cursor)` — 多条件动态筛选 + 游标分页 | B 端 22 字段列表，动态组合条件 |
| withdraw_rep.go | `CountPendingByReviewer(reviewerId)` — 待审核数量 | 特定条件计数 |
| manual_rep.go | `GetManualOrderPage(orderType, conds, cursor)` — 三种订单统一分页 | 补单/加款/减款三种表动态切换 |
| stat_rep.go | `AggregateChannelDaily(channelId, dateRange)` — 通道日统计聚合 | 需要 GROUP BY + SUM + COUNT |
| stat_rep.go | `AggregateRechargeStat(dateRange, currencyCode)` — 充值统计 | 需要跨字段聚合 |
| stat_rep.go | `AggregateWithdrawStat(dateRange, currencyCode)` — 提现统计 | 需要按审核阶段分段统计 |

### 3.8 cache/ — 缓存管理

```
cache/
└── keys.go     # Redis Key 定义 + 缓存辅助函数
```

| Key 模式 | 用途 | TTL |
|---------|------|-----|
| `finance:methods:recharge:{currency}` | 充值方式列表缓存 | 5 分钟 |
| `finance:methods:withdraw:{currency}` | 提现方式列表缓存 | 5 分钟 |
| `finance:channel:stats:{channelId}` | 通道成功率/平均时间缓存 | 1 分钟 |
| `finance:lock:withdraw:{orderId}` | 提现订单审核锁 | 5 分钟（审核完自动释放） |
| `finance:lock:manual:{orderId}` | 人工修正单审核锁 | 5 分钟 |
| `finance:daily:alert:{channelId}:{date}` | 通道每日累计金额（预警用） | 当日过期 |

**与 ser-wallet cache/ 的差异**：
- ser-wallet 缓存侧重**用户维度**（余额、币种偏好、操作分布式锁）
- ser-finance 缓存侧重**配置维度**（方式列表、通道统计）+ **审核锁**

### 3.9 gen/ — 代码生成入口

### 6.2 充值/提现记录的表归属说明

**关键问题**：充值记录和提现记录的数据存在 db_wallet 的 `recharge_order` 和 `withdraw_order` 表中（由钱包创建订单时写入），但 B 端的记录管理/审核功能在 ser-finance 负责。

**有两种方案**：

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| A：跨库直查 | ser-finance 直接读 db_wallet 的订单表 | 简单直接 | 跨库耦合、违反数据库隔离原则 |
| B：RPC 调用 | ser-finance 通过 RPC 调 ser-wallet 查询订单 | 服务边界清晰 | 多一层网络调用 |

**推荐方案 B**：通过 RPC 调用。ser-wallet 暴露 `GetRechargeOrderList`、`GetWithdrawOrderList` 等 RPC 接口供 ser-finance 调用。这意味着 ser-finance 自身的 gorm_gen 中**不包含** recharge_order 和 withdraw_order 的模型——这两张表的 CRUD 权限完全在 ser-wallet 侧。

**但实际实践中**：如果评审决定采用方案 A（跨库直查），则 ser-finance 的 gorm_gen 需要额外连接 db_wallet 生成对应模型，或者手动在 rep/ 层写跨库查询。具体方案需评审确认。

> 上方目录树中的 gorm_gen 已按方案 B 设计（只包含 db_finance 自身的 11 张表）。充值/提现记录子模块的 recharge_rep.go 和 withdraw_rep.go 在方案 B 下主要封装 RPC 调用的参数组装和结果转换。
