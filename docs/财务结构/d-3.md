## 三、逐层详解

### 3.1 根目录文件

| 文件 | 职责 | 工程规范参照 |
|------|------|------------|
| `main.go` | 服务启动入口：初始化日志 → 初始化 ETCD → 读取端口 → 创建 Service 实例 → 启动 Kitex Server | ser-item/main.go 完全一致 |
| `handler.go` | RPC 方法分发层：定义 `FinanceServiceImpl` 结构体，每个 Thrift 方法一行委托到 `internal/ser/` | ser-item/handler.go 完全一致 |
| `go.mod` | 模块声明 `module ser-finance`，依赖 `common/pkg`、`common/rpc` | 同其他 ser-xxx |
| `sql/01.table.sql` | ~14 张表的完整 CREATE TABLE DDL | ser-buser/sql/01.table.sql |

#### main.go 骨架逻辑

```
lg.InitKLog()
etcd.InitEtcd()
port := flag.String("port", etcd.DirectGet(ctx, namesp.EtcdFinancePort), "端口号")
flag.Parse()

// 创建所有 Service 实例
channelService    := ser.NewChannelService()
pollingService    := ser.NewPollingService()
methodService     := ser.NewMethodService()
depositService    := ser.NewDepositService()
withdrawService   := ser.NewWithdrawService()
callbackService   := ser.NewCallbackService()
manualService     := ser.NewManualService()
auditService      := ser.NewAuditService()
statsService      := ser.NewStatsService()
exportService     := ser.NewExportService()
siteConfigService := ser.NewSiteConfigService()
enumService       := ser.NewEnumService()

// 启动 Kitex Server
srv := financeservice.NewServer(
    &FinanceServiceImpl{...},
    rpc.InitRpcServerParams(namesp.EtcdFinanceService, utils.S2Int(ctx, *port))...,
)
srv.Run()
```

#### handler.go 骨架逻辑

```go
type FinanceServiceImpl struct {
    channelService    *ser.ChannelService
    pollingService    *ser.PollingService
    methodService     *ser.MethodService
    depositService    *ser.DepositService
    withdrawService   *ser.WithdrawService
    callbackService   *ser.CallbackService
    manualService     *ser.ManualService
    auditService      *ser.AuditService
    statsService      *ser.StatsService
    exportService     *ser.ExportService
    siteConfigService *ser.SiteConfigService
    enumService       *ser.EnumService
}

// 每个方法一行委托，零逻辑。例如：
func (s *FinanceServiceImpl) DepositChannelList(ctx, req) (resp, err) {
    return s.channelService.DepositChannelList(ctx, req)
}
func (s *FinanceServiceImpl) GetDepositMethods(ctx, req) (resp, err) {
    return s.methodService.GetDepositMethods(ctx, req)
}
```

### 3.7 internal/cache/ — Redis 缓存封装

| 文件 | 职责 | 缓存策略 |
|------|------|---------|
| `channel_cache.go` | 通道列表缓存、轮询当前指针（半小时切换） | Cache-Aside，TTL 3~5 分钟；轮询指针 Redis Hash |
| `method_cache.go` | 充值/提现方式列表缓存（按币种） | Cache-Aside，TTL 3~5 分钟，变更时删缓存 |

**通道轮询状态缓存说明**：轮询机制需要追踪"当前选中哪个通道"和"半小时切换计时"，这部分状态适合存 Redis（多实例共享），不适合存内存。

### 3.8 internal/domain/ — 领域辅助对象

| 文件 | 职责 | 被谁调用 |
|------|------|---------|
| `rpc_wallet.go` | 调用 ser-wallet RPC 的封装 — **这是财务与钱包联调的核心文件** | deposit_service（充值回调入账）、withdraw_service（冻结/解冻/扣款）、manual_service（加减款/补单入账） |
| `rpc_currency.go` | 调用 ser-wallet 币种配置 RPC（GetCurrencyInfo / GetExchangeRate） | method_service（关联币种）、stats_service（汇率换算） |
| `rpc_blog.go` | 调用 ser-blog.AddActLog 记录操作日志 | 所有 B 端写操作的 service |
| `rpc_buser.go` | 调用 ser-buser.QueryUserDetail 获取管理员姓名 | withdraw_service（展示审核人）、manual_service（展示操作人） |
| `channel_selector.go` | 通道选择算法 — 按配比权重轮询 + 通道可用性校验 + 半小时切换 | deposit_service（分配代收通道）、withdraw_service（分配代付通道） |
| `fee_calculator.go` | 手续费计算 — 固定金额模式 / 比例模式 / 阶梯模式 | deposit_service（充值手续费）、withdraw_service（提现手续费） |
| `timezone.go` | 时区转换 — 站点时区时间 → UTC+0 查库、UTC+0 结果 → 站点时区展示 | stats_service（统计查询全部经过时区转换） |
| `export_helper.go` | 导出工具 — 密码校验 + 谷歌验证码校验 + 生成 Excel/CSV | export_service |
| `third_party.go` | 三方通道 HTTP 对接封装 — 代收下单、代付下单、回调签名验证、金额校验 | deposit_service（向三方发起代收）、withdraw_service（向三方发起代付）、callback_service（校验回调签名） |

**`rpc_wallet.go` 详解**（与你最相关的文件）：

```
调用 ser-wallet 的方法清单：
  ├── WalletCredit(userId, currency, walletType, amount, bizType, orderNo)
  │     ← 充值到账入账 / 补单入账
  │
  ├── WalletDebit(userId, currency, walletType, amount, bizType, orderNo)
  │     ← 提现出款成功确认扣款
  │
  ├── WalletFreeze(userId, currency, amount, orderNo)
  │     ← 提现下单时冻结余额
  │
  ├── WalletUnfreeze(userId, currency, amount, orderNo)
  │     ← 提现审核驳回时解冻
  │
  ├── ManualCredit(userId, currency, walletType, amount, orderNo, remark)
  │     ← 人工加款审核通过
  │
  ├── ManualDebit(userId, currency, walletType, amount, orderNo, remark)
  │     ← 人工减款审核通过
  │
  └── GetWalletInfo(userId)
        ← 人工修正时查用户钱包全貌
```

### 3.9 internal/rep/ — 手写 Repository

> gorm_gen/repo 提供标准 CRUD，但审核流转、统计聚合等复杂逻辑必须手写。

| 文件 | 职责 | 关键操作 |
|------|------|---------|
| `channel_rep.go` | 通道管理 + 轮询配置 | 查询可用通道列表（含状态+金额范围过滤）；轮询配置批量更新（权重数组） |
| `method_rep.go` | 充值/提现方式关联查询 | 按币种查询方式列表 JOIN 通道（图标/金额范围/手续费/排序）；方式与通道的关联校验 |
| `deposit_rep.go` | 充值订单查询 + 导出 | 分页列表（多条件筛选：状态/币种/通道/时间）；导出数据批量提取；回调状态更新 |
| `withdraw_rep.go` | **最复杂** — 提现订单 + 审核 + 锁定 | 分页列表；三级审核状态流转（UPDATE WHERE current_status = X）；锁定/解锁/强制解锁；按审核级别查待审列表 |
| `manual_rep.go` | 人工修正单管理 | 补单/加款/减款各自列表查询；审核状态更新（校验审核人≠提交人） |
| `stats_rep.go` | 统计聚合查询 | GROUP BY date, currency_code 聚合（SUM/COUNT/AVG）；通道维度统计 JOIN 通道表；成功率计算 |
| `callback_rep.go` | 回调日志 | 插入回调记录（含原始报文）；按订单号查回调历史（排查问题用） |

**构造函数模式**（与 ser-item/internal/rep/ 一致）：

```go
func NewWithdrawRepo() *WithdrawRepo {
    db := cfg.InitMySQL()
    query.SetDefault(db)
    return &WithdrawRepo{
        db:    db,
        redis: cfg.InitRedis(),
    }
}
```

## 四、数据表设计概览

> 财务模块的表与钱包模块互补——钱包管余额和流水，财务管通道和订单。

### 4.1 表分级清单

**必须 (Must) — 6 张**

| 表名 | 定位 | 必须的理由 |
|------|------|-----------|
| `deposit_channel` | 代收通道主表 | 充值流程通道分配的数据源，无此表则通道轮询无法工作 |
| `withdraw_channel` | 代付通道主表 | 提现出款通道分配的数据源 |
| `deposit_method` | 充值方式配置 | C 端充值方式列表的数据源（ser-wallet 通过 RPC 从此获取） |
| `withdraw_method` | 提现方式配置 | C 端提现方式列表的数据源 |
| `deposit_order` | 充值订单（财务视角） | 承载完整的充值流程：用户→通道→三方→回调→入账，含通道信息/三方订单号/回调状态 |
| `withdraw_order` | 提现订单（财务视角） | 承载完整的提现流程：审核→出款→回调，含三级审核状态/审核人/审核时间 |

**应该 (Should) — 5 张**

| 表名 | 定位 | 应该的理由 |
|------|------|-----------|
| `channel_polling` | 通道轮询配置 | 需求明确要求按配比分配通道流量，需独立配置表管理轮询权重 |
| `manual_order` | 人工修正统一表 | 补单/加款/减款共用结构，通过 type 字段区分；审核人≠提交人的约束载体 |
| `audit_log` | 审核操作日志 | 三级审核每步操作需有记录（审核人/时间/结果/备注），用于审计追踪 |
| `callback_log` | 三方回调日志 | 记录每次三方回调的原始报文，排查"充值成功但未入账"等问题的关键依据 |
| `site_config` | 站点配置 | 时区设置（需求文档3-1"管理后台可设置站点时区"） |

**可选 (Optional) — 3 张**

| 表名 | 定位 | 可选的理由 |
|------|------|-----------|
| `deposit_daily_stats` | 充值日聚合 | 首版可从 deposit_order 实时聚合，量大后需预聚合提升查询性能 |
| `withdraw_daily_stats` | 提现日聚合 | 同上 |
| `channel_daily_stats` | 通道日聚合 | 同上 |

## 五、common/idl/ser-finance/ — IDL 文件详解

| 文件 | 定义内容 | 方法/结构体数量 |
|------|---------|---------------|
| `service.thrift` | FinanceService 主入口，include 其他 thrift，列出全部 RPC 方法 | ~63 个方法 |
| `channel.thrift` | DepositChannelListReq/Resp, ChannelEditReq, PollingConfigReq/Resp ... | ~16 个 struct |
| `method.thrift` | DepositMethodListReq/Resp, MethodAddReq, MethodEditReq ... | ~12 个 struct |
| `deposit.thrift` | DepositRecordListReq/Resp, DepositRecordItem, DepositExportReq ... | ~10 个 struct |
| `withdraw.thrift` | WithdrawRecordListReq/Resp, AuditReq, LockReq, WithdrawRecordItem ... | ~16 个 struct |
| `manual.thrift` | ManualSupplementReq, ManualCreditReq, ManualDebitReq, ManualAuditReq ... | ~14 个 struct |
| `stats.thrift` | ChannelStatsReq/Resp, DepositStatsReq/Resp, WithdrawStatsReq/Resp ... | ~12 个 struct |
| `finance_rpc.thrift` | GetDepositMethodsReq/Resp, CreateDepositOrderReq/Resp, GetOrderStatusReq/Resp ... | ~12 个 struct |

**`finance_rpc.thrift` 单独一个文件的原因**：

- RPC 接口参数结构与 B 端管理接口不同（B 端操作基于管理员权限，RPC 接口基于 userId + orderNo）
- ser-wallet 等外部服务调用时只需 include 这一个文件
- 与 ser-wallet 的 `wallet_rpc.thrift` 设计思路一致

**finance_rpc.thrift 核心接口说明**（这是你最需要关注的）：

| 接口 | 调用方 | 参数 | 返回 |
|------|--------|------|------|
| GetDepositMethods | ser-wallet | currencyCode | 充值方式列表（名称/图标/金额范围/手续费/排序） |
| GetWithdrawMethods | ser-wallet | currencyCode | 提现方式列表（名称/图标/限额/手续费） |
| CreateDepositOrder | ser-wallet | userId, currencyCode, amount, payMethod, bonusInfo | 平台订单号 + 支付URL/USDT地址 |
| CreateWithdrawOrder | ser-wallet | userId, currencyCode, amount, fee, payMethod, accountInfo | 平台订单号 |
| GetOrderStatus | ser-wallet | orderNo | 订单状态 + 金额 + 时间 |
| GetChannelInfo | ser-wallet | channelId | 通道手续费/限额信息 |

## 十、与你的 ser-wallet 联调关键点

> 这是整份文档的价值核心——你需要清楚**对方会在什么时机、调用你的什么接口、传什么参数、期望什么返回**。

### 10.1 财务调用钱包的 7 条 RPC 链路

| 触发时机 | 财务侧代码位置 | 调用你的接口 | 传参重点 | 你的响应 |
|---------|--------------|-----------|---------|---------|
| 充值回调到账 | callback_service → rpc_wallet | `WalletCredit` | userId, currency, walletType=中心, amount=到账金额, bizType="deposit", orderNo | 入账成功 + 入账后余额 |
| 补单审核通过 | manual_service → rpc_wallet | `WalletCredit` | 同上，bizType="supplement", orderNo=B+16位 | 入账成功 |
| 提现下单冻结 | withdraw_service → rpc_wallet | `WalletFreeze` | userId, currency, amount=提现+手续费, orderNo | 冻结成功 |
| 审核驳回解冻 | audit_service → rpc_wallet | `WalletUnfreeze` | userId, currency, amount, orderNo（同冻结时） | 解冻成功 |
| 出款成功扣款 | callback_service → rpc_wallet | `WalletDebit` | userId, currency, walletType=中心, amount, bizType="withdraw", orderNo | 扣款成功（实际是冻结金额消失） |
| 人工加款通过 | manual_service → rpc_wallet | `ManualCredit` | userId, currency, **walletType=指定4种之一**, amount, orderNo=A+16位, remark | 加款成功 |
| 人工减款通过 | manual_service → rpc_wallet | `ManualDebit` | userId, currency, walletType, amount, orderNo=M+16位, remark | 减款成功 |

### 10.2 钱包调用财务的 5 条 RPC 链路

| 触发时机 | 你的代码位置 | 调用财务的接口 | 传参重点 | 财务响应 |
|---------|------------|-----------|---------|---------|
| C 端打开充值页 | deposit_service → rpc_finance | `GetDepositMethods` | currencyCode | 方式列表（名称/图标/金额范围/手续费/排序） |
| C 端发起充值 | deposit_service → rpc_finance | `CreateDepositOrder` | userId, currency, amount, payMethod, (bonusInfo) | 平台订单号 + 支付URL/USDT地址 |
| C 端打开提现页 | withdraw_service → rpc_finance | `GetWithdrawMethods` | currencyCode | 方式列表（名称/图标/限额/手续费） |
| C 端发起提现 | withdraw_service → rpc_finance | `CreateWithdrawOrder` | userId, currency, amount, fee, payMethod, accountInfo | 平台订单号 |
| 查充值/提现状态 | deposit_service → rpc_finance | `GetOrderStatus` | orderNo | 订单状态 + 金额 + 时间 |

### 10.3 联调时序——以"用户充值"为例

```
时间线 ──────────────────────────────────────────────────────►

你的模块 (ser-wallet)                      同事模块 (ser-finance)
────────────────────                      ─────────────────────

① 用户打开充值页面
   deposit_service.GetDepositMethods
   ─── RPC ───────────────────────────►   method_service.GetDepositMethods
                                           → method_rep 查方式列表
   ◄── 返回方式列表 ──────────────────     ← 返回

② 用户选方式，输入金额，提交
   deposit_service.CreateDeposit
   → 重复拦截检查（本地）
   → 币种校验（本地）
   ─── RPC ───────────────────────────►   deposit_service.CreateDepositOrder
                                           → polling_service 轮询分配通道
                                           → third_party 向三方代收下单
   ◄── 返回订单号+支付URL ─────────────     ← 返回

③ 用户完成支付（三方页面）

④ 三方回调
                                           callback_service.DepositCallback
                                           → 校验签名 + 校验金额
                                           → 更新订单状态
   ◄── RPC ───────────────────────────     → rpc_wallet.WalletCredit(...)
   account_service.WalletCredit
   → FOR UPDATE 锁行
   → 余额 += amount
   → 写流水
   ─── 返回成功 ──────────────────────►     ← 入账完成

⑤ 用户查状态（可选轮询）
   deposit_service.GetDepositOrderStatus
   ─── RPC ───────────────────────────►   deposit_service.GetOrderStatus
   ◄── 返回"充值成功" ────────────────     ← 返回
```

### 10.4 联调时序——以"用户提现"为例

```
时间线 ──────────────────────────────────────────────────────►

你的模块 (ser-wallet)                      同事模块 (ser-finance)
────────────────────                      ─────────────────────

① 用户发起提现
   withdraw_service.CreateWithdraw
   → KYC 校验 (rpc_kyc)
   → 充值来源规则校验（本地）
   → 限额校验（本地）
   → 余额校验（本地）
   → 冻结余额（本地 DB 事务）
   ─── RPC ───────────────────────────►   withdraw_service.CreateWithdrawOrder
                                           → 创建提现订单（状态：审核中）
   ◄── 返回订单号 ────────────────────     ← 返回

② 三级审核（B 端管理员操作）
                                           audit_service.RiskAudit → 通过
                                           audit_service.FinanceAudit → 通过
                                           audit_service.PayoutAudit → 通过
                                           → third_party 向三方代付下单

③ 如果审核驳回
                                           audit_service → 驳回
   ◄── RPC ───────────────────────────     → rpc_wallet.WalletUnfreeze(...)
   account_service.WalletUnfreeze
   → 冻结金额 -= amount
   ─── 返回成功 ──────────────────────►     ← 解冻完成

④ 三方出款成功回调
                                           callback_service.WithdrawCallback
                                           → 更新订单状态 → 已完成
   ◄── RPC ───────────────────────────     → rpc_wallet.WalletDebit(...)
   account_service.WalletDebit
   → 冻结金额 -= amount（正式扣除）
   → 写流水
   ─── 返回成功 ──────────────────────►     ← 扣款完成
```

> 文档总结：ser-finance 工程由 **~88 个文件**组成（48 个手写 + ~40 个自动生成），分 6 个 Phase 逐步创建。核心文件是 `callback_service.go`（三方回调处理，连接充值入账和提现扣款）和 `audit_service.go`（三级审核流转），它们是该模块的心脏。所有目录结构和命名规范与现有工程（ser-item/ser-app/ser-buser）及 ser-wallet 完全一致。
>
> **对你的价值**：掌握此文档后，你在联调时将清楚——财务模块的 `rpc_wallet.go` 会在什么时机调用你的哪个接口、传什么参数、期望什么响应；你的 `rpc_finance.go` 需要调用财务的哪些接口、怎么传参、能拿到什么数据。12 条 RPC 链路，心中有数。
