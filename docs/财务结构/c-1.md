### 4.2 enum/ — 枚举层

| 文件 | 包含的枚举定义 | 来源（产品原型） |
|------|--------------|----------------|

| manual.go | `ManualCreditType`（运营补偿/运营奖励/财务修正/测试加款）、`ManualDebitType`（异常追回/风控处罚/财务冲正/费用调整）、`ManualOrderStatus`（待审核/审核拒绝/成功） | 人工修正-5/6.png 类型下拉 |

**枚举总量：~14 组**

**提现订单 7 种状态流转**（来自产品原型 提现记录-1.png）：

```
待风控审核 → 风控驳回
           → 待财务审核 → 财务驳回
                        → 待出款 → 出款失败
                                 → 出款成功
```

### 4.3 errs/ — 错误码层

| 文件 | 职责 |
|------|------|
| code.go | 按业务模块划分错误码段 |

**错误码分段规划（推测）**：

| 范围 | 模块 | 示例 |
|------|------|------|
| 6{xxx}000 ~ 099 | 通用 | 参数错误、权限不足 |
| 6{xxx}100 ~ 199 | 通道配置 | 通道不存在、通道已禁用 |
| 6{xxx}200 ~ 299 | 支付配置 | 方式名称重复、关联通道为空 |
| 6{xxx}300 ~ 399 | 充值 | 通道匹配失败、回调验签失败 |
| 6{xxx}400 ~ 499 | 提现 | 审核权限不足、订单已锁定 |
| 6{xxx}500 ~ 599 | 人工修正 | 原订单不存在、创建方不可审核 |
| 6{xxx}600 ~ 699 | 数据统计 | 导出验证失败 |

> 服务编号 `{xxx}` 待同事确认后填入。

### 4.4 cache/ — 缓存层

| 文件 | 缓存内容 | 缓存策略 | 为什么缓存 |
|------|---------|---------|----------|
| channel.go | 代收/代付通道启用列表 | 编辑/状态变更时失效 | 每笔充值/提现都要选通道，高频读取 |
| method.go | 充值/提现方式列表 | 编辑/状态变更时失效 | C端查充值方式是高频接口 |
| polling.go | 轮询参数（成功率系数/时间系数/预警系数） | 参数保存时失效 | 每次匹配通道都要用 |

**轮询算法核心逻辑**（来自产品原型 通道配置-3.png）：

```
输入：币种 + 支付方式 → 筛选出所有启用的代收通道

加权轮询 = 基础权重 × 成功率系数 × 时间系数 × 预警系数

成功率系数：单位时间内通道成功率 → 映射到配置的系数值
时间系数  ：通道平均完成时间 → 映射到配置的系数值
预警系数  ：累计金额达预警值 → 降低权重

最终：按调整后的权重概率选出一个通道
```

## 9. 与 ser-wallet 的结构对比

| 对比维度 | **ser-wallet（我的）** | **ser-finance（同事的）** |
|---------|---------------------|------------------------|
| 定位 | C端钱包 + B端币种配置 | B端财务后台管理 |
| 数据库表 | 10 | 14 |
| 接口数量 | 42（C端22 + B端5 + RPC15） | ~50（B端45 + RPC5） |
| rep 文件 | 10 | 14 |
| ser 文件 | 7 | 9 |
| enum 文件 | 6 | 7 |
| cache 文件 | 2 | 3 |
| 总文件数(手动) | 42 | 52 |
| 总文件数(含自动) | 73 | 95 |
| **端侧占比** | **C端为主，B端少量** | **B端为主，RPC对外少量** |
| **核心复杂度** | 余额安全（CAS+事务+幂等） | 审核流程（三级审核+锁定+轮询算法） |

**规模对比**：ser-finance 比 ser-wallet 更大（表多 4 张、接口多 8 个、文件多 22 个），因为财务后台管理的 B 端页面功能更多（通道+支付配置+记录+修正+统计 6 大块）。

## 10. 联调重点：双向 RPC 接口映射

### 10.1 钱包调财务（3 个 RPC）

| 钱包侧调用场景 | 财务侧提供的接口 | 数据流向 |
|-------------|---------------|---------|
| C端用户进入充值页 → 展示充值方式列表 | `GetDepositMethods(currencyCode)` → 返回方式列表+档位 | 财务 → 钱包 |
| C端用户进入提现页 → 展示提现方式列表 | `GetWithdrawalMethods(currencyCode)` → 返回方式列表+限额 | 财务 → 钱包 |
| C端用户点击充值 → 创建支付 | `MatchChannelAndCreatePayment(currency, method, amount)` → 轮询选通道→向三方发起→返回支付信息 | 钱包 → 财务 → 三方 |

### 10.2 财务调钱包（7 个 RPC）

| 财务侧调用场景 | 钱包侧提供的接口 | 钱包侧操作 |
|-------------|---------------|----------|
| 三方充值回调验证通过 | `CreditDeposit(userId, currency, amount, orderId)` | 中心钱包 + amount，记流水，幂等 |
| 提现出款成功 | `ConfirmWithdrawal(userId, currency, amount, orderId)` | frozen - amount（确认扣除） |
| 提现审核驳回/出款失败 | `UnfreezeBalance(userId, currency, amount, orderId)` | frozen - amount, available + amount |
| 补单审核通过 | `SupplementCredit(userId, currency, amount, bonusAmount, orderId)` | 中心+amount，奖励+bonusAmount |
| 加款审核通过 | `ManualCredit(userId, currency, walletType, amount, orderId, reason)` | 指定钱包类型 + amount |
| 减款审核通过 | `ManualDebit(userId, currency, walletType, amount, orderId, reason)` | 指定钱包类型 - amount |
| 减款前/审核辅助 | `GetUserBalance(userId, currency)` | 返回各钱包类型余额 |

### 10.3 联调时我需要关注的点

| 关注点 | 为什么 | 我需要做什么 |
|-------|-------|-----------|
| **财务的 IDL 何时定稿** | 我的 C 端充值/提现页依赖财务提供的方式列表接口 | 催同事出 IDL，或先 Mock |
| **回调归属谁** | 三方充值回调如果走财务，财务再调我入账；如果直接走我，我自己处理 | P0 待评审确认 |
| **提现审核后的 RPC 参数** | 确认扣款/解冻的 orderId 是钱包的还是财务的 | 对齐 orderId 传参规范 |
| **补单的 orderId 约定** | 补单入账的 orderId 是补单号（B+16位）还是原订单号 | 影响我的幂等去重逻辑 |
| **加减款的多钱包调用方式** | 一次审核可能对 4 种钱包类型各调一次，还是一次 RPC 传所有 | 影响我的接口参数设计 |
| **订单号编码谁生成** | 充值订单号 C+16位 是钱包生成还是财务生成 | 影响我的 deposit_order 表设计 |
| **财务查我余额的频率** | 审核页面加载时查一次 vs 每次操作都查 | 评估是否需要缓存优化 |
