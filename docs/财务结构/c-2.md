# 一、ser-finance 业务全景

## 1.1 财务管理模块做什么

```
财务管理模块 = 平台的"钱进钱出"管控中心

  对内：管理支付通道、支付方式、审核提现、人工修正、数据统计
  对外：对接第三方支付通道（代收/代付），接收回调
  对钱包：提供充值通道匹配、提现审核、触发余额变更的 RPC 能力

  一句话：钱包管"账"（余额），财务管"钱的进出路径"（通道+审核+统计）
```

# 三、各层目录职责详解

## 3.1 分层总览

```
请求方向（从上到下）：

  gate-back                         B端管理接口（通道/支付配置、记录、审核、统计）
         │
  handler.go                        接口委托层（纯透传，零逻辑）
         │
  internal/ser/                     业务编排层（通道匹配、审核流程、统计聚合）
         │
         ├──→ internal/adapter/     通道适配层（对接各三方支付通道 API）★ 财务特有
         │
         ├──→ internal/rep/         手写数据访问扩展
         │         │
         ├──→ internal/gorm_gen/    自动生成的标准 CRUD
         │         │
         └──→ rpc.WalletClient()    调钱包 RPC（Credit/Debit/Freeze/Unfreeze/QueryBalance）
                   │
              TiDB + Redis          存储层
```

## 3.2 目录职责表

| 目录 | 层级角色 | 职责边界 | 禁止做的事 |
|------|---------|---------|-----------|
| `handler.go` | 接口委托 | ~52 个方法一行委托到 ser 层 | 不写业务逻辑 |
| `internal/ser/` | 业务编排 | 通道匹配、审核流转、回调处理、统计聚合、调钱包 RPC | 不直接操作钱包余额 |
| `internal/adapter/` | 通道适配 | 封装各三方通道的 HTTP API 调用（创建订单、查询状态） | 不包含业务规则 |
| `internal/rep/` | 数据访问扩展 | 统计聚合查询、多条件筛选分页、跨表关联查询 | 不写业务逻辑 |
| `internal/cache/` | 缓存 | 通道列表、轮询规则、充值方式/提现方式的 Redis 缓存 | 不决定缓存策略 |
| `internal/gorm_gen/` | 标准 CRUD | 自动生成，每张表 19 个方法 | 禁止手动修改 |
| `internal/cfg/` | 基础设施 | MySQL/Redis 连接初始化 | 不包含业务配置 |
| `internal/errs/` | 错误码 | 全模块错误码常量定义 | 不写错误处理逻辑 |
| `internal/enum/` | 枚举注册 | 通道类型、订单状态、审核状态等下拉选项 | 不包含业务判断常量 |

## 3.3 adapter 层为什么是财务特有

```
普通服务（ser-app / ser-wallet）：
  只与内部服务通信（RPC），不对接三方 HTTP API

财务服务（ser-finance）：
  需要对接多个三方支付通道，每个通道有自己的：
    · API 地址（请求地址）
    · 认证方式（商户ID + 密钥 + AppKey + 公钥 + 私钥）
    · 请求格式（JSON / Form / XML）
    · 签名算法（MD5 / RSA / HMAC）
    · 回调验签方式

  如果把这些差异散在 ser 层，代码耦合严重。
  收敛到 adapter 层，每个通道一个文件，统一适配器接口：

    type ChannelAdapter interface {
        CreateOrder(ctx, params) (payUrl string, err error)    // 创建支付订单
        QueryOrder(ctx, orderNo) (status, err error)          // 查询订单状态
        VerifyCallback(ctx, body, sign) (result, err error)   // 验证回调签名
        Disburse(ctx, params) (result, err error)             // 发起出款
    }

  新增一个通道 = 新增一个 adapter 文件，不改 ser 层代码。
```

# 五、internal/adapter/ — 通道适配层详解

## 5.1 目录结构

```
internal/adapter/
├── adapter.go              ← 接口定义 + 适配器工厂
├── abcpix.go               ← ABCPIX 通道实现
├── vnpay.go                ← VNPay 通道实现
└── ...                     ← 后续新增通道，一个通道一个文件
```

## 5.2 为什么需要适配层

从产品原型的通道编辑弹窗可以看到，每个通道有自己的一套密钥体系：

| 配置项 | 说明 |
|-------|------|
| 请求地址 | 三方 API 的 base URL |
| 商户ID | 三方分配的商户标识 |
| 密钥 | 签名用密钥 |
| AppKey | 应用级别的密钥 |
| 支付编码 | 支付类型编码 |
| 平台公钥 | 平台自己的公钥（给三方验签用） |
| 公钥 | 三方的公钥（验证三方回调签名用） |
| 私钥 | 平台私钥（签名用） |

> 每个通道的 API 协议不同，不可能在 ser 层写一堆 if-else。adapter 层把差异封装起来，ser 层只调统一接口。

## 5.3 通道类型与支付方式对应关系（原型实证）

```
通道支付类型（代收通道原型可见）:
  · 唤醒（调起三方支付APP）
  · 原生（三方提供原生支付界面）
  · 混合（支持多种支付方式）
  · 扫码（生成二维码用户扫描）

充值方式（支付配置原型可见）:
  VND: QR / Momo / ZaloPay / VNPay / ShopeePay / TED
  IDR: QR / Momo / ZaloPay / VNPay

提现方式:
  VND: 对应提现通道
  IDR: 对应提现通道

每个充值方式关联多个代收通道（多选），由轮询引擎按权重匹配。
```

# 六、通道轮询匹配引擎详解（polling_ser.go）

这是财务模块最核心的算法模块，直接决定用户充值走哪个通道。

## 6.1 轮询规则的三维度系数

从产品原型中的"轮询参数配置"弹窗（3个Tab）可以看到：

```
┌──────────────────────────────────────────────────────────────────┐
│  轮询参数 = 成功率系数 × 时间系数 × 预警系数                       │
├────────────────────┬─────────────────┬───────────────────────────┤
│  成功率系数         │  时间系数        │  预警系数                  │
│                    │                 │                           │
│  · 单位时间（分钟）  │  · 单位时间（分钟）│  · 预警值到期：当日        │
│  · 样本最小订单数   │  · 样本最小订单数  │    代收通道累计金额        │
│  · 成功率区间→系数  │  · 平均完成时间   │    ≥ 预警金额              │
│                    │    区间→系数     │                           │
│  例：              │  例：            │  当日代收金额 ≥ 预警金额时   │
│  成功率 80~100%    │  时间 0~5秒      │  该通道权重降为 0           │
│    → 系数 1.0      │    → 系数 1.0   │  （相当于临时禁用）          │
│  成功率 60~80%     │  时间 5~10秒     │                           │
│    → 系数 0.8      │    → 系数 0.8   │                           │
│  成功率 <60%       │  时间 >10秒      │                           │
│    → 系数 0.5      │    → 系数 0.5   │                           │
└────────────────────┴─────────────────┴───────────────────────────┘

最终通道得分 = 通道基础权重 × 成功率系数 × 时间系数 × 预警系数
按得分降序，选得分最高的通道处理本次充值请求
```

## 6.2 轮询匹配流程

```
MatchChannel(currencyId, methodId, amount) 流程：

  Step 1: 获取该币种+支付方式关联的所有启用通道
  Step 2: 过滤掉单笔限额不满足的通道
  Step 3: 过滤掉今日累计金额已达预警值的通道
  Step 4: 对剩余通道计算综合得分
  Step 5: 按得分降序排列，选最高分通道
  Step 6: 如果无可用通道 → 返回"通道繁忙，请换一种方式"
```

# 七、三级审核流程详解（withdraw_ser.go）

提现审核是财务模块最复杂的业务流程，涉及三级审核+锁定机制。

## 7.1 提现状态机

```
                    ┌─────────┐
                    │ 提现创建  │
                    └────┬────┘
                         │
                    ┌────▼────┐
                    │风控审核中│ ←── 锁定/解锁/强制解锁 可操作
                    └────┬────┘
                    通过 │ 不通过 → Unfreeze → 状态"已拒绝"
                    ┌────▼────┐
                    │财务审核中│
                    └────┬────┘
                    通过 │ 不通过 → Unfreeze → 状态"已拒绝"
                    ┌────▼────┐
                    │出款审核中│
                    └────┬────┘
                    通过 │ 不通过 → Unfreeze → 状态"已拒绝"
                    ┌────▼────┐
                    │  待出款  │
                    └────┬────┘
                         │ 调通道 adapter 发起出款
                    ┌────▼────┐
                    │ 出款中   │
                    └────┬────┘
              成功 ──┤    ├── 失败
                    │    │
             ┌──────▼┐  ┌▼──────┐
             │出款成功│  │出款失败│
             │ Debit │  │Unfreeze│
             └───────┘  └────────┘
```

## 7.2 审核操作与钱包 RPC 的对应关系

| 审核结果 | 财务动作 | 调钱包 RPC | 钱包侧效果 |
|---------|---------|----------|-----------|
| 风控/财务/出款 审核拒绝 | 更新订单状态为"已拒绝" | Unfreeze | 冻结金额归还到可用余额 |
| 三级审核全部通过 | 调通道发起出款 | （等回调） | 无变化，仍处于冻结状态 |
| 通道出款成功 | 更新订单状态为"出款成功" | Debit | 从冻结金额中扣除（balance 和 frozen_amount 同时减少） |
| 通道出款失败 | 更新订单状态为"出款失败" | Unfreeze | 冻结金额归还 |

## 7.3 锁定机制

```
锁定：暂停该笔提现订单的审核流程，状态变为"已锁定"
解锁：恢复审核，状态回到锁定前的审核阶段
强制解锁：跳过当前操作人的解锁权限检查

应用场景：
  · 发现可疑交易，先锁定暂停审核，调查后再决定
  · 锁定期间不影响用户其他操作
  · 冻结金额在锁定期间持续保持
```

# 八、人工修正与钱包 RPC 交互详解（adjust_ser.go）

## 8.1 三种人工修正操作

```
┌─────────────────────────────────────────────────────────────────┐
│  充值补单（订单号 B+16位）                                        │
│  · 场景：用户已支付但系统未收到回调                                │
│  · 流程：输入原充值订单号 → 查询原订单信息 → 填写补单金额          │
│         → 提交 → 另一人审核 → 通过 → 调钱包 Credit 上分           │
│  · 约束：同一原订单只能补一次                                     │
│  · 审核展示：原订单金额、补单金额、原赠送金额、补单赠送金额、       │
│             订单稽核倍数、赠金稽核流水 等                          │
├─────────────────────────────────────────────────────────────────┤
│  人工加款（订单号 A+16位）                                        │
│  · 场景：运营补偿、活动奖励、财务修正、测试加款                    │
│  · 流程：输入用户编号查询 → 选加款类型 → 选币种 → 填各钱包金额     │
│         → 提交 → 另一人审核 → 通过 → 调钱包 Credit 上分           │
│  · 加款类型：运营补偿、运营奖励、财务修正、测试加款                 │
│  · 特点：可同时给多种钱包加款（中心/奖励/代理/主播各填金额）        │
├─────────────────────────────────────────────────────────────────┤
│  人工减款（订单号 M+16位）                                        │
│  · 场景：异常追回、风控处罚、财务冲正、费用调整                    │
│  · 流程：输入用户编号查询 → 选减款类型 → 选币种 → 填各钱包金额     │
│         → 提交 → 另一人审核 → 通过 → 调钱包 Debit 扣款            │
│  · 减款类型：异常追回、风控处罚、财务冲正、费用调整                 │
│  · 特点：展示当前余额、填减款金额、自动算减款后余额                 │
│  · 约束：余额不足时不可减款                                       │
└─────────────────────────────────────────────────────────────────┘

共同规则：创建人 ≠ 审核人（双人复核），审核状态：待审核 → 通过/拒绝
```

## 8.2 人工修正调钱包 RPC 的参数对应

| 操作 | 钱包 RPC | orderNo | walletType | 金额来源 |
|------|---------|---------|------------|---------|
| 补单通过 | Credit | B+16位 | 中心钱包=补单金额, 奖励钱包=补单赠送金额 | 审核弹窗填写 |
| 加款通过 | Credit | A+16位 | 中心/奖励/代理/主播 各自金额 | 创建弹窗多钱包金额 |
| 减款通过 | Debit | M+16位 | 中心/奖励/代理/主播 各自金额 | 创建弹窗多钱包金额 |

> **这是钱包开发者最关注的点**：人工修正一次操作可能触发多次 Credit/Debit（一个钱包类型一次），钱包的 Credit/Debit 需要支持指定 walletType。

# 十五、从钱包视角看：我需要关注财务的哪些点

## 15.1 必须对齐的 RPC 契约（13 个方法）

```
我调财务的 6 个方法（需要知道入参出参）:

  ① GetDepositMethods(currencyId) → []DepositMethod
  ② MatchDepositChannel(currencyId, methodId, amount) → ChannelInfo
  ③ CreateChannelOrder(channelId, amount, userInfo) → payUrl/address
  ④ GetWithdrawMethods(currencyId) → []WithdrawMethod
  ⑤ SubmitWithdrawRequest(orderNo, amount, receiverInfo) → auditId
  ⑥ QueryChannelOrderStatus(orderNo) → status

财务调我的 7 个方法（需要保证我的接口稳定）:

  ❶ Credit(orderNo, userId, currencyId, walletType, amount, flowType)
  ❷ Debit(orderNo, userId, currencyId, walletType, amount, flowType)
  ❸ Freeze(orderNo, userId, currencyId, walletType, amount)
  ❹ Unfreeze(orderNo, userId, currencyId, walletType, amount)
  ❺ QueryBalance(userId, currencyId) → []WalletBalance
  ❻ QueryWalletDetail(userId) → 详细钱包信息
  ❼ CreditReward（或通过 Credit walletType=2 实现）
```

## 15.2 必须理解的关键流程

| 流程 | 钱包关注点 | 和财务的交互点 |
|------|---------|-------------|
| 充值 | Credit 幂等 + 多次回调防重 | 财务回调触发 Credit，orderNo 由谁生成？ |
| 提现 | Freeze 时机 + Debit 扣冻结金额 | 是钱包先 Freeze 再告诉财务，还是财务调钱包 Freeze？ |
| 人工加款 | 一次加多个钱包类型 | 财务一次审核通过，可能调多次 Credit（每个 walletType 一次） |
| 人工减款 | 每个钱包余额校验 | 财务调 QueryBalance 先查，再调 Debit |
| 补单 | 原订单关联 + 金额可能不同 | 补单金额和原订单金额可能不一致，orderNo 用 B+16位 |

## 15.3 尚未确定的协作约定

| 约定项 | 选项 | 影响 |
|--------|------|------|
| 充值回调方式 | 同步 RPC / MQ | 影响 Credit 的调用方式和幂等设计 |
| orderNo 归属 | 钱包生成 / 财务生成 / 各自生成 | 影响幂等 key 设计 |
| Freeze 执行者 | 钱包提现时自己 Freeze / 财务调钱包 Freeze | 影响提现流程编排 |
| 人工修正多钱包加款 | 一次 RPC 传数组 / 多次 RPC 各传一个 | 影响 Credit/Debit 接口设计 |
| 超时重试策略 | 由调用方重试 / 由被调方保证幂等 | 双方都需要幂等，但重试策略需统一 |
