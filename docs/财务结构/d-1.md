# ser-finance 工程目录结构设计

> 文档定位：不涉及代码细节，只设计 ser-finance 模块的**文件目录骨架**——有哪些目录、有哪些文件、每个文件干什么。让人拿到这份文档后，对整个工程"长什么样"有完整的方向感。
>
> 身份说明：ser-finance 是同事负责的财务管理模块，本文档从**钱包开发者的协作视角**出发，基于产品原型和需求文档推导其工程结构。目的是让我方清楚对方模块的骨架，为联调做好准备。
>
> 设计依据：产品原型（通道配置 / 支付配置 / 财务管理 / 数据统计四大板块）+ ser-wallet 的结构设计（对标参照）+ ser-app / ser-buser / ser-bcom 的实际目录结构（工程规范）。

## 二、数字概览

| 维度 | ser-finance | 对标 ser-wallet | 对标 ser-app |
|------|------------|----------------|-------------|
| 顶层文件 | 6 | 6（一致） | 6（一致） |
| cfg/ | 2 | 2（一致） | 2（一致） |
| errs/ | 1 | 1（一致） | 1（一致） |
| enum/ | 7 | 7（一致） | 4 |
| gorm_gen/model/ | 10 | 10（同为 10 张表） | 2 |
| gorm_gen/query/ | 11 | 11（含 gen.go） | 3 |
| cache/ | 2 | 2（一致） | 1 |
| rep/ | 10 | 10（一致） | 2 |
| domain/ | 1 | 1（结构一致，内容不同） | 0 |
| ser/ | 9 | 11（钱包多 2 个） | 5 |
| **手写文件总计** | **~49** | **~51** | **~20** |

ser-finance 的规模与 ser-wallet 接近（49 vs 51），但业务域的切面不同：钱包偏"余额管理"、财务偏"通道管理 + 审核流程"。

**为什么 ser-finance 比 ser-wallet 少 2 个 Service：**
- 钱包有 `wallet_rpc_service.go`（RPC 供给）和 `exchange_rate_service.go`（汇率管理）——这两个业务在财务模块中没有对应物
- 财务的 RPC 供给逻辑内嵌在 `channel_match_service.go` 和 `payment_method_service.go` 中，不需要独立文件

## 三、顶层文件详解

### 3.1 目录树（顶层）

```
ser-finance/
├── main.go
├── handler.go
├── go.mod
├── go.sum
├── .golangci.yml
├── .gitignore
└── sql/
    └── 01.table.sql
```

### 3.2 逐文件说明

| 文件 | 职责 | 工程参照 |
|------|------|---------|
| **main.go** | 服务启动入口。初始化日志 → 加载 ETCD 配置 → 初始化 Redis → 初始化 MySQL → 创建各 Service 实例 → 创建 Kitex RPC Server → 启动服务 | ser-app/main.go 完全一致 |
| **handler.go** | RPC 方法分发层。定义 `FinanceServiceImpl` 结构体，持有所有 Service 实例引用，实现 IDL 中定义的 RPC 方法，每个方法体只做一件事：调用对应 Service 的方法 | ser-app/handler.go 完全一致 |
| **go.mod** | Go 模块声明。`module ser-finance`，依赖 `common/pkg`、`common/rpc` 等 | ser-app/go.mod 格式一致 |
| **go.sum** | 依赖校验文件（自动生成） | — |
| **.golangci.yml** | Go Lint 规则配置 | 复制 ser-app 的即可 |
| **.gitignore** | Git 忽略规则 | 复制 ser-app 的即可 |
| **sql/01.table.sql** | 全量建表 DDL（10 张表），用于 DBA 初始化数据库或开发本地建表 | ser-buser/sql/01.table.sql 先例 |

### 3.3 main.go 的初始化流程

```
main()
  │
  ├── 1. lg.InitKLog()                          ← 初始化日志
  ├── 2. etcd.LoadAndWatch("/slg/")              ← 加载 ETCD 配置
  ├── 3. cfg.InitRedis()                         ← Redis 连接
  ├── 4. cfg.InitMySQL()                         ← MySQL 连接 + GORM query.SetDefault
  ├── 5. cron.Init()                             ← 初始化定时任务调度器
  │
  ├── 6. 创建 Service 实例（依赖注入）
  │      ├── channelService
  │      ├── paymentMethodService
  │      ├── depositRecordService
  │      ├── withdrawRecordService
  │      ├── manualService
  │      ├── channelMatchService
  │      ├── statisticsService
  │      ├── enumService
  │      └── cronService
  │
  ├── 7. cronService.RegisterJobs()              ← 注册定时任务
  │
  ├── 8. srv := financeservice.NewServer(        ← 创建 Kitex RPC 服务
  │          &FinanceServiceImpl{...},
  │          rpc.InitRpcServerParams(...)
  │      )
  │
  └── 9. srv.Run()                               ← 启动服务
```

### 3.4 handler.go 的结构体

```
FinanceServiceImpl
  │
  ├── channelService          *ser.ChannelService
  ├── paymentMethodService    *ser.PaymentMethodService
  ├── depositRecordService    *ser.DepositRecordService
  ├── withdrawRecordService   *ser.WithdrawRecordService
  ├── manualService           *ser.ManualService
  ├── channelMatchService     *ser.ChannelMatchService
  ├── statisticsService       *ser.StatisticsService
  └── enumService             *ser.EnumService
```

**handler.go 中每个方法的模式（统一）：**

```
func (s *FinanceServiceImpl) CreateDepositChannel(ctx context.Context, req *ser_finance.CreateDepositChannelReq) (*ser_finance.CreateDepositChannelResp, error) {
    return s.channelService.CreateDepositChannel(ctx, req)
}
```

handler 只做转发，不做业务逻辑。

### 5.2 错误码段规划

> 服务编号待分配（设为 XXX），错误码前缀 `60XX`。以下以占位方式展示分段：

```
60XX 000 ~ 099    通用 / 代收通道
60XX 100 ~ 199    代付通道
60XX 200 ~ 249    充值方式配置
60XX 250 ~ 299    提现方式配置
60XX 300 ~ 399    充值记录 / 充值回调
60XX 400 ~ 499    提现记录 / 提现审核 / 提现回调
60XX 500 ~ 599    人工修正（补单 / 加款 / 减款）
60XX 600 ~ 699    通道匹配 / 轮询
60XX 700 ~ 799    数据统计
60XX 900 ~ 999    预留
```

**典型错误码示例：**

| 错误码 | 常量名 | 含义 |
|--------|--------|------|
| 60XX001 | `DepositChannelNotFound` | 代收通道不存在 |
| 60XX002 | `DepositChannelDisabled` | 代收通道已禁用 |
| 60XX003 | `ChannelNameAlreadyExist` | 通道名称已存在 |
| 60XX101 | `PayoutChannelNotFound` | 代付通道不存在 |
| 60XX102 | `PayoutChannelDisabled` | 代付通道已禁用 |
| 60XX201 | `DepositMethodNotFound` | 充值方式不存在 |
| 60XX251 | `WithdrawMethodNotFound` | 提现方式不存在 |
| 60XX301 | `DepositRecordNotFound` | 充值记录不存在 |
| 60XX302 | `DepositCallbackDuplicate` | 充值回调重复 |
| 60XX303 | `DepositCallbackSignInvalid` | 充值回调签名验证失败 |
| 60XX401 | `WithdrawRecordNotFound` | 提现记录不存在 |
| 60XX402 | `WithdrawAuditStatusInvalid` | 提现审核状态不合法 |
| 60XX403 | `WithdrawRecordLocked` | 提现订单已被锁定 |
| 60XX404 | `WithdrawPayoutFailed` | 提现出款失败 |
| 60XX501 | `ManualAdjustNotFound` | 人工修正记录不存在 |
| 60XX502 | `ManualSelfAuditForbidden` | 操作人不能审核自己的单 |
| 60XX503 | `ManualDebitExceedBalance` | 减款金额超过余额 |
| 60XX601 | `NoAvailableChannel` | 无可用通道 |
| 60XX602 | `ChannelMatchTimeout` | 通道匹配超时 |

## 六、internal/enum/ — 枚举常量

### 6.1 目录树

```
internal/enum/
├── enum_registry.go       # 枚举注册表
├── channel.go             # 通道类型、通道方向、通道状态
├── payment_method.go      # 支付方式类型、支付方式状态
├── deposit.go             # 充值订单状态
├── withdraw.go            # 提现订单状态、审核级别、审核动作
├── manual.go              # 人工修正类型、修正状态
└── audit.go               # 审核结果、锁定状态
```

### 6.2 逐文件说明

| 文件 | 包含的枚举 | 值定义 |
|------|-----------|--------|
| **enum_registry.go** | 枚举注册表，对接 i18n 翻译 | `map[string]func() []EnumOption` |
| **channel.go** | `ChannelDirection`：1=代收（充值用） 2=代付（提现用）；`ChannelStatus`：0=禁用 1=启用；`PaymentType`：1=银行转账 2=电子钱包 3=USDT 4=QR扫码 | 通道基础枚举 |
| **payment_method.go** | `MethodStatus`：0=禁用 1=启用 | 支付方式状态 |
| **deposit.go** | `DepositStatus`：0=待支付 1=支付中 2=成功 3=失败 4=超时关闭 | 充值状态机 |
| **withdraw.go** | `WithdrawStatus`：0=待审核 1=风控审核中 2=财务审核中 3=出款审核中 4=出款中 5=成功 6=失败 7=驳回；`AuditLevel`：1=风控 2=财务 3=出款；`AuditAction`：1=通过 2=驳回 | 提现状态机（三级审核） |
| **manual.go** | `ManualType`：1=充值补单 2=人工加款 3=人工减款；`ManualStatus`：0=待审核 1=审核通过 2=审核拒绝 3=已执行 4=已取消 | 人工修正枚举 |
| **audit.go** | `AuditResult`：1=通过 2=驳回；`LockStatus`：0=未锁定 1=已锁定 | 审核和锁定枚举 |

### 6.3 与 ser-wallet 枚举的对比

| 维度 | ser-wallet | ser-finance |
|------|-----------|-------------|
| 共同概念 | 充值状态、提现状态 | 充值状态、提现状态 |
| 关键差异 | **交易类型**（10 种 TxType）是核心 | **审核级别**（3 级 AuditLevel）是核心 |
| wallet 有但 finance 无 | WalletType、TxDirection、ExchangeStatus | — |
| finance 有但 wallet 无 | — | ChannelDirection、ManualType、LockStatus |

> 注意：两边的充值/提现状态枚举**值可能不完全一致**。钱包侧的状态更偏"用户视角"（待支付→成功/失败），财务侧的状态更偏"运营视角"（风控审核→财务审核→出款审核→出款→成功/失败）。联调时需约定状态映射关系。

| 文件 | 缓存的数据 | 缓存策略 | Redis Key |
|------|-----------|---------|-----------|
| **channel.go** | 各币种的启用通道列表（通道匹配高频读取） | Cache-Aside：读命中返回，未命中查 DB 回写；通道新增/编辑/启禁用时删除缓存 | `finance:channel:{direction}:{currencyId}` |
| **payment_method.go** | 各币种的支付方式列表（钱包 RPC 高频调用） | Cache-Aside：C 端展示充值/提现方式时高频读取，配置变更时删除缓存 | `finance:method:{type}:{currencyId}` |

**为什么只缓存这两个：**
- 通道配置和支付方式是"读多写极少"的典型场景——运营配好之后很少改动，但每次用户充值/提现都要查
- 充值/提现记录不缓存——写多读少，且需要以 DB 为准
- 审核数据不缓存——一致性要求高，不适合缓存
- 与 ser-wallet 的缓存策略一致：只缓存"配置类"数据，不缓存"交易类"数据

### 10.3 关键说明：withdraw_record.go

这是整个 rep 层**最复杂的文件**，因为提现记录涉及多维度状态管理：

```
提现记录的状态维度（三个独立状态字段）：
│
├── status（订单状态）
│     0=待审核 → 1=风控审核中 → 2=财务审核中 → 3=出款审核中
│     → 4=出款中 → 5=成功 / 6=失败 / 7=驳回
│
├── audit_level（当前审核级别）
│     1=风控 → 2=财务 → 3=出款
│     任一级驳回 → status=驳回
│
└── lock_status（锁定状态）
      0=未锁定 1=已锁定
      锁定期间不允许审核操作
```

**典型操作：**

```
UpdateAuditLevel(ctx, orderNo, nextLevel, operatorId)
    → UPDATE withdraw_record
      SET audit_level = ?, status = ?, updated_at = ?
      WHERE platform_order_no = ? AND audit_level = ? AND lock_status = 0
    → 条件更新：当前审核级别匹配且未锁定才能流转
```

## 十一、internal/domain/ — 领域核心逻辑

### 11.1 目录树

### 11.2 为什么单独建 domain 层

| 维度 | 说明 |
|------|------|
| **ser-app 有吗** | 没有。因为 ser-app 不对接三方服务 |
| **ser-wallet 的 domain 是什么** | `balance_engine.go`——封装余额变更事务，被 6 个 Service 调用 |
| **ser-finance 为什么需要** | 三方通道 API 调用被 deposit_record_service 和 withdraw_record_service 共同使用。不同通道有不同的 API 协议（请求地址、签名方式、回调格式），需要统一抽象 |
| **channel_gateway.go 的定位** | 封装"与三方支付通道交互"这一核心操作，对外提供统一接口，内部根据通道配置（config_json）选择不同的 API 调用逻辑 |

### 11.3 channel_gateway.go 的职责

```
ChannelGateway
  │
  ├── CreateDepositOrder(ctx, channel, order) (channelResp, error)
  │     │
  │     ├── ① 读取通道配置（API地址、密钥、商户号等）
  │     ├── ② 组装请求参数（不同通道格式不同）
  │     ├── ③ 计算签名（不同通道签名算法不同）
  │     ├── ④ 发起 HTTP 请求到三方
  │     └── ⑤ 解析响应，返回通道订单号 + 支付台面 URL
  │
  ├── CreatePayoutOrder(ctx, channel, order) (channelResp, error)
  │     │
  │     └── 与充值类似，发起出款请求到三方代付通道
  │
  ├── VerifyCallback(ctx, channel, rawBody, sign) (callbackData, error)
  │     │
  │     ├── ① 验证回调签名（防伪造）
  │     ├── ② 解析回调数据
  │     └── ③ 返回标准化的回调结果
  │
  └── QueryOrderStatus(ctx, channel, channelOrderNo) (status, error)
        │
        └── 主动查询通道订单状态（用于超时订单的补偿查询）
```

**被谁调用：**

| 调用方 Service | 调用场景 |
|---------------|---------|
| deposit_record_service.go | 充值时创建通道订单、处理充值回调 |
| withdraw_record_service.go | 出款时创建通道出款订单、处理出款回调 |
| cron_service.go | 定时查询超时订单的通道状态（补偿查询） |

### 11.4 与 ser-wallet 的 domain 层对比

| 维度 | ser-wallet domain | ser-finance domain |
|------|------------------|-------------------|
| 文件 | balance_engine.go | channel_gateway.go |
| 核心能力 | 余额变更事务（行锁 + 流水 + 幂等） | 三方通道 API 调用（签名 + 协议适配） |
| 被几个 Service 共用 | 6 个 | 3 个 |
| 是否涉及事务 | 是（DB 事务 + SELECT FOR UPDATE） | 否（HTTP 外部调用） |
| 复杂度来源 | 数据一致性 | 协议多样性（不同通道不同 API） |

### 15.2 结构一致的地方

- 目录分层完全一致：cfg / errs / enum / gen / gorm_gen / cache / rep / domain / ser
- 顶层文件完全一致：main.go + handler.go + go.mod + ...
- 初始化流程一致：日志 → ETCD → Redis → MySQL → Service 创建 → Kitex 启动
- 命名规范一致：`xxx_service.go` / Repository 用表名 / Filter struct / 构造函数

### 15.3 结构差异的地方

| 差异点 | ser-wallet | ser-finance |
|--------|-----------|-------------|
| domain/ 文件的性质 | 数据库事务操作（行锁、原子写入） | HTTP 外部调用（三方 API、签名验证） |
| RPC 方向 | **被调用方**（被财务/投注/结算等模块调用） | **双向**：被钱包调用 + 调用钱包 |
| 最复杂的流程 | 余额变更事务（modifyBalance） | 提现三级审核（风控→财务→出款） |
| 锁机制 | 数据库行锁（SELECT FOR UPDATE） | 业务锁（lock_status 字段） |
| 三方交互 | 无直接三方调用 | 核心职责就是对接三方支付通道 |

## 十六、财务模块与钱包模块的联调地图

> 这一节是本文档的核心价值——让钱包开发者清楚"哪些接口需要联调、数据怎么流动"。

### 16.1 双向 RPC 接口总览

```
                    ser-wallet                           ser-finance
                  ┌─────────────┐                     ┌─────────────┐
                  │             │ ──── RPC 调出 ────→  │             │
                  │  充值/提现   │  GetPaymentMethods   │  支付配置    │
                  │  页面展示    │  MatchDepositChannel  │  通道匹配    │
                  │             │  MatchWithdrawChannel │             │
                  │             │                      │             │
                  │             │ ←── RPC 被调 ─────   │             │
                  │  余额引擎    │  CreditBalance        │  充值回调    │
                  │  balance_   │  DebitBalance          │  出款回调    │
                  │  engine     │  UnfreezeBalance       │  审核驳回    │
                  │             │  (ManualAdjust)        │  人工修正    │
                  └─────────────┘                     └─────────────┘
```

### 16.2 RPC 接口明细

**钱包→财务（3 个接口）：**

| 接口 | 钱包调用时机 | 财务返回什么 | 关联文件 |
|------|------------|------------|---------|
| **GetPaymentMethods** | 用户进入充值/提现页面 | 可用支付方式列表（名称、icon、限额、排序） | payment_method_service.go |
| **MatchDepositChannel** | 用户提交充值订单后 | 最优通道信息（通道 ID、通道 URL、通道参数）或"无可用通道" | channel_match_service.go |
| **MatchWithdrawChannel** | 出款审核全部通过后 | 最优代付通道信息 | channel_match_service.go |

**财务→钱包（4 个接口）：**

| 接口 | 财务调用时机 | 钱包做什么 | 关联文件 |
|------|------------|-----------|---------|
| **CreditBalance** | 充值回调成功 / 人工加款审核通过 / 补单审核通过 | 入账（可用余额 +）+ 写流水 | deposit_record_service.go, manual_service.go |
| **DebitBalance** | 出款回调成功 / 人工减款审核通过 | 扣款（冻结余额 -）+ 写流水 | withdraw_record_service.go, manual_service.go |
| **UnfreezeBalance** | 审核驳回 / 出款回调失败 | 解冻（冻结余额 → 可用余额）+ 写流水 | withdraw_record_service.go |
| **ManualAdjust** | 人工加款/减款审核通过（多钱包类型同时操作） | 对多个钱包类型分别执行余额变更 | manual_service.go |

### 16.3 联调数据流时序

**充值完整链路：**

```
时序    钱包侧                              财务侧                            三方通道
 │
 T1     用户选择充值方式
 │      → RPC: GetPaymentMethods ─────→ 返回可用方式列表
 │
 T2     用户输入金额、提交
 │      → RPC: MatchDepositChannel ───→ 轮询算法匹配通道
 │                                      ← 返回通道信息
 │
 T3     创建 deposit_order（待支付）
 │      拉起三方支付台面
 │                                                                    用户支付
 T4                                     ← HTTP 回调（支付成功）←──────── 通道回调
 │                                      验签、更新 deposit_record
 │                                      → RPC: CreditBalance ────→
 │      收到入账请求
 │      执行 ModifyBalance（入账）
 │      更新 deposit_order = 成功
 │      写 transaction_log
```

**提现完整链路：**

```
时序    钱包侧                              财务侧                            三方通道
 │
 T1     用户申请提现
 │      本地校验（余额、限额、KYC）
 │      冻结可用余额
 │      创建 withdraw_order（待审核）
 │      写 freeze 流水
 │
 T2                                     B 端运营操作：
 │                                      风控审核 → 通过
 │                                      财务审核 → 通过
 │                                      出款审核 → 通过
 │                                      → RPC: MatchWithdrawChannel
 │                                      → domain/channel_gateway.CreatePayoutOrder ──→ 发起出款
 │
 T3a（驳回路径）                          任一级驳回：
 │                                      → RPC: UnfreezeBalance ──→
 │      收到解冻请求
 │      执行 ModifyBalance（解冻）
 │      更新 withdraw_order = 失败
 │
 T3b（成功路径）                          ← HTTP 回调（出款成功）←──────── 通道回调
 │                                      → RPC: DebitBalance ─────→
 │      收到扣款请求
 │      执行 ModifyBalance（扣款）
 │      更新 withdraw_order = 成功
 │
 T3c（失败路径）                          ← HTTP 回调（出款失败）←──────── 通道回调
 │                                      → RPC: UnfreezeBalance ──→
 │      收到解冻请求（同 T3a）
```

### 16.4 联调前需要双方约定的事项

| 约定事项 | 钱包侧关注 | 财务侧关注 | 影响 |
|---------|-----------|-----------|------|
| **幂等键格式** | `ref_order_no` 用什么格式？ | 生成的 platform_order_no 格式是什么？ | 钱包唯一索引依赖此键 |
| **回调参数结构** | CreditBalance/DebitBalance/UnfreezeBalance 的入参字段 | 回调后组装 RPC 请求的字段映射 | 入参不匹配会导致入账/扣款失败 |
| **错误码映射** | 钱包返回 6022xxx，财务怎么处理？ | 财务返回 60XXxxx，钱包怎么处理？ | 错误处理链路不清会导致静默失败 |
| **充值/提现状态映射** | 钱包的 DepositStatus vs 财务的 DepositStatus | 财务的状态流转更细（含审核级别） | 两边状态不一致会导致前端展示混乱 |
| **ManualAdjust 的入参** | 支持多钱包类型同时操作的参数结构 | 人工加款/减款按钱包类型拆分金额 | 影响 ManualAdjust 的实现方式 |
| **超时补偿机制** | 钱包的充值/提现超时扫描 vs 财务的超时补偿查询 | 谁先发现超时？谁主动触发？ | 双方都有 cron 扫描，需避免重复处理 |

## 十七、总结

### 一张表看懂全貌

| 层 | 目录 | 文件数 | 一句话职责 |
|----|------|--------|-----------|
| 入口 | 顶层 | 2（main + handler） | 启动服务 + 分发请求 |
| 配置 | cfg/ | 2 | MySQL + Redis 连接 |
| 契约 | errs/ + enum/ | 8 | 错误码 + 业务枚举 |
| 生成 | gen/ + gorm_gen/ | 22 | GORM 模型和查询 DSL |
| 缓存 | cache/ | 2 | 通道 + 支付方式缓存 |
| 数据 | rep/ | 10 | 每张表一个 Repository |
| 领域 | domain/ | 1 | 三方通道网关（签名 + 协议适配） |
| 业务 | ser/ | 9 | 每个业务域一个 Service |
| DDL | sql/ | 1 | 建表脚本 |
| **合计** | — | **~57** | — |

### 骨架设计的三个关键决策

1. **通道管理合并为一个 Service**——代收通道和代付通道的 CRUD 结构高度相似（字段差异 < 20%），合并在 `channel_service.go` 中避免重复，用 `ChannelDirection` 枚举区分方向
2. **新增 domain/channel_gateway.go**——三方通道 API 的协议差异是财务模块的核心复杂度，不同通道有不同的请求格式、签名方式、回调结构，统一封装避免散落在 Service 中
3. **人工修正共用一张表 + 一个 Service**——补单、加款、减款虽然是三种操作，但数据结构相同（用户 + 金额 + 原因 + 审核状态），用 `adjust_type` 字段区分，避免表和文件膨胀

### 与 ser-wallet 的定位总结

```
┌──────────────────────────────────────────────────────────┐
│                    "钱" 的完整生命周期                      │
│                                                          │
│   用户 ──→ 钱包（入口）──→ 财务（枢纽）──→ 三方通道（出口）  │
│          余额管理         通道管理+审核      真实资金流动      │
│          一致性保障        流程编排           协议适配          │
│                                                          │
│   ser-wallet 关注：                                        │
│     • 账户余额的准确性（行锁 + 事务 + 流水）                   │
│     • 用户侧的体验（展示、下单、查询）                          │
│     • 对外暴露 RPC 供其他模块调用余额操作                       │
│                                                          │
│   ser-finance 关注：                                       │
│     • 支付通道的稳定性（轮询 + 健康检查 + 预警）                 │
│     • 资金流的合规性（三级审核 + 操作日志 + 人工修正）             │
│     • 三方支付的协议适配（签名 + 回调 + 补偿查询）               │
│                                                          │
│   两者的交汇点：                                             │
│     • 充值入账（财务→钱包 CreditBalance）                     │
│     • 提现冻结/解冻/扣款（钱包↔财务 Freeze/Unfreeze/Debit）   │
│     • 人工调账（财务→钱包 ManualAdjust）                      │
│     • 支付方式查询（钱包→财务 GetPaymentMethods）              │
│     • 通道匹配（钱包→财务 MatchChannel）                      │
└──────────────────────────────────────────────────────────┘
```

> 掌握这份文档后，你在开发钱包模块时，对"充值回调后谁来调我"、"提现审核在哪里发生"、"人工加款请求从哪来"这些问题就有了清晰的答案。联调时可以对着第十六节的接口明细和时序图逐一对齐。
