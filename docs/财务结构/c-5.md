# ser-finance 工程结构设计（同事模块，联调参考）

> 视角：你是 ser-wallet 开发者，需要了熟于心同事的 ser-finance 模块结构
> 目的：搞清财务模块的代码组织方式，明确联调时"我调他什么、他调我什么、对应代码在哪里"
> 设计依据：接口评估文档（36个需实现接口 + 5个RPC调用）、表结构设计（9张表）、流程思路文档（跨模块联动全貌）、工程现有服务结构规范（ser-item/ser-buser/ser-app 四层分析）
> 立场：不是帮同事写代码，是让自己在联调时不被动 — 知道对方代码在哪、逻辑怎么走、出了问题去哪找

# 零、为什么你需要看这份文档

```
你开发 ser-wallet，同事开发 ser-finance。两个模块之间有 12 条 RPC 调用线：

  你调他 4 条:
    ① 支付配置查询     → C端进入充值/提现页面
    ② 创建渠道支付订单  → C端确认充值
    ③ 提交提现审核      → C端确认提现
    ④ 通道状态查询      → 充值前判断可用性

  他调你 7 条:
    ⑤ 充值入账          → 渠道充值回调成功后
    ⑥ 提现确认扣款      → 三级审核全部通过+渠道出款成功
    ⑦ 提现失败解冻      → 审核拒绝/出款失败
    ⑧ 人工加款          → 加款审核通过
    ⑨ 人工减款          → 减款审核通过
    ⑩ 人工补单入账      → 补单审核通过
    ⑪ 余额查询          → 创建减款单前校验

  他调外部 1 条:
    ⑫ 第三方渠道API     → 渠道下单/回调接收

不清楚对方结构 = 联调时两眼一黑
```

# 一、ser-finance 模块概况

## 1.1 基本数字

| 维度 | 数值 |
|------|------|
| 业务域 | 7 个（通道配置/支付配置/充值记录/提现记录+审核/人工修正/数据统计/RPC服务） |
| 接口数 | 36 个需实现（31 B端 + 5 RPC暴露） |

```
              ser-wallet          ser-finance
              ──────────          ───────────
  方法数       46                  36

  端           C端+B端+RPC         B端+RPC（无C端）
  IDL文件      7                   8
  外部依赖     调财务+用户+KYC     调钱包+用户+第三方渠道
  复杂度核心   余额精确性+并发     渠道对接+审核流程
```

# 三、分层详解 — 根目录层

## 3.1 main.go — 服务入口

与 ser-wallet 完全一致的启动模式：

```
lg.InitKLog() → etcd.InitEtcd() → flag.Parse(port)
  → ser.NewChannelService()
  → ser.NewPayConfigService()
  → ser.NewDepositService()
  → ser.NewWithdrawService()
  → ser.NewAdjustService()
  → ser.NewStatsService()
  → ser.NewRpcService()
  → financeservice.NewServer(&FinanceServiceImpl{...}).Run()
```

**注入 7 个 Service**：比 ser-wallet（6 个）多 1 个，因为财务的业务域更分散（通道和支付配置是独立管理页面）。

## 3.2 handler.go — RPC 纯路由层

```
单文件，36个方法，package main
所有方法一行委托，零业务逻辑
```

**方法分布**：

| 注入的Service | 方法数 | 对应IDL | 方法示例 |
|--------------|--------|---------|---------|
| channelService | 5 | channel.thrift | GetChannelList, CreateChannel, ToggleChannel |
| payConfigService | 4 | pay_config.thrift | GetPayConfigList, CreatePayConfig |
| depositService | 3 | finance_deposit.thrift | GetDepositRecords, ExportDeposits |
| withdrawService | 7 | finance_withdraw.thrift | GetWithdrawRecords, ReviewRisk, LockWithdraw |
| adjustService | 9 | adjustment.thrift | GetReplenishList, CreateCredit, ReviewDebit |
| statsService | 3 | finance_stats.thrift | GetChannelStats, GetDepositStats |
| rpcService | 5 | finance_rpc.thrift | QueryPayConfig, CreateChannelOrder |

## 4.2 ★ 与钱包关联的表 — 你必须清楚

财务的 9 张表中，有 4 张与你的 ser-wallet 存在**数据关联**：

```
  ser-wallet 表                ser-finance 表              关联方式
  ─────────────────           ─────────────────           ─────────
  deposit_order       ◄═══►   finance_deposit             order_no（平台订单号）
  withdraw_order      ◄═══►   finance_withdraw            order_no（平台订单号）
  (无对应表)          ◄────   manual_adjustment           adjust_no → 你的幂等键
  (无对应表)          ◄────   payment_config              RPC查询，不直接读表
```

### deposit_order ↔ finance_deposit 对照

```
你的 deposit_order（用户视角）         他的 finance_deposit（运营视角）
──────────────────────────            ──────────────────────────────
order_no        平台订单号              order_no         同一个
user_id         用户ID                 user_id          同一个
currency        币种                   currency         同一个
amount          充值金额               amount           同一个
status          1待支付/2成功/3失败     status           更细粒度的渠道状态
channel_order_no 渠道订单号            channel_order_no  渠道订单号
                                      channel_id        使用的通道ID（你不关心）
                                      channel_fee       通道手续费（你不关心）
                                      callback_data     渠道回调原始数据（你不关心）
bonus_activity_id 奖金活动（他不关心）
bonus_amount      奖金金额（他不关心）
```

**关键理解**：同一笔充值在两边各存一条记录。你管余额入账，他管渠道对接。通过 `order_no` 关联。

### withdraw_order ↔ finance_withdraw 对照

```
你的 withdraw_order（余额视角）        他的 finance_withdraw（审核视角）
──────────────────────────            ──────────────────────────────
order_no        平台订单号              order_no         同一个
status          1审核中/2成功/3失败     risk_status      风控审核状态
                                      finance_status   财务审核状态
                                      payout_status    出款审核状态
                                      lock_by          锁定审核人
                                      channel_id       出款通道
                                      channel_order_no 渠道出款单号
amount+fee      冻结金额               amount+fee       提现金额
```

**关键理解**：你只有 3 种终态（审核中/成功/失败），他有完整的审核流水（风控→财务→出款三级）。他审核完毕后调你的 RPC 告知结果。

### withdraw_service.go — WithdrawService（B端提现记录+三级审核） ★

```
定位: B端后台管理提现记录 + 执行风控/财务/出款三级审核
复杂度: ★★★ 高（三级审核状态机 + 锁定机制 + 渠道出款 + 回调钱包）
```

**这是 ser-finance 中最复杂的 Service，也是与你联调最密切的部分。**

| 依赖 | 用途 |
|------|------|
| FinanceWithdrawRepo | 提现记录CRUD + 审核状态更新 + 锁定/解锁 |
| WithdrawReviewRepo | 审核流水写入 |
| ChannelSelector | 出款通道选择 |
| CallbackHandler | 渠道出款回调处理 |
| **WalletRPC** | **★ 审核完成后调你的 WithdrawConfirm / WithdrawReject** |
| UserRPC | 查用户名 |
| ExportHelper | 导出 |

| 方法 | 说明 |
|------|------|
| GetWithdrawRecords | 提现记录列表 |
| GetWithdrawDetail | 提现记录详情 |
| ReviewRisk | ★ 风控审核（第一级） |
| ReviewFinance | ★ 财务审核（第二级） |
| ReviewPayout | ★ 出款审核（第三级）→ 通过后调渠道出款 → 成功后调你 |
| LockWithdraw | 锁定记录（防并发审核） |
| ExportWithdraws | 导出 |

**三级审核流程（重点理解）**：

```
你的钱包提交提现 → [RPC] → 财务创建 finance_withdraw（状态=待风控审核）
    │
    ▼
风控审核员 ReviewRisk
    ├── 通过 → 状态=待财务审核 + 写 withdraw_review 流水
    └── 拒绝 → 状态=已拒绝 + [RPC] → 你的 WithdrawReject
    │
    ▼
财务审核员 ReviewFinance
    ├── 通过 → 状态=待出款审核 + 写 withdraw_review 流水
    └── 拒绝 → 状态=已拒绝 + [RPC] → 你的 WithdrawReject
    │
    ▼
出款审核员 ReviewPayout
    ├── 通过 → 选择通道 → 调渠道API出款 → 等待渠道回调
    │           ├── 渠道成功 → [RPC] → 你的 WithdrawConfirm
    │           └── 渠道失败 → [RPC] → 你的 WithdrawReject
    └── 拒绝 → 状态=已拒绝 + [RPC] → 你的 WithdrawReject
```

**你被调用的时机**：
- `WithdrawConfirm`：只在最后一步"渠道出款成功"才触发
- `WithdrawReject`：任一级审核拒绝 或 渠道出款失败都会触发
- 每笔提现只会调你一次（通过或拒绝）

### ★ CreateChannelOrder 内部流程（你最需要理解的）

这是 C端用户确认充值后，你调财务创建渠道订单的完整内部流程：

```
你的 ser-wallet 调用: CreateChannelOrder(userId, currency, amount, method, orderNo)
    │
    └── 财务 rpc_service.go 内部处理:
        │
        ├── Step 1: 查 payment_config — 确认该充值方式有效
        │
        ├── Step 2: 获取绑定的通道ID列表
        │
        ├── Step 3: ChannelSelector 通道选择
        │   │   算法: 成功率×权重÷时间告警系数 → 加权随机选一个
        │   │   过滤: 状态=启用 + 单笔限额范围内 + 每日限额未超
        │   └── 选出最优通道
        │
        ├── Step 4: 调用第三方渠道API（HTTP）
        │   │   入参: 金额、币种、订单号、回调URL
        │   ├── USDT: 返回收款地址 + 二维码URL + 链网络
        │   └── 法币: 返回第三方支付页面跳转URL
        │
        ├── Step 5: 创建 finance_deposit 记录
        │   记录: 订单号、通道ID、渠道订单号、金额、状态=处理中
        │
        └── Step 6: 返回给你
            ├── 渠道订单号（你存到 deposit_order.channel_order_no）
            └── 支付信息（你透传给C端展示）
```

### ★ HandleDepositCallback 内部流程（渠道回调 → 调你入账）

```
第三方渠道回调: 支付成功通知
    │
    └── 财务 rpc_service.go (或 callback_handler.go) 内部处理:
        │
        ├── Step 1: 验签（通道密钥签名校验，防伪造）
        │
        ├── Step 2: 查 finance_deposit — 用渠道订单号匹配
        │
        ├── Step 3: 幂等校验 — 已处理过的直接返回成功
        │
        ├── Step 4: 更新 finance_deposit 状态 = 成功
        │   记录: 渠道回调原始数据、实际到账金额
        │
        ├── Step 5: [RPC] → 你的 ser-wallet.DepositCallback
        │   入参: order_no, actual_amount
        │   你的处理: 幂等 → 更新 deposit_order 状态 → 余额+ → 写流水
        │
        └── Step 6: 根据你的返回更新最终状态
            ├── 你返回成功 → finance_deposit 标记完成
            └── 你返回失败 → finance_deposit 标记异常，等人工处理
```

## 6.2 ★★★ wallet_rpc.go — 你最需要关注的文件

这个文件封装了财务调你的**全部 RPC 调用**。联调时出问题，优先看这里：

```
wallet_rpc.go 方法列表:

  方法                    调用你的接口              触发场景
  ───────────────────────────────────────────────────────────────
  DepositCallback()       wallet.DepositCallback    渠道充值成功回调后
  WithdrawConfirm()       wallet.WithdrawConfirm    三级审核全通过+出款成功

  ManualDebit()           wallet.ManualDebit        减款审核通过

  QueryBalance()          wallet.QueryBalance       创建减款单前
```

**每个方法的模式**：

```go
func (w *WalletRPCWrapper) DepositCallback(ctx, orderNo, amount) error {
    resp, err := rpc.WalletClient().DepositCallback(ctx, &ser_wallet.DepositCallbackReq{
        OrderNo: orderNo,
        Amount:  amount,
    })
    if err != nil {
        // RPC 调用失败 → 记录异常日志 → 标记待重试或人工介入
        return err
    }
    return nil
}
```

**联调调试时**：如果你的入账没执行，先看财务这个文件里的日志输出——确认他到底有没有调你、传了什么参数。

## 6.3 channel_selector.go — 通道选择算法

```
算法: 加权随机
输入: 支付配置绑定的通道ID列表
过滤: 状态=启用 ∧ 单笔金额在范围内 ∧ 每日累计未超限额
权重: success_rate × weight ÷ alert_coefficient
输出: 最优通道（channel_id + API配置）
```

这个算法决定了用户的充值/提现走哪个第三方渠道。你不参与这个逻辑，但需要知道：**如果用户充值失败率高，可能是通道选择的问题而不是你的代码问题。**

## 6.4 callback_handler.go — 渠道回调处理

```
职责:
  1. 接收第三方支付渠道的HTTP回调（充值成功/失败通知）
  2. 验签（通道密钥签名校验，防伪造回调）
  3. 解析回调数据（不同渠道格式可能不同）
  4. 匹配本地订单（用渠道订单号查 finance_deposit）
  5. 幂等校验（防重复回调）
  6. 更新财务记录状态
  7. 调用 wallet_rpc.DepositCallback 通知你入账

特殊说明:
  第三方回调是HTTP请求，可能需要额外的HTTP入口（Hertz端点）
  或者通过网关转发为RPC调用
  这是财务模块架构上需要确认的一个点
```

| 提现一直停在"审核中" | ser/rpc_service.go — 有没有收到 Confirm/Reject？ | ser/withdraw_service.go — 审核流程卡在哪一级？lock_by 是否没释放？ |
| 提现失败但余额没解冻 | ser/rpc_service.go WithdrawReject — 有没有收到？参数对吗？ | domain/wallet_rpc.go — 调你的 RPC 返回了什么错？ |
| 人工加款审核通过但余额没变 | ser/rpc_service.go ManualCredit — 收到了吗？幂等键(adjust_no)对得上吗？ | ser/adjust_service.go ReviewCredit — 审核通过后有调 wallet_rpc 吗？ |

## 9.3 联调顺序建议

```
阶段一：打通RPC连接
  你: common/rpc/rpc_client.go 注册 FinanceClient
  他: common/rpc/rpc_client.go 注册 WalletClient
  验证: 双方空接口能调通

阶段二：充值链路
  先调通: ① QueryPayConfig → 确认方式列表能拿到
  再调通: ② CreateChannelOrder → 确认渠道订单能创建
  最后: ⑤ DepositCallback → 他模拟回调，你验证入账

阶段三：提现链路
  先调通: ③ QueryPayConfig(type=提现) → 确认提现方式
  再调通: ④ SubmitWithdrawReview → 确认审核记录创建
  最后: ⑥⑦ WithdrawConfirm/Reject → 他走审核流程，你验证扣款/解冻

阶段四：人工修正
  ⑧⑨⑩ 人工加减款/补单 → 他创建+审核，你验证余额变动
  ⑪ QueryBalance → 他查你的余额
```

# 十、ser-finance 的枚举定义（你需要对齐的值）

联调时双方使用的枚举值必须一致。以下是 ser-finance 中你需要关心的枚举：

## 10.1 channel.go

```
ChannelType 通道类型:
  1 = 代收（充值通道）
  2 = 代付（提现通道）

ChannelStatus 通道状态:
  0 = 禁用
  1 = 启用
```

## 10.2 finance.go

```
PayConfigType 支付配置类型:
  1 = 充值
  2 = 提现
  → 你调 QueryPayConfig 时 configType 传这个值

FinanceDepositStatus 财务充值记录状态:
  1 = 处理中（等待渠道回调）
  2 = 成功（渠道确认+钱包入账）
  3 = 失败（渠道失败）
  4 = 异常（渠道成功但钱包入账失败 → 需人工介入）

WithdrawReviewLevel 审核级别:
  1 = 风控审核
  2 = 财务审核
  3 = 出款审核

WithdrawReviewResult 审核结果:
  1 = 通过
  2 = 拒绝

FinanceWithdrawStatus 财务提现总状态:
  1 = 待风控审核
  2 = 待财务审核
  3 = 待出款审核
  4 = 出款中（渠道处理中）
  5 = 成功（已回调钱包确认）
  6 = 失败（已回调钱包解冻）
  → 你的 withdraw_order 只有 1=审核中/2=成功/3=失败
  → 他的 finance_withdraw 有 6 种状态（更细粒度）

AdjustType 人工修正类型:
  1 = 补单（订单号前缀 B）
  2 = 加款（订单号前缀 A）
  3 = 减款（订单号前缀 M）

AdjustStatus 修正单状态:
  1 = 待审核
  2 = 审核通过
  3 = 审核拒绝
  4 = 执行成功（RPC调钱包成功）
  5 = 执行失败（RPC调钱包失败 → 需重试或人工）
```

# 十一、开发分期 → 联调时间窗口

## 11.1 ser-finance 预期开发分期

```
第一期：基础设施
  IDL + 生成桩代码 + 服务注册 + 建库建表 + 网关路由

第二期：通道+支付配置（自闭环）
  ChannelService + PayConfigService（纯B端CRUD）
  ★ 完成后你可以调通 QueryPayConfig

第三期：充值记录+回调处理
  DepositService + RpcService.CreateChannelOrder + 渠道回调处理
  ★ 完成后你可以调通 CreateChannelOrder
  ★ 完成后他可以调通你的 DepositCallback

第四期：提现审核+人工修正
  WithdrawService + AdjustService + 审核状态机
  ★ 完成后你可以调通 SubmitWithdrawReview
  ★ 完成后他可以调通你的 WithdrawConfirm/Reject + Manual*

第五期：统计+导出
  StatsService + 导出功能
  与你无关
```

## 11.2 联调时间窗口对照

```
              你的分期              他的分期              联调内容
              ──────              ──────              ────────
  你的第一二期  骨架+币种配置       他的第一二期          无联调（各自独立开发）
                                   基础+通道配置

  你的第三期    钱包+兑换           他的第二期            ★ 打通 QueryPayConfig
              （兑换是你独立验证     支付配置就绪            确认充值/提现方式可查
               余额操作的最佳时机）

  你的第四期    充值+提现           他的第三四期          ★★★ 全量联调
              （需要他的全部RPC）    充值回调+提现审核        充值链路端到端
                                   +人工修正              提现链路端到端
                                                         人工修正端到端
```

**关键信息**：你的第三期（钱包+兑换）不依赖财务，可以独立开发。联调集中在你的第四期，此时需要他的第二三四期全部就绪。

**风险**：如果他的进度慢于你，你的第四期（充值+提现）会被卡住。应对策略：
1. 尽早要到 `finance_rpc.thrift` 的 IDL 定义，你先基于 IDL 编码
2. 你可以写 mock 版本的 FinanceRPC 调用，在联调前用 mock 跑通本地逻辑
3. 你的 RpcService（被财务调的7个接口）可以独立开发和测试，不依赖他

# 十二、设计决策溯源

| 设计决策 | 选择 | 依据 |
|---------|------|------|
| 7 个 Service | 按业务域拆分 | 36 个方法分属 7 个独立管理页面，域间无数据共享 |
| 6 个 Repo | 按表分组 | 通道+账户共用 Repo，提现记录+审核流水共用 Repo，其余各自独立 |
| domain 层有 11 个文件 | 通道选择、回调处理、3 个 RPC 封装 | 财务模块的跨关注点（通道算法、外部渠道、钱包RPC、用户RPC）比纯CRUD多很多 |
| IDL 8 个文件 | 1 service + 7 entity | 7 个业务域各自独立 struct，RPC暴露单独一个文件（联调焦点清晰） |
| finance_rpc.thrift 独立 | RPC暴露与 B端管理分离 | 你只需要看这一个文件就知道怎么调他，降低沟通成本 |
| wallet_rpc.go 集中封装 | 所有调钱包的 RPC 在一个文件 | 联调排查时定位清晰——"他有没有调我" 只看这一个文件的日志 |
| 三级审核在 WithdrawService | 不拆独立 ReviewService | 审核是提现流程的一部分，与提现记录强绑定，拆分无收益 |
| StatsRepo 无专属表 | 聚合查询直接 GROUP BY | 初期数据量不大，实时聚合够用；后续量大再建统计表 |
| 回调处理在 domain 层 | 不在 Service 层 | 回调验签/解析是"技术关注点"不是"业务逻辑"，且可能被多个场景复用 |
