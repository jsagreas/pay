## 一、为什么要了解财务模块结构

### 1.1 联调必要性

```
你的模块(ser-wallet)和同事的模块(ser-finance)之间存在 双向 RPC 调用：

  ser-wallet ──►  ser-finance
    钱包调用财务 4 个 RPC（F1~F4）
    ├── F1 GetRechargeMethodList     充值方式列表
    ├── F2 GetWithdrawMethodList     提现方式列表
    ├── F3 GetWithdrawConfig         提现限额费率
    └── F4 SubmitWithdrawOrder       提交提现审核

  ser-finance ──►  ser-wallet
    财务调用钱包 6 个 RPC（E1~E6）
    ├── E1 GetBalance                查询余额（人工加减款弹窗）
    ├── E2 CreditWallet              加款（充值入账/补单/人工加款）
    ├── E3 DebitWallet               减款（人工减款）
    ├── E4 FreezeBalance             冻结（提现创建时由钱包侧调用，非财务直接调）
    ├── E5 UnfreezeBalance           解冻（审核驳回/出款失败）
    └── E6 ConfirmFrozenDeduction    确认扣除（出款成功）
```

### 1.2 不了解对方结构的风险

| 风险 | 具体场景 |
|------|---------|
| 接口字段对不齐 | F4(SubmitWithdrawOrder) 你传过去的字段对方要什么、返回什么 |
| 回调归属不清 | 充值回调到底走财务还是走钱包？影响你是否需要写回调处理 |
| 联调排期冲突 | 不知道对方哪些接口先开发完，你的充值/提现流程就无法跑通 |
| 错误码冲突 | 你用 6022XXX，对方用什么编号？透传错误码时需要区分 |

### 1.3 本文档的定位

```
✅ 做：基于产品需求推导财务模块的工程骨架
✅ 做：明确双方 RPC 接口的位置和归属
✅ 做：梳理财务模块内部的业务分层
⚠️ 注意：这是推测性设计，实际结构以同事的实现为准
❌ 不做：替同事做详细编码设计
```

### 2.2 与 ser-wallet 的结构对比

| 对比维度 | ser-wallet（你的模块） | ser-finance（同事模块） | 差异原因 |
|---------|----------------------|----------------------|---------|
| **服务端接口** | B端10 + C端30 + RPC13 ≈ **53** | B端~47 + RPC~4 ≈ **~51** | 财务纯B端，钱包偏C端 |
| **面向用户** | C端为主（充值/提现/兑换/记录） | B端为主（管理后台） | 业务定位不同 |
| **ser/ 文件数** | 7 个 | 8 个 | 财务子域更多（多了通道+轮询+人工修正） |
| **rep/ 文件数** | 10 个 | 11 个 | 表数量相近 |
| **cache/ 文件数** | 2 个 | 2 个 | 都是高频读取数据缓存 |
| **RPC 暴露** | 13 个（E1~E6 + R1~R7） | ~4 个（F1~F4） | 钱包被多方调用，财务主要被钱包调用 |
| **RPC 调用** | 10 个（调财务+币种+用户） | ~8 个（调钱包E1~E6 + 币种R1~R2） | 双向调用 |
| **核心难点** | 资金一致性（E1~E6） | 通道轮询算法 + 审核状态机 | 各有挑战 |

| 文件 | 包含的枚举 | 用途 |

### 4.3 errs/ — 错误码层

```
internal/errs/
└── code.go             # 全部错误码（60XXYYY — 服务编号待分配，假设023）
```

**预估错误码分段**：

| 段范围 | 所属模块 | 典型错误码 |
|--------|---------|-----------|
| 6023000~6023099 | 通道管理 | ChannelNotFound、ChannelNameDuplicate |
| 6023100~6023199 | 支付配置 | MethodNotFound、MethodChannelConflict |
| 6023200~6023299 | 充值业务 | NoAvailableChannel、CallbackVerifyFailed |
| 6023300~6023399 | 提现业务 | AuditPermissionDenied、DisbursementFailed |
| 6023400~6023499 | 人工修正 | SameAuditorNotAllowed、OriginalOrderNotFound |
| 6023500~6023599 | 数据统计 | InvalidDateRange |

> 服务编号假设为 023（接续 022 wallet）。实际编号以同事注册时确定。
> **联调注意**：财务返回的错误码前缀与你的 6022XXX 不同，透传给前端时需区分来源。

### 4.5 cache/ — Redis 缓存层

```
internal/cache/
├── channel_cache.go        # 通道配置 + 通道状态缓存
└── method_cache.go         # 充值方式 + 提现方式缓存
```

| 文件 | 缓存内容 | 缓存策略 | 被谁读取 |
|------|---------|---------|---------|
| channel_cache.go | 代收/代付通道列表及配置、通道启禁用状态 | 配置变更时主动失效 + TTL兜底 | polling_service（轮询算法高频读取通道数据） |
| method_cache.go | 各币种的充值方式列表、提现方式列表 | 配置变更时失效 + TTL兜底 | recharge_service/withdraw_service（F1/F2 RPC被钱包调用时返回） |

> **联调关系**：钱包调 F1/F2 时，财务模块从 method_cache 返回数据。如果缓存未更新，钱包拿到的是旧数据。

#### 通道轮询算法（polling_service.go）详解

> 这是财务模块中最复杂的算法逻辑

```
┌────────────────────────────────────────────────────────────────┐
│ 通道轮询算法                                                     │
│                                                                 │
│ 核心公式：                                                       │
│   轮询优先级 = W1×成功率系数 + W2×时间系数 + W3×预警系数            │
│                                                                 │
│ 其中：                                                           │
│   W1 + W2 + W3 = 100%（默认 50%:30%:20%）                       │
│                                                                 │
│ 成功率系数：                                                      │
│   ① 单位时间（分钟）内统计通道成功率                                │
│   ② 未达最小样本量 → 不计算                                       │
│   ③ 成功率落在哪个区间 → 查系数对照表得到系数值                      │
│                                                                 │
│ 时间系数：                                                        │
│   ① 单位时间内的平均完成时间                                       │
│   ② 未达最小样本量 → 默认0                                        │
│   ③ 平均时间(秒)落在哪个区间 → 查系数对照表                         │
│                                                                 │
│ 预警系数：                                                        │
│   ① 通道累计金额 ≥ 预警金额 → 触发预警系数                          │
│   ② 未达预警值 → 默认0                                            │
│                                                                 │
│ 调用时机：                                                        │
│   充值时 → 轮询代收通道                                            │
│   提现出款时 → 轮询代付通道                                        │
│   代收轮询和代付轮询结构完全一致                                    │
└────────────────────────────────────────────────────────────────┘
```

### 6.3 回调端点（设计待确认）

> 第三方支付通道的充值/出款回调需要 HTTP 端点接收。可能的实现方式：

| 方案 | 实现方式 | 优缺点 |
|------|---------|--------|
| 方案A | 通过 gate-font/gate-back 的 Open 路由透传 | 复用已有基础设施，但回调可能不走这两个网关 |
| 方案B | 独立回调网关 | 隔离回调和正常流量，但增加部署复杂度 |
| 方案C | ser-finance 内嵌 Hertz HTTP Server | 服务内同时跑 Kitex + Hertz，工程中无先例 |

> **联调影响**：如果回调走财务模块，充值成功后由财务调用你的 E2(CreditWallet) 入账。如果回调走钱包模块（如USDT链上确认），则你自己处理入账。评审时需确认。

### 8.2 订单表归属的设计决策

> **这是联调中最需要和同事对齐的架构问题之一**

| 问题 | 方案A（各自存储） | 方案B（共享数据库） |
|------|-----------------|------------------|
| 充值订单 | 钱包存 recharge_order（用户视角），财务存 recharge_order（通道视角） | 两个服务读写同一张表 |
| 提现订单 | 钱包存 withdraw_order（用户视角），财务存 withdraw_order（审核视角） | 两个服务读写同一张表 |
| 优点 | 服务解耦彻底，各自DB各自维护 | 数据一致性天然保证，无需同步 |
| 缺点 | 需要同步机制（RPC调用或消息队列） | 耦合度高，两个服务写同一张表容易冲突 |
| 工程先例 | 更符合微服务独立DB的理念 | 小团队快速开发时常见 |

> **建议评审时确认此点。** 如果各自存储：
> - 钱包订单 → 用户ID、币种、金额、状态（用户可见的3状态）
> - 财务订单 → 用户ID、币种、金额、通道信息、审核记录、详细7状态

### 8.3 通道表字段概要

| 字段分类 | 代收通道 | 代付通道 |
|---------|---------|---------|
| 基本信息 | 币种、通道名称（唯一）、支付类型 | 同 |
| 费率配置 | 代收费率（1位小数%） | 代付费率 + **单笔手续费** |
| 限额配置 | 单笔限额（最低~最高） | 同 |
| 权重排序 | 代收权重（整数） | 代付权重 |
| 预警 | 预警金额（暂不做） | 同 |
| 状态 | 启用/禁用（实时生效） | 同 |
| 密钥配置 | 请求地址、商户ID、密钥、AppKey、支付编码、平台公钥、公钥、私钥 | 同 |

> **入库方式**：通道由工程师代码入库（不是后台新增），B端后台仅做编辑和启禁用操作。

### 9.3 联调优先级

| 优先级 | 联调接口 | 原因 | 阻塞方 |
|--------|---------|------|--------|
| **P0** | F4 SubmitWithdrawOrder | 提现是最复杂的链路，双向调用最多 | 双向阻塞 |
| **P0** | E2 CreditWallet | 充值入账的唯一途径 | 钱包先完成，财务来调 |
| **P0** | E5 UnfreezeBalance + E6 ConfirmFrozenDeduction | 提现审核结果处理 | 钱包先完成，财务来调 |
| **P1** | F1 GetRechargeMethodList | 充值方式展示 | 财务先完成，钱包来调 |
| **P1** | F2 + F3 | 提现方式+限额 | 财务先完成，钱包来调 |
| **P1** | E1 GetBalance | 人工加减款查余额 | 钱包先完成，财务来调 |
| **P1** | E3 DebitWallet | 人工减款 | 钱包先完成，财务来调 |

> **关键结论**：钱包模块的 E1~E6 必须先于财务模块完成开发（财务依赖），而财务的 F1~F4 必须先于钱包的充值/提现流程完成。**建议两边同时启动，优先锁定 IDL 接口定义。**

## 十一、总结

### 11.1 关键认知

| 认知点 | 内容 |
|--------|------|
| **财务是纯B端模块** | 没有 C 端 HTTP 接口，C端数据全部由钱包通过 RPC 获取后返回前端 |
| **财务的核心算法** | 通道轮询（成功率×权重+时间×权重+预警×权重），直接影响充值/提现的通道选择 |
| **双向 RPC 依赖** | 你调 F1~F4（配置数据+提交审核），对方调 E1~E6（资金操作） |
| **回调归属待确认** | 银行/扫码回调大概率走财务 → 财务调 E2 入账；USDT链上回调可能直接到钱包 |
| **审核人≠提交人** | 人工修正必须双人操作（提交+审核不能是同一人），这是财务模块的硬规则 |
| **订单号规范** | 充值C+16/提现T+16/补单B+16/加款A+16/减款M+16，全系统唯一 |
| **导出需二次验证** | 充值/提现记录导出需验证后台密码（+可选谷歌验证码） |
| **统计时区** | UTC基准时间存储，按站点时区展示，查询时需要时区转换 |

### 11.2 你需要和同事提前对齐的事项

| # | 对齐项 | 优先级 | 原因 |
|---|--------|--------|------|
| 1 | **F1~F4 + E1~E6 的 IDL 字段定义** | P0 | 双方 RPC 接口的"合同"，不锁定无法编码 |
| 2 | **订单表归属方案** | P0 | 钱包各存一份 vs 共享DB，影响整体架构 |
| 3 | **充值回调归属** | P0 | 决定你是否需要写回调处理代码 |
| 4 | **服务编号** | P1 | 错误码前缀不能冲突（你的 6022 vs 对方的 60XX） |
| 5 | **开发优先级对齐** | P1 | E1~E6 和 F1~F4 哪边先完成，影响联调排期 |
| 6 | **F4 失败和超时处理** | P1 | 提交提现审核失败/超时时的回滚策略 |
| 7 | **通道匹配调用方式** | P2 | 充值时钱包直接请求三方 or 通过财务中转 |
