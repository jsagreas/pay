# ser-wallet IDL 设计方案

> 来源：12 份 c-4 分析文档 + 工程全量 IDL 扫描（20 个服务、70 个 thrift 文件逐一分析）
> 定位：IDL 先行——在编码之前，确定接口定义的文件拆分、命名规范、结构设计
> 原则：① 严格遵循现有工程 IDL 规范 ② 不过于死板（细节可随需求调整）③ 有依据有事实
> 范围：钱包模块（C端）+ 币种配置（B端）+ 供财务调用的 RPC 接口

---

## 一、现有工程 IDL 规范提炼

在设计之前，先从 20 个已有服务中提炼出已经稳定执行的规范，这是设计的基线。

### 1.1 已确认的硬性规范（必须遵循）

| 规范项 | 规则 | 来源 |
|--------|------|------|
| 目录命名 | `common/idl/ser-wallet/` | readme.md："文件夹名与微服务名保持一致" |
| namespace | `namespace go ser_wallet`（所有文件统一） | 全部 20 个服务均如此 |
| Service 命名 | `WalletService` | 去 `ser-` 前缀 + 首字母大写 + `Service` |
| 入口文件 | `service.thrift` | 所有服务唯一入口 |
| Service 唯一性 | 一个服务只有一个 `service WalletService {}` | 全部服务均如此 |
| struct 分离 | `service.thrift` 中不放任何 struct 定义 | 全部服务均如此 |
| 方法签名 | `module.XxxResp MethodName(1: module.XxxReq req)` | ser-app/ser-item/ser-live 等主流模式 |
| 方法名 | PascalCase 大驼峰 | 全部服务均如此 |
| Req/Resp 命名 | `XxxReq` / `XxxResp` | 主流且推荐 |
| 字段编号 | 从 1 开始连续递增 | 全部 |
| ID/时间戳 | `i64` | 全部 |
| 状态/类型/枚举值 | `i32` + 行尾注释 | 主流，不用 enum |
| 金额 | `i64`（最小单位整数） | 架构设计文档决定 |
| 同目录引用 | `include "xxx.thrift"` | 全部 |
| 跨服务引用 | `include "../ser-xxx/xxx.thrift"` | 全部 |
| 公共枚举 | `include "../common/enum.thrift"` | ser-app/ser-buser/ser-kyc/ser-mall 等 |

### 1.2 分页模式（统一模板）

```thrift
// 请求
struct PageXxxReq {
    // ... 业务筛选字段在前
    N: optional i32 pageNo;       // 页码，从1开始
    N+1: optional i32 pageSize;   // 每页条数
}

// 响应
struct PageXxxResp {
    1: list<XxxItem> pageList;    // 数据列表
    2: i64 total;                 // 总记录数
    3: i32 pageNo;                // 当前页码
    4: i32 pageSize;              // 每页记录数
}
```

### 1.3 required/optional 使用规则

```
Create 请求：必填字段 required，可选字段 optional
Update 请求：ID 用 required，可更新字段看业务
Page/Query 请求：筛选字段全部 optional
Delete/Status 请求：ID 用 required
Response：一般不标注
RPC供他接口：关键入参 required（如 userId、orderId）
```

### 1.4 注释规范

```thrift
// ===== 模块分组标题 =====
// 中文说明
module.XxxResp MethodName(1: module.XxxReq req)
```

### 1.5 CRUD 方法命名词汇表

| 操作 | 前缀 | 示例 |
|------|------|------|
| 创建 | `Create` | `CreateCurrency` |
| 编辑 | `Update` | `UpdateCurrency` |
| 删除 | `Delete` | `DeleteWithdrawalAccount` |
| 状态切换 | `Status` / `Toggle` | `StatusCurrency` |
| 分页查询（B端） | `Page` | `PageCurrency` |
| 列表查询（C端） | `List` / `Get...List` | `ListCurrency` |
| 详情 | `Detail` / `Get` | `DetailDepositOrder` |
| 批量获取 | `BatchGet` | `BatchGetWalletBalance` |
| 执行动作 | 动词 | `ExecuteExchange`, `FreezeForWithdrawal` |

## 二、文件拆分策略：设计多少个 IDL 文件

### 2.1 拆分依据

文件拆分不是随意的，需要有清晰的依据：

**依据一：现有工程的拆分粒度规律**

| 服务 | 接口数 | 文件数 | 拆分粒度 |
|------|--------|--------|----------|
| ser-app | ~15 | 5 | 按功能模块（banner/s3/home_category/content_strategy） |
| ser-buser | ~20 | 4 | 按实体（sys_menu/sys_role/sys_user） |
| ser-item | ~12 | 4 | 按功能（item/item_publish/s3） |
| ser-live | ~40 | 9 | 按模块+调用方（live_room/live_api/live_rpc 等） |
| ser-user | ~50 | 13 | 最细粒度（每个子域一个文件） |

规律：**接口越多，拆分越细。小服务 2-4 个文件，中大服务 4-9 个文件，超大服务 10+ 个文件。**

**依据二：ser-wallet 的接口规模**

根据接口评估文档：
- 币种配置域（B端）：~8 个接口
- 钱包域（C端）：~18 个接口
- 合计：~26 个接口

属于"中大服务"级别，参考 ser-live（~40 接口 / 9 文件）和 ser-buser（~20 接口 / 4 文件）。

**依据三：业务域边界**

ser-wallet 有两个明确的业务域：
1. **币种配置域**（B端管理）— 独立的管理功能，面向运营
2. **钱包域**（C端操作 + RPC 供他）— 面向用户 + 面向其他服务

钱包域内部又有自然的子分组：
- 基础能力（余额、币种列表）
- 充值
- 兑换
- 提现
- 交易记录
- RPC 供财务

### 2.2 拆分方案对比

#### 方案 A：4 文件（粗粒度）

```
service.thrift        ← Service 定义
currency.thrift       ← 币种配置域全部
wallet.thrift         ← 钱包域全部（充值+兑换+提现+余额）
rpc.thrift            ← RPC 供财务全部
```

- 优点：文件少，结构简单
- 缺点：wallet.thrift 会非常大（~18 个接口的 Req/Resp 结构体 ≈ 500-700 行），不利于协作和查找

#### 方案 B：6 文件（中粒度）⭐ 推荐

```
service.thrift        ← Service 定义
currency.thrift       ← 币种配置域
wallet.thrift         ← 钱包基础 + 充值 + 兑换 + 提现
transaction.thrift    ← 交易记录
rpc.thrift            ← RPC 供财务调用
model.thrift          ← 公共子结构体（被多个文件复用的）
```

- 优点：每个文件 150-300 行，可控；业务分组清晰；公共结构体不重复定义
- 缺点：比最精简多 2 个文件

#### 方案 C：8 文件（细粒度）

```
service.thrift
currency.thrift       ← 币种配置
wallet.thrift         ← 钱包基础（余额查询、币种列表）
deposit.thrift        ← 充值
exchange.thrift       ← 兑换
withdrawal.thrift     ← 提现
transaction.thrift    ← 交易记录
rpc.thrift            ← RPC 供财务
```

- 优点：每个文件最短（100-200 行），按功能精确定位
- 缺点：文件过多，充值/兑换/提现之间有共用的结构（如 WalletType、CurrencyInfo），可能导致跨文件引用增多

### 2.3 选择方案 B 的理由

| 考量 | 方案 B 的优势 |
|------|-------------|
| 工程规模匹配 | ~26 个接口 / 6 文件 ≈ 每文件 4-5 个接口，与 ser-app（15 接口 / 5 文件）、ser-item（12 接口 / 4 文件）的密度一致 |
| 内聚性 | 充值/兑换/提现都是"钱包核心操作"，放在同一个 wallet.thrift 中内聚度高。它们共用 WalletType、CurrencyCode 等概念，放一起减少跨文件引用 |
| model.thrift 避免重复 | 多个文件都需要用到的子结构体（如 CurrencyBrief、WalletInfo）抽到 model.thrift，不重复定义 |
| 参考先例 | ser-live 用 `live_rpc.thrift` 单独放 RPC 供他接口；ser-user 用 `user_rpc.thrift` 做同样的事。我们的 `rpc.thrift` 沿袭此模式 |
| 可维护性 | 6 个文件不多不少，每个文件职责清晰，改一个不影响其他 |
| 弹性空间 | 如果后续充值/提现变得特别复杂（比如多了 10 几个接口），可以从 wallet.thrift 拆出 deposit.thrift / withdrawal.thrift，渐进演进而不是一开始就过度拆分 |

## 三、6 个 IDL 文件职责定义

### 3.1 文件清单与职责

```
common/idl/ser-wallet/
├── service.thrift          ← 入口：WalletService 定义 + include + 方法列表
├── model.thrift            ← 公共子结构体：被多个文件复用的结构
├── currency.thrift         ← 币种配置域：B端 CRUD + RPC 供他查询
├── wallet.thrift           ← 钱包核心域：余额 + 充值 + 兑换 + 提现
├── transaction.thrift      ← 交易记录：C端记录列表 + 详情
└── rpc.thrift              ← RPC 供财务：余额变更相关的被调接口
```

### 3.2 每个文件的详细职责

#### (1) service.thrift — 服务入口

**只做两件事**：include 所有子文件 + 定义 WalletService 方法列表。不包含任何 struct。

```
职责：
  - namespace 声明
  - include 5 个子文件 + common/enum.thrift
  - 定义 WalletService，按域分组列出所有方法
  - 方法按 ===== 分组标题 ===== 归类

预估行数：~80 行
```

#### (2) model.thrift — 公共子结构体

**为什么需要这个文件**：有些结构体会被 currency.thrift、wallet.thrift、transaction.thrift 同时用到。如果不抽出来，要么重复定义，要么产生循环 include。

```
职责：
  - 被多个文件 include 的公共小结构体
  - 不包含任何 Req/Resp，只包含 "嵌套用的子 struct"

预估内容：
  - CurrencyBrief    ← 币种摘要（code/name/symbol/icon/decimals），余额查询、充值、提现、记录都要用
  - WalletBrief      ← 钱包摘要（walletType/balance/frozenAmount），余额查询响应中的子结构
  - OrderBrief       ← 订单摘要（orderNo/type/status/amount/time），记录列表中的子结构

预估行数：~60-80 行

参考先例：
  - ser-s3/model.thrift（公共 Model 结构）
  - ser-blog/model.thrift（公共 Model 结构）
  - ser-cron/model.thrift（公共 Model 结构）
```

#### (3) currency.thrift — 币种配置域

```
职责：
  - 币种 CRUD 的 Req/Resp（B端）
  - 汇率日志查询的 Req/Resp（B端）
  - 币种列表/汇率/基准币种查询的 Req/Resp（RPC 供他 + C端）

对应接口（~10 个）：
  B端管理：
    - CreateCurrency     新增币种
    - UpdateCurrency     编辑币种
    - StatusCurrency     启用/禁用
    - SetBaseCurrency    设置基准币种
    - PageCurrency       币种列表（分页）
    - PageRateLog        汇率日志（分页）

  RPC 供他 + C端：
    - GetCurrencyList      可用币种列表
    - GetCurrencyByCode    按代码查币种详情
    - GetExchangeRate      查汇率
    - GetBaseCurrency      查基准币种

struct 清单：
  CreateCurrencyReq / CreateCurrencyResp
  UpdateCurrencyReq / UpdateCurrencyResp
  StatusCurrencyReq / StatusCurrencyResp
  SetBaseCurrencyReq / SetBaseCurrencyResp
  PageCurrencyReq / PageCurrencyResp + CurrencyItem
  PageRateLogReq / PageRateLogResp + RateLogItem
  GetCurrencyListReq / GetCurrencyListResp
  GetCurrencyByCodeReq / GetCurrencyByCodeResp
  GetExchangeRateReq / GetExchangeRateResp
  GetBaseCurrencyResp

预估行数：~200-250 行
```

#### (4) wallet.thrift — 钱包核心域

```
职责：
  - 余额查询的 Req/Resp
  - 充值相关的 Req/Resp
  - 兑换相关的 Req/Resp
  - 提现相关的 Req/Resp
  - 提现账户管理的 Req/Resp

对应接口（~14 个）：
  余额：
    - GetWalletBalance         查余额
    - GetAvailableCurrencies   C端可用币种（带余额摘要）

  充值：
    - GetDepositMethods        充值方式查询
    - CreateDepositOrder       创建充值订单
    - GetUSDTDepositInfo       USDT充值信息
    - GetBonusActivities       奖金活动列表

  兑换：
    - GetExchangePreview       兑换预览（汇率+预估）
    - ExecuteExchange          执行兑换

  提现：
    - GetWithdrawalInfo        提现信息（限额+方式）
    - CreateWithdrawalOrder    创建提现订单
    - SaveWithdrawalAccount    保存提现账户
    - GetWithdrawalAccounts    查提现账户列表
    - DeleteWithdrawalAccount  删除提现账户

  辅助：
    - GetDepositOrderStatus    充值订单状态（轮询）

struct 清单：
  GetWalletBalanceReq / GetWalletBalanceResp
  GetAvailableCurrenciesResp + CurrencyWithBalance
  GetDepositMethodsReq / GetDepositMethodsResp + DepositMethod + AmountOption
  CreateDepositOrderReq / CreateDepositOrderResp
  GetUSDTDepositInfoReq / GetUSDTDepositInfoResp
  GetBonusActivitiesReq / GetBonusActivitiesResp + BonusActivity
  GetExchangePreviewReq / GetExchangePreviewResp
  ExecuteExchangeReq / ExecuteExchangeResp
  GetWithdrawalInfoReq / GetWithdrawalInfoResp + WithdrawalMethod
  CreateWithdrawalOrderReq / CreateWithdrawalOrderResp
  SaveWithdrawalAccountReq / SaveWithdrawalAccountResp
  GetWithdrawalAccountsReq / GetWithdrawalAccountsResp + WithdrawalAccountItem
  DeleteWithdrawalAccountReq / DeleteWithdrawalAccountResp
  GetDepositOrderStatusReq / GetDepositOrderStatusResp

预估行数：~350-450 行
```

**为什么充值/兑换/提现不拆分成 3 个文件**：

1. 它们共用大量概念（WalletType、CurrencyCode、userId 从 Context 获取等），放一起引用方便
2. 每个子功能的接口数不多（充值 4 个、兑换 2 个、提现 5 个），单独成文件太碎
3. 参考 ser-item：item 配置 + item 发布虽然是两个子域，但接口少的那个（item_publish ~4 个接口）仍然单独成文件，而我们这里每个子域也是 2-5 个，如果拆成 3 个文件每个只有 100-150 行，显得过于碎片化
4. 如果后续某个子域暴增（比如提现增加了 7 个新接口），再拆出来也很简单，渐进演进

#### (5) transaction.thrift — 交易记录

```
职责：
  - C端记录列表查询的 Req/Resp
  - 记录详情的 Req/Resp

对应接口（~2 个）：
  - GetTransactionRecords    记录列表（充值/提现/兑换/奖励，按类型筛选）
  - GetTransactionDetail     记录详情

struct 清单：
  GetTransactionRecordsReq / GetTransactionRecordsResp + TransactionItem
  GetTransactionDetailReq / GetTransactionDetailResp

预估行数：~80-120 行
```

**为什么单独一个文件**：
1. 交易记录是独立的查询功能，不涉及余额变更逻辑
2. ser-item 把 item 和 item_publish 分开，交易记录和钱包操作也有类似的"管理 vs 查询"区分
3. 后续可能增加 B 端的流水查询（管理后台看用户流水），独立文件方便扩展

#### (6) rpc.thrift — RPC 供财务调用

```
职责：
  - 被 ser-finance 调用的接口的 Req/Resp
  - 这些接口的共同特点：需要显式传 userId（不是从 Context 获取），金额操作需要幂等键

对应接口（~8 个）：
  余额查询：
    - GetUserWalletBalance       查指定用户余额（传 userId）

  充值上账：
    - CreditDeposit              充值入账（三方回调后财务调钱包）

  提现：
    - FreezeForWithdrawal        冻结提现金额
    - UnfreezeForWithdrawal      解冻（提现取消/审核拒绝）
    - ConfirmWithdrawalDeduct    确认扣款（出款成功）

  人工操作：
    - ManualCredit               人工加款
    - ManualDebit                人工减款

  差错调整：
    - CreditCorrection           差错修正（充值补单等）

struct 清单：
  GetUserWalletBalanceReq / GetUserWalletBalanceResp
  CreditDepositReq / CreditDepositResp
  FreezeForWithdrawalReq / FreezeForWithdrawalResp
  UnfreezeForWithdrawalReq / UnfreezeForWithdrawalResp
  ConfirmWithdrawalDeductReq / ConfirmWithdrawalDeductResp
  ManualCreditReq / ManualCreditResp
  ManualDebitReq / ManualDebitResp
  CreditCorrectionReq / CreditCorrectionResp

预估行数：~200-250 行
```

**为什么单独一个文件**：
1. **调用方不同**：wallet.thrift 面向 C 端用户（通过 gate-font），rpc.thrift 面向其他服务（通过 Kitex RPC）
2. **参数模式不同**：C 端接口的 userId 从 Context 获取，RPC 接口需要显式传 userId
3. **安全等级不同**：RPC 供他接口涉及直接余额操作，需要更严格的参数校验（幂等键 required、操作人 required 等）
4. **工程先例**：ser-user 的 `user_rpc.thrift`、ser-live 的 `live_rpc.thrift` 都是同样的做法——把"供其他服务调用"的接口独立成文件

## 四、service.thrift 完整骨架

```thrift
namespace go ser_wallet

include "model.thrift"
include "currency.thrift"
include "wallet.thrift"
include "transaction.thrift"
include "rpc.thrift"
include "../common/enum.thrift"

service WalletService {

    // ===== 币种配置（B端管理） =====
    // 新增币种
    currency.CreateCurrencyResp CreateCurrency(1: currency.CreateCurrencyReq req)
    // 编辑币种
    currency.UpdateCurrencyResp UpdateCurrency(1: currency.UpdateCurrencyReq req)
    // 启用/禁用币种
    currency.StatusCurrencyResp StatusCurrency(1: currency.StatusCurrencyReq req)
    // 设置基准币种
    currency.SetBaseCurrencyResp SetBaseCurrency(1: currency.SetBaseCurrencyReq req)
    // 币种列表（B端分页）
    currency.PageCurrencyResp PageCurrency(1: currency.PageCurrencyReq req)
    // 汇率日志列表（B端分页）
    currency.PageRateLogResp PageRateLog(1: currency.PageRateLogReq req)

    // ===== 币种配置（RPC 供他 + C端） =====
    // 可用币种列表
    currency.GetCurrencyListResp GetCurrencyList(1: currency.GetCurrencyListReq req)
    // 按代码查币种
    currency.GetCurrencyByCodeResp GetCurrencyByCode(1: currency.GetCurrencyByCodeReq req)
    // 查汇率
    currency.GetExchangeRateResp GetExchangeRate(1: currency.GetExchangeRateReq req)
    // 查基准币种
    currency.GetBaseCurrencyResp GetBaseCurrency()

    // ===== 钱包（余额） =====
    // 查余额
    wallet.GetWalletBalanceResp GetWalletBalance(1: wallet.GetWalletBalanceReq req)
    // C端可用币种（带余额摘要）
    wallet.GetAvailableCurrenciesResp GetAvailableCurrencies()

    // ===== 钱包（充值） =====
    // 充值方式查询
    wallet.GetDepositMethodsResp GetDepositMethods(1: wallet.GetDepositMethodsReq req)
    // 创建充值订单
    wallet.CreateDepositOrderResp CreateDepositOrder(1: wallet.CreateDepositOrderReq req)
    // USDT 充值信息
    wallet.GetUSDTDepositInfoResp GetUSDTDepositInfo(1: wallet.GetUSDTDepositInfoReq req)
    // 奖金活动列表
    wallet.GetBonusActivitiesResp GetBonusActivities(1: wallet.GetBonusActivitiesReq req)
    // 充值订单状态
    wallet.GetDepositOrderStatusResp GetDepositOrderStatus(1: wallet.GetDepositOrderStatusReq req)

    // ===== 钱包（兑换） =====
    // 兑换预览
    wallet.GetExchangePreviewResp GetExchangePreview(1: wallet.GetExchangePreviewReq req)
    // 执行兑换
    wallet.ExecuteExchangeResp ExecuteExchange(1: wallet.ExecuteExchangeReq req)

    // ===== 钱包（提现） =====
    // 提现信息（限额+方式）
    wallet.GetWithdrawalInfoResp GetWithdrawalInfo(1: wallet.GetWithdrawalInfoReq req)
    // 创建提现订单
    wallet.CreateWithdrawalOrderResp CreateWithdrawalOrder(1: wallet.CreateWithdrawalOrderReq req)
    // 保存提现账户
    wallet.SaveWithdrawalAccountResp SaveWithdrawalAccount(1: wallet.SaveWithdrawalAccountReq req)
    // 查提现账户列表
    wallet.GetWithdrawalAccountsResp GetWithdrawalAccounts(1: wallet.GetWithdrawalAccountsReq req)
    // 删除提现账户
    wallet.DeleteWithdrawalAccountResp DeleteWithdrawalAccount(1: wallet.DeleteWithdrawalAccountReq req)

    // ===== 交易记录 =====
    // 记录列表
    transaction.GetTransactionRecordsResp GetTransactionRecords(1: transaction.GetTransactionRecordsReq req)
    // 记录详情
    transaction.GetTransactionDetailResp GetTransactionDetail(1: transaction.GetTransactionDetailReq req)

    // ===== RPC 供财务调用 =====
    // 查指定用户余额
    rpc.GetUserWalletBalanceResp GetUserWalletBalance(1: rpc.GetUserWalletBalanceReq req)
    // 充值入账
    rpc.CreditDepositResp CreditDeposit(1: rpc.CreditDepositReq req)
    // 冻结提现金额
    rpc.FreezeForWithdrawalResp FreezeForWithdrawal(1: rpc.FreezeForWithdrawalReq req)
    // 解冻提现金额
    rpc.UnfreezeForWithdrawalResp UnfreezeForWithdrawal(1: rpc.UnfreezeForWithdrawalReq req)
    // 确认提现扣款
    rpc.ConfirmWithdrawalDeductResp ConfirmWithdrawalDeduct(1: rpc.ConfirmWithdrawalDeductReq req)
    // 人工加款
    rpc.ManualCreditResp ManualCredit(1: rpc.ManualCreditReq req)
    // 人工减款
    rpc.ManualDebitResp ManualDebit(1: rpc.ManualDebitReq req)
    // 差错修正
    rpc.CreditCorrectionResp CreditCorrection(1: rpc.CreditCorrectionReq req)

    // ===== 公共枚举 =====
    // 获取枚举选项
    enum.GetEnumOptionsResp GetEnumOptions(1: enum.GetEnumOptionsReq req)
}
```

**方法总计：26 个业务接口 + 1 个公共枚举 = 27 个**

## 六、关键设计决策与理由

### 6.1 为什么 C 端接口不传 userId

```
现有工程模式（已验证）：
  gate-font → FontAuthMiddleware 校验 Token → 注入 user_id 到 Context
  gate-font → Kitex RPC 调用 → TTHeader 透传 X-User-ID
  ser-* → tracer.GetUserId(ctx) 获取

所以 C 端接口（wallet.thrift）的 Req 不需要 userId 字段。
而 RPC 供他接口（rpc.thrift）的 Req 必须显式传 userId，因为调用方是 ser-finance，不是用户。
```

### 6.2 为什么金额字段用 i64 而不是 string 或 double

```
i64（最小单位整数）：
  - Go 映射为 int64，计算无精度损失
  - Thrift 编解码效率最高（固定 8 字节）
  - 与数据库 bigint 直接对应，无需转换
  - 架构设计文档已确定此方案

double 的问题：
  - 0.1 + 0.2 ≠ 0.3，金融场景不可接受

string 的问题：
  - 每次计算都需要 parse，性能差
  - 需要额外校验格式合法性
```

### 6.3 为什么枚举用 i32 + 注释而不用 Thrift enum

```
工程现状：
  20 个服务中只有 1 个（ser-contentcenter）使用了 Thrift enum
  其余全部用 i32 + 行尾注释

原因：
  - Thrift enum 生成的 Go 代码是 int64 类型别名，不如自定义 Go enum 灵活
  - 项目统一在 Go 层面定义 enum（internal/enum/*.go），提供校验+展示函数
  - IDL 层只关心传输类型（i32），语义由注释说明
  - 保持一致性，不破例
```

### 6.4 为什么 RPC 供他接口的 Resp 统一返回 balanceAfter

```
调用方（ser-finance）的典型使用场景：
  1. 调钱包 CreditDeposit 充值入账
  2. 拿到 balanceAfter 记录到自己的充值记录表
  3. 用于财务报表、对账

如果不返回，ser-finance 还要再调一次 GetUserWalletBalance 查余额，多一次 RPC。
所以在响应中直接返回变更后的余额，减少跨服务调用。

这也是金融系统的惯例——每次余额操作的结果都应该包含"操作后余额"。
```

### 6.5 为什么 rpc.thrift 所有请求都 required idempotencyKey

```
RPC 供财务调用的接口全部涉及余额变更，是资金安全的重灾区：
  - 网络超时重试可能导致重复调用
  - 财务审核通过后触发余额操作，如果重复会多扣/多加钱

idempotencyKey 设为 required 是强制约束：
  - 调用方必须传，不传编译不过
  - 服务端用它做幂等校验（数据库唯一键 + Redis 去重）
  - 不是 optional，不给"忘传"的机会

C 端接口的 idempotencyKey 设为 optional：
  - 充值/提现由服务端生成订单号作为幂等标识
  - 前端可以额外传一个防重复提交的 key
```

### 6.6 为什么 GetBaseCurrency() 没有 Req 参数

```
基准币种是全局唯一的配置（一次设置后不可更改），查询不需要任何参数。
Thrift 允许无参方法，工程中已有先例（虽然不多）。

如果后续需要加参数（比如按环境区分），加一个 Req 即可，兼容性无问题。
但目前确实不需要参数，不为了"预留"而加空 Req。
```

## 七、跨服务引用分析

### 7.1 ser-wallet 需要 include 的外部文件

```
include "../common/enum.thrift"    ← 公共枚举查询接口（GetEnumOptions）
```

只需要引用这一个外部文件。币种配置、钱包、交易记录的结构体全部自包含。

### 7.2 其他服务需要 include ser-wallet 的场景

```
ser-finance 如果要在自己的 IDL 中引用钱包的类型：
  include "../ser-wallet/rpc.thrift"     ← 引用 CreditDepositReq 等

但实际上 ser-finance 是作为 RPC 客户端调用 ser-wallet 的接口，
不需要在自己的 IDL 中 include 钱包的 thrift 文件。

Kitex 客户端直接使用 kitex_gen/ser_wallet/ 下生成的 Go 类型即可。
所以 ser-wallet 的 IDL 不会被其他服务的 IDL 直接 include。

除非将来有新的服务需要在 IDL 层面引用钱包的 struct（如 ser-report 的报表需要钱包交易结构），
到时候再按需 include。
```

### 7.3 跨服务调用的 Go 代码层面引用

```go
// ser-finance 调钱包的代码（Go 层面，不是 IDL 层面）：
import "common/pkg/kitex_gen/ser_wallet"
import "common/pkg/kitex_gen/ser_wallet/walletservice"

client, _ := walletservice.NewClient("ser-wallet", opts...)
resp, err := client.CreditDeposit(ctx, &ser_wallet.CreditDepositReq{...})
```

## 十一、弹性空间说明

以下是明确保留弹性的地方——需求评审后可能调整，当前不钉死：

| 可能调整 | 当前方案 | 调整方向 |
|---------|---------|---------|
| 奖金活动是否由钱包管理 | GetBonusActivities 在 wallet.thrift | 如果由活动模块管理，这个接口可能改为 RPC 调活动服务 |
| 提现是否需要交易密码 | CreateWithdrawalOrderReq 暂无密码字段 | 如果需要，加一个 `optional string tradePassword` |
| 充值方式是财务还是钱包查 | GetDepositMethods 在钱包层 | 如果完全由财务管理，可能改为钱包 RPC 调财务 |
| 兑换赠送规则 | ExecuteExchangeResp 有 bonusAmount | 赠送规则可能由运营另行配置，字段可能变化 |
| B 端订单查询 | 暂未放在钱包 IDL | 如果 B 端要在钱包模块查订单（非财务），需新增 PageDepositOrder / PageWithdrawalOrder |
| wallet.thrift 是否需要拆分 | 当前 14 个接口在一个文件 | 如果充值或提现接口暴增到 10+，可拆出 deposit.thrift / withdrawal.thrift |
| 各 struct 的具体字段 | 骨架已列出关键字段 | 需求评审后细化增减 |

## 十三、总结

```
文件数：  6 个（service + model + currency + wallet + transaction + rpc）
接口数：  35 个（含 1 个公共枚举）
结构体数：78 个（32 Req + 34 Resp + 12 Model/子结构）
预估总行数：~800-1000 行

设计原则：
  ① 严格遵循工程 20 个服务的已有 IDL 规范
  ② 中粒度拆分（不过度拆分，不过度合并）
  ③ 公共子结构体抽到 model.thrift 避免重复
  ④ RPC 供他接口独立成文件（先例：user_rpc / live_rpc）
  ⑤ 字段只定义骨架，保留弹性空间
  ⑥ C 端不传 userId，RPC 必传 userId + idempotencyKey
```
