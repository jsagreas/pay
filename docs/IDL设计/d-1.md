## 1. 设计前提：工程既有 IDL 规范提炼

通过扫描 ser-app、ser-user、ser-bcom、ser-mall 等已有服务的 IDL 源码，提炼出以下必须遵循的既有规范：

### 1.1 文件组织规范

| 规范 | 工程实证 | 说明 |
|------|---------|------|
| **service.thrift 为唯一主入口** | ser-app、ser-user、ser-bcom 均如此 | 只定义 `service XxxService {}`，通过 `include` 引入子文件的结构体 |
| **按业务概念拆分子文件** | ser-app：banner.thrift / home_category.thrift；ser-user：users.thrift / user_follow.thrift / user_blacklist.thrift | 一个 `.thrift` 文件 = 一个业务概念的全部结构体（Req + Resp + Entity） |
| **RPC 供给接口独立成文件** | ser-user：`user_rpc.thrift`（供其他服务调用的接口） | 与面向网关的接口分离，职责清晰 |
| **公共结构体走 common/** | `common/idl/common/enum.thrift` | 跨服务复用的结构体放公共目录 |

### 1.2 命名规范

| 维度 | 规范 | 工程实例 |
|------|------|---------|
| **namespace** | `namespace go ser_{服务名}` | `namespace go ser_app`、`namespace go ser_user` |
| **Service 名** | `{服务名}Service` | `AppService`、`UserService`、`BComService` |
| **创建方法** | `Create{Entity}` | `CreateBanner`、`CreateHomeCategory` |
| **更新方法** | `Update{Entity}` | `UpdateBanner`、`UpdateUser` |
| **删除方法** | `Delete{Entity}` | `DeleteBanner`、`DeleteClientBlackCountry` |
| **状态切换** | `Status{Entity}` | `StatusBanner`（独立方法，不合并到 Update） |
| **排序操作** | `Sort{Entity}` | `SortBanner` |
| **B 端分页** | `Page{Entity}` | `PageBanner`、`QueryUserPage` |
| **B 端详情** | `Detail{Entity}` | `DetailBanner` |
| **C 端列表** | `List{Entity}` 或 `Get{Entity}` | `ListBanner`、`GetUserDetail` |
| **CSV 导出** | `Query{Entity}Csv` | `QueryUserCsv`、`QueryLogCsv` |
| **批量查询** | `BatchGet{Entity}` | `BatchGetUserInfo`、`BatchGetUserDetail` |
| **请求结构体** | `{Action}{Entity}Req` | `CreateBannerReq`、`PageBannerReq` |
| **响应结构体** | `{Action}{Entity}Resp` | `CreateBannerResp`、`PageBannerResp` |
| **数据实体** | `{Entity}Item` 或 `{Entity}` | `BannerItem`、`User` |

### 1.3 字段规范

| 规范 | 工程实证 |
|------|---------|
| **ID 类型**：`i64` | 全工程统一，Snowflake 生成 |
| **时间戳**：`i64`（毫秒） | `createAt`、`updateAt`、`onlineTime` 均为 i64 |
| **金额**：`i64`（最小单位整数） | ser-mall 的 `price` 用 i64 |
| **状态/枚举**：`i32` | `status`、`jumpType`、`accountType` |
| **必填标注**：`required` | 创建/更新的核心字段 |
| **可选标注**：`optional` | 查询条件、非必填字段 |
| **HTTP 标注**：`api.body` | 所有字段均带 `(api.body = "fieldName")` |
| **分页请求**：`pageNo` + `pageSize`（i32） | 全工程统一命名 |
| **分页响应**：`pageList` + `total` + `pageNo` + `pageSize` | 全工程统一结构 |
| **空响应体**：`struct XxxResp {}` | 更新/删除/状态切换等无返回数据的操作 |

### 1.4 include 引用方式

```thrift
// 同服务内子文件
include "wallet.thrift"

// 公共库
include "../common/enum.thrift"

// 跨服务引用（已有先例）
include "../ser-kyc/user_kyc.thrift"   // ser-user 中引用了 ser-kyc
```

## 2. 钱包模块 IDL 的业务域划分

根据需求理解/d-1.md 和接口评估/d-1.md，钱包模块涉及的业务操作可归纳为以下几个独立的业务域：

| 业务域 | 核心操作 | 面向端 | 操作量级 |
|--------|---------|--------|---------|
| **钱包账户** | 余额查询、余额总览、余额变更（内部） | C 端查询 + RPC 供给 | 高频读、中频写 |
| **充值** | 充值方式查询、创建充值订单、充值回调处理 | C 端 + 回调 | 中频 |
| **提现** | 提现方式查询、创建提现申请、B 端审核、提现回调 | C 端 + B 端 | 中频 |
| **提现账户** | 银行卡/电子钱包绑定、查询、删除 | C 端 | 低频 |
| **兑换** | 汇率预览、执行兑换 | C 端 | 低频 |
| **币种配置** | 币种 CRUD、汇率管理、充值/提现配置 | B 端 | 低频 |
| **账单记录** | 充值/提现/兑换/奖励等各类记录分页查询 | C 端 + B 端 | 高频读 |
| **RPC 供给** | 加款/扣款/冻结/解冻/余额查询（供其他服务调用） | 内部 RPC | 中频 |

**关键观察：** 这 8 个业务域相互独立——每个域的结构体和方法自成一体，不存在交叉定义的情况。这与 ser-user 中 users / user_follow / user_blacklist / user_rpc 各自独立的模式完全一致。

## 3. 文件拆分方案与理由

### 3.1 拆分结果

| 文件 | 业务域 | 包含内容 | 拆分理由 |
|------|--------|---------|---------|
| **service.thrift** | 服务主入口 | `WalletService` 定义 + 全部 include | 工程统一模式，必须有 |
| **wallet.thrift** | 钱包账户 | 余额查询/总览相关结构体 | 钱包的核心实体，C 端最高频接口 |
| **currency.thrift** | 币种配置 | 币种 CRUD + 汇率管理 + 充提配置 | B 端独立管理页面，结构体自成一体 |
| **deposit.thrift** | 充值 | 充值方式/创建订单/回调处理 | 充值流程独立，结构体多（方式/订单/回调） |
| **withdraw.thrift** | 提现 | 提现方式/创建申请/审核/回调 | 提现流程独立且涉及 B 端审核 |
| **withdraw_account.thrift** | 提现账户 | 银行卡/电子钱包绑定管理 | 独立的实体管理，与提现订单分离 |
| **exchange.thrift** | 兑换 | 汇率预览/执行兑换 | 独立业务流程 |
| **record.thrift** | 账单记录 | 各类记录分页查询 | 多种记录类型的查询，结构体多但模式统一 |
| **wallet_rpc.thrift** | RPC 供给 | 加款/扣款/冻结/解冻/查余额 | 遵循 ser-user 的 user_rpc.thrift 先例，RPC 接口单独管理 |

**共 9 个文件（1 个主入口 + 8 个子文件）。**

### 3.2 拆分依据

**为什么是 9 个文件而不是更多或更少？**

| 对比方案 | 文件数 | 问题 |
|---------|--------|------|
| **全部塞进 1 个文件** | 1 | 违反工程规范（ser-user 有 13 个子文件）；单文件过大（预估 600+ 行）；不同域的结构体混在一起难以维护 |
| **按 C 端/B 端拆** | 3 | 一个接口可能同时面向 C 端和 B 端（如提现：C 端申请 + B 端审核）；按端拆会导致同一业务域的结构体被割裂 |
| **按表拆（一表一文件）** | 12 | 过度碎片化；很多表的操作是在同一个业务流程中组合使用的（如 deposit_order + deposit_config 都是充值域） |
| **按业务域拆（当前方案）** | 9 | 每个文件对应一个独立的业务概念，结构体内聚、文件间低耦合。与 ser-user（13 文件）、ser-app（5 文件）的粒度一致 |
| **将提现账户合并到提现** | 8 | 可行但不推荐——提现账户是独立的实体管理（CRUD），与提现订单流程在概念上不同。分开后各自更清晰 |

### 3.3 与工程已有服务的对标

| 服务 | 子文件数 | 方法数 | 对标 |
|------|---------|--------|------|
| ser-bcom | 3（base/black/white） | ~14 | 三个独立管理概念 |
| ser-app | 4（banner/s3/home_category/content_strategy） | ~16 | 四个独立管理概念 |
| ser-user | 10+（users/user_rpc/login_log/country/user_kyc/vip/blacklist/follow...） | ~30 | 多个独立管理概念 |
| **ser-wallet** | **8（wallet/currency/deposit/withdraw/withdraw_account/exchange/record/wallet_rpc）** | **预估 35-45** | 8 个独立业务域 |

ser-wallet 的复杂度介于 ser-app 和 ser-user 之间，8 个子文件合理。

## 6. service.thrift — 服务主入口

**职责：** 定义 `WalletService`，include 全部子文件，按业务域分组列出所有方法。

**设计要点：**
- 只有 service 定义和 include，不定义任何 struct
- 方法按业务域分组，每组用注释标注
- 方法签名引用子文件的 `{file}.{Struct}` 格式

```thrift
namespace go ser_wallet

include "wallet.thrift"
include "currency.thrift"
include "deposit.thrift"
include "withdraw.thrift"
include "withdraw_account.thrift"
include "exchange.thrift"
include "record.thrift"
include "wallet_rpc.thrift"
include "../common/enum.thrift"

service WalletService {

    // ==================== 钱包余额（C端） ====================

    // C端-获取用户钱包余额列表
    wallet.ListWalletResp ListWallet(1: wallet.ListWalletReq req)

    // C端-获取钱包总览（总资产折算）
    wallet.GetWalletOverviewResp GetWalletOverview(1: wallet.GetWalletOverviewReq req)

    // ==================== 钱包查询（B端） ====================

    // B端-分页查询用户钱包
    wallet.PageWalletResp PageWallet(1: wallet.PageWalletReq req)

    // B端-查询指定用户的钱包详情
    wallet.DetailWalletResp DetailWallet(1: wallet.DetailWalletReq req)

    // B端-手动调账
    wallet.ManualAdjustResp ManualAdjust(1: wallet.ManualAdjustReq req)

    // ==================== 币种配置（B端） ====================

    // B端-创建币种
    currency.CreateCurrencyResp CreateCurrency(1: currency.CreateCurrencyReq req)

    // B端-更新币种
    currency.UpdateCurrencyResp UpdateCurrency(1: currency.UpdateCurrencyReq req)

    // B端-启用/禁用币种
    currency.StatusCurrencyResp StatusCurrency(1: currency.StatusCurrencyReq req)

    // B端-分页查询币种
    currency.PageCurrencyResp PageCurrency(1: currency.PageCurrencyReq req)

    // B端-币种详情
    currency.DetailCurrencyResp DetailCurrency(1: currency.DetailCurrencyReq req)

    // C端-获取启用的币种列表
    currency.ListCurrencyResp ListCurrency(1: currency.ListCurrencyReq req)

    // B端-录入/更新汇率
    currency.SaveExchangeRateResp SaveExchangeRate(1: currency.SaveExchangeRateReq req)

    // C端-查询实时汇率
    currency.GetExchangeRateResp GetExchangeRate(1: currency.GetExchangeRateReq req)

    // B端-分页查询汇率日志
    currency.PageExchangeRateLogResp PageExchangeRateLog(1: currency.PageExchangeRateLogReq req)

    // B端-充值配置管理（创建/更新）
    currency.SaveDepositConfigResp SaveDepositConfig(1: currency.SaveDepositConfigReq req)

    // B端-提现配置管理（创建/更新）
    currency.SaveWithdrawConfigResp SaveWithdrawConfig(1: currency.SaveWithdrawConfigReq req)

    // ==================== 充值（C端 + B端） ====================

    // C端-获取充值方式列表
    deposit.ListDepositMethodResp ListDepositMethod(1: deposit.ListDepositMethodReq req)

    // C端-创建充值订单
    deposit.CreateDepositOrderResp CreateDepositOrder(1: deposit.CreateDepositOrderReq req)

    // C端-查询充值订单状态
    deposit.GetDepositOrderResp GetDepositOrder(1: deposit.GetDepositOrderReq req)

    // 内部-充值回调处理
    deposit.DepositCallbackResp DepositCallback(1: deposit.DepositCallbackReq req)

    // B端-分页查询充值订单
    deposit.PageDepositOrderResp PageDepositOrder(1: deposit.PageDepositOrderReq req)

    // B端-充值订单详情
    deposit.DetailDepositOrderResp DetailDepositOrder(1: deposit.DetailDepositOrderReq req)

    // ==================== 提现（C端 + B端） ====================

    // C端-获取提现方式列表
    withdraw.ListWithdrawMethodResp ListWithdrawMethod(1: withdraw.ListWithdrawMethodReq req)

    // C端-创建提现申请
    withdraw.CreateWithdrawOrderResp CreateWithdrawOrder(1: withdraw.CreateWithdrawOrderReq req)

    // C端-查询提现订单状态
    withdraw.GetWithdrawOrderResp GetWithdrawOrder(1: withdraw.GetWithdrawOrderReq req)

    // 内部-提现回调处理
    withdraw.WithdrawCallbackResp WithdrawCallback(1: withdraw.WithdrawCallbackReq req)

    // B端-分页查询提现订单
    withdraw.PageWithdrawOrderResp PageWithdrawOrder(1: withdraw.PageWithdrawOrderReq req)

    // B端-提现订单详情
    withdraw.DetailWithdrawOrderResp DetailWithdrawOrder(1: withdraw.DetailWithdrawOrderReq req)

    // B端-审核提现（通过/驳回）
    withdraw.AuditWithdrawOrderResp AuditWithdrawOrder(1: withdraw.AuditWithdrawOrderReq req)

    // ==================== 提现账户（C端） ====================

    // C端-绑定提现账户
    withdraw_account.CreateWithdrawAccountResp CreateWithdrawAccount(1: withdraw_account.CreateWithdrawAccountReq req)

    // C端-获取我的提现账户列表
    withdraw_account.ListWithdrawAccountResp ListWithdrawAccount(1: withdraw_account.ListWithdrawAccountReq req)

    // C端-删除提现账户
    withdraw_account.DeleteWithdrawAccountResp DeleteWithdrawAccount(1: withdraw_account.DeleteWithdrawAccountReq req)

    // C端-设置默认提现账户
    withdraw_account.SetDefaultWithdrawAccountResp SetDefaultWithdrawAccount(1: withdraw_account.SetDefaultWithdrawAccountReq req)

    // ==================== 兑换（C端 + B端） ====================

    // C端-兑换预览（输入金额→预估到账）
    exchange.ExchangePreviewResp ExchangePreview(1: exchange.ExchangePreviewReq req)

    // C端-执行兑换
    exchange.ExecuteExchangeResp ExecuteExchange(1: exchange.ExecuteExchangeReq req)

    // B端-分页查询兑换记录
    exchange.PageExchangeRecordResp PageExchangeRecord(1: exchange.PageExchangeRecordReq req)

    // B端-兑换记录详情
    exchange.DetailExchangeRecordResp DetailExchangeRecord(1: exchange.DetailExchangeRecordReq req)

    // ==================== 账单记录（C端 + B端） ====================

    // C端-分页查询账单记录（通用，支持按类型过滤）
    record.PageBillRecordResp PageBillRecord(1: record.PageBillRecordReq req)

    // B端-分页查询交易流水
    record.PageTransactionLogResp PageTransactionLog(1: record.PageTransactionLogReq req)

    // B端-交易流水详情
    record.DetailTransactionLogResp DetailTransactionLog(1: record.DetailTransactionLogReq req)

    // B端-导出交易流水CSV
    record.QueryTransactionLogCsvResp QueryTransactionLogCsv(1: record.PageTransactionLogReq req)

    // B端-导出充值订单CSV
    deposit.QueryDepositOrderCsvResp QueryDepositOrderCsv(1: deposit.PageDepositOrderReq req)

    // ==================== 枚举（公共） ====================

    // 获取枚举选项
    enum.GetEnumOptionsResp GetEnumOptions(1: enum.GetEnumOptionsReq req)

    // ==================== RPC接口（供其他服务调用） ====================

    // RPC-加款（充值到账/返奖/系统赠送等）
    wallet_rpc.CreditBalanceResp CreditBalance(1: wallet_rpc.CreditBalanceReq req)

    // RPC-扣款确认（冻结金额扣除）
    wallet_rpc.DebitBalanceResp DebitBalance(1: wallet_rpc.DebitBalanceReq req)

    // RPC-冻结余额
    wallet_rpc.FreezeBalanceResp FreezeBalance(1: wallet_rpc.FreezeBalanceReq req)

    // RPC-解冻余额
    wallet_rpc.UnfreezeBalanceResp UnfreezeBalance(1: wallet_rpc.UnfreezeBalanceReq req)

    // RPC-查询用户余额
    wallet_rpc.GetBalanceResp GetBalance(1: wallet_rpc.GetBalanceReq req)
```

## 14. wallet_rpc.thrift — 跨服务 RPC 接口

**职责：** 供财务、风控、结算等其他服务调用的余额操作接口。独立成文件，遵循 ser-user 的 `user_rpc.thrift` 先例。

**独立的关键理由：**
1. 面向网关的接口通过 Header 获取 userId（`tracer.GetUserId`），而 RPC 接口需要**显式传入 userId**
2. RPC 接口有**幂等性要求**（refOrderNo + txType 组合唯一），需求和约束与 C 端/B 端接口不同
3. 将来其他服务引用时，只需 `include "../ser-wallet/wallet_rpc.thrift"` 即可了解可调用的接口


## 15. 命名规范速查表

汇总 ser-wallet IDL 中使用的全部命名规范，与工程既有规范完全一致：

### 15.1 文件级

| 项 | 规范 | 示例 |
|----|------|------|
| 目录 | `common/idl/ser-{service}/` | `common/idl/ser-wallet/` |
| 主入口 | `service.thrift` | 唯一 service 定义 |
| 子文件 | `{业务概念}.thrift` | `wallet.thrift`、`deposit.thrift` |
| RPC 文件 | `{service}_rpc.thrift` | `wallet_rpc.thrift` |
| namespace | `namespace go ser_{service}` | `namespace go ser_wallet` |

### 15.2 Service 方法

| 操作 | 命名 | 参数 | 说明 |
|------|------|------|------|
| 创建 | `Create{Entity}` | `Create{Entity}Req` → `Create{Entity}Resp` | 返回 id |
| 更新 | `Update{Entity}` | `Update{Entity}Req` → `Update{Entity}Resp` | 空响应 |
| 删除 | `Delete{Entity}` | `Delete{Entity}Req` → `Delete{Entity}Resp` | 空响应 |
| 状态 | `Status{Entity}` | `Status{Entity}Req` → `Status{Entity}Resp` | 空响应 |
| B 端分页 | `Page{Entity}` | `Page{Entity}Req` → `Page{Entity}Resp` | 含 pageList/total/pageNo/pageSize |
| B 端详情 | `Detail{Entity}` | `Detail{Entity}Req` → `Detail{Entity}Resp` | 按 id 查询 |
| C 端列表 | `List{Entity}` | `List{Entity}Req` → `List{Entity}Resp` | 简化查询 |
| C 端单查 | `Get{Entity}` | `Get{Entity}Req` → `Get{Entity}Resp` | 按业务键查询 |
| 保存（创建或更新） | `Save{Entity}` | `Save{Entity}Req` → `Save{Entity}Resp` | 按唯一键 upsert |
| 审核 | `Audit{Entity}` | `Audit{Entity}Req` → `Audit{Entity}Resp` | 通过/驳回 |
| 导出 | `Query{Entity}Csv` | 复用分页 Req → `Query{Entity}CsvResp` | 返回 binary |
| 批量 | `BatchGet{Entity}` | `BatchGet{Entity}Req` → `BatchGet{Entity}Resp` | 返回 list |

### 15.3 字段类型

| 数据 | Thrift 类型 | 示例 |
|------|------------|------|
| ID | `i64` | `userId`、`currencyId`、`id` |
| 时间戳 | `i64` | `createAt`、`updateAt`（毫秒） |
| 金额 | `i64` | `amount`、`available`、`frozen`（最小单位整数） |
| 汇率 | `i64` | `exchangeRate`（×10^8 整数） |
| 状态 / 枚举 | `i32` | `status`、`txType`、`walletType` |
| 文本 | `string` | `orderNo`、`currencyCode`、`remark` |
| 布尔 | `bool` 或 `i32` | 工程偏好用 `i32`（1/0） |
| 列表 | `list<T>` | `list<WalletItem>` |
| 二进制 | `binary` | CSV 导出数据 |

### 15.4 字段标注

```thrift
// 必填字段
1: required i64 userId (api.body = "userId")

// 可选字段
2: optional i32 status (api.body = "status")

// 带默认值（较少使用）
3: i32 pageNo = 1 (api.body = "pageNo")
```

## 18. 方案对比：为什么不用其他拆法

### 18.1 方案 A：按服务拆分（3 个 service）

```
方案：拆成 WalletService + CurrencyService + FinanceService 三个独立 Kitex 服务
```

| 维度 | 评估 |
|------|------|
| 优点 | 微服务粒度更细，可独立部署 |
| 缺点 | 钱包余额变更和充值/提现/兑换是强耦合的（同一个 DB 事务），拆成不同服务会引入分布式事务问题。当前阶段没有独立部署的需求，过早拆分增加运维成本 |
| 结论 | **不采用**。一个 ser-wallet 服务内通过 IDL 文件拆分即可实现关注点分离 |

### 18.2 方案 B：单文件方案

```
方案：所有结构体和 service 放在一个 wallet.thrift 文件中
```

| 维度 | 评估 |
|------|------|
| 优点 | 最简单，不需要考虑文件拆分 |
| 缺点 | 预估 800-1000 行，难以维护；违反工程规范（ser-user 有 13 个子文件）；多人协作时冲突概率高 |
| 结论 | **不采用** |

### 18.3 方案 C：按端拆分（C 端 / B 端 / RPC）

```
方案：拆成 wallet_c.thrift（C端）+ wallet_b.thrift（B端）+ wallet_rpc.thrift（RPC）
```

| 维度 | 评估 |
|------|------|
| 优点 | 端的边界清晰 |
| 缺点 | 提现同时涉及 C 端（申请）和 B 端（审核），会导致同一业务概念被割裂到两个文件；数据实体（如 WithdrawOrderItem）需要在两个文件间共享或重复定义 |
| 结论 | **不采用**。按业务域拆分更自然 |

### 18.4 方案 D：当前方案（按业务域拆分 9 文件）

| 维度 | 评估 |
|------|------|
| 优点 | 每个文件内聚一个完整业务概念（实体 + C 端 + B 端操作全在一起）；与工程既有模式一致；文件粒度适中（70-180 行/文件） |
| 缺点 | 文件数量比方案 B/C 多（但与 ser-user 的 13 文件相比仍然适中） |
| 结论 | **采用** |

## 19. 总结

### 19.1 设计决策一览

| 决策项 | 选择 | 依据 |
|--------|------|------|
| 服务数量 | 1 个 WalletService | 余额操作需同 DB 事务，不拆微服务 |
| IDL 文件数 | 9 个（1 主入口 + 8 子文件） | 按业务域拆分，与 ser-user 模式一致 |
| 方法总数 | 49 个 | C 端 18 + B 端 24 + RPC 6 + 公共 1 |
| 结构体总数 | ~47 对 Req/Resp + 16 个 Entity | 每个方法对应一对 Req/Resp |
| 命名规范 | 完全沿用工程既有规范 | Create/Update/Delete/Status/Page/Detail/List/Get |
| 字段标注 | `api.body` + `required/optional` | 与 ser-app、ser-user 一致 |
| RPC 接口 | 独立 wallet_rpc.thrift | 沿用 ser-user 的 user_rpc.thrift 先例 |

### 19.2 文件清单

```
common/idl/ser-wallet/
├── service.thrift            ~60行    服务主入口（49个方法）
├── wallet.thrift             ~90行    钱包余额（C端+B端）
├── currency.thrift           ~180行   币种+汇率+充提配置（B端为主）
├── deposit.thrift            ~130行   充值（C端+B端+回调）
├── withdraw.thrift           ~150行   提现（C端+B端+审核+回调）
├── withdraw_account.thrift   ~70行    提现账户（C端）
├── exchange.thrift           ~90行    兑换（C端+B端）
├── record.thrift             ~120行   账单记录（C端+B端+导出）
└── wallet_rpc.thrift         ~80行    RPC供给（加/扣/冻/解冻/查余额）
                              ─────
                              ~970行   合计
```

### 19.3 弹性空间

当前方案保留了灵活调整的空间：

- **方法增减**：需求评审后可能增删接口，每个文件独立修改互不影响
- **字段调整**：struct 内部字段可自由调整，不影响文件结构
- **新增子文件**：如果后续出现新的业务概念（如优惠券、红包），新增一个 .thrift 文件 + 在 service.thrift 中 include 即可
- **跨服务引用**：其他服务需要调用钱包接口时，只需 `include "../ser-wallet/wallet_rpc.thrift"` 引入结构体定义
