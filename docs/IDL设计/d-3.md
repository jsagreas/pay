# IDL 设计方案 — ser-wallet 服务接口定义

> 定位：分析 ser-wallet 的 Thrift IDL 应该如何设计、拆分几个文件、每个文件的职责
> 核心原则：**遵循现有工程 IDL 规范，不发明新模式**；按业务域拆分文件，不过度也不过少
> 信息基础：common/idl/ 全目录扫描（21个服务目录、71个 .thrift 文件的实际模式）、接口评估/d-3.md（69个主责接口）、需求理解/d-3.md、架构设计/d-3.md
> 参照样本：ser-live（9文件37方法，最复杂）、ser-user（14文件，最多文件）、ser-app（5文件，中等）、ser-item（4文件，简单）
> 金额约束：IDL 中所有金额字段统一使用 **string** 类型（ADR-06: DECIMAL(20,8) + shopspring/decimal，Thrift 无 decimal 类型）
> 文档状态：预评审阶段（屏蔽具体字段细节，聚焦结构设计和分文件策略）

## 1. 设计决策与依据

### 1.1 决策1：一个 service 还是多个 service？

```
决策：一个 WalletService（单 service 定义）
否决：拆为 WalletService + CurrencyService 两个 service
```

**依据**：

| 维度 | 单 service | 多 service |
|------|-----------|-----------|
| 工程惯例 | ✅ ser-app、ser-live、ser-user 全部是单 service | 无先例 |
| 代码生成 | gen_kitex.sh 只识别单个 *service.thrift 入口 | 多 service 需改脚本 |
| handler.go | 一个 handler 文件对应一个 service 接口 | 多个 handler 拆分复杂 |
| 部署 | 一个进程 = 一个 service（项目统一模式） | 同一进程多 service 需额外配置 |
| 路由 | gate-font/gate-back 路由统一指向一个 service | 需要按 service 分发 |
| 架构设计/d-3.md | ADR-04 决策"币种配置与钱包同进程" | — |

**结论**：遵循现有工程的一致模式——一个服务一个 service 定义。钱包和币种配置虽然是两个子模块，但共处一个 ser-wallet 进程，对外只暴露一个 WalletService。

**工程验证**：

```
ser-live = 1 LiveService 包含：房间、观众、消费、审核、设置、直播间API、RPC  (9文件 37方法)
ser-user = 1 UserService 包含：用户、KYC、关注、黑名单、定制、扩展、RPC    (14文件)
ser-wallet = 1 WalletService 包含：币种配置、钱包核心、充值、提现、兑换、记录、RPC (8文件)
```

规模与 ser-live 相当，单 service 完全合理。

### 1.2 决策2：struct 文件拆分粒度

```
决策：按业务域拆分，7 个 struct 文件 + 1 个 service 文件 = 共 8 个文件
否决方案A：全部塞一个文件（太大，阅读维护困难）
否决方案B：每个接口一个文件（过度拆分，文件碎片化）
否决方案C：拆出 enum 独立文件（现有工程不用 Thrift enum）
```

**依据（现有工程对标）**：

| 现有参照 | 子模块数 | IDL文件数 | 文件/模块比 |
|---------|---------|----------|------------|
| ser-live | 7个子模块 | 9个文件 | 1.3 |
| ser-user | 6个子模块 | 14个文件 | 2.3 |
| ser-app | 4个子模块 | 5个文件 | 1.25 |
| ser-item | 3个子模块 | 4个文件 | 1.3 |
| ser-kyc | 2个子模块 | 3个文件 | 1.5 |
| **ser-wallet** | **7个子模块** | **8个文件** | **1.14** |

ser-wallet 的 8 个文件 ÷ 7 个子模块 = 1.14，在现有工程的合理范围内（1.1 ~ 2.3），偏紧凑。

### 1.3 决策3：枚举如何表示？

```
决策：不单独拆 enum 文件，枚举用 i32 + 注释内联定义
依据：现有工程所有服务都是 i32 + 注释，不使用 Thrift enum 关键字
```

**工程事实**（扫描全部 71 个 .thrift 文件确认）：

```thrift
// 现有工程的枚举表示方式（统一 i32 + 注释）
1: i32 status       // 状态：0禁用 1启用
2: i32 walletType   // 钱包类型：1中心 2奖励 3主播 4代理 5场馆
3: i32 direction    // 方向：1收入 2支出
4: i32 bizType      // 业务类型：1充值 2提现 3兑换 4投注 5礼物 6奖励 ...
```

不使用 Thrift `enum` 关键字的原因：
- 全工程保持一致性
- `i32` 在 Go 侧映射为 `int32`，直接可用
- 新增枚举值时无需修改 IDL（通过注释文档约束）
- 配合 `common/enum.thrift` 的 `GetEnumOptions()` 动态枚举查询已有模式

### 1.4 决策4：金额字段用什么类型？

```
决策：所有金额字段统一使用 string 类型
否决：double（浮点精度灾难）、i64（多币种最小单位换算极易出错）
```

**依据（ADR-06 + 技术挑战/d-3.md 挑战2+19）**：

```
Thrift 没有 decimal 类型
  → 可选方案：i64、double、string
  → i64：d-2 方案，VND 的 1=1 但 USDT 的 1=100000000，语义不直观
  → double：0.1+0.2≠0.3，资金场景绝对禁止
  → string：语义清晰，"1000.50000000" 就是 1000.5，无精度损失

数据流向：
  调用方 string → Thrift 传输 string → Service 层 decimal.NewFromString() → 精确计算
  Service 层 decimal.String() → Thrift 传输 string → 调用方展示

string 是 Stripe、Square 等一线支付公司 API 金额字段的标准做法。
```

**在 IDL 中的体现**：

```thrift
// ✅ 正确（d-3 方案）
struct WalletCreditReq {
    1: required i64    userId
    2: required string amount        // "5000.00000000" — string 传输
    3: required string currencyCode
    ...
}

// ❌ 错误
struct WalletCreditReq {
    2: required double amount   // 浮点精度灾难
    2: required i64    amount   // 多币种最小单位换算困难
}
```

### 1.5 决策5：是否需要 s3.thrift？

```
决策：不单独拆 s3.thrift，图标上传的 struct 放在 currency.thrift 中
依据：钱包只有币种图标上传一个 S3 场景，不值得单独拆文件
      参照 ser-app 的 s3.thrift 仅因有 Banner 多图片上传场景（多种 struct）才独立
```

## 2. 文件组织方案

### 2.1 最终方案

```
common/idl/ser-wallet/              # 8 个文件
├── service.thrift                  # 服务入口（纯 service 定义，零 struct）
├── currency.thrift                 # 币种配置域（B端 CRUD + RPC 查询 + 图标上传）
├── wallet.thrift                   # 钱包核心域（余额查询、钱包信息、B端钱包管理）
├── deposit.thrift                  # 充值域（C端充值流程）
├── withdraw.thrift                 # 提现域（C端提现流程 + 提现账户管理）
├── exchange.thrift                 # 兑换域（C端法币→BSB 兑换）
├── record.thrift                   # 记录查询域（C端 4 Tab 记录 + 详情）
└── wallet_rpc.thrift               # 对外 RPC 能力（供财务/游戏/直播/活动等服务调用）
```

### 2.2 每个文件的定位

| 文件 | 对应子模块 | 主要服务端 | 接口数（预估） | 说明 |
|------|----------|----------|-------------|------|
| service.thrift | — | — | 0（纯引用） | 只做 include + service 方法签名 |
| currency.thrift | 币种配置 | B端 + RPC | 15 | B端9个CRUD + RPC6个查询 |
| wallet.thrift | 钱包核心 | C端 + B端 | 4 | 余额查询、钱包信息、币种切换、B端查询 |
| deposit.thrift | 充值 | C端 | 5 | 充值方式、发起充值、状态查询、奖金活动、重复检测 |
| withdraw.thrift | 提现 | C端 | 7 | 提现方式、限额、发起提现、账户CRUD |
| exchange.thrift | 兑换 | C端 | 2 | 兑换预览、发起兑换 |
| record.thrift | 记录 | C端 | 5 | 4个Tab列表 + 详情 |
| wallet_rpc.thrift | 对外RPC | RPC | 18 | 入账/扣款/冻结/投注/礼物/奖励/查询/转账等 |
| **合计** | | | **~56** | 接口评估/d-3.md 中的 C端21+B端11+RPC暴露24=56 |

> 56 个暴露方法在一个 WalletService 中。参照 ser-live 有 37 个方法（9文件），ser-wallet 56 个方法（8文件）= 每文件平均 7 个方法，合理。

### 2.3 文件间 include 关系

```
service.thrift
  │
  ├── include "currency.thrift"
  ├── include "wallet.thrift"
  ├── include "deposit.thrift"
  ├── include "withdraw.thrift"
  ├── include "exchange.thrift"
  ├── include "record.thrift"
  └── include "wallet_rpc.thrift"

wallet_rpc.thrift
  └── include "wallet.thrift"        # RPC 返回中可能嵌套 WalletInfo 结构

其他文件之间无 include（扁平依赖，不交叉引用）
```

**重要约束**：struct 文件之间不互相 include（参照现有工程 ser-app/ser-item 的扁平模式）。如果多个文件需要共享 struct，提取到被共用的文件中（如 wallet.thrift 的 WalletInfo 可被 wallet_rpc.thrift 引用）。

## 3. 方案对比与选择

### 3.1 三种候选方案

| 方案 | 文件数 | 思路 | 优点 | 缺点 |
|------|--------|------|------|------|
| **A: 极简** | 3-4 | service + wallet + currency + rpc | 文件少，简单 | 单文件过大（wallet.thrift 300+ 行），充值/提现/兑换混杂 |
| **B: 业务域分离** | 8 | 按业务域一域一文件 | 职责清晰，独立维护 | 文件数多（但在工程合理范围） |
| **C: 极细** | 12+ | 每个子功能一文件 | 极致细粒度 | 过度拆分，碎片化 |

### 3.2 为什么选方案 B（8 文件）

```
判定标准1：现有工程的文件数/模块数比值范围 = 1.1 ~ 2.3
  方案A = 0.6 ← 偏低，说明文件太少
  方案B = 1.14 ← 合理范围内
  方案C = 1.7+ ← 合理范围内但不必要

判定标准2：单文件 struct 数量是否可维护
  方案A: wallet.thrift 预计 40+ struct ← 太大
  方案B: 每文件 5-15 个 struct ← 合理
  方案C: 每文件 2-5 个 struct ← 过碎

判定标准3：业务域边界清晰度
  充值、提现、兑换是三个完全独立的业务流程
  各自有独立的 Req/Resp/Entity 结构
  放同一个文件 = 阅读困难 + git diff 冲突
  分开 = 一人改充值不影响提现的 IDL 文件

结论：方案 B 在文件数、单文件大小、业务边界三个维度都处于最佳平衡点。
```

### 3.3 特殊考量：为什么充值/提现/兑换要分开？

```
看似相似实则不同：

deposit.thrift（充值）
  · 调用财务 → 获取充值方式、创建充值订单
  · 涉及奖金活动选择、重复转账检测
  · 回调入账由财务调 wallet_rpc 完成

withdraw.thrift（提现）
  · KYC 校验 → 提现账户管理 → 发起提现
  · 提现账户 CRUD（银行卡/电子钱包/USDT）是独立的实体
  · 涉及限额查询、冻结流程
  · 是 struct 最多的 C端域

exchange.thrift（兑换）
  · 纯进程内事务（不调外部服务）
  · 只有预览 + 执行两个接口
  · 涉及双币种扣入（法币扣 → BSB 入）

如果合并为一个 "transaction.thrift"：
  · 30+ struct 挤在一起，阅读困难
  · 改充值影响提现的 git 历史
  · 团队协作时增加冲突概率

ser-live 也是类似思路：live_room / live_consume / live_audience / live_audit 各自独立文件
```

## 5. currency.thrift — 币种配置域

### 5.1 定位

- **B端管理**：币种 CRUD、基准币种、汇率展示、日志查询
- **RPC 查询**：供钱包模块进程内调用 + 其他服务 RPC 调用
- 对应需求：币种配置需求文档全部 + 接口评估/d-3.md 第一章

### 5.2 struct 规划

```
currency.thrift 预计 struct 数量：~25 个

数据实体（列表项/详情）：
  · CurrencyItem         — 币种列表展示项（B端表格行）
  · CurrencyInfo         — 币种完整信息（RPC 返回）
  · RateItem             — 汇率展示行（实时/平台/入款/出款）
  · RateLogItem          — 汇率变更日志行
  · OperationLogItem     — 操作日志行
  · BaseCurrencyInfo     — 基准币种信息

请求/响应（B端 9 对）：
  · CurrencyPageReq / CurrencyPageResp          — 币种列表分页
  · CreateCurrencyReq / CreateCurrencyResp       — 新增币种
  · UpdateCurrencyReq / UpdateCurrencyResp       — 编辑币种
  · StatusCurrencyReq / StatusCurrencyResp       — 状态切换
  · GetBaseCurrencyReq / GetBaseCurrencyResp     — 查基准币种
  · SetBaseCurrencyReq / SetBaseCurrencyResp     — 设基准币种
  · RateListReq / RateListResp                   — 汇率展示
  · RateLogPageReq / RateLogPageResp             — 汇率日志分页
  · OperationLogPageReq / OperationLogPageResp   — 操作日志分页

请求/响应（RPC 6 对）：
```

### 10.4 为什么 4 个 Tab 不共用一套 PageReq/PageResp？

```
看似可以统一，实际不建议：

DepositRecordItem  有 methodName（充值方式）、payType（支付类型）
WithdrawRecordItem 有 accountNo（提现账号）、feeAmount（手续费）、auditStatus（审核状态）
ExchangeRecordItem 有 fromCurrency、toCurrency、exchangeRate（双币种+汇率）
RewardRecordItem   有 rewardType（奖励类型）、auditProgress（稽核进度）

如果统一为一个 RecordItem：
  · 大量 optional 字段，调用方不知道该读哪些
  · 每种类型只有 40-60% 的字段有值
  · 前端需要额外判断 recordType 来决定渲染逻辑

分开的好处：
  · 类型安全——编译期就知道有哪些字段
  · 前端按 Tab 直接用，不需要 if/switch
  · 与 ser-live 的 live_room / live_consume / live_audience 分文件同理
```

### 11.3 关键设计要点

```thrift
// 可选：引用 wallet.thrift 中的 WalletInfo 结构
include "wallet.thrift"

// 1. 所有资金操作的统一入参模式
struct WalletCreditReq {
    1: required i64    userId
    2: required string currencyCode
    3: required i32    walletType       // 1中心 2奖励 3主播 4代理 5场馆
    4: required string amount           // 入账金额 — string（DECIMAL精度）
    5: required i32    bizType          // 业务类型：1充值 2补单 3活动奖励 ...
    6: required string orderNo          // 业务订单号（幂等键）
    7: optional string remark           // 备注
}

// 2. 统一返回模式：变更后余额 + 操作结果
struct WalletCreditResp {
    1: string balanceAfter              // 变更后余额 — string
    2: i64    transactionId             // 流水ID
}

// 3. BetDebit 特殊：优先级扣款，先扣中心再扣奖励
struct BetDebitReq {
    1: required i64    userId
    2: required string currencyCode
    3: required string amount           // 总扣款金额
    4: required string orderNo          // 投注订单号（幂等键）
}

struct BetDebitResp {
    1: string centerDeducted            // 从中心钱包扣除的金额
    2: string bonusDeducted             // 从奖励钱包扣除的金额
    3: string centerBalanceAfter        // 中心钱包剩余
    4: string bonusBalanceAfter         // 奖励钱包剩余
}

// 4. BetCredit 特殊：按扣款时的比例分别返奖
struct BetCreditReq {
    1: required i64    userId
    2: required string currencyCode
    3: required string amount           // 总返奖金额
    4: required string orderNo          // 结算订单号
    5: required string centerRatio      // 中心钱包占比（如 "0.7"）
    6: required string bonusRatio       // 奖励钱包占比（如 "0.3"）
}

// 5. CheckBalance：前置校验，不扣款
struct CheckBalanceReq {
    1: required i64    userId
    2: required string currencyCode
    3: required string amount           // 需要的金额
    4: optional i32    walletType       // 不传=中心+奖励合计
}

struct CheckBalanceResp {
    1: bool   sufficient                // 是否充足
    2: string available                 // 可用余额
}

// 6. ManualCredit/Debit 需要指定目标钱包类型（4种都可能）
struct ManualCreditReq {
    1: required i64    userId
    2: required string currencyCode
    3: required i32    walletType       // 1中心 2奖励 3主播 4代理
    4: required string amount
    5: required string orderNo          // 财务审核订单号
    6: required string remark           // 人工操作必须有备注
}
```

### 11.4 为什么部分 RPC 接口要独立命名（BetDebit/GiftDebit/BonusCredit），而不全部用通用 WalletCredit/WalletDebit？

```
可以全部用 WalletCredit/WalletDebit + bizType 区分，但独立接口有明确优势：

1. BetDebit 有独特的"优先级扣款"逻辑
   · 先扣中心钱包，不足再扣奖励钱包
   · 返回 centerDeducted + bonusDeducted 两个字段
   · WalletDebit 的通用返回结构无法表达

2. BetCredit 有独特的"按比例返奖"逻辑
   · 入参需要 centerRatio + bonusRatio
   · WalletCredit 的通用入参无此字段

3. 调用方视角的清晰度
   · 游戏服务调 BetDebit() 语义清晰
   · 调 WalletDebit(bizType=4) 需要查文档确认 4 是什么

4. 后续演化独立性
   · BetDebit 未来可能加入"场馆钱包参与"逻辑
   · 独立接口可以自由扩展入参，不影响通用接口

5. 通用接口仍然保留
   · WalletCredit/WalletDebit 用于：充值入账、补单、提现扣款等"单钱包操作"场景
   · 特殊接口用于：投注/礼物/奖励等"有独特逻辑"的场景
   · 两者互不冲突
```

| 时间戳 | `i64` | 毫秒时间戳（项目统一） |
| 布尔 | `bool` | 是否充足、是否启用 |
| 列表 | `list<T>` | 列表项集合 |
| 字典 | `map<K,V>` | 批量查询结果 |

### 12.6 字段修饰符约定

| 场景 | 修饰符 | 说明 |
|------|--------|------|
| 请求的必填字段 | `required` | userId、amount、currencyCode 等 |
| 请求的筛选字段 | `optional` | status、startTime、endTime、pageNo 等 |
| 响应的所有字段 | 不标注或 `optional` | 保持灵活性（项目现有惯例） |

### 14.2 handler.go 的实现模式

```go
// ser-wallet/handler.go — 参照 ser-app/handler.go 模式
type WalletServiceImpl struct {
    currencySer  *ser.CurrencyService
    accountSer   *ser.AccountService
    depositSer   *ser.DepositService
    withdrawSer  *ser.WithdrawService
    exchangeSer  *ser.ExchangeService
    recordSer    *ser.RecordService
}

// 每个方法一行委托，零业务逻辑
func (s *WalletServiceImpl) CreateCurrency(ctx context.Context, req *ser_wallet.CreateCurrencyReq) (*ser_wallet.CreateCurrencyResp, error) {
    return s.currencySer.CreateCurrency(ctx, req)
}

func (s *WalletServiceImpl) WalletCredit(ctx context.Context, req *ser_wallet.WalletCreditReq) (*ser_wallet.WalletCreditResp, error) {
    return s.accountSer.Credit(ctx, req)
}
```
