## 一、为什么 IDL 设计是第一步

### 1.1 IDL 先行开发模式

> 依据：工程认知c-3.md 第五节

工程采用 **Thrift IDL 先行** 的开发模式：

```
定义 IDL (.thrift)
  → gen_kitex.sh 生成代码 (kitex_gen/)
    → handler.go 实现 Service 接口
      → internal/ser/ 业务逻辑
        → internal/rep/ 数据访问
```

**不定义 IDL 就没有代码可写**。所有 RPC 方法签名、请求/响应结构体都从 IDL 生成。

### 1.2 IDL 的三重角色

| 角色 | 说明 |
|------|------|
| **代码生成的源头** | handler.go 的接口签名、Request/Response 结构体、Client/Server 桩代码全部从 IDL 生成 |
| **跨团队的合同** | 财务模块调用 E1~E6、钱包调用 F1~F4，双方以 IDL 为接口契约 |
| **系统设计的骨架** | IDL 的文件拆分和方法分组反映了业务领域的边界划分 |

### 1.3 当前阶段 IDL 设计的原则

```
✅ 确定：文件数量、文件职责、方法分组、命名规范
✅ 确定：方法签名（方法名 + Req/Resp 结构体名）
⚠️ 屏蔽：具体字段定义（评审后细化，不钉死）
⚠️ 屏蔽：字段类型细节（如某个字段是 i32 还是 string 的讨论）
❌ 不做：编写完整可编译的 .thrift 文件（那是编码阶段的事）
```

## 二、工程现有 IDL 模式分析

### 2.1 现有服务 IDL 文件统计

> 依据：对 `/Users/mac/gitlab/common/idl/` 下 72 个 .thrift 文件的完整分析

| 服务 | 文件数 | 方法数（约） | 业务复杂度 | 文件拆分策略 |
|------|--------|------------|----------|------------|
| ser-user | 12 | 50+ | 高 | 按功能子域拆：users/follow/blacklist/kyc/rpc/... |
| ser-live | 9 | 60+ | 最高 | 按业务+接入方式拆：room/consume/audience/api/rpc |
| ser-app | 5 | 20+ | 中 | 按功能块拆：banner/category/strategy/s3 |
| ser-buser | 4 | 30+ | 中 | 按管理对象拆：menu/role/user |
| ser-item | 4 | 20+ | 中 | 按功能拆：item/publish/s3 |
| ser-exp | 4 | 15+ | 中 | 按实体拆：level/rules/user_exp |
| ser-rtc | 4 | 15+ | 中 | 按功能拆：rtc/mapping/proxy |
| ser-bcom | 4 | 15+ | 低 | 按功能拆：base/black/white |
| ser-ip | 3 | 10+ | 低 | 按类型拆：blackList/whiteList |
| ser-kyc | 3 | 10+ | 低 | 按角色拆：kyc(B端)/user_kyc(C端) |
| ser-i18n | 2 | 10+ | 低 | service + model |
| ser-s3 | 2 | 5+ | 低 | service + model |

### 2.2 提炼出的拆分规律

**规律一：每个服务有且仅有一个 `service.thrift` 作为入口**

```
service.thrift 的职责：
  ① 声明 namespace
  ② include 同目录下的所有其他 .thrift 文件
  ③ 定义唯一的 service XXXService { ... }
  ④ 所有 RPC 方法都在这一个 service 块中
```

**规律二：按业务子域拆分模型文件**

```
不按"请求/响应"拆分 — 不把所有 Req 放一个文件、所有 Resp 放一个文件
不按"B端/C端"拆分 — 同一业务域的 B端和 C端 struct 放在一起
按业务子域拆分 — 币种相关结构体放一个文件、充值相关放一个文件
```

**规律三：文件数量与方法数量正相关**

```
方法数 5~15   → 2~3 个文件（service + 1~2个model）
方法数 15~30  → 4~5 个文件
方法数 30~50  → 5~9 个文件
方法数 50+    → 9~12 个文件
```

**规律四：跨服务调用的接口用独立 `_rpc.thrift` 文件**

```
ser-live：live_rpc.thrift 单独存放 RPC 供外部调用的接口
ser-user：user_rpc.thrift 单独存放 RPC 供外部调用的接口

作用：
  ① 语义上明确"这些接口是给外部服务调用的"
  ② 其他服务可以通过 include 只关注自己需要的接口定义
  ③ 修改内部接口时不影响外部依赖的文件
```

### 2.3 关键规范提取（必须遵循）

| 规范项 | 规则 | 工程依据 |
|--------|------|---------|
| namespace | `namespace go ser_wallet` | 所有服务统一 `ser_` + 服务名 |
| 文件名 | 小写下划线：`wallet_core.thrift` | 工程统一 snake_case |
| 结构体名 | 大驼峰：`CreateRechargeOrderReq` | PascalCase 统一 |
| 字段名 | 小驼峰：`currencyCode`, `pageNo` | camelCase 统一 |
| 方法名 | 大驼峰：`GetWalletBalance` | PascalCase 统一 |
| 参数 | 固定 `1: XxxReq req` | 所有方法统一一个参数 |
| 注释 | `// 中文注释` 在方法上方 | 工程统一风格 |
| 字段分隔 | 分号 `;` | 新文件统一用分号 |
| 引用 | `include "xxx.thrift"` | 前缀 = 文件名去后缀 |

## 三、ser-wallet 的 IDL 设计挑战

### 3.1 接口规模

> 依据：接口评估c-3.md

| 分类 | 数量 | 说明 |
|------|------|------|
| 币种配置 B端 | 10 | 币种CRUD + 汇率管理 + 日志 |
| 币种配置 C端 | 3 | 币种列表 + 汇率查询 |
| 币种配置 RPC暴露 | 7 | 供其他模块调用的汇率/币种接口 |
| 钱包核心 C端 | 3 | 余额查询 + 币种切换 |
| 充值 C端 | 8 | 充值方式 + 创建订单 + 回调 |
| 兑换 C端 | 2 | 预览 + 创建兑换 |
| 提现 C端 | 9 | 提现订单 + 账户管理 |
| 记录 C端 | 5 | 4Tab记录 + 详情 |
| 钱包 RPC暴露 | 6 | E1~E6 核心资金操作 |
| **合计** | **~53** | 不含 RPC 调用依赖（那是 Client 端，不定义在自己的 IDL 中） |

**约 53 个方法**，对标工程中 ser-live（60+方法，9个文件）和 ser-user（50+方法，12个文件），ser-wallet 属于 **大型服务**。

### 3.2 两个上下文、一个 Service

> 依据：架构设计c-3.md 第三节领域边界

ser-wallet 包含两个限界上下文（币种配置 + 钱包业务），但只有一个 Kitex Service（`WalletService`）。IDL 拆分需要在文件层面反映两个上下文的边界，但 service.thrift 中只有一个 `service WalletService { ... }`。

```
coin/currency  →  配置域相关的 struct  →  几个 .thrift 文件
wallet/core    →  业务域相关的 struct  →  几个 .thrift 文件
                                          ↓
                                    service.thrift
                                  (唯一入口，include 全部)
```

### 3.3 需要考虑的设计问题

| 问题 | 分析 |
|------|------|
| **文件数量** | 太少（1~2个）→ 单文件过大，维护困难；太多（12+）→ 过度拆分，增加理解成本 |
| **拆分维度** | 按业务子域（推荐）还是按接入层（B/C/RPC）？ |
| **RPC 独立** | E1~E6 和 R1~R7 是否单独放一个 rpc.thrift？ |
| **公共结构体** | 币种信息、子钱包余额等被多个文件引用的结构体放哪？ |
| **实体结构体 vs Req/Resp** | 是否需要像 ser-user 那样单独定义实体（如 `CurrencyItem`、`WalletBalanceItem`）？ |

## 四、文件拆分方案评估

### 4.1 方案A：最小拆分（4 个文件）

```
common/idl/ser-wallet/
├── service.thrift       # 入口 + WalletService 定义
├── currency.thrift      # 币种配置全部（B端+C端+RPC）~20方法
├── wallet.thrift        # 钱包全部（余额+充值+兑换+提现+记录）~27方法
└── wallet_rpc.thrift    # 钱包 RPC 暴露 E1~E6  ~6方法
```

| 优点 | 缺点 |
|------|------|
| 文件数最少，结构简单 | currency.thrift 和 wallet.thrift 体量过大（各40+结构体） |
| 维护点少 | 充值/兑换/提现/记录/账户混在一起，难以定位 |
| — | 多人协作时频繁冲突（编辑同一文件） |

**评估：不推荐。** 方法数 53 个，4 个文件意味着每个模型文件平均承载 15~20 个方法的 Req/Resp，体量过大。工程中同等规模的 ser-live（60方法）用了 9 个文件。

### 4.2 方案B：按业务子域拆分（8 个文件）

```
common/idl/ser-wallet/
├── service.thrift       # 入口 + WalletService 定义
├── currency.thrift      # 币种配置（B端+C端）~13方法
├── currency_rpc.thrift  # 币种配置 RPC 暴露 R1~R7  ~7方法
├── wallet.thrift        # 钱包核心（余额+币种切换）~3方法
├── recharge.thrift      # 充值（方式+订单+回调+奖金）~8方法
├── exchange.thrift      # 兑换（预览+创建）~2方法
├── withdraw.thrift      # 提现（订单+限额+账户管理）~9方法
└── record.thrift        # 记录查询（4Tab+详情）~5方法
```

| 优点 | 缺点 |
|------|------|
| 每个文件职责清晰，按业务子域划分 | 文件数 8 个，中等 |
| 符合工程模式（ser-live 9个、ser-user 12个） | exchange.thrift 只有 2 个方法，略显单薄 |
| 查找方便：充值问题找 recharge.thrift | — |
| RPC 暴露接口独立（currency_rpc / wallet_rpc），语义明确 | — |
| 多人协作友好，修改不同文件不冲突 | — |

**评估：推荐。** 最符合工程既有模式。

### 4.3 方案C：细粒度拆分（11 个文件）

```
common/idl/ser-wallet/
├── service.thrift
├── currency.thrift          # 币种配置 B端
├── currency_rate.thrift     # 汇率管理
├── currency_rpc.thrift      # 币种 RPC
├── wallet.thrift            # 钱包核心
├── wallet_rpc.thrift        # 钱包 RPC（E1~E6）
├── recharge.thrift          # 充值
├── exchange.thrift          # 兑换
├── withdraw.thrift          # 提现订单
├── withdraw_account.thrift  # 提现账户管理
└── record.thrift            # 记录查询
```

| 优点 | 缺点 |
|------|------|
| 粒度最细，每个文件极度聚焦 | 11 个文件偏多，include 关系复杂 |
| — | 币种配置拆成 3 个文件有点碎 |
| — | exchange.thrift 和 withdraw_account.thrift 方法都很少（2~5个） |
| — | 超过工程中大多数服务的文件数 |

**评估：过度拆分。** 方法数 53 个配 11 个文件，平均每个文件不到 5 个方法。ser-user 有 50+方法配 12 文件，但它包含了 OpenAPI schema 定义等额外内容。对于纯 Req/Resp 定义的 ser-wallet 来说，11 个文件粒度过细。

### 4.4 方案D：合并 RPC（7 个文件）

```
common/idl/ser-wallet/
├── service.thrift
├── currency.thrift      # 币种配置全部（B+C+RPC）~20方法
├── wallet.thrift        # 钱包核心 + RPC暴露（余额+E1~E6）~9方法
├── recharge.thrift      # 充值 ~8方法
├── exchange.thrift      # 兑换 ~2方法
├── withdraw.thrift      # 提现 ~9方法
└── record.thrift        # 记录 ~5方法
```

| 优点 | 缺点 |
|------|------|
| 文件数少于方案B | RPC 接口混在 B/C 端接口中，不易识别 |
| 实现思路c-3.md 2.2节原始建议的方案 | 财务团队只需关注 E1~E6，但要去 wallet.thrift 里找，不方便 |
| — | currency.thrift 包含 20 个方法的结构体，偏大 |

**评估：可行但非最优。** RPC 接口没有独立文件在工程中也有先例（ser-app 就没有 rpc.thrift），但考虑到钱包的 RPC 暴露是跨团队的"合同"，独立出来更清晰。

### 4.5 方案对比总结

| 对比维度 | A(4文件) | B(8文件) | C(11文件) | D(7文件) |
|---------|---------|---------|----------|---------|
| 文件数 | 4 | **8** | 11 | 7 |
| 平均方法/文件 | 13~17 | **5~8** | 3~5 | 7~10 |
| 职责清晰度 | 低 | **高** | 最高 | 中 |
| 维护便利性 | 差 | **好** | 一般 | 好 |
| RPC 识别度 | 中 | **高** | 高 | 低 |
| 符合工程模式 | 偏少 | **最符合** | 偏多 | 略少 |
| 协作友好度 | 差 | **好** | 好 | 中 |
| **推荐** | ❌ | **✅** | ❌ | ⚠️ |

## 五、推荐方案：8 文件方案详解

### 5.1 文件结构

```
common/idl/ser-wallet/
│
├── service.thrift          # 入口文件：WalletService 全部方法定义
│                           # include 下面所有文件
│                           # 53 个方法签名
│
├── currency.thrift         # 币种配置（B端+C端）
│                           # 13 个方法的 Req/Resp + 公共实体
│                           # 包含：CurrencyItem、RateItem 等
│
├── currency_rpc.thrift     # 币种配置 RPC 暴露（供外部服务调用）
│                           # 7 个方法的 Req/Resp
│                           # 包含：R1~R7
│
├── wallet.thrift           # 钱包核心（余额+币种切换）
│                           # 3 个方法的 Req/Resp + 公共实体
│                           # 包含：WalletBalanceItem、SubWalletItem 等
│
├── recharge.thrift         # 充值业务
│                           # 8 个方法的 Req/Resp
│                           # 包含：RechargeOrderItem 等
│
├── exchange.thrift         # 兑换业务
│                           # 2 个方法的 Req/Resp
│                           # 包含：ExchangePreviewItem 等
│
├── withdraw.thrift         # 提现业务（订单+账户管理）
│                           # 9 个方法的 Req/Resp
│                           # 包含：WithdrawOrderItem、WithdrawAccountItem 等
│
└── record.thrift           # 记录查询
                            # 5 个方法的 Req/Resp
                            # 包含：RechargeRecordItem、WithdrawRecordItem 等
```

### 5.2 拆分决策说明

| 文件 | 方法数 | 约Req/Resp结构体数 | 拆分理由 |
|------|--------|-------------------|---------|
| currency.thrift | 13 | ~28（13对+2公共实体） | 币种配置是独立上下文，B端和C端的结构体共享实体（CurrencyItem） |
| currency_rpc.thrift | 7 | ~16（7对+2公共结构） | RPC 暴露是跨团队合同，独立文件便于对方只关注此文件 |
| wallet.thrift | 3 | ~8（3对+2公共实体） | 钱包核心余额是系统最重要的数据结构，独立便于重视 |
| recharge.thrift | 8 | ~18（8对+2实体） | 充值有 3 种方式（USDT/银行/扫码），结构体较多 |
| exchange.thrift | 2 | ~6（2对+2实体） | 虽然只有 2 个方法，但兑换涉及汇率试算、赠送计算等独特结构体 |
| withdraw.thrift | 9 | ~22（9对+4实体） | 提现最复杂：订单创建+限额查询+3种账户管理 |
| record.thrift | 5 | ~14（5对+4实体） | 记录查询的 4Tab 结构体各不相同（充值/提现/兑换/奖励各有字段差异） |

**为什么 exchange.thrift 只有 2 个方法也要独立**：

```
① 兑换的 Req/Resp 结构体与其他业务不共享
   → ExchangePreviewReq 包含 fromCurrency + fromAmount
   → ExchangePreviewResp 包含 toAmount + bonusAmount + lockedRate + expireTime
   → 这些结构体在充值/提现中不复用

② 兑换是独立的业务流程
   → 法币→BSB 单向兑换，与充值/提现的流程完全不同
   → 未来可能扩展（如 BSB→法币 反向兑换、多币种互兑）

③ 工程先例
   → ser-exp 的 vip_upgrade_rules.thrift 只有 3 个方法也单独一个文件
   → ser-bcom 的 white.thrift 也只有几个方法
   → 按业务域拆分时，方法数少不是合并的理由
```

**为什么 wallet_rpc.thrift（E1~E6）不单独拆出**：

```
分析过后决定将 E1~E6 的 Req/Resp 放在 wallet.thrift 中，理由：

① wallet.thrift 目前只有 3 个 C 端方法的结构体
   → 加上 E1~E6 的 6 个方法，总计 9 个方法，体量适中

② E1~E6 的结构体与 C 端余额结构体天然共享
   → GetBalance(E1) 的 Resp 和 GetWalletBalance(W1) 的 Resp 都包含 SubWalletItem
   → CreditWallet(E2) 的 Req 包含 walletType/currencyCode，与余额查询共享枚举

③ 如果需要也可以拆出
   → 评审后如果财务团队要求 E1~E6 完全独立，随时可拆成 wallet_rpc.thrift
   → 目前合并在 wallet.thrift 是更简洁的选择

④ service.thrift 中通过注释分组已足够区分
   → // ========= 钱包RPC供财务调用 (6个) =========
   → 方法签名层面清晰可辨
```

**但 currency_rpc.thrift 为什么要独立**：

```
① R1~R7 的调用方非常广泛
   → 不仅财务调用，竞彩域、用户域都可能调用
   → 独立文件降低了其他团队理解成本

② R1~R7 是纯查询接口，结构体与 B 端 CRUD 结构体不重叠
   → B端：PageCurrencyReq 分页请求，UpdateCurrencyReq 编辑请求
   → RPC：GetPlatformRateReq 只需 currencyCode，结构完全不同

③ 数量合理
   → 7 个方法 + 14 个结构体，单文件体量适中

④ 符合 ser-live、ser-user 的工程先例
   → live_rpc.thrift、user_rpc.thrift 都是独立文件
```

### 5.3 为什么是 8 个文件

```
对标分析：

ser-wallet（53方法，8文件）vs 工程中的大型服务：
  ser-live  （60+方法，9文件） → 比 live 少 1 个文件，合理
  ser-user  （50+方法，12文件）→ user 含 OpenAPI/s3/login_log 等额外文件
  ser-app   （20+方法，5文件） → wallet 方法数是 app 的 2.5 倍，文件数 1.6 倍

每文件平均承载：53/7 ≈ 7.6 个方法（排除 service.thrift）
  → 与 ser-live（60/8 ≈ 7.5）高度一致
```

**注释分组**：用 `// =========` 分隔符 + 分组名 + 方法数标注，共 9 个分组。方便快速定位。

**方法排列顺序**：按业务域分组，每个域内按操作类型排列（查询 → 创建 → 更新 → 状态变更）。

**公共枚举**：include `../common/enum.thrift`，复用工程已有的运行时枚举查询机制。

**总计 54 个方法**（53 + 1 个 GetEnumOptions 公共接口）。

## 八、命名规范与约定

### 8.1 方法命名模式

| 操作类型 | 前缀 | 示例 | 说明 |
|---------|------|------|------|
| B端分页查询 | `Page` | `PageCurrency`, `PageRateLog` | 返回 list + total + pageNo + pageSize |
| C端列表查询 | `List` | `ListEnabledCurrency`, `ListRechargeMethod` | 返回 list（不分页或简单列表） |
| 获取单个 | `Get` | `GetBaseCurrency`, `GetWalletBalance` | 返回单个实体或详情 |
| 创建 | `Create` | `CreateUsdtRecharge`, `CreateWithdraw` | 创建记录 |
| 更新/编辑 | `Update` | `UpdateCurrency` | 更新已有记录 |
| 状态变更 | `Status` | `StatusCurrency` | 启用/禁用 |
| 保存 | `Save` | `SaveBankAccount` | 创建或更新（首次=创建，已有=更新） |
| 切换 | `Switch` | `SwitchCurrency` | 用户级别的偏好切换 |
| 刷新 | `Refresh` | `RefreshRate` | 触发数据更新 |
| 导出 | `Export` | `ExportRateLog` | CSV/Excel 导出 |
| 校验/检查 | `Check`/`Is` | `CheckDuplicate`, `IsCurrencyEnabled` | 返回 bool 或状态 |
| 上传 | `Upload` | `UploadCurrencyIcon` | 文件上传 |

### 8.2 结构体命名模式

| 类型 | 命名规则 | 示例 |
|------|---------|------|
| 请求 | `{Action}{Entity}Req` | `PageCurrencyReq`, `CreateWithdrawReq` |
| 响应 | `{Action}{Entity}Resp` | `PageCurrencyResp`, `CreateWithdrawResp` |
| 列表项 | `{Entity}Item` | `CurrencyItem`, `RechargeMethodItem` |
| 详情/信息 | `{Entity}Info` | `BaseCurrencyInfo`, `WithdrawLimitInfo` |
| 嵌套子项 | `Sub{Entity}Item` | `SubWalletItem` |

### 8.3 分页请求标准字段

```thrift
// B端分页请求通用字段模式（具体字段按需增减）
struct PageXxxReq {
    1: optional i32 pageNo;           // 页码
    2: optional i32 pageSize;         // 每页条数
    3: optional string keyword;       // 搜索关键词（如有）
    4: optional i32 status;           // 状态筛选（如有）
    // ... 业务特定筛选字段
}

// 分页响应通用字段模式
struct PageXxxResp {
    1: list<XxxItem> pageList;        // 列表数据
    2: i64 total;                     // 总记录数
    3: i32 pageNo;                    // 当前页码
    4: i32 pageSize;                  // 每页条数
}
```

### 8.4 金额字段约定

```
所有金额字段在 IDL 中使用 string 类型传输

理由（技术挑战c-3.md 第五节）：
  → Thrift 的 double 存在精度丢失风险
  → JSON 序列化 double 可能产生科学计数法
  → string 全链路无损传输，Go 层用 shopspring/decimal 解析

示例：
  1: string amount;           // 金额 "100.00"
  2: string availableBalance; // 可用余额 "1234.56789012"
  3: string lockedRate;       // 锁定汇率 "0.00038168"
```

### 8.5 枚举值约定

```
IDL 中不使用 Thrift 原生 enum 关键字定义业务枚举

理由：
  → 工程中 enum 使用极少（仅 ser-contentcenter 用了）
  → 大多数服务通过 i32 + 注释表示状态值
  → 运行时枚举查询通过 GetEnumOptions 接口实现

示例：
  1: i32 walletType;      // 子钱包类型：1=中心 2=奖励 3=主播 4=代理
  2: i32 status;          // 状态：0=禁用 1=启用
  3: i32 rechargeType;    // 充值类型：1=USDT 2=银行转账 3=扫码
```

## 十一、IDL 设计决策记录

### ADR-IDL-001：单 Service vs 多 Service

**决策：一个 WalletService 包含全部方法。**

| 方案 | 说明 | 结论 |
|------|------|------|
| 单 Service | `service WalletService { ... 54方法 }` | ✅ 采用 |
| 多 Service | `service CurrencyService {}` + `service WalletService {}` | ❌ 排除 |

排除理由：
- gen_kitex.sh 生成时以 `*service.thrift` 作为入口，一个目录生成一套代码
- 工程中所有服务都是一个 service.thrift 中只定义一个 service
- 多 Service 需要多个 handler.go 或特殊处理，增加复杂度
- 通过注释分组 + 方法前缀已能清晰区分不同业务域

### ADR-IDL-002：金额用 string 不用 double

**决策：所有金额字段用 `string` 传输。**

```
string → Go string → decimal.NewFromString() → 精确计算 → .String() → string
全链路无精度丢失
```

### ADR-IDL-003：RPC 接口是否独立文件

**决策：**
- 币种 RPC（R1~R7）→ 独立为 `currency_rpc.thrift`（调用方广泛，需要独立合同文件）
- 钱包 RPC（E1~E6）→ 合并在 `wallet.thrift`（调用方单一且与核心余额共享实体）

### ADR-IDL-004：不使用 Thrift 原生 enum

**决策：用 `i32` + 注释表示枚举值，不用 `enum` 关键字。**

理由：
- 工程绝大多数服务都这样做
- 运行时枚举通过 `GetEnumOptions` 接口动态获取
- 编译期枚举不灵活，增减枚举值需要重新生成代码

### ADR-IDL-005：不定义全局 base.thrift

**决策：不创建 ser-wallet 自己的 base.thrift 公共定义文件。**

理由：
- 工程中没有全局 base.thrift 的先例
- 分页字段（pageNo/pageSize/total）在每个 Req/Resp 中直接定义，虽然重复但简单直接
- 引入公共 base 需要跨文件 include + 嵌套组合，增加理解成本
- 公共实体（如 CurrencyItem、SubWalletItem）放在对应的业务 .thrift 文件中，通过 include 引用
