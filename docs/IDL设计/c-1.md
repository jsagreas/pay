## 1. 设计思路与分析方法

### 1.1 IDL设计要回答的核心问题

| 问题 | 需要做的决策 |
|------|-------------|
| 拆几个文件？ | 一个大文件 vs 按域拆分 vs 按端拆分 |
| 每个文件放什么？ | struct定义归属、公共类型提取 |
| service怎么定义？ | 一个service还是多个、方法如何分组 |
| 命名怎么统一？ | 方法名、结构体名、字段名的规范 |
| 跨模块怎么引用？ | 币种配置会被其他服务include吗 |

### 1.2 分析依据

1. **工程现有模式**：扫描了全部71个IDL文件，提炼了统一规范（非凭空设计）
2. **接口需求清单**：前序分析中已梳理的43-46个接口（C端22 + B端5 + RPC对外16）
3. **参照服务规模**：ser-wallet接口量与ser-live（39方法/9文件）相当，属于"中大型服务"

### 1.3 设计原则

| 原则 | 说明 |
|------|------|
| **遵循工程规范** | 命名、结构、注释风格与现有19个服务保持一致 |
| **按业务域拆分** | 一个域一个文件，职责清晰，不混放 |
| **屏蔽实现细节** | IDL只定义"接口长什么样"，不涉及数据库表、缓存策略等内部实现 |
| **适度弹性** | 字段编号预留空间，optional多用少用required，细节可评审后调整 |
| **B/C分离** | B端和C端返回的数据结构可能不同（C端脱敏/精简），分开定义 |

## 2. 工程现有IDL规范提炼

通过分析全部71个IDL文件，提炼出以下统一规范：

### 2.1 文件组织规范

```
common/idl/ser-{service}/
├── service.thrift           # 主文件：namespace + include + service定义
├── {domain1}.thrift         # 域文件：struct定义（Req/Resp/数据结构）
├── {domain2}.thrift         # 域文件
├── {domain}_rpc.thrift      # RPC文件（可选）：供其他服务调用的接口结构
└── ...
```

**规律**：
- 每个服务**有且仅有一个** `service.thrift`，定义唯一的 `{Name}Service`
- 域文件只放 struct 定义，不放 service 声明
- 代码生成脚本只处理 `*service.thrift`，其他文件通过 include 被间接处理

### 2.2 命名规范

| 元素 | 格式 | 示例 |
|------|------|------|
| 文件名 | 小写下划线 | `wallet.thrift`、`currency_config.thrift` |
| namespace | `ser_{service}` | `namespace go ser_wallet` |
| Service名 | `{Name}Service` | `WalletService` |
| 方法名 | 大驼峰（动词+名词） | `CreateDepositOrder`、`GetUserBalance` |
| Struct名 | 大驼峰 + Req/Resp | `CreateDepositOrderReq`、`CreateDepositOrderResp` |
| 字段名 | 小驼峰 | `userId`、`currencyCode`、`walletType` |
| 数据结构名 | 大驼峰（名词） | `WalletInfo`、`CurrencyItem`、`DepositOrderItem` |

### 2.3 字段规范

| 规范 | 说明 | 示例 |
|------|------|------|
| 序号 | 从1开始连续编号 | `1: i64 id;` |
| 修饰符 | required=必填，optional=可选 | `1: required string amount;` |
| 注释 | 行尾中文注释 | `1: i64 userId; // 用户ID` |
| 金额 | **string类型** | `1: string amount; // 金额（string避免精度丢失）` |
| 时间 | **i64毫秒时间戳** | `1: i64 createAt; // 创建时间（毫秒）` |
| ID | **i64** | `1: i64 userId;` |
| 状态/枚举 | **i32** | `1: i32 status; // 1=待支付 2=成功 3=失败` |
| 分页 | `pageNo` + `pageSize` | required，带默认值 |
| api标签 | body/query | `(api.body="xxx", api.query="xxx")` |

### 2.4 分页查询标准模板

```thrift
// 请求
struct XxxPageReq {
    1: required i32 pageNo;       // 页码
    2: required i32 pageSize;     // 每页数量
    3: optional i32 status;       // 筛选条件...
}

// 响应
struct XxxPageResp {
    1: list<XxxItem> pageList;    // 数据列表
    2: i64 total;                 // 总记录数
    3: i32 pageNo;                // 当前页码
    4: i32 pageSize;              // 每页数量
}
```

### 2.5 Service方法签名模板

```thrift
service XxxService {
    // 中文注释说明方法用途
    domain.XxxResp MethodName(1: domain.XxxReq req)
}
```

- 单参数：`(1: XxxReq req)`
- 无参数：`()` —— 直接空括号
- 返回类型引用域文件：`domain.XxxResp`

## 3. 文件拆分方案与对比

### 3.1 方案A：极简拆分（3个文件）

```
ser-wallet/
├── service.thrift      # service定义
├── wallet.thrift       # 所有钱包+充提兑结构
└── wallet_rpc.thrift   # RPC对外结构
```

| 优点 | 缺点 |
|------|------|
| 文件少，好管理 | wallet.thrift会膨胀到500+行，难维护 |
| 适合小型服务 | 充值、提现、兑换混在一起，职责不清 |

**结论**：不适合。ser-wallet接口量40+，属于中大型服务，不适合极简拆分。

### 3.2 方案B：按业务域拆分（6个文件）

```
ser-wallet/
├── service.thrift          # service定义
├── wallet.thrift           # 钱包核心（余额、概览）
├── deposit.thrift          # 充值相关
├── withdrawal.thrift       # 提现相关
├── exchange.thrift         # 兑换相关
├── currency.thrift         # 币种配置（B端管理+对外查询）
└── wallet_rpc.thrift       # RPC对外接口（供其他服务调用）
```

| 优点 | 缺点 |
|------|------|
| 每个文件职责单一，边界清晰 | 文件数量中等 |
| 与产品原型的模块划分一致 | 需要管理include关系 |
| 单文件行数可控（100-200行） | — |
| 参照ser-live（9文件39方法），比例合理 | — |

**结论**：**推荐方案**。6个域文件 + 1个service = 7个文件，与工程中ser-live（9文件）、ser-user（13文件）规模一致。

### 3.3 方案C：按端拆分（按C/B/RPC分文件）

```
ser-wallet/
├── service.thrift
├── wallet_c.thrift         # C端所有接口
├── wallet_b.thrift         # B端所有接口
└── wallet_rpc.thrift       # RPC接口
```

| 优点 | 缺点 |
|------|------|
| 按调用方分类 | 同一个业务的充值Req/Resp被拆到不同文件 |
| 端隔离 | wallet_c.thrift会很大（22个接口的结构体） |
| — | 与工程现有模式不一致（现有按域拆分） |

**结论**：不推荐。按端拆分打断了业务内聚性，且与工程现有风格不一致。

### 3.4 方案对比总结

| 维度 | 方案A(3文件) | 方案B(7文件) | 方案C(4文件) |
|------|-------------|-------------|-------------|
| 与工程风格一致 | 一般 | **一致** | 不一致 |
| 业务内聚性 | 差 | **好** | 差 |
| 单文件可维护性 | 差(500+行) | **好(100-200行)** | 一般(300+行) |
| 参照服务比例 | 偏少 | **合理** | 偏少 |
| 多人协作友好 | 差 | **好(各改各的文件)** | 一般 |

**最终选择：方案B — 按业务域拆分**

## 4. 推荐方案：文件结构设计

### 4.1 文件清单

```
common/idl/ser-wallet/
│
├── service.thrift          # [1] 服务入口：include + WalletService 定义
│                           #     预计：~80行（纯方法声明）
│
├── wallet.thrift           # [2] 钱包核心：余额查询、概览、通用结构
│                           #     预计：~120行
│
├── deposit.thrift          # [3] 充值域：充值方式、创建订单、记录查询
│                           #     预计：~150行
│
├── withdrawal.thrift       # [4] 提现域：提现方式、创建订单、账户管理、记录
│                           #     预计：~200行（提现流程最复杂）
│
├── exchange.thrift         # [5] 兑换域：汇率查询、执行兑换、记录
│                           #     预计：~100行
│
├── currency.thrift         # [6] 币种配置：B端管理 + 对外查询
│                           #     预计：~150行
│
└── wallet_rpc.thrift       # [7] RPC对外：其他服务调用钱包的接口结构
                            #     预计：~200行（16个RPC方法的Req/Resp）
```

### 4.2 每个文件的职责边界

| 文件 | 放什么 | 不放什么 |
|------|--------|---------|
| service.thrift | service声明、include | 任何struct定义 |
| wallet.thrift | 余额查询、钱包概览、通用数据结构（WalletInfo等） | 充值/提现/兑换的具体业务 |
| deposit.thrift | 充值方式、创建充值、USDT支付页、充值记录 | 提现相关 |
| withdrawal.thrift | 提现方式、创建提现、提现账户、提现记录、限额 | 充值相关 |
| exchange.thrift | 兑换汇率、执行兑换、兑换预览、兑换记录 | 币种管理 |
| currency.thrift | B端币种管理（CRUD）、汇率日志、公共币种查询结构 | 钱包余额相关 |
| wallet_rpc.thrift | 其他服务调用的结构（余额操作、批量查询等） | C/B端接口的结构 |

### 4.3 与产品模块的对应关系

```
产品原型模块                    IDL文件
──────────────                ──────────
钱包首页/余额                → wallet.thrift
充值（方式选择→下单→记录）    → deposit.thrift
提现（方式选择→下单→账户→记录）→ withdrawal.thrift
兑换（汇率→执行→记录）       → exchange.thrift
币种管理（B端配置+汇率日志）  → currency.thrift
对外RPC（财务/投注调用）      → wallet_rpc.thrift
```

## 5. 各IDL文件职责与接口设计

### 5.1 service.thrift — 服务入口

**职责**：定义 WalletService，include所有域文件，声明全部方法。

**接口分组方式**（参照ser-live用注释分区块）：

```
WalletService
│
├── ========== C端 · 钱包基础 ==========
│   ├── GetCurrencyListWithBalance    // 获取币种列表及余额
│   ├── GetWalletBalance              // 查询当前币种钱包余额
│   └── GetWalletOverview             // 钱包概览/聚合信息
│
├── ========== C端 · 充值 ==========
│   ├── GetDepositMethods             // 获取充值方式列表
│   ├── CreateDepositOrder            // 创建充值订单
│   ├── GetUsdtPaymentInfo            // USDT支付页信息
│   ├── GetDepositOrderDetail         // 充值订单详情
│   ├── GetDepositRecordList          // 充值记录列表
│   └── CheckDuplicateDeposit         // 重复充值检查
│
├── ========== C端 · 提现 ==========
│   ├── GetWithdrawalMethods          // 获取提现方式列表
│   ├── CreateWithdrawalOrder         // 创建提现订单
│   ├── GetWithdrawalLimit            // 提现限额查询
│   ├── SaveWithdrawalAccount         // 保存提现账户
│   ├── GetWithdrawalAccount          // 获取提现账户
│   ├── GetWithdrawalOrderDetail      // 提现订单详情
│   └── GetWithdrawalRecordList       // 提现记录列表
│
├── ========== C端 · 兑换 ==========
│   ├── GetExchangeRate               // 获取兑换汇率
│   ├── ExecuteExchange               // 执行兑换
│   ├── PreviewExchange               // 兑换预览
│   └── GetExchangeRecordList         // 兑换记录列表
│
├── ========== C端 · 奖励 ==========
│   ├── GetRewardRecordList           // 奖励记录列表
│   └── GetBonusActivityList          // 可用奖金活动列表
│
├── ========== B端 · 币种管理 ==========
│   ├── QueryCurrencyList             // 币种列表查询（B端）
│   ├── UpdateCurrency                // 编辑币种配置
│   ├── SetBaseCurrency               // 设置基准币种
│   ├── QueryRateChangeLog            // 汇率日志列表
│   └── RefreshRealtimeRate           // 手动刷新实时汇率
│
└── ========== RPC · 对外接口 ==========
    ├── RpcGetUserBalance             // 查询用户余额
    ├── RpcBatchGetBalance            // 批量查询余额
    ├── RpcCreditDeposit              // 充值入账
    ├── RpcFreezeBalance              // 冻结余额
    ├── RpcUnfreezeBalance            // 解冻余额
    ├── RpcConfirmWithdrawal          // 确认提现扣款
    ├── RpcManualCredit               // 人工加款
    ├── RpcManualDebit                // 人工减款
    ├── RpcSupplementCredit           // 补单入账
    ├── RpcConsumeDebit               // 消费扣款
    ├── RpcRewardCredit               // 返奖入账
    ├── RpcGetExchangeRate            // 查询汇率
    ├── RpcGetCurrencyList            // 查询币种列表
    ├── RpcGetCurrencyDetail          // 查询币种详情
    └── RpcGetBaseCurrency            // 查询基准币种
```

**方法总计**：37个（C端22 + B端5 + RPC10余额操作 + RPC4币种查询 — 部分按评审可选调整）

### 5.2 wallet.thrift — 钱包核心域

**放什么**：钱包余额查询、概览、通用结构体。

**接口与结构设计**：

| 方法 | 请求 | 响应 | 说明 |
|------|------|------|------|
| GetCurrencyListWithBalance | 无参 | CurrencyBalanceListResp | C端首页：所有币种+对应余额 |
| GetWalletBalance | GetWalletBalanceReq | GetWalletBalanceResp | 指定币种的各类型钱包余额 |
| GetWalletOverview | 无参 | WalletOverviewResp | 钱包聚合信息 |

**核心结构体**：

```
// 币种余额项（C端展示用）
CurrencyBalanceItem
├── currencyCode    string    // 币种代码（USD、USDT、BSB）
├── currencyName    string    // 币种名称
├── currencyType    i32       // 币种类型 1=法币 2=加密 3=平台
├── icon            string    // 币种图标URL
├── totalBalance    string    // 总余额（各钱包求和）
├── decimalPlaces   i32       // 小数位数（展示精度）
└── thousandSep     string    // 千分位符

// 钱包余额信息
WalletBalanceInfo
├── walletType        i32     // 钱包类型 1-5
├── availableBalance  string  // 可用余额
├── frozenBalance     string  // 冻结余额
└── totalBalance      string  // 总余额

// 钱包概览
WalletOverviewResp
├── currencyList      list<CurrencyBalanceItem>    // 各币种余额
└── hasUnreadReward   bool                          // 是否有未读奖励
```

**设计说明**：
- 余额字段全部用 `string`（在Go层用decimal处理，前端直接展示）
- `CurrencyBalanceItem` 是C端展示用的聚合结构，屏蔽了内部分钱包类型的细节
- `WalletBalanceInfo` 是详情页用的明细结构，展示各钱包类型余额

### 5.3 deposit.thrift — 充值域

**放什么**：充值流程的所有接口结构。

**接口与结构设计**：

| 方法 | 请求 | 响应 | 说明 |
|------|------|------|------|
| GetDepositMethods | GetDepositMethodsReq | GetDepositMethodsResp | 充值方式列表+档位+汇率 |
| CreateDepositOrder | CreateDepositOrderReq | CreateDepositOrderResp | 创建充值订单 |
| GetUsdtPaymentInfo | GetUsdtPaymentInfoReq | GetUsdtPaymentInfoResp | USDT支付页信息 |
| GetDepositOrderDetail | GetDepositOrderDetailReq | DepositOrderDetailResp | 订单详情 |
| GetDepositRecordList | DepositRecordPageReq | DepositRecordPageResp | 充值记录分页 |
| CheckDuplicateDeposit | CheckDuplicateDepositReq | CheckDuplicateDepositResp | 重复充值检查 |

**核心结构体**：

```
// 充值方式项
DepositMethodItem
├── methodType     i32       // 方式类型 1=USDT 2=银行 3=电子钱包
├── methodName     string    // 方式名称
├── minAmount      string    // 最小充值金额
├── maxAmount      string    // 最大充值金额
├── currency       string    // 对应币种
├── exchangeRate   string    // 当前汇率（展示用）
└── presetAmounts  list<string>  // 预设金额档位

// 创建充值订单请求
CreateDepositOrderReq
├── currencyCode    string   // 充值币种
├── amount          string   // 充值金额
├── methodType      i32      // 充值方式
└── bonusActivityId i64      // 奖金活动ID（可选，0=不参与）

// 创建充值订单响应
CreateDepositOrderResp
├── orderId         string   // 订单号
├── paymentUrl      string   // 支付跳转URL（法币方式）
├── paymentType     i32      // 支付类型（1=URL跳转 2=USDT支付页）
└── expireTime      i64      // 订单过期时间

// USDT支付页信息
GetUsdtPaymentInfoResp
├── orderId         string   // 订单号
├── walletAddress   string   // 收款USDT地址
├── networkType     string   // 网络类型（TRC20/ERC20）
├── amount          string   // 待支付USDT金额
├── qrCodeUrl       string   // 二维码图片URL
└── expireTime      i64      // 过期时间

// 充值订单详情
DepositOrderDetailResp
├── orderId         string
├── currencyCode    string
├── amount          string
├── actualAmount    string   // 实际到账金额
├── exchangeRate    string   // 下单时汇率
├── methodType      i32
├── status          i32      // 1=待支付 2=成功 3=失败
├── createAt        i64
└── completeAt      i64

// 充值记录项（列表展示）
DepositRecordItem
├── orderId         string
├── currencyCode    string
├── amount          string
├── status          i32
├── methodType      i32
└── createAt        i64

// 充值记录分页
DepositRecordPageReq
├── pageNo          i32      // required
├── pageSize        i32      // required
├── currencyCode    string   // optional 筛选
└── status          i32      // optional 筛选

DepositRecordPageResp
├── pageList        list<DepositRecordItem>
├── total           i64
├── pageNo          i32
└── pageSize        i32

// 重复充值检查
CheckDuplicateDepositReq
├── currencyCode    string
└── amount          string

CheckDuplicateDepositResp
├── duplicateCount  i32      // 重复订单数量
└── allowCreate     bool     // 是否允许继续创建
```

### 5.4 withdrawal.thrift — 提现域

**放什么**：提现全流程（方式→下单→账户→记录），是最复杂的一个域文件。

**接口与结构设计**：

| 方法 | 请求 | 响应 | 说明 |

## 6. 公共结构与枚举设计

### 6.1 是否需要新增公共IDL文件？

**分析**：
- 工程现有的 `common/enum.thrift` 是通用枚举查询结构，钱包可以include它
- 钱包的枚举值（wallet_type、deposit_status等）**不需要**在IDL中定义为Thrift enum
- 原因：这些枚举值在Go代码的const中定义，前端通过枚举查询接口获取，IDL只用i32传递

**结论**：**不需要新增公共IDL文件**。钱包include现有的 `common/enum.thrift` 即可。

### 6.2 枚举值在IDL中的体现方式

工程的做法是**注释标注**，不用Thrift enum语法：

```thrift
// 正确做法（工程规范）：
1: i32 walletType;   // 钱包类型：1=中心 2=奖励 3=主播 4=代理 5=场馆

// 不采用的做法：
// enum WalletType { CENTER = 1, REWARD = 2, ... }  ← 工程中未使用Thrift enum
```

**原因**：
- Thrift enum生成的Go代码不够灵活（不支持中文label）
- 工程统一通过 `GetEnumOptions` 接口动态获取枚举，不硬编码在IDL中
- 注释已足够表达含义，实际值由Go const定义

### 6.3 钱包相关枚举值参照（注释用）

| 枚举名 | 值 | 含义 |
|--------|---|------|
| wallet_type | 1=中心 2=奖励 3=主播 4=代理 5=场馆 | 钱包类型 |
| currency_type | 1=法币 2=加密 3=平台 | 币种类型 |
| deposit_status | 1=待支付 2=成功 3=失败 | 充值状态 |
| withdrawal_status | 1=待风控 2=风控驳 3=待财务 4=财务驳 5=待出款 6=失败 7=成功 | 提现状态 |
| log_type | 1-12（详见数据模型文档） | 流水类型 |
| deposit_method_type | 1=USDT 2=银行 3=电子钱包 | 充值方式 |
| withdrawal_method_type | 1=USDT 2=银行卡 3=电子钱包 | 提现方式 |
| manual_reason_type | 1-8（详见数据模型文档） | 人工原因 |
| currency_status | 0=禁用 1=启用 | 币种状态 |

### 7.2 外部服务引用钱包

其他服务如果需要引用钱包的结构（如财务模块需要用到RPC请求结构），通过：

```thrift
// 在 ser-finance/service.thrift 中
include "../ser-wallet/wallet_rpc.thrift"
```

**设计要点**：`wallet_rpc.thrift` 是对外暴露的"契约"，其他域文件不需要被外部引用。

### 7.3 引用关系图

```
                    ┌──────────────────────┐
                    │ common/enum.thrift   │
                    └──────────┬───────────┘
                               │ include
                    ┌──────────┴───────────┐
                    │  service.thrift       │
                    │  (WalletService)      │
                    └──┬──┬──┬──┬──┬──┬────┘
          include      │  │  │  │  │  │
         ┌─────────────┘  │  │  │  │  └─────────────┐
         │     ┌──────────┘  │  │  └──────────┐     │
         │     │     ┌───────┘  └───────┐     │     │
         ▼     ▼     ▼                  ▼     ▼     ▼
     wallet deposit withdrawal    exchange currency wallet_rpc
     .thrift .thrift .thrift      .thrift  .thrift  .thrift
                                                      │
                                                      │ 被外部include
                                                      ▼
                                              ser-finance等服务
```

### 7.4 钱包内部文件之间是否需要互相引用？

**分析**：

| 场景 | 是否需要 | 说明 |
|------|---------|------|
| deposit 引用 wallet 的结构？ | 不需要 | 各域文件自包含各自的Req/Resp |
| exchange 引用 currency 的结构？ | 不需要 | 汇率信息在exchange中独立定义 |
| wallet_rpc 引用 wallet 的结构？ | 不需要 | RPC的结构独立定义（RpcWalletBalance vs WalletBalanceInfo） |

**结论**：6个域文件之间**没有互相引用**，只有 service.thrift 引用它们。这是最简洁的依赖关系。

## 8. 接口命名与分组规范

### 8.1 方法命名规范

| 操作类型 | 前缀 | 示例 |
|---------|------|------|
| 查询单个 | Get | `GetWalletBalance`、`GetDepositOrderDetail` |
| 查询列表 | Get...List / Query... | `GetDepositRecordList`、`QueryCurrencyList` |
| 创建 | Create | `CreateDepositOrder`、`CreateWithdrawalOrder` |
| 修改 | Update / Set | `UpdateCurrency`、`SetBaseCurrency` |
| 执行动作 | Execute / Refresh / Check / Save | `ExecuteExchange`、`RefreshRealtimeRate` |
| 预览 | Preview | `PreviewExchange` |
| RPC对外 | Rpc前缀 | `RpcGetUserBalance`、`RpcCreditDeposit` |

**RPC接口为什么加Rpc前缀**：
- 区分同名场景：`GetExchangeRate`（C端用）vs `RpcGetExchangeRate`（其他服务用）
- 参数不同：C端从context取userId，RPC需要显式传userId
- 返回不同：C端展示汇率+赠送比例，RPC返回入款/出款/平台三个汇率
- 参照工程已有模式：ser-user有 `users.thrift`（C端）和 `user_rpc.thrift`（RPC）的分离

### 8.2 C端 vs B端接口的命名区分

| 维度 | C端接口 | B端接口 |
|------|---------|---------|
| 命名 | `Get/Create/Execute` | `Query/Update/Set` |
| 参数 | 简洁（userId从ctx取） | 完整（含筛选条件） |
| 响应 | 精简（只返前端需要的） | 详细（含审计字段） |

**工程中B端接口使用Query前缀**（参照ser-bcom的QueryClientBlackCountry），C端使用Get前缀（参照ser-user的GetUserDetail）。钱包遵循这一约定。

### 8.3 分组策略（service.thrift中的注释分区块）

```
// ========== C端 · 钱包基础 ==========
// ========== C端 · 充值 ==========
// ========== C端 · 提现 ==========
// ========== C端 · 兑换 ==========
// ========== C端 · 奖励 ==========
// ========== B端 · 币种管理 ==========
// ========== RPC · 对外接口 ==========
```

参照 ser-live 的分区块注释风格（`// ========== 主播端接口 ==========`）。

| 开关/标记 | `bool` | 布尔值 | `true` |
| 列表 | `list<T>` | 数组 | `list<DepositRecordItem>` |
| 映射 | `map<K,V>` | 字典 | `map<i64, list<RpcWalletBalance>>` |
| 文本 | `string` | 通用字符串 | `"备注内容"` |

### 9.2 为什么金额必须用string

```
JSON传输问题：
  { "amount": 0.1 }  → JavaScript解析后可能变成 0.10000000000000001

string传输：
  { "amount": "0.1" } → 前端直接展示，Go端用decimal解析，全程零精度损失
```

**完整数据流**：
```
前端(string) → HTTP JSON(string) → Hertz网关(string) → Thrift IDL(string)
→ Go服务(decimal.NewFromString) → 计算(decimal运算) → GORM(string) → MySQL(DECIMAL)
```

### 9.3 required vs optional 使用原则

| 用法 | 场景 | 说明 |
|------|------|------|
| required | 创建操作的必要字段 | `amount`、`currencyCode` |
| optional | 查询筛选条件 | `status`、`startTime` |
| optional | 可选业务字段 | `bonusActivityId`、`remark` |
| optional | 可能为空的返回字段 | `rejectReason`（只有驳回才有） |

## 12. 总结

### 12.1 文件清单一览

| # | 文件 | 行数估算 | 职责 | Struct数量 |
|---|------|---------|------|-----------|
| 1 | service.thrift | ~80 | include + WalletService(42方法) | 0 |
| 2 | wallet.thrift | ~120 | 余额查询、概览、奖励记录 | ~10 |
| 3 | deposit.thrift | ~150 | 充值全流程 | ~12 |
| 4 | withdrawal.thrift | ~200 | 提现全流程（最复杂） | ~16 |
| 5 | exchange.thrift | ~100 | 兑换全流程 | ~8 |
| 6 | currency.thrift | ~150 | B端币种管理+汇率日志 | ~10 |
| 7 | wallet_rpc.thrift | ~200 | RPC对外接口 | ~25 |
| — | **合计** | **~1000** | — | **~81** |

### 12.2 设计决策总结

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 拆分方式 | 按业务域（7文件） | 与工程规范一致，业务内聚，规模合理 |
| Service数量 | 1个（WalletService） | 工程惯例：一个服务一个Service |
| 公共文件 | 不新增，复用common/enum.thrift | 枚举不在IDL中定义，注释标注即可 |
| 金额类型 | string | 避免精度丢失，全链路一致 |
| C/B/RPC结构 | 分开定义 | 参数和返回值不同，不能混用 |
| RPC命名 | Rpc前缀 | 区分同名方法，参照user_rpc.thrift模式 |
| 域文件间引用 | 无互引 | 保持最简依赖关系 |
| 对外暴露 | 仅wallet_rpc.thrift | 外部服务只需include这一个文件 |

### 12.3 弹性空间

以下内容在评审中可能调整，IDL设计已预留弹性：

| 可能调整项 | 影响 | 应对 |
|-----------|------|------|
| 接口增减 | service.thrift增删方法 | 方法独立，增删不影响其他 |
| 字段调整 | 域文件struct增删字段 | 序号连续编号，optional字段灵活增减 |
| RPC接口变化 | wallet_rpc.thrift调整 | 独立文件，不影响C/B端 |
| 币种配置独立服务 | currency.thrift可能移出 | 文件独立，可直接移到新服务目录 |
| 奖励模块扩展 | wallet.thrift可能拆出reward.thrift | 当前奖励接口只有2个，暂不拆分 |

### 12.4 下一步

IDL设计方案确定后，实施步骤：

1. **创建IDL目录**：`common/idl/ser-wallet/`
2. **编写IDL文件**：按本文档的结构定义编写7个文件
3. **运行代码生成**：`gen_kitex.sh` 生成Go代码到 `common/pkg/kitex_gen/ser_wallet/`
4. **搭建服务骨架**：基于生成代码创建 ser-wallet 的 handler/ser/rep 三层结构
5. **注册RPC客户端**：在 `common/rpc/rpc_client.go` 新增 `WalletClient()`
