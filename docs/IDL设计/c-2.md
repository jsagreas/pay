# ser-wallet IDL 设计方案

> 定位：在 IDL-first 开发模式下，如何设计 ser-wallet 的 Thrift IDL 文件结构
> 原则：符合现有工程 IDL 规范（21 个服务目录的实证归纳），不自创新规范
> 粒度：确定文件拆分策略 + 方法分组 + 类型组织，屏蔽具体字段细节（评审后再钉死）
> 依据：接口评估（~42 个方法）+ 工程 IDL 实证（ser-app/ser-live/ser-item 等参照）

---

# 一、核心决策：几个文件，怎么拆

## 1.1 现有工程的文件拆分规律

```
通过分析所有 21 个服务目录的 IDL 文件数，归纳出规律：

┌─────────────────┬──────────┬───────────────────────────────────────────┐
│ 服务              │ 类型文件数 │ 拆分逻辑                                    │
├─────────────────┼──────────┼───────────────────────────────────────────┤
│ ser-user         │ 12       │ 按子模块拆：user/auth/profile/address/...   │
│ ser-live         │ 8        │ 按角色/场景拆：api/room/audience/consume/...│
│ ser-app          │ 4        │ 按业务实体拆：banner/category/s3/strategy   │
│ ser-buser        │ 3        │ 按管理对象拆：menu/user/role                │
│ ser-item         │ 3        │ 按生命周期拆：item/publish/s3               │
│ ser-exp          │ 3        │ 按子模块拆                                  │
│ ser-bcom         │ 3        │ 按功能拆：enum/switch/base                  │
│ ser-rtc          │ 3        │ 按功能拆                                    │
│ ser-kyc          │ 2        │ 按内外拆：kyc_api/kyc_rpc                   │
│ ser-ip           │ 2        │ 按内外拆                                    │
│ ser-blog/i18n/s3 │ 1        │ 简单服务单文件                              │
│ ser-mall/socket  │ 0        │ 结构直接写在 service.thrift 中               │
├─────────────────┼──────────┼───────────────────────────────────────────┤
│ 规律             │          │ 方法数 >20 → 3~5 个文件                     │
│                  │          │ 方法数 10~20 → 2~3 个文件                   │
│                  │          │ 方法数 <10 → 0~1 个文件                     │
└─────────────────┴──────────┴───────────────────────────────────────────┘
```

## 1.2 ser-wallet 的文件拆分决策

```
ser-wallet 的特点：
  · 约 42 个方法（在所有现有服务中属于中大规模，接近 ser-user 的 12 文件）
  · 但方法不是同质化的——分属两个逻辑模块（钱包 + 币种配置）
  · 方法有清晰的三类受众：C端用户 / B端管理 / RPC服务间调用
  · 参照 ser-live（35+方法，8 个类型文件）的规模

拆分策略：按"业务子模块"拆分类型文件 + service.thrift 汇总

  不按"受众"拆（即不按 C端/B端/RPC 拆文件）的原因：
    · 现有工程中没有这种拆法（ser-live 的 C端/B端方法混在同一个类型文件中）
    · 同一个实体（如充值订单）的 C端查询和 B端查询共用类型定义
    · 按受众拆会导致类型定义跨文件引用，增加复杂度

  按"业务子模块"拆的好处：
    · 与 ser-app（banner/category/s3）、ser-buser（menu/user/role）模式一致
    · 一个文件内聚一个业务域的所有类型（请求/响应/模型）
    · 文件大小适中（每个 100~200 行，不会出现 ser-live 的 446 行巨文件）
```

## 1.3 最终文件清单

```
common/idl/ser-wallet/
├── service.thrift              ← 服务入口：定义 WalletService，包含所有方法签名
├── wallet.thrift               ← 钱包核心：余额查询、币种切换、余额操作(Credit/Debit/...)
├── deposit.thrift              ← 充值域：充值方式、创建订单、奖金活动、充值记录
├── withdraw.thrift             ← 提现域：提现方式、资格校验、创建订单、提现记录
├── exchange.thrift             ← 兑换域：汇率查询、创建兑换、兑换记录
├── currency.thrift             ← 币种配置：币种CRUD、汇率配置、汇率日志
└── record.thrift               ← 通用记录：记录详情、奖励记录（跨业务的通用查询）

共 7 个文件 = 1 个 service + 6 个类型定义文件
```

## 1.4 为什么是这 6 个类型文件

```
┌───────────────────┬──────────┬─────────────────────────────────────────┐
│ 文件               │ 方法数    │ 拆分依据                                  │
├───────────────────┼──────────┼─────────────────────────────────────────┤
│ wallet.thrift     │ ~8       │ 钱包核心实体：余额/钱包信息/余额操作        │
│                   │          │ 对标 ser-app 的 banner.thrift（核心实体）   │
├───────────────────┼──────────┼─────────────────────────────────────────┤
│ deposit.thrift    │ ~8       │ 充值流程独立且完整：方式→订单→奖金→记录     │
│                   │          │ 与提现/兑换无类型交叉                       │
├───────────────────┼──────────┼─────────────────────────────────────────┤
│ withdraw.thrift   │ ~7       │ 提现流程独立：方式→校验→订单→记录           │
│                   │          │ 与充值/兑换无类型交叉                       │
├───────────────────┼──────────┼─────────────────────────────────────────┤
│ exchange.thrift   │ ~4       │ 兑换相对简单但类型独立（有汇率快照等）       │
│                   │          │ 单独拆出可保持文件简洁                      │
├───────────────────┼──────────┼─────────────────────────────────────────┤
│ currency.thrift   │ ~11      │ 币种配置是独立的管理子模块（全是B端+RPC）    │
│                   │          │ 与钱包操作无类型交叉                       │
├───────────────────┼──────────┼─────────────────────────────────────────┤
│ record.thrift     │ ~4       │ 跨业务的通用记录查询（奖励记录、记录详情）    │
│                   │          │ 引用其他文件的模型做聚合返回                 │
└───────────────────┴──────────┴─────────────────────────────────────────┘
```

## 1.5 备选方案对比

```
方案A（选定）：6 个类型文件
  wallet / deposit / withdraw / exchange / currency / record
  ✓ 每个文件 100~200 行，阅读友好
  ✓ 业务边界清晰，一个文件=一个子域
  ✓ 与 ser-app（4文件/21方法）的密度比接近

方案B：4 个类型文件（合并版）
  wallet（含 deposit+withdraw+exchange） / currency / record / rpc
  ✗ wallet.thrift 会膨胀到 300+ 行（接近 ser-live 的 446 行巨文件）
  ✗ 充值/提现/兑换的类型混在一起，查找困难

方案C：3 个类型文件（最精简）
  wallet / currency / rpc
  ✗ wallet.thrift 会膨胀到 400+ 行
  ✗ rpc 文件混合了钱包 RPC 和币种 RPC

方案D：8+ 个类型文件（更细拆分）
  wallet / deposit / withdraw / exchange / currency / rate / record / admin
  ✗ 文件过多，每个只有 50~80 行，过于零碎
  ✗ 增加 include 依赖关系的复杂度

结论：方案A 在文件数量和内聚度之间取得最佳平衡
```

# 二、service.thrift — 服务入口设计

## 2.1 设计方案

```thrift
namespace go ser_wallet

include "wallet.thrift"
include "deposit.thrift"
include "withdraw.thrift"
include "exchange.thrift"
include "currency.thrift"
include "record.thrift"
include "../common/enum.thrift"

// 钱包服务（含币种配置）
service WalletService {

    // ==================== 钱包 · C端接口 ====================

    // 获取钱包信息（当前币种下各钱包余额）
    wallet.GetWalletInfoResp GetWalletInfo(1: wallet.GetWalletInfoReq req)
    // 获取用户可用币种列表
    wallet.GetUserCurrencyListResp GetUserCurrencyList(1: wallet.GetUserCurrencyListReq req)
    // 切换当前操作币种
    wallet.SwitchCurrencyResp SwitchCurrency(1: wallet.SwitchCurrencyReq req)

    // ==================== 充值 · C端接口 ====================

    // 获取充值方式列表
    deposit.GetDepositMethodsResp GetDepositMethods(1: deposit.GetDepositMethodsReq req)
    // 创建充值订单
    deposit.CreateDepositOrderResp CreateDepositOrder(1: deposit.CreateDepositOrderReq req)
    // 重复充值检查
    deposit.CheckDuplicateDepositResp CheckDuplicateDeposit(1: deposit.CheckDuplicateDepositReq req)
    // 充值订单状态查询
    deposit.GetDepositOrderStatusResp GetDepositOrderStatus(1: deposit.GetDepositOrderStatusReq req)
    // 获取充值奖金活动列表
    deposit.GetDepositBonusListResp GetDepositBonusList(1: deposit.GetDepositBonusListReq req)
    // 充值记录列表
    deposit.GetDepositRecordsResp GetDepositRecords(1: deposit.GetDepositRecordsReq req)

    // ==================== 提现 · C端接口 ====================

    // 获取提现方式列表
    withdraw.GetWithdrawMethodsResp GetWithdrawMethods(1: withdraw.GetWithdrawMethodsReq req)
    // 提现资格前置校验
    withdraw.CheckWithdrawEligibilityResp CheckWithdrawEligibility(1: withdraw.CheckWithdrawEligibilityReq req)
    // 创建提现订单
    withdraw.CreateWithdrawOrderResp CreateWithdrawOrder(1: withdraw.CreateWithdrawOrderReq req)
    // 提现记录列表
    withdraw.GetWithdrawRecordsResp GetWithdrawRecords(1: withdraw.GetWithdrawRecordsReq req)

    // ==================== 兑换 · C端接口 ====================

    // 获取兑换汇率及预估金额
    exchange.GetExchangeRateResp GetExchangeRate(1: exchange.GetExchangeRateReq req)
    // 创建兑换订单
    exchange.CreateExchangeOrderResp CreateExchangeOrder(1: exchange.CreateExchangeOrderReq req)
    // 兑换记录列表
    exchange.GetExchangeRecordsResp GetExchangeRecords(1: exchange.GetExchangeRecordsReq req)

    // ==================== 记录 · C端通用 ====================

    // 奖励记录列表
    record.GetRewardRecordsResp GetRewardRecords(1: record.GetRewardRecordsReq req)
    // 记录详情（通用，按类型+ID查）
    record.GetRecordDetailResp GetRecordDetail(1: record.GetRecordDetailReq req)

    // ==================== 钱包 · B端接口 ====================

    // 管理员查看用户钱包
    wallet.AdminGetUserWalletResp AdminGetUserWallet(1: wallet.AdminGetUserWalletReq req)
    // 管理员查看用户流水
    wallet.AdminGetUserFlowResp AdminGetUserFlow(1: wallet.AdminGetUserFlowReq req)

    // ==================== 币种配置 · B端接口 ====================

    // 币种配置列表
    currency.GetCurrencyListResp GetCurrencyList(1: currency.GetCurrencyListReq req)
    // 币种详情
    currency.GetCurrencyDetailResp GetCurrencyDetail(1: currency.GetCurrencyDetailReq req)
    // 新增币种
    currency.CreateCurrencyResp CreateCurrency(1: currency.CreateCurrencyReq req)
    // 编辑币种
    currency.UpdateCurrencyResp UpdateCurrency(1: currency.UpdateCurrencyReq req)
    // 启用/禁用币种
    currency.ToggleCurrencyStatusResp ToggleCurrencyStatus(1: currency.ToggleCurrencyStatusReq req)
    // 获取汇率配置
    currency.GetRateConfigResp GetRateConfig(1: currency.GetRateConfigReq req)
    // 更新汇率配置
    currency.UpdateRateConfigResp UpdateRateConfig(1: currency.UpdateRateConfigReq req)
    // 汇率变更日志
    currency.GetRateChangeLogResp GetRateChangeLog(1: currency.GetRateChangeLogReq req)

    // ==================== 钱包 · RPC接口（供其他服务调用）====================

    // 查询余额
    wallet.QueryBalanceResp QueryBalance(1: wallet.QueryBalanceReq req)
    // 查询钱包详情
    wallet.QueryWalletDetailResp QueryWalletDetail(1: wallet.QueryWalletDetailReq req)
    // 加款/上分
    wallet.CreditResp Credit(1: wallet.CreditReq req)
    // 扣款/下分
    wallet.DebitResp Debit(1: wallet.DebitReq req)
    // 冻结
    wallet.FreezeResp Freeze(1: wallet.FreezeReq req)
    // 解冻
    wallet.UnfreezeResp Unfreeze(1: wallet.UnfreezeReq req)
    // 奖励→中心转账
    wallet.TransferToCenterResp TransferToCenter(1: wallet.TransferToCenterReq req)

    // ==================== 币种配置 · RPC接口 ====================

    // 查询启用币种列表
    currency.GetAvailableCurrenciesResp GetAvailableCurrencies(1: currency.GetAvailableCurrenciesReq req)
    // 查询汇率
    currency.GetExchangeRateRpcResp GetExchangeRateRpc(1: currency.GetExchangeRateRpcReq req)
    // 查询币种信息
    currency.GetCurrencyInfoResp GetCurrencyInfo(1: currency.GetCurrencyInfoReq req)

    // ==================== 枚举（工程统一模式）====================

    // 获取枚举选项
    enum.GetEnumOptionsResp GetEnumOptions(1: enum.GetEnumOptionsReq req)
}
```

## 2.2 设计要点说明

```
① 分组方式：用 ========== 分隔符 + 注释标题
   参照: ser-live 的分组模式（成熟服务的实证做法）
   好处: 打开 service.thrift 一眼就能看到方法分布

② 分组顺序：C端 → B端 → RPC → 枚举
   逻辑: 按"谁调用"分组，最常维护的（C端）放前面
   C端内部再按业务线分：钱包基础 → 充值 → 提现 → 兑换 → 记录

③ 每个方法一行注释
   参照: ser-app 的模式（每个方法上方一行中文注释）
   效果: 新人看 service.thrift 就能理解每个方法的用途

④ 方法命名跟随现有规范
   动词前缀: Get/Create/Update/Toggle/Check/Query
   参照: ser-app 的 CreateBanner/PageBanner/StatusBanner 命名模式

⑤ include "../common/enum.thrift"
   复用共享的枚举机制（GetEnumOptions），与所有现有服务保持一致

⑥ 方法总数: 约 42 个
   C端: 18（钱包3 + 充值6 + 提现4 + 兑换3 + 记录2）
   B端: 10（钱包2 + 币种8）
   RPC: 10（钱包7 + 币种3）
   枚举: 1
   总计: 39 个确定 + 预留弹性
```

# 三、类型文件逐一设计

## 3.1 wallet.thrift — 钱包核心域

```
职责范围:
  · 钱包信息查询（C端）
  · 币种切换（C端）
  · 余额操作 RPC（Credit/Debit/Freeze/Unfreeze）
  · B端管理查询

为什么余额操作 RPC 的类型放在 wallet.thrift 而不是单独的 rpc.thrift:
  · 参照 ser-live: live_rpc.thrift 只有当 RPC 类型完全独立时才拆出
  · ser-wallet 的 RPC 方法（Credit/Debit 等）使用的基础类型（WalletInfo 等）
    和 C端方法是同一套，放在一起更内聚
  · 如果拆出 wallet_rpc.thrift，很多类型需要跨文件引用，反而更复杂
```

### 类型结构规划

```thrift
namespace go ser_wallet

// ==================== 模型 ====================

// 钱包信息（C端展示用）
struct WalletInfo {
    1: i64 currencyId;              // 币种ID
    2: string currencySymbol;       // 币种简称（VND/USDT/BSB）
    3: string currencyIcon;         // 币种图标
    4: string centerBalance;        // 中心钱包余额（展示值，string避免精度丢失）
    5: string rewardBalance;        // 奖励钱包余额
    6: string totalBalance;         // 资金钱包总余额（中心+奖励）
    7: string frozenAmount;         // 冻结金额
    8: string availableBalance;     // 可用余额
}

// 币种选项（切换列表用）
struct CurrencyOption {
    1: i64 currencyId;
    2: string symbol;
    3: string name;
    4: string icon;
    5: i32 currencyType;            // 1=法币 2=加密 3=平台币
    6: bool isCurrent;              // 是否当前选中
}

// 流水记录项（B端管理查看）
struct FlowItem {
    1: i64 id;
    2: i64 userId;
    3: string currencySymbol;
    4: i32 walletType;
    5: i32 flowType;
    6: string amount;
    7: string balanceBefore;
    8: string balanceAfter;
    9: string orderNo;
    10: string remark;
    11: i64 createAt;
}

// ==================== C端 · 请求/响应 ====================

struct GetWalletInfoReq {}

struct GetWalletInfoResp {
    1: WalletInfo walletInfo;
    2: string currentCurrencySymbol;
}

struct GetUserCurrencyListReq {}

struct GetUserCurrencyListResp {
    1: list<CurrencyOption> currencies;
}

struct SwitchCurrencyReq {
    1: required i64 currencyId;
}

struct SwitchCurrencyResp {}

// ==================== B端 · 请求/响应 ====================

struct AdminGetUserWalletReq {
    1: required i64 userId;
    2: optional i64 currencyId;     // 不传则返回所有币种
}

struct AdminGetUserWalletResp {
    1: list<WalletInfo> wallets;
}

struct AdminGetUserFlowReq {
    1: required i64 userId;
    2: optional i64 currencyId;
    3: optional i32 walletType;
    4: optional i32 flowType;
    5: optional i64 startTime;
    6: optional i64 endTime;
    7: optional i32 pageNo;
    8: optional i32 pageSize;
}

struct AdminGetUserFlowResp {
    1: list<FlowItem> pageList;
    2: i64 total;
    3: i32 pageNo;
    4: i32 pageSize;
}

// ==================== RPC · 请求/响应 ====================

struct QueryBalanceReq {
    1: required i64 userId;
    2: required i64 currencyId;
}

struct QueryBalanceResp {
    1: string centerBalance;
    2: string rewardBalance;
    3: string frozenAmount;
    4: string availableBalance;
}

struct QueryWalletDetailReq {
    1: required i64 userId;
    2: required i64 currencyId;
}

struct QueryWalletDetailResp {
    1: list<WalletInfo> wallets;    // 该币种下所有钱包类型
}

struct CreditReq {
    1: required string orderNo;
    2: required i64 userId;
    3: required i64 currencyId;
    4: required i32 walletType;
    5: required i64 amount;         // 最小货币单位整数
    6: required i32 flowType;
    7: optional string remark;
}

struct CreditResp {}

struct DebitReq {
    1: required string orderNo;
    2: required i64 userId;
    3: required i64 currencyId;
    4: required i32 walletType;
    5: required i64 amount;
    6: required i32 flowType;
    7: optional string remark;
}

struct DebitResp {}

struct FreezeReq {
    1: required string orderNo;
    2: required i64 userId;
    3: required i64 currencyId;
    4: required i64 amount;
}

struct FreezeResp {}

struct UnfreezeReq {
    1: required string orderNo;
    2: required i64 userId;
    3: required i64 currencyId;
    4: required i64 amount;
}

struct UnfreezeResp {}

struct TransferToCenterReq {
    1: required string orderNo;
    2: required i64 userId;
    3: required i64 currencyId;
    4: required i64 amount;
}

struct TransferToCenterResp {}
```

### 设计说明

```
① 金额类型选择: C端展示用 string，RPC 内部传输用 i64
   · GetWalletInfoResp.centerBalance = string "1000.50"（前端直接展示，不丢精度）
   · CreditReq.amount = i64 100050000（最小货币单位整数，内部计算用）
   · 参照架构设计: IDL 对外用 string，内部用 i64

② RPC 方法的 Req 使用 required:
   · Credit/Debit/Freeze 等资金操作的关键字段必须 required
   · 缺少任何一个字段都不应该执行
   · 参照 ser-item 的 ItemCreateReq（创建类方法全 required）

③ C端查询的 Req 大多为空结构体:
   · GetWalletInfoReq {} — 用户信息从 context 获取（tracer.GetUserId(ctx)）
   · 不通过请求参数传 userId — 防止越权
   · 参照 ser-app 的 ListHomeCategory()（无参数方法）

④ 空响应结构体:
   · CreditResp {} / SwitchCurrencyResp {} — 操作成功即返回空，失败走 BizErr
   · 参照 ser-app 的 DeleteBannerResp {} / UpdateBannerResp {}
```

## 3.2 deposit.thrift — 充值域

```
职责范围:
  · 充值方式获取
  · 创建充值订单
  · 重复充值检查
  · 充值订单状态查询
  · 奖金活动列表
  · 充值记录查询
```

### 类型结构规划

```thrift
namespace go ser_wallet

// ==================== 模型 ====================

// 充值方式
struct DepositMethod {
    1: i64 methodId;
    2: string methodName;           // 如"银行转账""TRC20""QRIS"
    3: string methodIcon;
    4: i32 methodType;              // 方式分类
    5: string minAmount;            // 最低充值额
    6: string maxAmount;            // 最高充值额
}

// 充值订单（记录列表项）
struct DepositOrderItem {
    1: i64 id;
    2: string orderNo;
    3: string methodName;
    4: string amount;
    5: string actualAmount;         // 实际到账
    6: i32 status;                  // 0=待支付 1=已完成 2=支付失败 3=已取消
    7: i64 createAt;
}

// 奖金活动
struct DepositBonusItem {
    1: i64 activityId;
    2: string activityName;
    3: string bonusRate;            // 奖金比例（如"10%"）
    4: string minDeposit;           // 最低充值门槛
    5: i32 auditMultiple;          // 稽核倍数
}

// ==================== 请求/响应 ====================

struct GetDepositMethodsReq {}

struct GetDepositMethodsResp {
    1: list<DepositMethod> methods;
}

struct CreateDepositOrderReq {

# 四、关键设计决策与依据

## 4.1 金额类型：string vs i64 的边界

```
决策:
  · C端接口（面向前端）: 金额用 string
  · RPC接口（服务间内部）: 金额用 i64

原因:
  · string 对前端友好: 前端 JavaScript 的 Number 类型有精度限制（2^53），
    USDT 金额 100000000（1USDT最小单位）在 JSON 中作为整数传输安全，
    但更大金额可能溢出 → 用 string 传输最安全
  · i64 对内部计算友好: Go 内部用 int64 计算，不需要额外转换
  · 参照 Stripe API: 对外 API 用 string/integer 混合，内部一律 integer

边界判定:
  ┌─────────────────────┬──────────────────────────┐
  │ string 场景           │ i64 场景                   │
  ├─────────────────────┼──────────────────────────┤
  │ 前端展示的余额        │ Credit/Debit 的 amount     │
  │ 前端提交的充值/提现金额│ Freeze/Unfreeze 的 amount  │
  │ 汇率数值              │ 内部服务间传递的金额        │
  │ 手续费展示            │ 流水金额                   │
  └─────────────────────┴──────────────────────────┘
```

## 4.2 C 端 Req 为何多数是空结构体

```
决策: GetWalletInfoReq {}、GetDepositMethodsReq {}、CheckWithdrawEligibilityReq {} 都是空的

原因:
  · userId 从 context 获取: C端请求经过 FontAuth → tracer.GetUserId(ctx) → 服务端知道是谁
  · currencyId 从设置获取: 服务端查 user_wallet_setting 表获取用户当前币种
  · 安全原则: 不让用户在请求中传 userId（防止越权），不让用户传 currencyId（以服务端记录为准）

例外:
  · CreateDepositOrderReq 有字段 — 因为需要用户选择的充值方式和金额
  · CreateWithdrawOrderReq 有字段 — 因为需要用户填写的收款信息
  · SwitchCurrencyReq 有 currencyId — 这是用户主动选择要切换到的币种
```

## 4.3 分页规范

```
决策: 与现有工程完全一致

入参:
  · pageNo（i32，optional）— 页码，从 1 开始
  · pageSize（i32，optional）— 每页条数
  · 放在请求结构的最后两个字段

出参:
  · pageList（list<T>）— 数据列表
  · total（i64）— 总记录数
  · pageNo（i32）
  · pageSize（i32）

参照: ser-app 的 PageBannerReq/Resp、ser-item 的 ItemQueryReq/Resp
```

## 4.4 required vs optional 的使用原则

```
决策:

  required 用于:
    · 创建类操作的必填字段（CreateDepositOrderReq.methodId）
    · RPC 操作的关键字段（CreditReq.orderNo/userId/amount）
    · 查询类操作的必须参数（GetCurrencyDetailReq.id）

  optional 用于:
    · 查询的筛选条件（GetDepositRecordsReq.status）
    · 更新操作的可选修改字段（UpdateCurrencyReq.name）
    · 分页参数（pageNo/pageSize — 不传则使用默认值）

  无修饰符（default）用于:
    · 模型结构中的展示字段（WalletInfo.centerBalance）
    · 响应结构中的所有字段

参照:
  · ser-app/banner.thrift: CreateBannerReq 用 required，PageBannerReq 的筛选用 optional
  · ser-item/item.thrift: ItemCreateReq 全 required，ItemQueryReq 的筛选用 optional
```

## 4.5 跨文件类型引用

```
本方案中各类型文件之间没有互相引用:
  · wallet.thrift 不引用 deposit.thrift（各自独立的模型）
  · deposit.thrift 不引用 withdraw.thrift
  · record.thrift 定义了自己的 RecordDetailInfo 聚合类型（不引用其他文件的订单类型）

这是有意为之:
  · 避免循环依赖
  · 各文件可独立阅读和修改
  · record.thrift 的 RecordDetailInfo 用 extraInfo(JSON string) 承载不同类型的扩展信息
    而不是引用 DepositOrderItem/WithdrawOrderItem（这样做需要 include）

如果未来确实需要跨文件引用:
  · Thrift 支持在类型文件中 include 其他类型文件
  · 参照 ser-app/content_strategy.thrift include "../ser-contentcenter/content.thrift"
  · 但当前设计刻意避免了这种需要
```

# 八、总结

```
核心决策:

  ① 7 个文件（1 service + 6 type）
     不多不少，每个文件 100~200 行，可读性好

  ② 按业务子模块拆分（不按受众拆）
     wallet / deposit / withdraw / exchange / currency / record
