# IDL 设计方案：ser-wallet（钱包 + 币种配置 + 财务）

> 视角：你负责 ser-wallet 的 IDL 设计，核心模块 = 钱包 + 币种配置，参考模块 = 财务（他人负责但 IDL 有联动）
> 方法论：IDL-First — 接口定义先行，gen_kitex.sh 生成桩代码，双方并行开发
> 依据：接口评估文档（81 接口）+ 工程现有 IDL 规范（20 个服务、69 个 .thrift 文件实际分析）
> 阶段：预评审，骨架确定、细节保留弹性

# 一、设计依据 — 从工程事实出发

## 1.1 现有 IDL 规范提炼

通过对工程 `common/idl/` 下 20 个服务目录、69 个 .thrift 文件的逐一分析，提炼出以下**已被验证的工程规范**：

### 目录与文件命名

| 规范项 | 工程实际 | 示例 |
|--------|----------|------|
| 目录名 | `ser-{name}` | `ser-item/`, `ser-buser/`, `ser-live/` |
| namespace | `namespace go ser_{name}` | `namespace go ser_item` |
| 入口文件 | 固定 `service.thrift`，只放 include + service 块，不放 struct | 所有 20 个服务均如此 |
| 实体文件 | 一个业务域一个文件，snake_case 命名 | `item.thrift`, `sys_user.thrift`, `live_room.thrift` |
| service 块 | 每个服务**只有一个** service 块 | LiveService(36方法)、UserService(31方法)，均单一 service |

### 方法签名规范

| 规范项 | 工程实际 | 示例 |
|--------|----------|------|
| 方法命名 | PascalCase，动词 + 名词 | `QueryItemPage`, `CreateItem`, `UpdateItem` |
| 入参 | `(1: entity.XxxReq req)` | `(1: item.ItemQueryReq req)` |
| 返回值 | struct 或 bool，不用 void | `item.ItemQueryResp`, `bool` |
| 无参方法 | 也存在，直接 `()` | `GetUserDetail()`, `UpgradeRule()` |

### 结构体规范

| 规范项 | 工程实际 | 示例 |
|--------|----------|------|
| 命名 | PascalCase + Req/Resp/Info 后缀 | `ItemQueryReq`, `ItemQueryResp`, `ItemInfo` |
| 字段命名 | camelCase | `pageNo`, `pageSize`, `itemCode`, `createAt` |
| 字段编号 | 从 1 开始连续递增 | `1: required i64 id` ... `5: optional string name` |
| required/optional | 必填用 required，筛选/可空用 optional | 请求体必填字段 required，查询条件 optional |

### 类型选择规范

| 数据类型 | Thrift 类型 | 工程依据 |
|----------|-------------|----------|
| ID（用户/订单/雪花） | `i64` | 全工程统一，雪花 ID 为 int64 |
| 时间戳 | `i64` | UnixMilli（毫秒），`createAt i64`, `updateAt i64` |
| 金额 | `i64` | BIGINT 最小单位存储（分/最小精度单位），ser-mall: `price i64`, ser-live: `totalRevenue i64` |
| 状态/枚举 | `i32` | 不用 Thrift enum 关键字，用 i32 + 注释说明取值 |
| 编码/名称 | `string` | `itemCode string`, `itemName string` |
| 开关/标识 | `bool` 或 `i32` | 新服务倾向 bool，旧服务用 i32（0/1） |
| 集合 | `list<T>` | `list<ItemInfo>`, `list<i64>` |

### 分页规范

**请求：**
```thrift
1: required i32 pageNo
2: required i32 pageSize
3: optional string xxx     // 筛选条件
```

**响应（主流写法，ser-item/ser-buser 风格）：**
```thrift
1: optional list<XxxInfo> pageList
2: optional i64 total
3: optional i32 pageNo
4: optional i32 pageSize
```

### include 规范

| 场景 | 写法 | 示例 |
|------|------|------|
| 同目录 | 直接文件名 | `include "wallet.thrift"` |
| 跨目录（common） | 相对路径 | `include "../common/enum.thrift"` |
| 跨目录（其他服务） | 相对路径 | `include "../ser-kyc/user_kyc.thrift"`（仅 entity 文件中使用） |

### 注释与注解

| 规范项 | 工程实际 |
|--------|----------|
| 方法注释 | `//` 单行注释，写在方法上方 |
| 分组注释 | `// ========== 模块名 ==========` 分隔区域（参考 ser-live） |
| HTTP 注解 | RPC-First 的服务不加 `api.body`/`api.query` 注解（ser-item, ser-buser 均无） |

## 1.2 可参考的工程标杆

| 参考服务 | 方法数 | 文件数 | 与 ser-wallet 的相似度 |
|----------|--------|--------|----------------------|
| **ser-live** | 36 | 9（1 service + 8 entity） | **高** — 方法数量接近，多端混合（B端+主播端+观众端+RPC），分组注释清晰 |
| **ser-buser** | 21 | 4（1 service + 3 entity） | **中** — 纯 B 端，entity 按业务域拆分（menu/role/user） |
| **ser-user** | 31 | 11（1 service + 10 entity） | **中** — C 端为主 + RPC，entity 文件最多 |
| **ser-item** | 10 | 4（1 service + 3 entity） | **低** — 方法少但文件结构最规范，适合做 template |

**结论**：ser-wallet 预估 45 个方法，体量介于 ser-live(36) 和更大之间。文件组织参考 ser-live 的分组方式 + ser-item 的命名规范。

# 二、文件组织方案

## 2.1 方案选择分析

### 方案 A：极简（3 文件）

```
ser-wallet/
├── service.thrift       1 service 块，45+ 方法
├── wallet.thrift        钱包所有 struct（余额/充值/提现/兑换/记录/RPC）
└── currency.thrift      币种配置所有 struct
```

**优势**：文件最少，结构最简单
**劣势**：wallet.thrift 会膨胀到 500+ 行，充值/提现/兑换/记录混在一起，查找困难

### 方案 B：按业务域拆分（6 文件）

```
ser-wallet/
├── service.thrift       1 service 块，45+ 方法，按组注释分区
├── currency.thrift      币种配置：CRUD + 汇率 + 日志 + RPC 暴露
├── wallet.thrift        钱包核心：余额查询/可用币种/钱包首页
├── deposit.thrift       充值：充值方式/创建订单/状态查询/奖金活动
├── withdraw.thrift      提现：提现方式/限额/账户信息/创建订单
├── exchange.thrift      兑换：汇率预览/创建兑换
└── (record 合并到 wallet.thrift 或各自订单文件中)
```

**优势**：每个文件职责单一，查找方便
**劣势**：文件数较多，但 ser-user(11 文件)和 ser-live(9 文件)已有先例，不算异常

### 方案 C：折中（5 文件）— 推荐

```
ser-wallet/
├── service.thrift       1 service 块，所有方法签名 + 分组注释
├── currency.thrift      币种配置模块（B 端 CRUD + 汇率 + 日志 + RPC 暴露）
├── wallet.thrift        钱包 C 端（首页/充值/兑换/提现/记录）
├── wallet_admin.thrift  钱包 B 端（后台余额查询/流水查询/冻结解冻）
└── wallet_rpc.thrift    钱包 RPC 暴露（入账/扣款/解冻/加减款/补单/余额查询）
```

**优势**：按"谁调用"的维度拆分，C 端/B 端/RPC 三种调用方的 struct 天然隔离
**劣势**：wallet.thrift 仍然较大（涵盖充值/提现/兑换/记录），但这些 struct 都是 C 端用户操作的，内聚性好

## 2.2 推荐方案：方案 B（按业务域拆分，6 文件）

**选择理由**：

1. **业务域天然隔离**：充值、提现、兑换是三条独立流程线，各有独立的订单表、状态机、请求响应结构，混在一个文件里会造成 struct 命名冲突压力（DepositCreateReq vs WithdrawCreateReq 都在同一文件显得臃肿）

2. **开发分期匹配**：
   - 第一期骨架 → 先建目录和 service.thrift
   - 第二期币种 → currency.thrift 完整定义
   - 第三期核心 → wallet.thrift + exchange.thrift + wallet_rpc.thrift
   - 第四期充提 → deposit.thrift + withdraw.thrift
   - 每一期只需关注对应的 entity 文件

3. **工程先例**：ser-live 有 8 个 entity 文件（live_room, live_api, live_rpc, live_consume 等），ser-user 有 10 个 entity 文件，6 个文件完全在工程规范范围内

4. **RPC 暴露独立**：wallet_rpc.thrift 单独一个文件，财务开发者只需要看这一个文件就知道自己要调什么接口，降低沟通成本。参考 ser-live 的 live_rpc.thrift 和 ser-user 的 user_rpc.thrift，都是 RPC 暴露独立成文件

## 2.3 最终文件结构

```
common/idl/ser-wallet/
├── service.thrift         主入口（include + 1 个 WalletService 块 + 分组注释）
├── currency.thrift        币种配置模块（B 端 + RPC 暴露）~13 个方法的 struct
├── wallet.thrift          钱包首页 + 记录查询（C 端通用）~7 个方法的 struct
├── deposit.thrift         充值模块（C 端）~6 个方法的 struct
├── withdraw.thrift        提现模块（C 端）~6 个方法的 struct
├── exchange.thrift        兑换模块（C 端）~2 个方法的 struct
└── wallet_rpc.thrift      钱包 RPC 暴露（供财务等模块调用）~8+3 个方法的 struct
```

**文件数**：7 个（1 service + 6 entity）
**对标**：ser-live 9 个、ser-user 11 个 — 体量合理

# 三、service.thrift 设计

## 3.1 整体结构

参考 ser-live/service.thrift 的分组注释风格：

```thrift
namespace go ser_wallet

include "currency.thrift"
include "wallet.thrift"
include "deposit.thrift"
include "withdraw.thrift"
include "exchange.thrift"
include "wallet_rpc.thrift"
include "../common/enum.thrift"

service WalletService {

    // ========== 币种配置 · B端管理 ==========

    // 币种列表查询（分页）
    currency.CurrencyPageResp QueryCurrencyPage(1: currency.CurrencyPageReq req)
    // 新增币种
    bool CreateCurrency(1: currency.CurrencyCreateReq req)
    // 编辑币种
    bool UpdateCurrency(1: currency.CurrencyUpdateReq req)
    // 币种启用/禁用
    bool ToggleCurrencyStatus(1: currency.CurrencyToggleReq req)
    // 币种详情
    currency.CurrencyDetailResp QueryCurrencyDetail(1: currency.CurrencyDetailReq req)
    // 汇率设置（新增/编辑）
    bool SaveExchangeRate(1: currency.ExchangeRateSaveReq req)
    // 汇率查询（B端管理页）
    currency.ExchangeRatePageResp QueryExchangeRatePage(1: currency.ExchangeRatePageReq req)
    // 汇率变更日志查询
    currency.RateLogPageResp QueryRateLogPage(1: currency.RateLogPageReq req)
    // 操作日志查询
    currency.OperationLogPageResp QueryOperationLogPage(1: currency.OperationLogPageReq req)

    // ========== 币种配置 · RPC暴露（供钱包/财务等模块调用） ==========

    // 获取可用币种列表
    currency.AvailableCurrencyListResp GetAvailableCurrencyList(1: currency.AvailableCurrencyListReq req)
    // 获取币种详情（单个/批量）
    currency.CurrencyInfoResp GetCurrencyInfo(1: currency.CurrencyInfoReq req)
    // 获取汇率（指定币种对BSB）
    currency.ExchangeRateResp GetExchangeRate(1: currency.ExchangeRateReq req)
    // 获取币种配置参数（精度/限额/费率/T+N）
    currency.CurrencyConfigResp GetCurrencyConfig(1: currency.CurrencyConfigReq req)

    // ========== 钱包 · C端首页 ==========

    // 钱包余额查询（指定币种，返回 4 个子钱包余额）
    wallet.WalletBalanceResp QueryWalletBalance(1: wallet.WalletBalanceReq req)
    // 可用币种列表查询（含各币种余额概览）
    wallet.UserCurrencyListResp QueryUserCurrencyList(1: wallet.UserCurrencyListReq req)

    // ========== 钱包 · C端充值 ==========

    // 获取可用充值方式列表
    deposit.DepositMethodListResp QueryDepositMethodList(1: deposit.DepositMethodListReq req)
    // 创建充值订单
    deposit.DepositCreateResp CreateDepositOrder(1: deposit.DepositCreateReq req)
    // USDT充值地址获取
    deposit.UsdtAddressResp GetUsdtDepositAddress(1: deposit.UsdtAddressReq req)
    // 充值订单状态查询
    deposit.DepositStatusResp QueryDepositStatus(1: deposit.DepositStatusReq req)
    // 可选奖金活动列表
    deposit.BonusActivityListResp QueryBonusActivityList(1: deposit.BonusActivityListReq req)
    // 重复转账预校验
    deposit.DuplicateCheckResp CheckDuplicateDeposit(1: deposit.DuplicateCheckReq req)

    // ========== 钱包 · C端兑换 ==========

    // 获取兑换汇率及预览
    exchange.ExchangePreviewResp PreviewExchange(1: exchange.ExchangePreviewReq req)
    // 创建兑换订单
    exchange.ExchangeCreateResp CreateExchangeOrder(1: exchange.ExchangeCreateReq req)

    // ========== 钱包 · C端提现 ==========

    // 获取可用提现方式列表
    withdraw.WithdrawMethodListResp QueryWithdrawMethodList(1: withdraw.WithdrawMethodListReq req)
    // 提现账户信息查询（已保存的银行卡/电子钱包）
    withdraw.WithdrawAccountResp QueryWithdrawAccount(1: withdraw.WithdrawAccountReq req)
    // 保存/更新提现账户信息
    bool SaveWithdrawAccount(1: withdraw.WithdrawAccountSaveReq req)
    // 创建提现订单
    withdraw.WithdrawCreateResp CreateWithdrawOrder(1: withdraw.WithdrawCreateReq req)
    // 提现限额查询
    withdraw.WithdrawLimitResp QueryWithdrawLimit(1: withdraw.WithdrawLimitReq req)
    // KYC状态查询（聚合转发）
    withdraw.KycStatusResp QueryKycStatus(1: withdraw.KycStatusReq req)

    // ========== 钱包 · C端记录 ==========

    // 充值记录列表
    wallet.RecordPageResp QueryDepositRecordPage(1: wallet.RecordPageReq req)
    // 提现记录列表
    wallet.RecordPageResp QueryWithdrawRecordPage(1: wallet.RecordPageReq req)
    // 兑换记录列表
    wallet.RecordPageResp QueryExchangeRecordPage(1: wallet.RecordPageReq req)
    // 奖励记录列表
    wallet.RecordPageResp QueryRewardRecordPage(1: wallet.RecordPageReq req)
    // 记录详情查询
    wallet.RecordDetailResp QueryRecordDetail(1: wallet.RecordDetailReq req)

    // ========== 钱包 · B端管理 ==========

    // 用户钱包余额查询（后台）
    wallet_rpc.AdminBalanceResp QueryUserBalanceAdmin(1: wallet_rpc.AdminBalanceReq req)
    // 用户钱包流水查询（后台）
    wallet_rpc.AdminFlowPageResp QueryUserFlowAdmin(1: wallet_rpc.AdminFlowPageReq req)
    // 钱包冻结/解冻
    bool ToggleWalletFreeze(1: wallet_rpc.WalletFreezeReq req)

    // ========== 钱包 · RPC暴露（供财务等模块调用） ==========

    // 充值入账
    bool DepositCredit(1: wallet_rpc.DepositCreditReq req)
    // 提现冻结（预扣）
    bool WithdrawFreeze(1: wallet_rpc.WithdrawFreezeReq req)
    // 提现确认扣款
    bool WithdrawConfirm(1: wallet_rpc.WithdrawConfirmReq req)
    // 提现失败解冻
    bool WithdrawUnfreeze(1: wallet_rpc.WithdrawUnfreezeReq req)
    // 人工加款
    bool ManualCredit(1: wallet_rpc.ManualCreditReq req)
    // 人工减款
    bool ManualDebit(1: wallet_rpc.ManualDebitReq req)
    // 人工补单入账
    bool ManualPatch(1: wallet_rpc.ManualPatchReq req)
    // 余额查询（RPC）
    wallet_rpc.RpcBalanceResp QueryBalanceRpc(1: wallet_rpc.RpcBalanceReq req)

    // ========== 通用 ==========

    // 枚举选项查询
    enum.GetEnumOptionsResp GetEnumOptions(1: enum.GetEnumOptionsReq req)
}
```

## 3.3 设计决策说明

**为什么 B 端管理（3 个方法）的 struct 放 wallet_rpc.thrift 而不是独立文件？**

B 端管理只有 3 个方法（余额查询、流水查询、冻结解冻），体量太小不值得单独建文件。且 B 端查询返回的数据结构（AdminBalanceResp、AdminFlowPageResp）与 RPC 暴露的 RpcBalanceResp 有相似性（都是查余额/操作余额），内聚在一个文件中合理。参考 ser-buser 的 sys_user.thrift 同时包含 B 端查询和 RPC 方法的 struct。

**为什么记录查询（5 个方法）的 struct 放 wallet.thrift 而不是独立 record.thrift？**

四类记录列表（充值/提现/兑换/奖励）可以共用同一个 RecordPageReq/RecordPageResp（通过 recordType 字段区分），struct 数量少（请求 1 个 + 响应 1 个 + Info 1 个 + 详情请求 1 个 + 详情响应 1 个 = 5 个 struct），放在 wallet.thrift 的尾部即可。如果后续四类记录的返回字段差异很大，再拆出 record.thrift 也来得及。

**为什么保留 `../common/enum.thrift` 引用？**

工程标配，ser-buser/ser-kyc/ser-mall/ser-app 都引用了。统一的枚举查询入口 `GetEnumOptions` 用于前端下拉框等场景。

# 四、Entity 文件设计

## 4.1 currency.thrift — 币种配置模块

```thrift
namespace go ser_wallet

// ==================== 币种 B端管理 ====================

// 币种列表查询
struct CurrencyPageReq {
    1: required i32 pageNo
    2: required i32 pageSize
    3: optional string currencyCode       // 筛选：币种代码
    4: optional i32 status                // 筛选：启用状态 1=启用 0=禁用
    5: optional i32 currencyType          // 筛选：币种类型 1=法币 2=数字货币
}

struct CurrencyPageResp {
    1: optional list<CurrencyInfo> pageList
    2: optional i64 total
    3: optional i32 pageNo
    4: optional i32 pageSize
}

struct CurrencyInfo {
    1: required i64 id
    2: required string currencyCode       // 币种代码（BSB/VND/USDT）
    3: required string currencyName       // 币种名称
    4: required i32 currencyType          // 1=法币 2=数字货币
    5: required string icon               // 图标URL
    6: required i32 precision             // 精度（小数位数）
    7: required i64 minDeposit            // 最小充值（最小单位）
    8: required i64 maxDeposit            // 最大充值
    9: required i64 minWithdraw           // 最小提现
    10: required i64 maxWithdraw          // 最大提现
    11: required i64 dailyWithdrawLimit   // 每日提现上限
    12: required i32 withdrawFeeType      // 手续费类型 1=固定 2=比例
    13: required i64 withdrawFeeValue     // 手续费值（固定=金额，比例=万分比）
    14: required i32 withdrawTDays        // T+N 天到账
    15: required i32 status               // 1=启用 0=禁用
    16: required i32 sortOrder            // 排序序号
    17: required i64 createAt             // 创建时间
    18: required i64 updateAt             // 更新时间
}

// 新增币种
struct CurrencyCreateReq {
    1: required string currencyCode
    2: required string currencyName
    3: required i32 currencyType
    4: required string icon
    5: required i32 precision
    6: optional i64 minDeposit
    7: optional i64 maxDeposit
    8: optional i64 minWithdraw
    9: optional i64 maxWithdraw
    10: optional i64 dailyWithdrawLimit
    11: optional i32 withdrawFeeType
    12: optional i64 withdrawFeeValue
    13: optional i32 withdrawTDays
    14: optional i32 sortOrder
}

// 编辑币种
struct CurrencyUpdateReq {
    1: required i64 id
    2: optional string currencyName
    3: optional string icon
    4: optional i32 precision
    5: optional i64 minDeposit
    6: optional i64 maxDeposit
    7: optional i64 minWithdraw
    8: optional i64 maxWithdraw
    9: optional i64 dailyWithdrawLimit
    10: optional i32 withdrawFeeType
    11: optional i64 withdrawFeeValue
    12: optional i32 withdrawTDays
    13: optional i32 sortOrder
}

// 启用/禁用
struct CurrencyToggleReq {
    1: required i64 id
    2: required i32 status                // 1=启用 0=禁用
}

// 币种详情查询
struct CurrencyDetailReq {
    1: required i64 id
}

struct CurrencyDetailResp {
    1: optional CurrencyInfo info
}

// ==================== 汇率管理 ====================

// 汇率设置
struct ExchangeRateSaveReq {
```

**设计说明**：

**RPC 暴露的核心设计原则 — 每个写方法必须带 refOrderNo（幂等键）**

这是技术挑战文档中"双层幂等"的 IDL 体现。所有 RPC 暴露的写接口（入账/冻结/扣款/解冻/加减款/补单）都 required 一个 `refOrderNo` 字段，它就是幂等键。调用方（财务）用自己的订单号作为 refOrderNo，钱包服务用 `balance_flow(ref_order_no, change_type)` 的唯一键保证幂等。

**为什么 RPC 接口要显式传 userId 而 C 端不传？**

C 端：userId 从 gateway 的 JWT 解析后注入 ctx（metainfo），service 层通过 `tracer.GetUserId(ctx)` 获取，IDL 不暴露。
RPC 暴露：调用方是财务服务，不经过 gateway，没有 JWT，必须显式传 userId。

**为什么返回 bool 而不是 struct？**

参考工程既有规范 — `bool UpdateItem(1: item.ItemUpdateReq req)`, `bool MenuCreate(1: sys_menu.MenuCreateDataReq req)`。简单的写操作返回 bool 是工程惯例。调用失败时走 Kitex 的 error 机制（BizErr），不需要在返回值里包装错误信息。

# 五、设计决策汇总

## 5.1 核心决策矩阵

| 决策项 | 选择 | 依据 |
|--------|------|------|
| Service 数量 | **1 个 WalletService** | 工程规范（每个 ser-xxx 一个 Service），ser-live 36 方法已验证可行 |
| 文件数量 | **7 个**（1 service + 6 entity） | 介于 ser-buser(4) 和 ser-live(9) 之间，体量匹配 |
| entity 拆分维度 | **按业务域**（currency/wallet/deposit/withdraw/exchange/wallet_rpc） | 充值/提现/兑换三条流程线独立，各有独立订单表和状态机 |
| namespace | `namespace go ser_wallet` | 遵循工程规范 `ser_{name}` |
| 金额类型 | `i64` | BIGINT 最小单位存储，工程已有先例（ser-mall price, ser-live revenue） |
| 状态/枚举类型 | `i32 + 注释` | 不用 Thrift enum 关键字（工程统一做法） |
| 分页规范 | `pageNo/pageSize` 请求 + `pageList/total/pageNo/pageSize` 响应 | 遵循 ser-item 风格（工程主流） |
| HTTP 注解 | **不加** | ser-wallet 是纯 RPC 服务，HTTP 由 gateway 负责，参考 ser-item、ser-buser |
| 方法注释 | `//` 单行注释 + `// ==========` 分组 | 参考 ser-live |
| C 端 userId | **ctx 获取，IDL 不暴露** | 工程统一做法（gateway JWT → metainfo → tracer） |
| RPC userId | **显式参数** | RPC 调用不经过 gateway，必须显式传 |
| 幂等键 | **refOrderNo required** | 所有 RPC 写接口必带，技术挑战中的双层幂等保障 |

## 5.3 关于财务模块 IDL 的说明

财务模块（36 个接口）由他人负责开发，IDL 也由他人定义。但存在以下 IDL 协调需求：

**你需要提供给财务的（你定义）：**
- `wallet_rpc.thrift` 中的全部 struct — 财务调用你的入账/扣款/解冻等接口需要
- 财务执行 gen_kitex.sh 后会基于你的 IDL 生成 `rpc.WalletClient()` 调用代码

**你需要财务提供给你的（他人定义）：**
- 支付配置查询接口的 IDL — 你的充值/提现页面获取可用方式
- 创建渠道支付订单接口的 IDL — 你的充值流程创建订单
- 提交提现审核接口的 IDL — 你的提现流程提交审核

**协调时间点：**
- wallet_rpc.thrift 应**最先定义**并提交到 common/idl/ser-wallet/（实现思路文档第二节确认），解除财务开发阻塞
- 财务的 IDL 定义后，你在 ser-wallet 中通过 `rpc.FinanceClient()` 调用，不需要在自己的 IDL 中定义财务的 struct

# 六、开发分期与 IDL 的关系

| 开发分期 | 需要完成的 IDL | 说明 |
|----------|--------------|------|
| **第一期：骨架搭建** | service.thrift（全部方法签名）+ 所有 6 个 entity 文件（至少 struct 骨架） | IDL 先行，全量定义后 gen_kitex.sh 一次性生成。方法签名先全部定义好，struct 字段可以后续微调 |
| **第二期：币种配置** | currency.thrift 字段确认 | 此时币种配置的 struct 字段应该已经终稿 |
| **第三期：钱包核心** | wallet.thrift + exchange.thrift + wallet_rpc.thrift 字段确认 | wallet_rpc.thrift 确认后通知财务 pull 代码 |
| **第四期：充值提现** | deposit.thrift + withdraw.thrift 字段确认 | 需要与财务协调 IDL 对齐 |

**关键认知**：Thrift IDL 修改后需要重新 gen_kitex.sh，生成的桩代码会更新。所以：
- 方法签名（service.thrift）尽早稳定，避免频繁重新生成
- struct 字段的增删不影响方法签名，只需重新生成，成本低
- 新增方法需要重新生成且可能影响 handler.go，成本略高

# 七、弹性空间 — 预评审阶段的不确定性

以下内容在预评审后可能调整，IDL 设计预留了弹性：

**可能新增的方法**

| 场景 | 可能新增 | 影响文件 |
|------|---------|---------|
| 提现白名单管理 | 2-3 个 B 端方法 | 新建 withdraw_admin.thrift 或并入 wallet_rpc.thrift |
| 对账接口 | 1-2 个 RPC 方法 | 并入 wallet_rpc.thrift |
| 钱包余额快照 | 1 个定时任务触发方法 | 并入 wallet_rpc.thrift |
| C端消息通知 | 0（走消息服务，不在钱包 IDL） | 无影响 |

**可能调整的 struct 字段**

| 场景 | 可能调整 | 影响 |
|------|---------|------|
| 限额规则变复杂 | WithdrawLimitResp 新增字段 | 新增 optional 字段即可 |
| 奖金活动规则变化 | BonusActivityItem 字段调整 | 修改已有字段或新增 |
| 审核层级变化 | 不影响钱包 IDL（审核在财务侧） | 无影响 |
| 新增币种类型 | CurrencyInfo 新增字段 | 新增 optional 字段 |

**核心结论**：当前 IDL 设计的 **方法签名层面** 高度稳定（45 个方法覆盖了所有已确认的需求），**struct 字段层面** 保留了 optional 弹性（预评审后的微调只需新增/修改字段，不需要重构 IDL 文件结构）。
