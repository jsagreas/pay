## 1. 设计决策与依据

### 1.1 决策1：一个 service 还是多个 service？

```
决策：一个 WalletService（单 service 定义）
否决：拆为 WalletService + CurrencyService 两个 service
```

**依据**：

| 维度 | 单 service | 多 service |
|------|-----------|-----------|
| 工程惯例 | ✅ ser-app、ser-live、ser-user 全部是单 service | 无先例 |
| 代码生成 | gen_kitex.sh 只识别单个 service.thrift | 多 service 需改脚本 |
| handler.go | 一个 handler 文件对应一个 service | 多个 handler 拆分复杂 |
| 部署 | 一个进程 = 一个 service | 同一进程多 service 需要额外配置 |
| 路由 | gate 路由统一指向一个 service | 需要按 service 分发 |

**结论**：遵循现有工程的一致模式——一个服务一个 service 定义。钱包和币种配置虽然是两个子模块，但共处一个 ser-wallet 进程，对外只暴露一个 WalletService。

**工程验证**：ser-live 包含房间、观众、消费、审核、设置等多个子模块，全部在一个 LiveService 中。ser-wallet 的子模块（钱包核心、充值、提现、兑换、记录、账户、币种配置）规模相当，同样适合单 service。

### 1.2 决策2：struct 文件拆分粒度

```
决策：按业务域拆分，7 个 struct 文件 + 1 个 service 文件 = 共 8 个文件
否决方案A：全部塞一个文件（太大，可维护性差）
否决方案B：每个接口一个文件（过度拆分，文件碎片化）
```

**依据**：

| 现有参照 | 子模块数 | IDL文件数 | 文件/模块比 |
|---------|---------|----------|------------|
| ser-live | 7个子模块 | 9个文件 | 1.3 |
| ser-user | 6个子模块 | 11个文件 | 1.8 |
| ser-app | 4个子模块 | 5个文件 | 1.25 |
| ser-item | 3个子模块 | 4个文件 | 1.3 |
| **ser-wallet** | **7个子模块** | **8个文件** | **1.14** |

ser-wallet 的 8 个文件 ÷ 7 个子模块 = 1.14，在现有工程的合理范围内（1.1 ~ 1.8）。

### 1.3 决策3：是否需要 enum 独立文件？

```
决策：不单独拆 enum 文件，枚举用 i32 + 注释内联定义
依据：现有工程所有枚举都是 i32 + 注释，不使用 Thrift enum 关键字
```

**工程事实**：
```thrift
// 现有工程的枚举方式（ser-kyc）
struct KycAudit {
    10: i32 status;  // 状态：1:去认证 2:审核中 3:已通过 4:重新认证
}
// 不使用 Thrift enum 关键字
```

枚举值在 Go 代码层定义为 const（`internal/enum/`），IDL 中只声明类型和注释。这样保持 IDL 与现有工程一致。

### 1.4 决策4：是否需要 s3.thrift？

```
决策：不需要。币种图标上传通过 gate-back 路由到现有 ser-s3 服务
依据：ser-wallet 不直接处理文件上传，图标 URL 作为字符串字段传入
```

**参照**：ser-app 的 `s3.thrift` 是因为 banner 图片需要通过 app service 中转上传。币种配置的图标上传可以在 gate-back 层直接调用 ser-s3，拿到 URL 后传入币种配置接口的 `iconUrl` 字段即可。如果评审后确认需要通过 ser-wallet 中转上传，再补充一个 s3.thrift 文件。

## 2. 文件组织方案

### 2.1 最终文件结构

```
common/idl/ser-wallet/
├── service.thrift          # 服务入口：WalletService 定义（include 所有 struct 文件）
├── wallet.thrift           # 钱包核心：钱包信息、余额、币种切换
├── recharge.thrift         # 充值域：充值方式、创建订单、回调、USDT地址
├── withdraw.thrift         # 提现域：提现方式、创建订单、提现账户管理
├── exchange.thrift         # 兑换域：兑换信息查询、执行兑换
```

## 3. 方案对比与选择

### 3.1 为什么不选其他方案

#### 方案X：按端拆分（c_api.thrift / b_api.thrift / rpc.thrift）

```
否决理由：
1. 现有工程无此模式（ser-live 按业务域拆分，不按端拆分）
2. 充值域的 struct 同时被 C 端接口和 RPC 接口引用，按端拆会导致交叉引用
3. 维护时不直觉：修改充值功能要同时改 c_api.thrift 和 rpc.thrift
```

#### 方案Y：只拆 2 个文件（wallet.thrift + currency.thrift）

```
否决理由：
1. wallet.thrift 会包含充值+提现+兑换+记录+账户+RPC 的所有 struct
   预估 50+ 个 struct 挤在一个文件，可维护性差
2. 参照 ser-live 的经验：当 struct 数量超过 20 个，就应该拆分
```

#### 方案Z：拆 12+ 个文件（每个功能点一个文件）

```
否决理由：
1. 过度碎片化，每个文件只有 2-3 个 struct
2. service.thrift 的 include 列表过长
3. 修改一个功能可能涉及多个文件
4. 现有工程最多 11 个文件（ser-user），且 ser-user 有社交/黑名单/VIP 等完全无关的域
```

### 3.2 8 个文件方案的优势

```
✅ 与现有工程一致（ser-live 9个文件，我们8个）
✅ 按业务域拆分，修改某个功能只改一个 struct 文件
✅ 每个文件 4-18 个 struct，不多不少
✅ 文件名即业务域名，一目了然
✅ wallet_rpc.thrift 专门承载对外能力，与 C/B 端接口分离清晰
✅ 新增功能容易定位该放哪个文件
```

## 4. service.thrift — 服务入口

### 4.1 文件职责

- 声明 namespace
- include 所有 struct 文件
- 定义唯一的 WalletService
- 方法按**注释分组**组织（参照 ser-live 模式）
- **不包含任何 struct 定义**

### 4.2 结构骨架

```thrift
namespace go ser_wallet

include "wallet.thrift"
include "recharge.thrift"
include "withdraw.thrift"
include "exchange.thrift"
include "record.thrift"
include "currency.thrift"
include "wallet_rpc.thrift"

service WalletService {

    // ========== C端接口：钱包信息 ==========
    wallet.GetWalletInfoResp        GetWalletInfo(1: wallet.GetWalletInfoReq req)
    wallet.SwitchCurrencyResp       SwitchCurrency(1: wallet.SwitchCurrencyReq req)

    // ========== C端接口：充值 ==========
    recharge.RechargeMethodListResp GetRechargeMethodList(1: recharge.RechargeMethodListReq req)
    recharge.CreateRechargeResp     CreateRechargeOrder(1: recharge.CreateRechargeReq req)
    recharge.RechargeCallbackResp   RechargeCallback(1: recharge.RechargeCallbackReq req)
    recharge.UsdtAddressResp        GetUsdtAddress(1: recharge.UsdtAddressReq req)
    recharge.RepeatCheckResp        CheckRepeatRecharge(1: recharge.RepeatCheckReq req)
    recharge.BonusListResp          GetBonusList(1: recharge.BonusListReq req)

    // ========== C端接口：提现 ==========
    withdraw.WithdrawMethodListResp GetWithdrawMethodList(1: withdraw.WithdrawMethodListReq req)
    withdraw.CreateWithdrawResp     CreateWithdrawOrder(1: withdraw.CreateWithdrawReq req)
    withdraw.AccountListResp        GetWithdrawAccountList(1: withdraw.AccountListReq req)
    withdraw.SaveAccountResp        SaveWithdrawAccount(1: withdraw.SaveAccountReq req)
    withdraw.DeleteAccountResp      DeleteWithdrawAccount(1: withdraw.DeleteAccountReq req)

    // ========== C端接口：兑换 ==========
    exchange.ExchangeInfoResp       GetExchangeInfo(1: exchange.ExchangeInfoReq req)
    exchange.DoExchangeResp         DoExchange(1: exchange.DoExchangeReq req)

    // ========== C端接口：记录 ==========
    record.RecordListResp           GetRecordList(1: record.RecordListReq req)
    record.RecordDetailResp         GetRecordDetail(1: record.RecordDetailReq req)

    // ========== B端接口：币种配置 ==========
    currency.PageCurrencyResp       PageCurrency(1: currency.PageCurrencyReq req)
    currency.CurrencyDetailResp     GetCurrencyDetail(1: currency.CurrencyDetailReq req)
    currency.CreateCurrencyResp     CreateCurrency(1: currency.CreateCurrencyReq req)
    currency.UpdateCurrencyResp     UpdateCurrency(1: currency.UpdateCurrencyReq req)
    currency.StatusCurrencyResp     StatusCurrency(1: currency.StatusCurrencyReq req)
    currency.SetBaseCurrencyResp    SetBaseCurrency(1: currency.SetBaseCurrencyReq req)
    currency.RateLogListResp        GetRateLogList(1: currency.RateLogListReq req)
    currency.RefreshRateResp        RefreshRate(1: currency.RefreshRateReq req)

    // ========== RPC接口：资金操作（供财务等服务调用） ==========
    wallet_rpc.CreditWalletResp     CreditWallet(1: wallet_rpc.CreditWalletReq req)
    wallet_rpc.DebitWalletResp      DebitWallet(1: wallet_rpc.DebitWalletReq req)
    wallet_rpc.GetBalanceResp       GetBalance(1: wallet_rpc.GetBalanceReq req)
    wallet_rpc.FreezeBalanceResp    FreezeBalance(1: wallet_rpc.FreezeBalanceReq req)
    wallet_rpc.UnfreezeBalanceResp  UnfreezeBalance(1: wallet_rpc.UnfreezeBalanceReq req)
    wallet_rpc.ConfirmDebitResp     ConfirmDebit(1: wallet_rpc.ConfirmDebitReq req)
    wallet_rpc.InitWalletResp       InitWallet(1: wallet_rpc.InitWalletReq req)
    wallet_rpc.BatchGetBalanceResp  BatchGetBalance(1: wallet_rpc.BatchGetBalanceReq req)
    wallet_rpc.GetWalletFlowResp    GetWalletFlow(1: wallet_rpc.GetWalletFlowReq req)
    wallet_rpc.GetCurrencyListResp  GetCurrencyList(1: wallet_rpc.GetCurrencyListReq req)
    wallet_rpc.GetExchangeRateResp  GetExchangeRate(1: wallet_rpc.GetExchangeRateReq req)
    wallet_rpc.GetBaseCurrencyResp  GetBaseCurrency(1: wallet_rpc.GetBaseCurrencyReq req)
}
```

### 4.3 设计要点

- 方法名 PascalCase，与现有工程一致（`CreateBanner` / `StatusBanner`）
- 每个方法 1 个入参（req），1 个返回值（resp），遵循现有模式
- 注释分组让 service 文件清晰可读
- 约 35 个方法，与 ser-live（~30方法）规模相当

## 5. wallet.thrift — 钱包核心

### 5.1 文件职责

承载 C 端用户查看钱包信息、切换币种相关的 struct。这是钱包最基础的能力。

### 5.2 包含的 struct 骨架

```thrift
namespace go ser_wallet

// ========== 钱包信息 ==========

struct GetWalletInfoReq {
    // 无需传参，userId 从 context 获取
}

struct GetWalletInfoResp {
    1: string currentCurrency;              // 用户当前选中的币种
    2: list<CurrencyWallet> walletList;     // 各币种钱包列表
}

struct CurrencyWallet {
    1: string currencyCode;                 // 币种代码
    2: string currencyName;                 // 币种名称
    3: string iconUrl;                      // 图标URL
    4: i64 fundBalance;                     // 资金余额（中心+奖励汇总，最小单位）
    5: string fundBalanceDisplay;           // 资金余额格式化展示
    6: i64 centerBalance;                   // 中心钱包余额
    7: string centerBalanceDisplay;         // 中心余额格式化
    8: i64 rewardBalance;                   // 奖励钱包余额
    9: string rewardBalanceDisplay;         // 奖励余额格式化
    // ... 展示规则字段
}

// ========== 币种切换 ==========

struct SwitchCurrencyReq {
    1: required string currencyCode;        // 目标币种代码
}

struct SwitchCurrencyResp {}
```

### 5.3 预估 struct 数：4-6 个

| struct | 说明 |
|--------|------|
| GetWalletInfoReq / Resp | 钱包信息查询 |
| CurrencyWallet | 单币种钱包数据模型 |
| SwitchCurrencyReq / Resp | 切换币种 |

## 6. recharge.thrift — 充值域

### 6.1 文件职责

承载充值相关的所有 struct：充值方式列表、创建充值订单、回调处理、USDT地址、重复转账检查、额外奖金。

### 6.2 包含的 struct 骨架

```thrift
namespace go ser_wallet

// ========== 充值方式 ==========

struct RechargeMethodListReq {
    1: required string currencyCode;
}

struct RechargeMethodListResp {
    1: list<RechargeMethod> methodList;
}

struct RechargeMethod {
    // 方式名、图标、档位列表、金额范围、类型（虚拟币/银行/扫码）
}

// ========== 创建充值订单 ==========

struct CreateRechargeReq {
    1: required string currencyCode;
    2: required i64 amount;                 // 充值金额（最小单位）
    3: required i64 methodId;               // 充值方式ID
    4: optional i64 bonusId;                // 选择的活动奖金ID
}

struct CreateRechargeResp {
    1: string orderNo;                      // 平台订单号
    2: i32 payType;                         // 支付类型 1三方跳转 2展示地址 3展示二维码
    3: optional string payUrl;              // 三方支付URL
    4: optional string payAddress;          // USDT充值地址
    5: optional string qrContent;           // 二维码内容
}

// ========== 充值回调 ==========

struct RechargeCallbackReq {
    // 通道回调参数（签名、订单号、金额、状态等）
    // 具体字段取决于通道接口约定
}

struct RechargeCallbackResp {
    1: bool success;
}

// ========== USDT 地址 ==========

struct UsdtAddressReq {
    1: required string chainType;           // TRC20/ERC20/BEP20
}

struct UsdtAddressResp {
    1: string address;                      // 充值地址
    2: string minAmount;                    // 最低充值金额展示
    3: i32 confirmations;                   // 到账确认数
}

// ========== 重复转账检查 ==========

struct RepeatCheckReq {
    1: required string currencyCode;
    2: required i64 amount;
}

struct RepeatCheckResp {
    1: i32 repeatCount;                     // 近期相同金额充值次数
    2: bool blocked;                        // 是否被拦截（>=3次）
    3: bool warned;                         // 是否需要警告（1-2次）
}

// ========== 额外奖金 ==========

struct BonusListReq {
    1: required string currencyCode;
    2: required i64 amount;                 // 充值金额
}

struct BonusListResp {
    1: list<BonusOption> bonusList;
}

struct BonusOption {
    // 奖金ID、比例、最低充值门槛、奖金金额展示等
}
```

### 6.3 预估 struct 数：10-14 个

充值是钱包最复杂的子模块（涉及方式列表、订单创建、回调、USDT地址、重复检查、奖金），struct 数量最多。

## 7. withdraw.thrift — 提现域

### 7.1 文件职责

承载提现 + 提现账户管理的所有 struct。把提现账户（子模块F）合并到提现域，因为提现账户仅服务于提现场景，放在一起更符合业务内聚。

### 7.2 包含的 struct 骨架

```thrift
namespace go ser_wallet

// ========== 提现方式 ==========

struct WithdrawMethodListReq {
    1: required string currencyCode;
}

struct WithdrawMethodListResp {
    1: list<WithdrawMethod> methodList;
    2: i32 kycStatus;                       // 用户KYC状态，前端据此展示
}

struct WithdrawMethod {
    // 方式名、图标、费率、最低/最高金额、关联提现方式类型
}

// ========== 创建提现订单 ==========

struct CreateWithdrawReq {
    1: required string currencyCode;
    2: required i64 amount;                 // 提现金额（最小单位）
    3: required i64 methodId;               // 提现方式ID
    4: required i64 accountId;              // 提现账户ID
}

struct CreateWithdrawResp {
    1: string orderNo;                      // 提现订单号
}

// ========== 提现账户管理 ==========

struct AccountListReq {}

struct AccountListResp {
    1: list<WithdrawAccount> accountList;
}

struct WithdrawAccount {
    1: i64 id;
    2: i32 accountType;                     // 1银行卡 2电子钱包 3USDT
    3: string accountNo;                    // 卡号/钱包号（脱敏展示）
    4: string accountName;                  // 持有人姓名（脱敏展示）
    5: optional string bankName;            // 银行名称
    6: optional string chainType;           // USDT链类型
    // ...
}

struct SaveAccountReq {
    1: optional i64 id;                     // 有值=编辑，无值=新增
    2: required i32 accountType;
    3: required string accountNo;
    4: required string accountName;
    5: optional string bankName;
    6: optional string chainType;
}

struct SaveAccountResp {
    1: i64 id;
}

struct DeleteAccountReq {
    1: required i64 id;
}

struct DeleteAccountResp {}
```

### 7.3 预估 struct 数：10-12 个

提现域 + 提现账户管理合并，struct 数量与充值域相当。

## 8. exchange.thrift — 兑换域

### 8.1 文件职责

承载法币→BSB兑换相关的 struct。兑换逻辑相对独立且简单。

### 8.2 包含的 struct 骨架

```thrift
namespace go ser_wallet

struct ExchangeInfoReq {
    1: required string currencyCode;        // 法币代码（VND/IDR/THB）
}

    1: string fromCurrency;                 // 源币种
    2: string toCurrency;                   // 目标币种（BSB）
    3: i64 availableBalance;                // 可兑换余额（最小单位）
    4: string availableBalanceDisplay;      // 可兑换余额展示
    5: string rateDisplay;                  // 汇率展示（如 "1 BSB = 2,300 VND"）
    6: i32 bonusPct;                        // 赠送比例（×100存储，如1000=10.00%）
```

## 11. wallet_rpc.thrift — 对外 RPC 能力

### 11.1 文件职责

承载 ser-wallet 向外部服务暴露的**资金操作原语 + 查询能力**的 struct。这些是其他服务（财务、用户、游戏）调用钱包时使用的接口。

### 11.2 为什么单独拆出来

```
理由：
1. RPC 接口的调用方是其他服务（财务、用户等），不是 C/B 端用户
2. RPC 的 Req/Resp 字段与 C/B 端接口不同（更偏数据层面，无展示字段）
3. 参照 ser-live 的 live_rpc.thrift、ser-user 的 user_rpc.thrift
4. 便于其他团队了解"ser-wallet 对外提供了哪些能力"
```

### 11.3 包含的 struct 骨架

```thrift

struct GetWalletFlowResp {
    1: list<WalletFlowItem> flowList;
    2: i64 total;
    3: i32 pageNo;
    4: i32 pageSize;
}

struct WalletFlowItem {
    // 流水ID、用户、币种、钱包类型、方向、业务类型、金额、前后余额、订单号、时间

// ==================== 币种配置能力（供其他服务查询） ====================

// ========== 获取可用币种列表 ==========

    1: optional i32 currencyType;           // 筛选类型，不传返回全部启用币种
```

## 12. 命名规范速查

### 12.1 文件命名

| 规则 | 示例 |
|------|------|
| 目录名 | `ser-wallet`（中划线，与所有现有服务一致） |
| 主文件 | `service.thrift`（固定名，gen_kitex.sh 依赖此名） |
| struct 文件 | `wallet.thrift`、`recharge.thrift`（小写，业务域名） |
| RPC 文件 | `wallet_rpc.thrift`（下划线分隔，参照 user_rpc.thrift / live_rpc.thrift） |

### 12.2 命名空间

```thrift
namespace go ser_wallet    // 所有文件统一
```

### 12.3 方法命名

| 操作 | 命名模式 | 示例 |
|------|---------|------|
| 创建 | Create + 实体 | `CreateRechargeOrder` `CreateCurrency` |
| 查询（分页） | Page + 实体 | `PageCurrency` |
| 查询（列表） | Get + 实体 + List | `GetRechargeMethodList` `GetRecordList` |
| 查询（详情） | Get + 实体 + Detail | `GetCurrencyDetail` `GetRecordDetail` |
| 查询（信息） | Get + 实体 + Info | `GetWalletInfo` `GetExchangeInfo` |
| 更新 | Update + 实体 | `UpdateCurrency` |
| 状态变更 | Status + 实体 | `StatusCurrency` |
| 删除 | Delete + 实体 | `DeleteWithdrawAccount` |
| 执行 | Do + 动作 | `DoExchange` |
| 保存 | Save + 实体 | `SaveWithdrawAccount` |
| 切换 | Switch + 实体 | `SwitchCurrency` |
| RPC 操作 | 动作 + 实体 | `CreditWallet` `FreezeBalance` |

### 12.4 Struct 命名

| 类型 | 命名模式 | 示例 |
|------|---------|------|
| 请求 | 方法名 + Req | `CreateRechargeReq` `GetBalanceReq` |
| 响应 | 方法名 + Resp | `CreateRechargeResp` `GetBalanceResp` |
| 列表请求 | 实体 + ListReq | `RechargeMethodListReq` `AccountListReq` |
| 列表响应 | 实体 + ListResp | `RechargeMethodListResp` `AccountListResp` |
| 数据模型 | 实体名 | `CurrencyWallet` `WithdrawAccount` `RateLogItem` |
| C端简化模型 | 实体 + C 后缀 | 如需区分 B/C 端展示（参照 `BannerItemC` 模式） |

### 12.5 字段规范

| 规则 | 示例 |
|------|------|
| 命名风格 | camelCase（与现有一致） |
| 金额字段 | `i64`（最小单位整数）+ 对应 `string display` 字段 |
| 时间字段 | `i64`（毫秒时间戳） |
| 状态字段 | `i32` + 注释说明枚举值 |
| 布尔字段 | `bool`（Thrift 原生支持） |
| 列表字段 | `list<T>` |
| 映射字段 | `map<K,V>` |
| 可选字段 | `optional` |
| 必传字段 | `required` |
| 字段编号 | 从 1 开始顺序递增，不跳号 |

## 14. 代码生成与工程集成

### 14.1 gen_kitex.sh 生成流程

```
执行命令（在 common/script/ 目录下）：
  ./gen_kitex.sh

脚本行为：
  1. 扫描 common/idl/ser-wallet/service.thrift
  2. 执行 kitex 代码生成：
     kitex -module common/pkg \
       -I common/idl/ser-wallet/ \
       -service ser_wallet \
       -gen-path ./kitex_gen/ \
       common/idl/ser-wallet/service.thrift
  3. 生成产物放入 common/pkg/kitex_gen/ser_wallet/

生成产物：
  common/pkg/kitex_gen/ser_wallet/
  ├── walletservice/          # 服务端/客户端代码
  │   ├── client.go
  │   ├── server.go
  │   ├── invoker.go
  │   └── walletservice.go
  ├── wallet.go               # wallet.thrift 的 Go struct
  ├── recharge.go             # recharge.thrift 的 Go struct
  ├── withdraw.go             # withdraw.thrift 的 Go struct
  ├── exchange.go             # exchange.thrift 的 Go struct
  ├── record.go               # record.thrift 的 Go struct
  ├── currency.go             # currency.thrift 的 Go struct
  ├── wallet_rpc.go           # wallet_rpc.thrift 的 Go struct
  └── k-ser_wallet.go         # service 定义
```

### 14.2 ser-wallet 需要的工程配置

```
步骤1：创建 IDL 目录和文件
  common/idl/ser-wallet/  （8个 .thrift 文件）

步骤2：执行代码生成
  cd common/script && ./gen_kitex.sh

步骤3：创建服务目录
  ser-wallet/
  ├── main.go           # 从生成的模板复制，加入初始化逻辑
  ├── handler.go        # 从生成的模板复制，实现各方法
  └── internal/         # 业务代码

步骤4：注册服务
  common/pkg/consts/namesp/namesp.go  → 添加 EtcdWalletService 常量
  common/rpc/rpc_client.go            → 添加 WalletClient() 函数
  go.work                              → 添加 ./ser-wallet

步骤5：注册网关路由
  gate-font/ → 添加 C 端钱包路由
  gate-back/ → 添加 B 端币种配置路由
```

### 14.3 IDL 迭代策略

```
初次开发时：
  一次性把 8 个文件的骨架建好
  struct 字段先定义核心必须的，预留 field number 空间
  gen_kitex.sh 生成后即可开始编码

需求变更时（评审后新增/删除接口）：
  修改对应的 struct 文件 → 在 service.thrift 增减方法 → 重新 gen_kitex.sh
  不影响其他 struct 文件（按域隔离的优势）

字段新增时：
  在 struct 末尾追加字段（新 field number）
  不修改已有字段的 number（Thrift 向后兼容）
```
