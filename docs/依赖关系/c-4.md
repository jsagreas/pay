### 依赖方向总结

| 调用方 | 被调用方 | 方向 | 说明 |
|--------|---------|------|------|
| 钱包 | 币种配置 | 单向 → | 钱包内部调用币种配置（同服务函数调用） |
| 钱包 | 财务管理 | → | 钱包查财务的支付配置（充值/提现方式） |
| 财务管理 | 钱包 | → | 财务通知钱包做余额变更（上账/冻结/扣款/加减款） |
| 财务管理 | 币种配置 | → | 财务查币种列表和汇率（RPC） |
| 钱包 | ser-kyc | → | 查KYC认证状态 |
| 钱包/币种配置 | ser-blog | → | 写操作审计日志（oneway） |
| 币种配置 | ser-buser | → | 操作人姓名解析 |
| 币种配置 | 三方汇率API | → | 定时拉取实时汇率 |

**关键发现：钱包和财务之间存在双向RPC依赖**，这是整个系统最核心的耦合点。

## 二、数据归属（谁拥有什么数据）

搞清楚数据归属才能搞清楚谁提供数据、谁消费数据：

```
┌─────────────────────────────────────────────────────────┐
│                    数 据 归 属 图                         │
├──────────────┬──────────────────┬───────────────────────┤
│   币种配置    │   多币种钱包     │     财务管理           │
│   (我负责)   │   (我负责)       │     (别人负责)         │
├──────────────┼──────────────────┼───────────────────────┤
│ · 币种定义    │ · 用户钱包余额   │ · 代收/代付通道配置    │
│ · 基准币种    │   (5种子钱包)    │ · 充值/提现方式配置    │
│ · 汇率配置    │ · 冻结金额       │ · 充值订单(C单)       │
│   (实时/平台/ │ · 提现账户信息   │ · 提现订单(T单)       │
│    入款/出款) │   (银行卡/电子   │ · 补单(B)/加款(A)/    │
│ · 汇率浮动%   │    钱包)         │   减款(M)订单         │
│ · 偏差阈值%   │ · 交易流水       │ · 通道轮询规则        │
│ · 汇率变更日志│   (余额变动记录) │ · 审核记录            │
│              │                  │ · 通道/充值/提现统计   │
└──────────────┴──────────────────┴───────────────────────┘
```

**关键结论**：
- **余额归钱包管**，财务不直接改余额，必须通过钱包RPC
- **订单归财务管**，充值/提现订单的创建和状态流转在财务侧
- **币种和汇率归币种配置管**，是所有模块的公共数据源
- **交易流水归钱包管**，每次余额变动钱包自己记一笔，用于C端记录展示和对账

## 三、接口依赖层级（从底层到顶层）

接口不是平等的，有严格的前后置关系。按依赖深度分5层：

```
Layer 4 (最顶层)    记录查询 / 汇率日志 / 数据统计
                          │ 依赖有数据产生
                          │
Layer 3 (回调层)    CreditDeposit / Freeze / Unfreeze / ConfirmDeduct / ManualCredit/Debit
                          │ 依赖订单和业务操作
                          │
Layer 2 (业务层)    CreateDepositOrder / ExecuteExchange / CreateWithdrawalOrder
                          │ 依赖方式查询和钱包基础
                          │
Layer 1 (查询层)    GetDepositMethods / GetWithdrawalInfo / GetExchangePreview / GetBonusActivities
                          │ 依赖钱包基础和外部配置
                          │
Layer 0 (基础层)    币种CRUD / 基准币种 / 汇率同步 / GetWalletBalance / GetAvailableCurrencies
                          │
                    ───── 地 基 ─────
```

### 各层详细说明

#### Layer 0：基础层（无内部依赖，最先实现）

| 接口 | 模块 | 端 | 依赖 | 说明 |
|------|------|-----|------|------|
| SetBaseCurrency | 币种配置 | B | 无 | 整个系统的第一步操作，其他一切依赖基准币种 |
| CreateCurrency | 币种配置 | B | SetBaseCurrency | 有基准币种后才能创建其他币种 |
| QueryCurrencyPage | 币种配置 | B | 有币种数据 | 列表展示 |
| UpdateCurrency | 币种配置 | B | 有币种数据 | 编辑 |
| ToggleCurrencyStatus | 币种配置 | B | 有币种数据 | 启用/禁用 |
| SyncRealtimeRate | 币种配置 | 定时 | 有币种数据 + 三方API | 汇率同步 |
| GetCurrencyList | 币种配置 | RPC | 有币种数据 | 供外部模块查询 |
| GetExchangeRate | 币种配置 | RPC | 有汇率数据 | 供外部模块查询 |
| GetBaseCurrency | 币种配置 | RPC | SetBaseCurrency已执行 | 供外部模块查询 |
| GetWalletBalance | 钱包 | C | 有币种数据(内部) | 用户看余额 |
| GetAvailableCurrencies | 钱包 | C | GetCurrencyList(内部) | 用户切币种 |

#### Layer 1：查询层（依赖 Layer 0 + 外部配置）

| 接口 | 模块 | 端 | 依赖 | 说明 |
|------|------|-----|------|------|
| GetDepositMethods | 钱包 | C | GetAvailableCurrencies + 财务.GetAvailableDepositMethods | 展示充值方式需要币种+财务支付配置 |
| GetWithdrawalInfo | 钱包 | C | GetAvailableCurrencies + 财务.GetAvailableWithdrawalMethods + ser-kyc.KycQuery | 展示提现方式需要币种+财务配置+KYC状态 |
| GetExchangePreview | 钱包 | C | GetWalletBalance + GetExchangeRate(内部) | 兑换预览需要余额+汇率 |
| GetBonusActivities | 钱包 | C | 外部活动模块(待确认) | 独立于其他钱包接口 |
| GetWithdrawalAccounts | 钱包 | C | 无强依赖(有就返回，没有返回空) | 提现账户列表 |
| SaveWithdrawalAccount | 钱包 | C | ser-kyc(姓名校验) | 首次保存提现账户 |

#### Layer 2：业务操作层（依赖 Layer 1，产生订单/交易）

| 接口 | 模块 | 端 | 依赖 | 说明 |
|------|------|-----|------|------|
| CreateDepositOrder | 钱包 | C | GetDepositMethods + GetWalletBalance(重复转账检查) | 创建充值订单 |
| GetUSDTDepositInfo | 钱包 | C | CreateDepositOrder(先有订单) 或独立(展示地址) | USDT收款地址 |
| ExecuteExchange | 钱包 | C | GetExchangePreview + GetWalletBalance(余额够) | 执行兑换 |
| CreateWithdrawalOrder | 钱包 | C | GetWithdrawalInfo + GetWithdrawalAccounts + GetWalletBalance(余额够) | 创建提现订单+冻结 |

#### Layer 3：回调/通知层（依赖 Layer 2 产生的订单，由财务触发）

| 接口 | 模块 | 端 | 前置条件 | 说明 |
|------|------|-----|---------|------|
| CreditDeposit | 钱包 | RPC供他 | 充值订单已创建 + 支付成功回调 | 财务通知钱包上账 |
| FreezeForWithdrawal | 钱包 | RPC供他 | 提现订单已创建 | 冻结提现金额 |
| UnfreezeForWithdrawal | 钱包 | RPC供他 | FreezeForWithdrawal已执行 + 审核拒绝 | 解冻返还 |
| ConfirmWithdrawalDeduct | 钱包 | RPC供他 | FreezeForWithdrawal已执行 + 出款成功 | 确认扣款 |
| ManualCredit | 钱包 | RPC供他 | 财务创建加款+审核通过 | 人工加款 |
| ManualDebit | 钱包 | RPC供他 | 财务创建减款+审核通过 | 人工减款 |
| CreditCorrection | 钱包 | RPC供他 | 财务创建补单+审核通过 | 补单上账 |
| GetUserWalletBalance | 钱包 | RPC供他 | 有用户钱包数据 | 财务查余额(加减款弹窗) |

#### Layer 4：结果查询层（依赖 Layer 2+3 产生的数据）

| 接口 | 模块 | 端 | 依赖 | 说明 |
|------|------|-----|------|------|
| GetTransactionRecords | 钱包 | C | Layer 2+3 产生的交易流水 | 有数据才能查 |
| GetTransactionDetail | 钱包 | C | GetTransactionRecords | 记录详情 |
| GetDepositOrderStatus | 钱包 | C | CreateDepositOrder | 轮询充值状态 |
| QueryRateLogPage | 币种配置 | B | SyncRealtimeRate产生的日志 | 有日志才能查 |

## 四、接口耦合分类

### 4.1 强耦合（必须成对/成组出现，缺一不可）

```
组1：充值生命周期
┌──────────────────────────────────────────────────┐
│ CreateDepositOrder ──→ [用户支付] ──→ CreditDeposit │
│        (创建)              (外部)        (上账)      │
│                                                    │
│ 缺 CreateDepositOrder：CreditDeposit 无订单可上账    │
│ 缺 CreditDeposit：订单永远停在"待支付"              │
└──────────────────────────────────────────────────┘

组2：提现生命周期（状态机）
┌──────────────────────────────────────────────────────────────┐
│ CreateWithdrawalOrder ──→ FreezeForWithdrawal                 │
│         │                        │                            │
│         │               ┌────────┴────────┐                   │
│         │               │                 │                   │
│         │        审核拒绝/出款失败    出款成功                  │
│         │               │                 │                   │
│         │               ▼                 ▼                   │
│         │     UnfreezeForWithdrawal  ConfirmWithdrawalDeduct  │
│         │        (解冻返还)            (确认扣款)              │
│         │                                                     │
│ 这4个接口形成完整的提现状态机：                                │
│   创建→冻结→(解冻 或 扣款)                                    │
│ 缺任何一个，提现流程断裂                                      │
└──────────────────────────────────────────────────────────────┘

组3：兑换预览+执行
┌──────────────────────────────────────────────┐
│ GetExchangePreview ──→ ExecuteExchange        │
│     (看汇率)              (执行)              │
│                                              │
│ 预览提供汇率快照，执行时需校验汇率未过期       │
└──────────────────────────────────────────────┘

组4：币种基础配置
┌──────────────────────────────────────────────┐
│ SetBaseCurrency ──→ CreateCurrency             │
│     (先定基准)        (再建币种)               │
│                                              │
│ 没有基准币种，币种间无法换算                   │
│ 没有币种数据，钱包无法运作                     │
└──────────────────────────────────────────────┘
```

### 4.2 松耦合（有关联但可独立工作）

| 接口A | 接口B | 关联方式 | 为什么松耦合 |
|-------|-------|---------|-------------|
| SaveWithdrawalAccount | CreateWithdrawalOrder | 提现需要账户 | 账户可以提前保存，不必提现时才保存 |
| GetBonusActivities | CreateDepositOrder | 充值可选奖金 | 不选奖金也能充值，奖金是可选项 |
| QueryRateLogPage | SyncRealtimeRate | 日志由同步产生 | 日志查询不影响同步，同步不依赖日志查询 |
| GetDepositOrderStatus | CreateDepositOrder | 轮询依赖订单 | 轮询是辅助体验，不影响充值核心流程 |

### 4.3 完全独立（互不影响，可并行开发）

```
独立组A（B端币种管理）          独立组B（C端提现账户）
┌──────────────────────┐     ┌──────────────────────┐
│ QueryCurrencyPage    │     │ SaveWithdrawalAccount │
│ UpdateCurrency       │     │ GetWithdrawalAccounts │
│ ToggleCurrencyStatus │     └──────────────────────┘
│ QueryRateLogPage     │
└──────────────────────┘     独立组C（人工修正）
                             ┌──────────────────────┐
独立组D（汇率RPC供他）        │ ManualCredit         │
┌──────────────────────┐     │ ManualDebit          │
│ GetCurrencyList      │     │ CreditCorrection     │
│ GetExchangeRate      │     │ GetUserWalletBalance  │
│ GetBaseCurrency      │     └──────────────────────┘
│ GetCurrencyByCode    │
└──────────────────────┘

这些组之间互不依赖，可以并行开发和测试
```

**提现状态机**：
```
                    创建订单
                       │
                       ▼
                  ┌─────────┐
                  │  已冻结  │ ← FreezeForWithdrawal
                  └────┬────┘
                       │
              ┌────────┼────────┐
              │        │        │
         审核拒绝   出款成功   出款失败
              │        │        │
              ▼        ▼        ▼
          ┌───────┐ ┌──────┐ ┌───────┐
          │已解冻  │ │已扣款 │ │已解冻  │
          │(返还) │ │(完成)│ │(返还) │
          └───────┘ └──────┘ └───────┘
              ↑                  ↑
        Unfreeze          Unfreeze
              ↑                  ↑
        审核拒绝触发        出款失败触发

最终态只有两种：已解冻(钱还给用户) 或 已扣款(钱出去了)
```

## 六、双向依赖分析（钱包 ↔ 财务）

这是架构设计中最需要注意的点：

```
┌──────────────────────┐         ┌──────────────────────┐
│     多币种钱包        │         │     财务管理          │
│                      │  →调用   │                      │
│ GetDepositMethods    │────────►│ GetAvailableDeposit  │
│   (充值方式查询)      │         │   Methods            │
│                      │         │                      │
│ GetWithdrawalInfo    │────────►│ GetAvailableWithdraw │
│   (提现信息查询)      │         │   alMethods          │
│                      │         │                      │
│                      │  ◄调用   │                      │
│ CreditDeposit        │◄────────│ 充值成功回调          │
│ FreezeForWithdrawal  │◄────────│ 创建提现订单          │
│ UnfreezeForWithdrawal│◄────────│ 审核拒绝/出款失败     │
│ ConfirmWithdrawDeduct│◄────────│ 出款成功              │
│ ManualCredit         │◄────────│ 加款审核通过          │
│ ManualDebit          │◄────────│ 减款审核通过          │
│ CreditCorrection     │◄────────│ 补单审核通过          │
│ GetUserWalletBalance │◄────────│ 加减款弹窗查余额      │
└──────────────────────┘         └──────────────────────┘

钱包→财务：2个RPC调用（查询类，同步）
财务→钱包：8个RPC调用（7个写操作+1个查询）

特征：
  - 钱包调财务：只有查询，不改财务数据
  - 财务调钱包：主要是写操作，改变钱包余额
  - 不存在循环调用（不会在同一个请求链路中互相调）
  - 钱包查财务是用户请求触发（同步）
  - 财务调钱包是回调/审核触发（异步事件驱动）
```

**为什么不会死锁**：
- 充值链路：用户→钱包→财务（查方式）→返回给用户，链路结束。后来财务回调→钱包（上账），是独立的新链路。
- 提现链路：用户→钱包→财务（查方式）→返回，链路结束。然后钱包冻结→财务审核→财务调钱包（解冻/扣款），也是独立链路。
- 不存在 A调B的同时B回调A 的情况。

## 八、总结

### 核心认识

1. **币种配置是地基**——所有模块都依赖它，必须最先做，且最先稳定
2. **兑换是最独立的功能**——不依赖财务模块，钱包内部闭环，可以最先跑通完整链路
3. **充值和提现对财务有强依赖**——财务的支付配置和通道管理不就绪，这两个功能无法完整联调
4. **提现是最复杂的功能**——涉及KYC+冻结/解冻状态机+三级审核，依赖链最长
5. **人工修正和记录查询是最后完善的**——依赖前面的功能产生数据
6. **钱包和财务的双向依赖不会导致循环调用**——查询和回调在不同链路上，时序上不重叠

### 对架构设计的建议

| 要点 | 说明 |
|------|------|
| **冻结/解冻必须原子化** | 提现冻结和余额扣减必须在事务内完成，不能出现冻结成功但记录失败的情况 |
| **RPC接口要幂等** | 财务调钱包的所有写操作（上账/冻结/解冻/扣款）必须支持幂等重试 |
| **余额变更走统一入口** | 不管是充值上账还是人工加款，所有改余额的操作必须走同一套流水记录逻辑，保证对账一致性 |
| **状态流转严格控制** | 冻结→只能解冻或扣款，不能跳过。未冻结不能扣款。已解冻不能再扣款。 |
| **汇率快照** | 兑换/充值/提现时要记录当时的汇率快照，不能事后用变化的汇率算 |

## 一、币种配置模块：所有模块的数据地基

### 1.1 币种配置提供了什么

币种配置是整个 ser-wallet 系统的**底层数据供应商**。根据币种配置需求文档 B3（币种配置）和 B4（汇率规则），它向外提供三类基础数据：

**第一类：币种定义数据**

币种配置 → 钱包：钱包需要知道当前平台支持哪些币种（BSB/VND/IDR/THB），每个币种的展示规则（千分位符号、小数位数、货币符号、图标），以及哪些币种是启用的。用户在C端切换币种（产品原型 D2 个人中心页的币种切换组件）、充值页按币种展示不同方式（产品原型 D3 充值中心），这些页面渲染所需的币种基础信息全部来自币种配置。

币种配置 → 财务管理：财务后台的几乎所有页面都有币种筛选下拉框（产品原型 F 系列，通道配置/支付配置/充值记录/提现记录/数据统计），这些下拉框的选项数据来自币种配置。此外，财务的充值/提现记录表格右侧有"基准币种换算列"（产品原型 F 充值记录/提现记录页），这个换算依赖币种配置提供的汇率。

**第二类：汇率数据**

币种配置 → 钱包（兑换场景）：用户在C端做法币→BSB兑换时（需求文档 A4，产品原型 D7 兑换页面），页面上展示的"1 BSB = X 法币"汇率来自币种配置的平台汇率。

币种配置 → 钱包（充值场景）：BSB充值时页面上显示的"≈ X USDT"换算（产品原型 D3 BSB充值页面金额档位旁的换算提示），汇率来自币种配置。

币种配置 → 钱包（提现场景）：BSB提现时展示的法币换算金额（产品原型 D9 BSB提现页面），汇率来自币种配置。

币种配置 → 财务管理（统计场景）：财务的数据统计模块（需求文档 E6）需要把各币种金额统一换算成基准币种（USDT）来做汇总，依赖币种配置的平台汇率和基准币种设置。

**第三类：基准币种**

币种配置 → 全局：基准币种（推荐USDT，需求文档 B3）一旦设定不可更改，是活动奖金计算、多币种投注统计、钱包资产统计、财务对账的统一换算基准。所有模块都间接依赖这个设定。

### 1.2 币种配置依赖什么

币种配置 ← 三方汇率API：汇率同步定时任务（需求文档 B4）需要从3家三方API获取实时汇率，取平均值。这是币种配置唯一的外部数据依赖。

币种配置 ← ser-blog：写操作后记录审计日志（工程规范要求，oneway调用）。

币种配置 ← ser-buser：B端列表中"操作人"字段需要解析为姓名（工程规范要求）。

### 1.3 币种配置的独立性

币种配置**不依赖**钱包模块，也**不依赖**财务管理模块。它是纯粹的数据提供方。这意味着：

- 币种配置可以**最先开发、最先测试、最先上线**
- 币种配置出问题 → 钱包和财务都受影响（全局影响）
- 钱包或财务出问题 → 不影响币种配置

## 二、多币种钱包模块：用户的资金操作中枢

### 2.1 钱包的上游依赖（钱包需要从谁那里拿数据）

**钱包 ← 币种配置（内部依赖）**

钱包和币种配置在同一个 ser-wallet 服务内，属于内部函数调用。钱包的以下功能依赖币种配置的数据：

- 用户打开钱包 → 需要可用币种列表（币种配置提供）
- 用户看余额 → 余额按币种隔离展示（需求文档 A2，币种隔离原则）→ 需要知道当前币种的展示规则（千分位、小数位）
- 用户兑换 → 需要平台汇率（币种配置提供）
- 充值/提现涉及金额换算 → 需要入款汇率/出款汇率（币种配置提供）

这层依赖是**强依赖、同步、高频**的——几乎每个钱包操作都要读币种数据。

**钱包 ← 财务管理（跨服务RPC依赖）**

根据财务需求文档 E2（支付配置），充值方式和提现方式的配置数据存在财务管理侧。钱包的以下功能依赖财务：

- 用户打开充值页面 → 需要知道当前币种有哪些充值方式、金额档位、费率（产品原型 D3，充值中心按币种展示不同方式）→ 这些数据在财务的"支付配置-充值方式"中配置
- 用户打开提现页面 → 需要知道当前币种有哪些提现方式、手续费、限额（产品原型 D9，提现方式按地区不同）→ 这些数据在财务的"支付配置-提现方式"中配置

这层依赖是**查询依赖**——钱包只读财务的配置数据，不修改。但如果财务的支付配置接口不可用，用户就看不到充值/提现方式，操作无法继续。

**钱包 ← ser-kyc（跨服务RPC依赖）**

根据需求文档 A5（KYC与提现关系），提现时需要检查用户的KYC认证状态：

- 未认证 → 弹窗："去认证"解锁全渠道，或"继续提现"仅USDT（产品原型 D8 提现触发认证弹窗）
- 已认证 → 直接展示全部提现方式
- 提现账户持有人姓名 = KYC认证姓名（产品原型 D9，姓名字段自动填充不可编辑）

这层依赖**仅在提现场景**存在。充值、兑换、记录查询都不需要KYC。

### 2.2 钱包的下游被依赖（谁需要调用钱包）

**钱包 ← 财务管理（跨服务RPC，财务主动调钱包）**

这是最重要的依赖关系。根据财务需求文档的多个章节，财务管理在以下场景需要操作钱包余额：

**充值上账**（需求文档 E3 充值流程）：三方支付回调到达财务 → 财务验签通过 → 财务调用钱包的 CreditDeposit → 充值金额进入中心钱包，如果有额外奖金则奖金进入奖励钱包。这是用户充值流程的最后一步，没有这个调用，用户付了钱但余额不会变。

**提现冻结**（需求文档 E4 提现流程）：用户提交提现 → 钱包创建订单并冻结金额 → 财务接管后续审核。冻结操作确保用户提现期间这笔钱不能被其他操作（消费、再次提现）使用。

**提现终态处理**（需求文档 E4 提现流程）：提现的最终结果有两种——成功出款（财务调用 ConfirmWithdrawalDeduct，钱包确认扣除冻结金额）或者失败/拒绝（财务调用 UnfreezeForWithdrawal，钱包解冻返还金额）。这两个接口是提现状态机的终态触发器。

**人工加款**（需求文档 E5 人工修正-人工加款）：财务人员选择用户 → 选币种 → 按钱包类型（中心/奖励/代理/主播）分别输入金额 → 另一人审核通过 → 财务调用钱包的 ManualCredit。产品原型中人工加款弹窗明确展示了4种钱包类型的金额输入框，说明加款是按子钱包粒度操作的。

**人工减款**（需求文档 E5 人工修正-人工减款）：逻辑同加款，但方向相反。需求文档明确指出"校验余额≥0"，即减款不能把余额扣成负数，余额不足的扣实际余额。

**补单上账**（需求文档 E5 人工修正-充值补单）：关联原充值订单，含补单金额和补单赠送，审核通过后调用钱包入账。

**查询用户余额**（产品原型 F 人工加减款弹窗）：财务人员在创建加款/减款时，弹窗中需要展示用户各子钱包的当前余额，作为操作参考。这要求钱包提供一个按用户查余额的RPC接口。

### 2.3 钱包内部的功能依赖链

钱包内部的四大功能之间也有前后置关系：

**充值 → 记录**：充值成功后会产生一条充值交易流水，记录模块才有数据可展示。

**兑换 → 记录**：兑换执行后产生兑换流水记录。

**提现 → 记录**：提现创建后产生提现记录（含进行中/成功/失败各状态）。

**充值 → 兑换的间接关系**：用户先充值法币 → 有了法币余额 → 才能做法币→BSB兑换。但从接口依赖角度，兑换接口不直接调用充值接口，它们通过余额数据间接关联。

**充值/兑换 → 提现的间接关系**：用户需要先有余额才能提现。需求文档 A5 的提现规则（法币充→法币提，USDT充→USDT提）说明提现方式受充值方式影响，但这是业务规则层面的依赖，不是接口调用层面的。

**兑换的独立性**：兑换功能（法币→BSB）是钱包内部闭环操作——只需要读取币种配置的汇率，然后操作用户自己的余额（扣法币，加BSB），**不涉及财务管理、不涉及外部通道、不涉及回调**。这使得兑换成为最容易独立开发和测试的功能。

## 三、财务管理模块：通道和审核的管控中枢

### 3.1 财务管理的上游依赖（财务需要从谁那里拿数据）

**财务 ← 币种配置（跨服务RPC依赖）**

财务管理的几乎所有B端页面都涉及币种维度：

- 通道配置按币种区分（需求文档 E1，代收/代付通道都有"币种"字段）→ 需要币种列表
- 支付配置按币种区分（需求文档 E2，充值方式/提现方式都有"币种"字段）→ 需要币种列表
- 充值/提现记录筛选含币种（需求文档 E3/E4）→ 需要币种列表
- 数据统计支持按币种筛选（需求文档 E6）→ 需要币种列表
- 记录表格右侧"基准币种换算列"（产品原型 F）→ 需要汇率 + 基准币种

**财务 ← 钱包（跨服务RPC依赖）**

如 2.2 节所述，财务在人工修正场景需要查询用户钱包余额，在充值/提现/人工修正审核通过后需要调用钱包做余额变更。

### 3.2 财务管理提供了什么给钱包

**财务 → 钱包（支付配置数据）**

根据需求文档 E2（支付配置），财务管理配置并提供：

- 每个币种有哪些充值方式（银行转账/电子钱包/USDT），每种方式关联哪个代收通道，金额档位是多少，费率是多少 → 钱包C端充值页面依赖这些数据来渲染UI
- 每个币种有哪些提现方式（银行卡/电子钱包各品牌），每种方式关联哪个代付通道，手续费是多少 → 钱包C端提现页面依赖这些数据来渲染UI

**财务 → 钱包（充值/提现的通道能力）**

用户发起充值后，具体走哪个通道、通道怎么轮询分配（需求文档 E1 通道轮询规则），这些是财务管理侧的能力。钱包不直接对接支付通道，中间隔着财务。

**财务 → 钱包（提现审核流程）**

用户提交提现后，三级审核流程（需求文档 E4：风控审核→财务审核→出款审核）由财务管理驱动。审核的最终结果（通过/拒绝）通过RPC反馈给钱包做余额终态处理。

### 3.3 财务管理的内部依赖链

财务管理内部也有前后置关系：

**通道配置 → 支付配置**：先有代收/代付通道，才能在支付配置中关联通道。

**支付配置 → 充值/提现流程**：先配置好充值方式和提现方式，用户才能在C端看到并使用。

**充值/提现流程 → 记录**：先有充值/提现发生，才有记录可查。

**充值/提现流程 → 数据统计**：统计是对历史记录的聚合分析。

**人工修正相对独立**：人工加减款不依赖充值/提现流程，可以随时对任何用户操作。但补单需要关联原充值订单（需求文档 E5：关联原充值订单），所以补单依赖有充值订单存在。

## 四、三模块间的依赖方向汇总

### 4.1 单向依赖（A依赖B，B不依赖A）

**币种配置 → 无**：币种配置不依赖钱包，不依赖财务。它是纯粹的被依赖方。

**钱包 → 币种配置**：钱包的所有币种相关操作都依赖币种配置的数据。单向依赖，币种配置不需要钱包的任何数据。

**财务 → 币种配置**：财务所有涉及币种的页面都依赖币种配置的数据。单向依赖。

**钱包 → ser-kyc**：只有提现场景需要。单向依赖，KYC不需要钱包的数据。

### 4.2 双向依赖（A依赖B，B也依赖A）

**钱包 ↔ 财务**：这是唯一的双向依赖。

钱包依赖财务的原因：
- 充值方式数据在财务（需求文档 E2 支付配置 → 充值方式）
- 提现方式数据在财务（需求文档 E2 支付配置 → 提现方式）

财务依赖钱包的原因：
- 充值回调后要改钱包余额（需求文档 E3 充值流程 → 上分）
- 提现审核后要改钱包冻结状态（需求文档 E4 提现流程 → 冻结/解冻/扣款）
- 人工修正要改钱包余额（需求文档 E5 人工加款/减款/补单）
- 人工修正创建时要读钱包余额（产品原型 F 人工加减款弹窗）

但两个方向的**性质不同**：

- 钱包→财务：**查询依赖**，同步读取配置数据，不改财务的数据
- 财务→钱包：**操作依赖**，异步事件驱动，改变钱包余额

### 4.3 依赖强度分级

**强依赖（缺失则功能不可用）**：
- 钱包对币种配置：没有币种数据，钱包所有页面无法渲染 → 强依赖
- 钱包(充值)对财务(支付配置)：没有充值方式数据，充值页面空白 → 强依赖
- 钱包(提现)对财务(支付配置)：没有提现方式数据，提现页面空白 → 强依赖
- 财务(充值回调)对钱包(CreditDeposit)：回调后无法上账，用户付了钱余额不变 → 强依赖
- 财务(提现终态)对钱包(Freeze/Unfreeze/Deduct)：审核后无法处理资金，钱永远冻着或不扣 → 强依赖

**中等依赖（缺失则部分功能降级）**：
- 钱包(提现)对ser-kyc：KYC不可用时，可以降级为仅允许USDT提现 → 中等依赖
- 财务(人工修正)对钱包(GetUserWalletBalance)：查余额不可用，弹窗中不展示余额，但创建操作本身不受影响 → 中等依赖
- 钱包(充值)对奖金活动模块：奖金活动不可用，用户照样能充值，只是没有额外奖金 → 中等依赖

**弱依赖（缺失不影响核心功能）**：
- 币种配置对ser-buser(操作人姓名解析)：不可用则列表中操作人显示ID而非姓名 → 弱依赖
- 所有模块对ser-blog(审计日志)：不可用则不记日志，不影响业务功能（oneway不阻塞） → 弱依赖
- 钱包(GetDepositOrderStatus)对CreateDepositOrder：状态轮询是辅助体验，不影响充值结果 → 弱依赖

## 六、接口前置/后置关系清单

以下用箭头链表示从最前置到最后置的关系，→ 表示"先于"或"是前置"。

### 6.1 必须严格按顺序的链

**币种配置启动链**：
SetBaseCurrency → CreateCurrency → SyncRealtimeRate → GetCurrencyList/GetExchangeRate 可用
（没有基准币种就不能创建币种，没有币种就没有汇率同步目标，没有汇率数据外部查询返回空）

**充值链**：
币种配置就绪 → GetAvailableCurrencies → GetDepositMethods(需财务配好充值方式) → CreateDepositOrder → [用户外部支付] → CreditDeposit → GetTransactionRecords
（每一步都严格依赖前一步完成）

**提现链**：
币种配置就绪 → GetAvailableCurrencies → GetWithdrawalInfo(需财务配好提现方式+KYC) → SaveWithdrawalAccount(如首次) → CreateWithdrawalOrder + FreezeForWithdrawal → [财务三级审核] → ConfirmWithdrawalDeduct 或 UnfreezeForWithdrawal → GetTransactionRecords
（最长的依赖链，跨越钱包、财务、KYC三个模块）

**兑换链**：
币种配置就绪 → GetAvailableCurrencies → GetExchangePreview → ExecuteExchange → GetTransactionRecords
（最短的完整业务链，不涉及财务模块）

### 6.2 没有先后关系、可独立并行的

**独立对A：币种B端管理 vs 钱包C端功能**
- QueryCurrencyPage / UpdateCurrency / ToggleCurrencyStatus / QueryRateLogPage 这组B端接口
- 与 GetWalletBalance / GetAvailableCurrencies 这组C端接口
- 两组之间无前后置关系，B端在管理币种的同时，C端可以独立查询余额

**独立对B：提现账户管理 vs 充值功能**
- SaveWithdrawalAccount / GetWithdrawalAccounts
- 与 GetDepositMethods / CreateDepositOrder / GetUSDTDepositInfo
- 用户可以先保存好提现账户，再去充值，两者没有关系

**独立对C：兑换 vs 充值**
- GetExchangePreview / ExecuteExchange
- 与 GetDepositMethods / CreateDepositOrder
- 只要用户有余额，兑换和充值是完全独立的操作

**独立对D：人工修正 vs C端用户操作**
- ManualCredit / ManualDebit / CreditCorrection
- 与充值/兑换/提现的所有C端接口
- 人工修正是B端运营行为，与用户在C端的操作完全独立，任何时候都可以执行

**独立对E：汇率日志 vs 业务操作**
- QueryRateLogPage 是B端查看历史汇率变更的只读页面
- 与所有业务操作（充值/兑换/提现/人工修正）完全独立

### 6.3 虽然独立但运行时互相影响的

**币种启用/禁用 与 C端所有操作**：接口层面没有直接调用关系，但如果后台禁用了某个币种，C端该币种下的所有操作都应该不可用。这不是接口依赖，而是数据状态依赖——C端每次查询时从币种配置读取的状态决定了用户能做什么。

**汇率更新 与 兑换/充值/提现**：汇率同步定时任务在后台静默运行，每次更新后，用户看到的汇率、充值换算、提现换算、兑换预览都会变化。但汇率更新不阻塞业务操作。需要注意的是：用户在预览阶段看到的汇率，和最终提交时的汇率可能已经变了，所以业务操作需要做**汇率快照**——记录下单时的汇率，不受后续汇率变动影响。

**人工加减款 与 用户余额**：人工加减款直接改变用户余额，可能影响用户正在进行的提现操作（比如用户正要提现，运营同时减款导致余额不足）。这不是接口依赖，而是**并发数据竞争**，需要在实现层面通过事务和锁来保证一致性。
