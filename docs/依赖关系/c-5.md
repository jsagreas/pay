# 模块间依赖调用关系分析

> 基于需求文档和产品原型，分析币种配置、多币种钱包、财务管理三模块的联动链路
> 目标：搞清楚谁先谁后、谁调谁、谁依赖谁，为架构设计提供依据

---

# 一、模块层级关系总览

三个模块不是平级的，存在明确的上下游层级关系：

```
┌─────────────────────────────────────────────────────┐
│                    Layer 0  基础数据层                │
│                                                     │
│   ┌─────────────┐          ┌─────────────────┐      │
│   │  币种配置    │          │  用户服务(KYC)   │      │
│   │  币种/汇率   │          │  (外部依赖)      │      │
│   └──────┬──────┘          └────────┬────────┘      │
│          │                          │                │
├──────────┼──────────────────────────┼────────────────┤
│          │       Layer 1  通道与配置层               │
│          ▼                          │                │
│   ┌─────────────────────────┐       │                │
│   │      财务管理             │       │                │
│   │  通道配置 → 支付配置      │       │                │
│   └──────────┬──────────────┘       │                │
│              │                      │                │
├──────────────┼──────────────────────┼────────────────┤
│              │       Layer 2  业务应用层             │
│              ▼                      ▼                │
│   ┌─────────────────────────────────────────┐       │
│   │            多币种钱包                     │       │
│   │  充值 / 兑换 / 提现 / 记录                │       │
│   └─────────────────────────────────────────┘       │
│                                                     │
├─────────────────────────────────────────────────────┤
│              Layer 3  回调与后置操作层               │
│                                                     │
│   财务(回调确认) ──RPC──→ 钱包(入账/扣款/解冻)       │
│   财务(人工修正) ──RPC──→ 钱包(加款/减款/补单)       │
│                                                     │
└─────────────────────────────────────────────────────┘
```

**层级含义**：
- Layer 0 必须先就绪，Layer 1 才能配置，Layer 2 才能运行
- Layer 3 是异步后置操作，由事件触发

# 二、模块间调用方向图

```
                    ┌──────────────┐
                    │   币种配置    │
                    │  (Layer 0)   │
                    └───┬──────┬───┘
                        │      │
            ┌───────────┘      └───────────┐
            │ RPC: 币种信息                 │ RPC: 汇率
            │ RPC: 币种配置参数             │
            ▼                               ▼
    ┌───────────────┐               ┌───────────────┐
    │   财务管理     │               │  多币种钱包    │
    │  (Layer 1)    │◄──────────────│  (Layer 2)    │
    └───────┬───────┘  RPC:         └───────────────┘
            │          · 支付配置查询         ▲
            │          · 创建渠道订单         │
            │          · 提交提现审核         │
            │                                │
            └────────────────────────────────┘
               RPC(回调/后置):
               · 充值入账
               · 提现确认扣款 / 失败解冻
               · 人工加款 / 减款 / 补单
               · 余额查询


    ┌──────────────┐
    │  用户服务     │◄──── 钱包 RPC调用: KYC状态/信息查询
    │  (外部依赖)  │
    └──────────────┘
```

**调用方向汇总**：

| 调用方 | → | 被调用方 | 调用内容 | 触发时机 |
|--------|---|----------|----------|----------|
| 钱包 | → | 币种配置 | 币种信息、汇率、配置参数 | C端页面加载、订单创建时 |
| 钱包 | → | 财务 | 支付配置查询、创建渠道订单、提交提现审核 | C端用户操作时 |
| 钱包 | → | 用户服务 | KYC状态/信息查询 | 提现前校验 |
| 财务 | → | 钱包 | 充值入账、提现扣款/解冻、人工加减款、余额查询 | 渠道回调、审核完成、人工修正时 |
| 财务 | → | 币种配置 | 汇率查询（BSB换算列展示） | B端记录/统计展示时 |

# 三、前置 / 后置 / 独立 / 耦合 接口分类

## 3.1 前置接口（必须先就绪，其他功能才能运行）

前置接口是整个系统的地基，如果它们不可用，下游功能全部瘫痪。

```
启动顺序：

[第一步] 币种配置.币种CRUD + 汇率设置
    │
    │  没有币种数据，钱包无法展示，财务无法配置
    ▼
[第二步] 财务.通道配置
    │
    │  没有通道，支付配置无法绑定通道，充值/提现无通道可走
    ▼
[第三步] 财务.支付配置
    │
    │  没有支付配置，C端不知道展示哪些充值/提现方式
    ▼
[第四步] 钱包C端功能就绪（充值/兑换/提现）
```

| 前置接口 | 所属模块 | 谁依赖它 | 为什么是前置 |
|----------|----------|----------|-------------|
| 币种列表/详情查询 | 币种配置 | 钱包、财务 | 没有币种元数据（精度/限额/状态），后续所有功能都算不了账 |
| 汇率查询 | 币种配置 | 钱包（兑换/充值/提现展示）、财务（BSB换算列） | 没有汇率，任何涉及BSB换算的操作都无法进行 |
| 币种配置参数查询 | 币种配置 | 钱包（限额校验/手续费计算/T+N展示） | 没有配置参数，充值提现的业务规则无法执行 |
| 支付配置查询 | 财务 | 钱包C端 | C端"可用充值方式"和"可用提现方式"的唯一数据来源 |
| 通道可用性 | 财务 | 钱包（间接）| 通道全挂了，充值/提现实际不可用 |
| KYC状态查询 | 用户服务 | 钱包（提现） | 法币提现的强制前置校验 |

## 3.2 后置接口（由事件触发，异步/回调性质）

后置接口不需要提前就绪，它们是被动触发的，由上游事件驱动。

```
触发链路：

[事件] 第三方渠道回调(充值成功)
    │
    ▼
[财务] 渠道回调处理 → 确认支付成功
    │
    ▼
[后置] 财务 ──RPC──→ 钱包.充值入账


[事件] B端审核人点击"通过"
    │
    ▼
[财务] 出款审核通过 → 调度渠道出款 → 渠道回调
    │
    ▼
[后置] 财务 ──RPC──→ 钱包.提现确认扣款 或 钱包.提现失败解冻


[事件] B端操作人创建加款/减款/补单 → 审核通过
    │
    ▼
[后置] 财务 ──RPC──→ 钱包.人工加款/减款/补单入账
```

| 后置接口 | 所属模块 | 谁触发它 | 触发时机 |
|----------|----------|----------|----------|
| 充值入账 | 钱包(RPC暴露) | 财务 | 第三方渠道回调确认充值成功后 |
| 提现确认扣款 | 钱包(RPC暴露) | 财务 | 多级审核全部通过 + 渠道出款成功后 |
| 提现失败解冻 | 钱包(RPC暴露) | 财务 | 任一级审核拒绝 或 渠道出款失败后 |
| 人工加款 | 钱包(RPC暴露) | 财务 | 人工修正-加款审核通过后 |
| 人工减款 | 钱包(RPC暴露) | 财务 | 人工修正-减款审核通过后 |
| 人工补单入账 | 钱包(RPC暴露) | 财务 | 人工修正-补单审核通过后 |

## 3.3 独立接口（不依赖其他模块实时响应）

这些接口是自闭环的，只读取自己模块的数据，不需要跨模块实时调用。

| 独立接口 | 所属模块 | 为什么独立 |
|----------|----------|-----------|
| 钱包余额查询 | 钱包 | 读自己的余额表，不依赖外部 |
| 充值/提现/兑换/奖励记录列表 | 钱包 | 读自己的订单/记录表 |
| 记录详情查询 | 钱包 | 同上 |
| 兑换订单创建 | 钱包 | 只依赖币种配置的汇率（可缓存），内部完成余额变动 |
| 可用币种列表 | 钱包 | 依赖币种配置但可缓存，不需要实时RPC |
| 币种CRUD | 币种配置 | 纯B端管理操作，操作自己的表 |
| 汇率设置 | 币种配置 | 同上 |
| 汇率日志/操作日志查询 | 币种配置 | 读自己的日志表 |
| 通道配置CRUD | 财务 | B端管理操作，操作自己的表 |
| 支付配置CRUD | 财务 | B端管理操作（引用通道数据但是自己模块内部的） |
| 充值记录列表/详情/导出 | 财务 | 读自己的记录表 |
| 提现记录列表/详情/导出 | 财务 | 读自己的记录表 |
| 人工修正列表查询 | 财务 | 读自己的修正记录表 |
| 数据统计（通道/充值/提现） | 财务 | 读自己的统计数据 |

**注意**：独立不代表完全无关联，只是"运行时不需要实时跨模块RPC调用"。部分接口的数据是由跨模块流程写入的（如充值记录是充值流程的产物），但查询时是自闭环的。

## 3.4 耦合接口（强绑定，改一个必须联动改另一个）

```
                   强耦合关系

    ┌──────────┐                    ┌──────────┐
    │  钱包     │ ◄═══ 充值链路 ═══► │  财务     │
    │ 创建订单  │ ────→ 创建渠道订单  │ 通道匹配  │
    │ 入账操作  │ ◄──── 回调入账     │ 回调处理  │
    └──────────┘                    └──────────┘

    ┌──────────┐                    ┌──────────┐
    │  钱包     │ ◄═══ 提现链路 ═══► │  财务     │
    │ 冻结余额  │ ────→ 提交审核     │ 多级审核  │
    │ 扣款/解冻 │ ◄──── 审核结果     │ 渠道出款  │
    └──────────┘                    └──────────┘

    ┌──────────┐                    ┌──────────┐
    │  钱包     │ ◄═══ 人工修正 ═══► │  财务     │
    │ 余额变动  │ ◄──── 修正执行     │ 修正审核  │
    └──────────┘                    └──────────┘
```

| 耦合对 | 模块A | 模块B | 耦合原因 | 影响 |
|--------|-------|-------|----------|------|
| 充值链路 | 钱包(创建订单) | 财务(通道匹配+渠道下单+回调) | 订单创建和入账分属两个模块，中间经过外部渠道 | 订单结构、回调协议、入账接口参数必须双方一致 |
| 提现链路 | 钱包(冻结+扣款/解冻) | 财务(审核+渠道出款+回调) | 冻结在钱包侧，审核和出款在财务侧，结果回写钱包 | 订单状态机、冻结/解冻接口、审核结果通知协议必须一致 |
| 人工修正 | 钱包(余额操作) | 财务(修正审核) | 财务审核通过后直接操作钱包余额（4个子钱包） | 子钱包类型枚举、金额精度、操作接口必须一致 |
| 支付配置消费 | 钱包(C端展示) | 财务(配置管理) | C端充值/提现方式完全由财务支付配置决定 | 数据结构（方式名称/图标/金额范围等）必须约定一致 |

**耦合的实际含义**：这些接口在设计RPC IDL（Kitex Thrift/Protobuf）时，必须双方协商确定参数结构，任何一方改动都需要通知对方同步修改。

# 四、核心业务链路详解

## 4.1 充值链路

```
用户操作                    钱包服务                      财务服务                    外部
─────────────────────────────────────────────────────────────────────────────────────────

[1] 进入充值页
        │
        ├──[C端]───→ 获取可用充值方式 ──[RPC]──→ 支付配置查询
        │                                          │
        │            ◄─── 返回方式列表 ◄────────────┘
        │
        ├──[C端]───→ 获取汇率(展示用) ──[RPC]──→ 币种配置.汇率查询
        │
[2] 选择方式、输入金额
        │
        ├──[C端]───→ 重复转账校验(可选) ──[内部]── 查本地pending订单
        │
        ├──[C端]───→ 选择奖金活动(可选)
        │
[3] 确认充值
        │
        ├──[C端]───→ 创建充值订单 ──[RPC]──→ 创建渠道支付订单
        │            (本地建订单)              (通道匹配+渠道下单)
        │                                          │
        │            ◄─── 返回支付信息 ◄────────────┘
        │            (跳转URL/收款地址)
        │
[4] 用户支付                                                      第三方渠道
        │                                                            │
        │                                                     [渠道处理支付]
        │                                                            │
        │                                    渠道回调处理 ◄──────────┘
        │                                          │
        │                                   [确认支付成功]
        │                                          │
[5]                    充值入账 ◄──────[RPC]────────┘
                    (余额增加)
                    (奖金→奖励钱包)
        │
[6] 收到入账通知
```

**链路中的关键依赖**：
- [1] 钱包 → 财务.支付配置：前置依赖，没有配置就没有充值方式
- [1] 钱包 → 币种配置.汇率：前置依赖，展示需要汇率
- [3] 钱包 → 财务.创建渠道订单：强耦合，订单参数/通道选择逻辑在财务侧
- [5] 财务 → 钱包.充值入账：后置回调，异步触发，必须幂等

## 4.2 提现链路

```
用户操作                    钱包服务                      财务服务                    外部
─────────────────────────────────────────────────────────────────────────────────────────

[1] 进入提现页
        │
        ├──[C端]───→ KYC状态查询 ──[RPC]──→ 用户服务.KYC查询
        │            (非USDT必须KYC)
        │
        ├──[C端]───→ 获取可用提现方式 ──[RPC]──→ 财务.支付配置查询
        │
        ├──[C端]───→ 提现限额查询 ──[内部/RPC]── 币种配置.限额/费率
        │
        ├──[C端]───→ 提现账户信息查询(已保存的银行卡/电子钱包)
        │
[2] 填写/确认提现信息
        │
        ├──[C端]───→ 保存提现账户(首次)
        │
[3] 确认提现
        │
        ├──[C端]───→ 创建提现订单
        │            ├── [内部] 冻结对应金额
        │            └── [RPC] ──→ 提交提现审核
        │                               │
        │                         [进入审核队列]
        │                               │
[4]                              风控审核(第一级)
                                       │
                              ┌────────┴────────┐
                              │                 │
                           [通过]             [拒绝]
                              │                 │
                        财务审核(第二级)    ┌────┘
                              │            │
                     ┌────────┴────────┐   │
                     │                 │   │
                  [通过]             [拒绝] │
                     │                 │   │
               出款审核(第三级)    ┌────┘   │
                     │            │        │
            ┌────────┴────────┐   │        │
            │                 │   │        │
         [通过]             [拒绝] │        │
            │                 │   │        │
[5]   调度渠道出款            │   │        │
            │                 │   │        │     第三方渠道
            ├─────────────────┼───┼────────┼────→ [渠道处理]
            │                 │   │        │         │
            │                 │   │        │    [回调结果]
            │                 │   │        │         │
      ┌─────┴─────┐          │   │        │    ◄────┘
      │           │          │   │        │
   [成功]       [失败]       │   │        │
      │           │          │   │        │
      ▼           ▼          ▼   ▼        ▼
[6] 钱包.提现确认扣款    钱包.提现失败解冻
    (冻结→正式扣除)      (冻结→解冻退回)
```

**链路中的关键依赖**：
- [1] 钱包 → 用户服务.KYC：前置强制校验，后端不能只靠前端拦截
- [1] 钱包 → 财务.支付配置：前置依赖
- [1] 钱包 → 币种配置.限额/费率：前置依赖
- [3] 钱包 → 财务.提交审核：强耦合，订单进入财务审核流
- [4] 财务内部三级审核：财务模块内部流程，钱包不参与
- [6] 财务 → 钱包.扣款/解冻：后置回调，由审核结果和渠道结果触发

**提现的特殊性**：它是所有链路中**跨度最长、涉及模块最多、状态流转最复杂**的流程。从用户提交到最终到账，可能经过：钱包→用户服务→财务（三级审核）→外部渠道→财务（回调）→钱包，跨越5个节点。

## 4.3 兑换链路

```
用户操作                    钱包服务
──────────────────────────────────────────────────

[1] 进入兑换页
        │
        ├──[C端]───→ 获取兑换汇率 ──[RPC]──→ 币种配置.汇率查询
        │
[2] 输入金额，确认兑换
        │
        ├──[C端]───→ 创建兑换订单
        │            ├── [内部] 扣减法币/USDT余额(中心钱包)
        │            ├── [内部] 增加BSB余额(中心钱包)
        │            └── [内部] 赠送部分→奖励钱包(如有)
        │
[3] 兑换完成
```

**兑换是最简单的链路**：
- 只依赖币种配置的汇率（可缓存）
- 不涉及财务模块、不涉及外部渠道
- 完全在钱包模块内部闭环完成
- 是三条核心链路中**唯一不与财务模块耦合的**

## 4.4 人工修正链路

```
B端操作                     财务服务                      钱包服务
──────────────────────────────────────────────────────────────────

[1] 操作员创建修正单
        │
        ├──[B端]───→ 创建补单/加款/减款
        │            (选择用户、子钱包、金额、原因)
        │                    │
        │              [RPC] ├──→ 钱包.余额查询(确认用户余额)
        │                    │
[2] 审核人审核
        │
        ├──[B端]───→ 审核通过/拒绝
        │                    │
        │              [通过] │
        │                    ▼
[3]                   财务 ──[RPC]──→ 钱包.人工加款/减款/补单
                                      (操作指定子钱包余额)
```

**链路特点**：
- 财务发起，钱包被动接受余额操作
- 财务需要先查钱包余额（减款时确认够不够扣）
- 审核通过后才执行余额变动，不是创建时就扣
- 可以操作4个子钱包中的任意一个

# 六、开发优先级建议（基于依赖链路推导）

根据上面的层级关系和前后置依赖，合理的开发顺序应该是：

```
Phase 1  基础数据层（无外部依赖，最先开发）
─────────────────────────────────────────
  ├── 币种配置 B端CRUD（币种/汇率）
  ├── 币种配置 RPC暴露（币种查询、汇率查询、配置参数查询）
  └── 钱包 数据模型（4子钱包 × 多币种 余额表、订单表、记录表）

Phase 2  钱包核心能力（依赖Phase 1）
─────────────────────────────────────────
  ├── 钱包余额查询（C端）
  ├── 可用币种列表（C端）
  ├── 兑换功能（C端）── 只依赖汇率，最简单的交易链路
  ├── 记录查询（C端）── 纯读操作
  └── 钱包RPC暴露（入账/扣款/解冻/加减款）── 供财务调用

Phase 3  充值/提现链路（依赖Phase 1 + 2 + 财务模块就绪）
─────────────────────────────────────────
  ├── 充值功能（依赖财务.支付配置 + 财务.创建渠道订单）
  ├── 提现功能（依赖财务.支付配置 + 财务.提交审核 + 用户服务.KYC）
  └── 充值回调入账 / 提现审核回写（与财务联调）

Phase 4  B端管理 + 联调完善
─────────────────────────────────────────
  ├── 钱包B端接口（用户钱包查询等）
  ├── 与财务模块联调（充值/提现/人工修正全链路）
  └── 非功能性：幂等/对账/异常检测
```

**关键结论**：
- 币种配置是一切的前提，必须最先开发
- 兑换是最独立的功能，可以最早验证钱包核心余额操作逻辑
- 充值和提现强依赖财务模块，如果财务还没开发，钱包侧可以先把RPC接口定义好（IDL先行），内部逻辑先实现，联调后置
- 钱包的RPC暴露接口（入账/扣款/解冻/加减款）是财务模块的下游依赖，需要尽早提供给财务方

> 信息来源：产品需求文档和原型图。

## 8.1 币种配置 → 钱包（币种配置被钱包依赖）

币种配置是钱包的**数据地基**，钱包几乎所有功能都要从币种配置拿数据。

**依赖线1：钱包 ← 币种配置（可用币种列表）**

钱包C端的个人中心页面有一个"币种切换"弹窗（原型：2-2个人中心），弹窗里展示当前平台支持的全部币种（BSB/VND/IDR/THB），每个币种旁边还显示该用户在该币种下的余额。这个币种列表从哪来？从币种配置模块获取。币种配置后台运营人员配置了哪些币种是启用的（需求文档3-3：后台配置页面），启用的才会出现在C端列表里。如果运营把某个币种禁用了，C端就不展示了。所以这条依赖是：**钱包展示什么币种，完全取决于币种配置模块的数据**。

**依赖线2：钱包 ← 币种配置（汇率）**

汇率在钱包里至少有四个地方被用到：
- 充值页面：用户充值法币时，展示"≈ XX BSB"的换算提示（需求文档5-1：充值方式总览）
- 兑换页面：用户法币兑换BSB时，页面展示"1 BSB = 23000 VND"（需求文档6：兑换模块）
- 提现页面：BSB提现时，展示"提现100 BSB ≈ 2,300,000 VND"的换算（需求文档7-4：BSB提现）
- 个人中心：币种切换弹窗里可能展示各币种余额的BSB等值

汇率由币种配置模块后台手动设定（需求文档4-2：汇率规则说明），不是实时市场汇率。钱包模块自己不产生汇率，只消费汇率。所以这条依赖是：**钱包所有涉及跨币种换算展示和计算的场景，全部依赖币种配置的汇率数据**。

**依赖线3：钱包 ← 币种配置（币种配置参数：精度、限额、手续费、T+N）**

钱包在处理金额时需要知道每个币种的小数精度（比如VND精度0位，USDT精度2位），这决定了金额的展示和计算精度。提现时需要知道单笔最低/最高限额、每日限额、手续费规则、T+N到账天数（需求文档7-5：提现限制，需求文档3-3：后台配置页面字段），这些全部在币种配置后台设定。钱包只是读取并执行这些规则。所以这条依赖是：**钱包执行业务规则（金额精度、限额校验、手续费计算、到账时间展示）的参数全部来自币种配置**。

**依赖性质**：单向依赖，钱包 ← 币种配置。币种配置不依赖钱包的任何数据。币种配置是纯粹的上游数据提供方。

---

## 8.2 财务管理 → 钱包（支付配置被钱包依赖）

**依赖线4：钱包 ← 财务（支付配置 — 可用充值方式）**

用户在C端点开充值页面时，看到的充值方式列表（比如BSB下看到"USDT充值""银行转账""电子钱包"三个选项）不是钱包代码里硬编码的，而是从财务模块的支付配置动态获取的（需求文档5-1：充值方式总览，财务需求2-2：支付配置）。运营在财务后台配置一条"BSB - USDT充值"的支付方式，绑定了对应的代收通道，设定了金额范围和排序权重。C端用户看到的充值方式列表，就是财务后台启用的支付配置的映射。所以这条依赖是：**钱包C端"用户能用什么方式充值"这个问题的答案，完全由财务的支付配置决定**。

**依赖线5：钱包 ← 财务（支付配置 — 可用提现方式）**

同理，提现方式也由财务的支付配置决定。但提现还叠加了一层钱包自己的业务规则：提现方式还受用户充值来源限制（需求文档7-1：只充法币只能法币提现，只充USDT只能USDT提现）。所以提现方式的最终结果是：**财务支付配置提供"平台支持哪些提现方式" → 钱包根据用户充值历史过滤 → 得到"该用户实际可用的提现方式"**。两层叠加，财务提供候选集，钱包做业务过滤。

**依赖线6：钱包 ← 财务（创建渠道支付订单）**

用户确认充值后，钱包需要把订单交给财务去对接第三方支付渠道（财务需求2-1：通道配置，2-3：充值流程）。因为通道的API接入、通道轮询选择、费率计算这些都是财务模块管的，钱包不直接跟第三方渠道打交道。钱包告诉财务"用户要充100 VND，走银行转账"，财务负责匹配最优通道、调用通道API、返回支付跳转链接或收款地址给钱包。所以这条依赖是：**钱包创建充值订单后，实际的支付渠道对接由财务完成，钱包拿到支付信息返回给C端**。

**依赖线7：钱包 ← 财务（提交提现审核）**

用户提交提现后，钱包先冻结用户余额（这步钱包自己完成），然后需要把提现订单提交给财务进入多级审核流程（财务需求2-4：提现流程）。风控审核 → 财务审核 → 出款审核，这三级审核全部在财务模块内完成，钱包不参与审核过程。所以这条依赖是：**钱包负责冻结余额和创建提现订单，但审核和出款由财务接管**。

**依赖性质**：钱包在充值和提现的"执行环节"强依赖财务。没有财务模块，用户可以看余额、可以兑换，但无法充值和提现。

---

## 8.3 钱包 → 财务（钱包被财务依赖 — 回调方向）

上面是钱包依赖财务，下面反过来，财务也依赖钱包。

**依赖线8：财务 → 钱包（充值入账）**

第三方支付渠道支付成功后会发回调通知给财务模块（财务需求2-3：充值流程图），财务确认回调合法性后，需要通知钱包模块给用户入账。这个"入账"操作只有钱包能做，因为用户余额在钱包模块管理。所以财务必须调用钱包的RPC接口来完成入账。这条依赖是：**财务收到渠道回调后，靠调用钱包的入账接口让用户余额增加，财务自己不能直接改用户余额**。

**依赖线9：财务 → 钱包（提现确认扣款 / 失败解冻）**

提现审核全部通过且渠道出款成功后，财务通知钱包把之前冻结的金额正式扣除（钱包.提现确认扣款）。如果审核被拒或出款失败，财务通知钱包解冻退回（钱包.提现失败解冻）。钱包在提现创建时只做了冻结操作，最终是扣还是退，取决于财务的审核和渠道结果。这条依赖是：**提现的最终结算（扣款或退回）由财务的审核结果驱动，但执行动作在钱包侧**。

**依赖线10：财务 → 钱包（人工加款/减款/补单）**

财务后台的人工修正功能（财务需求2-5：人工修正）允许操作人员直接给用户加钱、扣钱或补录充值。审核通过后，财务调用钱包的RPC接口执行余额操作。关键点是人工修正可以**指定操作哪个子钱包**（中心/奖励/代理/主播），所以钱包的加款/减款接口必须支持子钱包类型参数。这条依赖是：**财务的人工修正需要直接操作钱包余额，钱包必须暴露精确到子钱包粒度的余额操作接口**。

**依赖线11：财务 → 钱包（余额查询）**

财务在创建人工减款单时，需要先查看用户当前余额，确认余额充足（不能减成负数）。所以财务需要调用钱包的余额查询接口。这条依赖是辅助性的：**财务在人工修正场景下需要读取钱包余额作为操作前的校验依据**。

**依赖性质**：财务在回调和人工修正场景下反向依赖钱包。这是后置/异步方向的依赖，与上面钱包依赖财务（前置/同步方向）形成双向耦合。

---

## 8.4 币种配置 → 财务（币种配置被财务依赖）

**依赖线12：财务 ← 币种配置（汇率 — BSB换算列）**

财务后台的充值记录和提现记录列表页面（财务原型：充值记录、提现记录），每条记录除了显示原始币种金额外，还有一列"基准币种金额"也就是折算成BSB后的金额。这个折算需要汇率。所以财务的记录展示依赖币种配置的汇率数据。这条依赖是：**财务后台展示BSB换算列时需要读取币种配置的汇率**。

**依赖线13：财务 ← 币种配置（币种列表 — 筛选条件）**

财务后台的记录查询页面有"币种"筛选下拉框，下拉框里的选项来自币种配置模块。这是一个轻度依赖：**财务后台的筛选条件需要币种配置提供币种列表**。

**依赖性质**：轻度单向依赖，币种配置为财务提供基础数据。财务不依赖币种配置的实时响应，可以通过缓存降低耦合。

---

## 8.5 钱包 → 用户服务（外部依赖）

**依赖线14：钱包 ← 用户服务（KYC状态和信息）**

钱包的提现功能有一个硬性前置条件：非USDT提现必须完成KYC实名认证（需求文档7-1：提现模块）。用户点击提现时，钱包先检查用户的KYC状态，未认证则引导去认证页面，已认证才允许继续。此外，提现填写银行卡户名时，系统会自动填充KYC认证的真实姓名（只读），并且后续校验提现账户户名与KYC姓名是否一致。这条依赖是：**钱包提现的前置校验和户名一致性校验都需要读取用户服务的KYC数据**。

KYC本身的认证流程（提交证件、审核）属于用户服务的能力范畴，钱包只是消费KYC结果，不负责KYC审核逻辑。

---

## 8.6 三模块之间没有依赖的部分

明确列出哪些功能之间**不存在依赖**，避免过度设计耦合：

**币种配置 与 财务管理之间没有强耦合**：币种配置不需要知道财务的通道信息，财务也不需要回写币种配置。两者的关系是"财务读取币种配置的数据"这种单向弱依赖。

**兑换功能 与 财务管理之间没有依赖**：兑换是法币→BSB的内部操作，只涉及用户自己的余额在不同币种间的转换，不经过任何支付通道，不需要财务参与。兑换只依赖币种配置的汇率，是钱包模块内部闭环的功能。

**记录查询 与 任何外部模块之间没有运行时依赖**：充值记录、提现记录、兑换记录、奖励记录的查询都是读自己模块内的数据表，不需要实时调用其他模块。数据是在充值/提现/兑换流程中写入的，查询时是自闭环的。

**币种配置B端操作 与 其他模块没有运行时依赖**：币种的增删改、汇率的设置、日志的查询，都是币种配置模块自己的B端管理操作，不需要调用钱包或财务的接口。修改币种配置后，钱包和财务通过下次查询或缓存刷新感知变化。

**财务B端的内部管理操作 与 钱包大多没有运行时依赖**：通道配置CRUD、支付配置CRUD、记录查询、统计查询，这些都是财务模块自己的B端操作。唯一涉及钱包的是人工修正（审核通过后调用钱包余额操作）和提现审核（审核完成后通知钱包扣款/解冻）。

## 8.7 依赖关系总览（箭头描述）

将上面14条依赖线汇总，按方向归类：

### 钱包依赖的上游（钱包主动调用）

```
钱包 ← 币种配置.可用币种列表        用于C端币种切换、币种展示
钱包 ← 币种配置.汇率查询            用于充值/兑换/提现的跨币种换算
钱包 ← 币种配置.配置参数            用于精度计算、限额校验、手续费、T+N
钱包 ← 财务.支付配置查询            用于C端充值方式和提现方式展示
钱包 ← 财务.创建渠道支付订单        用于充值时对接第三方支付渠道
钱包 ← 财务.提交提现审核            用于提现时将订单推入审核流程
钱包 ← 用户服务.KYC状态/信息        用于提现前置校验和户名一致性校验
```

### 钱包被依赖的下游（钱包被动接受调用）

```
财务 → 钱包.充值入账                 渠道回调成功后触发
财务 → 钱包.提现确认扣款             审核通过+出款成功后触发
财务 → 钱包.提现失败解冻             审核拒绝或出款失败后触发
财务 → 钱包.人工加款                 加款审核通过后触发
财务 → 钱包.人工减款                 减款审核通过后触发
财务 → 钱包.人工补单入账             补单审核通过后触发
财务 → 钱包.余额查询                 人工修正创建时查询
```

### 币种配置被依赖（纯数据提供方）

```
钱包 ← 币种配置（3条依赖线：币种列表、汇率、配置参数）
财务 ← 币种配置（2条依赖线：汇率换算、币种筛选列表）
```

币种配置不依赖任何其他模块，是整个体系的最底层数据源。

### 无依赖关系的组合

```
币种配置 ←✕→ 财务    无强耦合，只有财务单向读取币种配置数据
兑换功能 ←✕→ 财务    兑换不经过支付渠道，内部闭环
记录查询 ←✕→ 外部    查询时自闭环，不需要实时跨模块调用
```

## 8.8 一句话总结每条依赖

| 编号 | 方向 | 一句话说明 |
|------|------|-----------|
| 1 | 钱包 ← 币种配置 | 钱包展示什么币种，由币种配置决定 |
| 2 | 钱包 ← 币种配置 | 钱包所有跨币种换算，依赖币种配置的汇率 |
| 3 | 钱包 ← 币种配置 | 钱包的限额/精度/手续费/T+N规则参数，来自币种配置 |
| 4 | 钱包 ← 财务 | C端用户看到什么充值方式，由财务的支付配置决定 |
| 5 | 钱包 ← 财务 | C端用户看到什么提现方式，由财务支付配置+钱包充值历史共同决定 |
| 6 | 钱包 ← 财务 | 充值实际对接第三方渠道的工作，由财务完成 |
| 7 | 钱包 ← 财务 | 提现审核和出款由财务接管，钱包只管冻结 |
| 8 | 财务 → 钱包 | 充值成功后，财务通知钱包入账 |
| 9 | 财务 → 钱包 | 提现最终结果（扣款或退回），由财务通知钱包执行 |
| 10 | 财务 → 钱包 | 人工加减款补单，财务审核通过后调钱包执行余额操作 |
| 11 | 财务 → 钱包 | 人工修正前，财务需查钱包余额做校验 |
| 12 | 财务 ← 币种配置 | 财务后台记录展示BSB换算列，需要汇率 |
| 13 | 财务 ← 币种配置 | 财务后台筛选下拉框，需要币种列表 |
| 14 | 钱包 ← 用户服务 | 提现前必须校验KYC，户名必须与KYC一致 |
