│   │   充值·兑换·提现·记录·余额查询·提现账户管理                    │    │

**层级结论**：
- **币种配置** 是最底层的基础设施，被钱包和财务共同依赖，自身不依赖业务层
- **钱包** 是面向用户的业务编排层，同时依赖币种配置和财务管理
- **财务管理** 是支付通道管理层，依赖币种配置，同时与钱包存在**双向依赖**（这是最关键的耦合点）

## 三、接口依赖分层分析

### 3.1 第一层：基础设施（币种配置）— 无外部业务依赖，最先就绪

币种配置是整个钱包体系的地基。**所有涉及币种、汇率、充值档位、提现限额的操作都依赖它先存在。**

```
币种配置提供的 RPC 接口（被其他模块调用）:

  GetActiveCurrencyList ──► 钱包（C端币种切换、余额概览）
                        ──► 财务（统计报表币种筛选）

  GetExchangeRate ──► 钱包（兑换计算、充值汇率展示、提现汇率计算）
                  ──► 财务（统计换算为基准币种）

  BatchGetExchangeRates ──► 钱包（多币种余额概览换算）
                        ──► 财务（批量统计）

  GetCurrencyDetail ──► 钱包（币种完整配置含档位限额）

  GetDepositWithdrawConfig ──► 钱包（充值档位展示、提现限额校验）
```

**结论：币种配置的 RPC 接口是前置依赖，必须最先开发就绪。没有它，钱包和财务的几乎所有业务都无法运转。**

### 3.3 第三层：耦合功能（存在双向/链式依赖，需要协调接口）

#### 3.3.1 充值链路 — 钱包↔财务双向依赖

这是最复杂的链路，存在**请求方向和回调方向的双向依赖**：

```
充值完整调用链路：

 用户(C端)          钱包(ser-wallet)           财务(ser-finance)         三方通道
    │                    │                         │                      │
    │  1.进入充值页面      │                         │                      │
    │───────────────────►│                         │                      │
    │                    │  2.获取充值方式列表        │                      │
    │                    │  [RPC] GetPaymentMethods │                      │
    │                    │────────────────────────►│                      │
    │                    │◄────────────────────────│                      │
    │◄───────────────────│  3.返回方式列表           │                      │
    │                    │                         │                      │
    │  4.选择金额+方式     │                         │                      │
    │  点击充值           │                         │                      │
    │───────────────────►│                         │                      │
    │                    │  5.重复转账检测            │                      │
    │                    │  (本地DB查询)             │                      │
    │                    │                         │                      │
    │                    │  6.匹配通道               │                      │
    │                    │  [RPC] MatchDepositChannel                      │
    │                    │────────────────────────►│                      │
    │                    │◄────────────────────────│                      │
    │                    │                         │                      │
    │                    │  7.创建充值订单(本地DB)    │                      │
    │                    │                         │                      │
    │                    │  8.向通道发起支付请求      │                      │
    │                    │────────────────────────►│─────────────────────►│
    │                    │                         │◄─────────────────────│
    │◄───────────────────│  9.返回三方支付页面       │                      │
    │                    │                         │                      │
    │  (用户在三方完成支付) │                         │                      │
    │                    │                         │                      │
    │                    │         10.三方回调支付结果│                      │
    │                    │◄────────────────────────│◄─────────────────────│
    │                    │         或 财务转发回调    │                      │
    │                    │                         │                      │
    │                    │  11.幂等校验+入账         │                      │
    │                    │  [本地] CreditBalance     │                      │
    │                    │  更新余额+生成流水        │                      │
    │                    │                         │                      │
    │◄───────────────────│  12.通知前端更新          │                      │
    │                    │                         │                      │
```

**充值链路的依赖关系**：

| 步骤 | 接口 | 所属模块 | 前置依赖 | 说明 |
|------|------|---------|---------|------|
| 2 | GetPaymentMethods | 财务→钱包调用 | 财务的支付配置CRUD已完成 | 钱包展示充值方式 |
| 5 | CheckDuplicateDeposit | 钱包本地 | 钱包充值订单表已设计 | 本地查询，无外部依赖 |
| 6 | MatchDepositChannel | 财务→钱包调用 | 财务的通道配置+轮询规则已完成 | 匹配最优通道 |
| 7 | CreateDepositOrder | 钱包本地 | 钱包订单表设计 | 本地创建订单 |
| 10 | DepositCallback | 钱包提供 | 钱包的回调处理逻辑 | 财务或三方直接回调 |
| 11 | CreditBalance | 钱包内部 | 钱包余额表+流水表设计 | **核心：必须幂等** |

**关键耦合点**：
- 步骤2、6：钱包**主动调用**财务（钱包是调用方）
- 步骤10：财务**回调**钱包（财务是调用方）
- 这就是**双向依赖**：充值发起时钱包调财务，充值完成时财务调钱包

#### 3.3.2 提现链路 — 钱包↔财务双向依赖（更复杂）

```
提现完整调用链路：

 用户(C端)          钱包(ser-wallet)           财务(ser-finance)       ser-kyc
    │                    │                         │                    │
    │  1.点击提现         │                         │                    │
    │───────────────────►│                         │                    │
    │                    │  2.查询KYC状态            │                    │
    │                    │  [RPC] KycQuery          │                    │
    │                    │─────────────────────────────────────────────►│
    │                    │◄─────────────────────────────────────────────│
    │                    │                         │                    │
    │◄───────────────────│  3.根据KYC状态展示       │                    │
    │  (未认证弹窗/       │    可用提现方式           │                    │
    │   已认证直接进入)    │                         │                    │
    │                    │                         │                    │
    │  4.填写信息+金额     │                         │                    │
    │  确认提现           │                         │                    │
    │───────────────────►│                         │                    │
    │                    │  5.校验提现限额           │                    │
    │                    │  (读取币种配置限额)       │                    │
    │                    │                         │                    │
    │                    │  6.冻结余额               │                    │
    │                    │  [本地] FreezeBalance     │                    │
    │                    │                         │                    │
    │                    │  7.创建提现订单            │                    │
    │                    │  提交给财务审核            │                    │
    │                    │────────────────────────►│                    │
    │                    │                         │                    │
    │◄───────────────────│  8.返回"提现申请已提交"   │                    │
    │                    │                         │                    │
    │                    │      === B端审核阶段 ===  │                    │
    │                    │                         │                    │
    │                    │  9a.审核驳回 → 调钱包解冻  │                    │
    │                    │◄────────────────────────│                    │
    │                    │  [RPC] UnfreezeBalance   │                    │
    │                    │                         │                    │
    │                    │  9b.审核通过+出款成功      │                    │
    │                    │◄────────────────────────│                    │
    │                    │  [RPC] DebitBalance      │                    │
    │                    │  (冻结→实际扣减)          │                    │
    │                    │                         │                    │
```

**提现链路的依赖关系**：

| 步骤 | 接口 | 调用方向 | 前置依赖 |
|------|------|---------|---------|
| 2 | KycQuery | 钱包→KYC | ser-kyc已就绪 |
| 5 | GetDepositWithdrawConfig | 钱包→币种配置 | 币种配置限额数据已配置 |
| 6 | FreezeBalance | 钱包内部 | 钱包余额表支持冻结字段 |
| 7 | 提交提现订单 | 钱包→财务 | 财务提现审核流程已就绪 |
| 9a | UnfreezeBalance | 财务→钱包 | 钱包提供解冻RPC |
| 9b | DebitBalance | 财务→钱包 | 钱包提供扣款RPC |

**关键耦合点**：
- 步骤7：钱包**推送**提现单给财务
- 步骤9a/9b：财务审核完成后**回调**钱包进行余额操作
- 同样是**双向依赖**

#### 3.3.3 人工修正链路 — 财务→钱包单向依赖

```
B端操作员            财务(ser-finance)           钱包(ser-wallet)
    │                    │                         │
    │  1.创建加款/减款单   │                         │
    │───────────────────►│                         │
    │                    │  2.提交审核               │
    │                    │  (操作人≠审核人)          │
    │                    │                         │
    │  3.审核通过          │                         │
    │───────────────────►│                         │
    │                    │  4.调用钱包调整余额        │
    │                    │  [RPC] ManualAdjust      │
    │                    │────────────────────────►│
    │                    │                         │  5.执行加款/减款
    │                    │                         │  生成流水记录
    │                    │◄────────────────────────│
    │                    │  6.更新修正单状态          │
    │◄───────────────────│                         │
```

**这条链路是单向的**：财务→钱包，钱包只需要提供 ManualAdjust RPC 接口即可。

## 四、接口前后置关系总结

### 4.1 前置接口（必须先就绪，否则后续功能无法运转）

```
优先级 P0 — 地基层：

  币种配置 B端 CRUD ──────────────────► 所有业务的前提
  │
  ├─ CreateCurrency / UpdateCurrency    (有币种数据才能操作)
  ├─ StatusCurrency                     (启用后C端才可见)
  ├─ UpdateExchangeRate                 (有汇率才能计算)
  └─ SaveDepositConfig / SaveWithdrawConfig (有档位限额才能校验)
      │
      ▼
  币种配置 RPC 接口 ──────────────────► 钱包和财务共同的数据源
  │
  ├─ GetActiveCurrencyList             (钱包展示、财务筛选)
  ├─ GetExchangeRate                   (兑换、充值、提现计算)
  └─ GetDepositWithdrawConfig          (充值档位、提现限额)
```

### 4.2 后置接口（依赖前置数据/接口就绪后才能开发联调）

```
优先级 P0 — 依赖地基就绪后：

  钱包 C端核心 ───────────────────────► 依赖币种配置RPC
  │
  ├─ GetWalletBalance / GetWalletOverview    (依赖：币种列表)
  ├─ GetDepositMethods                       (依赖：币种配置 + 财务支付配置)
  ├─ CreateDepositOrder                      (依赖：财务通道匹配)
  ├─ ExchangePreview / ExecuteExchange       (依赖：汇率)
  ├─ CheckWithdrawEligibility                (依赖：KYC + 币种限额配置)
  └─ CreateWithdrawOrder                     (依赖：币种限额 + 财务审核流程)

  钱包 RPC提供 ──────────────────────► 被财务回调，但自身可独立开发
  │
  ├─ CreditBalance                           (充值入账，被财务回调)
  ├─ FreezeBalance / UnfreezeBalance         (提现冻结/解冻，被财务回调)
  ├─ DebitBalance                            (提现扣款，被财务回调)
  ├─ ManualAdjust                            (人工修正，被财务回调)
  └─ DepositCallback                         (充值回调入口)
```

### 4.3 独立接口（无跨模块依赖，可随时开发）

```
完全独立：

  钱包记录模块（纯本地DB查询）：
  ├─ PageDepositRecords / DetailDepositRecord
  ├─ PageWithdrawRecords / DetailWithdrawRecord
  ├─ PageExchangeRecords
  └─ PageRewardRecords

  钱包提现账户管理（仅依赖KYC姓名）：
  ├─ ListWithdrawAccounts
  └─ BindWithdrawAccount

  钱包 B端查询（纯本地DB查询）：
  ├─ QueryUserWalletPage / QueryUserWalletDetail
  └─ QueryUserTransactionPage

  币种配置 B端操作日志（纯本地DB查询）：
  ├─ PageExchangeRateLog
  └─ PageCurrencyOperationLog
```

## 五、耦合度分析

### 5.1 模块间耦合矩阵

```
          ┌──────────┬──────────┬──────────┬──────────┬──────────┐
          │ 币种配置  │ 钱包      │ 财务管理  │ ser-kyc  │ ser-blog │
┌─────────┼──────────┼──────────┼──────────┼──────────┼──────────┤
│ 币种配置 │    —     │ 被调用    │ 被调用    │    ×     │ 调用     │
│         │          │ (5个RPC) │ (2个RPC) │          │ (审计)   │
├─────────┼──────────┼──────────┼──────────┼──────────┼──────────┤
│ 钱包     │ 调用     │    —     │ 双向依赖  │ 调用     │ 调用     │
│         │ (5个RPC) │          │(出2+入5) │ (1个RPC) │ (审计)   │
├─────────┼──────────┼──────────┼──────────┼──────────┼──────────┤
│ 财务管理 │ 调用     │ 双向依赖  │    —     │    ×     │ 调用     │
│         │ (2个RPC) │(出5+入2) │          │          │ (审计)   │
├─────────┼──────────┼──────────┼──────────┼──────────┼──────────┤
│ ser-kyc │    ×     │ 被调用    │    ×     │    —     │    ×     │
│         │          │ (1个RPC) │          │          │          │
├─────────┼──────────┼──────────┼──────────┼──────────┼──────────┤
│ ser-blog│ 被调用    │ 被调用    │ 被调用    │    ×     │    —     │
│         │ (oneway) │ (oneway) │ (oneway) │          │          │
└─────────┴──────────┴──────────┴──────────┴──────────┴──────────┘

读法：行调用列。如"钱包行-币种配置列 = 调用(5个RPC)"表示钱包调用币种配置5个RPC接口。
× = 无直接依赖
```

### 5.2 耦合程度评估

| 关系 | 耦合程度 | 原因 | 解耦建议 |
|------|---------|------|---------|
| **钱包 ↔ 财务** | **高耦合** | 充值和提现链路都是双向调用：钱包发起→财务处理→财务回调钱包 | 明确RPC接口契约，先定义IDL；回调接口设计幂等；考虑异步消息解耦（如用MQ替代同步回调） |
| 钱包 → 币种配置 | 中耦合（单向） | 钱包大量读取币种配置数据，但方向单一 | 币种配置数据变化低频，可缓存；接口幂等无状态 |
| 财务 → 币种配置 | 低耦合（单向） | 仅统计换算时读取汇率 | 同上，缓存即可 |
| 钱包 → ser-kyc | 低耦合（单向） | 仅提现时查一次KYC状态 | 单个RPC调用，简单依赖 |
| 全部 → ser-blog | 极低耦合 | oneway异步审计日志，不影响主链路 | 已有项目惯例，直接复用 |

### 5.3 钱包↔财务双向依赖的详细接口清单

**钱包调用财务（钱包主动发起，2个RPC出口）**：

| 接口 | 触发场景 | 同步/异步 |
|------|---------|---------|
| GetPaymentMethods | 用户进入充值页面，获取充值方式列表 | 同步 |
| MatchDepositChannel | 用户点击充值，匹配最优通道 | 同步 |

**财务调用钱包（财务主动发起，5个RPC出口）**：

| 接口 | 触发场景 | 同步/异步 | 幂等要求 |
|------|---------|---------|---------|
| CreditBalance | 充值回调成功，入账 | 同步 | **必须幂等** |
| FreezeBalance | 提现申请提交时 | 同步 | 幂等 |
| UnfreezeBalance | 提现审核驳回时 | 同步 | 幂等 |
| DebitBalance | 提现出款成功时 | 同步 | 幂等 |
| ManualAdjust | 人工加款/减款/补单审核通过时 | 同步 | 幂等 |

## 七、关键风险点

### 7.1 双向依赖导致的循环问题

钱包和财务互相调用，如果两个服务同时启动，可能出现：
- 钱包启动时需要财务的RPC客户端 → 但财务还没注册到ETCD
- 财务启动时需要钱包的RPC客户端 → 但钱包还没注册到ETCD

**应对**：项目使用 `sync.Once` 惰性初始化RPC客户端，调用时才创建连接，不会在启动时阻塞。但需确保ETCD服务发现的容错机制。

### 7.2 充值回调幂等性

需求文档明确要求充值回调必须幂等。这意味着 `DepositCallback` / `CreditBalance` 接口需要：
- 基于订单号去重
- 状态机校验（只有"待入账"状态才执行入账）
- 同一订单多次回调不产生重复流水

### 7.3 提现冻结-扣款的一致性

提现链路中余额经历：可用→冻结→扣减（或可用→冻结→解冻）。如果 FreezeBalance 成功但后续审核RPC调用失败，会导致资金悬挂。需要：
- 冻结记录关联提现订单号
- 超时未处理的冻结自动解冻机制
- 或使用事务消息保证最终一致性

### 7.4 币种配置变更的影响范围

币种被禁用后：C端不可充值、不可提现。但如果用户该币种有余额，需要弹提示。这意味着：
- 币种状态变更需要通知钱包（或钱包每次查询时实时校验）
- 汇率变更不影响已创建的订单（订单应锁定创建时的汇率快照）

## 九、三大模块依赖关系文字描述（基于产品需求文档和原型）

> 以下内容以"谁依赖谁、什么时候依赖、为什么依赖"的视角，用文字+箭头方式描述三个模块之间的每一条依赖关系。
> 信息来源：ser-wallet/tmp/ 下的产品需求文档和产品原型图。

### 9.1 币种配置模块 → 谁都不依赖（纯基础设施）

币种配置是整个钱包体系的数据地基。从产品需求文档来看，币种配置管理的是"币种元数据"和"汇率规则"，这些数据是静态配置性质的，不会反向调用任何业务模块。

**币种配置模块自身的数据来源**：

- 币种基础数据（代码、名称、类型、图标、国家、符号规则） → 开发导入+B端编辑
- 实时汇率 → 三方API接口拉取（3家取平均值）
- 平台汇率/入款汇率/出款汇率 → 基于实时汇率+阈值规则自动计算
- 汇率浮动%、阈值% → B端手动配置
- 充值档位、提现限额 → B端手动配置

以上所有数据来源都不涉及对钱包模块或财务模块的调用。因此：

**币种配置 → 不依赖钱包**
**币种配置 → 不依赖财务**
**币种配置 → 仅依赖三方汇率API（外部）和 ser-blog（审计日志，oneway）、ser-s3（图标上传）**

这意味着币种配置可以完全独立开发、独立测试、独立上线，不存在任何业务模块的前置阻塞。

---

### 9.2 钱包模块 → 依赖币种配置（单向，强依赖，高频读取）

钱包模块几乎所有面向C端的功能都依赖币种配置的数据。以下逐场景说明：

#### 场景1：用户打开钱包首页

用户打开个人中心钱包页面时，需要展示当前币种余额和币种切换列表。

- 钱包 → 币种配置：获取所有已启用的币种列表（代码、名称、图标、符号规则）
- 钱包 → 币种配置：获取各币种对BSB的汇率，用于余额概览中的"折合BSB总额"换算

**依据**：需求文档"4-1钱包总体设计"明确个人中心顶部展示余额，右上角有币种切换入口列出所有可用币种及余额。产品原型"2-1~2-3个人中心"展示了币种下拉列表。

#### 场景2：用户进入充值页面

用户选择某币种后进入充值界面，需要展示充值档位和汇率信息。

- 钱包 → 币种配置：获取当前币种的充值档位配置（快捷金额列表、最低档位）
- 钱包 → 币种配置：获取当前币种对BSB的**入款汇率**（仅BSB充值时展示"充值X BSB = Y USDT"的换算）

**依据**：需求文档"5-1充值方式总览"说"每个币种进来有默认充值方式及默认最小档位快捷金额"，产品原型"3-1~3-4充值币种切换"展示了不同币种的充值档位。

#### 场景3：用户执行兑换（法币→BSB）

兑换功能需要汇率进行计算。

- 钱包 → 币种配置：获取当前法币对BSB的**入款汇率**，用于计算"兑换获得BSB数量"
- 计算公式：兑换获得 = 输入法币金额 ÷ 入款汇率

**依据**：需求文档"6兑换模块"明确展示"1 BSB = X 法币"的汇率，以及兑换金额、兑换获得、赠送奖励、实际到账的完整计算。

#### 场景4：用户申请提现

提现时需要校验限额，BSB提现还需要汇率换算。

- 钱包 → 币种配置：获取当前币种的**提现限额配置**（单笔限额、单日限额、手续费规则）
- 钱包 → 币种配置：获取当前币种的**出款汇率**（BSB提现时需展示"提现X BSB = Y 法币/USDT"的换算）

**依据**：需求文档"7-5提现限制"说"单笔限额、单日限额、手续费规则"由后台配置。需求文档"7-4BSB提现"说"只有BSB提现金额输入框下有对应汇率展示"。

#### 场景5：币种被禁用后的影响

B端禁用某币种后，钱包需要实时感知。

- 钱包 → 币种配置：每次操作前校验币种状态是否为启用
- 如果币种已禁用：C端不可充值、不可提现，若用户该币种有余额，使用时弹出提示

**依据**：需求文档"3-3后台配置页面说明"明确"禁用后该币种在C端不可使用、不可充值、不可提现，若用户该币种有余额，使用时需弹出对应提示"。

**小结**：钱包 → 币种配置 是**单向读取**关系，钱包只读不写。币种配置的任何数据变更（汇率更新、币种禁用、限额调整）都可能影响钱包的业务表现，但币种配置不需要知道钱包的任何信息。

---

### 9.3 钱包模块 → 依赖财务管理（充值发起方向，钱包主动调用财务）

充值流程中，钱包需要从财务获取两类信息：可用的充值方式、匹配的支付通道。

#### 依赖点1：获取充值方式列表

- 钱包 → 财务：用户进入充值页面时，钱包将当前币种传给财务，财务返回该币种下已配置且启用的充值方式列表（如"银行转账"、"电子钱包"、"USDT"）以及每种方式的icon、排序
- 为什么钱包不自己管充值方式？因为充值方式与支付通道强关联，通道配置在财务模块，充值方式也必须由财务统一管理

**依据**：财务需求文档"2-2支付配置"明确充值方式配置包含"币种、充值方式名称、支付icon、充值档位/范围、关联通道、排序、状态"，这些配置在财务B端管理。

#### 依赖点2：匹配支付通道

- 钱包 → 财务：用户点击"去充值"后，钱包将币种+金额+充值方式传给财务，财务根据通道轮询规则（成功率系数×时间系数×预警系数）匹配最优通道，返回通道信息
- 匹配成功 → 钱包拿到通道信息后创建充值订单，向三方发起支付
- 匹配失败 → 钱包提示"充值通道异常失效，请更换充值方式"

**依据**：财务需求文档"2-1通道配置"说"通道匹配权重由三个系数决定：成功率系数、时间系数、预警系数"，财务需求文档"2-3充值记录"的流程图明确"校验规则 → 是否匹配到可用通道"这一步骤。

---

### 9.4 财务管理 → 依赖钱包（回调方向，财务主动调用钱包）

这是**反向依赖**，也是整个体系中耦合度最高的部分。财务在以下场景需要主动调用钱包的RPC接口来操作用户余额：

#### 依赖点1：充值成功入账

- 财务 → 钱包（CreditBalance）：三方支付通道回调支付成功后，财务通知钱包为用户入账
- 入账目标：充值金额进入**中心钱包**，额外奖金进入**奖励钱包**
- **幂等要求**：同一订单重复回调不得重复入账

**依据**：需求文档"5-6额外奖金选择规则"说"充值金额进入中心钱包，奖金进入奖励钱包"。需求文档"9非功能性需求"明确"充值回调必须做幂等处理，防止重复入账"。

#### 依赖点2：提现冻结余额

- 财务 → 钱包（FreezeBalance）：用户提交提现申请后，在进入三级审核之前，需要先冻结用户的可用余额
- 冻结的作用：防止用户在提现审核期间把钱花掉，导致审核通过后余额不足

**依据**：财务需求文档"2-4提现记录"流程图明确"校验：余额、限额、KYC → 冻结用户可用余额"。

#### 依赖点3：提现驳回解冻

- 财务 → 钱包（UnfreezeBalance）：风控审核/财务审核/出款审核任一环节不通过时，需要解冻之前冻结的余额，归还给用户
- 三级审核（风控审核 → 财务审核 → 出款审核）中任何一级驳回都触发解冻

**依据**：财务需求文档"2-4提现记录"流程图明确"不通过 → 余额解冻/订单失败"，三个审核节点都有这条分支。

#### 依赖点4：提现出款成功扣款

- 财务 → 钱包（DebitBalance）：三方代付通道出款成功后，将用户的冻结金额正式扣减（冻结→扣减）
- 如果出款失败 → 走解冻逻辑（同依赖点3）

**依据**：财务需求文档"2-4提现记录"流程图明确"是否成功？→ 是 → 订单审核状态为出款成功 → 订单状态为已完成"，同时钱包金额更新。

#### 依赖点5：人工加款/减款/补单

- 财务 → 钱包（ManualAdjust）：B端操作员创建加款/减款/补单单据，审核通过后调用钱包执行余额变更
- 加款可针对4类钱包分别设置金额（中心/奖励/主播/代理）
- 减款超过余额时需要前置校验提示

**依据**：财务需求文档"2-5人工修正"明确"审核通过后调用钱包服务加款/减款"，产品原型"人工修正1~10"展示了完整的加款减款操作流程和审核流程。

**小结**：财务 → 钱包 共5个RPC调用点，全部是**资金操作类**（入账/冻结/解冻/扣款/调整），这5个接口全部需要**幂等设计**，因为涉及真金白银，重复调用不能产生重复资金变动。

---

### 9.5 财务管理 → 依赖币种配置（单向，弱依赖，统计换算）

财务模块对币种配置的依赖相比钱包要弱得多，主要用于统计和展示：

#### 依赖点1：统计报表的币种换算

- 财务 → 币种配置：充值统计、提现统计、通道统计等报表需要将各币种金额统一换算为基准币种（BSB）进行汇总
- 使用的是**平台汇率**（非入款/出款汇率）

**依据**：币种配置需求文档"4-1汇率应用场景"明确"统计 → 平台汇率 → 平台充值、提现、账变等统计需要根据平台汇率转化为基准货币进行报表统计"。

#### 依赖点2：通道配置/支付配置的币种筛选

- 财务 → 币种配置：通道配置和支付配置中需要关联"币种"字段，B端下拉选择时需要获取可用币种列表

**依据**：财务需求文档"2-1通道配置"和"2-2支付配置"中，通道和支付方式都有"币种"字段。

### 9.6 三模块间完整依赖关系箭头总览

按调用方向整理（A → B 表示 A 调用 B）：

```
一、币种配置的对外供给（被调用）

  币种配置 ←── 钱包（5条依赖线）
    ├─ 获取已启用币种列表        ←── 钱包首页币种切换、余额概览
    ├─ 获取单币种汇率（入款/出款） ←── 充值汇率展示、兑换计算、提现换算
    ├─ 批量获取多币种汇率         ←── 多币种余额概览折合BSB
    ├─ 获取充值档位配置           ←── 充值页面快捷金额展示
    └─ 获取提现限额配置           ←── 提现时校验单笔/单日限额

  币种配置 ←── 财务（2条依赖线）
    ├─ 获取已启用币种列表        ←── 通道配置/支付配置中的币种下拉
    └─ 获取平台汇率              ←── 统计报表多币种换算为BSB

二、钱包的对外调用（钱包主动调用别人）

  钱包 → 币种配置：5条（见上）
  钱包 → 财务：2条
    ├─ 获取充值方式列表           → 用户进入充值页时展示可用方式
    └─ 匹配支付通道               → 用户点击充值时匹配最优通道
  钱包 → ser-kyc：1条
    └─ 查询KYC认证状态            → 提现前判断是否弹认证提示

三、钱包的对外供给（钱包被调用）

  钱包 ←── 财务（5条依赖线）
    ├─ 充值入账（CreditBalance）   ←── 三方回调充值成功后
    ├─ 冻结余额（FreezeBalance）   ←── 提现申请提交进入审核前
    ├─ 解冻余额（UnfreezeBalance） ←── 提现审核驳回时
    ├─ 扣款确认（DebitBalance）    ←── 提现出款成功后
    └─ 人工调整（ManualAdjust）    ←── 加款/减款/补单审核通过后

四、财务的对外调用（财务主动调用别人）

  财务 → 币种配置：2条（见上）
  财务 → 钱包：5条（见上）
  财务 → 三方支付通道：2条
    ├─ 发起代收请求               → 充值时向三方创建支付订单
    └─ 发起代付请求               → 提现出款时向三方发起打款

五、财务的对外供给（财务被调用）

  财务 ←── 钱包（2条依赖线）
    ├─ 获取充值方式列表           ←── 用户进入充值页
    └─ 匹配支付通道               ←── 用户点击充值
```

### 9.7 三条核心业务链路的文字描述

#### 链路一：充值链路

```
用户操作 → 钱包 → 币种配置 → 钱包 → 财务 → 三方 → 财务 → 钱包
```

文字描述：

1. 用户打开充值页面，**钱包 → 币种配置**获取当前币种的充值档位和汇率信息用于页面展示
2. 同时**钱包 → 财务**获取当前币种可用的充值方式列表（银行/电子钱包/USDT）
3. 用户选择金额和方式后点击"去充值"，**钱包（本地）** 执行重复转账拦截检测（同UserID+同币种+同金额的待支付订单不超过3笔）
4. 检测通过后，**钱包 → 财务**请求匹配最优支付通道
5. 匹配成功后，**钱包（本地）** 创建充值订单（状态=待支付），并向三方发起支付请求
6. 用户在三方页面完成支付，三方通道回调支付结果
7. **财务 → 钱包（CreditBalance）** 通知钱包入账：充值金额进中心钱包，奖金（如有）进奖励钱包
8. 钱包执行入账（**幂等校验**：基于订单号去重，仅"待入账"状态才执行），生成流水记录

**这条链路中的依赖流转**：
- 步骤1~2 是**钱包主动调用**币种配置和财务（请求方向）
- 步骤7 是**财务主动调用**钱包（回调方向）
- 请求方向和回调方向相反，形成**双向依赖**

#### 链路二：提现链路

```
用户操作 → 钱包 → KYC → 钱包 → 币种配置 → 钱包 → 财务 → 三方 → 财务 → 钱包
```

文字描述：

1. 用户点击提现，**钱包 → ser-kyc** 查询用户KYC认证状态
2. 未认证：弹窗提供"去认证"和"继续提现（仅USDT）"两个入口；已认证：直接进入提现选择页
3. 用户选择提现方式并输入金额，**钱包 → 币种配置**获取提现限额配置（单笔/单日），校验金额是否合规
4. BSB提现时，**钱包 → 币种配置**获取出款汇率，展示"提现X BSB ≈ Y 法币/USDT"
5. 用户确认提现，**钱包（本地）** 冻结对应余额（可用余额减少，冻结金额增加）
6. **钱包 → 财务**提交提现订单，进入三级审核流程
7. 财务侧执行：风控审核 → 财务审核 → 出款审核
   - 任一环节驳回：**财务 → 钱包（UnfreezeBalance）** 解冻余额，订单标记失败
   - 全部通过：财务向三方代付通道发起出款
8. 三方出款结果回调财务
   - 出款成功：**财务 → 钱包（DebitBalance）** 将冻结金额正式扣减，订单标记完成
   - 出款失败：**财务 → 钱包（UnfreezeBalance）** 解冻余额，退回给用户

**这条链路中的依赖流转**：
- 步骤1 钱包 → KYC（单向，仅查询）
- 步骤3~4 钱包 → 币种配置（单向，仅读取）
- 步骤6 钱包 → 财务（提交提现单）
- 步骤7~8 财务 → 钱包（解冻/扣款回调）
- 同样存在**钱包↔财务双向依赖**

#### 链路三：人工修正链路

```
B端操作 → 财务（创建+审核）→ 钱包（执行余额变更）
```

文字描述：

1. B端操作员在财务管理后台搜索用户ID，选择币种
2. 人工加款：对中心钱包/奖励钱包/主播钱包/代理钱包分别设置加款金额，选择原因（活动奖金补发/客诉赔偿/系统错误修正/运营奖励/测试充值/其他）
3. 人工减款：类似操作，选择原因（异常资金追回/风控处罚扣款/财务冲正/费用调整/其他），减款超余额时需确认提示
4. 充值补单：关联原充值订单，填写补单金额和赠送金额
5. 操作人提交后，审核人审核（**操作人 ≠ 审核人**）
6. 审核通过后，**财务 → 钱包（ManualAdjust）** 调用钱包执行实际的余额加减操作，钱包生成对应流水记录

**这条链路的依赖方向**：纯单向，**财务 → 钱包**。钱包只需要暴露一个ManualAdjust RPC接口，接受"用户ID+钱包类型+币种+操作类型(加/减)+金额+关联单号"即可。

### 9.8 依赖性质分类

将所有模块间依赖按性质归类：

**一、前置依赖（A必须先就绪，B才能工作）**

- 币种配置 → 钱包：**币种配置是钱包的前置依赖**。没有币种数据和汇率数据，钱包的余额展示、充值、兑换、提现都无法运转。币种配置的B端CRUD和RPC接口必须先于钱包业务开发完成。
- 币种配置 → 财务：**币种配置是财务的前置依赖**。通道配置、支付配置中的"币种"字段依赖币种配置数据，统计报表的汇率换算也依赖。但财务对币种配置的依赖程度弱于钱包。
- 财务支付配置 → 钱包充值展示：**财务的支付配置是钱包充值方式展示的前置依赖**。钱包C端展示哪些充值方式、什么排序、什么icon，数据源头在财务的支付配置中。如果财务还没配好支付方式，钱包充值页面无法展示任何内容。
- 财务通道配置 → 钱包充值下单：**财务的通道配置是钱包充值下单的前置依赖**。用户点击充值后需要匹配通道，通道数据在财务。没有配好通道，充值下单会返回"通道异常"。

**二、后置依赖（A先执行，B随后被触发）**

- 钱包充值订单创建 → 财务充值回调 → 钱包入账：充值完成后，财务接收三方回调，**然后**调用钱包入账。钱包入账是充值流程的**后置环节**。
- 钱包提现冻结 → 财务三级审核 → 钱包解冻/扣款：提现申请后余额先冻结，**然后**进入财务审核，审核结果**再**触发钱包的解冻或扣款。钱包的解冻/扣款是提现流程的**后置环节**。
- 财务人工修正审核通过 → 钱包余额调整：审核通过**然后**才调用钱包。钱包余额调整是人工修正的**后置环节**。

**三、互相独立（无直接依赖，可并行）**

- 钱包兑换模块 与 钱包充值模块：兑换是法币余额→BSB余额的本地操作（仅依赖汇率），充值是外部资金→余额的外部交互操作，二者互不依赖。
- 钱包记录模块 与 所有其他模块：记录模块是纯粹的本地DB查询（查充值记录、提现记录、兑换记录、奖励记录），不调用任何外部服务。
- 钱包提现账户管理 与 钱包充值模块：提现账户绑定（选银行+填账号）和充值流程没有任何关联。
- 财务数据统计 与 钱包C端功能：统计模块是B端独立的报表查询，不影响C端用户操作。
- 币种配置B端CRUD 与 财务B端CRUD：两个模块的B端管理操作互相独立，各自管各自的配置数据。

**四、耦合关系（双向依赖，需要接口契约协调）**

只有一对耦合关系：**钱包 ↔ 财务**。

耦合表现：
- 正向：钱包 → 财务（获取充值方式 + 匹配通道）— 充值发起阶段
- 反向：财务 → 钱包（入账 + 冻结 + 解冻 + 扣款 + 调整）— 充值完成/提现审核/人工修正阶段
- 同一条业务链路中出现"去"和"回"两个方向的调用，这就是双向依赖

耦合风险：
- 接口契约必须双方先行约定（IDL先定义好），否则联调时反复修改
- 回调接口（CreditBalance等5个）必须幂等设计，因为网络抖动可能导致重复调用
- 冻结→解冻/扣款之间如果出现状态丢失（如财务审核完成但RPC调用钱包失败），会导致资金悬挂

### 9.9 依赖关系一句话总结

用最简洁的箭头描述三大模块之间的全部依赖方向：

```
币种配置  ◄─── 钱包 ───► 财务
                         │
                         │  （反向回调）
                         ▼
                        钱包

即：
  币种配置 ←── 钱包（单向，钱包读取币种/汇率/限额）
  币种配置 ←── 财务（单向，财务读取币种/汇率用于统计）
  钱包 ──► 财务（正向，钱包请求充值方式和通道匹配）
  财务 ──► 钱包（反向，财务回调入账/冻结/解冻/扣款/调整）
```

**核心结论**：
- 币种配置是纯被依赖方，零对外依赖，开发优先级最高
- 钱包和财务存在双向依赖，是系统中唯一的耦合点，需优先约定RPC接口契约
- 其余依赖均为单向且松散，不构成开发阻塞
