# ser-wallet 模块间依赖关系分析

> 目的：梳理各模块、各接口之间的前后依赖、调用方向、耦合程度
> 用于：指导架构设计、开发顺序、接口优先级排定

---

## 一、初始化依赖层级（什么必须先存在）

系统不是一上来就能用，有严格的配置前置关系：

```
Layer 0 (地基)           Layer 1 (通道)          Layer 2 (方式)          Layer 3 (业务可用)
━━━━━━━━━━━━━━          ━━━━━━━━━━━━          ━━━━━━━━━━━━          ━━━━━━━━━━━━━━━━

┌──────────────┐        ┌──────────────┐       ┌──────────────┐       ┌────────────────┐
│  币种配置     │───────→│  代收通道     │──────→│  充值方式     │──────→│  用户可充值     │
│  (创建币种)   │        │  (创建通道)   │       │  (关联通道)   │       │                │
└──────────────┘        └──────────────┘       └──────────────┘       └────────────────┘
       │                                              │
       │                ┌──────────────┐       ┌──────────────┐       ┌────────────────┐
       └───────────────→│  代付通道     │──────→│  提现方式     │──────→│  用户可提现     │
                        │  (创建通道)   │       │  (关联通道)   │       │                │
                        └──────────────┘       └──────────────┘       └────────────────┘
       │                       │
       │                ┌──────────────┐                              ┌────────────────┐
       │                │  轮询规则     │─────────────────────────────→│  充值通道分配   │
       │                │  (配权重)     │                              │                │
       │                └──────────────┘                              └────────────────┘
       │
       │                                                              ┌────────────────┐
       └─────────────────────────────────────────────────────────────→│  用户可兑换     │
                        (汇率配置就绪)                                 │  (法币→BSB)    │
                                                                      └────────────────┘
```

### 层级说明

| 层级 | 内容 | 前置条件 | 负责人 |
|------|------|----------|--------|
| **L0 地基** | 创建币种、配置基准币种(USDT)、配置汇率规则 | 无 | 自己 |
| **L1 通道** | 创建代收/代付通道、配置轮询规则 | 币种必须先存在 | 他人(财务) |
| **L2 方式** | 创建充值方式(关联代收通道)、提现方式(关联代付通道) | 通道必须先存在 | 他人(财务) |
| **L3 业务** | 用户可以充值、提现、兑换 | L0+L1+L2全部就绪 | 自己(钱包) |

**结论：币种配置是整个系统的最底层依赖，必须最先开发完成。**

## 二、运行时调用链路（用户操作时谁调谁）

### 2.1 充值链路

```
用户                   钱包(ser-wallet)              财务管理模块                三方通道
 │                          │                           │                        │
 │  1.进入充值页             │                           │                        │
 │ ────────────────────────→│                           │                        │
 │                          │  2.GetDepositMethods()    │                        │
 │                          │ ─────────────────────────→│                        │
 │                          │  返回可用充值方式列表       │                        │
 │                          │ ←─────────────────────────│                        │
 │  3.展示充值方式列表        │                           │                        │
 │ ←────────────────────────│                           │                        │
 │                          │                           │                        │
 │  4.选方式+输入金额+选奖金  │                           │                        │
 │ ────────────────────────→│                           │                        │
 │                          │  5.[内部]重复转账拦截检查   │                        │
 │                          │  6.MatchDepositChannel()  │                        │
 │                          │ ─────────────────────────→│                        │
 │                          │  返回匹配的通道             │                        │
 │                          │ ←─────────────────────────│                        │
 │                          │  7.CreateChannelOrder()   │                        │
 │                          │ ─────────────────────────→│                        │
 │                          │                           │  8.创建通道订单          │
 │                          │                           │ ──────────────────────→│
 │                          │  返回支付链接/地址          │                        │
 │                          │ ←─────────────────────────│                        │
 │  9.跳转支付页面           │                           │                        │
 │ ←────────────────────────│                           │                        │
 │                          │                           │                        │
 │  10.用户完成支付           │                           │                        │
 │ ──────────────────────────────────────────────────────────────────────────────→│
 │                          │                           │  11.支付结果回调         │
 │                          │                           │ ←──────────────────────│
 │                          │  12.Credit() 上分          │                        │
 │                          │ ←─────────────────────────│                        │
 │                          │  [内部]余额+奖金入账       │                        │
 │  13.充值成功通知           │                           │                        │
 │ ←────────────────────────│                           │                        │
```

**调用方向总结：**
- 步骤2,6,7：钱包 → 财务（钱包主动调财务3次）
- 步骤12：财务 → 钱包（财务回调钱包1次）
- 步骤5：钱包内部逻辑
- 步骤8,11：财务 ↔ 三方通道（与钱包无关）

### 2.2 提现链路

```
用户                   钱包(ser-wallet)     KYC服务      财务管理模块           三方通道
 │                          │                │              │                    │
 │  1.进入提现页             │                │              │                    │
 │ ────────────────────────→│                │              │                    │
 │                          │ 2.GetKycStatus()│              │                    │
 │                          │ ──────────────→│              │                    │
 │                          │ 返回KYC状态     │              │                    │
 │                          │ ←──────────────│              │                    │
 │                          │                │              │                    │
 │                          │ 3.GetWithdrawMethods()        │                    │
 │                          │ ─────────────────────────────→│                    │
 │                          │ 返回可用提现方式               │                    │
 │                          │ ←─────────────────────────────│                    │
 │  4.展示提现页面           │                │              │                    │
 │ ←────────────────────────│                │              │                    │
 │                          │                │              │                    │
 │  5.选方式+金额+收款信息   │                │              │                    │
 │ ────────────────────────→│                │              │                    │
 │                          │ 6.[内部]校验:余额/限额/资金来源规则                  │
 │                          │ 7.[内部]Freeze冻结金额        │                    │
 │                          │ 8.SubmitWithdrawRequest()     │                    │
 │                          │ ─────────────────────────────→│                    │
 │  9.提现已提交,等待审核    │                │              │                    │
 │ ←────────────────────────│                │              │                    │
 │                          │                │              │                    │
 │                    ┌─────────────── 异步审核流程 ──────────────────┐            │
 │                    │     财务管理内部三级审核                       │            │
 │                    │     风控审核 → 财务审核 → 出款审核             │            │
 │                    └──────────────────────────────────────────────┘            │
 │                          │                │              │                    │
 │              ┌───── 审核通过 ─────┐         │              │  10.出款请求       │
 │              │                   │         │              │ ──────────────────→│
 │              │                   │         │              │  11.出款结果       │
 │              │                   │         │              │ ←──────────────────│
 │              │                   │         │              │                    │
 │              │  12.Debit() 扣款   │         │              │                    │
 │              │  钱包 ←──── 财务    │         │              │                    │
 │              └───────────────────┘         │              │                    │
 │                          │                │              │                    │
 │              ┌───── 审核拒绝 ─────┐         │              │                    │
 │              │  13.Unfreeze()解冻 │         │              │                    │
 │              │  钱包 ←──── 财务    │         │              │                    │
 │              └───────────────────┘         │              │                    │
```

**调用方向总结：**
- 步骤2：钱包 → KYC服务
- 步骤3,8：钱包 → 财务（钱包主动调2次）
- 步骤6,7：钱包内部逻辑
- 步骤12或13：财务 → 钱包（审核完成后回调1次）
- 审核流程：财务内部（与钱包无关，异步进行）

### 2.3 兑换链路

```
用户                   钱包(ser-wallet)           币种配置(同服务内部)
 │                          │                          │
 │  1.进入兑换页             │                          │
 │ ────────────────────────→│                          │
 │                          │  2.GetExchangeRate()     │
 │                          │  (内部Service调用,非RPC)  │
 │                          │ ────────────────────────→│
 │                          │  返回平台汇率+预估BSB     │
 │                          │ ←────────────────────────│
 │  3.展示汇率和预估金额     │                          │
 │ ←────────────────────────│                          │
 │                          │                          │
 │  4.确认兑换               │                          │
 │ ────────────────────────→│                          │
 │                          │  5.[内部]扣法币中心钱包    │
 │                          │  6.[内部]加BSB中心钱包     │
 │                          │  7.[内部]奖金加BSB奖励钱包 │
 │  8.兑换成功               │                          │
 │ ←────────────────────────│                          │
```

**调用方向总结：**
- 步骤2：钱包模块内部调用币种配置的Service方法（同一个ser-wallet进程）
- 步骤5,6,7：钱包内部余额操作
- **不依赖任何外部模块，完全自闭环**

### 2.4 人工修正链路（财务后台操作）

```
管理员A                 财务管理模块                  钱包(ser-wallet)
 │                          │                           │
 │  1.发起加款/减款/补单     │                           │
 │ ────────────────────────→│                           │
 │                          │  2.QueryBalance()         │
 │                          │ ─────────────────────────→│
 │                          │  返回用户当前余额           │
 │                          │ ←─────────────────────────│
 │                          │  3.创建待审核订单           │
 │  4.提交成功,等待审核      │                           │
 │ ←────────────────────────│                           │
 │                          │                           │
管理员B(≠管理员A)           │                           │
 │  5.审核通过               │                           │
 │ ────────────────────────→│                           │
 │                          │  6.Credit()/Debit()       │
 │                          │ ─────────────────────────→│
 │                          │  返回操作结果               │
 │                          │ ←─────────────────────────│
 │  7.审核完成               │                           │
 │ ←────────────────────────│                           │
```

**调用方向：** 始终是 财务 → 钱包（财务主动调，钱包被动响应）

## 三、接口依赖矩阵

### 3.1 前置接口（必须先就绪，后续才能工作）

```
优先级 P0 ──────────────────────────────────────────────────────────
│
│  ┌─────────────────────────────────────────────────────┐
│  │  币种配置模块 (B端)                                   │
│  │                                                     │
│  │  CreateCurrency ──→ 所有涉及币种的功能的前提          │
│  │  UpdateRateConfig ──→ 汇率相关功能的前提              │
│  │  定时汇率同步任务 ──→ 实时汇率数据的前提              │
│  │                                                     │
│  │  RPC: GetAvailableCurrencies ──→ 财务配通道/方式需要  │
│  │  RPC: GetExchangeRate ──→ 兑换/统计需要               │
│  │  RPC: GetCurrencyInfo ──→ 各处展示需要                │
│  └─────────────────────────────────────────────────────┘
│
│  这些接口是整个系统的地基。没有币种数据，
│  通道配不了，方式建不了，用户什么都做不了。
│
优先级 P1 ──────────────────────────────────────────────────────────
│
│  ┌─────────────────────────────────────────────────────┐
│  │  钱包核心能力 (RPC)                                   │
│  │                                                     │
│  │  QueryBalance ──→ 人工修正、B端查询的前提             │
│  │  Credit ──→ 充值成功、补单、人工加款的前提             │
│  │  Debit ──→ 提现完成、人工减款的前提                   │
│  │  Freeze / Unfreeze ──→ 提现流程的前提                │
│  └─────────────────────────────────────────────────────┘
│
│  这些是钱包的核心RPC能力。财务模块的充值回调、
│  提现审核、人工修正全都依赖这几个接口。
│  没有这些，财务模块的后半段流程全部断裂。
│
优先级 P2 ──────────────────────────────────────────────────────────
│
│  ┌─────────────────────────────────────────────────────┐
│  │  财务模块提供的前置能力 (他人负责,但钱包依赖)          │
│  │                                                     │
│  │  GetDepositMethods ──→ 钱包C端充值页的前提            │
│  │  GetWithdrawMethods ──→ 钱包C端提现页的前提           │
│  │  MatchDepositChannel ──→ 创建充值订单的前提           │
│  │  CreateChannelOrder ──→ 拉起支付的前提                │
│  │  SubmitWithdrawRequest ──→ 提现进入审核的前提         │
│  └─────────────────────────────────────────────────────┘
│
│  这些接口由财务模块提供。钱包的充值/提现功能
│  必须等这些接口就绪才能完整运行。
│
优先级 P3 ──────────────────────────────────────────────────────────
```

### 3.2 后置接口（依赖前面的数据产生后才有意义）

| 接口 | 前置条件 | 说明 |
|------|----------|------|
| GetDepositRecords | 有充值订单数据 | 没人充值过就是空列表 |
| GetWithdrawRecords | 有提现订单数据 | 同上 |
| GetExchangeRecords | 有兑换订单数据 | 同上 |
| GetRewardRecords | 有奖励发放数据 | 同上 |
| GetRecordDetail | 有订单数据 | 同上 |
| GetRateChangeLog | 有汇率变更记录 | 平台汇率更新过才有日志 |
| 财务-充值记录列表 | 有充值订单 | 纯查询，数据来自充值流程 |
| 财务-提现记录列表 | 有提现订单 | 纯查询，数据来自提现流程 |
| 财务-数据统计 | 有交易数据 | 聚合统计，无数据无意义 |
| AdminGetUserWallet | 用户有钱包 | 管理员查看用户资产 |

**这些接口本身无前置依赖，随时可以开发，但有意义的使用取决于业务数据的积累。**

### 3.3 独立接口（不依赖其他模块，可以独立开发测试）

```
┌─────────────────────────────────────────────────────────────┐
│  完全独立（与外部模块零依赖）                                  │
│                                                             │
│  钱包模块:                                                   │
│    · GetWalletInfo ──── 只读自己的库，查余额                  │

│    · GetExchangeRate ── 内部调币种配置Service                  │
│    · CreateExchangeOrder ── 内部余额操作，不涉及外部           │
│    · 四个Records接口 ── 只读自己的订单表                      │
│    · GetRecordDetail ── 只读自己的订单表                      │
│    · CheckDuplicateDeposit ── 只查自己的订单表                │
│                                                             │
│  币种配置模块:                                                │
│    · GetCurrencyConfigList ── 只读自己的配置表                │
│    · CreateCurrency ── 只写自己的配置表                       │
│    · UpdateCurrency ── 只写自己的配置表                       │
│    · ToggleCurrencyStatus ── 只写自己的配置表                 │
│    · GetRateConfig ── 只读自己的配置表                        │
│    · UpdateRateConfig ── 只写自己的配置表                     │
│    · GetRateChangeLog ── 只读自己的日志表                     │
│    · GetCurrencyDetail ── 只读自己的配置表                    │
│    · 三个RPC接口 ── 只读自己的数据对外提供                    │
│    · 定时汇率同步 ── 调第三方API+写自己的表                   │
│                                                             │
│  结论: 币种配置模块100%独立，钱包的兑换+余额+记录也独立。     │
│  这些可以最先开发，不需要等任何人。                            │
└─────────────────────────────────────────────────────────────┘
```

### 3.4 耦合接口（必须与其他模块联调才能完整工作）

```
┌─────────────────────────────────────────────────────────────┐
│  强耦合（双向依赖，必须联调）                                  │
│                                                             │
│  充值流程:                                                   │
│    钱包.CreateDepositOrder                                   │
│      ├──→ 财务.GetDepositMethods (获取方式)                  │
│      ├──→ 财务.MatchDepositChannel (匹配通道)                │
│      ├──→ 财务.CreateChannelOrder (创建通道订单)              │
│      └──← 财务.DepositCallback → 钱包.Credit (上分)          │
│                                                             │
│  提现流程:                                                   │
│    钱包.CreateWithdrawOrder                                  │
│      ├──→ 外部.GetKycStatus (KYC校验)                       │
│      ├──→ 财务.GetWithdrawMethods (获取方式)                 │
│      ├──→ 财务.SubmitWithdrawRequest (提交审核)              │
│      └──← 财务.审核完成 → 钱包.Debit/Unfreeze (扣款/解冻)    │
│                                                             │
│  人工修正:                                                   │
│    财务.人工加款/减款/补单                                     │
│      ├──→ 钱包.QueryBalance (查余额)                         │
│      └──→ 钱包.Credit/Debit (操作余额)                       │
│                                                             │
│  耦合度: 充值 > 提现 > 人工修正                               │
│  充值涉及4次跨模块调用，提现3次+异步回调，人工修正2次           │
└─────────────────────────────────────────────────────────────┘
```

## 四、接口耦合度分级

将所有接口按耦合程度分为四级：

### Level 0: 完全独立（可立即开发，不等任何人）

| 模块 | 接口 | 依赖 |
|------|------|------|
| 币种配置 | 全部8个B端接口 | 无外部依赖 |
| 币种配置 | 全部3个RPC接口 | 无外部依赖 |
| 币种配置 | 2个定时任务 | 仅依赖第三方汇率API |
| 钱包 | GetWalletInfo | 读自己的表 |
| 钱包 | SwitchCurrency | 写自己的表 |
| 钱包 | GetExchangeRate | 内部调币种配置Service |
| 钱包 | CreateExchangeOrder | 内部余额操作 |
| 钱包 | 4个Records接口 | 读自己的表 |
| 钱包 | GetRecordDetail | 读自己的表 |
| 钱包 | CheckDuplicateDeposit | 读自己的表 |

**小计：~20个接口，占总量近一半，可以完全并行开发**

### Level 1: 单向依赖（依赖他人提供数据，但自身逻辑完整）

| 模块 | 接口 | 依赖谁 | 依赖什么 |
|------|------|--------|----------|
| 钱包 | GetDepositMethods | 财务 | 充值方式列表数据 |
| 钱包 | GetWithdrawMethods | 财务 | 提现方式列表数据 |
| 钱包 | GetDepositBonusList | 活动系统(待定) | 奖金活动数据 |
| 钱包 | CheckWithdrawEligibility | ser-kyc | KYC状态 |
| 钱包 | AdminGetUserWallet | ser-buser | 用户信息 |

**小计：~5个接口，逻辑可以先写好，联调时对接数据源即可**

### Level 2: 双向耦合（需要联调才能完整工作）

| 模块 | 接口 | 耦合对象 | 耦合方式 |
|------|------|----------|----------|
| 钱包 | CreateDepositOrder | 财务(3个RPC) | 钱包调财务匹配通道+创建订单 |
| 钱包 | DepositCallback处理 | 财务→钱包Credit | 财务回调钱包上分 |
| 钱包 | CreateWithdrawOrder | 财务(1个RPC)+KYC | 钱包调财务提交审核 |
| 钱包 | 提现审核结果处理 | 财务→钱包Debit/Unfreeze | 财务回调钱包扣款/解冻 |

**小计：~4个接口，是最复杂的联调点**

### Level 3: 被动暴露（自身逻辑简单，但被多方依赖）

| 模块 | 接口 | 被谁调用 | 重要性 |
|------|------|----------|--------|

## 六、风险点提示

| 风险 | 说明 | 影响 |
|------|------|------|
| 财务模块进度 | 充值/提现的C端流程强依赖财务的6个RPC接口 | 财务没做好，钱包充值/提现就跑不通 |

| 通道回调可靠性 | 充值成功依赖通道回调→财务→钱包Credit | 回调丢失=用户充了钱但没到账 |
| 汇率API可用性 | 实时汇率依赖3个第三方API | 全挂时用最近一次缓存的汇率 |
| 余额操作并发 | Credit/Debit/Freeze可能并发调用 | 必须保证原子性，防止超扣或重复加 |

## 七、三大模块依赖关系文字描述

### 7.1 币种配置模块的依赖关系

**币种配置不依赖任何模块，但被其他所有模块依赖。**

币种配置是整个钱包体系的数据地基。它管理着平台支持哪些币种、每个币种的汇率怎么算、浮动比例多少。没有这些基础数据，后面的一切都无从谈起。

具体来说：

- 币种配置 → 钱包模块：钱包模块的兑换功能需要调用币种配置的汇率数据来计算法币兑BSB的到账金额。钱包展示给用户的可用币种列表也来源于币种配置中已启用的币种。这两者在同一个 ser-wallet 服务内部，属于内部 Service 调用，不走 RPC。

- 币种配置 → 财务管理模块：财务模块在配置代收/代付通道时，每个通道必须绑定一个币种，这个币种必须是币种配置中已经创建并启用的。财务模块的支付配置（充值方式、提现方式）同样按币种维度管理。财务的数据统计在做基准币种换算时也需要调用币种配置的汇率接口。这些是跨服务 RPC 调用。

- 币种配置不需要主动调用任何其他模块的接口。它唯一的外部依赖是第三方汇率 API（用于定时拉取实时汇率），这不属于模块间依赖。

**一句话总结：币种配置只被依赖，不依赖别人。是最底层的基础设施。**

### 7.2 钱包模块的依赖关系

**钱包模块向上服务C端用户，向下依赖币种配置和财务管理，同时被财务管理反向调用。**

钱包模块是用户直接打交道的模块。用户在App上看到的余额、充值、提现、兑换、交易记录，都是钱包模块提供的。但钱包模块并不是一个独立闭环的系统，它的很多能力需要依赖其他模块才能完成。

#### 钱包依赖币种配置（内部依赖）

钱包和币种配置同属一个 ser-wallet 服务。钱包在以下场景依赖币种配置的数据：

- 用户查看可用币种列表时 → 钱包调币种配置获取已启用的币种
- 用户进行法币兑BSB时 → 钱包调币种配置获取平台汇率来计算到账金额
- 用户切换币种时 → 需要确认目标币种是否在启用状态

因为是同一个服务内部的函数调用，没有网络开销，耦合度低，可靠性高。

#### 钱包依赖财务管理（跨服务强依赖）

这是最重要的依赖关系。钱包的充值和提现功能，核心链路上必须调用财务管理模块的接口：

**充值方向：钱包 → 财务**

- 用户进入充值页时，钱包需要调财务的 GetDepositMethods 获取当前币种有哪些充值方式可用（银行转账、QRIS、USDT等）。这些充值方式是财务模块在后台配置的，每种方式关联一个代收通道。
- 用户提交充值时，钱包需要调财务的 MatchDepositChannel 来匹配一个实际处理这笔充值的通道（按权重轮询）。如果匹配不到可用通道，充值就会失败。
- 匹配到通道后，钱包需要调财务的 CreateChannelOrder，让财务模块通过这个通道创建一笔三方支付订单，拿到支付页面的链接或二维码地址返回给用户。

**充值方向：财务 → 钱包**

- 用户在三方支付页面完成支付后，支付通道会发回调通知给财务模块。财务确认支付成功后，需要调钱包的 Credit 接口给用户上分（把钱加到用户的中心钱包）。如果用户选了奖金活动，奖金部分需要加到奖励钱包。

**提现方向：钱包 → 财务**

- 用户进入提现页时，钱包需要调财务的 GetWithdrawMethods 获取可用提现方式。
- 用户提交提现后，钱包先在内部冻结对应金额（防止用户把要提的钱花掉），然后调财务的 SubmitWithdrawRequest 把这笔提现订单提交到审核流程。

**提现方向：财务 → 钱包**

- 财务模块内部完成三级审核（风控→财务→出款）后，如果全部通过并且通道出款成功，财务需要调钱包的 Debit 接口来正式扣除用户余额（把之前冻结的金额真正扣掉）。
- 如果审核被拒绝或者通道出款失败，财务需要调钱包的 Unfreeze 接口解冻金额，把钱还给用户。

#### 钱包依赖其他服务（弱依赖）

- 钱包 → KYC服务：提现时需要查询用户的实名认证状态。法币和电子钱包提现必须完成KYC，USDT提现可以不做KYC。如果KYC服务不可用，只影响提现资格校验，不影响其他功能。
- 钱包 → 用户服务(ser-buser)：查询用户基本信息，用于B端管理员查看用户钱包时展示用户信息。

#### 钱包被财务反向调用

除了上面提到的 Credit/Debit/Unfreeze（充值提现流程中的回调），财务模块在人工修正场景下也会主动调钱包：

- 充值补单（漏单修复）→ 财务调钱包的 Credit 给用户补上分
- 人工加款（奖金/补偿）→ 财务调钱包的 Credit 给用户加钱
- 人工减款（纠错/处罚）→ 财务先调钱包的 QueryBalance 确认余额足够，再调 Debit 扣钱

**一句话总结：钱包向下依赖币种配置（内部）和财务管理（RPC），同时被财务管理反向调用来操作余额。充值和提现是双向耦合的，兑换是独立的。**

### 7.3 财务管理模块的依赖关系

**财务管理模块依赖币种配置的基础数据，依赖钱包的余额操作能力，同时被钱包调用来获取支付配置和通道能力。**

#### 财务依赖币种配置

财务模块的通道配置、支付配置、数据统计都是按币种维度管理的：

- 创建代收/代付通道时，必须指定该通道处理哪个币种的交易，这个币种必须在币种配置中存在且启用。
- 创建充值/提现方式时同理，每种方式对应一个币种。
- 配置通道轮询规则时，不同币种有独立的轮询规则，同样依赖币种数据。
- 数据统计做跨币种汇总时，可能需要调币种配置的汇率接口来做基准币种（USDT）换算。

#### 财务依赖钱包

财务模块本身不管理用户余额，所有涉及用户钱包操作的场景都必须通过钱包的RPC接口来完成：

- 充值成功后给用户上分 → 调钱包 Credit
- 提现审核通过且出款成功 → 调钱包 Debit
- 提现审核拒绝 → 调钱包 Unfreeze
- 人工加款审核通过 → 调钱包 Credit
- 人工减款审核通过 → 先调钱包 QueryBalance 确认余额，再调钱包 Debit
- 充值补单审核通过 → 调钱包 Credit

如果钱包的这些RPC接口不可用，财务模块虽然能完成审核流程，但最终的资金操作会失败。

#### 财务被钱包调用

钱包在用户充值和提现流程中需要调用财务模块提供的以下能力：

- 获取充值方式列表（支付配置数据）
- 获取提现方式列表（支付配置数据）
- 轮询匹配代收通道（通道调度能力）
- 创建通道订单（通道交互能力）
- 提交提现到审核流程（审核流程入口）
- 查询通道订单状态（订单状态同步）

**一句话总结：财务管理依赖币种配置的基础数据和钱包的余额操作能力，同时向钱包提供支付配置和通道调度能力。它是充值/提现链路的中间处理层。**

## 八、三模块关系的本质

把三个模块的关系抽象成一句话：

- **币种配置**是"规则制定者"——定义有哪些币种、汇率怎么算
- **钱包**是"用户代言人"——面向用户提供充值/提现/兑换/余额管理
- **财务管理**是"执行管家"——管理支付通道、处理资金流转、审核把关

它们的协作模式是：

```
币种配置(规则) ──提供基础数据──→ 钱包(用户侧) ←──操作余额── 财务管理(执行侧)
      │                            │                        ↑
      │                            │──获取支付配置+通道能力──→│
      │                                                     │
      └──────────提供币种和汇率数据─────────────────────────→│
```

**数据流方向总结：**

- 配置数据：币种配置 → 钱包 + 财务（单向，从底层往上流）
- 充值流程：钱包 → 财务 → 通道 → 财务 → 钱包（双向，完成一个完整闭环）
- 提现流程：钱包 → 财务 → 审核 → 通道 → 财务 → 钱包（双向，带审核的闭环）
- 兑换流程：钱包内部完成（不跨模块）
- 人工修正：财务 → 钱包（单向，管理员操作用户余额）
- 记录查询：各模块各自查自己的数据（不跨模块）

## 九、哪些功能之间互相独立

以下功能之间没有运行时依赖，可以完全并行开发和独立测试：

**独立组A：币种配置全模块**
- 币种CRUD、汇率配置、汇率日志、定时汇率同步
- 与其他功能零耦合，随时可以独立开发

**独立组B：钱包余额+币种切换**
- GetWalletInfo、SwitchCurrency
- 只操作自己的用户钱包表，不依赖外部

**独立组C：兑换功能**
- GetExchangeRate、CreateExchangeOrder
- 只依赖币种配置的汇率（同服务内部调用），不需要财务模块

**独立组D：交易记录查询**
- 四个Records接口 + GetRecordDetail
- 只读自己的订单表，与充值/提现的创建流程解耦

**独立组E：重复转账拦截**
- CheckDuplicateDeposit
- 只查自己的充值订单表

**独立组F：钱包核心RPC**
- Credit、Debit、Freeze、Unfreeze、QueryBalance
- 逻辑上是独立的余额操作原子能力，不主动依赖外部
- 被财务模块调用，但自身不需要调别人

**以上6组之间互相独立，总共约30个接口，占全部42个接口的70%以上。**

**互相耦合的功能（必须联调）：**

- 充值创建流程 ↔ 财务的支付配置+通道匹配+通道订单
- 充值回调处理 ↔ 财务的回调通知+钱包的Credit
- 提现创建流程 ↔ KYC校验+财务的支付配置+审核提交
- 提现结果处理 ↔ 财务的审核结果+钱包的Debit/Unfreeze

**这些耦合功能约10个接口，占30%，集中在充值和提现的完整链路上。**
