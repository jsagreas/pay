## 2. 模块间依赖方向图

```
                ┌─────────────┐
                │   币种配置    │  ← 最底层基建，被所有资金模块依赖
                │  (ser-wallet │
                │   内部模块)   │
                └──────┬───────┘
                       │
          ┌────────────┼────────────┐
          │ 提供币种/汇率  │            │ 提供币种信息
          ▼            │            ▼
   ┌──────────────┐    │    ┌──────────────────┐
   │  钱包模块     │    │    │    财务管理模块     │
   │ (ser-wallet  │◄───┼───►│   (ser-finance)  │
   │  核心模块)    │    │    │                  │
   └──────┬───────┘    │    └────────┬─────────┘
          │            │             │
          │            │             │
   ┌──────┴────────────┴─────────────┴──────┐
   │          外部调用方                      │
   │  游戏/投注 · 直播/礼物 · 活动/任务       │
   │  代理模块 · 场馆模块                     │
   └────────────────────────────────────────┘

箭头说明：
  A ──→ B  表示 A 提供能力给 B（A被B调用）
  A ◄──► B 表示双向依赖
```

## 3. 依赖层级总结

| 层级 | 模块 | 角色 | 依赖说明 |
|------|------|------|---------|
| L0（地基） | 币种配置 | 被依赖方 | 不依赖钱包和财务，被所有资金模块依赖 |
| L1（核心） | 钱包模块 | 双向依赖 | 依赖L0币种配置 + L1财务模块，同时被财务和外部服务调用 |
| L1（核心） | 财务管理 | 双向依赖 | 依赖L0币种配置 + L1钱包模块，同时被钱包调用 |
| L2（消费方） | 游戏/直播/活动等 | 调用方 | 单向依赖钱包的扣款/入账能力 |

# 第三部分：双向耦合分析（核心重点）

## 1. 钱包 ↔ 财务：双向强耦合

这是整个系统中**最核心的耦合点**，两个模块互为调用方和被调用方。

### 1.1 耦合接口对照表

```
┌──────────────────────────────────────────────────────────────────┐
│                    钱包 ◄══════► 财务 双向依赖                     │
│                                                                   │
│  ┌─── 钱包调用财务（5个核心）──────────────────────────────────┐  │
│  │                                                             │  │
│  │  充值链路:                                                  │  │
│  │    F3 获取充值方式 ──调用──→ 财务 R1 GetDepositMethods      │  │
│  │    F4 发起充值    ──调用──→ 财务 R3 CreateDepositOrder      │  │
│  │    F5 查询状态    ──调用──→ 财务 R5 GetOrderStatus          │  │
│  │                                                             │  │
│  │  提现链路:                                                  │  │
│  │    F8 获取提现方式 ──调用──→ 财务 R2 GetWithdrawMethods     │  │
│  │    F11 发起提现   ──调用──→ 财务 R4 CreateWithdrawOrder     │  │
│  └─────────────────────────────────────────────────────────────┘  │
│                                                                   │
│  ┌─── 财务调用钱包（7个核心）──────────────────────────────────┐  │
│  │                                                             │  │
│  │  充值回调:                                                  │  │
│  │    通道回调成功 → 财务 ──调用──→ 钱包 R1 WalletCredit       │  │
│  │                                                             │  │
│  │  提现流程:                                                  │  │
│  │    用户提交提现 → 财务 ──调用──→ 钱包 R3 WalletFreeze       │  │
│  │    审核驳回     → 财务 ──调用──→ 钱包 R4 WalletUnfreeze     │  │
│  │    出款成功     → 财务 ──调用──→ 钱包 R2 WalletDebit        │  │
│  │                                                             │  │
│  │  人工修正:                                                  │  │
│  │    补单/加款审核通过 → 财务 ──调用──→ 钱包 R5 ManualCredit   │  │
│  │    减款审核通过     → 财务 ──调用──→ 钱包 R6 ManualDebit     │  │
│  │    查看用户信息     → 财务 ──调用──→ 钱包 R8 GetWalletInfo   │  │
│  └─────────────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────────────┘
```

### 1.2 耦合带来的开发约束

| 约束 | 具体说明 |
|------|---------|
| **IDL契约必须先行对齐** | 12个双向RPC接口的请求/响应结构必须双方协商一致后再各自开发 |
| **不能完全串行开发** | 钱包不能等财务开发完再开始，反之亦然 |
| **建议mock并行** | 双方先定义IDL → 各自mock对方接口 → 并行开发 → 联调对接 |
| **错误处理约定** | 财务调用钱包入账失败时的重试/补偿策略需要预先约定 |

### 1.3 耦合的处理建议（开发顺序）

```
阶段1: 双方对齐IDL定义（接口契约）
  │
  ├─ 钱包侧：定义 WalletCredit/Debit/Freeze/Unfreeze/ManualCredit/ManualDebit/GetWalletInfo 的 thrift IDL
  ├─ 财务侧：定义 GetDepositMethods/GetWithdrawMethods/CreateDepositOrder/CreateWithdrawOrder/GetOrderStatus 的 thrift IDL
  │
阶段2: 各自实现核心（mock对方接口）
  │
  ├─ 钱包侧：先实现 RPC暴露接口（R1-R8）→ 再实现 C端接口（F1-F21，调用财务的部分先mock）
  ├─ 财务侧：先实现 RPC暴露接口（R1-R5）→ 再实现 B端接口（B1-B46，调用钱包的部分先mock）
  │
阶段3: 联调对接（去掉mock，接入真实RPC）
  │
  └─ 充值全链路 → 提现全链路 → 人工修正链路
```

## 2. 币种配置 → 钱包/财务：单向被依赖

```
┌──────────────────────────────────────────────────────┐
│          币种配置（单向输出，不依赖钱包和财务）          │
│                                                       │
│  R1 GetCurrencyInfo ────────→ 钱包（币种详情）         │
│                     ────────→ 财务（通道关联币种）      │
│                     ────────→ 游戏（币种转换）          │
│                                                       │
│  R2 GetEnabledCurrencyList ─→ 钱包C端（币种切换）      │
│                                                       │
│  R3 GetExchangeRate ────────→ 钱包（充值/提现/兑换汇率）│
│                     ────────→ 财务（统计换算）          │
│                     ────────→ 游戏（场馆汇率）          │
│                                                       │
│  R4 GetBaseCurrency ────────→ 财务（报表基准币种）      │
│                     ────────→ 统计（汇总换算基准）      │
└──────────────────────────────────────────────────────┘

特点：
  · 纯粹的"被依赖方"，不依赖钱包和财务的任何接口
  · 仅依赖外部三方汇率API（HTTP调用）和基础服务（ser-blog审计、ser-s3图标上传）
  · 因此可以最先独立开发完成
```

## 3. 外部服务 → 钱包：单向调用

```
┌──────────────────────────────────────────────────────────┐
│            外部服务对钱包的单向调用（钱包被动接收）          │
│                                                           │

│    → R10 BetDebit   (投注扣款：中心优先→奖励补充)          │
│    → R11 BetCredit  (投注返奖：按扣款比例分配)             │
│    → R12 TransferToVenue   (进入场馆转入，预留)            │
│    → R13 TransferFromVenue (退出场馆转出，预留)            │
│    → R9  CheckBalance (投注前余额校验)                     │
│                                                           │
```

```
  ║ 对齐  ║          ║              ║            ║      ║

关键并行点:
  · 你做币种配置时，他做通道配置 → 互不阻塞

  · 你做C端功能时，mock他的接口 → 互不阻塞
  · 最后联调阶段才需要双方接口都就绪
```

# 第八部分：风险点与注意事项

## 1. 数据一致性风险

```
风险场景: 充值回调
──────────────────
财务收到通道回调 → 调用钱包WalletCredit → 网络超时/失败
    │
    ├─ 如果财务已标记订单成功但钱包未入账 → 用户看到订单成功但余额没变
    ├─ 如果财务重试调用 → 钱包必须幂等处理，防止重复入账
    │
    └─ 约定：订单号作为幂等键，钱包侧必须做幂等校验

风险场景: 提现冻结
──────────────────
钱包调用财务CreateWithdrawOrder → 财务调用钱包WalletFreeze → 两步异常
    │
    ├─ 如果钱包发起成功但冻结失败 → 订单创建了但余额未冻结，用户可能超额提现
    ├─ 谁来保证原子性？
    │
    └─ 约定：提现下单应该是"钱包先冻结 → 再调财务创建订单"还是反过来？
            需要和财务开发者明确这个顺序
```

## 2. 循环依赖风险

```
当前设计中不存在循环调用:
  · 币种配置 → 不调用钱包/财务 ✅
  · 钱包 → 调用币种配置（本模块内部）✅
  · 钱包 ↔ 财务 → 双向调用但不存在A调B调A的循环链路 ✅
      钱包调用财务的场景（充值/提现下单）和财务调用钱包的场景（回调入账/审核后操作）
      是不同时间点的不同业务动作，不会形成同步循环调用

但需警惕:
  · 提现链路中"谁先调谁"的顺序必须明确（见上方风险场景）
  · 避免在同一个RPC调用链中出现 A→B→A 的情况
```

# 第九部分：三模块依赖关系文字详述

> 以下内容以产品需求文档和产品原型为信息源，从业务视角逐条梳理三模块间"谁依赖谁、为什么依赖、在哪个场景依赖"的完整关系。
> 箭头约定：A → B 表示 A 依赖 B（A 需要调用 B 提供的能力才能完成自己的职责）

---

## 一、币种配置模块的依赖关系

### 1.1 币种配置不依赖钱包和财务

币种配置是整个资金体系的**最底层基础设施**。从需求文档来看，它的职责是：定义平台支持哪些币种、每种币种的汇率如何计算、后台管理员如何配置这些参数。

这些职责**不需要调用钱包的任何能力**，也**不需要调用财务的任何能力**。币种配置的数据来源是：管理员在后台手动录入币种信息（币种配置需求文档3-3），以及定时从三方汇率API拉取实时汇率（币种配置需求文档4-2）。

因此，币种配置模块是一个**纯粹的"被依赖方"**，所有依赖箭头都是从外部指向它，没有从它指向外部的业务依赖。

### 1.2 币种配置仅依赖基础服务

币种配置 → ser-blog：操作日志记录。管理员对币种的新增、编辑、启用、禁用操作需要写入审计日志（币种配置需求文档3-4）。这是工程规范要求，不是业务依赖。

币种配置 → ser-s3：图标上传。币种需要配置图标（SVG/PNG，≤5k），上传走统一的S3文件存储服务（币种配置需求文档3-3编辑弹窗）。

币种配置 → 三方汇率API（HTTP）：汇率数据源。定时拉取3家三方汇率API取平均值，作为实时汇率的计算基础（币种配置需求文档4-2）。这是外部HTTP调用，不是内部RPC依赖。

### 1.3 被钱包依赖的具体场景

**钱包 → 币种配置（获取可用币种列表）**：用户在C端切换币种时（钱包产品原型3-1~3-4），需要知道平台当前启用了哪些币种。这个信息只有币种配置模块持有。如果币种配置的"可用币种列表"接口不存在，钱包C端的币种切换功能就无法实现——用户看不到可选的币种列表。

**钱包 → 币种配置（获取币种详情）**：钱包在处理充值、提现、兑换时，需要知道币种的具体属性——是法币还是加密货币？小数位数是多少？千分位符号是什么？（钱包需求文档4-1金额展示规则）。这些属性由币种配置统一管理。

**钱包 → 币种配置（获取汇率）**：这是最核心的依赖。钱包兑换功能（法币→BSB）需要入款汇率来计算兑换结果（钱包需求文档6）；BSB充值时需要入款汇率展示汇率换算（钱包需求文档5-2下方汇率展示）；BSB提现时需要出款汇率换算（钱包需求文档7-4）。入款汇率 = 实时汇率 × (1 + 汇率浮动%)，出款汇率 = 实时汇率 × (1 - 汇率浮动%)，这些计算逻辑在币种配置模块内部完成，钱包直接获取计算结果。

### 1.4 被财务依赖的具体场景

**财务 → 币种配置（获取币种信息）**：财务的通道配置和支付配置中，每个通道和支付方式都关联了特定币种（财务需求文档2-1通道配置编辑弹窗中有"币种"字段）。配置时需要从币种配置拉取币种列表作为下拉选项。

**财务 → 币种配置（获取基准币种）**：财务的统计报表需要把不同币种的金额统一换算为基准币种进行汇总（财务需求文档3-1时区处理+统计维度中明确"按币种维度统计"）。基准币种是USDT，这个信息由币种配置持有。

**财务 → 币种配置（获取平台汇率）**：财务统计中多币种金额换算为基准币种时使用的是平台汇率（币种配置需求文档4-1汇率应用场景表中"统计/报表/对账"一行明确写了"使用平台汇率"）。

### 1.5 小结

币种配置的依赖关系非常清晰：**只进不出**。它被钱包和财务（以及未来的游戏、统计等服务）依赖，自身不依赖任何业务模块。这种"基础设施"的定位意味着它**必须最先开发完成**，否则上层模块无从开展。

## 二、钱包模块的依赖关系

### 2.1 钱包依赖币种配置（向下依赖基建）

这部分依赖在上文"1.3 被钱包依赖的具体场景"中已阐述，此处从钱包视角做归纳。

钱包 → 币种配置，本质上是"应用层依赖基础设施层"。钱包的**每一个C端页面**都需要先获取当前币种信息，才能渲染内容和执行操作：

- 个人中心页（钱包产品原型2-1~2-3）：展示余额前需要知道当前选中的是哪个币种，该币种的金额格式如何展示（整数还是两位小数、千分位用点还是逗号）。
- 充值页（钱包产品原型3-1~3-4）：切换币种后充值方式列表要联动刷新，不同币种支持不同的充值渠道。
- 兑换页（钱包产品原型8）：需要获取入款汇率来实时计算"X法币 = Y BSB"。
- 提现页（钱包产品原型10-1~10-6）：BSB提现需要出款汇率换算，不同币种支持不同的提现方式。

**依赖性质**：单向强依赖。钱包必须依赖币种配置，但币种配置不需要知道钱包的存在。

### 2.2 钱包依赖财务（向外依赖协作方）

钱包并不直接对接三方支付通道，通道的管理、轮询、对接全部由财务模块负责。钱包在充值和提现场景中，需要调用财务提供的能力。

**充值场景中钱包 → 财务：**

- 钱包 → 财务（获取充值方式列表）：用户在充值页面看到的充值方式（USDT/银行转账/电子钱包）不是钱包自己管理的，而是财务的"支付配置"模块统一管理的（财务需求文档2-2支付配置）。钱包C端的充值方式列表页面（钱包产品原型5-x）需要调用财务来获取"当前币种下有哪些充值方式可用"。

- 钱包 → 财务（创建充值订单）：用户确认充值后，钱包需要调用财务来完成实际的订单创建。因为订单创建涉及通道轮询分配（财务需求文档2-1通道轮询规则：配比分配、半小时切换、2~5个通道）和三方通道对接（创建通道订单、获取支付页面URL），这些能力钱包不具备，由财务统一处理。

- 钱包 → 财务（查询订单状态）：用户从三方支付页面返回平台后（钱包产品原型7-1~7-4通用回调页面），需要查询充值订单的当前状态（成功/失败/处理中）。订单状态由财务维护。

**提现场景中钱包 → 财务：**

- 钱包 → 财务（获取提现方式列表）：与充值方式类似，提现方式也由财务的支付配置管理（财务需求文档2-2）。

- 钱包 → 财务（创建提现订单）：用户确认提现后，钱包将提现请求提交给财务，进入三级审核流程（风控→财务→出款，财务需求文档2-4提现流程）。钱包本身不参与审核，只负责提交。

**依赖性质**：钱包对财务的依赖集中在**充值下单和提现下单两条链路**上。如果财务的这些接口不可用，钱包的充值和提现功能就无法完成——用户只能看到页面，无法实际执行操作。但这不影响钱包的兑换功能和记录查询功能。

### 2.3 钱包依赖外部基础服务

**钱包 → ser-kyc（KYC状态查询）**：用户点击提现时，钱包需要先查询该用户是否完成了KYC实名认证（钱包需求文档7-1、钱包产品原型9-1~9-4）。未认证用户只能USDT提现，已认证用户可以选择全部提现方式。这个KYC状态不由钱包维护，需要调用KYC服务查询。

**钱包 → ser-user（获取用户信息）**：用户新增提现账户时，持有人姓名必须等于KYC认证姓名（钱包需求文档7-2），该姓名自动填充且不可编辑。钱包需要从用户服务获取KYC认证后的真实姓名。

**钱包 → 活动服务（查询奖金活动）**：充值时可以选择参加一个奖金活动（钱包需求文档5-6、钱包产品原型6-1~6-2）。可选的活动列表由活动服务管理，钱包调用获取。

**钱包 → ser-blog（审计日志）**：关键资金操作需要记录审计日志，这是工程规范要求。

**依赖性质**：这些依赖是**功能增强型**依赖。ser-kyc不可用时，最坏情况是所有提现都走未认证逻辑（仅USDT）；活动服务不可用时，充值页面不展示奖金选择即可。不像财务那样是"没有就完全不能工作"。

### 2.4 钱包被财务依赖（反向依赖，钱包作为被调用方）

这是双向耦合的另一面。财务在以下场景中需要调用钱包的能力：

**财务 → 钱包（充值入账）**：三方通道回调充值成功后，财务验证金额无误后需要调用钱包的入账接口，将资金计入用户的中心钱包（财务需求文档2-3充值流程的最后一步"调用钱包入账接口上分"）。这是充值链路的终点动作，如果钱包入账接口不可用，用户付了钱但余额不增加。

**财务 → 钱包（提现冻结）**：用户提交提现请求时，财务创建提现订单的同时需要冻结用户钱包中对应金额（财务需求文档2-4提现流程泳道图中明确标注"调用钱包冻结余额"）。冻结的目的是防止用户在审核期间重复提现导致超额。

**财务 → 钱包（提现解冻）**：提现审核被任何一级驳回时（财务需求文档2-4"任何一级驳回，订单状态变为已驳回"），需要调用钱包解冻之前冻结的金额，让用户可以重新使用这笔钱。

**财务 → 钱包（提现扣款确认）**：提现三级审核全部通过、通道出款成功后（财务需求文档2-4提现流程最后一步），需要调用钱包确认扣除之前冻结的金额。此时资金正式从用户账户中消失。

**财务 → 钱包（人工加款）**：管理员在财务后台发起人工加款并审核通过后（财务需求文档2-5-2），需要调用钱包的加款接口。加款需要指定目标钱包类型——是加到中心钱包、奖励钱包、主播钱包还是代理钱包（需求文档明确列出4种钱包可分别加款）。

**财务 → 钱包（人工减款）**：与加款对称，审核通过后调用钱包减款接口（财务需求文档2-5-3）。

**财务 → 钱包（充值补单）**：用户充值成功但系统未到账时，管理员新增补单并审核通过后（财务需求文档2-5-1），需要调用钱包入账，效果等同于正常充值入账。

**财务 → 钱包（查询用户钱包信息）**：管理员在人工修正时需要关联用户、查看其当前各钱包余额（财务需求文档2-5系列的操作页面中，选择用户后展示钱包信息）。

**依赖性质**：财务对钱包的依赖集中在**资金状态变更**上。财务是"发号施令方"（决定什么时候入账、冻结、解冻、扣款），钱包是"执行方"（实际操作余额和流水）。

### 2.5 钱包被游戏/直播/活动等模块依赖

这些外部模块对钱包的依赖是**单向的**——只有它们调用钱包，钱包不需要调用它们。

**游戏/投注 → 钱包（投注扣款）**：用户投注时需要从钱包扣除投注金额。关键业务规则是"优先扣中心钱包，不足部分再扣奖励钱包"（币种配置需求文档2-1资金钱包体系）。如果是中心+奖励混合投注，还需要记录各钱包的出资比例，供后续返奖使用。

**游戏/投注 → 钱包（投注返奖）**：赛事结果出来后，赢了的投注需要返奖。返奖的关键规则是"按投注时中心/奖励钱包的金额占比分别返奖"（币种配置需求文档2-1），这要求投注扣款时就记录了占比信息。

**直播/礼物 → 钱包（礼物消费扣款）**：用户在直播间送礼物时从中心钱包扣除对应金额。

**直播/礼物 → 钱包（主播收益入账）**：礼物打赏产生的主播收益入账到主播钱包（币种配置需求文档2-1钱包体系中"主播钱包"的资金来源是"礼物打赏收入、主播活动奖励"）。

**活动/任务 → 钱包（奖励入账）**：活动/任务奖励入账到奖励钱包（币种配置需求文档2-1钱包体系中"奖励钱包"的资金来源是"活动返奖、任务返奖、投注返奖"）。

**依赖性质**：这些调用方目前大多数还未开发或不在你的负责范围内。钱包只需要把RPC接口定义清楚、实现好，等它们按自己的开发节奏接入即可。优先级低于钱包↔财务的联动。

### 2.6 小结

钱包处于依赖关系的**中枢位置**：
- 向下依赖币种配置（获取基础数据）
- 横向双向依赖财务（充值/提现业务联动）
- 向上被游戏/直播/活动等多方依赖（提供资金操作能力）

这意味着钱包的**RPC暴露接口质量和稳定性**至关重要，因为多个模块的业务正确性直接取决于钱包的入账/扣款/冻结/解冻是否可靠。

## 三、财务管理模块的依赖关系

### 3.1 财务依赖币种配置（向下依赖基建）

**财务 → 币种配置（获取币种信息）**：财务的通道配置中每个代收/代付通道都关联特定币种（财务需求文档2-1通道配置列表中有"币种"字段），新增和编辑通道时需要从币种配置获取币种列表作为可选项。同理，支付配置中的充值方式和提现方式也都关联了币种（财务需求文档2-2支付配置列表中有"币种"字段）。

**财务 → 币种配置（获取基准币种和汇率）**：财务的数据统计模块需要把不同币种的金额统一换算为基准币种进行汇总展示（财务需求文档3-4/3-5充值统计和提现统计都是"按日期+币种维度统计"）。换算时使用平台汇率，基准币种从币种配置获取。

**依赖性质**：与钱包对币种配置的依赖类似，是"应用层依赖基础设施层"。没有币种信息，财务的通道配置和统计报表都无法正确运行。

### 3.2 财务依赖钱包（操作用户资金）

在上文"2.4 钱包被财务依赖"中已从钱包视角详述了所有场景。此处从财务视角归纳依赖的本质：

财务模块的核心职责是**管理支付通道和审核资金流转**，但它本身**不持有用户的钱包数据**。用户的余额、冻结金额、资金流水全部由钱包管理。因此，每当财务需要"动钱"时（充值入账、提现冻结/解冻/扣款、人工加减款），都必须调用钱包的接口。

具体来说：

- 充值成功后"上分"→ 需要钱包入账接口
- 提现发起时"锁钱"→ 需要钱包冻结接口
- 提现驳回时"还钱"→ 需要钱包解冻接口
- 出款成功后"扣钱"→ 需要钱包扣款接口
- 人工补单后"补钱"→ 需要钱包入账接口
- 人工加款后"给钱"→ 需要钱包加款接口
- 人工减款后"收钱"→ 需要钱包减款接口
- 人工操作时"看钱"→ 需要钱包余额查询接口

**依赖性质**：财务对钱包的依赖是"决策方对执行方"的依赖。财务负责判断"该不该动这笔钱"，钱包负责"实际去动这笔钱"。

### 3.3 财务依赖外部三方通道

**财务 → 三方支付通道（HTTP）**：财务的代收/代付功能最终需要对接三方支付渠道完成实际的收款和付款（财务需求文档2-3充值流程和2-4提现流程中都有"通道创建订单"和"通道回调"环节）。这是HTTP对外调用，不是内部RPC依赖。

**财务 → ser-blog（审计日志）**：同样是工程规范要求。

**财务 → ser-buser（用户名解析）**：财务后台展示操作人姓名时需要根据userId查询B端管理员姓名。

### 3.4 财务被钱包依赖（反向依赖，财务作为被调用方）

在上文"2.2 钱包依赖财务"中已详述。此处从财务视角补充：

财务需要暴露给钱包的核心能力是**支付方式查询和订单创建**：

- 充值方式查询：钱包C端要展示充值方式列表，数据来源是财务的支付配置
- 提现方式查询：同上，提现方式列表来源也是财务
- 充值下单：钱包C端提交充值请求后，财务负责通道分配、创建订单、对接三方
- 提现下单：钱包C端提交提现请求后，财务负责创建订单并启动审核流程
- 订单状态查询：钱包C端从三方页面返回后查询订单结果

这5个RPC接口是财务暴露给钱包的全部内容。财务并不像钱包那样被很多模块依赖——目前看来只有钱包需要调用财务的RPC。

### 3.5 小结

财务模块的依赖位置相对明确：
- 向下依赖币种配置（获取币种和汇率）
- 横向双向依赖钱包（发起资金操作指令 ↔ 提供支付能力给钱包C端）
- 向外依赖三方支付通道（实际收付款）
- 不被游戏/直播/活动等模块直接依赖（它们只和钱包打交道，不直接接触财务）

## 四、三模块间依赖关系的本质总结

### 4.1 依赖的分层结构

三模块之间的依赖从本质上看形成了**三层结构**：

**地基层 → 币种配置**：提供"什么币种存在、汇率是多少"的基础事实。这些是不可绕过的底层数据，所有涉及资金的模块都必须从这里获取。币种配置自身不需要知道"钱在谁那里"或"钱怎么流转"。

**核心操作层 → 钱包**：提供"用户的钱在哪里、能不能动、怎么动"的能力。钱包管理着用户的实际余额和资金流水，是所有"动钱"操作的唯一执行者。钱包需要从币种配置获取汇率来辅助计算，但计算完之后"加钱减钱"的动作完全由自己执行。

**业务流转层 → 财务**：提供"钱应该从哪个通道进来、经过怎样的审核流程出去"的管控能力。财务管理着支付通道和审核流程，但不直接持有用户资金。当它判断"这笔充值应该入账"或"这笔提现可以放行"时，必须调用钱包来执行实际的资金变动。

### 4.2 信息流和资金流的方向差异

理解三模块依赖关系时有一个关键区分点：**信息流和资金流的方向不同**。

**信息流**（配置数据、币种、汇率）：

- 币种配置 → 钱包（我有什么币种、汇率多少）
- 币种配置 → 财务（币种信息、基准币种、平台汇率）
- 财务 → 钱包（有哪些充值方式、提现方式、订单状态如何）

信息流的方向是**从基建到应用**，从提供方到消费方，没有双向性。

**资金操作指令流**（入账、扣款、冻结、解冻）：

- 财务 → 钱包（充值成功请入账、提现请冻结、审核通过请扣款、驳回请解冻）
- 游戏 → 钱包（投注请扣款、中奖请返奖）
- 直播 → 钱包（送礼请扣款、打赏请给主播入账）
- 活动 → 钱包（任务完成请发奖励）

资金操作指令流的方向是**从业务决策方到资金执行方**（钱包），是单向的。

**业务请求流**（用户发起操作）：

- 钱包C端 → 财务（用户要充值，请创建订单；用户要提现，请启动审核）

业务请求流的方向是**从用户入口到后台处理**，也是单向的。

正是"业务请求流"和"资金操作指令流"的方向相反，才造成了钱包和财务之间的**双向依赖**——钱包调用财务发起业务，财务调用钱包执行资金变动。但这两个方向发生在**不同的时间节点和不同的业务阶段**，不会形成同步循环。

### 4.3 各模块在充值场景中的角色分工

以一次完整充值为例，看三模块的协作关系（钱包需求文档5-7充值流程 + 财务需求文档2-3充值流程）：

1. **用户选择币种** → 钱包调用币种配置获取可用币种列表 → 币种配置提供信息
2. **用户选择充值方式** → 钱包调用财务获取充值方式列表 → 财务提供信息
3. **用户提交充值** → 钱包调用财务创建充值订单 → 财务分配通道、对接三方 → 三方返回支付页面
4. **用户完成支付** → 三方通道回调财务 → 财务验证金额、更新订单状态
5. **充值确认成功** → 财务调用钱包入账 → 钱包更新余额、记录流水

可以看到：**步骤1-3是钱包主导**（用户操作入口，向币种配置和财务获取信息），**步骤4-5是财务主导**（接收回调后反向调用钱包入账）。币种配置全程**只在步骤1被动提供数据**。

### 4.4 各模块在提现场景中的角色分工

以一次完整提现为例（钱包需求文档7-1提现流程 + 财务需求文档2-4提现流程）：

1. **用户点击提现** → 钱包调用ser-kyc查询KYC状态 → 决定可用提现方式
2. **用户选择提现方式** → 钱包调用财务获取提现方式列表 → 财务提供信息
3. **用户选择/新增账户** → 钱包内部处理（调用ser-user获取KYC姓名自动填充）
4. **用户提交提现** → 钱包调用财务创建提现订单 → 财务调用钱包冻结余额 → 订单进入审核中
5. **三级审核** → 财务后台操作（风控→财务→出款）
6a. **审核全部通过** → 财务调用三方通道出款 → 通道出款成功 → 财务调用钱包确认扣款
6b. **审核任一驳回** → 财务调用钱包解冻余额 → 用户可重新使用该金额

可以看到：**步骤1-4是钱包主导**，**步骤4（后半）-6是财务主导**。提现场景中的依赖更复杂，因为涉及"冻结→审核→扣款/解冻"这个异步多步骤流程。

### 4.5 各模块在兑换场景中的角色分工

兑换是三个核心场景中最简单的（钱包需求文档6）：

1. **用户输入兑换金额** → 钱包调用币种配置获取入款汇率 → 计算兑换结果和赠送金额
2. **用户确认兑换** → 钱包内部处理：扣法币（中心钱包）→ 加BSB兑换部分（中心钱包）→ 加BSB赠送部分（奖励钱包）→ 记录流水

**财务完全不参与**。兑换是钱包和币种配置两个模块独立完成的操作。这也是为什么兑换功能是最适合先开发的C端功能。

### 4.6 人工修正场景的角色分工

人工修正场景（财务需求文档2-5）的依赖方向与充值/提现相反——是**财务B端发起，钱包被动执行**：

1. **管理员发起补单/加款/减款** → 财务后台（B端操作）
2. **管理员选择目标用户** → 财务调用钱包查询用户钱包信息 → 展示当前余额
3. **另一管理员审核通过** → 财务调用钱包执行加款/减款/入账

这个场景中钱包的角色是纯粹的**执行者**，不做任何业务判断，只负责按指令修改余额和记录流水。审核和权限控制由财务负责。

## 五、模块内部的子功能依赖

### 5.1 币种配置模块内部

币种配置内部的子功能之间也有依赖关系：

**币种CRUD → 汇率体系**：必须先有币种数据，汇率才有"对谁"的对象。不可能为一个不存在的币种配置汇率浮动%和阈值。因此B端的币种新增/编辑操作是汇率功能的前置。

**三方汇率拉取 → 平台汇率 → 入款/出款汇率**：三方API拉取的是实时汇率（币种配置需求文档4-2），平台汇率初始值等于实时汇率，后续按偏差值和阈值的关系决定是否更新。入款汇率和出款汇率基于实时汇率加减浮动%计算。这三者是层级递推关系。

**汇率变更 → 汇率日志**：平台汇率因偏差超过阈值自动更新时，才会产生汇率日志记录（币种配置需求文档4-4）。汇率日志是汇率变更事件的后置产物。

**基准币种设置**：这是一次性操作（设置后不可更改，币种配置需求文档3-2），与其他功能无前后置关系，但建议在系统初始化时最先完成，因为后续统计和报表都依赖它。

### 5.2 钱包模块内部

**钱包表结构设计 → 所有资金操作**：5种钱包（中心/奖励/主播/代理/场馆）的表设计和流水表设计是所有后续功能的地基。没有表就没有数据存储，任何接口都无法工作。

**入账/扣款原子操作 → 冻结/解冻 → 业务专用接口**：入账和扣款是最底层的原子操作（对余额的加和减）。冻结/解冻是在原子操作之上增加"冻结态"的管理。投注扣款、礼物扣款、人工加减款等业务专用接口，底层都是在原子操作之上叠加各自的业务规则（如投注扣款的优先级逻辑）。

**充值功能内部**：获取充值方式列表 → 重复转账检测 → 发起充值 → 查询状态。这是一个前后顺序的链路，用户必须先看到方式列表才能选择，选择后才能提交，提交后才能查询结果。

**提现功能内部**：KYC校验 → 获取提现方式 → 获取/新增提现账户 → 获取限额 → 发起提现。其中"获取/新增提现账户"和"获取限额"之间**没有前后关系**，可以并行处理。但都必须在KYC校验之后和发起提现之前。

**记录查询**：四个Tab（充值/提现/兑换/奖励）之间**完全独立**，互不影响。它们与写操作之间是"先有数据才能查询"的后置关系。

### 5.3 财务模块内部

**通道配置 → 支付配置 → 充值/提现方式查询**：这是财务内部最核心的一条依赖链。支付配置中的充值方式/提现方式必须关联代收/代付通道（财务需求文档2-2编辑弹窗中有"代收通道/代付通道"下拉选择），因此通道配置是支付配置的前置。支付配置完成后，才有数据可供充值/提现方式查询RPC返回。

**通道配置 → 通道轮询规则**：轮询规则是通道配置的子功能，必须先有通道才能配置通道之间的轮询配比（财务需求文档2-1通道轮询规则）。

**充值记录/提现记录 → 数据统计**：统计是对历史订单数据的聚合计算（财务需求文档3-4/3-5），必须先有订单记录才有可统计的数据。统计功能是记录数据的后置消费者。

**提现记录 → 三级审核**：审核操作（通过/驳回/锁定/解锁）附着在提现记录上，必须先有提现订单才有审核对象。

**人工修正**：补单/加款/减款各自独立，互不影响。但都需要在钱包的入账/扣款/加款/减款RPC接口就绪后才能完成闭环。

## 六、依赖关系对开发策略的指导意义

### 6.1 最先动手的一定是币种配置

从上面的分析可以清晰看到：币种配置是所有依赖链的起点，没有它，钱包和财务的大部分功能都无法运行。而且币种配置不依赖任何业务模块，可以完全独立开发和测试。

### 6.2 钱包的RPC暴露接口是第二优先

钱包的入账/扣款/冻结/解冻/查询这组RPC接口，是财务模块调用钱包的全部触点。这组接口完成后，财务模块就可以开始联调充值回调、提现审核、人工修正等流程。越早实现这组接口，双方的联调阻塞就越少。

### 6.3 钱包C端中兑换最先、充值提现可稍后

兑换功能不依赖财务，只需要币种配置的汇率接口和钱包自身的入账/扣款能力。而充值和提现的C端功能需要调用财务的RPC，如果财务还没开发完这些接口，就需要mock。从开发效率角度，先做兑换，等财务的充值方式/提现方式查询和下单接口就绪后再做充值和提现，可以减少mock和返工。

### 6.4 记录查询可以收尾做

记录查询是纯读操作，不影响任何写链路，也不被任何模块依赖。它的唯一前置是"表中有数据"。可以在充值/提现/兑换功能都跑通后再统一实现。

### 6.5 双方需要最早对齐的是IDL契约而非实现

钱包和财务之间12个双向RPC接口，最关键的不是谁先实现完，而是双方对请求/响应结构达成一致。IDL定义好之后，各自可以mock对方接口独立开发，最后联调对接。如果IDL没对齐就各做各的，联调时大概率要返工调整字段。
