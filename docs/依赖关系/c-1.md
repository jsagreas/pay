### 1.2 双向依赖详解

钱包和财务之间的双向依赖是整个系统最关键的耦合点：

```
                    钱包 ──────────► 财务
                    │                 │
                    │  ① 查充值方式    │
                    │  ② 查提现方式    │
                    │  ③ 匹配通道支付  │
                    │                 │
                    ◄──────────── 财务
                    │                 │
                    │  ④ 充值入账     │
                    │  ⑤ 确认提现扣款  │
                    │  ⑥ 驳回解冻     │
                    │  ⑦ 补单入账     │
                    │  ⑧ 人工加款     │
                    │  ⑨ 人工减款     │
                    │  ⑩ 查询余额     │
```

**这不是循环依赖问题**：虽然两个模块互相调用，但调用场景完全不同——钱包调财务是为了**获取配置数据和发起支付**，财务调钱包是为了**操作用户余额**。在微服务架构中这是正常的协作模式，两边注册对方的 RPC 客户端即可。

## 二、三大业务链路的完整依赖链

### 2.1 充值链路

```
用户操作                    钱包模块                    财务模块                 币种配置           外部
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
                                                                              ┌──────────┐
                                                                              │ 三方API   │
                                                                              │ 汇率已就绪 │
                                                                              └────┬─────┘
                                                                                   │
[0] 前置条件                                                                        ▼
                                                        ┌──────────────┐    ┌──────────┐
                                                        │ B端已配置:    │    │ 币种已配置 │
                                                        │ 代收通道      │    │ 汇率已更新 │
                                                        │ 充值方式      │    │ 币种已启用 │
                                                        │ 轮询参数      │    └──────────┘
                                                        └──────────────┘

[1] 进入钱包          ┌─────────────────┐
    选择币种 ────────►│ 获取币种列表及余额 │──── RPC ────────────────────►│ 查币种列表/状态 │
                     └────────┬────────┘                              └────────────────┘
                              │
[2] 进入充值页        ┌───────▼─────────┐
                     │ 获取充值方式列表   │──── RPC ──►│ 查充值方式 │
                     │ (聚合: 方式+汇率) │──── RPC ────────────────────►│ 查汇率       │
                     └────────┬────────┘
                              │
[3] 选择奖金活动      ┌───────▼─────────┐
    (可选)           │ 获取奖金活动列表   │ ← 数据源待定
                     └────────┬────────┘
                              │
[4] 选择金额+方式     ┌───────▼─────────┐
    点击充值         │ 重复充值检查      │ ← 查本地待支付订单
                     └────────┬────────┘
                              │ 通过
                     ┌────────▼────────┐
                     │ 创建充值订单      │──── RPC ──►│ 匹配通道创建支付 │──►│ 三方通道 │
                     └────────┬────────┘             (轮询+创建)

                     │ 或跳转三方承载页   │ ← 三方页面
                     └─────────────────┘

                     ........用户完成支付........

[6] 通道回调                                         ┌──────────────┐
                                                     │ 接收通道回调   │──►│ 三方通道 │
                                                     │ 验签+更新记录  │
                                                     └───────┬──────┘
                                                             │ RPC
[7] 入账             ┌─────────────────┐                     │
                     │ 充值入账(RPC对外) │◄────────────────────┘
                     │ 中心钱包余额 +N   │
                     └────────┬────────┘
                              │
[8] 可查记录          ┌───────▼─────────┐
                     │ 充值记录列表/详情  │
                     └─────────────────┘
```

**依赖结论**：
- **[0] 是地基**：币种配置 + 通道配置 + 充值方式配置 必须先就绪
- **[1]-[4] 是顺序链**：每一步依赖前一步的结果
- **[6]-[7] 是跨模块紧耦合**：财务回调后必须成功调用钱包入账，失败则需要补偿/重试
- **[3] 奖金活动是可选分支**：不影响主链路

### 2.2 提现链路

```
用户操作                    钱包模块                    财务模块               外部
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
                                                                          ┌──────────┐
[0] 前置条件          需要: 用户有余额                 需要: 代付通道已配置   │ KYC已认证 │
                     需要: 币种已启用                 需要: 提现方式已配置   │ (或跳过)  │
                                                                          └──────────┘

[1] 点击提现          ┌─────────────────┐
                     │ 查询KYC状态      │──── RPC ────────────────────────►│ KYC服务  │
                     └────────┬────────┘
                              │
                     ┌────────▼────────┐
                     │ 未认证→弹窗选择   │
                     │ ├ "去认证" → KYC  │
                     │ └ "继续" → 仅USDT │
                     │ 已认证→直接进入    │
                     └────────┬────────┘
                              │
[2] 选择提现方式      ┌───────▼─────────┐
                     │ 获取提现方式列表   │──── RPC ──►│ 查提现方式 │
                     │ (聚合: 方式+限额) │──── RPC ──►│ 查汇率     │ (币种配置)
                     └────────┬────────┘
                              │
[3] 填写账户信息      ┌───────▼─────────────────────┐
    (首次)           │ 获取提现账户信息 ← 非首次复用   │
                     │ 保存提现账户信息 ← 首次填写     │
                     │ (姓名=KYC姓名, 只读)          │
                     └────────┬────────────────────┘
                              │
[4] 输入金额          ┌───────▼─────────┐
    确认提现         │ 提现限额查询      │──── RPC ──►│ 查方式配置 │
                     │ (单笔/单日/手续费) │
                     └────────┬────────┘
                              │ 校验通过
                     ┌────────▼────────┐
                     │ 创建提现订单      │

                     │ └ 生成订单       │
                     └────────┬────────┘
                              │
                     ┌────────▼────────┐
                     │ 提现状态回调弹窗   │ ← 告知用户"提现进行中"
                     └─────────────────┘

                     ........进入B端审核流程........

[5] 风控审核                                         ┌──────────────┐
                                                     │ 提现记录列表   │
                                                     │ 锁定订单      │
                                                     │ 风控审核       │
                                                     └───────┬──────┘
                                                             │
                                                     ┌───────▼──────┐
                                                     │ 通过 or 驳回  │
                                                     └───────┬──────┘
                                                             │
[6] 财务审核                                         ┌───────▼──────┐
    (风控通过后)                                      │ 财务审核      │
                                                     └───────┬──────┘
                                                             │
                                              ┌──────────────┼──────────────┐
                                              │              │              │
[7a] 审核通过        ┌─────────────────┐      │    ┌─────────▼────────┐     │
                     │ 确认提现扣款     │◄─── RPC ──┤ 发起代付         │     │
                     │ 冻结→正式扣除    │           │ (调用代付通道)    │     │
                     └─────────────────┘           └──────────────────┘     │
                                                                            │
[7b] 审核驳回        ┌─────────────────┐                                    │
                     │ 解冻退回余额     │◄────────────── RPC ────────────────┘
                     │ 冻结→退回用户    │
                     └─────────────────┘

[8] 可查记录          ┌─────────────────┐
                     │ 提现记录列表/详情  │
                     └─────────────────┘
```

**依赖结论**：
- **[0] 地基更多**：比充值多了 KYC 依赖和"用户必须有余额"的前提
- **[1] KYC 是硬性前置**：决定了用户能用哪些提现方式
- **[4] 冻结与创建是原子操作**：必须在同一事务内，不能"先创建订单再冻结"
- **[5]-[7] 审核链条严格顺序**：风控 → 财务 → 出款，中间任何环节驳回都要调钱包解冻
- **[7a] 和 [7b] 是互斥分支**：通过走扣款，驳回走解冻，不会同时发生

### 2.3 兑换链路

```
用户操作                    钱包模块                    币种配置
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[0] 前置条件          需要: 法币余额 > 0              需要: 汇率已就绪
                     需要: 在法币币种下(非BSB)

[1] 进入兑换页        ┌─────────────────┐
                     │ 获取兑换汇率      │──── RPC ──►│ 查汇率 │
                     │ (1 BSB = X 法币)  │
                     └────────┬────────┘
                              │
[2] 输入金额          ┌───────▼─────────┐
    (可选预览)        │ 兑换预览         │ ← 可前端基于汇率自行计算
                     │ 兑换获得+赠送+到账 │
                     └────────┬────────┘
                              │
[3] 确认兑换          ┌───────▼─────────┐
                     │ 执行兑换         │──── RPC ──►│ 查汇率(确认时再取一次) │
                     │ ├ 校验余额       │
                     │ ├ 扣法币余额     │
                     │ ├ 加BSB余额      │
                     │ │  (含赠送部分)   │
                     │ └ 生成兑换记录    │
                     └────────┬────────┘
                              │
[4] 兑换成功          ┌───────▼─────────┐
                     │ 3s成功提示       │
                     │ 兑换记录可查      │
                     └─────────────────┘
```

**依赖结论**：
- **最简单的链路**：只依赖币种配置的汇率，**不需要财务模块参与**
- **完全在钱包模块内闭环**：扣法币 + 加BSB 在同一事务内
- **唯一外部依赖**：币种配置的汇率查询

## 三、接口分层依赖模型

将所有接口按依赖深度分为 7 层。**上层依赖下层，同层互相独立**。

```
Layer 6 ─── 统计导出层 ─── 需要大量业务数据积累后才有意义
  │
Layer 5 ─── 人工修正层 ─── 基于已有订单和余额的纠错操作
  │
Layer 4 ─── 记录查询层 ─── 业务发生后的只读查询
  │
Layer 3 ─── 业务执行层 ─── 充值/提现/兑换的创建和执行
  │
Layer 2 ─── C端展示层 ─── 充值方式/提现方式/汇率展示
  │
Layer 1 ─── 基础查询层 ─── 余额查询/币种列表
  │
Layer 0 ─── 地基配置层 ─── 币种/通道/方式/汇率的B端配置
```

### Layer 0：地基配置层（零依赖，最先开发）

> 这些接口不依赖系统内任何其他接口，是所有后续功能的前提

| 接口 | 所属模块 | 依赖 | 说明 |
|------|---------|------|------|
| 基准币种设置 | 币种配置(B端) | 无 | 一次性操作，其他统计依赖它 |
| 币种列表查询 | 币种配置(B端) | 无 | 读DB，所有币种相关功能的源头 |
| 币种编辑 | 币种配置(B端) | 无 | 汇率浮动/阈值/图标/状态 |
| 汇率定时更新 | 币种配置(定时) | 三方API | 拉取实时汇率→阈值判断→更新 |
| 实时汇率获取 | 币种配置(B端) | 三方API | 手动触发获取最新汇率 |
| 汇率日志列表 | 币种配置(B端) | 无 | 读日志表 |
| 代收通道列表/编辑 | 财务(B端) | 无 | 通道配置，独立于钱包 |
| 代付通道列表/编辑 | 财务(B端) | 无 | 通道配置，独立于钱包 |
| 轮询参数查询/配置 | 财务(B端) | 无 | 通道轮询规则 |

**开发顺序提示**：这一层最先做，做完后系统才有"币种"和"通道"的概念。

### Layer 1：基础查询层（依赖 Layer 0）

> 需要币种和汇率数据存在后才能工作

| 接口 | 所属模块 | 依赖 Layer 0 的什么 | 说明 |
|------|---------|-------------------|------|
| 汇率查询(RPC对外) | 币种配置 | 币种+汇率数据 | 供钱包/财务调用的内部接口 |
| 币种列表查询(RPC对外) | 币种配置 | 币种数据 | 供钱包/财务调用 |
| 查询钱包余额 | 钱包(C端) | 币种必须存在 | 按币种查余额 |
| 获取币种列表及余额 | 钱包(C端) | 币种列表+状态 | 聚合币种信息+余额 |
| 充值方式新增/编辑 | 财务(B端) | 代收通道必须存在 | 关联通道 |
| 提现方式新增/编辑 | 财务(B端) | 代付通道必须存在 | 关联通道 |

### Layer 2：C端展示层（依赖 Layer 1）

> 需要基础数据层就绪后，才能为C端组装展示数据

| 接口 | 所属模块 | 依赖 Layer 1 的什么 | 说明 |
|------|---------|-------------------|------|
| 获取充值方式列表 | 钱包(C端) | 充值方式配置 + 汇率 | 聚合: RPC→财务(方式) + RPC→币种(汇率) |
| 获取提现方式列表 | 钱包(C端) | 提现方式配置 + KYC状态 | 聚合: RPC→财务(方式) + RPC→KYC(状态) |
| 获取兑换汇率 | 钱包(C端) | 汇率数据 | RPC→币种配置 |
| 提现限额查询 | 钱包(C端) | 提现方式配置 | 单笔/单日限额+手续费 |
| 获取可用奖金活动列表 | 钱包(C端) | 活动配置(源待定) | 充值时的奖金选择 |
| 保存/获取提现账户信息 | 钱包(C端) | 用户已存在 | 独立于支付方式，但提现流程需要 |
| 钱包概览信息 | 钱包(C端) | 余额 + 币种 | 个人中心聚合展示 |

### Layer 3：业务执行层（依赖 Layer 2）

> 核心业务操作，每个都依赖展示层的数据先就绪

| 接口 | 所属模块 | 依赖 Layer 2 的什么 | 关键约束 |
|------|---------|-------------------|---------|
| 创建充值订单 | 钱包(C端) | 充值方式 + 汇率 + 通道 | 调财务RPC匹配通道 |
| USDT支付页信息 | 钱包(C端) | 充值订单已创建 | 订单创建后才能展示 |
| 创建提现订单 | 钱包(C端) | 提现方式 + 限额 + KYC + 余额 + 账户 | 冻结余额(事务) |
| 执行兑换 | 钱包(C端) | 汇率 + 余额 | 扣法币+加BSB(事务) |
| 重复充值检查 | 钱包(C端) | 充值订单表 | 可合并到创建订单 |
| 兑换预览 | 钱包(C端) | 汇率 | 可前端计算 |
| 充值回调处理 | 财务(Webhook) | 充值订单已创建 + 通道密钥 | 验签后RPC调钱包入账 |

### Layer 4：后置结果层（依赖 Layer 3）

> 业务执行后产生的数据，只读查询

| 接口 | 所属模块 | 依赖 | 说明 |
|------|---------|------|------|
| 充值入账(RPC对外) | 钱包 | 充值回调触发 | 被财务调用，增加余额 |
| 确认提现扣款(RPC对外) | 钱包 | 提现审核通过 | 被财务调用，扣除冻结 |
| 解冻退回(RPC对外) | 钱包 | 提现审核驳回 | 被财务调用，解冻余额 |
| 充值记录列表 | 钱包(C端) | 充值订单存在 | 分页查询 |
| 提现记录列表 | 钱包(C端) | 提现订单存在 | 分页查询 |
| 兑换记录列表 | 钱包(C端) | 兑换记录存在 | 分页查询 |
| 奖励记录列表 | 钱包(C端) | 奖励记录存在 | 分页查询 |
| 充值/提现订单详情 | 钱包(C端) | 订单存在 | 单笔查询 |

## 四、耦合度分析

### 4.1 紧耦合（必须一起设计，不能独立变更）

```
紧耦合对                              原因                           处理策略
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
创建充值订单 ←→ 匹配通道创建支付       钱包创建订单时必须同步调财务      事务性：失败需回滚订单
                                       匹配通道，两者是一个原子流程

充值回调处理 ←→ 充值入账(RPC)         财务收到回调后必须成功调钱包      幂等+重试：回调可能重复
                                       入账，否则用户付了钱没到账

创建提现订单 ←→ 冻结余额              同一事务内完成，不能先创建后      DB事务：同一个事务内
                                       冻结，否则有时间窗口风险

提现审核通过 ←→ 确认扣款(RPC)         财务审核通过后必须调钱包扣除      消息可靠性：不能丢
提现审核驳回 ←→ 解冻退回(RPC)         财务驳回后必须调钱包解冻退回      消息可靠性：不能丢

补单/加款/减款审核 ←→ 余额变更(RPC)    审核通过后必须调钱包操作余额      消息可靠性：不能丢

执行兑换 ←→ 双币种余额变更            扣法币+加BSB必须在同一事务内      DB事务：同一个事务内
```

## 八、总结

### 8.1 一句话总结各模块定位

```
币种配置 ═══ 最底层的地基，所有金额计算的根基，不依赖任何业务模块
钱包模块 ═══ C端的门面和余额的管家，上接用户下接财务，是中间枢纽
财务管理 ═══ B端的后台和通道的管家，既给钱包提供配置数据，又消费钱包的余额接口
```

### 8.2 最重要的设计约束

| 约束 | 原因 | 影响 |
|------|------|------|
| 钱包RPC对外接口必须**幂等** | 财务调钱包入账/扣款可能重试 | 每个方法需要唯一请求ID去重 |
| 冻结+创建提现订单必须**同事务** | 否则存在冻结成功但订单未创建的窗口 | 不能跨服务事务，必须在钱包内完成 |
| 充值入账不能丢 | 用户付了钱必须到账 | 需要可靠消息机制或同步重试 |
| 汇率查询必须**高可用** | 充值/提现/兑换全依赖它 | 考虑缓存+降级策略 |
| 币种配置变更需**通知下游** | 币种禁用后钱包必须即时生效 | 考虑 ETCD Watch 或消息通知 |

### 8.3 风险最高的依赖点

### 9.1 币种配置模块的依赖关系

#### 它依赖谁

币种配置模块是三大模块中**依赖最少**的，几乎是一个自给自足的配置中心。它的外部依赖仅有：

**币种配置 → 三方汇率 API**：需求文档明确写了"通过3家三方API获取实时汇率，算平均值"。这是币种配置唯一的外部数据源依赖。汇率定时任务按配置频率拉取三方数据，计算偏差值，判断是否超过阈值来决定更不更新平台汇率。没有这个数据源，整个汇率体系就是静态的。

**币种配置 → 审计日志服务(BLog)**：币种编辑、基准币种设置这些操作需要记录审计日志。这是所有模块的通用依赖，异步调用，失败不阻塞主流程。

**币种配置 → S3服务**：币种编辑弹窗中有"币种图标上传（SVG/PNG格式，不超过5KB）"，上传图标走 S3 服务。

**币种配置 → B端用户服务(BUser)**：币种列表的"操作人"字段需要显示管理员名称，需查询 BUser 获取。

以上四个依赖都比较轻量，币种配置模块本身**不依赖钱包模块，也不依赖财务模块**。

#### 谁依赖它

这是重点——币种配置是整个资金体系的地基，被两个模块强依赖：

**钱包模块 → 币种配置**：钱包的几乎每个核心功能都需要币种配置提供数据。具体来说：

- 用户进入钱包首页切换币种时，"获取币种列表及余额"接口需要调用币种配置获取**当前哪些币种是启用的**。需求文档写了"开关禁用后，C端不可使用、不可充值、不可提现"，所以钱包必须实时感知币种的启用状态。

- 用户充值时，充值页面底部要展示汇率（"只有BSB充值金额输入框下有对应汇率展示"），钱包需要调用币种配置获取**入款汇率**。

- 用户提现时，BSB提现页面的金额输入框下要显示汇率，钱包需要调用币种配置获取**出款汇率**。

- 用户兑换法币→BSB时，兑换页面展示"1 BSB = X 法币"，计算"兑换获得+赠送+实际到账"，这些都需要币种配置提供**当前汇率**。兑换的核心计算完全依赖这个汇率数据。

- 金额展示精度也来源于币种配置——法定货币2位小数、加密货币8位小数的规则，以及千分位符号、小数点符号的差异，虽然这些可以前端处理，但数据源在币种配置。

**财务模块 → 币种配置**：财务模块的依赖集中在统计换算场景：

- 需求文档中充值记录、提现记录都有"基准币种订单金额"字段，需要用**平台汇率**将原币种金额换算为基准币种金额。

- 数据统计（代收/代付通道统计、充值统计、提现统计）的列表中都有"基准币种"系列的汇总字段，需要调用币种配置获取**基准币种代码和平台汇率**来做换算。

#### 币种配置的特殊性

需求文档中有一条关键规则："新增币种数据由开发写入，后台可编辑图标、开关状态"。也就是说币种新增不走接口，是DBA级别的操作。这意味着币种配置模块**只有编辑能力，没有新增能力**。币种是一个相对静态的配置，不会频繁变动。

另一条关键规则："基准币种只可设置一次，选择后无法修改，提交时需二级确认"。基准币种一旦确定（推荐USDT），就是一个常量，所有统计都基于它。

### 9.2 钱包模块的依赖关系

钱包模块是三大模块中**依赖关系最复杂**的，因为它是C端用户操作的入口，承上启下，既要从别的模块获取配置数据，又要对外暴露余额操作接口供别的模块调用。

#### 钱包依赖币种配置（数据获取方向）

**钱包 → 币种配置：查询币种列表及状态**

需求原型中"个人中心→钱包区域→点击币种切换"的页面展示了所有币种+余额+国旗图标+选中勾号。这个页面的数据源有两个：余额在钱包自己的数据库里，但币种列表（代码、名称、类型、国家、图标、是否启用）来自币种配置。如果某个币种在后台被禁用了，钱包这边就不应该再展示它。

**钱包 → 币种配置：查询汇率**

这是钱包对币种配置最频繁的调用。从需求文档和原型中可以追溯到以下场景全部需要汇率：

- 充值页面：BSB充值时展示"1BSB=0.1USDT"，选择越南银行时展示"1BSB=d2,620"，选择电子钱包时展示"1BSB=Rp1,530"。这些汇率来自币种配置的**入款汇率**。
- 提现页面：BSB提现金额输入框下显示汇率。这个汇率来自币种配置的**出款汇率**。
- 兑换页面：展示"1 BSB = X 法币"，并基于此汇率计算兑换获得、赠送、实际到账。这个汇率来自币种配置的**平台汇率**（或入款汇率，需评审确认）。
- 创建充值订单时：需要用汇率将用户输入的币种金额换算为支付金额。
- 创建提现订单时：BSB提现需要用汇率换算为法币出款金额。
- 执行兑换时：需要汇率来计算法币到BSB的兑换比例。

可以说，**钱包模块对币种配置的汇率查询接口是最高频的依赖**，如果汇率接口挂了，充值/提现/兑换全部瘫痪。

#### 钱包依赖财务管理（配置获取 + 支付执行方向）

**钱包 → 财务：查询充值方式列表**

需求原型中充值页面展示的"USDT（推荐）、越南银行、电子钱包扫码"这些充值方式，以及每种方式下的快捷金额档位（如VND的200,000/500,000/1,000,000...），这些数据的配置源头在财务模块的"支付配置→充值方式"。钱包模块C端的"获取充值方式列表"接口，内部需要调用财务的RPC获取这些配置。

为什么是钱包调财务而不是C端直接调财务？因为从需求原型看，充值页面是一个聚合页——既有余额（钱包数据）、又有充值方式（财务数据）、还有汇率（币种配置数据）、还有奖金活动（源待定）。由钱包作为C端聚合层统一提供数据，比让前端调多个服务更合理。

**钱包 → 财务：查询提现方式列表**

同理，需求原型中提现方式选择页展示的"银行卡提现（T+1）、GoPay（即时）、OVO（即时）..."这些提现方式，数据源在财务模块的"支付配置→提现方式"。钱包调用财务RPC获取当前币种可用的提现方式及其限额规则。

**钱包 → 财务：匹配通道并创建支付**

需求文档的充值流程泳道图写了"提交→服务端轮询匹配通道→创建订单→拉起支付"。这里的"轮询匹配通道"是财务模块的核心能力——财务掌握所有代收通道的配置（费率/限额/权重/密钥/状态），以及轮询参数（成功率系数/时间系数/预警系数）。钱包不具备也不应该具备通道匹配能力。

所以当用户在C端点击"去充值"时，钱包模块创建一个本地的充值订单后，必须调用财务模块的RPC来完成"选哪个通道+向三方发起支付请求+返回支付页信息"这一系列操作。

这个依赖是**充值流程中最关键的跨模块调用**，如果财务的通道匹配接口不可用，用户就无法完成充值。

#### 钱包依赖KYC服务

**钱包 → KYC服务：查询认证状态及姓名**

需求文档明确写了提现的KYC分级权限：
- 未认证 → 弹窗选择"去认证"或"继续提现（仅USDT）"
- 已认证 → 直接进入提现方式选择页

所以用户一点击"提现"，钱包就需要调用KYC服务查询该用户的认证状态。

另外，需求文档规定"账户持有人姓名=KYC认证姓名（提现页自动填充已实名的姓名且不可编辑）"。所以保存提现账户信息时，钱包也需要调KYC获取用户的认证姓名来做校验和自动填充。

这个依赖只在提现流程中出现，充值和兑换不涉及KYC。

#### 谁依赖钱包（被调用方向）

钱包模块对外暴露的RPC接口，主要是余额变更相关的操作，被**财务模块**大量调用：

**财务 → 钱包：充值入账**

需求文档的充值流程中"通道回调→验证金额→更新状态"这一步由财务处理（因为通道密钥和验签逻辑在财务），验证通过后财务需要调用钱包的RPC将充值金额入账到用户的中心钱包。这是"用户付了钱→钱到账"的最后一步，不能失败、不能重复。

**财务 → 钱包：确认提现扣款 / 解冻退回**

需求文档的提现流程中有多级审核（风控→财务），审核结果有两个分支：
- 通过：财务调钱包RPC，将冻结的金额正式扣除（变成真的花出去了）
- 驳回：财务调钱包RPC，将冻结的金额解冻退回（钱还给用户）

这两个调用是互斥的（一个订单要么通过要么驳回），但都是财务发起、钱包执行。

**财务 → 钱包：补单入账 / 人工加款 / 人工减款**

需求文档的人工修正模块中：
- 补单审核通过后 → 财务调钱包RPC → 补单金额入账到中心钱包
- 加款审核通过后 → 财务调钱包RPC → 按指定钱包类型增加余额（需求原型中加款支持同时给中心钱包、奖励钱包、代理钱包、主播钱包分别加款）
- 减款审核通过后 → 财务调钱包RPC → 按指定钱包类型扣减余额（需求原型中减款页面展示各钱包当前余额，减款金额不超余额）

**财务 → 钱包：查询用户余额**

需求原型中提现审核弹窗显示了用户的充值/提现历史统计信息，人工减款页面展示了用户各钱包的当前余额。这些数据需要财务调用钱包RPC查询。

#### 钱包模块内部的自依赖

钱包内部也有接口间的先后依赖：

- 创建提现订单 → 依赖 → 获取提现账户信息（非首次）或 保存提现账户信息（首次）。从需求原型看，用户选择提现方式后先看到账户信息页面，填完/确认后才到输入金额页面，最后才创建订单。所以账户信息是提现订单的前置。

- 创建充值订单 → 依赖 → 重复充值检查。需求文档写了"同一UserID+同一币种+同一充值金额"的待支付订单达到3笔时强制拦截。这个检查可以合并到创建订单接口内部，也可以独立为一个前置接口（点击充值时先检查）。

- USDT支付页信息 → 依赖 → 创建充值订单。USDT支付页展示的收款地址和二维码是在订单创建后才能确定的（可能与订单绑定），所以这是创建订单的后置接口。

- 各记录列表/详情 → 依赖 → 对应的业务已经发生过。没有充值过就没有充值记录，没有提现过就没有提现记录。这些是纯粹的后置只读接口。

### 9.3 财务管理模块的依赖关系

财务管理模块虽然是别人负责开发，但因为与钱包存在大量双向依赖，必须清楚它的依赖全貌。

#### 财务依赖币种配置

**财务 → 币种配置：查询基准币种和平台汇率**

需求文档中充值记录、提现记录都有"基准币种订单金额""基准币种活动优惠""基准币种手续费"等字段。数据统计模块的所有统计表格也都有基准币种换算列。这些字段的计算公式是：基准币种金额 = 原币种金额 ÷ 该币种对基准币种的平台汇率。

所以财务模块在生成订单记录和统计报表时，需要调用币种配置获取当前的平台汇率。这个依赖发生在**数据写入时**（创建订单记录时计算基准币种金额）和**数据查询时**（统计报表换算时）。

#### 财务依赖钱包

这是依赖最重的方向——财务模块有 **7 个场景**需要调用钱包的RPC。上面在"谁依赖钱包"部分已经详细描述了每个场景，这里从财务的视角再梳理一遍调用时机：

**财务充值回调处理时 → 钱包.充值入账**：三方通道支付成功回调到财务，财务验签+更新订单状态后，调钱包入账。这个调用发生在"充值流程的最后一步"，是异步的（由三方回调触发，不是用户操作触发）。

**财务提现审核时 → 钱包.确认扣款 或 钱包.解冻退回**：B端管理员在提现记录页面点击"审核"，弹出审核弹窗，选择"通过"或"驳回"后触发。需求原型中审核弹窗显示了订单详情+用户信息+历史统计。通过时调钱包确认扣款，驳回时调钱包解冻退回。这个调用发生在"提现流程的审核环节"。

**财务补单审核通过时 → 钱包.补单入账**：B端管理员在人工修正→补单列表中审核通过一条补单后触发。

**财务加款审核通过时 → 钱包.人工加款**：B端管理员在人工修正→加款列表中审核通过一条加款后触发。需求原型中加款支持给多个钱包类型同时加款，所以一次审核通过可能触发对钱包的多次余额增加调用（中心钱包+奖励钱包+...）。

**财务减款审核通过时 → 钱包.人工减款**：与加款类似，审核通过后调钱包扣减指定钱包类型的余额。

**财务审核/修正前 → 钱包.查询余额**：提现审核弹窗需要展示用户余额信息，减款页面需要展示各钱包当前余额确保减款不超余额。这些场景在操作前需要先查询钱包余额。

#### 谁依赖财务

**钱包 → 财务：获取充值/提现方式配置 + 通道匹配**

已在钱包依赖关系部分详细描述。这里补充一点：从需求原型看，财务模块的"支付配置"页面是这些数据的唯一管理入口——充值方式的新增/编辑（币种、方式名称、图标、金额档位/范围、关联通道、排序、状态）和提现方式的新增/编辑（币种、方式名称、图标、金额范围、关联通道、排序、状态）都在财务的B端页面完成。

也就是说，C端用户看到的"可以用什么方式充值/提现"，完全由财务模块的B端配置决定。钱包只是消费这些配置，不能自己生产。

#### 财务模块内部的自依赖

从需求文档中可以梳理出财务模块内部也有清晰的依赖链：

**通道配置 → 支付配置 → 充值/提现记录 → 人工修正 → 数据统计**

- 先有代收/代付通道（配好费率/限额/密钥），才能在支付配置中创建充值方式/提现方式并关联通道
- 先有充值方式/提现方式，才会产生充值/提现订单记录
- 先有充值订单记录，才能对它做补单
- 先有订单数据积累，才有统计分析的意义

这条链路的前半段（通道→支付配置）是配置阶段的事情，后半段（记录→修正→统计）是运营阶段的事情。

### 9.4 三模块间的调用方向总结

以下用文字箭头描述所有跨模块的调用方向，每条都对应需求文档中的具体场景：

#### 币种配置 → 其他模块

**无。** 币种配置不主动调用钱包或财务的任何接口。它是纯粹的"被查询方"。这意味着币种配置是整个系统中**最独立、最稳定**的模块，可以最先开发、最先稳定。

#### 钱包 → 币种配置（3条）

**钱包 → 币种配置.查询币种列表**：C端用户切换币种时，钱包需要知道哪些币种是启用的、各自的展示信息。对应需求原型中"币种切换页面：列出所有币种及对应余额（带国旗icon标识和选中勾号）"。

**钱包 → 币种配置.查询入款汇率**：C端用户充值时需要展示和计算汇率。对应需求文档"只有BSB充值金额输入框下有对应汇率展示及兑换所得的金额"，以及创建充值订单时的金额换算。

**钱包 → 币种配置.查询出款汇率**：C端用户提现BSB时需要展示和计算汇率。对应需求文档"只有BSB提现金额输入框下有对应汇率展示"。兑换时也需要汇率（"统一展示口径：1 BSB = X 法币"）。

#### 钱包 → 财务（3条）

**钱包 → 财务.查询充值方式列表**：C端充值页面需要展示当前币种可用的充值方式和快捷金额。对应需求原型中各币种充值页面展示的支付方式列表（如BSB页面的"USDT（推荐）、越南银行、电子钱包扫码"）。

**钱包 → 财务.查询提现方式列表**：C端提现方式选择页需要展示当前币种可用的提现方式和到账时间。对应需求原型中各国提现方式列表（如印尼的"GoPay即时、OVO即时、DANA即时、银行卡T+1"）。

**钱包 → 财务.匹配通道并创建支付**：用户点击"去充值"后，钱包需要财务来完成通道轮询和三方支付创建。对应需求文档充值流程泳道图中的"轮询匹配通道→拉起支付"步骤。

#### 钱包 → KYC服务（1条）

**钱包 → KYC.查询认证状态及姓名**：用户点击提现时判断KYC状态（决定能用哪些提现方式），以及填写提现账户时自动填充KYC认证姓名。对应需求文档"未认证状态：点击提现触发弹窗"和"账户持有人姓名=KYC认证姓名（提现页自动填充已实名的姓名且不可编辑）"。

#### 财务 → 钱包（7条）

**财务 → 钱包.充值入账**：充值回调验证通过后。对应需求文档充值流程的最后一步"通道回调确认后，需更新钱包余额"。

**财务 → 钱包.确认提现扣款**：提现审核通过后。对应需求文档提现流程中"人工审核（财务）通过"后的扣款动作。

**财务 → 钱包.解冻退回余额**：提现审核驳回后。对应需求文档"审核驳回后需退回冻结金额"。

**财务 → 钱包.补单入账**：补单审核通过后。对应需求文档人工修正模块"补单审核通过后入账到用户钱包"。

**财务 → 钱包.人工加款**：加款审核通过后。对应需求原型加款弹窗中"指定用户/钱包类型/金额加款"，审核通过后执行。

**财务 → 钱包.人工减款**：减款审核通过后。对应需求原型减款弹窗中"指定用户/钱包类型/金额减款"，审核通过后执行。

**财务 → 钱包.查询用户余额**：审核和修正操作前。对应需求原型中提现审核弹窗展示用户信息，以及减款页面"显示当前余额，减款金额不超过余额"。

#### 财务 → 币种配置（1条）

**财务 → 币种配置.查询平台汇率**：生成记录和统计报表时换算基准币种金额。对应需求文档中充值/提现记录的"基准币种订单金额"字段，以及数据统计中的基准币种系列列。

### 9.5 依赖方向的不对称性

从上面的梳理可以看出一个明显的**不对称性**：

**调用方向**：钱包调财务 3 条，财务调钱包 7 条。财务对钱包的依赖远重于钱包对财务的依赖。

**依赖性质的差异**：

钱包调财务的 3 条都是**读取配置数据**——查充值方式、查提现方式、匹配通道。这些调用是"我需要知道有哪些选项"，数据由财务管理，钱包只是消费方。即使财务接口暂时不可用，钱包可以用缓存或兜底方案应对。

财务调钱包的 7 条都是**写入操作（余额变更）**——入账、扣款、冻结、解冻、加款、减款。这些调用是"我需要改你的数据"，改的是用户的真金白银。一旦调用失败，要么用户付了钱没到账，要么用户的钱被锁死。**这些调用不能失败、不能重复、不能丢失**。

这个不对称性决定了：
- 钱包模块的 RPC 对外接口（余额变更系列）的**可靠性要求远高于**财务模块的 RPC 对外接口（配置查询系列）
- 钱包的余额变更接口必须做**幂等设计**（防重复调用）
- 财务调钱包的链路需要**可靠消息/重试机制**（防调用丢失）

### 9.6 独立域与交叉域

从需求文档的角度看，三大模块的功能可以分为"完全在自己域内闭环"和"必须跨域协作"两种：

#### 独立域（不需要跨模块协作的功能）

**币种配置的所有功能都是独立域**：币种列表查询、币种编辑、基准币种设置、汇率日志、汇率定时更新——这些全部在币种配置模块内闭环完成，不需要调用钱包或财务的任何接口。这使得币种配置可以**完全独立开发和测试**。

**钱包的兑换流程是独立域（对财务而言）**：用户兑换法币→BSB 的整个流程（获取汇率→输入金额→执行兑换→生成记录）只需要调用币种配置的汇率接口，**完全不涉及财务模块**。没有通道、没有三方支付、没有审核。扣法币+加BSB在钱包模块内的同一个数据库事务中完成。

**钱包的记录查询是独立域**：充值/提现/兑换/奖励记录的列表查询和详情查询，都是读取钱包自己的数据库，不需要调用任何外部服务。

**钱包的提现账户管理是独立域**：保存和获取提现账户信息（银行卡/电子钱包账户），数据存在钱包自己的数据库里，不需要调用财务。（但保存时需要调KYC获取姓名做校验。）

**财务的通道配置是独立域（对钱包而言）**：代收/代付通道的列表查询、编辑、轮询参数配置，都是财务模块内部的B端操作，与钱包无关。

**财务的数据统计是独立域（对钱包而言）**：统计报表的查询和导出是财务模块读自己的数据做聚合计算，不需要调用钱包（基准币种换算调的是币种配置，不是钱包）。

#### 交叉域（必须跨模块协作的功能）

**充值创建是钱包×财务的交叉域**：钱包负责C端交互和订单管理，财务负责通道匹配和三方支付。两个模块必须配合完成一笔充值的创建。

**充值入账是财务×钱包的交叉域**：财务负责接收和验证通道回调，钱包负责将金额入账到用户余额。

**提现全流程是钱包×财务×KYC的三方交叉域**：钱包负责C端交互+冻结余额+创建订单，KYC负责认证状态判断，财务负责审核+代付+调钱包扣款/解冻。提现是依赖链最长的业务流程。

**人工修正是财务×钱包的交叉域**：财务负责补单/加款/减款的发起和审核流程，钱包负责审核通过后的余额变更执行。

### 9.7 从需求原型反推的页面级依赖

最后从产品原型的**页面跳转关系**反推接口依赖，这是前端和后端都需要清楚的：

**个人中心 → 钱包区域**
- 显示当前币种余额 → 需要：查询余额
- 点击币种切换 → 需要：获取币种列表及余额
- 点击"充值" → 需要：判断登录状态（游客只能USDT）
- 点击"记录" → 需要：跳转记录页

**充值页面的数据组装**
- 余额区 → 需要：当前币种余额
- 奖金选择区 → 需要：获取奖金活动列表
- 支付方式区 → 需要：获取充值方式列表（→ 财务RPC）
- 快捷金额区 → 需要：充值方式配置中的档位数据（→ 财务RPC）
- 汇率展示 → 需要：获取汇率（→ 币种配置RPC）
- 点击"去充值" → 需要：重复检查 → 创建充值订单（→ 财务RPC匹配通道）

这个页面的数据来自 **4 个数据源**（钱包自身 + 财务 + 币种配置 + 活动），是聚合度最高的页面。

**提现页面的数据组装**
- 点击提现 → 需要：KYC状态（→ KYC RPC）
- 提现方式列表 → 需要：获取提现方式（→ 财务RPC）+ 限额信息
- 账户信息区 → 需要：获取/保存提现账户 + KYC姓名
- 金额输入区 → 需要：提现限额查询 + 汇率（BSB时）
- 点击"确认提现" → 需要：校验 → 冻结 → 创建订单

这个页面的数据来自 **3 个数据源**（钱包自身 + 财务 + KYC），并且有严格的步骤顺序（先KYC判断 → 再选方式 → 再填账户 → 最后提交）。

**兑换页面的数据组装**
- 汇率展示 → 需要：获取汇率（→ 币种配置RPC）
- 余额展示 → 需要：当前币种余额
- 点击"兑换" → 需要：执行兑换

数据来自 **2 个数据源**（钱包自身 + 币种配置），是最简单的页面。

**B端币种配置页面**
- 币种列表 → 需要：币种列表查询
- 编辑弹窗 → 需要：币种编辑
- 汇率日志Tab → 需要：汇率日志列表

数据来自 **1 个数据源**（币种配置自身），完全自闭环。
