
**依赖方向总结**：

```
币种配置 ← 钱包模块 → 财务模块
  ↑                      ↕ (双向RPC)
  └──── 财务模块 ─────────┘
```

- 币种配置是**纯被依赖方**（只被调用，不调用别人）
- 钱包模块是**中间枢纽**（调用币种配置和财务，同时被财务调用）
- 财务模块与钱包模块是**双向依赖**（互相调用RPC）

## 二、三个模块的接口依赖关系

### 2.1 币种配置模块 — 纯基础设施，零外部依赖

```
币种配置模块接口（21个）
│
├── B端 HTTP（10个）────── 零外部依赖，完全独立
│   ├── 币种列表/编辑/启禁用/图标上传
│   ├── 基准币种设置/查询
│   ├── 汇率刷新/汇率日志查询/导出
│   └── 操作日志查询
│
├── C端 HTTP（3个）────── 零外部依赖
│   ├── 获取已启用币种列表
│   ├── 获取指定币种汇率
│   └── 获取基准币种信息
│
├── RPC 暴露（7个）────── 零外部依赖（纯提供方）
│   ├── GetPlatformRate
│   ├── ConvertToBaseCurrency
│   ├── GetCurrencyConfig
│   ├── GetAllEnabledCurrencies
│   ├── GetTransactionRate
│   ├── SnapshotBettingRate
│   └── IsCurrencyEnabled
│
└── 定时任务（1个）────── 依赖三方汇率API（外部）
    └── 汇率自动刷新
```

**结论**：币种配置模块**没有任何内部模块依赖**，只依赖外部三方汇率API。它是整个系统的地基，可以最先独立开发和部署。

### 2.2 多币种钱包模块 — 核心枢纽，多方向依赖

```
钱包模块接口依赖图
│
├── C端 HTTP ── 钱包核心（3个）
│   ├── 查询余额 ─────────── 依赖：币种配置.GetCurrencyConfig（精度格式）
│   ├── 获取可用币种 ──────── 依赖：币种配置.GetAllEnabledCurrencies
│   └── 切换币种 ─────────── 依赖：币种配置.IsCurrencyEnabled
│
├── C端 HTTP ── 充值（8个）
│   ├── 获取充值方式列表 ──── 依赖：财务.GetRechargeMethodList
│   ├── 创建USDT充值订单 ─── 依赖：币种配置.GetTransactionRate（锁汇）
│   ├── 获取USDT地址/链 ──── 无内部依赖（读本地配置或三方）
│   ├── 创建银行充值订单 ──── 依赖：币种配置.GetTransactionRate + 财务.通道匹配
│   ├── 创建扫码充值订单 ──── 依赖：币种配置.GetTransactionRate + 财务.通道匹配

│   ├── 获取额外奖金列表 ──── 无外部依赖（或依赖运营活动模块）
│   └── 查询充值订单状态 ──── 无外部依赖（查本地订单表）
│
├── C端 HTTP ── 兑换（2个）

│
├── C端 HTTP ── 提现（9个）

│   ├── 创建提现订单 ──────── 依赖：币种配置.GetTransactionRate + 财务.SubmitWithdrawOrder
│   │                              + 本模块.FreezeBalance（自调用冻结）
│   ├── 查询提现限额费率 ──── 依赖：财务.GetWithdrawConfig
│   ├── 获取/保存银行账户 ── 无外部依赖（读写本地表）
│   ├── 获取/保存电子钱包 ── 无外部依赖（读写本地表）
│   └── 获取/保存USDT地址 ── 无外部依赖（读写本地表）
│
├── C端 HTTP ── 记录（5个）
│   ├── 充值记录分页 ──────── 无外部依赖（查本地订单表）
│   ├── 提现记录分页 ──────── 无外部依赖（查本地订单表）
│   ├── 兑换记录分页 ──────── 无外部依赖（查本地订单表）
│   ├── 奖励记录分页 ──────── 无外部依赖（查本地订单表）
│   └── 记录详情 ─────────── 无外部依赖（查本地订单表）
│
└── RPC 暴露（6个）
    ├── GetBalance ─────────── 无外部依赖（读本地钱包表）
    ├── CreditWallet ────────── 无外部依赖（写本地钱包表+流水表）
    ├── DebitWallet ─────────── 无外部依赖（写本地钱包表+流水表）
    ├── FreezeBalance ────────── 无外部依赖（写本地钱包表）
    ├── UnfreezeBalance ──────── 无外部依赖（写本地钱包表）
    └── ConfirmFrozenDeduction ─ 无外部依赖（写本地钱包表+流水表）
```

### 2.3 财务管理模块 — 与钱包双向依赖

```
财务模块（他人负责）与钱包的交互关系
│
├── RPC 暴露给钱包调用（4个）
```

**充值链路依赖总结**：

| 步骤 | 依赖的前置接口 | 依赖方向 |
|------|-------------|---------|

| 支付回调入账 | 钱包.CreditWallet | 财务→钱包 |

**提现链路依赖总结**：

| 步骤 | 依赖的前置接口 | 依赖方向 |
|------|-------------|---------|
| 查询KYC状态 | 用户.GetKYCStatus | 钱包→用户 |
| 获取提现方式 | 财务.GetWithdrawMethodList | 钱包→财务 |
| 获取限额费率 | 财务.GetWithdrawConfig | 钱包→财务 |
| 获取出款汇率 | 币种配置.GetTransactionRate | 钱包→币种配置 |
| 冻结余额 | 本地FreezeBalance | 本地 |
| 提交财务审核 | 财务.SubmitWithdrawOrder | 钱包→财务 |
| 审核驳回/出款失败 | 钱包.UnfreezeBalance | 财务→钱包 |
| 出款成功 | 钱包.ConfirmFrozenDeduction | 财务→钱包 |

**关键特征**：提现是整个系统中**依赖链最长、耦合度最高**的业务链路，同时涉及4个模块（钱包+币种配置+财务+用户），且钱包与财务存在**双向异步调用**。

### 3.3 兑换链路（法币→BSB）

```

    │            │               │
    │ 确认兑换    │               │
    │───────────►│               │
    │            │ GetTransactionRate (锁汇)
    │            │──────────────►│
```

**兑换链路依赖总结**：只依赖币种配置模块的汇率接口，**不涉及财务模块和用户模块**，是最简洁的业务链路。

### 3.4 人工修正链路（财务B端操作→影响钱包）

```
[B端管理员]    [财务模块]        [钱包模块]
    │            │               │
    │ 创建补单/   │               │
    │ 加款/减款   │               │
    │───────────►│               │
    │            │               │
```

## 五、双向依赖深度分析（钱包 ↔ 财务）

这是整个系统中**最关键也最容易出问题**的依赖关系。

```
                    钱包模块                    财务模块
                 ┌────────────┐             ┌────────────┐
  C端充值请求 ──►│            │             │            │
                 │  创建订单   │             │            │
                 │  (本地写入)  │             │            │
                 │            │  获取充值方式  │            │
                 │  ─────────────────────►  │ 充值方式配置 │
                 │            │◄──────────  │            │
                 │            │             │            │
                 │            │             │ 接收通道回调  │◄── 三方通道
                 │            │             │            │
                 │            │ CreditWallet │            │
                 │  入账操作  │◄──────────── │ 确认入账    │
                 │            │             │            │
                 └────────────┘             └────────────┘

  C端提现请求 ──►│            │             │            │
                 │  冻结余额   │ 获取提现方式  │            │
                 │  ─────────────────────►  │ 提现方式配置 │
                 │            │◄──────────  │            │
                 │            │             │            │
                 │            │ Submit订单   │            │
                 │  ─────────────────────►  │ 进入审核    │
                 │            │             │            │
                 │            │             │  风控/财务   │
                 │            │             │  审核判断    │
                 │            │             │            │
                 │            │ Unfreeze    │            │  (驳回)
                 │  解冻归还  │◄──────────── │            │
                 │            │             │            │
                 │            │ Confirm     │            │  (出款成功)
                 │  确认扣除  │◄──────────── │            │
                 │            │             │            │
                 └────────────┘             └────────────┘

双向调用方向：
  ►  钱包 → 财务：获取配置（同步）、提交订单（同步）
  ◄  财务 → 钱包：入账/冻结/解冻/扣除（异步回调触发）
```

**双向依赖的处理策略**：

| 方向 | 性质 | 时机 | 处理方式 |
|------|------|------|---------|
| 钱包→财务 | 同步查询配置 | 用户操作时 | 正常RPC调用，可加缓存降级 |
| 钱包→财务 | 同步提交订单 | 提现创建时 | 正常RPC调用，失败需回滚冻结 |
| 财务→钱包 | 异步回调触发 | 通道回调/审核完成时 | RPC调用，必须保证**幂等性** |

**核心风险**：
1. 提现创建时先冻结后提交财务，如果提交财务失败，**必须回滚解冻**
2. 财务回调钱包CreditWallet/Unfreeze等，**必须幂等**（同一订单多次回调不能重复入账/解冻）
3. 两个模块的开发节奏不同步时，需要**Mock对方的RPC**才能调试

## 七、跨团队协调节点

由于财务模块是他人负责，需要在以下节点进行**跨团队对齐**：

```
时间线 ────────────────────────────────────────────────────►

    ❶ IDL定义对齐              ❷ Mock联调          ❸ 真实联调
    │                         │                   │
    │  需要双方同时确认：       │  各自用Mock：       │  真实环境：
    │                         │                   │
    │  钱包暴露的6个RPC        │  钱包Mock财务RPC   │  充值回调
    │  ├ GetBalance            │  财务Mock钱包RPC   │  提现审核流程
    │  ├ CreditWallet          │                   │  人工修正流程
    │  ├ DebitWallet            │                   │  数据统计
    │  ├ FreezeBalance          │                   │
    │  ├ UnfreezeBalance        │                   │
    │  └ ConfirmFrozenDeduction │                   │
    │                         │                   │
    │  财务暴露的4个RPC        │                   │
    │  ├ GetRechargeMethodList  │                   │
    │  ├ GetWithdrawMethodList  │                   │
    │  ├ GetWithdrawConfig      │                   │
    │  └ SubmitWithdrawOrder    │                   │
    │                         │                   │
    ▼                         ▼                   ▼
  越早越好                 各自开发阶段           集成测试阶段
```

**核心结论**：
- **IDL接口定义**是两个团队并行开发的前提，越早锁定越好
- 锁定后各自可以**Mock对方RPC**独立开发
- **充值入账**和**提现审核**是联调的重点，涉及异步回调和状态流转

## 八、依赖关系矩阵（速查表）

### 纵轴 = 调用方，横轴 = 被调用方

```
调用方 ＼ 被调用方 │ 币种配置   │ 钱包模块   │ 财务模块   │ 用户模块
─────────────────┼──────────┼──────────┼──────────┼──────────
币种配置          │    —     │    ✗     │    ✗     │    ✗
钱包模块          │   ★★★   │    —     │   ★★    │    ★
财务模块          │    ★     │   ★★★   │    —     │    ✗
─────────────────┴──────────┴──────────┴──────────┴──────────

★★★ = 强依赖（多个接口，核心链路）
★★  = 中依赖（几个接口，重要链路）
★   = 弱依赖（1-2个接口，非核心）
✗   = 无依赖
```

### 具体调用关系数量

| 调用方 → 被调用方 | RPC接口数 | 核心链路 |
|------------------|----------|---------|
| 钱包 → 币种配置 | 4~5个 | 所有涉及汇率和币种的操作 |
| 钱包 → 财务 | 4个 | 充值方式、提现方式/配置、提交提现审核 |
| 钱包 → 用户 | 2个 | KYC状态、用户信息（仅提现） |
| 财务 → 钱包 | 6个 | 入账、加减款、冻结/解冻/确认扣除 |
| 财务 → 币种配置 | 2个 | 汇率换算（统计报表） |

## 九、总结

### 一句话概括每个模块的依赖特征

| 模块 | 特征 | 一句话 |
|------|------|--------|
| 币种配置 | **纯被依赖，零外部依赖** | 地基，最先做，做完了大家都能用 |
| 钱包核心+RPC | **纯被依赖，弱外部依赖** | 承重墙，做完了财务模块就能调 |
| 钱包-兑换 | **单向依赖币种配置** | 最简单的业务链路，第一个可以跑通的功能 |
| 钱包-充值 | **依赖币种配置+财务配置** | 核心业务，受财务模块RPC就绪影响 |
| 钱包-提现 | **依赖最多：币种+财务+用户** | 最复杂的链路，最后集成联调 |
| 钱包-记录 | **纯本地查询，依赖订单数据** | 技术简单，但需要订单跑通才能测 |
| 财务全部 | **强依赖钱包RPC** | 他人负责，但钱包RPC是他们的硬前置 |

### 最关键的三个认知

1. **币种配置是一切的地基** — 没有币种和汇率，其他模块寸步难行
2. **钱包的6个RPC是财务模块的命脉** — 这6个接口的IDL定义要尽早锁定，是跨团队协作的卡点
3. **提现是耦合度最高的链路** — 涉及4个模块的异步双向调用，是系统联调的最大风险点

## 10.1 币种配置模块的依赖关系

### 币种配置不依赖任何内部模块

币种配置模块是整个系统的基础设施层。从需求文档来看，它的所有功能都是"自给自足"的：

- B端管理员配置币种信息（编辑/启禁用/上传图标） → 读写自己的币种配置表，不需要调用钱包或财务的任何接口
- B端设置基准币种（USDT，一次性不可修改） → 写入自己的基准币种表，纯本地操作
- B端手动刷新实时汇率 → 调用外部三方汇率API取均值，不涉及内部模块
- 汇率自动刷新定时任务 → 同上，调用三方API后根据阈值判断是否更新平台汇率，写入本地汇率日志表
- C端获取已启用币种列表 → 查自己的币种配置表
- C端获取指定币种汇率 → 查自己的汇率数据

**结论**：币种配置模块唯一的外部依赖是三方汇率API（外部服务），对内部模块零依赖。

### 币种配置被谁依赖

**钱包模块 → 币种配置**（强依赖，高频调用）：

- 钱包首页查询余额时 → 需要调用币种配置获取货币精度规则（法币2位小数/加密8位）和千分位符号，才能正确格式化金额展示。依据：需求文档3-3后台配置页面说明中的千分位符号、小数点符号字段
- 钱包首页展示可用币种列表 → 需要调用币种配置获取所有已启用的币种。依据：需求文档4-2币种切换规则中"切换币种"依赖于可用币种列表
- 用户创建充值订单时 → 需要调用币种配置获取当前平台入款汇率，用于跨币种充值的汇率换算（例如用USDT充BSB）。依据：需求文档4-1汇率应用场景中"充值：跨币种充值时使用平台入款汇率"
- 用户创建充值/兑换/提现订单时 → 需要调用币种配置获取锁汇汇率，确保用户操作期间汇率不变。依据：需求文档4-1汇率应用场景中投注场景的汇率记录逻辑，充值/提现同理
- 兑换预览页面 → 需要调用币种配置获取平台汇率，用于展示"1 BSB = ₫2,620"。依据：需求文档6兑换模块中的汇率展示
- 提现时 → 需要调用币种配置获取平台出款汇率，用于BSB提现的汇率换算。依据：需求文档4-1汇率应用场景中"提现：跨币种提现时使用平台出款汇率"

**财务模块 → 币种配置**（弱依赖，统计场景）：

- 财务模块的数据统计报表 → 需要调用币种配置将各币种金额折算为基准币种（USDT）进行汇总。依据：需求文档3-2平台基准币种中"所有币种金额在账务层需换算为基准币种再进行统计"
- 财务模块的充值记录/提现记录列表 → 有"基准币种订单金额""基准币种手续费"等字段，需要汇率换算。依据：需求文档2-3充值记录列表中第22-24列

## 10.2 多币种钱包模块的依赖关系

### 钱包 → 币种配置（单向强依赖）

钱包模块对币种配置的依赖贯穿所有业务场景，具体分为两类：

**第一类：展示类依赖（每次页面加载都要调用）**

- C端钱包首页加载 → 币种配置.获取已启用币种列表 + 币种配置.获取币种配置（精度/符号/图标）
- C端充值页面加载 → 币种配置.获取指定币种汇率（BSB充值需要展示"汇率：1BSB=0.1USDT"）
- C端兑换页面加载 → 币种配置.获取平台汇率（展示"1 BSB = ₫2,620"）
- C端提现页面加载（BSB提现） → 币种配置.获取平台出款汇率（展示"汇率：1 BSB = ฿3.50"）

**第二类：交易类依赖（创建订单时必须调用）**

- 创建USDT充值订单 → 币种配置.获取交易锁汇汇率（用USDT充BSB时，锁定当前汇率计算应充USDT数量）
- 创建银行转账/扫码充值订单 → 币种配置.获取交易锁汇汇率（BSB充值选择法币通道时的汇率换算）
- 创建兑换订单 → 币种配置.获取交易锁汇汇率（法币兑换BSB的锁汇）
- 创建提现订单 → 币种配置.获取交易锁汇汇率（BSB提现换算为法币/USDT的锁汇）

**依赖性质**：同步调用，币种配置RPC不可用时钱包模块的C端功能基本瘫痪。需要在钱包侧做币种/汇率信息的本地缓存作为降级手段。

### 钱包 → 财务模块（单向中依赖）

钱包模块在充值和提现两个业务场景中依赖财务模块提供的配置信息和审核入口。

**充值场景的依赖**：

- C端用户进入充值页面 → 钱包调用财务.获取充值方式列表 → 财务根据当前币种+地区返回可用的充值方式（如越南：银行转账；印尼：QRIS扫码+银行等）。依据：需求文档5-1充值方式总览中不同币种对应不同充值方式；财务需求文档2-2支付配置-1充值方式中充值方式按币种配置
- 如果财务模块的充值方式配置接口不可用 → 用户看不到充值方式列表 → 无法充值

**提现场景的依赖**：

- C端用户进入提现页面 → 钱包调用财务.获取提现方式列表 → 财务根据币种返回可用的提现方式（如越南：银行卡T+1；印尼：GoPay即时/OVO即时/银行T+1等）。依据：需求文档7-1提现模块中不同地区的提现方式列表；财务需求文档2-2支付配置-2提现方式
- C端展示提现限制信息 → 钱包调用财务.获取提现配置 → 获取单笔限额、单日限额、手续费率、T+N到账时间。依据：需求文档7-5提现限制中"后台可配：单笔限额、单日限额、手续费率、T+N到账时间"
- C端用户确认提现 → 钱包先冻结用户余额（本地操作） → 然后调用财务.提交提现订单 → 提现订单进入财务的风控审核→财务审核流程。依据：财务需求文档2-4提现记录-1提现流程

**依赖性质**：同步调用。财务模块的配置接口不可用时，充值和提现的C端入口不可用，但钱包余额查询、兑换、记录查看等功能不受影响。

### 钱包 → 用户模块（弱依赖，仅提现场景）

- C端用户点击提现 → 钱包调用用户.获取KYC认证状态 → 判断该用户是否已完成KYC认证。依据：需求文档7-1提现模块中"KYC为可选非强制，但作为解锁高阶提现渠道的必要前提。未认证仅USDT提现，认证后全渠道"
- C端用户填写提现账户 → 钱包调用用户.获取用户信息 → 自动填充"账户持有人"字段为KYC认证姓名（置灰不可编辑）。依据：需求文档7-1提现模块中"账户持有人姓名=KYC认证姓名"

**依赖性质**：同步调用，但仅影响提现功能。KYC接口不可用时，可降级为默认"未认证"状态，仅允许USDT提现。

### 钱包被谁依赖

**财务模块 → 钱包（强依赖，6个RPC接口）**：

这是财务模块对钱包模块最核心的依赖，贯穿财务模块的充值确认、提现审核、人工修正三大业务。

**充值入账场景**：

- 三方通道回调充值成功 → 财务模块验证金额一致 → 财务调用钱包.CreditWallet → 将到账金额加入用户中心钱包，活动赠送金额加入用户奖励钱包。依据：财务需求文档2-3充值记录-1充值流程中"订单状态改为支付成功，打款到对应用户上分"

**提现审核场景**：

- 用户提交提现 → 钱包冻结余额 → 提现订单进入财务审核
- 风控审核驳回 → 财务调用钱包.UnfreezeBalance → 解冻归还冻结金额到用户可用余额。依据：财务需求文档2-4提现记录-1提现流程中"不通过→解冻用户冻结的金额，退回钱包余额"
- 财务审核驳回 → 同上，调用钱包.UnfreezeBalance
- 代付通道出款失败 → 财务调用钱包.UnfreezeBalance → 解冻归还
- 代付通道出款成功 → 财务调用钱包.ConfirmFrozenDeduction → 最终扣除冻结资金（真正出账）

**人工修正场景**：

- B端管理员创建人工加款/减款单 → 弹窗需要展示用户当前各子钱包余额 → 财务调用钱包.GetBalance → 获取中心钱包/奖励钱包/主播钱包/代理钱包的各自余额。依据：财务需求文档2-5人工修正-2加款/-3减款的创建弹窗中展示"当前余额"列
- 加款审核通过 → 财务调用钱包.CreditWallet → 按4种子钱包分别加入对应金额。依据：财务需求文档2-5人工修正-2人工加款的审核流程
- 减款审核通过 → 财务调用钱包.DebitWallet → 按4种子钱包分别扣减对应金额（余额不足则实际扣款=当前余额）。依据：财务需求文档2-5人工修正-3人工减款的审核流程
- 补单审核通过 → 财务调用钱包.CreditWallet → 将补单金额加入中心钱包，补单赠送金额加入奖励钱包。依据：财务需求文档2-5人工修正-1充值补单的审核流程

## 10.3 财务管理模块的依赖关系

### 财务 → 钱包（强依赖）

财务模块对钱包模块的依赖集中在"执行资金操作"这件事上。财务模块自身负责审核决策和流程控制，但实际的钱加减冻结解冻都由钱包模块完成。可以理解为：**财务是"决策者"，钱包是"执行者"**。

具体调用场景：

- 充值成功入账 → 财务调用 → 钱包.CreditWallet
- 提现审核各节点（驳回/失败） → 财务调用 → 钱包.UnfreezeBalance
- 提现出款成功 → 财务调用 → 钱包.ConfirmFrozenDeduction
- 补单审核通过 → 财务调用 → 钱包.CreditWallet
- 人工加款审核通过 → 财务调用 → 钱包.CreditWallet
- 人工减款审核通过 → 财务调用 → 钱包.DebitWallet
- 人工加/减款创建时查余额 → 财务调用 → 钱包.GetBalance

如果钱包的这6个RPC不可用，财务模块的充值无法入账、提现无法完成审核流程、人工修正无法执行。可以说**钱包RPC是财务模块的命脉**。

### 财务 → 币种配置（弱依赖）

财务模块对币种配置的依赖主要在统计和记录展示场景：

- 充值记录列表中有"基准币种订单金额""基准币种活动优惠""基准币种手续费"三列 → 需要调用币种配置.获取平台汇率或转换为基准币种接口来计算。依据：财务需求文档2-3充值记录列表字段22-24
- 提现记录列表中有"基准币种订单金额""基准币种手续费"两列 → 同上。依据：财务需求文档2-4提现记录
- 人工修正中补单/加款/减款列表都有"基准币种xxx金额"字段 → 同上
- 数据统计的汇总卡片金额都以基准币种（USDT）展示 → 需要汇率换算

**依赖性质**：非核心链路，如果币种配置不可用，充值/提现/审核流程不受影响，只是统计金额暂时无法换算展示。

### 财务模块内部依赖

从需求文档来看，财务模块内部也有明确的前后依赖：

- 支付配置（充值方式） → 依赖通道配置（代收通道）：充值方式需要"关联通道"，如果没有代收通道数据，充值方式无法正确配置。依据：财务需求文档2-2支付配置-1充值方式中"关联通道：请点击选择（多选）"
- 支付配置（提现方式） → 依赖通道配置（代付通道）：同上，提现方式需要关联代付通道。依据：财务需求文档2-2支付配置-2提现方式
- 充值记录 → 依赖充值方式配置 + 代收通道配置：充值记录中有"通道名称""充值方式""充值费率"字段，来源于通道和支付配置
- 提现记录 → 依赖提现方式配置 + 代付通道配置：同上
- 充值补单 → 依赖充值记录：创建补单时需要输入原充值订单号并查询回显。依据：财务需求文档2-5人工修正-1充值补单的创建弹窗
- 数据统计 → 依赖充值/提现记录数据：统计是对已有记录的聚合计算

所以财务模块内部的开发顺序也是：通道配置 → 支付配置 → 充值/提现记录 → 人工修正 → 数据统计。

### 财务被谁依赖

**钱包 → 财务**（中依赖，4个RPC接口）：

如上文10.2节所述，钱包在充值和提现场景中依赖财务提供的配置信息和审核入口。

## 10.4 三大模块间的双向依赖详解

整个系统中只有一对双向依赖：**钱包 ↔ 财务**。

### 钱包 → 财务方向（同步，用户操作时触发）

触发时机：C端用户主动操作（进入充值页面、进入提现页面、提交提现）

- 用户进入充值页面 → 钱包调用财务获取充值方式列表
- 用户进入提现页面 → 钱包调用财务获取提现方式列表 + 提现限额配置
- 用户确认提现 → 钱包冻结余额后调用财务提交提现订单

这个方向的调用是**同步的、由C端用户操作触发的**。

### 财务 → 钱包方向（异步，后台事件触发）

触发时机：三方通道回调、B端管理员审核操作

- 三方通道回调充值成功 → 财务确认后调用钱包CreditWallet入账
- B端管理员风控审核驳回提现 → 财务调用钱包UnfreezeBalance解冻
- B端管理员财务审核通过提现 → 通道出款 → 出款成功回调 → 财务调用钱包ConfirmFrozenDeduction确认扣除
- B端管理员审核通过补单/加款/减款 → 财务调用钱包CreditWallet/DebitWallet

这个方向的调用是**异步的、由后台事件驱动的**。

### 双向依赖的关键特征

两个方向的调用在时间上是分离的，不存在同一个请求链路中A调B再B调A的循环调用：

- 充值：用户提交(钱包→财务获取方式) → 用户支付 → 时间间隔 → 通道回调(财务→钱包入账)
- 提现：用户提交(钱包→财务提交审核) → 时间间隔 → 管理员审核(财务→钱包冻结/解冻)

所以这不是技术上的循环依赖，而是**业务流程上的前后协作**。钱包负责"发起和执行"，财务负责"中间过程的决策和控制"。

---

## 10.5 按业务场景的完整依赖链描述

### 场景一：用户充值（以VND银行转账充BSB为例）

1. 用户在钱包首页切换到BSB币种 → 钱包 → 币种配置.获取已启用币种列表 → 确认BSB可用
2. 用户点击"充值" → 钱包 → 财务.获取充值方式列表(BSB) → 返回：USDT(推荐) / 越南银行 / 电子钱包扫码
3. 用户选择"越南银行"，输入金额1000BSB → 钱包 → 币种配置.获取平台入款汇率(BSB/VND) → 展示"汇率：1BSB=₫2,620，充值₫2,620,000"
4. 用户选择额外奖金"50%奖金，最低BSB 1000" → 钱包校验金额≥门槛 → 通过
5. 用户点击"去充值" → 钱包执行重复转账检查(本地查询) → 无重复
6. 钱包 → 币种配置.获取交易锁汇汇率 → 锁定当前汇率
7. 钱包创建充值订单(C+16位)，写入本地DB → 钱包 → 三方通道创建订单 → 获取跳转URL
8. 返回用户，用户跳转三方页面完成支付
9. （时间间隔：用户支付中）
10. 三方通道回调支付结果 → 财务模块接收回调 → 验证金额一致
11. 财务 → 钱包.CreditWallet → 1000BSB入中心钱包 + 500BSB奖金入奖励钱包
12. 充值完成，用户余额更新

**依赖链**：币种配置(前置) → 财务配置(前置) → 钱包创建订单 → 三方通道 → 财务确认 → 钱包入账

### 场景二：用户提现（以VND银行卡提现为例）

1. 用户在钱包首页切换到VND → 钱包 → 币种配置.确认VND启用
2. 用户点击"提现" → 钱包 → 用户.获取KYC状态 → 已认证 → 无弹窗直接进入
3. 钱包 → 财务.获取提现方式列表(VND) → 返回：银行卡T+1
4. 钱包 → 财务.获取提现配置(VND) → 返回：单日限额₫2,500,000，单笔最低₫2,500,000，手续费1%
5. 用户选择"银行卡提现"，首次填写：银行名称(下拉)、银行账号、账户持有人(自动填充KYC姓名，置灰)
6. 用户输入金额₫7,000,000 → 钱包校验：≤单日限额？≥单笔最低？→ 通过
7. 钱包展示：提现₫7,000,000，手续费₫70,000，到账₫6,930,000
8. 用户点击"确认提现" → 钱包 → 币种配置.获取交易锁汇汇率 → 锁定汇率
9. 钱包执行FreezeBalance → 冻结₫7,000,000（本地操作，从可用余额移入冻结余额）
10. 钱包 → 财务.提交提现订单(T+16位) → 订单进入待风控审核
11. 返回用户"提现进行中"弹窗
12. （时间间隔：等待审核）
13. B端风控审核通过 → 进入待财务审核
14. B端财务审核通过 → 财务轮询匹配代付通道 → 向通道发起出款
15. 通道出款成功回调 → 财务 → 钱包.ConfirmFrozenDeduction → 最终扣除₫7,000,000冻结资金
16. 提现完成

**依赖链**：币种配置(前置) → 用户KYC(前置) → 财务配置(前置) → 钱包冻结+提交 → 财务审核(异步) → 财务出款(异步) → 钱包确认扣除

### 场景三：法币兑换BSB（以THB兑换为例）

1. 用户在THB充值页面点击"兑换" → 进入兑换页面
2. 钱包 → 币种配置.获取平台汇率(BSB/THB) → 展示"1 BSB = ฿3.50"
3. 用户输入兑换金额฿1,000 → 钱包计算：兑换获得 BSB 285.71，赠送10% BSB 28.57，实际到账 BSB 314.28
4. 用户点击"兑换" → 钱包 → 币种配置.获取交易锁汇汇率 → 锁定汇率
5. 钱包执行：扣减THB中心钱包฿1,000 + 增加BSB中心钱包285.71 + 增加BSB奖励钱包28.57（赠送部分需完成X倍流水）
6. 全部本地DB操作，一个事务内完成
7. 返回"兑换成功，获得BSB 314.28"

**依赖链**：币种配置(前置) → 钱包本地执行。**不涉及财务模块和用户模块**，这是最短的业务链路。

### 场景四：人工加款（B端操作）

1. B端管理员进入财务管理-人工修正-人工加款页面
2. 输入用户编号，点击查询 → 财务 → 钱包.GetBalance → 返回该用户VND币种下：中心钱包₫5,000,000 / 奖励钱包₫1,000,000 / 主播钱包₫0 / 代理钱包₫0
3. 管理员选择加款类型"运营补偿"，填写中心钱包加款₫500,000，奖励钱包加款₫100,000

5. （另一位管理员审核，审核人≠提交人）
6. 审核通过 → 财务 → 钱包.CreditWallet(中心钱包, ₫500,000) + 钱包.CreditWallet(奖励钱包, ₫100,000)
7. 用户钱包余额更新

**依赖链**：钱包.GetBalance(查余额) → 财务创建审核单 → 财务审核 → 钱包.CreditWallet(执行加款)

### 场景五：充值补单（B端操作）

1. B端管理员进入人工修正-充值补单页面
2. 输入原充值订单号(C+16位)，点击查询 → 财务查询本地充值订单表 → 回显订单信息（金额/状态/用户信息）

**依赖链**：财务本地查原订单 → 财务创建审核单 → 财务审核 → 钱包.CreditWallet(执行入账)

## 10.6 独立可并行开发的模块对

根据以上分析，可以识别出以下可以并行开发的组合：

**币种配置全部功能** 和 **财务通道配置** → 完全独立，互不依赖，可以同时开工

**钱包提现账户管理（6个接口）** 和 **钱包记录查询（5个接口）** → 都是纯本地CRUD，不依赖任何外部模块

**钱包RPC暴露层（6个接口）** 和 **币种配置RPC暴露层（7个接口）** → 都是读写自己的DB，互不依赖

**钱包兑换功能** 和 **财务支付配置** → 兑换只依赖币种配置，不涉及财务；支付配置只依赖通道配置，不涉及钱包

## 10.7 不可并行、必须串行的依赖链

**链条一：币种配置RPC → 钱包C端全部业务接口**

原因：钱包的每一个C端操作都需要查询币种信息或汇率，如果币种配置的RPC没有就绪，钱包的C端接口无法通过测试。

**链条二：财务通道配置 → 财务支付配置 → 钱包充值/提现流程**

原因：支付配置中的充值方式/提现方式需要关联已存在的代收/代付通道；钱包获取充值方式/提现方式列表又依赖财务的支付配置接口。

**链条三：钱包RPC暴露层 → 财务充值入账/提现审核/人工修正**

原因：财务模块在充值回调、提现审核、人工修正时必须调用钱包RPC操作余额，钱包RPC是这些财务功能的硬前置。

**链条四：充值/提现/兑换订单产生 → 记录查询有数据**

原因：记录模块虽然技术上独立（纯查表），但没有订单数据就没有可查的记录，功能测试无意义。
