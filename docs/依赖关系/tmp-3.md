# ser-wallet 模块间依赖关系与接口链路分析

> 分析时间：2026-02-26
> 分析依据：需求文档 + 产品原型 + 工程分析记录
> 分析目的：搞清楚模块间联动的前后关系，为架构设计提供依据

---

## 一、参与者全景（谁和谁有关系）

```
┌─────────────────────────────────────────────────────────────────┐
│                        ser-wallet 服务内部                        │
│                                                                 │
│   ┌──────────────────┐          ┌──────────────────┐            │
│   │   币种配置模块    │─────────→│    钱包模块       │            │
│   │  (Currency Config)│ 提供币种  │   (Wallet Core)  │            │
│   │                  │ 和汇率    │                  │            │
│   └──────────────────┘          └──────────────────┘            │
│           ↑                        ↑          ↑                 │
└───────────┼────────────────────────┼──────────┼─────────────────┘
            │                        │          │
     ┌──────┴──────┐         ┌───────┴───┐ ┌────┴────────────┐
     │ 三方汇率API  │         │ 财务模块   │ │ 其他业务模块     │
     │ (3家取平均)  │         │(他人开发)  │ │投注/直播/活动    │
     └─────────────┘         └───────────┘ └─────────────────┘
                                   ↑
                             ┌─────┴─────┐
                             │ 三方支付通道│
                             │(代收/代付) │
                             └───────────┘
```

**参与者清单：**

| 参与者 | 角色 | 和钱包的关系 |
|--------|------|-------------|
| 币种配置 | 同服务内部模块 | 钱包的地基，提供币种和汇率数据 |
| 财务模块 | 他人开发的独立服务 | **双向依赖**：钱包调它获取支付配置，它调钱包操作余额 |
| ser-kyc | 已有服务(8个RPC方法) | 钱包调它查KYC状态（提现时） |
| ser-user | 已有服务(35个RPC方法) | 钱包调它取用户信息（KYC姓名） |
| ser-blog | 已有服务 | 钱包调它记审计日志 |
| ser-s3 | 已有服务(11个RPC方法) | 钱包调它上传币种图标 |
| 活动模块 | 待定 | 钱包调它取可用奖金活动；它调钱包发放活动返奖 |
| 投注模块 | 待定 | 它调钱包扣款和返奖 |
| 直播模块 | 待定 | 它调钱包发放主播收入 |
| 三方汇率API | 外部 | 币种配置定时拉取汇率 |
| 三方支付通道 | 外部 | 财务模块对接，和钱包无直接关系 |

## 二、启动依赖顺序（什么必须先于什么存在）

```
Layer 0  ─────────────────────────────────────────────────────
  基础设施（已有）
  │  ETCD、Redis、TiDB、ser-blog、ser-s3、ser-i18n
  │  这些是所有服务的前提，不讨论
  │
Layer 1  ─────────────────────────────────────────────────────
  币种配置初始化
  │  ① 导入币种基础数据（开发脚本导入）
  │  ② 设置基准币种（USDT，一次性）
  │  ③ 启动汇率定时拉取任务（3家API取平均）
  │
  │  这一层不依赖任何业务模块，是纯粹的基础数据准备
  │  没有币种数据，钱包的一切操作都无法进行
  │
Layer 2  ─────────────────────────────────────────────────────
  钱包核心能力就绪
  │  ④ 钱包表结构就绪（用户钱包、钱包流水等）
  │  ⑤ 余额操作接口就绪（上账/扣款/冻结/解冻/查询）
  │
  │  这一层依赖 Layer 1（需要知道有哪些币种）
  │  但不依赖财务模块，是钱包自己的核心能力
  │
Layer 3  ─────────────────────────────────────────────────────
  财务模块配置就绪（他人负责）
  │  ⑥ 通道配置（代收/代付通道、轮询规则）
  │  ⑦ 支付配置（充值方式+档位、提现方式+费率）
  │
  │  这一层依赖 Layer 1（需要币种数据关联）
  │  和 Layer 2 平行，互不阻塞
  │
Layer 4  ─────────────────────────────────────────────────────
  C端业务流程可用
  │  ⑧ 充值流程（依赖 Layer 1 汇率 + Layer 3 支付配置 + Layer 2 上账能力）
  │  ⑨ 提现流程（依赖 Layer 1 汇率 + Layer 3 支付配置 + Layer 2 冻结/扣款 + ser-kyc）
  │  ⑩ 兑换流程（依赖 Layer 1 汇率 + Layer 2 余额操作）
  │  ⑪ 记录查询（依赖 ⑧⑨⑩ 产生的数据）
```

**关键结论：**
- Layer 1（币种配置）是一切的根基，必须最先完成
- Layer 2（钱包核心）和 Layer 3（财务配置）可以**并行开发**，互不阻塞
- Layer 4（C端流程）需要 Layer 1+2+3 全部就绪才能联调
- 兑换流程不依赖财务模块，是钱包内部闭环，可以最早跑通

## 三、接口间的前置/后置/独立/耦合关系

### 3.1 币种配置模块内部

```
SetBaseCurrency ──→ 绝对前置，必须最先执行，且只能执行一次
       │
       ▼
ImportCurrencies ──→ 前置，导入币种基础数据（开发脚本）
       │
       ▼
StartRateJob ──→ 前置，启动汇率定时拉取（依赖币种数据存在）
       │
       ├──→ GetCurrencyList ──→ 独立查询，随时可调
       ├──→ GetExchangeRate ──→ 独立查询，依赖 RateJob 在跑
       ├──→ EditCurrency ──→ 独立操作，依赖币种存在
       ├──→ StatusCurrency ──→ 有后置影响（禁用币种会影响C端所有操作）
       └──→ PageExchangeRateLog ──→ 独立查询，依赖有汇率变更记录
```

**前置关系链：** SetBaseCurrency → ImportCurrencies → StartRateJob → 其他接口可用
**独立接口：** GetCurrencyList、GetExchangeRate、EditCurrency、PageExchangeRateLog（互相之间无依赖）
**有副作用的：** StatusCurrency（禁用币种会级联影响C端）

### 3.2 钱包核心 RPC 接口（供其他模块调用）

```
                    ┌──────────────┐
                    │  GetBalance  │ ← 纯查询，完全独立，任何时候可调
                    └──────────────┘

  ┌──────────────────────────────────────────────────────┐
  │              余额变更类（互相独立，各自闭环）           │
  │                                                      │
  │  CreditWallet ── 上账（充值成功/投注返奖）             │
  │  RewardCredit ── 奖励入账（活动返奖→奖励钱包）         │
  │  AnchorIncome ── 主播收入（→主播钱包）                 │
  │  ManualCredit ── 人工加款                             │
  │  SupplementCredit ── 充值补单                         │
  │                                                      │
  │  以上5个接口互相独立，都是"给钱包加钱"                  │
  │  不存在前后依赖关系                                    │
  └──────────────────────────────────────────────────────┘

  ┌──────────────────────────────────────────────────────┐
  │              扣款类（互相独立）                        │
  │                                                      │
  │  DebitWallet ── 直接扣款（消费/投注）                  │
  │  BetDebit ── 投注扣款（需记录中心/奖励比例）           │
  │  ManualDebit ── 人工减款                              │
  │                                                      │
  │  以上3个接口互相独立，都是"从钱包扣钱"                  │
  └──────────────────────────────────────────────────────┘

  ┌──────────────────────────────────────────────────────┐
  │              冻结类（有严格前后关系）                   │
  │                                                      │
  │  FreezeBalance ──→ 前置（提现申请时冻结）              │
  │       │                                              │
  │       ├──→ ConfirmDebit ── 后置（出款成功，扣除冻结）   │
  │       │                                              │
  │       └──→ UnfreezeBalance ── 后置（出款驳回，解冻退回）│
  │                                                      │
  │  ConfirmDebit 和 UnfreezeBalance 二选一，互斥          │
  │  都必须在 FreezeBalance 之后才能调用                    │
  └──────────────────────────────────────────────────────┘
```

**关键发现：**
- 大部分 RPC 接口是**互相独立**的，可以并行开发
- 唯一有严格前后关系的是**冻结三件套**（冻结→确认扣除/解冻退回）
- GetBalance 完全独立，是被其他所有流程依赖的基础查询

### 3.3 C端业务流程的依赖链

#### 充值流程依赖链

```
用户进入充值页
       │
       ▼
GetWalletHome ─────────────────── 依赖：GetCurrencyList（币种配置）
       │                                  GetBalance（钱包核心）
       ▼
GetDepositMethods ─────────────── 依赖：财务模块.GetPaymentConfig（充值方式+档位）
       │                                  GetCurrencyList（币种过滤）
       ▼
GetBonusActivities ────────────── 依赖：活动模块.GetAvailableBonuses
       │                          独立于充值方式，可并行加载
       ▼
CheckDuplicateDeposit ─────────── 依赖：已有订单数据（钱包自己的数据）
       │                          前置于 CreateDepositOrder
       ▼
CreateDepositOrder ────────────── 依赖：① GetDepositMethods 已返回可用方式
       │                                 ② CheckDuplicateDeposit 通过
       │                                 ③ GetExchangeRate（BSB充值时）
       │                                 ④ 财务模块.CreatePayOrder（创建通道订单）
       ▼
等待三方回调
       │
       ▼
财务模块收到回调 → 更新订单状态
       │
       ▼
财务模块调 CreditWallet ────────── 依赖：充值订单成功
                                  后置于整个充值流程
```

**充值流程接口调用时序图：**

```
  │                │                 │               │               │             │

  │                │─GetCurrencyList→│               │               │             │
  │                │←──币种+汇率─────│               │               │             │
  │                │─GetBalance────→(内部)           │               │             │
  │←──首页数据─────│                 │               │               │             │
  │                │                 │               │               │             │
  │─GetDepositMethods→│             │               │               │             │
  │                │──────────────────────GetPaymentConfig──→│       │             │
  │                │←─────────────────────充值方式+档位──────│       │             │
  │←─充值方式列表──│                 │               │               │             │
  │                │                 │               │               │             │
  │─GetBonusActivities→│            │               │               │             │
```

#### 提现流程依赖链

```
用户进入提现页
       │
       ▼
KYC状态检查 ──────────────────── 依赖：ser-kyc.KycQuery
       │
       ├── 未认证 → 弹窗（去认证 / 继续提现仅USDT）
       │
       ▼
GetWithdrawMethods ────────────── 依赖：① 财务模块.GetPaymentConfig（提现方式+费率）
       │                                ② KYC状态（决定展示哪些方式）
       │                                ③ 充值历史（决定可用提现方式：充法币→提法币）
       ▼
GetWithdrawAccount ────────────── 依赖：用户选择了提现方式
       │                          独立查询（读取之前保存的账户信息）
       ▼
SaveWithdrawAccount ───────────── 依赖：① ser-user.GetUserInfo（KYC姓名自动填充）
       │                                ② 首次提现时触发
       │                          后续提现跳过此步骤
       ▼
CreateWithdrawOrder ───────────── 依赖：① KYC状态已确认
       │                                ② 提现方式已选择
       │                                ③ 账户信息已填写
       │                                ④ GetExchangeRate（BSB提现时）
       │                                ⑤ 提现限额校验通过
       │                                ⑥ FreezeBalance（冻结钱包余额）
       │                                ⑦ 财务模块.CreateWithdrawOrder
       ▼
等待审核（风控→财务→出款）
       │
       ├── 出款成功 → 财务调 ConfirmDebit（扣除冻结金额）
       │
       └── 审核驳回 → 财务调 UnfreezeBalance（解冻退回）
```

**提现流程的特殊之处：** CreateWithdrawOrder 是依赖最多的接口，有 7 个前置条件。这意味着：
- 任何一个前置环节出问题，提现都创建不了
- 架构设计时需要考虑部分前置失败的降级策略

#### 兑换流程依赖链

```
用户进入兑换页
       │
       ▼
GetExchangeRate ──────────────── 依赖：币种配置（同服务内部调用）
       │                          不依赖财务模块！
       ▼
DoExchange ───────────────────── 依赖：① 汇率数据
       │                                ② 用户法币钱包有余额
       │                                ③ 赠送比例配置（来源待确认）
       │
       │  内部操作（同一事务）：
       │  ├── 扣减法币中心钱包
       │  ├── 增加BSB中心钱包（兑换部分）
       │  └── 增加BSB奖励钱包（赠送部分）
       │
       ▼
   兑换完成（3秒内）
```

**兑换是最独立的流程：**
- 不依赖财务模块
- 不依赖 KYC
- 不依赖三方通道
- 纯粹是钱包内部的余额操作 + 币种配置的汇率数据
- **最适合作为第一个跑通的端到端流程**

#### 记录查询依赖链

```
GetRecordList ────────────────── 依赖：数据已经存在（充值/提现/兑换/奖励操作已发生）
       │                          独立查询接口，无前置调用
       ▼
GetRecordDetail ──────────────── 依赖：GetRecordList 返回了具体记录ID
                                  后置于列表查询
```

**记录查询完全独立**，不影响其他任何接口，也不被其他接口阻塞。但需要确认数据来源：
- 兑换记录、奖励记录 → 钱包自己的数据
- 充值记录、提现记录 → 可能从财务模块读取，需确认

## 四、双向依赖分析（最需要注意的）

### 4.1 钱包 ⟷ 财务模块（双向耦合）

这是整个系统中**唯一的双向依赖**，也是架构设计中最需要注意的。

```
        钱包调财务（C端流程触发）              财务调钱包（异步回调/审核触发）
        ─────────────────────                ─────────────────────
正向:   GetPaymentConfig ──────→             ←────── CreditWallet        :反向
        （获取充值/提现方式）                        （充值成功上账）

正向:   CreatePayOrder ────────→             ←────── FreezeBalance       :反向
        （创建通道订单）                             （注：也可能是钱包自己冻结）

                                             ←────── ConfirmDebit        :反向
                                                     （提现出款成功）

                                             ←────── UnfreezeBalance     :反向
                                                     （提现驳回退回）

                                             ←────── ManualCredit        :反向
                                                     （人工加款A+）

                                             ←────── ManualDebit         :反向
                                                     （人工减款M+）

                                             ←────── SupplementCredit    :反向
                                                     （充值补单B+）
```

**正向（钱包→财务）：** 2-3个接口，发生在C端用户操作时（同步调用）
**反向（财务→钱包）：** 6-7个接口，发生在异步回调或B端审核时（异步/回调调用）

**双向依赖的影响：**
- RPC Client 层面：钱包需要注册 FinanceClient，财务需要注册 WalletClient
- 启动顺序：不能有循环启动依赖，两个服务可以独立启动
- 接口契约：需要和财务模块开发者**提前约定 IDL**，否则联调时会出问题

### 4.2 钱包 → 外部模块（单向依赖）

```
钱包 ──→ ser-kyc      只有提现流程调用，查KYC状态
钱包 ──→ ser-user     只有提现流程调用，获取KYC姓名
钱包 ──→ ser-blog     B端操作时调用，记审计日志
钱包 ──→ ser-s3       B端编辑币种时调用，上传图标
钱包 ──→ 活动模块     充值流程调用，获取可用奖金（待确认）
```

这些都是**单向依赖**，钱包调用它们但它们不调用钱包。风险较低。

### 4.3 外部模块 → 钱包（被调用）

```
财务模块   ──→ 钱包    上账/扣款/冻结/解冻/人工修正
投注模块   ──→ 钱包    投注扣款/返奖
直播模块   ──→ 钱包    主播收入入账
活动模块   ──→ 钱包    活动返奖到奖励钱包
```

这些模块会调用钱包的 RPC 接口。钱包作为**被调用方**，接口设计需要：
- 幂等性（防止重复调用导致重复操作）
- 原子性（单次调用内的多表操作需事务保证）
- 通用性（接口设计要能适配多种调用场景）

## 五、接口开发优先级排序（基于依赖关系推导）

### 第一批：地基（无外部依赖，可立即开发）

```
┌─────────────────────────────────────────────────────────────┐
│  币种配置                                                     │
│  ├── SetBaseCurrency        设置基准币种                      │
│  ├── PageCurrency           币种列表查询                      │
│  ├── EditCurrency           编辑币种                          │
│  ├── StatusCurrency         启用/禁用                         │
│  └── 汇率定时任务            拉取三方API + 偏差计算 + 更新     │
│                                                              │
│  钱包核心RPC                                                  │
│  ├── GetBalance             查询余额                          │
│  ├── CreditWallet           上账                              │
│  ├── DebitWallet            扣款                              │
│  ├── FreezeBalance          冻结                              │
│  ├── UnfreezeBalance        解冻                              │
│  └── ConfirmDebit           确认扣除                          │
│                                                              │
│  特点：这些接口不依赖任何外部模块，只需要数据库和Redis         │
│        开发完成后，钱包的核心能力就具备了                      │
└─────────────────────────────────────────────────────────────┘
```

### 第二批：内部闭环（只依赖第一批）

```
┌─────────────────────────────────────────────────────────────┐
```

## 七、关键链路总结

### 7.1 三条核心业务链路

**链路一：充值链路（最长，涉及方最多）**
```
C端 → GetWalletHome → GetDepositMethods → [GetBonusActivities] → CheckDuplicate
    → CreateDepositOrder → 财务.CreatePayOrder → 三方通道 → 回调
    → 财务.UpdateOrder → 财务调 CreditWallet → 充值完成
```
参与方：5个（钱包、财务、活动、三方通道、币种配置）
耗时：从用户点击到余额变更可能数分钟到数小时（链上确认）

**链路二：提现链路（前置条件最多）**
```
C端 → KYC检查 → GetWithdrawMethods → [Get/Save WithdrawAccount]
    → CreateWithdrawOrder → FreezeBalance → 财务.CreateWithdrawOrder
    → 风控审核 → 财务审核 → 出款 → 财务调 ConfirmDebit/Unfreeze → 提现完成/退回
```
参与方：5个（钱包、财务、KYC、用户、三方通道）
耗时：从申请到出款可能数小时到数天（T+N）

**链路三：兑换链路（最短，内部闭环）**
```
C端 → GetExchangeRate → DoExchange → 兑换完成（3秒内）
```
参与方：1个（钱包自己，含币种配置）
耗时：秒级

### 7.2 设计建议（基于依赖分析得出）

| 建议 | 理由 |
|------|------|
| 兑换流程最先开发联调 | 不依赖外部模块，是验证钱包核心能力的最佳切入点 |
| 钱包核心RPC接口提前定义IDL | 财务/投注/直播都要调，IDL越早定越好 |
| 与财务模块提前约定接口契约 | 双向依赖，需要共同确认请求/响应结构和调用时序 |
| 充值/提现记录的数据归属需尽早确认 | 直接影响表结构设计和查询方案 |
| 冻结三件套设计为状态机 | Freeze→Confirm/Unfreeze 有严格状态流转，适合状态机模式 |
| 奖金活动接口做降级设计 | 活动模块不可用时，充值流程应该还能走通（只是没有奖金可选） |

## 八、三模块依赖关系文字描述（钱包 / 币种配置 / 财务管理）

> 以下围绕需求文档和产品原型中的实际业务场景，用文字 + 箭头描述三大模块之间的依赖流向。
> 箭头含义：`A → B` 表示 A 依赖 B / A 调用 B / A 需要 B 提供数据或能力

---

### 8.1 币种配置 → 钱包（单向供给关系）

**币种配置是钱包的地基，钱包的一切操作都建立在币种数据之上。**

**（1）币种列表供给**

币种配置 → 钱包首页（币种切换列表）

需求依据：钱包需求文档4.2节"币种切换规则"——钱包首页可切换当前币种，切换币种即切换余额、充值/支付方式、记录列表数据。产品原型2-2展示了四种币种切换列表（BSB/IDR/VND/THB，各显示余额和国旗图标）。这意味着钱包首页需要从币种配置获取当前启用的币种列表（代码、名称、图标、货币符号、千分位规则、小数点规则），缺少这些数据钱包首页无法渲染。

**（2）汇率数据供给**

币种配置 → 钱包充值（BSB充值时展示汇率和换算金额）
币种配置 → 钱包兑换（汇率展示和兑换计算）
币种配置 → 钱包提现（BSB提现时展示汇率和换算金额）

需求依据：
- 充值：需求文档5.1节规则说明第4条"只有BSB充值金额输入框下有对应汇率展示及兑换所得的金额"。原型5-2至5-4展示了BSB选中不同支付方式时的汇率（1BSB=0.1USDT / 1BSB=d2,620 / 1BSB=Rp1,530）
- 兑换：需求文档6.2节"统一展示口径：1 BSB = X 法币"。原型8展示了三种法币兑换页面各自的汇率
- 提现：需求文档7.4节"只有BSB提现金额输入框下有对应汇率展示"。原型10-4展示了BSB银行卡提现时汇率1BSB=B3.50

具体来说：
- 充值使用**入款汇率**（币种配置需求4.2节：入款汇率 = 实时汇率 × (1 + 汇率浮动%)）
- 提现使用**出款汇率**（币种配置需求4.2节：出款汇率 = 实时汇率 × (1 - 汇率浮动%)）
- 兑换使用**平台汇率**（实时汇率取平均后根据偏差阈值更新的汇率）

**（3）金额展示规则供给**

币种配置 → 钱包所有金额展示

需求依据：钱包需求文档4.1节明确了每种币种的千分位规则和小数点规则（VND/IDR整数、THB/BSB两位小数）。币种配置需求3.3节的后台配置字段包括"千分位符号""小数点符号"。币种配置需求4.3节规定法定货币保留2位小数、加密货币保留8位小数。钱包在展示任何金额时都需要根据这些配置规则来格式化。

**（4）币种状态约束**

币种配置（禁用币种）→ 钱包（该币种下所有功能不可用）

需求依据：币种配置需求3.3节"禁用后C端不可使用/充值/提现，有余额时需弹提示"。这意味着钱包在执行任何操作前，需要校验当前操作币种的启用状态。如果B端管理员禁用了某个币种，钱包要做降级处理。

**小结：** 币种配置对钱包是**纯供给关系**，币种配置不需要钱包的任何数据。反过来，钱包的几乎每一个接口都依赖币种配置——没有币种数据，钱包无法展示余额、无法充值、无法提现、无法兑换。

### 8.2 钱包 → 财务管理（钱包调用财务的场景）

**钱包在C端业务流程中需要调用财务模块获取支付配置和创建通道订单。**

**（1）充值方式获取**

钱包（C端充值页）→ 财务管理（支付配置-充值方式）

需求依据：财务管理需求2.21节"充值方式"描述了充值方式的后台配置（币种/充值方式/支付图标/充值档位/充值金额范围/关联通道/排序/状态）。产品原型"支付配置-1"展示了VND/IDR下的多种充值方式列表。钱包C端的充值页（原型3-1至3-4）展示的充值方式（USDT/越南银行/电子钱包扫码等）和档位金额，**数据源完全来自财务模块的支付配置**。钱包自己不维护充值方式数据。

**（2）提现方式获取**

钱包（C端提现页）→ 财务管理（支付配置-提现方式）

需求依据：财务管理需求2.22节"提现方式"描述了提现方式配置（币种/提现方式/提现图标/提现费率/大额提现费率/关联通道/排序/状态）。钱包C端提现页（原型9-4、10-1至10-4）展示的提现方式和手续费信息，数据源来自财务模块。例如原型9-4展示"即时到账，每日限额USDT 10,000，单笔最低USDT 10，手续费1%"，这些配置由财务模块管理。

**（3）充值订单创建**

钱包（CreateDepositOrder）→ 财务管理（通道匹配 + 创建通道订单）

需求依据：财务管理需求2.3节"充值流程"三泳道图描述——服务端：轮询匹配通道 → 创建平台订单。需求2.13节"通道轮询规则"明确了匹配逻辑（成功率系数/时间系数/预警系数综合评分排序）。当C端用户点击"去充值"时，钱包需要把订单信息传给财务模块，由财务模块负责：选择通道 → 创建通道订单 → 返回支付信息（三方承载页URL或USDT收款地址）。钱包不直接和三方支付通道交互。

**（4）提现订单创建**

钱包（CreateWithdrawOrder）→ 财务管理（创建代付订单）

需求依据：财务管理需求2.4节"提现流程"三泳道图——服务端：创建订单(待审核) → 风控审核 → 财务审核 → 出款(自动/人工)。当C端用户确认提现时，钱包需要：先冻结用户余额（自己做），然后把提现订单信息传给财务模块，由财务模块负责后续的审核和出款流程。

**小结：** 钱包调用财务的场景集中在**两个时间点**：
- 进入充值/提现页时 → 获取可用方式和配置（读操作）
- 用户确认充值/提现时 → 创建通道订单（写操作）

这些调用都是**同步调用**，发生在C端用户操作的实时链路上，对响应时间敏感。

---

### 8.3 财务管理 → 钱包（财务调用钱包的场景）

**财务模块在异步回调和B端审核操作后，需要调用钱包来实际变更用户余额。**

**（1）充值成功上账**

财务管理（充值回调处理）→ 钱包（CreditWallet 上账）

需求依据：财务管理需求2.3节充值流程——通道回传支付结果 → 服务端更新状态。钱包需求文档5.2节（USDT充值）"达到确认数后自动上账"，5.3节（银行转账）"支付状态回传平台 → 平台更新订单状态"。当三方通道回调告知支付成功后，财务模块处理完订单状态更新，需要调用钱包的上账接口给用户的中心钱包增加余额。这是充值流程的最后一步。

**（2）提现出款成功 → 确认扣除**

财务管理（出款成功）→ 钱包（ConfirmDebit 确认扣除冻结金额）

需求依据：财务管理需求2.4节提现流程——出款成功后需更新订单状态。钱包在创建提现订单时已经冻结了相应金额（FreezeBalance），当财务模块确认出款成功后，需要调用钱包的ConfirmDebit接口，将冻结金额正式扣除。

**（3）提现审核驳回 → 解冻退回**

财务管理（风控驳回/财务驳回）→ 钱包（UnfreezeBalance 解冻退回）

需求依据：财务管理需求2.4节提现流程——订单状态包含"风控驳回""财务驳回"。产品原型"提现记录-3"展示了审核弹窗（通过/驳回两态）。当提现被驳回时，之前冻结的金额需要退回给用户，财务模块调用钱包的UnfreezeBalance接口执行解冻。

**（4）充值补单**

财务管理（充值补单审核通过）→ 钱包（SupplementCredit 补单上账）

需求依据：财务管理需求2.51节"充值补单"——审核通过即"修正金额并入流水"。产品原型"人工修正-1至4"展示了补单的完整流程。当充值金额有误需要补差时，财务模块创建补单订单（B+前缀），双人审核通过后，调用钱包的补单接口补充差额到用户中心钱包。

**（5）人工加款**

财务管理（人工加款审核通过）→ 钱包（ManualCredit 加款到指定钱包）

需求依据：财务管理需求2.52节"人工加款"——加款类型包括运营补偿/运营奖励/财务修正/测试加款，"可同时向中心钱包/奖励钱包/代理钱包/主播钱包加款"，"如用户下没有该币种钱包，系统需自动创建"。产品原型"人工修正-5至7"展示了四个钱包分别加款的界面。审核通过后，财务模块调用钱包的加款接口，钱包需要根据传入的钱包类型分别操作。

**（6）人工减款**

财务管理（人工减款审核通过）→ 钱包（ManualDebit 从钱包扣款）

需求依据：财务管理需求2.53节"人工减款"——减款类型包括异常追回/风控处罚/财务冲正/费用调整，"当用户钱包余额不足时，从最高金额的钱包开始依次扣减""实际减款金额可能小于设定的减款金额"。产品原型"人工修正-8至10"展示了减款流程。审核通过后，财务模块调用钱包的减款接口，钱包需要实现跨钱包类型的级联扣减逻辑。

**小结：** 财务调用钱包的场景有**6类**，全部是**异步触发**——要么是三方通道回调后触发，要么是B端人工审核通过后触发。和钱包调财务的同步场景不同，这些调用不在C端用户的实时链路上，但对**数据一致性**要求极高（不能漏账、不能重复记账）。

### 8.4 三模块在具体业务场景中的协作流向

以下按每个C端业务场景，描述三个模块的完整协作链路。

#### 场景一：用户充值（以VND银行转账充值为例）

```
第1步：用户打开钱包首页，选择VND币种
  钱包 → 币种配置：获取币种列表（代码/名称/图标/货币符号/格式化规则）
  钱包内部：查询VND中心钱包余额 + VND奖励钱包余额，计算资金钱包余额

第2步：用户点击充值，进入充值页
  钱包 → 财务管理：获取VND下可用充值方式+档位（越南银行转账、档位列表、金额范围）
  注意：充值方式的排序、图标、档位全部由财务模块的支付配置决定

第3步：用户选择金额，点击"去充值"
  钱包内部：校验重复转账规则（同用户+同币种+同金额，查自己的订单数据）
  钱包 → 财务管理：传递订单信息（用户ID/币种/金额/充值方式），财务负责匹配通道+创建通道订单
  财务管理 → 三方通道：创建代收订单
  三方通道 → 财务管理：返回支付信息（三方承载页URL）
  财务管理 → 钱包：返回订单创建结果
  钱包 → C端：展示三方承载页（用户去三方页面付款）

第4步：用户在三方页面完成支付
  三方通道 → 财务管理：回调支付结果
  财务管理内部：更新充值订单状态为"支付成功"
  财务管理 → 钱包：调用CreditWallet，传入（用户ID/币种VND/金额/订单号）
  钱包内部：VND中心钱包余额 += 充值金额，写入钱包流水
```

关键依赖总结：
- 币种配置在第1步提供基础数据（**前置**，无此数据页面无法渲染）
- 财务管理在第2步提供充值方式（**前置**，无此数据用户不知道怎么充）
- 财务管理在第3步负责通道匹配和订单创建（**同步协作**）
- 财务管理在第4步回调时调用钱包上账（**异步后置**）

#### 场景二：用户充值BSB（涉及汇率换算）

```
第1步：用户在BSB币种下进入充值页
  钱包 → 币种配置：获取币种列表 + BSB与各法币的入款汇率
  钱包 → 财务管理：获取BSB下可用充值方式（USDT/越南银行/电子钱包等）

第2步：用户选择"越南银行"支付方式
  钱包 → 币种配置：获取BSB对VND的入款汇率（如 1BSB = d2,620）
  页面展示：充值 d262,000 ≈ BSB 100.00

第3步：用户确认充值
  钱包内部：记录入款汇率快照（锁定当时汇率，防止结算时汇率变动）
  钱包 → 财务管理：创建VND充值订单（实际支付币种是VND）

第4步：充值成功回调
  财务管理 → 钱包：调用CreditWallet，上账到BSB中心钱包
  钱包内部：按锁定的入款汇率换算，BSB中心钱包余额 += 100.00 BSB
```

特殊依赖：这个场景下**币种配置的入款汇率**在第2步和第3步都被使用——第2步用于展示，第3步用于计算锁定。如果汇率定时任务没有正常运行，BSB充值将无法进行。

#### 场景三：用户法币兑换BSB

```
第1步：用户在VND币种下点击兑换图标
  钱包 → 币种配置：获取VND对BSB的平台汇率（如 1BSB = d2,620）

第2步：用户输入兑换金额（如 d2,620,000）
  钱包内部：计算兑换获得BSB = 2,620,000 / 2,620 = 1,000 BSB
  钱包内部：计算赠送BSB = 1,000 × 10% = 100 BSB
  钱包内部：实际到账 = 1,000 + 100 = 1,100 BSB

第3步：用户确认兑换
  钱包内部（同一事务）：
    VND中心钱包余额 -= 2,620,000
    BSB中心钱包余额 += 1,000
    BSB奖励钱包余额 += 100（赠送部分，需完成X倍流水）
  写入兑换流水记录
```

关键依赖总结：
- 币种配置在第1步提供平台汇率（**唯一外部依赖**）
- **完全不依赖财务管理模块**——不需要支付配置、不需要通道匹配、不需要创建通道订单
- 这是三个C端场景中依赖最少的，纯粹在钱包服务内部闭环完成

#### 场景四：用户提现（以IDR银行卡提现为例）

```
第1步：用户在IDR币种下点击提现
  钱包 → ser-kyc：查询用户KYC认证状态
    → 未认证：弹窗提供"去认证"和"继续提现（仅USDT）"
    → 已认证：进入提现方式选择页

第2步：用户选择银行卡提现
  钱包 → 财务管理：获取IDR下可用提现方式+费率+限额
  钱包 → 币种配置：获取IDR金额格式化规则
  钱包内部：查询用户历史提现账户信息（首次需填写，后续复用）

第3步：用户填写/确认账户信息
  钱包 → ser-user：获取KYC认证姓名（自动填充"账户持有人"，不可编辑）
  钱包内部：保存提现账户信息（银行名称/账号/持有人/开户网点）

第4步：用户输入提现金额，确认提现
  钱包内部：校验单笔限额、单日限额（限额配置来自财务）
  钱包内部：FreezeBalance — 冻结IDR中心钱包对应金额
  钱包 → 财务管理：创建提现订单（传入用户ID/币种IDR/金额/账户信息/冻结流水号）

第5步：后台审核流程（B端）
  财务管理内部：风控审核 → 财务审核 → 出款
    → 出款成功：财务管理 → 钱包（ConfirmDebit，扣除冻结金额，IDR中心钱包正式减少）
    → 审核驳回：财务管理 → 钱包（UnfreezeBalance，解冻退回，IDR中心钱包恢复）
```

关键依赖总结：
- ser-kyc 在第1步提供认证状态（**前置门槛**，决定可用提现渠道）
- 财务管理在第2步提供提现方式和费率（**前置**）
- 币种配置在第2步提供格式化规则（**前置**）
- ser-user 在第3步提供KYC姓名（**前置**，账户持有人必须=KYC姓名）
- 财务管理在第4步接收提现订单（**同步协作**）
- 财务管理在第5步回调钱包确认或解冻（**异步后置**）
- 这是所有场景中**依赖链最长、参与方最多**的流程

#### 场景五：B端人工修正（加款/减款/补单）

```
人工加款流程：
  财务管理B端：操作员创建加款单（选择用户/币种/钱包类型/金额/类型）
  财务管理内部：另一名操作员审核通过（双人审核）
  财务管理 → 钱包：ManualCredit（用户ID/币种/钱包类型/金额）
  钱包内部：
    → 如果该用户没有指定币种的钱包 → 自动创建钱包
    → 指定钱包余额 += 金额
    → 写入流水（A+订单号前缀）

人工减款流程：
  财务管理B端：操作员创建减款单（选择用户/币种/金额/类型）
  财务管理内部：另一名操作员审核通过（双人审核）
  财务管理 → 钱包：ManualDebit（用户ID/币种/金额）
  钱包内部：
    → 从最高余额的钱包开始依次扣减（中心→奖励→主播→代理）
    → 余额不足时实际减款可能小于设定金额
    → 写入流水（M+订单号前缀）

充值补单流程：
  财务管理B端：操作员输入原充值订单号 → 查询到原订单信息
  财务管理B端：填写补单金额/赠送金额/稽核倍数
  财务管理内部：另一名操作员审核通过
  财务管理 → 钱包：SupplementCredit（用户ID/币种/补单金额/赠送金额）
  钱包内部：
    → 中心钱包余额 += 补单金额
    → 奖励钱包余额 += 赠送金额（需完成稽核流水）
    → 写入流水（B+订单号前缀）
```

关键依赖总结：
- 人工修正场景中，**主动权完全在财务管理模块**，钱包只是被调用方
- 钱包不需要调币种配置（因为财务模块已经确定了币种）
- 钱包不需要调任何外部服务，只需要操作自己的余额和流水
- 但钱包需要实现**特殊逻辑**：自动创建钱包（加款）、级联扣减（减款）

### 8.5 三模块依赖关系全景总结

#### 依赖方向汇总

```
币种配置 ——→ 钱包：单向供给（5条依赖线）
  ├── 币种列表（代码/名称/图标/符号/格式规则）→ 钱包首页、充值页、提现页、记录页
  ├── 入款汇率 → 充值（BSB充值时换算）
  ├── 出款汇率 → 提现（BSB提现时换算）
  ├── 平台汇率 → 兑换（法币→BSB换算）
  └── 币种状态 → 钱包所有操作（禁用时阻断）

钱包 ——→ 财务管理：C端流程触发（4条依赖线）
  ├── 获取充值方式+档位 → 充值页展示
  ├── 获取提现方式+费率+限额 → 提现页展示
  ├── 创建充值通道订单 → 充值确认时
  └── 创建提现订单 → 提现确认时

财务管理 ——→ 钱包：异步回调/审核触发（6条依赖线）
  ├── 充值成功 → CreditWallet（上账到中心钱包）
  ├── 出款成功 → ConfirmDebit（扣除冻结金额）
  ├── 审核驳回 → UnfreezeBalance（解冻退回）
  ├── 充值补单 → SupplementCredit（补差额+赠送）
  ├── 人工加款 → ManualCredit（加到指定钱包类型）
  └── 人工减款 → ManualDebit（级联扣减）
```

#### 依赖性质对比

| 依赖方向 | 性质 | 触发时机 | 调用方式 | 频率 | 核心关注点 |
|---------|------|---------|---------|------|-----------|
| 币种配置→钱包 | 数据供给 | 页面加载时 | 同服务内部调用 | 极高（每次打开页面） | 缓存策略，避免每次都查库 |
| 钱包→财务 | 业务协作 | 用户操作时 | RPC同步调用 | 中（充值/提现操作） | 响应速度，超时处理 |
| 财务→钱包 | 结果通知 | 回调/审核后 | RPC异步调用 | 低（订单级别） | 幂等性，数据一致性 |

#### 各模块的"给"与"要"

**币种配置（纯输出者）：**
- 给出去的：币种数据、汇率数据、格式化规则、币种状态
- 要别人的：只需要三方汇率API的数据（外部），不依赖钱包和财务的任何数据
- 定位：**基础设施层**，类似于配置中心

**钱包（中间枢纽）：**
- 给出去的：余额查询、余额变更（上账/扣款/冻结/解冻/确认）能力
- 要别人的：币种配置的一切数据、财务模块的支付配置和通道能力、KYC状态、用户信息
- 定位：**业务核心层**，承上（接受配置数据）启下（提供余额操作能力给所有业务模块）

**财务管理（业务合作者）：**
- 给出去的：支付配置（充值/提现方式）、通道能力（创建订单/匹配通道）、审核结果
- 要别人的：钱包的余额操作接口（CreditWallet/ConfirmDebit/UnfreezeBalance/Manual系列）
- 定位：**业务协作层**，管理资金进出的通道侧

#### 开发联调的依赖结论

```
第一阶段（可独立开发，无需联调）：
  币种配置全部功能 → 不依赖任何模块
  钱包核心RPC接口 → 只依赖币种配置（同服务内部）
  兑换流程 → 只依赖币种配置 + 钱包核心（同服务内部）

第二阶段（需要钱包和财务约定接口契约，可各自Mock开发）：
  钱包侧 Mock 财务接口 → 开发充值页、提现页的前端逻辑和本地流程
  财务侧 Mock 钱包接口 → 开发通道配置、支付配置、订单管理

第三阶段（必须联调）：
  充值全链路 → 钱包 + 财务 + 三方通道 三方联调
  提现全链路 → 钱包 + 财务 + KYC + 三方通道 四方联调
  人工修正 → 财务 + 钱包 双方联调
```
