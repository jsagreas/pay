## 三、调用方向分类（谁调谁）

### 3.1 单向依赖（A → B，B 不调 A）

```
币种配置 ──────► 钱包核心
   (钱包读取币种/汇率数据，币种配置不关心钱包)

钱包 ──────► ser-kyc
   (提现前查KYC状态，ser-kyc 不调钱包)

钱包 ──────► ser-user
   (获取用户信息做姓名校验，ser-user 不调钱包)

钱包 ──────► 三方汇率API
   (定时拉汇率，单向)

游戏/直播/礼物模块 ──────► 钱包RPC
   (投注扣款/返奖入账/打赏扣款，钱包不主动调它们)
```

### 3.2 双向依赖（A ↔ B，互相调用）

```
钱包 ◄──────► 财务管理
  │                │
  │ 钱包调财务:     │ 财务调钱包:
  │ · 获取支付方式   │ · CreditWallet (充值入账/补单/加款)
  │ · 通道匹配      │ · DebitWallet (减款)
  │ · 发起代付      │ · FreezeBalance (提现冻结)
  │                │ · UnfreezeBalance (驳回解冻)
  │                │ · DeductFrozenAmount (出款成功)
  │                │ · QueryWalletBalance (查余额)
  └────────────────┘
```

**这是最关键的双向依赖，也是架构设计时最需要注意的地方。**

解决双向依赖的方式：两个服务之间通过 RPC 接口解耦。钱包定义自己的 thrift IDL，财务定义自己的 thrift IDL，各自通过对方的 RPC Client 调用。不直接依赖对方的内部实现。

### 3.3 内部依赖（同服务内部模块间）

```
ser-wallet 服务内部:

币种配置模块 ──────► 钱包核心模块
  (同一个服务内部，不走RPC，直接函数调用)
  (汇率计算、币种校验、精度格式化 等直接调用币种配置的 service 层方法)
```

## 四、接口前置/后置关系详解

### 4.1 前置接口（必须先存在，其他接口才能工作）

这些接口是"地基"，如果没有它们，后续接口无法正常运行：

```
                    ┌──────────────────────┐
                    │  第一优先级：绝对前置   │
                    │  (没有这些，一切免谈)   │
                    └──────────┬───────────┘
                               │
            ┌──────────────────┼──────────────────┐
            ▼                  ▼                  ▼
    ┌──────────────┐  ┌──────────────┐  ┌───────────────┐
    │ SetBaseCurrency│  │ 币种数据初始化 │  │ 汇率定时任务   │
    │ (设基准币种)   │  │ (币种CRUD)    │  │ (拉取+计算)   │
    │              │  │              │  │               │
    │ 只能执行一次  │  │ 至少启用1个   │  │ 启动后持续运行  │
    │ 后续不可更改  │  │ 币种才能充提  │  │ 否则汇率为空   │
    └──────────────┘  └──────────────┘  └───────────────┘
           │                 │                  │
           └─────────────────┼──────────────────┘
                             │
                             ▼
                    全部后续接口才能正常工作
```

**前置接口清单：**

| 优先序 | 接口 | 为什么是前置 | 依赖它的后续接口 |
|--------|------|------------|----------------|
| **P0** | SetBaseCurrency | 基准币种是所有核算换算的"度量衡"，不设就没法统计对账 | ConvertAmount、所有涉及基准币种换算的统计 |
| **P0** | 币种 CRUD（GetCurrencyList / EditCurrency / ToggleCurrencyStatus） | 没有可用币种，钱包无法创建，充提无法进行 | 所有钱包操作 |
| **P0** | 汇率定时更新任务（ExchangeRateUpdate） | 没有汇率数据，充值/提现/兑换的金额换算全部失效 | CreateDepositOrder、CreateExchange、CreateWithdrawOrder |
| **P1** | 钱包账户初始化 | 用户注册时需要创建各币种的钱包账户 | 所有余额操作 |
| **P1** | 6个核心 RPC（Credit/Debit/Freeze/Unfreeze/DeductFrozen/QueryBalance） | 财务模块的充值入账、提现冻结、人工修正全部依赖这些 | 财务管理模块的所有资金操作 |
| **P2** | 财务：支付方式配置 | 钱包 C 端获取充值/提现方式列表依赖此数据 | GetDepositMethods、GetWithdrawMethods |
| **P2** | 财务：通道配置 | 创建充值订单需要匹配通道 | CreateDepositOrder |

### 4.2 后置接口（依赖前置接口的结果才能工作）

| 后置接口 | 它依赖什么前置 | 依赖方式 |
|---------|--------------|---------|
| GetWalletOverview | 币种数据 + 钱包账户 | 读币种列表确定展示哪些，读余额 |
| GetDepositMethods | 币种启用状态 + 财务支付方式配置 | 只展示已启用币种的已配置充值方式 |
| CreateDepositOrder | 汇率 + 财务通道匹配 + 重复转账检测 | 汇率算金额，通道匹配走财务RPC |
| GetExchangePreview | 汇率 | 汇率试算 |
| CreateExchange | 汇率 + 余额 | 按汇率换算 + 扣法币入BSB |
| GetWithdrawMethods | 币种启用状态 + KYC状态 + 财务提现方式配置 + 充值路由规则 | 根据KYC和历史充值方式过滤可用提现方式 |
| CreateWithdrawOrder | KYC状态 + 余额 + 提现限额配置 + 账户信息 | 多重校验后冻结余额 |
| 财务：充值入账 | 钱包 CreditWallet RPC | 支付成功后调钱包入账 |
| 财务：提现审核 | 钱包 Freeze/Unfreeze/Deduct RPC | 审核结果联动钱包冻结/解冻 |
| 财务：人工加减款 | 钱包 Credit/Debit/QueryBalance RPC | 先查余额再操作 |
| 财务：数据统计 | 订单数据 + 通道数据 + 汇率(换算为基准币种) | 所有统计数据都需要基准币种换算 |

### 4.3 独立接口（不依赖其他业务接口，可以独立工作）

| 独立接口 | 为什么独立 |
|---------|----------|
| GetTransactionList / GetTransactionDetail | 纯读操作，只查本地账变记录表 |
| GetWithdrawAccount / SaveWithdrawAccount | 提现账户信息的读写，不涉及资金操作 |
| GetExchangeRateLogs | 纯读操作，只查汇率日志表 |
| GetCurrencyDetail | 纯读操作，查单个币种配置 |
| 财务：通道配置 CRUD | 不涉及钱包，纯后台配置管理 |
| 财务：支付配置 CRUD | 不涉及钱包，纯后台配置管理 |
| 财务：充值记录查询/提现记录查询 | 纯读操作，查订单表 |
| 财务：锁定/解锁/强制解锁 | 只改订单的锁定状态，不涉及资金 |

**依赖关系梳理：**

| 步骤 | 依赖的前置 | 说明 |
|------|----------|------|
| ② 获取充值方式 | 财务：支付方式配置已完成 | 没配置就没有可选方式 |
| ④ 重复转账检测 | 订单表已存在 | 查同金额待处理订单 |
| ⑤ 读取汇率 | 汇率定时任务已运行 | BSB充值需要展示汇率 |
| ⑥ 通道匹配 | 财务：通道配置已完成 + 轮询规则已配 | 没有通道无法创建订单 |
| ⑩ 入账 | 钱包 CreditWallet RPC 已实现 | 财务回调后调钱包入账 |

| ⑩ 代付 | 通道配置已完成 + 通道可用 | 调三方通道 |
| ⑪ 回调处理 | DeductFrozen/UnfreezeBalance RPC 已实现 | 成功扣冻结，失败等人工 |

根据以上依赖分析，接口的建设顺序应该是"从地基往上盖"：

Phase 3 ─ C端功能（依赖 Phase 2，部分依赖财务模块）

## 一、三模块的角色定位与基本关系

**币种配置**是数据提供者，**钱包**是业务执行者，**财务管理**是运营管控者。

用一句话串联三者：运营人员先在**币种配置**里设好币种和汇率规则，用户通过**钱包**进行充值提现兑换等资金操作，运营人员再通过**财务管理**来审核提现、人工调账、查看统计报表。

三者的基本数据流向：

- 币种配置 → 钱包：钱包的一切金额计算都依赖币种配置提供的汇率、精度、币种状态
- 币种配置 → 财务管理：财务的统计报表需要汇率换算为基准币种
- 钱包 → 财务管理：钱包暴露 RPC 接口，财务调用它来操作用户余额
- 财务管理 → 钱包：财务把支付方式、通道能力提供给钱包，钱包在 C 端展示和使用
- 钱包 ← 财务管理：这是双向的，两个模块互相调用

**币种配置不依赖钱包和财务**——它是底层基础设施，谁来读它的数据它不关心。这意味着币种配置可以完全独立开发，不需要等其他模块。

## 二、币种配置 → 钱包：逐场景依赖分析

币种配置为钱包提供的核心数据有三类：**币种信息**、**汇率数据**、**显示规则**。以下按钱包的每个业务场景，说明它具体依赖币种配置的什么。

### 2.1 钱包首页（余额展示）

钱包首页需要知道当前平台有哪些可用币种 → **依赖币种配置的"已启用币种列表"**。

用户切换币种时，钱包需要知道每个币种的货币符号、千分位符号、小数精度 → **依赖币种配置的显示规则字段**。

依赖路径：`币种配置.已启用币种列表 → 钱包.首页币种切换`
依赖路径：`币种配置.精度/符号规则 → 钱包.金额格式化显示`

依据：[需求 3.2] 钱包按币种维度隔离，[需求 3.3] 切换币种立即切换余额和充提方式。

### 2.2 充值（金额换算）

BSB 充值时，用户输入的是 BSB 金额，但 USDT 充值需要按汇率换算显示"约等于多少 USDT" → **依赖币种配置的入款汇率**。

其他法币充值如果涉及跨币种（如用银行卡充 BSB），也需要汇率 → **依赖币种配置的入款汇率**。

依赖路径：`币种配置.入款汇率 → 钱包.充值金额换算展示`

依据：[需求 3.4.1] "只有 BSB 充值金额输入框下有对应汇率展示及兑换所得金额"，[需求 4.5.1] "充值（跨币种）使用入款汇率"。

### 2.3 兑换（汇率计算）

兑换页面展示 `1 BSB = X 法币` → **依赖币种配置的平台汇率**。

用户输入法币金额后计算能换多少 BSB → **依赖币种配置的汇率 + BSB 锚定价（10 BSB = 1 USDT）**。

依赖路径：`币种配置.平台汇率 → 钱包.兑换预览计算`
依赖路径：`币种配置.BSB锚定价 → 钱包.兑换到账金额计算`

依据：[需求 3.7] 兑换汇率显示格式 `1 BSB = X 法币`，[需求 4.3] BSB 价值 10 BSB = 1 USDT。

### 2.4 提现（出款汇率）

BSB 提现时需要按出款汇率换算为目标法币 → **依赖币种配置的出款汇率**。

依赖路径：`币种配置.出款汇率 → 钱包.提现金额换算`

依据：[需求 3.8.5] "BSB 提现金额输入框下方显示汇率"，[需求 4.5.1] "提现（跨币种）使用出款汇率"。

### 2.5 账变记录（币种过滤）

切换币种后记录列表立即刷新 → 钱包需要知道当前选中的币种代码来过滤数据 → **依赖币种配置的币种标识**。

依赖路径：`币种配置.币种代码 → 钱包.账变记录按币种过滤`

依据：[需求 3.9.1] "钱包当前币种 = 记录当前币种，切换币种立即切换全部记录数据集"。

### 2.6 小结：币种配置对钱包的影响范围

**币种配置的任何变动都可能影响钱包：**
- 禁用一个币种 → 该币种钱包不可充提，C 端不展示
- 修改汇率浮动 → 入款/出款汇率变化，影响所有充提金额
- 修改阈值 → 汇率更新频率变化
- 修改精度规则 → 金额显示格式变化

但**钱包的变动不影响币种配置**——钱包怎么用汇率数据，币种配置不关心。这是单向依赖。

---

## 三、钱包 → 财务管理：逐场景依赖分析

钱包依赖财务管理提供的数据主要是：**支付方式配置**和**通道能力**。以下按场景说明。

### 3.1 充值方式展示

用户进入充值页面，看到的充值方式列表（USDT/银行转账/电子钱包等）不是钱包自己定义的，而是**财务管理在后台配置的** → 钱包 C 端需要调用财务模块获取当前币种可用的充值方式。

依赖路径：`财务.充值方式配置数据 → 钱包.C端充值方式列表展示`

依据：[需求 5.3] 充值方式字段包括币种、名称、图标、档位、金额范围、关联通道、排序、状态——这些全是财务模块在 B 端配置的。

**如果财务模块的充值方式还没配好，钱包 C 端的充值页面就是空的，用户无法充值。**

### 3.2 充值下单（通道匹配）

用户确认充值后，钱包创建平台订单，但**订单需要匹配到一个具体的支付通道**才能跳转支付 → 通道匹配逻辑（轮询算法：成功率×时间×预警×权重）在**财务管理模块**内。

依赖路径：`钱包.创建充值订单 → 财务.通道轮询匹配 → 财务.创建通道侧订单 → 返回支付信息给钱包`

依据：[需求 5.4] 充值流程第 2 步"服务端：轮询算法计算通道分配"，[需求 5.2] 轮询规则"综合轮询得分 = 成功率系数 × 时间系数 × 预警系数 × 权重"。

**如果财务模块的通道配置不完善或通道全部不可用，钱包的充值订单创建会失败。**

### 3.3 提现方式展示

和充值类似，提现方式也是财务模块在后台配置的 → 钱包 C 端调用财务获取当前币种可用的提现方式。

依赖路径：`财务.提现方式配置数据 → 钱包.C端提现方式列表展示`

依据：[需求 5.3] 提现方式字段包括币种、名称、图标、状态。

**但钱包在展示之前还要做额外过滤**：根据用户 KYC 状态（未认证只能 USDT）和充值路由规则（怎么充的怎么提）。这两个过滤逻辑是钱包自己做的，不依赖财务。

依赖路径完整版：`财务.提现方式配置 → 钱包.获取全量方式 → 钱包.按KYC状态过滤 → 钱包.按充值路由规则过滤 → C端展示`

### 3.4 提现限额和手续费

提现的每日限额、单笔最低金额、手续费率 → 这些参数在哪配的？需求文档中没有明确说是财务 B 端配置还是钱包自己管理，但从产品原型来看，提现方式配置在财务管理模块内，限额和手续费大概率附属于提现方式配置。

依赖路径（推测）：`财务.提现方式配置(含限额/费率) → 钱包.提现校验(限额检查/手续费计算)`

依据：[需求 3.8.3] "每日提现限额 2,500,000 VND，单笔最低 2,500,000 VND，手续费 1%"——这是示例值，实际应该是可配置的。[需求 3.8.6] 提现校验需要检查限额。

**这是一个预评审待确认项：限额和手续费的配置归属。**

### 3.5 小结

钱包对财务管理的依赖集中在两个方面：
1. **配置数据**：充值方式、提现方式、限额、手续费 → 财务配好了钱包才能展示和使用
2. **通道能力**：通道匹配和通道订单创建 → 财务的通道功能就绪了钱包才能完成充值下单

**这些依赖的特点是：钱包是"读"财务的数据和"调"财务的服务，不会"写"财务的数据。**

## 四、财务管理 → 钱包：逐场景依赖分析

财务管理依赖钱包的核心场景全部是**资金操作**：需要调钱包的 RPC 接口来改变用户余额。

### 4.1 充值成功入账

三方支付通道回调到财务模块，财务确认支付成功后 → 需要把钱打到用户钱包里。

依赖路径：`三方通道回调 → 财务.确认支付成功 → 财务调钱包.CreditWallet(用户ID, 币种, 中心钱包, 到账金额, 充值订单号)`

如果用户选了奖金活动，奖金部分入奖励钱包：`财务调钱包.CreditWallet(用户ID, 币种, 奖励钱包, 奖金金额, 充值订单号, 稽核倍数, 稽核流水)`

依据：[需求 5.4] "支付成功：订单状态变更为支付成功 → 触发充值额度到钱包用户上"，[需求 6.1] CreditWallet 接口定义。

**如果钱包的 CreditWallet 接口不可用，充值支付成功了但钱入不了账，会造成用户投诉。这是最关键的依赖。**

### 4.2 提现冻结

严格来说，提现冻结发生在**用户提交提现申请的那一刻**，是钱包自己内部做的（C 端接口 CreateWithdrawOrder 内部调用 FreezeBalance）。但从财务的视角看，整个提现审核流程都依赖冻结已经成功。

依赖路径：`用户提交提现 → 钱包.FreezeBalance(冻结) → 订单进入待审核 → 财务.开始审核`

**财务能审核的前提是钱包已经冻结了余额。如果冻结失败（比如余额不足），订单不会创建，财务看不到这笔提现。**

### 4.3 提现审核驳回（风控/财务/出款任一环节）

审核驳回时，冻结的钱要还给用户。

依赖路径：`财务B端.审核驳回 → 财务调钱包.UnfreezeBalance(用户ID, 币种, 冻结金额, 提现订单号) → 用户可用余额恢复`

依据：[需求 5.5] "风控驳回 → 解冻钱包金额"，"财务驳回 → 解冻钱包金额"。

**UnfreezeBalance 必须和 FreezeBalance 配对使用。解冻的金额必须等于之前冻结的金额，订单号也要对应。这是强一致性要求。**

### 4.4 出款成功

通道代付成功，钱已经打给用户了，这时候要把冻结的金额正式扣除（从冻结余额中移除，不是从可用余额扣）。

依赖路径：`三方通道代付成功回调 → 财务.确认出款成功 → 财务调钱包.DeductFrozenAmount(用户ID, 币种, 金额, 提现订单号) → 冻结余额减少`

依据：[需求 5.5] "出款成功 → 扣除冻结金额"，[需求 6.1] DeductFrozenAmount 接口定义。

### 4.5 出款失败

通道代付失败，但钱**不自动解冻**，保持冻结状态等待人工处理。

依赖路径：`三方通道代付失败回调 → 财务.标记出款失败 → 不调钱包任何接口 → 等待人工介入`

人工介入后可能有两种走向：
- 重新发起代付 → 成功后走 4.4 的路径
- 判断为不可恢复 → 人工操作解冻（具体流程待评审确认）

依据：[需求 5.5] "出款失败 → 保持冻结，不自动解冻，需人工处理"。

**这个设计很谨慎：失败不自动解冻是为了防止在通道实际已出款但回调返回失败（网络问题等）的情况下，用户余额被错误释放后再次提现导致资金损失。**

### 4.6 充值补单

用户实际付了钱，但系统没收到通道回调，订单卡在"待支付"状态 → 运营在财务 B 端手动补单。

依赖路径：`财务B端.创建补单(关联原充值订单) → 审核通过 → 财务调钱包.CreditWallet(用户ID, 币种, 中心钱包, 补单金额+补单赠送, 补单订单号B+16, 稽核倍数) → 用户余额增加`

补单前，财务需要查用户当前余额来展示 → `财务调钱包.QueryWalletBalance`

依据：[需求 5.6.1] "补单成功 → 补单总金额（补单金额 + 补单赠送）记入用户钱包"。

### 4.7 人工加款

运营给用户手动加钱（补偿/奖励/测试等），**可以同时给多个子钱包加不同金额**。

依赖路径：
1. `财务B端.打开加款页面 → 财务调钱包.QueryWalletBalance → 展示用户当前各钱包余额`
2. `运营填写加款信息(中心钱包+X, 奖励钱包+Y, 主播钱包+Z)`
3. `审核通过 → 财务调钱包.CreditWallet(中心钱包, X) + CreditWallet(奖励钱包, Y) + CreditWallet(主播钱包, Z)`

**注意：一次加款操作可能触发多次 CreditWallet 调用（每个子钱包一次）。每次调用带各自的稽核倍数和稽核流水。**

依据：[需求 5.6.2] "可指定目标钱包类型及每个钱包的加款金额：中心/奖励/代理/主播"。

### 4.8 人工减款

和加款类似但方向相反，从用户钱包扣钱，**关键约束：减款金额不能超过当前余额**。

依赖路径：
1. `财务B端.打开减款页面 → 财务调钱包.QueryWalletBalance → 展示用户当前各钱包余额`
2. `运营填写减款信息 → 前端校验：减款后余额 >= 0`
3. `审核通过 → 财务调钱包.DebitWallet(中心钱包, X) + DebitWallet(奖励钱包, Y)`

**后端同样必须校验余额充足，不能只靠前端。**

依据：[需求 5.6.3] "减款金额不能超过当前余额（减款后余额 >= 0）"。

### 4.9 数据统计

财务模块的统计报表需要把所有币种的金额**统一换算为基准币种**来做汇总。

依赖路径：`财务.统计报表 → 钱包/币种配置.ConvertAmount(金额, 源币种, 基准币种) → 返回换算结果`

依据：[需求 4.3] "所有币种金额在核算层必须换算为基准币种进行统计"，[需求 5.7] 统计数据实时更新。

### 4.10 小结

财务对钱包的依赖全部是**"写操作"或"读余额"**：
- 写操作：CreditWallet（入账）、DebitWallet（扣款）、FreezeBalance（冻结）、UnfreezeBalance（解冻）、DeductFrozenAmount（扣冻结）
- 读操作：QueryWalletBalance（查余额）、ConvertAmount（金额换算）

**反过来说：如果钱包的这些 RPC 接口不可用，财务管理模块的充值入账、提现审核、人工修正功能全部瘫痪。钱包的 RPC 可用性直接决定了财务模块的业务可用性。**

## 五、钱包 ↔ 财务的双向依赖梳理

这是三模块关系中最复杂的部分。钱包和财务之间不是单向关系，而是**你中有我、我中有你**。

### 5.1 钱包依赖财务的 3 个点

**① 充值方式数据**
- 场景：C 端用户看到的"USDT充值/银行转账/电子钱包"等选项
- 方向：钱包 → 调财务 RPC → 获取当前币种的充值方式列表
- 时机：用户进入充值页面时
- 依据：[需求 5.3] 充值方式配置

**② 通道匹配和下单**
- 场景：用户确认充值后，需要匹配一个支付通道并创建通道侧订单
- 方向：钱包 → 调财务 RPC → 传入充值方式+金额 → 财务返回通道分配结果和支付信息
- 时机：用户点击"去充值"时
- 依据：[需求 5.2] 轮询规则，[需求 5.4] 充值流程

**③ 提现方式数据 + 代付发起**
- 场景：C 端用户看到的提现方式，以及提现审核通过后发起代付
- 方向：钱包 → 调财务 RPC → 获取提现方式 / 发起代付
- 时机：用户进入提现页面时 / 审核通过后
- 依据：[需求 5.3] 提现方式配置，[需求 5.5] 提现流程

### 5.2 财务依赖钱包的 6 个点

**① 充值入账**
- 场景：三方通道回调确认支付成功
- 方向：财务 → 调钱包 RPC CreditWallet → 余额增加
- 时机：收到通道支付成功回调时
- 依据：[需求 5.4] "支付成功 → 触发充值额度到钱包用户上"

**② 提现冻结/解冻/扣冻结**
- 场景：提现全流程的余额联动
- 方向：提现创建时钱包自己冻结；审核驳回时财务 → 调钱包 UnfreezeBalance；出款成功时财务 → 调钱包 DeductFrozenAmount
- 时机：提现流程的各个节点
- 依据：[需求 5.5] 提现状态流转中的每个"解冻"和"扣除冻结"

**③ 补单入账**
- 场景：运营手动补单
- 方向：财务 → 调钱包 CreditWallet
- 依据：[需求 5.6.1] "补单成功 → 补单总金额记入用户钱包"

**④ 人工加款**
- 场景：运营手动给用户加钱
- 方向：财务 → 调钱包 CreditWallet（可能多次，每个子钱包一次）
- 依据：[需求 5.6.2]

**⑤ 人工减款**
- 场景：运营手动从用户扣钱
- 方向：财务 → 调钱包 DebitWallet（可能多次）
- 依据：[需求 5.6.3]

**⑥ 查询余额**
- 场景：加款/减款/补单的表单页需要展示用户当前余额
- 方向：财务 → 调钱包 QueryWalletBalance
- 依据：[需求 5.6.2] "显示加款后各钱包余额"，[需求 5.6.3] "显示当前余额和减款后余额"

### 5.3 双向依赖的本质

把上面归纳一下：

```
钱包依赖财务的本质 = 钱包需要财务的"配置数据"和"通道能力"
财务依赖钱包的本质 = 财务需要钱包的"余额操作能力"
```

两者的依赖方向不同：
- 钱包 → 财务：**"告诉我用户能用什么方式充值/提现，帮我对接通道"**
- 财务 → 钱包：**"帮我把钱加到/减从/冻结/解冻用户的钱包里"**

**耦合的根源在于：充值和提现这两个业务流程横跨了两个模块。** 用户发起操作（钱包管），中间的通道对接和审核（财务管），最终的余额变动（又回到钱包）。

## 六、币种配置 → 财务管理：依赖分析

这个方向的依赖比较简单。

### 6.1 统计报表的金额换算

财务的统计报表需要跨币种汇总（比如"所有币种的充值总金额"），就需要把 VND、IDR、THB、BSB 都换算成基准币种（USDT）来相加。

依赖路径：`财务.统计数据汇总 → 币种配置.汇率数据 → 换算为基准币种`

依据：[需求 4.3] "所有币种金额在核算层必须换算为基准币种进行统计"。

### 6.2 充值/提现记录中的汇率字段

财务 B 端的充值记录列表有 24 个字段，其中包含"入款汇率""平台汇率""基准币种订单金额""基准币种活动优惠""基准币种手续费" → 这些字段的值来源于**订单创建时使用的汇率**（由币种配置提供）。

依赖路径：`币种配置.入款汇率/平台汇率 → 钱包.创建充值订单时记录汇率快照 → 财务.充值记录列表展示`

依据：[需求 5.4] 充值记录列表字段。

**注意：这个依赖是间接的——财务不直接调币种配置取汇率，而是通过订单数据中已经记录的汇率快照来展示。**

### 6.3 小结

币种配置对财务管理的影响是间接的、通过数据传递的，不是直接的 RPC 调用关系。

## 七、三模块整体依赖关系总结

### 7.1 依赖方向一览

```
币种配置 ───(提供币种/汇率数据)──────────────────→ 钱包
币种配置 ───(提供汇率，通过订单快照间接传递)──────→ 财务管理
钱包 ───(获取支付方式配置)──────────────────────→ 财务管理
钱包 ───(请求通道匹配/创建通道订单)──────────────→ 财务管理
钱包 ───(请求发起代付)────────────────────────→ 财务管理
财务管理 ───(充值入账 CreditWallet)────────────→ 钱包
财务管理 ───(提现解冻 UnfreezeBalance)──────────→ 钱包
财务管理 ───(出款扣冻结 DeductFrozenAmount)─────→ 钱包
财务管理 ───(人工加款 CreditWallet)────────────→ 钱包
财务管理 ───(人工减款 DebitWallet)─────────────→ 钱包
财务管理 ───(查余额 QueryWalletBalance)─────────→ 钱包
```

### 7.2 依赖强度排序

**最强依赖（断了就瘫痪）：**
- 财务 → 钱包 CreditWallet：充值成功后入不了账，用户投诉
- 财务 → 钱包 Freeze/Unfreeze/Deduct：提现流程全线中断
- 币种配置 → 钱包的汇率数据：没有汇率，充值/提现/兑换全部金额计算失效

**中等依赖（缺了功能不全）：**
- 钱包 → 财务的充值方式/提现方式配置：C 端展示不出方式列表，但不影响已有订单
- 钱包 → 财务的通道匹配：无法创建新充值订单，但不影响提现和兑换

**弱依赖（缺了有替代方案）：**
- 币种配置 → 财务的统计汇率换算：可以降级为不换算或用缓存汇率
- 财务 → 钱包 QueryWalletBalance：B 端表单少了余额展示，但不影响审核操作本身

### 7.3 哪些可以并行开发，哪些必须串行

**可以完全并行（互不影响）：**
- 币种配置全部功能 ‖ 财务通道配置/支付配置 ‖ 钱包提现账户管理 ‖ 钱包账变记录

**必须串行（先后关系）：**
- 币种配置先就绪 → 钱包的汇率相关功能才能开发测试
- 钱包 RPC 先就绪 → 财务的充值入账/提现审核/人工修正才能联调
- 财务支付方式配置先就绪 → 钱包 C 端充值/提现方式展示才能联调

**有先后但可以 mock：**
- 钱包 C 端充值下单 → 可以先 mock 财务的通道匹配返回，内部逻辑先跑通
- 财务人工加减款 → 可以先 mock 钱包的 CreditWallet/DebitWallet 返回，审核流程先跑通

### 7.4 循环依赖风险

**充值流程存在潜在的循环依赖：**

```
钱包.创建订单 → 调财务.匹配通道 → 财务内部处理 → 通道回调 → 财务.调钱包.入账
```

这个链路是：钱包→财务→（异步）→财务→钱包。但因为中间有异步（通道回调），所以不是同步循环，不会死锁。设计时要注意：
- 钱包调财务是**同步** RPC（等待通道分配结果）
- 财务调钱包入账是**异步**的（通道回调触发，与用户请求不在同一链路）
- 这意味着两个方向的调用不会在同一个请求链路中形成环

**提现流程不存在循环：**

```
钱包.冻结(内部) → 订单创建 → 财务.审核(人工) → 财务.调钱包.解冻/扣冻结
```

这里钱包的冻结是内部操作，不调财务；财务的审核结果再调钱包。方向是 钱包(内部)→等待→财务→钱包，中间隔了人工操作，没有同步循环。
