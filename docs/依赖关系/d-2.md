**分层逻辑说明**：
- **基础设施层**：外部服务和通用能力，被上层调用，自身无业务依赖

- **用户触达层**：HTTP网关路由，将用户请求转发到业务服务

## 二、模块间依赖方向图

```
                         ┌───────────────┐
                         │   币种配置     │
                         │  (数据基座)    │
                         └───┬───────┬───┘
                             │       │
                    提供币种  │       │ 提供汇率
                    列表/规则 │       │ /展示规则
                             │       │
                   ┌─────────▼─┐   ┌─▼───────────┐
                   │           │   │             │
                   │ ser-wallet│   │  财务管理    │
                   │  (钱包)   │   │  (他人负责)  │
                   │           │   │             │
                   └─────┬─┬───┘   └──┬──┬───────┘
                         │ │          │  │
           ┌─────────────┘ └──────────┘  │
           │  充值方式/通道匹配/出款提交   │
           │  ◀────────────────────────  │
           │                             │
           │  上账/扣款/冻结/解冻/余额     │
           │  ────────────────────────▶  │
           └─────────────────────────────┘

图例：
  ──▶  数据流方向（调用方 → 被调用方）
  币种配置 → 钱包/财务：单向供给（币种配置只被调用，不调用业务模块）
  钱包 ↔ 财务：双向依赖（耦合关系，需重点关注）
```

### 依赖性质分类

| 依赖方向 | 性质 | 耦合度 | 说明 |
|----------|------|--------|------|
| 币种配置 → 钱包 | **单向供给** | 低 | 币种配置是纯数据提供方，不依赖钱包 |
| 币种配置 → 财务 | **单向供给** | 低 | 同上 |
| 钱包 → 财务 | **功能依赖** | 中 | 钱包依赖财务的充值方式/提现方式/通道匹配能力 |
| 财务 → 钱包 | **数据回写** | 高 | 财务的充值成功/提现审核/人工修正都需要操作钱包余额 |
| 钱包 → ser-kyc | **校验依赖** | 低 | 提现前的KYC状态检查，只读调用 |
| 全部 → ser-blog | **旁路依赖** | 极低 | 异步oneway审计日志，不影响主流程 |

## 三、系统初始化顺序（Bootstrap Order）

> 系统从零启动时，各模块/接口必须按此顺序就绪，否则下游模块无法正常工作

```
Phase 0: 基础设施就绪
  ├── TiDB / Redis / ETCD / Kafka 集群启动
  ├── ser-blog（审计日志）启动
  ├── ser-s3（文件存储）启动
  └── ser-kyc（KYC服务）启动

Phase 1: 币种配置就绪（最底层业务前置）
  ├── [1] SetBaseCurrency        ← 设置基准币种（USDT），一次性，不可逆
  ├── [2] AddCurrency × N        ← 创建各币种（BSB/VND/IDR/THB/USDT）
  ├── [3] EditCurrency           ← 配置汇率浮动、偏值、展示规则
  ├── [4] 启动汇率定时拉取任务    ← 开始从三方API拉取实时汇率
  └── 此时可用的RPC能力：
       ├── GetCurrencyList       ← 下游可获取币种列表
       ├── GetExchangeRate       ← 下游可获取汇率
       ├── GetCurrencyDetail     ← 下游可获取展示规则
       └── GetBaseCurrency       ← 下游可获取基准币种

Phase 2: 钱包服务就绪 + 财务服务就绪（可并行）
  ├── ser-wallet 启动
  │   ├── 注册RPC服务（CreditWallet/DebitWallet/GetBalance/...）
  │   └── 依赖Phase1的币种配置RPC可用
  │
  └── 财务服务 启动
      ├── [5] 配置代收通道（开发写入 + 后台编辑参数）
      ├── [6] 配置代付通道
      ├── [7] 配置通道轮询规则
      ├── [8] 配置充值方式（关联通道+档位+金额范围）
      ├── [9] 配置提现方式（关联通道+费率）
      └── 依赖Phase1的币种配置RPC可用

Phase 3: 用户注册链路就绪
  └── [10] 用户注册 → ser-user 调用 ser-wallet.InitWallet
       └── 为新用户创建各币种钱包账户

Phase 4: C端业务可用
  └── 用户可以执行充值/提现/兑换/查记录
```

### 初始化依赖关系图

```
SetBaseCurrency ──▶ AddCurrency ──▶ EditCurrency ──▶ 启动汇率定时任务
       │                │                │                   │
       │                │                │                   │
       └────────────────┴────────────────┴───────────────────┘
                                │
                    币种配置 RPC 能力就绪
                                │
               ┌────────────────┼────────────────┐
               ▼                ▼                ▼
        ser-wallet 启动    财务服务启动      用户服务可调
               │                │           InitWallet
               │         ┌─────┴─────┐
               │         ▼           ▼
               │    配置代收/代付   配置充值/
               │    通道+轮询规则  提现方式
               │         │           │
               │         └─────┬─────┘
               │               │
               └───────┬───────┘
                       ▼
                C端业务全面可用
```

**关键结论**：
- 币种配置是**全局前置**，必须最先完成，其他模块全部依赖它
- 钱包和财务可以**并行启动**，但C端业务需要两者都就绪
- 财务的通道配置和支付配置属于**运营前置**，不配置则充值/提现无法使用

## 四、核心业务链路分析

### 4.1 充值链路（最复杂的跨模块链路）

```
用户操作              ser-wallet              币种配置            财务服务           三方通道
  │                      │                      │                  │                 │
  │  进入充值页           │                      │                  │                 │
  ├─────────────────────▶│                      │                  │                 │
  │                      │  GetCurrencyList     │                  │                 │
  │                      ├─────────────────────▶│                  │                 │
  │                      │  ◀─ 币种列表+展示规则 │                  │                 │
  │                      │                      │                  │                 │
  │                      │  GetRechargeMethodList                  │                 │
  │                      ├─────────────────────────────────────────▶                 │
  │                      │  ◀─ 充值方式+档位+金额范围              │                 │
  │  ◀─ 充值页面数据 ────│                      │                  │                 │
  │                      │                      │                  │                 │
  │  选择金额+奖金       │                      │                  │                 │
  │  提交充值            │                      │                  │                 │
  ├─────────────────────▶│                      │                  │                 │
  │                      │  CheckDuplicate      │                  │                 │
  │                      │  (钱包内部检查)       │                  │                 │
  │                      │                      │                  │                 │
  │                      │  GetExchangeRate（仅BSB充值时）          │                 │
  │                      ├─────────────────────▶│                  │                 │
  │                      │  ◀─ 入款汇率         │                  │                 │
  │                      │                      │                  │                 │
  │                      │  MatchChannel + CreateChannelOrder       │                 │
  │                      ├─────────────────────────────────────────▶│                 │
  │                      │  ◀─ 匹配通道+三方支付URL                │                 │
  │                      │                      │                  │                 │
  │                      │  创建平台充值订单     │                  │                 │
  │                      │  (状态=待支付)        │                  │                 │
  │  ◀─ 支付URL/地址 ───│                      │                  │                 │
  │                      │                      │                  │                 │
  │  完成支付 ───────────┼──────────────────────┼──────────────────┼────────────────▶│
  │                      │                      │                  │                 │
  │                      │  RechargeCallback（三方回调）            │                 │
  │                      │◀────────────────────────────────────────┼─────────────────│
  │                      │  验签+验金额          │                  │                 │
  │                      │  幂等检查             │                  │                 │
  │                      │  CreditWallet         │                  │                 │
  │                      │  (中心钱包上账)       │                  │                 │
  │                      │  写资金流水           │                  │                 │
  │                      │  更新订单状态=成功     │                  │                 │
  │                      │                      │                  │                 │
  │                      │  AddActLog (oneway)   │                  │                 │
  │                      ├──────────────────────┼──────────────────┼──▶ ser-blog     │
  │                      │                      │                  │                 │
```

**充值链路接口调用顺序**：

| 步骤 | 接口 | 所属模块 | 调用方向 | 前置依赖 |
|------|------|---------|---------|---------|
| 1 | GetCurrencyList | 币种配置 | 钱包→币种 | 币种配置已完成 |
| 2 | GetRechargeMethodList | 财务 | 钱包→财务 | 充值方式已配置 |
| 3 | CheckDuplicateTransfer | 钱包 | 内部 | 无 |
| 4 | GetExchangeRate | 币种配置 | 钱包→币种 | 汇率定时任务运行中（仅BSB） |
| 5 | MatchChannel | 财务 | 钱包→财务 | 代收通道+轮询规则已配置 |
| 6 | CreateRechargeOrder | 钱包 | 内部 | 步骤1-5全部完成 |
| 7 | RechargeCallback | 钱包 | 三方→钱包 | 用户完成支付 |
| 8 | CreditWallet | 钱包 | 内部(或财务→钱包) | 回调验证通过 |

**提现链路接口调用顺序**：

| 步骤 | 接口 | 所属模块 | 调用方向 | 前置依赖 |
|------|------|---------|---------|---------|
| 1 | GetKycStatus | ser-kyc | 钱包→KYC | 无 |
| 2 | GetWithdrawMethodList | 财务 | 钱包→财务 | 提现方式已配置 |
| 3 | GetExchangeRate | 币种配置 | 钱包→币种 | 仅BSB提现时 |
| 4 | FreezeBalance | 钱包 | 内部 | 余额充足 |
| 5 | CreateWithdrawOrder | 钱包 | 内部 | 步骤4成功 |
| 6 | 风控审核 | 财务 | B端操作 | 步骤5完成 |
| 7 | 财务审核 | 财务 | B端操作 | 步骤6通过 |
| 8 | SubmitPayout | 财务 | 财务→三方 | 步骤7通过 |
| 9a | ConfirmDebit | 钱包 | 财务→钱包 | 三方出款成功 |
| 9b | UnfreezeBalance | 钱包 | 财务→钱包 | 审核驳回或出款失败 |

### 4.3 兑换链路（最简单的链路，钱包内部闭环）

```
用户操作           ser-wallet              币种配置
  │                   │                      │
  │  进入兑换页        │                      │
  ├──────────────────▶│                      │
  │                   │  GetCurrencyList     │
  │                   ├─────────────────────▶│
  │                   │  ◀─ 币种列表         │
  │                   │                      │
  │                   │  GetExchangeRate     │
  │                   ├─────────────────────▶│
  │                   │  ◀─ 法币→BSB汇率     │
  │  ◀─ 兑换页面 ────│                      │
  │                   │                      │
  │  输入金额+确认     │                      │
  ├──────────────────▶│                      │
  │                   │  DebitWallet          │
  │                   │  (扣法币中心钱包)     │
  │                   │                      │
  │                   │  CreditWallet         │
  │                   │  (加BSB中心钱包)      │

  │                   │  CreditWallet         │
  │                   │  (赠送部分→BSB奖励钱包)│
  │                   │                      │
  │                   │  写兑换流水记录       │
  │  ◀─ 兑换成功 ────│                      │
```

**兑换链路特点**：
- **不依赖财务模块**，仅依赖币种配置获取汇率
- 钱包内部完成扣款+上账+流水，是一个**原子事务**
- 涉及同用户跨币种的两个钱包操作，需要事务保证

**记录链路特点**：
- **最轻量级链路**，只依赖币种配置获取展示规则
- 数据全部来自钱包本地的订单表和流水表
- 无跨模块写操作，无耦合风险

## 七、耦合风险评估与解耦建议

### 7.1 耦合风险等级

| 耦合对 | 风险等级 | 原因 | 影响范围 |
|--------|---------|------|---------|
| 钱包↔财务(充值) | **高** | 同一业务流程跨两服务，回调链路长 | 充值失败=用户资金受损 |
| 钱包↔财务(提现) | **高** | 生命周期跨两服务，冻结/解冻/确认三态转换 | 提现异常=资金冻结无法恢复 |
| 钱包↔财务(人工修正) | **中** | 单向调用但直接操作余额 | 误操作=余额错误 |
| 钱包→币种配置 | **低** | 单向只读调用，可缓存降级 | 汇率延迟=金额计算偏差 |
| 财务→币种配置 | **低** | 单向只读，仅影响报表展示 | 汇率延迟=统计数据偏差 |

### 7.2 高耦合场景的关键保障措施

```
■ 充值回调场景（财务→钱包 CreditWallet）

  风险：回调成功但上账失败 → 用户付了钱但余额没增加
  保障：
  ┌──────────────────────────────────────────────┐
  │ 1. 幂等设计：回调以订单号为幂等键，重复调用不重复上账 │
  │ 2. 事务保障：上账+写流水+更新订单状态在同一事务内     │
  │ 3. 补偿机制：定时对账任务扫描"已支付未上账"订单       │
  │ 4. 告警机制：回调失败触发告警，人工介入                │
  └──────────────────────────────────────────────┘

■ 提现冻结-扣款场景（双向多步骤）

  风险：冻结成功但审核/出款环节异常 → 用户余额永久冻结
  保障：
  ┌──────────────────────────────────────────────┐
  │ 1. 状态机严格控制：冻结→(解冻|确认扣款)二选一          │
  │ 2. 超时解冻：冻结超过N小时未完成自动解冻               │
  │ 3. 对账：冻结金额定时与提现订单状态交叉校验            │
  │ 4. 强制解锁：B端支持管理员手动解冻                     │
  └──────────────────────────────────────────────┘

■ 人工修正场景（财务→钱包 Credit/DebitWallet）

  风险：审核通过但钱包操作失败 → 修正单显示成功但余额未变
  保障：
  ┌──────────────────────────────────────────────┐
  │ 1. 同步调用+失败回滚：RPC调用失败则审核状态回退        │
  │ 2. 双人审核：操作人≠审核人，减少误操作                 │
  │ 3. 操作审计：全量记录到ser-blog                       │
  └──────────────────────────────────────────────┘
```

## 十、关键依赖链路总结

### 10.1 三条核心链路的依赖深度对比

```
充值链路（依赖深度：4层）
  用户 → 钱包 → 币种配置（汇率）
                → 财务（充值方式+通道匹配）→ 三方通道 → 回调 → 钱包上账
  跨模块调用次数：5次（2次读取配置 + 1次匹配通道 + 1次回调 + 1次上账）
  风险：链路最长，任一环节故障都导致充值失败

提现链路（依赖深度：5层）
  用户 → 钱包 → KYC（状态检查）
                → 币种配置（汇率）
                → 财务（提现方式）
                → 钱包（冻结）
                → 财务（审核×3）→ 三方通道 → 回调 → 钱包（确认/解冻）
  跨模块调用次数：7次
  风险：流程最长，审核环节引入人工操作不确定性

兑换链路（依赖深度：1层）
  用户 → 钱包 → 币种配置（汇率）→ 钱包内部完成
  跨模块调用次数：1次
  风险：最低，仅依赖汇率数据
```

### 10.2 模块自治能力评估

| 模块 | 完全独立接口数 | 弱依赖接口数 | 强依赖接口数 | 自治率 |
|------|-------------|-------------|-------------|--------|
| 币种配置 | 15/17 | 2/17 | 0/17 | **88%** |
| 钱包RPC | 9/9 | 0/9 | 0/9 | **100%** |
| 钱包C端 | 5/18 | 6/18 | 7/18 | **28%** |

**结论**：
- **币种配置**自治率极高，可以完全独立开发和测试
- **钱包RPC能力层**自治率100%，是纯粹的能力输出，不依赖外部
- **钱包C端**自治率最低，7个接口强依赖财务模块，是联调重点

## 十二、三模块的角色定位与依赖本质

### 12.1 币种配置 — 数据基座，被依赖方

币种配置模块的本质是一个**配置中心**。它自身不产生业务行为，不操作用户资产，不参与交易流程。它的全部价值在于**向外输出数据**：币种列表、汇率、展示规则、基准币种信息。

从需求文档看（来源：币种配置需求文档 1-币种配置说明.png ~ 4-4汇率日志.png），币种配置的职责边界非常清晰：

- 管理币种基础信息（代码、名称、类型、国籍、图标、千分位/小数点规则）
- 管理汇率体系（实时汇率、平台入款/出款汇率、汇率浮动、偏差阈值）
- 管理基准币种（USDT，一次性设置不可更改）
- 定时从三方API拉取汇率并维护汇率日志

因此，**币种配置不调用钱包，不调用财务**。它的依赖关系是纯粹的单向输出：

```
币种配置 → 钱包      （钱包读取币种列表、汇率、展示规则）
币种配置 → 财务      （财务读取币种列表、汇率用于统计换算）
币种配置 → 三方汇率API  （定时拉取实时汇率，这是唯一的外部数据输入）
币种配置 → ser-s3    （币种图标文件上传）
币种配置 → ser-blog  （操作审计日志）
```

**关键认知**：币种配置是整个资金链路的地基。没有币种数据，钱包不知道展示什么币种，不知道金额按什么格式显示，充值提现兑换不知道按什么汇率计算，财务报表不知道用什么基准换算。所以币种配置必须**最先完成开发和配置**。

---

### 12.2 钱包 — 资产核心，双重身份

钱包模块有两个身份：

**身份一：C端业务的直接承载者**。用户看到的充值页面、提现页面、兑换页面、记录页面，全部由钱包模块提供HTTP接口（通过gate-font网关暴露）。在这个身份下，钱包是**调用方**，它需要从币种配置拿数据，从财务拿充值/提现方式配置，从KYC服务拿认证状态。

**身份二：资产能力的RPC提供者**。对内部其他服务来说，钱包是"资金操作的唯一入口"。充值要上账？调钱包。提现要冻结？调钱包。人工加减款？调钱包。在这个身份下，钱包是**被调用方**，主要被财务模块调用。

从需求文档看（来源：钱包需求文档 4-1钱包总体设计.png ~ 9非功能性需求.png），钱包的依赖关系是：

```
作为调用方（身份一）：
  钱包 → 币种配置    （获取币种列表、汇率、展示规则）
  钱包 → 财务-支付配置 （获取充值方式列表、提现方式列表）
  钱包 → 财务-通道    （充值时匹配通道、创建通道订单）
  钱包 → ser-kyc     （提现前检查KYC认证状态）
  钱包 → ser-blog    （审计日志）

作为被调用方（身份二）：
  财务 → 钱包  （充值上账、提现冻结/解冻/确认扣款、人工加减款、余额查询）
  用户服务 → 钱包  （注册时初始化钱包）
  游戏/投注 → 钱包  （投注扣款、返奖上账）
```

**关键认知**：钱包的C端接口高度依赖财务模块的就绪程度。财务的充值方式没配好，钱包的充值页面就是空的；财务的代收通道没配好，用户提交充值就会失败。但钱包的RPC能力层是完全自治的，CreditWallet、DebitWallet、FreezeBalance这些核心操作不依赖任何外部服务。

---

### 12.3 财务管理 — 运营后台，通道与审核中枢

财务模块是纯B端的运营管理系统（来源：财务需求文档 1-文档说明.png ~ 3-3数据统计-提现统计.png），它不直接面向C端用户。它的核心职责有三个：

1. **配置管理**：管理代收/代付通道的参数，管理充值/提现方式的展示和规则
2. **交易审核**：提现订单的风控审核→财务审核→出款审核三级流程
3. **数据统计**：按通道、充值、提现维度汇总统计

财务模块的依赖关系：

```
作为调用方：
  财务 → 钱包       （上账、扣款、冻结、解冻、确认扣款、查余额）
  财务 → 币种配置    （币种列表用于筛选下拉、汇率用于报表基准换算）
  财务 → 三方支付通道 （代收/代付的实际支付通道对接）
  财务 → ser-blog   （审计日志）

作为被调用方：
  钱包 → 财务       （获取充值/提现方式配置、匹配充值通道）
```

**关键认知**：财务模块与钱包之间形成了**双向依赖**，这是整个系统最紧密的耦合点。充值场景：钱包先调财务匹配通道，财务收到回调后又调钱包上账。提现场景：钱包冻结余额，财务审核后又回调钱包确认扣款或解冻。这种双向依赖不可避免——它源自业务本身的性质，充值提现天然是跨越"用户资产"和"支付通道"两个领域的操作。

## 十三、按业务场景逐一描述依赖关系

### 13.1 充值场景的完整依赖链

> 需求来源：钱包需求 5-1充值方式总览.png ~ 5-7页面交互.png，财务需求 2-3充值流程.png

**场景概述**：用户在C端发起充值，选择充值方式和金额，通过三方支付通道完成付款，平台收到回调后给用户钱包上账。

**依赖链条**（按用户操作时序）：

**第一步：展示充值页面**

用户进入充值页面时，钱包需要做两件事——获取当前可用的币种列表，获取当前币种下的充值方式列表。前者来自币种配置，后者来自财务的支付配置。

- 钱包 → 币种配置（GetCurrencyList）：获取用户可切换的币种列表，含图标和展示规则。依据：需求文档 4-2币种切换规则.png 明确说明"切换后所有页面联动切换到对应币种数据"，这要求钱包始终知道有哪些可用币种。
- 钱包 → 财务（GetRechargeMethodList）：获取当前币种下可用的充值方式。依据：财务需求 2-2支付配置-1充值方式.png 中定义了充值方式的完整字段（名称、图标、档位、金额范围、关联通道、排序、状态），钱包原型（5-1~5-4切换充值方式.png）的底部弹出sheet展示的正是这些数据。

**第二步：用户选择金额**

用户选择快捷档位或输入自定义金额。如果是BSB充值，页面需要展示USDT→BSB的汇率换算。

- 钱包 → 币种配置（GetExchangeRate，入款汇率）：仅在BSB充值时需要。依据：钱包需求 5-2usdt充值.png 明确"仅在BSB充值界面显示汇率换算"。需要使用入款汇率（= 实时汇率 × (1 + 浮动%)），让用户知道1 USDT能充多少BSB。
- 如果用户选择额外奖金 → 钱包内部处理或调活动服务。依据：需求 5-6额外奖金选择规则.png，奖金与充值金额联动，有门槛校验。

**第三步：重复转账检查**

用户提交前，钱包需检查是否短时间内提交了相同的充值（同用户+同币种+同金额）。

- 钱包内部逻辑，不依赖外部模块。依据：需求 5-5重复转账拦截规则.png，第1-2笔弹窗警告可继续，第3笔直接拦截。

**第四步：创建充值订单**

用户确认提交后，钱包需要通过财务的通道轮询逻辑匹配一个最优的代收通道。

- 钱包 → 财务（MatchChannel）：传入币种、充值方式、金额，财务根据通道轮询规则（成功率系数+时间系数+预警系数）选择最优通道，创建通道订单，返回三方支付URL。依据：财务需求 2-1通道配置-3通道轮询规则.png 和 2-3充值流程.png 中描述"服务端轮询匹配充值通道"。
- 如果无匹配通道 → 订单状态=支付失败，提示用户"系统繁忙请更换充值方式"。
- 匹配成功 → 钱包创建本地充值订单（状态=待支付），用户端拉起三方支付页面。

**第五步：支付回调**

用户在三方支付页面完成付款后，三方通道回调平台。

- 三方通道 → 钱包（RechargeCallback）：携带通道订单号、金额、支付状态。依据：钱包需求 7-1~7-4通用回调.png 描述了验签→验金额→更新状态→上账的流程。
- 钱包内部执行CreditWallet：验证通过后，给用户的中心钱包上账（金额=充值金额），如有活动奖金则给奖励钱包上账。依据：币种配置需求 2-1资金钱包体系.png 明确中心钱包接收充值到账、奖励钱包接收活动赠送。
- 幂等保障：依据钱包需求 9非功能性需求.png，"充值回调必须幂等，重复回调不重复上账"。

**充值场景依赖总结**：

```
参与模块：币种配置、钱包、财务、三方通道
依赖链：
  币种配置 → 钱包（提供币种+汇率数据）
  财务 → 钱包（提供充值方式配置 + 通道匹配能力）
  三方通道 → 钱包（支付结果回调）
  钱包内部完成：重复检查 + 创建订单 + 上账

调用次序：
  钱包先调币种配置 → 再调财务获取方式和匹配通道 → 等待三方回调 → 自行上账
  即：币种配置(读) → 财务(读+写) → 三方(异步回调) → 钱包(写)
```

---

### 13.2 提现场景的完整依赖链

> 需求来源：钱包需求 7-1提现模块.png ~ 7-6页面交互.png，财务需求 2-4提现流程.png

**场景概述**：用户在C端发起提现，经过KYC校验、余额冻结、风控审核、财务审核、出款审核、三方出款，最终完成提现。这是跨模块交互最复杂、生命周期最长的场景。

**第一步：KYC校验**

用户点击提现按钮时，钱包首先检查用户的KYC认证状态。

- 钱包 → ser-kyc（GetKycStatus）：查询用户是否已完成实名认证。依据：需求 7-1提现模块.png 明确"未认证用户弹窗，'继续提现'仅USDT，'去认证'跳KYC"。
- 这个依赖是**弱依赖**：KYC服务不可用时，可以按"未认证"处理（仅放行USDT提现），不会阻断整个提现流程。

**第二步：展示提现方式**

KYC通过后，展示当前币种下的提现方式列表。

- 钱包 → 财务（GetWithdrawMethodList）：获取提现方式配置。依据：财务需求 2-2支付配置-2提现方式.png 定义了提现方式的字段（名称、图标、费率、关联通道、大额限制、排序）。
- 钱包 → 币种配置（GetExchangeRate，出款汇率）：仅在BSB提现时需要。依据：钱包需求 7-4bsb提现.png，"BSB提现时展示汇率换算，出款按出款汇率计算"。

**第三步：选择账户**

用户选择或填写提现账户（银行卡/电子钱包）。

- 钱包内部操作，提现账户的增删改查是钱包本地数据。依据：需求 7-2银行卡提现.png 和 7-3电子钱包提现.png，"首次填写后平台记录，后续复用"，"持有人姓名=KYC姓名不可编辑"。

**第四步：提交提现 + 冻结余额**

用户确认提交后，钱包冻结提现金额。

- 钱包内部执行FreezeBalance：将提现金额从可用余额冻结，冻结后用户不可消费这笔资金，但余额仍可见。依据：提现流程需要保证在审核期间用户不能重复花费这笔钱。
- 钱包创建本地提现订单（状态=待风控审核）。依据：财务需求 2-4提现流程.png 中"创建平台订单（状态=待风控审核）"。

**第五步至第七步：审核流程（财务模块内部）**

提现订单创建后，进入财务模块的审核流程。这部分由财务模块内部处理，钱包不直接参与。

- 风控审核：依据：财务需求 2-4提现流程.png，"风控规则判断 → 通过/驳回"。如果驳回 → 财务 → 钱包（UnfreezeBalance），解冻已冻结金额。
- 财务审核：依据：同上，"人工审核（财务审核）→ 通过/驳回"。如果驳回 → 财务 → 钱包（UnfreezeBalance）。
- 出款审核：依据：同上，审核通过后提交三方出款。

**第八步：三方出款结果**

- 出款成功 → 财务 → 钱包（ConfirmDebit）：将已冻结金额正式扣除，写扣款流水。
- 出款失败 → 财务 → 钱包（UnfreezeBalance）：解冻金额，恢复可用余额。

**提现场景依赖总结**：

```
参与模块：ser-kyc、币种配置、钱包、财务、三方通道
依赖链：
  ser-kyc → 钱包（KYC状态校验，弱依赖）
  币种配置 → 钱包（出款汇率，仅BSB）
  财务 → 钱包（提现方式配置 + 审核结果回写余额）
  三方通道 → 财务（出款结果回调）

调用次序：
  钱包调ser-kyc → 钱包调财务获取方式 → 钱包内部冻结 → 财务内部审核 → 财务调三方出款
  → 三方回调财务 → 财务调钱包(确认扣款或解冻)

特殊性：
  提现是唯一涉及"冻结→等待→确认/回滚"三态转换的场景。
  冻结由钱包自己执行，确认/回滚由财务触发调用钱包。
  这意味着钱包需要暴露 FreezeBalance + UnfreezeBalance + ConfirmDebit 三个RPC，
  且必须保证状态机的严格性：已冻结的金额只能走向"确认扣除"或"解冻恢复"二选一。
```

### 13.3 兑换场景的依赖链

> 需求来源：钱包需求 6兑换模块.png，币种配置需求 4-1汇率应用场景.png

**场景概述**：用户将法币（VND/IDR/THB）兑换为平台币BSB。这是三个核心场景中依赖最简单的。

**依赖关系描述**：

兑换操作仅涉及钱包和币种配置两个模块，**完全不依赖财务模块**。

- 钱包 → 币种配置（GetExchangeRate）：获取法币→BSB的兑换汇率。依据：需求 6兑换模块.png 明确"汇率统一口径：1 BSB = X 法币"。
- 钱包内部完成扣款+上账：扣法币中心钱包余额 → 加BSB中心钱包余额 → 赠送部分加BSB奖励钱包。依据：需求 6兑换模块.png 中"赠送奖励（如10%），赠送部分进入奖励钱包"。

```
兑换场景依赖链：
  币种配置 → 钱包（提供汇率）
  钱包内部完成全部资产操作
  与财务模块无关、与三方通道无关、与KYC无关
```

这个特点意味着兑换功能可以**独立于财务模块**先行开发和测试，只要币种配置的汇率RPC就绪即可。

### 13.5 人工修正场景的依赖链

> 需求来源：财务需求 2-5人工修正-1~4.png

**场景概述**：运营在B端后台执行充值补单、人工加款、人工减款操作。

**依赖关系描述**：

人工修正是财务模块的B端功能，但其核心操作全部是对钱包余额的修改，因此形成**财务→钱包的单向强依赖**。

**充值补单场景**：

- 财务内部创建补单记录 → 审核通过 → 财务 → 钱包（CreditWallet）：给用户的中心钱包补上应到账金额，如果有赠送金额则同时给奖励钱包上账。依据：财务需求 2-5人工修正-1补单.png，"补单审核通过后，资金分配到对应钱包"。

**人工加款场景**：

- 财务内部创建加款记录时 → 财务 → 钱包（GetBalance）：需要查询用户当前各钱包余额作为参考信息。依据：虽然加款不需要校验余额（加就行了），但B端弹窗需要展示当前状态。
- 审核通过 → 财务 → 钱包（CreditWallet × 4）：分别对中心钱包、奖励钱包、主播钱包、代理钱包各自加款。依据：财务需求 2-5人工修正-2加款.png，"分别指定4个钱包的加款金额"。

**人工减款场景**：

- 财务内部创建减款记录时 → 财务 → 钱包（GetBalance）：**必须查询余额**，因为减款弹窗需要"展示各钱包当前余额，减款后余额实时计算"，且需要处理余额不足的情况。依据：财务需求 2-5人工修正-3减款.png。
- 审核通过 → 财务 → 钱包（DebitWallet × 4）：分别对4个钱包扣款。

```
人工修正依赖链：
  财务 → 钱包（GetBalance，查余额）
  财务 → 钱包（CreditWallet/DebitWallet，操作余额）
  这是单向依赖：财务调钱包，钱包不调财务
  双人审核制（操作人≠审核人）是财务内部流程，不涉及钱包
```

## 十四、按依赖方向分组描述

### 14.1 币种配置 → 钱包（币种配置向钱包输出的能力）

> 这是最基础的依赖关系，贯穿钱包模块的几乎所有场景

**输出内容一：可用币种列表**

钱包的个人中心页面、币种切换功能、充值/提现/兑换页面，都需要知道当前平台支持哪些币种、哪些是启用状态、各币种的名称和图标是什么。

- 触发场景：用户进入钱包页面 → 钱包调币种配置获取可用币种列表
- 依据：需求 4-2币种切换规则.png，原型 3-1~3-4充值币种切换.png（下拉列表展示所有可用币种带图标和名称）
- 如果币种被后台禁用 → C端不可见该币种。依据：币种配置需求 3-3后台配置页面说明.png，"禁用币种后C端不可使用/充值/提现"

**输出内容二：汇率数据**

三个场景需要汇率：
1. BSB充值时展示USDT→BSB的**入款汇率**。依据：需求 5-2usdt充值.png
2. 法币兑换BSB时展示法币→BSB的**平台汇率**。依据：需求 6兑换模块.png
3. BSB提现时展示BSB→法币/USDT的**出款汇率**。依据：需求 7-4bsb提现.png

入款汇率 = 实时汇率 × (1 + 浮动%)，对平台有利（用户少得）。出款汇率 = 实时汇率 × (1 - 浮动%)，对平台有利（用户少拿）。这个计算逻辑在币种配置内部完成，钱包直接使用计算结果。

**输出内容三：金额展示规则**

每个币种有不同的千分位符号、小数点符号、小数位数。钱包所有涉及金额展示的地方（余额、充值金额、提现金额、记录金额）都需要按规则格式化。

- 依据：需求 4-1钱包总体设计.png，"VND/IDR用`.`千分位+整数，THB/BSB用`,`千分位+2位小数"

---

### 14.2 币种配置 → 财务（币种配置向财务输出的能力）

**输出内容一：币种筛选数据**

财务的所有列表页面（通道列表、充值记录、提现记录、补单列表、加减款列表、各统计页面）都有币种筛选条件。筛选下拉的数据来源于币种配置的可用币种列表。

- 依据：财务需求中每个列表页的搜索条件都包含"订单币种"字段

**输出内容二：基准币种换算汇率**

财务的充值记录、提现记录中有"基准币种订单金额""基准币种手续费"字段，数据统计的汇总也需要换算为基准币种。

- 依据：财务需求 2-3充值记录.png 列表第21个字段"基准币种订单金额"、2-4提现记录.png 列表第20和21字段

---

### 14.3 钱包 → 财务（钱包向财务请求的能力）

> 这是钱包作为C端业务承载者，需要从财务获取配置和通道能力

**请求内容一：充值方式配置**

C端的充值页面展示的充值方式（银行转账/USDT/电子钱包等），包括每种方式的名称、图标、快捷档位、金额范围，全部来自财务模块的支付配置。

- 钱包 → 财务（获取充值方式列表）
- 依据：财务需求 2-2支付配置-1充值方式.png 定义了完整配置；钱包原型 5-1~5-4 的充值方式切换sheet直接展示这些数据
- 如果财务未配置充值方式或全部禁用 → 钱包充值页面无可用方式，用户无法充值

**请求内容二：提现方式配置**

同理，C端提现页面的提现方式（各国银行卡/电子钱包等）来自财务的支付配置。

- 钱包 → 财务（获取提现方式列表）
- 依据：财务需求 2-2支付配置-2提现方式.png；钱包原型 10-1~10-6 各国提现页面
- 提现费率也由此配置决定，展示在提现确认页面的"手续费明细"中

**请求内容三：充值通道匹配**

用户提交充值订单后，钱包需要调用财务的通道轮询逻辑为本次充值匹配一个最优通道。

- 钱包 → 财务（匹配通道 + 创建通道订单）
- 依据：财务需求 2-3充值流程.png，"服务端轮询匹配充值通道"；2-1通道轮询规则.png 定义了成功率/时间/预警三维度的轮询策略
- 匹配结果：成功则返回通道信息和三方支付URL，失败则订单直接标记为支付失败

---

### 14.4 财务 → 钱包（财务调用钱包的能力）

> 这是钱包作为资产能力提供者，被财务调用来操作用户余额

**调用场景一：充值成功上账**

三方通道回调确认支付成功后，需要给用户钱包上账。

- 财务 → 钱包（CreditWallet）：传入用户ID + 币种 + 金额 + 订单号 + 钱包类型（中心钱包）。如果用户选择了活动奖金，还需要再调一次CreditWallet到奖励钱包。
- 依据：财务需求 2-3充值流程.png，"成功：验证金额→订单状态=支付成功→用户上账→更新钱包余额"

**调用场景二：提现生命周期（冻结→审核→确认/回滚）**

提现从创建到完成，财务需要多次调用钱包：

1. 创建提现订单时 → 财务 → 钱包（FreezeBalance）：冻结用户的提现金额
2. 风控驳回 或 财务驳回 → 财务 → 钱包（UnfreezeBalance）：解冻已冻结金额
3. 出款成功 → 财务 → 钱包（ConfirmDebit）：正式扣除已冻结金额，写扣款流水
4. 出款失败 → 财务 → 钱包（UnfreezeBalance）：解冻已冻结金额

- 依据：财务需求 2-4提现流程.png 完整定义了状态流转

**调用场景三：人工修正**

- 充值补单通过 → 财务 → 钱包（CreditWallet）
- 人工加款通过 → 财务 → 钱包（CreditWallet × 4个钱包类型）
- 人工减款创建时 → 财务 → 钱包（GetBalance，查4个钱包余额）
- 人工减款通过 → 财务 → 钱包（DebitWallet × 4个钱包类型）
- 依据：财务需求 2-5人工修正-1~4.png

**调用场景四：余额查询**

- 财务在人工加减款的创建弹窗中需要展示用户当前各钱包余额。
- 财务 → 钱包（GetBalance）
- 依据：财务原型 人工修正-5减款弹窗.png，展示"当前余额"和"减款后余额"

## 十五、依赖强度分级描述

基于以上逐场景分析，将所有依赖关系按强度分为三级：

### 15.1 强依赖（缺失则功能完全不可用）

**钱包 → 财务-支付配置**：钱包C端的充值方式列表和提现方式列表完全来自财务的支付配置。财务不配置 → 钱包充值/提现页面为空 → 用户无法充值提现。这是钱包C端最致命的外部依赖。

**钱包 → 财务-通道匹配**：充值订单创建必须匹配到代收通道。通道未配置或全部不可用 → 用户提交充值必定失败。

**财务 → 钱包-CreditWallet**：充值成功后必须调钱包上账。钱包不可用 → 用户付了钱但余额不增加 → 资金事故。

**财务 → 钱包-Freeze/Unfreeze/ConfirmDebit**：提现流程的资金安全保障。这三个接口任一不可用 → 提现资金可能永久冻结或异常扣除。

### 15.2 中等依赖（缺失则部分功能降级）

**钱包 → 币种配置-GetExchangeRate**：仅BSB充值、BSB提现、法币兑换三个场景需要汇率。法币充值/提现不需要汇率（1:1），所以即使汇率服务暂时不可用，法币充值提现仍可正常工作，只是BSB相关操作受影响。

**钱包 → ser-kyc**：KYC服务不可用时，可以按未认证处理，用户仍可通过USDT提现。功能降级但不完全中断。

**财务 → 币种配置-BatchGetExchangeRate**：统计报表的基准币种换算列无法计算，但其他原始币种数据仍可正常展示。

### 15.3 弱依赖（缺失不影响核心功能）

**钱包 → 币种配置-GetCurrencyList**：币种列表变动频率极低（新增币种是低频运营操作），可以本地缓存。即使币种配置服务临时不可用，用缓存数据完全可以支撑。

**钱包 → 币种配置-GetCurrencyDetail**：展示规则（千分位/小数位）在币种创建后几乎不会变，可以长期缓存。

**所有模块 → ser-blog**：审计日志采用oneway异步调用，日志服务故障不影响任何业务操作。

## 十六、三模块间数据流向总结

从数据流的角度，三个模块之间的关系可以这样描述：

**币种配置 → 向外输出配置数据**

```
币种配置输出的数据：
  → 钱包：币种代码/名称/图标/展示规则 → 用于C端页面展示
  → 钱包：入款汇率/出款汇率/平台汇率 → 用于充值/提现/兑换的金额换算
  → 财务：可用币种列表 → 用于B端筛选下拉
  → 财务：基准币种汇率 → 用于统计报表换算

币种配置接收的数据：
  ← 三方汇率API：实时汇率原始数据 → 定时拉取、均值计算、偏差检测
  ← B端运营：配置变更操作 → 汇率浮动/偏值/图标/启用禁用
```

**钱包 → 资金操作的唯一入口**

```
钱包输出的能力（通过RPC）：
  → 财务：CreditWallet（上账） → 充值成功/补单/加款时调用
  → 财务：DebitWallet（扣款） → 减款时调用
  → 财务：GetBalance（查余额） → 人工修正页面展示
  → 财务：FreezeBalance（冻结） → 提现创建时调用
  → 财务：UnfreezeBalance（解冻） → 提现驳回/失败时调用
  → 财务：ConfirmDebit（确认扣款） → 提现成功时调用
  → 用户服务：InitWallet（初始化） → 用户注册时调用

钱包请求的数据（通过RPC）：
  ← 币种配置：币种列表 + 汇率 + 展示规则
  ← 财务：充值方式列表 + 提现方式列表 + 通道匹配结果
  ← ser-kyc：KYC认证状态
```

**财务管理 → 通道与审核中枢**

```
财务输出的能力（通过RPC）：
  → 钱包：充值方式配置列表 → C端充值页面展示
  → 钱包：提现方式配置列表 → C端提现页面展示
  → 钱包：通道匹配 + 创建通道订单 → 充值订单创建时调用

财务请求的能力（通过RPC）：
  ← 钱包：CreditWallet/DebitWallet/Freeze/Unfreeze/ConfirmDebit/GetBalance
  ← 币种配置：GetCurrencyList/GetExchangeRate/BatchGetExchangeRate
  ← 三方支付通道：代收/代付实际支付

财务内部独立完成：
  通道参数配置、轮询规则配置、审核流程（风控→财务→出款）、
  数据统计汇总、操作日志、锁定/解锁
```

## 十七、一句话总结各模块定位

- **币种配置**：只读的配置中心。对外只提供数据查询，不操作业务状态，不写入用户数据。是最干净、最独立、最应该先完成的模块。

- **钱包**：资金操作的单一入口。所有涉及用户余额的增减冻结，无论是充值、提现、兑换、人工修正还是游戏投注，最终都必须通过钱包的RPC接口完成。钱包的RPC层没有外部依赖，但C端展示层强依赖财务配置。

- **财务管理**：运营侧的配置+审核+统计。自身不持有用户资产数据，通过调用钱包RPC间接操作余额。管理着充值提现的"方式"和"通道"——这些配置数据反过来被钱包C端所依赖。
