# 二、各层目录职责详解

## 2.1 分层总览

```
请求方向（从上到下）：

  gate-back / gate-font          网关层（HTTP → RPC 转发）
         │
  handler.go                     接口委托层（纯透传，零逻辑）
         │
  internal/ser/                  业务编排层（校验、编排、RPC调外部）
         │
         ├──→ internal/ledger/   账务核心层（余额变更唯一入口）★
         │         │
         └──→ internal/rep/  ←───┘  手写数据访问扩展
                   │
         internal/gorm_gen/      自动生成的标准 CRUD（19方法/表）
                   │
              TiDB + Redis       存储层
```

## 2.2 目录职责表

| 目录 | 层级角色 | 职责边界 | 禁止做的事 |
|------|---------|---------|-----------|
| `handler.go` | 接口委托 | 每个 IDL 方法对应一行委托调用，分发到 ser 层的具体 Service | 不写任何业务逻辑、不做参数转换 |
| `internal/ser/` | 业务编排 | 参数校验、业务规则判断、编排 ledger/rep/cache/外部RPC | 不直接 UPDATE wallet 表的 balance/frozen 字段 |
| `internal/ledger/` | 账务核心 | 余额变更的 5 个原子操作（事务+幂等+行锁） | 不处理业务规则（如KYC判断、限额判断） |
| `internal/rep/` | 数据访问扩展 | 手写 gorm_gen 不覆盖的查询（带锁查询、聚合统计、复杂联查） | 不写业务逻辑 |
| `internal/cache/` | 缓存 | Redis 缓存的读写封装（Cache-Aside 模式） | 不决定缓存策略（由 ser 层控制） |
| `internal/gorm_gen/` | 标准 CRUD | 自动生成，每张表 19 个方法 | 禁止手动修改（重新生成会覆盖） |
| `internal/cfg/` | 基础设施 | MySQL/Redis 连接初始化（sync.Once 单例） | 不包含业务配置 |
| `internal/errs/` | 错误码 | 全模块错误码常量定义 | 不写错误处理逻辑 |
| `internal/enum/` | 枚举注册 | 面向前端的下拉选项枚举注册 | 不包含业务逻辑判断用的常量 |
| `internal/gen/` | 代码生成 | gorm_gen 的入口程序（`go run` 执行） | 不用于运行时 |

# 三、根目录文件说明

## 3.1 main.go — 服务启动入口

```
启动流程（遵循 ser-app 模式）：

  main()
    ├── lg.InitKLog()                    日志初始化
    ├── etcd.LoadAndWatch("/slg/")       ETCD 配置加载 + 监听
    ├── cfg.InitRedis()                  Redis 连接
    ├── cfg.InitMySQL()                  MySQL 连接 + query.SetDefault(db)
    ├── 解析端口（命令行 / ETCD）
    ├── 创建 WalletServiceImpl 实例
    ├── walletservice.NewServer(impl, rpc.InitRpcServerParams(...))
    └── srv.Run()
```

> 与 ser-app 的区别：ser-app 的 MySQL 是懒初始化的（在 ser 层 sync.Once），ser-wallet 在 main 中就初始化，因为余额操作从启动即需要数据库连接。

## 3.2 handler.go — 接口委托层

```
结构模式：

  type WalletServiceImpl struct {
      walletSer    *ser.WalletService          // 钱包核心
      depositSer   *ser.DepositService          // 充值
      withdrawSer  *ser.WithdrawService         // 提现
      exchangeSer  *ser.ExchangeService         // 兑换
      currencySer  *ser.CurrencyService         // 币种配置
      rateSer      *ser.RateService             // 汇率
      recordSer    *ser.RecordService           // 记录查询
      enumSer      *ser.EnumService             // 枚举（如有）
  }

  // 每个方法一行委托（~42个方法）
  func (s *WalletServiceImpl) GetWalletInfo(ctx, req) → s.walletSer.GetWalletInfo(ctx, req)
  func (s *WalletServiceImpl) CreateDepositOrder(ctx, req) → s.depositSer.CreateDepositOrder(ctx, req)
  func (s *WalletServiceImpl) CreateCurrency(ctx, req) → s.currencySer.CreateCurrency(ctx, req)
  ...
```

> 参照：ser-app/handler.go 有 21 个方法的一一委托，ser-wallet 约 42 个，模式完全相同。

## 3.3 go.mod

```
依赖关系：

  module ser-wallet

  require (
      common/pkg  → 公共工具（ret, errs, tracer, etcd, db, redis, snowflake...）
      common/rpc  → RPC 客户端工厂（rpc.WalletClient, rpc.KycClient 等）
  )
```

> 与所有 ser-xxx 模块一致，只依赖 common/pkg 和 common/rpc。

# 四、internal/ser/ — 业务编排层详解

ser 层是手写代码量最大的目录，按业务子模块拆分为 7 个文件。

## 4.1 目录结构

```
internal/ser/
├── wallet_ser.go                ← 钱包核心（余额查询、币种切换、RPC被调接口）
├── deposit_ser.go               ← 充值业务（创建订单、查询充值方式）
├── withdraw_ser.go              ← 提现业务（资格校验、创建订单）
├── exchange_ser.go              ← 兑换业务（汇率计算、创建兑换）
```

# 五、internal/ledger/ — 账务核心层详解

## 5.1 目录结构

```
internal/ledger/
└── ledger.go                    ← 全部账务逻辑集中在一个文件
```

> 为什么只有一个文件：ledger 层只对外暴露 5 个方法，内部逻辑相似度极高（事务+幂等+行锁），不需要拆分。如果后续代码量超过 500 行，可拆分为 ledger.go（主逻辑） + ledger_exchange.go（兑换特殊逻辑）。

## 5.2 对外暴露的 5 个方法

| 方法 | 功能 | 事务内操作 | 调用场景 |
|------|------|-----------|---------|
| Credit | 增加余额 | 幂等检查 → 锁行 → balance += amount → 写流水 | 充值上分、奖金发放、人工加款 |
| Debit | 减少余额 | 幂等检查 → 锁行 → 余额校验 → balance -= amount → 写流水 | 提现成功扣款、人工减款、消费 |
| Freeze | 冻结金额 | 幂等检查 → 锁行 → 可用余额校验 → frozen += amount → 写流水 | 提现申请时冻结 |
| Unfreeze | 解冻金额 | 幂等检查 → 锁行 → 冻结余额校验 → frozen -= amount → 写流水 | 提现拒绝/失败时归还 |
| Exchange | 跨币种兑换 | 幂等检查 → 锁源钱包 → 锁目标钱包 → 3次余额变更 → 写3条流水 | 法币→BSB 兑换 |

## 5.3 ledger 在分层中的位置

```
               ┌─────────────────────────────────┐
ser 层         │  wallet_ser / deposit_ser / ...  │
               └──────────┬──────────────────────┘
                          │ 调用 5 个方法
               ┌──────────▼──────────────────────┐
ledger 层      │  Credit / Debit / Freeze /       │
               │  Unfreeze / Exchange              │
               │                                   │
               │  内部保证：                         │
               │  · 事务原子性（GORM Transaction）   │
               │  · 幂等（wallet_flow 唯一索引）     │
               │  · 并发安全（SELECT FOR UPDATE）     │
               │  · 审计（每次变更写 wallet_flow）    │
               └──────────┬──────────────────────┘
                          │ 操作两张表
               ┌──────────▼──────────────────────┐
数据层         │  user_wallet 表 + wallet_flow 表  │
               └─────────────────────────────────┘
```

## 5.4 为什么 ledger 层是钱包特有

```
普通服务（ser-app）：
  ser → repo（直接 CRUD 就行，数据写错了改回来就行）

钱包服务（ser-wallet）：
  ser → ledger → repo（余额变更必须经过 ledger 层）

原因：余额操作需要同时保证 4 个性质（原子性 + 幂等 + 并发安全 + 审计记录），
如果散在 ser 层的各个方法里，每个改余额的地方都要写事务 + 幂等 + 锁，
容易遗漏、难以审计。收敛到 ledger 层是钱包系统业界标准做法。

规则：ser 层禁止直接 UPDATE user_wallet 表的 balance/frozen_amount 字段。
```

# 六、internal/rep/ — 手写数据访问扩展详解

## 6.1 目录结构

```
internal/rep/
├── wallet_rep.go                ← 钱包表的扩展查询
├── order_rep.go                 ← 订单表群的扩展查询
└── currency_rep.go              ← 币种/汇率表的扩展查询
```

## 6.2 为什么需要 rep 层

gorm_gen 自动生成的 19 个标准方法覆盖了常见 CRUD，但有些场景它不覆盖：

| 文件 | 补充的查询类型 | 具体场景举例 |
|------|--------------|------------|
| wallet_rep.go | 带行锁查询 | `SELECT * FROM user_wallet WHERE ... FOR UPDATE`（ledger 层需要） |
| wallet_rep.go | 条件聚合 | 查询用户所有币种的余额汇总 |
| order_rep.go | 复杂条件分页 | 带多状态筛选 + 时间范围的订单分页查询 |
| order_rep.go | 资金来源分析 | 查询用户最近N天的充值记录用于提现判断 |
| currency_rep.go | 汇率日志批量写入 | 汇率变更时写 rate_change_log |
| currency_rep.go | 带状态过滤的币种列表 | 只查启用状态的币种（含汇率 JOIN） |

> 参照：ser-app 有 `rep/banner.go` 和 `rep/home_category.go`，ser-wallet 按业务域拆分为 3 个文件。

# 七、internal/cache/ — 缓存层详解

## 7.1 目录结构

```
internal/cache/
├── wallet_cache.go              ← 余额 + 用户币种选择 缓存
└── currency_cache.go            ← 币种列表 + 平台汇率 缓存
```

## 7.2 缓存项规划

| 文件 | 缓存 Key 模式 | 缓存内容 | TTL 方向 | 失效时机 |
|------|--------------|---------|---------|---------|
| wallet_cache.go | `wallet:balance:{uid}:{cid}` | 用户某币种余额 | ~5min | Credit/Debit/Freeze/Unfreeze 后主动 DEL |
| wallet_cache.go | `wallet:user_currency:{uid}` | 用户当前选中币种 | ~30min | SwitchCurrency 后主动 DEL |
| currency_cache.go | `wallet:currency:list` | 可用币种列表（全量） | ~10min | CreateCurrency/UpdateCurrency/StatusCurrency 后主动 DEL |
| currency_cache.go | `wallet:rate:{cid}` | 某币种的平台汇率 | ~5min | RecalculateRates 后主动 DEL |

> 缓存策略：Cache-Aside（读时回填，写时失效），金融数据以 DB 为权威来源。
