## 二、顶层文件详解（4 个需编写的文件）

### 2.1 文件总览

```
ser-wallet/
├── go.mod          模块定义 + 依赖声明
├── go.sum          依赖版本锁定（自动生成）
├── main.go         服务启动入口
└── handler.go      RPC 方法委托
```

### 2.2 逐文件职责

#### go.mod

| 属性 | 说明 |
|------|------|
| 包名 | `module ser-wallet` |
| Go 版本 | `go 1.25`（与 ser-app 保持一致） |
| replace 指令 | `replace common/pkg => ./../common/pkg`（工程规范） |
| 核心依赖 | Kitex、GORM、go-redis、sarama（Kafka）、nats.go（均通过 common/pkg 间接引入） |

参考：ser-app/go.mod、ser-item/go.mod

#### main.go

| 属性 | 说明 |
|------|------|
| 包名 | `package main` |
| 职责 | 整个服务的启动编排，零业务逻辑 |
| 启动顺序 | 见下方 |

**启动顺序（必须严格遵循）：**

```
① lg.InitKLog()                         → 初始化日志
② etcd.LoadAndWatch("/slg/")            → 加载 ETCD 配置 + 启动监听
③ cfg.InitRedis()                       → 初始化 Redis 连接
④ 创建 7 个 Service 实例                  → 依赖注入
⑤ appservice.NewServer(&WalletServiceImpl{...}) → 创建 Kitex 服务器
⑥ srv.Run()                             → 启动监听
```

参考：ser-app/main.go（第 20-58 行）

#### handler.go

| 属性 | 说明 |
|------|------|
| 包名 | `package main` |
| 职责 | 实现 IDL 定义的 WalletService 接口，每个方法一行委托 |
| 核心结构体 | `WalletServiceImpl`，持有 7 个 Service 指针 |
| 方法数量 | 33 个（与 IDL service.thrift 中的方法一一对应） |

**结构体字段 → Service 映射：**

| 字段 | 类型 | 委托的方法 | 方法数 |
|------|------|-----------|--------|
| `currencyService` | `*ser.CurrencyService` | SetBaseCurrency / CreateCurrency / UpdateCurrency / ToggleCurrencyStatus / PageCurrency / PageRateLog / GetCurrencyList / GetExchangeRate / GetBaseCurrency / GetCurrencyByCode | 10 |
| `walletService` | `*ser.WalletService` | GetWalletBalance / GetAvailableCurrencies | 2 |
| `balanceService` | `*ser.BalanceService` | GetUserWalletBalance / CreditDeposit / FreezeForWithdrawal / UnfreezeForWithdrawal / ConfirmWithdrawalDeduct / ManualCredit / ManualDebit / CreditCorrection | 8 |
| `depositService` | `*ser.DepositService` | GetDepositMethods / CreateDepositOrder / GetUSDTDepositInfo / GetBonusActivities / GetDepositOrderStatus | 5 |
| `exchangeService` | `*ser.ExchangeService` | GetExchangePreview / ExecuteExchange | 2 |
| `withdrawalService` | `*ser.WithdrawalService` | GetWithdrawalInfo / CreateWithdrawalOrder / SaveWithdrawalAccount / GetWithdrawalAccounts | 4 |
| `transactionService` | `*ser.TransactionService` | GetTransactionRecords / GetTransactionDetail | 2 |

**规范要求**：handler.go 中每个方法的函数体只有一行 `return s.xxxService.MethodName(ctx, req)`，绝对不写任何业务逻辑。

参考：ser-app/handler.go（每个方法均为单行委托）

## 三、internal/ 各子目录详解

### 3.1 分层架构总览

```
请求进入
    │
    ▼
handler.go                 零逻辑，纯委托
    │
    ▼
┌─────────────────────────────────────────────────────┐
│  internal/ser/            业务逻辑层                  │
│  ┌─────────────────────────────────────────────────┐ │
│  │ 参数校验 → 业务规则 → 编排 rep/ 和 domain/ 调用  │ │
│  └─────────────────────────────────────────────────┘ │
└──────┬─────────────────────────────┬────────────────┘
       │                             │
       ▼                             ▼
┌──────────────┐          ┌───────────────────┐
│ internal/rep/│          │ internal/domain/  │
│ 数据库读写    │          │ 外部 RPC 调用      │
│ 事务管理      │          │ 复杂业务校验       │
│ 锁操作        │          │ 汇率API对接        │
└──────┬───────┘          └───────────────────┘
       │
       ▼
┌──────────────────────┐
│ internal/gorm_gen/   │
│ [自动生成] 基础CRUD   │
└──────────────────────┘
       │
       ▼
   TiDB 数据库

辅助层（横向支撑）：
  internal/cfg/    → 数据库/Redis 连接
  internal/enum/   → 枚举常量
  internal/errs/   → 错误码
  internal/cache/  → Redis 缓存
  internal/utils/  → 工具函数
```

**分层规则（铁律）**：
- `ser/` 可以调 `rep/`、`domain/`、`cache/`
- `rep/` 只操作数据库，不调 `ser/` 和 `domain/`
- `domain/` 只封装外部调用，不调 `rep/`
- `cache/` 只操作 Redis，不调其他层

**gorm_gen.go 修正要点**：

当前内容仍引用 ser-app 的配置路径，需修改为：
```go
mysql_gen.GenCodeWithAll(etcdWalletDb, "ser-wallet", "")
```

**生成流程**：建表 → 修正 gorm_gen.go → 运行 `go run internal/gen/gorm_gen.go` → 自动生成 model/query/repo

**model 文件内容说明**（以 wallet.gen.go 为例）：

| 生成内容 | 说明 |

| `type Wallet struct` | 结构体字段与数据库列一一映射 |

| 字段类型映射 | `bigint` → `int64`，`varchar` → `string`，`tinyint` → `int32` |
| EncryptedString | 敏感字段（银行卡号等）自动使用 `EncryptedString` 类型 |

| ... | 以此类推 | ... |

规则：表名去掉下划线 → 全小写 → 拼接 `r` → 目录名和文件名同名。

### 3.6 rep/ —— 业务数据访问层

```
internal/rep/
├── currency.go                 币种配置表 + 汇率变更日志表
├── rate_log.go                 汇率变更日志表（独立查询逻辑）
├── wallet.go                   ★ 钱包余额表（含行锁 + 乐观锁）
├── transaction.go              交易流水表
├── deposit.go                  充值订单表
├── withdrawal.go               提现订单表
├── withdrawal_account.go       提现账户表
└── exchange.go                 兑换订单表
```

**rep/ 与 gorm_gen/repo/ 的区别**：

```
gorm_gen/repo/  →  提供最基础的 CRUD（Create/Update/Delete/GetOne/GetPage）
                   自动生成，不可修改，所有表的方法签名完全一样

rep/            →  在基础 CRUD 之上，编写本模块特有的业务查询逻辑
                   手动编写，包含事务、联合查询、锁操作等
```

#### 逐文件职责

| 文件 | 对应表 | 核心方法说明 |
|------|--------|-------------|
| currency.go | currency_config | 按状态/类型筛选分页、按 code 查询（加缓存穿透保护）、更新状态、设置基准币种（全局唯一校验） |
| rate_log.go | rate_change_log | 批量插入日志、按币种+时间范围分页查询 |
| **wallet.go** | wallet | **最关键文件**。包含：① `GetForUpdate(userId, currencyCode, walletType)` 行锁查询 ② `UpdateBalanceWithVersion(wallet, version)` 乐观锁更新 ③ `InitWallet(userId, currencyCode)` 首次创建钱包记录 ④ 事务封装方法 |
| transaction.go | wallet_transaction | 按用户+币种+类型分页查询（C端记录列表）、按订单号查详情、批量写入（一笔操作可能同时写多条流水） |

**wallet.go 特殊说明（余额安全三重防护在此实现）**：

```
防护层     实现位置              作用
───────────────────────────────────────────
行级锁     wallet.go             SELECT ... FOR UPDATE，事务内锁定目标行
乐观锁     wallet.go             UPDATE ... WHERE version=旧值，更新 0 行则冲突
幂等键     deposit.go 等         订单号唯一键，防止重复入账
```

### 3.8 domain/ —— 领域层（外部 RPC + 复杂校验）

```
internal/domain/
├── common.go              公共校验辅助
├── finance_rpc.go         封装调用 ser-finance
├── kyc_rpc.go             封装调用 ser-kyc
├── blog_rpc.go            封装审计日志
└── rate_provider.go       汇率 API 对接
```

#### 为什么需要 domain/ 层

直接在 ser/ 中调用 RPC 的问题：
1. ser/ 文件会被外部 RPC 的错误处理、重试、降级逻辑污染
2. 无法单独 Mock 外部服务做单元测试
3. 多个 Service 调同一个外部服务时，代码重复

domain/ 层的作用 = **隔离外部世界**，ser/ 只需要调 `domain.XxxRPC.DoSomething()` 即可。

参考：ser-item/internal/domain/user_service.go（封装调用 ser-user RPC）

#### 逐文件职责

| 文件 | 封装的外部服务 | 核心方法 |
|------|---------------|---------|
| common.go | 无（纯辅助） | HandleDBError / ValidateID / ValidatePagination / ValidateAmount（金额非负校验） |
| finance_rpc.go | ser-finance | QueryPaymentMethods / QueryChannelConfig / NotifyWithdrawal — 充值查通道、提现通知财务审核 |
| kyc_rpc.go | ser-kyc | CheckKycStatus — 调用 KycQuery 接口，返回用户是否 KYC 通过 |
| blog_rpc.go | ser-blog | AddOperationLog — 封装 AddActLog（oneway），所有 B 端写操作调用 |
| rate_provider.go | 3 家汇率 API | FetchRealTimeRates — 从 3 家 API 取汇率求均值，被汇率同步定时任务调用 |

**finance_rpc.go 特殊说明**：

ser-finance 模块当前尚不存在。这个文件的设计意义在于：
- 编码初期可以内部写 Mock 实现，不阻塞钱包主流程开发
- 等 ser-finance IDL 定型后，只需修改这一个文件的实现
- ser/ 层完全不感知底层是 Mock 还是真实 RPC

| 过期策略 | TTL 较长（如 10 分钟），ETCD 配置变更时主动清除 |

**为什么不缓存余额**：

参考：ser-app/internal/cache/banner.go（BannerCache 的 Get/Set/Del 模式）

```
          ▼             ▼                              ▼
  currency_service  wallet_service          transaction_service
          │             │
          │        ┌────┘  ★ 引用 CurrencyService
          │        │
          ▼        ▼
  ┌─────────────────────────────────────┐
  │ deposit_service   exchange_service  │
  │ withdrawal_service                  │
  └──────────┬──────────────────────────┘
             │
             │  ★ 全部调用 BalanceService 做余额变更
             ▼
      balance_service
      （余额变更唯一入口）
```

**调用方向规则**：
- `deposit/exchange/withdrawal_service` → `balance_service`（下级调上级统一入口）
- `wallet_service` → `currency_service`（同级跨域内部调用）
- `balance_service` 不调用其他 Service（它是最底层的核心引擎）

### 8.2 完整分层调用链路（以"创建提现订单"为例）

```
handler.go
  └── WithdrawalService.CreateWithdrawalOrder(ctx, req)
        │
        ├── domain.KycRPC.CheckKycStatus(userId)              → RPC 调 ser-kyc
        │
        ├── domain.FinanceRPC.QueryWithdrawalMethods(...)     → RPC 调 ser-finance
        │
        ├── BalanceService.FreezeForWithdrawal(...)            → 余额变更入口

        │     └── rep.TransactionRepo.Create(...)              → 写流水
        │
        ├── rep.WithdrawalRepo.Create(...)                     → 创建提现订单
        │
        ├── domain.FinanceRPC.NotifyWithdrawal(...)            → 通知财务审核
        │
        └── domain.BlogRPC.AddOperationLog(...)                → oneway 审计日志
```

## 十、与现有工程的规范对照

| 规范项 | ser-app 怎么做 | ser-item 怎么做 | ser-wallet 怎么做 | 一致性 |
|--------|---------------|----------------|------------------|--------|
| 顶层文件 | main.go + handler.go + go.mod | 同左 | 同左 | 完全一致 |
| handler 委托模式 | ServiceImpl 持有 Service 指针，一行委托 | 同左 | 同左 | 完全一致 |
| cfg/ 初始化 | sync.Once + ETCD 读配置 | 同左 | 同左 | 完全一致 |
| enum/ 命名 | 按实体拆文件（banner.go / category.go） | 单文件（item.go） | 按域拆文件（3 个） | 跟 ser-app |
| errs/ 结构 | 单文件 code.go，const 块分组 | 同左 | 同左 | 完全一致 |
| gorm_gen/ 结构 | model/ + query/ + repo/{entityr}/ | 同左 | 同左 | 完全一致 |
| rep/ 命名 | banner.go / home_category.go | item.go / item_publish.go | currency.go / wallet.go / ... | 完全一致 |
| ser/ 命名 | banner.go / s3_service.go（混合） | item_service.go / s3_service.go | 统一 xxx_service.go | 跟 ser-item |

| utils/ | 目录存在但空 | 无 | 有 amount.go（金额工具） | 按需使用 |
