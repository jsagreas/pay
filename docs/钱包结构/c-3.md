### 1.2 与 ser-app 的结构对比

| 目录层 | ser-app（参考） | ser-wallet（设计） | 差异原因 |
|--------|----------------|-------------------|---------|
| 根目录 | main.go + handler.go | main.go + handler.go | 完全一致 |
| cfg/ | mysql.go + redis.go | mysql.go + redis.go | 完全一致 |
| enum/ | 3个文件 + registry | 3个文件 + registry | 枚举种类不同，结构一致 |

| gen/ | gorm_gen.go | gorm_gen.go | 完全一致 |
| gorm_gen/ | 2表 → model/query/repo | 12表 → model/query/repo | 表数量 6 倍，但生成模式一致 |

| rep/ | 2个文件 | 10个文件 | 12表但部分合并（如日志共用） |
| ser/ | 5个文件 | 7个文件 | 7个业务子域 |

| 文件 | 职责 | 是否手写 | 参考 |

> MySQL 不在 main.go 初始化，在 rep 层首次访问时通过 `sync.Once` 延迟初始化（工程统一模式）

## 三、internal 分层详解

### 3.1 cfg/ — 基础设施配置层

```
internal/cfg/
├── mysql.go        # MySQL/TiDB 连接（sync.Once 单例）
└── redis.go        # Redis 连接（sync.Once 单例）
```

| 文件 | 职责 | ETCD 配置键 | 模式 |
|------|------|------------|------|
| mysql.go | 初始化 GORM DB 实例 + query.SetDefault() | `env.TidbAddr` `env.TidbPort` `env.TidbUsername` `env.TidbPassword` `namesp.EtcdWalletDb` | sync.Once 单例 |
| redis.go | 初始化 Redis Client | `env.RedisAddr` `env.RedisPassword` | sync.Once 单例 |

> 与 ser-app/internal/cfg/ 结构完全一致，只是 DB 名称从 `namesp.EtcdAppDb` 改为 `namesp.EtcdWalletDb`

### 3.2 enum/ — 业务枚举层

```
internal/enum/
├── wallet.go           # 子钱包类型 + 操作类型
├── currency.go         # 币种类型 + 币种状态
├── order.go            # 各类订单状态
└── enum_registry.go    # 枚举注册表（B端下拉选项）
```

| 文件 | 包含的枚举 | 用途 |
|------|-----------|------|
| wallet.go | `WalletType`（CENTER/REWARD/STREAMER/AGENT/VENUE）、`OperationType`（CREDIT/DEBIT/FREEZE/UNFREEZE/CONFIRM） | 余额操作、流水记录分类 |
| currency.go | `CurrencyType`（FIAT/CRYPTO/PLATFORM）、`CurrencyStatus`（ENABLED/DISABLED） | 币种配置管理、前端展示 |
| order.go | `RechargeStatus`（PENDING/SUCCESS/FAILED）、`WithdrawStatus`（7状态流转）、`ExchangeStatus` | 订单生命周期管理 |
| enum_registry.go | 注册以上枚举到统一 registry，供 B 端下拉接口动态获取 | B端前端页面下拉选项数据源 |

### 3.3 errs/ — 错误码层

```
internal/errs/
└── code.go             # 全部错误码（6022XXX）
```

**错误码分段设计**：

| 段范围 | 所属模块 | 典型错误码 |
|--------|---------|-----------|
| 6022000~6022099 | 币种配置 | CurrencyNotFound、BaseCurrencyAlreadySet |
| 6022100~6022199 | 钱包核心 | InsufficientBalance、DuplicateOrderNumber |
| 6022200~6022299 | 充值 | DuplicateRechargeBlocked、RechargeChannelUnavail |
| 6022300~6022399 | 兑换 | ExchangeAmountTooSmall |
| 6022400~6022499 | 提现 | DailyLimitExceeded、KYCRequired |
| 6022500~6022599 | 记录 | RecordNotFound |
| 6022600~6022699 | 预留 | — |

> 工程统一格式：`6` + `022`（服务编号）+ `XXX`（顺序号），参照 ser-app 的 `6011XXX`

### 3.4 gen/ — GORM 代码生成触发器

```
internal/gen/
└── gorm_gen.go         # 连接 DB → 扫描表 → 生成 model/query/repo
```

| 文件 | 职责 | 运行方式 | 产物目录 |
|------|------|---------|---------|
| gorm_gen.go | 调用 `mysql_gen.GenCodeWithAll()` 读取 wallet 数据库所有表结构，生成 Go 代码 | `go run internal/gen/gorm_gen.go`（手动触发） | gorm_gen/ 下的 model/ + query/ + repo/ |

> 当前文件引用了 `ser-app`，实施时需改为 `ser-wallet`

### 3.5 gorm_gen/ — 自动生成产物

```
internal/gorm_gen/                   ⚠️ 全部自动生成，禁止手动编辑

| 4 | rate_log | rate_log.gen.go | rate_log.gen.go | ratelogr/ | 汇率日志 |
| 5 | operation_log | operation_log.gen.go | operation_log.gen.go | operationlogr/ | 操作日志 |
| 6 | wallet_balance | wallet_balance.gen.go | wallet_balance.gen.go | walletbalancer/ | 钱包余额 |
| 7 | wallet_transaction | wallet_transaction.gen.go | wallet_transaction.gen.go | wallettransactionr/ | 资金流水 |
| 8 | recharge_order | recharge_order.gen.go | recharge_order.gen.go | rechargeorderr/ | 充值订单 |
| 9 | withdraw_order | withdraw_order.gen.go | withdraw_order.gen.go | withdraworderr/ | 提现订单 |
| 10 | exchange_record | exchange_record.gen.go | exchange_record.gen.go | exchangerecordr/ | 兑换流水 |

### 3.6 cache/ — Redis 缓存层

```
internal/cache/
├── currency_cache.go       # 币种配置 + 启用列表缓存
└── rate_cache.go           # 汇率数据 + 交易锁汇缓存
```

| 文件 | 缓存内容 | 缓存策略 | 调用方 |
|------|---------|---------|--------|
| currency_cache.go | 全量启用币种列表、单币种配置详情 | 配置变更时主动失效 + TTL兜底 | currency_service、wallet_core_service |
| rate_cache.go | 各币种平台汇率、交易锁汇快照 | 汇率刷新时更新 + 短TTL | rate_service、recharge_service、exchange_service、withdraw_service |

> 为什么只有 2 个缓存文件而不是按表拆分：
> - 缓存聚焦高频读取的数据（币种配置 + 汇率），而非所有表
> - 钱包余额不建议缓存（资金数据一致性要求高，直接查 DB）
> - 订单类数据查询频率低且状态多变，缓存收益不大

### 3.7 rep/ — 自定义仓储层

```
internal/rep/
├── currency_repo.go        # 币种配置表 CRUD + 分页 + 状态变更
├── base_currency_repo.go   # 基准币种读写（全表单条记录）
├── rate_repo.go            # 汇率读写 + rate_log 日志写入
├── wallet_repo.go          # 余额增减 + 冻结/解冻（事务操作）
├── transaction_repo.go     # 流水写入 + 幂等校验 + 分页查询
├── recharge_repo.go        # 充值订单 CRUD + 状态流转
├── withdraw_repo.go        # 提现订单 CRUD + 状态流转
├── exchange_repo.go        # 兑换流水写入 + 查询
├── account_repo.go         # 提现账户 CRUD（3种类型统一）
└── preference_repo.go      # 用户币种偏好读写
```

| 文件 | 操作的表 | 核心职责 | 关键注意点 |
|------|---------|---------|-----------|
| currency_repo.go | currency_config | 分页列表、编辑配置、启禁用 | 禁用时需校验关联业务 |
| base_currency_repo.go | base_currency | 读取/一次性写入 | 全表最多一条记录 |
| rate_repo.go | currency_rate + rate_log | 读写汇率、记录汇率变更日志 | 跨两张表但领域紧密 |
| wallet_repo.go | wallet_balance | 余额加/减/冻结/解冻/确认扣除 | **核心**：所有操作必须在事务内 |
| transaction_repo.go | wallet_transaction | 流水写入 + 幂等查重 + 分页 | 只写不改，审计追溯核心 |
| recharge_repo.go | recharge_order | 创建订单、更新状态、重复检查 | 重复转账拦截逻辑在此层 |
| withdraw_repo.go | withdraw_order | 创建订单、审核状态更新 | 状态机7步流转 |
| exchange_repo.go | exchange_record | 写入兑换流水 | 相对简单 |
| account_repo.go | withdraw_account | 按类型读写提现账户 | 首次写入后不可修改 |
| preference_repo.go | user_currency_preference | 读取/切换用户当前币种 | upsert 模式 |

> **设计原则**：rep 层是对 gorm_gen/query 的组合封装，添加业务级的查询条件（如 `.Where(DeleteAt.Eq(0))`）、事务管理、分页封装等。不包含业务逻辑判断。

### 3.8 ser/ — 业务逻辑层

```
internal/ser/
├── currency_service.go         # 币种配置全流程
├── rate_service.go             # 汇率管理 + 定时任务
├── wallet_core_service.go      # E1~E6 核心资金操作 + C端余额查询
├── recharge_service.go         # 三种充值方式全流程
├── exchange_service.go         # 法币→BSB 兑换
├── withdraw_service.go         # 提现全流程 + 账户管理
└── record_service.go           # 四Tab记录查询 + 详情
```

| 文件 | 承载接口数 | 接口分组 | 依赖的 repo | 依赖的 cache | RPC 外调 |
|------|-----------|---------|------------|-------------|---------|
| currency_service.go | **20** | B端10 + C端3 + RPC暴露7 | currency_repo + base_currency_repo + rate_repo | currency_cache | — |
| rate_service.go | **2** | 手动刷新B7 + 定时任务T1 | rate_repo | rate_cache | 第三方汇率API |
| wallet_core_service.go | **9** | C端3（W1~W3）+ RPC暴露6（E1~E6） | wallet_repo + transaction_repo + preference_repo | — | — |
| recharge_service.go | **8** | C端8（W4~W11） | recharge_repo + transaction_repo | rate_cache | 财务F1（充值方式） |
| exchange_service.go | **2** | C端2（W12~W13） | exchange_repo + transaction_repo | rate_cache | — |
| withdraw_service.go | **9** | C端9（W14~W22） | withdraw_repo + account_repo + transaction_repo | rate_cache | 财务F2~F4 + 用户D9~D10 |
| record_service.go | **5** | C端5（W23~W27） | recharge_repo + withdraw_repo + exchange_repo + transaction_repo | — | — |

> **为什么币种配置用一个 service 文件承载 20 个接口而不拆分**：
> - B端 CRUD、C端列表、RPC暴露三组接口操作的都是同一组表（currency_config + base_currency + currency_rate）
> - 它们共享大量内部方法（如 `getCurrencyByCode()`、`checkCurrencyEnabled()`）
> - 拆分后需要大量跨文件方法调用或重复代码
> - 工程先例：ser-app/ser/banner.go 一个文件承载了 banner 的全部 CRUD（8个方法）

> **wallet_core_service.go 是整个系统最关键的文件**：
> - E1~E6 六个资金操作（查余额、加款、减款、冻结、解冻、确认扣除）是所有业务的核心
> - 被 recharge_service、exchange_service、withdraw_service 内部复用
> - 被财务模块通过 RPC 外部调用
> - 所有资金变动的最终归口

## 十、总结

### 10.1 关键设计决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 一个服务还是两个 | 一个 ser-wallet | 币种配置和钱包数据紧耦合，同服务方法调用优于 RPC |
| IDL 文件数量 | 8 个文件 | 对标 ser-live（9个/60方法），最符合工程模式 |
| ser/ 文件拆分 | 7 个文件 | 7 个业务子域各一个文件 |
| rep/ 文件拆分 | 10 个文件 | 12 表中 rate_log 合并到 rate_repo，operation_log 按需归属 |
| cache/ 文件数 | 2 个文件 | 只缓存高频读取数据（币种+汇率），余额不缓存 |
| E1~E6 归属 | wallet.thrift（与C端合并） | wallet.thrift 总共 9 个方法，体量适中 |
| R1~R7 归属 | currency_rpc.thrift（独立） | 调用方广泛（跨多团队），独立文件降低理解成本 |

### 10.2 与工程其他服务的体量对比

```
ser-wallet（设计）：53方法 / 8IDL / 7ser / 10rep / 12表 / ~52手写文件
├── 对标：ser-live    60+方法 / 9IDL / 大型服务
├── 对标：ser-user    50+方法 / 12IDL / 大型服务
└── 参照：ser-app     20+方法 / 5IDL / 中型服务（本次参照的模板）
```

### 10.3 待评审后可能的变更

| 可能变更 | 影响范围 | 当前处理 |
|---------|---------|---------|
| 需要独立 wallet_rpc.thrift | IDL +1 文件 | 当前 E1~E6 合并在 wallet.thrift |
| 需要 B 端钱包管理页面 | gate-back handler +1 文件，ser/ 可能 +1 | 当前 B 端只有币种配置 |
| 回调接口归属确认 | 可能 ser/ 需新增回调处理逻辑 | 当前假设银行/扫码回调走财务模块 |
| 币种配置拆分为独立服务 | 全部结构需拆分为两个 ser- 目录 | 当前合并在一个服务内 |
| 记录接口合并 | record.thrift 方法数减少 | 当前按 4Tab 分 4 个接口 |

> 以上变更均为"增量调整"，不改变目录结构的基本骨架。目录层级、分层模式、命名规范不会因评审细节调整而变化。
