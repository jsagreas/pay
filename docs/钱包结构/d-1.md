# ser-wallet 工程目录结构设计

> 文档定位：不涉及代码细节，只设计 ser-wallet 模块的**文件目录骨架**——有哪些目录、有哪些文件、每个文件干什么。让人拿到这份文档后，对整个工程"长什么样"有完整的方向感。
>
> 设计依据：ser-app（5 Service / 2 Rep / 1 Cache）、ser-buser（4 Service / 3 Rep）、ser-bcom（5 Service / 3 Rep）的实际目录结构 + 钱包模块的 IDL 设计（8 业务域 / 49 接口）+ 表结构设计（10 张表）。

## 三、顶层文件详解

### 3.1 目录树（顶层）

```
ser-wallet/
├── main.go
├── handler.go
├── go.mod
├── go.sum
├── .golangci.yml
├── .gitignore
└── sql/
    └── 01.table.sql
```

### 3.2 逐文件说明

| 文件 | 职责 | 工程参照 |
|------|------|---------|
| **main.go** | 服务启动入口。初始化日志 → 加载 ETCD 配置 → 初始化 Redis → 初始化 MySQL → 创建各 Service 实例 → 创建 Kitex RPC Server → 启动服务 | ser-app/main.go 完全一致 |
| **handler.go** | RPC 方法分发层。定义 `WalletServiceImpl` 结构体，持有所有 Service 实例引用，实现 IDL 中定义的 49 个 RPC 方法，每个方法体只做一件事：调用对应 Service 的方法 | ser-app/handler.go 完全一致 |
| **go.mod** | Go 模块声明。`module ser-wallet`，依赖 `common/pkg`、`common/rpc` 等 | ser-app/go.mod 格式一致 |
| **go.sum** | 依赖校验文件（自动生成） | — |
| **.golangci.yml** | Go Lint 规则配置 | 复制 ser-app 的即可 |
| **.gitignore** | Git 忽略规则 | 复制 ser-app 的即可 |
| **sql/01.table.sql** | 全量建表 DDL（10 张表），用于 DBA 初始化数据库或开发本地建表 | ser-buser/sql/01.table.sql 先例 |

### 3.3 main.go 的初始化流程

```
main()
  │
  ├── 1. lg.InitKLog()                          ← 初始化日志
  ├── 2. etcd.LoadAndWatch("/slg/")              ← 加载 ETCD 配置
  ├── 3. cfg.InitRedis()                         ← Redis 连接
  ├── 4. cfg.InitMySQL()                         ← MySQL 连接 + GORM query.SetDefault
  ├── 5. cron.Init()                             ← 初始化定时任务调度器
  │
  ├── 6. 创建 Service 实例（依赖注入）
  │      ├── currencyService
  │      ├── exchangeRateService
  │      ├── walletService
  │      ├── depositService
  │      ├── withdrawService
  │      ├── withdrawAccountService
  │      ├── exchangeService
  │      ├── recordService
  │      ├── walletRpcService
  │      ├── enumService
  │      └── cronService
  │
  ├── 7. cronService.RegisterJobs()              ← 注册定时任务
  │
  ├── 8. srv := walletservice.NewServer(         ← 创建 Kitex RPC 服务
  │          &WalletServiceImpl{...},
  │          rpc.InitRpcServerParams(...)
  │      )
  │
  └── 9. srv.Run()                               ← 启动服务
```

### 3.4 handler.go 的结构体

```
WalletServiceImpl
  │
  ├── currencyService         *ser.CurrencyService
  ├── exchangeRateService     *ser.ExchangeRateService
  ├── walletService           *ser.WalletService
  ├── depositService          *ser.DepositService
  ├── withdrawService         *ser.WithdrawService
  ├── withdrawAccountService  *ser.WithdrawAccountService
  ├── exchangeService         *ser.ExchangeService
  ├── recordService           *ser.RecordService
  ├── walletRpcService        *ser.WalletRpcService
  └── enumService             *ser.EnumService
```

**handler.go 中每个方法的模式（统一）：**

```
func (s *WalletServiceImpl) CreateCurrency(ctx context.Context, req *ser_wallet.CreateCurrencyReq) (*ser_wallet.CreateCurrencyResp, error) {
    return s.currencyService.CreateCurrency(ctx, req)
}
```

handler 只做转发，不做业务逻辑。

## 九、internal/cache/ — 缓存层

### 9.1 目录树

```
internal/cache/
├── currency.go            # 币种列表缓存
└── exchange_rate.go       # 汇率缓存
```

### 9.2 逐文件说明

| 文件 | 缓存的数据 | 缓存策略 | Redis Key |
|------|-----------|---------|-----------|
| **currency.go** | 启用的币种列表（C 端高频读取） | Cache-Aside：读命中返回，未命中查 DB 回写；写入/修改时删除缓存 | `wallet:currency:list` |
| **exchange_rate.go** | 币种对的实时汇率 | Cache-Aside + 主动写入：汇率更新时同步写 Redis；读未命中查 DB 回写 | `wallet:rate:{fromId}:{toId}` |

**为什么只缓存这两个：**
- 余额数据不缓存——资金操作以 DB 为准，缓存余额有一致性风险
- 充值/提现订单不缓存——写多读少，缓存收益不大
- 币种和汇率是"读多写极少"的典型场景，缓存收益最高

### 10.2 逐文件说明

| 文件 | 对应的表 | 核心方法 | 特殊能力 |
|------|---------|---------|---------|
| **currency.go** | currency | `ExistsByCode` / `Create` / `UpdateFields` / `PageCurrency` / `ListEnabled` / `GetByID` | 编码唯一性校验 |
| **exchange_rate_log.go** | exchange_rate_log | `Create` / `GetLatestRate` / `PageRateLog` | 按币种对查最新生效汇率 |

| **deposit_order.go** | deposit_order | `Create` / `GetByOrderNo` / `UpdateStatus` / `PageDepositOrder` / `ListPendingTimeout` | `ListPendingTimeout` — 扫描超时待支付订单 |
| **deposit_config.go** | deposit_config | `GetByCurrencyId` / `Save` | 按币种唯一的配置（Upsert） |
| **withdraw_order.go** | withdraw_order | `Create` / `GetByOrderNo` / `UpdateStatus` / `PageWithdrawOrder` / `ListProcessingTimeout` / `UpdateAudit` | `UpdateAudit` — 条件更新审核状态（WHERE status=当前） |
| **withdraw_config.go** | withdraw_config | `GetByCurrencyId` / `Save` | 按币种唯一的配置（Upsert） |
| **withdraw_account.go** | withdraw_account | `Create` / `ListByUserId` / `GetByID` / `DeleteByID` / `SetDefault` / `CountByUserId` | 敏感字段自动加解密（EncryptedString） |
| **exchange_record.go** | exchange_record | `Create` / `GetByRecordNo` / `PageExchangeRecord` | 含汇率快照字段 |

### 10.3 关键说明：wallet_account.go

这是整个 rep 层**最关键的文件**，因为它包含行锁操作：

```
GetForUpdate(ctx, tx, userId, currencyId, walletType)
    → SELECT * FROM wallet_account
      WHERE user_id=? AND currency_id=? AND wallet_type=?
      FOR UPDATE

UpdateBalance(ctx, tx, id, availableDelta, frozenDelta)
    → UPDATE wallet_account
      SET available_balance = available_balance + ?,
          frozen_balance = frozen_balance + ?
      WHERE id = ?
```

这两个方法**只在事务内被调用**（由 domain/balance_engine.go 管理事务），不会被 Service 层直接调用。

## 十一、internal/domain/ — 领域核心逻辑

### 11.1 目录树

```
internal/domain/
└── balance_engine.go       # 余额引擎
```

### 11.2 为什么单独建 domain 层

| 维度 | 说明 |
|------|------|
| **ser-app 有吗** | 没有。因为 ser-app 的业务没有跨 Repository 的复杂事务 |
| **为什么 ser-wallet 需要** | 余额变更是"跨两张表的原子事务"（wallet_account + transaction_log），且被 6 个 Service 共同调用（充值、提现、兑换、返奖、调账、冻结/解冻）。如果每个 Service 各写一遍事务逻辑，会有大量重复且容易出错 |
| **balance_engine.go 的定位** | 封装"余额变更"这一个核心操作，对外提供统一的 `ModifyBalance()` 方法，内部管理事务、行锁、流水写入、幂等校验 |

### 11.3 balance_engine.go 的职责

```
BalanceEngine
  │
  └── ModifyBalance(ctx, params) error
        │
        ├── BEGIN 事务
        ├── ① 行锁：rep.WalletAccount.GetForUpdate(tx, ...)
        ├── ② 校验：余额是否充足（扣款/冻结场景）
        ├── ③ 更新：rep.WalletAccount.UpdateBalance(tx, ...)
        ├── ④ 写流水：rep.TransactionLog.Create(tx, ...)
        │      → 唯一索引自动保障幂等
        ├── COMMIT
        └── ⑤ 清除余额缓存（如有）
```

**被谁调用：**

| 调用方 Service | 调用场景 |
|---------------|---------|
| wallet_rpc_service.go | CreditBalance / DebitBalance / FreezeBalance / UnfreezeBalance |
| deposit_service.go | 充值到账加款 |
| withdraw_service.go | 提现申请冻结 / 驳回解冻 |
| exchange_service.go | 兑换扣源钱包 + 加目标钱包（同一事务内调两次） |
| wallet_service.go | B 端手动调账 |

### 12.4 cron_service.go 详解

| 项 | 说明 |
|----|------|
| **文件定位** | 不对应任何 IDL，是内部的定时任务管理器 |
| **为什么单独建文件** | 工程中定时任务的注册和执行逻辑应集中管理，避免散落在各个 Service 中 |
| **方法** | `RegisterJobs()` — 在 main.go 中调用一次，注册所有 cron 任务 |

**注册的定时任务：**

| 汇率刷新 | 按运营配置 | `exchangeRateService.RefreshFromApi()` |

## 十五、与工程已有服务的对标

| 维度 | ser-app | ser-buser | ser-bcom | **ser-wallet** |
|------|---------|-----------|----------|---------------|
| 业务域数 | 4 | 3 | 3 | **8** |
| IDL 方法数 | ~16 | ~20 | ~14 | **49** |
| 表数 | 2 | 4 | 3 | **10** |
| Service 数 | 5 | 4 | 5 | **11** |
| Repository 数 | 2 | 3 | 3 | **10** |
| Cache 文件数 | 1 | 0 | 0 | **2** |
| Domain 文件数 | 0 | 0 | 0 | **1** |
| Enum 文件数 | 4 | 3 | 3 | **7** |
| 手写文件总数 | ~20 | ~20 | ~18 | **~51** |
| 定时任务 | 无 | 无 | 无 | **5 个** |
| 复杂度等级 | 中 | 中 | 中 | **高** |

**差异的原因：**
- ser-wallet 有 8 个业务域（其他服务 3-4 个）
- ser-wallet 涉及资金操作，需要独立的 domain 层（余额引擎）
- ser-wallet 有定时任务（其他服务没有）
- ser-wallet 的枚举更多（交易类型就有 10 个值）

**没有差异的地方：**
- 目录结构规范完全一致（cfg / errs / enum / gen / gorm_gen / rep / ser）
- 分层模式完全一致（Handler → Service → Repository → Model）
- 配置初始化模式完全一致（sync.Once + ETCD）
- 命名规范完全一致

## 十六、总结

### 一张表看懂全貌

| 层 | 目录 | 文件数 | 一句话职责 |
|----|------|--------|-----------|
| 入口 | 顶层 | 2（main + handler） | 启动服务 + 分发请求 |
| 配置 | cfg/ | 2 | MySQL + Redis 连接 |
| 契约 | errs/ + enum/ | 8 | 错误码 + 业务枚举 |
| 生成 | gen/ + gorm_gen/ | 22 | GORM 模型和查询 DSL |
| 缓存 | cache/ | 2 | 币种 + 汇率缓存 |
| 数据 | rep/ | 10 | 每张表一个 Repository |
| 领域 | domain/ | 1 | 余额引擎（事务 + 行锁 + 流水） |
| 业务 | ser/ | 11 | 每个业务域一个 Service |
| DDL | sql/ | 1 | 建表脚本 |
| **合计** | — | **~59** | — |

### 骨架设计的三个关键决策

1. **新增 domain/ 层**（ser-app 没有）——因为余额引擎被 6 个 Service 共用，必须抽出来避免重复
2. **新增 cron_service.go**（ser-app 没有）——集中管理 5 个定时任务
3. **汇率管理独立为 exchange_rate_service.go**——虽然 IDL 中汇率在 currency.thrift 里，但汇率管理的逻辑（API 拉取、缓存刷新、日志记录）足够独立，拆出来避免 currency_service.go 过于臃肿
