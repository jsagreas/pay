# ser-wallet 工程结构设计

> 设计依据：4个现有服务的代码级结构分析（ser-item/ser-buser/ser-app/ser-live）
> 对标服务：ser-item（多Struct模式+domain层）+ ser-live（最大服务IDL组织方式）
> 业务规模：46个RPC方法、10张数据库表、7个IDL文件、6个业务域
> 原则：100%遵循现有工程规范，不引入新模式、不发明新结构

# 二、分层详解 — 根目录层

## 2.1 main.go — 服务入口

```
职责: 初始化链 → 创建Service实例 → 组装handler → 启动Kitex RPC Server
模式: 与 ser-item/main.go 完全一致
```

**初始化链**:

```
lg.InitKLog()
    → etcd.InitEtcd()
        → flag.Parse(port)
            → ser.NewXxxService() × 6
                → handler 组装
                    → walletservice.NewServer().Run()
```

**注入的 Service 实例**:

| Service | 构造函数 | 职责 |
|---------|---------|------|
| CurrencyService | `ser.NewCurrencyService()` | B端币种配置 |
| WalletService | `ser.NewWalletService()` | C端首页/余额/记录 |
| DepositService | `ser.NewDepositService()` | C端充值 |
| WithdrawService | `ser.NewWithdrawService()` | C端提现 |
| ExchangeService | `ser.NewExchangeService()` | C端兑换 |
| RpcService | `ser.NewRpcService()` | 财务回调RPC |

**选择多Struct而非单Struct的理由**:

ser-buser 用单个 `UserService` struct 涵盖 29 个方法（user/role/menu 共用一个 repo），因为这 3 个子域强耦合（共享 sys_user 表）。

ser-wallet 的 6 个子域**各自持有不同的 repo 和 domain 依赖**：
- CurrencyService 只需要 CurrencyRepo + CurrencyCache
- DepositService 需要 DepositRepo + BalanceRepo + FinanceRPC
- RpcService 需要 BalanceRepo + DepositRepo + WithdrawRepo

多 Struct 让每个 Service 的依赖关系清晰可见，而不是一个 struct 持有所有 repo 导致"谁都能调谁"。

---

## 2.2 handler.go — RPC 纯路由层

```
职责: 实现 Kitex 生成的 WalletServiceImpl 接口，每个方法一行代码委托给对应 Service
模式: 与所有现有服务一致 — 单文件，package main，零业务逻辑
方法数: 46 个（全部为一行转发）
```

**结构预览**:

```go
type WalletServiceImpl struct {
    currencyService *ser.CurrencyService
    walletService   *ser.WalletService
    depositService  *ser.DepositService
    withdrawService *ser.WithdrawService
    exchangeService *ser.ExchangeService
    rpcService      *ser.RpcService
}
```

**方法分布**:

| 注入的Service | 转发的方法数 | 对应IDL文件 | 方法示例 |
|--------------|-------------|------------|---------|
| currencyService | 9 | currency.thrift | GetCurrencyList, SetExchangeRate |
| walletService | 10 | wallet.thrift | GetWalletHome, GetBalanceList, GetDepositRecords |
| depositService | 7 | deposit.thrift | GetDepositMethods, CreateDeposit |
| withdrawService | 8 | withdraw.thrift | GetWithdrawMethods, CreateWithdraw |
| exchangeService | 5 | exchange.thrift | PreviewExchange, CreateExchange |
| rpcService | 7 | wallet_rpc.thrift | DepositCallback, WithdrawConfirm |

**为什么不拆分成多个 handler 文件**:

所有 4 个参考服务（含最大的 ser-live 36 方法）都使用单个 handler.go。handler 层是纯转发，每个方法 1 行代码，46 个方法约 200 行，可读性完全没问题。拆分 handler 反而破坏工程一致性。

**拆成 2 个文件的依据**: ser-buser 的 enum/ 有 3 个文件（buser.go + def.go + enum_registry.go），ser-app 有 4 个文件。按业务域拆分，避免单文件过长。ser-wallet 的钱包域和币种域枚举天然分离（不同表、不同接口使用）。

**暂不需要 enum_registry.go**: 枚举注册表是 B 端动态下拉菜单用的（ser-buser 的角色列表、ser-app 的分类列表）。ser-wallet 的枚举（币种类型、订单状态等）由前端写死映射，不需要后端动态提供选项。如后续有需求再追加。

## 4.2 三层输出对照

| 层 | 目录 | 每张表生成 | 使用方式 |
|----|------|-----------|---------|

| repo | gorm_gen/repo/xxxr/ | Create/Update/Delete/GetOneById/GetPageByConds... | ser层直接调用简单单条查询，复杂查询走rep层 |

# 五、分层详解 — 业务 Repository 层（rep/）

## 5.1 定位

```
位置: internal/rep/
性质: 手写，承载复杂业务查询（分页、JOIN、事务、乐观锁）
上层: 被 ser/ 层调用
下层: 使用 gorm_gen/query/ 的链式查询 + gorm_gen/model/ 的结构体
特征: 每个文件一个独立 Repo struct（区别于 ser-buser 的同struct多文件扩展）
```

## 5.2 文件 → Repo → 管辖表 → 典型方法

| 文件 | Struct | 管辖的表 | 核心方法（预估） |
|------|--------|---------|----------------|
| currency.go | CurrencyRepo | currency_config, exchange_rate, currency_log, rate_log | GetEnabledList, GetByCode, CreateCurrency, UpdateCurrency, SetRate, GetRate, PageCurrencyLog, PageRateLog |
| balance.go | BalanceRepo | user_balance, balance_flow | GetBalance, GetOrCreateBalance, CreditBalance(乐观锁+), DebitBalance(乐观锁-), FreezeBalance, UnfreezeBalance, ConfirmFreeze, WriteFlow, PageFlowByUser |
| deposit.go | DepositRepo | deposit_order | CreateOrder, UpdateStatus, GetByOrderNo, CountDuplicate, PageByUser, PageByAdmin |
| withdraw.go | WithdrawRepo | withdraw_order, withdraw_account | CreateOrder, UpdateStatus, GetByOrderNo, PageByUser, SaveAccount, GetAccount, PageByAdmin |
| exchange.go | ExchangeRepo | exchange_order | CreateOrder, GetByOrderNo, PageByUser |

## 5.3 为什么选择"多Struct"而非 ser-buser 的"单Struct多文件"

| 维度 | ser-buser（单 UserRepo） | ser-wallet 选择（多 Repo） |
|------|------------------------|---------------------------|
| 表的关联性 | sys_user/role/menu 强关联（JOIN） | 10张表分属不同业务域，极少JOIN |
| 初始化依赖 | 所有方法共享同一个 db + redis | 不同 Repo 可能有不同的缓存策略 |
| 事务边界 | 用户+角色常在同一事务 | 充值和提现永远不在同一事务 |
| Service 消费 | 1 个 Service 消费 1 个 Repo | 多个 Service 消费不同 Repo 组合 |

**关键区别**: CurrencyRepo 只被 CurrencyService 使用，DepositRepo 被 DepositService 和 RpcService 共用，BalanceRepo 被几乎所有 Service 共用。多 Struct 让这种"多对多"依赖关系通过构造函数注入清晰表达。

## 5.4 balance.go 特殊说明

这是**整个服务最核心的 repo 文件**。所有资金操作的 SQL 都在这里：

```
余额操作矩阵（balance.go 需要实现的原子操作）:

  操作          SQL核心                            调用场景
  ─────────────────────────────────────────────────────────────
  查余额        SELECT ... WHERE user+currency+type    所有Service
  加可用        available = available + amount          充值入账/兑换入/人工加
  减可用(乐观锁) available = available - amount          兑换扣/人工减
                WHERE available >= amount
  冻结          available -= N, frozen += N             提现创建
                WHERE available >= N
  解冻退回      available += N, frozen -= N             提现失败
  冻结确认扣除  frozen -= N                             提现成功
  写流水        INSERT balance_flow                     每次余额变动必写
```

## 6.4 Service 方法内部标准流程

参考 ser-item/ser-buser 的已验证模式，每个 Service 方法遵循统一结构：

```
1. 入口日志       lg.Log().CtxInfof(ctx, "方法名 ...")
2. 入参校验       domain.ValidateXxx → ret.BizErr(ctx, errs.XxxErr)
3. 缓存尝试       domain.CurrencyCache.Get（仅读操作）
4. 业务逻辑       rep.XxxRepo.DoSomething / domain.FinanceRPC.Call
5. 结果转换       domain.XxxMapper.ToResp
6. 异步善后       defer go { ClearCache / AddOperLog }（仅写操作）
7. 返回           return resp, nil
```

# 七、分层详解 — 领域层（domain/）

## 7.1 定位

```
位置: internal/domain/
性质: 手写，承载跨Service共享的"工具型"逻辑
上层: 被 ser/ 层调用
下层: 调用 common/rpc（跨服务RPC）+ common/cache（Redis）+ gorm_gen/model（结构体）
规则: domain 不依赖 rep 层，不直接操作数据库
```

**设计依据**: ser-item 有 9 个 domain 文件（common/cache/mapper/validator/log_builder/rpc_wrapper），是工程中唯一使用 domain 层的服务——因为它的跨关注点逻辑（缓存管理、DTO转换、校验接口、跨服务调用）足够复杂，值得独立。ser-wallet 的复杂度远超 ser-item（金额精度计算、多种缓存、3个外部RPC、6种订单DTO转换），domain 层是必须的。

## 10.2 层级职责一句话总结

| 层 | 目录 | 一句话 |
|----|------|--------|
| 入口 | main.go | 初始化链 + 服务启动，不含业务逻辑 |
| 路由 | handler.go | 46个方法 → 6个Service的纯转发 |
| 业务 | ser/ | 所有业务逻辑：入参校验、流程编排、事务协调、缓存策略 |
| 领域 | domain/ | 跨Service共享的工具：校验函数、金额计算、缓存管理、DTO转换、外部RPC封装 |
| 持久化 | rep/ | 手写的业务SQL：分页查询、JOIN、事务、乐观锁条件更新 |
| 基础 | gorm_gen/ | 自动生成：model结构体、query链式查询、通用CRUD |
| 配置 | cfg/ | MySQL + Redis 初始化（sync.Once 单例） |
| 常量 | enum/ + errs/ | 枚举定义 + 错误码定义 |
| 测试 | test/ | 集成测试（直连真实环境） |

# 十二、设计决策溯源

每个结构设计选择都有工程事实或需求依据：

| 设计决策 | 选择 | 工程依据 |
|---------|------|---------|
| handler 单文件 | 1 个 handler.go | ser-item(11方法)/ser-buser(29方法)/ser-app(19方法) 全部单文件 |
| Service 多Struct | 6 个独立 struct | ser-item 模式（3 struct），因为 6 个子域各自有独立的 repo 依赖组合 |
| domain 层存在 | 有 | ser-item 的 domain/(9文件) — 金额工具/缓存/mapper/RPC封装复杂度远超 ser-item |
| Repo 多Struct | 5 个独立 struct | 10张表分属不同业务域，极少 JOIN，各 Service 按需注入 |
| 枚举 2 个文件 | wallet.go + currency.go | ser-buser 3 个文件、ser-app 4 个文件，按域拆分是已验证模式 |
| 错误码分域段 | 每域间隔 100 | ser-app 模式（Banner=000+, Category=100+, Strategy=200+） |
| SQL 按模块拆 | 2 个文件 | 币种配置(第二期) 和 钱包(第三期) 不同阶段建表 |
| IDL 7 个文件 | 1 service + 6 entity | ser-live(9文件)/ser-user(11文件)，按业务域拆分是大服务标准做法 |
| test 按域拆 | 7 个文件 | ser-item(4文件)/ser-buser(3文件)，与 Service 1:1 对应 |
| gorm_gen.go 已存在 | 复用 | 已在 ser-wallet 目录内，指向 ser-app 的 ETCD DB key（后续改为 wallet key） |
| .golangci.yml 已存在 | 复用 | 与其他服务相同的 lint 规则集 |
