## 2. 根目录文件详解

ser-wallet 根目录 **3 个 Go 文件**，与所有现有服务完全一致：

### 2.1 文件清单

| 文件 | 职责 | 代码规模 |
|------|------|---------|
| `main.go` | 服务启动入口 | ~60-80 行 |
| `handler.go` | Kitex Handler 委托层 | ~200-300 行 |
| `go.mod` | 模块定义 | ~10-15 行 |

### 2.2 main.go

**职责**：按固定顺序初始化基础设施，创建 Service 实例，启动 Kitex RPC Server。

**启动序列**（遵循全工程统一模式）：

```
1. lg.InitKLog()                          → 初始化日志
2. etcd.InitEtcd()                        → 初始化 ETCD 连接
3. cfg.InitMysql()                        → 初始化 TiDB 连接 + gorm_gen 绑定
4. cfg.InitRedis()                        → 初始化 Redis 连接
5. 创建 Service 实例                       → ser.NewWalletService() 等
6. 启动汇率定时任务                         → rateSer.StartCron()
7. flag + ETCD 获取端口                    → 监听端口
8. walletservice.NewServer(handler, opts)  → 启动 Kitex Server
9. srv.Run()                              → 阻塞运行
```

**工程参照**：ser-app/main.go、ser-item/main.go（完全相同的结构，替换服务名即可）

### 2.3 handler.go

**职责**：实现 Kitex 生成的 `WalletService` 接口。每个方法是一行委托调用，**不包含任何业务逻辑**。

**结构**：

```
type WalletServiceImpl struct {
    walletSer   *ser.WalletService      // 钱包信息
    rechargeSer *ser.RechargeService     // 充值
    withdrawSer *ser.WithdrawService     // 提现
    exchangeSer *ser.ExchangeService     // 兑换
    recordSer   *ser.RecordService       // 记录
    accountSer  *ser.AccountService      // 提现账户
    currencySer *ser.CurrencyService     // 币种配置
    rateSer     *ser.RateService         // 汇率引擎
    rpcSer      *ser.RpcService          // 对外 RPC
}
```

每个接口方法格式统一：

```
func (s *WalletServiceImpl) GetWalletInfo(ctx, req) (resp, err) {
    return s.walletSer.GetWalletInfo(ctx, req)
}
```

**约 37 个方法**（C 端 ~17 + B 端 ~8 + RPC ~12），每个方法 3 行代码。

**工程参照**：ser-app/handler.go（AppServiceImpl 持有 ser 指针，一行委托）

### 2.4 go.mod

**职责**：声明模块路径和 Go 版本。

```
module ser-wallet

go 1.25.5
```

依赖解析由 go.work 统一管理，go.mod 保持极简。

**工程参照**：ser-app/go.mod（同样只有 module + go version，无 require）

## 3. internal/ 目录详解

### 3.1 总览

```
internal/
├── cfg/        基础设施初始化（2 文件）
├── enum/       枚举常量定义（1 文件）
├── errs/       错误码定义（1 文件）
├── domain/     资金操作原子层（6 文件）          ← ser-wallet 特有
├── ser/        Service 业务逻辑层（9 文件）
├── rep/        手写 Repository 层（5 文件）
├── cache/      缓存管理（1 文件）
├── gen/        代码生成入口（1 文件）
└── gorm_gen/   自动生成代码（~33 文件）
```

**手写文件数**：25 个
**自动生成文件数**：~33 个
**合计**：~58 个

### 3.2 cfg/ — 基础设施初始化

```
cfg/
├── mysql.go    # TiDB 连接初始化 + gorm_gen query 绑定
└── redis.go    # Redis 连接初始化
```

| 文件 | 职责 | 核心逻辑 |
|------|------|---------|
| mysql.go | 初始化数据库连接 | `sync.Once` → 从 ETCD 读 db_wallet 连接串 → `mysql.Init(opt)` → `query.SetDefault(db)` |
| redis.go | 初始化 Redis 连接 | `sync.Once` → 从 ETCD 读 Redis 配置 → `rds.Init(opt)` |

**工程参照**：ser-app/internal/cfg/mysql.go（完全相同模式，替换 namesp.EtcdWalletDb）

### 3.3 enum/ — 枚举常量定义

```
enum/
└── enums.go    # 全部枚举常量
```

| 枚举 | 值域 | 用途 |
|------|------|------|
| WalletType | 1=中心 / 2=奖励 / 3=主播 / 4=代理 / 5=场馆 | 钱包类型标识 |
| WalletStatus | 1=正常 / 2=冻结 / 3=关闭 | 钱包状态 |
| FlowDirection | 1=入账 / 2=出账 / 3=冻结 / 4=解冻 | 流水方向 |
| BizType | 1=充值 / 2=提现 / 3=兑换 / ... / 10=确认扣款 | 流水业务类型 |
| RechargeStatus | 1=待支付 / 2=成功 / 3=失败 / 4=取消 | 充值订单状态 |
| WithdrawStatus | 1=待风控 / 2=风控驳回 / ... / 7=出款成功 | 提现订单状态（7 种） |
| FreezeStatus | 1=冻结中 / 2=已确认扣款 / 3=已解冻 | 冻结明细状态 |
| CurrencyType | 1=法币 / 2=加密货币 / 3=平台币 | 币种分类 |
| AccountType | 1=银行卡 / 2=电子钱包 / 3=USDT 地址 | 提现账户类型 |

**为什么只用一个文件**：枚举总量不大（~10 组，~50 个常量值），放一个文件方便查找。全工程现有服务也是如此。

### 3.4 errs/ — 错误码定义

```
errs/
└── code.go     # 6022xxx 系列错误码
```

**错误码格式**：`6{NNN}{MMM}` — 其中 NNN=022（服务编号），MMM=子模块序号。

| 段号 | 范围 | 子模块 | 示例 |
|------|------|--------|------|
| 0 | 6022001-6022099 | 钱包核心 | 余额不足、钱包不存在、幂等冲突 |
| 1 | 6022100-6022199 | 充值 | 订单不存在、重复充值、回调金额不匹配 |
| 2 | 6022200-6022299 | 提现 | 超出限额、KYC 未认证、冻结失败 |
| 3 | 6022300-6022399 | 兑换 | 汇率获取失败、余额不足、币种不支持 |
| 4 | 6022400-6022499 | 记录 | 记录不存在、参数无效 |
| 5 | 6022500-6022599 | 提现账户 | 账户数量上限、账号格式无效 |
| 6 | 6022600-6022699 | 币种配置 | 币种已存在、基准币种已锁定 |
| 7 | 6022700-6022799 | 汇率 | API 拉取失败、汇率异常 |

**工程参照**：ser-app 用 `6011xxx`、ser-item 用 `6010xxx`，完全一致的分段规则。

### 3.5 domain/ — 资金操作原子层

```
domain/
├── credit.go          # CreditWallet — 上账原子操作
├── debit.go           # DebitWallet — 扣款原子操作
├── freeze.go          # FreezeBalance — 冻结原子操作
├── unfreeze.go        # UnfreezeBalance — 解冻原子操作
├── confirm_debit.go   # ConfirmDebit — 确认扣款原子操作
└── common.go          # 公共辅助函数
```

**为什么 ser-wallet 需要 domain/ 层（其他服务大多没有）**：

钱包的资金操作函数具有**双重身份**：
- 作为 **RPC 接口** 被财务等外部服务远程调用
- 作为 **内部函数** 被充值、提现、兑换等业务流程直接调用

如果放在 ser/ 层，会出现 Service 之间相互调用的问题。抽到 domain/ 层，所有 Service 都可以直接调用，不产生循环依赖。

| 文件 | 职责 | 核心保证 |
|------|------|---------|
| credit.go | 加钱到指定钱包 | 事务内：行锁 → 更新 balance → 写 flow → 幂等校验 |
| debit.go | 从指定钱包扣钱 | 事务内：行锁 → 余额校验 → 更新 balance → 写 flow |
| freeze.go | 按优先级冻结余额 | 事务内：锁中心→锁奖励 → 分配冻结额 → 写 freeze_detail → 写 flow |
| unfreeze.go | 按冻结明细解冻 | 事务内：查 detail → 锁行 → 减 frozen → 更新 detail → 写 flow |
| confirm_debit.go | 冻结转正式扣款 | 事务内：查 detail → 锁行 → 减 balance+frozen → 更新 detail → 写 flow |
| common.go | 公共辅助 | 加锁排序算法（currency_code ASC → wallet_type ASC）、流水记录构建器、幂等 key 生成 |

**工程参照**：ser-item/internal/domain/（item_validator.go、item_mapper.go 等），ser-wallet 的 domain 层更聚焦于资金原子操作。

### 3.6 ser/ — Service 业务逻辑层

```
ser/
├── wallet_ser.go      # 钱包信息、币种切换
├── recharge_ser.go    # 充值业务
├── withdraw_ser.go    # 提现业务
├── exchange_ser.go    # 兑换业务
├── record_ser.go      # 记录查询
├── account_ser.go     # 提现账户管理
├── currency_ser.go    # 币种配置 CRUD
├── rate_ser.go        # 汇率引擎 + 定时任务
└── rpc_ser.go         # 对外 RPC 接口逻辑
```

每个文件对应一个 Service struct，由 handler.go 持有并委托调用。

| 文件 | Service 结构体 | 包含的接口方法 | 依赖关系 |
|------|---------------|--------------|---------|
| wallet_ser.go | WalletService | GetWalletInfo, SwitchCurrency | domain/、cache/、gorm_gen/repo/ |
| recharge_ser.go | RechargeService | GetRechargeMethodList, CreateRechargeOrder, RechargeCallback, GetUsdtAddress, CheckRepeatRecharge, GetBonusList | domain/credit、财务 RPC |
| withdraw_ser.go | WithdrawService | GetWithdrawMethodList, CreateWithdrawOrder | domain/freeze、ser-kyc RPC、财务 RPC |
| exchange_ser.go | ExchangeService | GetExchangeInfo, DoExchange | domain/credit+debit、currency_ser（汇率） |
| record_ser.go | RecordService | GetRecordList, GetRecordDetail | gorm_gen/repo/（只读查询） |
| account_ser.go | AccountService | GetWithdrawAccountList, SaveWithdrawAccount, DeleteWithdrawAccount | ser-kyc RPC（KYC 姓名） |
| currency_ser.go | CurrencyService | PageCurrency, GetCurrencyDetail, CreateCurrency, UpdateCurrency, StatusCurrency, SetBaseCurrency, GetRateLogList | ser-blog RPC（审计日志） |
| rate_ser.go | RateService | RefreshRate, StartCron（内部方法） | 三方汇率 HTTP API × 3 |
| rpc_ser.go | RpcService | CreditWallet, DebitWallet, GetBalance, FreezeBalance, UnfreezeBalance, ConfirmDebit, InitWallet, BatchGetBalance, GetWalletFlow, GetCurrencyList, GetExchangeRate, GetBaseCurrency | domain/（全部 5 个原子函数） |

**文件拆分原则**：一个子模块 = 一个 Service 文件。rpc_ser.go 单独拆出是因为 RPC 接口的调用方是外部服务，与 C/B 端接口的调用方不同，逻辑边界清晰。

### 3.7 rep/ — 手写 Repository 层

```
rep/
├── wallet_rep.go      # 钱包相关自定义查询
├── flow_rep.go        # 流水相关自定义查询
├── order_rep.go       # 充值/提现订单自定义查询
├── exchange_rep.go    # 兑换记录自定义查询
└── currency_rep.go    # 币种配置自定义查询
```

**为什么需要手写 rep/ 层？**

gorm_gen 生成的 repo/ 提供标准 CRUD（Create、GetOneById、GetPageByConds 等），但以下场景需要手写：

| 文件 | 包含的自定义查询 | 为什么 gorm_gen 不够 |
|------|----------------|-------------------|
| wallet_rep.go | `LockWallet(tx, userId, currency, walletType)` — SELECT FOR UPDATE | gorm_gen 不生成悲观锁查询 |
| wallet_rep.go | `GetUserWalletSummary(userId, currency)` — 同时查中心 + 奖励余额 | 需要多行聚合 |
| flow_rep.go | `GetFlowsByUserCurrencyPage(userId, currency, bizTypes, cursor)` — 游标分页 | gorm_gen 只有 offset 分页 |
| flow_rep.go | `SumFlowsByWallet(walletId)` — 对账重算余额 | 需要 SUM 聚合 |
| order_rep.go | `GetRecentRecharges(userId, currency, amount, since)` — 重复转账检测 | 组合条件 + 时间窗口 |
| order_rep.go | `CountTodayWithdraw(userId)` — 当日提现次数和金额 | 日期范围 + SUM |
| exchange_rep.go | `GetExchangeRecordPage(userId, fromCurrency, cursor)` — 游标分页 | gorm_gen 不支持 |
| currency_rep.go | `GetAllEnabledCurrencies()` — 缓存预热用 | 虽然简单但语义明确更好 |

**工程参照**：ser-app/internal/rep/banner_repo.go（NewBannerRepo struct，使用 query.Q.Banner 进行复杂查询）

### 3.8 cache/ — 缓存管理

```
cache/
└── keys.go     # Redis Key 定义 + 缓存辅助函数
```

**职责**：统一管理 ser-wallet 的所有 Redis Key 定义和缓存读写辅助函数。

| Key 模式 | 用途 | TTL |
|---------|------|-----|
| `wallet:bal:{uid}:{currency}` | 用户余额缓存（中心 + 奖励） | 5 分钟 |
| `wallet:currency:list` | 启用币种列表缓存 | 10 分钟 |
| `wallet:rate:{currency}` | 平台汇率缓存（入款 + 出款） | 由汇率更新事件删除 |
| `wallet:user:currency:{uid}` | 用户当前选择的币种偏好 | 7 天 |
| `wallet:lock:withdraw:{uid}` | 提现操作分布式锁 | 30 秒 |
| `wallet:lock:exchange:{uid}` | 兑换操作分布式锁 | 30 秒 |
| `wallet:lock:recharge:{uid}` | 充值操作分布式锁 | 30 秒 |

**工程参照**：ser-app/internal/cache/（缓存管理独立目录），common/pkg/consts/redis_key/redis_key.go（全局 Key 定义示例）

### 3.9 gen/ — 代码生成入口

```
gen/
└── gorm_gen.go     # gorm_gen 配置
```

**职责**：连接数据库 → 调用 `mysql_gen.GenCodeWithAll()` → 自动生成 model/ + query/ + repo/。

**运行方式**：`cd ser-wallet && go run internal/gen/gorm_gen.go`

**生成结果**：写入 `internal/gorm_gen/` 目录。

**工程参照**：ser-app/internal/gen/gorm_gen.go（一模一样的模板，替换 namesp 和服务名）
