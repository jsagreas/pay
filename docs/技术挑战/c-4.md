# ser-wallet 技术挑战与解决方案

> 来源：7 份 c-4 分析文档 + 表结构设计 + 工程代码级分析 + 业界成熟实践
> 定位：识别钱包模块实现过程中的核心技术挑战，给出主流+可选的解决方案，并结合自身情况推荐最佳实践
> 范围：钱包模块（C端）+ 币种配置（B端）+ 与财务模块联调
> 原则：说清楚挑战是什么、为什么难、别人怎么解决、我们怎么选

## 总览：九大技术挑战

| 序号 | 挑战 | 难度 | 风险等级 | 核心一句话 |
|------|------|------|---------|-----------|
| 1 | 余额并发安全 | ★★★★★ | 极高 | 同一个用户同时操作余额，钱不能多不能少 |
| 2 | 幂等性保障 | ★★★★☆ | 高 | 三方回调重复推送、网络重试不能重复入账 |
| 3 | 对账一致性 | ★★★★☆ | 高 | 余额和流水必须永远对得上 |
| 4 | 跨服务数据一致性 | ★★★★☆ | 高 | 钱包和财务各管各的数据，不能丢不能错 |
| 5 | 金额精度 | ★★★☆☆ | 中 | 多币种多精度不能出现分/厘级误差 |
| 6 | 提现资金状态管理 | ★★★★☆ | 高 | 冻结/解冻/扣款状态机严格不可逆 |
| 7 | 汇率实时性与一致性 | ★★★☆☆ | 中 | 用户看到的汇率和执行的汇率必须一致 |
| 8 | 敏感数据安全 | ★★★☆☆ | 中 | 银行卡号、交易数据不能泄露 |
| 9 | 性能与可扩展性 | ★★★☆☆ | 中 | 流水表增长快，查询不能变慢 |

## 挑战一：余额并发安全

### 1.1 挑战描述

钱包的余额是所有功能的核心数据。多个操作可能同时修改同一个用户的同一个钱包余额：

```
场景 A：用户充值回调（+100）和用户提现冻结（-50）同时到达
场景 B：运营人工减款（-30）和用户兑换扣款（-80）同时到达
场景 C：三方重复推送两次充值回调（+100 +100，应该只入一次）
```

如果不加保护，可能出现：
- **丢失更新**：两个操作都读到 balance=1000，一个+100 一个-50，最终 balance=1050 或 950（丢失了另一个操作）
- **超额扣款**：余额 80，两个-50 并发执行，都校验通过，balance 变成 -20
- **重复入账**：同一笔充值入两次，用户凭空多了钱

### 1.2 业界解决方案对比

#### 方案 A：数据库行级锁（SELECT FOR UPDATE）⭐ 主流

```
原理：在数据库事务中对目标行加排他锁，其他事务读写同一行时阻塞等待
业界使用者：支付宝（早期）、微信支付、PayPal

优点：
  - 锁和数据操作在同一事务中，天然原子
  - 锁粒度精确（一个用户+一个币种+一种钱包 = 一把锁）
  - 事务结束自动释放，不存在锁泄漏
  - 实现简单，不引入外部依赖

缺点：
  - 持有锁期间占用数据库连接
  - 高并发同一行时，等待队列可能积压
  - 死锁风险（多个事务交叉锁不同行）

适用场景：单用户并发操作不高的情况（我们的场景）
```

#### 方案 B：乐观锁（version 字段）⭐ 主流

```
原理：不加锁，更新时校验版本号 WHERE version=旧值，更新0行则冲突重试
业界使用者：支付宝（account+version 双字段校验）、Stripe

优点：
  - 无锁等待，读不阻塞
  - 冲突率低时性能最好
  - 实现简单

缺点：
  - 高冲突时重试频繁，反而性能差
  - 需要上层实现重试逻辑
  - 单独使用无法防止脏读

适用场景：冲突率低的场景，常与行锁结合使用
```

#### 方案 C：Redis 分布式锁（SETNX + TTL）

```
原理：操作余额前先抢 Redis 锁，抢到才执行，执行完释放
业界使用者：部分电商支付、独立钱包服务

优点：
  - 不占数据库连接
  - 锁粒度灵活

缺点：
  - 锁和数据操作不在同一事务中，需要额外保证一致性
  - 锁续期/释放逻辑复杂（Redisson/Redlock）
  - Redis 故障影响核心资金操作
  - GC 停顿可能导致锁过期但操作未完成

适用场景：数据库连接是瓶颈的高并发场景
```

#### 方案 D：CAS 队列（串行化处理）

```
原理：同一用户的余额操作全部入队列，单线程串行处理
业界使用者：部分券商交易系统

优点：
  - 完全无并发问题
  - 吞吐量可控

缺点：
  - 延迟增加（排队等待）
  - 队列实现复杂（故障恢复、消息丢失）
  - 对实时操作（兑换）体验差

适用场景：批量处理为主的场景
```

### 1.3 我们的最佳实践：行锁 + 乐观锁 + 幂等键（三层防线）

```
选择理由：
  1. 单用户的并发操作本身不高——一个用户不会同时做 100 笔充值
  2. TiDB 支持行级锁（Percolator 事务模型），性能足够
  3. 三层各解决不同维度：行锁→事务内一致，乐观锁→跨事务兜底，幂等键→重复请求过滤
  4. 不引入 Redis 锁→核心资金操作只依赖数据库，Redis 挂了不影响

实现方式：

  第一层：数据库行锁（事务内强一致）
    SELECT balance, frozen_amount, version FROM wallet
      WHERE user_id=? AND currency_code=? AND wallet_type=?
      FOR UPDATE;
    → 锁定这一行，其他事务对同一行的操作阻塞等待

  第二层：乐观锁（跨事务冲突检测）
    UPDATE wallet SET balance=?, version=version+1
      WHERE id=? AND version=?;
    → 更新0行说明有并发冲突，返回错误由上层重试

  第三层：幂等键（重复请求过滤）
    wallet_transaction 唯一键 uk_order_wallet_tx(order_no, wallet_type, tx_type)
    → 同一订单同一钱包同一类型只能插入一条流水
    → 并发两个完全相同的请求，只有一个 INSERT 成功

放弃理由：
  - 不用 Redis 分布式锁：锁和事务分离增加一致性风险，当前规模不需要
  - 不用消息队列串行化：延迟不可接受（兑换是同步操作，用户等着看结果）
  - 不用 TCC/Saga：单服务内操作，数据库事务已够，不需要分布式事务框架

未来演进：
  - 如果单用户并发操作极高（> 100 TPS/用户），可在行锁之上加 Redis 前置过滤
  - 如果需要分库分表，行锁升级为分片内行锁 + 跨分片协调
```

## 挑战二：幂等性保障

### 2.1 挑战描述

金融系统的幂等性是**刚需中的刚需**——同一笔操作无论执行多少次，结果只生效一次。

```
必须幂等的场景：
  - 充值回调：三方支付可能重复推送回调（网络重试、补偿机制）
  - 提现冻结/解冻/扣款：财务侧可能因网络超时重试 RPC 调用
  - 人工加减款：审核通过后的 RPC 调用可能重试

不幂等的后果：
  - 充值重复入账：用户充 100 变成充 200，平台亏钱
  - 提现重复扣款：用户提 100 被扣 200，用户投诉
  - 人工加款重复：运营加 100 变 200，账不对
```

### 2.2 业界解决方案对比

#### 方案 A：唯一键 + 状态检查 ⭐ 主流

```
原理：
  1. 执行前查询：该订单号是否已处理过
  2. 已处理→直接返回成功
  3. 未处理→执行操作，利用数据库唯一键防并发穿透

业界使用者：支付宝、微信支付、Stripe（idempotency key）

实现：
  // 应用层检查
  exists := tx.WalletTransaction.Where(orderNo, walletType, txType).Exist()
  if exists { return success }
  // 执行操作 + 写流水
  // 数据库唯一键 uk_order_wallet_tx 兜底

优点：简单可靠、无外部依赖、数据库级防线
缺点：每次请求多一次查询（可接受）
```

#### 方案 B：幂等表（独立的去重表）

```
原理：单独维护一张 idempotency_key 表，操作前先插入 key，成功才执行
业界使用者：Stripe API（X-Idempotency-Key header）

实现：
  INSERT INTO idempotency_key (key, status) VALUES (?, 'processing')
  → 成功→执行操作→更新 status='completed'
  → 冲突→查询已有记录→返回之前的结果

优点：通用性强、可存储操作结果供重试查询
缺点：多一张表、多一次写入、需要清理过期数据
```

#### 方案 C：Redis 去重（SETNX + TTL）

```
原理：用 Redis SETNX 设置操作标记，成功才执行
业界使用者：部分中间件、消息消费去重

优点：速度快、无数据库开销
缺点：Redis 故障丢失去重标记、TTL 到期后可能重复
```

### 2.3 我们的最佳实践：应用层检查 + 数据库唯一键 + 状态机（三层）

```
选择理由：
  1. 订单号（order_no）是天然幂等键——每笔业务操作都有唯一订单号
  2. wallet_transaction 的唯一键 uk_order_wallet_tx 已经存在，零额外成本
  3. 状态机提供逻辑级防线——只有正确状态才允许操作

三层幂等架构：

  Layer 1：应用层快速拒绝
    → 查 wallet_transaction 是否已存在该 order_no 的记录
    → 存在→直接返回成功
    → 不存在→继续

  Layer 2：数据库唯一键（最终防线）
    → wallet_transaction 的 uk_order_wallet_tx
    → 并发 INSERT 时只有一个成功，另一个唯一键冲突

  Layer 3：业务状态机（逻辑防线）
    → 提现：只有"已冻结"状态才允许扣款或解冻
    → 充值：只有"待支付/进行中"才允许上账
    → 状态已终态的订单直接拒绝

放弃理由：
  - 不用独立幂等表：wallet_transaction 已承担此功能，不需要多建表
  - 不用 Redis 去重：Redis 不可靠（可能丢 key），资金操作的幂等必须落数据库
```

## 挑战三：对账一致性

### 3.1 挑战描述

金融系统的核心指标：**账不能错**。具体要求：

```
强一致性约束：
  1. 余额约束：wallet.balance = SUM(wallet_transaction.amount) WHERE status=已完成
  2. 流水约束：每条流水的 balance_after = balance_before + amount
  3. 链式约束：第N条流水的 balance_before = 第N-1条流水的 balance_after
  4. 终态约束：最后一条流水的 balance_after = wallet.balance
  5. 非负约束：balance >= 0，frozen_amount >= 0
```

如果这些约束被打破，说明出了 Bug 或数据被篡改。

### 3.2 业界解决方案对比

#### 方案 A：同事务写入（实时一致）⭐ 主流

```
原理：余额变更和流水写入在同一个数据库事务中完成
业界使用者：所有主流支付系统

实现：
  BEGIN
    UPDATE wallet SET balance=new_balance, version=version+1 WHERE ...
    INSERT wallet_transaction (balance_before, balance_after, amount, ...)
  COMMIT
  → 任何一步失败全部回滚

优点：最强一致性保障、实现简单
缺点：事务持有时间略长（两个 SQL）
```

#### 方案 B：事件溯源（Event Sourcing）

```
原理：不直接存余额，只存事件（流水），余额由事件重放计算得出
业界使用者：部分银行核心系统、Axon Framework

优点：天然一致（余额就是流水的函数）、完整审计追溯
缺点：查余额需要重放所有事件或维护快照、实现复杂、性能挑战
```

#### 方案 C：异步对账（定时校验）

```
原理：余额和流水独立写入，定时任务校验一致性
业界使用者：所有金融系统都有（作为兜底，非唯一手段）

实现：每天凌晨扫描所有用户，校验余额和流水一致性
优点：能发现同事务保障之外的异常（如数据库 Bug、手动改数据库）
缺点：事后发现，非实时，出了问题已经晚了
```

### 3.3 我们的最佳实践：同事务写入 + balance_before/after + 异步对账

```
选择理由：
  1. 同事务写入是金融系统标配，没有替代方案
  2. balance_before/after 是对账的关键字段——缺少它只能靠 SUM 重算
  3. 异步对账是兜底——即使代码完美也需要独立校验

实现：

  实时保障：
    BalanceEngine 每个操作内部：
      → 同一事务中：UPDATE wallet + INSERT wallet_transaction
      → 流水记录 balance_before = 变更前余额，balance_after = 变更前 + amount
      → 任何一步失败全部回滚

  定期校验（每天凌晨）：
    对每个有余额变动的用户：
      ① wallet.balance == 最新流水的 balance_after？
      ② 每条流水的 balance_after == balance_before + amount？
      ③ 第N条的 balance_before == 第N-1条的 balance_after？
      → 不一致则告警

放弃理由：
  - 不用事件溯源：太重，当前规模不需要，查余额需要快照维护
  - 不分离余额和流水写入：违反"同事务"原则，引入不一致风险
```

## 挑战四：跨服务数据一致性

### 4.1 挑战描述

钱包和财务是两个独立服务，各自有自己的数据库。资金操作跨越两个服务：

```
充值链路：财务收到回调 → RPC调钱包上账 → 如果RPC失败怎么办？
提现链路：钱包冻结成功 → 通知财务审核 → 如果通知失败怎么办？
提现终态：财务出款成功 → RPC调钱包扣款 → 如果RPC超时但实际已成功呢？
```

核心矛盾：**两个服务的数据变更不在同一个事务中**，可能出现一边成功一边失败。

### 4.2 业界解决方案对比

#### 方案 A：本地事务 + 幂等重试 ⭐ 主流

```
原理：
  每个服务只管好自己的事务
  跨服务通过幂等 RPC 重试保证最终一致

业界使用者：支付宝、微信支付、Stripe

实现：
  充值链路：
    财务本地事务：更新财务订单状态=成功
    → RPC 调钱包 CreditDeposit（幂等）
    → 失败？定时任务扫描未上账的成功订单，重试 RPC
    → 重试安全？CreditDeposit 幂等，重复调用不会重复入账

优点：简单可靠、无协调器开销、每个服务独立
缺点：有短暂不一致窗口（财务成功但钱包未入账）
```

#### 方案 B：TCC（Try-Confirm-Cancel）

```
原理：
  Try → 预留资源
  Confirm → 确认提交
  Cancel → 回滚
  需要事务协调器管理全局事务

业界使用者：蚂蚁金服（Seata/DTM）

优点：强一致性
缺点：
  - 三个阶段延迟高
  - 每个参与方需实现 Try/Confirm/Cancel 三个接口
  - 协调器是单点，复杂度高
  - 悬挂/空回滚/幂等问题需要处理
```

#### 方案 C：Saga（补偿事务）

```
原理：
  正向操作链：A→B→C
  任一步失败→反向补偿：C'→B'→A'

业界使用者：Netflix Conductor、Uber Cadence

优点：长事务支持好
缺点：
  - 每个操作需要设计补偿操作
  - 补偿操作本身可能失败（补偿的补偿？）
  - 中间状态对用户可见
```

#### 方案 D：可靠消息（事务消息表/Outbox Pattern）

```
原理：
  本地事务中同时写业务数据和消息表
  后台进程从消息表取出消息发到 MQ
  消费者幂等消费

业界使用者：RocketMQ 事务消息、Debezium CDC

优点：最终一致性好、解耦
缺点：增加消息中间件依赖、延迟增加
```

### 4.3 我们的最佳实践：本地事务 + 幂等重试

```
选择理由：
  1. 钱包和财务的跨服务调用模式天然适合：
     → 财务调钱包的所有 RPC 都是幂等的（order_no 唯一键）
     → 失败重试是安全的
  2. 不需要分布式事务协调器：
     → 两个服务之间不存在需要同时提交的场景
     → 充值：财务先提交自己的，再通知钱包（钱包失败就重试）
     → 提现：钱包先冻结，再通知财务（财务审核是异步的，不需要原子）
  3. 团队只有一个人，引入 TCC/Saga 框架是过度设计

具体实现：

  充值链路一致性保证：
    ┌────────────────┐     ┌────────────────┐
    │     财务服务     │     │     钱包服务     │
    │                │     │                │
    │ 1. 收到回调     │     │                │
    │ 2. 验签        │     │                │
    │ 3. 本地事务：   │     │                │
    │    更新订单=成功 │     │                │
    │ 4. RPC ────────┼────►│ CreditDeposit  │
    │                │     │ [幂等：查order]  │
    │    成功←────────┼─────│ [入账+写流水]   │
    │                │     │                │
    │ 如果步骤4失败：  │     │                │
    │ 5. 定时任务扫描  │     │                │
    │    未上账成功订单 │     │                │
    │ 6. 重试RPC ────┼────►│ 同上（幂等安全） │
    └────────────────┘     └────────────────┘

  提现链路一致性保证：
    ┌────────────────┐     ┌────────────────┐
    │     钱包服务     │     │     财务服务     │
    │                │     │                │
    │ 1. 创建订单     │     │                │
    │ 2. 冻结余额     │     │                │
    │ 3. 通知财务 ────┼────►│ 接受提现请求    │
    │                │     │ 三级审核...      │
    │                │     │                │
    │                │     │ 审核拒绝：       │
    │ Unfreeze ◄─────┼─────│ RPC（幂等）     │
    │ [解冻返还]     │     │                │
    │                │     │ 出款成功：       │
    │ ConfirmDeduct◄─┼─────│ RPC（幂等）     │
    │ [确认扣款]     │     │                │
    └────────────────┘     └────────────────┘

    关键：Unfreeze 和 ConfirmDeduct 互斥（状态机保证）
    财务重试安全：钱包通过订单状态判断是否已处理

放弃理由：
  - 不用 TCC：过度设计，协调器是额外的单点风险
  - 不用 Saga：补偿逻辑复杂，金融场景更适合幂等重试
  - 不用事务消息：引入 MQ 依赖，当前规模不需要
```

## 挑战五：金额精度

### 5.1 挑战描述

多币种系统面临的精度挑战：

```
不同币种精度不同：
  - VND（越南盾）：整数，无小数位，最小单位1盾
  - THB（泰铢）：2位小数，最小单位1萨当
  - USDT：8位小数，最小单位1聪（0.00000001）

精度陷阱：
  - 浮点运算：0.1 + 0.2 = 0.30000000000000004（IEEE 754）
  - 汇率计算：1 USDT = 25432.50 VND，用浮点乘除会积累误差
  - 截断/四舍五入：用户充 100.005 THB 到底算 100.00 还是 100.01？
  - 币种转换：法币→BSB→法币，两次汇率换算可能丢精度
```

### 5.2 业界解决方案对比

#### 方案 A：整数存储（最小单位）⭐ 主流

```
原理：所有金额以最小货币单位的整数存储和计算
业界使用者：Stripe（分）、支付宝（分）、微信支付（分）、Square

实现：
  法币（VND/IDR/THB）：× 100 存 bigint（分）
  加密货币（USDT）：× 10^8 存 bigint（聪）
  平台币（BSB）：× 100 存 bigint

  Go 实现：amount int64
  DB 存储：bigint
  IDL 传输：i64

优点：
  - 零精度丢失（整数运算 1+1=2，永远精确）
  - int64 运算速度是 decimal 库的 10x+
  - 索引效率：bigint > decimal

缺点：
  - 展示层需要按币种格式化（除以单位因子）
  - 不同币种的单位因子不同，需要查 currency_config.decimal_digits
```

#### 方案 B：Decimal 类型

```
原理：数据库用 DECIMAL(M,N)，Go 用 shopspring/decimal 库
业界使用者：传统银行系统

优点：直观，存的就是看到的
缺点：
  - Go decimal 库性能差（反射、字符串操作）
  - TiDB DECIMAL 索引效率低于 BIGINT
  - 精度仍需要显式定义 DECIMAL(20,8)，8 位不够就溢出
```

#### 方案 C：字符串存储

```
原理：金额以字符串存储，运算时用高精度库
业界使用者：极少数，特殊场景

优点：精度无限
缺点：性能极差、索引无法用、比较排序困难
```

### 5.3 我们的最佳实践：整数存储 + 币种精度配置 + 汇率放大 10^8

```
选择理由：
  1. Stripe/支付宝/微信支付都用整数存储，是金融系统事实标准
  2. Go 的 int64 范围 ±9.2×10^18，即使 ×10^8 也能存到 920 亿级金额
  3. 与工程现有模式一致（ser-item 的 base_price 用 int unsigned）

金额存储规则：
  currency_config.decimal_digits 定义每个币种的精度
  存储值 = 展示值 × 10^decimal_digits

  示例：
    VND：100000 盾 → 存 10000000（×100，但 VND 通常 0 小数位→存 100000）
    THB：1000.50 泰铢 → 存 100050（×100）
    USDT：0.05 USDT → 存 5000000（×10^8）
    BSB：100.00 BSB → 存 10000（×100）

汇率存储规则：
  所有汇率统一放大 10^8 存 bigint
  rate_platform = 实际汇率 × 10^8

  示例：
    1 USDT = 25432.50 VND → rate_platform = 2543250000000
    1 BSB = 0.10 USDT → rate_platform = 10000000

计算规则：
  兑换计算：target_amount = source_amount × rate / 10^8
  结果截断到目标币种的 decimal_digits，多余部分舍去（向零截断，平台不亏）

放弃理由：
  - 不用 decimal：Go decimal 库性能差，TiDB decimal 索引效率低
  - 不用字符串：无法索引和比较
```

## 挑战六：提现资金状态管理

### 6.1 挑战描述

提现是最复杂的业务链路——涉及冻结/解冻/确认扣款的状态机，且跨越钱包和财务两个服务：

```
状态机必须严格遵守的规则：
  - 冻结→只能解冻 或 确认扣款（二选一，互斥）
  - 已解冻→不能再扣款
  - 已扣款→不能再解冻
  - 未冻结→不能扣款也不能解冻
  - 同一笔提现的冻结金额不能被其他操作占用

违反的后果：
  - 冻结了没解冻也没扣款→用户的钱永远冻着（资损）
  - 冻结了但又被兑换扣走→提现失败时解冻发现余额不够（脏数据）
  - 解冻和扣款都执行了→用户先拿回钱又被扣走（双重操作）
```

### 6.2 业界解决方案对比

#### 方案 A：双字段分离（balance + frozen_amount）⭐ 主流

```
原理：
  可用余额（balance）和冻结余额（frozen_amount）分开存储
  冻结：balance -= amount, frozen_amount += amount
  解冻：balance += amount, frozen_amount -= amount
  确认扣款：frozen_amount -= amount

业界使用者：支付宝、微信支付、银行核心系统

优点：
  - 可用余额和冻结余额相互独立
  - 冻结的钱不会被消费/兑换操作扣走
  - 任何时刻 total_asset = balance + frozen_amount

缺点：
  - 需要维护两个字段的一致性
```

#### 方案 B：冻结记录表（独立 freeze_record）

```
原理：
  冻结时往 freeze_record 插一条记录
  解冻/扣款时更新或删除对应记录
  可用余额 = total_balance - SUM(freeze_record.amount)

优点：完整的冻结历史追溯
缺点：查可用余额需要 SUM 计算、性能差
```

#### 方案 C：预授权模式（Hold/Capture/Void）

```
原理：
  Hold → 预授权（冻结）
  Capture → 扣款（确认）
  Void → 撤销（解冻）

业界使用者：信用卡支付、Stripe PaymentIntent

与方案A本质相同，只是叫法不同。
```

### 6.3 我们的最佳实践：双字段分离 + 订单状态机 + 互斥校验

```
选择理由：方案 A 最简单直接，业界验证充分

实现：

  wallet 表：
    balance       → 可用余额（用户可以花的钱）
    frozen_amount → 冻结余额（提现中锁定的钱）

  状态流转：
    CreateWithdrawalOrder（W-11）→ Freeze：
      balance -= amount
      frozen_amount += amount
      withdrawal_order.status = 处理中

    UnfreezeForWithdrawal（W-R4）→ Unfreeze：
      前置检查：withdrawal_order.status 必须是 处理中/审核中/出款中
      balance += amount
      frozen_amount -= amount
      withdrawal_order.status = 已失败/已拒绝

    ConfirmWithdrawalDeduct（W-R5）→ ConfirmDeduct：
      前置检查：withdrawal_order.status 必须是 出款中
      前置检查：W-R4 未执行（status 不是已失败/已拒绝）
      frozen_amount -= amount
      withdrawal_order.status = 已完成

  互斥保证：
    W-R4 和 W-R5 都先查订单状态：
      → 已拒绝/已失败→说明 W-R4 已执行→W-R5 拒绝
      → 已完成→说明 W-R5 已执行→W-R4 拒绝
      → 只有"进行中/审核中/出款中"才允许操作
    → 数据库行锁保证查和改在同一事务中，不会并发穿透
```

## 挑战七：汇率实时性与一致性

### 7.1 挑战描述

```
核心矛盾：汇率是实时变动的，但用户操作需要稳定的汇率

问题场景：
  1. 用户在兑换预览页看到 1 BSB = 25000 VND
  2. 用户犹豫了 3 秒后点击确认
  3. 这 3 秒内汇率更新为 1 BSB = 25500 VND
  4. 如果用最新汇率执行：用户看到的和实际执行的不一致→投诉
  5. 如果用旧汇率执行：平台可能亏钱（汇率不利方向变动）

另一个问题：
  - 三方汇率 API 可能超时/返回异常值
  - 3 家 API 返回的汇率可能差异较大
  - 如何容错和取值？
```

### 7.2 业界解决方案对比

#### 方案 A：汇率快照 + TTL ⭐ 主流

```
原理：
  预览时锁定当前汇率，生成快照（存 Redis，TTL 5分钟）
  执行时用快照汇率，快照过期则拒绝要求重新预览

业界使用者：Wise（TransferWise）、Revolut、OKX

优点：
  - 用户体验一致：看到什么就是什么
  - TTL 限制风险窗口：5 分钟内汇率变动有限
  - 实现简单

缺点：
  - 快照期间汇率大幅变动时平台可能有损
  - 需要 Redis 支持
```

#### 方案 B：实时报价 + 滑点容忍

```
原理：
  每次操作都取最新汇率
  设置滑点阈值（如 ±0.5%），超出滑点提示用户确认

业界使用者：去中心化交易所（Uniswap）

优点：汇率最新
缺点：用户体验差（价格不停变）、前端实现复杂
```

#### 方案 C：固定窗口汇率

```
原理：每隔 N 分钟更新一次汇率，窗口期内汇率不变
业界使用者：传统银行外汇窗口

优点：简单、稳定
缺点：窗口切换瞬间可能有大跳变
```

### 7.3 我们的最佳实践：汇率快照 + TTL + 偏差阈值自动更新

```
选择理由：
  1. 汇率快照是外汇交易平台的标准做法
  2. 需求文档 B4 已定义了偏差阈值触发更新机制
  3. Redis 工程已有，快照存储零成本

实现：

  汇率同步（定时任务，每 5-10 分钟）：
    → 并发请求 3 家 API
    → 容错：3成功取均值 / 2成功取均值 / 1成功取该值 / 全失败跳过
    → 计算偏差 = |新实时 - 当前平台| / 新实时 × 100%
    → 偏差 < 阈值：只更新 realtime_rate
    → 偏差 ≥ 阈值：更新全部四个汇率 + 写 rate_change_log + 更新 Redis

  兑换快照机制：
    W-8 预览：
      → 读当前 platform_rate
      → 生成 snapshot_id = currency_code + timestamp + rate 的哈希
      → Redis SET wallet:exchange:snapshot:{snapshot_id} → rate_value，TTL 5分钟
      → 返回给前端

    W-9 执行：
      → 传入 snapshot_id
      → Redis GET 快照
      → 不存在（过期）→ 返回 6022301 ExchangeRateExpired
      → 存在 → 用快照汇率执行 → 执行完 DEL 快照（一次性使用）

  三方 API 适配器接口：
    type RateProvider interface {
        GetRate(ctx, base, target string) (rate int64, err error)
    }
    → CoinGeckoProvider / ExchangeRateAPIProvider / ...
    → 统一接口，方便替换和新增
```

## 挑战八：敏感数据安全

### 8.1 挑战描述

```
钱包系统涉及的敏感数据：
  - 银行卡号/电子钱包号（提现账户）
  - 用户余额（虽非隐私但属于财务数据）
  - 交易流水（用户行为轨迹）
  - KYC 认证姓名（个人信息）

安全威胁：
  - 数据库泄露：DBA 或攻击者直接看到明文卡号
  - API 越权：用户A操作用户B的钱包
  - 中间人攻击：RPC 通信被截获
  - 日志泄露：敏感数据被打印到日志中
```

### 8.2 业界解决方案对比

#### 方案 A：字段级加密（AES）⭐ 主流

```
原理：敏感字段写入数据库前 AES 加密，读出后解密
业界使用者：PCI-DSS 合规要求（所有支付系统）

工程已有实现：
  common/pkg/utils/crypto_util.go → EncryptedString 类型
  GORM 自动加解密（Valuer/Scanner 接口）
  AES-256-CBC 算法

适用范围：银行卡号、电子钱包号
不适用：需要模糊搜索的字段（如持有人姓名）
```

#### 方案 B：令牌化（Tokenization）

```
原理：用无意义的 Token 替代真实卡号，真实数据存在安全的 Vault 中
业界使用者：Stripe（PCI 合规的标准做法）

优点：应用层完全看不到真实卡号
缺点：需要额外的 Vault 服务、增加依赖
```

#### 方案 C：脱敏展示

```
原理：查询时只返回脱敏后的数据（如 **** **** **** 1234）
业界使用者：所有金融 App

实现：应用层格式化，不暴露完整卡号
适用：前端展示场景
```

### 8.3 我们的最佳实践：字段级加密 + 权限分层 + 脱敏展示

```
选择理由：
  1. 工程已有 EncryptedString 实现，零开发成本
  2. 网关层已有完整的认证授权体系
  3. 不需要独立 Vault 服务，AES 加密已满足合规要求

实现：

  存储安全：
    withdrawal_account.account_no → EncryptedString（AES-256-CBC）
    GORM Gen 配置 encryptedFields 注册此字段
    数据库存密文，应用层透明加解密

  访问安全：
    C端：userId 从 ctx 取（FontAuthMiddleware 注入），不从参数取
      → 用户只能操作自己的钱包，无法篡改 userId
    B端：URL 级 RBAC（ser-buser.CheckUrlAuth）
      → 不同角色看不同页面
    RPC：内网通信，服务间信任
      → Kitex TTHeader 传播 userId

  展示安全：
    卡号脱敏：6222 **** **** 1234（首4末4，中间掩码）
    余额：不脱敏（用户需要看到完整余额）
    日志：敏感字段不打印或脱敏打印
```

## 挑战九：性能与可扩展性

### 9.1 挑战描述

```
核心增长表：wallet_transaction
  假设日活 10 万用户，每人日均 3 笔操作 → 日增 30 万条
  年增 ~1 亿条
  C端记录页：按用户+币种+类型分页查询
  对账任务：扫描全表校验一致性

潜在性能瓶颈：
  1. wallet_transaction 表膨胀后，分页查询变慢
  2. 对账任务扫描耗时长，影响业务
  3. 行锁持有时间影响同一用户的并发操作
```

### 9.2 业界解决方案对比

#### 方案 A：索引优化 + 查询优化 ⭐ 首选

```
原理：精心设计索引覆盖主要查询模式
业界使用者：所有系统的第一步优化

实现：
  idx_user_curr_type(user_id, currency_code, tx_type) → C端记录页主查询
  idx_user_create_at(user_id, create_at) → 按时间分页
  分页用 ORDER BY id DESC LIMIT offset, size → 利用主键索引

优点：简单有效、零成本
缺点：数据量极大时索引也会变慢
```

#### 方案 B：冷热分离（数据归档）

```
原理：超过 N 天的数据归档到历史表，主表只保留近期数据
业界使用者：银行流水（一般保留近 6 个月）

实现：
  wallet_transaction → 主表（近 6 个月）
  wallet_transaction_archive → 归档表（历史数据）
  定时任务搬迁

优点：主表数据量可控
缺点：查历史数据需要跨表、归档任务需要运维
```

#### 方案 C：分区表

```
原理：按时间或用户 ID 分区
业界使用者：大规模交易系统

TiDB 支持 Range/Hash 分区：
  PARTITION BY RANGE (create_at) (
    PARTITION p2026q1 VALUES LESS THAN (unix_timestamp('2026-04-01')),
    PARTITION p2026q2 VALUES LESS THAN (unix_timestamp('2026-07-01')),
    ...
  )

优点：查询自动裁剪分区、管理数据方便
缺点：分区键选择影响查询效率、需要提前规划
```

#### 方案 D：读写分离

```
原理：写走主库，读走从库
业界使用者：所有大规模系统

注意：
  - 余额查询不能走从库（必须实时准确）
  - 记录查询可以走从库（允许秒级延迟）
  - TiDB 原生支持 Follower Read（读从节点）
```

### 9.3 我们的最佳实践：索引优化（首版）+ 冷热分离（后续）

```
选择理由：
  1. 首版数据量不大（日活万级），索引优化足够
  2. TiDB 本身就是分布式数据库，单表亿级数据没有 MySQL 那么紧张
  3. 过早优化是万恶之源——先跑通再优化

首版实现：
  - 精心设计索引（已在表结构设计中完成）
  - 分页查询用主键 id 做游标（深翻页优化）
  - 余额查询走 uk_user_curr_type 索引，单行查询 < 1ms
  - 连接池：MaxOpen=20, MaxIdle=10（工程默认，足够）
  - PrepareStmt=true（工程已开启，高频 SQL 预编译缓存）

后续扩展路径：
  数据量 < 1000 万：当前方案足够
  数据量 1000 万~1 亿：加冷热分离（6 个月归档）
  数据量 > 1 亿：按时间分区 + 冷热分离
  QPS > 1 万：读写分离（记录查询走从库）

不做的事：
  - 不分库分表：TiDB 本身就是分布式的，不需要应用层分片
  - 不用 Elasticsearch：流水查询是按用户+币种+类型的点查，不是全文检索
  - 不用缓存余额：强一致性要求，缓存一致性成本大于收益
```

## 总结：技术挑战决策速查表

| 挑战 | 我们的选择 | 业界对标 | 放弃的方案 |
|------|-----------|---------|-----------|
| 余额并发安全 | 行锁+乐观锁+幂等键 | 支付宝（行锁+乐观锁） | Redis 分布式锁、消息队列串行化 |
| 幂等性保障 | 应用层检查+唯一键+状态机 | Stripe（idempotency key） | 独立幂等表、Redis 去重 |
| 对账一致性 | 同事务写入+before/after+异步对账 | 所有金融系统标配 | 事件溯源 |
| 跨服务一致性 | 本地事务+幂等重试 | 支付宝/微信支付/Stripe | TCC、Saga、事务消息 |
| 金额精度 | 整数存储（最小单位） | Stripe/支付宝/微信支付 | decimal、字符串 |
| 提现状态管理 | 双字段分离+状态机+互斥校验 | 银行核心/支付宝 | 冻结记录表 |
| 汇率一致性 | 快照+TTL+偏差阈值 | Wise/Revolut | 实时报价+滑点 |
| 敏感数据安全 | AES 加密+权限分层+脱敏 | PCI-DSS 标准 | Vault 令牌化 |
| 性能扩展 | 索引优化（首版）→冷热分离（后续） | 渐进式优化 | 过早分库分表 |

### 核心原则
