## 二、挑战1：并发余额操作的正确性保证【P0】

### 2.1 挑战描述

```
核心问题：多个请求同时操作同一个用户的同一个钱包余额，如何保证不超扣、不丢失更新？

典型场景（来自我们的业务）：
  场景A：用户同时收到多笔投注返奖（批量结算回调）→ 同一账户被并发 credit
  场景B：用户快速点击充值，同时投注扣款 → 同一账户被并发 credit 和 debit
  场景C：提现冻结 + 投注扣款同时来 → 可用余额可能被超扣
  场景D：活动批量发奖 → 大量用户同时被 rewardCredit

风险：
  - 丢失更新：两个请求读到相同余额，各自计算后写回，后者覆盖前者
  - 超扣：两个扣款请求都通过余额检查（因为都读到旧余额），导致余额变负
  - 数据不一致：余额和流水不匹配
```

### 2.3 方案详细分析

**主流方案A：条件更新（推荐 — 我们的最佳实践）**

```
适用场景：单账户余额增减操作（credit / debit / freeze / unfreeze / confirmDebit）
原理：利用数据库单条 UPDATE 语句的天然原子性

扣款示例：
  UPDATE user_wallet
  SET available_balance = available_balance - ?amount,
      update_at = ?now
  WHERE user_id = ?uid
    AND currency_code = ?code
    AND wallet_type = ?type
    AND available_balance >= ?amount;   -- 关键：条件保护

  → affected_rows = 1：扣款成功
  → affected_rows = 0：余额不足或账户不存在

为什么这是我们的最佳实践：
  ✓ 现有工程已有此模式（ser-item 的 gorm.Expr + 条件更新）
  ✓ TiDB/MySQL InnoDB 单行 UPDATE 天然原子，不需要额外机制
  ✓ 无额外字段（不需要 version 列）
  ✓ 无外部依赖（不需要 Redis 锁）
  ✓ 性能最优（一条 SQL 搞定，锁持有时间最短）
  ✓ 用户级并发完全够用（不是双11秒杀场景）

Go 代码模式：
  result := tx.Model(&model.UserWallet{}).
      Where("user_id = ? AND currency_code = ? AND wallet_type = ?", uid, code, wType).
      Where("available_balance >= ?", amount).
      Updates(map[string]interface{}{
          "available_balance": gorm.Expr("available_balance - ?", amount),
          "update_at":         now,
      })
  if result.RowsAffected == 0 {
      return ErrInsufficientBalance
  }
```

**可选方案B：悲观锁（SELECT FOR UPDATE）**

```
适用场景：需要先读取余额做复杂判断，再执行多步操作的场景
典型用途：兑换操作（需同时操作两个币种账户）

示例：
  BEGIN;
  SELECT * FROM user_wallet WHERE user_id=? AND currency_code='VND' FOR UPDATE;
  SELECT * FROM user_wallet WHERE user_id=? AND currency_code='BSB' FOR UPDATE;
  -- 复杂计算（汇率换算、赠送比例等）
  UPDATE user_wallet SET ... WHERE ...;  -- 法币扣减
  UPDATE user_wallet SET ... WHERE ...;  -- BSB增加
  COMMIT;

为什么在兑换场景可以考虑：
  兑换涉及两个币种账户的联动操作，先锁定两行再操作可以避免中间状态
  但用条件更新也能实现（先扣法币，条件满足再加BSB，失败补偿）

为什么我们不首选：
  × 现有工程没有 FOR UPDATE 的使用模式
  × 锁持有时间更长（两次 SELECT + 两次 UPDATE）
  × 需要注意锁顺序避免死锁

我们的决策：兑换场景也用条件更新，在同一事务内顺序执行即可。
  事务内先扣法币（条件更新），再加BSB（无条件），如果扣法币失败直接回滚。
  单事务内的操作天然串行，不需要 FOR UPDATE。
```

**可选方案C：分布式锁（Redis Lock）**

```
适用场景：跨服务需要互斥的场景
典型用途：防止同一用户同时发起多笔提现

为什么我们当前不需要：
  × 所有余额操作都在同一数据库的同一事务内完成
  × 数据库行级锁已经是天然的"分布式锁"
  × 引入 Redis 锁增加复杂度（锁超时、锁续期、死锁检测、Redis 故障）
  × 当前用户级并发量不需要额外协调

什么时候引入：
  → 单账户秒级并发 > 1000 次写入（如热门主播秒级收到上千笔打赏）
  → 需要跨服务协调互斥（如钱包和投注同时操作同一余额）
  → 当前阶段不会遇到，预留即可
```

**可选方案D：乐观锁（version 字段）**

```
适用场景：读多写少，冲突概率低的场景
典型用途：Modern Treasury 的账本系统

为什么我们不选：
  × 需要在 user_wallet 表加 version 字段
  × 冲突时需要应用层重试逻辑
  × 条件更新已经解决了问题，引入 version 是多余的复杂度

什么时候考虑：
  → 如果业务逻辑变得复杂到无法用单条 UPDATE 表达
  → 当前阶段不会遇到
```

### 2.4 结合自身情况的最终决策

```
决策：条件更新（WHERE balance >= amount）+ 同一事务内的多步操作

依据：
  1. 现有工程已有此模式（ser-item 的 gorm.Expr），团队熟悉
  2. 单条 SQL 原子性 + 事务保护，已满足当前并发量级
  3. 不引入额外字段、不引入外部依赖，最小复杂度
  4. TiDB 支持 CHECK 约束，可加 CHECK (available_balance >= 0) 兜底
  5. 如果未来并发量级跃升，再评估是否引入分布式锁或账户分片
```

## 三、挑战2：幂等性——防止重复入账/扣款【P0】

### 3.1 挑战描述

```
核心问题：同一笔业务操作被重复执行时，如何保证只处理一次？

重复调用的来源（我们会遇到的真实场景）：
  来源A：财务模块回调重发 — 三方支付通道的重试机制，同一笔充值成功可能回调 2~3 次
  来源B：RPC 框架超时重试 — Kitex 默认有重试策略，网络抖动时会自动重试
  来源C：调用方不确定结果 — 财务模块调 CreditWallet 超时，不确定是否成功，再次调用
  来源D：前端重复点击 — 用户快速双击"兑换"按钮

风险：
  没有幂等 → 重复 CreditWallet → 用户余额多了一笔 → 平台亏钱
  没有幂等 → 重复 DebitWallet → 用户余额少了一笔 → 用户投诉
  没有幂等 → 重复 FreezeBalance → 冻结金额翻倍 → 实际余额被多冻
```

### 3.2 业界方案对比

```
┌──────────────────────┬────────────────────────┬──────────────────┬──────────────────┐
│ 方案                  │ 原理                   │ 代表系统         │ 适用场景         │
├──────────────────────┼────────────────────────┼──────────────────┼──────────────────┤
│ ① Idempotency-Key    │ 客户端传唯一 Key，     │ Stripe（业界标杆）│ 通用 API 层      │
│   请求头             │ 服务端缓存+唯一约束     │ UUID v4          │ 对外暴露的接口   │
├──────────────────────┼────────────────────────┼──────────────────┼──────────────────┤
│ ② 业务单号唯一索引    │ 用 orderNo 做唯一键，  │ 支付宝/微信支付  │ 业务层           │
│                      │ 插入冲突即为重复       │ 流水表唯一索引    │ 有业务单号的场景 │
├──────────────────────┼────────────────────────┼──────────────────┼──────────────────┤
│ ③ 独立幂等表          │ 单独的 idempotency_key│ Brandur Leach    │ 通用             │
```

### 3.3 方案详细分析

**主流方案A：业务单号唯一索引（推荐 — 我们的最佳实践）**

```
原理：在 wallet_flow 表上建 (order_no, flow_type) 唯一索引
  每次余额变更前先插入流水记录 → 唯一键冲突即为重复请求

实现流程：
  1. 接收请求，取出 order_no + flow_type
  2. 在事务内尝试 INSERT wallet_flow
     → 插入成功：首次请求，继续执行余额变更
     → 唯一键冲突（Duplicate Entry）：重复请求
       查出该 order_no 对应的流水记录
       直接返回"已处理"+ 上次结果
  3. 余额变更 + 流水写入在同一事务内提交

为什么 (order_no, flow_type) 是联合唯一：
  同一个 order_no 可能产生多种流水类型。例如提现订单：
    order_no=W20260226001, flow_type=4（冻结）     → 是一条流水
    order_no=W20260226001, flow_type=5（确认扣除）  → 是另一条流水
  两者都合法，但同一个 order_no + 同一个 flow_type 不能出现两次

为什么这是我们的最佳实践：
  ✓ 不需要额外的表（流水表本身就是幂等证据）
  ✓ 数据库层面保证（即使应用层有 bug，唯一约束也能兜底）
  ✓ 流水记录 = 幂等记录 = 审计记录，一石三鸟
  ✓ 架构设计方案已明确此方案（ADR #4）
```

**可选方案B：独立幂等表（Stripe 工程师模式）**

```
原理：创建专门的 idempotency_keys 表

  CREATE TABLE idempotency_keys (
      id              BIGINT AUTO_INCREMENT PRIMARY KEY,
      idempotency_key VARCHAR(64) NOT NULL,
      user_id         BIGINT NOT NULL,
      recovery_point  VARCHAR(32) NOT NULL DEFAULT 'started',
      request_params  JSON,
      response_code   INT,
      response_body   JSON,
      created_at      BIGINT UNSIGNED NOT NULL,
      UNIQUE KEY uk_user_key (user_id, idempotency_key)
  );

优势：
  + 可以存储完整的请求参数和响应结果
  + 支持分阶段恢复（recovery_point 标记处理到哪一步了）
  + 适合复杂的多步操作

为什么我们不首选：
  × 多一张表 = 多一份维护成本
  × 我们的操作相对简单（单事务内完成），不需要分阶段恢复
  × 流水表唯一索引已经满足需求

什么时候考虑引入：
  → 如果未来出现复杂的跨步骤操作，需要断点续传式的幂等恢复
  → 当前不需要
```

**可选方案C：Redis 前置去重（高并发补充方案）**

```
原理：在数据库操作前，先用 Redis SETNX 做快速去重

  key = "idempotent:{order_no}:{flow_type}"
  SETNX key "processing" EX 60
  → 设置成功：首次请求，继续处理
  → 设置失败：可能是重复请求，再查 DB 确认

为什么是"补充方案"而非"主方案"：
  Redis 不是持久化保证（Redis 宕机后 key 丢失，重启后重复请求会通过 Redis 检查）
  所以 Redis 做前置拦截（快速路径），数据库唯一索引做兜底（安全保证）

什么时候引入：
  → 当某个接口短时间内收到大量重复请求（如三方通道疯狂回调）
  → 用 Redis 拦截在前面，减少数据库压力
  → 当前阶段不需要
```

### 3.4 结合自身情况的最终决策

```
决策：wallet_flow 表 (order_no, flow_type) 唯一索引 — 作为幂等的唯一保障

补充约定：
  1. order_no 的生成规则：
     - 外部调用传入的（财务回调、投注扣款）→ 由调用方负责生成唯一 orderNo
     - 内部自发起的（兑换、冻结）→ 由 ser-wallet 用 snowflake ID 生成
  2. 幂等返回规则：
     - 重复请求返回"已处理" + 上次结果，不返回错误（调用方需要知道结果）
  3. 不引入独立幂等表，不引入 Redis 前置去重（当前阶段不需要，不过度设计）
```

## 四、挑战3：金额精度——资金系统的零误差要求【P0】

### 4.1 挑战描述

```
核心问题：如何保证从"用户输入金额"到"数据库存储"到"计算"到"展示"的全链路零精度损失？

为什么这是 P0 问题：
  0.1 + 0.2 = 0.30000000000000004  （IEEE 754 浮点数）
  在一个每天处理上万笔交易的钱包系统中，浮点误差会累积：
    每笔 0.01 的误差 × 10000 笔/天 × 365 天 = 36500 的累计误差
    对账永远对不上，审计无法通过

我们的特殊复杂度：
  多币种精度不同：VND/IDR 无小数，THB/BSB 2位小数，USDT 8位小数
  汇率精度更高：法币汇率 2位，加密货币汇率 8位
  涉及汇率计算：兑换/BSB充值/BSB提现都涉及乘除运算，误差会被放大
```

### 4.2 业界方案对比

```
┌──────────────────────┬──────────────────────────┬──────────────────────────┐
│ 方案                  │ 代表系统                  │ 优劣                      │
├──────────────────────┼──────────────────────────┼──────────────────────────┤
│ ① 整数最小单位       │ Stripe / PayPal /        │ ✓ 运算精确无损            │
│   (BIGINT)           │ 蚂蚁金服 / Modern Treasury│ ✓ 性能好（整数运算快）    │
│                      │                          │ × 需要转换逻辑            │
├──────────────────────┼──────────────────────────┼──────────────────────────┤
│ ② DECIMAL 定点数     │ 传统银行系统              │ ✓ 直觉易读                │
│   DECIMAL(20,8)      │ PostgreSQL/MySQL 原生支持 │ ✓ 无需转换                │
│                      │                          │ × 运算比整数慢            │
│                      │                          │ × Go 没有原生 decimal     │
├──────────────────────┼──────────────────────────┼──────────────────────────┤
│ ③ 字符串存储         │ 部分区块链系统            │ ✓ 精度无限                │
│   + 应用层计算       │                          │ × 计算需特殊库            │
│                      │                          │ × 排序/比较/聚合困难      │
└──────────────────────┴──────────────────────────┴──────────────────────────┘
```

### 4.3 方案详细分析

**主流方案A：整数最小单位 BIGINT（推荐 — 我们的最佳实践）**

```
原理：所有金额按币种最小单位存储为整数

  币种        小数位数    存储倍率    示例
  ────        ────────    ────────    ────
  VND/IDR     0           ×1         500,000 VND → 存 500000
  THB/BSB     2           ×100       1,500.50 THB → 存 150050
  USDT        8           ×10^8      1.00000001 USDT → 存 100000001

全链路精度方案：

  ┌───────────┐   string    ┌───────────┐   int64    ┌───────────┐
  │  用户输入  │ ──────────→ │  传输层    │ ─────────→ │  计算层    │
  │  "1500.50"│   (IDL)    │  string   │  (Go)     │  150050   │
  └───────────┘            └───────────┘            └─────┬─────┘
                                                          │ int64
                                                    ┌─────▼─────┐
                                                    │  存储层    │
                                                    │  BIGINT    │
                                                    │  150050   │
                                                    └─────┬─────┘
                                                          │
                                                    ┌─────▼─────┐
                                                    │  展示层    │
                                                    │ ÷100=1500.50
                                                    │ 按 decimal_digits
                                                    │ 格式化输出
                                                    └───────────┘

为什么传输用 string：
  Thrift 的 i64 精度足够，但 JSON 序列化后前端 JS Number 精度只有 2^53 ≈ 9×10^15
  用 string 传输避免前端精度丢失（这是业界标准，Stripe API 也用 string）

为什么这是我们的最佳实践：
  ✓ 业界标准（Stripe/PayPal/蚂蚁金服内部全部用整数存储）
  ✓ 现有工程一致（ser-item 的 base_price 用 int32 整数存储）
  ✓ Go 的 int64 运算精确无损，不需要引入第三方 decimal 库
  ✓ BIGINT 范围 ±9.2×10^18，即使存 USDT 的 satoshi 也绰绰有余
```

**汇率精度方案：**

```
汇率存储：BIGINT × 10^8（保留8位精度）

  示例（VND→USDT汇率，实际值 0.00003820）：
    存储值 = 0.00003820 × 10^8 = 3820
    读取时 = 3820 / 10^8 = 0.00003820

  示例（BSB→USDT汇率，实际值 0.10000000）：
    存储值 = 0.10000000 × 10^8 = 10000000
    读取时 = 10000000 / 10^8 = 0.10000000

  为什么统一 10^8：
    加密货币需要8位精度，法币2位就够，统一按最高精度存储避免多套方案

浮动比例/阈值精度：BIGINT × 10^4（保留2位百分比精度）

  示例（浮动比例 2.50%）：
    存储值 = 2.50 × 10^4 = 25000
    计算时 = 25000 / 10^4 = 2.50
    入款汇率 = 实时汇率 × (10^4 + 25000) / 10^4

兑换计算示例（完整精度链路）：
  用户用 500,000 VND 兑换 BSB
  平台汇率（1 BSB = 2,620 VND）存储为 rate = 262000000000 (2620 × 10^8)

  兑换公式：BSB数量 = VND金额 × 10^8 / rate × BSB倍率
    = 500000 × 10^8 / 262000000000 × 100
    = 50000000000000 / 262000000000 × 100
    = 190.83... × 100
    = 19083 （即 190.83 BSB）

  全程整数运算，最后一步才取整（向下取整，平台不亏）
```

**可选方案B：DECIMAL(20,8)**

```
优势：SQL 原生支持，直觉易读
劣势：
  Go 没有原生 decimal 类型，GORM 映射为 float64（精度丢失！）
  或映射为 string（需要额外转换逻辑）
  需要引入第三方库 shopspring/decimal 做计算
  与现有工程的 int32/int64 模式不一致

为什么我们不选：
  ✓ 现有工程已用整数模式（ser-item 的 base_price 是 int）
  ✓ Go 的整数运算天然精确，不需要额外库
  ✓ 架构设计方案已确定 BIGINT 整数方案（ADR #1）
```

### 4.4 结合自身情况的最终决策

```
决策：全链路整数存储方案

  金额：BIGINT × 币种最小单位倍率（VND ×1, THB ×100, USDT ×10^8）
  汇率：BIGINT × 10^8
  比例：BIGINT × 10^4
  传输：string（IDL 中金额字段一律 string 类型）
  计算：Go int64，全程整数运算，最后一步取整
  展示：按 currency_config.decimal_digits 配置除回去格式化

特别注意的坑：
  1. 除法精度丢失 — Go 整数除法直接截断，需要先乘后除（放大分子精度）
  2. 币种倍率映射 — 需要一个统一的工具函数 AmountToSmallest(amount, decimalDigits)
  3. 前端展示 — 千分位符号和小数点符号由后端返回，前端只做格式化
```

## 五、挑战4：跨服务数据一致性（无分布式事务）【P0】

### 5.1 挑战描述

```
核心问题：钱包操作涉及多个服务协作，如何在不使用分布式事务的情况下保证一致性？

真实场景：
  场景A - 创建提现订单：
    ser-wallet 冻结余额（本地事务）→ 调财务模块创建审核订单（RPC）
    如果 RPC 失败：余额已冻结但订单没创建 → 用户余额被"锁死"

  场景B - 充值上账：
    财务模块回调 ser-wallet.CreditWallet → ser-wallet 执行上账（本地事务）
    如果超时：财务不确定是否成功 → 重试 → 可能重复入账

  场景C - 提现出款成功：
    财务模块回调 ser-wallet.ConfirmDebit → ser-wallet 确认扣除冻结金额
    如果超时：财务不确定 → 重试 → 可能重复确认

为什么不用分布式事务：
  1. 现有工程没有引入任何分布式事务框架（无 Seata、无 DTM）
  2. 业界共识：微服务架构下 2PC 阻塞严重，TCC 复杂度高
  3. 蚂蚁金服、微信支付的公开分享都表示"避免分布式事务，用本地事务+最终一致性"
```

### 5.2 业界方案对比

```
┌──────────────────────┬──────────────────────────┬──────────────────┬──────────────────┐
│ 方案                  │ 原理                      │ 一致性           │ 复杂度           │
├──────────────────────┼──────────────────────────┼──────────────────┼──────────────────┤
│ ① 2PC（两阶段提交）   │ 协调者统一提交/回滚       │ 强一致           │ 极高（阻塞式）   │
├──────────────────────┼──────────────────────────┼──────────────────┼──────────────────┤
│ ② TCC                │ Try-Confirm-Cancel 三阶段 │ 准强一致         │ 高（三套接口）   │
├──────────────────────┼──────────────────────────┼──────────────────┼──────────────────┤
│ ③ Saga 编排           │ 本地事务链 + 补偿         │ 最终一致         │ 中等             │
├──────────────────────┼──────────────────────────┼──────────────────┼──────────────────┤
│ ④ 本地事务+幂等重试   │ 先做本地操作，外部调用    │ 最终一致         │ 低               │
│                      │ 失败则补偿/重试           │                  │ （最小依赖）     │
├──────────────────────┼──────────────────────────┼──────────────────┼──────────────────┤
│ ⑤ Outbox + 事件驱动  │ 本地事务写 outbox 表，    │ 最终一致         │ 中低             │
│                      │ 异步发布事件             │                  │ （需要消息中间件）│
└──────────────────────┴──────────────────────────┴──────────────────┴──────────────────┘
```

### 5.3 方案详细分析

**主流方案A：本地事务 + 补偿 + 幂等重试（推荐 — 我们的最佳实践）**

```
核心思路：
  先做本地可控的操作（冻结/记录），再做外部不可控的操作（RPC 调用）
  外部失败 → 补偿本地操作（解冻）
  外部超时 → 调用方重试 → 被调用方幂等保证

提现场景完整流程：

  ┌─ ser-wallet 本地 ─────────────────────────────────────────────────┐
  │ STEP 1: 开启事务                                                  │
  │ STEP 2: freezeCore(余额冻结) + 写流水 + 写冻结记录                  │
  │ STEP 3: 提交事务 ← 这一步完成后，用户余额已被冻结                    │
  └───────────────────────────────────────────────────────────────────┘
           │
           ▼ 事务外（RPC 调用）
  ┌─ 调财务模块 ─────────────────────────────────────────────────────┐
  │ STEP 4: rpc.FinanceClient().CreateWithdrawOrder()                │
  │   → 成功：流程正常，等待财务审核后回调                               │
  │   → 失败/超时：                                                    │
  │     STEP 5: unfreezeCore(余额解冻) ← 补偿操作，本地事务              │
  └───────────────────────────────────────────────────────────────────┘

  财务审核后回调：
  ┌─ 财务模块 → ser-wallet ─────────────────────────────────────────┐
  │ 审核通过+出款成功 → ConfirmDebit(orderNo) → 冻结金额永久扣除       │
  │ 审核驳回/出款失败 → UnfreezeBalance(orderNo) → 冻结金额退回可用     │
  │ 以上两个接口都有幂等保证（orderNo + 冻结状态机）                     │
  └───────────────────────────────────────────────────────────────────┘

为什么这是我们的最佳实践：
  ✓ 不引入任何分布式事务框架（与现有工程一致）
  ✓ 不引入消息中间件做一致性保证（RPC 重试 + 幂等已足够）
  ✓ "先冻结再调外部"的模式确保：
    冻结成功 + 外部成功 → 正常流程
    冻结成功 + 外部失败 → 解冻补偿（钱回来了）
    冻结失败 → 直接报错（钱没动）
    无论哪种情况，用户的钱都不会"凭空消失"
```

**可选方案B：TCC 模式**

```
原理：把冻结操作拆成 Try / Confirm / Cancel 三阶段

  Try    → 冻结余额（预留资源）
  Confirm → 确认扣除（提交资源）
  Cancel  → 解冻退回（释放资源）

实际上我们的冻结-确认/解冻本身就是 TCC 的思想！
  freezeCore   = Try
  confirmDebit = Confirm
  unfreeze     = Cancel

只是我们没有引入 TCC 框架（如 DTM / Seata-Go），
而是手动在 Service 层编排这三步。

为什么不用 TCC 框架：
  × 需要引入额外框架（DTM / Seata-Go），增加部署和运维复杂度
  × 框架需要注册 Try/Confirm/Cancel 三个接口，API 变多
  × 我们的场景简单（只有冻结-确认-解冻三步），手动编排即可
  × 现有工程没有使用任何 TCC 框架
```

**可选方案C：Outbox + 事件驱动**

```
原理：本地事务中同时写 outbox 表，异步发布事件到消息队列

  BEGIN;
    UPDATE user_wallet SET balance = balance - amount WHERE ...;
    INSERT INTO outbox (event_type, payload) VALUES ('BALANCE_DEBITED', '...');
  COMMIT;
  -- 异步进程读 outbox → 发布到 NATS/Kafka → 订阅方处理

优势：可靠的事件发布（本地事务保证 outbox 写入，异步保证发布）
劣势：需要 outbox 表 + 消费者 + 消息中间件

为什么我们当前不引入：
  × 余额变更是同步操作（调用方需要知道结果，不能异步）
  × 现有工程虽有 NATS JetStream，但用于异步场景
  × 引入 outbox 增加复杂度，当前不需要

什么时候考虑引入：
  → 余额变更后需要通知多个下游（统计/风控/推送）
  → 批量发奖等需要异步处理的场景
  → 当前预留即可
```

### 5.4 结合自身情况的最终决策

```
决策：本地事务 + 补偿 + 幂等重试

具体策略（按场景分）：

  充值上账：
    财务回调 → ser-wallet.CreditWallet (幂等 orderNo)
    超时重试由财务方负责
    ser-wallet 端保证幂等（流水表唯一索引）

  提现冻结+创建审核：
    ser-wallet 先冻结（本地事务）→ 调财务创建审核单
    失败 → 解冻补偿

  提现确认/退回：
    财务回调 → ConfirmDebit / UnfreezeBalance (幂等 orderNo + 冻结状态机)

  兑换：
    纯本地事务，不涉及外部调用，无一致性问题

  投注扣款/返奖：
    投注模块 → DebitWallet / CreditWallet (幂等 orderNo)
    重试由投注方负责
```

## 六、挑战5：缓存与数据库的一致性【P1】

### 6.1 挑战描述

```
核心问题：用户打开钱包看到的余额是缓存值，余额变更后缓存和数据库之间有短暂不一致窗口

典型场景：
  用户刚充值成功 → 回到钱包首页 → 看到的还是旧余额（缓存未更新）→ 用户恐慌
  用户提现冻结后 → 查余额 → 可用余额还是旧值 → 用户以为冻结没生效

不一致的危险范围：
  余额缓存：影响用户体验（看到错误余额）
  币种配置缓存：影响功能正确性（禁用的币种仍可操作）
  汇率缓存：影响计算准确性（用了过期汇率兑换）
```

### 6.2 业界方案对比

```
┌──────────────────────┬──────────────────────────┬──────────────────┬──────────────────┐
│ 方案                  │ 原理                      │ 一致性窗口       │ 适用场景         │
├──────────────────────┼──────────────────────────┼──────────────────┼──────────────────┤
│ ① Cache-Aside        │ 写DB→删缓存，读时回填     │ 短（毫秒级）     │ 读多写少         │
│ (先写DB再删缓存)     │                          │                  │ 余额/币种配置    │
├──────────────────────┼──────────────────────────┼──────────────────┼──────────────────┤
│ ② Write-Through      │ 写DB同时写缓存            │ 无               │ 一致性要求极高   │
│                      │                          │                  │ 银行核心系统     │
├──────────────────────┼──────────────────────────┼──────────────────┼──────────────────┤
│ ③ Write-Behind       │ 先写缓存，异步写DB        │ 无（缓存侧）     │ 写入极高频       │
│ (Write-Back)         │                          │ 但DB有延迟       │ 不适合资金系统   │
├──────────────────────┼──────────────────────────┼──────────────────┼──────────────────┤
│ ④ 不缓存（直读DB）    │ 每次都查数据库            │ 无               │ 强一致场景       │
│                      │                          │                  │ 低频操作         │
└──────────────────────┴──────────────────────────┴──────────────────┴──────────────────┘
```

### 6.3 方案详细分析

**主流方案A：Cache-Aside + 主动失效（推荐 — 我们的最佳实践）**

```
原理：
  写操作 → 数据库事务提交 → DEL Redis Key
  读操作 → 先读 Redis → 未命中则读 DB → 回填 Redis（带 TTL）

为什么用"删除"而不是"更新"缓存：
  并发写时，两个写操作的缓存更新顺序可能和数据库不一致（ABA 问题）
  DEL 是最安全的 — 下次读时自然拿到最新数据

为什么这是我们的最佳实践：
  ✓ 现有工程已有此模式（ser-item / ser-app 的缓存管理）
  ✓ 业界最经典、最成熟的缓存模式
  ✓ 不一致窗口只有"DEL 操作到下次读之间"的毫秒级

缓存分层策略（来自架构设计方案）：

  数据类型          │ Key 模式                        │ TTL     │ 失效策略
  ─────────        │ ──────                          │ ───     │ ──────
  用户余额          │ wallet:bal:{uid}:{code}         │ 30s     │ 余额变更后 DEL
  币种列表          │ wallet:currencies               │ 5min    │ B端编辑后 DEL
  币种详情          │ wallet:curr:{code}              │ 5min    │ B端编辑后 DEL
  汇率数据          │ wallet:rate:{code}              │ 定时覆盖 │ 定时任务写入新值
```

**关键补充：余额查询的读写分离策略**

```
展示场景（允许短暂不一致）：
  钱包首页查余额 → 走缓存 → 看到的余额可能有 30s 延迟 → 可接受
  记录列表查询 → 不缓存，直接查 DB → 数据量大，不适合缓存

操作场景（必须强一致）：
  扣款/冻结时的余额检查 → 不走缓存，由数据库 WHERE balance >= amount 保证
  兑换时的汇率获取 → 走缓存（定时任务刷新），误差在阈值控制范围内

核心原则：
  写操作的前置校验 → 靠数据库（权威数据源）
  读操作的数据展示 → 走缓存（提升性能）
  绝不用缓存值做资金操作的判断依据
```

**可选方案B：缓存击穿防护（Singleflight）**

```
问题：热门用户的余额缓存被 DEL 后，大量并发请求同时查库

方案 1 — Go singleflight（推荐如果需要）：
  var group singleflight.Group
  result, err, _ := group.Do(cacheKey, func() (interface{}, error) {
      return db.QueryBalance(userID, currCode)  // 只有一个 goroutine 执行
  })

方案 2 — Redis 分布式锁：
  缓存未命中 → SETNX lock_key → 获取锁的那个请求去查库回填 → 其他请求等待

为什么我们当前不引入：
  × 现有工程没有 singleflight 模式
  × 余额缓存 TTL 短（30s）+ 主动失效，击穿窗口极短
  × 单用户并发请求量不会极高
  × 不过度设计

什么时候引入：
  → 监控发现某些用户频繁触发缓存未命中导致 DB 压力
  → 在 cache 层加 singleflight 只需几行代码，随时可加
```

### 6.4 结合自身情况的最终决策

```
决策：Cache-Aside + 主动 DEL + 短 TTL

规则：
  1. 余额变更后，在事务提交后立即 DEL 缓存
  2. 缓存未命中时从 DB 读取并回填
  3. 写操作的余额校验不依赖缓存
  4. 当前不引入 singleflight，监控后按需加入
```

## 七、挑战6：安全防护——防超扣、防重放、防篡改【P0】

### 7.1 挑战描述

```
资金系统的安全要求远高于普通业务系统。攻击者的目标非常明确——钱。

我们面临的安全威胁：
  威胁1：超扣攻击 — 利用并发请求同时扣款，超出实际余额
  威胁2：重放攻击 — 截获一次成功的充值回调，重复发送实现多次入账
  威胁3：参数篡改 — 篡改请求中的 user_id / amount / currency 字段
  威胁4：越权访问 — 操作他人钱包余额
  威胁5：敏感数据泄露 — 银行账号、手机号等被明文存储/传输
```

### 7.2 逐项防护方案

**防超扣（P0）：**

```
方案层次：

  第一层 — 数据库条件更新（主防线）：
    UPDATE user_wallet SET available_balance = available_balance - ?
    WHERE available_balance >= ?
    → 数据库层面保证不会扣成负数

  第二层 — 数据库 CHECK 约束（兜底）：
    ALTER TABLE user_wallet ADD CONSTRAINT chk_balance
    CHECK (available_balance >= 0 AND frozen_balance >= 0);
    → TiDB 支持 CHECK 约束，即使应用层有 bug 也不会穿透

  第三层 — 冻结状态机互斥（防冻结金额被双重操作）：
    UPDATE wallet_freeze_record SET status = 3  -- 确认扣除
    WHERE order_no = ? AND status = 1           -- 只允许从"冻结中"状态转
    → Frozen → Confirmed 和 Frozen → Unfrozen 互斥
```

**防重放攻击（P0）：**

```
方案层次：

  第一层 — 幂等保证（核心防线）：
    wallet_flow 表 (order_no, flow_type) 唯一索引
    → 同一笔业务操作不可能入账两次

  第二层 — 回调签名校验（防伪造）：
    财务模块回调时带签名 → ser-wallet 验签
    → 防止第三方伪造回调请求

  第三层 — 请求时效校验（可选补充）：
    请求中带 timestamp → 校验是否在 5 分钟内
    → 过期请求直接拒绝

我们的决策：
  ✓ 必做：幂等（已在架构设计中确定）
  ✓ 必做：回调签名（财务模块联调时确认签名机制）
  ○ 可选：时效校验（如果财务模块有此机制则配合，无则暂不加）
```

**防参数篡改（P0）：**

```
方案：

  C端请求的 user_id 绝不信任客户端传入：
    user_id 从 context 中取 → tracer.GetUserId(ctx)
    由 gate-font 的 FontAuthMiddleware 从 Token 中提取并注入 context
    Handler 层不读取请求体中的 user_id 字段

  C端请求的金额校验：
    金额必须 > 0（防止传入负数变上账为扣款）
    金额不超过合理上限（防止 int64 溢出）
    币种必须存在且已启用

  B端操作的权限控制：
    gate-back 的 CheckUrlAuth 中间件确保管理员只能访问被授权的 URL
    所有 B 端写操作记录审计日志（调 ser-blog）
```

**防越权访问（P0）：**

```
方案：

  C端：user_id 从 Token 提取，无法伪造
  B端：URL 权限控制 + IP 白名单
  RPC：服务间调用通过 ETCD 服务发现，内网通信
    → 外部无法直接调用 ser-wallet 的 RPC 接口

已有机制（直接复用，不额外设计）：
  gate-font: BlackCheck → FontAuthMiddleware (Token + IP黑名单)
  gate-back: WhiteCheck → BackAuthMiddleware → CheckUrlAuth (Token + IP白名单 + URL权限)
```

**敏感数据加密（P1）：**

```
方案：

  提现账户的银行账号、手机号使用 EncryptedString 类型
  现有工程 ser-kyc 已有此模式 — GORM 自动加解密

  实现：
    type WithdrawAccount struct {
        BankAccountNo EncryptedString `gorm:"column:bank_account_no;comment:银行账号(加密)" json:"bank_account_no"`
        PhoneNumber   EncryptedString `gorm:"column:phone_number;comment:手机号(加密)" json:"phone_number"`
    }

  EncryptedString 的 GORM 钩子：
    写入时自动 AES 加密 → 存储为密文
    读取时自动 AES 解密 → 返回明文

  加密密钥管理：
    通过 ETCD 配置，不硬编码在代码中
    与 ser-kyc 使用相同的加密方案
```

### 7.3 结合自身情况的最终决策

```
安全防护总结：

  防超扣 → 条件更新 + CHECK 约束 + 冻结状态机 【自己实现】
  防重放 → 幂等唯一索引 + 回调签名验证 【自己实现 + 联调确认】
  防篡改 → user_id 服务端取 + 金额校验 【复用已有 + 自己补充】
  防越权 → 复用 gate 中间件链 【已有机制】
  数据加密 → EncryptedString 类型 【复用 ser-kyc 模式】
  审计日志 → B端操作调 ser-blog 【复用已有】

不需要额外引入：
  × WAF（Web 应用防火墙）— 由运维/基础设施团队管
  × 风控引擎 — 财务模块负责
  × 反欺诈 — 超出钱包模块边界
```

## 八、挑战7：状态机的严格性与容错【P1】

### 8.1 挑战描述

```
核心问题：充值订单、提现订单、冻结记录都有多种状态，状态流转必须严格按照预定义的路径进行

如果状态机不严格：
  冻结记录被同时"确认扣除"和"解冻退回" → 冻结金额出现正负偏差
  提现订单跳过"待风控审核"直接变成"出款成功" → 风控形同虚设
  充值订单已经"支付成功"又被改成"支付失败" → 已入账的钱被错误冲回

容错问题：
  订单卡在中间状态（如"待支付"超时不回调）→ 永远不会结束
  回调消息顺序错乱（先收到"成功"再收到"进行中"）→ 状态倒退
```

### 8.2 业界方案对比

```
┌──────────────────────┬──────────────────────────┬──────────────────┬──────────────────┐
│ 方案                  │ 原理                      │ 代表             │ 适用场景         │
├──────────────────────┼──────────────────────────┼──────────────────┼──────────────────┤
│ ① 状态机框架         │ 引入专业框架管理状态流转   │ Spring           │ Java 生态        │
│ (StateMachine)       │                          │ StateMachine     │ 复杂状态流       │
├──────────────────────┼──────────────────────────┼──────────────────┼──────────────────┤
│ ② WHERE 条件控制     │ SQL UPDATE 加上 WHERE     │ 大多数支付系统   │ 状态路径简单     │
│ (数据库层面)         │ status = expected_status  │                  │ 微服务场景       │
├──────────────────────┼──────────────────────────┼──────────────────┼──────────────────┤
│ ③ 事件溯源           │ 存储事件序列，派生当前状态│ 事件驱动系统     │ 需要完整审计     │
│ (Event Sourcing)     │                          │ 高合规金融系统   │ 复杂度极高       │
└──────────────────────┴──────────────────────────┴──────────────────┴──────────────────┘
```

### 8.3 方案详细分析

**主流方案A：WHERE 条件控制（推荐 — 我们的最佳实践）**

```
原理：状态流转用 UPDATE ... WHERE status = expected_status 保证

  示例 — 冻结→确认扣除：
    UPDATE wallet_freeze_record SET status = 3  -- Confirmed
    WHERE order_no = ? AND status = 1           -- 只允许从 Frozen 状态转

    affected_rows == 1 → 成功，状态合法
    affected_rows == 0 → 失败，状态不对（可能已被解冻/已被确认/不存在）

  示例 — 提现订单：风控通过 → 待财务审核：
    UPDATE withdraw_order SET status = 3        -- 待财务审核
    WHERE order_no = ? AND status = 1           -- 只允许从"待风控审核"状态转

三个状态机的转换规则表：

  冻结记录状态机：
    From(1:冻结中)  → To(2:已解冻)      Event: 驳回/失败
    From(1:冻结中)  → To(3:已确认扣除)   Event: 出款成功
    终态: 2, 3 — 不可逆

  充值订单状态机：
    From(1:待支付)   → To(2:进行中)      Event: 发起支付
    From(1:待支付)   → To(5:订单失效)    Event: 超时
    From(2:进行中)   → To(3:支付成功)    Event: 通道回调成功
    From(2:进行中)   → To(4:支付失败)    Event: 通道回调失败/超时
    终态: 3, 4, 5 — 不可逆

  提现订单状态机：
    From(1:待风控审核) → To(2:风控驳回)    Event: 风控驳回
    From(1:待风控审核) → To(3:待财务审核)  Event: 风控通过
    From(3:待财务审核) → To(4:财务驳回)    Event: 财务驳回
    From(3:待财务审核) → To(5:待出款)      Event: 财务通过
    From(5:待出款)     → To(6:出款成功)    Event: 通道出款成功
    From(5:待出款)     → To(7:出款失败)    Event: 通道出款失败
    驳回/失败 → 触发 UnfreezeBalance
    出款成功 → 触发 ConfirmDebit
    终态: 2, 4, 6, 7 — 不可逆

为什么这是我们的最佳实践：
  ✓ 不引入额外框架（Go 生态无主流状态机框架）
  ✓ 数据库层面保证（即使并发调用也不会出现状态错乱）
  ✓ 现有工程已有类似模式（状态字段 + WHERE 条件控制）
  ✓ 状态路径相对简单，不需要复杂的状态机框架
```

**容错补充：超时订单处理**

```
问题：订单卡在"待支付"/"进行中"状态，三方通道不回调

方案：定时任务扫描超时订单

  实现方式（两种可选）：

  方式A — 服务内部 goroutine（推荐当前阶段）：
    go func() {
        ticker := time.NewTicker(1 * time.Minute)
        for range ticker.C {
            expireStaleOrders(ctx)  // 扫描 create_at < now()-30min 且 status IN (1,2) 的订单
        }
    }()

  方式B — 通过 ser-cron 注册：
    如果 ser-cron 支持分钟级调度，注册一个定时任务

  超时处理逻辑：
    充值订单超时 → 标记为"订单失效"(5) → 不涉及余额操作（钱还没加）
    提现订单超时 → 视情况而定（可能需要人工介入）

为什么需要这个：
  三方通道可能丢回调、延迟回调
  如果订单永远停在中间状态，用户无法发起新订单（重复充值检测会拦截）
```

### 8.4 结合自身情况的最终决策

```
决策：WHERE status = expected 条件控制 + 定时超时扫描

规则：
  1. 所有状态流转必须通过 WHERE 条件保证合法性
  2. 终态不可逆（不允许从终态转回中间态）
  3. 充值订单超时扫描：30分钟无回调 → 标记失效
  4. 提现订单卡单：由 B 端人工处理（财务模块负责）
  5. 回调顺序错乱防护：终态忽略后续回调（WHERE status NOT IN (终态)）
```

## 九、挑战8：汇率管理——实时性、锁定、精度【P1】

### 9.1 挑战描述

```
核心问题：汇率是动态变化的，但交易执行需要确定的汇率

三个子问题：

  子问题1 — 实时性：
    三方 API 返回的汇率有延迟，平台汇率可能落后于市场
    如果汇率更新不及时 → 用户用有利的旧汇率兑换 → 平台亏损

  子问题2 — 锁定：
    用户看到汇率后点击兑换，但在这几秒内汇率可能已经变了
    如果用看到的旧汇率结算 → 可能不公平（对用户或对平台）

  子问题3 — 精度：
    汇率计算涉及多次乘除运算
    中间结果的精度丢失会被放大
    加密货币汇率需要 8 位小数精度
```

### 9.2 业界方案对比

```
┌──────────────────────┬──────────────────────────┬──────────────────┐
│ 方案                  │ 原理                      │ 适用场景         │
├──────────────────────┼──────────────────────────┼──────────────────┤
│ ① 定频刷新+阈值触发  │ 定时刷新实时汇率，        │ 中低频交易       │
│ (我们的需求)         │ 偏差超阈值才更新平台汇率   │ 平台类钱包       │
├──────────────────────┼──────────────────────────┼──────────────────┤
│ ② 实时报价+锁定      │ 每次交易实时报价，         │ 外汇交易平台     │
│ (Modern Treasury)    │ 报价锁定 N 分钟有效       │ 高频交易         │
├──────────────────────┼──────────────────────────┼──────────────────┤
│ ③ WebSocket 实时流    │ 订阅交易所实时汇率流      │ 专业交易所       │
│ (Binance/Coinbase)   │                          │ 做市商           │
└──────────────────────┴──────────────────────────┴──────────────────┘
```

### 9.3 方案详细分析

**主流方案A：定频刷新+阈值触发（推荐 — 我们的最佳实践）**

```
产品需求已明确的机制：
  1. 实时汇率 = 3家三方 API 取平均值
  2. 平台汇率 = 当偏差值 > 阈值时更新为实时汇率
  3. 入款汇率 = 实时汇率 × (1 + 浮动%)
  4. 出款汇率 = 实时汇率 × (1 - 浮动%)
  5. 偏差值 = |实时汇率 - 平台汇率| / 实时汇率 × 100

技术实现要点：

  定时刷新（建议每分钟一次）：
    遍历启用的非基准币种 →
      并行调 3 家三方 API →
      取平均值得实时汇率 →
      计算偏差值 →
      偏差 >= 阈值 → 更新四种汇率 + 写日志 →
      全部结果写 Redis 缓存

  API 容错策略：
    3 家全部成功 → 取平均值
    2 家成功 + 1 家超时 → 取 2 家平均值
    1 家成功 → 使用该值（风险稍高，记日志告警）
    全部失败 → 跳过本轮，保持上一轮数据，触发告警

  汇率写入的原子性：
    平台汇率、入款汇率、出款汇率必须在同一事务内更新
    不能出现"平台汇率更新了但入款汇率还是旧的"

  汇率锁定策略（兑换/BSB充值/BSB提现）：
    用户发起操作时 → 从 Redis 缓存读取当前汇率
    汇率写入订单/记录（作为"快照"）
    结算时用订单上的快照汇率，不用最新汇率
    这样即使操作过程中汇率变了，结算结果也是确定的
```

**可选方案B：实时报价+锁定（Modern Treasury / Goldman Sachs 模式）**

```
原理：每次交易前先调 "GetQuote" 获取报价，报价带有效期

  quote = {
      rate: 2620.00,
      source: "VND",
      target: "BSB",
      expires_at: "2026-02-26T12:05:00Z",  // 5分钟有效
      quote_id: "Q20260226001"
  }

  用户确认兑换时，传入 quote_id → 服务端校验报价未过期 → 用该报价汇率结算

优势：汇率变动风险最小化（每笔交易都用最新报价）
劣势：每笔交易多一次 API 调用，复杂度增加

为什么我们不首选：
  × 产品需求已定义了"阈值触发更新"机制，不需要每笔交易都报价
  × 我们的兑换频率不高（不是专业外汇平台）
  × 汇率快照写入订单已足够

什么时候考虑：
  → 如果平台扩展到高频兑换场景
  → 如果产品要求"兑换确认时再拿一次最新汇率"
```

**三方汇率 API 选择（待确认）：**

```
主流 API 推荐（需确认哪些适合我们的币种）：

  ┌──────────────────────┬──────────────────────┬──────────────────┐
  │ API                   │ 特点                 │ 免费额度         │
  ├──────────────────────┼──────────────────────┼──────────────────┤
  │ ExchangeRate-API      │ 法币汇率，简单易用   │ 1500次/月        │
  │ Open Exchange Rates   │ 法币+加密，历史数据   │ 1000次/月        │
  │ CurrencyLayer        │ 法币为主，可靠性高    │ 100次/月（免费版）│
  │ CoinGecko            │ 加密货币专精          │ 10000次/月       │
  │ Binance API           │ 加密货币实时价格      │ 高               │
  └──────────────────────┴──────────────────────┴──────────────────┘

  建议组合：法币用 2~3 家法币 API，加密货币用 1~2 家交易所 API
  具体选择需要结合 API Key 管理和成本确认
```

### 9.4 结合自身情况的最终决策

```
决策：
  1. 定频刷新（每分钟）+ 阈值触发更新平台汇率
  2. 汇率快照写入订单（兑换/充值/提现时锁定汇率）
  3. 三方 API 容错（3 家取平均，渐进降级）
  4. 汇率更新原子性（同一事务内更新四种汇率）
  5. 三方 API 具体选择待确认
```

## 十、挑战9：热点账户与写入性能【P2】

### 10.1 挑战描述

```
核心问题：当大量请求同时写同一个用户的同一行钱包记录时，数据库行级竞争成为瓶颈

我们可能遇到的热点场景：
  场景A：热门主播秒级收到大量打赏 → 同一个 anchor_wallet 行被高频写入
  场景B：活动批量发奖 → 短时间内大量用户的 reward_wallet 被写入
  场景C：投注批量结算 → 短时间内大量用户的 center_wallet 被写入

评估当前量级：
  普通用户：并发写入不会超过 10 TPS/用户（一个人不可能同时做 10 笔操作）
  热门主播：并发写入可能达到 100~500 TPS（百人同时打赏）
  批量结算：虽然总量大，但分散在不同用户（不同行），不存在单行热点

结论：当前阶段不需要特殊优化，但需要知道方案
```

### 10.2 业界方案

```
方案A — 账户分片（蚂蚁金服模式）：
  一个逻辑账户拆成 N 个子账户
  写入分散到不同行 → 并发度提升 N 倍
  读取时聚合 SUM(balance) → 读取变复杂
  适合：极高并发的结算账户（如平台收款账户）

方案B — 异步批量写入：
  写入请求先入队列（NATS/Kafka）
  消费者批量合并后统一写入
  适合：不需要即时返回结果的场景（批量发奖）

方案C — 减少锁竞争的代码优化（我们应该做的）：
  事务最小化：事务内只做必要操作（UPDATE wallet + INSERT flow）
  事务外做：缓存失效、日志记录、RPC 通知
  减少事务持有时间 = 减少行锁等待时间

我们的决策：
  当前：做好事务最小化，不引入账户分片和异步队列
  未来：如果监控发现单行写入瓶颈（TiDB 慢查询日志），再评估分片方案
  预留：批量发奖接口可以设计为异步模式（接收请求→入 NATS→消费者逐条处理）
```

## 十一、挑战10：对账与审计——资金安全的最后防线【P1】

### 11.1 挑战描述

```
核心问题：如何验证"余额数字"和"流水记录"是一致的？如何发现资金异常？

对账的必要性：
  即使有幂等、有事务、有条件更新，仍然可能因为未知 bug 导致数据不一致
  对账是"最后一道防线"——不是为了防止问题发生，而是为了发现问题已经发生

两级对账：
  内部对账：wallet 余额 vs wallet_flow 流水汇总
  外部对账：wallet 余额/订单 vs 财务模块记录（跨系统核对）
```

### 11.2 业界方案

**内部对账（我们必须做的）：**

```
原理：一个账户的当前余额 = 所有入账流水之和 - 所有出账流水之和

SQL 对账查询：
  SELECT
      uw.user_id,
      uw.currency_code,
      uw.wallet_type,
      uw.available_balance AS stored_balance,
      COALESCE(SUM(wf.amount), 0) AS computed_balance
  FROM user_wallet uw
  LEFT JOIN wallet_flow wf
      ON uw.user_id = wf.user_id
      AND uw.currency_code = wf.currency_code
      AND uw.wallet_type = wf.wallet_type
  GROUP BY uw.user_id, uw.currency_code, uw.wallet_type
  HAVING uw.available_balance != COALESCE(SUM(wf.amount), 0);

  -- 返回结果应为 0 行。如果有结果，说明余额和流水不一致。

流水表设计的对账友好性（已在表结构设计中考虑）：
  amount 字段：正数 = 入账，负数 = 出账
  balance_before / balance_after 字段：每条流水记录变更前后余额
  → 可以逐条追踪余额变化链条

对账频率：
  实时对账（每笔操作后）→ 太重，不推荐
  定时对账（每小时/每天）→ 推荐，作为定时任务
  人工触发对账（B端操作）→ 可选，提供给运营人员
```

**流水审计（我们必须做的）：**

```
原理：每条流水不可篡改，记录完整的操作链条

  wallet_flow 的审计字段：
    flow_id        — 唯一标识
    user_id        — 操作对象
    order_no       — 关联业务单号（可追溯到充值/提现/兑换等原始操作）
    flow_type      — 操作类型（充值/扣款/冻结/...）
    amount         — 变更金额
    balance_before — 变更前余额
    balance_after  — 变更后余额
    create_at      — 操作时间
    remark         — 备注（操作人/操作原因）

  审计追踪能力：
    给定任意时间点 → 通过 balance_before / balance_after 链条还原当时余额
    给定任意订单 → 通过 order_no 找到所有关联流水
    给定异常余额 → 通过对账发现 → 通过流水链定位问题
```

### 11.3 结合自身情况的最终决策

```
决策：
  1. wallet_flow 表记录 balance_before / balance_after（已在表结构中设计）
  2. 定时对账任务：每天凌晨运行，对比余额和流水汇总
  3. 对账不一致时：写告警日志，通知运营人员
  4. 外部对账：与财务模块联调时确认对账机制
  5. 优先级：对账是 P1，在核心功能上线后再实现
```

## 十二、挑战11：提现三重过滤的复杂联动【P1】

### 12.1 挑战描述

```
核心问题：提现方式的可用列表需要三个来源的数据做交集运算

三个过滤源：
  过滤1 — KYC 状态：未认证 → 仅 USDT；已认证 → 全渠道
  过滤2 — 充值历史：法币充 → 法币提；USDT充 → USDT提；混合充 → 法币提
  过滤3 — 财务配置：该币种下财务配置了哪些可用提现方式

最终可用方式 = 过滤1 ∩ 过滤2 ∩ 过滤3

复杂度在于：
  三个过滤源来自不同的数据源：
    KYC 状态 → 调 ser-kyc（RPC）
    充值历史 → 查自己的 deposit_order 表
    财务配置 → 调财务模块（RPC）或缓存
  三个 RPC/查询的失败处理策略不同
  交集运算后可能为空（用户看到"没有可用的提现方式"）
```

### 12.2 解决方案

```
实现思路：

  Step 1: 并行获取三个过滤源的数据
    kycResult    := rpc.KycClient().KycQuery(ctx, kycReq)     // 并行
    depositHist  := depositRep.GetCurrencyDepositHistory(ctx)  // 并行
    financeConf  := rpc.FinanceClient().GetWithdrawConfig(ctx) // 并行

  Step 2: 各自得到候选方式集合
    kycMethods     = kycStatusToMethods(kycResult)      // 未认证=[USDT], 已认证=[银行卡,电子钱包,USDT]
    historyMethods = depositHistoryToMethods(depositHist) // 法币充=[银行卡,电子钱包], USDT充=[USDT]
    configMethods  = financeConfigToMethods(financeConf)  // 从财务配置拿到的可用方式列表

  Step 3: 三者取交集
    availableMethods = intersection(kycMethods, historyMethods, configMethods)

  Step 4: 降级策略
    KYC 不可用 → 视为"未认证"（保守策略），仅保留 USDT
    充值历史查询失败 → 跳过此过滤（宽松策略），不限制
    财务配置不可用 → 返回错误（无法确定可用方式）

为什么并行获取：
  三个数据源互不依赖，并行可以减少总耗时（3 个 RPC 串行 = 3×100ms = 300ms，并行 ≈ 100ms）

特别注意：
  "混合充值 → 法币提现" 的判断：
    查 deposit_order 表，WHERE user_id = ? AND currency_code = ? AND status = 成功
    如果同时存在法币充值和 USDT 充值 → 判为混合 → 可提法币
    这个逻辑需要扫描用户的充值历史，考虑加索引优化
```

## 十三、挑战12：扣款分配规则（优先中心再奖励）【P1】

### 13.1 挑战描述

```
核心问题：消费扣款时要按"优先中心钱包，不足部分扣奖励钱包"的规则分配

产品需求：
  "资金钱包余额 = 中心钱包 + 奖励钱包"
  "消费时优先扣除中心钱包余额，再扣奖励钱包余额"
  "混合投注时需根据投注金额占比分开计返奖"

挑战点：
  1. 需要先查两个钱包的余额，再计算分配比例，再分别扣减
  2. 查和扣之间可能有并发操作改变了余额
  3. 扣款比例需要返回给调用方（投注模块需要知道各钱包扣了多少，用于返奖时按比例分配）
  4. 两个 UPDATE 必须在同一事务内（要么都成功，要么都失败）
```

### 13.2 解决方案

```
实现方案（在 debitCore 中实现）：

  func debitCore(tx *gorm.DB, userID int64, currCode string, totalAmount int64, orderNo string) (*DebitResult, error) {
      // 1. 在事务内查两个钱包余额
      centerWallet := getWalletInTx(tx, userID, currCode, WalletTypeCenter)
      rewardWallet := getWalletInTx(tx, userID, currCode, WalletTypeReward)

      totalBalance := centerWallet.AvailableBalance + rewardWallet.AvailableBalance
      if totalBalance < totalAmount {
          return nil, ErrInsufficientBalance
      }

      // 2. 计算分配
      centerDebit := min(centerWallet.AvailableBalance, totalAmount)
      rewardDebit := totalAmount - centerDebit

      // 3. 分别扣减（条件更新）
      if centerDebit > 0 {
          debitWallet(tx, userID, currCode, WalletTypeCenter, centerDebit)
      }
      if rewardDebit > 0 {
          debitWallet(tx, userID, currCode, WalletTypeReward, rewardDebit)
      }

      // 4. 写流水（两条：中心扣减 + 奖励扣减）
      // 5. 返回分配结果
      return &DebitResult{
          CenterDebit: centerDebit,
          RewardDebit: rewardDebit,
      }, nil
  }

并发安全保证：
  整个操作在同一事务内完成
  每个 debitWallet 内部使用条件更新（WHERE balance >= amount）
  如果在查和扣之间有并发操作改变了中心余额导致条件不满足 →
    centerDebit 的条件更新 affected_rows = 0 →
    需要重新计算分配（或直接报错让调用方重试）

返回值的重要性：
  DebitResult.CenterDebit 和 RewardDebit 会返回给投注模块
  投注返奖时：中心钱包扣了多少比例 → 返奖多少到中心；奖励扣了多少 → 返奖多少到奖励
  这个比例关系是产品需求明确要求的
```

## 十四、挑战13：与财务模块的联调协作【P1】

### 14.1 挑战描述

```
核心问题：充值和提现流程强依赖财务模块，但财务模块由他人负责，接口可能还没定

联调风险：
  1. 财务模块的 IDL 还没定义 → 我们无法生成调用代码
  2. 接口签名不一致 → 联调时大量修改
  3. 调用时序理解不一致 → 回调时机不对
  4. 错误码含义不一致 → 错误处理逻辑错误
```

### 14.2 解决方案

```
策略：Mock 先行 + 渐进式联调

Phase 1：Mock 开发（独立开发阶段）
  在 ser-wallet 内部写 Mock 方法模拟财务模块的响应

  // internal/mock/finance_mock.go
  func MockGetPaymentConfig(currCode string) *PaymentConfig {
      return &PaymentConfig{
          Methods: []PaymentMethod{
              {Code: "bank_transfer", Name: "银行转账", ...},
              {Code: "qris", Name: "QRIS扫码", ...},
          },
      }
  }

  Service 层通过配置开关决定调 Mock 还是真实 RPC：
  if cfg.UseMockFinance {
      return mock.MockGetPaymentConfig(currCode)
  } else {
      return rpc.FinanceClient().GetPaymentConfig(ctx, req)
  }

Phase 2：接口对齐（联调准备阶段）
  与财务负责人对齐 6 个 P0 接口：
    我调他们：GetPaymentConfig / CreatePayOrder / CreateWithdrawOrder
    他们调我：CreditWallet / ConfirmDebit / UnfreezeBalance

  对齐内容：
    ✓ 服务名、方法签名（IDL 方法名 + 请求/响应结构）
    ✓ 必传字段 + 可选字段
    ✓ 调用时序（谁先调谁，回调时机）
    ✓ 错误码含义 + 重试策略
    ✓ 签名/验签机制（回调安全）

Phase 3：联调替换（联调阶段）
  把 Mock 替换为真实 RPC 调用
  只需改一个配置开关，Service 层逻辑不变

优势：
  ✓ 钱包模块可以独立开发，不被财务模块阻塞
  ✓ Mock → 真实 RPC 切换成本极低
  ✓ 接口对齐在联调前完成，减少联调时的返工
```

```
  ✓ 本地事务+补偿 → 微服务架构的主流一致性方案
  ✓ Cache-Aside → 最经典的缓存模式
  ✓ WHERE 条件控制状态机 → 简单有效的工程实践

说不难也需要周全——因为结合自身业务有特殊性：
  ✓ 多币种精度差异（VND 0位小数 vs USDT 8位小数）→ 统一 BIGINT×10^8 方案
  ✓ 扣款分配规则（中心优先再奖励）→ 需要返回分配比例给投注模块
  ✓ 提现三重过滤（KYC × 充值历史 × 财务配置）→ 并行获取+交集运算
  ✓ 冻结-确认/解冻状态机 → 本质就是 TCC 的手动版
  ✓ 与财务模块联调 → Mock 先行，不被阻塞

最佳实践的核心原则：
  1. 能用数据库解决的不用中间件（条件更新 > Redis锁 > 分布式事务框架）
  2. 能不引入的依赖就不引入（不用 Seata/DTM/状态机框架）
  3. 先做对，再做快（正确性 > 性能 > 代码优雅）
  4. 监控发现问题后再优化（不预测性能瓶颈，用数据说话）
  5. 与现有工程保持一致（条件更新、Cache-Aside、EncryptedString 都是已有模式）
```
