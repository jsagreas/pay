## 二、挑战①：并发与超卖防护 🔴

### 2.1 挑战描述

钱包最核心的风险：**同一账户在高并发下余额被扣成负数（超卖）或被重复加款**。

场景还原：
- 用户同时发起提现和游戏消费，两个请求并发读到同一余额 1000，各扣 800，最终余额 = -600
- 投注返奖和充值回调同时到达，两个 Credit 操作并发写入，流水记录的 before_amt 相同，数据链断裂
- 兑换操作涉及两个币种的账户（扣法币 + 加 BSB），如果锁顺序不固定，两个并发兑换可能死锁

这不是理论问题——**Robinhood 2021 年就因为并发竞态导致负余额，被 FINRA 罚款 7000 万美元**。

### 2.2 业界方案对比

| 方案 | 原理 | 代表企业/系统 | 优点 | 缺点 | 适用场景 |
|------|------|-------------|------|------|---------|
| **① 悲观锁 SELECT FOR UPDATE** | 事务内先获取行锁，其他事务等待 | 蚂蚁金服内部钱包、大量传统金融系统 | 强一致保证，无重试成本，实现直观 | 持锁时间影响吞吐；锁粒度需控制 | 写多读少、冲突频繁场景 |
| **② 乐观锁（Version/CAS）** | 读时记版本号，写时 WHERE version=old_version | Shopify Balance、部分电商库存系统 | 无锁等待，读性能好 | 高冲突时重试风暴；ABA 问题需额外处理 | 读多写少、冲突罕见场景 |
| **③ Redis 分布式锁** | Redis SetNX/Redlock 获取全局互斥锁 | 微服务间资源竞争场景 | 跨数据源、跨服务 | Redis 单点/脑裂风险；锁续期复杂 | 跨数据库/跨服务操作 |
| **④ 数据库原子 UPDATE** | `UPDATE SET balance = balance - ? WHERE balance >= ?` 利用 DB 行锁 | Square Cash App 早期 | 极简，一条 SQL 解决 | 无法获知操作前的余额（影响流水记录） | 极简场景，不需要 before/after 审计 |
| **⑤ Actor 模型/队列串行化** | 每个账户对应一个 Actor/Channel，请求排队串行处理 | Akka 体系、部分游戏钱包 | 无锁，天然串行化 | 需要 Actor 框架支撑；单账户吞吐受限于单 Actor | 游戏币等高频小额场景 |

### 2.3 推荐方案

**主流方案（推荐）：① 悲观锁 SELECT FOR UPDATE**

理由：
1. 我们的余额操作是**写密集**场景（每次充值/消费/提现/兑换都改余额），乐观锁重试成本过高
2. 所有余额操作在同一 TiDB 数据库同一事务内，不需要跨库分布式锁
3. TiDB 完全兼容 MySQL 的 `SELECT ... FOR UPDATE` 行锁语义
4. 锁粒度精确到 `(user_id, currency_code, wallet_type)` 单行，不同用户/不同币种互不阻塞
5. 持锁时间极短（一次读 + 一次写 + 一次 INSERT 流水，微秒级），吞吐影响可忽略

**可选方案（备用）：② 乐观锁 Version 字段降级**

我们在 wallet_account 表中保留了 `version` 字段。当未来某些查询场景（如余额展示）不需要强一致锁时，可以用乐观锁降级，减少锁争用。但**所有资金变动操作必须用悲观锁**。

### 2.4 死锁防护

**固定锁序策略**（业界标准做法，MySQL 官方文档推荐）：

```
规则：多账户操作时，按 (user_id, currency_code, wallet_type) 字典序升序加锁

示例 — 兑换 VND → BSB：
  ① 先锁 (U001, BSB, center)    — B < V，BSB 先
  ② 再锁 (U001, VND, center)    — 然后 VND

示例 — 消费优先级扣款（中心+奖励）：
  ① 先锁 (U001, BSB, center)    — center=1
  ② 再锁 (U001, BSB, reward)    — reward=2

所有操作遵循同一个排序规则，消灭死锁的可能。
```

### 2.5 自身情况最佳实践总结

```
┌─────────────────────────────────────────────────────────────┐
│  并发控制最佳实践（结合自身技术栈）                               │
│                                                             │
│  核心：悲观锁（FOR UPDATE）                                   │
│  辅助：version 字段（乐观锁降级备用）                           │
│  死锁防护：固定锁序（字典序）                                   │
│  锁粒度：单行（user_id + currency_code + wallet_type）        │
│  持锁范围：最小化（读→校验→写→流水→COMMIT）                     │
│  不做：不用 Redis 分布式锁（单库场景不需要）                     │
│  不做：不用 Actor 模型（Go 标准库已够用，无需引入新框架）          │
│                                                             │
```

## 三、挑战②：数据一致性保障 🔴

### 3.1 挑战描述

钱包中最不可妥协的要求：**余额和流水必须 100% 一致**。

不一致的后果：
- 余额更新成功但流水没写入 → 审计断链，无法追踪资金去向
- 流水写入了但余额没更新 → 用户看到的余额和实际不符
- 充值回调到达但入账失败 → 用户已付款但余额未增加
- 提现审核通过但扣冻结失败 → 钱已出但余额未扣，平台资损

### 3.2 一致性分层

不是所有数据都需要最强一致性。过度追求强一致会牺牲性能和可用性：

| 一致性级别 | 数据范围 | 保障要求 | 出问题的代价 |
|-----------|---------|---------|------------|
| **强一致** | 余额 ↔ 流水 | 同一 DB 事务，要么都成功要么都回滚 | 资金数据错误，直接资损 |
| **最终一致** | 钱包 ↔ 财务（充值/提现跨服务） | 幂等 + 重试 + 补偿 | 短暂延迟但最终正确 |
| **弱一致** | Redis 缓存（币种配置/汇率） | TTL + 主动刷新 | 短暂读到旧数据，不影响资金安全 |

### 3.3 业界方案对比

| 方案 | 原理 | 代表 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|------|---------|
| **① 本地事务（单库 ACID）** | 余额+流水在同一数据库事务中操作 | 绝大多数单体/单库系统 | 最强一致性保障，最简单可靠 | 局限于单库范围 | 单服务内的数据操作 |
| **② 分布式事务（2PC/XA）** | 两阶段提交，协调多个数据源 | 传统银行核心系统 | 跨库强一致 | 性能差、可用性低、协调器单点 | 跨数据库强一致场景 |
| **③ TCC（Try-Confirm-Cancel）** | 业务层三阶段：预留→确认→取消 | 蚂蚁金服、美团支付 | 灵活性高，业务可控 | 需要手工编写三个阶段代码 | 跨服务资源预留场景 |
| **④ 本地消息表 + MQ** | 业务操作和消息写入同一本地事务，再由 MQ 传递 | 美团外卖、B站 | 可靠性高，解耦好 | 实现复杂，需要消息表+定时补偿 | 跨服务异步协调 |
| **⑤ 幂等 + 重试（最终一致）** | 所有操作幂等化，失败后安全重试直到成功 | Stripe、Airbnb | 实现简单，容错性好 | 需要等待，不是即时一致 | 跨服务调用 |
| **⑥ 定时对账（兜底校验）** | 定期比对多方数据，发现不一致人工介入 | 所有金融系统的标配 | 最后一道防线 | 发现时已是事后 | 作为补充，不能替代主方案 |

### 3.4 推荐方案（分层组合）

**服务内（余额↔流水）：① 本地事务**

这是最可靠的方案。余额更新和流水写入在同一个 GORM Transaction 中，TiDB 保证 ACID。没有任何理由在单库场景引入分布式事务。

**跨服务（钱包↔财务）：⑤ 幂等 + 重试 + ③ TCC（提现场景）**

- 充值链路：财务收到回调 → 调钱包 CreditWallet RPC → 幂等保证不重复入账 → 失败重试
- 提现链路：TCC 模式 — Freeze（预留）→ DeductFrozen（确认）/ Unfreeze（取消）
- 人工修正：同充值链路，审核通过后调 CreditWallet/DebitWallet RPC

**兜底（所有场景）：⑥ 定时对账**

每日凌晨执行余额 vs 流水核对，发现差异告警 + 人工介入。

**可选方案：④ 本地消息表**

如果后续充值回调链路的可靠性不够（比如 RPC 频繁超时），可以引入本地消息表模式。但初期 `幂等 + 重试` 已足够，不需要增加复杂度。

### 3.5 自身情况最佳实践总结

```
┌─────────────────────────────────────────────────────────────┐
│  数据一致性最佳实践                                            │
│                                                             │
│  第一层：本地事务（余额 + 流水同一 GORM Transaction）           │
│  第二层：幂等 + 重试（跨服务 RPC 调用）                        │
│  第三层：TCC 三步模式（提现冻结→扣除/解冻）                     │
│  第四层：定时对账（每日凌晨余额 vs 流水核对）                   │
│                                                             │
│  不做：不用 2PC/XA（性能差、TiDB 单库不需要）                  │
│  不做：初期不引入本地消息表（幂等+重试够用）                     │
│  不做：不用 Saga（提现三步已是简化版 TCC，不需要完整 Saga）      │
```

## 四、挑战③：幂等与重复操作防护 🔴

### 4.1 挑战描述

分布式环境下，重复请求无处不在：

- **网络超时重试**：RPC 调用超时，调用方重试，但实际第一次已经成功
- **回调重复推送**：三方支付通道的回调可能推送多次
- **前端重复点击**：用户快速点击两次"充值"按钮
- **消息队列重投**：NATS 消息消费失败重投，同一笔充值回调被消费两次
- **运维误操作**：手动执行脚本补偿时可能执行两次

**不处理幂等的后果**：同一笔充值到账两次（用户多拿了钱，平台亏钱）；同一笔提现被冻结两次（用户余额被多冻了）。

### 4.2 业界方案对比

| 方案 | 原理 | 代表 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|------|---------|
| **① 客户端幂等键（Idempotency Key）** | 客户端生成唯一 key，服务端记录已处理的 key | Stripe API | 标准化，可扩展到所有接口 | 依赖客户端正确传递 key | 开放 API |
| **② 数据库唯一约束** | INSERT 时利用唯一索引做冲突检测 | 所有需要去重的系统 | 100% 可靠，DB 层面保证 | 有网络往返延迟 | 最终兜底 |
| **③ Redis SetNX 快速拦截** | 利用 Redis 的原子 SET NX 操作做前置检测 | 大量互联网系统 | O(1) 极快，毫秒级拦截 | Redis 故障时失效 | 高频快速拦截 |
| **④ 幂等表（专门的去重表）** | 维护独立的幂等记录表 | 美团支付 | 独立管理，不侵入业务表 | 多一张表，多一次写入 | 需要记录幂等元数据的场景 |
| **⑤ 状态机约束** | 通过订单状态转换规则防止重复操作（如已完成不可再次完成） | 所有订单系统 | 业务语义清晰 | 仅适用于有状态流转的场景 | 订单类操作 |

### 4.3 推荐方案

**主流方案（推荐）：③ Redis SetNX + ② DB 唯一约束 双层防护**

```
                    请求到达
                       │
                       ▼
            ┌──────────────────────┐
            │  第一层：Redis SetNX   │  ← 快速拦截（O(1)，毫秒级）
            │  key = "wal:idem:    │
            │  {orderNo}:{op}:{wt}"│
            │  TTL = 24h           │
            │                      │
            │  命中 → 返回成功       │  99%+ 的重复请求在此拦截
            └──────────┬───────────┘
                       │ 未命中
                       ▼
            ┌──────────────────────┐
            │  第二层：DB 唯一索引   │  ← 最终兜底（事务内）
            │  wallet_transaction  │
            │  uk_order_op_wallet  │
            │  (order_no, op_type, │
            │   wallet_type)       │
            │                      │
            │  冲突 → 返回成功       │  Redis 失效时的兜底
            └──────────┬───────────┘
                       │ 插入成功
                       ▼
                   正常执行业务
```

为什么需要两层：
- **Redis 可能故障**：Redis 宕机或网络抖动时，第一层失效，DB 唯一约束兜底
- **DB 检查有延迟**：唯一约束需要 INSERT 尝试和网络往返，Redis 在前面快速拦截 99%+ 的重复请求
- 这是 **Stripe、Airbnb 均采用的标准双层策略**

**可选方案：① Idempotency Key Header**

如果未来开放 API 给第三方（如聚合平台），可以引入 Stripe 风格的 `Idempotency-Key` Header 标准。当前阶段只有内部 RPC 调用，orderNo 已作为天然幂等键，不需要额外 Header。

### 4.4 幂等键设计细节

```
幂等键三元组：(orderNo, opType, walletType)

为什么不是 orderNo 单字段？
  → 人工加款 A+16位订单号 可能同时操作中心和奖励两个钱包
  → 同一 orderNo 产生两条流水（walletType 不同）
  → 如果用 orderNo 单字段，第二条流水会被误判为重复

Redis 键格式：{wal:idem}:{orderNo}:{opType}:{walletType}
  示例：{wal:idem}:C20260227001234:credit:1

DB 唯一索引：UNIQUE INDEX uk_order_op_wallet (order_no, op_type, wallet_type)

TTL 策略：24h（覆盖最长的业务重试周期）

Redis 事务失败时的键清理：
  → 不主动删除 Redis 键
  → 重试时检查 DB 流水是否存在：
    · 存在 → 说明已成功，返回成功
    · 不存在 → 重置 Redis 键，重新执行
```

### 4.5 自身情况最佳实践总结

```
┌─────────────────────────────────────────────────────────────┐
│  幂等最佳实践                                                 │
│                                                             │
│  双层：Redis SetNX（快速拦截） + DB 唯一索引（最终兜底）       │
│  粒度：三元组 (orderNo, opType, walletType)                  │
│  TTL ：24 小时                                               │
│  辅助：订单状态机防止已完成订单二次处理                         │
│                                                             │
│  不做：不用独立幂等表（wallet_transaction 唯一索引已承担角色） │
│  不做：不在 Redis 键中包含 userId（orderNo 已全局唯一）       │
│                                                             │
│  业界依据：                                                   │
```

## 五、挑战④：资金安全与风控 🔴

### 5.1 挑战描述

钱包模块直接涉及用户的真金白银和平台资产，安全挑战涵盖多个层面：

- **超扣/超发**：余额扣成负数或凭空增加余额
- **提现欺诈**：恶意用户利用并发漏洞多次提现、使用虚假身份信息提现
- **薅羊毛**：通过反复充值领取奖金活动，兑换后立即提现
- **数据篡改**：直接修改数据库余额或流水记录
- **操作不可追溯**：B 端运营人员的加减款操作无法审计

### 5.2 业界方案对比

| 安全维度 | 方案 | 代表 | 说明 |
|---------|------|------|------|
| **余额保护** | DB 行锁 + 余额 >= 0 校验 | 所有资金系统 | 最基本的保障，在锁内校验 |
| **不可变账本** | 流水表 INSERT Only | Stripe Ledger、区块链 | 流水只追加不修改，任何修正通过反向流水实现 |
| **TCC 冻结模式** | Try-Confirm-Cancel | 蚂蚁金服、美团 | 提现不直接扣款，通过冻结预留资金 |
| **KYC 认证** | 实名制验证 | 所有持牌支付平台 | 法币提现必须完成实名认证 |
| **稽核/打码量** | 流水倍数门槛 | 博彩/游戏行业通用 | 奖金需完成 N 倍流水才可提现，防薅羊毛 |
| **限额控制** | 单笔/每日/累计限额 | 所有支付平台 | 限制单次和累计交易金额 |
| **操作审计** | 全量操作日志 + 操作人记录 | 金融行业合规要求 | 每一笔 B 端操作可追溯到人 |
| **对账验证** | 定时余额 vs 流水核对 | Stripe DQ Platform、支付宝 | 最后一道防线，发现不一致即告警 |

### 5.3 推荐方案（多层防护体系）

钱包安全不能依赖单一方案，必须构建**纵深防御**体系：

```
┌──────────────────────────────────────────────────────────────┐
│                    资金安全四层防护体系                           │
│                                                              │
│  第一层：接入安全（拦截非法请求）                                 │
│  ├── gate-font: Token 鉴权 + IP 黑名单                        │
│  ├── gate-back: Token + IP 白名单 + RBAC 权限控制              │
│  ├── RPC: Kitex 服务间通过 ETCD 服务发现（非公开端口）           │
│  └── 速率限制：单用户单接口 QPS 上限                             │
│                                                              │
│  第二层：业务安全（防止逻辑漏洞）                                 │
│  ├── 幂等控制：Redis + DB 双层防重复                             │
│  ├── 余额校验：FOR UPDATE 锁内校验 available >= amount          │
│  ├── 冻结机制：提现走 Freeze→Audit→DeductFrozen 三步           │
│  ├── KYC 校验：法币提现必须实名认证                              │
│  ├── 姓名一致性：提现账户持有人 = KYC 姓名                       │
│  ├── 提现路由：法币充值路径约束提现路径                           │
│  ├── 重复转账拦截：同金额同币种 3 笔上限                         │
│  ├── 稽核流水：奖金 N 倍打码量达标才能提现                       │
│  └── 每日限额：Redis 原子计数器                                 │
│                                                              │
│  第三层：数据安全（防止数据被篡改）                               │
│  ├── 不可变流水：wallet_transaction 只 INSERT                   │
│  ├── 软删除：delete_at 标记，不物理删除                          │
│  ├── 审计字段：create_by/update_by + BLog 操作日志              │
│  ├── 汇率日志：每次变更生成不可篡改记录                           │
│  └── TraceId：全链路追踪，每笔操作可追溯                         │
│                                                              │
│  第四层：对账安全（最后防线）                                     │
│  ├── 定时对账：余额 vs 流水 SUM                                 │
│  ├── 流水连续性：after_amt[N] == before_amt[N+1]               │
│  └── 差异告警 + 人工介入（不自动修正）                            │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

**可选增强方案（v2 考虑）**：

| 方案 | 说明 | 何时引入 |
|------|------|---------|
| 实时风控引擎 | 基于规则/ML 模型的实时交易风险评估 | 交易量达到一定规模后 |
| 异常行为检测 | 短时间大量小额充提、异常时段交易等模式识别 | 运营发现薅羊毛模式后 |
| 操作双人复核 | B 端大额操作需要两个人审批 | 管理层要求加强控制时 |
| 数据库审计日志 | TiDB 层面的 binlog 审计 | 合规审查要求时 |

### 5.4 出款失败保守策略

**这是一个关键的安全决策**：提现出款失败后是否自动解冻？

```
场景：用户提现 1000 → 冻结 1000 → 通道出款 → 通道返回"失败"

选项 A（激进）：自动解冻，用户余额恢复
  风险：通道实际已出款但回调错误（网络抖动），用户余额恢复后再次提现 = 平台双倍损失

选项 B（保守，推荐）：保持冻结，人工处理
  逻辑：出款失败的原因可能是真失败也可能是假失败（通道已出款但回调异常）
  处理：人工确认通道侧真实状态后决定解冻或扣冻结
  业界：Stripe、支付宝、微信支付均采用保守策略
```

**我们选择 B（保守策略）**——这是产品需求明确要求，也是业界标准做法。

```
┌─────────────────────────────────────────────────────────────┐
│  资金安全最佳实践                                              │
│                                                             │
│  核心原则：纵深防御，不依赖单一方案                              │
│  v1 必做：四层防护体系全部落地                                  │
│  出款失败：保持冻结不自动解冻，人工确认后处理                     │
│  数据修正：通过反向流水实现，永不直接修改余额或流水               │
│  审计要求：每笔操作必须有 traceId + userId + 完整的 before/after │
│                                                             │
│  业界依据：                                                   │
│  · Stripe 的"不可变账本"原则（Append-Only Ledger）            │
```

## 六、挑战⑤：多币种与汇率管理 🟡

### 6.1 挑战描述

多币种带来三个核心难题：

- **精度丢失**：不同币种精度不同（VND 无小数、THB 2 位、USDT 8 位），计算和存储过程中可能丢失精度
- **汇率波动**：从用户点击"确认"到系统处理完成有时间差，汇率可能已变化，导致"所见非所得"
- **多精度共存**：BSB 2 位、法币各不同、加密货币 8 位，同一张表需要容纳不同精度

### 6.2 业界方案对比

**精度问题**：

| 方案 | 原理 | 代表 | 优点 | 缺点 |
|------|------|------|------|------|
| **① DECIMAL 类型** | 数据库原生定点数，无精度丢失 | Stripe、Modern Treasury、银行系统 | 精确计算，直观可读 | 占用空间略大于整数 |
| **② BIGINT 最小单位** | 将金额转为最小单位整数存储（如分） | 部分游戏系统、早期支付系统 | 整数运算快 | 不同币种最小单位不同，转换复杂 |
| **③ 字符串存储** | 金额以字符串形式存储，需要时转换计算 | 少数系统 | 不丢精度 | 计算需转换，性能差，DB 无法做数值运算 |

**汇率波动**：

| 方案 | 原理 | 代表 | 优点 | 缺点 |
|------|------|------|------|------|
| **① Quote-Lock（汇率快照锁定）** | 交易时锁定汇率写入订单，后续用快照计算 | Stripe FX Quote、Airwallex、Wise | "所见即所得"，消灭汇率变动风险 | 需要在订单中存储汇率快照字段 |
| **② 实时汇率计算** | 每次计算都取最新汇率 | 简单系统 | 实现简单 | 前后金额可能不一致，引起纠纷 |
| **③ 汇率有效期窗口** | 汇率带有效期，过期后需重新获取 | 部分外汇交易平台 | 平衡实时性和确定性 | 实现复杂，需维护有效期状态 |

### 6.3 推荐方案

**精度：① DECIMAL(20,4) / DECIMAL(20,8)**

- 金额字段：`DECIMAL(20,4)` — 覆盖所有法币 + BSB（最多 2 位小数，预留到 4 位安全）
- 汇率字段：`DECIMAL(20,8)` — 覆盖加密货币精度（8 位小数）
- 百分比字段：`DECIMAL(10,4)` — 表示 xx.xx%
- **业务层统一使用 Go `shopspring/decimal` 库**做高精度运算，避免 float64 精度丢失

**汇率：① Quote-Lock 快照锁定**

```
用户发起涉及汇率的操作（兑换/充值/提现）：

  1. 读取当前汇率（从 Redis 缓存或 DB）
  2. 汇率快照写入订单记录：
     · deposit_order.rate_snapshot = 当时的入款汇率
     · withdraw_order.rate_snapshot = 当时的出款汇率
     · exchange_order.rate_snapshot = 当时的平台汇率
  3. 后续所有金额计算使用快照汇率
  4. 对账/统计也使用快照汇率

"所见即所得"— 用户确认时看到的金额 = 实际到账金额
```

**可选方案：③ 汇率有效期窗口**

如果后续产品需要更精细的汇率管理（比如大额兑换需要实时汇率），可以引入汇率有效期窗口。当前阶段 Quote-Lock 已够用。

### 6.4 三层汇率体系

```
实时汇率 ──→ 平台汇率 ──→ 入款/出款汇率

第一层：实时汇率（real_time_rate）
  · 来源：3 个三方 API 取平均值
  · 更新频率：可配置（如每 2 分钟）
  · 作用：反映真实市场价

第二层：平台汇率（platform_rate）
  · 触发：|实时 - 平台| / 实时 × 100% >= 阈值
  · 作用：平台内部核算基准，不频繁变动
  · 每次更新必须写 exchange_rate_log

第三层：入款/出款汇率
  · 入款 = 平台汇率 × (1 + 浮动%)
  · 出款 = 平台汇率 × (1 - 浮动%)
  · 浮动差价 = 平台汇兑收益

API 容错：
  · 3 个 API 任一失败 → 取可用的平均值
  · 全部失败 → 保持旧值不更新 + 告警
  · 单个 API 偏差过大 → 剔除异常值
```

### 6.5 自身情况最佳实践总结

```
┌─────────────────────────────────────────────────────────────┐
│  多币种与汇率最佳实践                                          │
│                                                             │
│  精度：DECIMAL(20,4) 金额 / DECIMAL(20,8) 汇率              │
│  运算：Go shopspring/decimal 库（禁用 float64 做金额计算）    │
│  汇率：Quote-Lock 快照锁定，订单中保存汇率快照                  │
│  体系：三层汇率（实时→平台→入款/出款）                          │
│  容错：3 API 取平均值，全失败保持旧值+告警                      │
│  BSB：锚定 USDT（10 BSB = 1 USDT），不从三方 API 获取         │
│                                                             │
│  不做：不用 BIGINT 最小单位存储（多币种精度不同，转换太复杂）    │
│  不做：不在运算中使用 float64（IEEE 754 浮点精度问题）          │
│                                                             │
│  业界依据：                                                   │
│  · Stripe/Wise/Airwallex 均使用 DECIMAL + Quote-Lock        │
│  · Go shopspring/decimal 是金融 Go 项目的事实标准            │
│  · 三方 API 多源聚合是汇率获取的业界标准做法                   │
└─────────────────────────────────────────────────────────────┘
```

## 七、挑战⑥：高性能与容量规划 🟢

### 7.1 挑战描述

钱包模块的性能挑战主要集中在三个方面：

- **热点账户**：某些高频用户（如大额玩家、主播）的账户被频繁读写，行锁争用
- **流水表增长**：wallet_transaction 是只追加的表，数据量持续增长，查询和写入性能逐渐退化
- **高频读取**：币种配置/汇率数据被所有接口反复读取，成为性能瓶颈

### 7.2 业界方案对比

**热点账户**：

| 方案 | 原理 | 代表 | 优点 | 缺点 |
|------|------|------|------|------|
| **① 行锁 + 最小持锁时间** | 事务内只做必要操作，快速释放锁 | 大量传统金融 | 简单可靠 | 单账户 TPS 有上限 |
| **② 异步合并记账** | 多笔小额操作合并为一笔批量操作 | 支付宝双十一、微信红包 | 极高吞吐 | 实时性降低，实现复杂 |
| **③ 影子余额（子账户拆分）** | 一个逻辑账户拆成 N 个物理子账户，分散锁争用 | 蚂蚁花呗 | 突破单行锁瓶颈 | 查询需聚合，一致性更复杂 |
| **④ 缓存余额 + 异步落库** | Redis 存余额，异步写 DB | 部分游戏币系统 | 极致性能 | 数据丢失风险，不适合真金白银 |

**流水表增长**：

| 方案 | 原理 | 代表 | 优点 | 缺点 |
|------|------|------|------|------|
| **① 单表 + 索引优化** | 合理索引，避免全表扫描 | 初期阶段的大部分系统 | 简单 | 数据量超限后性能下降 |
| **② TiDB 自动分片** | TiDB 底层自动将大表分散到多个 Region | TiDB 原生能力 | 透明分片，应用层无感 | 需要合理选择分片键 |
| **③ 应用层分表** | 按时间/用户 ID 拆分为多张物理表 | ShardingSphere、手动分表 | 控制单表大小 | 跨表查询复杂 |
| **④ 冷热分离** | 近期数据热表，历史数据归档冷表 | 大量运营系统 | 热表保持小而快 | 需要归档任务和冷数据查询方案 |

### 7.3 推荐方案

**热点账户：① 行锁 + 最小持锁时间（初期）**

当前业务规模下，单个用户的并发操作频率有限（充值/消费/提现不会像双十一秒杀那样）。悲观锁 + 最小持锁时间足够支撑。

```
最小持锁策略：
  1. 所有非 DB 操作（参数校验、Redis 幂等检查）在事务外完成
  2. 事务内只做：锁行 → 读余额 → 校验 → 写余额 → 写流水 → COMMIT
  3. 不在事务内做 RPC 调用、日志 IO、缓存操作
  4. 目标：单次事务持锁时间 < 5ms
```

**可选增强方案（v2，当 TPS 达到瓶颈时）**：

| 方案 | 触发条件 | 说明 |
|------|---------|------|
| ② 异步合并记账 | 单账户 TPS > 100/s | 对高频消费场景（如游戏投注），可以攒一批再记 |
| ③ 影子余额 | 单账户锁等待 > 10ms | 将热点账户拆成 N 个子账户分散锁争用 |

**流水表：① 单表 + ② TiDB 自动分片**

TiDB 的底层存储（TiKV）天然按 Region 分片。只要选好主键策略（Snowflake ID 避免写热点），初期不需要手动分表。

```
初期（< 1 亿条）：
  · 单表 + 合理索引
  · Snowflake ID 主键（分散写入到不同 Region，避免自增 ID 的写热点）
  · 索引：uk_order_op_wallet + idx_user_currency_type

中期（1-10 亿条）：
  · TiDB 自动分片已生效，性能不受影响
  · 可选冷热分离：90 天以上的流水归档到冷表

远期（> 10 亿条）：
  · TiDB 按 Region 自动扩展
  · 或引入应用层分表（按 user_id 分片）
```

### 7.4 高频读取优化

```
币种配置/汇率数据读取优化：

┌─────────────────────────────────────────────────────────┐
│  读取链路：业务层 → Redis 缓存 → DB（缓存未命中时回源）    │
│                                                         │
│  币种列表：{wal:cur}:list，TTL 5min，写时主动刷新         │
│  单个币种：{wal:cur}:{code}，TTL 5min，写时主动刷新      │
│  汇率数据：{wal:rate}:{code}，定时任务更新后刷新           │
│  基准币种：{wal:base}，无过期（设一次不变）                │
│                                                         │
│  缓存命中率目标：99%+                                    │
│  原因：币种/汇率数据写入频率极低（配置变更），读取频率极高   │
│                                                         │
│  不缓存余额：                                            │
│  · 余额是高频写的数据，缓存一致性风险大                    │
│  · 单行主键查询 TiDB 性能已足够                           │
│  · 缓存不一致 = 用户看到的余额和实际不符 = 投诉           │
└─────────────────────────────────────────────────────────┘
```

### 7.5 自身情况最佳实践总结

```
┌─────────────────────────────────────────────────────────────┐
│  高性能最佳实践                                                │
│                                                             │
│  热点账户：行锁 + 最小持锁时间（事务内 < 5ms）                │
│  流水表：Snowflake ID 主键 + TiDB 自动分片                   │
│  高频读：Redis 缓存币种/汇率（写时刷新 + TTL 兜底）           │
│  连接池：TiDB MaxOpen=20, MaxIdle=10（现有工程已配置）        │
│  预编译：GORM PrepareStmt=true（现有工程已配置）              │
│  RPC 复用：Kitex Mux 模式（现有工程已配置）                   │
│                                                             │
│  不做（避免过度设计）：                                        │
│  · 不缓存余额到 Redis                                       │
│  · 不做流水表分库分表（TiDB 自动分片够用）                    │
│  · 不做读写分离/CQRS（余额操作是强一致读写混合）              │
│  · 不做异步写流水（必须和余额同事务）                         │
│  · 不做多级缓存（Redis 单层够用）                             │
│                                                             │
│  业界依据：                                                   │
│  · TiDB 官方推荐 Snowflake ID 避免写热点                    │
│  · Stripe 初期也是单表 + 良好索引，规模增长后才分片          │
│  · "过早优化是万恶之源" — Donald Knuth                      │
└─────────────────────────────────────────────────────────────┘
```

## 八、挑战⑦：跨服务协调与容错 🟡

### 8.1 挑战描述

钱包不是一个孤立服务，它与财务、KYC、用户等模块有密切 RPC 交互。跨服务调用带来新的挑战：

- **RPC 超时**：财务模块调钱包 CreditWallet 超时，不知道钱包是否已入账
- **服务不可用**：KYC 服务临时不可用，提现校验无法完成
- **回调丢失**：三方支付通道的回调请求在网络传输中丢失
- **调用风暴**：大促期间大量充值回调同时到达，RPC 瞬间压力剧增

### 8.2 业界方案对比

| 方案 | 原理 | 代表 | 优点 | 缺点 |
|------|------|------|------|------|
| **① 幂等 + 安全重试** | 所有 RPC 幂等化，超时直接重试 | Stripe、AWS | 简单有效 | 需要所有接口支持幂等 |
| **② 熔断器（Circuit Breaker）** | 下游故障时快速失败，避免级联故障 | Hystrix、Sentinel、Kitex 内置 | 防止级联雪崩 | 需要合理配置阈值 |
| **③ 超时控制 + 重试策略** | 设置合理超时，指数退避重试 | gRPC/Kitex 内置 | 工业标准做法 | 重试次数和间隔需调优 |
| **④ 异步解耦（MQ）** | 用消息队列解耦同步调用 | 美团外卖、B 站 | 削峰填谷 | 增加复杂度，延迟增加 |
| **⑤ 降级策略** | 非核心依赖不可用时，降级处理 | Netflix、阿里 | 保证核心流程可用 | 降级后功能受限 |
| **⑥ 补偿任务（定时扫描）** | 定时扫描未完成的订单，重新处理 | 所有支付系统 | 兜底机制 | 有延迟 |

### 8.3 推荐方案

**主流方案（推荐）：① 幂等重试 + ③ 超时策略 + ⑤ 降级策略**

```
RPC 调用策略（每个跨服务调用）：

┌──────────────────────────────────────────────────────┐
│  配置项             │  推荐值               │ 说明     │
├──────────────────────────────────────────────────────┤
│  RPC 超时            │  3s（核心）/ 5s（非核心）│ 快速失败 │
│  重试次数            │  2 次                  │ 总共 3 次 │
│  重试策略            │  指数退避（100ms→200ms）│ 避免重试风暴│
│  幂等保证            │  每个 RPC 必须幂等      │ 安全重试  │
└──────────────────────────────────────────────────────┘

降级策略：

  · KYC 服务不可用 → 提现接口返回"服务暂时不可用，请稍后重试"
    （不能跳过 KYC 校验，安全不可降级）

  · 财务模块不可用 → 充值方式列表返回缓存数据或友好提示
    （充值下单暂不可用，但钱包余额查询、兑换等不受影响）

  · 汇率 API 全部不可用 → 使用最近一次成功获取的汇率 + 告警
    （平台汇率有缓存，短期不影响业务）
```

**可选方案（v2）：② 熔断器**

Kitex 内置了熔断能力。当财务模块故障率超过阈值时自动熔断，快速失败而不是每个请求都等超时。初期请求量不大时，简单的超时+重试已够用，熔断器可在流量增长后启用。

**可选方案（兜底）：⑥ 补偿任务**

定时扫描处于中间状态的订单（如"充值中"超过 30 分钟），触发补偿逻辑（如重新查询财务模块确认状态）。

### 8.4 双向依赖解耦

```
ser-wallet 和 ser-finance 的双向 RPC 调用是架构上必须处理的问题：

                ┌──────────────────┐
                │    IDL 合同层      │
                │ common/idl/       │
                │ ├── ser-wallet/   │  ← 钱包暴露的 RPC 合同
                │ └── ser-finance/  │  ← 财务暴露的 RPC 合同
                └────────┬─────────┘
                         │
          ┌──────────────┼──────────────┐
          ▼                             ▼
    ser-wallet                    ser-finance
    只依赖 IDL stub               只依赖 IDL stub
    (kitex_gen/ser_finance)       (kitex_gen/ser_wallet)
    通过 ETCD 发现对方             通过 ETCD 发现对方

解耦结果：两个服务在代码层面零依赖，仅通过 IDL + ETCD 交互
不会形成同步环路：
  · 充值：wallet→finance(同步) ... finance→wallet(异步回调触发)
  · 提现：wallet(内部冻结) ... finance→wallet(审核结果触发)
```

### 8.5 自身情况最佳实践总结

```
┌─────────────────────────────────────────────────────────────┐
│  跨服务协调最佳实践                                            │
│                                                             │
│  基础：所有 RPC 幂等化 + 超时 3-5s + 重试 2 次（指数退避）    │
│  解耦：IDL 合同 + ETCD 服务发现，零代码依赖                    │
│  降级：非核心依赖不可用时返回友好提示，安全相关不可降级          │
│  兜底：补偿任务定时扫描中间态订单                               │
│                                                             │
│  v2 考虑：Kitex 内置熔断器                                    │
│                                                             │
│  不做：不用 MQ 解耦核心资金流（余额操作必须同步 DB 事务）      │
│  不做：不自建服务注册发现（ETCD 已有）                         │
│                                                             │
│  业界依据：                                                   │
│  · Stripe 幂等 + 超时重试 处理所有跨服务调用                  │
│  · Netflix Hystrix 熔断器是微服务容错的工业标准               │
│  · CloudWeGo Kitex 内置超时/重试/熔断能力                    │
└─────────────────────────────────────────────────────────────┘
```

## 九、挑战⑧：分布式事务 🔴

### 9.1 挑战描述

钱包中有三个场景需要跨越普通本地事务的边界：

1. **提现冻结-扣款**：用户提现 → 冻结余额 → 财务审核 → 出款成功 → 扣冻结 / 审核驳回 → 解冻。中间有时间间隔（分钟到小时），不能用单个数据库事务覆盖
2. **兑换跨币种**：扣法币 + 加 BSB，涉及两个账户的余额变动，必须原子完成
3. **充值入账**：财务模块确认支付成功 → 调钱包 CreditWallet → 钱包入账。跨越两个服务的边界

### 9.2 业界方案对比

| 方案 | 原理 | 代表 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|------|---------|
| **① TCC（Try-Confirm-Cancel）** | 业务层三阶段控制 | 蚂蚁金服 | 灵活可控，无需中间件 | 需要三段代码 | 资源预留+确认/取消 |
| **② Saga** | 长事务拆分为多个本地事务 + 补偿 | Uber Cadence、Netflix | 适合长流程 | 补偿逻辑复杂 | 多步骤长流程 |
| **③ 2PC/XA** | 数据库层两阶段提交 | 传统银行核心 | 强一致 | 性能差、可用性低 | 跨库强一致 |
| **④ 本地事务（同库两行）** | 两个账户在同一数据库，一个事务搞定 | 大量单库系统 | 最简单、最可靠 | 局限于同库 | 同库跨表操作 |
| **⑤ 幂等 + 重试** | 每步操作幂等化，失败重试 | Stripe | 简单有效 | 有短暂不一致窗口 | 跨服务最终一致 |

### 9.3 推荐方案（按场景分治）

**场景 1 — 提现冻结/扣款：① TCC**

```
TCC 在提现中的映射：

  Try    = FreezeBalance      用户提交提现时，冻结可用余额
  Confirm = DeductFrozenAmount  出款成功后，扣除冻结金额
  Cancel  = UnfreezeBalance    审核驳回时，释放冻结回到可用

为什么是 TCC 而不是 Saga：
  · 冻结操作天然就是"资源预留"，完美映射 TCC 的 Try 阶段
  · 不需要"补偿回滚"——冻结状态本身就是可逆的
  · 每个阶段都是独立幂等的 RPC，不需要编排引擎

每步操作的幂等键：
  · FreezeBalance:       {wal:idem}:{orderNo}:freeze:{walletType}
  · DeductFrozenAmount:  {wal:idem}:{orderNo}:deduct_frozen:{walletType}
  · UnfreezeBalance:     {wal:idem}:{orderNo}:unfreeze:{walletType}
```

**场景 2 — 兑换跨币种：④ 本地事务**

兑换操作虽然涉及两个币种的账户，但这两个账户都在 **同一个 TiDB 数据库** 中。不需要分布式事务，一个本地事务就能搞定。

```
BEGIN TRANSACTION
  1. FOR UPDATE 锁定法币中心账户（按字典序先锁）
  2. FOR UPDATE 锁定 BSB 中心账户
  3. 校验法币余额 >= 兑换金额
  4. 法币可用余额 -= 兑换金额
  5. BSB 可用余额 += 兑换所得
  6. BSB 奖励余额 += 赠送金额（如有）
  7. 写入法币出账流水
  8. 写入 BSB 入账流水
  9. 写入 BSB 赠送流水（如有）
COMMIT

原子性保证：以上 9 步要么全部成功，要么全部回滚。
```

**场景 3 — 充值入账：⑤ 幂等 + 重试**

```
ser-finance                          ser-wallet
    │                                    │
收到三方回调                              │
支付成功     │                            │
    │── CreditWallet RPC ──────────────►│
    │   (orderNo=C20260227xxx)           │ 幂等检查
    │                                    │ DB事务(加余额+写流水)
    │◄── 成功 ────────────────────────│
    │                                    │
    │  如果 RPC 超时：                    │
    │── CreditWallet RPC（重试）────────►│
    │   (相同 orderNo)                   │ 幂等检查 → 已处理
    │◄── 成功（幂等返回）─────────────│

核心保障：幂等 + 重试 = 最终一致
```

### 9.4 自身情况最佳实践总结

```
┌─────────────────────────────────────────────────────────────┐
│  分布式事务最佳实践                                            │
│                                                             │
│  提现场景：TCC 三步模式（Freeze → DeductFrozen / Unfreeze）  │
│  兑换场景：本地事务（同库两行操作，无需分布式事务）              │
│  充值场景：幂等 + 重试（跨服务最终一致）                       │
│  消费优先级：本地事务（同库两行锁定，按字典序）                 │
│                                                             │
│  不做：不用 2PC/XA（性能差，TiDB 单库不需要）                 │
│  不做：不用 Saga 引擎（提现三步已是简化 TCC，不需要编排器）    │
│  不做：不引入 Seata 等分布式事务中间件（增加运维复杂度）       │
│                                                             │
│  业界依据：                                                   │
│  · 蚂蚁金服 TCC 广泛用于冻结/扣款场景                        │
│  · Stripe 充值入账用 幂等 + 重试                              │
│  · 同库操作用本地事务是工程共识                                │
└─────────────────────────────────────────────────────────────┘
```

## 十、挑战⑨：可观测性与故障定位 🟡

### 10.1 挑战描述

钱包涉及真金白银，一旦出现问题需要**快速定位**。但分布式环境下的故障排查面临：

- **链路穿透**：一笔充值从用户点击到入账，经过 gate-font → ser-wallet → ser-finance → 三方通道 → ser-finance → ser-wallet，跨 5 个节点，如何追踪？
- **异常告警**：余额不一致、对账差异、汇率 API 全部失败——需要实时发现而非事后排查
- **审计追溯**：运营人员给用户加了 10 万，需要追溯到谁批准的、谁操作的、什么时候操作的

### 10.2 业界方案对比

| 方案 | 代表 | 说明 |
|------|------|------|
| **分布式追踪（Tracing）** | OpenTelemetry、Jaeger、Zipkin | 全链路 TraceId 贯穿，可视化调用链 |

| **指标监控（Metrics）** | Prometheus + Grafana | 数值型指标采集，实时图表和告警 |
| **业务审计日志** | 自建审计表、BLog 框架 | 记录 who/what/when/result |

## 十一、挑战⑩：可扩展性与演进能力 🟢

### 11.1 挑战描述

当前的设计需要支持未来可能的变化：

- **新币种接入**：如果平台扩展到新市场（如菲律宾 PHP、马来西亚 MYR），系统是否能快速接入？
- **新钱包类型**：代理钱包、场馆钱包目前是预留，未来启用时能否无缝支持？
- **数据量增长**：用户增长后流水表数据量爆炸，系统是否能平滑扩展？
- **架构演进**：如果未来需要从"物化余额+不可变流水"升级到"完整复式记账"，改动有多大？

### 11.2 业界方案对比

| 方案 | 原理 | 代表 | 优点 | 缺点 |
|------|------|------|------|------|
| **① 配置驱动** | 新币种/新类型通过配置（DB/ETCD）添加，代码无需改动 | 所有成熟钱包系统 | 运营自主扩展 | 需要提前设计好配置化体系 |
| **② 枚举可扩展** | 使用字符串/宽范围整数而非严格枚举 | Stripe API | 新增类型不改代码 | 校验逻辑需额外处理 |
| **③ TiDB 弹性扩缩容** | 添加 TiKV 节点即可水平扩展 | TiDB 原生能力 | 应用层无感知 | 运维需要了解 TiDB |
| **④ 预留扩展字段** | 表中预留 JSON/Text 扩展字段 | 部分系统 | 灵活 | 不利于查询和索引 |

### 11.3 推荐方案

**新币种接入：① 配置驱动（已具备）**

```
新币种接入只需：
  1. currency_config 表中 INSERT 一条新币种配置（B 端运营操作）
  2. 配置汇率 API 来源、浮动比例、阈值
  3. 定时任务自动开始拉取该币种的汇率
  4. 用户端自动展示新币种

代码层面不需要改动——因为所有金额计算、精度处理都从 currency_config 读取配置
唯一可能需要改的：如果新币种有特殊规则（如 BSB 的不可外部充值），需要加业务判断
```

**新钱包类型：② 枚举可扩展**

```
wallet_type 使用 TINYINT，当前定义：
  1=中心, 2=奖励, 3=主播, 4=代理, 5=场馆

未来启用代理/场馆钱包：
  · 只需在代码枚举中解除注释
  · 账户延迟创建，首次使用时自动建
  · 余额操作的五个核心方法无需改动（它们不关心 walletType 的具体值）
```

**数据量增长：③ TiDB 弹性扩展**

```
TiDB 扩展路径：
  初期：3 TiKV 节点，单表足够
  中期：添加 TiKV 节点，TiDB 自动 Rebalance Region
  远期：如果需要进一步优化，可以用 TiDB 的 Placement Rules 做冷热分离

对应用层的影响：零。TiDB 的分片和扩展对上层透明。
```

**架构演进（物化余额 → 复式记账）：**

```
当前：物化余额 + 不可变流水

如果未来需要升级到复式记账：
  1. 新增"对手方账户"（平台收入账户、手续费账户、清算账户等）
  2. wallet_transaction 增加"对手方账户ID"字段
  3. 每笔操作产生两条流水（借方和贷方）
  4. 新增约束：所有分录之和 = 0

当前设计的演进友好性：
  · wallet_transaction 已有 direction 字段（收入/支出）→ 可映射为借/贷
  · before_amt / after_amt 审计链已建立 → 不需要重构
  · 主要改动：增加对手方账户体系 + 修改流水写入逻辑
  · 核心的余额操作方法接口不变，只是内部实现增加分录
```

## 十四、总结

### 14.1 核心认知

钱包模块的技术挑战**说难也不难**——不是要造火箭，每一个问题都有成熟的业界解决方案。关键在于：

1. **识别出真正致命的问题**（并发/一致性/幂等/安全），这些在 v1 必须完整解决
2. **选择匹配自身规模和技术栈的方案**，不是越高级越好（我们不需要蚂蚁金服的 OceanBase，TiDB + 本地事务已足够）
3. **纵深防御**，不依赖单一方案（幂等双层、安全四层、一致性分层）
4. **不过度设计**，初期用简单可靠的方案，在验证过后再按需升级

### 14.2 技术栈适配总结

| 挑战 | 我们的技术栈如何解决 |
|------|-------------------|
| 并发 | TiDB SELECT FOR UPDATE 行锁（完全兼容 MySQL） |
| 一致性 | GORM Transaction 本地事务（TiDB ACID） |
| 幂等 | Redis go-redis SetNX + TiDB 唯一索引 |
| 安全 | Hertz 中间件（鉴权/黑白名单）+ GORM Plugin（审计字段） |
| 汇率 | Redis 缓存 + ETCD 配置 + Go shopspring/decimal |
| 性能 | TiDB 自动分片 + Redis 缓存 + Kitex Mux |
| 跨服务 | Kitex RPC + ETCD 服务发现 + 幂等重试 |
| 分布式事务 | TCC（业务层实现）+ 本地事务 |
| 可观测 | Zap 结构化日志 + TraceId + BLog |
| 扩展 | TiDB 弹性扩展 + 配置驱动（currency_config） |

### 14.3 一句话总结

**钱包的技术挑战核心是"资金安全"——并发不超卖、数据不丢失、操作不重复、修改可追溯。我们选择的方案组合是：悲观锁保并发、本地事务保一致、双层幂等防重复、TCC 保提现、不可变流水保审计、定时对账兜底。所有方案都有业界成熟案例支撑，都能在我们的 CloudWeGo + TiDB + Redis 技术栈上直接落地，不需要引入额外中间件。**
