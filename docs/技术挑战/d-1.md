## 挑战 1：并发余额操作

### 问题描述

同一用户的同一个钱包账户可能被多个请求同时操作：用户在玩游戏时投注扣款，同时充值回调到达要入账，同时又可能有活动奖励要发放。如果不做并发控制：

```
经典超扣场景（Race Condition）：

  余额 = 100
  请求A（投注扣款50）：读到余额100，校验通过，100-50=50，写入50
  请求B（投注扣款80）：读到余额100，校验通过，100-80=20，写入20（覆盖A）
  结果：余额=20，但实际扣了50+80=130，超扣30

  或者反过来（丢失更新）：
  请求A写入50，请求B写入20，最终余额=20
  但如果B先写、A后写，最终余额=50
  两种结果都是错的。
```

**影响**：超扣 = 平台亏钱，少扣 = 用户白嫖。资金系统不允许任何一种。

### 方案对比

| 方案 | 原理 | 优点 | 缺点 | 适用场景 | 代表案例 |
|------|------|------|------|---------|---------|
| **悲观锁 SELECT FOR UPDATE** | 在事务内对行加排他锁，其他事务读写该行时阻塞 | 简单可靠，冲突时不需要重试 | 锁等待时间内其他请求阻塞 | 高冲突场景（同一用户频繁操作） | 支付宝/微信支付余额层 |
| 乐观锁 (version/CAS) | 读取时记录版本号，写入时校验版本号未变 | 无锁等待，并发读性能好 | 高冲突时大量重试，性能反而差 | 低冲突场景（商品库存秒杀不适合，余额也不适合） | 电商库存（低并发） |
| 数据库原子操作 UPDATE SET balance = balance - ? WHERE balance >= ? | 利用 DB 单条 SQL 的原子性 | 不需要显式事务和锁，简单 | 无法在同一事务中做复杂校验和关联写入 | 极简场景（无流水要求时可用） | 简单扣款场景 |
| Redis 分布式锁 + DB | 先用 Redis SetNX 抢锁，持锁期间操作 DB | 锁粒度可控，支持过期时间 | Redis 和 DB 之间无事务保证；锁超时后脑裂风险 | DB行锁不可用的场景 | 分布式限流/秒杀 |
| 内存队列串行化 | 同一用户的请求排队串行处理 | 无锁竞争，逻辑简单 | 单机瓶颈，扩展困难；进程重启丢请求 | 单体架构 | 早期支付系统 |

### 推荐方案：悲观锁 SELECT FOR UPDATE（主流）

**为什么选悲观锁**：
1. 钱包是**高冲突**场景——同一用户的余额可能被充值回调、投注扣款、活动奖励同时操作
2. 行锁持有时间极短（一次 SELECT + UPDATE + INSERT 在 TiDB 中通常 < 10ms），阻塞开销可忽略
3. 与 GORM Transaction 天然兼容，`query.Q.Transaction()` 内执行即可
4. 业界主流：**支付宝、微信支付、Stripe 的余额变更层均采用行锁方案**
5. TiDB 的行锁机制与 MySQL InnoDB 完全兼容

**实现要点**：
```
1. 锁的精确性：走 uk_user_currency_type 唯一索引精确锁单行
   → 不同用户、不同币种、不同钱包类型之间完全不阻塞
   → 锁粒度 = 一个用户的一个币种的一种钱包 = 一行

2. 事务边界最小化：事务内只做"读余额→校验→更新余额→写流水"
   → 不在事务内做 RPC 调用、不做复杂计算
   → 事务持有时间控制在个位毫秒

3. 死锁预防：兑换操作锁两行（源币种 + BSB），按 currency_id 升序加锁
   → 保证所有事务以相同顺序获取锁，杜绝死锁
```

**可选增强**：如果未来并发量极高（TPS > 5000 同一用户），可以在悲观锁前加一层 Redis 分布式锁做前置拦截，减少 DB 锁竞争。但当前阶段不需要。

## 挑战 2：操作幂等性

### 问题描述

网络环境下，同一个操作可能被重复执行：

```
场景1：三方支付回调重试
  三方通道因超时重试回调 → 财务模块收到两次 → 财务调钱包 CreditBalance 两次
  → 如果不做幂等 → 用户余额被加了两次 → 平台亏钱

场景2：用户网络抖动
  用户点击"兑换"按钮，网络超时 → 用户以为没成功，再点一次
  → 后端收到两次请求 → 两笔兑换都执行 → 用户多换了一份

场景3：服务重启中的消息重放
  RPC 调用在网络层重试 → 钱包服务收到重复请求
```

**影响**：重复入账 = 平台亏钱，重复扣款 = 用户投诉。

### 方案对比

| 方案 | 原理 | 优点 | 缺点 | 代表案例 |
|------|------|------|------|---------|
| **DB 唯一索引（推荐）** | 在流水表建唯一索引，重复写入触发冲突 | 与事务原子绑定，零额外依赖 | 依赖 DB，需要合理设计幂等键 | 支付宝/蚂蚁金服 |
| Redis SetNX 幂等表 | 用 Redis SetNX 判断请求是否已处理 | 性能高，不依赖 DB | Redis 与 DB 无事务关联，可能不一致 | 短信/邮件发送去重 |
| 幂等 Token（前端生成） | 前端生成唯一 Token 随请求发送，后端检查 Token 是否已使用 | 防止用户多次点击 | 需要前端配合；Token 存储和过期管理复杂 | 电商下单防重 |
| 状态机校验 | 操作前检查订单状态是否允许当前操作 | 业务语义清晰 | 单独不够，并发下仍可能穿透 | 辅助手段 |
| 请求去重表 | 独立的去重记录表，处理前先查是否已处理 | 通用性强 | 额外一张表的维护成本 | 通用幂等中间件 |

### 推荐方案：DB 唯一索引（主流）+ 状态机（辅助）

**为什么选 DB 唯一索引**：
1. 唯一索引冲突判断和余额变更在**同一事务**中——原子性天然保证
2. 如果用 Redis SetNX，Redis 成功但 DB 写入失败，该操作永远无法重试（假幂等）
3. 零额外依赖，性能等同于一次 INSERT
4. **蚂蚁金服资金核心的幂等方案就是 DB 唯一键**

**幂等键设计**：
```
transaction_log 表：
  UNIQUE KEY uk_ref_order_tx (ref_order_no, tx_type, wallet_type)

含义：同一笔业务单号 + 同一种交易类型 + 同一个钱包类型 → 只允许写入一次

为什么包含三个字段：
  ref_order_no 单独不够 → 兑换操作一笔单号会写多条流水（源币种扣减 + BSB入账 + 赠送入账）
  ref_order_no + tx_type 也不够 → BSB 入账可能分中心钱包和奖励钱包两行
  三字段联合 → 精确到每一条流水记录的粒度
```

**状态机辅助**：
```
订单状态更新用 WHERE status = 当前状态 条件：
  UPDATE deposit_order SET status = 3 WHERE order_no = ? AND status = 2
  → 如果返回 affected_rows = 0，说明状态已变（被其他请求处理了），直接返回
  → 双保险：即使唯一索引漏了，状态机也能兜住
```

## 挑战 3：数据一致性（余额 ↔ 流水 ↔ 订单）

### 问题描述

钱包系统的核心不变式（invariant）是：

```
不变式：余额的每一分变动，都必须有一条对应的流水记录。
  即：∑(流水变动金额) = 当前余额 - 初始余额(0)

如果违反这个不变式：
  - 余额变了但没有流水 → 无法审计，无法对账
  - 流水写了但余额没变 → 余额不对，用户投诉
  - 订单状态变了但余额和流水没跟上 → 订单已完成但钱没到账
```

### 方案对比

| 方案 | 原理 | 优点 | 缺点 | 代表案例 |
|------|------|------|------|---------|
| **本地事务（推荐）** | 余额更新 + 流水写入 + 订单状态变更在同一个 DB 事务中 | 强一致，不需要额外组件 | 仅限同一数据库内 | 支付宝/微信支付单库场景 |
| 分布式事务（Seata/DTM） | 跨库/跨服务的 2PC/TCC/SAGA | 支持跨库跨服务 | 复杂度高，性能开销大，引入新依赖 | 大型分布式系统 |
| 消息最终一致 | 操作后发消息，消费者异步完成关联操作 | 解耦，吞吐高 | 有窗口期不一致，补偿逻辑复杂 | 跨系统通知 |
| Event Sourcing | 以事件为第一公民，状态从事件回放得出 | 完美审计追溯 | 架构复杂度极高，团队学习成本大 | 金融清算系统 |

### 推荐方案：本地事务（主流）

**为什么选本地事务**：
1. ser-wallet 的余额表、流水表、订单表都在**同一个数据库**中——本地事务完全够用
2. 不需要引入 Seata/DTM 等分布式事务框架——当前工程没有这些组件，引入增加复杂度
3. 简单可靠，99% 的支付系统在单库内都用本地事务

**架构层面的一致性保障**：
```
核心设计：所有余额变更必须且只能通过 ModifyBalance 方法

ModifyBalance 内部（同一事务）：
  1. SELECT ... FOR UPDATE  → 锁定账户行
  2. 校验余额充足
  3. UPDATE wallet_account  → 修改余额
  4. INSERT transaction_log → 写入流水
  5. 事务提交

调用方无法绕过这个方法直接操作余额 → 架构层面强制余额与流水一致

上层业务（如充值入账）：
  query.Q.Transaction(func(tx *query.Query) error {
      engine.ModifyBalance(tx, params)        // 余额+流水
      tx.DepositOrder.Update(status=完成)     // 订单状态
      return nil
  })
  → 三者在同一事务中，要么全成功，要么全回滚
```

## 挑战 4：金额精度

### 问题描述

```
浮点数精度问题：
  0.1 + 0.2 = 0.30000000000000004（IEEE 754 标准）

在钱包场景下的灾难：
  用户充值 0.10 USDT + 0.20 USDT
  → 如果用 float64：余额 = 0.30000000000000004
  → 展示给用户 = 0.30 USDT（四舍五入后看起来没问题）
  → 但对账时：流水合计 0.30 ≠ 余额 0.30000000000000004 → 对账失败

汇率计算放大精度问题：
  BSB:VND 汇率 = 71
  用户充值 1000 VND → 1000 / 71 = 14.084507042253521... BSB
  → 不同的取整策略差一分钱，累积千万笔后就是几千块的差错
```

### 方案对比

| 方案 | 原理 | 优点 | 缺点 | 代表案例 |
|------|------|------|------|---------|
| **整数最小单位（推荐）** | 所有金额存储为最小单位的整数（分/聪） | 零精度损失，运算快，类型安全 | 需要展示层换算 | Stripe/支付宝/微信支付 |
| DECIMAL 类型 | DB 用 DECIMAL(20,8)，Go 用三方 decimal 库 | 直觉友好，直接存原值 | GORM 映射为 string/float64，类型转换易错；Go 无原生 decimal | 传统银行核心 |
| 字符串存储 | 金额存为字符串，运算时解析 | 无精度问题 | 无法做 DB 层聚合/比较；每次运算都要解析转换 | 极少使用 |
| float64 | 直接用浮点数 | 简单 | 精度灾难，绝对不能用于资金系统 | **禁止** |

### 推荐方案：整数最小单位（主流 / 业界标准）

**业界依据**：
- **Stripe**：所有金额以最小货币单位的整数表示（$1.00 = 100 cents）
- **支付宝/微信支付**：内部全部用分（整数）计算
- **以太坊**：Wei（1 ETH = 10^18 Wei），全部整数运算

**具体方案**：
```
每个币种有 decimal_places 和 multiplier 配置：
  VND:  decimal_places=0, multiplier=1       → 1000 VND → 存 1000
  IDR:  decimal_places=0, multiplier=1       → 1000 IDR → 存 1000
  THB:  decimal_places=2, multiplier=100     → 3.50 THB → 存 350
  BSB:  decimal_places=2, multiplier=100     → 1.00 BSB → 存 100
  USDT: decimal_places=8, multiplier=10^8    → 1.00 USDT → 存 100000000

存储：bigint（int64，最大 9.2×10^18，即使越南盾也绰绰有余）
计算：全程 int64 整数运算
展示：返回前端时 ÷ multiplier → 还原为用户看到的值

汇率同理：×10^8 整数存储，避免汇率除法的精度问题
```

**可选增强**：如果未来涉及极复杂的汇率计算（如多步链式换算），可以引入 `shopspring/decimal` Go 库做中间计算，最终结果仍存储为整数。但当前场景（单步汇率换算）int64 足够。

## 挑战 5：跨服务事务（钱包 ↔ 财务联动）

### 问题描述

钱包和财务是两个独立服务，各自有独立数据库。当业务流程横跨两个服务时，本地事务无法覆盖：

```
充值入账场景：
  1. 财务接收三方回调，确认支付成功
  2. 财务调钱包 CreditBalance → 钱包入账成功
  3. 财务更新自己的记录状态为"已入账"
  → 如果步骤3失败：钱已入账但财务还不知道 → 不一致

提现场景：
  1. 钱包冻结余额 + 创建提现订单
  2. 钱包通知财务有新的提现单
  → 如果步骤2失败：余额被冻结但财务不知道 → 钱被锁死
```

### 方案对比

| 方案 | 原理 | 优点 | 缺点 | 适用场景 | 代表案例 |
|------|------|------|------|---------|---------|
| **幂等 RPC + 重试 + 补偿（推荐）** | 每个操作幂等，失败时安全重试，超时后补偿 | 简单实用，不引入新组件 | 需要每个接口都做幂等 | 服务间操作可幂等的场景 | 支付宝/微信支付 |
| TCC (Try-Confirm-Cancel) | 分三阶段：预留→确认→取消 | 语义清晰，可控 | 三阶段增加复杂度和延迟 | 库存/余额的预扣场景 | 蚂蚁金服 TCC |
| SAGA | 每步操作有对应的补偿操作，失败时反向执行 | 长事务友好 | 补偿逻辑复杂，可能出现补偿失败 | 长流程编排 | Uber Cadence |
| 2PC (两阶段提交) | 协调者管理 prepare + commit | 强一致 | 性能差，协调者单点，阻塞严重 | 传统数据库跨库 | XA 协议 |
| 分布式事务框架（Seata/DTM） | 框架托管分布式事务 | 自动化 | 引入重依赖，学习成本高，当前工程无此组件 | 大型分布式系统 | 阿里 Seata |
| 本地消息表 + MQ | 操作和消息写入本地事务，异步消费保证最终一致 | 解耦，高吞吐 | 有延迟，消费者需幂等 | 跨系统通知 | 电商支付通知 |

### 推荐方案：幂等 RPC + 重试 + 定时补偿（主流 / 最佳实践）

**为什么不用 Seata/DTM**：
1. 当前工程没有分布式事务框架，引入是 0→1 的大事
2. 钱包和财务的交互模式是"请求-回调"，天然适合幂等+重试
3. **微信支付的跨系统交互也是基于幂等回调 + 定时对账，不用分布式事务框架**

**具体实现**：
```
原则：每个跨服务操作都做到"可重试不出错"

充值回调（财务 → 钱包）：
  1. 财务调钱包 CreditBalance(ref_order_no=xxx)
  2. 钱包基于 ref_order_no 幂等——已处理则直接返回成功
  3. 财务收到成功 → 更新自己状态
  4. 如果 RPC 超时 → 财务重试调用（安全，因为幂等）
  5. 如果持续失败 → 告警 + 人工介入

提现（钱包 → 财务）：
  1. 钱包冻结余额 + 创建提现订单（本地事务）
  2. 通知财务（RPC 或共享状态）
  3. 如果通知失败 → 定时任务扫描"已冻结但未通知"的订单 → 重试通知
  4. 财务审核完 → 调钱包 UnfreezeBalance/DebitBalance（幂等）

兜底：
  定时任务每 N 分钟扫描异常状态订单
  → 冻结超过 24 小时未审核 → 查询财务状态 → 补偿
  → 支付超时未回调 → 主动查询三方状态 → 补偿
```

**可选方案**：如果未来提现量极大，可以引入本地消息表模式——将"通知财务"这个操作作为一条本地消息写入事务，由后台任务异步投递。但当前阶段 RPC + 重试足够。

## 挑战 6：冻结-确认/取消模式

### 问题描述

提现流程存在时间跨度（用户申请 → 三级审核 → 出款），期间用户余额必须锁定，防止"申请提现1000但余额被花掉"：

```
如果不冻结：
  T0: 余额=1000, 用户申请提现1000
  T1: 余额仍=1000（提现还在审核中）
  T2: 用户投注花掉800, 余额=200
  T3: 提现审核通过, 要扣1000, 但余额只有200 → 扣款失败
  → 已出款但扣不了钱 → 平台亏800
```

### 方案对比

| 方案 | 原理 | 优点 | 缺点 | 代表案例 |
|------|------|------|------|---------|
| **冻结-确认/取消（推荐）** | 申请时 available→frozen，确认时 frozen 扣减，取消时 frozen→available | 简洁直观，两步搞定 | 冻结期间用户看到余额减少 | 支付宝/银行提现 |
| TCC 框架 | Try(预扣)/Confirm(确认)/Cancel(回滚) 由框架协调 | 自动化协调 | 框架依赖重 | 蚂蚁 TCC |
| 预扣记录表 | 独立表记录预扣信息，查询余额时扣除预扣 | 可用余额不变 | 查询余额变复杂（要减去所有预扣） | 少数银行核心 |

### 推荐方案：显式冻结字段（简化 TCC / 业界主流）

```
wallet_account 表双字段设计：
  available_balance — 可用余额
  frozen_balance    — 冻结余额

提现申请：
  available -= 1000
  frozen += 1000
  → 用户看到"可用余额"减少了，但"总余额"没变

审核通过+出款成功：
  frozen -= 1000
  → 钱真正离开系统

审核驳回/出款失败：
  frozen -= 1000
  available += 1000
  → 钱归还用户

优势：
  1. 不需要 TCC 框架，两个字段 + 三种操作搞定
  2. 用户的可用余额实时正确，不会超扣
  3. 操作全在本地事务内，强一致
```

**冻结超时兜底**：定时任务扫描 withdraw_order 中 status=待审核且 create_at 超过 N 小时的记录，查询财务侧状态做补偿。

## 挑战 7：回调可靠性

### 问题描述

充值链路中，三方支付结果通过回调传递：三方 → 财务 → 钱包。这条链路上任何一个节点故障都会导致"用户付了钱但余额没增加"。

```
可能的故障点：
  1. 三方回调丢失（网络故障/三方bug）
  2. 财务服务宕机，收不到回调
  3. 财务调钱包 CreditBalance 超时
  4. 钱包服务在处理过程中重启
```

### 方案对比

| 方案 | 原理 | 优点 | 缺点 | 代表案例 |
|------|------|------|------|---------|
| **回调 + 主动查询 + 补单（推荐）** | 回调为主，定时主动查询三方状态为辅，异常时人工补单 | 覆盖全面，简单实用 | 定时查询有延迟 | 支付宝/微信支付 |
| 消息队列保障 | 回调消息写入 MQ，消费者幂等处理 | 消息不丢（持久化） | 引入 MQ 依赖，消费延迟 | 大流量支付 |
| Webhook 重试机制 | 三方自带重试（指数退避） | 零开发成本 | 依赖三方的重试策略，不可控 | Stripe Webhook |

### 推荐方案：三层保障（主流）

```
第一层：回调（实时性）
  三方 → 财务接收回调 → 解析签名 → 调钱包 CreditBalance
  正常情况下 95%+ 的订单通过回调完成入账

第二层：主动查询（兜底）
  定时任务扫描 deposit_order 中状态=待支付/支付中且超过 N 分钟的订单
  → 主动向三方/财务查询订单状态
  → 如果已支付成功但未入账 → 触发入账（幂等安全）
  → 如果已超时 → 关闭订单

第三层：人工补单（最终兜底）
  以上都覆盖不了的异常情况
  → B端提供人工补单入口（财务模块已有此功能）
  → 审核通过后调钱包 CreditBalance（幂等）
```

## 挑战 8：热点账户

### 问题描述

当同一个用户在短时间内发生大量操作（如自动投注机器人、活动奖励批量发放），行锁导致所有请求排队：

```
用户A同时有10个请求操作同一钱包账户
→ 第1个拿到行锁，其他9个等待
→ 每个请求持锁10ms → 第10个请求等待90ms
→ 如果并发更高 → 等待超时 → 请求失败
```

### 方案对比

| 方案 | 原理 | 优点 | 缺点 | 适用场景 | 代表案例 |
|------|------|------|------|---------|---------|
| **短事务 + 快速释放锁（推荐）** | 把事务时间压到最短 | 简单有效 | 有并发上限 | 大多数场景（TPS < 1000 同一账户） | 通用实践 |
| 子账户拆分 | 一个逻辑账户拆多个子账户，随机写入 | 极大提高并发 | 查询余额需汇总；退款/提现时需合并 | 超高频写入（如平台总账户） | 支付宝平台账户 |
| 异步合并写入 | 内存中累积多次操作，定期批量刷入 DB | 极高吞吐 | 存在数据丢失风险，一致性降低 | 统计类非核心数据 | 消息计数 |
| 队列串行化 | 同一用户的请求放入队列顺序处理 | 无锁竞争 | 引入队列延迟，需要分布式队列 | 对延迟不敏感的场景 | 后台批处理 |

### 推荐方案：短事务优先（主流）+ 必要时子账户拆分（可选增强）

**当前阶段：短事务即可**

```
优化事务内操作，控制在个位毫秒：
  1. 事务外完成所有参数校验和计算
  2. 事务内只做：FOR UPDATE → UPDATE → INSERT
  3. 不在事务内做 RPC 调用、日志写入、复杂计算
  4. 审计日志用 oneway（不等响应）放在事务外

按 10ms/事务估算：
  单用户单账户 TPS ≈ 100
  这对于 C 端用户来说绰绰有余（用户不可能每秒操作100次）
```

**未来增强：子账户拆分**（仅在需要时启用）

```
适用场景：平台公共账户（如活动资金池）被大量用户同时操作

方案：
  将一个逻辑账户拆为 N 个子账户（如 10 个）
  写入时随机选择一个子账户
  查询余额时 SUM 所有子账户
  提现时按需合并

当前不需要：
  钱包的热点是用户个人账户，单用户 TPS 不会超过 100
  子账户适合平台级公共账户（当前需求无此场景）
```

## 挑战 9：汇率实时性与缓存一致性

### 问题描述

汇率数据被高频读取（每次充值/兑换/提现/余额展示都需要汇率），但不能缓存太久（汇率变化后用户看到的是旧汇率）：

```
矛盾：
  - 不缓存：每次操作都查 DB，QPS 高时 DB 压力大
  - 缓存时间长：汇率更新后用户仍看到旧值，用旧汇率交易
  - 缓存时间短：缓存失效频率高，缓存效果差
```

### 方案对比

| 方案 | 原理 | 优点 | 缺点 | 代表案例 |
|------|------|------|------|---------|
| **Cache-Aside + 主动写入（推荐）** | 更新 DB 后主动写 Redis；读取时先查 Redis | 一致性好，读性能高 | 需要更新时主动写缓存 | 通用实践 |
| 纯 Cache-Aside（删缓存） | 写 DB 后删 Redis，下次读时重建 | 简单 | 高频读场景下缓存Miss后所有请求回源DB | 低频变更数据 |
| TTL 短缓存 | 设置很短的 TTL（如 10 秒） | 自动过期，简单 | 频繁过期导致 DB 压力 | 非核心数据 |
| 本地缓存 + 订阅 | 进程内缓存，通过 Redis Pub/Sub 或 ETCD Watch 通知更新 | 零网络开销 | 多实例间一致性延迟 | 配置类数据 |

### 推荐方案：Cache-Aside + 主动写入（主流）

```
汇率数据的缓存策略：

  写入（汇率更新时）：
    1. 写 DB（currency 表的汇率字段 + exchange_rate_log）
    2. 主动写 Redis（不是删除，因为汇率读频率极高，避免缓存穿透）
    key: wallet:rate:{currency_code}
    value: JSON { platformRate, depositRate, withdrawRate, updatedAt }
    TTL: 与汇率拉取周期对齐（如拉取间隔 2 分钟，TTL 设 3 分钟）

  读取（业务操作时）：
    1. 查 Redis → 命中 → 返回
    2. 未命中 → 查 DB → 写入 Redis → 返回

  为什么汇率用"主动写"而不是"删除"：
    汇率是所有用户共享的热数据，删除后瞬间所有请求回源 DB（缓存击穿）
    主动写保证缓存永远有数据，不会出现击穿

  交易时汇率快照：
    创建订单时将当时的汇率写入订单的 rate_snapshot 字段
    后续入账/扣款按快照汇率计算，不受后续汇率变动影响
    → 解决"建单时71，入账时变72"的不一致问题
```

## 挑战 10：大表查询性能（流水表增长）

### 问题描述

`transaction_log` 是只增不删的流水表，随着用户量和操作频率增长，数据量会持续膨胀：

```
估算：
  10万活跃用户 × 每天平均 5 笔操作 = 每天 50 万条
  一年 ≈ 1.8 亿条
  三年 ≈ 5.4 亿条
  → 单表5亿行时，即使有索引，分页查询也会变慢
```

### 方案对比

| 方案 | 原理 | 优点 | 缺点 | 适用场景 | 代表案例 |
|------|------|------|------|---------|---------|
| **索引优化 + 分页策略（当前推荐）** | 合理索引 + 游标分页替代 OFFSET | 简单有效 | 数据量极大时仍有瓶颈 | 千万级 | 通用实践 |
| 分区表（时间分区） | 按月/季度自动分区 | 查询自动裁剪分区，老数据可归档 | 跨分区查询性能差 | 亿级 | TiDB 原生支持 |
| 冷热分离 | 近N天数据在主表（热），历史数据归档到归档表（冷） | 主表保持小体量 | 需要归档任务和查询路由 | 十亿级 | 支付宝流水 |
| 分库分表 | 按 user_id 分片 | 单表数据量可控 | TiDB 自带分布式不需要应用层分片 | MySQL 单机场景 | 传统支付 |

### 推荐方案：分阶段应对

```
阶段1（当前，数据量 < 1000 万行）：
  索引优化 + 标准分页
  → idx_user_currency_time 覆盖 C 端记录查询
  → 前端限制 pageSize ≤ 20
  → 足够支撑初期用户量

阶段2（千万到亿级）：
  TiDB 分区表
  → 按 create_at 时间范围分区（按月）
  → 查询时 TiDB 自动裁剪到对应分区
  → C 端查询通常只看近 N 天，自然命中最新分区

  ALTER TABLE transaction_log PARTITION BY RANGE(create_at) (
      PARTITION p202601 VALUES LESS THAN (1738368000000),
      PARTITION p202602 VALUES LESS THAN (1740787200000),
      ...
  );

阶段3（十亿级，远期）：
  冷热分离
  → 定期将 90 天前的数据迁移到归档表 transaction_log_archive
  → 查询路由：近期查主表，历史查归档表
  → B 端导出走归档表

TiDB 的优势：
  TiDB 本身是分布式数据库，数据自动分片到多个 TiKV 节点
  应用层不需要分库分表，天然比 MySQL 多一层扩展能力
  → 所以"分库分表"方案对我们不适用
```

## 挑战 11：敏感数据保护

### 问题描述

提现账户信息（银行卡号、手机号、USDT 地址、持有人姓名）是**敏感个人信息**，需要加密存储：

```
风险：
  数据库泄露 → 用户银行卡号/USDT地址暴露 → 法律责任 + 信任危机
  内部人员滥用 → 直接读 DB 看到明文 → 隐私违规
  日志泄露 → SQL日志包含明文银行卡号 → 信息外泄
```

### 方案对比

| 方案 | 原理 | 优点 | 缺点 | 代表案例 |
|------|------|------|------|---------|
| **应用层加密（GORM 钩子）（推荐）** | GORM 自定义类型在 Scan/Value 时自动加解密 | 对业务代码透明，现有工程已有组件 | 无法在 DB 层做模糊搜索 | 当前工程 EncryptedString |
| DB 层加密（TDE） | 数据库透明数据加密 | 对应用完全透明 | 防不了有 DB 账号的内部人员 | MySQL Enterprise |
| 独立加密服务（KMS） | 密钥托管在 KMS，加解密通过 API | 密钥管理安全 | 每次读写都要调 KMS，延迟增加 | AWS KMS/阿里云 KMS |
| 列级加密函数 | DB 的 AES_ENCRYPT/AES_DECRYPT | 简单 | 密钥硬编码在 SQL 中 | 简易场景 |

### 推荐方案：应用层 GORM 加密（现有工程方案）

```
当前工程已有完整的加密方案：

1. 加密组件：common/pkg/utils/crypto_util.go
   → EncryptedString 自定义类型
   → 实现 GORM 的 Scan/Value 接口
   → 写入时 AES-256-CBC 加密 → Base64 编码 → 存入 DB
   → 读取时 Base64 解码 → AES-256-CBC 解密 → 返回明文

2. 密钥管理：ETCD
   → /slg/conf/security/sensitive_data_key
   → /slg/conf/security/sensitive_data_iv
   → 启动时从 ETCD 加载，不硬编码在代码中

3. GORM Gen 配置：
   gen.FieldType("account_no", "utils.EncryptedString")
   gen.FieldType("holder_name", "utils.EncryptedString")

4. 对业务代码完全透明：
   account.AccountNo = "622588****1234"  // 直接赋值明文
   db.Create(account)                     // GORM 自动加密后存入
   db.Find(&account)                      // GORM 自动解密后返回明文

加密字段列表（withdraw_account 表）：
  account_no   — 银行卡号/手机号/USDT地址
  holder_name  — 持有人姓名
  → DB 中存储为密文，varchar(512) 足够存放加密后的 Base64 字符串
```

**可选增强**：
- 日志脱敏：在日志中不打印银行卡号/USDT地址的完整明文，只显示掩码（如 622588****1234）
- 密钥轮换：定期更换 AES 密钥时，需要批量重新加密已有数据

## 挑战 12：接口安全与防刷

### 问题描述

钱包接口直接涉及资金操作，是攻击的高价值目标：

```
攻击向量：
  1. 未授权访问：绕过认证直接调用充值/提现接口
  2. 参数篡改：修改充值金额、币种ID、用户ID
  3. 重放攻击：抓包后重放充值回调请求
  4. 并发攻击：同一时刻大量请求尝试超扣
  5. 频率攻击：高频调用创建订单接口消耗资源
```

### 方案对比与综合方案

| 安全层 | 方案 | 实现方式 | 当前工程支持 |
|--------|------|---------|-------------|
| **认证** | C端 Token 认证 | FontAuthMiddleware（Redis Token 校验） | **已有** |
| **认证** | B端 Token + URL 权限 | BackAuthMiddleware + CheckUrlAuth | **已有** |
| **IP管控** | C端黑名单 / B端白名单 | BlackCheck / WhiteCheck 中间件 | **已有** |
| **参数校验** | IDL required + 业务层校验 | Thrift required + 手动校验 | **需实现** |
| **防重放** | 幂等键 + 签名校验 | DB 唯一索引 + 回调签名验证（财务负责） | **需实现** |
| **频率限制** | Redis 限频 | SetNX + TTL 或 滑动窗口 | **需实现** |
| **审计追溯** | 全链路日志 | ser-blog.AddActLog + TraceID | **已有** |

### 推荐方案：分层防御

```
第1层：网关认证（已有）
  → C端请求必须携带合法 Token，FontAuthMiddleware 校验
  → B端请求必须携带合法 Token + URL 权限校验

第2层：参数校验（需实现）
  → IDL 中 required 字段 + 业务层二次校验
  → 金额 > 0，币种ID存在且启用，用户ID = Token中的用户ID（防止篡改）
  → 关键操作的 user_id 从 Token 解析（tracer.GetUserId），不接受前端传入

第3层：频率限制（需实现）
  → 充值建单：同一用户每分钟最多 N 次
  → 提现申请：同一用户每小时最多 N 次
  → 用 Redis INCR + EXPIRE 实现滑动窗口

第4层：幂等保护（已设计）
  → 所有资金操作基于 DB 唯一索引幂等
  → 防止重放和重试导致的重复操作

第5层：审计日志（已有）
  → 所有写操作通过 ser-blog.AddActLog 记录
  → TraceID 全链路可追溯
```

## 挑战 13：审计与合规追溯

### 问题描述

钱包涉及资金和个人信息，需要满足审计要求：

```
审计需求：
  1. 每笔资金变动可追溯（谁/什么时间/什么原因/变动前后余额）
  2. 每个操作员的操作可追溯（谁在B端做了什么修改）
  3. 异常操作可告警（大额/高频/非常规操作）
  4. 数据不可篡改（流水记录不可修改不可删除）
```

### 推荐方案：多层审计体系

```
第1层：交易流水（transaction_log）
  → 只增不改不删，永久保留
  → 每条记录包含：用户/币种/钱包/操作类型/金额/方向/变动后余额/关联单号/时间
  → balance_after 字段支持回放对账

第2层：操作行为日志（ser-blog）
  → 所有B端写操作调用 AddActLog 记录
  → 记录：操作人/操作类型/操作对象/关键标识/时间
  → 产品需求明确的操作类型：编辑/启用/禁用币种

第3层：审计字段六件套
  → 每张表（除只增日志表外）都有 create_by/create_by_name/create_at/update_by/update_by_name/update_at
  → UserPlugin 自动填充，开发者无需手动

第4层：汇率变更日志（exchange_rate_log）
  → 每次汇率更新都留痕
  → 保留触发时的完整参数快照（浮动/阈值/偏差/触发类型）
```

## 挑战 14：订单状态机管理

### 问题描述

充值和提现订单有多个状态和多种终态，状态流转必须严格单向，不允许跳跃或回退：

```
错误示例：
  1. 充值订单从"待支付"直接跳到"已完成"，跳过了支付中和入账步骤
  2. 提现订单从"已完成"回退到"审核中"（不可逆操作）
  3. 两个并发请求同时把订单从"待支付"推进到"支付中"，生成两个回调处理
```

### 方案对比

| 方案 | 原理 | 优点 | 缺点 | 代表案例 |
|------|------|------|------|---------|
| **DB 条件更新（推荐）** | UPDATE SET status=new WHERE status=old | 简单可靠，DB 层保证原子性 | 只有成功/失败两种结果 | 支付宝/微信支付 |
| 状态机框架 | 用 SM 框架定义合法转换 | 可视化、可验证 | 引入框架依赖 | Spring StateMachine |
| 版本号控制 | 每次更新版本号+1，写入时校验版本号 | 通用 | 本质是乐观锁，高冲突时重试 | 辅助手段 |

### 推荐方案：DB 条件更新（主流 / 最简单有效）

```
原则：状态更新必须用 WHERE status = 当前状态

充值状态机：
  待支付(0) → 支付中(1) → 待入账(2) → 已完成(3)
  待支付(0) → 已超时(5)
  支付中(1) → 已失败(4)

  实现：
  affected := tx.DepositOrder.Where("order_no = ? AND status = ?", orderNo, 0).
      Update("status", 1)
  if affected.RowsAffected == 0 {
      // 状态已变（被其他请求处理了），直接返回
      return nil
  }

提现状态机：
  待审核(0) → 审核中(1) → 出款中(2) → 已完成(3)
  任意环节 → 已失败(4)（+ 解冻余额）
  → 已取消(5)（+ 解冻余额）

好处：
  1. DB 层原子保证——并发请求只有一个能成功更新
  2. affected_rows=0 意味着"已被处理" → 直接返回（幂等）
  3. 不需要额外框架，一行 SQL 搞定
```

## 挑战 15：多钱包扣款优先级

### 问题描述

产品需求规定：**消费扣款优先级：先扣中心钱包余额，再扣奖励钱包余额**。返奖时还需要按投注比例分回：

```
难点：
  1. 跨钱包原子性：扣中心钱包和奖励钱包必须在同一事务中
  2. 比例计算：混合投注时记录各钱包的扣款比例
  3. 死锁风险：同时锁两个钱包账户（两行），需要保证锁顺序
```

### 推荐方案

```
扣款逻辑：
  请求扣减 X

  1. 事务内按 wallet_type 升序加锁（中心=1, 奖励=2）
     → SELECT ... FOR UPDATE WHERE user_id=? AND currency_id=? AND wallet_type=1
     → SELECT ... FOR UPDATE WHERE user_id=? AND currency_id=? AND wallet_type=2

  2. 计算扣减分配：
     A = 中心钱包可用余额
     B = 奖励钱包可用余额
     if A + B < X → 余额不足，回滚
     if A >= X → 只扣中心：modifyBalance(中心, debit, X)
     if A < X → 先扣光中心，差额扣奖励：
       modifyBalance(中心, debit, A)
       modifyBalance(奖励, debit, X - A)
       记录比例 ratio = A / X（返奖时用）

  3. 投注返奖时按比例分配：
     返奖 Y
     中心入账 = Y × ratio
     奖励入账 = Y × (1 - ratio)

死锁预防：
  所有涉及多行锁的操作，统一按 wallet_type 或 currency_id 升序加锁
  → 保证所有事务以相同顺序获取锁，杜绝死锁
```

## 挑战 16：汇率计算的精度与一致性

### 问题描述

```
汇率计算涉及除法，除法在整数运算中会丢失精度：
  1000 VND ÷ 71（汇率）= 14.084507... BSB
  整数运算：1000 ÷ 71 = 14（截断）→ 丢失 0.08 BSB

汇率一致性问题：
  T0: 用户看到汇率 71，预览兑换 1000 VND → 14.08 BSB
  T1: 汇率变为 72，用户点确认
  → 按 72 计算只有 13.89 BSB，比预览少了
  → 用户投诉
```

### 推荐方案

```
精度方案：
  汇率存储：×10^8 整数
  金额存储：最小单位整数

  计算公式（兑换 VND → BSB）：
    source_amount = 1000（VND，multiplier=1）
    rate = 7100000000（71.00 × 10^8）
    bsb_multiplier = 100（BSB 小数位 2）

    target_amount = source_amount × bsb_multiplier × 100000000 / rate
                  = 1000 × 100 × 100000000 / 7100000000
                  = 10000000000000 / 7100000000
                  = 1408（BSB 最小单位）→ 显示 14.08 BSB

  关键：先乘后除，避免除法截断

  取整规则：
    充值兑换（用户获得）→ 向下取整（truncate）→ 平台不亏
    提现换算（用户支出）→ 向上取整（ceiling）→ 平台不亏

汇率一致性方案：
  1. 预览时返回汇率，但不锁定
  2. 执行时重新获取最新汇率
  3. 锁定汇率写入订单 rate_snapshot
  4. 后续入账/扣款严格按 rate_snapshot 计算
  → 用户看到的最终金额以执行时为准
```

## 挑战 17：对账机制

### 问题描述

即使有事务和幂等保障，仍可能因 Bug、极端故障等原因出现数据不一致。需要有机制**发现**不一致：

```
对账场景：
  1. 内部对账：余额 vs 流水（本系统内）
  2. 渠道对账：我方订单 vs 三方通道订单（跨系统）
  3. 模块对账：我方订单 vs 财务方记录（跨服务）
```

### 方案对比

| 方案 | 原理 | 优点 | 缺点 | 代表案例 |
|------|------|------|------|---------|
| **流水回放对账（推荐）** | 按时间顺序回放流水，验证每笔 balance_after 正确 | 精确到每一笔 | 数据量大时耗时 | 银行核心 |
| 余额快照 + 增量校验 | 每天快照余额，对比当天流水增量 | 效率高 | 需要额外快照存储 | 支付宝 |
| 实时校验 | 每次操作后立即验证 | 即时发现 | 性能开销 | 关键交易 |

### 推荐方案：多层对账

```
第1层：流水回放对账（每日定时任务）
  对每个钱包账户：
    按时间顺序读取 transaction_log
    初始余额 = 0
    逐条累加：balance += amount × direction_sign
    最终结果与 wallet_account.available_balance 比较
    → 不一致 → 告警 + 人工排查

  实现要点：
    transaction_log 每条记录有 balance_after 字段
    回放时校验：上一笔 balance_after ± 本笔 amount = 本笔 balance_after
    → 任何一笔不等 → 定位到出错的那笔操作

第2层：订单维度对账（每日）
  充值：deposit_order 已完成金额合计 = 同期 transaction_log 充值类型金额合计
  提现：withdraw_order 已完成金额合计 = 同期 transaction_log 提现类型金额合计

第3层：渠道对账（按需）
  与财务模块定期核对订单状态和金额
  → 我方"已完成"但对方"未知" → 异常
  → 对方"已成功"但我方"待入账" → 补偿入账
```

## 挑战 18：异常兜底与告警

### 问题描述

线上运行时可能遇到各种异常，需要及时发现和处理：

```
关键异常场景：
  1. 提现冻结后财务一直不回调 → 资金被永久锁定
  2. 充值订单超时未关闭 → 占用重复充值检测的计数
  3. 汇率拉取全部失败 → 平台汇率过时
  4. 余额变为负数（不该发生，但如果发生说明有严重Bug）
  5. 大额操作（可能是误操作或攻击）
```

### 推荐方案：定时扫描 + 实时告警

```
定时扫描任务（内部 goroutine + time.Ticker）：

  1. 充值超时扫描（每 5 分钟）
     → 查 deposit_order WHERE status=0 AND create_at < 当前-30分钟
     → 更新为已超时

  2. 冻结超时扫描（每 30 分钟）
     → 查 withdraw_order WHERE status IN (0,1) AND create_at < 当前-24小时
     → 主动查询财务状态 → 补偿

  3. 汇率拉取监控（每次汇率任务执行后）
     → 如果三家 API 全部失败 → 记录失败日志 + 告警
     → 连续 N 次失败 → 升级告警

  4. 余额负数检查（每日）
     → 查 wallet_account WHERE available_balance < 0 OR frozen_balance < 0
     → 有结果 → 紧急告警

实时告警（在业务代码中）：

  5. 大额操作告警
     → 充值/提现金额超过阈值 → 记录日志 + 告警
     → 阈值可通过 ETCD 配置

  6. 高频操作告警
     → 同一用户短时间内操作超过阈值 → 记录日志 + 告警

多实例下的定时任务抢锁：
  ticker := time.NewTicker(interval)
  for range ticker.C {
      locked := rds.SetNX("wallet:task:{taskName}:lock", instanceId, interval)
      if locked {
          doTask()
      }
  }
  → 保证同一时刻只有一个实例执行
```

## 总结：挑战与方案速查表

| # | 挑战 | 主流方案 | 可选方案 | 本项目推荐 |
|---|------|---------|---------|-----------|
| 1 | 并发余额操作 | SELECT FOR UPDATE 悲观行锁 | 乐观锁/Redis锁/队列串行化 | **悲观行锁**（高冲突场景标准方案） |
| 2 | 操作幂等性 | DB 唯一索引 | Redis SetNX/幂等Token | **DB唯一索引**（与事务原子绑定） |
| 3 | 数据一致性 | 本地事务 | 分布式事务/Event Sourcing | **本地事务**（单库足够） |
| 4 | 金额精度 | 整数最小单位 | DECIMAL/字符串 | **整数最小单位**（Stripe/支付宝标准） |
| 5 | 跨服务事务 | 幂等RPC+重试+补偿 | Seata/DTM/本地消息表 | **幂等+补偿**（简单可控） |
| 6 | 冻结模式 | 显式冻结字段 | TCC框架/预扣表 | **显式冻结字段**（简化TCC） |
| 7 | 回调可靠性 | 回调+主动查询+补单 | MQ保障/Webhook重试 | **三层保障** |
| 8 | 热点账户 | 短事务快释放 | 子账户拆分/队列串行化 | **短事务**（当前够用） |
| 9 | 汇率缓存 | Cache-Aside+主动写入 | 本地缓存+订阅/短TTL | **主动写入**（高频读不穿透） |
| 10 | 大表性能 | 索引优化+分页 | 分区表/冷热分离 | **分阶段**（当前索引够，后续分区） |
| 11 | 敏感数据 | 应用层GORM加密 | TDE/KMS | **GORM加密**（已有组件） |
| 12 | 接口安全 | 分层防御 | WAF/API网关 | **分层防御**（已有中间件+按需补充） |
| 13 | 审计合规 | 流水+行为日志+审计字段 | Event Sourcing | **多层审计**（已有组件） |
| 14 | 状态机 | DB条件更新 | SM框架/版本号 | **DB条件更新**（一行SQL搞定） |
| 15 | 多钱包扣款 | 有序加锁+原子事务 | 队列串行化 | **有序加锁**（按type升序） |
| 16 | 汇率精度 | 先乘后除+快照锁定 | decimal库 | **整数运算+快照** |
| 17 | 对账 | 流水回放+订单维度 | 实时校验/快照增量 | **每日对账任务** |
| 18 | 异常兜底 | 定时扫描+实时告警 | 外部监控平台 | **定时扫描+告警** |

### 一句话总结

钱包模块的技术挑战本质上都是**资金安全 + 分布式可靠性 + 性能**三个方向的问题。市面上（支付宝、微信支付、Stripe）已有成熟方案，核心策略就是：**悲观锁保并发、唯一索引保幂等、本地事务保一致、整数保精度、幂等+补偿保跨服务、短事务保性能**。这些方案简单、可靠、经过亿级交易验证，完全适用于当前工程的技术栈和业务规模。
