## 2. 挑战1：并发余额操作的一致性

> **严重程度：P0 致命**
> 问题本质：多个请求同时操作同一用户的同一钱包余额时，如何保证最终余额正确

### 2.1 挑战描述

```
场景还原：用户 A 的 BSB 中心钱包余额 = 10000.00000000

时刻T1：充值回调上账 +5000（CreditWallet）
时刻T1：游戏结算扣款 -3000（DebitWallet）

两个请求并发到达，如果没有并发控制：

  线程1 读余额 → 10000
  线程2 读余额 → 10000
  线程1 更新余额 → 10000 + 5000 = 15000
  线程2 更新余额 → 10000 - 3000 = 7000   ← 覆盖了线程1！

最终余额 = 7000（正确应为 12000），丢失了 5000 的充值
```

### 2.2 主流方案

#### 方案A：数据库悲观锁 — SELECT FOR UPDATE（推荐）

```
使用者：支付宝、微信支付、Grab Financial、OVO
原理：在事务内先锁定余额行，其他事务排队等待

实现：
  BEGIN;
  SELECT * FROM wallet WHERE user_id=? AND currency_code=? AND wallet_type=?
    FOR UPDATE;                    -- 行级排他锁
  UPDATE wallet SET balance = balance + ?, version = version + 1 WHERE id = ?;
  INSERT INTO wallet_transaction (...);   -- 同事务写流水
  COMMIT;                          -- 释放锁

优点：
  ✅ 绝对安全，数据库层面保证串行化
  ✅ 实现简单，一行 SQL 加锁
  ✅ 锁粒度细（行级），不同用户互不影响
  ✅ TiDB 完全兼容（4.0+ 默认悲观事务模式）

缺点：
  ⚠️ 同一行高并发时有排队等待（但钱包场景同一用户并发不高）
  ⚠️ 有死锁风险（可通过固定加锁顺序规避）

性能数据：
  锁持有时间 ≈ 5-15ms（1次 UPDATE + 1次 INSERT）
  QPS 上限 ≈ 单用户 60-200 TPS（远超钱包操作频率）
```

#### 方案B：乐观锁 — CAS + Version

```
使用者：电商库存扣减、部分支付系统的低冲突场景
原理：不加锁，通过版本号检测冲突，冲突时重试

实现：
  SELECT id, balance, version FROM wallet WHERE ...;
  UPDATE wallet SET balance = balance + ?, version = version + 1
    WHERE id = ? AND version = ?;  -- CAS
  if affected_rows == 0 {
    // 冲突，重试
  }

优点：
  ✅ 无锁等待，低冲突场景性能好
  ✅ 无死锁风险

缺点：
  ⚠️ 高冲突场景大量重试，性能反而差
  ⚠️ 重试期间余额状态不确定，增加资金风险
  ⚠️ 需要额外的重试逻辑（退避策略、上限控制）
  ⚠️ 资金场景零容错，重试逻辑中的 bug 可能导致资金错误
```

### 2.3 可选方案

#### 方案C：数据库原子 UPDATE（无需先 SELECT）

```
使用者：部分简化场景
原理：直接用 SQL 原子操作更新余额

实现：
  UPDATE wallet SET balance = balance + ? WHERE user_id=? AND currency_code=? AND wallet_type=?
    AND balance + ? >= 0;  -- 扣款时检查余额充足

优点：
  ✅ 无需显式加锁，SQL 本身原子
  ✅ 性能最好

缺点：
  ⚠️ 无法获取 balance_before（写流水需要）
  ⚠️ 无法在更新前做复杂业务校验
  ⚠️ 不适合需要读-计算-写的场景（如兑换涉及汇率计算）
```

#### 方案D：应用层串行化队列

```
使用者：部分高频交易系统
原理：把同一用户的所有请求投入队列，单线程顺序处理

实现：
  使用 Redis List 或 Go Channel，按 user_id 分桶
  每个桶单 goroutine 消费

优点：
  ✅ 天然串行，无并发问题
  ✅ 适合超高频操作（如实时结算）

缺点：
  ⚠️ 架构复杂度高
  ⚠️ 需要处理队列积压、消费者故障
  ⚠️ 延迟增加（入队→等待→处理）
  ⚠️ 当前场景不需要（钱包操作不是超高频）
```

### 2.4 本项目最佳实践

```
✅ 采用方案A：数据库悲观锁（SELECT FOR UPDATE）+ 方案B 乐观锁 version 辅助

理由：
1. 资金操作零容错，悲观锁是最安全的选择
2. 同一用户的钱包操作并发量不高（非秒杀场景），排队时间极短
3. TiDB 完全支持悲观事务（默认模式）
4. 现有工程 query.Q.Transaction() 直接可用
5. 支付宝、微信支付等一线资金系统的标准选择
6. version 字段作为双重保障——即使悲观锁因某种原因失效，乐观锁也能兜底

工程落地：
  account_ser.go 中所有余额修改操作统一走以下模式：

  func (s *AccountSer) CreditInTx(ctx context.Context, tx *query.Query, params CreditParams) error {
      // Step1: SELECT FOR UPDATE 加行锁
      w, err := tx.Wallet.WithContext(ctx).
          Where(tx.Wallet.UserID.Eq(params.UserID)).
          Where(tx.Wallet.CurrencyCode.Eq(params.Currency)).
          Where(tx.Wallet.WalletType.Eq(params.WalletType)).
          Where(tx.Wallet.DeleteAt.Eq(0)).
          ForUpdate().
          First()

      // Step2: 余额计算（使用 shopspring/decimal 保证精度）
      newBalance := w.Balance.Add(params.Amount)

      // Step3: UPDATE wallet + version++
      // Step4: INSERT wallet_transaction（同事务）
      // Step5: COMMIT 释放锁
  }
```

## 3. 挑战2：金额精度与浮点陷阱

> **严重程度：P0 致命**
> 问题本质：计算机浮点数无法精确表示十进制小数，金融计算中 0.1 + 0.2 ≠ 0.3

### 3.1 挑战描述

```
浮点精度灾难示例：

  Go: 0.1 + 0.2 = 0.30000000000000004
  Go: 1.005 * 100 = 100.49999999999999（四舍五入后变成 100 而非 101）

在钱包场景中：
  - THB 有 2 位小数：1500.50 THB
  - USDT 有 8 位小数：1.50000000 USDT
  - 汇率计算涉及乘除：5000000 VND × 汇率 = ? BSB

如果用 float64 做金额计算：
  累积误差可能导致 1 分钱的偏差
  大量交易累积后偏差会放大
  余额 + 流水不一致（审计无法通过）
```

### 3.2 主流方案

#### 方案A：最小单位整数存储

```
使用者：Stripe、Square、支付宝、微信支付、Shopee、Grab
原理：将金额转换为最小精度单位的整数，所有计算在整数域完成

规则：
  VND/IDR（0位小数）：1:1 直接存储         5,000,000 VND → 5000000
  THB/BSB（2位小数）：×100 转为最小单位     1,500.50 THB → 150050
  USDT  （8位小数）：×10^8 转为最小单位     1.50000000 USDT → 150000000

存储类型：int64（Go）/ BIGINT（数据库）

优点：
  ✅ 整数运算零精度丢失
  ✅ CPU 原生整数运算，性能最优
  ✅ Go 原生 int64，无需第三方库
  ✅ 业界公认的金融计算标准方案

缺点：
  ⚠️ 需要在入口和出口做单位转换
  ⚠️ 跨币种计算时需注意精度系数不同
  ⚠️ 与现有工程金额字段（int32）有溢出风险需评估
```

#### 方案B：DECIMAL 类型 + 精确计算库（推荐）

```
使用者：传统银行系统、会计系统、部分现代支付系统
原理：数据库使用 DECIMAL(20,8) 存储，Go 侧使用 shopspring/decimal 保证计算精度

实现：
  数据库层：
    CREATE TABLE wallet (
      balance DECIMAL(20,8) NOT NULL DEFAULT 0.00000000
    );

  Go 层：
    import "github.com/shopspring/decimal"
    a := decimal.NewFromString("1500.50")
    b := decimal.NewFromString("200.25")
    c := a.Add(b)  // 1700.75（精确）

优点：
  ✅ 数据库层面精确，无精度丢失
  ✅ 直觉友好，存储值即为展示值（所见即所得）
  ✅ shopspring/decimal 是 Go 社区最成熟的精确计算库（GitHub 5.7k stars）
  ✅ 统一精度 DECIMAL(20,8) 简化设计——法币整数只是展示层截断
  ✅ 免去入库/出库的单位转换步骤

缺点：
  ⚠️ 引入 shopspring/decimal 第三方依赖
  ⚠️ decimal 计算性能低于原生整数运算（但钱包操作不是计算密集型）
  ⚠️ DB→Go 转换需要 GORM 自定义类型映射
  ⚠️ 与现有工程金额字段（int32）类型不一致
```

### 3.3 可选方案

#### 方案C：字符串存储 + 精确计算库

```
使用者：部分跨语言金融系统
原理：金额以字符串传输和存储，计算时使用精确计算库

优点：
  ✅ 任意精度，无上限
  ✅ 跨语言兼容性好

缺点：
  ⚠️ 不能用数据库做 SUM/比较等聚合运算
  ⚠️ 索引和排序无法正常工作
  ⚠️ 性能差（字符串解析开销）
```

### 3.4 本项目最佳实践

```
✅ 采用方案B：DECIMAL(20,8) + shopspring/decimal

理由：
1. 统一精度 DECIMAL(20,8) — USDT 需要 8 位小数，法币整数只是展示层截断
2. 所见即所得 — 数据库存 1500.50 就是 1500.50，不需要÷100 转换
3. shopspring/decimal 精确计算 — 避免浮点陷阱，业界成熟方案
4. DECIMAL 支持 SUM/比较/排序 — 对账 SQL 可以直接写 SUM(amount)
5. GORM 可以通过自定义类型映射 decimal.Decimal → DECIMAL(20,8) → decimal.Decimal
6. 虽然引入新依赖，但精确计算在资金场景是刚需，值得引入

DECIMAL(20,8) 容量分析：
  20位总长 - 8位小数 = 12位整数
  最大值 = 999,999,999,999.99999999
  以 USDT 计 ≈ 万亿级别，远超需求

shopspring/decimal 关键 API：
  decimal.NewFromString("1500.50")     // 从字符串创建（推荐）
  decimal.NewFromFloat(1500.50)        // 从浮点创建（尽量避免）
  a.Add(b)                             // 加法
  a.Sub(b)                             // 减法
  a.Mul(b)                             // 乘法
  a.Div(b)                             // 除法
  a.Cmp(b)                             // 比较（-1/0/1）
  a.IsNegative()                       // 是否为负
  a.StringFixed(8)                     // 固定 8 位小数输出
```

## 4. 挑战3：分布式环境下的幂等性

> **严重程度：P0 致命**
> 问题本质：网络超时/重试/回调重发导致同一请求被执行多次，如何保证只生效一次

### 4.1 挑战描述

```
典型灾难场景：

场景1 — 充值回调重复：
  三方支付回调通知"充值成功"
  第一次回调：CreditWallet +5000 → 余额 15000 ✅
  网络超时，三方没收到 200 OK
  第二次回调：CreditWallet +5000 → 余额 20000 ❌ 重复上账！

场景2 — RPC 超时重试：
  财务服务调用 CreditWallet 超时（但实际已执行成功）
  财务重试：再次调用 CreditWallet → 再次上账 ❌

场景3 — 人工加款审核：
  审核员点击"通过"后页面未刷新
  再次点击"通过" → 又调了一次 CreditWallet ❌
```

### 4.2 主流方案

#### 方案A：唯一约束 + 幂等键（推荐）

```
使用者：支付宝、微信支付、Stripe、PayPal
原理：为每次操作生成唯一幂等键，通过数据库 UNIQUE 约束保证只写入一次

实现层次：
  第一层（快速返回）：查 wallet_transaction 是否已存在相同 (biz_type, order_no, wallet_type)
    存在 → 直接返回成功（幂等响应）
  第二层（最终防线）：INSERT wallet_transaction 时 UNIQUE 约束
    冲突 → 并发重复，忽略唯一冲突错误返回成功

幂等键构成：UNIQUE(biz_type, order_no, wallet_type)
  示例："deposit" + "R20260227143052123456" + 1  → 充值到中心钱包

Stripe 官方实践：
  Stripe API 支持 Idempotency-Key header
  相同 key 的请求返回缓存的第一次响应
  key 有 24 小时有效期

优点：
  ✅ 数据库层面保证，无论应用层如何重试都安全
  ✅ 实现简单明确
  ✅ 无额外基础设施依赖
  ✅ wallet_transaction 流水表天然就需要这些字段

缺点：
  ⚠️ 需要调用方传递 biz_type + order_no（增加接口契约）
  ⚠️ UNIQUE 索引有微小的写入性能开销
```

#### 方案B：状态机幂等

```
使用者：所有有状态的业务系统
原理：通过订单状态机检查，已终态的操作直接返回成功

实现：
  充值回调：deposit_order.status ≠ 待支付 → 返回成功（已处理过）
  确认扣款：withdraw_order.status = 已完成 → 返回成功
  解冻操作：withdraw_order.status = 已驳回 → 返回成功

优点：
  ✅ 逻辑直觉清晰
  ✅ 不需要额外的幂等键字段

缺点：
  ⚠️ 只能防住"状态已变更"的重复，无法防住并发的首次重复
  ⚠️ 必须配合方案A使用，不能单独作为幂等方案
```

### 4.3 可选方案

#### 方案C：Redis 幂等缓存

```
使用者：作为辅助加速层
原理：用 Redis SETNX 记录已处理的请求ID，TTL 自动过期

实现：
  key = "idempotent:{biz_type}:{order_no}:{wallet_type}"
  if redis.SetNX(key, "1", 24h) == false {
    return success  // 已处理
  }
  // 执行业务逻辑
  // 失败时 DEL key

优点：
  ✅ 查询速度快（微秒级），可减少数据库查询
缺点：
  ⚠️ Redis 重启后缓存丢失，不是最终保障
  ⚠️ 需要与数据库 UNIQUE 配合使用
```

#### 方案D：Token 令牌机制

```
使用者：前端防重提交场景
原理：服务端生成一次性 token，提交时校验并消费

优点：
  ✅ 可精确控制前端重复提交
缺点：
  ⚠️ 只防前端重复，不防后端 RPC 重复
  ⚠️ 增加一次 HTTP 往返
```

### 4.4 本项目最佳实践

```
✅ 采用方案A + B 组合：两层幂等保障

第一层（事务外，0 锁竞争）：
  → 状态机快速检查：已终态的重复请求快速返回

第二层（事务内，最终防线）：
  → wallet_transaction 表 UNIQUE(biz_type, order_no, wallet_type)
  → 并发的首次重复请求由数据库拦截

为什么不需要方案C（Redis 缓存层）：
  - 状态机检查已经是快速路径，避免了大部分无效事务
  - UNIQUE 约束是最终保障，不依赖 Redis 可靠性
  - 减少 Redis 操作环节，简化架构

为什么不需要方案D（Token 令牌）：
  - RPC 接口的幂等由方案A+B保证
  - 前端重复提交由 Redis 分布式锁拦截（见挑战17）
```

## 5. 挑战4：跨服务数据一致性

> **严重程度：P1 严重**
> 问题本质：财务服务调用钱包 CreditWallet 时跨服务调用，如何保证两侧数据一致

### 5.1 挑战描述

```
跨服务一致性困境：

财务服务                          钱包服务
    │                                │
    ├── 收到回调，更新订单为"已支付"    │
    ├── RPC 调用 CreditWallet ──────▶│── 上账成功 ✅
    ├── 等待响应...                   │
    │   ← 网络超时（但钱包已上账）     │
    ├── 标记为"上账失败" ❌            │
    │                                │
    └── 数据不一致：钱包已加钱，但财务认为失败

或者反向：
    ├── RPC 调用 CreditWallet ──────▶│── 上账成功 ✅
    ├── 收到成功响应                   │
    ├── 更新本地订单状态...           │
    │   ← 本地事务失败 ❌             │
    │                                │
    └── 数据不一致：钱包已加钱，但财务订单还是"待上账"
```

### 5.2 主流方案

#### 方案A：同步 RPC + 幂等重试（推荐）

```
使用者：支付宝/微信的回调上账、Stripe Webhooks
原理：调用方失败后通过定时任务重试，被调用方保证幂等

流程：
  1. 财务收到回调 → 同步 RPC 调钱包 CreditWallet
  2. 成功 → 更新财务订单状态为"已上账"
  3. 失败/超时 → 财务订单保持"已支付待上账"
  4. 财务定时任务扫描"已支付待上账"订单 → 重新调 CreditWallet
  5. 钱包幂等保证 → 重复调用不会重复上账

核心保障：
  - 钱包侧幂等：即使被调100次，只会执行1次
  - 财务侧重试：保证最终一定会调成功
  - 最终一致性：可能有短暂的状态不一致，但最终一定一致

优点：
  ✅ 实现简单，无额外基础设施
  ✅ 业界最主流的跨服务一致性方案
  ✅ 支付宝/微信/Stripe 的回调+重试就是这个模式

缺点：
  ⚠️ 有短暂的不一致窗口（定时任务间隔）
  ⚠️ 需要财务侧实现重试逻辑（财务方的责任）
```

#### 方案B：本地消息表（Outbox Pattern）

```
使用者：大型金融系统、银行间转账
原理：调用方把"待执行的操作"先写入本地消息表，再由后台任务投递

流程：
  1. 财务在本地事务中：更新订单状态 + 写入 outbox 表
  2. 后台任务扫描 outbox → RPC 调钱包 → 成功后标记已投递

优点：
  ✅ 本地事务保证"状态+消息"原子性
缺点：
  ⚠️ 需要额外的 outbox 表和扫描任务
  ⚠️ 增加实现复杂度
  ⚠️ 对于当前场景是过度设计
```

### 5.3 可选方案

#### 方案C：TCC（Try-Confirm-Cancel）

```
使用者：分布式事务框架（Seata、DTM）
原理：Try冻结 → Confirm确认 → Cancel回滚

优点：
  ✅ 强一致性
缺点：
  ⚠️ 实现复杂度极高（每个操作需正向/反向两套逻辑）
  ⚠️ 需要分布式事务框架
  ⚠️ 当前项目完全不需要
```

#### 方案D：Saga 模式

```
使用者：长流程编排（订单→支付→物流→库存）
原理：多个本地事务+补偿事务，失败时逆序补偿

优点：
  ✅ 适合长链路编排
缺点：
  ⚠️ 补偿逻辑复杂
  ⚠️ 钱包上账是简单的点对点调用，Saga 过度
```

### 5.4 本项目最佳实践

```
✅ 采用方案A：同步 RPC + 幂等重试

理由：
1. 钱包与财务的交互模式简单：充值上账 / 提现冻结 / 确认扣款 / 解冻
2. 每个操作都有明确的幂等键 (biz_type + order_no + wallet_type)
3. 支付宝/微信/Stripe 的亿级日交易量也是这个模式
4. 方案B/C/D 在当前场景都是过度设计

钱包侧的职责（我们负责）：
  ✅ 保证所有 RPC 接口幂等
  ✅ 正确返回成功/失败状态码

财务侧的职责（对方负责）：
  ✅ 实现重试机制（定时任务扫描异常订单）
  ✅ 正确传递 biz_type + order_no
```

## 6. 挑战5：冻结-确认/解冻的原子性

> **严重程度：P0 致命**
> 问题本质：提现涉及冻结→等待审核→确认扣款/解冻三个阶段，如何保证每个阶段的原子性

### 6.1 挑战描述

```
提现冻结的三阶段挑战：

阶段1 — 冻结可能跨两个钱包：
  用户提现 1500 THB，中心钱包余额 1200，奖励钱包 500
  需要冻结：中心 1200 + 奖励 300 = 1500
  问题：冻中心成功但冻奖励失败 → 中心的 1200 被悬挂冻结

阶段2 — 解冻必须精确还原：
  审核驳回 → 需要解冻中心 1200 + 奖励 300
  问题：如果不记录每个钱包冻了多少，解冻时无法正确分配

阶段3 — 确认扣款的并发安全：
  出款成功回调 → DebitFrozen
  问题：回调可能重复，如何保证只扣一次？
```

### 6.2 主流方案

#### 方案A：冻结金额直接记录在订单 + 同事务操作（推荐）

```
使用者：大多数C端钱包系统
原理：
  - withdraw_order.frozen_amount 记录该订单总冻结金额
  - wallet.frozen 字段记录该钱包当前总冻结金额
  - 冻结/解冻/确认扣款都在同一事务内操作 wallet + 流水

实现要点：
  冻结事务：
    BEGIN;
    SELECT wallet WHERE user_id=? AND wallet_type=1 FOR UPDATE;  -- 锁中心钱包
    -- 检查可用余额 = balance - frozen
    UPDATE wallet SET frozen = frozen + ? WHERE wallet_type=1;
    INSERT INTO wallet_transaction (biz_type='withdraw_freeze', ...);
    INSERT INTO withdraw_order (frozen_amount=?, status=1, ...);
    COMMIT;

  解冻事务（审核驳回）：
    BEGIN;
    SELECT withdraw_order WHERE order_no=? FOR UPDATE;  -- 状态机检查
    -- status 必须是"审核中"才能解冻
    SELECT wallet WHERE ... FOR UPDATE;
    UPDATE wallet SET frozen = frozen - ?;
    INSERT INTO wallet_transaction (biz_type='withdraw_unfreeze', ...);
    UPDATE withdraw_order SET status=7;  -- 已驳回
    COMMIT;

  确认扣款事务（出款成功）：
    BEGIN;
    SELECT withdraw_order WHERE order_no=? FOR UPDATE;  -- 状态机检查
    SELECT wallet WHERE ... FOR UPDATE;
    UPDATE wallet SET balance = balance - ?, frozen = frozen - ?;
    INSERT INTO wallet_transaction (biz_type='withdraw', ...);
    UPDATE withdraw_order SET status=4;  -- 已完成
    COMMIT;

优点：
  ✅ 同事务保证原子性
  ✅ withdraw_order.frozen_amount 记录冻结详情
  ✅ 状态机防止重复操作
  ✅ 无需额外的冻结明细表（简化设计）

缺点：
  ⚠️ 如果提现涉及多钱包冻结，需要在订单中记录分配明细
```

#### 方案B：独立冻结明细表

```
使用者：支付宝/微信提现等涉及多钱包冻结的场景
原理：创建独立的 freeze_detail 表记录每个钱包的冻结分配

优点：
  ✅ 精确追踪每个钱包的冻结金额
  ✅ 适合复杂的跨钱包冻结场景

缺点：
  ⚠️ 多了一张表的 IO 开销
  ⚠️ 增加实现复杂度
```

### 6.3 本项目最佳实践

```
✅ 采用方案A：冻结金额记录在订单 + 同事务操作

理由：
1. 当前需求中提现只从中心钱包冻结（需求文档规则4：提现方式由充值来源决定）
2. 不涉及跨多钱包冻结的复杂场景（单钱包冻结）
3. withdraw_order.frozen_amount 足以记录冻结金额
4. 避免过早引入 freeze_detail 表增加复杂度

预留路径：
  如果未来需求变更需要跨钱包冻结 → 引入 freeze_detail 表
  当前架构设计不阻碍后续扩展

关键实现要点：
1. 冻结/解冻/确认都在同一事务内完成
2. withdraw_order.status 状态机防重复操作
3. wallet.frozen 字段独立于 balance（冻结不扣 balance）
4. 可提现余额 = balance - frozen（实时计算，不落字段）
```

## 7. 挑战6：缓存与数据库的一致性

> **严重程度：P1 严重**
> 问题本质：引入 Redis 缓存后，如何保证缓存与 DB 的数据一致

### 7.1 挑战描述

```
经典缓存不一致场景：

场景1 — 写后更新的竞态：
  线程1：事务提交（余额 1000→1500）
  线程2：事务提交（余额 1500→1800）
  线程2：SET cache = 1800
  线程1：SET cache = 1500    ← 旧值覆盖新值！

场景2 — 先删缓存再更新DB：
  线程1：DEL cache
  线程2：读缓存 miss → 读 DB（旧值 1000）→ SET cache = 1000
  线程1：UPDATE DB → 1500
  缓存停留在旧值 1000 ← 不一致！

场景3 — 先更新DB再删缓存（推荐方案的最坏情况）：
  线程1：UPDATE DB → 1500
  线程1：DEL cache（此时崩溃）
  缓存仍是旧值 ← 不一致（但 TTL 会自动修正）
```

### 7.2 主流方案

#### 方案A：Cache-Aside + 写后删除（推荐）

```
使用者：支付宝/微信/阿里巴巴/Facebook
原理：
  读：先读缓存 → 未命中则读DB → 写入缓存（带TTL）
  写：先更新DB（事务内）→ 事务提交后删除缓存

读余额：
  1. GET wallet:bal:{uid}:{currency}:{wallet_type}
  2. 命中 → 返回
  3. 未命中 → SELECT FROM wallet → SET 缓存（TTL 5min）→ 返回

写余额（CreditWallet/DebitWallet）：
  1. BEGIN → UPDATE wallet → INSERT wallet_transaction → COMMIT
  2. DEL wallet:bal:{uid}:{currency}:{wallet_type}  ← 事务提交后

为什么是"删"而不是"更新"：
  并发写时：两个线程都 DEL，下次读从 DB 取最新值（永远正确）
  如果是"更新"：线程乱序可能旧值覆盖新值

优点：
  ✅ 实现简单，最常用
  ✅ 并发安全（最差情况 TTL 后自动修正）
  ✅ 数据库始终是权威数据源

缺点：
  ⚠️ DEL 与事务提交之间有极短窗口可能读到旧缓存
  ⚠️ DEL 失败时需等 TTL 过期
```

#### 方案B：Write-Through（写穿透）

```
使用者：低写频率的配置类数据
原理：写操作同时更新 DB 和缓存

适用：币种配置、汇率数据（由定时任务统一更新）

优点：
  ✅ 缓存始终与 DB 同步（针对单写者场景）
缺点：
  ⚠️ 并发写时存在竞态
  ⚠️ 不适合余额这类高写频率数据
```

### 7.3 可选方案

#### 方案C：延迟双删

```
原理：删缓存 → 更新DB → 延迟N毫秒 → 再删一次缓存
优点：✅ 解决"先删后更新"的竞态问题
缺点：⚠️ 增加延迟，Cache-Aside + TTL 已足够
```

#### 方案D：订阅 Binlog 异步更新缓存

```
原理：监听 TiDB Binlog/CDC，变更后自动更新缓存
优点：✅ 解耦，业务代码不管缓存
缺点：⚠️ 重度基础设施依赖（CDC），当前规模不需要
```

### 7.4 本项目最佳实践

```
✅ 余额类数据：方案A Cache-Aside + 写后删除
✅ 配置类数据：方案B Write-Through（定时任务直接写 Redis）

各数据类型的缓存策略：

| 数据           | 策略          | TTL   | 更新方式            |
|---------------|---------------|-------|-------------------|
| 用户余额       | Cache-Aside   | 5min  | 写操作后 DEL        |
| 币种列表       | Cache-Aside   | 1h    | 配置变更后 DEL      |
|               | + 进程内缓存   | 10min | sync.Map 本地缓存   |
| 汇率数据       | Write-Through | 自然  | 定时任务直接写 Redis |
| 用户当前币种   | Cache-Aside   | 7d    | 切换时 SET          |
| 提现账户列表   | Cache-Aside   | 30min | 新增/删除后 DEL     |

DEL 失败的兜底：TTL 过期后缓存自然消失，下次读重新从 DB 加载
最差影响：用户看到的余额延迟最多 5 分钟更新（安全性不受影响，因为写操作走 DB）
```

## 8. 挑战7：汇率实时性与计算精度

> **严重程度：P2 重要**
> 问题本质：汇率是浮动的，如何保证获取的及时性、计算的精确性、以及"用户看到的汇率"="实际成交的汇率"

### 8.1 挑战描述

```
挑战维度：

1. 实时性：汇率每分钟都在变，但不能每次操作都调三方 API
2. 可靠性：三方 API 可能超时/返回异常值，不能影响业务
3. 精度：汇率计算涉及乘除，精度丢失可能导致偏差
4. 一致性：用户在充值/兑换页面看到的汇率，到提交时可能已变化
5. 偏差检测：三方汇率可能突然跳变（市场波动或API异常），需要告警
```

### 8.2 主流方案

#### 方案A：定时拉取 + 多源聚合 + 偏差检测（推荐）

```
使用者：Grab Financial、OVO、支付宝国际版
原理：定时从多个三方 API 拉取汇率，聚合后更新到平台

流程：
  ┌──────────┐  ┌──────────┐  ┌──────────┐
  │ API源-1   │  │ API源-2   │  │ API源-3   │
  └────┬─────┘  └────┬─────┘  └────┬─────┘
       │  并发请求(5s超时)  │              │
       └─────────┬─────────┘──────────────┘
                 ▼
        ┌────────────────┐
        │  聚合规则：      │
        │  排除离群值      │
        │  取有效值平均值  │
        └────────┬───────┘
                 ▼
        ┌────────────────────────────────┐
        │  偏差检测：                      │
        │  |新-旧|/旧×100 > 阈值 → 更新平台汇率 + 记日志 │
        │  正常 → 仅更新实时汇率           │
        └────────┬───────────────────────┘
                 ▼
        ┌────────────────────────────────┐
        │  计算入/出款汇率（不持久化）     │
        │  入款 = 实时汇率 × (1+浮动%)    │
        │  出款 = 实时汇率 × (1-浮动%)    │
        └────────────────────────────────┘

优点：
  ✅ 多源冗余，单个 API 挂不影响
  ✅ 偏差检测防止异常汇率影响业务
  ✅ 入/出款汇率分离，平台可控利差

缺点：
  ⚠️ 定时拉取有延迟（3-5分钟）
  ⚠️ 需要选择可靠的三方 API 源
```

#### 方案B：实时查询三方 API

```
使用者：外汇交易平台（对实时性要求极高）
优点：✅ 汇率最新
缺点：
  ⚠️ 每次操作增加外部 HTTP 调用
  ⚠️ 三方不可用时业务中断
  ⚠️ 互联网钱包不是外汇交易，不需要这么高的实时性
```

### 8.3 汇率计算精度方案

```
使用 shopspring/decimal 进行汇率计算：

示例：兑换 5,000,000 VND → BSB（汇率：1 BSB = 2,300 VND）

  rate := decimal.NewFromString("2300")     // 1 BSB = 2300 VND
  vndAmount := decimal.NewFromString("5000000")
  bsbAmount := vndAmount.Div(rate)           // 5000000 / 2300 = 2173.91304347...
  bsbRounded := bsbAmount.Round(8)           // 2173.91304347（保留8位）

  // 入款汇率计算
  floatRate := decimal.NewFromString("2.00")  // 浮动 2%
  depositRate := rate.Mul(decimal.NewFromString("1").Add(floatRate.Div(decimal.NewFromString("100"))))
  // depositRate = 2300 × 1.02 = 2346

精度保证：
  ✅ shopspring/decimal 使用任意精度十进制运算
  ✅ 除法结果保留足够小数位
  ✅ 汇率快照在订单中记录（exchange_rate 字段），事后可追溯
```

### 8.4 汇率一致性方案

```
用户看到的汇率 vs 实际成交的汇率：

方案：乐观展示 + 事务内重查
  1. 用户进入兑换页面 → 读缓存中的汇率展示
  2. 用户提交兑换 → 事务内重新读 DB 最新汇率
  3. 如果汇率变化超过阈值（如 0.5%）→ 返回"汇率已变化，请重新确认"
  4. 用户确认后以最新汇率成交
  5. 成交汇率快照存入订单记录

这样既保证了用户体验（不用每次刷新汇率），又保证了成交精度。
```

### 8.5 本项目最佳实践

```
✅ 定时拉取（3分钟一次）+ 多源聚合（3个三方 API）+ 偏差检测
✅ shopspring/decimal 全程精确计算
✅ 除法结果保留 8 位小数（DECIMAL(20,8) 统一精度）
✅ 入/出款汇率 = 实时汇率 ×（1 ± 浮动百分比），不持久化
✅ 提交时事务内重查最新汇率
✅ 成交汇率快照存入订单表 exchange_rate 字段
✅ 平台汇率更新时自动记录 currency_rate_log
```

## 10. 挑战9：死锁预防与事务性能

> **严重程度：P1 严重**
> 问题本质：多行加锁操作可能产生死锁，事务持锁时间过长会降低并发

### 10.1 挑战描述

```
死锁场景复现：

兑换操作需要同时锁两行钱包（VND中心 + BSB中心 + BSB奖励）：

  事务1（兑换 VND→BSB）：
    SELECT wallet WHERE currency=VND AND wallet_type=1 FOR UPDATE;  ← 锁VND中心
    SELECT wallet WHERE currency=BSB AND wallet_type=1 FOR UPDATE;  ← 等待BSB中心

  事务2（其他涉及BSB的操作）：
    SELECT wallet WHERE currency=BSB AND wallet_type=1 FOR UPDATE;  ← 锁BSB中心
    SELECT wallet WHERE currency=VND AND wallet_type=1 FOR UPDATE;  ← 等待VND中心

  → 死锁！事务1等事务2释放BSB，事务2等事务1释放VND
```

### 10.2 主流方案

#### 方案A：固定加锁顺序（推荐）

```
使用者：所有涉及多行锁的数据库应用（教科书级方案）
原理：所有事务按相同的全局顺序加锁，消除循环等待

规则：按 (currency_code ASC, wallet_type ASC) 排序后依次加锁

示例：兑换 VND → BSB
  先锁 BSB（字母序 B < V）中心(1)
  再锁 BSB 奖励(2)
  最后锁 VND 中心(1)

实现：
  func lockWalletsInOrder(ctx context.Context, tx *query.Query, keys []WalletKey) ([]*model.Wallet, error) {
      // 按 (currency_code, wallet_type) 排序
      sort.Slice(keys, func(i, j int) bool {
          if keys[i].CurrencyCode != keys[j].CurrencyCode {
              return keys[i].CurrencyCode < keys[j].CurrencyCode
          }
          return keys[i].WalletType < keys[j].WalletType
      })
      var wallets []*model.Wallet
      for _, k := range keys {
          w, err := tx.Wallet.WithContext(ctx).
              Where(tx.Wallet.UserID.Eq(k.UserID)).
              Where(tx.Wallet.CurrencyCode.Eq(k.CurrencyCode)).
              Where(tx.Wallet.WalletType.Eq(k.WalletType)).
              Where(tx.Wallet.DeleteAt.Eq(0)).
              ForUpdate().First()
          if err != nil { return nil, err }
          wallets = append(wallets, w)
      }
      return wallets, nil
  }

优点：
  ✅ 完全消除死锁可能
  ✅ 零额外开销
  ✅ 通过封装函数强制所有调用方遵守

缺点：
  ⚠️ 需要团队所有人遵守此规范（通过封装函数强制）
```

#### 方案B：死锁检测 + 自动重试

```
使用者：TiDB/MySQL 内置机制
原理：数据库检测到死锁后自动终止一个事务，应用层捕获后重试

TiDB 内置：检测到死锁返回错误码 1213 (ER_LOCK_DEADLOCK)
应用层：捕获此错误自动重试一次

优点：✅ 兜底方案
缺点：⚠️ 重试增加延迟，应作为兜底而非主策略
```

### 10.3 事务性能优化方案

```
核心原则：最小化锁持有时间

✅ 事务外做的事（无锁）：
  - 参数校验
  - 幂等键预检查（状态机快速路径）
  - 汇率查询（进程内缓存）
  - 金额计算（shopspring/decimal）
  - 订单号生成

✅ 事务内做的事（持锁）：
  - SELECT FOR UPDATE（加锁）
  - UPDATE wallet（更新余额/冻结）
  - INSERT wallet_transaction（写流水）
  - INSERT/UPDATE 业务订单表
  - COMMIT（释放锁）

❌ 绝对不在事务内做的事：
  - 外部 HTTP 调用（三方 API）
  - 跨服务 RPC 调用
  - Redis 操作（缓存删除放在事务提交后）
  - 文件 IO / 大量日志输出

锁持有时间目标：
  正常操作 < 10ms（1-2次 UPDATE + 1次 INSERT）
  兑换操作 < 20ms（2-3次 UPDATE + 2-3次 INSERT + 1次 INSERT 兑换记录）
```

### 10.4 本项目最佳实践

```
✅ 方案A（固定加锁顺序）作为主策略：封装 lockWalletsInOrder，所有多行操作统一使用
✅ 方案B（死锁检测）作为兜底：捕获 1213 错误码自动重试一次
✅ 事务内只做 DB 操作，所有非 DB 操作移到事务外
```

## 11. 挑战10：防超扣与资金安全校验

> **严重程度：P0 致命**
> 问题本质：余额扣减后不能为负，冻结金额不能超过可用余额，这些校验必须绝对可靠

### 11.1 挑战描述

```
超扣灾难场景：

场景1 — 余额为负：
  用户余额 1000，同时提现 800 和投注扣款 500
  如果并发控制有漏洞 → 余额变为 -300 → 资金损失

场景2 — 冻结超额：
  用户余额 1000，冻结 700（提现A），再冻结 500（提现B）
  可用余额 = 1000 - 700 = 300，但第二笔尝试冻结 500 → 应该拒绝

场景3 — 流水不一致：
  balance_before = 1000, amount = 500, balance_after = 1600
  1000 + 500 ≠ 1600 → 审计发现异常但已经晚了
```

### 11.2 主流方案

#### 方案A：多层防御体系（推荐）

```
使用者：所有正规资金系统

第一层 — 代码层校验（事务内，FOR UPDATE 锁保护下）：
  // 扣款前检查
  available := wallet.Balance.Sub(wallet.Frozen)
  if available.LessThan(amount) {
      return errors.New("余额不足")
  }

  // 冻结前检查
  available := wallet.Balance.Sub(wallet.Frozen)
  if available.LessThan(freezeAmount) {
      return errors.New("可用余额不足")
  }

  // 更新后防御性检查
  newBalance := wallet.Balance.Sub(amount)
  if newBalance.IsNegative() {
      return errors.New("余额异常：计算结果为负")  // 不应该触发，触发说明有 bug
  }

第二层 — 流水一致性校验（事务内）：
  txn := &WalletTransaction{
      BalanceBefore: wallet.Balance,
      Amount:        amount,
      BalanceAfter:  wallet.Balance.Add(amount),  // 入账
  }
  // 断言：before ± amount == after
  if direction == Credit && !txn.BalanceBefore.Add(txn.Amount).Equal(txn.BalanceAfter) {
      panic("流水一致性校验失败")  // 代码 bug，必须立即发现
  }

第三层 — 数据库约束（可选增强）：
  -- TiDB 5.0+ 支持 CHECK 约束
  ALTER TABLE wallet ADD CONSTRAINT chk_balance CHECK (balance >= 0);
  ALTER TABLE wallet ADD CONSTRAINT chk_frozen CHECK (frozen >= 0);

第四层 — 定期对账（运营级）：
  每日凌晨：SUM(所有入账流水) - SUM(所有出账流水) == 当前余额
  不等 → 告警 → 人工介入（不自动修正）

第五层 — wallet_transaction UNIQUE 索引兜底：
  同一笔业务绝不重复写流水（防幂等失败导致的重复入账）
```

### 11.3 本项目最佳实践

```
✅ 采用方案A：五层防御体系

必须实现：
  ✅ 第一层：代码层余额非负校验（事务内，FOR UPDATE 锁保护）
  ✅ 第二层：流水 before ± amount = after 一致性断言
  ✅ 第五层：wallet_transaction UNIQUE(biz_type, order_no, wallet_type)

建议实现：
  □ 第三层：TiDB CHECK 约束（需确认版本支持，不支持则跳过）
  □ 第四层：每日对账定时任务（可二期补充）

关键代码原则：
  - 所有扣款操作必须先查余额再扣款，不允许"盲扣"
  - 余额、冻结金额的修改必须在 FOR UPDATE 锁内
  - 余额计算结果为负时，不是返回错误而是 panic（说明有代码 bug）
```

## 14. 挑战13：审计追溯与对账

> **严重程度：P2 重要**
> 问题本质：所有资金操作必须可追溯，余额与流水必须可验证一致

### 14.1 挑战描述

```
审计追溯需求：
  1. 任何时刻的余额变动都能查到"谁在什么时候因为什么操作了多少钱"
  2. 当前余额 = 初始余额 + SUM(所有入账) - SUM(所有出账)
  3. 每条流水的 balance_before ± amount = balance_after 必须成立
  4. 异常时能快速定位哪条流水出了问题
```

### 14.2 主流方案

#### 方案A：Append-Only 流水 + before/after 快照（推荐）

```
使用者：支付宝/微信/Stripe
原理：每条流水记录变动前后的余额快照，形成可验证的链

wallet_transaction 设计：
  id | user_id | amount | balance_before | balance_after | biz_type | order_no | create_at

验证链：
  Txn-1: before=0,        amount=+5000, after=5000      ✅ 0+5000=5000
  Txn-2: before=5000,     amount=-2000, after=3000      ✅ 5000-2000=3000
  Txn-3: before=3000,     amount=+1000, after=4000      ✅ 3000+1000=4000
  当前余额 = 4000 = 最后一条流水的 balance_after          ✅

对账 SQL：
  -- 验证当前余额与流水汇总一致
  SELECT w.user_id, w.currency_code, w.wallet_type,
         w.balance AS current_balance,
         COALESCE(SUM(CASE WHEN t.direction=1 THEN t.amount ELSE -t.amount END), 0) AS calculated_balance
  FROM wallet w
  LEFT JOIN wallet_transaction t ON w.user_id = t.user_id
    AND w.currency_code = t.currency_code
    AND w.wallet_type = t.wallet_type
  GROUP BY w.id
  HAVING w.balance != calculated_balance;

核心规则：
  wallet_transaction 表 —— 只 INSERT，永不 UPDATE，永不 DELETE
  这是资金审计的基石原则
  修正通过新增反向流水实现（如人工减款对应一条 manual_debit 流水）
```

### 14.3 本项目最佳实践

```
✅ 方案A：Append-Only 流水 + before/after 快照

必须保证：
  1. wallet_transaction 只追加，永不修改删除
  2. 每条流水都有 balance_before 和 balance_after
  3. 事务内断言 before ± amount = after
  4. 所有B端写操作通过 ser-blog 上报审计日志
```

> **严重程度：P2 重要**
> 问题本质：wallet_transaction 只追加不删除，数据量会持续增长，如何保证查询性能

### 15.1 挑战描述

```
增长预估：

  C端：用户查自己的充值/提现/兑换记录（分页）

  对账：按日期范围汇总（全表扫描风险）
```

### 15.2 主流方案

#### 方案A：合理索引 + 游标分页（当前适用）


#### 方案A：分级降级策略（推荐）

```
| 三方汇率全不可用 | 保持最后一次成功的汇率，记录告警 | 汇率可能不是最新 |
```

```
使用者：Netflix Hystrix、阿里 Sentinel
```

```
不做熔断器的理由：
  - 调用链简单（钱包→财务/KYC/Blog）
```

### 20.3 Thrift IDL 金额传递方案

```
Thrift 没有 decimal 类型，金额通过 string 传递：

  struct CreditWalletReq {

      2: required string currencyCode
      3: required i32 walletType
```

```
✅ 方案A：GORM Gen FieldType 直接映射 decimal.Decimal
✅ Thrift IDL 金额字段统一用 string 类型
✅ JSON 响应金额用 string 传递 + 额外 display 字段
```

## 21. 挑战20：多钱包类型的优先级扣款

> **严重程度：P1 严重**
> 问题本质：投注扣款需要先扣中心钱包、不足部分扣奖励钱包，如何保证原子性和正确性

### 21.1 挑战描述

```
优先级扣款场景：

用户投注 1500 BSB：
  中心钱包余额 = 1000 BSB
  奖励钱包余额 = 800 BSB

扣款逻辑：
  先扣中心 1000（扣完）
  再扣奖励 500（还剩300）
  合计扣 1500 ✅

挑战：
  1. 需要同时锁两行钱包（原子性）
  2. 需要在两个钱包分别产生流水（审计）
  3. 返奖时需要按扣款时的比例分别返回对应钱包
  4. 流水表唯一索引需要支持同一 order_no 多个 wallet_type
```

### 21.2 主流方案

#### 方案A：同事务多钱包扣款 + 比例记录（推荐）

```
使用者：竞技游戏平台、虚拟经济系统
原理：在同一事务内按优先级依次扣款，记录各钱包的扣款比例

实现：
  func (s *AccountSer) BetDebit(ctx context.Context, params BetDebitParams) (*BetDebitResult, error) {
      return query.Q.Transaction(func(tx *query.Query) error {
          // Step1: 按固定顺序加锁（中心→奖励）
          centerWallet := lockWallet(tx, uid, currency, WalletTypeCenter)
          bonusWallet := lockWallet(tx, uid, currency, WalletTypeBonus)

          // Step2: 计算扣款分配
          centerDebit := decimal.Min(params.Amount, centerWallet.Available())
          bonusDebit := params.Amount.Sub(centerDebit)

          // Step3: 校验总余额
          if bonusDebit.GreaterThan(bonusWallet.Available()) {
              return errs.InsufficientBalance
          }

          // Step4: 分别扣款 + 分别写流水
          debitWallet(tx, centerWallet, centerDebit, "bet_debit", params.OrderNo)
          if bonusDebit.GreaterThan(decimal.Zero) {
              debitWallet(tx, bonusWallet, bonusDebit, "bet_debit", params.OrderNo)
          }

          // Step5: 返回扣款比例（用于后续返奖）
          result.CenterRatio = centerDebit.Div(params.Amount)
          result.BonusRatio = bonusDebit.Div(params.Amount)
          return nil
      })
  }

关键设计：
  - wallet_transaction UNIQUE(biz_type, order_no, wallet_type)
    → 同一笔投注在中心和奖励钱包各产生一条流水，不冲突
  - 返奖时按比例分别入账到中心和奖励钱包
```

### 21.3 本项目最佳实践

```
✅ 方案A：同事务多钱包扣款 + 比例记录

核心要点：
1. 同一事务锁两个钱包（按固定顺序防死锁）
2. 先扣中心、不足补奖励（需求文档规则2）
3. 记录扣款比例 → 返奖时按比例返
4. 唯一索引含 wallet_type → 支持同一 order_no 多条流水

返奖对应规则（需求文档规则）：
  投注 1500 BSB = 中心扣 1000 (66.7%) + 奖励扣 500 (33.3%)
  返奖 2000 BSB = 中心返 1334 (66.7%) + 奖励返 666 (33.3%)
```
