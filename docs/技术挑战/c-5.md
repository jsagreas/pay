# 二、致命级挑战

这类挑战如果处理不当，**直接导致资金损失**。没有商量余地，必须在编码前就确定方案。

---

## 2.1 挑战一：并发余额操作安全

### 问题本质

多个请求同时操作同一用户同一币种的余额时，如果不加控制，会出现**超扣**（余额变负）或**更新丢失**（两笔充值只入账一笔）。

```
经典场景：
  用户余额 = 1000
  请求A（兑换扣减800）和 请求B（提现冻结500）同时到达

  错误做法（先读后写）：
    A读余额=1000, B读余额=1000
    A判断1000>=800 通过, B判断1000>=500 通过
    A写余额=200, B写余额=500
    最终余额=500（A的扣减丢失了）或200（B的冻结丢失了）

  更糟的：
    A写余额=1000-800=200, B写余额=1000-500=500
    实际应该是：1000-800-500=-300 余额不足应该拒绝B
    但两笔都成功了，超扣了300
```

### 方案对比

| 方案 | 原理 | 优势 | 劣势 | 适用场景 |
|------|------|------|------|---------|
| **乐观锁（CAS条件更新）** | `UPDATE SET available=available-X WHERE available>=X` | 无显式锁、并发度高、无死锁、实现简单 | 竞争激烈时失败率高 | 钱包余额操作（推荐） |
| **悲观锁（SELECT FOR UPDATE）** | 先锁行再操作 | 保证读到最新值、支持复杂计算 | 锁持有时间长、死锁风险、并发差 | 需要先读后复杂计算再写 |
| **版本号乐观锁** | `UPDATE SET balance=X, version=version+1 WHERE version=V` | 通用、支持任意修改 | 需要额外version字段、失败需重试 | 通用并发控制 |
| **分布式锁（Redis/ETCD）** | 获取锁 → 操作 → 释放锁 | 跨服务安全 | 复杂、锁超时处理麻烦、性能差 | 跨服务/跨库场景 |
| **队列串行化** | 余额操作全部入队、单线程消费 | 彻底消除并发 | 延迟高、吞吐低、故障恢复复杂 | 对延迟不敏感的批量系统 |

### 业界主流做法

**支付宝/微信支付**等成熟钱包系统的余额操作核心模式：

1. **数据库层CAS条件更新**（最主流）：利用数据库行级锁的原子性，在UPDATE语句中带条件判断
2. **SELECT FOR UPDATE + 事务**（次主流）：需要先读余额做复杂计算的场景
3. **乐观锁 + 有限重试**：高并发下CAS失败后重试2-3次

### 我们的最佳实践

**选择：乐观锁（CAS条件更新），不重试**

```sql
-- 扣减余额：一条SQL原子完成"检查+扣减"
UPDATE user_balance
SET available = available - ?, update_at = ?
WHERE user_id = ? AND currency = ? AND wallet_type = ?
  AND available >= ?  -- 数据库层保证不超扣

-- 检查 affected rows: 0 = 余额不足，1 = 成功
```

**为什么这个方案最适合我们**：

```
1. 钱包的余额操作全是简单的加减
   不需要"先读余额做复杂计算再写"，直接 available ± amount 就够了
   这天然适合CAS条件更新模式

2. InnoDB UPDATE WHERE 内部会加行级排他锁
   但锁只在UPDATE语句执行的瞬间持有，微秒级释放
   远比 SELECT FOR UPDATE（锁到COMMIT）短

3. 不需要重试机制
   余额不足 affected_rows=0 → 直接返回"余额不足"
   不是"竞争失败可以重试"的场景
   充值入账（余额加）不存在失败场景

4. 不需要额外的version字段
   available >= amount 本身就是版本条件
   比version字段更有业务语义——它直接表达"够扣才扣"

5. 不引入分布式锁
   所有余额操作都在同一个数据库同一张表
   InnoDB的行锁就是最高效的"分布式锁"
```

**不选悲观锁的原因**：

```
SELECT FOR UPDATE 需要两条SQL（先SELECT再UPDATE）
事务持有行锁时间 = SELECT到COMMIT的全部时间
如果事务内还有写流水表等操作，锁持有时间更长
同一用户的并发请求全部串行等待

而 CAS UPDATE 一条SQL搞定，锁持有=单条UPDATE执行时间
```

## 2.2 挑战二：重复入账/扣款（幂等性）

### 问题本质

在分布式系统中，RPC调用可能因为以下原因被重复执行：

```
场景1：网络超时重试
  财务调钱包.充值入账 → 钱包执行成功 → 响应在网络中丢失
  财务认为失败 → 重试 → 钱包收到第二次调用
  如果不做幂等：同一笔充值入账两次 = 资损

场景2：渠道回调重发
  第三方渠道多次发送同一笔支付成功通知
  财务每次收到都调钱包入账

场景3：人工补偿操作
  运营人员重复点击"执行"按钮
```

### 方案对比

| 方案 | 原理 | 优势 | 劣势 | 适用场景 |
|------|------|------|------|---------|
| **应用层状态检查** | 先查订单/流水状态，已完成则直接返回 | 简单高效，99%请求在此拦截 | 极端并发可穿透 | 第一层快速拦截 |
| **数据库唯一键** | balance_flow 的 `(ref_order_no, change_type)` UNIQUE KEY | 绝对可靠，无法穿透 | 需要处理唯一键冲突异常 | 兜底防护 |
| **Redis SETNX** | `SET key NX EX ttl`，设置成功才执行 | 快速、支持过期 | Redis宕机或过期后失效 | 接口级去重 |
| **Token令牌机制** | 服务端下发token，客户端带token请求，用完即废 | 前端防重复提交 | 增加一次交互、不防服务间重试 | C端表单提交 |
| **幂等表** | 独立的幂等记录表，每次操作先插入记录 | 通用、可审计 | 额外一张表+一次查询 | 通用幂等框架 |

### 业界主流做法

**支付系统的标准做法是两层幂等**：

1. **第一层：业务状态检查**（快速路径）—— 查订单/流水状态，已完成直接返回
2. **第二层：唯一键约束**（安全兜底）—— 数据库层阻止重复写入

这在蚂蚁金服、Stripe、PayPal 等支付系统中都是标配。

### 我们的最佳实践

**选择：两层幂等（应用层 + 数据库唯一键）**

```
第一层（快速返回，拦截99.9%的重复请求）：

  充值入账方法入口：
    → 用 order_no 查 deposit_order 状态
    → 状态已是"成功" → 直接返回"成功"
    → 不执行任何余额操作

第二层（数据库兜底，拦截0.1%极端并发穿透）：

  balance_flow 表的唯一键：
    UNIQUE KEY uk_order_type (ref_order_no, change_type)

  即使两个并发请求同时穿透了第一层
  写流水时只有一个能成功，另一个触发唯一键冲突
  catch 唯一键冲突异常 → 返回"成功"（幂等语义）
```

**各接口的幂等键**：

```
接口              幂等键              第一层检查点
─────────────────────────────────────────────────
充值入账          平台订单号          deposit_order.status = 成功?
提现确认扣款      提现订单号          withdraw_order.status = 成功?
提现失败解冻      提现订单号          withdraw_order.status = 失败?
人工加款          修正单号            balance_flow 是否已有该单号记录?
人工减款          修正单号            同上
人工补单          补单号              deposit_order.status = 补单成功?
```

**为什么不用 Redis SETNX**：

```
1. 幂等键已经在数据库订单表和流水表中天然存在
   不需要额外引入Redis做幂等

2. Redis有失效风险（过期、宕机、主从切换丢数据）
   对于资金操作，幂等的兜底必须是数据库唯一键——它是持久化的、不会过期的

3. 增加Redis依赖反而增加故障点
```

## 2.3 挑战三：金额精度与溢出

### 问题本质

金融系统最经典的坑：**浮点数精度丢失**和**整数溢出**。

```
浮点精度问题：
  0.1 + 0.2 = 0.30000000000000004（IEEE 754浮点标准）
  如果用 float64 存余额，充值 0.1 + 0.2 后余额不是 0.3
  积累下来每笔交易丢几分钱 → 累计成资损

整数溢出问题：
  int64 最大值 = 9,223,372,036,854,775,807（约 9.2 × 10^18）
  VND 最小单位=1，1亿VND = 100,000,000
  即使用户有10亿VND余额 = 1,000,000,000
  距离 int64 上限还有 9 × 10^9 倍空间 → 实际不会溢出

  但要注意：SUM聚合时多用户余额累加可能逼近上限 → 实际也够用
```

### 方案对比

| 方案 | 原理 | 优势 | 劣势 | 适用场景 |
|------|------|------|------|---------|
| **BIGINT 最小单位整数** | 金额×10^精度后存为整数 | 零精度丢失、运算快、存储省 | 需要出入参做精度换算 | 大部分支付系统（推荐） |
| **DECIMAL(M,N)** | 数据库精确定点数 | 直接存读、无需换算 | 跨语言处理麻烦、Go无原生类型 | 传统银行系统 |
| **shopspring/decimal** | Go第三方精确小数库 | Go层面精确运算 | 性能比int64慢10-100倍、GORM需自定义映射 | 需要高精度小数计算 |
| **字符串存储** | 金额以字符串形式存储和传输 | 无精度问题 | 无法做数据库聚合运算、比较复杂 | API传输层（非存储层） |

### 业界主流做法

**全球主流支付系统几乎都用整数存储金额**：

- **Stripe**：所有金额以最小货币单位整数表示（如美元用分）
- **支付宝**：内部金额全部用分（long类型）
- **Shopify**：money_in_cents 字段名直白表达
- **ISO 4217**：国际货币标准定义了每种货币的最小单位

### 我们的最佳实践

**选择：BIGINT（int64）存最小单位整数**

```
存储规则（由 currency_config.precision 决定换算）：

  BSB   precision=2  →  10.50 BSB   存为  1050
  VND   precision=0  →  1,000,000 VND 存为 1000000
  USDT  precision=2  →  100.50 USDT  存为 10050
  IDR   precision=0  →  500,000 IDR  存为 500000

入参处理（C端/RPC传入展示金额 → 转为最小单位）：
  amount_internal = amount_display × 10^precision

出参处理（最小单位 → 展示金额）：
  amount_display = amount_internal / 10^precision

汇率计算（全程整数）：
  rate 也用 BIGINT 存储
  rate=230000000, rate_precision=4 → 实际汇率=23000
  兑换计算：to_amount = from_amount × 10^rate_precision / rate
```

**精度换算集中管理**：

```
设计要求：精度换算逻辑必须集中在一处（如 utils 层）
  不允许业务代码各处散落换算逻辑
  所有换算都通过统一函数：
    ToMinUnit(displayAmount, precision) → int64
    ToDisplayAmount(minUnit, precision) → string

这样即使精度规则调整，只改一处
```

**溢出防护（防御性编程）**：

```go
// 加法溢出检查（虽然实际不太可能触发，但防御性编程）
func SafeAdd(a, b int64) (int64, error) {
    if b > 0 && a > math.MaxInt64-b {
        return 0, errors.New("overflow")
    }
    if b < 0 && a < math.MinInt64-b {
        return 0, errors.New("underflow")
    }
    return a + b, nil
}
```

**为什么不用 DECIMAL**：

```
1. Go 没有原生 decimal 类型
   GORM 读 DECIMAL 列 → 需要用 shopspring/decimal 或 string 接收
   引入额外依赖，且 GORM Gen 自动生成的模型需要手动调整

2. 现有工程所有数值类型都是 int64
   ID、时间戳、统计量 → 全部 int64
   金额也用 int64，风格一致，不需要特殊处理

3. 数据库层 BIGINT 运算比 DECIMAL 快
   UPDATE SET available = available - ? 整数减法
   vs UPDATE SET available = available - ? DECIMAL减法
   差距虽小但在高并发下有意义

4. VND/IDR 这种"大数法币"天然无小数
   1 VND = 1 最小单位，BIGINT 完美匹配
```

# 三、严重级挑战

这类挑战如果处理不当，**可能导致资金卡死、系统不可用或数据不一致**。

---

## 3.1 挑战四：跨模块分布式一致性

### 问题本质

钱包和财务是两个独立服务、独立数据库。某些操作需要两个服务协同完成，但没有全局事务保证原子性。

```
最经典的场景——提现：

  Step 1: 钱包冻结余额（钱包DB操作 ✓）
  Step 2: 调财务.提交审核（RPC调用）

  如果 Step 2 失败了？
    → 用户的余额已经被冻结了
    → 但财务那边没有收到审核请求
    → 用户的钱"卡住了"——冻结了但没人审核
    → 必须补偿：解冻余额

  如果 Step 2 超时了（不知道成功还是失败）？
    → 更棘手：不能贸然解冻（万一财务其实收到了）
    → 也不能不管（万一财务确实没收到）
```

### 方案对比

| 方案 | 原理 | 优势 | 劣势 | 适用场景 |
|------|------|------|------|---------|
| **本地补偿（Saga模式简化版）** | 正向操作失败后执行逆向补偿 | 简单直接、无框架依赖 | 补偿也可能失败、需人工兜底 | 步骤少、补偿逻辑明确（推荐） |
| **Saga框架（Seata/DTM）** | 框架管理长事务的正向和补偿 | 自动编排补偿、可视化 | 引入重框架、学习成本高、运维复杂 | 跨多服务多步骤的复杂事务 |
| **TCC（Try-Confirm-Cancel）** | 预留资源→确认→取消三阶段 | 隔离性好 | 每个操作要写三套代码、侵入性强 | 资源预留型场景 |
| **本地消息表** | 写本地消息表→异步投递→下游消费 | 最终一致、可靠 | 延迟高、需要消息中间件、需消费者幂等 | 跨服务异步操作 |
| **事务消息（RocketMQ）** | 半消息→本地事务→确认→消费 | 强一致的异步方案 | 依赖特定MQ、复杂度高 | 大规模分布式事务 |
| **人工补单兜底** | 异常后人工核实+手动操作 | 最简单、成本最低 | 依赖人、有延迟 | 极低频异常场景 |

### 业界主流做法

**中小规模支付系统的标准做法**：

1. **本地补偿 + 人工兜底**（最主流）—— 绝大多数创业期/成长期支付系统
2. **Saga编排**（次主流）—— 大厂且跨多个服务多步骤时
3. **TCC**（特定场景）—— 高隔离性要求的金融核心

**支付宝早期也是本地补偿 + 定时对账 + 人工处理**，规模大了才引入分布式事务框架。

### 我们的最佳实践

**选择：本地补偿 + 人工补单兜底**

逐场景分析我们面临的跨模块一致性问题及补偿方案：

```
场景1：充值创建 → 调财务.创建渠道订单失败
  影响：本地订单已创建（待支付），但没有渠道订单
  补偿：本地订单标记失败，返回C端错误
  资金影响：无。充值是先支付后入账，此时没动过余额
  难度：★☆☆（简单，try-catch即可）

场景2：提现创建 → 冻结余额成功 → 调财务.提交审核失败
  影响：余额已冻结，但财务没收到审核请求，用户钱卡住
  补偿：立即解冻余额 + 本地订单标记失败 + 写回退流水
  资金影响：如果解冻成功则无影响
  如果解冻也失败（DB异常）：记日志告警，人工介入
  难度：★★☆（中等，需要在代码中处理补偿逻辑）

场景3：提现创建 → 冻结成功 → 调财务超时（不确定成功/失败）
  影响：不知道财务是否收到
  补偿策略：
    方案A（简单）：视为失败，解冻余额。财务如果实际收到了，
      它回调钱包时发现订单已失败，走异常处理
    方案B（严谨）：不立即处理，定时任务扫描"冻结超过N分钟
      但财务侧无对应记录"的订单，批量解冻
  推荐：方案A，简单优先。极端情况走人工补单

场景4：财务回调.充值入账 → 钱包DB异常
  影响：渠道已到账，但钱包没入账
  补偿：财务侧记录异常状态，运营通过"人工补单"功能补录
  这是需求文档明确设计的补单场景（订单号 B+16位）
  难度：★☆☆（靠产品功能兜底）
```

**为什么不引入 Seata/DTM**：

```
1. 我们的跨模块场景只有两个服务（钱包和财务），步骤最多3步
   Seata/DTM 是为5个以上服务、10步以上的复杂编排设计的
   杀鸡用牛刀，引入的复杂度远大于收益

2. 每个场景的补偿逻辑都很明确
   场景1：标记失败（一行代码）
   场景2：解冻余额（反向调一次DB）
   场景4：人工补单（产品功能已有）
   用try-catch + defer就能搞定，不需要框架

3. 引入Seata/DTM需要额外的TC（事务协调器）服务
   增加部署组件、运维成本、故障点
   现有技术栈没有，从0引入学习成本高

4. 当前团队规模和业务规模不需要
   "用人工能处理的异常" > "用框架自动处理的复杂度"
   这是务实的工程决策
```

## 3.2 挑战五：事务边界与连接池耗尽

### 问题本质

数据库事务是有代价的：**事务越长，锁持有时间越久，连接占用时间越长**。如果在事务内做了RPC调用，RPC超时3秒意味着数据库连接和行锁被占用3秒。

```
错误示范：

  BEGIN TRANSACTION
    → UPDATE balance（行锁住了）
    → RPC调用财务（等待3秒超时...）  ← 灾难！
    → 写流水
  COMMIT

  问题：
    - 行锁持有3秒 → 同一用户的其他操作全部排队等待
    - 数据库连接占用3秒 → GORM连接池默认10个连接
    - 10个并发请求全卡在RPC → 连接池耗尽 → 后续请求全部超时

  雪崩效应：
    RPC超时 → 连接池耗尽 → 所有请求超时 → 系统不可用
```

### 方案对比

| 方案 | 原理 | 优势 | 劣势 |
|------|------|------|------|
| **事务外做RPC，事务内只做DB操作** | 严格分离IO和事务 | 事务时间极短、连接快速释放 | 需要设计好数据预获取 |
| **事务超时设置** | 给事务设最大执行时间 | 防止无限持有连接 | 超时后事务回滚，需要重试 |
| **连接池监控 + 合理配置** | 监控连接使用率、调大连接池 | 提高并发容量 | 治标不治本 |

### 业界主流做法

**所有成熟支付系统的铁律：事务内不做IO（RPC、HTTP、Redis、日志打印等）。**

这不是某个团队的约定，而是数据库使用的基本原则。阿里中间件团队的开发规约、Google的代码审查标准都明确禁止"事务内远程调用"。

### 我们的最佳实践

**选择：严格遵循"RPC在事务外，DB操作在事务内"铁律**

```
以兑换为例（标准模式）：

  ┌── 事务之外（可以慢，不影响DB） ────────────────┐
  │ 1. [RPC] 获取最新汇率                          │
  │ 2. 计算兑换金额、赠送金额                       │
  │ 3. 预检查余额是否充足（快速判断，非精确校验）      │
  └────────────────────────────────────────────────┘
       │ 所有数据准备好
       ▼
  ┌── 事务之内（必须快，毫秒级完成） ──────────────┐
  │ 4. UPDATE 扣减源币种余额（乐观锁）              │
  │ 5. UPDATE 增加BSB余额                          │
  │ 6. UPDATE 增加BSB奖励余额（赠送）               │
  │ 7. INSERT 兑换订单                              │
  │ 8. INSERT 余额变动流水（3条）                    │
  └────────────────────────────────────────────────┘

  事务内 5 条 SQL，纯本地DB操作
  TiDB 执行时间预估：1-5ms
  连接占用时间 ≈ 5ms → 连接池利用率极高
```

**代码层面的保证**：

```
规则1：Rep 层方法不调 RPC
  所有 RPC 调用在 Service 层完成
  Rep 层只接收数据、执行 DB 操作
  这是架构分层自然保证的

规则2：事务方法的参数只接收值类型
  func (r *BalanceRep) ExecuteChanges(tx *query.Query, changes []BalanceChange) error
  changes 里全是算好的值，不需要再查外部

规则3：缓存操作也在事务外
  B端修改币种 → DB更新成功 → COMMIT → 然后再删Redis缓存
  defer + goroutine 异步删缓存
```

**连接池配置建议**：

```
参考现有工程的GORM初始化配置：
  MaxOpenConns = 20      → 最大连接数（单机20足够，扛千级QPS）
  MaxIdleConns = 10      → 空闲连接数
  ConnMaxLifetime = 1h   → 连接最大存活时间
  ConnMaxIdleTime = 10m  → 空闲连接最大存活时间
```

## 3.3 挑战六：账务一致性与对账

### 问题本质

钱包系统最核心的质量要求：**余额与流水必须严格一致**。任何时刻，用户的当前余额必须等于其所有流水变动的累计结果。

```
一致性要求：
  user_balance.available = SUM(balance_flow.amount) WHERE wallet_type=X AND currency=Y

  如果不等，说明存在以下可能：
  - 余额更新成功但流水写入失败（代码bug）
  - 流水重复写入（幂等失效）
  - 有余额操作绕过了流水记录（代码bug或人为篡改）
  - 并发问题导致的更新丢失
```

### 方案对比

| 方案 | 原理 | 优势 | 劣势 | 适用场景 |
|------|------|------|------|---------|
| **事务内同步记流水** | 余额变动和流水写入在同一个事务 | 强一致、最可靠 | 事务稍长 | 核心方案（推荐） |
| **异步记流水（消息队列）** | 余额变动成功后发消息，异步写流水 | 事务更短 | 消息丢失/延迟导致不一致 | 非金融场景 |
| **流水连续性校验** | 每条流水记录变动前后余额 | 可随时验证、发现异常 | 只能发现问题，不能预防 | 对账手段 |
| **定时对账任务** | 批量比对余额和流水 | 发现长尾问题 | 事后发现、有延迟 | 补充手段 |
| **余额快照 + 增量校验** | 每天记快照，快照+当天流水=新快照 | 可逐天回溯 | 额外存储、增加系统复杂度 | 大规模系统 |

### 业界主流做法

1. **事务内同步记流水**（基础保证，所有支付系统必做）
2. **流水连续性字段**（before_balance / after_balance，大部分支付系统有）
3. **T+1 对账**（定时任务，比对当天余额变动 vs 流水记录 vs 订单状态）

### 我们的最佳实践

**选择：事务内同步 + 流水连续性字段 + 后续可选定时对账**

**核心保证——事务原子性**：

```
每一次余额变动，以下操作在同一个数据库事务中完成：
  1. UPDATE user_balance（余额变动）
  2. INSERT balance_flow（流水记录，含 before_balance / after_balance）
  3. UPDATE *_order（订单状态更新，如有）

三者要么全部成功，要么全部回滚
不存在"余额改了但流水没写"的中间状态
```

**连续性校验能力（内建）**：

```
balance_flow 表的 before_balance 和 after_balance 字段提供自校验能力：

  flow_N.after_balance == flow_N+1.before_balance

  如果不等 → 说明有异常（漏记流水、重复记录、并发问题）

注意：
  这个校验需要限定 user_id + currency + wallet_type
  同一个用户同一个子钱包的流水应该是连续的
```

**before_balance 怎么获取**：

```
方案A（推荐）：在事务内，先读当前余额 → 记为 before_balance → 执行UPDATE →
  after_balance = before_balance + amount（或 before_balance - amount）

方案B：UPDATE 使用 RETURNING 子句（MySQL 不支持，TiDB 部分支持）

方案C：先 UPDATE 成功后，SELECT 读取更新后的余额 → 反推 before
  after_balance = 当前余额, before_balance = 当前余额 - amount

推荐方案A：事务内先SELECT再UPDATE，虽然多一次查询，
  但在乐观锁模式下（事务极短），性能影响可忽略
```

**不选异步记流水的原因**：

```
1. 资金流水绝对不能丢
   消息队列存在消息丢失的可能（即使有ACK机制）
   一旦丢了一条流水，对账就永远对不上

2. 同步写入的性能够用
   多写一条 INSERT balance_flow → 一次索引写入
   在 TiDB 上增加约 1-2ms 延迟
   对于钱包操作的响应时间（50-200ms），这个增量可以忽略

3. 简单可靠 > 性能极致
   这是金融系统的基本价值观
```

这类挑战**不会直接导致资损，但影响系统性能、用户体验或安全合规**。

## 4.1 挑战七：热点账户（高并发单用户）

### 问题本质

同一用户在短时间内发起大量操作（或多个外部服务同时回调操作同一用户余额），导致该用户的余额行成为"热点行"，行级锁竞争严重。

```
场景：
  用户A的余额行被以下操作同时争抢：
  - 充值入账回调（财务发起）
  - 兑换操作（用户发起）
  - 人工加款（运营发起）
  - 另一笔充值入账回调

  所有操作都要 UPDATE user_balance WHERE user_id=A
  InnoDB 行锁串行执行 → 排队等待

实际评估：
  对于泛娱乐钱包平台，单用户并发操作的概率较低
  不像证券交易所（同一只股票万人同时下单）
  但仍需考虑：运营批量操作、活动期间集中充值
```

### 方案对比

| 方案 | 原理 | 优势 | 劣势 | 适用场景 |
|------|------|------|------|---------|
| **不特殊处理（当前规模够用）** | InnoDB行锁自然串行 | 零额外复杂度 | 高并发下延迟增加 | 万级DAU（推荐当前阶段） |
| **余额分桶（子账户拆分）** | 一个用户余额拆成N个桶，分散锁竞争 | 大幅降低热点 | 查总余额需SUM、扣减需选桶、复杂度极高 | 超高频交易（证券/期货） |
| **异步合并（批量处理）** | 短时间内多笔操作合并为一笔 | 减少DB写入次数 | 延迟增加、需要中间状态管理 | 电商秒杀库存扣减 |
| **内存账本 + 异步持久化** | 余额在内存中操作，定时刷DB | 极高吞吐 | 宕机丢数据、一致性弱 | 游戏虚拟货币 |
| **队列排序** | 同一用户操作入队串行处理 | 有序、无冲突 | 增加延迟、需要队列基础设施 | 对顺序敏感的场景 |

### 业界主流做法

- **大部分中小支付系统**：不特殊处理，InnoDB行锁够用
- **支付宝（大规模）**：余额分桶 + 异步合并（双十一级别才需要）
- **游戏虚拟钱包**：内存账本 + 异步持久化（因为虚拟货币允许少量误差）

### 我们的最佳实践

**选择：当前阶段不特殊处理，架构上预留演进空间**

```
当前规模评估：
  - 数万级DAU
  - 单用户并发余额操作概率 < 1%
  - InnoDB行锁串行执行，单次UPDATE约1-5ms
  - 同一用户10笔并发操作排队：最长等待50ms
  - 完全在可接受范围内

什么时候需要升级：
  - 单用户每秒 > 100次余额操作（代理商结算场景）
  - 批量操作导致单行锁等待 > 1秒

架构预留：
  - user_balance 表的唯一键设计（user_id + currency + wallet_type）
    天然支持后续按user_id分桶（增加bucket_id字段即可扩展）
  - 统一余额变动操作的抽象设计
    后续改为分桶时只需修改 Rep 层，Service 层无感知
```

## 4.2 挑战八：缓存与DB数据一致性

### 问题本质

币种配置使用 Redis 缓存加速，但缓存和DB之间存在数据不一致窗口。B端修改了币种配置，缓存中还是旧数据。

```
经典场景：
  1. B端运营修改 VND 充值最低限额：1000 → 5000
  2. 更新DB成功
  3. 删除Redis缓存（异步执行中...）
  4. 此时C端用户查询币种配置 → 缓存未命中 → 查DB → 拿到新值 ✓

  竞态条件：
  1. B端修改 DB（新值=5000）
  2. C端查询 → 缓存命中（旧值=1000）✗ 这个窗口期使用了旧限额
  3. B端删除缓存
  4. 后续C端查询 → 缓存未命中 → 查DB → 拿到新值 ✓

  窗口期 = 步骤1到步骤3之间的时间（通常毫秒到秒级）
```

### 方案对比

| 方案 | 原理 | 一致性窗口 | 复杂度 | 适用场景 |
|------|------|-----------|--------|---------|
| **Cache-Aside（旁路缓存）** | 读：先缓存后DB；写：先DB后删缓存 | TTL时间内 | 低 | 读多写少（推荐） |
| **Write-Through（写穿透）** | 写操作同时更新DB和缓存 | 近零 | 中 | 写频率可控 |
| **Write-Behind（写回）** | 写操作只更新缓存，异步刷DB | 取决于刷盘频率 | 高 | 写频繁、容忍最终一致 |
| **发布/订阅（Pub/Sub）** | DB变更后发消息，所有节点刷缓存 | 消息延迟 | 高 | 多节点缓存同步 |
| **双删策略** | 先删缓存→更新DB→延迟再删一次缓存 | 极短 | 中 | 对一致性敏感 |

### 业界主流做法

**Cache-Aside（旁路缓存）是绝对主流**，Facebook 发表的论文、Redis 官方文档、阿里技术规范都推荐这种模式。

对于"读多写极少"的配置数据（币种配置变更频率=天级），TTL 5分钟的不一致窗口完全可以接受。

### 我们的最佳实践

**选择：Cache-Aside + 主动删缓存 + 短TTL**

```
读操作流程：
  1. 先查 Redis → 命中则反序列化返回
  2. 未命中 → 查 DB → 序列化写入 Redis（TTL=5分钟）→ 返回

写操作流程（B端修改币种配置/汇率）：
  1. 更新 DB
  2. 异步删除 Redis 缓存（defer + goroutine）
     → 参照 ser-item 的 ClearAllItemCache 模式
  3. 下次读时自动重建缓存

缓存Key设计（遵循现有 redis_key 规范）：
  wallet:currency:list              → 可用币种列表 JSON
  wallet:currency:detail:{code}     → 单个币种配置 JSON
  wallet:rate:{code}                → 单个币种汇率

TTL设计：
  币种列表/详情 → 5分钟（变更频率=天级）
  汇率 → 1-3分钟（变更频率=小时级，比币种配置略频繁）
```

**关键原则——余额数据绝不缓存**：

```
user_balance 表不做任何缓存。

理由：
  1. 余额是高频写数据，缓存与DB几乎永远不一致
  2. 用户看到过时余额（如充值后余额没变）→ 投诉/恐慌
  3. 余额查询走唯一键索引，TiDB 查询速度 < 1ms
  4. 缓存带来的性能收益远不如它带来的一致性风险

同理：订单状态、流水记录也不缓存
```

## 4.3 挑战九：敏感数据安全

### 问题本质

钱包系统涉及多种敏感数据：银行卡号、电子钱包ID、USDT地址、用户余额等。如果数据泄露或被篡改，后果严重。

```
需要保护的数据：
  1. 银行卡号（PCI DSS 合规要求加密存储）
  2. 电子钱包ID
  3. USDT钱包地址
  4. KYC信息（从用户服务获取后不应持久化）
  5. 余额数据（防止未授权查看）
```

### 方案对比

| 方案 | 原理 | 优势 | 劣势 | 适用场景 |
|------|------|------|------|---------|
| **应用层AES加密** | 写DB前加密，读DB后解密 | 简单、已有工具 | 加密字段无法做DB层模糊查询 | 银行卡号/钱包ID（推荐） |
| **数据库透明加密（TDE）** | 数据库层面加密存储文件 | 无代码改动 | 不防内部SQL查询泄露 | 磁盘级防护 |
| **列级加密（Vault）** | 独立密钥管理服务 | 密钥隔离、审计 | 增加依赖和延迟 | 大型金融系统 |
| **脱敏展示** | 展示时隐藏中间位 | 用户端安全 | 不防后端泄露 | API返回层 |

### 我们的最佳实践

**选择：应用层AES加密（复用现有工具）+ 脱敏展示**

```
加密存储（写入时）：
  withdraw_account.account_no → AES加密后存储

  工具：common/pkg/utils 已提供 AES 加密/解密函数
  密钥：从 ETCD /slg/conf/security/sensitive_data_key 读取

  加密后数据格式：Base64编码的密文
  查询方式：通过 user_id+currency+method 唯一键查询，不需要按account_no查询

脱敏展示（API返回时）：
  银行卡号：6222****1234（保留前4后4）
  电子钱包ID：09****7890（保留前2后4）
  USDT地址：TN7S****Ks3j（保留前4后4）

其他安全措施：
  1. SQL注入防护：GORM Gen 类型安全查询，不拼接SQL字符串
  2. C端用户只能操作自己的余额：tracer.GetUserId(ctx) 获取当前用户ID
  3. B端操作RBAC鉴权：BackAuthMiddleware + CheckUrlAuth
  4. 审计日志：所有B端写操作调 rpc.BLogClient().AddActLog()
```

## 4.4 挑战十：防刷与风控

### 问题本质

钱包系统面临多种欺诈和滥用风险：

```
风险场景：
  1. 重复转账：同一用户短时间发起大量相同金额充值（可能是洗钱/刷单）
  2. 提现滥用：大量小额提现绕过单笔限额
  3. 汇率套利：利用汇率更新间隙快速兑换获利
  4. 账户盗用：盗取他人账户发起提现
  5. 自动化攻击：脚本批量发起交易
```

### 方案对比

| 方案 | 原理 | 优势 | 劣势 | 适用场景 |
|------|------|------|------|---------|
| **业务规则硬编码** | 限额、重复检测等规则写在代码/配置中 | 简单直接 | 不灵活 | MVP阶段（推荐） |
| **规则引擎（Drools/GENGINE）** | 可配置的规则评估 | 灵活、可热更新 | 引入额外框架 | 规则频繁变更 |
| **实时风控系统** | 独立的风控服务，实时评估交易风险 | 专业、完善 | 架构复杂度极高 | 大型金融平台 |
| **频率限制（Rate Limiting）** | 限制单位时间内的操作次数 | 有效防自动化攻击 | 可能误伤正常用户 | API层面防护 |

### 我们的最佳实践

**选择：业务规则硬编码 + 需求明确的三级拦截**

```
1. 充值重复转账三级拦截（需求文档5-5明确要求）：

  查询：同用户 + 同币种 + 同金额 + 状态=处理中 的充值订单数

  0笔 → 通过
  1笔 → 返回提醒码，C端弹窗确认"您有一笔相同金额订单正在处理，是否继续？"
  2笔 → 返回警告码，建议修改金额
  ≥3笔 → 拦截，不允许继续

2. 提现限额校验：

  单笔限额：currency_config.withdraw_min/max
  每日限额：
    SELECT COALESCE(SUM(amount), 0) FROM withdraw_order
    WHERE user_id=? AND currency=? AND status IN (1,2)
    AND DATE(FROM_UNIXTIME(create_at/1000)) = CURDATE()

    今日已提现 + 本笔 > withdraw_daily → 拦截

3. KYC 后端强制校验：

  法币提现必须通过KYC
  不依赖C端拦截，后端创建提现订单时独立校验
  [RPC] → 用户服务.KYC状态查询 → 未认证 → 拦截

4. 户名一致性校验：

  提现账户户名 vs KYC认证姓名
  后端比对，不一致 → 拦截

5. 提现方式源头过滤（合规要求）：

  只充过法币 → 只能法币提现
  只充过USDT → 只能USDT提现
  法币+USDT → 只保留法币提现
  "资金从哪进就从哪出"

后续可演进：
  - 短时间大量充值/提现告警（ETCD 开关 + 计数器）
  - 异常IP/设备检测（复用现有 BlackCheck 中间件扩展）
  - 提现白名单（特殊用户绕过限制，需求提了但细节待定）
```

# 五、关注级挑战

这类挑战**当前阶段影响有限，但需要知道并预留演进空间**。

---

## 5.1 挑战十一：高可用与降级策略

### 问题本质

ser-wallet 依赖多个外部服务（财务、用户、Redis、TiDB）。任何一个不可用，都可能影响钱包功能。

### 方案对比与最佳实践

```
依赖不可用时的降级策略：

┌──────────────────────────────────────────────────────────────────────┐
│ 依赖           │ 不可用影响                  │ 降级策略              │
├──────────────────────────────────────────────────────────────────────┤
│ TiDB           │ 全部功能不可用              │ 无法降级，告警等恢复   │
│ Redis          │ 币种配置查询变慢（回源DB）   │ 自动回源DB，不影响功能 │
│ 财务RPC        │ 充值/提现创建失败           │ 返回"暂不可用"，       │
│                │                            │ 兑换/查询正常          │
│ 用户服务RPC    │ KYC校验失败                │ 提现不可用，其他正常   │
│ ETCD           │ 服务发现/配置不可用         │ 使用本地缓存续命       │
└──────────────────────────────────────────────────────────────────────┘
```

**关键设计原则**：

```
1. 功能独立降级，不互相牵连
   财务不可用 → 只影响充值/提现，不影响余额查询/兑换/记录查询
   这是单服务多模块的天然优势

2. RPC调用设超时，不无限等待
   现有工程 RPC 客户端默认超时 3 秒
   超时直接返回业务错误，不阻塞用户

3. Redis 降级为回源DB查询
   缓存不可用时，直接查 DB
   币种配置表数据量极小（<10条），查DB非常快

   实现：读缓存失败时 catch 错误 → 走DB → 不写缓存（Redis不可用）

4. 功能开关
   /slg/switch/wallet/deposit/enable
   /slg/switch/wallet/withdraw/enable
   /slg/switch/wallet/exchange/enable
   紧急情况下可通过ETCD关闭特定功能
```

## 5.2 挑战十二：数据增长与存储治理

### 问题本质

随着业务发展，balance_flow（流水表）和订单表会持续增长，影响查询性能。

### 数据量预估

```
假设：万级DAU，每日交易操作预估

  充值：1000笔/天 × 1条流水 = 1000条
  兑换：2000笔/天 × 3条流水 = 6000条
  提现：500笔/天 × 2条流水 = 1000条
  人工修正：50笔/天 × 1条流水 = 50条

  日增流水：约 8000 条
  月增流水：约 24 万条
  年增流水：约 300 万条

  日增订单：约 3550 条
  年增订单：约 130 万条

TiDB 单表千万级以内无需分表
按当前增速，3年不需要考虑分表
```

### 后续可选策略（不做，只做认知储备）

```
如果 balance_flow 超过千万行：
  → 按月分表：balance_flow_202603, balance_flow_202604
  → C端记录查询加时间范围限制（只查最近3个月）
  → 历史数据归档到冷存储

如果 user_balance 超过千万行（用户量爆发）：
  → 按 user_id 分表（mod 64 或 mod 256）
  → TiDB 原生支持 Range/Hash 分区，改动最小

当前阶段：不需要。TiDB 本身支持水平扩展，在百万级数据量下性能良好。
```

```
┌────────────────────────────────────────────────────────────────┐
│                                                                │
│  并发安全(2.1) ──→ 事务边界(3.2) ──→ 账务一致性(3.3)           │

│     │                                                          │
│     └──→ 跨模块一致性(3.1) ──→ 降级策略(5.1)                    │
│                                                                │
│  金额精度(2.3) ──→ 账务一致性(3.3) ──→ 对账(3.3)                │
```

# 七、已确认的架构决策与挑战的映射

以下是前序架构设计文档中已经做出的决策，以及它们分别解决了哪些挑战：

| 架构决策 | 解决的挑战 | 决策依据 |
|----------|-----------|---------|
| 乐观锁 CAS 更新 | 并发安全 | 简单高效、无死锁、现有工程无需额外组件 |
| BIGINT 最小单位整数 | 金额精度 | 零精度丢失、整数运算快、工程风格一致 |
| 双层幂等（应用层+DB唯一键） | 重复入账/扣款 | 99.9%快速拦截 + 0.1%数据库兜底 |
| 本地补偿 + 人工补单 | 跨模块一致性 | 场景简单、不需要分布式事务框架 |
| RPC在事务外、DB在事务内 | 事务边界/连接池 | 事务极短、连接快速释放 |
| 事务内同步写流水 | 账务一致性 | 强一致、不丢流水 |
| Cache-Aside + 短TTL | 缓存一致性 | 读多写少、不一致窗口可接受 |
| 余额不缓存 | 缓存一致性 | 资金数据必须实时准确 |
| 应用层AES加密 | 敏感数据安全 | 复用现有加密工具 |
| 单服务单库 | 高可用/运维 | 不过度设计、部署简单 |
| 不做分库分表 | 数据增长 | 当前规模不需要、TiDB原生扩展 |

# 八、技术方案验证路径

在编码实现时，建议按以下顺序验证各个挑战的方案是否有效：

```
验证阶段1：兑换功能（最简单的交易链路）
  验证点：
    ✓ 并发安全：同一用户同时兑换两笔
    ✓ 金额精度：VND大数 → BSB小数的换算正确性
    ✓ 事务原子性：扣减+增加+流水在同一事务
    ✓ 账务一致性：余额 == SUM(流水)
    ✓ before/after balance 连续性
  方法：写单元测试 + 并发压测脚本

验证阶段2：RPC暴露接口（入账/扣款/解冻）
  验证点：
    ✓ 幂等：同一请求调两次结果一致
    ✓ 并发幂等：同一请求两个goroutine同时调
    ✓ 唯一键冲突处理
  方法：写单元测试 + mock财务调用

验证阶段3：充值全链路
  验证点：
    ✓ 跨模块一致性：mock财务失败时订单状态正确
    ✓ 入账幂等：模拟财务重复回调
  方法：集成测试 + mock

验证阶段4：提现全链路
  验证点：
    ✓ 冻结+补偿：mock财务审核RPC失败后解冻
    ✓ 冻结→扣款：冻结金额正确扣除
    ✓ 冻结→解冻：冻结金额正确退回
    ✓ 限额/KYC校验
  方法：集成测试 + mock
```

# 九、总结：挑战清单与方案速查表

| # | 挑战 | 级别 | 方案选择 | 主流/可选 | 我们选的理由 |
|---|------|------|---------|----------|------------|
| 1 | 并发余额安全 | 致命 | CAS条件更新 | 主流 | 简单高效无死锁，够用 |
| 2 | 幂等性 | 致命 | 应用层+DB唯一键 | 主流 | 双层防护标配 |
| 3 | 金额精度 | 致命 | BIGINT最小单位 | 主流 | Stripe/支付宝同款 |
| 4 | 跨模块一致性 | 严重 | 本地补偿+人工补单 | 主流 | 场景简单不需框架 |
| 5 | 事务边界 | 严重 | RPC外/DB内铁律 | 主流 | 基本原则 |
| 6 | 账务一致性 | 严重 | 事务同步+before/after | 主流 | 强一致基础保证 |
| 7 | 热点账户 | 重要 | 当前不处理 | 看规模 | 万级DAU不需要 |
| 8 | 缓存一致性 | 重要 | Cache-Aside+主动删 | 主流 | 读多写少经典方案 |
| 9 | 数据安全 | 重要 | AES加密+脱敏 | 主流 | 复用现有工具 |
| 10 | 防刷风控 | 重要 | 业务规则硬编码 | 看阶段 | MVP够用 |
| 11 | 高可用降级 | 关注 | 功能独立降级 | 主流 | 单服务天然隔离 |
| 12 | 数据增长 | 关注 | 不处理，预留空间 | 看规模 | TiDB原生支持扩展 |

**核心结论**：

```
钱包系统的技术挑战确实不少，但市面上每一个挑战都有成熟方案。
我们的技术栈（Go + TiDB + Redis + Kitex）完全具备解决这些挑战的能力。

关键不在于"用什么高大上的技术"，而在于：
  1. 严格遵守几条铁律（乐观锁、事务边界、幂等双层、同步写流水）
  2. 不过度设计（不引入不需要的框架和组件）
  3. 在对的地方谨慎（资金操作）、在对的地方简单（配置管理）
  4. 预留演进空间，但不提前实现
```
