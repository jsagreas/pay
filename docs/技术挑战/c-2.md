# 二、资金安全类挑战（★★★）

## 挑战①：并发余额操作的数据一致性

### 问题描述

```
场景：用户 A 的中心钱包余额为 1000
  · 线程1: 充值回调 → Credit +500
  · 线程2: 消费扣款 → Debit -300

如果没有并发控制:
  线程1 读到 balance=1000, 计算 1000+500=1500
  线程2 读到 balance=1000, 计算 1000-300=700
  线程1 写入 balance=1500
  线程2 写入 balance=700  ← 覆盖了线程1的结果

结果: 余额=700，充值的500元凭空消失 → 资损
正确: 余额应该=1200 (1000+500-300)
```

### 方案对比

```

│              │ SELECT FOR UPDATE   │ WHERE version=?    │ RedLock/SET NX    │ 单账户单队列       │
├──────────────┼────────────────────┼───────────────────┼──────────────────┼───────────────────┤

### ★ 主流方案：数据库行锁（悲观锁）

```
工作原理:
  在同一个事务内:
    1. SELECT * FROM user_wallet WHERE user_id=? AND currency_id=? AND wallet_type=? FOR UPDATE
       → 这条 SQL 会锁定该行，其他事务读这行时会阻塞等待
    2. 执行业务逻辑（余额校验、计算新余额）
    3. UPDATE user_wallet SET balance = balance + ? WHERE id = ?
    4. INSERT INTO wallet_flow ...（记录流水）
    5. COMMIT → 释放锁

并发时的效果:
  线程1: 锁定行 → 读 1000 → 加 500 → 写 1500 → 提交(释放锁)
  线程2: 等待锁...... → 锁定行 → 读 1500 → 减 300 → 写 1200 → 提交(释放锁)
  结果: 1200 ✓ 正确

为什么适合本项目:
  · TiDB 完全兼容 MySQL 的 SELECT FOR UPDATE 语义（官方文档明确支持）
  · 钱包数据在同一个库内，行锁就够了，不需要分布式锁
  · C 端用户不会同时对同一个钱包发起多笔操作（并发度不高）
  · 实现最简单，出错概率最低——钱的事情，可靠比性能重要

行业参考:
  · 支付宝早期核心账务系统（《蚂蚁金服分布式架构金融实践》公开分享）
  · Stripe 内部余额系统（Stripe Engineering Blog 2019）
  · Square Cash 核心账务（强一致性优先策略）
```

### 可选方案 A：乐观锁（CAS）

```
工作原理:
  UPDATE user_wallet SET balance = balance + ?, version = version + 1
  WHERE id = ? AND version = ?
  → 如果 affected_rows = 0 说明被别人改过了，重试

何时升级到乐观锁:
  · 当行锁导致可观测的性能瓶颈（锁等待超 100ms 的 P99）
  · 我们已预留 version 字段，升级时不需要改表结构

注意事项:
  · 需要重试逻辑（通常 3 次），增加代码复杂度
  · 高争用时重试风暴可能更差（反复失败反复重试）
```

### 可选方案 B：应用层排队

```
工作原理:
  对同一个 user_id + currency_id 的写请求，在应用层排队串行处理
  可用 Go channel 或 Redis List 实现

何时考虑:
  · 出现热点账户问题（单账户 QPS > 100）
  · 当前阶段不需要，因为 C 端用户级别的并发不高
```

### 本项目最佳实践

```
阶段一（当前）: 行锁（FOR UPDATE）
  · 简单、可靠、满足当前业务并发量
  · 固定加锁顺序防死锁: (currency_id ASC, wallet_type ASC)

阶段二（性能瓶颈出现后）: 升级为乐观锁（CAS）
  · 利用已预留的 version 字段
  · 只需改 ledger 层内部实现，不影响上层

阶段三（热点账户出现后）: 排队 + 批量合并
  · 极端情况下才需要
```

## 挑战②：幂等性保障

### 问题描述

```
场景: 财务模块调用 ser-wallet 的 Credit 接口给用户充值上分
  · 第 1 次调用: 网络超时，但 ser-wallet 实际已执行成功
  · 财务模块没收到响应，认为失败了，发起重试
  · 第 2 次调用: 如果没有幂等控制 → 重复加款 → 资损

更复杂的场景:
  · 支付通道回调充值成功，回调了两次（通道侧重试）
  · 定时任务补偿时重复执行
  · 用户手速快双击了充值按钮
```

### ★ 主流方案：流水表唯一索引（一石二鸟）

```
核心思想:
  wallet_flow 表本来就是审计需要的，给它加唯一索引 uk_order_flow(order_no, wallet_type, flow_type)
  每次余额变更时，先 INSERT 流水 → 唯一索引冲突 = 已处理过 → 直接返回成功

执行流程（伪代码）:

  func (l *Ledger) Credit(ctx, params) error {
      return l.db.Transaction(func(tx) error {

          // Step 1: 尝试插入流水（幂等检查 + 审计记录，一石二鸟）
          flow := &WalletFlow{
              OrderNo:    params.OrderNo,
              WalletType: params.WalletType,
              FlowType:   params.FlowType,
              Amount:     params.Amount,
              // balance_before/after 暂时为 0，后续补全
          }
          err := tx.Create(flow).Error
          if IsDuplicateEntry(err) {
              // 唯一索引冲突 → 说明这笔操作已经处理过 → 幂等返回成功
              return nil
          }
          if err != nil {
              return err  // 其他错误
          }

          // Step 2: 锁定钱包行
          wallet := tx.Where(...).Set("gorm:query_option", "FOR UPDATE").First(...)

          // Step 3-5: 余额变更 + 补全流水的 before/after
          ...
      })
  }

为什么是最佳实践:
  · 零额外存储: 流水表本来就必须有（审计需求），不需要额外建幂等表
  · 事务原子性: INSERT 流水和 UPDATE 余额在同一个事务中
  · 数据库级保证: 唯一索引是 DB 引擎级别的保证，比应用层判断可靠
  · 重试安全: 调用方可以无脑重试，不会产生副作用

行业参考:
  · 支付宝账务系统："交易流水号 + 账务流水表唯一索引"（蚂蚁金服公开架构分享）
  · 微信支付："商户订单号幂等"（微信支付API文档明确说明）
  · Stripe: "Idempotency Keys"（Stripe API 文档，原理类似但用独立存储）
```

### 可选方案：Redis 前置幂等 + DB 流水表双保险

```
适用场景: 当需要快速拦截（不进 DB 事务就返回）

流程:
  1. 先检查 Redis: SETNX idempotent:{orderNo}:{walletType} TTL=24h
  2. Redis 已存在 → 直接返回成功（快速路径）
  3. Redis 不存在 → 进入 DB 事务 → INSERT 流水（唯一索引兜底）
  4. DB 事务成功 → 确认 Redis key 保留
  5. DB 事务失败 → DEL Redis key

何时引入:
  · 当大量重复请求频繁进入 DB 事务造成性能压力时
  · 当前阶段不需要——直接走 DB 唯一索引即可
```

## 挑战③：金额精度与计算安全

### 问题描述

```
经典问题:
  0.1 + 0.2 = 0.30000000000000004  （IEEE 754 浮点数标准）

在钱包场景中:
  · 用户充值 1000.50 VND，扣手续费 0.5%，计算手续费 = ?
  · 100 VND 兑换 BSB，汇率 1 VND = 0.000043 USDT，1 USDT = 10 BSB
    100 × 0.000043 × 10 = 0.043 BSB? 还是 0.042999...BSB?
  · USDT 精度 8 位: 1 USDT = 100000000 satoshi（最小单位）

如果使用 float64 存储和计算:
  多次计算后误差会累积，最终导致对账不平 → 合规问题 + 用户投诉
```

### 方案对比

```
┌──────────────────┬────────────────────┬───────────────────┬─────────────────────┐
│ 方案              │ 整数最小单位        │ decimal 库         │ 数据库 DECIMAL 类型   │
├──────────────────┼────────────────────┼───────────────────┼─────────────────────┤
│ 存储              │ BIGINT             │ VARCHAR/STRING     │ DECIMAL(M,D)        │
│ 计算方式          │ 纯整数运算          │ 任意精度库运算      │ DB 内精确运算        │
│ 精度              │ 无损（整数无精度问题）│ 无损（任意精度）    │ 无损（固定精度）     │
│ 计算性能          │ ★★★ (整数最快)    │ ★★☆ (略慢)       │ ★★☆ (DB计算慢)    │
│ 实现复杂度        │ ★★☆ (需转换逻辑)  │ ★★☆ (需引入库)   │ ★☆☆ (直接用)       │
│ 跨语言兼容        │ ★★★ (整数通用)    │ ★★☆ (各语言不同)  │ ★★★ (DB标准)      │
│ 谁在用            │ Stripe/支付宝/微信  │ 银行/高精度金融     │ 传统 ERP 系统       │

### ★ 主流方案：BIGINT 整数 + shopspring/decimal 汇率计算

```
核心策略: 存储用整数，计算用 decimal，展示用字符串

第一层 — 余额/金额存储（BIGINT 整数）:
  · 所有金额以该币种的最小货币单位存储
  · VND: decimals=0, 存 50000 → 展示 50,000 VND
  · USDT: decimals=8, 存 100000000 → 展示 1.00000000 USDT
  · BSB: decimals=8, 存 1000000000 → 展示 10.00000000 BSB
  · 转换公式: 展示值 = 数据库值 / 10^decimals

第二层 — 汇率存储与计算（VARCHAR + shopspring/decimal）:
  · 汇率存储: VARCHAR(32)，如 "0.00004312"（避免 DECIMAL 类型的精度上限）
  · 汇率计算:
    rate, _ := decimal.NewFromString("0.00004312")
    amount := decimal.NewFromInt(100)  // 100 VND
    result := amount.Mul(rate)         // 精确乘法
    → 无精度损失

第三层 — IDL 传输（string 类型）:
  · 金额在 RPC/HTTP 传输时使用 string
  · 前端收到 "1000.50" 自行格式化展示
  · 避免 JSON 序列化 float64 时的精度丢失

完整示例 — 100 VND 兑换 BSB:
  1. 读取汇率: platformRate = "0.00004312"（1 VND = 0.00004312 USDT）
  2. 读取锚定: bsbRate = "10"（1 USDT = 10 BSB）
  3. 计算:
     vndAmount = decimal.NewFromInt(100)              // 100 VND
     usdtValue = vndAmount.Mul(parseRate)             // 0.004312 USDT
     bsbValue  = usdtValue.Mul(parseBsbRate)          // 0.04312 BSB
     bsbInt    = bsbValue.Shift(8).IntPart()          // 4312000 (最小单位整数)
  4. 存储: exchange_order.to_amount = 4312000 (BIGINT)

行业参考:
  · Stripe: "All API requests expect amounts to be provided in a currency's smallest unit"
  · 支付宝: 所有金额以"分"为单位传输（CNY 场景）
  · 微信支付: amount 字段单位为"分"，类型 int64
  · shopspring/decimal: Go 生态最流行的精确计算库（GitHub 6k+ stars）
```

### 可选方案：纯 decimal 全链路

```
适用场景: 需要任意精度且性能要求不极端

做法: 存储用 DECIMAL(36,18)，Go 层全用 shopspring/decimal

缺点:
  · DECIMAL 类型在 TiDB 中的运算性能低于 BIGINT
  · ORM 映射需要自定义类型（GORM 原生不直接映射 decimal.Decimal）
  · 跨语言传输时仍然需要转 string

结论: 对本项目来说，BIGINT 整数足够且更简单
```

## 挑战④：跨币种兑换的事务原子性

### 问题描述

```
兑换操作涉及 3 个钱包行的余额变更:
  ① 法币中心钱包: balance -= fromAmount（扣法币）
  ② BSB 中心钱包:  balance += toAmount（加 BSB）
  ③ BSB 奖励钱包:  balance += bonusAmount（加奖金 BSB）

如果不在同一个事务中:
  · ① 成功但 ②③ 失败 → 扣了法币但 BSB 没到账 → 用户资损
  · ①② 成功但 ③ 失败 → 奖金丢失 → 用户投诉
  · ①②③ 部分成功 → 流水记录不完整 → 对账困难

同时还有死锁风险:
  · 请求A: 锁法币行 → 锁BSB中心行 → 锁BSB奖励行
  · 请求B: 锁BSB中心行 → 锁法币行 → ...
  → 两个请求互相等对方释放锁 → 死锁
```

### 方案对比

```
┌──────────────────┬────────────────────┬───────────────────┬─────────────────────┐

### ★ 主流方案：单库本地事务 + 固定加锁顺序

```
为什么可以用本地事务:
  · 钱包所有表（user_wallet / wallet_flow / deposit_order / ...）在同一个 TiDB 库中
  · 兑换操作涉及的 3 个钱包行都在 user_wallet 表中
  · 不跨库不跨服务 → 本地事务就能保证 ACID

固定加锁顺序（防死锁的关键）:
  规则: 按 (currency_id ASC, wallet_type ASC) 顺序加锁

  示例 — VND(id=2) → BSB(id=5) 的兑换:
    锁 ①: user_wallet WHERE user_id=U, currency_id=2, wallet_type=1 FOR UPDATE
    锁 ②: user_wallet WHERE user_id=U, currency_id=5, wallet_type=1 FOR UPDATE
    锁 ③: user_wallet WHERE user_id=U, currency_id=5, wallet_type=2 FOR UPDATE
    → (2,1) < (5,1) < (5,2)  ✓ 升序

  为什么不会死锁:
    · 所有事务都按同一个顺序获取锁
    · 不存在 A 等 B、B 等 A 的循环等待
    · 这是教科书级别的死锁预防策略

事务内部流程:

  BEGIN TRANSACTION:
    1. INSERT 3 条流水（幂等检查：任一条冲突 → 全部已处理 → ROLLBACK + 返回成功）
    2. 按顺序锁定 3 个钱包行
    3. 校验法币余额 ≥ 扣款金额
    4. 执行 3 次 UPDATE
    5. 补全 3 条流水的 balance_before/after
  COMMIT

行业参考:
  · MySQL 官方文档: "If you always acquire locks in a consistent order, deadlocks do not occur"
  · TiDB 官方文档: "TiDB supports pessimistic transactions since v3.0"
  · 支付宝内部规范: "多行更新必须按主键升序加锁"（公开分享）
```

### 可选方案：分步执行 + 补偿（SAGA 模式）

```
适用场景: 如果未来兑换涉及跨服务（如兑换需要调用外部交易所接口）

做法:
  Step 1: 扣法币（本地事务）
  Step 2: 加BSB（本地事务）→ 失败时: 补偿回退 Step 1
  Step 3: 加奖金（本地事务）→ 失败时: 补偿回退 Step 1+2

当前不需要: 因为兑换是纯内部操作，不跨服务，本地事务完全覆盖
```

### ★ 主流方案：定时对账脚本 + balance_before/after 校验

```
对账原理:
  wallet_flow 表每条记录都有 balance_before 和 balance_after
  对账公式: balance_after = balance_before + amount

层级一 — 单条流水自校验:
  遍历所有流水，检查每条: balance_after == balance_before + amount
  如果不等 → 说明流水记录本身有问题

层级二 — 流水链连续性校验:
  同一个 (user_id, currency_id, wallet_type) 下的流水按 create_at 排序
  检查: 第 N 条的 balance_before == 第 N-1 条的 balance_after
  如果不等 → 说明有流水缺失或顺序错误

层级三 — 流水与余额一致性校验:
  当前余额 = user_wallet.balance
  流水推导余额 = 该用户该钱包最后一条流水的 balance_after
  如果不等 → 说明余额被直接修改过（绕过了 ledger 层）

实现方式:
  · 利用项目已有的 XXL-JOB 或 robfig/cron 定时任务框架
  · 每天凌晨低峰期跑批
  · 差异记录写入对账结果表或发送告警

对账脚本伪代码:

  func ReconcileDaily(ctx context.Context, date time.Time) {
      // 获取当天所有有流水变动的钱包
      walletIds := repo.GetActiveWalletIds(date)

      for _, walletId := range walletIds {
          wallet := repo.GetWalletById(walletId)
          lastFlow := repo.GetLastFlow(walletId)

          if wallet.Balance != lastFlow.BalanceAfter {
              // 发现差异 → 记录并告警
              alert.Send("余额不一致", walletId, wallet.Balance, lastFlow.BalanceAfter)
          }
      }
  }

行业参考:
  · 支付宝: "日终对账" + "实时告警"（分层策略）
  · 微信支付: "对账单" API（商户日终对账）
  · Stripe: "Balance Transactions" API（完整流水链用于对账）
```

### 可选方案：ledger 层内置实时校验

```

    上一条流水的 balance_after + 本次 amount == 本次 balance_after

## 挑战⑦：缓存与数据库的一致性

### 问题描述

```
场景:
  1. 用户查余额 → 从 Redis 读到 1000（缓存）
  2. 充值回调 → DB 余额变为 1500 → DEL Redis key
  3. 但 DEL 操作在网络上丢失了（极端情况）
  4. 用户再查余额 → 还是 1000（过期缓存）→ 用户以为充值没成功

更复杂的场景（先删缓存后写DB的并发问题）:
  线程A: DEL cache → （线程B插入）→ UPDATE DB
  线程B: 读 cache miss → 读 DB(旧值) → SET cache(旧值)
  → 缓存永远是旧值（直到 TTL 过期）
```

### ★ 主流方案：Cache-Aside + 短 TTL + 查询不依赖缓存余额

```
策略:

  写路径: 先写 DB → 事务提交后 DEL 缓存 key
  读路径: 先查 Redis → miss 则查 DB → 回填 Redis（TTL=5min）

  关键设计:
    · 余额展示走缓存（快，允许短暂不一致）
    · 余额扣减走 DB（FOR UPDATE 实时读取，不走缓存）
    · 即使缓存不一致，也不会导致资金错误

  为什么短 TTL（5分钟）足够:
    · 充值/提现的结果通过订单状态判断，不依赖余额展示值
    · 即使缓存余额是旧值，用户看到充值订单"已完成"就知道成功了
    · 5 分钟后缓存自动过期，下次读取就是最新值

  极端情况处理:
    · DEL 失败: TTL 兜底，最多 5 分钟不一致
    · 并发写+读: 最坏情况是缓存到旧值，5 分钟后自愈
    · 完全可接受: 这是展示层面的不一致，不是资金层面的不一致

行业参考:
  · Facebook: "Look-Aside Cache Invalidation"（Cache-Aside 的发明者级别实践）
  · Stripe: Dashboard 余额展示允许短暂延迟，扣款永远实时查 DB
  · 微信支付: 商户余额查询走缓存，扣款走数据库
```

### 可选方案：延迟双删

```
流程:
  1. DEL cache
  2. UPDATE DB
  3. sleep(500ms)  // 等待可能的并发读把旧值写回缓存
  4. DEL cache     // 再删一次

优点: 缩小不一致窗口
缺点: 500ms 的延迟怎么确定？不同场景可能不够
结论: 对本项目来说，Cache-Aside + 短 TTL 已经够用，不需要双删
```

## 挑战⑨：RPC 超时与重试策略

### 问题描述

```
ser-wallet 与财务模块的交互是同步 RPC:

充值链路:
  ser-wallet → 调用财务模块的 MatchDepositChannel → 返回匹配的通道

如果 RPC 超时:
  · 场景 1: 请求到达了财务模块，财务模块处理成功了但响应丢失
    → ser-wallet 认为失败 → 用户看到"通道匹配失败"→ 但财务侧已经创建了通道订单
  · 场景 2: 请求没到达财务模块（网络断开）
    → ser-wallet 认为失败 → 安全，财务侧也没做任何事

问题: ser-wallet 无法区分这两种场景

更敏感的场景:
  财务模块 → 调用 ser-wallet 的 Credit → 超时
  → 财务模块不知道 Credit 是否成功 → 重试可能导致重复加款？
  → 不重试可能导致用户充值成功但余额没增加？
```

### 方案对比

### ★ 主流方案：接口幂等 + 指数退避重试

```
核心原则:
  所有涉及资金变更的接口必须幂等 → 调用方可以安全重试

ser-wallet 暴露的 Credit/Debit/Freeze/Unfreeze:
  · 已通过 wallet_flow 唯一索引实现幂等
  · 相同 orderNo + walletType + flowType 重复调用 → 返回成功，不产生副作用
  · 调用方（财务模块）超时后可以放心重试

ser-wallet 调用财务模块（MatchDepositChannel 等）:
  · 财务模块的接口也必须是幂等的（这是联调契约的一部分）
  · 超时后的策略:
    第 1 次重试: 等 100ms
    第 2 次重试: 等 300ms
    第 3 次重试: 等 900ms（指数退避）
    3 次都失败: 返回明确错误给用户（"通道繁忙，请稍后重试"）

重试的安全前提:
  · 只重试幂等接口
  · 只重试网络超时/连接失败/5xx 错误
  · 不重试 4xx 错误（参数错误重试也没用）
  · 不重试"余额不足"等业务错误（重试也不会改变结果）

Go 代码层面的实现:

  // 统一重试包装（方向性伪代码）
  func retryableRPC[T any](ctx context.Context, fn func() (T, error), maxRetry int) (T, error) {
      var result T
      var err error
      for i := 0; i <= maxRetry; i++ {
          result, err = fn()
          if err == nil {
              return result, nil
          }
          if !isRetryable(err) {
              return result, err  // 不可重试的错误直接返回
          }
          time.Sleep(time.Duration(math.Pow(3, float64(i))) * 100 * time.Millisecond)
      }
      return result, err
  }

行业参考:
  · AWS SDK: 所有 API 内置指数退避重试（默认 3 次）
  · gRPC: 官方推荐的重试策略（RetryPolicy with exponential backoff）
  · Stripe: Webhook 重试采用指数退避（1h → 2h → 4h → ... → 72h）
```

## 挑战⑩：汇率更新的实时性与一致性

### 问题描述

```
四层汇率体系:
  实时汇率 → 平台汇率 → 充值汇率 → 提现汇率

问题场景:
  · 用户 T0 时刻查看兑换页: 汇率 1 VND = 0.43 BSB（展示用平台汇率）
  · 用户 T1 时刻点击兑换: 平台汇率已更新为 0.41 BSB
  · 用户看到的是 0.43，实际成交 0.41 → 用户投诉"汇率不对"

另一个问题:
  · 定时任务每 5 分钟从第三方拉取实时汇率
  · 某次拉取失败 → 实时汇率停留在 5 分钟前的值
  · 市场剧烈波动时，5 分钟前的汇率和当前差距很大
```

### ★ 主流方案：展示与成交使用同一快照 + 有效期机制

```
策略:

  1. 展示汇率时返回一个"汇率快照 ID"或"汇率时间戳"
     GetExchangeRate 返回: { rate: "0.43", rateSnapshot: "1709020800000" }

  2. 用户提交兑换时带上这个快照标识
     CreateExchangeOrder 入参: { amount: 100, rateSnapshot: "1709020800000" }

  3. 后端校验:
     · 获取当前平台汇率
     · 对比快照时间: 如果超过 N 分钟（如 2 分钟）→ 汇率已过期，返回新汇率让前端刷新
     · 如果在有效期内 → 使用快照时的汇率成交
     · 或者: 对比汇率偏差，偏差超过 X% → 也返回过期

  4. 兑换订单记录实际使用的汇率（exchange_rate 字段）
     → 审计时可追溯

汇率拉取容错:
  · 第三方 API 失败: 保留上一次成功的值，记录告警日志
  · 连续失败 N 次: 触发告警（可能是 API key 过期或服务方故障）
  · 阈值更新机制: 只有实时汇率变动超过配置阈值才更新平台汇率
    → 避免微小波动频繁更新

行业参考:
  · OKX/Binance: 兑换页面展示"报价有效期 15 秒"，过期需刷新
  · Wise(TransferWise): "Your rate is guaranteed for X hours"
  · Stripe: 汇率锁定机制（Conversion rate locks for the duration of the payout）
```

# 四、安全与合规类挑战（★）

## 挑战⑪：接口防刷与限流

### 问题描述

```
钱包接口涉及资金操作，被恶意刷接口可能导致:
  · 高频创建充值订单 → 大量无效通道匹配请求 → 打垮财务模块
  · 高频调用兑换接口 → 利用汇率更新延迟套利
  · 高频查询余额 → 打垮缓存层 → 穿透到 DB
  · 暴力枚举提现方式 → 探测系统支持的银行列表
```

### ★ 主流方案：网关层限流 + 业务层频控

```
两层防护:

第一层 — 网关层限流（gate-font / gate-back 已有基础设施）:
  · 现有中间件: BlackCheck 黑名单检查（gate-font）
  · 补充: 基于 IP / UserID 的 QPS 限流
  · 策略:
    全局: 单 IP 不超过 100 req/s
    单用户: 不超过 20 req/s
    写操作: 不超过 5 req/s（充值/提现/兑换）

第二层 — 业务层频控（ser-wallet 内部）:
  · 创建充值订单: 同一用户 1 分钟内不超过 3 次
  · 创建提现订单: 同一用户 1 小时内不超过 5 次
  · 兑换: 同一用户 1 分钟内不超过 5 次
  · 实现: Redis INCR + EXPIRE（简单滑动窗口）

  -- Redis 频控伪代码:
  key := fmt.Sprintf("ratelimit:deposit:%d", userId)
  count := redis.Incr(key)
  if count == 1 {
      redis.Expire(key, 60)  // 首次设置 60 秒过期
  }
  if count > 3 {
      return BizErr("操作过于频繁，请稍后重试")
  }

行业参考:
  · Stripe: 所有 API 有速率限制（100 req/s per API key）
  · 支付宝: 网关层 + 业务层双重限流
  · Cloudflare: Rate Limiting as a Service（网关层面的标准实践）
```

### 可选方案：令牌桶 / 滑动窗口算法

```
更精细的限流:
  · 令牌桶: 允许突发流量，平滑长期速率
  · 滑动窗口: 精确计算时间窗口内的请求数
  · Go 库: golang.org/x/time/rate（官方令牌桶实现）

何时引入: 当简单的 Redis INCR 不能满足精细化限流需求时
```

## 挑战⑬：资金来源追踪与合规

### 问题描述

```
需求规则:
  · 纯法币充值用户 → 只能法币提现
  · 纯 USDT 充值用户 → 只能 USDT 提现
  · 混合充值用户 → 只能法币提现

技术挑战:
  · 怎么判定用户的"资金来源"？
  · 用户充了 100 VND，又充了 1 USDT，然后全部兑换成了 BSB
  · 此时用户提现，应该给法币还是 USDT？
  · 如果用户消费了一部分再提现呢？
```

### ★ 主流方案：资金来源标记 + 首次充值类型决定

```
方案设计:

  在 user_wallet_setting 或独立字段中记录用户的资金来源类型:

  fund_source_type:
    0 = 未充值过（新用户）
    1 = 纯法币充值
    2 = 纯 USDT 充值
    3 = 混合充值

  状态转换规则:
    首次法币充值: 0 → 1
    首次 USDT 充值: 0 → 2
    法币用户充了 USDT: 1 → 3
    USDT 用户充了法币: 2 → 3
    混合状态不可逆: 3 → 3（永远是混合）

  提现时的判定:
    fund_source_type == 1 → 法币提现
    fund_source_type == 2 → USDT 提现
    fund_source_type == 3 → 法币提现（需求规定）
    fund_source_type == 0 → 无充值记录，不允许提现

实现:
  · 每次充值成功时（Credit 回调），检查并更新 fund_source_type
  · 提现创建时（CreateWithdrawOrder），读取 fund_source_type 决定可用提现方式
  · fund_source_type 变更记录在 wallet_flow 中（可追溯）

行业参考:
  · 反洗钱（AML）: 资金来源追踪是金融合规的基本要求
  · Binance: 出金方式必须与入金方式一致（原路返回原则）
  · 传统银行: "Know Your Transaction (KYT)" 每笔交易标记资金来源
```

## 挑战⑭：防篡改与防重放攻击

### 问题描述

```
重放攻击:
  · 攻击者截获一个充值成功的回调请求
  · 重复发送这个请求 → 重复给用户加款

篡改攻击:
  · 攻击者拦截创建充值订单的请求
  · 修改 amount 字段 → 创建一个大额充值订单

参数枚举:
  · 攻击者遍历其他用户的 userId → 查看别人的余额和交易记录
```

### ★ 主流方案：签名验证 + 幂等 + 鉴权

```
三层防护:

第一层 — 签名验证（防篡改）:
  · 支付通道回调通常带有签名（HMAC/RSA）
  · 验证签名确认请求来自真实的支付通道，且内容未被篡改
  · 这是财务模块的职责（回调先到财务模块，验签后通过 RPC 通知钱包）

第二层 — 幂等（防重放）:
  · wallet_flow 唯一索引 → 同一 orderNo 重复请求不产生副作用
  · 已解决（挑战②的方案）

第三层 — 鉴权（防越权）:
  · C 端: gate-font 的 FontAuth 中间件从 Token 解析 userId
    → 用户只能操作自己的钱包，userId 从 Token 中获取而非请求参数
    → 攻击者无法伪造 userId
  · B 端: gate-back 的 BackAuth + RBAC 权限控制
    → 管理员只能执行被授权的操作
  · RPC 调用: 通过 Kitex 元数据传递上下文，服务间互信
    → 外部不能直接调用 RPC 接口

行业参考:
  · PCI DSS: 要求所有支付通知必须验证签名
  · OAuth 2.0: Token 鉴权防止越权访问
  · OWASP: Broken Access Control（2021 排名第一的安全风险）
```

### ★ 主流方案：结构化日志 + TraceID 串联 + 关键指标告警

```
三板斧:

第一板 — 结构化日志:

    klog.CtxInfof(ctx,
      "[LEDGER] op=%s orderNo=%s userId=%d currencyId=%d walletType=%d "+

  · 关键字段:
    - TraceID: 自动从 ctx 中获取（Kitex 透传）
    - OrderNo: 关联到具体订单

第三板 — 关键指标告警:
  · Credit/Debit 非业务性失败（系统错误）→ 立即告警
  · 余额出现负数（理论不应发生）→ 立即告警

  · RPC 调用超时率 > 1% → 告警

行业参考:
  · Google SRE: "The Four Golden Signals"（延迟/流量/错误/饱和度）
  · Uber: 全链路 TraceID（Jaeger 分布式追踪）
  · 支付宝: "交易号一查到底"（全链路日志关联）
```

## 挑战⑱：数据库连接池与事务超时

### 问题描述

```
行锁事务的潜在风险:
  · ledger 层的事务: 锁定行 → 执行业务 → 提交
  · 如果"执行业务"阶段调用了外部 RPC（不应该但可能误写）
  · RPC 超时 3 秒 → 事务持有行锁 3 秒 → 其他请求等待 3 秒
  · 多个这样的请求 → 连接池被占满 → 所有请求阻塞

即使不调用外部 RPC:
  · 慢 SQL → 事务执行时间变长 → 连接占用时间变长
  · 并发量突增 → 连接池不够用
```

### ★ 主流方案：事务超时 + 连接池配置 + ledger 内禁止 RPC

```
三个控制点:

第一 — 事务执行超时:
  · 在事务内设置 statement_timeout 或在应用层用 context.WithTimeout
  · 建议: 单笔余额操作事务 ≤ 500ms，兑换事务 ≤ 1s
  · 超时自动回滚 → 不会永久持有锁

  ctx, cancel := context.WithTimeout(ctx, 500*time.Millisecond)
  defer cancel()
  err := db.WithContext(ctx).Transaction(func(tx *gorm.DB) error { ... })

第二 — 连接池配置:
  · GORM 连接池参数:
    db.SetMaxOpenConns(100)      // 最大连接数
    db.SetMaxIdleConns(20)       // 最大空闲连接
    db.SetConnMaxLifetime(30m)   // 连接最大存活时间
    db.SetConnMaxIdleTime(10m)   // 空闲连接最大存活时间

第三 — ledger 层规范:
  · 刚性规则: ledger 层的方法内部禁止调用外部 RPC
  · 原因: RPC 延迟不可控，会拉长事务时间
  · 正确做法:
    ser 层: 先调外部 RPC 获取所需数据 → 再调 ledger 层
    ledger 层: 只做 DB 操作（查询 + 更新 + 流水），不调外部

行业参考:
  · HikariCP（Java 连接池）: "30 seconds is a reasonable connection timeout"
  · TiDB 官方: "建议设置 max_execution_time 防止长事务"
  · GORM 官方: "SkipDefaultTransaction: true — 跳过默认事务，手动管理"
```

# 八、总结

```
核心认知:

  钱包模块的技术挑战"说难也不难"：
    · 每个挑战在业界都有成熟的、经过大规模验证的解决方案
    · 支付宝/Stripe/微信支付 已经趟过所有的坑
    · 不需要发明新技术，选对方案并正确实现就好

  "说不难也需要周全考虑"：
    · 资金系统的特殊性: 一个 Bug 可能导致直接资损，不像前端 Bug 只是 UI 不好看
    · 需要多层防线: 幂等(防重复) + 事务(防部分失败) + 行锁(防并发) + 对账(防积累误差)
    · 每一层都不复杂，但组合在一起需要严谨思维

我们的策略:

  ① 选最简单可靠的方案（行锁 > 分布式锁，本地事务 > TCC，BIGINT > DECIMAL）
  ② 不过度设计（不做热点账户优化、不做分布式事务、不做复式记账）
  ③ 预留升级空间（version 字段、Mock 接口、ETCD 配置开关）
  ④ 分批实施（核心架构先行 → 业务逻辑跟进 → 运维能力补充 → 按需优化）

  简单的方案如果能解决问题，就是最好的方案。
  复杂度只在必要时引入，而不是预防性引入。
```
