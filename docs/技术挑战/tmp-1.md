## 一、挑战全景总览

### 1.1 为什么钱包模块的技术挑战比普通CRUD多

```
普通业务模块（如 ser-app Banner管理）：
  写入频率低、数据不涉及金额、错了可以直接改
  技术挑战：几乎没有

钱包模块：
  ① 涉及真金白银 → 错一分钱就是事故
  ② 多方并发操作同一余额 → 充值+消费+提现可能同时发生
  ③ 跨服务调用 → wallet ↔ finance 通过RPC通信，网络不可靠
  ④ 状态流转复杂 → 提现有8种状态，任何非法跳转都是漏洞
  ⑤ 外部依赖多 → 三方支付回调、区块链确认、汇率API
  ⑥ 合规要求高 → 每笔变动需审计、对账、防洗钱
```

### 1.2 挑战分级

| 级别 | 含义 | 判断标准 |
|------|------|---------|
| **P0 致命** | 出问题直接导致资金损失 | 余额错误、重复入账、双花 |
| **P1 严重** | 出问题导致业务不可用或数据不一致 | 状态错乱、对账失败、服务不可用 |
| **P2 重要** | 出问题导致性能下降或运维困难 | 查询慢、监控盲区、热点账户 |
| **P3 一般** | 出问题有备选方案或影响有限 | 缓存不一致(短暂)、日志丢失 |

### 1.3 挑战全景图


## 二、挑战一：并发余额操作的正确性

> 级别：P0 致命
> 场景：同一用户的余额被多个请求同时修改，导致"丢失更新"

### 2.1 问题本质

```
经典并发场景（Race Condition）：

用户余额 = 1000

  请求A（充值500）        请求B（消费300）
  ① 读取余额 = 1000      ① 读取余额 = 1000
  ② 计算 1000+500=1500   ② 计算 1000-300=700
  ③ 写入余额 = 1500      ③ 写入余额 = 700
                           ↑ 覆盖了请求A的结果！

结果：余额=700，丢失了500的充值 → 资金凭空蒸发

这在钱包系统中是零容忍的——任何丢失更新都意味着资金损失。
```

### 2.2 发生概率

```
我们的系统中，触发此场景的条件是：
  同一用户 + 同一币种 + 同一子钱包 + 同时发生余额变动

实际概率分析：
  - 充值回调 + 同时消费：中等概率（用户充值后立即消费）
  - 连续兑换操作：低概率（前端有loading防重，但不绝对）
  - finance重试入账 + 正常入账：低概率但后果严重（重复入账）
  - 同用户多设备同时操作：低概率

虽然概率不高，但只要发生一次就是资金事故，所以必须100%防御。
```

### 2.3 业界解决方案对比

#### 方案一：乐观锁（Optimistic Concurrency Control）——主流推荐

```
原理：
  账户表增加 version 字段，每次更新时带上版本号条件

  UPDATE t_wallet_account
  SET available_balance = available_balance + 500,
      version = version + 1
  WHERE id = 123 AND version = 10  -- 版本号必须匹配

  affected_rows == 1 → 成功
  affected_rows == 0 → 被其他请求先更新了 → 重试

优势：
  ✅ 无锁等待，高性能（大多数情况一次成功）
  ✅ 无死锁风险
  ✅ TiDB 原生支持乐观事务模型
  ✅ 实现简单（加一个 version 字段即可）

劣势：
  ⚠️ 极端高并发下重试次数增加
  ⚠️ 重试逻辑需要正确实现（不能无限重试）

适用场景：
  同一账户并发冲突率 < 10% → 绝大多数钱包系统

业界案例：
  支付宝账户余额更新、微信零钱变动、Stripe Balance API
```

#### 方案二：悲观锁（Pessimistic Locking / SELECT FOR UPDATE）——可选

```
原理：
  在事务中先锁定行，其他请求排队等待

  BEGIN;
  SELECT * FROM t_wallet_account WHERE id = 123 FOR UPDATE;  -- 加行锁
  -- 其他请求在此排队等待
  UPDATE t_wallet_account SET available_balance = ... WHERE id = 123;
  COMMIT;  -- 释放锁

优势：
  ✅ 绝对串行化，逻辑简单
  ✅ 不需要重试逻辑

劣势：
  ⚠️ 有死锁风险（多行锁顺序不一致时）
  ⚠️ 锁等待影响性能（高并发下瓶颈明显）
  ⚠️ TiDB 的悲观事务有额外开销（需要向 TiKV 发送加锁请求）
  ⚠️ 长事务导致锁持有时间长

适用场景：
  并发冲突率极高的场景（如秒杀库存扣减）

业界案例：
  银行核心系统（对延迟不敏感但对正确性要求极高）
```

#### 方案三：数据库原子操作（Atomic SQL）——轻量辅助

```
原理：
  利用 SQL 本身的原子性，不在应用层读再写

  UPDATE t_wallet_account
  SET available_balance = available_balance + 500
  WHERE id = 123 AND available_balance >= 0

  不先读取旧值，直接用 available_balance + N 原子操作

优势：
  ✅ 最简单，SQL 引擎保证原子性
  ✅ 无需额外字段或逻辑

劣势：
  ⚠️ 无法做业务校验（比如"余额是否充足"需要先读取）
  ⚠️ 无法获取操作前后的余额快照（流水表需要 balance_before/after）
  ⚠️ 单独使用不够，需要配合其他方案

适用场景：
  纯加款场景（credit）可以用原子 SQL，扣款场景仍需先查后改
```

#### 方案四：Redis 分布式锁——可选（不推荐作为主方案）

```
原理：
  操作余额前先获取 Redis 锁 wallet:lock:{user_id}:{currency}:{wallet_type}

  lock → 读取余额 → 校验 → 更新余额 → 写流水 → unlock

优势：
  ✅ 锁粒度灵活

劣势：
  ⚠️ Redis 锁和 DB 事务不在同一原子单元（锁成功但DB失败 → 不一致）
  ⚠️ Redis 不可用时整个系统不可用
  ⚠️ 锁过期但事务未完成 → 并发问题重现
  ⚠️ 引入额外网络依赖

适用场景：
  跨资源的协调场景（如多服务写同一资源），不适合单库操作
```

### 2.4 我们的最佳实践

```
主方案：乐观锁 + 有限重试（ADR-03 已决策）

原因：
  1. 我们是单用户维度的余额更新，并发冲突率低
  2. TiDB 乐观事务模型天然契合
  3. 不引入额外依赖（Redis锁），保持架构简洁
  4. 工程实现简单，version 字段 + CAS 更新 + 最多重试2次

代码模式：
  withRetry(ctx, 2, func() error {
      return tx.Transaction(func(tx *query.Query) error {
          // 读取 account + version
          // 校验余额
          // UPDATE WHERE version = oldVersion
          // 写流水
      })
  })

兜底：
  3 次都失败 → 返回错误码，让上游（finance/前端）决定是否重试
  监控重试率 → 如果持续 > 5%，考虑升级为悲观锁
```

## 三、挑战二：重复请求导致资金重复变动

> 级别：P0 致命
> 场景：finance 调 WalletCredit 超时后重试，wallet 执行了两次入账

### 3.1 问题本质

```
充值成功回调链路：

  三方支付 → ser-finance → RPC调 ser-wallet.WalletCredit(order_no=C123, amount=1000)

  情况1：正常
    finance调wallet → wallet入账1000 → 返回成功 → finance记录完成

  情况2：网络超时
    finance调wallet → wallet入账1000 → 响应在网络中丢失
    finance不知道wallet是否成功 → 重试调wallet → wallet又入账1000
    结果：用户余额多了1000 → 资金凭空产生

这是分布式系统中最经典的"至少一次投递 + 非幂等处理 = 灾难"问题。

不仅是充值，以下接口都有同样风险：
  WalletCreditReward（奖金入账）
  WalletManualAdd（人工加款）
  WalletManualSub（人工减款）
  DepositNotify（链上回调）
```

### 3.2 业界解决方案对比

#### 方案一：业务唯一键（流水表唯一索引）——主流推荐

```
原理：
  在 t_wallet_flow 表的 (related_order_no, flow_type) 上建 UNIQUE 索引
  入账操作在同一个 DB 事务中：①更新余额 ②插入流水
  重复请求 → 插入流水时唯一索引冲突 → 事务回滚 → 返回"已处理"

  Transaction {
      UPDATE wallet_account SET balance += 1000 WHERE ...
      INSERT INTO wallet_flow (order_no, flow_type, ...) -- 唯一索引保障
  }
  // 唯一索引冲突 → 整个事务自动回滚 → 余额不会被重复修改

优势：
  ✅ 余额更新和幂等检查在同一个 DB 事务中，原子性天然保证
  ✅ 不依赖额外组件
  ✅ 流水表本身就是幂等记录，一表两用
  ✅ 唯一索引是数据库引擎级别的保障，不存在 TOCTOU 竞态

劣势：
  ⚠️ 需要调用方传入全局唯一的业务单号
  ⚠️ 同一订单不同类型操作（入账+奖金）需要 flow_type 参与唯一约束

适用场景：
  所有写入操作有天然业务单号的系统

业界案例：
  支付宝流水号幂等、微信支付 out_trade_no 幂等、Stripe idempotency
```

#### 方案二：独立幂等表——可选

```
原理：
  建一张 t_idempotent_key 表，写入前先 INSERT(key)
  INSERT 成功 → 执行业务；INSERT 冲突 → 跳过

  Transaction {
      INSERT INTO t_idempotent_key (idempotent_key) VALUES (?)  -- 先占位
      UPDATE wallet_account ...
      INSERT INTO wallet_flow ...
  }

优势：
  ✅ 幂等逻辑与业务解耦
  ✅ 可复用于其他需要幂等的场景

劣势：
  ⚠️ 多维护一张表
  ⚠️ 本质和流水表唯一索引一样，多了一层抽象
  ⚠️ 如果幂等表和业务表不在同一库 → 分布式事务问题

适用场景：
  没有天然流水表的系统
```

#### 方案三：Redis SetNX 幂等——可选（不推荐单独使用）

```
原理：
  Redis.SetNX(key="wallet:credit:C123", value="1", TTL=24h)
  SET成功 → 执行业务；SET失败 → 跳过

优势：
  ✅ 速度快，可挡住大量重复请求

劣势：
  ⚠️ Redis 和 DB 不在同一个事务中
     可能出现：Redis SET成功但DB写入失败 → 后续重试被Redis挡住 → 入账丢失
  ⚠️ Redis 不可用时幂等失效
  ⚠️ TTL 过期后相同请求可能再次被执行

适用场景：
  作为流量层的快速去重（第一道防线），但不能作为唯一保障

改良用法：
  Redis SetNX 做第一层快速过滤 + DB唯一索引做第二层兜底
  这样大多数重复请求被Redis挡掉不到达DB，减少DB压力
```

#### 方案四：Token 机制（前端请求幂等）——可选辅助

```
原理：
  前端每次发起请求前先获取一个一次性 token
  后端校验 token 是否已使用过，已使用则拒绝

优势：
  ✅ 防止前端重复提交（双击、网络重试）

劣势：
  ⚠️ 只解决前端重复提交，不解决后端服务间的重试幂等
  ⚠️ 需要额外接口和存储

适用场景：
  C端用户操作的防重复提交（如充值/提现/兑换按钮）
```

### 3.3 我们的最佳实践

```
核心方案：流水表唯一索引（ADR-04 已决策）

  t_wallet_flow 的 (related_order_no, flow_type) UNIQUE INDEX
  所有资金变动操作在同一事务中：余额更新 + 流水插入
  唯一索引冲突 = 幂等拦截 → 事务回滚 → 返回成功

辅助方案（可选，视实际压力决定）：
  C端操作加前端防重复提交（loading + 短时间禁用按钮）
  高频接口加 Redis SetNX 做第一层快速过滤

为什么不用独立幂等表：
  流水表天然就是幂等记录，不需要再多维护一张表

为什么不依赖 Redis SetNX：
  Redis 和 DB 不在同一事务中，无法保证原子性
  DB 唯一索引是最终兜底，不可替代
```

## 四、挑战三：金额精度丢失

> 级别：P0 致命
> 场景：浮点运算导致 0.1 + 0.2 = 0.30000000000000004

### 4.1 问题本质

```
IEEE 754 浮点数无法精确表示十进制小数：

  Go 代码：
    a := 0.1
    b := 0.2
    fmt.Println(a + b)  // 输出 0.30000000000000004

  看似微小的误差在金融场景下会累积：
    10000 笔交易，每笔误差 0.000000001 → 累计误差 0.00001
    1000万笔交易 → 累计误差 0.01
    对账时一分钱对不上 → 人工排查成本极高

  更严重的场景——汇率换算：
    1 BSB = 27000 VND（越南盾）
    用户充值 100000 VND → 应得 3.703703703... BSB
    float64: 3.7037037037037037（丢失精度）
    正确:    3.70370370（按8位截断）

  精度丢失不仅是"差一点"的问题，更可能导致：
    - 对账不平（自动化对账失败需人工介入）
    - 用户投诉（"我充了10万VND怎么少了？"）
    - 审计风险（资金数据前后不一致）
```

### 4.2 业界解决方案对比

#### 方案一：DECIMAL + string + 高精度库——主流推荐

```
三层配合：
  数据库层：DECIMAL(30,8)
    MySQL/TiDB 的 DECIMAL 是精确十进制运算，不存在浮点误差
    30位整数 + 8位小数，覆盖法币(2位)和加密货币(8位)
    UNSIGNED 防止余额/金额为负（t_wallet_flow.amount 除外）

  传输层：string
    IDL 中金额字段用 string 类型（如 "100.50"）
    避免 Thrift double 在序列化/反序列化过程中的精度丢失
    前端 JavaScript 的 Number 精度只有 15-17 位有效数字，string 更安全

  计算层：shopspring/decimal 库
    Go 标准库没有 Decimal 类型，需要第三方库
    shopspring/decimal 是 Go 生态最主流的高精度十进制运算库
    支持加减乘除、舍入（RoundBank/RoundHalfUp）、比较、格式化

完整链路：
  前端传入 "100.50" → decimal.NewFromString("100.50")
  → 所有运算用 decimal 类型 → result.StringFixed(8)
  → DB 存入 DECIMAL(30,8) → 查出时用 string 传回前端

优势：
  ✅ 全链路无浮点参与，精度零丢失
  ✅ 业界标准方案（支付宝、Stripe、Wise 均采用）
  ✅ DECIMAL 有数据库引擎级别的精确保证
  ✅ shopspring/decimal 在 Go 生态使用广泛（30k+ stars）

劣势：
  ⚠️ string 解析有少量性能开销（可忽略，金额操作不是计算密集型）
  ⚠️ 开发时需要注意不能用 float64 做中间计算

业界案例：
  Stripe API（所有金额 string 传输）、Wise、Revolut
  AWS Billing API（DECIMAL 存储）
  支付宝/微信支付内部结算系统
```

#### 方案二：整数最小单位（int64 存"分"或"聪"）——可选

```
原理：
  法币：存"分"，1元 = 100分 → 100.50元存为 10050
  加密币：存"聪"，1 BTC = 100000000 聪

优势：
  ✅ 整数运算绝对无精度问题
  ✅ 性能最高（CPU 整数指令快于浮点）
  ✅ 存储空间小

劣势：
  ⚠️ 不同币种精度不同（法币2位 vs 加密8位）→ 放大倍数不统一
  ⚠️ 汇率换算的中间结果需要超高精度小数，int64 容易溢出
     例：100000 VND × 汇率 0.00003704 → 中间结果 3.704
     用 int64 存聪：370400000 聪 → 还需要按精度转换 → 易出错
  ⚠️ 前后端展示时需要频繁做放大/缩小转换
  ⚠️ 现有工程中 int64 模式用于礼物/钻石（单精度），不适合多精度混合场景

适用场景：
  单一币种或单一精度的系统（如纯人民币电商）

业界案例：
  微信支付（分为单位）、Google Play Billing（微单位）
```

#### 方案三：float64 + 格式化——不推荐

```
原理：
  用 float64 计算，最终结果通过格式化输出（如 %.2f）

劣势：
  ⚠️ 根本问题未解决，累积误差随交易量增长
  ⚠️ 对账噩梦
  ⚠️ 在金融/支付领域是公认的反模式

不适用于任何涉及资金的系统。
```

### 4.3 我们的最佳实践

```
采用方案一：DECIMAL(30,8) + string + shopspring/decimal（ADR-06 已决策）

补充规则：
  1. 舍入策略：统一使用银行家舍入（RoundBank），避免系统性偏差
     decimal.NewFromString("3.7037037037").RoundBank(8)

  2. 币种精度映射：
     法币（VND/IDR/THB）→ 展示2位小数，存储8位（展示层截断）
     加密币（USDT/BTC）→ 展示8位小数，存储8位
     平台币（BSB）→ 展示2位小数，存储8位

  3. 统一换算入口：
     所有涉及汇率换算的代码必须走 CurrencyService.CalcExchangeAmount()
     禁止各处自行 float64 计算

  4. Code Review 红线：
     金额相关代码中出现 float64 → 直接 block
     这是团队规范层面的保障
```

## 五、挑战四：跨服务数据一致性

> 级别：P1 严重
> 场景：wallet 和 finance 通过 RPC 通信，操作一方成功另一方失败

### 5.1 问题本质

```
充值入账链路（跨两个服务的数据库）：

  ser-finance:
    ① 收到三方支付成功回调
    ② 更新 finance 侧订单状态 = 已完成（写 db_finance）
    ③ RPC 调 wallet.WalletCredit 入账

  两种不一致场景：

  场景A：finance 成功但 wallet 失败
    finance 记录"已入账" → wallet 实际没入 → 用户投诉没收到钱

  场景B：wallet 成功但 finance 不知道
    wallet 入账成功 → RPC 响应超时 → finance 认为失败 → 再次调用
    如果没有幂等 → 重复入账

  这是经典的"分布式事务"问题。
  两个服务各有独立数据库，无法用本地事务保证原子性。
```

### 5.2 业界解决方案对比

#### 方案一：本地事务 + 重试 + 幂等（Best-Effort Delivery）——主流推荐

```
原理：
  不追求两阶段原子提交，而是通过"至少执行一次 + 幂等保障"达到最终一致

  finance 侧：
    1. 本地事务更新自己的状态
    2. 调 wallet.WalletCredit（可能失败）
    3. 失败则定时重试（指数退避，最多N次）
    4. 超过重试次数 → 进入人工处理队列

  wallet 侧：
    1. 所有资金接口都是幂等的（流水表唯一索引）
    2. 重复调用 → 返回成功（不重复入账）

  一致性保证：
    - 如果 wallet 成功 → 最终一致（finance 重试也不会重复入账）
    - 如果 wallet 持续失败 → 人工介入（这是极端情况，概率极低）
    - 定期对账兜底（finance 的订单状态 vs wallet 的流水记录）

优势：
  ✅ 不引入分布式事务框架
  ✅ 实现简单，与现有工程模式一致
  ✅ 幂等保障让重试安全无副作用
  ✅ 对账机制作为最终兜底

劣势：
  ⚠️ 不是实时强一致（有短暂不一致窗口）
  ⚠️ 极端情况需人工介入

适用场景：
  大多数互联网支付系统、钱包系统、电商交易系统

业界案例：
  支付宝（回调 + 重试 + 幂等 + 定时对账）
  Stripe（Webhook + Retry + Idempotency Key）
  Amazon Pay（At-Least-Once + Idempotent）
```

#### 方案二：事务消息（Transactional Outbox）——可选进阶

```
原理：
  发送方在本地事务中同时写业务数据 + 消息到 outbox 表
  后台线程/CDC 轮询 outbox 表投递到 MQ
  消费方消费消息并幂等处理

  finance 侧：
    Transaction {
        UPDATE order SET status = '已完成'
        INSERT INTO outbox (topic='wallet.credit', payload={...})
    }
    // 后台线程扫描 outbox → 投递到 NATS JetStream
    // wallet 消费消息 → 幂等入账

优势：
  ✅ 业务操作和消息发送的原子性
  ✅ 消费方故障时消息不丢（MQ 持久化）

劣势：
  ⚠️ 实现复杂度高（outbox 表 + 轮询/CDC + MQ）
  ⚠️ 延迟增加（经过 MQ 中间层）
  ⚠️ 需要额外组件和运维成本

适用场景：
  对一致性要求极高且消息量大的系统

业界案例：
  Debezium + Kafka（事件驱动架构）
  Axon Framework（CQRS/ES）
```

#### 方案三：分布式事务框架（Seata/DTM）——不推荐

```
原理：
  引入事务协调器，支持 AT/TCC/Saga 等模式

优势：
  ✅ 理论上可实现强一致

劣势：
  ⚠️ 引入新依赖（协调器需要高可用部署）
  ⚠️ 性能损耗大（两阶段提交的锁等待）
  ⚠️ 与现有工程栈（Kitex + TiDB）集成不成熟
  ⚠️ 明显过度设计

不适用于我们的场景。
```

### 5.3 我们的最佳实践

```
主方案：本地事务 + 重试 + 幂等（ADR-05 已决策）

具体模式：
  finance → wallet（同步 RPC）：
    所有资金操作（WalletCredit/Freeze/Unfreeze/Deduct 等）走同步 RPC
    wallet 侧全部幂等
    finance 侧超时后安全重试

  wallet → finance（异步 MQ）：
    提现创建后通知 finance 审核 → NATS JetStream
    消息投递失败 → NakWithDelay(5s) + MaxDeliver 重试
    超过最大重试 → 死信队列 + 告警

兜底：
  定期对账（每日凌晨）：
    比对 finance 侧订单状态 vs wallet 侧流水记录
    不一致 → 告警 → 人工用 WalletManualAdd/Sub 修正

该方案在"正确性 + 简洁性 + 可维护性"之间取得最佳平衡。
```

## 六、挑战五：订单状态机的正确流转

> 级别：P1 严重
> 场景：提现订单被非法跳转状态（如从"风控驳回"跳到"出款成功"）

### 6.1 问题本质

```
提现订单有8种状态，合法转换路径严格限定：

  1(审核中) → 2(风控通过) → 4(财务通过) → 6(出款中) → 7(出款成功)
            → 3(风控驳回)                            → 8(出款失败)
                           → 5(财务驳回)

非法跳转的危害：
  - 3(风控驳回) → 7(出款成功)：已驳回的提现被放行 → 资金外流
  - 7(出款成功) → 1(审核中)：已完成的订单重新进入审核 → 逻辑混乱
  - 同时收到"风控通过"和"风控驳回" → 状态不确定

状态机的正确性直接关系到资金安全。
```

### 6.2 业界解决方案对比

#### 方案一：数据库级 WHERE 守卫——主流推荐

```
原理：
  状态变更时 WHERE 条件带上"当前状态必须是合法的前置状态"

  UPDATE t_withdraw_order
  SET status = 4 (财务通过)
  WHERE order_no = 'T123' AND status = 2 (风控通过)

  affected_rows == 1 → 转换成功
  affected_rows == 0 → 当前状态不是预期的前置状态 → 拒绝

优势：
  ✅ 利用数据库的原子性保证，无竞态条件
  ✅ 实现极简，一个 WHERE 条件
  ✅ 对并发安全（两个请求同时尝试，只有一个能成功）

劣势：
  ⚠️ 状态转换规则散落在各处 SQL 中，不集中
  ⚠️ 如果状态很多，需要仔细枚举所有合法路径

适用场景：
  状态数量适中（< 20）的订单系统

业界案例：
  几乎所有订单系统都用此模式
```

#### 方案二：应用层状态机引擎——可选进阶

```
原理：
  用 Go 结构体定义状态转换图，所有状态变更必须通过状态机引擎

  transitions := map[State][]State{
      Reviewing:  {RiskPass, RiskDeny},
      RiskPass:   {FinPass, FinDeny},
      FinPass:    {Paying},
      Paying:     {Paid, PayFail},
  }

  func (sm *StateMachine) Transit(current, target State) error {
      allowed := transitions[current]
      if !contains(allowed, target) {
          return ErrIllegalTransition
      }
      return nil
  }

优势：
  ✅ 状态转换规则集中定义，一目了然
  ✅ 可生成状态转换图文档
  ✅ 可加入 hook（转换前后的回调逻辑）

劣势：
  ⚠️ 应用层校验不能替代数据库层 WHERE 守卫（两者应同时使用）
  ⚠️ 引入额外抽象

适用场景：
  状态复杂且需要可视化管理的系统
  推荐库：looplab/fsm（Go）
```

### 6.3 我们的最佳实践

```
双保险策略：

第一层：应用层枚举校验（快速失败）
  在 Service 层定义合法转换 map
  非法转换 → 直接返回错误码，不到达 DB

第二层：数据库 WHERE 守卫（最终保障）
  所有 status UPDATE 必须带 WHERE status = fromStatus
  即使应用层有 bug 漏过了，DB 层也能挡住

代码模式（与架构设计6.4一致）：
  func (r *OrderRepo) TransitStatus(ctx, orderNo, from, to) (bool, error) {
      result := o.Where(o.OrderNo.Eq(orderNo)).
                  Where(o.Status.Eq(from)).
                  UpdateSimple(o.Status.Value(to))
      return result.RowsAffected > 0, nil
  }

终态保护：
  提现终态 (3风控驳回/5财务驳回/7出款成功/8出款失败) → 不可再变
  充值终态 (3已完成/4已失败) → 不可再变
  冻结终态 (2已解冻/3已扣款) → 不可再变
```

## 七、挑战六：热点账户高并发写入

> 级别：P2 重要
> 场景：同一用户短时间内大量操作（如高频投注+同时充值+返奖），导致乐观锁频繁冲突

### 7.1 问题本质

```
乐观锁的前提是"冲突率低"。
但某些场景下同一账户的写入可能瞬时集中：

  用户 A 在 2 秒内发生以下操作：
    投注扣款（消费扣减中心钱包）
    返奖入账（返奖入奖励钱包）
    充值到账（回调入中心钱包）
    活动奖金（奖金入奖励钱包）

  如果 4 个请求同时到达且都操作同一个子钱包 → 乐观锁连续冲突
  最坏情况：每个请求重试 2 次仍冲突 → 返回错误

这在我们当前阶段发生的概率极低（非高频交易平台），
但需要提前了解解决方案以备扩展。
```

### 7.2 业界解决方案对比

#### 方案一：乐观锁 + 有限重试（当前足够）——主流

```
优势：
  ✅ 实现简单
  ✅ 低并发场景性能最优
  ✅ 监控重试率即可判断是否需要升级

劣势：
  ⚠️ 高并发下重试增多

适用场景：
  同一账户并发写入 < 10 QPS 的系统

我们的情况：
  当前用户规模下，同一用户同一子钱包的并发写入概率极低
  乐观锁 + 重试 2 次完全够用
```

#### 方案二：内存排队 + 合并写入——可选进阶

```
原理：
  将同一账户的多个操作在内存中排队，合并为一次 DB 写入

  请求1: +500, 请求2: -300, 请求3: +200
  合并为: +400，一次 DB UPDATE

优势：
  ✅ DB 写入减少，冲突降为0

劣势：
  ⚠️ 实现复杂（需要内存队列 + 合并逻辑）
  ⚠️ 进程重启时内存中未刷盘的操作可能丢失
  ⚠️ 分布式部署下需要按用户路由（同一用户请求到同一实例）
  ⚠️ 每个操作都需要独立的流水记录，合并写入后流水怎么处理是个问题

适用场景：
  高频交易系统（如股票撮合引擎）、游戏虚拟货币
  不适合我们当前阶段
```

#### 方案三：悲观锁降级——可选备用

```
原理：
  检测到乐观锁重试率持续 > 5%，动态切换为 SELECT FOR UPDATE 悲观锁

优势：
  ✅ 高冲突下不再重试，排队等待

劣势：
  ⚠️ 需要双模式代码
  ⚠️ 切换逻辑需要自动化或配置化

适用场景：
  乐观锁顶不住时的降级方案
```

#### 方案四：子账户拆分（Account Sharding）——可选极端方案

```
原理：
  将一个逻辑账户拆分为多个物理子账户
  用户 A 的中心钱包拆成 A-center-1, A-center-2, A-center-3
  写入时随机选一个子账户，冲突概率降为 1/N
  查余额时 SUM 所有子账户

优势：
  ✅ 写入并发提升 N 倍

劣势：
  ⚠️ 实现非常复杂
  ⚠️ 查余额需要聚合多个子账户
  ⚠️ 扣款时可能需要跨子账户操作
  ⚠️ 过度设计

适用场景：
  超大型支付系统（日千万级交易）
  完全不适合当前阶段
```

### 7.3 我们的最佳实践

```
当前阶段：乐观锁 + 重试 2 次（已足够）

监控指标：
  记录每次乐观锁重试次数
  如果"重试率 > 5%持续1小时" → 告警 → 评估是否需要升级

升级路线（预留，不提前实现）：
  阶段1（当前）：乐观锁 + 重试
  阶段2（冲突率 > 5%）：升级为悲观锁（SELECT FOR UPDATE）
  阶段3（单用户 QPS > 100）：内存排队或子账户拆分

注意：
  不同子钱包（中心/奖励/主播）天然隔离，减少了单账户冲突概率
  五种子钱包 = 天然的写入分散
```

## 八、挑战七：提现防双花

> 级别：P0 致命
> 场景：用户在提现审核期间利用余额做其他操作，导致"一笔钱用了两次"

### 8.1 问题本质

```
"双花"（Double Spending）是钱包系统最经典的安全问题。

场景：用户余额 10000
  Step1: 用户发起提现 10000
    如果只是"标记提现中"而不冻结余额
    → 可用余额仍然是 10000
  Step2: 用户同时发起消费 10000
    → 余额扣减成功（因为可用余额还在）
  Step3: 提现审核通过，出款 10000

结果：用户用 10000 的余额取走了 20000 → 平台资金损失

核心原因：提现期间余额没有被"锁住"。
```

### 8.2 业界解决方案对比

#### 方案一：冻结/解冻机制（Two-Balance Model）——主流推荐

```
原理：
  账户有两个余额字段：
    available_balance（可用余额）— 用户可操作
    frozen_balance（冻结余额）— 暂时不可操作

  提现发起 → freeze：available -= amount, frozen += amount
  审核通过 → deduct：frozen -= amount（最终扣除）
  审核驳回 → unfreeze：frozen -= amount, available += amount

  freeze 之后用户只能操作 available_balance，冻结的钱碰不到

优势：
  ✅ 逻辑清晰，天然防双花
  ✅ 审计清楚（冻结了多少一目了然）
  ✅ 解冻精确归还（不受审核期间其他操作影响）
  ✅ 业界标准模式

劣势：
  ⚠️ 需要维护两个余额字段的一致性
  ⚠️ 需要独立的冻结记录表追踪生命周期

适用场景：
  所有有审核流程的提现/出金系统

业界案例：
  所有银行、支付机构、交易所的标准做法
```

#### 方案二：状态锁定（单余额 + 状态标记）——可选（不推荐）

```
原理：
  账户只有一个 balance 字段
  有提现中时，设置 locked = true，所有操作被拒绝

劣势：
  ⚠️ 锁定期间用户无法做任何操作（体验极差）
  ⚠️ 如果提现金额 < 余额，剩余部分也被锁 → 不合理
  ⚠️ 多笔同时提现无法支持

不推荐。
```

#### 方案三：事后校验（余额充足才放行）——可选辅助

```
原理：
  不冻结，但在审核通过出款前再次校验余额是否充足
  不足则驳回

劣势：
  ⚠️ 审核期间余额可能已被消费 → 用户体验差（"我明明申请了提现怎么被驳回了"）
  ⚠️ 运营成本高（频繁驳回导致工单量大）

不推荐作为主方案，但可作为出款前的二次校验（Belt and Suspenders）。
```

### 8.3 我们的最佳实践

```
主方案：冻结/解冻机制（架构设计已采纳）

  t_wallet_account：available_balance + frozen_balance 双字段
  t_freeze_record：独立表追踪冻结生命周期（3种状态：冻结中/已解冻/已扣款）

  六种原子操作中的三个专门处理冻结：
    freeze：available → frozen（提现发起）
    unfreeze：frozen → available（审核驳回/出款失败）
    deduct：frozen → 0（出款成功，最终扣除）

  冻结记录状态机守卫：
    UPDATE t_freeze_record SET status=2 WHERE id=? AND status=1
    终态(2/3)不可再操作 → 防止"解冻了又扣款"或"扣款了又解冻"

辅助方案：
  出款前二次余额校验（verify frozen_balance >= amount）
  即使冻结机制正确，也做一次最终确认
```

## 九、挑战八：缓存与数据库的一致性

> 级别：P2 重要
> 场景：运营更新了币种配置，但C端读到的还是旧数据

### 9.1 问题本质

```
币种配置（t_currency）被缓存后：
  运营在B端修改了VND的汇率浮动比例：5% → 8%
  如果缓存没及时更新 → C端用户看到的充值兑换比例还是按5%计算
  → 用户多拿了钱 / 少拿了钱 → 投诉或损失

汇率数据（定时更新）：
  定时任务更新了平台汇率
  如果缓存没刷新 → 后续兑换/充值用的还是旧汇率
  → 汇差积累
```

### 9.2 业界解决方案对比

#### 方案一：Cache-Aside + 写后主动失效——主流推荐

```
原理：
  读：先查缓存 → 未命中 → 查DB → 写入缓存 → 返回
  写：先更新DB → 再删除缓存（而非更新缓存）

  为什么是"删除"而不是"更新"缓存：
    删除更安全——并发写时不会出现旧值覆盖新值的竞态
    下次读取时自然回源DB拿最新值

优势：
  ✅ 实现简单，模式成熟
  ✅ 与 ser-app 现有的 cache.DelAll() 模式一致
  ✅ 缓存不一致窗口极短（仅在DB更新后到缓存删除前的微秒级）

劣势：
  ⚠️ 多实例部署时，其他实例的进程内缓存无法被通知

适用场景：
  配置数据、低频变更数据

业界案例：
  大多数互联网系统的标准缓存策略
```

#### 方案二：延迟双删——可选增强

```
原理：
  更新DB → 删缓存 → sleep 500ms → 再删一次缓存

  为什么要"双删"：
    并发场景下可能出现：
    线程A删缓存 → 线程B读缓存miss → 线程B查DB(旧值) → 线程B写缓存(旧值) → 线程A更新DB
    第二次删除可以清理线程B写入的旧缓存

优势：
  ✅ 进一步降低不一致概率

劣势：
  ⚠️ 延迟增加（sleep）
  ⚠️ 币种配置变更频率极低，双删的收益有限

适用场景：
  对缓存一致性要求极高的热点数据
  我们的币种配置不需要（变更频率极低）
```

#### 方案三：Redis Pub/Sub 通知其他实例——可选（多实例进程内缓存）

```
原理：
  进程内缓存（sync.Map）在多实例部署时无法跨实例失效
  更新DB后通过 Redis Pub/Sub 广播通知所有实例刷新

优势：
  ✅ 进程内缓存也能及时失效

劣势：
  ⚠️ 额外的 Redis 订阅逻辑
  ⚠️ Pub/Sub 消息可能丢失（Redis Pub/Sub 不持久化）

替代方案：
  进程内缓存加短 TTL（如 30s），过期后自然回源
  对于币种配置这种极低频变更的数据，短TTL足够
```

### 9.3 我们的最佳实践

```
分层缓存策略：

第一层（进程内 sync.Map）：
  存储：币种配置列表、币种精度规则
  失效：写操作后主动清除本实例；加短 TTL（如 60s）自然过期覆盖多实例
  理由：变更频率极低（天/周级别），短TTL足够

第二层（Redis）：
  存储：汇率数据
  Key规范：wallet:rate:{currency_code}
  失效：定时任务更新后主动 DEL
  TTL：与定时任务间隔一致

什么不缓存：
  ❌ 用户余额（实时性要求高，缓存一致性成本 > 收益）
  ❌ 订单数据（写多读少，状态频繁变化）
  ❌ 冻结记录（正确性要求极高）

原则：涉及资金的实时数据不缓存，宁可多查一次DB。
```

## 十、挑战九：汇率数据的实时性与可靠性

> 级别：P2 重要
> 场景：三方汇率API全部不可用，或返回异常数据

### 10.1 问题本质

```
汇率定时任务需要从 3 家三方 API 获取实时汇率。
任何一个外部依赖都可能出问题：

  故障1：API 超时 → 拿不到数据
  故障2：API 返回异常值（如 1 USD = 0 VND）→ 汇率归零
  故障3：3 家 API 全部不可用 → 完全没有数据源
  故障4：API 返回值偏差过大（1 家返回正常值，另 2 家返回异常值）

如果异常汇率被写入 t_currency → 所有充值/提现/兑换都用错误汇率计算
→ 平台可能在几分钟内损失大量资金
```

### 10.2 业界解决方案对比

#### 方案一：多源聚合 + 异常值过滤 + 降级策略——主流推荐

```
多源聚合：
  对接 3+ 家汇率 API，取平均值（中位数更抗干扰）

异常值过滤：
  收到数据后校验：
  ① 值不能为 0 或负数
  ② 值不能偏离上一次平台汇率超过 X%（如 50%）
     偏离过大 → 丢弃该数据源
  ③ 3 家中只有 1 家异常 → 丢弃异常值取其余 2 家平均
     2 家异常 → 使用唯一正常值
     3 家都异常 → 跳过本轮更新 + 告警

降级策略：
  3 家成功 → 取平均（或中位数）
  2 家成功 → 取 2 家平均
  1 家成功 → 单源使用
  0 家成功 → 不更新，沿用上一次平台汇率 + 告警日志

业界案例：
  Wise（前 TransferWise）：多源汇率聚合 + 异常检测
  XE.com：全球汇率数据服务的标准做法
```

#### 方案二：熔断 + 人工覆盖——可选辅助

```
熔断：
  连续 N 次（如 5 次）获取汇率失败 → 进入熔断状态
  熔断期间停止自动更新 → 使用最近一次成功的平台汇率
  定期探测（每 5 分钟试一次）→ 恢复后自动退出熔断

人工覆盖：
  后台提供"手动设置汇率"能力
  当自动更新完全不可用时，运营可手动设置

适用场景：
  外部 API 大面积故障的极端情况
```

#### 方案三：汇率变化幅度校验（Rate Sanity Check）——推荐辅助

```
在写入 t_currency 前做最终校验：
  new_rate 与 current_platform_rate 的偏差不能超过阈值（如 10%）
  超过阈值 → 拒绝更新 + 告警

防止：
  API 被劫持返回异常值
  API Bug 导致返回值量级错误（如缺了小数点）
```

### 10.3 我们的最佳实践

```
定时任务内部逻辑：

  func updateRates() {
      for _, currency := range activeCurrencies {
          // 1. 并行请求 3 家 API
          rates := fetchFromAPIs(currency.Code)  // 3 goroutine 并行

          // 2. 过滤异常值
          validRates := filterAbnormal(rates, currency.PlatformRate)
          // 规则：> 0、偏离当前平台汇率不超过50%

          // 3. 降级聚合
          if len(validRates) == 0 {
              log.Warn("所有API返回异常，跳过更新", currency.Code)
              alertService.Notify(...)  // 告警
              continue
          }
          realRate := median(validRates)  // 取中位数

          // 4. 偏差检测
          deviation := abs(realRate - currency.PlatformRate) / currency.PlatformRate
          if deviation < currency.RateThreshold {
              continue  // 偏差未达阈值，不更新
          }

          // 5. 最终安全检查
          if deviation > 0.10 {  // 单次变化超过10%
              log.Error("汇率变化异常大", deviation)
              alertService.Notify(...)
              // 可选：仍然更新但告警，或拒绝更新
          }

          // 6. 写入 DB + 记录日志
          updateCurrencyRate(currency, realRate)
          insertRateLog(currency, realRate)
      }
  }

容错总结：
  - 多源冗余（3家API）
  - 异常值过滤（0值/负值/偏差过大）
  - 降级聚合（3→2→1→不更新）
  - 最终安全阀（单次变化>10%告警）
  - 沿用旧值兜底（不更新=用上一次值）
```

## 十一、挑战十：对账与资金安全审计

> 级别：P1 严重
> 场景：系统运行一段时间后，某个用户的余额与流水记录对不上

### 11.1 问题本质

```
对账是资金系统的"最终防线"——即使前面所有保障机制都到位，
也需要定期验证"钱到底对不对"。

核心公式（任何时刻都必须成立）：
  account.available_balance
  == 最后一条 flow 的 balance_after
  == SUM(所有影响 available 的 flow.amount)

如果不对账：
  - Bug 导致的余额偏差无法被发现（直到用户投诉）
  - 安全漏洞（如注入攻击修改余额）无法被发现
  - 审计时无法证明"每一分钱都有据可查"
```

### 11.2 业界解决方案对比

#### 方案一：实时校验 + 离线全量对账——主流推荐

```
实时校验（每次操作后）：
  在余额变动的同一事务中：
    newBalance = oldBalance + amount
    flow.balance_after = newBalance
    assert(flow.balance_after == account.available_balance)  // 运行时断言

  如果 assert 失败 → 日志 ERROR + 告警 + 事务正常提交（不回滚，避免影响用户）
  这能在第一时间发现数据层的不一致

离线全量对账（每日凌晨）：
  遍历 t_wallet_account 所有记录
  对每个账户：查 t_wallet_flow 最后一条记录的 balance_after
  比对：account.available_balance == flow.balance_after
  不一致 → 告警日志 + 通知财务

优势：
  ✅ 实时校验快速发现新问题
  ✅ 离线对账兜底，覆盖历史积累偏差
  ✅ 两层保障互补

劣势：
  ⚠️ 离线对账在数据量大时耗时长

业界案例：
  支付宝（T+1 日终对账）
  微信支付（实时+离线双对账）
  Stripe（Ledger Reconciliation）
```

#### 方案二：事件溯源（Event Sourcing）——可选进阶

```
原理：
  不存余额，余额由流水（事件）聚合计算得出
  account.balance = replay(all events)

优势：
  ✅ 天然一致（余额就是事件的函数）
  ✅ 完美审计能力

劣势：
  ⚠️ 查询余额需要聚合所有历史事件（性能差，需要物化视图）
  ⚠️ 架构复杂度极高
  ⚠️ 完全不适合当前工程栈

不推荐。
```

#### 方案三：跨服务对账（wallet vs finance）——推荐辅助

```
原理：
  比对 wallet 侧的订单/流水 与 finance 侧的订单状态
  例如：
    finance 记录"订单C123已入账" → wallet 应该有对应的入账流水
    wallet 有入账流水 → finance 应该有对应的成功订单

  找出差异 → 人工处理（补单/加减款）

适用场景：
  跨服务一致性校验
```

### 11.3 我们的最佳实践

```
三层对账体系：

第一层：实时断言（开发+测试阶段必启，上线后保留）
  每次余额变动事务中 assert balance_after == account.new_balance
  失败 → ERROR 日志 + 告警（不阻断业务）

第二层：内部对账（每日凌晨定时任务）
  遍历所有 wallet_account
  取最后一条 flow 的 balance_after vs 当前余额
  不一致 → 告警 + 生成对账差异报告

第三层：跨服务对账（每日凌晨或按需）
  finance 侧"已入账"的订单 → wallet 侧应有对应流水
  wallet 侧有流水 → finance 侧应有对应订单
  差异 → WalletManualAdd/Sub 人工修正

对账工具建设优先级：
  Phase 1（上线前）：实时断言
  Phase 2（上线后1周）：内部对账定时任务
  Phase 3（联调完成后）：跨服务对账
```

## 十二、挑战十一：接口安全与风控防护

> 级别：P3 一般（但不可忽视）
> 场景：恶意用户利用接口漏洞薅羊毛或注入非法数据

### 12.1 风险矩阵

```
┌──────────────────┬─────────────────────────┬──────────────────────┐
│ 风险类型          │ 攻击方式                  │ 影响                  │
├──────────────────┼─────────────────────────┼──────────────────────┤
│ 重复充值下单       │ 脚本高频调 CreateDeposit  │ 资源消耗/通道预警     │

│ 并发双花          │ 同时发起多笔提现            │ 资金超额提取          │
│ 越权操作          │ 用 A 的 token 操作 B 账户   │ 资金盗取             │
│ SQL 注入          │ 在输入字段注入SQL           │ 数据泄露/篡改        │
└──────────────────┴─────────────────────────┴──────────────────────┘
```

### 12.2 防御方案

```
四道防线（与架构设计10.1一致）：

第一道：网关层（gate-font/gate-back 已有）
  ✅ Token 鉴权（用户身份验证）
  ✅ 黑名单拦截
  ✅ 频率限制（如果网关有 rate limit 中间件）
  补充：关键接口加频率限制（如提现 1分钟最多3次）

第二道：参数校验层（Service 入口）
  ✅ 金额 > 0（用 decimal 比较，不能是0或负数）
  ✅ 金额 ≤ 上限（防止传入天文数字）
  ✅ 币种有效性校验（必须存在于 t_currency 且 status=启用）
  ✅ 用户归属校验（操作的 user_id 必须等于 token 中的 user_id）
  ✅ GORM 参数化查询（天然防 SQL 注入）

第三道：业务逻辑层
  ✅ 余额充足性校验
  ✅ 限额校验（单笔/单日）
  ✅ 状态机守卫
  ✅ 重复转账拦截
  ✅ KYC 状态校验

第四道：数据层
  ✅ UNSIGNED 字段（DB层面禁止负余额）
  ✅ 唯一索引（幂等保障）
  ✅ 乐观锁（并发保障）
```

## 十三、挑战十二：数据量增长下的查询性能

> 级别：P2 重要
> 场景：t_wallet_flow 只增不删，半年后数据量达到千万级，查询变慢

### 13.1 问题本质

```
流水表特性：
  只插入不更新不删除（审计要求）
  每笔资金变动都写一条记录
  如果日均 10 万笔 → 半年 1800 万条 → 一年 3600 万条

查询场景：
  C端用户查自己的流水记录（按币种+时间翻页）
  对账任务扫描全表
  按订单号追溯流水
```

### 13.2 业界解决方案对比

#### 方案一：合理索引 + TiDB 分布式能力——主流（当前足够）

```
TiDB 的天然优势：
  ✅ 自动按 Region 分片，数据量增长时自动扩展存储节点
  ✅ 不需要应用层分库分表（这是 TiDB 最大的价值之一）
  ✅ 索引查询性能在千万级仍然稳定

索引覆盖查询场景：
  idx(user_id, currency_code, wallet_type, create_at DESC)
    → 覆盖C端流水查询（WHERE user_id + currency + type ORDER BY create_at）

  idx(related_order_no)
    → 覆盖按订单追溯

  只要查询走索引，千万级数据不是问题。

适用场景：
  数据量 < 5 亿条
```

#### 方案二：冷热分离——可选进阶

```
原理：
  超过 N 个月的历史流水迁移到归档表（t_wallet_flow_archive）
  查询时默认查热表，需要历史数据时查归档表

优势：
  ✅ 热表保持小体量，查询速度稳定

劣势：
  ⚠️ 需要迁移脚本和定时任务
  ⚠️ 查询历史需要特殊逻辑

适用场景：
  数据量 > 5 亿条
```

#### 方案三：按时间分区（Partition）——可选

```
原理：
  t_wallet_flow 按 create_at 月份分区
  查询时自动裁剪（只扫描相关月份的分区）

  CREATE TABLE t_wallet_flow (...)
  PARTITION BY RANGE (create_at) (
      PARTITION p202601 VALUES LESS THAN (1738368000),
      PARTITION p202602 VALUES LESS THAN (1740787200),
      ...
  );

优势：
  ✅ 对应用透明，不改查询逻辑

劣势：
  ⚠️ TiDB 对分区表的支持有限制（某些索引不支持）
  ⚠️ 需要定期维护分区（添加新月份分区）

适用场景：
  时间范围查询为主的大表
```

### 13.3 我们的最佳实践

```
当前阶段：合理索引 + 信任 TiDB 分布式能力

  t_wallet_flow 的索引已覆盖所有查询场景（见表结构设计文档）
  TiDB 自动分片不需要应用层操心

预留措施：
  对账查询走只读副本（如果 TiDB 部署了 TiFlash 列存引擎）
  C端流水查询限制时间范围（如只查近3个月）
  监控慢查询 → 超过 200ms 的查询告警

升级路线：
  数据量 < 1亿：当前方案足够
  数据量 1亿-5亿：考虑冷热分离
  数据量 > 5亿：考虑时间分区 + 冷热分离
```

## 十四、挑战十三：多实例部署下的定时任务竞争

> 级别：P3 一般
> 场景：ser-wallet 部署 N 个实例，每个实例都启动了汇率定时任务，导致 N 次执行

### 14.1 问题本质

```
ser-wallet 部署 ≥ 2 个实例时，每个实例的 cron 都会触发汇率更新。
N 个实例同时执行：
  - 同一批 API 被调 N 次（浪费配额/触发限流）
  - 同一条汇率日志被写 N 次
  - 写入 t_currency 可能并发冲突
```

### 14.2 业界解决方案对比

#### 方案一：Redis 分布式锁——主流推荐

```
原理：
  执行前 Redis SETNX("wallet:lock:rate_update", instanceId, TTL=60s)
  抢到锁 → 执行 → 执行完释放（Lua CAS 释放）
  没抢到 → 跳过本轮

  func (c *CronJob) UpdateRates() {
      token := uuid.New()
      if !rds.SetNX("wallet:lock:rate_update", token, 60*time.Second) {
          return  // 其他实例已在执行
      }
      defer releaseLock("wallet:lock:rate_update", token)
      // 执行汇率更新逻辑
  }

优势：
  ✅ 现有工程已有 rds.SetNX + rds.Eval（Lua 释放锁）
  ✅ 实现简单
  ✅ 适合"只需要执行一次"的场景

劣势：
  ⚠️ Redis 不可用时无法获取锁（汇率更新被跳过）
  ⚠️ 锁超时但任务未完成 → 另一个实例可能重复执行（概率极低）

业界案例：
  几乎所有分布式定时任务的标准做法
```

#### 方案二：Leader 选举——可选进阶

```
原理：
  利用 ETCD 的 Leader Election，只有 Leader 实例执行定时任务

优势：
  ✅ 不需要每次执行都抢锁
  ✅ 稳定性更好（Leader 持续执行，Follower 不参与）

劣势：
  ⚠️ 实现复杂（ETCD Session + Keep-Alive + 选举逻辑）
  ⚠️ Leader 故障时需要重新选举（有短暂不可用窗口）

适用场景：
  需要持续执行（如流式消费）的场景，不适合周期性定时任务
```

#### 方案三：外部 Cron 服务（ser-cron）——可选

```
原理：
  项目已有 ser-cron 服务，定时任务统一在那里管理
  ser-cron 通过 RPC 调 ser-wallet 的汇率更新接口
  ser-cron 单实例或自己解决分布式问题

优势：
  ✅ 定时任务集中管理

劣势：
  ⚠️ 增加一次 RPC 调用
  ⚠️ 汇率更新逻辑与币种数据紧密耦合，拆到外部增加复杂度

适用场景：
  适合与业务逻辑解耦的定时任务
```

### 14.3 我们的最佳实践

```
采用方案一：Redis 分布式锁

  锁 Key：wallet:lock:rate_update
  TTL：60s（汇率更新通常 < 10s，留足余量）
  释放：Lua CAS（只有持锁者能释放）

  现有工程已具备 rds.SetNX + rds.Eval，零成本接入。
```

## 十五、挑战十四：跨模块RPC调用的容错与超时

> 级别：P2 重要
> 场景：finance 调 wallet 的 RPC 超时，不知道 wallet 是否成功

### 15.1 问题本质

```
RPC 调用有三种结果：
  ① 成功（收到明确的成功响应）
  ② 失败（收到明确的错误响应）
  ③ 未知（超时，不知道对方是否执行了）

情况③是最危险的：
  finance 调 WalletCredit 超时 → wallet 可能已经入账了，也可能没有
  如果直接重试 → 如果已入账 → 重复入账（需要幂等保障）
  如果不重试 → 如果没入账 → 用户少了钱
```

### 15.2 业界解决方案对比

#### 方案一：幂等接口 + 安全重试——主流推荐

```
原理：
  所有会修改余额的接口都做幂等
  调用方超时后直接重试（安全的，因为幂等）
  已执行 → 返回成功（不重复执行）
  未执行 → 正常执行

超时 → 重试 → 幂等保障 → 安全

这是分布式系统中解决"未知状态"的标准范式。
```

#### 方案二：查询确认 + 条件重试——可选辅助

```
原理：
  超时后先查询状态，再决定是否重试

  finance 调 WalletFreeze 超时 →
  先调 wallet.QueryFreezeStatus(order_no) →
    如果"已冻结" → 不重试
    如果"未找到" → 重试

优势：
  ✅ 更精确（不依赖幂等，明确知道状态）

劣势：
  ⚠️ 多一次 RPC 调用
  ⚠️ 查询和重试之间可能有时间差（如果此时原请求才处理完）

适用场景：
  不容易做幂等的接口（但我们所有接口都可以做幂等，所以不必要）
```

#### 方案三：异步通知 + 状态同步——可选

```
原理：
  关键操作完成后，wallet 主动推送结果给 finance（MQ/回调）
  finance 不完全依赖同步 RPC 的响应

  WalletCredit 执行后 → 发 NATS 消息 "wallet.credited"
  finance 订阅该消息 → 确认入账状态

优势：
  ✅ 解耦调用方和被调用方

劣势：
  ⚠️ 增加 MQ 依赖
  ⚠️ 消息可能延迟或丢失

适用场景：
  作为辅助通知机制，不作为主链路
```

### 15.3 我们的最佳实践

```
主方案：幂等接口 + 安全重试（ADR-05 已决策）

  所有 finance → wallet 的资金RPC 接口都是幂等的
  Kitex 客户端配置 3s 超时 + 最多重试2次
  超时后直接重试，wallet 侧幂等保障安全

  rpc.WalletClient() 配置：
    client.WithRPCTimeout(3 * time.Second)
    client.WithRetryPolicy(retry.NewFailurePolicy())  // 失败重试

辅助：
  超过重试次数仍失败 → 进入 finance 的人工处理队列
  对账兜底 → 每日比对订单状态 vs 流水状态
```

## 十六、挑战十五：敏感数据保护

> 级别：P3 一般
> 场景：数据库被攻击或内部人员查看到银行卡号、密钥等敏感信息

### 16.1 敏感数据识别

```
我们系统中的敏感数据：

  t_withdraw_account:
    bank_account（银行卡号）
    chain_address（区块链地址）
    e_wallet_phone（电子钱包手机号）

  t_currency（如果存汇率API密钥的话，但建议放ETCD不放DB）

  t_withdraw_order / t_deposit_order:
    channel_order_no（三方通道订单号，非直接敏感但属于商业机密）
```

### 16.2 业界解决方案

#### 方案一：字段级加密（应用层）——推荐

```
原理：
  敏感字段在写入 DB 前加密，读取时解密
  密钥存储在 ETCD 或 KMS（不在代码中硬编码）

  bank_account_encrypted = AES_GCM_Encrypt(bank_account, key)
  存入 DB 时用密文
  读取时解密展示（B端全部展示，C端脱敏展示）

优势：
  ✅ 数据库泄露也不会暴露明文
  ✅ 符合 PCI-DSS 等支付合规要求

劣势：
  ⚠️ 加密字段无法做精确查询（需要额外的 hash 索引）
  ⚠️ 加解密有少量性能开销

适用字段：
  bank_account、chain_address、e_wallet_phone
```

#### 方案二：展示层脱敏——推荐辅助

```
C端展示时脱敏：
  银行卡号：**** **** **** 1234（只显示后4位）
  手机号：138****5678
  链上地址：0x1234...abcd（首尾保留，中间省略）

这不替代加密存储，但减少前端泄露风险。
```

### 16.3 我们的最佳实践

```
阶段性方案：

Phase 1（上线前）：
  展示层脱敏（C端强制，B端可选）
  API密钥/密钥存 ETCD，不进 DB

Phase 2（合规要求时）：
  bank_account、chain_address、e_wallet_phone 字段级 AES-GCM 加密
  加密密钥存 ETCD 配置中心

原则：
  先做展示层脱敏（成本最低、效果明显）
  根据合规要求决定是否做存储加密
```

### 17.2 挑战之间的依赖关系

```
挑战3（精度） ──→ 挑战9（汇率） ──→ 挑战10（对账）
                                        ↑
挑战1（并发） ──→ 挑战6（热点） ──→ 挑战10（对账）
      │
      └──→ 挑战2（幂等） ──→ 挑战4（跨服务一致性） ──→ 挑战14（RPC容错）
                                    │
                              挑战5（状态机） ──→ 挑战7（防双花）

说明：
  精度是地基 → 汇率和对账都依赖精度正确
  并发和幂等是核心 → 跨服务一致性建立在这两者之上
  状态机正确性 → 冻结机制的正确性
```

### 17.3 "没那么难"的原因

```
以上 15 个挑战看似很多，但实际上：

1. 大多数有成熟的业界标准方案，不需要创新，照搬即可
   乐观锁、唯一索引幂等、DECIMAL、冻结机制 → 都是教科书级的

2. TiDB 帮我们解决了很多分布式问题
   自动分片、乐观事务支持、水平扩展 → 不需要分库分表

3. 现有工程栈已有很多基础设施
   Redis(锁/缓存)、ETCD(配置/服务发现)、NATS(MQ)、ser-blog(审计)
   不需要从零搭建

4. 我们的规模在可控范围
   不是支付宝/微信支付那种亿级QPS
   大多数方案选"主流简单方案"即可，不需要"极端高并发方案"

5. 关键决策已在 ADR 中确定
   7 个 ADR 覆盖了最核心的架构选择，方向清晰

所以核心策略是：
  用成熟的主流方案 → 不创新
  在现有工程栈上扩展 → 不引入新依赖
  先做最小可用版本 → 监控指标决定是否升级
```

## 十八、技术选型决策速查表

### 18.1 已确定的技术选型

| 挑战领域 | 选型 | 替代方案 | 选择依据 |
|---------|------|---------|---------|
| 并发控制 | 乐观锁(version) | 悲观锁(FOR UPDATE) | ADR-03：冲突率低，TiDB原生支持 |
| 幂等保障 | 流水表唯一索引 | Redis SetNX / 独立幂等表 | ADR-04：与余额更新同一事务 |
| 记账模型 | 增强单式记账 | 复式记账 | ADR-02：用户钱包不是银行核心 |
| 金额精度 | DECIMAL+string+decimal | int64最小单位 | ADR-06：多精度混合+汇率换算 |
| 跨服务通信 | 同步RPC+异步NATS | 全RPC / 全MQ / Seata | ADR-05：资金操作需知结果 |
| 币种+钱包 | 同进程(ser-wallet) | 拆两个服务 | ADR-07：避免分布式事务 |
| 账户模型 | 多子账户(user×currency×type) | 扁平账户 | ADR-01：五种子钱包隔离 |

### 18.2 Go 库选型

| 用途 | 推荐库 | 理由 |
|------|--------|------|
| 高精度十进制运算 | shopspring/decimal | Go 生态最主流（30k+ stars），支持银行家舍入 |
| 定时任务 | robfig/cron/v3 | Go 生态标准 cron 库，ser-cron 已在使用 |
| UUID/雪花ID | 工程已有 snowflake | 订单号生成：前缀 + 雪花ID |
| 状态机 | 自建（map + WHERE） | looplab/fsm 也可以，但状态不多自建足够 |
| 分布式锁 | Redis SetNX + Lua | 现有 rds 封装已支持，不引入新依赖 |

### 18.3 "不做什么"清单

```
以下是经过分析后明确不需要做的事情：

❌ 不引入分布式事务框架（Seata/DTM） → 本地事务+幂等+对账足够
❌ 不做复式记账 → 增强单式记账满足需求
❌ 不做账户拆分（Account Sharding）→ 当前并发量不需要
❌ 不用 float64 做金额计算 → DECIMAL+decimal 库
❌ 不做实时流式对账 → 离线批量对账足够
❌ 不做事件溯源（Event Sourcing）→ 架构复杂度不值得
❌ 不缓存用户余额 → 实时查DB更安全
❌ 不用 Redis 做幂等的唯一保障 → DB唯一索引才是最终兜底
❌ 不分库分表 → TiDB 自动分片
❌ 不做消息驱动的余额更新 → 同步RPC+幂等更可靠

"不做什么"和"做什么"同样重要——避免过度设计是工程纪律。
```

## 总结

```
钱包模块的技术挑战可以归结为一个核心问题：

    如何在分布式环境下保证"每一分钱都准确无误"

这个核心问题被分解为 15 个具体挑战，但解决方案都有成熟的业界实践：

  资金正确性 → 乐观锁 + 幂等 + 冻结 + DECIMAL
  系统可靠性 → 重试 + 降级 + 对账 + 告警
  性能可扩展 → TiDB + 合理索引 + 分层缓存
  安全合规性 → 四道防线 + 审计日志 + 脱敏

我们不需要发明任何新技术，只需要把业界成熟方案正确落地。
关键不在于"用什么"，而在于"每一步都做对"。
```
