## 2. 挑战1：并发余额操作的一致性

> **严重程度：P0 致命**
> 问题本质：多个请求同时操作同一用户的同一钱包余额时，如何保证最终余额正确

### 2.1 挑战描述

```
场景还原：用户 A 的 BSB 中心钱包余额 = 10000

时刻T1：充值回调上账 +5000（CreditWallet）
时刻T1：游戏结算扣款 -3000（DebitWallet）

两个请求并发到达，如果没有并发控制：

  线程1 读余额 → 10000
  线程2 读余额 → 10000
  线程1 更新余额 → 10000 + 5000 = 15000
  线程2 更新余额 → 10000 - 3000 = 7000   ← 覆盖了线程1！

最终余额 = 7000（正确应为 12000），丢失了 5000 的充值
```

### 2.2 主流方案

#### 方案A：数据库悲观锁 — SELECT FOR UPDATE（推荐）

```
使用者：支付宝、微信支付、Grab Financial、OVO
原理：在事务内先锁定余额行，其他事务排队等待

实现：
  BEGIN;
  SELECT * FROM wallet WHERE user_id=? AND currency_code=? AND wallet_type=?
    FOR UPDATE;                    -- 行级排他锁
  UPDATE wallet SET balance = balance + ? WHERE id = ?;
  INSERT INTO wallet_flow (...);   -- 同事务写流水
  COMMIT;                          -- 释放锁

优点：
  ✅ 绝对安全，数据库层面保证串行化
  ✅ 实现简单，一行 SQL 加锁
  ✅ 锁粒度细（行级），不同用户互不影响
  ✅ TiDB 完全兼容（4.0+ 默认悲观事务模式）

缺点：
  ⚠️ 同一行高并发时有排队等待（但钱包场景同一用户并发不高）
  ⚠️ 有死锁风险（可通过固定加锁顺序规避）

性能数据：
  锁持有时间 ≈ 5-15ms（1次 UPDATE + 1次 INSERT）
  QPS 上限 ≈ 单用户 60-200 TPS（远超钱包操作频率）
```

#### 方案B：乐观锁 — CAS + Version

```
使用者：电商库存扣减、部分支付系统的低冲突场景
原理：不加锁，通过版本号检测冲突，冲突时重试

实现：
  SELECT id, balance, version FROM wallet WHERE ...;
  UPDATE wallet SET balance = balance + ?, version = version + 1
    WHERE id = ? AND version = ?;  -- CAS
  if affected_rows == 0 {
    // 冲突，重试
  }

优点：
  ✅ 无锁等待，低冲突场景性能好
  ✅ 无死锁风险

缺点：
  ⚠️ 高冲突场景大量重试，性能反而差
  ⚠️ 重试期间余额状态不确定，增加资金风险
  ⚠️ 需要额外的重试逻辑（退避策略、上限控制）
  ⚠️ 资金场景零容错，重试逻辑中的 bug 可能导致资金错误
```

### 2.3 可选方案

#### 方案C：数据库原子 UPDATE（无需先 SELECT）

```
使用者：部分简化场景
原理：直接用 SQL 原子操作更新余额

实现：
  UPDATE wallet SET balance = balance + ? WHERE user_id=? AND currency_code=? AND wallet_type=?
    AND balance + ? >= 0;  -- 扣款时检查余额充足

优点：
  ✅ 无需显式加锁，SQL 本身原子
  ✅ 性能最好

缺点：
  ⚠️ 无法获取 before_balance（写流水需要）
  ⚠️ 无法在更新前做复杂业务校验
  ⚠️ 不适合需要读-计算-写的场景（如兑换涉及汇率计算）
```

#### 方案D：应用层串行化队列

```
使用者：部分高频交易系统
原理：把同一用户的所有请求投入队列，单线程顺序处理

实现：
  使用 Redis List 或 Go Channel，按 user_id 分桶
  每个桶单 goroutine 消费

优点：
  ✅ 天然串行，无并发问题
  ✅ 适合超高频操作（如实时结算）

缺点：
  ⚠️ 架构复杂度高
  ⚠️ 需要处理队列积压、消费者故障
  ⚠️ 延迟增加（入队→等待→处理）
  ⚠️ 当前场景不需要（钱包操作不是超高频）
```

### 2.4 本项目最佳实践

```
✅ 采用方案A：数据库悲观锁（SELECT FOR UPDATE）

理由：
1. 资金操作零容错，悲观锁是最安全的选择
2. 同一用户的钱包操作并发量不高（非秒杀场景），排队时间极短
3. TiDB 完全支持悲观事务（默认模式）
4. 实现最简单，现有工程 query.Q.Transaction() 直接可用
5. 支付宝、微信支付等一线资金系统的标准选择
6. Go 代码实现清晰，不需要重试逻辑的额外复杂度

工程落地：
  func CreditWalletInTx(ctx context.Context, tx *query.Query, params CreditParams) error {
      // SELECT FOR UPDATE 加行锁
      w, err := tx.Wallet.WithContext(ctx).
          Where(tx.Wallet.UserID.Eq(params.UserID)).
          Where(tx.Wallet.CurrencyCode.Eq(params.Currency)).
          Where(tx.Wallet.WalletType.Eq(params.WalletType)).
          ForUpdate().
          First()
      // ... 余额计算 + UPDATE + INSERT flow
  }
```

## 3. 挑战2：金额精度与浮点陷阱

> **严重程度：P0 致命**
> 问题本质：计算机浮点数无法精确表示十进制小数，金融计算中 0.1 + 0.2 ≠ 0.3

### 3.1 挑战描述

```
浮点精度灾难示例：

  Go: 0.1 + 0.2 = 0.30000000000000004
  Go: 1.005 * 100 = 100.49999999999999（四舍五入后变成 100 而非 101）

在钱包场景中：
  - THB 有 2 位小数：1500.50 THB
  - USDT 有 8 位小数：1.50000000 USDT
  - 汇率计算涉及乘除：5000000 VND × 汇率 = ? BSB

如果用 float64 做金额计算：
  累积误差可能导致 1 分钱的偏差
  大量交易累积后偏差会放大
  余额 + 流水不一致（审计无法通过）
```

### 3.2 主流方案

#### 方案A：最小单位整数存储（推荐）

```
使用者：Stripe、Square、支付宝、微信支付、Shopee、Grab
原理：将金额转换为最小精度单位的整数，所有计算在整数域完成

规则：
  VND/IDR（0位小数）：1:1 直接存储         5,000,000 VND → 5000000
  THB/BSB（2位小数）：×100 转为最小单位     1,500.50 THB → 150050
  USDT  （8位小数）：×10^8 转为最小单位     1.50000000 USDT → 150000000

存储类型：int64（Go）/ BIGINT（数据库）

优点：
  ✅ 整数运算零精度丢失
  ✅ CPU 原生整数运算，性能最优
  ✅ Go 原生 int64，无需第三方库
  ✅ 业界公认的金融计算标准方案
  ✅ Stripe API 文档明确要求金额以 "smallest currency unit" 传递

缺点：
  ⚠️ 需要在入口和出口做单位转换
  ⚠️ 跨币种计算时需注意精度系数不同

Stripe 官方文档原文：
  "All API requests expect amounts to be provided in a currency's smallest unit.
   For example, to charge 10 USD, provide an amount value of 1000 (that is, 1000 cents)."
```

#### 方案B：DECIMAL 类型

```
使用者：传统银行系统、会计系统
原理：数据库使用 DECIMAL(20,8) 存储，保证数据库层面的精确

实现：
  CREATE TABLE wallet (
    balance DECIMAL(20,8) NOT NULL DEFAULT 0.00000000
  );

优点：
  ✅ 数据库层面精确，无精度丢失
  ✅ 直觉友好，存储值即为展示值

缺点：
  ⚠️ Go 没有原生 decimal 类型，需要引入 shopspring/decimal 库
  ⚠️ DB → Go 的转换过程可能引入精度问题
  ⚠️ 计算性能低于整数运算
  ⚠️ 与现有工程金额字段（int32/int64）不一致
```

### 3.3 可选方案

#### 方案C：字符串存储 + 精确计算库

```
使用者：部分跨语言金融系统
原理：金额以字符串传输和存储，计算时使用精确计算库

实现：
  import "github.com/shopspring/decimal"
  a := decimal.NewFromString("0.1")
  b := decimal.NewFromString("0.2")
  c := a.Add(b)  // "0.3"

优点：
  ✅ 任意精度，无上限
  ✅ 跨语言兼容性好

缺点：
  ⚠️ 性能差（字符串解析开销）
  ⚠️ 引入第三方依赖
  ⚠️ 不适合高频计算场景
```

### 3.4 本项目最佳实践

```
✅ 采用方案A：最小单位整数存储

理由：
1. 与现有工程一致（ser-item 的 BasePrice 用 int32）
2. Go 原生 int64，零依赖
3. Stripe/支付宝/微信等全部采用此方案
4. 整数运算性能最优，汇率计算全部在整数域完成
5. int64 最大 9.2×10^18，以 USDT 8位小数算可表示 922 亿 USDT，远超需求

精度系数统一存在 currency_config 表：
  VND: decimal_digits=0, precision_factor=1
  THB: decimal_digits=2, precision_factor=100
  BSB: decimal_digits=2, precision_factor=100
  USDT: decimal_digits=8, precision_factor=100000000

转换公式：
  入库：db_amount = display_amount × precision_factor
  出库：display_amount = db_amount / precision_factor
```

## 4. 挑战3：分布式环境下的幂等性

> **严重程度：P0 致命**
> 问题本质：网络超时/重试/回调重发导致同一请求被执行多次，如何保证只生效一次

### 4.1 挑战描述

```
典型灾难场景：

场景1 — 充值回调重复：
  三方支付回调通知"充值成功"
  第一次回调：CreditWallet +5000 → 余额 15000 ✅
  网络超时，三方没收到 200 OK
  第二次回调：CreditWallet +5000 → 余额 20000 ❌ 重复上账！

场景2 — RPC 超时重试：
  财务服务调用 CreditWallet 超时（但实际已执行成功）
  财务重试：再次调用 CreditWallet → 再次上账 ❌

场景3 — 人工加款审核：
  审核员点击"通过"后页面未刷新
  再次点击"通过" → 又调了一次 CreditWallet ❌
```

### 4.2 主流方案

#### 方案A：唯一约束 + 幂等键（推荐）

```
使用者：支付宝、微信支付、Stripe、PayPal
原理：为每次操作生成唯一幂等键，通过数据库 UNIQUE 约束保证只写入一次

实现层次：
  第一层（快速返回）：查 wallet_flow 是否已存在相同 idempotent_key
    存在 → 直接返回成功（幂等）
  第二层（最终防线）：INSERT wallet_flow 时 UNIQUE 约束
    冲突 → 并发重复，忽略错误返回成功

幂等键格式：{biz_type}:{ref_order_no}:{wallet_type}
  示例："1:R20260226001:1"  → 充值:订单号:中心钱包

Stripe 官方实践：
  Stripe API 支持 Idempotency-Key header
  相同 key 的请求返回缓存的第一次响应
  key 有 24 小时有效期

优点：
  ✅ 数据库层面保证，无论应用层如何重试都安全
  ✅ 实现简单明确
  ✅ 无额外基础设施依赖

缺点：
  ⚠️ 需要调用方传递幂等键（增加接口契约）
  ⚠️ UNIQUE 索引有微小的写入性能开销
```

#### 方案B：状态机幂等

```
使用者：所有有状态的业务系统
原理：通过订单/冻结的状态机检查，已终态的操作直接返回成功

实现：
  充值回调：订单状态 ≠ 待支付 → 返回成功（已处理过）
  确认扣款：冻结状态 = 已确认 → 返回成功
  解冻操作：冻结状态 = 已解冻 → 返回成功

优点：
  ✅ 逻辑直觉清晰
  ✅ 不需要额外的幂等键字段

缺点：
  ⚠️ 只能防住"状态已变更"的重复，无法防住并发的首次重复
  ⚠️ 必须配合方案A使用，不能单独作为幂等方案
```

### 4.3 可选方案

#### 方案C：Redis 幂等缓存

```
使用者：作为辅助层使用
原理：用 Redis SETNX 记录已处理的请求ID，TTL 自动过期

实现：
  key = "idempotent:{biz_type}:{order_no}"
  if redis.SetNX(key, "1", 24h) == false {
    return success  // 已处理
  }
  // 执行业务逻辑
  // 失败时 DEL key

优点：
  ✅ 查询速度快（微秒级）
  ✅ 可减少数据库查询

缺点：
  ⚠️ Redis 重启后缓存丢失，不是最终保障
  ⚠️ 需要与数据库 UNIQUE 配合使用
  ⚠️ 增加了 Redis 依赖
```

#### 方案D：Token 令牌机制

```
使用者：前端防重提交场景
原理：服务端生成一次性 token，提交时校验并消费

实现：
  GET  /api/recharge/token → 返回一次性 token
  POST /api/recharge/create { token: "xxx", ... }
  服务端：检查 token 是否有效 → 消费 token → 执行业务

优点：
  ✅ 可精确控制前端重复提交

缺点：
  ⚠️ 只防前端重复，不防后端 RPC 重复
  ⚠️ 增加一次 HTTP 往返
```

### 4.4 本项目最佳实践

```
✅ 采用方案A + B 组合：两层幂等保障

第一层：状态机快速检查（事务外，0 锁竞争）
  → 已终态的重复请求快速返回
第二层：UNIQUE(idempotent_key, delete_at) 最终防线（事务内）
  → 并发的首次重复请求由数据库拦截

为什么不需要方案C（Redis 缓存层）：
  - 状态机检查已经是快速路径，避免了大部分无效事务
  - UNIQUE 约束是最终保障，不依赖 Redis 可靠性
  - 减少 Redis 操作环节，简化架构

为什么不需要方案D（Token 令牌）：
  - RPC 接口的幂等由方案A+B保证，前端重复提交由 Redis 分布式锁拦截（见挑战17）
```

## 5. 挑战4：跨服务数据一致性

> **严重程度：P1 严重**
> 问题本质：财务服务调用钱包 CreditWallet 时跨服务调用，如何保证两侧数据一致

### 5.1 挑战描述

```
跨服务一致性困境：

财务服务                          钱包服务
    │                                │
    ├── 收到回调，更新订单为"已支付"    │
    ├── RPC 调用 CreditWallet ──────▶│── 上账成功 ✅
    ├── 等待响应...                   │
    │   ← 网络超时（但钱包已上账）     │
    ├── 标记为"上账失败" ❌            │
    │                                │
    └── 数据不一致：钱包已加钱，但财务认为失败

或者反向：
    ├── RPC 调用 CreditWallet ──────▶│── 上账成功 ✅
    ├── 收到成功响应                   │
    ├── 更新本地订单状态...           │
    │   ← 本地事务失败 ❌             │
    │                                │
    └── 数据不一致：钱包已加钱，但财务订单还是"待上账"
```

### 5.2 主流方案

#### 方案A：同步 RPC + 幂等重试（推荐）

```
使用者：支付宝/微信的回调上账、Stripe Webhooks
原理：调用方失败后通过定时任务重试，被调用方保证幂等

流程：
  1. 财务收到回调 → 同步 RPC 调钱包 CreditWallet
  2. 成功 → 更新财务订单状态为"已上账"
  3. 失败/超时 → 财务订单保持"已支付待上账"
  4. 财务定时任务扫描"已支付待上账"订单 → 重新调 CreditWallet
  5. 钱包幂等保证 → 重复调用不会重复上账

核心保障：
  - 钱包侧幂等：即使被调100次，只会执行1次
  - 财务侧重试：保证最终一定会调成功
  - 最终一致性：可能有短暂的状态不一致，但最终一定一致

优点：
  ✅ 实现简单，无额外基础设施
  ✅ 业界最主流的跨服务一致性方案
  ✅ 支付宝/微信/Stripe 的回调+重试就是这个模式

缺点：
  ⚠️ 有短暂的不一致窗口（定时任务间隔）
  ⚠️ 需要财务侧实现重试逻辑（财务方的责任）
```

#### 方案B：本地消息表（Outbox Pattern）

```
使用者：大型金融系统、银行间转账
原理：调用方把"待执行的操作"先写入本地消息表，再由后台任务投递

流程：
  1. 财务在本地事务中：
     - 更新订单状态为"已支付"
     - 写入 outbox 表：{action: "credit_wallet", payload: {...}}
  2. 后台任务扫描 outbox 表 → RPC 调钱包 → 成功后标记已投递
  3. 钱包幂等保证

优点：
  ✅ 本地事务保证"支付状态+消息"原子性
  ✅ 不依赖 RPC 的同步成功

缺点：
  ⚠️ 需要额外的 outbox 表和扫描任务
  ⚠️ 增加了实现复杂度
  ⚠️ 对于当前场景是过度设计
```

### 5.3 可选方案

#### 方案C：TCC（Try-Confirm-Cancel）

```
使用者：分布式事务框架（Seata、DTM）
原理：
  Try：冻结资源
  Confirm：确认执行
  Cancel：回滚释放

优点：
  ✅ 强一致性
缺点：
  ⚠️ 实现复杂度极高（每个操作需要正向/反向两套逻辑）
  ⚠️ 需要分布式事务框架支持
  ⚠️ 当前项目完全不需要
```

#### 方案D：Saga 模式

```
使用者：长流程编排（订单→支付→物流→库存）
原理：拆分为多个本地事务+补偿事务，失败时逆序执行补偿

优点：
  ✅ 适合长链路编排
缺点：
  ⚠️ 补偿逻辑复杂
  ⚠️ 钱包上账是简单的点对点调用，Saga 过度
```

### 5.4 本项目最佳实践

```
✅ 采用方案A：同步 RPC + 幂等重试

理由：
1. 钱包与财务的交互模式简单：充值上账/提现冻结/确认扣款/解冻
2. 每个操作都有明确的幂等键
3. 支付宝/微信/Stripe 的亿级日交易量也是这个模式
4. 方案B/C/D 在当前场景都是过度设计

钱包侧的职责（我们负责）：
  ✅ 保证所有 RPC 接口幂等
  ✅ 正确返回成功/失败状态码

财务侧的职责（对方负责）：
  ✅ 实现重试机制（定时任务扫描异常订单）
  ✅ 正确传递幂等键
```

## 6. 挑战5：冻结-确认/解冻的原子性

> **严重程度：P0 致命**
> 问题本质：提现涉及冻结→等待审核→确认扣款/解冻三个阶段，如何保证每个阶段的原子性

### 6.1 挑战描述

```
提现冻结的三阶段挑战：

阶段1 — 冻结可能跨两个钱包：
  用户提现 1500 THB，中心钱包余额 1200，奖励钱包 500
  需要冻结：中心 1200 + 奖励 300 = 1500
  问题：冻中心成功但冻奖励失败 → 中心的 1200 被悬挂冻结

阶段2 — 解冻必须精确还原：
  审核驳回 → 需要解冻中心 1200 + 奖励 300
  问题：如果不记录每个钱包冻了多少，解冻时无法正确分配

阶段3 — 确认扣款的并发安全：
  出款成功回调 → ConfirmDebit
  问题：回调可能重复，如何保证只扣一次？
```

### 6.2 主流方案

#### 方案A：冻结明细表 + 同事务操作（推荐）

```
使用者：支付宝/微信提现、银行转账
原理：
  - 冻结时：同事务创建 freeze_detail 记录 + 更新 wallet.frozen_amount
  - 解冻时：根据 freeze_detail 精确还原每个钱包的冻结金额
  - 确认时：根据 freeze_detail 精确扣减每个钱包的余额和冻结金额

实现要点：
  冻结事务：
    BEGIN;
    SELECT wallet WHERE user_id=? AND wallet_type=CENTER FOR UPDATE;
    SELECT wallet WHERE user_id=? AND wallet_type=REWARD FOR UPDATE;  // 按固定顺序加锁
    UPDATE wallet SET frozen_amount += 1200 WHERE wallet_type=CENTER;
    UPDATE wallet SET frozen_amount += 300  WHERE wallet_type=REWARD;
    INSERT freeze_detail (wallet_type=CENTER, freeze_amount=1200, status=冻结中);
    INSERT freeze_detail (wallet_type=REWARD, freeze_amount=300,  status=冻结中);
    INSERT withdraw_order (...);
    COMMIT;

  解冻事务：
    BEGIN;
    SELECT freeze_detail WHERE order_no=? AND status=冻结中 FOR UPDATE;
    // 遍历明细，精确解冻每个钱包
    UPDATE wallet SET frozen_amount -= 1200 WHERE wallet_type=CENTER;
    UPDATE wallet SET frozen_amount -= 300  WHERE wallet_type=REWARD;
    UPDATE freeze_detail SET status=已解冻 WHERE order_no=?;
    COMMIT;

优点：
  ✅ 精确追踪每个钱包的冻结金额
  ✅ 解冻/确认扣款时有据可依
  ✅ freeze_detail 本身提供审计能力
  ✅ 状态机防止重复操作

缺点：
  ⚠️ 多了一张表的 IO 开销（但提现频率不高，可以接受）
```

#### 方案B：单钱包简化冻结（不适用本项目）

```
适用场景：只有一种钱包类型的简单系统
原理：直接在 wallet 表记录冻结金额，不需要明细表

优点：
  ✅ 简单

缺点：
  ⚠️ 不适用于本项目（提现可能跨中心+奖励两个钱包）
  ⚠️ 多笔提现同时冻结时无法区分各笔的冻结量
```

### 6.3 本项目最佳实践

```
✅ 采用方案A：冻结明细表 + 同事务操作

关键实现要点：
1. 冻结涉及多行钱包 → 必须按固定顺序加锁（防死锁）
2. freeze_detail 记录每个钱包的冻结分配 → 解冻时精确还原
3. freeze_detail.status 状态机 → 防止重复确认/解冻
4. 冻结+订单创建同事务 → 防止"冻了钱但没建单"的悬挂状态

消费优先级规则（需求硬性要求）：
  冻结时：先冻中心钱包（可用余额），不足部分冻奖励钱包
  这与消费扣款优先级一致：中心 → 奖励
```

## 7. 挑战6：缓存与数据库的一致性

> **严重程度：P1 严重**
> 问题本质：引入 Redis 缓存后，如何保证缓存与 DB 的数据一致

### 7.1 挑战描述

```
经典缓存不一致场景：

场景1 — 写后更新的竞态：
  线程1：事务提交（余额 1000→1500）
  线程2：事务提交（余额 1500→1800）
  线程2：SET cache = 1800
  线程1：SET cache = 1500    ← 旧值覆盖新值！

场景2 — 先删缓存再更新DB：
  线程1：DEL cache
  线程2：读缓存 miss → 读 DB（旧值 1000）→ SET cache = 1000
  线程1：UPDATE DB → 1500
  缓存停留在旧值 1000 ← 不一致！

场景3 — 先更新DB再删缓存：
  线程1：UPDATE DB → 1500
  线程1：DEL cache（此时崩溃）
  缓存仍是旧值 ← 不一致（但 TTL 会自动修正）
```

### 7.2 主流方案

#### 方案A：Cache-Aside + 写后删除（推荐）

```
使用者：支付宝/微信/阿里巴巴/Facebook
原理：
  读：先读缓存 → 未命中则读DB → 写入缓存
  写：先更新DB（事务内）→ 事务提交后删除缓存

流程（以余额为例）：
  读余额：
    1. GET wallet:bal:{uid}:{currency}
    2. 命中 → 返回
    3. 未命中 → SELECT FROM wallet → SET 缓存（TTL 5min）→ 返回

  写余额（CreditWallet/DebitWallet）：
    1. BEGIN → UPDATE wallet → INSERT flow → COMMIT
    2. DEL wallet:bal:{uid}:{currency}  ← 事务提交后执行

为什么是"删"而不是"更新"：
  并发写时：
    线程1：DEL cache
    线程2：DEL cache
    下次读：从 DB 读到最新值（永远正确）

  如果是"更新"：
    线程2：SET cache = 1800（正确）
    线程1：SET cache = 1500（旧值覆盖！）

优点：
  ✅ 实现简单，最常用
  ✅ 并发安全（最差情况是 TTL 后自动修正）
  ✅ 数据库始终是权威数据源

缺点：
  ⚠️ DEL 与 UPDATE DB 之间有极短窗口可能读到旧缓存
  ⚠️ DEL 失败时需等 TTL 过期（可通过异步补偿解决）
```

#### 方案B：Write-Through（写穿透）

```
使用者：低写频率的配置类数据
原理：写操作同时更新 DB 和缓存

适用：币种配置、汇率数据（由定时任务统一更新）

优点：
  ✅ 缓存始终与 DB 同步
缺点：
  ⚠️ 并发写时存在竞态（同场景1）
  ⚠️ 不适合余额这类高写频率数据
```

### 7.3 可选方案

#### 方案C：延迟双删

```
使用者：对一致性要求极高的场景
原理：删缓存 → 更新DB → 延迟N毫秒 → 再删一次缓存

实现：
  DEL cache
  UPDATE DB
  sleep(500ms)
  DEL cache  // 第二次删除，清除读线程可能写入的旧值

优点：
  ✅ 解决了"先删后更新"的竞态问题
缺点：
  ⚠️ 延迟增加（500ms 是额外开销）
  ⚠️ 实现复杂度增加
  ⚠️ 对于余额场景，Cache-Aside + TTL 已足够
```

#### 方案D：订阅 Binlog 异步更新缓存

```
使用者：大型分布式系统（阿里 Canal + MQ）
原理：监听数据库 Binlog，数据库变更后自动更新/删除缓存

优点：
  ✅ 解耦，业务代码不需要管缓存
缺点：
  ⚠️ 重度基础设施依赖（Canal / Debezium）
  ⚠️ 引入 MQ 增加复杂度
  ⚠️ 当前项目规模完全不需要
```

### 7.4 本项目最佳实践

```
✅ 采用方案A：Cache-Aside + 写后删除（余额、提现账户等）
✅ 采用方案B：Write-Through（汇率数据，由定时任务直接写 Redis）

各数据类型的缓存策略：

| 数据           | 策略          | TTL   | 更新方式            |
|---------------|---------------|-------|-------------------|
| 用户余额       | Cache-Aside   | 5min  | 写操作后 DEL        |
| 币种列表       | Cache-Aside   | 1h    | 配置变更后 DEL      |
|               | + 本地缓存     | 10min | sync.Map           |
| 汇率数据       | Write-Through | 自然  | 定时任务直接写       |
| 用户当前币种   | Cache-Aside   | 7d    | 切换时 SET          |

DEL 失败的兜底：TTL 过期后缓存自然消失，下次读重新从 DB 加载
最差影响：用户看到的余额延迟最多 5 分钟更新（安全性不受影响，因为写操作走 DB）
```

## 8. 挑战7：汇率实时性与计算精度

> **严重程度：P2 重要**
> 问题本质：汇率是浮动的，如何保证获取的及时性、计算的精确性、以及"用户看到的汇率"="实际成交的汇率"

### 8.1 挑战描述

```
挑战维度：

1. 实时性：汇率每分钟都在变，但不能每次操作都调三方 API
2. 可靠性：三方 API 可能超时/返回异常值，不能影响业务
3. 精度：汇率计算涉及乘除，浮点精度可能导致 1 分钱偏差
4. 一致性：用户在充值/兑换页面看到的汇率，到提交时可能已变化
5. 偏差检测：三方汇率可能突然跳变（市场波动或API异常），需要告警
```

### 8.2 主流方案

#### 方案A：定时拉取 + 多源聚合 + 偏差检测（推荐）

```
使用者：Grab Financial、OVO、支付宝国际版
原理：定时从多个三方 API 拉取汇率，聚合后更新到平台

流程：
  ┌──────────┐  ┌──────────┐  ┌──────────┐
  │ API源-1   │  │ API源-2   │  │ API源-3   │
  └────┬─────┘  └────┬─────┘  └────┬─────┘
       │  并发请求(5s超时)  │              │
       └─────────┬─────────┘──────────────┘
                 ▼
        ┌────────────────┐
        │  聚合规则：      │
        │  排除离群值      │
        │  取有效值中位数  │
        └────────┬───────┘
                 ▼
        ┌────────────────┐
        │  偏差检测：      │
        │  |新-旧|/旧 > 阈值 → 告警 │
        │  正常 → 仅更新实时汇率      │
        │  超阈值 → 更新平台汇率      │
        └────────┬───────┘
                 ▼
        ┌────────────────┐
        │  计算入/出款汇率 │
        │  入款 = 平台率 × (1+浮动%) │
        │  出款 = 平台率 × (1-浮动%) │
        └────────────────┘

优点：
  ✅ 多源冗余，单个 API 挂不影响
  ✅ 偏差检测防止异常汇率影响业务
  ✅ 入/出款汇率分离，平台可控利差

缺点：
  ⚠️ 定时拉取有延迟（3-5分钟）
  ⚠️ 需要选择可靠的三方 API 源
```

#### 方案B：实时查询三方 API

```
使用者：外汇交易平台（对实时性要求极高）
原理：每次用户操作时实时查三方 API 获取最新汇率

优点：
  ✅ 汇率最新
缺点：
  ⚠️ 每次操作增加外部 HTTP 调用（5s 超时风险）
  ⚠️ 三方 API 不可用时业务中断
  ⚠️ API 调用频率限制（Rate Limit）
  ⚠️ 互联网钱包不是外汇交易，不需要这么高的实时性
```

### 8.3 汇率计算精度方案

#### 整数域汇率计算（推荐）

```
核心规则：汇率也用 int64 存储，×10^8 转为整数

示例：1 BSB = 2,300 VND
  存储值 real_rate = 2300 × 10^8 = 230,000,000,000

兑换计算（5,000,000 VND → BSB）：
  // 第一步：VND 金额（最小单位）÷ 汇率 = BSB 金额
  // vnd_amount = 5000000 （VND 精度系数=1，最小单位=1VND）
  // bsb_rate_per_vnd = 230000000000 （1 BSB = 2300 VND，×10^8）

  // 整数域计算：
  bsb_raw = vnd_amount × 10^8 / bsb_rate_per_vnd
         = 5000000 × 100000000 / 230000000000
         = 500000000000000 / 230000000000
         = 2173 （BSB 最小单位，即 21.73 BSB）

  // 精度保证：先乘后除，避免中间结果精度丢失
  // 溢出检查：5000000 × 10^8 = 5×10^14 < 9.2×10^18 (int64 max) ✅
```

### 8.4 汇率一致性方案

```
用户看到的汇率 vs 实际成交的汇率：

方案：乐观展示 + 事务内重查
  1. 用户进入兑换页面 → 读缓存中的汇率展示
  2. 用户提交兑换 → 事务内重新读 DB 最新汇率
  3. 如果汇率变化超过阈值（如 0.5%）→ 返回"汇率已变化，请重新确认"
  4. 用户确认后以最新汇率成交

这样既保证了用户体验（不用每次刷新汇率），又保证了成交精度。
```

### 8.5 本项目最佳实践

```
✅ 定时拉取（3分钟一次）+ 多源聚合（3个三方 API）+ 偏差检测
✅ 汇率全程 int64 × 10^8 整数域计算
✅ 先乘后除避免中间精度丢失
✅ 入/出款汇率 = 平台汇率 ×（1 ± 浮动百分比）
✅ 提交时事务内重查最新汇率
```

## 9. 挑战8：热点账户问题

> **严重程度：P2 重要**
> 问题本质：当某个用户（或系统账户）被高频操作时，行级锁导致所有请求排队

### 9.1 挑战描述

```
热点账户场景（当前项目可能触及）：

场景1 — 系统平台账户（如果有的话）：
  所有用户充值都对"平台收款账户"加钱 → 所有 CreditWallet 排队

场景2 — 高活跃用户：
  某用户频繁操作（充值+游戏结算+兑换+提现）→ 同一行频繁加锁

场景3 — 活动期间集中操作：
  活动开始 → 大量用户同时兑换 → 如果涉及统一的 BSB 汇率更新 → 锁竞争

本项目实际分析：
  - 当前设计中没有"系统平台账户"（钱包是用户级别的）
  - 同一用户的操作频率有限（不可能每秒充值10次）
  - 热点账户在当前阶段不是主要瓶颈
  - 但需要了解方案，以备业务增长后应对
```

### 9.2 主流方案

#### 方案A：用户级别行锁 + 无系统汇总账户（当前适用）

```
使用者：大多数 C 端钱包系统
原理：钱包按 (user_id, currency_code, wallet_type) 隔离，不同用户完全不冲突

性能分析：
  同一用户同一钱包的锁竞争 = 极低（用户不会秒级多操作）
  不同用户 = 零冲突
  这是行级锁的天然优势

判断：当前阶段无热点账户问题 ✅
```

#### 方案B：拆分账户（缓冲层）

```
使用者：支付宝/微信的核心热点账户
原理：将一个热点账户拆分为 N 个子账户，请求随机分配到子账户

实现：
  platform_account_1 → balance: 1000
  platform_account_2 → balance: 1000
  platform_account_3 → balance: 1000
  ...
  总余额 = SUM(所有子账户)

  充值上账：随机选一个子账户加钱
  查总余额：SUM 或 异步聚合

优点：
  ✅ 把单行锁分散为 N 行锁，吞吐量提升 N 倍
  ✅ 支付宝/微信核心账户的经典方案

缺点：
  ⚠️ 复杂度大增（查余额需聚合、扣款需跨子账户协调）
  ⚠️ 当前项目完全不需要
```

### 9.3 可选方案

#### 方案C：异步合并记账

```
使用者：高频投注/结算系统
原理：把多笔小额操作合并为一笔批量操作

实现：
  收集 100ms 窗口内的所有 CreditWallet 请求
  合并为一笔：总金额 = SUM(各笔金额)
  单次事务写入

优点：
  ✅ 极大降低锁竞争
缺点：
  ⚠️ 增加延迟（100ms 窗口）
  ⚠️ 单笔失败影响整批
  ⚠️ 当前项目无此需求
```

#### 方案D：内存记账 + 异步落盘

```
使用者：游戏虚拟货币系统、计数器系统
原理：在 Redis 中操作余额，定时同步到数据库

实现：
  INCRBY wallet:bal:123:BSB 5000
  定时任务：读 Redis → 写 DB

优点：
  ✅ 极高性能
缺点：
  ⚠️ Redis 崩溃可能丢失数据
  ⚠️ 不适合真实资金场景
  ⚠️ 资金安全零容错，不可接受
```

### 9.4 本项目最佳实践

```
✅ 当前阶段：方案A（用户级行锁，无热点问题）

预留意识：
  - 如果未来引入"平台汇总账户"→ 考虑方案B（拆分账户）
  - 如果未来引入高频游戏结算 → 考虑方案C（异步合并）
  - 方案D 永远不用（真实资金不可能放 Redis 做主存储）

当前设计天然规避热点的原因：
  1. 没有系统级汇总账户
  2. 每个用户独立钱包行，互不影响
  3. 同一用户的操作频率有限
  4. TiDB 行级锁粒度足够细
```

## 10. 挑战9：死锁预防与事务性能

> **严重程度：P1 严重**
> 问题本质：多行加锁操作可能产生死锁，事务持锁时间过长会降低并发

### 10.1 挑战描述

```
死锁场景复现：

兑换操作需要同时锁两行钱包（VND + BSB）：

  事务1（兑换 VND→BSB）：
    SELECT * FROM wallet WHERE currency=VND FOR UPDATE;  ← 锁VND
    SELECT * FROM wallet WHERE currency=BSB FOR UPDATE;  ← 等待BSB

  事务2（兑换 BSB→...或其他涉及BSB的操作）：
    SELECT * FROM wallet WHERE currency=BSB FOR UPDATE;  ← 锁BSB
    SELECT * FROM wallet WHERE currency=VND FOR UPDATE;  ← 等待VND

  → 死锁！事务1等事务2释放BSB，事务2等事务1释放VND

事务性能问题：
  锁持有时间 = SELECT FOR UPDATE → 业务计算 → UPDATE → INSERT → COMMIT
  如果在事务内做了外部调用（HTTP/RPC）→ 锁持有时间大幅增加 → 其他请求排队
```

### 10.2 主流方案

#### 方案A：固定加锁顺序（推荐）

```
使用者：所有涉及多行锁的数据库应用（教科书级方案）
原理：所有事务按相同的全局顺序加锁，消除循环等待

规则：按 (currency_code ASC, wallet_type ASC) 排序后依次加锁

示例：兑换 VND → BSB
  先锁 BSB（字母序 B < V）
  再锁 VND

示例：提现冻结涉及中心+奖励
  先锁 wallet_type=1（中心）
  再锁 wallet_type=2（奖励）

实现：
  func lockWalletsInOrder(tx *query.Query, wallets []WalletKey) {
      // 按 (currency_code, wallet_type) 排序
      sort.Slice(wallets, func(i, j int) bool {
          if wallets[i].CurrencyCode != wallets[j].CurrencyCode {
              return wallets[i].CurrencyCode < wallets[j].CurrencyCode
          }
          return wallets[i].WalletType < wallets[j].WalletType
      })
      for _, w := range wallets {
          tx.Wallet.Where(...).ForUpdate().First()  // 依次加锁
      }
  }

优点：
  ✅ 完全消除死锁可能
  ✅ 零额外开销
  ✅ 实现简单（排序后循环加锁）

缺点：
  ⚠️ 需要团队所有人遵守此规范（通过封装函数强制）
```

#### 方案B：死锁检测 + 自动重试

```
使用者：TiDB/MySQL 内置机制
原理：数据库检测到死锁后自动终止一个事务，应用层捕获后重试

实现：
  err := tx.Transaction(func(tx *query.Query) error { ... })
  if isDeadlockError(err) {
      // 重试
      return tx.Transaction(func(tx *query.Query) error { ... })
  }

TiDB 内置死锁检测：
  TiDB 检测到死锁后会返回错误码 1213 (ER_LOCK_DEADLOCK)
  应用层可以捕获此错误并重试

优点：
  ✅ 兜底方案，即使没有固定顺序也不会永久死锁
缺点：
  ⚠️ 重试增加延迟和复杂度
  ⚠️ 应该是兜底而非主策略
```

### 10.3 事务性能优化方案

```
核心原则：最小化锁持有时间

✅ 事务外做的事（无锁）：
  - 参数校验
  - 幂等键预检查
  - 汇率查询（进程内调用）
  - 金额计算
  - 订单号生成

✅ 事务内做的事（持锁）：
  - SELECT FOR UPDATE（加锁）
  - UPDATE balance（更新余额）
  - INSERT wallet_flow（写流水）
  - INSERT/UPDATE 业务表
  - COMMIT（释放锁）

❌ 绝对不在事务内做的事：
  - 外部 HTTP 调用（三方 API）
  - 跨服务 RPC 调用
  - Redis 操作
  - 文件 IO
  - 大量日志输出

锁持有时间目标：
  正常操作 < 10ms（1-2次 UPDATE + 1次 INSERT）
  兑换操作 < 20ms（2-3次 UPDATE + 2-3次 INSERT + 1次 INSERT 兑换记录）
```

### 10.4 本项目最佳实践

```
✅ 方案A（固定加锁顺序）作为主策略：封装 lockWalletsInOrder 函数，所有涉及多行的操作统一使用
✅ 方案B（死锁检测）作为兜底：捕获 1213 错误码自动重试一次
✅ 事务内只做 DB 操作，所有非 DB 操作移到事务外
```

## 11. 挑战10：防超扣与资金安全校验

> **严重程度：P0 致命**
> 问题本质：余额扣减后不能为负，冻结金额不能超过余额，这些校验必须绝对可靠

### 11.1 挑战描述

```
超扣灾难场景：

场景1 — 余额为负：
  用户余额 1000，同时提现 800 和扣款 500
  如果并发控制有漏洞 → 余额变为 -300 → 资金损失

场景2 — 冻结超额：
  用户余额 1000，冻结 700（提现A），再冻结 500（提现B）
  可用余额 = 1000 - 700 = 300，但第二笔尝试冻结 500 → 应该拒绝

场景3 — 流水不一致：
  before_balance = 1000, amount = 500, after_balance = 1600
  1000 + 500 ≠ 1600 → 审计发现异常但已经晚了
```

### 11.2 主流方案

#### 方案A：多层防御体系（推荐）

```
使用者：所有正规资金系统

第一层 — 代码层校验（事务内，锁保护下）：
  // 扣款前检查
  available := wallet.Balance - wallet.FrozenAmount
  if available < amount {
      return errors.New("余额不足")
  }

  // 冻结前检查
  available := wallet.Balance - wallet.FrozenAmount
  if available < freezeAmount {
      return errors.New("可用余额不足")
  }

  // 更新后检查（防御性编程）
  newBalance := wallet.Balance - amount
  if newBalance < 0 {
      return errors.New("余额异常：计算结果为负")  // 不应该触发，触发说明有 bug
  }

第二层 — 流水一致性校验（事务内）：
  flow := &WalletFlow{
      BeforeBalance: wallet.Balance,
      Amount:        amount,
      AfterBalance:  wallet.Balance + amount,  // 入账
  }
  // 校验：before + amount == after（入账）或 before - amount == after（出账）
  if flow.Direction == IN && flow.BeforeBalance + flow.Amount != flow.AfterBalance {
      panic("流水一致性校验失败")  // 这是代码 bug，必须立即发现
  }

第三层 — 数据库约束（可选增强）：
  -- TiDB 支持 CHECK 约束（5.0+）
  ALTER TABLE wallet ADD CONSTRAINT chk_balance CHECK (balance >= 0);
  ALTER TABLE wallet ADD CONSTRAINT chk_frozen CHECK (frozen_amount >= 0);
  ALTER TABLE wallet ADD CONSTRAINT chk_frozen_le_balance CHECK (frozen_amount <= balance);

第四层 — 定期对账（运营级）：
  每日凌晨：SUM(所有入账流水) - SUM(所有出账流水) == 当前余额
  不等 → 告警 → 人工介入（不自动修正）

优点：
  ✅ 四层防线，任何一层漏过都有下一层兜住
  ✅ 代码层校验 + 悲观锁 = 核心保障
  ✅ 流水一致性 = 开发期的 bug 检测器
  ✅ DB 约束 = 最终防线
  ✅ 对账 = 事后发现机制
```

### 11.3 本项目最佳实践

```
✅ 采用方案A：四层防御体系

必须实现：
  ✅ 第一层：代码层余额非负校验（事务内，FOR UPDATE 锁保护）
  ✅ 第二层：流水 before + amount = after 一致性断言
  ✅ 第三层：TiDB CHECK 约束（需确认 TiDB 版本支持情况，不支持则跳过）

建议实现：
  □ 第四层：每日对账定时任务（可二期补充）

关键代码原则：
  - 所有扣款操作必须先查余额再扣款，不允许"盲扣"
  - 余额、冻结金额的修改必须在 FOR UPDATE 锁内
  - 余额计算结果为负时，不是返回错误而是 panic（说明有 bug）
```

## 14. 挑战13：审计追溯与对账

> **严重程度：P2 重要**
> 问题本质：所有资金操作必须可追溯，余额与流水必须可验证一致

### 14.1 挑战描述

```
审计追溯需求：
  1. 任何时刻的余额变动都能查到"谁在什么时候因为什么操作了多少钱"
  2. 当前余额 = 初始余额 + SUM(所有入账) - SUM(所有出账)
  3. 每条流水的 before_balance + amount = after_balance（入账）必须成立
  4. 异常时能快速定位哪条流水出了问题

对账需求：
  1. 内部对账：余额 vs 流水汇总
  2. 外部对账：平台记录 vs 三方渠道记录（财务负责，钱包提供数据）
```

### 14.2 主流方案

#### 方案A：Append-Only 流水 + before/after 快照（推荐）

```
使用者：支付宝/微信/Stripe
原理：每条流水记录变动前后的余额快照，形成可验证的链

wallet_flow 表设计：
  id | user_id | amount | before_balance | after_balance | idempotent_key | create_at

验证链：
  Flow-1: before=0,     amount=+5000, after=5000     ✅ 0+5000=5000
  Flow-2: before=5000,  amount=-2000, after=3000     ✅ 5000-2000=3000
  Flow-3: before=3000,  amount=+1000, after=4000     ✅ 3000+1000=4000
  当前余额 = 4000 = 最后一条流水的 after_balance     ✅

对账 SQL：
  -- 验证当前余额与流水汇总一致
  SELECT w.user_id, w.currency_code, w.wallet_type,
         w.balance AS 当前余额,
         COALESCE(SUM(CASE WHEN f.direction=1 THEN f.amount ELSE -f.amount END), 0) AS 流水计算余额
  FROM wallet w
  LEFT JOIN wallet_flow f ON w.user_id = f.user_id
    AND w.currency_code = f.currency_code
    AND w.wallet_type = f.wallet_type
  GROUP BY w.id
  HAVING w.balance != 流水计算余额;

优点：
  ✅ 每条流水自验证（before + amount = after）
  ✅ 流水链可完整重建任意时刻余额
  ✅ 只追加不修改，防篡改

规则：
  wallet_flow 表 —— 只 INSERT，永不 UPDATE，永不 DELETE
  这是资金审计的基石原则
```

### 14.3 可选方案

#### 方案B：每日余额快照表

```
适用场景：数据量大时加速历史余额查询
原理：每日零点记录所有用户的余额快照

实现：
  CREATE TABLE wallet_daily_snapshot (
    user_id BIGINT, currency_code VARCHAR(10), wallet_type TINYINT,
    snapshot_date DATE, balance BIGINT, frozen_amount BIGINT
  );
  每日零点定时任务：INSERT INTO wallet_daily_snapshot SELECT ...

优点：
  ✅ 查历史余额无需扫描全部流水
  ✅ 对账基准点

分类：在表结构设计中归为"可选"表
```

### 14.4 本项目最佳实践

```
✅ 方案A：Append-Only 流水 + before/after 快照

必须保证：
  1. wallet_flow 只追加，永不修改删除
  2. 每条流水都有 before_balance 和 after_balance
  3. 事务内校验 before + amount = after
  4. 所有写操作通过 ser-blog 上报审计日志

可选增强（二期）：
  □ 每日对账定时任务
  □ wallet_daily_snapshot 快照表
```

## 15. 挑战14：流水表的持续增长与查询性能

> **严重程度：P2 重要**
> 问题本质：wallet_flow 只追加不删除，数据量会持续增长，如何保证查询性能

### 15.1 挑战描述

```
增长预估：
  假设 10万活跃用户，每人每天 5 笔操作
  每日新增：500,000 条流水
  每月新增：15,000,000 条
  一年后：180,000,000 条（1.8 亿）

查询场景：
  C端：用户查自己的充值/提现/兑换记录（分页）
  B端：运营查询某用户的完整流水（分页）
  对账：按日期范围汇总（全表扫描风险）
```

### 15.2 主流方案

#### 方案A：合理索引 + 分页优化（当前适用）

```
使用者：绝大多数初期到中期的互联网系统
原理：通过索引覆盖常见查询，避免全表扫描

索引设计：
  KEY idx_user_currency_time (user_id, currency_code, create_at)  -- C端查记录
  KEY idx_ref_order (ref_order_no)                                 -- 按订单号查流水
  UNIQUE KEY uk_idempotent (idempotent_key, delete_at)             -- 幂等

C端分页优化（游标分页替代 OFFSET）：
  -- 传统 OFFSET 分页（慢，OFFSET 越大越慢）：
  SELECT * FROM wallet_flow WHERE user_id=? ORDER BY id DESC LIMIT 20 OFFSET 100;

  -- 游标分页（恒定性能）：
  SELECT * FROM wallet_flow
  WHERE user_id=? AND currency_code=? AND id < ?  -- 上一页最后一条的 id
  ORDER BY id DESC LIMIT 20;

优点：
  ✅ 简单，无额外基础设施
  ✅ 游标分页性能恒定，不随数据量增长
  ✅ 当前数据量完全可以承受
```

#### 方案B：TiDB 分区表

```
使用者：中大规模系统
原理：按时间（月/季）分区，查询自动路由到对应分区

实现：
  CREATE TABLE wallet_flow (...)
  PARTITION BY RANGE (create_at) (
    PARTITION p202602 VALUES LESS THAN (1740787200000),  -- 2026-03
    PARTITION p202603 VALUES LESS THAN (1743465600000),  -- 2026-04
    ...
  );

优点：
  ✅ 按时间范围查询只扫描对应分区
  ✅ 老分区可归档/压缩
  ✅ TiDB 原生支持

缺点：
  ⚠️ 需要定时创建新分区
  ⚠️ 跨分区查询性能可能下降
  ⚠️ 当前数据量不需要
```

### 15.3 可选方案

#### 方案C：冷热分离（归档策略）

```
使用者：大型系统（数据量超亿级）
原理：超过一定期限的流水移到归档表/冷存储

实现：
  wallet_flow        -- 近3个月数据（热数据）
  wallet_flow_archive -- 3个月以上数据（冷数据）

  定时任务：每月初将3个月前的数据迁移到归档表

优点：
  ✅ 热表数据量可控
  ✅ 查询性能稳定

缺点：
  ⚠️ 迁移逻辑复杂
  ⚠️ 跨表查询需要特殊处理
  ⚠️ 当前阶段不需要
```

#### 方案D：分库分表

```
使用者：超大规模系统（十亿+）
原理：按 user_id 分片

优点：
  ✅ 理论无上限
缺点：
  ⚠️ TiDB 本身是分布式数据库，内部已做了数据分片
  ⚠️ 应用层分库分表在 TiDB 上完全不需要
```

### 15.4 本项目最佳实践

```
✅ 当前阶段：方案A（合理索引 + 游标分页）

理由：
1. TiDB 天然支持水平扩展，单表亿级数据不是瓶颈
2. 合理索引覆盖所有常见查询
3. 游标分页替代 OFFSET，性能恒定
4. 不做提前优化，等数据量真正到瓶颈时再考虑分区

预留路径：
  数据量 > 5000万 → 考虑方案B（分区表，按月分区）
  数据量 > 5亿    → 考虑方案C（冷热分离）
  方案D 在 TiDB 架构下不需要
```

## 16. 挑战15：服务降级与容错

> **严重程度：P3 一般**
> 问题本质：依赖的外部服务（财务、KYC、三方API）不可用时，钱包如何保证核心功能正常

### 16.1 挑战描述

```
依赖故障影响分析：

| 依赖 | 故障影响 | 频率 |
|------|---------|------|
| 财务服务 | 充值方式/提现方式获取失败，充值提现不可用 | 低 |
| ser-kyc | 提现前 KYC 校验失败 | 低 |
| ser-blog | 审计日志丢失 | 低 |
| 三方汇率API | 汇率不更新 | 中 |
| Redis | 缓存不可用，所有请求打 DB | 低 |
| TiDB | 全部服务不可用 | 极低 |
```

### 16.2 主流方案

#### 方案A：分级降级策略（推荐）

```
使用者：所有高可用系统

核心原则：核心功能不因非核心依赖故障而停止

降级策略表：

| 故障场景 | 降级方案 | 用户影响 |
|----------|---------|---------|
| 财务不可用 | 充值/提现返回"服务暂忙，请稍后重试" | 充值提现暂停，余额查询/兑换/记录正常 |
| ser-kyc 不可用 | 提现降级为仅允许 USDT 提现（无需KYC） | 部分提现方式不可用 |
| ser-blog 不可用 | 审计日志记录到本地文件，不影响主流程 | 无感知 |
| 三方汇率全不可用 | 保持最后一次成功的汇率，记录告警 | 汇率可能不是最新，但业务可用 |
| Redis 不可用 | 跳过缓存直接查 DB | 性能下降，功能正常 |

实现要点：
  // 非核心依赖的调用模式
  kycStatus, err := kycClient.GetStatus(ctx, userID)
  if err != nil {
      lg.Log().CtxWarnf(ctx, "KYC服务不可用，降级处理: %v", err)
      kycStatus = KycUnverified  // 降级为未认证，只能 USDT 提现
  }

  // 审计日志的 fire-and-forget 模式
  go func() {
      defer func() { recover() }()  // 绝不影响主流程
      blogClient.AddActLog(ctx, logReq)
  }()
```

#### 方案B：熔断器模式（Circuit Breaker）

```
使用者：Netflix Hystrix、阿里 Sentinel
原理：连续失败达到阈值后自动"断开"，不再发起调用，定时探测恢复

状态机：
  Closed（正常）→ 连续失败5次 → Open（熔断）→ 等30秒 → Half-Open（探测）
  Half-Open：放一个请求试探 → 成功则 Close，失败则继续 Open

优点：
  ✅ 防止级联故障
  ✅ 快速失败，不浪费资源等超时
缺点：
  ⚠️ 增加实现复杂度
  ⚠️ 当前服务调用链不深，简单超时+降级已足够
  ⚠️ 架构设计文档明确说"当前不做熔断器"
```

### 16.3 本项目最佳实践

```
✅ 方案A：分级降级策略

核心原则：
  - TiDB 挂了 → 全停（无法降级，这是底层存储）
  - Redis 挂了 → 降级（跳过缓存走 DB）
  - 外部服务挂了 → 根据业务重要性分级降级
  - 审计日志挂了 → 完全不影响（fire-and-forget）

不做熔断器的理由：
  - 当前调用链简单（钱包→财务/KYC/Blog）
  - 3秒超时 + 降级逻辑已足够
  - 架构设计文档已明确"不过度设计"
  - 未来如果调用链变深再引入（common/pkg 中可以随时加）
```

## 17. 挑战16：多币种金额展示与换算

> **严重程度：P3 一般**
> 问题本质：不同币种的展示格式完全不同，整数存储的金额如何正确格式化展示

### 17.1 挑战描述

```
展示格式差异：

| 币种 | 千分位符 | 小数点符 | 小数位 | 示例 |
|------|---------|---------|--------|------|
| VND  | .       | —       | 0      | 5.000.000 |
| IDR  | .       | —       | 0      | 500.000 |
| THB  | ,       | .       | 2      | 1,500.50 |
| BSB  | ,       | .       | 2      | 200.00 |
| USDT | ,       | .       | 8      | 1.50000000 |

存储值到展示值的转换：
  150050 → THB → 1,500.50
  5000000 → VND → 5.000.000
  150000000 → USDT → 1.50000000

复杂度来源：
  1. 每个币种的精度系数不同（1/100/10^8）
  2. 千分位符和小数点符可能互为对方（VND 用.做千分位，THB 用.做小数点）
  3. 金额比较和排序必须在整数域进行，展示在 API 出口统一格式化
```

### 17.2 主流方案

#### 方案A：服务端格式化 + 币种配置驱动（推荐）

```
使用者：Stripe、支付宝国际版
原理：所有金额在 API 响应中由服务端统一格式化

实现思路：
  1. 币种配置表存储展示规则（thousands_sep, decimal_sep, decimal_digits, precision_factor）
  2. 封装统一格式化函数
  3. API 响应中包含：原始值（int64）+ 格式化值（string）

  func FormatAmount(amount int64, cfg *CurrencyConfig) string {
      // 1. 整数→小数
      value := float64(amount) / float64(cfg.PrecisionFactor)
      // 2. 按小数位格式化
      formatted := strconv.FormatFloat(value, 'f', int(cfg.DecimalDigits), 64)
      // 3. 添加千分位符
      return addThousandsSeparator(formatted, cfg.ThousandsSep, cfg.DecimalSep)
  }

API 响应格式：
  {
    "balance": 150050,           // 原始值（int64，用于计算）
    "balance_display": "1,500.50" // 格式化值（string，用于展示）
  }

优点：
  ✅ 前端无需关心格式化逻辑
  ✅ 所有端（App/H5/Web）展示一致
  ✅ 格式规则可通过币种配置动态调整
```

#### 方案B：前端格式化

```
使用者：部分轻量系统
原理：API 只返回原始整数值和币种代码，前端根据币种自行格式化

优点：
  ✅ 服务端简单
缺点：
  ⚠️ 多端（App/H5/Web）需要各自实现格式化逻辑
  ⚠️ 容易出现端间不一致
  ⚠️ 格式规则变更需要所有端更新
```

### 17.3 本项目最佳实践

```
✅ 方案A：服务端格式化 + 币种配置驱动

API 响应同时返回：
  - 原始整数值：前端用于计算、比较、传参
  - 格式化字符串：前端直接展示

格式化函数封装在 domain 层：
  domain/amount.go → FormatAmount()、ParseAmount()
  不在 Service 层做格式化，保持 Service 层的数据是原始值
  在 Handler 层（API 出口）统一调用 FormatAmount
```

## 18. 挑战17：并发防重提交

> **严重程度：P3 一般**
> 问题本质：用户快速双击"提交"按钮，或网络延迟导致重复发起请求

### 18.1 挑战描述

```
典型场景：
  - 用户点击"充值"按钮，网络慢没有反馈，再点一次 → 两笔充值订单
  - 用户点击"提现"按钮，App 卡顿，点了三次 → 三笔提现请求

与幂等性的区别：
  幂等性（挑战3）：同一笔业务操作被重复执行，保证只生效一次
  防重提交（本挑战）：阻止用户在短时间内发起多笔不同的操作
```

### 18.2 主流方案

#### 方案A：Redis 分布式锁（推荐）

```
使用者：电商下单、支付系统
原理：同一用户同一操作类型，短时间内只允许一个请求通过

实现：
  key = "wallet:lock:{operation}:{user_id}"
  // 如 "wallet:lock:withdraw:123"

  if !rds.SetNX(key, "1", 30*time.Second) {
      return BizErr(ctx, errs.OperationTooFrequent)  // "操作太频繁，请稍后重试"
  }
  defer rds.Del(key)

  // ... 执行业务逻辑

优点：
  ✅ 实现简单
  ✅ 已有工程实现（common/pkg/rds/redis_util.go）
  ✅ 有效防止用户快速重复操作

注意：
  - TTL 设为 30s（覆盖业务处理时间）
  - 业务执行完毕后 DEL key（不等 TTL 过期）
  - 这是"优化层"，不是"安全层"（安全层是数据库幂等）
```

#### 方案B：前端节流/防抖

```
使用者：所有前端
原理：按钮点击后禁用，等响应返回后恢复

实现：前端按钮点击后立即 disable，loading 状态

优点：
  ✅ 用户体验好（即时反馈）
缺点：
  ⚠️ 只是前端措施，不防绕过
  ⚠️ 后端仍需有防护
```

### 18.3 本项目最佳实践

```
✅ Redis 分布式锁（后端）+ 前端防抖（前端）双保险

后端锁粒度：
  充值创建：wallet:lock:recharge:{user_id}    TTL=30s
  提现创建：wallet:lock:withdraw:{user_id}    TTL=30s
  兑换提交：wallet:lock:exchange:{user_id}    TTL=30s

防御层次：
  第1层：前端按钮防抖（前端负责）
  第2层：Redis 分布式锁（拦截短时间重复请求）
  第3层：数据库幂等约束（终极保障）
```

## 19. 挑战18：充值回调的可靠性

> **严重程度：P1 严重**
> 问题本质：三方支付回调可能延迟/丢失/重复/伪造，如何保证上账可靠且安全

### 19.1 挑战描述

```
回调异常场景：

1. 回调延迟：用户已支付，但回调 10 分钟后才到 → 用户投诉没到账
2. 回调丢失：三方没发回调或被网络吞了 → 钱已扣但没上账
3. 回调重复：三方重发 → 重复上账（已由幂等性解决）
4. 回调伪造：攻击者伪造回调请求 → 凭空上账
5. 金额篡改：回调中的金额被篡改 → 充 100 上账 10000
```

### 19.2 主流方案

#### 方案A：验签 + 金额校验 + 幂等 + 主动轮询（推荐）

```
使用者：支付宝/微信/Stripe Webhooks
原理：多层防御确保回调可靠

防御层次：
  ┌──────────────────────────────────────────────────┐
  │ 第1层：签名验证                                    │
  │  三方回调带签名参数，用密钥验签                       │
  │  验签失败 → 拒绝，记录告警                           │
  ├──────────────────────────────────────────────────┤
  │ 第2层：金额校验                                    │
  │  回调金额 vs 订单金额，必须一致                       │
  │  不一致 → 拒绝，记录告警                             │
  ├──────────────────────────────────────────────────┤
  │ 第3层：幂等保证                                    │
  │  wallet_flow UNIQUE(idempotent_key) 防重复上账       │
  │  重复回调 → 返回成功但不重复执行                      │
  ├──────────────────────────────────────────────────┤
  │ 第4层：主动轮询（兜底）                              │
  │  定时任务扫描"待支付"超时订单 → 主动查三方支付结果      │
  │  已支付 → 主动上账；未支付 → 标记超时关闭              │
  └──────────────────────────────────────────────────┘

注意：回调处理和主动轮询主要是**财务模块**的职责
钱包模块的职责：
  ✅ CreditWallet 幂等 → 无论被调多少次都安全
  ✅ 金额参数校验 → amount > 0
  ✅ 正确返回结果 → 让财务知道上账成功/失败
```

### 19.3 本项目最佳实践

```
钱包侧职责明确：
  ✅ CreditWallet RPC 接口幂等
  ✅ 参数校验（user_id > 0, amount > 0, currency 合法）
  ✅ 正确的成功/失败状态码

财务侧职责（对方负责，但需要了解以便联调）：
  ✅ 回调验签
  ✅ 金额校验
  ✅ 主动轮询兜底
  ✅ 重试逻辑
```

```
│                                                               │

│     └── 保证重复操作只执行一次                                   │
```

### 21.4 不做/暂不做的事（同样重要）

```
┌───────────────────────────────────────────────────────────────┐
│               当前阶段明确不做的事（避免过度设计）                  │
│                                                               │
│  ✗ 不做分库分表 → TiDB 天然分布式，无需应用层分片                 │
│  ✗ 不做 CQRS → 单服务读写量不大                                 │
│  ✗ 不做 Event Sourcing → 余额+流水模型已满足审计                 │
│  ✗ 不做 Saga/TCC → 同步RPC+幂等已满足跨服务一致性                │
│  ✗ 不做熔断器 → 调用链不深，超时+降级已足够                       │
│  ✗ 不做异步消息队列 → 当前无高吞吐异步场景                        │
│  ✗ 不做拆分账户 → 无热点账户问题                                 │
│  ✗ 不做多级缓存 → Cache-Aside + 单级 Redis 已满足               │
│  ✗ 不做复式记账 → 用户钱包不是银行账务                            │
│  ✗ 不做流水表分区 → 当前数据量不需要                              │
│                                                               │
│  原则：不做不代表不能做，而是当前阶段不需要                        │
│  每一项都有明确的"何时需要做"的触发条件                            │
│  达到触发条件时可以低成本引入（架构已预留扩展点）                   │
└───────────────────────────────────────────────────────────────┘
```
