## 二、币种配置域：接口内部调用链路

### 2.1 基准币种设置链路

```
运营人员(B端) → gate-back → CC-1 SetBaseCurrency
```

**内部调用**：
- CC-1 不调用任何其他接口
- CC-1 → EXT-4 ser-blog.AddActLog（记录操作日志，oneway）

**被谁依赖**：
- CC-R3 GetBaseCurrency 返回的数据来自 CC-1 写入的基准币种记录
- CC-2 CreateCurrency 需要基准币种已设置（汇率换算的基准）
- CC-T1 SyncRealtimeRate 需要基准币种已设置（换算基准）

**调用链路说明**：SetBaseCurrency 是整个系统的**第一个操作**。在基准币种设置之前，币种创建无法完成汇率换算关系的建立。这是一个一次性操作，执行后记录不可更改。

> 依据：币种配置需求文档 B3——基准币种推荐USDT，配置后不可更改

### 2.2 币种CRUD链路

```
运营人员(B端) → gate-back → CC-2 CreateCurrency
                            CC-3 UpdateCurrency
                            CC-4 ToggleCurrencyStatus
                            CC-5 QueryCurrencyPage
```

**CC-2 CreateCurrency 内部调用**：
- 读取基准币种信息（内部函数，判断基准是否已设置）
- CC-2 → EXT-4 ser-blog.AddActLog（oneway）

**CC-3 UpdateCurrency 内部调用**：
- 读取当前币种记录（内部数据库查询）
- 更新汇率浮动%、阈值%等配置字段
- CC-3 → EXT-4 ser-blog.AddActLog（oneway）

**CC-4 ToggleCurrencyStatus 内部调用**：
- 读取当前状态，翻转为启用/禁用
- CC-4 → EXT-4 ser-blog.AddActLog（oneway）

**CC-5 QueryCurrencyPage 内部调用**：
- 数据库分页查询（内部）
- CC-5 → EXT-5 ser-buser.QueryUserByIds（解析操作人姓名）

**CC-2/3/4 之间的关系**：三者操作同一张币种表的不同字段，互不调用，但有数据依赖——CC-3 和 CC-4 操作的记录必须先由 CC-2 创建。

**被谁依赖**：
- CC-2 创建的币种数据 → CC-R1 GetCurrencyList 返回 → 钱包 W-2 展示给用户 → 财务所有页面币种筛选
- CC-4 禁用某币种 → 影响 CC-R1 返回的可用币种列表 → 影响C端用户能选择的币种

> 依据：币种配置需求文档 B3（18个配置字段）+ 产品原型 C1/C2（列表页+编辑弹窗）

### 2.3 汇率同步链路

```
定时触发 → CC-T1 SyncRealtimeRate
```

**CC-T1 内部调用**：
1. 调用三方汇率API-A，获取汇率
2. 调用三方汇率API-B，获取汇率
3. 调用三方汇率API-C，获取汇率
4. 计算三家平均值 = 实时汇率
5. 读取每个币种的当前平台汇率和阈值配置（内部数据库）
6. 计算偏差值 = |实时 - 平台| / 实时 × 100%
7. 如果偏差 ≥ 阈值：
   - 更新平台汇率 = 实时汇率
   - 更新入款汇率 = 实时汇率 × (1 + 汇率浮动%)
   - 更新出款汇率 = 实时汇率 × (1 - 汇率浮动%)
   - 写入汇率变更日志（CC-6 QueryRateLogPage 查的就是这条记录）

**不调用任何RPC**：三方汇率API是HTTP外部调用，不是项目内的RPC服务。

**被谁依赖**：
- CC-T1 更新的汇率数据 → CC-R2 GetExchangeRate 返回 → 钱包兑换/充值/提现时读取
- CC-T1 产生的日志记录 → CC-6 QueryRateLogPage 展示

> 依据：币种配置需求文档 B4——实时汇率3家取均值，偏差≥阈值触发更新

### 2.4 币种配置RPC供他链路

```
财务管理/其他服务 → RPC → CC-R1 GetCurrencyList
                         CC-R2 GetExchangeRate
                         CC-R3 GetBaseCurrency
                         CC-R4 GetCurrencyByCode
```

这4个接口是**纯读操作**，内部只做数据库查询或缓存读取，不调用任何其他接口。

**CC-R1 GetCurrencyList 被调用场景**：
- 财务管理所有页面的币种下拉筛选（通道配置/支付配置/充值记录/提现记录/统计）
- 钱包内部通过函数调用获取（不走RPC）

**CC-R2 GetExchangeRate 被调用场景**：
- 财务管理统计报表的基准币种换算
- 财务管理充值/提现记录表格的基准换算列
- 钱包内部兑换/充值/提现时获取汇率（不走RPC）

**CC-R3 GetBaseCurrency 被调用场景**：
- 财务管理统计报表确定换算基准

> 依据：财务管理产品原型 F 系列——所有页面均含币种筛选和基准换算列

## 三、钱包域：C端接口内部及跨模块调用链路

### 3.1 钱包基础接口

#### W-1 GetWalletBalance（查余额）

```
用户(C端) → gate-font → W-1 GetWalletBalance
                              │
                              ├→ [内部] 读取币种展示规则（千分位/小数位）← 币种配置数据
                              ├→ [内部] 读取用户当前币种下各子钱包余额（中心/奖励/主播）
                              └→ 返回：资金钱包余额(=中心+奖励) + 各子钱包明细
```

**不调用任何外部RPC**。纯粹读取钱包自己的数据库 + 币种配置的展示规则（同服务内部函数调用）。

**被谁调用/关联**：
- C端几乎每个页面都会调 W-1（个人中心、充值页、提现页、兑换页顶部余额展示）
- W-4 CreateDepositOrder 内部调 W-1 做重复转账检查（查当前余额和待处理订单）
- W-9 ExecuteExchange 内部调 W-1 做余额充足性校验
- W-11 CreateWithdrawalOrder 内部调 W-1 做余额充足性校验

> 依据：产品原型 D2 个人中心——钱包余额展示；需求文档 B2——资金钱包余额=中心+奖励

#### W-2 GetAvailableCurrencies（可用币种列表）

```
用户(C端) → gate-font → W-2 GetAvailableCurrencies
                              │
                              ├→ [内部] 调币种配置获取已启用币种列表（函数调用，非RPC）
                              └→ 返回：可用币种列表 + 每个币种的图标/符号/展示规则 + 用户当前选中币种
```

**不调用外部RPC**。币种配置在同服务内，直接函数调用。

**被谁调用/关联**：
- C端币种切换组件调 W-2
- W-3 GetDepositMethods、W-10 GetWithdrawalInfo 隐含依赖 W-2（需要知道当前币种是什么）

> 依据：产品原型 D2——币种切换组件(BSB/VND/IDR/THB)

### 3.2 充值链路接口

#### W-3 GetDepositMethods（充值方式查询）

```
用户(C端) → gate-font → W-3 GetDepositMethods
                              │
                              ├→ [RPC] 调财务 EXT-1 GetAvailableDepositMethods
                              │         传入：当前币种代码
                              │         返回：该币种下可用充值方式列表
                              │               每种方式的金额档位、费率、关联通道信息
                              │
                              ├→ [内部] 调币种配置获取入款汇率（函数调用）
                              │         用于BSB充值时展示"≈ X USDT"换算
                              │
                              └→ 返回：充值方式列表(USDT/银行/电子钱包) + 金额档位 + 汇率换算
```

**关键跨模块调用**：W-3 → EXT-1（财务管理RPC）。如果财务的支付配置接口不可用，充值页面无法渲染。

> 依据：产品原型 D3——充值中心按币种展示不同充值方式和金额档位；需求文档 E2——充值方式配置在财务侧

#### W-6 GetBonusActivities（奖金活动列表）

```
用户(C端) → gate-font → W-6 GetBonusActivities
                              │
                              ├→ [RPC?] 调外部活动模块（数据源待确认）
                              │          传入：当前币种、充值金额
                              │          返回：可用活动列表 + 奖金比例 + 最低门槛
                              │
                              └→ 返回：活动列表，每项含名称/比例/门槛/描述
```

**特殊说明**：这个接口的数据源当前不明确，项目21个服务中没有活动管理服务。可能需要后续确认。

**与 W-4 的关系**：W-6 是 W-4 CreateDepositOrder 的前置查询——用户在奖金浮窗中选了某个活动后，将活动ID传给 W-4。但 W-6 不是必须的，用户可以不选奖金直接充值。

> 依据：产品原型 D5——奖金选择半屏浮窗；需求文档 A3——额外奖金选择规则

#### W-4 CreateDepositOrder（创建充值订单）

```
用户(C端) → gate-font → W-4 CreateDepositOrder
                              │
                              │  传入：币种 + 充值方式 + 金额 + 奖金活动ID(可选)
                              │
                              ├→ [内部] 重复转账拦截检查
                              │         查询条件：同UserID + 同币种 + 同金额 + 状态=待支付/进行中
                              │         1-2笔 → 返回警告标记（客户端弹窗，可继续）
                              │         ≥3笔 → 直接拒绝，返回拦截错误
                              │
                              ├→ [内部] 奖金门槛校验（如果选了奖金活动）
                              │         充值金额 < 活动最低门槛 → 拒绝
                              │
                              ├→ [内部] 创建钱包侧充值记录（状态=待支付）
                              │         生成订单号 C+16位
                              │
                              ├→ [RPC?] 通知财务分配通道（具体交互方式待确认）
                              │         财务侧：轮询匹配代收通道 → 创建财务侧订单
                              │
                              ├→ EXT-4 ser-blog.AddActLog（oneway）
                              │
                              └→ 返回：订单信息 + 支付页面信息（三方跳转URL 或 USDT地址）
```

**关键说明**：
- 重复转账拦截是 W-4 内部逻辑，不是独立接口，不调用外部服务
- USDT充值走 W-5 获取地址，非USDT走三方承载页
- 奖金关联记录奖金活动ID和比例，实际奖金入账在 W-R2 CreditDeposit 时执行

> 依据：需求文档 A3——重复转账拦截(同UserID+同币种+同金额，3笔上限)；额外奖金规则(单笔一个活动，门槛校验)

#### W-5 GetUSDTDepositInfo（USDT收款信息）

```
用户(C端) → gate-font → W-5 GetUSDTDepositInfo
                              │
                              │  传入：网络类型(TRC20/ERC20/BEP20)
                              │
                              ├→ [内部/外部?] 获取收款地址（地址管理方式待确认）
                              │
                              └→ 返回：收款地址 + 二维码数据 + 网络类型 + 复制按钮文案
```

**与 W-4 的关系**：用户先通过 W-4 创建充值订单选择USDT方式，然后 W-5 返回该次充值的收款地址。或者 W-5 独立返回平台的固定收款地址（取决于地址是固定的还是每笔订单独立的，待确认）。

**不调用财务RPC**：USDT充值是平台自己承载的（需求文档 A3——"平台承载页=是"），不走三方支付通道。

> 依据：产品原型 D4——USDT支付页(网络选择+收款地址+QR码)

#### W-7 GetDepositOrderStatus（充值订单状态轮询）

```
用户(C端) → gate-font → W-7 GetDepositOrderStatus
                              │
                              │  传入：订单号(C+16位)
                              │
                              ├→ [内部] 查询订单当前状态
                              │
                              └→ 返回：订单状态(待支付/进行中/已完成/已失败/已关闭)
```

**纯内部查询**，不调用任何外部RPC。客户端在用户支付后定时轮询此接口等待支付结果。

**与 W-4 的关系**：W-7 查询的是 W-4 创建的订单。W-4 是前置。

**与 W-R2 的关系**：当财务回调触发 W-R2 CreditDeposit 后，订单状态变为"已完成"，W-7 下次轮询就能拿到最终结果。

> 依据：产品原型 D6——充值进行中弹窗(查看详情/继续转账)

#### W-R2 CreditDeposit（充值上账，RPC供财务调用）

```
财务管理 → RPC → W-R2 CreditDeposit
                      │
                      │  传入：订单号 + 充值金额 + 币种 + 用户ID + 奖金信息(活动ID/比例/金额)
                      │
                      ├→ [内部] 幂等校验：该订单是否已上账（防重复回调）
                      │
                      ├→ [内部] 充值金额 → 中心钱包余额增加
                      │
                      ├→ [内部] 如果有额外奖金：
                      │         奖金金额(=充值金额×比例) → 奖励钱包余额增加
                      │         记录稽核要求（X倍流水）
                      │
                      ├→ [内部] 写入交易流水（充值类型）
                      │         W-14 GetTransactionRecords 查的就是这条流水
                      │
                      ├→ [内部] 更新订单状态 → 已完成
                      │         W-7 GetDepositOrderStatus 下次轮询返回"已完成"
                      │
                      ├→ EXT-4 ser-blog.AddActLog（oneway）
                      │
                      └→ 返回：上账结果(成功/失败)
```

**关键说明**：
- **幂等性**是硬要求——三方回调可能推送多次，W-R2 必须保证同一订单只入账一次
- W-R2 同时影响 W-1 的返回值（余额变了）和 W-7 的返回值（状态变了）
- W-R2 是充值链路的**终点接口**

> 依据：需求文档 A7——幂等性(充值回调防重复入账)；需求文档 E3——充值流程最后一步"上分"

### 3.3 充值链路完整调用链

将上述接口串联成完整链路：

```
W-2 → W-3 → W-6(可选) → W-4 → W-5(USDT时) → [用户支付] → W-R2 → W-7(轮询) → W-14/W-15(查记录)

详细展开：

1. 用户打开钱包
   → W-2 GetAvailableCurrencies（获取可用币种）
   → W-1 GetWalletBalance（展示余额）

2. 用户进入充值页面
   → W-3 GetDepositMethods（获取充值方式+档位）
       内部调用：财务 EXT-1 + 币种配置汇率

3. 用户选择奖金活动（可选）
   → W-6 GetBonusActivities（获取活动列表）

4. 用户提交充值
   → W-4 CreateDepositOrder（创建订单+重复拦截+奖金关联）
       如果USDT → W-5 GetUSDTDepositInfo（获取收款地址）
       如果银行/电子钱包 → 返回三方跳转URL

5. 用户完成支付后，客户端轮询
   → W-7 GetDepositOrderStatus（等待状态变更）

6. 三方回调到达财务管理 → 财务验签 → 财务调用钱包
   → W-R2 CreditDeposit（上账：金额→中心钱包，奖金→奖励钱包）

7. 用户查看充值记录
   → W-14 GetTransactionRecords（充值Tab）
   → W-15 GetTransactionDetail（订单详情）
```

**模块间调用次数**：整条充值链路中，钱包调财务1次（W-3→EXT-1 查充值方式），财务调钱包1次（→W-R2 上账）。

### 3.4 兑换链路接口

#### W-8 GetExchangePreview（兑换预览）

```
用户(C端) → gate-font → W-8 GetExchangePreview
                              │
                              │  传入：源币种(法币) + 兑换金额
                              │
                              ├→ [内部] 调币种配置获取平台汇率（函数调用）
                              │         1 BSB = X 法币
                              │
                              ├→ [内部] 计算兑换获得 = 金额 / 汇率
                              │
                              ├→ [内部] 计算赠送额 = 兑换获得 × 赠送比例(如10%)
                              │
                              ├→ [内部] 计算实际到账 = 兑换获得 + 赠送额
                              │
                              └→ 返回：汇率 + 兑换获得 + 赠送额 + 实际到账 + 汇率快照ID
```

**不调用任何外部RPC**。全部是钱包内部计算 + 币种配置内部函数调用。

> 依据：产品原型 D7——兑换BSB页面(余额→输入金额→兑换获得+赠送10%→实际到账)

#### W-9 ExecuteExchange（执行兑换）

```
用户(C端) → gate-font → W-9 ExecuteExchange
                              │
                              │  传入：源币种 + 金额 + 汇率快照ID
                              │
                              ├→ [内部] 校验汇率快照是否过期（防止预览后汇率已变）
                              │
                              ├→ [内部] 校验源币种余额 ≥ 兑换金额
                              │         调 W-1 逻辑获取当前余额
                              │
                              ├→ [内部] 事务操作：
                              │         扣减 源币种中心钱包余额（减）
                              │         增加 BSB中心钱包余额（加，兑换获得部分）
                              │         增加 BSB奖励钱包余额（加，赠送部分）
                              │         记录赠送部分的稽核要求（X倍流水）
                              │
                              ├→ [内部] 写入交易流水（兑换类型）
                              │
                              ├→ EXT-4 ser-blog.AddActLog（oneway）
                              │
                              └→ 返回：兑换结果 + 新余额
```

**不调用任何外部RPC**。兑换是**钱包内部闭环操作**。

**W-8 与 W-9 的关系**：W-8 是预览（只读），W-9 是执行（写操作）。W-8 返回的汇率快照ID传给 W-9 用于校验。如果用户在预览后长时间不操作，汇率可能已变，W-9 应拒绝并要求重新预览。

> 依据：需求文档 A4——兑换模块(法币→BSB，赠送奖励需X倍流水)

### 3.5 兑换链路完整调用链

```
W-2 → W-1 → W-8 → W-9 → W-14/W-15

详细展开：

1. 用户在法币币种下进入兑换页面
   → W-1 GetWalletBalance（展示当前法币余额）

2. 用户输入兑换金额
   → W-8 GetExchangePreview（计算兑换结果预览）
       内部调用：币种配置平台汇率（函数调用）

3. 用户确认兑换
   → W-9 ExecuteExchange（执行兑换）
       内部：扣法币 + 加BSB（中心+奖励）

4. 用户查看兑换记录
   → W-14 GetTransactionRecords（兑换Tab）
```

**模块间调用次数**：整条兑换链路**0次跨模块RPC调用**。全部在钱包+币种配置内部完成。

### 3.6 提现链路接口

#### W-10 GetWithdrawalInfo（提现信息聚合）

```
用户(C端) → gate-font → W-10 GetWithdrawalInfo
                               │
                               │  传入：当前币种
                               │
                               ├→ [RPC] 调 EXT-3 ser-kyc.KycQuery
                               │         传入：用户ID
                               │         返回：KYC认证状态(已认证/未认证) + 认证姓名
                               │
                               ├→ [RPC] 调 EXT-2 财务.GetAvailableWithdrawalMethods
                               │         传入：当前币种代码
                               │         返回：该币种下可用提现方式列表 + 手续费 + 限额
                               │
                               ├→ [内部] 根据KYC状态过滤提现方式：
                               │         未认证 → 只保留USDT提现方式
                               │         已认证 → 保留全部方式
                               │
                               ├→ [内部] 调币种配置获取出款汇率（函数调用）
                               │         用于BSB提现时展示法币换算
                               │
                               └→ 返回：KYC状态 + 可用提现方式列表 + 限额规则 + 汇率
```

**跨模块调用最多的接口**：同时调用 ser-kyc（查认证状态）和 财务管理（查提现方式），是整个钱包中外部依赖最重的接口。

> 依据：需求文档 A5——KYC与提现关系(未认证仅USDT)；产品原型 D8——提现触发认证弹窗

#### W-12 SaveWithdrawalAccount（保存提现账户）

```
用户(C端) → gate-font → W-12 SaveWithdrawalAccount
                               │
                               │  传入：账户类型(银行卡/电子钱包) + 账户信息(卡号/钱包号/银行名)
                               │
                               ├→ [内部] 账户持有人 = KYC认证姓名（自动填充）
                               │         需要读取用户KYC信息（内部缓存 或 调 ser-kyc）
                               │
                               ├→ [内部] 保存到数据库
                               │
                               └→ 返回：保存成功 + 账户ID
```

**与 W-13 的关系**：W-12 保存，W-13 查询。首次提现时调 W-12 保存账户，后续提现调 W-13 获取已保存账户自动填充。

**与 W-11 的关系**：W-12 可以在提现前独立调用（提前保存账户），也可以在提现流程中调用。W-11 CreateWithdrawalOrder 需要指定提现账户ID，所以 W-12 或 W-13 是 W-11 的前置。

> 依据：需求文档 A5——首次填写银行信息，后续自动填充；账户持有人=KYC认证姓名

#### W-13 GetWithdrawalAccounts（查询已保存账户）

```
用户(C端) → gate-font → W-13 GetWithdrawalAccounts
                               │
                               │  传入：当前币种（不同币种可能有不同账户）
                               │
                               ├→ [内部] 查询该用户该币种下已保存的提现账户列表
                               │
                               └→ 返回：账户列表(含类型/账号/银行名/持有人姓名)
```

**纯内部查询**，不调用外部RPC。

#### W-11 CreateWithdrawalOrder（创建提现订单）

```
用户(C端) → gate-font → W-11 CreateWithdrawalOrder
                               │
                               │  传入：币种 + 金额 + 提现方式 + 账户ID
                               │
                               ├→ [内部] 提现规则校验：
                               │         法币充→法币提，USDT充→USDT提，混合→法币提
                               │         检查用户充值历史确定允许的提现方式
                               │
                               ├→ [内部] 限额校验：
                               │         金额 ≥ 单笔最低限额
                               │         金额 ≤ 单笔最高限额
                               │         今日已提现 + 本笔 ≤ 每日限额
                               │
                               ├→ [内部] 余额校验：
                               │         调 W-1 逻辑确认余额 ≥ 提现金额
                               │
                               ├→ [内部] 事务操作：
                               │         创建提现订单（T+16位）
                               │         调 W-R3 FreezeForWithdrawal 逻辑冻结对应金额
                               │         写入交易流水（提现类型，状态=处理中）
                               │
                               ├→ [通知] 将提现请求提交给财务管理进入审核流程
                               │
                               ├→ EXT-4 ser-blog.AddActLog（oneway）
                               │
                               └→ 返回：订单号 + 预计到账时间
```

**关键说明**：
- W-11 内部包含冻结逻辑（W-R3 FreezeForWithdrawal 的逻辑），不是W-11调W-R3的RPC——因为它们在同一个服务内
- 冻结和订单创建必须在**同一个数据库事务**内完成
- 提现规则校验（法币充→法币提）需要读取用户的充值历史来判断

> 依据：需求文档 A5——提现规则(充值方式决定提现方式) + 提现限制(单笔/单日限额)

#### W-R3 FreezeForWithdrawal（提现冻结，RPC供财务调用）

```
财务管理 → RPC → W-R3 FreezeForWithdrawal
                      │
                      │  传入：用户ID + 币种 + 金额 + 订单号
                      │
                      ├→ [内部] 幂等校验：该订单是否已冻结
                      │
                      ├→ [内部] 冻结操作：
                      │         可用余额减少，冻结余额增加
                      │         保证：可用余额 ≥ 0
                      │
                      └→ 返回：冻结结果(成功/余额不足)
```

**说明**：W-R3 既作为 W-11 内部创建提现时的冻结逻辑，也作为 RPC 接口暴露给财务管理直接调用。当从 W-11 内部调用时走函数调用；当财务需要独立冻结时走RPC。

#### W-R4 UnfreezeForWithdrawal（提现解冻，RPC供财务调用）

```
财务管理 → RPC → W-R4 UnfreezeForWithdrawal
                      │
                      │  传入：订单号 + 解冻原因(审核拒绝/出款失败)
                      │
                      ├→ [内部] 幂等校验：该订单是否已处理过终态
                      │
                      ├→ [内部] 解冻操作：
                      │         冻结余额减少，可用余额恢复
                      │
                      ├→ [内部] 更新交易流水状态 → 已拒绝/已失败
                      │
                      ├→ EXT-4 ser-blog.AddActLog（oneway）
                      │
                      └→ 返回：解冻结果
```

**触发条件**：财务管理的风控审核拒绝、财务审核拒绝、或通道出款失败后调用。

> 依据：需求文档 E4——提现流程(审核拒绝→解冻返还)

#### W-R5 ConfirmWithdrawalDeduct（提现确认扣款，RPC供财务调用）

```
财务管理 → RPC → W-R5 ConfirmWithdrawalDeduct
                      │
                      │  传入：订单号
                      │
                      ├→ [内部] 幂等校验：该订单是否已处理过终态
                      │
                      ├→ [内部] 状态校验：该订单必须处于"已冻结"状态
                      │
                      ├→ [内部] 确认扣款：
                      │         冻结余额减少（最终扣除）
                      │         该笔资金从用户账户永久移出
                      │
                      ├→ [内部] 更新交易流水状态 → 已完成
                      │
                      ├→ EXT-4 ser-blog.AddActLog（oneway）
                      │
                      └→ 返回：扣款结果
```

**触发条件**：财务管理收到通道出款成功回调后调用。这是提现链路的**终点接口**。

**状态约束**：W-R5 只能操作"已冻结"状态的订单。如果订单已被 W-R4 解冻，W-R5 必须拒绝。反之亦然。**W-R4 和 W-R5 互斥**——同一笔订单只能走其中一条路径。

> 依据：需求文档 E4——提现流程(出款成功→确认扣款)

### 3.7 提现链路完整调用链

```
W-2 → W-1 → W-10 → W-13 → W-12(首次) → W-11 → [财务三级审核] → W-R4 或 W-R5 → W-14/W-15

详细展开：

1. 用户进入提现页面
   → W-1 GetWalletBalance（展示余额）
   → W-10 GetWithdrawalInfo（KYC状态+方式+限额）
       RPC调用：ser-kyc.KycQuery + 财务.GetAvailableWithdrawalMethods
       内部调用：币种配置出款汇率

2. 如果未认证KYC
   → 客户端弹窗："去认证"或"仅USDT提现"
   → 如果去认证：跳转KYC流程（ser-kyc负责，不在我模块内）

3. 用户选择提现方式
   → W-13 GetWithdrawalAccounts（获取已保存账户）
   → 如果无账户 → W-12 SaveWithdrawalAccount（首次保存）

4. 用户输入金额提交
   → W-11 CreateWithdrawalOrder（规则校验+限额校验+冻结+创建订单）
       内部：W-R3 冻结逻辑 + 通知财务

5. 财务管理接管
   → 风控审核 → 财务审核 → 出款审核 → 通道出款
   （全部在财务模块内，不调用钱包接口）

6. 审核/出款结果
   → 拒绝/失败：财务调用 W-R4 UnfreezeForWithdrawal（解冻返还）
   → 成功：财务调用 W-R5 ConfirmWithdrawalDeduct（确认扣款）

7. 用户查看提现记录
   → W-14 GetTransactionRecords（提现Tab）
   → W-15 GetTransactionDetail（详情含状态/失败原因）
```

**模块间调用次数**：钱包调外部2次（W-10→EXT-2 查提现方式 + W-10→EXT-3 查KYC），财务调钱包1次（W-R4 或 W-R5 终态处理）。

### 3.8 人工修正链路接口

#### W-R1 GetUserWalletBalance（查用户余额，RPC供财务调用）

```
财务管理 → RPC → W-R1 GetUserWalletBalance
                      │
                      │  传入：用户ID + 币种
                      │
                      ├→ [内部] 查询该用户该币种下各子钱包余额
                      │         中心钱包 / 奖励钱包 / 主播钱包 / (代理钱包) / (场馆钱包)
                      │
                      └→ 返回：各子钱包当前余额
```

**与 W-1 的区别**：W-1 是C端用户查自己的余额（从token取用户ID），W-R1 是B端财务人员查指定用户的余额（传入用户ID参数）。底层查询逻辑一样，但权限校验和入口不同。

**触发场景**：财务人员打开人工加款/减款创建弹窗时调用，展示目标用户各钱包当前余额作为操作参考。

> 依据：财务产品原型 F——人工加款/减款弹窗中展示各钱包余额

#### W-R6 ManualCredit（人工加款，RPC供财务调用）

```
财务管理 → RPC → W-R6 ManualCredit
                      │
                      │  传入：用户ID + 币种 + 各钱包加款金额
                      │         {中心钱包: X, 奖励钱包: Y, 主播钱包: Z, 代理钱包: W}
                      │         + 加款类型(运营补偿/运营奖励/财务修正/测试加款)
                      │         + 订单号(A+16位，财务侧生成)
                      │         + 操作备注
                      │
                      ├→ [内部] 幂等校验：该订单号是否已处理
                      │
                      ├→ [内部] 按钱包类型分别增加余额
                      │
                      ├→ [内部] 为每个有变动的子钱包写入交易流水
                      │
                      ├→ EXT-4 ser-blog.AddActLog（oneway）
                      │
                      └→ 返回：加款结果
```

> 依据：需求文档 E5——人工加款(按钱包类型分别输入金额，加款类型枚举)

#### W-R7 ManualDebit（人工减款，RPC供财务调用）

```
财务管理 → RPC → W-R7 ManualDebit
                      │
                      │  传入：用户ID + 币种 + 各钱包减款金额
                      │         {中心钱包: X, 奖励钱包: Y, 主播钱包: Z, 代理钱包: W}
                      │         + 减款类型(异常追回/风控处罚/财务冲正/费用调整)
                      │         + 订单号(M+16位)
                      │         + 操作备注
                      │
                      ├→ [内部] 幂等校验
                      │
                      ├→ [内部] 余额校验：每个子钱包减款金额 ≤ 当前余额
                      │         余额不足的 → 实际减款金额 = 当前余额（不报错，扣能扣的）
                      │
                      ├→ [内部] 按钱包类型分别扣减余额
                      │
                      ├→ [内部] 写入交易流水
                      │
                      ├→ EXT-4 ser-blog.AddActLog（oneway）
                      │
                      └→ 返回：减款结果(含每个钱包实际扣减金额)
```

**与 W-R6 的区别**：方向相反（加→减），且有余额兜底逻辑（余额不足扣实际余额而非报错）。

> 依据：需求文档 E5——人工减款(校验余额≥0)

#### W-R8 CreditCorrection（补单上账，RPC供财务调用）

```
财务管理 → RPC → W-R8 CreditCorrection
                      │
                      │  传入：原充值订单号 + 补单订单号(B+16位)
                      │         + 补单金额 + 补单赠送金额
                      │         + 稽核倍数 + 稽核流水要求
                      │
                      ├→ [内部] 幂等校验
                      │
                      ├→ [内部] 补单金额 → 中心钱包
                      │
                      ├→ [内部] 补单赠送 → 奖励钱包（含稽核要求记录）
                      │
                      ├→ [内部] 写入交易流水
                      │
                      ├→ EXT-4 ser-blog.AddActLog（oneway）
                      │
                      └→ 返回：上账结果
```

**与 W-R2 CreditDeposit 的相似性**：都是"金额进中心钱包，赠送/奖金进奖励钱包"的模式。区别在于 W-R2 是正常充值触发，W-R8 是人工补单触发。

> 依据：需求文档 E5——充值补单(关联原充值订单，含补单金额/赠送/稽核)

## 四、跨模块调用汇总

### 4.1 钱包调用外部（我主动调别人）

| 钱包接口 | 调用的外部接口 | 所属服务 | 调用时机 | 调用方式 |
|---------|-------------|---------|---------|---------|
| W-3 GetDepositMethods | EXT-1 GetAvailableDepositMethods | 财务管理 | 用户打开充值页 | 同步RPC |
| W-10 GetWithdrawalInfo | EXT-2 GetAvailableWithdrawalMethods | 财务管理 | 用户打开提现页 | 同步RPC |
| W-10 GetWithdrawalInfo | EXT-3 KycQuery | ser-kyc | 用户打开提现页 | 同步RPC |
| 所有写操作 | EXT-4 AddActLog | ser-blog | 写操作完成后 | 异步oneway |

**总计**：2个财务RPC + 1个KYC RPC + 1个审计日志。钱包对外调用少，且全是**查询类**，不修改外部数据。

### 4.2 外部调用钱包（别人主动调我）

| 外部调用方 | 钱包接口 | 调用时机 | 调用方式 |
|-----------|---------|---------|---------|
| 财务(充值回调) | W-R2 CreditDeposit | 三方支付成功回调验签后 | 异步RPC |
| 财务(创建提现) | W-R3 FreezeForWithdrawal | 提现订单进入审核前 | 同步RPC |
| 财务(审核拒绝) | W-R4 UnfreezeForWithdrawal | 风控/财务审核拒绝时 | 异步RPC |
| 财务(出款失败) | W-R4 UnfreezeForWithdrawal | 通道出款失败回调时 | 异步RPC |
| 财务(出款成功) | W-R5 ConfirmWithdrawalDeduct | 通道出款成功回调时 | 异步RPC |
| 财务(人工加款) | W-R6 ManualCredit | 加款审核通过时 | 异步RPC |
| 财务(人工减款) | W-R7 ManualDebit | 减款审核通过时 | 异步RPC |
| 财务(补单) | W-R8 CreditCorrection | 补单审核通过时 | 异步RPC |
| 财务(查余额) | W-R1 GetUserWalletBalance | 加减款弹窗打开时 | 同步RPC |

**总计**：8个RPC接口被财务调用（7个写 + 1个读）。全部是事件驱动的——财务侧发生了某件事之后通知钱包做对应操作。

### 4.3 币种配置供外部调用

| 外部调用方 | 币种配置接口 | 调用时机 | 调用方式 |
|-----------|------------|---------|---------|
| 财务(所有页面) | CC-R1 GetCurrencyList | 页面加载时 | 同步RPC |
| 财务(金额换算) | CC-R2 GetExchangeRate | 记录展示/统计计算时 | 同步RPC |
| 财务(统计报表) | CC-R3 GetBaseCurrency | 报表换算时 | 同步RPC |

**总计**：3个RPC被财务调用，全是**只读查询**。

## 五、钱包内部接口互相调用关系

以下梳理钱包域内部接口之间的调用关系（同服务内函数调用，非RPC）。

### 5.1 W-1 GetWalletBalance（被其他接口内部复用）

W-1 的余额查询逻辑是钱包内部的**高频复用能力**：

- W-4 CreateDepositOrder → 内部调用 W-1 逻辑做重复转账拦截（查同币种待处理订单）
- W-9 ExecuteExchange → 内部调用 W-1 逻辑做余额充足性校验
- W-11 CreateWithdrawalOrder → 内部调用 W-1 逻辑做余额充足性校验+限额校验

注意：这里说的"内部调用W-1逻辑"不是说W-4去调W-1的HTTP接口，而是W-4和W-1共享底层的余额查询函数。在代码层面是同一个 `getBalance(userID, currency)` 函数。

### 5.2 交易流水写入（被多个写操作复用）

钱包有一个统一的"写交易流水"内部函数，被以下接口调用：

- W-R2 CreditDeposit → 写充值流水
- W-9 ExecuteExchange → 写兑换流水
- W-11 CreateWithdrawalOrder → 写提现流水（初始状态）
- W-R4 UnfreezeForWithdrawal → 更新提现流水（拒绝/失败）
- W-R5 ConfirmWithdrawalDeduct → 更新提现流水（完成）
- W-R6 ManualCredit → 写加款流水
- W-R7 ManualDebit → 写减款流水
- W-R8 CreditCorrection → 写补单流水

这些流水是 W-14/W-15 记录查询的数据源。所有余额变动都经过这个统一入口，保证对账一致性。

### 5.3 余额变更（被多个写操作复用）

余额增减的底层函数被以下接口调用：

增加余额的接口：W-R2（充值上账）、W-R6（人工加款）、W-R8（补单上账）、W-R4（解冻返还）、W-9（兑换加BSB）
减少余额的接口：W-R7（人工减款）、W-R5（确认扣款）、W-9（兑换扣法币）
冻结相关接口：W-R3（冻结，可用→冻结）、W-R4（解冻，冻结→可用）、W-R5（扣款，冻结→移除）

### 5.4 接口互调关系图

```
内部共享函数层：
  getBalance()        ← W-1, W-4, W-9, W-11 调用
  addBalance()        ← W-R2, W-R4, W-R6, W-R8, W-9 调用
  deductBalance()     ← W-R5, W-R7, W-9 调用
  freezeBalance()     ← W-R3, W-11 调用
  unfreezeBalance()   ← W-R4 调用
  writeTransaction()  ← W-R2, W-R4, W-R5, W-R6, W-R7, W-R8, W-9, W-11 调用

C端接口层（不互调）：
  W-1 ~ W-15 之间不存在直接的接口互调关系。
  它们通过共享底层函数间接关联。

  例外：W-11 CreateWithdrawalOrder 内部包含 W-R3 FreezeForWithdrawal 的逻辑
  （因为创建订单和冻结必须在同一事务中）

RPC供他层（不互调）：
  W-R1 ~ W-R8 之间不存在互调关系。
  每个RPC接口独立处理来自财务的一种操作。
```
