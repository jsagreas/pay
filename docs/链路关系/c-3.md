## 二、币种配置模块内部调用链路

币种配置模块内部各接口之间的调用关系相对简单，因为它是基础设施层，接口之间耦合度低。

### 2.1 B端接口内部链路

```
B5(设置基准币种)
  └→ 写入基准币种表（一次性操作）
  └→ 后续影响：B1列表中"基准币种"行禁止禁用操作（B3需读取基准币种判断）

B2(编辑币种配置)
  └→ 修改汇率浮动%、阈值%、精度等参数
  └→ 触发 B10(操作日志查询) 中的日志写入 ← 编辑操作产生操作日志记录

B3(启用/禁用币种)
  └→ 前置校验：读取基准币种信息（基准币种不可禁用）
  └→ 禁用时：需检查是否有关联业务在用
  └→ 触发操作日志写入

B7(手动刷新实时汇率)
  └→ 调用3家三方汇率API取均值
  └→ 计算偏差值 = (实时汇率 - 平台汇率) / 实时汇率 × 100%
  └→ 判断偏差值 ≥ 阈值% → 是 → 更新平台汇率 → 写入 B8(汇率日志) 记录
  └→ 否 → 平台汇率不变，不产生汇率日志

B1(币种列表查询)
  └→ 读取币种配置表 + 关联读取当前实时汇率/平台汇率/入款汇率/出款汇率/偏差值
  └→ 入款汇率和出款汇率是根据平台汇率和汇率浮动%实时计算的
```

**B端接口内部关系图**：

```
B5(设置基准币种) ──影响──→ B3(启禁用) ──产生──→ B10(操作日志)
                                ↑
B2(编辑配置) ──────产生──→ B10(操作日志)
      │
      └─修改汇率浮动/阈值─→ 影响 B1(列表) 中入款汇率/出款汇率计算

B7(手动刷新汇率) ──触发阈值判断─→ 可能更新平台汇率 ──产生──→ B8(汇率日志)
                                                              ↓
                                                      B9(汇率日志导出)
```

### 2.2 定时任务与B端的关系

```
T1(汇率自动刷新) ──执行逻辑与 B7 完全一致──→ 同样触发阈值判断 → 可能更新平台汇率 → 写入汇率日志
  │
  └→ T1 是定时自动执行，B7 是手动触发，两者调用同一套核心逻辑
```

### 2.3 C端接口与B端的关系

```
C1(获取已启用币种列表) ←读取── B3(启禁用) 决定的启用状态
C2(获取指定币种汇率)   ←读取── B7/T1 更新的平台汇率，以及 B2 配置的汇率浮动%
C3(获取基准币种信息)   ←读取── B5 设置的基准币种数据
```

### 2.4 RPC接口与内部数据的关系

```
R1(GetPlatformRate) ←读取── B7/T1 更新的平台汇率 + B2 配置的汇率浮动%
  └→ 返回入款汇率和出款汇率

R2(ConvertToBaseCurrency) ←读取── R1 的平台汇率 + B5 的基准币种
  └→ 金额 × 汇率 → 基准币种USDT金额

R3(GetCurrencyConfig) ←读取── B2 编辑的精度/千分位/图标等配置
  └→ 返回完整的币种配置信息

R4(GetAllEnabledCurrencies) ←读取── B3 设置的启用状态
  └→ 返回所有status=启用的币种列表

R5(GetTransactionRate) ←读取── R1 的实时平台汇率
  └→ 锁定当前汇率返回，用于交易期间汇率固定

R7(IsCurrencyEnabled) ←读取── B3 设置的启用状态
  └→ 返回布尔值，轻量级校验
```

**币种配置模块总结**：所有RPC接口都是读取B端管理员配置的数据+定时任务更新的汇率数据，属于"配置写入 → 数据读出"的单向数据流。模块内部不存在复杂的接口互调。

## 三、钱包模块内部接口调用链路

这是本文档的核心部分。钱包模块内部接口之间存在明确的调用依赖关系。

### 3.1 钱包核心层（W1~W3）— 被所有业务接口依赖

```
W1(查询钱包余额)
  └→ 读取钱包余额表（中心+奖励+主播+代理各子钱包余额）
  └→ 外部调用：币种配置.R3(GetCurrencyConfig) → 获取精度/千分位规则用于格式化
  └→ 计算资金钱包余额 = 中心钱包余额 + 奖励钱包余额
  └→ 不调用其他钱包内部接口

W2(获取用户可用币种列表)
  └→ 外部调用：币种配置.R4(GetAllEnabledCurrencies) → 获取平台启用币种列表
  └→ 查询本地用户币种偏好表 → 标记当前选中币种
  └→ 不调用其他钱包内部接口

W3(切换当前操作币种)
  └→ 外部调用：币种配置.R7(IsCurrencyEnabled) → 校验目标币种是否已启用
  └→ 更新本地用户当前币种设置
  └→ 不调用其他钱包内部接口
  └→ 影响：切换后所有后续操作（充值/提现/兑换/记录）都基于新币种
```

### 3.2 充值接口组（W4~W11）— 内部调用关系

```
W4(获取充值方式列表)
  ├→ 外部调用：财务.F1(GetRechargeMethodList) → 按当前币种+地区过滤可用充值方式
  ├→ 内部关系：W4 的结果决定了用户能选择 W5/W7/W8 中的哪一个
  └→ 不调用其他钱包内部接口

W5(创建USDT充值订单) —— 完整内部调用链
  ├→ 第1步：内部调用 W9(重复转账检查) → 判断是否有同UserID+同币种+同金额的待支付订单
  │   ├→ 1-2笔 → 返回软拦截提示，可继续
  │   ├→ ≥3笔 → 返回硬拦截，阻止创建
  │   └→ 0笔 → 继续
  ├→ 第2步：外部调用 币种配置.R5(GetTransactionRate) → 锁定当前汇率
  ├→ 第3步：生成平台充值订单号(C+16位) → 写入本地充值订单表
  ├→ 第4步：返回USDT收款地址+链网络信息（可能调用 W6 的逻辑获取地址）
  └→ 注意：USDT充值不走三方通道，是平台自有承载页，到账由 CB2(链上确认回调) 触发

W6(获取USDT充值地址及链网络)
  └→ 读取本地USDT地址配置或调用区块链服务获取地址
  └→ 返回TRC20(默认)/ERC20/BEP20地址+二维码数据
  └→ 不调用其他钱包内部接口
  └→ W5 在创建订单时可能复用 W6 的地址获取逻辑

W7(创建银行转账充值订单) —— 完整内部调用链
  ├→ 第1步：内部调用 W9(重复转账检查) → 同 W5
  ├→ 第2步：外部调用 币种配置.R5(GetTransactionRate) → 锁定当前汇率
  ├→ 第3步：生成平台充值订单号(C+16位) → 写入本地充值订单表
  ├→ 第4步：调用三方支付通道创建订单 → 获取第三方跳转URL
  │         (通道选择可能通过财务模块的轮询算法匹配)
  └→ 第5步：返回三方跳转URL给C端

W8(创建扫码充值订单) —— 调用链路与 W7 完全一致
  ├→ 第1步：内部调用 W9(重复转账检查)
  ├→ 第2步：外部调用 币种配置.R5(GetTransactionRate)
  ├→ 第3步：生成订单号(C+16位) → 写入本地订单表
  ├→ 第4步：调用三方通道 → 获取二维码数据+倒计时
  └→ 第5步：返回二维码数据给C端

W9(重复转账检查)
  └→ 查询本地充值订单表：WHERE user_id=X AND currency=Y AND amount=Z AND status IN (待支付,进行中)
  └→ 统计匹配订单数 → 返回拦截等级（0=放行/1-2=软拦截/≥3=硬拦截）
  └→ 纯本地操作，不调用其他接口
  └→ 依据：需求文档5-5重复转账拦截规则
  └→ 适用范围：仅银行转账和电子钱包(W7/W8)需要检查，USDT链上(W5)不适用
  └→ ⚠ 注意：虽然USDT不存在重复下单问题，但从接口设计统一性角度，W5也可调用W9（直接返回放行）

W10(获取额外奖金选项列表)
  └→ 可能读取本地奖金活动配置表，或调用运营活动模块RPC
  └→ 返回可用奖金列表（含最低充值门槛+奖金比例+流水倍数）
  └→ 不调用其他钱包内部接口
  └→ 与 W5/W7/W8 的关系：用户选择奖金后，创建订单时将奖金信息写入订单

W11(查询充值订单状态)
  └→ 读取本地充值订单表的状态字段
  └→ 纯本地查询，不调用其他接口
```

**充值接口组内部调用关系图**：

```
W4(充值方式列表) ──用户选择──→ W5(USDT) / W7(银行) / W8(扫码)
                                  │           │           │
                                  ▼           ▼           ▼
                              W9(重复转账检查) ←──────────┘
                                  │
                              W10(奖金列表) ← 用户可选奖金
                                  │
                              创建订单 → 写入本地DB
                                  │
                              W11(订单状态查询) ← 用户轮询
                                  │
                              CB1/CB2(回调) ← 三方/链上通知到账
```

### 3.3 兑换接口组（W12~W13）— 内部调用关系

```
W12(兑换预览/汇率试算)
  ├→ 外部调用：币种配置.R1(GetPlatformRate) → 获取入款汇率（法币→BSB方向）
  ├→ 本地计算：兑换金额、兑换获得BSB、赠送X%、实际到账BSB
  ├→ 内部依赖：需要读取用户当前币种的余额（复用 W1 的余额查询逻辑）
  │   └→ 用于判断输入金额是否超出余额（"已达当前可兑换上限"）
  └→ 只做展示计算，不产生订单

W13(创建兑换订单)
  ├→ 第1步：外部调用 币种配置.R5(GetTransactionRate) → 锁定兑换汇率
  ├→ 第2步：再次校验余额是否足够（复用 W1 余额查询逻辑 或直接读表）
  ├→ 第3步：在同一个事务中执行三个操作——
  │   ├→ 扣减源币种中心钱包余额（复用 E3 DebitWallet 的核心逻辑）
  │   ├→ 增加BSB中心钱包余额（复用 E2 CreditWallet 的核心逻辑）
  │   └→ 赠送部分增加BSB奖励钱包余额（复用 E2 CreditWallet 的核心逻辑）
  ├→ 第4步：生成兑换流水记录
  └→ 第5步：返回兑换成功结果
```

**关键点**：W13 内部复用了 E2(CreditWallet) 和 E3(DebitWallet) 的核心余额操作逻辑。从Service层设计看，兑换服务内部调用了钱包核心服务的加款/减款方法。这不是RPC调用，而是同一个服务内部的方法调用。

```
W12(兑换预览) ──用户确认──→ W13(创建兑换订单)
                                │
                    ┌───────────┼───────────┐
                    ▼           ▼           ▼
             DebitWallet逻辑  CreditWallet逻辑  CreditWallet逻辑
             (扣源币种)      (加BSB中心钱包)   (加BSB奖励钱包)
                    │           │           │
                    └───────────┼───────────┘
                                ▼
                          写入兑换流水记录
```

### 3.4 提现接口组（W14~W22）— 内部调用关系

```
W14(获取提现方式列表)
  ├→ 外部调用：用户.D9(GetKYCStatus) → 判断是否已完成KYC认证
  │   ├→ 未认证 → 仅返回USDT提现方式
  │   └→ 已认证 → 返回全部提现方式（银行卡+电子钱包+USDT）
  ├→ 外部调用：财务.F2(GetWithdrawMethodList) → 按币种获取可用提现方式
  └→ 两个外部结果取交集后返回给C端

W15(创建提现订单) —— 最复杂的内部调用链
  ├→ 第1步：校验提现限额
  │   └→ 外部调用 财务.F3(GetWithdrawConfig) → 获取单笔限额/单日限额
  │   └→ 本地查询今日已提现总额 → 判断是否超出限额
  ├→ 第2步：外部调用 币种配置.R5(GetTransactionRate) → 锁定出款汇率
  │   └→ 仅BSB提现时需要（BSB→法币/USDT换算），法币直提不需要
  ├→ 第3步：计算手续费
  │   └→ 手续费 = 提现金额 × 手续费率（来自 W16/F3 的配置）
  │   └→ 到账金额 = 提现金额 - 手续费
  ├→ 第4步：内部调用 E4(FreezeBalance) 的核心逻辑
  │   └→ 冻结用户对应金额：从可用余额移入冻结余额
  │   └→ 这是同一服务内部的方法调用，不是RPC
  ├→ 第5步：生成平台提现订单号(T+16位) → 写入本地提现订单表
  ├→ 第6步：外部调用 财务.F4(SubmitWithdrawOrder) → 提交到财务审核流程
  │   └→ ⚠ 如果此步失败 → 必须回滚第4步的冻结（调用 UnfreezeBalance 逻辑）
  └→ 第7步：返回"提现进行中"结果给C端

  ⚠ 关键事务设计：第4步（冻结）和第6步（提交财务）之间的一致性——
    成功路径：冻结成功 → 提交财务成功 → 返回成功
    失败路径：冻结成功 → 提交财务失败 → 回滚解冻 → 返回失败

W16(查询提现限额及费率)
  └→ 外部调用：财务.F3(GetWithdrawConfig) → 获取限额/费率/T+N配置
  └→ 返回给C端展示
  └→ 不调用其他钱包内部接口

W17(获取银行卡账户信息)
  └→ 读取本地提现账户表 → 返回已保存的银行卡信息
  └→ 纯本地查询

W18(保存银行卡账户信息)
  ├→ 外部调用：用户.D10(GetUserInfo) → 获取KYC姓名用于自动填充持有人
  ├→ 校验：持有人姓名必须与KYC姓名一致（忽略大小写和多余空格）
  └→ 写入本地提现账户表（首次填写，后续只读不可自行修改）

W19(获取电子钱包账户信息)
  └→ 同 W17 逻辑

W20(保存电子钱包账户信息)
  └→ 同 W18 逻辑

W21(获取USDT提现地址)
  └→ 同 W17 逻辑

W22(保存USDT提现地址)
  └→ 类似 W18 逻辑，但不需要比对KYC姓名（USDT地址不涉及持有人）
```

**提现接口组内部调用关系图**：

```
W14(提现方式列表) ──用户选择──→ W17/W19/W21(获取账户信息)
       │                              │
       │                     首次? ──→ W18/W20/W22(保存账户信息)
       │                              │
       ▼                              ▼
W16(限额费率查询) ──────────→ W15(创建提现订单)
                                │
                    ┌───────────┤
                    ▼           ▼
              FreezeBalance   SubmitWithdrawOrder
              (冻结余额-内部)  (提交财务-外部)
                    │           │
                    │       失败? ──→ UnfreezeBalance(回滚解冻-内部)
                    ▼
              写入提现订单表(T+16位)
```

### 3.5 记录接口组（W23~W27）— 内部调用关系

```
W23(充值记录分页)
  └→ 读取本地充值订单表 → 按当前币种过滤 → 分页返回
  └→ 不调用其他接口
  └→ 数据来源：W5/W7/W8 创建的充值订单 + CB1/CB2 更新的状态

W24(提现记录分页)
  └→ 读取本地提现订单表 → 按当前币种过滤 → 分页返回
  └→ 不调用其他接口
  └→ 数据来源：W15 创建的提现订单 + 财务审核/出款结果更新的状态

W25(兑换记录分页)
  └→ 读取本地兑换流水表 → 按当前币种过滤 → 分页返回
  └→ 不调用其他接口
  └→ 数据来源：W13 创建的兑换记录

W26(奖励记录分页)
  └→ 读取本地奖励流水表 → 按当前币种过滤 → 分页返回
  └→ 不调用其他接口
  └→ 数据来源：活动奖金入账、任务奖励等（E2 CreditWallet 写入奖励钱包时产生）

W27(记录详情查询)
  └→ 根据订单号+类型读取对应表的详情
  └→ 不调用其他接口
```

**记录接口组不存在内部互调**，每个接口独立查询各自的数据表。但它们的数据来源分别依赖其他接口组产生的订单数据。

### 3.6 RPC暴露层（E1~E6）— 内部调用关系

```
E1(GetBalance)
  └→ 读取钱包余额表 → 返回各子钱包余额明细
  └→ 纯本地读操作，不调用其他接口
  └→ 被 W12(兑换预览)/W13(兑换订单)/W15(提现订单) 内部复用余额查询逻辑

E2(CreditWallet)
  ├→ 往指定用户的指定子钱包增加余额
  ├→ 写入钱包余额表 + 写入资金流水表（流水记录：操作类型/金额/订单号/时间/操作前余额/操作后余额）
  ├→ 纯本地写操作，不调用其他接口
  └→ 被 W13(兑换) 内部复用加款逻辑
  └→ 被财务模块通过RPC调用：充值入账/补单/加款场景

E3(DebitWallet)
  ├→ 从指定用户的指定子钱包扣减余额
  ├→ 校验：余额不足时实际扣款=当前余额（依据：需求文档2-5人工减款）
  ├→ 写入钱包余额表 + 写入资金流水表
  ├→ 纯本地写操作，不调用其他接口
  └→ 被 W13(兑换) 内部复用减款逻辑
  └→ 被财务模块通过RPC调用：人工减款场景

E4(FreezeBalance)
  ├→ 从用户可用余额中冻结指定金额 → 可用余额减少，冻结余额增加
  ├→ 校验：可用余额 ≥ 冻结金额
  ├→ 写入钱包余额表（可用余额、冻结余额两个字段变更）
  ├→ 纯本地写操作，不调用其他接口
  └→ 被 W15(创建提现订单) 内部复用冻结逻辑

E5(UnfreezeBalance)
  ├→ 解冻归还：冻结余额减少，可用余额恢复
  ├→ 写入钱包余额表 + 写入资金流水表
  ├→ 纯本地写操作，不调用其他接口
  └→ 被 W15 提交财务失败时内部调用回滚
  └→ 被财务模块通过RPC调用：审核驳回/出款失败场景

E6(ConfirmFrozenDeduction)
  ├→ 最终扣除冻结资金：冻结余额减少，确认出账
  ├→ 写入钱包余额表 + 写入资金流水表
  ├→ 纯本地写操作，不调用其他接口
  └→ 被财务模块通过RPC调用：出款成功场景
```

**RPC暴露层内部关系图**：

```
                 ┌─── E2(CreditWallet) ←──── 财务调用(充值入账/补单/加款)
                 │                     ←──── W13(兑换)内部复用
                 │
                 ├─── E3(DebitWallet)  ←──── 财务调用(人工减款)
E1(GetBalance) ──┤                     ←──── W13(兑换)内部复用
(余额查询基础)   │
                 ├─── E4(FreezeBalance) ←── W15(提现)内部复用
                 │         │
                 │         ▼ (冻结后续)
                 ├─── E5(UnfreezeBalance) ←── 财务调用(驳回/出款失败)
                 │                        ←── W15失败回滚
                 │
                 └─── E6(ConfirmFrozenDeduction) ←── 财务调用(出款成功)

注意：E4→E5→E6 是提现生命周期的三个阶段，不是互相调用，而是由不同触发方在不同时间点分别调用
```

### 3.7 回调接口（CB1~CB2）— 内部调用关系

```
CB1(充值通道回调通知) —— 归属待评审确认
  ├→ 如果回调到钱包模块：
  │   ├→ 验证回调签名 + 幂等校验（同一订单号不重复处理）
  │   ├→ 更新本地充值订单状态
  │   ├→ 内部调用 E2(CreditWallet) → 订单金额入中心钱包
  │   ├→ 如有奖金 → 再次调用 E2(CreditWallet) → 奖金入奖励钱包
  │   └→ 写入资金流水记录
  │
  └→ 如果回调到财务模块（大概率）：
      └→ 财务验证后通过RPC调用钱包 E2(CreditWallet) → 同上入账逻辑

CB2(USDT链上确认回调)
  ├→ 接收区块链监控服务的确认通知
  ├→ 幂等校验
  ├→ 更新本地充值订单状态
  ├→ 内部调用 E2(CreditWallet) → USDT金额入中心钱包
  └→ 写入资金流水记录
```

## 四、钱包模块 → 币种配置模块 调用链路

以下列出钱包模块中每个需要调用币种配置RPC的接口，说明在什么时机调用、调用哪个RPC、传入什么参数、获取什么结果。

### 4.1 调用 R3(GetCurrencyConfig) 的接口

```
W1(查询钱包余额) ──调用──→ R3(GetCurrencyConfig)
  时机：每次加载钱包首页
  传入：当前币种代码
  获取：千分位符号、小数点符号、小数精度（法币2位/加密8位）、币种图标
  用途：格式化余额金额展示（如 VND: 9.000.000 / BSB: 1,000,000.00）
  依据：需求文档4-1钱包总体设计中的金额展示规则
```

### 4.2 调用 R4(GetAllEnabledCurrencies) 的接口

```
W2(获取用户可用币种列表) ──调用──→ R4(GetAllEnabledCurrencies)
  时机：加载钱包首页、进入币种切换页面
  传入：无（获取全量已启用币种）
  获取：所有已启用的币种列表（含币种代码、名称、类型、图标）
  用途：展示可切换的币种列表
  依据：需求文档4-2币种切换规则
```

### 4.3 调用 R7(IsCurrencyEnabled) 的接口

```
W3(切换当前操作币种) ──调用──→ R7(IsCurrencyEnabled)
  时机：用户点击切换币种时
  传入：目标币种代码
  获取：布尔值（是否已启用）
  用途：防止切换到已被B端禁用的币种
  依据：需求文档3-3后台配置说明中"禁用后对应币种在C端不可使用"
```

### 4.4 调用 R1(GetPlatformRate) 的接口

```
W12(兑换预览) ──调用──→ R1(GetPlatformRate)
  时机：用户进入兑换页面
  传入：源币种代码(法币) + 目标币种代码(BSB)
  获取：平台入款汇率
  用途：展示"1 BSB = ₫2,620"，计算兑换预览金额
  依据：需求文档6兑换模块中的汇率展示

W4(获取充值方式列表) 或充值页面加载时 ──调用──→ R1(GetPlatformRate)
  时机：BSB充值页面加载时需要展示汇率信息
  传入：当前币种 + BSB
  获取：平台入款汇率
  用途：BSB充值金额输入框下展示"汇率：1BSB=0.1USDT"
  依据：需求文档5-1充值方式总览中"只有BSB充值金额输入框下有对应汇率展示"
```

### 4.5 调用 R5(GetTransactionRate) 的接口

```
W5(创建USDT充值订单) ──调用──→ R5(GetTransactionRate)
  时机：用户点击"去充值"后，创建订单前
  传入：源币种+目标币种+方向(入款)
  获取：锁定的交易汇率
  用途：锁定汇率写入订单，确保支付期间汇率不变
  依据：需求文档4-1汇率应用场景

W7(创建银行转账充值订单) ──调用──→ R5(GetTransactionRate)
  时机/用途：同 W5

W8(创建扫码充值订单) ──调用──→ R5(GetTransactionRate)
  时机/用途：同 W5

W13(创建兑换订单) ──调用──→ R5(GetTransactionRate)
  时机：用户点击"兑换"后
  传入：源币种(法币)+目标币种(BSB)+方向(入款)
  获取：锁定的兑换汇率
  用途：锁定汇率后执行兑换操作
  依据：需求文档6兑换模块

W15(创建提现订单) ──调用──→ R5(GetTransactionRate)
  时机：用户点击"确认提现"后
  传入：源币种+目标币种+方向(出款)
  获取：锁定的出款汇率
  用途：BSB提现需要按出款汇率换算为法币/USDT
  依据：需求文档4-1汇率应用场景中"提现：跨币种提现时使用平台出款汇率"
  备注：仅BSB提现需要，法币直提不涉及跨币种换算
```

### 4.6 调用汇总

```
钱包接口 ──→ 币种配置RPC 调用关系汇总

W1  ──→ R3(GetCurrencyConfig)        展示格式化
W2  ──→ R4(GetAllEnabledCurrencies)   币种列表
W3  ──→ R7(IsCurrencyEnabled)         币种校验
W5  ──→ R5(GetTransactionRate)        充值锁汇
W7  ──→ R5(GetTransactionRate)        充值锁汇
W8  ──→ R5(GetTransactionRate)        充值锁汇
W12 ──→ R1(GetPlatformRate)           兑换预览汇率
W13 ──→ R5(GetTransactionRate)        兑换锁汇
W15 ──→ R5(GetTransactionRate)        提现锁汇

共涉及币种配置的4个RPC接口：R1/R3/R4/R5/R7
被调用频率：R5最高（每次创建订单都调用），R4/R3次之（页面加载），R1/R7最低
```

## 五、钱包模块 → 财务模块 调用链路

### 5.1 调用 F1(GetRechargeMethodList) 的接口

```
W4(获取充值方式列表) ──调用──→ F1(GetRechargeMethodList)
  时机：C端用户进入充值页面
  传入：当前币种代码 + 用户地区
  获取：可用充值方式列表（如 BSB: USDT推荐/银行/扫码；VND: 越南银行）
  用途：展示充值方式供用户选择
  依据：需求文档5-1充值方式总览 + 财务需求文档2-2支付配置-1充值方式

  返回数据结构（推测）：
  - 充值方式名称、支付图标、支付类型、充值档位(快捷金额)、充值金额范围(最小~最大)、排序
```

### 5.2 调用 F2(GetWithdrawMethodList) 的接口

```
W14(获取提现方式列表) ──调用──→ F2(GetWithdrawMethodList)
  时机：C端用户进入提现页面
  传入：当前币种代码
  获取：可用提现方式列表（如 VND: 银行卡T+1；IDR: GoPay即时/OVO即时/银行T+1）
  用途：结合KYC状态过滤后展示提现方式
  依据：需求文档7-1提现模块 + 财务需求文档2-2支付配置-2提现方式

  W14 内部逻辑：
  KYC已认证 → 返回 F2 的全部结果
  KYC未认证 → 从 F2 结果中仅保留USDT提现方式
```

### 5.3 调用 F3(GetWithdrawConfig) 的接口

```
W16(查询提现限额及费率) ──调用──→ F3(GetWithdrawConfig)
  时机：C端用户进入提现方式详情页
  传入：当前币种代码 + 提现方式
  获取：单笔限额(最低~最高)、单日限额、手续费率(%)、T+N到账时间
  用途：展示提现须知（"每日提现限额xxx，单笔最低xxx，手续费x%，到账T+1个工作日"）
  依据：需求文档7-5提现限制

W15(创建提现订单) ──调用──→ F3(GetWithdrawConfig)
  时机：用户点击"确认提现"后的校验阶段
  传入：同上
  获取：限额配置
  用途：校验提现金额是否超出单笔/单日限额，计算手续费
  注意：W15可能直接复用W16的调用结果（如果在同一页面会话中已缓存）
```

### 5.4 调用 F4(SubmitWithdrawOrder) 的接口

```
W15(创建提现订单) ──调用──→ F4(SubmitWithdrawOrder)
  时机：冻结余额成功后
  传入：提现订单信息（订单号T+16位、用户ID、币种、金额、手续费、到账金额、提现方式、账户信息、锁定汇率）
  获取：提交结果（成功/失败）
  用途：将提现订单提交到财务模块的风控审核→财务审核流程
  依据：财务需求文档2-4提现记录-1提现流程

  ⚠ 失败处理：
  如果 F4 调用失败（网络超时/财务服务不可用）
  → 必须回滚冻结操作 → 调用本地 UnfreezeBalance 逻辑
  → 返回用户"提现提交失败，请稍后重试"

  ⚠ 幂等设计：
  如果 F4 调用超时，无法确定是否成功
  → 不能直接解冻（可能财务已接收）
  → 需要查询财务侧订单状态后决定
```

### 5.5 调用汇总

```
钱包接口 ──→ 财务RPC 调用关系汇总

W4  ──→ F1(GetRechargeMethodList)   获取充值方式
W14 ──→ F2(GetWithdrawMethodList)   获取提现方式
W15 ──→ F3(GetWithdrawConfig)       校验提现限额
W16 ──→ F3(GetWithdrawConfig)       展示提现限额
W15 ──→ F4(SubmitWithdrawOrder)     提交提现审核

共涉及财务模块的4个RPC接口：F1/F2/F3/F4
```

## 七、财务模块 → 钱包模块 调用链路

这是反方向的调用，由财务模块（他人负责）发起，调用我们暴露的RPC接口。虽然不是我们主动调用的，但必须清楚每个场景的调用链路，因为我们需要确保RPC接口的正确性和幂等性。

### 7.1 充值入账场景

```
三方通道回调 ──→ 财务模块接收回调 ──→ 财务验证金额一致性
  │
  ├→ 验证通过 ──调用──→ 钱包.E2(CreditWallet)
  │   第一次调用：订单金额 → 中心钱包
  │   传入：user_id, wallet_type=CENTER, amount=订单金额, order_no=C+16位, source=RECHARGE
  │
  │   如有活动奖金 ──调用──→ 钱包.E2(CreditWallet)
  │   第二次调用：奖金金额 → 奖励钱包
  │   传入：user_id, wallet_type=REWARD, amount=奖金金额, order_no=C+16位, source=RECHARGE_BONUS
  │
  └→ 验证不通过 → 订单标记支付失败，不调用钱包RPC

幂等要求：同一订单号(C+16位)重复回调时，E2 必须幂等处理，不能重复入账
依据：财务需求文档2-3充值记录-1充值流程 + 需求文档9非功能性需求
```

### 7.2 提现审核场景

```
提现生命周期中财务调用钱包的完整链路：

[创建提现] ── 钱包侧已冻结(E4) ── 订单提交到财务(F4)
  │
  ├→ [风控审核-驳回]
  │   财务 ──调用──→ 钱包.E5(UnfreezeBalance)
  │   传入：user_id, currency, amount=冻结金额, order_no=T+16位
  │   效果：冻结余额减少，可用余额恢复
  │
  ├→ [风控审核-通过] → [财务审核-驳回]
  │   财务 ──调用──→ 钱包.E5(UnfreezeBalance)
  │   传入/效果：同上
  │
  ├→ [风控审核-通过] → [财务审核-通过] → [代付通道出款-失败]
  │   财务 ──调用──→ 钱包.E5(UnfreezeBalance)
  │   传入/效果：同上
  │
  └→ [风控审核-通过] → [财务审核-通过] → [代付通道出款-成功]
      财务 ──调用──→ 钱包.E6(ConfirmFrozenDeduction)
      传入：user_id, currency, amount=冻结金额, order_no=T+16位
      效果：冻结余额减少，确认出账（资金真正离开用户钱包）

提现订单一生只会被调用一次 E5 或 E6（互斥）：
  驳回/出款失败 → E5(解冻归还)
  出款成功 → E6(确认扣除)

幂等要求：同一订单号的E5/E6必须幂等，防止重复解冻或重复扣除
依据：财务需求文档2-4提现记录-1提现流程
```

### 7.3 充值补单场景

```
B端管理员创建补单(B+16位) → 另一管理员审核
  │
  ├→ 审核通过 ──调用──→ 钱包.E2(CreditWallet)
  │   第一次调用：补单订单金额 → 中心钱包
  │   传入：user_id, wallet_type=CENTER, amount=补单金额, order_no=B+16位, source=SUPPLEMENT
  │
  │   如有补单赠送 ──调用──→ 钱包.E2(CreditWallet)
  │   第二次调用：补单赠送金额 → 奖励钱包
  │   传入：user_id, wallet_type=REWARD, amount=补单赠送, order_no=B+16位, source=SUPPLEMENT_BONUS
  │
  └→ 审核拒绝 → 不调用钱包RPC

依据：财务需求文档2-5人工修正-1充值补单
```

### 7.4 人工加款场景

```
B端管理员创建加款(A+16位)
  │
```

### 7.5 人工减款场景


### 7.6 调用汇总

```
财务 ──→ 钱包RPC 调用关系汇总

充值成功入账     ──→ E2(CreditWallet)  × 1~2次
充值补单审核通过 ──→ E2(CreditWallet)  × 1~2次
人工加款审核通过 ──→ E2(CreditWallet)  × 1~4次
人工减款创建     ──→ E1(GetBalance)    × 1次
人工加款创建     ──→ E1(GetBalance)    × 1次
人工减款审核通过 ──→ E3(DebitWallet)   × 1~4次
提现驳回/出款失败──→ E5(UnfreezeBalance)    × 1次
提现出款成功     ──→ E6(ConfirmFrozenDeduction) × 1次

E2(CreditWallet) 被调用频率最高 — 覆盖充值入账+补单+加款三个场景
E1(GetBalance) 被调用频率次之 — 人工加减款创建时查余额
E5/E6 仅在提现场景的最终阶段被调用
```

### 10.1 钱包模块内部的层次调用关系

### 10.2 三条最关键的调用链路

**第一条（依赖链最长）**：提现完整链路
```
W14 →(D9+F2) → W16 →(F3) → W18 →(D10) → W15 →(R5+E4+F4) → 等待 → 财务审核 →(E5/E6)
涉及4个模块：钱包 + 币种配置 + 财务 + 用户
```

**第二条（调用频率最高）**：充值链路
```
W4 →(F1) → W9 → W5/W7/W8 →(R5) → 等待 → CB1/CB2 →(E2)
涉及3个模块：钱包 + 币种配置 + 财务
```

**第三条（最简洁）**：兑换链路
```
W12 →(R1) → W13 →(R5+E3+E2)
仅涉及2个模块：钱包 + 币种配置
```

### 10.4 内部接口复用关系总结

钱包模块的Service层设计应体现以下内部复用：

```
WalletCoreService（核心服务 — 对应 E1~E6）
  ├── GetBalance()      ← ExchangeService.Preview() 调用
  ├── Credit()          ← ExchangeService.Execute() 调用
  │                     ← RechargeCallbackService.Handle() 调用
  ├── Debit()           ← ExchangeService.Execute() 调用
  ├── Freeze()          ← WithdrawService.Create() 调用
  ├── Unfreeze()        ← WithdrawService.Create() 失败回滚调用
  └── ConfirmDeduction()  （仅通过RPC被财务调用，内部不使用）

RechargeService（充值服务 — 对应 W4~W11）
  └── DuplicateCheckService.Check()  ← W5/W7/W8 创建前调用

ExchangeService（兑换服务 — 对应 W12~W13）
  ├── WalletCoreService.GetBalance()
  ├── WalletCoreService.Debit()
  └── WalletCoreService.Credit()
```
