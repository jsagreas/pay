# 接口调用链路关系（模块内 + 模块间）

> 基于：接口评估（d-1.md）+ 需求理解（d-1.md）+ 依赖关系（d-1.md）+ 产品原型
> 目的：逐接口梳理钱包模块每个接口的内部调用链和外部依赖，以及与财务模块的跨模块调用链路
> 视角：**我负责钱包+币种配置模块**，需要清楚每个接口"调了谁、被谁调、内部经过哪些步骤"

## 一、全局调用链路概览

### 1.1 钱包模块对外调用关系

```
钱包模块的每个接口，对外最多涉及以下4类调用目标：

  ┌─────────────────────────────────────────────────────┐
  │                  ser-wallet（钱包模块）                │
  │                                                      │
  │  C端接口(22个) + B端接口(5个) + RPC提供接口(8个)       │
  │                                                      │
  │      调用出口：                   被调用入口：          │
  │      ├─ 币种配置 RPC (5条)        ├─ 财务回调 (5条)    │
  │      ├─ 财务 RPC (2条)            └─ 三方/财务回调(1条)│
  │      ├─ ser-kyc RPC (1条)                             │
  │      └─ ser-blog oneway (1条)                         │
  └─────────────────────────────────────────────────────┘
```

### 1.2 调用关系分类说明

- **内部调用**：同一个 ser-wallet 服务内，接口之间、接口与内部方法之间的调用（如 CreateWithdrawOrder 内部调用 FreezeBalance 的底层逻辑）
- **RPC调出**：钱包作为调用方，通过 RPC 调用其他服务（币种配置、财务、KYC、审计日志）
- **RPC被调**：其他服务（主要是财务）通过 RPC 调用钱包暴露的接口
- **DB操作**：接口对本地数据库的读写（标记为 [DB读] 或 [DB写]）

## 二、钱包模块 C 端接口逐一调用链路

> 每个接口按"用户触发 → 接口入口 → 内部步骤 → 外部调用 → 返回"的顺序描述

### C1 GetWalletBalance — 当前币种钱包余额

**触发场景**：用户打开钱包首页、切换币种后刷新余额

**调用链路**：
```
用户请求(币种ID)
  → [DB读] 查询该用户在该币种下的所有钱包账户（中心/奖励/主播/代理）
  → 汇总各钱包余额（可用余额、冻结余额）
  → 返回：资金钱包余额(中心+奖励)、各钱包明细
```

**外部调用**：无。纯本地 DB 查询。

**内部依赖**：依赖钱包账户表（wallet_account）中有该用户该币种的记录。如果用户首次使用某币种，可能需要自动初始化账户记录（懒创建）。

**被谁复用**：
- C15 `CreateWithdrawOrder` 内部校验余额时会用到相同的余额查询逻辑
- R6 `GetUserBalance` 是这个查询逻辑的 RPC 版本

---

### C2 GetWalletOverview — 所有币种余额概览

**触发场景**：用户点击币种切换下拉时，展示所有币种的余额列表

**调用链路**：
```
用户请求
  → [RPC调出] 币种配置.GetActiveCurrencyList → 获取所有启用币种列表
  → [DB读] 批量查询该用户在所有启用币种下的钱包账户余额
  → [RPC调出] 币种配置.BatchGetExchangeRates → 获取各币种对BSB汇率
  → 计算每个币种余额折合BSB值，计算总资产
  → 返回：[{币种代码, 币种名称, 图标, 余额, 折合BSB}, ...]，总资产(BSB)
```

**外部调用**：
- 币种配置.GetActiveCurrencyList — 获取币种列表
- 币种配置.BatchGetExchangeRates — 获取汇率用于总资产换算

**内部依赖**：同 C1 的余额查询逻辑，只是批量执行

### C3 GetDepositMethods — 获取充值方式列表

**触发场景**：用户进入充值页面时展示可用的充值方式（银行转账/电子钱包/USDT）

**调用链路**：
```
用户请求(币种ID)
  → [RPC调出] 财务.GetPaymentMethods(币种ID, 类型=充值) → 获取充值方式列表
  → [RPC调出] 币种配置.GetDepositWithdrawConfig(币种ID) → 获取充值档位配置
  → 组装返回数据：每种充值方式的名称、icon、排序、对应的充值档位
  → 如果是BSB币种，额外获取汇率展示
    → [RPC调出] 币种配置.GetExchangeRate(BSB→USDT) → 获取入款汇率
  → 返回：[{方式名称, icon, 档位列表, 汇率(BSB时)}, ...]
```

**外部调用**：
- 财务.GetPaymentMethods — **关键依赖**，充值方式数据源在财务
- 币种配置.GetDepositWithdrawConfig — 充值档位/快捷金额
- 币种配置.GetExchangeRate — 仅BSB充值时需要展示汇率

**说明**：这个接口是钱包与财务在充值流程中的**第一个交互点**。财务的支付配置必须先配好，这个接口才能返回数据。

### C4 GetDepositBonusList — 获取充值可选活动奖金

**触发场景**：充值页面中"请选择您的奖金"浮窗展示可用奖金活动

**调用链路**：

**外部调用**：

**说明**：需求文档明确奖金选择功能存在，但数据归属（活动系统 or 钱包本地）未最终确定。当前按P1优先级处理。

### C5 CreateDepositOrder — 创建充值订单（银行/扫码类）

**触发场景**：用户选好金额和方式后点击"去充值"

**调用链路**：
```
用户请求(币种ID, 充值方式, 金额, 奖金活动ID(可选))
  │
  ├─ 步骤1：重复转账检测 [内部调用 C7 CheckDuplicateDeposit 的逻辑]
  │   → [DB读] 查询同UserID+同币种+同金额的"待支付/进行中"订单数
  │   → 0笔：继续
  │   → 1~2笔：返回提示但允许继续
  │   → 3笔：拦截，返回错误码，不创建订单
  │
  ├─ 步骤2：校验充值金额
  │   → [RPC调出] 币种配置.GetDepositWithdrawConfig(币种ID) → 获取最低档位
  │   → 校验金额 ≥ 最低档位
  │
  ├─ 步骤3：奖金门槛校验（如选了奖金活动）
  │   → 校验充值金额 ≥ 奖金活动最低充值门槛
  │
  ├─ 步骤4：匹配支付通道
  │   → [RPC调出] 财务.MatchDepositChannel(币种ID, 充值方式, 金额)
  │   → 返回通道信息（通道ID, 通道参数）
  │   → 匹配失败 → 返回"充值通道异常失效，请更换充值方式"
  │
  ├─ 步骤5：创建充值订单
  │   → [DB写] 写入充值订单表（状态=待支付，锁定创建时汇率快照）
  │   → [RPC调出] ser-blog.AddActLog(oneway) → 记录操作日志
  │
  ├─ 步骤6：向三方发起支付
  │   → 通过匹配到的通道参数，构造三方支付请求
  │   → 获取三方返回的支付页面URL / 支付信息
  │
  └─ 返回：订单号、三方支付页面URL（前端跳转）
```

**外部调用**：
- 币种配置.GetDepositWithdrawConfig — 校验最低充值金额
- 财务.MatchDepositChannel — **关键依赖**，匹配支付通道
- ser-blog.AddActLog — 审计日志

**内部调用**：
- 复用 C7 重复转账检测的核心逻辑
- 写入充值订单表（deposit_order）

**说明**：这是充值流程中**最重要的接口**，串联了校验→通道匹配→建单→发起支付的完整链路。

### C6 GetUsdtDepositAddress — USDT充值获取收款地址

**触发场景**：用户选择USDT充值方式后展示收款地址和二维码

**调用链路**：
```
用户请求(币种ID, 网络类型=TRC20/ERC20/BEP20)
  │
  ├─ 步骤1：重复转账检测 [内部调用 C7 逻辑]
  │   → 与 C5 相同的检测逻辑（但USDT链上充值的重复检测可能豁免，
  │     需求文档说"不适用于USDT链上充值"，待评审确认是否在此步骤跳过）
  │
  ├─ 步骤2：获取/生成收款地址
  │   → [DB读] 查询该用户是否已分配该网络的收款地址
  │   → 未分配：[RPC调出/三方API] 向链上服务请求分配新地址
  │   → 已分配：直接复用
  │
  ├─ 步骤3：创建充值订单
  │   → [DB写] 写入充值订单表（状态=待支付，类型=USDT链上）
  │   → 锁定创建时汇率快照
  │
  └─ 返回：网络类型、收款地址、二维码数据、风险提示文案
```

**外部调用**：
- 链上服务/三方API — 获取收款地址（具体对接方式待评审确认）
- 与 C5 共用重复检测逻辑（但USDT可能豁免）

**区别于C5**：C5 是银行/扫码类充值（跳转三方支付页面），C6 是USDT链上充值（展示收款地址让用户自行转账）。两者建单逻辑类似但支付方式不同。

### C7 CheckDuplicateDeposit — 重复转账检测

**触发场景**：可被 C5/C6 内部调用，也可作为独立接口在前端点击充值前预检

**调用链路**：
```
请求(用户ID, 币种ID, 充值金额)
  → [DB读] 查询 deposit_order 表
    条件：user_id = X AND currency_id = X AND amount = X AND status IN (待支付, 进行中)
  → 统计匹配订单数 N
  → 返回：{count: N, canProceed: N < 3, message: 对应文案}
```

**外部调用**：无。纯本地 DB 查询。

**内部关系**：被 C5 和 C6 在创建订单前内部调用。也可以作为独立C端接口供前端在用户点击"去充值"前预检。

### C9 ExchangePreview — 兑换预览

**触发场景**：用户在兑换页面输入金额后实时展示计算结果

**调用链路**：
```
请求(源币种ID, 目标币种=BSB, 金额)
  → [RPC调出] 币种配置.GetExchangeRate(源币种→BSB) → 获取入款汇率
  → 计算：兑换获得BSB = 金额 ÷ 入款汇率
  → 计算：赠送BSB = 兑换获得BSB × 赠送比例%（赠送比例从哪里来？待确认，可能是配置或活动）
  → 计算：实际到账BSB = 兑换获得 + 赠送
  → 返回：{汇率, 兑换获得, 赠送比例, 赠送金额, 实际到账, 流水倍数要求}
```

**外部调用**：
- 币种配置.GetExchangeRate — 获取兑换汇率

**说明**：此接口是**纯计算+展示**，不产生任何DB写操作。赠送比例的数据来源需评审确认（币种配置中管理 or 活动系统）。

### C10 ExecuteExchange — 执行兑换

**触发场景**：用户确认兑换后点击"兑换"按钮

**调用链路**：
```
请求(源币种ID, 目标币种=BSB, 金额)
  │
  ├─ 步骤1：获取并锁定汇率
  │   → [RPC调出] 币种配置.GetExchangeRate(源币种→BSB) → 获取入款汇率
  │   → 锁定当前汇率作为本次兑换快照
  │
  ├─ 步骤2：校验余额
  │   → [DB读] 查询源币种中心钱包可用余额
  │   → 余额不足 → 返回错误"已达当前可兑换上限"
  │
  ├─ 步骤3：执行余额变更（事务内）
  │   → [DB写] 扣减源币种中心钱包余额
  │   → [DB写] 增加BSB中心钱包余额（兑换获得部分）
  │   → [DB写] 增加BSB奖励钱包余额（赠送部分，需完成流水后可提现）
  │   → [DB写] 生成兑换流水记录（exchange_record表）
  │
  ├─ 步骤4：审计日志
  │   → [RPC调出] ser-blog.AddActLog(oneway) → 记录兑换操作
  │
  └─ 返回：兑换成功 + 明细（兑换金额、获得BSB、赠送BSB、实际到账）
```

**外部调用**：
- 币种配置.GetExchangeRate — 获取汇率
- ser-blog.AddActLog — 审计日志

**内部调用**：
- 余额查询逻辑（同 C1）
- 余额变更逻辑（扣减+增加，与 R1 CreditBalance / R4 DebitBalance 的底层余额操作逻辑复用）

**关键要求**：步骤3的多表写操作必须在**同一个DB事务**中执行，保证源币种扣减与BSB增加的原子性。

### C12 CheckWithdrawEligibility — 提现前置校验

**触发场景**：用户点击"提现"按钮时，前端先调此接口判断是否可提现

**调用链路**：
```
请求(用户ID, 币种ID)
  │
  ├─ 步骤1：查询KYC认证状态
  │   → [RPC调出] ser-kyc.KycQuery(用户ID) → 返回认证状态（未认证/审核中/已认证）
  │
  ├─ 步骤2：查询可用余额
  │   → [DB读] 查询该币种中心钱包可用余额
  │
  ├─ 步骤3：查询今日已用提现额度
  │   → [DB读] 统计今日该币种已提现成功+审核中的总金额
  │   → [RPC调出] 币种配置.GetDepositWithdrawConfig(币种ID) → 获取单日限额
  │   → 计算剩余可提额度
  │
  └─ 返回：{kycStatus, 可用余额, 单笔限额, 单日限额, 今日已用, 剩余可提}
    前端根据kycStatus决定：
      未认证 → 弹窗（"去认证" / "继续提现仅USDT"）
      已认证 → 直接进入提现选择页
```

**外部调用**：
- ser-kyc.KycQuery — 查KYC状态
- 币种配置.GetDepositWithdrawConfig — 获取提现限额配置

### C14 BindWithdrawAccount — 绑定提现账户

- ser-kyc.KycQuery — 获取KYC认证姓名
- ser-blog.AddActLog — 审计日志

### C15 CreateWithdrawOrder — 创建提现申请

**触发场景**：用户填完信息、确认金额后点击"确认提现"

**调用链路**：
```
请求(用户ID, 币种ID, 提现方式, 提现账户ID, 金额)
  │
  ├─ 步骤1：金额校验
  │   → [RPC调出] 币种配置.GetDepositWithdrawConfig(币种ID) → 获取限额和手续费规则
  │   → 校验：金额 ≥ 单笔最低限额
  │   → 校验：金额 ≤ 单笔最高限额
  │   → [DB读] 统计今日已提现金额 → 校验：今日累计+本次 ≤ 单日限额
  │
  ├─ 步骤2：余额校验
  │   → [DB读] 查询该币种可用余额（中心钱包）
  │   → 校验：可用余额 ≥ 提现金额 + 手续费
  │
  ├─ 步骤3：冻结余额（事务内）
  │   → [DB写] 可用余额减少，冻结金额增加
  │   → 冻结金额 = 提现金额 + 手续费
  │   → [DB写] 生成冻结流水记录
  │
  ├─ 步骤4：创建提现订单
  │   → [DB写] 写入 withdraw_order 表（状态=待审核，关联冻结记录）
  │   → BSB提现时锁定当前出款汇率快照
  │
  ├─ 步骤5：提交给财务审核
  │   → [RPC调出] 财务接收提现单（具体接口待财务模块确认，可能是异步消息/直接入财务表）
  │   → 或者：钱包只建单，财务通过查询/轮询获取待审核订单
  │
  ├─ 步骤6：审计日志
  │   → [RPC调出] ser-blog.AddActLog(oneway) → 记录提现申请
  │
  └─ 返回：提现申请已提交 + 订单号
```

**外部调用**：
- 币种配置.GetDepositWithdrawConfig — 限额和手续费校验
- 财务（提交审核方式待确认）
- ser-blog.AddActLog — 审计日志

**内部调用**：
- 余额查询逻辑（同 C1）
- 冻结逻辑（与 R2 FreezeBalance 的底层逻辑一致，但此处是钱包内部自行调用，不经过RPC）

**关键说明**：步骤3的冻结和步骤4的建单必须在**同一个DB事务**内，保证冻结与订单的原子性。冻结失败则不建单，建单失败则回滚冻结。

## 三、钱包模块 RPC 提供接口逐一调用链路

> 这些接口是钱包暴露给外部模块（主要是财务）调用的。每个接口都需要**幂等设计**。

### R1 CreditBalance — 充值入账

**调用方**：财务模块（充值回调成功后调用）

**调用链路**：
```
财务调用(用户ID, 币种ID, 钱包类型, 金额, 订单号, 业务类型=充值)
  │
  ├─ 步骤1：幂等校验
  │   → [DB读] 按订单号查询是否已入账
  │   → 已入账 → 直接返回成功（幂等，不重复操作）
  │
  ├─ 步骤2：订单状态校验
  │   → [DB读] 查询充值订单状态
  │   → 仅"待入账"状态才执行入账
  │   → 其他状态 → 返回错误
  │
  ├─ 步骤3：执行入账（事务内）
  │   → [DB写] 增加指定钱包账户的可用余额
  │   → [DB写] 生成入账流水记录（transaction_log）
  │   → [DB写] 更新充值订单状态为"已完成"
  │
  ├─ 步骤4：审计日志
  │   → [RPC调出] ser-blog.AddActLog(oneway)
  │
  └─ 返回：入账结果
```

**内部调用**：
- 余额增加的底层方法（与 C10 ExecuteExchange 中增加BSB余额共用）
- 流水记录写入方法

**关键要求**：
- **幂等**：基于订单号去重，同一订单多次调用只入账一次
- **事务**：余额变更、流水记录、订单状态更新在同一事务内

---

### R2 FreezeBalance — 冻结余额

**调用方**：财务模块（提现申请进入审核时）、投注系统（未来）

**调用链路**：
```
调用方(用户ID, 币种ID, 钱包类型, 金额, 关联单号)
  │
  ├─ 步骤1：幂等校验
  │   → [DB读] 按关联单号查询是否已冻结
  │   → 已冻结 → 返回成功
  │
  ├─ 步骤2：余额校验
  │   → [DB读] 查询可用余额 ≥ 冻结金额
  │   → 不足 → 返回错误
  │
  ├─ 步骤3：执行冻结（事务内）
  │   → [DB写] 可用余额 -= 冻结金额
  │   → [DB写] 冻结余额 += 冻结金额
  │   → [DB写] 生成冻结流水记录
  │
  └─ 返回：冻结结果
```

**内部关系**：C15 CreateWithdrawOrder 内部的冻结步骤与此接口的底层逻辑一致。区别在于 C15 是钱包内部直接调用底层方法（不经过RPC），R2 是外部通过RPC调用。建议抽取公共的冻结逻辑方法，C15和R2都调用它。

---

### R3 UnfreezeBalance — 解冻余额

**调用方**：财务模块（提现审核驳回/出款失败时）

**调用链路**：
```
调用方(用户ID, 币种ID, 钱包类型, 金额, 关联单号)
  │
  ├─ 步骤1：幂等校验
  │   → [DB读] 按关联单号查询冻结记录状态
  │   → 已解冻 → 返回成功
  │
  ├─ 步骤2：执行解冻（事务内）
  │   → [DB写] 冻结余额 -= 解冻金额
  │   → [DB写] 可用余额 += 解冻金额
  │   → [DB写] 更新冻结记录状态为"已解冻"
  │   → [DB写] 更新关联提现订单状态
  │   → [DB写] 生成解冻流水记录
  │
  └─ 返回：解冻结果
```

---

### R4 DebitBalance — 提现扣款确认

**调用方**：财务模块（提现出款成功后）

**调用链路**：
```
调用方(用户ID, 币种ID, 钱包类型, 金额, 关联单号)
  │
  ├─ 步骤1：幂等校验
  │   → [DB读] 按关联单号查询是否已扣款
  │   → 已扣款 → 返回成功
  │
  ├─ 步骤2：执行扣款（事务内）
  │   → [DB写] 冻结余额 -= 扣款金额（从冻结中扣减，非可用余额）
  │   → [DB写] 更新冻结记录状态为"已扣款"
  │   → [DB写] 更新关联提现订单状态为"已完成"
  │   → [DB写] 生成扣款流水记录
  │
  └─ 返回：扣款结果
```

**说明**：DebitBalance 操作的是**冻结余额**而非可用余额。正常流程是：FreezeBalance（冻结）→ DebitBalance（从冻结中扣除）。如果审核驳回则走 UnfreezeBalance（冻结→恢复为可用）。

---

### R5 ManualAdjust — 人工调整余额

**调用方**：财务模块（人工加款/减款/补单审核通过后）

**调用链路**：
```
调用方(用户ID, 币种ID, 钱包类型, 操作类型=加款/减款, 金额, 原因, 关联单号)
  │
  ├─ 步骤1：幂等校验
  │   → [DB读] 按关联单号查询是否已执行
  │   → 已执行 → 返回成功
  │
  ├─ 步骤2：减款时校验余额
  │   → 操作类型=减款：[DB读] 查询可用余额 ≥ 减款金额
  │   → 不足 → 返回错误（但财务侧应该已前置校验，这里是兜底）
  │
  ├─ 步骤3：执行余额变更（事务内）
  │   → 加款：[DB写] 可用余额 += 金额
  │   → 减款：[DB写] 可用余额 -= 金额
  │   → [DB写] 生成调整流水记录（记录原因、关联单号）
  │
  ├─ 步骤4：审计日志
  │   → [RPC调出] ser-blog.AddActLog(oneway)
  │
  └─ 返回：调整结果
```

**说明**：此接口可能需要支持**批量操作**——财务人工加款时可以同时对4类钱包分别设置金额。可以设计为单次调用传入多个钱包类型+金额的数组，在同一事务中执行。

---

### R6 GetUserBalance — 查询用户余额（RPC版）

**调用方**：财务模块、其他需要查询余额的服务

**调用链路**：
```
调用方(用户ID, 币种ID)
  → [DB读] 查询该用户该币种下所有钱包账户的余额
  → 返回：{可用余额, 冻结余额, 各钱包类型明细}
```

**外部调用**：无。纯本地 DB 查询。

**内部关系**：与 C1 GetWalletBalance 查询逻辑一致，区别在于 C1 是 HTTP 接口（从Token取用户ID），R6 是 RPC 接口（参数传入用户ID）。

---

### R7 BatchGetUserBalances — 批量查询用户余额

**调用链路**：
```
调用方(用户ID列表, 币种ID)
  → [DB读] 批量查询余额
  → 返回：map<userId, balanceInfo>
```

**外部调用**：无。

---

### R8 DepositCallback — 充值回调处理

**调用方**：财务模块转发三方支付回调，或三方直接回调

**调用链路**：
```
回调请求(订单号, 支付状态, 实际金额, 通道订单号, ...)
  │
  ├─ 步骤1：签名/来源验证
  │   → 验证回调请求的合法性
  │
  ├─ 步骤2：订单状态校验
  │   → [DB读] 查询充值订单
  │   → 订单不存在 → 返回错误
  │   → 订单已完成 → 返回成功（幂等）
  │
  ├─ 步骤3：根据支付状态处理
  │   → 支付成功：
  │     → [内部调用] R1 CreditBalance 的底层逻辑 → 执行入账
  │     → 如有关联奖金活动：
  │       → [DB写] 充值金额入中心钱包
  │       → [DB写] 奖金金额入奖励钱包（需记录流水倍数要求）
  │   → 支付失败/超时：
  │     → [DB写] 更新订单状态为失败/超时
  │
  ├─ 步骤4：审计日志
  │   → [RPC调出] ser-blog.AddActLog(oneway)
  │
  └─ 返回：处理结果
```

**内部调用**：
- R1 CreditBalance 的底层入账逻辑（非RPC调用，直接调用内部方法）

**关键说明**：R8 是 R1 的上层封装。如果财务直接调 R1 CreditBalance 入账，那 R8 主要处理三方直接回调的场景（需要解析三方回调格式）。如果所有回调都经过财务模块统一转发，R8 可以简化为与 R1 合并。

## 五、钱包模块内部公共方法复用关系

> 以下是钱包模块内部多个接口共用的底层逻辑方法，不是独立接口，但是链路分析中的关键节点。

### 5.1 余额查询方法（getBalance）

```
被以下接口调用：
  C1  GetWalletBalance         → 直接调用
  C2  GetWalletOverview        → 批量调用
  C10 ExecuteExchange          → 扣减前校验
  C12 CheckWithdrawEligibility → 提现前校验
  C15 CreateWithdrawOrder      → 提现前校验
  R2  FreezeBalance            → 冻结前校验
  R5  ManualAdjust             → 减款前校验
  R6  GetUserBalance           → RPC版直接调用
```

### 5.2 余额变更方法（modifyBalance）

```
底层操作：可用余额/冻结余额的增减 + 流水记录生成，必须在事务内

被以下接口调用（通过不同参数区分操作类型）：
  R1  CreditBalance   → modifyBalance(type=credit, target=available)
  R2  FreezeBalance   → modifyBalance(type=freeze, source=available, target=frozen)
  R3  UnfreezeBalance → modifyBalance(type=unfreeze, source=frozen, target=available)
  R4  DebitBalance    → modifyBalance(type=debit, source=frozen)
  R5  ManualAdjust    → modifyBalance(type=credit/debit, target=available)
  C10 ExecuteExchange → modifyBalance(type=debit + credit, 跨币种两次调用)
  C15 CreateWithdrawOrder → modifyBalance(type=freeze) [内部调用，非RPC]
  R8  DepositCallback → 通过R1逻辑间接调用
```

### 5.3 幂等校验方法（checkIdempotent）

```
被以下接口调用：
  R1  CreditBalance   → 按订单号校验
  R2  FreezeBalance   → 按关联单号校验
  R3  UnfreezeBalance → 按关联单号校验
  R4  DebitBalance    → 按关联单号校验
  R5  ManualAdjust    → 按关联单号校验
  R8  DepositCallback → 按订单号校验
```

### 5.4 提现限额校验方法（checkWithdrawLimit）

```
被以下接口调用：
  C12 CheckWithdrawEligibility → 展示限额信息
  C15 CreateWithdrawOrder      → 提现前校验
  C16 GetWithdrawLimit         → 查询限额信息

三者都需要：
  → 调用币种配置.GetDepositWithdrawConfig 获取限额规则
  → 查询今日已提现金额
  → 计算剩余可提额度
```

## 六、跨模块调用链路（完整业务场景视角）

> 从用户操作到最终完成，跨越多个模块的完整调用链路

### 6.1 充值全链路

```
[C端用户]
    │
    │ ① 打开充值页面
    ▼
[钱包 C3 GetDepositMethods]
    │── RPC ──→ [财务 GetPaymentMethods] ── 返回充值方式列表
    │── RPC ──→ [币种配置 GetDepositWithdrawConfig] ── 返回充值档位
    │── RPC ──→ [币种配置 GetExchangeRate] ── BSB时返回汇率
    │
    │ ② 用户选择金额+方式，点击充值
    ▼
[钱包 C5 CreateDepositOrder]
    │── 内部 ──→ [C7逻辑 重复转账检测] ── DB查询
    │── RPC ──→ [币种配置 GetDepositWithdrawConfig] ── 校验最低金额
    │── RPC ──→ [财务 MatchDepositChannel] ── 匹配支付通道
    │── DB写 ──→ 创建充值订单（状态=待支付）
    │── HTTP ──→ [三方支付通道] ── 发起支付请求
    │
    │ ③ 用户在三方完成支付
    ▼
[三方支付通道]
    │── 回调 ──→ [财务 接收回调]
    │                │
    │                │ ④ 财务处理回调
    │                ▼
    │           [财务 → 钱包 R8 DepositCallback 或 R1 CreditBalance]
    │                │── 内部 ──→ 幂等校验
    │                │── DB写 ──→ 增加余额 + 生成流水 + 更新订单状态
    │
    │ ⑤ 前端轮询
    ▼
[钱包 C8 GetDepositOrderStatus]
    │── DB读 ──→ 返回最新订单状态（成功/待支付/失败）
```

**钱包模块在此链路中涉及的接口**：C3 → C5(含C7) → R8/R1 → C8
**跨模块RPC调用数**：4次调出（2次币种配置 + 2次财务）+ 1次被调入（财务回调入账）

---

### 6.2 USDT充值全链路

```
[C端用户]
    │
    │ ① 选择USDT充值
    ▼
[钱包 C3 GetDepositMethods]
    │── 同上，获取充值方式（USDT在列表中）
    │
    │ ② 选择网络(TRC20)，输入金额
    ▼
[钱包 C6 GetUsdtDepositAddress]
    │── DB读/三方 ──→ 获取收款地址
    │── DB写 ──→ 创建充值订单
    │── 返回：收款地址 + 二维码
    │
    │ ③ 用户链上转账，区块链确认
    ▼
[链上监听服务 / 三方通道]
    │── 回调 ──→ [财务或直接回调钱包]
    │                │
    │                ▼
    │           [钱包 R8/R1 入账]
    │                │── 同充值入账逻辑
```

**区别于6.1**：不经过通道匹配（MatchDepositChannel），收款地址由链上服务管理。

### 6.3 兑换全链路

```
[C端用户]
    │
    │ ① 进入兑换页面，输入金额
    ▼
[钱包 C9 ExchangePreview]
    │── RPC ──→ [币种配置 GetExchangeRate] ── 获取入款汇率
    │── 计算 ──→ 兑换获得 + 赠送 + 实际到账
    │── 返回展示
    │
    │ ② 确认兑换
    ▼
[钱包 C10 ExecuteExchange]
    │── RPC ──→ [币种配置 GetExchangeRate] ── 再次获取汇率并锁定
    │── DB读 ──→ 校验源币种余额
    │── DB写（事务）──→ 扣减源币种余额 + 增加BSB中心钱包 + 增加BSB奖励钱包(赠送)
    │── DB写 ──→ 生成兑换记录
    │── oneway ──→ [ser-blog 审计日志]
```

**钱包模块涉及的接口**：C9 → C10
**跨模块RPC调用数**：2次调出（2次币种配置获取汇率），无被调入
**特点**：兑换是钱包内部完全自闭环的链路，不涉及财务模块

### 6.4 提现全链路

```
[C端用户]
    │
    │ ① 点击提现
    ▼
[钱包 C12 CheckWithdrawEligibility]
    │── RPC ──→ [ser-kyc KycQuery] ── 查KYC状态
    │── RPC ──→ [币种配置 GetDepositWithdrawConfig] ── 查限额
    │── DB读 ──→ 查余额 + 今日已用额度
    │── 返回：KYC状态 + 限额信息
    │
    │ ② （如需绑定提现账户）
    ▼
[钱包 C14 BindWithdrawAccount]
    │── RPC ──→ [ser-kyc KycQuery] ── 获取KYC姓名自动填充
    │── DB写 ──→ 保存提现账户
    │
    │ ③ 选择方式、输入金额、确认提现
    ▼
[钱包 C15 CreateWithdrawOrder]
    │── RPC ──→ [币种配置 GetDepositWithdrawConfig] ── 限额校验
    │── DB读 ──→ 余额校验
    │── DB写（事务）──→ 冻结余额 + 创建提现订单（状态=待审核）
    │── oneway ──→ [ser-blog 审计日志]
    │
    │ ④ 进入财务三级审核（B端操作，非钱包模块内部逻辑）
    ▼
[财务模块 — 风控审核 → 财务审核 → 出款审核]
    │
    ├─ 审核驳回：
    │   → [财务 → 钱包 R3 UnfreezeBalance]
    │       │── 幂等校验
    │       │── DB写（事务）──→ 解冻余额 + 更新订单状态
    │
    ├─ 审核通过 + 出款成功：
    │   → [财务 → 钱包 R4 DebitBalance]
    │       │── 幂等校验
    │       │── DB写（事务）──→ 扣减冻结金额 + 更新订单状态为已完成
    │
    └─ 审核通过 + 出款失败：
        → [财务 → 钱包 R3 UnfreezeBalance]
            │── 同审核驳回逻辑
```

**钱包模块涉及的接口**：C12 → C14(可选) → C15 → R3/R4(被财务回调)
**跨模块RPC调用数**：
- 调出：2次KYC + 2次币种配置 = 4次
- 被调入：1~2次（审核通过走R4，驳回/失败走R3）

### 6.6 充值补单全链路

## 八、接口间数据流转关系

> 关注"数据从哪来、经过哪、到哪去"

### 8.1 充值订单的生命周期数据流

```
① 创建阶段：
   C5 CreateDepositOrder → 写入 deposit_order（状态=待支付）

② 支付阶段：
   三方通道处理支付

③ 回调阶段：
   R8 DepositCallback → 更新 deposit_order（状态=成功/失败）
                      → R1逻辑 → 写入 wallet_account（余额增加）
                               → 写入 transaction_log（入账流水）

④ 查询阶段：
   C8 GetDepositOrderStatus → 读取 deposit_order
   C17 PageDepositRecords   → 读取 deposit_order
   C18 DetailDepositRecord  → 读取 deposit_order + transaction_log
```

### 8.2 提现订单的生命周期数据流

```
① 创建阶段：
   C15 CreateWithdrawOrder → 写入 wallet_account（冻结余额增加，可用减少）
                           → 写入 freeze_record（冻结记录）
                           → 写入 withdraw_order（状态=待审核）
                           → 写入 transaction_log（冻结流水）

② 审核阶段（财务模块）：
   风控→财务→出款 三级审核

③ 审核结果回调：
   驳回/出款失败：
     R3 UnfreezeBalance → 更新 wallet_account（解冻，恢复可用余额）
                        → 更新 freeze_record（状态=已解冻）
                        → 更新 withdraw_order（状态=失败）
                        → 写入 transaction_log（解冻流水）

   出款成功：
     R4 DebitBalance → 更新 wallet_account（冻结余额减少）
                     → 更新 freeze_record（状态=已扣款）
                     → 更新 withdraw_order（状态=已完成）
                     → 写入 transaction_log（扣款流水）

④ 查询阶段：
   C19 PageWithdrawRecords   → 读取 withdraw_order
   C20 DetailWithdrawRecord  → 读取 withdraw_order + transaction_log
```

### 8.3 兑换的数据流

```
① 预览阶段（不写DB）：
   C9 ExchangePreview → 读取汇率 → 计算 → 返回展示

② 执行阶段：
   C10 ExecuteExchange → 更新源币种 wallet_account（余额减少）
                       → 更新BSB wallet_account（中心钱包余额增加）
                       → 更新BSB wallet_account（奖励钱包余额增加，赠送部分）
                       → 写入 exchange_record
                       → 写入 transaction_log（扣减+入账两条流水）

③ 查询阶段：
   C21 PageExchangeRecords → 读取 exchange_record
```

|-------|---------|---------|------------|

**结论**：
- 币种配置是自己负责的模块，所有RPC自给自足，不存在外部阻塞
- 与财务模块的联调只有**2个出口**（GetPaymentMethods、MatchDepositChannel），阻塞面可控
- 钱包提供给财务的**7个RPC入口**全部可以独立开发和测试，不依赖财务模块的进度
- 真正需要等财务联调的场景只有**充值方式展示**和**充值下单通道匹配**

### 9.3 独立可测试 vs 需联调的接口分组

**完全独立（可自行开发+测试，无外部阻塞）— 25个**：
```
C1, C7, C8, C9, C10, C13, C14, C16, C17~C22（14个C端）
B1~B5（5个B端）
R1~R8（8个RPC提供，可用mock财务请求测试）
= 共27个（占全部35个的77%）
```

**需币种配置先就绪（自己负责，优先开发）— 6个**：
```

C5 CreateDepositOrder    ← 需充值金额校验
C12 CheckWithdrawEligibility ← 需提现限额
C15 CreateWithdrawOrder  ← 需提现限额
```

**需财务模块联调（真正的外部阻塞点）— 2个C端接口用到的2条RPC**：

```
C5 CreateDepositOrder   ← 需财务 MatchDepositChannel
（C11 GetWithdrawMethods 如果也走财务的 GetPaymentMethods，则+1）
```
