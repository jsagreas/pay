# 接口调用链路关系（逐接口追踪）

> 资料依据：需求理解/d-3.md + 接口评估/d-3.md + 存储会话/c-3.md（工程实证）
> 范围：钱包模块（主责）+ 币种配置模块（主责）+ 财务管理模块（他人负责，理解调用关系）
> 目的：逐接口追踪完整调用链路，明确"从哪来、经过什么、到哪去、调了谁、谁调了我"
>
> 链路标记约定：
> - `→` 调用方向（A → B 表示 A 调用 B）
> - `[内部]` 模块内部逻辑
> - `[跨模块]` 跨模块RPC调用
> - `[外部]` 调用外部三方服务
> - `[DB]` 数据库读写操作
> - 每个接口标注入口来源（gate-font / gate-back / RPC被调用）

---

# 第一部分：币种配置模块 — 接口调用链路

> 币种配置是 ser-wallet 服务内部的子模块，B端通过 gate-back 路由进入，RPC通过 Kitex 直接调用。

---

## B1 币种列表查询

**入口**：gate-back `/admin/api/wallet/currency/list` → ser-wallet RPC

**完整链路**：
```
B端管理员请求
  → gate-back 路由匹配
    → BackAuthMiddleware 鉴权（校验Token + RBAC菜单权限）
      → rpc.Handler → ser-wallet.CurrencyList()
        → [内部] 参数校验（分页参数pageNo/pageSize）
        → [DB] 查询币种表，条件：delete_at=0，支持按币种类型/状态筛选
        → [内部] 组装响应（币种编码/名称/类型/图标URL/汇率浮动%/阈值%/状态）
      ← 返回分页结果
    ← ret.HttpOne 统一响应
  ← HTTP 200 {traceId, code, data}
```

**调用了谁**：无跨模块调用，纯本地DB查询
**谁调用了我**：仅B端管理员通过gate-back

---

## B2 新增币种

**入口**：gate-back `/admin/api/wallet/currency/add` → ser-wallet RPC

**完整链路**：
```
B端管理员请求
  → gate-back 路由匹配
    → BackAuthMiddleware 鉴权
      → rpc.Handler → ser-wallet.CurrencyAdd()
        → [内部] 参数校验（编码唯一性、名称非空、类型合法、浮动%/阈值%范围）
        → [DB] 查重：相同币种编码是否已存在
        → [内部] 如果有图标上传
          → [跨模块] ser-s3.Upload() 上传图标文件（SVG/PNG≤5k）
          ← 返回图标URL
        → [DB] 插入币种记录（雪花ID、审计字段create_by/create_at）
        → [跨模块] ser-blog.AddActLog() 记录操作日志（操作类型：新增币种）
      ← 返回成功
    ← ret.HttpOne 统一响应
```

**调用了谁**：
- ser-s3 → 图标上传（条件调用，有图标时才调）
- ser-blog → 操作审计日志

**谁调用了我**：仅B端管理员

---

## B3 编辑币种

**入口**：gate-back `/admin/api/wallet/currency/edit` → ser-wallet RPC

**完整链路**：
```
B端管理员请求
  → gate-back 鉴权
    → ser-wallet.CurrencyEdit()
      → [内部] 参数校验（币种ID必填、可编辑字段：名称/类型/图标/浮动%/阈值%/状态）
      → [DB] 查询原记录（校验存在性）
      → [内部] 币种编码不可修改（需求文档3-3明确）
      → [内部] 如果有新图标上传
        → [跨模块] ser-s3.Upload() 上传新图标
      → [DB] 更新币种记录（审计字段update_by/update_at）
      → [跨模块] ser-blog.AddActLog() 记录操作日志（含变更前后对比）
    ← 返回成功
```

**调用了谁**：ser-s3（条件）、ser-blog
**谁调用了我**：仅B端管理员

---

## B4 币种状态切换

**入口**：gate-back `/admin/api/wallet/currency/status` → ser-wallet RPC

**完整链路**：
```
B端管理员请求
  → gate-back 鉴权
    → ser-wallet.CurrencyStatusSwitch()
      → [内部] 参数校验（币种ID、目标状态：启用/禁用）
      → [DB] 查询原记录
      → [内部] 状态变更校验（如：禁用某币种前是否需要检查是否有用户持有该币种余额）
      → [DB] 更新状态字段
      → [跨模块] ser-blog.AddActLog() 记录操作日志
    ← 返回成功
```

**关键影响**：币种禁用后，钱包C端调用 GetEnabledCurrencyList 将不再返回该币种，用户无法选择该币种进行操作。

**调用了谁**：ser-blog
**谁调用了我**：仅B端管理员

## B5 基准币种查询 / B6 基准币种设置

**入口**：gate-back → ser-wallet RPC

**B5 查询链路**：
```
B端管理员请求
  → gate-back 鉴权
    → ser-wallet.GetBaseCurrency()
      → [DB] 查询平台配置表中的基准币种设置
      → [内部] 未设置时返回空/默认提示
    ← 返回基准币种信息（编码、名称）或"未设置"
```

**B6 设置链路**：
```
B端管理员请求
  → gate-back 鉴权
    → ser-wallet.SetBaseCurrency()
      → [DB] 查询是否已设置过基准币种
      → [内部] 已设置 → 拒绝修改，返回错误"基准币种设置后不可更改"（需求文档3-2）
      → [内部] 未设置 → 校验币种编码有效性（是否存在且启用）
      → [DB] 写入基准币种配置
      → [跨模块] ser-blog.AddActLog()
    ← 返回成功
```

**调用了谁**：ser-blog（仅B6设置时）
**谁调用了我**：B端管理员（B5/B6）；币种配置RPC R4 GetBaseCurrency 内部复用B5的查询逻辑

---

## B7 汇率展示查询

**入口**：gate-back → ser-wallet RPC

**完整链路**：
```
B端管理员请求
  → gate-back 鉴权
    → ser-wallet.ExchangeRateList()
      → [DB] 查询所有启用币种
      → [内部] 对每个币种计算/获取：
        → 实时汇率（最近一次从三方API拉取的值）
        → 平台汇率（当前生效的平台汇率）
        → 入款汇率 = 实时汇率 × (1 + 汇率浮动%)
        → 出款汇率 = 实时汇率 × (1 - 汇率浮动%)
        → 偏差值 = |实时 - 平台| / 实时 × 100
      → [内部] 组装列表
    ← 返回所有币种的汇率一览表
```

**调用了谁**：无跨模块调用，纯内部计算 + DB查询
**谁调用了我**：仅B端管理员

---

## B8 汇率日志查询

**入口**：gate-back → ser-wallet RPC

**完整链路**：
```
B端管理员请求
  → gate-back 鉴权
    → ser-wallet.ExchangeRateLogList()
      → [DB] 查询汇率日志表，条件：按币种/时间范围筛选，分页
      → [内部] 日志记录包含：变更时间、币种、变更前汇率、变更后汇率、触发偏差值、阈值
    ← 返回分页日志列表
```

**数据来源**：汇率日志由定时任务（C3）在发现偏差超阈值并更新平台汇率时自动写入，不由管理员手动产生。

**调用了谁**：无
**谁调用了我**：仅B端管理员

## B9 操作日志查询

**入口**：gate-back → ser-wallet RPC

**完整链路**：
```
B端管理员请求
  → gate-back 鉴权

```

**调用了谁**：可能调用ser-blog查询（取决于日志存储方案）
**谁调用了我**：仅B端管理员

## C3 三方汇率定时拉取（定时任务）

**入口**：定时任务触发（XXL-JOB 或 robfig/cron）

**完整链路**：
```
定时任务触发（建议频率：1-5分钟）
  → ser-wallet.SyncExchangeRates()
    → [DB] 查询所有启用的非平台币种（法币 + 加密货币，排除BSB）
    → 对每个币种:
      → [外部] 调用三方汇率API-A 获取汇率
      → [外部] 调用三方汇率API-B 获取汇率
      → [外部] 调用三方汇率API-C 获取汇率
      → [内部] 取三家平均值 → 得到实时汇率
      → [DB] 更新币种的实时汇率记录
      → [内部] 计算偏差值 = |实时汇率 - 当前平台汇率| / 实时汇率 × 100
      → [内部] 偏差值 >= 该币种阈值?
        → 是：
          → [DB] 更新平台汇率 = 当前实时汇率
          → [DB] 写入汇率日志（变更前、变更后、偏差值、阈值）
        → 否：
          → 不更新平台汇率，跳过
    ← 任务完成
```

**调用了谁**：三方汇率API（HTTP外部调用，3家取均值）
**谁调用了我**：定时任务调度器
**产出物**：更新实时汇率 + 条件更新平台汇率 + 条件写入汇率日志

**入口**：被其他服务通过 Kitex RPC 调用


## R3 GetExchangeRate（RPC暴露）

**入口**：被其他服务通过 Kitex RPC 调用

**完整链路**：
```
调用方（钱包兑换/充值/提现、财务统计、游戏场馆）
  → ser-wallet.GetExchangeRate(currencyCode, rateType)
    → [DB] 查询该币种的实时汇率和平台汇率
    → [DB] 查询该币种的汇率浮动%
    → [内部] 按rateType计算:
      rateType=平台   → 返回平台汇率
      rateType=入款   → 返回 实时汇率 × (1 + 浮动%)
      rateType=出款   → 返回 实时汇率 × (1 - 浮动%)
      rateType=实时   → 返回实时汇率
  ← 返回指定类型的汇率值
```

**调用了谁**：无
**谁调用了我**：
- 钱包F4发起充值（BSB充值时展示汇率，用入款汇率）
- 钱包F11发起提现（BSB提现时展示汇率，用出款汇率）
- 钱包F15兑换预览（法币→BSB汇率计算，用入款汇率）
- 钱包F16发起兑换（实际汇率计算）
- 财务统计报表（换算为基准币种，用平台汇率）
- 游戏场馆（进入时用入款汇率，退出时用进入时快照）

# 第二部分：多币种钱包模块 — C端接口调用链路

> C端接口从 gate-font 进入，经 FontAuthMiddleware 认证后转发到 ser-wallet RPC。

---

## F1 获取钱包余额信息

**入口**：gate-font `/api/wallet/balance` → ser-wallet RPC

**完整链路**：
```
C端用户请求（个人中心页加载时）
  → gate-font 路由匹配
    → FontAuthMiddleware 认证（校验Token，从Redis获取用户信息）
      → rpc.Handler → ser-wallet.GetWalletBalance()
        → [内部] 从ctx提取userId
        → [内部] 从请求参数获取当前币种编码（currencyCode）
        → [DB] 查询该用户+该币种下的钱包记录:
          → 中心钱包余额
          → 奖励钱包余额
          → 冻结金额（提现审核中的冻结）
        → [内部] 计算:
          → 资金钱包余额 = 中心钱包余额 + 奖励钱包余额（需求文档2-1）
          → 可提现余额 = 中心钱包余额 - 冻结金额（需求文档7-5）
        → [内部] 按币种类型格式化金额（VND/IDR整数用.千分位，THB/BSB两位小数用,千分位）
      ← 返回余额信息
    ← ret.HttpOne 统一响应
```

**调用了谁**：无跨模块调用，纯本地DB查询 + 内部计算
**谁调用了我**：C端用户（个人中心页、每次币种切换后）

---

## F2 获取用户可用币种列表

**入口**：gate-font `/api/wallet/currency/list` → ser-wallet RPC

**完整链路**：
```
C端用户请求（币种切换时）
  → gate-font FontAuth认证
    → ser-wallet.GetUserCurrencyList()
      → [内部] 调用本模块币种配置的能力（同一服务内部调用，非跨服务RPC）
        → 等效于 R2 GetEnabledCurrencyList 的逻辑
        → [DB] 查询启用状态的币种列表
      → [内部] 组装返回（编码、名称、图标URL、类型）
    ← 返回可用币种列表
```

**调用了谁**：本模块内部（币种配置子模块），不需要跨服务RPC因为在同一个ser-wallet中
**谁调用了我**：C端用户

**关键说明**：因为币种配置和钱包在同一个 ser-wallet 服务中，F2获取币种列表不需要走RPC网络调用，直接在进程内调用币种配置的查询逻辑即可。

## F3 获取充值方式列表

**入口**：gate-font `/api/wallet/deposit/methods` → ser-wallet RPC

**完整链路**：
```
C端用户请求（打开充值页面时）
  → gate-font FontAuth认证
    → ser-wallet.GetDepositMethods()
      → [内部] 从ctx提取userId，从参数获取currencyCode
      → [跨模块] 财务服务.GetDepositMethods(currencyCode)
        → 财务内部：查询支付配置表，按币种过滤启用的充值方式
        → 财务内部：关联代收通道信息（图标、金额范围、手续费、排序）
        ← 返回充值方式列表
      → [内部] 按排序字段排序
      → [内部] 组装C端展示数据（方式名称、图标、金额范围、手续费说明）
    ← 返回充值方式列表
```

**调用了谁**：
- **财务服务 GetDepositMethods** — 充值方式的数据源在财务的支付配置中（财务需求文档2-2）

**谁调用了我**：C端用户

---

## F4 发起充值

**入口**：gate-font `/api/wallet/deposit/create` → ser-wallet RPC

**完整链路**：
```
C端用户请求（确认充值提交时）
  → gate-font FontAuth认证
    → ser-wallet.CreateDeposit()
      → [内部] 从ctx提取userId
      → [内部] 参数校验：currencyCode、充值方式编码、金额、（可选）奖金活动ID

      → [内部] 币种校验
        → [DB] 查询该币种是否启用（本模块内部，币种配置逻辑）

      → [内部] 重复转账检测（需求文档5-5）
        → [DB] 查询该用户+该币种+该金额的近期充值记录数
        → 1-2笔：返回警告标记（由前端弹窗确认）
        → ≥3笔：直接拒绝，返回错误"重复转账已拦截"

      → [内部] 如果选择了奖金活动
        → [跨模块] 活动服务.CheckBonusEligibility(activityId, amount)
          → 校验充值金额是否满足最低门槛
          ← 返回是否满足 + 奖金比例

      → [内部] BSB充值时获取汇率
        → [DB] 查询入款汇率（本模块内部，币种配置逻辑，等效R3）

      → [跨模块] 财务服务.CreateDepositOrder(userId, currencyCode, amount, payMethod, ...)
        → 财务内部：通道轮询分配（配比权重 → 选择代收通道）
        → 财务内部：创建平台订单（状态：待支付）
        → 财务内部：调用三方通道创建订单
          → [外部] 三方通道API → 返回支付URL 或 USDT接收地址
        ← 返回：平台订单号 + 支付URL/USDT地址/二维码

      → [DB] 记录充值订单（本地记录，关联财务平台订单号）
      → [内部] 如果选了奖金活动，记录关联关系（活动ID+充值订单号+奖金比例）
    ← 返回支付信息（支付URL / USDT地址+二维码 / 链类型）
```

**调用了谁**：
- **财务服务 CreateDepositOrder** — 核心调用，通道分配+订单创建+三方对接（财务需求文档2-3充值流程）
- **活动服务 CheckBonusEligibility** — 条件调用，选了奖金活动时校验（需求文档5-6）
- 本模块币种配置 — 币种校验+汇率（进程内调用）

**谁调用了我**：C端用户

## F11 发起提现

**入口**：gate-font `/api/wallet/withdraw/create` → ser-wallet RPC

**完整链路**：
```
C端用户请求（确认提现提交时）
  → gate-font FontAuth认证
    → ser-wallet.CreateWithdraw()
      → [内部] 从ctx提取userId
      → [内部] 参数校验：currencyCode、提现方式、账户ID、提现金额

      → [内部] KYC校验（需求文档7-1）
        → [跨模块] ser-kyc.GetKycStatus(userId)
          ← 返回KYC状态（未认证/已认证）
        → [内部] 未认证 + 非USDT提现方式 → 拒绝，返回"法币提现需先完成KYC认证"

      → [内部] 提现方式校验（需求文档7-1提现规则）
        → [DB] 查询用户充值来源
        → [内部] 验证提现方式是否符合充值来源规则
          （法币充→法币提，USDT充→USDT提，混合充→法币提）

      → [内部] 限额校验（需求文档7-5）
        → [DB] 查询提现限额配置（单笔最低/最高、单日限额）
        → [DB] 查询今日已提现金额
        → [内部] 校验：金额 >= 单笔最低 AND 金额 <= 单笔最高 AND 今日累计+本次 <= 单日限额

      → [内部] 余额校验
        → [DB] 查询中心钱包可提现余额 = 余额 - 冻结金额
        → [内部] 可提现余额 >= 提现金额 + 手续费?

      → [内部] 手续费计算
        → [DB] 查询手续费配置（固定金额 或 比例）
        → [内部] 计算实际手续费

      → [内部] BSB提现时汇率换算
        → [DB] 获取出款汇率（本模块币种配置逻辑，等效R3）

      → [DB] 冻结余额（中心钱包可用余额减少，冻结金额增加）
      → [DB] 记录冻结流水

      → [跨模块] 财务服务.CreateWithdrawOrder(userId, currencyCode, amount, fee, accountInfo, ...)
        → 财务内部：创建提现订单（状态：审核中）
        → 财务内部：进入三级审核队列
        ← 返回平台订单号

      → [DB] 记录提现订单（本地记录，关联财务平台订单号、冻结记录ID）
      → [跨模块] ser-blog.AddActLog() 记录关键操作日志
    ← 返回提现订单信息（订单号、预计到账金额、手续费）
```

**调用了谁**：
- **ser-kyc GetKycStatus** — KYC状态校验（需求文档7-1）
- **财务服务 CreateWithdrawOrder** — 创建提现订单进入审核（财务需求文档2-4）
- **ser-blog AddActLog** — 审计日志
- 本模块币种配置 — 出款汇率（进程内调用）

**谁调用了我**：C端用户

**关键说明**：此接口的链路最长、逻辑最复杂，涉及KYC校验→提现规则校验→限额校验→余额校验→手续费计算→冻结余额→调用财务创建订单，共7个关键步骤。

## R1 WalletCredit（通用入账）

**入口**：被财务服务/活动服务等通过 Kitex RPC 调用

**完整链路**：
```
调用方（财务充值回调 / 财务补单审核通过 / 活动服务发奖等）
  → ser-wallet.WalletCredit(userId, currencyCode, walletType, amount, bizType, orderNo)
    → [内部] 参数校验（userId、金额>0、币种有效、钱包类型有效）
    → [内部] 幂等校验（需求文档9"充值回调幂等"）
      → [DB] 查询流水表：是否已存在相同orderNo的入账记录
      → 已存在 → 返回成功（幂等，不重复入账）
    → [DB] 【事务开始】
      → 查询并锁定用户钱包记录（SELECT ... FOR UPDATE）
      → 增加指定钱包类型的余额（walletType对应的余额字段 += amount）
      → 插入资金流水记录（类型：入账，方向：收入，业务类型：bizType，关联订单号：orderNo）
    → [DB] 【事务提交】
  ← 返回成功 + 入账后余额
```

**调用了谁**：无跨模块调用
**谁调用了我**：
- 财务服务 → 充值成功入账（bizType=充值，walletType=中心钱包）
- 财务服务 → 补单入账（bizType=补单，walletType=中心钱包）
- 活动服务 → 通过 R16 BonusCredit 间接（或直接调WalletCredit，bizType=活动奖励，walletType=奖励钱包）

## R2 WalletDebit（通用扣款）

**入口**：被财务服务等通过 Kitex RPC 调用

**完整链路**：
```

  → ser-wallet.WalletDebit(userId, currencyCode, walletType, amount, bizType, orderNo)
    → [内部] 参数校验
    → [内部] 幂等校验：同orderNo是否已扣款

    → [DB] 【事务提交】
  ← 返回成功 + 扣款后余额
```

**调用了谁**：无
**谁调用了我**：
- 财务服务 → 提现出款成功确认扣款（此时实际是扣除冻结金额，见R3/R4说明）

## R10 BetDebit（投注扣款）

**完整链路**：
```
调用方（游戏/投注服务）
  → ser-wallet.BetDebit(userId, currencyCode, amount, orderNo)
    → [内部] 参数校验 + 幂等校验
    → [DB] 【事务开始】
      → 查询并锁定用户中心钱包 + 奖励钱包
      → [内部] 优先级扣款逻辑（需求文档2-1资金钱包体系）：
        → 中心钱包可扣 = min(中心钱包余额, amount)
        → 剩余 = amount - 中心钱包可扣
        → 奖励钱包可扣 = min(奖励钱包余额, 剩余)
        → 如果 中心钱包可扣 + 奖励钱包可扣 < amount → 余额不足，回滚
      → 扣除中心钱包（中心钱包可扣金额）
      → 扣除奖励钱包（奖励钱包可扣金额）
      → 插入流水记录（记录中心/奖励各扣了多少 → 用于返奖时按比例分配）
      → [内部] 记录扣款比例：
        中心占比 = 中心钱包可扣 / amount
        奖励占比 = 奖励钱包可扣 / amount
    → [DB] 【事务提交】
  ← 返回成功 + 扣款比例信息（供返奖时使用）
```

**谁调用了我**：游戏/投注服务

## R11 BetCredit（投注返奖）

# 第四部分：跨模块完整链路（财务调用钱包的场景）

> 以下链路从财务服务的动作开始，追踪到钱包服务的响应。
> 虽然财务模块不是你负责，但你需要清楚"别人怎么调你的接口"。

---

## 链路A：充值成功 → 入账

```
三方支付通道回调
  → 财务服务接收回调（HTTP回调接口）
    → [财务内部] 校验回调签名、校验金额
    → [财务内部] 更新充值订单状态 → 充值成功
    → [财务内部] 调用钱包入账
      → [跨模块] ser-wallet.WalletCredit(
          userId     = 订单中的用户ID,
          currency   = 订单币种,
          walletType = 中心钱包,
          amount     = 到账金额（扣除通道手续费后）,
          bizType    = "deposit",
          orderNo    = 平台充值订单号
        )
        → 钱包执行入账（见R1链路）
      ← 入账成功
    → [财务内部] 如果该充值关联了奖金活动
      → [跨模块] ser-wallet.BonusCredit(
          userId     = 同上,
          currency   = 同上,
          amount     = 奖金金额,
          bizOrderNo = 充值订单号,
          auditMultiplier = X倍流水要求
        )
        → 钱包执行奖励入账（见R16链路）
      ← 奖励入账成功
  ← 财务返回回调应答给三方通道
```

---

## 链路B：提现下单 → 冻结

```
C端用户提交提现（经由钱包F11）
  → 钱包F11完成本地校验后
    → [跨模块] 财务服务.CreateWithdrawOrder(...)
      → [财务内部] 创建提现订单（状态：审核中）
      → [财务内部] 需要冻结用户余额
        → [跨模块] ser-wallet.WalletFreeze(
            userId   = 用户ID,
            currency = 币种,
            amount   = 提现金额 + 手续费,
            orderNo  = 提现订单号
          )
          → 钱包执行冻结（见R3链路）
        ← 冻结成功
      ← 返回提现订单号
    → 钱包F11记录本地提现订单
  ← 返回给C端用户
```

**注意**：这里存在一个待确认点——冻结是在钱包F11内部先做（钱包先冻结再调财务建单），还是在财务CreateWithdrawOrder内部做（财务建单时回调钱包冻结）。两种方案的链路不同，需与财务开发者对齐。上述链路描述的是"财务建单时调用钱包冻结"的方案。

---

## 链路C：提现审核驳回 → 解冻

```
B端审核人员操作驳回（在财务后台）
  → 财务服务.RejectWithdrawOrder(orderNo, rejectReason)
    → [财务内部] 更新提现订单状态 → 已驳回
    → [财务内部] 需要解冻用户余额
      → [跨模块] ser-wallet.WalletUnfreeze(
          userId   = 订单中的用户ID,
          currency = 订单币种,
          amount   = 冻结金额,
          orderNo  = 提现订单号
        )
        → 钱包执行解冻（见R4链路）
      ← 解冻成功
  ← 驳回完成
```

---

## 链路D：提现出款成功 → 确认扣款

```
三方代付通道回调出款成功
  → 财务服务接收出款回调
    → [财务内部] 校验回调
    → [财务内部] 更新提现订单状态 → 已完成
    → [财务内部] 需要确认从用户钱包扣款（将冻结金额正式扣除）
      → [跨模块] ser-wallet.WalletDebit(
          userId     = 用户ID,
          currency   = 币种,
          walletType = 中心钱包,
          amount     = 冻结金额,
          bizType    = "withdraw",
          orderNo    = 提现订单号
        )
        → 钱包执行扣款：将冻结金额转为正式扣除（余额不变，冻结金额减少）
      ← 扣款成功
  ← 返回回调应答
```

**说明**：提现扣款的实际效果是"冻结金额消失"而不是"可用余额减少"，因为可用余额在冻结时就已经减少了。

## 链路E：人工加款 → 钱包入账

```
B端管理员发起人工加款（财务后台）
  → 财务服务.CreateManualCredit(targetUserId, currency, walletType, amount, remark)
    → [财务内部] 创建人工加款单（状态：待审核）
    → [财务内部] 查看目标用户钱包信息
      → [跨模块] ser-wallet.GetWalletInfo(targetUserId)
        → 钱包返回用户各钱包余额（见R8链路）
    ← 展示给管理员确认

另一个管理员审核通过
  → 财务服务.ApproveManualCredit(orderNo)
    → [财务内部] 校验审核人≠提交人（需求文档2-5-2）
    → [财务内部] 更新加款单状态 → 已通过
    → [财务内部] 调用钱包执行加款
      → [跨模块] ser-wallet.ManualCredit(
          userId     = 目标用户ID,
          currency   = 币种,
          walletType = 指定的钱包类型（中心/奖励/主播/代理），
          amount     = 加款金额,
          orderNo    = 加款单号（A+16位随机数）,
          remark     = 加款备注
        )
        → 钱包执行加款（见R5链路）
      ← 加款成功
  ← 审核完成
```

---

## 链路F：人工减款 → 钱包扣款

```
结构与链路E对称，差异点：
  - 调用的是 ser-wallet.ManualDebit（R6）
  - 钱包执行减款（余额减少，插入流水）
  - 订单号前缀为M（M+16位随机数）
```

## 链路G：充值补单 → 钱包入账

```
B端管理员新增补单 → 另一管理员审核通过
  → 财务服务.ApproveDepositSupplement(orderNo)
    → [财务内部] 更新补单状态 → 已通过
    → [财务内部] 调用钱包入账
      → [跨模块] ser-wallet.WalletCredit(
          userId     = 目标用户ID,
          currency   = 币种,
          walletType = 中心钱包,
          amount     = 补单金额,
          bizType    = "supplement",
          orderNo    = 补单号（B+16位随机数）
        )
        → 钱包执行入账（见R1链路）
      ← 入账成功
  ← 审核完成
```
