# 接口调用链路关系分析

> 来源：需求理解/d-2.md、接口评估/d-2.md、依赖关系/d-2.md 综合分析
> 分析目标：逐接口梳理模块内部调用链 + 跨模块调用链，清楚每个接口"内部做了什么、调了谁、被谁调"
> 分析范围：我方负责的钱包模块（ser-wallet）+ 币种配置模块，以及与财务模块（他人负责）的交互链路
> 标注约定：
>   [内部] = 模块内部操作（DB读写、本地逻辑）
>   [→ 币种配置] = 调用币种配置模块的RPC
>   [→ 财务] = 调用财务模块的RPC
>   [→ ser-xxx] = 调用其他微服务
>   [← 财务] = 被财务模块调用

---

# 第一部分：钱包模块（ser-wallet）接口调用链路

## 一、C端接口调用链路

> 以下每个接口从"用户发起HTTP请求"到"返回响应"的完整调用路径
> 路由层：gate-font → ser-wallet handler → 内部逻辑 + 外部RPC

---

### C-1. 获取钱包信息

> 需求依据：原型 2-1~2-3个人中心.png，需求 4-1钱包总体设计.png
> 场景：用户进入个人中心/钱包页面，展示当前币种的余额和可用币种列表

**调用链路**：

```
用户请求(当前币种/用户ID)
  │
  ├── [→ 币种配置] GetCurrencyList
  │     获取所有启用状态的币种（代码/名称/图标/展示规则）
  │     用途：返回币种切换下拉列表数据
  │
  ├── [内部] 查询钱包表
  │     根据用户ID + 当前币种，查询各子钱包余额
  │     ├── 中心钱包余额
  │     ├── 奖励钱包余额
  │     ├── 冻结金额（提现中的冻结）
  │     └── 资金钱包余额 = 中心 + 奖励（计算字段）
  │
  ├── [→ 币种配置] GetCurrencyDetail
  │     获取当前币种的千分位符/小数位/小数点符
  │     用途：前端按规则格式化金额显示
  │
  └── 返回响应
        ├── 当前币种信息
        ├── 资金余额（格式化后）
        ├── 中心钱包余额 + 奖励钱包余额
        └── 可用币种列表
```

**内部操作**：DB读取（钱包余额表）
**外部调用**：币种配置 × 2（GetCurrencyList + GetCurrencyDetail）
**被谁调用**：无（C端用户直接触发）

### C-2. 切换币种

> 需求依据：需求 4-2币种切换规则.png
> 场景：用户在钱包页面切换当前操作币种

**调用链路**：

```
用户请求(目标币种代码)
  │
  ├── [内部] 校验目标币种是否为启用状态
  │     可查本地缓存的币种列表，或调币种配置校验
  │
  ├── [内部] 更新用户当前币种偏好
  │     写入用户设置表或Redis缓存
  │
  └── 返回响应
        └── 切换成功 + 新币种信息
```

**内部操作**：DB/Redis写入（用户币种偏好）
**外部调用**：无（或币种校验可用缓存）
**被谁调用**：无
**备注**：这是最轻量的接口，纯内部操作，切换后前端重新请求钱包信息接口刷新数据

### C-3. 获取充值方式列表

> 需求依据：需求 5-1充值方式总览.png，原型 5-1~5-4切换充值方式.png
> 场景：用户进入充值页面，展示当前币种下可用的充值方式

**调用链路**：

```
用户请求(当前币种)
  │
  ├── [→ 财务] GetRechargeMethodList(币种)
  │     获取当前币种下所有启用的充值方式
  │     返回数据：
  │     ├── 方式名称（如"银行转账""USDT-TRC20"）
  │     ├── 支付图标URL
  │     ├── 充值档位列表（如 100/500/1000/5000）
  │     ├── 金额范围（最低~最高）
  │     └── 排序序号
  │     来源：财务需求 2-2支付配置-1充值方式.png
  │
  ├── [内部] 游客/登录状态判断
  │     游客：过滤仅保留USDT充值方式
  │     登录用户：返回全部充值方式
  │     来源：需求 5-1充值方式总览.png "游客仅可用USDT充值"
  │
  └── 返回响应
        └── 充值方式列表（已过滤+已排序）
```

**内部操作**：用户状态判断 + 列表过滤
**外部调用**：财务 × 1（GetRechargeMethodList）
**被谁调用**：无
**关键依赖**：**强依赖财务**。财务未配置充值方式 → 此接口返回空列表 → 用户无法充值

### C-4. 创建充值订单

> 需求依据：需求 5-7页面交互.png，财务需求 2-3充值流程.png
> 场景：用户选好充值方式+金额+奖金后提交订单

**调用链路**：

```
用户请求(币种/充值方式/金额/奖金活动ID)
  │
  ├── [内部] 参数校验
  │     ├── 金额是否在该充值方式的范围内
  │     ├── 金额是否匹配档位（如果是快捷档位）
  │     └── 用户状态校验（封禁等）
  │
  ├── [内部] 重复转账检查（可内联或前端先调C-14独立接口）
  │     查近期订单：同用户+同币种+同金额
  │     ├── 命中1-2笔 → 返回警告标记（前端弹窗确认）
  │     └── 命中≥3笔 → 直接拦截，返回错误
  │     来源：需求 5-5重复转账拦截规则.png
  │
  ├── [→ 币种配置] GetExchangeRate(USDT→BSB, 入款汇率)  ← 仅BSB充值时
  │     获取入款汇率，计算用户实际到账的BSB数量
  │     来源：需求 5-2usdt充值.png "仅BSB充值界面显示汇率换算"
  │
  ├── [内部] 奖金校验（如果用户选择了活动）
  │     ├── 校验充值金额是否达到活动门槛
  │     ├── 计算奖金金额
  │     └── 来源：需求 5-6额外奖金选择规则.png
  │
  ├── [→ 财务] MatchChannel(币种/充值方式/金额)
  │     请求财务模块轮询匹配最优代收通道
  │     ├── 匹配成功 → 返回通道信息 + 三方支付URL/充值地址
  │     └── 匹配失败 → 返回错误"系统繁忙，请更换充值方式"
  │     来源：财务需求 2-3充值流程.png + 2-1通道轮询规则.png
  │
  ├── [内部] 创建平台充值订单
  │     ├── 生成平台订单号
  │     ├── 订单状态 = 待支付
  │     ├── 记录：用户ID/币种/金额/充值方式/通道信息/奖金信息/入款汇率
  │     └── 写入充值订单表
  │
  ├── [→ ser-blog] AddActLog(oneway)
  │     异步记录操作日志
  │
  └── 返回响应
        ├── 平台订单号
        ├── 三方支付URL（银行/扫码类）或 充值地址（USDT类）
        └── 订单金额/到账金额预估
```

**内部操作**：参数校验 + 重复检查 + 奖金校验 + 创建订单（DB写入）
**外部调用**：币种配置 × 0~1（仅BSB） + 财务 × 1（MatchChannel） + ser-blog × 1
**被谁调用**：无
**关键依赖**：**强依赖财务的MatchChannel**。通道不可用则充值必定失败。这是钱包C端最核心的跨模块调用点。

### C-5. 充值回调

> 需求依据：需求 7-1~7-4通用回调.png，财务需求 2-3充值流程.png
> 场景：三方支付通道完成支付后回调平台

**调用链路**：

```
三方通道回调请求(通道订单号/金额/支付状态/签名)
  │
  ├── [内部] 验证签名
  │     使用通道配置的密钥验签，防伪造回调
  │
  ├── [内部] 查询平台订单
  │     通过通道订单号关联查出平台充值订单
  │
  ├── [内部] 幂等检查
  │     ├── 订单已处理（状态≠待支付）→ 直接返回成功，不重复上账
  │     └── 订单未处理 → 继续
  │     来源：需求 9非功能性需求.png "充值回调必须幂等"
  │
  ├── [内部] 验证金额
  │     回调金额与订单金额对比，允许一定偏差或严格匹配
  │
  ├── [内部] 事务操作（必须原子性）
  │     ├── CreditWallet(用户ID, 币种, 中心钱包, 充值金额)
  │     │     中心钱包余额 += 充值金额
  │     │
  │     ├── CreditWallet(用户ID, 币种, 奖励钱包, 奖金金额)  ← 仅有活动奖金时
  │     │     奖励钱包余额 += 奖金金额
  │     │     来源：币种配置需求 2-1资金钱包体系.png "奖励钱包接收活动赠送"
  │     │
  │     ├── 写入资金流水记录
  │     │     流水类型=充值，关联订单号，变动前余额→变动后余额
  │     │     来源：需求 9非功能性需求.png "账务流水与余额强一致"
  │     │
  │     └── 更新订单状态 = 支付成功
  │
  ├── [→ ser-blog] AddActLog(oneway)
  │
  └── 返回响应给三方通道
        └── success（通知三方停止重试）
```

**内部操作**：验签 + 幂等检查 + 验金额 + 事务（上账+写流水+更新订单）
**外部调用**：ser-blog × 1
**被谁调用**：三方支付通道（HTTP回调）
**关键点**：这是资金安全的关键接口。CreditWallet在此场景是**模块内部调用**（同进程内的函数调用），不是RPC。上账、写流水、更新状态必须在同一数据库事务中完成。

### C-6. 获取提现方式列表

> 需求依据：需求 7-1提现模块.png，原型 10-1~10-6提现页面.png
> 场景：用户进入提现页面，展示当前币种下可用的提现方式

**调用链路**：

```
用户请求(当前币种)
  │
  ├── [→ ser-kyc] GetKycStatus(用户ID)
  │     查询用户KYC认证状态
  │     来源：需求 7-1提现模块.png "未认证弹窗，继续提现仅USDT"
  │
  ├── [→ 财务] GetWithdrawMethodList(币种)
  │     获取当前币种下所有启用的提现方式
  │     返回数据：
  │     ├── 方式名称（如"银行卡""GoPay""PromptPay"）
  │     ├── 支付图标URL
  │     ├── 提现费率（%）
  │     ├── 单笔限额（最低/最高）
  │     └── 排序序号
  │     来源：财务需求 2-2支付配置-2提现方式.png
  │
  ├── [内部] 根据KYC状态过滤
  │     ├── 未认证 → 仅保留USDT提现方式
  │     └── 已认证 → 返回全部提现方式
  │
  ├── [→ 币种配置] GetExchangeRate(BSB→法币/USDT, 出款汇率)  ← 仅BSB币种时
  │     获取出款汇率用于页面展示换算
  │     来源：需求 7-4bsb提现.png "BSB提现时展示汇率换算"
  │
  └── 返回响应
        ├── KYC状态
        ├── 提现方式列表（已过滤+已排序）
        └── 出款汇率（BSB时）
```

**内部操作**：KYC状态判断 + 列表过滤
**外部调用**：ser-kyc × 1 + 财务 × 1 + 币种配置 × 0~1（BSB时）
**关键依赖**：强依赖财务的GetWithdrawMethodList

### C-7. 创建提现订单

> 需求依据：需求 7-6页面交互.png，财务需求 2-4提现流程.png
> 场景：用户选好提现方式+金额+账户后提交提现

**调用链路**：

```
用户请求(币种/提现方式/金额/提现账户ID)
  │
  ├── [内部] 参数校验
  │     ├── 金额范围校验（单笔最低/最高）
  │     ├── 每日提现次数校验
  │     ├── 每日提现金额上限校验
  │     └── 来源：需求 7-5提现限制.png
  │
  ├── [→ ser-kyc] GetKycStatus(用户ID)
  │     再次校验KYC（防止绕过前端直接调接口）
  │     ├── 未认证 + 非USDT方式 → 拒绝
  │     └── 已认证 → 继续
  │
  ├── [内部] 查询提现账户信息
  │     ├── 校验提现账户是否属于当前用户
  │     ├── 校验持有人姓名 = KYC认证姓名
  │     └── 来源：需求 7-2银行卡提现.png "持有人姓名=KYC姓名不可编辑"
  │
  ├── [内部] 查询可用余额
  │     当前币种中心钱包 + 奖励钱包的可用余额（排除已冻结部分）
  │     可用余额 < 提现金额 → 拒绝
  │
  ├── [→ 币种配置] GetExchangeRate(出款汇率)  ← 仅BSB提现时
  │     计算出款金额（BSB → 实际出款法币/USDT）
  │
  ├── [内部] 计算手续费
  │     手续费 = 提现金额 × 费率（费率来自提现方式配置）
  │     实际到账 = 提现金额 - 手续费
  │
  ├── [内部] 冻结余额 — FreezeBalance
  │     事务内操作：
  │     ├── 可用余额 -= 提现金额
  │     ├── 冻结金额 += 提现金额
  │     └── 消费扣款优先级：先扣中心钱包，不足部分扣奖励钱包
  │         来源：币种配置需求 2-1资金钱包体系.png "消费优先扣中心钱包"
  │
  ├── [内部] 创建提现订单
  │     ├── 生成平台订单号
  │     ├── 订单状态 = 待风控审核
  │     ├── 记录：用户ID/币种/金额/提现方式/账户信息/出款汇率/手续费
  │     └── 写入提现订单表
  │
  ├── [→ ser-blog] AddActLog(oneway)
  │
  └── 返回响应
        ├── 订单号
        ├── 提现金额/手续费/预计到账金额
        └── 订单状态（待审核）
```

**内部操作**：参数校验 + KYC二次校验 + 账户校验 + 余额校验 + 冻结 + 创建订单
**外部调用**：ser-kyc × 1 + 币种配置 × 0~1（BSB时） + ser-blog × 1
**被谁调用**：无
**关键点**：冻结余额是模块**内部操作**，不是RPC调用。此时FreezeBalance作为内部函数在同一事务中执行。后续的审核流程转入财务模块，与钱包C端无关。

### C-8. 获取兑换信息

> 需求依据：需求 6兑换模块.png，原型 8兑换BSB.png
> 场景：用户进入兑换页面，展示法币→BSB的汇率和相关信息

**调用链路**：

```
用户请求(当前法币币种)
  │
  ├── [内部] 校验当前币种是否为法币
  │     BSB没有兑换入口，仅法币可兑换
  │     来源：需求 6兑换模块.png "仅法币钱包有兑换入口"
  │
  ├── [→ 币种配置] GetExchangeRate(当前法币→BSB)
  │     获取兑换汇率（1 BSB = X 法币）
  │     来源：需求 6兑换模块.png "汇率统一口径"
  │
  ├── [内部] 查询当前法币中心钱包可用余额
  │     用户最多可兑换的金额
  │
  ├── [内部] 获取赠送比例配置
  │     如赠送10%，来自活动/运营配置
  │     来源：需求 6兑换模块.png "赠送奖励如10%"
  │
  └── 返回响应
        ├── 兑换汇率（1 BSB = X 法币）
        ├── 当前法币可用余额
        └── 赠送比例 + 赠送规则说明
```

**内部操作**：币种校验 + 余额查询 + 赠送配置读取
**外部调用**：币种配置 × 1（GetExchangeRate）
**特点**：不依赖财务模块

### C-9. 执行兑换

> 需求依据：需求 6兑换模块.png
> 场景：用户确认兑换，法币→BSB

**调用链路**：

```
用户请求(法币币种/兑换金额)
  │
  ├── [内部] 参数校验
  │     ├── 当前币种是否为法币
  │     └── 兑换金额是否在范围内
  │
  ├── [→ 币种配置] GetExchangeRate(法币→BSB)
  │     获取实时兑换汇率（再次获取，防止页面停留期间汇率变动）
  │
  ├── [内部] 计算兑换结果
  │     ├── BSB主金额 = 法币金额 / 汇率
  │     ├── BSB赠送金额 = BSB主金额 × 赠送比例
  │     └── 标记赠送部分的稽核流水要求（X倍流水后可提现）
  │
  ├── [内部] 事务操作（原子性，跨两个币种钱包）
  │     ├── DebitWallet(用户ID, 法币, 中心钱包, 法币金额)
  │     │     法币中心钱包余额 -= 兑换金额
  │     │
  │     ├── CreditWallet(用户ID, BSB, 中心钱包, BSB主金额)
  │     │     BSB中心钱包余额 += BSB主金额
  │     │
  │     ├── CreditWallet(用户ID, BSB, 奖励钱包, BSB赠送金额)
  │     │     BSB奖励钱包余额 += BSB赠送金额
  │     │     来源：需求 6兑换模块.png "赠送部分进入奖励钱包"
  │     │
  │     ├── 写入资金流水 × 3（法币扣款 + BSB主入账 + BSB赠送入账）
  │     │
  │     └── 创建兑换记录
  │           记录：法币币种/金额/BSB数量/赠送数量/汇率/时间
  │
  ├── [→ ser-blog] AddActLog(oneway)
  │
```

## 二、钱包 RPC 接口调用链路（供他方调用）

> 以下接口由其他服务通过Kitex RPC调用，不经过HTTP网关
> 这些接口是钱包模块的**核心能力输出**

---

### RPC-1. CreditWallet（上账/加款）

> 调用方：财务（充值成功/补单通过/人工加款通过）
> 需求依据：财务需求 2-3充值流程.png、2-5人工修正.png

**调用链路**：

```
RPC请求(用户ID/币种/钱包类型/金额/业务类型/关联订单号)
  │
  ├── [内部] 参数校验
  │     ├── 金额 > 0
  │     ├── 币种有效
  │     └── 钱包类型有效（中心/奖励/主播/代理）
  │
  ├── [内部] 事务操作
  │     ├── 查询当前余额（加锁 SELECT FOR UPDATE）
  │     ├── 余额 += 金额
  │     ├── 写入资金流水（流水类型=业务类型，变动前→变动后）
  │     └── 更新余额表
  │
  └── 返回响应
        └── 操作成功 + 变动后余额
```

**内部操作**：事务（加锁查余额 + 更新余额 + 写流水）
**外部调用**：无
**特点**：**完全自治**，不依赖任何外部服务。这是资金安全的核心操作，必须保证事务原子性和流水一致性。

---

### RPC-2. DebitWallet（扣款/减款）

> 调用方：财务（人工减款通过）、游戏（投注扣款）

**调用链路**：

```
RPC请求(用户ID/币种/钱包类型/金额/业务类型/关联订单号)
  │
  ├── [内部] 参数校验
  │
  ├── [内部] 事务操作
  │     ├── 查询当前余额（加锁）
  │     ├── 余额 < 金额 → 返回余额不足错误
  │     ├── 余额 -= 金额
  │     ├── 写入资金流水
  │     └── 更新余额表
  │
  └── 返回响应
        └── 操作成功 + 变动后余额
```

**内部操作**：事务（加锁查余额 + 余额检查 + 扣减 + 写流水）
**外部调用**：无
**特点**：与CreditWallet对称，**完全自治**

---

### RPC-3. GetBalance（余额查询）

> 调用方：财务（人工修正页面）、游戏、风控

**调用链路**：

```
RPC请求(用户ID/币种)
  │
  ├── [内部] 查询余额表
  │     返回该用户该币种下所有钱包类型的余额：
  │     ├── 中心钱包余额
  │     ├── 奖励钱包余额
  │     ├── 主播钱包余额
  │     ├── 代理钱包余额
  │     ├── 冻结金额
  │     └── 资金钱包余额（中心+奖励）
  │
  └── 返回响应
        └── 各钱包余额明细
```

**内部操作**：DB查询
**外部调用**：无
**特点**：纯只读，**完全自治**

---

### RPC-4. FreezeBalance（冻结）

> 调用方：提现流程（创建提现订单时）
> 注意：在C-7创建提现订单中是内部调用；如果提现由财务侧发起则是RPC调用

**调用链路**：

```
RPC请求(用户ID/币种/金额/关联订单号)
  │
  ├── [内部] 事务操作
  │     ├── 查询可用余额（加锁）
  │     ├── 可用余额 < 金额 → 返回余额不足
  │     ├── 按扣款优先级冻结：先冻中心钱包，不足部分冻奖励钱包
  │     │     来源：币种配置需求 2-1 "消费优先扣中心钱包"
  │     ├── 可用余额 -= 金额
  │     ├── 冻结金额 += 金额
  │     ├── 写入冻结流水
  │     └── 更新余额表
  │
  └── 返回响应
        └── 冻结成功 + 冻结记录ID
```

**内部操作**：事务（余额检查 + 冻结操作 + 写流水）
**外部调用**：无
**特点**：**完全自治**。冻结后余额可见但不可消费。

---

### RPC-5. UnfreezeBalance（解冻）

> 调用方：财务（提现驳回/出款失败时）

**调用链路**：

```
RPC请求(用户ID/币种/金额/关联订单号)
  │
  ├── [内部] 事务操作
  │     ├── 查询冻结记录（加锁）
  │     ├── 冻结金额 -= 金额
  │     ├── 可用余额 += 金额（恢复到冻结前的钱包）
  │     ├── 写入解冻流水
  │     └── 更新余额表
  │
  └── 返回响应
        └── 解冻成功
```

**内部操作**：事务（解冻 + 恢复余额 + 写流水）
**外部调用**：无
**特点**：与FreezeBalance配对，**完全自治**

---

### RPC-6. ConfirmDebit（确认扣款）

> 调用方：财务（出款成功后）

**调用链路**：

```
RPC请求(用户ID/币种/金额/关联订单号)
  │
  ├── [内部] 事务操作
  │     ├── 查询冻结记录（加锁）
  │     ├── 冻结金额 -= 金额（冻结金额正式清除）
  │     ├── 不恢复可用余额（钱已经出去了）
  │     ├── 写入确认扣款流水
  │     └── 更新余额表
  │
  └── 返回响应
        └── 确认成功
```

**内部操作**：事务（清除冻结 + 写流水）
**外部调用**：无
**特点**：与UnfreezeBalance互斥——同一笔冻结只能走ConfirmDebit或UnfreezeBalance之一。**完全自治**。

---

### RPC-7. InitWallet（钱包初始化）

> 调用方：ser-user（用户注册时）

```
  │     ├── 奖励钱包（余额=0）

  │     └── 代理钱包（余额=0，预留）
  │
  └── 返回响应
        └── 初始化成功
```

**内部操作**：批量DB写入
**外部调用**：币种配置 × 1（GetCurrencyList）
**特点**：唯一有外部依赖的RPC接口

## 三、C端接口之间的内部调用关系

> 有些C端接口会被其他C端接口的逻辑内部复用（函数级别调用，非RPC）

```
C-4 创建充值订单
  ├── 内部复用 C-14 重复转账检查 的逻辑
  │    （可作为独立函数被C-4直接调用，也可前端先调C-14再调C-4）
  └── 内部复用 RPC-1 CreditWallet 的逻辑 ← 不在C-4中，在C-5回调中

C-5 充值回调
  ├── 内部调用 RPC-1 CreditWallet 函数（同进程内，非RPC）
  │    中心钱包上账
  └── 内部调用 RPC-1 CreditWallet 函数
       奖励钱包上账（如有活动奖金）

C-7 创建提现订单
  └── 内部调用 RPC-4 FreezeBalance 函数（同进程内，非RPC）
       冻结提现金额

C-9 执行兑换
  ├── 内部调用 RPC-2 DebitWallet 函数
  │    扣法币中心钱包
  ├── 内部调用 RPC-1 CreditWallet 函数
  │    加BSB中心钱包
  └── 内部调用 RPC-1 CreditWallet 函数
       加BSB奖励钱包（赠送部分）
```

**说明**：CreditWallet / DebitWallet / FreezeBalance 既是对外暴露的RPC接口，也是模块内部可复用的核心函数。当C端接口需要操作余额时，直接在同进程内调用这些函数（不走网络RPC），保证事务性。

### B-7. 手动刷新汇率

> 需求依据：需求 4-1汇率应用场景.png

**调用链路**：

```
B端请求(币种代码 或 全部)
  │
  ├── [→ 三方汇率API] HTTP请求 × 3家
  │     并发调用3家汇率API
  │     来源：需求 4-2汇率规则.png "从3家三方汇率API获取，取均值"
  │
  ├── [内部] 计算均值
  │     实时汇率 = (API-1 + API-2 + API-3) / 3
  │
  ├── [内部] 偏差检测
  │     偏差值 = |新实时汇率 - 当前平台汇率| / 新实时汇率 × 100%
  │     ├── 偏差值 < 偏值阈值 → 仅更新实时汇率，不动平台汇率
  │     └── 偏差值 ≥ 偏值阈值 → 更新平台汇率 + 重算入款/出款汇率
  │         来源：需求 4-2汇率规则.png "偏差值≥阈值时更新平台汇率"
  │
  ├── [内部] 写入汇率日志
  │     记录：触发原因=手动刷新/操作人/变动前→变动后/时间
  │
  ├── [→ ser-blog] AddActLog(oneway)
  │
  └── 返回响应
        └── 刷新结果（更新了哪些币种的汇率）
```

**外部调用**：三方API × 3 + ser-blog × 1
**备注**：此逻辑与定时任务的汇率拉取逻辑相同，可复用同一函数

```

  │
  ├── 通过 → 状态变为"待出款"
  └── 驳回 →
       ├── [财务 → 钱包] UnfreezeBalance(同上)

  │
  └── 三方出款结果回调

  │     └── 状态变为"出款成功"
  │
  └── 失败 →
        ├── [财务 → 钱包] UnfreezeBalance(同上)
```

## 十、调用链路核心结论

### 10.1 钱包模块内部的"核心函数复用"

CreditWallet、DebitWallet、FreezeBalance、UnfreezeBalance、ConfirmDebit 这5个函数是钱包模块的资产操作原语。它们既作为RPC接口对外暴露，也被C端接口在进程内直接调用：

```
RPC对外暴露（被财务/游戏等调用）：
  财务 ──RPC──→ CreditWallet / DebitWallet / FreezeBalance / ...

C端接口内部复用（同进程函数调用）：
  C-5 充值回调 ──函数调用──→ CreditWallet
  C-7 创建提现 ──函数调用──→ FreezeBalance
  C-9 执行兑换 ──函数调用──→ DebitWallet + CreditWallet
```

**设计要点**：这些函数的实现必须同时支持两种调用场景。作为RPC被调用时，事务由调用方管理（或函数自身开事务）；作为内部函数被调用时，需要能加入外层事务。

### 10.2 与财务模块的交互接口清单

**我方调用财务（3个RPC接口）**：
1. GetRechargeMethodList — 获取充值方式配置
2. GetWithdrawMethodList — 获取提现方式配置
3. MatchChannel — 充值时匹配代收通道

**财务调用我方（6个RPC接口）**：
1. CreditWallet — 上账（充值/补单/加款）
2. DebitWallet — 扣款（减款）
