# ser-wallet 接口调用链路关系

> 信息来源：ser-wallet/tmp/ 下的需求文档和产品原型
> 分析范围：钱包模块（主责）+ 币种配置（主责）的每个接口调用链路，以及与财务模块的跨模块调用链路
> 分析日期：2026-02-26

---

## 一、钱包模块 C端接口调用链路

> 以下每个接口描述：谁调用它 → 它内部做了什么 → 它调用了谁 → 返回什么
> 标记说明：【本地】= 操作钱包自己的数据库，【RPC→X】= 调用X模块的RPC接口

---

### C1：获取币种列表及余额

**触发场景**：用户在个人中心点击币种切换 icon，进入币种选择页面

**调用链路**：

```
C端请求 → gate-font(/api) → ser-wallet.GetCurrencyListWithBalance
    │
    ├──【RPC→币种配置】GetCurrencyList
    │       获取所有启用的币种信息（代码/名称/类型/国家/图标/状态）
    │
    ├──【本地】查询该用户在每个币种下的余额
    │       SELECT currency, SUM(balance) FROM wallet
    │       WHERE user_id = ? GROUP BY currency
    │
    └── 合并：币种信息 + 对应余额 → 返回列表
```

**返回数据**：币种代码、名称、图标、余额、是否当前选中

**内部无状态变更**，纯读取操作。

---

### C2：查询当前币种钱包余额

**触发场景**：用户进入钱包首页、充值页、提现页时展示当前币种余额

**调用链路**：

```
C端请求 → gate-font(/api) → ser-wallet.GetBalance
    │
    └──【本地】查询该用户指定币种下各钱包类型余额
            SELECT wallet_type, available_balance, frozen_balance
            FROM wallet WHERE user_id = ? AND currency = ?

            组装返回：
            ├── 中心钱包余额
            ├── 奖励钱包余额
            ├── 主播钱包余额（如有）
            └── 资金钱包余额 = 中心 + 奖励
```

**纯本地操作**，不调用任何外部服务。

---

### C3：获取充值方式列表

**触发场景**：用户进入充值页面时，展示当前币种可用的充值方式、快捷金额、汇率

**调用链路**：

```
C端请求 → gate-font(/api) → ser-wallet.GetDepositMethods
    │
    ├──【RPC→财务】GetDepositMethods(currency)
    │       获取该币种下配置的充值方式列表
    │       返回：方式名称、图标、支付类型、快捷金额档位、金额范围、排序、推荐标记
    │
    ├──【RPC→币种配置】GetExchangeRate(currency → BSB)
    │       获取当前入款汇率（充值页底部展示"1BSB=0.1USDT"用）
    │       仅当币种为BSB或涉及跨币种展示时才需要
    │
    └── 合并：充值方式列表 + 汇率信息 → 返回
```

**关键点**：这个接口是两个外部RPC调用的聚合。如果财务模块的充值方式查询接口不可用，这个接口整体失败。

### C4：创建充值订单

**触发场景**：用户选择充值方式和金额后点击"去充值"

**调用链路**：

```
C端请求(金额+方式+奖金活动ID) → gate-font(/api) → ser-wallet.CreateDepositOrder
    │
    ├── 1.【本地】参数校验
    │       ├── 金额 >= 该方式最低限额
    │       ├── 币种是否启用
    │       └── 如选了奖金活动：金额 >= 活动最低充值门槛
    │
    ├── 2.【本地】重复充值检查（可能合并在此处）
    │       查询：同用户 + 同币种 + 同金额 的待支付订单数量
    │       ├── 1-2笔 → 允许继续（前端已弹窗提醒过）
    │       └── >=3笔 → 拒绝创建，返回错误
    │
    ├── 3.【RPC→币种配置】GetExchangeRate(currency)
    │       获取入款汇率，计算实际支付金额
    │       如BSB充值：支付金额 = 充值金额 × 入款汇率
    │
    ├── 4.【本地】生成充值订单
    │       ├── 订单号：C + 16位随机数（Snowflake或随机）
    │       ├── 状态：待支付
    │       ├── 记录：用户ID、币种、金额、方式、汇率快照、奖金活动信息
    │       └── INSERT INTO deposit_order ...
    │
    ├── 5.【RPC→财务】MatchChannelAndCreatePayment(币种, 方式, 金额)
    │       财务执行：
    │       ├── 加权轮询算法选择代收通道
    │       ├── 用通道密钥向三方发起支付请求
    │       └── 返回：支付页URL/收款信息/通道订单号
    │
    ├── 6.【本地】更新订单
    │       UPDATE deposit_order SET channel_order_id = ?, payment_info = ?
    │
    ├── 7.【RPC→BLog】AddActLog（异步审计）
    │
    └── 返回：支付页信息（跳转URL / USDT地址 / 二维码数据）
```

**关键点**：
- 步骤 4 和步骤 5 构成紧耦合——创建了本地订单后必须成功调用财务匹配通道，否则订单变成"死单"
- 汇率在步骤 3 获取后应作为快照保存到订单中，不依赖后续汇率变动
- 步骤 5 如果失败，需要将步骤 4 创建的订单标记为失败或删除

### C5：USDT支付页信息

**触发场景**：充值方式为USDT时，展示平台承载的支付页面

**调用链路**：

```
C端请求(订单ID) → gate-font(/api 或 /open) → ser-wallet.GetUsdtPaymentInfo
    │
    ├──【本地】查询充值订单
    │       SELECT * FROM deposit_order WHERE order_id = ?
    │       校验：订单存在且状态为待支付
    │
    └──【本地】获取USDT收款信息
            ├── 收款地址（平台统一地址 或 按订单分配的地址）
            ├── 网络类型（默认TRC20，可选ERC20/BEP20）
            └── 生成二维码数据
```

**纯本地操作**。USDT收款地址可能来自配置或地址池，不涉及外部RPC。

### C6：获取提现方式列表

**触发场景**：用户点击"提现"后（如已通过KYC判断），进入提现方式选择页

**调用链路**：

```
C端请求 → gate-font(/api) → ser-wallet.GetWithdrawalMethods
    │
    ├──【RPC→KYC服务】GetKycStatus(userId)
    │       获取用户KYC认证状态
    │       ├── 未认证 → 过滤：只保留USDT提现方式
    │       └── 已认证 → 不过滤：全渠道可用
    │
    ├──【RPC→财务】GetWithdrawalMethods(currency)
    │       获取该币种下配置的提现方式列表
    │       返回：方式名称、图标、金额范围、限额规则(单笔/单日)、手续费率、到账时间、排序
    │
    ├──【RPC→币种配置】GetExchangeRate(currency)
    │       如当前币种为BSB，获取出款汇率（BSB提现页需展示）
    │
    └── 按KYC状态过滤方式列表 + 附加汇率 → 返回
```

**关键点**：这个接口聚合了三个外部RPC（KYC + 财务 + 币种配置）。其中KYC决定了"能用哪些方式"，财务提供了"有哪些方式"，币种配置提供了"汇率是多少"。

### C7：创建提现订单

**触发场景**：用户填完提现信息和金额后点击"确认提现"

**调用链路**：

```
C端请求(金额+方式+账户信息引用) → gate-font(/api) → ser-wallet.CreateWithdrawalOrder
    │
    ├── 1.【RPC→KYC服务】GetKycStatus(userId)
    │       二次确认KYC状态（防止C6到C7之间状态变化）
    │       未认证且方式非USDT → 拒绝
    │
    ├── 2.【本地】校验提现账户
    │       查询已保存的提现账户信息，确认存在且完整
    │
    ├── 3.【本地+RPC】校验提现限额
    │       ├── 单笔限额：金额 >= 最低 且 <= 最高（方式配置中获取，可能已缓存自C6）
    │       ├── 单日限额：查询今日已提现总额 + 本次 <= 单日上限
    │       └── 如超限 → 返回具体提示（"超出每日提现限额"/"低于单笔最低"）
    │
    ├── 4.【本地】校验余额
    │       查询可用余额（非冻结部分）>= 提现金额
    │
    ├── 5.【RPC→币种配置】GetExchangeRate(currency)
    │       如BSB提现：用出款汇率计算实际出款金额
    │
    ├── 6. 计算手续费和到账金额
    │       手续费 = 提现金额 × 手续费率
    │       到账金额 = 提现金额 - 手续费
    │
    ├── 7.【本地 — DB事务】创建订单 + 冻结余额（必须在同一事务内）
    │       BEGIN TRANSACTION
    │       ├── UPDATE wallet SET available -= amount, frozen += amount
    │       │       WHERE user_id = ? AND currency = ? AND available >= amount
    │       │       （乐观锁：affected_rows == 0 则余额不足）
    │       ├── INSERT INTO withdrawal_order
    │       │       订单号：T + 16位随机数
    │       │       状态：待风控审核
    │       │       记录：用户ID/币种/金额/手续费/到账金额/汇率快照/方式/账户引用
    │       └── INSERT INTO transaction_log（流水记录：冻结）
    │       COMMIT
    │
    ├── 8.【RPC→BLog】AddActLog（异步审计）
    │
    └── 返回：订单信息 + "提现进行中"状态
```

**关键点**：
- 步骤 7 是整个提现流程中最关键的操作——冻结余额和创建订单必须在同一个数据库事务内
- 如果先创建订单再冻结，存在时间窗口：订单已创建但余额未冻结，此时如果用户并发操作可能超额
- 如果先冻结再创建订单，冻结成功但创建订单失败则需要回滚冻结
- 同一事务是最安全的方案

### C8：保存提现账户信息

**触发场景**：用户首次提现时填写银行卡/电子钱包账户信息

**调用链路**：

```
C端请求(银行名称+账号+持有人+开户网点/手机号...) → gate-font(/api) → ser-wallet.SaveWithdrawalAccount
    │
    ├──【RPC→KYC服务】GetKycInfo(userId)
    │       获取KYC认证姓名
    │
    ├──【本地】姓名一致性校验
    │       将用户提交的"账户持有人姓名"与KYC姓名对比
    │       规则：忽略大小写 + 忽略多余空格
    │       不一致 → 拒绝保存
    │
    ├──【本地】保存账户信息
    │       INSERT INTO withdrawal_account
    │       字段：用户ID/币种/方式类型/银行名称/账号/持有人/开户网点/手机号...
    │
    └── 返回：保存成功
```

### C10：获取兑换汇率

**触发场景**：用户进入兑换页面时，展示当前法币→BSB的汇率和赠送比例

**调用链路**：

```
C端请求 → gate-font(/api) → ser-wallet.GetExchangeRateForSwap
    │
    ├──【RPC→币种配置】GetExchangeRate(当前法币 → BSB)
    │       获取汇率，用于展示"1 BSB = X 法币"
    │
    ├──【本地或配置】获取赠送比例
    │       如10%赠送（数据源待评审确认：可能来自活动配置或固定配置）
    │
    └── 返回：汇率 + 赠送比例
```

### C11：执行兑换

**触发场景**：用户在兑换页面输入金额后点击"兑换"

**调用链路**：

```
C端请求(法币金额) → gate-font(/api) → ser-wallet.ExecuteExchange
    │
    ├── 1.【RPC→币种配置】GetExchangeRate(当前法币 → BSB)
    │       取最新汇率（不用C10缓存的，防止汇率已变）
    │
    ├── 2.【本地】校验余额
    │       当前法币可用余额 >= 兑换金额
    │       超出 → 返回"已达当前可兑换上限"
    │
    ├── 3. 计算兑换结果
    │       兑换获得BSB = 法币金额 ÷ 汇率
    │       赠送BSB = 兑换获得BSB × 赠送比例
    │       实际到账BSB = 兑换获得 + 赠送
    │
    ├── 4.【本地 — DB事务】双币种余额变更
    │       BEGIN TRANSACTION
    │       ├── UPDATE wallet SET available -= 法币金额
    │       │       WHERE user_id=? AND currency=当前法币 AND wallet_type=中心钱包
    │       ├── UPDATE wallet SET available += 兑换获得BSB
    │       │       WHERE user_id=? AND currency=BSB AND wallet_type=中心钱包
    │       ├── UPDATE wallet SET available += 赠送BSB
    │       │       WHERE user_id=? AND currency=BSB AND wallet_type=奖励钱包
    │       │       （赠送部分进奖励钱包，需完成流水后才能提现）
    │       ├── INSERT INTO exchange_record（兑换记录）
    │       └── INSERT INTO transaction_log × 3（法币扣除+BSB入账+赠送入账）
    │       COMMIT
    │
    ├── 5.【RPC→BLog】AddActLog（异步审计）
    │
    └── 返回：兑换成功 + 兑换获得/赠送/实际到账
```

**关键点**：
- 步骤 4 涉及两个币种三条余额变更（扣法币、加BSB到中心、加赠送BSB到奖励），必须在同一事务内
- 赠送部分进**奖励钱包**而非中心钱包，这是需求文档明确的规则
- 汇率在步骤 1 实时获取，不用兑换预览时的缓存值，防止汇率变化导致计算偏差

### C12-C13：充值记录列表 / 提现记录列表

**调用链路**：

```
C端请求(分页参数) → gate-font(/api) → ser-wallet.GetDepositRecords / GetWithdrawalRecords
    │
    └──【本地】分页查询
            SELECT * FROM deposit_order / withdrawal_order
            WHERE user_id = ? AND currency = 当前币种
            ORDER BY create_time DESC
            LIMIT ? OFFSET ?

            返回：金额/方式/时间/状态/操作入口
```

**纯本地操作**，无外部依赖。记录跟随当前选中币种，切换币种即切换数据。

### C14：钱包概览/聚合信息

```
C端请求 → gate-font(/api) → ser-wallet.GetWalletOverview
    │
```

可能与C2合并为一个接口，区别在于是否需要币种的展示信息。

### C19：提现限额查询

**调用链路**：

```
C端请求(币种+方式) → gate-font(/api) → ser-wallet.GetWithdrawalLimits
    │
    ├──【RPC→财务 或 缓存】获取该方式的限额配置
    │       单笔最低/最高、单日限额、手续费率
    │       （如C6已调过，可从缓存获取）
    │
    ├──【本地】查询今日已提现总额
    │       SELECT SUM(amount) FROM withdrawal_order
    │       WHERE user_id = ? AND currency = ? AND DATE(create_time) = TODAY
    │       AND status NOT IN (风控驳回, 财务驳回, 出款失败)
    │
    └── 计算并返回：单笔限额/单日限额/今日已用/今日剩余/手续费规则
```

### C21：重复充值检查

**调用链路**：

```
C端请求(金额) → gate-font(/api) → ser-wallet.CheckDuplicateDeposit
    │
    └──【本地】查询待支付订单数量
            SELECT COUNT(*) FROM deposit_order
            WHERE user_id = ? AND currency = ? AND amount = ?
            AND status IN (待支付, 进行中)

            ├── 0笔 → 正常
            ├── 1-2笔 → 返回提醒（含订单数量）
            └── >=3笔 → 返回拦截（禁止继续）
```

**纯本地操作**。此接口可能合并到C4（创建充值订单）内部作为前置校验。


## 二、钱包模块 RPC对外接口调用链路

> 这些接口定义在 ser-wallet 的 IDL 中，由其他服务（主要是财务模块）通过 RPC 调用。
> 每个接口都涉及余额变更，是资金安全的核心。

---

### R1：GetUserBalance（查询用户余额）

**调用方**：财务模块（提现审核时查看用户余额、减款前查余额）

**调用链路**：

```
财务模块 → RPC → ser-wallet.GetUserBalance(userId, currency, walletType?)
    │
    └──【本地】查询余额
            SELECT wallet_type, available_balance, frozen_balance
            FROM wallet WHERE user_id = ? AND currency = ?
            （如指定walletType则精确查，否则返回所有钱包类型）
```

**纯读取**，无副作用。

---

### R2：CreditDeposit（充值入账）

**调用方**：财务模块（三方通道回调验证通过后）

**完整跨模块链路**：

```
三方通道回调 → 财务.充值回调Webhook
    │
    ├── 财务：验签 + 校验金额匹配 + 更新财务记录
    │
    └── 财务 → RPC → ser-wallet.CreditDeposit(userId, currency, amount, orderId)
            │
            ├── 1.【本地】幂等检查
            │       查询该 orderId 是否已入账
            │       已入账 → 直接返回成功（幂等）
            │
            ├── 2.【本地 — DB事务】入账
            │       BEGIN TRANSACTION
            │       ├── UPDATE wallet SET available += amount
            │       │       WHERE user_id=? AND currency=? AND wallet_type=中心钱包
            │       ├── UPDATE deposit_order SET status = 支付成功
            │       │       WHERE order_id = ?
            │       └── INSERT INTO transaction_log（流水：充值入账）
            │       COMMIT
            │
            ├── 3.【RPC→BLog】AddActLog（异步审计）
            │
            └── 返回：入账成功
```

**关键点**：幂等性——通道可能重复回调，必须通过 orderId 去重，重复调用返回成功而不重复入账。

---

### R3：FreezeBalance（冻结余额）

**调用方**：钱包内部（C7创建提现订单时）/ 未来投注模块

**调用链路**：

```
调用方 → ser-wallet.FreezeBalance(userId, currency, walletType, amount, reason)
    │
    └──【本地 — DB事务】
            UPDATE wallet
            SET available -= amount, frozen += amount
            WHERE user_id=? AND currency=? AND wallet_type=?
            AND available >= amount  ← 乐观锁，防止超额冻结

            affected_rows == 0 → 余额不足，拒绝冻结
            affected_rows == 1 → 冻结成功
```

通常被C7内部事务包含，不作为独立RPC对外暴露。但如果投注模块需要冻结余额（锁定投注金额），则需要独立暴露。

---

### R4：UnfreezeBalance（解冻退回）

**调用方**：财务模块（提现审核驳回后）

**完整跨模块链路**：

```
B端管理员审核驳回 → 财务.提现审核(orderId, 驳回, 原因)
    │
    ├── 财务：更新财务记录状态 → 风控驳回/财务驳回
    │
    └── 财务 → RPC → ser-wallet.UnfreezeBalance(userId, currency, amount, orderId)
            │
            ├── 1.【本地】幂等检查
            │       该 orderId 是否已解冻过
            │
            ├── 2.【本地 — DB事务】解冻
            │       BEGIN TRANSACTION
            │       ├── UPDATE wallet SET frozen -= amount, available += amount
            │       │       WHERE user_id=? AND currency=? AND wallet_type=中心钱包
            │       ├── UPDATE withdrawal_order SET status = 驳回
            │       │       WHERE order_id = ?
            │       └── INSERT INTO transaction_log（流水：提现驳回解冻）
            │       COMMIT
            │
            ├── 3.【RPC→BLog】AddActLog
            │
            └── 返回：解冻成功
```

---

### R5：ConfirmWithdrawal（确认提现扣款）

**调用方**：财务模块（提现审核通过、出款成功后）

**完整跨模块链路**：

```
B端管理员审核通过 → 财务.提现审核(orderId, 通过)
    │
    ├── 财务：更新财务记录 → 发起代付（调三方通道出款）
    │
    └── 财务 → RPC → ser-wallet.ConfirmWithdrawal(userId, currency, amount, orderId)
            │
            ├── 1.【本地】幂等检查
            │
            ├── 2.【本地 — DB事务】确认扣款
            │       BEGIN TRANSACTION
            │       ├── UPDATE wallet SET frozen -= amount
            │       │       WHERE user_id=? AND currency=? AND wallet_type=中心钱包
            │       │       （冻结金额减少，可用余额不变——钱真正"出去"了）
            │       ├── UPDATE withdrawal_order SET status = 出款成功
            │       │       WHERE order_id = ?
            │       └── INSERT INTO transaction_log（流水：提现扣款）
            │       COMMIT
            │
            ├── 3.【RPC→BLog】AddActLog
            │
            └── 返回：扣款成功
```

**关键区别**：R4（解冻）是 frozen→available（钱退给用户），R5（扣款）是 frozen→消失（钱真的走了）。

---

### R6：ManualCredit（人工加款）

**调用方**：财务模块（人工加款审核通过后）

**完整跨模块链路**：

```
B端管理员创建加款 → 审核通过 → 财务.加款审核(orderId, 通过)
    │
    └── 财务 → RPC → ser-wallet.ManualCredit(userId, currency, walletType, amount, orderId, reason)
            │
            ├── 1.【本地】幂等检查（按orderId去重）
            │
            ├── 2.【本地 — DB事务】加款
            │       BEGIN TRANSACTION
            │       ├── UPDATE wallet SET available += amount
            │       │       WHERE user_id=? AND currency=? AND wallet_type=?
            │       │       注意：walletType可能是中心/奖励/主播/代理中的任一个
            │       └── INSERT INTO transaction_log（流水：人工加款，记录原因类型）
            │       COMMIT
            │
            ├── 3.【RPC→BLog】AddActLog
            │
            └── 返回：加款成功
```

**关键点**：需求原型中加款支持同时给多个钱包类型加款。如果一次审核涉及多个钱包，财务可能发起多次RPC调用（每个钱包类型一次），或钱包提供批量接口。

---

### R7：ManualDebit（人工减款）

**调用方**：财务模块（人工减款审核通过后）

**调用链路**：

```
财务 → RPC → ser-wallet.ManualDebit(userId, currency, walletType, amount, orderId, reason)
    │
    ├── 1.【本地】幂等检查
    │
    ├── 2.【本地】校验余额
    │       该钱包类型可用余额 >= 减款金额
    │       不足 → 返回错误（需求原型："减款金额不超过余额"）
    │
    ├── 3.【本地 — DB事务】减款
    │       BEGIN TRANSACTION
    │       ├── UPDATE wallet SET available -= amount
    │       │       WHERE user_id=? AND currency=? AND wallet_type=?
    │       │       AND available >= amount
    │       └── INSERT INTO transaction_log（流水：人工减款，记录原因类型）
    │       COMMIT
    │
    ├── 4.【RPC→BLog】AddActLog
    │
    └── 返回：减款成功
```

---

### R8：SupplementCredit（补单入账）

**调用方**：财务模块（充值补单审核通过后）

**完整跨模块链路**：

```
B端管理员查到原充值订单 → 创建补单(金额/赠送/稽核倍数) → 审核通过
    │
    └── 财务 → RPC → ser-wallet.SupplementCredit(userId, currency, amount, bonusAmount, originalOrderId, supplementOrderId)
            │
            ├── 1.【本地】幂等检查（按supplementOrderId去重）
            │
            ├── 2.【本地 — DB事务】补单入账
            │       BEGIN TRANSACTION
            │       ├── UPDATE wallet SET available += amount
            │       │       WHERE user_id=? AND currency=? AND wallet_type=中心钱包
            │       ├── 如有赠送金额：
            │       │   UPDATE wallet SET available += bonusAmount
            │       │       WHERE user_id=? AND currency=? AND wallet_type=奖励钱包
            │       └── INSERT INTO transaction_log × N（流水）
            │       COMMIT
            │
            ├── 3.【RPC→BLog】AddActLog
            │
            └── 返回：补单入账成功
```

## 三、币种配置模块接口调用链路

> 币种配置是我的主责模块之一。以下梳理其每个接口的调用链路。

---

### B1：币种列表查询（B端）

```
B端请求(筛选条件) → gate-back(/admin/api) → ser-wallet.GetCurrencyList
    │
    ├──【本地】查询币种表
    │       SELECT * FROM currency_config
    │       WHERE (code=? OR 1=1) AND (type=? OR 1=1) AND (status=? OR 1=1)
    │
    ├──【本地】计算衍生字段
    │       对每条记录：
    │       ├── 入款汇率 = 平台汇率 × (1 + 汇率浮动%)
    │       └── 出款汇率 = 平台汇率 × (1 - 汇率浮动%)
    │
    └── 返回：20个字段的完整列表
```

---

### B2：币种编辑（B端）

```
B端请求(币种代码, 汇率浮动, 阈值, 图标文件, 状态) → gate-back(/admin/api) → ser-wallet.EditCurrency
    │
    ├── 1.【本地】校验
    │       ├── 汇率浮动：2位小数，单位%
    │       ├── 阈值：2位小数，单位%
    │       ├── 如要禁用：校验不是基准币种（基准无法禁用）
    │       └── 图标：SVG/PNG，不超过5KB
    │
    ├── 2.【RPC→S3服务】Upload（如有图标文件）
    │       上传图标，获取URL
    │
    ├── 3.【本地】更新币种配置
    │       UPDATE currency_config
    │       SET rate_floating=?, threshold=?, icon=?, status=?
    │       WHERE code = ?
    │
    ├── 4.【本地】重新计算入款/出款汇率
    │       入款汇率 = 平台汇率 × (1 + 新浮动%)
    │       出款汇率 = 平台汇率 × (1 - 新浮动%)
    │
    ├── 5.【RPC→BLog】AddActLog
    │
    └── 返回：编辑成功
```

---

### B3：基准币种设置（B端）

```
B端请求(基准币种代码) → gate-back(/admin/api) → ser-wallet.SetBaseCurrency
    │
    ├── 1.【本地】校验
    │       ├── 基准币种是否已设置过 → 已设置则拒绝（不可更改）
    │       └── 选择的币种代码是否存在
    │
    ├── 2.【本地】写入基准币种配置
    │       INSERT/UPDATE system_config SET base_currency = ?
    │
    ├── 3.【RPC→BLog】AddActLog
    │
    └── 返回：设置成功（后续该接口不再可用）
```

---

### B4：汇率日志列表（B端）

```
B端请求(时间范围, 币种代码) → gate-back(/admin/api) → ser-wallet.GetRateChangeLogs
    │
    └──【本地】查询汇率日志表
            SELECT * FROM rate_change_log
            WHERE (update_time BETWEEN ? AND ?) AND (currency_code = ? OR 1=1)
            ORDER BY update_time DESC
            返回：序号/日志编号/更新时间/币种代码/实时汇率/平台汇率/入款汇率/出款汇率
```

---

### B5：实时汇率手动刷新（B端）

```
B端请求 → gate-back(/admin/api) → ser-wallet.RefreshRealtimeRate
    │
    ├──【HTTP→三方API】调用3家汇率API
    │       API_1 → rate_1
    │       API_2 → rate_2
    │       API_3 → rate_3
    │       实时汇率 = (rate_1 + rate_2 + rate_3) / 3
    │
    ├──【本地】更新实时汇率缓存/DB
    │       UPDATE currency_config SET realtime_rate = ?
    │
    └── 返回：最新实时汇率值
```

注意：手动刷新只更新实时汇率的展示值，**不自动触发平台汇率更新**。平台汇率更新由定时任务根据阈值判断。

---

### CR1：GetExchangeRate（RPC对外）

**调用方**：钱包模块（充值/提现/兑换时）、财务模块（统计换算时）

```
调用方 → RPC → ser-wallet.GetExchangeRate(fromCurrency, toCurrency)
    │
    └──【本地】查询汇率数据
            SELECT platform_rate, rate_floating FROM currency_config
            WHERE code = ?

            计算返回：
            ├── 平台汇率
            ├── 入款汇率 = 平台汇率 × (1 + 汇率浮动%)
            └── 出款汇率 = 平台汇率 × (1 - 汇率浮动%)
```

**高频接口**，被钱包的充值/提现/兑换和财务的统计多个场景调用。应考虑缓存。

---

### CR2：GetCurrencyList（RPC对外）

**调用方**：钱包模块（展示可用币种）、财务模块（记录中展示币种信息）

```
调用方 → RPC → ser-wallet.GetCurrencyList(statusFilter?)
    │
    └──【本地】查询启用的币种列表
            SELECT code, name, type, country, icon, symbol,
                   thousand_separator, decimal_separator, status
            FROM currency_config
            WHERE status = 启用（或不筛选）
```

---

### 汇率定时更新任务

**触发方式**：定时任务（cron / XXL-Job）

```
定时触发 → ser-wallet.ScheduledRateUpdate
    │
    ├── 遍历所有启用的币种（BSB除外，BSB汇率写死）
    │       │
    │       ├──【HTTP→三方API】获取该币种实时汇率（3家取平均）
    │       │
    │       ├──【本地】计算偏差值
    │       │       当前入款汇率 = 平台汇率 × (1 + 浮动%)
    │       │       当前出款汇率 = 平台汇率 × (1 - 浮动%)
    │       │       入款偏差 = |(实时汇率 - 当前入款汇率) / 实时汇率| × 100%
    │       │       出款偏差 = |(实时汇率 - 当前出款汇率) / 实时汇率| × 100%
    │       │
    │       ├── 判断：入款偏差 >= 阈值% 或 出款偏差 >= 阈值%
    │       │       │
    │       │       ├── 未超阈值 → 跳过该币种，不更新
    │       │       │
    │       │       └── 超过阈值 →
    │       │               │
    │       │               ├──【本地】更新平台汇率
    │       │               │       UPDATE currency_config
    │       │               │       SET platform_rate = 实时汇率, realtime_rate = 实时汇率
    │       │               │       （新的入款/出款汇率随之变化）
    │       │               │
    │       │               └──【本地】生成汇率日志
    │       │                       INSERT INTO rate_change_log
    │       │                       (日志编号, 更新时间, 币种代码, 实时汇率, 平台汇率, 入款汇率, 出款汇率)
    │       │
    │       └── 继续下一个币种
    │
    └──【RPC→BLog】AddActLog（记录本次定时任务执行）
```

## 四、跨模块完整链路（端到端）

> 将用户操作从头到尾走一遍，看数据在三个模块之间怎么流转。

---

### 4.1 一笔充值的完整生命周期

```
阶段        操作                          模块            方向              说明
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

前置        运营配置代收通道                财务B端          —               费率/限额/密钥
前置        运营配置充值方式                财务B端          —               关联通道/档位/状态
前置        运营配置币种和汇率              币种配置B端       —               浮动/阈值/状态
前置        汇率定时更新                   币种配置          → 三方API       3家取平均

[1]         用户进入钱包选币种              钱包C端          → 币种配置       查可用币种
[2]         用户进入充值页                  钱包C端          → 财务(RPC)      查充值方式
                                          钱包C端          → 币种配置(RPC)   查汇率
[3]         用户选奖金活动(可选)            钱包C端          → 待定           查活动列表
[4]         用户点击"去充值"               钱包C端          本地             检查重复充值
                                          钱包C端          → 币种配置(RPC)   取汇率快照
                                          钱包C端          本地             创建充值订单
                                          钱包C端          → 财务(RPC)      匹配通道创建支付
[5]         用户在支付页完成支付            三方/平台页        —              用户操作
[6]         三方通道回调                   —                → 财务           回调到财务webhook
                                          财务             本地             验签+更新记录
                                          财务             → 钱包(RPC)      充值入账
[7]         钱包入账                       钱包             本地             幂等校验
                                          钱包             本地             余额+N，订单→成功
[8]         用户查看充值记录                钱包C端          本地             查订单表
```

**数据流转**：用户操作 → 钱包 → 财务(通道) → 三方 → 财务(回调) → 钱包(入账)。形成一个"去程走财务、回程也走财务、最终落地钱包"的环形链路。

---

### 4.2 一笔提现的完整生命周期

```
阶段        操作                          模块            方向              说明
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

前置        运营配置代付通道                财务B端          —               费率/限额/密钥
前置        运营配置提现方式                财务B端          —               关联通道/限额/状态
前置        用户已完成KYC认证              KYC服务           —               姓名/证件

[1]         用户点击"提现"                 钱包C端          → KYC(RPC)       查认证状态
                                          如未认证 → 弹窗：去认证 / 继续(仅USDT)
[2]         用户选提现方式                  钱包C端          → 财务(RPC)      查提现方式
                                          钱包C端          → 币种配置(RPC)   查汇率(BSB时)
[3]         用户填写/确认账户信息            钱包C端          → KYC(RPC)       获取姓名做校验
                                          钱包C端          本地             保存/回显账户
[4]         用户确认提现                    钱包C端          本地             校验限额+余额
                                          钱包C端          → 币种配置(RPC)   取汇率(BSB时)
                                          钱包C端          本地             【事务】冻结+创建订单
[5]         B端风控审核                     财务B端          → 钱包(RPC)      查用户余额(辅助判断)
                                          ├── 通过 → 进入财务审核
                                          └── 驳回 → 财务 → 钱包(RPC)      解冻退回
[6]         B端财务审核                     财务B端          → 钱包(RPC)      查用户余额(辅助判断)
                                          ├── 通过 → 发起代付
                                          └── 驳回 → 财务 → 钱包(RPC)      解冻退回
[7]         代付出款                        财务             → 三方通道       发起代付
                                          ├── 成功 → 财务 → 钱包(RPC)      确认扣款
                                          └── 失败 → 财务 → 钱包(RPC)      解冻退回
[8]         用户查看提现记录                 钱包C端          本地             查订单表
```

**数据流转**：用户操作 → 钱包(冻结+建单) → 财务(审核) → 钱包(扣款/解冻)。提现链路最长，经过三次钱包↔财务的交互。

### 4.3 一笔兑换的完整生命周期

```
阶段        操作                          模块            方向              说明
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

前置        汇率已就绪                     币种配置          —               平台汇率已更新
前置        用户有法币余额                  钱包             —               —

[1]         用户进入兑换页                  钱包C端          → 币种配置(RPC)   查汇率
[2]         用户输入金额                    前端             本地             基于汇率计算预览
[3]         用户点击"兑换"                 钱包C端          → 币种配置(RPC)   再次取最新汇率
                                          钱包C端          本地             校验余额
                                          钱包C端          本地             【事务】扣法币+加BSB+加赠送
[4]         兑换成功                        钱包C端          本地             返回结果
```

**数据流转**：用户操作 → 钱包 → 币种配置(取汇率) → 钱包(事务完成)。**全程不涉及财务模块**，是最简洁的链路。

### 5.2 调用关系对 service 层设计的启示

从上面的共享方法梳理可以看出，钱包模块的 service 层应至少包含以下职责单元：

```
internal/ser/
├── wallet_ser.go         ← 余额操作核心（getBalance/addBalance/deductBalance/
│                            freezeBalance/unfreezeBalance/confirmDeduct）
│                            + createTransactionLog + idempotencyCheck
│
├── deposit_ser.go        ← 充值业务（C3获取方式/C4创建订单/C5支付页/C21重复检查）
│                            调用 wallet_ser 做余额操作
│                            调用 rate_ser 做汇率获取
│
├── withdrawal_ser.go     ← 提现业务（C6获取方式/C7创建订单/C8-C9账户管理/C19限额）
│                            调用 wallet_ser 做余额操作（冻结）
│                            调用 rate_ser 做汇率获取
│
├── exchange_ser.go       ← 兑换业务（C10汇率/C11执行/C22预览）
│                            调用 wallet_ser 做余额操作（扣法币+加BSB）
│                            调用 rate_ser 做汇率获取
│
├── record_ser.go         ← 记录查询（C12-C18所有记录/详情接口）
│                            纯查询，不调用 wallet_ser
│
├── rate_ser.go           ← 汇率服务（封装对币种配置RPC的调用，可加Redis缓存）
│                            被 deposit/withdrawal/exchange_ser 共同调用
│
└── rpc_ser.go            ← RPC对外入口（R1-R8所有被外部调用的方法）
                             调用 wallet_ser 做余额操作
```
