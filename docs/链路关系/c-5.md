# 接口调用链路关系

> 基于需求文档和产品原型，梳理钱包模块内部接口之间的调用关系，以及与币种配置、财务管理、用户服务之间的跨模块调用链路
> 重点：你负责的钱包+币种配置模块的内部链路 + 与财务模块的跨模块链路

---

# 一、钱包模块内部接口调用链路

## 1.1 钱包首页场景

### 链路A：用户进入钱包首页 / 个人中心

```
C端请求: 获取钱包首页数据
    │
    ├─[1]→ 可用币种列表查询
    │       ├── [内部] 读取用户当前选中币种
    │       └── [RPC] → 币种配置.获取可用币种列表（拿币种名称/图标/状态）
    │
    └─[2]→ 钱包余额查询（当前币种）
            ├── [内部] 读取余额表（中心钱包 + 奖励钱包 + 代理钱包 + 主播钱包）
            └── [RPC] → 币种配置.获取汇率（如需展示BSB等值换算）
```

说明：用户一进入钱包页面，至少触发2个接口。可用币种列表和余额查询是**并行独立**的，不存在先后依赖。但两者都依赖币种配置模块的RPC接口作为数据源。

### 链路B：用户切换币种

```
C端请求: 切换币种（如 VND → BSB）
    │
    └─→ 钱包余额查询（新币种）
        ├── [内部] 读取新币种下的4个子钱包余额
        └── 返回后，C端整体刷新页面（充值方式、记录等全部切换为新币种上下文）
```

说明：币种切换本身不需要写操作，只是一次新币种维度的余额查询。但切换后C端会重新拉取充值方式、记录等数据，触发后续链路。

## 1.2 充值场景

### 链路C：进入充值页面

```
C端请求: 进入充值页面
    │
    ├─[1]→ 获取可用充值方式列表
    │       └── [RPC] → 财务.支付配置查询（传入当前币种 + 类型=充值 + 端=App/H5）
    │           └── 返回: 方式列表（名称/图标/金额范围/快捷金额/排序）
    │
    └─[2]→ 可选奖金活动列表（可选，页面有奖金入口时才调）
            └── [内部/RPC] 读取当前可参与的充值奖金活动配置
```

说明：获取充值方式 和 奖金活动列表 是**并行独立**的。充值方式是前置必须接口，没有它页面无法展示方式列表。奖金活动是辅助接口，不影响充值主流程。

### 链路D：用户选择充值方式 → 输入金额 → 点击确认

这是充值的核心链路，分为三段：预校验 → 创建订单 → 等待结果。

```
[第一段：预校验]

C端请求: 确认充值（金额=100, 方式=银行转账, 币种=VND）
    │
    ├─[1]→ 重复转账预校验
    │       └── [内部] 查询本用户 + 本币种 + 同金额 的处理中订单数量
    │           ├── 0笔: 直接通过
    │           ├── 1笔: 返回提醒（C端弹窗，用户可选择继续）
    │           ├── 2笔: 返回警告（C端弹窗，建议修改金额）
    │           └── ≥3笔: 返回拦截（C端强制引导修改金额或方式）
    │
    │   注：此步可独立接口，也可合并到创建订单接口的前置校验逻辑中
    │

[第二段：创建订单]

    ├─[2]→ 创建充值订单
    │       ├── [内部] 参数校验（金额范围、币种状态）
    │       ├── [RPC] → 币种配置.获取币种配置参数（精度、限额）── 做限额校验
    │       ├── [内部] 生成平台订单号，写入充值订单表（状态=待支付）
    │       ├── [内部] 如选了奖金活动 → 记录活动关联（待充值成功后发放）
    │       └── [RPC] → 财务.创建渠道支付订单
    │               ├── 入参: 用户ID、币种、金额、支付方式ID、平台订单号
    │               └── 返回: 渠道订单号 + 支付信息（跳转URL / USDT收款地址 / 二维码）
    │
    └── 返回C端: 支付信息（跳转到第三方页面 或 展示USDT地址）


[第三段：等待支付结果]

C端: 轮询 或 等待通知
    │
    └─[3]→ 充值订单状态查询
            └── [内部] 读取订单表状态（待支付 / 成功 / 失败 / 超时）
```

**创建充值订单内部的调用顺序**（串行依赖）：

```
参数校验 → 币种配置参数校验（RPC）→ 重复转账校验（内部）→ 写订单表 → 调财务创建渠道订单（RPC）→ 返回
```

这里面**币种配置参数校验**和**重复转账校验**可以并行，但都必须在**写订单表**之前完成。**调财务创建渠道订单**必须在订单表写入后，因为需要传平台订单号。

### 链路E：充值成功回调（RPC被调用）

```
财务模块收到第三方渠道回调 → 确认支付成功
    │
    └── [RPC调用] → 钱包.充值入账
            ├── [内部] 幂等校验（用平台订单号去重，防止重复入账）
            ├── [内部] 更新充值订单状态 → 成功
            ├── [内部] 中心钱包余额 + 充值金额
            ├── [内部] 如有奖金活动 → 奖励钱包余额 + 奖金金额
            ├── [内部] 写入余额变动流水记录
            └── 返回: 成功/失败（失败原因）
```

**充值入账内部的调用顺序**（严格串行，事务内）：

```
幂等校验 → 更新订单状态 → 增加余额 → 写流水记录 → 提交事务
```

这些步骤必须在**同一个数据库事务**内完成，保证余额和流水的一致性。幂等校验必须第一步，避免重复操作。

## 1.3 兑换场景

### 链路F：进入兑换页面

```
C端请求: 进入兑换页面
    │
    └─[1]→ 获取兑换汇率及预览
            └── [RPC] → 币种配置.获取汇率（当前法币/USDT 对 BSB）
                └── 返回: "1 BSB = 23000 VND"，赠送比例等
```

说明：兑换页面只需要一个接口：汇率查询。这是所有业务场景中**前置依赖最少**的。

### 链路G：用户输入金额 → 确认兑换

```
C端请求: 创建兑换订单（从VND兑换BSB, 金额=230000 VND）
    │
    └─→ 创建兑换订单
        ├── [RPC] → 币种配置.获取汇率（实时获取，不用缓存，防止汇率已变）
        ├── [内部] 计算: 230000 VND ÷ 23000 = 10 BSB
        ├── [内部] 计算赠送: 10 BSB × 5% = 0.5 BSB（如有赠送活动）
        ├── [内部] 校验: 用户VND中心钱包余额 ≥ 230000
        ├── [内部] 事务操作:
        │       ├── 扣减 VND 中心钱包余额 230000
        │       ├── 增加 BSB 中心钱包余额 10
        │       ├── 增加 BSB 奖励钱包余额 0.5（赠送部分）
        │       ├── 写兑换订单记录
        │       └── 写余额变动流水（VND扣减 + BSB增加 + BSB奖励增加）
        └── 返回: 兑换成功/失败
```

**兑换订单内部的调用顺序**（严格串行，事务内）：

```
获取最新汇率（RPC）→ 金额计算 → 余额校验 → [事务开始] → 扣减源币种 → 增加BSB → 增加赠送 → 写订单 → 写流水 → [事务提交]
```

关键点：**获取汇率必须在事务之前**（RPC调用不能在数据库事务内，会拉长事务时间）。计算结果进入事务后直接执行，不再二次查汇率。

兑换是唯一一个**完全不涉及财务模块**的交易链路。它只依赖币种配置的汇率，剩下全是钱包模块内部操作。

## 1.4 提现场景

### 链路H：进入提现页面

```
C端请求: 进入提现页面
    │
    ├─[1]→ KYC状态查询
    │       └── [RPC] → 用户服务.KYC状态查询（传入用户ID）
    │           ├── 已认证: 继续
    │           └── 未认证: C端引导去KYC认证页面（非USDT提现不允许继续）
    │
    ├─[2]→ 获取可用提现方式列表
    │       ├── [RPC] → 财务.支付配置查询（传入当前币种 + 类型=提现）
    │       │       └── 返回: 平台支持的提现方式候选集
    │       └── [内部] 根据用户充值历史过滤
    │               ├── 只充过法币 → 只保留法币方式
    │               ├── 只充过USDT → 只保留USDT方式
    │               └── 混合充值 → 只保留法币方式
    │
    ├─[3]→ 提现限额查询
    │       └── [RPC] → 币种配置.获取币种配置参数（限额/手续费/T+N）
    │           └── 返回: 单笔最低/最高、每日限额、手续费规则、T+N天数
    │
    └─[4]→ 提现账户信息查询
            └── [内部] 查询用户已保存的银行卡/电子钱包信息
                ├── 有记录: 返回已保存信息（后续页面只读展示）
                └── 无记录: 返回空（后续页面展示填写表单）
```

说明：进入提现页面触发4个接口。其中[1][2][3][4]可以**并行调用**，互不依赖。但[1]KYC校验的结果决定了用户能否继续操作——如果KYC未通过，其他接口的返回结果用户看不到（被引导去认证了）。

**获取可用提现方式**内部有两步串行依赖：先拿财务的候选集 → 再用充值历史过滤。这两步不能并行，因为过滤需要候选集作为输入。

### 链路I：用户填写/确认提现信息 → 提交提现

```
C端请求: 创建提现订单（金额=500000 VND, 方式=银行转账, 账户信息=xxx）
    │
    └─→ 创建提现订单
        ├── [1] [RPC] → 币种配置.获取币种配置参数 → 限额校验
        │       ├── 单笔金额 在 最低~最高 范围内？
        │       └── 今日累计提现 + 本笔 ≤ 每日限额？
        │
        ├── [2] [RPC] → 用户服务.KYC信息查询 → 户名一致性校验
        │       └── 提现账户户名 与 KYC姓名 比对（忽略大小写和首尾空格）
        │
        ├── [3] [内部] 余额校验
        │       └── 中心钱包余额 ≥ 提现金额 + 手续费？
        │
        │   注：[1][2][3]可并行执行，全部通过才继续
        │
        ├── [4] [内部] 冻结余额（余额表扣减可用余额，增加冻结余额）
        │
        ├── [5] [内部] 生成平台订单号，写入提现订单表（状态=审核中）
        │
        ├── [6] [内部] 保存/更新提现账户信息（首次填写时写入，后续自动回显）
        │
        └── [7] [RPC] → 财务.提交提现审核
                ├── 入参: 用户ID、币种、金额、提现方式、平台订单号、账户信息
                └── 返回: 审核单号
```

**创建提现订单内部的调用顺序**：

```
[可并行] 限额校验(RPC) + 户名校验(RPC) + 余额校验(内部)
    │
    │ 全部通过
    ▼
[串行] 冻结余额 → 写订单表 → 保存账户信息 → 提交财务审核(RPC)
```

关键点：**冻结余额**必须在**提交财务审核之前**完成。如果先提交审核再冻结，中间用户可能把钱花了，导致冻结失败。**提交财务审核**必须在订单写入之后，因为要传订单号。

### 链路J：提现审核结果回调（RPC被调用）

**审核通过 + 出款成功**：

```
财务模块: 三级审核全部通过 + 渠道出款成功
    │
    └── [RPC调用] → 钱包.提现确认扣款
            ├── [内部] 校验订单状态 = 审核中（防止重复操作）
            ├── [内部] 冻结余额 → 正式扣除（冻结字段减少，不影响可用余额）
            ├── [内部] 更新提现订单状态 → 成功
            ├── [内部] 写余额变动流水记录
            └── 返回: 成功/失败
```

**审核拒绝 或 出款失败**：

```
财务模块: 任一级审核拒绝 或 渠道出款失败
    │
    └── [RPC调用] → 钱包.提现失败解冻
            ├── [内部] 校验订单状态 = 审核中
            ├── [内部] 冻结余额 → 解冻退回（冻结字段减少，可用余额恢复）
            ├── [内部] 更新提现订单状态 → 失败（记录失败原因）
            ├── [内部] 写余额变动流水记录
            └── 返回: 成功/失败
```

**确认扣款和失败解冻内部的调用顺序**（严格串行，事务内）：

```
订单状态校验 → [事务开始] → 余额操作（扣除或解冻）→ 更新订单状态 → 写流水 → [事务提交]
```

## 1.5 记录查询场景

### 链路K：查询记录列表

```
C端请求: 充值记录列表（币种=VND, 分页）
    │
    └─→ 充值记录列表查询
        └── [内部] 按用户ID + 币种 + 分页参数查询充值订单表
            └── 返回: 订单列表（订单号/金额/状态/时间/方式）
```

```
C端请求: 提现记录列表 / 兑换记录列表 / 奖励记录列表
    │
    └─→ 对应记录列表查询（同上，查不同的表/不同的类型筛选条件）
```

### 链路L：查询记录详情

```
C端请求: 记录详情（订单号=xxx）
    │
    └─→ 记录详情查询
        └── [内部] 按订单号查询对应表的完整字段
            └── 返回: 完整信息（金额/状态/时间/方式/渠道/失败原因等）
```

说明：记录查询是**完全自闭环**的链路，不依赖任何外部RPC调用。所有数据在充值/提现/兑换/奖金发放流程中已经写入了本地表，查询时直接读取。

## 1.6 人工修正场景（RPC被调用）

### 链路M：人工加款

```
财务模块: 加款审核通过
    │
    └── [RPC调用] → 钱包.人工加款
            ├── 入参: 用户ID、币种、金额、子钱包类型（中心/奖励/代理/主播）、修正单号、原因
            ├── [内部] 幂等校验（修正单号去重）
            ├── [内部] [事务开始]
            │       ├── 指定子钱包余额 + 金额
            │       ├── 写余额变动流水（类型=人工加款，关联修正单号）
            │       └── [事务提交]
            └── 返回: 成功/失败
```

### 链路N：人工减款

```
财务模块: 减款审核通过
    │
    └── [RPC调用] → 钱包.人工减款
            ├── 入参: 同加款 + 需校验余额
            ├── [内部] 幂等校验
            ├── [内部] 余额校验（指定子钱包余额 ≥ 减款金额）
            ├── [内部] [事务开始]
            │       ├── 指定子钱包余额 - 金额
            │       ├── 写余额变动流水（类型=人工减款）
            │       └── [事务提交]
            └── 返回: 成功/失败（余额不足则失败）
```

### 链路O：人工补单

```
财务模块: 补单审核通过
    │
    └── [RPC调用] → 钱包.人工补单入账
            ├── 入参: 用户ID、币种、金额、原充值订单号、补单号
            ├── [内部] 幂等校验（补单号去重）
            ├── [内部] 查找原充值订单（确认存在且状态为失败/超时）
            ├── [内部] [事务开始]
            │       ├── 中心钱包余额 + 金额
            │       ├── 更新原充值订单状态 → 补单成功
            │       ├── 写余额变动流水（类型=人工补单，关联原订单+补单号）
            │       └── [事务提交]
            └── 返回: 成功/失败
```

### 链路P：余额查询（供财务调用）

```
财务模块: 创建人工修正前确认余额
    │
    └── [RPC调用] → 钱包.余额查询
            ├── 入参: 用户ID、币种（可选）、子钱包类型（可选）
            └── [内部] 读取余额表
                └── 返回: 余额信息（可用余额/冻结余额，按子钱包分组）
```

# 二、币种配置模块内部接口调用链路

币种配置模块的接口链路相对简单，绝大多数是独立的CRUD操作。

### 链路Q：币种管理（B端）

```
B端请求: 币种列表 / 新增 / 编辑 / 启用禁用
    │
    └─→ 对应CRUD操作
        ├── [内部] 读写币种配置表
        └── [内部] 写操作日志表（谁/什么时间/改了什么/改前值/改后值）
```

### 链路R：汇率管理（B端）

```
B端请求: 设置汇率（如 1 BSB = 23000 VND → 1 BSB = 23500 VND）
    │
    └─→ 汇率设置
        ├── [内部] 更新汇率配置表
        └── [内部] 写汇率变更日志（币种/旧值/新值/操作人/时间）
```

### 链路S：RPC被调用（供其他模块查询）

```
其他模块（钱包/财务）RPC调用:
    │
    ├─→ 获取可用币种列表 → [内部] 读币种配置表（status=启用）
    ├─→ 获取币种详情 → [内部] 读币种配置表（指定币种）
    ├─→ 获取汇率 → [内部] 读汇率配置表（指定币种对BSB）
    └─→ 获取配置参数 → [内部] 读币种配置表（精度/限额/费率/T+N字段）
```

说明：币种配置的RPC接口全部是**读操作**，不存在写操作的RPC暴露。其他模块只能读取币种配置数据，不能通过RPC修改。修改只能通过B端管理界面操作。

# 三、与财务模块的跨模块调用链路

## 3.1 钱包主动调用财务的链路

### 链路T1：获取充值/提现方式

```
钱包C端 → 钱包服务 → [RPC] → 财务.支付配置查询
    │
    ├── 触发时机: 用户进入充值页面 或 提现页面
    ├── 调用方向: 钱包 → 财务（同步调用，等待返回）
    ├── 入参: 币种、类型（充值/提现）、端（App/H5/Web）
    ├── 出参: 方式列表（名称/图标/金额范围/快捷金额/排序）
    └── 失败处理: 如果财务服务不可用，充值/提现页面无法展示方式列表，功能降级
```

### 链路T2：创建渠道支付订单（充值）

```
钱包C端创建充值订单 → 钱包服务 → [RPC] → 财务.创建渠道支付订单
    │
    ├── 触发时机: 用户确认充值后，钱包本地订单已创建
    ├── 调用方向: 钱包 → 财务（同步调用，等待返回）
    ├── 入参: 用户ID、币种、金额、支付方式ID、钱包平台订单号
    ├── 出参: 渠道订单号 + 支付信息（跳转URL 或 USDT收款地址）
    ├── 财务内部处理: 通道轮询匹配 → 调用第三方渠道API → 获取支付信息
    └── 失败处理: 财务返回通道不可用 → 钱包订单标记失败，提示用户更换方式
```

### 链路T3：提交提现审核

```
钱包C端创建提现订单 → 钱包服务（冻结余额后）→ [RPC] → 财务.提交提现审核
    │
    ├── 触发时机: 用户确认提现 + 钱包冻结余额成功后
    ├── 调用方向: 钱包 → 财务（同步调用，等待返回审核单号）
    ├── 入参: 用户ID、币种、金额、提现方式、钱包订单号、账户信息
    ├── 出参: 审核单号
    ├── 财务内部处理: 创建提现审核记录，进入风控→财务→出款审核队列
    └── 失败处理: 财务接收失败 → 钱包解冻余额，订单标记失败
```

**注意**：链路T3有一个关键的**补偿逻辑**——如果提交审核这步RPC调用失败了（网络超时、财务服务宕机等），钱包已经冻结了余额但财务没收到审核请求。此时必须解冻余额，否则用户的钱就"卡住了"。这是典型的分布式事务问题，需要重点关注。

## 3.2 财务主动调用钱包的链路

### 链路T4：充值入账回调

```
第三方渠道 → 财务.回调处理 → [RPC] → 钱包.充值入账
    │
    ├── 触发时机: 第三方渠道回调通知财务充值成功
    ├── 调用方向: 财务 → 钱包（同步调用）
    ├── 入参: 用户ID、币种、金额、钱包订单号、渠道订单号
    ├── 钱包内部处理: 幂等校验 → 增加余额 → 更新订单 → 写流水（见链路E）
    ├── 出参: 成功/失败
    └── 失败处理: 钱包返回失败 → 财务记录异常，等待人工补单
```

### 链路T5：提现确认/拒绝

```
财务.审核完成 → [RPC] → 钱包.提现确认扣款 或 钱包.提现失败解冻
    │
    ├── 触发时机: 三级审核全部通过+渠道出款成功 → 调确认扣款
    │             任一级审核拒绝 或 渠道出款失败 → 调失败解冻
    ├── 调用方向: 财务 → 钱包（同步调用）
    ├── 入参: 钱包订单号（或冻结流水号）
    ├── 钱包内部处理: 见链路J
    └── 失败处理: 钱包返回失败 → 财务记录异常，告警人工介入
```

### 链路T6：人工修正操作

```
财务.人工修正审核通过 → [RPC] → 钱包.人工加款/减款/补单
    │
    ├── 触发时机: B端操作人创建修正单 → 审核人审核通过后
    ├── 调用方向: 财务 → 钱包（同步调用）
    ├── 入参: 用户ID、币种、金额、子钱包类型、修正单号、原因
    ├── 钱包内部处理: 见链路M/N/O
    └── 失败处理: 钱包返回失败（如减款余额不足）→ 财务记录修正失败
```

### 链路T7：余额查询

```
财务.创建人工修正 → [RPC] → 钱包.余额查询
    │
    ├── 触发时机: B端操作人创建减款单前查看用户余额
    ├── 调用方向: 财务 → 钱包（同步调用，只读）
    ├── 入参: 用户ID、币种、子钱包类型
    └── 出参: 余额信息（可用/冻结）
```

    ├── 钱包.创建充值订单（内部写表）
    ├── 钱包 ──RPC──→ 财务.创建渠道支付订单

# 六、关键设计要点（从链路分析中得出）

## 6.1 必须事务保证的操作

以下操作涉及余额变动，必须在同一个数据库事务内完成，不能拆分：

1. **充值入账**：余额增加 + 订单状态更新 + 流水记录 → 同一事务
2. **兑换**：源币种扣减 + BSB增加 + 赠送增加 + 订单记录 + 流水 → 同一事务
3. **提现冻结**：可用余额减少 + 冻结余额增加 → 同一事务
4. **提现确认**：冻结余额扣除 + 订单状态更新 + 流水 → 同一事务
5. **提现解冻**：冻结余额减少 + 可用余额恢复 + 订单状态更新 + 流水 → 同一事务
6. **人工加减款**：余额变动 + 流水 → 同一事务

## 6.2 必须幂等的接口

以下接口可能被重复调用（网络重试、回调重发等），必须保证幂等：

1. **充值入账**（财务回调可能重发）→ 用平台订单号做幂等键
2. **提现确认扣款/失败解冻**（财务可能重试）→ 用订单号做幂等键
3. **人工加减款/补单**（财务可能重试）→ 用修正单号做幂等键

## 6.3 需要补偿机制的链路

以下链路涉及跨模块调用，存在中间状态不一致的风险：

1. **创建提现订单**：钱包冻结余额成功 → 提交财务审核失败 → 必须解冻余额（补偿）
2. **充值创建**：钱包创建订单成功 → 财务创建渠道订单失败 → 必须将订单标记失败（补偿）
3. **人工减款**：财务审核通过 → 调用钱包减款失败（余额不足等）→ 财务需标记修正失败（补偿）
