# ser-wallet 接口调用链路关系

> 目的：梳理每个接口在业务流程中的完整调用链路——谁触发、经过哪些接口、什么顺序、数据怎么流转
> 视角：以"我负责的模块"（钱包 + 币种配置）为中心，区分模块内和跨模块调用
> 依据：需求文档 + 产品原型图（约133张）的分析结果
> 说明：箭头 → 表示调用方向，← 表示返回方向

# 第一部分：模块内部接口调用关系

> 这部分梳理"我的模块内部"接口之间的调用和依赖关系
> 同属 ser-wallet 服务，内部调用走 Service 层方法，不走 RPC

## 一、钱包模块内部调用关系

### 1.1 钱包模块接口全景

钱包模块共约31个接口，按调用关系可以分为3层：

```
┌─────────────────────────────────────────────────────────────────────┐
│  C端接口层（用户直接操作触发）                                         │
│                                                                     │
│  GetWalletInfo   SwitchCurrency   GetDepositMethods   GetWithdraw-  │
│  GetExchangeRate CreateExchange   CreateDeposit       Methods       │
│  GetDepositBonus CheckDuplicate   CreateWithdraw      CheckWith-    │
│  GetDeposit/Withdraw/Exchange/RewardRecords           drawEligib-   │
│  GetRecordDetail GetDepositOrderStatus                ility         │
├─────────────────────────────────────────────────────────────────────┤
│  内部Service层（被C端接口和RPC接口共同调用）                           │
│                                                                     │
│  WalletService       CurrencyService(币种配置)     OrderService     │
│  ·查询余额            ·获取可用币种                 ·创建订单         │
│  ·加款(上分)          ·获取汇率                    ·查询订单         │
│  ·扣款(下分)          ·获取币种信息                 ·更新订单状态     │

```

### 1.2 C端接口之间的内部调用关系

以下梳理每个C端接口在处理用户请求时，内部调用了哪些Service方法：

#### GetWalletInfo（获取钱包信息）

```
用户请求
  │
  ▼
GetWalletInfo
  │
  ├──→ CurrencyService.GetUserCurrentCurrency()  // 获取用户当前选中的币种
  │
  ├──→ WalletService.GetBalance(userId, currencyId)  // 查询中心钱包余额
  │
  ├──→ WalletService.GetBalance(userId, currencyId, rewardType)  // 查询奖励钱包余额
  │
  └──→ 组装返回：资金钱包余额 = 中心 + 奖励
```

调用链路：纯内部，不跨模块。

#### SwitchCurrency（切换币种）

```
用户请求(目标币种ID)
  │
  ▼
SwitchCurrency
  │
  ├──→ CurrencyService.GetCurrencyInfo(currencyId)  // 校验目标币种是否存在且启用
  │         │
  │         └── 币种不存在或已禁用 → 返回错误
  │
  ├──→ WalletService.EnsureWalletExists(userId, currencyId)  // 确保该币种下用户钱包已初始化
  │         │
  │         └── 如果没有 → 自动创建（中心钱包 + 奖励钱包，余额为0）
  │
  └──→ UserService.UpdateCurrentCurrency(userId, currencyId)  // 更新用户的当前币种
```

调用链路：纯内部。注意 EnsureWalletExists 是懒初始化——用户第一次切到某个币种时才创建钱包。

#### GetExchangeRate（获取兑换汇率）

```
用户请求(源币种=当前法币)
  │
  ▼
GetExchangeRate
  │
  ├──→ CurrencyService.GetUserCurrentCurrency()  // 获取当前币种
  │
  ├──→ CurrencyService.GetPlatformRate(currencyId)  // 获取该法币的平台汇率
  │         │
  │         └── 平台汇率 = 最近一次超过阈值时更新的汇率
  │
  └──→ 计算返回：1单位法币 = X个BSB（基于平台汇率 + BSB锚定 10BSB=1USDT）
```

调用链路：内部调用币种配置的Service。不跨模块。

#### CreateExchangeOrder（创建兑换订单）

```
用户请求(法币金额)
  │
  ▼
CreateExchangeOrder
  │
  ├──→ CurrencyService.GetPlatformRate(currencyId)  // 获取当前平台汇率
  │
  ├──→ 计算到账BSB = 法币金额 × 平台汇率 / BSB_USDT锚定比
  │    计算奖金BSB = 到账BSB × 10%（兑换奖金比例）
  │
  ├──→ WalletService.GetBalance(userId, currencyId, centerType)  // 检查法币余额
  │         │
  │         └── 余额不足 → 返回错误
  │
  ├──→ WalletService.Debit(userId, currencyId, centerType, amount)  // 扣法币中心钱包
  │
  ├──→ WalletService.Credit(userId, BSB_ID, centerType, bsbAmount)  // 加BSB中心钱包
  │
  ├──→ WalletService.Credit(userId, BSB_ID, rewardType, bonusAmount)  // 奖金加BSB奖励钱包
  │
  └──→ OrderService.CreateExchangeRecord(...)  // 创建兑换记录
```

调用链路：内部完成，涉及3次余额操作（1扣2加），这3个操作必须在同一个事务中。不跨模块。

**关键点：兑换涉及跨币种操作（扣A币种，加B币种），要保证事务原子性。**

#### CheckDuplicateDeposit（重复转账拦截检查）

```
用户请求(币种, 金额)
  │
  ▼
CheckDuplicateDeposit
  │
  └──→ OrderService.CountPendingDeposits(userId, currencyId, amount)
           │
           ├── count = 0 → 正常通过
           ├── count = 1~2 → 返回警告（前端弹窗让用户确认）
           └── count ≥ 3 → 返回阻断（禁止提交）
```

调用链路：纯内部，只查订单表。

#### GetDepositBonusList（获取充值奖金活动列表）

```
用户请求(当前币种)
  │
  ▼
GetDepositBonusList
  │
  └──→ ActivityService.GetAvailableBonusActivities(currencyId)
           │
           └── 返回：活动名称、奖金比例、最低充值门槛、稽核倍数
```

调用链路：活动数据的来源待确认（可能是同服务配置表，也可能调用活动系统RPC）。

#### GetDepositRecords / GetWithdrawRecords / GetExchangeRecords / GetRewardRecords

```
用户请求(分页参数, 时间范围)
  │
  ▼
GetXxxRecords
  │
  ├──→ CurrencyService.GetUserCurrentCurrency()  // 获取当前币种
  │
  └──→ OrderService.ListRecords(userId, currencyId, type, page)  // 按类型查订单表
           │
           └── 返回分页列表
```

调用链路：四个记录接口结构一致，只是 type 参数不同，纯内部查询。

#### GetRecordDetail（记录详情）

```
用户请求(订单类型, 订单ID)
  │
  ▼
GetRecordDetail
  │
  └──→ OrderService.GetOrderDetail(type, orderId)
           │
           └── 根据订单号前缀(C/T/B/A/M)判断类型，查对应表
```

调用链路：纯内部。

### 1.3 RPC接口的内部实现链路

这些RPC接口被外部模块（主要是财务管理）调用，但它们的内部实现只操作自己的数据：

#### Credit（加款/上分）

```
外部RPC请求(userId, currencyId, walletType, amount, orderNo, reason)
  │
  ▼
Credit
  │
  ├──→ 幂等校验：OrderService.CheckDuplicate(orderNo)  // 防止同一笔订单重复加款
  │         │
  │         └── 已处理过 → 直接返回成功（幂等）
  │
  ├──→ WalletService.Credit(userId, currencyId, walletType, amount)
  │         │
  │         └── 增加指定钱包的余额（乐观锁/行锁保证并发安全）
  │
  └──→ OrderService.CreateFlowRecord(...)  // 创建流水记录
```

#### Debit（扣款/下分）

```
外部RPC请求(userId, currencyId, walletType, amount, orderNo, reason)
  │
  ▼
Debit
  │
  ├──→ 幂等校验：OrderService.CheckDuplicate(orderNo)
  │
  ├──→ WalletService.GetBalance(userId, currencyId, walletType)  // 查余额
  │         │
  │         └── 余额不足 → 返回错误
  │
  ├──→ WalletService.Debit(userId, currencyId, walletType, amount)
  │         │
  │         └── 减少指定钱包的余额
  │
  └──→ OrderService.CreateFlowRecord(...)
```

#### Freeze（冻结金额）

```
外部RPC请求(userId, currencyId, amount, orderNo)
  │
  ▼
Freeze
  │
  ├──→ WalletService.GetBalance(userId, currencyId, centerType)  // 查可用余额
  │         │
  │         └── 可用余额不足 → 返回错误
  │
  ├──→ WalletService.Freeze(userId, currencyId, amount)
  │         │
  │         └── 可用余额减少amount，冻结金额增加amount（总余额不变）
  │
  └──→ OrderService.CreateFreezeRecord(...)
```

#### Unfreeze（解冻金额）

```
外部RPC请求(userId, currencyId, amount, orderNo)
  │
  ▼
Unfreeze
  │
  ├──→ WalletService.Unfreeze(userId, currencyId, amount)
  │         │
  │         └── 冻结金额减少amount，可用余额增加amount
  │
  └──→ OrderService.CreateUnfreezeRecord(...)
```

#### QueryBalance（查询余额）

```
外部RPC请求(userId, currencyId)
  │
  ▼
QueryBalance
  │
  └──→ WalletService.GetBalance(userId, currencyId)
           │
           └── 返回：中心钱包余额、奖励钱包余额、冻结金额、可用余额
```

**关键发现：**

- `WalletService.GetBalance` 是使用频率最高的内部方法，被 C端接口、RPC接口、以及内部校验逻辑共同调用
- `CurrencyService` 的方法是同服务内部调用（不走RPC），被钱包模块的多个接口调用
- `OrderService` 负责所有订单/流水的读写，是数据持久化层的统一入口
- C端接口之间没有直接互调关系（A接口不会调B接口），它们通过共享底层 Service 间接关联

## 二、币种配置模块内部调用关系

### 2.1 币种配置接口全景

币种配置模块共约11个接口 + 2个定时任务，全部独立无外部依赖：

```
┌─────────────────────────────────────────────────────────────────┐
│  B端接口层（管理员操作触发）                                       │
│                                                                 │
│  GetCurrencyConfigList  CreateCurrency  UpdateCurrency          │
│  ToggleCurrencyStatus   GetRateConfig   UpdateRateConfig       │
│  GetRateChangeLog       GetCurrencyDetail                      │
├─────────────────────────────────────────────────────────────────┤
│  内部Service层                                                   │
│                                                                 │
│  CurrencyConfigService    RateService                          │
│  ·币种CRUD                ·获取/更新实时汇率                      │
│  ·币种状态管理             ·获取/更新平台汇率                      │
│  ·币种信息查询             ·计算充值/提现汇率                      │
│                           ·记录汇率变更日志                       │
├─────────────────────────────────────────────────────────────────┤
│  RPC接口层（被外部模块调用）                                       │
│                                                                 │
│  GetAvailableCurrencies  GetExchangeRate  GetCurrencyInfo      │
├─────────────────────────────────────────────────────────────────┤
│  定时任务                                                        │
│                                                                 │
│  SyncThirdPartyRates     CheckRateThreshold                    │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 B端接口的内部调用链路

#### CreateCurrency（新增币种）

```
管理员请求(名称, 简称, icon, 充值浮动%, 提现浮动%, 阈值%)
  │
  ▼
CreateCurrency
  │
  ├──→ CurrencyConfigService.CheckDuplicate(简称)  // 校验简称是否重复
  │
  ├──→ S3Service.Upload(icon文件)  // 上传币种图标（唯一的外部调用：ser-s3）
  │
  └──→ CurrencyConfigService.Create(...)  // 写入币种配置表
```

#### UpdateCurrency（编辑币种）

```
管理员请求(currencyId, 修改字段)
  │
  ▼
UpdateCurrency
  │
  ├──→ CurrencyConfigService.GetById(currencyId)  // 查当前配置
  │
  ├──→ [如果修改了icon] S3Service.Upload(新icon)
  │
  ├──→ CurrencyConfigService.Update(...)  // 更新配置
  │
  └──→ [如果修改了浮动比例或阈值] RateService.RecalculateRates(currencyId)
           │
           └── 重新计算该币种的充值汇率和提现汇率
```

#### ToggleCurrencyStatus（启用/禁用币种）

```
管理员请求(currencyId, 目标状态)
  │
  ▼
ToggleCurrencyStatus
  │
  ├──→ CurrencyConfigService.GetById(currencyId)  // 查当前状态
  │
  ├──→ [禁用前] 检查是否有关联的活跃通道/方式（可选的安全校验）
  │         │
  │         └── 如果有活跃关联 → 提示风险/需确认
  │
  └──→ CurrencyConfigService.UpdateStatus(currencyId, status)
```

**重要影响：禁用一个币种后，财务模块关联该币种的通道和支付方式理论上应该也无法使用。这是一个跨模块联动效应，但不是接口调用，而是数据层面的影响。**

#### UpdateRateConfig（更新汇率配置）

```
管理员请求(currencyId, 新阈值%, 新充值浮动%, 新提现浮动%)
  │
  ▼
UpdateRateConfig
  │
  ├──→ CurrencyConfigService.GetRateConfig(currencyId)  // 查当前配置
  │
  ├──→ CurrencyConfigService.UpdateRateConfig(...)  // 更新配置
  │
  └──→ RateService.RecalculateRates(currencyId)
           │
           ├── 新充值汇率 = 实时汇率 × (1 + 新充值浮动%)
           └── 新提现汇率 = 实时汇率 × (1 - 新提现浮动%)
```

### 2.3 定时任务的内部调用链路

#### SyncThirdPartyRates（定时拉取第三方汇率）

```
定时触发（分钟级）
  │
  ▼
SyncThirdPartyRates
  │
  ├──→ CurrencyConfigService.GetAllActiveCurrencies()  // 获取所有启用的法币
  │
  ├──→ 对每个法币币种:
  │      │
  │      ├──→ ThirdPartyAPI_1.GetRate(currency→USDT)  // 第三方API 1
  │      ├──→ ThirdPartyAPI_2.GetRate(currency→USDT)  // 第三方API 2
  │      ├──→ ThirdPartyAPI_3.GetRate(currency→USDT)  // 第三方API 3
  │      │
  │      ├──→ 计算平均值 = (rate1 + rate2 + rate3) / 3
  │      │
  │      └──→ RateService.UpdateRealtimeRate(currencyId, avgRate)  // 更新实时汇率
  │
  └──→ 触发 CheckRateThreshold（检查是否需要更新平台汇率）
```

#### CheckRateThreshold（汇率阈值检查）

```
由 SyncThirdPartyRates 触发
  │
  ▼
CheckRateThreshold
  │
  ├──→ 对每个法币币种:
  │      │
  │      ├──→ RateService.GetRealtimeRate(currencyId)    // 当前实时汇率
  │      ├──→ RateService.GetPlatformRate(currencyId)    // 当前平台汇率
  │      │
  │      ├──→ 计算偏差 = |实时 - 平台| / 实时 × 100%
  │      │
  │      ├──→ [偏差 > 阈值]
  │      │      ├──→ RateService.UpdatePlatformRate(currencyId, 新平台汇率)
  │      │      ├──→ RateService.RecalculateRates(currencyId)  // 联动更新充值/提现汇率
  │      │      └──→ RateService.CreateChangeLog(...)  // 记录变更日志
  │      │
  │      └──→ [偏差 ≤ 阈值] → 不更新，跳过
```

**两个定时任务的关系：SyncThirdPartyRates → 触发 → CheckRateThreshold，是串行执行关系。**

### 2.4 币种配置模块接口间的内部依赖总结

```
B端接口                         依赖链路
────────                        ────────

CreateCurrency ───────────→ CurrencyConfigService.Create
                              (外部: S3Service.Upload)

UpdateCurrency ───────────→ CurrencyConfigService.Update
                              RateService.RecalculateRates
                              (外部: S3Service.Upload)

ToggleCurrencyStatus ─────→ CurrencyConfigService.UpdateStatus

GetRateConfig ────────────→ CurrencyConfigService.GetRateConfig (纯读)

UpdateRateConfig ─────────→ CurrencyConfigService.UpdateRateConfig
                              RateService.RecalculateRates

GetRateChangeLog ─────────→ RateService.ListChangeLogs (纯读)

GetCurrencyDetail ────────→ CurrencyConfigService.GetById (纯读)

GetCurrencyConfigList ────→ CurrencyConfigService.ListAll (纯读)


定时任务                        依赖链路
────────                        ────────

SyncThirdPartyRates ──────→ 三方汇率API (外部HTTP)
                              RateService.UpdateRealtimeRate
                              → 串行触发 CheckRateThreshold

CheckRateThreshold ───────→ RateService.GetRealtimeRate
                              RateService.GetPlatformRate
                              RateService.UpdatePlatformRate
                              RateService.RecalculateRates
                              RateService.CreateChangeLog
```

**关键发现：**

- 币种配置模块内部，`RateService.RecalculateRates` 是一个关键联动方法——被 UpdateCurrency、UpdateRateConfig、CheckRateThreshold 三处调用
- 任何修改浮动比例或触发阈值更新的操作，都会通过 RecalculateRates 联动更新充值/提现汇率
- 模块唯一的外部依赖是 S3（icon上传）和三方汇率API，与其他业务模块完全解耦

## 三、钱包模块与币种配置模块的内部调用关系

> 钱包和币种配置同属 ser-wallet 服务，它们之间是 Service 层内部调用，不走 RPC

```
钱包模块的接口                调用方向              币种配置的 Service 方法
──────────────              ──────────            ────────────────────

GetWalletInfo ──────────→ CurrencyService.GetUserCurrentCurrency()
SwitchCurrency ─────────→ CurrencyService.GetCurrencyInfo()
GetExchangeRate ────────→ CurrencyService.GetPlatformRate()
CreateExchangeOrder ────→ CurrencyService.GetPlatformRate()
GetUserCurrencyList ────→ CurrencyService.GetAvailableCurrencies()
4个Records接口 ─────────→ CurrencyService.GetUserCurrentCurrency()

反向调用: 无（币种配置不调用钱包模块的任何方法）
```

**调用方向是单向的：钱包 → 币种配置，从不反向。**

这些调用的共同特点：
1. 都是读操作（查币种信息、查汇率），不涉及写币种配置的数据
2. 调用频率高（几乎每个C端接口都需要获取当前币种信息）
3. 可以考虑对 GetUserCurrentCurrency 做进程内缓存，减少重复查询

### 4.2 充值提交阶段

```
用户选择充值方式 + 输入金额 + 选择奖金活动(可选) → 点击提交
  │
  ▼
[C端] CreateDepositOrder(currencyId, methodId, amount, bonusActivityId?)
  │
  │  ① 重复转账拦截检查（内部）
  ├──→ OrderService.CountPendingDeposits(userId, currencyId, amount)
  │         │
  │         ├── count ≥ 3 → 直接返回"阻断，请勿重复提交"
  │         └── count = 1~2 → 检查前端是否传了"用户已确认"标记
  │
  │  ② 匹配支付通道（跨模块）
  ├──→ [RPC] 财务模块.MatchDepositChannel(currencyId, methodId, amount)
  │         │
  │         │  财务模块内部处理:
  │         ├── 查该币种+方式类型对应的轮询规则
  │         ├── 筛选满足金额范围且未超日交易上限的启用通道
  │         ├── 按权重轮询选择一个通道
  │         │     ├── 选中 → 返回通道信息
  │         │     └── 全部不可用 → 返回"通道繁忙"错误

  │

  ├──→ [RPC] 财务模块.CreateChannelOrder(channelId, userId, amount, orderNo)
  │         │
  │         │  财务模块内部处理:
  │         ├── 生成内部订单记录（状态：待支付）
  │         ├── 调用三方通道API创建支付订单
  │         │     ├── 成功 → 获取支付链接/二维码地址
  │         │     └── 失败 → 返回通道错误
  │         └── 返回: {thirdPartyOrderNo, payUrl/qrCode, expireTime}
  │
  │  ④ 创建钱包侧充值订单（内部）
  ├──→ OrderService.CreateDepositOrder(userId, currencyId, amount,
  │         methodId, channelId, bonusActivityId, orderNo格式="C+16位")
  │
  └──→ 返回给用户: {orderNo, payUrl/qrCode, expireTime}
```

### 4.3 用户支付 + 回调阶段

```
用户在三方支付页面完成支付
  │
  ▼
三方通道 → 发送支付结果回调 → 财务管理模块
  │
  │  财务模块内部处理:
  ├── 验签、校验订单
  ├── 更新财务侧订单状态
  │
  │  跨模块回调（财务 → 钱包）:
  ├──→ [RPC] ser-wallet.Credit(userId, currencyId, centerType, actualAmount, orderNo)

  │         │  钱包内部处理:
  │         ├── 幂等校验（防重复上分）
  │         ├── WalletService.Credit → 中心钱包余额增加
  │         ├── OrderService.UpdateDepositStatus(orderNo, "已完成")

  │
  │  [如果选了奖金活动] 财务/活动系统额外调用:
  ├──→ [RPC] ser-wallet.CreditReward(userId, currencyId, rewardType, bonusAmount, orderNo)

  │         │  钱包内部处理:
  │         ├── WalletService.Credit → 奖励钱包余额增加

  │
  └── 充值流程完成
```

### 5.1 提现页面加载阶段

```
用户打开提现页面
  │

[C端] CheckWithdrawEligibility(currencyId)
  │
  │  ① KYC校验（跨服务）
  ├──→ [RPC] ser-kyc.GetKycStatus(userId)
  │         │

  │         └── 返回: {allowFiat: bool, allowUSDT: bool}
  │
  │  ③ 综合判断
  ├──→ 组合逻辑:
  │      · 法币/电子钱包提现 + 未KYC → 返回"需要先完成KYC"

  │      · 纯法币充值用户 → 只允许法币提现方式
  │      · 混合充值用户 → 只允许法币提现方式
  │
  └──→ 返回: {eligible: bool, allowedTypes: [...], kycRequired: bool}
```

```
[C端] GetWithdrawMethods(currencyId)
  │
  │  跨模块 RPC 调用:
  ├──→ [RPC] 财务模块.GetWithdrawMethods(currencyId)
  │         │
  │         │  财务模块内部处理:
  │         ├── 查支付配置表：筛选该币种下已启用的提现方式
  │         └── 返回: [{方式名称, 方式类型, icon, 排序值}]

### 5.2 提现提交阶段

```
用户选择提现方式 + 输入金额 + 填写收款信息 → 点击提交
  │
  ▼
[C端] CreateWithdrawOrder(currencyId, methodId, amount, receiverInfo)
  │
  │  ① 余额校验（内部）
  ├──→ WalletService.GetBalance(userId, currencyId, centerType)
  │         │
  │         └── 可用余额 < amount → 返回"余额不足"
  │
  │  ② 限额校验（内部）
  ├──→ OrderService.GetTodayWithdrawTotal(userId, currencyId)
  │         │
  │         ├── 单笔超限 → 返回"超过单笔限额"
  │         └── 今日累计超限 → 返回"超过每日限额"
  │
  │  ③ 资金来源规则校验（内部）
  ├──→ OrderService.AnalyzeDepositSources(userId)
  │         │
  │         └── 方式类型不匹配资金来源 → 返回"提现方式不符合规则"
  │
  │  ④ 冻结金额（内部）
  ├──→ WalletService.Freeze(userId, currencyId, amount)
  │         │
  │         └── 可用余额减少，冻结金额增加
  │
  │  ⑤ 创建钱包侧提现订单（内部）
  ├──→ OrderService.CreateWithdrawOrder(userId, currencyId, amount,
  │         methodId, receiverInfo, orderNo格式="T+16位")
  │
  │  ⑥ 提交到审核流程（跨模块）
  ├──→ [RPC] 财务模块.SubmitWithdrawRequest(orderNo, userId, currencyId,
  │         amount, methodId, receiverInfo)
  │         │
  │         │  财务模块内部处理:
  │         ├── 创建提现审核记录（状态：风控审核中）
  │         └── 返回: {financeOrderNo, status}
  │
  └──→ 返回给用户: {orderNo, status="审核中"}
```

```
                          财务管理模块内部

  │
  ├── 通过 → 状态变为"财务审核中"

  │     │          │
  │     │          ▼
  │     │   出款审核员审核
  │     │     │

  │     │     │          │
  │     │     │          ▼

  │
  └── 拒绝 → ⑧

### 5.4 审核结果回调钱包

    │         │
    │         │  钱包内部处理:
    │         ├── 幂等校验

## 十、调用链路的关键特征总结

### 10.1 调用频率分级

| 频率 | 接口 | 说明 |
|------|------|------|
| **高频** | GetWalletInfo, GetExchangeRate, QueryBalance | 用户每次打开钱包/财务查余额时调用 |
| **中频** | GetDepositMethods, GetWithdrawMethods, Records查询 | 用户进入充值/提现/记录页时调用 |
| **低频** | CreateDepositOrder, CreateWithdrawOrder, CreateExchangeOrder | 用户实际操作时才触发 |
| **极低频** | Credit, Debit, Freeze, Unfreeze | 充值成功/提现完成时才触发 |
| **定时** | SyncThirdPartyRates, CheckRateThreshold | 分钟级自动触发 |
| **极低频** | 人工修正相关 | 管理员手动操作，频率最低 |

### 10.2 调用链路复杂度排名

| 排名 | 业务场景 | 跨模块RPC次数 | 涉及的服务 | 复杂度评估 |
|------|----------|-------------|-----------|-----------|
| 1 | **充值** | 4~5次 | 钱包↔财务↔三方通道 | 最高（双向+异步回调） |
| 2 | **提现** | 4次 | 钱包↔财务↔KYC↔三方通道 | 高（双向+三级审核+异步） |
| 3 | **人工修正** | 2次 | 财务→钱包 | 中（单向，但有双人审核） |
| 4 | **兑换** | 0次 | 仅ser-wallet内部 | 低（完全自闭环） |
| 5 | **余额/记录查询** | 0次 | 仅ser-wallet内部 | 最低（纯读操作） |

## 十一、开发和联调的实操建议

### 11.1 可以独立开发（不等财务模块）的接口

第一批（约30个接口，占70%）：

```
币种配置全部 ──→ 11个接口 + 2个定时任务（完全独立）

钱包纯内部接口:
  GetWalletInfo          ──→ 只读余额
  SwitchCurrency         ──→ 只写当前币种
  GetExchangeRate        ──→ 内部调币种Service
  CreateExchangeOrder    ──→ 内部事务操作
  4个Records + Detail    ──→ 只读订单表
  CheckDuplicateDeposit  ──→ 只读订单表
  GetDepositOrderStatus  ──→ 只读订单表

钱包RPC接口（被动能力）:
  QueryBalance / Credit / Debit / Freeze / Unfreeze
  QueryWalletDetail / TransferToCenter / CreditReward
  ──→ 只操作自己的数据，不需要调外部
```

### 11.2 需要等财务模块联调的接口

第二批（约10个接口，占30%）：

```
充值相关:
  GetDepositMethods      ──→ 等财务提供 GetDepositMethods RPC
  GetDepositBonusList    ──→ 等活动系统（待确认来源）
  CreateDepositOrder     ──→ 等财务提供 MatchChannel + CreateChannelOrder
  充值回调处理            ──→ 等财务回调钱包 Credit

提现相关:
  CheckWithdrawEligibility ──→ 等 ser-kyc 提供 GetKycStatus
  GetWithdrawMethods       ──→ 等财务提供 GetWithdrawMethods
  CreateWithdrawOrder      ──→ 等财务提供 SubmitWithdrawRequest
  提现审核结果处理          ──→ 等财务回调钱包 Debit/Unfreeze
```

### 11.3 联调策略建议

```
阶段一: 独立开发 + Mock 联调
━━━━━━━━━━━━━━━━━━━━━━━━━━━
  · 开发所有独立接口（第一批30个）
  · 充值/提现接口的框架先写好，财务调用的部分用 Mock 返回
  · Mock 数据: 固定的充值方式列表、固定的通道匹配结果
  · 好处: 可以先跑通全流程，不阻塞

阶段二: 真实联调
━━━━━━━━━━━━━━━
  · 财务模块接口就绪后，替换 Mock 为真实 RPC 调用
  · 联调重点: 充值全链路 → 提现全链路 → 人工修正
  · 最关键的联调点: 充值回调上分(Credit)、提现审核扣款(Debit/Unfreeze)
  · 这两个回调是异步的，需要特别验证幂等性和并发安全
```
