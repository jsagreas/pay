## 第1章 工程概览与技术栈

### 1.1 项目定位

TG-Services Wallet-Server 是一个**专用于 TRON 链上虚拟币操作**的区块链钱包服务。与参考A（P9游戏内部流转钱包）完全不同，本项目直接对接真实区块链网络，处理链上资产的充值、提现、归集。

**核心价值**：这是我们能参考到的**唯一一份完整的区块链充值链路实现**，对 ser-wallet 未来接入 USDT 充值具有直接指导意义。

### 1.2 技术栈清单

| ORM | JPA + Hibernate | Spring默认 | 自动DDL |

## 第5章 区块链地址生成全链路（密钥管理核心）

### 5.1 地址生成流程

> 源码：`WalletTrxBusiness.createWallet()` L99-136

**完整流程**：

```
Step 1: 检查用户是否已有地址
    trxWalletService.findAddressByUserId(userId)
    → 已存在则直接返回，保证幂等

Step 2: 生成椭圆曲线密钥对
    ECKeyPair ecKeyPair = Keys.createEcKeyPair()
    → 使用 web3j 的 secp256k1 曲线生成
    → 产出：privateKey（256位） + publicKey

Step 3: 创建标准钱包文件
    WalletFile walletFile = Wallet.createStandard(passphrase, ecKeyPair)
    → 得到以太坊格式地址（40位hex，无前缀）

Step 4: 转换为 TRON 地址
    finalAddress = fromHexAddress("41" + originAddress)
    → "41"前缀 = TRON主网标识
    → Base58Check编码 → 得到 T 开头的TRON地址

Step 5: 提取私钥十六进制
    keyStore = Numeric.toHexStringNoPrefix(ecKeyPair.getPrivateKey())

Step 6: 加密私钥
    enKey = encrypt(keyStore, passphrase + finalAddress)
    → AES/ECB/PKCS5Padding
    → 密钥 = SHA1(passphrase + address) 取前16字节
    → passphrase 是通过 AWS KMS 解密的助记词

Step 7: 持久化
    trxWalletService.save(userId, userName, enKey, finalAddress)
    → 存储：userId + 加密私钥 + TRON地址

Step 8: 异步回写用户服务
    trxWalletService.updateUserAddress(trxUserWalletRequest)
    → @Async HTTP POST 到 user-server 更新用户钱包地址
```

### 5.2 地址格式转换

```
以太坊地址(hex40) → 加 "41" 前缀 → Base58Check编码 → TRON地址(T开头)
示例：0x1234...abcd → 411234...abcd → Base58Check → TLvmjV7uj8V2TZAQLrEDhduUzzwfoPCxy8
```

**TRON地址特征**：以 T 开头，34个字符，Base58编码

### 5.3 私钥加密方案详解

```
加密流程：
  原始私钥(hex) → AES加密 → Base64编码 → 存入DB

密钥派生：
  passphrase + address → UTF-8 bytes → SHA-1 → 取前16字节 → AES-128密钥

解密流程：
  DB.enPrivateKey → Base64解码 → AES解密 → 原始私钥(hex)
  其中 AES密钥 = SHA-1(passphrase + address)[0:16]
```

**安全层级**：
1. 第一层：AES对称加密（私钥→密文）
2. 第二层：AES密钥的 passphrase 通过 AWS KMS 加密存储在配置文件
3. 运行时：AWS KMS 解密 passphrase → SHA-1 派生 AES Key → AES 解密私钥

> **用户原话**："务必要区块链充值链路过程搞清楚"

### 6.1 充值全链路概览

TRC20（如USDT）充值不是传统的"调接口扣款"，而是**链上确认**模式。流程跨越 `user-server` 和 `wallet-server` 两个服务，涉及链上查询和异步处理。

### 6.3 Step①：获取充值地址

**触发**：用户首次充值时，前端调用 wallet-server

```
POST wallet/create
Body: { "userId": "123", "userName": "test" }
```

> 源码：`WalletController.createWallet()` → `WalletTrxBusiness.createWallet()`

**逻辑**：
1. 查询用户是否已有地址（幂等保护）
2. 无则生成新地址（详见第5章）
3. 返回 TRON 地址给前端展示

### 6.4 Step②：链外转账

这一步完全在链上发生，用户从交易所/其他钱包转 USDT 到展示的 TRON 地址。**wallet-server 不参与此步骤**。

### 6.5 Step③：用户确认充值

**触发**：用户在前端输入 txId（交易哈希）并点击确认

> 源码：`user-server/VirtualCurrencyController.java`

```java
@PostMapping("charge")
public ResponseEntity charge(@RequestBody SsOrderVirtualCurrencyChargeVO vo) {
    // Redisson公平锁，防止重复提交
    RLock rLock = redissonClient.getFairLock("USER_CHARGEVIRTUALCURRENCY_" + order.getId());
    if (rLock.tryLock(3, 5, TimeUnit.SECONDS)) {
        // 创建充值订单（状态=WAIT）
        // 异步触发验证
        asyncVirtualDepositService.checkCrypotcurrencyOrder(order);
    }
}
```

**关键设计**：
- Redisson公平锁按订单ID加锁，防止重复提交
- 订单创建后立即异步验证，不阻塞用户响应

### 6.6 Step④：异步验证链上余额

## 第7章 TRC20 转账底层实现（智能合约调用详解）

### 7.1 TRC20 转账流程概览

> 源码：`WalletTrxBusiness.trc20Transaction()` L209-290

TRC20转账本质是**调用代币智能合约的 transfer 方法**，流程分为：构造→打包→签名→广播→确认。

### 7.2 完整步骤分解

```
Step 1: 地址校验
    validateAddress(to) → POST TRON_NODE/wallet/validateaddress
    → 验证接收地址格式合法性（Base58Check格式）

Step 2: 构造智能合约调用参数
    map.put("contract_address", toHexAddress(tokenContract))  // USDT合约地址(hex)
    map.put("function_selector", "transfer(address,uint256)")  // ERC20标准方法签名
    map.put("owner_address", toHexAddress(from))               // 调用者地址(hex)
    map.put("call_value", 0)                                   // 不附带TRX
    map.put("fee_limit", 50000000L)                            // 最大手续费50TRX

Step 3: ABI参数编码
    BigInteger integer = needSendAmount.multiply(10^decimal).toBigInteger()
    String parameter = encoderAbi(toHexAddress(to).substring(2), integer)
    → Web3j FunctionEncoder: address + uint256 → 64字节hex参数

Step 4: 触发智能合约（获取未签名交易）
    POST TRON_NODE/wallet/triggersmartcontract
    → 返回 JSON { transaction: { raw_data: {...}, ... } }

Step 5: 打包交易
    transaction.raw_data.data = hex("trc20")  // 标记交易类型
    Protocol.Transaction tx = TronUtils.packTransaction(json, false)
    → Protobuf 序列化

Step 6: 签名（分支处理）
    if (出款地址 == dispensingAddress) {
        // 提现场景 → 调用外部加签系统
        bytes = sendSign(signUrl, tx, ssOrderId, from)
    } else {
        // 归集场景 → 本地私钥签名
        bytes = signTransaction2Byte(tx.toByteArray(), ByteArray.fromHexString(keyStore))
    }

Step 7: 广播已签名交易
    POST TRON_NODE/wallet/broadcasthex { transaction: signedTxHex }
    → 返回 { code: "SUCCESS", txid: "abc123..." }

Step 8: 返回交易哈希
    trxWalletResult.setTxId(txId)
```

### 7.3 签名逻辑详解

**本地签名**（归集用）：

```java
private static byte[] signTransaction2Byte(byte[] transaction, byte[] privateKey) {
    ECKey ecKey = ECKey.fromPrivate(privateKey);       // 从私钥恢复EC密钥
    Protocol.Transaction tx = Protocol.Transaction.parseFrom(transaction);
    byte[] rawdata = tx.getRawData().toByteArray();     // 取raw_data
    byte[] hash = Sha256Sm3Hash.hash(rawdata);          // SHA-256哈希
    byte[] sign = ecKey.sign(hash).toByteArray();       // ECDSA签名
    return tx.toBuilder()
        .addSignature(ByteString.copyFrom(sign))        // 附加签名
        .build().toByteArray();
}
```

**外部加签**（提现用）：
```java
private byte[] sendSign(String signUrl, Protocol.Transaction tx, String ssOrderId, String from) {
    byte[] txId = Sha256Sm3Hash.hash(tx.getRawData().toByteArray());
    String signTxid = Base64.getEncoder().encodeToString(txId);

    // 构造加签请求
    Map signMap = { orderNo, outAddressNo, chainType: "TRON", hashId: signTxid };

    // RSA加密请求体
    String enSign = RSAUtil.encrypt(JSON.toJSONString(signMap), rsaPublicKey);

    // 发送到加签系统
    POST signUrl/api/sign/signTransactionV2
    Header: PLAT_CODE = 平台编码
    Body: RSA加密后的请求

    // 返回签名值
    byte[] deSign = Base64.decode(response.data);

    // 组装完整交易
    return tx.toBuilder().addSignature(ByteString.copyFrom(deSign)).build().toByteArray();
}
```

### 7.4 ABI编码详解

```java
public String encoderAbi(String Address, BigInteger amount) {
    ArrayList<Type> inputParameter = new ArrayList<>();
    inputParameter.add(new org.web3j.abi.datatypes.Address(Address));  // 20字节地址
    inputParameter.add(new Uint256(amount));                            // 32字节金额
    return FunctionEncoder.encodeConstructor(inputParameter);
    // 输出：64字节hex = 32字节地址填充 + 32字节uint256金额
}
```

**示例**：转100 USDT给 TLvmjV7...
```
参数编码 =
  0000000000000000000000004141...abcd (地址填充到32字节)
  0000000000000000000000000000000000000000000000000000000005F5E100 (100*10^6)
```

## 第10章 资金归集/汇总全链路

### 10.1 什么是归集

归集（Collection/Sweep）是将分散在各用户独立地址上的资产，统一转移到公司主地址（冷钱包/热钱包）的过程。

**为什么需要归集**：
- 每个用户有独立链上地址，充值资金分散
- 集中到主地址便于资金管理和出款
- 降低热钱包被盗风险（用户地址余额清零）

### 10.2 单笔归集流程

> 源码：`TrxDriver.singleCollect()` L175-228

```

Step 2: 获取出款地址（加签系统配置）
    gasAddress = config.findByType("sign").get("DISPENDADDRESS")

Step 3: 查询用户地址余额（如果未传入金额）
    trxBalance = getAccountBalance(fromAddress, coinName)

Step 4: 余额检查
    if (trxBalance <= 0) return null  // 余额不足不归集

Step 5: Gas补充（TRC20专用）
    if (coinName != TRX) {
        trxBalance_of_user = getAccountBalance(fromAddress, "TRX")
        if (trxBalance_of_user < 30 TRX) {
            // 从Gas地址转30TRX到用户地址（支付TRC20转账手续费）

        }
    }

Step 6: 执行归集转账

        null,           // 私钥从DB解密

Step 7: 记录归集信息

    → 写入 trx_wallet_collect 表
    → 写入 TrxWalletOrder 表（order_type=RECHARGE）
```

### 10.3 批量归集（定时任务）

## 第11章 提现全链路

### 11.1 提现流程概览

提现是充值的逆向操作：平台内部扣款 → 链上转账 → 确认。

### 11.2 提现完整流程

> 源码：`user-server/VirtualCurrencyWithdrawBusiness.withdraw()` + `AsyncVirtualWithdrawService.withdrawUSDT()`

```
Step 1: 用户发起提现请求
    POST /virtual/withdraw { orderId }
    → Redisson公平锁(orderId)
    → 检查订单状态、余额
    → 冻结提现金额 doFreezeAmount(userId, amount, FREEZE)

Step 2: 异步处理提现
    @Async withdrawVirtualCurrencyTask()
    → 获取用户提现地址
    → 调用 wallet-server 发起链上转账

Step 3: wallet-server 执行提现转账
    POST wallet/withdrawTraction {
        userId, toAddress, amount, coinName, ssOrderId, postType
    }

    TrxDriver.withdrawTraction():
    a) 幂等检查：查询Order表是否已存在该ssOrderId
       → 已存在则直接返回txId
    b) 余额检查：查询出款地址链上余额是否足够
    c) Gas补充：TRC20需要TRX手续费，不足则补充
    d) 执行转账：使用外部加签系统签名
       → createTransactionForTrx(fromAddress=dispensingAddress)
    e) 记录订单：save到TrxWalletOrder表

Step 4: user-server 确认到账
    → 每2分钟定时任务轮询
    → POST wallet/getTransactionInfoById 查询交易状态
    → 交易确认后：
       a) 解冻金额 doFreezeAmount(userId, amount, UNFREEZE)
       b) 扣减真实余额 reduceRealAmount(order)
       c) 扣减手续费 doBalanceFeeBusiness(order)
       d) 更新订单状态 → SUCCESS
       e) 发送MQ统计通知
```

### 11.3 提现与充值的关键区别

| 维度 | 充值 | 提现 |
|------|------|------|
| 资金方向 | 链上→平台 | 平台→链上 |
| 签名方式 | 无需签名(只是查询) | **外部加签系统**（安全隔离） |
| 出款地址 | 用户独立地址 | **公司出款地址**(dispensingAddress) |
| 余额变更 | addBalance | freezeAmount → reduceRealAmount |
| 安全级别 | 低（只需验证） | 高（冻结+加签+确认） |


**AWS KMS 的价值**：
- 密钥从不以明文出现在磁盘上
- 加解密操作有审计日志
- 密钥轮换由AWS管理
- 即使服务器被入侵，配置文件中的密文无法直接使用

### 13.3 AES加密参数

| 参数 | 值 | 说明 |
|------|------|------|
| 算法 | AES | 对称加密 |
| 模式 | ECB | **安全性较弱，不推荐** |
| 填充 | PKCS5Padding | 标准填充 |
| 密钥长度 | 128位 | SHA-1取前16字节 |
| 密钥派生 | SHA-1(secret) | **不够安全，应用PBKDF2** |

## 第14章 外部加签系统（提现安全隔离）

### 14.1 设计思想

提现操作涉及资金外流，安全要求最高。本项目将**签名操作从 wallet-server 隔离到独立的加签系统**：

```
wallet-server (构造交易) → RSA加密请求 → 加签系统(签名) → 返回签名值 → 广播
```

### 14.2 加签流程

> 源码：`WalletTrxBusiness.sendSign()` L292-321

```
1. 计算交易哈希
   txId = SHA256(transaction.rawData)
   signTxid = Base64(txId)

2. 构造请求
   { orderNo, outAddressNo, chainType: "TRON", hashId: signTxid }

3. RSA加密
   enSign = RSAUtil.encrypt(JSON.toJSONString(signMap), rsaPublicKey)
   rsaPublicKey 来自 TrxWalletConfig表 (type="sign", code="RSA")

4. 发送请求
   POST signUrl/api/sign/signTransactionV2
   Header: PLAT_CODE = 平台编码
   Header: Content-Type = application/json
   Body: RSA加密后的字符串
   Timeout: 60秒

5. 获取签名
   response.code == 200 → Base64.decode(response.data) → 签名字节

6. 组装完整交易
   transaction.addSignature(签名) → 完整可广播交易
```

### 14.3 安全评价

| 优点 | 说明 |
|------|------|
| 签名隔离 | 出款私钥不在wallet-server内存中 |
| 请求加密 | RSA加密防篡改 |
| 平台标识 | PLAT_CODE头防伪造 |
| 订单关联 | orderNo可审计追踪 |

| 不足 | 说明 |
|------|------|
| 无双向认证 | wallet-server到加签系统的通信未用mTLS |
| 签名结果无验证 | 拿到签名后直接广播，未验证签名有效性 |
| 单点故障 | 加签系统不可用则所有提现瘫痪 |

## 第18章 精华清单（值得借鉴 G01-G16）

### G01. 链上余额实时查询，不在本地存储余额

**出处**：`TrxWalletEntity` 无 amount 字段
**做法**：每次查询余额都直接调用链上RPC
**价值**：避免链上余额与本地余额不一致的问题
**借鉴方式**：对于链上资产，以链为准；平台内部账本可以缓存但需定期对账

### G02. 地址生成幂等保护

**出处**：`WalletTrxBusiness.createWallet()` L110-115
**做法**：`findAddressByUserId(userId)` 先查后创建
**价值**：避免一个用户生成多个地址导致资金分散
**借鉴方式**：ser-wallet 地址生成也必须幂等（数据库 UNIQUE 更佳）

### G03. 私钥双层加密（AES + AWS KMS）

**出处**：`WalletTrxBusiness.createWallet()` L123 + `AwsKmsDriver`
**做法**：私钥→AES→DB，AES密钥的种子→KMS→配置文件
**价值**：即使DB泄露或配置文件泄露，单层泄露无法解密私钥
**借鉴方式**：必须借鉴，ser-wallet 可用 Vault 替代 AWS KMS

### G04. 签名隔离——提现专用外部加签系统

**出处**：`WalletTrxBusiness.sendSign()` + `trc20Transaction()` L248
**做法**：归集=本地签名，提现=外部加签系统
**价值**：出款私钥不在wallet-server内存中，降低被盗风险
**借鉴方式**：高度推荐。ser-wallet 提现应采用独立签名服务

### G05. Gas自动补充机制

**出处**：`TrxDriver.singleCollect()` L192-206
**做法**：TRC20操作前自动检查TRX余额，不足则从Gas地址补充
**价值**：TRC20转账需要TRX作Gas，自动补充避免操作卡死
**借鉴方式**：直接复用此模式，阈值可配置化

### G06. 提现幂等——订单号查重

**出处**：`TrxDriver.withdrawTraction()` L298-303
**做法**：`findBySsOrderId(ssOrderId)` 已存在则返回已有txId
**价值**：防止重复提现（重复链上转账=资金损失）
**借鉴方式**：借鉴思路，但应改为 UNIQUE 约束 + 数据库层面保证

### G07. 冻结先行的提现模式

**出处**：`user-server/VirtualCurrencyWithdrawBusiness.withdraw()`
**做法**：提现请求→冻结金额→异步链上转账→确认后扣款
**价值**：防止提现期间余额被其他操作消耗
**借鉴方式**：与核心方案/d-3.md C6 完全一致，必须采用

### G08. 定时任务兜底轮询

**出处**：`HandleVirtualCurrencyTask` 每2分钟
**做法**：轮询所有WAIT状态订单，重新检查链上状态
**价值**：异步回调可能丢失，定时轮询是最终一致性保障
**借鉴方式**：必须采用。ser-wallet 可用 NATS JetStream 替代 RabbitMQ + 定时轮询兜底

### G09. 合约枚举化管理

**出处**：`TRXContractEnum`
**做法**：代币合约地址、精度、名称用枚举集中管理
**价值**：避免散落在代码各处的魔法值
**借鉴方式**：ser-wallet 应在数据库配置表中管理，枚举仅做类型约束

### G10. ABI编解码使用web3j成熟库

**出处**：`Trc20.java`、`WalletTrxBusiness.encoderAbi()`
**做法**：使用 web3j 的 FunctionEncoder/FunctionReturnDecoder
**价值**：避免手动编解码出错（ABI规范复杂）
**借鉴方式**：Go生态对应库为 `go-ethereum/accounts/abi`

### G11. 地址验证前置

**出处**：`WalletTrxBusiness.validateAddress()` + `trc20Transaction()` L217
**做法**：转账前先调用 TRON 节点验证地址格式
**价值**：避免向无效地址转账导致资金永久丢失
**借鉴方式**：必须借鉴，ser-wallet 所有涉及地址的操作都应前置验证

### G12. 交易状态链上确认

**出处**：`WalletTrxBusiness.getTransactionInfoById()`
**做法**：通过 txId 查询链上交易，检查 `contractRet == "SUCCESS"`
**价值**：广播成功不等于交易成功，必须链上确认
**借鉴方式**：必须借鉴。还应增加区块确认数检查（如等待20个区块）

### G13. 归集两种模式（即时+定时）

**出处**：`CollectTypeEnum` + `TrxDriver`
**做法**：充值后即时归集 + 每日定时批量归集
**价值**：即时归集保安全，定时归集扫漏网之鱼
**借鉴方式**：双模式都需要，ser-wallet 可配置归集阈值

### G14. TRON地址格式转换封装

**出处**：`WalletTrxBusiness.toHexAddress()`, `fromHexAddress()`, `Base58`
**做法**：统一封装 T地址(Base58) ↔ 41hex地址 的转换
**价值**：TRON RPC用hex格式，用户端用T格式，转换频繁
**借鉴方式**：Go实现需封装类似工具函数

### G15. 归集/提现Gas记录入订单表

**出处**：`TrxDriver.singleCollect()` L203-204
**做法**：Gas补充转账也记录到TrxWalletOrder表
**价值**：每笔链上交易都有记录，便于对账审计
**借鉴方式**：必须借鉴。Gas消耗也是成本，需要完整记录

### G16. 配置表管理动态参数

**出处**：`TrxWalletConfig` 表
**做法**：出款地址、RSA公钥、平台编码等存DB而非硬编码
**价值**：运行时可调整，无需重启服务
**借鉴方式**：ser-wallet 币种配置模块已有此设计，可复用

## 第19章 糟粕清单（需要规避 B01-B14）

### B01. AES/ECB模式不安全

**出处**：`WalletTrxBusiness.encrypt()` L185
**代码**：`Cipher.getInstance("AES/ECB/PKCS5Padding")`
**问题**：ECB模式相同明文产生相同密文，存在模式泄露风险
**我们应**：使用 AES/GCM 或 AES/CBC+随机IV

### B02. SHA-1密钥派生不安全

**出处**：`WalletTrxBusiness.getKey()` L197-202
**代码**：`MessageDigest.getInstance("SHA-1")` + `Arrays.copyOf(digest, 16)`
**问题**：SHA-1已被攻破，且无盐值、无迭代，容易暴力破解
**我们应**：使用 PBKDF2 或 Argon2 进行密钥派生

### B03. 订单表无UNIQUE约束

**出处**：`TrxWalletOrderEntity` 表定义
**问题**：`ss_order_id` 无 UNIQUE 索引，并发提现可能插入重复记录
**我们应**：对 (ss_order_id) 或 (biz_type, order_no) 加 UNIQUE 约束

### B04. 订单表无状态字段

**出处**：`TrxWalletOrderEntity` 表定义
**问题**：没有 status 字段，无法标记 pending/success/failed
**我们应**：订单必须有完整的状态机（参考核心方案 C7）

### B05. 金额精度 decimal(16,2) 不够

**出处**：`TrxWalletOrderEntity.amount` + `TrxWalletCollectEntity.amount`
**问题**：USDT精度6位、CXT精度18位，存2位小数会丢失精度
**我们应**：使用 DECIMAL(36,18) 或至少 DECIMAL(20,8)

### B06. Driver与Business职责边界模糊

**出处**：`TrxDriver` 做业务编排（Gas补充、余额检查、归集循环）
**问题**：Driver 名为"驱动"却做了业务编排，Business 名为"业务"却做底层链上操作
**我们应**：清晰分层：Service（业务编排）→ Client（链上操作）→ Utils（工具）

### B07. 所有跨服务调用无重试

**出处**：全项目 HTTP 调用
**问题**：网络抖动、瞬时超时会导致操作静默失败
**我们应**：关键操作（链上广播、余额查询）需要指数退避重试

### B08. 异常被吞——catch后无补偿

**出处**：`WalletTrxBusiness.trc20Transaction()` L286-288
**代码**：`catch (Exception e) { log.error("代币转账失败", e); } return null;`
**问题**：转账失败仅打日志，调用方收到null后可能未做正确处理
**我们应**：抛出明确异常 + 调用方做补偿处理

### B09. 批量归集全量遍历无优化

**出处**：`TrxDriver.collectTrc20()` L241
**代码**：`trxWalletService.findAll()` 遍历所有钱包
**问题**：大量用户余额为0也会逐一查询链上余额，浪费Gas和RPC配额
**我们应**：先链上批量查询余额 > 0 的地址，再归集

### B10. 配置中硬编码私钥到application.yml

**出处**：`wallet.trx.gasPrivateKey` 等配置
**问题**：虽然KMS加密了，但密文在git仓库中，环境间配置共享
**我们应**：使用环境变量或 Vault 动态注入，不落配置文件

### B11. 创建KmsClient未复用

**出处**：`AwsKmsDriver.encryptData()` + `decryptData()` 每次调用都 `createKmsClient()`
**问题**：每次加解密都创建新的KMS客户端，建立新的HTTPS连接
**我们应**：单例复用KmsClient（或连接池），初始化一次

### B12. TRON地址硬编码在枚举

**出处**：`TRXContractEnum.CXT` 硬编码合约地址
**问题**：更换合约需要改代码重新部署
**我们应**：合约地址存数据库配置表，运行时读取

### B13. 无链上确认数检查

**出处**：`getTransactionInfoById()` 仅检查 contractRet==SUCCESS
**问题**：TRON虽然是DPOS（出块快、确定性高），但最安全做法仍需等待一定区块确认
**我们应**：至少等待19个区块（1个超级代表轮次）确认

### B14. 日志中泄露敏感信息

**出处**：`WalletTrxBusiness.createWallet()` 打印地址信息
**问题**：生产日志可能泄露用户地址关联关系
**我们应**：地址脱敏打印（只打前6位+后4位）

## 第20章 Bug 级问题（D01-D06）

### D01. 地址生成非原子操作——竞态条件

**出处**：`WalletTrxBusiness.createWallet()` L110-127

```java
// L110: 先查
String address = trxWalletService.findAddressByUserId(userId);
if(StringUtils.isNotBlank(address)) return ...;  // 已存在

// L116-127: 后创建（中间无锁！）
ECKeyPair ecKeyPair = Keys.createEcKeyPair();
...
trxWalletService.save(userId, userName, enKey, finalAddress);
```

**Bug**：两个并发请求同时到达，都通过 L110 检查（都为空），然后各生成一个地址，导致同一用户两个地址，其中一个成为孤儿地址（有资产但无法归集）。

**修复方案**：
- 方案A：userId 加分布式锁（Redisson）
- 方案B：DB UNIQUE(userId) 约束 + 乐观失败重试

### D02. Gas补充后无等待——归集交易可能失败

**出处**：`TrxDriver.singleCollect()` L196-205

```java
// 补充Gas
createTransactionForTrx(gasPassword, 30TRX, gasAddress, fromAddress, TRX, ssOrderId);
// 立即执行归集（不等Gas到账！）
trxWalletResult = createTransactionForTrx(null, trxBalance, fromAddress, mainAddress, ...);
```

**Bug**：Gas补充的TRX转账需要链上确认（约3-6秒），但代码补完Gas后立即执行归集转账。此时Gas可能还未到账，导致归集因Energy不足失败。

**修复方案**：Gas补充后应等待确认（轮询链上状态）或延迟执行归集。

### D03. 提现createTransactionForTrx第一个参数传错

**出处**：`TrxDriver.withdrawTraction()` L329

```java
result = createTransactionForTrx(fromAddress, sendAmount, fromAddress, toAddress, tokenContract, ssOrderId);
//                               ↑ 这里传了fromAddress作为privateKey！
```

**对比正确调用**：`createTransactionForTrx(null, ...)` 或 `createTransactionForTrx(privateKey, ...)`

**Bug**：`fromAddress`（地址字符串）被传入 `privateKey` 参数位，但因为提现走外部加签（不需要本地私钥），且 `createTransactionForTrx` 内部当 `privateKey` 非空时直接使用，所以在TRC20提现场景：进入 `trc20Transaction` 后，签名走 `sendSign`（外部加签），`keyStore` 参数被忽略，实际不会报错。但如果代码逻辑变更（如归集改提现出款地址），会导致签名失败。

**风险等级**：中（当前不触发，但属于隐患代码）


**Bug**：生产代码中保留了测试 main 方法，包含助记词明文和加密私钥。任何能访问源码的人可以运行此方法解密私钥。

**严重性**：高（安全漏洞，如果地址中仍有资金则可直接被盗）

### D05. getTrxWalletAccount接口暴露加密能力

**出处**：`WalletController.getTrxWalletAccount()` L120-140

```java
@PostMapping("getTrxWalletAccount")
public ResponseEntity<TrxWalletAccountVO> getTrxWalletAccount(...) {
    trxWalletAccountVO.setMainAddress(awsKmsDriver.encryptData(request.getMainAddress()));
    trxWalletAccountVO.setCollectAddress(awsKmsDriver.encryptData(request.getCollectAddress()));
    ...
}
```

**Bug**：此接口接收明文地址和私钥，返回KMS加密后的密文。任何能调用此接口的人都能利用它加密任意数据（Oracle攻击向量）。且没有任何鉴权保护。

**严重性**：高（应作为内部初始化工具，不应暴露为HTTP API）

### D06. 归集CompanyAddress未解密就用于记录

**出处**：`TrxDriver.reverseNoticeCollectDetail()` L262

```java
walletTrxBusiness.reverseNoticeCollectDetail(userId, fromAddress, companyAddress, ...);
//                                                                 ↑ KMS密文！
```

**对比**：`singleCollect()` L179 中 `awsKmsDriver.decryptData(companyAddress)` 解密后用于链上操作，但 `reverseNoticeCollectDetail` 传入的是原始密文字段。

**Bug**：归集记录表中存储的 toAddress 是 KMS 密文而非明文地址，导致对账时无法直接识别。

### 21.1 架构映射

| TG-Services (Java) | ser-wallet (Go) | 映射策略 |

| RabbitMQ | NATS JetStream | 消息队列 |
| JPA | GORM Gen | ORM |
| Redisson | go-redis | 分布式锁 |

### 21.2 充值链路映射

```
TG-Services 充值链路：
  用户提交txId → user-server创建订单 → @Async验证 → 轮询链上 → 入账 → 归集

ser-wallet 充值链路（建议设计）：
  用户提交txId → order-service创建订单(NATS) → ser-wallet.chain监听
  → 链上验证(gotron-sdk/gRPC) → 入账(GORM事务+FOR UPDATE) → 归集(NATS延迟消息)
  → 定时任务兜底(每2分钟)
```

### 21.3 必须改进的设计点

| # | TG原设计 | ser-wallet改进 |
|---|----------|----------------|
| 1 | HTTP轮询验证 | gRPC流式监听 + HTTP轮询兜底 |
| 2 | AES/ECB | AES/GCM + Vault托管 |
| 3 | SHA-1密钥派生 | PBKDF2/Argon2 |
| 4 | 无UNIQUE约束 | UNIQUE(chain, tx_hash) |
| 5 | decimal(16,2) | DECIMAL(36,18) |
| 6 | 无状态机 | 完整状态机(PENDING→CONFIRMING→SUCCESS→COLLECTED) |
| 7 | 无区块确认数 | 可配置确认数（TRON≥19块，ETH≥12块） |
| 8 | 全量归集 | 余额>阈值才归集 |
| 9 | 无重试 | 指数退避重试+死信队列 |
| 10 | 单链硬编码 | 多链抽象（Chain接口，TRON/ETH/BSC实现） |

### 21.4 可直接复用的设计模式

| # | 模式 | 说明 |
|---|------|------|
| 1 | 链上余额为准 | 不在本地存链上余额 |
| 2 | 冻结先行提现 | freeze → 链上转 → confirm → deduct |
| 3 | Gas自动补充 | TRC20操作前自动补TRX |
| 4 | 签名服务隔离 | 提现签名与钱包服务分离 |
| 5 | 即时+定时归集 | 双模式归集策略 |
| 6 | 2分钟轮询兜底 | 最终一致性保障 |
| 7 | 30分钟超时取消 | 防止订单永久挂起 |
