## 1. 工程全景概述

### 1.1 工程定位

tg-services 是一个基于 Telegram Bot 的泛娱乐平台后端系统，包含 18+ 个微服务模块。其中 `wallet-server` 是**专门的 TRON 区块链钱包服务**，负责：

- 为每个用户生成独立的 TRON 链上地址
- 监控用户地址上的 TRC20 代币余额
- 将用户地址上的资产归集到公司主地址
- 从出款地址向用户外部地址执行提现
- 查询链上交易状态

### 1.2 与参考A的本质区别

| 维度 | 参考A (p9-services wallet-service) | 参考B (tg-services wallet-server) |
|------|-----------------------------------|-----------------------------------|
| **钱包模型** | 中心化余额钱包（DB记账） | 区块链链上钱包（每用户独立地址） |
| **资产存储** | 数据库 user_coin 表 | TRON 区块链 + DB 记录 |
| **充值方式** | 内部消息队列触发加币 | 用户向链上地址转入 → 归集到主地址 |
| **提现方式** | 内部扣币 + 通知商户 | 从出款地址发起链上真实转账 |
| **核心挑战** | 并发余额一致性 | 私钥安全 + 链上交易可靠性 + Gas费管理 |
| **技术复杂度** | 中等（传统CRUD） | 高（密码学 + 区块链协议 + 外部签名） |

## 2. 技术栈选型分析

### 2.1 核心技术栈

| 层次 | 技术选择 | 版本/说明 |
|------|---------|----------|
| **框架** | Spring Boot | 2.5.3 |
| **ORM** | Spring Data JPA + Hibernate | ddl-auto: update |
| **数据库** | MySQL 5 (InnoDB) | cloud_wallet 库 |
| **缓存** | Redis (Lettuce) | 6.x 兼容 |
| **消息队列** | RabbitMQ | 备用，wallet-server 本身未大量使用 |
| **区块链SDK** | tron-sdk + web3j | TRON Java SDK + 以太坊 ABI 工具 |
| **密码学** | BouncyCastle + AES + RSA | 多层加密体系 |
| **云服务** | AWS KMS | AP_SOUTHEAST_1 区域 |
| **HTTP客户端** | hutool HttpRequest + RestTemplate | 链上RPC调用 |
| **API文档** | Swagger 2 | 开发环境启用 |
| **连接池** | HikariCP | 4-16 连接 |
| **安全** | Spring Security Basic Auth | baisha/baisha |

### 2.2 关键依赖库

```
org.tron.common.crypto.ECKey          — TRON 椭圆曲线签名
org.tron.protos.Protocol.Transaction  — TRON 协议 protobuf 定义
org.web3j.crypto.ECKeyPair            — 以太坊兼容密钥对生成
org.web3j.crypto.Wallet               — 以太坊兼容钱包文件
org.web3j.abi.FunctionEncoder         — Solidity ABI 编码器
org.web3j.abi.datatypes.*             — ABI 数据类型
com.google.protobuf.ByteString        — protobuf 字节处理
software.amazon.awssdk.services.kms   — AWS KMS SDK
```

### 2.3 选型评价

**合理之处：**
- web3j 用于 ABI 编码是成熟选择，TRON 兼容以太坊的 ABI 标准
- AWS KMS 作为系统级密钥管理是企业级最佳实践
- 外部签名服务分离了签名权限，是安全设计的正确方向

**值得商榷：**
- JPA ddl-auto: update 不适合生产环境（应使用 Flyway/Liquibase）
- hutool HttpRequest 混合 RestTemplate 使用，HTTP 客户端不统一
- Spring Security Basic Auth 过于简陋（服务间应使用 mTLS 或 JWT）
- 缺少链路追踪（无 Sleuth/Zipkin/SkyWalking 集成）

### 3.2 分层职责矩阵

| 层次 | 职责 | 是否包含业务逻辑 | 是否包含事务 |
|------|------|:---:|:---:|
| Controller | 参数接收、响应包装 | 轻量路由 | 否 |
| Driver | 流程编排、条件判断 | **是（主要）** | 否 |
| Business | 底层实现、链上交互 | **是（核心）** | 否 |
| Service | CRUD + 数据访问 | 否 | **是（@Transactional）** |
| Repository | SQL执行 | 否 | 继承Service事务 |

### 3.3 架构特征

**关键设计决策：**
1. **Driver 与 Business 的分工**：TrxDriver 负责流程编排（归集、提现的步骤），WalletTrxBusiness 负责底层实现（具体怎么和链交互）
2. **无分布式锁**：wallet-server 内部没有使用 Redisson 或其他分布式锁（与参考A形成对比）
3. **幂等依赖查询**：提现通过 `findBySsOrderId` 查询已有订单来实现幂等，而非唯一索引
4. **异步回调 user-server**：钱包创建后通过 `@Async` 异步通知 user-server 更新地址

### 4.2 模型说明

**trx_wallet（用户钱包表）** — 一个用户一条记录，存储链上地址和加密私钥
- 核心：用户 → 地址 的映射
- 安全：只存加密后的私钥（enPrivateKey）

**trx_wallet_order（交易订单表）** — 所有链上交易的记录
- 充值归集、提现、Gas费补充等所有交易都记录
- ssOrderId 关联 user-server 的 SsOrder

**trx_wallet_collect（归集记录表）** — 专门记录归集操作
- 区分单笔即时归集和定时批量归集
- 独立于 order 表，提供归集维度的查询视角

**trx_wallet_config（动态配置表）** — 运行时可变配置
- 存储签名服务相关配置：出款地址、RSA公钥、平台编码等
- type='sign' 的配置项用于外部签名服务交互

### 5.2 关键设计点

**Assets 表的乐观锁：**
```java
@Version
private Integer version;
```
- 所有余额修改操作通过 `version` 字段实现乐观锁
- JPA 的 `@Version` 注解自动在 UPDATE 时追加 `WHERE version = ?`
- 如果并发冲突，抛出 `OptimisticLockException`

**BalanceChange 的唯一约束（幂等核心）：**
```java
@Table(uniqueConstraints = {
    @UniqueConstraint(columnNames = {"change_type", "relate_id", "user_id", "order_type"})
})
```
- 数据库级别保证同一笔业务操作只能产生一条余额变动
- 即使应用层重试，也不会导致重复加减款

**三种余额类型：**
- `balance`：主余额
- `balanceNn`：牛牛游戏余额
- `balanceBacc`：百家乐余额
- 各自独立增减，通过业务类型路由

### 8.1 四类地址角色

这是整个区块链钱包的核心架构设计——通过角色分离实现资金安全：

```
┌─────────────────────────────────────────────────────────────────┐
│                        多地址钱包架构                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌──────────────┐    用户充值          ┌──────────────────┐     │
│  │ 用户地址 (N个) │ ←────────────────── │ 用户链上转入       │     │
│  │ per-user生成  │                     └──────────────────┘     │
│  │ 私钥加密存DB  │                                              │
│  └──────┬───────┘                                              │
│         │ 归集（转出到公司地址）                                   │
│         ▼                                                       │

│  └──────────────┘                                              │
│                                                                 │
│  ┌──────────────┐    提现（转出到用户外部地址）                     │
│  │ 出款地址       │ ────────────────→ 用户提供的外部地址             │

│  │ 配置在DB     │   私钥不在wallet-server                        │
│  └──────────────┘                                              │
│                                                                 │

│  │ Gas 手续费地址 │ ────→ 补充TRX到用户地址（归集前）                │
│  │ (Gas Addr)   │ ────→ 补充TRX到出款地址（提现前）                │
│  │ 私钥KMS加密   │                                              │
│  └──────────────┘                                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 8.2 地址配置方式

| 地址角色 | 配置位置 | 加密方式 | 私钥管理 |
|---------|---------|---------|---------|
| 用户地址 | DB trx_wallet 表 | AES(key, passphrase+address) | 本地解密 |
| 公司归集地址 | application.yml companyAddress | AWS KMS | KMS解密 |
| 归集助记词/密码 | application.yml mainCollectPassphrase | AWS KMS | KMS解密后用于AES解密用户私钥 |
| 出款地址 | DB trx_wallet_config (code=disPendAddress) | 明文 | **外部签名服务持有私钥** |
| Gas地址 | application.yml gasAddress | AWS KMS | KMS解密 |
| Gas私钥 | application.yml gasPrivateKey | AWS KMS | KMS解密 |

### 8.3 安全设计的核心思想

**三层安全分离：**

1. **用户私钥** — AES加密存DB，解密需要 `passphrase + address`，passphrase 存 KMS
2. **公司/Gas私钥** — 直接由 KMS 加密保护，不以明文形式出现
3. **出款私钥** — **不在 wallet-server 中**，由独立的外部签名服务持有

**设计意图：** 即使 wallet-server 被攻破，攻击者也无法直接动用出款地址的资金（因为私钥在外部签名服务）。归集地址的私钥虽然可以通过 KMS 解密获取，但归集操作是把资金汇入公司主地址，风险可控。

## 9. 私钥管理与加密体系

### 9.1 用户私钥加密流程

```
生成密钥对:
  ECKeyPair ecKeyPair = Keys.createEcKeyPair();
  ↓
提取私钥(hex):
  keyStore = Numeric.toHexStringNoPrefix(ecKeyPair.getPrivateKey());
  ↓
AES加密:
  enKey = encrypt(keyStore, passphrase + finalAddress);
  ↓
存入DB:
  trxWalletService.save(userId, userName, enKey, finalAddress);
```

### 9.2 AES 加密实现细节

```java
// 算法：AES/ECB/PKCS5Padding
// 密钥派生：SHA-1(secret) → 取前16字节

public static String encrypt(String strToEncrypt, String secret) throws Exception {
    // 1. 密钥派生
    byte[] bytes = secret.getBytes("UTF-8");      // secret = passphrase + address
    MessageDigest sha = MessageDigest.getInstance("SHA-1");
    byte[] digest = sha.digest(bytes);
    byte[] keyBytes = Arrays.copyOf(digest, 16);   // 取前16字节 = AES-128

    // 2. AES加密
    SecretKeySpec secretKeySpec = new SecretKeySpec(keyBytes, "AES");
    Cipher cipher = Cipher.getInstance("AES/ECB/PKCS5Padding");
    cipher.init(Cipher.ENCRYPT_MODE, secretKeySpec);

    // 3. Base64编码输出
    return Base64.getEncoder().encodeToString(cipher.doFinal(strToEncrypt.getBytes("UTF-8")));
}
```

**密钥组成：** `secret = passphrase + address`
- `passphrase`：通过 AWS KMS 解密获取的助记词（系统级密钥）
- `address`：用户的 TRON 地址（每个用户不同 → 每个用户的 AES 密钥不同）

### 9.3 私钥解密流程（用于归集时签名）

```java
// TrxWalletService.getPrivateKeyByAddress()
public String getPrivateKeyByAddress(String address, String password) {
    // 1. 从DB读取加密私钥
    String keyStore = trxWalletRepository.findEnPrivateKeyByAddress(address).getEnPrivateKey();
    // 2. AES解密：password = KMS解密后的passphrase
    keyStore = AES.decrypt(keyStore, password + address);
    return keyStore;  // 返回明文hex私钥
}
```

### 9.4 安全评价

**优点：**
- 每个用户使用不同的 AES 密钥（passphrase + address），单个密文泄露不影响其他
- 系统级 passphrase 托管在 AWS KMS，不以明文存储
- 出款地址私钥完全不在本系统中

**缺陷：**
- AES/ECB 模式不安全（ECB 模式的固有缺陷：相同明文产生相同密文）
- 应使用 AES/CBC 或 AES/GCM 模式
- SHA-1 已被认为不够安全（NIST 2011年弃用），应使用 SHA-256
- 密钥只取 SHA-1 的前 16 字节 = AES-128，强度偏低（应 AES-256）
- 明文私钥在内存中处理，无 secure wipe（内存清除）机制

## 10. AWS KMS 系统级密钥管理

### 10.1 KMS 使用场景

| 操作 | 加密/解密 | 保护对象 |
|------|----------|---------|
| 钱包创建 | 解密 | passphrase（用于AES加密用户私钥的组成部分） |
| 归集操作 | 解密 | companyAddress（公司归集目标地址） |
| 归集操作 | 解密 | collectPassword（用于解密用户私钥） |
| 归集/提现 Gas | 解密 | gasAddress（Gas地址） |
| 归集/提现 Gas | 解密 | gasPassword（Gas私钥） |
| 账户配置 | 加密 | 管理端传入的各类地址和私钥 → KMS加密后存配置 |

### 10.2 KMS 实现

```java
@Component
public class AwsKmsDriver {
    @Value("${aws.accessKeyId}")
    private String accessKeyId;
    @Value("${aws.secretAccessKey}")
    private String secretAccessKey;
    @Value("${aws.keyId}")
    private String keyId;

    // 区域：新加坡 (AP_SOUTHEAST_1)
    // SDK：software.amazon.awssdk.services.kms
    // 每次操作创建新的 KmsClient → 用完关闭

    public String encryptData(String noEncryptData) {
        KmsClient kmsClient = createKmsClient();
        EncryptRequest request = EncryptRequest.builder()
            .keyId(keyId)
            .plaintext(SdkBytes.fromByteArray(noEncryptData.getBytes()))
            .build();
        EncryptResponse response = kmsClient.encrypt(request);
        return Base64.getEncoder().encodeToString(
            response.ciphertextBlob().asByteArray());
    }

    public String decryptData(String afterEncryptionData) {
        KmsClient kmsClient = createKmsClient();
        DecryptRequest request = DecryptRequest.builder()
            .ciphertextBlob(SdkBytes.fromByteArray(
                Base64.getDecoder().decode(afterEncryptionData)))
            .keyId(keyId)
            .build();
        DecryptResponse response = kmsClient.decrypt(request);
        return new String(response.plaintext().asByteArray());
    }
}
```

### 10.3 KMS 使用问题

**问题1：每次操作创建新的 KmsClient**
- `createKmsClient()` 每次都 `KmsClient.builder().build()`
- 应该使用单例或连接池，频繁创建/关闭 TCP 连接开销大

**问题2：无缓存机制**
- 每次归集每个用户都要调用 `awsKmsDriver.decryptData(companyAddress)`
- 1000个用户归集 = 1000次 KMS API 调用（companyAddress解密）
- 应该在服务启动时解密一次并缓存

**问题3：异常处理不当**
- KMS 异常只 `log.info`（应该 `log.error`），且返回 null
- 调用方没有对 null 做防御性判断

## 11. 外部签名服务（Sign Service）设计

### 11.1 设计动机

出款地址（Dispensing Address）持有大量资金，其私钥不应存放在 wallet-server 中。通过独立的签名服务实现**签名权限分离**：

```
wallet-server                    Sign Service
    │                                │
    │  1. 构造交易 → 计算txHash       │
    │  2. RSA加密请求体               │
    │  ────────────────────────────→  │
    │  POST /api/admin/chain/sign     │
    │  Header: PLAT_CODE              │
    │  Body: RSA(JSON{orderNo,        │
    │        outAddressNo, chainType, │
    │        hashId})                 │
    │                                │  3. 验证RSA → 解密
    │                                │  4. 用出款私钥签名txHash
    │                                │  5. 返回签名值
    │  ←────────────────────────────  │
    │  {code: 200, data: Base64Sign}  │
    │                                │
    │  6. 组装完整交易（tx + signature）│
    │  7. 广播到TRON网络               │
    │                                │
```

### 11.2 签名请求实现

```java
private byte[] sendSign(String signUrl, Protocol.Transaction tx,
                        String ssOrderId, String from) {
    // 1. 从 trx_wallet_config 读取签名服务配置
    List<TrxWalletConfigEntity> list = trxWalletConfigRepository.findByType("sign");

    // 2. 计算交易哈希
    byte[] txId = Sha256Sm3Hash.hash(tx.getRawData().toByteArray());
    String signTxid = Base64.getEncoder().encodeToString(txId);

    // 3. 构造签名请求
    Map<String, Object> signMap = new HashMap<>();
    signMap.put("orderNo", ssOrderId);          // 订单号
    signMap.put("outAddressNo", from);          // 出款地址
    signMap.put("chainType", "TRON");           // 链类型
    signMap.put("hashId", signTxid);            // 待签名的交易哈希

    // 4. RSA加密请求体
    String enSign = RSAUtil.encrypt(
        JSONObject.toJSONString(signMap),
        list.stream().filter(obj -> "rsa".equals(obj.getCode()))
            .findFirst().get().getValue()  // RSA公钥
    );

    // 5. 发送请求
    HttpResponse response = HttpRequest.post(signUrl + "/api/admin/chain/sign")
        .header("PLAT_CODE", list.stream()
            .filter(obj -> "platCode".equals(obj.getCode()))
            .findFirst().get().getValue())      // 平台编码
        .body(enSign)
        .timeout(60000)
        .execute();

    // 6. 解析签名结果
    JSONObject jsonObject = JSONObject.parseObject(response.body());
    byte[] deSign = null;
    if ("200".equals(jsonObject.get("code"))) {
        deSign = Base64.getDecoder().decode(jsonObject.getString("data"));
    }

    // 7. 组装完整交易（交易体 + 签名）
    return null == deSign ? null :
        tx.toBuilder().addSignature(ByteString.copyFrom(deSign)).build().toByteArray();
}
```

### 11.3 签名路由策略

```java
// trc20Transaction 中的签名分支：
if (deDispensingAddress.equals(from)) {
    // 出款地址 → 调用外部签名服务
    bytes = sendSign(signUrl, tx, ssOrderId, from);
} else {
    // 其他地址（用户地址归集）→ 本地签名
    bytes = signTransaction2Byte(tx.toByteArray(), ByteArray.fromHexString(keyStore));
}
```

**关键区分：**
- **出款地址转账**（提现）：必须通过外部签名服务（私钥不在本地）
- **用户地址转账**（归集）：本地签名（私钥AES解密后在内存中）

### 11.4 设计评价

**优点：**
- 出款私钥物理隔离，即使 wallet-server 被攻破也无法盗取出款资金
- RSA 加密传输，防止中间人窃取签名请求
- 带 PLAT_CODE 鉴权头，多租户安全

**缺陷：**
- 签名服务是单点故障（如果不可用，所有提现阻塞）
- 60秒超时过长（区块链交易有时效性，超时后交易可能已过期）
- 没有重试机制
- RSA 512位密钥过短（应至少 2048 位）

### 12.2 地址生成的密码学细节

```
ECKeyPair (secp256k1 椭圆曲线)
  │
  ├─ 私钥 (256 bit 随机数)
  │   → Numeric.toHexStringNoPrefix() → 64字符hex字符串
  │   → AES加密 → enPrivateKey (Base64)
  │
  └─ 公钥 (512 bit)
      → Keccak-256 哈希 → 取后20字节 → 以太坊地址
      → 加前缀 "41" → TRON hex地址
      → Base58Check 编码 → TRON 可读地址 (T开头)
```

**以太坊地址 → TRON 地址的转换：**
- 以太坊地址：`0x` + 20字节hex = 42字符
- TRON hex地址：`41` + 20字节hex = 42字符（`41` 替代 `0x`）
- TRON Base58地址：Base58Check(`41` + 20字节) = T开头的34字符

### 12.3 问题分析

1. **无并发保护**：两个请求同时为同一用户创建钱包，可能生成两个地址
   - `findAddressByUserId` 和 `save` 之间没有锁
   - user_id 没有唯一索引（只是普通索引）
   - 应该添加 `@UniqueConstraint(columnNames = "user_id")` 或分布式锁

2. **异步通知无保证**：`@Async updateUserAddress` 如果失败，user-server 不知道用户地址
   - 没有重试机制
   - 没有补偿机制

3. **密钥在内存中**：`keyStore`（明文私钥）在方法栈中，方法结束后由 GC 回收，但回收前可能被内存dump获取

### 13.3 ABI 编码细节

```java
// transfer(address,uint256) 的 ABI 编码
public String encoderAbi(String Address, BigInteger amount) {
    ArrayList<Type> inputParameter = new ArrayList<>();
    inputParameter.add(new org.web3j.abi.datatypes.Address(Address));  // 目标地址
    inputParameter.add(new Uint256(amount));                            // 金额（最小单位）
    return FunctionEncoder.encodeConstructor(inputParameter);
    // 输出：64字符hex(address padding to 32 bytes) + 64字符hex(uint256)
}
```

**金额换算：**
```java
// amount = 用户可读金额（如 100 USDT）
// needSendAmount = 100
// decimal = 6 (USDT)
// integer = 100 * 10^6 = 100000000（链上实际传输的最小单位）
BigInteger integer = needSendAmount.multiply(BigDecimal.TEN.pow(
    TRXContractEnum.getDecimalByTokenContract(tokenContract))).toBigInteger();
```

### 13.4 本地签名实现

```java
private static byte[] signTransaction2Byte(byte[] transaction, byte[] privateKey)
        throws InvalidProtocolBufferException {
    ECKey ecKey = ECKey.fromPrivate(privateKey);               // 从私钥构建ECKey
    Protocol.Transaction transaction1 = Protocol.Transaction.parseFrom(transaction);
    byte[] rawdata = transaction1.getRawData().toByteArray();  // 交易原始数据
    byte[] hash = Sha256Sm3Hash.hash(rawdata);                 // SHA-256哈希
    byte[] sign = ecKey.sign(hash).toByteArray();              // ECDSA签名
    return transaction1.toBuilder()
        .addSignature(ByteString.copyFrom(sign))               // 附加签名
        .build().toByteArray();
}
```

### 13.5 fee_limit 说明

```java
map.put("fee_limit", 50000000L);  // 50 TRX = 50,000,000 SUN
```

- fee_limit 是交易愿意消耗的最大 TRX（Gas费上限）
- TRC20 转账实际 Gas 消耗约 5-15 TRX
- 设置 50 TRX 是安全余量
- 注释中原来是 10000000（10 TRX），后改为 50000000（50 TRX）

## 14. 核心链路三：TRX主币转账（trxOfflineSignature）

### 14.1 与 TRC20 转账的区别

| 维度 | TRC20 转账 | TRX 主币转账 |
|------|-----------|-------------|
| 调用接口 | /wallet/triggersmartcontract | createTransaction（本地构造） |
| 合约调用 | transfer(address,uint256) | 无（原生转账） |
| 签名方式 | 本地 或 外部签名 | **仅外部签名** |
| 广播方式 | /wallet/broadcasthex | WalletApi.broadcastTransaction |
| 使用场景 | USDT/CXT归集、提现 | Gas费补充 |

### 14.2 完整流程

```java
public TrxWalletResult trxOfflineSignature(String keyStore, String from,
        String to, BigDecimal needSendAmount, String ssOrderId) {
    // 1. 地址解码
    byte[] fromAddress = WalletApi.decodeFromBase58Check(from);
    byte[] toHexAddress = WalletApi.decodeFromBase58Check(to);

    // 2. 金额转换（TRX精度=6）
    amount = needSendAmount.multiply(BigDecimal.TEN.pow(TRXContractEnum.TRX.getDecimal()));

    // 3. 构造交易（本地，非RPC）
    Protocol.Transaction transaction = createTransaction(
        fromAddress, toHexAddress, amount.longValueExact());

    // 4. 计算交易哈希
    byte[] txIdHash = Sha256Sm3Hash.hash(transaction.getRawData().toByteArray());

    // 5. 调用外部签名服务
    byte[] bytes = sendSign(signUrl, transaction, ssOrderId, from);

    // 6. 广播交易
    String hash = ByteArray.toHexString(txIdHash);
    boolean success = WalletApi.broadcastTransaction(bytes);

    // 7. 返回
    if (success) return TrxWalletResult{txId: hash};
    return null;
}
```

### 14.3 注意点

**TRX 主币转账始终调用外部签名服务**。代码中可以看到，即使 keyStore 参数被传入，实际签名也是走 `sendSign()`。这是因为 TRX 主币转账主要用于 Gas 费补充，from 地址是 Gas 地址或出款地址，它们的私钥都不在本地。

**但这里有一个 Bug**：如果需要从用户地址转 TRX（不太常见但理论上可能），这个方法也会调用外部签名服务，但外部签名服务不持有用户私钥，会签名失败。

## 15. 核心链路四：单笔归集（singleCollect）

### 15.1 归集是什么？

**归集 = 把用户地址上的代币转移到公司主地址**

用户充值时，将代币转入自己的专属地址。但这些代币分散在成百上千个地址上，需要定期或即时地"归拢"到公司的一个主地址，方便统一管理和提现。

### 15.4 问题分析

1. **Gas 转账和代币转账不是原子操作**
   - Gas 转账成功但代币归集失败 → 30 TRX 滞留在用户地址
   - 没有回滚或补偿机制
   - 下次归集会再次检查 TRX 余额，如果 > 30 TRX 则不再补充（自愈）

2. **Gas 转账后没有等待确认**
   - TRX 转入用户地址后立即发起 TRC20 转账
   - TRON 链上确认需要约 3 秒（1个区块）
   - 如果 Gas 转账尚未被确认，TRC20 转账可能因 Gas 不足失败
   - **应该加入等待/轮询机制**

3. **变量命名混乱**
   - `newGasAddress` 实际是从 `disPendAddress`（出款地址）配置中读取
   - 但 Gas 补充用的是 `gasPassword`（Gas私钥）从 `gasAddress` 发送
   - 两段代码中 Gas 来源不一致，可能是不同版本的遗留

4. **全额归集**
   - 将用户地址上的代币全部归集（`trxBalance` = 全部余额）
   - 没有预留手续费或最小保留金额的概念

### 16.3 性能问题分析

**严重的串行瓶颈：**

- 1次 TRC20 余额查询（HTTP RPC）
- 1次 TRX 余额查询（HTTP RPC，Gas 检查）
- 可能 1次 Gas 补充（链上交易 3-6秒）
- 1次代币归集交易（链上交易 3-6秒）
- 2次 DB 写入

**保守估计：每个用户 5-10秒，1000用户 = 83-167分钟**

1. **串行执行**：没有并行处理
2. **每个用户都查询全部配置**：`trxWalletConfigRepository.findByType("sign")` 在 singleCollect 中被反复调用
3. **KMS 重复调用**：每次循环都解密 `companyAddress`

5. **无进度跟踪**：1000个用户归集过程中无法知道进度
6. **无并发锁**：多实例部署时可能重复归集

- 链上余额查询和实际转账之间有时间差
- 并发提现可能导致余额检查时足够但实际转账时不足
- TRON 链上不会出现余额透支（交易会失败），但可能浪费 Gas

### 17.3 提现代码Bug分析

```java
// TrxDriver.withdrawTraction() 第329行
result = createTransactionForTrx(fromAddress, sendAmount, fromAddress,
    toAddress, tokenContract, ssOrderId);
```

**第一个参数应该是 privateKey，但传入的是 fromAddress（出款地址字符串）**

进入 `createTransactionForTrx`：
```java
public TrxWalletResult createTransactionForTrx(String privateKey, ...) {
    String keyStore = privateKey;
    if (StringUtils.isBlank(privateKey)) {
        keyStore = trxWalletService.getPrivateKeyByAddress(from, password);
    }
    ...
}
```

由于 `fromAddress` 不是 blank，`keyStore` 被设置为地址字符串。然后：
- 如果是 TRC20，进入 `trc20Transaction(keyStore, from, to, ...)`
- 在 `trc20Transaction` 中，判断 `from == dispensingAddress`，走外部签名
- **外部签名不使用 keyStore 参数**，所以虽然 keyStore 是错误值，但不影响结果

**结论：** 代码意外地"正确工作"，但参数语义完全错误。如果出款地址不走外部签名（比如配置变更），会导致使用地址字符串作为私钥来签名，必然失败。

  3. 返回余额数值

- 高频调用可能触发 TRON 节点限流

### 20.1 充值全流程（最重要的链路）

这是从用户角度看到的完整充值过程：

```

```

### 20.2 充值链路中的关键时间节点

```
T0: 用户发起 USDT 转账（交易所/外部钱包）
T1: TRON 链上确认（T0 + 3秒~3分钟）
T2: 平台检测到充值（手动 或 定时扫描）
T3: 管理员审核（人工环节，时间不定）
T4: 触发归集（Gas补充 + 代币转账）
T5: 归集链上确认（T4 + 3秒~3分钟）
T6: user-server 确认并增加余额
T7: 用户余额可用
```

**关键观察：**
- 从 T0 到 T7，最快约 1 分钟（即时归集 + 无需审核）
- 最慢可能 1 天+（等待定时归集 + 人工审核）
- **系统没有自动检测链上充值的机制**——不监听链上事件
- 依赖人工确认或定时归集，用户体验不够好

### 20.3 充值链路的缺失环节

**当前系统缺失的关键能力：**

1. **链上充值监听**：没有 WebSocket 或轮询机制监听用户地址的入账事件
   - 改进方案：使用 TronGrid Event API 或 TronWeb 的事件订阅
   - 或定时轮询用户地址的 TRC20 转入记录

2. **自动入账**：充值确认完全依赖人工或定时任务
   - 改进方案：监听到入账后自动创建订单 + 自动归集

3. **充值金额核验**：归集时没有校验链上实际到账金额与订单金额是否一致
   - 如果用户声称充值 100 USDT 但实际只转了 50，系统可能按 100 入账

4. **多次充值合并**：如果用户在两次归集之间充值了多次，所有金额会在一次归集中全部转走
   - 需要有机制区分哪些金额对应哪个订单

## 21. Gas 费管理机制

### 21.1 为什么需要 Gas 管理？

TRON 链上的 TRC20 代币转账（USDT 等）需要消耗 TRX 作为能量/带宽费用。用户地址上可能只有 USDT 没有 TRX，所以需要先补充 TRX。

### 21.2 Gas 费补充策略

```
检查条件：用户地址/出款地址的 TRX 余额 < 30 TRX
补充金额：固定 30 TRX
补充来源：Gas 地址（gasAddress）
补充目标：归集 → 用户地址；提现 → 出款地址
```

### 21.3 Gas 费参数

```java
private static final BigDecimal FEE_BAL = new BigDecimal("30");  // 30 TRX 阈值
// fee_limit = 50000000 (50 TRX) — TRC20 转账的最大 Gas 上限
// 实际 TRC20 转账 Gas 消耗约 5-15 TRX
```

### 21.4 Gas 费补充的两个场景

**场景1：归集时（singleCollect）**
```java
// Gas 来源
from: gasAddress (KMS解密)
privateKey: gasPassword (KMS解密)
to: 用户地址 (fromAddress)
amount: 30 TRX
// 使用 gasPassword 直接签名，不走外部签名服务
```

**场景2：提现时（withdrawTraction）**
```java
// Gas 来源
from: gasAddress (KMS解密)
privateKey: gasPassword (KMS解密)
to: 出款地址 (dispensingAddress)
amount: 30 TRX
// 同样直接签名
```

### 21.5 Gas 管理问题

1. **固定 30 TRX 的浪费**
   - 实际 TRC20 转账只需 5-15 TRX
   - 每次补充 30 TRX 可能有剩余
   - 归集后剩余的 TRX 留在用户地址上（无法归集 TRX 主币）

2. **Gas 地址余额耗尽风险**
   - 1000 用户归集，每人补充 30 TRX = 30,000 TRX
   - 没有 Gas 地址余额预警机制
   - Gas 地址耗尽 → 所有归集/提现阻塞

3. **Gas 补充后立即转账**
   - Gas 补充交易确认需要 3 秒
   - 没有等待确认就立即发起代币转账
   - 可能导致代币转账因 Gas 不足失败

4. **不归集 TRX 主币**
   - 定时任务只归集 USDT，不归集 TRX
   - 用户地址上残留的 TRX（Gas 补充的剩余）无法回收

## 24. user-server 资产并发控制——Redisson + 乐观锁

### 24.1 双重并发控制

```java
// UserAssetsBusiness.java
public void doBalanceBusiness(User user, BalanceVO vo) {
    // 第一层：Redisson 分布式锁
    RLock lock = redissonClient.getFairLock("USER_ASSETS:" + userId);
    lock.lock();
    try {
        // 第二层：JPA @Version 乐观锁（在 Assets 表上）
        doIncomeBalance(user, vo);  // 或 doReduceBalance
    } finally {
        lock.unlock();
    }
}
```

### 24.2 与参考A的对比

| 维度 | 参考A (p9-services) | 参考B (tg-services) |
|------|--------------------|--------------------|
| 分布式锁 | Redisson lock (20s/30s) | Redisson fairLock (无超时?) |
| 数据库锁 | SQL WHERE 条件防负数 | JPA @Version 乐观锁 |
| 幂等保护 | 应用层 order_no 查询 | DB唯一约束 (change_type, relate_id, user_id, order_type) |
| 锁粒度 | userId 级别 | userId 级别 |

### 24.3 参考B的优势

1. **公平锁（FairLock）**：保证等待顺序，避免饥饿
2. **乐观锁**：JPA `@Version` 比 SQL WHERE 条件更规范
3. **DB级幂等**：唯一约束比应用层查询更可靠

## 25. user-server 幂等性设计——BalanceChange 唯一约束

### 25.1 唯一约束定义

```java

})

    private Integer orderType;   // 1=用户订单, 2=注单

}
```

### 25.2 幂等机制

```
同一笔充值操作(changeType=1, relateId=订单ID, userId=用户ID, orderType=1)
  第一次写入 → 成功
  第二次写入 → 唯一约束冲突 → 抛异常 → 不会重复加款
```

### 25.3 这是参考B中最值得采纳的设计

**为什么数据库级幂等优于应用级幂等：**
1. 应用层查询有时间窗口（check-then-act 非原子）
2. 数据库唯一索引是原子操作，无竞态条件
3. 即使应用层逻辑有Bug，数据库也能兜底

**对 ser-wallet 的启示：**
- 在余额变动流水表上设计唯一约束
- 约束字段 = (业务类型 + 业务ID + 用户ID + 订单类型)
- 配合乐观锁实现最强保障

## 26. Smart Contract 交互细节——ABI 编码与链上调用

### 26.1 TRC20 的 transfer 调用

TRC20 代币（如 USDT）基于 TRC20 标准（类似 ERC20），其转账需要调用智能合约的 `transfer` 方法。

**Solidity 方法签名：**
```solidity
function transfer(address to, uint256 amount) public returns (bool);
```

**ABI 编码过程：**
```java
// 1. 方法选择器（function_selector）
"transfer(address,uint256)"
// TRON节点会自动计算 keccak256 哈希并匹配

// 2. 参数编码
String parameter = encoderAbi(toHexAddress(to).substring(2), amount);
// address: 左补零到 32 字节 (64 hex chars)
// uint256: 左补零到 32 字节 (64 hex chars)
// 总共 128 hex chars

// 3. 金额单位转换
// 用户金额 * 10^decimal = 链上最小单位
// 100 USDT * 10^6 = 100,000,000
```

### 26.2 triggersmartcontract API

```json
POST https://nile.trongrid.io/wallet/triggersmartcontract
{
    "contract_address": "41...(USDT合约hex地址)",
    "function_selector": "transfer(address,uint256)",
    "parameter": "0000...address...0000...amount...",
    "owner_address": "41...(发送方hex地址)",
    "call_value": 0,
    "fee_limit": 50000000
}
```

**返回：**
```json
{
    "result": {"result": true},
    "transaction": {
        "txID": "...",
        "raw_data": {...},
        "raw_data_hex": "..."
    }
}
```

### 26.3 交易打包与广播

```java
// 1. 打包：JSON → Protocol.Transaction (protobuf)
Protocol.Transaction tx = TronUtils.packTransaction(json, false);

// 2. 添加备注
transaction.raw_data.data = hex("trc20")

// 3. 签名：ECDSA secp256k1
byte[] hash = Sha256Sm3Hash.hash(tx.getRawData().toByteArray());
byte[] sign = ecKey.sign(hash).toByteArray();
signedTx = tx.toBuilder().addSignature(ByteString.copyFrom(sign)).build();

// 4. 广播
POST trxUrl/wallet/broadcasthex {transaction: hex(signedTx)}
→ {code: "SUCCESS", txid: "..."}
```

### 26.4 TRON vs Ethereum 的差异

| 维度 | Ethereum | TRON |
|------|----------|------|
| 地址前缀 | 0x | 41 (hex) / T (Base58) |
| 编码格式 | RLP | Protobuf |
| 共识机制 | PoW/PoS | DPoS (19个超级代表) |
| 确认时间 | 12秒 | 3秒 |
| Gas模型 | gas * gasPrice | 能量(Energy) + 带宽(Bandwidth) |
| ABI标准 | ERC20 | TRC20 (兼容ERC20 ABI) |

**代码中利用了 TRON 对以太坊的兼容性：**
- 密钥生成使用 web3j（以太坊库）
- ABI编码使用 web3j FunctionEncoder
- 只需要在地址转换时加 "41" 前缀

### 29.2 动态配置（trx_wallet_config 表）

| type | code | 含义 | 使用场景 |
|------|------|------|---------|
| sign | disPendAddress | 出款地址 | 提现时的发送方 |

| sign | platCode | 平台编码 | 签名请求Header |

### 29.3 双轨设计的意义

- **静态配置**：核心地址和密钥（不频繁变更，安全性高）
- **动态配置**：签名服务相关（可能需要热更新RSA密钥或出款地址）

### 29.4 问题

1. **命名混乱**：
   - `mainCollectAddress` vs `companyAddress` — 两个配置含义模糊
   - `mainCollectPassphrase` 名为"助记词"但实际是 AES 密钥的一部分
   - `mainDispensingAddress`/`mainDispensingPrivateKey` 被注释掉但配置文件中还有

2. **动态配置无缓存**：
   - 每次使用都查数据库
   - `findByType("sign")` 在归集循环中被频繁调用

3. **安全隐患**：
   - AWS credentials 明文存在 application.yml 中
   - 应使用 AWS IAM Role 或 环境变量

## 31. 架构优劣势深度评估

### 31.1 架构优势（值得学习）

| # | 优势 | 评分 | 说明 |
|---|------|------|------|
| 1 | 多地址架构 | 9/10 | 用户、归集、出款、Gas 四类地址角色清晰分离 |
| 2 | 私钥分级保护 | 8/10 | 三层加密：AES(用户) + KMS(系统) + 外部签名(出款) |
| 3 | 外部签名服务 | 9/10 | 出款私钥物理隔离，安全设计标杆 |
| 4 | 币种合约枚举 | 7/10 | 统一管理币种、精度、合约地址 |
| 5 | DB级幂等(user-server) | 9/10 | BalanceChange 唯一约束，最强幂等保障 |
| 6 | 乐观锁(user-server) | 8/10 | JPA @Version 标准实现 |
| 7 | 归集机制设计 | 7/10 | 单笔即时 + 定时批量，覆盖两种场景 |
| 8 | Gas 费自动管理 | 7/10 | 自动检查 + 自动补充，无需人工干预 |

### 31.2 架构缺陷（需要改进）

| # | 缺陷 | 严重度 | 说明 |
|---|------|--------|------|
| 1 | 无链上充值监听 | **高** | 充值完全依赖人工/定时发现，用户体验差 |
| 2 | AES/ECB模式 | **高** | ECB模式有已知安全缺陷，应用CBC/GCM |
| 3 | RSA 512位 | **高** | 可暴力破解，应 ≥ 2048位 |
| 4 | Gas补充后不等确认 | **高** | 可能导致后续交易失败 |
| 5 | 串行归集 | **中** | 1000用户归集耗时100+分钟 |
| 6 | 无分布式锁(wallet-server) | **中** | 并发请求可能重复创建钱包/重复提现 |
| 7 | ssOrderId无唯一索引 | **中** | 查询级幂等有竞态窗口 |
| 8 | KMS无缓存 | **中** | 每次操作都调KMS API，性能差、成本高 |
| 9 | 异常处理粗糙 | **中** | 大量 catch + log + return null |
| 10 | JPA ddl-auto:update | **中** | 生产环境不安全（可能意外修改表结构） |
| 11 | AWS credentials明文 | **低** | 应使用IAM Role或环境变量 |
| 12 | 重复代码 | **低** | AES加密在两处实现 |
| 13 | 命名混乱 | **低** | 变量名与实际含义不符 |

### 31.3 安全风险评估

| 风险 | 等级 | 影响 | 缓解建议 |
|------|------|------|---------|
| AES/ECB 模式 | 高 | 加密私钥可能被分析 | 改用 AES/GCM |
| RSA 512 | 高 | 签名请求可被伪造 | 升级到 RSA 2048+ |
| 私钥内存驻留 | 中 | 内存dump可获取私钥 | 使用后立即清零 |
| AWS credentials 明文 | 中 | yml泄露等于KMS失守 | IAM Role |
| Basic Auth | 中 | 服务间认证脆弱 | mTLS / JWT |
| 无审计日志 | 低 | 安全事件难追溯 | 增加操作审计 |

## 33. 对 ser-wallet 的参考价值总结

### 33.1 直接采纳的设计（可直接复用）

| # | 设计 | 来源 | 原因 |
|---|------|------|------|
| 1 | 多地址角色分离 | 8.1节 | 归集、出款、Gas 角色分明是业界标准 |
| 2 | 外部签名服务架构 | 11节 | 出款私钥物理隔离，安全必需 |
| 3 | DB级幂等(唯一约束) | 25节 | 最强幂等保障，直接用于余额变动表 |
| 4 | 归集双模式(即时+定时) | 15-16节 | 覆盖用户体验和运营效率两种需求 |
| 5 | Gas 自动补充 | 21节 | TRC20转账的必要机制 |
| 6 | 币种合约枚举 | 7.1节 | 统一管理合约地址、精度、币种名 |
| 7 | 乐观锁(@Version/version) | 24节 | 配合分布式锁使用的第二层保护 |
| 8 | 交易记录双视角(order+collect) | 4节 | 不同维度查询需求 |

### 33.2 需要改进后采纳的设计

| # | 设计 | 改进方向 | 原因 |
|---|------|---------|------|
| 1 | AES 私钥加密 | ECB→GCM, SHA-1→SHA-256, 128→256 | 安全缺陷不可上线 |
| 2 | RSA 签名请求加密 | 512→2048+位 | 512位可被暴力破解 |
| 3 | KMS 密钥管理 | 添加缓存 + 使用IAM Role | 性能和安全 |
| 4 | Gas 费补充 | 补充后等待确认再转账 | 否则后续交易可能失败 |
| 5 | 串行归集 | 改为并行(带限流) | 1000用户串行不可接受 |
| 6 | 定时归集 | 增加链上事件监听 | 用户体验要求快速入账 |
| 7 | 余额查询 | 增加短期缓存 | 避免高频调用TRON节点 |
| 8 | 异常处理 | 统一错误码 + 异常分类 | 不能所有错误都return null |

### 33.3 不采纳的设计

| # | 设计 | 原因 |
|---|------|------|
| 1 | JPA ddl-auto:update | 生产环境不安全，使用 Flyway/Migrate |
| 2 | Basic Auth | 安全级别太低 |
| 3 | hutool + RestTemplate 混用 | HTTP客户端应统一 |
| 4 | 查询级幂等(findBySsOrderId) | 有竞态窗口，应使用唯一索引 |
| 5 | @Async 异步通知(无重试) | 关键操作应有可靠消息机制 |
| 6 | AWS credentials 明文配置 | 安全隐患 |
| 7 | AES重复实现 | 代码维护问题 |
| 8 | Driver直接注入Repository | 违反分层原则 |

### 33.4 核心结论

**参考B 的核心价值是区块链交互的完整实现**：

1. **钱包地址生成**（ECKeyPair → TRON地址 → 加密存储）的完整链路
2. **TRC20 智能合约调用**（ABI编码 → triggersmartcontract → 签名 → 广播）的完整链路
3. **归集机制**（余额检查 → Gas补充 → 代币转账 → 记录）的完整链路
4. **外部签名服务**（RSA加密 → 请求 → 签名值组装 → 广播）的完整链路

这四条链路是区块链钱包的核心骨架，参考B提供了**可工作的完整实现**（虽然有安全缺陷）。ser-wallet 可以在此骨架上增强安全性、改进性能、完善错误处理。

**一句话总结：** 参考B 的"做什么"和"怎么做"都值得参考，但"做的安全不安全"需要全面升级。

### C.2 改进后采纳 🔧

| # | 设计项 | 问题 | 改进方案 |
|---|--------|------|---------|
| 1 | AES私钥加密 | ECB模式+SHA-1+128位 | AES-256-GCM + SHA-256 |
| 2 | RSA签名请求加密 | 512位太短 | RSA-2048 或 Ed25519 |
| 3 | AWS KMS使用 | 无缓存+频繁创建Client | 单例Client + 启动时缓存解密 |
| 4 | Gas补充流程 | 不等确认就转账 | 补充后轮询确认(最多30s) |
| 5 | 批量归集 | 串行执行 | 并行(Go goroutine天然优势) |
| 6 | 充值检测 | 无链上监听 | 添加TronGrid事件监听/轮询 |
| 7 | 余额查询 | 无缓存 | Redis短期缓存(30s) |
| 8 | 异常处理 | catch+return null | 统一错误码体系+异常传播 |
| 9 | 幂等(wallet-server) | 查询级(无唯一索引) | ssOrderId加唯一索引 |
| 10 | 配置加密 | AWS credentials明文 | 环境变量/Vault/IAM Role |

### C.3 不采纳 ❌

| # | 设计项 | 不采纳理由 |
|---|--------|----------|
| 1 | JPA ddl-auto:update | 生产不安全，用GORM AutoMigrate+版本管理 |
| 2 | Basic Auth认证 | 安全级别太低，用JWT/mTLS |
| 3 | hutool+RestTemplate混用 | HTTP客户端不统一，Go用统一client |
| 4 | @Async无重试通知 | 关键操作需可靠消息(MQ/重试) |
| 5 | AES重复实现 | 统一到一处 |
| 6 | Driver直接注入Repository | 严格分层 |
| 7 | 串行遍历所有用户归集 | Go用goroutine池并行 |
| 8 | 变量命名混乱 | 严格命名规范 |
