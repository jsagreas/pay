### 2.2 职责边界

| 服务 | 职责 | 数据归属 |
|------|------|---------|
| **wallet-server** | 链上操作：创建钱包、转账、余额查询、交易确认、归集 | TrxWallet, TrxWalletOrder, TrxWalletCollect, TrxWalletConfig |
| **user-server** | 业务逻辑：充值/提现流程编排、余额增减、冻结解冻、订单状态机、账变记录 | Assets, BalanceChange, SsOrder |
| **web-server** | 网关转发 + TG Bot 通知 | 无 |
| **backend-server** | 后台管理 + 统计接收 | 无（虚拟币场景） |

### 2.3 关键设计决策

**决策1：钱包余额双轨制**
- **链上余额**：由 wallet-server 直接查询 TRON 全节点获取，代表用户地址上的真实链上资产
- **平台余额**：由 user-server 的 Assets 表管理，代表用户在平台可用于投注/提现的额度
- 两者通过"充值归集 → 增加平台余额"和"提现 → 减少平台余额"联动

**决策2：Polling 而非 Event-Driven**
- 充值确认采用**定时任务轮询**（每2分钟 `@Scheduled(cron = "0 */2 * * * ?")`），而非区块链事件监听
- **原因推测**：TRON 全节点 HTTP API 不支持 WebSocket 推送；轮询实现简单，对少量用户足够
- **代价**：最长2分钟延迟，高并发时 N 个订单串行查询

**决策3：服务间 HTTP 直连**
- user-server → wallet-server 通过 `HttpClient4Util.doPost()` 直接调用，无服务发现
- wallet-server → 加签系统 通过 HTTP + RSA 加密
- 原来设计有 RabbitMQ，但代码中已被注释，改为 HTTP 直接调用 backend-server

## 第3章 数据模型与表结构

### 3.1 wallet-server 表结构（4张表）

#### 表1：TrxWallet（用户链上钱包）

```sql
CREATE TABLE trx_wallet (
    id          BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id     VARCHAR(10)  COMMENT '会员账号',         -- INDEX
    user_name   VARCHAR(30)  COMMENT '会员姓名',
    en_private_key VARCHAR(250) COMMENT '加密后的私钥',   -- ★ AES-256-ECB 加密存储
    address     VARCHAR(100) COMMENT '钱包地址',         -- TRON Base58Check 格式（T开头）
    create_time DATETIME,
    update_time DATETIME
);
CREATE INDEX idx_user_id ON trx_wallet(userId);
```

**关键点**：
- userId 是 VARCHAR(10) 而非 BIGINT → 与 user-server 的 BIGINT userId 不一致，需要 String.valueOf() 转换
- en_private_key 存储加密后的私钥，密文长度约 88 字符（Base64编码的 AES 输出）
- 一个用户一个地址（1:1 映射），不支持多地址
- **缺失**：无 UNIQUE 约束在 userId 上 → 理论上可创建重复钱包（代码层通过 findAddressByUserId 做幂等检查）

#### 表2：TrxWalletOrder（链上交易记录）

```sql
CREATE TABLE trx_wallet_order (
    id           BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id      VARCHAR(10)  COMMENT '会员账号',
    tx_id        VARCHAR(200) COMMENT '交易id（TRON txHash）',  -- INDEX
    ss_order_id  VARCHAR(200) COMMENT '充值/提现记录id',
    amount       DECIMAL(16,2) COMMENT '金额',
    from_address VARCHAR(250) COMMENT '发送地址',
    to_address   VARCHAR(250) COMMENT '接收地址',
    coin_name    VARCHAR(10)  COMMENT '币种类型（USDT/TRX/CXT）',
    order_type   TINYINT(2)   COMMENT '收付方向（1充值/2提现）',
    post_type    TINYINT(2)   COMMENT '调用方式（1:接口/2:定时任务）',
    create_time  DATETIME,
    update_time  DATETIME
);
CREATE INDEX idx_user_id ON trx_wallet_order(user_id);
CREATE INDEX idx_order_type ON trx_wallet_order(order_type);
CREATE INDEX idx_tx_id ON trx_wallet_order(tx_id);
```

**关键点**：
- 金额用 DECIMAL(16,2)，只保留2位小数 → USDT实际精度是6位，会丢失精度
- ss_order_id 关联 user-server 的 SsOrder.id
- **幂等查询**：`findBySsOrderId` 且加了 `coin_name = 'USDT'` 条件（硬编码）
- **缺失**：ss_order_id 无 UNIQUE 约束 → 依赖查询幂等，而非 DB 约束幂等

#### 表3：trx_wallet_collect（归集记录）

```sql
CREATE TABLE trx_wallet_collect (
    id              BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id         VARCHAR(10)  COMMENT '会员账号',       -- INDEX
    tx_id           VARCHAR(200) COMMENT '交易id',         -- INDEX
    collect_address VARCHAR(250) COMMENT '汇款账号',
    amount          DECIMAL(16,2) COMMENT '汇款金额',
    collect_type    TINYINT(2)   COMMENT '归集类型',        -- INDEX（1=单笔充值归集, 2=批量定时归集）
    create_time     DATETIME,
    update_time     DATETIME
);
CREATE INDEX idx_user_id ON trx_wallet_collect(userId);
CREATE INDEX idx_collect_type ON trx_wallet_collect(collectType);
CREATE INDEX idx_tx_id ON trx_wallet_collect(txId);
```

**关键点**：归集类型区分单笔充值触发 vs 每日定时批量归集

#### 表4：TrxWalletConfig（钱包配置）

```sql
CREATE TABLE trx_wallet_config (
    id    BIGINT AUTO_INCREMENT PRIMARY KEY,
    type  VARCHAR(30)   COMMENT '类型',
    code  VARCHAR(50)   COMMENT '标识code',
    value VARCHAR(1000) COMMENT '标识value'   -- ★ 存储 RSA 公钥、出款地址等敏感配置
);
```

**关键点**：
- 配置化的出款地址（dispensing address）和 RSA 公钥，不硬编码在配置文件中
- type='sign' 下存储：出款地址(DISPENDADDRESS)、RSA公钥(RSA)、平台码(PLATCODE)

### 3.2 user-server 表结构（3张核心表）

#### 表5：Assets（用户平台资产）

```sql
CREATE TABLE assets (
    id             BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id        BIGINT(20) UNIQUE COMMENT '会员ID',     -- ★ UNIQUE 约束
    balance        DECIMAL(16,2) DEFAULT 0 COMMENT '余额',
    balance_nn     DECIMAL(16,2) DEFAULT 0 COMMENT '牛牛余额',
    balance_bacc   DECIMAL(16,2) DEFAULT 0 COMMENT '百家乐余额',
    freeze_amount  DECIMAL(16,2) DEFAULT 0 COMMENT '冻结余额',
    play_money     DECIMAL(16,2) DEFAULT 0 COMMENT '打码量',
    credit_amount  DECIMAL(16,2) DEFAULT 0 COMMENT '信用额度',
    version        INT DEFAULT 1,                           -- ★ @Version 乐观锁
    user_type      TINYINT(2) DEFAULT 1 COMMENT '1正式 2测试 3机器人',
    agent_id       BIGINT(11) COMMENT '当前代理ID',
    agent_level    TINYINT(2) COMMENT '代理层级',
    -- ...省略 agent_id1~5, channel_name1~5 等冗余字段
    create_time    DATETIME,
    update_time    DATETIME
);
```

**关键点**：
- **三余额模型**：balance（主余额）、balanceNn（牛牛余额）、balanceBacc（百家乐余额）→ 游戏隔离
- **冻结余额**：freezeAmount，提现时冻结，链上确认后解冻并扣减
- **@Version 乐观锁**：Hibernate 自动管理，每次更新 version+1，并发冲突时抛异常
- **打码量**：playMoney，充值时按 flowMultiple 倍率增加，用于流水要求判断
- **冗余字段**：代理层级(agent_id1~5)、渠道(channel_name1~5) 冗余存储 → 报表查询方便但数据一致性维护成本高

#### 表6：BalanceChange（账变记录）

```sql
CREATE TABLE balance_change (
    id               BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id          BIGINT(20) COMMENT '会员ID',
    relate_id        BIGINT(20) COMMENT '关联ID（订单ID）',
    change_type      TINYINT(2) COMMENT '变动类型',
    balance_type     TINYINT(2) COMMENT '收支类型 1收入 2支出',
    before_amount    DECIMAL(16,2) COMMENT '交易前金额',
    amount           DECIMAL(16,2) COMMENT '变化金额',
    after_amount     DECIMAL(16,2) COMMENT '交易后金额',
    remark           VARCHAR(100) COMMENT '备注信息',
    business_type    TINYINT(2) COMMENT '业务类型',
    order_type       TINYINT(2) COMMENT '订单类型 1用户订单 2注单',
    -- ...省略 agent/channel 冗余字段
    UNIQUE KEY uk_change (change_type, relate_id, user_id, order_type)  -- ★★ 四字段唯一约束
);
```

**关键点**：
- **★★ 四字段唯一约束**：`(changeType, relateId, userId, orderType)` — 这是本工程最有价值的幂等设计
- before_amount / after_amount 形成余额快照链 → 可追溯任意时刻余额
- changeType 枚举：TELEGRAM_CHARGE(充值)、TELEGRAM_WITHDRAW(提现)、FEE(手续费)、BET(下注)、SETTLEMENT(派奖) 等

#### 表7：SsOrder（充提订单）

```sql
CREATE TABLE ss_order (
    id               BIGINT AUTO_INCREMENT PRIMARY KEY,
    order_num        VARCHAR(32) COMMENT '订单编号',        -- INDEX
    user_id          BIGINT(20) COMMENT '会员ID',          -- INDEX
    amount           DECIMAL(16,2) COMMENT '总金额',
    real_amount      DECIMAL(16,2) COMMENT '实际到账金额',
    order_type       TINYINT(2) COMMENT '1充值 2提现',      -- INDEX
    order_status     TINYINT(2) DEFAULT 1 COMMENT '1等待 2成功 3失败 4取消', -- INDEX
    flow_multiple    DECIMAL(16,2) DEFAULT 1 COMMENT '流水倍数',
    virtual_currency VARCHAR(20) COMMENT '虚拟币种（USDT）',
    fee_amount       DECIMAL(16,2) DEFAULT 0 COMMENT '手续费',
    payee_address    VARCHAR(200) COMMENT '收款地址',
    tx_id            VARCHAR(200) COMMENT '交易记录Id（链上 txHash）',
    nick_name        VARCHAR(64) COMMENT '会员昵称',
    complete_time    DATETIME COMMENT '完成时间',
    -- ...省略审核/锁单/代理等字段
    create_time      DATETIME,
    update_time      DATETIME
);
```

**关键点**：
- **amount vs realAmount**：提现场景 amount=总额、realAmount=到账额（amount - feeAmount）
- **订单状态机**：1(VIRTUAL_CURRENCY_WAIT) → 2(ORDER_SUCCESS) / 3(ORDER_FAIL) / 4(ORDER_CANCEL)
- **flowMultiple（流水倍数）**：充值时设置，playMoney = realAmount × flowMultiple
- tx_id 在创建时为空，归集上链后回填

### 5.2 加密方案详解

```java
// WalletTrxBusiness.java:183-202

// 加密流程
public static String encrypt(String strToEncrypt, String secret) {
    SecretKeySpec secretKeySpec = new SecretKeySpec(getKey(secret), "AES");
    Cipher cipher = Cipher.getInstance("AES/ECB/PKCS5Padding");  // ★ ECB 模式
    cipher.init(Cipher.ENCRYPT_MODE, secretKeySpec);
    return Base64.getEncoder().encodeToString(cipher.doFinal(strToEncrypt.getBytes("UTF-8")));
}

// 密钥派生
private static byte[] getKey(String secret) {
    byte[] bytes = secret.getBytes("UTF-8");
    MessageDigest sha = MessageDigest.getInstance("SHA-1");       // SHA-1 哈希
    byte[] digest = sha.digest(bytes);
    return Arrays.copyOf(digest, 16);                             // 取前16字节 = AES-128
}
```

**分析**：
- secret = `passphrase + address`，每个地址的加密密钥不同（地址绑定）
- passphrase 本身通过 AWS KMS 加密存储在配置中
- **AES/ECB 模式缺陷**：ECB 模式不使用 IV，相同明文产生相同密文 → 但私钥本身是随机的，实际影响有限
- **SHA-1 + 截断16字节 = AES-128**：实际只有128位安全性，非标称的256位
- **解密**：`decrypt(enKey, passphrase + address)` → 还原64字符 hex 私钥

public String encryptData(String plaintext) {
    EncryptRequest req = new EncryptRequest()
        .withKeyId(keyId)            // AWS KMS 密钥 ID
        .withPlaintext(ByteBuffer.wrap(plaintext.getBytes()));
    EncryptResult result = kmsClient.encrypt(req);
    return Base64.getEncoder().encodeToString(result.getCiphertextBlob().array());
}

public String decryptData(String ciphertext) {
    DecryptRequest req = new DecryptRequest()
        .withCiphertextBlob(ByteBuffer.wrap(Base64.getDecoder().decode(ciphertext)));
    DecryptResult result = kmsClient.decrypt(req);
    return new String(result.getPlaintext().array());
}

## 第10章 余额引擎与账变机制

### 10.1 UserAssetsBusiness 核心模式

```java
// UserAssetsBusiness.java:76-109 — doBalanceBusiness

public ResponseEntity doBalanceBusiness(User user, BalanceVO vo) {
    RLock fairLock = redisson.getFairLock(RedisConstants.USER_ASSETS + user.getId());
    try {
        boolean res = fairLock.tryLock(WAIT_TIME, UNLOCK_TIME, TimeUnit.SECONDS);
        if (res) {
            if (INCOME == vo.getBalanceType()) {
                ResponseEntity response = doIncomeBalance(user, vo);
                fairLock.unlock();
                return response;
            }
            if (EXPENSES == vo.getBalanceType()) {
                ResponseEntity response = doReduceBalance(user, vo);
                fairLock.unlock();
                return response;
            }
        }
    } catch (Exception e) {
        fairLock.unlock();   // ★ catch 中 unlock，但 finally 缺失
    }
    return ResponseUtil.fail();
}
```

### 10.2 收入流程（充值上分）

```
doIncomeBalance(user, vo)
  │
  ├── 1. 查询当前 Assets
  │   assets = findAssetsByUserId(user.getId())
  │
  ├── 2. 构建 BalanceChange（账变记录）
  │   balanceChange.beforeAmount = assets.getBalance()
  │   balanceChange.afterAmount  = assets.getBalance() + vo.getAmount()
  │   balanceChange.UK = (changeType, relateId, userId, orderType)  ← 唯一约束
  │
  ├── 3. ★ 先写账变
  │   balanceChangeService.save(balanceChange)
  │   // 如果 UK 冲突 → 抛异常 → 幂等保障
  │
  └── 4. 后更余额
      assetsService.doIncreaseBalanceById(amount, assetsId)
      // SQL: UPDATE assets SET balance = balance + ? WHERE id = ?
```

### 10.3 支出流程（提现扣减）

```
doReduceBalance(user, vo)
  │
  ├── 1. 查询当前 Assets
  │
  ├── 2. 余额校验
  │   if 提现类型(WITHDRAW/TELEGRAM_WITHDRAW):
  │     balance >= amount  ← 不扣冻结（因为提现时已经冻结了）
  │   else 其他类型(下注):
  │     balance - freezeAmount >= amount  ← 可用余额 = balance - 冻结
  │
  ├── 3. ★ 先扣余额（支出先扣钱）
  │   assetsService.doReduceBalanceById(amount, assetsId)
  │   // SQL: UPDATE assets SET balance = balance - ? WHERE id = ?
  │
  └── 4. 后写账变
      balanceChangeService.save(balanceChange)
```

**收入 vs 支出的操作顺序差异**：
- **收入**：先写账变 → 后加余额（保守：账变失败不加余额，不会多给钱）
- **支出**：先扣余额 → 后写账变（保守：扣了钱一定记账，不会丢钱）
- 两种都是"保护平台资产"的思路

## 第11章 并发控制与一致性机制

### 11.1 锁机制全景

| 场景 | 锁类型 | 锁粒度 | Key 模式 | 等待/持有 |
|------|--------|--------|---------|-----------|
| 充值轮询 | Redisson FairLock | 订单级 | `deposit_task_ss_order_id:{orderId}` | 0s/10s |
| 提现轮询 | Redisson FairLock | 订单级 | `withdraw_task_ss_order_id:{orderId}` | 0s/UNLOCK_TIME |
| 余额变更 | Redisson FairLock | 用户级 | `user_assets:{userId}` | WAIT_TIME/UNLOCK_TIME |
| Assets 更新 | @Version 乐观锁 | 行级 | Hibernate 自动 | — |

### 11.2 充值幂等保障（多层防护）

```
Layer 1: Redisson FairLock (orderId)
  → tryLock(0, 10, SECONDS)
  → 0秒等待：拿不到锁直接跳过，等下一轮定时任务
  → 防止同一订单被多个线程并行处理

Layer 2: 订单状态 CAS
  → UPDATE ss_order SET status=SUCCESS WHERE id=? AND status=WAIT
  → count==1 才继续，防止重复上分

Layer 3: BalanceChange 唯一约束
  → UK(changeType, relateId, userId, orderType)
  → 相同充值订单的同类型账变只能写入一次
  → 数据库级别兜底

Layer 4: 业务代码检查
  → if (StringUtils.isEmpty(txId)) → 分支1（归集）
  → else → 分支2（确认）
  → 不同阶段走不同逻辑，不会重复归集
```

### 11.3 提现幂等保障

```
Layer 1: Redisson FairLock (orderId)
  → 同充值

Layer 2: wallet-server ssOrderId 查询幂等
  → TrxWalletOrderEntity order = trxWalletOrderRepository.findBySsOrderId(ssOrderId)
  → if (order != null) return existing txId
  → 防止重复发起链上提现

Layer 3: 订单状态检查
  → isOrderStatus(orderId, ORDER_SUCCESS) → 如果已成功，跳过
  → "重複提現" 日志

Layer 4: BalanceChange UK
  → 同充值
```

### 14.1 值得借鉴的设计

| # | 设计 | 出处 | 为什么好 | 借鉴方式 |

| 5 | **Gas 地址独立** | gasAddress 与出款地址/公司地址分离 | 职责清晰，Gas 地址只负责充 TRX，不持有大额资产 | 直接采用 |
| 6 | **提现 ssOrderId 查询幂等** | TrxDriver.withdrawTraction:298-303 | 先查后做，已有记录直接返回 txId，不重复上链 | 采用并加 UNIQUE 约束强化 |
| 7 | **收入先账变后余额，支出先余额后账变** | UserAssetsBusiness | 保守策略保护平台资产 | 参考思路，但统一为先扣余额后写账变（原子事务内） |
| 8 | **before/after 余额快照链** | BalanceChange.beforeAmount/afterAmount | 完整审计链，可追溯任意时刻余额 | 直接采用 |
| 9 | **配置化出款地址** | TrxWalletConfig 表存储出款地址 | 切换出款地址无需改代码重部署 | 采用（我们用币种配置表实现） |
| 10 | **FairLock + 0秒等待** | deposit() 中 tryLock(0, 10, SECONDS) | 拿不到锁就跳过，不阻塞线程池，等下一轮定时任务自然重试 | 采用思路 |

### 14.2 需要改进的设计

| # | 改进点 | 我们的方案 |
|---|--------|-----------|
| 1 | Polling 2分钟延迟 | 考虑 TRONGrid 事件订阅 / Webhook / 缩短轮询间隔 |
| 2 | 归集-上分无原子性 | 归集成功后立即写入"待上分"状态，确保补偿 |
| 3 | 四步提现无事务 | 统一事务或 Saga 编排 |
| 4 | AES-ECB 模式 | 使用 AES-GCM（AEAD），提供完整性保护 |
| 5 | 无超时订单告警 | 30分钟超时后自动告警 + 人工介入流程 |
| 6 | 批量归集无隔离 | 每用户独立 try-catch + 失败列表汇总 |
| 7 | 无链上 Gas 确认等待 | Gas 发送后等待确认再归集 |
| 8 | DECIMAL(16,2) 精度 | 使用 BIGINT 最小单位存储（我们已确定） |

> **核心发现**：区块链充值是两阶段异步流程（归集→确认→上分），提现是冻结-发送-确认-解冻四步流程
> **最大价值**：BalanceChange 四字段 UK 幂等设计 + 签名外置架构 + 归集两阶段分离
> **最大风险**：归集-上分非原子 + 提现四步无事务 + 金额精度丢失
