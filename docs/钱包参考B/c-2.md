## 第1章 工程概览与技术栈

### 1.1 项目背景

tg-services 是一个面向 Telegram Bot 的在线博彩平台后端，采用 Java + Spring Boot 微服务架构，共 11+ 个微服务。wallet-server 是其中专门负责 **TRON 区块链钱包管理** 的独立服务。

**核心业务场景**: 用户通过 Telegram Bot 发起 USDT(TRC20) 充值/提现，wallet-server 负责：
1. 为每个用户生成独立的 TRON 链上地址
2. 监听用户地址余额变化，确认充值到账
3. 从公司出金地址执行提现转账
4. 定时将分散在用户地址上的资金归集到公司主地址

### 1.2 技术栈全景

| 层次 | 技术选型 | 版本/说明 |
|------|---------|----------|
| 基础框架 | Spring Boot | 2.5.3 |
| ORM | Spring Data JPA + Hibernate | ddl-auto: update，自动建表 |
| 数据库 | MySQL 5 (InnoDB) | cloud_wallet 库 |
| 缓存 | Redis + Lettuce | database: 11 |
| 消息队列 | RabbitMQ | 配置了但实际代码中改用 HTTP 调用替代 |
| 区块链SDK-gRPC | wallet-cli (TRON官方) | GrpcClient → TRON fullnode |
| 区块链SDK-HTTP | 手写 HttpClient | TRON HTTP API (nile.trongrid.io) |
| 加密库 | web3j (ABI编码) + BouncyCastle + SpongyCastle | ABI编解码、签名 |
| 密钥托管 | AWS KMS | AP_SOUTHEAST_1 区域 |
| 私钥加密 | AES/ECB/PKCS5Padding + SHA-1 | 自实现 AES 工具类 |
| 外部签名 | 独立签名服务 (34.92.121.9:80) | RSA 加密通信 |
| 定时任务 | Spring @Scheduled | cron 表达式 |
| API文档 | Swagger | swagger-ui |
| 连接池 | HikariCP | max-pool-size: 16 |

### 2.2 调用链路总览

```
WalletController (REST)
    ↓
WalletTrxBusiness (业务编排：地址生成、转账、余额查询)
    ↓ ←→ TrxDriver (归集/提现编排)
    ↓
TrxWalletService / OrderService / CollectService (数据持久化)
    ↓
TrxWalletRepository... (Spring Data JPA)

区块链交互分两条通道：
通道A (gRPC): TronUtil → GrpcClient → TRON FullNode
通道B (HTTP): WalletTrxBusiness → TRON HTTP API (trongrid.io)
```

### 2.3 关键设计特征

1. **双通道区块链交互**: gRPC (通过 wallet-cli) 和 HTTP (手写 HttpClient) 并存
2. **单 Controller 全接口**: 6 个 API 端点全在一个 WalletController 里
3. **Business 层重逻辑**: WalletTrxBusiness 承担了大量直接的区块链交互逻辑
4. **无事务保护的链上操作**: 区块链操作和数据库操作之间缺乏事务一致性保障
5. **配置即密钥**: 所有敏感地址、私钥通过 AWS KMS 加密后存储在 application.yml

## 第3章 数据模型与表结构

### 3.1 BaseEntity（公共基类）

```java
// 源码: model/BaseEntity.java
@MappedSuperclass
@EntityListeners(AuditingEntityListener.class)
public abstract class BaseEntity implements Serializable {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;                // 自增主键

    @CreatedDate
    @Column(updatable = false)
    private Date createTime;        // 创建时间（JPA审计自动填充）

    @CreatedBy
    private String createBy;        // 创建人（JPA审计自动填充）

    @LastModifiedDate
    private Date updateTime;        // 更新时间

    @LastModifiedBy
    private String updateBy;        // 更新人
}
```

**时区**: JsonFormat 配置 `timezone = "GMT+7"`（泰国曼谷时区）

### 3.2 TrxWallet（用户钱包表）

```sql
CREATE TABLE TrxWallet (
    id             BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id        VARCHAR(10)  COMMENT '会员账号',
    user_name      VARCHAR(30)  COMMENT '会员姓名',
    en_private_key VARCHAR(250) COMMENT '加密后的私钥',
    address        VARCHAR(100) COMMENT '钱包地址',
    create_time    DATETIME,
    create_by      VARCHAR(255),
    update_time    DATETIME,
    update_by      VARCHAR(255),
    INDEX idx_userId (userId)
);
```

**核心要点**:
- **userId 是 VARCHAR(10)**: 不是 BIGINT，说明与 user-server 的 userId (Long) 之间需要类型转换
- **en_private_key**: AES加密后的私钥（Base64编码），密钥 = SHA1(passphrase + address)[:16]
- **一个用户一个地址**: 没有多链/多币种区分，因为所有 TRC20 代币共享同一个 TRON 地址
- **无软删除字段**: 没有 delete_at 字段

### 3.3 TrxWalletOrder（钱包订单表）

```sql
CREATE TABLE TrxWalletOrder (
    id             BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id        VARCHAR(10)  COMMENT '会员账号',
    tx_id          VARCHAR(200) COMMENT '交易id（TRON链上txHash）',
    ss_order_id    VARCHAR(200) COMMENT '充值/提现记录id（关联user-server的SsOrder.id）',
    amount         DECIMAL(16,2) COMMENT '金额',
    from_address   VARCHAR(250) COMMENT '发送地址',
    to_address     VARCHAR(250) COMMENT '接收地址',
    coin_name      VARCHAR(10)  COMMENT '币种类型（USDT/TRX/CXT）',
    order_type     TINYINT(2)   COMMENT '收付方向：1充值 2提现',
    post_type      TINYINT(2)   COMMENT '调用方式：1接口方式 2定时任务方式',
    create_time    DATETIME,
    create_by      VARCHAR(255),
    update_time    DATETIME,
    update_by      VARCHAR(255),
    INDEX idx_user_id (user_id),
    INDEX idx_order_type (order_type),
    INDEX idx_tx_id (tx_id)
);
```

**核心要点**:
- **ss_order_id**: 关联 user-server 的充提订单，是跨服务幂等的关键字段
- **amount DECIMAL(16,2)**: 只保留2位小数，对于 CXT（18位精度）会有精度损失
- **post_type**: 区分是用户主动触发(1)还是定时任务自动处理(2)
- **无状态字段**: 订单只有记录，没有 status，无法追踪交易确认状态

### 3.4 TrxWalletCollect（归集记录表）

```sql
CREATE TABLE trx_wallet_collect (
    id              BIGINT AUTO_INCREMENT PRIMARY KEY,

    update_by       VARCHAR(255),
    INDEX idx_userId (userId),
    INDEX idx_collectType (collectType),
```

## 第5章 钱包地址生成链路（完整拆解）

### 5.1 触发入口

```
用户注册/首次充值 → user-server → POST /wallet/create → WalletController.createWallet()
```

### 5.2 完整流程（代码级）

**源码位置**: `WalletTrxBusiness.createWallet()` (business/WalletTrxBusiness.java)

```
步骤1: 查询用户是否已有钱包地址
    trxWalletService.findAddressByUserId(userId)
    → TrxWalletRepository.findAddressByUserId(userId)
    → 如果已存在，直接返回现有 address

步骤2: 生成 ECKeyPair（以太坊兼容密钥对）
    ECKeyPair ecKeyPair = Keys.createEcKeyPair()
    → 底层使用 java.security.SecureRandom 生成 256位 私钥
    → 导出 publicKey (椭圆曲线 secp256k1)

步骤3: 创建 WalletFile（web3j 密钥文件格式）
    Wallet.createStandard(passphrase, ecKeyPair)
    → passphrase 来源: TronUtil.key = "e7f86b50-97cb-4e56-898d-45de4a00d6db"
    → 生成包含加密私钥的 JSON 钱包文件

步骤4: 获取以太坊格式地址并转换为 TRON 格式
    address = walletFile.getAddress()        // 以太坊 hex 地址(40字符)
    finalAddress = "41" + address             // 添加 TRON 地址前缀 0x41
    finalAddress = encode58Check(finalAddress) // Base58Check 编码 → "T" 开头

步骤5: AES 加密私钥
    privateKey = ecKeyPair.getPrivateKey().toString(16)
    enKey = AES.encrypt(privateKey, passphrase + finalAddress)

    加密细节:
    → AES/ECB/PKCS5Padding
    → 密钥 = SHA-1(passphrase + address) 取前16字节
    → 输出 = Base64(ciphertext)

步骤6: 持久化到数据库
    trxWalletService.save(userId, userName, enKey, finalAddress)
    → INSERT INTO TrxWallet (userId, userName, enPrivateKey, address, ...)

步骤7: 异步更新 user-server 的钱包地址
    trxWalletService.updateUserAddress(request)
    → @Async POST http://user-server/updateWalletAddressById
    → 将 TRON 地址写回用户表
```

### 5.3 地址格式转换详解

```
以太坊私钥 (256bit hex)
    ↓ secp256k1
以太坊公钥 (512bit hex)
    ↓ Keccak-256 取后20字节
以太坊地址 (40 hex chars): "aB1234...cD5678"
    ↓ 加前缀 0x41
TRON hex地址 (42 hex chars): "41aB1234...cD5678"
    ↓ SHA-256 两次取校验码 + Base58
TRON Base58地址 (34 chars): "TJCnKsPa7y5okkXvQAidZBzqx3QyQ6sxMW"
```

### 5.4 安全评估

| 项目 | 评估 | 说明 |
|------|------|------|
| 密钥生成 | 合格 | web3j Keys.createEcKeyPair() 使用 SecureRandom |
| 私钥加密 | ⚠️ 不佳 | AES/ECB 模式不安全（ECB模式不能隐藏数据模式） |
| passphrase | ⚠️ 硬编码 | `TronUtil.key` 是硬编码的固定 UUID |
| 异步更新 | ⚠️ 无重试 | 异步写回 user-server 失败不重试，可能导致用户看不到地址 |

```java
// → POST trongrid.io/wallet/validateaddress

    "function_selector": "transfer(address,uint256)",
    "parameter": parameter,                 // ABI编码后的参数

    "call_value": 0,
    "visible": false
};
POST trongrid.io/wallet/triggersmartcontract → 返回未签名交易

// Step 5: 打包交易（JSON → Protobuf）

// → SHA-256(rawData) → ECKey.sign(hash) → 附加签名
```

### 6.9 充值链路关键设计总结

| 设计点 | 实现方式 | 评价 |
|--------|---------|------|
| 充值模式 | 每用户独立地址 + 轮询确认 | 标准做法，安全隔离 |
| 确认机制 | 定时任务轮询(2分钟) | 合格，但延迟较高 |
| 两阶段处理 | 第一轮获取txId，第二轮确认 | 巧妙，避免长时间等待 |
| 并发控制 | Redisson FairLock + 0秒tryLock | 合格，避免重复处理 |
| 订单状态CAS | WHERE orderStatus=WAIT SET SUCCESS | 安全，防止重复发放 |
| 超时取消 | 30分钟超时 → 系统取消 | 必要的安全阀 |
| Gas代付 | TRX不足30时从gasAddress代转 | 必要，TRC20需要TRX做gas |

## 第8章 归集链路（资金汇聚）

### 8.1 归集的业务含义

用户向各自独立的TRON地址充值USDT后，资金分散在大量用户地址上。**归集** = 将分散资金统一转移到公司主地址，便于：
1. 集中管理和出金
2. 减少被攻击的风险面
3. 统一财务对账

```java
    // 但collectType = COLLECT (定时任务归集)
```

### 8.4 归集关键问题

| 问题 | 说明 |
|------|------|
| **无批量优化** | 遍历所有钱包逐个归集，用户量大时非常慢 |
| **无余额过滤** | 没有先批量查余额再决定归集哪些，浪费gas |
| **串行执行** | 没有并发处理，一个失败不影响后续但耗时长 |
| **Gas费开销** | 每笔TRC20归集需要消耗约30 TRX gas，大量用户时gas费可观 |
| **无最小归集阈值** | `minSumAmount = 0`，任何余额都会触发归集 |

```java
        .keyId(keyId)                          // mrk-2017a36cde86493d...

        .build();

// 解密
public String decryptData(String afterEncryptionData) {
    SdkBytes sdkBytes = SdkBytes.fromByteArray(Base64.getDecoder().decode(afterEncryptionData));
    DecryptRequest req = DecryptRequest.builder()
        .ciphertextBlob(sdkBytes)
        .keyId(keyId)
        .build();
    DecryptResponse resp = kmsClient.decrypt(req);

}
```

```java
// 源码: util/AES.java

// 密钥派生: SHA-1 → 取前16字节 → AES-128
private static byte[] getKey(String secret) {
    byte[] bytes = secret.getBytes("UTF-8");
    MessageDigest sha = MessageDigest.getInstance("SHA-1");
    byte[] digest = sha.digest(bytes);
    return Arrays.copyOf(digest, 16);  // 取前16字节
}

// 加密: AES/ECB/PKCS5Padding
public static String encrypt(String plaintext, String secret) {
    SecretKeySpec keySpec = new SecretKeySpec(getKey(secret), "AES");
    Cipher cipher = Cipher.getInstance("AES/ECB/PKCS5Padding");
    cipher.init(Cipher.ENCRYPT_MODE, keySpec);
    return Base64.getEncoder().encodeToString(cipher.doFinal(plaintext.getBytes("UTF-8")));
}
```

## 第14章 缺陷发现与风险评估

### 14.1 P0 级缺陷（必须修复）

#### P0-1: AES/ECB 模式加密用户私钥

**位置**: `util/AES.java:19`
```java
private static final String model = "AES/ECB/PKCS5Padding";
```

**问题**: ECB(Electronic Codebook)模式对相同明文块产生相同密文块，不提供语义安全性。攻击者可以通过模式分析推断私钥信息。

**应改为**: `AES/CBC/PKCS5Padding` 或 `AES/GCM/NoPadding`，并使用随机IV。

#### P0-2: passphrase 硬编码在源码中

**位置**: `client/trx/TronUtil.java:26`
```java
public static final String key = "e7f86b50-97cb-4e56-898d-45de4a00d6db";
```

**问题**: 这个 passphrase 用于所有用户私钥的 AES 加密。源码泄露 = 配合数据库泄露可解密所有用户私钥 = 全部资产被盗。

**应改为**: 从 AWS KMS 或 HSM 获取，绝不能硬编码。

#### P0-3: 归集/提现操作无事务保护

**位置**: `driver/TrxDriver.java:singleCollect()`, `withdrawTraction()`

**问题**: 链上转账成功但数据库记录失败的场景：
```
1. 链上转账执行成功 → 返回 txId ✅
2. trxWalletCollectService.save() 失败 ❌
3. trxWalletOrderService.save() 失败 ❌
→ 资金已转移但无记录，对账无法追溯
```

**应改为**: 先写入数据库记录(status=PENDING)，再执行链上操作，最后更新状态。

#### P0-4: 充值余额检查与归集非原子

**位置**: `AsyncVirtualDepositService.checkCrypotcurrencyOrder()` + `TrxDriver.singleCollect()`

**问题**:
```
时刻T1: user-server 查询链上余额 = 100 USDT ✅
时刻T2: 另一个进程/服务也对同一地址发起归集
时刻T3: user-server 发起归集，但余额已不足
→ 归集交易失败，但订单状态已在处理中
```

### 14.2 P1 级缺陷（建议修复）

#### P1-1: TrxWalletOrder 无状态字段

**问题**: 订单表只记录了插入时的信息，没有 `status` 字段追踪交易确认状态。导致：
- 无法直接从 wallet-server 查询哪些交易待确认
- 必须每次去链上查询确认状态
- 无法处理链上交易失败的情况

#### P1-2: 金额精度 DECIMAL(16,2) 不足

**位置**: `TrxWalletOrderEntity.java:43`, `TrxWalletCollectEntity.java:41`

**问题**: CXT 代币精度为 18 位，但数据库只存 2 位小数。TRC20 的 USDT 虽然精度为 6 位，但也只保留了 2 位，存在精度损失。

#### P1-3: AWS 凭据明文存储在 application.yml

**位置**: `application.yml:91-93`
```yaml
aws:
  accessKeyId: YOUR_AWS_ACCESS_KEY_ID
  secretAccessKey: YOUR_AWS_SECRET_ACCESS_KEY
```

**问题**: 凭据直接存储在配置文件中，被纳入版本控制。

#### P1-4: 每次 KMS 操作都创建新客户端

**位置**: `driver/AwsKmsDriver.java:98-108`
```java
private KmsClient createKmsClient() {
    return KmsClient.builder()
        .region(Region.AP_SOUTHEAST_1)
        .credentialsProvider(StaticCredentialsProvider.create(awsCreds))
        .build();
}
```

**问题**: 每次加解密都创建新的 KmsClient，开销大且可能导致连接耗尽。应使用单例。

### 14.3 P2 级缺陷（改进建议）

#### P2-1: 归集无最小阈值和并发优化

**问题**: `minSumAmount = 0`，即使0.01 USDT也会触发归集，消耗30 TRX gas。应设合理阈值。

#### P2-2: 服务间调用全部硬编码 HTTP

**问题**: 无服务发现、无负载均衡、无熔断、无重试。生产环境不够健壮。

#### P2-3: HttpClient 不复用

**位置**: `util/TronUtils.java:450-468`
```java
CloseableHttpClient httpClient = HttpClients.createDefault();  // 每次创建新client
```

**问题**: 每次 HTTP 调用创建新的 HttpClient，不复用连接池。

#### P2-4: 签名服务单点

**位置**: `application.yml:42`
```yaml
sign:
  url: http://34.92.121.9:80
```

**问题**: 签名服务是固定IP、HTTP协议、单点部署。是提现链路的单点故障。

## 第15章 借鉴清单与规避清单

### 15.1 值得借鉴的设计（5项）

| # | 设计点 | 来源 | 借鉴价值 |
|---|--------|------|---------|
| 1 | **每用户独立链上地址** | WalletTrxBusiness.createWallet() | 充值天然隔离，无需中心化分配，资金归属清晰 |
| 2 | **两阶段充值确认** | AsyncVirtualDepositService.checkCrypotcurrencyOrder() | 第一轮获取txId、第二轮确认结果，避免长时间阻塞，可重入设计 |
| 3 | **提现幂等检查(ssOrderId)** | TrxDriver.withdrawTraction() | 用外部订单ID做幂等键，查重后直接返回已有txId，防止重复出金 |
| 4 | **Gas费自动代付** | TrxDriver.singleCollect() / withdrawTraction() | TRC20需要TRX做gas，自动检测并从gasAddress补充，用户无感 |
| 5 | **AWS KMS 保护公司密钥** | AwsKmsDriver + application.yml | 公司地址、出金地址、gas地址的私钥用KMS加密存储，启动时解密 |

### 15.2 必须规避的设计（7项）

| # | 缺陷 | 来源 | 规避策略 |
|---|------|------|---------|
| 1 | **AES/ECB加密用户私钥** | AES.java | 我们使用 AES-256-GCM + 随机IV/nonce |
| 2 | **passphrase硬编码** | TronUtil.key | 从配置中心/KMS获取，不入源码 |
| 3 | **链上操作无事务保护** | TrxDriver.singleCollect() | 先写DB(PENDING) → 链上操作 → 更新DB状态 |
| 4 | **订单表无状态字段** | TrxWalletOrderEntity | 我们的链上交易记录表必须有 status 字段 |
| 5 | **金额精度不足** | DECIMAL(16,2) | 我们使用 BIGINT 存储最小单位(如 satoshi/sun) |
| 6 | **归集无阈值无并发** | CollectTrxJob/collectTrc20 | 设最小归集阈值 + 批量查余额 + 并发归集 |
| 7 | **全HTTP无服务发现** | 所有跨服务调用 | 我们用 Kitex RPC + ETCD 服务发现 |

### 15.3 需要调整借鉴的设计（5项）

| # | 原设计 | 调整方向 |
|---|--------|---------|
| 1 | **定时轮询确认(2分钟)** | 保持轮询机制但增加WebSocket/链上事件监听作为辅助，降低确认延迟 |
| 2 | **Redisson FairLock(0秒tryLock)** | 借鉴分布式锁思路，但在Go中用Redis分布式锁实现，且增加重试而非直接放弃 |
| 3 | **独立签名服务(提现)** | 借鉴热冷钱包分离思想，但签名服务要HTTPS+mTLS+高可用 |
| 4 | **单表存储所有币种订单** | 保留但增加 status 字段、扩展 chain_type 字段支持多链 |
| 5 | **异步更新user-server地址** | 改为同步确认，失败则重试，确保用户能立即看到地址 |
