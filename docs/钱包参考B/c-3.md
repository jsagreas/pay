### 3.2 各表详细分析

#### TrxWallet — 用户钱包表

```sql
CREATE TABLE TrxWallet (
    id          BIGINT AUTO_INCREMENT PRIMARY KEY,
    userId      VARCHAR(10) COMMENT '会员账号',    -- 索引
    userName    VARCHAR(30) COMMENT '会员姓名',
    enPrivateKey VARCHAR(250) COMMENT '加密后的私钥', -- AES加密存储
    address     VARCHAR(100) COMMENT '钱包地址',    -- TRON T-address
    create_time DATETIME,
    create_by   VARCHAR(255),
    update_time DATETIME,
    update_by   VARCHAR(255),
    INDEX idx_userId (userId)
);
```

**关键设计决策**：
- **每用户一个独立链上地址**：userId → address 一对一映射
- **私钥加密存储**：enPrivateKey 用AES加密，密钥=SHA1(passphrase+address)[:16]
- **无余额字段**：余额通过链上实时查询获取，不在本地存储

**问题标注**：
- `userId VARCHAR(10)` — 长度仅10字符，对大用户量不友好
- 无UNIQUE约束在userId或address上 — 理论上可能重复创建
- 私钥和地址在同一张表 — 一旦DB泄露，配合passphrase可解密所有私钥

#### TrxWalletOrder — 交易订单表

```sql
CREATE TABLE TrxWalletOrder (
    id          BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id     VARCHAR(10) COMMENT '会员账号',     -- 索引
    tx_id       VARCHAR(200) COMMENT '交易id',      -- 索引，链上txHash
    ss_order_id VARCHAR(200) COMMENT '充值/提现记录id', -- 业务订单号
    amount      DECIMAL(16,2) COMMENT '金额',       -- !!!仅2位小数
    from_address VARCHAR(250) COMMENT '发送地址',
    to_address  VARCHAR(250) COMMENT '接收地址',
    coin_name   VARCHAR(10) COMMENT '币种类型',     -- TRX/USDT/CXT
    order_type  TINYINT(2) COMMENT '收付方向',      -- 1充值 2提现
    post_type   TINYINT(2) COMMENT '1:接口方式,2:定时任务方式',
    create_time DATETIME,
    create_by   VARCHAR(255),
    update_time DATETIME,
    update_by   VARCHAR(255),
    INDEX idx_user_id (user_id),
    INDEX idx_order_type (order_type),
    INDEX idx_tx_id (tx_id)
);
```

**关键设计决策**：
- **ssOrderId**：业务系统的订单号，用于提现幂等性检查
- **txId**：链上交易哈希，用于交易状态确认
- **orderType**：1=充值（归集入账），2=提现
- **postType**：区分手动触发(1)和定时任务触发(2)

**问题标注**：
- `DECIMAL(16,2)` — **虚拟币仅2位小数是严重缺陷**，USDT有6位，TRX有6位，CXT有18位
- ssOrderId无UNIQUE索引 — 幂等性靠查询而非约束保障
- 无交易状态字段 — 无法追踪交易是pending/confirmed/failed

#### TrxWalletCollect — 归集记录表

```sql
CREATE TABLE trx_wallet_collect (
    id              BIGINT AUTO_INCREMENT PRIMARY KEY,
    userId          VARCHAR(10) COMMENT '会员账号',  -- 索引
    txId            VARCHAR(200) COMMENT '交易id',   -- 索引
    collectAddress  VARCHAR(250) COMMENT '汇款账号', -- from地址
    amount          DECIMAL(16,2) COMMENT '汇款金额',
    collectType     TINYINT(2) COMMENT '归集类型',   -- 1单笔即时 2定时任务
    create_time     DATETIME,
    INDEX idx_userId (userId),
    INDEX idx_collectType (collectType),
    INDEX idx_txId (txId)
);
```

**关键设计决策**：
- 归集记录独立表，与订单表分离
- collectType区分触发方式

**问题标注**：
- 缺少toAddress（归集目标地址）
- 缺少coinName（币种类型）
- 同样`DECIMAL(16,2)`精度问题

#### TrxWalletConfig — 动态配置表


## 4. 地址生成与私钥管理

### 4.1 钱包地址生成完整流程

```
createWallet(userId, userName)
  │
  ├── 1. 查询已有地址：trxWalletRepository.findAddressByUserId(userId)
  │   └── 若已存在 → 直接返回（幂等保护）
  │
  ├── 2. 生成密钥对：ECKeyPair ecKeyPair = Keys.createEcKeyPair()
  │   └── Web3j的ECDSA secp256k1密钥对生成
  │
  ├── 3. 创建标准钱包文件：Wallet.createStandard(passphrase, ecKeyPair)
  │   └── Web3j标准钱包格式，包含scrypt KDF
  │
  ├── 4. 转换为TRON地址：
  │   ├── walletFile.getAddress() → 以太坊格式hex地址
  │   ├── 添加TRON前缀："41" + originAddress
  │   └── fromHexAddress("41" + hex) → Base58Check编码 → T-address
  │
  ├── 5. 私钥加密存储：
  │   ├── 提取裸私钥hex：Numeric.toHexStringNoPrefix(ecKeyPair.getPrivateKey())
  │   ├── 构造加密密钥：passphrase + finalAddress（KMS解密后的助记词+地址拼接）
  │   └── AES加密：encrypt(privateKeyHex, passphrase + address) → enKey
  │
  ├── 6. 持久化：trxWalletService.save(userId, userName, enKey, finalAddress)
  │
  └── 7. 异步回调：@Async updateUserAddress() → POST /user/updateWalletAddressById
      └── 通知用户服务更新钱包地址
```

### 4.2 TRON地址编码原理

```
以太坊地址:  0x + 20字节hex (40字符)
TRON地址:   "41" + 20字节hex → Base58Check编码 → T-开头的34字符

Base58Check编码过程:
  1. payload = 0x41 + address_bytes (21字节)
  2. checksum = SHA256(SHA256(payload))[:4]
  3. result = Base58Encode(payload + checksum)

示例: 41 + a614f803b6fd780986a42c78ec9c7f77e6ded13c
    → Base58Check → TRrQtwH9Z4kPZeuMMhvQ5ecL7Syy8vPqbq
```

### 4.3 私钥加密方案详解

```java
// 加密算法：AES/ECB/PKCS5Padding（128位密钥）
// 密钥派生：SHA-1(secret) → 取前16字节

public static String encrypt(String strToEncrypt, String secret) throws Exception {
    // secret = passphrase + address（如"heart essence...drill" + "TRrQtw...pbq"）
    byte[] bytes = secret.getBytes("UTF-8");
    MessageDigest sha = MessageDigest.getInstance("SHA-1");
    byte[] digest = sha.digest(bytes);          // SHA-1: 20字节
    byte[] key = Arrays.copyOf(digest, 16);     // 截取前16字节作为AES-128密钥

    SecretKeySpec secretKeySpec = new SecretKeySpec(key, "AES");
    Cipher cipher = Cipher.getInstance("AES/ECB/PKCS5Padding");
    cipher.init(Cipher.ENCRYPT_MODE, secretKeySpec);
    return Base64.getEncoder().encodeToString(cipher.doFinal(strToEncrypt.getBytes("UTF-8")));
}
```

**加密密钥组成**：
```
passphrase = "heart essence drill cave negative machine weather erosion civil general holiday three"
              （12个英文单词的助记词，通过AWS KMS加密存储在application.yml中）
address    = "TRrQtwH9Z4kPZeuMMhvQ5ecL7Syy8vPqbq"
              （用户的TRON地址，每个用户不同）

最终密钥 = SHA1(passphrase + address)[:16]
```

**安全分析**：
- **优点**：每个用户的AES密钥不同（因为address不同），单个密钥泄露不影响其他用户
- **缺点**：passphrase是全局共享的，一旦泄露+DB泄露 → 所有私钥可被解密
- **缺点**：AES/ECB模式没有IV，相同明文产生相同密文
- **缺点**：SHA-1已被认为不安全，应使用SHA-256或PBKDF2

## 5. 区块链充值完整链路

### 5.1 充值模式：独立地址 + 归集

本工程采用**独立充值地址模式**，这是虚拟币充值的标准架构：

```
┌─────────┐    USDT     ┌──────────────┐   归集   ┌───────────────┐
│  用户    │ ─────────→  │ 用户独立地址   │ ─────→  │ 公司主地址     │
│(外部钱包) │            │ (per-user)    │         │ (companyAddr)  │
└─────────┘             └──────────────┘         └───────────────┘
                              ↑
                         系统生成并管理
                         私钥加密存储
```

## 6. 归集（Collection/Sweep）机制

### 6.1 归集的核心意义

独立充值地址模式下，用户的USDT分散在各自地址中，必须**归集**到公司主地址才能：
- 统一管理资金
- 降低出款时的gas消耗
- 便于财务对账

### 6.2 定时批量归集

```java
// TrxDriver.singleCollect() — 单笔归集核心逻辑
public TrxWalletResult singleCollect(String fromAddress, String userId,
        String coinName, String ssOrderId, BigDecimal reChargeAmount,
        int collectCode, int postType) {

    // 1. 确定归集金额
    BigDecimal trxBalance = reChargeAmount;
    if (null == reChargeAmount) {
        trxBalance = getAccountBalance(fromAddress, coinName); // 链上查询
    }

    // 2. 余额检查（>0才归集）
    if (NumberUtil.isGreaterOrEqual(minSumAmount, trxBalance)) {
        log.error("余额不足0");
        return null;
    }

    // 3. TRC20代币：检查并补充gas
    String tokenContract = TRXContractEnum.getByTokenContractByContractName(coinName);
    if (!TRXContractEnum.TRX.getContract().equalsIgnoreCase(tokenContract)) {
        BigDecimal addTrx = getAccountBalance(fromAddress, TRXContractEnum.TRX.getContract());
        if (FEE_BAL.compareTo(addTrx) > 0) { // TRX余额 < 30
            // 从gasAddress向用户地址转30 TRX
            TrxWalletResult gasResult = createTransactionForTrx(
                awsKmsDriver.decryptData(gasPassword),
                FEE_BAL, newGasAddress, fromAddress,
                TRXContractEnum.TRX.getContract(), ssOrderId);
            // 记录gas转账订单
            trxWalletOrderService.save(userId, newGasAddress, fromAddress,
                gasResult.getTxId(), ssOrderId, FEE_BAL,
                TrxOrderTypeEnum.RECHARGE.getCode(), "TRX", postType);
        }
    }

    // 4. 执行归集转账（用户地址 → 主地址）
    TrxWalletResult result = createTransactionForTrx(
        null,  // null表示需要从DB获取私钥并解密
        trxBalance, fromAddress, mainAddress, tokenContract, ssOrderId);

    // 5. 记录归集信息
    reverseNoticeCollectDetail(userId, fromAddress, result.getTxId(),
        trxBalance, collectCode, ssOrderId, coinName, postType);

    return result;
}
```

### 6.5 归集的关键问题

| 问题 | 说明 | 影响 |
|------|------|------|
| **gas确认未等待** | 补充gas后立即发起TRC20转账，gas交易可能还未确认 | TRC20转账可能因gas不足失败 |
| **无重试机制** | 单笔失败直接return null，不影响下一笔 | 失败的归集可能被遗漏 |
| **全表扫描** | `findAll()`加载所有钱包 | 用户量大时性能问题 |
| **串行执行** | for循环逐个处理 | 1000个用户地址可能需要数小时 |
| **无gas回收** | 补充30 TRX后若TRC20转账失败，gas不会回收 | TRX资产损耗 |
| **ssOrderId不唯一** | gas转账和代币转账使用同一ssOrderId | 订单号冲突 |

```java
        }
    }
```

### 7.3 提现的关键安全设计

**出款地址私钥不在本服务**：

## 8. Gas费管理策略

### 8.1 Gas费基本概念

TRON链上的Gas费（能量+带宽）规则：
- **TRX主币转账**：仅消耗带宽（bandwidth），通常免费（新账户除外）
- **TRC20代币转账**：消耗能量（energy），需要约30-50 TRX的能量费用
- **智能合约调用**：消耗能量，费用由`fee_limit`限制

### 8.2 本工程Gas策略

### 8.4 Gas策略问题

| 问题 | 详情 |
|------|------|
| 固定30 TRX | 实际TRC20转账消耗约5-15 TRX，30 TRX偏多但留有余量 |
| 无gas回收 | 补gas后若后续操作失败，TRX留在用户地址无法自动回收 |
| 无等待确认 | gas转账后立即发起TRC20转账，可能gas还未到账 |
| gas地址余额未监控 | 无告警机制，gas地址TRX耗尽将导致所有归集/提现失败 |
| fee_limit固定50 TRX | `fee_limit: 50000000L`（50 TRX），TRC20转账的max fee |

### 9.2 本地签名（归集用）

```java
// WalletTrxBusiness.signTransaction2Byte()
private static byte[] signTransaction2Byte(byte[] transaction, byte[] privateKey)
        throws InvalidProtocolBufferException {
    ECKey ecKey = ECKey.fromPrivate(privateKey);
    Protocol.Transaction tx = Protocol.Transaction.parseFrom(transaction);
    byte[] rawdata = tx.getRawData().toByteArray();
    byte[] hash = Sha256Sm3Hash.hash(rawdata);        // SHA256哈希
    byte[] sign = ecKey.sign(hash).toByteArray();       // ECDSA签名
    return tx.toBuilder()
        .addSignature(ByteString.copyFrom(sign))        // 附加签名到交易
        .build().toByteArray();
}
```

### 9.3 外部签名服务（提现用）

```java
// WalletTrxBusiness.sendSign()
private byte[] sendSign(String signUrl, Protocol.Transaction tx,
        String ssOrderId, String from) {
    // 1. 从DB获取签名服务配置
    List<TrxWalletConfigEntity> list = trxWalletConfigRepository.findByType(CommonConstant.SIGN);

    // 2. 计算交易哈希
    byte[] txId = Sha256Sm3Hash.hash(tx.getRawData().toByteArray());
    String signTxid = Base64.getEncoder().encodeToString(txId);

    // 3. 构造签名请求
    Map<String, Object> signMap = new HashMap<>();
    signMap.put("orderNo", ssOrderId);
    signMap.put("outAddressNo", from);
    signMap.put("chainType", "TRON");
    signMap.put("hashId", signTxid);  // 交易哈希（待签名内容）

    // 4. RSA加密请求体
    String rsaPublicKey = list.stream()
        .filter(obj -> "rsa".equals(obj.getCode()))
        .findFirst().get().getValue();
    String enSign = RSAUtil.encrypt(
        JSONObject.toJSONString(signMap), rsaPublicKey); // RSA公钥加密

    // 5. 发送签名请求
    HttpResponse response = HttpRequest
        .post(signUrl + "/api/admin/chain/sign")
        .header("PLAT_CODE", platCode)    // 平台商户号
        .body(enSign)                      // 加密后的请求体
        .timeout(60000)                    // 60秒超时
        .execute();

    // 6. 解析签名结果
    JSONObject result = JSONObject.parseObject(response.body());
    if ("200".equals(result.get("code"))) {
        byte[] deSign = Base64.getDecoder().decode(result.getString("data"));
        // 将签名附加到交易
        return tx.toBuilder()
            .addSignature(ByteString.copyFrom(deSign))
            .build().toByteArray();
    }
    return null;
}
```

### 9.4 签名路由决策

```java
// WalletTrxBusiness.trc20Transaction() 中的签名路由
String deDispensingAddress = ...; // 从DB获取出款地址
```

**设计思想**：
- 出款地址的私钥由独立签名服务管理，wallet-server不持有 → **分离权限**
- 用户充值地址的私钥在wallet-server管理 → 归集效率高
- 签名服务通过RSA加密通信 → 防止中间人攻击

### 10.2 ABI编码详解

```java
// transfer(address,uint256) 的ABI编码
// Web3j的FunctionEncoder将参数编码为EVM标准格式

public String encoderAbi(String hexAddress, BigInteger amount) {
    ArrayList<Type> inputParameter = new ArrayList<>();
    inputParameter.add(new org.web3j.abi.datatypes.Address(hexAddress)); // 32字节
    inputParameter.add(new Uint256(amount));                              // 32字节
    return FunctionEncoder.encodeConstructor(inputParameter);
    // 返回64字节hex字符串（不含function selector，selector在function_selector参数中）
}

// 示例：
// 转账100 USDT到TRrQtwH9Z4kPZeuMMhvQ5ecL7Syy8vPqbq
// hexAddress = "a614f803b6fd780986a42c78ec9c7f77e6ded13c"  （去掉41前缀）
// amount = 100 * 10^6 = 100000000  （USDT精度6位）
// 编码后 = 000000000000000000000000a614f803b6fd780986a42c78ec9c7f77e6ded13c
//          0000000000000000000000000000000000000000000000000000000005f5e100
```

### 10.3 TRC20余额查询

```java
// Trc20.balanceOf() — 查询TRC20代币余额
public static BigDecimal balanceOf(String address, String contract) {
    String addressHex = TronUtil.toHexAddress(address);
    String param = encoderBalanceOfAbi(addressHex.substring(2));

    // 调用合约只读方法（不消耗能量）
    String result = SmartContract.triggerConstantContract(
        address, contract, "balanceOf(address)", param);

    BigDecimal balance = decoderBalanceOf(result);  // 解码uint256
    Integer decimals = decimals(contract);           // 查询合约精度
    return balance.divide(BigDecimal.TEN.pow(decimals)); // 还原为人类可读
}
```

## 11. 金额精度与单位转换

### 11.1 链上金额表示

TRON链上金额以**最小单位**（整数）表示：
- TRX: 1 TRX = 1,000,000 SUN（10^6）
- USDT: 1 USDT = 1,000,000 最小单位（10^6）
- CXT: 1 CXT = 10^18 最小单位

### 11.2 代码中的精度处理

```java
// 发送时：人类可读金额 → 链上整数
BigInteger integer = needSendAmount
    .multiply(BigDecimal.TEN.pow(decimals))  // 乘以10^6或10^18
    .toBigInteger();                          // 转为整数

// 查询时：链上整数 → 人类可读金额
balance = balanceN.divide(BigDecimal.TEN.pow(TRXContractEnum.TRX.getDecimal()));
// 1000000 / 10^6 = 1.0 TRX

// TRX主币转账时：
amount = needSendAmount.multiply(BigDecimal.TEN.pow(TRXContractEnum.TRX.getDecimal()));
transaction = createTransaction(fromAddress, toAddress, amount.longValueExact());
// longValueExact() 会在有小数部分时抛出ArithmeticException
```

### 11.3 精度问题汇总

| 问题 | 详情 | 影响 |
|------|------|------|
| DB精度`DECIMAL(16,2)` | 仅2位小数 | USDT需6位、CXT需18位，存储时精度丢失 |
| Java使用BigDecimal | 计算时精度足够 | 但存DB时被截断 |
| 未设置BigDecimal RoundingMode | divide操作可能抛ArithmeticException | 除不尽时崩溃 |
| longValueExact | TRX转账金额转long | 若计算有误产生小数则崩溃 |

## 12. 密钥安全架构

### 12.1 多层密钥保护体系

```
┌─────────────────────────────────────────────────────────────┐
│                     Layer 1: AWS KMS                         │
│  application.yml中的敏感值 → KMS加密后的密文                   │
│  (companyAddress, passphrase, gasAddress, gasPrivateKey)      │
│  解密需要：AWS accessKeyId + secretAccessKey + keyId          │
└─────────────────────┬───────────────────────────────────────┘
                      │ decryptData()
                      ▼
┌─────────────────────────────────────────────────────────────┐
│                     Layer 2: AES-128                          │
│  用户私钥 → AES/ECB/PKCS5Padding加密                          │
│  密钥 = SHA1(passphrase + address)[:16]                      │
│  passphrase来自Layer 1解密                                    │
└─────────────────────┬───────────────────────────────────────┘
                      │ decrypt()
                      ▼
┌─────────────────────────────────────────────────────────────┐
│                     Layer 3: DB Config                        │
│  出款地址 → TrxWalletConfig表                                 │
│  (运行时可修改，无需重启服务)                                    │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                     Layer 4: External Signing                │
│  出款私钥 → 独立签名服务                                       │
│  通信加密：RSA公钥加密请求体                                    │
│  认证：PLAT_CODE商户号 + RSA密钥对                             │
└─────────────────────────────────────────────────────────────┘
```

### 12.2 密钥分类与存储位置

| 密钥类型 | 存储位置 | 保护方式 | 用途 |
|---------|---------|---------|------|
| AWS accessKeyId/secretAccessKey | application.yml | **明文！** | KMS API认证 |
| KMS keyId | application.yml | 明文（本身不敏感） | KMS密钥标识 |
| 公司主地址 | application.yml → KMS密文 | KMS加密 | 归集目标地址 |
| passphrase（助记词） | application.yml → KMS密文 | KMS加密 | 用户私钥AES加密密钥的一部分 |
| Gas地址+私钥 | application.yml → KMS密文 | KMS加密 | Gas费操作 |
| 用户私钥 | MySQL trx_wallet.en_private_key | AES加密 | 归集签名 |
| 出款地址 | MySQL trx_wallet_config | 明文 | 提现出款 |
| 出款私钥 | 外部签名服务 | RSA加密通信 | 提现签名 |
| RSA公钥 | MySQL trx_wallet_config | 明文 | 与签名服务通信 |

### 12.3 安全问题清单

| 严重度 | 问题 | 说明 |
|--------|------|------|
| **P0** | AWS凭证明文存储 | application.yml中`accessKeyId`和`secretAccessKey`明文 |
| **P0** | AES/ECB模式 | ECB没有IV，相同明文产生相同密文，不安全 |
| **P1** | SHA-1密钥派生 | SHA-1已不安全，应使用PBKDF2/Argon2 |
| **P1** | 私钥和地址同表 | DB泄露+passphrase泄露 = 全部私钥暴露 |
| **P1** | RSA-512 | RSAUtil使用512位密钥，已可被破解 |
| **P2** | Web安全完全禁用 | `anyRequest().permitAll()` + `ignoring("**")` |
| **P2** | KmsClient未复用 | 每次加解密创建新连接，效率低 |

## 16. 幂等性与并发控制

### 16.1 幂等性措施

| 操作 | 幂等方式 | 实现 | 可靠性 |
|------|---------|------|--------|
| 创建钱包 | 查询已有地址 | `findAddressByUserId(userId)` | 中等（无UNIQUE约束，并发可能重复） |
| 提现 | 查询ssOrderId | `findBySsOrderId(ssOrderId)` + `coin_name='USDT'` | 低（硬编码USDT、无DB约束） |
| 归集 | 无 | - | **无幂等保护** |
| gas补充 | 无 | - | **无幂等保护** |

### 16.2 并发控制

**本工程没有任何并发控制机制**：
- 无分布式锁
- 无乐观锁
- 无数据库行锁
- 无Redis锁

**风险场景**：
1. **双花风险**：同一提现请求并发到达 → ssOrderId查询还没结果 → 两笔都执行
2. **重复归集**：定时归集+手动归集同时触发 → 同一地址被归集两次
3. **重复创建钱包**：并发请求 → 同一userId创建两个地址

### 16.3 事务管理

```java
// Service层类级别@Transactional
@Transactional(rollbackFor = Exception.class)
public class TrxWalletService { ... }

@Transactional(rollbackFor = Exception.class)
public class TrxWalletOrderService { ... }

@Transactional(rollbackFor = Exception.class)
public class TrxWalletCollectService { ... }
```

**问题**：
- `@Transactional`放在类级别，所有方法（包括查询）都开启事务
- Driver/Business层无事务管理 → 归集的"save collect + save order"不在同一事务中
- `reverseNoticeCollectDetail`中两次save操作可能出现一成功一失败

## 18. 借鉴取舍评估（对标ser-wallet）

### 18.1 值得借鉴（8项）

| # | 借鉴项 | 原工程实现 | ser-wallet落地方式 |
|---|--------|-----------|-------------------|
| 1 | **独立充值地址模式** | 每用户一个TRON地址，私钥加密存储 | 核心架构照搬。Go实现：每用户生成独立链上地址，私钥AES-256-GCM加密后存DB |

| 8 | **链上余额实时查询** | 余额不在本地存储，每次链上查询 | 参考+改进：链上查询为准，本地缓存为辅（Redis缓存+TTL），查询走链上，展示可用缓存 |

### 18.2 需要调整（7项）

| # | 调整项 | 原实现问题 | ser-wallet调整方案 |
|---|--------|-----------|-------------------|
| 1 | **金额精度** | DECIMAL(16,2)，仅2位小数 | 使用DECIMAL(30,18) UNSIGNED，满足所有主流币种精度需求 |
| 2 | **gas等待确认** | gas补充后不等确认就执行TRC20转账 | 补充gas后轮询确认（最多30秒/10次），确认后再发起代币转账 |
| 3 | **串行归集** | for循环串行处理所有用户 | 并发归集：将用户地址分批，每批并行处理（goroutine + semaphore控制并发数） |
| 4 | **AES加密模式** | AES/ECB/PKCS5Padding，无IV | 升级为AES-256-GCM（带IV+认证标签），每次加密生成随机IV |
| 5 | **密钥派生** | SHA-1截取前16字节 | 改用PBKDF2-HMAC-SHA256，迭代100000次，盐值=address |
| 6 | **事务边界** | Service类级别@Transactional | 精确控制事务边界，归集的collect+order记录在同一事务中 |
| 7 | **错误处理** | catch(Exception) + return null | 定义业务错误码体系，区分"余额不足"/"签名失败"/"网络超时"等 |

### 18.3 需要规避/丢弃（10项）

| # | 规避项 | 原实现 | 为什么丢弃 |
|---|--------|--------|-----------|
| 1 | **无并发控制** | 全局无锁 | 金融系统不可接受。ser-wallet必须有分布式锁+乐观锁双保险 |
| 2 | **无充值检测** | 完全靠定时归集发现充值 | 用户体验极差（最长24小时）。ser-wallet需实现TRC20 Transfer事件监听或高频轮询 |
| 3 | **无交易状态追踪** | 订单表无status字段 | 金融对账基础。ser-wallet订单表必须有完整状态机（pending→broadcasting→confirmed→failed） |
| 4 | **AWS凭证明文** | application.yml直接写死 | 使用环境变量/Vault/K8S Secret，绝不在配置文件写明文凭证 |
| 5 | **Web安全全禁** | `anyRequest().permitAll()` | ser-wallet必须有认证（JWT/Token）+ 接口权限控制 |
| 6 | **RSA-512** | RSAUtil使用512位密钥 | 最低RSA-2048，推荐ECDSA P-256 |
| 7 | **无参数校验** | Controller无校验 | ser-wallet使用Hertz binding+validator，所有入参严格校验 |
| 8 | **异常吞没** | 多处catch后return null | 必须有结构化错误返回，日志+监控+告警 |
| 9 | **全表扫描归集** | `findAll()`加载全部用户 | 分页+条件查询（仅归集有余额的地址），或维护"待归集"队列 |
| 10 | **JPA ddl-auto: update** | Hibernate自动建表 | 生产环境绝不用ddl-auto，必须Migration管理（Go: golang-migrate/goose） |

> **总结**：本工程是一个功能完整但工程质量一般的TRON虚拟币钱包实现。其核心价值在于提供了区块链钱包的**完整业务流程参考**（地址生成→私钥管理→余额查询→归集→提现→交易确认），以及**分层密钥安全架构**（KMS→AES→外部签名）。但在并发控制、金额精度、充值检测、交易状态追踪、安全防护等方面存在明显不足。ser-wallet应取其架构精华（独立地址模式、Gas管理、密钥分层、签名分离），大幅升级安全和可靠性（并发锁、事务管理、幂等约束、充值监听、状态机、重试机制）。
