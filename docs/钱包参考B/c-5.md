## 2.1 TrxWallet（钱包地址表 — wallet-server / cloud_wallet 库）

```sql
CREATE TABLE TrxWallet (
    id              BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id         VARCHAR(10)    NOT NULL COMMENT '会员账号',
    user_name       VARCHAR(30)    COMMENT '会员姓名',
    en_private_key  VARCHAR(250)   NOT NULL COMMENT '加密后的私钥',
    address         VARCHAR(100)   NOT NULL COMMENT 'TRON 钱包地址',
    create_time     DATETIME,
    create_by       VARCHAR(255),
    update_time     DATETIME,
    update_by       VARCHAR(255),
    INDEX idx_user_id (user_id)
);
```

**关键特征**：
- 每用户一个 TRON 地址，`userId` 建索引但非唯一约束（代码层幂等控制）
- 私钥用 AES 加密存储，密钥 = `KMS密码 + 钱包地址`（地址参与加密盐值）

# 三、区块链充值全链路 — 从用户发起到余额到账

**这是本文的核心章节**。完整拆解 USDT 充值从链上到账户余额变动的每一步。

## 3.1 充值模型概述 — "先转后归"模式

```
核心思路：充值 ≠ 直接转到公司账户

  传统法币充值：用户 → 公司账户（一步到位）

  虚拟币充值（本工程）：
    步骤 1：用户将 USDT 转到自己专属的 TRON 地址（链上操作，用户自行完成）
    步骤 2：系统检测到用户地址有余额（定时轮询 or 用户点确认）
    步骤 3：系统将 USDT 从用户地址归集到公司地址（链上操作，系统自动完成）
    步骤 4：等待链上交易确认
    步骤 5：确认后，给用户账户加余额（链下操作）

  关键：用户地址的私钥由系统生成并加密保管，用户只知道地址不知道私钥
```

# 四、归集模型 — 用户地址→公司地址的资金汇聚

## 4.1 归集的业务含义

```
为什么需要归集（Collection/Aggregation）？

  每个用户有独立的 TRON 地址 → 资金分散在 N 个地址上
  公司需要统一管理资金 → 必须把分散资金归集到一个地址

  │ 用户B地址   │──── USDT ────→│    │  (大量 USDT)  │

  └───────────┘               │
  ┌───────────┐               │
  │ 用户C地址   │──── USDT ────→┘

  └───────────┘
```

# 五、提现全链路 — 从冻结到链上确认

## 5.1 提现流程概述

```
提现 vs 充值的核心区别：
  充值：链上资金流入（用户→系统），先有链上操作后有余额变动
  提现：链上资金流出（系统→用户），先有余额冻结后有链上操作

提现流程：
  1. 用户申请提现 → 创建 SsOrder(type=提现)
  2. 冻结用户余额(freezeAmount += amount)
  3. 后台审核通过
  4. 从公司出款地址向用户提现地址发送 USDT
  5. 等待链上确认
  6. 确认后：解冻 + 扣减余额 + 扣手续费
```

## 5.2 提现关键代码

# 六、钱包地址生成 — EC 密钥对 + AES 加密

## 6.1 地址生成流程

```java
// WalletTrxBusiness.createWallet()
public WalletAddressVO createWallet(TrxUserWalletRequest request) {
    String userId = request.getUserId();

    // 1. 幂等检查：已有地址直接返回
    String existingAddress = trxWalletService.findAddressByUserId(userId);
    if (existingAddress != null) {
        return new WalletAddressVO(userId, existingAddress);
    }

    // 2. 生成 EC 密钥对（与以太坊兼容）
    ECKeyPair ecKeyPair = Keys.createEcKeyPair();  // Web3j

    // 3. 生成标准钱包文件
    WalletFile walletFile = Wallet.createStandard(passphrase, ecKeyPair);

    // 4. 转换为 TRON 地址格式（41 前缀 → T 前缀）
    String originAddress = walletFile.getAddress();  // ETH 格式（无 0x）
    String finalAddress = fromHexAddress("41" + originAddress);  // TRON 格式

    // 5. 提取私钥并 AES 加密
    String privateKeyHex = Numeric.toHexStringNoPrefix(ecKeyPair.getPrivateKey());
    String encryptedKey = encrypt(privateKeyHex, kmsPassword + finalAddress);
    // 加密盐 = KMS密码 + 钱包地址（每个地址的密钥不同）

    // 6. 持久化
    trxWalletService.save(userId, userName, encryptedKey, finalAddress);

    // 7. 同步更新 user-server 的用户记录
    trxWalletService.updateUserAddress(request);
    // 调用 user-server 更新 User.walletAddress

    return new WalletAddressVO(userId, finalAddress);
}
```

## 6.2 私钥加密方案

```
加密算法：AES/ECB/PKCS5Padding
加密密钥：SHA-1(KMS密码 + TRON地址) → 取前 16 字节

解密时：
  1. 从 AWS KMS 解密获取 passphrase
  2. 拼接：passphrase + address
  3. SHA-1 哈希取前 16 字节作为 AES 密钥
  4. AES 解密 enPrivateKey 得到明文私钥
```

**评价**：
- AES/ECB 模式不如 AES/GCM 安全（ECB 相同明文产生相同密文），但私钥是随机的所以实际影响有限
- 地址参与密钥生成是好设计 — 每个地址的加密密钥不同
- **弱点**：如果 KMS 密码泄露 + 数据库泄露 = 所有私钥泄露

## 7.3 UserAssetsBusiness 的余额操作

```java
// UserAssetsBusiness.doBalanceBusiness() 核心方法
public void doBalanceBusiness(BalanceVO balanceVO) {
    Long userId = balanceVO.getUserId();

    // 获取 Redisson 分布式锁
    String lockKey = "USER_SERVER_WALLET_USER_ASSETS_BUSINESS" + userId;
    RLock rLock = redisUtil.getFairLock(lockKey);

    try {
        if (rLock.tryLock(10, TimeUnit.SECONDS)) {
            Assets assets = assetsService.getAssetsByUserId(userId);

            BigDecimal beforeAmount = assets.getBalance();
            BigDecimal amount = balanceVO.getAmount();

            if (balanceVO.getBalanceType() == INCOME) {
                // 加钱
                assetsService.doIncreaseBalanceById(amount, assets.getId());
            } else {
                // 减钱
                int count = assetsService.doReduceBalanceById(amount, assets.getId());
                if (count == 0) {
                    throw new RuntimeException("余额不足");
                }
            }

            // 刷新获取最新 balance
            assetsService.doRefresh(assets);
            BigDecimal afterAmount = assets.getBalance();

            // 记录余额变动（BalanceChange）
            BalanceChange change = new BalanceChange();
            change.setBeforeAmount(beforeAmount);
            change.setAmount(amount);
            change.setAfterAmount(afterAmount);
            change.setBalanceType(balanceVO.getBalanceType());
            change.setChangeType(balanceVO.getChangeType());
            change.setRelateId(balanceVO.getRelateId());
            balanceChangeService.save(change);
        }
    } finally {
        rLock.unlock();
    }
}
```

```java
// Assets.java
@Version
@Column(name = "version")
private Integer version = 1;
```

虽然 Assets 实体上有 `@Version` 注解，但大部分余额操作使用的是 `@Query` 手写 SQL，**绕过了 JPA 的自动版本检查**。只有通过 JPA 的 `save()` 方法更新时才会触发版本校验。

## 8.3 SQL 级条件检查（数据库层）

```sql
-- reduceBalanceById：WHERE balance >= amount（防超扣）
-- doReduceFreezeAmountById：WHERE freezeAmount >= amount（防超解冻）
```

## 8.4 三层保护评价

```
┌──────────────┬────────────────────┬────────────────┐
│ 层级          │ 机制                │ 保护范围        │
├──────────────┼────────────────────┼────────────────┤
│ 应用层        │ Redisson FairLock  │ 同用户串行化     │
│ ORM 层        │ JPA @Version       │ save 操作       │
│ SQL 层        │ WHERE >= condition │ 余额非负         │
└──────────────┴────────────────────┴────────────────┘
```

**判定：调整**
> 思路正确（多层保护），但实际 @Version 被绕过了。我们的方案用乐观锁（WHERE version = V）统一覆盖所有写操作更简洁。

# 九、区块链交互技术 — TRON RPC 详解

## 9.1 TRON 链交互方式

| 操作 | 协议 | 端点 | 说明 |
|------|------|------|------|
| 查询 TRX 余额 | HTTP | /wallet/getaccount | 返回 balance 字段（Sun 单位） |
| 查询 TRC-20 余额 | HTTP | /wallet/triggerconstantcontract | 调用合约 balanceOf |
| 创建 TRC-20 交易 | HTTP | /wallet/triggersmartcontract | 调用合约 transfer |
| 广播已签名交易 | HTTP | /wallet/broadcasthex | 提交到网络 |
| 查询交易状态 | HTTP | /wallet/gettransactionbyid | 确认 contractRet |
| 验证地址格式 | HTTP | /wallet/validateaddress | Base58 格式校验 |
| 创建 TRX 转账 | gRPC | createTransaction | 原生 TRX 转账 |
| 广播交易 | gRPC | broadcastTransaction | 原生广播 |

## 9.2 TRC-20 转账的技术步骤

```
TRC-20 转账（以 USDT 为例）实际上是调用 USDT 智能合约的 transfer 方法：

1. 构造合约调用参数：
   contract_address = USDT 合约地址（hex 格式）
   function_selector = "transfer(address,uint256)"
   parameter = ABI 编码（目标地址 + 金额）
   owner_address = 发送方地址（hex 格式）
   fee_limit = 50000000（50 TRX）

2. 调用 /wallet/triggersmartcontract 获取未签名交易

3. 签名：
   - 归集：本地用 AES 解密的用户私钥签名
   - 提现：发送到外部签名服务签名

4. 广播：调用 /wallet/broadcasthex 提交签名后的交易

5. 获取 txId（交易哈希）

6. 后续查询 /wallet/gettransactionbyid 确认状态
```

## 9.3 金额精度处理

```java
// TRC-20 金额转换（USDT 6 位小数）
BigInteger integer = needSendAmount
    .multiply(BigDecimal.TEN.pow(6))  // 乘以 10^6
    .toBigInteger();                   // 转为整数

// TRX 余额转换（6 位小数）
balance = balanceN.divide(BigDecimal.TEN.pow(6));  // 除以 10^6

// 地址格式转换
// TRON: T-prefix Base58  ↔  41-prefix Hex
// ETH:  0x-prefix Hex
// 生成时：ETH hex → 加 41 前缀 → Base58 编码 → T-prefix
```

## 10.1 任务总览

| 任务 | 位置 | Cron | 功能 |
|------|------|------|------|

## 10.2 充值检查任务详解

```java
@Scheduled(cron = "0 */2 * * * ?", zone = "Asia/Bangkok")
public void checkVirtualCurrencyTask() {
    virtualCurrencyBusiness.checkChargeCrypotcurrencyOrder("USDT", 30);
}

// checkChargeCrypotcurrencyOrder 内部：
// 1. 查找 30 分钟内创建的 VIRTUAL_CURRENCY_WAIT 充值订单
// 2. 超过 30 分钟的 → 标记为 ORDER_SYSTEM_CANCLE（自动取消）
// 3. 30 分钟内的 → 逐个调用 asyncVirtualDepositService.deposit()
//    → deposit() 内部获取分布式锁（orderId 维度）防止重复处理
//    → 走完整充值链路（查余额→归集→确认→到账）
```

**设计要点**：
- 每 2 分钟轮询一次，最多 15 次检查机会（30 分钟 / 2 分钟）
- 每次检查都是全量扫描所有待处理订单
- 分布式锁防止多实例重复处理同一订单

# 十一、安全架构

## 11.1 多层安全体系

```
┌─────────────────────────────────────────────────┐
│                安全架构全景                        │
│                                                  │
│  Layer 1: AWS KMS（密钥管理）                     │
│  ├── 系统级密钥加密（公司地址私钥、Gas 地址私钥）    │
│  ├── KMS 密码加密（passphrase）                   │
│  └── 运行时按需解密，内存中使用                     │
│                                                  │
│  Layer 2: AES 本地加密（用户私钥保护）              │
│  ├── 每个用户私钥单独加密                           │
│  ├── 加密密钥 = SHA-1(KMS密码 + 地址)             │
│  └── 即使 DB 泄露也无法直接获取私钥                 │
│                                                  │
│  Layer 3: RSA 通信加密（签名服务）                  │
│  ├── 提现请求用 RSA 公钥加密                       │
│  ├── 签名服务用 RSA 私钥解密                       │
│  └── 防止中间人攻击                                │
│                                                  │
│  Layer 4: 外部签名服务（提现控制）                   │
│  ├── 提现（资金流出）必须通过外部签名服务            │
│  ├── wallet-server 无法独立执行提现                 │
│  └── 即使 wallet-server 被入侵也无法提走资金        │
│                                                  │
│  Layer 5: Spring Security（接口保护）              │
│  └── WebSecurityConfig 配置接口访问控制             │
└─────────────────────────────────────────────────┘
```

## 11.2 评价

**优点**：
- 分层安全设计思路正确
- 提现独立签名服务是金融级标配
- 用户私钥加密存储，密钥与地址绑定

**不足**：
- AES/ECB 模式安全性略弱（应改用 GCM/CBC）
- AWS KMS 密钥和 accessKey 硬编码在 application.yml
- 日志中未对敏感信息脱敏

# 十四、关键缺陷与风险

## 14.1 高危问题

### H1：充值金额以链上余额为准，非用户指定金额

```
问题：归集时转移的是用户地址的全部 USDT 余额，不是订单的 realAmount
      singleCollect 中 reChargeAmount 若为 null，则查询链上全额余额
场景：用户充值 100 USDT，但地址上已有 200 USDT（之前残留）
      → 系统归集 200 USDT 到公司地址
      → 但订单只加 100 USDT 余额
      → 多出的 100 USDT 去向不明
```

### H2：Gas 补充与 USDT 归集不原子

```
问题：先转 30 TRX gas 费，再归集 USDT，两步不原子
场景：Gas 转入成功 → 归集 USDT 失败 → 30 TRX 白白送入用户地址
      用户理论上可以提走这 30 TRX（如果知道私钥）
      虽然私钥在系统手中，但是设计不够严谨
```

### H3：链上交易确认无区块确认数检查

```
问题：只检查 contractRet == "SUCCESS"，不检查区块确认数
      交易可能被链上回滚（虽然 TRON 极少发生）
正确做法：等待 N 个区块确认（TRON 通常 19 个区块确认）
```

### H4：轮询而非事件驱动

```
问题：每 2 分钟轮询检查充值状态
      用户等待时间最长 2 分钟（加上链上确认时间）
优化方向：
  - WebSocket 监听区块链事件
  - TronGrid 的 Event API
  - 减小轮询间隔（如 30 秒）
```

## 14.2 中危问题

| 编号 | 问题 | 影响 |
|------|------|------|
| M1 | application.yml 中 AWS 密钥明文 | 安全风险 |
| M2 | AES/ECB 模式（应改 GCM） | 加密强度不足 |
| M3 | HttpClient4Util 同步调用区块链 | 阻塞线程 |
| M4 | 归集失败无重试机制 | 资金可能长期滞留用户地址 |
| M5 | 定时归集遍历所有用户（findAll） | 用户量大时性能差 |
| M6 | 日志未脱敏（打印地址/金额） | 安全审计风险 |

# 十五、取舍对照表 — 借鉴 / 调整 / 规避 完整决策

## 15.1 借鉴项

### B1："先转后归"的充值模型

```
决策：借鉴 ✅
原因：每用户独立地址 + 系统归集到公司地址，是行业标准做法
好处：
  - 用户转账目标地址唯一，不会转错
  - 自动识别充值金额（链上余额变化）
  - 公司资金集中管理
```

### B2：Gas 补充机制

```
决策：借鉴 ✅
原因：TRC-20 转账需要 TRX 作为 gas 费，这是 TRON 链的固有限制
实现要点：
  - 专用 Gas 地址（大量 TRX）
  - 归集前自动检查 TRX 余额
  - 不足则先补充 TRX
```

### B3：外部签名服务（提现安全）

```
决策：借鉴 ✅
原因：金融级安全设计，私钥不在业务服务器
好处：即使 wallet-server 被入侵也无法直接转走资金
我们方案：虚拟币提现必须通过独立签名服务
```

### B4：链上/链下双服务分离

```
决策：借鉴 ✅
原因：区块链操作（慢、不可靠）和业务操作（快、需事务）解耦
wallet-server：纯链上操作（生成地址/转账/查询）
user-server：纯链下操作（余额管理/订单管理/流水记录）
```

### B5：BalanceChange 的 before/after 设计

```
决策：借鉴 ✅
原因：变动前后余额记录，审计追溯能力强
同参考工程A的 coinFrom/coinTo
```

### B6：订单状态条件更新（幂等保护）

```
决策：借鉴 ✅
原因：UPDATE ... WHERE order_status = OLD_STATUS，防止重复处理
SQL 级幂等 > 应用层幂等
```

### B7：钱包创建幂等检查

```
决策：借鉴 ✅
原因：创建前先查是否已存在，已存在直接返回
我们方案：INSERT ON DUPLICATE KEY 更原子
```

### B8：提现时先检查已有订单

```
决策：借鉴 ✅
原因：withdrawTraction 先查 findBySsOrderId，已有则直接返回 txId
防止重复发起链上交易
```

## 15.2 调整项

### A1：轮询改为事件驱动 + 轮询兜底

```
决策：调整 ⚡
原因：2 分钟轮询太慢，用户体验差
我们方案：
  - 主路径：区块链事件监听（WebSocket / Event API）
  - 兜底：30 秒轮询
  - 用户触发：点击"已充值"按钮即时检查
```

### A2：金额存储从 BigDecimal 改为 BIGINT

```
决策：调整 ⚡
原因：与我们整体方案一致，BIGINT 存储分/厘
链上金额转换：USDT 链上 6 位小数 × 10^6 = 整数
```

### A3：AES/ECB 改为 AES/GCM

```
决策：调整 ⚡
原因：GCM 提供认证加密（AEAD），防篡改 + 防重放
```

### A4：同步 HTTP 改为异步非阻塞

```
决策：调整 ⚡
原因：区块链 RPC 调用耗时可能数秒，同步调用阻塞线程
我们方案：Go goroutine 天然异步，但仍需设置合理超时
```

### A5：全表扫描归集改为增量扫描

```
决策：调整 ⚡
原因：findAll() 遍历所有用户钱包效率低
我们方案：记录上次归集时间，只扫描有新交易的地址
```

## 15.3 规避项

### X1：AWS 密钥硬编码在配置文件

```
决策：规避 ❌
原因：配置文件泄露 = KMS 密钥泄露 = 所有私钥泄露
我们方案：密钥通过环境变量 / Vault / K8s Secret 注入
```

### X2：链上确认无区块确认数

```
决策：规避 ❌
原因：仅检查 contractRet=SUCCESS 不够，需要 N 个区块确认
我们方案：充值确认要求 ≥ 19 个区块确认（TRON 标准）
```

### X3：充值归集全额（非订单金额）

```
决策：规避 ❌
原因：归集金额 ≠ 充值金额可能导致金额不匹配
我们方案：归集金额以订单金额为准，多余部分标记为异常
```

### X4：Gas 补充与代币归集不原子

```
决策：规避 ❌（在设计上做防护）
我们方案：
  - Gas 补充失败 → 整体归集不执行
  - Gas 补充成功但归集失败 → 记录异常 + 下次归集时重试
  - Gas 地址 TRX 余额告警
```

# 总结

## 一句话评价

参考工程B 是一个 **区块链充值链路完整、安全设计较好** 的实现 — 在密钥管理和签名隔离方面优于参考工程A，**区块链交互层是我们最需要借鉴的部分**。

## 核心借鉴价值（与参考工程A互补）

| 维度 | 参考工程A (p9-backend) | 参考工程B (tg-services) |
|------|---------------------|----------------------|
| 核心价值 | 传统钱包余额管理 | 区块链虚拟币充值链路 |
| 借鉴重点 | 流水设计/职责划分 | 充值归集/密钥管理/签名服务 |
| 安全评分 | 3/10 | 8/10 |
| 总体评分 | 4.8/10 | 6.2/10 |

## 参考工程B 的核心借鉴清单

1. **"先转后归"充值模型** — 行业标准，直接采用
2. **Gas 补充机制** — TRC-20 必需，直接采用
3. **外部签名服务** — 金融级安全，建议采用
4. **链上/链下双服务分离** — 架构合理，借鉴思路
5. **订单状态条件更新** — SQL 级幂等保护
6. **提现订单去重** — 防止重复发起链上交易
7. **定时归集 + 即时归集** — 双模式互补
8. **AWS KMS 密钥管理** — 系统级密钥保护

## 参考工程B 的核心规避清单

1. **无区块确认数检查** — 必须等待 N 个确认
2. **归集全额而非订单金额** — 金额不匹配隐患
3. **AWS 密钥硬编码** — 严重安全风险
4. **AES/ECB 加密模式** — 改用 GCM
5. **同步轮询 2 分钟间隔** — 改为事件驱动 + 短轮询
