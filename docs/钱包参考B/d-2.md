### 2.2 分层设计

```
wallet-server 内部分层：

Controller层 → WalletController (6个端点)
     │
Driver层    → TrxDriver (高层编排：归集、提现、余额查询)
     │
Business层  → WalletTrxBusiness (核心：钱包生成、交易构造、签名、广播)
     │
Client层    → Trc20 / SmartContract / TronUtil (底层链交互)
     │
Service层   → TrxWalletService / TrxWalletOrderService (数据持久化)
     │
Repository层→ JPA Repository (数据库操作)
```

**架构特点：**
- wallet-server 是**纯链交互服务**，不管理用户余额（余额在链上）
- user-server 负责**业务编排**（创建订单、检查余额、触发归集、入账）
- 两个服务通过 **HTTP POST** 通信，非RPC

## 3. 数据模型与表结构

### 3.1 TrxWallet（用户钱包地址表）

```sql
CREATE TABLE TrxWallet (
    id            BIGINT AUTO_INCREMENT PRIMARY KEY,
    userId        VARCHAR(10) COMMENT '会员账号',    -- ★ 只有10字符
    userName      VARCHAR(30) COMMENT '会员姓名',
    enPrivateKey  VARCHAR(250) COMMENT '加密后的私钥', -- AES加密存储
    address       VARCHAR(100) COMMENT '钱包地址',    -- TRON T开头Base58地址
    create_time   TIMESTAMP,
    update_time   TIMESTAMP,
    INDEX idx_userId (userId)
);
-- JPA @Entity, Hibernate ddl-auto=update 自动建表
```

**关键：**
- 一个用户一个链上地址，**不区分币种**（同一个TRON地址可接收TRX和所有TRC20代币）
- `enPrivateKey` 是 AES-ECB 加密后的私钥，密钥 = `passphrase + address`
- 私钥解密需要 passphrase（来自AWS KMS解密的配置值）

### 3.2 TrxWalletOrder（交易记录表）

```sql
CREATE TABLE TrxWalletOrder (
    id            BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id       VARCHAR(10) COMMENT '会员账号',
    tx_id         VARCHAR(200) COMMENT '链上交易哈希',
    ss_order_id   VARCHAR(200) COMMENT '业务订单号（关联user-server的SsOrder）',
    amount        DECIMAL(16,2) COMMENT '金额',
    from_address  VARCHAR(250) COMMENT '发送地址',
    to_address    VARCHAR(250) COMMENT '接收地址',
    coin_name     VARCHAR(10) COMMENT '币种: TRX/USDT/CXT',
    order_type    TINYINT(2) COMMENT '1=充值(RECHARGE) 2=提现(WITHDRAW)',
    post_type     TINYINT(2) COMMENT '1=API调用 2=定时任务',
    create_time   TIMESTAMP,
    update_time   TIMESTAMP,
    INDEX idx_user (user_id),
    INDEX idx_type (order_type),
    INDEX idx_txid (tx_id)
);
```

**关键：**
- `tx_id` = 链上交易哈希（SHA256），是链上唯一标识
- `ss_order_id` = 业务侧订单号，关联 user-server 的 SsOrder 表
- `amount` 用 `DECIMAL(16,2)` — 只支持2位小数（USDT通常6位，精度不够）

### 5.2 充值步骤逐一拆解

#### 步骤①② — 用户声明充值 + 创建订单

```
用户在Telegram Bot中输入充值金额（如100 USDT）
→ user-server 创建 SsOrder:
    - userId = 用户ID
    - amount = 100 (申报金额)
    - realAmount = 100 (实际金额)
    - orderStatus = VIRTUAL_CURRENCY_WAIT
    - orderType = CHARGE_ORDER
    - virtualCurrency = "USDT"
```

**关键：此时用户还没有转币，只是声明了意向金额。**

#### 步骤③ — 用户链上转账

```
用户使用自己的TRON钱包（如TronLink）
将100 USDT转到系统分配的充值地址（用户专属地址）

系统不监听链上事件！
等待用户点击确认 或 定时任务轮询。
```

**关键：系统没有区块扫描监听！依赖用户主动确认 + 定时轮询。**

#### 步骤④⑤⑥ — 确认充值 + 余额校验

```java
// AsyncVirtualDepositService.checkCrypotcurrencyOrder()
// 1. 查订单
SsOrder ssOrder = ssOrderService.findById(orderId);
// 2. 如果还没有txId（首次进入）
if (StringUtils.isEmpty(txId)) {
    // 3. 调wallet-server查链上余额
    balance = walletBusiness.getBalance(userId, virtualCurrency);
    // 4. 余额校验：链上余额 >= 申报金额
    if (balance.compareTo(realAmount) == -1) {
        return "馀额 小於 充值数量";
    }
    // 5. 余额足够，触发归集
    txId = walletBusiness.getTransferResult(virtualCurrency, orderId, userId, realAmount, postType);
    // 6. 保存txId
    ssOrderService.updateTxIdById(txId, orderId);
    return "获取到txId后,下次进入";
}
```

**流程特点：**
- 第一次进入：查余额→触发归集→获取txId→保存→返回等待
- 第二次进入：有txId→查交易确认→入账

#### 步骤⑦⑧ — Gas费补充

```java
// TrxDriver.singleCollect() — 归集前检查Gas费
// TRC20转账需要TRX作为手续费
BigDecimal addTrx = getAccountBalance(fromAddress, TRXContractEnum.TRX.getContract());
if (FEE_BAL.compareTo(addTrx) > 0) {  // FEE_BAL = 30 TRX
    // 从Gas地址向用户地址转30TRX（手续费）
    trxWalletResult = createTransactionForTrx(
        awsKmsDriver.decryptData(gasPassword),  // Gas地址私钥
        FEE_BAL,                                  // 30 TRX
        newGasAddress,                            // Gas地址
        fromAddress,                              // 用户地址
        TRXContractEnum.TRX.getContract(),        // TRX主币
        ssOrderId
    );
}
```

**为什么需要补Gas？**
- TRON网络上TRC20转账需要消耗能量(Energy)或TRX
- 用户充值地址只有USDT，没有TRX
- 系统从专门的Gas地址向用户地址转30 TRX作为手续费
- 30 TRX ≈ 足够3-5次TRC20转账

#### 步骤⑨ — 归集：用户地址→公司地址

```java
// TrxDriver.singleCollect() — 执行归集转账
trxWalletResult = createTransactionForTrx(
    null,                // privateKey=null → 从DB解密获取
    trxBalance,          // 归集全部余额
    fromAddress,         // 用户地址（发送方）
```

### 6.1 地址生成流程

```java

// 5. 提取私钥（hex格式）

// 6. AES加密私钥
```

### 6.2 地址格式转换

```
以太坊地址:  0x + 40个hex字符
    ↓ 添加TRON前缀
TRON hex:    41 + 40个hex字符 (42个hex)
    ↓ Base58Check编码
TRON地址:    T + 33个Base58字符 (34字符，如 TXYZop...)
```

### 6.3 私钥加密方案

```
明文私钥(hex)
    ↓ AES-ECB / PKCS5Padding
    ↓ 密钥 = SHA1(passphrase + address)[0:16]
加密私钥(Base64)
    ↓ 存入 TrxWallet.enPrivateKey

解密时:
    enPrivateKey
    ↓ AES-ECB解密
    ↓ 密钥 = SHA1(passphrase + address)[0:16]
    明文私钥(hex)
```

**passphrase** 本身存储在 `application.yml`，用 AWS KMS 加密：
```yaml
wallet.trx.mainCollectPassphrase: AQICAHgY...（KMS密文）
```
运行时通过 `awsKmsDriver.decryptData(passphrase)` 解密。

## 10. 外部签名服务

### 10.1 设计原理

```
wallet-server 构造交易 → RSA加密请求 → 签名服务 → 签名 → 返回
```

**目的：** 出款地址（大资金池）的私钥与wallet-server隔离，即使wallet-server被攻破也无法直接提走大额资金。

### 10.2 签名流程

```java
// WalletTrxBusiness.sendSign()

// 1. 计算交易哈希
byte[] txId = Sha256Sm3Hash.hash(tx.getRawData().toByteArray());
String signTxid = Base64.getEncoder().encodeToString(txId);

// 2. 构造签名请求
Map<String, Object> signMap = new HashMap<>();
signMap.put("orderNo", ssOrderId);
signMap.put("outAddressNo", from);
signMap.put("chainType", "TRON");
signMap.put("hashId", signTxid);

// 3. RSA加密请求体
String enSign = RSAUtil.encrypt(JSONObject.toJSONString(signMap), rsaPublicKey);

// 4. HTTP POST到签名服务
HttpResponse response = HttpRequest.post(signUrl + "/api/admin/chain/sign")
    .header("PLAT_CODE", platCode)
    .body(enSign)
    .timeout(60000)  // 60秒超时
    .execute();

// 5. 解析签名结果
JSONObject jsonObject = JSONObject.parseObject(response.body());
if ("200".equals(jsonObject.get("code"))) {
    byte[] deSign = Base64.getDecoder().decode(jsonObject.getString("data"));
    // 6. 组装完整签名交易
    return tx.toBuilder().addSignature(ByteString.copyFrom(deSign)).build().toByteArray();
}
```

### 10.3 签名策略矩阵

| 场景 | 签名方式 | 私钥位置 | 安全等级 |
|------|---------|---------|---------|
| 归集（用户→公司） | 本地签名 | DB (AES加密) | ★★★ |
| 补Gas（Gas→用户） | 外部签名 | 签名服务 | ★★★★ |
| 提现（出款→用户外部） | 外部签名 | 签名服务 | ★★★★★ |

## 11. AWS KMS密钥管理

### 11.1 加密层次

```
第1层：应用层AES加密
  用户私钥 → AES-ECB(passphrase+address) → enPrivateKey (存DB)

第2层：配置层KMS加密
  passphrase/gasPrivateKey等 → AWS KMS加密 → application.yml (密文)

第3层：运行时解密
  application.yml密文 → AWS KMS解密 → 内存中明文
  enPrivateKey + 明文passphrase → AES解密 → 明文私钥 (瞬时使用)
```

```java
// AwsKmsDriver
// 加密
public String encryptData(String noEncryptData) {
    KmsClient kmsClient = createKmsClient();
    EncryptRequest encryptRequest = EncryptRequest.builder()
        .keyId(keyId)  // KMS主密钥ID
        .plaintext(SdkBytes.fromByteArray(data))
        .build();
    EncryptResponse response = kmsClient.encrypt(encryptRequest);
    return Base64.getEncoder().encodeToString(response.ciphertextBlob().asByteArray());
}

// 解密
public String decryptData(String afterEncryptionData) {
    KmsClient kmsClient = createKmsClient();
    DecryptRequest decryptRequest = DecryptRequest.builder()
        .ciphertextBlob(SdkBytes.fromByteArray(Base64.getDecoder().decode(data)))
        .keyId(keyId)
        .build();
    DecryptResponse decryptResponse = kmsClient.decrypt(decryptRequest);
    return new String(decryptResponse.plaintext().asByteArray());
}
```

## 15. 安全设计分析

### 15.1 多层安全体系

```
┌─────────────────────────────────────────────────────┐
│ 第1层：传输安全                                      │
│  签名服务调用使用RSA加密请求体                        │
│  服务间通信内网（无HTTPS）                           │
├─────────────────────────────────────────────────────┤
│ 第2层：存储安全                                      │
│  用户私钥 = AES-ECB(privateKey, passphrase+address) │
│  敏感配置 = AWS KMS加密                              │
├─────────────────────────────────────────────────────┤
│ 第3层：权限隔离                                      │
│  出款私钥不在wallet-server（签名服务独立）            │
│  用户私钥在wallet-server（归集用）                   │
├─────────────────────────────────────────────────────┤
│ 第4层：业务安全                                      │
│  订单乐观锁防重复入账                                │
│  Redisson分布式锁防并发                              │
│  提现余额校验                                       │
│  提现订单幂等（查已有txId直接返回）                   │
└─────────────────────────────────────────────────────┘
```

### 15.2 安全弱点

| 问题 | 严重性 | 说明 |
|------|--------|------|
| AES-ECB模式 | ★★★★ | ECB是最弱的AES模式，同一明文相同密文，应用CBC/GCM |
| 服务间无认证 | ★★★ | user-server调wallet-server无API Key/Token |
| AWS AK/SK硬编码 | ★★★★★ | 写在application.yml，应使用IAM角色或环境变量 |
| Spring Security默认账号 | ★★★ | baisha/baisha 硬编码 |
| 私钥解密在内存 | ★★ | 合理但需注意GC敏感数据清理 |

## 16. 精华提炼：值得借鉴的设计

### 16.1 签名权限分离 ★★★★★

```
归集（小额，用户→公司）：wallet-server本地签名
提现（大额，公司→外部）：外部签名服务

精华：出款私钥与应用服务器物理隔离，即使应用被攻破也无法提走大额资金。
```

**我们的应用：** 提现出款必须走独立签名服务，私钥不在业务服务中。

### 16.2 Gas费自动补充 ★★★★★

```
TRC20转账需要TRX手续费
→ 系统自动检查用户地址TRX余额
→ 不足时从Gas地址补30 TRX
→ 然后执行TRC20归集
```

**我们的应用：** 充值归集前必须自动检查Gas费，Gas不足自动补充。这个是容易忽略的关键步骤。

### 16.3 充值=声明+归集 模型 ★★★★

```
不是"用户转到公司地址"
而是"用户转到自己的专属地址 → 系统归集到公司地址"

优点：
1. 每个用户独立地址，方便追溯
2. 可精确知道谁充了多少
3. 归集时机可控
```

**我们的应用：** 采用同样的模型——用户独立地址+系统归集。

### 16.4 提现订单幂等 ★★★★

```java
// 提现前先查订单表
TrxWalletOrderEntity order = trxWalletOrderRepository.findBySsOrderId(ssOrderId);
if (ObjectUtil.isNotEmpty(order)) {
    result.setTxId(order.getTxId());
    return result;  // 直接返回已有txId
}
```

**我们的应用：** 所有链上操作必须先查DB幂等，避免重复广播交易。

### 16.5 USDT合约地址配置化 ★★★★

```java
// TRXContractEnum
USDT(2, WalletUtil.getContractByCoin("USDT"), "USDT", 6, "TRC20")
// 从YAML加载合约地址，不硬编码
```

**我们的应用：** 合约地址通过配置管理，方便测试网/主网切换。

### 16.6 订单状态乐观锁 ★★★★

```java
// 更新订单状态时加WHERE条件
updateOrderStatusAndCompleteTime(
    newStatus, completeTime, txId, orderId, userId, virtualCurrency,
    currentStatus  // WHERE orderStatus = currentStatus
);
// 返回count=1表示成功，count=0表示状态已变（被其他线程处理了）
```

### 16.7 超时自动取消 ★★★

```
充值订单超过2小时未完成 → 系统自动取消
防止大量WAIT订单积压
```

### 16.8 两阶段充值（查余额→归集→确认）★★★★

```
阶段1：查余额 + 触发归集 → 获得txId
阶段2：查txId确认 → 入账

两阶段通过txId是否存在来区分
确保不会在链上交易未确认时就入账
```

## 17. 糟粕识别：需要规避的问题

### 17.1 P0级（严重）

#### 17.1.1 无区块扫描/事件监听 ★★★★★

**问题：** 完全依赖轮询检查余额，没有区块链事件监听。

**后果：**
- 充值到账延迟（最长5分钟轮询间隔）
- 如果用户不点"确认"，要等定时任务
- 无法实时感知链上转入
- 如果节点API故障，轮询失败会延迟更久

**我们的对策：** 必须实现区块扫描 + 事件监听（后台服务扫描区块中与我们地址相关的TRC20 Transfer事件），轮询作为兜底。

#### 17.1.2 金额精度不足 ★★★★

**问题：** `DECIMAL(16,2)` 只有2位小数。

**后果：**
- USDT-TRC20精度是6位小数（1 USDT = 1,000,000 最小单位）
- 用户充值 99.123456 USDT 会被截断为 99.12
- 差额 0.003456 USDT 直接丢失

**我们的对策：** 使用BIGINT最小单位存储（USDT精度6位，1 USDT = 1000000）。

#### 17.1.3 AES-ECB加密模式 ★★★★

**问题：** ECB模式不安全——相同明文产生相同密文，可被分析。

**我们的对策：** 使用AES-256-GCM（带认证的加密模式）。

#### 17.1.4 AWS凭证硬编码 ★★★★★

**问题：** AWS Access Key和Secret Key直接写在application.yml。

**后果：** 代码泄露 = 密钥泄露 = KMS完全可访问 = 所有加密数据裸奔。

**我们的对策：** 使用环境变量/Vault/K8s Secret注入敏感配置。

### 17.2 P1级（重要）

#### 17.2.1 服务间调用无重试/降级

**问题：** user-server调wallet-server全部是单次HTTP POST，无重试、无超时配置、无降级。

#### 17.2.2 归集失败无重试

**问题：** 归集交易失败（Gas不足、网络超时）直接返回null，无重试队列。

#### 17.2.3 交易广播无确认等待

**问题：** 广播交易后直接返回txId，不等待链上确认。如果交易被丢弃（nonce冲突等），txId对应的交易可能永远不会被确认。

#### 17.2.4 充值金额以声明为准

**问题：** 用户声明充值100 USDT，实际链上只转了50 USDT，系统归集50但入账100。

**具体代码问题：**
```java
// checkCrypotcurrencyOrder() 中
balance = walletBusiness.getBalance(userId, virtualCurrency);
if (balance.compareTo(realAmount) == -1) {
    return "馀额 小於 充值数量";  // 余额<声明 → 失败
}
// 但如果 balance >= realAmount，归集的是 realAmount 而不是 balance
txId = walletBusiness.getTransferResult(virtualCurrency, orderId, userId, realAmount, postType);
```

实际上 `getTransferResult` 调的是 `collectTrc20`，归集参数是 `amount=realAmount`。如果链上余额>声明金额，多余部分留在用户地址不归集（下次归集定时任务会处理）。但如果链上余额<声明金额（用户只转了部分），校验会失败返回错误。

#### 17.2.5 无交易重发机制

**问题：** 归集/提现交易如果广播失败或被TRON网络丢弃，没有重发逻辑。

### 17.3 P2级（优化建议）

| 问题 | 说明 |
|------|------|
| 所有地址findAll遍历 | 归集时查所有用户地址，数据量大时性能差 |
| 日志打印过多 | JSON序列化大对象写日志 |
| 无链上Gas估算 | 固定补30 TRX，可能浪费或不够 |
| UUID做订单号 | 归集的ssOrderId用UUID，可读性差 |
| 无交易Gas费记录 | 归集/提现的实际Gas消耗未记录 |
