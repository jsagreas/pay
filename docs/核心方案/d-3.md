## 第 1 章 引言：为什么钱包≠CRUD

### 1.1 普通CRUD vs 金融级服务

| 维度 | 普通CRUD | 钱包金融级 |
|------|---------|-----------|
| **数据正确性** | 偶尔错一条可修 | 差1分钱=事故 |
| **并发模型** | 最终一致即可 | 强一致，不容忍Lost Update |
| **幂等要求** | 无或弱幂等 | 所有写操作必须幂等 |
| **精度要求** | 整数或无所谓 | 小数精度到0.00000001 |
| **事务范围** | 单表单库 | 跨表、跨服务、跨流程 |
| **状态管理** | 无状态或简单开关 | 多步状态机，每步可失败需补偿 |
| **安全敏感度** | 信息泄露 | 直接资金损失 |
| **审计要求** | 日志即可 | 不可篡改流水+链式校验 |

4. **落地骨架**——伪代码/代码模式模板，可直接指导编码

## 第 3 章 [P0-C1] 并发安全与余额一致性

### 3.1 问题定义

**场景**：同一用户的同一钱包，多个操作并发到达：
- 充值回调入账(+5000) 与 投注扣款(-3000) 同时执行
- 两笔投注(-800 和 -600) 同时扣款

**如果不处理**：
```
初始余额: 10000

T1: Read balance=10000, 计算 10000+5000=15000, Write 15000
T2: Read balance=10000, 计算 10000-3000=7000,  Write 7000
                                                      ↑
                                            最终结果: 7000
                                            正确结果: 12000
                                            丢失: 5000（Lost Update）
```

**严重性**：P0 — 直接资金损失或超发，不可自动修复

### 3.2 方案比选

| 方案 | 原理 | 优势 | 劣势 | 适用场景 |
|------|------|------|------|---------|
| **A: SELECT FOR UPDATE（悲观锁）** | 事务内行级排他锁，串行化同一行的写操作 | 零冲突丢失、实现简单、TiDB原生支持 | 锁等待增加延迟(10-20ms) | **金融系统首选** |
| B: CAS+Version（乐观锁） | 读取version→更新时WHERE version=X | 无锁竞争、高吞吐 | 高冲突时大量重试失败，金融系统不可接受 | 电商库存（允许下单失败） |
| C: 原子UPDATE不SELECT | `UPDATE SET balance=balance+amount` | 极简、性能最优 | 无法获取balance_before（流水记录需要）、无法做复杂校验 | 简单计数器 |
| D: 应用层串行化队列 | 按user_id分片到单线程队列 | 完全无锁、极致吞吐 | 架构复杂度高、需要消息队列、当前阶段过度设计 | 超大规模（百万QPS） |

### 3.3 最优方案：悲观锁 + 版本号双重保护

**推荐方案：A+B混合 = SELECT FOR UPDATE + version字段**

**推荐理由**：
1. 支付宝、微信支付、Stripe、Grab Financial 均采用此模式
2. TiDB 4.0+ 默认悲观事务模式，原生适配
3. 锁持有时间 ~10-20ms，钱包操作频率（用户级）远低于此阈值
4. version字段作为最后一道防线，防御代码缺陷

**方案细节**：

```
┌─────────────────────────────────────────────────────┐
│                   单次余额操作事务                     │
│                                                      │
│  BEGIN TRANSACTION                                   │
│    │                                                 │
│    ├─ ① SELECT * FROM wallet                        │
│    │   WHERE user_id=? AND currency_code=?           │
│    │   AND wallet_type=? FOR UPDATE;   ← 行级排他锁  │
│    │                                                 │
│    ├─ ② 业务校验（余额充足？金额合法？）                │
│    │   balance - frozen >= debit_amount              │
│    │                                                 │
│    ├─ ③ UPDATE wallet SET                            │
│    │   balance = new_balance,                        │
│    │   version = version + 1     ← 版本号自增        │
│    │   WHERE id=? AND version=old_version;           │
│    │                                                 │
│    │   if affected_rows != 1 → ROLLBACK + 异常告警   │
│    │                                                 │
│    ├─ ④ INSERT INTO wallet_transaction(              │
│    │     balance_before, balance_after, ...)          │
│    │                                    ← 不可变流水  │
│    │                                                 │
│  COMMIT                                              │
└─────────────────────────────────────────────────────┘
```

**关键约束**：
- **所有**余额变更操作（Credit/Debit/Freeze/Unfreeze/DebitFrozen）必须走此模式
- 事务内禁止RPC调用（防止锁持有时间不可控）
- 事务内禁止慢查询（只操作已锁定的行）
- version字段不匹配时 → ROLLBACK + 告警（说明有并发绕过了FOR UPDATE，理论上不会发生）

### 3.4 落地骨架（伪代码）

```go
// account_rep.go — 所有余额操作的唯一入口
func (r *AccountRepo) ExecuteBalanceOp(ctx context.Context, req BalanceOpReq) (*BalanceOpResp, error) {
    var resp *BalanceOpResp

    err := r.db.WithContext(ctx).Transaction(func(tx *gorm.DB) error {
        // ① 幂等快检（详见第4章）
        existing, err := r.checkIdempotent(tx, req.BizType, req.OrderNo, req.WalletType)
        if err != nil {
            return err
        }
        if existing != nil {
            resp = existing // 幂等返回历史结果
            return nil
        }

        // ② FOR UPDATE 锁定钱包行
        var wallet model.Wallet
        err = tx.Set("gorm:query_option", "FOR UPDATE").
            Where("user_id = ? AND currency_code = ? AND wallet_type = ?",
                req.UserID, req.CurrencyCode, req.WalletType).
            First(&wallet).Error
        if err != nil {
            return err // 钱包不存在或DB错误
        }

        // ③ 业务校验
        balanceDec := decimal.NewFromString(wallet.Balance)
        frozenDec := decimal.NewFromString(wallet.Frozen)
        amountDec := decimal.NewFromString(req.Amount)
        available := balanceDec.Sub(frozenDec)

        if req.Direction == DirectionExpense && available.LessThan(amountDec) {
            return ErrInsufficientBalance
        }

        // ④ 计算新余额
        var newBalance decimal.Decimal
        switch req.Direction {
        case DirectionIncome:
            newBalance = balanceDec.Add(amountDec)
        case DirectionExpense:
            newBalance = balanceDec.Sub(amountDec)
        }

        // ⑤ UPDATE + version自增
        result := tx.Model(&model.Wallet{}).
            Where("id = ? AND version = ?", wallet.ID, wallet.Version).
            Updates(map[string]interface{}{
                "balance": newBalance.String(),
                "version": wallet.Version + 1,
            })
        if result.RowsAffected != 1 {
            return ErrVersionConflict // 理论上不会触发，安全兜底
        }

        // ⑥ INSERT 不可变流水
        txnRecord := &model.WalletTransaction{
            UserID:        req.UserID,
            CurrencyCode:  req.CurrencyCode,
            WalletType:    req.WalletType,
            Direction:     req.Direction,
            Amount:        req.Amount,
            BalanceBefore: wallet.Balance,
            BalanceAfter:  newBalance.String(),
            FrozenBefore:  wallet.Frozen,
            FrozenAfter:   wallet.Frozen, // 非冻结操作不变
            BizType:       req.BizType,
            OrderNo:       req.OrderNo,
            Remark:        req.Remark,
            CreateAt:      time.Now().UnixMilli(),
        }
        if err := tx.Create(txnRecord).Error; err != nil {
            return err // 唯一索引冲突=幂等兜底
        }

        resp = &BalanceOpResp{
            BalanceBefore: wallet.Balance,
            BalanceAfter:  newBalance.String(),
            TransactionID: txnRecord.ID,
        }
        return nil
    })

    return resp, err
}
```

| 验证项 | 方法 | 预期结果 |
|--------|------|---------|
| 并发Lost Update | 10个goroutine同时对同一钱包+1000 | 最终余额=初始+10000，无丢失 |

## 第 4 章 [P0-C2] 幂等性设计

### 4.1 问题定义

**场景**：网络超时或调用方重试导致同一操作重复到达：
- 财务调钱包WalletCredit超时 → 财务重试 → 钱包收到2次入账
- 用户快速双击"提现"按钮 → 2个提现请求到达

**如果不处理**：
```
充值回调: WalletCredit(orderNo="D20260302001", amount=10000)
  第1次: balance 0 → 10000 ✓
  第2次（重试）: balance 10000 → 20000 ✗ ← 重复入账！
```

**严重性**：P0 — 重复入账=资金超发，重复扣款=用户资金损失

### 4.2 方案比选

| 方案 | 原理 | 优势 | 劣势 | 适用场景 |
|------|------|------|------|---------|
| **A: DB唯一索引（强幂等）** | UNIQUE(biz_type,order_no,wallet_type)阻止重复INSERT | 100%可靠、DB级保证、无额外组件 | 需要流水表设计支持 | **金融写操作首选** |
| B: Redis SetNX预检 | 操作前SetNX(key,TTL)，存在则拒绝 | 性能极高、微秒级 | Redis宕机=失效、TTL过期=失效 | 防重复提交（非关键路径） |
| C: 业务状态检查 | 先查询订单状态，已处理则跳过 | 直觉简单 | 并发窗口（检查→执行之间仍可能重复） | 单机低并发 |

### 4.3 最优方案：双层幂等 = Redis快检 + DB唯一索引兜底

**推荐方案：L1(Redis) + L2(DB UNIQUE)双层保护**

```
┌──────────────────────────────────────────────────┐
│              双层幂等防护模型                       │
│                                                   │
│  请求到达(biz_type, order_no, wallet_type)         │
│     │                                             │
│     ▼                                             │
│  ┌─────────────────────────────┐                  │
│  │ L1: Redis快速检查            │ ← 99%重复请求    │
│  │ Key: idempotent:{biz}:{no}  │   在此被拦截      │
│  │ TTL: 24h                     │                  │
│  │                              │                  │
│  │ EXISTS? → YES → 返回历史结果  │                  │
│  │          → NO  → 继续         │                  │
│  └─────────────────────────────┘                  │
│     │                                             │
│     ▼                                             │
│  ┌─────────────────────────────┐                  │
│  │ L2: DB UNIQUE约束            │ ← 最终防线       │
│  │ wallet_transaction表         │   Redis失效时     │
│  │ UNIQUE(biz_type,order_no,    │   仍100%拦截     │
│  │        wallet_type)          │                  │
│  │                              │                  │
│  │ INSERT成功 → 正常流程        │                  │
│  │ Duplicate → 返回已有记录     │                  │
│  └─────────────────────────────┘                  │
└──────────────────────────────────────────────────┘
```

**为什么需要双层**：
- L1(Redis)：性能优先，99%重复请求在微秒级被拦截，不进入数据库事务
- L2(DB UNIQUE)：可靠性兜底，Redis宕机/TTL过期/缓存不一致时仍100%有效
- 单独L1不够：Redis非持久化（重启丢失）、TTL过期后失效
- 单独L2不够：每次都进入事务+INSERT尝试→性能浪费

### 4.4 幂等键设计规范

```
┌──────────────────────────────────────────────────────────┐
│                    幂等键三元组                            │
│                                                          │
│   (biz_type, order_no, wallet_type)                      │
│                                                          │
│   biz_type  ── 业务类型，区分不同操作来源                   │
│   order_no  ── 业务方生成的唯一订单号                      │
│   wallet_type── 钱包类型(1=中心,2=奖励...)                │
│                                                          │
│   同一笔投注可能同时扣中心钱包和奖励钱包                    │
│   → 2条流水记录，biz_type和order_no相同                   │
│   → 但wallet_type不同 → 不冲突                           │
│   → 这就是为什么三元组中包含wallet_type                    │
└──────────────────────────────────────────────────────────┘
```

**order_no格式与归属**：

| biz_type | 前缀 | 生成方 | 格式示例 |
|----------|------|--------|---------|
| deposit | D | 钱包 | D20260302143052A8X7 |
| withdraw | W | 钱包 | W20260302143052B3K9 |
| exchange | E | 钱包 | E20260302143052C1M4 |
| supplement | B | 财务 | B20260302143052D5N2 |
| manual_credit | A | 财务 | A2026030214305200001 |
| manual_debit | M | 财务 | M2026030214305200002 |
| bet | BET | 游戏 | BET20260302143052F7P8 |
| gift | GFT | 直播 | GFT20260302143052G2Q1 |
| bonus | ACT | 活动 | ACT20260302143052H4R6 |

**关键原则**：
- **谁发起操作，谁生成order_no** — 调用方负责唯一性
- 不同前缀天然不碰撞，无需全局序列器
- 钱包只使用三元组做幂等，不关心order_no的具体生成算法
- 重复调用必须传**相同的order_no** — 改变order_no=新操作

### 4.5 幂等返回行为

```
┌────────────────────────────────────────────────────┐
│               幂等返回的正确行为                     │
│                                                    │
│  首次调用: WalletCredit(orderNo="D001", amt=1000)  │
│    → 执行入账 → 返回 {code:0, balance_after:1000}  │
│                                                    │
│  重复调用: WalletCredit(orderNo="D001", amt=1000)  │
│    → 检测到已存在 → 返回 {code:0, balance_after:1000}│
│    → 注意: code=0(成功)，不是错误码                 │
│    → 因为调用方需要知道"这笔操作已完成"             │
│                                                    │
│  ⚠️ 错误做法: 返回错误 {code:6022201, "重复操作"}   │
│    → 调用方无法区分"已成功"和"真的失败"              │
│    → 财务模块会以为失败而继续重试 → 死循环            │
└────────────────────────────────────────────────────┘
```

### 4.6 验证标准

| 验证项 | 方法 | 预期结果 |
|--------|------|---------|
| 相同order_no重复调用 | 同一请求调用3次 | 余额只变化1次，后2次返回code=0 |
| Redis宕机幂等 | 关闭Redis后重复调用 | DB层UNIQUE拦截，仍幂等 |
| 不同wallet_type | 同order_no+同biz_type操作不同钱包 | 两笔均成功（三元组不同） |
| TTL过期重试 | 24h后重试相同order_no | DB层仍拦截 |

## 第 5 章 [P0-C3] 金额精度与计算安全

### 5.1 问题定义

**场景**：Go的float64无法精确表示十进制小数
```go
// Go float64 精度陷阱
fmt.Println(0.1 + 0.2)          // 0.30000000000000004
fmt.Println(1.005 * 100)        // 100.49999999999999
fmt.Println(0.3 - 0.1 - 0.2)   // -2.7755575615628914e-17
```

**如果不处理**：
- 累积误差导致对账不平（差几分钱）
- 汇率换算放大精度误差（1 USDT = 25000 VND，0.00000001的误差 × 25000 = 0.00025）
- 多次加减后 balance_before ≠ 上一条 balance_after（流水链断裂）

**严重性**：P0 — 对账不平=财务事故，流水链断裂=审计失败

### 5.2 方案比选

| 方案 | 原理 | 优势 | 劣势 | 适用场景 |
|------|------|------|------|---------|
| **A: shopspring/decimal** | 任意精度十进制，基于big.Int实现 | 零精度损失、API丰富、Stripe/PayPal标准 | 性能比float64慢~10x（仍在ns级，不影响） | **金融系统首选** |
| B: int64最小单位 | 1元=100分=10000厘 | 整数运算快、无精度问题 | 跨币种精度不同（VND 0位/USDT 8位）转换易出错 | 单币种系统 |
| C: math/big.Float | 标准库大浮点数 | 无需第三方依赖 | API不如decimal友好、还是二进制浮点 | 科学计算 |

### 5.3 最优方案：全链路Decimal保证

**推荐方案：A = shopspring/decimal + 全链路string传输**

```
┌──────────────────────────────────────────────────────────┐
│                全链路金额精度保证                          │
│                                                          │
│  [IDL层]     amount string          ← string传输零损失    │
│      ↓                                                   │
│  [Service层] decimal.NewFromString(amount)  ← 解析       │
│      ↓                                                   │
│  [计算层]    d1.Add(d2).Mul(rate).Round(precision)       │
│      ↓                                    ← decimal运算  │
│  [存储层]    DECIMAL(20,8) NOT NULL         ← DB精确存储  │
│      ↓                                                   │
│  [展示层]    FormatByCurrency(decimal, "VND") → "1.000"  │
│                                    ← 按币种精度格式化     │
└──────────────────────────────────────────────────────────┘
```

**各层精度规则**：

| 层级 | 类型 | 精度 | 说明 |
|------|------|------|------|
| IDL传输 | string | 原始精度 | "100.50000000"，不做任何截断 |
| Go运算 | decimal.Decimal | 任意精度 | 中间运算不截断 |
| DB存储 | DECIMAL(20,8) | 8位小数 | 最大12位整数+8位小数 |
| 汇率存储 | DECIMAL(20,8) | 8位小数 | 汇率可能极小(如0.00004) |
| 展示输出 | string | 按币种配置 | VND→0位, THB→2位, USDT→8位 |

**精度截断规则**（仅在最终写入DB前执行）：

```go
// 不同币种的精度配置
var precisionMap = map[string]int32{
    "VND": 0,  // 越南盾：整数
    "IDR": 0,  // 印尼盾：整数
    "THB": 2,  // 泰铢：2位小数
    "BSB": 2,  // 平台币：2位小数
    "USDT": 8, // USDT：8位小数
}

// 截断规则: RoundDown（向零截断，不四舍五入）
// 理由: 金融系统永远向用户不利方向截断，防止精度膨胀
result := amount.RoundDown(precision)
```

### 5.4 禁忌清单

```
⛔ 绝对禁止:
1. var balance float64          — 精度灾难
2. amount * 100 转int           — 跨币种精度不同，必错
3. fmt.Sprintf("%.2f", amount)  — float→string丢精度
4. json.Number直接运算          — 本质还是float
5. 中间运算截断精度              — 必须最终步骤才截断

✅ 必须遵守:
1. 所有金额字段 → decimal.Decimal
2. 所有传输 → string
3. 所有存储 → DECIMAL(20,8)
4. 所有比较 → d1.Equal(d2)，不用 ==
5. 所有截断 → RoundDown，不用Round
```

### 5.5 验证标准

| 验证项 | 方法 | 预期结果 |
|--------|------|---------|
| 精度不丢失 | 0.1+0.2 用decimal计算 | 精确等于0.3 |
| 链式流水 | 100笔随机加减后检查链 | 每条balance_before=上条balance_after |
| 汇率换算 | 0.00000001 USDT × 25000 | 精确等于0.00025000 |
| 极端金额 | 999999999999.99999999 存取 | DB存取无损 |

## 第 6 章 [P0-C4] 余额负数五层防线

### 6.1 问题定义

**场景**：任何代码缺陷或并发漏洞导致 balance < 0 或 available < 0

**如果不处理**：
- 用户余额为负 = 平台损失（用户花了不存在的钱）
- 负余额无法通过正常流程恢复（只能人工介入）

**严重性**：P0 — 余额为负是金融系统的终极红线

### 6.2 最优方案：五层纵深防御

```
┌──────────────────────────────────────────────────────────┐
│                   余额负数五层防线                         │
│                                                          │
│  L1: 参数层校验                                          │
│  ┌─────────────────────────────────────────────────┐    │
│  │ amount > 0 ？ currency_code 有效？ wallet_type 合法？ │    │
│  │ → 非法参数直接拒绝，不进入任何业务逻辑               │    │
│  └─────────────────────────────────────────────────┘    │
│                          ↓ 通过                          │
│  L2: 幂等层拦截                                          │
│  ┌─────────────────────────────────────────────────┐    │
│  │ (biz_type, order_no, wallet_type) 已存在？           │    │
│  │ → 重复操作快速拒绝（防止重复扣款）                    │    │
│  └─────────────────────────────────────────────────┘    │
│                          ↓ 通过                          │
│  L3: 事务层校验（核心防线）                               │
│  ┌─────────────────────────────────────────────────┐    │
│  │ SELECT ... FOR UPDATE → 锁定行                      │    │
│  │ available = balance - frozen                        │    │
│  │ available >= debit_amount ？                        │    │
│  │ → 不足则 ROLLBACK + 返回余额不足错误                 │    │
│  └─────────────────────────────────────────────────┘    │
│                          ↓ 通过                          │
│  L4: 更新层保护                                          │
│  ┌─────────────────────────────────────────────────┐    │
│  │ UPDATE ... WHERE version=old_version                │    │
│  │ affected_rows == 1 ？                               │    │
│  │ → 不等于1则 ROLLBACK + 版本冲突告警                  │    │
│  └─────────────────────────────────────────────────┘    │
│                          ↓ 通过                          │
│  L5: DB层兜底                                            │
│  ┌─────────────────────────────────────────────────┐    │
│  │ UNIQUE(biz_type, order_no, wallet_type)             │    │
│  │ → 万一L2失效（Redis宕机），DB层仍阻止重复写入        │    │
│  │                                                     │    │
│  │ ※ 可选追加: CHECK(balance >= 0) 约束                │    │
│  │   TiDB支持CHECK约束（5.7+兼容模式）                  │    │
│  │   作为代码缺陷的最终安全网                            │    │
│  └─────────────────────────────────────────────────┘    │
└──────────────────────────────────────────────────────────┘
```

### 6.3 每层防线的失效场景与下一层接管

| 层 | 正常拦截 | 可能失效场景 | 下一层接管 |
|----|---------|-------------|-----------|
| L1 | 非法参数 | 调用方传了合法但不应该的参数 | L2/L3 |
| L2 | 重复操作 | Redis宕机 | L5(DB UNIQUE) |
| L3 | 余额不足 | 代码Bug跳过了检查 | L4(version)兜底 |
| L4 | 并发冲突 | 极端情况version正好匹配 | L5(DB UNIQUE) |
| L5 | 重复INSERT | 理论上不会被穿透 | CHECK约束最终兜底 |

### 6.4 available计算公式

```
可用余额 = balance - frozen

其中:
  balance = 账户总余额（只在 Credit/Debit 时变化）
  frozen  = 冻结金额（只在 Freeze/Unfreeze/DebitFrozen 时变化）

扣款校验: available >= debit_amount
冻结校验: available >= freeze_amount
```

**关键区分**：
- Credit(入账)：balance += amount，frozen 不变
- Debit(直接扣款)：balance -= amount，frozen 不变
- Freeze(冻结)：balance 不变，frozen += amount
- Unfreeze(解冻)：balance 不变，frozen -= amount
- DebitFrozen(从冻结扣)：balance -= amount，frozen -= amount

## 第 7 章 [P1-C5] 分布式事务与跨服务一致性

### 7.1 问题定义

**场景**：钱包和财务是独立服务，操作需要跨服务协调：
- 充值：财务确认支付成功 → 调钱包入账 → 入账失败怎么办？
- 提现：钱包冻结 → 调财务建单 → 建单失败怎么办？

**如果不处理**：
- 财务记录"已入账"但钱包余额没增加 → 用户投诉
- 钱包冻结了但财务没有对应订单 → 余额永久锁死

### 7.2 方案比选

| 方案 | 原理 | 优势 | 劣势 | 适用场景 |
|------|------|------|------|---------|
| **A: 同步RPC + 幂等重试** | 调用方同步调用，失败则幂等重试 | 实现最简单、维护成本最低 | 调用方需要可靠重试机制 | **2服务简单编排** |
| B: Saga（编排式） | 每步有补偿操作，失败执行逆向补偿 | 理论完备 | 代码量3x、补偿逻辑复杂、调试困难 | 多服务长链路(5+服务) |
| C: TCC（Try-Confirm-Cancel） | 两阶段预留/确认/取消 | 强一致 | 侵入性极大、需要所有服务适配TCC接口 | 银行核心系统 |
| D: 本地消息表 | 写操作+消息同事务，消费方拉取 | 最终一致 | 延迟大、需要消息中间件 | 高吞吐异步场景 |

### 7.3 最优方案：同步RPC + 幂等重试 + 定时补偿

**推荐方案：A = 同步RPC + 幂等 + Cron补偿**

**推荐理由**：
1. Stripe、支付宝均验证：2服务间同步+幂等足够
2. 钱包↔财务只有2个服务参与，不是5+服务长链路
3. Saga/TCC带来的3x代码复杂度在当前阶段是过度设计
4. 幂等重试 = 调用方重试 + 被调用方幂等，天然满足最终一致

**各场景的一致性策略**：

```
┌──────────────────────────────────────────────────────────┐
│              场景1: 充值回调入账                           │
│                                                          │
│  财务确认三方回调 → 财务调 WalletCredit(orderNo)          │

│                  ├────┤                                  │
│                  │失败│ → 财务记录待补偿                   │
│                  │    │   → Cron每5分钟扫描待入账订单重试   │

│                                                          │
│  ⚠️ 一致性保证:                                          │
│  - 钱包幂等 → 重试不会多入账                              │
│  - 财务持续重试 → 最终一定入账                            │
│  - 最坏情况: 用户等待数分钟（Cron周期）后看到余额           │
└──────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────┐
│              场景2: 提现冻结后建单                         │

│                        │    │     有 → 保持冻结            │

│                                                          │
```

## 第 8 章 [P1-C6] 冻结-确认/解冻原子性

### 8.1 问题定义

**场景**：提现流程涉及"冻结→审核→扣款/解冻"的多步操作，中间任一步失败需要保证状态一致

```
正常流程(审核通过):
  Freeze(800) → [审核] → DebitFrozen(800)
  balance: 1000 → 1000 → 200
  frozen:  0    → 800  → 0
  available: 1000 → 200 → 200 ✓

正常流程(审核驳回):
  Freeze(800) → [审核] → Unfreeze(800)
  balance: 1000 → 1000 → 1000
  frozen:  0    → 800  → 0
  available: 1000 → 200 → 1000 ✓

异常流程(冻结成功但后续失败):
  Freeze(800) → [建单失败] → ???
  balance: 1000 → 1000
  frozen:  0    → 800    ← 资金被锁死！
```

### 8.2 最优方案：先冻后建 + 孤儿冻结巡检

**时序选择：方案A — 先冻后建（Freeze First）**

```
用户发起提现
  │
  ├─ ① 钱包校验(KYC/限额/余额)
  │
  ├─ ② Wallet.Freeze(amount)    ← 先冻结
  │     成功 → 继续
  │     失败 → 直接返回错误（安全，什么都没发生）
  │
  ├─ ③ Finance.CreateWithdrawOrder(...)  ← 后建单
  │     成功 → 正常流程，等待审核
  │     失败 → 立即 Wallet.Unfreeze(amount) 补偿
  │              ├─ Unfreeze成功 → 返回用户"提现失败请重试"
  │              └─ Unfreeze也失败 → 标记为"待补偿"，Cron处理
  │
  └─ 返回结果

为什么不先建后冻:
  建单成功 + 冻结失败 → 审核通过 → 出款 → 钱出去了但余额未冻
  → 资金损失！不可逆！
```

**孤儿冻结巡检Cron**：

```
每5分钟执行:
  1. 扫描 withdraw_order 表: status='pending' AND create_at < now()-10min
  2. 对比 wallet.frozen 是否有对应冻结
  3. 若有冻结但无财务订单 → 自动Unfreeze
  4. 若有冻结且有财务订单 → 正常等待（不处理）
  5. 记录补偿日志
```

### 8.3 Freeze/Unfreeze/DebitFrozen 操作模式

```go
// Freeze: frozen += amount, balance不变
func Freeze(wallet, amount):
    available := wallet.Balance - wallet.Frozen
    if available < amount: return ErrInsufficientBalance
    wallet.Frozen += amount     // 冻结增加
    // balance不变
    INSERT wallet_transaction(direction=freeze, ...)

// Unfreeze: frozen -= amount, balance不变
func Unfreeze(wallet, amount):
    if wallet.Frozen < amount: return ErrFrozenInsufficient
    wallet.Frozen -= amount     // 冻结减少
    // balance不变
    INSERT wallet_transaction(direction=unfreeze, ...)

// DebitFrozen: balance -= amount, frozen -= amount
func DebitFrozen(wallet, amount):
    if wallet.Frozen < amount: return ErrFrozenInsufficient
    wallet.Balance -= amount    // 总余额减少
    wallet.Frozen -= amount     // 冻结减少
    // available不变（因为balance和frozen同步减少）
    INSERT wallet_transaction(direction=debit_frozen, ...)
```

## 第 9 章 [P1-C7] 状态机与生命周期管理

### 9.1 问题定义

**场景**：订单存在多个状态，状态间的转换必须严格受控，非法转换会导致业务混乱

**如果不处理**：
- "已失效"的充值订单被回调入账 → 重复入账
- "已驳回"的提现订单被出款 → 资金损失
- 状态直接跳跃（待支付→已完成） → 丢失中间处理

### 9.2 最优方案：显式状态机 + 转换白名单

**充值订单状态机**：

```
                     ┌───────────────┐
                     │   pending     │ (待支付)
                     └───────┬───────┘
                             │
                    用户发起支付
                             │
                     ┌───────▼───────┐
                     │  processing   │ (支付中)
                     └───┬───────┬───┘
                         │       │
                  三方回调成功  三方回调失败/超时过期
                         │       │
                  ┌──────▼──┐ ┌──▼──────┐
                  │ success  │ │ failed  │
                  │ (成功)   │ │ (失败)  │
                  └─────────┘ └─────────┘

合法转换白名单:
  pending    → processing  (用户发起支付)
  pending    → failed      (超时未支付，Cron标记)
  processing → success     (三方回调确认)
  processing → failed      (三方回调失败/超时)

⛔ 非法转换:
  success → 任何状态      (终态不可变)
  failed → 任何状态       (终态不可变)
  processing → pending    (不可逆)
```

**提现订单状态机**：

```
┌───────────┐  冻结+建单   ┌───────────┐
│  pending   │ ──────────→ │ reviewing  │ (审核中)
└───────────┘              └─────┬──────┘
                                 │
                    ┌────────────┼────────────┐
                  驳回           │          审核通过
                    │            │            │
              ┌─────▼─────┐     │     ┌──────▼──────┐
              │ rejected   │     │     │ approved    │ (待出款)
              │ (已驳回)   │     │     └──────┬──────┘
              └───────────┘     │            │
                                │    三方出款结果
                  锁定(异常)     │     ┌──────┼──────┐
                    │            │   成功     │    失败
              ┌─────▼─────┐     │     │      │      │
              │  locked    │←───┘  ┌──▼───┐  │  ┌───▼────┐
              │ (锁定)    │       │success│  │  │ failed  │
              └───────────┘       │(完成) │  │  │ (失败)  │
                                  └──────┘  │  └────────┘
                                            │
                                       人工处理
                                            │
                                    ┌───────▼───────┐
                                    │  locked       │
                                    │ (需人工介入)   │
                                    └───────────────┘

合法转换白名单:
  reviewing → rejected   (任一级审核驳回 → 触发Unfreeze)
  reviewing → approved   (全部审核通过)
  approved  → success    (三方出款成功 → 触发DebitFrozen)
  approved  → failed     (三方出款失败 → 触发Unfreeze)
  approved  → locked     (出款异常不确定 → 人工介入)

余额联动:
  reviewing进入 → Freeze已执行
  rejected     → Unfreeze
  success      → DebitFrozen
  failed       → Unfreeze
  locked       → 保持冻结（等人工处理）
```

### 9.3 代码实现模式

```go
// 状态转换白名单定义
var depositTransitions = map[string][]string{
    "pending":    {"processing", "failed"},
    "processing": {"success", "failed"},
    "success":    {},  // 终态
    "failed":     {},  // 终态
}

var withdrawTransitions = map[string][]string{
    "reviewing": {"rejected", "approved"},
    "approved":  {"success", "failed", "locked"},
    "rejected":  {},  // 终态
    "success":   {},  // 终态
    "failed":    {},  // 终态
    "locked":    {"success", "failed"},  // 人工处理后可转
}

// 状态转换校验函数
func ValidateTransition(table string, from string, to string) error {
    var transitions map[string][]string
    switch table {
    case "deposit_order":
        transitions = depositTransitions
    case "withdraw_order":
        transitions = withdrawTransitions
    }

    allowed, ok := transitions[from]
    if !ok {
        return fmt.Errorf("unknown status: %s", from)
    }

    for _, a := range allowed {
        if a == to {
            return nil  // 合法转换
        }
    }

    return fmt.Errorf("illegal transition: %s → %s", from, to)
}

// UPDATE时WHERE带上当前状态，防止并发状态跳跃
result := tx.Model(&DepositOrder{}).
    Where("id = ? AND status = ?", orderID, currentStatus).
    Update("status", newStatus)
if result.RowsAffected != 1 {
    return ErrStatusConflict  // 状态已被其他操作改变
}
```

## 第 10 章 [P1-C8] 死锁预防

### 10.1 问题定义

**场景**：兑换操作需要同时锁定两个钱包（法币扣款 + BSB加款），不同请求锁定顺序不一致会死锁

```
请求A: 兑换 VND → BSB
  Lock VND wallet → Wait BSB wallet lock

请求B: 兑换 THB → BSB (或其他涉及BSB的操作)
  Lock BSB wallet → Wait VND wallet lock

  → 死锁！互相等待
```

### 10.2 最优方案：固定锁序 + 超时检测

**规则：按 (currency_code ASC, wallet_type ASC) 排序后依次加锁**

```go
// 兑换操作: 需要同时锁定 source wallet 和 target wallet
func (r *ExchangeRepo) Execute(ctx context.Context, req ExchangeReq) error {
    return r.db.WithContext(ctx).Transaction(func(tx *gorm.DB) error {
        // ① 确定锁序: currency_code 字典序排列
        walletKeys := []WalletKey{
            {UserID: req.UserID, CurrencyCode: req.SourceCurrency, WalletType: req.SourceWalletType},
            {UserID: req.UserID, CurrencyCode: req.TargetCurrency, WalletType: req.TargetWalletType},
        }
        sort.Slice(walletKeys, func(i, j int) bool {
            if walletKeys[i].CurrencyCode != walletKeys[j].CurrencyCode {
                return walletKeys[i].CurrencyCode < walletKeys[j].CurrencyCode
            }
            return walletKeys[i].WalletType < walletKeys[j].WalletType
        })

        // ② 按固定顺序依次 FOR UPDATE
        wallets := make(map[string]*model.Wallet)
        for _, key := range walletKeys {
            var w model.Wallet
            err := tx.Set("gorm:query_option", "FOR UPDATE").
                Where("user_id=? AND currency_code=? AND wallet_type=?",
                    key.UserID, key.CurrencyCode, key.WalletType).
                First(&w).Error
            if err != nil {
                return err
            }
            wallets[key.CurrencyCode+":"+fmt.Sprint(key.WalletType)] = &w
        }

        // ③ 两个钱包都锁定后执行扣款+入账
        // ... (业务逻辑)

        return nil
    })
}
```

**额外防御：捕获死锁错误重试一次**

```go
err := r.Execute(ctx, req)
if isMySQLDeadlock(err) {
    // TiDB/MySQL error 1213 = deadlock
    log.Warn("deadlock detected, retry once", "err", err)
    err = r.Execute(ctx, req) // 重试一次
}
```

### 10.3 其他涉及多行锁的场景

| 场景 | 涉及行数 | 锁序规则 |
|------|---------|---------|
| 兑换(法币→BSB) | 2-3行（源钱包+目标中心+目标奖励） | currency_code ASC → wallet_type ASC |
| 投注扣款(中心+奖励) | 2行 | wallet_type ASC (1=中心先, 2=奖励后) |
| 投注返奖(中心+奖励) | 2行 | wallet_type ASC |

## 第 11 章 [P1-C9] 多钱包优先级扣款与比例返还

### 11.1 问题定义

**场景**：用户投注1500 BSB，中心钱包1000，奖励钱包800

```
问题1: 扣款顺序？
  → 中心优先，不足部分从奖励扣

问题2: 如果这笔投注赢了，返奖3000 BSB，怎么分？
  → 按扣款比例分配
  → 中心扣1000/1500 = 66.7%，奖励扣500/1500 = 33.3%
  → 中心返 3000×66.7% = 2000，奖励返 3000×33.3% = 1000
```

### 11.2 最优方案：原子扣款 + 比例快照 + 比例返还

```
┌────────────────────────────────────────────────────────┐
│                多钱包优先级扣款流程                       │
│                                                        │
│  Input: user_id, currency=BSB, total_amount=1500       │
│                                                        │
│  ① 同一事务内 FOR UPDATE 锁定两个钱包                   │
│     center_wallet:  balance=1000, type=1               │
│     reward_wallet:  balance=800,  type=2               │
│     (按wallet_type ASC锁序)                            │
│                                                        │
│  ② 优先级扣款计算                                      │
│     center_debit = min(center.available, 1500) = 1000  │
│     remain = 1500 - 1000 = 500                         │
│     reward_debit = min(reward.available, 500) = 500    │
│     remain = 500 - 500 = 0   ← 凑齐                   │
│     if remain > 0: return ErrInsufficientBalance       │
│                                                        │
│  ③ 执行扣款（同一事务）                                 │
│     center.balance -= 1000                             │
│     reward.balance -= 500                              │
│     INSERT 2条 wallet_transaction                      │
│                                                        │
│  ④ 返回比例快照                                        │
│     ratio = {                                          │
│       "center": "0.6667",   // 1000/1500               │
│       "reward": "0.3333"    // 500/1500                │
│     }                                                  │
│     → 调用方（游戏模块）必须存储此ratio                  │
│                                                        │
│  Output: {success, ratio, txn_ids}                     │
└────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────┐
│                比例返还流程                              │
│                                                        │
│  Input: user_id, currency=BSB, win_amount=3000,        │
│         ratio={"center":"0.6667","reward":"0.3333"}    │
│                                                        │
│  ① 按比例计算各钱包入账金额                             │
│     center_credit = 3000 × 0.6667 = 2000.10           │
│     reward_credit = 3000 × 0.3333 = 999.90            │
│                                                        │
│  ② 尾差处理（保证 center + reward = total）             │
│     center_credit = 3000 - reward_credit = 2000.10     │
│     → 尾差归中心钱包（对用户有利）                       │
│                                                        │
│  ③ 同一事务内两个钱包入账                               │
│     center.balance += 2000.10                          │
│     reward.balance += 999.90                           │
│     INSERT 2条 wallet_transaction                      │
└────────────────────────────────────────────────────────┘
```

### 11.3 关键规则

1. **中心优先**：先扣中心，不足部分从奖励补
2. **比例快照**：扣款时返回ratio，调用方必须存储
3. **返还必传ratio**：返奖时必须传入ratio，钱包按此分配
4. **尾差归中心**：精度截断的尾差统一归入中心钱包
5. **全部在一个事务**：扣两个钱包 + 写两条流水 = 一个事务

## 第 14 章 [P2-C12] 性能取舍与可扩展性

### 14.1 余额不缓存的决策与论证

**决策：ADR-02 — 不缓存余额到Redis**

```
┌─────────────────────────────────────────────────────────┐
│                为什么余额不能缓存？                       │
│                                                         │
│  假设使用Cache-Aside:                                    │
│                                                         │
│  T0: balance=100 (DB=100, Cache=100)                    │
│  T1: 请求A: 读Cache得100, 准备扣80                       │
│  T2: 请求B: 读Cache得100, 准备扣80                       │
│  T3: 请求A: DB扣款成功, DB=20, 删Cache                   │
│  T4: 请求B: DB扣款... balance=20 >= 80? NO → 被拦截 ✓    │
│                                                         │
│  看起来没问题？但是:                                      │
│                                                         │
│  T3': 请求A: DB扣款成功, DB=20                           │
│  T3'': 请求C: 读DB得20, 回填Cache=20                     │
│  T4': 请求A: 删Cache（但请求C刚回填了！）                 │
│       → 删除的是请求C回填的Cache                          │
│       → Cache=空 → 下次读DB → OK                         │
│                                                         │
│  真正的问题:                                             │
│  T3''': 请求A: DB扣款成功, DB=20, 写Cache=20             │
│  T3'''': 请求D: 入账+5000, DB=5020                      │
│  T4'''': 请求D: 写Cache=5020                            │
│  T5: 请求A的Cache写延迟到达: Cache=20 (覆盖了5020!)      │
│       → Cache=20, DB=5020 → 不一致！                     │
│                                                         │
│  金融系统: 不一致=透支风险 → 不可接受                     │
│                                                         │
│  性能够用吗？                                            │
│  TiDB PK查询: <1ms                                      │
│  用户级操作频率: ~1次/秒                                  │
│  结论: 不需要缓存，DB直查足够                             │
│                                                         │
│  参考: Stripe、支付宝核心资金系统均不缓存余额              │
└─────────────────────────────────────────────────────────┘
```

### 14.2 币种配置缓存策略

**与余额不同，币种配置适合缓存**：
- 变更频率极低（管理员手动操作，日均<10次）
- 读取频率极高（每笔交易都需要币种信息）
- 不一致后果可控（最多用了旧的精度/汇率几秒钟）

```
缓存策略: Cache-Aside + 写时删除

写入流程:
  ① 更新DB
  ② 删除Redis缓存
  ③ 下次读取时回填缓存

缓存Key与TTL:
  currency:list          TTL=5min    (全部启用币种列表)
  currency:info:{code}   TTL=5min    (单币种详情)
  currency:rate:{code}   TTL=3min    (汇率，更短TTL保鲜)
  base_currency          TTL=24h     (基础币种，几乎不变)
```

### 14.3 流水表增长治理

**问题**：wallet_transaction 是append-only，增长最快

```
预估增长:
  每日活跃用户10万
  每用户日均操作: 5笔（投注+入账+查询触发的流水）
  每日新增: 50万条
  每月: 1500万条
  每年: 1.8亿条

  ├─ 短期(首版): TiDB自动分片，无需人工分表
  ├─ 中期(3-6月): 按create_at分区（月分区）
  ├─ 长期(1年+): 冷热分离
  │   ├─ 热数据: 最近3个月(TiDB)
  │   └─ 冷数据: 归档到TiDB冷存储或ClickHouse
  └─ 索引优化:
      ├─ 主查询: idx_user_currency_time(user_id, currency_code, create_at)
```

### 14.4 锁等待与热点账户

```

  │   ├─ 方案B: 分桶账户(一个逻辑账户拆成N个子账户)
  │   └─ 方案C: 异步归集(先记录，定期批量汇总)
  └─ 当前判断: 无需提前优化，先跑起来再看
```

## 第 15 章 [P2-C13] 汇率精度与三源聚合

### 15.1 问题定义

**场景**：汇率来自3个第三方API，可能数据异常（某API返回极端值/返回失败）

### 15.2 最优方案：三源中位数 + 偏离告警 + 降级策略

```
┌────────────────────────────────────────────────────────┐
│               汇率三源聚合算法                           │
│                                                        │
│  每3分钟(Cron)对每个启用币种执行:                        │
│                                                        │
│  ① 并发请求3个汇率API                                  │
│     API-A: rate_a = 25100.50                           │
│     API-B: rate_b = 25098.00                           │
│     API-C: rate_c = 25200.00 ← 偏高                   │
│                                                        │
│  ② 计算中位数(排序取中间值)                             │
│     sorted: [25098.00, 25100.50, 25200.00]             │
│     median = 25100.50                                  │
│                                                        │
│  ③ 异常检测(偏离中位数>X%)                              │
│     API-C偏离: |25200-25100.5|/25100.5 = 0.4%         │
│     if 偏离 > threshold(如2%): 剔除该API，用另外2个均值  │
│                                                        │
│  ④ 与平台汇率比对                                      │
│     platform_rate = 25080.00 (上一次设定值)             │
│     变化: |25100.5-25080|/25080 = 0.08%               │
│     if 变化 > fluctuation%(如5%):                      │
│       → 自动更新平台汇率 = median                      │
│       → 写入 currency_rate_log 审计表                  │
│       → 发送告警通知运营确认                            │
│     else:                                              │
│       → 保持原平台汇率不变                              │
│                                                        │
│  ⑤ 降级策略                                            │
│     1个API失败: 用另外2个均值                           │
│     2个API失败: 用剩余1个                               │
│     3个全失败: 本轮跳过,沿用上次汇率 + P2告警            │
│     连续3轮全失败: P1告警 + 考虑暂停兑换功能             │
└────────────────────────────────────────────────────────┘
```

### 15.3 三种汇率的区别与使用场景

| 汇率类型 | 用途 | 更新频率 | 来源 |
|---------|------|---------|------|
| **实时汇率(real_rate)** | 三方API最新值，仅参考 | 每3分钟 | 三源聚合中位数 |
| **平台汇率(platform_rate)** | 充值/提现/兑换实际使用 | 变化超阈值时更新 | real_rate触发更新 |
| **入金汇率(inbound_rate)** | 充值时 法币→BSB 换算 | = platform_rate × (1 + fluctuation%) | 计算值 |
| **出金汇率(outbound_rate)** | 提现时 BSB→法币 换算 | = platform_rate × (1 - fluctuation%) | 计算值 |

**fluctuation%**：配置在currency表中，每个币种独立，代表平台的汇率差价利润

### B.1 统一余额操作模板（第3章的完整扩展）

```go
// ============================================================
// account_repo.go — 钱包模块最关键的文件
// 所有余额变更的唯一入口，集成了 C1+C2+C3+C4 四个P0方案
// ============================================================

type BalanceOpReq struct {
    UserID       int64
    CurrencyCode string
    WalletType   int32
    Direction    int32   // 1=income, 2=expense
    OpType       string  // credit/debit/freeze/unfreeze/debit_frozen
    Amount       string  // decimal string
    BizType      string
    OrderNo      string
    Remark       string
}

type BalanceOpResp struct {
    BalanceBefore string
    BalanceAfter  string
    FrozenBefore  string
    FrozenAfter   string
    TransactionID int64
    Idempotent    bool  // true=本次为幂等返回（已存在）
}

func (r *AccountRepo) ExecuteBalanceOp(ctx context.Context, req *BalanceOpReq) (*BalanceOpResp, error) {
    // [C4-L1] 参数层校验
    if err := validateParams(req); err != nil {
        return nil, err
    }

    // [C2-L1] Redis幂等快检
    cacheKey := fmt.Sprintf("idempotent:%s:%s:%d", req.BizType, req.OrderNo, req.WalletType)
    cached, err := r.redis.Get(ctx, cacheKey).Result()
    if err == nil && cached != "" {
        return unmarshalCachedResp(cached), nil  // 幂等快返回
    }

    var resp *BalanceOpResp

    // 进入事务
    err = r.db.WithContext(ctx).Transaction(func(tx *gorm.DB) error {
        // [C4-L2] / [C2-L2] 事务内幂等检查
        var existingTxn model.WalletTransaction
        err := tx.Where("biz_type=? AND order_no=? AND wallet_type=?",
            req.BizType, req.OrderNo, req.WalletType).
            First(&existingTxn).Error
        if err == nil {
            // 已存在 → 幂等返回
            resp = &BalanceOpResp{
                BalanceBefore: existingTxn.BalanceBefore,
                BalanceAfter:  existingTxn.BalanceAfter,
                TransactionID: existingTxn.ID,
                Idempotent:    true,
            }
            return nil
        }

        // [C1] FOR UPDATE 锁定钱包行
        var wallet model.Wallet
        err = tx.Set("gorm:query_option", "FOR UPDATE").
            Where("user_id=? AND currency_code=? AND wallet_type=?",
                req.UserID, req.CurrencyCode, req.WalletType).
            First(&wallet).Error
        if err != nil {
            return fmt.Errorf("wallet not found: %w", err)
        }

        // [C3] Decimal精度计算
        balance := decimal.RequireFromString(wallet.Balance)
        frozen := decimal.RequireFromString(wallet.Frozen)
        amount := decimal.RequireFromString(req.Amount)
        available := balance.Sub(frozen)

        // [C4-L3] 业务校验
        newBalance := balance
        newFrozen := frozen

        switch req.OpType {
        case "credit":
            newBalance = balance.Add(amount)
        case "debit":
            if available.LessThan(amount) {
                return ErrInsufficientBalance
            }
            newBalance = balance.Sub(amount)
        case "freeze":
            if available.LessThan(amount) {
                return ErrInsufficientBalance
            }
            newFrozen = frozen.Add(amount)
        case "unfreeze":
            if frozen.LessThan(amount) {
                return ErrFrozenInsufficient
            }
            newFrozen = frozen.Sub(amount)
        case "debit_frozen":
            if frozen.LessThan(amount) {
                return ErrFrozenInsufficient
            }
            newBalance = balance.Sub(amount)
            newFrozen = frozen.Sub(amount)
        }

        // [C4-L4] UPDATE + version自增
        result := tx.Model(&model.Wallet{}).
            Where("id=? AND version=?", wallet.ID, wallet.Version).
            Updates(map[string]interface{}{
                "balance":   newBalance.String(),
                "frozen":    newFrozen.String(),
                "version":   wallet.Version + 1,
                "update_at": time.Now().UnixMilli(),
            })
        if result.RowsAffected != 1 {
            return ErrVersionConflict
        }

        // [C4-L5] INSERT流水（UNIQUE约束最终兜底）
        txnRecord := &model.WalletTransaction{
            UserID:        req.UserID,
            CurrencyCode:  req.CurrencyCode,
            WalletType:    req.WalletType,
            Direction:     req.Direction,
            Amount:        req.Amount,
            BalanceBefore: wallet.Balance,
            BalanceAfter:  newBalance.String(),
            FrozenBefore:  wallet.Frozen,
            FrozenAfter:   newFrozen.String(),
            BizType:       req.BizType,
            OrderNo:       req.OrderNo,
            Remark:        req.Remark,
            CreateAt:      time.Now().UnixMilli(),
        }
        if err := tx.Create(txnRecord).Error; err != nil {
            return err
        }

        resp = &BalanceOpResp{
            BalanceBefore: wallet.Balance,
            BalanceAfter:  newBalance.String(),
            FrozenBefore:  wallet.Frozen,
            FrozenAfter:   newFrozen.String(),
            TransactionID: txnRecord.ID,
            Idempotent:    false,
        }
        return nil
    })

    if err != nil {
        return nil, err
    }

    // [C2-L1] 事务成功后回填Redis缓存
    if resp != nil && !resp.Idempotent {
        cacheData, _ := json.Marshal(resp)
        r.redis.Set(ctx, cacheKey, string(cacheData), 24*time.Hour)
    }

    return resp, nil
}
```

```go
// cron/orphan_freeze.go — 集成 C6(冻结原子性) + C10(容错补偿)
func (c *OrphanFreezeCleanup) Run(ctx context.Context) {
    // 查询"待确认"状态且超时的提现记录
    threshold := time.Now().Add(-10 * time.Minute).UnixMilli()

    orders, err := c.withdrawRepo.FindPendingBefore(ctx, threshold)
    if err != nil {
        log.Error("orphan freeze scan failed", "err", err)
        return
    }

    for _, order := range orders {
        // 检查财务侧是否有对应订单
        hasFinanceOrder, err := c.financeRPC.CheckOrderExists(ctx, order.FinanceOrderNo)
        if err != nil {
            log.Warn("finance check failed, skip", "order", order.OrderNo)
            continue // 查不到就跳过，下轮再检
        }

        if !hasFinanceOrder {
            // 无对应财务订单 → 孤儿冻结 → 自动解冻
            _, err := c.accountRepo.ExecuteBalanceOp(ctx, &BalanceOpReq{
                UserID:       order.UserID,
                CurrencyCode: order.CurrencyCode,
                WalletType:   order.WalletType,
                OpType:       "unfreeze",
                Amount:       order.FreezeAmount,
                BizType:      "orphan_unfreeze",
                OrderNo:      "ORPHAN_" + order.OrderNo, // 保证幂等
            })
            if err != nil {
                log.Error("orphan unfreeze failed", "order", order.OrderNo, "err", err)
            } else {
                log.Info("orphan freeze cleaned", "order", order.OrderNo)
                // 更新本地提现订单状态为cancelled
                c.withdrawRepo.UpdateStatus(ctx, order.ID, "reviewing", "cancelled")
            }
        }
    }
}
```

> **本文档总结**：
>
> 钱包模块在编码实现阶段面临**15个核心挑战**，其中**4个P0级（资金安全红线）**必须在Phase 2之前100%解决。
>
> 这15个挑战并非孤立，而是高度关联：并发安全(C1)是根基，幂等(C2)和精度(C3)是弹药，五层防线(C4)是集大成。跨服务一致性(C5)依赖幂等(C2)，冻结原子性(C6)依赖补偿(C10)。
>
> 所有方案均来自业界验证的最优实践（Stripe/支付宝/微信支付），没有过度设计（不用Saga/TCC），没有不足设计（不缺任何必要防护层）。
>
> **一句话总结**：FOR UPDATE保一致，双层幂等防重复，Decimal保精度，五层防线防透支，先冻后建保安全，Cron补偿保最终一致。
