## 议题 1：并发安全——余额超卖与竞态条件

### 1.1 为什么不是普通CRUD

普通CRUD：读一条记录、改一个字段、存回去。并发时最多数据不一致，不会造成资金损失。

钱包场景：用户余额1000，同时来了两个扣款请求各600。如果两个请求都先读到余额1000，都判断"够扣"，都执行减600，余额变成-200。**用户凭空多出了200的消费额度，平台亏钱。**

### 1.2 不解决的后果

- 余额超卖（变负数）= 资金安全事故
- 冻结超扣（frozen变负数）= 提现数据混乱
- 同一笔充值入账两次 = 平台损失

### 1.3 解决方案：SELECT FOR UPDATE 悲观行锁

**选型决策（ADR-1）**：

| 方案 | 适用场景 | 我们的场景 | 结论 |
|------|---------|-----------|------|
| 乐观锁(Version) | 读多写少，冲突率低 | 余额写密集，冲突率高 | 淘汰：重试成本高 |
| Redis分布式锁 | 跨服务/跨DB锁 | 单DB操作 | 淘汰：Redis挂=锁失效 |
| **SELECT FOR UPDATE** | 单DB，写密集 | 完全匹配 | **采用** |

**为什么选悲观锁而非乐观锁？**
- 钱包余额是高争用资源：充值、扣款、冻结、解冻可能同时到达
- 乐观锁在争用下退化为"读→计算→写→冲突→重读→重算→重写→又冲突..."循环
- 悲观锁：排队等 < 10ms，一次成功，不用重试

### 1.4 代码级落地

```go
// Balance Engine 核心写路径
func (e *BalanceEngine) ModifyBalance(ctx context.Context, p ModifyBalanceParams) error {
    // ① 事务外：幂等检查（不占锁时间）
    if e.isIdempotent(ctx, p.RefOrderNo, p.TxType, p.WalletType) {
        return nil // 已处理过，直接返回成功
    }

    // ② 进入事务
    return query.Q.Transaction(func(tx *query.Query) error {
        // ③ 行锁：精确锁定一行
        account, err := tx.WalletAccount.WithContext(ctx).
            Where(tx.WalletAccount.UserID.Eq(p.UserID)).
            Where(tx.WalletAccount.CurrencyCode.Eq(p.CurrencyCode)).
            Where(tx.WalletAccount.WalletType.Eq(p.WalletType)).
            ForUpdate(). // SELECT ... FOR UPDATE
            First()
        if err != nil {
            return err // 账户不存在
        }

        // ④ 校验+计算（在锁内完成，保证读到的是最新值）
        newAvail, newFrozen, err := e.compute(account, p.OpType, p.Amount)
        if err != nil {
            return err // BALANCE_NOT_ENOUGH 或 FROZEN_NOT_ENOUGH
        }

        // ⑤ 更新余额（SQL级原子表达式，双重保险）
        result, err := tx.WalletAccount.WithContext(ctx).
            Where(tx.WalletAccount.ID.Eq(account.ID)).
            UpdateSimple(
                tx.WalletAccount.AvailableBalance.Value(newAvail),
                tx.WalletAccount.FrozenBalance.Value(newFrozen),
            )
        if err != nil || result.RowsAffected == 0 {
            return ErrUpdateFailed
        }

        // ⑥ 写流水（同一事务，原子提交）
        return tx.TransactionLog.WithContext(ctx).Create(&model.TransactionLog{
            UserID:          p.UserID,
            CurrencyCode:    p.CurrencyCode,
            WalletType:      p.WalletType,
            TxType:          p.TxType,
            RefOrderNo:      p.RefOrderNo,
            BeforeAvailable: account.AvailableBalance,
            BeforeFrozen:    account.FrozenBalance,
            ChangeAmount:    p.Amount,
            AfterAvailable:  newAvail,
            AfterFrozen:     newFrozen,
            Remark:          p.Remark,
        })
    })
}
```

**锁的粒度和性能**：
- 锁粒度 = 唯一索引 `uk_user_currency_type(user_id, currency_code, wallet_type)` = 精确一行
- 不同用户、不同币种、不同钱包类型 → 互不阻塞
- 锁持有时间目标 < 5ms（事务内只做读→算→写→插入→提交）
- 单账户TPS约100（足够当前规模，远超实际需求）

**额外防线：SQL级非负守卫（参考Java钱包检验模式）**：

```go
// compute 方法中，扣减类操作的计算
func (e *BalanceEngine) compute(acc *model.WalletAccount, opType int8, amount int64) (int64, int64, error) {
    switch opType {
    case OpCredit: // 入账：无条件
        return acc.AvailableBalance + amount, acc.FrozenBalance, nil

    case OpDebit: // 扣减：available必须够
        if acc.AvailableBalance < amount {
            return 0, 0, ErrBalanceInsufficient
        }
        return acc.AvailableBalance - amount, acc.FrozenBalance, nil

    case OpFreeze: // 冻结：available必须够
        if acc.AvailableBalance < amount {
            return 0, 0, ErrBalanceInsufficient
        }
        return acc.AvailableBalance - amount, acc.FrozenBalance + amount, nil

    case OpUnfreeze: // 解冻：frozen必须够
        if acc.FrozenBalance < amount {
            return 0, 0, ErrFrozenInsufficient
        }
        return acc.AvailableBalance + amount, acc.FrozenBalance - amount, nil

    case OpDebitFrozen: // 扣冻结：frozen必须够，资金离开系统
        if acc.FrozenBalance < amount {
            return 0, 0, ErrFrozenInsufficient
        }
        return acc.AvailableBalance, acc.FrozenBalance - amount, nil
    }
    return 0, 0, ErrInvalidOpType
}
```

## 议题 2：幂等性——重复请求防护

### 2.1 为什么不是普通CRUD

普通CRUD：重复创建一条记录，最多多一行数据，删掉就行。

钱包场景：充值回调到了两次（网络重传），如果不做幂等，用户余额多加一次。**100块变200块，平台白送100块。**

### 2.2 不解决的后果

- 充值重复入账 = 平台资金损失
- 提现重复扣款 = 用户资金损失 + 投诉
- 手动调账重复执行 = 账务混乱

### 2.3 解决方案：DB唯一索引为核心 + Redis快速拦截

**为什么不是只用Redis？** Redis有TTL过期——24小时后幂等键消失，迟到的重试就挡不住。DB唯一索引是永久的，和业务数据在同一个事务里原子提交。

**为什么不是只用DB？** 每次都走DB查询太慢。Redis SetNX是O(1)，拦截99%的重复请求，DB唯一索引只兜底那1%。

### 2.4 代码级落地

**幂等键三元组设计**：

```
唯一索引：uk_ref_order_tx(ref_order_no, tx_type, wallet_type)

为什么是三个字段而非单个order_no？

场景1：手动加款(orderNo=A123, 中心+奖励)
  → CreditBalance(A123, manual_add, walletType=1) ← 允许
  → CreditBalance(A123, manual_add, walletType=2) ← 允许
  → 如果只用orderNo，第二次调用会被误拦截

场景2：提现冻结+解冻(同一个orderNo=T456)
  → FreezeBalance(T456, freeze, walletType=1)    ← 允许
  → UnfreezeBalance(T456, unfreeze, walletType=1) ← 允许
  → 如果只用orderNo+walletType，解冻会被误拦截

结论：(order_no, tx_type, wallet_type) 三者组合才能唯一标识一笔操作
```

**双层幂等实现**：

```go
// Layer 1: Redis快速拦截
func (e *BalanceEngine) isIdempotent(ctx context.Context, orderNo string, txType int8, walletType int8) bool {
    key := fmt.Sprintf("{wal:idem}:%s:%d:%d", orderNo, txType, walletType)
    // SetNX: 设置成功=第一次请求，设置失败=重复请求
    ok, err := redis.Client.SetNX(ctx, key, "1", 24*time.Hour).Result()
    if err != nil {
        // Redis故障，降级到DB层检查，不阻塞业务
        return false
    }
    return !ok // SetNX返回false = key已存在 = 幂等命中
}

// Layer 2: DB唯一索引兜底（在transaction_log表上）
// CREATE UNIQUE INDEX uk_ref_order_tx ON transaction_log(ref_order_no, tx_type, wallet_type);
// INSERT时如果触发唯一冲突 → 捕获错误 → 返回成功（非错误）

func (e *BalanceEngine) handleDuplicateKeyError(err error) error {
    if isDuplicateKeyError(err) {
        return nil // 幂等命中，视为成功
    }
    return err
}
```

**Redis故障时的恢复逻辑**：

```go
// 场景：Redis设置了幂等键，但DB事务回滚了
// 下次重试时：Redis说"已处理"，但DB里没有流水记录
// 解决：Redis幂等命中时，再查一次DB确认

func (e *BalanceEngine) isIdempotent(ctx context.Context, orderNo string, txType int8, walletType int8) bool {
    key := fmt.Sprintf("{wal:idem}:%s:%d:%d", orderNo, txType, walletType)
    ok, err := redis.Client.SetNX(ctx, key, "1", 24*time.Hour).Result()
    if err != nil {
        return false // Redis不可用，降级到DB
    }
    if !ok {
        // Redis说已存在，但需要确认DB确实有这条记录
        exists, _ := e.txnRepo.ExistsByOrderKey(ctx, orderNo, txType, walletType)
        if !exists {
            // DB里没有 → 上次事务失败了 → 删掉Redis键 → 让本次重新执行
            redis.Client.Del(ctx, key)
            return false
        }
        return true // DB确认存在，确实是重复请求
    }
    return false // 第一次请求
}
```

## 议题 3：事务一致性——余额与流水的原子绑定

### 3.1 为什么不是普通CRUD

普通CRUD：更新一个表成功就行，失败就报错。

钱包场景：**余额变了但流水没写** = 对账永远对不上 = 无法审计 = 财务灾难。**流水写了但余额没变** = 用户看到扣了钱但余额没少 = 投诉。

### 3.2 铁律

```
★ 余额变更和流水记录必须在同一个DB事务内 ★
没有例外。没有"先改余额后补流水"。没有"MQ异步写流水"。

同一事务内：余额UPDATE和流水INSERT要么同时成功，要么同时回滚。
对账公式永远成立：最新流水.after_available == wallet_account.available_balance
```

### 3.3 解决方案：本地事务（GORM Transaction）

```go
// 所有余额操作遵循这个模式——没有例外
query.Q.Transaction(func(tx *query.Query) error {
    // Step 1: 锁行+读余额
    account, err := tx.WalletAccount...ForUpdate().First()

    // Step 2: 计算新余额
    newAvail, newFrozen, err := compute(account, opType, amount)

    // Step 3: 更新余额
    tx.WalletAccount...UpdateSimple(...)

    // Step 4: 写流水（同一事务）
    tx.TransactionLog...Create(&model.TransactionLog{
        BeforeAvailable: account.AvailableBalance,
        AfterAvailable:  newAvail,
        // ...
    })

    // 两者一起COMMIT或一起ROLLBACK
    return nil
})
```

### 3.4 transaction_log 表设计要点

```sql
CREATE TABLE transaction_log (
    id              BIGINT UNSIGNED NOT NULL,        -- Snowflake ID（非自增，分布式友好）
    user_id         BIGINT UNSIGNED NOT NULL,
    currency_code   VARCHAR(10) NOT NULL,
    wallet_type     TINYINT NOT NULL,
    tx_type         TINYINT NOT NULL,                -- 1=deposit, 2=withdraw_freeze, ...
    ref_order_no    VARCHAR(32) NOT NULL,            -- 关联订单号
    direction       TINYINT NOT NULL,                -- 1=收入, 2=支出
    change_amount   BIGINT NOT NULL,                 -- 变动金额（正数）
    before_available BIGINT NOT NULL,                -- 操作前可用
    before_frozen   BIGINT NOT NULL,                 -- 操作前冻结
    after_available BIGINT NOT NULL,                 -- 操作后可用
    after_frozen    BIGINT NOT NULL,                 -- 操作后冻结
    remark          VARCHAR(256) DEFAULT '',
    created_at      BIGINT UNSIGNED NOT NULL,        -- 毫秒时间戳

    PRIMARY KEY (id),
    UNIQUE KEY uk_ref_order_tx (ref_order_no, tx_type, wallet_type),  -- 幂等索引
    KEY idx_user_currency_type_time (user_id, currency_code, tx_type, created_at),
```

**不可变性约束**（代码级强制）：
- 无 `update_at` 字段 → 永不UPDATE
- 无 `delete_at` 字段 → 永不DELETE
- Repository层不提供Update/Delete方法
- 代码Review检查项：transaction_log只有Create，没有Update/Delete

## 议题 4：金额精度——不丢一分钱

### 4.1 为什么不是普通CRUD

普通CRUD：金额存个float、算个乘除，差几分钱无所谓。

钱包场景：
- `0.1 + 0.2 = 0.30000000000000004`（IEEE 754浮点）
- 用户充值 100.10 USDT，系统存了 100.09999999...，对账差0.00000001
- 千万用户×千万次操作 = 积少成多的巨大损失

### 4.2 解决方案：BIGINT 最小单位整数存储

**选型决策（ADR-2）**：

| 方案 | DB类型 | 运算方式 | 精度风险 | 结论 |
|------|--------|---------|---------|------|
| float64 | DOUBLE | Go原生浮点 | ★★★★★ 灾难级 | 淘汰 |
| DECIMAL(20,8) | DECIMAL | shopspring/decimal | ★☆☆☆☆ 极低 | 可选 |
| **BIGINT最小单位** | BIGINT | Go int64 | **零** | **采用** |

**最终决定：BIGINT**。理由：
- 整数运算零精度损失——`500000 + 300000 = 800000`，永远精确
- Go原生int64，无需引入第三方库做基础运算
- Stripe、支付宝、微信支付均采用最小单位整数

### 4.3 币种精度配置表

```
currency_config 表中的字段：
  decimal_digits  TINYINT  -- 该币种小数位数
  multiplier      BIGINT   -- 最小单位倍率

VND:  decimal_digits=0, multiplier=1        (1 VND  = 1)
IDR:  decimal_digits=0, multiplier=1        (1 IDR  = 1)
THB:  decimal_digits=2, multiplier=100      (1 THB  = 100)
BSB:  decimal_digits=2, multiplier=100      (1 BSB  = 100)
USDT: decimal_digits=8, multiplier=10^8     (1 USDT = 100000000)
```

**用户看到的 vs 系统存的**：

```
用户看到：充值 500.00 THB
系统存的：amount = 50000  (500 × 100)

用户看到：充值 10.50000000 USDT
系统存的：amount = 1050000000  (10.5 × 10^8)

用户看到：充值 500000 VND
系统存的：amount = 500000  (500000 × 1)
```

### 4.4 代码级落地

```go
// 前端→后端：接收用户输入的显示金额，转为BIGINT
func DisplayToInternal(displayAmount string, currencyCode string) (int64, error) {
    cfg := GetCurrencyConfig(currencyCode)
    d, err := decimal.NewFromString(displayAmount)
    if err != nil {
        return 0, ErrInvalidAmount
    }
    // 乘以倍率，截断取整（不四舍五入，截断保护平台）
    result := d.Mul(decimal.NewFromInt(cfg.Multiplier)).IntPart()
    if result <= 0 {
        return 0, ErrInvalidAmount
    }
    return result, nil
}

// 后端→前端：BIGINT转为显示金额
func InternalToDisplay(internalAmount int64, currencyCode string) string {
    cfg := GetCurrencyConfig(currencyCode)
    d := decimal.NewFromInt(internalAmount)
    result := d.Div(decimal.NewFromInt(cfg.Multiplier))
    return result.StringFixed(int32(cfg.DecimalDigits))
}
```

**汇率存储**：

```
所有汇率统一乘以 10^8 存为 BIGINT

1 USDT = 25,620 VND
  → 存储：rate = 2562000000000 (25620 × 10^8)

兑换计算：
  用户兑换 500000 VND → BSB
  BSB_rate = 250000000000 (2500 × 10^8，即1 BSB = 2500 VND)
  BSB_amount = 500000 × 10^8 / 250000000000 = 200 (×100最小单位 = 20000)

计算规则：先乘后除，防止中间结果丢失精度
```

**shopspring/decimal 使用场景**：

```go
// 只在涉及除法的中间计算时使用decimal，最终结果转回int64
func CalculateExchange(fiatAmount int64, rate int64) int64 {
    fiat := decimal.NewFromInt(fiatAmount)
    r := decimal.NewFromInt(rate)
    base := decimal.NewFromInt(1e8) // 汇率倍率

    // BSB = fiatAmount × 10^8 / rate
    bsb := fiat.Mul(base).Div(r)

    // 截断取整（用户收到的金额向下截断，保护平台）
    return bsb.IntPart()
}
```

### 4.5 int64 溢出检查

```
int64 最大值：9,223,372,036,854,775,807 (约 9.2 × 10^18)

最大场景估算：
  1亿用户 × 每人 10亿 USDT (不可能) = 10^17 最小单位
  远小于 int64 上限

但中间计算可能溢出（先乘后除的"乘"步骤）：
  fiatAmount = 10^12 (大额VND)
  × 10^8 (汇率倍率) = 10^20 > int64上限

解决：中间计算使用 decimal.Decimal，最终结果转 int64
```

### 4.6 行业参考

- **Stripe**：所有金额以最小单位整数传输（cents），API返回integer
- **支付宝**：内部使用分为单位的BIGINT
- **微信支付**：金额单位为分(BIGINT)
- **Coinbase**：加密货币使用最小单位整数（satoshi/wei）

## 议题 5：跨服务一致性——没有分布式事务怎么办

### 5.1 为什么不是普通CRUD

普通CRUD：操作一个数据库，本地事务搞定。

钱包场景：充值链路跨3个服务——wallet创建订单 → finance对接渠道 → 第三方支付 → finance收回调 → wallet入账。任何一步失败或超时，数据怎么保持一致？

### 5.2 选型决策（ADR-4）

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| Seata/DTM | 自动化分布式事务 | 工程无先例、引入复杂度、性能差 | 淘汰 |
| 本地消息表 | 可靠 | 维护成本高、延迟 | 备选 |
| **幂等RPC + 超时重试 + 定时补偿** | 简单、工程有先例、微信支付同方案 | 最终一致非强一致 | **采用** |

### 5.3 三板斧——每条跨服务链路都用这套

**第一斧：幂等RPC**
```
每个写入RPC都自带幂等键(refOrderNo)
调用方可以安全地无限重试
钱包侧的幂等三元组保证不会重复执行
```

**第二斧：超时重试**
```
超时时间：3s (核心路径) / 5s (非核心)
重试次数：2次（总共3次尝试）
退避策略：指数退避 100ms → 200ms
安全前提：幂等保证重试安全
```

**第三斧：定时补偿**
```
定时任务扫描"应该完成但还没完成"的订单
主动查询对方状态 → 补偿处理
三层回调保障：主动回调(95%) → 定时补偿(4%) → 人工补单(1%)
```

### 5.4 三条跨服务链路的一致性方案

**充值链路（最终一致）**：

```
wallet创建订单 →(同步RPC)→ finance匹配渠道
  ↓ 成功：创建pending_payment订单
  ↓ 失败：不创建订单（无脏数据）

      ⋯⋯ 用户支付 ⋯⋯

finance收到回调 →(同步RPC)→ wallet.CreditBalance
  ↓ 成功：入账完成
  ↓ 超时：finance重试CreditBalance（幂等安全）
  ↓ 回调丢失：定时补偿扫描 → 主动查渠道状态 → 补偿入账

一致性保证：
  ① 幂等：同一订单号重复入账 = 只执行一次
  ② 补偿：回调丢失由定时任务兜底
  ③ 人工：最终兜底，运营创建补单
```

**提现链路（TCC模式）**：

```
Try:     wallet冻结余额(available→frozen) + 创建订单（同一DB事务）
Confirm: finance代付成功 → wallet.DebitBalance(从frozen扣除)
Cancel:  finance审核拒绝 → wallet.UnfreezeBalance(frozen→available)

一致性保证：
  ① 原子冻结：冻结和建单在同一事务，不可能出现"冻结了但没建单"
  ② 每步幂等：Freeze/Unfreeze/DebitFrozen各有独立幂等键
  ③ 保守策略：代付失败保持冻结，不自动解冻（防止渠道实际已付款）
```

**手动调账链路（幂等循环）**：

```
finance审核通过 → 循环调用wallet.CreditBalance/DebitBalance
  每次调用带相同orderNo + 不同walletType
  幂等三元组保证不重复
  超时重试安全

原子性选项：
  方案A：ManualAdjust原子RPC（一个事务搞定多个钱包）
  方案B：分多次调用（部分失败需额外处理）
  推荐方案A
```

### 5.5 行业参考

- **微信支付**：核心资金链路使用"幂等+重试+补偿"三板斧，不用分布式事务框架
- **支付宝**：TCC模式用于担保交易（冻结→确认→取消）
- **Stripe**：Webhook重试机制 + Idempotency Key

## 议题 6：状态机正确性——提现七态三终态

### 6.1 为什么不是普通CRUD

普通CRUD：status字段随便改，最多改错了修回来。

提现场景：status决定了用户的钱在哪——在可用余额？在冻结？已离开系统？**status改错了 = 钱的归属出错 = 资金事故。**

### 6.2 解决方案：严格单向状态机 + 行级CAS

```go
// 状态定义和合法转换
var validTransitions = map[int8][]int8{
    StatusPendingRisk:   {StatusRiskRejected, StatusPendingFinance},
    StatusPendingFinance:{StatusFinanceRejected, StatusPendingPayout},
    StatusPendingPayout: {StatusPayoutFailed, StatusPayoutSuccess},
    // 终态不在map中 = 不可再转换
}

// 状态转换方法：UPDATE ... WHERE status = expected
func (r *WithdrawOrderRepo) TransitionStatus(ctx context.Context, orderNo string, fromStatus, toStatus int8) (bool, error) {
    result, err := query.Q.WithdrawOrder.WithContext(ctx).
        Where(query.Q.WithdrawOrder.OrderNo.Eq(orderNo)).
        Where(query.Q.WithdrawOrder.Status.Eq(fromStatus)). // CAS：当前状态必须匹配
        UpdateSimple(query.Q.WithdrawOrder.Status.Value(toStatus))
    if err != nil {
        return false, err
    }
    return result.RowsAffected > 0, nil
    // RowsAffected=0 说明当前状态不是fromStatus → 被别的流程先改了 → 等价于幂等
}
```

**并发审核竞态保护**：

```go
// 两个管理员同时审核同一单
// Admin A: TransitionStatus(T001, PendingRisk, RiskRejected)  → RowsAffected=1 ✓
// Admin B: TransitionStatus(T001, PendingRisk, PendingFinance) → RowsAffected=0 ✗（已被A改了）

// 加上 lock_by 机制：
// Admin A锁单 → UPDATE SET lock_by=A WHERE lock_by=0 AND order_no=T001
// Admin B锁单 → RowsAffected=0（已被A锁了）
```

**每个状态转换对应的余额操作**：

```
1→2 (风控拒绝):    UnfreezeBalance (frozen→available)
1→3 (风控通过):    无余额操作
3→4 (财务拒绝):    UnfreezeBalance (frozen→available)
3→5 (财务通过):    无余额操作（等待代付）
5→6 (代付失败):    ★ 无余额操作（保持冻结，等人工）
5→7 (代付成功):    DebitBalance(txType=withdraw_confirm, 从frozen扣除)

代码绑定：状态转换和余额操作在同一个方法中调用，不允许分离
```

### 6.3 行业参考

- **支付宝**：交易状态机严格单向，不可逆转
- **Stripe**：Payment Intent 状态机（requires_payment_method → processing → succeeded/failed）

## 议题 7：死锁预防——跨币种兑换的多行锁

### 7.1 为什么不是普通CRUD

普通CRUD：操作一行数据，不存在死锁。

兑换场景：一个事务要同时锁VND中心钱包和BSB中心钱包。如果线程A先锁VND再锁BSB，线程B先锁BSB再锁VND → **死锁**。

### 7.2 解决方案：固定字典序加锁

```go
// 兑换操作：固定按 (currency_code, wallet_type) 升序加锁
func (s *ExchangeService) Execute(ctx context.Context, p ExchangeParams) error {
    return query.Q.Transaction(func(tx *query.Query) error {
        // 确定加锁顺序：按字典序排列
        // BSB(B) < VND(V)，所以先锁BSB，再锁VND
        lockOrder := sortAccountKeys([]AccountKey{
            {CurrencyCode: "BSB", WalletType: 1}, // center
            {CurrencyCode: "BSB", WalletType: 2}, // reward（如有加赠）
            {CurrencyCode: p.FromCurrency, WalletType: 1}, // center
        })

        // 按排定顺序逐个加锁
        accounts := make(map[string]*model.WalletAccount)
        for _, key := range lockOrder {
            acc, err := tx.WalletAccount.WithContext(ctx).
                Where(...).ForUpdate().First()
            accounts[key.String()] = acc
        }

        // 所有行都已锁定，安全执行跨币种操作
        // ... 扣VND、加BSB中心、加BSB奖励 ...
        return nil
    })
}

// 排序函数
func sortAccountKeys(keys []AccountKey) []AccountKey {
    sort.Slice(keys, func(i, j int) bool {
        if keys[i].CurrencyCode != keys[j].CurrencyCode {
            return keys[i].CurrencyCode < keys[j].CurrencyCode
        }
        return keys[i].WalletType < keys[j].WalletType
    })
    return keys
}
```

**规则**：项目中所有需要锁多行的操作（兑换、优先级扣款），都必须遵循这个排序规则。

### 7.3 行业参考

- **MySQL官方文档**：推荐固定加锁顺序防止死锁
- **银行清算系统**：按账号升序加锁

## 议题 8：汇率系统——四层汇率+外部API容错

### 8.1 为什么不是普通CRUD

普通CRUD：配置项改了就改了，影响不大。

汇率场景：汇率决定了"用户充100 VND换多少BSB"。**汇率更新出错 = 所有充值/提现/兑换金额全错 = 全平台资金风险。**

### 8.2 四层汇率体系

```
第1层：实时汇率 (realtime_rate)
  来源：3个第三方API取均值
  频率：每5-15分钟更新
  用途：B端展示"当前市场汇率"

第2层：平台汇率 (platform_rate)
  来源：当实时汇率与上次平台汇率偏差 >= 阈值时更新
  用途：兑换计算基准

第3层：充值汇率 (deposit_rate)
  公式：platform_rate × (1 + float_pct)
  用途：充值时的汇率（用户多付一点，平台的汇差收入）

第4层：提现汇率 (withdraw_rate)
  公式：platform_rate × (1 - float_pct)
  用途：提现时的汇率（用户少收一点）
```

### 8.3 外部API三级容错

```go
func (s *RateService) FetchRealtimeRates(ctx context.Context) (map[string]int64, error) {
    results := make([]map[string]int64, 0, 3)
    errors := make([]error, 0)

    // 并行调用3个API
    var wg sync.WaitGroup
    var mu sync.Mutex
    for _, api := range s.rateAPIs {
        wg.Add(1)
        go func(a RateAPI) {
            defer wg.Done()
            rates, err := a.Fetch(ctx)
            mu.Lock()
            defer mu.Unlock()
            if err != nil {
                errors = append(errors, err)
            } else {
                results = append(results, rates)
            }
        }(api)
    }
    wg.Wait()

    switch len(results) {
    case 3: return average(results), nil          // 3个都成功：取均值
    case 2: return average(results), nil          // 1个失败：取2个均值
    case 1:
        alert.Warn("仅1个汇率API可用")
        return results[0], nil                    // 2个失败：用剩下1个+告警
    case 0:
        alert.Critical("所有汇率API不可用")
        return nil, ErrAllRateAPIsFailed          // 全失败：保旧值+告警
    }
    return nil, ErrUnexpected
}
```

### 8.4 Quote-Lock：汇率快照锁定


### 8.5 偏差阈值机制

```go
// 只有偏差超过阈值才更新平台汇率（防频繁更新）
func (s *RateService) ShouldUpdatePlatformRate(realtimeRate, platformRate int64, thresholdPct int64) bool {
    if platformRate == 0 {
        return true // 首次运行，直接设置
    }
    // deviation = |realtime - platform| / platform × 10000（万分比）
    diff := abs(realtimeRate - platformRate)
    deviation := diff * 10000 / platformRate
    return deviation >= thresholdPct
}
```

### 9.1 第一层：接入安全

```
④ 单笔最小/最大限额
```

```
同userID + 同currencyCode + 同amount + status=pending
  >= 3条：硬拦截
  1-2条：软提醒

```

**敏感数据加密**：

**流水不可篡改**：

## 议题 10：性能工程——不拖慢核心链路

### 10.1 最小持锁策略

```
★ 事务内只做：锁行→读→算→写→插入→提交
★ 事务外做：参数校验、幂等检查、日志记录、缓存操作、RPC调用

反模式（参考Java钱包的教训）：
  ✗ 事务内调RPC → 网络超时=长时间持锁
  ✗ 事务内写日志文件 → IO阻塞=持锁时间不可控
  ✗ 事务内更新缓存 → Redis超时=持锁时间延长
```

### 10.2 缓存策略

| 数据 | 是否缓存 | 策略 | 理由 |
|------|---------|------|------|
| **余额** | **否** | 每次查DB | 资金数据宁慢勿错；行锁查询<1ms |
| 币种配置 | 是 | Redis TTL 5min + 修改时主动刷新 | 高读低写 |
| 汇率 | 是 | Redis，定时任务更新后主动写入 | 高读，更新有规律 |
| 幂等键 | 是 | Redis SetNX 24h | 快速拦截重复 |
| 每日提现额度 | 是 | Redis INCRBY原子累加 | 高频写入 |
| 基准币种 | 是 | Redis无过期 | 设一次不变 |

### 10.3 索引设计要点

```sql
-- wallet_account: 唯一行定位（所有余额操作的锁目标）
UNIQUE KEY uk_user_currency_type (user_id, currency_code, wallet_type)

-- transaction_log: 幂等索引（最频繁的写入检查）
UNIQUE KEY uk_ref_order_tx (ref_order_no, tx_type, wallet_type)

-- transaction_log: C端记录查询（分页场景）
KEY idx_user_currency_type_time (user_id, currency_code, tx_type, created_at)

-- deposit_order: 重复充值检测
KEY idx_user_currency_status (user_id, currency_code, status)

-- deposit_order: 超时订单扫描
KEY idx_status_create_at (status, created_at)
```

### 10.4 Snowflake ID vs 自增ID

```
transaction_log 使用 Snowflake ID 而非 AUTO_INCREMENT
理由：
  ① 分布式友好：未来分表/分库无需改ID生成策略
  ② TiDB性能：自增ID导致写入集中在同一个Region热点
     Snowflake ID分散写入，充分利用TiDB多Region并行
  ③ 安全：不暴露自增序列（防推测数据量）
```

## 议题 12：可扩展性——新币种零代码接入

## 附录B：参考项目教训总结

### Java法币钱包（参考项目A）——采纳与规避

| 采纳 | 规避 |
|------|------|
| SQL原子表达式 `amount = amount + ?` | 无幂等索引（MQ重发=重复记录） |
| Freeze/Deduct/Rollback三操作模型 | 锁在Controller层（Service层无保护） |
| Before/After余额快照 | 批量操作只锁第一个用户 |
| 懒创建钱包（首次操作时创建） | Rollback不检查frozen>=amount |
| SQL非负守卫 `AND available >= ?` | 10+冗余字段写入每条流水 |

### TRC20虚拟币充值（参考项目B）——采纳与规避

| 采纳 | 规避 |
|------|------|
| 多地址架构（收款/归集/出款/Gas分离） | 提现5步无事务包裹 |
| 两阶段充值确认（链上确认后再入账） | RSA 512位密钥（不安全） |
| 外部签名服务分离（出款密钥不在应用内存） | AES/ECB模式（应用AES/GCM） |
| Gas费自动补充 | fee_limit硬编码 |
| 链交互层三层抽象 | 异常被吞（catch后只print） |

## 附录C：12个议题的一句话速记

```
1. 并发安全   → SELECT FOR UPDATE 悲观行锁，锁一行 < 5ms
2. 幂等性     → DB唯一索引(order+type+wallet)为核心，Redis SetNX为加速
3. 事务一致性  → 余额UPDATE和流水INSERT同一事务，无例外
4. 金额精度   → BIGINT最小单位整数，shopspring/decimal做除法
5. 跨服务一致  → 幂等RPC + 超时重试 + 定时补偿（三板斧）
6. 状态机     → UPDATE WHERE status=expected（CAS），严格单向
7. 死锁预防   → 多行锁按(currency_code, wallet_type)字典序
8. 汇率系统   → 四层汇率 + 三API容错 + Quote-Lock快照
9. 安全纵深   → 接入→业务→数据→对账，四层防御
10. 性能工程  → 最小持锁 + 余额不缓存 + Snowflake ID
11. 可观测性  → TraceID全链路 + [BALANCE_OP]结构化日志
12. 可扩展性  → 配置驱动 + Balance Engine无感知 + 预留复式记账
```
