## 一、挑战全景总览

### 1.1 为什么钱包不是普通CRUD

普通CRUD：请求进来 → 校验参数 → 读写数据库 → 返回结果。失败了最多重试一次，数据错了改回来就行。

钱包CRUD：每一笔操作都是**不可逆的资金变动**。多扣一分钱是资损，少扣一分钱是漏洞，重复扣款是事故。区别在于：

| 维度 | 普通CRUD | 钱包操作 |
|------|---------|---------|
| 并发影响 | 最多脏读 | 余额被多扣/少扣 |
| 重复请求 | 幂等覆盖即可 | 重复扣款=资损 |
| 精度要求 | 前端展示级 | 分/厘级零误差 |
| 失败处理 | 返回错误 | 冻结态必须有人收尾 |
| 审计要求 | 日志即可 | 每笔变动可追溯、可对账 |

## 二、P0-并发余额操作安全

### 2.1 问题描述

**场景**：用户A同时发起两笔消费请求（比如下注+打赏），两个请求同时读到余额=100元，各扣50元，最终余额=50元（应该=0元）。这就是经典的**并发超扣**问题。

**影响面**：直接资损，平台亏钱。

**发生概率**：高——多端操作（APP+H5）、RPC内部调用（投注服务+打赏服务同时扣款）都可能触发。

### 2.2 方案对比

| 方案 | 原理 | 优点 | 缺点 | 适用性 |
|------|------|------|------|--------|
| ① 悲观锁 SELECT FOR UPDATE | 行级锁，先锁再改 | 强一致，实现简单 | 并发高时锁等待 | **采用** |
| ② 乐观锁 version | WHERE version=old_ver | 无锁等待 | 高冲突时大量重试 | 不适合（资金操作重试代价高） |
| ③ CAS UPDATE SET balance=balance-X WHERE balance>=X | 原子扣减 | 无锁，性能好 | 无法获取变更前后快照 | 不适合（流水记录需要before/after） |
| ④ Redis分布式锁 | 先抢锁再操作 | 跨服务锁 | 引入Redis强依赖 | 不适合（单DB事务够用） |

### 2.3 选定方案：悲观锁（SELECT FOR UPDATE）

**决策理由**：
1. 钱包所有余额变动在**同一个TiDB实例**的**同一个事务**内完成，不需要跨数据库锁
2. 必须获取`balance_before`和`balance_after`写入流水，CAS做不到
3. 正常业务并发不会很高（不是秒杀场景），锁等待在可接受范围

**关键事实**：当前整个代码库**没有任何地方使用FOR UPDATE**，钱包将是第一个引入行级锁的服务。

### 2.4 实现代码（伪代码级）

```go
// internal/ledger/ledger.go

// Debit 扣减余额（统一入口，所有扣减操作必须经过此方法）
func (l *Ledger) Debit(ctx context.Context, tx *gorm.DB, param DebitParam) error {
    // 第一步：锁定钱包行（SELECT FOR UPDATE）
    var wallet model.UserWallet
    err := tx.WithContext(ctx).
        Clauses(clause.Locking{Strength: "UPDATE"}).
        Where("user_id = ? AND currency_id = ? AND wallet_type = ? AND delete_at = 0",
            param.UserID, param.CurrencyID, param.WalletType).
        First(&wallet).Error
    if err != nil {
        if errors.Is(err, gorm.ErrRecordNotFound) {
            return errs.WalletNotFound
        }
        return err
    }

    // 第二步：余额校验（锁定后的余额是准确的）
    if wallet.Balance < param.Amount {
        return errs.BalanceInsufficient  // 错误码: 6023201
    }

    // 第三步：扣减余额
    newBalance := wallet.Balance - param.Amount
    err = tx.WithContext(ctx).
        Model(&wallet).
        Update("balance", newBalance).Error
    if err != nil {
        return err
    }

    // 第四步：写入流水（幂等保障在这里，见第三章）
    flow := model.WalletFlow{
        UserID:        param.UserID,
        CurrencyID:    param.CurrencyID,
        WalletType:    param.WalletType,
        FlowType:      FlowTypeDebit,
        Amount:        -param.Amount,  // 扣减为负数
        BalanceBefore: wallet.Balance,
        BalanceAfter:  newBalance,
        OrderNo:       param.OrderNo,
        BizType:       param.BizType,
        Remark:        param.Remark,
    }
    return tx.WithContext(ctx).Create(&flow).Error
}
```

### 2.5 GORM FOR UPDATE 语法要点

```go
// 正确写法 — clause.Locking
import "gorm.io/gorm/clause"

db.Clauses(clause.Locking{Strength: "UPDATE"}).Where(...).First(&wallet)
// 生成SQL: SELECT * FROM user_wallet WHERE ... FOR UPDATE

// 错误写法 — 不要用Raw SQL拼接，会绕过GORM链式调用
// db.Raw("SELECT * FROM user_wallet WHERE ... FOR UPDATE").Scan(&wallet) ❌
```

### 2.6 锁的粒度与范围控制

```
规则1: 锁的粒度 = 行级（WHERE条件命中唯一索引 uk_user_currency_type）
规则2: 锁的持有时间 = 整个事务（从SELECT FOR UPDATE到tx.Commit()）
规则3: 锁的顺序 = 固定按(user_id, currency_id, wallet_type)排序（防死锁，见第六章）
规则4: 事务超时 = 500ms（防止长事务占锁，见第十九章）
```

### 2.7 涉及接口

| 接口 | 锁操作 | 说明 |
|------|--------|------|
| RPC: WalletDebit | Debit | 外部服务扣款 |
| RPC: WalletCredit | Credit | 外部服务加款 |
| RPC: WalletFreeze | Freeze | 冻结余额 |
| RPC: WalletUnfreeze | Unfreeze | 解冻余额 |
| C端: 兑换 | Exchange(Debit+Credit) | 扣A币+加B币 |
| B端: 手动加减款 | Credit/Debit | 管理员操作 |

## 三、P0-操作幂等性保障

### 3.1 问题描述

**场景**：支付服务通过RPC调用`WalletCredit`给用户充值，网络超时后重试 → 如果没有幂等保障，用户余额被加了两次。

**影响面**：直接资损（多加=平台亏，多扣=用户投诉）。

**发生概率**：高——RPC本身有超时重试机制（Kitex默认配置），网络抖动是常态。

### 3.2 方案对比

| 方案 | 原理 | 优点 | 缺点 | 适用性 |
|------|------|------|------|--------|
| ① DB唯一索引 | 流水表唯一键拦截重复 | 强一致，零成本 | 需要调用方传递唯一单号 | **采用** |
| ② Redis幂等键 | SET NX + TTL | 性能好 | Redis宕机=幂等失效 | 不适合（资金不允许弱保障） |
| ③ 幂等表 | 独立表记录已处理ID | 灵活 | 多一张表多一次查询 | 不需要（流水表已经承担此职责） |

### 3.3 选定方案：wallet_flow唯一索引

**核心设计**：`wallet_flow` 表上建唯一索引 `uk_order_wallet_flow(order_no, wallet_type, flow_type, delete_at)`。

**含义**：同一笔订单（order_no）、同一个钱包类型（wallet_type）、同一种操作（flow_type）只能产生一条流水。第二次INSERT会被MySQL/TiDB的唯一约束拦截（Error 1062: Duplicate entry）。

### 3.4 为什么是这三个字段的组合

```
order_no   = "PAY202603021234"    -- 支付订单号（调用方生成，全局唯一）
wallet_type = 1                   -- 中心钱包
flow_type   = 1                   -- 入账(Credit)

组合唯一 → 这笔支付订单只能往中心钱包入账一次

为什么不只用order_no？
→ 一笔兑换订单会产生两条流水：A币扣减(Debit) + B币增加(Credit)
→ 如果只用order_no做唯一，第二条流水就插不进去了
→ 所以需要 order_no + wallet_type + flow_type 三字段组合
```

### 3.5 实现代码

```go
// internal/ledger/ledger.go — 幂等处理模式

func (l *Ledger) Credit(ctx context.Context, tx *gorm.DB, param CreditParam) error {
    // ... 锁定钱包行（同Debit）...

    // 写入流水 — 幂等核心
    flow := model.WalletFlow{
        OrderNo:    param.OrderNo,
        WalletType: param.WalletType,
        FlowType:   FlowTypeCredit,
        // ... 其他字段 ...
    }
    err := tx.WithContext(ctx).Create(&flow).Error

    // 判断是否重复插入
    if err != nil {
        if isDuplicateKeyError(err) {
            // 幂等命中：这笔操作已经处理过了
            // 返回成功（不是返回错误！）
            return nil  // 或返回 ErrAlreadyProcessed 让上层决定
        }
        return err  // 真正的数据库错误
    }

    // 流水插入成功，执行余额变更
    // ... 更新余额 ...
    return nil
}

// isDuplicateKeyError 检测MySQL/TiDB唯一键冲突
func isDuplicateKeyError(err error) bool {
    // MySQL Error 1062: Duplicate entry
    return strings.Contains(err.Error(), "1062")
}
```

### 3.6 关键设计决策

```
决策1: 先插流水，再改余额
  → 流水INSERT是幂等校验点，必须先于余额变更
  → 如果先改余额后插流水，重复请求第一步就多加了钱

决策2: 重复请求返回成功，不返回错误
  → 调用方视角：幂等=无论调几次，效果一样
  → 如果返回错误，调用方会以为失败，可能触发补偿逻辑导致混乱

决策3: delete_at参与唯一索引
  → 软删除后同一组合可以重新创建
  → 唯一索引定义：uk_order_wallet_flow(order_no, wallet_type, flow_type, delete_at)
  → delete_at=0表示有效记录，>0表示已删除（不再占用唯一位）
```

### 3.7 幂等键规范（与调用方约定）

| 调用方 | order_no格式 | 示例 | 说明 |
|--------|-------------|------|------|
| 支付服务 | 支付订单号 | PAY20260302xxxx | 充值入账 |
| 投注服务 | 投注单号 | BET20260302xxxx | 下注扣款 |
| 结算服务 | 结算单号 | STL20260302xxxx | 返奖入账 |
| 钱包自身 | 兑换订单号 | EXC20260302xxxx | 币种兑换 |
| 管理后台 | 操作单号 | ADJ20260302xxxx | 手动调账 |

**约束**：调用方**必须**传递全局唯一的order_no，且重试时传递**相同的**order_no。这是幂等生效的前提。

## 四、P0-数据一致性（事务策略）

### 4.1 问题描述

**场景**：充值操作需要同时完成三件事——① 更新余额、② 写入流水、③ 更新订单状态。如果②成功③失败，余额变了但订单还是"处理中"，数据不一致。

**更严重的场景**：跨服务调用。支付服务通知钱包充值成功，钱包本地事务提交了，但返回RPC响应时网络断了，支付服务以为失败又重试 → 这就需要幂等（第三章）兜底。

### 4.2 方案对比

| 方案 | 原理 | 优点 | 缺点 | 适用性 |
|------|------|------|------|--------|
| ① 本地事务 | 单库ACID | 强一致，简单可靠 | 只能管本地数据 | **采用** |
| ② 分布式事务(Seata/DTM) | 2PC/TCC/Saga | 跨库强一致 | 极大复杂度，团队没经验 | **明确不引入** |
| ③ 最终一致性(MQ) | 事件驱动 | 解耦 | 需要MQ基础设施 | **当前不引入** |
| ④ 幂等RPC+对账 | 重试+定期校验 | 实现简单 | 有短暂不一致窗口 | **跨服务采用** |

### 4.3 选定方案：本地事务（内部）+ 幂等RPC+对账（跨服务）

**本地策略**（ser-wallet内部）：
- 钱包的所有表（user_wallet, wallet_flow, deposit_order, withdraw_order, exchange_order, freeze_record）在**同一个TiDB实例**
- 一笔操作涉及的所有写入放在**一个GORM事务**里，要么全成功要么全回滚
- 不引入分布式事务框架（Seata/DTM），团队没有运维经验，收益不值得

**跨服务策略**（钱包 ↔ 支付/投注/结算）：
- 所有RPC接口做到幂等（第三章方案）
- 调用方超时后可安全重试
- 日终对账发现不一致（第七章方案）

### 4.4 事务实现模式

```go
// 项目统一事务模式（已有工程实践：ser-buser中使用）
func (s *WalletSer) Deposit(ctx context.Context, req *DepositReq) error {
    return s.db.WithContext(ctx).Transaction(func(tx *gorm.DB) error {
        // 所有操作使用同一个tx，保证原子性

        // 1. 通过ledger加款（内部会SELECT FOR UPDATE + 写流水）
        err := s.ledger.Credit(ctx, tx, CreditParam{
            UserID:     req.UserID,
            CurrencyID: req.CurrencyID,
            WalletType: WalletTypeCenter,
            Amount:     req.Amount,
            OrderNo:    req.OrderNo,
            BizType:    BizTypeDeposit,
        })
        if err != nil {
            return err  // 触发回滚
        }

        // 2. 更新充值订单状态
        err = tx.Model(&model.DepositOrder{}).
            Where("order_no = ? AND status = ?", req.OrderNo, OrderStatusPending).
            Update("status", OrderStatusSuccess).Error
        if err != nil {
            return err  // 触发回滚
        }

        return nil  // 全部成功，提交事务
    })
}
```

### 4.5 事务纪律清单

```
纪律1: 事务内禁止RPC调用
  → RPC可能超时，会拖长事务持锁时间
  → 正确做法：RPC调用在事务外，事务内只做DB操作

纪律2: 事务内禁止耗时计算
  → 汇率计算、参数校验等在事务前完成
  → 事务内只做：锁行 → 校验余额 → 改余额 → 写流水

纪律3: 事务超时设置500ms
  → 超过500ms强制回滚，防止慢SQL持锁
  → TiDB pessimistic-txn-max-age 配合使用

纪律4: 一个事务只锁一个用户的钱包
  → 批量操作拆成多个独立事务
  → 例外：兑换操作锁同一用户的两个币种钱包（按固定顺序，见第六章）
```

### 4.6 不引入分布式事务的理由（明确决策）

| 原因 | 说明 |
|------|------|
| 团队经验 | 工程团队没有Seata/DTM运维经验，贸然引入风险高 |
| 架构复杂度 | 2PC需要TC（事务协调器），增加单点故障 |
| 性能影响 | 分布式事务的锁范围更大，性能下降50%+ |
| 业务可替代 | 幂等RPC+对账已经能解决跨服务一致性 |
| 行业实践 | 大多数支付系统也是本地事务+最终一致性，而非强一致 |

## 五、P0-金额精度与溢出防护

### 5.1 问题描述

**场景**：用户充值100.50 VND，系统用float64存储，经过几次加减乘除后变成100.4999999999998 → 显示异常、对账不平。

**更严重的场景**：汇率计算。1 USDT = 25,000 VND，兑换0.001 USDT → 25 VND。如果用浮点数运算，结果可能是24.9999999 → 截断后少给1 VND。

### 5.2 方案对比

| 方案 | 存储 | 计算 | 精度 | 适用性 |
|------|------|------|------|--------|
| ① BIGINT + Decimal库 | 整数(分/厘) | shopspring/decimal | 零精度损失 | **采用** |
| ② DECIMAL(20,8) | 数据库定点数 | 数据库运算 | 数据库级精度 | 不采用（Go运算仍有问题） |
| ③ float64 | 浮点 | 浮点运算 | 有精度损失 | **禁止** |
| ④ string存储 | 字符串 | 转换后运算 | 取决于运算方式 | 过度设计 |

### 5.3 选定方案：BIGINT存储 + shopspring/decimal计算

**存储层**：
```sql
-- user_wallet表
balance       BIGINT NOT NULL DEFAULT 0   -- 余额（最小货币单位整数）
frozen_amount BIGINT NOT NULL DEFAULT 0   -- 冻结金额（最小货币单位整数）

-- wallet_flow表
amount        BIGINT NOT NULL             -- 变动金额（最小货币单位整数，负数=扣减）
balance_before BIGINT NOT NULL            -- 变动前余额
balance_after  BIGINT NOT NULL            -- 变动后余额

-- currency_rate表
rate          VARCHAR(32) NOT NULL        -- 汇率（字符串存储，避免精度损失）
```

**最小货币单位换算**：

| 币种 | 最小单位 | 1元=? | 示例 |
|------|---------|-------|------|
| USDT | 1e-8 (聪) | 100,000,000 | 1.5 USDT = 150,000,000 |
| VND | 1 (盾) | 1 | 50,000 VND = 50,000 |
| IDR | 1 (盾) | 1 | 100,000 IDR = 100,000 |
| THB | 1 (萨当) | 100 | 50.25 THB = 5,025 |
| BSB | 平台币最小单位 | 待定 | 待产品确认 |

### 5.4 shopspring/decimal使用规范

```go
import "github.com/shopspring/decimal"

// ✅ 正确用法
rate := decimal.RequireFromString("25000.12345678")  // 从字符串创建
amount := decimal.NewFromInt(150000000)               // 从整数创建
result := amount.Mul(rate)                            // 精确乘法
intResult := result.IntPart()                         // 截断取整

// ❌ 禁止用法
rate := decimal.NewFromFloat(25000.12345678)  // float已经损失精度了！
amount := 100.50                               // 禁止用float64存储金额
```

### 5.5 溢出防护

```go
// BIGINT最大值: 9,223,372,036,854,775,807 (约 9.2 * 10^18)
// USDT最大值: 92,233,720,368.54775807 USDT（约920亿美元，足够）
// VND最大值: 9,223,372,036,854,775,807 VND（约3.6亿美元，足够）

// 但仍需防护溢出
func safeAdd(a, b int64) (int64, error) {
    if b > 0 && a > math.MaxInt64 - b {
        return 0, errs.AmountOverflow  // 正溢出
    }
    if b < 0 && a < math.MinInt64 - b {
        return 0, errs.AmountOverflow  // 负溢出
    }
    return a + b, nil
}
```

### 5.6 金额校验规则

```go
// 统一金额校验（在handler/ser层，事务前执行）
func validateAmount(amount int64) error {
    if amount <= 0 {
        return errs.InvalidAmount        // 金额必须为正数
    }
    if amount > MaxSingleAmount {
        return errs.AmountExceedLimit    // 单笔上限
    }
    return nil
}
```

**关键原则**：所有金额在进入ledger层之前完成校验，ledger层不做业务校验，只做余额充足性检查。

## 六、P1-跨币种兑换原子性

### 6.1 问题描述

**场景**：用户用100 USDT兑换2,500,000 VND。需要同时完成：
1. USDT钱包扣减100（Debit）
2. VND钱包增加2,500,000（Credit）
3. 写入兑换订单
4. 写入两条流水

如果步骤1成功、步骤2失败 → USDT扣了但VND没到账 → 资损。

**额外风险**：如果用户同时发起两笔兑换（USDT→VND 和 VND→USDT），两个事务分别锁USDT和VND → **死锁**。

### 6.2 解决方案：单事务 + 固定锁顺序

**原子性**：两个钱包的变更在**同一个本地事务**内完成。

**防死锁**：多钱包锁定时，按`(currency_id, wallet_type)`的固定顺序加锁。

### 6.3 实现代码

```go
// internal/ledger/ledger.go

// Exchange 币种兑换（原子操作）
func (l *Ledger) Exchange(ctx context.Context, tx *gorm.DB, param ExchangeParam) error {
    // 第一步：确定锁顺序（按currency_id升序，防死锁）
    type lockTarget struct {
        CurrencyID int64
        WalletType int8
        IsDebit    bool
    }
    targets := []lockTarget{
        {param.FromCurrencyID, param.WalletType, true},   // 扣减方
        {param.ToCurrencyID, param.WalletType, false},     // 增加方
    }
    // 按currency_id排序
    sort.Slice(targets, func(i, j int) bool {
        return targets[i].CurrencyID < targets[j].CurrencyID
    })

    // 第二步：按固定顺序锁定两个钱包
    wallets := make(map[int64]*model.UserWallet)
    for _, t := range targets {
        var wallet model.UserWallet
        err := tx.WithContext(ctx).
            Clauses(clause.Locking{Strength: "UPDATE"}).
            Where("user_id = ? AND currency_id = ? AND wallet_type = ? AND delete_at = 0",
                param.UserID, t.CurrencyID, t.WalletType).
            First(&wallet).Error
        if err != nil {
            return err
        }
        wallets[t.CurrencyID] = &wallet
    }

    // 第三步：校验源钱包余额
    fromWallet := wallets[param.FromCurrencyID]
    if fromWallet.Balance < param.FromAmount {
        return errs.BalanceInsufficient
    }

    // 第四步：执行扣减+增加
    // 扣减源币种
    tx.Model(fromWallet).Update("balance", fromWallet.Balance - param.FromAmount)
    // 增加目标币种
    toWallet := wallets[param.ToCurrencyID]
    tx.Model(toWallet).Update("balance", toWallet.Balance + param.ToAmount)

    // 第五步：写入两条流水（共用同一个order_no，通过flow_type区分）
    // Debit流水
    tx.Create(&model.WalletFlow{
        OrderNo: param.OrderNo, FlowType: FlowTypeDebit,
        CurrencyID: param.FromCurrencyID,
        Amount: -param.FromAmount,
        BalanceBefore: fromWallet.Balance,
        BalanceAfter: fromWallet.Balance - param.FromAmount,
    })
    // Credit流水
    tx.Create(&model.WalletFlow{
        OrderNo: param.OrderNo, FlowType: FlowTypeCredit,
        CurrencyID: param.ToCurrencyID,
        Amount: param.ToAmount,
        BalanceBefore: toWallet.Balance,
        BalanceAfter: toWallet.Balance + param.ToAmount,
    })

    return nil
}
```

### 6.4 死锁防护规则

```
规则: 所有需要锁定多个钱包行的操作，必须按以下顺序加锁：
  排序键 = (currency_id ASC, wallet_type ASC)

示例:
  用户同时发起 USDT(id=1)→VND(id=3) 和 VND(id=3)→USDT(id=1)
  两个事务都按 currency_id 升序 → 都先锁id=1再锁id=3
  → 第二个事务在锁id=1时等待第一个事务释放 → 串行化，无死锁
```

## 七、P1-对账机制

### 7.1 问题描述

**场景**：经过一天的运行，钱包服务处理了上万笔交易。如何确认每一笔都是正确的？余额有没有多加少扣？订单状态有没有不一致？

### 7.2 解决方案：日终批量对账

```
对账时机: 每日凌晨（低峰期）
对账维度:
  ① 余额对账: SUM(wallet_flow.amount) WHERE user_id=X == user_wallet.balance
  ② 订单对账: 钱包流水 vs 支付/投注/结算方的流水
  ③ 流水完整性: 每笔流水的balance_after == 下一笔的balance_before
```

### 7.3 对账实现

```go
// internal/cron/reconciliation.go（定时任务）

// ReconcileBalance 余额对账
func (c *CronJob) ReconcileBalance(ctx context.Context) {
    // 获取昨日有流水变动的所有用户
    userIDs := c.flowRepo.GetActiveUserIDs(ctx, yesterday)

    for _, userID := range userIDs {
        // 1. 查询用户所有钱包当前余额
        wallets := c.walletRepo.GetByUserID(ctx, userID)

        for _, wallet := range wallets {
            // 2. 计算流水累计金额
            flowSum := c.flowRepo.SumAmount(ctx, userID, wallet.CurrencyID, wallet.WalletType)

            // 3. 对比
            if flowSum != wallet.Balance {
                // 不一致！记录差异 + 告警
                c.alertService.Send(ctx, AlertBalanceMismatch{
                    UserID:     userID,
                    WalletType: wallet.WalletType,
                    CurrencyID: wallet.CurrencyID,
                    Expected:   flowSum,
                    Actual:     wallet.Balance,
                    Diff:       wallet.Balance - flowSum,
                })
            }
        }
    }
}

// ReconcileFlowChain 流水链完整性校验
func (c *CronJob) ReconcileFlowChain(ctx context.Context) {
    // 获取昨日所有流水，按(user_id, currency_id, wallet_type, create_at)排序
    flows := c.flowRepo.GetDailyFlows(ctx, yesterday)

    // 校验相邻流水的 balance_after[n] == balance_before[n+1]
    for i := 1; i < len(flows); i++ {
        if flows[i].BalanceBefore != flows[i-1].BalanceAfter {
            c.alertService.Send(ctx, AlertFlowChainBreak{
                FlowID:   flows[i].ID,
                Expected: flows[i-1].BalanceAfter,
                Actual:   flows[i].BalanceBefore,
            })
        }
    }
}
```

### 7.4 与支付团队的对账约定

```
钱包侧输出:
  - 每日流水汇总文件（按order_no分组，含金额/币种/时间）
  - 格式待与支付团队确认

支付侧输出:
  - 每日充值/提现明细（按order_no分组）

比对逻辑:
  - 以order_no为关联键，匹配金额和状态
  - 差异分类：金额不符 / 钱包有支付无 / 支付有钱包无 / 状态不一致
```

## 八、P1-热点账户竞争

### 8.1 问题描述

**场景**：平台运营账户、活动奖池账户，可能被大量并发请求同时操作。SELECT FOR UPDATE在高并发下会导致大量锁等待，吞吐量下降。

### 8.2 解决策略：监控优先，按需优化

**当前阶段策略**：不过度设计，先监控锁等待指标。

**理由**：
1. 钱包服务主要操作的是**用户个人钱包**，天然按user_id分散，不太会产生热点
2. 平台账户操作频率相对较低（每笔充值/提现才触发一次）
3. 过早引入子账户/异步聚合会增加大量复杂度

**监控指标**：
```
指标1: TiDB锁等待时间 → P99 > 100ms 告警
指标2: 单行锁竞争次数 → 同一user_wallet行1秒内被锁>10次 告警
指标3: 事务超时回滚次数 → 突增告警
```

**预案**（如果监控触发告警）：

| 阶段 | 措施 | 复杂度 | 效果 |
|------|------|--------|------|
| 第一步 | 缩短事务持锁时间（优化SQL） | 低 | 提升30% |
| 第二步 | 平台账户拆子账户（按业务类型） | 中 | 提升5-10倍 |
| 第三步 | 异步聚合（先记流水，定期汇总余额） | 高 | 彻底解决 |

## 九、P2-汇率缓存一致性

### 9.1 问题描述

**场景**：运营在后台修改了USDT→VND的汇率，但C端用户看到的还是旧汇率（Redis缓存未更新）→ 用户按旧汇率兑换 → 平台资损或用户投诉。

### 9.2 解决方案：Cache-Aside + 主动写入

```
汇率查询链路（4层优先级）:
  ① 实时汇率（第三方接口） → 定时任务每分钟拉取 → 写入DB + Redis
  ② 平台汇率（运营设置） → B端接口修改时 → 写入DB + 主动刷Redis
  ③ 充值汇率（锁定快照） → 充值时取当前汇率快照
  ④ 提现汇率（锁定快照） → 提现审核通过时取当前汇率快照

缓存策略:
  - 读：先查Redis → 命中返回 → 未命中查DB → 写回Redis
  - 写（运营修改）：先更新DB → 再删除/覆盖Redis缓存
  - 写（定时任务）：先更新DB → 直接覆盖Redis
  - TTL：5分钟（即使主动刷新失败，5分钟后自动过期回源）
```

### 9.3 缓存键设计

```
wallet:rate:{fromCurrencyId}:{toCurrencyId}     → 最新汇率
wallet:rate:deposit:{fromCurrencyId}:{toCurrencyId}  → 充值汇率
wallet:rate:withdraw:{fromCurrencyId}:{toCurrencyId} → 提现汇率
wallet:currency:{currencyId}                     → 币种配置
wallet:currency:list                              → 启用的币种列表
```

### 9.4 汇率更新时序

```
运营修改汇率:
  B端 → handler → ser.UpdateRate() →
    ① tx.Update(rate) → DB写入成功
    ② redis.Del(cacheKey) → 删除旧缓存（下次查询回源）
    ③ 写入rate_change_log → 审计记录

  如果②失败（Redis故障）:
    → 不影响DB数据 → 等TTL过期(5min) → 自动回源 → 最终一致
    → 不做回滚（DB已经是正确的，缓存最终会一致）
```

## 十、P2-RPC超时与重试

### 10.1 问题描述

**场景**：投注服务调用`WalletDebit`扣款，5秒无响应（网络抖动/TiDB慢查询）→ 投注服务超时 → 重试一次 → 但第一次请求其实已经成功扣款了。

### 10.2 解决方案：幂等RPC + 合理超时配置

**已有保障**：第三章的幂等机制确保重试不会重复扣款。

**补充措施**：

```
① 超时配置:
  - RPC客户端超时: 3秒（注意：已发现工程代码中注释写3s实际20min的bug，需修正）
  - RPC服务端处理: 目标 < 500ms（ledger事务）
  - DB事务超时: 500ms

② 重试策略:
  - 调用方负责重试（钱包不主动重试上游）
  - 建议重试策略: 指数退避，最多3次
    第1次重试: 100ms后
    第2次重试: 500ms后
    第3次重试: 2000ms后
  - 幂等保障: 重试使用相同order_no

③ 超时后的状态判断:
  - 调用方超时后不确定结果 → 查询接口确认
  - 钱包提供 QueryByOrderNo 接口（RPC暴露）
  - 调用方先查询确认状态，再决定重试或补偿
```

## 十一、P2-流水表膨胀与分页

### 11.1 问题描述

**场景**：wallet_flow是写入量最大的表。每笔充值/提现/兑换/投注/返奖/打赏都会产生流水。按每日10万笔估算，一年3600万条，三年超1亿。

**影响**：
- C端查询流水列表变慢
- 对账任务扫全表超时
- 深度分页（OFFSET 100000）性能急剧下降

### 11.2 解决方案

**短期（编码阶段就做）**：

```
① 游标分页替代OFFSET分页:
  - C端流水列表使用 cursor（上一页最后一条的ID）而非 page_no
  - SQL: WHERE id < #{lastId} ORDER BY id DESC LIMIT #{pageSize}
  - 性能: O(1) vs OFFSET的O(N)

② 索引优化:
  - 主查询索引: idx_user_currency_wallet_create(user_id, currency_id, wallet_type, create_at)
  - 对账索引: idx_create_at(create_at) — 按时间范围扫描
  - 幂等索引: uk_order_wallet_flow(order_no, wallet_type, flow_type, delete_at)
```

**中期（数据量触发时做）**：

```
③ TiDB自动分区:
  - 按create_at时间范围分区（按月）
  - TiDB原生支持Range Partitioning，对应用透明
  - 查询自动路由到对应分区，不扫全表

④ 冷热分离:
  - 3个月内: 热数据，在线查询
  - 3个月外: 冷数据，归档到历史表或对象存储
  - C端只查热数据，管理后台/对账可查历史表
```

### 11.3 游标分页实现

```go
// internal/rep/wallet_flow_rep.go

// ListByUserCursor 游标分页查询流水（C端）
func (r *WalletFlowRep) ListByUserCursor(ctx context.Context, param FlowListParam) ([]model.WalletFlow, error) {
    query := r.db.WithContext(ctx).
        Where("user_id = ? AND currency_id = ? AND delete_at = 0",
            param.UserID, param.CurrencyID)

    // 可选筛选
    if param.WalletType > 0 {
        query = query.Where("wallet_type = ?", param.WalletType)
    }
    if param.BizType > 0 {
        query = query.Where("biz_type = ?", param.BizType)
    }

    // 游标条件（首次查询不传cursor）
    if param.Cursor > 0 {
        query = query.Where("id < ?", param.Cursor)
    }

    var flows []model.WalletFlow
    err := query.Order("id DESC").Limit(param.PageSize).Find(&flows).Error
    return flows, err
}
```

**B端管理后台**仍可使用传统分页（OFFSET），因为操作频率低且有权限管控。

## 十二、P3-敏感数据保护

### 12.1 问题描述

钱包涉及用户资产信息，部分字段需要加密存储：
- 银行卡号、USDT钱包地址（提现信息）
- 用户余额快照（对账文件）

### 12.2 解决方案：应用层AES加密

```
加密范围:
  - 提现订单中的 receiver_info（银行卡号/钱包地址）→ AES-256-GCM
  - 对账文件 → 加密后存储

不加密范围:
  - balance / amount 等数值字段 → 需要参与SQL计算
  - user_id 等关联字段 → 需要索引查询

实现方式:
  - 写入时: plaintext → AES加密 → 存入DB
  - 读取时: DB读出 → AES解密 → 返回
  - 密钥管理: 配置中心（ETCD）+ 环境变量，不硬编码
```

```go
// pkg/crypto/aes.go
func Encrypt(plaintext string, key []byte) (string, error) {
    block, _ := aes.NewCipher(key)
    gcm, _ := cipher.NewGCM(block)
    nonce := make([]byte, gcm.NonceSize())
    io.ReadFull(rand.Reader, nonce)
    ciphertext := gcm.Seal(nonce, nonce, []byte(plaintext), nil)
    return base64.StdEncoding.EncodeToString(ciphertext), nil
}

func Decrypt(ciphertextB64 string, key []byte) (string, error) {
    ciphertext, _ := base64.StdEncoding.DecodeString(ciphertextB64)
    block, _ := aes.NewCipher(key)
    gcm, _ := cipher.NewGCM(block)
    nonceSize := gcm.NonceSize()
    nonce, ciphertext := ciphertext[:nonceSize], ciphertext[nonceSize:]
    plaintext, err := gcm.Open(nil, nonce, ciphertext, nil)
    return string(plaintext), err
}
```

## 十四、P3-订单状态机

### 14.1 问题描述

**场景**：提现订单状态流转 `待审核→审核通过→支付处理中→成功/失败`。如果两个管理员同时审核同一笔，一个通过一个拒绝 → 状态混乱。

### 14.2 解决方案：条件UPDATE（状态机守卫）

```go
// 状态流转必须验证当前状态
result := tx.Model(&model.WithdrawOrder{}).
    Where("id = ? AND status = ?", orderID, StatusPending). // 状态守卫
    Update("status", StatusApproved)

if result.RowsAffected == 0 {
    // 当前状态不是Pending → 已经被别人处理了
    return errs.OrderAlreadyProcessed
}
```

### 14.3 状态机定义

```
充值订单(deposit_order):
  待处理(0) → 成功(1)
  待处理(0) → 失败(2)
  （注：充值由支付回调触发，状态简单）

提现订单(withdraw_order):
  待审核(0) → 审核通过(1) → 支付处理中(2) → 成功(3)
  待审核(0) → 审核拒绝(4)
  支付处理中(2) → 失败(5)
  失败(5) → 待审核(0)  [重新提交]

兑换订单(exchange_order):
  待处理(0) → 成功(1)
  待处理(0) → 失败(2)
  （注：兑换是实时操作，通常同步完成）

冻结记录(freeze_record):
  冻结中(1) → 已扣减(2)  [冻结→确认扣减]
  冻结中(1) → 已解冻(3)  [冻结→取消解冻]
```

### 14.4 合法状态转换表

```go
// internal/ser/order_state.go

var validTransitions = map[int32][]int32{
    // 提现订单
    WithdrawPending:    {WithdrawApproved, WithdrawRejected},
    WithdrawApproved:   {WithdrawProcessing},
    WithdrawProcessing: {WithdrawSuccess, WithdrawFailed},
    WithdrawFailed:     {WithdrawPending}, // 允许重新提交

    // 冻结记录
    FreezeActive:       {FreezeDeducted, FreezeUnfrozen},
}

func isValidTransition(from, to int32) bool {
    targets, ok := validTransitions[from]
    if !ok {
        return false
    }
    for _, t := range targets {
        if t == to {
            return true
        }
    }
    return false
}
```

## 十五、P3-多钱包扣款优先级

### 15.1 问题描述

**场景**：用户有中心钱包100元和奖励钱包50元，下注80元 → 先扣哪个？扣完一个不够怎么办？

### 15.2 解决方案：有序扣减 + 比例记录

```
扣款优先级（产品规则）:
  1. 奖励钱包优先（流水要求低）
  2. 中心钱包兜底（真金白银）

扣减逻辑:
  需扣80元:
    ① 奖励钱包余额50 → 扣50 → 不足
    ② 中心钱包余额100 → 扣30 → 足够
    ③ 实际: 奖励扣50 + 中心扣30 = 80 ✓

返奖时:
  按原始扣减比例返还（奖励:中心 = 50:30）
  → 需要在流水/订单中记录扣减比例
```

### 15.3 实现要点

```go
// 多钱包扣减的锁顺序
// 必须按 wallet_type 升序加锁（与第六章防死锁规则一致）
// 假设 wallet_type: 1=中心, 2=奖励, 3=主播, 4=代理, 5=场馆
// 扣减顺序可以和锁顺序不同，但加锁必须按固定顺序

func (l *Ledger) MultiDebit(ctx context.Context, tx *gorm.DB, param MultiDebitParam) error {
    // 1. 按wallet_type升序锁定所有参与的钱包
    walletTypes := []int8{WalletTypeCenter, WalletTypeReward}
    sort.Slice(walletTypes, func(i, j int) bool { return walletTypes[i] < walletTypes[j] })

    wallets := make(map[int8]*model.UserWallet)
    for _, wt := range walletTypes {
        // SELECT FOR UPDATE 加锁
        // ...
    }

    // 2. 按业务优先级扣减（奖励优先 → 中心兜底）
    remaining := param.TotalAmount
    deductions := make(map[int8]int64) // wallet_type → 扣减金额

    // 先扣奖励
    rewardWallet := wallets[WalletTypeReward]
    if rewardWallet.Balance > 0 {
        deduct := min(rewardWallet.Balance, remaining)
        deductions[WalletTypeReward] = deduct
        remaining -= deduct
    }

    // 不够则扣中心
    if remaining > 0 {
        centerWallet := wallets[WalletTypeCenter]
        if centerWallet.Balance < remaining {
            return errs.BalanceInsufficient
        }
        deductions[WalletTypeCenter] = remaining
    }

    // 3. 执行扣减 + 写入流水（每个钱包一条流水）
    for wt, amount := range deductions {
        // UPDATE balance, INSERT flow...
    }

    return nil
}
```

## 十六、P3-汇率精度与快照

### 16.1 问题描述

**场景**：用户发起兑换时看到的汇率是25,000，点击确认后实际用的汇率变成了25,100（后台刚好更新了）→ 用户投诉。

### 16.2 解决方案：汇率快照锁定

```
兑换流程:
  1. 用户查询汇率（API返回当前汇率 + 有效时间窗口）
  2. 用户确认兑换（请求中携带rate_snapshot_id）
  3. 服务端校验:
     - rate_snapshot_id是否有效
     - 是否在有效时间窗口内（如5分钟）
     - 如果汇率已变动超过阈值（如0.5%）→ 提示用户重新确认

快照存储:
  - 在rate_change_log中有完整变更历史
  - 兑换订单中记录 exchange_rate 字段（快照值）
  - 审计时可以追溯每笔兑换的实际汇率
```

### 16.3 汇率计算规范

```go
// 统一汇率计算函数
func CalcExchangeAmount(fromAmount int64, rateStr string) (int64, error) {
    // 必须用decimal计算，禁止float64
    amount := decimal.NewFromInt(fromAmount)
    rate := decimal.RequireFromString(rateStr)

    result := amount.Mul(rate)

    // 向下取整（保守策略，不多给用户）
    return result.IntPart(), nil
}

// ❌ 禁止写法
func badCalc(fromAmount int64, rate float64) int64 {
    return int64(float64(fromAmount) * rate)  // 精度损失！
}
```

### 17.1 需要监控的异常场景

| 场景 | 触发条件 | 监控方式 | 处理策略 |

## 十九、P3-数据库连接池与事务超时

### 19.1 问题描述

钱包服务大量使用事务+行锁，如果事务长时间不提交：
- 数据库连接被占用不释放 → 连接池耗尽 → 服务不可用
- 行锁长时间持有 → 其他请求全部排队 → 超时雪崩

### 19.2 解决方案

```go
// 事务超时设置
func newTransactionContext(ctx context.Context) (context.Context, context.CancelFunc) {
    return context.WithTimeout(ctx, 500 * time.Millisecond)
}

// 使用方式
func (s *WalletSer) Deposit(ctx context.Context, req *DepositReq) error {
    txCtx, cancel := newTransactionContext(ctx)
    defer cancel()

    return s.db.WithContext(txCtx).Transaction(func(tx *gorm.DB) error {
        // 整个事务必须在500ms内完成
        // 超时后ctx.Done() → GORM自动回滚
        // ...
    })
}
```

### 19.3 连接池配置建议

```go
// GORM连接池配置
sqlDB, _ := db.DB()
sqlDB.SetMaxOpenConns(100)      // 最大连接数
sqlDB.SetMaxIdleConns(20)       // 最大空闲连接数
sqlDB.SetConnMaxLifetime(time.Hour) // 连接最大存活时间
sqlDB.SetConnMaxIdleTime(10 * time.Minute) // 空闲连接最大存活时间
```

### 19.4 Ledger纪律总结

```
Ledger层是钱包的心脏，所有余额变动的唯一入口。纪律清单:

✅ MUST:
  - 所有参数在进入Ledger前已完成校验
  - 事务内只做DB操作（锁行→校验→改余额→写流水）
  - 事务超时500ms
  - 多钱包操作按固定顺序加锁
  - 流水是幂等检查点，先INSERT流水再UPDATE余额

❌ MUST NOT:
  - 事务内调用RPC
  - 事务内做耗时计算
  - 绕过Ledger直接修改user_wallet表
  - 使用float64做金额计算
  - 手动管理事务的Commit/Rollback（用GORM的Transaction回调）
```
