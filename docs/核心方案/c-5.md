# 二、并发安全 — 防超扣方案

## 2.1 问题本质

```
场景：用户A余额 1000，同时发起兑换(扣800)和提现(扣500)
两个请求并发到达同一行 user_balance 记录

错误做法（先查后扣，100%会超扣）：
  请求1：SELECT available → 1000 → 1000 >= 800 ✓ → UPDATE available = 200
  请求2：SELECT available → 1000 → 1000 >= 500 ✓ → UPDATE available = 500
  结果：两个都成功了！最终余额取决于谁后提交（200或500），但实际应该只有一个成功

  根因：SELECT 和 UPDATE 之间存在时间窗口，并发读取到了同一个快照值
```

## 2.2 方案：乐观锁（CAS 条件更新）

**核心思路：把"检查"和"扣减"合并为一条 SQL，让数据库引擎保证原子性。**

```sql
-- 扣减余额：一条SQL同时完成"余额够不够"和"扣减"
UPDATE user_balance
SET available = available - ?
WHERE user_id = ? AND currency = ? AND wallet_type = ?
  AND available >= ?    -- ← 这个 WHERE 条件就是"乐观锁"

-- 判断 affected rows：
--   rows = 1 → 扣减成功（余额充足，扣减已完成）
--   rows = 0 → 扣减失败（余额不足，什么都没改）
```

**并发场景验证：**

```
初始状态：available = 1000

请求1：UPDATE SET available = available - 800 WHERE available >= 800
请求2：UPDATE SET available = available - 500 WHERE available >= 500

TiDB/MySQL InnoDB 行级锁串行化执行：
  请求1先执行：1000 >= 800 ✓ → available = 200，rows = 1
  请求2后执行：200 >= 500 ✗ → 不更新，rows = 0

  或者反过来：
  请求2先执行：1000 >= 500 ✓ → available = 500，rows = 1
  请求1后执行：500 >= 800 ✗ → 不更新，rows = 0

  无论哪个先执行，都不会超扣。100% 安全。
```

## 2.3 为什么不选悲观锁（SELECT FOR UPDATE）

| 对比项 | 乐观锁（CAS） | 悲观锁（FOR UPDATE） |
|--------|-------------|-------------------|
| SQL 数量 | 1条（UPDATE WHERE） | 2条（SELECT FOR UPDATE + UPDATE） |
| 锁持有时间 | 一条SQL执行时间（微秒级） | 从SELECT到事务提交（毫秒级） |
| 死锁风险 | 无（不持锁） | 有（多行操作时可能死锁） |
| 并发度 | 极高 | 被锁行串行 |
| 代码复杂度 | 低（1条SQL + 判断rows） | 高（需锁管理+超时+重试） |
| 适用场景 | 简单加减运算 | 需要"先读再算再写"的复杂场景 |
| 工程一致性 | ser-item 范式一致 | 工程中无先例 |

**结论：钱包余额操作全部是简单加减，不需要"先读出来在应用层做复杂计算再写回"，乐观锁是最优选择。**

## 2.4 Go + GORM 伪代码

```go
// 扣减余额（兑换扣减/提现冻结/人工减款）
func deductBalance(tx *gorm.DB, userID int64, currency string,
    walletType int, amount int64) error {

    result := tx.Model(&model.UserBalance{}).
        Where("user_id = ? AND currency = ? AND wallet_type = ?",
            userID, currency, walletType).
        Where("available >= ?", amount).  // ← 乐观锁条件
        Update("available", gorm.Expr("available - ?", amount))

    if result.RowsAffected == 0 {
        return errs.ErrInsufficientBalance  // 余额不足
    }
    return result.Error
}

// 增加余额（充值入账/兑换增加/人工加款）
func creditBalance(tx *gorm.DB, userID int64, currency string,
    walletType int, amount int64) error {

    result := tx.Model(&model.UserBalance{}).
        Where("user_id = ? AND currency = ? AND wallet_type = ?",
            userID, currency, walletType).
        Update("available", gorm.Expr("available + ?", amount))

    // 增加操作不需要 available >= 条件，但要检查行是否存在
    if result.RowsAffected == 0 {
        // 行不存在 → 懒创建（见第九章）
        return lazyCreateAndCredit(tx, userID, currency, walletType, amount)
    }
    return result.Error
}

// 冻结余额（提现专用：available减 + frozen加，同一条SQL原子完成）
func freezeBalance(tx *gorm.DB, userID int64, currency string,
    walletType int, amount int64) error {

    result := tx.Model(&model.UserBalance{}).
        Where("user_id = ? AND currency = ? AND wallet_type = ?",
            userID, currency, walletType).
        Where("available >= ?", amount).  // ← 乐观锁
        Updates(map[string]interface{}{
            "available": gorm.Expr("available - ?", amount),
            "frozen":    gorm.Expr("frozen + ?", amount),
        })

    if result.RowsAffected == 0 {
        return errs.ErrInsufficientBalance
    }
    return result.Error
}
```

## 2.5 必须遵守的铁律

```
铁律一：所有扣减/冻结操作必须走 WHERE available >= amount
  → 绝不允许先 SELECT 余额在 Go 代码里判断再 UPDATE
  → 并发场景下 SELECT 和 UPDATE 之间的时间窗口 = 超扣风险

铁律二：通过 RowsAffected == 0 判断失败
  → 不通过 err != nil 判断（WHERE 不满足不是 error，是 rows=0）
  → 0 行影响 = 余额不足，必须回滚事务

铁律三：增加操作无需 WHERE available >= 条件
  → 充值入账、人工加款只做 available + amount
  → 但必须处理行不存在的情况（懒创建，见第九章）
```

## 2.6 覆盖的接口（13个）

```
需要乐观锁（扣减/冻结）的接口：
  ① CreateExchangeOrder     — VND available - sourceAmount
  ② CreateWithdrawOrder     — available - amount, frozen + amount
  ③ ManualDebit (RPC)       — available - amount
  ④ WithdrawConfirm (RPC)   — frozen - amount

需要增加余额（无乐观锁但需懒创建）的接口：
  ⑤ DepositCredit (RPC)     — available + amount
  ⑥ WithdrawUnfreeze (RPC)  — available + amount, frozen - amount
  ⑦ ManualCredit (RPC)      — available + amount
  ⑧ ManualPatch (RPC)       — available + amount
  ⑨ CreateExchangeOrder     — BSB available + targetAmount
  ⑩ CreateExchangeOrder     — BSB奖励 available + giftAmount

包含在事务内但间接涉及的：
  ⑪⑫⑬ 各订单创建（订单表写入在同一事务中）
```

# 三、幂等性保障 — 双层防护精确实现

## 3.1 问题本质

```
场景：财务调用 DepositCredit(充值入账) → 网络超时 → 财务重试
  第一次：钱包收到 → 余额 +100 → 写流水 → 返回成功 → 网络丢包
  第二次：钱包再次收到 → 如果不做幂等 → 余额再 +100 → 资损

同一笔充值入账两次 = 用户凭空多了 100 = 直接资损
同一笔提现确认两次 = 用户余额被多扣 100 = 直接资损
```

## 3.2 方案：双层防护

```
┌─────────────────────────────────────────────────────────────┐
│                      第一层：应用层快速返回                     │
│                                                              │
│  进入方法 → 用 refOrderNo 查 订单/流水 状态                    │
│  ├─ 已处理完成 → 直接返回 true（不做任何操作，毫秒级返回）       │
│  └─ 未处理/处理中 → 继续执行业务逻辑                           │
│                                                              │
│  目的：正常重试场景快速返回，避免走到事务层                      │
│  覆盖：99.9% 的重试场景                                       │
├─────────────────────────────────────────────────────────────┤
│                      第二层：DB 唯一键兜底                      │
│                                                              │
│  balance_flow 表：UNIQUE KEY (ref_order_no, change_type)      │
│  ├─ 正常写入 → 成功                                           │
│  └─ 唯一键冲突（Duplicate Key）→ 捕获异常 → 返回 true          │
│                                                              │
│  目的：极端并发场景（两个请求同时穿透第一层）的最终兜底           │
│  覆盖：0.1% 的极端并发场景                                    │
└─────────────────────────────────────────────────────────────┘
```

## 3.3 Go 伪代码（以 DepositCredit 为例）

```go
func (s *RpcService) DepositCredit(ctx context.Context,
    req *wallet_rpc.DepositCreditReq) (bool, error) {

    // ===== 第一层：应用层幂等校验 =====
    order, err := s.depositRepo.GetByOrderNo(ctx, req.RefOrderNo)
    if err != nil {
        return false, errs.WrapDBError(err)
    }
    if order != nil && order.Status == enum.DepositStatusSuccess {
        // 已入账，直接返回成功（幂等）
        return true, nil
    }
    if order == nil {
        // 订单不存在（不是我们创建的订单号），拒绝
        return false, errs.ErrOrderNotFound
    }
    // order.Status == pending → 继续处理

    // ===== 参数校验 =====
    if req.UserId <= 0 || req.Amount <= 0 || req.Currency == "" {
        return false, errs.ErrInvalidParam
    }

    // ===== 事务操作 =====
    err = s.db.Transaction(func(tx *gorm.DB) error {

        // 1. 更新订单状态 → success
        result := tx.Model(&model.DepositOrder{}).
            Where("order_no = ? AND status = ?", req.RefOrderNo, enum.DepositStatusPending).
            Update("status", enum.DepositStatusSuccess)
        if result.RowsAffected == 0 {
            // 并发：另一个请求已处理，跳过（双重检查）
            return nil
        }

        // 2. 增加余额
        if err := creditBalance(tx, req.UserId, req.Currency,
            int(req.WalletType), req.Amount); err != nil {
            return err
        }

        // 3. 写流水（第二层幂等兜底）
        flow := &model.BalanceFlow{
            UserId:        req.UserId,
            Currency:      req.Currency,
            WalletType:    int8(req.WalletType),
            ChangeType:    enum.ChangeTypeDeposit,
            Amount:        req.Amount,
            BeforeBalance: beforeBal,  // 事务内查获
            AfterBalance:  beforeBal + req.Amount,
            RefOrderNo:    req.RefOrderNo,
        }
        if err := tx.Create(flow).Error; err != nil {
            if isDuplicateKeyError(err) {
                // ===== 第二层：唯一键冲突 → 幂等兜底 =====
                // 说明另一个并发请求已经写了流水，直接返回成功
                return nil  // 不返回error，事务会提交（但前面的操作也会被跳过）
            }
            return err
        }

        return nil
    })

    if err != nil {
        return false, err
    }
    return true, nil
}

// 唯一键冲突判断
func isDuplicateKeyError(err error) bool {
    // MySQL/TiDB error code 1062 = Duplicate entry
    var mysqlErr *mysql.MySQLError
    if errors.As(err, &mysqlErr) {
        return mysqlErr.Number == 1062
    }
    return false
}
```

## 3.4 幂等键设计

```
balance_flow 表：UNIQUE KEY (ref_order_no, change_type)

为什么是"订单号 + 变动类型"双字段组合键，而不是只有订单号？

原因：同一个订单号可能产生多条流水
  例如：兑换订单 EX001 产生 3 条流水
    ① ref_order_no=EX001, change_type=EXCHANGE_DEDUCT   (VND扣减)
    ② ref_order_no=EX001, change_type=EXCHANGE_CREDIT   (BSB增加)
    ③ ref_order_no=EX001, change_type=EXCHANGE_GIFT     (BSB赠送)

  如果只用 ref_order_no 做唯一键，第②③条会冲突
  用 (ref_order_no, change_type) 组合，同一订单不同类型的流水可以共存
  但同一订单同一类型不能写两次 → 幂等保证
```

## 3.5 8 个 RPC 写接口的幂等键映射

| RPC 接口 | 幂等键字段 | change_type 枚举 | 幂等目标 |
|----------|----------|-----------------|---------|
| DepositCredit | refOrderNo (充值订单号) | DEPOSIT_CREDIT | 不重复入账 |
| WithdrawFreeze | refOrderNo (提现订单号) | WITHDRAW_FREEZE | 不重复冻结 |
| WithdrawConfirm | refOrderNo (提现订单号) | WITHDRAW_CONFIRM | 不重复确认扣款 |
| WithdrawUnfreeze | refOrderNo (提现订单号) | WITHDRAW_UNFREEZE | 不重复解冻 |
| ManualCredit | refAdjustNo (修正单号) | MANUAL_CREDIT | 不重复加款 |
| ManualDebit | refAdjustNo (修正单号) | MANUAL_DEBIT | 不重复减款 |
| ManualPatch | refAdjustNo (修正单号) | MANUAL_PATCH | 不重复补单 |
| QueryBalanceRpc | — | — | 纯读，无需幂等 |

## 3.6 边界条件处理

```
边界一：第一层和第二层之间的并发窗口
  请求A：第一层查到 status=pending → 继续
  请求B：第一层查到 status=pending → 继续
  请求A：事务内更新 status=success → 写流水成功
  请求B：事务内更新 status=success → RowsAffected=0（已被A改了）→ 跳过
  → 安全：第二个请求被订单状态的乐观更新挡住

边界二：第一层被穿透 + 订单状态更新也被穿透
  极端情况：两个请求完全同时执行，都通过了第一层和订单状态更新
  → 第二层兜底：写流水时唯一键冲突，第二个写入失败
  → 需要：捕获 DuplicateKey 异常后整个事务回滚（余额+100 被回滚）

  ★ 注意：这里 DuplicateKey 后应该让事务回滚（return err），
    然后在事务外再做一次幂等检查返回 true。
    而不是在事务内吞掉异常让事务提交（那样余额加了但流水没写）。

修正后的精确处理：
  事务内 DuplicateKey → return err → 事务回滚
  事务外捕获 DuplicateKey → 再查一次订单状态 → 已成功 → 返回 true
```

# 四、事务一致性 — 边界铁律与统一余额变动方法

## 4.1 问题本质

```
场景：兑换操作事务内有 5 个写操作
  ① VND available - 230000
  ② BSB available + 1000
  ③ BSB 奖励 available + 50
  ④ INSERT exchange_order
  ⑤ INSERT balance_flow x3

如果步骤①成功、步骤⑤失败：
  → 余额扣了但没有流水记录 → 对账永远差 → 无法追溯

如果步骤①②③成功、步骤④失败：
  → 余额变了但没有订单记录 → 无从查证这笔变动来源

必须保证：5 个操作全部成功 或 全部回滚，不允许中间状态。
```

## 4.2 三条铁律

### 铁律一：余额 + 订单 + 流水 = 同一事务

```
绝对不允许的代码模式：

  // ✗ 错误：余额和流水不在同一事务
  updateBalance(db, ...)       // 自动提交
  insertFlow(db, ...)          // 又一次自动提交
  // 如果第二个失败，余额变了但流水没记录

正确模式：

  db.Transaction(func(tx *gorm.DB) error {
      updateBalance(tx, ...)   // 事务内
      insertOrder(tx, ...)     // 事务内
      insertFlow(tx, ...)      // 事务内
      return nil               // 全部成功才提交
  })
```

### 铁律二：扣减操作必须用乐观锁

```
绝对不允许的代码模式：

  // ✗ 错误：先查后扣
  balance := getBalance(tx, userId, currency)
  if balance.Available >= amount {
      tx.Update("available", balance.Available - amount)
  }
  // 并发场景：两个goroutine同时查到相同的 balance.Available → 都判断够 → 超扣

正确模式（见第二章）：

  tx.Where("available >= ?", amount).
     Update("available", gorm.Expr("available - ?", amount))
  // 数据库引擎保证原子性
```

### 铁律三：RPC 在事务之外

```
绝对不允许的代码模式：

  // ✗ 错误：RPC 在事务内
  db.Transaction(func(tx *gorm.DB) error {
      rate := rpc.GetExchangeRate(ctx)    // RPC 可能超时 3-10 秒
      updateBalance(tx, ...)               // 事务持有了 3-10 秒
      return nil
  })
  // 风险：
  //   RPC 超时 → 事务持有 10 秒 → 数据库连接被占 10 秒
  //   并发 100 个请求 → 100 个连接被占 → 连接池耗尽 → 全服务不可用

正确模式：

  // RPC 在事务外
  rate := rpc.GetExchangeRate(ctx)         // 事务外，超时不影响DB连接
  kycStatus := rpc.KycQuery(ctx)           // 事务外

  // 数据准备完毕后再开事务（事务持有时间 = 纯DB操作时间 = 毫秒级）
  db.Transaction(func(tx *gorm.DB) error {
      updateBalance(tx, ...)   // 毫秒
      insertOrder(tx, ...)     // 毫秒
      insertFlow(tx, ...)      // 毫秒
      return nil
  })
```

## 4.3 统一余额变动方法 — 核心抽象

**所有余额操作（充值/兑换/提现/冻结/解冻/人工）最终都归结为一个原子操作：
"修改某行余额 + 记录一条流水"。**

```go
// BalanceChangeParam — 单次余额变动参数
type BalanceChangeParam struct {
    UserID     int64
    Currency   string
    WalletType int
    ChangeType int               // 枚举：充值/兑换扣减/冻结/解冻/...
    Amount     int64             // 正数=增加，负数=扣减（或用direction区分）
    RefOrderNo string            // 幂等键：关联订单号
    Remark     string            // 备注（可选）
}

// ExecuteBalanceChange — 统一余额变动（必须在事务内调用）
func ExecuteBalanceChange(tx *gorm.DB, param BalanceChangeParam) error {

    // Step 1：查变动前余额（用于写流水的 before_balance）
    var balance model.UserBalance
    err := tx.Where("user_id = ? AND currency = ? AND wallet_type = ?",
        param.UserID, param.Currency, param.WalletType).
        First(&balance).Error

    if err == gorm.ErrRecordNotFound {
        // 懒创建（见第九章）
        balance, err = lazyCreate(tx, param.UserID, param.Currency, param.WalletType)
    }
    if err != nil {
        return err
    }

    beforeAvailable := balance.Available

    // Step 2：更新余额（根据类型决定操作方式）
    switch {
    case isDeductType(param.ChangeType):
        // 扣减类：乐观锁
        result := tx.Model(&model.UserBalance{}).
            Where("user_id = ? AND currency = ? AND wallet_type = ?",
                param.UserID, param.Currency, param.WalletType).
            Where("available >= ?", param.Amount).
            Update("available", gorm.Expr("available - ?", param.Amount))
        if result.RowsAffected == 0 {
            return errs.ErrInsufficientBalance
        }

    case isCreditType(param.ChangeType):
        // 增加类：直接加
        tx.Model(&model.UserBalance{}).
            Where("user_id = ? AND currency = ? AND wallet_type = ?",
                param.UserID, param.Currency, param.WalletType).
            Update("available", gorm.Expr("available + ?", param.Amount))

    case isFreezeType(param.ChangeType):
        // 冻结类：available减 + frozen加
        result := tx.Model(&model.UserBalance{}).
            Where("user_id = ? AND currency = ? AND wallet_type = ?",
                param.UserID, param.Currency, param.WalletType).
            Where("available >= ?", param.Amount).
            Updates(map[string]interface{}{
                "available": gorm.Expr("available - ?", param.Amount),
                "frozen":    gorm.Expr("frozen + ?", param.Amount),
            })
        if result.RowsAffected == 0 {
            return errs.ErrInsufficientBalance
        }

    case isUnfreezeType(param.ChangeType):
        // 解冻类：available加 + frozen减
        tx.Model(&model.UserBalance{}).
            Where("user_id = ? AND currency = ? AND wallet_type = ?",
                param.UserID, param.Currency, param.WalletType).
            Updates(map[string]interface{}{
                "available": gorm.Expr("available + ?", param.Amount),
                "frozen":    gorm.Expr("frozen - ?", param.Amount),
            })

    case isConfirmType(param.ChangeType):
        // 确认扣款类（提现确认）：仅 frozen减
        tx.Model(&model.UserBalance{}).
            Where("user_id = ? AND currency = ? AND wallet_type = ?",
                param.UserID, param.Currency, param.WalletType).
            Update("frozen", gorm.Expr("frozen - ?", param.Amount))
    }

    // Step 3：计算变动后余额
    afterAvailable := calculateAfterBalance(beforeAvailable, param.ChangeType, param.Amount)

    // Step 4：写流水（幂等兜底靠唯一键）
    flow := &model.BalanceFlow{
        UserID:        param.UserID,
        Currency:      param.Currency,
        WalletType:    int8(param.WalletType),
        ChangeType:    int8(param.ChangeType),
        Amount:        signedAmount(param.ChangeType, param.Amount),
        BeforeBalance: beforeAvailable,
        AfterBalance:  afterAvailable,
        RefOrderNo:    param.RefOrderNo,
        Remark:        param.Remark,
    }
    if err := tx.Create(flow).Error; err != nil {
        return err  // 唯一键冲突也返回error，让事务回滚
    }

    return nil
}
```

# 五、分布式一致性 — RPC 失败补偿方案

## 5.1 问题本质

```
全模块只有一个场景需要补偿 — CreateWithdrawOrder：

  步骤1（事务内）：冻结余额成功 → available -500, frozen +500
  步骤2（事务外）：调 finance.SubmitWithdrawReview(RPC)
    ├─ 成功 → 等待审核结果回调 → 正常流程
    └─ 失败 → ★ 用户的 500 被冻住了！但审核单没创建 → 资金卡死

为什么其他接口不需要补偿：
  - 充值创建：先创建订单后调 finance.CreateChannelOrder
    → 失败时订单标记 failed 即可，余额从未变动
  - 兑换：完全闭环无 RPC
  - RPC 被调接口：余额操作是最终步骤，不存在"做完余额后还需要调别人"
```

## 5.2 补偿方案详细设计

```
CreateWithdrawOrder 的精确流程：

  ┌─────────────────────────────────────────────────────┐
  │  事务外：并行校验（限额 + KYC + 余额预检）              │
  └─────────────────────┬───────────────────────────────┘
                        │ 全部通过
                        ▼
  ┌─────────────────────────────────────────────────────┐
  │  事务内（毫秒级，纯DB操作）：                            │
  │    ① 冻结余额：available -= X, frozen += X（乐观锁）   │
  │    ② 写提现订单：status = reviewing                    │
  │    ③ 写冻结流水                                        │
  └─────────────────────┬───────────────────────────────┘
                        │ 事务提交成功
                        │ ★ 到此刻：用户的钱已被冻结 ★
                        ▼
  ┌─────────────────────────────────────────────────────┐
  │  事务外：调 RPC                                        │
  │    resp, err := finance.SubmitWithdrawReview(req)      │
  │                                                        │
  │    ├─ err == nil（成功）                                │
  │    │   → 存 resp.ReviewNo 到订单                       │
  │    │   → 返回"提现已提交，等待审核"                      │
  │    │                                                   │
  │    └─ err != nil（失败）★★★ 进入补偿流程 ★★★           │
  │        │                                               │
  │        ▼                                               │
  │   ┌─── 补偿事务 ──────────────────────────────────┐   │
  │   │  ① 解冻余额：available += X, frozen -= X        │   │
  │   │  ② 订单状态 → failed                            │   │
  │   │  ③ 写回退流水（change_type = WITHDRAW_COMPENSATE）│   │
  │   └───────────────────────┬─────────────────────────┘   │
  │                           │                             │
  │     ├─ 补偿成功 → 返回"提现失败，余额已退回"             │
  │     └─ 补偿失败 → ★ 极端情况 ★                         │
  │         → 记 ERROR 日志（包含 userId/orderNo/amount）    │
  │         → 触发告警（通知运维/DBA）                        │
  │         → 返回"系统异常，请联系客服"                      │
  │         → 人工介入修复                                   │
  └─────────────────────────────────────────────────────┘
```

```go
func (s *WithdrawService) CreateWithdrawOrder(ctx context.Context,
    req *withdraw.CreateWithdrawOrderReq) (*withdraw.CreateWithdrawOrderResp, error) {

    // ... 并行校验（省略）...

    orderNo := utils.GenerateSnowflakeID()

    // ===== 冻结事务 =====
    err := s.db.Transaction(func(tx *gorm.DB) error {
        // 冻结余额
        if err := freezeBalance(tx, userId, req.Currency, 1, amount); err != nil {
            return err
        }
        // 创建订单
        order := &model.WithdrawOrder{
            OrderNo: orderNo, Status: enum.WithdrawStatusReviewing, ...
        }
        if err := tx.Create(order).Error; err != nil {
            return err
        }
        // 写冻结流水
        if err := insertFreezeFlow(tx, userId, req.Currency, amount, orderNo); err != nil {
            return err
        }
        return nil
    })
    if err != nil {
        return nil, err  // 冻结失败，事务已回滚，无需补偿
    }

    // ===== 调 RPC（事务外）=====
    resp, rpcErr := rpc.FinanceClient().SubmitWithdrawReview(ctx, &SubmitReq{
        UserId: userId, Currency: req.Currency, Amount: amount,
        Fee: fee, OrderNo: orderNo, AccountInfo: accountInfo,
    })

    if rpcErr != nil {
        // ===== 补偿流程 =====
        compensateErr := s.compensateWithdrawFreeze(ctx, userId, req.Currency, amount, orderNo)
        if compensateErr != nil {
            // ★ 极端情况：补偿也失败
            logger.Error("CRITICAL: withdraw compensate failed",
                "userId", userId, "orderNo", orderNo, "amount", amount,
                "rpcErr", rpcErr, "compensateErr", compensateErr)
            // 触发告警
            alerter.SendCritical("提现补偿失败", userId, orderNo, amount)
            return nil, errs.ErrSystemError
        }
        return nil, errs.WrapBizError(rpcErr, "提现失败，余额已退回")
    }

    // RPC 成功，存审核单号
    s.withdrawRepo.UpdateReviewNo(ctx, orderNo, resp.ReviewNo)

    return &withdraw.CreateWithdrawOrderResp{OrderNo: orderNo}, nil
}

// 补偿方法
func (s *WithdrawService) compensateWithdrawFreeze(ctx context.Context,
    userId int64, currency string, amount int64, orderNo string) error {

    return s.db.Transaction(func(tx *gorm.DB) error {
        // 解冻余额
        tx.Model(&model.UserBalance{}).
            Where("user_id = ? AND currency = ? AND wallet_type = 1", userId, currency).
            Updates(map[string]interface{}{
                "available": gorm.Expr("available + ?", amount),
                "frozen":    gorm.Expr("frozen - ?", amount),
            })

        // 订单标记失败
        tx.Model(&model.WithdrawOrder{}).
            Where("order_no = ?", orderNo).
            Update("status", enum.WithdrawStatusFailed)

        // 写补偿流水
        tx.Create(&model.BalanceFlow{
            UserID: userId, Currency: currency, WalletType: 1,
            ChangeType: int8(enum.WithdrawCompensate),
            Amount: amount,  // 正数=退回
            RefOrderNo: orderNo,
        })

        return nil
    })
}
```

## 5.4 为什么不用分布式事务框架（Seata/DTM）

| 对比项 | 本地事务+补偿（当前方案） | Seata/DTM |
|--------|-------------------|-----------|
| 需要补偿的场景 | **仅 1 个**（提现冻结→RPC失败） | 适合多个场景 |
| 引入成本 | **零**（无新组件） | 需部署TC Server，引入SDK |
| 运维复杂度 | **零**（无额外运维） | 需维护 TC 高可用 |
| 代码复杂度 | 一个补偿方法 | 注册分支事务+回滚接口 |
| 工程一致性 | 与现有22个服务一致 | 新引入，无先例 |
| 性能影响 | **零**（补偿只在失败时执行） | 全局事务有额外协调开销 |

**结论：就一个需要补偿的场景，用不着上分布式事务框架。本地补偿够用。**

# 六、金额精度 — BIGINT 全链路零浮点方案

## 6.1 问题本质

```
浮点数的经典陷阱（Go 语言验证）：
  var a float64 = 0.1
  var b float64 = 0.2
  fmt.Println(a + b)           // 0.30000000000000004
  fmt.Println(a + b == 0.3)   // false

如果用 float64 存余额：
  余额 = 0.1 + 0.2 = 0.30000000000000004
  和预期的 0.3 不等 → 对账差异
  1万次操作后累积误差可能到分/角级别 → 资损
```

## 6.2 方案：BIGINT 最小单位整数存储

```
核心规则：所有金额在全链路中以"最小单位整数"表示

  "10.50 BSB"（精度2位）  → 数据库存 1050（int64）
  "100000 VND"（精度0位） → 数据库存 100000（int64）
  "50.00 USDT"（精度2位） → 数据库存 5000（int64）

全链路统一：
  ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
  │ 前端展示  │    │ Thrift   │    │ Go 代码  │    │  TiDB   │
  │  10.50   │ →  │ i64=1050│ →  │ int64   │ →  │ BIGINT  │
  │  BSB     │    │         │    │ =1050   │    │ =1050   │
  └─────────┘    └─────────┘    └─────────┘    └─────────┘

  展示时换算：amount / 10^precision = 展示金额
  1050 / 10^2 = 10.50 BSB
```

# 七、整数除法与汇率计算 — 取整策略与精度损失控制

## 7.1 问题本质

```
兑换场景：用户用 230000 VND 兑换 BSB，汇率 1 BSB = 23000 VND

如果 BSB precision = 2（最小单位 = 0.01 BSB = 1）：
  targetAmount = sourceAmount * 10^bsbPrecision / rate
  targetAmount = 230000 * 100 / 23000 = 1000（= 10.00 BSB）✓ 整除

如果 BSB precision = 2，用户充值 235000 VND：
  targetAmount = 235000 * 100 / 23000 = 1021.739...
  整数除法截断：1021（= 10.21 BSB）
  丢失：0.739... 个最小单位 ≈ 0.00739 BSB

问题：整数除法必然存在截断，谁来承受这个损失？
```

## 7.2 方案：向下取整（Truncate），平台侧承受微损失

```
策略选择：

  向上取整（Ceil）：用户多拿，平台损失 → ✗ 不可接受
  向下取整（Truncate/Floor）：平台多留，用户略少 → ✓ 行业标准
  四舍五入（Round）：双向偏差，审计困难 → ✗ 不推荐

结论：统一使用向下取整（Go 整数除法默认行为就是 Truncate）
```

## 7.3 汇率计算的精确公式

```go
// 兑换金额计算（全整数运算）
func CalculateExchangeAmount(sourceAmount int64, rate int64,
    ratePrecision int, targetPrecision int) (targetAmount int64) {

    // rate 存储也是整数：
    // "1 BSB = 23000 VND" 如果 ratePrecision=0，rate=23000
    // "1 BSB = 23000.50 VND" 如果 ratePrecision=2，rate=2300050

    // 公式：targetAmount = sourceAmount * 10^targetPrecision / rate * 10^ratePrecision
    // 分子分母同时扩大以减少精度损失：

    numerator := sourceAmount * pow10(targetPrecision) * pow10(ratePrecision)
    denominator := rate

    targetAmount = numerator / denominator  // Go 整数除法 = 自动向下取整

    return targetAmount
}

// 赠送金额计算
func CalculateGiftAmount(targetAmount int64, giftRate int64) int64 {
    // giftRate 以万分比表示：500 = 5%
    return targetAmount * giftRate / 10000  // 向下取整
}

// 手续费计算
func CalculateWithdrawFee(amount int64, feeType int, feeValue int64) int64 {
    switch feeType {
    case enum.FeeTypeFixed:
        return feeValue  // 固定金额
    case enum.FeeTypePercent:
        // feeValue 以万分比表示：200 = 2%
        return amount * feeValue / 10000  // 向下取整
    }
    return 0
}
```

## 7.4 验证方法

```
每笔兑换后验证等式：
  sourceAmount（VND扣减）= targetAmount * rate / 10^(targetPrecision + ratePrecision) + 截断损失

  截断损失 = sourceAmount - targetAmount * rate / 10^(...)

  要求：0 <= 截断损失 < 1 个最小单位
  如果截断损失 < 0 → Bug（用户多拿了）→ 必须修复
  如果截断损失 >= 1 个最小单位 → Bug（计算公式有误）→ 必须修复
```

## 7.5 溢出风险检查

```
int64 最大值：9,223,372,036,854,775,807（约 9.2 × 10^18）

最大金额场景：
  用户充值 100,000,000 VND（1亿越南盾，约 4000 美元，合理上限）
  存储值：100000000（1亿，远小于 int64 上限）

最大计算场景：
  numerator = sourceAmount * 10^targetPrecision * 10^ratePrecision
  = 100000000 * 100 * 1
  = 10,000,000,000（100亿，远小于 int64 上限）

结论：当前业务规模下，int64 不会溢出。但代码中仍应加防御性检查：
  if sourceAmount > math.MaxInt64 / (pow10(targetPrecision) * pow10(ratePrecision)) {
      return 0, errs.ErrAmountOverflow
  }
```

# 八、缓存一致性 — Cache-Aside 精确实现

## 8.1 策略划分

```
需要缓存的数据：
  ● 币种配置列表（currency_config）— 读多写少，B端修改频率低
  ● 汇率配置（exchange_rate）— 读多写少，但比币种改得频繁

不缓存的数据（铁律）：
  ● user_balance 余额 — 金融数据必须实时准确，直接查DB
  ● balance_flow 流水 — 写多读少，无缓存价值
  ● 订单表 — 状态频繁变更，缓存价值低
```

## 8.2 Cache-Aside 模式实现

```
读流程：
  ① 查 Redis → 命中 → 反序列化返回（毫秒级）
  ② Redis 未命中 → 查 DB → 序列化写入 Redis（设 TTL）→ 返回

写流程（B端修改币种/汇率时）：
  ① 写 DB（事务内）
  ② 删 Redis 缓存（事务提交后）
  ③ 下次查询自动重建缓存（Cache-Aside 的 lazy rebuild）
```

## 8.3 Go 伪代码

```go
// currency_cache.go

const (
    CacheCurrencyList = "wallet:currency:list"       // 全部启用币种
    CacheCurrencyInfo = "wallet:currency:info:%s"     // 单币种详情
    CacheExchangeRate = "wallet:rate:%s"              // 币种汇率

    TTLCurrencyList = 5 * time.Minute
    TTLExchangeRate = 3 * time.Minute
)

// GetAvailableCurrencyList — 带缓存的币种列表查询
func GetAvailableCurrencyList(ctx context.Context) ([]*model.CurrencyConfig, error) {

    // Step 1：查 Redis
    cacheKey := CacheCurrencyList
    data, err := rds.Get(ctx, cacheKey)
    if err == nil && data != "" {
        var list []*model.CurrencyConfig
        if json.Unmarshal([]byte(data), &list) == nil {
            return list, nil  // 缓存命中
        }
    }

    // Step 2：缓存未命中，查 DB
    list, err := currencyRepo.GetEnabledList(ctx)
    if err != nil {
        return nil, err
    }

    // Step 3：写入 Redis（异步写入不阻塞主流程）
    go func() {
        bytes, _ := json.Marshal(list)
        rds.Set(context.Background(), cacheKey, string(bytes), TTLCurrencyList)
    }()

    return list, nil
}

// InvalidateCurrencyCache — B端修改后删除缓存
func InvalidateCurrencyCache(ctx context.Context, currencyCode string) {
    rds.Del(ctx, CacheCurrencyList)
    rds.Del(ctx, fmt.Sprintf(CacheCurrencyInfo, currencyCode))
}

// InvalidateRateCache — 汇率修改后删除缓存
func InvalidateRateCache(ctx context.Context, currencyCode string) {
    rds.Del(ctx, fmt.Sprintf(CacheExchangeRate, currencyCode))
}
```

## 8.4 一致性分析

```
已知的不一致窗口（可接受）：

  时刻T1：B端修改汇率 23000 → 24000（写DB）
  时刻T2：删 Redis 缓存
  时刻T3：C端查汇率 → Redis 未命中 → 查DB拿到 24000 → 重建缓存

  窗口：T1 到 T2 之间（毫秒级），如果有C端查询走了旧缓存
  影响：用户看到旧汇率做兑换预览，但提交兑换时会重新获取最新汇率
  结论：无资损风险，仅影响预览展示，可接受

极端场景（删缓存失败）：
  TTL 兜底：最多 5 分钟后缓存自动过期
  TTL 是最终一致性的安全网，不依赖删除操作的 100% 成功

为什么不用 "先删缓存再写DB" ？
  如果先删缓存，在写DB之前有请求来了，会把旧数据重新写入缓存
  "先写DB再删缓存" 是 Cache-Aside 的标准模式
```

# 九、懒创建 — 余额行不存在时的原子处理

## 9.1 问题本质

```
user_balance 表设计：每个用户每个币种每个子钱包一行（三元组唯一键）
最多 5 币种 × 4 子钱包 = 20 行/用户

何时创建这些行？两个选择：
  方案A：注册时批量创建（20行 × 每个新用户）
    缺点：大部分用户可能只用 1 个币种，19 行浪费
    缺点：新增币种后，所有老用户需要补建行（数据迁移）

  方案B：首次使用时懒创建（用到才建）
    优点：按需创建，无浪费
    优点：新增币种自动适配（首次操作该币种时自动建行）
    缺点：写操作中需要处理"行不存在"的情况

结论：选方案B（懒创建），这也是钱包系统的行业标准做法。
```

## 9.2 懒创建触发场景

```
触发懒创建的时机：
  ① 充值入账（DepositCredit）— 用户首次充值某币种
  ② 兑换（CreateExchangeOrder）— BSB行可能不存在（首次兑换）
  ③ 人工加款（ManualCredit）— 给某个子钱包加款
  ④ 人工补单（ManualPatch）— 补单到某个子钱包

不触发懒创建的时机：
  ✗ 余额查询 — 行不存在 = 余额为0，直接返回0，不创建
  ✗ 扣减操作 — 行不存在 = 余额为0 = 余额不足，直接报错
```

## 9.3 方案：INSERT ON DUPLICATE KEY UPDATE

```sql
-- MySQL/TiDB 标准方案：UPSERT
INSERT INTO user_balance (user_id, currency, wallet_type, available, frozen)
VALUES (?, ?, ?, ?, 0)
ON DUPLICATE KEY UPDATE available = available + VALUES(available)

-- 行不存在 → INSERT 新行，available = 传入值
-- 行已存在 → UPDATE，available += 传入值
-- 利用三元组唯一键判断行是否存在
-- 原子操作，无并发问题
```

## 9.4 Go + GORM 实现

```go
// lazyCreateAndCredit — 懒创建+增加余额（原子操作）
func lazyCreateAndCredit(tx *gorm.DB, userID int64, currency string,
    walletType int, amount int64) error {

    // 使用原生 SQL（GORM 的 Clauses 方式）
    balance := &model.UserBalance{
        UserID:     userID,
        Currency:   currency,
        WalletType: int8(walletType),
        Available:  amount,
        Frozen:     0,
    }

    result := tx.Clauses(clause.OnConflict{
        Columns:   []clause.Column{{Name: "user_id"}, {Name: "currency"}, {Name: "wallet_type"}},
        DoUpdates: clause.Assignments(map[string]interface{}{
            "available": gorm.Expr("available + ?", amount),
        }),
    }).Create(balance)

    return result.Error
}
```

## 9.5 与乐观锁的协同

```
场景：首次兑换，BSB 余额行不存在

  步骤①：扣 VND（行存在，乐观锁正常工作）
    UPDATE user_balance SET available = available - 230000
    WHERE user_id=? AND currency='VND' AND wallet_type=1 AND available >= 230000
    → rows = 1 ✓

  步骤②：加 BSB 中心（行可能不存在）
    先尝试 UPDATE ... available = available + 1000
    → rows = 0（行不存在）
    → 走懒创建：INSERT ... ON DUPLICATE KEY UPDATE available = available + 1000
    → 成功

  步骤③：加 BSB 奖励（行可能不存在）
    同步骤②

★ 在 ExecuteBalanceChange 中的处理：
  增加类操作先 UPDATE，如果 rows=0，则走 lazyCreateAndCredit
  这个判断在统一余额变动方法内完成，上层业务无需感知
```

# 十、流水连续性与对账 — 数据完整性保障

## 10.1 问题本质

```
钱包系统最怕的不是单笔出错（单笔出错有事务回滚兜底），
而是长期存在不被发现的差错。

对账公式：
  当前余额 = 初始余额(0) + SUM(所有流水变动)

  如果不等 → 说明某次操作的余额更新或流水记录有遗漏
  → 资金黑洞（说不清钱去哪了）→ 审计灾难
```

## 10.2 流水连续性保障

```
balance_flow 表的关键设计：

  before_balance: 变动前余额（从当前行的available读取，在事务内）
  after_balance:  变动后余额（计算得出）

  连续性校验规则：
    第N条流水的 after_balance = 第N+1条流水的 before_balance

    如果不等 → 中间有"跳过"（余额被修改但没记录流水）→ Bug
```

## 10.3 before_balance 的精确获取

```
★ 关键问题：before_balance 从哪里取？

  错误方式：先 SELECT available，再做 UPDATE，再写流水
    → 并发场景下 SELECT 到的值可能已被另一个事务修改

  正确方式：在事务内，余额更新后，用 RETURNING 或 再 SELECT 获取更新后的值
    → TiDB/MySQL 不支持 UPDATE RETURNING
    → 方案：事务内先 SELECT FOR SHARE（共享锁读），拿到 before_balance
    → 然后 UPDATE（行级锁升级），再写流水
    → 整个过程在同一事务内，before_balance 是准确的

  更简单的方案（推荐）：
    事务内：
      ① SELECT available FROM user_balance WHERE ... FOR UPDATE
         → 拿到 before_balance（顺便锁行）
      ② UPDATE available = available - amount WHERE available >= amount
         → 执行扣减
      ③ after_balance = before_balance - amount
         → 直接计算
      ④ INSERT balance_flow (before_balance, after_balance, ...)

    ★ 注意：这里用了 FOR UPDATE，但目的不是做悲观锁（不是用它做并发控制），
      而是获取准确的 before_balance。并发控制仍然靠步骤②的 WHERE available >= amount。

    ★ 如果想完全避免 FOR UPDATE，替代方案：
      在乐观锁 UPDATE 的同时，用应用层计算 after_balance = before - amount
      但这需要先查 before（可能被并发修改），所以还是在事务内查更可靠。
```

## 10.4 对账方案（基础版）

```
对账定时任务（建议第一期后补充，或三期末实现）：

  // 每天凌晨执行一次全量对账
  func DailyReconciliation() {

      // 查所有有余额记录的用户
      users := getAllUsersWithBalance()

      for _, user := range users {
          for _, currency := range getAllCurrencies() {
              for _, walletType := range [1,2,3,4] {

                  // 当前余额
                  currentBalance := getBalance(user, currency, walletType)

                  // 流水累计
                  flowSum := sumAllFlows(user, currency, walletType)

                  // 校验
                  if currentBalance.Available != flowSum {
                      // ★ 不一致 → 记录异常 → 告警
                      alerter.SendWarning("对账差异",
                          user, currency, walletType,
                          currentBalance.Available, flowSum,
                          currentBalance.Available - flowSum,
                      )
                  }
              }
          }
      }
  }
```

# 十二、热点行与性能 — 高频操作的数据库压力

## 12.1 热点场景分析

```
热点行定义：同一行数据在短时间内被大量并发请求更新

钱包的热点场景：
  ① 同一用户快速操作：用户快速点击兑换 5 次 → 5 个请求并发更新同一行 user_balance
     → 频率有限（用户操作速度有上限），TiDB 行级锁串行处理，可接受

  ② 同一用户的充值回调：用户发起 5 笔充值，渠道回调几乎同时到达
     → 5 个 DepositCredit 并发更新同一用户同一币种的余额
     → 频率有限（不可能同时有大量充值），可接受

  ③ 不存在"万人同时操作同一行"的场景：
     → 每个用户的余额行是独立的，用户A和用户B的操作不会冲突
     → 不像秒杀场景（万人抢同一个库存行）

结论：当前业务规模下，热点行不是瓶颈。
```

## 12.2 查询性能优化

```
高频查询接口：
  ① QueryWalletBalance — 用户每次打开钱包都调
  ② QueryUserCurrencyList — 币种切换时调

优化方案：
  ① user_balance 的索引设计：
     UNIQUE KEY (user_id, currency, wallet_type)
     → 查询条件完全匹配索引，单行查询，微秒级

  ② 币种列表走 Redis 缓存（TTL 5min）
     → 命中率 > 99%（B端极少修改币种）

  ③ 余额不缓存 → 直接查DB
     → 单用户单币种最多 4 行（4 个子钱包），数据量极小

  ④ 记录查询的分页性能：
     balance_flow / deposit_order / withdraw_order
     INDEX (user_id, currency) — 按用户+币种分页查
     → 加上时间范围条件避免全表扫描
```

## 12.3 数据库连接池保障

```
铁律三的直接收益（RPC 在事务外）：

  事务持有连接时间 = 纯 DB 操作时间 = 1-5 毫秒
  不包含 RPC 等待时间（可能 100-3000 毫秒）

  按 TiDB 默认连接池 100 个连接：
    ✓ 事务 5ms：每秒可处理 20000 个事务
    ✗ 事务含RPC 500ms：每秒只能处理 200 个事务

  差距 100 倍。铁律三不是"最佳实践"，是"生存底线"。
```

# 十三、状态机正确性 — 订单生命周期管理

## 13.1 三张订单表的状态机

```
deposit_order 充值订单：

  pending ───→ success         渠道回调 → 财务确认 → 入账
     │
     ├──────→ failed           创建渠道订单失败 / 渠道返回失败
     │
     └──────→ timeout          超时未支付
                │
                └──→ patch_success   人工补单成功


withdraw_order 提现订单：

  reviewing ──→ success        三级审核通过 + 出款成功
     │
     ├────────→ failed         审核拒绝 / 出款失败 / 提交审核RPC失败(补偿)
     │
     └────────→ cancelled?     用户取消（待确认是否需要）


exchange_order 兑换订单：

  success                      即时完成（事务内直接写 success）
  failed                       余额不足/系统异常（理论上不会写入DB，直接返回错误）
```

## 13.2 状态转换的防护

```go
// 状态转换只允许特定路径，禁止任意跳转

// 充值订单状态转换校验
func ValidateDepositStatusTransition(current, target int) error {
    validTransitions := map[int][]int{
        enum.DepositPending: {enum.DepositSuccess, enum.DepositFailed, enum.DepositTimeout},
        enum.DepositTimeout: {enum.DepositPatchSuccess},
        // ★ success/failed/patch_success 是终态，不允许再变
    }

    targets, ok := validTransitions[current]
    if !ok {
        return fmt.Errorf("terminal state %d cannot transition", current)
    }
    for _, t := range targets {
        if t == target {
            return nil
        }
    }
    return fmt.Errorf("invalid transition: %d → %d", current, target)
}
```

```sql
-- rows = 0 → 当前状态已不是 pending（被其他请求改了）
-- rows = 1 → 转换成功
```

# 十四、错误处理与防御性编程 — 金融级编码规范

## 14.1 错误分类

```
钱包系统的错误分为四类：

  类型一：业务错误（可预期，用户操作导致）
    → 余额不足、超过限额、KYC未通过、币种已禁用
    → 处理：返回明确的业务错误码（6022xxx），前端展示给用户
    → 不记告警日志

  类型二：外部错误（可预期，外部系统导致）
    → 财务 RPC 超时、KYC 服务不可用
    → 处理：降级/补偿/重试
    → 记 WARN 日志

  类型三：数据错误（不可预期，数据不一致）
    → 订单不存在（应该存在时）、余额为负（理论不可能）
    → 处理：拒绝操作 + ERROR 日志 + 告警
    → 需要人工排查

  类型四：系统错误（不可预期，基础设施故障）
    → DB 连接失败、Redis 连接失败
    → 处理：返回系统错误 + ERROR 日志
```

## 14.3 防御性编程清单

```

  ✓ currency 非空
  ✓ currency 在启用列表中

  ✓ refOrderNo 非空（RPC 写接口必传）

零值防御：
  ✓ 金额为 0 时拒绝操作（不允许 0 元充值/提现/兑换）
  ✓ 汇率为 0 时拒绝兑换（除以 0 = panic）

负值防御：
  ✓ 所有金额字段在写入前校验 >= 0
  ✓ 余额操作后如果 available < 0 → 说明有Bug → 告警（理论上乐观锁已防护）
```

# 附录C：方案选型对比（为什么选 A 不选 B）

| 议题 | 选择方案 | 放弃方案 | 核心理由 |
|------|---------|---------|---------|
| 并发控制 | **乐观锁**（CAS WHERE >= ） | 悲观锁（SELECT FOR UPDATE） | 无死锁、1条SQL、并发度高、工程一致 |
| 幂等保障 | **双层**（应用层+DB唯一键） | 只应用层 / 只DB层 | 应用层快速返回+DB层极端兜底，两层互补 |
| 分布式事务 | **本地事务+补偿** | Seata/DTM | 只有1个补偿场景，不值得引入新组件 |
| 金额存储 | **BIGINT 整数** | DECIMAL / float | 零精度丢失、运算更快、全链路一致 |
| 除法取整 | **向下取整**（Truncate） | 四舍五入 / 向上取整 | 行业标准，平台微损可控，不会让用户多拿 |
| 缓存策略 | **Cache-Aside** | Write-Through / Write-Behind | 标准模式，读多写少场景最优，TTL兜底一致性 |
| 余额初始化 | **懒创建** | 注册时批量创建 | 按需创建无浪费，新增币种自动适配 |
| 余额缓存 | **不缓存** | Redis 缓存 | 金融数据必须实时准确，缓存带来一致性风险 |
| before_balance | **事务内查询** | 乐观计算 | 事务内查准确，乐观计算有并发偏差风险 |
| 加密方式 | **AES（已有工具）** | RSA / 国密 | 工程已有 EncryptedString，零引入 |
| 对账方案 | **定时任务全量对账** | 实时流式对账 | 第一期够用，按需迭代 |
| 状态转换 | **白名单校验**（允许的转换路径） | 无校验 | 防止非法状态跳转，如 success → pending |
