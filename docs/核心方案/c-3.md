## 第一章 幂等性

### 1.1 问题本质

金融系统最怕的不是失败，而是**成功了两次**。钱包的每一笔余额变动（充值入账、提现扣款、
兑换划转、手动调账）都可能因为以下原因被重复触发：

- **网络重传**：HTTP/RPC 超时后客户端重试，服务端已处理但响应丢失
- **消息重投**：Kafka 消费失败后重试，业务逻辑已执行但 offset 未提交
- **人为误操作**：B 端管理员重复点击提交按钮
- **跨模块回调**：支付系统回调通知重复推送

如果没有幂等保护，一次充值可能入账两次，一次扣款可能扣两次 —— 这在金融系统中是致命 Bug。

### 1.2 解决方案：orderNo + flow_type UNIQUE 索引

**核心思路**：每笔业务操作携带全局唯一的业务订单号 `orderNo`，结合流水类型 `flow_type`
在 `t_wallet_flow` 表建立 UNIQUE 索引。数据库层面保证同一笔业务只能产生一条流水记录。

**表级约束**：
```sql
-- t_wallet_flow 表
UNIQUE INDEX uk_order_flow (order_no, flow_type)
```

**orderNo 生成规则**：
```
前缀 + 雪花ID(16位)
C + snowflakeId  -- 充值订单  (C2310150000000001)
T + snowflakeId  -- 兑换订单  (T2310150000000002)
E + snowflakeId  -- 提现订单  (E2310150000000003)
B + snowflakeId  -- 奖励发放  (B2310150000000004)
A + snowflakeId  -- 手动调增  (A2310150000000005)
M + snowflakeId  -- 手动调减  (M2310150000000006)
```

### 1.3 伪代码

```go
// ======== 幂等入账（以 RPC 充值入账为例） ========

func (s *WalletService) WalletCredit(ctx context.Context, req *wallet.WalletCreditReq) (*wallet.WalletCreditResp, error) {
    // 1. 参数校验
    if req.OrderNo == "" || req.UserId == 0 {
        return nil, ret.BizErr(ctx, errs.ParamErr, "参数缺失")
    }

    // 2. 幂等检查：先查流水表是否已存在
    existing, err := flowRepo.GetByOrderNoAndType(ctx, req.OrderNo, enum.FlowTypeDeposit)
    if err != nil {
        return nil, ret.BizErr(ctx, errs.SystemErr, "查询流水失败")
    }
    if existing != nil {
        // 已处理过，直接返回成功（幂等）
        lg.Log().CtxInfof(ctx, "幂等命中，orderNo=%s 已入账", req.OrderNo)
        return &wallet.WalletCreditResp{}, nil
    }

    // 3. 进入事务（详见第二章）
    err = query.Q.Transaction(func(tx *query.Query) error {
        // 3a. 乐观锁更新余额（详见第四章）
        // 3b. 插入流水记录 —— UNIQUE 索引兜底
        //     如果此时并发的另一个相同 orderNo 请求也通过了步骤2的检查，
        //     INSERT 会因为 UNIQUE 约束报 Duplicate entry 错误，事务回滚
        return nil
    })

    if err != nil {
        if isDuplicateEntryError(err) {
            // UNIQUE 冲突 = 并发幂等，安全返回成功
            return &wallet.WalletCreditResp{}, nil
        }
        return nil, ret.BizErr(ctx, errs.SystemErr, "入账失败")
    }

    return &wallet.WalletCreditResp{}, nil
}

// isDuplicateEntryError 判断是否为 MySQL UNIQUE 冲突
func isDuplicateEntryError(err error) bool {
    var mysqlErr *mysql.MySQLError
    if errors.As(err, &mysqlErr) {
        return mysqlErr.Number == 1062 // Duplicate entry
    }
    return false
}
```

### 1.4 关键细节

| 细节 | 说明 |
|------|------|
| 两层防御 | 应用层先查（快速幂等返回），DB 层 UNIQUE 兜底（防并发穿透） |
| 幂等返回值 | 幂等命中时返回**成功**而非错误，对调用方透明 |
| flow_type 联合 | 同一个 orderNo 在不同 flow_type 下可能合法存在（如充值+奖励各一条） |
| 雪花 ID 唯一性 | 基于 `utils.GenerateSnowflakeID()`，mutex 保护，毫秒内 4096 序列 |
| RPC 场景 | ser-finance 调用 WalletCredit 时携带其侧生成的 orderNo，天然幂等键 |

## 第二章 事务一致性（本地事务）

### 2.1 问题本质

钱包的核心操作是"**余额变动 + 流水记录**"，这两步必须原子完成：

- 如果余额扣了但流水没写 → 账不平，对账时无法追溯
- 如果流水写了但余额没扣 → 资金凭空产生，系统泄漏
- 如果两步都成功但中间穿插了另一个并发操作 → 数据竞争，余额错乱

这不是简单的"包一个 Transaction 就行了"——需要考虑事务粒度、锁的范围、
失败后的状态一致性。

### 2.2 解决方案：GORM Transaction + 原子三步

**本项目事务基座**（来自 `ser-app/internal/rep/banner.go` 实际使用）：
```go
query.Q.Transaction(func(tx *query.Query) error {
    // 所有操作使用 tx 而不是 query.Q
    // 任何一步返回 error，整个事务自动回滚
    return nil
})
```

**注意**：项目 GORM 配置了 `SkipDefaultTransaction: true`（见 `common/pkg/db/mysql/mysql.go:44`），
所以普通的单条 CRUD 不会自动包事务，需要手动开启。这是正确的性能优化 —— 只在真正需要原子性
的地方手动包事务。

### 2.3 钱包核心事务模板：余额变动原子三步

每一次余额变动（不论充值/扣款/冻结/解冻），都遵循相同的事务模板：

```go
// ======== 通用余额变动事务模板 ========

func (s *WalletService) changeBalance(ctx context.Context, param *BalanceChangeParam) error {
    return query.Q.Transaction(func(tx *query.Query) error {
        // === Step 1: 查询当前账户（事务内读，获取最新 version） ===
        account, err := tx.WalletAccount.WithContext(ctx).
            Where(tx.WalletAccount.UserID.Eq(param.UserID)).
            Where(tx.WalletAccount.WalletType.Eq(param.WalletType)).
            Where(tx.WalletAccount.CurrencyCode.Eq(param.CurrencyCode)).
            First()
        if err != nil {
            return fmt.Errorf("查询账户失败: %w", err)
        }

        // === Step 2: 乐观锁更新余额（CAS: version 必须匹配） ===
        newBalance, err := calculateNewBalance(account.Balance, param.Amount, param.Direction)
        if err != nil {
            return err // 余额不足等业务错误
        }

        result, err := tx.WalletAccount.WithContext(ctx).
            Where(tx.WalletAccount.ID.Eq(account.ID)).
            Where(tx.WalletAccount.Version.Eq(account.Version)). // 乐观锁条件
            UpdateSimple(
                tx.WalletAccount.Balance.Value(newBalance),
                tx.WalletAccount.Version.Value(account.Version + 1),
                tx.WalletAccount.UpdateAt.Value(time.Now().UnixMilli()),
            )
        if err != nil {
            return fmt.Errorf("更新余额失败: %w", err)
        }
        if result.RowsAffected == 0 {
            return ErrOptimisticLock // 版本冲突，需要重试
        }

        // === Step 3: 插入流水记录（UNIQUE 索引保幂等） ===
        flow := &model.WalletFlow{
            OrderNo:      param.OrderNo,
            UserID:       param.UserID,
            WalletType:   param.WalletType,
            CurrencyCode: param.CurrencyCode,
            FlowType:     param.FlowType,
            Amount:       param.Amount,
            BalanceBefore: account.Balance,
            BalanceAfter:  newBalance,
            CreateAt:     time.Now().UnixMilli(),
        }
        if err := tx.WalletFlow.WithContext(ctx).Create(flow); err != nil {
            return fmt.Errorf("插入流水失败: %w", err)
        }

        return nil // 三步全成功，事务提交
    })
}
```

### 2.4 事务粒度原则

| 原则 | 说明 |
|------|------|
| 最小化事务范围 | 事务内只做 DB 操作，RPC 调用/Redis 操作放事务外 |
| 不在事务内做网络调用 | 避免事务持锁时间过长，死锁风险 |
| 单库事务 | 钱包只操作自己的库，不跨库事务 |
| 失败即回滚 | 任何一步返回 error，整个事务自动回滚 |
| 事务后才发消息 | Kafka 消息在事务提交成功后才发送 |

### 2.5 典型事务边界示例

```
充值入账完整流程：
  ┌── 事务外 ──┐
  │ 1. 参数校验    │
  │ 2. 幂等检查    │
  ├── 事务内 ──┤
  │ 3. 查账户      │  ← query.Q.Transaction 开始
  │ 4. 乐观锁改余额 │
  │ 5. 写流水      │
  ├── 事务外 ──┤  ← 事务提交成功
  │ 6. 更新缓存    │
  │ 7. 发 Kafka    │
  └──────────┘

  为什么第6、7步放事务外？
  - 如果放事务内：Redis/Kafka 超时 → 整个事务回滚 → 钱没入账
  - 放事务外：DB 已成功 → 缓存/消息失败可补偿（最终一致）
```

## 第三章 跨模块事务补偿

### 3.1 问题本质

钱包模块不是孤岛，提现流程涉及：
1. **ser-wallet**：冻结余额 → 写冻结流水
2. **ser-finance**：提交提现申请 → 审核 → 调用支付渠道
3. **ser-wallet**：收到结果后 → 扣款（成功）或 解冻（失败）

这是一个**跨模块的分布式事务**——不可能用本地 Transaction 包住，
因为 ser-wallet 和 ser-finance 是不同的服务、不同的数据库。

核心矛盾：**冻结成功但提交失败**怎么办？钱被冻住了，用户既不能用也不能提。

### 3.2 解决方案：冻结-提交-回滚 三段式 + 最终一致性

```
              ser-wallet                       ser-finance
              ─────────                        ───────────
用户发起提现 ──→ ① 冻结余额（本地事务）
                 balance -= amount
                 frozen_amount += amount
                 写冻结流水
              ──→ ② RPC: SubmitWithdraw ──────→ 接收提现申请
                                                  创建提现订单
                                              ←── 返回 success/fail
              ←── ③ 判断结果
                  ├─ 成功：更新本地订单状态=已提交
                  └─ 失败：立即回滚解冻 ──→ ④ 解冻余额（本地事务）
                                              frozen_amount -= amount
                                              balance += amount
                                              写解冻流水

              ... 异步等待（可能数分钟到数天） ...

              ←── ⑤ RPC回调: WalletDeduct ←── 审核通过+打款成功
                  扣减冻结金额（本地事务）       通知钱包扣款
                  frozen_amount -= amount
                  写扣款流水

              ←── ⑤' RPC回调: WalletUnfreeze ←── 审核拒绝/打款失败
                  解冻余额（本地事务）              通知钱包解冻
                  frozen_amount -= amount
                  balance += amount
                  写解冻流水
```

### 3.3 伪代码

```go
// ======== 提现发起：冻结 + RPC 提交 + 失败回滚 ========

func (s *WalletService) Withdraw(ctx context.Context, req *wallet.WithdrawReq) (*wallet.WithdrawResp, error) {
    // 1. 参数校验、KYC 检查（详见第十一章）
    // ...

    // 2. 生成提现订单号
    orderNo := "E" + utils.GenerateSnowflakeString()

    // 3. 本地事务：冻结余额
    err := s.freezeBalance(ctx, &FreezeParam{
        UserID:       req.UserId,
        WalletType:   enum.WalletCenter,
        CurrencyCode: req.CurrencyCode,
        Amount:       req.Amount,
        OrderNo:      orderNo,
    })
    if err != nil {
        return nil, err // 冻结失败（余额不足等），直接返回
    }

    // 4. RPC 调用 ser-finance 提交提现申请 —— 在事务外
    submitResp, err := rpc.FinanceClient().SubmitWithdraw(ctx, &finance.SubmitWithdrawReq{
        OrderNo:      orderNo,
        UserId:       req.UserId,
        Amount:       req.Amount,
        CurrencyCode: req.CurrencyCode,
        // ... 其他参数
    })

    // 5. RPC 失败：立即回滚解冻
    if err != nil || submitResp == nil {
        lg.Log().CtxErrorf(ctx, "提现提交失败，开始回滚解冻, orderNo=%s, err=%v", orderNo, err)
        unfreezeErr := s.unfreezeBalance(ctx, &UnfreezeParam{
            UserID:       req.UserId,
            WalletType:   enum.WalletCenter,
            CurrencyCode: req.CurrencyCode,
            Amount:       req.Amount,
            OrderNo:      orderNo,
        })
        if unfreezeErr != nil {
            // 解冻也失败 → 严重异常，告警 + 人工介入
            lg.Log().CtxErrorf(ctx, "【严重】提现回滚解冻失败, orderNo=%s, err=%v", orderNo, unfreezeErr)
            // TODO: 发告警通知（钉钉/飞书），记录异常表
        }
        return nil, ret.BizErr(ctx, errs.WithdrawSubmitFail, "提现提交失败")
    }

    // 6. 成功：更新本地提现记录状态
    // ...

    return &wallet.WithdrawResp{OrderNo: orderNo}, nil
}
```

### 3.4 补偿机制清单

| 场景 | 本地状态 | 远端状态 | 补偿策略 |
|------|---------|---------|---------|
| 冻结成功 + RPC 超时 | frozen | 未知 | 先尝试查询远端状态；无法确认则自动解冻 + 告警 |
| 冻结成功 + RPC 返回失败 | frozen | 失败 | 立即解冻（如上伪代码） |
| 冻结成功 + RPC 成功 + 远端后续失败 | frozen | 已提交→失败 | 等待远端回调 WalletUnfreeze |
| 冻结成功 + 解冻也失败 | frozen | 失败 | **告警 + 人工介入**（最终兜底） |

### 3.5 悬挂预防

**什么是悬挂**：冻结操作还没执行，解冻回调先到了。

```
时间线：
  T1: 用户发起提现
  T2: 钱包收到请求，准备冻结（但还没执行）
  T3: 网络抖动，ser-finance 认为超时，发回 "失败" 回调
  T4: 钱包收到 WalletUnfreeze 回调 → 找不到冻结记录 → ???
  T5: 冻结操作终于执行成功 → 余额被冻住，再也没人来解冻
```

**解决方案**：在 WalletUnfreeze 中，如果找不到对应的冻结流水记录，
先插入一条"空解冻"标记；冻结操作执行时检查是否已有解冻标记，
如果有则不冻结。

```go
func (s *WalletService) WalletUnfreeze(ctx context.Context, req *wallet.WalletUnfreezeReq) error {
    // 查找冻结流水
    freezeFlow, err := flowRepo.GetByOrderNoAndType(ctx, req.OrderNo, enum.FlowTypeFreeze)
    if err != nil {
        return ret.BizErr(ctx, errs.SystemErr)
    }

    if freezeFlow == nil {
        // 冻结还没发生 → 插入"空解冻"标记，防止后续冻结
        lg.Log().CtxWarnf(ctx, "悬挂预防：冻结记录不存在，插入空解冻标记, orderNo=%s", req.OrderNo)
        return flowRepo.InsertUnfreezeMarker(ctx, req.OrderNo)
    }

    // 正常解冻流程...
    return s.unfreezeBalance(ctx, ...)
}
```

## 第四章 并发控制 — 乐观锁

### 4.1 问题本质

同一个用户的钱包账户，可能在同一时刻被多个操作并发修改：

- 用户正在充值入账，同时收到一笔奖励发放
- 用户正在提现冻结，同时有一笔兑换完成
- B 端手动调账，同时 C 端有自动入账

如果不加控制，经典的"**丢失更新**"问题：
```
线程A: 读余额=100 → 计算 100+50=150 → 写入150
线程B: 读余额=100 → 计算 100+30=130 → 写入130
结果: 余额=130（A的+50被覆盖丢失了！应该是180）
```

### 4.2 解决方案：version 字段 CAS 更新

**`t_wallet_account` 表设计**：
```sql
version  INT NOT NULL DEFAULT 0 COMMENT '乐观锁版本号，每次更新+1'
```

**核心 SQL**：
```sql
UPDATE t_wallet_account
SET balance = ?, version = version + 1, update_at = ?
WHERE id = ? AND version = ?   -- CAS: 只有版本匹配才更新
```

如果 `RowsAffected == 0`，说明在我读和写之间，有另一个操作改了余额，版本号已变。

### 4.3 伪代码：带重试的乐观锁

```go
const maxRetry = 3

func (s *WalletService) creditWithOptimisticLock(ctx context.Context, param *CreditParam) error {
    for attempt := 0; attempt < maxRetry; attempt++ {
        err := query.Q.Transaction(func(tx *query.Query) error {
            // 1. 读取当前账户
            account, err := tx.WalletAccount.WithContext(ctx).
                Where(tx.WalletAccount.UserID.Eq(param.UserID)).
                Where(tx.WalletAccount.WalletType.Eq(param.WalletType)).
                Where(tx.WalletAccount.CurrencyCode.Eq(param.CurrencyCode)).
                First()
            if err != nil {
                return err
            }

            // 2. 计算新余额（使用 decimal 精确计算，详见第六章）
            oldBal, _ := decimal.NewFromString(account.Balance)
            delta, _ := decimal.NewFromString(param.Amount)
            newBal := oldBal.Add(delta)

            // 3. 乐观锁 CAS 更新
            result, err := tx.WalletAccount.WithContext(ctx).
                Where(tx.WalletAccount.ID.Eq(account.ID)).
                Where(tx.WalletAccount.Version.Eq(account.Version)).
                UpdateSimple(
                    tx.WalletAccount.Balance.Value(newBal.String()),
                    tx.WalletAccount.Version.Value(account.Version + 1),
                    tx.WalletAccount.UpdateAt.Value(time.Now().UnixMilli()),
                )
            if err != nil {
                return err
            }
            if result.RowsAffected == 0 {
                return ErrOptimisticLock // 版本冲突
            }

            // 4. 写流水
            return tx.WalletFlow.WithContext(ctx).Create(buildFlow(account, param, newBal))
        })

        if err == nil {
            return nil // 成功
        }
        if !errors.Is(err, ErrOptimisticLock) {
            return err // 非冲突错误，不重试
        }

        // 乐观锁冲突，重试
        lg.Log().CtxWarnf(ctx, "乐观锁冲突，重试 %d/%d, userId=%d", attempt+1, maxRetry, param.UserID)
    }

    return ret.BizErr(ctx, errs.SystemBusy, "系统繁忙，请稍后重试")
}
```

### 4.4 为什么用乐观锁而非悲观锁

| 对比 | 乐观锁 (version CAS) | 悲观锁 (SELECT FOR UPDATE) |
|------|--------|--------|
| 适用场景 | 冲突概率低（钱包场景，同一用户并发操作频率不高） | 冲突概率高、每次都会竞争 |
| 性能 | 不持锁，并发友好 | 持行锁，阻塞其他事务 |
| 死锁风险 | 无 | 有（多个 FOR UPDATE 交叉） |
| 实现复杂度 | 需要重试逻辑 | 简单但要注意锁顺序 |
| 本项目选择 | **采用** | 不采用 |

选择理由：同一用户的钱包操作虽然可能并发，但频率不会太高（不是秒杀场景），
乐观锁冲突概率 < 1%，3 次重试足够覆盖。

## 第五章 并发控制 — 分布式锁

### 5.1 问题本质

某些操作需要全局互斥，单靠数据库行级锁无法满足：

- **汇率同步定时任务**：多个实例部署时，同一时刻只能有一个实例执行汇率拉取
- **币种配置缓存刷新**：避免多个实例同时刷新导致缓存击穿
- **提现限额统计重置**：日限额/月限额在零点重置时，只需执行一次

### 5.2 解决方案：Redis SetNX

**本项目已有组件**（`common/pkg/rds/redis_util.go:107-114`）：
```go
func SetNX(key string, value interface{}, expiration time.Duration) bool {
    success, err := client.SetNX(ctx, key, value, expiration).Result()
    // ...
    return success
}
```

### 5.3 伪代码：汇率同步互斥锁

```go
const (
    rateSyncLockKey = "wallet:lock:rate_sync"
    rateSyncLockTTL = 30 * time.Second  // 锁超时时间
)

func (s *RateSyncService) SyncExchangeRates(ctx context.Context) error {
    // 1. 尝试获取分布式锁
    lockValue := utils.GenerateSnowflakeString() // 唯一值，用于安全释放
    if !rds.SetNX(rateSyncLockKey, lockValue, rateSyncLockTTL) {
        lg.Log().CtxInfof(ctx, "汇率同步：其他实例正在执行，跳过")
        return nil
    }

    // 2. 确保释放锁（Lua 脚本保原子性：只释放自己的锁）
    defer func() {
        luaScript := `
            if redis.call("GET", KEYS[1]) == ARGV[1] then
                return redis.call("DEL", KEYS[1])
            else
                return 0
            end
        `
        rds.Eval(ctx, luaScript, []string{rateSyncLockKey}, lockValue)
    }()

    // 3. 执行汇率同步逻辑
    rates, err := s.fetchRatesFromSource(ctx)
    if err != nil {
        return err
    }

    // 4. 批量更新汇率表
    return s.batchUpdateRates(ctx, rates)
}
```

### 5.4 锁设计要点

| 要点 | 说明 |
|------|------|
| 锁值唯一 | 用 snowflake ID 作为锁值，释放时验证，避免误删其他实例的锁 |
| 超时保底 | TTL 设为 30s，即使持锁实例崩溃，锁也会自动释放 |
| Lua 原子释放 | GET + DEL 必须原子操作，否则有 ABA 问题 |
| 非阻塞 | 获取不到锁就跳过（幂等任务），不自旋等待 |

### 5.5 钱包模块分布式锁清单

| 锁 Key 模式 | 用途 | TTL | 获取失败策略 |
|------------|------|-----|------------|
| `wallet:lock:rate_sync` | 汇率同步定时任务 | 30s | 跳过本次 |
| `wallet:lock:rate_cache_refresh` | 汇率缓存刷新 | 10s | 跳过，用旧缓存 |
| `wallet:lock:daily_limit_reset` | 每日限额重置 | 60s | 跳过本次 |

## 第六章 金额精度

### 6.1 问题本质

IEEE 754 浮点数在计算机中是不精确的：
```go
0.1 + 0.2 = 0.30000000000000004  // 不是 0.3！
```

在金融系统中，哪怕 0.00000001 的误差，累积百万笔后都会造成巨额资金差异。
而且钱包需要处理多种币种——法币（VND 小数位少但数值大）、加密货币（USDT 精度高）、
平台币（BSB），精度需求各不同。

### 6.2 解决方案：三层精度保障

```
                            全链路金额精度保障

  ┌─────────────┐    ┌──────────────┐    ┌──────────────┐
  │  传输层       │    │  计算层         │    │  存储层         │
  │  string 类型   │    │  decimal 库     │    │  DECIMAL(30,8) │
  │  IDL: string  │ →  │  shopspring    │ →  │  MySQL 定点数    │
  │  JSON: "1.5"  │    │  /decimal      │    │  精确存储        │
  └─────────────┘    └──────────────┘    └──────────────┘
```

**传输层**：所有金额字段在 IDL/JSON 中定义为 `string` 类型
```thrift
// IDL 示例
struct WalletCreditReq {
    4: required string amount;       // "100.50" 而不是 double
    5: required string exchangeRate; // "23456.78"
}
```

**计算层**：引入 `shopspring/decimal`（当前代码库未有，需新增依赖）
```go
import "github.com/shopspring/decimal"

// 所有金额计算使用 decimal
amount, _ := decimal.NewFromString("100.50")
rate, _ := decimal.NewFromString("23456.78")
result := amount.Mul(rate)  // 精确乘法
resultStr := result.StringFixed(8) // 保留8位小数
```

**存储层**：MySQL DECIMAL(30,8)
```sql
balance       DECIMAL(30,8) UNSIGNED NOT NULL DEFAULT 0.00000000 COMMENT '可用余额'
frozen_amount DECIMAL(30,8) UNSIGNED NOT NULL DEFAULT 0.00000000 COMMENT '冻结金额'
```

### 6.3 伪代码：精确余额计算

```go
// calculateNewBalance 精确计算新余额
// direction: 1=增加(入账) 2=减少(扣款)
func calculateNewBalance(currentBalance string, changeAmount string, direction int) (string, error) {
    current, err := decimal.NewFromString(currentBalance)
    if err != nil {
        return "", fmt.Errorf("当前余额格式错误: %s", currentBalance)
    }

    change, err := decimal.NewFromString(changeAmount)
    if err != nil {
        return "", fmt.Errorf("变动金额格式错误: %s", changeAmount)
    }

    // 金额必须为正数
    if change.LessThanOrEqual(decimal.Zero) {
        return "", fmt.Errorf("变动金额必须为正数: %s", changeAmount)
    }

    var newBalance decimal.Decimal
    switch direction {
    case 1: // 增加
        newBalance = current.Add(change)
    case 2: // 减少
        newBalance = current.Sub(change)
        if newBalance.LessThan(decimal.Zero) {
            return "", ErrInsufficientBalance // 余额不足
        }
    default:
        return "", fmt.Errorf("无效方向: %d", direction)
    }

    return newBalance.StringFixed(8), nil // 统一保留8位小数
}
```

### 6.4 汇率换算精度

```go
// convertAmount 汇率换算
// 法币 → 平台币(BSB) 示例：100 VND ÷ 23456.78 VND/BSB = ? BSB
func convertAmount(fiatAmount string, exchangeRate string) (string, error) {
    amount, _ := decimal.NewFromString(fiatAmount)
    rate, _ := decimal.NewFromString(exchangeRate)

    if rate.IsZero() {
        return "", fmt.Errorf("汇率不能为零")
    }

    // 法币 → BSB: 除以汇率
    bsbAmount := amount.Div(rate)

    // 截断到8位小数（不四舍五入，用 Truncate 防止多给）
    bsbAmount = bsbAmount.Truncate(8)

    return bsbAmount.String(), nil
}
```

### 6.5 精度红线

| 规则 | 说明 |
|------|------|
| 永不使用 float64 做金额计算 | 即使"看起来精度够"也不行 |
| 传输用 string | 避免 JSON 序列化精度丢失 |
| 截断不四舍五入 | 兑换时用 `Truncate`，永远不多给用户 |
| 存储 DECIMAL(30,8) | 30 位总长 + 8 位小数，覆盖所有币种场景 |
| 对账用字符串比较 | 不做浮点相等判断 |

## 第七章 冻结/解冻/扣款互斥

### 7.1 问题本质

提现场景中，`frozen_amount` 字段被三个操作共享：
- **冻结** (Freeze)：balance → frozen_amount（用户发起提现）
- **解冻** (Unfreeze)：frozen_amount → balance（审核拒绝/打款失败）
- **扣款** (Deduct)：frozen_amount → 0（打款成功，真正扣除）

如果这三个操作对同一笔订单并发执行（如回调重复推送），可能出现：
- 冻结了但重复扣款 → 多扣了
- 解冻和扣款同时执行 → 钱既还给了用户又被扣了

### 7.2 解决方案：订单状态机 + 乐观锁 + 流水唯一约束 三重防护

**第一层：订单状态机**
```
提现订单状态流转：

  [已冻结] ──→ [已提交] ──→ [处理中] ──→ [已完成]   ← 正常路径
     │            │            │
     │            ↓            ↓
     └────→ [已取消]     [已拒绝/已失败]              ← 异常路径
                            │
                            ↓
                        [已解冻]                      ← 终态
```

每个状态只允许特定的操作：
```go
var validTransitions = map[int32][]int32{
    StatusFrozen:     {StatusSubmitted, StatusCancelled},
    StatusSubmitted:  {StatusProcessing, StatusRejected},
    StatusProcessing: {StatusCompleted, StatusFailed},
    StatusRejected:   {StatusUnfrozen},
    StatusFailed:     {StatusUnfrozen},
}

func canTransit(from, to int32) bool {
    allowed, ok := validTransitions[from]
    if !ok {
        return false
    }
    for _, s := range allowed {
        if s == to {
            return true
        }
    }
    return false
}
```

**第二层：乐观锁更新订单状态**
```go
func (s *WalletService) transitOrderStatus(ctx context.Context, tx *query.Query,
    orderNo string, fromStatus, toStatus int32) error {

    if !canTransit(fromStatus, toStatus) {
        return fmt.Errorf("非法状态转换: %d → %d", fromStatus, toStatus)
    }

    result, err := tx.WithdrawOrder.WithContext(ctx).
        Where(tx.WithdrawOrder.OrderNo.Eq(orderNo)).
        Where(tx.WithdrawOrder.Status.Eq(fromStatus)). // 乐观锁：状态必须匹配
        UpdateSimple(
            tx.WithdrawOrder.Status.Value(toStatus),
            tx.WithdrawOrder.UpdateAt.Value(time.Now().UnixMilli()),
        )
    if err != nil {
        return err
    }
    if result.RowsAffected == 0 {
        // 状态已变更（被其他操作抢先了）
        return ErrStatusAlreadyChanged
    }
    return nil
}
```

**第三层：流水唯一约束兜底**
```sql
UNIQUE INDEX uk_order_flow (order_no, flow_type)
-- 同一笔订单的冻结/解冻/扣款流水各只能有一条
```

### 7.3 完整的扣款伪代码

```go

    // 1. 幂等检查
    existFlow, _ := flowRepo.GetByOrderNoAndType(ctx, req.OrderNo, enum.FlowTypeDeduct)
    if existFlow != nil {
        return nil // 已扣过，幂等返回
    }

    // 2. 事务：状态转换 + 扣减冻结金额 + 写流水
    return query.Q.Transaction(func(tx *query.Query) error {
        // 2a. 状态机校验 + 乐观锁转换

        if err != nil {
            if errors.Is(err, ErrStatusAlreadyChanged) {

            }
            return err
        }

        // 2b. 扣减冻结金额
        account, _ := tx.WalletAccount.WithContext(ctx).
            Where(/* userId, walletType, currencyCode */).First()

        frozenBal, _ := decimal.NewFromString(account.FrozenAmount)
        deductAmt, _ := decimal.NewFromString(req.Amount)
        newFrozen := frozenBal.Sub(deductAmt)

        if newFrozen.LessThan(decimal.Zero) {
            return fmt.Errorf("冻结金额异常：冻结=%s, 扣减=%s", account.FrozenAmount, req.Amount)
        }

        result, _ := tx.WalletAccount.WithContext(ctx).
            Where(tx.WalletAccount.ID.Eq(account.ID)).
            Where(tx.WalletAccount.Version.Eq(account.Version)).
            UpdateSimple(
                tx.WalletAccount.FrozenAmount.Value(newFrozen.StringFixed(8)),
                tx.WalletAccount.Version.Value(account.Version + 1),
                tx.WalletAccount.UpdateAt.Value(time.Now().UnixMilli()),
            )
        if result.RowsAffected == 0 {
            return ErrOptimisticLock
        }

        // 2c. 写扣款流水（UNIQUE 兜底）
        return tx.WalletFlow.WithContext(ctx).Create(&model.WalletFlow{
            OrderNo:  req.OrderNo,
            FlowType: enum.FlowTypeDeduct,
            Amount:   req.Amount,
            // ...
        })
    })
}
```

## 第八章 汇率一致性 — 锁价快照

### 8.1 问题本质

用户发起兑换时看到的汇率是 `23456.78 VND/BSB`，但从用户点击确认到订单真正处理，
可能经过了几秒甚至几分钟。如果这期间汇率从 23456 变成了 24000，
用同一金额换到的 BSB 就不一样了。

**如果不锁价**：
- 用户看到 "100 VND = 0.00426 BSB"，点确认
- 处理时汇率已变，实际只给了 0.00416 BSB
- 用户投诉："我看到的不是这个数！"

### 8.2 解决方案：下单锁价 + 快照不可变

```
用户看到汇率 → 发起兑换请求 → 服务端记录当时的汇率快照 → 整个订单生命周期使用快照汇率
                                    ↓
                            t_exchange_order
                            ├── exchange_rate: "23456.78"  (锁定)
                            ├── rate_snapshot_at: 1234567890 (锁定时间)
                            └── 订单完成前，此汇率不变
```

### 8.3 伪代码

```go
func (s *WalletService) Exchange(ctx context.Context, req *wallet.ExchangeReq) (*wallet.ExchangeResp, error) {
    // 1. 获取当前有效汇率
    rate, err := s.getCurrentRate(ctx, req.FromCurrency, req.ToCurrency)
    if err != nil {
        return nil, ret.BizErr(ctx, errs.RateUnavailable, "汇率获取失败")
    }

    // 2. 汇率时效性校验（汇率更新时间不能超过阈值）
    if time.Now().UnixMilli()-rate.UpdateAt > rateMaxAgeMs {
        return nil, ret.BizErr(ctx, errs.RateExpired, "汇率已过期，请刷新")
    }

    // 3. 用锁定的汇率精确计算
    fromAmount, _ := decimal.NewFromString(req.Amount)
    rateValue, _ := decimal.NewFromString(rate.WithdrawRate) // 使用提现/充值方向对应的汇率
    toAmount := fromAmount.Div(rateValue).Truncate(8)

    // 4. 生成订单号
    orderNo := "T" + utils.GenerateSnowflakeString()

    // 5. 事务：扣源钱包 + 加目标钱包 + 写订单 + 写两条流水
    err = query.Q.Transaction(func(tx *query.Query) error {
        // 5a. 扣源钱包余额（乐观锁）
        if err := s.debitAccount(ctx, tx, req.UserId, req.FromWallet, req.FromCurrency,
            req.Amount, orderNo, enum.FlowTypeExchangeOut); err != nil {
            return err
        }

        // 5b. 加目标钱包余额（乐观锁）
        if err := s.creditAccount(ctx, tx, req.UserId, req.ToWallet, req.ToCurrency,
            toAmount.String(), orderNo, enum.FlowTypeExchangeIn); err != nil {
            return err
        }

        // 5c. 写兑换订单（锁定汇率快照）
        return tx.ExchangeOrder.WithContext(ctx).Create(&model.ExchangeOrder{
            OrderNo:        orderNo,
            UserID:         req.UserId,
            FromCurrency:   req.FromCurrency,
            ToCurrency:     req.ToCurrency,
            FromAmount:     req.Amount,
            ToAmount:       toAmount.String(),
            ExchangeRate:   rate.WithdrawRate,    // ← 锁定的汇率
            RateSnapshotAt: rate.UpdateAt,          // ← 汇率快照时间
            Status:         enum.ExchangeCompleted,
            CreateAt:       time.Now().UnixMilli(),
        })
    })

    if err != nil {
        return nil, err
    }

    return &wallet.ExchangeResp{
        OrderNo:  orderNo,
        ToAmount: toAmount.String(),
        Rate:     rate.WithdrawRate,
    }, nil
}
```

### 8.4 汇率四层体系

```
外部数据源 → real_rate    (真实市场汇率，定时拉取)
             ↓
         platform_rate    (平台基准汇率 = real_rate × 平台系数)
             ↓
         ┌── deposit_rate   (充值汇率 = platform_rate × 充值系数)
         └── withdraw_rate  (提现汇率 = platform_rate × 提现系数)

每一层汇率独立存储、独立版本、独立缓存。
用户看到的始终是 deposit_rate 或 withdraw_rate，不直接接触 real_rate。
```

### 8.5 汇率缓存策略

```go
const (
    rateCacheKey    = "wallet:rate:%s_%s"     // wallet:rate:VND_BSB
    rateCacheTTL    = 5 * time.Minute         // 缓存5分钟
    rateMaxAgeMs    = 10 * 60 * 1000          // 汇率最大有效期10分钟
)

func (s *WalletService) getCurrentRate(ctx context.Context, from, to string) (*RateInfo, error) {
    // 1. 先查 Redis 缓存
    cacheKey := fmt.Sprintf(rateCacheKey, from, to)
    cached := rds.Get(cacheKey)
    if cached != "" {
        var rate RateInfo
        if json.Unmarshal([]byte(cached), &rate) == nil {
            return &rate, nil
        }
    }

    // 2. 缓存未命中，查 DB
    rate, err := rateRepo.GetLatestRate(ctx, from, to)
    if err != nil {
        return nil, err
    }

    // 3. 写入缓存
    rds.Set(ctx, cacheKey, utils.MarshalStr(ctx, rate), rateCacheTTL)

    return rate, nil
}
```

## 第九章 异步消息可靠性

### 9.1 问题本质

钱包模块中 Kafka 消息用于：
- **汇率同步通知**：汇率更新后通知依赖方
- **余额变动通知**：通知 C 端 WebSocket 推送
- **对账事件**：生成对账数据

消息丢失 = 数据不一致，消息重复 = 回到幂等问题。

### 9.2 现有工程基座分析

**生产者**（`common/pkg/kafka/producer.go`）：
- 异步发送（`AsyncProducer`）
- 幂等选项（`Idempotent: true`，需要 `RequiredAcks = -1`）
- 10 次重试，1 秒间隔
- 3 秒发送超时

**消费者**（`common/pkg/kafka/consumer.go`）：
- 消费失败自动重试 3 次（递归调用 `handleExe`）
- 超过 3 次记录 Error 日志
- 支持手动提交 offset（`AutoCommit: false` + `sess.Commit()`）

### 9.3 钱包消息策略

```go
// ======== 生产者：事务提交后再发消息 ========
```

```go
// ======== 消费者：幂等消费 + 手动提交 ========

func handleBalanceChangeEvent(msgBytes []byte) error {
    var event BalanceChangeEvent
    if err := json.Unmarshal(msgBytes, &event); err != nil {
        // 反序列化失败，重试也没用，直接丢弃 + 告警
        lg.Log().CtxErrorf(context.Background(), "消息反序列化失败: %v", err)
        return nil // 返回 nil 避免无限重试
    }

    // 幂等检查：基于 orderNo 判断是否已处理
    if isEventProcessed(event.OrderNo) {
        return nil // 已处理，跳过
    }

    // 执行业务逻辑（如推送 WebSocket 通知）
    if err := pushNotification(event); err != nil {
        return err // 返回 error，触发重试（最多3次）
    }

    // 标记已处理
    markEventProcessed(event.OrderNo)
    return nil
}
```

### 9.4 消息可靠性保障矩阵

| 阶段 | 风险 | 保障措施 |
|------|------|---------|
| 发送前 | 事务成功但消息未发 | 事务后发送；失败记日志，对账补偿 |
| 发送中 | 消息丢失 | RequiredAcks=-1（全副本确认）+ 10 次重试 |
| 发送后 | 消息重复 | 消费端幂等（orderNo 去重） |
| 消费中 | 处理失败 | 3 次自动重试 + Error 日志告警 |
| 消费后 | offset 提交失败 | 手动提交 + 消费幂等（重复消费安全） |

## 第十章 防重复转账 / 重复回调

### 10.1 问题本质

支付系统的回调通知（充值成功、提现结果）可能重复到达：
- 支付渠道主动重试
- 网络抖动导致超时重发
- 人工补发通知

如果不做拦截，同一笔充值可能入账多次。

### 10.2 解决方案：多层拦截

**第一层：Redis 短期去重**
```go
const callbackDedupePrefix = "wallet:callback:dedup:"
const callbackDedupeTTL    = 24 * time.Hour

func isCallbackDuplicate(orderNo string, callbackType string) bool {
    key := callbackDedupePrefix + orderNo + ":" + callbackType
    // SetNX: 首次设置成功返回 true（未重复），已存在返回 false（重复）
    return !rds.SetNX(key, "1", callbackDedupeTTL)
}
```

**第二层：流水表 UNIQUE 索引**（与第一章一致）
即使 Redis 去重失效（如 Redis 故障恢复后缓存丢失），
DB 层的 `uk_order_flow (order_no, flow_type)` 也会拦截。

**第三层：订单状态机校验**
已完成的订单不接受重复操作。

### 10.3 完整回调处理流程

```go
func (s *WalletService) WalletCredit(ctx context.Context, req *wallet.WalletCreditReq) (*wallet.WalletCreditResp, error) {
    // Layer 1: Redis 快速去重
    if isCallbackDuplicate(req.OrderNo, "credit") {
        lg.Log().CtxInfof(ctx, "回调去重命中(Redis), orderNo=%s", req.OrderNo)
        return &wallet.WalletCreditResp{}, nil
    }

    // Layer 2: DB 流水查询去重
    existing, _ := flowRepo.GetByOrderNoAndType(ctx, req.OrderNo, enum.FlowTypeDeposit)
    if existing != nil {
        lg.Log().CtxInfof(ctx, "回调去重命中(DB), orderNo=%s", req.OrderNo)
        return &wallet.WalletCreditResp{}, nil
    }

    // Layer 3: 事务处理（UNIQUE 索引终极兜底）
    err := s.creditWithOptimisticLock(ctx, &CreditParam{
        OrderNo: req.OrderNo,
        // ...
    })

    if err != nil {
        if isDuplicateEntryError(err) {
            return &wallet.WalletCreditResp{}, nil // 并发去重
        }
        return nil, err
    }

    return &wallet.WalletCreditResp{}, nil
}
```

## 第十一章 KYC 服务降级

### 11.1 问题本质

提现操作需要检查用户 KYC 状态：
- **未认证用户**：只能 USDT 提现
- **已认证用户**：所有渠道均可

但 KYC 服务（`ser-kyc`）是外部依赖，可能：
- 响应超时
- 返回异常
- 服务整体不可用

**如果 KYC 不可用就完全阻断提现**，用户体验极差。
**如果 KYC 不可用就放行所有**，安全风险极高。

### 11.2 解决方案：安全降级 + 缓存兜底

```go
const (
    kycCacheKey = "wallet:kyc:status:%d"  // userId
    kycCacheTTL = 30 * time.Minute
)

func (s *WalletService) checkKYCForWithdraw(ctx context.Context, userId int64, channel string) error {
    // 1. 先查 Redis 缓存
    cacheKey := fmt.Sprintf(kycCacheKey, userId)
    cachedStatus := rds.Get(cacheKey)

    var kycVerified bool

    if cachedStatus != "" {
        // 缓存命中
        kycVerified = cachedStatus == "1"
    } else {
        // 2. 缓存未命中，RPC 查询 ser-kyc
        resp, err := rpc.KycClient().GetUserKycStatus(ctx, &kyc.GetKycStatusReq{
            UserId: userId,
        })

        if err != nil {
            // 3. KYC 服务不可用 → 安全降级策略
            lg.Log().CtxWarnf(ctx, "KYC 服务降级, userId=%d, err=%v", userId, err)
            // 降级策略：只允许 USDT 提现（最保守的权限）
            if channel != "USDT" {
                return ret.BizErr(ctx, errs.KycServiceUnavailable,
                    "身份验证服务暂时不可用，当前仅支持USDT提现")
            }
            return nil // USDT 提现放行
        }

        kycVerified = resp.Status == enum.KycVerified

        // 4. 写入缓存
        statusStr := "0"
        if kycVerified {
            statusStr = "1"
        }
        rds.Set(ctx, cacheKey, statusStr, kycCacheTTL)
    }

    // 5. 业务规则判断
    if !kycVerified && channel != "USDT" {
        return ret.BizErr(ctx, errs.KycNotVerified, "未完成身份认证，仅支持USDT提现")
    }

    return nil
}
```

### 11.3 降级策略矩阵

| KYC 状态 | 服务可用 | 服务不可用 |
|----------|---------|-----------|
| 已认证 | 全渠道放行 | 使用缓存（30min内有效）→ 全渠道放行 |
| 未认证 | 仅 USDT | 仅 USDT |
| 未知（缓存也没有） | — | **仅 USDT**（安全默认值） |

核心原则：**降级时收紧权限，不放大权限**。宁可让已认证用户暂时只能 USDT 提现，
也不能让未认证用户绕过 KYC 检查。

## 第十二章 余额安全兜底

### 12.1 问题本质

余额字段出现负数 = 金融事故。不论代码逻辑多么"正确"，
都需要多层兜底防止**余额变为负数**。

### 12.2 四层防护

**第一层：应用层 decimal 校验**
```go
// 扣款前检查
if newBalance.LessThan(decimal.Zero) {
    return ErrInsufficientBalance
}
```

**第二层：DB 字段 UNSIGNED 约束**
```sql
balance DECIMAL(30,8) UNSIGNED NOT NULL DEFAULT 0.00000000
-- UNSIGNED: MySQL 层面拒绝负数，即使应用层漏了检查
-- 如果 UPDATE 导致负数，MySQL 会报错，事务回滚
```

**第三层：事务内一致性检查**
```go
// 事务提交前做最终校验
err = query.Q.Transaction(func(tx *query.Query) error {
    // ... 余额变动操作 ...

    // 最终校验：重新读取，确认余额非负
    updated, _ := tx.WalletAccount.WithContext(ctx).
        Where(tx.WalletAccount.ID.Eq(account.ID)).First()
    bal, _ := decimal.NewFromString(updated.Balance)
    if bal.LessThan(decimal.Zero) {
        return fmt.Errorf("【严重】余额异常为负, userId=%d, balance=%s", userId, updated.Balance)
    }
    return nil
})
```

**第四层：定时对账**
```go
// 每日对账任务：检查所有账户
// 余额 + 冻结金额 = 所有入账流水 - 所有出账流水
func (s *ReconcileService) DailyReconcile(ctx context.Context) {
    accounts, _ := accountRepo.ListAll(ctx)
    for _, acc := range accounts {
        expected := calculateExpectedBalance(acc.UserID, acc.WalletType, acc.CurrencyCode)
        actual, _ := decimal.NewFromString(acc.Balance)
        frozen, _ := decimal.NewFromString(acc.FrozenAmount)
        total := actual.Add(frozen)

        if !total.Equal(expected) {
            lg.Log().CtxErrorf(ctx, "【对账异常】userId=%d, wallet=%d, 预期=%s, 实际=%s",
                acc.UserID, acc.WalletType, expected.String(), total.String())
            // 发送告警通知
        }
    }
}
```

```go
// 正确：批量查询
accounts, _ := accountRepo.BatchGetByUserIds(ctx, userIds)

    accountMap[acc.UserID] = acc

```

**分页查询索引**：

```sql
INDEX idx_user_currency_time (user_id, currency_code, create_at DESC)
```

### 13.3 缓存策略总览

| 缓存对象 | Key 模式 | TTL | 更新策略 |
|----------|---------|-----|---------|
| 币种配置列表 | `wallet:currency:config:list` | 10min | 配置变更时主动删除 |
| 单币种配置 | `wallet:currency:config:{code}` | 10min | 同上 |
| 用户余额概览 | `wallet:balance:{userId}` | 5min | 余额变动后主动删除 |
| 汇率 | `wallet:rate:{from}_{to}` | 5min | 汇率同步后主动更新 |
| KYC 状态 | `wallet:kyc:status:{userId}` | 30min | KYC 变更回调时删除 |

**缓存一致性策略**：
```
写操作流程：
  1. 事务更新 DB
  2. 事务提交成功
  3. 删除缓存（Del，不是 Set）
  4. 下次读取时自动回源 DB 并重建缓存

为什么用"删除"而不是"更新"？
  - 避免并发写导致缓存与 DB 不一致
  - 删除是幂等的，更新不是
```

### 14.3 金额合理性校验

```go
func validateAmount(amount string, currencyConfig *CurrencyConfig) error {
    amt, err := decimal.NewFromString(amount)
    if err != nil {
        return fmt.Errorf("金额格式错误")
    }

    // 必须为正数
    if amt.LessThanOrEqual(decimal.Zero) {
        return fmt.Errorf("金额必须为正数")
    }

    // 不超过币种最大值
    maxAmt, _ := decimal.NewFromString(currencyConfig.MaxAmount)
    if amt.GreaterThan(maxAmt) {
        return fmt.Errorf("超过单笔最大金额限制")
    }

    // 不低于币种最小值
    minAmt, _ := decimal.NewFromString(currencyConfig.MinAmount)
    if amt.LessThan(minAmt) {
        return fmt.Errorf("低于单笔最小金额限制")
    }

    // 小数位数检查
    if amt.Exponent() < -int32(currencyConfig.DecimalPlaces) {
        return fmt.Errorf("小数位数超过币种限制")
    }

    return nil
}
```
