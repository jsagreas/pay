## 议题一：并发余额操作的原子性

### 1.1 不做会怎样

```
场景：用户余额=1000，充值回调+100 和 提现冻结-500 同时到达

  线程A（充值）：读balance=1000 → 计算1000+100=1100 → 写入1100
  线程B（提现）：读balance=1000 → 计算1000-500=500  → 写入500

  最终balance=500（充值丢失）或1100（提现丢失）
  正确值应该是600

后果：丢失更新 → 用户资金无故增减 → 资损 → 生产事故
```

### 1.2 方案：三层并发防线

```
第一层：数据库行锁（SELECT ... FOR UPDATE）
  → 同一事务内，其他事务对同一行的读写阻塞等待
  → 锁粒度：user_id + currency_code + wallet_type = 一把锁
  → 锁范围：一个用户的一个币种的一个钱包类型
  → 不同用户之间零互斥，不同币种之间零互斥

第二层：乐观锁（version字段）
  → UPDATE ... WHERE id=? AND version=? → 更新0行说明被别人改过
  → 跨事务的兜底：万一行锁在某种极端情况失效，乐观锁还能拦住

第三层：幂等键（唯一约束）
  → wallet_transaction 的 uk_order_wallet_tx
  → 两个完全相同的请求，只有一个INSERT成功
  → 数据库级最终兜底
```

### 1.3 代码骨架

```go
func (e *BalanceEngine) Add(ctx context.Context, op *BalanceOp) error {
    return e.repo.WithTransaction(func(tx *query.Query) error {

        // ═══ 第三层：幂等检查（最先执行，快速拒绝重复请求）═══
        exists, _ := tx.WalletTransaction.
            Where(tx.WalletTransaction.OrderNo.Eq(op.OrderNo)).
            Where(tx.WalletTransaction.WalletType.Eq(op.WalletType)).
            Where(tx.WalletTransaction.TxType.Eq(op.TxType)).
            First()
        if exists != nil {
            return nil // 已处理，幂等返回成功
        }

        // ═══ 第一层：行锁（锁住目标钱包行）═══
        wallet, err := tx.Wallet.
            Where(tx.Wallet.UserID.Eq(op.UserID)).
            Where(tx.Wallet.CurrencyCode.Eq(op.CurrencyCode)).
            Where(tx.Wallet.WalletType.Eq(op.WalletType)).
            ForUpdate(). // ← 行锁：其他事务对此行的操作将阻塞
            First()
        if err != nil {
            // 钱包不存在 → 自动创建（首次充值该币种）
            wallet = e.autoCreateWallet(tx, op)
        }

        // ═══ 计算新余额 ═══
        newBalance := wallet.Balance + op.Amount

        // ═══ 第二层：乐观锁更新 ═══
        result, _ := tx.Wallet.
            Where(tx.Wallet.ID.Eq(wallet.ID)).
            Where(tx.Wallet.Version.Eq(wallet.Version)). // ← 乐观锁条件
            UpdateSimple(
                tx.Wallet.Balance.Value(newBalance),
                tx.Wallet.Version.Value(wallet.Version + 1), // ← version+1
                tx.Wallet.UpdateAt.Value(time.Now().Unix()),
            )
        if result.RowsAffected == 0 {
            return ErrConcurrentConflict // 乐观锁冲突，调用方可重试
        }

        // ═══ 写流水（同一事务内）═══
        return tx.WalletTransaction.Create(&model.WalletTransaction{
            UserID:        op.UserID,
            CurrencyCode:  op.CurrencyCode,
            WalletType:    op.WalletType,
            TxType:        op.TxType,
            Amount:        op.Amount,
            BalanceBefore:  wallet.Balance,    // 变更前
            BalanceAfter:   newBalance,         // 变更后
            OrderNo:        op.OrderNo,
            Status:         StatusCompleted,
            Remark:         op.Remark,
            CreateAt:       uint64(time.Now().Unix()),
        })
    })
}
```

### 1.4 关键参数

```
行锁等待超时：TiDB默认 innodb_lock_wait_timeout = 50s（足够）
乐观锁重试：上层最多重试2次，重试间隔50ms（指数退避）
事务超时：GORM事务默认不超时，建议设置 context.WithTimeout(ctx, 5*time.Second)
锁粒度：user_id + currency_code + wallet_type（不会误锁其他用户/币种）
```

## 议题二：幂等性——同一笔操作只能生效一次

### 2.1 不做会怎样

```
场景A：三方支付重复推送充值回调（网络重试、补偿机制）
  → 财务收到2次回调 → 2次RPC调CreditDeposit → 如果不幂等：入账2次
  → 用户充100变200，平台凭空亏100

场景B：财务调钱包RPC超时 → 实际已成功 → 财务重试
  → 如果不幂等：又入一次 → 双倍入账

场景C：网络抖动导致响应丢失，调用方认为失败而重试
  → 同上

后果：重复入账/重复扣款 → 资损
```

### 2.2 方案：三层幂等架构

```
Layer 1（应用层快速拒绝）：查wallet_transaction是否已有该order_no
  → 有 → 直接返回成功（响应时间 ~1ms）
  → 无 → 继续

Layer 2（数据库唯一键兜底）：wallet_transaction的uk_order_wallet_tx
  → 两个并发INSERT，只有一个成功，另一个唯一键冲突
  → 冲突 → 查已有记录 → 返回成功

Layer 3（业务状态机防线）：订单status字段
  → 只有正确状态才允许操作
  → 已完成 → 拒绝再次操作
```

### 2.3 代码骨架——RPC入口的通用幂等模式

```go
// 所有RPC写操作都遵循这个模式
func (s *RpcFinanceService) CreditDeposit(ctx context.Context, req *CreditDepositReq) (*CreditDepositResp, error) {

    // ═══ 参数校验 ═══
    if req.OrderNo == "" || req.UserID == 0 || req.Amount <= 0 {
        return nil, ErrInvalidParams
    }

    // ═══ Layer 1：应用层幂等检查 ═══
    existingTx, _ := s.transactionRepo.FindByOrderNoAndType(ctx, req.OrderNo, WalletTypeCenter, TxTypeDeposit)
    if existingTx != nil {
        // 已经处理过了，直接返回成功（幂等语义）
        return &CreditDepositResp{
            Success:       true,
            WalletBalance: existingTx.BalanceAfter,
        }, nil
    }

    // ═══ 执行余额操作（BalanceEngine内部有Layer 2和Layer 3）═══
    err := s.balanceEngine.Add(ctx, &BalanceOp{
        UserID:       req.UserID,
        CurrencyCode: req.CurrencyCode,
        WalletType:   WalletTypeCenter,
        Amount:       req.DepositAmount,
        TxType:       TxTypeDeposit,
        OrderNo:      req.OrderNo,
    })

    if err != nil {
        // ═══ Layer 2处理：唯一键冲突 = 另一个并发请求已成功 ═══
        if isDuplicateKeyError(err) {
            return &CreditDepositResp{Success: true}, nil
        }
        return nil, err
    }

    // ═══ 如果有奖金，继续入奖励钱包 ═══
    if req.BonusAmount > 0 {
        _ = s.balanceEngine.Add(ctx, &BalanceOp{
            UserID:       req.UserID,
            CurrencyCode: req.CurrencyCode,
            WalletType:   WalletTypeReward,
            Amount:       req.BonusAmount,
            TxType:       TxTypeBonus,
            OrderNo:      req.OrderNo,
        })
    }

    // ═══ 更新充值订单状态 ═══
    s.depositOrderRepo.UpdateStatus(ctx, req.OrderNo, OrderStatusCompleted)

    return &CreditDepositResp{Success: true}, nil
}

// isDuplicateKeyError 判断是否是唯一键冲突
func isDuplicateKeyError(err error) bool {
    // MySQL error 1062: Duplicate entry
    return strings.Contains(err.Error(), "1062") ||
           strings.Contains(err.Error(), "Duplicate entry")
}
```

## 议题三：余额与流水的强一致性

### 3.1 不做会怎样

```
场景：余额更新成功但流水写入失败（或反过来）
  → 余额变了但没有流水记录 → 对账发现数字对不上
  → 更严重：有人直接改了数据库余额但没有流水 → 无法追溯

后果：
  账不对 → 无法审计 → 无法定位问题 → 监管合规失败
```

### 3.2 方案：同事务写入 + balance_before/after + 异步对账

```
铁律：余额变更和流水写入永远在同一个数据库事务中，不可分离。

  BEGIN
    UPDATE wallet SET balance = new_balance ...     ← 改余额
    INSERT wallet_transaction (                     ← 写流水
      balance_before = old_balance,
      balance_after  = new_balance,
      amount         = delta
    )
  COMMIT

  任何一步失败 → 自动ROLLBACK → 余额和流水都不变

对账约束（5条铁律）：
  ① wallet.balance == 该用户最新一条流水的 balance_after
  ② 每条流水：balance_after == balance_before + amount
  ③ 第N条流水的 balance_before == 第N-1条的 balance_after（链式）
  ④ balance >= 0（非负约束）
  ⑤ frozen_amount >= 0（非负约束）
```

### 3.3 代码骨架——对账任务

```go
// DailyReconciliation 每日对账（凌晨执行）
func (s *ReconciliationService) DailyReconciliation(ctx context.Context) (*ReconciliationReport, error) {
    report := &ReconciliationReport{}

    // 取所有有余额的钱包
    wallets, _ := s.walletRepo.FindAllWithBalance(ctx)

    for _, wallet := range wallets {
        // 约束①：余额 == 最新流水的balance_after
        latestTx, _ := s.txRepo.FindLatest(ctx, wallet.UserID, wallet.CurrencyCode, wallet.WalletType)
        if latestTx != nil && latestTx.BalanceAfter != wallet.Balance {
            report.AddMismatch(wallet, "余额与最新流水不一致",
                fmt.Sprintf("wallet.balance=%d, latest_tx.balance_after=%d", wallet.Balance, latestTx.BalanceAfter))
        }

        // 约束②+③：流水链式校验
        txList, _ := s.txRepo.FindByWallet(ctx, wallet.UserID, wallet.CurrencyCode, wallet.WalletType)
        for i, tx := range txList {
            // 约束②
            if tx.BalanceAfter != tx.BalanceBefore + tx.Amount {
                report.AddMismatch(wallet, "流水before/after不一致", fmt.Sprintf("tx_id=%d", tx.ID))
            }
            // 约束③
            if i > 0 && tx.BalanceBefore != txList[i-1].BalanceAfter {
                report.AddMismatch(wallet, "流水链式断裂", fmt.Sprintf("tx_id=%d", tx.ID))
            }
        }

        // 约束④⑤
        if wallet.Balance < 0 {
            report.AddMismatch(wallet, "余额为负", fmt.Sprintf("balance=%d", wallet.Balance))
        }
        if wallet.FrozenAmount < 0 {
            report.AddMismatch(wallet, "冻结为负", fmt.Sprintf("frozen=%d", wallet.FrozenAmount))
        }
    }

    // 有不一致 → 告警
    if report.HasMismatch() {
        s.alertService.SendUrgentAlert(ctx, "对账异常", report.String())
    }

    return report, nil
}
```

## 议题四：提现冻结三阶段状态机

### 4.1 不做会怎样

```
场景A：冻结了但既没解冻也没扣款（状态悬空）
  → 用户的钱永远被冻着 → 投诉 → 资损

场景B：解冻和确认扣款都执行了
  → 钱先退回来又被扣走 → balance变负数 → 数据混乱

场景C：没冻结就扣款
  → 从可用余额扣 → 但可用余额可能已被用户花掉 → 余额不足
```

### 4.2 方案：双字段分离 + 订单状态机 + 互斥校验

```
钱包表设计：
  balance       → 可用余额（用户可以花的钱）
  frozen_amount → 冻结余额（提现中锁定的钱）
  总资产 = balance + frozen_amount

三阶段状态流转（不可逆）：

  ┌─────────────┐        ┌──────────────┐
  │  Freeze      │        │  Unfreeze    │
  │  冻结         │───────→│  解冻退回     │
  │              │  失败   │              │
  │ balance -=X  │        │ balance +=X  │
  │ frozen  +=X  │        │ frozen  -=X  │
  └──────┬───────┘        └──────────────┘
         │
         │ 成功
         ▼
  ┌──────────────┐
  │ ConfirmDeduct│
  │ 确认扣款      │
  │              │
  │ frozen  -=X  │
  │ (balance不变) │  ← 余额已在Freeze时减了，这里只减frozen
  └──────────────┘

  互斥关系：Unfreeze 和 ConfirmDeduct 只能二选一
  保证机制：withdrawal_order.status 状态机
```

### 4.3 代码骨架

```go
// Freeze 冻结可用余额 → 冻结余额
func (e *BalanceEngine) Freeze(ctx context.Context, op *BalanceOp) error {
    return e.repo.WithTransaction(func(tx *query.Query) error {

        // 幂等
        if e.alreadyProcessed(tx, op) {
            return nil
        }

        wallet, _ := tx.Wallet.Where(/*...*/).ForUpdate().First()

        // ═══ 核心校验：可用余额够不够冻结 ═══
        available := wallet.Balance - wallet.FrozenAmount
        if available < op.Amount {
            return ErrInsufficientBalance // 6022001
        }

        // ═══ 冻结：balance减少，frozen增加 ═══
        result, _ := tx.Wallet.Where(tx.Wallet.ID.Eq(wallet.ID), tx.Wallet.Version.Eq(wallet.Version)).
            UpdateSimple(
                tx.Wallet.Balance.Value(wallet.Balance - op.Amount),
                tx.Wallet.FrozenAmount.Value(wallet.FrozenAmount + op.Amount),
                tx.Wallet.Version.Value(wallet.Version + 1),
            )
        if result.RowsAffected == 0 {
            return ErrConcurrentConflict
        }

        // 写流水
        return e.writeTransaction(tx, wallet, op, -(op.Amount))
    })
}

// ConfirmDeduct 确认扣除冻结余额（资金永久移出平台）
func (e *BalanceEngine) ConfirmDeduct(ctx context.Context, op *BalanceOp) error {
    return e.repo.WithTransaction(func(tx *query.Query) error {

        if e.alreadyProcessed(tx, op) {
            return nil
        }

        wallet, _ := tx.Wallet.Where(/*...*/).ForUpdate().First()

        // ═══ 核心校验：冻结余额够不够扣 ═══
        if wallet.FrozenAmount < op.Amount {
            return ErrInsufficientFrozen // 6022004
        }

        // ═══ 互斥校验：提现订单状态必须允许确认扣款 ═══
        order, _ := tx.WithdrawalOrder.Where(tx.WithdrawalOrder.OrderNo.Eq(op.OrderNo)).First()
        if order.Status != OrderStatusProcessing && order.Status != OrderStatusPaying {
            return ErrOrderStatusInvalid // 6022003 (已解冻/已完成)
        }

        // ═══ 确认扣款：只减frozen，balance不变 ═══
        result, _ := tx.Wallet.Where(tx.Wallet.ID.Eq(wallet.ID), tx.Wallet.Version.Eq(wallet.Version)).
            UpdateSimple(
                tx.Wallet.FrozenAmount.Value(wallet.FrozenAmount - op.Amount),
                tx.Wallet.Version.Value(wallet.Version + 1),
            )
        if result.RowsAffected == 0 {
            return ErrConcurrentConflict
        }

        // 更新订单终态
        tx.WithdrawalOrder.Where(tx.WithdrawalOrder.OrderNo.Eq(op.OrderNo)).
            UpdateSimple(tx.WithdrawalOrder.Status.Value(OrderStatusCompleted))

        return e.writeTransaction(tx, wallet, op, -(op.Amount))
    })
}

// Unfreeze 解冻退回（审核拒绝/出款失败）
func (e *BalanceEngine) Unfreeze(ctx context.Context, op *BalanceOp) error {
    return e.repo.WithTransaction(func(tx *query.Query) error {

        if e.alreadyProcessed(tx, op) {
            return nil
        }

        wallet, _ := tx.Wallet.Where(/*...*/).ForUpdate().First()

        // ═══ 互斥校验：不能是已完成的订单 ═══
        order, _ := tx.WithdrawalOrder.Where(tx.WithdrawalOrder.OrderNo.Eq(op.OrderNo)).First()
        if order.Status == OrderStatusCompleted {
            return ErrOrderAlreadyCompleted // 已扣款，不能再解冻
        }
        if order.Status == OrderStatusRefunded || order.Status == OrderStatusRejected {
            return nil // 已解冻过，幂等返回
        }

        // ═══ 解冻：balance恢复，frozen减少 ═══
        result, _ := tx.Wallet.Where(tx.Wallet.ID.Eq(wallet.ID), tx.Wallet.Version.Eq(wallet.Version)).
            UpdateSimple(
                tx.Wallet.Balance.Value(wallet.Balance + op.Amount),
                tx.Wallet.FrozenAmount.Value(wallet.FrozenAmount - op.Amount),
                tx.Wallet.Version.Value(wallet.Version + 1),
            )
        if result.RowsAffected == 0 {
            return ErrConcurrentConflict
        }

        // 更新订单终态
        tx.WithdrawalOrder.Where(tx.WithdrawalOrder.OrderNo.Eq(op.OrderNo)).
            UpdateSimple(tx.WithdrawalOrder.Status.Value(OrderStatusRejected))

        return e.writeTransaction(tx, wallet, op, op.Amount) // 正数=退回
    })
}
```

## 议题五：跨服务最终一致性

### 5.1 不做会怎样

```
场景：财务验签成功后调钱包CreditDeposit → RPC超时 → 实际已入账
  财务认为失败 → 不重试 → 钱包已入账但财务状态还是"待入账"
  → 用户看到钱到了，但财务对不上账

场景：钱包冻结成功 → 通知财务进入审核 → 通知失败
  → 用户钱冻着但没人审核 → 永远冻结
```

### 5.2 方案：本地事务 + 幂等重试（不用TCC/Saga）

```
为什么不用TCC/Saga：
  ① 钱包和财务之间不存在需要"同时原子提交"的场景
  ② TCC需要每个接口实现Try/Confirm/Cancel三套逻辑，复杂度×3
  ③ Saga的补偿操作本身可能失败，引入"补偿的补偿"问题
  ④ 一个人开发，不引入分布式事务协调器

为什么本地事务+幂等重试就够：
  充值：财务先提交自己的 → 再调钱包 → 失败就重试 → 钱包幂等
  提现：钱包先冻结 → 再通知财务 → 失败就重试 → 最终都能成功
  关键：所有跨服务RPC都是幂等的，重试永远安全
```

### 5.3 代码骨架——财务侧的重试任务（参考）

```
// 这段是财务侧的逻辑，列出来是为了你理解全貌

// RetryUnconfirmedDeposits 定时扫描：充值回调成功但钱包入账未确认的订单
func (s *FinanceRetryTask) RetryUnconfirmedDeposits() {
    // 查找：财务状态=已收款 但 钱包未入账 的订单
    orders := s.orderRepo.FindByStatus("paid_but_not_credited")

    for _, order := range orders {
        // 用相同的order_no重试（钱包侧幂等，不会重复入账）
        resp, err := s.walletRPC.CreditDeposit(&CreditDepositReq{
            OrderNo:      order.WalletOrderNo,
            UserID:       order.UserID,
            Amount:       order.Amount,
        })
        if err == nil && resp.Success {
            s.orderRepo.MarkCredited(order.ID)
        }
        // 失败则下次定时任务继续重试
    }
}
```

## 议题六：金额精度零损失

### 6.1 不做会怎样

```
场景A：用float存金额
  float64: 0.1 + 0.2 = 0.30000000000000004
  → 用户看到余额0.30000000000000004 → 不信任

场景B：汇率计算用float
  1 USDT = 25432.50 VND
  100 USDT → 应该是 2543250 VND
  float64: 100 * 25432.50 = 2543250.0（碰巧对了）
  但: 0.1 USDT * 25432.50 = 2543.2500000000003（错了）

场景C：截断方向错误
  用户兑换得到 99.999 BSB，向上取整=100 → 平台多给了0.001
  百万次操作累计 → 平台亏钱

后果：精度丢失 → 余额显示异常 → 对账不平 → 累积误差 → 资损
```

### 6.2 方案：整数存储 + 统一精度规则

```
存储规则（数据库 + 传输 + 内存，全链路统一）：
  法币（VND/IDR/THB）：× 100   → 保留2位小数精度 → int64
  加密货币（USDT）   ：× 10^8  → 保留8位小数精度 → int64
  平台币（BSB）      ：× 100   → 保留2位小数精度 → int64
  汇率              ：× 10^8  → 统一放大         → int64

计算规则（核心：先乘后除，向零截断）：
  兑换：target = source_amount * rate / 10^8
  截断：结果直接丢弃小数部分（int64自动截断）
  方向：向零截断 = 平台不亏（用户拿到的不会比正确值多）
```

### 6.3 代码骨架

```go
// ═══ 精度常量 ═══
const (
    RateScale    int64 = 100_000_000 // 汇率放大因子 10^8
    FiatScale    int64 = 100          // 法币放大因子（2位小数）
    CryptoScale  int64 = 100_000_000 // 加密币放大因子（8位小数）
    BSBScale     int64 = 100          // 平台币放大因子
)

// GetScaleByCode 根据币种获取精度因子
func GetScaleByCode(currencyCode string) int64 {
    // 实际应从currency_config表读取decimal_digits，这里简化
    switch currencyCode {
    case "USDT":
        return CryptoScale
    case "BSB":
        return BSBScale
    default: // VND, IDR, THB
        return FiatScale
    }
}

// ExchangeCalc 兑换计算（法币→BSB）
// sourceAmount: 法币金额（已放大存储的int64）
// rate: 汇率（×10^8）
// 返回：BSB金额（已放大存储的int64）
func ExchangeCalc(sourceAmount int64, rate int64) int64 {
    // ═══ 先乘后除，避免中间结果溢出 ═══
    // 使用 big.Int 或拆分计算防止int64溢出
    //
    // 简单场景：source * rate / RateScale
    // 但如果 source * rate 超过 int64.Max 就溢出了
    //
    // 安全做法：用math/big
    result := new(big.Int).Mul(big.NewInt(sourceAmount), big.NewInt(rate))
    result.Div(result, big.NewInt(RateScale))
    return result.Int64() // 向零截断（big.Int.Div的行为）
}

// FormatForDisplay 展示层格式化
func FormatForDisplay(amount int64, decimalDigits int) string {
    scale := int64(math.Pow10(decimalDigits))
    integer := amount / scale
    fraction := amount % scale

    // 格式化：加千分位，保留指定小数位
    if decimalDigits == 0 {
        return formatWithThousands(integer)
    }
    return fmt.Sprintf("%s.%0*d", formatWithThousands(integer), decimalDigits, fraction)
}
```

### 6.4 溢出防护

```
int64最大值：9,223,372,036,854,775,807（约9.2×10^18）

最大金额场景评估：
  USDT ×10^8 → 能存到 92,233,720,368 USDT（920亿美元）→ 够用
  VND ×100   → 能存到 92,233,720,368,547,758 VND → 够用
  汇率 ×10^8 → 能表示到 92,233,720,368 → 够用

乘法溢出风险：
  最大金额(~10^18) × 最大汇率(~10^13) = 10^31 → 超过int64
  → 必须用 math/big 做中间计算
```

```go
// Preview 兑换预览：锁定汇率生成快照

func (s *ExchangeService) Execute(ctx context.Context, req *ExchangeExecuteReq) error {

    snapshot := &ExchangeSnapshot{}
    err := s.redis.Get(ctx, "wallet:exchange:snapshot:"+req.SnapshotID, snapshot)
    if err != nil {
        return ErrExchangeRateExpired // 6022301 快照过期，请重新预览
    }

    // ═══ 删除快照（一次性使用，防止同一快照被用两次）═══

    if deleted == 0 {
        return ErrExchangeRateExpired // 并发请求，另一个已使用
    }

    // ═══ 用快照中的汇率执行兑换（不取最新汇率）═══
    return s.executeWithSnapshot(ctx, snapshot)
}
```

## 议题八：兑换事务的多行锁安全

### 8.1 不做会怎样

```
兑换需要在同一事务中操作3个钱包行：
  1. 扣 VND 中心钱包
  2. 加 BSB 中心钱包
  3. 加 BSB 奖励钱包

如果不规范锁顺序：
  事务A：先锁VND中心 → 再锁BSB中心
  事务B：先锁BSB中心 → 再锁VND中心
  → 死锁！两个事务互相等对方释放
```

### 8.2 方案：统一锁顺序

```
规则：同一事务中操作多个钱包行时，按以下顺序排序后依次加锁：
  排序键 = (user_id, currency_code, wallet_type) 字典序

示例（兑换 VND→BSB）：
  锁1：(user_id=100, BSB, type=1)      ← BSB < VND 字典序
  锁2：(user_id=100, BSB, type=2)      ← type=1 < type=2
  锁3：(user_id=100, VND, type=1)      ← VND

所有事务都按这个顺序 → 永远不会交叉 → 零死锁
```

```go
// ExecuteExchange 执行兑换（一个事务操作3个钱包）
func (s *ExchangeService) executeWithSnapshot(ctx context.Context, snap *ExchangeSnapshot) error {
    return s.repo.WithTransaction(func(tx *query.Query) error {

        // ═══ 按规范顺序组织操作（BSB先于VND）═══
        // 操作1：加BSB中心钱包
        if err := s.balanceEngine.AddInTx(tx, ctx, &BalanceOp{
            CurrencyCode: "BSB", WalletType: WalletTypeCenter,
            Amount: snap.TargetAmount, TxType: TxTypeExchangeIn,
            OrderNo: snap.OrderNo,
        }); err != nil {
            return err
        }

        // 操作2：加BSB奖励钱包（赠送部分）
        if snap.BonusAmount > 0 {
            if err := s.balanceEngine.AddInTx(tx, ctx, &BalanceOp{
                CurrencyCode: "BSB", WalletType: WalletTypeReward,
                Amount: snap.BonusAmount, TxType: TxTypeExchangeBonus,
                OrderNo: snap.OrderNo,
            }); err != nil {
                return err
            }
        }

        // 操作3：扣法币中心钱包（最后扣，因为VND > BSB字典序）
        if err := s.balanceEngine.DeductInTx(tx, ctx, &BalanceOp{
            CurrencyCode: snap.CurrencyCode, WalletType: WalletTypeCenter,
            Amount: snap.SourceAmount, TxType: TxTypeExchangeOut,
            OrderNo: snap.OrderNo,
        }); err != nil {
            return err // 余额不足 → 整个事务回滚
        }

        // 创建兑换订单记录
        return tx.ExchangeOrder.Create(&model.ExchangeOrder{/*...*/})
    })
```

```go
// wallet表的fund_source字段维护规则：
// 每次充值到账时更新：
//   充值方式=法币(银行/电子钱包) → fund_source = FIAT(1)
//   充值方式=USDT              → fund_source = CRYPTO(2)
//   已有FIAT再充USDT           → fund_source = MIXED(3)
//   已有CRYPTO再充法币          → fund_source = MIXED(3)
//   从未充过                    → fund_source = NONE(0)

// ═══ 充值入账时更新fund_source ═══
func (s *DepositService) updateFundSource(tx *query.Query, wallet *model.Wallet, depositMethod string) {
    var newSource int8
    isLegalTender := depositMethod == "BANK_TRANSFER" || depositMethod == "E_WALLET"

    switch wallet.FundSource {
    case FundSourceNone:
        if isLegalTender {
            newSource = FundSourceFiat
        } else {
            newSource = FundSourceCrypto
        }
    case FundSourceFiat:
        if !isLegalTender {
            newSource = FundSourceMixed
        } else {
            return // 已经是法币，不变
        }
    case FundSourceCrypto:
        if isLegalTender {
            newSource = FundSourceMixed
        } else {
            return // 已经是加密，不变
        }
    case FundSourceMixed:
        return // 已混合，不变
    }
    tx.Wallet.Where(tx.Wallet.ID.Eq(wallet.ID)).UpdateSimple(tx.Wallet.FundSource.Value(newSource))
}

// ═══ 提现时校验 ═══
func (s *WithdrawalService) validateFundSource(wallet *model.Wallet, withdrawMethod string) error {
    isLegalTender := withdrawMethod == "BANK_TRANSFER" || withdrawMethod == "E_WALLET"

    switch wallet.FundSource {
    case FundSourceNone:
        return ErrNoDepositHistory // 从没充过钱，不能提
    case FundSourceFiat:
        if !isLegalTender {
            return ErrFundSourceMismatch // 法币充的→只能法币提
        }
    case FundSourceCrypto:
        if isLegalTender {
            return ErrFundSourceMismatch // USDT充的→只能USDT提
        }
    case FundSourceMixed:
        if !isLegalTender {
            return ErrFundSourceMismatch // 混合→只能法币提
        }
    }
    return nil // 校验通过
}
```

### 11.1 方案：状态转换表 + 校验函数

```go
// ═══ 充值订单状态机 ═══
// 待支付(1) → 支付中(2) → 已完成(3)
//                        → 已失败(4)
//          → 已关闭(5)   （超时未支付）

// ═══ 提现订单状态机 ═══
// 处理中(1) → 审核中(2) → 出款中(3) → 已完成(4)
//                                    → 已失败(5)
//                       → 已拒绝(6)
//          → 已拒绝(6)

var validTransitions = map[int8][]int8{
    // 充值
    DepositPending:    {DepositProcessing, DepositClosed},
    DepositProcessing: {DepositCompleted, DepositFailed},
    DepositCompleted:  {}, // 终态
    DepositFailed:     {}, // 终态
    DepositClosed:     {}, // 终态
    // 提现
    WithdrawProcessing: {WithdrawAuditing, WithdrawRejected},
    WithdrawAuditing:   {WithdrawPaying, WithdrawRejected},
    WithdrawPaying:     {WithdrawCompleted, WithdrawFailed},
    WithdrawCompleted:  {}, // 终态
    WithdrawFailed:     {}, // 终态
    WithdrawRejected:   {}, // 终态
}

// ValidateTransition 校验状态转换合法性
func ValidateTransition(currentStatus, targetStatus int8) error {
    allowed, ok := validTransitions[currentStatus]
    if !ok {
        return ErrUnknownStatus
    }
    for _, s := range allowed {
        if s == targetStatus {
            return nil
        }
    }
    return fmt.Errorf("非法状态转换: %d → %d (error: 6022003)", currentStatus, targetStatus)
}
```

## 议题十五：高频查询的缓存策略

### 15.1 该缓存什么，不该缓存什么

```
═══ 缓存（变化慢读取频繁）═══

  wallet:currency:list
    内容：所有启用币种列表
    TTL：无（B端修改时主动清除）
    场景：C端几乎每个页面都要读

  wallet:rate:{currency_code}
    内容：该币种的4个汇率值
    TTL：无（汇率同步任务覆盖写入）
    场景：兑换预览、充提金额换算

  wallet:currency:base
    内容：基准币种代码
    TTL：永不过期
    场景：统计报表换算

  wallet:exchange:snapshot:{id}
    内容：兑换汇率快照
    TTL：5分钟
    场景：兑换预览→执行之间的汇率锁定

═══ 不缓存（变化快 或 一致性要求高）═══

  余额：每次查数据库。
    原因：缓存一致性维护成本 > 性能收益。
    数据库单行查询走唯一索引 < 1ms，不值得缓存。

  交易流水：每次查数据库。
    原因：分页查询走索引够快，缓存分页数据复杂。

  订单状态：每次查数据库。
    原因：状态流转频繁（待支付→成功/失败），缓存同步成本高。
```

### 15.2 缓存更新策略代码骨架

```go
// B端修改币种配置后，清除缓存
func (s *CurrencyService) AfterUpdate(ctx context.Context) {
    s.redis.Del(ctx, "wallet:currency:list")
    // 不设TTL，下次C端请求时缓存穿透到DB，重新写入
}

// C端读币种列表时，先读缓存
func (s *CurrencyService) GetEnabledList(ctx context.Context) ([]*model.CurrencyConfig, error) {
    // 读缓存
    var list []*model.CurrencyConfig
    if err := s.redis.Get(ctx, "wallet:currency:list", &list); err == nil {
        return list, nil // cache hit
    }

    // 缓存未命中，查DB
    list, err := s.currencyRepo.FindAllEnabled(ctx)
    if err != nil {
        return nil, err
    }

    // 写回缓存（无TTL，等B端修改时主动清除）
    s.redis.Set(ctx, "wallet:currency:list", list, 0)
    return list, nil
}
```

```sql
CREATE INDEX idx_user_curr_type ON wallet_transaction (user_id, currency_code, tx_type);
```

```go
// ═══ 充值方式查询：降级返回空列表 ═══
func (s *DepositService) GetMethods(ctx context.Context, currencyCode string) ([]*DepositMethod, error) {
    // 设置超时（不让财务RPC拖垮我）
    rpcCtx, cancel := context.WithTimeout(ctx, 3*time.Second)
    defer cancel()

    methods, err := s.financeRPC.GetAvailableDepositMethods(rpcCtx, currencyCode)
    if err != nil {

    }
    return methods, nil
}

// ═══ KYC查询：降级为"未认证"（只能USDT提现）═══
func (s *WithdrawalService) getKYCStatus(ctx context.Context, userID int64) int8 {
```

## 议题十八：事务超时与死锁预防

### 18.1 事务超时

```go
// 所有BalanceEngine操作都加context超时
func (s *SomeService) SomeMethod(ctx context.Context, req *Req) error {
    // ═══ 事务操作设置超时上限 ═══
    txCtx, cancel := context.WithTimeout(ctx, 5*time.Second)
    defer cancel()

    return s.balanceEngine.Add(txCtx, &BalanceOp{/*...*/})
}

// 如果5秒内事务没完成 → context取消 → GORM自动回滚 → 返回错误
// 5秒对于一个事务内3-12条SQL绰绰有余（正常50ms内完成）
// 超时说明有异常（死锁等待/网络问题），及时释放资源
```

### 18.2 死锁预防

```
规则①：同一事务中操作多行时，按固定顺序加锁
  排序键 = (user_id, currency_code, wallet_type) 字典序
  所有代码路径都遵守这个顺序 → 零死锁

规则②：事务中不做外部调用
  ✗ 事务内调用RPC（如果RPC慢，行锁就被长时间持有）
  ✗ 事务内调用Redis（Redis故障会阻塞事务）
  ✗ 事务内调用HTTP
  ✓ 事务内只做数据库操作

规则③：事务尽量短小
  减少锁持有时间 → 减少并发等待 → 减少死锁概率
  每个BalanceEngine操作目标：< 50ms
```


### 19.2 开发阶段检查清单

```
每写一个接口前问自己：

  □ 这个接口修改余额了吗？
    → 是：必须走BalanceEngine，不允许直接UPDATE wallet

  □ 这个接口会被重复调用吗？
    → 是（RPC供他调用）：必须用order_no做幂等
    → 是（C端提交）：看是否需要业务层防重

  □ 这个接口操作多个钱包行吗？
    → 是：按字典序加锁，在同一事务内

  □ 这个接口涉及金额计算吗？
    → 是：用int64，汇率用big.Int做中间计算

  □ 这个接口使用了userId吗？
    → C端：从ctx取，不从参数取
    → RPC：从参数取（内网信任）

  □ 这个接口调用了外部RPC吗？
    → 是：加超时(3s)，加降级策略

  □ 这个接口的事务中有外部调用吗？
    → 有：移到事务外面！事务内只做DB操作

  □ 这个接口返回了敏感数据吗？
    → 是：脱敏后返回（卡号掩码）
```

### 19.3 代码Review时的关键检查项

```
═══ 资金安全红线（任何一条不满足就不能合并）═══

  ✗ 直接UPDATE wallet.balance而不经过BalanceEngine
  ✗ 事务外独立写wallet_transaction（必须和余额在同一事务）
  ✗ RPC写操作没有幂等检查
  ✗ C端接口从请求参数获取userId
  ✗ 金额计算使用float64
  ✗ 事务内调用外部RPC/Redis/HTTP
  ✗ 订单状态变更没有校验当前状态
  ✗ 提现扣款没有校验冻结金额
  ✗ 银行卡号明文存储

═══ 性能底线（不满足需要说明原因）═══

  ✗ 余额查询结果被缓存到Redis
  ✗ 使用OFFSET做深翻页
  ✗ 事务超过100ms没有context超时
  ✗ 外部RPC没有超时设置
```
