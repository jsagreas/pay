## 二、挑战1：幂等性设计 — 重复请求不重复执行

### 2.1 问题本质

网络不可靠。RPC调用可能因超时被重试，三方回调可能因未收到ACK被重发。如果CreditWallet被调两次，用户余额就凭空多了一倍。**这是资金系统的头号天敌。**

### 2.2 哪些操作必须幂等

| 操作 | 幂等键 | 重复调用的正确行为 |
|------|--------|------------------|
| CreditWallet | `{bizType}:{refOrderNo}:{walletType}` | 返回首次结果，不重复上账 |
| DebitWallet | `{bizType}:{refOrderNo}:{walletType}` | 返回首次结果，不重复扣款 |
| FreezeBalance | `freeze:{refOrderNo}` | 返回首次结果，不重复冻结 |
| UnfreezeBalance | `unfreeze:{refOrderNo}` | 返回首次结果，不重复解冻 |
| ConfirmDebit | `confirm:{refOrderNo}` | 返回首次结果，不重复确认 |
| InitWallet | `init:{userId}` | 返回成功，不重复创建 |
| 充值回调 | `callback:{platformOrderNo}` | 返回成功，不重复处理 |

### 2.3 最优方案：INSERT-before-UPDATE + DB唯一约束

```
方案核心思想：
  在事务内，先尝试 INSERT 流水记录（带唯一幂等键），
  如果INSERT因唯一约束冲突失败 → 说明已处理过 → 直接返回，不执行UPDATE余额。
  如果INSERT成功 → 说明是首次 → 继续执行UPDATE余额。

  幂等判断和业务执行在同一个事务内，不存在"判断完还没执行就被另一个请求插进来"的问题。
```

**Go伪代码**：

```go
func (s *WalletRpcService) CreditWallet(ctx context.Context, req *CreditWalletReq) (*CreditWalletResp, error) {
    // 1. 参数校验（事务外）
    if req.Amount <= 0 {
        return nil, errs.ErrInvalidAmount
    }

    // 2. 构造幂等键
    idempotentKey := fmt.Sprintf("%d:%s:%d", req.BizType, req.RefOrderNo, req.WalletType)

    var resp *CreditWalletResp

    // 3. 事务执行
    err := db.Transaction(func(tx *gorm.DB) error {

        // 3a. 悲观锁定钱包记录
        wallet, err := lockWallet(tx, req.UserId, req.CurrencyCode, req.WalletType)
        if err != nil {
            return err
        }

        // 3b. 构造流水记录
        flow := &WalletFlow{
            UserId:        req.UserId,
            CurrencyCode:  req.CurrencyCode,
            WalletType:    req.WalletType,
            Direction:     DirectionIn, // 入账
            BizType:       req.BizType,
            Amount:        req.Amount,
            BeforeBalance: wallet.Balance,
            AfterBalance:  wallet.Balance + req.Amount,
            RefOrderNo:    req.RefOrderNo,
            IdempotentKey: idempotentKey,
            Remark:        req.Remark,
        }

        // 3c. 尝试插入流水（幂等检查点）
        if err := tx.Create(flow).Error; err != nil {
            if isDuplicateKeyError(err) {
                // ★ 已处理过，查出已有流水返回
                existFlow, _ := queryFlowByIdempotentKey(tx, idempotentKey)
                resp = &CreditWalletResp{
                    AfterBalance: existFlow.AfterBalance,
                    FlowId:       existFlow.Id,
                }
                return nil // 幂等返回，不是错误
            }
            return err // 其他DB错误
        }

        // 3d. 更新余额
        result := tx.Model(wallet).
            Update("balance", gorm.Expr("balance + ?", req.Amount))
        if result.Error != nil {
            return result.Error
        }

        resp = &CreditWalletResp{
            AfterBalance: wallet.Balance + req.Amount,
            FlowId:       flow.Id,
        }
        return nil
    })

    return resp, err
}
```

### 2.4 方案对比：为什么不选其他方案

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| **INSERT-before-UPDATE + DB唯一约束** | 原子性天然保证、无额外组件 | 需要处理唯一约束错误 | **✅ 选用** |
| Redis SETNX 预检查 | 速度快 | Redis和DB不在同一事务，存在窗口期：Redis判断未处理→Redis挂→DB已执行 | ❌ 不够可靠 |
| 业务层先SELECT检查 | 简单 | 并发窗口期：两个请求同时SELECT都没查到→都执行INSERT | ❌ 有并发漏洞 |
| 乐观锁（version字段） | 无锁竞争 | wallet_flow是INSERT不是UPDATE，乐观锁不适用 | ❌ 场景不匹配 |

### 2.5 边界条件与陷阱

```
陷阱1：幂等返回值必须和首次一致
  重复调用时不能返回空或错误，必须查出首次操作的结果返回。
  否则调用方以为失败了会继续重试。

陷阱2：isDuplicateKeyError 的判断要准确
  TiDB/MySQL 的唯一约束冲突错误码是 1062 (ErrDupEntry)。
  不能把所有错误都当成幂等，否则真正的DB错误会被吞掉。

陷阱3：幂等键的粒度要对
  错误示例：key = refOrderNo（不区分钱包类型）
  后果：充值上账到中心钱包后，奖金上账到奖励钱包被误判为重复
  正确：key = {bizType}:{refOrderNo}:{walletType}

陷阱4：DELETE软删除后的幂等键冲突
  wallet_flow 的唯一约束是 (idempotent_key, delete_at)。
  delete_at=0 时才生效。如果某天需要物理删除旧流水，
  必须确保幂等键的设计不会跨时间段冲突。
```

### 2.6 幂等键格式规范

```
CreditWallet / DebitWallet:
  key = "{bizType}:{refOrderNo}:{walletType}"
  示例："1:R20260302103000123456:1"  → 充值:订单号:中心钱包
  示例："4:R20260302103000123456:2"  → 奖金:订单号:奖励钱包

FreezeBalance:
  key = "freeze:{refOrderNo}"
  示例："freeze:W20260302120000654321"

UnfreezeBalance:
  key = "unfreeze:{refOrderNo}"
  示例："unfreeze:W20260302120000654321"

ConfirmDebit:
  key = "confirm:{refOrderNo}"
  示例："confirm:W20260302120000654321"

InitWallet:
  key = "init:{userId}:{currencyCode}:{walletType}"
  示例："init:10001:VND:1"
```

## 三、挑战2：并发控制与锁策略 — 同一时刻谁先谁后

### 3.1 问题本质

同一个用户可能同时有多个余额变动操作：
- 充值回调上账 + 游戏扣款同时到达
- 两笔提现同时发起冻结
- 兑换扣法币和加BSB在事务内锁多行

如果不控制并发，读到的余额是过时的，写入时可能超扣或重复加。

### 3.2 最优方案：数据库行级悲观锁（SELECT ... FOR UPDATE）

```go
// lockWallet — 统一的钱包加锁函数，所有余额变动操作都通过这个函数获取钱包记录
func lockWallet(tx *gorm.DB, userId int64, currencyCode string, walletType int32) (*Wallet, error) {
    var wallet Wallet
    err := tx.Set("gorm:query_option", "FOR UPDATE").
        Where("user_id = ? AND currency_code = ? AND wallet_type = ? AND delete_at = 0",
            userId, currencyCode, walletType).
        First(&wallet).Error
    if err != nil {
        if errors.Is(err, gorm.ErrRecordNotFound) {
            return nil, errs.ErrWalletNotFound
        }
        return nil, err
    }
    return &wallet, nil
}
```

### 3.3 方案对比：为什么不选其他方案

| 方案 | 优点 | 缺点 | 适用场景 | 结论 |
|------|------|------|---------|------|
| **DB行级悲观锁** | 和事务天然结合、实现简单、TiDB支持好 | 锁等待期间占DB连接 | 单用户并发不高、操作必须在事务内 | **✅ 选用** |
| Redis分布式锁 | 锁获取快、不占DB连接 | 和DB事务不在一个层面，有窗口期；锁超时难设 | 跨DB/跨服务的协调 | ❌ 画蛇添足 |
| 乐观锁(version) | 无锁等待 | 冲突后需重试，充值高峰期同用户多笔回调→重试雪崩 | 冲突率低的场景 | ❌ 风险高 |
| 无锁(CAS原子更新) | 极高性能 | 无法保证 before_balance 的正确性 | 不需要记录前后余额的场景 | ❌ 不满足需求 |

**为什么悲观锁不会成为瓶颈？**

```
性能分析：
  - 锁的粒度：单用户 × 单币种 × 单钱包类型 = 一行
  - 不同用户的操作锁不同行，互不影响
  - 同一用户的余额变动本来就应该串行（不然before_balance没意义）
  - 单个事务持锁时间：INSERT flow + UPDATE wallet ≈ 2-5ms
  - 单用户并发峰值：即使10个并发，排队也只需50ms

  不是秒杀场景。用户不会每秒充值100次。悲观锁完全够用。
```

### 3.4 关键规则：加锁顺序（防死锁）

兑换操作需要在一个事务内锁2-3个钱包记录。如果不同事务以不同顺序加锁，会死锁。

```
死锁场景（错误示例）：
  事务A：锁VND中心 → 锁BSB中心（兑换：扣VND加BSB）
  事务B：锁BSB中心 → 锁VND中心（另一笔兑换，顺序反了）
  → 事务A持有VND锁等BSB锁，事务B持有BSB锁等VND锁 → 死锁

正确方案：全局统一加锁顺序
```

```go
// lockWallets — 按固定顺序锁定多个钱包（防死锁）
// 排序规则：先按 currency_code 字母序，再按 wallet_type 数字序
func lockWallets(tx *gorm.DB, userId int64, targets []WalletTarget) ([]*Wallet, error) {
    // 1. 排序（固定顺序 = 不死锁）
    sort.Slice(targets, func(i, j int) bool {
        if targets[i].CurrencyCode != targets[j].CurrencyCode {
            return targets[i].CurrencyCode < targets[j].CurrencyCode
        }
        return targets[i].WalletType < targets[j].WalletType
    })

    // 2. 按排序后的顺序逐个加锁
    wallets := make([]*Wallet, 0, len(targets))
    for _, t := range targets {
        w, err := lockWallet(tx, userId, t.CurrencyCode, t.WalletType)
        if err != nil {
            return nil, err
        }
        wallets = append(wallets, w)
    }
    return wallets, nil
}
```

```
兑换操作的加锁顺序示例（VND→BSB）：
  排序后：BSB(1) → BSB(2) → VND(1)
  即：先锁BSB中心 → 再锁BSB奖励 → 最后锁VND中心
  不管是VND→BSB还是THB→BSB，只要币种代码不变，加锁顺序永远一致。
```

### 3.5 事务超时设置

```go

// 超过5秒自动回滚，防止长事务占锁

    Context: context.WithTimeout(ctx, 5*time.Second),
```

## 四、挑战3：防资金异常 — 余额绝对不能为负

### 4.1 问题本质

扣款操作（DebitWallet、FreezeBalance）如果校验不严，可能导致余额为负。一旦余额为负，所有后续业务逻辑都会混乱。

### 4.2 最优方案：三层防线

```
第一层：代码层 — 读余额后判断
  wallet := lockWallet(...)
  available := wallet.Balance - wallet.FrozenAmount
  if available < req.Amount {
      return errs.ErrInsufficientBalance  // 明确拒绝
  }

第二层：SQL层 — WHERE条件兜底
  UPDATE wallet
  SET balance = balance - ?
  WHERE id = ? AND balance >= ?
  // 如果被并发修改导致balance < amount，WHERE不匹配，RowsAffected=0

第三层：事务层 — 悲观锁防止读到过期余额
  FOR UPDATE 确保读到的balance是最新的，
  读和写在同一个事务内，中间不会有其他操作修改balance。
```

**Go伪代码**：

```go
func (s *WalletRpcService) DebitWallet(ctx context.Context, req *DebitWalletReq) (*DebitWalletResp, error) {
    // ... 幂等键构造 ...

    err := db.Transaction(func(tx *gorm.DB) error {
        // 第三层：悲观锁
        wallet, err := lockWallet(tx, req.UserId, req.CurrencyCode, req.WalletType)
        if err != nil {
            return err
        }

        // 第一层：代码层校验
        available := wallet.Balance - wallet.FrozenAmount
        if available < req.Amount {
            return errs.ErrInsufficientBalance
        }

        // ... 插入流水（幂等检查）...

        // 第二层：SQL层兜底
        result := tx.Model(&Wallet{}).
            Where("id = ? AND balance >= ?", wallet.Id, req.Amount).
            Update("balance", gorm.Expr("balance - ?", req.Amount))

        if result.RowsAffected == 0 {
            // 理论上不应该走到这里（因为有悲观锁），但作为最终安全网
            return errs.ErrInsufficientBalance
        }

        return nil
    })
    return resp, err
}
```

### 4.3 为什么需要三层都有

```
只有第一层（代码层）：
  并发场景下，两个请求同时读到 balance=100，各自扣80，
  两个都通过了代码校验，最终 balance = -60。

只有第二层（SQL层）：
  WHERE balance >= 80 确实能防止超扣，但此时 before_balance 已经写错了
  （流水中记录的 before_balance 可能不是最新的），审计不通过。

只有第三层（悲观锁）：
  悲观锁保证了读写串行化，但如果代码里忘记判断余额直接扣了，
  还是会出现负数（悲观锁只防并发，不防逻辑错误）。

三层组合才是水密的：
  第三层保证读到最新值 → 第一层做业务判断 → 第二层是最终安全网。
```

## 五、挑战4：事务边界划分 — 什么必须在一起，什么可以分开

### 5.1 核心原则

```
原则：余额变动 + 流水写入 必须在同一个事务内。
      余额变了但流水没写 = 无法审计 = 资金黑洞。
      流水写了但余额没变 = 幂等键已占用但操作没生效 = 后续请求被误拒。
```

### 5.2 事务内 vs 事务外：明确划分

**必须在事务内的操作（原子性要求）**：

| 操作 | 事务内必须包含 | 原因 |
|------|-------------|------|
| CreditWallet | SELECT FOR UPDATE + INSERT flow + UPDATE balance | 三者缺一不可 |
| DebitWallet | SELECT FOR UPDATE + 余额校验 + INSERT flow + UPDATE balance | 校验和更新必须基于同一份数据 |
| FreezeBalance | SELECT FOR UPDATE + 余额校验 + INSERT freeze_detail + UPDATE frozen_amount | 冻结明细和冻结金额必须一致 |
| UnfreezeBalance | SELECT freeze_detail + SELECT FOR UPDATE + UPDATE frozen_amount + UPDATE freeze_detail.status | 还原必须精确 |
| ConfirmDebit | SELECT freeze_detail + SELECT FOR UPDATE + UPDATE balance + UPDATE frozen_amount + INSERT flow + UPDATE freeze_detail.status | 正式扣款必须原子 |
| doExchange | 扣法币(lock+flow+update) + 加BSB(lock+flow+update) + exchange_record | 多钱包原子 |

**可以在事务外的操作（不影响资金安全）**：

| 操作 | 为什么可以事务外 |
|------|-----------------|
| 更新 recharge_order.status | 订单状态是信息性的，即使更新失败，余额已经正确变动，可以通过对账修复 |
| 更新 withdraw_order.status | 同上 |
| RPC调用 ser-blog.AddActLog | 异步oneway，审计日志不影响资金操作 |
| 删除Redis缓存 | 缓存不一致最多导致短暂展示旧数据，不影响资金 |
| 发送MQ/NATS通知 | 通知失败可重发，不影响已完成的资金操作 |

### 5.3 事务边界代码模板

```go
func (s *RechargeService) HandleRechargeCallback(ctx context.Context, req *CallbackReq) error {
    // ========== 事务外：前置校验 ==========
    order, err := s.repo.GetRechargeOrder(ctx, req.PlatformOrderNo)
    if err != nil {
        return err
    }
    if order.Status != OrderStatusPending {
        return nil // 已处理，幂等返回
    }

    // ========== 事务内：资金操作（原子） ==========
    err = s.db.Transaction(func(tx *gorm.DB) error {
        // CreditWallet 主金额
        if err := s.creditWalletInTx(tx, CreditReq{
            UserId: order.UserId, CurrencyCode: order.CurrencyCode,
            WalletType: WalletTypeCenter, Amount: req.ActualAmount,
            BizType: BizTypeRecharge, RefOrderNo: order.OrderNo,
        }); err != nil {
            return err
        }

        // CreditWallet 奖金（如有）
        if order.BonusAmount > 0 {
            if err := s.creditWalletInTx(tx, CreditReq{
                WalletType: WalletTypeReward, Amount: order.BonusAmount,
                BizType: BizTypeBonus, RefOrderNo: order.OrderNo + ":bonus",
            }); err != nil {
                return err
            }
        }
        return nil
    })

    if err != nil {
        return err
    }

    // ========== 事务外：后续处理（允许失败，可重试/修复） ==========

    // 更新订单状态（非原子，失败不影响余额）
    _ = s.repo.UpdateRechargeOrderStatus(ctx, order.OrderNo, OrderStatusSuccess)

    // 删除相关缓存
    _ = s.cache.DeleteUserBalanceCache(ctx, order.UserId, order.CurrencyCode)

    return nil
}
```

### 5.4 关键陷阱：事务内不能有RPC调用

```
❌ 错误做法：
  db.Transaction(func(tx) {
      wallet := lockWallet(tx, ...)        // 拿到锁
      rate := rpc.WalletClient().GetExchangeRate(ctx, ...)  // RPC调用！
      // ... 使用rate计算 ...
      tx.Update(wallet, ...)               // 更新余额
  })

  问题：RPC调用可能耗时数秒，期间悲观锁一直持有，阻塞其他操作。
        如果RPC超时，事务超时回滚，但锁已经占用了很久。

✅ 正确做法：
  // 事务外获取汇率
  rate := rpc.WalletClient().GetExchangeRate(ctx, ...)

  // 事务内只做DB操作
  db.Transaction(func(tx) {
      wallet := lockWallet(tx, ...)
      // ... 使用rate计算 ...
      tx.Update(wallet, ...)
  })

原则：事务越短越好。所有非DB操作（RPC、Redis、计算）都放在事务外。
     事务内只有：SELECT FOR UPDATE → INSERT → UPDATE。
```

## 六、挑战5：冻结三阶段一致性 — 冻多少解多少扣多少

### 6.1 问题本质

提现冻结可能跨两个钱包（中心80 + 奖励20 = 冻结100）。解冻时必须精确还原（中心解冻80 + 奖励解冻20），不能多不能少。如果冻结明细记录丢失或不准确，解冻/确认扣款时就无法正确执行。

### 6.2 最优方案：freeze_detail 表精确记录 + 状态互斥校验

**冻结操作的完整逻辑**：

```go
func (s *WalletRpcService) FreezeBalance(ctx context.Context, req *FreezeReq) error {
    return s.db.Transaction(func(tx *gorm.DB) error {
        // 1. 幂等检查
        exists, err := checkFreezeExists(tx, req.RefOrderNo)
        if exists {
            return nil // 已冻结，幂等返回
        }

        remainAmount := req.Amount
        var details []*FreezeDetail

        // 2. 先冻中心钱包
        centerWallet, err := lockWallet(tx, req.UserId, req.CurrencyCode, WalletTypeCenter)
        if err != nil {
            return err
        }
        centerAvailable := centerWallet.Balance - centerWallet.FrozenAmount

        if centerAvailable >= remainAmount {
            // 中心够冻
            details = append(details, &FreezeDetail{
                UserId: req.UserId, CurrencyCode: req.CurrencyCode,
                WalletType: WalletTypeCenter, OrderNo: req.RefOrderNo,
                FreezeAmount: remainAmount, Status: FreezeStatusFrozen,
            })
            centerWallet.FrozenAmount += remainAmount
            remainAmount = 0
        } else if centerAvailable > 0 {
            // 中心不够，冻光中心可用部分
            details = append(details, &FreezeDetail{
                WalletType: WalletTypeCenter, FreezeAmount: centerAvailable,
                OrderNo: req.RefOrderNo, Status: FreezeStatusFrozen,
                // ... 其他字段 ...
            })
            centerWallet.FrozenAmount += centerAvailable
            remainAmount -= centerAvailable
        }
        // 中心可用=0则不冻中心，全部由奖励承担

        // 3. 如果还有剩余，冻奖励钱包
        if remainAmount > 0 {
            rewardWallet, err := lockWallet(tx, req.UserId, req.CurrencyCode, WalletTypeReward)
            if err != nil {
                return err
            }
            rewardAvailable := rewardWallet.Balance - rewardWallet.FrozenAmount

            if rewardAvailable < remainAmount {
                return errs.ErrInsufficientBalance // 中心+奖励都不够
            }

            details = append(details, &FreezeDetail{
                WalletType: WalletTypeReward, FreezeAmount: remainAmount,
                OrderNo: req.RefOrderNo, Status: FreezeStatusFrozen,
                // ... 其他字段 ...
            })
            rewardWallet.FrozenAmount += remainAmount
        }

        // 4. 批量写入冻结明细
        if err := tx.Create(&details).Error; err != nil {
            return err
        }

        // 5. 更新钱包冻结金额
        for _, d := range details {
            tx.Model(&Wallet{}).
                Where("user_id = ? AND currency_code = ? AND wallet_type = ?",
                    req.UserId, req.CurrencyCode, d.WalletType).
                Update("frozen_amount", gorm.Expr("frozen_amount + ?", d.FreezeAmount))
        }

        return nil
    })
}
```

**解冻操作——按明细精确还原**：

```go
func (s *WalletRpcService) UnfreezeBalance(ctx context.Context, req *UnfreezeReq) error {
    return s.db.Transaction(func(tx *gorm.DB) error {
        // 1. 查冻结明细
        var details []FreezeDetail
        tx.Where("order_no = ? AND status = ? AND delete_at = 0",
            req.RefOrderNo, FreezeStatusFrozen).Find(&details)

        if len(details) == 0 {
            return nil // 已解冻或不存在，幂等返回
        }

        // 2. 逐条还原
        for _, d := range details {
            wallet, err := lockWallet(tx, d.UserId, d.CurrencyCode, d.WalletType)
            if err != nil {
                return err
            }

            // 还原冻结金额
            wallet.FrozenAmount -= d.FreezeAmount
            if wallet.FrozenAmount < 0 {
                wallet.FrozenAmount = 0 // 防御性编程，不应该出现
            }

            tx.Model(wallet).Update("frozen_amount", wallet.FrozenAmount)

            // 更新明细状态
            tx.Model(&d).Update("status", FreezeStatusUnfrozen)
        }

        // 3. 写解冻流水
        // ... INSERT wallet_flow direction=4(解冻) ...

        return nil
    })
}
```

### 6.3 互斥校验：UnfreezeBalance 和 ConfirmDebit 不可同时执行

```go
// 在 UnfreezeBalance 和 ConfirmDebit 开头都加这个校验
func checkFreezeDetailStatus(tx *gorm.DB, orderNo string) ([]FreezeDetail, error) {
    var details []FreezeDetail
    tx.Where("order_no = ? AND delete_at = 0", orderNo).Find(&details)

    for _, d := range details {
        switch d.Status {
        case FreezeStatusFrozen:
            continue // 正常，可以操作
        case FreezeStatusConfirmed:
            return nil, errs.ErrAlreadyConfirmed // 已确认扣款，不能再解冻
        case FreezeStatusUnfrozen:
            return nil, nil // 已解冻，幂等返回
        }
    }
    return details, nil
}
```

### 6.4 校验不变式（每次操作后都应该满足的条件）

```
不变式1：wallet.frozen_amount = SUM(freeze_detail.freeze_amount WHERE status=1)
  即钱包的冻结金额 = 该钱包所有"冻结中"明细的冻结金额之和。
  任何操作后如果这个等式不成立，说明有bug。

不变式2：wallet.balance >= wallet.frozen_amount >= 0
  余额永远 >= 冻结金额 >= 0。
  如果 frozen_amount > balance，说明冻结逻辑有bug（冻了比余额还多的钱）。

不变式3：一笔订单的 freeze_detail 只能全部是 status=1 或全部是 status=2 或全部是 status=3
  不允许同一订单的明细处于不同终态。
```

## 七、挑战6：数据精度与计算安全 — 一分钱都不能算错

### 7.1 问题本质

浮点数计算有精度损失。0.1 + 0.2 ≠ 0.3 在几乎所有编程语言中都成立。钱包系统如果用浮点数，迟早会算错一分钱，积少成多就是大事故。

### 7.2 最优方案：全链路BIGINT整数运算

```
存储规范：所有金额以最小单位整数存储

| 币种 | 最小单位 | 精度系数 | 示例 |
|------|---------|---------|------|
| VND  | 1 VND   | 1       | 100万VND = 存1000000 |
| IDR  | 1 IDR   | 1       | 100万IDR = 存1000000 |
| THB  | 0.01 THB| 100     | 150.50 THB = 存15050 |
| BSB  | 0.01 BSB| 100     | 100.00 BSB = 存10000 |
| USDT | 10^-8   | 10^8    | 1 USDT = 存100000000 |

存储类型：BIGINT（8字节，最大约 9.2 × 10^18）
运算方式：全部整数加减乘除，零精度损失
展示转换：在API返回前统一调用 amountToDisplay() 转为前端展示格式
```

### 7.3 汇率计算：乘除顺序决定精度

```
汇率存储：×10^8 转整数
  1 USDT = 25000 VND → rate = 2500000000000 (25000 × 10^8)
  1 BSB = 0.1 USDT  → rate = 10000000 (0.1 × 10^8)

兑换计算公式：
  toAmount = fromAmount × rate / RATE_PRECISION

  其中 RATE_PRECISION = 10^8 = 100000000
```

**关键：先乘后除，不能先除后乘**

```go
const RatePrecision = 100_000_000 // 10^8

// ✅ 正确：先乘后除
func ConvertAmount(fromAmount int64, rate int64) int64 {
    // 1. 先乘（可能很大，但BIGINT能hold住）
    product := fromAmount * rate
    // 2. 后除（向下取整）
    return product / RatePrecision
}

// ❌ 错误：先除后乘
func ConvertAmountWrong(fromAmount int64, rate int64) int64 {
    // 先除会丢失精度
    partial := fromAmount / RatePrecision  // 如果fromAmount < 10^8，这里直接=0
    return partial * rate
}
```

### 7.4 溢出防护

```
溢出风险分析：
  fromAmount 最大值：假设用户有10亿VND = 1,000,000,000
  rate 最大值：汇率 25000 × 10^8 = 2,500,000,000,000

  product = 1,000,000,000 × 2,500,000,000,000 = 2.5 × 10^21

  BIGINT 最大值 ≈ 9.2 × 10^18

  溢出了！
```

**解决方案**：用 `math/big` 或先除后乘的安全版本

```go
import "math/big"

func SafeConvertAmount(fromAmount int64, rate int64) int64 {
    a := new(big.Int).SetInt64(fromAmount)
    r := new(big.Int).SetInt64(rate)
    p := new(big.Int).SetInt64(RatePrecision)

    // a * r / p
    result := new(big.Int).Mul(a, r)
    result.Div(result, p)

    if !result.IsInt64() {
        // 溢出，不应该出现在正常业务中
        return 0 // 或panic/log
    }
    return result.Int64()
}
```

**或者：分步计算避免溢出**

```go
func SafeConvertAmountV2(fromAmount int64, rate int64) int64 {
    // 将rate拆成 整数部分 + 小数部分
    // rate = 2500000000000 = 25000 × 10^8
    // intPart = rate / RatePrecision = 25000
    // fracPart = rate % RatePrecision = 0
    intPart := rate / RatePrecision
    fracPart := rate % RatePrecision

    result := fromAmount * intPart
    if fracPart > 0 {
        result += fromAmount * fracPart / RatePrecision
    }
    return result
}
```

### 7.5 取整方向规范

```
规则：所有对用户不利的取整一律向下取整（截断），所有对用户有利的取整一律向上取整。

实际操作中的简化规则：
  - 兑换：toAmount 向下取整（用户少得一点，平台兜底差额）
  - 手续费：fee 向上取整（用户多付一点手续费）
  - 赠送：bonus 向下取整（平台少送一点）

原因：
  向下取整 = 截断 = Go 整数除法的默认行为
  向上取整 = (a + b - 1) / b

代码：
```

```go
// 向下取整（默认，Go整数除法天然行为）
func FloorDiv(a, b int64) int64 {
    return a / b
}

// 向上取整
func CeilDiv(a, b int64) int64 {
    return (a + b - 1) / b
}

// 手续费计算：fee = amount × feeRate / 10000（向上取整）
func CalcFee(amount int64, feeRate int32) int64 {
    return CeilDiv(amount * int64(feeRate), 10000)
}
```

### 7.6 展示转换工具

```go
// amountToDisplay — 将最小单位整数转为展示字符串
// amount=15050, precisionFactor=100, decimalDigits=2, thousandsSep=',', decimalSep='.'
// → "150.50"
func amountToDisplay(amount int64, cfg *CurrencyConfig) string {
    intPart := amount / cfg.PrecisionFactor
    fracPart := amount % cfg.PrecisionFactor

    intStr := formatWithThousandsSep(intPart, cfg.ThousandsSep)
    if cfg.DecimalDigits == 0 {
        return intStr
    }

    fracStr := fmt.Sprintf("%0*d", cfg.DecimalDigits, fracPart)
    return intStr + string(cfg.DecimalSep) + fracStr
}
```

## 八、挑战7：状态机设计 — 只允许合法的状态流转

### 8.1 问题本质

提现订单有7种状态。如果代码里随意更新状态（比如从"出款成功"更新回"待风控"），数据就混乱了。必须用状态机约束合法的转换路径。

### 8.2 提现订单状态机

```
合法转换矩阵（行=当前状态，列=目标状态，✓=允许，✗=禁止）：

              → 1待风控  2风控驳回  3待财务  4财务驳回  5待出款  6出款失败  7出款成功
1.待风控       ✗        ✓          ✓        ✗         ✗       ✗         ✗
2.风控驳回     ✗        ✗          ✗        ✗         ✗       ✗         ✗  (终态)
3.待财务       ✗        ✗          ✗        ✓         ✓       ✗         ✗
4.财务驳回     ✗        ✗          ✗        ✗         ✗       ✗         ✗  (终态)
5.待出款       ✗        ✗          ✗        ✗         ✗       ✓         ✓
6.出款失败     ✗        ✗          ✗        ✗         ✗       ✗         ✗  (终态)
7.出款成功     ✗        ✗          ✗        ✗         ✗       ✗         ✗  (终态)
```

### 8.3 代码实现：状态机校验器

```go
// 提现订单状态常量
const (
    WithdrawStatusPendingRisk    = 1 // 待风控
    WithdrawStatusRiskRejected   = 2 // 风控驳回（终态）
    WithdrawStatusPendingFinance = 3 // 待财务
    WithdrawStatusFinRejected    = 4 // 财务驳回（终态）
    WithdrawStatusPendingPayout  = 5 // 待出款
    WithdrawStatusPayoutFailed   = 6 // 出款失败（终态）
    WithdrawStatusPayoutSuccess  = 7 // 出款成功（终态）
)

// validTransitions — 合法的状态转换映射
var validWithdrawTransitions = map[int32][]int32{
    WithdrawStatusPendingRisk:    {WithdrawStatusRiskRejected, WithdrawStatusPendingFinance},
    WithdrawStatusPendingFinance: {WithdrawStatusFinRejected, WithdrawStatusPendingPayout},
    WithdrawStatusPendingPayout:  {WithdrawStatusPayoutFailed, WithdrawStatusPayoutSuccess},
    // 终态不允许任何转换
}

func CanTransition(currentStatus, targetStatus int32) bool {
    allowedTargets, ok := validWithdrawTransitions[currentStatus]
    if !ok {
        return false // 当前状态是终态或未知状态
    }
    for _, t := range allowedTargets {
        if t == targetStatus {
            return true
        }
    }
    return false
}

// 使用方式：每次更新状态前必须校验
func (s *WithdrawService) UpdateOrderStatus(ctx context.Context, orderNo string, newStatus int32) error {
    order, err := s.repo.GetWithdrawOrder(ctx, orderNo)
    if err != nil {
        return err
    }

    if !CanTransition(order.Status, newStatus) {
        return fmt.Errorf("非法状态转换: %d → %d, orderNo=%s", order.Status, newStatus, orderNo)
    }

    // 用 WHERE 条件兜底：只有当前状态匹配才更新
    result := s.db.Model(&WithdrawOrder{}).
        Where("order_no = ? AND status = ?", orderNo, order.Status).
        Update("status", newStatus)

    if result.RowsAffected == 0 {
        return errs.ErrStatusChanged // 被并发修改了
    }
    return nil
}
```

## 九、挑战8：缓存一致性 — 改了配置用户立刻生效

### 9.1 问题本质

币种配置和汇率通过Redis缓存提升读性能。但B端修改了配置后，如果缓存不更新，C端用户看到的还是旧数据。

### 9.2 最优方案：Cache-Aside + 先写DB后删缓存

```
写操作流程（B端修改币种配置）：
  1. 写入DB（UPDATE currency_config ...）
  2. 删除相关Redis缓存（DEL wallet:currency:list, DEL wallet:currency:{code}）
  3. 下次读请求发现缓存不存在 → 回源DB → 写入缓存

读操作流程（C端查币种列表）：
  1. 读Redis缓存（GET wallet:currency:list）
  2. 缓存命中 → 直接返回
  3. 缓存未命中 → 查DB → 写入Redis（SET ... EX 3600）→ 返回
```

### 9.3 为什么是"先写DB后删缓存"，不是其他方案

| 方案 | 问题 | 结论 |
|------|------|------|
| **先写DB后删缓存** | 极端情况下有短暂不一致（删缓存后、新请求重建缓存前，另一个旧请求写入旧缓存）。但概率极低且窗口极短。 | **✅ 选用（工业标准）** |
| 先删缓存后写DB | 删完缓存、写DB前，另一个请求读到DB旧数据写入缓存 → 缓存长期不一致 | ❌ 有明显漏洞 |
| 写DB后更新缓存 | 并发写时可能把旧数据覆盖新数据（A先改、B后改，但B的缓存先写入、A的后写入 → 缓存是A的旧数据） | ❌ 有并发问题 |
| 双删策略 | 先删缓存→写DB→延迟再删一次。可以解决第一种方案的极端问题，但引入延迟删除的复杂度。 | 可选（对我们场景不需要） |

### 9.4 缓存击穿/穿透/雪崩防护

```
缓存击穿（热点key过期瞬间大量请求打DB）：
  场景：wallet:currency:list 过期，同时100个请求查币种列表
  方案：singleflight — 同一个key同一时刻只允许一个请求回源DB
```

```go
import "golang.org/x/sync/singleflight"
```

## 十、挑战9：跨模块最终一致性 — 我改了你没改怎么办

### 10.1 问题本质

充值链路：ser-finance 调 ser-wallet.CreditWallet 成功了，但 ser-finance 更新自己的订单状态失败了。结果：用户余额加了，但 ser-finance 的记录还显示"待处理"。

提现链路：ser-wallet 冻结成功了，但通知 ser-finance 失败了。结果：用户余额冻结了，但 ser-finance 不知道有这笔订单。

### 10.2 核心原则：钱包只管余额，不管对方的事

```
设计哲学：
  1. ser-wallet 的 CreditWallet/DebitWallet/Freeze/Unfreeze/Confirm
     只负责"余额变动+流水写入"的原子性。
  2. 调用方（ser-finance）自己负责"调用钱包成功后更新自己的状态"。
  3. 如果调用方更新自己的状态失败了，那是调用方的问题，
     调用方应该有自己的重试机制或补偿机制。

  我们的接口是幂等的，所以调用方可以安全重试。
  这是最简洁、最清晰的责任划分。
```

### 10.3 场景分析与应对

**场景1：充值回调 — ser-finance调CreditWallet成功，但ser-finance自己状态更新失败**

```
时间线：
  T1: 三方回调到达 ser-finance
  T2: ser-finance 调 ser-wallet.CreditWallet → 成功（用户余额已增加）
  T3: ser-finance 更新自己的订单状态 → 失败（DB异常）

结果：用户余额已增加（正确），但 ser-finance 记录还是"处理中"

修复方案：
  - ser-finance 应有重试机制：T3失败后重试更新状态
  - 如果 ser-finance 重试时又调了一次 CreditWallet → 幂等返回，不重复上账
  - 最终兜底：对账任务扫描"状态不一致"的订单进行修复
```

**场景2：提现冻结 — ser-wallet冻结成功，但通知ser-finance失败**

```
时间线：
  T1: 用户提交提现
  T2: ser-wallet FreezeBalance → 成功（余额已冻结）
  T3: ser-wallet 创建 withdraw_order → 成功
  T4: ser-wallet 通知 ser-finance → 失败（RPC超时/MQ发送失败）

结果：用户余额已冻结，订单已创建，但 ser-finance 不知道

修复方案（关键）：
```

## 十一、挑战10：异常与容错 — 三方挂了/超时了/返回乱了

### 11.1 RPC调用超时处理

| 调用 | 超时设置 | 超时后处理 |
|------|---------|-----------|
| ser-finance.GetRechargeMethodList | 3秒 | 返回"充值功能暂不可用"，不缓存错误结果 |
| ser-finance.MatchChannelAndCreateOrder | 10秒 | **关键**：标记充值订单为"创建失败"，让用户重新提交 |
| ser-finance.GetWithdrawConfig | 3秒 | 返回"提现功能暂不可用" |
| ser-kyc.KycQuery | 3秒 | 返回"暂无法校验KYC"，阻止提现 |
| ser-blog.AddActLog | 1秒 | oneway，超时不影响主流程 |

### 11.2 MatchChannelAndCreateOrder 超时的特殊处理

```
这是最棘手的超时场景：

  T1: ser-wallet 调 ser-finance.MatchChannelAndCreateOrder
  T2: ser-finance 收到请求，调三方API成功，通道订单已创建
  T3: ser-finance 返回响应
  T4: 但 ser-wallet 的 10秒超时已到，认为失败

  结果：三方那边通道订单已创建（钱可能会被扣），
        但 ser-wallet 不知道，标记充值订单为失败。
        如果三方后续回调成功 → CreditWallet 找不到对应订单？

解决方案：
  1. 充值订单创建时，状态 = "待确认"（不是直接"待支付"）
  2. 如果 MatchChannel 超时，标记订单为"通道确认中"
  3. 如果三方后续回调来了 → 按平台订单号匹配 → 正常上账
  4. 如果超时后一段时间没有回调 → 定时任务标记为过期
  5. 不重试 MatchChannel（可能创建重复通道订单）
```

```go
func (s *RateService) FetchRates(ctx context.Context, currencyCode string) (int64, error) {
    type rateResult struct {
        rate int64
        err  error
    }

    apis := []func(context.Context, string) (int64, error){
        s.fetchFromBinance,
        s.fetchFromCoinGecko,
        s.fetchFromThirdAPI,
    }

    // 并发调用，每个API独立超时5秒
    ch := make(chan rateResult, len(apis))
    for _, api := range apis {
        go func(fn func(context.Context, string) (int64, error)) {
            apiCtx, cancel := context.WithTimeout(ctx, 5*time.Second)
            defer cancel()
            rate, err := fn(apiCtx, currencyCode)
            ch <- rateResult{rate, err}
        }(api)
    }

    // 收集结果
    var rates []int64
    for i := 0; i < len(apis); i++ {
        r := <-ch
        if r.err == nil && r.rate > 0 {
            rates = append(rates, r.rate)
        }
    }

    if len(rates) == 0 {
        return 0, errs.ErrAllRateAPIsFailed // 全部失败，不更新
    }

    // 过滤异常值（偏离均值>30%的排除）
    rates = filterOutliers(rates, 0.3)

    if len(rates) == 0 {
        return 0, errs.ErrAllRatesAbnormal
    }

    // 取平均值
    var sum int64
    for _, r := range rates {
        sum += r
    }
    return sum / int64(len(rates)), nil
}

func filterOutliers(rates []int64, threshold float64) []int64 {
    if len(rates) <= 1 {
        return rates
    }

    // 计算均值
    var sum int64
    for _, r := range rates {
        sum += r
    }
    avg := float64(sum) / float64(len(rates))

    // 过滤偏离均值超过threshold的
    var filtered []int64
    for _, r := range rates {
        deviation := math.Abs(float64(r)-avg) / avg
        if deviation <= threshold {
            filtered = append(filtered, r)
        }
    }
    return filtered
}
```

### 12.1 充值回调验签

```
充值回调是 open 接口（/open/wallet/rechargeCallback），不需要登录态。
任何人都可以伪造请求调这个接口。如果不做验签，攻击者可以伪造"充值成功"通知给用户加钱。

  2. 根据通道类型选择对应的验签算法（MD5/SHA256/RSA）
  3. 验签失败 → 返回失败响应，记录日志告警
  4. IP白名单（网关层或代码层校验来源IP）

方案B伪代码：
```

```go
func (s *RechargeService) VerifyCallback(ctx context.Context, req *CallbackReq) error {
```

```go
// 加密
func Encrypt(plaintext string, key []byte) (string, error) {
    block, _ := aes.NewCipher(key)
    gcm, _ := cipher.NewGCM(block)
    nonce := make([]byte, gcm.NonceSize())
    io.ReadFull(rand.Reader, nonce)
    ciphertext := gcm.Seal(nonce, nonce, []byte(plaintext), nil)
    return base64.StdEncoding.EncodeToString(ciphertext), nil
}

// 解密
func Decrypt(encoded string, key []byte) (string, error) {
    data, _ := base64.StdEncoding.DecodeString(encoded)
    block, _ := aes.NewCipher(key)
    gcm, _ := cipher.NewGCM(block)
    nonceSize := gcm.NonceSize()
    nonce, ciphertext := data[:nonceSize], data[nonceSize:]
    plaintext, err := gcm.Open(nil, nonce, ciphertext, nil)
    return string(plaintext), err
}
```

## 十四、议题间关联关系 — 它们不是孤立的

```
12个挑战不是独立的，它们相互依赖、相互制约：

幂等性 ←依赖→ 并发控制
  幂等检查（INSERT唯一约束）本身就需要在事务内执行，
  事务内的悲观锁保证了幂等检查的原子性。

并发控制 ←依赖→ 事务边界
  悲观锁的范围 = 事务的范围。
  事务越大锁越久，事务越小越安全。

冻结三阶段 ←依赖→ 状态机
  freeze_detail 的状态机（冻结中→已确认/已解冻）
  是提现订单状态机的子状态机。

数据精度 ←影响→ 缓存一致性
  汇率缓存如果返回旧值，兑换计算就会用旧汇率。
  TTL设5分钟是精度和性能的平衡。

跨模块一致性 ←依赖→ 幂等性
  调用方可以安全重试的前提是我方接口幂等。
  幂等性是跨模块最终一致性的基石。

安全防护 ←补充→ 幂等性
  即使攻击者伪造了回调请求，
  幂等键唯一约束也能防止重复上账（最后一道防线）。
```

### 15.1 每个写操作必须遵循的模板

```go
func XxxOperation(ctx context.Context, req *XxxReq) (*XxxResp, error) {
    // ====== 第一段：参数校验（事务外）======
    if err := validateParams(req); err != nil {
        return nil, err
    }

    // ====== 第二段：获取外部数据（事务外）======
    // 如果需要RPC调用、Redis读取等，在这里做
    // 不要把RPC调用放在事务内！

    // ====== 第三段：事务执行（原子）======
    var resp *XxxResp
    err := db.Transaction(func(tx *gorm.DB) error {
        // 3a. 悲观锁定钱包
        wallet, err := lockWallet(tx, ...)
        if err != nil {
            return err
        }

        // 3b. 业务校验（余额充足？状态合法？）
        if wallet.Balance < req.Amount {
            return errs.ErrInsufficientBalance
        }

        // 3c. 幂等检查（INSERT流水）
        flow := buildFlow(wallet, req)
        if err := tx.Create(flow).Error; isDuplicateKeyError(err) {
            resp = buildIdempotentResp(tx, idempotentKey)
            return nil
        }

        // 3d. 更新余额
        result := tx.Model(wallet).Where("id = ? AND balance >= ?", wallet.Id, req.Amount).
            Update("balance", gorm.Expr("balance - ?", req.Amount))
        if result.RowsAffected == 0 {
            return errs.ErrInsufficientBalance
        }

        resp = buildResp(wallet, flow)
        return nil
    })

    // ====== 第四段：后续处理（事务外，允许失败）======
    if err == nil {
        _ = updateOrderStatus(ctx, ...)  // 非原子
        _ = deleteCache(ctx, ...)         // 非原子
        _ = sendNotification(ctx, ...)    // 非原子
    }

    return resp, err
}
```

| `CeilDiv(a, b)` | 向上取整除法 | 手续费 |

| `CanTransition(current, target)` | 状态机校验 | 订单状态更新 |
| `randomTTL(base)` | 带抖动的缓存TTL | 缓存写入 |
