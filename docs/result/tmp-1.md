**1.2 文档背景**

- 钱包体系作为平台的核心基础能力之一，用于承载用户在平台内的资金流转与消费行为，需同时满足娱乐消费与igaming投注两类业务场景。

**1.4 使用对象**

- 本文档面向公司内部研发团队使用，主要包括但不限于：

  - 交互/UI设计师：用于理解功能边界与交互目标
  - 后端开发工程师：用于系统架构设计、接口实现与权限控制
  - 前端开发工程师：用于页面结构、交互逻辑与权限适配
  - 测试工程师：用于测试用例设计、功能验证与回归测试

### 图片8：`2-1资金钱包体系6.png`

- **主播钱包：** 用于承接平台向主播、创作者结算的各类收益资金的钱包账户
  - 主播通过直播、连麦、短视频等内容场景获得的分成收益，统一聚至主播钱包。主播钱包内资金支持提现，也可以转入中心钱包进行消费。

**主播钱包链路：**
- 类型：礼物打赏、主播活动/任务奖励 → 主播钱包 → 可转入中心钱包（默认提现） → 可提现

- **代理钱包(预留)：** 代理专属资金钱包，用于承接下级分佣收益并支持提现

**代理钱包链路：**
- 类型：代理分成 → 代理钱包 → 可转入中心钱包（默认提现） → 可提现

- **场馆钱包(预留)：** 用户进入igaming场馆时的临时资金钱包，分为单一钱包模式和自动转账钱包模式

**场馆钱包链路：**
- 投注 → 场馆钱包 → 退出场馆 → 中心钱包+活动钱包
- 中心钱包+活动钱包余额 → 场馆钱包 → 退出场馆 → 中心钱包+活动钱包

### 图片11：`3-2平台基准币种.png`

**3.2 平台基准币种**

- 基准币种定义：系统内部统一使用一个**基准币种**进行账务换算
  - 推荐使用：USDT，可配置，配置后无法更改
  - **BSB价值：10BSB = 1USDT，BSB汇率锚定USDT**
- 使用场景
  - 活动奖金计算
  - 多币种投注与返奖统计
  - 多币种钱包资产统计
  - 财务对账与统计报表

所有币种金额在账务层需换算为基准币种再进行统计

### 图片18：`4-1汇率应用场景.png`

**四、汇率规则说明**

**4.1 汇率应用场景**

- 充值：跨币种充值时使用平台入款汇率，例如充值通道支持USDT充BSB
- 提现：跨币种提现时使用平台出款汇率，例如平台内到BSB直接提USDT
- 投注：游戏不支持当前钱包币种，进入游戏时根据平台入款汇率转换为可投注币种，并记录该汇率，从游戏退出时使用该汇率转换回钱包币种
- 统计：平台充值、提现、账变等统计需要根据平台汇率转化为基准货币进行报表统计

### 图片19：`4-2汇率规则说明1.png`

**4.2 汇率规则**

- 实时汇率：通过三方API接口获取，后台展示，平台汇率的基准来源，可约定实时汇率频率，并发测试期间可低频约1天1次；上线根据运营需求设定
- 通过3家三方获取汇率，算平均值作为实时，多家三方也可以避免接口API停用或挂掉的问题
- 平台汇率：平台汇率刷新时的实时汇率，初始平台汇率为空时，直接取实时汇率值为平台汇率浮动%在后台配置
- 平台入款汇率 = 实时汇率 * (1 + 汇率浮动%)
- 平台出款汇率 = 实时汇率 * (1 - 汇率浮动%)
- 汇率偏差：实时汇率与平台汇率的偏差值
  - 偏差值 = (实时汇率 - 平台汇率) / 实时汇率 * 100
- 阈值：阈值可配置，当偏差值 > 阈值%时，更新平台汇率

- 某个币种偏差值>阈值时，更新该币种的平台汇率出入款汇率，并生成日志，未超过阈值的币种不更新

**举例：**

例1：BSB: VDN = 1 : 71
- 平台70 → 实时71 → 入款72
- 汇率浮动 = (71 - 70) / 71 * 1.4%
- 出款68 → 实时71 → 入款72

例2：BSB: VDN = 1 : 69
- 汇率浮动 = (69 - 70) / 69 = 1.4%
- 出款68 → 实时69 → 入款72（保持不变，因为未达到阈值...部分内容）

**九、非功能性需求（Non-Functional Requirements）：**

- **对账一致性：** 明确"账务流水"与"账户余额"的强一致性校验机制。

- 不支持用户自行修改提现账户
- 异常情况需人工介入
- 大额 / 异常频率触发人工审核

### 2.13 通道轮询规则

- 通道轮询仅针对代收（充值）方向，代付暂不需要轮询逻辑，如后续代付也需要轮询再补充前端界面说明
- 产品上轮询策略概述：在轮询模式下，系统根据"成功率优先"的策略，在同类型可用的通道中进行自动轮询选择
- 系统处理逻辑：通道优先级通过权重+成功率确定

**轮询规则涉及参数解释和配置规则说明：**

- 权重基础排序：所有同类型启用通道按照配置的权重值进行基础排序，权重越高优先级越高
- 成功率动态调整：系统实时监控各个通道的支付成功率，对于下降到指定阈值以下的通道，系统自动降低其优先级
- 一级优先：成功率 > 阈值，二级优先：成功率下降，三级切换：通道暂时移出
- 如轮询规则配置不完整无通道可选，在后台可切换强制下选通道（强制使用最新优先级排序通道，不论成功率是否降低）
- 同一方式关联多条通道时，按配置的权重值依次尝试，权重相同随机选择
- 轮询策略配置分为三个维度指标：成功率系数、时间系数、预警系数
- 成功率、时间、预警，三个系数 = 一个总分数，优先级按照总分数排序，分数越高优先级越高
