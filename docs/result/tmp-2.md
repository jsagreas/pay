## 三、模块A — 多币种钱包需求详析

### 3.1 项目背景与目标

平台业务涵盖 **直播 + 游戏 + 打赏 + 多国市场**，需建立清晰、安全、可扩展的多币种钱包系统，核心能力:
- 多币种充提
- 游戏投注与结算
- 礼物打赏与主播收入
- 提现与合规风控（KYC）
- 资金流水追踪与对账

**产品目标:**
1. 支持多币种钱包体系（法币 / BSB 平台币）
2. 清晰资产分层，避免资金逻辑混乱
3. 不同币种对应不同充提方式
4. 提供完整可追溯的资金记录
5. 提现合规（KYC + 风控）

**适用范围:** iOS / Android / H5，用户角色: 普通用户 + 主播（签约/非签约）

### 3.2 钱包币种结构

钱包按 **币种维度隔离**，每个币种独立余额和账本。

**当前支持币种:**

| 币种 | 千分位符号 | 小数规则 | 汇率锚定 |
|------|-----------|---------|---------|
| BSB（平台币） | `,` | 2位小数 | 1 BSB = 0.1 USDT（即 10 BSB = 1 USDT） |
| VND（越南盾） | `.` | 整数（无小数） | 实时汇率 |
| IDR（印尼盾） | `.` | 整数（无小数） | 实时汇率 |
| THB（泰铢） | `,` | 2位小数 | 实时汇率 |

### 3.3 币种切换规则

- 钱包首页可切换当前币种
- **当前选中币种 = 所有操作的唯一数据源**（充值/提现/兑换/账变记录）
- 切换币种 **立即切换**: 余额、充提方式、记录列表
- **不支持跨币种混合显示和操作**

### 3.4 充值模块

#### 3.4.1 充值方式总览

| 类型 | 方式 | 平台承载页? | 状态回调 | 备注 |
|------|------|-----------|---------|------|
| 虚拟币 | USDT（TRC20/ERC20） | 是 | 三方支付通道回调 | 链上转账 |
| 银行 | 银行转账 | 否 | 三方支付通道回调 | 三方通道 |
| 电子钱包 | QRIS / PromptPay 扫码 | 否 | 三方支付通道回调 | 三方通道 QR码 + 倒计时 |

**各币种充值方式:**
- **BSB**: USDT（推荐）+ 银行转账 + 电子钱包扫码，金额输入框下方显示汇率
- **VND**: 仅越南银行转账
- **IDR**: 电子钱包扫码（推荐）+ 印尼银行
- **THB**: 电子钱包扫码（推荐）+ 泰国银行

**通用规则:**
1. 每个币种有默认充值方式及默认最小档位快捷金额
2. 充值金额输入框支持手动填写（不得小于最低档位）
3. 点击"去充值"后生成订单匹配通道
4. 只有 BSB 充值金额输入框下有对应汇率展示及兑换所得金额

#### 3.4.2 USDT 充值

- 仅在支持 USDT 支付方式的币种下可见
- **游客（未登录）也支持 USDT 充值**
- 充值单位: 钱包币种单位，系统自动按汇率换算为 USDT
- 网络选择: TRC20（默认），可展开 ERC20 / BEP20
- 显示: 网络类型、收款地址（可复制）、QR码
- 到账: 区块链监控服务确认，通常 **2-10 分钟**

#### 3.4.3 银行转账充值

- 根据当前币种显示对应国家的银行转账
- 流程: 用户选择金额 → 点击充值 → 平台创建订单 → 获取并展示转账信息（三方承载页）→ 支付状态回传
- 转账信息: 银行名称、户名、账号、金额、转账备注（UserID + 时间戳）

#### 3.4.4 QRIS / PromptPay 扫码充值

- 印尼 / 泰国法币钱包下显示，为默认推荐方式
- 三方承载页: QR码 + 倒计时 + 支付方式 + 金额（可复制）+ 备注
- 订单过期后 QR码失效

### 3.5 重复转账拦截规则

**识别维度:** 同一 UserID + 同一币种 + 同一充值金额
**状态条件:** 订单处于"待支付"或"处理中"
**适用范围:** 银行转账、电子钱包支付（**不适用于 USDT 链上充值**）

| 待处理订单数 | 弹窗内容 | 按钮1 | 按钮2 | 行为 |
|------------|---------|-------|-------|------|
| 1-2笔 | "您有N笔同等金额交易正在进行中，建议先处理。是否继续创建新充值？" | 查看详情 | 继续充值 | 标准弹窗 |
| 3笔（上限） | "您有3笔同等金额交易正在进行中，请优先完成" | 查看详情 | 确认 | **强拦截**，不创建新订单 |

### 3.6 奖金/优惠选择机制

- **组件位置:** 余额显示区下方、支付方式区上方
- 默认显示"无奖金"或"当前最佳匹配活动"
- 点击展开半屏浮层，列出所有可用奖金活动
- **选择奖金后:**
  - 快捷金额档位仅显示满足活动门槛的档位
  - 每个档位右上角显示奖金百分比（如 +50%）
  - 输入区右侧显示"额外+XXX"
- **门槛校验:** 手动输入金额低于活动最低门槛时，toast 提示
- **一笔充值订单只能参与一个奖金活动**
- 强制提示: "您的充值额与奖金将入账充值钱包，需完成 xx 倍流水要求"

### 3.7 兑换模块（法币 → BSB）

**兑换范围:**
- VND / IDR / THB → BSB（BSB 充值界面无兑换入口）
- 只允许兑换当前币种的余额

**汇率显示:** 统一格式 `1 BSB = X 法币`
- 1 BSB = 2,620 VND
- 1 BSB = Rp 1,530 IDR
- 1 BSB = 3.50 THB

**兑换计算:**
- 输入: 当前法币金额（超过余额提示"已达当前可兑换上限"）
- 兑换所得: BSBxxxx.xx
- **赠送奖金**（如 10%）
- **实际到账 BSB = 兑换所得 + 赠送奖金**
- 赠送部分需完成 **X 倍流水** 后才能提现

### 3.8 提现模块

#### 3.8.1 KYC 与提现关系

| 状态 | 行为 | 可用提现方式 |
|------|------|------------|
| 未认证 | 点击提现触发弹窗:"去认证" / "继续提现" | 仅 USDT 提现 |
| 已认证 | 直接进入提现方式选择页 | 法币/电子钱包/USDT 全部渠道 |

**KYC 认证字段:** 姓名（必填）、国家、手机号、证件类型、证件号码（必填）、上传证件正反面
**认证成功后不可更改**

#### 3.8.2 提现规则（基于币种路由）

| 充值方式 | 允许的提现方式 |
|---------|--------------|
| 法币充值 | 法币提现 |
| USDT 充值 | USDT 提现 |
| 法币 + USDT 充值 | 法币提现 |
| 法币 → BSB | 法币提现 |

#### 3.8.3 银行卡提现

- **首次提现字段:** 银行名称（必填）、银行账号（必填）、账户持有人（灰色，默认 KYC 姓名）、开户网点（选填）、提现金额
- **非首次:** 自动填充已保存信息（只读），仅需输入金额
- **到账时间:** T+1 个工作日
- **示例限额:** 每日提现限额 d 2,500,000 VND，单笔最低 d 2,500,000 VND，手续费 1%

#### 3.8.4 电子钱包提现

- **支持:** GoPay / OVO / DANA / ShopeePay（印尼），PromptPay（泰国）
- **首次提现字段:** 钱包类型、手机号、账户持有人（灰色 KYC 姓名）、提现金额
- **到账时间:** 即时到账
- **账户持有人名称必须与 KYC 认证姓名一致**（忽略大小写、忽略多余空格、非完全等值字符串比对）

#### 3.8.5 BSB 提现

- **支持:** USDT / 各大银行 / GoPay / OVO / DANA / PromptPay 等
- BSB 提现金额输入框下方显示汇率
- 显示: 提现金额、手续费、到账金额（按目标法币显示）

#### 3.8.6 提现校验

- 提现金额超过每日限额: 弹窗"您已超出当日提现限额，请修改提现金额"
- 低于单笔最低: 弹窗"未达到单笔最低提现金额，请修改提现金额"
- 账户信息不完整 → 确认提现按钮灰色不可点击

### 3.9 账变记录模块

#### 3.9.1 记录分类

4个 Tab: **提现 / 充值 / 兑换 / 奖励**

**核心规则（强制）:**
- 钱包当前币种 = 记录当前币种
- 切换币种**立即切换全部记录数据集**
- **不支持跨币种混合记录**

#### 3.9.2 列表字段

| 类型 | 字段 |
|------|------|
| 提现 | 金额、方式、时间、状态（已完成/审核中/失败）、操作 |
| 充值 | 金额、方式、时间、状态（已完成/充值中/失败）、操作 |
| 兑换 | 兑换金额、赠送%、实际到账、时间 |
| 奖励 | 奖励金额、奖励类别、时间 |

#### 3.9.3 详情页

- 所有金额币种**必须与列表页币种一致**
- 字段: 订单号、金额明细、状态、时间、方式、失败原因（如有）
- 充值失败原因: 支付超时、支付异常、银行卡问题
- 提现失败原因: 信息错误、系统维护、超出限额
- 失败订单提供"更换其他充值/提现方式"按钮

### 3.10 非功能性需求

- **对账一致性:** 必须建立"账务流水记录"与"账户余额"之间的强一致性校验机制
- **幂等性:** 充值回调**必须实现幂等处理**，防止重复入账
- **KYC 强制验证**（法币提现渠道）
- **提现账户白名单**
- **姓名强一致性校验**（账户持有人 = KYC 姓名）
- **用户不允许自行修改提现账户**
- **异常情况需人工干预**
- **大额 / 异常频率触发人工审核**

## 四、模块B — 币种配置需求详析

### 4.1 资金钱包体系

#### 4.1.1 钱包结构

| 钱包类型 | 用途 | 状态 |
|---------|------|------|
| **中心钱包** | 用户主资金钱包，处理充值、投注扣款、投注返奖、提现 | 已实现 |
| **奖励钱包** | 活动/任务奖励临时钱包，完成稽核流水后可转入中心钱包 | 已实现 |
| **主播钱包** | 主播各类收入（直播/语聊/短视频/场馆佣金）统一结算 | 已实现 |
| **代理钱包** | 代理专属，接收下级佣金，支持提现 | 预留 |
| **场馆钱包** | 进入 iGaming 场馆时的临时资金钱包 | 预留 |

#### 4.1.2 消费优先级

- **资金钱包余额 = 中心钱包余额 + 奖励钱包余额**
- **消费优先级: 先扣中心钱包，再扣奖励钱包**
- 中心钱包投注返奖归中心钱包；奖励钱包投注返奖归奖励钱包
- **混合投注时，返奖按投注金额比例拆分**

#### 4.1.3 使用场景

- 购买礼物、购买虚拟道具（头像框/坐骑/靓号等）
- 娱乐场消费（也可稽核）
- 投注（电子/真人/彩票/体育）
- 提现

#### 4.1.4 获得途径

- 充值（直接充值法币或虚拟币）
- 投注返奖
- 活动/任务返奖

### 4.2 币种类型划分

| 币种类型 | 说明 | 示例 |
|---------|------|------|
| **法定货币** | 国家/地区官方货币 | IDR、VND、THB |
| **加密货币** | 区块链或平台支持的数字资产 | USDT、BTC、ETH |
| **平台货币** | 平台专属货币 | BSB |

- 配置层统一为"币种"，规则层做差异化
- 每个币种对应一个币种资金钱包，不同币种钱包独立

### 4.3 基准币种

- **定义:** 系统内部统一使用的一种基准货币，用于所有核算换算
- **推荐:** USDT，可配置但**配置后不可更改**
- **BSB 价值:** 10 BSB = 1 USDT，BSB 汇率锚定 USDT

**基准币种用途:**
- 活动奖金计算
- 多币种投注与返奖统计
- 多币种钱包资产统计
- 财务对账与统计报表
- **所有币种金额在核算层必须换算为基准币种进行统计**

### 4.5 汇率规则

#### 4.5.1 汇率应用场景

| 场景 | 使用的汇率 |
|------|----------|
| 充值（跨币种） | 入款汇率 |
| 提现（跨币种） | 出款汇率 |
| 投注（币种不支持时） | 入款汇率转入，同汇率转出 |
| 统计 | 平台汇率换算为基准币种 |

#### 4.5.2 汇率获取与计算

**实时汇率:**
- 从 **3 个三方汇率 API** 获取，取**平均值**作为实时汇率
- 目的: 避免单一 API 宕机或卡死问题
- 频率: 可配置（开发测试可低频，上线后按运营需求设定）

**平台汇率:**
- **初始状态:** 平台汇率为空时，直接设置实时汇率为平台汇率
- 后续: 通过偏差阈值机制触发更新

**计算公式:**

```
入款汇率 = 实时汇率 × (1 + 汇率浮动%)
出款汇率 = 实时汇率 × (1 - 汇率浮动%)
偏差值 = |实时汇率 - 平台汇率| / 实时汇率 × 100%
```

#### 4.5.3 阈值更新机制

```
实时汇率更新 → 平台汇率是否存在？
  ├── 否 → 以实时汇率计算平台汇率
  └── 是 → 偏差 >= 阈值%？
              ├── 是 → 更新平台汇率（入款+出款），生成日志
              └── 否 → 平台汇率保持不变
```

**重要:** 偏差检查必须同时评估入款和出款偏差；**任一** >= 阈值即触发更新。
**所有平台汇率更新必须生成对应的汇率日志。**

**示例:**
```
BSB:VND = 1:71
平台汇率 70，实时汇率 71: 偏差 = (71-70)/71 ≈ 1.4%
若阈值 = 1.4%:
  → 平台汇率更新为 71
  → 入款汇率 = 71 × (1+1.4%) ≈ 72
  → 出款汇率 = 71 × (1-1.4%) ≈ 70
```

#### 4.5.4 汇率显示规则

| 币种类型 | C端显示汇率 | 系统端汇率 |
|---------|-----------|----------|
| 法定货币 | 保留 **2 位小数** | 保留 **2 位小数**，不足补 0，超出四舍五入 |
| 加密货币 | 保留 **8 位小数** | 保留 **8 位小数**，不足补 0，超出四舍五入 |

### 4.6 汇率日志

- **位置:** 操作日志页面新增 [汇率日志] Tab
- **触发:** 任何币种平台汇率变更时生成日志
- **字段:** 序号、日志编号、更新时间、币种代码、实时汇率、平台汇率、入款汇率、出款汇率
- **筛选:** 更新时间（日期范围）、币种代码（下拉）

## 五、模块C — 财务管理需求详析（依赖分析重点）

> 此模块由他方负责开发，我方重点分析其与钱包模块的交互依赖。

### 5.1 模块功能列表

1. 通道配置（代收通道 / 代付通道 / 轮询规则）
2. 支付配置（充值方式 / 提现方式）
3. 充值记录
4. 提现记录
5. 人工修正（充值补单 / 人工加款 / 人工减款）
6. 数据统计（通道统计 / 充值统计 / 提现统计）

### 5.2 通道配置

#### 代收/代付通道
- 通道由开发团队在集成新通道时录入，后台仅可编辑已有数据
- 可编辑字段: 费率（1位小数%）、单笔限额（min-max）、权重（整数）、预警金额、状态、配置参数
- 配置参数: 请求地址、商户ID、密钥、AppKey、支付编码、公钥/私钥

#### 轮询规则
- 综合轮询得分 = **成功率系数 × 时间系数 × 预警系数 × 权重**
- 得分最高的通道获取下一笔交易
- 代收和代付通道独立配置轮询参数

### 5.3 支付配置

#### 充值方式
- 字段: 币种、充值方式名称、支付图标（SVG/PNG ≤10KB）、充值档位、充值金额范围、关联通道（多选）、排序、状态
- 一个充值方式可关联多个通道
- 同币种下充值方式名称唯一

#### 提现方式
- 字段: 币种、提现方式名称、支付图标（SVG/PNG ≤10KB）、状态
- 比充值方式简单（无档位/范围/通道关联）
- 同币种下提现方式名称唯一

### 5.4 充值记录

#### 订单号格式: **C + 16位随机数**

#### 充值流程（三泳道: 客户端 / 服务端 / 通道）

1. 客户端: 用户选择币种 → 选择充值方式 → 输入金额 → 提交订单
2. 服务端: 轮询算法计算通道分配 → 创建平台订单（状态=待支付）
3. 通道: 创建通道侧订单号 → 返回支付结果
4. 客户端: 跳转支付页 → 用户支付
5. 通道: 返回支付结果给服务端
6. **服务端（支付成功）: 订单状态变更为支付成功 → 触发充值额度到钱包用户上**
7. 服务端（支付失败）: 订单状态=支付失败 → 通知客户端

**关键钱包交互: 充值成功时，系统将到账总金额（订单金额 + 活动赠送）记入用户钱包**

#### 列表字段（24个）

序号、创建时间、完成时间、平台订单号、通道订单号、用户编号、昵称、通道名称、订单币种、充值方式、订单状态（待支付/支付成功/支付失败）、订单金额、活动类型、活动优惠、充值费率、手续费、到账总金额、入款汇率、支付币种、支付金额、平台汇率、基准币种订单金额、基准币种活动优惠、基准币种手续费

### 5.5 提现记录

#### 订单号格式: **T + 16位随机数**

#### 提现状态流转（7个状态）

```
待风控审核 → 风控驳回（终态）
           → 待财务审核 → 财务驳回（终态）
                        → 待出款 → 出款失败（需人工处理）
                                  → 出款成功（终态）
```

#### 提现流程（三泳道）

1. 客户端: 用户选择提现方式、输入信息、提交订单
2. **服务端: 创建订单 → 立即冻结用户钱包金额** → 状态=待风控审核
3. 风控审核:
   - 通过 → 待财务审核
   - **驳回 → 解冻钱包金额**
4. 财务审核:
   - 通过 → 待出款，计算手续费，发送到通道
   - **驳回 → 解冻钱包金额**
5. 出款审核:
   - 通过 → 发送到支付通道
   - **驳回 → 解冻钱包金额**
6. 通道返回结果:
   - **成功 → 扣除冻结金额**
   - **失败 → 保持冻结，不自动解冻，需人工处理**

#### 锁定/解锁机制

- 锁定: 锁定提现订单，其他操作员不可同时处理
- 解锁: 释放锁定
- 强制解锁: 管理员覆盖释放

### 5.6 人工修正

#### 5.6.1 充值补单

**用途:** 用户充值实际成功但系统未收到通道回调（订单停留在待支付），运营手动补单入账。

- **补单订单号:** B + 16位随机数
- 仅**待支付**状态的订单可补单
- 创建补单时需查询原订单信息
- **补单成功 → 补单总金额（补单金额 + 补单赠送）记入用户钱包**
- 包含**稽核倍数**和**稽核流水**字段

#### 5.6.2 人工加款

**用途:** 手动给用户钱包加款（运营补偿/运营奖励/财务修正/测试加款）。

- **加款订单号:** A + 16位随机数
- **可指定目标钱包类型及每个钱包的加款金额:**
  - 中心钱包
  - 奖励钱包
  - 代理钱包
  - 主播钱包
- 显示加款后各钱包余额
- 每个钱包类型有独立的稽核倍数和稽核流水
- **加款成功 → 按指定金额分别加入对应钱包**

#### 5.6.3 人工减款

**用途:** 手动从用户钱包扣款（异常追回/风控处罚/财务冲正/费用调整）。

- **减款订单号:** M + 16位随机数
- **可指定目标钱包类型及每个钱包的减款金额:**
  - 中心钱包
  - 奖励钱包
  - 代理钱包
  - 主播钱包
- 显示当前余额和减款后余额
- **减款金额不能超过当前余额**（减款后余额 >= 0）
- **减款成功 → 按指定金额分别从对应钱包扣除**

### 5.7 数据统计

- 统计数据**实时更新**，服务端数据使用 **UTC+0 基准时间**
- 运营人员可设置站点时区，查询时间和显示时间按站点时区调整
- 站点时区配置: 设置页 → 通用 → 站点时区

#### 通道统计
- 代收/代付通道统计
- 汇总卡片: 订单总数、成功订单数、成功率、平均入/出款时间、订单总金额、成功金额、净入账/出账金额、手续费

#### 充值统计
- 汇总: 订单总数、成功订单数、充值人数、成功率、手续费、成功充值人数、人均充值金额

#### 提现统计
- 汇总: 订单总数、成功订单数、提现人数、成功率、平台手续费、通道手续费、待出款金额、出款失败金额
- 详细分状态: 待风控审核/风控驳回/待财务审核/财务驳回/待出款/出款失败

## 七、关键业务规则速查

### 7.1 钱包核心规则

1. 按币种维度隔离，不支持跨币种混合操作
2. 消费优先级: 中心钱包 → 奖励钱包
3. 混合投注时返奖按比例拆分
4. BSB 锚定: 10 BSB = 1 USDT
5. 账务流水与余额必须强一致性
6. 充值回调必须幂等处理

### 7.2 提现核心规则

1. 法币充值 → 法币提现; USDT 充值 → USDT 提现
2. 法币+USDT 充值 → 法币提现
3. 法币 → BSB → 法币提现
4. 未KYC → 仅USDT提现; 已KYC → 全渠道
5. 账户持有人 = KYC姓名（忽略大小写和多余空格）
6. 首次提现需填写账户信息，后续自动填充

### 7.3 汇率核心规则

1. 基准币种（USDT）设置一次不可更改
2. BSB 汇率/代码/类型/状态不可配置
3. 实时汇率 = 3个三方API取平均值
4. 平台汇率仅在偏差 >= 阈值%时更新
5. 入款汇率 = 实时 × (1+浮动%); 出款汇率 = 实时 × (1-浮动%)
6. 法定货币显示2位小数; 加密货币显示8位小数

### 7.4 重复转账拦截

1. 识别: 同UserID + 同币种 + 同金额
2. 仅适用于银行转账和电子钱包（USDT链上不适用）
3. 1-2笔: 警告可继续; 3笔: 强拦截不创建

### 7.5 稽核/流水要求

1. 充值奖金和兑换赠送均有稽核倍数要求
2. 补单和人工加款也记录稽核倍数和稽核流水
3. 必须完成规定倍数流水后才能提现
