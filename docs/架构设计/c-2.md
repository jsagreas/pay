# ser-wallet 架构设计

> 定位：钱包 + 多币种配置 + 配合财务模块联调的整体技术架构方案
> 原则：安全、高并发、高性能、不过度设计、业界主流
> 基础：完全基于现有工程基础设施（Kitex + GORM + TiDB + Redis + ETCD）
> 依据：业界钱包/支付系统通用架构模式 + 项目代码实证

## 1.2 核心架构决策

| 决策项 | 选择 | 理由 |
|-------|------|------|
| 账户模型 | 余额账户制（非复式记账） | 业务复杂度适中，不需要完整的复式记账系统 |
| 余额存储 | 数据库为准，Redis 为辅 | 金融数据以 DB 为权威来源，缓存加速读 |
| 并发控制 | 数据库行锁（悲观锁）为主 | 简单可靠，TiDB 行锁成熟，不过度设计 |
| 幂等方案 | 流水表唯一索引 | 复用审计流水，不额外建表 |
| 金额精度 | BIGINT 整数（最小货币单位） | 无浮点误差，性能好，业界主流 |
| 事务模型 | 本地事务（单库） | 钱包数据在一个库内，不需要分布式事务 |
| 消息机制 | 同步 RPC 为主 | 现有工程全部是同步 RPC，保持一致 |

# 二、内部分层架构

## 2.1 四层架构

> 遵循现有工程的分层模式（handler → ser → rep/repo），在钱包场景下明确每层职责

```
┌─────────────────────────────────────────────────────────────────┐
│  handler.go（接口层）                                             │
│  · 纯委托，不含业务逻辑                                           │
│  · 按方法一一委托到对应的 ser 层 Service                            │
├─────────────────────────────────────────────────────────────────┤
│  internal/ser/（业务编排层）                                       │
│  · 业务规则校验（余额是否足够、资金来源规则、KYC 校验等）             │
│  · 编排多个原子操作为一个业务流程                                   │
│  · 调用外部 RPC（财务模块、KYC 服务等）                             │
│  · 调用 ledger 层执行余额变更                                      │
│  · 缓存读写策略控制                                               │
├─────────────────────────────────────────────────────────────────┤
│  internal/ledger/（账务核心层）★ 新增层，钱包特有                    │
│  · 余额变更的原子操作（Credit / Debit / Freeze / Unfreeze）        │
│  · 事务管理（保证余额变更 + 流水记录在同一事务中）                    │
│  · 幂等控制（基于 orderNo 防重）                                   │
│  · 并发安全（行锁）                                               │
│  · 这一层是资金安全的最后防线，不允许被绕过                          │
├─────────────────────────────────────────────────────────────────┤
│  internal/gorm_gen/ + internal/rep/（数据访问层）                   │
│  · gorm_gen：自动生成的标准 CRUD（19 个方法 / 表）                  │
│  · rep：手写的复杂查询扩展（如带锁查询、聚合统计）                   │
└─────────────────────────────────────────────────────────────────┘
```

## 2.2 为什么需要 ledger 层

现有服务（ser-app 等）只有 ser → repo 两层就够了，因为它们的写操作是简单的 CRUD。但钱包不同：

```
普通服务的写操作:
  ser 层: 校验 → repo.Create/Update → 缓存失效 → 审计日志
  数据安全要求: 普通（写错了改回来就行）

钱包的余额操作:
  ser 层: 业务校验 → ??? → 缓存失效 → 审计日志
  这个 ??? 需要保证:
    · 幂等：同一笔 orderNo 不能重复执行
    · 原子：余额变更 + 流水记录同成功同失败
    · 并发：两个并发的 Credit 不能导致余额错误
    · 审计：变更前后的余额必须准确记录

如果把这些逻辑散在 ser 层的各个方法里，会导致:
  · 代码重复（每个改余额的地方都要写事务 + 幂等 + 锁）
  · 容易遗漏（某个方法忘了加锁或幂等）
  · 难以审计（安全相关逻辑分散在各处）
```

因此，把余额操作收敛到一个独立的 ledger 层，是钱包系统业界的标准做法。所有余额变更只能通过 ledger 层完成，ser 层不能直接操作余额表。

## 2.3 分层调用规则

```
规则一：handler → ser（一一委托）
规则二：ser → ledger（余额相关操作必须走 ledger）
规则三：ser → rep/repo（非余额的普通读写直接走 repo）
规则四：ser → 外部 RPC（调财务/KYC 等）
规则五：ledger → rep/repo（在事务内操作 wallet 表 + flow 表）
规则六：禁止 ser 层直接 UPDATE wallet 表的 balance/frozen 字段
```

# 三、数据架构

## 3.1 金额类型：统一使用整数

```
决策：所有金额在数据库中使用 BIGINT，以该币种的最小货币单位存储

  VND: 1 VND = 1（越南盾没有"分"的概念，最小单位就是1盾）
  IDR: 1 IDR = 1（印尼卢比同理）
  THB: 1 THB = 100（泰铢有 satang，1 THB = 100 satang）
  USDT: 1 USDT = 100000000（USDT 精度 8 位，1 USDT = 10^8 最小单位）
  BSB: 1 BSB = 100000000（平台币，对齐 USDT 精度）

每个币种在 currency_config 表中记录精度(decimals)字段：
  VND: decimals=0    → 数据库存 50000 → 展示 50,000 VND
  USDT: decimals=8   → 数据库存 100000000 → 展示 1.00000000 USDT
  BSB: decimals=8    → 数据库存 1000000000 → 展示 10.00000000 BSB

IDL 传输: i64（整数形式传输）或 string（带小数点的展示形式传输）
  建议：IDL 用 string 传输（前端友好），内部用 int64 计算
  转换公式：展示值 = 数据库值 / 10^decimals

好处：
  · 杜绝浮点精度问题（金融系统的刚性要求）
  · 计算性能好（整数运算远快于 decimal）
  · 数据库存储高效（BIGINT 8 字节 vs DECIMAL 动态长度）
  · 业界标准做法（Stripe/支付宝/微信支付都用最小单位整数）
```

## 3.2 核心表设计

### 3.2.1 user_wallet — 用户钱包表（最核心）

```sql
CREATE TABLE user_wallet (
    id              BIGINT       PRIMARY KEY,             -- 雪花 ID
    user_id         BIGINT       NOT NULL,                -- 用户 ID
    currency_id     BIGINT       NOT NULL,                -- 币种 ID
    wallet_type     TINYINT      NOT NULL,                -- 钱包类型: 1=中心 2=奖励 3=主播
    balance         BIGINT       NOT NULL DEFAULT 0,      -- 余额（最小货币单位）
    frozen_amount   BIGINT       NOT NULL DEFAULT 0,      -- 冻结金额
    version         BIGINT       NOT NULL DEFAULT 0,      -- 乐观锁版本号（预留）
    status          TINYINT      NOT NULL DEFAULT 1,      -- 1=正常 2=冻结 3=禁用
    create_by       BIGINT       DEFAULT 0,
    create_at       BIGINT       DEFAULT 0,
    update_by       BIGINT       DEFAULT 0,
    update_at       BIGINT       DEFAULT 0,
    delete_at       BIGINT       NOT NULL DEFAULT 0,      -- 软删除

    UNIQUE KEY uk_user_currency_type (user_id, currency_id, wallet_type, delete_at)
) ENGINE=InnoDB;

-- 设计要点：
-- · 唯一索引保证一个用户+一个币种+一个钱包类型只有一行记录
-- · balance 和 frozen_amount 不允许为负（应用层保证）
-- · 可用余额 = balance - frozen_amount（计算值，不存储）
-- · version 字段预留给乐观锁场景（当前主用行锁）
```

### 3.2.2 wallet_flow — 流水表（审计核心 + 幂等载体）

```sql
CREATE TABLE wallet_flow (
    id              BIGINT       PRIMARY KEY,             -- 雪花 ID
    user_id         BIGINT       NOT NULL,
    currency_id     BIGINT       NOT NULL,
    wallet_type     TINYINT      NOT NULL,
    flow_type       TINYINT      NOT NULL,                -- 1=充值 2=提现 3=兑换出 4=兑换入
                                                          -- 5=奖励 6=冻结 7=解冻 8=人工加 9=人工减
                                                          -- 10=消费 11=转入(奖励→中心)
    amount          BIGINT       NOT NULL,                -- 变动金额（正数=增加 负数=减少）
    balance_before  BIGINT       NOT NULL,                -- 变动前余额
    balance_after   BIGINT       NOT NULL,                -- 变动后余额
    order_no        VARCHAR(32)  NOT NULL,                -- 关联业务订单号
    biz_type        VARCHAR(32)  DEFAULT '',               -- 业务类型标识
    remark          VARCHAR(256) DEFAULT '',
    create_at       BIGINT       DEFAULT 0,

    UNIQUE KEY uk_order_wallet (order_no, wallet_type),   -- 幂等索引：同一订单+同一钱包类型只能写一条
    INDEX idx_user_currency (user_id, currency_id, create_at)
) ENGINE=InnoDB;

-- 设计要点：
-- · 这张表是 append-only 的，只写不改不删
-- · uk_order_wallet 唯一索引同时承担幂等去重的功能
-- · balance_before/after 记录变更快照，用于对账和审计
-- · 一笔兑换订单会产生 3 条流水（扣法币 + 加BSB中心 + 加BSB奖励）
--   它们的 order_no 相同但 wallet_type 不同，所以唯一索引不冲突
```

# 四、账务核心架构（ledger 层）

> 这是整个架构中最关键的部分，决定了资金安全

## 4.1 ledger 层的 5 个原子操作

ledger 层只暴露 5 个方法，所有余额变更必须通过它们完成：

```
┌─────────────────────────────────────────────────────────────────┐
│  ledger 层对外暴露的 5 个方法                                      │
│                                                                 │
│  Credit(ctx, params)    — 增加余额（充值上分/奖金发放/人工加款）    │
│  Debit(ctx, params)     — 减少余额（提现扣款/人工减款/消费）        │
│  Freeze(ctx, params)    — 冻结金额（提现时冻结可用余额）            │
│  Unfreeze(ctx, params)  — 解冻金额（提现拒绝/失败时归还）          │
│  Exchange(ctx, params)  — 兑换（跨币种原子操作：扣A币种+加B币种）   │
│                                                                 │
│  参数结构:                                                       │
│  · OrderNo    string    — 业务订单号（幂等 key）                  │
│  · UserID     int64     — 用户 ID                                │
│  · CurrencyID int64     — 币种 ID                                │
│  · WalletType int       — 钱包类型                                │
│  · Amount     int64     — 金额（最小单位整数，正数）               │
│  · FlowType   int       — 流水类型                                │
│  · Remark     string    — 备注                                   │
└─────────────────────────────────────────────────────────────────┘
```

## 4.2 单笔余额变更的内部流程（以 Credit 为例）

```
Credit(ctx, params) 内部流程:

  ┌─────────────────────── 数据库事务 ──────────────────────────┐
  │                                                             │
  │  Step 1: 幂等检查                                           │
  │    INSERT INTO wallet_flow (order_no, wallet_type, ...)     │
  │    如果 uk_order_wallet 冲突 → 说明已处理过 → 直接返回成功    │
  │                                                             │
  │  Step 2: 锁定钱包行                                         │
  │    SELECT * FROM user_wallet                                │
  │    WHERE user_id=? AND currency_id=? AND wallet_type=?      │
  │    FOR UPDATE                                               │
  │                                                             │
  │  Step 3: 记录变更前余额                                      │
  │    balanceBefore = wallet.balance                            │
  │                                                             │
  │  Step 4: 执行余额变更                                       │
  │    UPDATE user_wallet SET balance = balance + ?              │
  │    WHERE id = ?                                             │
  │                                                             │
  │  Step 5: 补全流水记录                                        │
  │    UPDATE wallet_flow SET balance_before=?, balance_after=?  │
  │    WHERE id = ?  (补全 Step 1 中创建的流水记录)               │
  │                                                             │
  └─────────────────────── 事务提交 ──────────────────────────┘

  Step 6: 缓存失效（事务提交后）
    del wallet:balance:{userId}:{currencyId}

  返回成功
```

## 4.3 为什么用"先写流水再更新余额"的顺序

```
这个顺序的设计意图：

  · 先 INSERT 流水：利用唯一索引实现幂等。如果这条 INSERT 成功，说明是首次执行。
    如果唯一索引冲突，说明是重复请求，直接跳出事务返回成功。
    这比"先查后写"更安全，避免了并发窗口期的重复执行。

  · 后 UPDATE 余额：在流水已经记录的前提下执行。
    如果余额更新失败，整个事务回滚，流水也一起回滚。

  · 最后补全流水的 balance_before/after：因为在 INSERT 流水时还不知道锁定后的
    实际余额值，所以先写入不完整的流水记录，锁定后补全。

  替代方案——"先查后写"的问题：
    1. 先 SELECT 检查 order_no 是否存在
    2. 不存在 → UPDATE 余额 → INSERT 流水
    问题：两个并发请求可能同时通过 Step 1 的检查，导致重复执行。
    虽然可以用分布式锁解决，但不如唯一索引这种数据库层面的保证可靠。
```

## 4.4 Debit 和 Freeze 的特殊处理

```
Debit（扣款）在 Step 4 有额外校验:
  · 锁定后检查: wallet.balance - wallet.frozen_amount >= amount
  · 不满足 → 事务回滚，返回"余额不足"
  · UPDATE user_wallet SET balance = balance - ? WHERE id = ?

Freeze（冻结）在 Step 4:
  · 锁定后检查: wallet.balance - wallet.frozen_amount >= amount
  · UPDATE user_wallet SET frozen_amount = frozen_amount + ? WHERE id = ?
  · 总余额不变，但可用余额减少

Unfreeze（解冻）在 Step 4:
  · 锁定后检查: wallet.frozen_amount >= amount
  · UPDATE user_wallet SET frozen_amount = frozen_amount - ? WHERE id = ?

提现成功时的 Debit 特殊处理:
  · 提现扣款扣的是冻结金额，不是可用余额
  · UPDATE user_wallet SET balance = balance - ?, frozen_amount = frozen_amount - ?
  · 所以提现成功的 Debit 和普通消费的 Debit 走不同的分支
```

## 4.5 Exchange（兑换）的事务设计

```
兑换涉及跨币种的 3 次余额变更，是最复杂的原子操作:

  ┌───────────────────── 单个数据库事务 ─────────────────────────┐
  │                                                              │
  │  ① 幂等检查: INSERT 3 条流水记录（order_no 相同，wallet_type 不同） │
  │     如果任一条冲突 → 说明已处理过 → 回滚返回成功                 │
  │                                                              │
  │  ② 锁定源币种钱包行（法币中心钱包）FOR UPDATE                   │
  │     检查可用余额 >= 扣款金额                                    │
  │                                                              │
  │  ③ 锁定目标币种钱包行（BSB 中心钱包）FOR UPDATE                 │
  │                                                              │
  │  ④ 锁定目标币种钱包行（BSB 奖励钱包）FOR UPDATE                 │
  │                                                              │
  │  ⑤ 执行 3 次余额变更:                                         │
  │     · 法币中心钱包 balance -= fromAmount                       │
  │     · BSB 中心钱包 balance += toAmount                         │
  │     · BSB 奖励钱包 balance += bonusAmount                      │
  │                                                              │
  │  ⑥ 补全 3 条流水的 balance_before/after                        │
  │                                                              │
  └───────────────────── 事务提交 ────────────────────────────────┘

  ⑦ 缓存失效: 清除两个币种的余额缓存

  加锁顺序（防止死锁）:
    固定按 (currency_id, wallet_type) 升序加锁
    例如: 法币(id=2)中心(type=1) → BSB(id=5)中心(type=1) → BSB(id=5)奖励(type=2)
```

# 五、并发控制架构

## 5.1 并发控制策略选型

```
对比三种方案:

┌──────────────┬──────────────────┬───────────────────┬──────────────────┐
│ 方案          │ 数据库行锁(FOR   │ 乐观锁(version)   │ Redis 分布式锁    │
│              │ UPDATE)          │                   │                  │
├──────────────┼──────────────────┼───────────────────┼──────────────────┤
│ 一致性保证    │ 强一致            │ 强一致(重试后)     │ 最终一致          │
│ 实现复杂度    │ 低               │ 中(需重试逻辑)     │ 中(需续期/释放)   │
│ 性能          │ 中               │ 高(无锁等待)       │ 高               │
│ 死锁风险      │ 有(需固定加锁顺序)│ 无                │ 有(需 TTL 兜底)  │
│ 适合场景      │ 写多/争用中等     │ 写少读多           │ 跨资源协调       │
│ 业界选择      │ 支付宝/蚂蚁账务  │ 电商库存           │ 分布式限流       │
├──────────────┼──────────────────┼───────────────────┼──────────────────┤
│ 本项目选择    │ ★ 主选           │ 预留(version字段)  │ 不采用           │
└──────────────┴──────────────────┴───────────────────┴──────────────────┘

选择数据库行锁的理由:
  · 钱包数据在单库中，行锁就能解决问题，不需要分布式锁
  · TiDB 兼容 MySQL 的 SELECT FOR UPDATE 语义
  · 实现最简单，出错概率最低
  · 不过度设计：当前业务并发量不需要乐观锁的高吞吐
  · version 字段预留了升级到乐观锁的能力，但当前不需要
```

## 5.2 加锁顺序规范（防死锁）

```
规则: 当需要锁定多行时，按 (currency_id ASC, wallet_type ASC) 顺序加锁

示例 — 兑换操作需要锁 3 行:
  · VND(id=2) 中心钱包(type=1)
  · BSB(id=5) 中心钱包(type=1)
  · BSB(id=5) 奖励钱包(type=2)

  加锁顺序: (2,1) → (5,1) → (5,2)  ✓
  而非:     (5,2) → (2,1) → (5,1)  ✗ 可能死锁

单笔 Credit/Debit 只锁一行，不存在死锁问题。
只有兑换操作需要多行加锁，必须遵循此规则。
```

## 5.3 热点账户问题（预留方案，当前不实施）

```
什么是热点账户:
  如果某个用户频繁收到充值回调（如平台运营账户），同一个钱包行被频繁锁定，
  后续请求会排队等待，导致性能瓶颈。

当前阶段的判断:
  · C 端用户的钱包操作并发度不高（一个用户不会同时充值+提现+兑换）
  · 被财务调用的 Credit/Debit 频率也不高（充值成功/提现完成的事件低频）
  · 当前不需要解决热点账户问题

如果未来出现:
  方案 A: 排队 + 批量合并（把多个 Credit 合并为一次 UPDATE）
  方案 B: 影子账户（拆分为多个子账户，分散锁争用）
  方案 C: 异步入账（先返回成功，后台异步操作余额）

这些方案留到真正出现性能瓶颈时再引入，现在不做。
```

# 六、缓存架构

## 6.1 缓存分层

```
                    读请求
                      │
                      ▼
              ┌──────────────┐
              │  Redis 缓存   │  ← 高频读的第一道屏障
              │  (热数据)     │
              └──────┬───────┘
                     │ cache miss
                     ▼
              ┌──────────────┐
              │  TiDB 数据库  │  ← 权威数据源
              │  (全量数据)   │
              └──────────────┘

写请求永远直接写 DB，写成功后失效缓存（Cache-Aside 模式）
```

## 6.2 各类数据的缓存策略

```
┌────────────────┬──────────┬──────────────┬──────────────────────────┐
│ 数据类型        │ 是否缓存  │ TTL          │ 失效时机                  │
├────────────────┼──────────┼──────────────┼──────────────────────────┤
│ 用户余额        │ 是       │ 5 分钟       │ Credit/Debit/Freeze 后    │
│ 可用币种列表    │ 是       │ 10 分钟      │ 币种 CRUD 后              │
│ 平台汇率        │ 是       │ 5 分钟       │ 定时任务覆盖更新           │
│ 用户当前币种    │ 是       │ 30 分钟      │ SwitchCurrency 后         │
│ 币种配置详情    │ 是       │ 10 分钟      │ 更新币种配置后             │
├────────────────┼──────────┼──────────────┼──────────────────────────┤
│ 订单数据        │ 否       │ —           │ —                        │
│ 冻结金额        │ 否       │ —           │ —                        │
│ 流水记录        │ 否       │ —           │ —                        │
└────────────────┴──────────┴──────────────┴──────────────────────────┘

不缓存的原因:
  · 订单数据：写多读少，且状态频繁变更，缓存收益低
  · 冻结金额：必须从 DB 读取保证准确性，涉及资金安全
  · 流水记录：append-only 数据，查询走 DB 索引即可
```

## 6.3 余额缓存的一致性保证

```
问题: 缓存余额和数据库余额可能不一致

方案: 写后失效 + 短 TTL 双重保证

  Credit/Debit 执行后:
    1. DB 事务提交成功
    2. 立即 DEL 缓存 key（写后失效）
    3. 下次读请求 cache miss 时从 DB 加载最新值

  如果 DEL 失败（极端情况）:
    · TTL=5min 后缓存自动过期
    · 5 分钟内可能读到旧余额
    · 这是可以接受的：充值/提现的结果通过订单状态判断，不依赖余额展示

  不采用"写后更新缓存"的原因:
    · 并发写入时可能把旧值覆盖新值
    · 删除比更新更安全（宁可 cache miss 重新加载）
```

## 6.4 Redis Key 规范

```
wallet:bal:{userId}:{currencyId}           — 余额汇总（中心+奖励+冻结）
wallet:cur:list                            — 启用的币种列表
wallet:cur:info:{currencyId}               — 单个币种配置
wallet:rate:{currencyId}                   — 平台汇率
wallet:user:cur:{userId}                   — 用户当前选中的币种 ID
```

# 七、安全架构

## 7.1 资金安全的四道防线

```
第一道: 幂等（防重复操作）
  · wallet_flow 表的 uk_order_wallet 唯一索引
  · 数据库层面的硬保证，比应用层判断可靠
  · 任何导致余额变更的操作都必须经过此检查

第二道: 事务（防部分失败）
  · 余额变更 + 流水记录在同一个 DB 事务中
  · 要么全部成功，要么全部回滚
  · 不存在"钱加了但没记录"或"记录了但钱没加"

第三道: 行锁（防并发冲突）
  · SELECT FOR UPDATE 锁定钱包行后再操作
  · 两个并发的 Credit 不会导致余额丢失更新
  · 固定加锁顺序防死锁

第四道: 对账（防积累误差）
  · wallet_flow 表记录每次变更前后余额
  · 可以通过流水反算：初始余额 + 所有流水 = 当前余额
  · 定期对账脚本校验一致性（后期补充）
```

## 7.2 敏感数据处理

```
需要考虑的敏感字段:
  · 提现收款信息（银行卡号、电子钱包账号、USDT 地址）

处理方式:
  · 存储: 使用 AES 加密后存储（common/pkg/utils 已有 AES 工具）
  · 展示: B 端展示时部分脱敏（如银行卡号只展示后 4 位）
  · 传输: HTTPS + IDL string 类型（不在日志中打印明文）

  参考现有项目的加密字段映射:
  > common/pkg/db/cmd/generate.go 中 encryptedFields 定义了 user_kyc 表的
  > phone/name/id_number 字段使用 utils.EncryptedString 自动加解密。
  > wallet 的收款信息可以参照此模式。
```

## 7.3 审计日志

```
两层审计:

第一层: wallet_flow 表（细粒度，每笔余额变更）
  · 自动记录 balance_before/balance_after
  · 谁操作的（通过 context 中的 UserID）
  · 什么时候操作的（create_at）
  · 关联哪个业务订单（order_no）

第二层: ser-blog 审计日志（粗粒度，管理员操作）
  · B 端管理操作（币种 CRUD、管理员查看用户钱包）走 oneway 审计上报
  · 遵循现有工程的 rpc.BLogClient().AddActLog() 模式

RPC 被调操作（Credit/Debit 等）不需要上报 ser-blog:
  · 因为调用方（财务模块）会在自己那边记录审计
  · 钱包侧有 wallet_flow 详细记录，不重复审计
```

# 八、接口分类与路由架构

## 8.1 接口按安全等级分类

```
等级 S（核心资金操作）:
  · Credit / Debit / Freeze / Unfreeze / Exchange
  · 特征: 涉及余额变更，必须走 ledger 层，必须有幂等保证

  · GetWalletInfo / Records / GetExchangeRate / 币种配置查询
  · 特征: 只读，无副作用，可以走缓存
  · 错误处理: 缓存 miss 兜底查 DB

等级 C（管理配置操作）:
  · 币种 CRUD / 汇率配置
  · 特征: 低频，B 端操作，标准 CRUD
  · 错误处理: 标准的参数校验 + 业务规则校验
```

# 九、跨模块集成架构

## 9.1 与财务模块的集成契约

```
通信方式: 同步 RPC（与现有工程一致，不引入消息队列）

钱包暴露给财务的 RPC 方法（被调）:
┌──────────────────┬─────────────────────────────────────────────────┐
│ 方法              │ 契约要点                                         │
├──────────────────┼─────────────────────────────────────────────────┤
│ QueryBalance     │ 入: userId, currencyId                           │
│                  │ 出: 中心余额, 奖励余额, 冻结金额, 可用余额         │
│                  │ 只读，无副作用                                    │
├──────────────────┼─────────────────────────────────────────────────┤
│ Credit           │ 入: orderNo, userId, currencyId, walletType,     │
│                  │     amount, flowType, remark                     │
│                  │ 出: success                                      │
│                  │ 幂等: 相同 orderNo+walletType 重复调用返回成功     │
│                  │ 失败: 只有系统异常时才返回失败                      │
├──────────────────┼─────────────────────────────────────────────────┤
│ Debit            │ 入: 同 Credit                                    │
│                  │ 出: success                                      │
│                  │ 幂等: 同 Credit                                  │
│                  │ 失败: 余额不足返回明确错误码                       │
├──────────────────┼─────────────────────────────────────────────────┤
│ Freeze           │ 入: orderNo, userId, currencyId, amount          │
│                  │ 出: success                                      │
│                  │ 失败: 可用余额不足返回明确错误码                    │
├──────────────────┼─────────────────────────────────────────────────┤
│ Unfreeze         │ 入: orderNo, userId, currencyId, amount          │
│                  │ 出: success                                      │
│                  │ 失败: 冻结金额不足返回明确错误码                    │
└──────────────────┴─────────────────────────────────────────────────┘

关键约定:
  · orderNo 的唯一性由调用方保证（财务模块用自己的订单号）
  · 所有方法都是幂等的：重复调用不会产生副作用
  · 超时时间：RPC 默认 3s，余额操作通常在 50ms 内完成
  · 错误处理：调用方收到超时/系统错误时可以安全重试
```

## 9.2 钱包调用财务的集成方式

```
钱包调用财务的 RPC 方法（调出）:
  · GetDepositMethods / GetWithdrawMethods — 透传返回
  · MatchDepositChannel / CreateChannelOrder — 充值核心链路
  · SubmitWithdrawRequest — 提现核心链路

集成策略:
  1. 在 ser 层定义 interface（面向接口编程）
  2. 实现真实的 RPC 调用版本
  3. 实现 Mock 版本（用于独立开发和单元测试）
  4. 通过依赖注入切换

示意:
  type FinanceClient interface {
      GetDepositMethods(ctx, req) (resp, error)
      MatchDepositChannel(ctx, req) (resp, error)
      // ...
  }

  type realFinanceClient struct{}    // 真实 RPC
  type mockFinanceClient struct{}    // Mock 数据

好处:
  · 开发阶段不依赖财务模块的进度
  · 单元测试可以 Mock 所有外部依赖
  · 联调时只需切换注入即可
```

# 十一、架构约束与原则

## 11.1 刚性约束（不可违反）

```
① 所有余额变更必须经过 ledger 层
   · ser 层不允许直接 UPDATE user_wallet 的 balance/frozen_amount 字段
   · 唯一例外：wallet 初始化（SwitchCurrency 时懒创建余额为 0 的记录）

② 所有余额变更必须在事务内完成
   · 余额 UPDATE + 流水 INSERT 在同一个事务中
   · 不允许事务外操作余额

③ 所有余额变更必须有流水记录
   · wallet_flow 表是审计和对账的唯一依据
   · 不允许绕过流水直接修改余额

④ 所有会导致余额变更的 RPC 方法必须幂等
   · 基于 orderNo + walletType 的唯一索引保证
   · 重复调用返回成功，不产生副作用

⑤ 余额和冻结金额不允许为负
   · 在锁定行后、执行扣减前检查
   · 不满足条件直接返回错误，不执行操作
```

## 11.2 柔性原则（推荐遵循）

```
① 缓存只加速读，不参与写路径
   · 写操作永远直接写 DB
   · 缓存在写成功后失效
   · 缓存 miss 时从 DB 加载

② 外部 RPC 失败不应导致资金问题
   · 充值流程中，财务 RPC 失败 → 不创建充值订单 → 不影响余额
   · 提现流程中，财务 RPC 失败 → 不冻结金额 → 不影响余额
   · KYC 服务不可用 → 降级策略，不阻断所有提现

③ 宁可拒绝操作也不执行错误操作
   · 余额不确定时不扣款
   · 幂等不确定时不执行
   · 网络超时时返回明确错误让调用方重试
```

# 十二、不做什么（不过度设计）

```
明确不在当前阶段引入的设计:

✗ 分布式事务（TCC/SAGA）
  理由: 钱包数据在同一个 TiDB 库中，本地事务足够

✗ 分库分表
  理由: TiDB 本身是分布式数据库，自动处理数据分片

✗ 消息队列驱动的余额操作
  理由: 增加复杂度且现有工程全部是同步 RPC

✗ 复式记账（双边记账）
  理由: 业务场景不需要（没有平台账户和用户账户之间的资金转移核算）

✗ Redis 余额缓存写入
  理由: 余额以 DB 为准，Redis 只做读缓存，不参与写

✗ 热点账户优化（影子账户/异步合并）
  理由: 当前并发量不需要，预留了 version 字段未来可升级

✗ 独立的幂等服务/去重表
  理由: wallet_flow 表的唯一索引已经承担了幂等功能

✗ 独立的对账服务
  理由: 当前通过流水表可以手动或脚本对账，不需要独立服务

以上每一项都是"可以但不需要"的设计。
如果未来业务规模增长到需要的程度，可以在现有架构上渐进式引入。
```

# 总结

```
架构核心思想:

  "以数据库事务为核心，以流水表为载体，以行锁为保障，
   用最简单的手段解决资金安全问题。"

关键架构决策:
  ① 引入 ledger 层收敛所有余额操作 → 资金安全的单一入口
  ② 流水表唯一索引实现幂等 → 一石二鸟（审计 + 防重）
  ③ 数据库行锁处理并发 → 简单可靠不过度设计
  ④ 金额用最小单位整数 → 杜绝浮点精度问题
  ⑤ 本地事务足够 → 不引入分布式事务的复杂度
  ⑥ 缓存只加速读 → 写路径不经过 Redis
  ⑦ 不做复式记账 → 业务复杂度不需要
  ⑧ 预留升级空间 → version 字段、可插拔的财务接口

不过度设计:
  不搞分布式事务、不搞消息队列驱动、不搞热点账户优化、
  不搞复式记账、不搞独立幂等服务——
  这些都是"可以但当前不需要"的东西。

  简单的方案如果能解决问题，就是最好的方案。
```
