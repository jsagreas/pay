## 一、架构设计原则

### 1.1 六条设计原则

| 序号 | 原则 | 说明 | 在本系统中的体现 |
|------|------|------|-----------------|
| 1 | **遵循现有工程模式** | 不创造新架构范式，复用已验证的技术栈和代码组织方式 | 沿用 CloudWeGo (Kitex+Hertz) + GORM Gen + ETCD + Redis 全套 |
| 2 | **资金安全第一** | 任何架构决策在"性能"与"安全"冲突时，优先安全 | 幂等、事务、防重、审计流水，不可妥协 |
| 3 | **不过度设计** | 只解决当前需求确定存在的问题，不为假想的未来需求增加复杂度 | 不引入分布式事务框架、不引入事件溯源、不引入额外消息中间件（除非证明必须） |
| 4 | **数据库是最后防线** | 缓存可以丢失，消息可以重试，但 DB 必须是最终一致的 truth | 余额变更 + 流水写入必须在同一个 DB 事务内完成 |
| 5 | **最小依赖原则** | 新模块尽可能少地依赖外部模块，减少级联故障面 | 核心资金操作(E1~E6)零外部依赖，纯本地 DB 操作 |
| 6 | **渐进式复杂度** | 先用最简单的方案跑通，遇到瓶颈再升级 | 先用数据库行锁保并发安全，后续有瓶颈再考虑 Redis 分布式锁 |

### 1.2 业界参考

本架构设计参考以下业界成熟模式：

- **支付系统三层架构**（接入层 → 业务层 → 核心账务层）— 来自 Stripe、支付宝等支付系统的公开设计
- **钱包账务双写模式**（余额变更 + 流水记录同事务）— 金融系统标准做法
- **幂等性设计**（业务唯一键去重）— 所有涉及资金变动的系统必备
- **冻结-确认-解冻三阶段模式**（TCC 的简化版，不引入 TCC 框架）— 提现场景标准做法

### 2.2 架构分层说明

| 层次 | 组件 | 职责 | 技术选型 |
|------|------|------|---------|
| **接入层** | gate-font / gate-back | HTTP路由、认证鉴权、IP过滤、请求转发 | Hertz HTTP |
| **服务层** | ser-wallet | 业务逻辑、领域规则、RPC 暴露 | Kitex RPC |
| **核心账务层** | WalletCoreService(E1~E6) | 余额变更、流水记录、幂等控制 | Go + GORM + DB事务 |
| **持久层** | TiDB | 数据存储、事务保障、行级锁 | TiDB (MySQL兼容) |
| **缓存层** | Redis | 配置缓存、幂等辅助、分布式锁(备用) | Redis UniversalClient |
| **配置层** | ETCD | 服务注册发现、配置热更新、功能开关 | ETCD v3 |
| **定时层** | XXL-JOB | 汇率定时刷新 | XXL-JOB |

### 2.3 关键架构特征

```
① 两层网关 + 单一服务
   gate-font(C端) ──RPC──→ ser-wallet
   gate-back(B端) ──RPC──→ ser-wallet
   单一 ser-wallet 服务内部通过 Package 划分领域

② 三层业务分离
   接口层(handler.go) → 业务层(ser/*.go) → 仓储层(rep/*.go)
   每层职责明确，不跨层调用

③ 核心账务层独立
   WalletCoreService 是系统最关键的一层
   供内部业务复用 + 供外部财务模块 RPC 调用
   设计上无任何外部依赖，纯本地 DB 操作
```

### 3.2 上下文间通信规则

| 通信方向 | 方式 | 说明 |
|---------|------|------|
| 币种配置 → 钱包业务 | **进程内方法调用** | 同一个 ser-wallet 服务内部，直接调用 Service 方法 |
| 钱包业务 → 财务模块 | **Kitex RPC** | 跨服务调用，通过 ETCD 服务发现 |
| 财务模块 → 钱包业务 | **Kitex RPC** | 跨服务调用，财务调用 E1~E6 |
| 钱包业务 → 用户模块 | **Kitex RPC** | 跨服务调用，查 KYC/用户信息 |

### 3.3 为什么不拆成两个服务

> 依据：实现思路c-3.md 第一节先决问题

**决策：币种配置和钱包放在同一个 `ser-wallet` 服务中，内部分 Package。**

| 考量维度 | 同一服务 | 拆两个服务 |
|---------|---------|-----------|
| **调用开销** | 进程内方法调用，纳秒级 | RPC 调用，毫秒级 + 网络开销 |
| **事务一致性** | 同 DB 实例，原生事务 | 需分布式事务或补偿 |
| **基础设施成本** | 一套 ETCD 配置、一个端口、一个 DB 连接池 | 两套全部独立 |
| **工程先例** | ser-app 内部有 banner + category + enum 等多领域 | 工程中无此先例 |
| **代码隔离** | 通过 Go Package 隔离，import 路径清晰 | 进程级隔离 |
| **运维复杂度** | 一个服务部署 | 两个服务部署、监控、告警 |

**结论**：在当前团队规模（你一个人负责）和业务复杂度下，合并为一个服务是最优解。拆分带来的解耦收益不足以覆盖运维成本增加。

### 4.3 Service 层内部依赖关系

```
┌──────────────────────────────────────────────────┐
│                   handler.go                      │
│                                                  │
│  持有以下 Service 实例（main.go 中依赖注入）：     │
│                                                  │
│  currencyService    ← 币种配置（B端+C端+RPC）    │
│  rateService        ← 汇率管理                   │
│  walletCoreService  ← E1~E6 核心资金操作         │
│  balanceService     ← W1~W3 余额+币种            │
│  rechargeService    ← W4~W11 充值                │
│  exchangeService    ← W12~W13 兑换               │
│  withdrawService    ← W14~W16 提现               │
│  accountService     ← W17~W22 账户管理           │
│  recordService      ← W23~W27 记录               │
└──────────────────────────────────────────────────┘

Service 层内部调用关系（单向依赖，无循环）：

  currencyService ← rateService (汇率需要读币种配置)
        ↑
  currencyService ← balanceService (余额查询需要币种精度)
        ↑
  walletCoreService ← exchangeService (兑换复用 E2/E3)
        ↑
  walletCoreService ← withdrawService (提现复用 E4/E5)
        ↑
  walletCoreService ← rechargeService (充值回调复用 E2)

关键设计：
  walletCoreService 不依赖任何其他 Service
  它是依赖链的最底层，只依赖 rep 层
  这保证了核心资金操作的独立性和可测试性
```

### 4.4 handler.go 结构设计

```go
// handler.go — 纯委派层
type WalletServiceImpl struct {
    // 币种配置上下文
    currencyService   *ser.CurrencyService
    rateService       *ser.RateService
    // 钱包业务上下文
    walletCoreService *ser.WalletCoreService
    balanceService    *ser.BalanceService
    rechargeService   *ser.RechargeService
    exchangeService   *ser.ExchangeService
    withdrawService   *ser.WithdrawService
    accountService    *ser.AccountService
    recordService     *ser.RecordService

### 5.3 核心表设计

#### 5.3.1 wallet_balance — 钱包余额表 ⚠️ 最核心

> 依据：需求理解c-3.md 2.1.2节 五种子钱包

```
设计原则：
  ① 一行 = 一个用户 + 一个币种 + 一个子钱包类型的余额
  ② 唯一约束 (user_id, currency_code, wallet_type) 保证不会出现重复行
  ③ available_balance 和 frozen_balance 都用 DECIMAL，精度足够覆盖法币和加密货币
  ④ 不使用乐观锁版本号（通过 SQL 原子更新或 SELECT FOR UPDATE 保证并发安全）

核心字段：
  id                BIGINT AUTO_INCREMENT PRIMARY KEY
  user_id           BIGINT NOT NULL             -- 用户ID
  currency_code     VARCHAR(16) NOT NULL        -- 币种代码 (USDT/VND/IDR/THB/BSB)
  wallet_type       TINYINT NOT NULL            -- 子钱包类型 (1=CENTER 2=REWARD 3=STREAMER 4=AGENT)
  available_balance DECIMAL(24,8) NOT NULL DEFAULT 0  -- 可用余额
  frozen_balance    DECIMAL(24,8) NOT NULL DEFAULT 0  -- 冻结余额
  create_by         BIGINT NOT NULL DEFAULT 0
  create_at         BIGINT NOT NULL DEFAULT 0
  update_by         BIGINT NOT NULL DEFAULT 0
  update_at         BIGINT NOT NULL DEFAULT 0
  delete_at         BIGINT NOT NULL DEFAULT 0

索引：
  UNIQUE KEY uk_user_currency_wallet (user_id, currency_code, wallet_type, delete_at)
  INDEX idx_user_id (user_id)

设计理由：
  - DECIMAL(24,8)：24位总精度、8位小数。覆盖法币2位和加密货币8位，整数部分16位足以承载任何金额
  - wallet_type 用 TINYINT 而非 VARCHAR：节省存储、索引更快，代理钱包(4)预留
  - 场馆钱包(5)预留但当前不创建行，按需扩展
  - available_balance 不允许为负数（通过 SQL 条件或应用层校验）
```

#### 5.3.2 wallet_transaction — 资金流水表 ⚠️ 最核心

> 依据：需求理解c-3.md 3.6节 对账一致性

```
设计原则：
  ① 只写不改（INSERT ONLY），是系统审计和对账的核心依据
  ② (order_no + operation_type) 唯一索引，天然保证幂等
  ③ 记录操作前余额和操作后余额，形成完整的余额变动链条

核心字段：
  id                BIGINT AUTO_INCREMENT PRIMARY KEY
  user_id           BIGINT NOT NULL
  currency_code     VARCHAR(16) NOT NULL
  wallet_type       TINYINT NOT NULL
  operation_type    TINYINT NOT NULL            -- 操作类型 (见下方枚举)
  amount            DECIMAL(24,8) NOT NULL      -- 操作金额 (正数)
  direction         TINYINT NOT NULL            -- 方向 1=入账 2=出账
  before_balance    DECIMAL(24,8) NOT NULL      -- 操作前余额
  after_balance     DECIMAL(24,8) NOT NULL      -- 操作后余额
  order_no          VARCHAR(32) NOT NULL        -- 关联订单号
  source            VARCHAR(64) DEFAULT ''      -- 来源标识 (RECHARGE/WITHDRAW/EXCHANGE/MANUAL 等)
  remark            VARCHAR(256) DEFAULT ''     -- 备注
  create_at         BIGINT NOT NULL DEFAULT 0
  delete_at         BIGINT NOT NULL DEFAULT 0

索引：
  UNIQUE KEY uk_order_operation (order_no, operation_type)  -- 幂等键
  INDEX idx_user_currency (user_id, currency_code)
  INDEX idx_create_at (create_at)

operation_type 枚举：
  1=充值入账  2=充值奖金入账  3=提现冻结  4=提现解冻  5=提现确认扣除
  6=兑换扣款  7=兑换入账  8=兑换赠送  9=人工加款  10=人工减款
  11=补单入账  ...可扩展

设计理由：
  - INSERT ONLY：流水表绝不 UPDATE 和 DELETE，每条记录就是一个不可变的事实
  - before_balance + after_balance：形成链条，可以验证任何时间点的余额正确性
  - 唯一索引 (order_no, operation_type)：同一个订单+同一种操作只能有一条记录
    例如：充值订单C001 + 操作类型=充值入账 → 唯一 → 重复调用 E2 不会产生第二条
  - amount 始终为正数，方向由 direction 字段表示
```

## 六、资金安全架构

> 这是整个架构设计中最关键的部分。资金系统的核心不是性能，是安全。

### 6.1 三道防线模型

```
┌─────────────────────────────────────────────────────────┐
│                    资金安全三道防线                         │
│                                                         │
│  第一道防线：幂等控制                                      │
│  ────────────────                                       │
│  同一笔操作无论调用多少次，结果只生效一次                    │
│  实现：流水表 (order_no + operation_type) 唯一索引          │
│                                                         │
│  第二道防线：事务一致性                                     │
│  ────────────────                                       │
│  余额变更和流水写入必须在同一个 DB 事务中                    │
│  不会出现"余额变了但流水没写"或"流水写了但余额没变"           │
│  实现：GORM Gen query.Q.Transaction()                    │
│                                                         │
│  第三道防线：审计可追溯                                     │
│  ────────────────                                       │
│  每一笔资金变动都有完整的流水记录（INSERT ONLY）             │
│  记录操作前余额和操作后余额，形成可验证的链条                 │
│  对账脚本可以随时验证：余额 = 初始值 + 所有流水的净变动       │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 6.2 幂等机制详细设计

> 依据：实现思路c-3.md 6.3节

```
幂等流程（以 E2 CreditWallet 为例）：

调用方传入：user_id + currency_code + wallet_type + amount + order_no + source

Step 1: 检查流水表
  SELECT 1 FROM wallet_transaction
  WHERE order_no = ? AND operation_type = ?
  LIMIT 1

Step 2a: 已存在 → 说明已处理过 → 直接返回成功（不报错不重复操作）
Step 2b: 不存在 → 继续执行

Step 3: 在同一个事务中
  ① UPDATE wallet_balance SET available_balance = available_balance + ? ...
  ② INSERT INTO wallet_transaction (order_no, operation_type, amount, ...)

Step 4: 唯一索引 (order_no, operation_type) 作为最后兜底
  如果 Step 1 查询时并发通过（极端情况），Step 3② 的 INSERT 会触发唯一索引冲突
  → 事务回滚 → 重新走 Step 1 → 发现已存在 → 返回成功

双重保障：
  应用层检查 (Step 1) + 数据库唯一索引 (Step 4)
  即使应用层检查存在并发窗口，数据库层面也能兜住
```

### 6.3 事务边界设计

> 依据：实现思路c-3.md 6.5节

```
以下操作必须在同一个 DB 事务内完成：

┌─────────────────────────────────────────────────┐
│ E2 CreditWallet (加款事务)                       │
│ BEGIN                                           │
│   ① 查流水表确认幂等                              │
│   ② UPDATE wallet_balance SET available += amount│
│   ③ INSERT wallet_transaction (记录流水)          │
│ COMMIT                                          │
└─────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────┐
│ E4 FreezeBalance (冻结事务)                       │
│ BEGIN                                           │
│   ① 查流水表确认幂等                              │
│   ② SELECT ... FOR UPDATE (锁定余额行)           │
│   ③ 校验 available_balance >= freeze_amount      │
│   ④ UPDATE wallet_balance SET                    │
│       available -= amount, frozen += amount      │
│   ⑤ INSERT wallet_transaction (记录流水)          │
│ COMMIT                                          │
└─────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────┐
│ W13 兑换事务（最复杂的单一事务）                    │
│ BEGIN                                           │
│   ① SELECT ... FOR UPDATE (锁定源币种余额行)      │
│   ② 校验余额充足                                  │
│   ③ UPDATE 源币种 available -= from_amount        │
│   ④ UPDATE BSB中心钱包 available += to_amount     │
│   ⑤ UPDATE BSB奖励钱包 available += bonus_amount  │
│   ⑥ INSERT wallet_transaction × 3条流水           │
│   ⑦ INSERT exchange_record                       │
│ COMMIT                                          │
└─────────────────────────────────────────────────┘

设计原则：
  - 所有事务都在单个 DB 实例内完成，无需分布式事务
  - 不引入 TCC 框架，用原生 GORM 事务即可
  - 事务粒度尽量小，只包含必须原子的操作
```

### 6.4 提现的冻结-提交一致性

> 依据：链路关系c-3.md 3.4节 提现接口组

```
提现流程中存在跨服务操作：冻结(本地) → 提交财务(RPC)

问题：如果冻结成功但提交财务 RPC 失败，资金被锁怎么办？

解决方案：补偿模式（非分布式事务）

W15 创建提现订单的执行序列：
  Step 1: 本地事务 → 冻结余额 + 写入提现订单(status=FREEZED)
  Step 2: 调用财务 RPC F4(SubmitWithdrawOrder)
  Step 3a: F4 成功 → 更新本地订单 status=SUBMITTED → 完成
  Step 3b: F4 失败 → 本地事务 → 解冻余额 + 更新订单 status=SUBMIT_FAILED

伪代码：
  // Step 1: 冻结（本地事务）
  err := freezeAndCreateOrder(ctx, req)
  if err != nil { return err }

  // Step 2: 提交财务
  resp, err := rpc.FinanceClient().SubmitWithdrawOrder(ctx, submitReq)

  // Step 3: 根据结果处理
  if err != nil || !resp.Success {
      // 补偿：解冻
      unfreezeErr := unfreezeAndUpdateOrder(ctx, orderNo, SUBMIT_FAILED)
      if unfreezeErr != nil {
          // 解冻也失败：记录告警日志，人工介入
          lg.CtxErrorf(ctx, "提现补偿解冻失败 orderNo=%s err=%v", orderNo, unfreezeErr)
      }
      return err
  }
  updateOrderStatus(ctx, orderNo, SUBMITTED)

关键设计决策：
  不使用分布式事务框架（Seata/DTM等），原因：
  ① 提现场景频率不高（不是秒级高并发）
  ② 补偿逻辑简单明确（失败就解冻）
  ③ 引入分布式事务框架的运维成本 >> 手写补偿逻辑
  ④ 遵循"不过度设计"原则
```

### 6.5 防超额扣款

```
扣款操作（E3/E4）必须防止余额变为负数：

方案：SQL 层面条件更新

-- E3 DebitWallet (减款)
UPDATE wallet_balance
SET available_balance = available_balance - ?
WHERE user_id = ? AND currency_code = ? AND wallet_type = ?
  AND available_balance >= ?    -- 余额不足时 affected rows = 0
  AND delete_at = 0

-- E4 FreezeBalance (冻结)
UPDATE wallet_balance
SET available_balance = available_balance - ?,
    frozen_balance = frozen_balance + ?
WHERE user_id = ? AND currency_code = ? AND wallet_type = ?
  AND available_balance >= ?    -- 余额不足时 affected rows = 0
  AND delete_at = 0

判断 affected rows：
  = 1 → 操作成功
  = 0 → 余额不足或记录不存在 → 返回 BizErr

特殊规则（E3）：
  需求要求"人工减款余额不足时实际扣款=当前余额"
  → 这种场景走 SELECT FOR UPDATE 读取当前余额，再决定实际扣款金额
```

## 七、并发控制架构

### 7.1 并发场景分析

| 场景 | 并发源 | 频率 | 风险等级 |
|------|--------|------|---------|
| 同一用户多次充值回调 | 三方通道重复回调 | 低频但一定会发生 | 高（重复入账） |
| 同一用户同时充值和提现 | 用户多端操作 | 中频 | 中（余额竞争） |
| 同一用户同时兑换 | 用户快速点击 | 低频 | 高（余额双花） |
| 财务同时人工加款和减款 | 管理员误操作 | 极低频 | 中 |
| 多用户批量充值入账 | 充值高峰 | 高频但不同用户 | 低（不同行无竞争） |

### 7.2 并发控制策略

> 依据：实现思路c-3.md 6.4节

```
策略分两层：

第一层：数据库行级锁（核心手段）
────────────────────────────
  方案A：原子 SQL 更新（优先使用）
    UPDATE wallet_balance
    SET available_balance = available_balance + ?
    WHERE user_id = ? AND currency_code = ? AND wallet_type = ?

    数据库自动加行级锁，同一行的并发 UPDATE 会串行执行
    适用场景：CreditWallet（加款）、ConfirmFrozenDeduction（确认扣除）

  方案B：SELECT FOR UPDATE 悲观锁
    BEGIN;
    SELECT * FROM wallet_balance WHERE ... FOR UPDATE;
    -- 业务逻辑校验（如余额是否充足）
    UPDATE wallet_balance SET ...;
    INSERT INTO wallet_transaction ...;
    COMMIT;

    适用场景：DebitWallet（需要判断余额是否充足）、FreezeBalance（需要判断+双字段更新）、
             兑换（需要跨币种操作）

第二层：幂等唯一索引（兜底手段）
────────────────────────────
  wallet_transaction 表的 UNIQUE(order_no, operation_type)
  即使并发请求都通过了应用层幂等检查，DB 唯一约束也会阻止重复写入

不使用 Redis 分布式锁：
  当前阶段不需要。原因：
  ① 所有资金操作的并发竞争发生在同一行数据上，DB 行锁天然解决
  ② Redis 分布式锁引入了额外的故障点（Redis 宕机 → 锁失效）
  ③ 渐进式复杂度原则：先用 DB 锁，遇到性能瓶颈再升级
  ④ 如果后续确实需要（如余额缓存一致性），再引入 Redis Lua 脚本
```

### 7.3 各操作的并发安全总结

| 操作 | 并发控制方式 | 幂等方式 | 事务范围 |
|------|------------|---------|---------|
| E1 GetBalance | 无需（纯读） | 不适用 | 无事务 |
| E2 CreditWallet | 原子 SQL 更新 | 流水唯一索引 | 余额更新 + 流水写入 |
| E3 DebitWallet | SELECT FOR UPDATE | 流水唯一索引 | 余额更新 + 流水写入 |
| E4 FreezeBalance | SELECT FOR UPDATE | 流水唯一索引 | available减 + frozen增 + 流水 |
| E5 UnfreezeBalance | SELECT FOR UPDATE | 流水唯一索引 | frozen减 + available增 + 流水 |
| E6 ConfirmFrozenDeduction | 原子 SQL 更新 | 流水唯一索引 | frozen减 + 流水 |
| W13 兑换 | SELECT FOR UPDATE | 兑换流水唯一 | 多表操作 + 多条流水 |

## 九、缓存架构设计

### 9.1 缓存决策矩阵

> 依据：实现思路c-3.md 第十一节

| 数据 | 是否缓存 | 理由 | 策略 |
|------|---------|------|------|
| **钱包余额** | **不缓存** | 写密集、强一致性要求，缓存与 DB 不一致风险极高 | 直接读 DB |
| **资金流水** | **不缓存** | 只写不读（记录查询走分页 DB 查询） | 直接读 DB |
| 已启用币种列表 | 缓存 | 读多写少，B端启禁用操作低频 | Redis TTL 5~10分钟 |
| 币种配置详情 | 缓存 | 格式化余额每次都用，变更极低频 | Redis TTL 5~10分钟 |
| 当前平台汇率 | 缓存 | 每次创建订单都查，定时任务更新 | Redis TTL = 定时任务间隔 |
| 基准币种信息 | 缓存 | 一次设置永不变 | Redis TTL 长期 |
| 充值/提现订单状态 | 不缓存 | 状态变更频繁且关键 | 直接读 DB |

### 9.2 缓存实现模式

> 依据：工程认知c-3.md 7.5节缓存失效策略

```
遵循工程已有模式：

读操作 — 缓存旁路（Cache Aside）：
  func GetEnabledCurrencies(ctx) {
      // 1. 查缓存
      data := cache.Get(ctx, key)
      if data != nil { return data }
      // 2. 缓存未命中 → 查 DB
      data = repo.ListEnabled(ctx)
      // 3. 写缓存（空结果也缓存，防穿透）
      cache.Set(ctx, key, data, TTL)
      return data
  }

写操作 — 全量清除：
  func UpdateCurrency(ctx, req) {
      // 1. 更新 DB
      repo.Update(ctx, ...)
      // 2. 清除所有币种相关缓存
      cache.DelAll(ctx)
  }

缓存 Key 规范：
  wallet:currency:list          — 已启用币种列表
  wallet:currency:config:{code} — 单个币种配置
  wallet:rate:platform:{code}   — 平台汇率
  wallet:base:currency          — 基准币种信息
```

### 9.3 余额不缓存的设计理由

```
为什么不把余额放 Redis？

  ① 余额是高频写数据（每次充值/提现/兑换都会变）
  ② DB 更新成功但 Redis 更新失败 → 用户看到错误余额 → 资金纠纷
  ③ Redis 更新成功但 DB 更新失败 → 用户以为有钱但实际没有
  ④ 要保证 DB 和 Redis 双写一致性，需要额外的复杂机制（如 Lua 脚本、消息队列）
  ⑤ TiDB 单行读取性能足够（主键查询 < 1ms），不构成瓶颈

什么时候考虑缓存余额？
  当单用户余额查询 QPS > 1000 时（目前不会到达）
  到达时的方案：Redis + DB 双写 + Lua 脚本保证原子性
  遵循"渐进式复杂度"原则，当前不做
```

## 十二、部署与扩缩容

### 12.1 部署架构

```
ser-wallet 部署方式：

  与现有服务保持一致：
  ① 单实例部署（开发/测试阶段）
  ② 多实例部署（生产环境，通过 ETCD 服务注册自动负载均衡）

  多实例安全性：
  ─────────────
  ser-wallet 的所有资金操作通过 DB 行锁保证并发安全
  多实例场景下，两个实例同时处理同一用户的资金操作
  → 数据库行锁自动串行化，不需要应用层额外处理
  → 幂等唯一索引兜底，不需要 Redis 分布式锁

  Kitex 服务注册：
  ─────────────
  // main.go
  srv := walletservice.NewServer(impl,
      rpc.InitRpcServerParams(namesp.EtcdWalletService, port)...,
  )
  // 自动注册到 ETCD，客户端通过 ETCD 发现 + 负载均衡
```

### 12.2 扩缩容策略

```
当前阶段：单实例足够
─────────────────
  理由：
  - 用户量和交易量预估有限
  - Kitex RPC + TiDB 单实例可支撑数千 QPS
  - 币种配置是读多写少，更不构成瓶颈

扩容触发条件：
─────────────
  ① DB 连接池利用率持续 > 80% → 增加 ser-wallet 实例 + 提高连接池上限
  ② RPC 请求延迟 P99 > 500ms → 分析瓶颈（DB慢查询?汇率API?）
  ③ 充值/提现并发量激增 → 多实例（DB 行锁天然支持）

不需要考虑的扩容：
───────────────
  ① 数据库分库分表 — TiDB 本身支持水平扩展，应用层无需关心
  ② 读写分离 — 当前规模不需要
  ③ 消息队列削峰 — 资金操作需要同步返回结果，不适合异步
```

## 十三、架构约束与边界

### 13.1 架构约束（不可违反）

| 约束 | 说明 |
|------|------|
| **金额用 DECIMAL** | 全链路（DB/Go/IDL）使用 DECIMAL/string，禁止 float64 计算金额 |
| **余额变更必须有流水** | 不存在"只改余额不写流水"的代码路径 |
| **流水表只写不改** | INSERT ONLY，禁止 UPDATE/DELETE |
| **所有查询加软删除过滤** | `.Where(DeleteAt.Eq(0))` 不可遗漏 |
| **幂等键不可缺失** | 所有资金操作必须传入 order_no |
| **handler.go 零逻辑** | 纯委派，不做任何判断 |
| **不引入新中间件** | 不引入 Seata/DTM/新的消息队列，用现有基础设施 |

### 13.2 架构边界（不负责的范围）

```
本架构不覆盖：
  ① 财务模块内部架构（通道管理、审核流程、统计报表）— 他人负责
  ② 区块链监听服务（USDT链上确认）— 独立子系统，只提供回调接口
  ③ 三方支付通道对接细节 — 通道由财务模块管理，钱包只调用
  ④ 用户模块改造 — KYC/UserInfo 是已有接口，不需要改
  ⑤ 场馆钱包实现 — 预留字段，当前不实现
  ⑥ 稽核流水系统 — 需求中提到但不在钱包模块内
```

### 13.3 需要评审确认的架构决策点

| 决策点 | 当前建议 | 备选方案 | 需确认原因 |
|--------|---------|---------|-----------|
| 一个服务还是两个 | 一个 ser-wallet | 拆 ser-wallet + ser-coincfg | 团队可能有不同偏好 |
| 充值回调归属 | 大概率在财务模块 | 在钱包模块 | 影响回调验签和入账触发方式 |
| 余额精度 | DECIMAL(24,8) | DECIMAL(20,2) 法币单独 | 法币不需要8位小数 |
| 提现补偿方式 | 同步补偿（失败即解冻） | 异步补偿（定时任务扫描） | 依赖财务RPC的稳定性预期 |
| 兑换赠送来源 | 硬编码百分比 | ETCD 可配 | 运营是否需要灵活调整 |

## 十四、架构决策记录

### ADR-001: 服务合并

- **决策**：币种配置和钱包合并为单一 ser-wallet 服务
- **理由**：团队单人负责、数据关联紧密、工程先例(ser-app)支持、减少运维成本
- **后果**：内部通过 Package 隔离领域，如未来需要拆分可沿 Package 边界拆

### ADR-002: 不使用分布式事务框架

- **决策**：使用本地 DB 事务 + 同步补偿，不引入 Seata/DTM
- **理由**：提现场景频率低、补偿逻辑简单、运维成本高、不过度设计
- **后果**：提现失败补偿需要在代码中显式处理，极端失败场景需人工介入

### ADR-003: 不使用 Redis 分布式锁

- **决策**：使用 DB 行级锁（原子 SQL + SELECT FOR UPDATE）
- **理由**：资金竞争发生在同一行数据、DB 行锁天然串行化、减少故障点
- **后果**：如果后续需要缓存余额，需要重新评估锁策略

### ADR-004: 不使用消息队列处理充值入账

- **决策**：充值入账通过同步 RPC 调用 E2
- **理由**：入账需要实时反馈结果（成功/失败）、幂等机制已覆盖重试场景
- **后果**：如果三方回调量级极大（>万级/秒），可能需要引入 Kafka 削峰

### ADR-005: 余额不缓存

- **决策**：余额直接读 DB，不放 Redis
- **理由**：写密集、强一致性、TiDB 单行读性能足够、缓存一致性成本高
- **后果**：每次查询余额都是 DB 查询，如果用户频繁刷新可能有压力（可通过限流解决）

### ADR-006: 金额使用 DECIMAL + string

- **决策**：DB 用 DECIMAL(24,8)，Go 用 `shopspring/decimal` 库，IDL 用 string
- **理由**：float64 有精度丢失，金融系统行业标准
- **后果**：需要在各层做类型转换，Go 的 decimal 库有一定性能开销（可接受）

### 15.1 架构关键词

```
单服务多领域 | DB事务保一致 | 行锁保并发 | 幂等保安全 | 流水保可追溯

### 15.2 架构全景一句话

```
ser-wallet 是一个基于 CloudWeGo 体系的单体微服务，

对内通过方法调用被充值/兑换/提现业务复用。

┌────────────── 资金安全全链路 ──────────────────────┐
│                                                  │
│  请求入口                                          │
│    ↓                                             │
│  ① Token 认证（gate 层）                          │
│    ↓                                             │
│  ② 功能开关检查（ETCD，Service 层入口）            │
│    ↓                                             │
│  ③ 业务规则校验（余额充足、限额、KYC）              │
│    ↓                                             │
│  ④ 幂等检查（流水表查 order_no）                   │
│    ↓                                             │
│  ⑤ DB 事务执行（余额变更 + 流水写入）              │
│    ↓                                             │
│  ⑥ 唯一索引兜底（并发场景最后防线）                 │
│    ↓                                             │
│  ⑦ 操作日志 + 资金流水（审计追溯）                  │
│                                                  │
└──────────────────────────────────────────────────┘
```

---

> 本架构设计基于已有工程代码、需求文档、接口评估、链路关系、依赖关系、工程认知、流程思路、实现思路八份文档综合分析得出。
> 所有技术选型均沿用现有工程基座，不引入新的中间件或框架。
> 设计目标：安全优先、不过度设计、业界主流、遵循现有模式。
> 当前阶段：预评审，待评审确认后可进入 IDL 定义和编码阶段。
