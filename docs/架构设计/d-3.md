**架构决策说明**：

| 决策 | 选择 | 理由 |
|------|------|------|
| 是否新增独立网关/BFF | **否** | 现有 gate-font/gate-back 机制成熟，新增路由组即可，不过度设计 |
| 是否引入独立配置中心 | **否** | ETCD 已承担此职责，币种配置走 DB + Redis 缓存 |
| 是否引入消息中间件 | **复用 NATS JetStream** | 工程已有完整 NATS 封装（Publish/Subscribe/PullBatch），用于异步事件 |
| 是否引入 API Gateway | **否** | Hertz 网关已具备限流/鉴权/路由能力，不额外引入 Kong/Envoy |
| 是否独立部署币种配置 | **否** | 与钱包同进程，进程内调用零网络开销，业务高度内聚 |

## 1.2 架构核心原则

| 原则 | 含义 | 在本架构中的体现 |
|------|------|----------------|
| **会计恒等式** | 每笔变动必有对等的流水记录，余额 = 所有流水的累计结果 | t_wallet_transaction 流水表，balance_before/after |
| **事务原子性** | 余额变更与流水记录在同一 DB 事务中 | GORM Transaction + FOR UPDATE |
| **幂等性** | 同一操作重复执行结果不变 | (biz_type, order_no) 唯一索引 |
| **最终一致性** | 跨服务操作允许短暂不一致，但最终一致 | 同步 RPC + 失败重试 + 补单兜底 |
| **不过度设计** | 不引入当前不需要的组件 | 不用 Saga、不用分布式事务框架、不用独立账务引擎 |

## 2.2 为什么 account_ser 是核心

业界资金系统的标准做法是将**账户操作（余额增减）**从具体业务（充值/提现/投注）中抽离出来，形成独立的核心层：

```
                   ┌─────────────────────┐
                   │    account_ser.go   │  ← 账户核心（资金原子操作）
                   │                     │
                   │  Credit()           │  ← 通用入账
                   │  Debit()            │  ← 通用扣款
                   │  Freeze()           │  ← 冻结
                   │  Unfreeze()         │  ← 解冻
                   │  DebitFrozen()      │  ← 从冻结中扣除
                   │  BetDebit()         │  ← 优先级扣款（专用）
                   │  BetCredit()        │  ← 按比例返奖（专用）
                   │  GetBalance()       │  ← 余额查询
                   └──────────┬──────────┘
                              │
          ┌───────────────────┼───────────────────┐
          │                   │                   │
    deposit_ser          withdraw_ser        exchange_ser
    充值成功入账→         提现冻结→           法币扣款→
    Credit()             Freeze()            Debit() + Credit()
                         审核驳回→
                         Unfreeze()
                         出款成功→
                         DebitFrozen()

    ┌─────────────────────────────────────────────┐
    │  外部 RPC 调用方直接使用 account_ser 方法      │
    │                                              │
    │  财务→WalletCredit    = Credit()             │
    │  财务→WalletFreeze    = Freeze()             │
    │  游戏→BetDebit        = BetDebit()           │
    │  直播→GiftDebit       = Debit(中心钱包)       │
    │  活动→BonusCredit     = Credit(奖励钱包)      │
    └──────────────────────────────────────────────┘
```

**核心设计原则**：
- account_ser 是**唯一能修改余额的地方**，任何路径的资金变动都必须经过它
- 它不关心"为什么变" — 由调用方通过 `biz_type` 和 `order_no` 告知
- 它只关心"变多少、从哪个钱包变、是否幂等"
- 这是业界支付系统/钱包系统的标准分层（支付宝/微信支付/Stripe 都是类似模式）

# 第三部分：数据架构

## 3.1 数据库选型与使用方式

```
存储组件        角色                    使用方式
─────────      ────                    ────────
TiDB           主存储（唯一数据源）      wallet_db 独立数据库
               余额、流水、订单、配置    GORM + GORM Gen
                                        事务 + 行级锁

Redis          缓存 + 分布式控制         币种/汇率缓存
               (非数据源)               操作防重（SetNX）
                                        余额不缓存

ETCD           服务注册 + 基础设施配置   /slg/serv/wallet/port
               (非业务数据)             /slg/serv/wallet/db
                                        DB/Redis连接信息
```

**关键决策 — 余额存 TiDB，不存 Redis**：

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| 余额存 Redis，异步落库 | 读写极快 | 宕机丢数据、一致性无保障、对账困难 | **否决** |
| 余额存 TiDB，Redis 缓存余额 | 读快 | 缓存与 DB 不一致导致超扣或展示错误 | **否决** |
| **余额只存 TiDB，不缓存** | 强一致、对账可靠 | 每次读 DB | **采用** |

理由：资金数据的一致性优先级远高于读性能。TiDB 行级查询（主键/唯一索引）耗时 <1ms，C 端用户级 QPS 完全可承受。业界支付系统（支付宝/微信）核心账户余额也是 DB 实时读取。

## 3.2 核心表模型

### 3.2.1 表关系总览


### 3.2.2 t_wallet — 钱包余额主表（全系统最核心）

```sql
CREATE TABLE t_wallet (
    id            BIGINT       NOT NULL AUTO_INCREMENT,
    user_id       BIGINT       NOT NULL COMMENT '用户ID',
    currency_code VARCHAR(10)  NOT NULL COMMENT '币种编码(VND/IDR/THB/USDT/BSB)',
    wallet_type   TINYINT      NOT NULL COMMENT '钱包类型(1中心/2奖励/3主播/4代理/5场馆)',
    balance       DECIMAL(20,8) NOT NULL DEFAULT 0 COMMENT '可用余额',
    frozen        DECIMAL(20,8) NOT NULL DEFAULT 0 COMMENT '冻结金额',
    version       INT          NOT NULL DEFAULT 0 COMMENT '乐观锁版本号',
    create_by     BIGINT       NOT NULL DEFAULT 0,
    create_at     BIGINT       NOT NULL DEFAULT 0,
    update_by     BIGINT       NOT NULL DEFAULT 0,
    update_at     BIGINT       NOT NULL DEFAULT 0,
    delete_at     BIGINT       NOT NULL DEFAULT 0,
    PRIMARY KEY (id),
    UNIQUE KEY uk_user_currency_type (user_id, currency_code, wallet_type, delete_at),
    KEY idx_user_id (user_id)
) COMMENT '钱包余额主表';
```

**设计依据**：

| 字段 | 设计决策 | 理由 |
|------|---------|------|
| `DECIMAL(20,8)` | 统一精度 | USDT 需 8 位小数，法币整数只是展示层截断，存储层统一 |
| `balance + frozen` | 双字段模型 | 可提现余额 = balance - frozen，冻结不影响 balance 本身 |
| `version` | 乐观锁辅助 | 配合 FOR UPDATE 悲观锁双重保障 |
| `uk_user_currency_type` | 三维唯一 | 同一用户+同一币种+同一钱包类型只有一行 |
| `delete_at=0` | 项目规范 | 软删除标识，0 表示未删除 |

### 3.2.3 t_wallet_transaction — 资金流水表（对账核心）

```sql
CREATE TABLE t_wallet_transaction (
    id              BIGINT        NOT NULL AUTO_INCREMENT,
    user_id         BIGINT        NOT NULL,
    currency_code   VARCHAR(10)   NOT NULL,
    wallet_type     TINYINT       NOT NULL,
    direction       TINYINT       NOT NULL COMMENT '方向(1收入/2支出)',
    amount          DECIMAL(20,8) NOT NULL COMMENT '变动金额(正数)',
    balance_before  DECIMAL(20,8) NOT NULL COMMENT '变动前余额',
    balance_after   DECIMAL(20,8) NOT NULL COMMENT '变动后余额',
    frozen_before   DECIMAL(20,8) NOT NULL DEFAULT 0 COMMENT '变动前冻结',
    frozen_after    DECIMAL(20,8) NOT NULL DEFAULT 0 COMMENT '变动后冻结',
    biz_type        VARCHAR(32)   NOT NULL COMMENT '业务类型(deposit/withdraw/exchange/bet/gift/bonus/...)',
    order_no        VARCHAR(64)   NOT NULL COMMENT '业务订单号(幂等键)',
    remark          VARCHAR(256)  NOT NULL DEFAULT '',
    create_at       BIGINT        NOT NULL DEFAULT 0,
    delete_at       BIGINT        NOT NULL DEFAULT 0,
    PRIMARY KEY (id),
    UNIQUE KEY uk_biz_order (biz_type, order_no, wallet_type),
    KEY idx_user_currency (user_id, currency_code, create_at),
    KEY idx_order_no (order_no)
) COMMENT '资金流水表';
```

**关键设计**：

| 设计点 | 说明 |
|--------|------|
| `uk_biz_order` | **幂等保障的最终兜底** — DB 层面保证同一业务+同一订单+同一钱包不会有两条流水 |
| `balance_before/after` | **对账核心** — 任意时点可通过流水链验证余额正确性 |
| `frozen_before/after` | 冻结/解冻操作的快照，提现审计需要 |
| 只增不改不删 | 流水是审计凭证，逻辑删除都不做，即使"修正"也是新增反向流水 |

### 3.2.4 金额精度架构

```
┌──────────────────────────────────────────────────────────────┐
│                      金额精度分层策略                          │
│                                                               │
│  RPC 接口层 (IDL)                                             │
│    · 金额字段类型: string                                      │
│    · 原因: Thrift 无 decimal 类型，string 无精度损失            │
│    · 示例: "1000000.00", "0.00000001"                         │
│                                                               │
│  Service 层 (Go)                                              │
│    · 金额运算: shopspring/decimal                              │
│    · 原因: Go float64 有精度问题(0.1+0.2≠0.3)                 │
│    · 所有金额接收后立即 decimal.NewFromString() 转换            │
│    · 所有运算用 decimal.Add/Sub/Mul/Div                        │
│    · 禁止 float64 参与任何金额计算                              │
│                                                               │
│  数据库层 (TiDB)                                               │
│    · 字段类型: DECIMAL(20,8)                                   │
│    · 原因: 统一精度，USDT 需 8 位，法币向上兼容                  │
│    · GORM 映射: Go 侧对应 decimal.Decimal 或 string            │
│                                                               │
│  展示层 (响应给前端)                                            │
│    · 按币种配置格式化: VND/IDR→整数, THB/BSB→2位, USDT→8位     │
│    · 千分位: VND/IDR用".", THB/BSB/USDT用","                   │
│    · 格式化在 Service 层完成，返回给网关的已是格式化后的 string    │
│                                                               │
│  注意: shopspring/decimal 需新增依赖（现有工程未使用）            │
│  现有工程金额用 int32 (ser-item 道具价格)，资金系统精度要求更高    │
└──────────────────────────────────────────────────────────────┘
```

## 3.3 数据库连接池配置

现有工程默认：MaxIdle=10, MaxOpen=20, MaxLifeTime=1h。

资金服务因为事务+FOR UPDATE 会持有连接更久，建议适当提高：

```
ser-wallet 建议连接池:
  MaxIdleConns:    20     (空闲连接保持更多，减少新建开销)
  MaxOpenConns:    50     (资金操作事务持连接时间较长)
  MaxLifeTime:     1h     (保持不变)
```

不需要读写分离（`dbresolver`）—— 资金操作全部走主库，读写分离反而引入延迟一致性风险。

# 第四部分：资金安全架构

> 这是整个架构中最核心的部分。资金系统的架构质量 = 安全机制的严密程度。

## 4.1 账户操作的标准事务模型

每一次余额变动（无论来自哪个业务场景），都遵循同一个事务模型：

```
account_ser.Credit(ctx, req) / Debit(ctx, req) / Freeze(ctx, req) / ...
  │
  ▼
┌─────────────────────────────────────────────────────────────────┐
│  Step 1: 幂等校验 (查流水表)                                     │
│    SELECT id FROM t_wallet_transaction                           │
│    WHERE biz_type = ? AND order_no = ? AND wallet_type = ?       │
│    LIMIT 1                                                       │
│                                                                  │
│    → 存在: 直接返回成功 (幂等，不重复执行)                         │
│    → 不存在: 进入事务                                             │
│                                                                  │
│  Step 2: 开启 DB 事务                                            │
│    query.Q.Transaction(func(tx *query.Query) error {             │
│                                                                  │
│      Step 3: 行级锁定目标钱包                                    │
│        SELECT balance, frozen, version FROM t_wallet             │
│        WHERE user_id=? AND currency_code=? AND wallet_type=?     │
│        FOR UPDATE                                                │
│                                                                  │
│        → 不存在: 自动创建钱包记录 (懒初始化)                      │
│                                                                  │
│      Step 4: 业务校验                                            │
│        · Credit: amount > 0 ✓                                    │
│        · Debit: balance >= amount ? ✓ : 返回余额不足              │
│        · Freeze: balance - frozen >= amount ? ✓ : 余额不足       │
│        · Unfreeze: frozen >= amount ? ✓ : 冻结不足               │
│                                                                  │
│      Step 5: 更新余额                                            │
│        UPDATE t_wallet SET balance=?, frozen=?, version=version+1│
│        WHERE id=? AND version=?                                  │
│                                                                  │
│      Step 6: 写入流水                                            │
│        INSERT INTO t_wallet_transaction (                        │
│          user_id, currency_code, wallet_type, direction, amount, │
│          balance_before, balance_after, frozen_before, frozen_after,│
│          biz_type, order_no, create_at                           │
│        )                                                         │
│                                                                  │
│    }) ← 事务提交 or 回滚                                         │
│                                                                  │
│  Step 7: 返回结果 (成功 + 变更后余额)                             │
└─────────────────────────────────────────────────────────────────┘
```

**为什么是这个模型**：

| 机制 | 业界依据 | 在本架构中的实现 |
|------|---------|----------------|
| 幂等校验在事务外 | 快速拒绝重复请求，不浪费 DB 事务资源 | SELECT 流水表，O(1) 唯一索引查找 |
| FOR UPDATE 悲观锁 | 支付宝/微信支付标准做法 | 锁定目标钱包行，防并发修改 |
| version 乐观锁 | 双重保障 | UPDATE WHERE version=?，防止极端情况下的脏写 |
| 流水与余额同事务 | 会计准则 | 事务保证原子性，不可能出现余额变了但流水没记 |
| 唯一索引兜底 | 工程安全网 | 即使应用层幂等校验并发漏判，DB 层拒绝重复插入 |

## 4.2 FOR UPDATE 在现有工程中的引入

**现状**：通过 agent 深度扫描确认，当前工程中**没有**任何 FOR UPDATE 的使用。这是因为既有业务（Banner、道具、用户管理）不涉及并发余额操作。

**引入方式**：在手写的 `rep/account_rep.go` 中实现，使用 GORM 原生 API：

```go
// rep/account_rep.go
func (r *AccountRepo) LockWallet(tx *gorm.DB, ctx context.Context,
    userId int64, currencyCode string, walletType int32,
) (*model.Wallet, error) {
    var wallet model.Wallet
    err := tx.WithContext(ctx).
        Where("user_id = ? AND currency_code = ? AND wallet_type = ? AND delete_at = 0",
            userId, currencyCode, walletType).
        Clauses(clause.Locking{Strength: "UPDATE"}).
        First(&wallet).Error
    return &wallet, err
}
```

**不用 GORM Gen 的 query 模式**：因为 GORM Gen 生成的查询构建器不直接支持 FOR UPDATE 子句，手写 Repository 更灵活。这与项目中手写 rep/ 处理复杂查询的既有模式一致。

## 4.3 幂等性架构

```
所有资金操作接口统一幂等设计:

调用方 (财务/游戏/直播/活动)
  │
  │ 携带 order_no (调用方生成，全局唯一)
  ▼
ser-wallet RPC 接口
  │
  │  ① 查流水表: (biz_type, order_no, wallet_type) 是否存在?
  │     ├─ 存在 → 返回成功 + 已有结果 (幂等)
  │     └─ 不存在 → 执行事务 → 插入流水 (事务内)
  │
  │  ② 如果并发穿透了 ① (极端情况)
  │     └─ 流水表唯一索引 uk_biz_order 拒绝重复插入
  │        → 事务回滚 → 返回成功 (二次幂等)
  ▼
绝不可能出现同一笔操作执行两次

order_no 生成规则 (建议):
  · 由调用方生成，格式: {前缀}{时间戳}{随机数}
  · 例: D20260226143052xxxx (充值), W20260226xxxx (提现)
  · 钱包不生成、不关心格式，只做唯一性校验
```

## 4.4 并发安全模型

```
场景: 用户同时发起提现 + 投注 (两个 RPC 请求并发到达)

时间线:
  T1  提现请求 → ser-wallet → Freeze()
  T1  投注请求 → ser-wallet → BetDebit()

并发保护:

请求A (Freeze)                    请求B (BetDebit)
  │                                  │
  ▼                                  ▼
BEGIN TX                          BEGIN TX
  │                                  │
SELECT ... FOR UPDATE             SELECT ... FOR UPDATE
WHERE user_id=1                   WHERE user_id=1
AND currency='BSB'                AND currency='BSB'
AND wallet_type=1                 AND wallet_type=1
  │                                  │
  ▼                                  ▼
获得行锁 ✓                        等待行锁 ⏳ (被A阻塞)
  │                                  │
balance=100, frozen=0                │ (等待中...)
frozen += 50                         │
UPDATE ... SET frozen=50             │
INSERT 流水                          │
COMMIT ✓                             │
  │                                  ▼
释放行锁 ──────────────────→     获得行锁 ✓
                                    │
                                 balance=100, frozen=50
                                 可用 = 100-50 = 50
                                 BetDebit 50 ✓ (余额够)
                                 balance -= 50
                                 UPDATE ... SET balance=50
                                 INSERT 流水
                                 COMMIT ✓

结果: balance=50, frozen=50, 正确且一致
```

**TiDB 行锁特性**：TiDB 支持标准的 `SELECT ... FOR UPDATE` 行级锁，在悲观事务模式下行为与 MySQL InnoDB 一致。TiDB 默认悲观事务模式（v3.0.8+），无需额外配置。

## 4.5 安全机制总览

```
┌─────────────────────────────────────────────────────────────────┐
│                     资金安全五层防护                              │
│                                                                  │
│  L1  接口层: 参数校验                                            │
│       · amount > 0, currency 有效, walletType 有效               │
│       · userId 从中间件 Token 获取，不信任前端传参                 │
│                                                                  │
│  L2  业务层: 幂等校验                                            │

│       · 重复请求直接返回成功                                      │
│                                                                  │
│  L3  事务层: FOR UPDATE 行级锁                                   │

│       · 事务内校验余额充足性                                      │
│                                                                  │
│  L4  更新层: 乐观锁 version                                      │

│                                                                  │
│  L5  数据层: 唯一索引                                            │
│       · uk_biz_order 流水表唯一索引                               │
│       · DB 层面拒绝重复流水，最终兜底                              │
│                                                                  │
│  任何一笔资金操作必须穿过全部5层才能成功                           │
└─────────────────────────────────────────────────────────────────┘
```

**为什么资金操作用同步 RPC，不用消息队列**：

| 对比 | 同步 RPC | 异步消息 |
|------|---------|---------|
| 结果确认 | 调用方立即知道成功/失败 | 需要额外确认机制 |
| 一致性 | 强一致 | 最终一致（有延迟窗口） |
| 复杂度 | 简单直接 | 需要消息幂等、消费确认、死信处理 |
| 适用场景 | 资金操作（要求实时确认） | 通知类/统计类（允许延迟） |

**结论**：所有资金变动走同步 RPC + 幂等，事件通知走 NATS JetStream。这是业界主流做法（同步保安全，异步做扩展）。

## 6.2 为什么不用 Redis 分布式锁库

现有工程中没有使用 redsync 等分布式锁库（agent 扫描确认），只使用 `rds.SetNX()`。

本架构不引入新的锁库，原因：

| 场景 | SetNX 足够吗 | 理由 |
|------|-------------|------|
| 用户操作防重 | 足够 | 只需防双击，10 秒 TTL 自动过期，最坏情况只是多等 10 秒 |
| 余额并发操作 | 不用 Redis 锁 | DB 的 FOR UPDATE 行级锁才是权威保障 |
| 定时任务防重 | 足够 | SetNX 抢占即可 |

**Redis 锁只做"快速挡板"，DB 锁才做"资金安全"。这是两个不同层面的问题，不要混淆。**

# 第九部分：投注扣款/返奖架构

## 9.1 优先级扣款模型

```
┌──────────────────────────────────────────────────────────────────┐
│  BetDebit — 投注优先级扣款                                        │
│                                                                   │
│  需求: "先扣中心钱包，不足部分再扣奖励钱包"                        │
│  特殊: 需要在一个事务中操作两个钱包，并记录扣款比例                 │
│                                                                   │
│  事务模型:                                                        │
│                                                                   │
│  BEGIN TX                                                         │
│    ① SELECT ... FROM t_wallet                                     │
│       WHERE user_id=? AND currency=? AND wallet_type=1(中心)      │
│       FOR UPDATE                                                  │
│                                                                   │
│    ② SELECT ... FROM t_wallet                                     │
│       WHERE user_id=? AND currency=? AND wallet_type=2(奖励)      │
│       FOR UPDATE                                                  │
│                                                                   │
│    ③ 计算分配:                                                    │
│       centerDebit = min(centerBalance, totalAmount)               │
│       bonusDebit  = min(bonusBalance, totalAmount - centerDebit)  │
│       if centerDebit + bonusDebit < totalAmount → 回滚,余额不足   │
│                                                                   │
│    ④ 更新两个钱包余额                                             │
│    ⑤ 写入两条流水记录                                             │
│  COMMIT                                                           │
│                                                                   │
│  返回: {centerRatio, bonusRatio}                                  │
│  例: 投注100, center扣80, bonus扣20 → {0.8, 0.2}                 │
│                                                                   │
│  调用方(游戏)必须持久化此比例，用于 BetCredit 按比例返奖            │
│                                                                   │
│  BetCredit — 按比例返奖:                                          │
│    centerCredit = winAmount × centerRatio                         │
│    bonusCredit  = winAmount × bonusRatio                          │
│    分别入账到中心/奖励钱包 (同一事务)                               │
└──────────────────────────────────────────────────────────────────┘
```

**为什么两个钱包在同一个事务中锁定**：
- TiDB（和 MySQL）的行级锁是事务级别的，同一事务可以锁多行
- 锁定顺序固定为 wallet_type 升序（1 中心 → 2 奖励），防止死锁
- 两行在同一张表（t_wallet），不涉及跨库事务

# 第十四部分：架构决策记录（ADR）

## ADR-01: 不使用分布式事务框架

**背景**：钱包↔财务存在双向 RPC 调用，是否需要引入 TCC/Saga/Seata 等分布式事务框架？

**决策**：**不引入**

**理由**：
- 钱包的资金操作（入账/扣款/冻结）是**单服务内的本地事务**，不涉及跨库
- 钱包↔财务的交互模式是"同步 RPC + 幂等 + 补单兜底"，已能保证最终一致性
- 分布式事务框架引入大量复杂度（协调者/参与者/补偿逻辑），属于过度设计
- 业界同类系统（中小规模支付系统）普遍采用"本地事务 + 幂等 + 补单"模式

## ADR-02: 余额不缓存

**背景**：是否将用户余额缓存到 Redis 提升读性能？

**决策**：**不缓存**

**理由**：
- 缓存与 DB 的不一致会导致超扣或展示错误，资金场景不可接受
- TiDB 主键/唯一索引查询 <1ms，用户级 QPS 下性能足够
- 缓存更新策略（Cache-Aside / Write-Through / Write-Behind）在并发扣款场景下都有一致性缺陷
- 业界支付系统核心账户都是实时读 DB

## ADR-03: 使用 FOR UPDATE 悲观锁而非纯乐观锁

**背景**：并发余额操作用悲观锁（FOR UPDATE）还是乐观锁（CAS version）？

**决策**：**FOR UPDATE 为主，version 为辅**

**理由**：
- 乐观锁在高冲突场景下重试率高（投注高峰期同一用户多次投注），影响响应时间
- 悲观锁保证获得锁后必定成功，无需重试逻辑
- FOR UPDATE 的锁等待时间在毫秒级别（单行锁），对响应时间影响极小
- version 字段作为安全网保留，防止极端情况下的脏写
- 业界支付系统标准做法

## ADR-04: 币种配置与钱包同进程

**背景**：币种配置是否独立部署为单独的微服务？

**决策**：**不独立，与 ser-wallet 同进程**

**理由**：
- 币种配置是钱包的"内部基础设施"，每次资金操作都需要查汇率/币种信息
- 如果独立部署，每次操作多一次 RPC 调用（增加 3~10ms 延迟 + 网络不稳定风险）
- 同进程函数调用 ~0.001ms，性能差 1000 倍以上
- 币种配置的接口量小（18 个），不足以独立成服务
- 对外依然通过 Kitex RPC 暴露（给财务/游戏等服务使用），只是"对内零开销"

## ADR-05: NATS 用于事件通知，不用于资金操作

**背景**：是否通过 NATS 消息队列传递资金操作指令？

**决策**：**资金操作走同步 RPC，事件通知走 NATS**

**理由**：
- 资金操作需要实时确认（调用方必须知道入账/扣款是否成功）
- 消息队列的确认机制（ACK/NAK/重投递）在资金场景下增加复杂度
- NATS 适合用于"结果广播"（余额变动通知、充值成功通知），而非"操作指令"
- 现有 NATS JetStream 封装完善（Publish/Subscribe/PullBatch/延迟投递/死信队列），用于通知绰绰有余

## ADR-06: 金额使用 decimal 而非 int64 分/厘

**背景**：金额内部表示用 decimal 还是 int64（分/厘最小单位）？

**决策**：**decimal**

**理由**：
- 多币种系统中，不同币种的最小单位不同（VND 无小数、USDT 8 位小数）
- 如果用 int64 表示最小单位，VND 的 1 = 1，USDT 的 1 = 0.00000001，换算极易出错
- decimal 语义清晰：1.00 就是 1，0.00000001 就是 0.00000001
- shopspring/decimal 是 Go 生态中最成熟的 decimal 库，百万级项目使用

# 第十五部分：架构总结

## 15.1 一句话架构

> **单服务（ser-wallet）内的分层架构，DB 本地事务保证资金安全，同步 RPC + 幂等实现跨服务协作，Redis 做配置缓存和防重，NATS 做事件通知。不引入分布式事务、不缓存余额、不过度拆分服务。**

## 15.2 架构关键数字

| 指标 | 数值 | 说明 |
|------|------|------|
| 服务数 | 1 (ser-wallet) | 币种配置和钱包同进程 |
| 数据库 | 1 (wallet_db) | TiDB 独立数据库 |
| 核心表 | 10 张 | 见 3.2 |
| RPC 接口 | ~69 个 | C端21 + B端11 + RPC暴露24 + RPC调用13 |
| DB 连接池 | 50 | MaxOpen，事务场景需要更多 |
| RPC 超时 | 3 秒 | 项目统一配置 |
| 缓存 TTL | 3~5 分钟 | 币种/汇率缓存 |
| 防重 TTL | 10 秒 | SetNX 用户操作防重 |
| 安全层数 | 5 层 | 参数校验 → 幂等 → 行锁 → 乐观锁 → 唯一索引 |

## 15.3 架构检查清单

| 检查项 | 满足 | 依据 |
|--------|------|------|
| 安全 | 五层防护（幂等+事务+行锁+乐观锁+唯一索引） | 第四部分 |
| 高并发 | FOR UPDATE 行级锁，只锁目标行不锁表 | 4.2, 4.4 |
| 高性能 | 余额读 DB <1ms，币种/汇率 Redis 缓存，进程内调用零网络 | 3.1, 6.1, ADR-04 |
| 不过度设计 | 不引入分布式事务/独立配置中心/API Gateway/额外锁库 | ADR-01~06 |
| 业界主流 | 本地事务+幂等+补单，悲观锁，decimal金额，流水表对账 | 支付宝/微信支付同类架构 |
| 现有工程兼容 | 复用 Kitex/Hertz/GORM Gen/ETCD/Redis/NATS 全套 | 第一部分 |
