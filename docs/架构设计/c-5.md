# 架构设计：钱包 + 币种配置模块

> 视角：在现有工程体系下，按业界成熟标准设计钱包系统架构
> 原则：安全、高并发、高性能、不过度设计、业界主流
> 依据：需求文档 + 产品原型 + 现有工程代码级分析（ser-item/ser-buser/ser-app/common全栈）
> 定位：粗粒度架构决策，为后续编码提供方向

---

# 一、架构定位

## 1.1 这是什么系统

这是一个**多币种电子钱包系统**，核心能力是管理用户资产（余额）、处理资金进出（充值/提现）和内部流转（兑换）。

从业界视角看，它属于**支付系统的账户子系统**——不直接对接支付渠道（财务模块负责），但管理用户的资金账本。类似支付宝的"余额"模块，或游戏平台的虚拟钱包系统。

## 1.2 架构设计的核心矛盾

钱包系统的核心矛盾是：**余额操作的强一致性要求 vs 分布式微服务下的最终一致性现实**。

- 余额必须精确——多一分少一分都是事故
- 但充值和提现跨越钱包和财务两个独立服务，天然存在分布式事务问题

架构设计的本质就是在这个矛盾中找到**工程上可落地、业务上可接受**的平衡点。

## 1.3 不过度设计的边界

以下是**不做**的事情（当前阶段不需要，规模到了再考虑）：

```
不做：分库分表（用户量不到千万级不需要）
不做：Event Sourcing（事件溯源模式，复杂度太高，CRUD够用）
不做：CQRS 读写分离（单表读写不是瓶颈）
不做：分布式事务框架（Seata/DTM，杀鸡不用牛刀，补偿机制够用）
不做：独立的账务中台（业务规模不到，一个 ser-wallet 服务够用）
不做：实时对账系统（初期人工 + 定时任务对账即可）
```

## 2.2 为什么是单服务而不是拆分

**决策：钱包和币种配置放在同一个 ser-wallet 服务中。**

依据：

1. **现有工程惯例**：ser-buser（31方法）、ser-app（21方法）都是单服务多职责。ser-wallet 预估45个方法，在同一量级

2. **数据内聚**：币种配置和钱包余额在同一个数据库中，余额操作需要读取币种参数（精度/限额），放在同一个服务内可以走**本地方法调用**而非 RPC，减少网络开销和故障点

3. **事务边界**：兑换操作涉及"扣减法币 + 增加BSB"两个币种的余额变动，必须在同一个数据库事务中完成。如果币种和钱包拆成两个服务两个库，这个事务就变成分布式事务，复杂度暴增

4. **部署简单**：一个进程、一个端口、一个 ETCD 注册，运维成本低

**什么时候才需要拆**：当 ser-wallet 的 QPS 远超单机处理能力，或者团队规模扩大到需要独立迭代时。当前阶段不需要。

## 2.3 与外部服务的调用关系

```
ser-wallet 主动调用（出方向）:
  → 财务服务    支付配置查询 / 创建渠道订单 / 提交提现审核
  → 用户服务    KYC状态查询 / KYC信息查询
  → i18n服务    错误码翻译（ret框架自动处理）
  → blog服务    审计日志（操作日志记录）

ser-wallet 被调用（入方向）:
  ← 财务服务    充值入账 / 提现确认 / 提现解冻 / 人工加减款补单 / 余额查询
  ← 其他服务    余额查询（如果有其他业务需要读余额）
```

# 三、分层架构

## 3.1 四层架构

遵循现有工程的分层惯例（从 ser-item、ser-buser、ser-app 代码验证），ser-wallet 采用四层架构：

```
┌────────────────────────────────────────────────────────┐
│  RPC Handler 层（handler.go）                           │
│  职责：接收 Kitex RPC 请求，一行委托到 Service 层        │
│  规则：不写任何业务逻辑                                  │
├────────────────────────────────────────────────────────┤
│  Service 层（internal/ser/*.go）                        │
│  职责：业务编排 — 校验、调外部RPC、组装参数、调Repo       │
│  规则：事务在这层发起，跨模块调用在这层协调               │
├────────────────────────────────────────────────────────┤
│  Repository 层（internal/rep/*.go）                     │
│  职责：数据访问 — 复杂查询、事务内多表操作                │
│  规则：基于 GORM Gen 生成的 query 扩展自定义查询          │
├────────────────────────────────────────────────────────┤
│  Model 层（internal/gorm_gen/model/*.go）               │
│  职责：数据结构 — 表模型定义                             │
│  规则：由 gorm_gen.go 自动生成，不手写                   │
└────────────────────────────────────────────────────────┘
```

## 3.2 各层职责详解

### Handler 层

```go
// handler.go — 纯粹的委托，零逻辑
// 参照 ser-item/handler.go、ser-buser/handler.go 的写法

type WalletServiceImpl struct {
    currencyService *ser.CurrencyService
    balanceService  *ser.BalanceService
    depositService  *ser.DepositService
    withdrawService *ser.WithdrawService
    exchangeService *ser.ExchangeService
    recordService   *ser.RecordService
    manualService   *ser.ManualService
}

// 每个方法都是一行委托
func (s *WalletServiceImpl) CreateDeposit(ctx context.Context, req *ser_wallet.CreateDepositReq) (*ser_wallet.CreateDepositResp, error) {
    return s.depositService.CreateDeposit(ctx, req)
}
```

依据：`ser-item/handler.go` 全部方法都是一行委托，`ser-buser/handler.go`（169行31方法）同样模式。

### Service 层

Service 层是**业务编排的核心**，负责：
- 参数校验（第一道关）
- 调用外部 RPC（币种配置、财务、用户服务）
- 组织数据、计算业务逻辑
- 发起事务（调用 Rep 层的事务方法）
- 处理异常和补偿

```
internal/ser/
├── currency_service.go      币种CRUD + 汇率管理 + RPC暴露查询
├── balance_service.go       余额查询 + 余额变动原子操作（核心）
├── deposit_service.go       充值全流程（创建订单→调财务→入账回调）
├── withdraw_service.go      提现全流程（校验→冻结→提交审核→确认/解冻回调）
├── exchange_service.go      兑换全流程（汇率→计算→事务内余额变动）
├── record_service.go        记录查询（充值/提现/兑换/奖励列表+详情）
└── manual_service.go        人工修正（加款/减款/补单 — RPC被调用）
```

依据：
- ser-buser 按业务域拆分为 user_service.go（516行）、menu_service.go（387行）、role_service.go（145行）、enum_service.go（49行），共4个文件
- ser-app 拆分为 banner.go、home_category_service.go、s3_service.go、enum_service.go、content_strategy.go，共5个文件
- 按业务域拆文件，每个文件聚焦一个子领域，是现有工程的统一范式

### Repository 层

Rep 层负责：
- 复杂的 SQL 查询（多条件、JOIN、聚合）
- 事务操作的具体执行
- 扩展 GORM Gen 自动生成方法不够用的场景

```
internal/rep/
├── currency_rep.go          币种/汇率表的自定义查询
├── balance_rep.go           余额表的原子操作（乐观锁更新、冻结/解冻）
├── deposit_rep.go           充值订单表的自定义查询
├── withdraw_rep.go          提现订单表的自定义查询
├── exchange_rep.go          兑换订单表的自定义查询
└── record_rep.go            流水/记录表的自定义查询
```

依据：
- ser-buser 有 user.go（465行）、menu.go（332行）、role.go（348行）共3个 rep 文件
- ser-item 有 item.go（298行）、item_publish.go（267行）共2个 rep 文件
- rep 层使用 `query.Q.XXX` 做类型安全查询，事务使用 `query.Q.Transaction(func(tx *query.Query) error {...})`

## 3.3 层间调用规则

```
Handler  → 只调 Service（一行委托）
Service  → 调 Service（同模块内可互调，如 DepositService 调 BalanceService）
Service  → 调 Rep（数据访问）
Service  → 调外部 RPC（rpc.XxxClient()）
Rep      → 调 GORM Gen query（数据库操作）
Rep      → 不调外部 RPC（数据层不应有远程调用）
```

**关键约束：Rep 层不调 RPC**。所有远程调用在 Service 层完成，数据准备好后传给 Rep 层执行本地 DB 操作。这保证事务内不包含 RPC 调用。

# 四、核心数据架构

## 4.1 数据库选择

使用现有的 **TiDB**（MySQL 兼容），一个独立数据库 `ser_wallet`。

不需要额外引入其他存储。TiDB 本身支持水平扩展，当前阶段不需要分库分表。

## 4.2 表设计架构

按数据性质分三类表：

```
┌────────────────────────────────────────┐
│ 第一类：配置表（低频写、高频读、可缓存）   │
│                                        │
│  currency_config   币种配置表           │
│  exchange_rate     汇率配置表           │
│  currency_log      配置操作日志表        │
│  rate_log          汇率变更日志表        │
├────────────────────────────────────────┤
│ 第二类：账本表（高频写、强一致、不可缓存）  │
│                                        │
│  user_balance      用户余额表（核心）     │
│  balance_flow      余额变动流水表        │
├────────────────────────────────────────┤
│ 第三类：订单表（中频写、按状态流转）       │
│                                        │
│  deposit_order     充值订单表            │
│  withdraw_order    提现订单表            │
│  exchange_order    兑换订单表            │
│  withdraw_account  提现账户信息表         │
└────────────────────────────────────────┘
```

## 4.3 余额表 — 整个系统的心脏

**设计决策：单表、三元组唯一键、可用+冻结双字段**

```sql
CREATE TABLE user_balance (
    id           BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id      BIGINT       NOT NULL,
    currency     VARCHAR(10)  NOT NULL,           -- BSB / VND / IDR / THB / USDT
    wallet_type  TINYINT      NOT NULL,           -- 1中心 2奖励 3代理 4主播
    available    BIGINT       NOT NULL DEFAULT 0, -- 可用余额（最小单位）
    frozen       BIGINT       NOT NULL DEFAULT 0, -- 冻结余额
    version      INT          NOT NULL DEFAULT 0, -- 乐观锁版本号（备用）
    create_by    BIGINT,
    create_at    BIGINT,
    update_by    BIGINT,
    update_at    BIGINT,

    UNIQUE KEY uk_user_currency_wallet (user_id, currency, wallet_type)
) ENGINE=InnoDB;
```

**架构决策及理由**：

**金额存储用 BIGINT（最小单位整数）而非 DECIMAL**

```
理由：
  - 整数运算无精度丢失风险（不存在 0.1+0.2≠0.3 的问题）
  - 数据库层面 BIGINT 比较和加减更快
  - 现有工程所有 ID 和时间戳都是 int64，风格一致
  - VND 最小单位就是1（1 VND = 1），USDT 最小单位是 0.01（1分=1）
  - 展示层做精度换算：available / 10^precision → 展示金额

存储规则：
  BSB   → precision=2 → 存储 1050 = 展示 10.50 BSB
  VND   → precision=0 → 存储 1000000 = 展示 1,000,000 VND
  USDT  → precision=2 → 存储 10050 = 展示 100.50 USDT
```

**可用余额和冻结余额分两个字段而非拆表**

```
理由：
  - 提现冻结操作需要原子性地 "available减+frozen加"，同一行更新最简单
  - 不需要冻结明细表（冻结流水靠 balance_flow 记录追溯）
  - 解冻时直接 "frozen减+available加" 回退

如果未来需要多种冻结类型（提现冻结、风控冻结、活动冻结...）：
  再拆一张 frozen_detail 表记录冻结明细，但 user_balance 上的 frozen 字段始终是总冻结额
  当前阶段只有提现冻结一种场景，不需要拆
```

## 4.4 余额变动流水表 — 审计和对账的底线

```sql
CREATE TABLE balance_flow (
    id             BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id        BIGINT       NOT NULL,
    currency       VARCHAR(10)  NOT NULL,
    wallet_type    TINYINT      NOT NULL,
    change_type    TINYINT      NOT NULL,           -- 变动类型枚举
    amount         BIGINT       NOT NULL,           -- 变动金额（正=增 负=减）
    before_balance BIGINT       NOT NULL,           -- 变动前可用余额
    after_balance  BIGINT       NOT NULL,           -- 变动后可用余额
    ref_order_no   VARCHAR(32)  NOT NULL DEFAULT '',-- 关联订单号
    remark         VARCHAR(256) DEFAULT '',
    create_at      BIGINT,

    INDEX idx_user_currency (user_id, currency),
    INDEX idx_ref_order (ref_order_no),
    UNIQUE KEY uk_order_type (ref_order_no, change_type) -- 幂等兜底
) ENGINE=InnoDB;
```

**架构意义**：

这张表是**不可篡改的资金流水账本**。每一笔余额变动都必须有一条对应的流水记录。

`before_balance` 和 `after_balance` 的存在让你可以随时验证：任意两条相邻流水的 `after_balance(上一条) = before_balance(下一条)`。如果不等，说明有异常。这就是**流水连续性校验**——钱包系统最基本的对账手段。

`uk_order_type` 唯一键是**幂等的数据库级兜底**。即使应用层的幂等判断在极端并发下穿透了，数据库层也会阻止重复写入。

## 4.5 索引策略

```
余额表：
  uk_user_currency_wallet    → 所有余额查询和更新的必经之路
  无需额外索引               → 余额表查询一定走唯一键

流水表：
  idx_user_currency          → C端按用户+币种查记录列表
  idx_ref_order              → 通过订单号反查流水
  uk_order_type              → 幂等兜底

订单表：
  uk_order_no                → 订单号唯一键
  idx_user_currency_status   → 用户+币种+状态的组合查询（如查处理中订单数）
  idx_create_at              → 按时间范围查询
```

不过度建索引。遵循原则：**查询驱动索引，不是先建再说**。

# 五、余额操作架构 — 最核心的设计

## 5.1 设计目标

余额操作必须同时满足三个属性：

```
正确性：任何并发场景下余额都不会出错（不超扣、不重复入账）
一致性：余额变动和流水记录严格一致（不出现余额对不上流水的情况）
可追溯：每一笔变动都有完整的流水记录，支持审计和对账
```

## 5.2 统一余额变动模型

所有涉及余额变动的业务场景（充值入账、兑换、提现冻结/扣除/解冻、人工加减款），最终都收敛到一个**统一的余额变动操作**。

```
┌──────────────────────────────────────────────────┐
│           统一余额变动操作（事务内）                 │
│                                                  │
│  输入：                                          │
│    UserID    用户ID                              │
│    Currency  币种                                │
│    WalletType 子钱包类型                          │
│    Amount    变动金额（正=加 负=减）               │
│    ChangeType 变动类型枚举                        │
│    RefOrderNo 关联订单号                          │
│                                                  │
│  操作流程：                                       │
│    1. 查询当前余额（SELECT FOR UPDATE 或乐观锁）    │
│    2. 校验余额充足（扣减场景）                      │
│    3. 更新余额（带乐观锁条件）                      │
│    4. 写入 balance_flow 流水                      │
│    5. 返回操作前后余额                             │
│                                                  │
│  保证：                                          │
│    步骤 3 和 4 在同一个数据库事务内                 │
│    步骤 3 使用乐观锁防并发超扣                     │
│    步骤 4 的唯一键防幂等穿透                       │
└──────────────────────────────────────────────────┘
```

上层业务方法只需组装参数调用这个统一操作：

```
充值入账    → BalanceChange{Amount=+100, Type=充值入账, Order=充值订单号}
兑换扣减    → BalanceChange{Amount=-230000, Type=兑换扣减, Order=兑换订单号}
兑换增加    → BalanceChange{Amount=+10, Type=兑换增加, Order=兑换订单号}
提现冻结    → BalanceChange{Available=-500, Frozen=+500, Type=提现冻结, Order=提现订单号}
人工加款    → BalanceChange{Amount=+100, Type=人工加款, Order=修正单号}
```

## 5.3 并发安全方案

**主方案：乐观锁（CAS 更新）**

```sql
-- 增加余额（充值入账、人工加款）
UPDATE user_balance
SET available = available + ?,
    update_at = ?
WHERE user_id = ? AND currency = ? AND wallet_type = ?

-- 扣减余额（兑换扣减、人工减款）
UPDATE user_balance
SET available = available - ?,
    update_at = ?
WHERE user_id = ? AND currency = ? AND wallet_type = ?
  AND available >= ?    -- 乐观锁条件：余额充足才执行

-- 提现冻结
UPDATE user_balance
SET available = available - ?,
    frozen = frozen + ?,
    update_at = ?
WHERE user_id = ? AND currency = ? AND wallet_type = ?
  AND available >= ?    -- 乐观锁条件

-- 提现确认扣除
UPDATE user_balance
SET frozen = frozen - ?,
    update_at = ?
WHERE user_id = ? AND currency = ? AND wallet_type = ?
  AND frozen >= ?       -- 冻结金额充足

-- 提现解冻退回
UPDATE user_balance
SET available = available + ?,
    frozen = frozen - ?,
    update_at = ?
WHERE user_id = ? AND currency = ? AND wallet_type = ?
  AND frozen >= ?
```

通过 `AND available >= ?` 条件，数据库层保证不会出现负余额。检查受影响行数：0行 = 余额不足。

**为什么选乐观锁而非悲观锁（SELECT FOR UPDATE）**：

```
乐观锁优势：
  - 不持有行锁，并发度高
  - 单条 UPDATE 语句，原子性由数据库保证
  - 无死锁风险
  - InnoDB 的 UPDATE WHERE 在执行时会加行级排他锁，
    但锁的持有时间极短（只在 UPDATE 语句执行期间）

悲观锁劣势：
  - SELECT FOR UPDATE 需要先查再改，两条语句
  - 事务持有行锁时间更长（从 SELECT 到 COMMIT 之间）
  - 并发场景下锁等待增加

什么时候才需要悲观锁：
  - 需要先读余额做复杂计算再更新时（但钱包的增减操作用 available ± amount 就够了）
  - 当前场景乐观锁完全够用
```

## 5.4 事务边界设计

```
原则：RPC 调用在事务之外，数据库操作在事务之内

以兑换为例（最清晰的模式）：

  ┌─ 事务之外 ──────────────────────────┐
  │ 1. [RPC] 获取最新汇率               │
  │ 2. 计算兑换金额和赠送金额            │
  │ 3. 校验源币种余额是否充足（快速预检） │
  └──────────────────────────────────────┘
        │
        ▼
  ┌─ 事务之内 ──────────────────────────┐
  │ 4. 扣减源币种中心钱包 available      │
  │ 5. 增加BSB中心钱包 available         │
  │ 6. 增加BSB奖励钱包 available（赠送）  │
  │ 7. 写兑换订单记录                    │
  │ 8. 写余额变动流水（3条）             │
  └──────────────────────────────────────┘

以充值入账（RPC被调用）为例：

  ┌─ 事务之外 ──────────────────────────┐
  │ 1. 幂等校验（查订单状态）             │
  │    已入账 → 直接返回成功              │
  └──────────────────────────────────────┘
        │
        ▼
  ┌─ 事务之内 ──────────────────────────┐
  │ 2. 更新充值订单状态 → 成功            │
  │ 3. 增加中心钱包 available             │
  │ 4. 如有奖金：增加奖励钱包 available   │
  │ 5. 写余额变动流水                    │
  └──────────────────────────────────────┘
```

**事务必须尽可能短**。原则上事务内只做 SQL 操作，不做任何 IO（RPC、Redis、日志打印等）。

依据：`ser-item/internal/rep/item.go` 中的事务使用 `query.Q.Transaction(func(tx *query.Query) error {...})`，事务内只有 GORM 操作，没有 RPC 调用。这是现有工程的正确范式。

# 六、幂等架构

## 6.1 幂等的范围

**所有 RPC 暴露的写接口必须幂等**。这是金融系统的底线要求。

```
接口                幂等键               幂等判断方式
────────────────────────────────────────────────────
充值入账            平台订单号            查订单状态是否已成功
提现确认扣款        提现订单号            查订单状态是否已成功
提现失败解冻        提现订单号            查订单状态是否已失败
人工加款            修正单号              查流水表是否已有该单号记录
人工减款            修正单号              同上
人工补单            补单号                同上
```

## 6.2 两层防护

```
第一层：应用层快速判断
  进入方法后，先用幂等键查询当前状态
  已完成 → 直接返回成功，不执行任何操作
  好处：绝大多数重复请求在这里就返回了，不会走到DB写操作

第二层：数据库唯一键兜底
  balance_flow 表的 uk_order_type (ref_order_no, change_type) 唯一键
  即使两个并发请求同时穿透了第一层，写流水时唯一键冲突
  catch 唯一键冲突异常 → 返回成功
  好处：无论如何不会写入重复流水

两层结合：应用层拦截99.9%的重复请求，数据库层兜底0.1%的极端并发
```

# 七、跨模块交互架构

## 7.1 与财务模块的交互模式

钱包和财务的交互有两种模式：

**模式A：钱包主动调财务（同步 RPC 请求-响应）**

```
适用场景：获取充值方式、创建渠道订单、提交提现审核

特点：
  - 钱包发起请求，等待财务响应
  - 如果财务不可用，钱包相应功能降级（无法充值/提现）
  - 超时时间：3秒（现有 RPC 客户端默认配置）

异常处理：
  - 财务超时 → 返回用户"服务暂忙，请稍后重试"
  - 财务返回业务错误 → 透传错误码给前端
```

**模式B：财务回调钱包（同步 RPC，由财务发起）**

```
适用场景：充值入账、提现确认/解冻、人工加减款

特点：
  - 财务发起请求，钱包执行余额操作
  - 钱包接口必须幂等（财务可能重试）
  - 钱包必须快速响应（控制在100ms以内）

异常处理：
  - 钱包返回失败 → 财务记录异常，告警人工介入
  - 钱包超时 → 财务会重试，钱包靠幂等保证安全
```

## 7.2 补偿机制

跨模块调用存在中间状态不一致的风险。处理策略：

```
场景1：创建提现 → 钱包冻结成功 → 调财务提交审核失败
  补偿：钱包本地立即解冻
  实现：在 Service 层 try-catch，RPC 失败后执行解冻

场景2：创建充值 → 钱包建订单成功 → 调财务创建渠道订单失败
  补偿：钱包订单标记失败
  无资金风险：充值是先支付后入账，此时没有动过余额

场景3：财务回调充值入账 → 钱包DB异常导致入账失败
  补偿：财务侧记录异常，运营通过"人工补单"功能补录
  这是需求文档明确的补单场景（订单号 B+16位）
```

**不引入分布式事务框架**。原因：
- 场景1和2可以在本地代码里用 try-catch 解决（本地补偿）
- 场景3有明确的人工补偿机制（补单功能）
- 引入 Seata/DTM 的复杂度远大于这三种简单补偿的实现成本

# 八、缓存架构

## 8.1 缓存分层

```
┌───────────────────────────────────────────────┐
│            不缓存（强一致数据）                  │
│                                               │
│  user_balance    用户余额                      │
│  *_order         各类订单状态                   │
│  balance_flow    流水记录                       │
│                                               │
│  理由：资金数据必须实时准确，任何缓存都            │
│       可能导致用户看到错误余额或错误状态           │
├───────────────────────────────────────────────┤
│            Redis 缓存（读多写少数据）             │
│                                               │
│  币种列表            TTL 5分钟                  │
│  币种详情/配置参数    TTL 5分钟                  │
│  汇率                TTL 1-3分钟                │
│                                               │
│  更新策略：B端修改操作成功后主动删除缓存           │
│           下次查询自动重建（Cache-Aside 模式）     │
│                                               │
│  理由：币种配置数据变更频率极低（天级），           │
│       但被查询频率极高（C端每次操作都查）          │
└───────────────────────────────────────────────┘
```

## 8.2 缓存 Key 设计

```
遵循现有工程的 redis_key 命名规范（common/pkg/consts/redis_key/）:

wallet:currency:list             → 可用币种列表 JSON
wallet:currency:detail:{code}    → 单个币种配置 JSON
wallet:rate:{code}               → 单个币种汇率值
```

## 8.3 缓存实现模式

参照 `ser-item/internal/domain/item_cache.go` 的缓存模式：

```
读操作：
  1. 先查 Redis
  2. 命中 → 反序列化返回
  3. 未命中 → 查 DB → 序列化写入 Redis（带 TTL）→ 返回

写操作（B端修改币种/汇率）：
  1. 更新 DB
  2. 删除对应 Redis Key（异步，defer + goroutine）
  3. 下次读时自动重建缓存
```

依据：`ser-item` 中 `ClearAllItemCache` 使用 `go s.cacheManager.ClearAllItemCache(ctx)` 异步清除缓存，写操作不等缓存清除完成。

# 九、安全架构

## 9.1 认证与鉴权

**完全复用现有网关中间件链，不需要额外设计**：

```
C端请求（gate-font）：
  SetHeaderInfo → BlackCheck → FontAuthMiddleware → Handler
  - Token 认证通过后，user_id 写入 context（metainfo）
  - 所有 ser-wallet 的 C端方法通过 tracer.GetUserId(ctx) 获取当前用户
  - 用户只能操作自己的余额（代码中校验 user_id 一致性）

B端请求（gate-back）：
  SetHeaderInfo → WhiteCheck → BackAuthMiddleware → Handler
  - IP白名单 + Token认证 + URL权限（RBAC）
  - 运营只能访问被授权的管理页面
```

依据：`common/pkg/middleware/gate-auth.go` 中的 FontAuthMiddleware 和 BackAuthMiddleware 已完整实现。

## 9.2 资金安全

```
防超扣：乐观锁条件 AND available >= amount
防重入：幂等两层防护（应用层 + 数据库唯一键）
防篡改：C端用户只能操作自己的余额（所有操作从 context 获取 user_id）
防越权：提现户名与 KYC 姓名强制一致性校验（后端校验，不依赖前端）
防刷单：重复转账三级拦截（同用户+同币种+同金额+处理中订单数判断）
```

## 9.3 数据安全

```
敏感信息加密：银行卡号、电子钱包ID 使用 AES 加密存储
  工具：common/pkg/utils 已提供 AES 加密工具
  加密密钥：从 ETCD /slg/conf/security/ 读取

操作审计：所有B端写操作调用 rpc.BLogClient().AddActLog() 记录审计日志
  参照：ser-item 中每个写操作都有审计日志调用

SQL注入：使用 GORM Gen 的类型安全查询，不拼接 SQL 字符串
```

# 十五、架构总结

## 一句话描述

**基于现有 CloudWeGo 微服务体系，采用四层分层架构，以"统一余额变动操作 + 乐观锁 + 事务 + 双层幂等"为核心安全机制，单服务单库部署，不引入额外组件，不过度设计。**

## 架构决策清单

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 服务粒度 | 单 ser-wallet 服务 | 数据内聚+事务简单+工程惯例 |
| 分层 | Handler→Service→Rep→Model | 现有工程统一范式 |
| 金额存储 | BIGINT 最小单位 | 无精度风险+整数运算快 |
| 并发安全 | 乐观锁（UPDATE WHERE available>=amount） | 简单高效+无死锁 |
| 事务边界 | RPC在外、DB操作在内 | 事务尽可能短 |
| 幂等 | 应用层状态判断 + DB唯一键兜底 | 双层防护 |
| 跨模块一致性 | 本地补偿 + 人工补单兜底 | 不引入分布式事务框架 |
| 缓存 | 币种配置用Redis，余额不缓存 | 读多写少缓存 + 资金强一致 |
| 安全 | 复用网关中间件链 + 后端强制校验 | 不信任前端 |
| 技术栈 | 不引入新组件 | 用现有的够用 |
