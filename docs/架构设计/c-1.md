## 一、架构设计总纲

### 1.1 我们要设计什么

一套**资金域微服务架构**，在现有 CloudWeGo（Hertz + Kitex）+ ETCD + GORM/gen 的工程体系下，承载三个业务模块的协作：

| 模块 | 部署形态 | 面向端 | 核心职责 |
|------|---------|--------|---------|
| **钱包模块** | ser-wallet 子模块 | C端 | 用户余额管理、充值/提现/兑换/记录；对外暴露余额变更 RPC |
| **币种配置** | ser-wallet 子模块 | B端 | 币种参数/汇率机制/汇率日志；对外暴露汇率和币种查询 RPC |
| **财务管理** | 独立服务（别人负责） | B端 | 通道/支付配置/审核/人工修正/统计；审核通过后调钱包 RPC 执行余额变更 |

### 1.2 架构设计的出发点

**不是从零造轮子，而是在已验证的工程体系上扩展。** 现有平台已运行 12+ 微服务（ser-app、ser-item、ser-user 等），技术选型、通信模式、分层规范、代码生成流程都已成熟。架构设计的核心工作是：

1. **沿用已有基础设施**：Kitex RPC、Hertz 网关、ETCD 配置、GORM/gen 代码生成
2. **针对资金域特性做增强**：事务保障、幂等机制、余额安全、汇率管理
3. **合理划分模块边界**：钱包和币种配置合并部署（降低汇率查询延迟），与财务模块 RPC 协作
4. **预留扩展但不提前实现**：代理钱包/场馆钱包预留枚举，不提前建表不提前编码

### 1.3 设计原则

| 原则 | 说明 | 在本架构中的体现 |
|------|------|----------------|
| **安全第一** | 资金系统容不得错误，宁可慢不可错 | 事务原子性、幂等去重、乐观锁防超额、流水强审计 |
| **高并发** | C端充值/提现/兑换可能突发并发 | 乐观锁替代悲观锁、Redis 缓存热数据、连接池复用 |
| **高性能** | 汇率查询是高频接口，不能每次都查 DB | 本地缓存 + Redis 缓存、计算值不落库、读写分离路径 |
| **不过度设计** | 需求未定型的部分不提前实现 | 代理/场馆钱包只预留枚举、消息队列视需求引入、不做分库分表 |
| **业界主流** | 参考支付/钱包领域的通用实践 | 冻结-扣款两阶段、事务流水一体、幂等 orderId、汇率快照 |

### 2.2 架构分层说明

整个资金域系统从上到下分为 **5 层**：

```
Layer 5 ── 接入层（网关）
           gate-font: C端用户请求入口，Token认证 + IP黑名单
           gate-back: B端管理请求入口，Token认证 + RBAC + IP白名单

Layer 4 ── 服务层（RPC 服务）
           ser-wallet: 钱包 + 币种配置（本方主责）
           ser-finance: 财务管理（别人负责）
           ser-kyc/ser-blog/ser-s3: 辅助服务（已有）

Layer 3 ── 业务逻辑层（ser/ 目录）
           deposit_ser / withdrawal_ser / exchange_ser: 业务编排
           wallet_ser: 余额操作核心（事务 + 幂等 + 乐观锁）
           rate_ser: 汇率查询封装（DB + 缓存）
           currency_ser: 币种配置 CRUD
           rpc_ser: RPC 对外统一入口
           record_ser: 记录查询

Layer 2 ── 数据访问层（rep/ 目录 + gorm_gen/）
           gorm_gen: 自动生成的 Model/Query/Repo
           rep: 手写的复杂查询封装

Layer 1 ── 基础设施层
           MySQL/TiDB: 持久化存储（余额、订单、流水、配置）
           Redis: 缓存（汇率、币种列表）+ 分布式锁
           ETCD: 配置中心 + 服务注册发现
```

### 2.3 为什么这样分层

| 决策 | 理由 | 依据 |
|------|------|------|
| **钱包和币种配置合并在 ser-wallet** | 钱包几乎每个接口都要查汇率/币种，合并后汇率查询变成本地调用（ns 级）而非跨进程 RPC（ms 级），减少 50%+ 的汇率查询延迟和故障点 | 链路关系分析：22 个 C端接口中 12 个需要查汇率/币种 |
| **财务管理独立服务** | 财务模块涉及通道密钥、三方回调验签等安全敏感操作，独立部署便于安全隔离；且由不同开发者负责，代码归属清晰 | 工程实际：不同开发者负责不同服务 |
| **服务间通过 RPC 协作** | 沿用现有 Kitex RPC + ETCD 服务发现模式，无需引入额外通信机制 | 工程认知：所有服务均通过 Kitex RPC 通信 |
| **不引入消息队列** | 当前业务场景下，充值回调由财务同步 RPC 调钱包即可保证一致性；引入 MQ 增加复杂度但收益有限 | 不过度设计原则；现有服务中 NATS/Kafka 主要用于日志和推送，非核心资金链路 |
| **不做分库分表** | 初期用户量和数据量不足以触发分表阈值（单表 < 1000万行），过早分表增加事务复杂度 | 不过度设计原则；TiDB 本身支持水平扩展 |

```
               └──────┬───────┘
                      │
               ┌──────▼───────┐
               │   rate_ser   │ ← 汇率查询封装（被 deposit/withdrawal/exchange 共用）
               │              │
               │  getRate     │ ← 获取指定币种汇率（优先 Redis → 回落 DB）
               │  getList     │ ← 获取启用币种列表
               │  calcDeposit │ ← 计算入款汇率
               │  calcWithdraw│ ← 计算出款汇率
               └──────────────┘
```

**调用规则（严格遵守，不可违反）**：

| 规则 | 说明 |
|------|------|
| wallet_ser 不调任何其他 ser | 它是最底层，只操作 DB |
| rate_ser 不调 wallet_ser | 它是独立的汇率工具层 |
| 业务 ser 可调 wallet_ser 和 rate_ser | 业务编排层依赖底层 |
| rpc_ser 只调 wallet_ser | RPC 对外入口，直接委托余额操作 |
| record_ser 直接查 DB | 纯读取，不经过 wallet_ser |
| handler.go 不写任何逻辑 | 纯委托，一行转发 |

### 3.3 handler.go 委托模式

遵循工程规范，handler 是纯委托层：

```
WalletServiceImpl {
    walletSer     *ser.WalletService      ← 余额核心
    depositSer    *ser.DepositService      ← 充值
    withdrawalSer *ser.WithdrawalService   ← 提现
    exchangeSer   *ser.ExchangeService     ← 兑换
    recordSer     *ser.RecordService       ← 记录
    currencySer   *ser.CurrencyService     ← 币种配置
    rpcSer        *ser.RpcService          ← RPC对外
}

每个 IDL 方法 → 一行委托到对应 ser
```

依据：工程认知文档中 ser-app 的 handler.go 每个方法一行委托到 ser 层。

| 数据库 | MySQL (TiDB) | 沿用现有工程统一选型，TiDB 兼容 MySQL 协议，支持水平扩展 |

### 4.3 核心表：wallet（用户钱包余额表）

这是整个系统最关键的表，所有余额操作最终都落到这张表上。

**唯一约束**：`(user_id, currency_code, wallet_type)` 三元组唯一——同一用户 + 同一币种 + 同一钱包类型 = 唯一一行记录。

**余额双字段设计**：

```
┌─────────────────────────────────────────────────────┐
│                    wallet 表                         │
│                                                     │
│  user_id + currency_code + wallet_type → 唯一行      │
│                                                     │
│  available_balance  ── 可用余额（用户可直接使用）       │
│  frozen_balance     ── 冻结余额（提现待审核时冻结）     │
│                                                     │
│  不变量: available + frozen = 该钱包的真实余额          │
│  任何操作都不能打破这个等式                             │
└─────────────────────────────────────────────────────┘
```

**为什么用"双字段"而非"单余额 + 冻结表"**：

| 方案 | 优点 | 缺点 |
|------|------|------|
| **双字段（采用）** | 一次查询获得完整余额信息；余额操作在单行 UPDATE 内完成，天然原子 | 冻结明细需额外查 withdrawal_order |
| 单余额 + 冻结表 | 可追踪每笔冻结明细 | 查余额需 JOIN 或两次查询；解冻需更新两张表增加事务复杂度 |

业界主流钱包系统（支付宝/微信支付）均采用**双字段**方案，冻结明细由订单表承担。

**金额精度**：

```
available_balance DECIMAL(20,8) NOT NULL DEFAULT 0
frozen_balance    DECIMAL(20,8) NOT NULL DEFAULT 0
```

- `DECIMAL(20,8)`：整数部分最大 12 位（万亿级），小数部分 8 位（满足加密货币精度）
- 法币虽然只需 2 位小数，但统一用 8 位避免表结构因币种不同而分化
- **Thrift IDL 中用 string 传输，Go 代码中用 shopspring/decimal 计算，DB 中用 DECIMAL 存储**

**并发安全**（乐观锁）：

```sql
-- 扣减/冻结操作，WHERE 条件保障不超额
UPDATE wallet
SET available_balance = available_balance - :amount,
    frozen_balance = frozen_balance + :amount,
    update_at = :now
WHERE user_id = :userId
  AND currency_code = :currency
  AND wallet_type = :walletType
  AND available_balance >= :amount     -- 关键：乐观锁
  AND delete_at = 0

-- 检查 affected_rows:
-- 0 → 余额不足，返回错误
-- 1 → 操作成功
```

为什么用乐观锁而非悲观锁（SELECT ... FOR UPDATE）：

| 方案 | 适用场景 | 理由 |
|------|---------|------|
| **乐观锁（采用）** | 冲突概率低、读多写少 | 不加行锁，并发性能高；同一用户同一时刻并发操作余额的概率极低 |
| 悲观锁 | 冲突概率高 | 加行锁阻塞并发，在"同一用户同时充值+提现"这种低概率场景下过重 |

依据：业界钱包系统（如 Stripe、PayPal 后端）对余额变更普遍采用乐观锁 + 单行 CAS 更新。

### 4.4 核心表：transaction_log（余额变更流水表）

**设计原则**：每一笔余额变动，无论来源（充值/提现/兑换/人工修正/补单），都必须在同一事务内写入一条流水记录。

```
┌──────────────────────────────────────────────┐
│              transaction_log                  │
│                                              │
│  id               BIGINT PK AUTO_INCREMENT   │
│  user_id          BIGINT NOT NULL            │
│  currency_code    VARCHAR(10) NOT NULL       │
│  wallet_type      INT NOT NULL               │
│  log_type         INT NOT NULL               │  ← 操作类型枚举
│  amount           DECIMAL(20,8) NOT NULL     │  ← 变更金额（正数）
│  direction        INT NOT NULL               │  ← 1=收入 2=支出
│  balance_before   DECIMAL(20,8) NOT NULL     │  ← 变更前余额
│  balance_after    DECIMAL(20,8) NOT NULL     │  ← 变更后余额
│  order_id         VARCHAR(50) NOT NULL       │  ← 关联业务单号（幂等依据）
│  remark           VARCHAR(200)               │
│  create_at        BIGINT NOT NULL            │
│                                              │
│  INDEX idx_user_currency (user_id, currency_code, create_at)  │
│  UNIQUE INDEX uk_order_id (order_id, log_type)                │
└──────────────────────────────────────────────┘
```

**为什么 order_id + log_type 做联合唯一索引**：

同一笔业务单号（如提现订单 T123456）可能产生两条流水——冻结时一条、确认扣款时一条。`order_id` 单独不唯一，`order_id + log_type` 才唯一。这个唯一索引同时承担**幂等去重**的职责，省掉独立的幂等表。

依据：业界主流做法是用流水表的唯一索引做幂等，而非独立幂等表。独立幂等表增加了表数量和维护成本，且需要额外的清理策略。

### 4.5 币种配置表设计

```
┌──────────────────────────────────────────────┐
│              currency_config                  │
│                                              │
│  id               BIGINT PK AUTO_INCREMENT   │
│  code             VARCHAR(10) UNIQUE         │  ← 币种代码（VND/IDR/THB/BSB/USDT）
│  name             VARCHAR(50)                │
│  type             INT                        │  ← 1=法币 2=加密 3=平台币
│  country          VARCHAR(50)                │  ← 国家（加密货币="全球"）
│  icon_url         VARCHAR(200)               │
│  currency_symbol  VARCHAR(10)                │  ← 货币符号（d/Rp/B/$）
│  thousand_sep     VARCHAR(5)                 │  ← 千分位符号
│  decimal_sep      VARCHAR(5)                 │  ← 小数点符号
│  decimal_places   INT                        │  ← 小数位数
│  realtime_rate    DECIMAL(20,8)              │  ← 实时汇率（三方API取均值）
│  platform_rate    DECIMAL(20,8)              │  ← 平台汇率（阈值触发后更新）
│  rate_floating    DECIMAL(5,2)               │  ← 汇率浮动百分比
│  threshold        DECIMAL(5,2)               │  ← 阈值百分比
│  sort_order       INT DEFAULT 0              │
│  status           INT DEFAULT 1              │  ← 0=禁用 1=启用
│  (通用审计字段)                                │
│                                              │
│  入款汇率 = platform_rate × (1 + rate_floating/100)  ← 计算值，不存储  │
│  出款汇率 = platform_rate × (1 - rate_floating/100)  ← 计算值，不存储  │
└──────────────────────────────────────────────┘
```

**为什么入款/出款汇率是计算值不存储**：

- 存储冗余：入款/出款汇率完全由 `platform_rate` 和 `rate_floating` 决定，存储是冗余的
- 一致性风险：如果存储了，修改 `rate_floating` 时必须同时更新三个字段，忘了就不一致
- 计算极轻：一次乘法运算（ns 级），不值得为此增加存储和同步逻辑

依据：汇率浮动本质上是一个函数关系 `f(platform_rate, floating) = deposit_rate / withdraw_rate`，函数关系应该在应用层计算，不应该在存储层冗余。

### 4.6 订单表设计要点

充值订单（deposit_order）和提现订单（withdrawal_order）的共同设计要点：

**订单号编码**：

| 类型 | 前缀 | 格式 | 示例 |
|------|------|------|------|
| 充值 | C | C + SnowflakeID | C7381920346112 |
| 提现 | T | T + SnowflakeID | T7381920346113 |
| 补单 | B | 财务生成 | B7381920346114 |
| 加款 | A | 财务生成 | A7381920346115 |
| 减款 | M | 财务生成 | M7381920346116 |

使用 `utils.GenerateSnowflakeID()` 生成（已有轮子），不用随机数。雪花 ID 有序（按时间递增），对 B+Tree 索引友好。

**汇率快照**：

```
rate_snapshot DECIMAL(20,8) NOT NULL  -- 订单创建时的汇率
```

订单创建时保存当时的汇率到订单记录中，后续入账/出款金额以此汇率为准。这是资金安全的基本要求——**不依赖后续汇率变动**。

依据：业界支付系统（Stripe、PayPal）均在订单创建时锁定汇率，不允许回溯修改。

## 五、余额安全架构

这是整个系统的核心中的核心。资金系统的安全性要求远高于普通业务系统。

### 5.1 事务保障

**原则：凡涉及余额变更的操作，余额 UPDATE + 流水 INSERT 必须在同一个数据库事务内完成。**

```
BEGIN TRANSACTION
├── 1. 幂等检查（INSERT ... ON DUPLICATE KEY → 如已存在则跳过后续）
├── 2. 余额 UPDATE（可能多条：如兑换涉及扣法币 + 加 BSB + 加赠送）
├── 3. 流水 INSERT（每个余额变动对应一条流水，记录 before/after）
├── 4. 订单状态 UPDATE（如有关联订单）
COMMIT
```

使用 gorm/gen 提供的事务支持：

```go
query.Q.Transaction(func(tx *query.Query) error {
    // 所有操作使用 tx，不使用 query.Q
    // 任何一步返回 error 则自动 rollback
    return nil
})
```

**为什么不用分布式事务（如 Saga/TCC）**：

钱包的余额变更和流水记录在**同一个数据库**内，用本地事务（ACID）即可保证原子性。分布式事务的场景是"跨数据库/跨服务"的数据一致性，当前架构中余额和流水共库，不需要引入额外复杂度。

如果未来拆分数据库（如流水表独立库），才需要考虑分布式事务。但当前阶段——**不过度设计**。

### 5.2 幂等机制

**场景**：财务模块调用钱包 RPC 入账时，可能因网络超时重试，导致同一笔入账被调用多次。

**方案**：基于 order_id + log_type 唯一索引做幂等。

```
任何余额变更 RPC 请求进入
    │
    ├── 在事务内尝试 INSERT transaction_log (order_id, log_type, ...)
    │
    ├── 唯一索引冲突（Duplicate Key）
    │   └── 说明已处理过 → 查询已有记录，返回成功（幂等响应）
    │
    └── INSERT 成功
        └── 继续执行余额 UPDATE → COMMIT
```

**为什么不用独立幂等表**：

- transaction_log 的 `(order_id, log_type)` 唯一索引天然承担了幂等去重的职责
- 少一张表 = 少一份维护成本
- 幂等检查和流水写入在同一事务内，天然原子
- 业界主流：很多支付系统用流水表的唯一键做幂等，而非独立幂等表

### 5.3 乐观锁防超额

见 4.3 节 wallet 表的 WHERE 条件设计。补充一个完整的操作流程：

```
freezeBalance(tx, userId, currency, walletType, amount):
    1. UPDATE wallet SET
         available_balance = available_balance - amount,
         frozen_balance = frozen_balance + amount
       WHERE user_id=? AND currency_code=? AND wallet_type=?
         AND available_balance >= amount AND delete_at=0

    2. 检查 affected_rows:
       - 0 → return BizErr(InsufficientBalance)
       - 1 → 继续（在同一事务内写流水）

    // 不需要先 SELECT 再 UPDATE，一步完成
    // 这是 CAS (Compare-And-Swap) 模式的数据库实现
```

**为什么不先 SELECT 检查余额再 UPDATE**：

- SELECT + UPDATE 之间有时间窗口，并发请求可能在 SELECT 通过后、UPDATE 前同时通过检查
- 单条 UPDATE 带 WHERE 条件是原子的（数据库行锁保证）
- 检查 affected_rows = 0/1 即可判断操作是否成功

### 5.4 核心不变量

```
┌─────────────────────────────────────────────────────────────────┐
│                      核心不变量                                   │
│                                                                 │
│  对于每个 (user_id, currency_code, wallet_type) 组合:             │
│                                                                 │
│  INV-1: available_balance >= 0                                  │
│         可用余额不能为负（乐观锁 WHERE available >= amount 保证）   │
│                                                                 │
│  INV-2: frozen_balance >= 0                                     │
│         冻结余额不能为负                                          │
│                                                                 │
│  INV-3: 每一笔 available/frozen 变动都有对应的 transaction_log    │
│         流水与余额变更在同一事务内，不存在"只改了余额没记流水"        │
│                                                                 │
│  INV-4: transaction_log 的 SUM(amount * direction) =             │
│         wallet 当前 available + frozen                           │
│         流水正向累计 = 当前余额（可用于定期对账校验）                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 5.5 资金操作的六个原子方法

wallet_ser 对外暴露 6 个余额变更方法，覆盖所有资金操作场景：

```
┌────────────────────────────────────────────────────────────────────────┐
│ 方法              │ SQL 语义                        │ 使用场景          │
├────────────────────────────────────────────────────────────────────────┤
│ addBalance        │ available += amount              │ 充值入账/加款/补单│
│ deductBalance     │ available -= amount              │ 人工减款/兑换扣币 │
│                   │ WHERE available >= amount        │                  │
│ freezeBalance     │ available -= amount              │ 创建提现订单      │
│                   │ frozen += amount                 │                  │
│                   │ WHERE available >= amount        │                  │
│ unfreezeBalance   │ frozen -= amount                 │ 提现驳回/出款失败 │
│                   │ available += amount              │                  │
│ confirmDeduct     │ frozen -= amount                 │ 提现出款成功      │
│                   │ (钱真正离开系统)                  │                  │
│ getBalance        │ SELECT available, frozen         │ 余额查询（只读）  │
└────────────────────────────────────────────────────────────────────────┘

所有写方法的共同特征:
├── 在数据库事务内执行
├── 同事务内写 transaction_log
├── RPC 入口方法先做幂等检查
└── 返回操作后的余额信息
```

## 六、汇率架构

### 6.1 四层汇率模型

```
┌──────────────────────────────────────────────────────────────┐
│                      汇率数据流                                │
│                                                              │
│  [外部]  三方汇率 API (3家)                                    │
│             │                                                │
│             │ 定时任务拉取，取平均值                              │
│             ▼                                                │
│  [Layer 1]  实时汇率 (realtime_rate)                           │
│             │  存入 currency_config.realtime_rate              │
│             │  代表: 当前市场真实汇率                            │
│             │                                                │
│             │ 偏差值 = |实时 - 平台| / 实时 × 100%              │
│             │ 当偏差值 >= 阈值% → 触发更新                       │
│             ▼                                                │
│  [Layer 2]  平台汇率 (platform_rate)                           │
│             │  存入 currency_config.platform_rate              │
│             │  代表: 平台采用的基础汇率（不随市场实时波动）         │
│             │                                                │
│             ├── × (1 + rate_floating%)                        │
│             │                                                │
│  [Layer 3]  入款汇率 (计算值，不存储)                            │
│             │  用于: 充值、投注                                  │
│             │  用户充值时获得略多（平台让利）                       │
│             │                                                │
│             ├── × (1 - rate_floating%)                        │
│             │                                                │
│  [Layer 4]  出款汇率 (计算值，不存储)                            │
│             │  用于: 提现                                      │
│             │  用户提现时获得略少（平台利差）                       │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 6.2 汇率缓存策略

汇率查询是高频接口（C端充值/提现/兑换都要调），需要缓存优化。

```
┌────────────────────────────────────────────────────────┐
│                    rate_ser 缓存策略                      │
│                                                        │
│  读取路径（展示类接口：币种列表、充值页汇率展示等）：        │
│                                                        │
│  请求 → Redis 缓存命中？                                 │
│          ├── 命中 → 直接返回（ms 级）                     │
│          └── 未命中 → 查 DB → 写入 Redis → 返回           │
│                                                        │
│  Redis Key: wallet:rate:{currency_code}                 │
│  Redis TTL: 30秒 ~ 1分钟                                │
│                                                        │
│  写入/失效路径：                                          │
│  ├── 定时任务更新平台汇率后 → 主动删除相关 Key             │
│  ├── B端编辑币种配置后 → 主动删除相关 Key                  │
│  └── TTL 自然过期                                       │
│                                                        │
│  执行类接口（创建订单、执行兑换等）：                       │
│  ├── 直接查 DB 取最新汇率                                │
│  └── 不用缓存值（防止缓存与DB不一致导致资金误差）           │
│                                                        │
└────────────────────────────────────────────────────────┘
```

**为什么执行类接口不用缓存**：

展示类接口（看一眼汇率）可以容忍短暂的缓存延迟。但执行类接口（基于汇率计算入账/出款金额）必须用最新值——30 秒的缓存延迟可能导致用户以"旧汇率"下单，产生资金差异。

### 6.3 BSB 特殊处理

BSB（平台币）的汇率逻辑与法币不同：

```
BSB 锚定 USDT: 10 BSB = 1 USDT

这个关系写死在代码中，不走配置:
├── currency_config 中 BSB 行的 platform_rate = 10（对 USDT）
├── BSB 不参与定时汇率更新
├── BSB 的 rate_floating 和 threshold 字段无意义
└── BSB 对法币的汇率通过 USDT 间接计算:
    BSB:VND = USDT:VND / 10
```

### 6.4 三方汇率 API 接入策略

```
┌────────────────────────────────────────────────────────┐
│                  定时汇率更新任务                         │
│                                                        │
│  触发: cron 定时（频率可配，初期 1天1次，上线后按需调整）    │
│                                                        │
│  FOR EACH 启用的非BSB币种:                               │
│    1. 并发请求 3 家 API (goroutine + WaitGroup)           │
│       ├── API_1 返回 rate_1                              │
│       ├── API_2 返回 rate_2                              │
│       └── API_3 返回 rate_3                              │
│                                                        │
│    2. 容错: 某个 API 失败则用剩余可用的取平均               │
│       至少 1 个可用才更新，全部失败则跳过本轮               │
│                                                        │
│    3. realtime_rate = avg(可用的 rates)                   │
│                                                        │
│    4. 阈值判断:                                          │
│       偏差值 = |realtime - platform| / realtime × 100%    │
│       IF 偏差值 >= threshold%:                           │
│         ├── UPDATE platform_rate = realtime_rate         │
│         ├── INSERT rate_change_log（记录变更）            │
│         └── 删除 Redis 缓存                              │
│       ELSE:                                             │
│         └── 只更新 realtime_rate，不动 platform_rate     │
│                                                        │
│  API 地址/密钥存 ETCD，支持运行时动态更换                   │
└────────────────────────────────────────────────────────┘
```

### 7.2 RPC 对外接口契约

这 8 个方法是财务模块调用钱包的核心接口，**必须优先确定、优先实现**。

| 方法 | 语义 | 幂等 | 事务 | 余额变更 |
|------|------|------|------|---------|
| R1 GetUserBalance | 查询余额 | - | - | 无（纯读取） |
| R2 CreditDeposit | 充值入账 | orderId 去重 | 余额+流水+订单 | available + N |
| R3 FreezeBalance | 冻结余额 | orderId 去重 | 余额+流水 | available - N, frozen + N |
| R4 UnfreezeBalance | 解冻退回 | orderId 去重 | 余额+流水+订单 | frozen - N, available + N |
| R5 ConfirmWithdrawal | 确认扣款 | orderId 去重 | 余额+流水+订单 | frozen - N |
| R6 ManualCredit | 人工加款 | orderId 去重 | 余额+流水 | 指定钱包 available + N |
| R7 ManualDebit | 人工减款 | orderId 去重 | 余额+流水 | 指定钱包 available - N |
| R8 SupplementCredit | 补单入账 | orderId 去重 | 余额+流水 | 中心 +N, 奖励 +赠送 |

**统一请求字段**：

```
所有余额变更方法的请求至少包含:
├── userId      (i64)    ← 操作对象
├── currency    (string) ← 币种代码
├── walletType  (i32)    ← 钱包类型（部分方法需要）
├── amount      (string) ← 变更金额
├── orderId     (string) ← 唯一业务单号（幂等依据）
└── reason      (string) ← 操作原因（审计用）
```

**统一响应字段**：

```
所有余额变更方法的响应至少包含:
├── success            (bool)    ← 操作是否成功
├── availableBalance   (string)  ← 操作后可用余额
├── frozenBalance      (string)  ← 操作后冻结余额
└── idempotent         (bool)    ← 是否为幂等返回（重复调用时 = true）
```

### 9.1 文件组织

```

├── service.thrift         ← 服务入口: include 所有子文件, 定义 WalletService
├── wallet.thrift          ← 余额/币种列表/钱包概览
```

**按功能域拆分 IDL 文件**的理由：

- 各文件职责清晰，单文件不超过 200 行
- 评审后如果决定拆分为独立服务（如币种配置独立），直接拿走对应文件即可
- 多人协作时不会产生 IDL 文件冲突

### 9.2 IDL 字段约定

| 约定 | Thrift 类型 | Go 生成类型 | 依据 |
|------|------------|------------|------|
| 金额 | string | string → shopspring/decimal 计算 | 避免浮点精度丢失 |
| 时间 | i64 | int64 (毫秒时间戳) | 工程 IDL 规范 |
| 枚举 | i32 | int32 | 工程 IDL 规范 |
| 分页请求 | pageNo(i32) + pageSize(i32) | int32 | 工程分页模式 |
| 分页响应 | pageList(list) + total(i64) + pageNo(i32) + pageSize(i32) | - | 工程分页模式 |
| 命名 | {Action}{Entity}Req / Resp | - | 工程命名规范 |

## 十一、缓存架构

### 11.1 缓存使用场景

| 场景 | 缓存 Key | TTL | 写入时机 | 失效时机 |
|------|---------|-----|---------|---------|
| 汇率查询 | `wallet:rate:{code}` | 30s-1min | 查 DB 后写入 | 汇率更新后主动删除 |
| 币种列表 | `wallet:currency:list` | 1-5min | 查 DB 后写入 | 币种配置编辑后主动删除 |
| 用户余额 | **不缓存** | - | - | - |

**用户余额为什么不缓存**：

- 余额是高频变更的数据（充值/提现/兑换都会改），缓存一致性难保证
- 余额查询是单行 WHERE 查询（命中唯一索引），DB 查询本身很快（< 1ms）
- 缓存余额在并发余额变更时会出现"显示余额与实际余额不一致"的体验问题
- 业界主流钱包系统（支付宝/微信支付）的余额查询也不走缓存

### 11.2 缓存模式

采用 **Cache-Aside** 模式（旁路缓存）：

```
读: 先查 Redis → 命中则返回 → 未命中则查 DB → 写入 Redis → 返回
写: 先写 DB → 成功后删除 Redis（不更新缓存，下次读时自然回填）
```

为什么不用 Write-Through（写穿透）：

- Write-Through 需要在每次写 DB 时同步更新缓存，增加写路径复杂度
- 汇率/币种配置的写入频率极低（汇率更新可能几小时一次），Cache-Aside 的"删除 + 下次读回填"更简单
- 不过度设计

## 十二、定时任务架构

### 12.1 任务清单

| 任务 | 触发方式 | 优先级 | 说明 |
|------|---------|--------|------|
| 汇率定时更新 | cron 定时 | 核心 | 拉取三方 API → 阈值判断 → 更新平台汇率 → 写日志 → 清缓存 |


不在 ser-wallet 内部自建 cron 调度，遵循工程已有的集中调度模式。

## 十三、安全架构

### 13.1 传输层安全

| 层级 | 安全措施 | 来源 |
|------|---------|------|
| C端 → gate-font | HTTPS + Token 认证 + IP 黑名单 | 已有基础设施 |
| B端 → gate-back | HTTPS + Token 认证 + RBAC 鉴权 + IP 白名单 | 已有基础设施 |
| 网关 → 微服务 | Kitex TTHeader 传递认证信息 + 内网通信 | 已有基础设施 |

### 13.2 业务层安全

| 安全要求 | 实现方式 |
|---------|---------|
| KYC 强制校验 | 法币/电子钱包提现必须通过 KYC 认证，通过 RPC 调 ser-kyc 校验 |
| 提现姓名一致性 | 提现账户持有人 = KYC 认证姓名（忽略大小写和空格后比对） |
| 充值回调幂等 | 基于 orderId 去重，重复回调不重复入账 |
| 余额不可超额 | 乐观锁 WHERE available >= amount |
| 冻结与订单原子 | 同一数据库事务 |
| 流水强审计 | 每笔余额变更同事务内写 transaction_log |
| 汇率快照 | 订单创建时锁定汇率，不依赖后续变动 |
| 导出安全 | B端导出需密码 + 谷歌验证码 |

### 13.3 敏感数据

| 数据 | 处理方式 |
|------|---------|
| 提现银行卡号 | 存储全量，展示时脱敏（已有 `utils.mask_util.go`） |
| KYC 证件信息 | ser-kyc 管理，钱包只查状态和姓名，不存证件 |
| 通道密钥 | 财务模块管理，钱包不接触 |

## 十四、扩展性设计

### 14.1 当前预留但不实现的

| 预留项 | 预留方式 | 不实现的理由 |
|--------|---------|------------|
| 代理钱包 | 枚举值 wallet_type=4 | 需求文档灰字标注"暂不制作" |
| 场馆钱包 | 枚举值 wallet_type=5 | 需求文档灰字标注"暂不制作" |
| 消费扣款/返奖 | IDL 方法预留 | 随投注模块同步实现 |
| 分库分表 | 不预留 | 数据量未触发阈值，TiDB 支持水平扩展 |

### 14.2 未来可能的扩展方向

| 方向 | 触发条件 | 扩展方案 |
|------|---------|---------|
| 币种配置独立服务 | 评审决定 / 团队扩大 | IDL 已按功能域分文件，拆分成本低 |
| 流水表分表 | 单表超过 1000 万行 | 按 user_id 取模分表（TiDB 原生支持） |
| 余额缓存 | 极高 QPS（> 10000/s） | Redis 缓存余额 + 写操作后失效 |
| 消息队列 | 充值回调需要解耦 | 引入 NATS/Kafka 做异步入账 |
| 对账系统 | 上线稳定后 | 定时任务比对 transaction_log 累计 vs wallet 余额 |

### 14.3 为什么当前不做这些

**不过度设计原则**：每一项扩展都有明确的触发条件。在条件未触发之前，增加的复杂度 > 收益。等到需要时再加，成本也不高（架构已预留了扩展空间）。

## 十六、架构决策记录

以下汇总本方案中的关键架构决策及其理由，便于评审讨论。

| # | 决策 | 选择 | 理由 | 替代方案及否决理由 |
|---|------|------|------|------------------|
| 1 | 钱包和币种配置是否合并 | **合并在 ser-wallet** | 汇率查询变本地调用（ns 级 vs ms 级），减少 50%+ 延迟和故障点 | 独立服务：增加网络开销，当前规模不值得 |
| 2 | 余额双字段 vs 冻结表 | **双字段** | 单行 UPDATE 原子操作，查询简单 | 冻结表：增加 JOIN 和事务复杂度 |
| 3 | 乐观锁 vs 悲观锁 | **乐观锁** | 同用户并发余额操作概率低，不加行锁性能高 | 悲观锁：不必要的锁开销 |
| 4 | 幂等表 vs 流水表唯一索引 | **流水表唯一索引** | 少一张表，幂等检查和流水写入天然在同一事务 | 独立幂等表：额外维护成本 |
| 5 | 入款/出款汇率存储 vs 计算 | **计算值不存储** | 避免冗余，避免一致性风险 | 存储：修改浮动时需同步三字段 |
| 6 | 用户余额是否缓存 | **不缓存** | 高频变更难保一致，单行查询本身很快 | 缓存：一致性复杂度高 |
| 7 | 是否引入消息队列 | **当前不引入** | 同步 RPC 满足需求，MQ 增加复杂度 | MQ：当前规模收益不够 |
| 8 | 是否分库分表 | **当前不分** | 数据量未触发阈值，TiDB 支持扩展 | 分表：过早增加事务复杂度 |
| 9 | 定时任务方式 | **ser-cron 调度 + RPC** | 沿用工程已有集中调度模式 | 内部 cron：违反统一调度规范 |
| 10 | 订单号生成 | **雪花 ID** | 有序、对索引友好、已有轮子 | 随机数：无序，索引性能差 |

| 3 | C端充值/提现方式查询路径 | A: C端 → 钱包 → 财务(RPC) / B: C端 → 财务(直接) | A: 钱包作聚合层；B: 钱包少接口但前端多调一个服务 |

## 十八、总结

### 18.1 架构全景一句话

**在现有 CloudWeGo + ETCD + GORM/gen 工程体系上，新增 ser-wallet 服务（编号 022），将钱包和币种配置合并部署，通过 6 个余额原子方法 + 事务 + 幂等 + 乐观锁构建资金安全核心，与财务模块通过 8 个 RPC 接口双向协作，汇率采用四层模型 + 阈值更新 + Redis 缓存，不引入消息队列不分库分表，在安全和性能之间取得平衡。**

### 18.2 架构关键数字

```
服务数量:        1 个 (ser-wallet，编号 022)
IDL 方法数:      ~36 个 (C端 22 + B端 5 + RPC 8 + 通用 1)
数据库表数:      10 张
余额原子方法:    6 个 (add/deduct/freeze/unfreeze/confirm/get)
RPC 对外接口:    8 个 (R1-R8，供财务调用)
RPC 调外部:      ~7 个 (财务 3 + KYC 1 + BLog 1 + S3 1 + BUser 1)
C端路由:         ~20 个
B端路由:         ~5 个
错误码区间:      6022000 ~ 6022599
缓存 Key 类型:   2 个 (汇率 + 币种列表)
定时任务:        2 个 (汇率更新 + 订单超时)
开发阶段:        4 个 (骨架 → 地基 → 业务 → 完善)
```

### 18.3 架构安全红线

```
绝对不可违反:
├── 余额变更必须在数据库事务内
├── 余额变更必须同事务写流水
├── RPC 余额变更必须幂等
├── 余额扣减必须用乐观锁 WHERE available >= amount
├── 冻结和创建订单必须同一事务
├── 订单创建时必须保存汇率快照
├── 金额存储用 DECIMAL，传输用 string，计算用 shopspring/decimal
└── 不允许 float/double 参与金额运算
```
