# ser-wallet 架构设计方案

> 设计时间：2026-02-26
> 设计原则：安全、高并发、高性能、不过度设计、业界主流
> 设计依据：产品需求文档 + 产品原型 + 现有工程实际代码模式 + 业界钱包/支付系统通用架构
> 设计边界：我负责钱包+币种配置，配合财务模块联调，整体作为一个和"钱"相关的系统来设计

---

## 一、架构定位：ser-wallet 在整个资金体系中的角色

### 1.1 资金体系全景

钱包、币种配置、财务管理三个模块共同构成平台的**资金体系**。在业界，这个体系通常分为三层：

```
┌─────────────────────────────────────────────────────────────┐
│                     资金体系三层架构                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  【账户层】ser-wallet                                        │
│   管理用户账户余额、记录每一笔资金变动                         │
│   职责：知道每个用户有多少钱，每一分钱怎么来的怎么走的          │
│   类比：银行的核心账务系统                                    │
│                                                             │
│  【配置层】ser-wallet（币种配置子模块）                        │
│   管理币种基础数据、汇率体系                                  │
│   职责：知道平台支持什么币种，各币种之间怎么换算                │
│   类比：银行的外汇牌价管理                                    │
│                                                             │
│  【通道层】财务模块（他人负责）                                │
│   管理支付通道、充提订单审核、资金进出                         │
│   职责：知道钱从哪个通道进来、从哪个通道出去                    │
│   类比：银行的支付清算系统                                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 职责边界

```
ser-wallet 只管「账」，不管「通道」：
  ✓ 用户有多少钱（余额管理）
  ✓ 钱怎么来的怎么走的（流水记录）
  ✓ 币种怎么换算（汇率管理）
  ✓ 冻结多少、可用多少（状态管理）
  ✗ 钱从哪个支付通道进来（财务模块管）
  ✗ 用哪个通道出款（财务模块管）
  ✗ 通道费率和轮询规则（财务模块管）

这个边界意味着：
  - ser-wallet 不直接和三方支付通道交互
  - ser-wallet 不存储通道配置、通道密钥
  - ser-wallet 通过 RPC 调用财务模块来对接通道能力
  - 财务模块通过 RPC 调用 ser-wallet 来变更用户余额
```

## 二、整体架构：分层 + 分域

### 2.1 架构分层

在现有工程的 handler → ser → rep → gorm_gen 四层基础上，针对钱包的资金属性，在 Service 层内部进一步划分：

```
┌──────────────────────────────────────────────────────────────┐
│  接入层  │  gate-font (C端)  │  gate-back (B端)  │  RPC     │
│          │  中间件链路        │  中间件链路        │  直接调用 │
└──────────┴──────┬────────────┴────────┬──────────┴────┬─────┘
                  │                     │               │
┌─────────────────▼─────────────────────▼───────────────▼─────┐
│  Handler 层  │  handler.go — 薄封装，IDL方法 → Service 方法   │
└──────────────┴──────────────────────┬───────────────────────┘
                                      │
┌─────────────────────────────────────▼───────────────────────┐
│  业务编排层  │  internal/ser/                                 │
│  (Service)  │  deposit_service / withdraw_service /           │
│             │  exchange_service / record_service              │
│             │  组合调用下层能力 + 对接外部模块                   │
└─────────────┴──────────────────────┬───────────────────────-─┘
                                     │
┌────────────────────────────────────▼────────────────────────┐
│  账户核心层  │  internal/ser/wallet_core.go                   │
│  (Core)     │  creditCore / debitCore / freezeCore /          │
│             │  unfreezeCore / confirmDebitCore                │
│             │  幂等、事务、并发控制全在这一层                    │
│             │  对外 RPC（CreditWallet 等）也直接调这一层        │
├─────────────┼─────────────────────────────────────────────── ┤
│  配置核心层  │  internal/ser/currency_core.go                  │
│  (Config)   │  币种CRUD / 汇率计算 / 汇率刷新                  │
│             │  为上层所有操作提供币种和汇率数据                   │
└─────────────┴──────────────────────┬────────────────────────┘
                                     │
┌────────────────────────────────────▼────────────────────────┐
│  数据访问层  │  internal/rep/                                  │
│  (Repo)     │  wallet_rep / currency_rep / flow_rep /         │
│             │  deposit_rep / withdraw_rep / exchange_rep      │
└─────────────┴──────────────────────┬────────────────────────┘
                                     │
┌────────────────────────────────────▼────────────────────────┐
│  生成代码层  │  internal/gorm_gen/  (model + query + repo)     │
│  (Gen)      │  自动生成，禁止手动编辑                           │
└─────────────┴───────────────────────────────────────────────┘
```

**关键设计决策：为什么在 Service 层拆分出"账户核心层"？**

```
业界依据：支付/钱包系统的标准做法是将「账务引擎」独立为核心层。
  - 蚂蚁金服的资金核心系统分为"账务核心"和"业务层"
  - Stripe 架构将 balance transactions 作为独立的核心模块
  - 所有成熟支付系统都将余额变更操作原子化并集中管理

工程依据：现有工程中 ser-item 的 domain 层已有类似思路（item_cache.go / item_log_builder.go）。
  wallet_core 承担的角色类似于 domain 层，但更偏向事务性操作。

实际价值：
  - C端充值调 creditCore，RPC 财务回调也调 creditCore，逻辑完全一致
  - 幂等校验、并发控制、事务管理集中在一处，不会遗漏
  - 上层业务编排层只关心"业务流程"，不关心"余额怎么安全地变更"
```

### 2.2 业务域划分

```
ser-wallet 内部分为 5 个业务域：

┌──────────────────────────────────────────────────────────┐
│                    ser-wallet 业务域                      │
│                                                          │
│  ┌──────────────┐  ┌────────────┐  ┌────────────────┐   │
│  │ 币种配置域    │  │ 账户域     │  │  交易域         │   │
│  │              │  │            │  │                │   │
│  │ 币种管理     │  │ 钱包账户    │  │ 充值业务       │   │
│  │ 汇率管理     │  │ 余额操作    │  │ 提现业务       │   │
│  │ 汇率日志     │  │ 冻结管理    │  │ 兑换业务       │   │
│  │              │  │ 流水记录    │  │ 订单管理       │   │
│  └──────┬───────┘  └─────┬──────┘  └───────┬────────┘   │
│         │                │                  │            │
│         │        ┌───────▼────────┐         │            │
│         └───────→│  共享：缓存/   │←────────┘            │
│                  │  幂等/错误码   │                       │
│                  └────────────────┘                       │
│                                                          │
│  ┌──────────────┐  ┌────────────────────────────────┐    │
│  │ 记录查询域    │  │ 定时任务域                      │    │
│  │              │  │                                │    │
│  │ 交易记录     │  │ 汇率定时刷新                    │    │
│  │ 记录详情     │  │                                │    │
│  └──────────────┘  └────────────────────────────────┘    │
└──────────────────────────────────────────────────────────┘

域间依赖规则：
  币种配置域 → 被所有域依赖，自身无依赖（最底层）
  账户域    → 依赖币种配置域，被交易域和外部RPC依赖（核心层）
  交易域    → 依赖币种配置域 + 账户域 + 外部模块RPC（最上层）
  记录查询域 → 只读，依赖各域的数据表
  定时任务域 → 依赖币种配置域

  严禁循环依赖：账户域不能依赖交易域，币种配置域不能依赖账户域。
```

## 三、数据架构：账户模型设计

这是钱包系统最核心的架构决策。

### 3.1 账户模型

**业界标准做法**：用户资金按「账户」维度管理，每个账户是一个独立的余额容器。

```
账户结构：
  一个用户 × 一个币种 × 一个钱包类型 = 一个账户

  例如用户 10001：
    账户1：user_id=10001, currency=VND,  wallet_type=center   → 可用余额=500000, 冻结=0
    账户2：user_id=10001, currency=VND,  wallet_type=reward   → 可用余额=50000,  冻结=0
    账户3：user_id=10001, currency=BSB,  wallet_type=center   → 可用余额=1000,   冻结=200
    账户4：user_id=10001, currency=BSB,  wallet_type=reward   → 可用余额=100,    冻结=0
    账户5：user_id=10001, currency=IDR,  wallet_type=center   → 可用余额=8000000,冻结=0
    ...

为什么这样设计：
  ✓ 币种之间完全隔离（需求要求"不支持跨币种混合"）
  ✓ 钱包类型之间独立（中心/奖励/主播/代理各自的余额互不干扰）
  ✓ 查询高效（按 user_id + currency + type 精准定位一行）
  ✓ 并发隔离（不同币种/类型的操作互不阻塞）
```

### 3.2 核心表结构方向

**① user_wallet — 账户表（核心中的核心）**

```
设计思路：
  主键：自增 ID
  唯一索引：(user_id, currency_code, wallet_type) — 保证同一维度只有一个账户
  余额字段：available_balance（可用）, frozen_balance（冻结）
  金额类型：BIGINT（存最小单位整数，如分/最小货币单位）

为什么金额用 BIGINT 整数：
  业界标准：Stripe、PayPal、蚂蚁金服的账务系统内部全部用整数存储
  工程依据：现有工程 ser-item 的 base_price 字段是 int32 类型（整数存储）
  原因：浮点运算有精度丢失问题（0.1 + 0.2 ≠ 0.3），资金系统绝不允许
  方案：
    VND/IDR（整数币种）：1 VND = 1（最小单位就是 1）
    THB/BSB（2位小数币种）：1.50 THB = 150（存放"分"）
    USDT（8位小数币种）：1.00000001 USDT = 100000001（存 satoshi）
  展示时按币种配置的小数位规则除回去
```

**② wallet_flow — 流水表（账务凭证）**

```
设计思路：
  每次余额变更写一条流水，是对账和审计的唯一凭证。
  业界称为 "transaction journal" 或 "ledger entry"。

核心字段方向：
  flow_id         — 雪花ID
  user_id         — 用户
  currency_code   — 币种
  wallet_type     — 钱包类型
  flow_type       — 流水类型（充值/扣款/冻结/解冻/确认扣除/兑换/返奖/加款/减款/...）
  amount          — 变更金额（正数=入账，负数=出账）
  balance_before  — 变更前余额
  balance_after   — 变更后余额
  order_no        — 关联业务订单号（幂等依据）
  remark          — 备注

  唯一索引：(order_no, flow_type) — 幂等保证

为什么流水表这么重要：
  1. 对账：sum(amount) WHERE flow_type IN (入账类) - sum(amount) WHERE flow_type IN (出账类) = 当前余额
  2. 审计：任何余额变更都有迹可循
  3. 幂等：order_no 唯一索引，重复写入直接冲突
  4. 排查：出了问题可以回溯每一笔变更
```

**③ currency_config — 币种配置表**

```
核心字段方向：
  currency_code    — 币种代码（PK 或唯一）
  currency_name    — 名称
  currency_type    — 类型（法定/加密/平台）
  icon_url         — 图标
  symbol           — 货币符号
  thousand_sep     — 千分位符号
  decimal_sep      — 小数点符号
  decimal_digits   — 小数位数
  real_time_rate   — 实时汇率（三方API平均值）
  platform_rate    — 平台汇率
  deposit_rate     — 入款汇率
  withdraw_rate    — 出款汇率
  float_rate       — 浮动比例
  threshold        — 阈值
  is_base          — 是否基准币种
  status           — 启用/禁用

  存储精度：汇率字段用 BIGINT，乘以 10^8 存储（保留8位精度）
```

**④~⑧ 业务表（充值订单/提现订单/兑换记录/提现账户/汇率日志）**

设计原则与上述一致：金额用 BIGINT，雪花ID做订单号，软删除用 deleted_at。

### 3.3 金额精度方案

```
┌──────────────────────────────────────────────────────┐
│              金额精度统一方案                          │
│                                                      │
│  存储层：BIGINT（最小单位整数）                         │
│  传输层：string（IDL 中金额字段用 string 类型）         │
│  计算层：Go int64 运算                                │
│  展示层：按 decimal_digits 配置格式化                   │
│                                                      │
│  转换示例（THB，2位小数）：                             │
│    用户看到：B 1,500.50                               │
│    传输值：  "150050"                                  │
│    存储值：  150050  (BIGINT)                          │
│    计算：    150050 + 50000 = 200050                   │
│    展示：    200050 / 100 = 2000.50 → "B 2,000.50"    │
│                                                      │
│  转换示例（VND，0位小数）：                             │
│    用户看到：d 500,000                                 │
│    传输值：  "500000"                                  │
│    存储值：  500000  (BIGINT)                          │
│    计算：    500000 - 100000 = 400000                  │
│    展示：    400000 → "d 400,000"                      │
│                                                      │
│  为什么传输用 string：                                  │
│    Thrift 的 i64 最大 9.2×10^18，足够                  │
│    但 JSON 序列化后前端 JS Number 精度只有 2^53         │
│    用 string 传输避免前端精度丢失                       │
│    （这是业界标准做法，Stripe API 也用 string）         │
└──────────────────────────────────────────────────────┘
```

## 四、并发控制架构

钱包系统最大的技术挑战是并发写入——多个请求同时操作同一个账户的余额。

### 4.1 并发场景分析

```
高并发场景：
  用户同时收到多笔返奖（投注结算批量回调）→ 同一账户被并发 credit
  用户快速点击充值 + 投注同时扣款 → 同一账户被并发 credit 和 debit
  活动批量发奖 → 大量用户同时被 rewardCredit

低并发但高风险场景：
  提现冻结 + 投注扣款同时来 → 可用余额可能被超扣
  人工加款 + 充值上账同时来 → 必须都成功，不能丢
```

### 4.2 并发控制方案：乐观更新 + 行级条件

**方案选择依据：**

```
方案对比：
┌──────────────┬──────────────────┬───────────────────────┐
│ 方案         │ 适用场景          │ 现有工程是否使用       │
├──────────────┼──────────────────┼───────────────────────┤
│ 悲观锁       │ 高冲突写入        │ 未使用                │
│ SELECT FOR   │ 银行转账          │                       │
│ UPDATE       │                  │                       │
├──────────────┼──────────────────┼───────────────────────┤
│ 乐观锁       │ 中等冲突          │ 未使用 version 字段    │
│ version 字段  │ 电商库存          │ 但用了条件更新模式     │
├──────────────┼──────────────────┼───────────────────────┤
│ 条件更新      │ 数值增减          │ ✓ ser-item 使用       │
│ WHERE >=     │ 余额/库存扣减     │ gorm.Expr + Add()     │
│ + gorm.Expr  │                  │                       │
├──────────────┼──────────────────┼───────────────────────┤
│ 分布式锁      │ 跨服务互斥        │ 未使用                │
│ Redis Lock   │                  │                       │
└──────────────┴──────────────────┴───────────────────────┘

选择：条件更新（WHERE balance >= amount）+ 事务

理由：
  1. 现有工程已有此模式（ser-item 的 gorm.Expr + Add/Sub）
  2. 不需要额外的 version 字段（减少表复杂度）
  3. 数据库层面保证原子性（单条 UPDATE 语句天然原子）
  4. 性能优于 SELECT FOR UPDATE（减少锁持有时间）
  5. 能满足当前并发量级（用户级别的操作，非亿级秒杀）
```

### 4.3 具体实现模式

**扣款（debit）：**

```go
// 核心思路：UPDATE ... SET balance = balance - amount WHERE balance >= amount
// 通过 affected rows 判断是否成功

result := tx.Model(&model.UserWallet{}).
    Where("user_id = ? AND currency_code = ? AND wallet_type = ?", userID, currCode, walletType).
    Where("available_balance >= ?", amount).  // 关键：余额充足条件
    Updates(map[string]interface{}{
        "available_balance": gorm.Expr("available_balance - ?", amount),
        "updated_at":        now,
    })

if result.RowsAffected == 0 {
    // 两种可能：账户不存在 or 余额不足
    // 查一下区分原因
    return ErrInsufficientBalance
}
// 成功，继续写流水
```

**上账（credit）：**

```go
// 上账不需要 WHERE >= 条件（加钱不会失败）
// 但需要处理账户不存在的情况

result := tx.Model(&model.UserWallet{}).
    Where("user_id = ? AND currency_code = ? AND wallet_type = ?", userID, currCode, walletType).
    Updates(map[string]interface{}{
        "available_balance": gorm.Expr("available_balance + ?", amount),
        "updated_at":        now,
    })

if result.RowsAffected == 0 {
    // 账户不存在，自动创建
    tx.Create(&model.UserWallet{
        UserID:           userID,
        CurrencyCode:     currCode,
        WalletType:       walletType,
        AvailableBalance: amount,
        FrozenBalance:    0,
        // ...
    })
}
```

**冻结（freeze）：**

```go
// 从可用转到冻结，原子操作
result := tx.Model(&model.UserWallet{}).
    Where("user_id = ? AND currency_code = ? AND wallet_type = 'center'", userID, currCode).
    Where("available_balance >= ?", amount).
    Updates(map[string]interface{}{
        "available_balance": gorm.Expr("available_balance - ?", amount),
        "frozen_balance":    gorm.Expr("frozen_balance + ?", amount),
        "updated_at":        now,
    })
```

### 4.4 为什么不用分布式锁

```
当前阶段不需要：
  1. 所有余额操作最终都是单库单表的 UPDATE，数据库行级锁已经足够
  2. ser-wallet 是单服务多实例部署，但操作的是同一个数据库，
     数据库本身就是"分布式锁"的天然实现
  3. 引入 Redis 分布式锁会增加复杂度（锁超时、锁续期、死锁检测）
  4. 当前用户级并发不会极高（不是双11秒杀场景）

什么时候需要：
  如果未来出现单账户极高频并发写入（如热门主播秒级收到上千笔打赏），
  可以在 wallet_core 层加一层 Redis 分布式锁做入口限流，
  但这是优化手段，不是初始设计。不过度设计。
```

## 五、幂等架构

### 5.1 为什么是架构级关注点

```
钱包系统中，幂等不是"可选优化"，是"安全底线"：
  - 财务回调可能重发（三方通道重试机制）
  - RPC 框架超时重试（Kitex 默认行为）
  - 网络抖动导致调用方不确定是否成功，再次发起

没有幂等 → 重复入账 → 平台亏钱
没有幂等 → 重复扣款 → 用户投诉
```

### 5.2 幂等方案

```
策略：order_no 作为幂等键，在流水表中用唯一索引保证

┌─────────────────────────────────────────────────────┐
│  幂等处理流程（每个写操作的第一步）                     │
│                                                     │
│  1. 接收请求，取出 order_no                          │
│  2. 在事务内，尝试 INSERT wallet_flow (order_no, ...)│
│     → 成功（首次请求）：继续执行余额变更              │
│     → 唯一键冲突（重复请求）：                       │
│        查出该 order_no 对应的流水记录                 │
│        直接返回上次的处理结果                         │
│  3. 余额变更 + 流水写入在同一事务内提交                │
│                                                     │
│  唯一索引：wallet_flow (order_no, flow_type)         │
│  为什么加 flow_type：同一个 order_no 可能既有冻结又有 │
│  确认扣除，flow_type 不同算不同操作                   │
└─────────────────────────────────────────────────────┘
```

## 六、状态机架构

### 6.1 冻结状态机

```
                   freeze
    ┌─────────┐  ──────────→  ┌──────────────┐
    │  正常    │               │  冻结中       │
    │(无冻结) │               │  (Frozen)     │
    └─────────┘               └───┬──────┬────┘
                                  │      │
                    unfreeze      │      │  confirmDebit
                   (驳回/失败)    │      │  (出款成功)
                                  │      │
                   ┌──────────────▼┐  ┌──▼──────────────┐
                   │  已解冻        │  │  已确认扣除      │
                   │ (Unfrozen)    │  │ (Confirmed)     │
                   │ 余额退回可用   │  │ 余额永久减少     │
                   └───────────────┘  └─────────────────┘

约束：
  - Frozen → Unfrozen 和 Frozen → Confirmed 互斥
  - 终态不可逆（Unfrozen/Confirmed 后不能再变）
  - 每个 order_no 只能对应一条冻结记录
```

## 七、缓存架构

### 7.1 缓存分层

```
┌────────────────────────────────────────────────────────────────┐
│                      缓存架构                                  │
│                                                                │
│  热数据（高频读，低频变）                                        │
│  ┌─────────────────────────────────────────────────────┐       │
│  │  币种列表         wallet:currencies       TTL 5min  │       │
│  │  币种详情         wallet:curr:{code}      TTL 5min  │       │
│  │  汇率数据         wallet:rate:{code}      定时覆盖   │       │
│  │  基准币种         wallet:base_currency    TTL 1h    │       │
│  └─────────────────────────────────────────────────────┘       │
│  特点：变更由B端管理员触发，频率极低。                             │
│  失效策略：B端写操作后主动 DEL。                                  │
│                                                                │
│  温数据（中频读，中频变）                                        │
│  ┌─────────────────────────────────────────────────────┐       │
│  │  用户余额         wallet:bal:{uid}:{code} TTL 30s   │       │
│  └─────────────────────────────────────────────────────┘       │
│  特点：每次打开钱包首页读，每次余额变更写。                        │
│  失效策略：余额变更后主动 DEL（不更新，下次读时回填）。             │
│  防击穿：短 TTL + 主动失效已足够，无需 singleflight               │
│  （现有工程无此模式，不额外引入）                                  │
│                                                                │
│  不缓存的数据                                                   │
│  ┌─────────────────────────────────────────────────────┐       │
│  │  流水记录 / 订单记录 / 提现账户                       │       │
│  └─────────────────────────────────────────────────────┘       │
│  特点：读频率不高（用户主动查），数据量大，不适合缓存。              │
│  直接查数据库 + 分页。                                           │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 7.2 缓存一致性保证

```
模式：先写数据库，再删缓存（Cache-Aside Pattern）
  这是业界最成熟的缓存模式，现有工程 ser-item/ser-app 已在使用。

流程：
  写操作 → 数据库事务提交 → DEL Redis Key
  读操作 → 先读 Redis → 未命中则读 DB → 回填 Redis

为什么不用"先删缓存再写数据库"：
  并发场景下，删完缓存、还没写完数据库时，另一个读请求会把旧数据回填缓存。

为什么不用"写数据库同时更新缓存"：
  并发写时，两个写操作的缓存更新顺序可能和数据库不一致，导致缓存脏数据。
  DEL 操作是最安全的——下次读时自然拿到最新数据。
```

## 八、事务架构

### 8.1 事务边界原则

```
原则：事务只在 ser-wallet 自己的数据库内开，不做跨服务分布式事务。

业界依据：
  微服务架构下，跨服务分布式事务（2PC/TCC/Saga）复杂度极高，
  大多数成熟系统选择"本地事务 + 最终一致性"。
  蚂蚁金服、微信支付的公开架构分享都明确表示避免分布式事务。

现有工程依据：
  GORM 配置 SkipDefaultTransaction: true（显式事务控制）
  现有事务模式：query.Q.Transaction() 和 db.Transaction()
  没有引入任何分布式事务框架（无 Seata、无 DTM）
```

### 8.2 事务场景清单

```
场景一：单钱包操作（上账/扣款）
  事务范围：UPDATE user_wallet + INSERT wallet_flow
  一个事务，两条 SQL。

场景二：冻结/解冻/确认
  事务范围：UPDATE user_wallet + INSERT/UPDATE freeze_record + INSERT wallet_flow
  一个事务，三条 SQL。

场景三：兑换（跨币种双向操作）
  事务范围：
    UPDATE user_wallet (法币, center)  SET balance -= X
    UPDATE user_wallet (BSB, center)   SET balance += Y
    UPDATE user_wallet (BSB, reward)   SET balance += Z
    INSERT wallet_flow (法币扣减)
    INSERT wallet_flow (BSB入账)
    INSERT wallet_flow (BSB赠送入账)
  一个事务，六条 SQL。虽然操作多，但都是同一个数据库内，没问题。

场景四：创建提现订单（本地冻结 + 调财务）
  事务范围：冻结在一个本地事务内完成。
  调财务是事务外的 RPC 调用。
  如果调财务失败 → 执行 unfreezeCore（另一个本地事务）补偿。
  这不是分布式事务，是"本地事务 + 补偿"模式。
```

### 8.3 跨服务一致性保证

```
模式：本地事务 + 异步通知 + 幂等重试

充值场景：
  财务模块收到三方回调 → 调 ser-wallet.CreditWallet
  如果调用超时：财务模块重试调用
  ser-wallet 端：幂等校验保证不重复入账
  最终一致性：只要不断重试，最终一定会成功入账

提现场景：
  ser-wallet 冻结余额 → 调财务模块创建审核订单
  如果调用失败：ser-wallet 执行解冻（本地补偿）
  财务审核后回调 → 调 ser-wallet.ConfirmDebit 或 UnfreezeBalance
  ser-wallet 端：冻结状态机 + 幂等校验保证正确性

关键原则：
  1. 先做本地不可逆操作（冻结余额），再调外部
  2. 外部调用失败可以补偿（解冻）
  3. 外部回调通过幂等保证不重复处理
  4. 不需要额外的消息队列来保证一致性（RPC 重试 + 幂等已足够）
```

## 九、安全架构

### 9.1 接入层安全

```
C端安全（已有机制，直接复用）：
  gate-font 中间件链：BlackCheck → FontAuthMiddleware
  - IP黑名单：拦截恶意 IP
  - Token校验：验证用户登录态
  - 用户ID注入：从 Token 中提取，不信任客户端传入的 user_id

B端安全（已有机制，直接复用）：
  gate-back 中间件链：WhiteCheck → BackAuthMiddleware
  - IP白名单：只允许办公网络 IP
  - Token校验：验证管理员登录态
  - URL权限校验：CheckUrlAuth，管理员只能访问被授权的 URL

钱包特有补充：
  无。现有安全机制已覆盖接入层需求。不额外设计。
```

### 9.2 业务层安全

```
资金安全规则（在 Service 层实现）：

① 用户ID以服务端为准
  C端请求的 user_id 从 context 中取（tracer.GetUserId(ctx)）
  不允许 C端请求体中传 user_id（防止篡改操作他人钱包）

② 余额不允许为负
  扣款/冻结使用 WHERE available_balance >= amount 条件
  数据库层面兜底：available_balance 和 frozen_balance 字段加 CHECK >= 0
  （TiDB 支持 CHECK 约束）

③ 所有写操作有幂等保证
  防止重复入账/扣款（见第五节）

④ 冻结状态机严格互斥
  防止冻结金额被同时确认和解冻（见第六节）

⑤ KYC 校验后端强制
  提现时的 KYC 状态校验在 ser-wallet 服务端执行
  账户持有人姓名与 KYC 姓名的一致性校验在服务端执行
  不依赖前端（前端置灰只是 UX 手段，不是安全手段）

⑥ B端操作审计
  所有 B 端写操作调 ser-blog 记录审计日志
  基准币种设置需二级确认
  币种禁用需提示有余额用户
```

### 9.3 数据层安全

```
① 敏感字段加密
  提现账户的银行账号、手机号使用 EncryptedString 类型
  （现有工程 ser-kyc 已有此模式，GORM 自动加解密）

② 数据库访问控制
  ser-wallet 通过 ETCD 获取数据库连接信息
  数据库用户只授予 ser_wallet 库的权限
  不与其他服务共享数据库用户

③ 软删除
  所有表用 deleted_at 字段，不硬删除
  保证数据可追溯（资金系统的数据永远不应该被物理删除）
```

## 十、性能架构

### 10.1 读性能

```
高频读接口：GetWalletHome / GetCurrencyList / GetBalance / GetExchangeRate

优化手段：
  1. Redis 缓存（见第七节）
     余额、币种、汇率全部缓存
     命中率预期 > 90%（大部分请求不到数据库）

  2. 数据库索引
     user_wallet: (user_id, currency_code, wallet_type) 唯一索引 → 精准定位一行
     wallet_flow: (user_id, currency_code, created_at) 索引 → 记录查询分页
     deposit_order: (user_id, currency_code, status, created_at) → 充值记录+重复检测
     withdraw_order: 同理

  3. 查询优化
     记录列表查询只读必要字段（SELECT 指定列，不 SELECT *）
     分页用 LIMIT + OFFSET（数据量可控范围内足够）
```

### 10.2 写性能

```
写操作特点：频率远低于读，但每笔都要求正确。

优化手段：
  1. 事务最小化
     事务内只做必要操作（UPDATE wallet + INSERT flow）
     事务外再做缓存失效、日志记录等

  2. 减少锁竞争
     不同用户的操作天然不竞争（不同行）
     同一用户不同币种不竞争（不同行）
     只有同一用户同一币种同一类型的并发写才竞争

  3. 批量操作
     批量查询余额用 IN 查询而不是循环单条查
     批量返奖可考虑异步化（NATS 消息 → 消费者依次处理）
```

### 10.3 不过度设计

```
以下方案在当前阶段不引入：

× 读写分离
  当前数据量和并发量不需要。TiDB 本身支持分布式读写，
  单库已能满足。未来数据量增长时 TiDB 可透明扩展。

× 分库分表
  同理。TiDB 自带分布式能力，不需要应用层分片。

× 消息队列解耦余额操作
  余额变更要求实时响应（调用方需要知道结果），
  不适合异步化。同步 RPC + 幂等重试 是最简单可靠的方案。

× 独立的账务微服务
  当前钱包模块的体量不需要再拆分。
  币种配置、账户管理、交易业务放在同一个服务内，
  减少服务间调用，降低运维复杂度。
```

## 十一、集成架构：与外部模块的交互模式

### 11.1 同步 RPC 调用

```
ser-wallet 主动调用（请求-响应模式）：

  ser-wallet ──RPC──→ 财务模块
    GetPaymentConfig / CreatePayOrder / CreateWithdrawOrder
    特点：在 C端用户操作链路上，需要等结果
    超时：3秒（Kitex 默认）
    失败处理：返回错误给用户（"暂不可用"）

  ser-wallet ──RPC──→ ser-kyc / ser-user / ser-blog / ser-s3
    KycQuery / GetUserInfoExpend / CreateLog / GetPresignUrl
    特点：已有服务，接口稳定
    超时：3秒
    失败处理：根据场景降级（KYC失败→保守策略，日志失败→不阻断）
```

### 11.2 被调用（请求-响应模式）

```
外部 ──RPC──→ ser-wallet：

  财务模块 → CreditWallet / ConfirmDebit / UnfreezeBalance / Manual系列
    特点：异步触发（回调/审核后），但调用本身是同步的
    幂等：必须，order_no 唯一
    返回：调用方需要知道结果（成功/失败/已处理）

  投注模块 → DebitWallet / CreditWallet / GetBalance
    特点：投注链路实时性要求高
    性能：GetBalance 走缓存，DebitWallet 单次数据库事务

  直播/活动 → AnchorIncome / RewardCredit / GetBalance
    特点：频率中等
```

### 11.3 异步事件（可选扩展点）

```
当前阶段不引入异步消息，但架构上预留：

  如果未来需要：
    余额变更后发 NATS 事件 → 通知统计模块更新报表
    大额操作发 NATS 事件 → 通知风控模块
    活动批量发奖 → 通过 NATS 队列消费，避免瞬时高并发

  现有工程已有 NATS JetStream 基础设施：
    common/pkg/nats/ 提供了 ConsumerConfig、死信队列等完整能力
    需要时直接复用，无需额外搭建

  但当前不引入的原因：
    余额变更是同步操作（调用方需要知道结果）
    统计需求还不明确
    不过度设计
```

## 十三、架构决策记录（ADR）

将本方案中的关键架构决策集中记录，便于后续回溯。

| # | 决策 | 选项 | 选择 | 理由 |
|---|------|------|------|------|
| 1 | 金额存储类型 | DECIMAL / BIGINT整数 | **BIGINT整数** | 业界标准（Stripe/蚂蚁），避免浮点精度问题，现有工程int32模式一致 |
| 2 | 金额传输类型 | i64 / string | **string** | 避免前端 JS Number 精度丢失，业界标准做法 |
| 3 | 并发控制 | 悲观锁 / 乐观锁(version) / 条件更新 | **条件更新** | 现有工程已有模式（gorm.Expr），性能优，复杂度低 |
| 4 | 幂等实现 | 独立幂等表 / 流水表唯一索引 | **流水表唯一索引** | 减少一张表，流水本身就是幂等证据 |
| 5 | 事务策略 | 分布式事务 / 本地事务+补偿 | **本地事务+补偿** | 现有工程无分布式事务框架，业界微服务主流做法 |
| 6 | 缓存策略 | Write-Through / Cache-Aside | **Cache-Aside** | 现有工程已有模式（ser-item/ser-app），成熟可靠 |
| 7 | 缓存失效 | 更新缓存 / 删除缓存 | **删除缓存** | 并发安全，避免脏数据，业界标准 |
| 8 | 状态机实现 | 状态机框架 / WHERE条件 | **WHERE条件** | 不引入额外依赖，数据库天然支持，复杂度低 |
| 9 | 服务拆分 | 钱包+币种在同一服务 / 拆两个服务 | **同一服务** | 币种配置和钱包紧耦合，拆分增加RPC调用，无收益 |
| 10 | 异步消息 | 当前引入NATS / 预留不引入 | **预留不引入** | 当前需求同步RPC已满足，不过度设计 |
| 11 | 分布式锁 | Redis锁 / 不用 | **不用** | 数据库行级条件更新已够，当前并发量不需要 |
| 12 | 读写分离 | 主从分离 / 单库 | **单库** | TiDB 分布式架构自带，不需应用层处理 |
| 13 | 汇率精度 | DECIMAL(20,8) / BIGINT×10^8 | **BIGINT×10^8** | 与金额存储方案一致，整数运算更快更安全 |
| 14 | 敏感数据 | 明文 / 加密存储 | **加密（EncryptedString）** | 银行账号等敏感字段加密，复用 ser-kyc 现有模式 |

## 十四、架构质量属性总结

```
安全性 ✓
  接入层：复用现有 IP检查 + Token校验 + URL权限（已验证成熟）
  业务层：user_id 服务端取、余额非负约束、幂等、状态机互斥、KYC强校验
  数据层：敏感字段加密、软删除不物理删、独立数据库权限

高并发 ✓
  读：Redis 缓存（命中率 > 90%）+ 数据库索引
  写：条件更新（无全局锁）+ 不同用户天然隔离 + 事务最小化
  上限：单账户级别的写并发由数据库行级处理，足够

高性能 ✓
  读路径：Redis → DB（大部分在 Redis 就返回）
  写路径：单次数据库事务（2~6条SQL）+ 缓存失效（1次DEL）
  RPC：Kitex 多路复用 + 3秒超时 + TTHeader 追踪

不过度设计 ✓
  不引入：分布式事务、分布式锁、读写分离、分库分表、消息队列解耦余额
  不拆分：钱包和币种配置放同一服务
  不抽象：状态机用 WHERE 条件，不引入框架
  一句话：能用数据库解决的不用中间件，能不拆的不拆

业界主流 ✓
  账户模型：用户×币种×类型 = 一个账户（标准多币种钱包模型）
  金额处理：整数存储 + string 传输（Stripe/PayPal/蚂蚁标准）
  并发控制：条件更新 WHERE balance >= amount（电商/支付通用模式）
  幂等：业务单号唯一索引（所有支付系统标配）
  缓存：Cache-Aside + DEL（最经典的缓存模式）
  一致性：本地事务 + 幂等重试（微服务架构主流做法）
```
