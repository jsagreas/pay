### 1.2 三个模块的架构关系

这不是三个独立系统，而是**一套资金系统的三个切面**：

```
币种配置 = 基础数据层    提供"度量衡"（币种定义+汇率规则）
钱包     = 账务核心层    提供"账本"（余额管理+交易流水）
财务管理 = 运营管控层    提供"管道"（支付通道+审核流程）
```

架构约束：
- **余额只有钱包能改**。财务管理不直接操作余额，必须通过钱包 RPC
- **通道只有财务能管**。钱包不直接对接三方支付，必须通过财务的通道
- **汇率只有币种配置能出**。钱包和财务都从币种配置获取汇率，不各自维护

这种分权设计是金融系统的基本原则——**关注点分离 + 职责单一**。

### 1.3 为什么是一个服务而不是多个

币种配置和钱包在同一个 ser-wallet 中，原因：

| 考量 | 合并 | 拆分 |
|------|------|------|
| 调用频率 | 钱包每个操作都要读汇率/币种，内部函数调用 0 延迟 | 拆成两个服务就是 RPC 调用，每次增加 1-3ms 网络开销 |
| 数据耦合 | 钱包的余额展示强依赖币种的展示规则（小数位/千分位/符号） | 拆开后两个服务还是紧密耦合，分了也白分 |
| 运维复杂度 | 1 个服务部署 | 2 个服务部署 + 2 倍配置 + 2 倍监控 |
| 团队边界 | 就你一个人做 | 没有拆分的组织动力 |
| 未来扩展 | 代码层面按包隔离，未来需要拆时改下 import 即可 | — |

**结论：不拆。代码层面按包隔离，物理上一个服务。业界称为"模块化单体"（Modular Monolith），是微服务过度拆分反思后的主流实践。**

> 依据：工程中 ser-app 包含 Banner + HomeCategory + ContentStrategy 三个域在一个服务内，已有先例

### 2.2 金额存储策略

**业界标准做法：整数存储，应用层换算。**

```
存储层：bigint（最小单位）
  法币（VND/IDR/THB）：× 100（分），精度 2 位小数
  加密货币（USDT）：× 10^8（聪），精度 8 位小数
  平台币（BSB）：× 100，精度 2 位小数

传输层：i64（Thrift IDL），和存储层一致
展示层：按币种的 decimal_digits 配置格式化
```

为什么不用 decimal：
1. TiDB 的 `bigint` 性能优于 `decimal`，索引更高效
2. Go 的 `int64` 运算比 `decimal` 库快一个数量级
3. 整数不存在精度丢失
4. 支付宝、微信支付、Stripe 都是整数存储（单位：分）

**汇率存储**：汇率是比率，不可避免有小数。统一放大 10^8 存 bigint，应用层计算后截断。

```
例：1 USDT = 25432.50 VND
存储：rate_platform = 2543250000000（= 25432.50 × 10^8）
计算：vnd_amount = usdt_amount × rate / 10^8
```

### 2.3 wallet 表——架构核心

```sql
CREATE TABLE `wallet` (
    `id`              bigint unsigned NOT NULL AUTO_INCREMENT,
    `user_id`         bigint unsigned NOT NULL COMMENT '用户ID',
    `currency_code`   varchar(16) NOT NULL COMMENT '币种编码',
    `wallet_type`     tinyint unsigned NOT NULL COMMENT '钱包类型 1中心 2奖励 3主播 4代理 5场馆',
    `balance`         bigint NOT NULL DEFAULT 0 COMMENT '可用余额（最小单位）',
    `frozen_amount`   bigint NOT NULL DEFAULT 0 COMMENT '冻结金额（最小单位）',
    `version`         bigint unsigned NOT NULL DEFAULT 0 COMMENT '乐观锁版本号',
    `status`          tinyint NOT NULL DEFAULT 1 COMMENT '1正常 2冻结',
    `create_by`       bigint unsigned DEFAULT 0,
    `update_by`       bigint unsigned DEFAULT 0,
    `create_at`       bigint unsigned NOT NULL DEFAULT 0,
    `update_at`       bigint unsigned NOT NULL DEFAULT 0,
    `delete_at`       bigint unsigned NOT NULL DEFAULT 0,
    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_user_curr_type` (`user_id`,`currency_code`,`wallet_type`,`delete_at`),
    KEY `idx_user_id` (`user_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_bin COMMENT='用户钱包';
```

**关键设计点**：

**`version` 字段**——乐观锁。这是并发控制的核心，下面第四章展开。

**`balance` + `frozen_amount` 分离**——提现冻结时 `balance` 减少、`frozen_amount` 增加；出款成功时 `frozen_amount` 减少。两个字段分开记录，任何时刻 `balance + frozen_amount = 用户实际资产`。

**唯一键 `uk_user_curr_type`**——保证一个用户一个币种一个钱包类型只有一条记录。加了 `delete_at` 是为了软删除后允许重建。

### 2.4 wallet_transaction 表——对账生命线

```sql
CREATE TABLE `wallet_transaction` (
    `id`               bigint unsigned NOT NULL AUTO_INCREMENT,
    `user_id`          bigint unsigned NOT NULL,
    `currency_code`    varchar(16) NOT NULL,
    `wallet_type`      tinyint unsigned NOT NULL,
    `tx_type`          tinyint unsigned NOT NULL COMMENT '1充值 2兑换 3提现 4加款 5减款 6补单 7冻结 8解冻',
    `amount`           bigint NOT NULL COMMENT '变动金额（正=增加，负=减少）',
    `balance_before`   bigint NOT NULL COMMENT '变更前余额',
    `balance_after`    bigint NOT NULL COMMENT '变更后余额',
    `order_no`         varchar(32) NOT NULL COMMENT '关联订单号',
    `status`           tinyint NOT NULL DEFAULT 1 COMMENT '1已完成 2处理中 3已失败 4已拒绝',
    `remark`           varchar(256) DEFAULT '' COMMENT '备注',
    `create_by`        bigint unsigned DEFAULT 0,
    `update_by`        bigint unsigned DEFAULT 0,
    `create_at`        bigint unsigned NOT NULL DEFAULT 0,
    `update_at`        bigint unsigned NOT NULL DEFAULT 0,
    `delete_at`        bigint unsigned NOT NULL DEFAULT 0,
    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_order_wallet_tx` (`order_no`,`wallet_type`,`tx_type`,`delete_at`),
    KEY `idx_user_curr_type` (`user_id`,`currency_code`,`tx_type`),
    KEY `idx_order_no` (`order_no`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_bin COMMENT='钱包交易流水';
```

**`balance_before` + `balance_after`**——这是金融系统的标配。对账时只需要：
```
遍历流水：每条流水的 balance_after 是否等于 balance_before + amount？
最后一条流水的 balance_after 是否等于 wallet 表的 balance？
```
如果不一致，说明数据被非法修改，立刻告警。

**唯一键 `uk_order_wallet_tx`**——同一个订单、同一个钱包类型、同一个交易类型只能有一条流水。这是数据库级的幂等保障。

> 依据：支付宝/微信支付的交易流水都包含变更前后余额；TiDB 的唯一键可直接作为幂等屏障

### 3.2 引擎接口设计

```go
// BalanceEngine 余额变更引擎，所有资金操作的唯一入口
type BalanceEngine struct {
    walletRepo      *rep.WalletRepo
    transactionRepo *rep.TransactionRepo
}

// BalanceOp 余额操作参数
type BalanceOp struct {
    UserID       int64
    CurrencyCode string
    WalletType   int32
    Amount       int64   // 正数
    TxType       int32
    OrderNo      string
    Remark       string
}

// Add 增加可用余额
func (e *BalanceEngine) Add(ctx context.Context, op *BalanceOp) error

// Deduct 扣减可用余额（余额不足返回错误）
func (e *BalanceEngine) Deduct(ctx context.Context, op *BalanceOp) error

// DeductBestEffort 尽力扣减（余额不足扣实际余额，用于人工减款）
func (e *BalanceEngine) DeductBestEffort(ctx context.Context, op *BalanceOp) (actualAmount int64, err error)

// Freeze 冻结：可用余额 → 冻结余额
func (e *BalanceEngine) Freeze(ctx context.Context, op *BalanceOp) error

// Unfreeze 解冻：冻结余额 → 可用余额
func (e *BalanceEngine) Unfreeze(ctx context.Context, op *BalanceOp) error

// ConfirmDeduct 确认扣除冻结余额（资金永久移出）
func (e *BalanceEngine) ConfirmDeduct(ctx context.Context, op *BalanceOp) error
```

### 3.3 引擎内部执行流程（以 Add 为例）

```
func (e *BalanceEngine) Add(ctx, op) error {
    return e.walletRepo.WithTransaction(func(tx) error {

        // 第一步：幂等检查
        exists := tx.WalletTransaction.Where(orderNo=op.OrderNo, walletType=op.WalletType, txType=op.TxType).Exist()
        if exists {
            return nil  // 已处理，直接成功
        }

        // 第二步：锁定 wallet 行
        wallet := tx.Wallet.Where(userId, currencyCode, walletType).ForUpdate().First()
        if wallet == nil {
            // 首次操作该币种该钱包类型，自动创建
            wallet = autoCreateWallet(tx, op)
        }

        // 第三步：计算新余额
        newBalance := wallet.Balance + op.Amount

        // 第四步：乐观锁更新
        affected := tx.Wallet.Where(id=wallet.ID, version=wallet.Version).
            UpdateSimple(balance=newBalance, version=wallet.Version+1)
        if affected == 0 {
            return ErrConcurrentConflict  // 乐观锁冲突，上层重试
        }

        // 第五步：写流水
        tx.WalletTransaction.Create(&WalletTransaction{
            UserID:        op.UserID,
            CurrencyCode:  op.CurrencyCode,
            WalletType:    op.WalletType,
            TxType:        op.TxType,
            Amount:        op.Amount,
            BalanceBefore:  wallet.Balance,
            BalanceAfter:   newBalance,
            OrderNo:       op.OrderNo,
            Status:        StatusCompleted,
        })

        return nil
    })
}
```

**为什么行锁 + 乐观锁双保险**：

- `FOR UPDATE`（行锁）：在同一事务内防止其他事务读到旧数据
- `version`（乐观锁）：跨事务兜底——如果两个请求都读到了同一行，只有一个能更新成功

业界对比：
- 支付宝：行锁 + 乐观锁（amount + version 双重校验）
- Stripe：行锁 + 幂等键
- 本设计：行锁 + 乐观锁 + 幂等键（三重保障）

> 依据：工程中 GORM 已启用 `SkipDefaultTransaction: true`（mysql.go:44），手动事务管理更高效

## 四、并发控制架构

### 4.1 并发场景分析

钱包面临的并发场景：

| 场景 | 频率 | 风险 | 示例 |
|------|------|------|------|
| 同一用户同时操作 | 中等 | 高 | 用户充值回调和提现冻结同时到达 |
| 财务回调重复 | 低 | 极高 | 三方支付重复推送充值成功回调 |
| 运营和用户同时操作 | 低 | 高 | 运营人工减款和用户同时提现 |
| 多币种并发 | 低 | 低 | 不同币种操作互不影响（锁粒度到 user+currency+walletType） |

### 4.2 三层并发防线

```
第一层：数据库行锁（事务内强一致）
  SELECT ... FOR UPDATE WHERE user_id=? AND currency_code=? AND wallet_type=?
  锁粒度：用户 + 币种 + 钱包类型
  作用：同一事务内保证读写的原子性
  自动释放：事务 COMMIT/ROLLBACK 后释放

第二层：乐观锁（跨事务冲突检测）
  UPDATE wallet SET balance=?, version=version+1 WHERE id=? AND version=?
  作用：如果两个事务并发执行，第二个会因为 version 不匹配而更新 0 行
  冲突处理：返回错误，调用方重试

第三层：幂等键（重复请求过滤）
  wallet_transaction 唯一键：order_no + wallet_type + tx_type
  作用：即使并发两个完全相同的请求，数据库唯一键也会拒绝第二个
  冲突处理：INSERT 失败 → 查询已有记录 → 返回成功
```

### 4.3 为什么不用 Redis 分布式锁

| 方案 | 优点 | 缺点 |
|------|------|------|
| **数据库行锁（选用）** | 锁和数据操作在同一事务中，天然原子；锁粒度精确；无需维护额外组件 | 持有锁期间占用数据库连接 |
| Redis 分布式锁 | 不占用数据库连接 | 锁和数据操作不在同一事务中，需要额外保证一致性；锁续期/释放复杂；Redis 故障影响可用性 |
| Redis + 数据库双重锁 | 两层保护 | 过度设计，增加复杂度和延迟 |

**结论：当前规模用数据库行锁就够了。** 理由：

1. wallet 表按 `user_id + currency_code + wallet_type` 锁定，锁粒度非常细——一个用户的一种币种的一种钱包类型才是一把锁，用户间完全不互相阻塞
2. TiDB 支持行级锁（基于 Percolator 事务模型），性能足够
3. 单用户的并发操作本身就不高——一个用户不会同时做 100 笔充值
4. 减少了对 Redis 的依赖——Redis 故障不影响核心资金操作

**如果未来用户量级增长到需要的程度**，可以在数据库行锁之上加 Redis 前置过滤（先抢 Redis 锁，抢到了再走数据库事务），但现在不加——不过度设计。

> 依据：工程中 Redis 的 `rds.SetNX` 可用但不作为资金操作的主锁；TiDB 官方推荐行锁用于金融场景

### 5.1 幂等分层

│                  幂等架构三层                          │
│                                                      │

## 六、一致性架构

### 6.1 一致性分级

不是所有数据都需要强一致，分级对待：

| 数据 | 一致性级别 | 原因 | 实现方式 |
|------|-----------|------|---------|
| **余额** | **强一致** | 钱不能多不能少 | 数据库事务 + 行锁 |
| **交易流水** | **强一致** | 必须和余额同步 | 和余额在同一事务中写入 |
| **订单状态** | **强一致** | 状态机不能乱 | 数据库事务 + 状态校验 |
| 币种配置 | 最终一致 | 配置变更可以容忍秒级延迟 | DB 是源，Redis 缓存 + TTL |
| 汇率 | 最终一致 | 汇率本身就在波动 | 定时同步，缓存覆盖 |
| 审计日志 | 最终一致 | 不影响业务 | oneway RPC，丢了不影响 |

### 6.2 余额和流水的强一致保证

```
每次余额变更的原子操作：

BEGIN
  1. SELECT wallet ... FOR UPDATE         ← 锁行
  2. 校验余额
  3. UPDATE wallet SET balance=?, version=version+1  ← 改余额
  4. INSERT wallet_transaction (balance_before, balance_after)  ← 写流水
COMMIT

如果第3步成功但第4步失败 → 自动回滚，第3步也撤销
如果中间任何一步失败 → 全部回滚，余额和流水都不变
```

**核心原则：余额变更和流水写入永远不分离。** 这是金融系统的铁律——有余额变动就必须有流水记录，有流水记录就必须有余额变动。

### 6.3 跨服务一致性（钱包 ↔ 财务）

钱包和财务之间不需要分布式事务。原因：

```
充值链路：
  财务收到回调 → 财务更新自己的订单状态 → RPC 调钱包 W-R2 上账
  如果 W-R2 失败 → 财务重试（W-R2 幂等，重试安全）

提现链路：
  钱包冻结成功 → 通知财务审核
  财务出款成功 → RPC 调钱包 W-R5 确认扣款
  如果 W-R5 失败 → 财务重试（W-R5 幂等，重试安全）
```

**模式：本地事务 + 幂等重试**，不需要 TCC/Saga/XA。这是业界处理跨服务资金操作最主流的方式——每个服务管好自己的事务，跨服务通过幂等重试保证最终一致。

> 依据：微信支付、支付宝的跨服务资金操作均采用本地事务 + 幂等重试模式，不用分布式事务

### 7.2 C端关键安全措施

**用户只能操作自己的钱包**：

## 八、性能架构

### 8.1 读写分离思路

```
写操作（低频高权重）：
  余额变更 → 走主库，行锁事务
  订单创建 → 走主库
  目标：正确性 > 性能

读操作（高频低权重）：
  余额查询 → 走主库（余额必须实时准确，不走从库）
  币种/汇率 → 走 Redis 缓存
  交易记录 → 走主库（分页查询走索引）
  目标：毫秒级响应
```

**为什么余额查询不走缓存**：

余额变动频繁，缓存一致性维护成本高（每次余额变更都要更新缓存，还有并发覆盖风险）。直接走 TiDB，索引命中 `uk_user_curr_type`，单行查询 < 1ms。不值得为了省这 1ms 引入缓存一致性问题。

### 8.2 缓存架构

只缓存**变动频率低、读取频率高**的数据：

```
┌──────────────────────────────────────────────────────────┐
│                       Redis 缓存层                        │
│                                                           │
│  wallet:currency:list                                     │
│    内容：全量启用币种列表（JSON）                            │
│    来源：currency_config 表                                │
│    写入：B端修改币种后主动更新                              │
│    TTL：不设（主动失效）                                    │
│    场景：C端每次打开钱包/充值/提现都要读                     │
│                                                           │
│  wallet:rate:{currency_code}                              │
│    内容：该币种四个汇率值                                   │
│    来源：SyncRealtimeRate 定时任务                          │
│    写入：汇率同步任务每次执行后覆盖                          │
│    TTL：不设（定时覆盖）                                    │
│    场景：兑换预览、充值换算、提现换算                        │
│                                                           │
│  wallet:exchange:snapshot:{id}                            │
│    内容：兑换预览时的汇率快照                               │
│    来源：W-8 GetExchangePreview                            │
│    写入：预览时写入                                         │
│    TTL：5分钟                                              │
│    场景：W-9 ExecuteExchange 执行时校验                     │
│                                                           │
│  不缓存：                                                  │
│    余额（强一致，直接查DB）                                  │
│    交易流水（走索引够快）                                    │
│    订单数据（状态流转频繁）                                  │
└──────────────────────────────────────────────────────────┘
```

### 8.3 数据库性能优化

**连接池**：工程默认 MaxOpen=20, MaxIdle=10（mysql.go）。对 ser-wallet 足够——即使峰值并发，行锁保证了同一用户的操作串行化，连接争抢有限。

**PrepareStmt**：工程已开启（mysql.go:41）。对钱包模块特别有利——余额查询和更新是高频重复 SQL，预编译缓存直接命中。

**索引覆盖**：
- `wallet` 表：`uk_user_curr_type` 覆盖所有余额查询
- `wallet_transaction` 表：`idx_user_curr_type` 覆盖 C端记录列表查询
- `deposit_order` / `withdrawal_order`：`uk_order_no` 覆盖订单号查询

**SQL 优化要点**：
- 余额更新只改 `balance`/`frozen_amount`/`version`，不改其他字段 → 减少锁持有时间
- 交易记录分页查询用 `ORDER BY id DESC LIMIT offset, size` → 利用主键索引
- 软删除条件 `delete_at = 0` 必须在每个查询中 → 索引前缀包含此条件

### 8.4 RPC 性能

```
工程已有的 RPC 优化（rpc/init.go）：
  - MuxTransport：连接多路复用，一条 TCP 连接承载多个请求
  - MuxConnection(1)：客户端复用一条连接
  - TTHeader：元数据在 Header 中传播，不侵入业务负载
  - 3秒超时：防止下游卡死拖垮上游

ser-wallet 不需要额外的 RPC 优化。Kitex 的连接复用 + TiDB 的行锁级并发
已经能支撑数万 QPS 的余额操作。
```

> 依据：Kitex 官方 benchmark：单连接多路复用可达 10万+ QPS

## 九、事件驱动架构

### 9.1 哪些场景需要异步事件

```
不是所有操作都要发事件。只在以下场景使用：

必须异步的：
  - 审计日志（oneway RPC → ser-blog）         已有机制
  - 汇率同步（定时任务触发）                    用 cron

可以考虑异步的：
  - 充值成功后通知客户端刷新余额                用 WebSocket/长连接推送
  - 余额变动后的统计数据更新                    用 Kafka 事件

不需要异步的：
  - 余额变更本身                               必须同步事务
  - 订单创建                                   必须同步返回结果
  - 兑换执行                                   必须同步返回新余额
```

### 9.2 事件架构（如果需要）

```
工程已有 Kafka 和 NATS 两套消息系统：

Kafka（持久化，高吞吐）：
  适合：余额变动事件流，用于下游统计/风控/报表
  Topic：wallet.balance.changed
  Key：userId（保证同一用户的事件有序）
  消息体：{userId, currencyCode, walletType, txType, amount, balanceBefore, balanceAfter, orderNo, timestamp}

NATS（低延迟，实时推送）：
  适合：实时通知客户端余额变化
  Subject：wallet.notify.{userId}

当前阶段建议：
  先不引入 Kafka/NATS 事件。审计日志用 ser-blog oneway RPC 已够。
  等功能跑通后，如果有统计/风控/推送需求，再加事件层。
  原因：事件驱动增加了系统复杂度（消息丢失、重复消费、顺序保证），
  在核心功能稳定之前引入是过度设计。
```

> 依据：工程中 Kafka Producer 已配置 `Idempotent: true`（幂等生产者），未来引入时数据可靠性有保障

## 十、定时任务架构

### 10.1 需要的定时任务

| 任务 | 频率 | 职责 | 实现方式 |
|------|------|------|---------|
| 汇率同步 | 每 5-10 分钟 | 从三方 API 拉取汇率，偏差超阈值时更新 | cron |
| 充值订单超时关闭 | 每 5 分钟 | 扫描超过 30 分钟仍为"待支付"的订单，改为"已关闭" | cron |
| 对账检查 | 每天凌晨 | 校验 wallet.balance 和 wallet_transaction 的一致性 | cron / xxljob |

### 10.2 汇率同步任务设计

```
SyncRealtimeRate：
  1. 读取所有启用的币种列表
  2. 对每个币种，并发请求三方汇率 API
  3. 容错策略：
     3家成功 → 取平均
     2家成功 → 取2家平均
     1家成功 → 取该1家
     全失败 → 跳过本轮
  4. 计算偏差值
  5. 偏差 ≥ 阈值 → 更新四个汇率 + 写日志 + 更新 Redis 缓存
  6. 偏差 < 阈值 → 只更新实时汇率

三方 API 适配器接口：
  type RateProvider interface {
      GetRate(ctx, baseCurrency, targetCurrency string) (rate int64, err error)
  }

  具体实现：CoinGeckoProvider, ExchangeRateAPIProvider, ...
  通过接口抽象，方便替换和新增提供商
```

### 10.3 对账任务设计

```
DailyReconciliation：
  对每个有余额变动的用户：
    1. 查询 wallet.balance
    2. 查询该用户最新一条 wallet_transaction.balance_after
    3. 两者应该相等，不等则告警
    4. 计算当天所有流水的 amount 之和，验证等于 balance 的变化量
    5. 输出对账报告

  这是金融系统的标配——即使代码完美，也要有独立的对账机制做兜底校验。
```

### 11.2 依赖注入方式

遵循工程既有模式（ser-app 的 NewXxxService 构造器注入）：

```go
// main.go 中组装
func main() {
    // 基础设施
    cfg.InitMySQL()
    cfg.InitRedis()

    // 缓存层
    currencyCache := cache.NewCurrencyCache()

    // 引擎层
    balanceEngine := ser.NewBalanceEngine()

    // 业务层
    currencyService := ser.NewCurrencyService(currencyCache)
    walletService := ser.NewWalletService(currencyCache)
    depositService := ser.NewDepositService(balanceEngine, currencyCache)
    exchangeService := ser.NewExchangeService(balanceEngine, currencyCache)
    withdrawalService := ser.NewWithdrawalService(balanceEngine, currencyCache)
    rpcFinanceService := ser.NewRpcFinanceService(balanceEngine)
    transactionService := ser.NewTransactionService()

    // handler 组装
    srv := walletservice.NewServer(
        &WalletServiceImpl{
            currencyService:    currencyService,
            walletService:      walletService,
            depositService:     depositService,
            exchangeService:    exchangeService,
            withdrawalService:  withdrawalService,
            rpcFinanceService:  rpcFinanceService,
            transactionService: transactionService,
        },
        rpc.InitRpcServerParams(namesp.EtcdWalletService, port)...,
    )
    srv.Run()
}
```

**BalanceEngine 作为共享依赖**注入到多个 Service 中——DepositService/ExchangeService/WithdrawalService/RpcFinanceService 都通过 BalanceEngine 操作余额，保证所有资金操作走同一条路。

### 11.3 兑换场景的架构执行流

用一个具体场景展示各层如何协作：

```
用户点击"确认兑换"
  │
  ▼
gate-font → rpc.Handler(ctx, c, rpc.WalletClient().ExecuteExchange)
  │
  ▼
handler.go: WalletServiceImpl.ExecuteExchange(ctx, req)
  │  纯委托
  ▼
ser/exchange_service.go: ExchangeService.Execute(ctx, req)
  │
  ├→ 1. 校验汇率快照（cache/currency_cache → Redis GET snapshot）
  │     快照过期 → return BizErr(ExchangeRateExpired)
  │
  ├→ 2. 校验余额
  │     rep/wallet_repo → DB SELECT balance WHERE user+currency+type
  │     不足 → return BizErr(BalanceInsufficient)
  │
  ├→ 3. 事务执行（BalanceEngine 编排多步操作）
  │     rep/wallet_repo.WithTransaction(func(tx) {
  │         balanceEngine.DeductInTx(tx, 法币中心钱包, 兑换金额)
  │         balanceEngine.AddInTx(tx, BSB中心钱包, 兑换获得)
  │         balanceEngine.AddInTx(tx, BSB奖励钱包, 赠送部分)
  │     })
  │     任何一步失败 → 全部回滚
  │
  ├→ 4. 删除已使用的汇率快照（cache → Redis DEL snapshot）
  │
  ├→ 5. 审计日志（domain/blog_service → oneway RPC ser-blog）
  │
  └→ 6. 返回成功 + 新余额
```

## 十二、可观测性架构

### 12.1 链路追踪

```
工程已有完整的链路追踪体系（tracer.go + TTHeader）：

用户请求 → gate-font 生成 TraceID → TTHeader 传播 → ser-wallet → 日志自动关联

每条日志都带 TraceID，通过 TraceID 可以串联一次请求经过的所有服务和操作。

ser-wallet 需要做的：
  - 所有日志用 lg.Log().CtxInfof(ctx, ...) ← 自动附加 TraceID
  - 关键操作记录详细日志：余额变更前后、订单状态流转、RPC 调用结果
  - 异常情况用 CtxErrorf 记录完整上下文
```

### 12.2 关键监控指标

```
业务指标（日志 + 对账）：
  - 余额变更成功/失败次数（按 tx_type 分类）
  - 充值/提现订单数量和金额（按币种、按状态）
  - 兑换成功率
  - 幂等命中率（重复请求被过滤的比例）
  - 乐观锁冲突率（version 不匹配的次数）
  - 对账差异（应为 0）

系统指标（工程已有的访问日志中间件）：
  - RPC 方法耗时
  - 数据库查询耗时
  - Redis 缓存命中率
```

## 十三、容错与降级

### 13.1 外部依赖降级策略

| 依赖 | 故障影响 | 降级策略 |
|------|---------|---------|
| 财务 RPC (GetDepositMethods) | 充值页无法渲染 | 返回空列表 + 提示"暂时无法充值" |
| 财务 RPC (GetWithdrawalMethods) | 提现页无法渲染 | 返回空列表 + 提示"暂时无法提现" |
| KYC RPC (KycQuery) | 无法判断认证状态 | 按"未认证"处理（仅允许USDT提现） |
| ser-blog (AddActLog) | 审计日志丢失 | oneway 不阻塞业务，日志本地也打一份 |
| 三方汇率 API | 汇率无法更新 | 使用上次的平台汇率，本轮跳过 |
| Redis | 缓存不可用 | 回退到数据库直查（币种列表/汇率） |

### 13.2 核心操作不降级

```
以下操作必须成功，不能降级：
  - 余额变更（BalanceEngine）→ 数据库事务，只依赖 TiDB
  - 订单创建 → 数据库操作，只依赖 TiDB
  - 幂等校验 → 数据库唯一键，只依赖 TiDB

设计原则：核心资金操作只依赖数据库，不依赖 Redis、不依赖 Kafka、
不依赖任何外部服务。数据库挂了系统不可用——这是可接受的（也没别的选择）。
Redis/Kafka 挂了只影响缓存和异步通知，资金操作不受影响。
```

## 十四、架构决策记录

### 14.1 已确定的决策

| 编号 | 决策 | 理由 | 备选方案 | 否决理由 |
|------|------|------|---------|---------|
| AD-1 | 币种配置和钱包在同一个服务 | 调用频率高、数据耦合紧、一个人开发 | 拆两个服务 | 增加网络开销和运维复杂度 |
| AD-2 | 金额用 bigint 存最小单位 | 无精度丢失、性能好、业界标准 | decimal | Go decimal 库性能差、TiDB decimal 索引效率低 |
| AD-3 | 余额操作统一入口（BalanceEngine） | 对账一致性、并发安全、代码复用 | 各接口自己管余额 | 流水遗漏风险、并发控制不一致 |
| AD-4 | 数据库行锁 + 乐观锁 | 锁和数据在同一事务，天然原子 | Redis 分布式锁 | 锁和事务分离，额外一致性成本 |
| AD-5 | 流水记录 balance_before/after | 对账必需，审计追溯 | 只记 amount | 无法独立验证余额正确性 |
| AD-6 | 跨服务用本地事务+幂等重试 | 简单可靠、业界主流 | TCC/Saga 分布式事务 | 过度设计，引入协调器复杂度 |
| AD-7 | 余额不缓存 | 强一致性要求、变动频繁、索引够快 | Redis 缓存余额 | 缓存一致性问题大于性能收益 |
| AD-8 | 币种/汇率走 Redis 缓存 | 数据量小、变动低频、读取高频 | 每次查 DB | 不必要的 DB 压力 |
| AD-9 | 当前不引入 Kafka/NATS 事件 | 核心功能稳定优先 | 一开始就事件驱动 | 增加复杂度，分散精力 |
| AD-10 | 汇率合并存在 currency_config 表 | 一对一关系、数据量极小 | 拆 exchange_rate 表 | JOIN 增加复杂度，无收益 |

### 14.2 架构约束

```
1. 余额变更必须且只能通过 BalanceEngine —— 禁止任何代码直接 UPDATE wallet 表
2. 每次余额变更必须同事务写流水 —— 禁止余额变了但没有流水
3. 流水必须记录 balance_before 和 balance_after —— 禁止只记 amount
4. C端接口的 userId 必须从 ctx 获取 —— 禁止从请求参数取
5. RPC写操作必须幂等 —— 禁止不检查 order_no 就直接执行
6. 敏感字段必须加密存储 —— 禁止明文存卡号
7. 所有写操作必须记审计日志 —— 通过 ser-blog oneway
```

## 十五、架构总结

```
一句话：在现有 CloudWeGo + TiDB + Redis 技术栈上，
用"模块化单体 + 余额变更引擎 + 行锁乐观锁 + 幂等三层防线 + 本地事务"
构建一个安全、高性能、可维护的多币种钱包系统。

不引入：分布式事务、分库分表、事件溯源、CQRS、Redis分布式锁
原因：当前规模不需要，引入只增加复杂度

核心设计：
  BalanceEngine = 所有钱经过的唯一通道
  行锁+乐观锁+幂等键 = 三层并发防线
  balance_before/after = 对账生命线
  本地事务+幂等重试 = 跨服务一致性

技术选型对照：

  需求              方案                    业界参考
  ─────────        ─────────              ─────────
  余额变更原子性    DB事务(行锁+乐观锁)     支付宝/微信支付
  跨服务一致性      本地事务+幂等重试        Stripe/支付宝
  并发控制         行锁+乐观锁             支付宝(account+version)
  金额存储         bigint最小单位           Stripe/支付宝/微信支付
  幂等性           应用层+唯一键+状态机      Stripe idempotency key
  对账             before/after流水        所有金融系统标配
  服务拆分         模块化单体               Shopify Balance(模块化单体)
  缓存策略         只缓存配置不缓存余额      业界共识
```
