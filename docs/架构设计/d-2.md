### 1.2 设计原则

| 原则 | 具体含义 | 在本架构中的体现 |
|------|---------|----------------|
| **安全第一** | 资金操作零容错，宁可慢不可错 | 悲观锁、同事务、幂等性、流水审计 |
| **业界主流** | 不造轮子，用成熟方案 | 余额+流水模型、冻结三阶段、最小单位存储 |
| **契合现有工程** | 不破坏现有架构，用现有基础设施 | 遵循 Kitex/Hertz/GORM/ETCD/Redis 全家桶 |
| **不过度设计** | 不做当前不需要的东西 | 不做分库分表、不做CQRS、不做Event Sourcing |
| **高性能** | 在安全前提下追求性能 | 热数据缓存、进程内调用、批量操作 |
| **高并发** | 核心操作在并发下正确 | 行级锁、Redis分布式锁、幂等去重 |

## 2. 核心架构决策与依据

### 决策1：单服务多子模块（不拆分微服务）

```
决策：钱包 + 币种配置 = 一个 ser-wallet 服务
否决：拆为 ser-wallet + ser-currency 两个服务
```

**依据**：

| 维度 | 单服务 | 拆分服务 | 结论 |
|------|--------|---------|------|
| 开发效率 | 一人负责全部，无协调成本 | 需要维护两套 IDL、两套部署 | 单服务胜 |
| 调用性能 | 币种/汇率查询是进程内函数调用，0 延迟 | 每次查汇率走 RPC（3s 超时、网络开销） | 单服务胜 |
| 事务边界 | 兑换(扣法币+查汇率+加BSB)可在一个事务中 | 必须先 RPC 查汇率再本地事务，汇率可能变 | 单服务胜 |
| 数据一致性 | 同数据库，强一致 | 跨库，最终一致 | 单服务胜 |
| 故障隔离 | 汇率定时任务异常可能影响钱包 | 隔离好 | 拆分胜 |
| 扩容独立性 | 整体扩容 | 各自扩容 | 拆分胜 |

**结论**：当前阶段（一人开发、用户量未知），单服务的优势远大于拆分。故障隔离通过代码层面的 goroutine 隔离+panic recover 解决。未来用户量激增需要独立扩容时，因为子模块之间已经通过 Service 层接口解耦，可以低成本拆分。

**工程验证**：现有 ser-app 服务包含 banner、category、contentStrategy 等多个子模块，一个服务多子模块是已有惯例。

### 决策2：余额+流水模型（不用复式记账）

```
决策：简单余额字段 + 资金流水表
否决：复式记账（Double-Entry Bookkeeping）
```

**依据**：

| 维度 | 余额+流水 | 复式记账 |
|------|----------|---------|
| 复杂度 | 简单直观，一张余额表一张流水表 | 需要科目体系、借贷平衡、多张账务表 |
| 适用场景 | 用户钱包（查余额、看流水） | 银行/券商（监管合规要求） |
| 性能 | 查余额就读一行 | 查余额需聚合所有流水 |
| 一致性保证 | 余额+流水同事务写入 | 借贷必须平衡，约束更强 |
| 审计能力 | 流水表提供完整审计轨迹 | 天然审计能力 |
| 当前需求 | 满足 | 过度 |

**结论**：本项目是游戏/社交平台的用户钱包，不是银行。余额+流水模型是互联网行业钱包的标准做法（支付宝/微信钱包内部也是余额+流水，只在财务对账层面做复式处理）。流水表的 before_balance/after_balance 字段提供了充分的审计能力。

**关键约束**：余额变更与流水写入必须在**同一数据库事务**中。这是余额+流水模型的安全基石。

### 决策3：最小单位整数存储（不用 DECIMAL）

```
决策：金额用 int64 存储，以各币种最小精度单位为基准
否决：用 DECIMAL(20,8) 存储原始金额
```

**依据**：

| 维度 | int64 最小单位 | DECIMAL(20,8) |
|------|--------------|---------------|
| 计算精度 | 整数运算，零精度丢失 | 数据库内精确，但 Go 浮点转换可能丢失 |
| 运算性能 | CPU 原生整数运算，最快 | 需要 decimal 库或数据库计算 |
| 代码简洁 | Go 原生 int64，无需引入第三方库 | 需要 shopspring/decimal 或字符串来回转 |
| 存储空间 | 8字节 | 9-16字节 |
| 溢出风险 | int64 最大 9.2×10^18，以最小单位算足够 | 无溢出风险 |
| 行业实践 | 支付宝/微信/Stripe/Square 均用最小单位整数 | 银行/金融数据库用 DECIMAL |

**工程验证**：现有工程的金额字段（ser-item 的 BasePrice）使用 int32。钱包用 int64 是同一思路的升级版。

**最小单位定义**：

| 币种 | 展示精度 | 最小单位 | 换算 | 示例 |
|------|---------|---------|------|------|
| VND | 0位小数 | 1 VND | 1:1 | 5,000,000 VND → 存 5000000 |
| IDR | 0位小数 | 1 IDR | 1:1 | 500,000 IDR → 存 500000 |
| THB | 2位小数 | 0.01 THB (satang) | ×100 | 1,500.50 THB → 存 150050 |
| BSB | 2位小数 | 0.01 BSB | ×100 | 200.00 BSB → 存 20000 |
| USDT | 8位小数 | 0.00000001 USDT | ×10^8 | 1.50000000 USDT → 存 150000000 |

**精度系数（precision_factor）存在币种配置表中**，由币种的 decimal_digits 字段推导：`precision_factor = 10^decimal_digits`

**int64 溢出分析**：
- int64 最大值 = 9,223,372,036,854,775,807
- USDT（8位小数）：最大可表示 92,233,720,368.54775807 USDT ≈ 922亿 USDT → 远超需求
- VND（0位小数）：最大可表示 9,223,372,036,854,775,807 VND ≈ 922京 VND → 远超需求

### 决策4：悲观锁（不用乐观锁）

```
决策：余额更新使用 SELECT ... FOR UPDATE 行级悲观锁
否决：乐观锁（version 字段 + CAS）
```

**依据**：

| 维度 | 悲观锁 (FOR UPDATE) | 乐观锁 (version + CAS) |
|------|---------------------|----------------------|
| 写冲突场景 | 一次加锁即成功，无需重试 | 冲突时需重试，高并发下大量重试 |
| 资金安全 | 锁内操作原子性保证，不可能超扣 | 重试逻辑复杂，极端情况可能丢失更新 |
| 实现复杂度 | 一行 SQL 加锁，简单直接 | 需要重试循环、退避策略、重试上限 |
| 性能 | 锁持有时间短（<10ms），可接受 | 无锁等待，但重试消耗可能更大 |
| 死锁风险 | 有，但可通过固定加锁顺序避免 | 无死锁 |
| 行业实践 | 支付宝/微信余额操作均用悲观锁 | 高读低写场景（库存秒杀的读） |

**结论**：资金操作是典型的**写密集、冲突概率高、零容错**场景。悲观锁是业界资金系统的标准选择。乐观锁的重试机制在资金场景中是不必要的复杂度，且重试期间余额状态不确定，增加风险。

**TiDB 支持验证**：TiDB 兼容 MySQL，完全支持 `SELECT ... FOR UPDATE` 行级锁（悲观事务模式，TiDB 4.0+ 默认启用）。

### 决策5：冻结三阶段（不用直接扣款）

```
决策：提现采用 冻结 → 确认扣款/解冻 三阶段
否决：直接扣款 + 失败回补
```

**依据**：这是业界提现/出款场景的标准模型（支付宝提现到银行卡、微信提现均采用此模式）。直接扣款后出款失败需要"加回"操作，引入额外风险（加回失败怎么办？加回多了怎么办？）。冻结模型天然规避了这些问题：余额始终正确，只是可用/冻结状态在变。

### 4.2 核心调用流程（以 CreditWallet 为例）

```
财务 RPC 请求
  │
  ▼
handler.go :: CreditWallet(ctx, req)
  │  // 纯透传，不做逻辑
  ▼
ser/wallet_ser.go :: CreditWallet(ctx, req)
  │
  ├── 1. 参数校验（user_id > 0, amount > 0, currency 非空）
  │
  ├── 2. 幂等检查
  │     查 wallet_flow 表：是否已存在 (ref_order_no, biz_type) 记录
  │     已存在 → 直接返回成功（幂等）
  │
  ├── 3. 数据库事务 ──────────────────────────────────────┐
  │     │                                                 │
  │     ├── SELECT * FROM wallet                          │
  │     │   WHERE user_id=? AND currency_code=?           │
  │     │   AND wallet_type=? FOR UPDATE                  │
  │     │   // 行级锁，防并发                              │
  │     │                                                 │
  │     ├── wallet.balance += amount                      │
  │     │   UPDATE wallet SET balance=? WHERE id=?        │
  │     │                                                 │
  │     ├── INSERT INTO wallet_flow (                     │
  │     │     user_id, currency_code, wallet_type,        │
  │     │     direction=IN, biz_type, amount,             │
  │     │     before_balance, after_balance,              │
  │     │     ref_order_no, idempotent_key               │
  │     │   )                                             │
  │     │                                                 │
  │     └── COMMIT ───────────────────────────────────────┘
  │
  ├── 4. 清除缓存
  │     DEL wallet:bal:{uid}:{currency}
  │
  ├── 5. 审计日志（oneway 异步）
  │     ser-blog.AddActLog(...)
  │
  └── 6. 返回响应
```

```
│  withdraw_ser（提现）                                     │
│    创建提现订单时:                                         │
│    wallet_ser.FreezeBalanceInTx(tx, ...) ← 同事务内调用    │
│                                                          │
│  exchange_ser（兑换）                                     │
│    执行兑换时:                                             │
│    wallet_ser.DebitWalletInTx(tx, ...)   ← 同事务扣法币    │
│    wallet_ser.CreditWalletInTx(tx, ...)  ← 同事务加BSB    │
│                                                          │
│  wallet_ser（钱包核心）                                    │
│    对外 RPC 入口:  CreditWallet(ctx, req)  ← 独立事务      │
│    对内事务入口:  CreditWalletInTx(tx, ...) ← 嵌入调用方事务│
└──────────────────────────────────────────────────────────┘
```

**核心设计：资金操作原语的双入口模式**

```go
// 对外 RPC 入口 — 独立开启事务
func (s *WalletService) CreditWallet(ctx context.Context, req *CreditWalletReq) error {
    // 幂等检查（事务外）
    // ...
    return query.Q.Transaction(func(tx *query.Query) error {
        return s.CreditWalletInTx(ctx, tx, params)
    })
}

// 对内事务入口 — 嵌入调用方的事务
func (s *WalletService) CreditWalletInTx(ctx context.Context, tx *query.Query, params CreditParams) error {
    // SELECT FOR UPDATE
    // UPDATE balance
    // INSERT flow
    return nil
}
```

**为什么需要双入口**：
- **RPC 调用**（财务→钱包上账）：独立事务，因为调用方是跨服务的
- **内部调用**（充值回调上账、兑换扣加款）：嵌入调用方事务，保证原子性

这是资金系统的**业界标准做法**：同一个原子操作，对外是独立事务 RPC，对内是事务内函数调用。

**工程验证**：参照 ser-app 的 BannerRepo.ShiftAndCreate()，已有在事务内执行多步操作的模式。

## 6. 资金安全架构

### 6.1 资金操作的事务边界

```
┌─────────────────────────────────────────────────────────┐
│                    事务边界规则                            │
│                                                          │
│  规则1：余额变更 + 流水写入 = 必须同事务                    │
│                                                          │
│  规则2：冻结 + 订单创建 = 必须同事务                        │
│  （提现时，冻结成功但订单创建失败 = 资金悬空）                │
│                                                          │
│  规则3：兑换的扣+加 = 必须同事务                            │
│  （扣了法币但加BSB失败 = 资金丢失）                          │
│                                                          │
│  规则4：RPC 跨服务调用 = 不在同事务中                        │
│  （财务→钱包上账 是独立事务，通过幂等保证最终一致性）           │
└─────────────────────────────────────────────────────────┘
```

### 6.2 事务模板（对应现有工程模式）

```go
// 使用现有工程的 query.Q.Transaction() 模式
func (s *WalletService) DoExchange(ctx context.Context, req *ExchangeReq) error {
    // 1. 事务外：获取汇率（进程内调用币种配置）
    rate, err := s.currencySer.GetExchangeRate(ctx, req.FromCurrency)
    if err != nil {
        return err
    }

    // 2. 事务外：计算金额（纯领域逻辑）
    bsbAmount := domain.CalcExchange(req.FromAmount, rate)
    bonusAmount := domain.CalcBonus(bsbAmount, bonusRate)

    // 3. 事务内：原子操作
    return query.Q.Transaction(func(tx *query.Query) error {
        // 3.1 扣法币（中心钱包）
        if err := s.DebitWalletInTx(ctx, tx, DebitParams{
            UserID: req.UserID,
            Currency: req.FromCurrency,
            WalletType: enum.WalletCenter,
            Amount: req.FromAmount,
            BizType: enum.BizExchange,
            RefOrderNo: exchangeOrderNo,
        }); err != nil {
            return err  // 回滚
        }

        // 3.2 加BSB（中心钱包）
        if err := s.CreditWalletInTx(ctx, tx, CreditParams{
            UserID: req.UserID,
            Currency: "BSB",
            WalletType: enum.WalletCenter,
            Amount: bsbAmount,
            BizType: enum.BizExchange,
            RefOrderNo: exchangeOrderNo,
        }); err != nil {
            return err  // 回滚
        }

        // 3.3 加BSB赠送（奖励钱包，有赠送时）
        if bonusAmount > 0 {
            if err := s.CreditWalletInTx(ctx, tx, CreditParams{
                UserID: req.UserID,
                Currency: "BSB",
                WalletType: enum.WalletReward,
                Amount: bonusAmount,
                BizType: enum.BizBonus,
                RefOrderNo: exchangeOrderNo,
            }); err != nil {
                return err  // 回滚
            }
        }

        // 3.4 创建兑换记录
        return tx.ExchangeRecord.WithContext(ctx).Create(&model.ExchangeRecord{...})
    })
}
```

### 6.3 资金操作的完整性校验

```
运行时校验：
├── before_balance + amount = after_balance（入账）
├── before_balance - amount = after_balance（出账）
├── balance >= 0（任何操作后余额不可为负）
└── frozen_amount >= 0 且 frozen_amount <= balance

可选的定期对账：
├── 每日凌晨：SUM(所有入账流水) - SUM(所有出账流水) = 当前余额
├── 若不等 → 告警（不自动修正，人工介入）
└── 对账频率和策略待评审后确认
```

## 7. 并发控制架构

### 7.1 数据库行级锁

```
场景：同一用户的同一钱包并发更新余额

方案：SELECT ... FOR UPDATE

执行流程：
  事务1：SELECT * FROM wallet WHERE user_id=1 AND currency_code='BSB'
         AND wallet_type=1 FOR UPDATE;   ← 加行锁
  事务2：SELECT * FROM wallet WHERE user_id=1 AND currency_code='BSB'
         AND wallet_type=1 FOR UPDATE;   ← 等待事务1释放锁
  事务1：UPDATE ... COMMIT;              ← 释放锁
  事务2：获得锁，继续执行

性能影响：
  - 锁持有时间 = 1次UPDATE + 1次INSERT(流水) ≈ 5-15ms
  - 同一用户同一钱包的并发不高（不是秒杀场景）
  - 不同用户的操作完全不冲突（行级锁粒度）
```

### 7.2 死锁预防

```
风险场景：兑换操作涉及两个币种的钱包行

预防策略：固定加锁顺序

规则：永远按 (currency_code ASC, wallet_type ASC) 的顺序加锁

示例：兑换 VND → BSB
  ① 先锁 BSB-CENTER（字母序 BSB < VND）
  ② 再锁 VND-CENTER

所有涉及多行加锁的操作都遵循同一顺序，消除死锁可能。
```

### 7.3 Redis 分布式锁（补充场景）

```
场景：防止同一用户并发创建提现订单

方案：Redis SetNX（已有工程实现：common/pkg/rds/redis_util.go）

流程：
  key = "wallet:lock:withdraw:{user_id}"
  if !rds.SetNX(key, "1", 30*time.Second) {
      return BizErr(ctx, errs.OperationTooFrequent)
  }
  defer rds.Del(key)
  // ... 执行提现逻辑

为什么需要 Redis 锁 + DB 锁两层：
  - Redis 锁：防止同一用户短时间内发起多笔提现（业务层拦截）
  - DB 锁：防止余额并发更新导致数据不一致（数据层保护）
  - Redis 锁是优化（减少无效的 DB 事务），DB 锁是保障（即使 Redis 锁失效也不会超扣）
```

## 8. 缓存架构

### 8.1 缓存层次

```
                    请求
                     │
                ┌────▼────┐
                │ 进程缓存  │  币种列表（sync.Map, 10min TTL）
                │ (可选)    │  减少 Redis 调用
                └────┬────┘
                     │ miss
                ┌────▼────┐
                │  Redis   │  余额、汇率、币种配置
                │          │
                └────┬────┘
                     │ miss
                ┌────▼────┐
                │  TiDB   │  最终数据源
                │          │
                └─────────┘
```

### 8.2 缓存策略

| 数据 | 读写比 | 缓存策略 | TTL | 更新方式 |
|------|--------|---------|-----|---------|
| 用户余额 | 高读中写 | Cache-Aside | 5min | 写操作后 DEL（不是 SET） |
| 币种列表 | 极高读极低写 | Cache-Aside + 本地缓存 | 1h | 配置变更后 DEL |
| 汇率数据 | 高读低写 | Write-Through | 自然过期 | 定时任务直接写 Redis |
| 用户当前币种 | 中读低写 | Cache-Aside | 7d | 切换时 SET |

### 8.3 缓存一致性

**写后删除（而非写后更新）**：

```
为什么删除而不是更新：

  场景：并发 CreditWallet
  线程1：事务提交（余额 1000→1500）
  线程2：事务提交（余额 1500→1800）

  如果用"写后更新"：
    线程1：SET cache = 1500
    线程2：SET cache = 1800（正确）
    但如果执行顺序反了：
    线程2：SET cache = 1800
    线程1：SET cache = 1500 ← 覆盖了线程2的正确值！

  如果用"写后删除"：
    线程1：DEL cache
    线程2：DEL cache
    下次读：从 DB 读到最新值 1800（永远正确）
```

**余额缓存特殊处理**：

```
读余额时：
  1. 先读 Redis
  2. Redis miss → 读 TiDB（不加锁，普通 SELECT）
  3. 写入 Redis（SET + 5min TTL）
  4. 返回

写余额时（CreditWallet/DebitWallet/Freeze/Unfreeze）：
  1. 事务内更新 TiDB
  2. 事务提交后 DEL Redis
  3. 不在事务内操作 Redis（避免事务回滚但缓存已删的问题）

注意：DEL 在事务提交后执行。如果 DEL 失败（Redis 异常），余额缓存会在 TTL 后自动过期。
最差情况：用户看到的余额延迟 5 分钟更新。这在安全性和可用性之间是合理的权衡。
```

## 9. 跨服务通信架构

### 9.1 通信模式选择

```
┌──────────────────────────────────────────────────────────┐
│  同步 RPC（Kitex）— 需要实时响应的场景                      │

│                                                          │

├──────────────────────────────────────────────────────────┤
│  暂不使用 NATS/Kafka — 当前阶段不需要                       │
│                                                          │
│  理由：                                                    │
│  1. 资金操作需要同步确认，异步消息增加复杂度无收益             │
│  2. 当前没有"高吞吐异步处理"的场景                           │
│  3. 审计日志用 oneway RPC 已足够                            │
│  4. 后续如有高吞吐场景（如游戏投注结算）再引入                 │
│                                                          │
│  工程验证：NATS JetStream 和 Kafka 在 common/pkg/ 中已有    │
│  完整实现，需要时可随时引入，不是技术障碍                       │
```

### 9.2 RPC 调用容错

```
所有 RPC 调用遵循：

超时：3秒（现有工程统一配置，common/rpc/init.go）
重试：Kitex 默认不重试（资金操作不应该框架层自动重试）
降级策略：

  钱包 → 币种配置（进程内调用）：
    无 RPC 开销，无需额外容错
    缓存 miss 时直接查 DB

  钱包 → 财务（获取充值/提现方式）：
    失败 → 返回错误给前端"服务暂时不可用"
    可选优化：缓存上次成功的方式列表（TTL 5min），财务不可用时用缓存

  钱包 → ser-kyc（KYC状态）：
    失败 → 可降级为"允许 USDT 提现"（最安全的默认行为）

  财务 → 钱包（CreditWallet等）：
    失败 → 财务侧重试（财务负责保证最终一致性）
    钱包侧幂等保证重复调用安全

  钱包 → ser-blog（审计日志）：
    oneway 异步，失败仅记录本地日志，不影响主流程
```

### 9.3 跨服务数据一致性

```
场景：充值成功上账

方案A（简单方案，推荐）：
  财务收到回调 → 同步 RPC 调钱包 CreditWallet → 成功则更新订单状态
  失败 → 财务设定重试（定时任务扫描"已支付未上账"订单，重新调 CreditWallet）
  钱包幂等保证 → 重复调用无副作用

方案B（如果需要更强保证）：
  引入本地消息表（outbox pattern）
  财务写"上账消息"到本地表 → 定时任务扫描发送 → 钱包幂等接收
  但当前阶段过度设计，不推荐

选择方案A的理由：
  1. 简单，实现成本低
  2. 同步 RPC + 幂等重试 在绝大多数互联网资金系统中已足够
  3. 支付宝/微信等也是类似模式（回调+重试+幂等）
```

### 10.2 汇率计算的精度处理

```
汇率存储：统一用 int64，乘以 10^8 转整数

示例：1 USDT = 25,380.50 VND
  存储值 = 25380.50 × 10^8 = 2,538,050,000,000

汇率计算（全部在整数域）：

  入款汇率 = real_rate × (100 + rate_float_pct) / 100
  出款汇率 = real_rate × (100 - rate_float_pct) / 100

  注：rate_float_pct 也是 ×100 存储
  如 5.00% 存为 500，计算时：real_rate × (10000 + 500) / 10000

兑换金额计算：

  用户输入 5,000,000 VND 兑换 BSB
  法币金额（最小单位）= 5000000（VND 无小数，精度系数1）
  汇率（×10^8）= 2,538,050,000,000（1 USDT = 25,380.50 VND）
  BSB 锚定：10 BSB = 1 USDT

  计算过程（整数域）：
  usdt_amount = vnd_amount × 10^8 / vnd_rate  // 先算USDT
  bsb_amount = usdt_amount × 10                // 再乘10得BSB
  // 最后根据 BSB 的 precision_factor 转为最小单位
```

## 11. 幂等性架构

### 11.1 幂等策略总览

```
┌─────────────────────────────────────────────────────────┐
│                   幂等性保证层次                           │
│                                                          │
│  第一层：业务语义幂等（idempotent_key 唯一约束）            │
│  ┌─────────────────────────────────────────────────┐    │
│  │ wallet_flow 表：UNIQUE(idempotent_key, delete_at)│    │
│  │                                                  │    │
│  │ idempotent_key 格式：{biz_type}:{ref_order_no}   │    │
│  │   :{wallet_type}                                 │    │
│  │                                                  │    │
│  │ 示例：                                            │    │
│  │ "1:R202602261234:1"  充值:订单号:中心钱包          │    │
│  │ "5:A202602261234:1"  人工加款:订单号:中心钱包       │    │
│  │ "5:A202602261234:2"  人工加款:订单号:奖励钱包       │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  第二层：状态机幂等（订单/冻结状态检查）                     │
│  ┌─────────────────────────────────────────────────┐    │
│  │ 充值回调：订单状态≠待支付 → 直接返回成功             │    │
│  │ 解冻操作：冻结状态=已解冻 → 直接返回成功             │    │
│  │ 确认扣款：冻结状态=已确认 → 直接返回成功             │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  执行顺序：先查状态（事务外）→ 再查唯一约束（事务内）        │
│  状态检查是"快速返回"的优化，唯一约束是"最终防线"            │
└─────────────────────────────────────────────────────────┘
```

### 11.2 幂等实现模板

```go
func (s *WalletService) CreditWallet(ctx context.Context, req *CreditWalletReq) error {
    // 第一层：状态机快速检查（事务外，减少锁竞争）
    idempotentKey := fmt.Sprintf("%d:%s:%d", req.BizType, req.RefOrderNo, req.WalletType)
    exists, _ := walletFlowr.Exists(ctx,
        query.Q.WalletFlow.IdempotentKey.Eq(idempotentKey),
        query.Q.WalletFlow.DeleteAt.Eq(0),
    )
    if exists {
        return nil // 幂等：已处理过，直接返回成功
    }

    // 第二层：事务内操作（UNIQUE 约束是最终防线）
    return query.Q.Transaction(func(tx *query.Query) error {
        // ... SELECT FOR UPDATE + UPDATE + INSERT
        // 如果 INSERT wallet_flow 因 UNIQUE 冲突失败 → 说明并发重复 → 也返回成功
    })
}
```

## 12. 错误处理与降级架构

### 12.1 错误分类

```
┌─────────────────────────────────────────────────────────┐
│                       错误分类                            │
│                                                          │
│  业务错误（可预期，返回明确错误码给前端）：                   │
│  ├── 余额不足 → 6022001                                  │
│  ├── KYC 未认证 → 6022200                                │
│  ├── 提现次数超限 → 6022201                               │
│  ├── 重复转账拦截 → 6022100                               │
│  └── 币种已禁用 → 6022600                                │
│                                                          │
│  系统错误（不可预期，记录日志 + 返回通用错误给前端）：         │
│  ├── 数据库超时/连接失败 → 通用系统错误                     │
│  ├── Redis 不可用 → 降级走 DB                             │
│  ├── 三方汇率 API 超时 → 保持旧汇率                       │
│  └── RPC 调用超时 → 具体场景具体降级                       │
│                                                          │
│  资金安全错误（绝不能静默处理）：                            │
│  ├── 余额计算为负 → 回滚事务 + 告警                       │
│  ├── 流水 before/after 不匹配 → 回滚事务 + 告警            │
│  └── 冻结明细找不到 → 拒绝操作 + 告警                      │
└─────────────────────────────────────────────────────────┘
```

### 12.2 降级策略

| 故障场景 | 降级方案 | 影响 |
|----------|---------|------|
| Redis 不可用 | 跳过缓存直接查 DB | 性能下降，功能不受影响 |
| 币种缓存过期 | 查 DB 重建缓存 | 一次慢查询 |
| 三方汇率 API 全部不可用 | 保持旧汇率，记录告警 | 汇率不更新但业务可用 |
| 财务服务不可用 | 充值/提现返回"服务暂忙" | 充值提现暂停，其他功能正常 |
| ser-kyc 不可用 | 提现降级为仅允许 USDT | 部分提现方式不可用 |
| ser-blog 不可用 | 审计日志缺失，记录本地日志 | 不影响业务 |

## 14. 安全架构

### 14.1 资金安全

| 安全措施 | 实现方式 | 防御目标 |
|----------|---------|---------|
| 行级锁 | SELECT FOR UPDATE | 防并发超扣 |
| 幂等键唯一约束 | wallet_flow.UNIQUE(idempotent_key) | 防重复上账/扣款 |
| 流水审计 | before_balance/after_balance 记录 | 可追溯、可审计 |
| 余额非负校验 | 代码层 + 数据库 CHECK（如TiDB支持） | 防余额为负 |
| 冻结三阶段 | freeze→confirm/unfreeze | 防提现过程中资金不一致 |
| 充值回调验签 | 三方通道签名验证 | 防伪造回调 |
| 操作审计日志 | ser-blog 异步上报 | 所有写操作可追溯 |

### 14.2 接口安全

```
C 端接口安全（gate-font）：
  ├── IP 黑名单检查（BlackCheck 中间件）
  ├── Token 认证（FontAuthMiddleware）
  ├── 用户身份通过 tracer.GetUserId(ctx) 从 Token 解析，不可伪造
  └── 所有金额、币种参数服务端校验（不信任前端传值）

B 端接口安全（gate-back）：
  ├── IP 白名单检查（WhiteCheck 中间件）
  ├── Token 认证 + RBAC 鉴权（BackAuthMiddleware）
  └── 操作日志全记录

RPC 接口安全：
  ├── 内网 RPC，不暴露到公网
  ├── 参数校验（amount > 0, user_id > 0, currency 合法）
  └── 幂等性保证
```

### 14.3 数据安全

```
敏感数据：
  ├── 提现账户信息（银行卡号、持有人姓名）
  │     → 存储时考虑加密（参照现有 user_kyc 表的加密模式：utils.EncryptedString）
  └── 用户余额
        → 不在日志中打印具体余额（或脱敏处理）
```

**工程验证**：现有工程在 `common/pkg/db/cmd/generate.go` 中已有加密字段支持（`encryptedFields` 映射），ser-wallet 可复用此机制。

## 15. 架构约束与边界

### 15.1 当前架构不做的事（不过度设计）

| 不做的事 | 理由 |
|----------|------|
| 分库分表 | 当前用户量未知，TiDB 本身支持水平扩展，不需要应用层分片 |
| CQRS（读写分离架构） | 一个服务内的读写量不大，标准读写同库即可 |
| Event Sourcing | 余额+流水模型已满足审计需求，事件溯源是过度设计 |
| Saga 分布式事务 | 跨服务一致性通过幂等+重试解决，Saga 引入复杂度不值得 |
| 消息队列（NATS/Kafka） | 当前无高吞吐异步场景，同步 RPC 足够 |
| 多级缓存（L1+L2） | 进程缓存只用于币种列表这类极低频变更数据，不需要复杂多级缓存 |
| 熔断器 | 当前服务间调用链不深，简单超时+降级即可 |
| 读写分离（TiDB 从库） | 初期不需要，TiDB 支持但增加运维复杂度 |

### 15.2 未来可扩展的方向（预留但不实现）

| 扩展方向 | 何时需要 | 预留设计 |
|----------|---------|---------|
| 币种配置拆分为独立服务 | 多团队开发、需独立扩容时 | Service 层接口解耦，拆分时改 RPC 调用即可 |
| 引入消息队列 | 游戏投注结算等高吞吐场景 | NATS/Kafka 已有工程实现，接入成本低 |
| 对账系统 | 线上运营需要定期对账 | wallet_flow 的 before/after_balance 天然支持 |
| 限流 | 高并发压力超过 DB 承受能力时 | Redis 令牌桶 或 网关层限流 |
| 国际化错误消息 | 多语言上线时 | 错误码 + ser-i18n 的 Redis 哈希表，框架已有 |

### 15.3 架构检查清单

```
✅ 资金安全
  ├── 余额+流水同事务 → 保证一致性
  ├── 行级锁 → 防并发超扣
```
