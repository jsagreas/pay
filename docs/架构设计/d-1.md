# 钱包体系架构设计方案

> 范围：ser-wallet（钱包+币种配置）+ 与 ser-finance（财务）的联动架构
> 约束：基于现有工程体系（Hertz+Kitex+GORM+TiDB+Redis+ETCD+NATS），不引入新技术栈
> 原则：安全第一、高并发支撑、高性能、不过度设计、业界主流方案

---

### 1.1 为什么这样分层

| 层 | 职责 | 依据 |
|---|------|------|
| handler.go | 接收 RPC 请求，委托给 ser 层，不含业务逻辑 | 工程惯例：ser-app/handler.go 每个方法只有一行委托 |
| ser/ | 核心业务逻辑、参数校验、流程编排、调用外部 RPC | 工程惯例：ser-app/internal/ser/ 所有业务逻辑集中于此 |
| rep/ | 数据库 CRUD，使用 GORM Gen 生成的 query DSL | 工程惯例：ser-app/internal/rep/ 纯数据访问 |
| cfg/ | DB/Redis 连接初始化（sync.Once 单例） | 工程惯例：ser-app/internal/cfg/ |
| cache/ | Redis 缓存读写，封装缓存策略 | 工程实例：ser-app/internal/cache/banner.go |

不引入额外的 domain 层或 event 层——现有工程没有这些抽象，保持一致性比引入新模式更重要。

### 1.2 ser-wallet 内部子模块划分

```
ser-wallet/internal/ser/
├── currency_service.go       ← 币种CRUD、状态切换
├── rate_service.go           ← 汇率拉取、阈值判断、自动更新
├── wallet_engine.go          ← 余额引擎（getBalance / modifyBalance）★核心★
├── deposit_service.go        ← 充值流程编排
├── withdraw_service.go       ← 提现流程编排
├── exchange_service.go       ← 兑换流程编排
├── record_service.go         ← 记录查询
└── account_service.go        ← 提现账户管理
```

币种配置和钱包业务在同一进程内，相互调用走**内部方法**，不走 RPC。对外（财务调用币种汇率、财务调用余额操作）走 RPC。

## 二、余额引擎架构（核心中的核心）

### 2.1 为什么需要单独的余额引擎

钱包系统中所有业务（充值、兑换、提现、人工修正、投注扣款、返奖）最终都归结为余额的增减。如果各业务各自操作余额，必然出现不一致。业界标准做法是**抽取一个统一的余额操作层**，所有上层业务通过它操作余额。

### 2.2 引擎设计

```
wallet_engine.go 对外只暴露两个方法：

┌─────────────────────────────────────────────────┐
│              WalletEngine                        │
│                                                  │
│  GetBalance(userId, currencyId, walletType)      │
│    → 读余额，不存在则懒创建                       │
│                                                  │
│  ModifyBalance(tx, params ModifyBalanceParams)   │
│    → 在调用方传入的事务tx内操作余额+写流水         │
│                                                  │
└─────────────────────────────────────────────────┘

ModifyBalanceParams {
    UserID       int64
    CurrencyID   int64
    WalletType   int32
    OpType       int32   // credit / debit / freeze / unfreeze / debit_frozen
    Amount       int64   // 最小单位整数
    RefOrderNo   string  // 关联单号（幂等键）
    TxType       int32   // 流水类型（充值/提现/兑换/冻结/调整...）
    Remark       string
}
```

### 2.3 并发安全：悲观行锁

余额操作必须防止并发超扣。方案：**SELECT ... FOR UPDATE 行锁**。

```
ModifyBalance 内部流程：

  1. 在事务 tx 内执行：
     SELECT available_balance, frozen_balance
     FROM wallet_account
     WHERE user_id=? AND currency_id=? AND wallet_type=?
     FOR UPDATE
     → 对该行加排他锁，其他事务读写该行时阻塞等待

  2. 应用层校验：
     debit:        available >= amount?
     freeze:       available >= amount?
     debit_frozen: frozen >= amount?

  3. 更新余额

  4. 写入 transaction_log（同一事务内）

  5. 事务提交 → 行锁释放
```

**为什么用悲观锁而不是乐观锁**：
- 钱包场景是**高冲突**场景——同一用户的余额可能被充值回调和用户消费同时操作
- 乐观锁（version 字段）在高冲突下重试率高，反而降低性能
- 悲观锁虽然有锁等待，但锁持有时间极短（一次简单的 UPDATE），实际开销小
- 业界主流支付系统（支付宝、微信支付的余额变更层）均采用行锁方案

**现有工程兼容性**：当前工程 MySQL 配置 `SkipDefaultTransaction: true`（common/pkg/db/mysql/mysql.go），意味着普通操作不自动包事务。余额引擎的事务需要**显式开启**，使用 `query.Q.Transaction()` 或 `db.Transaction()`——这和 ser-app/internal/rep/banner.go 中的 `ShiftAndCreate` 事务模式一致。

### 2.4 幂等方案：唯一索引

```
transaction_log 表上建唯一索引：
  UNIQUE INDEX uk_ref_order (ref_order_no, tx_type)

ModifyBalance 执行写入 transaction_log 时：
  → 写入成功 → 正常继续
  → 唯一索引冲突 → 说明该操作已执行过 → 直接返回成功（幂等）
```

**为什么用 DB 唯一索引而不是 Redis SetNX**：
- 唯一索引是事务的一部分，和余额变更在同一事务中，**原子性天然保证**
- 如果用 Redis SetNX 做幂等判断，Redis 和 DB 之间没有事务保证，可能出现：SetNX 成功但 DB 写入失败，导致该操作永远无法重试
- 唯一索引方案零额外依赖，性能等同于一次 INSERT

### 2.5 流水必写：不可绕过的约束

```
ModifyBalance 内部强制写 transaction_log，
调用方无法绕过。这是架构层面的硬约束：

  余额变更和流水写入在同一个事务中
  → 事务成功 = 余额变了 + 流水写了
  → 事务失败 = 余额没变 + 流水也没写
  → 不可能出现"余额变了但没有流水"的情况
```

### 2.6 金额精度方案

```
所有金额字段用 bigint 存储最小单位：

  VND:  decimal_places=0, multiplier=1      → 存1000 = 1000 VND
  IDR:  decimal_places=0, multiplier=1      → 存1000 = 1000 IDR
  THB:  decimal_places=2, multiplier=100    → 存350  = 3.50 THB
  BSB:  decimal_places=2, multiplier=100    → 存100  = 1.00 BSB
  USDT: decimal_places=8, multiplier=1e8   → 存100000000 = 1.00000000 USDT

每个币种的 multiplier 在 currency 表中配置。
所有内部计算用 int64 整数运算，仅在返回给前端时 ÷ multiplier 还原为展示值。
```

**为什么不用 decimal**：
- Go 标准库没有 decimal 类型，引入三方库增加复杂度
- TiDB/MySQL 的 DECIMAL 在 GORM 中映射为 string 或 float64，类型转换容易出错
- 整数运算性能更高，且天然不存在浮点精度问题
- 业界主流：支付宝/微信支付/Stripe 均用最小单位整数

## 三、数据架构

### 3.1 表结构总览

```
ser-wallet 数据库（单库，与币种配置共用）

核心账务表：
├── wallet_account        用户钱包账户（余额存储）
└── transaction_log       流水记录（只增不改不删）

业务订单表：
├── deposit_order         充值订单
├── withdraw_order        提现订单
├── exchange_record       兑换记录
└── withdraw_account      提现账户

基础配置表：
├── currency              币种配置
└── exchange_rate_log     汇率变更日志
```

### 3.2 关键索引设计

```
wallet_account:
  PRIMARY KEY (id)
  UNIQUE INDEX uk_user_currency_type (user_id, currency_id, wallet_type)
  → 保证每个用户每个币种每种钱包类型只有一行
  → SELECT ... FOR UPDATE 按此索引精确定位行，锁粒度最小

transaction_log:
  PRIMARY KEY (id)
  UNIQUE INDEX uk_ref_order (ref_order_no, tx_type)
  → 幂等键，防止重复操作
  INDEX idx_user_currency_time (user_id, currency_id, created_at DESC)
  → C端记录查询走这个索引

deposit_order:
  PRIMARY KEY (id)
  UNIQUE INDEX uk_order_no (order_no)
  INDEX idx_user_currency_status (user_id, currency_id, status)
  → 重复充值检测：WHERE user_id=? AND currency_id=? AND amount=? AND status IN (...)

withdraw_order:
  PRIMARY KEY (id)
  UNIQUE INDEX uk_order_no (order_no)
  INDEX idx_user_currency_status (user_id, currency_id, status)
  INDEX idx_status (status)
  → 财务按状态批量查询待审核订单

currency:
  PRIMARY KEY (id)
  UNIQUE INDEX uk_code (currency_code, deleted_at)
  → 币种代码唯一
```

### 3.3 读写分离考量

当前阶段**不需要**读写分离：
- TiDB 本身支持水平扩展，单库足够承载初期用户量
- 钱包的核心操作（余额变更）是写密集型，读写分离对写操作无帮助
- 记录查询可通过索引+分页优化

**后续扩展点**：如果流水表数据量增长到千万级，可以按 user_id 或 time 做分区表，TiDB 原生支持。

## 四、缓存架构

### 4.1 缓存策略总则

```
可缓存（低频变更、非资金数据）：
├── 启用币种列表          → Redis String, TTL 5分钟, 变更时主动删除
├── 币种详情（含汇率）     → Redis Hash, TTL 与汇率更新周期对齐
└── 充值档位/提现限额     → Redis Hash, 配置变更时主动删除

不可缓存（资金安全相关）：
├── 用户余额              → 每次必查DB（行锁场景下缓存无意义）
├── 订单状态              → 状态机变更频繁且必须实时
└── 今日已提现金额        → 每次操作都可能变
```

### 4.2 缓存模式

采用**Cache-Aside**（旁路缓存），与现有工程一致（ser-app/internal/cache/banner.go 即此模式）：

```
读取：
  1. 查 Redis → 命中 → 返回
  2. 未命中 → 查 DB → 写入 Redis（带 TTL）→ 返回

写入（配置变更时）：
  1. 写 DB
  2. 删 Redis（不是更新，而是删除，让下次读时重建）

为什么删而不更新：
  → 删除更简单，不会出现"DB写成功但Redis更新失败导致不一致"
  → 最坏情况是下次多查一次DB，不影响正确性
```

### 4.3 币种汇率缓存的特殊处理

```
汇率是对外 RPC 供给的高频读数据（钱包每次充值/兑换/提现都要读），缓存收益大。

方案：
  key: wallet:rate:{currency_code}
  value: JSON { platformRate, depositRate, withdrawRate, updatedAt }
  TTL: 与汇率拉取周期对齐（如拉取间隔2分钟，TTL设3分钟）

  汇率更新时（rate_service 定时任务）：
    1. 写 DB
    2. 主动写 Redis（这里用写而不是删，因为汇率读频率极高，避免瞬间全量回源DB）

  读取时：
    1. 查 Redis → 有 → 返回
    2. 没有 → 查 DB → 写 Redis → 返回
```

## 六、充值链路架构

### 6.1 架构选择：同步建单 + 异步回调

```
充值采用业界标准的"同步建单、异步回调"模式：

  ┌───────┐   同步      ┌──────────┐   同步     ┌──────────┐   同步    ┌──────┐
  │ 前端  │──────────►│ ser-wallet │──────────►│ser-finance│─────────►│ 三方 │
  │       │  建单请求   │  校验+建单 │  匹配通道  │ 通道匹配  │  发起支付 │ 通道 │
  │       │◄──────────│  返回URL   │◄──────────│  返回结果  │◄─────────│      │
  └───────┘   订单号    └──────────┘           └────┬─────┘          └──┬───┘
                             ▲                       │                    │
                             │    异步回调入账         │    异步回调结果     │
                             └───────────────────────┘◄───────────────────┘
```

**为什么不用消息队列做回调**：
- 当前工程 NATS/Kafka 虽已实现但**尚未在任何业务中使用**
- 充值回调是**关键资金链路**，第一版用同步 RPC 更可控、更易排查
- RPC 调用有明确的超时和重试，行为可预期
- 后续如果并发量大导致回调堆积，再考虑引入消息队列做削峰

### 6.2 充值订单状态机

```
                  创建
                   │
                   ▼
              ┌─────────┐
              │ 待支付   │──── 超时未支付 ────► 已超时
              └────┬────┘
                   │ 三方回调
                   ▼
              ┌─────────┐
              │ 支付中   │──── 支付失败 ─────► 已失败
              └────┬────┘
                   │ 支付成功
                   ▼
              ┌─────────┐
              │ 待入账   │
              └────┬────┘
                   │ CreditBalance
                   ▼
              ┌─────────┐
              │ 已完成   │
              └─────────┘

状态流转规则：
  → 只允许向后流转，不允许回退
  → 每次状态变更用 WHERE status = 当前状态 条件更新，防止并发跳跃
```

## 七、提现链路架构

### 7.1 架构选择：两阶段提交（冻结-确认/取消）

```
提现采用业界经典的"冻结→确认/取消"模式（类似 TCC 的简化版）：

阶段一：Try（冻结）
  → 用户申请提现
  → 钱包校验+冻结余额（available → frozen）
  → 创建提现订单
  → 推送给财务

阶段二A：Confirm（扣款）
  → 财务审核通过 + 三方出款成功
  → 财务调钱包 DebitBalance（frozen → 扣减）

阶段二B：Cancel（解冻）
  → 财务审核驳回 或 三方出款失败
  → 财务调钱包 UnfreezeBalance（frozen → available）
```

**为什么这个模式**：
- 提现从申请到出款有**时间跨度**（三级审核），必须冻结防止用户在审核期间把钱花掉
- 冻结-确认/取消 是业界处理这类场景的标准模式
- 不需要引入分布式事务框架（Seata/DTM），当前场景用显式 RPC 调用完全可控

### 7.2 冻结超时兜底

```
异常场景：冻结后财务从未回调（网络故障/财务服务挂掉）
→ 冻结资金悬挂，用户看到余额减少但提现没下文

兜底方案：
  → 定时任务扫描 withdraw_order
  → 条件：status=待审核/审核中 AND created_at < 当前时间-N小时
  → 对超时订单：查询财务侧状态
    → 财务无此单 → 自动解冻
    → 财务有但还在处理 → 跳过
    → 财务审核完但回调失败 → 补偿执行

实现方式：内部 goroutine ticker，每隔 N 分钟扫描一次
N 的具体值待定，初期可设 24 小时（宽松兜底）
```

## 八、兑换架构

```
兑换是服务内部自闭环，架构最简单：

  ExchangePreview：纯计算，不写DB
  ExecuteExchange：一个DB事务搞定

  事务内操作：
    1. 读取源币种余额 (FOR UPDATE)
    2. 读取BSB余额 (FOR UPDATE)
    3. 校验源币种余额充足
    4. 扣减源币种 available
    5. 增加BSB中心钱包 available
    6. 增加BSB奖励钱包 available（赠送部分）
    7. 写 exchange_record
    8. 写 transaction_log × 3条

  注意：步骤1和2分别锁两行，需要保证锁顺序一致（如按 currency_id 升序加锁），
  防止死锁。这是多行锁的标准实践。
```

## 九、与财务模块的联动架构

### 9.1 接口契约优先

```
钱包和财务的联动架构不是技术选型问题，而是接口契约问题。

核心契约：
┌──────────────────────────────────────────────────────────┐
│                   IDL 接口契约                             │
│                                                           │
│  财务 → 钱包（5条）    │    钱包 → 财务（2条）             │
│  CreditBalance         │    GetPaymentMethods             │
│  FreezeBalance         │    MatchDepositChannel           │
│  UnfreezeBalance       │                                  │
│  DebitBalance          │                                  │
│  ManualAdjust          │                                  │
│                        │                                  │
│  全部要求：             │    全部要求：                     │
│  ├─ 幂等（ref_order_no）│    ├─ 超时3秒                    │
│  ├─ 事务内执行          │    └─ 失败有明确错误码             │
│  └─ 返回明确状态码      │                                  │
└──────────────────────────────────────────────────────────┘
```

### 9.2 充值回调的对接方式

```
方案（推荐）：三方 → 财务 → 钱包（RPC）

理由：
  1. 三方回调需要解析各通道的回调格式（签名验证等），这是财务的职责
  2. 钱包只需要接受一个标准化的 CreditBalance 请求
  3. 钱包不需要暴露 HTTP 端点给外部，安全性更好
  4. 如果三方直接回调钱包，钱包就需要了解每个通道的回调协议，职责混乱
```

### 9.3 提现订单的推送方式

```
方案选择需要和财务方确认，有三种可行方案：

方案A：共享表（最简单）
  钱包建单写入 withdraw_order 表，财务直接读这张表
  → 优点：零对接成本
  → 缺点：财务服务直连钱包DB，耦合度高

方案B：RPC通知（推荐）
  钱包建单后 RPC 通知财务有新提现单
  → 优点：解耦，财务可以有自己的数据表
  → 缺点：需要财务暴露一个接收接口

方案C：消息队列（后续考虑）
  钱包建单后发 NATS 消息，财务订阅消费
  → 优点：完全异步解耦
  → 缺点：当前工程没有业务使用 NATS 的先例，引入有风险

第一版推荐方案B，简单可控。
```

## 十、安全架构

### 10.1 资金安全

| 安全措施 | 实现方式 | 保护的风险 |
|---------|---------|-----------|
| **行锁** | SELECT FOR UPDATE | 并发超扣 |
| **事务** | GORM Transaction | 余额与流水不一致 |
| **幂等** | transaction_log.ref_order_no 唯一索引 | 重复入账/重复扣款 |
| **状态机** | WHERE status=当前状态 条件更新 | 状态跳跃/重复操作 |
| **流水必写** | ModifyBalance 内部强制写流水 | 余额变了但无记录 |
| **汇率快照** | 订单表存 rate_snapshot | 汇率变动导致金额不一致 |

### 10.2 接口安全

| 安全措施 | 实现方式 | 依据 |
|---------|---------|------|
| **C端认证** | gate-font FontAuthMiddleware | 已有中间件，Redis Token 校验 |
| **B端认证+鉴权** | gate-back BackAuthMiddleware + CheckUrlAuth | 已有中间件，Token + URL 权限 |
| **IP管控** | BlackCheck(C端) / WhiteCheck(B端) | 已有中间件，调 ser-ip |
| **参数校验** | IDL required + 业务层二次校验 | Thrift required 标签 + 手动校验 |
| **审计日志** | ser-blog.AddActLog (oneway) | 已有组件，所有写操作记录 |
| **链路追踪** | TraceID + 8个标准Header | 已有 tracer 组件 |

### 10.3 敏感数据

```
提现账户信息（银行账号/手机号/USDT地址）属于敏感数据。

方案：使用现有工程的 AES-256 加密方案
  → common/pkg/utils/crypto_util.go 已实现 EncryptedString 类型
  → GORM 的 Scan/Value 钩子自动加解密
  → 数据库存密文，应用层自动透明处理

敏感字段列表：
  withdraw_account.account_no  → 加密
  withdraw_account.holder_name → 加密
```

## 十一、性能架构

### 11.1 热点优化

```
热点1：用户余额查询（高频读）
  → 不缓存，每次读DB
  → 优化手段：uk_user_currency_type 唯一索引，点查 < 1ms
  → TiDB 本身有 SQL 缓存层，热点数据大概率在内存中

热点2：币种/汇率查询（高频读，所有用户共享）
  → Redis 缓存，TTL 与更新周期对齐
  → 一次缓存 Miss 后所有请求都走缓存

热点3：充值重复检测（每次充值前查）
  → 走 idx_user_currency_status 索引，点查 < 1ms
  → 不缓存（状态频繁变化）

热点4：流水分页查询（C端记录页）
  → 走 idx_user_currency_time 索引
  → 标准分页：LIMIT + OFFSET，前端限制 pageSize ≤ 20
```

### 11.2 连接池配置

```
MySQL（TiDB）：
  MaxOpenConns: 20（现有默认值，初期够用）
  MaxIdleConns: 10
  ConnMaxLifetime: 1h

Redis：
  复用现有 common/pkg/rds 的配置
  ReadTimeout: 2s, WriteTimeout: 2s

后续如果并发上来（TPS > 1000），可按需调大 MaxOpenConns。
TiDB 水平扩展能力意味着 DB 不太可能成为瓶颈。
```

### 11.3 不做的优化（不过度设计）

| 方案 | 为什么不做 |
|------|---------|
| 读写分离 | 初期数据量不大，TiDB 自身扩展能力足够 |
| 分库分表 | TiDB 本身是分布式数据库，不需要应用层分片 |
| 本地缓存 (sync.Map) | 余额数据不能缓存，配置数据 Redis 足够 |
| 异步入账（MQ） | 资金链路不能异步，必须同步确认 |
| Event Sourcing | 当前 CRUD 模式 + 流水表已满足审计需求 |
| CQRS | 查询和写入量级相当，不需要分开 |

## 十二、可靠性架构

### 12.1 服务可用性

```
ser-wallet 是有状态的（连接DB和Redis），但本身无状态可水平扩展：
  → ETCD 注册多实例，Kitex 客户端自动负载均衡
  → 任一实例挂掉，其他实例接管请求

关键依赖的容错：
  TiDB 挂了      → 所有操作失败，返回错误码（不能降级，资金不能出错）
  Redis 挂了     → 缓存失效，回源DB（配置查询变慢但不影响正确性）
                 → 分布式锁失效，退化为DB行锁（本来就是主要锁机制）
  ETCD 挂了      → 已连接的RPC客户端继续工作（连接已建立），新客户端无法发现
  ser-kyc 挂了   → 提现KYC校验失败，返回错误码，用户暂时无法提现
  ser-blog 挂了  → 审计日志丢失（oneway不阻塞主流程），需后续补录
  ser-finance 挂了 → 充值方式展示失败、通道匹配失败，用户暂时无法充值
                   → 已在途的提现回调失败，由定时任务兜底补偿
```

### 12.2 数据一致性保障

```
钱包系统数据一致性分级：

强一致（事务保证）：
  ├── 余额变更 + 流水写入：同一DB事务
  ├── 冻结 + 建单：同一DB事务
  └── 订单状态变更：WHERE 条件保证状态机单向

最终一致（补偿保证）：
  ├── 充值回调失败 → 财务侧重试（幂等安全）
  ├── 提现回调失败 → 定时任务扫描补偿
  └── 审计日志 → oneway丢失后可从流水表重建
```

## 十四、架构决策记录（ADR）

### ADR-1：一个服务 vs 拆分多个服务

```
决策：币种配置和钱包放在同一个 ser-wallet 服务内
理由：
  1. 币种配置的主要调用方就是钱包，内部调用比RPC高效
  2. 共享DB事务更简单（兑换时读汇率和扣余额在一个事务内）
  3. 工程先例：ser-item 内含 item + item_publish 两个子模块
  4. 初期团队规模（一个人负责），一个服务更易管理
后续：如果币种配置的调用方增多（N个服务都需要），可考虑独立为 ser-currency
```

### ADR-2：悲观锁 vs 乐观锁

```
决策：余额操作使用 SELECT FOR UPDATE 悲观行锁
理由：
  1. 余额是高冲突资源（同一用户的充值回调和消费可能并发）
  2. 乐观锁在高冲突下重试率高，性能反而差
  3. 行锁持有时间极短（毫秒级），阻塞开销可忽略
  4. 业界标准：支付宝/微信支付的余额层均用悲观锁
```

### ADR-3：幂等方案选择

```
决策：DB唯一索引（transaction_log.ref_order_no）
备选：Redis SetNX
理由：
  1. 唯一索引和余额变更在同一事务中，原子性天然保证
  2. Redis SetNX 与 DB 没有事务关联，可能出现不一致
  3. 零额外依赖，不增加复杂度
```

### ADR-4：同步RPC vs 消息队列

```
决策：第一版用同步RPC对接财务，不引入消息队列
理由：
  1. 当前工程 NATS/Kafka 无业务使用先例，引入风险不可控
  2. 资金链路需要同步确认，异步增加不确定性
  3. RPC 超时和错误处理更直接
  4. 定时任务兜底可覆盖 RPC 调用失败的场景
后续：如果充值回调量大（TPS > 1000），考虑 NATS JetStream 做削峰
```

### ADR-5：金额精度方案

```
决策：bigint 存储最小单位整数
备选：decimal 字段 / 字符串存储
理由：
  1. 整数运算无浮点精度问题
  2. Go 原生 int64 足够大（最大 9.2×10^18，即使越南盾也绰绰有余）
  3. GORM 对 decimal 的映射不够友好（string or float64 转换）
  4. 业界标准：Stripe/支付宝/微信支付均用最小单位整数
```

### ADR-6：缓存 vs 不缓存余额

```
决策：用户余额不缓存，每次读DB
理由：
  1. 余额操作用行锁，缓存反而增加不一致风险
  2. 唯一索引点查 < 1ms，缓存收益极小
  3. 缓存一致性维护成本（每次变更都要同步更新/删除）大于收益
  4. 资金数据"宁可慢一点，不可错一分"
```

## 十六、总结

### 架构核心选择

| 决策点 | 选择 | 业界依据 |
|-------|------|---------|
| 余额并发控制 | SELECT FOR UPDATE 行锁 | 支付宝/微信支付标准 |
| 幂等保障 | DB唯一索引 (ref_order_no) | 与事务原子绑定，最可靠 |
| 金额存储 | bigint 最小单位整数 | Stripe/支付宝/微信支付标准 |
| 提现模式 | 冻结→确认/取消（简化TCC） | 金融系统标准的两阶段模式 |
| 充值模式 | 同步建单+异步回调 | 支付行业标准 |
| 流水保障 | 余额变更和流水写入同一事务 | 会计准则：有借必有贷 |
| 模块通信 | 同步RPC | 资金链路不能异步 |
| 缓存策略 | 配置缓存/余额不缓存 | 资金数据安全优先 |
| 服务划分 | 单服务双子模块 | 匹配团队规模和耦合度 |

### 不做什么

| 不做 | 原因 |
|------|------|
| 读写分离 | TiDB 分布式架构自带扩展 |
| 消息队列 | 第一版保持简单，同步RPC可控 |
| Event Sourcing | 流水表已满足审计，不需要事件溯源 |
| CQRS | 读写量级相当 |
| 分布式事务框架 | 显式RPC+幂等+补偿已足够 |
| 余额缓存 | 安全 > 性能 |
