# ser-wallet 编码前置信息清单 — 缺什么就做不了什么

> 文档定位：系统梳理钱包多币种模块开启编码前，必须提前准备好但目前确实缺失的**业务层面前置信息与资源**。
>
> 不包含什么：数据库连接、Redis/ETCD 配置、工具类、框架组件等工程基础设施——这些已具备，不在本文讨论范围。
>
> 判定标准：如果缺少某项信息，则对应功能的编码**无法启动或无法完成**，不是"做得不好"而是"做不了"。
>

## 2. 第一类：三方服务对接信息

> 这一类信息来自外部服务商或公司商务/运维决策，我们后端无法自行补齐，必须提前拿到。

---

### 2.1 三方汇率 API 接入信息

**需求依据**：币种配置需求文档 4-2 —— "接入 3 家三方汇率 API，取平均值"

**为什么缺了做不了**：币种配置模块的核心能力之一是汇率定时同步（Phase 1），需要向三方 API 发 HTTP 请求拉取汇率数据。没有 API 信息，整个汇率模块无法编码。

**需要的具体信息**：

| 信息项 | 说明 | 状态 |
|--------|------|------|
| API 服务商选型（3 家） | 推荐：CurrencyLayer、ExchangeRate-API、Fixer.io 等 | ❌ 未确定 |
| 每家的 API Key | 注册后获得的访问密钥 | ❌ 未获取 |
| 每家的 API 文档/Endpoint | 请求 URL、请求参数、响应 JSON 格式 | ❌ 未获取 |
| 调用频率/配额限制 | 免费版通常限制 100-1000 次/月，是否需要付费版 | ❌ 未确认 |
| 支持的币种对列表 | 是否支持 VND/IDR/THB 对 USDT 的汇率 | ❌ 未确认 |
| 返回精度 | 小数位数，影响存储精度（DECIMAL(20,8) 是否够用） | ❌ 未确认 |

**影响的功能**：
- B 端汇率查看（实时汇率展示）
- 汇率定时同步任务
- 平台汇率自动更新（偏差值 ≥ 阈值触发）
- 兑换汇率计算
- 充值/提现跨币种换算

**阻塞阶段**：**Phase 1**（币种配置模块）

**退路方案**：如果 API 信息迟迟拿不到，可以先用 Mock 数据模拟汇率（硬编码 3 条固定汇率），汇率同步模块的代码框架先写好，预留适配层接口。但汇率浮动、阈值判断等逻辑无法端到端验证。

### 2.2 USDT 区块链对接信息

**需求依据**：多币种钱包需求文档 5-2 —— "平台展示 USDT 接收地址和二维码，支持 TRC20/ERC20/BEP20 三种链，通过区块链监听确认到账"

**为什么缺了做不了**：USDT 充值不是走传统三方支付回调，而是区块链链上监听。没有链上接收地址和监听服务，USDT 充值功能完全无法实现。

**需要的具体信息**：

| 信息项 | 说明 | 状态 |
|--------|------|------|
| 平台 USDT 收款地址 | TRC20、ERC20、BEP20 各一个（或多个） | ❌ 未提供 |
| 收款地址的私钥管理方案 | 热钱包/冷钱包策略，私钥谁持有 | ❌ 未确定 |
| 区块链监听服务方案 | 自建节点？接入第三方？（如 BlockCypher、Alchemy、Tatum） | ❌ 未确定 |
| 如果接入第三方监听服务的凭证 | API Key、Webhook 回调 URL 配置 | ❌ 未获取 |
| 各链确认数要求 | TRC20（通常 19 确认）、ERC20（通常 12 确认）、BEP20（通常 15 确认） | ❌ 未确认 |
| USDT 最低充值金额 | 链上 gas 费有最低经济性，低于某值充值没有意义 | ❌ 未确认 |
| 到账后入账币种 | 按 1:1 入 USDT 钱包？还是按汇率换算为其他币种？ | ❌ 未确认 |

**影响的功能**：
- C 端 USDT 充值页面（展示地址+二维码）
- USDT 到账入账链路
- 游客 USDT 充值（需求文档 5-2 特别提到）
- USDT 提现出款通道

**阻塞阶段**：**Phase 4**（充值模块）

**退路方案**：USDT 充值与法币充值走完全不同的技术路径。如果区块链信息迟迟未定，可以先实现法币充值链路（走三方支付通道），USDT 充值独立排期。但需注意：如果 USDT 是用户的主要充值方式，延迟实现会严重影响上线。

### 2.3 三方支付通道接口信息

**需求依据**：财务管理需求文档 2-1/2-2 —— 代收通道、代付通道配置；充值/提现流程中跳转三方支付页面

**为什么与钱包模块相关**：虽然三方通道对接由财务模块（ser-finance）负责，但钱包模块需要知道通道的关键信息来**设计接口参数和展示逻辑**。

**需要的具体信息**：

| 信息项 | 说明 | 钱包模块为什么需要 | 状态 |
|--------|------|-------------------|------|
| 各国支持的充值方式列表 | 越南：银行转账/电子钱包；印尼：银行/QRIS；泰国：银行/PromptPay | 钱包 C 端展示充值方式列表需从 ser-finance 获取 | ❌ 未最终确认 |
| 充值金额范围（各方式） | 每种充值方式的最低/最高充值金额 | 钱包 C 端金额输入校验 | ❌ 未确认 |
| 提现金额范围（各方式） | 每种提现方式的单笔/单日限额 | 钱包 C 端提现金额校验 | ❌ 未确认 |
| 通道回调参数格式 | 充值成功/提现出款回调的参数结构 | 钱包接收财务转发的回调信息 | ❌ 待 ser-finance IDL 对齐 |
| 充值订单失效时间 | 需求说 10 分钟，但不同方式是否一致 | 钱包侧订单状态管理 | ❌ 待确认 |

**阻塞阶段**：**Phase 4**（充值/提现模块）

**退路方案**：Phase 1-3 不依赖通道信息，可正常推进。Phase 4 开始时，至少需要 ser-finance 的 IDL 契约定义好，钱包侧基于契约编码，通道细节由财务侧屏蔽。

### 2.4 各国银行列表与电子钱包列表

**需求依据**：多币种钱包需求文档 7-2/7-3 —— 提现时需选择银行/电子钱包，原型 10-1~10-6 展示了各国差异

**为什么缺了做不了**：提现账户绑定功能需要用户选择银行或电子钱包。如果没有各国银行/钱包的标准列表，前端下拉选项无法渲染，后端校验也无法实现。

**需要的具体信息**：

| 国家 | 需要的银行列表 | 需要的电子钱包列表 | 状态 |
|------|-------------|-------------------|------|
| 越南 | 越南各主要银行名称+银行编码（如 VietinBank、Vietcombank、BIDV...） | MoMo、ZaloPay、VNPay 等 | ❌ 未提供 |
| 印尼 | 印尼各主要银行名称+银行编码（如 BCA、BNI、Mandiri、BRI...） | OVO、DANA、GoPay 等 | ❌ 未提供 |
| 泰国 | 泰国各主要银行名称+银行编码（如 Bangkok Bank、KBank、SCB...） | PromptPay（扫码） | ❌ 未提供 |

**每条银行记录需要的字段**：
- 银行编码（标准 SWIFT 或本地编码）
- 银行名称（本地语言 + 英文）
- 银行图标（如有）
- 银行卡号格式（位数、是否有校验规则）
- 是否支持提现

**影响的功能**：
- C 端银行卡提现（F9/F10 绑定提现账户）
- C 端电子钱包提现
- 提现账户 withdraw_account 表的数据校验

**阻塞阶段**：**Phase 4**（提现模块）

**退路方案**：提现模块可以先实现 USDT 提现（不需要银行列表），银行/电子钱包提现等银行列表确认后再补充。或者先用一个简化列表（只写几个主要银行名称），后续再完善。

## 3. 第二类：跨模块 RPC 契约信息

> 这一类信息需要与其他模块的开发者对齐。不是"不存在"而是"还没有对齐定义"。

---

### 3.1 ser-finance（财务服务）RPC 接口契约

**需求依据**：跨模块依赖关系 —— 钱包↔财务 12 个双向 RPC 接口

**为什么缺了做不了**：钱包与财务之间存在深度双向依赖。钱包调财务 5 个接口（获取充值/提现方式、创建订单、查询状态），财务调钱包 7 个接口（入账/冻结/解冻/扣款/人工加减款/查余额）。没有 IDL 契约，联调无法进行。

**需要对齐的具体内容**：

#### 3.1.1 钱包调财务的接口（我需要知道他的接口长什么样）

| 接口 | 我需要知道的 | 状态 |
|------|------------|------|
| GetDepositMethods | 请求参数（币种编码？地区？）、响应结构（方式列表字段） | ❌ 待对齐 |
| CreateDepositOrder | 请求参数（金额、币种、方式编码、奖金活动ID）、响应（平台订单号、跳转URL/USDT地址） | ❌ 待对齐 |
| GetOrderStatus | 请求参数（订单号）、响应（订单状态枚举、金额明细） | ❌ 待对齐 |
| GetWithdrawMethods | 同 GetDepositMethods 结构 | ❌ 待对齐 |
| CreateWithdrawOrder | 请求参数（金额、币种、方式编码、收款账户信息）、响应（订单号） | ❌ 待对齐 |

#### 3.1.2 财务调钱包的接口（我需要确认我暴露的接口参数他能接受）

| 接口 | 需确认的 | 状态 |
|------|---------|------|
| WalletCredit | order_no 由谁生成？金额用 string？biz_type 枚举列表？ | ❌ 待对齐 |
| WalletFreeze | 冻结金额是否包含手续费？ | ❌ 待确认 |
| WalletUnfreeze | 驳回原因是否传过来存到流水？ | ❌ 待对齐 |
| WalletDebit | 出款成功后，实际到账金额、手续费等字段传什么？ | ❌ 待对齐 |
| ManualCredit/Debit | 管理员信息怎么传？指定钱包类型的枚举值？ | ❌ 待对齐 |
| GetWalletInfo | 返回哪些字段？所有钱包类型还是指定类型？ | ❌ 待对齐 |

**阻塞阶段**：**Phase 2**（钱包暴露 RPC）到 **Phase 4**（充值/提现联调）

**退路方案**：钱包侧可以先单方面定义好自己暴露的 RPC IDL（wallet_rpc.thrift），发给财务开发确认。财务侧的接口可以先用 Mock 客户端对接。但最晚在 Phase 4 之前必须完成双方 IDL 对齐。

### 3.4 投注/直播/活动模块的 RPC 调用格式

**需求依据**：依赖关系/d-3.md 第五部分 —— 游戏/投注调用 BetDebit、直播调用 GiftDebit、活动调用 BonusCredit

**为什么缺了做不了**：钱包暴露的 RPC 接口（如 BetDebit 优先级扣款、BetCredit 按比例返奖）的参数设计，取决于调用方能传什么信息。

**需要对齐的具体内容**：

| 调用方 | 需要对齐的 | 影响钱包接口设计 | 状态 |
|--------|----------|----------------|------|
| 投注模块 | 投注扣款时传什么？注单号格式、币种、扣款金额、是否需要传各钱包扣款比例 | BetDebit 参数结构 | ❌ 待对齐 |
| 投注模块 | 返奖时传什么？中心/奖励钱包的返奖比例由谁计算？ | BetCredit 参数结构 | ❌ 待对齐 |
| 直播模块 | 礼物消费的订单号格式、扣款走 Debit 还是单独接口 | GiftDebit 参数结构 | ❌ 待对齐 |
| 活动模块 | 奖励入账的活动流水号格式、入账钱包类型由谁指定 | BonusCredit 参数结构 | ❌ 待对齐 |

**阻塞阶段**：**Phase 5**（扩展 RPC）

**退路方案**：Phase 5 较晚，有充足时间对齐。但建议在 Phase 2（钱包核心 RPC）设计阶段就先定好通用参数结构（biz_type、order_no、amount、wallet_type、currency_code），后续各调用方只需按格式传参。

## 4. 第三类：产品评审待确认规则

> 这一类信息需要产品经理在评审会上明确拍板。不确认就意味着核心逻辑存在歧义，编码时只能靠猜。

---

### 4.1 提现手续费计算规则

**需求依据**：多币种钱包需求文档 7-5 —— "手续费：后台可配置（固定金额或比例）"

**缺失内容**：

| 问题 | 为什么必须确认 |
|------|--------------|
| 固定金额 OR 比例 OR 两者都支持？ | 影响手续费计算逻辑和配置表字段设计 |
| 手续费是"内扣"还是"外扣"？ | 内扣：冻结100，到账98，手续费2；外扣：冻结102，到账100，手续费2。影响冻结金额计算 |
| 不同币种手续费规则是否独立？ | VND 和 USDT 的手续费计算可能不同 |
| 是否有手续费封顶/保底？ | 比例手续费是否有最高/最低限制 |

**影响**：冻结金额计算 → 余额校验 → 流水记录拆分 → 提现订单字段。**编码 withdraw_ser.go 的前提**。

---

### 4.2 提现冻结时序

**需求依据**：评审反馈 P0-11

**缺失内容**：先冻结余额再创建财务订单（方案 A）？还是先创建财务订单再冻结（方案 B）？

| 方案 | 风险 |
|------|------|
| A: 先冻结再建单 | 冻结成功但建单失败 → 需要补偿解冻 |
| B: 先建单再冻结 | 建单成功但冻结失败 → 订单存在但钱没冻，审核通过会出款但没扣钱 |

**影响**：F11 发起提现接口的核心调用编排。方案 B 有资金安全风险，开发倾向方案 A。

---

### 4.3 充值来源决定提现方式的精确规则

**需求依据**：多币种钱包需求文档 7-1 —— "法币 + USDT 混合充值 → 只能法币提现"

**缺失内容**：

| 问题 | 为什么必须确认 |
|------|--------------|
| "充值来源"的判定维度？ | 按当前币种？按历史所有充值记录？按最近一笔？ |
| 混合充值的 USDT 部分永远无法 USDT 提现？ | 用户先充 100 USDT 再充 1000 VND，100 USDT 无法提回？ |
| 充值来源判定的时间窗口？ | 是全生命周期还是某个时间段内？ |

**影响**：提现方式过滤逻辑（F8 接口）。需要查询 deposit_order 表中的历史充值记录来判定可用提现方式，查询条件取决于规则定义。

---

### 4.4 稽核流水倍数 X 的确定

**需求依据**：币种配置需求文档 2-1 —— "奖励钱包资金需完成 X 倍投注流水后才能转入中心钱包"

**缺失内容**：

| 问题 | 为什么必须确认 |
|------|--------------|
| X 是全局固定值还是按活动/场景独立配置？ | 影响是在 reward_record 上记录 audit_multiplier 还是独立配置表 |
| 充值奖金的 X 和兑换赠送的 X 相同吗？ | 如果不同，需要分开记录 |
| 稽核完成后是自动转入中心钱包还是用户手动？ | 影响是否需要定时任务或转账接口 |
| 稽核进度如何计算？ | 需要从投注模块获取投注流水数据 |

**影响**：reward_record 表结构 + 稽核逻辑 + 钱包间转账功能。

---

### 4.5 兑换赠送比例是否可配置

**需求依据**：多币种钱包需求文档 6 —— "兑换金额额外赠送 10%"

**缺失内容**：10% 是硬编码还是后台可配置？如果可配，是全局统一还是按币种独立？是否有赠送封顶？

**影响**：如果可配置，需要新增配置表和 B 端管理接口；如果固定，直接常量定义。

---

### 4.6 禁用币种的影响处理

**需求依据**：币种配置需求文档 3-3 —— 币种有"启用/禁用"切换

**缺失内容**：禁用后，已持有该币种余额的用户怎么办？正在进行中的充值/提现订单怎么处理？

**影响**：币种状态切换接口是否需要前置校验余额和订单。

---

### 4.7 游客 USDT 充值的身份关联方案

**需求依据**：多币种钱包需求文档 5-2 —— "游客也可以 USDT 充值"

**缺失内容**：游客没有 user_id。充值的钱存在哪？注册后如何关联？永远不注册怎么处理？

**影响**：挑战 wallet 表的三维唯一性模型 (user_id, currency_code, wallet_type)。如果首版不做游客充值，需产品明确确认。

---

### 4.8 BSB 提现的完整出金链路

**需求依据**：多币种钱包需求文档 7-4 —— "BSB 提现时展示汇率换算（BSB → USDT → 用户到账金额）"

**缺失内容**：BSB → USDT → 出款，中间的兑换步骤用什么汇率？提现时 BSB 来源是法币充值兑换的，是否遵循"法币充值→法币提现"规则？

**影响**：BSB 提现接口的完整编排逻辑。

---

### 4.9 重复转账拦截的时间窗口

**需求依据**：多币种钱包需求文档 5-5 —— "同 UserID + 同币种 + 同金额，第 3 笔强制拦截"

**缺失内容**：时间窗口多长？（10 分钟？1 小时？24 小时？不限时间？）窗口重置机制？

**影响**：查询 deposit_order 的 WHERE 条件、索引设计。没有时间窗口则无法编写 SQL。

---

### 4.10 幂等键 order_no 的格式约定

**需求依据**：评审反馈 P0-13

**缺失内容**：order_no 由调用方生成还是钱包生成？不同业务场景的 order_no 格式是否统一？

**影响**：wallet_transaction 表唯一索引 `(biz_type, order_no, wallet_type)` 的字段长度、字符集设计。各调用方传入的 order_no 如果格式不统一或可能碰撞，幂等机制会失效。

---

### 4.11 主播钱包的创建条件

**需求依据**：评审反馈 P0-02

**缺失内容**：所有用户都有主播钱包？还是仅认证主播身份的用户才有？主播身份如何判定？

**影响**：wallet 表数据量（用户数 × 币种数 × 钱包类型数）和懒创建策略。

---

### 4.12 充值/提现限额的归属模块

**需求依据**：评审反馈 P0-08

**缺失内容**：限额是在财务侧支付配置中管理，还是钱包侧也有独立限额体系？两边是否会重复校验？

**影响**：是否需要新增限额配置表、B 端限额管理接口。如果财务侧全管，钱包侧透传即可，不需要建表。

| 补单 | B | B + 16位随机数（需求文档已定义） | ser-finance |

**需要做的**：在 Phase 0 阶段召集相关开发者确认此规范，写入共享文档。

**建议**（按项目既有规范 服务编号 × 1000）：

| ser-wallet | 022 | 6022000 ~ 6022999 |

**需要做的**：与财务开发者确认各自服务编号，写入工程的错误码登记表。
