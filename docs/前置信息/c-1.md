## 2. 财务模块接口契约

### 2.1 为什么缺了不行

钱包和财务是双向调用关系。钱包做充值需要问财务"有哪些支付方式"，做提现需要让财务"执行出款"。没有财务的接口定义，钱包的充值和提现流程就是断的——中间有个黑盒。

### 2.2 需要财务提供的接口（钱包 → 财务方向）

| # | 接口 | 钱包什么时候调 | 需要知道的信息 |
|---|------|-------------|---------------|
| F1 | **获取充值方式列表** | 用户进入充值页面 | Req：currency（币种）<br>Resp：方式列表（类型、名称、限额、预设金额档位） |
| F2 | **匹配支付通道并创建支付** | 用户确认充值下单 | Req：currency、amount、methodType<br>Resp：paymentUrl（跳转地址）或收款信息 |
| F3 | **获取提现方式列表** | 用户进入提现页面 | Req：currency<br>Resp：方式列表（类型、名称、限额、手续费率） |
| F4 | **发起提现出款** | 财务审核通过后 | Req：orderId、amount、currency、accountInfo<br>Resp：出款请求ID |
| F5 | **查询出款状态** | 定时补偿任务 | Req：orderId<br>Resp：status（处理中/成功/失败） |

**具体需要拿到的东西**：
- 财务模块的 **Thrift IDL 文件**（或至少接口签名+字段定义）
- 每个接口的 **错误码定义**（失败时返回什么错误码，钱包怎么处理）
- 财务服务的 **ETCD注册名**（用于服务发现）

**找谁要**：财务模块负责人

### 2.3 需要财务调用的接口（财务 → 钱包方向）

这部分由钱包定义，已在 `wallet_rpc.thrift` 中设计好。但需要和财务确认：

| # | 接口 | 财务什么时候调 | 需要对齐的内容 |
|---|------|-------------|---------------|
| R1 | RpcCreditDeposit | 充值回调验签成功后 | orderId 格式、幂等重试机制 |
| R2 | RpcFreezeBalance | 钱包内部调用（如果冻结逻辑在钱包侧） | — |
| R3 | RpcUnfreezeBalance | 提现驳回时 | orderId 关联、退回时效 |
| R4 | RpcConfirmWithdrawal | 出款成功回调后 | 状态同步机制 |
| R5 | RpcManualCredit | 人工加款操作 | 操作单号格式、审计字段 |
| R6 | RpcManualDebit | 人工减款操作 | 同上 |
| R7 | RpcSupplementCredit | 充值补单 | 补单单号格式、赠送处理 |

**需要对齐的关键问题**：
- 财务调钱包入账失败了怎么重试？重试间隔和次数约定
- orderId 的生成方由谁？（建议：充值订单号由钱包生成，出款单号由财务生成）

## 3. KYC服务接口契约

### 3.1 为什么缺了不行

提现前必须校验用户KYC认证状态。如果不知道KYC接口的具体字段，提现的前置校验就写不出来。另外保存提现账户时需要校验"持卡人姓名与KYC实名一致"，必须知道KYC返回姓名的字段名和格式。

### 3.2 需要KYC服务提供的信息

| # | 接口 | 钱包什么时候调 | 需要知道的信息 |
|---|------|-------------|---------------|
| K1 | **GetKycStatus** | 用户进入提现页面 | Req：userId<br>Resp：认证状态（未认证/待审核/已认证/已拒绝）<br>**具体问题**：状态枚举值是什么？ |
| K2 | **GetKycInfo** | 保存提现账户时校验姓名 | Req：userId<br>Resp：真实姓名（realName）、证件类型、国籍<br>**具体问题**：姓名字段名叫什么？是firstName+lastName还是fullName？ |

**具体需要拿到的东西**：
- KYC的 **Thrift IDL 文件**（或接口签名）
- 认证状态的 **枚举值定义**
- 姓名字段的 **格式和编码**（全大写？支持Unicode？）
- KYC服务的 **ETCD注册名**（工程中已有 `EtcdKycService`，需确认是否正确）

**找谁要**：KYC模块负责人

| R2 | **每家API的地址** | REST API的URL | 未知 |
| R3 | **每家API的认证方式** | API Key / Token / 免费？ | 未知 |

| R8 | **费用** | 按月/按次收费？预算多少？ | 未评估 |

### 4.3 常见的汇率API候选

以下仅供选型参考，需要决策人确认：

| API | 特点 | 加密货币支持 | 价格 |
|-----|------|------------|------|
| ExchangeRate-API | 简单易用，覆盖法币 | 不支持 | 免费/付费 |
| CoinGecko | 加密货币专长 | 支持USDT | 免费/付费 |

- 法币汇率API（1-2家）：覆盖VND、IDR、THB对USD的汇率

**找谁要**：技术负责人 / 产品经理（需要确认预算和选型）

## 5. USDT链上收款信息

### 5.1 为什么缺了不行

用户选择USDT充值时，钱包需要返回一个收款地址和二维码给用户。没有收款地址，USDT充值页面就是空白——用户没有地方可以转币。

### 5.2 需要准备的信息

| # | 信息项 | 说明 | 依赖的决策 |
|---|--------|------|-----------|
| U1 | **收款地址方案** | 统一地址 / 用户独立地址 / 订单临时地址 | 评审P1-5待确认 |
| U2 | **支持的网络类型** | TRC20(TRON)？ERC20(以太坊)？BEP20(BSC)？ | 产品确认 |
| U3 | **平台USDT收款地址** | 至少一个主地址（如果是统一地址方案） | 运维/财务提供 |
| U4 | **地址的私钥管理方式** | 私钥存在哪？HSM？KMS？配置文件？ | 安全团队确认 |
| U5 | **链上到账确认机制** | 谁来监听链上交易？钱包自己监听？三方服务？ | 架构决策 |
| U6 | **到账确认数** | TRC20要几个确认？ERC20要几个？ | 安全策略 |
| U7 | **USDT充值的最小金额** | 链上gas费考虑，太小金额不划算 | 产品确认 |

### 5.3 不同方案的信息需求差异

| 方案 | 需要准备的 | 复杂度 |
|------|-----------|--------|
| **统一地址** | 1个收款地址 + 用户通过备注/金额区分 | 低（但可能错账） |
| **地址池** | 预生成N个地址 + 分配管理逻辑 + 每个地址的私钥 | 中 |
| **三方托管** | 接入Fireblocks/BitGo等托管服务的API | 中高（需API密钥+费用） |

**后端建议**：如果USDT充值走三方支付通道（和法币充值一样由财务模块统一管），那么钱包不需要直接处理链上逻辑，只需要展示通道返回的地址即可。这种情况下此项的前置信息可以转嫁给财务模块。

**找谁要**：产品经理（方案决策）+ 运维/安全（地址和私钥）+ 财务模块（如果走统一通道）

| P7 | **充值订单过期时间** | 各方式是否相同？USDT是否更长？ | 产品确认 |

### 6.3 如果钱包需要直接处理回调（方案B）

| # | 信息项 | 说明 |

## 9. 评审决策结论

### 9.1 为什么缺了不行

以下7个问题的答案直接决定了代码怎么写。不同答案会导致完全不同的代码路径和模块边界。

### 9.2 必须拍板的决策

| # | 决策项 | 影响 | 如果不确认会怎样 |
|---|--------|------|-----------------|
| D1 | **充值回调归属**（财务/钱包） | 钱包是否需要处理三方回调的验签逻辑 | 写了可能白写（职责不归我），不写可能漏了（职责归我） |
| D2 | **币种配置归属**（子模块/独立服务） | IDL放在ser-wallet还是单独建服务 | IDL文件组织方式完全不同 |
| D3 | **充值方式数据流**（钱包聚合/前端分调） | 钱包是否需要写"获取充值方式"的聚合接口 | 接口数量差2-3个 |
| D4 | **奖金活动归属** | 钱包是否需要建活动配置表和CRUD接口 | 可能多建3-4张表、多写5-6个接口 |
| D5 | **提现审核流程细节** | 状态机有几个状态、流转规则是什么 | 提现功能的核心逻辑无法编写 |
| D6 | **提现账户生命周期** | 是否需要编辑/删除接口 | 接口数量差2个 |
| D7 | **人工加减款审核** | 是否需要审核状态表和审核流程 | 直接执行 vs 审核流程，代码复杂度差3倍 |

**找谁要**：产品经理 + 跨部门评审会
