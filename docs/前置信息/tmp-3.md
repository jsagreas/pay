## 一、三方汇率 API 信息 🔴

### 1.1 缺什么

币种配置需求 4.2 明确要求"实时汇率通过 **3家三方 API 取平均值**"。这是汇率定时刷新任务的核心数据来源。没有三方汇率 API 的接入信息，汇率体系完全无法运转。

**需要以下具体信息：**

| 前置信息 | 说明 | 为什么必须 |
|---------|------|-----------|
| 3 家 API 服务商名称 | 如 ExchangeRate-API、CurrencyLayer、Open Exchange Rates、Fixer、CoinGecko 等 | 不同API的请求格式、响应结构、支持的币种完全不同，代码层面无法通用 |
| 每家 API 的 API Key / Secret | 鉴权凭据 | 无 Key 无法调用，绝大多数汇率API免费版有调用次数限制 |
| API 请求文档 / Endpoint | 具体URL、请求参数格式、响应字段映射 | 编码时需要知道"汇率值在响应JSON的哪个字段" |
| 支持的币种范围 | 是否支持 VND、IDR、THB、USDT、BSB 全部币种 | BSB 是平台自创币种，三方API肯定不支持，BSB汇率锚定USDT需要单独处理 |
| 调用频率限制 | 免费版通常限制 100-1500次/月，付费版不同套餐不同限制 | 如果每分钟刷新×4个币种×3个API = 每天 17,280 次请求，免费版大概率不够 |
| 计费方式 | 是否需要公司采购付费套餐 | 涉及预算审批 |

### 1.2 为什么不能Mock绕过

汇率数据影响所有涉及金额换算的场景：

```
充值（BSB充值时汇率换算）→ 入款汇率 = 实时汇率 × (1+浮动%)
提现（BSB提现时汇率换算）→ 出款汇率 = 实时汇率 × (1-浮动%)
兑换（法币→BSB换算）    → 平台汇率
B端（汇率展示、偏差值计算）→ 实时汇率 vs 平台汇率
财务统计（基准币种折算）   → 平台汇率
```

**开发前期可以 Mock 固定汇率值（如 1 USDT = 15800 VND）来跑通流程，但联调和上线前必须接入真实API。** 如果API选择延迟，会直接阻塞阶段六（汇率定时刷新任务）的实际测试。

### 1.3 需要谁提供

| 责任方 | 提供内容 |
|--------|---------|
| **产品/运营** | 确定用哪 3 家汇率 API 服务商 |
| **运维/财务** | 采购 API 套餐、获取 API Key |
| **后端（我）** | 拿到 Key 和文档后编码对接 |

### 1.4 影响范围

- 币种配置模块的汇率定时刷新功能（直接阻塞）
- C端兑换页汇率展示（间接阻塞）
- BSB充值/提现的汇率换算（间接阻塞）
- B端币种配置页的实时汇率展示（间接阻塞）

## 二、USDT 链上充值的区块链对接信息 🔴

### 2.1 缺什么

需求文档 5.2 明确"USDT 充值：TRC20/ERC20/BEP20 网络，链上监听到账，达到确认数后自动上账"。这意味着平台需要：

| 前置信息 | 说明 | 为什么必须 |
|---------|------|-----------|
| **收款钱包地址**（TRC20/ERC20/BEP20 各一个或多个） | 用户看到的"收款地址 + 二维码"（原型4展示了地址可复制+二维码） | 没有收款地址，USDT充值页面无法展示任何有效信息 |
| **私钥管理方案** | 谁持有私钥？是自管还是托管？用什么钱包系统？ | 私钥安全是资金安全的根基 |
| **链上监听服务** | 是自建还是使用三方服务（如 BlockCypher、Alchemy、Moralis、NOWNodes 等）？ | 需要监听链上转账事件才能确认到账 |
| **区块确认数** | TRC20 需要几个确认？ERC20 需要几个？BEP20 需要几个？ | 确认数决定"到账延迟"和"安全性" |
| **节点/API 接入信息** | 区块链节点 RPC URL 或 三方 API Key | 查询链上交易状态的基础 |
| **链上充值服务是谁负责** | 是钱包团队自己做？还是有独立的区块链服务团队？还是外包给三方？ | 决定是"我自己写链上监听代码"还是"对接别人的回调接口" |

### 2.2 为什么不能Mock绕过

USDT 充值是需求中明确的三种充值方式之一，且有以下特殊性：

```
特殊1：USDT是所有币种都可能用到的支付方式（不只BSB，VND/IDR/THB下也可能有USDT充值入口）
特殊2：游客登录状态也支持 USDT 充值（需求5.2），是唯一不需要登录就能充值的方式
特殊3：收款地址展示是平台自有承载页（不是三方跳转），页面交互和数据完全由钱包控制
特殊4：到账机制是区块链监听（不是三方支付回调），和银行转账/扫码充值完全不同的链路
```

**开发前期可以 Mock 收款地址和到账回调来跑通流程，但这个功能的真实可用离不开链上基础设施。**

### 2.3 关键决策点

这是一个**架构级决策**，需要在编码前明确：

```
方案A：平台自建链上监听
  → 需要：区块链节点接入、自建监听服务、地址管理系统
  → 优势：完全自控，无三方费用
  → 劣势：开发量大，安全要求极高

方案B：对接三方链上充值服务（如 BlockCypher Webhooks / Alchemy Notify / NOWPayments 等）
  → 需要：三方服务 API Key、Webhook 回调地址配置
  → 优势：省开发量，三方负责链上安全
  → 劣势：依赖三方，有手续费

方案C：对接三方支付网关（支持加密货币收款的支付网关，如 Coinbase Commerce / BitPay / NOWPayments）
  → 需要：网关 API Key、回调配置
  → 优势：最省事，和银行转账类似的对接模式
  → 劣势：费率最高，且可能是财务模块的职责范围（和三方支付通道归同一层）
```

**不论哪种方案，都需要外部信息（Key/地址/服务商）才能编码。**

### 2.4 需要谁提供

| 责任方 | 提供内容 |
|--------|---------|
| **CTO / 技术负责人** | 确定链上充值方案（自建 vs 三方） |
| **区块链团队（如有）** | 提供收款地址、监听服务接口文档 |
| **运维** | 配置节点/API Key |
| **产品** | 确认区块确认数、到账时间要求 |

### 2.5 影响范围

- USDT 充值功能（直接阻塞）
- C端 USDT 支付页面（收款地址+二维码展示）
- 充值到账回调链路（链上监听→通知→上账）

## 三、财务模块接口契约 🔴

### 3.1 缺什么

钱包和财务模块存在**双向 RPC 依赖**。充值和提现两大核心流程都强依赖财务模块。目前财务模块的 IDL 和接口定义**完全不存在**（common/idl/ 下没有财务模块目录）。

**钱包调财务（3 个接口，必须知道方法签名）：**

| 接口 | 场景 | 需要知道什么 |
|------|------|-------------|
| GetPaymentConfig | C端充值页获取充值方式列表 + C端提现页获取提现方式列表 | 服务名、方法签名、Req/Resp结构（返回哪些字段：方式名/图标/档位/金额范围/关联通道/排序/费率/限额） |
| CreatePayOrder | 用户确认充值，创建通道订单 | 方法签名、传入字段（用户ID/币种/金额/充值方式/奖金活动ID）、返回字段（通道订单号/支付信息/三方承载页URL） |
| CreateWithdrawOrder | 用户确认提现，创建审核订单 | 方法签名、传入字段（用户ID/币种/金额/提现方式/账户信息/冻结流水号）、返回字段（财务订单号） |

**财务调钱包（6 个接口，我提供IDL给对方）：**

| 接口 | 场景 | 需要对齐什么 |
|------|------|-------------|
| CreditWallet | 充值成功上账 | 必传字段定义（userID/currencyCode/amount/orderNo/creditType） |
| ConfirmDebit | 出款成功确认扣除 | 必传字段、调用时序 |
| UnfreezeBalance | 驳回/失败解冻退回 | 必传字段、触发条件 |
| ManualCredit | 人工加款 | walletType 枚举值约定 |
| ManualDebit | 人工减款 | 返回值（实际减款金额） |
| SupplementCredit | 充值补单 | 补单金额+赠送金额+稽核倍数 |

### 3.2 核心卡点

```
卡点不是"财务模块代码写没写完"，而是"接口契约定没定下来"。

有契约 → 我可以 Mock 财务接口，本地独立开发充值/提现流程
无契约 → 我连 Mock 返回什么结构都不知道，充值页和提现页的字段都不确定
```

**特别是 GetPaymentConfig 这个接口**——它决定了 C 端充值页和提现页展示什么内容。产品原型上展示的充值方式（USDT/银行转账/扫码）、快捷金额档位、提现方式（银行卡/电子钱包）、费率/限额信息，**全部来自这个接口**。接口没定，Mock 都不知道 Mock 什么字段。

### 3.3 需要谁提供

| 责任方 | 提供内容 | 紧急度 |
|--------|---------|--------|
| **财务模块开发者** | 财务服务名（ETCD注册名）、IDL 文件位置（或共同约定） | 阶段四前必须 |
| **后端组长** | 牵头组织钱包+财务接口对齐会 | 越早越好 |
| **我（钱包开发）** | 提供钱包 RPC 对外接口的 IDL（wallet_rpc.thrift）给财务参考 | 阶段一完成后立即给 |

### 3.4 可替代方案

可以先用 Mock 推进（阶段一到三不依赖财务），但**阶段四（充值）和阶段五（提现）开始前必须拿到接口契约**。建议在阶段一完成后（IDL 定义好）立即安排与财务模块的对齐会。

### 3.5 影响范围

- C 端充值方式列表展示（直接阻塞）
- C 端提现方式列表展示（直接阻塞）
- 充值订单创建流程（直接阻塞）
- 提现订单创建流程（直接阻塞）
- 充值成功上账的回调集成（联调阻塞）
- 提现审核结果的回调集成（联调阻塞）
- 人工修正（加款/减款/补单）的回调集成（联调阻塞）

## 四、三方支付通道回调地址与密钥 🟡

### 4.1 缺什么

虽然三方支付通道的对接是**财务模块的职责**（钱包不直接和三方通道交互），但充值和提现流程中有一个环节需要钱包知道：

| 前置信息 | 说明 | 为什么需要关注 |
|---------|------|---------------|
| 回调通知的格式 | 财务模块收到三方回调后，通过什么方式通知钱包（RPC 调用？消息队列？） | 决定钱包的 CreditWallet 是被 RPC 同步调用还是通过消息订阅 |
| 通道订单号格式 | 财务传给钱包的通道订单号是什么格式 | 影响订单关联和对账 |
| 充值状态同步方式 | C端用户在等待支付结果，钱包怎么知道充值已成功？轮询财务？还是财务主动推？ | 影响前端轮询策略和接口设计 |

### 4.2 说明

这部分严格来说是财务模块的事（见第三节），但因为涉及回调链路的设计，钱包需要提前知道：**财务模块成功收到三方回调后，是通过 RPC 同步调用 CreditWallet 来通知钱包的吗？** 如果是异步消息（NATS/Kafka），钱包的接收方式就不同了。

### 4.3 需要谁提供

| 责任方 | 提供内容 |
|--------|---------|
| **财务模块开发者** | 回调通知方式、通道订单号格式 |
| **与第三节一并解决** | 接口对齐会上一起确认 |

### 4.4 影响范围

- 充值成功上账链路的实现方式（RPC 还是 MQ）
- 提现审核结果通知链路
- 前端充值状态轮询策略

## 五、活动模块接口契约 🟡

### 5.1 缺什么

充值模块的"额外奖金选择"功能（需求 5.6）依赖活动模块：

| 前置信息 | 说明 | 影响 |
|---------|------|------|
| 活动模块服务名 | ETCD 注册的服务名，用于 RPC 调用 | 代码中需要初始化对应的 RPC Client |
| GetAvailableBonuses 接口 | 获取当前可用的奖金活动列表 | 充值页的奖金选择区域数据来源 |
| ValidateBonus 接口 | 校验奖金活动有效性，返回奖金金额和流水倍数 | 创建充值订单时需要 |
| 活动模块 IDL | 接口定义文件 | 代码生成和调用 |

### 5.2 附加问题

兑换模块的"赠送比例"（如 10%）和"流水倍数要求"——**这个配置从哪来？** 需求文档只说了有赠送，没说谁来配这个参数：

```
可能的来源：
A. 活动模块配置 → 兑换赠送也走活动体系
B. B端币种配置中新增字段 → 钱包自己管
C. 运营后台独立配置 → 需要新增B端接口

不同来源决定不同的代码实现方式，必须确认。
```

### 5.3 需要谁提供

| 责任方 | 提供内容 |
|--------|---------|
| **活动模块开发者** | 服务名、IDL、接口文档 |
| **产品** | 确认兑换赠送比例的配置归属 |

### 5.4 可替代方案

充值奖金可以降级处理：**活动模块不可用时，充值流程应该还能走通，只是没有奖金可选。** 所以这不阻塞充值核心流程，但功能完整性需要活动接口到位。兑换赠送比例如果活动模块还没开发，可以先在 B 端币种配置中预留字段。

### 5.5 影响范围

- 充值页"请选择您的奖金"功能（功能阻塞，但可降级为无奖金）
- 兑换页赠送比例展示和计算（功能阻塞，但可用固定值 Mock）

## 七、TiDB 数据库创建 🔴

### 7.1 缺什么

ser-wallet 需要独立的数据库实例（库名建议 `ser_wallet`，遵循已有命名规范）：

| 前置信息 | 说明 |
|---------|------|
| TiDB 集群连接信息 | 开发环境的 TiDB 地址、端口、账号、密码 |
| 库是否已创建 | 需要 DBA 或运维在 TiDB 中 CREATE DATABASE ser_wallet |
| 库的字符集和排序规则 | 建议 utf8mb4 + utf8mb4_general_ci（对齐已有服务） |

### 7.2 为什么不能跳过

```
没有库 → GORM 无法连接 → gorm_gen.go 无法执行 → 无法生成 model/query/repo → 无法编写任何 Service 层代码
```

这是一个**硬阻塞**，没有库就没有数据层，没有数据层就没有业务逻辑。

## 总结

**16 项前置信息中：**

- 🔴 **5 项硬阻塞**：ETCD端口、TiDB建库、稽核流水规则、财务接口契约、USDT链上方案——其中前 3 项阻塞阶段一，后 2 项阻塞阶段四
- 🟡 **5 项卡点**：三方支付回调方式、活动模块接口、币种初始化数据、提现规则口径、兑换赠送配置来源——可以 Mock 先行但联调前必须到位
- 🟢 **6 项需要**：错误码编号、B端权限、i18n翻译、投注/直播接口约定、加密方案确认——不紧急但过程中需要陆续补齐
- 🔴 **1 项阻塞（三方汇率API）**：阻塞阶段六但不阻塞前五个阶段，有缓冲时间

**核心结论：工程基础设施100%就绪，缺的不是框架和工具，缺的是业务层面的外部信息和跨团队契约。** 框架、中间件、工具类、数据库连接、缓存、代码生成、RPC框架——这些全部是现成的。真正阻塞编码的是：产品规则未定（稽核流水）、外部资源未获取（汇率API、链上基础设施）、跨团队接口未对齐（财务模块）。

把这 5 个 🔴 解决掉，编码就可以无阻碍推进。
