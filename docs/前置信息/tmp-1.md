## 二、USDT链上充值 — 区块链相关

### 2.1 为什么需要

需求文档4.3.1明确描述：
- USDT充值是**平台自己的页面**（不跳三方），需要自己生成收款地址和二维码
- 支持三条链：TRC20（默认）、ERC20、BEP20
- 到账机制：区块链监听服务确认到账 → 自动上账

**工程现状扫描结论**：整个代码仓库中**没有任何**区块链/USDT/TRC20/ERC20/钱包地址/私钥相关的代码、配置或依赖。这是一个完全从零开始的领域。

### 2.2 具体缺失项

#### ① 收款地址生成方案

| 问题 | 说明 |
|------|------|
| **是什么** | 用户充值时展示的USDT收款地址从哪来 |
| **两种方案** | A：对接第三方地址托管服务（如 Fireblocks、BitGo）→ 需要API凭证；B：自建HD钱包派生地址 → 需要主私钥 + 派生规则 |
| **需要提供** | 确定方案A或B + 对应的API文档/密钥/主私钥 |
| **谁提供** | 运维/区块链团队 或 商务签约的三方托管服务商 |
| **被阻塞的代码** | `service/deposit/usdt.go` — 创建充值订单时生成收款地址 |

每条链（TRC20/ERC20/BEP20）的地址生成方式可能不同，需要三条链分别确认。

#### ② 区块链到账监听服务

| 问题 | 说明 |
|------|------|
| **是什么** | 监听链上交易，确认用户转账到指定地址后通知钱包服务入账 |
| **需要确认** | 是自建区块链节点监听？还是对接三方监控服务（如 Tatum、BlockCypher）？ |
| **需要提供** | 监听服务的回调格式、回调地址配置方式、确认数（几个区块确认算到账）、回调签名验证方式 |
| **谁提供** | 区块链团队 或 三方服务商 |
| **被阻塞的代码** | `handler/callback/usdt_callback.go` — 接收链上确认回调 → 触发入账 |

**关键参数需确认**：
- TRC20 确认数（通常19个块）
- ERC20 确认数（通常12个块）
- BEP20 确认数（通常15个块）
- 最低充值金额（链上gas费考虑）
- 小额粉尘攻击防御策略

#### ③ USDT提现 — 热钱包管理

| 问题 | 说明 |
|------|------|
| **是什么** | 用户提现USDT时，平台需要从热钱包向用户地址发起链上转账 |
| **需要提供** | 热钱包地址（TRC20/ERC20/BEP20各一个）、私钥管理方案（HSM？KMS？软件签名？）、转账API或SDK |
| **谁提供** | 安全团队 + 运维团队 |
| **被阻塞的代码** | 提现出款环节（由 finance 驱动，但 wallet 需要知道地址有效性校验规则） |

**安全要求**：私钥绝对不能明文存储在代码或配置中，必须确认密钥管理方案。

### 2.3 如果暂时没有怎么办

USDT相关功能是**可以延后**但**不可省略**的。如果区块链基础设施短期内无法就绪：
- 币种配置、钱包余额体系、兑换模块可以先行开发（不依赖区块链）
- USDT充值/提现代码可以先写接口框架 + 预留回调入口，区块链对接作为Phase 5-6再接入
- 但**表结构和IDL必须一开始就把USDT相关字段预留好**，否则后面加字段改动更大

## 三、银行转账/电子钱包充值 — 三方支付通道相关

### 3.1 为什么需要

需求文档4.3.2 / 4.3.3描述：
- 银行转账充值：平台向支付通道创建订单 → **跳转三方托管页** → 用户完成支付 → 三方回调
- 电子钱包充值：类似银行转账，跳三方页面展示二维码

需求文档5.2（财务管理-通道配置）明确需要配置：请求地址、商户ID、密钥、AppKey、支付编码、公钥、私钥

**工程现状扫描结论**：代码仓库中**没有任何**支付通道/支付网关集成代码。唯一的"网关"是短信网关COSMS，与支付无关。

### 3.2 具体缺失项

#### ① 代收通道（收钱）商务签约 + API文档

| 问题 | 说明 |
|------|------|
| **是什么** | 代收通道负责处理用户的充值付款（银行转账、电子钱包扫码） |
| **需要提供的信息** | 服务商名称、API文档（创建订单/查询订单/回调格式）、商户ID、密钥/AppKey/公私钥、支持的支付方式列表、测试环境（沙箱）账号 |
| **按币种至少需要** | VND（越南）：至少1家代收通道；IDR（印尼）：至少1家；THB（泰国）：至少1家 |
| **谁提供** | 商务/财务团队签约后交付 |
| **被阻塞的代码** | ser-finance 的通道对接层（但 wallet 的充值下单接口需要知道支付URL的返回格式来设计响应结构） |

**需要的具体凭证清单（每个通道）**：

```
1. 商户ID（Merchant ID）
2. API密钥（Secret Key / API Key）
3. AppKey / AppID
4. 支付编码（Payment Code）— 通道内部对支付方式的编号
5. 公钥（用于验签）
6. 私钥（用于签名）
7. 回调通知URL配置方式
8. 请求地址（API Endpoint）— 生产环境 + 测试环境
9. IP白名单要求
```

#### ② 各国银行列表

| 问题 | 说明 |
|------|------|
| **是什么** | 用户提现选择银行时的下拉列表数据 |
| **需要提供** | 各国家支持的银行名称 + 银行代码（Bank Code），通常由代收/代付通道提供支持列表 |
| **涉及国家** | 越南、印尼、泰国 |
| **谁提供** | 支付通道服务商的文档中通常附带，或产品团队整理 |
| **被阻塞的功能** | 银行卡提现 — 银行名称下拉选择（需求文档4.7.3） |

#### ③ 电子钱包渠道确认

| 问题 | 说明 |
|------|------|
| **是什么** | 需求文档列举了多种电子钱包，需确认哪些是第一期必须支持的 |
| **已提及的钱包** | 印尼：GoPay、OVO、DANA、ShopeePay、QRIS；泰国：PromptPay |
| **需要确认** | 第一期支持哪些？是否每种钱包需要单独对接？还是通过统一的代收通道（如QRIS聚合）？ |
| **谁确认** | 产品 + 商务 |

### 3.3 如果暂时没有怎么办

- 充值下单的**接口框架和订单表**可以先做，但「创建支付订单→拿到支付URL」的核心逻辑无法实现
- 本地开发可以用mock回调模拟充值到账流程，但联调必须有真实通道
- **建议**：先确认至少1个代收通道的沙箱环境，优先跑通一个币种的充值全链路

## 四、提现出款 — 代付通道相关

### 4.1 为什么需要

需求文档4.7.6描述提现流程的最后一步：
> 审核通过 → 出款（代付通道）→ 成功/失败回调

需求文档5.2（通道配置）也明确了代付通道的配置项：通道名称、币种、代付费率、单笔限额、成功率等。

### 4.2 具体缺失项

#### ① 代付通道（付钱）商务签约 + API文档

| 问题 | 说明 |
|------|------|
| **是什么** | 代付通道负责将平台资金转出到用户银行卡/电子钱包 |
| **需要提供的信息** | 与代收通道类似（API文档、凭证、沙箱环境），但代付有额外要求 |
| **额外要求** | 代付余额查询API（平台在通道中的余额）、代付结果回调格式、代付限额规则、结算周期 |
| **按币种至少需要** | VND至少1家、IDR至少1家、THB至少1家 |
| **谁提供** | 商务/财务团队 |
| **被阻塞的代码** | ser-finance 的出款逻辑（wallet负责冻结/解冻/扣款，出款由finance执行） |

**注意**：代收和代付可能是同一家服务商的不同API，也可能是不同服务商。需要商务确认具体安排。

#### ② 提现手续费规则的精确定义

| 问题 | 说明 |
|------|------|
| **是什么** | 需求文档4.7.5提到"手续费率(%)"后台可配，但没有给出初始值 |
| **需要确认** | 各币种各提现方式的手续费率（%）、是否有最低手续费（如手续费不低于X元）、手续费的计算基数（提现金额 or 到账金额） |
| **谁提供** | 产品经理 |
| **影响** | 影响提现金额计算逻辑和前端展示 |

### 4.3 wallet模块的直接影响

虽然代付对接主要是 finance 的工作，但 wallet 需要知道：
1. 代付回调后，finance 会以什么格式通知 wallet 执行「解冻+扣款」或「解冻+归还」
2. 代付通道的提现限额是否需要 wallet 在下单时做前置校验
3. 提现到账时间（T+N）的 N 值是按通道配置还是固定

## 五、汇率服务 — 三方汇率数据源相关

### 5.1 为什么需要

汇率贯穿整个业务：
- **币种配置**：各币种对BSB的基准汇率需要定时从外部获取
- **兑换模块**：法币→BSB兑换时的实时汇率计算
- **充值展示**：BSB充值页面展示汇率和兑换所得金额
- **提现展示**：BSB提现时展示汇率换算
- **USDT充值**：用户输入法币金额 → 系统换算USDT数量

需求文档提到汇率有四层体系：三方基准 → 平台调整 → B端手动覆盖 → 最终用户可见汇率。**第一层"三方基准"是外部依赖**。

**工程现状扫描结论**：代码仓库中**没有任何**汇率API、外汇数据源的集成代码或配置。

### 5.2 具体缺失项

#### ① 汇率数据源选型 + API凭证

| 问题 | 说明 |
|------|------|
| **是什么** | 自动获取法币汇率（VND/IDR/THB 对 USD 或对 BSB）的数据源 |
| **常见服务商** | ExchangeRate-API、Open Exchange Rates、Fixer.io、CurrencyLayer 等 |
| **需要提供** | 选定的服务商名称、API Key、请求频率限制、计费方式、API文档 |
| **谁决定** | 技术负责人 + 产品（涉及成本） |
| **被阻塞的代码** | `cron/rate_sync.go` — 定时拉取基准汇率并存入 `t_exchange_rate` 表 |

#### ② USDT汇率数据源

| 问题 | 说明 |
|------|------|
| **是什么** | USDT/USD 的实时汇率（USDT虽然锚定1美元但有波动），以及 USDT 对各法币的汇率 |
| **常见服务商** | CoinGecko API、CoinMarketCap API、Binance API 等 |
| **需要提供** | 选定的服务商 + API Key（部分免费接口有频率限制） |
| **被阻塞的代码** | USDT充值时金额换算（需求文档4.3.1：用户输入法币金额 → 系统根据汇率换算USDT） |

#### ③ 汇率更新频率确认

| 问题 | 说明 |
|------|------|
| **需要确认** | 法币汇率多久更新一次？USDT汇率多久更新一次？ |
| **影响** | 决定定时任务的cron表达式、API调用次数（影响计费）、缓存TTL |
| **谁确认** | 产品经理 |
| **建议** | 法币汇率：每小时或每4小时；USDT汇率：每5-15分钟（波动较大时） |

### 5.3 如果暂时没有怎么办

- 币种配置模块中，汇率字段可以先用**手动配置**的方式（B端手动填写），定时任务的「拉取三方汇率」逻辑先留空或mock
- 但**兑换模块**强依赖汇率准确性，没有真实汇率数据源之前，兑换功能只能用mock测试
- **建议**：选型不难（几分钟），关键是申请API Key + 确认计费方案。这是成本决策，需要技术负责人拍板

## 六、精度计算库 — shopspring/decimal

### 6.1 为什么需要

钱包系统的每一笔金额计算都涉及精度问题：
- 数据库用 `DECIMAL(30,8)` 存储 — 精度没问题
- Go代码中如果用 `float64` 做运算 — **必然**出现精度丢失（如 0.1+0.2≠0.3）
- 汇率换算（除法）、手续费计算（乘法）、余额增减 — 全是高风险场景

架构设计文档ADR-06已决策：**数据库DECIMAL + Go string传输 + shopspring/decimal运算** 三位一体方案。

**工程现状扫描结论**：当前项目 `go.mod` 中**没有** `shopspring/decimal` 依赖。现有代码中金额字段使用 `int32/int64`（适用于整数币种如VND/IDR的分表示，但不适用于USDT的小数精度）。

### 6.2 具体缺失项

| 问题 | 说明 |
|------|------|
| **是什么** | Go语言的高精度十进制运算库 `github.com/shopspring/decimal` |
| **为什么不能用float64** | `float64` 的 IEEE 754 标准无法精确表示十进制小数，资金计算绝对不可接受 |
| **为什么不能只用整数** | USDT有8位小数（如0.00000001 USDT），BSB/法币的汇率也有小数，整数方案需要统一放大倍数，复杂度高且易出错 |
| **需要做什么** | 在 `ser-wallet/go.mod` 中引入依赖，并在 `common/` 中封装金额运算工具函数 |
| **谁审批** | 技术负责人（引入新第三方库通常需要审批） |
| **被阻塞的代码** | 所有涉及金额运算的代码：余额增减、汇率换算、手续费计算、兑换计算 |

### 6.3 这个最容易解决

```bash
# 一条命令的事，但需要团队确认
go get github.com/shopspring/decimal
```

**建议**：这是最容易推进的前置项，建议在评审会上直接确认，会后立即引入。

## 七、币种与业务规则 — 产品侧待确认参数

### 7.1 为什么需要

币种配置模块是整个钱包的地基。表 `t_currency_config` 需要初始化数据，但很多参数值需求文档只说了「后台可配」，没有给出初始值。

### 7.2 具体缺失项

#### ① 各币种基础参数

| 参数 | 说明 | 需求文档是否给出 | 需要产品确认 |
|------|------|----------------|-------------|
| 支持的币种列表 | VND/IDR/THB/BSB/USDT 还有其他？ | 部分提及，未明确完整列表 | 第一期精确列表 |

## 八、跨模块协议 — 与财务模块联调约定

### 8.1 为什么需要

wallet 和 finance 之间有大量 RPC 调用和异步消息，双方必须就接口协议达成一致才能各自编码。

**这不是"有没有"的问题，而是"定没定"的问题**。技术组件（RPC框架、MQ）都有了，但协议内容需要双方坐下来定。

### 8.2 具体需要对齐的协议

#### ① wallet → finance 的调用

| 场景 | 需要定义的协议 |
|------|---------------|
| 充值下单（如果wallet主导） | wallet 调 finance 的什么方法？入参出参？finance 返回支付URL的格式？ |
| 查询支付配置 | wallet 查 finance 的充值方式/提现方式配置，RPC 方法名？返回结构？ |
| 查询通道余额 | wallet 是否需要查询代付通道余额？ |

#### ② finance → wallet 的调用（已在IDL设计中定义，但需双方确认）

| 场景 | 计划的RPC方法 | 需确认 |
|------|-------------|--------|
| 充值到账入账 | `WalletCredit` | 入参格式、幂等键格式、错误码约定 |
| 提现冻结金额 | `WalletFreeze` | 冻结标识、超时自动解冻策略 |
| 提现解冻归还 | `WalletUnfreeze` | 解冻后是否需要写流水 |
| 提现确认扣款 | `WalletDeduct` | 扣款=解冻金额还是可能不等？ |
| 人工加款 | `WalletManualCredit` | 加款目标子钱包、操作员信息 |
| 人工减款 | `WalletManualDebit` | 减款范围、强制扣款还是余额校验 |

#### ③ 异步消息协议

| 场景 | 需要定义 |
|------|---------|
| 提现创建后通知finance | NATS subject名称、消息体JSON格式、重试策略 |
| 充值状态变更通知 | 是否需要？格式？ |
| 余额变动事件 | 是否对外广播？谁消费？ |

### 8.3 对编码的影响

- **Phase 1-4**（脚手架、币种配置、余额体系、兑换）**不依赖**这个协议，可以先行开发
- **Phase 5-6**（充值入账、提现冻结/出款）**强依赖**双方协议确认
- **建议**：在 Phase 1-4 开发期间，双方约一次30分钟的接口对齐会，产出一份接口协议文档

## 九、评审待定项 — 架构决策依赖

### 9.1 评审反馈文档中的5个必须拍板问题

这些在评审反馈文档中已详细分析，这里提炼对编码的直接影响：

| 问题 | 如果不定 | 影响哪些代码 |
|------|---------|-------------|
| 充值流程调用方向（wallet主导 vs finance主导） | IDL方法设计可能要改、充值下单逻辑完全不同 | 充值子模块全部代码 |
| 充值/提现配置数据存哪个库 | 表建错位置要迁移、缓存策略不同 | 配置查询接口、缓存层 |
| 提现创建后怎么通知finance | 同步RPC vs 异步MQ，代码结构完全不同 | 提现子模块的通知逻辑 |
| USDT回调打到谁 | 决定wallet要不要写回调handler | USDT回调接口 |
| 人工加减款针对哪些子钱包 | 影响RPC入参设计和余额操作范围 | 人工操作RPC接口 |

### 9.2 对编码的影响

- **如果评审会能快速定论**：直接按结论编码，零返工
- **如果评审会不定**：Phase 1-4不受影响（这些决策只影响Phase 5-7），但建议**不晚于Phase 4完成前**确认，否则Phase 5开始会被阻塞

| Phase | 模块 | 需要的前置信息 | 目前状态 | 能否开工 |

| **3** | 子钱包余额体系 | shopspring/decimal | ⚠️ 需确认 | ✅ 核心逻辑不依赖外部 |

### 11.4 核心结论

```
能立即编码的（不被阻塞）：
  ✅ 工程脚手架搭建
  ✅ IDL定义和代码生成
  ✅ 数据库建表
  ✅ 币种配置CRUD（汇率拉取mock）
  ✅ 子钱包余额体系（开户/查询/增减/冻结/解冻）
  ✅ 兑换模块框架（汇率mock）
  ✅ 记录查询模块

被阻塞但可预留接口的：
  ⚠️ 汇率定时拉取（等API Key）
  ⚠️ 充值下单到三方（等通道凭证 + 接口协议）
  ⚠️ 提现出款（等通道凭证 + 接口协议）
  ⚠️ USDT充值/提现（等区块链方案）

完全无法动工的：
  ❌ 充值回调处理（没有通道回调格式无法写）
  ❌ USDT到账确认（没有监听服务无法对接）
  ❌ 跨模块联调（finance接口协议未定）
```

**一句话总结**：先推 Phase 1-4 把内部闭环做扎实，同时并行推进三方签约和接口协议，Phase 5-7 在前置信息就绪后快速接入。
