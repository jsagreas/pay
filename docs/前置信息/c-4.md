## 二、产品/运营需确认（6项）

这些不是技术问题，是业务规则的缺失。规则不定就没法写代码。

### P-1. USDT充值地址和收款方案

**现状**：代码中无任何USDT/区块链相关实现（`common/pkg/third/` 只有 ip_third.go）
**缺失内容**：
- USDT收款地址从哪来？公司自建冷热钱包？还是接第三方托管（如 Fireblocks、Bitgo）？
- 收款确认逻辑：自己跑链上监听节点？还是第三方回调？
- 支持哪些网络？TRC-20 / ERC-20 / BEP-20？
- 私钥管理方案（这个敏感，必须产品+安全+运维三方定）

**阻塞范围**：USDT充值链路（充值4个接口中的2个）
**谁来给**：产品确认方案 → 安全/运维提供密钥管理方案

### P-2. 汇率数据源API选择

**现状**：代码中无任何汇率API接入（`common/pkg/third/` 无相关文件）
**缺失内容**：
- 需求说"3家API取均值"，具体是哪3家？（CoinGecko？Binance？CurrencyLayer？Fixer？）
- API Key / 账号 / 调用频率限制 / 费用情况
- 每家API的请求格式和响应格式
- 降级策略：3家挂了1家怎么办？2家呢？

**阻塞范围**：汇率同步定时任务、兑换功能的实时汇率
**谁来给**：产品选定供应商 → 运维申请账号 → 我来对接

### P-3. 支付通道信息和回调规范

**现状**：代码中无支付相关第三方接入
**缺失内容**：
- 法币充值接的是哪些支付通道？（各国本地支付 + 银行转账）
- 法币提现接的是哪些代付通道？
- 每个通道的：接入文档、商户号、密钥、回调URL格式、签名规则
- 通道是钱包对接还是财务模块对接？（依赖关系分析中提到财务管通道，但需确认）

**阻塞范围**：充值/提现全链路
**谁来给**：产品确认通道 → 财务模块开发者/运营提供接入信息

### P-4. 金额档位与预设值数据源

**现状**：需求原型中展示了充值金额档位（如 100/500/1000/5000），但未说明是写死还是可配置
**缺失内容**：
- 各币种的充值金额预设档位从哪来？ETCD配？数据库配？写死？
- 各币种的最低充值/提现金额、最高限额由谁定？
- 这些数据是运营在B端配，还是初始化写入？

**阻塞范围**：充值方式查询、提现方式查询接口的返回数据结构
**谁来给**：产品定方案

### P-5. 流水要求（打码量）的计算规则

**现状**：需求提到奖励钱包需要完成流水要求才能转出，但计算规则不明确
**缺失内容**：
- 流水倍数怎么定？（比如奖金 × N 倍 = 需完成流水）
- 哪些下注算有效流水？（最低赔率？特定玩法？）
- 流水数据从哪个模块获取？ser-wallet 自己算还是投注模块推过来？

**阻塞范围**：wager_requirement 表设计、奖励钱包转出逻辑
**谁来给**：产品+投注模块开发者

### P-6. BSB（平台币）与USDT的固定比例

**现状**：需求提到 BSB 与 USDT 按固定比例兑换，但未给出具体比例
**缺失内容**：
- BSB:USDT = 1:1？还是其他？
- 这个比例是永久固定还是可调？
- 如果可调，谁来调？B端配置？

**阻塞范围**：兑换功能的换算逻辑
**谁来给**：产品确认

## 三、第三方/外部依赖（4项）

已确认代码中**完全没有**这些接入，需要从零建立。

### E-1. 汇率API账号和密钥

**依赖前置**：P-2 确认供应商后才能申请
**需要拿到**：
- 3家汇率服务商的 API Key
- 各自的请求频率限制（影响定时任务间隔设计）
- 沙箱/测试环境的凭证（开发阶段用）

**存放位置**：应存入 ETCD（`/wallet/third/rate_api/` 命名空间下），参考现有 `ip_third.go` 从 ETCD 读取凭证的模式

### E-2. USDT相关凭证

**依赖前置**：P-1 确认方案后才能获取
**需要拿到**（视方案而定）：
- 如果自建：热钱包私钥、冷钱包地址、链上节点RPC地址
- 如果第三方托管：API Key、Webhook Secret、商户ID

**存放位置**：ETCD 加密存储 + 私钥不入 ETCD，走 KMS 或硬件安全模块

### E-3. 支付通道凭证

**依赖前置**：P-3 确认通道后才能获取
**需要拿到**：
- 各通道的商户号、API Key、签名密钥
- 测试环境沙箱凭证
- 回调IP白名单

**归属说明**：如果财务模块管通道对接，这些凭证由财务模块开发者管理，我只需要知道 RPC 接口即可。但如果充值回调直接打到钱包，则钱包需要知道验签密钥。

### E-4. S3/OSS 上传路径约定

**现状**：工程中已有 S3 上传能力（ser-s3 模块），但币种图标的上传路径/bucket/前缀未定义
**需要确认**：
- 币种图标上传到哪个 bucket 的哪个前缀？（如 `icons/currency/`）
- 图标格式和大小限制（需求说 SVG/PNG ≤5KB，但需确认 S3 策略是否限制）

**谁来给**：与 ser-s3 模块开发者或运维确认
