## 三、外部服务凭证类（没有就完全无法对接外部系统）

### ① 三方汇率API — P0 致命阻塞

**阻塞功能**：币种配置模块的汇率自动刷新（T1）、手动刷新（B7）、整个汇率体系

**需求依据**：
- 需求理解c-3.md 2.4.2节："通过**3家三方API取汇率**，算平均值作为实时汇率"
- 需求理解c-3.md 2.4.2节："多家三方也可以避免接口API停用或挂掉的问题"
- 流程思路c-3.md 2.3节：汇率自动更新流程明确"调用3家三方汇率API"

**需要确认的信息**：

| 信息项 | 说明 | 为什么必须 |
|--------|------|-----------|
| 选择哪3家汇率API服务商 | 如 ExchangeRate-API / Open Exchange Rates / CurrencyLayer / Fixer 等 | 不同服务商的接口格式、认证方式、返回字段完全不同，无法编写适配代码 |
| 每家API的Key/Secret | API认证凭证 | 没有凭证无法调通任何一家API |
| 各家API的请求格式与返回格式 | URL模板、参数规范、响应JSON结构 | 需要编写3套适配器，格式不同就无法解析 |
| 支持的币种对列表 | 如 USDT/VND、USDT/IDR、USDT/THB | 需确认3家API是否都支持项目所需的币种对 |
| 调用频率限制（Rate Limit） | 免费/付费版每分钟/每天调用上限 | 决定定时任务刷新频率的上限 |
| 费用预算 | 免费版是否够用，是否需要付费订阅 | 免费版通常限制较多（如每月1000次），需提前确认 |

**阻塞影响**：
- 没有汇率数据 → 平台汇率为空 → 入款汇率和出款汇率无法计算
- 入款汇率为空 → 充值页面无法展示汇率信息（BSB充值必须展示"1BSB=0.1USDT"）
- 出款汇率为空 → 提现无法计算到账金额
- 兑换预览无法展示"1 BSB = ₫2,620"

**结论**：**汇率是整个系统的血液，没有汇率，币种配置模块的核心价值归零，后续钱包模块的充值/兑换/提现全部受阻。**

**提供方**：产品/运营团队选型 → 技术团队评估确认 → 运营采购API Key

### ② USDT区块链监控服务 — P0 致命阻塞

**阻塞功能**：USDT充值到账确认（CB2）、USDT充值地址生成（W6）

**需求依据**：
- 需求理解c-3.md 3.2.2节："到账机制：支付状态回传平台，通过**区块链监听服务**确认到账，达到确认数后自动上账"
- 需求理解c-3.md 3.2.1节：USDT充值"区块链监听确认到账"
- 流程思路c-3.md 3.3.2节："区块链监控服务 → 确认到账 → 通知钱包模块"

**需要确认的信息**：

| 信息项 | 说明 | 为什么必须 |
|--------|------|-----------|
| 区块链监控方案选型 | 自建节点监听 / 三方服务（如 Tatum、BlockCypher、Alchemy、Moralis） | 方案不同，回调数据结构、认证方式、接入方式完全不同 |
| 监控服务API凭证 | 三方服务的API Key / Webhook Secret | 没有凭证无法注册链上监听 |
| 支持的网络类型 | TRC20（Tron）/ ERC20（Ethereum）/ BEP20（BSC） | 需确认监控服务是否覆盖需求中的3种网络 |
| 回调数据格式 | Webhook回调的JSON结构（交易hash、确认数、金额、地址） | 需要编写回调解析和处理逻辑 |
| 确认数阈值 | 多少个区块确认后认为到账（TRC20通常19确认、ERC20通常12确认） | 直接影响到账判断逻辑和用户等待时间 |
| 回调URL配置方式 | 如何注册回调地址，是否支持动态注册 | 需要在代码中实现注册逻辑 |

**阻塞影响**：
- 没有区块链监控 → USDT链上转账无法感知 → 用户充值后永远不到账
- USDT是所有币种的推荐充值方式（BSB充值默认USDT推荐），阻塞最核心的充值链路
- 游客状态也支持USDT充值（需求理解3.2.2节），这是最基础的充值入口

**结论**：**USDT充值是平台最核心的充值入口（游客也可用），没有区块链监控服务就完全无法实现。**

**提供方**：技术团队选型 + 运营采购

**需要确认的信息**：

| 信息项 | 说明 | 为什么必须 |
|--------|------|-----------|
| 选择哪些支付通道服务商 | 如 Xendit（东南亚）/ Midtrans / PayPal 等 | 决定了API对接规范 |
| 各通道的API文档 | 接口地址、请求格式、认证方式、回调格式 | 需要按通道编写适配器 |
| 商户ID + 密钥 + AppKey | 每个通道的认证凭证 | 没有凭证无法调通 |
| 公钥/私钥对 | 用于回调签名验证和请求签名 | 回调验签是必须的安全要求 |

| 测试环境（Sandbox）信息 | 测试模式的API地址和测试凭证 | 开发阶段需要沙箱环境调试 |

**注意**：此部分信息大概率由**财务模块团队**负责对接（因为通道配置归财务管理），但钱包模块需要了解回调数据结构来做入账处理（若USDT之外的充值回调直接到钱包模块的话）。

**阻塞影响**：
- 没有通道 → 银行转账充值和扫码充值无法使用（VND/IDR/THB的主要充值入口）
- 没有通道 → 提现出款无法完成（所有提现最终都需要通道执行）
- **可Mock阶段**：钱包侧可先Mock财务模块的F1~F4接口开发，但联调前通道必须就绪

**结论**：**虽然通道由财务模块管理，但它是整个充值/提现链路的关键外部依赖。钱包模块可Mock先行，但系统级联调前必须就绪。**

**提供方**：财务模块团队对接 → 运营/商务采购通道服务

### ⑤ 财务模块IDL接口定义 — P0 致命阻塞

**阻塞功能**：充值方式列表（W4）、提现方式列表（W14）、提现配置（W16）、提交提现审核（W15）

**需求依据**：
- 接口评估c-3.md 2.4节：钱包RPC调用依赖——D5 GetRechargeMethodList、D6 GetWithdrawMethodList、D7 GetWithdrawConfig、D8 SubmitWithdrawOrder
- 依赖关系c-3.md 第七节：跨团队协调节点——"阶段一：IDL接口锁定（越早越好）"
- 实现思路c-3.md 2.5节："以下接口必须最先锁定……F1~F4（财务4个RPC）钱包充值/提现流程的硬依赖"

**需要确认的信息**：

| 信息项 | 说明 | 为什么必须 |
|--------|------|-----------|
| F1 GetRechargeMethodList 的 Thrift IDL | 请求参数（币种、地区等）和响应结构（方式名称、图标URL、金额范围、档位等） | 钱包C端充值方式列表页完全依赖此返回数据来渲染 |
| F2 GetWithdrawMethodList 的 Thrift IDL | 请求参数和响应结构（提现方式名称、到账时间标签等） | 钱包C端提现方式列表页完全依赖此数据 |
| F3 GetWithdrawConfig 的 Thrift IDL | 响应结构（单笔限额最低/最高、单日限额、手续费率%、T+N天数） | 钱包提现页面"提现须知"和金额校验逻辑完全依赖此数据 |
| F4 SubmitWithdrawOrder 的 Thrift IDL | 请求参数（订单号、用户ID、币种、金额、提现方式、账户信息、锁定汇率等） | 钱包冻结余额后需调用此接口提交审核，参数不对齐则提现流程中断 |
| 财务模块的Thrift命名空间 | 如 `namespace go ser_finance` | 需要在IDL中正确引用 |
| 财务模块的服务名和端口 | ETCD中的注册信息 | RPC Client需要知道目标服务的发现路径 |

**阻塞影响**：
- 没有F1 IDL → 充值方式列表接口无法编写请求和解析响应
- 没有F4 IDL → 提现的"冻结→提交审核"链路断裂，提现功能完全不可用
- **可Mock开发**：钱包侧可通过接口抽象注入Mock实现，但Mock的数据结构也必须基于双方约定的IDL

**结论**：**F1~F4的IDL定义是钱包模块开发充值和提现功能的硬前置，是跨团队协作的第一个卡点。建议评审后第一时间双方对齐。**

**提供方**：财务模块开发团队 + 本方共同确认

### ⑥ 用户模块KYC接口信息 — P1 严重阻塞

**阻塞功能**：提现方式展示（W14）、提现账户持有人自动填充（W18/W20/W22）

**需求依据**：
- 需求理解c-3.md 3.4.1节："KYC为可选非强制……未认证仅USDT提现，认证后全渠道"
- 需求理解c-3.md 3.4.3节："账户持有人姓名 = KYC认证姓名（提现页自动填充已实名的姓名且**不可编辑**）"
- 接口评估c-3.md 2.4节：D9 GetKYCStatus、D10 GetUserInfo

**需要确认的信息**：

| 信息项 | 说明 | 为什么必须 |
|--------|------|-----------|
| KYC状态查询接口的RPC定义 | 请求参数（用户ID）、响应结构（认证状态枚举、认证姓名、证件类型等） | 钱包提现入口需要判断用户KYC状态决定展示哪些提现渠道 |
| KYC认证姓名的获取方式 | 是通过D9一起返回，还是需要单独调D10获取 | 提现页面需自动填充"账户持有人"字段 |
| KYC状态枚举值定义 | 如 0=未认证、1=审核中、2=已认证、3=认证失败 | 需要在代码中做状态判断分支 |
| 用户模块的RPC Client是否已注册 | common/rpc/rpc_client.go 中是否已有 UserClient() | 工程中应该已有，但需确认接口是否覆盖KYC查询 |
| KYC接口的服务编号和端口 | ETCD注册路径 | RPC调用需要服务发现 |

**阻塞影响**：
- 没有KYC状态接口 → 无法判断用户是否已认证 → 提现方式列表展示逻辑缺失
- 没有认证姓名 → 提现账户"持有人"字段无法自动填充 → 违反产品需求"灰显不可编辑"要求
- **可降级开发**：可先硬编码为"未认证"状态，仅展示USDT提现，待接口就绪后切换

**结论**：**KYC接口是提现功能的前置依赖，但影响范围相对集中（仅提现模块），可降级Mock先行。**

**提供方**：用户模块开发团队

**结论**：**奖金活动数据源是充值体验的重要组成部分，但不阻塞充值主流程。建议评审时确认归属，先行开发可跳过此部分。**

### 9.3 最关键的认知

1. **汇率API和USDT基础设施是最优先级** — 它们阻塞币种配置和充值两大核心模块
2. **财务模块IDL对齐是跨团队协作的第一卡点** — 不锁定IDL就无法Mock开发
3. **币种种子数据看似简单但影响全局** — 没有基础数据全系统空转
4. **大多数P1项可通过Mock/默认值先行** — 不会完全阻塞主干开发进度
5. **推动节奏**：评审后立即启动4项 → Phase 0~1期间推动4项 → Phase 3前全部P0/P1就绪
