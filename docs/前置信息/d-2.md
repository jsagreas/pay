## 2. 按功能点的完整分析

### 2.1 USDT 充值功能

这是前置信息最密集的功能点——涉及区块链和三方支付双重外部依赖。

#### 缺失项 ①：USDT 充值地址的来源方案

| 项目 | 说明 |
|------|------|
| **需要什么** | 用户在 USDT 充值页面看到一个链上地址（TRC20/ERC20/BEP20），往这个地址转 USDT 后平台检测到账并上账 |
| **为什么必须** | 没有地址，USDT 充值页面就是空白，用户无处可转 |
| **现状** | 产品文档只说"平台生成充值地址"，但没说地址从哪来 |
| **可能的方案** | A. 平台自建链上钱包（需要私钥、节点 RPC）<br>B. 接入三方钱包服务商（如 Fireblocks、BitGo、MixPay）→ 需要服务商的 API 地址、AppKey、Secret<br>C. 使用交易所提供的商户收款 API（如 Binance Pay）→ 需要商户账号、API Key |
| **阻塞级别** | **硬阻塞** — 地址从哪来、用什么方案，直接决定充值接口的代码逻辑 |
| **需要谁提供** | 产品/运营确定方案 → 技术对接获取凭证 |

#### 缺失项 ②：链上到账检测机制

| 项目 | 说明 |
|------|------|
| **需要什么** | 用户转 USDT 到充值地址后，系统如何感知"钱到了"？ |
| **为什么必须** | 不知道到账就无法触发上账流程（CreditWallet） |
| **可能的方案** | A. 三方钱包服务商提供 Webhook 回调 → 需要回调地址配置、签名验证规则<br>B. 平台自行监听链上交易 → 需要 RPC 节点地址（如 Tron FullNode、Ethereum Infura）<br>C. 财务模块统一对接 → 钱包只管接收 CreditWallet 调用 |
| **阻塞级别** | **硬阻塞** — 决定充值流程的核心链路 |
| **需要谁提供** | 取决于缺失项①的方案选择 |

#### 缺失项 ③：USDT 充值的链上配置参数

| 参数 | 说明 | 现状 |
|------|------|------|
| 最低充值金额 | 页面展示"最低 XX USDT" | 产品文档提到但未给具体值 |
| 到账确认数 | "需 XX 个区块确认" | 产品原型有展示但未定值（TRC20 通常 19 确认，ERC20 通常 12 确认） |
| 支持的链类型 | TRC20 / ERC20 / BEP20 哪些上线？ | 产品提到三种，但是否全部上线第一期？ |

**阻塞级别**：软阻塞 — 可以先写框架，值留配置化，但测试前必须填

### 2.2 法币充值功能（银行/电子钱包）

#### 缺失项 ④：财务模块的充值相关 IDL 接口

| 项目 | 说明 |
|------|------|
| **需要什么** | 钱包调财务的 3 个 RPC 接口的 Thrift IDL 定义 |
| **为什么必须** | 充值流程中钱包需要调财务获取充值方式列表、请求支付通道、接收充值回调通知——没有 IDL 就无法生成客户端代码，无法编译 |
| **具体需要的接口** | |

```
接口 1：GetRechargeMethodList
  入参：currency_code（币种）、user_type（游客/注册用户）
  出参：方式列表（名称、图标URL、档位列表、金额范围、排序）

接口 2：RequestPayChannel（请求支付通道/创建三方订单）
  入参：order_no、currency_code、amount、recharge_method、user_id
  出参：pay_url（三方支付URL或充值地址）、channel_order_no

接口 3：充值结果通知（财务→钱包）
  方式：财务调钱包的 CreditWallet RPC（这个IDL由我方定义，已设计完成）
  但需确认：回调数据中财务会传什么参数？order_no 格式？附加信息？
```

| **阻塞级别** | **硬阻塞**（Phase 3 充值联调无法开展） |
| **需要谁提供** | 财务模块开发同事 |

#### 缺失项 ⑤：充值奖金活动的数据来源

| 项目 | 说明 |
|------|------|
| **需要什么** | 充值页面展示"额外奖金选择"（如充 100 送 10，充 500 送 80），这些奖金规则的数据从哪来？ |
| **为什么必须** | 充值接口需要拿到奖金规则才能展示给用户，用户选了奖金后提交充值需要把 bonus_type、bonus_amount、bonus_turnover 写入订单 |
| **现状** | 产品文档描述了奖金的展示和选择规则，但没有说数据存在哪个服务 |
| **可能的来源** | A. 财务模块的充值方式配置里附带奖金档位<br>B. 独立的活动系统（ser-activity）提供<br>C. 钱包本地 ETCD/DB 配置 |
| **阻塞级别** | **硬阻塞** — 不知道从哪取数据，代码里调谁的接口都不确定 |
| **需要谁提供** | 产品确认来源 → 对应模块提供接口 |

#### 缺失项 ⑥：充值订单超时时间

| 项目 | 说明 |
|------|------|
| **需要什么** | 用户创建充值订单后不支付，多久自动取消？ |
| **为什么必须** | 代码里需要写定时任务扫描超时订单并更新状态为"已取消" |
| **现状** | 产品文档完全没提 |
| **需要的值** | 银行/电子钱包类的超时时间（建议 30 分钟）、USDT 链上类的超时时间（建议不设或 24 小时） |
| **阻塞级别** | 软阻塞 — 可以先写定时任务框架，超时值从配置读，但必须有值才能跑 |
| **需要谁提供** | 产品确认 |

#### 缺失项 ⑦：重复转账拦截的精确参数

| 参数 | 说明 | 现状 |
|------|------|------|
| 时间窗口 | 同用户+同币种+同金额的判断范围（1小时？24小时？） | 产品未定义 |
| 笔数阈值 | 1-2笔警告、3笔拦截——已失败/取消的算不算？ | 未说明 |
| 金额精度 | 100.00 和 100.01 算"同金额"吗？ | 未说明 |

**阻塞级别**：软阻塞 — 可以写查重逻辑框架，但 WHERE 条件里的时间窗口不确定就跑不对

**需要谁提供**：产品确认

| **现状** | 产品给了模糊规则（法币充值→法币提现，USDT充值→USDT提现，混充→法币提现），但混合场景歧义大 |

#### 缺失项 ⑫：风控审核的对接方式

| 项目 | 说明 |
|------|------|
| **需要什么** | 提现订单"待风控审核"这一步，风控审核由谁做？钱包需要对接什么？ |

| **现状** | 产品文档有"风控审核"环节但未说明系统归属 |
| **阻塞级别** | 软阻塞 — 可以先把订单创建为"待风控审核"状态，后续审核流转是别人的事，但需要确认钱包是否有主动通知义务 |
| **需要谁提供** | 架构师/产品确认 |

### 2.5 汇率引擎（定时拉取三方汇率）

#### 缺失项 ⑯：三方汇率 API 的服务商和接入凭证

这是整个币种配置模块最关键的外部依赖。

| 项目 | 说明 |
|------|------|
| **需要什么** | 3 家三方汇率 API 的完整接入信息 |
| **为什么必须** | 汇率引擎的核心逻辑就是调这 3 家 API 取汇率 → 聚合 → 偏差检测 → 更新。没有 API 就没有汇率数据，整个兑换、BSB 充值汇率展示、BSB 提现汇率换算全部瘫痪 |
| **每家 API 需要的信息** | |

```
每家服务商需要：
  1. API 请求地址（URL）
  2. 认证方式（API Key / OAuth / 无认证）
  3. 请求参数格式（REST/GraphQL、入参格式）
  4. 响应结构（JSON 字段映射：哪个字段是汇率值）
  5. 支持的币种对（是否覆盖 VND/IDR/THB vs USDT）
  6. 调用频率限制（每分钟/每小时调用上限）
  7. 费用（免费额度？超出怎么收费？）

常见候选服务商（供参考）：
  - ExchangeRate-API（exchangerate-api.com）
  - CurrencyLayer（currencylayer.com）
  - Open Exchange Rates（openexchangerates.org）
  - Fixer.io
  - CoinGecko（加密货币汇率）
  - Binance API（USDT 相关汇率）
```

| **阻塞级别** | **硬阻塞** — 没有 API 凭证，汇率定时任务写出来也跑不了，整个汇率相关功能全部空转 |
| **需要谁提供** | 产品/运营选定服务商 → 技术完成注册和接入 |

#### 缺失项 ⑰：汇率更新的业务参数

| 参数 | 说明 | 现状 |
|------|------|------|
| 拉取频率 | 每 1 分钟？5 分钟？30 分钟？ | 产品未定 |
| 偏差阈值 | 偏差 ≥ X% 时触发平台汇率更新，X 是多少？ | 产品未定 |
| 是否按币种单独配阈值 | VND 用 2%，THB 用 1%？还是全局统一？ | 产品未定 |
| 汇率浮动百分比初始值 | 每个币种的初始浮动比例（影响入款/出款汇率） | 产品/运营未提供 |
| 汇率展示精度 | 法币保留 2 位小数，加密货币 8 位——但中间计算保留多少？ | 需确认（建议全程 int64×10^8 不截断） |

**阻塞级别**：软阻塞 — 代码框架可以写（从 ETCD 读配置），但没有值就测不了

**需要谁提供**：产品/运营提供初始值

### 2.6 币种配置管理（B 端）

#### 缺失项 ⑱：5 个币种的完整初始化数据

| 项目 | 说明 |
|------|------|
| **需要什么** | 系统上线前，currency_config 表需要 5 行初始数据（BSB/VND/IDR/THB/USDT） |
| **每行需要填的值** | |

```
每个币种需要确认：
  - currency_code：币种代码（确定：BSB/VND/IDR/THB/USDT）
  - currency_name：显示名称（如"越南盾""泰铢""平台币"）
  - currency_type：类型（法币/加密/平台）
  - nationality：国籍/地区代码
  - icon_url：币种图标（谁设计？谁上传？）
  - decimal_digits：小数位数（VND=0, IDR=0, THB=2, BSB=2, USDT=8）
  - precision_factor：精度因子（VND=1, IDR=1, THB=100, BSB=100, USDT=10^8）
  - thousands_sep：千分位符号（VND=".", THB=","）
  - decimal_sep：小数点符号（THB="."）
  - rate_float_pct：汇率浮动百分比（需运营提供初始值）
  - deviation_pct：偏差阈值百分比（需运营提供初始值）
  - status：初始状态（全部启用？还是先上部分？）
```

### 2.10 产品级业务决策（影响代码分支走向）

以下不是"具体的值"，而是"方案选择"——不同选择导致完全不同的代码实现。

#### 缺失项 ㉔：USDT 的定位

| 选择 | 代码影响 |
|------|---------|
| A. USDT 是独立币种钱包 | wallet 表要建 currency_code='USDT' 的行，用户有 USDT 余额 |
| B. USDT 仅是充提通道 | 不建 USDT 钱包，充 USDT 直接换算到 BSB，提 USDT 从 BSB 钱包按汇率换算 |

**阻塞级别**：硬阻塞 — wallet 表结构和 InitWallet 逻辑依赖此决策
**需要谁提供**：产品定论

#### 缺失项 ㉕：钱包创建时机

| 选择 | 代码影响 |
|------|---------|
| A. 注册时全量创建 | InitWallet 一次创建 5 币种 × 2 类型 = 10 行 |
| B. 首次使用时懒创建 | 每个查余额/操作余额的地方都要判空 + 自动创建 |

**阻塞级别**：硬阻塞 — 影响 InitWallet 逻辑和全部余额操作的代码模式
**需要谁提供**：产品/架构确认

#### 缺失项 ㉖：游客充值方案

| 选择 | 代码影响 |
|------|---------|
| A. 真游客（无 user_id） | 需要临时账户体系、合并逻辑——代码量和风险都很大 |
| B. "注册但未 KYC"的用户 | 不需要额外处理，走正常流程，只是限制充值方式为 USDT |

**阻塞级别**：硬阻塞 — A 方案需要额外的数据模型
**需要谁提供**：产品确认

#### 缺失项 ㉗：充值流程的职责边界

| 选择 | 代码影响 |
|------|---------|
| A. 钱包管订单 + 上账 | 需要写 CreateRechargeOrder 接口、存订单到 db_wallet、调财务请求通道 |
| B. 钱包只管上账 | 不需要 recharge_order 表（订单在财务库），C 端充值页面数据从财务获取 |

**阻塞级别**：硬阻塞 — 决定是否需要 recharge_order 表和充值相关的 C 端接口
**需要谁提供**：产品/架构定论

#### 缺失项 ㉘：提现订单的归属

| 选择 | 代码影响 |
|------|---------|
| A. 提现订单在钱包库 | withdraw_order 表在 db_wallet，C 端提现记录查钱包库 |

**阻塞级别**：硬阻塞 — 决定 withdraw_order 表是否需要以及 C 端提现记录接口怎么写
**需要谁提供**：产品/架构定论
