## 一、第三方服务凭证（没有则功能完全无法实现）

### 1.1 汇率 API 服务（3 套）

**影响功能**：币种汇率更新定时任务（T-1）、充值金额换算、提现金额换算、兑换计算、所有涉及跨币种的金额展示

**需求依据**：
> 需求理解 4.4："定时从 3 个三方汇率 API 获取数据，取平均值"
> 实现思路 9.1："并发调 3 个三方汇率 API，过滤失败的，取成功的平均值作为实时汇率"
> 技术挑战："全部 API 失败 → 保持旧值 + 告警日志"

**具体需要的信息**：

| 信息项 | 说明 | 示例 |
|--------|------|------|
| API 服务商选择（3 个） | 需确定用哪 3 家汇率数据源 | CoinGecko / ExchangeRate-API / CurrencyLayer 等 |
| API Key × 3 | 每个服务商的认证凭证 | `apikey=sk-xxxxxxxxxx` |
| API Endpoint × 3 | 请求地址和参数格式 | `GET https://api.xxx.com/v1/rates?base=USDT&symbols=VND,IDR,THB` |
| 支持的币种对 | 确认 3 家都支持 VND/IDR/THB/USDT 互转报价 | 部分 API 可能不支持冷门法币 |
| 请求频率限制 | 免费/付费套餐的 QPS 和日调用上限 | 免费版可能限制 100 次/天，不够用 |
| 响应格式 | JSON 结构差异，需要写 3 套解析逻辑 | 每家返回的字段名、层级不同 |

**缺失后果**：
- 所有币种的汇率字段为空，无法计算充值/提现/兑换金额
- 平台汇率、入款汇率、出款汇率均无数据来源
- 整个汇率管理子模块完全瘫痪

### 1.2 USDT 区块链服务

**影响功能**：USDT 充值信息获取（C-4 GetUSDTDepositInfo）、链上充值到账检测

**需求依据**：
> 需求理解 3.2："支持 3 种网络（TRC20/ERC20/BEP20），默认 TRC20"
> 需求理解 3.2："展示收款地址+QR码，用户在链上转账，2~10 分钟到账"
> 接口评估 6 号待确认："C-4 GetUSDTDepositInfo 的收款地址是平台统一地址还是每用户独立地址"

**具体需要的信息**：

| 信息项 | 说明 | 为什么缺了不行 |
|--------|------|--------------|
| 平台收款钱包地址 | TRC20/ERC20/BEP20 各一个（或多个） | 没有地址，C 端无法展示"请转到此地址" |
| 收款地址类型决策 | 统一地址 vs 每用户独立地址 | 决定链上监控方案完全不同 |
| 链上监控方案 | 自建节点 / 第三方区块链 API（TronGrid, Infura, BSCScan） | 没有监控手段，链上转账无法被系统感知 |
| 监控服务 API Key | 对应监控方案的凭证 | 无凭证无法调用链上查询 |
| 确认数阈值 | 多少个区块确认算到账（TRC20 通常 19 个） | 影响到账时间和安全策略 |
| USDT 合约地址 | 各链上 USDT 的合约地址 | 监控时需要过滤特定合约的 Transfer 事件 |

**补充说明**：如果 USDT 充值走的是三方支付通道（财务模块代收通道支持 USDT），而非平台自己监控链上，则此项转为「财务模块代收通道配置」范畴。但无论哪种方案，都需要确认方案本身。

**缺失后果**：
- USDT 充值页面无法展示收款地址和二维码
- 即使用户在链上转了 USDT，系统无法感知到账
- C-4 接口返回空数据

**影响功能**：法币充值（C-3）、法币提现（C-8）、充值回调、代付出款

**需求依据**：

> 实现思路 10.1-5："充值回调链路：三方→财务→(RPC)→钱包"

**虽然不由我方直接持有，但需确认**：

| 确认项 | 说明 | 为什么必须确认 |
|--------|------|--------------|
| 代收通道（充值）是否已接入 | 至少 1 个通道可用 | 无代收通道 → 法币充值功能完全不可用 |

**缺失后果**：
- 即使我方钱包核心代码写完，充值和提现的完整链路无法跑通

## 二、跨模块 RPC 接口定义（没有则无法编写调用代码）

### 2.1 财务模块 RPC 接口定义

**影响功能**：充值方式获取（C-2）、创建充值订单的通道匹配（C-3）、提现方式获取（C-7）、发起代付（提现出款）

**需求依据**：
> 接口评估 6.3："钱包调财务的 RPC：GetPaymentMethods / MatchAndCreateChannelOrder / InitiateChannelPayout / GetChannelOrderStatus"
> 实现思路 13.1："6 个核心 RPC 的入参出参（IDL 定义）— 待定义 — 最高重要"

**具体需要对方提供**：

| RPC 接口 | 我方调用场景 | 需要的信息 |
|----------|------------|-----------|
| `GetPaymentMethods` | 用户打开充值/提现页面 → 展示可用的支付方式列表 | 入参：币种代码、方式类型(充值/提现)；出参：方式列表(名称/图标/金额范围/档位) |
| `MatchAndCreateChannelOrder` | 用户确认充值 → 匹配最优通道并创建通道订单 | 入参：平台订单号、金额、支付方式；出参：通道订单号、支付跳转URL/参数 |
| `InitiateChannelPayout` | 提现审核通过 → 发起代付出款 | 入参：提现订单号、金额、收款账户信息；出参：通道代付单号 |
| `GetChannelOrderStatus` | 查询充值/提现通道订单的最新状态 | 入参：通道订单号；出参：状态(成功/失败/处理中) |

**关键信息**：这 4 个 RPC 的 Thrift IDL 定义必须由财务模块开发者提供。没有 IDL 定义，我方无法生成调用代码，对应功能无法编写。

**过渡方案**：Phase 3 之前可以 Mock，但 Phase 5/6 必须有真实 IDL。

## 四、跨模块协作约定（没有则接口对不上）

### 4.1 提现冻结发起方确认

**影响功能**：提现订单创建（C-8）的冻结时机和调用链路

**需求依据**：
> 实现思路 10.1-4："提现冻结由谁发起：我方内部冻结还是财务模块调 FreezeBalance？— 推测我方内部冻结"

**两种方案的代码完全不同**：

| 方案 | 流程 | 代码影响 |
|------|------|---------|
| A. 我方内部冻结 | 用户提交提现 → ser-wallet 自行冻结余额 → 创建提现订单 → 通知财务模块审核 | CreateWithdrawOrder 内部包含 Freeze 逻辑 |
| B. 财务模块冻结 | 用户提交提现 → ser-wallet 创建提现订单(不冻结) → 财务模块收到后调 FreezeBalance | CreateWithdrawOrder 不做冻结，等 RPC 回调 |

**当前推测方案 A**，但必须与财务开发者确认。否则可能出现：
- 方案 A 实现后对方也冻了一次 → 双重冻结
- 方案 B 实现后对方不冻 → 用户余额未冻结就进入审核，期间可能被消费
