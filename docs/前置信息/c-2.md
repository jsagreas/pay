# 二、🔴 硬阻塞项（6 项）

## ❶ 财务模块 RPC 接口契约

### 缺失什么

**经代码实证确认：`common/idl/` 目录下不存在任何财务管理模块的 IDL 定义文件，`namesp.go` 和 `rpc_client.go` 中也没有财务相关服务注册。**

财务管理模块是钱包模块最大的外部依赖——充值和提现的完整链路需要跨模块 RPC 协作。但目前：
- 没有财务模块的 IDL 文件
- 没有方法签名定义
- 没有请求/响应结构定义
- 没有错误码约定
- 没有超时/重试策略约定

### 具体需要的接口契约

**钱包需要调用财务的 6 个 RPC 方法：**

| 方法 | 钱包在什么时候调 | 需要确认的字段 |
|------|----------------|---------------|
| GetDepositMethods | C 端充值页加载 | 入参：currencyId? 出参：方式列表结构（id/name/type/icon/limits?） |
| MatchDepositChannel | 用户提交充值时 | 入参：currencyId+methodId+amount? 出参：channelId+channelInfo? |
| CreateChannelOrder | 匹配通道后创建订单 | 入参：channelId+amount+userInfo? 出参：支付链接/地址/二维码? |
| GetWithdrawMethods | C 端提现页加载 | 入参：currencyId? 出参：方式列表结构 |
| SubmitWithdrawRequest | 用户提交提现后 | 入参：金额+收款信息+钱包冻结记录? 出参：审核单号? |
| QueryChannelOrderStatus | 查通道订单状态 | 入参：orderNo? 出参：状态枚举? |

**财务需要调用钱包的 7 个 RPC 方法：**

| 方法 | 财务在什么时候调 | 需要对齐的约定 |
|------|----------------|---------------|
| Credit | 充值成功/补单/人工加款 | orderNo 由谁生成？walletType 怎么传？ |
| Debit | 提现完成/人工减款 | 提现扣款是扣 frozen 还是 balance？ |
| Freeze | 创建提现订单时 | 还是钱包自己 Freeze？ |
| Unfreeze | 提现被拒绝时 | 解冻金额必须和冻结金额一致吗？ |
| QueryBalance | 人工修正前查余额 | 返回哪些钱包类型的余额？ |
| QueryWalletDetail | B 端查用户钱包 | 分页？返回哪些字段？ |
| CreditReward | 奖金发放 | 这个是否需要？还是 Credit 的 walletType=2 就够了？ |

### 为什么是硬阻塞

- **没有这个契约，充值 C 端功能完全无法编码**：GetDepositMethods/MatchDepositChannel/CreateChannelOrder 的入参出参都不知道，代码怎么写？
- **没有这个契约，提现 C 端功能完全无法编码**：同理
- **Mock 策略需要知道数据结构才能 Mock**：不知道 GetDepositMethods 返回什么结构，连 Mock 数据都造不出来

### 阻塞范围

```
直接阻塞的接口（~13 个，约 30% 的总接口量）:
  C 端: GetDepositMethods / CreateDepositOrder / GetWithdrawMethods
        CreateWithdrawOrder / CheckWithdrawEligibility（部分）
  RPC: Credit / Debit / Freeze / Unfreeze / QueryBalance
       QueryWalletDetail / CreditReward
  B 端: 无直接阻塞，但 AdminGetUserWallet 的展示字段可能涉及
```

### 谁提供 / 怎么解决

| 方案 | 说明 |
|------|------|
| **最优** | 和财务模块负责人一起定义双方的 IDL，双方各自生成代码 |
| **次优** | 钱包先定义被调接口（Credit/Debit 等）的 IDL，财务先定义被调接口的 IDL，各自先做 |
| **兜底** | 钱包先定义自己这边所有 IDL（含 Mock 版的财务调用接口），联调时再对齐修改 |

### 需要对齐的关键约定（超出 IDL 字段范围）

| 约定项 | 为什么必须提前确认 |
|--------|-------------------|
| 回调方式 | 充值成功后财务是同步 RPC 调 Credit，还是 MQ？影响钱包的 Credit 实现方式 |
| orderNo 归属 | 谁生成 orderNo？财务用自己的订单号还是钱包的？影响幂等设计 |
| Freeze 执行者 | 提现时是钱包自己 Freeze 后告诉财务冻结金额，还是财务调钱包 Freeze？影响流程设计 |
| 重试策略 | RPC 超时后谁重试？间隔多久？影响幂等测试场景 |
| 错误码透传 | 财务调 Credit 失败时，钱包返回的错误码财务怎么处理？需统一规范 |

## ❷ 第三方汇率 API 选型与凭证

### 缺失什么

产品需求明确要求：**"从 3 个第三方 API 获取汇率，取平均值更新实时汇率"**（来源：币种配置需求文档）

但当前完全不知道：

| 缺失项 | 说明 |
|--------|------|
| **哪 3 个 API** | CoinGecko? Binance? CoinMarketCap? ExchangeRate-API? Fixer.io? 具体是哪三个？ |
| **API Key / 凭证** | 这些 API 几乎都需要注册获取 API Key |
| **请求格式** | 每个 API 的 endpoint、参数格式、认证方式各不相同 |
| **返回格式** | 每个 API 返回的 JSON 结构不同，需要分别解析 |
| **调用频率限制** | 免费版通常有 rate limit（如每分钟 30 次），付费版需要费用预算 |
| **支持的币种对** | 是否支持 VND/USDT、IDR/USDT、THB/USDT 的直接查询 |
| **凭证存放位置** | 放 ETCD？放环境变量？放代码？需要确认 |

### 为什么是硬阻塞

- **币种配置模块的定时任务 `SyncThirdPartyRates` 是整个汇率体系的数据源头**
- 没有汇率 API → 没有实时汇率 → 没有平台汇率 → 没有充值/提现汇率 → 兑换功能也无法正确计算
- 这不是"可以 Mock"的场景——开发时可以用假数据，但 API 集成代码本身需要知道接口格式才能写

### 阻塞范围

```
直接阻塞:
  · SyncThirdPartyRates 定时任务（无法编码 HTTP 请求和响应解析）
  · CheckRateThreshold（依赖实时汇率数据）

间接阻塞（可先用假数据跑通，但最终必须替换）:
  · GetExchangeRate / CreateExchangeOrder
  · 所有涉及汇率展示的 C 端接口
```

### 谁提供 / 怎么解决

- **选型决策**：需要产品/运营或技术负责人确认使用哪 3 个 API 平台
- **账号注册**：注册 API Key，确认付费方案
- **技术对接**：拿到文档后编写 3 个 adapter（适配器模式，统一输出格式）

## ❹ 奖励钱包稽核规则与执行方

### 缺失什么

产品需求多次提到：
> "奖金进入奖励钱包，需要完成对应流水要求后才能转入中心钱包"
> "奖金需完成 X 倍流水才能转入中心钱包"
> "兑换可获得 10% 额外 BSB 奖金，但奖金部分进入奖励钱包，需完成 X 倍流水"

但关键规则全部未定：

| 缺失项 | 为什么必须知道 |
|--------|--------------|
| **"流水"具体指什么？** | 是用户在平台的消费金额（投注额）？还是充值金额？还是所有交易金额？ |
| **X 倍的 X 是多少？** | 固定值？还是每个活动单独配置？ |
| **谁来跟踪和计算流水？** | 钱包自己算？还是有独立的稽核服务？如果是投注额，需要从竞彩/游戏模块获取数据 |
| **达标后谁触发转账？** | 用户手动点"转到中心钱包"？还是系统自动转？ |
| **TransferToCenter 的触发条件** | 怎么校验用户是否完成了稽核要求？查哪个数据源？ |

### 为什么是硬阻塞

- 奖励钱包是核心钱包类型之一，充值奖金和兑换奖金都进这里
- **不知道稽核规则 → 不知道 TransferToCenter 的校验逻辑怎么写 → 奖励钱包的资金出口完全没法实现**
- 如果"流水"是投注额，钱包模块需要调用竞彩/游戏模块的 RPC 来获取——而这个 RPC 也不知道存不存在

### 阻塞范围

```
直接阻塞:
  · TransferToCenter（奖励钱包→中心钱包转账）的校验逻辑
  · 奖励记录中"稽核状态"字段的计算逻辑

间接影响:
  · 奖励记录查询（稽核进度怎么展示？）
  · reward_record 表结构（需要存稽核相关字段）
```

### 谁提供 / 怎么解决

- **需要产品确认**："流水"的业务定义、X 的具体值或配置来源
- **需要技术确认**：如果流水=投注额，需要知道从哪个服务获取用户投注数据

## ❻ 服务编号与 ETCD 注册信息

### 缺失什么

**经代码实证确认：`namesp.go` 中当前最高编号 021（ser-mall），016 空缺，ser-wallet 未注册。**

| 缺失项 | 影响 |
|--------|------|
| **服务编号** | 用 022 还是 016？决定错误码前缀（6022xxx vs 6016xxx） |
| **ETCD 端口配置** | `/slg/serv/wallet/port` 的值是多少？ |
| **ETCD 数据库名** | `/slg/serv/wallet/db` 的值是多少？TiDB 中要创建哪个数据库？ |
| **TiDB 数据库创建** | 数据库还不存在，需要 DBA 或有权限的人创建 |

### 为什么是硬阻塞

- 没有服务编号 → 不能定义错误码 → errs/code.go 写不了
- 没有 ETCD 端口 → 服务启动不了
- 没有数据库 → GORM Gen 跑不了 → model/query/repo 生不了 → 所有数据访问代码写不了

### 谁提供 / 怎么解决

- **服务编号**：问团队约定，022 还是 016
- **端口号**：找运维或团队约定，避免端口冲突
- **数据库**：找 DBA 在 TiDB 中创建数据库，配置 ETCD key-value

### 缺失什么

币种配置模块是自己负责的 B 端功能，CRUD 代码可以先写。但系统初始化时需要预置币种数据：

| 币种 | 需要确认的参数 |

| IDR | decimals=0？浮动比例？阈值？ |
| THB | decimals=2（有 satang 概念）？还是 decimals=0？浮动比例？阈值？ |

### 为什么是软阻塞（不是硬阻塞）

- CRUD 代码不依赖具体参数值，可以先写
- 表结构中 float_rate/threshold_rate 字段类型是确定的（万分比整数）
- 开发测试时可以用合理估计值（如 float 50 = 万分之五十 = 0.5%）
- **但上线前必须由运营确认真实参数值**

### 谁提供

产品/运营团队提供各币种的浮动比例、阈值等业务参数

## ❿ 提现方式对应的收款信息字段定义

### 缺失什么

提现时用户需要填写收款信息，不同提现方式的收款字段不同：

| 提现方式 | 需要用户填什么？ | 是否明确？ |
|----------|----------------|-----------|
| 银行卡（VND） | 银行名称+卡号+姓名？还有分行信息？ | 产品原型有"选择银行+卡号+姓名"，但越南银行列表从哪来？ |
| 银行卡（IDR） | 同上 | 印尼银行列表？ |
| 银行卡（THB） | 同上 | 泰国银行列表？ |
| 电子钱包（GoPay/OVO/DANA/ShopeePay） | 钱包类型+账号？还需要其他？ | 基本明确 |
| PromptPay（THB） | PromptPay ID？手机号？身份证号？ | 需确认 |
| USDT 链上 | 网络选择+钱包地址 | 基本明确 |

核心问题：**各国银行列表数据从哪来？**
- 是硬编码在前端？
- 是钱包模块维护一张银行列表表？
- 是财务模块提供？
- 是从第三方服务获取？

- withdraw_order 表的 receiver_info 字段可以先用 JSON 存储（已设计为 TEXT 类型）
- 收款信息的具体字段不影响 ledger 层的 Freeze/Debit 核心逻辑
- 但提现页面的表单字段展示、后端校验规则、AES 加密哪些字段，都依赖这个定义

### 谁提供

产品确认各国提现方式的具体表单字段 + 银行列表数据来源

## ⓬ 老用户资金来源初始状态

### 缺失什么

提现有"资金来源规则"：
> 纯法币充值 → 只能法币提现
> 纯 USDT 充值 → 只能 USDT 提现
> 混合充值 → 只能法币提现

这个规则需要追踪每个用户的充值历史来判定。对于**新系统上线时已经存在的老用户**：
- 他们之前的充值记录在旧系统，新系统的 deposit_order 表里没有
- 他们的资金来源类型怎么初始化？

### 为什么是软阻塞

- 新用户完全不受影响（从第一笔充值开始追踪）
- 老用户问题可以在上线前通过数据迁移脚本解决
- 但如果不提前考虑，上线后老用户提现可能被错误拒绝

### 谁提供

和产品/运维确认老用户迁移策略

## ⓭ 定时任务调度方式

### 缺失什么

需求中有两个定时任务：
- `SyncThirdPartyRates`（拉取汇率，分钟级）
- `CheckRateThreshold`（阈值检查，伴随汇率拉取）

**经代码实证确认：项目支持三种调度方式：**
- robfig/cron（进程内调度）
- XXL-JOB（分布式调度）
- ser-cron 服务（编号 020，通过 RPC 触发）

但不确定：
- ser-wallet 的部署方式是单实例还是多实例？
- 如果多实例，用 cron 会导致每个实例都执行（重复拉取汇率）
- 应该用 XXL-JOB 还是 ser-cron？

### 为什么是软阻塞

- 开发阶段可以先用 robfig/cron（最简单）
- 上线前根据部署方式切换为 XXL-JOB 或 ser-cron

### 谁提供

和运维/团队确认部署方式

## 不能开始编码前必须搞定的 TOP 3

```
1. 🔴 ❻ 服务编号 + 数据库创建 + ETCD 配置
   → 没有这个，骨架都起不来
   → 找团队 5 分钟能定，找运维 10 分钟能建库
   → 最快解决的硬阻塞

2. 🔴 ❶ 和财务模块负责人对齐 RPC 契约
   → 影响 30% 的功能实现
   → 至少先对齐 Credit/Debit/Freeze/Unfreeze 的签名
   → 建议约一次 30 分钟的对齐会议

3. 🔴 ❷ 确认第三方汇率 API
   → 影响币种配置模块的核心定时任务
   → 需要产品定方向 + 注册 API Key + 阅读接口文档
```
