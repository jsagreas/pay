| T+N到账天数 | ? | ? | ? | ? | ? |

**为什么不能先写代码后填数据**：

- **USDT 精度**尤其关键：链上支持 6 位小数，但平台允许到几位？这影响最小充值金额和展示格式。

**哪个功能被阻塞**：

- C端金额格式化逻辑（展示几位小数）
- 充值/提现限额校验
- 手续费计算
- 单元测试的测试数据

**谁提供**：产品经理定义，运营确认具体数值

## 2.2 【产品】BSB 的定位 — 展示币还是结算币

**缺什么**：BSB 在平台中的核心定位。

**两种可能**：
- **结算币**：所有消费（投注/打赏等）统一用 BSB → 充值法币后**必须先兑换**成 BSB 才能使用 → 兑换是"必经之路"
- **展示币**：BSB 只是展示层的统一计量单位 → 法币可以直接消费 → 兑换是"可选操作"

**为什么是前置信息而非"开发时再说"**：
- 如果 BSB 是结算币：充值入账后可能需要**自动兑换**逻辑，中心钱包存的是 BSB 不是法币
- 如果 BSB 是展示币：中心钱包存法币原始金额，BSB 余额只是汇率换算后的展示值
- 这两种方案的数据模型、余额计算、兑换流程设计**完全不同**，开发到一半改方向等于推翻重来

**哪个功能被阻塞**：
- `user_balance` 表的余额存储含义（存法币还是 BSB？）
- 兑换流程是"可选功能"还是"核心必经路径"
- 充值入账时是否需要自动触发兑换
- 投注/打赏等其他模块的对接方式

**谁提供**：产品经理明确

---

## 2.3 【产品】汇率的管理粒度

**缺什么**：汇率配置的具体规则。

**需要明确的3个子问题**：

**(a) 买入/卖出汇率是否分开？**
- 单向：`1 BSB = 23000 VND`，兑换和提现展示用同一个值
- 双向：买入汇率 `1BSB = 23000 VND`（用户兑换时用），卖出汇率 `1BSB = 22500 VND`（提现展示时用）
- 这决定了 `exchange_rate` 表是存 1 个 rate 字段还是 2 个（buy_rate / sell_rate）

**(b) BSB 对 USDT 的汇率怎么管？**
- 手动配置（和法币一样）？→ 运营需要多频繁更新？USDT 价格波动大
- 接入外部行情自动更新？→ 需要对接第三方行情 API（不在当前需求范围内，但需要确认）

**(c) 汇率修改后对进行中操作的影响？**
- 用户在兑换预览页看到汇率后停留 5 分钟再确认，此时汇率已变。用提交时最新汇率，还是页面展示的汇率？
- 如果要求"页面展示什么就用什么"：需要汇率锁定机制（token 锁 + TTL），开发成本完全不同

**哪个功能被阻塞**：
- `exchange_rate` 表结构（1字段 vs 2字段）
- 兑换计算逻辑
- 提现展示逻辑
- 汇率缓存策略

**谁提供**：产品经理定义规则

**必须逐条确认**：

| (b) | 兑换从哪个子钱包扣、BSB 入哪个子钱包？赠送入哪个？ | 兑换事务中的 3 条余额操作 |

```
  - Request 结构：用户ID、币种、金额、支付方式ID、平台订单号 — 是否还需要其他字段？
  - Response 结构：
    - 渠道订单号（字段名和类型）
    - 支付信息（USDT：收款地址+二维码URL+链网络；法币：第三方支付页面URL）
    - 这些支付信息是统一一个 map 返回还是分字段返回？
  - 超时时间：这个 RPC 调用的预期响应时间？（渠道对接可能很慢，是否需要设置长超时？）
```

### 接口 C：提交提现审核

```
触发场景：C端确认提现，钱包冻结余额后调用
我需要知道：
  - Request 结构：用户ID、币种、金额、手续费、提现方式、平台订单号、提现账户信息
    - "提现账户信息"怎么传？整个结构体嵌套进去？还是只传 withdraw_account_id？
    - 是否需要传 KYC 信息？还是财务自己去查用户服务？
  - Response 结构：审核单号（类型？i64 还是 string？）
```

**为什么这是最大阻塞项**：
- 充值流程的 Step 1（获取充值方式）和 Step 5（创建渠道订单）直接依赖接口 A 和 B
- 提现流程的 Step 7（提交审核）直接依赖接口 C
- 没有 IDL 定义 → 钱包无法 import → 编译不过 → 第四期完全启动不了

**最小推进方案**：如果财务模块开发进度慢，至少需要先提供 IDL 文件（只有 struct 和 service 定义），钱包可以先基于 IDL 编码，联调时再对接真实实现。

## 4.2 【财务模块负责人】财务回调钱包的 RPC 协议 ★★★

**缺什么**：财务调用钱包 RPC 接口时的协议约定。

**钱包暴露给财务的 7 个 RPC 接口**（已在钱包 IDL 设计中定义）：

| 接口 | 触发时机 |
|------|---------|
| 充值入账 DepositCallback | 渠道充值回调成功后 |
| 提现确认扣款 WithdrawConfirm | 审核通过+出款成功后 |

| 人工加款 ManualCredit | 加款审核通过后 |
| 人工减款 ManualDebit | 减款审核通过后 |
| 人工补单 ManualReplenish | 补单审核通过后 |
| 余额查询 QueryBalance | 创建减款单前/任何需要查余额时 |

| 待对齐项 | 具体问题 |
|---------|---------|

## 4.4 【产品】手续费的计算方向

**缺什么**：提现手续费从余额扣还是从提现金额扣。

**两种方案**：

| 方案 | 用户操作 | 手续费 5 | 余额变化 | 实际到账 |
|------|---------|---------|---------|---------|
| A：从提现金额扣 | 提现 100 | 5 | -100 | 95 |
| B：从余额额外扣 | 提现 100 | 5 | -105 | 100 |

**为什么不能两个都支持**：代码层面虽然能做，但 C 端展示逻辑完全不同（方案 A 展示"到账金额"，方案 B 展示"总扣款金额"），余额校验条件也不同（方案 A 校验 `available >= amount`，方案 B 校验 `available >= amount + fee`），必须选一个。

**延伸问题**：
- 手续费有没有最低/最高限制？（比例制手续费 2%，提现 100 万收 2 万，有封顶吗？）
- 手续费算出来有小数时，向上取整还是向下取整？

**哪个功能被阻塞**：
- 提现接口的余额校验条件
- 提现冻结金额计算（冻结 amount 还是 amount + fee）
- C端提现页面的手续费展示逻辑

**谁提供**：产品经理
