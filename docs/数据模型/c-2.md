# 一、从业务概念到数据模型的映射总览

## 1.1 映射思路

```
产品原型中的业务概念              映射为数据模型              映射理由
──────────────────              ──────────────              ─────────

用户的钱包余额                → user_wallet 表              每人×每币种×每钱包类型=一行记录
每一笔余额变动                → wallet_flow 表              append-only 流水，审计+幂等双重载体
用户当前选中的币种            → user_wallet_setting 表       全局单一数据源，影响所有页面展示

充值订单（页面→支付→回调）    → deposit_order 表            记录充值生命周期
提现订单（页面→冻结→审核）    → withdraw_order 表           记录提现生命周期+收款信息
兑换订单（法币→BSB 即时完成） → exchange_order 表           记录跨币种兑换快照

币种基础配置（名称/icon/精度） → currency_config 表          运营后台管理的币种元数据
四层汇率数据（实时/平台/充值/提现）→ currency_rate 表        每币种一行，存储四层汇率
平台汇率变更记录              → rate_change_log 表           阈值触发或手动修改时记录变更

充值奖金活动配置              → deposit_bonus_config 表      运营配置的奖金活动（来源待确认）
奖励稽核跟踪                  → reward_audit 表             跟踪奖金流水达标进度（需求待细化）
```

```
├─────────────── 交易订单域 ────────────────┤
│                                           │

│  withdraw_order        提现订单           │
│  exchange_order        兑换订单           │

├─────────────── 币种配置域 ────────────────┤
│                                           │
│  currency_config       币种配置           │
│  currency_rate         汇率数据           │
│  rate_change_log       汇率变更日志       │
│                                           │
├─────────────── 待确认域 ─────────────────┤
│                                           │
│  deposit_bonus_config  充值奖金活动配置    │

└───────────────────────────────────────────┘

确定模型: 9 张表（必须有，可直接建表）
```

```
──────                   ──────   ──────────
```

## 2.3 充值订单状态（DepositStatus）

```
业务流转: 待支付 → 已完成 / 支付失败 / 已取消

枚举值    状态          触发条件                     是否终态
──────   ────          ────────                     ──────
0        待支付        创建订单后                     否
1        已完成        通道回调支付成功                是
2        支付失败      通道回调支付失败                是
3        已取消        超时未支付/用户取消              是
```

## 2.4 提现订单状态（WithdrawStatus）

```
业务流转: 审核中 → [风控通过] → [财务通过] → [出款通过] → 出款中 → 已完成 / 已拒绝 / 出款失败 / 已取消

枚举值    状态          触发条件                     是否终态
──────   ────          ────────                     ──────
0        审核中        提交提现后（含风控/财务/出款三级审核） 否
1        已完成        出款成功，余额已扣减              是
2        已拒绝        任一级审核拒绝，已解冻              是
3        出款失败      通道出款失败，已解冻                是
4        已取消        用户或系统取消，已解冻              是
5        已锁定        管理员锁定暂停审核                 否
```

# 三、钱包核心域模型

## 3.1 user_wallet — 用户钱包账户表

### 业务来源

```
产品原型映射:
  · 个人中心页 → 显示当前币种下的余额
  · 每个用户 × 每个币种 × 每种钱包类型 = 一行记录
  · 钱包结构: 中心钱包 + 奖励钱包 + 主播钱包（本期）
  · 资金钱包余额 = 中心钱包 + 奖励钱包
  · 可用余额 = balance - frozen_amount（计算值，不存储）
  · 消费扣款顺序: 先扣中心钱包，不够再扣奖励钱包

核心规则:
  · 余额和冻结金额不允许为负（应用层保证）
  · 所有余额变更必须经过 ledger 层
  · 并发控制使用 SELECT FOR UPDATE 行锁
```

### DDL

```sql
CREATE TABLE user_wallet (
    id              BIGINT       NOT NULL PRIMARY KEY                COMMENT '主键（雪花ID）',
    user_id         BIGINT       NOT NULL                            COMMENT '用户ID',
    currency_id     BIGINT       NOT NULL                            COMMENT '币种ID（关联 currency_config.id）',
    wallet_type     TINYINT      NOT NULL                            COMMENT '钱包类型: 1=中心 2=奖励 3=主播 4=代理 5=场馆',
    balance         BIGINT       NOT NULL DEFAULT 0                  COMMENT '余额（最小货币单位，BIGINT 整数）',
    frozen_amount   BIGINT       NOT NULL DEFAULT 0                  COMMENT '冻结金额（提现冻结时增加）',
    version         BIGINT       NOT NULL DEFAULT 0                  COMMENT '乐观锁版本号（预留，当前主用行锁）',
    status          TINYINT      NOT NULL DEFAULT 1                  COMMENT '状态: 1=正常 2=冻结 3=禁用',
    create_by       BIGINT       NOT NULL DEFAULT 0                  COMMENT '创建人ID',
    create_at       BIGINT       NOT NULL DEFAULT 0                  COMMENT '创建时间（毫秒时间戳）',
    update_by       BIGINT       NOT NULL DEFAULT 0                  COMMENT '更新人ID',
    update_at       BIGINT       NOT NULL DEFAULT 0                  COMMENT '更新时间（毫秒时间戳）',
    delete_at       BIGINT       NOT NULL DEFAULT 0                  COMMENT '删除时间（毫秒时间戳，0=未删除）',

    UNIQUE KEY uk_user_currency_type (user_id, currency_id, wallet_type, delete_at),
    INDEX idx_user_id (user_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户钱包账户表';
```

### 索引说明

```
uk_user_currency_type:
  · 唯一索引，保证 一个用户 + 一个币种 + 一种钱包类型 只有一行
  · delete_at 加入唯一索引是为了支持软删除后重建（0=未删除时唯一，删除后不冲突）
  · 这个索引同时覆盖了最常用的查询: WHERE user_id=? AND currency_id=? AND wallet_type=?

idx_user_id:
  · 支持"查询某用户所有钱包"的场景（如管理员查看、币种切换时懒创建检查）
```

### 关键操作与此表的关系

```
操作              对此表的影响
────             ──────────

SwitchCurrency    懒创建: 如果 (user_id, currency_id, type=1) 不存在则 INSERT
Credit            UPDATE balance = balance + amount WHERE id=? (行锁内)
Debit             UPDATE balance = balance - amount WHERE id=? (行锁内，需检查可用余额≥amount)
Freeze            UPDATE frozen_amount = frozen_amount + amount WHERE id=? (行锁内)
Unfreeze          UPDATE frozen_amount = frozen_amount - amount WHERE id=? (行锁内)
提现成功扣款      UPDATE balance = balance - amount, frozen_amount = frozen_amount - amount
Exchange          同一事务内操作 3 行（扣法币中心 + 加BSB中心 + 加BSB奖励）
```

## 3.2 wallet_flow — 钱包流水记录表

### 业务来源

```
产品原型映射:
  · 交易记录页 → 奖励记录tab中的流水明细
  · 每一笔余额变动都写一条流水（充值/提现/兑换/奖励/冻结/解冻/人工加减/消费/稽核转入）
  · 流水是 append-only 的，只写不改不删

双重职责:
  ① 审计: 记录 balance_before → balance_after，可用于对账
  ② 幂等: uk_order_wallet 唯一索引，同一 order_no + wallet_type 不能重复写入
```

### DDL

```sql
CREATE TABLE wallet_flow (
    id              BIGINT       NOT NULL PRIMARY KEY                COMMENT '主键（雪花ID）',
    user_id         BIGINT       NOT NULL                            COMMENT '用户ID',
    currency_id     BIGINT       NOT NULL                            COMMENT '币种ID',
    wallet_type     TINYINT      NOT NULL                            COMMENT '钱包类型: 1=中心 2=奖励 3=主播',
    flow_type       TINYINT      NOT NULL                            COMMENT '流水类型: 1=充值 2=提现 3=兑换出 4=兑换入 5=奖励 6=冻结 7=解冻 8=人工加 9=人工减 10=消费 11=稽核转入 12=补单',
    amount          BIGINT       NOT NULL                            COMMENT '变动金额（正数=增加 负数=减少）',
    balance_before  BIGINT       NOT NULL DEFAULT 0                  COMMENT '变动前余额',
    balance_after   BIGINT       NOT NULL DEFAULT 0                  COMMENT '变动后余额',
    frozen_before   BIGINT       NOT NULL DEFAULT 0                  COMMENT '变动前冻结金额（冻结/解冻操作时记录）',
    frozen_after    BIGINT       NOT NULL DEFAULT 0                  COMMENT '变动后冻结金额',
    order_no        VARCHAR(32)  NOT NULL                            COMMENT '关联业务订单号（幂等key的一部分）',
    biz_type        VARCHAR(32)  NOT NULL DEFAULT ''                 COMMENT '业务类型标识（扩展用）',
    remark          VARCHAR(256) NOT NULL DEFAULT ''                 COMMENT '备注',
    create_at       BIGINT       NOT NULL DEFAULT 0                  COMMENT '创建时间（毫秒时间戳）',

    UNIQUE KEY uk_order_wallet (order_no, wallet_type),
    INDEX idx_user_currency_time (user_id, currency_id, create_at),
    INDEX idx_user_flow_type (user_id, flow_type, create_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='钱包流水记录表';
```

### 索引说明

```
uk_order_wallet:
  · 幂等核心: 同一业务订单号 + 同一钱包类型，只允许一条流水
  · 场景: 财务模块重复调用 Credit(orderNo="C123", walletType=1)
    → 第二次 INSERT 因唯一索引冲突失败 → 说明已处理 → 返回成功（幂等）
  · 兑换订单产生 3 条流水，order_no 相同但 wallet_type 不同（法币中心=1, BSB中心=1, BSB奖励=2）
    → 因为 currency_id 不同所以实际指向不同钱包行，但唯一索引只看 order_no + wallet_type
    → 注意: 兑换的 BSB中心(type=1) 和法币中心(type=1) 虽然 wallet_type 相同，
      但 order_no 可以通过后缀区分（如 E123-OUT, E123-IN, E123-BONUS），
      或者 wallet_type 维度改为包含 currency_id

  ★ 设计调整: 为了兑换场景更清晰，唯一索引改为:
    UNIQUE KEY uk_order_flow (order_no, wallet_type, flow_type)
    这样同一订单的"兑换出(type=3)"和"兑换入(type=4)"不会冲突

idx_user_currency_time:
  · 支持"查询某用户某币种的流水列表"（记录页分页查询）

idx_user_flow_type:
  · 支持"查询某用户某类型的流水"（如只看充值流水、只看奖励流水）
```

### 设计要点

```
① 只写不改不删（append-only）
   · 没有 update_by/update_at/delete_at 字段
   · 流水一旦写入就是永久记录
   · 如果需要"冲正"，写一条反向流水（如误加款后写一条减款流水）

② frozen_before / frozen_after 字段
   · 只在 flow_type=6(冻结) 和 flow_type=7(解冻) 时有意义
   · 其他类型时这两个字段为 0
   · 用于审计冻结金额的变动轨迹

③ 一笔兑换产生多条流水
   · 法币中心钱包: flow_type=3(兑换出), amount=-N
   · BSB中心钱包:  flow_type=4(兑换入), amount=+M
   · BSB奖励钱包:  flow_type=5(奖励), amount=+B
   · 三条流水的 order_no 相同，通过 flow_type 区分
```

## 4.2 withdraw_order — 提现订单表

### 业务来源

```
产品原型映射:
  · 提现页 → KYC校验 → 选择提现方式 → 输入金额 + 收款信息 → 提交
  · 提现审核: 风控 → 财务 → 出款（三级审核，财务模块负责）
  · 交易记录页 → 提现记录tab
  · B端财务管理 → 提现记录列表 → 24个展示字段

订单号格式: T + 16位随机数

状态流转:
  审核中(0) ──全部通过+出款成功──→ 已完成(1)
           ──任一级拒绝─────────→ 已拒绝(2)
           ──出款失败───────────→ 出款失败(3)
           ──用户/系统取消──────→ 已取消(4)
           ──管理员锁定────────→ 已锁定(5) ──解锁──→ 审核中(0)

关键规则:
  · 提交时冻结金额（从可用余额中扣出，放入冻结池）
  · 完成时从冻结金额中扣减（不是从可用余额扣）
  · 拒绝/失败时解冻归还
  · 收款信息需要加密存储（敏感数据）
```

## 4.3 exchange_order — 兑换订单表

### 业务来源

```
产品原型映射:
  · BSB兑换页 → 选择法币 → 输入金额 → 显示汇率和预估BSB → 确认兑换
  · 交易记录页 → 兑换记录tab

  · 汇率快照: 使用下单时刻的平台汇率，不是查询时的
```

### 字段设计说明

```

  · 整数存储，无精度问题

  · 提现汇率 = 实时汇率 × (1 - withdraw_float_rate / 10000)

限额字段（min_deposit / max_withdraw 等）:
  · 产品原型中提现限额要求: 单笔限额 + 每日限额
  · 充值限额由财务模块的充值方式配置控制，但币种级别也可以有上限
  · 0 = 无限制，简化业务逻辑判断

create_by_name / update_by_name:
  · B端管理表，参照 Banner 等表的模式，带操作人名称
  · 方便B端列表直接展示"创建人"列，无需额外查询
```

## 5.2 currency_rate — 汇率数据表

### 业务来源

```
产品原型映射:
  · 四层汇率体系:
    实时汇率 → 从3个第三方API取平均值，分钟级更新
    平台汇率 → 基于实时汇率，偏差超阈值时更新
    充值汇率 → 实时汇率 × (1 + 充值浮动%)
    提现汇率 → 实时汇率 × (1 - 提现浮动%)

  · 每个法币币种一行记录（1:1 关联 currency_config）
  · BSB 和 USDT 不需要汇率记录（BSB锚定=10:1 USDT，USDT是基准）
```

### DDL

```sql

    id              BIGINT       NOT NULL PRIMARY KEY                COMMENT '主键（雪花ID）',
    currency_id     BIGINT       NOT NULL                            COMMENT '币种ID（关联 currency_config.id）',
    realtime_rate   VARCHAR(32)  NOT NULL DEFAULT '0'                COMMENT '实时汇率（3个API平均值，币种/USDT）',
    platform_rate   VARCHAR(32)  NOT NULL DEFAULT '0'                COMMENT '平台汇率（超阈值时从实时汇率更新）',
    deposit_rate    VARCHAR(32)  NOT NULL DEFAULT '0'                COMMENT '充值汇率（实时×(1+浮动%)）',
    withdraw_rate   VARCHAR(32)  NOT NULL DEFAULT '0'                COMMENT '提现汇率（实时×(1-浮动%)）',
    rate_source_1   VARCHAR(32)  NOT NULL DEFAULT '0'                COMMENT 'API源1的汇率值',
    rate_source_2   VARCHAR(32)  NOT NULL DEFAULT '0'                COMMENT 'API源2的汇率值',
    rate_source_3   VARCHAR(32)  NOT NULL DEFAULT '0'                COMMENT 'API源3的汇率值',
    last_sync_at    BIGINT       NOT NULL DEFAULT 0                  COMMENT '最近一次同步时间（毫秒时间戳）',
    update_at       BIGINT       NOT NULL DEFAULT 0                  COMMENT '更新时间（毫秒时间戳）',

    UNIQUE KEY uk_currency (currency_id)
```

### 设计要点

```

  · 如果某个源异常可以快速定位

last_sync_at:
  · 区分于 update_at，记录的是"最近一次成功同步第三方汇率"的时间
  · 如果长时间未同步（第三方全部不可用），可以用此字段告警

  · 这张表由定时任务自动维护
  · 初始化时随 currency_config 一起创建
  · 不存在人工创建场景
```

# 九、关键设计决策汇总

## 9.1 金额为什么用 BIGINT 而不是 DECIMAL

```
决策: 所有金额字段使用 BIGINT，以最小货币单位的整数存储

对比:
┌──────────────┬─────────────────┬───────────────────┐
│ 方案          │ BIGINT 整数      │ DECIMAL(20,8)     │
├──────────────┼─────────────────┼───────────────────┤
│ 存储          │ 固定 8 字节     │ 可变长度            │
│ 计算性能      │ 整数运算，极快   │ 十进制运算，较慢     │
│ 精度          │ 无精度问题       │ 无精度问题          │
│ 比较/排序     │ 原生高效         │ 需额外处理          │
│ 索引性能      │ 好              │ 一般                │
│ 业界参照      │ Stripe/支付宝/  │ 部分银行系统         │
│              │ 微信支付         │                    │
│ Go 类型       │ int64           │ shopspring/decimal  │
│ IDL 类型      │ i64 或 string   │ string             │
│ 转换复杂度    │ 展示时除以10^N  │ 无需转换             │
├──────────────┼─────────────────┼───────────────────┤
│ 本项目选择    │ ★ 采用          │ 不采用              │
└──────────────┴─────────────────┴───────────────────┘

核心转换逻辑（每个币种一个 decimals 值）:
  展示值 = 数据库值 / 10^decimals
  数据库值 = 展示值 × 10^decimals

示例:
  用户充值 100.50 THB (decimals=2)
  → 数据库存储: 10050
  → IDL 传输: "10050"(i64) 或 "100.50"(string)
  → 前端展示: 100.50 THB
```

## 9.2 汇率为什么用 VARCHAR 而不是数值类型

```
决策: 汇率字段使用 VARCHAR(32) 存储

理由:
  · 汇率精度需求极高（如 1 VND = 0.0000000395 USDT）
  · DECIMAL 的精度需要预先指定，不同币种精度差异大
  · VARCHAR 存储原始精度，无截断风险
  · Go 内部使用 shopspring/decimal 计算，结果 .String() 存入 DB
  · 读取时 decimal.NewFromString(dbValue) 恢复精度

不用 DOUBLE/FLOAT 的理由:
  · 浮点数存在精度丢失（0.1 + 0.2 ≠ 0.3）
  · 金融场景的硬性要求: 不允许任何精度误差
```

## 9.3 流水表幂等索引为什么选 (order_no, wallet_type, flow_type)

```
决策: wallet_flow 的唯一索引为 (order_no, wallet_type, flow_type)

而不是简单的 (order_no):
  · 一笔兑换订单产生 3 条流水（扣法币、加BSB中心、加BSB奖励）
  · 3 条流水的 order_no 相同
  · 需要通过 wallet_type + flow_type 区分

而不是 (order_no, wallet_type):
  · 兑换时，源法币中心钱包 wallet_type=1，目标BSB中心钱包 wallet_type=1
  · 仅 wallet_type 不足以区分
  · 加入 flow_type 后: (E123, 1, 3=兑换出) vs (E123, 1, 4=兑换入) 不冲突
```

## 9.4 为什么 user_wallet 不用自增主键

```
决策: 所有表主键使用雪花 ID（BIGINT），不使用 AUTO_INCREMENT

理由:
  · 项目统一规范: 所有现有服务都使用 GenerateSnowflakeID()
  · TiDB 环境: AUTO_INCREMENT 在分布式数据库中可能有性能问题
  · 雪花 ID 自带时间排序能力
  · 无需数据库层面的自增锁
```

## 9.5 软删除字段为什么是 BIGINT 0 而不是 GORM 的 DeletedAt

```
决策: delete_at BIGINT NOT NULL DEFAULT 0

而不是 GORM 内置的 gorm.DeletedAt (time.Time):
  · 项目统一规范: 所有现有表（Banner、Item 等）都使用 BIGINT 时间戳
  · 0 = 未删除，非 0 = 删除时间的毫秒时间戳
  · 可以参与唯一索引（如 uk_symbol(symbol, delete_at)）
  · NULL 不能有效参与唯一索引约束，而 0 可以
```
