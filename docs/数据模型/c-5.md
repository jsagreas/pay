# 一、设计规范 — 遵循现有工程惯例

以下规范从 ser-item、ser-buser、ser-app 等模块的 15 个 GORM 模型中提取，所有新表严格遵循。

## 1.1 字段类型规范

```
类型映射关系（Go ↔ MySQL/TiDB）：

  Go int64              → BIGINT          主键ID、用户ID、金额、时间戳
  Go int32              → INT             状态、枚举、排序权重等小数值
  Go string             → VARCHAR(N)      币种代码、订单号、备注等
  Go *int64             → BIGINT NULL     可选的ID、可选的金额
  Go *int32             → INT NULL        可选的状态/枚举
  Go *string            → VARCHAR NULL    可选的字符串
```

**规则**：
- **必填字段**用非指针类型（`int64`、`string`），GORM 标签加 `not null`
- **可选字段**用指针类型（`*int64`、`*string`），GORM 标签不加 `not null`
- **金额字段**一律 `int64` 对应 `BIGINT`，存储最小单位整数（无 DECIMAL）
- **时间戳字段**一律 `int64`，存储毫秒级时间戳（由 UserPlugin 自动填充）

依据：`ser-item/internal/gorm_gen/model/item.gen.go` — `CreateAt int64`、`UpdateAt int64`；
`ser-buser/internal/gorm_gen/model/sys_user.gen.go` — 同样模式。

## 1.2 GORM 标签规范

```go
// 主键
ID int64 `gorm:"column:id;primaryKey;autoIncrement:true;comment:主键ID" json:"id"`

// 必填字段
FieldName int64 `gorm:"column:field_name;not null;comment:中文说明" json:"field_name"`

// 可选字段
FieldName *string `gorm:"column:field_name;comment:中文说明" json:"field_name"`

// 有默认值的字段
FieldName int64 `gorm:"column:field_name;not null;default:0;comment:中文说明" json:"field_name"`
```

**规则**：
- `column:xxx` 用蛇形命名
- `json:"xxx"` 与 column 保持一致
- `comment:` 写中文说明（生成的 DDL 中会带到 MySQL COMMENT）
- 主键统一 `primaryKey;autoIncrement:true`

## 1.3 审计字段规范

```go
// 所有表统一携带以下审计字段（由 UserPlugin 自动填充）
CreateBy *int64 `gorm:"column:create_by;comment:创建人ID" json:"create_by"`
CreateAt int64  `gorm:"column:create_at;not null;comment:创建时间（毫秒时间戳）" json:"create_at"`
UpdateBy *int64 `gorm:"column:update_by;comment:更新人ID" json:"update_by"`
UpdateAt int64  `gorm:"column:update_at;not null;comment:更新时间（毫秒时间戳）" json:"update_at"`
```

**规则**：
- 不手动赋值，由 `common/pkg/db/user_plugin.go` 在 GORM Create/Update 回调中自动注入
- C 端操作的 CreateBy/UpdateBy 来自 `tracer.GetUserId(ctx)`
- B 端操作的 CreateBy/UpdateBy 来自 `tracer.GetAdminId(ctx)`

## 1.4 软删除规范

```go
// 需要软删除的表加以下字段
DeleteAt int64  `gorm:"column:delete_at;not null;default:0;comment:删除时间（0=未删除）" json:"delete_at"`
DeleteBy *int64 `gorm:"column:delete_by;comment:删除人ID" json:"delete_by"`
```

**规则**：
- `delete_at = 0` 表示未删除，非零值表示删除时间戳
- 不是所有表都需要软删除 — 配置表需要（可恢复），流水/订单表不需要（不允许删除）

## 1.5 表命名规范

```
表名用蛇形小写，不加 t_ 前缀
Go 常量: const TableNameXxx = "xxx"
Go 方法: func (*Xxx) TableName() string { return TableNameXxx }
```

依据：`ser-item` 表名 `item`、`item_publish`；`ser-buser` 表名 `sys_user`、`sys_role`。

## 1.6 ID 生成规范

```
主键 ID：数据库自增（autoIncrement:true）
订单号：雪花算法生成（utils.GenerateSnowflakeID()）
```

依据：`common/pkg/utils/snowflake.go` 提供全局雪花 ID 生成。

# 三、第一类：配置表

## 3.1 currency_config — 币种配置表

### 业务来源

需求文档"币种配置模块"中的 B 端币种管理页面。运营人员配置每个币种的基本信息、充值参数、提现参数、状态等。这些参数直接影响 C 端行为（限额、精度、是否可用）。

对应需求要素：
- 基本信息（名称/代码/图标/精度） ← 产品原型币种配置页面
- 充值配置（限额/启用） ← 产品原型充值参数区域
- 提现配置（限额/手续费/T+N） ← 产品原型提现参数区域
- 状态控制（启用/禁用） ← 产品原型状态开关
- 排序 ← 产品原型排序字段

### Go Struct

```go
const TableNameCurrencyConfig = "currency_config"

type CurrencyConfig struct {

    Icon           *string `gorm:"column:icon;comment:币种图标URL" json:"icon"`
    Precision      int32   `gorm:"column:precision;not null;default:2;comment:小数精度（小数位数）" json:"precision"`
    DepositMin     int64   `gorm:"column:deposit_min;not null;default:0;comment:充值单笔最低限额（最小单位）" json:"deposit_min"`
    DepositMax     int64   `gorm:"column:deposit_max;not null;default:0;comment:充值单笔最高限额（最小单位）" json:"deposit_max"`
    DepositEnable  int32   `gorm:"column:deposit_enable;not null;default:1;comment:充值启用状态 0=禁用 1=启用" json:"deposit_enable"`
    WithdrawMin    int64   `gorm:"column:withdraw_min;not null;default:0;comment:提现单笔最低限额（最小单位）" json:"withdraw_min"`
    WithdrawMax    int64   `gorm:"column:withdraw_max;not null;default:0;comment:提现单笔最高限额（最小单位）" json:"withdraw_max"`
```

### DDL

```sql
CREATE TABLE currency_config (
    id               BIGINT       PRIMARY KEY AUTO_INCREMENT COMMENT '主键ID',
    currency_code    VARCHAR(10)  NOT NULL COMMENT '币种代码（BSB/VND/IDR/THB/USDT）',
    currency_name    VARCHAR(50)  NOT NULL COMMENT '币种名称',
    currency_type    INT          NOT NULL COMMENT '币种类型 1=法币 2=加密货币 3=平台币',
    icon             VARCHAR(256)          COMMENT '币种图标URL',
    `precision`      INT          NOT NULL DEFAULT 2 COMMENT '小数精度（小数位数）',
    deposit_min      BIGINT       NOT NULL DEFAULT 0 COMMENT '充值单笔最低限额（最小单位）',
    deposit_max      BIGINT       NOT NULL DEFAULT 0 COMMENT '充值单笔最高限额（最小单位）',
    deposit_enable   INT          NOT NULL DEFAULT 1 COMMENT '充值启用状态 0=禁用 1=启用',
    withdraw_min     BIGINT       NOT NULL DEFAULT 0 COMMENT '提现单笔最低限额（最小单位）',
    withdraw_max     BIGINT       NOT NULL DEFAULT 0 COMMENT '提现单笔最高限额（最小单位）',
    withdraw_daily   BIGINT       NOT NULL DEFAULT 0 COMMENT '提现每日总额上限（最小单位）',
    fee_type         INT          NOT NULL DEFAULT 0 COMMENT '手续费类型 0=无 1=固定金额 2=百分比',
    fee_value        BIGINT       NOT NULL DEFAULT 0 COMMENT '手续费值（固定=最小单位金额 百分比=万分比）',
    arrive_days      INT          NOT NULL DEFAULT 0 COMMENT '提现到账天数（T+N的N）',
    withdraw_enable  INT          NOT NULL DEFAULT 1 COMMENT '提现启用状态 0=禁用 1=启用',
    sort_order       INT          NOT NULL DEFAULT 0 COMMENT '排序权重（越大越靠前）',
    status           INT          NOT NULL DEFAULT 1 COMMENT '币种状态 0=禁用 1=启用',
    create_by        BIGINT                COMMENT '创建人ID',
    create_at        BIGINT       NOT NULL COMMENT '创建时间',
    update_by        BIGINT                COMMENT '更新人ID',
    update_at        BIGINT       NOT NULL COMMENT '更新时间',
    delete_at        BIGINT       NOT NULL DEFAULT 0 COMMENT '删除时间（0=未删除）',
    delete_by        BIGINT                COMMENT '删除人ID',
    UNIQUE KEY uk_currency_code (currency_code, delete_at)
) ENGINE=InnoDB COMMENT='币种配置表';
```

### 字段说明

| 字段 | 业务含义 | 设计说明 |
|------|---------|---------|
| currency_code | 币种唯一标识 | 与 delete_at 联合唯一，支持删除后重建同代码币种 |
| precision | 小数精度 | VND=0（无小数），BSB=2（两位小数），USDT=2。决定 BIGINT ↔ 展示金额的换算倍数：`展示值 = BIGINT值 / 10^precision` |
| deposit_min/max | 充值限额 | 存储最小单位整数。例 VND precision=0：min=100000 表示最低 100,000 VND |
| fee_type + fee_value | 手续费规则 | 两个字段组合表达：type=1 时 value 是固定金额（最小单位）；type=2 时 value 是万分比（如 50 = 0.5%） |
| arrive_days | T+N | 提现到账天数，0 表示即时到账 |
| status | 全局启禁用 | 禁用后 C 端该币种完全不可见、不可操作 |

### 索引策略

```
uk_currency_code (currency_code, delete_at)  → 唯一键，币种代码不重复
  加 delete_at 是为了支持软删除后重建同代码币种

无需额外索引：
  - B端管理查询一般全量加载（币种数量极少，<10条），不需要按条件索引
  - C端读取走 Redis 缓存，缓存未命中时全量查 DB 重建
```

## 3.2 exchange_rate — 汇率配置表

### 业务来源

需求文档"汇率体系"中的汇率管理功能。格式统一为"1 BSB = X 法币"。运营手动配置，不对接外部实时汇率。

与 currency_config 分表的理由：
- 汇率变更频率高于币种基本配置（运营可能每天调汇率，但很少改币种信息）
- 汇率日志只需记录汇率变更，不需要夹带币种配置字段的变更
- 分开后缓存粒度更细（汇率缓存 TTL 可以更短）

### Go Struct

```go
const TableNameExchangeRate = "exchange_rate"

type ExchangeRate struct {
    ID           int64  `gorm:"column:id;primaryKey;autoIncrement:true;comment:主键ID" json:"id"`
    CurrencyCode string `gorm:"column:currency_code;not null;comment:币种代码" json:"currency_code"`
    Rate         int64  `gorm:"column:rate;not null;comment:汇率值（1 BSB = rate/10^rate_precision 该币种）" json:"rate"`
    RatePrecision int32 `gorm:"column:rate_precision;not null;default:4;comment:汇率精度（小数位数）" json:"rate_precision"`
    GiftRatio    int32  `gorm:"column:gift_ratio;not null;default:0;comment:兑换赠送比例（万分比，如500=5%）" json:"gift_ratio"`
    Status       int32  `gorm:"column:status;not null;default:1;comment:状态 0=禁用 1=启用" json:"status"`
    CreateBy     *int64 `gorm:"column:create_by;comment:创建人ID" json:"create_by"`
    CreateAt     int64  `gorm:"column:create_at;not null;comment:创建时间" json:"create_at"`
    UpdateBy     *int64 `gorm:"column:update_by;comment:更新人ID" json:"update_by"`
    UpdateAt     int64  `gorm:"column:update_at;not null;comment:更新时间" json:"update_at"`
}

func (*ExchangeRate) TableName() string { return TableNameExchangeRate }
```

### DDL

```sql
CREATE TABLE exchange_rate (
    id              BIGINT       PRIMARY KEY AUTO_INCREMENT COMMENT '主键ID',
    currency_code   VARCHAR(10)  NOT NULL COMMENT '币种代码',
    rate            BIGINT       NOT NULL COMMENT '汇率值（1 BSB = rate/10^rate_precision 该币种）',
    rate_precision  INT          NOT NULL DEFAULT 4 COMMENT '汇率精度（小数位数）',
    gift_ratio      INT          NOT NULL DEFAULT 0 COMMENT '兑换赠送比例（万分比，如500=5%）',
    status          INT          NOT NULL DEFAULT 1 COMMENT '状态 0=禁用 1=启用',
    create_by       BIGINT                COMMENT '创建人ID',
    create_at       BIGINT       NOT NULL COMMENT '创建时间',
    update_by       BIGINT                COMMENT '更新人ID',
    update_at       BIGINT       NOT NULL COMMENT '更新时间',
    UNIQUE KEY uk_currency_code (currency_code)
) ENGINE=InnoDB COMMENT='汇率配置表';
```

### 字段说明

| 字段 | 业务含义 | 设计说明 |
|------|---------|---------|
| rate | 汇率数值 | 用 BIGINT 存储，避免浮点精度问题。实际汇率 = rate / 10^rate_precision。例：1 BSB = 23000 VND → rate=230000000, rate_precision=4 → 230000000/10000=23000 |
| rate_precision | 汇率精度 | 不同币种汇率的有效小数位不同（VND 是整数，USDT 可能需要4位小数）。这个字段让汇率存储精度可配 |
| gift_ratio | 兑换赠送比例 | 需求"兑换送5%"对应 gift_ratio=500（万分之500）。赠送金额进奖励钱包。用万分比而非百分比是为了支持更精细的配置（如 0.5% = 50） |
| status | 汇率启禁用 | 禁用后该币种无法兑换。注意：这里不需要软删除，汇率记录不删除只禁用 |

### 为什么汇率也用 BIGINT 而不用 DECIMAL

```
理由同金额字段：
  - 数据库层整数比较和运算更快
  - 无浮点精度问题（23000.50 VND/BSB 不会出现 23000.4999999）
  - 与金额字段统一用整数运算，计算过程全整数链路

汇率换算公式（全整数）：
  BSB金额(最小单位) = 法币金额(最小单位) × 10^rate_precision / rate
  法币金额(最小单位) = BSB金额(最小单位) × rate / 10^rate_precision

  举例：1000 VND 兑换 BSB
  rate=230000000, rate_precision=4
  BSB = 1000 × 10000 / 230000000 = 10000000 / 230000000 ≈ 0（整数截断丢失精度）

  修正：实际应先放大再除，使用 Go 的 big.Int 或调整运算顺序避免溢出和截断
  这个细节在编码时处理，数据模型层面只需确保存储类型是 BIGINT
```

## 3.3 currency_log — 币种操作日志表

### 业务来源

需求文档"操作行为全部记录操作日志"。B 端对币种配置的每一次修改操作都要记录：谁、什么时间、改了什么字段、改前值、改后值。

### Go Struct

```go
const TableNameCurrencyLog = "currency_log"

type CurrencyLog struct {
    ID           int64   `gorm:"column:id;primaryKey;autoIncrement:true;comment:主键ID" json:"id"`
    CurrencyCode string  `gorm:"column:currency_code;not null;comment:币种代码" json:"currency_code"`
    Action       int32   `gorm:"column:action;not null;comment:操作类型 1=新增 2=编辑 3=启用 4=禁用 5=删除" json:"action"`
    FieldName    *string `gorm:"column:field_name;comment:变更字段名" json:"field_name"`
    OldValue     *string `gorm:"column:old_value;comment:变更前值" json:"old_value"`
    NewValue     *string `gorm:"column:new_value;comment:变更后值" json:"new_value"`
    Remark       *string `gorm:"column:remark;comment:备注" json:"remark"`
    OperatorId   int64   `gorm:"column:operator_id;not null;comment:操作人ID" json:"operator_id"`
    OperatorName *string `gorm:"column:operator_name;comment:操作人名称" json:"operator_name"`
    CreateAt     int64   `gorm:"column:create_at;not null;comment:操作时间" json:"create_at"`
}

func (*CurrencyLog) TableName() string { return TableNameCurrencyLog }
```

### DDL

```sql
CREATE TABLE currency_log (
    id             BIGINT       PRIMARY KEY AUTO_INCREMENT COMMENT '主键ID',
    currency_code  VARCHAR(10)  NOT NULL COMMENT '币种代码',
    action         INT          NOT NULL COMMENT '操作类型 1=新增 2=编辑 3=启用 4=禁用 5=删除',
    field_name     VARCHAR(50)           COMMENT '变更字段名',
    old_value      VARCHAR(256)          COMMENT '变更前值',
    new_value      VARCHAR(256)          COMMENT '变更后值',
    remark         VARCHAR(256)          COMMENT '备注',
    operator_id    BIGINT       NOT NULL COMMENT '操作人ID',
    operator_name  VARCHAR(50)           COMMENT '操作人名称',
    create_at      BIGINT       NOT NULL COMMENT '操作时间',
    INDEX idx_currency_code (currency_code),
    INDEX idx_create_at (create_at)
) ENGINE=InnoDB COMMENT='币种操作日志表';
```

### 设计说明

- **纯追加（Append-Only）**：日志只写不改不删，没有 update/delete 字段
- **operator_name 冗余存储**：避免查日志时还要 RPC 查用户名。操作人改名不影响历史日志的展示
- **一次编辑多字段变更写多条记录**：每个变更字段一条 log。也可选择把变更内容 JSON 化存一条，视 B 端查询需求而定，这个细节编码时确认

## 3.4 rate_log — 汇率变更日志表

### 业务来源

## 4.1 user_balance — 用户余额表（整个系统的心脏）

### 业务来源

需求文档"四个子钱包 × 多币种 = 余额矩阵"。这是整个钱包系统最核心的数据模型。

映射关系：
```
现实世界                          →  数据模型
─────────────────────────────────────────────────
用户张三                          →  user_id = 10086
张三的中心钱包                     →  wallet_type = 1
```

```sql
    UNIQUE KEY uk_user_currency_wallet (user_id, currency, wallet_type)
) ENGINE=InnoDB COMMENT='用户余额表';
```

### 核心设计决策

**为什么金额用 BIGINT 不用 DECIMAL**

```
存储规则（参照 currency_config.precision）：

  币种    precision    存储1000.50的方式        BIGINT值
  ────────────────────────────────────────────────
  BSB       2         1000.50 × 10^2         = 100050
  VND       0         1000000                = 1000000
  USDT      2         1000.50 × 10^2         = 100050
  IDR       0         1000000                = 1000000

展示还原：
  展示值 = BIGINT值 / 10^precision
  100050 / 100 = 1000.50 BSB ✓
  1000000 / 1 = 1,000,000 VND ✓
```

**为什么 available 和 frozen 不分表**

提现冻结操作需要原子地 `available -= amount, frozen += amount`，同一行更新最简单最安全。解冻反向操作同理。如果分表需要跨表事务，复杂度暴增但收益为零。

**为什么保留 version 字段**

当前主方案是行级乐观锁（`WHERE available >= amount`），不依赖 version。但保留 version 字段，以备未来更复杂的并发场景（如需要"读-改-写"模式时可切换为 version 乐观锁）。字段存着不用只占 4 字节，不碍事。

**为什么没有 delete_at**

余额记录不允许删除。用户注册后不管是否使用钱包，余额行始终存在（余额为0也是有效状态）。

### 索引策略

```
uk_user_currency_wallet (user_id, currency, wallet_type)
  → 所有余额查询和更新的必经之路
  → 覆盖全场景：
    查某用户所有余额：WHERE user_id = ?（走联合索引前缀）
    查某用户某币种余额：WHERE user_id = ? AND currency = ?（走联合索引前缀）
    更新精确匹配：WHERE user_id = ? AND currency = ? AND wallet_type = ?（走唯一键）

不需要额外索引 — 余额表查询永远走唯一键或其前缀
```

```sql
    UNIQUE KEY uk_order_type (ref_order_no, change_type)
) ENGINE=InnoDB COMMENT='余额变动流水表';
```

### 核心设计决策

**before_balance + after_balance 的审计价值**

```
连续性校验：任意两条相邻流水（同 user_id + currency + wallet_type）
  上一条的 after_balance 必须等于下一条的 before_balance

  flow_1: before=0,     amount=+100, after=100      ✓ (充值入账)
  flow_2: before=100,   amount=-80,  after=20       ✓ (兑换扣减, 100→20)
  flow_3: before=20,    amount=+50,  after=70       ✓ (人工加款, 20→70)

如果出现 flow_3.before ≠ flow_2.after，说明有异常（遗漏流水或并发bug）
这是钱包系统最基本的对账手段
```

**uk_order_type 唯一键的幂等兜底作用**

```
唯一键：(ref_order_no, change_type)

含义：同一个订单号 + 同一种变动类型只能有一条流水

场景：充值入账（change_type=1），同一个充值订单号只能入账一次
  第一次写入 → 成功
  第二次写入（重复回调）→ 唯一键冲突 → 被数据库拦截

这是幂等的第二层防护（第一层是应用层状态判断）
```

**为什么没有 update/delete 字段**

流水是**不可变记录**。一旦写入不允许修改和删除。如果写错了，只能追加一条"冲正"流水来纠正，不能改历史记录。这是财务系统的基本原则。

### 索引策略

```
idx_user_currency (user_id, currency)
  → C端按用户+币种查记录列表（"我的充值记录""我的兑换记录"）
  → 也覆盖对账查询：查某用户某币种的全部流水

idx_ref_order (ref_order_no)
  → 通过订单号反查流水（运营排查问题时用）

idx_create_at (create_at)
  → 按时间范围查询（B端统计、导出时用）

uk_order_type (ref_order_no, change_type)
  → 幂等兜底唯一键
```

### 设计说明

**为什么汇率和赠送比例要快照存储**

```
兑换订单创建时的汇率和赠送比例锁定在订单记录中，不引用 exchange_rate 表的当前值。

理由：
  - 运营随时可能调整汇率和赠送比例
  - 历史订单必须按创建时的汇率展示和审计
  - 如果引用当前值，汇率变化后历史订单的金额逻辑就对不上了
  - rate_precision 也要快照，因为精度可能调整（虽然可能性低）
```

**兑换订单在事务中的角色**

```
一次兑换操作在事务内完成 3~4 件事：
  1. 扣减源币种中心钱包 available
  2. 增加 BSB 中心钱包 available
  3. 增加 BSB 奖励钱包 available（如有赠送）
  4. 写入兑换订单记录（本表）
  5. 写入余额变动流水（balance_flow 2~3 条）

全部在同一个 DB 事务内，要么全成功要么全回滚
```

**account_no AES 加密存储**

```
银行卡号和电子钱包 ID 属于敏感信息，不能明文存储。
使用 common/pkg/utils 已有的 AES 加密工具：
  写入时：加密后存储
  读取时：解密后返回
加密密钥从 ETCD 配置中心获取

依据：架构设计文档安全架构章节明确要求
```

**唯一键 uk_user_currency_method**

```

场景：张三的 VND 银行转账账户只有一条，修改时覆盖更新

当前阶段按需求"后续自动回显上次信息"，一条就够
```

```
可选方案：把银行卡信息 JSON 化存一个字段
  {"bank_name":"XXX", "account_no":"加密值", ...}

不选的理由：
  - 加密字段不方便在 JSON 内单独处理
  - 查询时无法按字段索引（如果未来需要按银行名查）
  - 显式字段更符合工程现有模式（没有一个 GORM model 用 JSON 存储结构化数据）
```

# 九、从数据模型到下一步 — 你应该做什么

## 9.1 建库建表

```
1. 在 TiDB 创建数据库 ser_wallet
2. 执行上面 10 张表的 DDL
3. ETCD 配置 /slg/serv/wallet/db = "ser_wallet"
```

## 9.2 GORM Gen 生成代码

```
1. 在 ser-wallet/internal/gen/gorm_gen.go 中配置表名列表
2. 运行 go run internal/gen/gorm_gen.go
3. 自动生成：
   - internal/gorm_gen/model/*.gen.go   → 就是上面的 Go Struct（自动生成的会完全匹配表结构）
   - internal/gorm_gen/query/*.gen.go   → 类型安全的查询 API
   - internal/gorm_gen/repo/*.gen.go    → 20+ 标准 CRUD 方法

4. 生成后的 model 文件是只读的（不手改）
5. 需要扩展的查询在 internal/rep/ 目录手写
```

## 9.3 枚举定义

```
在 ser-wallet/internal/enum/enums.go 中定义上面第六节的所有枚举常量
在 IDL 的 common_types.thrift 或 wallet.thrift 中同步定义 enum
保持 Go 代码和 IDL 的枚举值完全一致
```

## 9.4 开始编码的顺序

```
数据模型确认 → 建库建表 → GORM Gen → IDL 定义 → gen_kitex → 骨架搭建 → 币种CRUD → ...

参照实现思路文档的四期开发计划
```

# 十、数据模型决策清单

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 金额存储 | BIGINT 最小单位 | 无精度风险 + 整数运算快 + 工程风格一致 |
| 汇率存储 | BIGINT + precision | 同金额，避免浮点，precision 可配 |
| 手续费比例 | 万分比（int32） | 比百分比更精细，支持 0.5% = 50 |
| 赠送比例 | 万分比（int32） | 同手续费 |
| 时间字段 | int64 毫秒时间戳 | 工程 UserPlugin 统一写入 |
| 敏感信息 | AES 加密 VARCHAR | 银行卡号/钱包地址加密存储 |
| 软删除 | 仅配置表有 | 订单/流水/余额不允许删除 |
| 外键约束 | 不使用 | 微服务架构，应用层保证一致性 |
| 分表 | 当前不做 | 单表到千万级再考虑 |
| 唯一键含 delete_at | 仅 currency_config | 支持删除后重建同代码币种 |
