## 一、映射思路：怎么从产品需求推导出数据模型

### 1.1 推导方法

产品原型和需求文档描述的是**现实世界的行为和规则**，数据模型是**代码世界的承载容器**。两者之间的映射遵循三个步骤：

```
第一步：识别实体
  产品文档中反复出现的"名词" → 数据库表
  例："用户的钱包余额" → wallet 表
       "充值订单" → deposit_order 表
       "币种配置" → currency_config 表

第二步：提取属性
  产品原型的页面字段 + 需求文档的业务规则 → 表的列
  例：充值记录详情页显示"订单号、金额、状态、时间、方式" → 这5个就是 deposit_order 的核心字段
       需求说"余额不能是负数" → wallet 表 balance 字段加 CHECK 或应用层约束

第三步：建立关系
  产品文档中的"动词"描述了实体间的关系
  例："用户充值产生订单" → wallet 和 deposit_order 通过 user_id 关联
       "充值成功后余额增加" → deposit_order 的终态触发 wallet_transaction 的写入
```

### 1.2 我识别出的全部实体

从136张产品原型和6份分析文档中，识别出以下**必须建模**的实体：

| 序号 | 现实概念 | 数据模型（表名） | 来源依据 |
|------|---------|---------------|---------|
| 1 | 平台支持的币种及其汇率 | `currency_config` | 需求 B3（18个配置字段）+ B4（汇率规则） |
| 2 | 汇率每次变更的历史记录 | `rate_change_log` | 需求 B4（汇率日志页） |
| 3 | 用户在某个币种某个子钱包下的余额 | `wallet` | 需求 B2（5种子钱包）+ A2（币种隔离） |
| 4 | 每一笔余额变动的流水记录 | `wallet_transaction` | 需求 A6（4类记录）+ A7（对账一致性） |
| 5 | 用户发起的充值订单 | `deposit_order` | 需求 A3（充值模块）+ E3（充值流程） |
| 6 | 用户发起的提现订单 | `withdrawal_order` | 需求 A5（提现模块）+ E4（提现流程） |
| 7 | 用户保存的提现账户信息 | `withdrawal_account` | 需求 A5（提现账户管理）+ D9（自动填充） |
| 8 | 用户发起的兑换订单 | `exchange_order` | 需求 A4（兑换模块）+ D7（兑换BSB页） |

### 1.3 为什么是这8张表

**逐一论证**：

1. **currency_config** —— 币种配置需求 B3 有18个配置字段，汇率规则 B4 有4种汇率。币种和汇率是1:1关系（每个币种只有一套当前汇率），合一张表（架构决策 AD-10）。
2. **rate_change_log** —— 产品原型 C3 有"汇率日志"Tab页，需求 B4 明确"偏差≥阈值时生成汇率变更日志"。这是只增不改的历史表。
3. **wallet** —— 需求 B2 定义了5种子钱包体系，每个用户+每个币种+每个钱包类型 = 一条余额记录。这是资金系统的核心。
4. **wallet_transaction** —— 需求 A7 要求"对账一致性"，架构决策 AD-5 要求记录 balance_before/balance_after。每次余额变动必须有对应流水，C端记录页（D10）的数据源。
5. **deposit_order** —— 充值是一个有状态的异步流程（待支付→进行中→已完成/已失败/已关闭），需要独立表追踪生命周期。C端充值记录详情（D10）需要订单级信息。
6. **withdrawal_order** —— 提现有三级审核（风控→财务→出款），状态流转最复杂（处理中→审核中→出款中→已完成/已失败/已拒绝），且涉及冻结/解冻/扣款状态机。
7. **withdrawal_account** —— 需求 A5 明确"首次提现填写银行信息，后续自动填充"。用户可保存多个提现账户（按币种），需要独立表管理。
8. **exchange_order** —— 兑换虽然是同步操作，但C端有独立的"兑换记录"Tab（D10），且兑换涉及跨币种操作（源币种→目标币种），需要记录汇率快照、赠送金额等兑换特有信息。wallet_transaction 记录余额变动，exchange_order 记录业务详情。

> 依据：实现思路 c-4 §三 表结构设计思路 + 架构设计 c-4 §二 数据架构

### 2.3 交易类型 (tx_type)

来源：流程思路 c-4 §五 + C端记录页4个Tab（充值/提现/兑换/奖励）

| 值 | 名称 | 触发来源 | 金额方向 |
|----|------|---------|---------|
| 1 | 充值 | W-R2 CreditDeposit | + 增加 |
| 2 | 兑换扣出 | W-9 ExecuteExchange | - 减少（源币种） |
| 3 | 兑换转入 | W-9 ExecuteExchange | + 增加（目标币种） |
| 4 | 提现冻结 | W-11 / W-R3 | - 减少（可用→冻结） |
| 5 | 提现解冻 | W-R4 UnfreezeForWithdrawal | + 增加（冻结→可用） |
| 6 | 提现扣款 | W-R5 ConfirmWithdrawalDeduct | - 减少（冻结额扣除） |
| 7 | 人工加款 | W-R6 ManualCredit | + 增加 |
| 8 | 人工减款 | W-R7 ManualDebit | - 减少 |
| 9 | 补单 | W-R8 CreditCorrection | + 增加 |
| 10 | 奖金 | W-R2（充值奖金）/ W-9（兑换赠送） | + 增加（奖励钱包） |

```go
const (
    TxTypeDeposit          = 1  // 充值
    TxTypeExchangeOut      = 2  // 兑换扣出
    TxTypeExchangeIn       = 3  // 兑换转入
    TxTypeWithdrawFreeze   = 4  // 提现冻结
    TxTypeWithdrawUnfreeze = 5  // 提现解冻
    TxTypeWithdrawDeduct   = 6  // 提现扣款
    TxTypeManualCredit     = 7  // 人工加款
    TxTypeManualDebit      = 8  // 人工减款
    TxTypeCorrection       = 9  // 补单
    TxTypeBonus            = 10 // 奖金/赠送
)
```

**设计说明**：兑换拆成"扣出"和"转入"两条流水（因为涉及两个币种两个钱包，每条流水必须归属到一个确定的 user_id + currency_code + wallet_type）。提现拆成"冻结""解冻""扣款"三种类型（对应状态机的三个阶段，每个阶段都有余额变动需要记录）。

### 2.4 充值订单状态 (deposit_status)

来源：流程思路 c-4 §八 订单状态机

| 值 | 名称 | 颜色 | 说明 |
|----|------|------|------|
| 1 | 待支付 | 默认 | 订单已创建，等待用户支付 |
| 2 | 进行中 | 橙色 | 用户已发起支付，等待回调 |
| 3 | 已完成 | 绿色 | 支付成功，余额已入账 |
| 4 | 已失败 | 红色 | 支付失败 |
| 5 | 已关闭 | 灰色 | 超时未支付，自动关闭 |

```
状态流转：
  1(待支付) → 2(进行中) → 3(已完成)
                        → 4(已失败)
  1(待支付) → 5(已关闭)
```

### 2.5 提现订单状态 (withdrawal_status)

来源：流程思路 c-4 §八 提现订单状态流转

| 值 | 名称 | 资金状态 | 说明 |
|----|------|---------|------|
| 1 | 处理中 | 已冻结 | 订单已创建，金额已冻结 |
| 2 | 审核中 | 已冻结 | 进入财务审核队列 |
| 3 | 出款中 | 已冻结 | 审核通过，通道出款中 |
| 4 | 已完成 | 已扣除 | 出款成功，冻结金额永久扣除 |
| 5 | 已失败 | 已返还 | 出款失败，冻结金额返还 |
| 6 | 已拒绝 | 已返还 | 审核拒绝，冻结金额返还 |

```
状态流转：
  1(处理中) → 2(审核中) → 3(出款中) → 4(已完成)
                                     → 5(已失败)
                        → 6(已拒绝)

终态：4(已完成) = 资金永久移出
     5(已失败) / 6(已拒绝) = 资金返还用户
```

## 三、数据模型详细定义

### 3.1 currency_config —— 币种配置（含汇率）

#### 现实世界含义

**这张表回答一个问题：平台支持哪些币种，每个币种的规则和当前汇率是什么？**

运营人员在B端后台管理币种（产品原型 C1/C2 币种配置列表页+编辑弹窗，18个字段）。汇率定时从三方API同步（需求 B4）。所有涉及金额展示、汇率换算的地方都读这张表。

#### 字段定义

| 字段名 | 类型 | 约束 | 现实含义 | 来源 |
|--------|------|------|---------|------|
| id | bigint unsigned | PK, AUTO_INCREMENT | 主键 | 工程规范 |
| currency_code | varchar(16) | NOT NULL | 币种代码（BSB/VND/IDR/THB/USDT） | 原型 C2 字段1 |
| currency_name | varchar(64) | NOT NULL | 币种名称（越南盾/印尼盾/...） | 原型 C2 字段2 |
| currency_type | tinyint unsigned | NOT NULL | 币种类型（1法定/2加密/3平台） | 原型 C2 字段3 |
| country_code | varchar(16) | DEFAULT '' | 关联国家代码（VN/ID/TH） | 原型 C2 字段4 |
| icon_url | varchar(512) | DEFAULT '' | 币种图标URL（SVG/PNG，≤5KB） | 原型 C2 字段5 |
| currency_symbol | varchar(16) | NOT NULL | 货币符号（₫/Rp/฿/$） | 原型 C2 字段6 |
| thousands_separator | varchar(4) | NOT NULL DEFAULT ',' | 千分位符号（VND/IDR用点号，THB/BSB用逗号） | 需求 A2 金额展示差异 |
| decimal_separator | varchar(4) | NOT NULL DEFAULT '.' | 小数点符号 | 需求 A2 |
| decimal_digits | tinyint unsigned | NOT NULL DEFAULT 2 | 小数位数（法币2位，加密8位） | 需求 A2 + B4 精度规则 |

| deviation_threshold | int unsigned | NOT NULL DEFAULT 0 | 偏差阈值百分比（× 100，如 300 = 3.00%） | 原型 C2 字段 |
| sort_value | int unsigned | NOT NULL DEFAULT 1000 | 排序值 | 原型 C1 表格排序列 |
| status | tinyint unsigned | NOT NULL DEFAULT 1 | 状态（1启用/2禁用） | 原型 C1 状态列 |
| create_by | bigint unsigned | DEFAULT 0 | 创建人ID | 工程规范 |
| update_by | bigint unsigned | DEFAULT 0 | 更新人ID | 工程规范 |
| create_at | bigint unsigned | NOT NULL DEFAULT 0 | 创建时间戳 | 工程规范 |
| update_at | bigint unsigned | NOT NULL DEFAULT 0 | 更新时间戳 | 工程规范 |
| delete_at | bigint unsigned | NOT NULL DEFAULT 0 | 删除时间戳（0=未删除） | 工程规范 |

#### CREATE TABLE SQL

```sql
CREATE TABLE `currency_config` (
    `id`                   bigint unsigned NOT NULL AUTO_INCREMENT COMMENT '主键ID',
    `currency_code`        varchar(16) NOT NULL COMMENT '币种代码(BSB/VND/IDR/THB/USDT)',
    `currency_name`        varchar(64) NOT NULL COMMENT '币种名称',
    `currency_type`        tinyint unsigned NOT NULL COMMENT '币种类型 1法定 2加密 3平台',
    `country_code`         varchar(16) DEFAULT '' COMMENT '国家代码(VN/ID/TH)',
    `icon_url`             varchar(512) DEFAULT '' COMMENT '币种图标URL',
    `currency_symbol`      varchar(16) NOT NULL COMMENT '货币符号(₫/Rp/฿/$)',
    `thousands_separator`  varchar(4) NOT NULL DEFAULT ',' COMMENT '千分位符号',
    `decimal_separator`    varchar(4) NOT NULL DEFAULT '.' COMMENT '小数点符号',
    `decimal_digits`       tinyint unsigned NOT NULL DEFAULT 2 COMMENT '小数位数(法币2/加密8)',
    `unit_factor`          bigint unsigned NOT NULL DEFAULT 100 COMMENT '最小单位因子(法币100/加密100000000)',
    `is_base`              tinyint unsigned NOT NULL DEFAULT 0 COMMENT '是否基准币种 0否 1是',
    `rate_realtime`        bigint unsigned NOT NULL DEFAULT 0 COMMENT '实时汇率(×10^8)',
    `rate_platform`        bigint unsigned NOT NULL DEFAULT 0 COMMENT '平台汇率(×10^8)',
    `rate_deposit`         bigint unsigned NOT NULL DEFAULT 0 COMMENT '入款汇率(×10^8)',
    `rate_withdrawal`      bigint unsigned NOT NULL DEFAULT 0 COMMENT '出款汇率(×10^8)',
    `rate_float_percent`   int unsigned NOT NULL DEFAULT 0 COMMENT '汇率浮动%(×100, 150=1.50%)',
    `deviation_threshold`  int unsigned NOT NULL DEFAULT 0 COMMENT '偏差阈值%(×100, 300=3.00%)',
    `sort_value`           int unsigned NOT NULL DEFAULT 1000 COMMENT '排序值',
    `status`               tinyint unsigned NOT NULL DEFAULT 1 COMMENT '状态 1启用 2禁用',
    `create_by`            bigint unsigned DEFAULT 0 COMMENT '创建人ID',
    `update_by`            bigint unsigned DEFAULT 0 COMMENT '更新人ID',
    `create_at`            bigint unsigned NOT NULL DEFAULT 0 COMMENT '创建时间',
    `update_at`            bigint unsigned NOT NULL DEFAULT 0 COMMENT '更新时间',
    `delete_at`            bigint unsigned NOT NULL DEFAULT 0 COMMENT '删除时间(0=未删除)',
    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_currency_code` (`currency_code`, `delete_at`),
    KEY `idx_status` (`status`),
    KEY `idx_is_base` (`is_base`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_bin COMMENT='币种配置(含汇率)';
```

#### 索引说明

- `uk_currency_code`：币种代码全局唯一，加 delete_at 允许软删除后重建。CC-2 CreateCurrency 校验重复用。
- `idx_status`：C端查询已启用币种列表（W-2）、B端按状态筛选（CC-5）。
- `idx_is_base`：快速定位基准币种（CC-R3）。

#### 关键约束

- **全局只有一条 is_base=1 的记录**，应用层保证（SetBaseCurrency 校验已设置则拒绝）。
- **汇率放大 10^8 存整数**，避免浮点精度问题。例：1 USDT = 25432.50 VND → rate_platform = 2543250000000。
- **rate_float_percent 和 deviation_threshold 放大 100 存整数**（2位小数精度）。

#### 数据量预估

极少，4-6条（BSB/VND/IDR/THB/USDT）。全量缓存在 Redis 中。

### 3.2 rate_change_log —— 汇率变更日志

#### 现实世界含义

**运营在B端的"汇率日志"Tab页看到的每一条记录。** 每当定时任务检测到实时汇率偏差超过阈值并更新平台汇率时，就写一条日志。

#### 字段定义

```sql
    `total_withdrawal` bigint NOT NULL DEFAULT 0 COMMENT '累计提现(最小单位)',
    `fund_source`      tinyint unsigned NOT NULL DEFAULT 0 COMMENT '资金来源 0未知 1法币 2USDT 3混合',

    `create_by`        bigint unsigned DEFAULT 0 COMMENT '创建人ID',

    KEY `idx_user_id` (`user_id`)

```

#### 索引说明

- `uk_user_curr_type`：**核心唯一键**——保证一个用户+一个币种+一个钱包类型只有一条记录。也是 BalanceEngine 中 `SELECT ... FOR UPDATE` 的锁定目标。
- `idx_user_id`：W-R1 GetUserWalletBalance 按用户查所有子钱包余额。

#### 关键设计点

**`balance` 用 bigint（有符号）而非 bigint unsigned**：虽然余额逻辑上不应为负，但使用有符号类型可以在应用层检测到异常（如余额变负说明有Bug），而 unsigned 在计算时可能因溢出导致隐蔽错误。

**`frozen_amount` 独立字段**：提现冻结时 `balance -= amount, frozen_amount += amount`；出款成功时 `frozen_amount -= amount`；出款失败时 `frozen_amount -= amount, balance += amount`。任意时刻 `balance + frozen_amount = 用户在该钱包的实际资产`。

**`fund_source` 资金来源追踪**：实现"法币充→法币提，USDT充→USDT提"的合规规则（需求 A5）。用户首次充值时标记来源，后续充值如果方式不同则标记为"混合"。提现时根据此字段决定允许的提现方式。

**`version` 乐观锁**：BalanceEngine 更新余额时 `UPDATE ... SET balance=?, version=version+1 WHERE id=? AND version=?`，affected=0 说明并发冲突需重试。

**`total_deposit / total_withdrawal` 累计统计**：用于提现限额校验（每日已提现额度）和数据统计。这是冗余字段（可以从流水表计算），但高频查询时避免了聚合计算的开销。

#### 数据量预估

用户数 × 币种数 × 活跃钱包类型数。假设10万用户平均使用2个币种、2个钱包类型 → 约40万条。

#### 对账逻辑

```
对账验证公式：
  每条流水：balance_after == balance_before + amount  ✓
  最新流水的 balance_after == wallet.balance           ✓
  所有流水的 SUM(amount) == wallet.balance - 初始余额   ✓
```

#### 数据量预估

这是增长最快的表。假设10万用户每天平均5笔操作 → 每天50万条 → 每年约1.8亿条。未来可能需要按时间分表或归档历史数据。

#### 索引说明

- `uk_order_no`：订单号唯一。
- `uk_snapshot_id`：**快照ID唯一**——保证一个汇率快照只能执行一次兑换，这是兑换的幂等防线。
- `idx_user_create`：C端兑换记录列表按时间倒序分页。

#### 兑换无状态

兑换是同步操作，执行成功就是成功（写入记录），失败就回滚（不写入）。不存在中间状态，所以没有 status 字段。记录存在即代表兑换成功。

## 六、每个业务场景写哪些表

理清"一次操作影响哪些数据"——这是后续编码时事务边界的依据。

### 6.1 充值成功（W-R2 CreditDeposit）

```
一个事务内：
  1. UPDATE wallet SET balance += 充值额 WHERE user+currency+中心钱包  (addBalance)
  2. INSERT wallet_transaction (充值流水, 中心钱包)
  3. 如果有奖金：
     UPDATE wallet SET balance += 奖金额 WHERE user+currency+奖励钱包  (addBalance)
     INSERT wallet_transaction (奖金流水, 奖励钱包)
  4. UPDATE deposit_order SET status = 已完成
```

### 6.2 兑换执行（W-9 ExecuteExchange）

```
一个事务内：
  1. UPDATE wallet SET balance -= 法币额 WHERE user+法币+中心钱包     (deductBalance)
  2. INSERT wallet_transaction (兑换扣出流水, 法币中心)
  3. UPDATE wallet SET balance += BSB额 WHERE user+BSB+中心钱包       (addBalance)
  4. INSERT wallet_transaction (兑换转入流水, BSB中心)
  5. 如果有赠送：
     UPDATE wallet SET balance += 赠送额 WHERE user+BSB+奖励钱包     (addBalance)
     INSERT wallet_transaction (奖金流水, BSB奖励)
  6. INSERT exchange_order (兑换业务详情)
```

### 6.3 提现创建（W-11 CreateWithdrawalOrder）

```
一个事务内：
  1. UPDATE wallet SET balance -= 额, frozen_amount += 额 WHERE user+currency+中心钱包  (freezeBalance)
  2. INSERT wallet_transaction (提现冻结流水)
  3. INSERT withdrawal_order (状态=处理中)
```

### 6.4 提现成功（W-R5 ConfirmWithdrawalDeduct）

```
一个事务内：
  1. UPDATE wallet SET frozen_amount -= 额 WHERE user+currency+中心钱包  (confirmDeduct)
  2. INSERT wallet_transaction (提现扣款流水)
  3. UPDATE withdrawal_order SET status = 已完成
```

### 6.5 提现失败/拒绝（W-R4 UnfreezeForWithdrawal）

```
一个事务内：
  1. UPDATE wallet SET frozen_amount -= 额, balance += 额 WHERE user+currency+中心钱包  (unfreezeBalance)
  2. INSERT wallet_transaction (提现解冻流水)
  3. UPDATE withdrawal_order SET status = 已失败/已拒绝
```

### 6.6 人工加款（W-R6 ManualCredit）

```
一个事务内，对每个有金额的子钱包：
  1. UPDATE wallet SET balance += 额 WHERE user+currency+walletType  (addBalance)
  2. INSERT wallet_transaction (加款流水)
```

### 6.7 人工减款（W-R7 ManualDebit）

```
一个事务内，对每个有金额的子钱包：
  1. UPDATE wallet SET balance -= min(额, 当前余额) WHERE ...  (deductBestEffort)
  2. INSERT wallet_transaction (减款流水, 记录实际扣减额)
```

### 6.8 补单（W-R8 CreditCorrection）

```
一个事务内：
  1. UPDATE wallet SET balance += 补单额 WHERE user+currency+中心钱包  (addBalance)
  2. INSERT wallet_transaction (补单流水, 中心钱包)
  3. 如果有赠送：
     UPDATE wallet SET balance += 赠送额 WHERE user+currency+奖励钱包  (addBalance)
     INSERT wallet_transaction (奖金流水, 奖励钱包)
```

## 七、字段设计决策说明

### 7.1 为什么金额用 bigint 不用 decimal

| 维度 | bigint | decimal |
|------|--------|---------|
| 精度 | 无损（整数运算） | 也无损（定点数） |
| Go映射 | int64，原生运算 | 需要第三方 decimal 库 |
| TiDB性能 | 索引高效 | 索引效率略低 |
| 业界实践 | 支付宝/微信支付/Stripe | 传统银行系统 |
| 展示处理 | 除以 unit_factor 再格式化 | 直接格式化 |

结论：**bigint 更适合互联网金融系统**，计算快、无精度陷阱、与 Go 原生类型对齐。

> 依据：架构决策 AD-2

### 7.2 为什么汇率放大 10^8

汇率是两个币种的比率，必然有小数。例如 1 USDT = 25432.50 VND。如果直接用 float64/double 存储，会有浮点精度丢失。放大 10^8 后存为整数 2543250000000，计算时 `result = amount * rate / 100000000`，全程整数运算。

选择 10^8 是因为加密货币精度到 8 位小数（BTC 的 satoshi），统一用 10^8 可以覆盖所有币种精度。

### 7.3 为什么 wallet_transaction 拆分兑换为两条流水

兑换是跨币种操作：扣VND（法币中心钱包）+ 加BSB（BSB中心钱包）。每条流水必须归属到一个确定的 `user_id + currency_code + wallet_type`，所以拆成"兑换扣出"（VND中心）和"兑换转入"（BSB中心）两条。这样每条流水的 balance_before/balance_after 都是对应钱包的准确值，对账时才能逐条校验。

### 7.4 为什么需要 exchange_order 独立表

wallet_transaction 记录的是**余额变动事实**（谁的余额变了多少），而 exchange_order 记录的是**业务详情**（源币种、目标币种、汇率、赠送比例、稽核要求）。一笔兑换产生 2-3 条流水（扣法币 + 加BSB + 可能加奖励），但业务上是一笔操作，需要一个地方存完整的业务信息。

这和充值/提现的模式一致：deposit_order 存业务详情，wallet_transaction 存余额变动。

### 7.5 为什么 wallet 表有 fund_source 字段

需求 A5 提现规则："法币充的钱→只能法币提，USDT充的钱→只能USDT提，混合→法币提"。提现时需要知道用户的资金来源。

两种实现方式：
- A. 每次提现时查 deposit_order 历史记录推导 → 查询成本高
- B. 在 wallet 表维护冗余的 fund_source 标记 → 充值时更新，提现时直接读

选择方案B。充值入账时（W-R2）更新规则：
- 首次充值：fund_source = 充值方式对应值（法币=1，USDT=2）
- 再次充值：如果新方式和已有不同 → fund_source = 3（混合）

## 九、数据模型总结

### 一句话概括

```
8张表 = 1张配置表(currency_config) + 1张日志表(rate_change_log)
       + 1张余额表(wallet) + 1张流水表(wallet_transaction)
       + 3张订单表(deposit/withdrawal/exchange_order)
       + 1张账户表(withdrawal_account)
```

### 表的分类

| 类别 | 表 | 特征 |
|------|-----|------|
| **配置类**（低频写、高频读） | currency_config | 运营管理，全量缓存 |
| **日志类**（只增不改） | rate_change_log | 审计追溯 |
| **核心账务类**（高频读写、强一致） | wallet, wallet_transaction | BalanceEngine 统一操作 |
| **业务订单类**（有状态机） | deposit_order, withdrawal_order | 生命周期管理 |
| **业务记录类**（写入即终态） | exchange_order | 同步操作，无状态机 |
| **用户信息类**（低频写） | withdrawal_account | 敏感数据加密 |

### 数据量级预估

| 表 | 预估量级 | 增长模式 |
|----|---------|---------|
| currency_config | 个位数 | 几乎不增长 |
| rate_change_log | 万/年 | 缓慢增长 |
| wallet | 十万~百万 | 随用户增长 |
| wallet_transaction | **亿/年** | 最快增长，未来可能需归档 |
| deposit_order | 百万/年 | 随业务增长 |
| withdrawal_order | 百万/年 | 随业务增长 |
| exchange_order | 百万/年 | 随业务增长 |
| withdrawal_account | 十万 | 随用户增长 |
