# 数据模型设计（从产品需求到代码载体的完整映射）

> 来源：需求理解 + 接口评估 + 架构设计 + 实现思路 + 工程认知 + 流程思路 + 现有工程模型实证
> 定位：把产品需求中的"现实世界描述"映射为可直接落地的"数据模型"，作为需求层到代码层的桥梁
> 原则：每个模型都有产品需求依据 + 工程规范依据；遵循现有工程的字段约定和命名惯例

---

## 一、映射思路：从现实世界到数据模型

### 1.1 为什么需要数据模型

产品需求描述的是"用户做什么"（充值、提现、兑换），代码实现的是"数据怎么流转"。两者之间需要一个**载体**把现实概念抽象为可存储、可查询、可计算的结构——这就是数据模型。

### 1.2 映射关系全景

```
产品概念（现实世界）                 数据模型（代码世界）
─────────────────────────────────────────────────────────
币种（VND/IDR/THB/BSB/USDT）    →  Currency 模型
汇率（实时/平台/入出款）           →  Currency 字段 + ExchangeRateLog 模型
用户的钱包余额                    →  WalletAccount 模型
每笔资金变动记录                  →  TransactionLog 模型
充值订单                         →  DepositOrder 模型
提现订单                         →  WithdrawOrder 模型
兑换操作                         →  ExchangeRecord 模型
绑定的银行卡/电子钱包              →  WithdrawAccount 模型

钱包类型（中心/奖励/主播/代理）    →  WalletType 枚举
币种类型（法币/加密/平台）         →  CurrencyType 枚举
订单状态（待支付/成功/失败）       →  各 Status 枚举
操作类型（入账/扣款/冻结/解冻）    →  OpType 枚举
交易类型（充值/提现/兑换/奖励）    →  TxType 枚举
```

### 1.3 模型分类

按职责域将数据模型分为四类：

| 分类 | 模型 | 职责 |
|------|------|------|
| **配置域** | Currency, ExchangeRateLog | 基础数据，变化频率低，被所有业务依赖 |
| **账务域** | WalletAccount, TransactionLog | 资金核心，强一致性要求，是整个系统的心脏 |
| **订单域** | DepositOrder, WithdrawOrder, ExchangeRecord | 业务过程记录，有状态机流转 |
| **账户域** | WithdrawAccount | 用户绑定的外部收款信息 |

## 二、工程约定（数据模型的通用规范）

以下规范来自现有工程的实际代码（ser-app、ser-item、ser-buser 模型实证），ser-wallet 必须遵循。

### 2.1 字段类型约定

| 约定项 | 规范 | 依据 |
|--------|------|------|
| 主键 | `id bigint NOT NULL AUTO_INCREMENT` → Go: `int64` | 全部现有模型一致 |
| 时间字段 | `bigint NOT NULL`，存毫秒时间戳 → Go: `int64` | ser-app Banner/HomeCategory 模型 |
| 软删除 | `delete_at bigint NOT NULL DEFAULT 0`，0=未删除 | ser-app/ser-item 模型 |
| 金额字段 | `bigint NOT NULL DEFAULT 0`，存最小单位整数 | 架构设计 ADR-5 决策 |
| 审计字段 | `create_by`/`create_by_name`/`create_at`/`update_by`/`update_by_name`/`update_at` | ser-app Banner 模型 + UserPlugin 自动填充 |
| 状态字段 | `tinyint NOT NULL DEFAULT 0` 或 `int NOT NULL DEFAULT 0` | 全部现有模型 |
| 字符串字段 | `varchar(N)` + `*string`（可空）或 `string`（非空） | GORM Gen FieldNullable=true |
| 枚举存储 | `tinyint` 或 `int`，应用层定义常量 | ser-app/ser-buser |

### 2.2 命名约定

| 约定项 | 规范 | 示例 |
|--------|------|------|
| 表名 | 全小写下划线 | `wallet_account`, `deposit_order` |
| 列名 | 全小写下划线 | `user_id`, `currency_id`, `available_balance` |
| Go结构体 | 大驼峰 | `WalletAccount`, `DepositOrder` |
| Go字段 | 大驼峰 | `UserID`, `CurrencyID`, `AvailableBalance` |
| GORM tag | `gorm:"column:xxx;..."` + `json:"xxx"` | 与列名一致 |
| 唯一索引 | `uk_` 前缀 | `uk_user_currency_type` |
| 普通索引 | `idx_` 前缀 | `idx_user_currency_time` |

### 2.3 审计字段自动填充机制

```
写入时（UserPlugin 回调自动执行）：
  CREATE → 自动填充 create_by, create_by_name, create_at, update_by, update_by_name, update_at
  UPDATE → 自动填充 update_by, update_by_name, update_at

来源：context 中的 tracer.GetUserId(ctx) 和 tracer.GetUserName(ctx)
时间：time.Now().UnixMilli()
```

**注意**：RPC 调用场景（如财务调钱包的 CreditBalance），context 中可能没有 B 端用户信息，此时 create_by 填入系统标识或操作发起方信息，需要在业务代码中手动设置。

### 2.4 GORM 代码生成约定

```
模型不手写，全部由 GORM Gen 从数据库表结构自动生成：
  1. 先在 TiDB 中 CREATE TABLE
  2. 执行 go run internal/gen/gorm_gen.go
  3. 自动产出 internal/gorm_gen/model/{table}.gen.go（模型结构体）
  4. 自动产出 internal/gorm_gen/query/{table}.gen.go（类型安全查询 DSL）
  5. 自动产出 internal/gorm_gen/repo/{table}r/{table}r.go（25个标准 CRUD 方法）

所以本文档定义的是"表结构"（DDL），Go 模型由 DDL 决定，自动生成。
```

#### 产品需求来源

| 金额展示规则（千分位/小数位） | 多币种钱包需求文档/4-1 |

#### 表结构定义

```sql
CREATE TABLE `currency` (
    `id`               BIGINT       NOT NULL AUTO_INCREMENT COMMENT '主键ID',

    -- 币种基本信息
    `currency_code`    VARCHAR(10)  NOT NULL COMMENT '币种代码（VND/IDR/THB/USDT/BSB）',
    `currency_name`    VARCHAR(50)  NOT NULL COMMENT '币种名称（越南盾/印尼盾/泰铢/...）',
    `currency_type`    TINYINT      NOT NULL COMMENT '币种类型：1法定货币 2加密货币 3平台货币',
    `country`          VARCHAR(50)  NOT NULL DEFAULT '' COMMENT '所属国家（越南/印尼/泰国/...）',
    `icon_url`         VARCHAR(512) NOT NULL DEFAULT '' COMMENT '币种图标URL（SVG/PNG，≤5KB）',

    -- 展示规则
    `symbol`           VARCHAR(10)  NOT NULL DEFAULT '' COMMENT '货币符号（d/Rp/฿/$/...）',
    `thousands_sep`    VARCHAR(5)   NOT NULL DEFAULT ',' COMMENT '千分位符号（逗号或点）',
    `decimal_sep`      VARCHAR(5)   NOT NULL DEFAULT '.' COMMENT '小数位符号（点或逗号）',
    `decimal_places`   TINYINT      NOT NULL DEFAULT 0 COMMENT '小数位数（VND=0, THB=2, BSB=2, USDT=8）',
    `multiplier`       BIGINT       NOT NULL DEFAULT 1 COMMENT '最小单位换算倍率（10^decimal_places）',

    -- 汇率相关（对基准币种 BSB）
    `real_time_rate`   BIGINT       NOT NULL DEFAULT 0 COMMENT '实时汇率（×10^8存储）',
    `platform_rate`    BIGINT       NOT NULL DEFAULT 0 COMMENT '平台汇率（×10^8存储）',
    `deposit_rate`     BIGINT       NOT NULL DEFAULT 0 COMMENT '入款汇率（×10^8存储）',
    `withdraw_rate`    BIGINT       NOT NULL DEFAULT 0 COMMENT '出款汇率（×10^8存储）',
    `rate_float_pct`   INT          NOT NULL DEFAULT 0 COMMENT '汇率浮动百分比（×100存储，如140=1.40%）',
    `threshold_pct`    INT          NOT NULL DEFAULT 0 COMMENT '汇率更新阈值百分比（×100存储）',

    -- 状态与排序
    `status`           TINYINT      NOT NULL DEFAULT 0 COMMENT '状态：0禁用 1启用',
    `sort_order`       INT          NOT NULL DEFAULT 0 COMMENT '排序值（越小越靠前）',

    -- 审计字段（UserPlugin 自动填充）
    `create_by`        BIGINT       NOT NULL DEFAULT 0 COMMENT '创建人ID',
    `create_by_name`   VARCHAR(64)  NOT NULL DEFAULT '' COMMENT '创建人名称',
    `create_at`        BIGINT       NOT NULL DEFAULT 0 COMMENT '创建时间（毫秒时间戳）',
    `update_by`        BIGINT       NOT NULL DEFAULT 0 COMMENT '更新人ID',
    `update_by_name`   VARCHAR(64)  NOT NULL DEFAULT '' COMMENT '更新人名称',
    `update_at`        BIGINT       NOT NULL DEFAULT 0 COMMENT '更新时间（毫秒时间戳）',
    `delete_at`        BIGINT       NOT NULL DEFAULT 0 COMMENT '删除时间（毫秒，0=未删除）',

    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_code` (`currency_code`, `delete_at`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='币种配置表';
```

#### 字段映射说明

| 产品概念 | 表字段 | 设计说明 |
|----------|--------|---------|
| 币种代码（VND/IDR/...） | `currency_code` | 唯一键，联合 delete_at 支持软删除后重建 |
| 币种名称 | `currency_name` | 展示用 |
| 法定/加密/平台 | `currency_type` | 枚举 1/2/3，对应 CurrencyType 常量 |
| 币种图标（SVG/PNG ≤5KB） | `icon_url` | 上传到 ser-s3 后存 URL |
| 千分位符号（VND用点，THB用逗号） | `thousands_sep` | C端格式化展示用 |
| 小数位数（VND=0，BSB=2，USDT=8） | `decimal_places` | 控制精度 |
| 最小单位换算倍率 | `multiplier` | VND=1, THB=100, BSB=100, USDT=100000000 |
| 实时汇率 | `real_time_rate` | ×10^8 存储，避免浮点精度问题 |
| 平台汇率 = 实时汇率（阈值触发更新） | `platform_rate` | 同上 |
| 入款汇率 = 平台汇率 × (1 + 浮动%) | `deposit_rate` | 同上 |
| 出款汇率 = 平台汇率 × (1 - 浮动%) | `withdraw_rate` | 同上 |
| 汇率浮动% | `rate_float_pct` | ×100 存储，1.40% → 存 140 |
| 阈值% | `threshold_pct` | ×100 存储 |
| 启用/禁用 | `status` | 禁用后 C 端不可用，但余额保留 |

#### 汇率存储精度说明

```
为什么汇率用 bigint ×10^8 存储：
  - 法币汇率如 BSB:VND = 1:71，精度要求小数点后2位 → ×10^8 绰绰有余
  - 加密货币汇率如 BSB:USDT = 10:1，精度要求小数点后8位 → ×10^8 刚好覆盖
  - 与金额字段统一用整数运算，全链路无浮点
  - 计算时：实际汇率 = rate_value / 100000000

举例：
  BSB:VND 实时汇率 = 71.00
  → real_time_rate = 7100000000（71 × 10^8）
  浮动 1.40%
  → rate_float_pct = 140
  入款汇率 = 71 × (1 + 0.014) = 71.994
  → deposit_rate = 7199400000
```

### 3.2 BaseCurrencyConfig（基准币种配置）

#### 产品需求来源

- 基准币种设置是一次性操作，提交后不可修改（币种配置需求文档/3-2）
- BSB 价值：10BSB = 1USDT，锚定 USDT

#### 设计方案

不独立建表，采用系统配置表或 ETCD 配置存储：

```
方案：ETCD 配置键
  /slg/serv/wallet/base_currency_code = "USDT"
  /slg/serv/wallet/base_currency_set  = "1"    ← 是否已设置，1=已设置不可改

理由：
  1. 只有一条数据，不值得建表
  2. 设置后不可修改，不需要历史记录
  3. ETCD 的 LoadAndWatch 机制支持热读取
  4. BSB 的汇率锚定（10BSB=1USDT）是硬编码逻辑，不需要存储
```

### 3.3 ExchangeRateLog（汇率变更日志）

#### 产品需求来源

| 需求点 | 来源 |
|--------|------|
| 汇率日志字段定义 | 币种配置需求文档/4-4 |
| 所有平台汇率更新后需生成日志 | 币种配置需求文档/4-2 |

#### 表结构定义

```sql
CREATE TABLE `exchange_rate_log` (
    `id`               BIGINT       NOT NULL AUTO_INCREMENT COMMENT '主键ID',

    -- 关联信息
    `currency_id`      BIGINT       NOT NULL COMMENT '关联币种ID',
    `currency_code`    VARCHAR(10)  NOT NULL COMMENT '币种代码（冗余，避免查询join）',

    -- 汇率快照
    `real_time_rate`   BIGINT       NOT NULL COMMENT '更新时的实时汇率（×10^8）',
    `platform_rate`    BIGINT       NOT NULL COMMENT '更新后的平台汇率（×10^8）',
    `deposit_rate`     BIGINT       NOT NULL COMMENT '更新后的入款汇率（×10^8）',
    `withdraw_rate`    BIGINT       NOT NULL COMMENT '更新后的出款汇率（×10^8）',
    `rate_float_pct`   INT          NOT NULL COMMENT '当时的汇率浮动百分比（×100）',
    `threshold_pct`    INT          NOT NULL COMMENT '当时的阈值百分比（×100）',
    `deviation_pct`    INT          NOT NULL COMMENT '触发更新时的实际偏差值（×100）',

    -- 触发来源
    `trigger_type`     TINYINT      NOT NULL DEFAULT 1 COMMENT '触发类型：1自动（定时任务） 2手动（后台操作）',

    -- 时间（只增不改，不需要 update 审计字段）
    `create_at`        BIGINT       NOT NULL DEFAULT 0 COMMENT '日志生成时间（毫秒时间戳）',

    PRIMARY KEY (`id`),
    INDEX `idx_currency_time` (`currency_id`, `create_at` DESC),
    INDEX `idx_code_time` (`currency_code`, `create_at` DESC)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='汇率变更日志表';
```

#### 设计说明

```
这是只增不改不删的日志表：
  - 每次平台汇率更新时写入一条记录
  - 冗余 currency_code 避免分页查询时 join currency 表
  - 冗余 rate_float_pct/threshold_pct 保留当时的配置快照
  - deviation_pct 记录实际偏差值，便于事后分析
  - trigger_type 区分自动更新和手动更新
  - 不需要 update/delete 审计字段
```

## 四、账务域模型（核心中的核心）

### 4.1 WalletAccount（钱包账户）

#### 产品需求来源

| 需求点 | 来源 |
|--------|------|
| 5类钱包（中心/奖励/主播/代理/场馆） | 币种配置需求文档/2-1 |
| 每个币种一套独立余额 | 多币种钱包需求文档/4-1 |
| 资金钱包余额 = 中心 + 奖励 | 币种配置需求文档/2-1 |
| 冻结余额（提现时） | 需求流程 — 提现冻结→确认/取消 |
| 消费扣款优先级：先中心后奖励 | 币种配置需求文档/2-1 |

#### 表结构定义

```sql
CREATE TABLE `wallet_account` (
    `id`                 BIGINT     NOT NULL AUTO_INCREMENT COMMENT '主键ID',

    -- 定位维度：用户 × 币种 × 钱包类型 = 一行
    `user_id`            BIGINT     NOT NULL COMMENT '用户ID',
    `currency_id`        BIGINT     NOT NULL COMMENT '关联币种ID',
    `wallet_type`        TINYINT    NOT NULL COMMENT '钱包类型：1中心 2奖励 3主播 4代理 5场馆',

    -- 余额（bigint 最小单位整数）
    `available_balance`  BIGINT     NOT NULL DEFAULT 0 COMMENT '可用余额（最小单位）',
    `frozen_balance`     BIGINT     NOT NULL DEFAULT 0 COMMENT '冻结余额（最小单位）',

    -- 版本号（预留，当前未使用，如果未来需要乐观锁可启用）
    `version`            INT        NOT NULL DEFAULT 0 COMMENT '版本号（预留）',

    -- 审计字段
    `create_by`          BIGINT     NOT NULL DEFAULT 0 COMMENT '创建人ID',
    `create_by_name`     VARCHAR(64) NOT NULL DEFAULT '' COMMENT '创建人名称',
    `create_at`          BIGINT     NOT NULL DEFAULT 0 COMMENT '创建时间（毫秒时间戳）',
    `update_by`          BIGINT     NOT NULL DEFAULT 0 COMMENT '更新人ID',
    `update_by_name`     VARCHAR(64) NOT NULL DEFAULT '' COMMENT '更新人名称',
    `update_at`          BIGINT     NOT NULL DEFAULT 0 COMMENT '更新时间（毫秒时间戳）',
    `delete_at`          BIGINT     NOT NULL DEFAULT 0 COMMENT '删除时间（毫秒，0=未删除）',

    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_user_currency_type` (`user_id`, `currency_id`, `wallet_type`, `delete_at`),
    INDEX `idx_user_id` (`user_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='钱包账户表';
```

#### 关键设计说明

```
1. 一行一账户原则：
   user_id + currency_id + wallet_type 唯一确定一个账户
   例如：用户123 的 VND 中心钱包 = 一行
         用户123 的 VND 奖励钱包 = 另一行
         用户123 的 BSB 中心钱包 = 又一行

2. 余额双字段设计：
   available_balance = 可用余额（可以花的钱）
   frozen_balance = 冻结余额（提现申请中，不可花不可提）
   产品展示的"钱包余额" = available_balance + frozen_balance
   但用户实际可操作的只有 available_balance

3. 并发控制：
   余额修改通过 SELECT ... FOR UPDATE 行锁（走 uk_user_currency_type 唯一索引）
   锁粒度 = 单个用户的单个币种的单个钱包类型 = 一行
   不同用户、不同币种之间完全不阻塞

4. 懒创建：
   用户首次操作某个币种时，如果账户不存在则自动创建（available=0, frozen=0）
   不需要注册时批量预创建全部币种的账户

5. 金额存储举例：
   用户有 1000 VND（decimal_places=0, multiplier=1）
   → available_balance = 1000

   用户有 3.50 THB（decimal_places=2, multiplier=100）
   → available_balance = 350

   用户有 1.00 BSB（decimal_places=2, multiplier=100）
   → available_balance = 100

   用户有 1.00000000 USDT（decimal_places=8, multiplier=10^8）
   → available_balance = 100000000
```

#### 产品概念到字段的映射

| 产品说法 | 对应数据 |
|----------|----------|
| 资金钱包余额 | SUM(available + frozen) WHERE wallet_type IN (1,2) AND currency_id=? |
| 中心钱包余额 | available_balance WHERE wallet_type=1 |
| 奖励钱包余额 | available_balance WHERE wallet_type=2 |
| 主播钱包余额 | available_balance WHERE wallet_type=3 |
| 所有币种余额概览 | GROUP BY currency_id，每个币种的 SUM(available + frozen) |
| 消费扣款优先级 | 业务逻辑控制：先扣 wallet_type=1，不够再扣 wallet_type=2 |

### 4.2 TransactionLog（交易流水）

#### 产品需求来源

| 需求点 | 来源 |
|--------|------|
| 完整可追溯的资金记录 | 需求目标第4条 |
| 对账一致性 | 非功能性需求 |
| 充值/提现/兑换/奖励 四类记录 | 多币种钱包需求文档/8-1 |
| 余额变更必须有流水 | 架构设计 — 流水必写约束 |
| 幂等键（ref_order_no） | 架构设计 — ADR-3 |

#### 表结构定义

```sql
CREATE TABLE `transaction_log` (
    `id`                 BIGINT       NOT NULL AUTO_INCREMENT COMMENT '主键ID',

    -- 关联维度
    `user_id`            BIGINT       NOT NULL COMMENT '用户ID',
    `currency_id`        BIGINT       NOT NULL COMMENT '币种ID',
    `wallet_type`        TINYINT      NOT NULL COMMENT '钱包类型：1中心 2奖励 3主播 4代理 5场馆',

    -- 操作信息
    `op_type`            TINYINT      NOT NULL COMMENT '操作类型：1入账 2扣款 3冻结 4解冻 5冻结扣款',
    `tx_type`            TINYINT      NOT NULL COMMENT '交易类型：1充值 2提现 3兑换入 4兑换出 5奖励 6人工加款 7人工减款 8投注扣款 9返奖 10冻结 11解冻 12冻结扣款',
    `amount`             BIGINT       NOT NULL COMMENT '变动金额（最小单位，正数）',
    `direction`          TINYINT      NOT NULL COMMENT '方向：1收入(+) 2支出(-)',

    -- 余额快照（变动后的余额，用于对账）
    `balance_after`      BIGINT       NOT NULL COMMENT '变动后可用余额（最小单位）',
    `frozen_after`       BIGINT       NOT NULL COMMENT '变动后冻结余额（最小单位）',

    -- 关联单号（幂等键）
    `ref_order_no`       VARCHAR(64)  NOT NULL COMMENT '关联业务单号（充值单号/提现单号/兑换单号/修正单号）',

    -- 描述
    `remark`             VARCHAR(255) NOT NULL DEFAULT '' COMMENT '备注说明',

    -- 时间（只增不改不删）
    `create_at`          BIGINT       NOT NULL DEFAULT 0 COMMENT '创建时间（毫秒时间戳）',

    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_ref_order_tx` (`ref_order_no`, `tx_type`, `wallet_type`),
    INDEX `idx_user_currency_time` (`user_id`, `currency_id`, `create_at` DESC),
    INDEX `idx_user_tx_type_time` (`user_id`, `tx_type`, `create_at` DESC)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='交易流水表';
```

#### 关键设计说明

```
1. 只增不改不删：
   这是审计和对账的数据根基，任何一条记录写入后永远不修改、不删除。
   没有 update_at / delete_at 字段。

2. 幂等键设计：
   uk_ref_order_tx (ref_order_no, tx_type, wallet_type)
   同一笔业务单号 + 同一种交易类型 + 同一个钱包类型 → 只允许写入一次
   重复写入时 DB 报唯一索引冲突 → 应用层捕获后视为幂等成功

   为什么包含 wallet_type：
   兑换操作一笔 ref_order_no 会同时影响源币种和 BSB 两个账户，
   tx_type 分别是"兑换出(4)"和"兑换入(3)"，但如果 BSB 有赠送部分进入奖励钱包，
   同一个 tx_type=兑换入 会写入中心和奖励两行，需要 wallet_type 区分。

3. direction 字段：
   amount 始终为正数，由 direction 决定正负：
   direction=1 (收入) → 余额增加
   direction=2 (支出) → 余额减少
   这比存负数更清晰，查询时不容易混淆。

4. balance_after/frozen_after：
   每笔流水记录变动后的余额快照。
   对账时可以按时间顺序回放：上一笔的 balance_after +/- 本笔 amount = 本笔 balance_after
   如果不等，说明数据不一致。

5. C端记录查询：
   充值记录 → WHERE tx_type=1 (充值) AND user_id=? AND currency_id=?
   提现记录 → WHERE tx_type IN (2,10,11,12) AND ...
   兑换记录 → WHERE tx_type IN (3,4) AND ...
   奖励记录 → WHERE tx_type=5 AND ...
   走 idx_user_currency_time 索引
```

#### tx_type 枚举映射

| 产品场景 | tx_type值 | op_type值 | direction |
|----------|-----------|-----------|-----------|
| 充值到账 | 1 充值 | 1 入账 | 1 收入 |
| 提现冻结 | 10 冻结 | 3 冻结 | 2 支出（可用减少） |
| 提现解冻（驳回/失败） | 11 解冻 | 4 解冻 | 1 收入（可用恢复） |
| 提现扣款（出款成功） | 12 冻结扣款 | 5 冻结扣款 | 2 支出 |
| 兑换-源币种扣减 | 4 兑换出 | 2 扣款 | 2 支出 |
| 兑换-BSB到账 | 3 兑换入 | 1 入账 | 1 收入 |
| 活动奖励到账 | 5 奖励 | 1 入账 | 1 收入 |
| 人工加款 | 6 人工加款 | 1 入账 | 1 收入 |
| 人工减款 | 7 人工减款 | 2 扣款 | 2 支出 |
| 投注扣款 | 8 投注扣款 | 2 扣款 | 2 支出 |
| 投注返奖 | 9 返奖 | 1 入账 | 1 收入 |

#### 关键设计说明

```

   创建即完成，没有 status 字段。
   如果兑换事务失败，记录根本不会写入（整个事务回滚）。

2. 一笔兑换产生的流水：
   - transaction_log: tx_type=4(兑换出), 源币种中心钱包, direction=2
   - transaction_log: tx_type=3(兑换入), BSB中心钱包, direction=1
   - transaction_log: tx_type=3(兑换入), BSB奖励钱包, direction=1 （如果有赠送）

3. bonus_ratio 赠送比例：
   ×100 存储（而不是×10000），因为产品文档中赠送比例是整数百分比（如10%），精度到百分之一即可。
   10% → 存 1000（实际是 10.00%）

4. C端兑换记录展示：
   兑换金额 = source_amount / source_currency.multiplier
   赠送 = bonus_ratio / 100 %
   实际到账 = (target_amount + bonus_amount) / bsb.multiplier
```

    `delete_at`          BIGINT       NOT NULL DEFAULT 0 COMMENT '删除时间（毫秒，0=未删除）',

    PRIMARY KEY (`id`),
    INDEX `idx_user_currency_type` (`user_id`, `currency_id`, `account_type`, `delete_at`),
    INDEX `idx_user_id` (`user_id`, `delete_at`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='提现账户表';
```

#### 关键设计说明

```
1. 三种账户类型复用一张表：
   银行卡用 bank_name, bank_branch, account_no, holder_name
   电子钱包用 ewallet_type, account_no(手机号), holder_name
   USDT用 network_type, account_no(地址), holder_name
   未使用的字段默认空字符串，不影响查询和展示。

2. 敏感数据加密：
   account_no 和 holder_name 使用 EncryptedString 类型。
   GORM 代码生成时指定：
     gen.FieldType("account_no", "utils.EncryptedString")
     gen.FieldType("holder_name", "utils.EncryptedString")
   写入时自动 AES-256 加密，读取时自动解密，对业务代码透明。
   数据库中存储的是密文（base64编码后的加密数据），varchar(512) 足够。

3. 不支持用户自行修改：
   产品需求明确"不可自改提现账户"。
   如果需要更换账户，走客服或后台操作流程（软删除旧的+绑定新的）。

4. holder_name 自动填充：
   绑定提现账户时，前端不传 holder_name，后端从 KYC 查询结果自动填充。
   holder_name 必须与 KYC 认证姓名一致。
```

## 九、索引策略汇总

### 9.1 核心索引与使用场景

| 表 | 索引 | 使用场景 |
|---|-------|---------|
| wallet_account | `uk_user_currency_type` | SELECT FOR UPDATE 行锁精确定位 + 唯一性保证 |
| wallet_account | `idx_user_id` | 用户所有币种余额查询 |
| transaction_log | `uk_ref_order_tx` | 幂等校验（唯一索引冲突=重复操作） |
| transaction_log | `idx_user_currency_time` | C端记录分页（按币种+时间倒序） |
| transaction_log | `idx_user_tx_type_time` | C端分Tab记录查询（按交易类型筛选） |
| deposit_order | `uk_order_no` | 订单号查询 |
| deposit_order | `idx_user_currency_amount_status` | 重复充值检测 |
| deposit_order | `idx_status_create` | 定时任务扫描超时订单 |
| withdraw_order | `uk_order_no` | 订单号查询 |
| withdraw_order | `idx_status` | 财务按状态批量查询待审核订单 |
| withdraw_order | `idx_status_create` | 定时任务扫描超时冻结订单 |
| currency | `uk_code` | 币种代码唯一 |
| exchange_rate_log | `idx_currency_time` | B端汇率日志分页 |
| withdraw_account | `idx_user_currency_type` | 用户提现时查询已绑定账户 |

### 9.2 索引设计原则

```
1. 唯一索引用于业务约束（uk_前缀）：
   防止重复数据，同时提供精确查询能力

2. 联合索引遵循最左前缀原则：
   idx_user_currency_time (user_id, currency_id, create_at DESC)
   → 支持 WHERE user_id=? AND currency_id=?
   → 支持 WHERE user_id=? AND currency_id=? ORDER BY create_at DESC
   → 不支持 WHERE currency_id=?（缺少最左列）

3. 不过度建索引：
   写入密集的流水表只建必要索引
   每个额外索引都增加写入开销（INSERT时维护B+Tree）

4. 覆盖查询优化：
   重复充值检测的 idx_user_currency_amount_status 把 amount 和 status 都包含进去，
   查询可以直接在索引上完成，不需要回表。
```

## 十二、数据模型与余额引擎的关系

### 12.1 余额引擎只操作两个模型

```
余额引擎（wallet_engine.go）对外暴露两个方法：
  GetBalance  → 读 WalletAccount
  ModifyBalance → 写 WalletAccount + 写 TransactionLog（同一事务）

所有上层业务（充值/提现/兑换/人工修正/投注/返奖）
都通过 ModifyBalance 操作余额，不直接操作 WalletAccount 表。

这意味着：
  WalletAccount + TransactionLog 是余额引擎的"私有模型"
  上层业务通过引擎接口间接操作，不绕过引擎直接写这两张表
```

### 12.2 ModifyBalance 参数到模型字段的映射

```
ModifyBalanceParams {
    UserID       → WalletAccount.user_id + TransactionLog.user_id
    CurrencyID   → WalletAccount.currency_id + TransactionLog.currency_id
    WalletType   → WalletAccount.wallet_type + TransactionLog.wallet_type
    OpType       → TransactionLog.op_type + 决定余额加减逻辑
    Amount       → TransactionLog.amount + 余额变动量
    RefOrderNo   → TransactionLog.ref_order_no（幂等键）
    TxType       → TransactionLog.tx_type
    Remark       → TransactionLog.remark
}

执行后自动填充：
    TransactionLog.direction    ← 由 OpType 决定
    TransactionLog.balance_after ← 变动后的 available_balance
    TransactionLog.frozen_after  ← 变动后的 frozen_balance
    TransactionLog.create_at     ← 当前毫秒时间戳
```

## 十三、数据生命周期

### 13.1 各模型数据增长预估

| 模型 | 增长频率 | 是否可归档 | 说明 |
|------|---------|-----------|------|
| Currency | 极低（几十条） | 否 | 币种配置，手动管理 |
| ExchangeRateLog | 低-中（每天×币种数） | 可（按时间） | 定时任务生成 |
| WalletAccount | 中（用户数×币种数×钱包类型） | 否 | 余额实时数据 |
| TransactionLog | **高**（每次资金操作一条） | 可（按时间） | 增长最快的表 |
| DepositOrder | 中-高 | 可（按时间） | 每次充值一条 |
| WithdrawOrder | 中 | 可（按时间） | 每次提现一条 |
| ExchangeRecord | 中 | 可（按时间） | 每次兑换一条 |
| WithdrawAccount | 低（每用户×每币种×每类型≈1条） | 否 | 绑定后复用 |

### 13.2 大表应对策略

```
TransactionLog 是增长最快的表，预计千万级后需要考虑：

短期（<1000万行）：
  → 不做任何特殊处理，TiDB 索引足够

中期（1000万-1亿行）：
  → TiDB 原生分区表（按 create_at 范围分区）
  → 或 TiDB 自动分片（基于 user_id Hash）

长期（>1亿行）：
  → 冷热数据分离：N个月前的历史数据归档到独立表/独立库
  → C端只查近N个月数据，B端提供历史数据查询入口
```

  → 是否有遗漏的字段或多余的字段
  → 与财务方确认跨模块字段（channel_id 类型等）

步骤2：编写 DDL 并在 TiDB 中建表
  → 按本文档的 CREATE TABLE 语句执行
  → 确保索引和注释完整

步骤3：执行 GORM 代码生成
  → go run internal/gen/gorm_gen.go
  → 自动产出 model + query + repo

步骤4：编写 IDL（基于模型字段定义 RPC 请求/响应结构）
  → 模型字段 → IDL 字段的映射
  → 注意：IDL 中金额返回给前端时需要除以 multiplier

步骤5：实现余额引擎
  → 基于 WalletAccount + TransactionLog 两个模型
  → GetBalance + ModifyBalance

步骤6：逐个实现业务模块
  → 币种配置（Currency + ExchangeRateLog）
  → 兑换（ExchangeRecord，自闭环最先可验证）
  → 充值（DepositOrder）
  → 提现（WithdrawOrder + WithdrawAccount）
```

### 14.2 弹性空间（可能调整的部分）

| 可能调整的点 | 当前设计 | 可能的变化 |
|-------------|---------|-----------|
| 奖金配置数据归属 | DepositOrder.bonus_activity_id 关联外部 | 可能钱包自己维护奖金配置表 |
| 充值/提现档位配置 | 通过 RPC 读取（可能挂在 Currency 或独立表） | 评审确认后可能增加 deposit_config / withdraw_config 表 |
| 冻结记录是否独立表 | 合并在 TransactionLog 中（tx_type=10/11/12） | 如果需要更丰富的冻结管理，可能拆为 freeze_record 表 |
| 兑换赠送比例配置 | ExchangeRecord.bonus_ratio 每次传入 | 可能需要一张兑换配置表 |
| 稽核流水配置 | 暂未建模 | 如果奖励钱包流水稽核逻辑明确后可能新增表 |

## 十五、总结

### 模型清单

| 序号 | 模型名 | 表名 | 域 | 核心用途 |
|------|--------|------|---|---------|
| 1 | Currency | currency | 配置域 | 币种基本信息+汇率+展示规则 |
| 2 | ExchangeRateLog | exchange_rate_log | 配置域 | 汇率变更审计日志 |
| 3 | WalletAccount | wallet_account | 账务域 | 用户余额存储（核心中的核心） |
| 4 | TransactionLog | transaction_log | 账务域 | 资金流水（审计根基） |
| 5 | DepositOrder | deposit_order | 订单域 | 充值订单全生命周期 |
| 6 | WithdrawOrder | withdraw_order | 订单域 | 提现订单全生命周期 |
| 7 | ExchangeRecord | exchange_record | 订单域 | 兑换操作记录 |
| 8 | WithdrawAccount | withdraw_account | 账户域 | 用户绑定的提现收款信息 |

### 枚举清单

| 序号 | 枚举名 | 值域 | 用途 |
|------|--------|------|------|
| 1 | WalletType | 1-5 | 钱包类型 |
| 2 | CurrencyType | 1-3 | 币种类型 |
| 3 | OpType | 1-5 | 余额操作类型 |
| 4 | TxType | 1-12 | 交易流水类型 |
| 5 | DepositStatus | 0-5 | 充值订单状态 |
| 6 | WithdrawStatus | 0-5 | 提现订单状态 |
| 7 | AuditStatus | 0-3 | 审核进度 |
| 8 | WithdrawAccountType | 1-3 | 提现账户类型 |
| 9 | PaymentMethod | 1-3 | 充值方式 |
| 10 | RateTriggerType | 1-2 | 汇率更新触发类型 |

### 设计原则

```
1. 每张表都有明确的产品需求来源，不臆造
2. 金额全部 bigint 最小单位整数，全链路无浮点
3. 汇率 ×10^8 存储，统一精度
4. 流水表只增不改不删，是对账和审计的数据根基
5. 幂等通过 DB 唯一索引保证，与事务原子绑定
6. 敏感数据（账号/姓名）AES-256 加密存储
7. 遵循现有工程的全部字段约定和命名惯例
8. GORM 模型不手写，全部从 DDL 自动生成
```
