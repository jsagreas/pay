# 第二部分：币种配置域（L0 底座）

> 现实世界：平台支持多种货币（法币/加密/平台币），每种货币有汇率、浮动比例、阈值等配置
> 数据模型：4张表承载币种管理和汇率管理的完整数据需求

## 2.1 t_currency — 币种主表

### 2.1.1 现实世界 → 数据模型映射

| 现实世界概念 | 数据字段 | 映射说明 |
|-------------|---------|---------|
| 币种编码（VND/USDT/BSB） | currency_code | 大写字母 3-5 位，创建后不可修改，全局唯一标识 |
| 币种名称（越南盾/泰达币） | currency_name | 展示用名称，可编辑 |
| 币种类型（法币/加密/平台） | currency_type | 枚举：1法币 2加密货币 3平台货币 |
| 币种图标 | icon_url | SVG/PNG 文件上传后的 S3 URL，≤5k |
| 汇率浮动百分比 | float_rate | 用于计算入款/出款汇率，每个币种独立配置 |
| 偏差阈值百分比 | threshold | 实时汇率与平台汇率偏差超此值时自动更新平台汇率 |
| 展示小数位数 | decimal_places | VND/IDR=0, THB/BSB=2, USDT=8 |
| 千分位符号 | thousand_sep | VND/IDR=".", THB/BSB/USDT="," |
| 启用/禁用状态 | status | 1启用 2禁用，影响C端可用币种列表 |
| 排序权重 | sort_order | C端币种切换列表的排列顺序 |

### 2.1.2 DDL

```sql
CREATE TABLE t_currency (
    id              BIGINT        NOT NULL AUTO_INCREMENT COMMENT '主键ID',
    currency_code   VARCHAR(10)   NOT NULL COMMENT '币种编码(VND/IDR/THB/USDT/BSB)，创建后不可修改',
    currency_name   VARCHAR(64)   NOT NULL COMMENT '币种名称',
    currency_type   TINYINT       NOT NULL COMMENT '币种类型(1法币/2加密货币/3平台货币)',
    icon_url        VARCHAR(256)  NOT NULL DEFAULT '' COMMENT '币种图标URL(SVG/PNG≤5k)',
    float_rate      DECIMAL(10,4) NOT NULL DEFAULT 0 COMMENT '汇率浮动百分比(如5.00表示5%)',
    threshold       DECIMAL(10,4) NOT NULL DEFAULT 0 COMMENT '偏差阈值百分比(偏差≥此值时更新平台汇率)',
    decimal_places  TINYINT       NOT NULL DEFAULT 2 COMMENT '金额展示小数位数(VND/IDR=0,THB/BSB=2,USDT=8)',
    thousand_sep    VARCHAR(4)    NOT NULL DEFAULT ',' COMMENT '千分位分隔符(VND/IDR=".",其他=",")',
    sort_order      INT           NOT NULL DEFAULT 0 COMMENT '排序权重，数值越小越靠前',
    status          TINYINT       NOT NULL DEFAULT 2 COMMENT '状态(1启用/2禁用)，新增默认禁用',
    create_by       BIGINT        NOT NULL DEFAULT 0 COMMENT '创建人ID(UserPlugin自动填充)',
    create_at       BIGINT        NOT NULL DEFAULT 0 COMMENT '创建时间戳(UserPlugin自动填充)',
    update_by       BIGINT        NOT NULL DEFAULT 0 COMMENT '更新人ID(UserPlugin自动填充)',
    update_at       BIGINT        NOT NULL DEFAULT 0 COMMENT '更新时间戳(UserPlugin自动填充)',
    delete_at       BIGINT        NOT NULL DEFAULT 0 COMMENT '软删除(0=未删除,非0=删除时间戳)',
    PRIMARY KEY (id),
    UNIQUE KEY uk_currency_code (currency_code, delete_at),
    KEY idx_status (status),
    KEY idx_currency_type (currency_type)
) COMMENT '币种主表';
```

### 2.1.3 字段设计依据

| 字段 | 设计决策 | 需求依据 |
|------|---------|---------|
| currency_code | VARCHAR(10)，唯一索引含 delete_at | 币种编码创建后不可修改（需求文档3-3），软删除后允许重建同编码 |
| currency_type | TINYINT 枚举 | 三种币种类型（需求文档3-1）：法币/加密/平台 |
| float_rate | DECIMAL(10,4) | 汇率浮动%（需求文档4-1），精度到万分之一足够 |
| threshold | DECIMAL(10,4) | 偏差阈值%（需求文档4-2），与 float_rate 同精度 |
| decimal_places | TINYINT | 金额展示小数位数（需求文档4-3），值域 0-8 |
| thousand_sep | VARCHAR(4) | 千分位符号（需求文档4-3），VND/IDR 用"."，其他用"," |
| status 默认=2 | 新增币种默认禁用 | 需管理员手动启用后才对C端可见（流程思路2.1.2） |

### 2.1.4 初始化数据

```sql
('THB', '泰铢',      1, 2.0000, 5.0000, 2, ',', 3, 1),
```

## 2.2 t_currency_rate — 汇率表

### 2.2.1 现实世界 → 数据模型映射

| 现实世界概念 | 数据字段 | 映射说明 |
|-------------|---------|---------|
| 某币种的实时汇率 | rate_type=1 | 3家三方API取均值，定时更新 |
| 某币种的平台汇率 | rate_type=2 | 偏差超阈值时从实时汇率拷贝过来 |
| 汇率值（1 基准币种 = X 目标币种） | rate_value | 例：1 USDT = 25000 VND |
| 入款汇率 / 出款汇率 | 不落表，实时计算 | 入款=实时×(1+float%), 出款=实时×(1-float%) |

### 2.2.2 DDL

```sql
CREATE TABLE t_currency_rate (
    id              BIGINT        NOT NULL AUTO_INCREMENT COMMENT '主键ID',
    currency_code   VARCHAR(10)   NOT NULL COMMENT '币种编码',
    rate_type       TINYINT       NOT NULL COMMENT '汇率类型(1实时汇率/2平台汇率)',
    rate_value      DECIMAL(20,8) NOT NULL DEFAULT 0 COMMENT '汇率值(1基准币种=X本币种)',
    api_a_rate      DECIMAL(20,8) NOT NULL DEFAULT 0 COMMENT 'API-A返回的汇率(仅实时类型有效)',
    api_b_rate      DECIMAL(20,8) NOT NULL DEFAULT 0 COMMENT 'API-B返回的汇率(仅实时类型有效)',
    api_c_rate      DECIMAL(20,8) NOT NULL DEFAULT 0 COMMENT 'API-C返回的汇率(仅实时类型有效)',
    sync_at         BIGINT        NOT NULL DEFAULT 0 COMMENT '最近一次同步时间戳',
    create_by       BIGINT        NOT NULL DEFAULT 0,
    create_at       BIGINT        NOT NULL DEFAULT 0,
    update_by       BIGINT        NOT NULL DEFAULT 0,
    update_at       BIGINT        NOT NULL DEFAULT 0,
    delete_at       BIGINT        NOT NULL DEFAULT 0,
    PRIMARY KEY (id),
    UNIQUE KEY uk_code_type (currency_code, rate_type, delete_at),
    KEY idx_currency_code (currency_code)
) COMMENT '汇率表';
```

### 2.2.3 设计说明

| 设计点 | 说明 | 依据 |
|--------|------|------|
| rate_type 只有 1 和 2 | 入款/出款汇率是实时计算的（实时汇率×浮动%），不需要持久化 | 架构设计7.1"入款/出款汇率不持久化" |
| api_a/b/c_rate | 记录三方API各自返回的原始值，便于排查汇率异常 | 需求文档4-2"接入3家三方汇率API" |
| 唯一索引 (code, type, delete_at) | 每个币种每种类型只有一条记录 | 一个币种同时只有一个实时汇率和一个平台汇率 |
| BSB 特殊处理 | BSB 锚定 USDT（10BSB=1USDT），汇率固定写入，不需要三方同步 | 需求文档3-1"10BSB=1USDT" |
| DECIMAL(20,8) | 与钱包余额统一精度 | 架构设计3.2.4 金额精度架构 |

### 2.2.4 初始化数据

```sql
-- BSB固定锚定汇率（BSB相对于USDT的汇率：1 USDT = 10 BSB）
INSERT INTO t_currency_rate (currency_code, rate_type, rate_value) VALUES
('BSB', 1, 10.00000000),  -- BSB实时汇率=固定值
('BSB', 2, 10.00000000);  -- BSB平台汇率=固定值

-- 其他币种汇率由定时任务写入，此处不预置
```

## 2.3 t_currency_rate_log — 汇率变更日志

### 2.3.1 现实世界 → 数据模型映射

| 现实世界概念 | 数据字段 | 映射说明 |
|-------------|---------|---------|
| 平台汇率发生自动更新 | 一条日志记录 | 仅在偏差超阈值触发更新时记录，不是每次同步都记录 |
| 变更前后的汇率值 | rate_before / rate_after | 审计追踪 |
| 触发原因（偏差值和阈值） | deviation / threshold | 便于事后复盘 |

### 2.3.2 DDL

```sql
CREATE TABLE t_currency_rate_log (
    id              BIGINT        NOT NULL AUTO_INCREMENT COMMENT '主键ID',
    currency_code   VARCHAR(10)   NOT NULL COMMENT '币种编码',
    rate_before     DECIMAL(20,8) NOT NULL COMMENT '变更前平台汇率',
    rate_after      DECIMAL(20,8) NOT NULL COMMENT '变更后平台汇率(=当前实时汇率)',
    realtime_rate   DECIMAL(20,8) NOT NULL COMMENT '触发时的实时汇率',
    deviation       DECIMAL(10,4) NOT NULL COMMENT '偏差值百分比',
    threshold       DECIMAL(10,4) NOT NULL COMMENT '触发时的阈值百分比',
    trigger_type    TINYINT       NOT NULL DEFAULT 1 COMMENT '触发类型(1自动偏差更新/2手动更新)',
    create_at       BIGINT        NOT NULL DEFAULT 0,
    delete_at       BIGINT        NOT NULL DEFAULT 0,
    PRIMARY KEY (id),
    KEY idx_currency_code (currency_code, create_at),
    KEY idx_create_at (create_at)
) COMMENT '汇率变更日志';
```

### 2.3.3 设计说明

| 设计点 | 说明 | 依据 |
|--------|------|------|
| 只记录平台汇率变更 | 实时汇率每 1-5 分钟更新一次，不记日志（量太大）；平台汇率按偏差阈值触发更新，才记录 | 需求文档4-4"汇率日志" |
| 无 update_by/update_at | 日志表只增不改，不需要更新审计字段 | 日志天然只增不改 |
| 保留 deviation + threshold | 便于 B 端汇率日志查询页面展示"触发原因" | 需求文档4-4 |
| trigger_type | 区分自动触发和手动更新（预留） | 扩展性考虑 |

## 2.4 t_base_currency — 基准币种配置

### 2.4.1 现实世界 → 数据模型映射

| 现实世界概念 | 数据字段 | 映射说明 |
|-------------|---------|---------|
| 平台统一的基准换算币种 | currency_code | 推荐 USDT，设置后不可更改 |
| 设置状态（是否已设置） | 记录是否存在 | 表中有记录=已设置，无记录=未设置 |

### 2.4.2 DDL

```sql
CREATE TABLE t_base_currency (
    id              BIGINT        NOT NULL AUTO_INCREMENT COMMENT '主键ID',
    currency_code   VARCHAR(10)   NOT NULL COMMENT '基准币种编码(推荐USDT)',
    create_by       BIGINT        NOT NULL DEFAULT 0,
    create_at       BIGINT        NOT NULL DEFAULT 0,
    update_by       BIGINT        NOT NULL DEFAULT 0,
    update_at       BIGINT        NOT NULL DEFAULT 0,
    delete_at       BIGINT        NOT NULL DEFAULT 0,
    PRIMARY KEY (id)
) COMMENT '基准币种配置(全局唯一)';
```

### 2.4.3 设计说明

| 设计点 | 说明 | 依据 |
|--------|------|------|
| 全局唯一 | 表中最多只有一条有效记录（delete_at=0） | 需求文档3-2"基准币种设置后不可更改" |
| 不可更改的实现 | 业务层校验：查询有效记录存在则拒绝再次设置 | 实现思路2.1.4"基准币种设置流程" |
| 为什么单独建表 | 基准币种是全局配置，放在 t_currency 中不合适（它不是币种属性而是平台属性） | 架构清晰，职责分离 |

# 第三部分：账户核心域（全系统最核心）

> 现实世界：用户在某个币种下有某种类型的钱包，钱包有余额、冻结金额；每一次余额变动都有完整记录
> 数据模型：t_wallet + t_wallet_transaction 构成整个资金系统的核心

## 3.1 t_wallet — 钱包余额主表

### 3.1.1 现实世界 → 数据模型映射

| 现实世界概念 | 数据字段 | 映射说明 |
|-------------|---------|---------|
| 一个用户 | user_id | 用户唯一标识 |
| 选了某个币种 | currency_code | VND/IDR/THB/USDT/BSB |
| 某种类型的钱包 | wallet_type | 1中心/2奖励/3主播/4代理/5场馆 |
| 这三者组合后有一个唯一的余额 | balance | 该用户该币种该钱包类型的可用余额 |
| 提现审核期间的冻结金额 | frozen | 独立于 balance 的冻结字段 |
| 可提现余额 | balance - frozen | 实时计算，不落字段 |
| 资金钱包余额（C端展示） | 中心.balance + 奖励.balance | 需求文档2-1"对用户展示的总余额" |
| 并发控制版本 | version | 乐观锁，配合 FOR UPDATE 双重保障 |

### 3.1.2 DDL

```sql
CREATE TABLE t_wallet (
    id              BIGINT        NOT NULL AUTO_INCREMENT COMMENT '主键ID',
    user_id         BIGINT        NOT NULL COMMENT '用户ID',
    currency_code   VARCHAR(10)   NOT NULL COMMENT '币种编码(VND/IDR/THB/USDT/BSB)',
    wallet_type     TINYINT       NOT NULL COMMENT '钱包类型(1中心/2奖励/3主播/4代理/5场馆)',
    balance         DECIMAL(20,8) NOT NULL DEFAULT 0.00000000 COMMENT '可用余额',
    frozen          DECIMAL(20,8) NOT NULL DEFAULT 0.00000000 COMMENT '冻结金额(提现审核期间)',
    version         INT           NOT NULL DEFAULT 0 COMMENT '乐观锁版本号',
    create_by       BIGINT        NOT NULL DEFAULT 0,
    create_at       BIGINT        NOT NULL DEFAULT 0,
    update_by       BIGINT        NOT NULL DEFAULT 0,
    update_at       BIGINT        NOT NULL DEFAULT 0,
    delete_at       BIGINT        NOT NULL DEFAULT 0,
    PRIMARY KEY (id),
    UNIQUE KEY uk_user_currency_type (user_id, currency_code, wallet_type, delete_at),
    KEY idx_user_id (user_id),
    KEY idx_user_currency (user_id, currency_code)
) COMMENT '钱包余额主表';
```

### 3.1.3 字段设计依据

| 字段 | 设计决策 | 需求/架构依据 |
|------|---------|-------------|
| `DECIMAL(20,8)` | 统一精度 | USDT 需 8 位小数，法币整数只是展示层截断，存储层统一（架构设计3.2.4） |
| `balance + frozen` 双字段 | 冻结不扣 balance，可提现=balance-frozen | 提现审核期间用户看到的"总余额"不变，"可提现余额"减少（架构设计8.1） |
| `version` | 乐观锁辅助 | 配合 FOR UPDATE 悲观锁双重保障（架构设计4.1 Step5） |
| `uk_user_currency_type` | 三维唯一 | 同一用户+同一币种+同一钱包类型只有一行（实现思路3.2） |
| 唯一索引含 `delete_at` | 软删除后允许重建 | 项目规范 delete_at=0 为未删除 |
| 无 CHECK 约束 | TiDB 支持 CHECK 但 GORM Gen 不感知 | 在应用层 account_ser 中校验 balance≥0 和 frozen≥0 |

### 3.1.4 三维唯一性说明

```
用户 Alice (user_id=1001) 的钱包矩阵：

              VND        IDR        THB        USDT       BSB
中心(1)    [row_1]    [row_2]    [row_3]    [row_4]    [row_5]
奖励(2)    [row_6]    [row_7]    [row_8]    [row_9]    [row_10]
主播(3)    [row_11]   [row_12]   [row_13]   [row_14]   [row_15]
代理(4)    (预留)     (预留)     (预留)     (预留)     (预留)
场馆(5)    (预留)     (预留)     (预留)     (预留)     (预留)

每个格子 = 一行 t_wallet 记录
每行通过 (user_id=1001, currency_code='VND', wallet_type=1) 唯一定位

首期需要的行：中心(1) + 奖励(2) + 主播(3) = 最多 5币种 × 3类型 = 15行/用户
按懒创建策略，用户首次涉及某币种某钱包时才创建，不预创建所有组合
```

### 3.1.5 冻结模型的数据变化

```
提现全生命周期中 t_wallet 的字段变化：

[初始] balance=10000, frozen=0       → 可提现=10000
       ↓ Freeze(5000)
[冻结] balance=10000, frozen=5000    → 可提现=5000
       ↓
       ├─ Unfreeze(5000) [驳回]
       │  balance=10000, frozen=0    → 可提现=10000（恢复）
       │
       └─ DebitFrozen(5000) [出款成功]
          balance=5000, frozen=0     → 可提现=5000（余额正式减少）

关键：Freeze 只增 frozen 不扣 balance
     DebitFrozen 同时扣 balance 和 frozen
     Unfreeze 只减 frozen
```

### 3.2.3 字段设计依据

| 字段/约束 | 设计决策 | 依据 |
|----------|---------|------|
| `uk_biz_order_wallet` | **幂等保障的最终兜底** | DB层面保证同一业务+同一订单+同一钱包不会有两条流水（架构设计4.3） |
| `balance_before/after` | **对账核心** | 需求文档9"任何资金变动都要有流水记录"，任意时点可通过流水链验证余额正确性 |
| `frozen_before/after` | 冻结/解冻操作的快照 | 提现审计需要追踪冻结金额变化 |
| `direction` | 独立字段而非正负 amount | 避免负数歧义，方向明确 |
| amount 始终正数 | 规避负数运算复杂性 | 业界标准做法 |
| 只增不改不删 | 流水是审计凭证 | "修正"通过新增反向流水实现，不修改已有记录（架构设计3.2.3） |
| 无 update_by/update_at | 只增不改，不需要更新审计字段 | 流水表天然只增不改 |
| `idx_user_currency_time` | C端记录列表查询 | 需求文档8-2"只展示当前币种的记录，按时间倒序" |
| `idx_user_biz` | 按业务类型筛选用户流水 | C端奖励记录Tab等场景 |

### 3.2.4 唯一索引含 wallet_type 的原因

```
为什么是 UNIQUE(biz_type, order_no, wallet_type) 而不是 UNIQUE(biz_type, order_no)？

场景：BetDebit 投注扣款
  一笔投注(order_no=BET001)需要同时：
    - 从中心钱包扣80 → 产生一条流水 (biz_type=bet_debit, order_no=BET001, wallet_type=1)
    - 从奖励钱包扣20 → 产生一条流水 (biz_type=bet_debit, order_no=BET001, wallet_type=2)

  如果唯一索引不含 wallet_type，第二条流水会被拒绝。
  所以唯一索引必须是 (biz_type, order_no, wallet_type) 三字段组合。

同理：BetCredit 返奖、兑换（法币扣+BSB加+BSB奖励加）都涉及同一 order_no 多个 wallet_type。
```

### 3.2.5 流水链路验证原理

```
对账公式：

  wallet.balance == transaction[最新一条].balance_after

  且对每一条流水 T[n]：
    T[n].balance_before == T[n-1].balance_after
    T[n].balance_after  == T[n].balance_before + T[n].amount  (如果 direction=1)
    T[n].balance_after  == T[n].balance_before - T[n].amount  (如果 direction=2)

如果任何一条流水不满足以上条件，说明数据异常，需要人工介入。
这就是 balance_before/after 存在的核心价值。
```

```
GORM Gen 默认行为:
  DECIMAL(20,8) → Go string

业务层处理方式:
  1. 从 model 读取 string 值
  2. decimal.NewFromString(model.Balance) → decimal.Decimal
  3. 用 decimal.Decimal 做所有运算 (Add/Sub/Mul/Div)
  4. 运算结果 .String() → 回写 model

禁止:
  ✗ 直接用 float64 做金额运算
  ✗ 直接用 string 做金额比较 (如 "100.00" > "9.00" 是字母序比较)
  ✗ 跳过 decimal 库直接拼接 SQL 做运算

正确示例:
  balanceDec, _ := decimal.NewFromString(wallet.Balance)
  amountDec, _ := decimal.NewFromString(req.Amount)
  if balanceDec.LessThan(amountDec) {
      return errors.New("余额不足")
  }
  newBalance := balanceDec.Sub(amountDec)
  wallet.Balance = newBalance.String()
```

| delete_at = Unix 时间戳 | 已删除，值为删除时间 |

| t_wallet_transaction 不做软删除 | 流水是审计凭证，永远不删除 |

# 第十部分：数据量估算

## 10.1 各表增长预估

| 表 | 增长模式 | 估算依据 |
|----|---------|---------|
| t_currency | 极低频 | 初始 5 条，每新增一个国家/币种才增长，预计 <20 条 |
| t_currency_rate | 极低频 | 每币种 2 条（实时+平台），预计 <40 条 |
| t_currency_rate_log | 低频 | 偏差触发才写入，每币种每天 0-5 条，预计每月 <500 条 |

| t_wallet | 随用户增长 | 每用户最多 5币种×3类型=15 行（懒创建实际更少），10万用户≈50-100万行 |
| t_wallet_transaction | **高增长** | 每次余额变动一条，投注+充值+提现+礼物等，10万DAU×每人每天10笔≈100万条/天 |
| t_deposit_order | 中等增长 | 与充值频率相关，预计每天 1-10 万条 |
| t_withdraw_order | 低增长 | 提现频率低于充值，预计每天 0.1-1 万条 |
| t_exchange_order | 低增长 | 兑换频率较低，预计每天 0.1-1 万条 |
| t_withdraw_account | 低增长 | 每用户 1-5 个账户，10万用户≈10-50万行 |
| t_reward_record | 中等增长 | 与活动/投注频率相关，预计每天 1-10 万条 |

## 10.2 高增长表的应对策略

| 表 | 策略 | 说明 |
|----|------|------|
| t_wallet_transaction | 分区/归档 | 按月分区，历史数据定期归档到冷存储，TiDB 天然支持水平扩展 |
| t_deposit_order | 时间索引 | C端只查近期记录，B端查询走索引，TiDB 分布式存储自动负载均衡 |
| t_wallet | 主键/唯一索引查询 | 行级查询 <1ms，10万用户量级 TiDB 轻松应对 |

# 第十一部分：数据模型与业务场景的对应关系

## 11.1 充值场景涉及的表

```
用户发起充值 → 充值成功：

  写入：
    t_deposit_order          ← 创建充值订单(待支付)
    t_deposit_order          ← 更新订单状态(成功)
    t_wallet                 ← 中心钱包余额 += 到账金额
    t_wallet_transaction     ← 充值入账流水 (biz_type=deposit)
    t_reward_record          ← 如有奖金活动，记录奖励
    t_wallet                 ← 奖励钱包余额 += 奖金金额 (如有)
    t_wallet_transaction     ← 奖金入账流水 (biz_type=deposit_bonus)

  读取：
    t_currency               ← 币种校验
    t_currency_rate          ← 汇率获取（跨币种时）
    t_deposit_order          ← 重复转账检测
```

## 11.2 提现场景涉及的表

```
用户发起提现 → 审核 → 出款：

  写入：
    t_wallet                 ← 冻结余额 (frozen += amount)
    t_wallet_transaction     ← 冻结流水 (biz_type=withdraw_freeze)
    t_withdraw_order         ← 创建提现订单(审核中)

  审核驳回：
    t_wallet                 ← 解冻余额 (frozen -= amount)
    t_wallet_transaction     ← 解冻流水 (biz_type=withdraw_unfreeze)
    t_withdraw_order         ← 更新订单状态(已驳回)

  出款成功：
    t_wallet                 ← 正式扣除 (balance -= amount, frozen -= amount)
    t_wallet_transaction     ← 扣款流水 (biz_type=withdraw)
    t_withdraw_order         ← 更新订单状态(已完成)

  读取：
    t_withdraw_account       ← 收款账户信息
    t_deposit_order          ← 充值来源判断（决定提现方式）
    t_currency_rate          ← 出款汇率（BSB提现为法币时）
```

## 11.3 兑换场景涉及的表

```
用户法币→BSB兑换（单一DB事务，最独立）：

  写入（同一事务内）：
    t_wallet                 ← 法币中心钱包 -= from_amount
    t_wallet_transaction     ← 法币扣款流水 (biz_type=exchange_debit)
    t_wallet                 ← BSB中心钱包 += to_amount
    t_wallet_transaction     ← BSB入账流水 (biz_type=exchange_credit)
    t_wallet                 ← BSB奖励钱包 += bonus_amount
    t_wallet_transaction     ← BSB赠送流水 (biz_type=exchange_bonus)
    t_exchange_order         ← 兑换订单记录
    t_reward_record          ← 赠送部分的稽核要求记录

  读取：
    t_currency               ← 币种校验（必须是法币）
    t_currency_rate          ← 入款汇率
    t_wallet                 ← 法币余额校验 (FOR UPDATE)
```

## 11.4 投注扣款/返奖场景涉及的表

```
投注扣款 BetDebit（同一事务锁两行）：

  写入：
    t_wallet                 ← 中心钱包 -= centerDebit (FOR UPDATE)
    t_wallet                 ← 奖励钱包 -= bonusDebit  (FOR UPDATE)
    t_wallet_transaction     ← 中心扣款流水 (biz_type=bet_debit, wallet_type=1)
    t_wallet_transaction     ← 奖励扣款流水 (biz_type=bet_debit, wallet_type=2)

投注返奖 BetCredit（按原始比例返奖）：

  写入：
    t_wallet                 ← 中心钱包 += centerCredit
    t_wallet                 ← 奖励钱包 += bonusCredit
    t_wallet_transaction     ← 中心返奖流水 (biz_type=bet_credit, wallet_type=1)
    t_wallet_transaction     ← 奖励返奖流水 (biz_type=bet_credit, wallet_type=2)
```

## 12.2 自动生成 vs 手写的边界

```
自动生成（gorm_gen/）:
  ├── model/    表结构体（所有11张表）
  ├── query/    类型安全查询构建器 + Transaction 支持
  └── repo/     ~20个标准CRUD方法/表
                Create / Update / SoftDelete / FindById / FindByPage / ...

手写（rep/）:
  ├── account_rep.go      FOR UPDATE 锁定 + 余额更新 + 流水写入（同一事务）
  ├── currency_rep.go     汇率批量更新 + 偏差判断
  ├── withdraw_rep.go     充值来源聚合查询 + 限额累计查询
  └── exchange_rep.go     双币种原子转移事务（锁3行）

分界线:
  · 单表简单CRUD → 用 gorm_gen/repo
  · 涉及事务/FOR UPDATE/多表JOIN/复杂聚合 → 手写 rep/
  · 决不修改 gorm_gen/ 下的代码（会被重新生成覆盖）
```

# 第十三部分：数据模型关键约束总结

## 13.1 数据完整性约束

| 约束 | 层面 | 实现方式 | 目的 |
|------|------|---------|------|
| 币种编码唯一 | DB | UK (currency_code, delete_at) | 防止重复币种 |
| 钱包三维唯一 | DB | UK (user_id, currency_code, wallet_type, delete_at) | 同用户同币种同类型只有一行 |
| 流水幂等 | DB | UK (biz_type, order_no, wallet_type) | 同一操作不产生重复流水 |
| 订单号唯一 | DB | UK (order_no, delete_at) | 订单不重复 |
| 余额非负 | 应用 | account_ser 校验 balance≥0, frozen≥0 | 防止超扣 |
| 金额精度 | DB+应用 | DECIMAL(20,8) + shopspring/decimal | 防止精度丢失 |
| 流水只增不改 | 应用 | 业务层约定，不提供 UPDATE/DELETE 流水的接口 | 审计追踪 |
| 基准币种不可改 | 应用 | 业务层校验已设置则拒绝 | 需求文档3-2 |

## 13.2 并发安全约束

| 约束 | 实现方式 | 场景 |
|------|---------|------|
| 行级锁 | SELECT ... FOR UPDATE on t_wallet | 所有余额变更操作 |
| 乐观锁 | UPDATE ... WHERE version=? | 二重保障 |
| 操作防重 | Redis SetNX | 用户快速双击（提现/兑换） |
| 锁定顺序 | wallet_type 升序（1→2→3） | BetDebit/兑换 同时锁多行时防死锁 |

## 13.3 数据安全约束

| 约束 | 实现方式 | 场景 |
|------|---------|------|
| 敏感数据脱敏 | 展示时只显示后4位 | 银行卡号、USDT地址 |
| userId 来源可信 | 从 Token 中间件获取，不接受前端传参 | 所有C端接口 |
| 审计追踪 | UserPlugin + ser-blog.AddActLog | 所有B端写操作 |
