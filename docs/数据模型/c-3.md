## 二、通用规范

> 依据：工程认知c-3.md 3.4节 GORM + TiDB 规范

### 2.1 字段通用规范

| 规范项 | 约定 | 说明 |
|--------|------|------|
| **主键** | `id BIGINT AUTO_INCREMENT PRIMARY KEY` | 所有表统一 |
| **软删除** | `delete_at BIGINT NOT NULL DEFAULT 0` | 0=未删除，非零=删除时间戳(毫秒) |
| **审计字段** | `create_by BIGINT`, `create_at BIGINT`, `update_by BIGINT`, `update_at BIGINT` | GORM 审计插件自动填充 |
| **金额字段** | `DECIMAL(24,8)` | 24位总精度、8位小数，覆盖法币2位和加密货币8位 |
| **百分比字段** | `DECIMAL(8,4)` | 如汇率浮动%、手续费率%，保留4位小数 |
| **汇率字段** | `DECIMAL(24,8)` | 与金额同精度，防止精度丢失 |
| **字符串** | `VARCHAR(N)` | N 按实际最大长度设定，不用 TEXT |
| **时间** | `BIGINT` | 毫秒级时间戳，不用 DATETIME |
| **布尔** | `TINYINT` | 0=否/禁用，1=是/启用 |
| **字符集** | `utf8mb4` | TiDB 默认 |

### 2.2 Go 代码层规范

| 规范项 | 约定 | 说明 |
|--------|------|------|
| 金额类型 | `shopspring/decimal` 库的 `decimal.Decimal` | 不用 float64 |
| IDL 金额 | `string` | Thrift IDL 中金额用 string 传输，两端解析 |
| GORM 模型 | `FieldNullable: true` 生成 | optional 字段生成 `*type` 指针类型 |
| 仓储查询 | 每次必须 `.Where(DeleteAt.Eq(0))` | 软删除过滤不可遗漏 |

### 2.3 审计字段（所有表共有，GORM 插件自动填充）

```
字段名      类型     默认值  说明
─────────  ──────   ──────  ───────────────────────────
create_by  BIGINT   0       创建人ID（tracer.GetUserId）
create_at  BIGINT   0       创建时间戳(毫秒)
update_by  BIGINT   0       最后更新人ID
update_at  BIGINT   0       最后更新时间戳(毫秒)
delete_at  BIGINT   0       软删除时间戳，0=未删除
```

### 4.1 currency_config — 币种配置表

### 4.3 currency_rate — 当前汇率表

> 现实映射：每个币种当前的实时汇率和平台汇率快照
> 来源：需求理解c-3.md 2.4节 汇率体系
> 特征：每币种一条，定时任务或手动刷新时更新

```
┌──────────────────────────────────────────────────────────────────┐
│ currency_rate — 当前汇率表                                        │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  字段名              类型             约束         说明            │
│  ─────────────────  ──────────────  ──────────  ──────────────  │
│  id                 BIGINT          PK, AI      主键             │
│  currency_code      VARCHAR(16)     UNIQUE, NN  币种代码          │
│  real_time_rate     DECIMAL(24,8)   DEFAULT 0   实时汇率(3家API均值)│
│  platform_rate      DECIMAL(24,8)   DEFAULT 0   平台汇率          │
│  deposit_rate       DECIMAL(24,8)   DEFAULT 0   入款汇率(计算后存储)│
│  withdraw_rate      DECIMAL(24,8)   DEFAULT 0   出款汇率(计算后存储)│
│  deviation_pct      DECIMAL(8,4)    DEFAULT 0   当前偏差值(%)      │
│  last_refresh_at    BIGINT          DEFAULT 0   最后刷新时间戳     │
│  ─── 审计字段 ───                                                 │
│  create_by          BIGINT          DEFAULT 0                    │
│  create_at          BIGINT          DEFAULT 0                    │
│  update_by          BIGINT          DEFAULT 0                    │
│  update_at          BIGINT          DEFAULT 0                    │
│  delete_at          BIGINT          DEFAULT 0                    │
│                                                                  │
├──────────────────────────────────────────────────────────────────┤
│ 索引：                                                            │
│  UNIQUE KEY uk_currency_code (currency_code, delete_at)          │
├──────────────────────────────────────────────────────────────────┤
│ 汇率计算公式（来源：需求理解c-3.md 2.4.2节）：                      │
│  deposit_rate  = platform_rate × (1 + rate_float_pct / 100)     │
│  withdraw_rate = platform_rate × (1 - rate_float_pct / 100)     │
│  deviation_pct = |real_time_rate - platform_rate| / real_time_rate × 100  │
│                                                                  │
│ 更新规则：                                                        │
│  ① 初次：platform_rate 为空时，直接取 real_time_rate              │
│  ② 后续：deviation_pct ≥ threshold_pct 时更新 platform_rate       │
│  ③ 入款/出款偏差任一 ≥ 阈值即更新（需求红字强调）                    │
│  ④ 各币种独立判断，互不影响                                        │
└──────────────────────────────────────────────────────────────────┘
```

**设计决策**：deposit_rate 和 withdraw_rate 可以运行时计算也可以存储。这里选择**存储**，理由：
- 减少每次 RPC 查询时的计算开销（R1/R5 被高频调用）
- 定时任务刷新时一次算好写入，查询时直接返回
- 需要在汇率日志中记录当时的入款/出款汇率，存储更方便

## 九、记录查询领域模型

> 记录查询不新增表。C端记录页面的4个Tab数据分别来自已有表。

### 9.1 记录数据来源映射

| Tab | 数据源表 | 查询条件 | 列表字段 |
|-----|---------|---------|---------|
| 充值 | `recharge_order` | user_id + currency_code | 金额/方式/时间/状态 |
| 提现 | `withdraw_order` | user_id + currency_code | 金额/方式/时间/状态 |
| 兑换 | `exchange_record` | user_id + from_currency | 兑换金额/赠送%/实际到账/时间 |
| 奖励 | `wallet_transaction` | user_id + currency_code + direction=IN + operation_type IN (2,8,9,11,12) | 奖励金额/奖励类别/时间 |

### 9.2 奖励记录说明

> 来源：需求理解c-3.md 3.5.3节

奖励记录不是独立的表，而是从 `wallet_transaction` 中按条件过滤：

```
奖励记录 = wallet_transaction WHERE
  user_id = ?
  AND currency_code = ?
  AND direction = 1 (入账)
  AND operation_type IN (
    2,   -- 充值奖金入账
    8,   -- 兑换赠送入账
    9,   -- 人工加款（运营奖励类型）
    11,  -- 补单入账
    12   -- 补单赠送入账
  )
  AND delete_at = 0
ORDER BY create_at DESC

列表字段映射：
  "奖励金额" → amount
  "奖励类别" → operation_type (枚举展示名)
  "时间" → create_at
```

### 10.2 数据写入链路

```
充值场景的数据写入链路：
  用户充值 → 创建 recharge_order (status=0)
  回调入账 → 更新 recharge_order (status=1)
           → 写入 wallet_transaction (operation_type=1, order_no=C...)
           → 更新 wallet_balance (available +=)
           → (如有奖金) 写入 wallet_transaction (operation_type=2)
           → (如有奖金) 更新 wallet_balance (REWARD.available +=)

兑换场景的数据写入链路：
  用户兑换 → 同一事务内：
    写入 wallet_transaction × 3条 (扣法币+加BSB+赠送)
    更新 wallet_balance × 3行
    创建 exchange_record

提现场景的数据写入链路：
  用户提现 → 更新 wallet_balance (available -=, frozen +=)
           → 写入 wallet_transaction (operation_type=3)
           → 创建 withdraw_order (status=0)
  审核驳回 → 更新 wallet_balance (frozen -=, available +=)
           → 写入 wallet_transaction (operation_type=4)
           → 更新 withdraw_order (status=1/3/5)
  出款成功 → 更新 wallet_balance (frozen -=)
           → 写入 wallet_transaction (operation_type=5)
           → 更新 withdraw_order (status=6)
```

### 10.3 各表的读写频率特征

| 表 | 写频率 | 读频率 | 写场景 | 读场景 |
|---|--------|--------|--------|--------|

| base_currency | 一次 | 中(可缓存) | B端设置 | 各模块读取 |
| currency_rate | 低(定时任务) | 高(可缓存) | 刷新汇率 | 每次创建订单 |

| operation_log | 低 | 低(B端查看) | B端操作时 | B端日志查询 |
| **wallet_balance** | **高(每笔资金操作)** | **高** | 充值/提现/兑换 | 余额查询 |

## 十二、IDL 层数据模型

> IDL（Thrift）中的数据结构是网络传输层的模型，与数据库模型有对应关系但不完全相同

### 12.1 IDL 与 DB 模型的差异

| 差异点 | DB 模型 | IDL 模型 |
|--------|---------|---------|
| 金额类型 | DECIMAL(24,8) | string（字符串传输，两端解析） |
| 时间类型 | BIGINT(毫秒时间戳) | i64 |
| 审计字段 | create_by/create_at/update_by/update_at/delete_at | 响应中可选择性包含或不包含 |
| 分页 | FindByPage(offset, limit) | PageReq(pageNo, pageSize) / PageResp(pageList, total, pageNo, pageSize) |
| 敏感字段 | 明文存储 | 响应中脱敏（如银行卡号） |

### 12.2 核心 IDL 结构体示例

```thrift
// wallet.thrift
namespace go ser_wallet

// 余额查询响应 — 对应 W1(查询钱包余额)
struct WalletBalanceItem {
    1: string currencyCode;          // 币种代码
    2: i8 walletType;                // 子钱包类型
    3: string walletTypeName;        // 子钱包名称(展示用)
    4: string availableBalance;      // 可用余额(string传输)
    5: string frozenBalance;         // 冻结余额
}

struct GetWalletBalanceResp {
    1: string totalBalance;              // 资金钱包总余额 = CENTER + REWARD
    2: list<WalletBalanceItem> wallets;  // 各子钱包明细
    3: string currencySymbol;            // 币种符号(展示用)
    4: string thousandSep;               // 千分位符号
    5: string decimalSep;                // 小数点符号
    6: i8 decimalPrecision;              // 小数精度
}

// 核心RPC — 对应 E2(CreditWallet)
struct CreditWalletReq {
    1: required i64 userId;              // 用户ID
    2: required string currencyCode;     // 币种代码
    3: required i8 walletType;           // 子钱包类型
    4: required string amount;           // 加款金额(string)
    5: required string orderNo;          // 关联订单号(幂等键)
    6: required i8 operationType;        // 操作类型
    7: optional string source;           // 来源标识
    8: optional string remark;           // 备注
}

struct CreditWalletResp {
    1: string beforeBalance;             // 操作前余额
    2: string afterBalance;              // 操作后余额
}
```
