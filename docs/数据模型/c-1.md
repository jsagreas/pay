## 一、映射方法论

### 1.1 从产品需求到数据模型的转换过程

```
产品原型 / 需求文档
    │
    │ 第一步：识别实体（名词提取）
    │ "用户有钱包" → wallet 实体
    │ "用户发起充值" → deposit_order 实体
    │ "币种有汇率配置" → currency_config 实体
    │
    │ 第二步：识别属性（形容词/值提取）
    │ "钱包有可用余额和冻结余额" → available_balance, frozen_balance
    │ "充值订单有金额、状态、时间" → amount, status, create_at
    │
    │ 第三步：识别关系（动词/关联提取）
    │ "一个用户有多个钱包（按币种+类型区分）" → user_id + currency_code + wallet_type 唯一
    │ "提现订单关联提现账户" → withdrawal_order.account_id → withdrawal_account.id
    │
    │ 第四步：识别约束（业务规则提取）
    │ "余额不能为负" → available_balance >= 0（乐观锁保证）
    │ "基准币种只能设一次" → system_config 中只允许 INSERT 一次
    │ "同一 orderId 不能重复入账" → transaction_log (order_id, log_type) 唯一索引
    │
    ▼
数据模型（表 + 字段 + 索引 + 约束 + 枚举）
```

### 1.2 工程通用字段模板

根据工程认知文档（11.1 节），**每张表**都包含以下通用字段：

```sql
-- 通用审计字段（每张表必须包含）
id              BIGINT        NOT NULL AUTO_INCREMENT  COMMENT '主键ID',
create_by       BIGINT        NOT NULL DEFAULT 0       COMMENT '创建人ID',
create_by_name  VARCHAR(50)   NOT NULL DEFAULT ''      COMMENT '创建人名称',
create_at       BIGINT        NOT NULL DEFAULT 0       COMMENT '创建时间（毫秒时间戳）',
update_by       BIGINT        NOT NULL DEFAULT 0       COMMENT '更新人ID',
update_by_name  VARCHAR(50)   NOT NULL DEFAULT ''      COMMENT '更新人名称',
update_at       BIGINT        NOT NULL DEFAULT 0       COMMENT '更新时间（毫秒时间戳）',
delete_at       BIGINT        NOT NULL DEFAULT 0       COMMENT '软删除（0=未删除，非0=删除时间戳）',
PRIMARY KEY (id)
```

依据：工程认知 `11.1 表结构设计规范` — 每张表都应包含这组字段，时间用 BIGINT 毫秒时间戳，软删除用 delete_at。

**后续各表 DDL 中用 `-- [通用审计字段]` 标记代替重复列出。**

## 三、表 ① wallet — 用户钱包余额表

### 3.1 需求来源

| 需求概念 | 映射为 | 来源 |
|---------|--------|------|
| 用户在平台上有钱包 | wallet 表 | 需求理解 1.1 |
| 每个币种独立隔离 | currency_code 字段 | 需求理解 2.1："互不干扰" |
| 按用途分 5 个口袋 | wallet_type 字段 | 需求理解 2.2：中心/奖励/主播/代理/场馆 |
| 余额可冻结（提现时） | available_balance + frozen_balance | 需求理解 3.2：提现审核流程 |
| 资金余额 = 中心 + 奖励 | 应用层计算 | 需求理解 2.2："C端展示的余额是这两者之和" |

### 3.2 DDL

```sql
CREATE TABLE `wallet` (
  `id`                  BIGINT        NOT NULL AUTO_INCREMENT  COMMENT '主键ID',
  `user_id`             BIGINT        NOT NULL                 COMMENT '用户ID',
  `currency_code`       VARCHAR(10)   NOT NULL                 COMMENT '币种代码（VND/IDR/THB/BSB/USDT）',
  `wallet_type`         INT           NOT NULL                 COMMENT '钱包类型 1中心 2奖励 3主播 4代理 5场馆',
  `available_balance`   DECIMAL(20,8) NOT NULL DEFAULT 0       COMMENT '可用余额',
  `frozen_balance`      DECIMAL(20,8) NOT NULL DEFAULT 0       COMMENT '冻结余额（提现待审核时冻结）',
  -- [通用审计字段]
  `create_by`           BIGINT        NOT NULL DEFAULT 0       COMMENT '创建人ID',
  `create_by_name`      VARCHAR(50)   NOT NULL DEFAULT ''      COMMENT '创建人名称',
  `create_at`           BIGINT        NOT NULL DEFAULT 0       COMMENT '创建时间（毫秒时间戳）',
  `update_by`           BIGINT        NOT NULL DEFAULT 0       COMMENT '更新人ID',
  `update_by_name`      VARCHAR(50)   NOT NULL DEFAULT ''      COMMENT '更新人名称',
  `update_at`           BIGINT        NOT NULL DEFAULT 0       COMMENT '更新时间（毫秒时间戳）',
  `delete_at`           BIGINT        NOT NULL DEFAULT 0       COMMENT '软删除（0=未删除）',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_user_currency_type` (`user_id`, `currency_code`, `wallet_type`, `delete_at`),
  KEY `idx_user_id` (`user_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户钱包余额表';
```

### 3.3 字段说明

| 字段 | 类型 | 需求来源 | 说明 |
|------|------|---------|------|
| user_id | BIGINT | 需求理解 1.3：用户角色 | 钱包所属用户 |
| currency_code | VARCHAR(10) | 需求理解 2.1：4种币种 + USDT | 如 "VND"/"IDR"/"THB"/"BSB"/"USDT" |
| wallet_type | INT | 需求理解 2.2：5种钱包 | 枚举见 2.1 节 |
| available_balance | DECIMAL(20,8) | 需求理解 2.2：可直接使用的余额 | 乐观锁条件：WHERE available >= amount |
| frozen_balance | DECIMAL(20,8) | 需求理解 3.2：提现冻结 | 提现待审核时 available→frozen |

### 3.4 索引说明

| 索引 | 类型 | 用途 |
|------|------|------|
| uk_user_currency_type | UNIQUE | 保证同用户+同币种+同类型只有一行；查余额时命中 |
| idx_user_id | 普通 | C1 接口查用户所有币种余额时使用 |

### 3.5 核心不变量

```
对于每一行 wallet 记录:
  available_balance >= 0
  frozen_balance >= 0
  (available_balance + frozen_balance) = 该钱包的真实余额
```

### 3.6 GORM/gen 生成后的 Go Model 预期

```go
type Wallet struct {
    ID               int64            `gorm:"column:id;primaryKey;autoIncrement:true"`
    UserID           int64            `gorm:"column:user_id;not null"`
    CurrencyCode     string           `gorm:"column:currency_code;not null"`
    WalletType       int32            `gorm:"column:wallet_type;not null"`
    AvailableBalance *decimal.Decimal  `gorm:"column:available_balance;not null;default:0"`
    FrozenBalance    *decimal.Decimal  `gorm:"column:frozen_balance;not null;default:0"`
    CreateBy         int64            `gorm:"column:create_by;not null;default:0"`
    CreateByName     string           `gorm:"column:create_by_name;not null;default:''"`
    CreateAt         int64            `gorm:"column:create_at;not null;default:0"`
    UpdateBy         int64            `gorm:"column:update_by;not null;default:0"`
    UpdateByName     string           `gorm:"column:update_by_name;not null;default:''"`
    UpdateAt         int64            `gorm:"column:update_at;not null;default:0"`

}
```

注：DECIMAL 字段 gorm/gen 可能生成为 `*string` 或 `*decimal.Decimal`（取决于 gen 配置），业务层统一用 `shopspring/decimal` 运算。

## 四、表 ② transaction_log — 余额变更流水表

### 4.1 需求来源

| 需求概念 | 映射为 | 来源 |
|---------|--------|------|
| 每笔余额变动必须可追溯 | transaction_log 表 | 需求理解 3.5：对账一致性 |
| 充值/提现/兑换/人工修正都是流水 | log_type 枚举 | 链路关系 5.1：createTransactionLog 被所有余额变更调用 |
| 幂等：同一 orderId 不能重复操作 | (order_id, log_type) 唯一索引 | 架构设计 5.2 |
| C端记录查询需要按币种筛选 | idx_user_currency 索引 | 链路关系 C12-C16 |

### 4.2 DDL

```sql
CREATE TABLE `transaction_log` (
  `id`                BIGINT        NOT NULL AUTO_INCREMENT  COMMENT '主键ID',
  `user_id`           BIGINT        NOT NULL                 COMMENT '用户ID',
  `currency_code`     VARCHAR(10)   NOT NULL                 COMMENT '币种代码',
  `wallet_type`       INT           NOT NULL                 COMMENT '钱包类型',
  `log_type`          INT           NOT NULL                 COMMENT '流水类型（见枚举2.5）',
  `amount`            DECIMAL(20,8) NOT NULL                 COMMENT '变更金额（正数）',
  `direction`         INT           NOT NULL                 COMMENT '方向 1=收入 2=支出',
  `balance_before`    DECIMAL(20,8) NOT NULL                 COMMENT '变更前可用余额',
  `balance_after`     DECIMAL(20,8) NOT NULL                 COMMENT '变更后可用余额',
  `order_id`          VARCHAR(50)   NOT NULL                 COMMENT '关联业务单号（幂等依据）',
  `remark`            VARCHAR(500)  NOT NULL DEFAULT ''      COMMENT '备注（操作原因/说明）',
  -- [通用审计字段]
  `create_by`         BIGINT        NOT NULL DEFAULT 0       COMMENT '创建人ID',
  `create_by_name`    VARCHAR(50)   NOT NULL DEFAULT ''      COMMENT '创建人名称',
  `create_at`         BIGINT        NOT NULL DEFAULT 0       COMMENT '创建时间（毫秒时间戳）',
  `update_by`         BIGINT        NOT NULL DEFAULT 0       COMMENT '更新人ID',
  `update_by_name`    VARCHAR(50)   NOT NULL DEFAULT ''      COMMENT '更新人名称',
  `update_at`         BIGINT        NOT NULL DEFAULT 0       COMMENT '更新时间（毫秒时间戳）',
  `delete_at`         BIGINT        NOT NULL DEFAULT 0       COMMENT '软删除（0=未删除）',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_order_log_type` (`order_id`, `log_type`),
  KEY `idx_user_currency_time` (`user_id`, `currency_code`, `create_at`),
  KEY `idx_user_type_time` (`user_id`, `log_type`, `create_at`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='余额变更流水表';
```

### 4.3 字段说明

| 字段 | 类型 | 需求来源 | 说明 |
|------|------|---------|------|
| user_id | BIGINT | - | 余额变更的用户 |
| currency_code | VARCHAR(10) | 需求理解 2.1 | 变更涉及的币种 |
| wallet_type | INT | 需求理解 2.2 | 变更涉及的钱包类型 |
| log_type | INT | 链路关系 5.1 | 12 种流水类型，见枚举 2.5 |
| amount | DECIMAL(20,8) | - | 变更金额，始终为正数 |
| direction | INT | - | 1=收入（余额增加）2=支出（余额减少） |
| balance_before | DECIMAL(20,8) | 架构设计 5.4 | 变更前的 available_balance |
| balance_after | DECIMAL(20,8) | 架构设计 5.4 | 变更后的 available_balance |
| order_id | VARCHAR(50) | 架构设计 5.2 | 关联的业务单号（充值单/提现单/加款单等） |
| remark | VARCHAR(500) | 链路关系 R6/R7 | 操作原因，人工加款/减款时必填 |

### 4.4 索引说明

| 索引 | 类型 | 用途 |
|------|------|------|
| uk_order_log_type | UNIQUE | **幂等去重**：同一 order_id + 同一 log_type 不允许重复插入 |
| idx_user_currency_time | 普通 | C端记录查询：按用户+币种+时间排序分页 |
| idx_user_type_time | 普通 | 按流水类型筛选：如只看充值流水、只看提现流水 |

### 4.5 幂等机制

```
同一笔充值订单 C123456 只能产生一条 log_type=1（充值入账）的流水
同一笔提现订单 T789012 会产生两条流水：
  - log_type=2（提现冻结）— 创建订单时
  - log_type=3（提现扣款）— 出款成功时
  或：
  - log_type=2（提现冻结）— 创建订单时
  - log_type=4（提现解冻）— 驳回时

(order_id, log_type) 唯一索引保证每种操作只执行一次
```

## 五、表 ③ deposit_order — 充值订单表

### 5.1 需求来源

| 需求概念 | 映射为 | 来源 |
|---------|--------|------|
| 用户充值产生订单 | deposit_order 表 | 需求理解 3.1 |
| 订单号 C + 16位 | order_no 字段 | 需求理解 7.2 |
| 充值有方式、金额、状态 | method_type, amount, status | 需求理解 3.1 |
| 奖金活动选择 | bonus_activity_id, bonus_ratio, bonus_amount | 需求理解 3.1：奖金活动选择规则 |
| B端充值记录 24 个字段 | 表字段覆盖 | 需求理解 5.3 |

### 5.2 DDL

```sql
CREATE TABLE `deposit_order` (
  `id`                    BIGINT          NOT NULL AUTO_INCREMENT  COMMENT '主键ID',
  `order_no`              VARCHAR(50)     NOT NULL                 COMMENT '平台订单号（C+SnowflakeID）',
  `user_id`               BIGINT          NOT NULL                 COMMENT '用户ID',
  `user_nickname`         VARCHAR(100)    NOT NULL DEFAULT ''      COMMENT '用户昵称（冗余，方便B端展示）',
  `currency_code`         VARCHAR(10)     NOT NULL                 COMMENT '订单币种代码',
  `method_type`           INT             NOT NULL                 COMMENT '充值方式类型（见枚举2.7）',

### 6.3 核心流转

```
创建订单: status = 1（待风控审核）
    │

    ├── 风控通过: status → 3（待财务审核）
    │       ├── 财务驳回: status → 4 → 钱包解冻(R4)
    │       ├── 财务通过: status → 5（待出款）
    │       │       ├── 出款失败: status → 6 → 钱包解冻(R4)
    │       │       └── 出款成功: status → 7 → 钱包确认扣款(R5)
```

└────────────────────────────────────────────────────────────────────────────┘
```

**关联说明**：

| 关联 | 类型 | 说明 |
|------|------|------|
| wallet.currency_code → currency_config.code | 逻辑外键（不建物理外键） | 钱包的币种由币种配置表定义 |
| transaction_log.order_id → 各订单表.order_no | 逻辑外键 | 流水关联到具体业务订单 |
| withdrawal_order.account_id → withdrawal_account.id | 逻辑外键 | 提现订单引用账户信息 |
| rate_change_log.currency_code → currency_config.code | 逻辑外键 | 汇率日志属于哪个币种 |

**不建物理外键的原因**：工程规范中所有服务都不使用数据库外键约束，由应用层保证数据一致性。这在微服务架构中是标准做法——物理外键会限制分库分表和独立部署。

## 十五、数据量估算与索引策略

### 15.1 数据量预估

| 表 | 增长模式 | 预估量级（半年） | 说明 |
|----|---------|---------------|------|
| wallet | 用户数 × 币种数 × 钱包类型 | 万级 | 每用户最多 5币种 × 5类型 = 25行 |
| transaction_log | 每笔操作至少1条 | **百万级** | 增长最快的表，充值/提现/兑换/奖励每次都写 |
| deposit_order | 每笔充值1条 | 十万级 | |
| withdrawal_order | 每笔提现1条 | 十万级 | |
| withdrawal_account | 每用户×方式1条 | 万级 | |
| exchange_record | 每笔兑换1条 | 十万级 | |
| reward_record | 每笔奖励1条 | 十万级 | |
| currency_config | 固定5条左右 | 个位数 | 初始写入，极少变更 |
| rate_change_log | 每次汇率更新1条/币种 | 千级 | 每天几次 × 几个币种 |
| system_config | 固定几条 | 个位数 | |

### 15.2 transaction_log 的特殊考量

这是数据量增长最快的表。索引策略：

- `uk_order_log_type`：幂等查询（高频写入时的唯一性检查）
- `idx_user_currency_time`：C端记录查询（按时间倒序分页）
- `idx_user_type_time`：按流水类型筛选

如果未来单表超过 1000 万行，可考虑按 `user_id` 范围分表。但当前阶段不过度设计。

### 16.2 数据模型的核心检验标准

在建表和编码前，用以下标准检验数据模型是否正确：

```
✓ 每个产品原型页面上的字段，都能在某张表中找到对应
✓ 每个 C端列表页的展示字段，都能通过单表查询获得（不需要复杂 JOIN）
✓ 每个 B端列表页的 20+ 字段，都能在表中直接映射
✓ 每个枚举值都有明确的需求来源
✓ 金额字段统一 DECIMAL(20,8)
✓ 时间字段统一 BIGINT 毫秒时间戳
✓ 每张表都有通用审计字段
✓ wallet 表的双字段设计能覆盖冻结/解冻/扣款所有场景
✓ transaction_log 的唯一索引能满足幂等要求
✓ 订单表的汇率快照字段确保不依赖后续汇率变动
```
