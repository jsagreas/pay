## 1. 映射方法论

### 1.1 从现实世界到数据模型的转换思路

```
产品需求（现实世界描述）
  │
  ├── "用户有多个币种的钱包"       ──→  wallet 表（按用户+币种+类型唯一）
  ├── "每笔资金变动都要有记录"     ──→  wallet_flow 表（append-only 流水）
  ├── "提现时余额要冻结"           ──→  freeze_detail 表（冻结明细）
  ├── "用户可以充值"               ──→  recharge_order 表（充值订单）
  ├── "用户可以提现"               ──→  withdraw_order 表（提现订单）
  ├── "法币可以兑换BSB"            ──→  exchange_record 表（兑换记录）
  ├── "用户绑定提现账户"           ──→  withdraw_account 表（提现账户）
  ├── "后台管理币种参数"           ──→  currency_config 表（币种配置）
  ├── "汇率变动要有日志"           ──→  rate_log 表（汇率变动日志）
  └── "设置平台基准币种"           ──→  base_currency_config 表（基准币种）
```

### 1.2 转换原则

| 原则 | 说明 | 示例 |
|------|------|------|
| **一个业务实体 = 一张表** | 独立的业务概念独立建表，不混合 | 充值订单和提现订单分开，不合一张"订单表" |
| **现实属性 = 表字段** | 产品原型中展示的信息对应字段 | 充值记录详情页展示的每个信息点都有字段承载 |
| **业务规则 = 约束/索引** | 唯一性、非空等规则转为数据库约束 | "同用户同币种同钱包类型唯一" → UNIQUE 索引 |
| **状态流转 = 状态字段+枚举** | 产品流程中的状态节点转为枚举值 | 提现订单状态：待审核→审核通过→出款中→成功/失败 |
| **关联关系 = 外键字段** | 实体间的关联用 ID 字段逻辑关联（不用物理外键） | 流水表的 wallet_id 关联钱包表 |
| **金额 = int64 最小单位** | 架构决策：所有金额用最小单位整数存储 | 1500.50 THB → 存 150050（×100） |
| **汇率 = int64 ×10^8** | 架构决策：汇率统一乘以 10^8 转整数 | 1 BSB = 2300 VND → 存 230000000000 |

后文每张表的字段都标注了需求来源，格式如下：

## 2. 模型规范约定

### 2.1 GORM 模型规范（基于现有工程代码分析）

| 规范项 | 约定 | 工程依据 |
|--------|------|---------|
| 主键 | `int64`, `primaryKey;autoIncrement:true` | ser-app Banner、ser-item Item 等 |
| 时间戳 | `int64` 毫秒时间戳 | ser-app Banner 模型 |
| 软删除 | `delete_at int64 NOT NULL DEFAULT 0`，0=未删除 | ser-app Banner 模型 |
| 可空字段 | 指针类型 `*string`, `*int64` | ser-buser SysUser 模型 |
| 加密字段 | `utils.EncryptedString` | common/pkg/utils/crypto_util.go |
| Tag 格式 | `gorm:"column:xxx;not null;comment:xxx" json:"xxx"` | 全工程统一 |
| 字段命名 | snake_case（数据库列名 = Go 字段的 snake 形式） | 全工程统一 |
| 关联方式 | 逻辑外键（ID 字段），不使用 GORM 原生关联 | ser-buser sys_user_role 等 |
| 状态字段 | `int32` 类型，配合枚举常量 | ser-app、ser-item 等 |

### 2.2 金额存储规范（架构决策）

| 币种 | 展示精度 | precision_factor | 换算规则 | 存储示例 |
|------|---------|------------------|---------|---------|
| VND | 0位小数 | 1 | 1:1 | 5,000,000 VND → 存 `5000000` |
| IDR | 0位小数 | 1 | 1:1 | 500,000 IDR → 存 `500000` |
| THB | 2位小数 | 100 | ×100 | 1,500.50 THB → 存 `150050` |
| BSB | 2位小数 | 100 | ×100 | 200.00 BSB → 存 `20000` |
| USDT | 8位小数 | 100000000 | ×10^8 | 1.50000000 USDT → 存 `150000000` |

**Go 模型中金额字段统一使用 `int64`**，数据库列类型为 `BIGINT NOT NULL DEFAULT 0`。

### 2.3 通用基础字段模板

每张表都包含以下基础字段：

```go
// 每张表的标准尾部字段
CreateAt  int64  `gorm:"column:create_at;not null;comment:创建时间（毫秒时间戳）" json:"create_at"`
UpdateAt  int64  `gorm:"column:update_at;not null;comment:更新时间（毫秒时间戳）" json:"update_at"`
DeleteAt  int64  `gorm:"column:delete_at;not null;default:0;comment:删除标记（0=未删除，非0=删除时间戳）" json:"delete_at"`
```

## 4. 核心模型 — 钱包与资金流水

### 4.1 wallet 表 — 用户钱包

**现实世界描述**：
> "钱包按币种维度隔离，每个币种独立余额与账本。每个币种下有多个子钱包：中心钱包、奖励钱包、主播钱包、代理钱包、场馆钱包。" — [需求 2.1 钱包结构]

**转换逻辑**：
- "按币种隔离" → `currency_code` 字段
- "多个子钱包" → `wallet_type` 字段（枚举区分）
- "独立余额" → `balance` 字段
- "提现时冻结" → `frozen_amount` 字段
- "可用余额 = 余额 - 冻结" → 计算字段，不额外存储
- "同用户同币种同类型唯一" → UNIQUE 约束

#### DDL

```sql
CREATE TABLE wallet (
    id              BIGINT       NOT NULL AUTO_INCREMENT COMMENT '主键',
    user_id         BIGINT       NOT NULL COMMENT '用户ID',
    currency_code   VARCHAR(10)  NOT NULL COMMENT '币种代码(BSB/VND/IDR/THB/USDT)',
    wallet_type     TINYINT      NOT NULL COMMENT '钱包类型(1中心/2奖励/3主播/4代理/5场馆)',
    balance         BIGINT       NOT NULL DEFAULT 0 COMMENT '余额（最小单位整数）',
    frozen_amount   BIGINT       NOT NULL DEFAULT 0 COMMENT '冻结金额（最小单位整数）',
    status          TINYINT      NOT NULL DEFAULT 1 COMMENT '状态(1正常/2冻结/3关闭)',
    create_at       BIGINT       NOT NULL COMMENT '创建时间（毫秒时间戳）',
    update_at       BIGINT       NOT NULL COMMENT '更新时间（毫秒时间戳）',
    delete_at       BIGINT       NOT NULL DEFAULT 0 COMMENT '删除标记（0=未删除）',
    PRIMARY KEY (id),
    UNIQUE KEY uk_user_currency_type (user_id, currency_code, wallet_type, delete_at),
    KEY idx_user_id (user_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户钱包';
```

#### Go Model

```go

    CurrencyCode string `gorm:"column:currency_code;not null;comment:币种代码(BSB/VND/IDR/THB/USDT)" json:"currency_code"`
    WalletType   int32  `gorm:"column:wallet_type;not null;comment:钱包类型(1中心/2奖励/3主播/4代理/5场馆)" json:"wallet_type"`
    Balance      int64  `gorm:"column:balance;not null;default:0;comment:余额（最小单位整数）" json:"balance"`
    FrozenAmount int64  `gorm:"column:frozen_amount;not null;default:0;comment:冻结金额（最小单位整数）" json:"frozen_amount"`
    Status       int32  `gorm:"column:status;not null;default:1;comment:状态(1正常/2冻结/3关闭)" json:"status"`

}

```

#### 字段与需求的映射

| 字段 | 需求来源 | 说明 |
|------|---------|------|
| user_id | [需求 2.1] 每个用户有钱包 | 关联 ser-user 的用户ID |
| currency_code | [需求 2.1] 钱包按币种维度隔离 | BSB/VND/IDR/THB/USDT |
| wallet_type | [需求 2.1] 中心/奖励/主播/代理/场馆 | 枚举值，详见第10章 |
| balance | [需求 2.1] 独立余额 | int64 最小单位，[架构决策3] |
| frozen_amount | [需求 5.1] 提现需冻结余额 | 可用余额 = balance - frozen_amount |
| status | [需求 5.1] 禁用币种后不可操作 | 正常/冻结/关闭 |

#### 关键约束

- **UNIQUE (user_id, currency_code, wallet_type, delete_at)**：保证同一用户同一币种同一钱包类型只有一行有效记录
- **balance >= 0**：代码层强校验，余额不允许为负
- **frozen_amount >= 0 且 frozen_amount <= balance**：代码层校验

#### 数据量预估

假设 10万用户 × 5币种 × 2钱包类型（中心+奖励）= 100万行（可控）

### 4.2 wallet_flow 表 — 资金流水

**现实世界描述**：
> "账务流水与余额必须强一致，不允许出现余额与流水不一致的情况。" — [需求 9 非功能性需求]
> "记录分四大类tab：充值记录、提现记录、兑换记录、奖励记录。每条记录显示：类型、金额、状态、时间。" — [需求 6.1 记录分类]

**转换逻辑**：
- "每笔资金变动都有记录" → 每次 CreditWallet / DebitWallet / Freeze / Unfreeze 都写一条流水
- "余额与流水强一致" → before_balance / after_balance 字段，同事务写入
- "充值回调必须幂等" → idempotent_key 唯一约束
- "记录详情展示订单号/金额/状态/时间" → 对应字段

#### DDL

```sql
CREATE TABLE wallet_flow (
    id              BIGINT       NOT NULL AUTO_INCREMENT COMMENT '主键',
    user_id         BIGINT       NOT NULL COMMENT '用户ID',
    currency_code   VARCHAR(10)  NOT NULL COMMENT '币种代码',
    wallet_type     TINYINT      NOT NULL COMMENT '钱包类型',
    direction       TINYINT      NOT NULL COMMENT '方向(1入账/2出账/3冻结/4解冻)',
    biz_type        TINYINT      NOT NULL COMMENT '业务类型(1充值/2提现/3兑换/4奖金/5人工加款/6人工减款/7补单/8冻结/9解冻/10确认扣款)',
    amount          BIGINT       NOT NULL COMMENT '变动金额（最小单位，正数）',
    before_balance  BIGINT       NOT NULL COMMENT '变动前余额',
    after_balance   BIGINT       NOT NULL COMMENT '变动后余额',
    ref_order_no    VARCHAR(64)  NOT NULL DEFAULT '' COMMENT '关联业务订单号',
    idempotent_key  VARCHAR(128) NOT NULL COMMENT '幂等键',
    remark          VARCHAR(256) NOT NULL DEFAULT '' COMMENT '备注',
    create_at       BIGINT       NOT NULL COMMENT '创建时间（毫秒时间戳）',
    update_at       BIGINT       NOT NULL COMMENT '更新时间（毫秒时间戳）',
    delete_at       BIGINT       NOT NULL DEFAULT 0 COMMENT '删除标记',
    PRIMARY KEY (id),
    UNIQUE KEY uk_idempotent (idempotent_key, delete_at),
    KEY idx_user_currency_time (user_id, currency_code, create_at),
    KEY idx_ref_order (ref_order_no)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='资金流水';
```

#### 字段与需求的映射

| 字段 | 需求来源 | 说明 |
|------|---------|------|
| direction | [需求 6.1] 记录分充值/提现/兑换/奖励 | 入账/出账/冻结/解冻 |
| biz_type | [需求 6.1] 区分充值/提现/兑换/奖励等 | 10 种业务类型，详见第10章 |
| amount | [需求 6.2] 每条记录显示金额 | 正数，方向由 direction 表达 |
| before_balance / after_balance | [需求 9] 余额与流水强一致 | 可用于审计和余额重建 |
| ref_order_no | [需求 8.4] 详情页展示订单号 | 关联充值/提现/兑换订单号 |
| idempotent_key | [需求 9] 充值回调必须幂等 | 格式：`{biz_type}:{ref_order_no}:{wallet_type}` |

#### 关键设计

- **只追加、不更新、不删除**（append-only ledger）— 这是资金流水表的核心原则
- **idempotent_key 唯一约束**：幂等性的最终防线，同一笔业务操作不会产生重复流水
- **before_balance / after_balance**：形成链式审计轨迹，可验证余额连续性
- **索引 (user_id, currency_code, create_at)**：支撑 C 端记录列表的分页查询

#### 数据量预估

每笔资金操作至少 1 条流水，活跃用户每日 5-20 笔 → 日增量约 50万-200万行（高增长表，需关注）

### 4.3 freeze_detail 表 — 冻结明细

**现实世界描述**：
> "提现采用 冻结 → 确认扣款/解冻 三阶段。" — [架构决策5]
> "消费扣款优先级：先扣中心钱包，不足再扣奖励钱包。" — [需求 2.2]

**为什么需要这张表**：
提现冻结时可能跨两个钱包（中心钱包 1200 + 奖励钱包 300 = 总冻结 1500）。解冻/确认扣款时需要精确知道每个钱包冻结了多少，按明细还原。如果没有这张表，解冻时无法正确分配回每个钱包。

#### DDL

```sql
CREATE TABLE freeze_detail (
    id              BIGINT       NOT NULL AUTO_INCREMENT COMMENT '主键',
    user_id         BIGINT       NOT NULL COMMENT '用户ID',
    currency_code   VARCHAR(10)  NOT NULL COMMENT '币种代码',
    wallet_type     TINYINT      NOT NULL COMMENT '钱包类型',
    order_no        VARCHAR(64)  NOT NULL COMMENT '关联提现订单号',
    freeze_amount   BIGINT       NOT NULL COMMENT '冻结金额（最小单位）',
    status          TINYINT      NOT NULL DEFAULT 1 COMMENT '状态(1冻结中/2已确认扣款/3已解冻)',
    create_at       BIGINT       NOT NULL COMMENT '创建时间（毫秒时间戳）',
    update_at       BIGINT       NOT NULL COMMENT '更新时间（毫秒时间戳）',
    delete_at       BIGINT       NOT NULL DEFAULT 0 COMMENT '删除标记',
    PRIMARY KEY (id),
    KEY idx_order_no (order_no),
    KEY idx_user_currency (user_id, currency_code)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='冻结明细';
```


```
① FreezeBalance 创建 2 条 freeze_detail：

③ 或审核驳回 → UnfreezeBalance：
   - 中心: status=1→3, wallet.frozen_amount -= 1200（可用余额恢复）
   - 奖励: status=1→3, wallet.frozen_amount -= 300（可用余额恢复）
```

#### 兑换金额计算示例

```
用户兑换 5,000,000 VND → BSB（汇率 1 BSB = 2,300 VND，赠送 10%）

from_amount = 5000000（VND最小单位，precision_factor=1）
exchange_rate = 230000000000（2300 × 10^8）
to_amount = 5000000 × 10^8 / 230000000000 × 100 = 217391（2173.91 BSB × 100）
bonus_rate = 1000（10.00%）
bonus_amount = 217391 × 1000 / 10000 = 21739（217.39 BSB × 100）

结果：扣 5,000,000 VND → 中心钱包 +2173.91 BSB → 奖励钱包 +217.39 BSB
```

### 9.3 base_currency_config 表 — 基准币种配置

**现实世界描述**：
> "推荐设置USDT为基准币种。基准币种一旦设置不可更改（影响全局计算逻辑）。BSB锚定基准币种：10 BSB = 1 USDT。" — [需求（二）4]

**为什么独立建表**：
基准币种是全局唯一的配置项，且设置后不可更改。如果放在 currency_config 中用标记字段表示，逻辑较分散。独立建表语义清晰：这张表永远只有一行数据。

#### DDL

```sql
CREATE TABLE base_currency_config (
    id               BIGINT       NOT NULL AUTO_INCREMENT COMMENT '主键',
    currency_code    VARCHAR(10)  NOT NULL COMMENT '基准币种代码(推荐USDT)',
    bsb_anchor_rate  BIGINT       NOT NULL DEFAULT 0 COMMENT 'BSB锚定汇率（×10^8，如10BSB=1USDT→存1000000000）',
    set_by           BIGINT       NOT NULL DEFAULT 0 COMMENT '设置人ID',
    create_at        BIGINT       NOT NULL COMMENT '创建时间（毫秒时间戳）',
    update_at        BIGINT       NOT NULL COMMENT '更新时间（毫秒时间戳）',
    delete_at        BIGINT       NOT NULL DEFAULT 0 COMMENT '删除标记',
    PRIMARY KEY (id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='基准币种配置';
```

#### Go Model


#### 字段与需求的映射

| 字段 | 需求来源 | 说明 |
|------|---------|------|
| currency_code | [需求（二）4] 推荐 USDT | 设置后不可更改 |
| bsb_anchor_rate | [需求（二）4] 10 BSB = 1 USDT | 存 1000000000（10 × 10^8）|
| set_by | B 端操作审计 | 记录是谁设置的 |

#### 业务约束

- 此表最多只有 **1 行** 有效数据（delete_at = 0）
- 设置后 **不可修改**（代码层检查 `SELECT COUNT(*) WHERE delete_at=0 > 0 → 拒绝`）
- 读取时使用进程缓存，启动时加载一次即可

## 12. 索引设计策略

### 12.1 索引总览

| 表 | 索引名 | 字段 | 类型 | 用途 |
|----|--------|------|------|------|
| wallet | uk_user_currency_type | (user_id, currency_code, wallet_type, delete_at) | UNIQUE | 业务唯一约束 |

| freeze_detail | idx_order_no | (order_no) | INDEX | 按提现订单查冻结明细 |
| freeze_detail | idx_user_currency | (user_id, currency_code) | INDEX | 按用户查冻结 |
| recharge_order | uk_order_no | (order_no, delete_at) | UNIQUE | 订单号唯一 |
| recharge_order | idx_user_currency_time | (user_id, currency_code, create_at) | INDEX | C 端充值记录 |

| withdraw_order | idx_user_daily | (user_id, create_at) | INDEX | 每日提现限制 |
| exchange_record | uk_order_no | (order_no, delete_at) | UNIQUE | 订单号唯一 |
| exchange_record | idx_user_currency_time | (user_id, from_currency, create_at) | INDEX | C 端兑换记录 |
| withdraw_account | idx_user_currency | (user_id, currency_code, delete_at) | INDEX | 用户账户列表 |
| currency_config | uk_code | (currency_code, delete_at) | UNIQUE | 币种代码唯一 |
| currency_config | idx_status_sort | (status, sort_order) | INDEX | C 端币种列表 |
| rate_log | idx_currency_time | (currency_code, create_at) | INDEX | B 端按币种时间查 |

### 12.2 索引设计原则

1. **UNIQUE 索引都带 delete_at**：配合软删除，删除后的记录不影响唯一性约束
2. **C 端查询索引都以 (user_id, currency_code, create_at) 结尾**：统一的记录列表查询模式
3. **不建过多索引**：每张表控制在 3-5 个索引内，避免写性能下降
4. **联合索引遵循最左匹配**：查询条件的字段顺序与索引顺序一致

### 13.2 数据更新频率

| 表 | 更新频率 | 热点字段 |
|----|---------|---------|
| wallet | **高频** — 每次资金操作 | balance, frozen_amount, update_at |
| wallet_flow | **不更新** — append-only | — |
| freeze_detail | **低频** — 仅状态变更 | status, update_at |
| recharge_order | **低频** — 回调时更新一次 | status, actual_amount, paid_at, channel_order_no |
| withdraw_order | **低频** — 审核各阶段更新 | status, reviewer fields, completed_at |
| exchange_record | **不更新** — 创建即终态 | — |
| withdraw_account | **极低频** — 用户修改账户 | — |
| currency_config | **低频** — B 端编辑 + 汇率定时更新 | real_rate, platform_rate, deposit_rate, withdraw_rate |
| rate_log | **不更新** — append-only | — |
| base_currency_config | **不更新** — 设置后锁定 | — |

### 13.3 数据删除策略

- **全部使用软删除**（delete_at 标记）
- 资金相关数据（wallet、wallet_flow、freeze_detail、订单表）**永不物理删除**
- 提现账户可软删除（用户主动删除已保存的账户）
- 币种配置可软删除（但实际业务中极少删除，通常只禁用）
