#### TrxWallet（用户链上钱包地址表）

## 4. TRON 地址生成与私钥管理

### 4.1 地址生成流程

```java
// WalletTrxBusiness.createWallet()
public WalletAddressVO createWallet(TrxUserWalletRequest request) {
    String userId = request.getUserId();

    // 1. 检查是否已有地址（幂等）
    String existingAddress = trxWalletService.findAddressByUserId(userId);
    if (existingAddress != null) {
        return existingVO;  // 已存在直接返回
    }

    // 2. 解密主密码（AWS KMS）
    String passphrase = awsKmsDriver.decryptData(this.passphrase);

    // 3. 生成 ECDSA 密钥对（Web3j）
    ECKeyPair ecKeyPair = Keys.createEcKeyPair();

    // 4. 创建标准钱包文件
    WalletFile walletFile = Wallet.createStandard(passphrase, ecKeyPair);

    // 5. 将以太坊格式地址转为 TRON 格式（41 前缀 → T 前缀）
    String originAddress = walletFile.getAddress();    // 0x 开头
    String tronAddress = fromHexAddress("41" + originAddress);  // T 开头

    // 6. 提取私钥并加密存储
    String privateKey = Numeric.toHexStringNoPrefix(ecKeyPair.getPrivateKey());
    String encryptedKey = encrypt(privateKey, passphrase + tronAddress);
    //      ↑ AES-ECB 加密，密钥 = SHA-1(passphrase + address) 前 16 字节

    // 7. 持久化到 DB
    trxWalletService.save(userId, userName, encryptedKey, tronAddress);

    // 8. 异步通知 user-server 更新地址映射
    trxWalletService.updateUserAddress(request);

    return walletAddressVO;
}
```

### 4.2 地址格式转换

```
以太坊地址:  0x1234...abcd （20 字节）
      ↓  添加 0x41 前缀
TRON hex:   411234...abcd （21 字节）
      ↓  Base58Check 编码
TRON 地址:  T1234...XYZ  （Base58Check 格式）
```

**TRON 地址规则**：
- 前缀字节：`0x41`（十进制 65）
- 编码：Base58Check（双 SHA-256 校验 + Base58）
- 总长度：21 字节地址 + 4 字节校验 = 25 字节 → Base58 编码约 34 个字符

### 4.3 私钥加密方案

```
                    ┌─────────────────────────────────────┐
                    │         私钥加密存储方案               │
                    │                                      │
                    │  passphrase (主密码)                  │
                    │  ↓ AWS KMS 加密存储在 application.yml │
                    │  ↓ 运行时 KMS 解密得到明文             │
                    │                                      │
                    │  AES 密钥 = SHA-1(passphrase + address) │
                    │  ↓ 取前 16 字节                       │
                    │                                      │
                    │  加密: AES-ECB(privateKey, aesKey)    │
                    │  结果: Base64 编码后存入 DB            │
                    └─────────────────────────────────────┘

解密流程（转账时）:
1. 从 DB 读取 enPrivateKey
2. AWS KMS 解密 passphrase
3. AES 密钥 = SHA-1(passphrase + address)[:16]
4. AES-ECB 解密得到私钥明文
5. 用私钥签名交易
```

**安全评估**：
- AES-ECB 模式不够安全（无 IV，相同明文产生相同密文）→ 我方应使用 AES-GCM
- passphrase 通过 AWS KMS 保护 → 合理，我方可用类似云 KMS 方案
- 私钥与地址绑定（作为 AES 密钥的一部分）→ 增加安全性

## 7. 归集机制（Collect）

### 7.1 归集概念

**归集 = 将用户充值地址上的 USDT 转移到平台主地址（公司钱包）**

每个用户有独立的充值地址。用户充值后 USDT 留在用户地址上，需要通过归集操作将其转到平台统一管理的主地址。

### 7.2 单笔即时归集（充值触发）

```java
// TrxDriver.singleCollect()
public TrxWalletResult singleCollect(String fromAddress, String userId,
        String coinName, String ssOrderId, BigDecimal reChargeAmount, ...) {

    String mainAddress = awsKmsDriver.decryptData(companyAddress);  // 解密主地址

    // 1. 获取用户地址上的代币余额
    BigDecimal trxBalance = reChargeAmount;
    if (reChargeAmount == null) {
        trxBalance = getAccountBalance(fromAddress, coinName);
    }

    // 2. 余额校验
    if (trxBalance <= 0) return null;

    // 3. 如果是 TRC20 代币，检查用户地址的 TRX 余额（Gas 费）
    if (!isTRX(tokenContract)) {
        BigDecimal addTrx = getAccountBalance(fromAddress, "TRX");

            // 从 Gas 地址发送 30 TRX 到用户地址（补充手续费）
            createTransactionForTrx(gasPrivateKey, 30, gasAddress, fromAddress, TRX);
        }
    }

    // 4. 执行 TRC20 转账：用户地址 → 主地址
    TrxWalletResult result = createTransactionForTrx(
        null,          // null → 从 DB 解密用户私钥
        trxBalance,    // 全额转出
        fromAddress,   // 用户地址
        mainAddress,   // 公司主地址
        tokenContract  // USDT 合约地址
    );

    // 5. 记录归集信息
    reverseNoticeCollectDetail(userId, fromAddress, txId, trxBalance, ...);
    //    → trxWalletCollectService.save()   // 归集记录
    //    → trxWalletOrderService.save()     // 订单记录
}
```

### 8.2 外部签名服务调用

```java
// WalletTrxBusiness.sendSign() — 提现时调用
private byte[] sendSign(String signUrl, Protocol.Transaction tx,
        String ssOrderId, String from) {
    // 1. 计算交易哈希
    byte[] txIdHash = Sha256Sm3Hash.hash(tx.getRawData().toByteArray());
    String hashId = Base64.getEncoder().encodeToString(txIdHash);

    // 2. 构造签名请求
    Map<String, Object> signMap = new HashMap<>();
    signMap.put("orderNo", ssOrderId);
    signMap.put("outAddressNo", from);
    signMap.put("chainType", "TRON");
    signMap.put("hashId", hashId);

    // 3. RSA 加密请求体
    String encryptedBody = RSAUtil.encrypt(
        JSON.toJSONString(signMap),
        rsaPublicKey  // 从 TrxWalletConfig 表读取
    );

    // 4. HTTP POST 到签名服务
    HttpResponse response = HttpRequest.post(signUrl + "/sign")
        .header("PLAT_CODE", platCode)
        .body(encryptedBody)
        .timeout(60000)
        .execute();

    // 5. 获取签名值
    JSONObject result = JSON.parseObject(response.body());
    if ("200".equals(result.get("code"))) {
        byte[] signature = Base64.getDecoder().decode(result.getString("data"));
        // 将签名附加到交易
        return tx.toBuilder()
            .addSignature(ByteString.copyFrom(signature))
            .build().toByteArray();
    }
    return null;
}
```

### 8.3 本地私钥签名

```java
// WalletTrxBusiness.signTransaction2Byte() — 归集时调用
private static byte[] signTransaction2Byte(byte[] transaction, byte[] privateKey) {
    ECKey ecKey = ECKey.fromPrivate(privateKey);
    Protocol.Transaction tx = Protocol.Transaction.parseFrom(transaction);

    // SHA-256 哈希交易原始数据
    byte[] rawdata = tx.getRawData().toByteArray();
    byte[] hash = Sha256Sm3Hash.hash(rawdata);

    // ECDSA 签名
    byte[] sign = ecKey.sign(hash).toByteArray();

    // 将签名附加到交易
    return tx.toBuilder()
        .addSignature(ByteString.copyFrom(sign))
        .build().toByteArray();
}
```

### 8.4 签名安全设计总结

| 场景 | 签名方式 | 原因 |
|------|---------|------|
| 归集（用户→主地址） | 本地私钥签名 | 用户私钥在本服务 DB 中 |
| 提现（出款地址→用户） | 外部签名服务 | 出款私钥不在本服务中，由专门的签名服务管理 |
| Gas 充值（Gas→用户） | 本地私钥签名 | Gas 私钥在 application.yml（KMS 加密） |

**借鉴价值**：出款地址的私钥不存放在业务服务中，通过独立签名服务隔离，降低私钥泄露风险。这是生产环境的最佳实践。

## 10. Gas 费管理策略

## 11. 密钥安全体系

### 11.1 三层加密架构

```
第一层：AWS KMS

├── 解密方式：运行时调用 AWS KMS API
└── 安全级别：最高（HSM 保护）

第二层：AES 加密
├── 加密对象：用户私钥
├── 密钥来源：SHA-1(KMS解密的passphrase + 用户TRON地址)[:16]
├── 加密算法：AES/ECB/PKCS5Padding
├── 存储位置：TrxWallet 表 en_private_key 字段
└── 安全级别：中等（ECB 模式不够安全）

第三层：RSA 加密
├── 加密对象：签名服务通信内容
├── 密钥来源：TrxWalletConfig 表存储的 RSA 公钥
├── 用途：与外部签名服务的通信加密
└── 安全级别：高（非对称加密）
```

### 11.2 我方适配建议

| 参考工程做法 | 我方建议 |
|-------------|---------|
| AWS KMS | 可用阿里云 KMS / 自建 Vault |
| AES-ECB | 改用 AES-GCM（带 IV 和认证标签） |
| SHA-1 派生密钥 | 改用 PBKDF2 或 Argon2 |
| RSA 通信加密 | 可沿用或改用 mTLS |

## 17. 不宜照搬清单与理由

| # | 不宜照搬项 | 理由 |
|---|-----------|------|
| 1 | **AES-ECB 加密模式** | ECB 不安全（相同明文产生相同密文），应用 AES-GCM |
| 2 | **SHA-1 派生密钥** | SHA-1 已不推荐用于安全场景，应用 PBKDF2/Argon2 |
| 3 | **BigDecimal 金额** | Go 使用 int64 最小单位，更高效更安全 |
| 4 | **Hibernate DDL auto=update** | 生产环境不应自动修改表结构 |
| 5 | **HTTP POST 替代 MQ** | 可靠性不如 MQ，应走 NATS/Kafka |
| 6 | **明文密码在 application.yml** | 数据库密码等应通过 KMS 或 Vault 管理 |
| 7 | **硬编码币种枚举** | TRXContractEnum 每加币种需改代码，我方用动态配置 |
| 8 | **wallet-cli JAR 依赖** | 嵌入式 JAR 维护困难，Go 应用原生 TRON SDK |
| 9 | **TRON 测试网配置** | nile.trongrid.io 是测试网，生产需切主网 |
| 10 | **直接查 findAll** | collectTrc20 中 findAll 全量查用户，大规模时性能问题 |
| 11 | **缺少确认数检查** | 只查交易状态，不检查区块确认数，可能有重组风险 |
| 12 | **无链上事件监听** | 仅靠定时轮询，延迟 5 分钟。应增加事件监听实时感知 |

```go
param := abiEncode("transfer(address,uint256)",
```
