# 参考工程B钱包模块深度分析

> 文档定位：对生产运行的 Java 参考工程（tg-services）中钱包相关模块的**完整深度分析**，提取设计思想、数据模型、流程逻辑、技术选型等可借鉴信息。重点关注 **TRC20 虚拟币充值完整实现思路**。仅作参考，不照搬照抄，以我方需求为准。
>
> 分析范围：wallet-server（核心区块链钱包）+ wallet-third-server（三方游戏钱包）+ user-server（订单管理/余额账本）+ casino-*（后台/前台/代理）+ module-common（共享枚举/常量/VO）
>
> 参考工程路径：`/Users/mac/gitlab/z-tmp/tg-services`

## 二、数据模型分析

### 2.1 wallet-server 数据模型（链上钱包层）

#### 表1：TrxWallet（用户热钱包）

```
表名：TrxWallet
一用户一行，存储用户的 TRON 链上地址和加密私钥
```

| 字段 | 类型 | 说明 | 对标我方 |
|------|------|------|---------|
| id | BIGINT(PK) | 主键（BaseEntity 继承） | — |
| userId | VARCHAR | 用户 ID（业务主键） | wallet_address.user_id |
| userName | VARCHAR | 用户名 | — |
| address | VARCHAR | TRON 钱包地址（T 开头 Base58Check） | wallet_address.address |
| enPrivateKey | VARCHAR | AES 加密的私钥 | wallet_address.encrypted_key |
| createTime / updateTime | DATETIME | 审计时间 | — |
| createBy / updateBy | VARCHAR | 操作人 | — |

**私钥加密公式**：`AES.encrypt(privateKey, passphrase + address)`

> **评估**：一人一地址模式是虚拟币钱包标准设计。私钥用 AES 加密存数据库，密钥由 passphrase + address 组合，是合理的 KDF 方案。但比行业最佳实践（HSM/MPC）弱。我方可借鉴基本模式，密钥管理方案需单独评估。

#### 表2：TrxWalletOrder（链上交易订单）

```
表名：TrxWalletOrder
每笔链上交易一行，覆盖充值归集和提现两种方向
```

| 字段 | 类型 | 说明 | 对标我方 |
|------|------|------|---------|
| id | BIGINT(PK) | 主键 | — |
| userId | VARCHAR | 用户 ID | crypto_transaction.user_id |
| txId | VARCHAR | 链上交易哈希 | crypto_transaction.tx_hash |
| ssOrderId | VARCHAR | 关联业务订单号 | crypto_transaction.ref_order_no |
| amount | DECIMAL | 交易金额 | crypto_transaction.amount |
| fromAddress | VARCHAR | 发送方地址 | crypto_transaction.from_address |
| toAddress | VARCHAR | 接收方地址 | crypto_transaction.to_address |
| coinName | VARCHAR | 币种名称（TRX/USDT/CXT） | crypto_transaction.currency_code |
| orderType | INT | 1=充值归集 2=提现 | crypto_transaction.tx_type |
| postType | INT | 1=API调用 2=定时任务 | crypto_transaction.trigger_source |

**索引设计**：userId, orderType, txId 三个独立索引

> **评估**：orderType + postType 双分类设计值得借鉴。orderType 区分业务方向（充值/提现），postType 区分触发来源（用户操作/系统定时）。但缺少**交易状态字段**（pending/confirmed/failed），状态依赖调用链上 RPC 实时查询，这是一个设计弱点。

#### 表3：trx_wallet_collect（归集记录）

```
表名：trx_wallet_collect
用户热钱包 → 公司集中地址 的归集流水
```

| 字段 | 类型 | 说明 | 对标我方 |
|------|------|------|---------|
| id | BIGINT(PK) | 主键 | — |
| userId | VARCHAR | 用户 ID | collect_record.user_id |
| txId | VARCHAR | 归集交易哈希 | collect_record.tx_hash |
| collectAddress | VARCHAR | 源地址（用户钱包） | collect_record.from_address |
| amount | DECIMAL | 归集金额 | collect_record.amount |
| collectType | INT | 1=实时单笔 2=定时批量 | collect_record.collect_type |

**索引设计**：userId, collectType, txId 三个独立索引

> **评估**：归集与普通交易分表记录是好的设计，归集是平台内部操作，不应与用户充提混在一张表里。collectType 区分实时和定时也是合理的。我方可借鉴此分表思路。

#### 表4：TrxWalletConfig（系统配置）

```
表名：TrxWalletConfig
type + code 键值对形式存储加密配置项
```

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT(PK) | 主键 |
| type | VARCHAR | 配置分类（如 "sign"） |
| code | VARCHAR | 配置键（如 "rsa", "disPendAddress", "platCode"） |
| value | VARCHAR | 配置值（AWS KMS 加密） |

> **评估**：用数据库存加密配置是可行方案，但 type+code 二级分类粒度较粗。我方可用配置中心（ETCD）替代，更灵活也更安全。

### 2.2 user-server 数据模型（平台账本层）

#### 表5：SsOrder（主订单表）

```
表名：ss_order（推断）
充值和提现共用一张订单表，通过 orderType 区分
```

| 字段 | 类型 | 说明 | 对标我方 |
|------|------|------|---------|
| id | BIGINT(PK) | 主键 | deposit_order.id / withdraw_order.id |
| orderNum | VARCHAR | 订单号 | deposit_order.order_no |
| userId | BIGINT | 会员 ID | — |
| tgUserId | VARCHAR | Telegram 用户 ID | — |
| amount | DECIMAL | 订单金额 | deposit_order.amount |
| realAmount | DECIMAL | 实际到账金额 | deposit_order.real_amount |
| feeAmount | DECIMAL | 手续费 | withdraw_order.fee_amount |
| orderType | INT | 1=充值 2=提现 | 我方充提分表 |
| orderStatus | INT | 订单状态（见下方状态机） | deposit_order.status |
| virtualCurrency | VARCHAR | 虚拟币类型（USDT） | — |
| txId | VARCHAR | 链上交易哈希 | deposit_order.tx_hash |
| payeeAddress | VARCHAR | 收款地址（提现用） | withdraw_order.to_address |
| flowMultiple | DECIMAL | 流水倍数要求 | — |
| userType | INT | 1=正式 2=体验 3=机器人 4=试玩 | — |
| completeTime | DATETIME | 完成时间 | — |

**订单状态机**：

```
                     ┌─ ORDER_SUCCESS(2) ───── 完成
                     │
VIRTUAL_CURRENCY_WAIT(1) ─┼─ ORDER_FAILED(3) ───── 失败
   （初始状态）        │
                     ├─ ORDER_SYSTEM_CANCLE(99) ── 超时取消
                     │
                     ├─ ORDER_WAIT(1) ── 待审核（法币路径）
                     ├─ ORDER_AUDIT_ING(3) ── 审核中
                     ├─ ORDER_AUDIT_NO(4) ── 审核拒绝
                     ├─ ORDER_WITHDRAW_ING(5) ── 待出款
                     └─ ORDER_WITHDRAW_LOCK(6) ── 出款中
```

> **评估**：充提共用一张订单表简化了查询但耦合严重——充值字段（flowMultiple）和提现字段（payeeAddress、feeAmount）混在一起，会有大量空字段。**我方充提分表的设计更合理**。但状态机中虚拟币路径（WAIT→SUCCESS/CANCLE）只有3个状态，非常精简，值得借鉴。

#### 表6：Assets（用户资产）

```
表名：assets（推断）
用户余额账本，一人一行
```

| 字段 | 类型 | 说明 | 对标我方 |
|------|------|------|---------|
| id | BIGINT(PK) | 主键 | wallet_account.id |
| userId | BIGINT | 用户 ID | wallet_account.user_id |
| balance | DECIMAL | 主钱包余额 | wallet_account.available_balance |
| balanceBacc | DECIMAL | 百家乐子钱包 | — |
| balanceNn | DECIMAL | 牛牛子钱包 | — |
| freezeAmount | DECIMAL | 冻结金额 | wallet_account.frozen_balance |
| playMoney | DECIMAL | 流水金额（打码量） | — |
| creditAmount | DECIMAL | 信用额度 | — |
| version | INT | 乐观锁版本号 | wallet_account.version |

**余额模型**：

```
参考工程B：
  balance = 可用余额（主钱包）
  balanceBacc = 百家乐子钱包余额
  balanceNn = 牛牛子钱包余额
  freezeAmount = 冻结金额
  可提余额 = balance - freezeAmount

与参考工程A对比：
  参考A：amount(总额含冻结) - freeze_amount = 可用
  参考B：balance(可用) + freezeAmount(冻结) = 总额

我方设计：
  available_balance(可用) + frozen_balance(冻结) = 总额
```

> **评估**：参考B的余额模型与我方设计一致——可用和冻结分离存储。使用 JPA 乐观锁（version 字段）做并发控制，但比 DB 行锁弱（高并发下会频繁冲突重试）。**我方 DB 行锁方案更可靠**。游戏子钱包（balanceBacc/balanceNn）是"转账钱包"模式的产物，我方不需要。

#### 关键步骤详解

**步骤⑦ 查询链上余额（WalletBusiness.getBalance）**：

```
user-server → POST wallet-server:9600/wallet/getBalance
  请求：{ userId: "12345", currency: "USDT" }

wallet-server 内部：
  1. 根据 userId 查 TrxWallet 获取用户地址
  2. 调用 SmartContract.triggerConstantContract()
     → 发送 balanceOf(address) 到 USDT 合约
     → 解码 Uint256 返回值
     → 除以 10^6（USDT 6位精度）
  3. 返回 BigDecimal 余额
```

**步骤⑩⑪⑫⑬⑭ 归集转账（TrxDriver.singleCollect → WalletTrxBusiness.trc20Transaction）**：

```
TrxDriver.singleCollect() 核心逻辑：

1. 获取目标归集地址（KMS 解密 companyAddress）
2. 查询用户地址的 TRX 余额
3. 如果 TRX < 30：
   → 从 gasAddress 转 30 TRX 到用户地址（作为 Gas 费）
   → 等待 Gas 转账上链确认
4. 从用户地址转全部 USDT 到归集地址

WalletTrxBusiness.trc20Transaction() 核心逻辑：

1. 地址转换：Base58Check → Hex（T开头 → 41开头）
2. 构造合约调用参数：
   contract_address: USDT 合约地址（hex）
   function_selector: "transfer(address,uint256)"
   parameter: ABI 编码（目标地址 + 金额）
   fee_limit: 50,000,000 SUN（50 TRX 上限）
3. POST /wallet/triggersmartcontract → 获取未签名交易
4. 签名：
   hash = SHA256(transaction.rawData)
   signature = ECKey.sign(hash)
5. POST /wallet/broadcasthex → 广播已签名交易
6. 返回 txId
```

**步骤⑱ 到账入金（AsyncVirtualDepositService.updateOrderStatusAndSendMQ）**：

```
1. SsOrder.orderStatus = ORDER_SUCCESS(2)
2. SsOrder.completeTime = now
3. Assets.balance += realAmount（入金到主余额）
4. Assets.playMoney += realAmount * flowMultiple（增加打码量）
5. 写 BalanceChange 流水记录
6. MQ 通知 casino-backend 更新统计
7. 通知前端显示充值成功
```

**提现签名双路径**：

```

  3. POST {signUrl}/api/admin/chain/sign（带 PLAT_CODE 头）
  4. 解密返回签名
  5. 附加签名到交易

  1. ECKey.fromPrivate(privateKeyBytes)
  2. hash = Sha256Sm3Hash.hash(rawData)
  3. signature = ecKey.sign(hash)
  4. transaction.addSignature(signature)
```

> **评估**：提现使用外部签名服务是安全的最佳实践——出款私钥不在 wallet-server 内存中。我方可借鉴此分离设计。

```

    │   └── status = VIRTUAL_CURRENCY_WAIT
```

### 3.5 钱包地址生成流程

```
WalletTrxBusiness.createWallet()：

1. 检查用户是否已有钱包（userId 查 TrxWallet）
   └── 已有 → 返回现有地址

2. 生成密钥对：
   ECKeyPair ecKeyPair = Keys.createEcKeyPair()
   → 使用 web3j 库生成 secp256k1 密钥对

3. 创建钱包文件：
   WalletFile = Wallet.createStandard(passphrase, ecKeyPair)
   → 提取原始地址

4. 地址转换：
   TRON 地址 = fromHexAddress("41" + originAddress)
   → "41" 是 TRON 主网地址前缀
   → Base58Check 编码得到 "T" 开头地址

5. 加密私钥：
   privateKeyHex = Numeric.toHexStringNoPrefix(ecKeyPair.getPrivateKey())
   enPrivateKey = AES.encrypt(privateKeyHex, passphrase + address)

6. 入库：
   TrxWalletEntity { userId, userName, address, enPrivateKey }

7. 异步通知 user-server 更新钱包地址：
   POST /user/updateWalletAddressById
```

> **评估**：地址生成使用 web3j 的 ECKeyPair（secp256k1）是标准做法。"41" 前缀是 TRON 特有的链 ID 标识。私钥 AES 加密后存库是基本方案，但密钥明文曾经在内存中——生产环境应考虑 HSM 生成+存储。

## 五、安全机制分析

### 5.1 密钥管理三层架构

```
第一层：AWS KMS（系统级）
├── 加密对象：application.yml 中的敏感配置
├── 加密内容：companyAddress, gasAddress, gasPrivateKey, dispensingAddress 等
├── 使用方式：运行时 KMS SDK 调用解密
└── 安全等级：★★★★★（云厂商托管）

第二层：RSA（传输级）
├── 加密对象：外部签名服务的请求体
├── 公钥来源：TrxWalletConfig 表（code="rsa"）
├── 使用场景：提现签名请求
└── 安全等级：★★★★（非对称加密传输）

第三层：AES（存储级）
├── 加密对象：用户私钥（enPrivateKey 字段）
├── 密钥构造：passphrase + address（每用户唯一）
├── 算法：AES/ECB/PKCS5Padding + SHA-1 派生 16 字节密钥
├── 安全隐患：ECB 模式较弱，SHA-1 已过时
└── 安全等级：★★★（基本防护）
```

**安全隐患清单**：

| 编号 | 问题 | 严重度 | 说明 |
|------|------|--------|------|
| S1 | AES 使用 ECB 模式 | 中 | ECB 不使用 IV，相同明文产生相同密文，应改用 CBC/GCM |
| S2 | SHA-1 密钥派生 | 中 | SHA-1 已被证明不安全，应改用 PBKDF2/scrypt/Argon2 |
| S3 | AWS 密钥明文在 yml | 高 | accessKeyId/secretAccessKey 明文写在配置文件中 |
| S4 | 外部签名服务 HTTP 调用 | 高 | `http://34.92.121.9:80` 非 HTTPS，签名传输可被拦截 |
| S5 | 热钱包私钥在库内 | 中 | 数据库泄露 = 全部用户私钥泄露（虽有 AES 加密） |
| S6 | 无 API 限频 | 中 | WalletController 接口无 Rate Limiting |

> **评估**：安全机制有基本防护但存在明显短板。我方实现时应：（1）AES 改用 GCM 模式；（2）密钥派生改用 scrypt/Argon2；（3）配置敏感项使用 Vault/ETCD 加密存储；（4）签名服务必须 HTTPS + mTLS；（5）考虑 MPC 或 HSM 替代数据库存私钥。

### 5.2 分布式锁使用

```
wallet-third-server（三方游戏操作）：
  锁键：add_money_{userId}
  等待：30秒
  持有：60秒
  场景：三方游戏上下分（防并发双花）

user-server（充提操作）：
  锁键1：user_server_deposit_task_ss_order_id_{orderId}
  锁键2：user_server_withdraw_task_ss_order_id_{orderId}
  锁键3：prevent_time:virtualCurrencyCharge::{orderId}（3秒防重）
  场景：防止同一订单并发处理
```

> **评估**：per-order 锁粒度合理（不是 per-user 全局锁，不会阻塞同一用户的不同订单）。3 秒防重是额外保险层。我方用 DB 行锁 + Redis 防重可达到同等效果。

### 9.1 代码级缺陷

### 9.2 架构级缺陷

| 编号 | 问题 | 说明 |
|------|------|------|
| A1 | **充值检测靠轮询** | 2分钟轮询链上状态，延迟高且消耗 RPC 配额。应改为监听链上 Transfer 事件 |
| A2 | **无幂等性保障** | 虽有 Redis 标记，但无数据库级幂等（如唯一索引 ssOrderId + orderType） |
| A3 | **wallet-server 无事务保障** | 多表操作（TrxWalletOrder + TrxWalletCollect）无 @Transactional 注解 |
| A4 | **跨服务一致性弱** | user-server 与 wallet-server 之间无 Saga/TCC 补偿机制，链上成功+入金失败 = 资金丢失 |
| A5 | **归集与入金分离** | 归集成功不等于入金成功——归集记录在 wallet-server，入金操作在 user-server，中间靠 API 调用衔接，断链风险大 |

> **评估**：最严重的是 A4 和 A5——链上操作不可回滚，如果归集成功但 user-server 入金失败（网络断开/服务宕机），用户资金"消失"在归集地址中。生产环境中这类问题通过**对账系统**（reconciliation）兜底。我方设计应预留对账能力。

## 十一、技术实现细节

### 11.1 TRC20 ABI 编码

```
函数签名：transfer(address,uint256)

ABI 编码规则（Web3j）：
  address 参数：32字节左填充（去掉 41 前缀）
  uint256 参数：32字节大端整数

  示例 —— 转 100 USDT 到 TQZ... 地址：
  parameter =
    0000000000000000000000004173...（address 32字节）
    00000000000000000000000000000000000000000000000005F5E100（100 * 10^6）
```

### 11.2 TRON 地址格式转换

```
Base58Check 地址（用户可见）：
  TUyKqdNXNEzAvjiBWQJ3ooQFJoSYUAqLrR

Hex 地址（RPC 调用使用）：
  41 + 后续40个hex字符

转换公式：
  Base58Check → Hex：Base58.decode(address) → 取 [0:21] 字节 → Hex编码
  Hex → Base58Check：Hex解码 → 计算双SHA256校验和 → Base58.encode(payload + checksum[:4])

"41" 前缀 = TRON 主网标识符（类似 Ethereum 的 "0x"）
```

### 11.3 交易签名流程

```
1. 获取未签名交易（两种方式）：
   TRX 原生：createTransaction(from, to, amount) → Protocol.Transaction
   TRC20：POST /wallet/triggersmartcontract → JSON → packTransaction()

2. 设置时间戳和过期：
   timestamp = current milliseconds
   expiration = timestamp + 60_000（1分钟有效期）

3. 计算哈希：
   rawData = transaction.getRawData().toByteArray()
   hash = Sha256Sm3Hash.hash(rawData)
   → SHA-256（标准路径）或 SM3（国密路径，代码支持但未启用）

4. 签名：
   ecKey = ECKey.fromPrivate(privateKeyBytes)
   signature = ecKey.sign(hash).toByteArray()
   → secp256k1 椭圆曲线签名

5. 组装：
   signedTx = transaction.toBuilder()
     .addSignature(ByteString.copyFrom(signature))
     .build()

6. 广播：
   POST /wallet/broadcasthex
   body: { transaction: Hex.encode(signedTx.toByteArray()) }
   → 返回: { result: true, code: "SUCCESS", txid: "..." }
```

### 11.4 智能合约调用

```
只读调用（查余额/精度）：
  SmartContract.triggerConstantContract(ownerAddress, contractAddress, methodStr, data)
  → WalletApi.triggerCallContract() → GrpcClient.triggerConstantContract()
  → 返回 constant_result（不上链、不耗 Gas）

写入调用（转账）：
  SmartContract.triggerContract(ownerAddress, privateKey, contractAddress, methodStr, data, feeLimit)
  → WalletApi.triggerCallContract() → 获取未签名交易
  → 设置 fee_limit（最大 Gas 消耗上限）
  → 签名 + 广播
  → 返回 txId
```

## 十三、总结与镜像确认

### 13.1 最有价值的借鉴点

**排名前5**：

```
1. TRC20 充值全链路实现
   → 地址生成 → 用户转账 → 余额查询 → 归集转账 → 交易确认 → 入金
   → 这是我方最需要的完整实现参考

2. 四层地址分离体系
   → 热钱包 / 集中 / 出款 / Gas 四种角色明确分工
   → 安全架构的基石

3. Gas 费自动补充机制
   → TRC20 操作必须有 TRX 做 Gas
   → 自动检测 + 自动补充是必要设计

4. 归集双模式（定时 + 实时）
   → 大额即时归集 + 日常定时归集
   → 兼顾安全和效率

5. 外部签名服务分离
   → 出款私钥不在业务系统内
   → 最后一道安全防线
```
