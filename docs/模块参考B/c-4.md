## 三、数据模型深度分析

### 3.1 wallet-server 表结构（4 张表）

#### 表 1：TrxWallet —— 用户热钱包

```sql
CREATE TABLE TrxWallet (
    id              BIGINT       PRIMARY KEY AUTO_INCREMENT,
    user_id         VARCHAR(10)  NOT NULL,             -- 用户 ID
    user_name       VARCHAR(30),                       -- 用户名
    address         VARCHAR(100) NOT NULL,             -- ★ TRON 地址（T 开头 Base58Check）
    en_private_key  VARCHAR(250) NOT NULL,             -- ★ AES 加密的私钥
    create_time     DATETIME,
    create_by       VARCHAR(64),
    update_time     DATETIME,
    update_by       VARCHAR(64),
    INDEX idx_user_id (user_id)
);
-- 设计：一个用户一个 TRON 地址，私钥 AES 加密存储
-- 加密 Key = SHA1(passphrase + address)[0:16]
```

#### 表 2：TrxWalletOrder —— 交易记录

```sql
CREATE TABLE TrxWalletOrder (
    id              BIGINT       PRIMARY KEY AUTO_INCREMENT,
    user_id         VARCHAR(10)  NOT NULL,
    tx_id           VARCHAR(200),                      -- ★ TRON 链上交易哈希
    ss_order_id     VARCHAR(200),                      -- ★ 业务系统订单 ID（幂等 key）
    amount          DECIMAL(16,2) NOT NULL,            -- 交易金额
    from_address    VARCHAR(250),                      -- 发送方地址
    to_address      VARCHAR(250),                      -- 接收方地址
    coin_name       VARCHAR(10),                       -- 币种（TRX/USDT/CXT）
    order_type      TINYINT(2),                        -- 1=充值 2=提现
    post_type       TINYINT(2),                        -- 1=API 调用 2=定时任务
    create_time     DATETIME,
    create_by       VARCHAR(64),
    update_time     DATETIME,
    update_by       VARCHAR(64),
    INDEX idx_user_id (user_id),
    INDEX idx_order_type (order_type),
    INDEX idx_tx_id (tx_id)
);
```

#### 表 3：trx_wallet_collect —— 归集记录

```sql
CREATE TABLE trx_wallet_collect (
    id              BIGINT       PRIMARY KEY AUTO_INCREMENT,
    user_id         VARCHAR(10)  NOT NULL,
    tx_id           VARCHAR(200),                      -- 归集交易哈希
    collect_address VARCHAR(250),                      -- 来源地址
    amount          DECIMAL(16,2),                     -- 归集金额
    collect_type    TINYINT(2),                        -- 1=单笔即时 2=定时任务
    create_time     DATETIME,
    create_by       VARCHAR(64),
    update_time     DATETIME,
    update_by       VARCHAR(64),
    INDEX idx_user_id (user_id),
    INDEX idx_collect_type (collect_type),
    INDEX idx_tx_id (tx_id)
);
```

#### 表 4：TrxWalletConfig —— 配置存储

```sql
CREATE TABLE TrxWalletConfig (
    id              BIGINT       PRIMARY KEY AUTO_INCREMENT,
    type            VARCHAR(30),                       -- 配置类型（如 "sign"）
    code            VARCHAR(50),                       -- 配置编码（如 "rsa"/"platCode"/"disPendAddress"）
    value           VARCHAR(1000),                     -- 配置值（可加密）
    create_time     DATETIME,
    create_by       VARCHAR(64),
    update_time     DATETIME,
    update_by       VARCHAR(64)

-- 存储：RSA 公钥、平台编码、出款地址等动态配置
```

### 3.2 user-server 关键表（3 张）

#### 表 5：Assets —— 用户资产（余额）

```sql
CREATE TABLE assets (
    id              BIGINT       PRIMARY KEY AUTO_INCREMENT,
    user_id         BIGINT       NOT NULL UNIQUE,      -- ★ 用户 ID（唯一）
    balance         DECIMAL(16,2) DEFAULT 0,            -- ★ 主余额
    freeze_amount   DECIMAL(16,2) DEFAULT 0,            -- 冻结金额
    play_money      DECIMAL(16,2) DEFAULT 0,            -- 打码量（流水要求）
    credit_amount   DECIMAL(16,2) DEFAULT 0,            -- 信用额度
    version         INT          DEFAULT 1,             -- ★ 乐观锁版本号
    -- 省略其他游戏分余额字段
);
-- 设计：一个用户一条 Assets 记录，乐观锁控制并发
```

#### 表 6：SsOrder —— 充值/提现订单

```sql
CREATE TABLE ss_order (
    id              BIGINT       PRIMARY KEY AUTO_INCREMENT,
    user_id         BIGINT       NOT NULL,
    order_num       VARCHAR(64),                       -- 订单号
    amount          DECIMAL(16,2),                     -- 申请金额
    real_amount     DECIMAL(16,2),                     -- ★ 实际到账金额（链上确认后填入）
    virtual_currency VARCHAR(20),                      -- ★ 虚拟币种（USDT/TRX/CXT）
    order_type      TINYINT,                           -- ★ 1=充值 2=提现
    order_status    TINYINT,                           -- ★ 1=等待 2=成功 3=失败 4=取消
    tx_id           VARCHAR(200),                      -- ★ TRON 链上交易哈希
    payee_address   VARCHAR(200),                      -- 提现目标地址
    fee_amount      DECIMAL(16,2),                     -- 手续费
    flow_multiple   DECIMAL(10,2),                     -- ★ 流水倍数（打码要求）
    complete_time   DATETIME,                          -- 完成时间
    -- 省略代理层级、渠道等字段
    INDEX idx_user_id (user_id),
    INDEX idx_order_type (order_type),
    INDEX idx_order_status (order_status)
);
```

#### 表 7：User —— 用户账户

    -- 省略代理层级等字段
);

### 3.3 与我们表结构的对比

| 维度 | 参考工程 B | 我们的设计 | 评估 |
|------|-----------|-----------|------|
| **钱包地址** | TrxWallet 独立表存链上地址+私钥 | 我们的 wallet 表不存链上地址 | ★ 需要新增：链上地址存储 |
| **余额管理** | Assets 表管余额（平台内部），链上余额实时查 | wallet 表管余额 | 思路一致 |
| **订单管理** | SsOrder 表管充值/提现订单 | deposit_order + withdrawal_order 分表 | 我们更细分 |
| **交易记录** | TrxWalletOrder 记录链上交易 | wallet_transaction 记录平台流水 | 不同层级：链上 vs 平台 |
| **归集记录** | trx_wallet_collect 独立表 | 我们暂未设计 | ★ 需要评估是否需要 |
| **配置管理** | TrxWalletConfig 表存加密配置 | ETCD 配置中心 | 不同方案，功能等价 |
| **金额精度** | DECIMAL(16,2) / BigDecimal | BIGINT（最小单位） | 我们更精确 |
| **乐观锁** | Assets.version（@Version） | 待定（行锁 / 乐观锁） | 可参考 |
| **私钥存储** | AES 加密存 DB + AWS KMS | 待定 | ★ 需要设计安全方案 |

### 4.2 充值流程分阶段解析

#### 阶段 1：钱包地址生成（用户注册时）

```
触发：用户首次注册
入口：POST /wallet/create

WalletTrxBusiness.createWallet():
  1. 幂等校验：findAddressByUserId(userId) → 已存在则直接返回
  2. 解密 passphrase（AWS KMS）
  3. 生成密钥对：Keys.createEcKeyPair()（secp256k1 曲线）
  4. 创建钱包文件：Wallet.createStandard(passphrase, ecKeyPair)
  5. 提取 hex 地址，拼接 TRON 前缀 "41"
  6. 转换为 Base58Check 格式："T" 开头的地址
  7. 提取私钥 hex：Numeric.toHexStringNoPrefix(ecKeyPair.getPrivateKey())
  8. AES 加密私钥：encrypt(privateKeyHex, passphrase + address)
  9. 持久化：TrxWallet(userId, address, enPrivateKey)
  10. 异步通知 user-server 更新 User.walletAddress
```

**我们的平移方案**：

| 步骤 | 参考工程 | 我们的实现方向 |
|------|---------|-------------|
| 密钥对生成 | web3j Keys.createEcKeyPair() | Go: crypto/ecdsa + secp256k1 |
| 地址格式 | "41" + hex → Base58Check | 同 TRON 标准，Go 实现 Base58Check |
| 私钥加密 | AES/ECB + SHA1 密钥推导 | AES/GCM（更安全）+ ETCD 管理密码 |
| 私钥存储 | DB 加密存储 | DB 加密存储 + 可选 KMS |
| 异步通知 | @Async HTTP POST | Kitex RPC（如需跨服务）或同服务内调用 |

#### 阶段 2：用户链上转账（平台外部）

```
用户操作：
  1. 在 C 端看到平台生成的 TRON 地址 + 二维码
  2. 打开交易所或钱包 App
  3. 向该地址转入 USDT（TRC20）
  4. 等待 TRON 网络确认（通常 1-3 分钟）

平台不需要做任何操作——链上转账由用户自行完成
地址上的 USDT 余额会自动增加
```

#### 阶段 3：充值确认 + 归集（核心链路）

```
触发方式（2 种并行）：
  A. 用户主动点击"确认充值" → POST /virtual/charge
  B. 定时任务每 2 分钟轮询 → HandleVirtualCurrencyTask

AsyncVirtualDepositService.checkCrypotcurrencyOrder():

第一次进入（txId 为空）：
  ├── 查链上余额：walletBusiness.getBalance(userId, "USDT")
  │     → wallet-server → Trc20.balanceOf(address, contract)
  │     → 返回 BigDecimal 余额
  │
  ├── 校验余额 >= realAmount
  │
  ├── 发起归集：walletBusiness.getTransferResult(...)
  │     → wallet-server POST /wallet/collectTrc20
  │     → TrxDriver.singleCollect()
  │         ├── 检查 TRX 余额 >= 30（Gas 费）
  │         │   不足 → 从 gasAddress 转 30 TRX 过来
  │         │
  │         ├── 执行 TRC20 transfer
  │         │   from: 用户地址
  │         │   to: 主归集地址（companyAddress）
  │         │   → triggersmartcontract → 签名 → broadcasthex
  │         │
  │         ├── 记录 trx_wallet_collect + TrxWalletOrder
  │         └── 返回 txId
  │
  └── 更新 SsOrder.txId
      返回"下次进入"

第二次进入（txId 已有）：
  ├── 查链上交易状态：walletBusiness.getTransactionInfoById(orderId)
  │     → wallet-server → /wallet/gettransactionbyid
  │     → 检查 ret[0].contractRet == "SUCCESS"
  │
  ├── 如果确认成功：
  │     └── updateOrderStatusAndSendMQ()
  │           ├── 订单状态：WAIT → SUCCESS
  │           ├── ★ 上账：Assets.balance += realAmount
  │           ├── 打码量：Assets.playMoney += realAmount × flowMultiple
  │           └── 发 MQ 通知后台统计
  │
  └── 如果未确认：
        等待下次轮询
```

**关键设计思想提炼**：

1. **两阶段确认**：先归集（拿到 txId），再确认（检查链上状态）
2. **双触发机制**：用户主动 + 定时轮询，确保不遗漏
3. **Gas 费预充**：TRC20 转账需要 TRX 作为 Gas，自动从专用 Gas 地址补充
4. **归集模式**：用户地址上的币不直接用于余额，而是归集到主地址后再上账
5. **幂等保障**：ssOrderId 作为幂等 key，重复调用不会重复归集

#### 阶段 4：定时归集（补充机制）

```
触发：每天 08:00（曼谷时间）
入口：CollectTrxJob.collectUSDT()

TrxDriver.collectTrc20():
  遍历所有用户钱包（TrxWallet.findAll()）
  对每个用户地址：
    跳过主归集地址本身
    调用 singleCollect(userAddress, "USDT", UUID, COLLECT, 2)
    继续处理下一个（单个失败不影响整体）
```

这是**安全兜底机制**——即使用户没点确认、定时任务也没轮询到，每天定时归集确保资金不留在用户地址上。

### 5.3 安全分级

| 地址类型 | 私钥存储 | 签名方式 | 安全等级 | 风险暴露面 |
|---------|---------|---------|---------|-----------|
| 用户地址 | DB（AES 加密） | 本地签名 | 中 | wallet-server 被入侵可提取 |
| 主归集地址 | AWS KMS 加密配置 | 不需要签名（只收款） | 高 | 无签名风险 |
| 出款地址 | 外部签名服务 | RSA 加密请求 → 外部签名 | 最高 | 签名服务物理隔离 |
| Gas 地址 | AWS KMS 加密配置 | 本地签名 | 中 | 仅用于小额 Gas 补充 |

**核心安全思想**：

1. **出款地址私钥不在业务服务器** —— 即使 wallet-server 被完全攻破，攻击者也无法直接提走出款地址的资金
2. **用户地址的余额应尽快归集** —— 减少分散在大量热钱包中的资金量
3. **AWS KMS 管理敏感配置** —— 服务启动时解密，运行时明文只在内存中

## 七、TRC20 智能合约交互 —— 技术实现细节

### 7.1 TRC20 余额查询

```
Trc20.balanceOf(address, contractAddress):
  1. 地址转换：T 开头 Base58 → 41 开头 hex
  2. ABI 编码：encoderBalanceOfAbi(hexAddress)
     · 函数签名：balanceOf(address)
     · 参数：address 类型（Web3j Address）
  3. 调用只读合约：triggerConstantContract(ownerAddr, contractAddr, "balanceOf(address)", data)
     · TRON RPC: POST /wallet/triggerconstantcontract
  4. 解码返回值：decoderBalanceOf(hexResult)
     · 返回类型：Uint256
  5. 精度转换：result / 10^decimals
     · USDT: 10^6
     · CXT: 10^18
```

### 7.2 TRC20 转账

```
WalletTrxBusiness.trc20Transaction(keyStore, from, to, tokenContract, amount, ssOrderId):
  1. 验证接收地址：POST /wallet/validateaddress
  2. ABI 编码转账参数：
     · 函数：transfer(address,uint256)
     · 参数：toAddress (hex去"41") + amount * 10^decimals
  3. 触发智能合约：
     POST /wallet/triggersmartcontract
     {
       contract_address: tokenContract(hex),
       function_selector: "transfer(address,uint256)",
       parameter: ABI编码结果,
       owner_address: from(hex),
       call_value: 0,
       fee_limit: 50000000  // 50 TRX
     }
  4. 解析交易对象：TronUtils.packTransaction(transaction_json)
  5. 签名：
     if (from == dispensingAddress):
       sendSign(signUrl, tx, ssOrderId, from)  // 外部签名
     else:
       signTransaction2Byte(tx.toByteArray(), keyStore)  // 本地签名
  6. 广播：POST /wallet/broadcasthex
     { transaction: hex编码的签名交易 }
  7. 检查结果：code == "SUCCESS" → 返回 txId
```

### 7.3 外部签名服务交互

```
WalletTrxBusiness.sendSign(signUrl, tx, ssOrderId, from):
  1. 计算交易哈希：Sha256Sm3Hash.hash(tx.getRawData().toByteArray())
  2. Base64 编码哈希
  3. 构建请求体：
     {
       "orderNo": ssOrderId,
       "outAddressNo": fromAddress,
       "chainType": "TRON",
       "hashId": base64Hash
     }
  4. RSA 加密请求体：RSAUtil.encrypt(json, rsaPublicKey)
     · RSA 公钥从 TrxWalletConfig 读取
  5. HTTP POST 签名服务：
     URL: signUrl + "/api/admin/chain/sign"
     Headers: Content-Type=JSON, PLAT_CODE=平台编码
     Body: { "encData": rsaEncryptedJson }
  6. 解析返回的 Base64 签名
  7. 附加签名到交易：tx.addSignature(ByteString.copyFrom(signature))
  8. 返回签名后的交易字节
```

### 7.4 平移到 Go 的技术方案

| 操作 | Java 实现 | Go 平移方案 |
|------|----------|------------|
| 密钥对生成 | web3j Keys.createEcKeyPair() | go-ethereum crypto.GenerateKey() |
| 地址格式 | "41"+hex → Base58Check | github.com/mr-tron/base58 + 自实现 checksum |
| ABI 编码 | web3j FunctionEncoder | go-ethereum abi.Pack() |
| TRON RPC | HTTP POST (HttpClient4Util) | net/http 或 go-resty |
| 交易签名 | ECKey (TRON SDK) | 自实现 secp256k1 签名 或 go-tron-sdk |
| SHA256 | Sha256Sm3Hash | crypto/sha256 |
| Base58 | 自实现 Base58.java | github.com/mr-tron/base58 |
| RSA 加密 | RSAUtil（PKCS1） | crypto/rsa |
| AES 加密 | AES/ECB（不安全） | AES/GCM（更安全的模式） |

## 十二、与我们充值流程的完整对照

### 12.1 我们的 USDT 充值流程设计（融合参考工程 B 思想）

```
用户在 C 端选择 USDT 充值
  │
  ▼
① ser-wallet: CreateDepositOrder
  ├── 生成订单号
  ├── 查用户是否有 TRON 地址（无则生成）
  │     → 密钥对生成 + Base58Check 地址
  │     → AES/GCM 加密私钥存 DB
  ├── 创建 deposit_order（状态=待支付）
  └── 返回：TRON 充值地址 + 二维码

  ▼
② 用户在外部交易所/钱包转 USDT 到地址
   （平台不参与，等待链上确认）

  ▼
③ 确认机制（双触发）
  ├── 路径 A：用户在 C 端点击"已充值"
  │     → ser-wallet 接收确认请求
  └── 路径 B：定时任务（每 2 分钟）
        → 扫描 status=待支付 的 USDT 订单

  ▼
④ ser-wallet: 链上余额查询
  ├── 调用 TRON HTTP API: /wallet/triggerconstantcontract
  │     → balanceOf(userAddress)
  ├── 返回余额
  └── 校验余额 >= 订单金额

  ▼
⑤ ser-wallet: 归集操作
  ├── 检查用户地址 TRX >= 30（Gas 费）
  │     不足 → 从 Gas 地址转 TRX
  ├── 执行 TRC20 transfer
  │     from: 用户地址
  │     to: 主归集地址
  │     → triggersmartcontract → 签名 → broadcasthex
  ├── 获得 txId
  ├── 记录 wallet_transaction（归集类型）
  └── 更新 deposit_order.tx_id

  ▼
⑥ 定时任务: 确认链上状态
  ├── 查 TRON: /wallet/gettransactionbyid
  │     → contractRet == "SUCCESS"
  ├── 如果成功：
  │     ├── 更新 deposit_order 状态 → 已完成
  │     ├── 余额上账：wallet.balance += amount
  │     ├── 写入 wallet_transaction（充值类型）
  │     └── 如有奖金活动：奖金钱包入账 + 记录稽核要求
  └── 如果未确认：等待下次轮询

  ▼
⑦ 用户看到余额更新
```

## 十五、结论

### 15.1 参考工程 B 的核心价值

参考工程 B 给我们提供了一套**完整的 TRC20 虚拟币充值生产实现**，其核心价值在于：

1. **完整的链上充值链路**：从地址生成 → 用户转入 → 余额查询 → 归集 → 链上确认 → 上账，每个环节都有清晰的代码实现
2. **多地址安全架构**：用户地址/归集地址/出款地址/Gas 地址的职责分离是区块链钱包的行业最佳实践
3. **Gas 费自动管理**：TRC20 转账需要 TRX 作为 Gas 的自动检测和补充机制
4. **两阶段确认模式**：先归集拿到 txId，再查链上确认，保证了充值的可靠性
5. **双触发保障**：用户主动 + 定时轮询双重机制，确保每笔充值都能被处理

### 15.2 参考工程 B 的局限性

1. **安全实践有缺陷**：AES/ECB 模式不安全、AWS 密钥明文配置、安全认证全部关闭
2. **无审核流程**：提现直接执行，没有人工审核环节
3. **无冻结机制**：提现没有冻结/解冻流程
4. **无多币种余额**：用户只有一个 balance，不支持多币种分别管理
5. **归集时机单一**：只有每天定时归集 + 用户确认时归集，缺少灵活的触发策略
6. **缺少对账机制**：没有链上余额与平台余额的对账逻辑
