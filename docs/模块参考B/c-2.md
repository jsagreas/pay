## 3.2 TrxWallet — 用户链上钱包

```sql
CREATE TABLE TrxWallet (
    id              BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id         VARCHAR(10)   COMMENT '会员账号',         -- INDEX
    user_name       VARCHAR(30)   COMMENT '会员姓名',
    en_private_key  VARCHAR(250)  COMMENT '加密后的私钥',     -- AES(privateKey, passphrase+address)
    address         VARCHAR(100)  COMMENT '钱包地址',         -- TRON Base58 地址（T开头）
    create_time     DATETIME,
    update_time     DATETIME,
    ...
);
CREATE INDEX idx_userId ON TrxWallet(userId);
```

**核心设计**：
- **一人一地址**：每个 userId 对应唯一一个 TRON 链上地址
- **私钥加密落盘**：`en_private_key = AES_ECB(hex(privateKey), SHA1(passphrase+address)[:16])`
- **密码来源**：`passphrase` 是 BIP39 助记词，存储在 application.yml 中（经 AWS KMS 加密）
- **地址格式**：41 + hex(publicKey) → Base58Check 编码 → T 开头的 TRON 地址

## 3.3 TrxWalletOrder — 交易订单记录

```sql
CREATE TABLE TrxWalletOrder (
    id              BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id         VARCHAR(10)   COMMENT '会员账号',         -- INDEX
    tx_id           VARCHAR(200)  COMMENT '链上交易ID',       -- INDEX（TRON txHash）
    ss_order_id     VARCHAR(200)  COMMENT '充值/提现记录id',  -- 业务方订单号（幂等键）
    amount          DECIMAL(16,2) COMMENT '金额',
    from_address    VARCHAR(250)  COMMENT '发送地址',
    to_address      VARCHAR(250)  COMMENT '接收地址',
    coin_name       VARCHAR(10)   COMMENT '币种类型',         -- TRX / USDT / CXT
    order_type      TINYINT(2)    COMMENT '收付方向',         -- 1=充值 2=提现
    post_type       TINYINT(2)    COMMENT '调用方式',         -- 1=接口触发 2=定时任务
    create_time     DATETIME,
    update_time     DATETIME,
    ...
);
CREATE INDEX idx_user_id ON TrxWalletOrder(user_id);
CREATE INDEX idx_order_type ON TrxWalletOrder(order_type);
CREATE INDEX idx_tx_id ON TrxWalletOrder(tx_id);
```

**核心设计**：
- **双 ID 设计**：`tx_id`（链上交易哈希，链上唯一）+ `ss_order_id`（业务方订单号，业务幂等键）
- **金额精度**：`DECIMAL(16,2)` — 仅保留 2 位小数（注意：TRX/USDT 链上是 6 位精度，CXT 是 18 位）
- **收付区分**：`order_type` 区分充值/提现，不做方向反转（from→to 始终是链上真实方向）
- **触发来源追溯**：`post_type` 记录是 API 触发还是定时任务触发，方便审计

## 3.4 TrxWalletConfig — 钱包配置（加签系统参数）

```sql
CREATE TABLE TrxWalletConfig (
    id     BIGINT AUTO_INCREMENT PRIMARY KEY,
    type   VARCHAR(30)    COMMENT '类型',       -- 分组键（如 "sign"）
    code   VARCHAR(50)    COMMENT '标识code',   -- 参数名
    value  VARCHAR(1000)  COMMENT '标识value',  -- 参数值
    ...

```

**实际存储内容**（type="sign" 分组下）：

| code | 含义 | 示例 |
|------|------|------|
| `rsa` | 加签系统 RSA 公钥 | RSA 公钥字符串 |
| `platCode` | 平台识别码 | 请求头 PLAT_CODE |

**设计意图**：将加签系统相关的敏感配置从 application.yml 移到数据库，支持运行时更新。提现出款地址也存在这里而非配置文件——因为出款地址可能需要运营更换。

## 3.5 trx_wallet_collect — 归集记录

```sql
CREATE TABLE trx_wallet_collect (
    id              BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id         VARCHAR(10)   COMMENT '会员账号',   -- INDEX
```

# 四、三类系统地址与密钥安全体系

## 4.1 三类地址设计

```
┌──────────────────────────────────────────────────────┐
│                     地址体系                          │
├──────────────────┬──────────────┬────────────────────┤
│ 地址类型         │ 配置来源     │ 职责                │
├──────────────────┼──────────────┼────────────────────┤
│ mainCollect      │ yml(KMS加密) │ 归集收款地址        │
│ Address          │              │ 用户资金归集到此    │
├──────────────────┼──────────────┼────────────────────┤
│ mainDispensing   │ DB(Config表) │ 提现出款地址        │
│ Address          │              │ 用户提现从此出款    │
├──────────────────┼──────────────┼────────────────────┤
│ gasAddress       │ yml(KMS加密) │ 手续费充值地址      │
│                  │              │ 给用户地址补TRX gas │
├──────────────────┼──────────────┼────────────────────┤
│ companyAddress   │ yml(KMS加密) │ 公司主地址          │
│                  │              │ 归集目标地址        │
└──────────────────┴──────────────┴────────────────────┘
```

**设计要点**：
1. **归集地址 ≠ 出款地址**：归集（收款）和提现（出款）使用不同地址，实现资金流向隔离
2. **出款地址存 DB**：因为出款需要调用外部加签系统签名，地址和加签系统绑定，支持运营更换
3. **gasAddress 独立**：专门用于给用户地址转 TRX 作为 gas 费，避免从主钱包暴露大额地址

## 4.2 多层密钥安全架构

```
┌─────────────────────────────────────────────────────────────┐
│                    密钥安全分层                               │
├─────────────┬───────────────────────────────────────────────┤
│ 第1层       │ AWS KMS（服务端加密）                           │
│             │ 用途：加密 application.yml 中的所有敏感配置      │
│             │ 包括：归集地址、归集密码、gas地址、gas私钥等     │
│             │ Region: AP_SOUTHEAST_1                         │
├─────────────┼───────────────────────────────────────────────┤
│ 第2层       │ AES/ECB/PKCS5Padding（私钥落盘加密）           │
│             │ 用途：用户私钥存入 DB 前加密                    │
│             │ 密钥：SHA-1(passphrase + address)[:16]         │
│             │ passphrase = KMS解密后的助记词                  │
├─────────────┼───────────────────────────────────────────────┤
│ 第3层       │ RSA（加签系统通信加密）                         │
│             │ 用途：提现时与外部签名服务通信                   │
│             │ 流程：请求体 RSA 加密 → 签名值 Base64 返回     │
├─────────────┼───────────────────────────────────────────────┤
│ 第4层       │ 签名服务物理隔离                                │
│             │ 出款私钥不在 wallet-server 内存中               │
│             │ 由独立的加签系统持有并完成签名                   │
│             │ wallet-server 只拿到签名后的 bytes 用于广播     │
└─────────────┴───────────────────────────────────────────────┘
```

**安全分离原则**：
- **归集签名**：wallet-server 本地持有用户私钥（AES 解密后），本地签名
- **提现签名**：wallet-server **不持有出款地址私钥**，必须调用外部加签系统
- **判断逻辑**：`if(deDispensingAddress.equals(from))` → 调外部加签；否则 → 本地签名

这是一个重要的安全设计——归集只是把用户的钱归到公司地址，即使被攻击也只是归集失败（钱还在用户地址）；而提现是从公司出款，一旦私钥泄露将直接资金损失，所以必须隔离。

# 五、核心链路 ① — 创建钱包

```
用户注册 → user-server → POST /wallet/create
                            ↓
                    WalletController
                            ↓
                    WalletTrxBusiness.createWallet()
                            ↓
                ┌───────────────────────────┐
                │ 1. 查询 userId 是否已存在  │ → 已有则直接返回地址
                │ 2. ECKeyPair 生成密钥对    │ → Keys.createEcKeyPair()
                │ 3. 生成 WalletFile         │ → Wallet.createStandard(passphrase, ecKeyPair)
                │ 4. 提取地址                │ → "41" + hex(address) → Base58Check
                │ 5. 提取私钥hex             │ → Numeric.toHexStringNoPrefix(privateKey)
                │ 6. AES 加密私钥            │ → encrypt(privateKeyHex, passphrase+address)
                │ 7. 保存到 TrxWallet 表     │ → save(userId, userName, enKey, address)
                │ 8. 异步通知 user-server    │ → @Async updateUserAddress(HTTP POST)
                └───────────────────────────┘
                            ↓
                返回 WalletAddressVO{userId, userName, address}
```

**代码级细节**：

```java
// 密钥生成（Web3j 库）
ECKeyPair ecKeyPair = Keys.createEcKeyPair();
WalletFile walletFile = Wallet.createStandard(newPassphrase, ecKeyPair);

// TRON 地址推导（41前缀 = TRON 主网标识）
String originAddress = walletFile.getAddress();           // ETH 格式hex地址
String finalAddress = fromHexAddress("41" + originAddress); // TRON Base58Check 地址

// 私钥加密
String keyStore = Numeric.toHexStringNoPrefix(ecKeyPair.getPrivateKey());
String enKey = encrypt(keyStore, newPassphrase + finalAddress);
// encrypt 实现：AES/ECB/PKCS5Padding，密钥 = SHA-1(secret)[:16]
```

**关键观察**：
1. TRON 地址 = ETH 地址加 "41" 前缀后 Base58Check 编码
2. 密钥对使用以太坊标准库 Web3j 生成（TRON 底层密码学与 ETH 相同，都是 secp256k1）
3. passphrase 从 AWS KMS 解密获取：`awsKmsDriver.decryptData(passphrase)`
4. 创建后异步通知 user-server 更新用户的钱包地址（`@Async`）
5. 防重设计：先查 userId 是否已有地址，有则直接返回，不重复创建

# 六、核心链路 ② — TRC20 充值归集（最核心的借鉴价值）

## 6.1 整体充值模型

```
  用户充值不是"充到平台"，而是"充到自己的链上地址"
  平台通过"归集"把用户地址上的代币转到公司主地址

  ┌────────┐    链上转账     ┌──────────────┐
  │ 用户   │ ────────────→  │ 用户专属      │
  │ 外部   │   TRC20 USDT   │ TRON 地址     │
  │ 钱包   │                │ (wallet-server│
  └────────┘                │  创建的)      │
                            └───────┬───────┘
                                    │ 归集（collect）
                                    ↓
                            ┌──────────────┐
                            │ 公司主地址    │
                            │ companyAddr   │
                            │ (mainCollect) │
                            └──────────────┘
```

## 6.2 单笔归集流程（singleCollect）

这是充值的核心链路，由 `TrxDriver.singleCollect()` 编排：

```
POST /wallet/collectTrc20（或定时任务触发）
        ↓
TrxDriver.singleCollect(fromAddress, userId, coinName, ssOrderId, amount, ...)
        ↓
┌──────────────────────────────────────────────────────────────────┐
│ Step 1: 确定归集金额                                              │
│   if(reChargeAmount == null)                                      │
│       trxBalance = getAccountBalance(fromAddress, coinName)       │
│   // 若未指定金额，则查链上实际余额                                │
│                                                                    │
│ Step 2: 余额检查                                                   │
│   if(minSumAmount >= trxBalance) → 余额≤0，跳过                    │
│                                                                    │
│ Step 3: Gas 费检查（仅 TRC20 代币需要）                            │
│   if(币种 ≠ TRX) {                                                 │
│       查询该地址 TRX 余额                                          │
│       if(TRX < 30) {                                               │
│           从 gasAddress 转 30 TRX 到用户地址  ← ★ gas 补充         │
│           记录 gas 转账订单                                        │
│       }                                                            │
│   }                                                                │
│                                                                    │
│ Step 4: 执行归集转账                                                │
│   createTransactionForTrx(null, balance, fromAddr, companyAddr,   │
│                            tokenContract, ssOrderId)               │
│   // null = 私钥从 DB 获取（归集用收款密钥）                       │
│                                                                    │

└──────────────────────────────────────────────────────────────────┘
```

## 6.3 Gas 补充机制详解

TRC20 代币转账需要消耗 TRX 作为 gas 费。如果用户地址上没有足够的 TRX，转账会失败。

```
FEE_BAL = 30 TRX（阈值常量）

检查逻辑：
1. 查询用户地址的 TRX 余额
2. if TRX < 30:
   - 从 gasAddress（专用 gas 钱包）转 30 TRX 到用户地址
   - 使用 gasAddress 的私钥签名（本地签名，非加签系统）
   - 记录这笔 gas 转账到订单表
3. 然后才执行 TRC20 代币转账
```

**为什么是 30 TRX**：TRC20 转账 fee_limit 设置为 50,000,000 SUN（= 50 TRX），实际消耗远小于此，30 TRX 提供了足够的余量。如果之前归集时有剩余 TRX，本次可能不需要再补充。

## 6.4 定时批量归集

```java
// CollectTrxJob.java
@Scheduled(cron = "0 0 8 * * ?", zone="Asia/Bangkok")
public void collectUSDT(){
    TrcCollectRequest req = new TrcCollectRequest();
    req.setCoinName("USDT");     // 只归集 USDT

}
```

`TrxDriver.collectTrc20()` 流程：
```
1. 解密公司主地址
2. 查询所有用户钱包 → trxWalletService.findAll()
3. 遍历每个用户钱包:

     // 2 = 定时任务方式
```

**定时归集时间**：每日曼谷时间（GMT+7）08:00，只归集 USDT。

**缺陷观察**：
1. **无并发控制**：遍历全量用户串行归集，如果用户量大会非常慢
2. **无失败重试**：单个用户归集失败只 log，不重试
3. **无增量检测**：每次扫全量，未检查余额是否大于 0 前就尝试归集（虽然 singleCollect 内部会检查）
4. **单币种**：只归集 USDT，不归集 CXT

**提现与归集的关键区别**：

| 维度 | 归集（充值） | 提现 |
|------|-------------|------|
| 资金方向 | 用户地址 → 公司主地址 | 出款地址 → 用户提供的地址 |

# 八、TRC20 链上交易技术实现

## 8.1 TRC20 代币转账（trc20Transaction）

三步流程：`triggersmartcontract → 签名 → broadcasthex`

```
Step 1: 构建智能合约调用
  POST {trxUrl}/wallet/triggersmartcontract
  Body: {
    contract_address: hex(tokenContract),      // 合约地址（如 USDT 合约）
    function_selector: "transfer(address,uint256)",  // ERC20 标准 transfer
    parameter: encoderAbi(to, amount),         // ABI 编码参数
    owner_address: hex(from),                  // 调用者地址
    call_value: 0,                             // 不附带 TRX
    fee_limit: 50000000                        // 最大 50 TRX gas
  }

Step 2: 打包 + 签名
  transaction.raw_data.data = hex("trc20")     // 标记交易类型
  Protocol.Transaction tx = TronUtils.packTransaction(json)

  if(from == dispensingAddress):
      bytes = sendSign(signUrl, tx, ssOrderId, from)   // ★ 外部加签
  else:
      bytes = signTransaction2Byte(tx, privateKey)      // 本地签名

Step 3: 广播
  POST {trxUrl}/wallet/broadcasthex
  Body: { transaction: hex(signedBytes) }
  Result: { code: "SUCCESS", txid: "xxx..." }
```

## 8.2 TRX 原生币转账（trxOfflineSignature）

```

Step 2: 外部加签（提现始终走加签系统）
```

## 8.3 外部加签系统通信（sendSign）

```java
// 1. 计算交易哈希
byte[] txId = Sha256Sm3Hash.hash(tx.getRawData().toByteArray());
String signTxid = Base64.encode(txId);

// 2. 构建请求体
Map signMap = {
    "orderNo":      ssOrderId,
    "outAddressNo": fromAddress,
    "chainType":    "TRON",
    "hashId":       signTxid
};

// 3. RSA 加密请求体
String encrypted = RSAUtil.encrypt(JSON.toJSONString(signMap), rsaPublicKey);
// rsaPublicKey 从 TrxWalletConfig 表获取

// 4. 发送请求
POST {signUrl}/api/admin/chain/sign
Headers: {
    Content-Type: application/json,
    PLAT_CODE: platCode       // 平台码，从 Config 表获取
}
Body: encrypted               // RSA 加密后的字符串

// 5. 解析响应
Response: { code: "200", data: "Base64签名值" }
byte[] signature = Base64.decode(response.data);

// 6. 组装完整签名交易
return tx.toBuilder().addSignature(ByteString.copyFrom(signature)).build().toByteArray();
```

# 十一、wallet-third-server（第三方游戏钱包代理）

### 定位

独立服务（端口 9601），负责与第三方游戏平台（汇旺百家乐/牛牛）的钱包对接。

### 核心职责

- 用户上分/下分：调用第三方 API 将平台余额转入/转出游戏平台
- 余额同步：查询第三方平台余额，与平台余额比对
- 游戏记录：同步第三方游戏注单

### 与 wallet-server 的关系

**完全独立**。wallet-third-server 处理的是"平台余额 ↔ 第三方游戏余额"的转换，不涉及链上操作。wallet-server 处理的是"链上资产 ↔ 平台余额"的充提。两者通过 user-server 的余额间接关联。

**对我们的借鉴价值**：较低。我们的第三方游戏对接有专门的真人视讯/游戏模块负责。但其上分/下分的余额一致性保障思路可参考。

# 十四、设计模式与实践总结

## 14.1 值得借鉴的设计（6 项）

### ① 归集模型（最核心借鉴点）

**模式**：用户获得独立链上地址 → 用户充值到自己地址 → 平台将代币归集到公司主地址
**价值**：
- 每个用户有独立的充值地址，无需标注 memo/tag 区分来源
- 归集到公司主地址统一管理，方便出款和财务统计
- 充值确认由归集操作触发（或定时扫描触发），而非实时链上监听

### ② 签名权分离

**模式**：归集（低风险）本地签名 vs 提现（高风险）外部加签系统
**价值**：
- 出款私钥不在业务服务内存中，即使 wallet-server 被攻破也无法转走公司资金
- 加签系统独立部署，可有更严格的网络隔离和审计
- RSA 加密通信，防止中间人篡改签名请求

### ③ Gas 自动补充

**模式**：TRC20 转账前检查 TRX 余额 → 不足则从 gasAddress 自动补充
**价值**：归集/提现全自动化，无需人工干预 gas 费问题

### ④ 多层密钥保护

**模式**：AWS KMS（配置层）→ AES（存储层）→ RSA（通信层）→ 物理隔离（签名层）
**价值**：纵深防御，任一层被攻破都不会直接导致私钥泄露

### ⑤ 双 ID 幂等设计

**模式**：`ss_order_id`（业务订单号）+ `tx_id`（链上交易哈希）
**价值**：
- 提现前查 ssOrderId 是否已存在 → 防重复提现
- txId 用于链上确认 → 通过 `gettransactionbyid` 验证交易结果

### ⑥ 归集与交易双表记录

**模式**：每次归集同时写 collect 表（归集视角）和 order 表（交易视角）
**价值**：不同角度审计，collect 表聚焦"哪些用户的钱归集了"，order 表聚焦"链上发生了什么交易"

## 14.2 需要改进/规避的设计（7 项）

### ① 金额精度丢失风险

**问题**：`DECIMAL(16,2)` 只保留 2 位小数，但 TRX/USDT 链上精度是 6 位，CXT 是 18 位
**影响**：小额交易可能被截断，例如 0.001 USDT 链上有效但数据库记录为 0.00
**我们的做法**：使用 BIGINT 存储最小单位整数（如 USDT = amount * 10^6），彻底避免浮点精度问题

### ② 缺乏分布式锁

**问题**：归集和提现操作没有加分布式锁，Redis 虽然配置了但未在钱包操作中使用
**风险**：并发归集可能导致 gas 重复补充或重复广播同一笔交易
**我们的做法**：Redis 分布式锁保护关键操作（参考工程 A 的 Redisson 模式）

### ③ 定时归集全量扫描

**问题**：每次定时归集遍历所有用户钱包，不判断余额直接尝试（虽然内部会检查，但浪费链上 API 调用）
**改进方向**：
- 先批量查余额，过滤余额 > 0 的地址再归集
- 或引入链上 Event 监听（Webhook），充值时主动触发归集

### ④ AES/ECB 模式安全性不足

**问题**：私钥加密使用 `AES/ECB/PKCS5Padding`，ECB 模式相同明文产生相同密文，存在模式泄露风险
**改进**：应使用 `AES/GCM` 或 `AES/CBC` + IV，提供更好的语义安全性

### ⑤ 错误处理粗糙

**问题**：大量 catch 只 log 不重试，不区分可重试错误（网络超时）和不可重试错误（余额不足）
**尤其是**：归集失败不重试，链上交易如果广播成功但记录失败会导致数据不一致

### ⑥ 缺乏链上交易确认机制

**问题**：归集/提现广播后立即记录订单，但链上交易可能失败（回滚）
**现状**：只有 `getTransactionInfoById` 端点供外部查询，没有自动的交易确认轮询
**改进**：应有异步 Job 定期扫描 pending 状态的交易，确认链上最终状态

### ⑦ userId 字段过短

**问题**：`VARCHAR(10)` 限制 userId 最大 10 字符，对于大平台可能不够
**我们的做法**：BIGINT 类型用户 ID，无长度限制

## 15.2 关键设计决策对比

| 维度 | 参考工程 B | 我们的 ser-wallet | 评估 |
|------|-----------|------------------|------|
| 钱包余额 | 不维护（只查链上） | 平台内部余额账本（BIGINT） | 我们范围更大 |
| 充值模型 | 归集模式（链上→归集→确认） | 需设计 | 借鉴归集思路 |
| 提现模型 | 外部签名服务 | 需设计 | 借鉴签名分离 |
| 金额存储 | DECIMAL(16,2) | BIGINT（最小单位） | 我们更安全 |
| 密钥管理 | AWS KMS + AES | 待定 | 借鉴分层思路 |
| 并发控制 | 无 | Redis 分布式锁 | 我们更完善 |
| 交易确认 | 仅按需查询 | 需设计 | 需改进 |
| 多链支持 | 仅 TRON | 可能需多链 | 需更灵活的抽象 |

## 15.3 我们需要额外设计的部分

参考工程 B 不涉及但我们必须处理的：

1. **平台内部余额变动**：充值确认后需要增加用户平台余额（走 Ledger 层）
2. **充值检测**：参考工程依赖外部系统通知或定时扫描触发 collectTrc20，我们可能需要链上 Event 监听或定期轮询
3. **提现审核**：参考工程提现是直接执行，我们可能需要管理端审核流程
4. **多币种/多链**：参考工程只有 TRON 链 3 个币，我们可能需要抽象链操作接口支持未来多链
5. **充值/提现状态机**：参考工程没有显式的状态流转，只有最终记录。我们需要 pending→confirmed→completed 状态流转
6. **对账机制**：链上数据与平台数据的定期对账

# 十七、TRC20 充值完整链路（提炼版，供我们复用）

基于参考工程的实现，提炼出适合我们 Go 项目的 TRC20 充值完整链路：

```
┌────────────────────────────────────────────────────────────────┐
│              TRC20 充值归集完整链路（理想设计）                   │
│                                                                  │
│  Phase 1: 钱包创建                                               │
│  ────────────────                                                │
│  用户注册 → 生成 secp256k1 密钥对                                │
│          → 推导 TRON 地址（41+hex → Base58Check）                │
│          → 私钥加密存储（AES/GCM + 密钥管理服务）                │
│          → 绑定 userId ↔ address 映射                            │
│                                                                  │
│  Phase 2: 充值检测                                               │
│  ────────────────                                                │
│  方案 A（参考工程做法）：定时扫描所有用户地址余额                │
│  方案 B（推荐）：TronGrid Event API / Webhook 监听               │
│  方案 C：由上游系统通知触发归集                                  │
│                                                                  │
│  Phase 3: 归集执行                                               │
│  ────────────────                                                │
│  3.1 检查用户地址 TRC20 余额（triggerConstantContract）          │
│  3.2 if 余额 > 最小归集阈值:                                    │
│      3.2.1 检查 TRX gas 余额                                    │
│      3.2.2 if TRX < 阈值: 从 gasWallet 补充 TRX                 │
│      3.2.3 构建 TRC20 transfer 交易                              │
│      3.2.4 解密用户私钥 → 本地签名                               │
│      3.2.5 broadcasthex 广播交易                                 │
│      3.2.6 记录交易（pending 状态）                              │
│                                                                  │
│  Phase 4: 交易确认                                               │
│  ────────────────                                                │
│  4.1 异步 Job 轮询 pending 交易                                  │
│  4.2 gettransactionbyid 查询链上状态                             │
│  4.3 if SUCCESS → 更新状态为 confirmed → 增加用户平台余额        │
│  4.4 if FAILED → 标记失败 → 告警 → 人工介入                     │
│                                                                  │
│  Phase 5: 平台入账                                               │
│  ────────────────                                                │
│  5.1 调用 Ledger 层增加用户余额                                  │
│  5.2 记录账变流水（变动类型=虚拟币充值）                         │
│  5.3 通知用户充值到账                                            │
└────────────────────────────────────────────────────────────────┘
```

**与参考工程的差异**：
- 参考工程没有 Phase 4（交易确认）和 Phase 5（平台入账），因为它只负责链上操作
- 我们需要补全这两个阶段，将链上归集与平台余额变动衔接起来

# 十九、总结

## 19.1 参考工程 B 的核心价值

这个工程最大的价值在于提供了一套 **完整的、经过生产验证的 TRC20 虚拟币充提链路**：

1. **创建钱包**：密钥生成 → TRON 地址推导 → 加密存储
2. **归集充值**：余额查询 → Gas 补充 → 合约调用 → 签名 → 广播 → 记录
3. **提现**：幂等检查 → 余额校验 → Gas 补充 → 外部加签 → 广播 → 记录
4. **安全体系**：KMS + AES + RSA + 签名分离 四层纵深

## 19.2 我们的借鉴策略

**直接复用思路**：归集模型、签名分离、Gas 补充、TRC20 交互三步骤
**改造升级**：金额精度（BIGINT）、加密算法（AES/GCM）、并发控制（分布式锁）、交易确认（状态机）
**自主设计**：平台入账联动、充值检测机制、提现审核流程、多链抽象、对账机制

核心原则不变：**借鉴参考，不无脑照搬。**这个参考工程提供了链上操作的骨架和思路，我们需要在此基础上叠加平台账本层的设计，并修补其在安全性、并发控制、精度处理上的不足。
