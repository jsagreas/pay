### 1.3 与模块参考A（p9-services）对比

| 维度 | 参考A（p9-services） | 参考B（tg-services） |
|------|---------------------|---------------------|
| 定位 | 传统博彩平台 | Telegram博彩平台 |
| 钱包核心 | 内部余额系统（充扣冻结） | **链上钱包（TRON区块链）** |

**结论**：两个参考工程互补——A提供余额安全机制，B提供链上充提实现。

## 4. 数据模型深度分析

### 4.1 表结构总览

wallet-server 共 4 张表，全部围绕 TRON 链上钱包操作：

```
┌─────────────────┐     ┌─────────────────────┐
│   TrxWallet     │     │   TrxWalletOrder    │
│ (用户钱包地址)   │     │ (链上交易记录)       │
├─────────────────┤     ├─────────────────────┤
│ userId          │────→│ userId              │
│ userName        │     │ txId (链上交易哈希)   │
│ enPrivateKey    │     │ ssOrderId (业务订单)  │
│ address         │     │ amount              │
└─────────────────┘     │ fromAddress         │
                        │ toAddress           │
┌─────────────────┐     │ coinName            │
│ TrxWalletConfig │     │ orderType (1充/2提)  │
│ (钱包配置)       │     │ postType (1接口/2定时)│
├─────────────────┤     └─────────────────────┘
│ type            │
│ code            │     ┌─────────────────────┐
│ value           │     │ trx_wallet_collect  │
└─────────────────┘     │ (归集记录)           │
                        ├─────────────────────┤
                        │ userId              │
                        │ txId                │
                        │ collectAddress      │
                        │ amount              │
                        │ collectType (1单/2定)│
                        └─────────────────────┘
```

### 4.2 TrxWallet（用户热钱包表）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint (auto) | 主键 |
| userId | varchar(10) | 会员账号 |
| userName | varchar(30) | 会员姓名 |
| enPrivateKey | varchar(250) | **AES加密后的私钥** |
| address | varchar(100) | 钱包地址（T开头Base58格式） |
| createTime | datetime | 创建时间 |
| updateTime | datetime | 更新时间 |

**索引**：userId

**关键特征**：
- 每个用户一个唯一的 TRON 地址（1:1映射）
- 私钥 AES 加密存储，密码 = SHA1(passphrase + address)[0:16]
- 不存余额！余额实时查链

### 4.3 TrxWalletOrder（链上交易记录表）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint (auto) | 主键 |
| userId | varchar(10) | 会员账号 |
| txId | varchar(200) | **链上交易哈希**（唯一标识） |
| ssOrderId | varchar(200) | 业务系统订单号（关联 user-server 的 ss_order） |
| amount | decimal(16,2) | 金额 |
| fromAddress | varchar(250) | 发送地址 |
| toAddress | varchar(250) | 接收地址 |
| coinName | varchar(10) | 币种（TRX/USDT/CXT） |
| orderType | tinyint(2) | 1=充值/归集 2=提现 |
| postType | tinyint(2) | 1=接口调用 2=定时任务 |
| createTime | datetime | 创建时间 |

**索引**：user_id, order_type, tx_id

**特殊查询**：`findBySsOrderId` 使用 native SQL 并硬编码 `coin_name = 'USDT'`

### 4.4 TrxWalletConfig（钱包配置表）

| 字段 | 类型 | 说明 |
|------|------|------|
| type | varchar(30) | 类型标识（如 "sign"） |
| code | varchar(50) | 配置项标识（如 "rsa", "platCode", "disPendAddress"） |
| value | varchar(1000) | 配置值 |

**用途**：存储签名服务相关配置（RSA公钥、商户号、下发地址）

### 4.5 trx_wallet_collect（归集记录表）

| 字段 | 类型 | 说明 |

| collectAddress | varchar(250) | 归集来源地址 |
| amount | decimal(16,2) | 归集金额 |
| collectType | tinyint(2) | 1=单笔即时 2=定时任务 |

### 5.2 关键步骤代码溯源

```java
// AsyncVirtualDepositService.java
String key = USER_SERVER_DEPOSIT_TASK_SS_ORDER_ID + orderId;
RLock fairLock = redisson.getFairLock(key);
boolean res = fairLock.tryLock(0L, UNLOCK_TIME, TimeUnit.SECONDS);
```

- 锁粒度：单个订单ID

| **非实时到账** | 用户转币后需点击确认，系统主动查验余额 | ★★★ 需要理解 |
| **定时+手动双轮驱动** | 每2分钟定时扫描 + 用户手动触发 | ★★★ 核心模式 |
| **余额判断→归集** | 先查余额≥订单金额，再执行归集转账 | ★★★ 安全逻辑 |
| **Gas费自动补充** | TRC20转账需TRX作Gas，<30TRX自动补 | ★★★ 必须实现 |
| **归集=充值确认** | 将用户地址的币归集到公司地址=确认到账 | ★★★ 设计精髓 |
| **超时取消** | 30分钟无链上到账→系统自动取消订单 | ★★ 合理策略 |
| **幂等保护** | Redis key防重 + ssOrderId去重 | ★★★ 必须有 |

### 6.2 提现与充值的关键差异

| 维度 | 充值 | 提现 |
|------|------|------|
| 链上方向 | 用户地址 → 归集地址 | **下发地址 → 用户指定地址** |
| 签名方式 | 本地签名（用户私钥） | **外部签名服务**（下发地址私钥不在本服务） |
| 余额检查 | 查用户链上地址余额 | 查下发地址余额 |
| 冻结 | 无（先查后转） | **先冻结用户DB余额** |
| 确认机制 | 归集成功=到账 | 链上确认=成功 → 扣余额+扣手续费 |
| 失败处理 | 超时取消 | **解冻余额 + 取消订单** |

### 6.3 提现双重签名安全设计

```
提现签名流程：
1. wallet-server 构建交易（triggersmartcontract）
2. 计算交易哈希 txIdHash = SHA256(rawData)
3. Base64编码哈希
4. RSA加密请求体（含 orderNo, outAddressNo, chainType, hashId）
5. 发送到外部签名服务 POST /api/admin/chain/sign
6. 签名服务持有下发地址私钥，独立签名
7. 返回签名值（Base64编码）
8. wallet-server 组装签名到交易体
9. 广播到TRON网络
```

**设计思想**：下发地址的私钥不在 wallet-server 中，而是在独立的签名服务中。这是**冷热分离**的典型实践——即使 wallet-server 被攻破，也无法动用下发资金。

### 7.1 什么是归集

### 7.3 归集流程细节

```java
// TrxDriver.singleCollect() 核心逻辑
public TrxWalletResult singleCollect(fromAddress, userId, coinName, ssOrderId, reChargeAmount, ...) {
    // 1. 获取公司归集地址
    String newMainAddress = awsKmsDriver.decryptData(companyAddress);

    // 2. 获取下发地址（Gas费来源）
    String newGasAddress = list.stream()
        .filter(obj -> obj.getCode().equals("disPendAddress")).findFirst().get().getValue();

    // 3. 查询或使用传入的余额
    if (reChargeAmount == null) {
        trxBalance = getAccountBalance(fromAddress, coinName);
    }

    // 4. 余额检查
    if (minSumAmount >= trxBalance) return null;  // 余额<=0不归集

    // 5. Gas费补充（仅TRC20需要）
    if (!isTRX(tokenContract)) {
        BigDecimal trxBalance = getAccountBalance(fromAddress, "TRX");
        if (30 > trxBalance) {
            // 从Gas地址转30TRX到用户地址
            createTransactionForTrx(gasPassword, 30, gasAddress, fromAddress, "trx", ssOrderId);
        }
    }

    // 6. 执行归集转账：用户地址 → 公司地址
    result = createTransactionForTrx(null, trxBalance, fromAddress, mainAddress, tokenContract, ssOrderId);

    // 7. 记录归集表 + 订单表
    reverseNoticeCollectDetail(userId, fromAddress, txId, trxBalance, collectCode, ssOrderId, coinName, postType);
}
```

```java
// WalletTrxBusiness.trc20Transaction() 核心流程

// 1. 地址转换：T开头 → 41开头hex
String hexTo = toHexAddress(to);
String hexContract = toHexAddress(tokenContract);
String hexFrom = toHexAddress(from);

// 2. ABI编码 transfer(address,uint256) 参数
BigInteger integer = amount.multiply(10^decimals).toBigInteger();
String parameter = encoderAbi(hexTo.substring(2), integer);
// 使用 web3j FunctionEncoder.encodeConstructor([Address, Uint256])

// 3. 构建triggersmartcontract请求体
map.put("contract_address", hexContract);
map.put("function_selector", "transfer(address,uint256)");
map.put("parameter", parameter);
map.put("owner_address", hexFrom);
map.put("call_value", 0);
map.put("fee_limit", 50000000L);  // 50 TRX最大Gas

// 4. POST /wallet/triggersmartcontract 获取未签名交易
String getInfo = HttpClient4Util.doPost(trxUrl + "/wallet/triggersmartcontract", body);
JSONObject transaction = jsonObject.getJSONObject("transaction");

// 5. 附加data字段（备注）
transaction.getJSONObject("raw_data").put("data", Hex.toHexString("trc20".getBytes()));

// 6. 打包为protobuf Transaction对象
Protocol.Transaction tx = TronUtils.packTransaction(transaction.toJSONString(), false);

// 7. 签名（分两种路径）
if (fromAddress == dispensingAddress) {
    // 提现：调用外部签名服务
    bytes = sendSign(signUrl, tx, ssOrderId, from);
} else {
    // 归集：本地私钥签名
    bytes = signTransaction2Byte(tx.toByteArray(), ByteArray.fromHexString(keyStore));
}

// 8. 广播
hashMap.put("transaction", ByteArray.toHexString(bytes));
String raw = HttpClient4Util.doPost(trxUrl + "/wallet/broadcasthex", body);

// 9. 检查结果
String isSuc = result.getString("code");  // "SUCCESS" 或 null
String txId = result.getString("txid");
```

### 8.4 钱包地址生成流程

```java
// WalletTrxBusiness.createWallet()

// 1. 生成椭圆曲线密钥对
ECKeyPair ecKeyPair = Keys.createEcKeyPair();  // web3j

// 2. 创建标准钱包文件
WalletFile walletFile = Wallet.createStandard(passphrase, ecKeyPair);

// 3. 获取原始地址（hex格式）
String originAddress = walletFile.getAddress();  // 无0x前缀

// 4. 转换为TRON地址格式
String tronAddress = fromHexAddress("41" + originAddress);
// "41" + 20字节 → Base58Check编码 → "T"开头

// 5. 提取私钥（hex）
String privateKeyHex = Numeric.toHexStringNoPrefix(ecKeyPair.getPrivateKey());

// 6. AES加密私钥
String encryptedKey = encrypt(privateKeyHex, passphrase + tronAddress);
// AES/ECB/PKCS5Padding, key = SHA1(passphrase+address)[0:16]

// 7. 存储
trxWalletService.save(userId, userName, encryptedKey, tronAddress);
```

### 8.5 TRON地址格式

```
hex地址:   41 + 20字节 = 42个hex字符
           例: 41a614f803b6fd780986a42c78ec9c7f77e6ded13c

Base58Check: 对hex地址做 SHA256→SHA256→取前4字节校验→Base58编码
             结果：T + 33个字符
             例: TRrQtwH9Z4kPZeuMMhvQ5ecL7Syy8vPqbq
```

| **B3** | Gas费自动补充 | `TrxDriver.singleCollect()` 中30TRX阈值检查 | 阈值可配置化改进后采用 |
| **B4** | 地址生成流程 | `WalletTrxBusiness.createWallet()` | ECKeyPair→TRON地址的完整流程，Go版本对等实现 |
| **B5** | TRC20智能合约交互 | `Trc20.java` + `SmartContract.java` | balanceOf/transfer/decimals调用模式采用 |
| **B6** | 冷热签名分离 | `sendSign()` + 外部签名服务 | 下发地址私钥隔离的安全设计采用 |
| **B7** | 双轮驱动充值确认 | 定时扫描（2分钟）+ 用户手动确认 | 确保不遗漏的策略采用 |
| **B8** | 超时取消机制 | `VirtualCurrencyBusiness.timeoutOrder()` 30分钟超时 | 超时阈值可配置化后采用 |
| **B9** | 多地址体系 | 用户地址 + 归集地址 + 下发地址 + Gas地址 | 四种地址角色模型采用 |

### 12.2 中等价值借鉴（★★）

| 编号 | 设计点 | 参考代码 | 借鉴方式 |
|------|--------|---------|---------|
| **M1** | 私钥AES加密存储 | `WalletTrxBusiness.encrypt()` | 加密思路可用，但ECB模式需改为CBC/GCM |
| **M2** | AWS KMS配置加密 | `AwsKmsDriver` | 如果我们也用AWS可以参考，否则替换为Vault |
| **M3** | 订单ssOrderId去重 | `TrxWalletOrderRepository.findBySsOrderId()` | 去重思路可用，但应加唯一索引 |
| **M4** | 充提订单分离 | wallet-server记链上交易 + user-server记业务订单 | 职责分离思路可参考 |
| **M5** | TrxWalletConfig配置表 | 签名服务配置存DB | 灵活但缺少缓存，可用ETCD替代 |

### 12.3 不建议采用（★）

| 编号 | 设计点 | 原因 |
|------|--------|------|
| **N1** | HTTP直调代替RPC | 无类型安全、无自动发现、无负载均衡；我们用Kitex更好 |
| **N2** | FastJSON序列化 | 已知安全漏洞多；Go项目用标准json或sonic |
| **N3** | AES/ECB模式 | ECB模式不安全（相同明文产生相同密文）；应用CBC或GCM |
| **N4** | 无连接池的HTTP调用 | HttpClient4Util每次创建连接；Go用http.Client复用 |
| **N5** | Spring Security全放行 | `web.ignoring().antMatchers("**")` 等于没有安全 |
| **N6** | 密码硬编码在yml | 数据库密码、Redis密码、RabbitMQ密码明文 |
| **N7** | findBySsOrderId硬编码USDT | Native SQL中 `coin_name = 'USDT'` 写死 |

## 13. 参考工程存在的问题

### 13.1 安全问题

| 问题 | 严重度 | 代码位置 | 说明 |
|------|--------|---------|------|
| AES使用ECB模式 | 高 | `WalletTrxBusiness.encrypt()` | 相同明文→相同密文，可被分析 |
| RSA-512密钥长度 | 高 | `RSAUtil.java` | 512位RSA已不安全，应≥2048位 |
| 密码明文存储 | 高 | `application.yml` | DB/Redis/MQ密码明文 |
| Spring Security全放行 | 中 | `WebSecurityConfig.java` | `antMatchers("**")` 无防护 |
| AWS凭证明文 | 高 | `application.yml` | accessKeyId/secretAccessKey明文 |
| 私钥密码推导可预测 | 中 | `encrypt(key, passphrase + address)` | passphrase全局统一+address公开 |

### 13.2 健壮性问题

| 问题 | 代码位置 | 说明 |
|------|---------|------|
| 无HTTP超时配置 | `HttpClient4Util.doPost()` 调用处 | 链上调用可能长时间阻塞 |
| 无重试机制 | `trc20Transaction()` 等 | 广播失败无重试 |
| 无幂等保护（弱） | `TrxWalletOrder` 表 | ssOrderId无唯一索引 |
| Gas费竞态条件 | `singleCollect()` | 多个归集并发时Gas可能不足 |
| 定时归集无并发控制 | `collectTrc20()` | 遍历全部用户，单线程顺序执行 |
| 异常被吞 | 多处 `catch(Exception e)` | 打印日志但不重试不告警 |

### 13.3 设计问题

| 问题 | 说明 |
|------|------|
| 不存余额 | 每次查余额都走链上RPC，性能差且依赖链上可用性 |
| 无链上事件监听 | 只有轮询模式，没有WebSocket/事件订阅 |
| 缺少对账机制 | 无链上←→DB的自动对账 |
| config表滥用 | 签名配置存DB但无缓存，每次提现都查DB |
| 单一归集地址 | companyAddress单点，若被封冻则所有资金受影响 |

## 15. 对我们ser-wallet的具体建议

### 15.1 TRC20充提模块设计建议

基于参考B的分析，对我们 ser-wallet 的 TRC20 充提功能提出以下具体建议：

#### R1. 四地址体系（必须）

```
┌──────────────────────────────────────────────┐
│ 用户充值地址（per-user）                       │
│ - 每个用户独立生成                              │
│ - 私钥AES-GCM加密存DB                         │
│ - 用于接收用户链上转账                          │
├──────────────────────────────────────────────┤
│ 归集地址（公司地址）                             │
│ - 所有用户充值最终归集到此                       │
│ - 私钥不在wallet服务中                          │
├──────────────────────────────────────────────┤
│ 下发地址（提现地址）                             │
│ - 提现出款从此地址发起                           │
│ - 私钥通过签名服务隔离                           │
├──────────────────────────────────────────────┤
│ Gas地址（手续费地址）                            │
│ - 专门为TRC20操作提供Gas(TRX)                  │
│ - 需要监控余额，低于阈值告警                     │
└──────────────────────────────────────────────┘
```

#### R2. 充值流程建议（结合参考A+B）

```
1. 用户请求充值 → 返回用户专属TRON地址
2. 用户链上转USDT到该地址
3. 双轮检测：
   a. 定时扫描（每2分钟）所有待处理订单
   b. 用户手动确认（可选）
4. 检测到余额 ≥ 充值金额：
   a. 检查Gas余额，不足则从Gas地址补充
   b. 归集：用户地址 → 归集地址（本地私钥签名）
   c. 等待链上确认
5. 确认成功：
   a. 更新订单状态
   b. 增加用户DB余额（用参考A的CAS安全机制）
   c. 记录transaction_log（含balance_before/balance_after）
6. 超时处理：30分钟（可配置）未到账自动取消
```

#### R3. 提现流程建议

```
1. 用户请求提现 → 冻结DB余额
2. 创建提现订单（status=WAIT）
3. 检查下发地址余额 ≥ 提现金额
4. Gas检查 + 补充
5. 构建交易 → 签名服务签名 → 广播
6. 等待链上确认：
   - 成功：扣DB余额 + 扣手续费 + 解冻 + 订单完成
   - 失败/超时：解冻余额 + 取消订单
```

#### R4. 安全改进点

| 参考B做法 | 我们的改进 |
|----------|----------|
| AES/ECB | **AES/GCM**（带认证的加密） |
| RSA-512 | **RSA-2048** 或 **ECDSA** |
| 密码明文yml | **ETCD加密存储** 或 **Vault** |
| 无HTTP超时 | **设置合理超时**（链上查询10s，广播30s） |
| 无重试 | **指数退避重试**（广播失败时） |
| 无幂等索引 | **唯一索引** on (ss_order_id, coin_name) |

#### R5. 新增表结构建议

基于参考B的4张表 + 参考A的设计经验，建议我们的表设计：

```sql
-- 1. 链上钱包表（对应 TrxWallet）
-- 已在我们的表结构设计中考虑，新增以下字段：
-- chain_type: 链类型（TRON/ETH/...）扩展性
-- wallet_status: 1=正常 2=冻结 3=废弃

-- 2. 链上交易记录表（对应 TrxWalletOrder，但更完善）
-- 建议增加：
-- chain_type: 链类型
-- block_number: 区块高度
-- confirm_count: 确认数
-- gas_fee: 实际Gas消耗
-- retry_count: 重试次数
-- error_msg: 错误信息

-- 3. 归集记录表（对应 trx_wallet_collect）
-- 可以合并到交易记录表，用 order_type 区分

-- 4. 链上配置表
-- 建议用 ETCD 替代 DB 配置表
```

#### R6. 与参考A方案的整合

```
我们的ser-wallet应该同时借鉴A和B：

参考A（p9-services）提供：
├─ 余额安全机制（CAS乐观锁 + SQL WHERE检查）
├─ 冻结/解冻/回滚三步模式
├─ balance_before/balance_after 审计字段
├─ balance_type 收支方向字段
└─ 幂等唯一索引

参考B（tg-services）提供：
├─ TRC20完整充提链路
├─ 归集机制设计
├─ Gas费自动补充
├─ 冷热签名分离
├─ 多地址体系
└─ 定时扫描+手动确认双轮驱动

整合结果 = A的余额安全 + B的链上操作 → 我们的ser-wallet
```
