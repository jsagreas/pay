## 2. 架构分层与模块关系

### 2.1 四层调用链

```
用户/admin请求
    ↓
Controller层（路由 + 参数校验）
    ↓
Driver层（业务编排 + 流程控制 + 幂等检查）
    ↓
Business层（区块链原子操作：生成钱包、构建交易、签名、广播、查余额）
    ↓
Service层（DB持久化 + 私钥解密 + 异步通知）→ Repository → MySQL
```

**关键设计决策：**
- Driver与Business职责分离：Driver负责"做什么"（流程编排），Business负责"怎么做"（区块链API调用）
- 外部签名服务通过Business层的`sendSign()`方法对接，与本地签名在同一层做分支判断
- Service层同时承担DB持久化和私钥解密（解密需要密码+地址作为AES密钥）

### 2.2 跨服务交互

```
wallet-server (9600)
    ├── → TRON全节点 (https://nile.trongrid.io)     # 区块链RPC
    ├── → 外部签名服务 (http://34.92.121.9:80)      # 提现交易签名
    ├── → AWS KMS (AP_SOUTHEAST_1)                    # 敏感配置解密
    ├── → user-server (9300)                          # 用户地址回写
    └── ← admin-server                               # 管理后台调用
```

### 2.3 4类地址角色

| 地址角色 | 配置来源 | 用途 | 私钥存放 |
|---------|---------|------|---------|
| Company Address（companyAddress） | application.yml（KMS加密） | 企业结算/金库地址 | 不在此系统 |

## 3. 数据模型（4张表）

### 3.1 trx_wallet（用户钱包表）

```sql
CREATE TABLE trx_wallet (
    id          BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id     VARCHAR(10)  INDEX,       -- 用户ID
    user_name   VARCHAR(30),              -- 用户名
    en_private_key VARCHAR(250),          -- AES加密后的私钥（Base64）
    address     VARCHAR(100),             -- TRON Base58地址（T开头）
    created_time DATETIME,                -- 继承BaseEntity
    updated_time DATETIME                 -- 继承BaseEntity
);
```

**设计要点：**
- 一个用户一个钱包地址（userId唯一，createWallet先查后建）
- 私钥加密存储：AES-ECB加密，密钥 = SHA1(passphrase + address)取前16字节
- 不存储助记词或原始keystore文件，仅存加密后的hex私钥

### 3.2 trx_wallet_order（交易订单表）

```sql
CREATE TABLE trx_wallet_order (
    id          BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id     VARCHAR(10)  INDEX,       -- 用户ID
    tx_id       VARCHAR(200) INDEX,       -- 链上交易hash
    ss_order_id VARCHAR(200),             -- 业务方订单号（幂等键）
    amount      DECIMAL(16,2),            -- 交易金额
    from_address VARCHAR(250),            -- 发送方地址
    to_address   VARCHAR(250),            -- 接收方地址
    coin_name   VARCHAR(10),              -- 币种名（TRX/USDT/CXT）
    order_type  TINYINT(2)   INDEX,       -- 1=充值 2=提现
    post_type   TINYINT(2),               -- 1=API调用 2=定时任务
    created_time DATETIME,
    updated_time DATETIME
);
```

**设计要点：**
- 三索引策略：user_id、tx_id、order_type
- ssOrderId用于幂等检查（提现前先查订单是否已存在）
- DECIMAL(16,2)精度 — 仅2位小数，对于TRX(6位)和USDT(6位)可能丢失精度
- Gas费补充也会产生order记录（fromAddress=gasAddress, coinName=TRX, orderType=RECHARGE）

### 3.3 trx_wallet_collect（归集记录表）

```sql
CREATE TABLE trx_wallet_collect (
    id              BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id         VARCHAR(10)  INDEX,       -- 用户ID
    tx_id           VARCHAR(200) INDEX,       -- 链上交易hash
    collect_address VARCHAR(250),             -- 归集源地址
    amount          DECIMAL(16,2),            -- 归集金额
    collect_type    TINYINT(2)   INDEX,       -- 1=单笔及时归集 2=定时任务归集
    created_time    DATETIME,
    updated_time    DATETIME
);
```

**设计要点：**
- 归集记录同时写入collect表和order表（双表冗余记录）
- collect_type区分触发方式：API主动触发 vs 定时批量

### 3.4 trx_wallet_config（配置表）

```sql
CREATE TABLE trx_wallet_config (
    id    BIGINT AUTO_INCREMENT PRIMARY KEY,
    type  VARCHAR(30),      -- 配置分类（如 "SIGN"）
    code  VARCHAR(50),      -- 配置项编码（如 "DISPENDADDRESS"、"RSA"、"PLATCODE"）
    value VARCHAR(1000),    -- 配置值
    created_time DATETIME,
```

**设计要点：**
- 通用KV配置模式，按type分组
- SIGN类型下存储：出款地址(DISPENDADDRESS)、RSA公钥(RSA)、平台编码(PLATCODE)
- 动态配置优于yml硬编码 — 可不重启更换出款地址

### 5.1 完整流程图

```
    ↓
[Step 5] 加密私钥

    ↓

    ↓

    trxWalletService.updateUserAddress(request);  // @Async → POST to user-server
    ↓
返回 WalletAddressVO { userId, userName, address }
```

### 5.2 关键技术细节

**5.2.1 地址生成原理**

```
ECDSA secp256k1 → 私钥(256bit) + 公钥
    ↓
Keccak-256(公钥) → 取后20字节 = 以太坊地址(hex)
    ↓
加前缀 "41" + 以太坊地址(hex) = TRON原始地址(hex)
    ↓
Base58Check编码 = TRON可读地址("T"开头，如 TLvmjV7uj8V2TZAQLrEDhduUzzwfoPCxy8)
```

- TRON与以太坊共享相同的椭圆曲线(secp256k1)，地址区别仅在于前缀（以太坊=0x, TRON=41）和编码格式（以太坊=hex, TRON=Base58Check）
- 同一个私钥在两条链上生成不同格式但数学等价的地址

**5.2.2 私钥加密方案**

```
明文私钥(hex string)
    ↓ AES-ECB-PKCS5Padding
加密密钥 = SHA1(passphrase + TRON地址)[:16字节]
    ↓
Base64编码后的密文 → 存入DB的en_private_key字段
```

- passphrase从application.yml读取（KMS加密存储），运行时KMS解密
- 每个用户的加密密钥不同（因为address不同），即使passphrase泄露也不能批量解密
- AES-ECB模式安全性较弱（同样的明文块产生同样的密文块），但私钥长度通常为32字节=2个AES块，实际影响有限

**5.2.3 幂等保护**

```java
String address = trxWalletService.findAddressByUserId(userId);
if (StringUtils.isNotBlank(address)) {
    // 已存在，直接返回
    walletAddressVO.setAddress(address);
    return walletAddressVO;
}
```

- 先查后建模式 — 但缺少并发保护（无分布式锁，理论上可能重复创建）
- 由于用户注册是低频操作，实际风险较低

### 5.3 Go项目复用价值

| 步骤 | 复用策略 | 说明 |
|------|---------|------|
| EC密钥对生成 | ✅ 直接复用思路 | Go用`crypto/ecdsa` + secp256k1曲线 |
| TRON地址格式 | ✅ 直接复用 | "41" + keccak256(pubkey)[-20:] → Base58Check |
| 私钥加密存储 | ⚠️ 改造后复用 | 改用AES-GCM替代ECB，增加随机IV |
| 先查后建幂等 | ⚠️ 增强 | 加分布式锁 或 DB唯一约束(user_id) |
| 异步通知用户服务 | ✅ 复用思路 | Go用goroutine或MQ异步回写 |

## 6. TRC20转账流程（核心重点）

> 源码位置：`WalletTrxBusiness.java:209-290`
> 这是本工程最有价值的实现，需要我方借鉴复用

### 6.1 完整流程图

```
调用方: TrxDriver.createTransactionForTrx(..., tokenContract, ssOrderId)
    ↓
WalletTrxBusiness.trc20Transaction(keyStore, from, to, tokenContract, amount, ssOrderId)
    ↓
[Step 1] 地址校验
    validateAddress(to)  →  POST TRON_URL/wallet/validateaddress
    ← 要求返回 "Base58check format" + result=true
    ↓
[Step 2] 合约校验
    tokenContract必须以"T"开头（TRC20合约地址格式）
    ↓
[Step 3] 构建智能合约调用参数
    contract_address:    toHexAddress(tokenContract)  // T→41格式
    function_selector:   "transfer(address,uint256)"   // ERC20标准方法
    parameter:           encoderAbi(toHex(to), amount × 10^decimal)
    owner_address:       toHexAddress(from)
    call_value:          0                             // 不随调用转TRX
    fee_limit:           50000000                      // 最大50 TRX手续费
    ↓
[Step 4] 调用TRON全节点API构建交易
    POST TRON_URL/wallet/triggersmartcontract
    ← 返回 { transaction: { raw_data: {...}, txID: "..." } }
    ↓
[Step 5] 注入自定义数据
    transaction.raw_data.data = Hex("trc20")  // 备注标识
    ↓
[Step 6] 打包交易（序列化为Protocol.Transaction）
    Protocol.Transaction tx = TronUtils.packTransaction(json, false);
    ↓
[Step 7] 签名（关键分支）
    ├── from == DispensingAddress (出款地址)
    │   → 调用外部签名服务: sendSign(signUrl, tx, ssOrderId, from)
    │   // 提现走外部签名，私钥不在本系统
    │
    └── from != DispensingAddress (归集/其他地址)
        → 本地签名: signTransaction2Byte(tx.toByteArray(), privateKey)
        // 归集走本地签名，私钥从DB解密获取
    ↓
[Step 8] 广播交易
    POST TRON_URL/wallet/broadcasthex  { transaction: signedHex }
    ← 返回 { code: "SUCCESS", txid: "..." }
    ↓
[Step 9] 结果判断
    ├── code == null 或 "SUCCESS" → 成功，返回txId
    └── 其他 → 失败，记录错误日志，返回null
```

### 6.2 关键代码解析

**6.2.1 ABI编码（ERC20/TRC20通用）**

```java
// WalletTrxBusiness.java:478-483
public String encoderAbi(String Address, BigInteger amount) {
    ArrayList<Type> inputParameter = new ArrayList<>();
    inputParameter.add(new org.web3j.abi.datatypes.Address(Address));  // 收款地址（去"41"前缀）
    inputParameter.add(new Uint256(amount));                           // 金额（最小单位）
    return FunctionEncoder.encodeConstructor(inputParameter);
    // 输出: 64字节hex = 32字节地址(左补0) + 32字节金额(左补0)
}
```

- 使用Web3j的`FunctionEncoder`做ABI编码 — 这是标准做法，Go中可用go-ethereum的abi包等价实现
- 注意：`function_selector`由TRON节点处理，`parameter`只需编码参数部分

```java
    // 归集：from是用户地址 → 私钥在本地DB → 本地签名
    bytes = signTransaction2Byte(tx.toByteArray(), ByteArray.fromHexString(keyStore));
}
```

这是**安全设计的核心**：出款私钥与归集私钥分离存储、分离签名。

**6.2.5 本地签名实现**

```java
// WalletTrxBusiness.java:151-159
private static byte[] signTransaction2Byte(byte[] transaction, byte[] privateKey) {
    ECKey ecKey = ECKey.fromPrivate(privateKey);            // 从私钥恢复ECKey
    Protocol.Transaction tx = Protocol.Transaction.parseFrom(transaction);
    byte[] rawdata = tx.getRawData().toByteArray();         // 获取raw_data
    byte[] hash = Sha256Sm3Hash.hash(rawdata);              // SHA256哈希
    byte[] sign = ecKey.sign(hash).toByteArray();           // ECDSA签名
    return tx.toBuilder()
        .addSignature(ByteString.copyFrom(sign))            // 附加签名到交易
        .build().toByteArray();
}
```

Go等价实现：`crypto/ecdsa.Sign(hash, privateKey)` + protobuf序列化

## 7. Gas费管理机制

> 源码位置：`TrxDriver.java:191-206`（归集场景）、`TrxDriver.java:308-324`（提现场景）

### 7.1 设计原理

TRC20代币转账需要消耗TRX作为Energy/带宽费用。如果发送方地址TRX余额不足，交易会失败。因此系统需要一个"加油站"机制：

```
[Gas Address] ─── 30 TRX ───→ [用户地址 or 出款地址]
                                    ↓ 有了TRX作为手续费
                              [用户地址] ─── USDT ───→ [归集地址]
                              或
                              [出款地址] ─── USDT ───→ [用户提现地址]
```

### 7.2 核心逻辑

```java
// 常量定义
private static final BigDecimal FEE_BAL = new BigDecimal("30"); // 30 TRX

// 检查逻辑（归集和提现完全相同的代码段）
if (!TRXContractEnum.TRX.getContract().equalsIgnoreCase(tokenContract)) {
    // 只有TRC20代币需要检查（TRX自身不需要额外Gas）
    BigDecimal addTrx = getAccountBalance(fromAddress, TRXContractEnum.TRX.getContract());
    if (FEE_BAL.compareTo(addTrx) > 0) {
        // TRX余额 < 30 → 从Gas地址补充30 TRX
        trxWalletResult = createTransactionForTrx(
            awsKmsDriver.decryptData(gasPassword),  // Gas地址私钥
            FEE_BAL,                                  // 固定30 TRX
            gasAddress,                               // from = Gas地址
            fromAddress,                              // to = 需要Gas的地址
            TRXContractEnum.TRX.getContract(),        // 转TRX
            ssOrderId
        );
        if (StringUtils.isBlank(trxWalletResult.getTxId())) {
            return null;  // Gas补充失败，整个流程终止
        }
        // Gas补充也记录到order表
        trxWalletOrderService.save(userId, gasAddress, fromAddress, ...);
    }
}
```

### 7.3 设计评价

**优点：**
- 自动化Gas管理，用户无需持有TRX
- Gas补充失败会终止后续操作，避免无Gas的TRC20转账失败
- Gas操作也有order记录，可追溯

**缺陷：**
- 固定30 TRX阈值 — TRON网络Gas价格波动时可能不够或浪费
- Gas补充和代币转账是两笔独立交易 — Gas到账需要确认，但代码中没有等待确认就立即执行代币转账
- 没有Gas地址余额监控/告警 — Gas地址余额耗尽会导致所有TRC20操作停摆

### 7.4 Go项目改进建议

| 问题 | 改进方案 |
|------|---------|
| 固定阈值 | Gas阈值放到配置表，可动态调整 |
| 无确认等待 | Gas补充后等待1-2个区块确认（约3-6秒），再执行代币转账 |
| 无余额监控 | Gas地址余额低于阈值时触发告警（钉钉/Telegram Bot） |
| 无批量优化 | 批量归集时先统计需要Gas的地址，一次性从Gas地址批量转出 |

### 8.3 归集流程中的数据流

```

    ↓ 定时归集

    ↓ 记录
trx_wallet_collect表: { userId, txId, collectAddress, amount, collectType=2 }
trx_wallet_order表:   { userId, txId, ssOrderId, amount, fromAddress, toAddress, orderType=1, postType=2 }

```

### 8.4 设计评价

**优点：**
- 每日定时归集保证资金不长期滞留用户热钱包
- 归集失败不影响其他用户（单个try-catch隔离）
- 双表记录（collect表+order表）便于对账

**缺陷：**
- `findAll()`全量查询所有钱包 — 钱包数量大时内存压力大，应分页或流式处理
- 没有最小归集金额阈值 — 金额极小时Gas费可能超过归集金额
- 没有归集状态追踪 — 归集交易广播后无确认流程，不知道是否真正到账
- 归集是顺序执行 — 用户量大时耗时过长，应考虑并发

### 10.2 外部签名调用流程

```java
private byte[] sendSign(String signUrl, Protocol.Transaction tx, String ssOrderId, String from) {
    // 1. 计算交易hash
    byte[] txId = Sha256Sm3Hash.hash(tx.getRawData().toByteArray());
    String signTxid = Base64.getEncoder().encodeToString(txId);

    // 2. 构建请求体
    Map<String, Object> signMap = new HashMap<>();
    signMap.put("orderNo", ssOrderId);        // 订单号
    signMap.put("outAddressNo", from);        // 出款地址
    signMap.put("chainType", "TRON");         // 链类型
    signMap.put("hashId", signTxid);          // 交易hash(Base64)

    // 3. RSA加密请求体（安全传输）
    String enSign = RSAUtil.encrypt(
        JSONObject.toJSONString(signMap),
        rsaPublicKey  // 从trx_wallet_config表获取
    );

    // 4. 发起HTTP请求
    HttpResponse response = HttpRequest.post(signUrl + "/api/admin/chain/sign")
        .header("PLAT_CODE", platCode)  // 平台标识
        .body(enSign)                    // RSA加密后的body
        .timeout(60000)                  // 60秒超时
        .execute();

    // 5. 解析签名结果
    JSONObject jsonObject = JSONObject.parseObject(response.body());
    byte[] deSign = null;
    if ("200".equals(jsonObject.get("code"))) {
        deSign = Base64.getDecoder().decode(jsonObject.getString("data"));
    }

    // 6. 组装完整签名交易
    return deSign == null ? null :
        tx.toBuilder().addSignature(ByteString.copyFrom(deSign)).build().toByteArray();
}
```

### 10.3 安全设计分析

| 层面 | 机制 | 评价 |
|------|------|------|
| 传输层 | RSA加密请求体 | ✅ 即使HTTP明文传输也不暴露交易内容 |
| 身份认证 | PLAT_CODE头部标识 | ⚠️ 简单标识，非强认证 |
| 私钥隔离 | 出款私钥仅在签名服务内 | ✅ wallet-server完全不接触出款私钥 |
| 审计 | 通过orderNo关联业务订单 | ✅ 签名服务可记录每笔签名请求 |
| 超时 | 60秒超时 | ✅ 合理（考虑网络延迟和签名服务负载） |

### 10.4 Go项目复用建议

- ✅ **复用外部签名分离架构** — 出款私钥不在主业务系统，通过RPC调用签名服务
- ⚠️ **改进传输安全** — 使用mTLS替代RSA加密body，或改用gRPC+TLS
- ⚠️ **改进身份认证** — PLAT_CODE改为JWT或HMAC签名
- ✅ **复用请求结构** — chainType + hashId + orderNo的签名请求模型通用性好

```java
// 加密密钥生成
private static byte[] getKey(String secret) {
    byte[] bytes = secret.getBytes("UTF-8");
    MessageDigest sha = MessageDigest.getInstance("SHA-1");
    byte[] digest = sha.digest(bytes);       // SHA-1 = 20字节
    return Arrays.copyOf(digest, 16);         // 截取前16字节 = AES-128
}

// 加密
public static String encrypt(String plaintext, String secret) {
    SecretKeySpec keySpec = new SecretKeySpec(getKey(secret), "AES");
    Cipher cipher = Cipher.getInstance("AES/ECB/PKCS5Padding");
    cipher.init(Cipher.ENCRYPT_MODE, keySpec);
    return Base64.encode(cipher.doFinal(plaintext.getBytes("UTF-8")));
}
```

## 17. 设计亮点

### 亮点1：签名权限分离

**提现签名 ≠ 归集签名**

| | 提现 | 归集 |
|---|------|------|
| 签名方式 | 外部签名服务 | 本地签名 |
| 私钥位置 | 签名服务内部 | 本系统DB（AES加密） |
| 安全等级 | 高（私钥不离开签名服务） | 中（私钥在内存中解密） |
| 适用场景 | 大额出款 | 用户热钱包归集 |

**核心思想：出款私钥的泄露风险远大于用户热钱包私钥**（因为出款地址持有大量资金），所以出款签名必须隔离到独立服务。这是**最值得复用的架构决策**。

### 亮点2：Gas费自动管理

系统自动检测并补充TRC20交易所需的TRX Gas费，用户无需关心底层技术细节。

### 亮点3：动态出款地址

出款地址从DB配置表（而非yml）读取，支持运行时切换，无需重启服务。在出款地址被标记/风控时可快速替换。

### 亮点4：KMS双层防护

yml配置级别用KMS加密（保护passphrase等系统密钥），DB级别用AES加密（保护用户私钥）。即使DB被脱裤，没有passphrase也无法解密私钥。

### 亮点5：归集双表记录

归集操作同时写入collect表（专用）和order表（通用），兼顾了归集专项查询和整体交易流水查询的需求。

### 亮点6：提现幂等保护

通过ssOrderId先查后写，防止重复提现。虽然实现不够严谨（缺DB唯一约束），但体现了幂等意识。

## 18. 设计缺陷与风险点

### 缺陷等级定义
- 🔴 致命：可直接导致资金损失
- 🟠 严重：可导致业务异常或数据不一致
- 🟡 一般：影响性能或可维护性

---

### 🔴 缺陷1：提现并发无锁保护

```java
// TrxDriver.withdrawTraction() — 无任何锁机制
BigDecimal balance = getAccountBalance(fromAddress, coinName);  // 查余额
if (NumberUtil.isGreater(sendAmount, balance)) return null;     // 校验
// --- 并发窗口：多个提现请求同时通过校验 ---
result = createTransactionForTrx(...);                          // 执行转账
```

**风险：** 多个提现请求并发执行，全部通过余额校验后都广播交易，导致出款超额。

**影响：** 出款地址USDT被掏空。

**修复建议：** 出款操作加分布式锁（按出款地址维度），或使用提现队列串行处理。

### 🔴 缺陷2：Gas补充无确认等待

```java
// Gas补充后立即执行TRC20转账
trxWalletResult = createTransactionForTrx(..., FEE_BAL, gasAddress, fromAddress, TRX, ...);
// 没有等待Gas到账确认
trxWalletResult = createTransactionForTrx(null, trxBalance, fromAddress, mainAddress, USDT, ...);
```

**风险：** Gas TRX转账广播成功≠上链确认。如果Gas交易未被确认就执行TRC20转账，TRC20转账会因Gas不足失败。

**影响：** 归集失败，Gas费白付。

**修复建议：** Gas补充后轮询`getTransactionInfoById`确认到账，或预留充足等待时间。

### 🟠 缺陷3：幂等保护不完整

```java
// 提现有幂等（先查ssOrderId）
TrxWalletOrderEntity order = trxWalletOrderRepository.findBySsOrderId(ssOrderId);

// 归集无幂等 — 同一地址可能被重复归集
// 钱包创建无幂等保护（仅先查后建，无并发锁）
```

**风险：** 归集定时任务执行期间如果手动触发归集，可能重复归集同一地址。钱包创建并发时可能为同一用户创建两个钱包（DB无唯一约束保护）。

### 🟠 缺陷4：DECIMAL(16,2)精度不足

```sql
amount DECIMAL(16,2) -- 仅2位小数
```

- TRX最小单位是SUN（10^-6），USDT也是6位小数，CXT是18位小数
- DECIMAL(16,2)存储链上精确金额时会丢失精度

**影响：** 小额交易金额可能被截断为0.00，账目不平。

### 🟠 缺陷5：全量查询无分页

```java
List<TrxWalletEntity> trxWalletEntityList = trxWalletService.findAll();
for (TrxWalletEntity entity : trxWalletEntityList) {
    // 逐个归集...
}
```

**风险：** 用户量增长后OOM或归集耗时过长。

### 🟡 缺陷6：AES-ECB模式安全性弱

ECB模式相同明文块产生相同密文块，存在模式泄露风险。应使用AES-GCM或AES-CBC。

### 🟡 缺陷7：AWS KMS客户端不复用

```java
public String encryptData(String noEncryptData) {
    KmsClient kmsClient = KmsClient.builder()...build();  // 每次新建
    try { ... }
    finally { kmsClient.close(); }
}
```

每次加解密都创建新的KMS客户端，浪费TCP连接和认证开销。

### 🟡 缺陷8：错误处理不统一

部分方法返回null表示失败（getAccountBalance、createTransactionForTrx），部分返回空对象（TrxWalletResult()），调用方需要分别判断null和空txId。

### 🟡 缺陷9：createTransactionForTrx的privateKey参数含义混乱

```java
// 提现时调用
result = createTransactionForTrx(fromAddress, sendAmount, fromAddress, toAddress, ...);
// 第一个参数本应是privateKey，这里传的是fromAddress（地址字符串）
// 因为提现走外部签名不需要本地私钥，但参数语义混乱
```
