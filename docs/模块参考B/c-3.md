## 四、数据模型完整分析

### 4.1 wallet-server 数据库（cloud_wallet）— 4张表

#### 表1：TrxWallet — 用户链上钱包地址

```
TrxWallet（核心：一用户一地址）
├── id              BIGINT AUTO_INCREMENT PK
├── userId          VARCHAR(10) INDEXED     ← 用户ID（唯一索引应有但代码未强制）
├── userName        VARCHAR(30)             ← 用户名
├── address         VARCHAR(100)            ← TRON地址（T开头Base58格式）
├── enPrivateKey    VARCHAR(250)            ← AES加密的私钥
├── createTime      TIMESTAMP               ← 创建时间
├── updateTime      TIMESTAMP               ← 更新时间
├── createBy        VARCHAR                 ← 创建者
└── updateBy        VARCHAR                 ← 更新者
```

**设计要点：**
- 每个用户生成一个独立的TRON地址
- 私钥用 `AES(SHA1(passphrase + address)[0:16])` 加密后存储
- 地址格式：`T` 开头的 Base58Check 编码

#### 表2：TrxWalletOrder — 链上交易记录

```
TrxWalletOrder（所有链上交易的记录）
├── id              BIGINT PK
├── userId          VARCHAR(10) INDEXED     ← 用户ID
├── txId            VARCHAR(200) INDEXED    ← ★TRON交易哈希（链上唯一标识）
├── ssOrderId       VARCHAR(200)            ← 业务系统订单号
├── amount          DECIMAL(16,2)           ← 交易金额
├── fromAddress     VARCHAR(250)            ← 发送方地址
├── toAddress       VARCHAR(250)            ← 接收方地址
├── coinName        VARCHAR(10) INDEXED     ← 币种名称（TRX/USDT/CXT）
├── orderType       TINYINT(2) INDEXED      ← 1=充值, 2=提现
├── postType        TINYINT(2)              ← 1=API触发, 2=定时任务
└── createTime      TIMESTAMP               ← 创建时间
```

**设计要点：**
- txId 是链上交易的唯一标识，用于后续确认交易是否成功
- orderType 区分充值和提现
- postType 区分手动触发和自动任务触发

#### 表3：trx_wallet_collect — 余额归集记录

```
trx_wallet_collect（用户地址→主地址的归集记录）
├── id              BIGINT PK
├── userId          VARCHAR(10) INDEXED     ← 用户ID
├── txId            VARCHAR(200) INDEXED    ← 归集交易哈希
├── collectAddress  VARCHAR(250)            ← 归集来源地址
├── amount          DECIMAL(16,2)           ← 归集金额
├── collectType     TINYINT(2) INDEXED      ← 1=即时归集, 2=定时归集
└── createTime      TIMESTAMP               ← 创建时间
```

#### 表4：TrxWalletConfig — 钱包配置（加密存储）

```
TrxWalletConfig（系统级配置，KMS加密）
├── id              BIGINT PK
├── type            VARCHAR(30) INDEXED     ← 配置类型（如"sign"）
├── code            VARCHAR(50) INDEXED     ← 配置键（如"RSA","PLATCODE","disPendAddress"）
└── value           VARCHAR(1000)           ← 配置值（AWS KMS加密后的JSON）
```

### 4.2 user-server 数据库（cloud_user）— 钱包相关表

#### 表5：Assets — 用户资产（内部余额）

```
Assets（用户的内部账户余额）
├── id              BIGINT PK
├── userId          BIGINT UNIQUE           ← 用户ID（唯一约束）
├── balance         DECIMAL(16,2)           ← 主余额（充值后增加，投注/提现减少）
├── balanceNn       DECIMAL(16,2)           ← 牛牛游戏余额
├── balanceBacc     DECIMAL(16,2)           ← 百家乐游戏余额
├── freezeAmount    DECIMAL(16,2)           ← 冻结金额（提现中冻结）
├── playMoney       DECIMAL(16,2)           ← 打码量（流水要求）
├── creditAmount    DECIMAL(16,2)           ← 信用额度
├── version         INT                     ← ★乐观锁版本号
├── userType        INT                     ← 用户类型
└── agentId1~5      BIGINT                  ← 5级代理ID
```

**核心设计：**
- **乐观锁**（version字段）防并发冲突
- **打码量**（playMoney）= 充值金额 × 流水倍数，必须完成才能提现
- **冻结金额**：提现时先冻结，链上确认后才扣减

#### 表6：SsOrder — 充值/提现订单

```
SsOrder（核心订单表，充值和提现共用）
├── id              BIGINT PK
├── orderId         VARCHAR                 ← 订单ID
├── orderNum        VARCHAR INDEXED         ← 订单编号（U + 雪花ID）
├── userId          BIGINT INDEXED          ← 用户ID
├── tgUserId        VARCHAR INDEXED         ← Telegram用户ID
├── nickName        VARCHAR INDEXED         ← 昵称
├── amount          DECIMAL(16,2)           ← 订单金额
├── realAmount      DECIMAL(16,2)           ← 实际到账金额
├── feeAmount       DECIMAL(16,2)           ← 手续费（提现时扣除）
├── orderType       INT INDEXED             ← ★ 1=充值, 2=提现
├── orderStatus     INT INDEXED             ← ★ 状态枚举（见下）
├── virtualCurrency VARCHAR                 ← 虚拟币类型（USDT/TRX/CXT）
├── payeeAddress    VARCHAR                 ← 收款地址（提现时的目标地址）
├── txId            VARCHAR                 ← ★链上交易哈希（关键关联字段）
├── flowMultiple    DECIMAL                 ← 流水倍数
├── isLock          INT INDEXED             ← 锁定标记（0/1）
├── completeTime    DATETIME                ← 完成时间
├── auditUser       VARCHAR                 ← 审核人
├── auditTime       DATETIME                ← 审核时间
├── agentId         BIGINT                  ← 代理ID
├── agentLevel      INT                     ← 代理层级
└── 审计字段...
```

**订单状态枚举：**

```
  值  |  状态                    |  含义
 -----+--------------------------+-------------------
   1  | ORDER_WAIT               | 等待处理
   2  | ORDER_SUCCESS            | ★交易成功（终态）
   3  | ORDER_SYSTEM_CANCLE      | 系统取消（超时/失败）
   4  | ORDER_USER_CANCLE        | 用户取消
   8  | VIRTUAL_CURRENCY_WAIT    | ★虚拟币等待确认（初始态）
```

#### 表7：BalanceChange — 余额变动流水

```
BalanceChange（每笔余额变动的审计日志）
├── id              BIGINT PK
├── userId          BIGINT INDEXED          ← 用户ID
├── relateId        BIGINT                  ← 关联ID（订单ID）
├── relateOrderNum  VARCHAR INDEXED         ← 关联订单号
├── changeType      INT INDEXED             ← ★变动类型（见下枚举）
├── balanceType     INT INDEXED             ← 1=收入, 2=支出
├── beforeAmount    DECIMAL                 ← 变动前余额
├── amount          DECIMAL                 ← 变动金额
├── afterAmount     DECIMAL                 ← 变动后余额
├── businessType    INT INDEXED             ← 业务类型
├── nickName        VARCHAR INDEXED
├── tgUserId        VARCHAR INDEXED
├── agentId         BIGINT INDEXED
└── agentAccount    VARCHAR INDEXED
唯一约束: (changeType, relateId, userId, orderType)
```

## 七、钱包地址生成机制

### 7.1 地址生成流程

```java
// WalletTrxBusiness.createWallet()

// 1. 生成ECDSA密钥对
ECKeyPair ecKeyPair = Keys.createEcKeyPair();
// 使用 web3j 的 secp256k1 椭圆曲线

// 2. 创建钱包文件（获取标准格式）
String passphrase = AwsKmsDriver.decryptData(mainCollectPassphrase);
// passphrase 是 KMS 加密后的口令
WalletFile walletFile = Wallet.createStandard(passphrase, ecKeyPair);

// 3. 获取原始地址（以太坊格式 0x...）
String rawAddress = walletFile.getAddress();

// 4. 转换为TRON地址
// 以太坊地址前加 41（TRON的地址前缀）
String hexAddress = "41" + rawAddress;
// Base58Check编码 → 得到 T 开头的TRON地址
String tronAddress = Base58.encode58Check(ByteArray.fromHexString(hexAddress));
// 例如: TUyKqdNXNEzAvjiBWQJ3ooQFJoSYUAqLrR

// 5. 加密私钥
String privateKeyHex = ecKeyPair.getPrivateKey().toString(16);
String aesSecret = SHA1(passphrase + tronAddress).substring(0, 16);
String encryptedKey = AES.encrypt(privateKeyHex, aesSecret);

// 6. 持久化
TrxWalletEntity entity = new TrxWalletEntity();
entity.setUserId(userId);
entity.setUserName(userName);
entity.setAddress(tronAddress);
entity.setEnPrivateKey(encryptedKey);
TrxWalletService.save(entity);

// 7. 异步回调用户服务
TrxWalletService.updateUserAddress(userId, tronAddress);
// → HTTP POST user-server:/user/updateWalletAddressById
```

### 7.2 地址格式转换

```
以太坊地址:  0x 7f4d...a3b2  （20字节）
     ↓ 加前缀 41
TRON Hex:    41 7f4d...a3b2  （21字节）
     ↓ SHA256 两次取前4字节校验
TRON Base58:  T Uyq...SUAQLR （34字符，T开头）

关键函数映射：
  toHexAddress("TUyq...") → "417f4d..."     用于链上API调用
  toViewAddress("417f...") → "TUyq..."      用于用户展示
  encode58Check(bytes) → "TUyq..."          编码
  decode58Check("TUyq...") → bytes          解码验证
```

## 八、余额归集机制

### 8.1 归集的业务含义

```
                 ★为什么需要归集？

用户充值时，USDT转到用户的独立TRON地址上
  → 但资金分散在N个用户地址上
  → 平台需要集中管理资金（提现出金、运营等）
  → 归集 = 把各用户地址的USDT汇总到主地址
```

### 8.2 两种归集模式

```
模式1：即时归集（SINGLE，充值时触发）
  触发时机：用户确认充值 → collectTrc20(postType=1)
  特点：充值到账即归集，资金快速集中

模式2：定时归集（COLLECT，定时任务触发）
  触发时机：每日8:00 AM(曼谷时间) → CollectTrxJob.collectUSDT()
  特点：批量处理，遍历所有用户钱包

共同流程：
  1. 查用户TRC20余额
  2. 检查Gas费（需要≥30 TRX）
  3. 不足则从Gas钱包补充TRX
  4. 执行TRC20转账：用户地址→主地址
  5. 记录归集记录和订单
```

### 8.3 Gas费管理

```
TRC20转账需要TRX作为Gas费（类似以太坊的ETH Gas）

Gas地址（gasAddress）：专门提供Gas费的地址
  - 地址和私钥通过AWS KMS加密存储
  - 需要人工定期给Gas地址充TRX

Gas费阈值：30 TRX
  - 每次归集前检查用户地址的TRX余额
  - 不足30 TRX → 从Gas地址转30 TRX过来
  - 这笔转账也记录为一个FEE类型的TrxWalletOrder

手续费限制：
  - TRC20转账 fee_limit = 50 TRX (50,000,000 SUN)
  - Trc20.java 中另一个限制 = 10 TRX (10,000,000 SUN)
```

## 九、密钥安全与加密体系

### 9.1 三层加密架构

```
第一层：AWS KMS（传输+存储）
  ┌──────────────────────────────┐
  │  存储：                      │
  │  - 主归集地址                │
  │  - 主归集口令（passphrase）  │
  │  - Gas地址 + 私钥            │
  │  - 出金地址 + 私钥            │
  │                              │
  │  流程：                      │
  │  配置值 → KMS加密 → 存yml    │
  │  启动时 → 读yml → KMS解密    │
  └──────────────────────────────┘

第二层：AES加密（用户私钥存储）
  ┌──────────────────────────────┐
  │  加密：                      │
  │  secret = SHA1(passphrase    │
  │           + address)[0:16]   │
  │  encrypted = AES-ECB(        │
  │    privateKey, secret)       │
  │                              │
  │  存储到 TrxWallet.enPrivateKey│
  │                              │
  │  解密：                      │
  │  secret = SHA1(passphrase    │
  │           + address)[0:16]   │
  │  privateKey = AES-ECB-Dec(   │
  │    encrypted, secret)        │
  └──────────────────────────────┘

第三层：RSA加密（外部签名服务通信）
  ┌──────────────────────────────┐
  │  场景：提现交易签名           │
  │  流程：                      │
  │  1. 构造签名请求JSON          │
  │  2. RSA公钥加密请求           │
  │  3. POST → 签名服务           │
  │  4. 返回签名字节              │
  │  5. 附加到交易并广播          │
  └──────────────────────────────┘
```

### 9.2 安全评估

| 维度 | 做法 | 评价 |
|------|------|------|
| 私钥存储 | AES加密后存DB | 中等安全，生产应考虑HSM |
| AES模式 | ECB模式 | **不安全** — 应改用GCM/CBC |
| 密钥派生 | SHA1截取16字节 | 较弱 — 应用PBKDF2/Argon2 |
| 配置加密 | AWS KMS | 良好实践 |
| 提现签名 | 外部签名服务 | 良好实践（隔离出金私钥） |
| 地址验证 | TRON API validateaddress | 正确做法 |

### 10.4 交易签名流程

```
两种签名路径：

路径A：本地签名（充值归集、Gas费转账）
  1. 从DB读取 enPrivateKey
  2. AES解密得到原始私钥
  3. ECKey.fromPrivate(privateKeyBytes)
  4. ecKey.sign(SHA256(rawData))
  5. 将签名附加到交易protobuf
  6. 广播

路径B：外部签名（提现出金）
  1. 构造签名请求：{ orderNo, outAddressNo, chainType, hashId }
  2. RSA公钥加密请求JSON
  3. POST http://签名服务:80/api/admin/chain/sign
  4. 接收返回的签名字节
  5. 将签名附加到交易protobuf
  6. 广播
```

```java
@Scheduled(cron = "0 */2 * * * ?")
public void checkVirtualCurrencyTask() {
    // 每2分钟执行一次
    VirtualCurrencyBusiness.checkChargeCrypotcurrencyOrder("USDT", 30);
}

// VirtualCurrencyBusiness.checkChargeCrypotcurrencyOrder()
public void checkChargeCrypotcurrencyOrder(String currency, int timeoutMinutes) {
    Date now = new Date();
    Date timeout = now - timeoutMinutes分钟;

    // 1. 超时订单 → 自动取消
    List<SsOrder> timeoutOrders = ssOrderService.findTimeoutOrders(
        currency, CHARGE_ORDER, VIRTUAL_CURRENCY_WAIT, timeout前的订单
    );
    for (SsOrder order : timeoutOrders) {
        order.setOrderStatus(ORDER_SYSTEM_CANCLE);
        ssOrderService.save(order);
    }

    // 2. 未超时订单 → 检查链上状态
    List<SsOrder> pendingOrders = ssOrderService.findPendingOrders(
        currency, CHARGE_ORDER, VIRTUAL_CURRENCY_WAIT, timeout内的订单
    );
    for (SsOrder order : pendingOrders) {
        // 加分布式锁
        asyncVirtualDepositService.deposit(orderId, currency, triggerByTg=false);
        // → checkCrypotcurrencyOrder() → 查余额/归集/确认
    }
}
```

## 十四、设计亮点与不足

### 14.1 设计亮点

#### ✅ 亮点1：完整的TRC20充值链路

- 地址生成 → 余额检测 → Gas补充 → 归集转账 → 链上确认 → 到账
- 端到端打通，经过生产验证
- **我们最核心的参考价值所在**

#### ✅ 亮点2：Gas费自动管理

- 检测用户地址TRX是否充足
- 不足时自动从Gas钱包补充
- 独立记录Gas费订单
- **我们必须实现类似机制**

#### ✅ 亮点3：外部签名服务隔离

- 出金（提现）使用独立的外部签名服务
- 出金私钥不存在业务系统中
- RSA加密通信
- **安全等级最高的做法**

#### ✅ 亮点4：双触发模式（用户+定时）

- 用户主动确认 → 即时响应
- 定时任务轮询 → 兜底确认
- 两种模式互补，不会漏单
- **可靠性设计**

#### ✅ 亮点5：提现先冻结后扣减

- 创建提现订单时冻结余额
- 链上确认成功后才真正扣减
- 失败/超时自动解冻
- **与参考工程A的freeze模型一致**

#### ✅ 亮点6：幂等性保护

- 提现时检查ssOrderId是否已存在
- BalanceChange唯一约束防重复记账
- 分布式锁 + Redis防重复提交
- **金融系统的正确做法**

#### ✅ 亮点7：乐观锁保护资产

- Assets表使用@Version乐观锁
- 并发更新时自动检测冲突
- 比悲观锁性能更好

### 14.2 设计不足

#### ❌ 不足1：AES使用ECB模式

```java
// AES.java
Cipher.getInstance("AES/ECB/PKCS5Padding")
// ECB模式不安全 — 相同明文产生相同密文
```

- **我们应该用**：AES-256-GCM 或 AES-256-CBC

#### ❌ 不足2：SHA1密钥派生

```java
// AES.encrypt 中
byte[] raw = SHA1(secret).substring(0, 16).getBytes();
// SHA1已被证明不安全，且截取方式简陋
```

- **我们应该用**：PBKDF2 或 Argon2 派生密钥

#### ❌ 不足3：无链上事件监听

- 充值确认依赖**定时轮询**（每2分钟）
- 没有监听TRON链上的TRC20 Transfer事件
- 延迟高（最多2分钟）且消耗资源
- **我们可以考虑**：监听链上事件 + 轮询兜底的混合模式

#### ❌ 不足4：无重试机制

- 广播交易失败只记日志，不重试
- 链上查询失败静默返回
- **我们应该**：加入重试+补偿机制

#### ❌ 不足5：金额精度只有2位小数

```java
DECIMAL(16,2)  // 只有2位小数
```

- USDT标准精度是6位小数
- 存在精度丢失风险
- **我们的设计**：DECIMAL(24,8)

#### ❌ 不足6：AWS凭证硬编码

```yaml
accessKeyId: REDACTED_AWS_ACCESS_KEY_ID
secretAccessKey: ...
```

- 凭证直接写在配置文件中
- **应该用**：环境变量、IAM角色或Vault

#### ❌ 不足7：缺少对账机制

- 没有链上余额与内部余额的定期对账
- 缺少差异报警
- **我们应该**：定时对账Job + 差异告警
