## 二、数据模型分析

### 2.1 核心表结构

#### 表 1：TrxWallet（用户链上钱包）

```sql
-- 每个用户一个 TRON 链上地址
CREATE TABLE TrxWallet (
    id          BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id     VARCHAR(10)  NOT NULL,     -- 用户 ID
    user_name   VARCHAR(30),               -- 用户名
    en_private_key VARCHAR(250),           -- AES 加密后的私钥
    address     VARCHAR(100),              -- TRON 地址（T 开头 Base58）
    create_time DATETIME,
    update_time DATETIME,
    create_by   VARCHAR(255),
    update_by   VARCHAR(255),
    INDEX idx_user_id (user_id)
);
```

**设计要点**：
- 一用户一地址（1:1 关系）
- 私钥加密存储：`AES(privateKeyHex, SHA1(passphrase + address)[:16])`
- passphrase 来自 AWS KMS 解密
- 地址格式：TRON Base58Check（41 前缀 → T 开头）

#### 表 2：TrxWalletOrder（充值/提现交易记录）

```sql
CREATE TABLE TrxWalletOrder (
    id          BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id     VARCHAR(10),               -- 用户 ID
    tx_id       VARCHAR(200),              -- 链上交易哈希（txId）
    ss_order_id VARCHAR(200),              -- 内部订单号
    amount      DECIMAL(16,2),             -- 交易金额
    from_address VARCHAR(250),             -- 发送地址
    to_address  VARCHAR(250),              -- 接收地址
    coin_name   VARCHAR(10),               -- 币种名（TRX/USDT/CXT）
    order_type  TINYINT(2),                -- 1=充值 2=提现
    post_type   TINYINT(2),                -- 1=API触发 2=定时任务
    create_time DATETIME,
    update_time DATETIME,
    create_by   VARCHAR(255),
    update_by   VARCHAR(255),
    INDEX idx_user_id (user_id),
    INDEX idx_order_type (order_type),
    INDEX idx_tx_id (tx_id)
);
```

**设计要点**：
- `tx_id` = TRON 链上交易哈希，**唯一标识链上交易**
- `ss_order_id` = 内部业务订单号，**关联 user-server 的 SsOrder**
- `order_type` 区分充值/提现方向
- `post_type` 区分触发来源（人工 API / 定时任务）
- 充值和提现共用同一张表

#### 表 3：trx_wallet_collect（代币归集记录）

### 2.3 关键设计观察

| 维度 | 对方设计 | 分析 |
|------|---------|------|
| 余额存储位置 | user-server 的 User 表（balance 字段） | **链上余额 ≠ 平台余额**，两者独立 |
| 链上资产 | wallet-server 管理（TrxWallet + TrxWalletOrder） | 仅管理链上操作，不管平台余额 |
| 帐变记录 | user-server 的 BalanceChange 表 | 充值确认后才写帐变 |
| 交易记录 | wallet-server 的 TrxWalletOrder 表 | 链上操作即记录，不管是否确认 |
| **职责分离** | wallet-server=链上操作 / user-server=平台余额 | **清晰的关注点分离** |

## 三、TRC20 充值全链路

### 3.1 充值架构概览

```
TRC20 充值不是"用户主动调用"——而是"链上转账后，系统轮询发现并确认"

核心思路：
  1. 用户注册时生成专属 TRON 地址
  2. 用户在外部钱包（如 TronLink）向该地址转 USDT
  3. 系统定时轮询检查该地址余额变化
  4. 发现余额变化 → 确认充值 → 平台加款
  5. 归集：将用户地址的 USDT 转到公司总地址
```

### 3.2 步骤一：钱包地址生成

```
API: POST /wallet/create
入参: { userId, userName }
出参: { address, userId, userName }

流程：
┌─────────────────────────────────────────────────────────────┐
│ WalletController.createWallet()                              │
│   └→ WalletTrxBusiness.createWallet()                        │
│       │                                                      │
│       ├─ 1. 检查用户是否已有地址（幂等）                       │
│       │     findAddressByUserId(userId)                       │
│       │     → 已存在则直接返回                                │
│       │                                                      │
│       ├─ 2. 生成 EC 密钥对                                   │
│       │     ECKeyPair ecKeyPair = Keys.createEcKeyPair()     │
│       │     → web3j 的密钥生成（secp256k1 椭圆曲线）          │
│       │                                                      │
│       ├─ 3. 创建 TRON 地址                                   │
│       │     WalletFile walletFile = Wallet.createStandard(   │
│       │         passphrase, ecKeyPair)                        │
│       │     originAddress = walletFile.getAddress()           │
│       │     finalAddress = fromHexAddress("41" + origin)      │
│       │     → "41" 前缀 = TRON 主网地址标识                   │
│       │     → fromHexAddress = Base58Check 编码 → T 开头      │
│       │                                                      │
│       ├─ 4. 加密私钥                                         │
│       │     keyStore = ecKeyPair.getPrivateKey() 的十六进制    │
│       │     enKey = AES_ECB(keyStore, SHA1(passphrase +       │
│       │                     address)[:16])                    │
│       │     → passphrase 从 AWS KMS 解密获得                  │
│       │                                                      │
│       ├─ 5. 持久化钱包记录                                   │
│       │     trxWalletService.save(userId, userName,           │
│       │         enKey, finalAddress)                          │
│       │                                                      │
│       └─ 6. 异步回调 user-server                             │
│             updateUserAddress(request)                        │
│             → POST user-server/user/updateWalletAddressById  │
└─────────────────────────────────────────────────────────────┘
```

**密钥加密方案细节**：
```
加密密钥 = SHA-1(passphrase + address) 的前 16 字节
加密算法 = AES/ECB/PKCS5Padding
加密对象 = 私钥的十六进制字符串
存储形式 = Base64 编码的密文

解密时：
  privateKeyHex = AES_decrypt(enPrivateKey, SHA1(passphrase + address)[:16])
  passphrase = AWS_KMS_decrypt(config.mainCollectPassphrase)
```

### 3.3 步骤二：用户链上转账（平台外部）

### 3.5 充值确认的关键判断逻辑

```java
// AsyncVirtualDepositService.checkCrypotcurrencyOrder() 核心逻辑：
BigDecimal balance = walletBusiness.getBalance(userId, "USDT");  // 查链上余额
if (balance >= realAmount) {
    // 余额充足 → 触发归集
    BalanceTransferResultVO result = walletBusiness.getTransferResult(...);
    if (result.getTransferResult()) {
        // 归集成功 → 确认充值
        updateOrderStatusAndSendMQ(order, result);
    }
}
```

### 4.2 提现签名分离设计

```
关键安全设计：出款私钥不在 wallet-server 本地！

提现（出款）→ 调用外部签名系统
  签名系统地址: http://34.92.121.9:80/api/admin/chain/sign
  通信加密: RSA 公钥加密请求体
  请求参数: { orderNo, outAddressNo, chainType="TRON", hashId }
  hashId = Base64(SHA256(交易 rawData))
  返回: Base64 编码的签名值

归集（内部）→ 本地签名
  使用存储在 DB 的加密私钥
  解密后直接 ECDSA 签名

原因：出款涉及资金外流，必须由独立的签名系统审批控制
      归集是内部转账（用户地址→公司地址），安全性要求低于出款
```

### 4.3 提现 vs 归集签名差异

```java
// WalletTrxBusiness.trc20Transaction() 中的关键分支：
if (deDispensingAddress.equals(from)) {
    // 出款地址发起的交易 → 调用外部签名系统
    bytes = sendSign(signUrl, tx, ssOrderId, from);
} else {
    // 其他地址（用户地址归集）→ 本地签名
    bytes = signTransaction2Byte(tx.toByteArray(),
            ByteArray.fromHexString(keyStore));
}
```

## 五、代币归集机制

### 5.1 什么是归集

```
问题：每个用户有独立的 TRON 地址，用户充值后 USDT 散落在各个地址上
解决：定期将所有用户地址的 USDT 归集到公司统一地址
目的：① 集中管理资金 ② 方便出款 ③ 降低风险
```

```java
BigInteger integer = needSendAmount
    .multiply(BigDecimal.TEN.pow(TRXContractEnum.getDecimalByTokenContract(tokenContract)))
    .toBigInteger();

// 2. ABI 编码 transfer(address, uint256)
String parameter = encoderAbi(toHexAddress(to).substring(2), integer);
// → 去掉 "41" 前缀，拼接 address + uint256 参数

map.put("contract_address", toHexAddress(tokenContract));
map.put("function_selector", "transfer(address,uint256)");
map.put("parameter", parameter);
map.put("owner_address", toHexAddress(from));
map.put("fee_limit", 50000000L);  // 50 TRX

// 4. 发送到 TRON API
String result = HttpClient4Util.doPost(trxUrl + "/wallet/triggersmartcontract", json);

// 5. 打包交易（JSON → Protobuf）
Protocol.Transaction tx = TronUtils.packTransaction(transaction.toJSONString(), false);

// 6. 签名（本地 or 外部签名系统）
byte[] bytes = signTransaction2Byte(tx.toByteArray(), privateKeyBytes);

// 7. 广播
String raw = HttpClient4Util.doPost(trxUrl + "/wallet/broadcasthex", signedTxHex);
```

## 十一、设计优点与不足评估

### 11.1 值得借鉴的设计（精华）

| # | 设计点 | 详情 | 我们的采纳方式 |
|---|--------|------|--------------|
| 1 | **充值轮询模式** | 不依赖链上回调，主动轮询余额变化 | ★ 核心借鉴——可靠且简单 |
| 2 | **充值=归集** | 确认充值的同时把资金归集到公司地址 | 资金集中管理最佳实践 |
| 3 | **Gas 费自动补充** | TRC20 归集前自动检查并补充 TRX | 必须采纳——否则 TRC20 转不出去 |
| 4 | **签名分离** | 出款用外部签名系统，归集用本地签名 | 安全架构可参考 |
| 5 | **三层加密** | AWS KMS + AES + RSA 分层保护 | 密钥分层管理思路 |
| 6 | **ssOrderId 幂等** | 防止重复上链操作 | 必须采纳 |
| 7 | **职责分离** | wallet-server=链上 / user-server=平台余额 | 清晰的关注点分离 |
| 8 | **代币合约枚举** | TRXContractEnum 统一管理合约信息 | 动态配置化改进 |
| 9 | **一用户一地址** | 简单直接，链上可追踪 | 地址管理基础模式 |
| 10 | **30 分钟超时** | 防止订单无限挂起 | 可配置超时阈值 |

### 11.2 需要规避的设计（不足）

| # | 问题 | 详情 | 我们如何改进 |
|---|------|------|------------|
| 1 | **轮询效率低** | 每 2 分钟遍历所有待充值订单 | 结合链上事件监听（TronGrid Event API）优化 |
| 2 | **无区块扫描** | 不扫描链上区块/事件，纯余额比对 | 补充区块扫描确认交易来源 |
| 3 | **归集串行** | 批量归集逐个处理，不并行 | 并发归集 + 错误隔离 |
| 4 | **金额精度** | DECIMAL(16,2) 仅 2 位小数 | 区块链场景需要 6~18 位精度 |
| 5 | **无交易确认数** | 不检查链上确认数（区块数） | 等待 N 个确认块再入账 |
| 6 | **余额≠真实充值** | 仅比较余额是否 >= 金额，不追踪具体交易 | 需解析链上转账记录精确匹配 |
| 7 | **无重入保护** | 归集过程中如果 Gas 转账成功但主转账失败，Gas 被消耗 | 需要状态机管理多步操作 |
| 8 | **AWS KMS 强依赖** | 每次操作都调 KMS API | 启动时解密一次缓存于内存 |
| 9 | **硬编码签名 URL** | `http://34.92.121.9:80` 写死在 yml | 配置中心管理 |
| 10 | **无事件通知** | 充值成功后不主动推送，等轮询发现 | WebSocket / SSE 实时通知 |
| 11 | **catch 吞异常** | 多处 `catch (Exception e) { log.error }` 静默失败 | 错误需要上报和告警 |
| 12 | **无对账机制** | 没有链上-平台的定期对账 | 需要每日对账任务 |

### 12.2 核心模式平移

#### 模式 1：地址生成

```go
// Java: Keys.createEcKeyPair() + "41" 前缀 + Base58Check
// Go:

import (
    "crypto/ecdsa"
    "crypto/rand"
    "github.com/ethereum/go-ethereum/crypto"
    "github.com/mr-tron/base58"
)

func GenerateTronAddress() (address string, privateKeyHex string, err error) {
    // 1. 生成 secp256k1 密钥对
    privateKey, err := ecdsa.GenerateKey(crypto.S256(), rand.Reader)

    // 2. 取公钥 → Keccak256 → 取后 20 字节
    pubBytes := crypto.FromECDSAPub(&privateKey.PublicKey)
    hash := crypto.Keccak256(pubBytes[1:])
    addressBytes := hash[len(hash)-20:]

    // 3. 加 0x41 前缀（TRON 主网）
    tronAddr := append([]byte{0x41}, addressBytes...)

    // 4. Base58Check 编码
    address = base58.CheckEncode(tronAddr)
    privateKeyHex = hex.EncodeToString(crypto.FromECDSA(privateKey))
    return
}
```

```go
    })

    // 3. 解析 Uint256 返回值

    return balance, nil
}
```

#### 模式 3：充值轮询

```go
// Java: @Scheduled(cron="0 */2 * * * ?") + 查余额 + 归集
// Go:

func (s *DepositScanner) Start() {
    ticker := time.NewTicker(2 * time.Minute)
    for range ticker.C {
        s.checkPendingDeposits()
    }
}

func (s *DepositScanner) checkPendingDeposits() {
    orders := s.repo.FindPendingDepositOrders()
    for _, order := range orders {
        if time.Since(order.CreatedAt) > 30*time.Minute {
            s.cancelOrder(order)
            continue
        }
        balance := s.tronClient.GetTRC20Balance(order.Address, usdtContract)
        if balance.Cmp(order.Amount) >= 0 {
            txId, err := s.collect(order)
            if err == nil {
                s.confirmDeposit(order, txId)
            }
        }
    }
}
```

```
   └── 私钥加密存储（参考对方 AES 方案）

   └── 出款额度限制 + 审批流程
```
