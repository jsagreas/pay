### 1.3 核心设计原则

- **IDL先行**：所有接口先在Thrift IDL中定义，再生成代码
- **网关透传**：网关不处理业务逻辑，只负责认证、路由、转发
- **ETCD驱动**：配置、服务发现、端口分配全部通过ETCD
- **代码生成**：Model/Query/Repo层由gorm/gen自动生成，手写业务逻辑层
- **公共复用**：所有基础能力在common模块中，服务模块不重复造轮子

## 3. Common 模块 — 公共基础设施

### 3.1 RPC 通用 Handler（网关→服务的桥梁）

> 来源：`/Users/mac/gitlab/common/rpc/handler.go`

提供 3 种泛型 Handler 函数，网关handler中直接使用：

```go
// 1. 标准Handler —— 有请求体的接口
func Handler[REQ any, RESP any](ctx, c, rpcMethod)
// 流程：BindAndValidate请求 → 调用RPC方法 → ret.HttpOne统一返回

// 2. 无参Handler —— 无请求体的接口
func HandlerNoParam[RESP any](ctx, c, rpcMethod)
// 流程：直接调用RPC方法 → ret.HttpOne统一返回

// 3. CSV下载Handler —— 文件导出接口
func HandlerWithCSV[REQ, RESP any](ctx, c, rpcMethod)
// 流程：BindAndValidate → 调用RPC → 检查是否实现CSVResponse接口 → 是则返回文件/否则返回JSON
```

**网关handler中的调用极其简洁**（以banner为例）：

```go
// gate-back/biz/handler/app/banner.go
func CreateBanner(ctx context.Context, c *app.RequestContext) {
    rpc.Handler(ctx, c, rpc.AppClient().CreateBanner)  // 一行搞定
}
```

### 3.4 统一返回格式

> 来源：`/Users/mac/gitlab/common/pkg/ret/project_ret.go`

**HTTP 标准返回结构**：
```json
{
    "traceId": "uuid-string",
    "code": 0,          // 0=成功, 1=严重异常, 6xxxxxx=业务错误
    "msg": "",           // 成功为空，异常为i18n翻译后的消息
    "data": {}           // 业务数据
}
```

**核心函数**：
```go
// HTTP层
ret.HttpOk(ctx, data)                     // 成功返回
ret.HttpErr(ctx, err, i18nClient)          // 错误返回（自动翻译错误码）
ret.HttpOne(ctx, data, i18nClient, err)    // 统一返回（根据err判断走Ok还是Err）

// RPC层（服务内部使用）
ret.BizErr(ctx, errCode, "错误描述")        // 抛出业务错误
ret.RpcOne(ctx, data, err, defaultCode)    // RPC统一返回
ret.GetBizErrCode(err, defaultCode)        // 提取错误码
```

**BizErr 原理**：底层使用 Kitex 的 `kerrors.NewBizStatusError(code, msg)` 封装错误，HTTP网关通过 `kerrors.FromBizStatusError` 解包后自动调用 ser-i18n 翻译错误码为多语言消息。

### 3.5 链路追踪

> 来源：`/Users/mac/gitlab/common/pkg/tracer/tracer.go`

在网关入口自动生成并注入以下Header，贯穿整个请求链路：

```
X-Trace-Id      → UUID（唯一请求标识）
X-Client-IP     → 客户端IP
X-User-Token    → 用户Token
X-User-ID       → 用户ID（int64）
X-User-Name     → 用户名/昵称
X-User-Language → 客户端语言偏好
X-User-Device-ID   → 设备标识
X-User-Device-Type → 设备类型（Android/iOS/H5）
```

**同时自动设置 CORS 头**，包括 `Access-Control-Expose-Headers: Content-Disposition`。

### 3.7 认证中间件

> 来源：`/Users/mac/gitlab/common/pkg/middleware/gate-auth.go`

| 中间件 | 网关 | 作用 | 原理 |
|--------|------|------|------|
| `BlackCheck()` | gate-font | IP黑名单校验 | 调用 `rpc.IpClient().OnBlackList(ctx)`，在黑名单中则拒绝 |
| `WhiteCheck()` | gate-back | IP白名单校验 | 调用 `rpc.IpClient().OnWhiteList(ctx)`，不在白名单则拒绝 |
| `FontAuthMiddleware()` | gate-font | C端Token认证 | Redis key = `user:token:{token}` → 解析 `CUserTokenPayload{ID,NickName}` |
| `BackAuthMiddleware()` | gate-back | B端Token+URL鉴权 | Redis验证Token → 解析 `BUserTokenPayload{ID,UserName}` → 调用 `rpc.BUserClient().CheckUrlAuth(fullPath)` |

**关键区别**：
- C端认证只校验Token有效性（登录即可访问）
- B端认证在Token之上还有URL级别的RBAC权限校验

## 4. 双网关模块 — gate-back / gate-font

### 4.1 gate-back（B端后台网关）

> 来源：`/Users/mac/gitlab/gate-back/`

**main.go 初始化顺序**：
1. `lg.InitHLog()` — 初始化日志
2. `etcd.LoadAndWatch("/slg/conf/")` — 加载配置+Watch变更
3. `middleware.InitRedis()` — 初始化Redis（认证依赖）
4. `mys3.Init(...)` — 初始化S3（S3重定向依赖）
5. 获取端口（命令行参数 > ETCD `EtcdGateBackPort`）
6. 创建Hertz服务器 + 注册Tracer + 设置20MB请求缓冲
7. `my_router.InitMyRouter(h)` — 初始化路由组
8. `register(h)` → `GeneratedRegister(h)` — 注册所有模块路由
9. `h.Spin()` — 启动服务

**路由组定义**（`biz/router/my_router/my_router.go`）：
```go
Open = h.Group("/admin/open", middleware.SetHeaderInfo(), middleware.WhiteCheck())
Api  = h.Group("/admin/api",  middleware.SetHeaderInfo(), middleware.WhiteCheck(), middleware.BackAuthMiddleware())
```

**URL路径规范**：`/admin/{open|api}/{模块名}/{功能}/{动作}`
- 示例：`/admin/api/app/banner/create`
- 示例：`/admin/open/buser/user/login`

**当前已注册的模块**（14个）：
buser、bcom、i18n、blog、user、enum、kyc、item、app、exp、s3、live、shortvideos、rtc

### 4.3 两个网关的核心差异

| 维度 | gate-back (B端) | gate-font (C端) |
|------|-----------------|-----------------|
| URL前缀 | `/admin/{group}` | `/{group}` |
| IP策略 | 白名单（必须在内） | 黑名单（不在就放行） |
| 认证方式 | Token + URL权限(RBAC) | Token认证即可 |
| Token Redis Key | 直接用token作key | `user:token:{token}` |
| Token载荷 | `BUserTokenPayload{ID,UserName}` | `CUserTokenPayload{ID,NickName}` |
| 模块数 | 14 | 8 |

### 4.4 新模块在网关中的注册步骤

以钱包模块为例：

**步骤1：创建 handler 文件**

### 6.3 类型定义规范

> 来源：`/Users/mac/gitlab/common/idl/ser-app/banner.thrift` 实际代码

```thrift
namespace go ser_wallet

// ===== 实体结构 =====
struct WalletItem {
    1:  i64 id;
    2:  i64 userId;
    3:  string currencyCode;   // 币种代码
    4:  i32 walletType;        // 钱包类型：1中心 2奖励 3主播 4代理 5场馆
    5:  string balance;        // 余额（字符串避免精度丢失）
    6:  i32 status;            // 状态：0冻结 1正常
    7:  i64 createAt;          // 创建时间（毫秒时间戳）
    8:  i64 updateAt;          // 更新时间
}

// ===== 请求/响应 =====
struct GetBalanceReq {
    1: optional string currencyCode;  // 币种代码，不传则返回所有
}

struct GetBalanceResp {
    1: list<WalletItem> wallets;
}

// 分页查询
struct PageRecordReq {
    1: optional string currencyCode;
    2: optional i32 recordType;       // 交易类型：-1全部
    3: optional i64 startTime;
    4: optional i64 endTime;
    5: optional i32 pageNo;           // 页码从1开始
    6: optional i32 pageSize;         // 每页条数
}

struct PageRecordResp {
    1: list<RecordItem> pageList;
    2: i64 total;
    3: i32 pageNo;
    4: i32 pageSize;
}
```

**Thrift类型注意事项**：
- `required` 表示必填字段
- `optional` 表示可选字段（Go中生成为指针类型）
- 金额使用 `string` 类型（避免浮点精度问题）
- 时间使用 `i64` 毫秒时间戳
- 分页响应固定包含 `pageList + total + pageNo + pageSize`
- 无参方法直接 `MethodName()` 不写参数（参考 `ListHomeCategory()`）

```
   SetHeaderInfo() → 提取ClientIP/Lang/DeviceID/DeviceType/Token
   BlackCheck()    → 调用 rpc.IpClient().OnBlackList() 检查IP
   FontAuthMiddleware() → Redis验证Token → 注入UserID/UserName到metainfo

3. 路由匹配 → wallet.GetBalance handler

4. Handler调用
   rpc.Handler(ctx, c, rpc.WalletClient().GetBalance)
   ├── BindAndValidate(req)    // 绑定JSON请求体
   ├── rpc.WalletClient()      // sync.Once获取客户端
   │   └── 通过ETCD发现 wallet_service 地址
   └── WalletClient.GetBalance(ctx, req)  // Kitex RPC调用
       ├── TTHeader传递 trace_id/user_id/user_name 等元数据
       └── 3秒超时

5. ser-wallet 服务接收
   handler.go → WalletServiceImpl.GetBalance(ctx, req)
   └── walletService.GetBalance(ctx, req)  // 委托给ser层

6. ser层业务逻辑
   ├── 从ctx中获取userId（tracer.GetUserId(ctx)）
   ├── 查询数据库
   ├── 可能调用其他RPC（如 rpc.XXXClient().Method()）
   └── 返回 (resp, nil) 或 ret.BizErr(ctx, errCode, msg)

7. 响应返回
   ret.HttpOne(ctx, resp, rpc.I18nClient(), err)
   ├── 成功 → {"traceId":"xxx","code":0,"data":{...}}
   └── 失败 → 自动通过I18n翻译错误码 → {"traceId":"xxx","code":6022001,"msg":"余额不足"}
```

## 11. 数据库层规范

### 11.1 表结构设计规范

> 来源：对 ser-app banner 表结构的分析

**通用字段**（每张表都应包含）：
```sql
id           BIGINT       PRIMARY KEY AUTO_INCREMENT
create_by    BIGINT       NOT NULL COMMENT '创建人ID'
create_by_name VARCHAR(50) NOT NULL COMMENT '创建人名称'
create_at    BIGINT       NOT NULL COMMENT '创建时间（毫秒时间戳）'
update_by    BIGINT       NOT NULL COMMENT '更新人ID'
update_by_name VARCHAR(50) NOT NULL COMMENT '更新人名称'
update_at    BIGINT       NOT NULL COMMENT '更新时间（毫秒时间戳）'
delete_at    BIGINT       NOT NULL DEFAULT 0 COMMENT '删除时间（0=未删除）'
```

**设计要点**：
- 时间字段使用 `BIGINT` 存毫秒时间戳（不用 datetime/timestamp）
- 软删除使用 `delete_at`（0=未删除，非0=删除时的时间戳）
- 金额字段使用 `DECIMAL(20,8)` 或 `VARCHAR`（避免浮点精度问题）
- 主键使用自增 `BIGINT`，业务中需要分布式ID时使用 `utils.GenerateSnowflakeID()`
- nullable字段在gorm/gen中会生成指针类型（`*string`）

### 11.2 Repository 层编码规范

> 来源：`/Users/mac/gitlab/ser-app/internal/rep/banner.go` 实际代码

```go
package rep

type WalletRepo struct{}

func NewWalletRepo() *WalletRepo { return &WalletRepo{} }

// 查询（使用类型安全的query builder）
func (r *WalletRepo) GetByUserAndCurrency(ctx context.Context, userId int64, code string) (*model.Wallet, error) {
    w := query.Q.Wallet
    wallet, err := w.WithContext(ctx).
        Where(w.UserID.Eq(userId)).
        Where(w.CurrencyCode.Eq(code)).
        Where(w.DeleteAt.Eq(0)).
        First()
    if err != nil {
        if errors.Is(err, gorm.ErrRecordNotFound) {
            return nil, nil  // 未找到返回nil，不返回error
        }
        return nil, err
    }
    return wallet, nil
}

// 分页查询
func (r *WalletRepo) PageRecords(ctx context.Context, filter *RecordFilter) ([]*model.Record, int64, error) {
    q := query.Q.TransactionRecord.WithContext(ctx).
        Where(query.Q.TransactionRecord.DeleteAt.Eq(0))

    // 条件过滤（动态构建）
    if filter.UserID > 0 {
        q = q.Where(query.Q.TransactionRecord.UserID.Eq(filter.UserID))
    }
    if filter.CurrencyCode != "" {
        q = q.Where(query.Q.TransactionRecord.CurrencyCode.Eq(filter.CurrencyCode))
    }
    if filter.RecordType > 0 {
        q = q.Where(query.Q.TransactionRecord.RecordType.Eq(filter.RecordType))
    }

    // 排序+分页
    q = q.Order(query.Q.TransactionRecord.CreateAt.Desc())
    records, total, err := q.FindByPage(filter.Offset, filter.Limit)
    return records, total, err
}

// 事务操作
func (r *WalletRepo) TransferBalance(ctx context.Context, fromID, toID int64, amount string) error {
    return query.Q.Transaction(func(tx *query.Query) error {
        w := tx.Wallet.WithContext(ctx)
        // 1. 扣减来源钱包
        _, err := w.Where(tx.Wallet.ID.Eq(fromID)).
            UpdateSimple(tx.Wallet.Balance.Sub(amount))
        if err != nil {
            return err
        }
        // 2. 增加目标钱包
        _, err = w.Where(tx.Wallet.ID.Eq(toID)).
            UpdateSimple(tx.Wallet.Balance.Add(amount))
        return err
    })
}
```

## 17. 关键注意事项与红线

### 17.1 IDL 相关

- **修改IDL后必须重新执行 gen_kitex.sh**，否则代码和IDL不一致
- **不要手动修改 kitex_gen/ 下的生成代码**
- **不要手动修改 gorm_gen/ 下的生成代码**（model/query/repo 都是生成的）
- service.thrift 中的 namespace 必须是 `namespace go ser_wallet`（下划线连接）
- IDL文件之间通过 `include "xxx.thrift"` 引用，路径是相对路径

### 17.2 编码相关

- handler.go 只做委托，**不允许在handler中写业务逻辑**
- ser 层是核心业务逻辑层，**所有校验、事务、缓存操作都在这里**
- rep 层只做数据库操作，**不包含业务判断**
- 错误返回使用 `ret.BizErr(ctx, errCode, msg)`，**不要直接 return errors.New()**
- 金额计算使用 `decimal` 库或字符串，**绝对禁止用 float64**
- 时间统一使用毫秒时间戳（`time.Now().UnixMilli()`），**不使用 time.Time**

### 17.3 ETCD 相关

- 配置前缀必须以 `/slg/` 开头
- 服务端口和DB名走 `/slg/serv/{shortname}/` 路径
- 全局配置走 `/slg/conf/` 路径
- 新增 ETCD key 后要在 namesp.go 中注册常量

### 17.4 RPC 相关

- RPC 超时统一 3 秒，**不要在业务代码中自行设置超时**
- 客户端通过 ETCD 服务发现，**不要硬编码服务地址**
- 新增 RPC 客户端必须在 `common/rpc/rpc_client.go` 中注册
- 被调用方的幂等性由被调用方自己保证

### 17.5 网关相关

- B端路由前缀 `/admin/api` 或 `/admin/open`
- C端路由前缀 `/api` 或 `/open`
- 所有接口统一使用 `POST` 方法
- 路由注册在 `register.go` 的 `GeneratedRegister` 函数中集中管理
- 新模块必须同时在两个网关中注册（如果B端和C端都有接口的话）

### 17.6 数据库相关

- 所有表必须有软删除字段 `delete_at`
- 查询时必须加 `delete_at = 0` 条件
- 修改数据前检查 `delete_at = 0`（防止操作已删除的数据）
- gorm/gen 生成的 query 使用 `WithContext(ctx)` 传递上下文
