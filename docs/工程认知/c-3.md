### 1.2 请求完整链路

```
客户端 HTTP 请求
  → Hertz 网关 (gate-font / gate-back)
    → SetHeaderInfo 中间件（提取 IP、语言、设备、Token 注入 metainfo）
    → BlackCheck / WhiteCheck 中间件（IP 黑/白名单过滤）
    → FontAuthMiddleware / BackAuthMiddleware（Token 认证，注入 UserID）
    → 路由 Handler 调用 rpc.Handler[REQ, RESP](ctx, c, rpc.XxxClient().SomeMethod)
      → Handler 绑定&校验 HTTP Body → 调用 Kitex RPC 方法
        → Kitex Client 通过 ETCD 解析服务地址 → TTHeader 协议发送请求
          → Kitex Server 接收，访问日志中间件记录，分发到 handler
            → handler.go 纯委派 → ser 服务层（校验→仓储→缓存失效→日志）
              → rep 仓储层（GORM Gen 类型安全查询）
              → cache 缓存层（Redis 读写/失效）
            ← 返回 IDL 响应结构
        ← RPC 响应回传
      → ret.HttpOne 包装为统一 JSON: R{TraceId, Code, Msg, Data}
    ← HTTP 响应返回客户端
```

#### 3.1.4 泛型 Handler — `handler.go`

三种泛型 Handler 桥接 HTTP 到 RPC：

**Pattern 1: `Handler[REQ, RESP]()` — 标准 RPC 代理（最常用）**
```go
func Handler[REQ any, RESP any](
    ctx context.Context,
    c *app.RequestContext,
    rpcMethod func(context.Context, REQ, ...callopt.Option) (RESP, error),
)
```
流程：反射创建 REQ → `c.BindAndValidate` → 调用 rpcMethod → `ret.HttpOne(ctx, resp, I18nClient(), err)`

**Pattern 2: `HandlerNoParam[RESP]()` — 无参数**
```go
func HandlerNoParam[RESP any](
    ctx context.Context,
    c *app.RequestContext,
    rpcMethod func(context.Context, ...callopt.Option) (RESP, error),
)
```
跳过绑定，直接调用 RPC 方法。

**Pattern 3: `HandlerWithCSV[REQ, RESP]()` — CSV/文件下载**
```go
func HandlerWithCSV[REQ any, RESP any](
    ctx context.Context,
    c *app.RequestContext,
    rpcMethod func(context.Context, REQ, ...callopt.Option) (RESP, error),
)
```
如果响应实现了 `CSVResponse` 接口（`GetFilename() string` + `GetData() []byte`），返回文件下载；否则回退到 JSON。

#### 3.2.2 缓存与监听 — `etcd_cache.go`

**核心结构：**
```go
type Watcher struct {
    cache     atomic.Value              // 存储 map[string]string
    watchPath string                    // 监听前缀
    hookMap   map[string]func(val string) // 变更回调
}
```

**公共 API：**
```go
func LoadAndWatch(prefix string)                      // 创建默认 configWatcher
func CustomWatcher(prefix string) *Watcher            // 创建自定义 watcher
func Get(ctx context.Context, key string) string       // 从默认缓存读
func GetDef(ctx, key, defaultVal string) string        // 带默认值读
func Hook(key string, f func(val string))              // 注册变更回调
```

**监听机制：**
1. `downAndWatch()` 全量 GET 前缀下所有 KV，存入 `map[string]string`（通过 `atomic.Value.Store`）
2. 启动 goroutine `watchLoop()` 监听 `client.Watch(prefix, WithPrefix())`
3. 每次事件使用 **Copy-on-Write**：加载当前 map → 复制新 map → 应用 PUT/DELETE → 存储新 map
4. 更新后检查 `hookMap`，触发已注册的回调


#### 3.3.3 Token 认证

**C端认证 — `FontAuthMiddleware()`：**
```
1. 提取 token: tracer.GetUserToken(ctx)
2. token 为空 → abort: errs.TokenNotExist
3. Redis 查找: rds.Get("user:token:" + token)
4. Redis 返回空 → abort: errs.TokenAnalysisFailed
5. 反序列化为 session.CUserTokenPayload
6. payload.ID == 0 → abort: errs.TokenAnalysisFailed
7. 注入上下文:
   - X-User-ID = base.ID
   - X-User-Name = base.NickName
8. c.Next(ctx)
```

**B端认证 — `BackAuthMiddleware()`：**
- 类似流程但使用 `session.BUserTokenPayload`（`{ID, UserName}`）
- **额外** URL 级别权限检查：`rpc.BUserClient().CheckUrlAuth(ctx, c.FullPath())`

**Token 结构体**（`/Users/mac/gitlab/common/pkg/session/login.go`）：

### 5.2 Thrift IDL 规范

#### 5.2.1 命名空间规范

共享类型：
```thrift
namespace go common
```

#### 5.2.2 服务接口命名

```thrift
service AppService { ... }
service WalletService { ... }
service CoinCfgService { ... }
```

#### 5.2.3 请求/响应命名模式

严格的 `{Action}{Entity}Req` / `{Action}{Entity}Resp` 规范：

| 动作前缀 | 用途 | 请求 | 响应 |
|---------|------|------|------|
| `Create` | 创建实体 | `CreateBannerReq` | `CreateBannerResp` |

| `Page` | B端分页列表 | `PageBannerReq` | `PageBannerResp` |
| `Detail` | 单条详情 | `DetailBannerReq` | `DetailBannerResp` |
| `List` | C端完整列表 | `ListBannerReq` | `ListBannerResp` |
| `Get` | 通用获取 | `GetEnumOptionsReq` | `GetEnumOptionsResp` |

```thrift
struct PageXxxResp {
```

## 七、核心设计模式总结

### 7.1 数据流：Handler → Service → Repo → Cache

```
[Kitex RPC] → handler.go (纯委派，零逻辑)
    → ser/{service}.go (校验、业务规则、编排)
        → rep/{entity}.go (GORM Gen 查询、软删除、事务)
        → cache/{entity}.go (Redis 读写/失效)
        ← ser/{service}.go (模型→IDL 响应转换、审计日志)
    ← handler.go
[Kitex RPC 响应]
```

### 7.2 各层错误处理

| 层 | 模式 | 示例 |
|----|------|------|
| **Handler** | 无错误处理；直接透传 | `return s.service.Method(ctx, req)` |
| **Service** | `ret.BizErr(ctx, errs.XXX, "message")` | `ret.BizErr(ctx, errs.BannerNotFound)` |
| **Repository** | 返回原始 GORM error；未找到返回 `nil, nil` | `if gorm.ErrRecordNotFound` |
| **Cache** | 缓存未命中返回 `nil`（不传播错误） | `if data == "" { return nil }` |
| **Common** | `6000XXX` 跨切面错误 | `errs.ParamErr` (6000002) |

### 7.3 分页标准

**IDL 层：**
```thrift
struct PageXxxReq {
    N: optional i32 pageNo;      // 从 1 开始
    N+1: optional i32 pageSize;  // 15/30/50/100
}
```

**Service 层归一化：**
```go
pageNo := 1; pageSize := 15  // 默认值
if req.PageNo != nil && *req.PageNo > 0 { pageNo = int(*req.PageNo) }
if req.PageSize != nil && *req.PageSize > 0 { pageSize = int(*req.PageSize) }
if pageSize > 100 { pageSize = 100 }  // 硬上限
```

**Repository 层：**
```go
offset := (pageNo - 1) * pageSize
list, total, err := q.Order(...).FindByPage(offset, pageSize)
```

### 7.4 软删除规范

- **列名**：`delete_at` 或 `deleted_at`（毫秒时间戳）
- **未删除值**：`0`（非 NULL）
- **查询过滤**：`.Where(b.DeleteAt.Eq(0))` — **每次查询都必须加**
- **删除操作**：设置 `delete_at = time.Now().UnixMilli()`（从不 DELETE 行）
- **生成的 repo 函数是硬删除**，软删除在手写 rep 层实现

### 7.5 缓存失效策略

**模式：写时全量清除（无选择性失效）**
- 每次写操作调用 `cache.DelAll()`，枚举删除所有可能的 cache key 组合
- C端读使用缓存旁路：Redis 优先 → 未命中查 DB → 写 Redis
- 空结果也缓存（防缓存穿透）
- 缓存 TTL：Banner 列表 5分钟，枚举选项 1小时

### 7.6 事务模式

使用 GORM Gen 类型化事务：
```go
query.Q.Transaction(func(tx *query.Query) error {
    // tx.Banner 提供事务内的类型安全访问
    if _, err := tx.Banner.WithContext(ctx).Where(...).UpdateSimple(...); err != nil {
        return err  // 自动回滚
    }
    return tx.Banner.WithContext(ctx).Omit(tx.Banner.ID).Create(entity)
    // nil 返回自动提交
})
```

### 7.7 单例初始化模式

所有基础设施组件（ETCD、Redis、MySQL、ES、S3、NATS、Kafka、Snowflake、RPC Client）都使用 `sync.Once` 实现线程安全的延迟初始化。
