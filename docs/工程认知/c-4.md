#### 2.1.9 ETCD 配置管理

### 2.2 gate-back —— B端网关

**职责**：所有后台管理请求的统一入口，路由到各业务服务的 RPC 方法。

#### 2.2.1 路由架构

**文件**：`/Users/mac/gitlab/gate-back/biz/router/my_router/my_router.go`

```go
func InitMyRouter(h *server.Hertz) {
    // B端不需要认证的接口（如登录）
    Open = h.Group("/admin/open", middleware.SetHeaderInfo(), middleware.WhiteCheck())
    // B端需要认证 + URL鉴权的接口
    Api = h.Group("/admin/api", middleware.SetHeaderInfo(), middleware.WhiteCheck(), middleware.BackAuthMiddleware())
    // S3 文件直接重定向
    s3Group := h.Group("/s3")
    s3Group.GET("/*path", middleware.RedirectS3())
}
```

**中间件链**：
- `/admin/open/*` → SetHeaderInfo → WhiteCheck（IP白名单）
- `/admin/api/*` → SetHeaderInfo → WhiteCheck → BackAuthMiddleware（Token验证 + URL权限检查）

**BackAuthMiddleware 的两步验证**：
1. 从 Redis 读取 Token (`user:token:{token}`) 解析为 `BUserTokenPayload`
2. 调用 `rpc.BUserClient().CheckUrlAuth(ctx, c.FullPath())` 做 URL 级别的 RBAC 权限校验

### 2.3 gate-font —— C端网关

**职责**：所有用户端请求的统一入口。

#### 2.3.1 与 gate-back 的核心差异

| 维度 | gate-back（B端） | gate-font（C端） |
|------|------------------|------------------|
| URL前缀 | `/admin/open/*`, `/admin/api/*` | `/open/*`, `/api/*` |
| IP策略 | `WhiteCheck()` — 白名单放行 | `BlackCheck()` — 黑名单拦截 |
| 鉴权中间件 | `BackAuthMiddleware()` — Token + URL权限 | `FontAuthMiddleware()` — 仅Token验证 |
| Token载荷 | `BUserTokenPayload{ID, UserName}` | `CUserTokenPayload{ID, UID, NickName, ...}` |
| 请求体限制 | 20MB (`WithMaxRequestBodySize(20<<20)`) | 默认 |

**文件**：`/Users/mac/gitlab/gate-font/biz/router/my_router/my_router.go`

```go
func InitMyRouter(h *server.Hertz) {
    Open = h.Group("/open", middleware.SetHeaderInfo(), middleware.BlackCheck())
    Api = h.Group("/api", middleware.SetHeaderInfo(), middleware.BlackCheck(), middleware.FontAuthMiddleware())
    s3Group := h.Group("/s3")
    s3Group.GET("/*path", middleware.RedirectS3())
}
```

**开发 ser-wallet 时**，C端路由应为：
```
/api/wallet/balance                    ← 查询余额
/api/wallet/deposit/methods            ← 充值方式列表
/open/wallet/currencies                ← 币种列表（可能无需鉴权）
```

#### 2.3.2 FontAuthMiddleware 鉴权流程

```go
// 1. 从 ctx 获取 Token（SetHeaderInfo 已提取到 ctx）
token := tracer.GetUserToken(ctx)

// 2. Redis 查询 Token
data := rds.Get("user:token:" + token)

// 3. 解析为 C端载荷
var base session.CUserTokenPayload  // {ID, UID, NickName, LastLoginIp, ...}
json.Unmarshal([]byte(data), &base)

// 4. 注入 UserID 到上下文（后续 RPC 自动传播）
ctx = metainfo.WithPersistentValue(ctx, tracer.UserIdHeader, strconv.FormatInt(base.ID, 10))
```

这意味着 ser-wallet 的 RPC 方法里直接 `tracer.GetUserId(ctx)` 就能拿到当前用户ID。

#### 3.1.2 IDL 命名规范（从既有代码归纳）

| 元素 | 规范 | 示例 |
|------|------|------|
| namespace | `namespace go ser_{模块名}` | `namespace go ser_wallet` |
| 服务名 | `{模块}Service` (PascalCase) | `WalletService` |
| 方法名 | PascalCase，动词+名词 | `CreateCurrency`, `PageBalance` |
| 请求结构 | `{Action}{Entity}Req` | `CreateCurrencyReq` |
| 响应结构 | `{Action}{Entity}Resp` | `CreateCurrencyResp` |
| 字段名 | camelCase | `currencyCode`, `pageNo` |
| 文件拆分 | 按功能域拆分子文件 | `wallet.thrift`, `currency.thrift` |
| 引用方式 | 相对路径 include | `include "../common/enum.thrift"` |

#### 3.1.3 请求/响应结构规范

**参考**：`/Users/mac/gitlab/common/idl/ser-app/banner.thrift`

```thrift

// 创建请求：required 标识必填
struct CreateBannerReq {
    1: required string bannerName
    2: required i32 jumpType
    3: optional string jumpTarget        // optional 标识选填
    10: required i32 sortValue
}

struct CreateBannerResp {
    1: i64 id
}

// 分页请求：固定 pageNo/pageSize 在高编号位
struct PageBannerReq {
    1: required string pagePosition
    2: optional string bannerName        // 筛选条件用 optional
    3: optional i32 status
    10: optional i32 pageNo              // 分页参数
    11: optional i32 pageSize
}

// 分页响应：固定四字段
struct PageBannerResp {
    1: list<BannerItem> pageList
    2: i64 total
    3: i32 pageNo
    4: i32 pageSize
}

// 列表项结构
struct BannerItem {
    1: i64 id
    2: string bannerName
    // ...
}
```

**Thrift 字段类型映射**：

| Thrift | Go | 场景 |
|--------|-----|------|
| `i32` | `int32` | 状态、页码、页大小 |
| `i64` | `int64` | ID、金额（分）、时间戳 |
| `string` | `string` | 文本 |
| `bool` | `bool` | 开关 |
| `binary` | `[]byte` | 文件内容 |
| `list<T>` | `[]*T` | 列表 |
| `map<K,V>` | `map[K]V` | 字典 |

**注解用法**（参考 `common/idl/readme.md`）：
```thrift
struct SomeReq {
    1: i64 id (api.query="id", api.body="id")          // 同时支持 query 和 body
    2: string name (go.tag='form:"name"')                // 表单字段
    3: binary content (go.tag='form:"content"')          // 文件上传
}
```

```go
type UserService struct{}

func (s *UserService) GetUserMapByIds(ctx context.Context, userIds []int64) (map[int64]string, error) {
    if len(userIds) == 0 {
        return make(map[int64]string), nil
    }
    userMap, err := rpc.BUserClient().QueryUserByIds(ctx,
        &ser_buser.RpcSysUserReq{Ids: userIds})
    if err != nil {

    }
    return userMap, nil
}
```

**关键原则**：
- 通过 `rpc.XxxClient()` 获取客户端，无需手动管理连接

- 3秒超时，失败时应有优雅降级（不强依赖）

### 5.2 必须遵守的规范

**1. 软删除**：所有表必须有 `delete_at` 字段（0=未删除，非0=删除时间戳），查询条件必须带 `WHERE delete_at = 0`。

**2. 审计字段**：所有表必须有 `create_by`、`update_by`、`create_at`、`update_at`。GORM UserPlugin 会自动填充，但表结构必须有这些列。

**3. 错误码不用 0**：
```go
// ❌ 错误
return nil, ret.BizErr(ctx, 0, "xxx")
// ✅ 正确
return nil, ret.BizErr(ctx, errs.WalletNotFound, "钱包不存在")
```
`project_ret.go:33` 明确写了：`禁止使用0值作为错误码，框架不支持`。

**4. 日志必须带 ctx**：
```go
// ❌ 错误
log.Println("something happened")
// ✅ 正确
lg.Log().CtxInfof(ctx, "something happened")  // 自动关联 TraceID
```

**5. RPC 方法签名固定模式**：
```go
// IDL 生成的接口：
Method(ctx context.Context, req *ReqType) (resp *RespType, err error)
// handler 层必须严格按此签名实现
```

**6. 分页参数校验**：
```go
if req.GetPageNo() <= 0 || req.GetPageSize() <= 0 {
    return nil, ret.BizErr(ctx, errs.ParamErr, "pageNo/pageSize 必须大于0")
}
```

### 5.3 既有模块的代码模式（必须对齐）

**Service 层标准模式**（从 ser-app 归纳）：

```go
func (s *SomeService) Create(ctx context.Context, req *ser_wallet.CreateSomeReq) (*ser_wallet.CreateSomeResp, error) {
    // 1. 参数校验
    if req == nil || req.SomeField == "" {
        return nil, ret.BizErr(ctx, errs.ParamErr, "参数缺失")
    }

    // 2. 日志记录入参
    b, _ := json.Marshal(req)
    lg.Log().CtxInfof(ctx, "Some(Create) 入参=%s TraceId=%s", b, tracer.GetTraceID(ctx))

    // 3. 业务逻辑（查重、规则校验等）
    exists, err := s.repo.ExistsByCode(ctx, req.Code)
    if err != nil {
        return nil, ret.BizErr(ctx, errs.DatabaseErr, err.Error())
    }
    if exists {
        return nil, ret.BizErr(ctx, errs.CodeAlreadyExist)
    }

    // 4. 数据库操作
    id, err := s.repo.Create(ctx, &model.SomeModel{
        Code: req.Code,
        Name: req.Name,
        // ... 注意不需要手动设置 CreateBy/CreateAt
    })
    if err != nil {
        return nil, ret.BizErr(ctx, errs.DatabaseErr, err.Error())
    }

    // 5. 返回结果
    return &ser_wallet.CreateSomeResp{Id: id}, nil
}
```

## 十、总结

开发 ser-wallet 模块，核心认知可以浓缩为：

1. **IDL 先行**：一切从 Thrift IDL 开始，生成代码是骨架
2. **改 common 4 个文件**：namesp.go + rpc_client.go + IDL目录 + go.work
3. **服务内部严格分层**：handler(零逻辑) → ser(业务) → rep(数据) → gorm_gen(生成)
4. **网关只做路由**：一行 `rpc.Handler(ctx, c, rpc.WalletClient().Method)` 解决
5. **上下文自动传播**：TraceID、UserID 等通过 TTHeader 全链路穿透，不需要参数传递
6. **用 common 的工具**：不造轮子，雪花ID、分页、错误返回、日志都有现成的
7. **遵守建表规范**：审计四字段 + 软删除 + 毫秒时间戳 + utf8mb4_bin
8. **错误码有体系**：6022xxx，按模块分100的块
