# 二、架构模式与调用链路

## 2.1 整体架构

```
客户端(App/H5/Web)
    │
    ▼
Hertz HTTP网关 ─── gate-back(B端) 或 gate-font(C端)
    │
    │  Kitex RPC（TTHeader元数据传播）
    ▼
Kitex微服务 ─── ser-xxx
    │
    ├── TiDB（GORM）
    ├── Redis（缓存/会话）
    ├── ETCD（配置/注册/开关）
    ├── Kafka/NATS（异步消息）
    └── S3/ES（存储/搜索）
```

## 2.2 HTTP请求完整生命周期（以B端认证POST为例）

```
客户端请求 POST /admin/api/item/page
    │
    ├── [1] IGlTracer.Start()
    │       生成UUID → 设为X-Trace-Id → 写入metainfo（全链路透传）
    │       设置CORS头
    │
    ├── [2] HertzAccLogStr()
    │       记录请求日志：trace_id + method + path
    │
    ├── [3] SetHeaderInfo() 中间件
    │       提取Header：ClientIP, Language, DeviceID, DeviceType
    │       Token：先查X-User-Token头 → 后查session_token Cookie
    │       全部写入metainfo.WithPersistentValue（跨RPC透传）
    │
    ├── [4] WhiteCheck() 中间件（B端独有）
    │       调用 rpc.IpClient().OnWhiteList(ctx) → ser-ip
    │       ETCD开关检查 → 开关关闭则放行
    │       不在白名单 → 拒绝（IpNotInWhiteTable）
    │
    ├── [5] BackAuthMiddleware()（B端独有）
    │       Redis GET {token} → 反序列化 BUserTokenPayload{ID, UserName}
    │       写入metainfo：UserID, UserName
    │       调用 rpc.BUserClient().CheckUrlAuth(ctx, fullPath) → RBAC鉴权
    │
    ├── [6] Handler处理
    │       rpc.Handler(ctx, c, rpc.ItemClient().QueryItemPage)
    │       内部：reflect初始化REQ → BindAndValidate → RPC调用 → ret.HttpOne统一返回
    │
    └── HTTP 200 返回 {"traceId":"xxx", "code":0, "msg":"", "data":{...}}
```

## 2.3 B端与C端网关差异

| 维度 | gate-back（B端） | gate-font（C端） |
|------|-----------------|-----------------|
| 路由前缀 | /admin/open, /admin/api | /open, /api |
| IP安全策略 | WhiteCheck（白名单，默认拒绝） | BlackCheck（黑名单，默认放行） |
| 认证中间件 | BackAuthMiddleware | FontAuthMiddleware |
| Redis Token键 | `{token}`（无前缀） | `user:token:{token}` |
| Token载荷 | BUserTokenPayload{ID, UserName} | CUserTokenPayload{ID, UID, IP, Time, Device, NickName} |
| URL权限校验 | CheckUrlAuth（RBAC菜单权限） | 无 |
| MaxBody | 20MB | 默认 |
| 路由数 | ~145（14模块） | ~69（8模块） |

## 2.4 8个上下文传播Header

```
X-Trace-Id         链路追踪ID（UUID）
X-Client-IP        客户端真实IP
X-User-Token       用户Token
X-User-ID          用户ID（认证后写入）
X-User-Name        用户名（认证后写入）
X-User-Language    语言偏好
X-User-Device-ID   设备ID
X-User-Device-Type 设备类型
```

传播机制：metainfo.WithPersistentValue() → Kitex transmeta.ClientTTHeaderHandler序列化到RPC元数据 → 下游transmeta.ServerTTHeaderHandler反序列化。任何RPC服务中通过 `tracer.GetUserId(ctx)` 等函数读取。

# 三、关键代码模式（有据可依）

## 3.1 main.go 启动模式

依据：`ser-item/main.go`

```go
func main() {
    lg.InitKLog()                           // 初始化日志
    etcd.InitEtcd()                         // 初始化ETCD

    port := flag.String("port",             // 端口：命令行优先，ETCD兜底
        etcd.DirectGet(ctx, namesp.EtcdItemPort), "端口号")
    flag.Parse()

    // 创建业务Service实例
    itemService := ser.NewItemService()
    itemPublishService := ser.NewItemPublishService()
    s3Service := ser.NewS3Service()

    // 创建Kitex服务器
    srv := itemservice.NewServer(
        &ItemServiceImpl{                   // handler.go中定义的实现
            itemService:        itemService,
            itemPublishService: itemPublishService,
            s3Service:          s3Service,
        },
        rpc.InitRpcServerParams(            // 公共服务参数
            namesp.EtcdItemService,         // ETCD服务名
            utils.S2Int(ctx, *port),        // 端口号
        )...,
    )
    srv.Run()
}
```

启动顺序：`InitKLog → InitEtcd → 获取端口 → 创建Service → NewServer → Run`

## 3.2 handler.go 薄委托模式

依据：`ser-item/handler.go`

```go
type ItemServiceImpl struct {
    itemService        *ser.ItemService
    itemPublishService *ser.ItemPublishService
    s3Service          *ser.S3Service
}

func (s *ItemServiceImpl) QueryItemPage(ctx context.Context, req *ser_item.ItemQueryReq) (resp *ser_item.ItemQueryResp, err error) {
    return s.itemService.QueryItemPage(ctx, req)    // 一行委托

    return s.itemService.CreateItem(ctx, req)       // 一行委托

// ... 所有方法都是一行委托到internal/ser/
```

handler.go只做一件事：把Kitex接口方法委托给`internal/ser/`中的业务Service。不写任何业务逻辑。

## 3.3 cfg/mysql.go 数据库初始化模式

依据：`ser-item/internal/cfg/mysql.go`

```go
var (
    db   *gorm.DB
    once sync.Once
)

func InitMySQL() *gorm.DB {
    once.Do(func() {
        ctx := context.Background()
        opt := &mysql.InitOption{
            Addr:     etcd.DirectGet(ctx, env.TidbAddr) + ":" + etcd.DirectGet(ctx, env.TidbPort),
            Username: etcd.DirectGet(ctx, env.TidbUsername),
            Password: etcd.DirectGet(ctx, env.TidbPassword),
            Database: etcd.DirectGet(ctx, namesp.EtcdItemDB),  // 数据库名从ETCD取
        }
        mysql.Init(opt)
        db = mysql.DB
    })
    return db
}
```

全部连接参数从ETCD读取，`sync.Once`保证只初始化一次。`mysql.Init()`内部注册了`UserPlugin`，自动填充`create_by/update_by/create_at/update_at`审计字段。

## 3.6 网关Handler模式

依据：`gate-back/biz/handler/item/item.go`

```go
func QueryItemPage(ctx context.Context, c *app.RequestContext) {
    rpc.Handler(ctx, c, rpc.ItemClient().QueryItemPage)   // 一行搞定
}

func CreateItem(ctx context.Context, c *app.RequestContext) {
    rpc.Handler(ctx, c, rpc.ItemClient().CreateItem)       // 一行搞定
}
```

网关Handler全部是一行代理，调用`rpc.Handler`泛型桥接函数。内部自动完成：`reflect初始化请求 → BindAndValidate → RPC调用 → ret.HttpOne统一返回`。

三种变体（依据`common/rpc/handler.go`）：
- `rpc.Handler[REQ,RESP]` — 标准（有请求体）
- `rpc.HandlerNoParam[RESP]` — 无请求体
- `rpc.HandlerWithCSV[REQ,RESP]` — CSV文件下载

少数Handler有自定义逻辑（如Login需要额外SetCookie），此时手动调用`c.BindAndValidate` + RPC + `ret.HttpOne`。

## 4.2 IDL命名规范

依据：现有20个服务IDL的命名模式

```thrift
namespace go ser_wallet                                     // 下划线风格

service WalletService {                                     // PascalCase + Service后缀
    wallet.CreateDepositResp CreateDeposit(1: wallet.CreateDepositReq req)   // PascalCase动词+资源
    bool UpdateCurrency(1: currency.UpdateCurrencyReq req)                   // 布尔返回用bool
}
```

| 规范项 | 标准 | 依据 |
|--------|------|------|
| namespace | `namespace go ser_{name}` | 全部20个IDL一致 |
| Service命名 | `{Domain}Service`（PascalCase） | AppService, ItemService, BUserService |
| 方法命名 | `{动词}{资源}`（PascalCase） | Create/Update/Delete/Query/Get/Page/Toggle/List |
| 请求结构 | `{Action}{Resource}Req` | ItemQueryReq, ItemCreateReq |
| 响应结构 | `{Action}{Resource}Resp` | ItemQueryResp, ItemCreateResp |
| 字段标注 | required/optional | required为必填，optional为可选 |
| 时间字段 | `i64` | 毫秒时间戳 |
| ID字段 | `i64` | Snowflake int64 |

# 七、统一响应与错误处理

## 7.1 响应结构

依据：`common/pkg/ret/project_ret.go`

```go
type R struct {
    TraceId string         `json:"traceId"`
    Code    errs.ErrorType `json:"code"`     // 0=成功，非零=错误码
    Msg     string         `json:"msg"`      // 成功为空，失败为翻译后的错误信息
    Data    interface{}    `json:"data"`     // 业务数据
}
```

所有HTTP响应都返回状态码200，业务错误通过Code字段区分。错误消息自动通过i18n服务翻译。

## 7.2 错误传播链路

```
微服务业务层发现错误
    → ret.BizErr(ctx, code, msg)                    // 包装为Kitex BizStatusError
    → Kitex RPC传播到网关
    → 网关 ret.HttpOne() 检测到err
    → ret.HttpErr() 提取错误码
    → 调用 I18nClient().LangTransOne(ctx, code)     // i18n翻译
    → 返回 {code: 6010003, msg: "道具不存在"}
```

# 十、已知问题与注意事项

代码分析中发现的问题，开发ser-wallet时需注意避免：

1. **RPC Server日志时序Bug**（`common/rpc/init.go:49-64`）：日志打印在`next()`调用之前，cost永远≈0，err永远nil。这是已有bug，不需要ser-wallet修复，但知道有这个问题。

2. **软删除方式不统一**：有的表用时间戳DeleteAt，有的用枚举，有的物理删除。ser-wallet应统一使用时间戳方式（delete_at=0存活，非零=删除时间戳）。

3. **ETCD前缀不统一**：有服务用`/slg/`，有的用`/slg/conf/`。ser-wallet应遵守主流模式：服务配置用`/slg/serv/wallet/`，共享配置直接用env包中的常量。

4. **B/C端Token Redis Key格式不同**：B端直接用token字符串做key，C端用`user:token:{token}`前缀。开发时注意区分。

5. **审计日志（ser-blog）**：所有有写操作的服务都应调用`rpc.BLogClient().AddActLog()`记录审计日志（oneway异步调用）。
