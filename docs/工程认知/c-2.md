#### 2.1.1 ret/ — 统一响应封装

> 依据：`common/pkg/ret/project_ret.go`

**响应结构体**：
```go
type R struct {
    TraceId string         `json:"traceId"`  // 链路追踪ID
    Code    errs.ErrorType `json:"code"`     // 0=成功, 1=严重异常, 其他=业务错误码
    Msg     string         `json:"msg"`      // 成功时为空
    Data    interface{}    `json:"data"`     // 业务数据
}
```

**HTTP 侧函数**（网关出口使用）：
| 函数 | 用途 | 调用场景 |
|------|------|---------|
| `HttpOk(ctx, data)` | 成功响应 | code=0 |
| `HttpErr(ctx, err, i18nClient)` | 错误响应（自动 i18n 翻译） | 从 BizStatusError 提取 code → 调 LangTransOne 翻译 → 翻译失败兜底用 BizMessage |
| `HttpOne(ctx, data, i18nClient, err)` | 懒式封装（有err走错误，无err走成功） | Handler 中最常用 |

**RPC 侧函数**（服务内部使用）：
| 函数 | 用途 | 调用场景 |
|------|------|---------|
| `BizErr(ctx, code, msg...)` | 构造 Kitex BizStatusError | 业务校验失败时 return |
| `BizErrExt(ctx, code, extra, msg...)` | 带附加信息的业务错误 | 需传递额外信息时 |
| `RpcOne[T](ctx, result, err, code)` | 泛型懒式错误包装 | 调用下游时的错误转换 |
| `GetBizErrCode(err, defaultCode)` | 提取错误中的业务码 | 错误码透传场景 |

**关键机制**：
- 所有 HTTP 响应统一返回 200 状态码，业务异常通过 JSON body 的 code 字段区分
- `BizErr` 中禁止使用 0 值作为错误码（框架不支持，代码中有显式检查）
- `BizErr` 的 msg 参数在启用 i18n 后计划被 code 推导的多语言内容替代，所以 msg 可以写得随意
- 错误翻译回退链：目标语言翻译 → 空字符串时兜底用 BizMessage → 都失败返回 "translate err"

#### 2.1.3 tracer/ — 链路追踪

> 依据：`common/pkg/tracer/`

**8 个上下文传递头**：
```
X-Trace-Id        — UUID 每请求唯一（网关 IGlTracer.Start 生成）
X-Client-IP       — 客户端真实IP
X-User-Token      — 认证令牌
X-User-ID         — 用户ID（认证中间件解析后注入）
X-User-Name       — 用户名（认证中间件解析后注入）
X-User-Language   — 语言偏好
X-User-Device-ID  — 设备ID
X-User-Device-Type — 设备类型
```

**取值函数**：
- `GetTraceID(ctx) string`
- `GetUserId(ctx) int64`
- `GetUserToken(ctx) string`
- `GetClientIp(ctx) string`
- `GetUserLang(ctx) string`
- `GetUserName(ctx) string`
- `GetDeviceID(ctx) string`
- `GetDeviceType(ctx) string`

**传递机制**：通过 `metainfo.WithPersistentValue()` 写入 context → Kitex `transmeta.ClientTTHeaderHandler` 自动序列化到 RPC 元数据 → 下游 `transmeta.ServerTTHeaderHandler` 反序列化回 context

**新模块中的使用**：直接通过上述函数从 ctx 获取，无需额外配置，TTHeader 在 `InitRpcServerParams` 和 `InitRpcClientParams` 中已统一配置。

#### 2.1.4 middleware/ — 中间件

> 依据：`common/pkg/middleware/`

| 中间件 | 作用 | 用于 | RPC 调用 |
|--------|------|------|----------|
| `SetHeaderInfo()` | 提取请求头（IP/语言/设备/Token）写入 metainfo | B+C 端 | 无 |

**Token 载荷差异**：

#### 2.1.5 etcd/ — 配置中心

> 依据：`common/pkg/etcd/`

**核心使用方式**：
```go
// 启动时一次性加载 + 监听
etcd.LoadAndWatch("/slg/")  // 加载前缀下所有配置到本地缓存

// 读取（零网络，从本地原子缓存读取）
val := etcd.Get(ctx, "/slg/conf/redis/addr")

// 带默认值读取
val := etcd.GetDef(ctx, "/slg/conf/xxx", "default")

// 变更回调

// 直读（绕过缓存，每次网络请求）
val := etcd.DirectGet(ctx, key)
etcd.DirectPut(ctx, key, val)
```

**实现细节**：原子 Copy-On-Write 缓存（`atomic.Value`），PUT/DELETE 事件自动处理，热更新无需重启。

```go
func Handler[REQ, RESP](ctx, c, rpcMethod) {
    reflect初始化REQ（处理指针/非指针类型）
    → c.BindAndValidate(req)
    → rpcMethod(ctx, req)

// 2. 无参数版：跳过绑定
func HandlerNoParam[RESP](ctx, c, rpcMethod)

func HandlerWithCSV[REQ, RESP](ctx, c, rpcMethod)
```

### 4.1 gen_kitex.sh — IDL → RPC 桩代码

> 依据：`/Users/mac/gitlab/common/script/gen_kitex.sh`（71行 Bash 脚本）

**执行方式**：在项目根目录执行 `bash common/script/gen_kitex.sh`

**完整流程**：

```
1. 定位路径
   SCRIPT_DIR → common/script/
   WORKSPACE_DIR → 项目根目录（/Users/mac/gitlab/）
   IDL_BASE_DIR → common/idl/

2. 检查 kitex 命令是否安装

3. cd 到 common/pkg/ 目录

4. 遍历 common/idl/ 下每个子目录
   for service_name in $(ls "$IDL_BASE_DIR")
       │
       ├── 查找 *service.thrift 文件
       │
       ├── 【保护机制】如果 ser-xxx/ 已存在 handler.go + main.go：
       │   cp handler.go → common/pkg/handler.go（临时保护）
       │   cp main.go → common/pkg/main.go（临时保护）
       │
       ├── 执行核心命令：
       │   kitex -module common/pkg \
       │         -I $IDL_BASE_DIR/$service_name \
       │         -service $service_name \
       │         -gen-path ./kitex_gen/ \
       │         $rel_thrift_path
       │
       ├── 【恢复机制】如果之前有保护：
       │   cp common/pkg/handler.go → ser-xxx/handler.go（恢复）
       │   cp common/pkg/main.go → ser-xxx/main.go（恢复）
       │
       └── 清理副产物：
           rm bootstrap.sh kitex_info.yaml build.sh handler.go main.go scripts/
```

**生成目标**：`common/pkg/kitex_gen/ser_xxx/` 下的 RPC 桩代码

**关键要点**：
- 脚本会扫描**所有**服务的 IDL，不是只生成某一个
- handler.go 和 main.go 有保护机制：已存在的不会被覆盖
- 全新服务首次执行时，Kitex 会在 `ser-xxx/` 下生成 handler.go + main.go 骨架
- 后续 IDL 变更重新执行时，只更新 `kitex_gen/` 下的桩代码，不覆盖业务代码

```go
func (s *BannerService) CreateBanner(ctx context.Context, req *ser_app.CreateBannerReq) (*ser_app.CreateBannerResp, error) {
    // 1. 参数校验
    if req.Name == "" {
        return nil, ret.BizErr(ctx, errs.ReqParamErr, "名称不能为空")
    }

    // 2. 业务规则校验（查重等）
    exists, err := bannerr.Exists(ctx, query.Q.Banner.Name.Eq(req.Name))
    if err != nil { return nil, ret.BizErr(ctx, errs.BannerBaseError) }
    if exists { return nil, ret.BizErr(ctx, errs.BannerNameAlreadyExist) }

    // 3. 构造实体
    entity := &model.Banner{...}

    // 4. 持久化（使用自动生成的 repo 方法）
    err = bannerr.Create(ctx, entity)
    if err != nil { return nil, ret.BizErr(ctx, errs.BannerBaseError) }

    // 5. 缓存失效（如有）
    s.cache.InvalidateAll()

    // 6. 审计日志（oneway 异步，不影响主链路）
    rpc.BLogClient().AddActLog(ctx, &ser_blog.AddActLogReq{
        Module: "banner",
        Action: "create",
        Detail: utils.MarshalStr(entity),
    })

    // 7. 返回
    return &ser_app.CreateBannerResp{Id: entity.ID}, nil
}
```

**关键规范**：
- 错误返回使用 `ret.BizErr(ctx, code, msg...)` 构造 Kitex BizStatusError
- 审计日志使用 `rpc.BLogClient().AddActLog()`（oneway，不等待返回）
- 缓存失效在写操作后立即执行
- `tracer.GetUserId(ctx)` 获取当前操作人 ID
- `tracer.GetTraceID(ctx)` 获取链路追踪 ID

## 七、端到端请求生命周期

### 7.1 B端已认证请求完整链路

以 `POST /admin/api/xxx/entity/create` 为例：

```
[1] 客户端 → Hertz HTTP Server

[2] IGlTracer.Start()
    生成 UUID → 设置 X-Trace-Id → 写入 metainfo → 设置 CORS

[3] HertzAccLogStr()
    记录请求日志（trace_id + method + path）

[4] SetHeaderInfo()
    提取 IP/Language/DeviceID/DeviceType → 读 Token（Header优先，Cookie兜底）
    全部写入 metainfo.WithPersistentValue

[5] WhiteCheck()
    rpc.IpClient().OnWhiteList(ctx) → ser-ip 查 sys_white_list
    ├─ 开关关闭 → 直接放行
    ├─ 不在白名单 → Abort(IpNotInWhiteTable)
    └─ RPC 错误 → Abort（fail-closed 安全策略）

[6] BackAuthMiddleware()
    6a. 获取 token → 为空则 Abort(TokenNotExist)
    6b. Redis GET {token} → 反序列化 BUserTokenPayload
    6c. UserID == 0 → Abort(TokenAnalysisFailed)
    6d. 注入 X-User-ID, X-User-Name 到 metainfo
    6e. rpc.BUserClient().CheckUrlAuth(ctx, fullPath)
        ├─ super_admin("1"或"2") → 直接通过
        └─ 普通角色 → 查 sys_role_menu + sys_menu 匹配 URL

[7] 路由 Handler（一行代码）
    rpc.Handler(ctx, c, rpc.XxxClient().Method)

    Handler[REQ,RESP] 内部：
    7a. reflect 初始化 REQ
    7b. c.BindAndValidate(req)
    7c. rpcMethod(ctx, req) — Kitex RPC 调用
        └─ TTHeader 自动携带 TraceId/UserID/UserName 等
    7d. ret.HttpOne(ctx, resp, I18nClient(), err)
        ├─ 成功 → {traceId, code:0, data:resp}
        └─ 失败 → I18n 翻译错误码 → {traceId, code:N, msg:"多语言消息"}

[8] ser-xxx 处理
    handler.go → internal/ser/ → internal/gorm_gen/repo/ 或 internal/rep/
    → GORM → TiDB（UserPlugin 自动审计 CreateBy/UpdateBy）
    → rpc.BLogClient().AddActLog() [oneway 审计]

[9] HTTP 200 响应（所有业务异常也返回 200，靠 code 字段区分）
```

## 八、关键设计规范汇总

### 8.1 数据模型规范

| 规范项 | 标准 | 依据 |
|-------|------|------|
| ID 生成 | 雪花算法 int64 | utils/snowflake.go |
| 时间字段 | int64 毫秒时间戳 | model 中用 `*int64` |
| 审计字段 | `create_by`, `update_by`, `create_at`, `update_at` | 所有 model 结构体 |
| 软删除 | `delete_at=0` 未删除，非零=删除时间戳 | 查询条件 `.Where(t.DeleteAt.Eq(0))` |
| 删除人 | `delete_by` 记录操作人 | model 字段 |
| 可空字段 | GORM Gen 配置 `FieldNullable=true`，生成指针类型 | generate.go:69 |

### 8.2 接口规范

| 规范项 | 标准 |
|-------|------|
| HTTP 方法 | 全部 POST |
| 路由格式 | `/{module}/{entity}/{action}` |
| 统一响应 | `{traceId, code, msg, data}` |
| 分页入参 | `pageNo` + `pageSize` |
| 分页出参 | `{data, total, pageTotal, pageNo, pageSize}` |
| 错误码 | `6{NNN}{MMM}` 格式 |
| 链路追踪 | X-Trace-Id 贯穿全链路 |
| 审计日志 | 所有写操作 oneway 上报 ser-blog |

### 8.3 初始化模式

| 场景 | 模式 | 依据 |
|------|------|------|
| RPC 客户端 | sync.Once 惰性单例 | rpc_client.go |
| MySQL 连接 | sync.Once 惰性单例 | cfg/mysql.go |
| Redis 连接 | sync.Once 惰性单例 | cfg/redis.go |
| ETCD 配置 | LoadAndWatch 一次性加载 + 持续监听 | main.go |

### 8.4 缓存模式

| 模式 | 说明 | 示例 |
|------|------|------|
| Cache-Aside | 先查缓存，miss 查 DB 写缓存 | Banner 列表 |
| 空结果缓存 | 防穿透 | Banner 空列表也缓存 |
| 写后失效 | 写操作后主动清除相关缓存 | Banner 清 18 种组合 |
| TTL 兜底 | 即使不主动失效，也有 TTL 自动过期 | 5分钟 ~ 24小时 |

## 十、必须遵守的约束（不能重复造轮子）

### 10.1 必须复用的公共能力

| 能力 | 提供方 | 禁止自行实现 |

| 日志 | common/pkg/lg | 禁止直接使用 fmt/log |
| Redis | common/pkg/rds | 禁止引入其他 Redis 库 |

| MySQL | common/pkg/db/mysql | 禁止直接使用 sql 包 |
| S3 | common/pkg/mys3 | 禁止引入其他 S3 SDK |

| Handler 桥接 | rpc.Handler/HandlerNoParam/HandlerWithCSV | 禁止自行绑定参数+返回 |

### 10.2 禁止事项

- 禁止在 `gorm_gen/` 目录下手动修改文件（会被重新生成覆盖）
- 禁止在 handler.go 中编写业务逻辑（保持薄委托）

- 禁止在 ETCD 监听路径中使用硬编码字符串（使用 env 和 namesp 常量）
- 禁止在 HTTP 响应中使用非 200 状态码表示业务错误
- 禁止跳过审计日志（所有写操作必须上报 ser-blog）

## 十一、已发现的代码问题与注意点

> 以下问题基于代码实证，供开发时规避参考

### 11.1 确认的代码问题

1. **RPC 服务端日志时序问题**（`common/rpc/init.go:49-64`）
   - 现状：先记录 `time.Since(start)` 和 `err`，然后才调用 `next(ctx, req, resp)`
   - 后果：cost 永远 ≈ 0，err 永远 = nil
   - 应在 `next()` 之后记录

2. **gate-back UpdateUser 使用了错误的 Handler**
   - 使用了 `HandlerWithCSV` 而非 `Handler`
   - 更新用户操作应返回 JSON，不是 CSV

3. **ser-ip DeleteBlackList 审计日志标签错误**
   - 删除黑名单时，审计日志写的是 "whitelist" 而非 "blacklist"

4. **sys_user_role 表字段注释互换**
   - user_id 注释写"角色ID"，role_id 注释写"菜单ID"
   - 实际代码使用正确，仅注释错误

### 11.2 设计差异注意点

5. **B端/C端 Token Redis Key 格式不一致**
   - B端：裸 token → C端：`user:token:` + token
   - 了解此差异以避免混淆

6. **rpc_client.go 中编号注释不一致**
   - item 注释写 009 但 namesp.go 中 009 是 exp
   - content_center 在 rpc_client.go 标 016 但 namesp.go 标 013
   - 以 namesp.go 为准

7. **菜单级联递归无深度限制**
   - getAllChildIds/getAllParentIds 理论上无限递归
   - 依赖数据正确性（无环）
