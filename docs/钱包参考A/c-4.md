# 第一部分：全景认知

## 1. 项目概览与钱包定位

### 1.1 项目背景

参考工程是一个 **真人视讯（Live Casino）平台**，采用 B2B 模式，为多个商户（Merchant）提供真人荷官游戏服务。

```
核心业务链：
商户接入 → 玩家注册/登录 → 入金(上分) → 下注 → 结算(派彩) → 出金(下分)
```

### 1.2 钱包在系统中的角色

```
┌─────────────────────────────────────────────────────────┐
│                    系统服务全景                            │
├─────────────┬───────────────┬───────────────────────────┤
│  接入层      │  业务层         │  游戏层                    │
│             │               │                           │
│ api-service │ wallet-service│ baccarat-service(百家乐)   │
│ gateway     │ user-service  │ roulette-service(轮盘)     │
│             │ auth-service  │ dragon-tiger-service(龙虎)  │
│             │ merchant-svc  │ black-jack-service(21点)   │
│             │ order-service │ ...共16个游戏服务            │
│             │ report-svc    │ common-service(游戏公共)    │
│             │               │ websocket-service(WS推送)  │
├─────────────┴───────────────┴───────────────────────────┤
│  管理层：admin-service, admin-merchant-line-service       │
│  基础设施：gateway-server, file-service, system-business  │
│  任务层：p9-job-executor                                 │
└─────────────────────────────────────────────────────────┘
```

**wallet-service 的定位**：
- 全系统唯一的资金操作入口
- 管理玩家账户余额（可用+冻结）
- 记录所有账变流水
- 被 7+ 个服务通过 Feign 调用

### 1.3 业务场景覆盖

| 场景 | 资金方向 | 触发方 | 钱包操作 |
|------|---------|--------|---------|
| 商户上分（充值） | 商户→平台→玩家 | api-service | addCoin(income) |
| 商户下分（提现） | 玩家→平台→商户 | api-service | addCoin(expenses) |
| 下注 | 玩家→平台 | game-service(MQ) | addCoin(expenses) 或 freeze |
| 结算派彩 | 平台→玩家 | game-service(MQ) | addCoin(income) |
| 结算回滚 | 修正 | game-service(MQ) | addCoin(调整) |
| 重算/二次结算 | 修正 | game-service(MQ) | addCoin(调整) |
| 打赏 | 玩家→荷官 | websocket-service | addCoin(expenses) |
| 闪电投注 | 快速下注 | game-service(MQ) | addCoin(expenses) |

## 2. 技术栈全景

| 维度 | 参考工程 | 我们的 ser-wallet |
|------|---------|------------------|
| **语言** | Java 17 | Go 1.21+ |
| **框架** | Spring Boot 2.7.4 + Spring Cloud | CloudWeGo Hertz(HTTP) + Kitex(RPC) |
| **ORM** | MyBatis Plus 3.5.3 | GORM + gorm_gen |
| **数据库** | MySQL(兼容TiDB) | TiDB |
| **缓存** | Redis + Redisson 3.17.7 | Redis |
| **消息队列** | RabbitMQ (AMQP) | Kafka/NATS(预留) |
| **服务发现** | K8s DNS (Feign硬编码URL) | ETCD |
| **服务通信** | OpenFeign (HTTP/JSON) | Kitex (Thrift RPC) |
| **分布式锁** | Redisson RLock | Redis SETNX / Redlock |
| **API文档** | Springdoc OpenAPI 1.6.8 | Thrift IDL |
| **序列化** | FastJSON2 + Jackson | Thrift Binary/JSON |
| **部署** | Docker (JIB) + K8s | Docker + K8s |
| **监控** | Actuator + Prometheus | Prometheus + OpenTelemetry |
| **日志** | Logback + Fluentd | zap/zerolog |

### 关键技术差异评价

**Feign vs Kitex RPC**：参考工程用 HTTP Feign 做服务间通信，延迟相对高（HTTP序列化+反序列化），但简单直观。我们用 Kitex Thrift RPC，二进制协议性能更好，接口契约更严格（IDL编译时校验）。

**MyBatis Plus vs GORM**：两者都是成熟ORM，MyBatis Plus 的 XML Mapper 灵活但冗余（参考工程 CoinRecordMapper.xml 284行，三个查询90%代码重复）。GORM + gorm_gen 类型安全但自定义 SQL 相对麻烦。

**RabbitMQ vs Kafka**：参考工程用 RabbitMQ 做点对点消费（一条消息只被一个消费者处理），适合下注扣款这种精确一次的场景。我们预留 Kafka，更适合事件流和多消费者场景。

# 第二部分：架构设计

## 4. 分层架构

### 4.1 实际分层

```
┌─────────────────────────────────────────┐
│           Controller 层                  │
│  WalletController + CoinRecordController│
│  职责：参数校验、分布式锁获取/释放、调用Service │
├─────────────────────────────────────────┤
│           Service 层                     │
│  CoinRecordServiceImpl (核心)            │
│  UserCoinServiceImpl (查询)              │
│  CoinRecordDataServiceImpl (数据查询)     │
│  职责：业务逻辑、事务管理、MQ发送          │
├─────────────────────────────────────────┤
│           Repository 层                  │
│  UserCoinRepository + CoinRecordRepository│
│  职责：MyBatis Plus CRUD + 自定义XML SQL  │
├─────────────────────────────────────────┤
│           数据层                          │
│  MySQL/TiDB + Redis + RabbitMQ          │
└─────────────────────────────────────────┘
```

### 4.2 分层特点

**优点**：
- 标准三层架构，结构清晰
- Service 接口与实现分离，契约明确

**问题**：
- **没有 Domain 层**：所有业务逻辑都在 Service 层，没有独立的领域模型（如 BalanceEngine）
- **Controller 承担过多职责**：分布式锁的获取/释放放在 Controller 层（WalletController 的 addCoin/freezeCoin 等方法都有 20+ 行的锁模板代码）
- **DTO 混用**：CoinRecordDTO 既是数据库实体也作为内部传输对象
- **没有 BO/DO 分离**：直接用 VO 做 Service 入参，DTO 做数据库操作，缺少中间的业务对象

### 4.3 与我们的 ser-wallet 对比

| 层次 | 参考工程 | 我们的设计 |
|------|---------|-----------|
| 接入层 | Controller（HTTP REST） | Handler（Hertz HTTP + Kitex RPC） |
| 业务层 | Service | Service + BalanceEngine(领域核心) |
| 数据层 | Repository(MyBatis Plus) | Repository(GORM + gorm_gen) |
| 领域模型 | **缺失** | BalanceEngine 统一引擎 |
| 锁管理 | Controller 层 | BalanceEngine 内部 |

## 6. 双钱包模式——转账钱包与免转钱包

### 6.1 两种模式定义

```java
public enum WalletTypeEnum {
    TRANSFER(1, "转账钱包"),       // 余额存在本地 user_coin 表
    FREE_TRANSFER(2, "免转钱包"),  // 余额存在商户（三方）系统
}
```

### 6.2 转账钱包（Transfer Wallet）

```
玩家余额 = 本地 user_coin.amount 字段
资金操作 = 直接修改本地数据库

商户上分 → api-service → wallet-service.addCoin → UPDATE user_coin SET amount = amount + X
游戏下注 → game-service → RabbitMQ → wallet-service.addCoin → UPDATE user_coin SET amount = amount - X
游戏结算 → game-service → RabbitMQ → wallet-service.addCoin → UPDATE user_coin SET amount = amount + X
商户下分 → api-service → wallet-service.addCoin → UPDATE user_coin SET amount = amount - X
```

### 6.3 免转钱包（Seamless/Free-Transfer Wallet）

```
玩家余额 = 商户自己维护（不在本地数据库）
资金操作 = HTTP 调用商户的加扣款接口

游戏下注 → game-service → RabbitMQ → wallet-service.addCoin
         → 调用 apiResource.amountTransfer → 商户HTTP接口 → 返回结果
         → 成功则记录本地账变(coinFrom/coinTo=null)，失败则标记失败
```

### 6.4 双钱包模式评价

**设计意图**：适应不同商户的技术能力——技术强的商户用免转模式（余额自管），技术弱的用转账模式（平台代管）。

**优点**：
- 灵活适配多种商户接入模式
- 统一入口（addCoin），内部按 walletType 分流

**缺陷**：
- **if-else 分流导致代码膨胀**：CoinRecordServiceImpl.addCoin() 方法 170+ 行，转账钱包 vs 免转钱包的逻辑混在一起
- **免转钱包本地无余额**：getWallet 对免转钱包直接返回 amount=0，实际余额需要另外调商户接口获取——查询一致性差
- **coinFrom/coinTo 为 null**：免转钱包的账变记录不记录变更前后金额，审计不完整

### 6.5 对我们的启示

**我们不需要双钱包模式**。我们的场景是 B2C 平台自营，不存在"商户代管余额"的需求。但这个设计提醒我们：
- 保持核心操作入口统一是正确的（类似我们的 BalanceEngine）
- 不同业务场景的分流应该在 Service 层用 Strategy 模式，而不是在一个大方法里 if-else

# 第三部分：数据模型

## 7. 表结构完整定义

### 7.1 user_coin 表（用户钱包账户）

```sql
-- 源文件：UserCoinDTO.java + UserCoinMapper.xml
CREATE TABLE `user_coin` (
  `id`            BIGINT AUTO_INCREMENT PRIMARY KEY,  -- 主键
  `user_id`       VARCHAR(255),       -- 玩家ID
  `user_name`     VARCHAR(255),       -- 玩家名称
  `amount`        DECIMAL(18,8),      -- 可用余额
  `freeze_amount` DECIMAL(18,8),      -- 冻结金额
  `user_state`    INT,                -- 玩家游戏状态
  `game_type`     INT,                -- 正在进行的游戏类型
  `wallet_type`   INT,                -- 1=转账钱包, 2=免转钱包
  `currency`      VARCHAR(50),        -- 币种代码
  `remark`        TEXT,               -- 备注
  `creator`       BIGINT,             -- 创建人ID
  `updater`       BIGINT,             -- 更新人ID
  `created_time`  DATETIME,           -- 创建时间
  `updated_time`  DATETIME            -- 更新时间
);
```

**SQL 操作定义**（UserCoinMapper.xml）：

```sql
-- 减余额（带条件保护）
UPDATE user_coin b
SET b.amount = b.amount - #{amount}, b.updated_time = #{updateTime}
WHERE b.id = #{id}
  -- 结算回滚(6)和取消(5)允许负数，其余不允许
  AND (b.amount - #{amount}) >= 0   -- 仅非回滚/取消时

-- 加余额
UPDATE user_coin b

UPDATE user_coin b

WHERE b.id = #{id}
```

|------|------|---------|

### 8.2 存在的问题

#### 问题1：user_coin 表缺少关键索引和约束

```
【缺失】没有 user_id 唯一索引
       → 理论上同一用户可能被插入多条 user_coin 记录
       → selectOne(new QueryWrapper(userCoinDTO)) 如果有多条会报错

【缺失】没有 version 乐观锁字段
       → 完全依赖 Redisson 分布式锁，如果锁失效无兜底

【缺失】amount 字段没有 NOT NULL DEFAULT 0 约束
       → 代码层面有保护，但数据库层面缺乏约束
```

#### 问题2：coin_record 表缺少唯一约束

```
【缺失】order_no 没有唯一索引
       → 幂等性完全靠应用层 findCoinResult 查询
       → 并发场景下理论上可能插入重复账变

【缺失】没有 (user_id, order_no) 组合唯一索引
       → 数据库层面无幂等保护

【冗余】大量商户/代理冗余字段
       → merchant_id, merchant_name, merchant_no, agent_id, agent_name, agent_no, path
       → 这些信息应该通过 JOIN 获取，而不是冗余存储
       → 好处：查询不需要 JOIN，性能好
       → 坏处：如果商户/代理信息变更，历史记录不一致
```

#### 问题3：金额精度选择

```
DECIMAL(18,8) → 最大 9999999999.99999999

【评价】
- 8位小数：对法币过多（一般2位），对加密货币可能不够（比特币聪=10^8刚好，但链上gas可能更精确）
- 统一用一种精度：简单但不精确
- 与我们的设计差异：我们用 BIGINT 存最小单位（法币×100，加密×10^8），运算零精度损失
```

#### 问题4：user_id 用 VARCHAR(255)

```
【问题】用字符串存 userId，索引效率低于 BIGINT
       查询条件里大量 LIKE '%userId%'（模糊搜索），全表扫描
       与我们用 int64 userId 的设计不同
```

#### 问题5：没有状态字段

```
【缺失】coin_record 没有 status 字段
       → 无法区分"处理中"、"已完成"、"已失败"
       → 对于免转钱包的异步操作，无法追踪状态

【缺失】user_coin 没有 status/disabled 字段
       → 无法冻结/禁用整个钱包账户
       → 只有 user_state（玩家游戏状态），不是钱包状态
```

### 12.2 关键代码片段解读

**余额充足性校验**：

```java
// CoinRecordServiceImpl.java:366-384
private void insufficientBalance(CoinRecordVO coinRecordVO, UserCoinDTO userCoin) {
    // 仅对 转出/下注/打赏 校验余额
    // 结算回滚/取消/重算 不校验（允许负数）
    if (TRANSFER_OUT.equals(coinType) || BET.equals(coinType) || SEND_REWARD.equals(coinType)) {
        BigDecimal balance = userCoinAmount.subtract(freezeAmount);  // 可用 = 总额 - 冻结
        if (balance.compareTo(amount) < 0) {
            throw new MayaDefaultException(ErrorCode.AMOUNT_NOT_ENOUGH);
        }
    }
}
```

**减余额SQL的防负保护**：

### 12.3 流程评价

**优点**：
- 双重余额保护：Java 层 `insufficientBalance()` + SQL 层 `(amount - #{amount}) >= 0`
- 首次入金自动创建钱包记录（lazy initialization）
- 结算回滚/取消允许负数——业务上合理（重算可能减到负数后再加回）

**缺陷**：

```
【严重】查询余额和扣款不是原子操作
  step1: SELECT user_coin WHERE userId = ?  → 得到 amount = 100
  step2: 判断 100 >= 50                     → 通过
  step3: UPDATE user_coin SET amount = amount - 50

  在 step1 和 step3 之间，如果另一个线程已经扣了 60，
  那 step3 的 SQL 会因为 (40 - 50) < 0 而影响行数=0。
  代码检查了 i==0 会抛异常——但这是"悲观退出"而不是"提前拦截"。

  分布式锁（Redisson）保护了这个间隙，但如果锁失效呢？
  → 没有 SELECT FOR UPDATE 行锁兜底
  → 没有 version 乐观锁兜底
  → 安全性完全依赖 Redisson

【严重】事务内调用外部服务
  addCoin() 标注了 @Transactional，但内部调用了：
  - commonService.getUserInfoByUserId() → Feign HTTP 到 user-service
  - 免转钱包还调用 apiResource.amountTransfer() → HTTP 到三方

  如果 Feign 调用超时或失败：
  - 事务可能长时间不提交，数据库连接被占用
  - 如果 amountTransfer 成功但后续 coin_record.insert 失败 → 资金不一致

【中等】没有幂等校验
  addCoin 入口没有根据 orderNo 做幂等检查
  如果同一个 orderNo 被调用两次，会插入两条 coin_record 并扣/加两次余额
  → 幂等依赖调用方（如 api-service 的 doTransferAmount 先调 findCoinResult 检查）
  → 但 MQ 消费者（AddCoinListener）没有做幂等检查！

【轻微】BeanUtils.copyProperties 过多
  VO → DTO 大量使用 Spring 的 BeanUtils.copyProperties
  → 性能损耗（反射）
  → 字段名不匹配时静默丢失数据
  → 不如用 MapStruct 或手动 set
```

### 13.1 冻结流程 (freezeCoin)

### 13.3 回滚冻结 (rollbackFreezeBalance)

```

2. 【不校验冻结金额是否足够！】
3. UPDATE user_coin SET freeze_amount = freeze_amount - X
4. 【不产生账变记录】

资金变化：
  可用余额：不变
  冻结金额：-X
  总金额：不变
```

### 13.4 三阶段评价

**优点**：
- 冻结/确认/回滚三步走是标准的资金冻结模式
- 确认扣除产生账变，冻结和回滚不产生——减少流水噪音

**缺陷**：

```
【严重】冻结不减可用余额的设计问题

  user_coin 表的 amount 字段含义不明确：
  - getWallet 返回时：amount = amount - freezeAmount（可用余额 = 总额 - 冻结）
  - addCoin 的余额校验时：balance = amount - freezeAmount
  - subBalance SQL：直接减 amount

  实际上 amount 存的是"总余额"而不是"可用余额"！
  这导致 amount 字段的语义混乱：
  - 冻结后 amount 不变（因为总额确实没变）
  - 确认扣除时 amount 减少（总额减少）
  - 但 subBalance 在非冻结场景直接减 amount

  → 可用余额 = amount - freezeAmount 这个计算散落在代码各处，没有封装
  → 如果某处忘了减 freezeAmount，就会出现超额扣款

【严重】rollbackFreezeBalance 不校验冻结余额
  如果回滚金额 > 冻结金额，freeze_amount 会变负数
  → 数据库没有 CHECK 约束
  → 代码没有校验

【中等】freezeCoin 和 rollbackFreezeBalance 不产生账变
  冻结/解冻作为资金状态变更，没有留痕
  → 审计时无法追溯冻结历史
  → 我们的设计应该每个操作都记录流水

【中等】三个操作没有用 @Transactional
  freezeCoin 有 @Transactional
  subFreezeBalance 没有 @Transactional（但有两步操作：update + insert）
  rollbackFreezeBalance 没有 @Transactional
  → subFreezeBalance 的 update 和 insert 不在同一个事务里！
```

## 15. 批量操作与MQ消费

### 15.1 MQ消费者——AddCoinListener

```
源文件：AddCoinListener.java（148行，4个监听方法）

Queue 1: ADD_COIN_QUEUE
  → NoticeAddCoinMessage(): 单笔 addCoin + sendUserBalance
  → 加 Redisson 锁

Queue 2: ADD_COIN_ORDER_QUEUE
  → noticeDeductCoinOrderMessage(): 批量 deductCoinBatch
  → 不加锁！直接调用 deductCoinBatch
  → 失败时回滚订单状态：orderResource.updateOrdersToInitialStatus()

Queue 3: ADD_COIN_NOT_NOTICE_BALANCE_QUEUE
  → addCoinNotNoticeBalance(): 批量 deductCoinBatch
  → 不加锁，不发余额通知
  → 失败时回滚订单状态

Queue 4: ADD_COIN_RECORD_AND_NOTICE_BALANCE_QUEUE
  → addCoinRecordAndNoticeBalanceQueue(): 批量 changeCoinAndNoticeBalance
  → 不加锁
```

### 15.2 MQ消费评价

```
【严重】Queue 2/3/4 的消费者不加分布式锁
  同一个用户的消息可能被并发消费
  deductCoinBatch → 循环调用 deductCoin → 只记账变不改余额
  → 这个 deductCoin 和 addCoin 不同！它只 INSERT coin_record，不 UPDATE user_coin
  → 所以并发安全？——只对免转钱包安全，转账钱包如果也走这里就有问题

【严重】没有消息幂等校验
  MQ 消费者收到消息后直接执行，没有检查 orderNo 是否已处理
  如果 RabbitMQ 重投消息（消费者异常未 ACK），同一笔账变会重复执行
  → 转账钱包会重复加/减余额！

【中等】异常处理粗糙
  catch(Exception e) 后只记日志，没有重试/补偿/死信队列
  对于金额操作，消息丢失意味着资金差异

【轻微】消息体用 JSON 字符串
  直接 JSON.parseObject(jsonStr, CoinRecordVO.class)
  没有消息版本/类型标识，未来扩展困难
```

## 16. 并发控制——Redisson 分布式锁

### 16.1 锁实现

```java
// 锁 key: "SERVER_NAME:addCoin:{userId}"
RLock rLock = redisUtil.getLock(RedisLockKey.getAddCoinLockKey(userId));
try {
    if (rLock.tryLock(20000, 30000L, TimeUnit.MILLISECONDS)) {
        // 20秒等待获取锁，30秒自动释放
        // 执行业务逻辑...
    } else {
        log.info("锁超时：{}", orderNo);
    }
} catch (Exception e) {
    log.error(e.getMessage(), e);
} finally {
    if (Objects.nonNull(rLock) && rLock.isLocked() && rLock.isHeldByCurrentThread()) {
        rLock.unlock();
    }
}
```

### 16.2 锁使用统计

| 场景 | 锁类型 | 锁key | 等待/过期 |
|------|--------|-------|----------|
| addCoin | 可重入锁 | addCoin:{userId} | 20s/30s |
| addCoinList | 可重入锁 | addCoin:{firstUserId} | 20s/30s |
| freezeCoin | 可重入锁 | addCoin:{userId} | 20s/30s |
| subFreezeBalance | 可重入锁 | addCoin:{userId} | 20s/30s |
| rollbackFreezeBalance | 可重入锁 | addCoin:{userId} | 20s/30s |
| transferAmount(api-svc) | 公平锁 | transferAmount:{merchantNo}:{userId} | 配置化 |
| MQ NoticeAddCoinMessage | 可重入锁 | addCoin:{userId} | 20s/30s |

### 16.3 锁策略评价

**优点**：
- 同一用户的所有资金操作共享一把锁（addCoin:userId），避免并发冲突
- api-service 的 transferAmount 用公平锁（getFairLock），保证 FIFO 顺序
- 锁释放前检查 isHeldByCurrentThread()，避免释放别人的锁

**缺陷**：

```
【严重】单层防线
  只有 Redisson 分布式锁一层防护
  如果 Redis 主从切换导致锁丢失 → 并发操作可能同时执行
  → 没有 DB 行锁（SELECT FOR UPDATE）兜底
  → 没有乐观锁（version 字段）兜底
  → 对比我们的设计：三层防线（DB行锁 + 乐观锁 + 幂等key）

【中等】锁粒度统一
  所有操作共享 addCoin:{userId} 这一把锁
  查询余额也不加锁（可能读到脏数据）
  冻结和账变用同一把锁（冻结不影响账变的场景也被阻塞）

【中等】30秒自动释放
  如果业务逻辑执行超过30秒（如免转钱包调三方超时），锁提前释放
  → 后续操作在无锁状态下执行
  → Redisson 有看门狗（watchdog）可以自动续期，但需要配置

【轻微】MQ消费者的锁策略不统一
  Queue 1 加锁，Queue 2/3/4 不加锁
  → 同一个用户，Queue 1 的消息和 Queue 2 的消息可能并发执行
```

## 17. 事务管理

### 17.1 @Transactional 使用情况

| 方法 | @Transactional | 事务内操作 | 事务内外部调用 |
|------|---------------|-----------|--------------|
| addCoin | ✅ rollbackFor=Exception | UPDATE + INSERT | Feign×2(user, api) |
| addCoinBatch | ✅ 默认 | 循环调用 addCoin | 间接调外部 |
| deductCoin | ✅ rollbackFor=Throwable | INSERT | Feign×1(user) |
| deductCoinBatch | ✅ 默认 | 循环调用 deductCoin | 间接调外部 |
| changeCoinAndNoticeBalance | ✅ 默认 | 循环调用 deductCoin | 间接调外部 |
| freezeCoin | ✅ 默认 | SELECT + UPDATE | 无 |
| subFreezeBalance | **❌ 无事务** | SELECT + UPDATE + INSERT | Feign×1(user) |
| rollbackFreezeBalance | **❌ 无事务** | SELECT + UPDATE | 无 |

### 17.2 事务评价

```
【严重】subFreezeBalance 没有 @Transactional
  该方法执行：
  1. SELECT user_coin → 获取钱包
  2. UPDATE user_coin → 减余额+减冻结
  3. createOrder() → INSERT coin_record（还要 Feign 获取用户信息）

  如果步骤2成功但步骤3失败：
  → 余额已扣但没有账变记录
  → 资金对不上！

【严重】事务内的 Feign 调用
  addCoin 在 @Transactional 内调用 commonService.getUserInfoByUserId()
  → 该方法内部调用 Feign HTTP 到 user-service
  → 如果 user-service 响应慢，事务长时间占用数据库连接
  → 如果 user-service 超时后事务回滚，但此前的 Feign 副作用无法回滚

【中等】@Transactional 的 rollbackFor 不统一
  addCoin: rollbackFor = Exception.class
  deductCoin: rollbackFor = Throwable.class
  addCoinBatch: 默认（仅 RuntimeException 和 Error）
  → 如果抛出 checked Exception，addCoinBatch 不会回滚
```

## 18. 幂等性实现

### 18.1 当前幂等机制

```
层次1：api-service 的 transferAmount
  → 调用 findCoinResult(merchantId, merchantNo, orderNo)
  → 如果已有记录，抛出 ORDER_SUBMITTED
  → 这是应用层幂等，有并发窗口

层次2：数据库层
  → coin_record 表的 order_no 字段 **没有唯一索引**
  → 无数据库层幂等保护

层次3：MQ 消费者
  → 没有任何幂等检查
  → 消息重投会重复执行
```

### 18.2 幂等评价

```
【严重】幂等性严重不足

1. 数据库层面：
   - coin_record.order_no 没有唯一约束
   - 即使两个并发请求通过了应用层 findCoinResult 检查，都会插入成功
   → 我们的设计：uk_order_wallet_tx 唯一索引，数据库层面绝对防重

2. 应用层面：
   - 只有 api-service 的 transferAmount 有幂等检查
   - wallet-service 自身的 addCoin 入口没有幂等检查
   - 任何直接调用 addCoin 的服务（game-service、websocket-service）都可能重复操作
   → 我们的设计：BalanceEngine 内部统一幂等检查

3. MQ层面：
   - 完全没有幂等机制
   - 消息重投 = 重复扣款/加款
   → 我们的设计：MQ 消费前根据 orderNo 查询是否已处理
```

## 19. 安全机制——MD5签名验证

### 19.1 签名验证流程

```
仅在 api-service（三方接口）层使用

其中 apiSign 是每个商户的密钥，存在 merchant_detail 表
```

## 20. RabbitMQ 四队列架构

### 20.1 队列定义

```
Queue 1: ADD_COIN_QUEUE
  用途：单笔加减额度（下注扣款、结算加款等）
  特点：加分布式锁，发送余额变更通知
  生产者：游戏服务

Queue 2: ADD_COIN_ORDER_QUEUE
  用途：批量扣减注单（一次下注多个注单）
  特点：不加锁，失败回滚订单状态
  生产者：游戏服务

Queue 3: ADD_COIN_NOT_NOTICE_BALANCE_QUEUE
  用途：批量扣减，不通知余额变更
  特点：不加锁，不发通知，失败回滚订单
  生产者：游戏服务（不需要实时更新前端余额的场景）

Queue 4: ADD_COIN_RECORD_AND_NOTICE_BALANCE_QUEUE
  用途：账变记录 + 通知余额变更
  特点：不加锁
  生产者：游戏服务
```

### 20.2 队列设计评价

```
【问题】四个队列功能高度重叠

本质上是两个维度的组合：
  维度1：加锁 vs 不加锁
  维度2：通知 vs 不通知

可以用一个队列 + 消息头标记代替四个队列：
  message.header["lock"] = true/false
  message.header["notify"] = true/false

【借鉴】异步解耦游戏与钱包
  游戏服务通过 MQ 异步调用钱包——这个模式值得借鉴
  好处：
  - 游戏服务不等待钱包操作完成
  - 钱包服务可以按自己的节奏消费
  - 游戏服务不直接依赖钱包服务（解耦）

  但也有问题：
  - 异步意味着下注可能延迟失败
  - 需要补偿机制（失败后回滚订单）
  → 我们的设计中 RPC 同步调用更适合强一致性场景（下注必须实时知道成功/失败）
```

## 27. 优势分析——值得借鉴的设计

### 27.1 可借鉴——设计理念层

| # | 设计 | 描述 | 借鉴价值 | 我们如何采纳 |
|---|------|------|---------|-------------|
| 1 | 统一入口 | 所有资金操作走 addCoin 一个入口 | ★★★★★ | 我们的 BalanceEngine 更好，按操作类型分方法 |
| 2 | 冻结三阶段 | freeze → confirm/rollback 标准模式 | ★★★★★ | 直接采用，但我们每步都记流水 |
| 3 | 双重余额保护 | Java校验 + SQL WHERE条件 | ★★★★ | 采用，我们还多一层乐观锁 |
| 4 | MQ异步解耦 | 游戏服务通过MQ异步调钱包 | ★★★★ | 参考，但我们的投注场景用同步RPC |
| 5 | 余额变更通知 | 操作后MQ通知WebSocket推送前端 | ★★★★ | 采用，但推送时带上最新余额 |
| 6 | 可用余额=总额-冻结 | 分开存储更清晰 | ★★★★ | 采用，但要封装计算逻辑 |

### 27.2 可借鉴——实现细节层

| # | 细节 | 描述 | 借鉴价值 |

## 28. 缺陷分析——需要规避的问题

### 28.1 P0 严重缺陷（绝对要规避）

| # | 缺陷 | 风险 | 我们的规避方案 |
|---|------|------|--------------|
| 1 | **无数据库层幂等保护** | 并发重复扣款/加款 | uk_order_wallet_tx 唯一索引 |
| 2 | **无DB行锁** | Redisson失效时并发穿透 | SELECT FOR UPDATE + version乐观锁 |
| 3 | **MQ消费无幂等** | 消息重投重复执行 | 消费前查orderNo是否已处理 |
| 4 | **subFreezeBalance无事务** | 扣款成功但流水丢失 | 所有资金操作统一@Transactional |
| 5 | **事务内Feign调用** | 长事务+不可回滚的外部副作用 | 用户信息通过ctx传入，不在事务内调外部 |
| 6 | **rollback不校验冻结余额** | freeze_amount可能变负 | 每步操作都校验前置条件 |

### 28.2 P1 中等缺陷（需要改进）

| # | 缺陷 | 风险 | 我们的改进 |
|---|------|------|-----------|
| 7 | 冻结/回滚不记账变 | 审计不完整 | 所有资金状态变更都记流水 |
| 8 | Feign Client定义分散 | 接口变更漏改 | Thrift IDL统一定义 |
| 9 | 用户信息缓存未写入 | 每次操作都调user-service | 配置信息缓存+事务外获取 |
| 10 | 免转钱包余额=0 | 查询不准确 | 不适用（我们没有免转模式） |
| 11 | @Transactional rollbackFor 不统一 | 部分异常不回滚 | 统一配置 rollbackFor=Exception.class |
| 12 | 四个MQ队列功能重叠 | 维护成本高 | 统一队列+消息头标记 |

### 28.3 P2 轻微缺陷（可以优化）

| # | 缺陷 | 说明 | 我们的优化 |
|---|------|------|-----------|
| 13 | userId VARCHAR(255) | 索引效率低 | int64 userId |
| 14 | DECIMAL(18,8) 一刀切 | 法币8位小数过多 | BIGINT最小单位，零精度损失 |
| 15 | BeanUtils.copyProperties | 反射性能差 | Go结构体直接赋值 |
| 16 | XML Mapper大量重复SQL | getCoinRecordPage/List/Count 90%相同 | GORM查询构造器 |
| 17 | 全部POST不RESTful | 查询也用POST | HTTP GET for queries, RPC for mutations |
| 18 | 返回类型不统一 | ResponseVO vs ResponseEntity | 统一响应结构 |

## 30. 借鉴策略总结——取舍清单

### 30.1 直接采用（✅ Take）

| # | 设计 | 理由 |
|---|------|------|
| 1 | amount + freeze_amount 双字段 | 行业标准，清晰直观 |
| 2 | coin_from/coin_to 快照 | 审计必需，变更前后金额存流水 |
| 3 | balance_type 收支方向 | 查询友好，分类清晰 |
| 4 | 冻结三阶段模式 | 已在我们设计中采用 |
| 5 | SQL层防负保护 | WHERE (amount - X) >= 0 兜底 |
| 6 | 首次入金自动建钱包 | lazy init 减少空钱包 |
| 7 | 结算回滚允许负数 | 业务场景需要 |
| 8 | 余额变更通知MQ广播 | 实时推送前端 |
| 9 | 操作/查询Controller分离 | 职责清晰 |

### 30.2 改进采用（🔧 Improve）

| # | 参考设计 | 我们的改进 | 理由 |
|---|---------|-----------|------|
| 1 | Redisson单层锁 | DB行锁+乐观锁+幂等key三层 | 安全性倍增 |
| 2 | 应用层幂等查询 | DB唯一索引绝对防重 | 消除并发窗口 |
| 3 | addCoin统一入口 | BalanceEngine按操作类型分方法 | 语义更清晰 |
| 4 | coin_record宽表 | wallet_transaction精简+关联 | 减少冗余 |
| 5 | 冻结/回滚不记流水 | 所有操作都记流水 | 审计完整 |
| 6 | 事务内Feign调用 | 事务仅包DB操作 | 避免长事务 |
| 7 | 4个MQ队列 | 统一队列+消息标记 | 简化维护 |
| 8 | 商户信息1天缓存 | 精确TTL+主动失效 | 更及时 |
| 9 | md5Sign不为空判断来源 | ctx中token/userId判断 | 更可靠 |

### 30.3 完全不采用（❌ Reject）

| # | 设计 | 理由 |
|---|------|------|
| 1 | 双钱包模式(转账/免转) | 我们是B2C自营，不需要 |
| 2 | DECIMAL(18,8) 金额存储 | 我们用BIGINT最小单位，零精度损失 |
| 3 | userId VARCHAR | 我们用 int64，索引效率更高 |
| 4 | Feign HTTP 服务通信 | 我们用 Kitex RPC，性能更好 |
| 5 | MyBatis XML 重复SQL | GORM查询构造器更DRY |
| 6 | 全POST不RESTful | 查询用GET/RPC用Thrift |
| 7 | Controller层管理锁 | Domain层内部管理 |
| 8 | BeanUtils反射复制 | Go结构体直接赋值 |
| 9 | 无version乐观锁 | 必须有 |
| 10 | 无order_no唯一索引 | 必须有 |

### 30.4 核心认知总结

```
参考工程的钱包模块是一个"能用但有隐患"的实现：
- 核心业务逻辑正确（加减冻结解冻回滚，流程对）
- 数据模型基本合理（amount/freezeAmount/coinFrom/coinTo，结构对）
- 但安全防线薄弱（单层锁、无DB幂等、MQ无防重）
- 代码组织不够优雅（大方法、事务内外部调用、DTO混用）

对我们的启示：
1. 业务模型可以参考（冻结三阶段、账变类型枚举、余额双字段）
2. 安全机制必须升级（三层并发防线、DB级幂等、事务边界严格）
3. 架构设计需要改进（Domain层、BalanceEngine统一入口）
4. 细节实现各取所长（lazy建钱包、SQL防负、余额通知）

一句话总结：
参考工程的"做什么"值得借鉴，"怎么做"需要全面升级。
```
