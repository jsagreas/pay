### 1.3 与我方项目的核心差异

| 维度 | 参考工程(P9) | 我方工程(ser-wallet) |
|------|-------------|---------------------|
| **语言** | Java 17 | Go 1.21+ |
| **框架** | Spring Boot + Spring Cloud | CloudWeGo Hertz + Kitex |
| **ORM** | MyBatis Plus | GORM + Gen |
| **RPC** | OpenFeign (HTTP) | Kitex (Thrift RPC) |
| **消息队列** | RabbitMQ | NATS JetStream(规划) |
| **分布式锁** | Redisson RLock | Redis SetNX(规划) |

### 3.2 架构特征

| 特征 | 描述 | 评判 |
|------|------|------|
| **三层架构** | Controller→Service→Repository | ✅ 清晰标准 |
| **锁在Controller层** | 分布式锁在Controller获取/释放 | ⚠️ 有争议（见第6章分析） |
| **双路钱包** | TRANSFER(本地)和FREE_TRANSFER(三方) | ✅ 思路可借鉴 |
| **异步消息** | 投注扣款通过RabbitMQ异步处理 | ✅ 值得借鉴 |
| **Service层偏厚** | 核心Service 522行，混合了验证、RPC、DB操作 | ⚠️ 可以更细分 |

### 3.3 取舍评判

**✅ 精华**：
- 三层架构分层清晰，Controller不含业务逻辑
- 双路钱包设计（TRANSFER/FREE_TRANSFER）思路灵活，用walletType枚举区分处理路径
- 余额查询返回available(=amount-freezeAmount)而非raw amount

**❌ 糟粕**：
- Service层过于臃肿（522行单文件），缺少domain层抽象
- 没有Repository层的自定义手写Rep（所有DB操作在Service中编排）
- Controller直接管理锁的获取/释放，增加Controller复杂度

**⚖️ 我方调整**：
- 我方采用handler→service→domain→repository四层，比P9多一层domain
- 我方锁逻辑放在repository事务层（FOR UPDATE），不在handler层

## 第 4 章 数据模型与表结构

### 4.1 核心表结构

#### 4.1.1 user_coin（用户钱包表）

```sql
CREATE TABLE user_coin (
  id          BIGINT AUTO_INCREMENT PRIMARY KEY,
  user_id     VARCHAR(?) NOT NULL,      -- 玩家ID（注意：是String类型！）
  user_name   VARCHAR(?),               -- 玩家名
  amount      DECIMAL(?,?) NOT NULL,    -- 账户余额（总额）
  freeze_amount DECIMAL(?,?) DEFAULT 0, -- 冻结金额
  user_state  INT,                      -- 玩家状态
  game_type   INT,                      -- 当前游戏类型
  wallet_type INT,                      -- 1=转账钱包, 2=免转钱包
  currency    VARCHAR(?),               -- 币种
  remark      VARCHAR(?),               -- 备注
  created_time DATETIME,
  updated_time DATETIME,
  creator     BIGINT,
  updater     BIGINT
);
```

**特点分析**：

| 字段 | 分析 | 评判 |
|------|------|------|
| `user_id` VARCHAR | 字符串类型userId，与我方int64不同 | ⚖️ 各有优劣，我方用int64更高效 |
| `amount` + `freeze_amount` | 双字段设计：total=amount, available=amount-freeze | ✅ 思路相同，可借鉴 |
| 无`version`字段 | 没有乐观锁保护 | ❌ 缺陷：并发安全隐患 |
| 无`wallet_type`多钱包 | 每个用户只有1条记录（不分中心/奖励/主播） | ❌ 我方需要多钱包 |
| 无唯一约束 | 没有UNIQUE(user_id, currency, wallet_type) | ❌ 缺陷：可能重复创建钱包 |
| `game_type`字段 | 钱包中记录当前游戏类型 | ❌ 不应该：钱包不应关心游戏状态 |
| `currency`字段 | 支持多币种，但每用户只有一种 | ⚠️ 我方需要多币种多钱包 |

### 4.2 与我方表结构对比

| 维度 | P9 | 我方设计 | 差异评判 |
|------|-----|---------|---------|
| 钱包表维度 | user_id单维 | (user_id, currency_code, wallet_type)三维 | 我方更完备 |
| 唯一约束 | 无 | UNIQUE(biz_type, order_no, wallet_type) | 我方有DB级幂等 |
| version字段 | 无 | 有 | 我方双重保护 |
| 流水表 | 可更新(有updater字段) | append-only(只INSERT) | 我方更安全 |
| 冻结相关 | frozen_before/after缺失 | 有frozen_before/after | 我方更完整 |
| 表数量 | 3张(钱包+流水+消息) | 11张(全链路) | 我方业务更复杂 |
| 订单存储 | MongoDB(注单) | TiDB(统一) | 我方更简洁 |

### 4.3 取舍评判

**✅ 精华**：
- amount + freeze_amount 双字段设计，available = amount - freeze_amount
- coin_from/coin_to 余额快照对（等同于balance_before/after）
- 消息追踪表 add_coin_message 保证异步可靠性

**❌ 糟粕**：
- coin_record表**无UNIQUE约束**，无法在DB层防止重复写入 → 幂等性依赖应用层
- user_coin表**无version字段**，并发更新无乐观锁保护
- coin_record表设计为**可更新**（有updater字段），流水应该是不可变的
- 大量冗余字段（merchant_name/agent_name等）增加写入开销
- game_type 字段放在钱包表中是不当耦合

**⚖️ 我方调整**：
- 保持三维钱包(user_id × currency × wallet_type)
- 流水表必须append-only，不设updater字段
- UNIQUE约束作为幂等最终防线
- 不在钱包表中放游戏状态字段

## 第 5 章 核心业务逻辑剖析

### 5.1 addCoin — 核心入账/扣款方法

**位置**: `CoinRecordServiceImpl.java:169-341`

**流程分解**：

```
addCoin(CoinRecordVO) 入口
  │
  ├─ ① 校验 coinType > 0
  │
  ├─ ② 获取用户信息（Feign → user-service，带缓存）
  │
  ├─ ③ 判断钱包类型
  │     │
  │     ├─ TRANSFER(本地钱包):
  │     │   ├─ 查询 user_coin（SelectOne by userId）
  │     │   │   ├─ 存在:
  │     │   │   │   ├─ 校验余额（仅转出/投注/打赏校验）
  │     │   │   │   ├─ if income: addBalance(amount) → coinTo = old + value
  │     │   │   │   ├─ if expense: subBalance(amount) → coinTo = old - value
  │     │   │   │   │   └─ subBalance返回0 → 余额不足异常
  │     │   │   │   └─ 设置 coinFrom/coinTo/coinAmount
  │     │   │   │
  │     │   │   └─ 不存在（首次入账）:
  │     │   │       ├─ 校验不能是支出类型
  │     │   │       ├─ INSERT user_coin (amount=coinValue)
  │     │   │       └─ 设置 coinFrom=0, coinTo=coinValue
  │     │   │
  │     │   └─ INSERT coin_record（流水记录）
  │     │
  │     └─ FREE_TRANSFER(免转钱包):
  │         ├─ 调用三方API: apiResource.amountTransfer()
  │         ├─ 成功: 记录三方流水 + 通知余额变更
  │         ├─ 失败: 记录失败 + return false
  │         └─ INSERT coin_record（无coinFrom/coinTo）
  │
  └─ return true
```

### 5.2 代码级问题标注

**问题1: 读取余额与更新非原子操作（L186-L208）**

```java
// 先查询（此时未加行锁）
UserCoinDTO userCoinDTO1 = userCoinRepository.selectOne(new QueryWrapper(userCoinDTO));
// ... 若干行后才更新
userCoinRepository.addBalance(balanceVO);  // 或 subBalance
```

**问题**：selectOne和addBalance之间存在时间窗口。虽然Controller层有Redis分布式锁，但如果锁获取/释放出现异常，这个窗口就是并发漏洞。

**与我方对比**：我方使用SELECT FOR UPDATE在事务内锁定行，读取和更新是原子的。

---

**问题2: subBalance的SQL防护依赖条件判断（UserCoinMapper.xml:4-12）**

```xml
<update id="subBalance">
    update user_coin b
    set b.amount = b.amount - #{vo.amount},
        b.updated_time = #{vo.updateTime}
    where b.id = #{vo.id}
    <!-- coinType != 6(结算回滚) and coinType != 5(取消) 时才检查 -->
    <if test=" vo.coinType != 6 and vo.coinType != 5 ">
        AND (b.amount - #{vo.amount}) >= 0
    </if>
</update>
```

**亮点**：通过SQL条件`AND (amount - debit) >= 0`在DB层防止透支。
**问题**：coinType=5(取消)和coinType=6(结算回滚)可以扣成负数！这是业务决策（回滚可能导致负数），但缺少注释说明原因。
**评判**：✅ SQL层防透支思路好；⚠️ 允许负数的特殊情况应更明确标注。

---

**问题3: 余额校验不完整（L366-L383）**

```java
private void insufficientBalance(CoinRecordVO coinRecordVO, UserCoinDTO userCoin) {
    // 仅对 转出/投注/打赏 三种类型校验余额
    if (TRANSFER_OUT == coinType || BET == coinType || SEND_REWARD == coinType) {
        BigDecimal balance = userCoinAmount.subtract(freezeAmount);
        if (balance.compareTo(amount) < 0) {
            throw new MayaDefaultException(ErrorCode.AMOUNT_NOT_ENOUGH);
        }
    }
}
```

**问题**：只对3种类型校验，其他支出类型（如结算回滚）跳过校验。如果新增coinType忘记加入校验列表，可能导致透支。

**与我方对比**：我方在repository层统一对所有debit操作做`available >= amount`检查，不分类型。

---

**问题4: 钱包懒创建无并发保护（L212-L226）**

```java
// 钱包不存在时新建
userCoinDTO.setAmount(coinRecordVO.getCoinValue());
userCoinRepository.insert(userCoinDTO);
```

**问题**：如果两个并发请求同时发现钱包不存在，都尝试INSERT，且user_coin表无UNIQUE(user_id)约束，会导致同一用户创建两个钱包记录。

**与我方对比**：我方user_coin表有UNIQUE(user_id, currency_code, wallet_type)约束，即使并发INSERT也只会有一个成功。

---

**问题5: coinTo计算使用Java内存中的旧值（L199,L207）**

```java
// income时
coinTo = userCoinDTO1.getAmount().add(coinRecordVO.getCoinValue());
// expense时
coinTo = userCoinDTO1.getAmount().subtract(coinRecordVO.getCoinValue());
```

**问题**：coinTo的计算基于selectOne时读取的旧余额值。如果此时有另一个并发操作修改了余额（即使有Redis锁，锁失效时也可能发生），coinTo就不等于DB中的实际余额。

**与我方对比**：我方在FOR UPDATE事务内读取余额并计算，保证原子一致。

### 5.3 取舍评判

**✅ 精华**：
- `subBalance`的SQL层防透支（`AND (amount - debit) >= 0`） — 好思路
- 先查后判再操作的三步模式 — 思路清晰
- TRANSFER/FREE_TRANSFER双路分发 — 灵活可扩展
- 新用户首次入账自动创建钱包 — 懒创建合理

**❌ 糟粕**：
- 读取余额与更新不在同一行锁内 — 并发安全漏洞
- coinTo基于内存旧值计算 — 可能与DB实际值不一致
- 钱包懒创建无UNIQUE保护 — 可能重复创建
- 余额校验只针对3种类型 — 不够通用
- Service层522行臃肿 — 缺少domain层拆分

## 第 6 章 并发控制与锁策略

### 6.1 P9的锁方案：Redisson分布式锁

**位置**: `WalletController.java:60-78`

```java
// 获取分布式锁
RLock rLock = redisUtil.getLock(RedisLockKey.getAddCoinLockKey(userId));
try {
    if (rLock.tryLock(20000, 30000L, TimeUnit.MILLISECONDS)) {
        // 临界区：余额操作
        boolean result = coinRecordService.addCoin(coinRecordVO);
        // ...
    } else {
        log.info("锁超时：{}", coinRecordVO.getOrderNo());
    }
} catch (Exception e) {
    log.error(e.getMessage(), e);
} finally {
    if (rLock.isLocked() && rLock.isHeldByCurrentThread()) {
        rLock.unlock();
    }
}
```

**锁参数**：
- 锁Key: `add.coin.lock.key.{userId}` — 按用户ID粒度
- 等待时间: 20秒（tryLock超时）
- 持有时间: 30秒（leaseTime自动释放）
- 锁类型: RLock（可重入锁）

### 6.2 锁策略分析

| 维度 | P9方案 | 分析 |
|------|--------|------|
| 锁粒度 | 用户级(userId) | ✅ 合理，同一用户串行，不同用户不阻塞 |
| 锁位置 | Controller层 | ⚠️ 有争议：锁应该尽量靠近数据操作 |
| 锁实现 | Redisson RLock | ✅ 成熟方案，自动续期 |
| 锁超时 | 30秒 | ⚠️ 偏长，正常余额操作应<1秒 |
| 无DB锁 | 没有FOR UPDATE | ❌ 缺陷：Redis锁失效时无兜底 |
| 无version | 没有乐观锁 | ❌ 缺陷：无第二层保护 |

### 6.3 P9方案的风险

```
风险场景: Redis集群故障导致锁失效

T1: 请求A获取Redis锁成功，开始addCoin
T2: Redis主节点宕机，锁信息丢失
T3: 请求B获取Redis锁成功（因为锁已丢失）
T4: 请求A和请求B同时操作同一用户余额
    → 请求A: selectOne得到amount=1000, 计算1000+500=1500
    → 请求B: selectOne得到amount=1000, 计算1000-800=200
    → 请求A: addBalance → amount=1500
    → 请求B: subBalance → amount=1500-800=700
    → 正确结果应该是1000+500-800=700 ← 碰巧正确

但如果:
    → 请求B先执行: subBalance → amount=200
    → 请求A后执行: addBalance → amount=200+500=700 ← 也正确

真正的问题:
    → 请求A: selectOne得到amount=1000, coinTo计算=1500
    → 请求B: selectOne得到amount=1000, coinTo计算=200
    → 请求B先完成: amount=200, coin_record.coinTo=200 ✓
    → 请求A后完成: amount=700, coin_record.coinTo=1500 ✗ ← coinTo与实际不一致！
    → 流水链断裂！
```

**P9的subBalance/addBalance使用原子SQL**（`amount = amount + X`），所以余额本身不会错。但**coinTo/coinFrom记录会错**，因为它们基于selectOne时的旧值计算。

### 6.4 与我方方案对比

| 维度 | P9 | 我方 |
|------|-----|------|
| 第一层 | Redis分布式锁 | Redis SetNX快检(幂等) |
| 第二层 | 无 | SELECT FOR UPDATE(行锁) |
| 第三层 | 无 | version字段(乐观锁) |
| 第四层 | SQL条件防透支 | SQL条件防透支 |
| 第五层 | 无 | DB UNIQUE约束 |
| coinTo准确性 | 基于内存旧值（可能不一致） | 在事务内原子计算（100%一致） |

### 6.5 取舍评判

**✅ 精华**：
- 按用户ID粒度的分布式锁 — 正确的锁粒度
- tryLock非阻塞等待 — 避免无限等待
- finally块保证释放 — 标准锁使用模式
- isLocked+isHeldByCurrentThread双判断 — 安全释放

**❌ 糟粕**：
- 仅依赖Redis单层锁，无DB层兜底
- 锁在Controller层，与事务层分离 — 锁和事务不在同一边界
- 无version字段作为最后防线
- coinTo基于旧值计算 — 流水可能不一致

**⚖️ 我方调整**：
- 主防线用FOR UPDATE(DB层)，不依赖外部Redis锁
- Redis仅用于幂等快检（L1），不作为并发控制主手段
- version字段作为安全兜底

## 第 7 章 冻结/解冻状态机

### 7.1 P9的冻结模型

P9的冻结主要服务于**投注场景**：
- 用户下注 → freezeCoin(冻结投注额)
- 结算赢了 → rollbackFreezeBalance(回滚冻结) + addCoin(入账赢利)
- 结算输了 → subFreezeBalance(从冻结中扣款)
- 注单取消 → rollbackFreezeBalance(回滚冻结)

### 7.2 三个冻结操作对比

| 操作 | amount变化 | freeze_amount变化 | 产生流水 | 用途 |
|------|-----------|------------------|---------|------|
| `freezeCoin` | 不变 | +增加 | **不产生** | 投注冻结 |
| `subFreezeBalance` | -减少 | -减少 | **产生** | 结算扣款 |
| `rollbackFreezeBalance` | 不变 | -减少 | **不产生** | 取消/回滚 |

### 7.3 问题分析

**问题1: freezeCoin不产生流水**

```java
// freezeCoin方法（L425-L446）
public ErrorCode freezeCoin(CoinFreezeVO coinFreezeVO) {
    // ... 校验 ...
    userCoinRepository.addFreezeBalance(balanceVO);
    return ErrorCode.SUCCESS;
    // 注意：没有INSERT coin_record！
}
```

**问题**：冻结操作不写流水记录，审计链不完整。如果冻结后系统崩溃，无法知道这笔冻结是什么时候、因为什么原因发生的。

**同样，rollbackFreezeBalance也不产生流水**（L476-L493）。

**与我方对比**：我方所有余额变更（包括冻结/解冻）都产生wallet_transaction流水，保证审计完整性。

---

**问题2: freezeCoin校验使用amount而非available**

```java
if (userCoinDTO.getAmount().compareTo(coinFreezeVO.getFreezeAmount()) < 0) {
    return ErrorCode.USER_WALLET_NOT_ENOUGH_BALANCE;
}
```

**问题**：校验`amount >= freezeAmount`，但没有减去已冻结的部分。如果用户已经冻结了500，总额1000，再冻结600应该失败（available=500<600），但这里1000>=600会通过。

**这是一个Bug**。正确应该是：`(amount - freeze_amount) >= new_freeze_amount`

**与我方对比**：我方在FOR UPDATE事务内计算`available = balance - frozen`，校验`available >= freeze_amount`。

---

**问题3: subFreezeBalance无事务注解**

```java
// subFreezeBalance方法（L448-L473）
@Override
public ErrorCode subFreezeBalance(CoinFreezeRecordVO coinFreezeRecordVO) {
    // ... 校验 ...
    userCoinRepository.subFreezeBalance(balanceVO);  // UPDATE
    createOrder(coinFreezeRecordVO, userCoinDTO);     // INSERT
    return ErrorCode.SUCCESS;
}
```

**问题**：方法没有`@Transactional`注解！UPDATE和INSERT不在同一事务中。如果`subFreezeBalance`成功但`createOrder`失败，余额已扣但没有流水记录。

**与我方对比**：我方所有余额操作在同一事务中（balance更新 + 流水INSERT原子完成）。

### 7.4 取舍评判

**✅ 精华**：
- 冻结/扣款/回滚三操作分离 — 职责清晰
- freeze_amount独立字段 — 可用余额动态计算
- 三个独立SQL操作明确（addFreeze/subFreeze/rollbackFreeze）

**❌ 糟粕**：
- freezeCoin不产生流水 — 审计链断裂
- freezeCoin校验用amount而非available — **Bug**
- subFreezeBalance缺少@Transactional — **Bug**
- rollbackFreezeBalance不产生流水 — 审计链断裂

## 第 10 章 幂等性设计（或缺失）

### 10.1 P9的幂等现状

**结论：P9钱包模块基本没有幂等保护。**

| 接口 | 幂等措施 | 风险 |
|------|---------|------|
| addCoin | 无 | 重复调用=重复入账/扣款 |
| addCoinBatch | 无 | 批量中每笔都可能重复 |
| freezeCoin | 无 | 重复冻结=金额多冻 |
| subFreezeBalance | 无 | 重复调用=多次扣款 |
| rollbackFreezeBalance | 无 | 重复回滚=冻结变负数 |
| MQ消费 | 无 | 消息重投=重复处理 |

**唯一的"幂等"出现在API层**（api-service的transferAmount接口）：

```java
// api-service中的CoinRecordController

walletResource.findCoinResult(orderNo)
// 如果已存在 → 返回已有结果
// 如果不存在 → 执行转账
```

但这是应用层检查，有并发窗口。

### 10.2 问题严重性

对于**投注/结算**场景（P9的主场景），由于游戏引擎控制频率，实际重复的概率低。但对于**充值/提现**场景（我方的核心场景），回调可能重复、RPC可能超时重试，幂等是**必须的**。

### 10.3 取舍评判

**❌ 这是P9最大的缺陷**：
- coin_record表无UNIQUE约束
- 无Redis幂等检查
- MQ消费无去重
- 整个钱包服务几乎是"调一次执行一次"

**⚖️ 我方必须做到**：
- L1: Redis幂等快检
- L2: DB UNIQUE(biz_type, order_no, wallet_type)兜底
- 所有MQ消费前先检查幂等

- THIRD_ADD_AND_SUB_FAIL — 三方转账失败
- PARAM_ERROR (9990) — 参数错误

### 12.2 问题分析

**问题1: freezeCoin返回ErrorCode而非抛异常**

```java
public ErrorCode freezeCoin(CoinFreezeVO coinFreezeVO) {
    if (userCoinDTO == null) {
        return ErrorCode.USER_WALLET_NOT_EXIST;  // 返回而非抛异常
    }
    // ...
}
```

而addCoin中：
```java
throw new MayaDefaultException(ErrorCode.AMOUNT_NOT_ENOUGH);  // 抛异常
```

**问题**：同一Service层，有的方法返回ErrorCode，有的抛异常 — 风格不统一。

### 12.3 取舍评判

**✅ 精华**：
- 统一ResponseVO格式
- 全局异常处理器
- 枚举化错误码

**❌ 糟粕**：
- 错误处理风格不统一（有时返回ErrorCode，有时throw）
- catch中吞掉异常（MQ监听器）
- 错误码范围不够细分（钱包和其他服务混用）

### 15.1 精华（值得借鉴）

|------|------|--------|------|---------|
| **B01** | 无DB层幂等 | coin_record无UNIQUE | 重复操作无法在DB层拦截 | UNIQUE(biz_type,order_no,wallet_type) |

## 第 16 章 对我方ser-wallet的借鉴策略

### 16.1 直接采纳的设计

| 采纳项 | 来源 | 我方实现方式 |
|--------|------|-------------|
| available = balance - frozen | G04 | wallet表设计完全一致 |
| SQL原子加减 | G02 | GORM的`gorm.Expr("balance + ?", amount)` |
| SQL防透支条件 | G03 | UPDATE WHERE `(balance - frozen - amount) >= 0` |
| 懒创建钱包 | G11 | 首次入账时INSERT，UNIQUE约束保证幂等 |
| 余额变更通知 | G07 | NATS JetStream推送WebSocket |
| 冗余字段查询 | G13 | wallet_transaction存完整业务信息 |
| 统一响应格式 | G14 | ret.R{TraceId,Code,Msg,Data} |

### 16.2 改进后采纳的设计

| 改进项 | P9原始做法 | 我方改进做法 |
|--------|-----------|-------------|
| 分布式锁 | Redisson锁在Controller(单层) | Redis SetNX仅做幂等快检，主防线在DB FOR UPDATE |
| 冻结操作 | 不写流水 | 所有操作写流水 |
| 余额校验 | 仅对3种类型 | 所有debit操作统一校验 |
| 消息消费 | 无幂等 | 消费前三元组幂等检查 |
| 错误处理 | 混合return+throw | 统一通过error返回 |

### 16.3 完全不采纳的设计

| 不采纳项 | 原因 |
|---------|------|
| 无UNIQUE约束 | 这是DB层幂等的最终防线，不能没有 |
| 无version字段 | 乐观锁是并发安全的最后兜底 |
| RPC在事务内 | 事务内禁止网络调用 |
| 流水表可更新 | 流水必须是不可变的 |
| MySQL+MongoDB双存储 | 我方TiDB统一存储 |
| Feign HTTP RPC | 我方Kitex Thrift RPC性能更优 |

### 16.4 额外总结：P9缺失而我方必须有的

| 必须项 | P9没有 | 我方设计中已规划 |
|--------|--------|----------------|
| 多币种多钱包 | 单币种单钱包 | (user_id, currency_code, wallet_type)三维 |

| 汇率管理 | 无 | 三源聚合+中位数+偏离告警 |

| KYC校验 | 无 | 提现前RPC查KYC等级 |
| 限额管理 | 无 | 充值/提现限额+日累计 |
| 死锁预防 | 不涉及（单钱包） | 固定锁序(currency ASC, wallet_type ASC) |

> **本文档总结**：
>
> P9参考工程是一个生产运行的真人博弈平台钱包模块，技术栈为Java/Spring Boot/MyBatis Plus/Redis/RabbitMQ。
>
> **核心价值**：其SQL原子加减、分布式锁粒度、available计算模型、异步消息解耦、双路钱包设计等思路值得借鉴。
>
> **核心缺陷**：无DB层幂等保护、无version字段、读写不在同一行锁内、冻结校验Bug、流水非不可变等问题需要规避。
>
> **核心差异**：P9是"游戏内循环钱包"（只有投注/结算），我方是"全链路资金钱包"（充值/提现/兑换/投注...），复杂度差异巨大。P9的架构适合简单场景，但不足以支撑我方的完整资金链路需求。
>
> **一句话**：P9给了我们一个"最简版钱包"的参考基线，我们在此基础上需要补齐幂等、多层锁、状态机、多币种多钱包、充值提现链路等全部能力。
