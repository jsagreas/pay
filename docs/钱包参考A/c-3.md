## 第二章 数据模型

### 2.1 核心表：user_coin（用户钱包）

```sql
CREATE TABLE user_coin (
    id            BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id       VARCHAR(64) NOT NULL,       -- 用户 ID（字符串类型！）
    user_name     VARCHAR(64),                -- 用户名
    amount        DECIMAL NOT NULL,           -- 可用余额
    freeze_amount DECIMAL DEFAULT 0,          -- 冻结金额
    user_state    INT,                        -- 用户游戏状态
    game_type     INT,                        -- 当前游戏类型
    wallet_type   INT NOT NULL,               -- 钱包类型：1=转账 2=免转
    currency      VARCHAR(16),                -- 币种
    remark        VARCHAR(256),               -- 备注
    created_time  DATETIME,
    updated_time  DATETIME,
    creator       INT,
    updater       INT
);
```

**关键设计决策**：
- `user_id` 是 **String 类型**（非 Long），适配三方商户系统的异构 ID
- `amount` 和 `freeze_amount` 两个字段管理余额，总余额 = amount（含冻结），可用 = amount - freeze_amount
- **每用户一行记录**（非每币种一行），单币种设计
- **无 version 字段**（无乐观锁）
- **无 delete_at 字段**（无软删除）

### 2.2 核心表：coin_record（流水记录）

```sql
CREATE TABLE coin_record (
    id            BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id       VARCHAR(64) NOT NULL,
    user_name     VARCHAR(64),
    nick_name     VARCHAR(64),
    user_type     INT,                        -- 0=正式 1=试玩
    user_belong   INT,                        -- 用户归属
    merchant_id   BIGINT,                     -- 商户 ID
    merchant_no   VARCHAR(64),                -- 商户编号
    merchant_name VARCHAR(128),               -- 商户名称
    agent_id      BIGINT,                     -- 代理 ID
    agent_no      VARCHAR(64),                -- 代理编号
    agent_name    VARCHAR(128),               -- 代理名称
    path          VARCHAR(512),               -- 上级层级链
    order_no      VARCHAR(64),                -- 订单号
    coin_type     INT NOT NULL,               -- 账变类型（1-9 枚举）
    coin_detail   INT,                        -- 账变明细
    coin_value    DECIMAL NOT NULL,           -- 变动金额（绝对值）
    coin_from     DECIMAL,                    -- 变动前余额
    coin_to       DECIMAL,                    -- 变动后余额
    coin_amount   DECIMAL,                    -- 当前余额
    balance_type  INT,                        -- 1=收入 2=支出
    remark        VARCHAR(512),
    created_time  DATETIME,
    updated_time  DATETIME,
    creator       BIGINT,
    updater       BIGINT
);
```

**关键设计决策**：
- **宽表设计**：流水表冗余了大量用户/商户/代理信息（user_name, merchant_name, agent_name, path 等），
  查询时无需 JOIN，但写入成本高
- `coin_from` / `coin_to` 记录了变动前后的余额快照 —— 这是审计的核心字段
- `coin_amount` 和 `coin_to` 存在**语义重叠**（实际代码中赋值逻辑有时不一致）
- **order_no 没有 UNIQUE 索引**（这是一个重要缺陷，后文详述）

### 2.3 表结构对比评估

| 维度 | 参考工程（user_coin） | 我们（t_wallet_account） | 评估 |
|------|---------------------|------------------------|------|
| 用户 ID 类型 | String | int64 | 我们更优（内部系统统一 int64） |
| 多币种支持 | 单币种/单行 | 多币种/多行（user_id+wallet_type+currency 联合） | 我们更优 |
| 子钱包 | 无 | 5 种子钱包类型 | 我们更丰富 |
| 乐观锁 | 无 version 字段 | version 字段 CAS | 我们更安全 |
| 冻结模型 | amount 含冻结 | balance + frozen_amount 独立 | 语义不同，各有道理 |
| 软删除 | 无 | delete_at 毫秒戳 | 我们有 |
| DECIMAL 精度 | 未指定 (p,s) | DECIMAL(30,8) UNSIGNED | 我们更严格 |

| 维度 | 参考工程（coin_record） | 我们（t_wallet_flow） | 评估 |
|------|----------------------|---------------------|------|
| 宽表冗余 | 大量冗余（merchant/agent/path） | 精简（仅 userId + 核心字段） | 我们更精简 |
| 幂等唯一索引 | **无**（order_no 无 UNIQUE） | order_no + flow_type UNIQUE | 我们更安全 |
| 余额快照 | coin_from / coin_to | balance_before / balance_after | 思路一致，可借鉴 |
| 变动方向 | balance_type 枚举 | flow_type + amount 正负 | 各有优劣 |

### 3.2 接口设计特点

**优点**：
- 写操作和查询操作分在两个 Controller，职责清晰
- 所有写操作都加了分布式锁，防并发
- 读写分离的意识（查询接口无锁）

**缺点**：
- **全部使用 POST**，包括查询接口 —— 不符合 RESTful 规范
- **返回值不统一**：有的返回 `ResponseEntity<ResponseVO>`，有的返回 `ResponseVO<?>`
- `/addCoin` 一个接口同时处理加钱和减钱（通过 balanceType 区分） —— 职责过载
- **无鉴权/权限标识**：所有接口都是内部 Feign 调用，依赖网络隔离而非接口级鉴权

## 第四章 核心业务流程

### 4.1 addCoin — 万能加减币方法

这是整个钱包服务的**核心方法**，所有余额变动都走这一个入口。

```
addCoin(coinRecordVO) 流程图：

  ┌─ 1. 校验 coinType > 0 ──────────────────────────────────────┐
  │                                                               │
  ├─ 2. getUserInfoByUserId(userId) ← Feign 调用 + Redis 缓存    │
  │      获取：walletType, merchantId, agentId, currency 等       │
  │                                                               │
  ├─ 3. 判断 walletType                                          │
  │      ├─ TRANSFER (转账钱包, type=1) ──→ 走本地余额逻辑       │
  │      └─ FREE_TRANSFER (免转钱包, type=2) ──→ 走三方API        │
  │                                                               │
  │  ====== TRANSFER 分支 ======                                  │
  ├─ 4. selectOne(userId) 查 user_coin                            │
  │      ├─ 存在 ──→ 继续                                        │
  │      └─ 不存在 ──→ 新建 user_coin（首次充值自动开户）          │
  │                                                               │
  ├─ 5. insufficientBalance() 余额校验                            │
  │      仅对 TRANSFER_OUT/BET/SEND_REWARD 校验                   │
  │      可用余额 = amount - freezeAmount >= coinValue             │
  │                                                               │
  ├─ 6. 根据 balanceType 执行 SQL                                 │
  │      ├─ income(1): addBalance → amount += coinValue            │
  │      └─ expenses(2): subBalance → amount -= coinValue          │
  │         subBalance 返回 0 ──→ 余额不足，抛异常回滚             │
  │                                                               │
  ├─ 7. 计算 coinFrom/coinTo，填充流水 DTO                        │
  │                                                               │
  └─ 8. coinRecordRepository.insert(coinRecordDTO) 写流水          │
```

### 4.2 关键代码逐行分析

**余额校验（insufficientBalance）** — 源码 `CoinRecordServiceImpl.java:366-384`

```java
// 只对 转出/下注/打赏 三种类型做余额校验
if (TRANSFER_OUT.equals(coinType) || BET.equals(coinType) || SEND_REWARD.equals(coinType)) {
    BigDecimal balance = userCoinAmount.subtract(freezeAmount); // 可用 = 总额 - 冻结
    if (balance.compareTo(amount) < 0) {
        throw new MayaDefaultException(ErrorCode.AMOUNT_NOT_ENOUGH);
    }
}
```

**问题**：结算回滚（coinType=6）和取消（coinType=5）**不做余额校验**，
可以减到负数。注释明确说"重算后可以减为负数"。这是一个**业务妥协**——
结算回滚本质上是撤回之前的派彩，即使用户余额已经花完了也必须执行。

**SQL 层防护（subBalance）** — 源码 `UserCoinMapper.xml:4-13`

```xml
UPDATE user_coin b
SET b.amount = b.amount - #{vo.amount},
    b.updated_time = #{vo.updateTime}
WHERE b.id = #{vo.id}
  <!-- coinType != 6(结算回滚) AND coinType != 5(取消) 时才校验 -->
  <if test="vo.coinType != 6 and vo.coinType != 5">
      AND (b.amount - #{vo.amount}) >= 0
  </if>
```

**这是一个精妙的设计**：SQL 层通过条件判断，对不同业务场景施加不同约束。
普通扣款必须 >= 0，但特殊场景（结算回滚）可以突破限制。
通过 `RowsAffected == 0` 判断余额不足。

### 4.3 冻结三操作

| 操作 | 方法 | SQL 语义 | 产生流水 |
|------|------|---------|---------|
| 冻结 | freezeCoin | freeze_amount += X | **否** |
| 扣冻结 | subFreezeBalance | amount -= X, freeze_amount -= X | **是** |
| 回滚冻结 | rollbackFreezeBalance | freeze_amount -= X | **否** |

**冻结语义详解**：

```
初始状态: amount=1000, freeze_amount=0
         可用=1000, 冻结=0, 总额=1000

freezeCoin(200):
         amount=1000, freeze_amount=200
         可用=800, 冻结=200, 总额=1000
         ※ amount 不变！冻结只是标记，不扣总额

subFreezeBalance(200) 真扣款:
         amount=800, freeze_amount=0
         可用=800, 冻结=0, 总额=800
         ※ 同时减两个字段，总额减少

或 rollbackFreezeBalance(200) 回滚:
         amount=1000, freeze_amount=0
         可用=1000, 冻结=0, 总额=1000
         ※ 只减冻结，恢复原状
```

**与我们设计的对比**：
- 参考工程：`总额 = amount`（amount 含冻结），`可用 = amount - freeze_amount`
- 我们的设计：`总额 = balance + frozen_amount`，`可用 = balance`
- **本质相同**，只是字段语义不同。我们的 balance 直接就是可用余额，语义更直观。

### 4.4 批量操作

```java
@Transactional
public boolean addCoinBatch(List<CoinRecordVO> coinRecordVOList) {
    for (CoinRecordVO coinRecordVO : coinRecordVOList) {
        boolean result = addCoin(coinRecordVO);
        if (!result) {
            throw new MayaDefaultException("余额处理失败");
        }
    }
    return true;
}
```

**问题**：批量操作在一个大事务里循环调用 addCoin，每次都做 Feign 调用获取用户信息。
如果 100 笔批量操作，事务持锁时间 = 100 × (Feign 调用 + DB 操作)，
可能导致**长事务**和**锁超时**。

## 第五章 并发控制

### 5.1 分布式锁策略

**锁工具**：Redisson（Redis 客户端，支持分布式锁、可重入锁等）

**锁 Key**：`add.coin.lock.key.{userId}` — 按用户级别加锁

**锁参数**：
```java
rLock.tryLock(20000, 30000L, TimeUnit.MILLISECONDS)
//            ↑ 等锁20秒   ↑ 持锁30秒自动释放
```

**锁位置**：Controller 层（不在 Service 层）

```java
// WalletController.addCoin()
RLock rLock = redisUtil.getLock(RedisLockKey.getAddCoinLockKey(userId));
try {
    if (rLock.tryLock(20000, 30000L, TimeUnit.MILLISECONDS)) {
        boolean result = coinRecordService.addCoin(coinRecordVO);
        if (result) {
            coinRecordService.sendUserBalance(coinRecordVO.getUserId());
        }
    }
} finally {
    if (rLock.isLocked() && rLock.isHeldByCurrentThread()) {
        rLock.unlock();
    }
}
```

### 5.2 锁设计分析

**优点**：
- **用户级粒度**：不同用户互不阻塞，并发友好
- **统一锁 Key**：所有余额变动操作（加/减/冻/解冻）使用同一个锁 Key，
  保证同一用户的所有操作串行化
- **锁超时保护**：30s 自动释放，防止死锁
- **安全释放**：`isLocked() && isHeldByCurrentThread()` 确保只释放自己的锁

**缺点**：
- **锁在 Controller 层**：MQ 消费者（AddCoinListener）也需要加锁，
  导致锁逻辑分散在多处（Controller + Listener），容易遗漏
- **等锁 20s 太长**：金融操作等 20 秒对用户体验影响大
- **无乐观锁兜底**：如果 Redis 故障导致锁失效，没有数据库层面的第二道防线
  （没有 version 字段 CAS 更新）
- **MQ 消费者的锁逻辑不完整**：`noticeDeductCoinOrderMessage` 方法没有加锁，
  批量扣款操作无分布式锁保护

### 5.3 对比评估

| 维度 | 参考工程 | 我们的设计 | 对比 |
|------|---------|-----------|------|
| 锁类型 | Redisson RLock（可重入） | Redis SetNX + Lua 释放 | Redisson 功能更丰富，但我们更轻量 |
| 锁粒度 | userId 级 | userId 级（余额操作）+ 全局级（汇率同步） | 我们更细化 |
| 第二道防线 | 无 | version 字段乐观锁 | **我们更安全** |
| 锁位置 | Controller | Service 层事务内 | 各有利弊 |

## 第六章 事务管理

### 6.1 事务注解使用

```java
@Transactional(rollbackFor = Throwable.class)  // addCoin 方法
@Transactional(rollbackFor = Exception.class)   // deductCoin 方法（inconsistent!）
@Transactional                                  // freezeCoin（无 rollbackFor，默认只回滚 RuntimeException）
```

**问题**：
- **rollbackFor 不一致**：有的写 `Throwable.class`，有的写 `Exception.class`，有的不写。
  建议统一为 `Throwable.class` 或 `Exception.class`
- **freezeCoin 的 @Transactional 没有指定 rollbackFor**：
  如果冻结操作中抛出 checked exception（如 IOException），事务不会回滚

### 6.2 事务边界问题

**严重问题：事务内包含 Feign 远程调用**

```java
@Transactional(rollbackFor = Exception.class)
public boolean addCoin(CoinRecordVO coinRecordVO) {
    // 事务开始
    UserInfoVO userInfoVO = commonService.getUserInfoByUserId(coinRecordVO.getUserId());
    //                      ↑ 这是 Feign 远程调用（或 Redis 缓存读取）！
    //                        如果 user-service 超时 5s，事务持锁 5s

    // ... 然后才做 DB 操作
    userCoinRepository.addBalance(balanceVO);
    coinRecordRepository.insert(coinRecordDTO);
    // 事务结束
}
```

**风险**：
1. Feign 调用超时（默认 5-30s）→ 事务持 DB 连接不释放 → 连接池耗尽
2. Feign 调用成功但 DB 操作失败回滚 → 远程调用已经发出无法撤回
3. 高并发时，大量事务因为等 Feign 响应而阻塞 → 连接池打满

**正确做法**：Feign 调用应放在事务**外部**。先获取所有需要的外部数据，
再进入事务做原子 DB 操作。

### 6.3 另一个事务问题

**subFreezeBalance 方法没有 @Transactional**：

```java
@Override
public ErrorCode subFreezeBalance(CoinFreezeRecordVO coinFreezeRecordVO) {
    // 没有 @Transactional ！！
    userCoinRepository.subFreezeBalance(balanceVO);  // SQL 1: 扣余额+扣冻结
    createOrder(coinFreezeRecordVO, userCoinDTO);     // SQL 2: 写流水
}
```

这意味着如果 SQL 1 成功但 SQL 2 失败（写流水失败），余额已经扣了但没有流水记录。
虽然 GORM 的 `SkipDefaultTransaction: true` 在我们项目里也有类似风险，
但至少我们在核心操作上显式使用了 `query.Q.Transaction`。

## 第七章 金额精度

### 7.1 Java BigDecimal 使用模式

```java

private BigDecimal freezeAmount;
private BigDecimal coinValue;
private BigDecimal coinFrom;
private BigDecimal coinTo;
```

所有金额字段统一使用 `BigDecimal`，Java 层面精度安全。

### 7.2 计算模式

```java

BigDecimal coinTo = userCoinDTO1.getAmount().add(coinRecordVO.getCoinValue());  // 加法
```

### 7.3 精度评估

**优点**：
- 全链路 BigDecimal，无 float/double
- SQL 使用 `b.amount + #{vo.amount}` 直接在 DB 层做算术运算

**缺点**：
- **未指定 DECIMAL 精度**：DDL 中没有看到 `DECIMAL(p,s)` 的精度定义，

## 第八章 幂等性设计

### 8.1 现状：**缺乏系统性幂等保护**

分析所有写操作：

| 操作 | 幂等保护 | 评估 |
|------|---------|------|
| addCoin | 无 UNIQUE 索引、无 orderNo 去重 | **高风险** |
| addCoinList | 同上 | **高风险** |
| freezeCoin | 无 | **中风险** |
| subFreezeBalance | 无 | **高风险** |
| rollbackFreezeBalance | 无 | **中风险** |
| MQ NoticeAddCoinMessage | 无消息去重 | **高风险** |
| MQ noticeDeductCoinOrderMessage | 无消息去重 | **高风险** |

### 8.2 唯一的"类幂等"机制

```java
// queryCoinRecordByOrderNo — 按 orderNo 查询
// 但这只是查询接口，不是自动去重，需要调用方主动先查再决定
```

实际代码中，没有任何地方在 addCoin 前检查"这个 orderNo 是否已经处理过"。
coin_record 表的 order_no 字段**没有 UNIQUE 索引**，
同一个 orderNo 可以插入多条记录。

### 8.3 风险场景

```
场景：MQ 消息重投
1. settlement-service 发送结算消息到 ADD_COIN_QUEUE
2. wallet-service 消费成功，addCoin 执行完毕
3. 但 ACK 还没提交时，consumer 崩溃
4. RabbitMQ 重投消息
5. 新的 consumer 再次消费 → addCoin 再次执行
6. 用户余额多加了一次！
```

这是参考工程中**最严重的设计缺陷**之一。

### 8.4 对我们的启示

**必须借鉴（反面教材）**：
- 流水表必须建 `order_no + flow_type` UNIQUE 索引
- 所有写操作入口必须做幂等检查（先查后写 + UNIQUE 兜底）
- MQ 消费必须是幂等的

### 10.2 缓存策略

```java
// CommonServiceImpl — 用户信息缓存
String redisKey = RedisConstants.CURRENCY_LIMIT_ID_MERCHANT_ID + userId;
Object obj = redisUtil.get(redisKey);  // 先查 Redis
```

## 第十一章 冻结/解冻/扣款深度分析

### 11.1 SQL 层实现对比

```sql
-- addFreezeBalance: 冻结
SET freeze_amount = freeze_amount + #{amount}
WHERE id = #{id}
-- 注意：没有校验 freeze_amount + amount <= total_amount 的防溢出逻辑
-- 应用层 freezeCoin() 虽然做了 amount >= freezeAmount 的校验，
-- 但是在并发场景下，两个冻结请求可能同时通过应用层校验，
-- 然后两个 SQL 都执行成功，冻结金额超过总额

-- subFreezeBalance: 扣减冻结
SET amount = amount - #{amount},
    freeze_amount = freeze_amount - #{amount}
WHERE id = #{id}
-- 注意：没有 AND freeze_amount >= #{amount} 的防止冻结变负数的校验
-- 也没有 AND (amount - #{amount}) >= 0 的余额非负校验

-- rollbackFreezeBalance: 回滚冻结
SET freeze_amount = freeze_amount - #{amount}
WHERE id = #{id}
-- 同样没有 freeze_amount >= #{amount} 的校验
```

### 11.2 冻结的并发安全隐患

```
并发场景：
  余额: amount=1000, freeze_amount=0

  线程A: freezeCoin(800) → 应用层校验通过（1000 >= 800）
  线程B: freezeCoin(800) → 应用层校验通过（1000 >= 800）

  线程A: SQL执行 → freeze_amount = 0 + 800 = 800 ✓
  线程B: SQL执行 → freeze_amount = 800 + 800 = 1600 ✗ (超过总额!)

  结果: amount=1000, freeze_amount=1600 → 冻结比总额还多！
```

虽然有分布式锁保护（freezeCoin 在 Controller 层加了锁），
但如果锁获取失败或者异常释放，就会出现这个问题。
没有 DB 层面的兜底约束。

### 11.3 与我们的对比

| 维度 | 参考工程 | 我们的设计 |
|------|---------|-----------|
| 冻结校验 | 仅应用层 | 应用层 + DB UNSIGNED |
| 并发防护 | 仅分布式锁 | 分布式锁 + 乐观锁 + UNIQUE |
| 回滚校验 | 无（可冻结变负数） | 有（decimal 计算 + UNSIGNED） |
| 冻结产生流水 | 否 | 是（冻结也记录流水） |

## 第十二章 双钱包模式

### 12.1 转账钱包 vs 免转钱包

参考工程支持两种钱包模式，这是一个有趣的设计：

**转账钱包（TRANSFER, type=1）**：
- 余额存储在钱包服务本地（user_coin 表）
- 所有加减币操作直接更新本地数据
- 用户在下注前需要先把钱"转入"钱包

**免转钱包（FREE_TRANSFER, type=2）**：
- 余额**不在**钱包服务本地存储
- 余额由三方商户系统管理
- 加减币通过 api-service → 三方 API 完成
- 本地只记录流水（coin_from/coin_to 为 null）
- 查询余额返回 0（需要去三方查实时余额）

### 12.2 免转钱包的实现

```java
// addCoin 中免转钱包的处理
} else {
    // 先调用免转商户加减余额
    ThirdOrderRequestVO req = new ThirdOrderRequestVO();
    req.setTransferNo(OrderUtil.getOrderNo("T", 11));
    // ... 构造请求参数

    ThirdOrderResponseVO resp = apiResource.amountTransfer(req);

    switch (ErrorCode.nameOfCode(resp.getCode())) {
        case SUCCESS:
            // 成功 → 记录三方转账记录 + 写本地流水
            orderResource.saveThirdOrderRecord(thirdOrderRecordDTO);
            break;
        default:
            // 失败 → 不写流水，返回 false
            return false;
    }

    // 写本地流水（coinFrom/coinTo 都是 null）
    coinRecordRepository.insert(coinRecordDTO);
}
```

### 12.3 对我们的借鉴

我们的场景不涉及"免转钱包"（没有三方商户托管余额的需求），
但这个模式提供了一个启示：**钱包服务可以作为"代理"**，
对外提供统一的加减币接口，内部根据账户类型路由到不同的实现。

不过我们的子钱包（中心/奖励/主播/代理/场馆）都是本地管理，
不需要这种代理模式。这个设计**弃用**。

## 第十三章 安全与校验

### 13.1 参数校验

```java
// Controller 层手动校验
if (StringUtils.isEmpty(coinRecordVO.getUserId())) {
    return ResponseUtil.fail(ErrorCode.PARAM_ERROR);
}
if (coinRecordVO.getCoinValue().compareTo(BigDecimal.ZERO) == 0) {
```

**问题**：
- 没有使用 JSR-303 `@Valid` 注解，全靠手写 if 判断
- 校验逻辑分散在 Controller 和 Service 中，不集中
- 没有校验 coinValue 为负数的情况（`compareTo(ZERO) == 0` 只拦截了等于 0）

### 13.2 异常处理

```java
// Controller 层
try {
    if (rLock.tryLock(...)) {
        boolean result = coinRecordService.addCoin(coinRecordVO);
    }
} catch (Exception e) {
    log.error(e.getMessage(), e);

// 所有异常被 catch 后返回通用失败

```

**问题**：
- catch Exception 太宽泛，隐藏了具体错误
- 锁获取失败和业务异常返回**同一个错误码**（FAIL），调用方无法区分
- MQ 消费者中的部分 catch 直接**吞异常**，不重试不告警

## 第十四章 优缺点全景评估

### 14.1 值得借鉴的设计（精华）

| # | 设计点 | 具体实现 | 为什么好 |
|---|--------|---------|---------|
| 1 | SQL 层条件扣款 | `AND (amount - #{amount}) >= 0` + RowsAffected 判断 | DB 层兜底，比纯应用层校验更安全 |
| 2 | 分布式锁粒度 | userId 级别，同一用户串行 | 并发友好且保证一致性 |
| 3 | 流水快照 | coin_from/coin_to 记录变动前后余额 | 审计必须，故障排查利器 |
| 4 | 统一锁 Key | 加/减/冻/解冻用同一个锁 | 不同操作间互斥，简单有效 |
| 5 | 余额通知解耦 | 通过 MQ Fanout 广播余额变化 | 钱包不依赖推送服务 |
| 6 | 自动开户 | 首次充值时自动创建 user_coin | 无需单独的"注册钱包"步骤 |
| 7 | 结算回滚可负数 | coinType=5,6 时跳过余额校验 | 业务现实需要，灵活处理特殊场景 |
| 8 | 安全锁释放 | `isLocked() && isHeldByCurrentThread()` | 防止释放别人的锁 |

### 14.2 需要规避的缺陷（糟粕）

| # | 缺陷 | 具体表现 | 风险等级 | 为什么差 |
|---|------|---------|---------|---------|
| 1 | **无幂等保护** | order_no 无 UNIQUE 索引，无去重逻辑 | **致命** | 消息重投/重试会导致重复入账 |
| 2 | **事务内远程调用** | @Transactional 内包含 Feign 调用 | **严重** | 长事务、连接池耗尽、数据不一致 |
| 3 | **无乐观锁** | 没有 version 字段 | **严重** | 分布式锁失效后无第二道防线 |
| 4 | **subFreezeBalance 无事务** | 扣余额和写流水非原子 | **严重** | 扣了钱没流水 = 账不平 |
| 5 | **冻结无溢出校验** | SQL 层没有 freeze <= amount 约束 | **高** | 并发冻结可超额 |
| 6 | **MQ 消费者锁不一致** | 4 个消费者只 1 个加锁 | **高** | 并发保护有漏洞 |
| 7 | **catch Exception 吞异常** | MQ 消费失败无重试无告警 | **中** | 数据丢失无感知 |
| 8 | **Feign 无降级** | user-service 不可用则全部瘫痪 | **中** | 单点依赖 |
| 9 | **事务 rollbackFor 不一致** | 有的 Throwable 有的 Exception 有的不写 | **低** | checked exception 漏回滚 |
| 10 | **SQL 查询 SQL 注入风险** | `${vo.createdTimeBy}` 直接拼接 | **中** | 排序字段未白名单校验 |
| 11 | **XML 大量重复 SQL** | getCoinRecordPage/List/Count 三段完全重复 | **低** | 维护成本高 |

### 14.3 设计不足但可接受的妥协

| # | 设计点 | 现状 | 为什么可接受 |
|---|--------|------|------------|
| 1 | 全 POST 接口 | 查询也用 POST | 内部 Feign 调用，不对外暴露 |
| 2 | 宽表冗余 | 流水表冗余了 merchant/agent 信息 | 避免 JOIN，查询性能好 |
| 3 | 贫血模型 | DTO 无业务逻辑 | 项目规模小，够用 |
| 4 | 单币种设计 | 每用户一行 | 当时业务场景确实只需要单币种 |

### 15.2 调（改进后借鉴）

| # | 原始设计 | 问题 | 我们的改进 |
|---|---------|------|-----------|
| 1 | 分布式锁在 Controller | 逻辑分散，MQ 容易遗漏 | 锁逻辑封装在 Service 层公共方法中 |
| 2 | 批量操作大事务 | N 次循环 × Feign 调用 | 先批量获取外部数据，再进入事务 |

| 4 | 冻结只校验应用层 | 并发可超额 | 应用层 + DB UNSIGNED + 乐观锁 |

| 6 | 余额通知在事务外 | 有时在事务内发 MQ | 统一：事务提交成功后再发通知 |

### 15.3 弃（不采用）

| # | 设计点 | 为什么弃用 |
|---|--------|-----------|
| 1 | 无幂等保护 | 金融系统致命缺陷，我们必须有 UNIQUE 索引 |
| 2 | 事务内远程调用 | 违反分布式事务基本原则，我们严格隔离 |
| 3 | 无乐观锁 | 分布式锁 + 乐观锁双层防护是我们的底线 |
| 4 | 免转钱包模式 | 我们的子钱包全部本地管理，无三方托管需求 |

| 7 | MyBatis XML 手写 SQL | 我们用 GORM Gen 类型安全查询 |
| 8 | subFreezeBalance 无事务 | 余额变动 + 流水必须在同一事务内 |
| 9 | 宽表冗余策略 | 我们流水表精简，查询时按需 JOIN 或 RPC 补充 |
| 10 | user_id String 类型 | 我们统一 int64，内部系统无需适配异构 ID |

## 第十六章 总结：从参考工程看到了什么

### 16.1 参考工程的本质

这是一个为**游戏平台定制的单币种轻量钱包**，核心功能是：
- 为下注/结算/充值/提现/打赏提供余额操作
- 支持转账钱包和免转钱包两种模式
- 通过分布式锁保证并发安全

**它的定位**是"够用就好"——约 1600 行代码支撑了全部功能，
但在金融安全的几个核心维度上存在明显不足。

### 16.2 我们 ser-wallet 已经超越的部分

| 维度 | 参考工程 | 我们 |
|------|---------|------|
| 幂等性 | 无 | orderNo + flow_type UNIQUE 三层防护 |
| 并发控制 | 仅分布式锁 | 分布式锁 + 乐观锁双层 |
| 金额精度 | BigDecimal（无 DECIMAL 精度定义） | string + decimal + DECIMAL(30,8) UNSIGNED |
| 事务边界 | 事务内混合远程调用 | 严格隔离（事务内只做 DB，外部调用放事务外） |
| 多币种 | 单币种 | 多币种 + 多子钱包 + 汇率系统 |
| 跨模块补偿 | 无 | 冻结→提交→回滚三段式 + 悬挂预防 |
| 状态机 | 无 | 充值 4 态 / 提现 8 态 / 冻结 3 态 |
| 降级策略 | 无 | KYC 降级 + 缓存兜底 |

### 16.3 参考工程给我们的最大启示

**不是技术实现，而是"什么不该做"的反面教材**：

1. **幂等不是可选项**，是金融系统的生命线
2. **事务边界**要像手术一样精确，多一步少一步都可能致命
3. **分布式锁不是万能的**，必须有第二道防线（乐观锁 / UNIQUE 约束）
4. **MQ 消费必须幂等**，消息中间件不保证 exactly-once
5. **异常不能吞**，金融系统的每一个异常都可能是资金安全事件

同时也有几个实用的工程实践值得直接借用：
- SQL 条件更新（`AND amount >= X`）是简洁有效的余额防护
- 流水的 before/after 快照是审计必备
- 统一锁 Key 让不同操作间自然互斥
- MQ Fanout 广播余额变化是优雅的解耦方式
