### 2.2 架构特点分析

**（1）Controller 层 = 分布式锁 + 委托**

Controller 层并非纯路由转发，而是承担了分布式锁的获取与释放。这是一个**有争议**的设计决策：

```java
// WalletController.java:60-78 — 锁在 Controller 层
RLock rLock = redisUtil.getLock(RedisLockKey.getAddCoinLockKey(userId));
try {
    if (rLock.tryLock(20000, 30000L, TimeUnit.MILLISECONDS)) {
        boolean result = coinRecordService.addCoin(coinRecordVO);
        // ...
    }
} finally {
    rLock.unlock();
}
```

> **评价**：锁应该在 Service 层内部管理，Controller 层不应感知并发控制细节。这导致了如果有 MQ 消费也需要 addCoin，则必须在 Listener 中重复锁逻辑（实际代码确认了这一点——AddCoinListener:48-66 完全重复了锁模式）。

**（2）没有 Balance Engine 抽象**

所有余额变更操作散落在 `CoinRecordServiceImpl` 中的多个方法里（`addCoin`、`deductCoin`、`freezeCoin`、`subFreezeBalance`、`rollbackFreezeBalance`），没有统一的余额变更入口。

> **评价**：这是一个明显的设计缺失。我们的 ser-wallet 已设计了 `BalanceEngine.ModifyBalance()` 作为唯一入口，这是正确的方向。

**（3）双钱包模式**

系统支持两种钱包类型：
- `TRANSFER`（转账钱包，code=1）：余额存在 wallet-service 的 `user_coin` 表，所有加减操作走本地数据库
- `FREE_TRANSFER`（免转钱包，code=2）：余额存在三方商户侧，加减操作通过 API 调用商户的 `addDeductionUrl`

> **评价**：这是 B2B 游戏平台的特殊需求（包网模式），我们的 ser-wallet 目前只需要类似"转账钱包"的模式——余额在我方管理。但双钱包的抽象思路值得参考。

**（4）VO/DTO 分层但混用**

- `p9-core-lib` 中定义共享 VO（跨服务传输用）
- `wallet-service` 中定义 DTO（数据库映射用）
- 但很多地方 VO 直接作为 Controller 入参传到 Service 再 BeanUtils.copyProperties 到 DTO

> **评价**：BeanUtils.copyProperties 是 Java 常见模式但效率低且不安全（字段名不匹配时静默忽略）。我们的 Go 工程应避免这种"隐式映射"，用明确的构造或转换函数。

## 第3章 数据模型与表结构

### 3.1 user_coin 表（用户钱包）

**源码定位**：`wallet-service/.../dto/UserCoinDTO.java` + `UserCoinMapper.xml`

```
表名: user_coin
继承: BasicDataBaseDTO (id, creator, updater, created_time, updated_time)

字段:
┌──────────────┬──────────────┬──────────────────────────────────┐
│ 字段名        │ Java 类型     │ 说明                              │
├──────────────┼──────────────┼──────────────────────────────────┤
│ id           │ Long (AUTO)  │ 自增主键                           │
│ user_id      │ String       │ 玩家 ID                           │
│ user_name    │ String       │ 玩家名称                           │
│ amount       │ BigDecimal   │ 总金额（含冻结）                    │
│ freeze_amount│ BigDecimal   │ 冻结金额                           │
│ user_state   │ Integer      │ 游戏状态                           │
│ game_type    │ Integer      │ 正在进行的游戏                      │
│ wallet_type  │ Integer      │ 1=转账钱包, 2=免转钱包              │
│ currency     │ String       │ 币种                              │
│ remark       │ String       │ 备注                              │
│ creator      │ Long         │ 创建人                             │
│ updater      │ Long         │ 更新人                             │
│ created_time │ Date         │ 创建时间                           │
│ updated_time │ Date         │ 更新时间                           │
└──────────────┴──────────────┴──────────────────────────────────┘

关键约束（从代码推断，无 DDL 文件）:
- id: AUTO_INCREMENT 主键
- user_id: 应有唯一索引（但代码中 selectOne 用 QueryWrapper 无限定条件，暗示 user_id 唯一）
- 无 user_id+currency 复合索引（每个用户只有一个钱包，不支持多币种多钱包）
```

**余额语义**：
```
可用余额 = amount - freeze_amount
总余额   = amount
冻结余额 = freeze_amount
```

> **重要发现**：`amount` 字段存储的是**总金额**（可用+冻结），而非纯可用余额。这与很多系统用 `available_balance + frozen_balance = total_balance` 三字段模式不同。只用两个字段，可用余额靠计算得出。

> **关键观察**：
> 1. **coin_from / coin_to 是余额快照**——这是审计链（我们的 `before_balance / after_balance`），优秀实践
> 2. **coin_amount 字段冗余**——代码中有时赋值为 `coinTo`，有时为 `freezeAmount`，语义不清晰
> 3. **大量冗余字段**——`user_name / nick_name / merchant_name / agent_name / path` 全部冗余存储在每条账变记录中。这是**反范式设计**，查询友好但写入臃肿（每次账变都要 RPC 获取用户信息再填充）
> 4. **无唯一约束**——`order_no` 没有唯一索引，意味着**没有数据库层面的幂等保护**

### 3.4 账变类型枚举全表

**CoinChangeEnum（coin_type）**：

|------|------|------|---------|------|

| 3 | BET | 下注 | -（支出） | 用户投注扣款 |

> **对比我们的设计**：我们的 OpType 更抽象（Credit/Debit/Freeze/Unfreeze/DebitFrozen），而这里的枚举是业务语义的。两种设计各有优劣——我们的更通用，他们的更直观。建议我们保留 OpType 作为底层抽象，上层业务用 tx_type 映射具体业务场景。

### 5.2 余额变更 SQL 详解

**源码定位**：`UserCoinMapper.xml:4-43`

```xml
<!-- 减余额（条件扣减） -->
<update id="subBalance">
    UPDATE user_coin b
    SET b.amount = b.amount - #{vo.amount},
        b.updated_time = #{vo.updateTime}
    WHERE b.id = #{vo.id}
    <!-- 结算回滚(6) 和 取消(5) 允许扣为负数 -->
    <if test="vo.coinType != 6 and vo.coinType != 5">
        AND (b.amount - #{vo.amount}) >= 0
    </if>
</update>

<!-- 加余额（无条件） -->
<update id="addBalance">
    UPDATE user_coin b
    SET b.amount = b.amount + #{vo.amount},
        b.updated_time = #{vo.updateTime}
    WHERE b.id = #{vo.id}
</update>

<!-- 增加冻结额 -->
<update id="addFreezeBalance">
    UPDATE user_coin b
    SET b.freeze_amount = b.freeze_amount + #{vo.amount},
        b.updated_time = #{vo.updateTime}
    WHERE b.id = #{vo.id}
</update>

<!-- 扣除冻结（确认提款）：总额减少 + 冻结减少 -->
<update id="subFreezeBalance">
    UPDATE user_coin b
    SET b.amount = b.amount - #{vo.amount},
        b.freeze_amount = b.freeze_amount - #{vo.amount},
        b.updated_time = #{vo.updateTime}
    WHERE b.id = #{vo.id}
</update>

<!-- 回滚冻结（取消提款）：仅冻结减少，总额不变 -->
<update id="rollbackFreezeBalance">
    UPDATE user_coin b
    SET b.freeze_amount = b.freeze_amount - #{vo.amount},
        b.updated_time = #{vo.updateTime}
    WHERE b.id = #{vo.id}
</update>
```

> **核心观察**：
> 1. `subBalance` 使用 **条件 UPDATE**（`AND (amount - #{amount}) >= 0`）防止扣为负数——这是一种**乐观锁**思路，但这里配合外层的 Redisson 分布式锁，实际是**双保险**
> 2. 结算回滚(6)和取消(5)特殊处理：允许余额扣为负数——这是业务规则（回滚可能在用户已消费余额后发生）
> 3. `subFreezeBalance` 同时减 amount 和 freeze_amount——对应"确认冻结扣款"
> 4. `rollbackFreezeBalance` 只减 freeze_amount——对应"取消冻结归还可用"
> 5. **所有 UPDATE 都用 `WHERE id = #{id}`**，而非 `WHERE user_id = #{userId}`——因为前面已经 selectOne 拿到了 id

### 5.3 余额校验逻辑

**源码定位**：`CoinRecordServiceImpl.java:366-384`

```java
private void insufficientBalance(CoinRecordVO coinRecordVO, UserCoinDTO userCoin) {
    // 仅对 转出/下注/打赏 校验余额
    if (TRANSFER_OUT / BET / SEND_REWARD) {
        BigDecimal amount = coinRecordVO.getCoinValue();        // 要扣的金额
        BigDecimal userCoinAmount = userCoin.getAmount();       // 总余额
        BigDecimal freezeAmount = userCoin.getFreezeAmount();   // 冻结额
        BigDecimal balance = userCoinAmount.subtract(freezeAmount); // 可用余额
        if (balance.compareTo(amount) < 0) {
            throw new MayaDefaultException(ErrorCode.AMOUNT_NOT_ENOUGH);
        }
    }
}
```

> **注意**：
> 1. 结算(4)、注单取消(5)、结算回滚(6)、重算(7) 不校验余额——因为是系统操作，可能导致余额为负
> 2. 这里的校验是在 Java 代码层面的"预检"，数据库层面还有 `(amount - #{amount}) >= 0` 的保护
> 3. **潜在风险**：校验和扣款之间不是原子操作（虽然有锁保护，但如果锁失效...）

### 5.4 冻结/解冻流程

**冻结（freezeCoin）**：

```
1. 查询 user_coin WHERE userId = ?
2. 校验: amount >= freezeAmount（总余额 >= 要冻结的金额）
3. UPDATE: freeze_amount += freezeAmount
4. 不产生账变记录
```

**确认扣除冻结（subFreezeBalance）**：

```
1. 查询 user_coin WHERE userId = ?
2. 校验: freeze_amount >= 要扣除的冻结金额
3. UPDATE: amount -= freezeAmount, freeze_amount -= freezeAmount
4. 产生账变记录（createOrder 方法）
```

**回滚冻结（rollbackFreezeBalance）**：

```
1. 查询 user_coin WHERE userId = ?
2. UPDATE: freeze_amount -= freezeAmount
3. 不产生账变记录
```

> **映射到我们的 OpType**：
> - freezeCoin → `Freeze`（available -= amount, frozen += amount）
> - subFreezeBalance → `DebitFrozen`（frozen -= amount, 资金离开系统）
> - rollbackFreezeBalance → `Unfreeze`（frozen -= amount, available += amount）
>
> **但这里有一个语义差异**：参考工程的 `amount` 是总额不变（freeze 时），而 `subFreezeBalance` 是总额减少。我们的设计中 `available + frozen = total` 是恒等式，更清晰。

## 第6章 并发控制与一致性机制

### 6.1 分布式锁策略

**锁实现**：Redisson RLock（基于 Redis 的可重入分布式锁）

```
锁 Key 模式: add.coin.lock.key.{userId}
尝试获取超时: 20 秒
锁持有上限: 30 秒
锁粒度: 用户级别（同一用户的所有钱包操作串行化）
```

**加锁位置统计**：

| 位置 | 锁 Key | 说明 |
|------|--------|------|
| WalletController.addCoin | userId | HTTP 同步调用 |
| WalletController.addCoinList | 取第一个元素的 userId | 批量操作 |
| WalletController.freezeCoin | userId | 冻结 |
| WalletController.subFreezeBalance | userId | 确认冻结扣款 |
| WalletController.rollbackFreezeBalance | userId | 回滚冻结 |
| AddCoinListener.NoticeAddCoinMessage | userId | MQ 消费 |
| api-service CoinRecordController.transferAmount | merchantNo:userId | API 层额外锁 |

> **问题清单**：
> 1. **锁位置不统一**：5个 Controller 方法 + 1个 MQ Listener 都各自加锁，**代码重复 6 次**
> 2. **addCoinList 锁只取第一个元素的 userId**——如果批量中有不同 userId，其他用户的操作**没有锁保护**
> 3. **api-service 额外加了一层锁**（不同 Key 模式 `api-service:transferAmount:{merchantNo}:{userId}`），造成同一操作被两层锁序列化——性能浪费且可能死锁
> 4. **锁超时 20 秒太长**——在高并发下，大量请求排队等锁
> 5. **锁持有 30 秒也太长**——如果 addCoin 内部调用免转钱包的 Feign→API→三方HTTP，可能真的需要这么久，但这暴露了架构问题

### 6.2 事务策略

| 方法 | @Transactional | 说明 |
|------|---------------|------|
| addCoin | `rollbackFor = Exception.class` | ✓ 正确 |
| addCoinBatch | `@Transactional`（无 rollbackFor） | ⚠ rollbackFor 缺失 |
| deductCoin | `rollbackFor = Throwable.class` | ✓ 正确 |
| deductCoinBatch | `@Transactional`（无 rollbackFor） | ⚠ rollbackFor 缺失 |
| freezeCoin | `@Transactional`（无 rollbackFor） | ⚠ rollbackFor 缺失 |
| subFreezeBalance | **无 @Transactional** | ❌ 严重缺陷！ |
| rollbackFreezeBalance | **无 @Transactional** | ❌ 缺失但影响较小 |
| changeCoinAndNoticeBalance | `@Transactional`（无 rollbackFor） | ⚠ rollbackFor 缺失 |

> **严重问题**：
> 1. `subFreezeBalance` 没有 `@Transactional` —— 它内部做了两步操作：(1) `userCoinRepository.subFreezeBalance()`（UPDATE 余额）+ (2) `createOrder()`（INSERT 账变记录）。如果步骤2失败，步骤1已经提交，导致**余额已扣但无账变记录**。
> 2. 多处 `@Transactional` 没有指定 `rollbackFor`，默认只回滚 `RuntimeException`，checked exception 不回滚。

## 第9章 冻结解冻机制深度分析

### 9.1 冻结模型

参考工程用的是**两字段模型**：

```
user_coin:
  amount        = 总额（available + frozen）
  freeze_amount = 冻结额

available（可用）= amount - freeze_amount  // 代码中计算得出
```

| 操作 | amount 变化 | freeze_amount 变化 | 可用余额变化 | 产生账变 |
|------|-----------|-------------------|------------|---------|
| 充值(+100) | +100 | 不变 | +100 | ✓ |
| 下注(-50) | -50 | 不变 | -50 | ✓ |
| 冻结(30) | 不变 | +30 | -30 | × |
| 确认冻结扣除(30) | -30 | -30 | 不变 | ✓ |
| 回滚冻结(30) | 不变 | -30 | +30 | × |

### 9.2 我们的三字段模型对比

```
wallet_account:
  available_balance = 可用余额
  frozen_balance    = 冻结余额
  total_balance     = 总余额（冗余，= available + frozen）

| 操作 | available 变化 | frozen 变化 | total 变化 |
|------|-------------|-----------|-----------|
| Credit(+100) | +100 | 不变 | +100 |
| Debit(-50) | -50 | 不变 | -50 |
| Freeze(30) | -30 | +30 | 不变 |
| DebitFrozen(30) | 不变 | -30 | -30 |
| Unfreeze(30) | +30 | -30 | 不变 |
```

> **对比结论**：
> - 参考工程的两字段模型**更简洁**，但可用余额需要计算
> - 我们的三字段模型**更直观**，任何时候看到的 available_balance 就是可以用的，无需减法计算
> - 数据库层面，三字段可以直接对 available_balance 做 `>= 0` 约束，两字段需要对 `amount - freeze_amount` 做约束（更复杂）
> - **建议**：保持我们的三字段模型——直观性和约束友好性更好

### 9.3 冻结校验缺陷

```java
// freezeCoin 方法中:
if (userCoinDTO.getAmount().compareTo(coinFreezeVO.getFreezeAmount()) < 0) {
    return ErrorCode.USER_WALLET_NOT_ENOUGH_BALANCE;
}
```

> **问题**：校验的是 `amount >= freezeAmount`（总额 >= 冻结金额），但正确应该是 `(amount - freeze_amount) >= 要冻结的金额`，即可用余额 >= 要冻结的金额。如果用户已有部分冻结，这个校验就会**允许超额冻结**。

**举例**：
- amount=100, freeze_amount=80, 可用余额=20
- 请求冻结 50
- 校验: 100 >= 50 → 通过 ✓（但实际可用只有 20！）
- 结果: freeze_amount 变成 130 > amount，可用余额变为 -30

> **这是一个严重 BUG**。我们的设计中必须校验 `available_balance >= freeze_amount`。

## 第10章 安全设计分析

### 10.1 认证与授权

| 层级 | 机制 | 实现 |
|------|------|------|
| 外部 API | MD5 签名 | `SecurityUtil.paramSigns(apiSign, params...)` |
| 内部 Feign | 无认证 | K8s 网络隔离（依赖内网安全） |
| 管理后台 | 未在钱包代码中体现 | 依赖 admin-service 统一认证 |

> **评价**：
> 1. MD5 签名已被认为不够安全（碰撞攻击），行业标准已转向 HMAC-SHA256
> 2. 内部服务间无认证是常见做法，但不够安全——一旦内网被攻破，攻击者可以直接调用钱包接口
> 3. 我们的设计应在 RPC 层面加入 Service Token 或 mTLS

### 10.2 金额安全

| 措施 | 是否实现 | 评价 |
|------|---------|------|
| BigDecimal 精度 | ✓ | 正确使用 BigDecimal，避免浮点误差 |
| 条件扣减 (>=0) | ✓ | SQL 层保护，防止余额为负 |
| 余额预检 | ✓ | Java 层先检查再操作 |
| 负数金额校验 | ✓ | coinValue.compareTo(ZERO)==0 拒绝 |
| 幂等保护 | ✗ | **严重缺失** — 无唯一索引 |
| 金额上限校验 | ✗ | 无单笔/单日限额 |
| 审计追踪 | △ | coin_from/coin_to 有但 coinAmount 语义混乱 |

### 10.3 防重放/幂等分析

```
addCoin 方法内部:
  1. 无 Redis SetNX 幂等检查
  2. 无 DB 唯一索引 (order_no 无唯一约束)
  3. 唯一的幂等保护在 api-service 层:
     findCoinResult(merchantId, orderNo) — 查询是否已有记录
     但这是 SELECT 检查，存在 TOCTOU 竞态
```

> **评价**：这是参考工程**最大的安全隐患**。如果同一笔订单的 addCoin 请求被发送两次（网络重试、MQ 重复投递），将导致**重复入账**。
>
> 我们的设计中有**双层幂等**：
> 1. Redis SetNX（快速拦截，24h TTL）
> 2. DB 唯一索引 `uk_ref_order_tx(ref_order_no, tx_type, wallet_type)`
>
> 这是必须坚持的设计。
