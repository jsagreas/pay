## 二、钱包模块定位与边界

### 2.1 职责范围

| 职责 | 是否包含 | 说明 |
|------|---------|------|
| 余额管理 | ✅ | 单一user_coin表管理可用+冻结余额 |
| 账变记录 | ✅ | coin_record表记录所有变动流水 |
| 冻结/解冻 | ✅ | freeze_amount字段 + 3个冻结接口 |
| 充值/提现 | ❌ | 充值=商户"转入"，提现=商户"转出"（无独立充提流程） |
| 币种配置 | ❌ | currency字段存储但无独立币种管理模块 |
| 汇率管理 | ❌ | 完全不涉及 |
| 对账 | ❌ | 无对账机制 |
| 下注扣款 | ✅（通过MQ） | order-service直接操作user_coin表 + 发MQ给wallet |
| 结算加款 | ✅（通过MQ） | 游戏服务结算后通过MQ触发wallet加款 |

### 2.2 业务模型

**核心概念**：该工程的"钱包"本质上是**游戏平台的筹码管理系统**，不是传统金融钱包。

```
两种钱包模式:
  ① 转账钱包(TRANSFER, code=1): 余额存在本地DB(user_coin表)
     - 充值：商户→本平台转入（addCoin, coinType=1）
     - 提现：本平台→商户转出（addCoin, coinType=2）
     - 下注：直接扣减本地余额
     - 结算：直接增加本地余额

  ② 免转钱包(FREE_TRANSFER, code=2): 余额存在第三方(商户侧)
     - 充值：不经过本地DB
     - 下注：调用第三方API扣减
     - 结算：调用第三方API增加
     - 本地只记录账变流水，不维护余额

关键理解: 这里的"充值""提现"不是我们理解的法币充值/提现
而是商户(包网方)与平台之间的筹码上分/下分操作
```

### 2.3 与我方ser-wallet的本质区别

| 维度 | 参考工程wallet-service | 我方ser-wallet |
|------|----------------------|---------------|
| 定位 | 游戏筹码管理（B2B2C） | 用户资产钱包（B2C） |
| 用户模型 | 商户下的玩家 | 直接用户 |
| 多币种 | currency字段但单钱包 | 多币种×多钱包类型 |
| 充提 | 商户上下分 | 法币/虚拟币充值提现 |
| 汇率 | 无 | 4层汇率体系 |
| 兑换 | 无 | 跨币种兑换 |
| 结算层 | 无独立层 | ledger层（统一余额变更入口） |

## 四、核心业务逻辑实现

### 4.1 addCoin — 核心加减款方法

**文件**：`CoinRecordServiceImpl.java:169-341`

这是整个钱包模块最核心的方法，承担所有资金变动。逻辑拆解如下：

```
addCoin(coinRecordVO) 入口:
  ├─ 参数校验: coinType > 0
  ├─ 获取用户信息: commonService.getUserInfoByUserId()  ← Feign调user-service
  ├─ 冗余字段回填: 把userName/nickName/merchantId等写入coinRecordDTO
  │
  ├─ [分支1] 转账钱包(walletType=1):
  │   ├─ 查询user_coin: selectOne(userId)
  │   ├─ [存在钱包]
  │   │   ├─ 余额校验: insufficientBalance()  ← 只对转出/下注/打赏校验
  │   │   ├─ [收入] userCoinRepository.addBalance(id, amount)
  │   │   ├─ [支出] userCoinRepository.subBalance(id, amount)
  │   │   │         └─ subBalance返回0 → throw余额不足
  │   │   ├─ 计算coinTo/coinFrom
  │   │   └─ coinRecordRepository.insert()  ← 写入流水
  │   └─ [不存在钱包]
  │       ├─ 检查必须是收入(不能首笔就扣款)
  │       ├─ userCoinRepository.insert()  ← 创建钱包
  │       └─ coinRecordRepository.insert()  ← 写入流水
  │
  └─ [分支2] 免转钱包(walletType=2):
      ├─ 查询关联订单: orderResource.findOrderByOrderNo()
      ├─ 查询商户详情: commonService.getMerchantDetailVO()
      ├─ 组装三方请求: ThirdOrderRequestVO
      ├─ 调用三方API: apiResource.amountTransfer()  ← 远程HTTP调用
      ├─ [成功] 记录三方订单 + 写入流水 + 发送余额变更MQ
      └─ [失败] 记录三方订单(UNDONE状态) + return false
```

### 4.2 关键代码段精读

**余额校验（insufficientBalance）**：

```java
// CoinRecordServiceImpl.java:366-384
private void insufficientBalance(CoinRecordVO coinRecordVO, UserCoinDTO userCoin) {
    // 只对 转出/下注/打赏 做余额校验
    if (TRANSFER_OUT / BET / SEND_REWARD) {
        BigDecimal balance = userCoinAmount.subtract(freezeAmount);  // 可用=总额-冻结
        if (balance.compareTo(amount) < 0) {
            throw new MayaDefaultException(ErrorCode.AMOUNT_NOT_ENOUGH);
        }
    }
    // 注意：结算回滚(code=6)和取消(code=5)允许余额变负！
}
```

**⚠️ 关键缺陷**：余额校验和余额扣减不是原子操作！校验时读到的余额和扣减时的余额可能不一致。虽然有Redis分布式锁保护，但锁的可靠性取决于Redis。

**SQL扣减（subBalance）**：

```xml
<!-- UserCoinMapper.xml:4-13 -->
<update id="subBalance">
    update user_coin b
    set b.amount = b.amount - #{vo.amount},
        b.updated_time = #{vo.updateTime}
    where b.id = #{vo.id}
    <!-- 结算回滚和取消不校验余额 -->
    <if test="vo.coinType != 6 and vo.coinType != 5">
        AND (b.amount - #{vo.amount}) >= 0
    </if>
</update>
```

**⚠️ 设计亮点**：SQL层面的`AND (amount - X) >= 0`是CAS防超扣保底，即使锁失效也不会扣成负数。但结算回滚(6)和取消(5)可以扣为负数——这是有意为之的业务决策。

### 4.3 批量加减款

```java
// addCoinBatch: 循环调用addCoin，有一笔失败则throw异常触发整体回滚
@Transactional
public boolean addCoinBatch(List<CoinRecordVO> coinRecordVOList) {
    for (CoinRecordVO coinRecordVO : coinRecordVOList) {
        boolean result = addCoin(coinRecordVO);
        if (!result) {
            throw new MayaDefaultException("余额处理失败");
        }
    }
    return true;
}
```

**⚠️ 问题**：addCoin内部对免转钱包调用了远程API(apiResource.amountTransfer)，这意味着**事务内包含RPC调用**——违反事务纪律！远程调用可能超时，导致事务长时间不释放。

## 五、并发控制方案

### 5.1 方案：Redisson分布式锁

该工程使用**Redis分布式锁（Redisson RLock）**作为并发控制的唯一手段。

**锁的使用模式**（所有写操作通用）：

```java
// WalletController.java:60-78 — 典型锁使用
RLock rLock = redisUtil.getLock(RedisLockKey.getAddCoinLockKey(userId));
try {
    if (rLock.tryLock(20000, 30000L, TimeUnit.MILLISECONDS)) {  // 等待20s，持有30s
        boolean result = coinRecordService.addCoin(coinRecordVO);
        // ...
    } else {
        log.info("锁超时：{}", orderNo);  // 获取锁失败只打日志
    }
} catch (Exception e) {
    log.error(e.getMessage(), e);  // 异常只打日志
} finally {
    if (rLock != null && rLock.isLocked() && rLock.isHeldByCurrentThread()) {
        rLock.unlock();
    }
}
return ResponseUtil.fail(ErrorCode.FAIL);  // 获取锁失败返回通用错误
```

### 5.2 锁的Key设计

```
钱包操作锁: "addCoin.lock.key.{userId}"   → getAddCoinLockKey()
下注操作锁: "bet.lock.key.{userId}"       → getBetLockKey()
```

**两把不同的锁**！钱包操作和下注操作使用不同的锁key。

### 5.3 锁的参数

| 参数 | 钱包操作 | 下注操作(order-service) | 说明 |
|------|---------|----------------------|------|
| waitTime | 20秒 | 10秒 | 获取锁的等待时间 |
| leaseTime | 30秒 | 10秒 | 锁的自动释放时间 |
| 锁粒度 | userId | userId | 用户级别 |

### 5.4 并发控制评价

| 评价项 | 评分 | 分析 |
|--------|------|------|
| 方案选择 | ⭐⭐ | Redis锁不如DB行锁可靠（Redis宕机=锁失效） |
| 锁粒度 | ⭐⭐⭐⭐ | userId级别，粒度合理 |
| 锁等待时间 | ⭐⭐ | 20秒太长，说明内部处理可能很慢 |
| 获取锁失败处理 | ⭐ | 只返回通用错误FAIL，调用方不知道是什么原因 |
| 两锁不一致 | ⭐ | wallet用addCoin锁，order用bet锁，不同锁不互斥！ |
| 无数据库层保底 | ⭐⭐ | subBalance有CAS(`AND amount-X>=0`)算半个保底 |

**⚠️ 严重问题**：`addCoin.lock.key`和`bet.lock.key`是两把不同的锁！这意味着wallet-service在加款的同时，order-service可以在另一把锁下直接操作user_coin表扣款，存在并发冲突风险。

## 六、冻结/解冻模型

### 6.1 三个冻结操作

```
① freezeCoin       → 冻结：freeze_amount增加，amount不变，总额不变，不产生账变
② subFreezeBalance → 解冻扣款：amount减少，freeze_amount减少，总额减少，产生账变
③ rollbackFreezeBalance → 回滚冻结：freeze_amount减少，amount不变，总额不变，不产生账变
```

### 6.2 冻结模型图示

```
初始状态: amount=1000, freeze_amount=0, 可用=1000

① freezeCoin(200):
   amount=1000, freeze_amount=200, 可用=800
   （可用=amount-freeze_amount，展示层计算）

② 两个出口:
   a) subFreezeBalance(200) 确认扣除:
      amount=800, freeze_amount=0, 可用=800（产生账变）
   b) rollbackFreezeBalance(200) 取消冻结:
      amount=1000, freeze_amount=0, 可用=1000（不产生账变）
```

### 6.3 冻结模型评价

| 评价项 | 评分 | 分析 |
|--------|------|------|
| 模型清晰度 | ⭐⭐⭐ | 三个操作职责明确 |
| 可用余额计算 | ⭐⭐ | amount-freeze_amount在应用层计算，不是DB字段 |
| 冻结记录 | ❌ | 没有独立的冻结记录表！不知道谁冻结了多少 |
| 部分解冻 | ⭐ | 无法追踪哪笔冻结对应哪笔解冻 |
| 超额冻结 | ⭐⭐ | freezeCoin有校验amount >= freezeAmount |

**⚠️ 核心缺陷**：没有freeze_record表，只有一个汇总的freeze_amount字段。无法追踪：
- 谁冻结了这笔钱（哪个订单、哪个操作）
- 冻结了多少笔、每笔多少
- 如果多笔冻结并行，部分解冻的金额对应关系

我方ser-wallet的freeze_record表设计是明确的改进点。

## 七、消息队列集成

### 7.1 RabbitMQ队列清单

该工程大量使用RabbitMQ做异步通信，钱包相关队列有4个：

| 队列名 | 监听方 | 用途 | 说明 |
|--------|-------|------|------|
| ADD_COIN_QUEUE | wallet-service | 单笔加减款 | 结算后触发 |
| ADD_COIN_ORDER_QUEUE | wallet-service | 批量扣款（下注） | 含失败回滚逻辑 |
| ADD_COIN_NOT_NOTICE_BALANCE_QUEUE | wallet-service | 批量扣款（不通知余额） | 静默扣款 |
| ADD_COIN_RECORD_AND_NOTICE_BALANCE_QUEUE | wallet-service | 只写账变+通知余额 | 不改余额 |
| EXCHANGE_NOTICE_USER_BALANCE_CHANGE | websocket-service | 余额变更通知推送 | WebSocket推送前端 |

### 7.2 MQ消息流转

```
游戏结算:
  game-service → (MQ: ADD_COIN_QUEUE) → wallet-service.AddCoinListener
    → addCoin() 加款
    → sendUserBalance() → (MQ: NOTICE_USER_BALANCE_CHANGE) → websocket推送

下注扣款:
  order-service → (MQ: ADD_COIN_ORDER_QUEUE) → wallet-service.AddCoinListener
    → deductCoinBatch() 批量扣款
    → [失败] orderResource.updateOrdersToInitialStatus() 回滚订单
    → sendUserBalance() → websocket推送
```

### 7.3 MQ可靠性机制

```
① RabbitMQ Confirm机制:
   WallRabbitTemplateConfig.java 配置了 ConfirmCallback
   → 消息发送成功: updateMessageStatus(SUCCESS)
   → 消息发送失败: updateMessageStatus(FAILURE)
   → 通过 add_coin_message 表追踪

② 问题: 只有发送确认，没有消费确认(ACK)！
   → 如果消费者处理失败，消息丢失
   → 没有死信队列(DLQ)机制
   → 没有重试机制
```

### 7.4 MQ评价

| 评价项 | 评分 | 分析 |
|--------|------|------|
| 异步解耦 | ⭐⭐⭐⭐ | 结算/下注通过MQ解耦，钱包不直接依赖游戏 |
| 消息可靠性 | ⭐⭐ | 有confirm无ACK，有消息追踪表但无重试 |
| 消息幂等 | ❌ | 消费者没有幂等处理！重复消费会重复加款 |
| 错误处理 | ⭐⭐ | 下注失败会回滚订单，但结算失败只打日志 |

### 8.3 ⚠️ order-service直接操作wallet的DB

**这是最严重的架构问题**。

order-service不通过wallet-service的API，而是**直接import了user_coin的Repository和Mapper**，自己操作user_coin表：

```java
// order-service/OrderAddCoinListener.java:434
userCoinRepository.addBalance(balanceVO);  // order-service直接加余额！

// order-service/UserCoinMapper.xml
<update id="subBalance">
    update user_coin b set b.amount = b.amount - #{vo.amount}
    where b.user_id = #{vo.id} AND (b.amount - #{vo.amount}) >= 0
</update>
```

**问题**：
1. 余额变更不经过wallet-service → 无法统一管控
2. order-service的锁key是`bet.lock.key`，wallet-service的是`addCoin.lock.key` → 两把不同的锁不互斥
3. 同一张表被两个服务直接操作 → 微服务边界被打破

```

    → [免转钱包] apiResource.amountTransfer → 三方 + insert coin_record
    → sendUserBalance → MQ → websocket推送

下注:
  游戏前端 → websocket → game-service → order-service

    → [免转钱包] apiResource.amountTransfer → 三方
    → MQ(ADD_COIN_ORDER_QUEUE) → wallet-service.deductCoinBatch
    → insert coin_record (只写流水)

结算:
  game-service.OrderSettlementServiceImpl
    → MQ(ADD_COIN_QUEUE) → wallet-service.addCoin(coinType=4, income)
    → addBalance + insert coin_record
    → sendUserBalance → websocket推送

冻结(投注前):
  order-service → wallet-service.freezeCoin
    → addFreezeBalance(freeze_amount增加)

解冻扣款(确认投注):
  order-service → wallet-service.subFreezeBalance

  game-service → MQ(GAME_CANCEL_AND_SKIP) → order-service.OrderAddCoinListener
```

## 十一、代码质量与缺陷分析

### 11.1 严重缺陷（P0级）

#### 缺陷1：order-service直接操作wallet的DB

```
位置: order-service/OrderAddCoinListener.java:434
     order-service/UserCoinMapper.xml
问题: order-service绕过wallet-service直接读写user_coin表
影响: 微服务边界被打破，并发锁不一致，无法统一审计
严重度: P0 — 架构级缺陷
```

#### 缺陷2：缺少幂等保障

```
位置: CoinRecordServiceImpl.java:229 (coinRecordRepository.insert)
问题: coin_record的order_no没有唯一索引
     MQ消费者没有幂等检查（NoticeAddCoinMessage）
影响: 重复消费会导致重复加款 → 直接资损
严重度: P0 — 资损风险
```

#### 缺陷3：事务内包含远程调用

```
位置: CoinRecordServiceImpl.java:169 (@Transactional) + :309 (apiResource.amountTransfer)
问题: addCoin方法标注@Transactional，内部调用了apiResource.amountTransfer(远程HTTP)
影响: 远程调用超时 → 事务长时间不释放 → 连接池耗尽
严重度: P0 — 稳定性风险
```

#### 缺陷4：两把不同的锁

```
位置: WalletController使用 addCoin.lock.key.{userId}
     OrderAddCoinListener使用 bet.lock.key.{userId}
问题: 两个不同的锁key不互斥
影响: wallet加款和order扣款可以同时操作同一用户的余额
严重度: P0 — 并发安全漏洞
```

### 11.2 中等缺陷（P1级）

#### 缺陷5：获取锁失败的处理过于简单

```
位置: WalletController.java:68-70, 80
问题: 获取锁失败只打日志+返回通用错误
     调用方不知道是"锁超时"还是"处理失败"
     addCoin方法的异常被catch后返回fail，吞掉了异常信息
修正: 应区分锁超时错误和业务处理错误，返回不同错误码
```

#### 缺陷6：余额校验和余额扣减不原子

```
位置: CoinRecordServiceImpl.java:370-384 (insufficientBalance)
     然后:198 (addBalance/subBalance)
问题: 先读余额做校验，再执行UPDATE，中间有窗口
保底: subBalance的SQL有CAS条件(AND amount-X>=0)，算部分保底
改进: 应该在SQL层面一步完成校验+扣减
```

#### 缺陷7：无冻结记录表

```
位置: freezeCoin/subFreezeBalance/rollbackFreezeBalance
问题: freeze_amount只是一个汇总值，无法追踪哪笔冻结对应哪笔解冻
影响: 多笔冻结并行时无法精确管理
改进: 应增加freeze_record表记录每笔冻结
```

#### 缺陷8：MQ消费无ACK确认

```
位置: AddCoinListener.java
问题: @RabbitListener默认是autoAck
     如果消费处理中异常，消息已经被ACK，不会重投
     且消费端无幂等检查
影响: 消息可能丢失或重复消费
改进: 应使用manual ACK + 幂等校验
```

### 11.3 轻微缺陷（P2级）

#### 缺陷9：大量冗余字段

```
位置: CoinRecordDTO — merchant_name, agent_name, user_name, nick_name等
问题: 流水表冗余了用户/商户/代理的名称信息
     如果用户改名，历史流水的名称不会更新
影响: 数据不一致（但流水通常不需要更新，可接受）
评价: 反范式设计，查询方便但存储浪费
```

#### 缺陷10：user_id使用String类型

```
位置: UserCoinDTO.userId — String类型
问题: String比Long多占存储空间，索引效率更低
影响: 性能损耗（小规模可忽略）
```

#### 缺陷11：缺少代码注释

```
位置: 全局
问题: addCoin方法300+行没有方法注释
     分支逻辑复杂但缺少流程说明
影响: 可维护性差
```

#### 缺陷12：coinFrom/coinTo计算时机

```
位置: CoinRecordServiceImpl.java:199-211
问题: coinTo=当前余额+/-变动金额，但这个"当前余额"是addBalance/subBalance执行前读到的
     如果在锁外有并发，这个值可能不准确
保底: 有Redis锁保护，正常情况下准确
```

## 十三、借鉴清单（取其精华）

### ✅ 借鉴1：SQL层CAS保底机制

```
参考工程做法:
  UPDATE user_coin SET amount = amount - X WHERE id = ? AND (amount - X) >= 0
  → 即使应用层锁失效，SQL层面也不会扣成负数

我方借鉴:
  在ledger层的Debit方法中，除了SELECT FOR UPDATE，
  UPDATE语句也加上 AND balance >= amount 的CAS条件
  → 双重保障：行锁是主力，CAS是保底

实现:
  UPDATE user_wallet
  SET balance = balance - #{amount}
  WHERE id = #{id} AND balance >= #{amount}
  → 如果RowsAffected=0则说明余额不足或被并发扣走
```

**价值**：这是一个低成本高收益的防护措施，即使在极端情况下（行锁被绕过、事务隔离级别降低）也不会超扣。

### ✅ 借鉴2：可用余额=总额-冻结（展示层计算）

```
参考工程做法:
  amount字段存储总余额（含冻结部分）
  freeze_amount字段存储冻结部分
  可用余额 = amount - freeze_amount（在getWallet时计算返回）

我方可考虑:
  user_wallet表保持 balance + frozen_amount 两个字段
  查询时返回 available = balance - frozen_amount
  → 冻结时只增加frozen_amount，不动balance
  → 好处：解冻回滚时只减frozen_amount，不改balance（原子性更好）
```

**注意**：需要和产品确认"余额"的定义——是总余额还是可用余额。

### ✅ 借鉴3：余额变更后WebSocket推送

```
参考工程做法:
  addCoin成功后 → sendUserBalance()
  → RabbitMQ → EXCHANGE_NOTICE_USER_BALANCE_CHANGE
  → websocket-service → 前端实时更新

我方借鉴:
  虽然我方不用RabbitMQ，但"余额变更后主动推送前端"是好的用户体验
  可以在RPC响应中带上最新余额，或者通过现有WebSocket通道推送
```

### ✅ 借鉴4：特殊业务场景允许余额为负

```
参考工程做法:
  结算回滚(coinType=6)和取消(coinType=5)的subBalance不校验>=0
  → 允许余额变负（因为是修正操作，必须执行）

我方借鉴:
  手动调账(管理员操作)也可能需要余额变负的能力
  应在ledger层留一个 force=true 的开关
  → 正常业务：force=false，严格校验余额
  → 管理员调账/结算修正：force=true，允许负余额
```

### ✅ 借鉴5：新用户首笔必须是入账

```
参考工程做法:
  checkCoinChange() — 如果用户没有钱包记录且是支出操作 → 报错

我方借鉴:
  用户钱包不存在时应该自动创建（首次充值时）
  但确实应该校验首笔不能是扣款
  可在 initWallet() 时同步创建
```

## 十四、规避清单（去其糟粕）

### ❌ 规避1：跨服务直接操作DB

```
参考工程问题:
  order-service直接import了UserCoinRepository和UserCoinMapper.xml
  → 绕过wallet-service直接读写user_coin表

我方必须规避:
  所有余额变更必须经过ser-wallet的RPC接口
  → 投注扣款: 投注服务 → RPC WalletDebit → ser-wallet.ledger.Debit
  → 结算加款: 结算服务 → RPC WalletCredit → ser-wallet.ledger.Credit
  → 绝不允许其他服务直接操作user_wallet表
```

### ❌ 规避2：使用Redis分布式锁管理余额

```
参考工程问题:
  依赖Redisson RLock做余额并发控制
  → Redis宕机/主从切换 → 锁失效 → 资损

我方已规避:
  使用DB行锁(SELECT FOR UPDATE)
  → 与事务绑定，不依赖外部组件
  → 只要数据库正常，锁就不会失效
```

### ❌ 规避3：无幂等保障

```
参考工程问题:
  coin_record.order_no无唯一索引
  MQ消费者无幂等检查
  → 重复消费/重试 → 重复加款

我方已规避:
  wallet_flow表 uk_order_wallet_flow(order_no, wallet_type, flow_type, delete_at)
  → 先INSERT流水（唯一约束拦截重复）→ 再UPDATE余额
```

### ❌ 规避4：事务内远程调用

```
参考工程问题:
  @Transactional标注的addCoin方法内调用apiResource.amountTransfer()
  → 远程HTTP超时 → 事务长时间不释放 → 连接池耗尽

我方已规避:
  事务纪律：事务内禁止RPC调用
  → RPC在事务外部执行
  → 事务内只做DB操作(锁行→校验→改余额→写流水)
```

### ❌ 规避5：无冻结记录追踪

```
参考工程问题:
  只有一个freeze_amount汇总字段，无法追踪每笔冻结

我方已规避:
  freeze_record表独立记录每笔冻结
  → 可追踪：谁冻结的、冻结多少、什么时候解冻
  → 支持部分解冻、超时自动解冻等高级场景
```

### ❌ 规避6：MQ消费无幂等无重试

```
参考工程问题:
  RabbitMQ消费者autoAck，无幂等校验，无死信队列
  → 消息丢失或重复消费都不可恢复

我方已规避:
  不引入MQ → 同步RPC + 幂等接口 + 对账机制
  → 更简单可靠，出问题可追踪
```

### 🔄 调整1：金额存储策略

```
参考工程: BigDecimal(DECIMAL) → Java原生精确
我方现设计: BIGINT(最小货币单位整数) → 零精度风险

  但要注意：不同币种的"最小单位"不同

  → VND/IDR: 1:1（本身就是整数）
  → THB: 1e-2 → *100
```

```
我方设计: 需要根据自身业务定义
```

### 🔄 调整3：流水金额字段设计

```
参考工程: coin_value(绝对值) + balance_type(1收入/2支出)
我方设计: amount(正/负数，负=扣减)

保持我方设计:
  正/负数更直观：SUM(amount) = 余额变动总和
  不需要额外的balance_type字段
  计算更简单：balance_after = balance_before + amount

但借鉴一个点:
  参考工程有coin_amount字段（当前总金额），我方不需要
  我方的balance_after已经满足此需求
```

### 🔄 调整4：添加CAS保底到ledger层

```
我方原设计:
  SELECT FOR UPDATE → 读余额 → 校验 → UPDATE balance = newBalance

优化后:
  SELECT FOR UPDATE → 读余额 → 校验 → UPDATE balance = balance - X WHERE balance >= X
  → RowsAffected == 0 → 说明被并发扣走 → return error
  → 双重保障

具体代码调整:
  // 原来
  tx.Model(&wallet).Update("balance", wallet.Balance - param.Amount)

  // 优化
  result := tx.Model(&wallet).
      Where("balance >= ?", param.Amount).
      Update("balance", gorm.Expr("balance - ?", param.Amount))
  if result.RowsAffected == 0 {
      return errs.BalanceInsufficient
  }
```

### 🔄 调整5：余额变更通知机制

```
参考工程: MQ → WebSocket推送
我方调整: 由于不引入MQ，考虑以下方案：

方案A: RPC响应携带最新余额
  → WalletDebit/Credit的Response中直接返回最新余额
  → 调用方（投注/结算服务）拿到后通过WebSocket推送

方案B: 钱包服务内调用推送服务
  → ledger操作完成后，主动调用WebSocket服务的RPC
  → 推送余额变更消息给前端

方案C: 定时轮询（不推荐）
  → 前端定时查询余额
  → 实时性差，浪费资源

建议: 方案A最简单，先用A，后续如果需要实时性再升级到B
```
