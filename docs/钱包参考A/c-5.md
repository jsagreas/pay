# 二、数据模型 — 表结构、字段语义、索引推断

## 2.1 user_coin 表（钱包主表）

```sql
-- 从 UserCoinDTO.java + UserCoinMapper.xml 推断
CREATE TABLE user_coin (
    id            BIGINT AUTO_INCREMENT PRIMARY KEY,   -- 继承自 BasicDataBaseDTO
    user_id       VARCHAR(64)    NOT NULL,              -- 玩家 ID（业务主键）
    user_name     VARCHAR(64),                          -- 玩家名称
    amount        DECIMAL(18,2)  NOT NULL DEFAULT 0,    -- 总金额（含冻结）
    freeze_amount DECIMAL(18,2)  NOT NULL DEFAULT 0,    -- 冻结金额
    user_state    INT,                                  -- 玩家游戏状态
    game_type     INT,                                  -- 当前游戏类型
    wallet_type   TINYINT        NOT NULL,              -- 1=转账钱包, 2=免转钱包
    currency      VARCHAR(16),                          -- 币种（CNY/USD/TWD/INR/BRL）
    remark        VARCHAR(255),                         -- 备注
    creator       BIGINT,                               -- 创建人
    updater       BIGINT,                               -- 更新人
    created_time  DATETIME       NOT NULL,              -- 创建时间
    updated_time  DATETIME                              -- 更新时间
);
```

**关键特征**：
- **单用户单钱包**：`user_id` 是唯一业务键，每个用户只有一个钱包行
- **无多币种隔离**：`currency` 字段存在但没有作为钱包隔离维度，所有币种共用一个 amount
- **无版本号字段**：没有 `version` 或 `updated_at` 用于乐观锁
- **BigDecimal 存储**：使用 DECIMAL 类型，非整数存储

# 三、余额模型 — amount / freezeAmount 的语义与数学关系

## 3.1 核心语义

```
参考工程的余额模型：

  amount       = 总金额（包含冻结部分）
  freezeAmount = 已冻结金额
  可用余额      = amount - freezeAmount

  ┌─────────────────────────────────────────┐
  │              amount（总金额）              │
  │  ┌─────────────────┬──────────────────┐  │
  │  │   可用余额        │  freezeAmount    │  │
  │  │ (amount-freeze)  │   （冻结）        │  │
  │  └─────────────────┴──────────────────┘  │
  └─────────────────────────────────────────┘
```

**关键源码证据**（UserCoinServiceImpl.java → getWallet）：

```java
// 查询返回可用余额时：减去冻结金额
if (userCoinDTO1.getFreezeAmount() != null) {
    userCoinVO1.setAmount(userCoinDTO1.getAmount().subtract(userCoinDTO1.getFreezeAmount()));
}
```

## 3.2 与我们设计的对比

```
我们的余额模型：

  available    = 可用余额（直接存储，直接使用）
  frozen       = 冻结金额
  总金额       = available + frozen

  ┌─────────────────────────────────────────┐
  │           总金额 = available + frozen     │
  │  ┌─────────────────┬──────────────────┐  │
  │  │   available      │    frozen        │  │
  │  │ （可直接使用）     │   （冻结）        │  │
  │  └─────────────────┴──────────────────┘  │
  └─────────────────────────────────────────┘
```

## 3.3 语义差异影响分析

| 操作 | 参考工程 | 我们的方案 |
|------|---------|----------|
| 充值入账 | amount += X | available += X |
| 普通扣款 | amount -= X (需检查 amount-freeze >= X) | available -= X (WHERE available >= X) |
| 冻结 | freezeAmount += X (amount 不变) | available -= X, frozen += X |
| 冻结扣减（确认） | amount -= X, freezeAmount -= X | frozen -= X |
| 冻结回滚 | freezeAmount -= X (amount 不变) | frozen -= X, available += X |
| 查询可用余额 | 需要**运算** (amount - freezeAmount) | **直接读取** available |

**核心差异**：
- 参考工程：冻结时 `amount` 不变，可用余额需要实时计算
- 我们：冻结时 `available` 立即减少，可用余额直接读取

**判定：调整**
> 我们的模型更优。原因：
> 1. 查询频率远高于写入频率，available 直接可读避免每次都做减法运算
> 2. SQL 层面的并发安全更简单：`WHERE available >= X` 直接判断，无需 `WHERE (amount - freeze_amount) >= X`
> 3. 对账公式更直观：`total = available + frozen`，天然恒等

# 四、五大 SQL 操作 — 余额变动的原子单元

参考工程在 `UserCoinMapper.xml` 中定义了 5 个原子 SQL 操作，这是整个钱包系统的基石。

## 4.1 subBalance — 扣减余额

```xml
<update id="subBalance">
    update user_coin b
    set b.amount = b.amount - #{vo.amount},
        b.updated_time = #{vo.updateTime}
    where b.id = #{vo.id}
    <!-- 结算回滚(6)和取消(5)允许扣为负数 -->
    <if test=" vo.coinType != 6 and vo.coinType != 5 ">
        AND (b.amount - #{vo.amount}) >= 0
    </if>
</update>
```

**分析**：
- 通过 `WHERE (amount - X) >= 0` 防止余额扣为负数 — 这是一种**行级 CAS 保护**
- 但特殊场景（结算回滚 coinType=6、取消 coinType=5）跳过此检查，允许负余额
- **问题**：这里的 CAS 检查的是 `amount`（总金额），不是可用余额。如果有冻结金额，理论上可以扣到可用余额为负但总金额仍 >= 0 的状态
- 返回 `int`（受影响行数），调用方通过 `i == 0` 判断余额不足

**评价**：SQL 级防超扣的思路正确，但 **只检查 amount 不检查 available 是缺陷**。

## 4.2 addBalance — 增加余额

```xml
<update id="addBalance">
    update user_coin b
    set b.amount       = b.amount + #{vo.amount},
        b.updated_time = #{vo.updateTime}
    where b.id = #{vo.id}
</update>
```

**分析**：
- 无条件加钱，无额外检查 — 合理，加钱不需要余额校验
- 通过 `WHERE id = ?` 定位行

## 4.3 addFreezeBalance — 冻结（增加冻结金额）

```xml
<update id="addFreezeBalance">
    update user_coin b
    set b.freeze_amount = b.freeze_amount + #{vo.amount},
        b.updated_time  = #{vo.updateTime}
    where b.id = #{vo.id}
</update>
```

**分析**：
- 仅增加 `freeze_amount`，`amount` 不变
- **无任何条件检查** — 没有 `WHERE (amount - freeze_amount) >= X`
- 前置检查在 Java 代码中做（`freezeCoin` 方法），但两步不原子

## 4.4 subFreezeBalance — 冻结扣减（确认提现/扣款）

```xml
<update id="subFreezeBalance">
    update user_coin b
    set b.amount        = b.amount - #{vo.amount},
        b.freeze_amount = b.freeze_amount - #{vo.amount},
        b.updated_time  = #{vo.updateTime}
    where b.id = #{vo.id}
</update>
```

**分析**：
- 同时减少 `amount` 和 `freeze_amount` — 确认冻结后真正扣款
- 语义：钱从"冻结态"变为"已扣除"
- **无条件检查**：不检查 `freeze_amount >= X`，依赖前置校验

## 4.5 rollbackFreezeBalance — 冻结回滚

```xml
<update id="rollbackFreezeBalance">
    update user_coin b
    set b.freeze_amount = b.freeze_amount - #{vo.amount},
        b.updated_time  = #{vo.updateTime}
    where b.id = #{vo.id}
</update>
```

**分析**：
- 仅减少 `freeze_amount`，`amount` 不变 — 冻结金额释放回可用
- **无条件检查**

## 4.6 五操作汇总与对比

```
┌────────────────────┬──────────────┬───────────────────┬────────────────┐
│ 操作                │ amount 变化   │ freeze_amount 变化 │ SQL 级安全检查  │
├────────────────────┼──────────────┼───────────────────┼────────────────┤
│ subBalance         │ -= X         │ 不变               │ 有(条件性)      │
│ addBalance         │ += X         │ 不变               │ 无              │
│ addFreezeBalance   │ 不变          │ += X              │ 无              │
│ subFreezeBalance   │ -= X         │ -= X              │ 无              │
│ rollbackFreeze     │ 不变          │ -= X              │ 无              │
└────────────────────┴──────────────┴───────────────────┴────────────────┘
```

**判定：调整**
> SQL 层面的防超扣思路值得借鉴（subBalance 中的 `WHERE (amount - X) >= 0`），但：
> 1. 安全检查应覆盖所有写操作，而非仅 subBalance
> 2. 冻结操作应在 SQL 级检查 `(amount - freeze_amount) >= X`，而非依赖应用层
> 3. 我们的方案通过乐观锁 `WHERE available >= X AND version = V` 更彻底

# 五、核心业务链路 — addCoin 全流程深度拆解

`addCoin` 是参考工程的核心方法，所有余额变动都通过此方法。约 120 行，是整个钱包的心脏。

## 5.1 调用入口

```
HTTP POST /webapi/wallet/addCoin
    → WalletController.addCoin()
        → 获取 Redisson 分布式锁（userId 维度）
        → CoinRecordServiceImpl.addCoin()
        → sendUserBalance()（MQ 通知余额变更）
```

## 5.2 addCoin 内部分支逻辑

```
addCoin(CoinRecordVO)
│
├── 校验 coinType > 0
├── 查询用户信息 → commonService.getUserInfoByUserId()
│
├── 分支 1：转账钱包 (walletType == 1)
│   ├── 查询 user_coin 记录
│   ├── 若存在：
│   │   ├── insufficientBalance() — 验证可用余额
│   │   ├── 收入(balanceType=1) → addBalance SQL
│   │   ├── 支出(balanceType=2) → subBalance SQL (返回0则抛异常)
│   │   ├── 记录 coinFrom / coinTo
│   │   └── 插入 coin_record 流水
│   └── 若不存在：
│       ├── checkCoinChange() — 新钱包不能首次就支出
│       ├── 创建 user_coin 记录（INSERT）
│       └── 插入 coin_record 流水
│
└── 分支 2：免转钱包 (walletType == 2)
    ├── 调用第三方 API → apiResource.amountTransfer()
    ├── 成功 → 记录三方订单 + 插入 coin_record
    └── 失败 → 记录失败状态 + return false
```

## 5.3 关键代码片段注解

### 5.3.1 余额不足校验（insufficientBalance）

```java
private void insufficientBalance(CoinRecordVO coinRecordVO, UserCoinDTO userCoin) {
    // 仅对特定操作校验：转出/下注/打赏
    if (WalletUtilEnum.CoinChangeEnum.TRANSFER_OUT.getCode().equals(coinRecordVO.getCoinType())
            || WalletUtilEnum.CoinChangeEnum.BET.getCode().equals(coinRecordVO.getCoinType())
            || WalletUtilEnum.CoinChangeEnum.SEND_REWARD.getCode().equals(coinRecordVO.getCoinType())) {
        BigDecimal amount = coinRecordVO.getCoinValue();
        BigDecimal userCoinAmount = userCoin.getAmount();
        BigDecimal freezeAmount = userCoin.getFreezeAmount();
        BigDecimal balance = userCoinAmount.subtract(freezeAmount);  // 可用余额
        if (balance.compareTo(amount) < 0) {
            throw new MayaDefaultException(ErrorCode.AMOUNT_NOT_ENOUGH);
        }
    }
}
```

**问题**：
1. 先读后写不原子 — 读取 amount 和 freeze_amount 后，到 subBalance 执行前可能已被其他请求修改
2. 依赖分布式锁来保护这个间隙 — 如果锁失效，就有超扣风险
3. 仅对 3 种 coinType 校验，其他支出场景（如结算回滚）不校验 — 有意为之（允许负余额）

### 5.3.2 新用户钱包创建

```java
// 用户无 user_coin 记录时：
```

**问题**：
1. **非原子创建** — `selectOne` + `insert` 不是原子操作，并发时可能重复创建
2. 虽然有分布式锁保护，但锁粒度是 userId，理论上同一用户的并发请求被串行化
3. **未使用 INSERT ON DUPLICATE KEY UPDATE** — 如果锁失效，无数据库级兜底

# 六、冻结模型 — 三阶段生命周期

## 6.1 冻结三阶段

```
阶段 1：冻结（freezeCoin）
  amount 不变，freezeAmount += X
  可用余额 = amount - freezeAmount（减少了 X）
  不产生账变记录

阶段 2a：确认扣减（subFreezeBalance）
  amount -= X，freezeAmount -= X
  可用余额 = amount - freezeAmount（不变）
  产生账变记录

阶段 2b：回滚（rollbackFreezeBalance）
  amount 不变，freezeAmount -= X
  可用余额 = amount - freezeAmount（增加了 X）
  不产生账变记录
```

## 6.2 冻结前校验

```java
// CoinRecordServiceImpl.freezeCoin()
if (userCoinDTO.getAmount().compareTo(coinFreezeVO.getFreezeAmount()) < 0) {
    return ErrorCode.USER_WALLET_NOT_ENOUGH_BALANCE;
}
```

**严重缺陷**：检查的是 `amount >= freezeAmount`，但应该检查 `(amount - 已冻结的freezeAmount) >= 新冻结金额`。

**举例说明**：
```
初始状态：amount = 1000, freezeAmount = 800, 可用 = 200
请求冻结 500
当前检查：1000 >= 500 ✓ — 允许冻结！
冻结后：amount = 1000, freezeAmount = 1300, 可用 = -300 ← 负数！
```

这意味着冻结操作可以把可用余额冻成负数。在有分布式锁保护的情况下不太可能并发触发，但 **校验逻辑本身就是错的**。

**判定：规避**
> 我们的方案中，冻结检查的是 `WHERE available >= X`，直接在 SQL 级别保证不会冻成负数。

## 6.3 解冻扣减前校验

```java
// CoinRecordServiceImpl.subFreezeBalance()
if (userCoinDTO.getFreezeAmount().compareTo(coinFreezeRecordVO.getFreezeAmount()) < 0) {
    return ErrorCode.USER_WALLET_NOT_ENOUGH_BALANCE;
}
```

**分析**：检查 `freezeAmount >= 要扣减的冻结金额`，逻辑正确。但同样是应用层检查，非 SQL 原子操作。

## 6.4 与我们方案对比

| 阶段 | 参考工程 | 我们的方案 |
|------|---------|----------|
| 冻结 | freezeAmount += X | available -= X, frozen += X |
| 冻结校验 | Java: amount >= X（有 Bug） | SQL: WHERE available >= X（原子） |
| 确认 | amount -= X, freezeAmount -= X | frozen -= X |
| 确认校验 | Java: freezeAmount >= X | SQL: WHERE frozen >= X（原子） |
| 回滚 | freezeAmount -= X | frozen -= X, available += X |
| 流水 | 仅确认产生，冻结/回滚不产生 | 全部产生（可配） |

# 七、并发控制 — Redisson 分布式锁方案剖析

## 7.1 锁的使用模式

所有写操作（WalletController 中）都使用同一个锁模式：

```java
RLock rLock = redisUtil.getLock(RedisLockKey.getAddCoinLockKey(userId));
try {
    if (rLock.tryLock(20000, 30000L, TimeUnit.MILLISECONDS)) {
        // 业务逻辑
    } else {
        log.info("锁超时：{}", orderNo);
    }
} catch (Exception e) {
    log.error(e.getMessage(), e);
} finally {
    if (Objects.nonNull(rLock) && rLock.isLocked() && rLock.isHeldByCurrentThread()) {
        rLock.unlock();
    }
}
```

**锁参数**：
- **等待超时**：20 秒 — 获取锁最多等 20 秒
- **持有超时**：30 秒 — 锁自动释放时间
- **锁粒度**：userId — 同一用户的所有写操作串行化

## 7.2 分布式锁 vs 乐观锁 深度对比

```
┌────────────────────┬──────────────────────┬──────────────────────┐
│ 维度                │ 分布式锁（参考工程）    │ 乐观锁（我们的方案）   │
├────────────────────┼──────────────────────┼──────────────────────┤
│ 实现方式            │ Redis Redisson       │ DB CAS (version)     │
│ 锁粒度              │ userId               │ 余额行               │
│ 并发串行化          │ 是（完全串行）         │ 否（失败重试）         │
│ 性能               │ 低（串行等待20s）      │ 高（并行尝试）         │
│ 强依赖              │ Redis 可用性          │ 无额外依赖            │
│ Redis 宕机影响      │ 锁失效=超扣风险       │ 无影响                │
│ 死锁风险            │ 有（超时兜底30s）      │ 无                    │
│ 代码侵入性          │ 高（每个Controller）   │ 低（统一在仓储层）     │
│ 多用户并发          │ 不同用户不阻塞         │ 不同用户不阻塞         │
│ 同用户并发          │ 完全串行              │ CAS 竞争，失败重试     │
│ 单点故障            │ Redis 主从延迟=锁丢失  │ 数据库自身保障         │
│ 适用场景            │ 业务逻辑复杂，需长事务  │ 短事务，高吞吐         │
└────────────────────┴──────────────────────┴──────────────────────┘
```

## 7.3 参考工程的锁使用问题

### 问题 1：锁与事务边界不匹配

```java
// Controller 持锁
if (rLock.tryLock(...)) {
    boolean result = coinRecordService.addCoin(coinRecordVO);  // @Transactional
    if (result) {
        coinRecordService.sendUserBalance(coinRecordVO.getUserId());
    }
}
// finally unlock

// 时序问题：
// 1. 获取锁 ✓
// 2. @Transactional 开始
// 3. SQL 执行
// 4. @Transactional 提交
// 5. sendUserBalance (MQ)
// 6. 释放锁
//
// 如果步骤 4 之后、步骤 6 之前崩溃？
// → 事务已提交但锁未释放（Redis TTL 30s 后自动释放）
// → 影响不大，30s 后恢复
```

### 问题 2：获取锁失败时的处理

```java
if (rLock.tryLock(20000, 30000L, TimeUnit.MILLISECONDS)) {
    // 成功
} else {
    log.info("锁超时：{}", coinRecordVO.getOrderNo());
}
// 获取锁失败 → 仅打日志 → 返回 ResponseUtil.fail(ErrorCode.FAIL)
// 调用方无法区分是"锁超时"还是"业务失败"
```

### 问题 3：order-service 直接操作 user_coin 表

order-service 有自己的 `UserCoinMapper.xml`，直接对 `user_coin` 表执行 `subBalance` 和 `addBalance`。但 order-service 中**没有使用分布式锁**，这意味着 wallet-service 和 order-service 可能同时修改同一行数据，分布式锁形同虚设。

**判定：规避**
> 我们的方案：
> 1. 使用乐观锁（CAS），无 Redis 依赖
> 2. 安全性在 SQL 层面保证，无论从哪个服务操作都有效
> 3. 余额操作只在 wallet 服务内部，不暴露直接 SQL 给其他服务

## 8.1 队列定义

## 8.2 ADD_COIN_QUEUE 消费逻辑

```java
@RabbitListener(queues = {MqConstants.ADD_COIN_QUEUE})
public void NoticeAddCoinMessage(String jsonStr) {
    CoinRecordVO coinRecordVO = JSONObject.parseObject(jsonStr, CoinRecordVO.class);
    if (coinRecordVO.getCoinValue().compareTo(BigDecimal.ZERO) == 0) return;

    RLock rLock = redisUtil.getLock(RedisLockKey.getAddCoinLockKey(coinRecordVO.getUserId()));
    try {
        if (rLock.tryLock(20000, 30000L, TimeUnit.MILLISECONDS)) {
            boolean isSuc = coinRecordService.addCoin(coinRecordVO);
            if (isSuc) {
                coinRecordService.sendUserBalance(coinRecordVO.getUserId());
            }
        }
    } finally { unlock(); }
}
```

## 8.3 ADD_COIN_ORDER_QUEUE 消费逻辑（批量注单）

```java
@RabbitListener(queues = {MqConstants.ADD_COIN_ORDER_QUEUE})
public void noticeDeductCoinOrderMessage(String jsonStr) {
    List<CoinRecordVO> coinRecordVOList = JSON.parseArray(jsonStr, CoinRecordVO.class);
    try {
        boolean result = coinRecordService.deductCoinBatch(coinRecordVOList);
        if (!result) {
            // 扣款失败 → 回滚订单状态
            orderResource.updateOrdersToInitialStatus(orderNos);
        }
        coinRecordService.sendUserBalance(userId);
    } catch (Exception e) {
        // 异常 → 也回滚订单状态
        orderResource.updateOrdersToInitialStatus(orderNos);
    }
}
```

**注意**：此队列**不持锁**。`deductCoinBatch` 只是写账变记录（`deductCoin`），不修改余额。

## 8.4 余额变更通知

```java
// 通过 MQ 通知前端余额变化
public void sendUserBalance(String userId) {
    UserBalanceUpdateVO vo = new UserBalanceUpdateVO();
    vo.setUserId(userId);
    Message message = new Message(objectMapper.writeValueAsBytes(vo), messageProperties);
    rabbitTemplate.convertAndSend(MqConstants.EXCHANGE_NOTICE_USER_BALANCE_CHANGE, "", message);
}
```

**用途**：通知 WebSocket 网关推送余额变更给前端页面实时更新。

## 8.5 MQ 架构评价

| 维度 | 评价 |
|------|------|
| 解耦 | ✅ 余额变动与通知解耦，合理 |
| 可靠性 | ⚠️ 消息消费失败无死信队列处理 |
| 幂等 | ❌ MQ 消费无幂等保护，重复消费 = 重复加减钱 |
| 事务一致性 | ⚠️ 非事务消息，DB 提交后 MQ 可能发送失败 |
| 监控 | ⚠️ 仅日志，无消费失败告警 |

**判定：借鉴思路，调整实现**
> MQ 异步解耦的架构思路值得借鉴（尤其是余额变更通知）。但：
> 1. 我们暂不引入 MQ，用同步 RPC 保证一致性
> 2. 余额通知可以考虑 Redis Pub/Sub 或 Kafka，而非 RabbitMQ
> 3. 如果引入 MQ，必须保证消费幂等（利用 coin_record 的唯一键）

```java
// 用户信息：Redis 缓存（但 getUserInfoByUserId 中没有设置 TTL！）
public UserInfoVO getUserInfoByUserId(String userId) {
```

## 11.2 逐项详解

### ① 并发安全（6/10）

**实现**：Redisson 分布式锁 + subBalance 的 `WHERE (amount - X) >= 0`

**优点**：
- 双层保护：应用层锁 + SQL 级条件
- 同用户串行化，彻底避免并发冲突

**不足**：
- Redis 宕机 = 锁失效 = 仅靠 SQL 条件兜底（此时只有 subBalance 一个操作有 SQL 级保护）
- SQL 条件只检查 amount（总金额），不检查可用余额
- order-service 绕过分布式锁直接操作 user_coin 表

### ② 幂等性（2/10）

**实现**：几乎没有幂等保护

**证据**：
- `coin_record` 表的 `order_no` 没有唯一索引
- 同一 orderNo 可以多次调用 addCoin，每次都会成功插入流水并修改余额
- MQ 消息重复消费 = 重复加减钱
- 无请求幂等 token 或业务去重逻辑

**风险**：**极高** — 网络重试、MQ 重投都会导致重复扣款/加款

### ③ 事务一致性（7/10）

**实现**：`@Transactional(rollbackFor = Exception.class)`

**优点**：
- addCoin 中余额变动和流水记录在同一事务内
- 事务失败自动回滚

**不足**：
- 部分方法（rollbackFreezeBalance）没有 @Transactional
- sendUserBalance（MQ 通知）在事务外，事务提交后 MQ 发送失败 = 通知丢失

### ④ 分布式一致性（3/10）

**实现**：MQ 异步 + 失败回滚订单状态

**证据**（AddCoinListener）：
```java
boolean result = coinRecordService.deductCoinBatch(coinRecordVOList);
if (!result) {
    orderResource.updateOrdersToInitialStatus(collect);  // 回滚
}
```

**不足**：
- 回滚操作（updateOrdersToInitialStatus）本身也可能失败 — 无重试/补偿
- 无 saga 模式、无本地消息表、无事务消息
- MQ 消息丢失 = 业务丢失

### ⑤ 金额精度（5/10）

**实现**：Java BigDecimal

**优点**：BigDecimal 比 float/double 精确得多

**不足**：
- 未指定精度（scale）和舍入模式（RoundingMode）
- BigDecimal 运算需要显式指定 `setScale(2, RoundingMode.HALF_UP)`，源码中多处使用 `add`/`subtract` 未指定
- DECIMAL 存储在 MySQL 中虽然精确，但应用层运算仍有隐患

### ⑥ 整数除法/汇率（0/10）

未涉及汇率计算。DashboardCurrencyEnum 中有多币种（CNY/USD/TWD/INR/BRL），但只用于报表展示切换，非实际汇率换算。

### ⑦ 流水连续性/对账（6/10）

**优点**：
- coin_record 记录 coinFrom/coinTo/coinAmount，具备单笔追溯能力
- 宽表设计利于后台查询对账

**不足**：
- 无自动对账机制（daily balance reconciliation）
- coinFrom/coinTo 是应用层填入而非 DB 层保证，有被篡改/错填风险
- 免转钱包的 coinFrom/coinTo 被设为 null，断链

### ⑧ 状态机正确性（3/10）

**实现**：无显式状态机

**分析**：余额操作的状态流转完全隐含在代码逻辑中，无状态枚举、无状态转移校验、无非法状态检测。

### ⑨ 缓存一致性（4/10）

**实现**：商户信息 Redis 缓存 1 天，用户信息缓存有 Bug

**缺陷**：
- `getUserInfoByUserId` 只读缓存不写缓存
- 商户信息缓存无失效机制（商户修改后最长 1 天才更新）
- 无缓存穿透/击穿/雪崩保护

### ⑩ 懒创建（5/10）

**实现**：`selectOne → if null → insert`，依赖分布式锁防并发

**不足**：
- 非原子操作
- 未使用 INSERT ON DUPLICATE KEY UPDATE
- 如果分布式锁失效，并发创建可能产生重复行（取决于 userId 是否有唯一索引）

### ⑪ 热点行/性能（5/10）

**分析**：分布式锁串行化同一用户的操作，自然避免了热点行竞争。但代价是：
- 同用户请求排队等待（最长 20 秒）
- 高并发场景（如赛事结算批量操作同一用户）性能瓶颈

### ⑫ 敏感数据安全（2/10）

**未见任何安全措施**：
- 用户余额明文存储
- 账变金额明文传输
- 日志中打印完整金额和用户信息（`log.info("coinRecordVO:{}", JSON.toJSONString(coinRecordVO))`）
- 唯一安全机制：getWallet 对 API 来源的非正式用户返回 0（防刷）

### ⑬ 错误处理（4/10）

**模式**：catch Exception → log.error → return false / throw MayaDefaultException

**不足**：
- 大量 `catch (Exception e)` 粗粒度捕获
- 部分异常被吞掉（`catch (Exception e) { log.error(...);}` 无 rethrow）
- MQ 消费失败无死信队列，消息直接丢失
- 错误信息不够结构化，调用方难以区分失败原因

### ⑭ 统一抽象（3/10）

**现状**：
- `addCoin` 方法 ~120 行，承担了余额校验、余额变动、流水记录、新钱包创建、免转钱包三方调用等多重职责
- 转账钱包和免转钱包在同一方法中 if-else 分支
- 无统一的 "余额操作" 抽象，每种操作各自实现

# 十二、关键缺陷与风险 — 深层问题清单

## 12.1 致命级问题（生产出事就是资损）

### F1：无幂等保护

```
严重度：★★★★★
影响：重复扣款/加款
场景：
  1. HTTP 超时重试 → addCoin 被调两次
  2. MQ 重投（网络抖动） → NoticeAddCoinMessage 消费两次
  3. 手动重发（运维操作）→ 同一笔订单多次处理
根因：coin_record.order_no 无唯一索引，addCoin 无幂等检查
```

### F2：冻结校验 Bug

```
严重度：★★★★☆
影响：可用余额可被冻成负数
场景：amount=1000, freezeAmount=800, 请求冻结 500
      检查 1000 >= 500 通过 → 冻结后 freezeAmount=1300, 可用=-300
根因：freezeCoin 检查的是 amount >= X，应该检查 (amount - freezeAmount) >= X
```

### F3：order-service 直接操作 user_coin

```
严重度：★★★★☆
影响：绕过分布式锁的并发修改
场景：wallet-service 和 order-service 同时对同一 userId 的 user_coin 执行 SQL
      wallet-service 持有锁，order-service 不持有锁 → 数据不一致
根因：user_coin 表被多个服务直接操作，违反领域边界
```

## 12.2 高危级问题

### H1：MQ 消费无幂等

```
严重度：★★★☆☆
影响：RabbitMQ 重投 → 余额重复加减
场景：消费成功但 ACK 未返回 → RabbitMQ 重投 → 消费者再次处理
```

### H2：SQL 注入风险

```
严重度：★★★☆☆
影响：潜在 SQL 注入
位置：CoinRecordMapper.xml → getCoinRecordPage
      ORDER BY s.created_time ${vo.createdTimeBy}, s.id ${vo.createdTimeBy}
      使用 ${} 而非 #{}，createdTimeBy 直接拼接进 SQL
```

### H3：用户信息缓存失效

```
严重度：★★☆☆☆
影响：每次 addCoin 都调用 user-service RPC
位置：CommonServiceImpl.getUserInfoByUserId
根因：读取缓存但不写回缓存
```

## 12.3 中危级问题

| 编号 | 问题 | 影响 |
|------|------|------|
| M1 | addCoinList 锁粒度错误 | 只锁第一个 userId，多用户操作不安全 |
| M2 | rollbackFreezeBalance 无 @Transactional | 但只有一个 SQL 操作，影响有限 |
| M3 | 日志明文打印用户余额 | 安全审计风险 |
| M4 | CoinRecordMapper.xml 中 selectRecentWithDraw 在 Repository 中无对应方法 | 死代码 |
| M5 | BigDecimal 运算未指定精度 | 边界场景可能产生非预期精度 |

# 十三、取舍对照表 — 借鉴 / 调整 / 规避 完整决策

## 13.1 借鉴项（直接可用或微调即可）

### B1：账变流水的 coinFrom / coinTo 设计

```
决策：借鉴 ✅
原因：记录变动前后余额，提供审计追溯能力，对账时可以验证流水连续性
调整：我们用 BIGINT 存储，字段名改为 balance_before / balance_after 更语义化
```

### B2：写查分离的 Controller 职责划分

```
决策：借鉴 ✅
原因：写操作和查询操作分开，职责清晰
对应：WalletController（写）+ CoinRecordController（查）→ 我们也可以类似划分 handler
```

### B3：SQL 级余额条件检查（subBalance 的 WHERE 子句）

```
决策：借鉴 ✅
原因：WHERE (amount - X) >= 0 是数据库级别的最后一道防线
调整：我们的乐观锁方案已包含此能力（WHERE available >= X AND version = V）
```

### B4：余额变更通知解耦

```
决策：借鉴 ✅
原因：余额变更后异步通知前端是通用模式
调整：不用 RabbitMQ，考虑 Redis Pub/Sub 或 WebSocket 直推
```

### B5：coin_record 宽表设计（冗余用户信息）

```
决策：借鉴 ✅
原因：账变记录需要独立可查，不依赖 JOIN 外部表
注意：写入时冗余字段必须可靠获取，避免为 null
```

### B6：冻结三阶段生命周期模型

```
决策：借鉴 ✅
原因：冻结→确认/回滚 的三阶段模型是金融系统标准做法
调整：具体实现用我们的余额模型（available + frozen 分字段存储）
```

## 13.2 调整项（思路可取，实现需改进）

### A1：并发控制从分布式锁改为乐观锁

```
决策：调整 ⚡
原因：
  - 分布式锁依赖 Redis，增加外部依赖和单点风险
  - 串行化牺牲吞吐量
  - 无法保护跨服务操作（order-service 绕过锁）
我们方案：乐观锁 CAS（WHERE version = V AND available >= X），无外部依赖
```

### A2：金额存储从 BigDecimal 改为 BIGINT

```
决策：调整 ⚡
原因：
  - BigDecimal 在 Java 中安全，但序列化/反序列化时有精度风险
  - 不同语言的 Decimal 实现不一致
  - 整数运算永远不会有精度问题
我们方案：BIGINT 存储分/厘，全链路零浮点
```

### A3：余额模型从 amount(总) 改为 available(可用)

```
决策：调整 ⚡
原因：
  - 可用余额是最高频查询的值，直接存储避免每次计算
  - SQL 校验更简单：WHERE available >= X
  - 对账公式更直观：total = available + frozen
详见第三章分析
```

### A4：懒创建从 select+insert 改为 INSERT ON DUPLICATE KEY UPDATE

```
决策：调整 ⚡
原因：参考工程依赖分布式锁保护创建，如果锁失效则可能重复创建
我们方案：数据库级原子操作，无论什么情况都安全
```

### A5：MQ 消费增加幂等保护

```
决策：调整 ⚡
原因：如果我们未来引入 MQ，消费端必须有幂等机制
方案：利用 coin_record 的业务唯一键（userId + orderNo + coinType）做去重
```

## 13.3 规避项（明确不采用的做法）

### X1：分布式锁作为唯一并发控制手段

```
决策：规避 ❌
原因：
  - Redis 宕机 = 锁失效 = 并发不安全
  - 跨服务无法统一（order-service 绕过锁）
  - 串行化不必要地降低吞吐
```

### X2：无幂等保护

```
决策：规避 ❌
原因：金融系统不做幂等等于裸奔
我们方案：双层幂等 — 应用层 Redis 去重 + DB 唯一键兜底
```

### X3：冻结校验 Bug（amount >= freezeAmount）

```
决策：规避 ❌
原因：可导致可用余额为负
我们方案：SQL 级 WHERE available >= X
```

### X4：多服务直接操作钱包表

```
决策：规避 ❌
原因：
  - 违反领域边界
  - 分布式锁在一个服务用，另一个不用，形同虚设
  - 后期维护和重构困难
我们方案：余额操作只通过 wallet 服务 RPC 暴露
```

### X5：catch Exception 吞异常

```
决策：规避 ❌
原因：异常被吞掉导致：
  - 问题排查困难
  - 可能丢失余额操作（异常后 return false，调用方可能不处理）
我们方案：精确异常类型 + 结构化错误码 + 关键操作 panic 兜底
```

### X6：日志明文打印余额和用户信息

```
决策：规避 ❌
原因：安全审计不合规
我们方案：日志中金额脱敏，敏感字段加密
```

### X7：addCoin 上帝方法（单方法多职责）

```
决策：规避 ❌
原因：120+ 行的 addCoin 承担了验证、加减钱、创建钱包、三方调用、流水记录等多重职责
我们方案：统一 ExecuteBalanceChange 方法 + 策略模式（按操作类型分发）
```

# 总结

## 一句话评价

参考工程的钱包模块是一个 **功能完整但安全防护薄弱** 的实现 — 基本业务链路跑通了，但在金融系统的核心挑战（幂等、并发、精度、对账）上存在显著不足。

## 核心借鉴价值

1. **冻结三阶段模型**的业务语义（冻结→确认/回滚）
2. **coinFrom / coinTo 审计追溯**的流水设计
3. **写查分离**的接口职责划分
4. **SQL 级条件扣减**作为最后防线的思路
5. **余额变更通知**的异步解耦架构

## 核心规避清单

1. **无幂等保护** — 金融系统最大禁忌
2. **分布式锁作为唯一并发控制** — 单点依赖 + 跨服务失效
3. **冻结校验 Bug** — 可用余额可冻成负数
4. **多服务直接操作钱包表** — 领域边界破坏
5. **BigDecimal 无精度约束** — 边界场景隐患
6. **日志明文打印金额** — 安全不合规
7. **上帝方法 addCoin** — 120 行多职责混杂

## 定量评分

```
总体评分：4.8 / 10

分项：
  功能完整性：7/10（基本链路齐全，缺对账）
  安全防护：  3/10（幂等缺失是致命伤）
  代码质量：  5/10（可运行但不够健壮）
  可扩展性：  4/10（单钱包模型，硬编码多）
  可维护性：  5/10（结构清晰但抽象不足）
  生产就绪度：4/10（功能就绪，安全未就绪）
```
