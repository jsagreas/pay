## 2. 工程架构设计

### 2.1 分层架构

```
┌─────────────────────────────────────────────────────┐
│  Controller层 (WalletController)                     │
│  职责：参数校验 + 分布式锁获取/释放 + 调用Service     │
│  特点：锁逻辑放在Controller，非Service层              │
├─────────────────────────────────────────────────────┤
│  Service层 (CoinRecordServiceImpl / UserCoinServiceImpl) │
│  职责：核心业务逻辑 + 事务管理(@Transactional)         │
│  特点：单个Service类承载所有业务（God Class倾向）       │
├─────────────────────────────────────────────────────┤
│  Repository层 (UserCoinRepository / CoinRecordRepository) │
│  职责：数据库操作 = MyBatis Plus BaseMapper + 自定义XML │
│  特点：余额操作用XML自定义SQL，查询用BaseMapper         │
├─────────────────────────────────────────────────────┤
│  Feign层 (4个FeignClient接口)                         │
│  职责：跨服务HTTP调用                                 │
│  特点：无降级(fallback)、无超时配置、无重试策略        │
├─────────────────────────────────────────────────────┤
│  MQ层 (AddCoinListener)                              │
│  职责：异步消费下注扣款/结算加款消息                   │
│  特点：消费者内部也获取分布式锁                       │
└─────────────────────────────────────────────────────┘
```

### 2.2 关键架构决策

| 决策 | 具体做法 | 评价 |
|------|---------|------|
| 锁的位置 | Controller层获取Redisson RLock | **有争议** — 锁粒度和业务绑定不够紧密 |
| 事务范围 | Service方法上 @Transactional | **合理** — 但部分方法内含RPC调用，违反事务原则 |
| 钱包类型 | 运行时查用户walletType走不同分支 | **灵活** — 但if/else散落各处，扩展性差 |
| 余额操作 | XML自定义SQL的原子UPDATE | **精华** — 数据库层保证原子性 |
| 冻结模型 | freeze_amount字段 + 3个操作 | **基础可用** — 但无冻结记录表 |
| 幂等设计 | **无** — 无幂等键/去重机制 | **严重缺陷** — 重复请求会重复加钱 |

## 3. 数据模型与表结构

### 3.1 user_coin 表（用户钱包主表）

```sql
CREATE TABLE user_coin (
    id          BIGINT AUTO_INCREMENT PRIMARY KEY,  -- 主键
    user_id     VARCHAR(64),       -- 用户ID（字符串类型）
    user_name   VARCHAR(128),      -- 用户名
    amount      DECIMAL(?,?),      -- 账户总金额（BigDecimal）
    freeze_amount DECIMAL(?,?),    -- 冻结金额
    user_state  INT,               -- 用户游戏状态
    game_type   INT,               -- 正在进行的游戏类型
    wallet_type INT,               -- 钱包类型：1=转账钱包, 2=免转钱包
    currency    VARCHAR(16),       -- 币种
    remark      VARCHAR(256),      -- 备注
    -- 以下继承BasicDataBaseDTO
    creator     BIGINT,
    updater     BIGINT,
    created_time DATETIME,
    updated_time DATETIME
);
```

**关键分析：**
- **amount** = 总余额（含冻结），可用余额 = amount - freeze_amount
- **单行单用户** — 一个userId对应一行，无多钱包概念
- **无币种约束** — currency字段仅记录，无唯一索引(userId+currency)
- **BigDecimal存储** — Java BigDecimal映射MySQL DECIMAL
- **无版本号字段** — 无乐观锁支持
- **user_state/game_type** — 游戏状态字段冗余在钱包表（职责越界）

### 3.2 coin_record 表（账变流水表）

```sql
CREATE TABLE coin_record (
    id            BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id       VARCHAR(64),       -- 用户ID
    user_name     VARCHAR(128),      -- 用户名
    user_type     INT,               -- 0=正式 1=试玩
    nick_name     VARCHAR(128),      -- 昵称
    user_belong   INT,               -- 用户所属
    merchant_id   BIGINT,            -- 商户ID
    merchant_name VARCHAR(128),      -- 商户名称
    merchant_no   VARCHAR(64),       -- 商户编号
    agent_no      VARCHAR(64),       -- 代理编号
    agent_id      BIGINT,            -- 代理ID
    agent_name    VARCHAR(128),      -- 代理名称
    path          VARCHAR(512),      -- 上级列表
    order_no      VARCHAR(128),      -- 订单号
    coin_type     INT,               -- 账变类型（9种）
    coin_detail   INT,               -- 账变明细（8种）
    coin_value    DECIMAL(?,?),      -- 变动金额（绝对值，始终>0）
    coin_from     DECIMAL(?,?),      -- 变动前余额
    coin_to       DECIMAL(?,?),      -- 变动后余额
    coin_amount   DECIMAL(?,?),      -- 当前金额（冗余）
    balance_type  INT,               -- 1=收入 2=支出
    remark        VARCHAR(512),      -- 备注
    creator       BIGINT,
    updater       BIGINT,
    created_time  DATETIME,
    updated_time  DATETIME
);
```

**关键分析：**
- **宽表设计** — 冗余了user_name/nick_name/merchant_name/agent_name等维度信息
- **order_no 非唯一** — 同一个订单号可以产生多条账变（下注+结算）
- **无幂等键** — 没有 unique index 防止重复插入
- **coin_from/coin_to** — 记录变动前后余额快照（审计用），但免转钱包时为null
- **coin_amount ≈ coin_to** — 字段语义重叠，代码中 `coinAmount = coinTo`
- **只INSERT不UPDATE** — 流水表是追加写，符合金融系统惯例
- **无deleted/status字段** — 无软删除

### 3.3 表结构对比

| 维度 | 参考A (user_coin + coin_record) | 我们的设计 (14张表) |
|------|-------------------------------|-------------------|
| 钱包表 | 1张，userId单行 | wallet + wallet_account(userId×currency多行) |
| 流水表 | 1张 coin_record | wallet_flow（追加写，不可更改） |
| 冻结记录 | **无独立表** | wallet_freeze（独立冻结记录） |
| 币种配置 | 无 | currency_config + exchange_rate |
| 提现订单 | 无（归order-service管） | withdraw_order（含状态机） |
| 金额精度 | BigDecimal（DECIMAL） | BIGINT最小单位整数 |
| 幂等保护 | **无** | idempotent_key UNIQUE INDEX |
| 多币种 | 单币种（currency仅标记） | 多币种（一个用户多个wallet_account） |

## 5. 核心业务逻辑深度拆解

### 5.1 addCoin — 核心加减款方法 (CoinRecordServiceImpl:169-341)

**这是整个钱包的核心方法，承载90%的资金变动逻辑。**

#### 5.1.1 转账钱包分支 (walletType=1，本地DB管理余额)

```
入参: CoinRecordVO { userId, coinValue, coinType, balanceType, orderNo, ... }
  │
  ├── 1. 校验 coinType 合法 (>0)
  ├── 2. RPC→user-service 获取用户信息 (getUserInfoByUserId)
  ├── 3. 赋值用户信息到 coinRecordDTO
  ├── 4. 判断 walletType
  │
  ├── [转账钱包 walletType=1]
  │   ├── 5. 查询 user_coin 记录 (selectOne by userId)
  │   │
  │   ├── [记录存在]
  │   │   ├── 6. insufficientBalance() 校验（仅转出/下注/打赏校验余额）
  │   │   ├── 7a. 如果 income → userCoinRepository.addBalance()
  │   │   ├── 7b. 如果 expenses → userCoinRepository.subBalance()
  │   │   │       └── ★ subBalance返回0表示余额不足 → 抛异常
  │   │   ├── 8. 计算 coinFrom(变动前) / coinTo(变动后)
  │   │   └── 9. INSERT coin_record
  │   │
  │   └── [记录不存在 = 新用户]
  │       ├── 6'. checkCoinChange() — 新用户不能是支出
  │       ├── 7'. INSERT user_coin (初始余额=coinValue)
  │       └── 8'. INSERT coin_record
  │
  └── [免转钱包 walletType=2] → 见5.1.2
```

**源码关键片段分析（转账钱包分支）：**

```java
// CoinRecordServiceImpl.java:182-230
if (Objects.equals(userInfoVO.getWalletType(), WalletTypeEnum.TRANSFER.getCode())) {
    UserCoinDTO userCoinDTO1 = userCoinRepository.selectOne(...);

    if (userCoinDTO1 != null) {
        // ★ 余额校验只在特定coinType生效
        insufficientBalance(coinRecordVO, userCoinDTO1);

        // ★ 加钱：直接SQL UPDATE amount = amount + ?
        if (Objects.equals(coinRecordVO.getBalanceType(), BalanceTypeEnum.income.getCode())) {
            userCoinRepository.addBalance(balanceVO);
            coinTo = userCoinDTO1.getAmount().add(coinRecordVO.getCoinValue());
        }
        // ★ 扣钱：SQL UPDATE amount = amount - ? WHERE (amount - ?) >= 0
        if (Objects.equals(coinRecordVO.getBalanceType(), BalanceTypeEnum.expenses.getCode())) {
            int i = userCoinRepository.subBalance(balanceVO);
            if (i == 0) {
                throw new MayaDefaultException(ErrorCode.AMOUNT_NOT_ENOUGH);
            }
            coinTo = userCoinDTO1.getAmount().subtract(coinRecordVO.getCoinValue());
        }
    } else {
        // 新用户：创建钱包
        checkCoinChange(coinRecordVO);  // 不能支出
        userCoinRepository.insert(userCoinDTO);
    }
    // 写账变记录
    coinRecordRepository.insert(coinRecordDTO);
}
```

**★ 关键发现 — coinFrom/coinTo 的计算Bug：**

```java
// 第211行：coinFrom 赋值为查询时的amount
coinRecordDTO.setCoinFrom(userCoinDTO1.getAmount());
// 第199行：coinTo = 查询时amount + coinValue
coinTo = userCoinDTO1.getAmount().add(coinRecordVO.getCoinValue());
```

这里有**并发时数据不准确**的问题：
- `userCoinDTO1.getAmount()` 是SELECT时的快照值
- 但实际UPDATE后的amount可能因其他并发请求改变
- 虽然有Redisson锁保护，但锁粒度是userId级别，同一用户的coinFrom/coinTo在高并发下是准确的
- 但如果锁失效或MQ消费和HTTP请求并发（都用同一个锁），就可能出现偏差

#### 5.1.2 免转钱包分支 (walletType=2，余额由三方管理)

```
  └── [免转钱包 walletType=2]
      ├── 5. 根据coinType确定gameTypeId和playType
      │   ├── SEND_REWARD → gameTypeId=100, playType="GIFT"
      │   └── 其他 → RPC→order-service 查订单获取 gameTypeId/playType
      ├── 6. RPC→merchant-service 获取商户详情（含三方API地址）
      ├── 7. 封装 ThirdOrderRequestVO（transferNo/totalAmount/transferType）
      ├── 8. RPC→api-service 调用三方加扣款 (amountTransfer)
      │   ├── [成功] → 保存三方订单记录 + MQ通知余额变更
      │   └── [失败] → 保存失败记录 → return false
      └── 9. INSERT coin_record
```

**免转钱包的特殊处理：**
- coinFrom/coinTo 设为null（因为余额不在本地）
- 需要先调三方API完成资金操作，再写本地账变
- 三方调用失败时：记录失败状态，但**不回滚**，直接返回false

### 5.3 insufficientBalance — 余额充足性校验 (CoinRecordServiceImpl:366-384)

```java
private void insufficientBalance(CoinRecordVO coinRecordVO, UserCoinDTO userCoin) {
    // ★ 只在以下三种操作时校验余额：
    //   - TRANSFER_OUT(2) 转出
    //   - BET(3) 下注
    //   - SEND_REWARD(8) 打赏

    // ★ 可用余额 = amount - freezeAmount
    BigDecimal balance = userCoinAmount.subtract(freezeAmount);
    if (balance.compareTo(amount) < 0) {
        throw new MayaDefaultException(ErrorCode.AMOUNT_NOT_ENOUGH);
    }

    // ★ 以下操作不校验余额，允许扣为负数：
    //   - SETTLEMENT(4) 结算
    //   - ORDER_CANCEL(5) 取消
    //   - SETTLEMENT_ROLLBACK(6) 结算回滚
    //   - SETTLEMENT_SECOND(7) 二次结算
}
```

**精华点：** 结算/回滚允许扣为负数，这是游戏结算业务的合理设计——玩家可能在下注后充值又提走，结算时余额不足但仍需记录。

### 5.4 addCoinBatch / deductCoinBatch — 批量操作

```java
@Transactional
public boolean addCoinBatch(List<CoinRecordVO> coinRecordVOList) {
    for (CoinRecordVO coinRecordVO : coinRecordVOList) {
        boolean result = addCoin(coinRecordVO);  // ★ 循环调用单笔
        if (!result) {
            throw new MayaDefaultException("余额处理失败");
        }
    }
    return true;
}
```

**问题：** 批量=循环单笔，每次addCoin内部都RPC查用户信息，N笔就N次RPC。性能差。

## 6. 并发控制与分布式锁

### 6.1 锁策略

```
┌─────────────────────────────────────────────────┐
│  Redisson Distributed Lock                       │
│  Key模式: "add.coin.lock.key.{userId}"           │
│  等待时间: 20秒 (tryLock等待获取)                 │
│  持有时间: 30秒 (自动释放，避免死锁)              │
│  锁类型: RLock (非公平可重入锁)                   │
└─────────────────────────────────────────────────┘
```

### 6.2 锁使用位置分析

| 调用入口 | 锁Key | 分析 |
|---------|-------|------|
| addCoin (HTTP) | userId | ✅ 合理 |
| addCoinList (HTTP) | list[0].userId | ⚠️ 只锁第一个用户 |
| freezeCoin (HTTP) | userId | ✅ 合理 |
| subFreezeBalance (HTTP) | userId | ✅ 合理 |
| rollbackFreezeBalance (HTTP) | userId | ✅ 合理 |
| NoticeAddCoinMessage (MQ) | userId | ✅ 合理，与HTTP共用同一把锁 |
| noticeDeductCoinOrderMessage (MQ) | **无锁** | ⚠️ 调deductCoinBatch无锁 |
| addCoinNotNoticeBalance (MQ) | **无锁** | ⚠️ 调deductCoinBatch无锁 |
| addCoinRecordAndNoticeBalance (MQ) | **无锁** | ⚠️ 调changeCoinAndNoticeBalance无锁 |

### 6.3 锁粒度评估

```
优点：
├── userId粒度 — 同用户串行，不同用户并行
├── HTTP和MQ共用同一把锁Key — 保证一致性
├── tryLock超时机制 — 避免永久阻塞
└── finally中安全释放 — isLocked + isHeldByCurrentThread双重检查

缺点（严重）：
├── [P0] 部分MQ消费者无锁 — deductCoinBatch直接操作余额无锁保护
├── [P0] addCoinList只锁一个userId — 批量含多用户时数据不安全
├── [P1] 无数据库悲观锁兜底 — 如果Redis锁故障，数据无保护
├── [P1] 30秒持有时间 — 如果addCoin中三方API超时>30秒，锁提前释放
└── [P2] 锁key太简单 — 不区分操作类型，所有写操作串行化
```

### 6.4 与我们的方案对比

| 维度 | 参考A | 我们的设计 |
|------|------|-----------|
| 锁类型 | Redis分布式锁（Redisson） | DB悲观锁 + Redis缓存锁 |
| 锁粒度 | userId | userId+walletId |
| 兜底方案 | 无 | SQL WHERE amount >= ? 三层防线 |
| 死锁预防 | 超时自动释放 | 锁排序 + 超时 |
| 锁位置 | Controller层 | Service层（更内聚） |

## 7. 冻结模型设计

### 7.1 冻结三操作

```
冻结(freeze)：   freeze_amount += N    （可用余额减少，总额不变）
确认扣除(sub)：  amount -= N, freeze_amount -= N （总额和冻结都减少）
回滚(rollback)： freeze_amount -= N    （恢复可用余额，总额不变）
```

### 7.2 SQL实现

```xml
<!-- 冻结：增加冻结金额 -->
<update id="addFreezeBalance">
    UPDATE user_coin b
    SET b.freeze_amount = b.freeze_amount + #{vo.amount},
        b.updated_time  = #{vo.updateTime}
    WHERE b.id = #{vo.id}
</update>

<!-- 确认扣除：总额和冻结都减 -->
<update id="subFreezeBalance">
    UPDATE user_coin b
    SET b.amount        = b.amount - #{vo.amount},
        b.freeze_amount = b.freeze_amount - #{vo.amount},
        b.updated_time  = #{vo.updateTime}
    WHERE b.id = #{vo.id}
</update>

<!-- 回滚冻结：只减冻结 -->
<update id="rollbackFreezeBalance">
    UPDATE user_coin b
    SET b.freeze_amount = b.freeze_amount - #{vo.amount},
        b.updated_time  = #{vo.updateTime}
    WHERE b.id = #{vo.id}
</update>
```

### 7.3 冻结模型缺陷分析

| 问题 | 严重性 | 说明 |
|------|--------|------|
| 无冻结记录表 | ★★★★ | 没有独立的freeze记录，无法追溯谁冻结的、什么时候冻结的 |
| 冻结校验不足 | ★★★★ | addFreezeBalance无条件限制——`amount >= freeze_amount + new_freeze`未校验 |
| freeze可超额 | ★★★★ | 理论上freeze_amount可以 > amount（SQL无WHERE限制） |
| 回滚无下限校验 | ★★★ | rollbackFreezeBalance可将freeze_amount减为负数 |
| 确认扣除无@Transactional | ★★★★★ | subFreezeBalance方法未加事务注解！amount和freeze_amount的扣减非原子 |
| 无关联订单号 | ★★★ | 冻结操作没有关联order_no，无法知道是哪笔订单冻结的 |

### 7.4 对比我们的设计

```
参考A冻结模型:
  user_coin.freeze_amount (一个数字)
  └── 无记录、无关联、无校验

我们的冻结模型:
  wallet_account.frozen_amount (主表冗余字段)
  wallet_freeze (独立冻结记录表)
  ├── freeze_id
  ├── wallet_id + account_id
  ├── order_no (关联业务订单)
  ├── freeze_amount
  ├── status (frozen/unfrozen/confirmed)
  ├── freeze_type (withdraw/bet/manual)
  └── created_at / updated_at
```

**结论：** 参考A的冻结模型过于简单，生产环境有资金安全风险。我们的三阶段冻结+独立记录表设计更健壮。

### 8.2 消费者逻辑拆解

**NoticeAddCoinMessage（单笔加减款）— 最完整的消费者：**

```java
    CoinRecordVO coinRecordVO = JSONObject.parseObject(jsonStr, CoinRecordVO.class);
    // 1. 空值/零值过滤
    // 2. 获取Redisson锁（与HTTP端共用同一把锁）
    // 3. 调addCoin（完整加减款逻辑）
    // 4. 成功后发送余额变更通知
    // 5. finally释放锁
}
```

**noticeDeductCoinOrderMessage（下注扣款订单）— 无锁消费者：**
```java
@RabbitListener(queues = {MqConstants.ADD_COIN_ORDER_QUEUE})
public void noticeDeductCoinOrderMessage(String jsonStr) {
    List<CoinRecordVO> coinRecordVOList = JSON.parseArray(jsonStr, CoinRecordVO.class);
    // ★ 无分布式锁！
    boolean result = coinRecordService.deductCoinBatch(coinRecordVOList);
    if (!result) {
        // 失败：回调order-service恢复订单初始状态
        orderResource.updateOrdersToInitialStatus(collect);
    }
    coinRecordService.sendUserBalance(userId);
}
```

### 8.3 MQ设计评价

| 维度 | 评价 | 说明 |
|------|------|------|
| 异步解耦 | ✅ 好 | 下注/结算通过MQ异步处理，不阻塞游戏流程 |
| 消息幂等 | ❌ 无 | 没有消息去重机制，重复消费会重复加钱 |
| 消费确认 | ⚠️ 默认 | 使用RabbitMQ默认ACK，异常时消息可能丢失或重复 |
| 失败处理 | ⚠️ 弱 | 扣款失败只是通知order-service恢复状态，无重试/死信 |
| 锁一致性 | ⚠️ 不一致 | 部分消费者有锁，部分无锁 |
| 消息格式 | ⚠️ JSON字符串 | 直接传JSON，无消息版本号/时间戳/幂等键 |

### 8.4 对比我们的设计

我们目前没有MQ，依赖RPC同步调用。参考A使用MQ的场景给我们的启示：

| 场景 | 参考A方案 | 我们应该怎么做 |

## 10. 余额变更SQL策略

### 10.1 五个核心SQL操作

```xml
<!-- ① 加钱：无条件，直接加 -->
UPDATE user_coin SET amount = amount + #{amount} WHERE id = #{id}

<!-- ② 扣钱：有条件，余额不能为负（结算回滚/取消除外） -->
UPDATE user_coin SET amount = amount - #{amount} WHERE id = #{id}
  <!-- 结算回滚(6)/取消(5) 不加条件 -->
  <if test="coinType != 6 and coinType != 5">
    AND (amount - #{amount}) >= 0
  </if>

<!-- ③ 增加冻结 -->
UPDATE user_coin SET freeze_amount = freeze_amount + #{amount} WHERE id = #{id}

<!-- ④ 确认扣冻结：同时减总额和冻结额 -->
UPDATE user_coin SET amount = amount - #{amount}, freeze_amount = freeze_amount - #{amount} WHERE id = #{id}

<!-- ⑤ 回滚冻结：只减冻结额 -->
UPDATE user_coin SET freeze_amount = freeze_amount - #{amount} WHERE id = #{id}
```

### 10.2 SQL设计精华

| 设计点 | 说明 | 评价 |
|--------|------|------|
| 原子UPDATE | `amount = amount + ?` 而非先查再改 | ★★★★★ 数据库层保证原子性 |
| 条件扣款 | `AND (amount - ?) >= 0` 防负数 | ★★★★★ SQL层兜底防线 |
| 结算豁免 | coinType=5/6 时不校验余额 | ★★★★ 业务合理 |
| 按ID定位 | WHERE id = #{id} 主键定位 | ★★★★ 性能好 |

### 10.3 SQL设计缺陷

| 缺陷 | 严重性 | 说明 |
|------|--------|------|
| ③无上限校验 | ★★★★ | 冻结可以无限增加，freeze > amount |
| ④无下限校验 | ★★★★ | 确认扣冻结可将freeze减为负 |
| ⑤无下限校验 | ★★★ | 回滚可将freeze减为负 |
| 无updated_by | ★★ | 更新时不记录操作人 |
| 无version | ★★ | 无乐观锁版本号 |

### 10.4 对比我们的设计

```
参考A:  Redis分布式锁 + SQL原子UPDATE（amount = amount ± ?）
        └── 两层防线（锁 + SQL条件）

我们:   DB悲观锁(SELECT FOR UPDATE) + SQL原子UPDATE + 代码层校验
        └── 三层防线（代码 + SQL + 锁）
```

**我们比参考A多一层保护：**
1. 代码层：`if balance < amount → reject`
2. SQL层：`WHERE (amount - ?) >= 0`
3. 锁层：`SELECT ... FOR UPDATE` 防止并发读脏数据

```java
String redisKey = RedisConstants.CURRENCY_LIMIT_ID_MERCHANT_ID + userId;
Object obj = redisUtil.get(redisKey);
if (obj instanceof UserInfoVO) return (UserInfoVO) obj;
// cache miss → RPC查询 → ★ 没有set回Redis！
```

**严重Bug：** 用户信息查询(`getUserInfoByUserId`)命中缓存时返回，但cache miss后**只查不写回Redis**。这意味着用户信息缓存只在其他服务写入时生效，wallet-service本身每次都要RPC查询。

### 13.2 缓存评价

| 维度 | 评价 |
|------|------|
| 商户缓存 | ✅ Read-Through，TTL 1天 |
| 用户缓存 | ❌ 只读不写，形同虚设 |
| 余额缓存 | ❌ 无 — 每次查DB |
| 缓存穿透保护 | ❌ 无 — null值未缓存 |
| singleflight | ❌ 无 — 热key可能雪崩 |

**对我们的启示：** 我们不需要免转钱包模式（需求不同），但"同一API支持多模式"的思路值得参考——如果未来需要支持不同类型的钱包（如奖金钱包、虚拟币钱包），可以用策略模式替代if/else。

## 16. 精华提炼：值得借鉴的设计

### 16.1 SQL层原子余额操作 ★★★★★

```xml
UPDATE user_coin SET amount = amount + #{amount} WHERE id = #{id}
UPDATE user_coin SET amount = amount - #{amount} WHERE id = #{id}
  AND (amount - #{amount}) >= 0  -- 条件扣款
```

**借鉴要点：**
- 永远不要 `SELECT amount → amount ± value → UPDATE amount = newValue`
- 始终用 `UPDATE amount = amount ± value` 原子操作
- 扣款加条件 `AND (amount - ?) >= 0`，数据库层兜底
- 特殊业务（结算回滚）豁免条件校验

**我们的应用：** GORM中实现为
```go
db.Model(&WalletAccount{}).
    Where("id = ? AND (balance - ?) >= 0", id, amount).
    Update("balance", gorm.Expr("balance - ?", amount))
```

### 16.2 分布式锁的释放模式 ★★★★

```java
finally {
    if (Objects.nonNull(rLock) && rLock.isLocked() && rLock.isHeldByCurrentThread()) {
        rLock.unlock();
    }
}
```

**三重检查：**
1. `Objects.nonNull(rLock)` — 防null
2. `rLock.isLocked()` — 确认锁还在
3. `rLock.isHeldByCurrentThread()` — 确认是当前线程持有

**我们的应用：** Go中defer释放DB悲观锁时也需类似安全检查

### 16.3 结算/回滚允许余额为负 ★★★★

```xml
<if test="vo.coinType != 6 and vo.coinType != 5">
    AND (b.amount - #{vo.amount}) >= 0
</if>
```

**业务洞察：** 游戏场景中，玩家下注后余额被用于其他操作（提现），结算时余额可能不够。参考A的做法是允许扣为负数，后续通过人工或系统处理。

**我们的应用：** 这个逻辑合理，但需要配套：
- 负数余额告警
- 负数余额限制（不能继续下注/提现）
- 运营处理流程

### 16.4 HTTP和MQ共用同一把锁 ★★★★

```java
// HTTP端 (WalletController):
RLock rLock = redisUtil.getLock(RedisLockKey.getAddCoinLockKey(userId));
// MQ端 (AddCoinListener):
RLock rLock = redisUtil.getLock(RedisLockKey.getAddCoinLockKey(userId));
// 同一个key → 互斥
```

**关键：** 不管是HTTP请求还是MQ消费，对同一用户的余额操作都要串行。这避免了HTTP充值和MQ结算同时修改余额的竞争。

### 16.5 余额变更MQ通知 ★★★

```java
// 每次余额变更后，广播通知其他服务
rabbitTemplate.convertAndSend(
    MqConstants.EXCHANGE_NOTICE_USER_BALANCE_CHANGE, "", message);
```

**用途：** WebSocket服务收到广播后推送前端更新余额显示。

**我们的应用：** 我们也需要余额变更通知——通过RPC回调或WebSocket推送。

### 16.6 宽表冗余设计 ★★★

coin_record表冗余了user_name/merchant_name/agent_name等字段。

**优点：** 查询账变时不需要JOIN用户表/商户表，查询性能好。
**缺点：** 如果用户改名，历史记录不一致。

**结论：** 对于金融流水表，冗余是合理的——流水应该记录"当时的状态"。

## 17. 糟粕识别：需要规避的问题

### 17.1 P0级（严重，必须规避）

#### 17.1.1 无幂等机制 ★★★★★

**问题：** 没有任何幂等防护。如果addCoin被重复调用（网络重试、MQ重投、前端重复提交），会重复加钱/扣钱。

**证据：**
- coin_record表无 UNIQUE INDEX on (order_no + coin_type) 或 idempotent_key
- 代码中无任何去重逻辑
- MQ消费者无消息去重

**后果：** 生产环境下必然出现资金不一致。

**我们的对策：** 必须实现 INSERT-before-UPDATE 幂等模式 + idempotent_key UNIQUE INDEX

#### 17.1.2 subFreezeBalance无事务 ★★★★★

**问题：** `subFreezeBalance`方法未加`@Transactional`注解。

```java
@Override  // ← 无 @Transactional！
public ErrorCode subFreezeBalance(CoinFreezeRecordVO coinFreezeRecordVO) {
    // Step 1: SQL扣余额和冻结
    userCoinRepository.subFreezeBalance(balanceVO);
    // Step 2: 写账变记录
    createOrder(coinFreezeRecordVO, userCoinDTO);  // ← 如果这步失败，余额已扣但无记录
}
```

**后果：** 如果createOrder（INSERT coin_record）失败，余额已被扣减但没有账变记录，资金不可追溯。

#### 17.1.3 三方API在事务内 ★★★★★

**问题：** addCoin免转分支中，三方HTTP调用在`@Transactional`事务范围内。

**后果：**
- 三方API响应慢 → 数据库连接长时间占用 → 连接池耗尽
- 三方API超时后事务回滚 → 但三方可能已成功执行 → 资金不一致

#### 17.1.4 部分MQ消费者无锁 ★★★★★

**问题：** `noticeDeductCoinOrderMessage`和`addCoinNotNoticeBalance`直接调用deductCoinBatch，无分布式锁保护。

**后果：** 如果同一用户的多条MQ消息并发消费，可能导致余额计算错误。

### 17.2 P1级（重要，强烈建议规避）

#### 17.2.1 God Class设计

**问题：** CoinRecordServiceImpl承载所有业务逻辑（522行），包含：
- 加减款、批量操作、冻结/解冻/回滚、账变查询、MQ通知、用户信息填充
- 转账/免转两种模式混在同一个方法
- 无职责分离

**我们的对策：** 按职责拆分为：
- WalletService（余额操作核心）
- WalletFlowService（流水记录）
- WalletFreezeService（冻结管理）
- WalletQueryService（查询）

#### 17.2.2 锁放Controller层

**问题：** 分布式锁逻辑散落在Controller的每个方法中，大量重复代码。

**我们的对策：** 锁应在Service层或通过AOP/拦截器统一管理。

#### 17.2.3 Feign无降级无超时

**问题：** 4个FeignClient接口均无fallback配置，user-service挂了整个钱包不可用。

**我们的对策：** Kitex RPC配置超时(2s) + 重试(1次) + 降级(缓存兜底)

#### 17.2.4 错误码被吞

**问题：** Controller的`catch(Exception)`吞掉了Service层的具体错误码，统一返回FAIL。

**我们的对策：** 全局异常处理器，精确映射业务异常到HTTP响应码。

#### 17.2.5 用户信息缓存只读不写

**问题：** CommonServiceImpl.getUserInfoByUserId cache miss后不写回Redis。

#### 17.2.6 冻结无上下限校验

**问题：** freeze_amount可以超过amount，也可以被减为负数。

### 17.3 P2级（优化建议）

| 问题 | 说明 |
|------|------|
| 游戏状态冗余在钱包表 | user_state/game_type不应该在user_coin表 |
| BigDecimal存储 | Java有BigDecimal方便，但存在精度配置不统一风险 |
| coin_amount ≈ coin_to | 两个语义重叠字段 |
| 查询SQL大量重复 | getCoinRecordPage和getCoinRecordList几乎一样 |
| BeanUtils.copyProperties | 大量浅拷贝，效率差且有隐式字段遗漏风险 |
| 日志信息过多 | JSON.toJSONString序列化大对象写日志，性能差 |

## 19. 落地建议清单

### 19.1 直接采纳（精华）

| # | 参考A的设计 | 我们如何采纳 |
|---|-----------|------------|
| 1 | SQL原子UPDATE `amount = amount ± ?` | GORM的 `gorm.Expr("balance - ?", amount)` |
| 2 | 扣款条件 `AND (amount - ?) >= 0` | WHERE子句加入条件 |
| 3 | 结算回滚允许负数 | 特定flow_type豁免条件校验 |
| 4 | 锁释放三重检查 | defer中检查锁状态 |
| 5 | HTTP/MQ共用锁Key | RPC入口统一加锁 |
| 6 | 余额变更后通知 | 事务提交后异步推送 |
| 7 | 流水表宽表冗余 | wallet_flow冗余关键维度信息 |
| 8 | 只INSERT不UPDATE流水 | wallet_flow追加写，禁止更新 |

### 19.2 改良采纳（取其思路，改进实现）

| # | 参考A的做法 | 我们的改良 |
|---|-----------|-----------|
| 1 | Redis分布式锁在Controller层 | DB悲观锁在Service层（更内聚） |
| 2 | 全局catch+通用错误码 | 精确异常映射+模块化错误码 |
| 3 | if/else切换钱包类型 | 策略模式（如需多模式时） |
| 4 | BeanUtils.copyProperties | Go struct直接赋值或工具函数 |
| 5 | BigDecimal存储 | BIGINT最小单位（更安全） |
| 6 | 单个CoinRecordService | 拆分为4+个专职Service |
| 7 | Feign同步RPC | Kitex RPC + 超时 + 重试 + 降级 |

### 19.3 坚决规避（糟粕）

| # | 参考A的问题 | 我们的规避方案 |
|---|-----------|--------------|
| 1 | **无幂等机制** | idempotent_key UNIQUE + INSERT-before-UPDATE |
| 2 | **事务内RPC/HTTP** | 事务只包DB操作，RPC在事务外 |
| 3 | **冻结无记录表** | wallet_freeze独立表 |
| 4 | **冻结无上下限** | SQL加WHERE条件 |
| 5 | **subFreezeBalance无事务** | 所有写操作必须在事务内 |
| 6 | **MQ消费者无锁** | 所有写入口统一加锁 |
| 7 | **错误码被吞** | 全局异常处理，精确返回 |
| 8 | **Feign无降级无超时** | 所有RPC配超时+重试+降级 |
| 9 | **用户缓存只读不写** | 标准Cache-Aside：查→写回 |
| 10 | **God Class** | 按职责拆分Service |

### 19.4 编码优先级建议

基于参考A的经验教训，我们编码时的防御性优先级：

```
P0 必须（资金安全）:
├── 幂等机制（idempotent_key）
├── SQL条件扣款（WHERE balance >= amount）
├── 事务边界正确（无RPC在事务内）
├── 所有写入口加锁
└── 冻结上下限校验

P1 重要（系统健壮）:
├── RPC超时+降级
├── 错误码精确传递
├── 缓存正确实现
└── 流水表追加写

P2 优化（代码质量）:
├── Service职责拆分
├── 锁抽象统一管理
└── 查询性能优化
```
