## 1. 开场定调：为什么需要这次沟通

### 1.1 背景

钱包（ser-wallet）和支付/财务（ser-finance）是平台资金体系的**左右手**：

```
                        外部世界
                           │
                    ┌──────┴───────┐
                    │   三方通道     │
                    │ (代收/代付)    │
                    └──────┬───────┘
                           │ HTTP 回调
                           ▼
                    ┌──────────────┐
                    │  ser-finance  │  ← 支付侧：通道管理、订单管理、审核管理、三方对接
                    │  (支付/财务)  │
                    └───┬──────┬───┘
                        │      │
              RPC 调用  │      │  RPC 回调
              (5个方法) │      │  (7个方法)
                        │      │
                        ▼      ▼
                    ┌──────────────┐
                    │  ser-wallet   │  ← 钱包侧：余额管理、流水记录、币种配置、C端展示
                    │  (钱包)       │
                    └──────────────┘
```

**任何一方对另一方的理解有偏差，都可能导致资金事故。** 这次沟通的目标是：

1. **消除盲区**：双方对对方的接口、参数、状态机、异常处理有一致的理解
2. **建立契约**：形成书面的 IDL 定义和协议文档，后续以此为准开发
3. **确认时序**：关键操作（冻结、入账、扣款）的先后顺序必须明确
4. **约定规范**：订单号格式、错误码范围、超时重试策略等工程规范对齐

## 2. 双方角色与职责边界确认

> **为什么先聊这个**：之前的需求分析中发现，充值/提现流程中有些环节的归属模糊（比如限额校验两边都有能力做，订单创建两边都可能做），需要先把"谁管什么"理清。

### 2.1 职责对照表

| 能力域 | ser-wallet（钱包） | ser-finance（支付/财务） | 边界说明 |
|--------|-------------------|------------------------|---------|
| **余额管理** | ✅ 全权负责 | ❌ 不碰余额 | 所有余额变动（增/减/冻结/解冻）只能通过钱包 RPC |
| **流水记录** | ✅ 全权负责 | ❌ 有自己的订单但不记余额流水 | 钱包流水 = 账本，财务订单 = 业务单据 |
| **币种/汇率** | ✅ 全权负责 | 📥 消费方 | 财务需要查币种信息和汇率时调钱包的币种配置 RPC |
| **充值方式配置** | ❌ 不管 | ✅ 全权负责 | 钱包 C 端展示充值方式时调财务 RPC 获取 |
| **提现方式配置** | 📝 二次过滤 | ✅ 基础配置 | 财务管基础配置，钱包按充值来源规则做二次过滤 |
| **通道配置/轮询** | ❌ 不管 | ✅ 全权负责 | 通道选择算法完全由财务内部实现 |
| **三方支付对接** | ❌ 不管 | ✅ 全权负责 | 三方 HTTP 调用、签名校验、回调处理都在财务 |
| **充值订单** | 📝 本地副本 | ✅ 主订单 | 钱包记录关联信息（用户维度），财务管完整订单生命周期 |
| **提现订单** | 📝 本地副本 + 冻结 | ✅ 主订单 + 审核 | 钱包负责冻结/解冻/扣款，财务负责订单状态和三级审核 |
| **三级审核** | ❌ 不管 | ✅ 全权负责 | 风控→财务→出款三级审核链完全在 ser-finance 内部 |
| **人工修正** | 📥 被调方（执行加减款） | ✅ 发起方（创建修正单+审核） | 财务审核通过后调钱包 RPC 执行入账/扣款 |
| **C 端展示** | ✅ 全权负责 | ❌ 不管 C 端 | 充值页/提现页/余额页/记录页都由钱包出接口 |
| **B 端管理** | ✅ 币种+汇率管理 | ✅ 通道+订单+审核管理 | 各管各的 B 端功能 |

### 2.2 需要当场确认的边界模糊点

**【议题 2-A】充值订单到底由谁创建？**

```
现有理解：
  用户点击"充值" → 钱包收到请求 → 钱包调 finance.CreateDepositOrder() → 财务创建主订单

  钱包侧同时创建一条本地充值订单记录（关联 finance_order_no），用于：
    - C 端充值记录列表展示
    - 本地幂等校验
    - 充值来源追踪（影响后续提现方式过滤）

❓ 确认：这个流程对吗？还是前端直接调财务的接口跳过钱包？
```

**【议题 2-B】提现方式过滤的职责划分**

```
现有理解：
  1. 财务侧管理基础配置：哪些提现方式可用、每种方式的限额/手续费
  2. 钱包侧做二次过滤：按"充值来源"规则过滤（法币充值→法币提现，USDT充值→USDT提现）

❓ 确认：
  - 财务返回的方式列表是"该币种所有启用的提现方式"？
  - 钱包在此基础上根据用户充值历史做过滤？
  - 还是财务那边就已经有类似的过滤逻辑？
```

**【议题 2-C】B 端"用户钱包查询"归属哪个模块？**

```
需求文档描述财务管理后台有"查看用户余额"的功能（用于人工加减款前确认余额）。

❓ 确认：
  - 方案 A：财务 B 端直接调钱包 RPC GetWalletInfo 获取余额展示
  - 方案 B：钱包提供独立的 B 端余额查询页面，财务 B 端跳转过去

  我们倾向方案 A（更简单），需确认。
```

## 3. RPC 接口契约对齐（12 个双向接口）

> **这是本次沟通的核心重点。** 12 个接口的参数、返回值、语义必须双方达成一致，否则联调时必出问题。

### 3.1 钱包暴露给财务调用的 7 个 RPC

#### ① WalletCredit — 余额入账（充值到账/补单入账）

```
场景：财务收到三方支付成功回调后，调此接口给用户入账
频率：每笔成功充值调一次（核心链路，不可降级）

入参（钱包侧设计）：
  - user_id     i64     必填    用户ID
  - currency_code string 必填    币种编码（如 VND/USDT/BSB）
  - wallet_type  i32    必填    钱包类型（1=中心 2=奖励）
  - amount       string 必填    入账金额（字符串，避免精度丢失）
  - biz_type     string 必填    业务类型（deposit/supplement/activity...）
  - order_no     string 必填    幂等键（平台订单号）
  - remark       string 选填    备注

返回值：
  - code         i32           0=成功, 非0=失败
  - msg          string        错误描述
  - balance_after string       入账后的余额（用于财务侧记录）
  - transaction_id i64         流水ID

幂等保证：
  钱包侧通过 UNIQUE(biz_type, order_no, wallet_type) 保证。
  重复调用返回 code=0 + 已有流水的 balance_after（不会重复入账）。
```

**❓ 需要支付侧确认的问题：**

| 序号 | 问题 | 背景 |
|------|------|------|
| 3-1-A | wallet_type 由谁指定？ | 充值入账默认进中心钱包(1)，但充值奖金进奖励钱包(2)。财务侧传还是钱包侧自动判断？ |
| 3-1-B | amount 是否含手续费？ | 用户充值 100 VND，通道收了 2 VND 手续费，amount 传 100 还是 98？ |
| 3-1-C | biz_type 枚举对齐 | 钱包侧定义的业务类型：deposit(充值)/supplement(补单)/manual_credit(人工加款)，财务侧传什么值？ |
| 3-1-D | order_no 就是财务的平台充值订单号？ | 确认 order_no = 财务的 deposit_order.order_no，还是另外生成？ |

#### ② WalletDebit — 余额扣款（提现出款确认）

```
场景：财务确认三方代付通道出款成功后，调此接口从用户冻结金额中正式扣除
频率：每笔成功提现调一次

入参（钱包侧设计）：
  - user_id       i64     必填
  - currency_code  string  必填
  - wallet_type    i32     必填    通常=1（中心钱包）
  - amount         string  必填    扣款金额
  - biz_type       string  必填    业务类型（withdraw）
  - order_no       string  必填    幂等键（平台提现订单号）
  - remark         string  选填

返回值：
  - code           i32     0=成功
  - balance_after  string  扣款后余额

内部逻辑：
  这不是直接扣可用余额，而是：
    frozen_amount -= amount （减少冻结）
    balance -= amount （总余额减少）
  因为在提现发起时余额已经冻结了。
```

**❓ 需要支付侧确认的问题：**

| 序号 | 问题 | 背景 |
|------|------|------|
| 3-2-A | amount 含不含手续费？ | 用户提现 100，手续费 2，冻结的是 102 还是 100？扣款时传 102 还是 100？ |
| 3-2-B | 如果是从冻结扣款，需要区分操作类型吗？ | 钱包内部区分 debit_frozen（扣冻结）和 debit_available（扣可用），是否需要入参区分？ |

---

#### ③ WalletFreeze — 冻结余额（提现申请时冻结）

```
场景：提现流程中冻结用户余额，防止审核期间用户把钱花掉
频率：每笔提现申请调一次

入参：
  - user_id        i64     必填
  - currency_code   string  必填
  - wallet_type     i32     必填    通常=1
  - amount          string  必填    冻结金额
  - order_no        string  必填    关联提现订单号（幂等键）

返回值：
  - code            i32
  - frozen_balance  string  冻结后的冻结总额
  - available       string  冻结后的可用余额

内部逻辑：
  校验：available_balance >= amount
  操作：frozen_amount += amount

  注意：balance（总额）不变，只是 frozen 增加、available 减少。
  available = balance - frozen_amount
```

**❓ 需要支付侧确认的问题：**

| 序号 | 问题 | 关键 |
|------|------|------|
| 3-3-A | **冻结由谁发起？（P0-11）** | 方案A：钱包 F11 接口内部自己冻结 → 再调 finance.CreateWithdrawOrder。方案B：钱包调 finance.CreateWithdrawOrder → 财务建单时回调 wallet.Freeze。**钱包侧强烈倾向方案 A**，详见 §5.2。 |
| 3-3-B | 冻结金额 = 提现金额 + 手续费？还是仅提现金额？ | 这取决于 P0-16 手续费处理方案。如果手续费"在内"，冻结=提现金额；如果"在外"，冻结=提现+手续费。 |

#### ④ WalletUnfreeze — 解冻余额（提现审核驳回）

```
场景：提现订单在三级审核中被任一级驳回，需要解冻用户余额
频率：每笔驳回的提现订单调一次

入参：
  - user_id        i64     必填
  - currency_code   string  必填
  - wallet_type     i32     必填
  - amount          string  必填    解冻金额（= 原冻结金额）
  - order_no        string  必填    关联提现订单号

返回值：
  - code            i32
  - frozen_balance  string  解冻后的冻结总额
  - available       string  解冻后的可用余额

内部逻辑：
  frozen_amount -= amount
  available 自动增加（balance 不变，frozen 减少）
```

**❓ 需要支付侧确认的问题：**

| 序号 | 问题 | 背景 |
|------|------|------|
| 3-4-A | 解冻是同步 RPC 还是异步消息？（P1-02） | 钱包倾向同步 RPC（简单可靠），但如果财务审核系统是异步流水线处理，可能更适合异步。需对齐。 |
| 3-4-B | 驳回原因是否需要传给钱包？ | 钱包可以在解冻流水的 remark 中记录驳回原因，方便用户查看记录详情。 |
| 3-4-C | 部分解冻是否存在？ | 正常场景下解冻金额 = 原冻结金额（全额解冻）。是否存在部分驳回？如审核通过 80% 出款、驳回 20%？ |

### 3.2 财务暴露给钱包调用的 5 个 RPC

> **这 5 个接口的 IDL 目前不存在（ser-finance IDL 未创建），是我们最大的阻塞项。** 需要本次会议确定参数结构。

#### ① GetDepositMethods — 获取充值方式列表

```
场景：用户打开充值页面，钱包调此接口获取该币种可用的充值方式
频率：每次打开充值页调一次（高频）

钱包侧期望的入参：
  - currency_code  string  必填    用户选择的币种

钱包侧期望的返回：
  - methods  list<DepositMethod>
    - method_id      i64       方式ID（后续创建订单时传回）
    - method_name    string    方式名称（如"银行转账""USDT-TRC20"）
```

### 5.2 提现冻结时序——必须当场拍板（P0-11）

**这是本次沟通最重要的技术决策之一。**

```
方案 A：先冻结再建单（钱包侧强烈推荐）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  流程：wallet.Freeze() → finance.CreateWithdrawOrder()

  优点：
  ✅ 资金安全优先——先保护再建单
  ✅ 如果建单失败，余额只是暂时冻结（安全），定时任务可自动解冻
  ✅ 支付宝/微信/Stripe 的标准做法

  缺点：
  ⚠️ 建单失败时有"僵尸冻结"风险，需要补偿机制

  补偿方案：
  定时任务每 5 分钟扫描：已冻结 > 10分钟但无对应财务订单的数据 → 自动解冻

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
方案 B：先建单再冻结（不推荐）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  流程：finance.CreateWithdrawOrder() → wallet.Freeze()

  致命缺陷：
  ❌ 建单成功但冻结失败 → 审核通过后出款，但钱没冻住 → 资金损失
  ❌ 这个损失是不可逆的（钱已经出去了）
```

**❓ 请支付侧明确表态：接受方案 A 吗？**

## 7. 幂等与数据一致性策略约定

> 这是资金系统的生命线。必须双方都理解并遵守。

### 7.1 双层幂等防护

```
                           三方通道
                              │
                         回调（可能多次）
                              │
                              ▼
                    ┌──────────────────┐
                    │   ser-finance     │
                    │                  │
         第1层防线 → │  ① 查订单状态      │
                    │  if status=成功:  │ ← 已处理过，直接返回200
                    │    return 200    │
                    │  else:           │
                    │    处理回调逻辑    │
                    │    调钱包入账     │
                    └───────┬──────────┘
                            │
                      RPC（可能重试）
                            │
                            ▼
                    ┌──────────────────┐
                    │   ser-wallet     │
                    │                  │
         第2层防线 → │  ② DB唯一索引     │
                    │  UNIQUE(biz_type │ ← 重复insert直接冲突
                    │  ,order_no       │
                    │  ,wallet_type)   │
                    │                  │
                    │  冲突 → 返回成功  │ ← 不报错，返回已有余额
                    └──────────────────┘
```

**❓ 需要支付侧确认：**

| 编号 | 约定 | 确认 |
|------|------|------|
| **7-A** | 财务侧在调钱包入账前，先检查本地订单状态（第 1 层防线） | 是否已实现？ |
| **7-B** | 财务侧重试 WalletCredit 时用相同的 order_no | 必须保证——换 order_no 会导致重复入账 |
| **7-C** | 钱包侧 WalletCredit 重复调用返回 code=0（成功） | 即幂等返回，不报错。财务侧不应把"已入账"当作异常处理。 |

### 7.2 各场景一致性策略

| 场景 | 策略 | 责任方 | 补偿机制 |
|------|------|--------|---------|

## 9. 限额/手续费/配置归属问题

### 9.1 充值/提现限额由谁配置？（P0-08）

```
场景：用户充值 VND，最低 100,000 VND，最高 500,000,000 VND
场景：用户提现 VND，单笔最高 200,000,000 VND，每日最高 1,000,000,000 VND

❓ 这些限额配置在哪里？

选项 A：完全在财务侧（支付方式配置里已有限额字段）
  → 钱包 C 端展示限额时调 GetDepositMethods/GetWithdrawMethods 获取
  → 钱包侧不重复存储限额配置
  → 优点：一处配置，无同步问题
  → 缺点：钱包依赖财务 RPC 才能做校验

选项 B：钱包侧也有一套限额配置
  → 二次校验，更安全
  → 缺点：两处配置可能不一致

选项 C：限额在财务侧配置，但作为 GetDepositMethods 返回值带给钱包缓存
  → 钱包侧缓存限额做前置校验（快），财务侧做最终校验（准）

钱包侧倾向选项 C，请支付侧确认。
```

### 9.2 提现手续费计算（P0-09 + P0-16）

## 12. 异常场景与补偿机制约定

### 12.1 异常场景清单

| 场景 | 触发条件 | 后果 | 补偿方案 | 责任方 |
|------|---------|------|---------|--------|
| 充值回调丢失 | 三方通道未推送或网络异常 | 用户已付款但未到账 | 财务定时轮询三方状态 + 人工补单入口 | 财务 |
| 充值回调重复推送 | 三方多次推送/网络重试 | 可能重复入账 | 双层幂等（财务查状态+钱包唯一索引） | 双方 |
| WalletCredit RPC 超时 | 网络抖动/钱包服务繁忙 | 财务不确定是否入账成功 | 财务重试（相同 order_no）+ 钱包幂等 | 财务重试 |
| 提现冻结成功但建单失败 | 财务 RPC 异常 | 余额被悬挂冻结 | 钱包定时任务检测孤立冻结自动解冻 | 钱包 |
| 提现审核驳回解冻失败 | 钱包 RPC 异常 | 余额未解冻（用户资金被锁） | 财务重试 UnfreezeWallet + 定时任务 | 财务重试 |
| 提现出款成功但扣款失败 | WalletDebit RPC 异常 | 钱出去了但余额没扣 | 财务重试 WalletDebit + 人工介入 | 🔴 最危险 |
| 人工加减款入账失败 | 钱包 RPC 异常 | 审核通过但未执行 | 财务重试 + 人工重新操作 | 财务 |

### 12.2 RPC 超时与重试策略建议

```
┌───────────────────────────────┬─────────┬──────────┬───────────────┐
│          接口方向               │ 超时时间  │ 重试次数  │   重试间隔      │
├───────────────────────────────┼─────────┼──────────┼───────────────┤
│ 钱包→财务（查询类）              │  3s     │  0       │  不重试        │
│ 钱包→财务（创建订单）            │  5s     │  0       │  不重试,返回错误│
│ 财务→钱包（入账/扣款/冻结/解冻）  │  3s     │  3       │ 1s→2s→4s 退避 │
│ 钱包→ser-kyc（查询）            │  3s     │  0       │  不重试        │
│ 钱包→ser-user（查询）           │  3s     │  0       │  不重试        │
└───────────────────────────────┴─────────┴──────────┴───────────────┘

核心原则：
  · 写操作（入账/扣款/冻结/解冻）→ 由调用方负责重试，被调方保证幂等
  · 查询操作 → 不重试，直接返回错误给用户
  · 所有 RPC 都不应超过 5s（避免级联超时）
```
