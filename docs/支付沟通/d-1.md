# 支付沟通准备文档（D-26）

> **文档定位**：钱包模块负责人参加与支付/财务模块技术沟通会议的全面准备材料
> **核心目标**：消除双方盲区，暴露所有依赖和问题，敲定接口契约和技术约定
> **会议性质**：技术讨论交流，非需求评审——重点是"怎么做"而非"做不做"
> **基于**：24份分析文档 + 产品原型 + 工程代码扫描

---

## 第一章：会议背景与关键事实

### 1.1 关键事实（双方必须知道的前提）

1. **ser-wallet 和 ser-finance 都是全新模块**——工程中零足迹：
   - `common/idl/` 下无 ser-wallet/ 和 ser-finance/ 目录
   - `namesp.go` 无 wallet 和 finance 的 ETCD 常量（当前最高编号 #023 ffmpeg）
   - `rpc_client.go` 无 WalletClient() 和 FinanceClient() 工厂方法
   - 两边都从零开始，意味着**接口契约必须先于编码敲定**

2. **双向 RPC 依赖**——钱包调财务，财务也调钱包：
   - 钱包 → 财务：获取支付方式、匹配渠道创建订单（充值/提现页面需要）
   - 财务 → 钱包：入账、扣款、冻结、解冻、查余额（回调/审核/手动调整需要）
   - 不是单向依赖，**双方都是对方的上游**

3. **没有分布式事务框架**——项目明确不用 Seata/DTM：
   - 跨服务一致性靠：幂等RPC + 超时重试 + 定时补偿（"三板斧"）
   - 这意味着双方必须严格约定幂等键格式和重试策略

4. **双方都处于预评审阶段**——大方向已定，细节可能变：
   - 今天的沟通目标是敲定"确定性骨架"，非每个字段的最终定义
   - 但 P0 级约定必须当场敲定，否则双方都无法开始编码

### 1.2 双方模块职责边界

```
┌─────────────────────────────────────────────────────────────────┐
│                    完整资金链路                                    │
│                                                                   │
│  用户 ──→ [ser-wallet: 创建订单/冻结]                              │
│              │                                                    │
│              ├──→ [ser-finance: 渠道匹配/支付对接/审核]             │
│              │         │                                          │
│              │         ├──→ [第三方支付渠道]                       │
│              │         │         │                                │
│              │         ←── 回调 ─┘                                │
│              │                                                    │
│              ←── [ser-finance: RPC调钱包入账/解冻/扣冻结]          │
│                                                                   │
│  ser-wallet 职责：余额管理、订单生命周期、用户交互                   │
│  ser-finance 职责：渠道对接、资金审核、回调处理、手动调整             │
└─────────────────────────────────────────────────────────────────┘
```

| 维度 | ser-wallet（我方） | ser-finance（对方） |
|------|-------------------|-------------------|
| 核心职责 | 多币种余额管理、币种配置、汇率 | 支付渠道对接、资金审核、回调处理 |
| 数据主权 | wallet_account, currency_config, transaction_log | deposit_method, withdraw_method, channel_order, channel_config |
| 对外暴露 | 9个RPC（余额操作+币种查询） | 3-4个RPC（支付方式+渠道匹配+发起代付） |
| 用户交互 | C端23个HTTP接口（充值/提现/兑换页面） | 主要是B端管理（审核/对账/渠道管理） |
| 定时任务 | 汇率更新、对账校验 | 回调补偿、渠道状态巡检 |

## 第二章：我方需要对方提供什么（钱包 → 财务 RPC）

### 2.1 接口总览

我方共需调用财务模块 **3-4个RPC接口**，按业务场景分组：

#### D-01：GetPaymentMethods — 获取支付方式列表

- **调用场景**：用户打开充值页面（C-2）、用户打开提现页面（C-7）
- **调用频率**：高频（每次用户打开页面都调一次）
- **我方入参需求**：

```
GetPaymentMethodsReq {
    currencyCode: string    // "VND" / "THB" / "IDR" / "USDT"
    type: string            // "deposit" 或 "withdraw"
}
```

- **我方期望返回**：

```
GetPaymentMethodsResp {
    methods: list<PaymentMethodInfo> {
        methodId: i64            // 方式唯一ID（我方创建订单时回传）
        methodName: string       // 显示名称（如"银行转账"、"电子钱包"）
        methodIcon: string       // 图标URL
        methodType: i32          // 1=银行卡 / 2=电子钱包 / 3=USDT
        quickAmounts: list<i64>  // 快捷金额选项（如 [100000, 500000, 1000000]）
        minAmount: i64           // 单笔最小金额
        maxAmount: i64           // 单笔最大金额
        feeRate: optional string // 手续费率（如有）
        dailyLimit: optional i64 // 每日限额（如有）
        sortOrder: i32           // 前端排序序号
        isRecommended: bool      // 是否推荐标签
    }
}
```

- **需要确认的问题**：
  - [ ] methodId 由财务生成还是配置平台？我方只需透传回去
  - [ ] quickAmounts 是最小单位整数还是字符串？
  - [ ] 充值和提现共用一个接口 + type 区分，还是拆成两个独立接口？
  - [ ] 我方需要缓存这个结果吗？还是每次实时调？如果缓存，失效策略是什么？
  - [ ] 财务不可用时，我方应该返回缓存数据还是直接报错？

#### D-02：MatchAndCreateChannelOrder — 匹配渠道并创建渠道订单

- **调用场景**：用户确认充值、提交订单时（C-3 CreateDepositOrder）
- **调用频率**：中频（每次充值提交调一次）
- **这是充值流程的关键路径**——这个调用失败 = 整个充值订单创建失败

- **我方入参需求**：

```
MatchAndCreateChannelOrderReq {
    depositMethodId: i64    // 用户选择的支付方式ID（从D-01返回中获取）
    currencyCode: string    // 币种
    amount: i64/string      // 充值金额（待定类型）
    platformOrderNo: string // 平台订单号（C+16位，我方生成）
    userId: i64             // 用户ID
}
```

- **我方期望返回**：

```
MatchAndCreateChannelOrderResp {
    channelOrderNo: string       // 渠道侧订单号（我方存储关联）
    channelName: string          // 渠道名称（记录用）
    paymentType: i32             // 1=跳转支付 / 2=扫码 / 3=链上地址
    paymentUrl: optional string  // 跳转URL（type=1时）
    qrCodeUrl: optional string   // 二维码内容/URL（type=2时）
    walletAddress: optional string // 链上地址（type=3，USDT时）
    expireTime: i64              // 支付过期时间戳
}
```

- **需要确认的问题**：
  - [ ] 渠道匹配失败（所有渠道都不可用），返回错误码是什么？我方需要给用户什么提示？
  - [ ] 是否需要我方先创建平台订单再调这个接口？还是调成功后再创建？（**顺序关键**）
  - [ ] platformOrderNo 对方需要存储吗？后续回调时会回传这个号吗？
  - [ ] USDT 链上充值走这个接口还是走独立流程？
  - [ ] 超时时间建议多少？3s 还是 5s？（渠道对接可能慢）

#### D-03：InitiateChannelPayout — 发起渠道代付（提现出金）

- **调用场景**：提现审核通过后，发起实际出金
- **核心疑问：这个接口谁来调？**

```
场景A（推荐）：财务审核通过后，财务自己内部发起代付
  → 钱包不需要调这个接口
  → 钱包只需被动等待财务回调结果（DeductFrozen 或 Unfreeze）

场景B：财务审核通过后，通知钱包，钱包再调这个接口
  → 钱包需要调这个接口
  → 多一次跨服务调用，增加复杂度
```

- **需要确认的问题**：
  - [ ] **提现代付是财务自行发起还是钱包触发？** 这直接决定接口设计
  - [ ] 如果是场景A，财务审核通过后如何通知钱包状态变更？推模式(RPC回调) 还是 拉模式(钱包定时轮询)？
  - [ ] 代付成功/失败的回调链路是什么？第三方→财务→(RPC)→钱包？

#### D-04：GetChannelOrderStatus — 查询渠道订单状态

- **调用场景**：用户手动刷新订单状态（C-16）、定时补偿任务扫描超时订单
- **调用频率**：低频
- **入参**：channelOrderNo（从 D-02 返回中获取）
- **返回**：status + updateTime + actualAmount(可选)

- **需要确认的问题**：
  - [ ] 这个接口存在吗？还是我方直接看充值订单状态就行（财务通过回调已更新）？
  - [ ] 如果存在，状态枚举值怎么定义？

## 第三章：对方需要我方提供什么（财务 → 钱包 RPC）

### 3.1 接口总览

对方共需调用我方 **9个RPC接口**（5个写 + 4个读），全部走 Kitex Thrift RPC：

### 3.2 写入类接口（5个，资金操作）

#### R-01：CreditBalance — 入账（加钱）

- **对方调用场景**：
  - 充值回调成功 → 中心钱包入账（opType=deposit）
  - 充值有活动奖励 → 奖励钱包入账（opType=deposit_bonus，附带审核信息）
  - 补单审核通过 → 中心钱包入账（opType=supplement）
  - 加款审核通过 → 最多4个钱包入账（opType=manual_add，同一 orderNo 不同 walletType）

```
CreditBalanceReq {
    1: required i64    userId
    2: required string currencyCode    // "VND"/"THB"/"USDT"/"BSB"
    3: required i32    walletType      // 1=中心 / 2=奖励 / 3=主播 / 4=代理
    4: required i64    amount          // 最小单位整数（待定）
    5: required string refOrderNo      // 来源订单号（C+16/B+16/A+16）
    6: required i32    txType          // 交易类型枚举
    7: optional string remark          // 备注
    8: optional i32    auditMultiplier // 审核倍率（仅奖励钱包）
    9: optional i64    auditTurnover   // 需完成流水（仅奖励钱包）
}

CreditBalanceResp {
    // 空响应，成功=无错误，失败=业务错误码
}
```

- **幂等机制**：DB 唯一索引 `uk_ref_order_tx(ref_order_no, tx_type, wallet_type)`
  - 重复调用 → 触发唯一索引冲突 → 我方返回**成功**（非错误）
  - 对方超时重试是安全的
- **加款场景特殊说明**：同一个 A+16 订单号，对方可能调4次（4个walletType），这不是重复——幂等键包含 walletType

#### R-02：DebitBalance — 扣款（减钱）

- **对方调用场景**：
  - 减款审核通过 → 从指定钱包扣除（opType=manual_deduct）
  - 提现最终确认 → 从冻结余额扣除（opType=withdraw_confirm）

```
DebitBalanceReq {
    1: required i64    userId
    2: required string currencyCode
    3: required i32    walletType
    4: required i64    amount
    5: required string refOrderNo
    6: required i32    txType
    7: optional string remark
}
```

- **余额不足**：返回 `BALANCE_NOT_ENOUGH` 错误
- **对方注意**：减款审核页面应先调 R-06 查余额，确认够再提交审核

#### R-03：FreezeBalance — 冻结余额

- **设计共识**：提现冻结由**钱包内部完成**（创建提现订单时原子操作：冻结 + 创建订单在同一个DB事务）
- **但仍然暴露为RPC**：以备其他场景需要（如风控系统临时冻结）
- **操作**：available -= amount, frozen += amount（总余额不变）

```
FreezeBalanceReq {
    1: required i64    userId
    2: required string currencyCode
    3: required i32    walletType
    4: required i64    amount
    5: required string refOrderNo
}
```

#### R-04：UnfreezeBalance — 解冻余额

- **对方调用场景**：
  - 风控审核拒绝 → 解冻，资金回到可用余额
  - 财务审核拒绝 → 解冻，资金回到可用余额
- **操作**：frozen -= amount, available += amount（总余额不变，钱还给用户）

```
UnfreezeBalanceReq {
    1: required i64    userId
    2: required string currencyCode
    3: required i32    walletType
    4: required i64    amount
    5: required string refOrderNo      // 原提现订单号 T+16
}
```

- **冻结不足**：返回 `FROZEN_NOT_ENOUGH` 错误

#### R-05：ManualAdjust — 多钱包原子调整

- **对方调用场景**：B端手动加款/减款，一次操作多个钱包类型
- **设计目标**：替代多次 CreditBalance/DebitBalance 调用，保证原子性

```
ManualAdjustReq {
    1: required i64    userId
    2: required string currencyCode
    3: required string refOrderNo      // 调整单号
    4: required list<AdjustItem> adjustments
}

AdjustItem {
    1: required i32    walletType      // 1=中心 / 2=奖励 / 3=主播 / 4=代理
    2: required i32    direction       // 1=加 / 2=减
    3: required i64    amount
}

ManualAdjustResp {}
```

- **原子性**：所有调整在一个DB事务内完成，任一钱包余额不足 → 全部回滚
- **需要确认的问题**：
  - [ ] 是否需要这个原子接口？还是对方分多次调 CreditBalance/DebitBalance 就行？
  - [ ] 如果分多次调，部分成功部分失败怎么处理？（如中心钱包加成功，奖励钱包加失败）

### 3.3 读取类接口（4个）

#### R-06：GetBalance — 查询用户余额

```
GetBalanceReq {
    1: required i64    userId
    2: optional string currencyCode    // 不传=返回所有币种
}

GetBalanceResp {
    1: required list<WalletBalanceInfo> wallets
}

WalletBalanceInfo {
    1: required string currencyCode
    2: required i32    walletType
    3: required i64    availableBalance
    4: required i64    frozenBalance
    5: required i64    totalBalance
}
```

- **对方使用场景**：补单/加款/减款表单展示当前余额
- **注意**：余额数据不走缓存，每次实时查DB（"资金数据宁慢勿错"）

#### R-07：BatchGetBalance — 批量查询余额

- **对方使用场景**：运营报表、用户列表展示
- **优先级**：P2，可后期实现

#### R-08：GetActiveCurrencyList — 获取启用币种列表

- **对方使用场景**：配置支付方式时需要知道平台支持哪些币种
- **返回**：币种代码、名称、精度、状态、汇率等

#### R-09：GetExchangeRate — 获取汇率

- **对方使用场景**：统计报表中的跨币种金额换算
- **返回**：指定币种对基础币种(USDT)的当前汇率

## 第四章：核心业务链路协作详解

### 4.1 充值链路（Deposit Chain）

```
时序图：

用户                ser-wallet              ser-finance              第三方渠道
 │                     │                       │                       │
 │──打开充值页面──→     │                       │                       │
 │                     │──GetPaymentMethods──→  │                       │
 │                     │←──方式列表──────────── │                       │
 │←──展示充值页面──── │                       │                       │
 │                     │                       │                       │
 │──确认充值──────→    │                       │                       │
 │                     │ 1.参数校验              │                       │
 │                     │ 2.重复充值检测           │                       │
 │                     │ 3.锁定汇率快照           │                       │
 │                     │──MatchAndCreate──────→ │                       │
 │                     │                       │──创建渠道订单──────→   │
 │                     │                       │←──支付信息────────── │
 │                     │←──渠道订单号+支付信息── │                       │
 │                     │ 4.创建平台订单(status=pending)                   │
 │←──跳转支付──────── │                       │                       │
 │                     │                       │                       │
 │──────────在第三方页面支付──────────────────────────────────→          │
 │                     │                       │                       │
 │                     │                       │←──支付成功回调──────  │
 │                     │                       │ 1.验签                 │
 │                     │                       │ 2.更新渠道订单状态      │
 │                     │←──CreditBalance(R-01)──│                       │
 │                     │ 入账：available += amount                       │
 │                     │ 写交易流水                                       │
 │                     │──返回成功──────────→   │                       │
 │                     │                       │──返回success──────→   │
```

**关键协作点**：

| 序号 | 协作点 | 需要确认 |
|------|-------|---------|
| ① | GetPaymentMethods 入参/出参字段 | 对方确认字段定义 |
| ② | MatchAndCreateChannelOrder 入参需要带什么 | platformOrderNo 对方是否存储和回传 |
| ③ | 创建平台订单的时机：先调渠道成功后再创建？还是先创建再调渠道？ | **顺序必须对齐** |
| ④ | 回调时对方回传什么参数调 CreditBalance | 完整参数映射 |
| ⑤ | 充值有活动奖励时，是对方算好奖励金额调两次 CreditBalance，还是我方算？ | **职责边界** |
| ⑥ | USDT链上充值是否走同一套流程？区别在哪？ | 链上 vs 法币差异 |

### 4.2 提现链路（Withdrawal Chain）

```
时序图：

用户                ser-wallet              ser-finance              第三方渠道
 │                     │                       │                       │
 │──打开提现页面──→     │                       │                       │
 │                     │──GetPaymentMethods──→  │                       │
 │                     │ + ser-kyc.KycQuery     │                       │
 │                     │←──提现方式列表──────── │                       │
 │←──展示提现页面──── │                       │                       │
 │                     │                       │                       │
 │──提交提现──────→    │                       │                       │
 │                     │ 1.KYC校验 + 姓名匹配    │                       │
 │                     │ 2.限额校验（单笔+每日）  │                       │
 │                     │ 3.原子操作：             │                       │
 │                     │   available -= amount   │                       │
 │                     │   frozen += amount      │                       │
 │                     │   创建提现订单(T+16)     │                       │
 │                     │──推送订单到财务审核──→   │                       │
 │←──提现申请已提交── │                       │                       │
 │                     │                       │                       │
 │                     │          ┌─ 风控审核 ─┐                       │
 │                     │          │ 通过 / 拒绝│                       │
 │                     │          └────────────┘                       │
 │                     │                       │                       │
 │                     │  [拒绝]                │                       │
 │                     │←──UnfreezeBalance(R-04)│                       │
 │                     │  frozen -= amount       │                       │
 │                     │  available += amount    │                       │
 │                     │                       │                       │
 │                     │  [通过→财务审核→通过]    │                       │
 │                     │                       │──发起代付──────────→  │
 │                     │                       │                       │
 │                     │                       │←──代付成功回调──────  │
 │                     │←──DebitBalance(R-02)── │                       │
 │                     │  frozen -= amount       │  (扣除冻结,资金离开)    │
 │                     │                       │                       │
 │                     │                       │←──代付失败──────────  │
 │                     │  [保持冻结，等待人工处理]  │                       │
```

**关键协作点**：

| 序号 | 协作点 | 需要确认 |
|------|-------|---------|
| ① | 提现订单创建后如何推送到财务审核？RPC？消息队列？状态轮询？ | **通知机制** |
| ② | 审核拒绝时，对方调 UnfreezeBalance 的参数 | refOrderNo = 原提现订单号 T+16 |
| ③ | 代付成功后，对方调什么接口？DebitBalance(从frozen扣)？还是需要独立的DeductFrozenBalance？ | **接口选择** |
| ④ | 代付失败处理：保持冻结等人工？还是自动解冻？ | **行业惯例是保持冻结** |

### 4.4 三层回调保障机制

充值和提现都依赖第三方回调，但回调可能丢失，需要三层保障：

```
第一层：第三方主动回调（覆盖95%+场景）
  → 财务接收回调 → 验签 → 调钱包入账/扣冻结

第二层：定时补偿扫描（覆盖4%场景）
  → 谁扫？财务扫还是钱包扫？
  → 扫到超时订单 → 主动查询第三方状态 → 补偿处理
  → 需要调用 D-04 GetChannelOrderStatus

第三层：B端人工补单（覆盖1%场景）
  → 运营在财务B端创建补单 → 审核通过 → 调钱包CreditBalance
```

- **需要确认**：
  - [ ] 第二层补偿扫描由哪方执行？建议财务执行（因为渠道状态在财务侧）

## 第十章：我方的底线与弹性空间

### 10.1 不可让步的底线

| 底线 | 理由 |
|------|------|
| 余额操作必须走 RPC，不允许财务直接操作钱包DB | 数据主权、事务一致性、审计追踪 |
| 幂等机制由钱包侧实现（DB唯一索引） | 幂等保护必须在数据写入侧，而非调用侧 |
| 提现冻结由钱包内部原子操作完成 | 冻结+创建订单必须同一事务，跨服务做不到原子 |
| CreditBalance/DebitBalance 的写入语义不可降级为异步 | 资金操作必须同步确认 |
| DUPLICATE_ORDER 返回成功语义 | 幂等设计核心约定，否则调用方重试逻辑会出错 |

### 10.2 可以灵活协商的部分

| 弹性点 | 备选方案 |
|--------|---------|
| 金额 RPC 传输类型 | 我方倾向 i64 但可接受 string（我方做转换层） |
| ManualAdjust 原子接口 | 如对方不需要，可退回到多次 CreditBalance/DebitBalance |
| 提现订单状态维护者 | 可以我方维护，也可以双方各维护一份（以我方为准） |
| GetChannelOrderStatus 是否需要 | 如回调足够可靠，可以延后实现 |
| 提现推送机制 | RPC/MQ/轮询 都可以讨论 |

**如果对方坚持 string**：
- 我方在 RPC 层增加 adapter，入参 string→i64 转换，出参 i64→string 转换
- 转换层统一在钱包侧实现，内部始终用 i64
- 代价：每次调用多一层转换，但精度安全

**如果对方坚持 DECIMAL**：
- DB 层面我方不接受 DECIMAL（ADR-2已决定 BIGINT）
- 但 Thrift i64 和 string 都能承载，这是传输层问题不是存储层问题
- 最终数据进入钱包后都会转成 BIGINT

### 11.2 谁来维护提现订单状态

**如果对方想要维护**：
- 可以各维护一份，但必须约定**钱包侧状态为最终真实状态**
- 对方维护的是"审核流程状态"，我方维护的是"资金状态"
- 两个状态机可以不完全一致，但必须有映射表

**如果对方不想维护**：
- 对方审核结果通过 RPC 通知我方，我方更新状态
- 更简单，推荐

### 11.3 代付失败处理

**如果对方想自动解冻**：
- 风险：渠道可能实际已付出但回调返回失败（网络问题），自动解冻=用户可重复提现=平台亏损
- 我方建议：保持冻结，人工核实后再决定解冻还是扣除
- 这是行业标准做法（支付宝、微信支付、Stripe 都这么做）

### 11.4 回调参数映射

**可能的分歧**：对方回调调 CreditBalance 时，参数从哪来？
- platformOrderNo → refOrderNo：必须回传
- amount：用渠道实际到账金额还是平台订单金额？（可能有差异）
- txType：对方负责传正确的枚举值，我方校验
