## 一、开场定调：双方角色和协作全景

### 1.1 三句话说清楚我们各自干什么

```
钱包模块（我负责）= "账本 + 记账员"
  → 管用户余额、管交易流水、管币种配置、管汇率
  → 所有资金变动最终都在钱包这里完成余额增减和流水记录

支付/财务模块（你们负责）= "管道 + 搬运工 + 审核员"
  → 管支付通道、管三方对接、管回调验签、管审核流程
  → 所有资金进出平台的"物理搬运"都在财务这里完成

币种配置（我负责）= "度量衡"
  → 管币种定义、汇率规则、精度规范
  → 钱包和财务都从这里获取币种和汇率信息
```

**一句话总结分工：你们负责把钱搬进搬出平台（对接三方支付），我负责在平台内部记好账（余额管理）。**

## 二、议题一：充值链路——订单创建与回调谁来做

### 2.1 我理解的充值完整链路

```
C端用户点充值
  │
  ▼
① 钱包侧（C-4 CreateDepositOrder）
  ├→ 执行重复转账拦截（查钱包历史订单：同用户+同币种+同金额）
  ├→ 关联奖金活动（如有）
  ├→ 创建钱包侧预订单（C+16位订单号，status=待分配通道）
  │
  ▼
② ???（这里是关键问题）
  ├→ 方案A：钱包RPC调财务 → 财务分配通道+对接三方 → 返回支付链接
  ├→ 方案B：钱包创建完订单后，前端直接调财务接口去发起支付
  ├→ 方案C：其他？
  │
  ▼
③ 用户跳转三方支付页面完成支付
  │
  ▼
④ 三方支付推送回调
  ├→ 回调接收方：财务（验签+更新通道记录）
  ├→ 财务验签通过后 → RPC调钱包 R-6 CreditDeposit → 钱包上账
  │
  ▼
⑤ 钱包完成上账
  ├→ 充值金额 → 中心钱包
  ├→ 奖金金额 → 奖励钱包
  ├→ 创建交易流水
  └→ 更新充值订单状态 → 已完成
```

### 2.2 需要和支付组澄清的问题

**问题2.2.1：步骤②——钱包创建完预订单后，如何触发三方支付？**

```
我的疑问：
  用户选好充值方式和金额后，需要"分配支付通道"并"对接三方获取支付链接"。
  分配通道和对接三方是你们（财务）的职责。
  但这个动作是：
    A. 钱包后端RPC调财务后端？（后端→后端）
    B. 前端先调钱包创建预订单，再调财务发起支付？（前端→后端×2）
    C. 钱包创建完订单后异步通知财务？（后端→MQ→后端）

  如果是A：
    - 钱包需要知道调你们哪个RPC方法（AllocateChannelAndPay？）
    - 钱包传什么参数给你们？（订单号+用户ID+币种+金额+充值方式？）
    - 你们返回什么给钱包？（支付链接/收款信息？）
    - 这个RPC调用超时怎么处理？（三方API可能慢）

  如果是B：
    - 前端需要拿到钱包的预订单号，再传给财务
    - 两次调用之间的原子性怎么保证？（钱包创建了订单但财务发起失败）
    - 财务需要校验这个预订单号是否合法吗？

  这个问题是充值链路的核心，必须搞清楚。
```

**问题2.2.2：充值回调确认由财务接收？**

```
我的理解：
  三方支付回调 → 打到财务的HTTP端点 → 财务验签 → 更新通道记录 → RPC调钱包上账

  如果这个理解正确：
  - 钱包不需要暴露任何HTTP回调端点（钱包只提供RPC供财务调用）
  - 回调验签所需的通道密钥/签名规则归财务管理
  - 如果回调推送的订单号和钱包预订单号不一致，怎么映射？
    （你们有自己的支付订单号？和钱包的C+16位订单号是什么关系？）

  请确认：
  ① 回调确实是你们接收吗？
  ② 你们调我的 CreditDeposit 时，传的是钱包的订单号还是你们的订单号？
  ③ 如果你们验签通过但调我的RPC失败（网络超时），你们的重试策略是什么？
```

**问题2.2.3：充值回调可能的异常场景**

```
想讨论几个充值异常场景的处理方式：

  场景A：用户创建了订单但没有支付（三方没有回调）
    → 这个订单最终怎么关闭？是你们超时关闭还是我这边超时关闭？
    → 超时时间是多久？（我理解15-30分钟？）

  场景B：三方重复推送回调（同一笔支付推了2次甚至10次）
    → 我这边CreditDeposit基于订单号做幂等（同一订单号只入一次账）
    → 你们那边需要也做去重吗？还是直接都推给我让我过滤？

  场景C：用户支付了但三方回调丢失或延迟
    → 是否有补偿机制？（主动查询三方订单状态？）
    → 补偿查询是你们做还是需要钱包配合？

  场景D：回调金额和订单金额不一致（用户实际付多了或少了）
    → 这个校验在你们那边做还是传到我这边？
    → 如果不一致怎么处理？（拒绝入账？按实际金额入账？）
```

## 三、议题二：提现链路——冻结/审核/出款的职责切分

### 3.1 我理解的提现完整链路

```
C端用户提交提现
  │
  ▼
① 钱包侧（C-9 CreateWithdrawalOrder）
  ├→ 校验KYC状态
  ├→ 校验限额（单笔/单日）
  ├→ 校验余额充足
  ├→ 校验资金来源合规（法币充→法币提，USDT充→USDT提）
  ├→ BalanceEngine.Freeze()：立即冻结提现金额
  ├→ 创建提现订单（T+16位，status=处理中）
  │
  ▼
② 通知财务进入审核流程 ← 这里怎么通知？
  ├→ 方案A：钱包RPC调财务 SubmitForReview
  ├→ 方案B：财务定时拉取钱包的待审核提现订单
  ├→ 方案C：通过消息队列（Kafka/NATS）
  │
  ▼
③ 财务三级审核（风控→财务→出款）
  │
  ├→ 审核拒绝 → 财务RPC调钱包 R-8 UnfreezeForWithdrawal
  │              钱包解冻余额，订单状态→已拒绝
  │
  └→ 审核通过 → 财务发起出款（对接三方支付）
      │
      ├→ 出款成功回调 → 财务RPC调钱包 R-9 ConfirmWithdrawalDeduct
      │                  钱包确认扣除冻结额，订单状态→已完成
      │
      └→ 出款失败回调 → 财务RPC调钱包 R-8 UnfreezeForWithdrawal
                         钱包解冻余额，订单状态→已失败
```

### 3.2 需要和支付组澄清的问题

**问题3.2.1：步骤②——提现订单如何进入你们的审核流程？**

```
钱包创建了提现订单并冻结了余额之后，你们怎么知道有一笔新的提现要审核？

  方案A：我主动推给你们（RPC调用）
    → 需要你们提供一个接口（如 SubmitWithdrawalForReview）
    → 参数：提现订单号、用户ID、币种、金额、提现方式、提现账户信息
    → 如果这个RPC调用失败了怎么办？（钱包已冻结但财务不知道有这笔提现）

  方案B：你们主动来拉（定时查询/事件监听）
    → 我提供一个RPC查询接口（如 GetPendingWithdrawals）
    → 你们定时拉取待处理的提现订单
    → 实时性较差，但解耦更好

  方案C：消息队列（Kafka/NATS）
    → 钱包发消息到topic，财务消费
    → 实时性好，解耦也好，但需要保证消息不丢

  我倾向方案A或C，需要你们确认。
```

**问题3.2.2：冻结时机确认——提交即冻结**

```
我的设计：用户提交提现的那一刻就冻结余额。

  理由：
  - 如果审核通过后才冻结，审核期间用户可能把钱花掉，审核通过时余额不足
  - 提交即冻结保证"审核通过一定有钱"

  对你们的影响：
  - 你们审核拒绝时，需要调我的 UnfreezeForWithdrawal 解冻
  - 你们出款失败时，也需要调我的 UnfreezeForWithdrawal 解冻
  - 也就是说，凡是提现最终不成功，你们都要负责通知我解冻

  请确认：
  ① 你们认可"提交即冻结"的方案吗？
  ② 你们有没有场景是"先审核再冻结"的？
  ③ 审核拒绝/出款失败后，你们调我解冻的时效性保证？（审核拒绝后多久调我？）
```

**问题3.2.3：三级审核的具体流程**

```
需求文档提到"风控审核→财务审核→出款审核"三级。
但从我的角度，只需要知道两个终态：

  终态1：审核/出款成功 → 你们调我 ConfirmWithdrawalDeduct（确认扣款）
  终态2：审核/出款失败 → 你们调我 UnfreezeForWithdrawal（解冻退回）

  中间的三级审核过程对我透明（我不关心是哪一级拒绝的）。

  请确认：
  ① 我的理解对吗？我只需要接收你们的终态通知？
  ② 还是你们每一级审核状态变更都需要通知我？（如果需要我更新订单状态为"风控审核中/财务审核中/出款中"？）
  ③ 审核锁定/解锁功能是怎么回事？（我看到需求里有"锁定/解锁/强制解锁"）
      → 这个锁定是锁定审核记录不让别人处理？还是锁定用户不让再提现？
```

**问题3.2.4：出款回调流程**

```
出款（三方代付）成功/失败的回调，处理流程是什么？

  我的理解：
  三方代付回调 → 打到财务 → 财务验签 → 成功则调我ConfirmDeduct / 失败则调我Unfreeze

  需要确认：
  ① 出款回调也是你们接收？（和充值回调一样？）
  ② 如果三方代付超时没有回调怎么处理？（你们主动查询？）
  ③ 你们调我ConfirmDeduct时，传的订单号是我的T+16位还是你们自己的？
```

### 4.1 币种查询接口（4个，钱包→财务方向，你们调我查询）

**R-1 GetCurrencyList（获取已启用币种列表）**

```
你们的使用场景：

  - 通道配置页面的币种选择
  - 支付配置页面的币种选择

我的返回：
  [
    {
      currency_code: "USDT",       // 币种编码
      currency_name: "Tether",      // 币种名称

      decimal_digits: 8,            // 小数位数

  ]

  需要确认：
  ① 这些字段够用吗？你们还需要别的字段吗？
  ② 你们需要禁用的币种吗？（默认只返回启用的）
  ③ 你们调用频率大概多少？（我好评估是否需要给你们返回缓存版本）
```

**R-2 GetExchangeRate（获取指定币种汇率）**

```
你们的使用场景：
  - 金额换算（法币⟷基准币种）
  - 统计报表换算

我的返回：
  {
    currency_code: "VND",
    rate_realtime: 2543250000000,   // 实时汇率（×10^8）
    rate_platform: 2540000000000,   // 平台汇率（×10^8）
    rate_deposit: 2552000000000,    // 入款汇率（×10^8）
    rate_withdrawal: 2528000000000  // 出款汇率（×10^8）
  }

  注意：所有汇率值都是 ×10^8 的 bigint。

**R-6 CreditDeposit（充值入账）——最重要的接口**

## 五、议题四：数据归属——订单表建在谁那边

### 5.1 问题背景

```
充值订单和提现订单的数据，在哪个服务的数据库中建表存储？

  现状：
  - C端有"充值记录"和"提现记录"页面 → 数据展示在钱包模块
  - B端有"充值记录管理"和"提现记录管理"页面 → 数据展示在财务模块
  - 两边都要展示同一份数据
```

### 5.2 我的方案建议

```
方案（推荐）：各自建表，各管各的维度

  钱包侧建"平台订单表"：
    - deposit_order / withdrawal_order
    - 管订单号、用户ID、金额、币种、状态、时间
    - 状态由钱包维护：待支付→支付中→已完成/已失败/已关闭
    - C端记录查询直接查自己的表

  财务侧建"通道订单表"：
    - 管通道ID、三方订单号、通道费率、审核记录、出款记录
    - 状态由财务维护：待分配→已分配→回调成功/回调失败
    - B端记录管理查自己的表

  关联关系：
    两边通过钱包的 order_no（C+16/T+16）做外键关联
    你们的表里存一份我的 order_no，我的表里也可以存一份你们的 channel_order_no

  好处：
  - 各管各的表，不双写，不需要跨服务同步状态
  - 数据一致性通过 RPC 调用保证（终态通知）
  - C端查充值记录不需要调财务（我查自己的表）
  - B端查充值管理不需要调钱包（你们查自己的表+需要用户信息时调我查）

  代价：
  - 两边数据可能短暂不一致（回调到你们但还没通知我的窗口期）
  - 但这个窗口很短（毫秒级），且最终一致

  需要确认：
  ① 你们认可"各自建表"的方案吗？
  ② 你们的B端充值/提现记录页面需要展示哪些钱包侧的信息？
     → 如果需要用户余额等信息，可以调我的 R-5 GetUserBalance
  ③ 你们的表会存我的 order_no 吗？（方便对账）
```

## 七、议题六：USDT充值——特殊链路问题

### 7.1 USDT充值和法币充值的区别

```
法币充值（银行转账/电子钱包）：
  → 跳转三方支付页面 → 用户完成支付 → 三方回调 → 入账
  → 纯线上流程，你们对接三方完成

USDT充值：
  → 展示收款地址+二维码 → 用户链上转账 → 链上确认 → 入账
  → 涉及区块链交互（地址生成、链上监听、到账确认）
```

### 7.2 需要和支付组澄清的问题

```
问题7.2.1：USDT收款地址由谁管理？

  可能的方案：
  A. 有独立的区块链服务（链上监听+地址管理）→ 和法币走不同链路
  B. USDT也走三方支付通道（如 NOWPayments/Coinbase Commerce）→ 和法币一样的流程
  C. 平台固定地址（人工管理）→ 钱包从配置读取
  D. 你们（财务）管理USDT收款地址

  不同方案对我的影响：
  - 方案A/D：我的C-5（USDT充值地址获取）需要调你们或区块链服务获取地址
  - 方案B：USDT充值和法币充值走同样的流程（你们返回三方跳转页面），
           我的C-5接口就不需要了
  - 方案C：地址写在配置里，我直接返回

  这个直接决定了USDT充值的接口设计。请告知你们的方案。

### 10.1 金额单位规范

```
提议：双方统一使用 bigint 最小单位，不传小数

  法定货币（VND/IDR/THB）：× 100（保留2位小数精度）
    → 1000.50 VND = 100050（bigint i64）
  加密货币（USDT）：× 10^8（保留8位小数精度）
    → 100.50000000 USDT = 10050000000（bigint i64）
  平台币（BSB）：× 100
    → 50.00 BSB = 5000（bigint i64）

  汇率：统一 × 10^8
    → 1 USDT = 25432.50 VND → rate = 2543250000000（bigint i64）

  传输层：全部用 i64（Thrift）
  展示层：各自按 decimal_digits 格式化给前端

  需要确认：你们是否也用这套规范？如果你们用 float/decimal，我们需要在哪一层做转换？
```

### 10.2 币种编码规范

```
统一使用 currency_config.currency_code，由币种配置表定义：
  - USDT / VND / IDR / THB / BSB

  你们的通道配置、支付方式配置中的币种字段，也统一用这个编码。
  避免出现一方用"VND"一方用"vnd"或"704"的情况。

  需要确认：你们的现有设计中币种编码是什么格式？
```

### 10.3 订单号规范

```
钱包侧订单号规则：
  C + 16位（充值订单）     → 如 C1234567890123456
  T + 16位（提现订单）     → 如 T1234567890123456
  B + 16位（补单）         → 如 B1234567890123456
  A + 16位（人工加款）     → 如 A1234567890123456
  M + 16位（人工减款）     → 如 M1234567890123456
  E + 16位（兑换订单）     → 如 E1234567890123456

  16位生成规则：时间戳+随机数/雪花ID（具体算法待定）

  问题：
  ① 你们有自己的订单号规范吗？格式是什么？
  ② 我们双方的订单号需要互相存储吗？（你们表里存我的order_no，我表里存你们的channel_order_no）
  ③ 人工加减款和补单的订单号由谁生成？

     → 还是你们调我的时候我自己生成？

## 十一、议题十：错误码分段与异常处理

### 11.1 错误码分段

```
提议：
  钱包模块：6022xxx 段
  财务模块：6023xxx 段

  钱包常见错误码：
  - 6022001：余额不足
  - 6022002：订单已处理（幂等拒绝）
  - 6022003：订单状态不允许此操作
  - 6022004：冻结金额不足
  - 6022005：币种不存在或已禁用
  - 6022006：超过限额
  - 6022007：KYC未认证
  - 6022008：资金来源不符合提现规则

  需要确认：
  ① 你们的错误码段是 6023xxx 吗？
  ② 你们调我的RPC时，需要处理哪些钱包错误码？
     → 6022001（余额不足）→ 你们需要在B端展示"用户余额不足"
     → 6022002（已处理）→ 你们无需处理（代表幂等成功）
```

### 11.2 异常处理约定

```
  场景1：你们调我的RPC超时了
  → 建议：你们用同一个 order_no 重试，我这边幂等处理
  → 不要用新的 order_no 重试！否则会导致重复操作

  场景2：你们调我的RPC返回错误
  → 区分"可重试"和"不可重试"：
     - 可重试：网络超时、服务暂时不可用
     - 不可重试：余额不足、订单已处理、币种禁用
  → 需要确认：你们需要我在返回结构中标记 retryable 吗？

  场景3：我调你们的RPC超时了
  → 充值方式/提现方式查询是只读操作，超时后我降级返回空列表
  → 不影响资金安全，只影响用户体验（看不到充值/提现选项）

  场景4：你们回调我上账成功了，但你们自己更新状态失败了
  → 你们需要保证自己的状态更新和调我的RPC是原子的？
  → 还是你们先调我再更新自己？（如果你们更新失败但我已入账怎么办？）
```

## 十三、议题十二：对账与差错处理

### 13.1 对账需求

```
钱包和财务之间需要对账：

  对账维度1（充值）：
    钱包的充值订单数 / 充值总金额 vs 你们的支付成功数 / 支付总金额
    → 应该完全一致

  对账维度2（提现）：
    钱包的提现完成数 / 提现总金额 vs 你们的出款成功数 / 出款总金额
    → 应该完全一致

  对账维度3（人工加减款）：
    钱包的人工操作流水 vs 你们的审核通过记录
    → 应该完全一致

  需要讨论：
  ① 对账是谁来做？你们做还是我做还是各自做？
  ② 对账频率？每日？实时？
  ③ 对不上（差异）怎么处理？报警+人工核查？
  ④ 需要提供什么对账接口？（批量查询某时间段内的订单？）
```

### 13.2 差错处理

```
  场景A：充值回调到你们了但调我入账失败
    → 你们重试 → 还是失败 → 怎么处理？
    → 是否需要人工补单流程？（R-12 CreditCorrection就是为这个场景准备的）

  场景B：提现出款成功了但调我确认扣款失败
    → 你们重试 → 还是失败 → 怎么处理？
    → 钱已经出去了但我这边还是冻结状态，最终需要人工确认

  场景C：提现审核拒绝了但调我解冻失败
    → 你们重试 → 还是失败 → 怎么处理？
    → 用户的钱被冻住了但实际已经审核拒绝

  对于场景B和C，需要有兜底机制：
  ① 你们是否有定时重试任务（扫描"已通知但未确认"的订单）？
  ② 我们是否需要一个对账任务来发现不一致？
  ③ 人工介入的入口在哪？（你们B端？运维后台？）
