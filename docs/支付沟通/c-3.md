## 一、先对齐的大前提

### 1.1 职责划分——谁管什么

在开始细节讨论之前，**必须先把大边界画清楚**。我理解的分工是：

```
┌─────────────────────────────────────────────────────────────────────┐
│  钱包模块（我们）                                                     │
│                                                                      │
│  管的事：                                                             │
│  ① 用户余额（各子钱包的加减冻结解冻）                                  │
│  ② 币种配置（币种CRUD、汇率管理、精度规则）                            │
│  ③ 兑换（法币→BSB，纯内部操作）                                       │
│  ④ 充值/提现的C端入口页面数据组装                                      │
│  ⑤ 资金流水记录（不可变审计记录）                                      │
│  ⑥ 提现账户管理（银行卡/电子钱包/USDT地址的保存复用）                   │
│                                                                      │
│  不管的事：                                                            │
│  ✗ 三方通道对接（代收/代付SDK）                                        │
│  ✗ 通道配置和轮询                                                     │
│  ✗ 充值/提现审核流程                                                   │
│  ✗ 风控判定                                                           │
│  ✗ 对账报表                                                           │
└─────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────┐
│  支付/财务模块（你们）                                                 │
│                                                                      │
│  管的事：                                                             │
│  ① 通道配置（代收通道、代付通道、密钥、费率、限额、权重）               │
│  ② 支付方式配置（充值方式、提现方式，关联通道）                         │
│  ③ 三方通道API对接（创建订单、回调处理、签名验证）                      │
│  ④ 通道轮询/选择（权重分配、降级、故障转移）                           │
│  ⑤ 审核流程（风控审核→财务审核）                                       │
│  ⑥ 充值/提现记录管理（B端列表、状态流转、导出）                        │
│  ⑦ 人工修正审核                                                       │
│  ⑧ 数据统计（通道统计、充值统计、提现统计）                            │
│                                                                      │
│  不管的事：                                                            │
│  ✗ 用户余额变动（这是钱包的事）                                        │
│  ✗ 币种/汇率配置（这是钱包的事）                                       │
│  ✗ C端钱包页面逻辑（余额概览、记录查询）                               │
└─────────────────────────────────────────────────────────────────────┘
```

**需要当场确认**：
> 以上分工是否准确？有没有哪块职责归属有分歧？

### 1.2 一句话关系

- 钱包是**执行者**——你让我加钱我加钱、让我冻钱我冻钱、让我扣钱我扣钱
- 支付是**决策者**——你决定什么时候加、加多少、什么时候冻、什么时候扣
- 我们之间的沟通桥梁是**RPC接口 + 订单号**

### 1.3 双向依赖关系

```
钱包 → 支付（我们调你们 4个RPC）：
  获取充值方式、获取提现方式、获取提现配置、提交提现订单

支付 → 钱包（你们调我们 9个RPC）：
  入账、奖金入账、冻结、解冻、扣款、人工加款、人工减款、查余额、查币种配置

双向依赖 = 互相阻塞，所以IDL定义要尽早锁定
```

## 二、充值链路 — 7个必须对齐的问题

### 问题C1：三方通道由谁对接？

**背景**：用户充值时需要向三方支付通道创建订单、获取支付页面URL。

**我的理解**：支付模块负责对接三方SDK，钱包不直接调三方。

**需要确认**：
```
方案A（我认为的）：
  用户 → 钱包创建本地订单 → 钱包RPC调支付"创建通道订单" → 支付调三方 → 返回paymentUrl
  钱包拿到paymentUrl返回给前端

方案B（可能的另一种）：
  用户 → 钱包创建本地订单 → 钱包直接调三方SDK → 返回paymentUrl
  钱包自己处理三方对接

你们是哪种？如果是方案A，我需要你们提供一个RPC：CreateChannelOrder
如果是方案B，我需要你们给我通道配置信息（密钥、请求地址等）
```

### 问题C2：充值回调（Webhook）由谁接收？

**背景**：用户支付完成后，三方通道会回调通知支付结果。

**需要确认**：
```
方案A（我假设的）：
  三方通道 → 回调支付模块 → 支付验签+验证金额 → 支付调钱包 WalletCredit 入账
  钱包不接触回调

方案B：
  三方通道 → 回调钱包 → 钱包验签 → 钱包自己入账 → 通知支付更新记录
  钱包直接处理回调

我倾向方案A——通道是你们管的，密钥是你们的，回调验签也应该你们做。
但这意味着你们回调处理完后，必须调我的 WalletCredit RPC 完成入账。
```

**关键问题**：如果是方案A，你们调 WalletCredit 时传什么参数？
```
WalletCreditReq {
    userId: i64         // 哪个用户
    currencyCode: string // 什么币种
    amount: string       // 入账金额（这个金额是用户的原始充值金额？还是扣完手续费的？）
    orderNo: string      // 订单号（用你们的还是我们的？）
    remark: string       // 备注
}
```

### 问题C3：充值手续费由谁扣？

**背景**：三方通道有代收费率（如1%），这笔费用的处理方式影响入账金额。

**需要确认**：
```
场景：用户充值 ₫1,000,000，通道代收费率 1%

方案A：用户全额到账，平台承担手续费
  → WalletCredit(amount = "1000000")  用户收到100万
  → 通道实际给平台 99万，平台亏1万

方案B：手续费从用户充值金额中扣除
  → WalletCredit(amount = "990000")  用户收到99万
  → 通道费在支付侧记录

方案C：手续费额外收取
  → 用户实际支付 101万，到账100万

你们的产品需求是哪种？这决定了你们调 WalletCredit 时传的 amount 值
```

### 问题C4：充值订单在谁的库？

**背景**：充值过程中有"创建订单→支付→回调→入账"多个阶段，订单状态需要有地方维护。

**需要确认**：
```
方案A：两边都存
  钱包库：t_deposit_order（C端查记录用）
  支付库：充值记录表（B端管理、通道对账用）
  两张表通过 orderNo 关联

方案B：只存一边
  只在支付库存，钱包通过RPC查询

我倾向方案A——因为C端用户查充值记录是高频操作，走RPC查你们的库太慢。
但如果两边都存，需要约定：谁先创建订单？订单号谁生成？状态怎么同步？
```

### 问题C5：充值订单号谁生成？格式是什么？

**需要确认**：
```
我们的设计：钱包生成订单号，格式 C+16位（如 C2026030200000001）
你们有没有自己的订单号格式要求？
通道侧还有一个 channel_order_no（三方返回的）

所以一笔充值最多有3个订单号：
  ① 钱包订单号（C+16位）—— 我们生成
  ② 支付平台订单号 —— 你们生成？还是复用我们的？
  ③ 通道订单号 —— 三方返回

需要约定：你们调 WalletCredit 时，orderNo 传哪一个？
```

### 问题C6：重复转账拦截逻辑谁做？

**背景**：产品要求"3分钟内同金额充值"做拦截（软拦截：弹窗警告；硬拦截：≥3笔禁止）。

**需要确认**：
```
这个拦截是在C端提交时钱包自己检查（查本地 t_deposit_order）？
还是由支付模块在创建通道订单时检查？
还是两边都检查？

我的想法：钱包在创建订单前先检查本地表——同用户、同金额、3分钟内、状态为待支付/处理中的订单数量。
如果 ≥1 且 <3 → 软拦截（返回警告，前端弹窗确认后可继续）
如果 ≥3 → 硬拦截（直接拒绝）

你们需要做额外的拦截逻辑吗？
```

### 问题C7：充值奖金活动数据从哪来？

**背景**：用户充值时可以选择"额外奖金活动"（如充值送50%），这个活动列表由谁管？

**需要确认**：
```
方案A：活动由支付/运营模块管理
  → 钱包通过RPC获取活动列表：GetBonusActivityList(currencyCode) → list<BonusActivity>
  → 需要你们提供这个接口

方案B：活动由钱包模块自己管理
  → 钱包有 t_bonus_activity 表
  → 充值时钱包自己查询可用活动

方案C：有独立的活动/运营模块管理
  → 钱包和支付都从活动模块获取

这块是谁的职责？奖金计算规则谁执行？
```

## 三、提现链路 — 9个必须对齐的问题

### 问题W1：提现方式和限额配置是你们管的对吧？

**确认**：
```
钱包C端"提现页面"需要展示：
  - 可用提现方式（银行卡/电子钱包/USDT）
  - 单笔最低/最高限额
  - 单日限额
  - 手续费率
  - T+N到账时间

这些数据我理解全部由支付模块B端配置（财务需求文档2-2支付配置-2提现方式），
我通过RPC来取：

  GetWithdrawMethodList(currencyCode) → 可用提现方式列表
  GetWithdrawConfig(currencyCode, methodType) → 限额、费率、到账时间

请确认这两个RPC的入参出参设计：

  GetWithdrawMethodListReq {
      currencyCode: string     // 币种
  }
  GetWithdrawMethodListResp {
      methods: list<WithdrawMethod>
          methodType: i32      // 1银行卡 2电子钱包 3USDT
          methodName: string   // 展示名称
          iconUrl: string      // 图标
          subTypes: list<string>  // 如 ["GoPay","OVO","DANA"]（电子钱包子类型）
          arrivalTime: string  // "T+1"、"即时"
  }

  GetWithdrawConfigReq {
      currencyCode: string
      withdrawMethodType: i32  // 1银行卡 2电子钱包 3USDT
  }
  GetWithdrawConfigResp {
      singleMinAmount: string  // 单笔最低
      singleMaxAmount: string  // 单笔最高
      dailyMaxAmount: string   // 单日限额
      feeRate: string          // 手续费率（"1.00"表示1%）
      fixedFee: string         // 固定手续费（如果有）
      arrivalDays: i32         // T+N中的N
  }

你们觉得这个设计合理吗？有没有遗漏字段？
```

### 问题W2：手续费谁算？

**背景**：用户输入提现金额后，页面需要展示"手续费X元，实际到账Y元"。

**需要确认**：
```
我的方案：钱包自己算
  ① 钱包从你们获取费率（feeRate）
  ② 钱包本地计算：feeAmount = amount × feeRate / 100
  ③ 钱包本地计算：actualAmount = amount - feeAmount
  ④ 提交提现时，钱包把三个值都传给你们

你们需要重新验算吗？
还是说手续费由你们计算后返回给我？
```

### 问题W3：提现订单创建后怎么传给你们？

**背景**：用户确认提现后，钱包先冻结余额，然后需要把提现订单交给支付模块进入审核。

**需要确认**：
```
方案A（同步RPC，我目前假设的）：
  钱包冻结余额 → rpc.FinanceClient().SubmitWithdrawOrder(req) → 同步返回成功/失败
  如果RPC失败 → 钱包回滚解冻

方案B（异步MQ）：
  钱包冻结余额 → 发MQ消息 → 支付消费消息后处理
  不需要等待同步返回

方案A更安全（失败可以立即回滚），方案B吞吐更高。
你们倾向哪种？

如果是方案A，SubmitWithdrawOrder 的入参出参：

  SubmitWithdrawOrderReq {
      orderNo: string           // 提现单号 T+16位（我们生成）
      userId: i64
      currencyCode: string
      amount: string            // 提现金额
      feeAmount: string         // 手续费
      actualAmount: string      // 到账金额
      withdrawMethod: i32       // 1银行卡 2电子钱包 3USDT
      accountInfo: object       // 提现账户信息（银行名/卡号/户名 等）
      exchangeRate: string      // 锁汇汇率（跨币种场景）
  }
  SubmitWithdrawOrderResp {
      success: bool
      financeOrderNo: string    // 你们侧的订单号（如果有）
  }

需要确认：这个结构够不够？你们还需要什么字段？
```

### 问题W4：提现审核结果怎么通知钱包？

**背景**：提现进入你们的审核流程后（风控审核→财务审核→出款），每个结果都需要通知钱包执行对应操作。

**需要确认**：
```
审核驳回/出款失败：你们调我的 WalletUnfreeze(orderNo) → 我解冻归还
出款成功：你们调我的 WalletDeduct(orderNo) → 我最终扣除冻结资金

这里的 orderNo 用哪个？
  → 我建议用钱包生成的提现订单号（T+16位），因为我的冻结记录关联的就是这个号

你们同意吗？

另外，你们通知我的时机是什么？
  风控驳回后立即调？还是有延迟？
  出款成功的回调你们收到后多久调我？
```

### 问题W5：单日限额的"已用额度"谁统计？

**背景**：提现有单日限额（如 dailyMaxAmount = 50000000VND），需要校验"今天已经提了多少"。

**需要确认**：
```
方案A：钱包统计
  钱包查本地 t_withdraw_order 表，统计今天该用户该币种已创建的提现订单总额
  优点：不依赖RPC
  缺点：可能和你们的统计有微小差异（如果有取消订单等）

方案B：你们统计
  你们在 GetWithdrawConfig 返回值中增加 dailyUsed 字段
  需要我传 userId

方案C：两边都统计，以某一方为准

你们怎么想？
```

### 问题W6：提现账户信息传给你们什么？

**背景**：用户提现时的银行卡/电子钱包/USDT地址信息，钱包保存在 t_withdraw_account 表。

**需要确认**：
```
SubmitWithdrawOrder 中的 accountInfo，你们需要哪些字段？

银行卡提现：
  bankName: string        // 银行名称
  bankAccount: string     // 银行账号
  bankBranch: string      // 开户行
  accountHolder: string   // 账户持有人（= KYC认证姓名）

电子钱包提现：
  eWalletType: string     // GoPay/OVO/DANA/...
  eWalletPhone: string    // 绑定手机号
  accountHolder: string   // 持有人

USDT提现：
  chainAddress: string    // 链上地址
  chainNetwork: string    // TRC20/ERC20/BEP20

够不够？你们代付通道需要额外字段吗？
```

### 问题W7：部分提现场景——中心+奖励混合提现

**背景**：用户提现10万，但中心钱包只有7万，奖励钱包有3万（且已完成流水要求）。

**需要确认**：
```
是否支持混合提现？如果支持：
  ① 钱包冻结中心7万 + 奖励3万 = 两条冻结记录
  ② 提交给你们的是一笔提现单10万
  ③ 出款成功时，你们调两次 WalletDeduct？还是一次传两个子钱包金额？

这个场景需要双方都理解清楚。
```

### 问题W8：提现时KYC校验是谁的活？

**背景**：未认证用户只能USDT提现，已认证用户全渠道可用。

**需要确认**：
```
我的方案：钱包负责
  ① 钱包调 ser-kyc.KycQuery 获取KYC状态
  ② 根据状态过滤可用提现方式（status≠3 → 只保留USDT选项）
  ③ 然后再调你们的 GetWithdrawMethodList 获取具体方式

还是说你们在 GetWithdrawMethodList 内部就做了KYC过滤？
需要我把KYC状态传给你们吗？

我倾向我自己做过滤，因为KYC逻辑简单，而且银行卡提现时
"账户持有人"要用KYC认证姓名——这个数据本来就在我手上。
```

### 问题W9：提现币种规则——你们知道这条规则吗？

**背景**：产品有一条硬规则——充值方式决定提现方式。

```
| 用户充值方式 | 允许的提现方式 |
|-------------|-------------|
| 法币充值     | 法币提现     |
| USDT充值    | USDT提现    |
| 法币+USDT混合| 法币提现    |
| 法币→兑换BSB | 法币提现     |
```

**需要确认**：
```
这条规则在哪里校验？
  方案A：钱包校验（钱包知道用户历史充值方式）
  方案B：你们校验（你们有充值记录）
  方案C：两边都校验

需要约定清楚，否则可能出现"用户USDT充值后走银行卡提现"的漏洞。
```

## 四、人工修正 — 4个必须对齐的问题

### 问题M1：人工修正审核通过后，你们怎么调我？

**需要确认具体调用逻辑**：
```

  你们调 WalletCredit(orderNo=B补单号, userId, currencyCode, amount=补单金额)
```

```
  ├── 我们 Mock 你们的 4 个 RPC（硬编码返回充值方式/提现配置等）
  └── 你们 Mock 我们的 9 个 RPC（WalletCredit返回success等）

阶段3：单接口联调
  ├── 先调通 GetCurrencyConfig（最简单，纯查询）
  ├── 再调通 GetUserBalance（纯查询）
  └── 再调通 WalletCredit（有数据写入，验证幂等）

阶段4：链路联调
  ├── 充值全链路：用户充值 → 三方回调 → 入账
  ├── 提现全链路：用户提现 → 冻结 → 审核 → 出款 → 扣除/解冻
  └── 人工修正：创建 → 审核 → 加款/减款

阶段5：异常场景验证
  ├── 充值回调重复 → 幂等
  ├── 提现冻结后提交失败 → 回滚解冻
  ├── 出款失败 → 解冻归还
  └── 人工减款余额不足 → 部分执行
```

## 十四、会议沟通话术备忘

### 14.1 开场白

> "我们钱包模块和支付模块是双向依赖关系——你们调我们9个RPC做资金操作，我们调你们4个RPC拿配置和提交订单。今天的目的是把所有交叉点摊开讲清楚，让两边没有盲区。
>
> 我这边梳理了几大块问题：**充值链路7个、提现链路9个、人工修正4个、USDT专项5个**，加上**RPC接口契约、数据归属、异常场景、对账机制**。我们一个个过。"

### 14.2 关键诉求（必须达成的共识）

```
① 充值回调由谁接收 → 定死
② 三方通道由谁对接 → 定死
③ 充值/提现订单两边都存还是只存一边 → 定死
④ 你们的 IDL 什么时候能输出 → 给时间承诺
⑤ 联调环境什么时候能通 → 给时间承诺
```

### 14.3 底线（不能让步的）

```
① 所有资金操作通过我们的 RPC 执行 → 你们不能直接改我们的余额表
② 幂等通过 orderNo 保证 → 你们每次调用必须传同一个 orderNo，不能重试换号
③ 金额类型用 string → 不接受 int64/float64，这是精度安全的底线
④ 冻结三件套互斥 → 同一个 orderNo 只能 Unfreeze 或 Deduct，二选一
```

### 14.4 可以让步/商量的

```
① 充值订单存在哪个库 → 可以只在你们那边存，我通过RPC查（虽然性能差些）
② 手续费谁算 → 可以你们算好了传给我，也可以我自己算
③ 对账粒度 → 先粗后细，不用一开始就做到订单级实时对账
④ Mock数据结构 → 你们先给大致字段，IDL后续可以微调
```
