## 一、沟通目标与核心诉求

### 本次沟通要解决的 3 个核心问题

1. **接口契约**：双方各暴露什么 RPC 接口、入参出参是什么、幂等怎么做、超时重试怎么处理
2. **链路归属**：充值/提现/人工修正这三条链路中，每一步谁负责、数据流向如何、回调谁接收
3. **开发解耦**：财务模块未就绪前，钱包如何 Mock 你们的接口独立开发；你们如何 Mock 我们的接口独立开发

### 我方（钱包）的核心诉求

- 明确你们提供给我的 **5~6 个 RPC 接口**的入参、出参、错误码
- 明确充值回调的接收方和通知链路
- 明确提现审核流程中，每个状态变更时你们调我哪个接口
- 确认订单号规则和幂等键约定
- 拿到你们的 Mock 接口或 IDL 草案，我方可以先行开发

### 预期对方（支付）的诉求

- 明确我方暴露的 **8~11 个 RPC 接口**的入参、出参、错误码
- 明确 WalletCredit / WalletFreeze / WalletDeduct / WalletUnfreeze 各自的精确语义
- 确认余额操作的幂等机制和重复调用的返回行为
- 拿到我方的 IDL 草案，支付方可先行开发

## 二、双方职责边界总览

### 2.1 一张表讲清谁管什么

| 职责域 | 钱包 (ser-wallet) | 支付/财务 (ser-finance) | 交叉点 |
|--------|-------------------|------------------------|--------|
| **用户余额管理** | ✅ 独占（4种子钱包） | ❌ 无权直接改余额 | 支付通过 RPC 调钱包的加减款接口 |
| **币种配置+汇率** | ✅ 独占（B端管理+定时更新） | 读取：通过 RPC 获取汇率 | 支付方下单时需要用我方汇率 |
| **充提订单（平台侧）** | ✅ 创建和管理 | 读取/关联 | 订单号由钱包生成，双方共用 |
| **通道配置+通道订单** | ❌ | ✅ 独占 | 钱包请求匹配通道，支付返回支付信息 |
| **三方回调接收+验签** | ❌ | ✅ 独占（持有通道密钥） | 支付验签后 RPC 通知钱包入账/出款结果 |
| **充值通道轮询匹配** | ❌ | ✅（成功率+时间+预警系数加权） | 钱包传币种+方式+金额，支付返回通道 |
| **提现三级审核** | ❌ | ✅（风控→财务→出款） | 审核结果通过 RPC 通知钱包冻结/解冻/扣款 |
| **人工修正（补单/加减款）** | 被动执行加减款 | ✅ 创建+审核流程 | 审核通过后支付 RPC 调钱包执行 |
| **数据统计/报表** | ❌（仅提供余额查询） | ✅ 独占 | 支付方可能需要批量查余额 |
| **对账** | ✅ 钱包侧流水 | ✅ 通道侧流水 | 双方日对账比对，差异→补单 |

### 2.2 一句话总结边界

> **钱包管余额和流水（只进不改），支付管通道和审核。钱包不碰通道，支付不碰余额。双方通过 RPC 接口交互，以订单号为纽带。**

## 三、我需要你们提供的接口（钱包→支付）

> 以下是钱包模块需要调用支付模块的 RPC 接口，**请支付组确认接口名称、入参、出参、错误码**

### 接口 F-1：GetDepositMethods — 获取充值方式列表

| 项目 | 内容 |
|------|------|
| **调用时机** | C端用户打开充值页面 |
| **入参** | currencyCode (string) — 当前币种代码 |
| **期望返回** | 该币种下所有已启用的充值方式列表 |
| **返回字段（期望）** | methodId, methodName, methodType(银行转账/电子钱包/USDT链上/扫码), icon, minAmount, maxAmount, feeRate, fixedAmounts(档位列表), sortOrder |
| **调用频率** | 高（每次进入充值页） |
| **缓存策略** | 钱包侧缓存 5 分钟，减少 RPC 压力 |
| **关联奖金活动** | 返回值中是否包含关联的充值奖金活动？还是钱包需要另调活动模块？ |

**需要确认**：
- [ ] 返回的金额字段是什么单位？最小货币单位整数 还是 带小数的字符串？
- [ ] 充值方式的 methodType 枚举值列表？
- [ ] 是否支持按金额范围过滤（用户在VND下看到的方式和在USDT下看到的不同）？
- [ ] 充值档位（如 100/500/1000/5000）在你们配置还是我们配置？

### 接口 F-2：MatchDepositChannel — 充值通道匹配

| 项目 | 内容 |
|------|------|
| **调用时机** | 用户确认充值金额后，创建充值订单时 |
| **入参** | currencyCode, methodId, amount, userId, platformOrderNo(我方生成的C+16位订单号) |
| **期望返回** | 匹配到的通道信息 + 三方支付信息 |
| **返回字段（期望）** | channelId, channelCode, thirdPartyOrderNo, payUrl(跳转URL) / qrCodeData(二维码) / walletAddress(USDT地址), expireTime |
| **调用频率** | 中（每次用户下单） |
| **超时时间** | 建议 5s，超时钱包返回"通道繁忙" |

**需要确认**：
- [ ] 通道匹配和创建三方订单是**一个接口**还是**两个接口**？（先 MatchChannel 再 CreateChannelOrder，还是合并？）
- [ ] 如果所有通道都不可用（限额/维护/预警），返回什么错误码？
- [ ] 你们的轮询算法（成功率系数+时间系数+预警系数）需要我传什么参数？还是你们内部自己计算？
- [ ] 三方返回的支付URL/二维码，我直接透传给C端，格式是什么？

### 接口 F-3：GetWithdrawMethods — 获取提现方式列表

| 项目 | 内容 |
|------|------|
| **调用时机** | C端用户打开提现页面 |
| **入参** | currencyCode |
| **期望返回** | 该币种下所有已启用的提现方式列表 |
| **返回字段（期望）** | methodId, methodName, methodType, icon, feeRate, singleMin, singleMax, dailyMax, arrivalDays(T+N), sortOrder |
| **补充说明** | 钱包侧会根据用户 KYC 状态和资金来源规则二次过滤（未认证仅USDT、法币充值只能法币提现等） |

**需要确认**：
- [ ] 限额数据（单笔最小/最大、单日限额）在你们配置还是我们配置？
- [ ] 手续费率是你们返回还是我们配置？（涉及前端展示和实际扣费）
- [ ] 你们是否能在返回时就考虑用户的资金来源规则进行预过滤？还是我方自己过滤？

### 接口 F-4：SubmitWithdrawRequest — 提交提现出款请求

| 项目 | 内容 |
|------|------|
| **调用时机** | 用户提交提现申请，钱包冻结余额后 |
| **入参** | platformOrderNo(T+16位), userId, currencyCode, amount, methodId, receiverInfo(收款信息), fundSourceType(资金来源类型) |
| **期望返回** | financeAuditId(你们的审核单号), status(已进入审核队列) |

**需要确认**：
- [ ] receiverInfo 的格式怎么传？JSON 字符串？还是你们定义结构体？
- [ ] 你们是否需要钱包传汇率信息？（提现时使用的出款汇率）
- [ ] 提交后，你们的三级审核（风控→财务→出款）每一步状态变更会通知我吗？还是只通知最终结果？
- [ ] 你们的审核单号和我们的订单号是什么映射关系？

### 接口 F-5：QueryChannelOrderStatus — 查询通道订单状态（可选）

| 项目 | 内容 |
|------|------|
| **调用时机** | 充值回调丢失时的兜底查询、C端轮询充值状态 |
| **入参** | platformOrderNo 或 thirdPartyOrderNo |
| **期望返回** | status(待支付/支付中/成功/失败/超时), actualAmount(实际到账金额) |
| **是否必需** | 如果你们保证回调一定送达（重试机制），此接口可降级为 P2 |

**需要确认**：
- [ ] 你们的回调机制是否有重试？重试几次？间隔多久？
- [ ] 如果回调全部失败，是否有补偿机制（如定时扫描未完成订单）？
- [ ] 还是说钱包需要自己轮询此接口做兜底？

### 接口 F-6：GetDepositMethodDetail — 获取单个充值方式详情（待确认是否需要）

| 项目 | 内容 |
|------|------|
| **场景** | 用户选定充值方式后，加载详细配置（档位、限额、手续费明细） |

## 四、你们需要调我的接口（支付→钱包）

> 以下是钱包模块暴露给支付/财务模块调用的 RPC 接口，**我方负责实现，请支付组确认是否满足你们的需求**

### 接口 W-1：WalletCredit — 充值入账

```
场景：充值成功回调后、补单审核通过后
入参：
  - userId     int64   用户ID
  - currencyId int64   币种ID
  - walletType int8    子钱包类型（1=中心，固定）
  - amount     int64   入账金额（最小货币单位整数）
  - orderNo    string  平台订单号（幂等键）—— 充值用 C+16位，补单用 B+16位
  - bizType    int8    业务类型：1=充值 2=补单
返回：
  - success    bool
  - message    string  "ok" / "already_processed" / "error_xxx"
幂等机制：
  - wallet_flow 表唯一索引 (orderNo, walletType, flowType)
  - 重复调用返回 "already_processed"（不是错误），调用方收到后不再重试
```

### 接口 W-2：WalletCreditReward — 奖金入账

```
场景：充值选了奖金活动，充值成功后额外发奖金
入参：
  - userId          int64
  - currencyId      int64
  - amount          int64   奖金金额
  - auditMultiplier int32   流水倍数要求（如 3 = 需完成 3 倍流水才可提现）
  - orderNo         string  幂等键
  - bizType         int8    1=充值奖金 2=活动奖金
返回：同 W-1
说明：奖金入奖励钱包（walletType=2），不入中心钱包。有流水稽核约束。
```

**需要确认**：
- [ ] 奖金金额由你们计算好传给我，还是你们传奖金比例由我计算？
- [ ] 流水倍数（auditMultiplier）是你们传还是活动模块传？
- [ ] 一笔充值只能关联一个奖金活动？还是可以叠加多个？

### 接口 W-3：WalletFreeze — 冻结余额

```
场景：提现发起时，锁定提现金额不可消费
入参：
  - userId      int64
  - currencyId  int64
  - walletType  int8    从哪个子钱包冻结（通常 1=中心）
  - amount      int64   冻结金额
  - orderNo     string  提现订单号（T+16位，幂等键）
返回：
  - success     bool
  - message     string
行为：available -= amount, frozen += amount（总余额不变，只是组成变化）
前置条件：available >= amount，否则返回余额不足错误
```

**需要确认**：
- [ ] 冻结由我方在用户提交提现时内部调用，还是由你们调用？（我方当前假设：我方内部调用）
- [ ] 你们是否有需要主动冻结用户余额的场景？（如风控紧急冻结）

### 接口 W-4：WalletUnfreeze — 解冻余额

```
场景：提现被驳回（风控/财务任一阶段）、出款失败
入参：
  - userId      int64
  - currencyId  int64
  - walletType  int8
  - amount      int64   解冻金额
  - orderNo     string  同冻结时的提现订单号（幂等键）
返回：同上
行为：frozen -= amount, available += amount（资金归还用户可用余额）
```

**需要确认**：
- [ ] 你们在以下哪些节点会调用 Unfreeze？请逐一确认：
  - 风控审核驳回 → 调 Unfreeze ✅?
  - 财务审核驳回 → 调 Unfreeze ✅?
  - 代付通道出款失败 → 调 Unfreeze ✅?
  - 代付通道超时无响应 → 调 Unfreeze？还是等通道最终回调？
- [ ] 部分解冻是否存在？（冻结 1000，只解冻 500？我方当前假设不存在）

### 接口 W-5：WalletDeduct — 出款确认扣款

```
场景：提现代付成功，最终确认扣除用户资金
入参：
  - userId      int64

```

## 五、充值链路：逐步拆解谁做什么

### 完整时序（每一步标注负责方）

```
步骤    负责方      动作                               输入→输出
────────────────────────────────────────────────────────────────────────
T0      C端用户     打开充值页
T1      钱包        调 [F-1] GetDepositMethods          currencyCode → 方式列表
T2      钱包        将方式列表返回C端展示
T3      C端用户     选择充值方式+输入金额+（可选）选奖金活动
T4      钱包        重复转账检测（内部）                  检查是否有同币种同金额待处理订单
T5      钱包        校验金额范围（min ≤ amount ≤ max）
T6      钱包        生成平台订单号：C + 16位随机数
T7      钱包        调 [F-2] MatchDepositChannel         currencyCode, methodId, amount, orderNo
                                                         → channelId, payUrl/qrCode, expireTime
T8      钱包        写入 deposit_order 表（status=待支付）
T9      钱包        返回支付信息给C端（payUrl/二维码/地址）
T10     C端用户     跳转三方页面完成支付
────────── 分割线：以下为异步回调 ──────────
T11     三方通道     支付结果回调 → 发送到【支付模块】
T12     支付        验签（用通道密钥验证回调签名）
T13     支付        更新通道侧订单状态
T14     支付        调 [W-1] WalletCredit(orderNo, amount)  → 钱包入账
T15     钱包        ledger.Credit → 余额增加 + 写流水
T16     钱包        更新 deposit_order 状态 → 成功
T17     钱包        返回成功给支付
T18     支付        （如选了奖金活动）调 [W-2] WalletCreditReward → 奖金入奖励钱包
T19     钱包        ledger.Credit(奖励钱包) → 奖励余额增加 + 写流水
```


| 处理方 | 动作 |
|--------|------|

**需要确认**：□ 支付有轮询兜底  □ 钱包需要做兜底  □ 仅靠补单

### 场景 E-2：WalletCredit 调用超时（支付→钱包）

```
支付收到充值回调，调钱包 WalletCredit 超时，不知道钱包是否已入账
```

| 处理 | 说明 |
|------|------|
| 支付侧 | 用同一个 orderNo 重试（最多3次，指数退避） |
| 钱包侧 | 幂等保证——重复调用返回"已处理"，不会重复入账 |
| 最终兜底 | 日对账发现差异 → 补单 |

### 场景 E-3：提现提交失败（冻结后调支付失败）

```
钱包已冻结用户余额，但调支付 SubmitWithdrawRequest 失败
```

| 处理 | 说明 |
|------|------|
| 钱包侧 | **立即解冻**，withdraw_order 状态标记为"提交失败" |
| 用户侧 | 返回错误"提现提交失败，请稍后重试" |
| 不能 | 留着冻结不管——用户资金会被锁死 |

### 场景 E-4：提现出款后，WalletDeduct 调用失败

```
支付确认代付成功，但调钱包 WalletDeduct 超时
```

| 处理 | 说明 |
|------|------|
| 支付侧 | 重试（同 orderNo，钱包幂等） |
| 如果 3 次都失败 | 支付标记为"待补扣"，日对账补处理 |
| 风险 | 用户冻结金额不会自动解冻（因为钱确实已出），日对账会发现 |

### 场景 E-5：审核驳回后，WalletUnfreeze 调用失败

```
支付审核驳回提现，但调钱包 WalletUnfreeze 失败
```

| 处理 | 说明 |
|------|------|
| 支付侧 | 重试（钱包幂等） |
| 如果持续失败 | 告警 → 人工介入解冻 |
| 风险 | 用户资金被锁，必须及时处理 |

### 场景 E-6：用户提现期间同时充值

```
用户有 1000 可用 + 500 冻结，同时来了一笔充值入账
```

| 处理 | 说明 |
|------|------|
| 钱包侧 | 正常处理——充值 Credit 和提现 Freeze 在行锁层面串行化 |
| 结果 | 两笔操作都会成功，不会互相影响 |

### 场景 E-7：并发多笔提现

```
用户余额 1000，同时提交两笔提现各 800
```

| 处理 | 说明 |
|------|------|
| 钱包侧 | 行锁串行化：第一笔冻结成功（available=200, frozen=800），第二笔冻结失败（余额不足） |
| 无风险 | 不会出现超冻 |
