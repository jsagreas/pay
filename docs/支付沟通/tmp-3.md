              │    ManualDebit         人工减款               │

              │    (GetBalance)        查余额（可选）          │
              └─────────────────────────────────────────────┘
```

### 1.2 核心认知

```
我们是"钱的保管员"——只管加减冻结余额，不碰支付通道
你们是"钱的搬运工"——负责对接三方通道、处理回调、管理审核流程

充值链路 = 用户入口(我) → 方式列表(你提供) → 创建订单(联合) → 用户支付(你对接) → 回调上账(你调我)
提现链路 = 用户入口(我) → 方式列表(你提供) → 冻结+创建订单(联合) → 审核出款(你负责) → 结果回调(你调我)

双方是"联合作战"关系，不是"我做完扔给你"的串行关系。
```

### 1.3 关键数据

```
钱包需要调支付的接口：3-4个（全部是P0，缺任何一个充提流程都跑不通）
支付需要调钱包的接口：6-7个（我已有IDL定义，可以提前给你们）
双方联合的业务链路：充值 + 提现（平台最核心的资金流入/流出通道）
```

```

 2     [钱包→支付]  调 GetPaymentConfig 获取充值方式列表          ← 本次重点议题
 3     [钱包]      展示充值方式+档位+汇率(BSB时)
 4     [钱包]      用户选择方式、输入金额
 5     [钱包]      执行重复转账检测（纯钱包内部逻辑）
 6     [钱包]      BSB场景：锁定入款汇率快照
 7     [钱包→支付]  调 CreatePayOrder 创建通道订单                ← 本次重点议题
 8     [支付]      匹配支付通道 → 创建代收订单
 9     [支付→钱包]  返回支付信息（跳转URL / 收款地址）
10     [钱包]      返回给前端，用户跳转支付
11     [用户→三方]  用户在三方页面完成支付
12     [三方→支付]  三方通道回调支付结果
13     [支付]      验签 → 更新订单状态
14     [支付→钱包]  调 CreditWallet 给用户上账                   ← 本次重点议题
15     [钱包]      幂等校验 → 增加余额 → 写流水
```

```
步骤    负责方       动作                                      备注
────    ──────      ──────                                    ──────
 1     [钱包]      用户打开提现页面
 2     [钱包]      调 ser-kyc 校验 KYC 状态
 3     [钱包→支付]  调 GetWithdrawConfig 获取提现方式列表          ← 本次重点议题
 4     [钱包]      三重过滤：KYC状态 × 充值历史 × 支付配置
 5     [钱包]      展示提现方式+费率+限额
 6     [钱包]      用户填写提现信息、输入金额
 7     [钱包]      校验限额（单笔/单日金额/单日次数）
 8     [钱包]      冻结余额 FreezeBalance（可用→冻结）            ← 时序议题
 9     [钱包]      创建钱包侧提现订单
10     [钱包→支付]  调 CreateWithdrawOrder 提交审核                ← 本次重点议题
11     [支付]      风控审核
12     [支付]      财务审核
13     [支付]      匹配代付通道 → 执行出款
14a    [支付→钱包]  出款成功 → 调 ConfirmDebit 确认扣除冻结        ← 本次重点议题
14b    [支付→钱包]  审核驳回/出款失败 → 调 UnfreezeBalance 解冻     ← 本次重点议题
15     [钱包]      更新冻结状态（终态）→ 更新订单状态
```

# 第二部分：我需要支付组提供的

## 三、充值方式配置接口（GetPaymentConfig）

### 3.1 我的需求

```
场景：C端用户打开充值页面，需要展示"可用的充值方式列表"
     这个列表的数据100%来自支付侧——充值方式、快捷金额档位、金额范围、关联通道全是你们配置的
     钱包只负责"拿到你的数据→展示给用户"

调用时机：每次用户打开充值页面（或切换币种时）
调用频率：中频（用户级操作，非批量）
实时性要求：需要实时数据（不能缓存太久，配置变更需及时生效）
```

### 3.2 我需要你们返回什么字段（最小必须集）

```
请确认以下字段是否可以提供，以及字段命名和类型：

必须字段：
  ① 方式ID (methodId)         — 唯一标识，钱包创建订单时回传给你们
  ② 方式名称 (methodName)     — "银行转账" / "USDT" / "PIX扫码" 等
  ③ 方式图标 (methodIcon)     — URL，前端展示用
  ④ 金额范围-最低 (minAmount) — i64最小单位整数
  ⑤ 金额范围-最高 (maxAmount) — i64最小单位整数
  ⑥ 排序权重 (sort)           — 前端展示顺序
  ⑦ 方式状态 (status)         — 启用/禁用

应该有字段：
  ⑧ 快捷金额档位 (tiers)      — list<i64>，如 [10000, 50000, 100000, 500000]
  ⑨ 默认选中方式 (isDefault)   — 哪个方式默认高亮
  ⑩ 方式类型标识 (methodType)  — 1=银行转账 2=扫码 3=USDT 4=电子钱包...
  ⑪ 需要的额外信息 (extraFields) — 告诉钱包前端该收集哪些表单字段

可选字段：
  ⑫ 通道维护提示 (maintenanceMsg) — 如果某方式暂不可用，展示原因
  ⑬ 手续费信息 (feeInfo)         — 充值是否收手续费？收的话怎么展示？
```

### 3.3 我需要传什么给你们（请求参数）

```
我这边能提供：
  ① currencyCode (string)  — 当前操作币种（BRL/VND/USDT等）
  ② bizType (i32)          — 1=充值 2=提现（复用同一接口 or 拆两个接口？）
  ③ uid (i64)              — 用户ID（如果方式列表需要按用户差异化展示的话）
  ④ platform (i32)         — 平台标识（如果多平台的话）

问题：
  → 充值方式和提现方式是同一个接口（传 bizType 区分），还是拆成两个独立接口？
  → 是否需要传 uid？（即：同一币种对不同用户展示的方式可能不同？）
```

### 3.4 具体问题（需要你们回答）

```
Q3-1：接口方法名是什么？你们的服务在 ETCD 中的注册名是什么？
      → 我这边需要在 rpc_client.go 中注册你们的 Client 才能调用

Q3-2：充值方式列表是否会按币种不同返回不同的方式？
      比如 BRL 只有 PIX 和银行转账，VND 有银行转账和扫码，USDT 是链上充值？

Q3-3：如果你们的接口暂时不可用（超时/报错），充值页面应该怎么降级？
      我这边目前的设计是：直接展示"充值通道维护中"的友好提示
      你们有没有建议提供一个兜底的静态配置？

Q3-4：方式列表的返回中，USDT链上充值和法币充值是同一个结构体吗？
      USDT充值需要展示的是"收款地址+二维码+链网络"，和法币充值"跳转三方页面"完全不同
```

## 四、创建通道订单接口（CreatePayOrder）

### 4.1 我的需求

```
场景：用户确认充值金额和方式后，钱包调支付创建通道订单
     支付侧负责：匹配通道 → 创建代收订单 → 返回支付信息
     钱包拿到支付信息后返回给前端

调用时机：用户点击"确认充值"按钮
调用频率：低频（每次充值操作一次）
响应时间要求：<3秒（用户在等待）
```

### 4.2 我要传给你们什么

```
我这边能提供的信息：
  ① uid (i64)              — 用户ID
  ② currencyCode (string)  — 充值币种
  ③ amount (i64)           — 充值金额（最小单位整数）
  ④ methodId (i64)         — 充值方式ID（从 GetPaymentConfig 返回的）
  ⑤ walletOrderNo (string) — 钱包侧生成的订单号（C+16位）
  ⑥ exchangeRate (i64)     — 汇率快照（BSB充值时，钱包在创建时锁定的入款汇率）
  ⑦ bonusActivityId (i64)  — 关联的奖金活动ID（可选，用户未选奖金时为0）
  ⑧ bonusAmount (i64)      — 奖金金额（可选）

问题：
  → 你们还需要什么额外字段？比如用户IP、设备信息、回调URL？
  → 回调URL是你们固定配置的还是我每次传给你们？
```

### 4.3 我需要你们返回什么

```
必须返回：
  ① channelOrderNo (string)  — 通道订单号（你们的单号，用于后续查询和回调匹配）
  ② payType (i32)            — 支付方式类型（决定前端如何展示）
  ③ expireAt (i64)           — 订单过期时间戳

根据支付方式类型不同，需要返回不同的支付信息：
  银行转账 / 扫码支付：
    ④ payUrl (string)         — 三方承载页跳转URL
  USDT链上充值：
    ⑤ depositAddress (string) — 收款地址
    ⑥ chainNetwork (string)   — 链网络（TRC20/ERC20/BEP20）
    ⑦ qrCodeUrl (string)      — 收款二维码URL

问题：
  → 这些不同支付方式的返回结构，是一个统一的 Resp 结构（用 optional 区分），还是拆成多个方法？
  → 如果创建失败（通道不可用/参数校验失败），错误码体系是什么？
  → USDT链上充值的地址是你们管理还是需要我们提供？
```

### 4.4 具体问题

```
Q4-1：创建通道订单失败时的重试策略？
      用户点了"确认充值"但你们返回错误——钱包应该自动重试还是让用户手动重试？
      如果重试，用同一个 walletOrderNo 还是生成新的？

Q4-2：订单过期后的处理？
      订单有过期时间，过期后用户完成支付了（三方回调到了），你们还会回调我上账吗？
      还是直接丢弃？这个边界需要明确。

Q4-3：USDT充值的最小确认数是多少？
      TRC20 一般要求多少个区块确认才算到账？这个数字由你们定还是钱包配置？

Q4-4：创建订单时是否需要"预锁定通道"？
      即：用户选了方式A，在支付前方式A的通道维护了——应该怎么处理？
```

## 五、提现方式配置接口（GetWithdrawConfig）

### 5.1 我的需求

```
场景：C端用户打开提现页面，需要展示"可用的提现方式列表"
     和充值方式类似，提现方式+费率+限额全是支付侧配置的

调用时机：用户打开提现页面（已通过KYC校验后）
```

### 5.2 我需要你们返回什么字段

```
必须字段：
  ① methodId (i64)         — 提现方式ID
  ② methodName (string)    — "银行卡" / "电子钱包" / "USDT"
  ③ methodIcon (string)    — 图标URL
  ④ feeType (i32)          — 手续费类型：1=固定 2=比例 3=固定+比例
  ⑤ feeFixed (i64)         — 固定手续费金额（最小单位）
  ⑥ feeRate (string)       — 比例手续费率（如 "0.02" 表示2%）
  ⑦ minAmount (i64)        — 单笔最低提现
  ⑧ maxAmount (i64)        — 单笔最高提现
  ⑨ dailyLimit (i64)       — 每日提现限额（0=不限）
  ⑩ dailyCount (i32)       — 每日提现次数限制（0=不限）
  ⑪ status (i32)           — 启用/禁用

应该有字段：
  ⑫ methodType (i32)       — 类型标识：1=银行卡 2=电子钱包 3=USDT
  ⑬ requiredFields         — 需要用户填写的字段列表
  ⑭ arrivalTime (string)   — 预计到账时间描述（如 "1-3个工作日"）
```

### 5.3 具体问题

```
Q5-1：提现方式是否和充值方式共用 GetPaymentConfig 接口（传 bizType=2 区分），
      还是独立接口 GetWithdrawConfig？

Q5-2：提现限额（单笔/单日金额/次数）是按方式维度配置，还是按币种维度？还是按用户等级？
      → 这影响我传什么参数给你们

Q5-3：手续费计算是在你们侧还是我这边？
      → 即：用户输入提现1000，手续费2%，实际到账980
      → 这个980是你们算好返回给我，还是我自己按你们的费率算？
      → 手续费从用户余额额外扣还是从提现金额中扣？

Q5-4：KYC过滤逻辑由谁负责？
      需求规定：未KYC用户只能USDT提现。这个过滤逻辑是——
      → 方案A：你们的接口直接按用户维度返回过滤后的方式（那你们也需要查KYC状态）
      → 方案B：你们返回全量方式，我这边根据KYC状态自行过滤
      → 我倾向方案B（KYC校验已在钱包侧实现，不需要你们重复依赖 ser-kyc）
```

## 六、提现审核订单接口（CreateWithdrawOrder / SubmitWithdraw）

### 6.1 我的需求

```
场景：用户确认提现后，钱包已冻结余额，需要把提现申请提交给支付侧进入审核流程
调用时机：用户点击"确认提现" → 钱包冻结余额成功后 → 调你们
```

### 6.2 我要传给你们什么

```
  ① uid (i64)               — 用户ID
  ② currencyCode (string)   — 提现币种
  ③ amount (i64)             — 提现金额（最小单位整数）
  ④ feeAmount (i64)          — 手续费金额
  ⑤ actualAmount (i64)       — 实际到账金额 = amount - feeAmount

## 十八、冻结谁来发起？

### 18.1 两种理解

```
理解A（我的方案）：
  用户确认提现 → 钱包自己冻结(FreezeBalance) → 然后调支付提交审核
  → 好处：冻结在用户提交瞬间完成，无时间窗口
  → 坏处：如果调支付失败，需要钱包自己回滚解冻

理解B（另一种可能）：
  用户确认提现 → 钱包调支付提交审核 → 支付回调钱包冻结(FreezeBalance)
  → 好处：冻结和审核由支付统一管理
  → 坏处：从用户提交到冻结之间有时间窗口（可能余额被其他操作消耗）

我强烈建议理解A，因为：
  → 如果冻结延迟，用户可能在提现申请后投注扣光余额，冻结就会失败
  → 先冻结后提交，即使提交失败也能自动解冻恢复
  → 冻结是余额操作，应该由钱包主导

请支付组确认：你们接受"钱包先冻结，再调你们提交审核"的时序吗？
```

  2. 创建订单时钱包锁定当时的入款汇率 = 2620
  3. 传给支付的 CreatePayOrder 中需要包含这个汇率快照吗？
  4. 回调上账时，是按汇率快照计算还是按实时汇率？

问题：
  → 汇率快照是我传给你们在创建通道订单时用，还是你们不需要？
  → 上账金额是我在回调时自己按快照算，还是你们在回调时告诉我BSB数量？
```

### 20.2 BSB提现场景

```
用户提BSB时：
  1. 钱包展示出款汇率（1 BSB = 3.50 THB），用户确认
  2. 冻结的是BSB余额
  3. 实际出款的是THB（BSB按汇率换算）

问题：
  → 出款金额的计算在我这边还是你们那边？
  → 汇率快照由我传给你们？
  → 如果提交提现到出款之间经过了审核（可能几小时），汇率变了——用锁定的还是最新的？
```

## 二十一、orderNo 规则怎么定？

### 21.1 我这边的规则

```
钱包侧生成的订单号：
  充值订单：C + 16位（如 C2026030200000001）
  提现订单：T + 16位
  兑换订单：E + 16位
  人工加款：A + 16位
  人工减款：M + 16位
  补单：    B + 16位
```

### 21.2 需要对齐的

```
Q21-1：你们调我的 CreditWallet/ConfirmDebit/UnfreezeBalance 时传的 orderNo——
       是用我给你们的 walletOrderNo（钱包侧单号），还是你们自己的 channelOrderNo（支付侧单号）？
       → 这非常关键！因为 orderNo 是我做幂等校验的唯一键
       → 如果用我的单号：你们需要在回调时把我的单号传回来
       → 如果用你们的单号：你们需要保证全局唯一，且和我的单号不冲突

Q21-2：人工修正的 orderNo 由你们生成还是我生成？
       → 我的理解：你们B端审核通过后生成单号，传给我
       → 单号前缀需要区分类型（A=加款/M=减款/B=补单），你们那边有这个规则吗？

Q21-3：订单号的生成规则需要统一吗？
       → 比如：钱包用 C/T/E/A/M/B 前缀，支付用 P 前缀？
       → 确保双方的订单号不会重复
```

### 22.1 充值异常场景

  → 谁以谁为准？我倾向以回调的实际金额为准
```

### 22.2 提现异常场景

```
场景4：提现冻结成功但调支付CreateWithdrawOrder失败
  → 我会自动解冻（unfreezeCore）
  → 需要你们确认：你们侧有没有创建半成品订单？需要清理吗？

场景5：支付调ConfirmDebit失败
  → 出款已经成功了，但冻结没有确认扣除
  → 资金实际已出但钱包账面冻结金额还在
  → 你们会重试吗？有没有对账兜底？

场景6：支付调UnfreezeBalance失败

### 22.3 我的建议

```
双方约定一个对账方案：
  ① 每日对账：你们提供"已上账清单"（充值成功），我提供"已入账清单"，比对差异
  ② 或者：你们维护一个"待钱包处理"队列，失败的自动重试
  ③ 极端情况：人工介入处理（通过人工加款/减款补偿）

关键原则：
  → 宁可多打少赔不打多赔——如果不确定是否上账成功，先查再操作
  → 我这边有 GetBalance 和流水查询接口，你们可以随时核实

# 第五部分：潜在风险与边界案例

## 二十四、必须讨论的边界场景

### 场景1：币种被禁用时的在途订单

```
情况：用户创建了BRL充值订单（待支付），管理员禁用了BRL，用户完成支付后三方回调到达

问题：
  → BRL已禁用，CreditWallet还应该上账吗？
  → 我的建议：已创建的订单应该继续完成（到账后上账），但不允许新建BRL订单
  → 即：禁用对"新订单"生效，对"在途订单"不生效
  → 支付侧同意吗？
```

### 场景2：提现"充什么提什么"规则

```
情况：用户充了VND→兑换成BSB→想提VND

问题：
  → 提现方式列表的过滤逻辑涉及用户充值历史
  → 充值历史数据在谁那边？如果充值订单归支付管理，那我查提现方式时还得调你们查充值历史？
  → 还是说你们在返回提现方式时就已经按用户过滤了？
  → 这个逻辑需要明确分工
```

### 场景3：充值方式列表为空

```
情况：支付侧GetPaymentConfig返回空列表（所有通道维护中）

问题：
  → 我的充值页面展示"暂无可用充值方式"——这个文案需要你们确认
  → 是否需要支付侧返回 maintenanceMsg 告诉用户原因？
  → 建议：至少返回一个状态码区分"没配置"和"全维护中"
```

### 场景4：充值超额

```
情况：用户通过CreatePayOrder创建了充值100的订单，但在三方页面支付了200

问题：
  → 三方回调告诉你们实际支付了200——你们回调CreditWallet时传amount=200还是100？
  → 这个差额怎么处理？
  → 如果传200：用户多充了100，正常上账
  → 如果传100：用户损失了100，三方那边会退吗？
```

### 场景5：并发提现

```
情况：用户快速点击两次"确认提现"

问题：
  → 第一次冻结成功→调支付成功→提现已提交
  → 第二次冻结可能也成功（如果余额够）→调支付→又提了一笔
  → 我这边的防重方式：重复转账检测（同用户+同币种+同金额）
  → 你们侧是否也有去重逻辑？还是完全依赖我？
```

### 场景6：汇率刷新时的充值并发

```
情况：
  用户A 14:00:00 打开充值页面，看到汇率 1BSB = 2620VND
  14:00:05 汇率刷新，变成 1BSB = 2680VND
  14:00:10 用户A点击确认充值——用旧汇率2620还是新汇率2680？

我的方案：
  → 用户打开页面时获取的汇率有效期5分钟
  → 创建订单时校验汇率是否过期
  → 过期则返回"汇率已更新请刷新页面"
  → 未过期则用锁定的快照汇率
  → 你们侧创建通道订单时需要我传这个快照汇率吗？
```

### 场景7：支付服务长时间不可用

```
情况：支付模块宕机2小时

影响：
  → 充值功能完全不可用（页面无法展示方式列表）
  → 提现功能完全不可用（无法提交审核）
  → 但：钱包核心功能正常（余额查询、兑换、RPC暴露接口）

问题：
  → 你们有SLA承诺吗？（如99.9%可用性）
  → 我这边是否需要缓存上次获取的方式列表作为降级展示？
  → 如果缓存降级：用户基于缓存数据发起充值，但通道实际已维护——创建订单会失败
  → 结论：可能不适合缓存降级，而是直接友好提示"暂不可用"更安全
```
