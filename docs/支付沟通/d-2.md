## 一、开场：双方关系一句话说清

```
支付组管"钱从哪来到哪去"（通道、三方、出入款）
钱包组管"账上有多少钱"（余额、流水、冻结）

用户充值：用户 → 支付组匹配通道 → 三方支付成功 → 钱包组上账
用户提现：用户 → 钱包组冻结 → 支付组审核出款 → 成功/失败 → 钱包组确认扣/解冻
```

我们之间是**双向强依赖**：
- 钱包充值/提现的核心链路，离开支付组的4个接口跑不通
- 支付组的每一笔资金操作（入账、冻结、解冻、扣款），都必须调钱包的RPC才能改余额

所以这次沟通的目标：**把两边的接口契约、流程边界、数据规范全部对齐，不留任何灰色地带。**

## 二、我需要你提供的接口（钱包 → 支付）

### 2.1 总览

我需要支付组暴露给我 **4个必须接口 + 2个待定接口**：

| # | 接口名称 | 优先级 | 调用场景 | 频率 |
|---|---------|:------:|---------|------|
| 1 | GetRechargeMethodList | **必须** | 用户进充值页面 | 高 |
| 2 | MatchChannelAndCreateOrder | **必须** | 用户提交充值 | 高 |
| 3 | GetWithdrawMethodList | **必须** | 用户进提现页面 | 高 |
| 4 | GetWithdrawConfig | **必须** | 用户提交提现时校验 | 中 |
| 5 | NotifyNewWithdrawOrder | **待定** | 创建提现订单后通知你们 | 中 |
| 6 | GetRechargeOrderStatus | **待定** | 充值订单状态主动查询 | 低 |

缺前4个，C端充值和提现完全不可用。

### 2.2 接口1：GetRechargeMethodList — 获取充值方式列表

**我在什么场景调这个接口**：
用户进入充值页面，我需要展示"当前币种下有哪些充值方式"（银行转账/扫码/USDT等）。这些数据在你们的配置表里，不在我这里。

**我传给你什么**：

| 参数 | 类型 | 说明 |
|------|------|------|
| currencyCode | string | 用户当前操作的币种（VND/IDR/THB/BSB/USDT） |

**我期望你返回什么**：

| 字段 | 类型 | 说明 | 我用来干什么 |
|------|------|------|------------|
| methodId | int64 | 充值方式ID | 用户选择后传回给你创建订单 |
| methodName | string | 方式名称（"银行转账"/"QRIS"/"USDT-TRC20"） | C端展示 |
| methodType | int32 | 方式类型（1银行/2电子钱包/3扫码/4虚拟币） | C端分组展示 |
| iconUrl | string | 图标 | C端展示 |
| minAmount | int64 | 最低充值金额（最小单位整数） | 前端校验 |
| maxAmount | int64 | 最高充值金额 | 前端校验 |
| fixedAmounts | list\<int64\> | 快捷金额档位列表（如[100000, 200000, 500000]） | C端快捷选择按钮 |
| sort | int32 | 排序值 | C端排列顺序 |

**需要确认的问题**：
1. **这个接口是否需要传 userId？** 有些充值方式可能只对特定用户开放（如VIP才能用某种方式），这种限制在你们那里做还是我这里做？
2. **游客状态怎么处理？** 产品说游客只能USDT充值。这个过滤逻辑谁来做？我传一个 isGuest 标记给你？还是你返回全部方式我来过滤？
3. **充值方式关联的通道信息，需要返回给我吗？** 我理解通道对用户不可见，但我是否需要拿到 channelId 传回给创建订单接口？

### 2.3 接口2：MatchChannelAndCreateOrder — 匹配通道+创建通道订单

**我在什么场景调这个接口**：
用户选好充值方式、输入金额、选好奖金后，点击提交。我这边先创建平台充值订单（status=待支付），然后调你这个接口匹配通道、向三方发起代收请求、拿到支付URL返回给前端。

**我传给你什么**：

| 参数 | 类型 | 说明 |
|------|------|------|
| platformOrderNo | string | 我方生成的平台订单号（R+时间戳+随机数） |
| userId | int64 | 用户ID |
| currencyCode | string | 充值币种 |
| methodId | int64 | 用户选择的充值方式ID |
| amount | int64 | 充值金额（最小单位整数） |
| callbackUrl | string | 充值成功后的回调地址（**需要确认是否由我传**） |

**我期望你返回什么**：

| 字段 | 类型 | 说明 | 我用来干什么 |
|------|------|------|------------|
| channelOrderNo | string | 通道订单号 | 存到我的 recharge_order.channel_order_no |
| channelName | string | 匹配到的通道名称 | 存到我的 recharge_order.channel_name |
| payType | int32 | 支付类型（1跳转URL/2展示地址/3展示二维码） | 前端根据类型做不同展示 |
| payUrl | string | 三方支付URL（payType=1时有值） | 前端跳转 |
| payAddress | string | 收款地址（payType=2时，USDT充值） | 前端展示地址+复制 |
| qrContent | string | 二维码内容（payType=3时） | 前端生成二维码 |
| expireAt | int64 | 支付过期时间（毫秒时间戳） | 前端倒计时 |

**需要确认的问题**：
1. **订单号谁生成？** 我的方案：我生成平台订单号（R开头），你生成通道订单号。后续回调用哪个号做关联？
2. **匹配失败怎么处理？** 所有通道都不可用时返回什么错误码？我需要告诉用户"当前充值方式暂不可用"
3. **USDT充值地址谁生成？** USDT充值不走三方通道，地址是你们生成的还是我们自己生成的？还是有独立的链上地址服务？
4. **这个接口的超时预期是多少？** 你向三方发起代收请求可能有网络延迟，我设超时5秒够吗？
5. **通道费率信息需要返回吗？** 我的 recharge_order 有 fee_rate/fee_amount 字段，这些数据你返回给我还是回调时带回？

### 2.4 接口3：GetWithdrawMethodList — 获取提现方式列表

**我在什么场景调这个接口**：
用户进入提现页面，我需要展示当前币种可用的提现方式。在调你之前，我会先查KYC状态——如果用户未认证，我只展示USDT提现。

**我传给你什么**：

| 参数 | 类型 | 说明 |
|------|------|------|
| currencyCode | string | 用户当前操作的币种 |

**我期望你返回什么**：

| 字段 | 类型 | 说明 | 我用来干什么 |
|------|------|------|------------|
| methodId | int64 | 提现方式ID | 用户选择后传回给我创建订单 |
| methodName | string | 方式名称 | C端展示 |
| methodType | int32 | 方式类型（1银行卡/2电子钱包/3USDT） | C端分组 |
| iconUrl | string | 图标 | C端展示 |
| feeType | int32 | 费率类型（1百分比/2固定金额/3混合） | 手续费计算 |
| feeRate | int32 | 手续费率（×10000，如 50=0.50%） | 手续费计算 |
| feeFixed | int64 | 固定手续费（最小单位） | 手续费计算 |
| minAmount | int64 | 单笔最低提现金额 | 前端校验 |
| maxAmount | int64 | 单笔最高提现金额 | 前端校验 |

**需要确认的问题**：
1. **提现限额配置在谁那里？** 日次数限制、日累计限额、大额审核阈值——这些参数归你管还是归我管？
2. **KYC过滤谁来做？** 未认证用户只能USDT提现。我在我这里过滤（只展示USDT），还是你返回全部方式+KYC要求标记？
3. **手续费计算逻辑在谁那里执行？** 我理解应该是我这边算（拿到你的费率后），然后用户看到"提现100元，手续费0.5元，实际到账99.5元"。对吗？

### 2.5 接口4：GetWithdrawConfig — 获取提现限额和费率配置

**我在什么场景调这个接口**：
用户输入提现金额点击提交时，我需要校验：金额是否在范围内、今天是否超次数/超限额、手续费多少。

**我传给你什么**：

| 参数 | 类型 | 说明 |
|------|------|------|
| currencyCode | string | 币种 |
| methodId | int64 | 提现方式ID |
| userId | int64 | 用户ID（用于查询今日已提现次数/金额） |

**我期望你返回什么**：

| 字段 | 类型 | 说明 |
|------|------|------|
| minAmount | int64 | 单笔最低 |
| maxAmount | int64 | 单笔最高 |
| dailyMaxAmount | int64 | 每日累计上限 |
| dailyMaxCount | int32 | 每日次数上限 |
| todayUsedAmount | int64 | 今日已用额度 |
| todayUsedCount | int32 | 今日已用次数 |
| feeRate | int32 | 手续费率 |
| feeMin | int64 | 最低手续费 |
| largeAmountThreshold | int64 | 大额审核触发金额 |

**需要确认的问题**：
1. **这个接口和接口3有重复字段（费率/限额），是否合并？** 还是说接口3只返回方式列表+基础信息，接口4才返回详细配置？
2. **"今日已提现次数/金额"谁来统计？** 提现订单在我的表里（withdraw_order），我可以自己统计。但如果你也需要限制，那统计口径必须一致。
3. **大额提现审核阈值：** 超过阈值的订单是自动进入人工审核，还是直接拒绝？审核逻辑在你们那里？

### 2.6 接口5（待定）：NotifyNewWithdrawOrder — 新提现订单通知

**背景**：
用户提现 → 我这边冻结余额 + 创建 withdraw_order → 然后需要通知你们"有一笔新的提现订单待审核"。

**要讨论的问题**：
1. **用RPC还是用消息队列？**
   - 方案A：我RPC调你 NotifyNewWithdrawOrder
   - 方案B：我发 NATS/Kafka 消息，你消费
   - 我倾向方案B（解耦，你可以按自己的节奏消费），但需要你确认
2. **通知需要带什么数据？** 我可以带：orderNo、userId、currencyCode、amount、methodId、accountInfo
3. **还是说你直接轮询我的订单表？** 你定期查 withdraw_order status=待风控 的记录？（不推荐，延迟大）

### 2.7 接口6（待定）：GetRechargeOrderStatus — 充值订单状态查询

**背景**：
充值订单创建后，用户可能长时间没操作。我需要知道这笔订单在三方那边是什么状态（是否超时、是否支付中）。

**要讨论的问题**：
1. **你们会有主动状态推送吗？** 除了支付成功的回调，超时/取消你们会通知我吗？
2. **我是否需要一个主动查询接口？** 传 platformOrderNo 或 channelOrderNo，查当前状态
3. **或者我设置订单超时？** 比如30分钟未回调就自动标记为"失败/过期"，不依赖你的状态

## 三、你需要调我的接口（支付 → 钱包）

### 3.1 总览

你需要调用我的 **8个RPC接口**：

| # | 接口名称 | 场景 | 性质 |
|---|---------|------|------|
| 1 | CreditWallet | 充值到账/补单/人工加款 | 写（余额增加） |
| 2 | DebitWallet | 人工减款 | 写（余额减少） |
| 3 | FreezeBalance | 人工冻结 | 写（冻结） |
| 4 | UnfreezeBalance | 提现驳回/出款失败/人工解冻 | 写（解冻） |
| 5 | ConfirmDebit | 提现出款成功 | 写（确认扣） |
| 6 | GetBalance | 人工操作前展示余额 | 读 |
| 7 | GetCurrencyList | 查可用币种 | 读 |
| 8 | GetExchangeRate | 报表汇率换算 | 读 |

### 3.2 重点接口说明

#### CreditWallet — 你最常调的接口

你在以下场景需要调我：

| 你的场景 | bizType（我定义的业务类型枚举） | 目标钱包 |
|---------|------|---------|
| 充值回调成功 → 入账 | 1（充值） | 中心钱包 |
| 充值回调成功 + 有奖金 → 奖金入账 | 4（奖金） | 奖励钱包 |
| 补单审核通过 → 补录入账 | 7（补单） | 中心钱包 |
| 人工加款审核通过 → 加款 | 5（人工加款） | 指定钱包 |

**你需要传的参数**：
```
userId      int64   目标用户ID
currencyCode string  币种
walletType   int32   钱包类型（1中心/2奖励）
amount       int64   金额（最小单位正整数）
bizType      int32   业务类型（用我定义的枚举）
refOrderNo   string  你的业务订单号（充值订单号/补单号/加款单号）
remark       string  备注（可选）
```

**我返回给你**：
```
afterBalance int64   操作后余额
flowId       int64   流水ID
```

**关键约定**：
- **refOrderNo 是幂等键**：同一个 refOrderNo 重复调我，不会重复上账，返回首次操作结果。这是你的安全保障——回调重试不怕重复。
- **walletType 必须明确传**：充值进中心钱包=1，奖金进奖励钱包=2，不能传错。
- **amount 必须是最小单位正整数**：VND 100万 = 传 1000000，THB 150.50 = 传 15050。

#### UnfreezeBalance / ConfirmDebit — 提现终态

你在审核提现订单后需要调我：

| 你的审核结果 | 调我哪个接口 | 效果 |
|------------|-----------|------|
| 风控驳回 | UnfreezeBalance | 冻结金额全部解冻，用户可用余额恢复 |
| 财务驳回 | UnfreezeBalance | 同上 |
| 出款失败 | UnfreezeBalance | 同上 |
| 出款成功 | ConfirmDebit | 冻结金额正式扣除，余额减少 |

**你需要传的参数**：
```
userId       int64   用户ID
currencyCode string  币种
refOrderNo   string  提现订单号（与 FreezeBalance 时传的一致）
```

**关键约定**：
- **refOrderNo 必须和冻结时一致**：我通过 refOrderNo 查找冻结明细，知道每个钱包冻了多少
- **UnfreezeBalance 和 ConfirmDebit 互斥**：同一笔只能调一个，调了解冻就不能再调确认
- **幂等**：重复调不会出错

### 3.3 接口IDL我来定义

这些接口的 Thrift IDL 由我来定义（在 common/idl/ser-wallet/wallet_rpc.thrift 中）。我会提前给你们 IDL 文件，你们在代码中通过 `rpc.WalletClient()` 调用。

**我需要你们配合**：
- 在 common/rpc/rpc_client.go 中注册 WalletClient（或者我来注册，你们直接用）
- 按我定义的 bizType 枚举传值，不要自己发明新的类型

## 四、核心流程边界：谁干什么必须对齐

### 4.1 充值流程 — 逐步拆解

```
步骤1  用户进入充值页面，选择币种
       → 谁干：钱包（展示当前币种，调支付的 GetRechargeMethodList）

步骤2  用户选择充值方式、输入金额、选择奖金
       → 谁干：钱包（前端交互 + 奖金列表展示）

步骤3  用户点击提交
       → 谁干：钱包（创建 recharge_order status=待支付）
       → 然后：钱包调支付的 MatchChannelAndCreateOrder

步骤4  支付匹配通道、向三方发起代收请求
       → 谁干：支付（内部轮询匹配通道 → 调三方API → 返回支付URL）

步骤5  前端跳转三方支付页面 / 展示充值地址
       → 谁干：前端（根据 payType 决定展示方式）

步骤6  用户在三方完成支付

步骤7  ★★★ 三方回调通知支付成功 ★★★
       → 【核心问题】回调到谁？
       → 方案A：回调到支付，支付验签后调钱包 CreditWallet
       → 方案B：回调到钱包，钱包自己验签后 CreditWallet
       → 【必须本次讨论敲定】

步骤8  上账：CreditWallet 中心钱包 + 奖金到奖励钱包
       → 谁干：钱包（不管回调到谁，最终由钱包执行上账）

步骤9  更新订单状态
       → 谁干：钱包更新 recharge_order status=成功
       → 支付是否也有自己的订单表需要更新？
```

**必须敲定**：
- 步骤7：回调地址是支付的还是钱包的？
- 如果回调到支付：支付验签后用什么方式通知钱包？（RPC调 CreditWallet + 带上充值结果？）
- 如果回调到钱包：验签需要通道的密钥，密钥在支付那里，钱包怎么拿到验签规则？

### 4.2 提现流程 — 逐步拆解

```
步骤1  用户进入提现页面
       → 钱包：查KYC（调ser-kyc）+ 查提现方式（调支付 GetWithdrawMethodList）

步骤2  用户选方式、输金额、选账户、提交
       → 钱包：校验限额（调支付 GetWithdrawConfig）
       → 钱包：FreezeBalance 冻结余额
       → 钱包：创建 withdraw_order（status=待风控）

步骤3  ★★★ 通知支付有新提现订单 ★★★
       → 【核心问题】怎么通知？RPC/MQ/轮询？
       → 通知里带什么数据？

步骤4  风控审核
       → 谁干：支付（还是有独立风控模块？）
       → 审核结果：通过 → 进入步骤5 | 驳回 → 调钱包 UnfreezeBalance

步骤5  财务审核
       → 谁干：支付
       → 审核结果：通过 → 进入步骤6 | 驳回 → 调钱包 UnfreezeBalance

步骤6  发起三方出款
       → 谁干：支付（匹配代付通道 → 调三方出款API）

步骤7  三方出款结果
       → 成功 → 支付调钱包 ConfirmDebit
       → 失败 → 支付调钱包 UnfreezeBalance

步骤8  更新订单状态
       → 钱包：更新 withdraw_order 状态
       → 支付：是否也有自己的订单/审核记录？
```

**必须敲定**：
- 步骤3：通知机制

## 六、灰色地带：必须这次会上敲定

### 6.1 充值回调归谁接收？

| 方案 | 流程 | 优点 | 缺点 |
|------|------|------|------|
| A：回调到支付 | 三方→支付验签→支付调钱包CreditWallet | 验签逻辑在支付内闭环，通道密钥不需要给钱包 | 多一跳RPC |
| B：回调到钱包 | 三方→钱包验签→钱包自己CreditWallet | 少一跳，钱包掌握全流程 | 钱包需要知道通道密钥和验签规则 |

**我的倾向**：方案A。理由：
- 通道密钥/验签逻辑是支付域的东西，不应该散到钱包
- 不同通道验签规则不一样，支付统一处理更合理
- 缺点是多一跳RPC，但充值回调不是高频操作

**但是**：如果用方案A，支付调我 CreditWallet 时需要传什么数据？我列一下：
```
你调 CreditWallet 时需要告诉我：
- userId           谁充的值
- currencyCode     充的什么币种
- walletType = 1   进中心钱包
- amount           实际到账金额（扣除手续费后的）
- bizType = 1      充值
- refOrderNo       平台充值订单号（R开头那个）

如果有奖金，你还需要第二次调 CreditWallet：
- walletType = 2   进奖励钱包
- amount           奖金金额
- bizType = 4      奖金
- refOrderNo       平台充值订单号 + "-bonus" 后缀（区分幂等键）
```

**等等，奖金入账谁来触发？** 奖金信息在我的 recharge_order 里（bonus_amount），你回调成功后只需要告诉我"订单R123456支付成功了"，然后我自己去查奖金信息做第二笔上账。这样你不需要知道奖金逻辑。

→ 那方案A的流程就是：**支付告诉我"哪个订单成功了+实际金额"，我自己处理上账+奖金。**

### 6.2 提现订单状态谁维护？

```
                            → 7.出款成功（终态）
```

**问题**：状态流转由谁驱动？
- 我创建订单后 status=1（待风控）

**我的倾向**：你审核完通知我（RPC或MQ），我自己更新状态。理由：
- 不应该跨模块直接写对方的数据库
- 我的表我维护，你负责审核逻辑和调我的冻结/解冻接口

**具体方案**：
```
你审核完后调我的某个接口（或发消息）：
  通过风控 → 通知我更新状态为3
  风控驳回 → 调我 UnfreezeBalance + 通知我更新状态为2
  财务通过 → 通知我更新状态为5
  财务驳回 → 调我 UnfreezeBalance + 通知我更新状态为4
  出款成功 → 调我 ConfirmDebit + 通知我更新状态为7
  出款失败 → 调我 UnfreezeBalance + 通知我更新状态为6
```

**或者更简单**：你调 UnfreezeBalance/ConfirmDebit 时带上"驳回原因/审核人"等信息，我自己根据调用类型更新状态？
