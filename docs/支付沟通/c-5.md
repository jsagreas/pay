# 会前须知 — 为什么要开这个会

钱包和支付是**双向强依赖**关系：

```
钱包调支付：4 个 RPC 接口（充值方式查询、创建渠道订单、提现方式查询、提交提现审核）
支付调钱包：8 个 RPC 接口（充值入账、提现确认/解冻、人工加减款/补单、余额查询）
```

这 12 个 RPC 接口构成充值/提现两条完整链路的"胶水"。**任何一个接口的参数理解偏差，联调时都会出问题**。

本次会议目标：
1. 对齐双方的职责边界（谁管什么、谁做什么）
2. 逐接口确认请求/响应参数结构
3. 明确回调协议（时序、重试、幂等）
4. 暴露所有待确认的业务问题
5. 约定 IDL 交付时间线

# 一、职责边界 — 谁管什么

## 1.1 充值链路的职责切分

```
┌─────────────────────────────────────────────────────────────────┐
│                    充值全链路（从左到右）                          │
│                                                                 │
│  C端用户                 钱包(我)              支付(你)    渠道    │
│  ──────                 ────────             ────────    ────    │
│  选币种                  校验参数                                 │
│  选充值方式  ──请求──→   展示方式列表 ←RPC查── 支付配置             │
│  输金额                  创建本地订单                              │
│  确认充值   ──请求──→   调支付创建   ──RPC──→ 创建渠道订单 ──→ 渠道 │
│                         存 channelOrderNo                        │
│  跳转支付   ←──返回──   透传支付信息 ←──返回── payUrl/address      │
│  完成支付                                      ←── 渠道回调       │
│                                    校验签名                       │
│             ←RPC入账──  余额+订单+流水          确认到账通知我      │
│  看到到账                事务操作完成                              │
└─────────────────────────────────────────────────────────────────┘
```

**关键边界**：
- **钱包负责**：用户交互、本地订单管理、余额操作（入账/冻结/解冻）、流水记录
- **支付负责**：支付方式配置管理、渠道对接、渠道订单管理、回调签名校验、三级审核
- **交界点**：通过 RPC 接口连接，以 `orderNo`（平台订单号）为纽带关联

## 1.2 提现链路的职责切分

```
┌─────────────────────────────────────────────────────────────────┐
│                    提现全链路（从左到右）                          │
│                                                                 │
│  C端用户            钱包(我)                 支付(你)       渠道   │
│  ──────            ────────                ────────       ────   │
│  点提现             KYC前置校验                                    │
│  看限额             查限额+提现方式  ←RPC── 提现方式配置            │
│  填账户信息          保存提现账户                                   │
│  确认提现  ─请求─→  冻结余额(乐观锁)                               │
│                     生成提现订单                                    │
│                     调支付提交审核  ──RPC──→ 接收审核单              │
│                                             风控审核                │
│                                             财务审核                │
│                                             出款审核                │
│                                             选通道  ──→ 渠道出款   │
│                                                        ←── 回调   │
│           ←RPC确认── 确认扣款/解冻  ←──RPC── 通知结果              │
│  看到结果            事务操作完成                                   │
└─────────────────────────────────────────────────────────────────┘
```

**关键边界**：
- **钱包负责**：KYC校验（调ser-kyc）、余额冻结/解冻/确认、提现账户管理、按充值来源过滤提现方式
- **支付负责**：支付方式配置管理、三级审核流程、通道选择与出款、渠道回调处理
- **冻结→提交审核的补偿**：如果调支付提交审核 RPC 失败，**由钱包负责立即解冻**（补偿机制）

## 1.3 人工修正链路

```
┌─────────────────────────────────────────────────────────────────┐
│                    人工修正链路                                    │
│                                                                 │
│  B端操作员            支付(你)                   钱包(我)          │
│  ──────              ────────                  ────────          │
│  发起加款/减款/补单    创建修正单                                   │
│  审核通过             审核流程完成                                  │
│                      调钱包执行    ──RPC──→    操作余额            │
│                                               写流水记录          │
│                                   ←──返回──   返回成功/失败       │
└─────────────────────────────────────────────────────────────────┘
```

**关键边界**：
- **支付负责**：修正单的创建、审核流程（"操作人≠审核人"合规要求）
- **钱包负责**：余额操作的原子性、幂等性
- **减款前**：支付应先调 `QueryBalanceRpc` 确认用户余额充足

## 2.3 接口 C：提交提现审核（SubmitWithdrawReview）

**调用场景**：C端用户确认提现后，钱包先冻结余额（available - amount, frozen + amount），再调支付提交审核。

# 三、支付需要调钱包的接口 — "你需要我给你什么"

## 3.1 钱包 RPC 暴露的 8 个接口概览

```
方法名              触发场景                操作              幂等键
──────────────     ────────────           ──────            ──────
DepositCredit      渠道充值回调成功        available + amt    refOrderNo
WithdrawFreeze     钱包创建提现时内部调    available - / frozen +  refOrderNo
WithdrawConfirm    审核通过+出款成功       frozen - amt       refOrderNo
WithdrawUnfreeze   审核拒绝/出款失败       available + / frozen -  refOrderNo
ManualCredit       加款审核通过            available + amt    refAdjustNo
ManualDebit        减款审核通过            available - amt    refAdjustNo
ManualPatch        补单审核通过            available + amt    refAdjustNo
QueryBalanceRpc    任何需查余额时          纯读取             无
```

**统一设计原则**（需支付方知晓并遵循）：

```
1. 所有写接口带 refOrderNo / refAdjustNo（幂等键）
2. userId 显式传参（RPC 不经网关，没有 JWT）
3. 返回 bool（成功=true，失败通过 Kitex BizErr 返回错误码+错误信息）
4. walletType 取值：1=中心 2=奖励 3=代理 4=主播
5. 金额单位：i64 最小单位整数（和钱包侧一致）
6. 我方保证幂等：同一 refOrderNo 重复调用直接返回 true
```

## 3.2 接口详细参数 — DepositCredit（充值入账）

```
DepositCreditReq {
    userId       i64       必填    用户ID
    currency     string    必填    币种
    amount       i64       必填    入账金额（最小单位整数）
    refOrderNo   string    必填    关联订单号（钱包的 deposit_order.order_no）= 幂等键
    walletType   i32       必填    目标子钱包（充值默认=1中心钱包）
    // 问题: 你传的 refOrderNo 就是钱包之前传给你的 orderNo？还是你自己的渠道订单号？
    // 问题: 如果充值有奖金活动，奖金金额和奖金目标钱包由谁告诉谁？
}
```

**钱包侧操作**：
1. 用 refOrderNo 查 deposit_order → 已入账 → 直接返回 true（幂等）
2. 事务内：deposit_order.status = 成功 + user_balance.available += amount + 写 balance_flow
3. 如有奖金活动：奖励钱包额外 +giftAmount + 写奖金流水

## 3.3 接口详细参数 — WithdrawConfirm / WithdrawUnfreeze

```
WithdrawConfirmReq {
    userId       i64       必填
    currency     string    必填
    amount       i64       必填    确认扣款金额
    refOrderNo   string    必填    关联提现订单号 = 幂等键
    walletType   i32       必填    子钱包（提现默认=1中心钱包）
}

WithdrawUnfreezeReq {
    userId       i64       必填
    currency     string    必填
    amount       i64       必填    解冻金额
    refOrderNo   string    必填
    walletType   i32       必填
    failReason   string    可选    失败原因（展示给用户看/记录在流水中）
    // 问题: failReason 格式是什么？错误码+文本？还是纯文本？
}
```

**钱包侧操作**：
- Confirm：`frozen -= amount` + 更新订单状态=成功 + 写流水
- Unfreeze：`available += amount, frozen -= amount` + 更新订单状态=失败 + 写回退流水

## 3.5 接口详细参数 — QueryBalanceRpc

# 四、回调协议 — 最容易出错的地方

## 4.1 需要对齐的 6 个协议问题

这些问题**不对齐就会出 Bug**，是联调最常见的坑：

### 问题 1：调用方式 — 同步 RPC 还是异步消息

```
场景：渠道回调你充值成功 → 你通知我入账

方案 A（同步 RPC）：你收到渠道回调 → 同步调我 DepositCredit → 等我返回 → 你更新你的状态
方案 B（异步消息）：你收到渠道回调 → 更新你的状态 → 发消息 → 消费者调我 DepositCredit

我这边 8 个 RPC 接口都支持同步调用。
你那边是哪种方案？如果是消息队列，用什么中间件？
```

### 问题 2：重试策略

```
如果你调我 DepositCredit 网络超时（你没收到我的返回），你怎么处理？

选项 A：立即重试（我方幂等，直接返回 true）
选项 B：延迟重试（几秒后重试）
选项 C：人工介入（标记异常，运营查看）
选项 D：定时任务扫描（补偿机制）

需要确认：
  - 最多重试几次？
  - 重试间隔多久？
  - 超过最大重试次数后怎么处理？
  - 每次重试的参数完全一致吗？（我方幂等依赖 refOrderNo 不变）
```

### 问题 3：幂等约定的细节

```
我方保证：同一 refOrderNo + changeType 的组合，重复调用直接返回 true（不重复操作余额）。

你方需保证：
  - 每次重试传的 refOrderNo 不变（不能每次生成新的）
  - 金额等核心参数不变（不能第一次传 100，重试传 200）
  - 如果参数变了，我会当作新的请求处理（可能导致重复入账）

确认：你能保证重试参数不变吗？
```

### 问题 4：回调顺序

```
场景：同一用户有 3 笔提现在审核中，审核结果出来了。

你调我的顺序是：
  A：按审核完成顺序（先审完先调，可能乱序）
  B：按提交顺序（FIFO，保证顺序）
  C：无顺序保证

我方通过乐观锁控制并发，不依赖顺序。但如果你保证顺序，我可以做额外校验。
```

### 问题 5：失败原因格式

```
你调我 WithdrawUnfreeze 时，failReason 字段的格式是什么？

选项 A：纯文本（如"风控审核拒绝：疑似异常交易"）
选项 B：错误码+文本（如"RISK_REJECT:疑似异常交易"）
选项 C：结构化JSON

这个内容最终会展示给用户看（提现记录详情的"失败原因"字段），
所以需要是用户可理解的语言，而不是系统内部日志。
需不需要做多语言？（VND用户看越南语、THB用户看泰语）
```

### 问题 6：超时与兜底

```
场景：钱包冻结余额 → 调你 SubmitWithdrawReview → 你创建了审核单 → 但72小时没有任何回调

我方设计：冻结就一直冻着，等你回调。不主动超时（怕和你的审核流程冲突）。

问题：
  - 你的审核流程有没有最长时限？（超过X天自动拒绝 → 调我解冻？）
  - 如果你们系统故障长时间没回调，有没有兜底机制？
  - 是否需要我提供"查询审核状态"的场景？（我轮询你 or 用户查进度）
```

## 4.2 状态同步 — 两张表的状态对照

```
钱包 deposit_order 状态          支付 finance_deposit 状态
────────────────────           ─────────────────────────
pending（待支付）               → created（已创建）
                                 processing（支付中/渠道处理中）
success（已入账）              ← paid（已支付） → 触发 DepositCredit
failed（已失败）               ← failed（支付失败）
expired（已过期）              ← expired（渠道超时）
patched（补单成功）            ← patched（人工确认到账）
```

```
钱包 withdraw_order 状态        支付 finance_withdraw 状态
────────────────────           ─────────────────────────
reviewing（审核中）             → risk_pending（待风控审核）
                                 risk_passed → finance_pending（待财务审核）
                                 finance_passed → payout_pending（待出款审核）
                                 payout_passed → processing（出款中）
success（已完成）              ← success（出款成功） → 触发 WithdrawConfirm
failed（已失败/已拒绝）        ← rejected/failed → 触发 WithdrawUnfreeze
```

**关键点**：我方只有 3-5 种终态，你方有更细粒度的中间态。我不需要知道你的全部中间态，但需要你在达到终态时准确调我。

### 需要确认的状态映射问题

| # | 问题 |
|---|------|
| S1 | 支付失败和渠道超时，你是分开通知我还是统一为"failed"？ |
| S2 | 提现被风控拒绝 vs 被财务拒绝 vs 被出款拒绝 — 对我来说都是"解冻"，但 failReason 不同，你怎么区分？ |
| S3 | 充值订单你那边有超时机制吗？超时后渠道延迟到账你怎么处理？ |
| S4 | 补单场景：你调我 ManualPatch，我要不要更新 deposit_order 的状态？ |

# 七、金额单位与精度 — 必须完全一致

## 7.1 金额传输约定

```
双方约定（需确认）：

1. 所有金额字段统一使用 i64（Thrift）/ BIGINT（数据库）

  "10.50 BSB"（假设精度=2） → 传输/存储 1050
```

## 13.1 技术风险
