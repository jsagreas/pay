# ser-wallet 接口统计（整合版）

> 整合来源：接口评估(11版迭代)、IDL设计、RPC暴露分析、RPC调用分析、链路关系、依赖关系、需求理解
> 输出时间：2026-02-28
> 分类标记：[必须] = P0核心链路必备 | [应有] = P1完整体验需要 | [商榷] = P2待确认或可后置

路由规范（工程约定，全部POST）：

```
C端（gate-font）：POST /api/wallet/xxx（登录态） | POST /open/wallet/xxx（无需登录）
B端（gate-back）：POST /admin/api/wallet/xxx（登录态）
```

## 一、C端 HTTP 接口（gate-font → ser-wallet）

共 20 个：13 必须 + 5 应有 + 2 商榷

### 1.1 钱包核心（2个）

1. [必须] `POST /api/wallet/getWalletInfo` — **获取钱包信息**
   - 返回当前币种下各子钱包余额（中心/奖励/资金汇总）、可用币种列表、当前币种配置
   - 内部调用：币种配置 GetCurrencyList + GetCurrencyDetail

2. [必须] `POST /api/wallet/switchCurrency` — **切换币种**
   - 切换用户当前操作币种，后续所有操作基于新币种上下文
   - 纯钱包内部操作，无外部依赖

### 1.2 充值（6个）

3. [必须] `POST /api/wallet/getRechargeMethodList` — **获取充值方式列表**
   - 按当前币种返回可用充值方式（银行卡/电子钱包/USDT等）及金额区间
   - 外部依赖：ser-finance.GetRechargeMethodList（强依赖，财务模块未建）

4. [必须] `POST /api/wallet/createRechargeOrder` — **创建充值订单**
   - 选择充值方式 + 金额 + 奖金，创建平台订单，返回三方支付URL/信息
   - 外部依赖：ser-finance.MatchChannelAndCreateOrder + 币种配置.GetExchangeRate

5. [必须] `POST /open/wallet/rechargeCallback` — **充值回调**（无需登录）
   - 三方支付通道结果回调，验签 + 幂等校验 + 上账（CreditWallet）+ 写流水
   - 关键：必须幂等，重复回调不可重复上账

6. [必须] `POST /api/wallet/getUsdtAddress` — **获取USDT充值地址**
   - 按链类型（TRC20/ERC20/BEP20）返回充值地址

7. [应有] `POST /api/wallet/checkRepeatRecharge` — **重复转账检查**
   - 同用户 + 同币种 + 同金额短时间内重复检测，达3笔触发拦截提示

8. [应有] `POST /api/wallet/getBonusList` — **获取额外奖金列表**
   - 充值金额联动的活动奖金选项，选择后随充值一起发放到奖励钱包

### 1.3 提现（5个）

9. [必须] `POST /api/wallet/getWithdrawMethodList` — **获取提现方式列表**
   - 按当前币种 + KYC状态返回可用提现方式（未认证用户仅USDT）
   - 外部依赖：ser-kyc.KycQuery + ser-finance.GetWithdrawMethodList

10. [必须] `POST /api/wallet/createWithdrawOrder` — **创建提现订单**
    - 选择提现方式 + 金额 + 账户，KYC校验 → 余额冻结（FreezeBalance）→ 创建订单
    - 外部依赖：ser-kyc.KycQuery + ser-finance.GetWithdrawConfig

11. [必须] `POST /api/wallet/getWithdrawAccountList` — **提现账户列表**
    - 获取用户已保存的提现银行卡 / 电子钱包账户列表

12. [应有] `POST /api/wallet/saveWithdrawAccount` — **保存提现账户**
    - 新增/编辑银行卡或电子钱包账户，持有人姓名须匹配KYC认证姓名

13. [应有] `POST /api/wallet/deleteWithdrawAccount` — **删除提现账户**
    - 软删除，删前校验账户归属权

### 1.4 兑换（2个）

14. [必须] `POST /api/wallet/getExchangeInfo` — **获取兑换信息**
    - 返回法币→BSB实时汇率、可兑换余额、赠送比例
    - 内部调用：币种配置.GetExchangeRate

15. [必须] `POST /api/wallet/doExchange` — **执行兑换**
    - 法币→BSB原子操作：扣法币钱包（DebitWallet）+ 加BSB钱包（CreditWallet）
    - 事务一致性要求，内部调用币种配置.GetExchangeRate

### 1.5 记录（2个）

16. [必须] `POST /api/wallet/getRecordList` — **记录列表**
    - 按分类（充值/提现/兑换/奖励）分页查询订单记录
    - 内部调用：币种配置.GetCurrencyDetail（展示格式化用）

17. [必须] `POST /api/wallet/getRecordDetail` — **记录详情**
    - 单条订单完整信息，按订单ID + 类型查询
    - 纯本地DB查询，无外部依赖

### 1.6 辅助（3个）

18. [应有] `POST /api/wallet/getKycStatus` — **KYC状态查询**
    - 透传 ser-kyc 查询用户认证状态，提现前前端预校验用
    - 外部依赖：ser-kyc.KycQuery

19. [商榷] `POST /api/wallet/cancelRechargeOrder` — **取消充值订单**
    - 用户主动取消待支付状态的充值订单
    - 商榷：待支付订单可设自动过期，主动取消是否需要待产品确认

20. [商榷] `POST /api/wallet/cancelWithdrawOrder` — **取消提现订单**
    - 用户在待审核阶段取消提现，触发 UnfreezeBalance
    - 商榷：取消提现涉及解冻流程，产品未明确是否允许用户端取消

## 二、B端 HTTP 接口（gate-back → ser-wallet）

共 11 个：7 必须 + 2 应有 + 2 商榷

### 2.1 币种配置管理（9个）

1. [必须] `POST /admin/api/wallet/setBaseCurrency` — **设置基准币种**
   - 首次设置平台基准币种（推荐USDT），一旦设置不可更改
   - 全局影响：BSB锚定（10 BSB = 1 USDT）、活动奖金计算、财务统计换算

2. [必须] `POST /admin/api/wallet/createCurrency` — **新增币种**
   - 创建新币种配置（代码/名称/图标/千分位/小数位/汇率浮动/偏值）
   - 内部调用：ser-s3 上传图标 + ser-blog 写操作日志

3. [必须] `POST /admin/api/wallet/updateCurrency` — **编辑币种**
   - 修改汇率浮动比例、偏值阈值、图标、显示规则等
   - 内部调用：ser-blog 写操作日志

4. [必须] `POST /admin/api/wallet/statusCurrency` — **启用/禁用币种**
   - 切换币种状态，禁用后C端不可见、不可操作
   - 内部调用：ser-blog 写操作日志

5. [必须] `POST /admin/api/wallet/pageCurrency` — **币种列表**
   - 分页查询所有币种配置（含实时汇率、状态、创建时间）

6. [必须] `POST /admin/api/wallet/getRateLogList` — **汇率日志列表**
   - 查询汇率变动历史（定时/手动触发的汇率更新记录）

7. [必须] `POST /admin/api/wallet/getOperationLogList` — **操作日志列表**
   - 查询币种配置变更记录
   - 内部调用：ser-blog.GetActLogList
   - 备注：当前IDL未定义此方法，编码阶段需补充

8. [应有] `POST /admin/api/wallet/getCurrencyDetail` — **币种详情**
   - 查看单个币种完整配置信息（汇率浮动、偏值、显示规则等）

9. [应有] `POST /admin/api/wallet/refreshRate` — **手动刷新汇率**
   - 手动触发并发调3个三方汇率API，取平均值更新平台汇率
   - 内部调用：ser-blog 写操作日志

### 2.2 可选（2个）

10. [商榷] `POST /admin/api/wallet/exportRateLog` — **汇率日志导出**
    - 导出汇率变动记录 CSV 文件
    - 商榷：初期列表接口够用，导出按运营实际反馈再定

11. [商榷] `POST /admin/api/wallet/batchStatusCurrency` — **批量启用/禁用**
    - 批量切换多个币种状态
    - 商榷：币种总量有限（5-10个），批量场景不大

## 三、对外暴露 RPC 接口（ser-wallet → 其他模块调用）

共 12 个：9 必须 + 2 应有 + 1 商榷

### 3.1 资金操作原语（5个，写操作，全部幂等）

1. [必须] `CreditWallet(CreditWalletReq) → CreditWalletResp` — **上账/加款**
   - 增加指定钱包余额 + 写资金流水，幂等键 = refOrderNo
   - 调用方：财务（充值入账/补单/人工加款）、活动（奖金发放）、兑换（BSB到账）

2. [必须] `DebitWallet(DebitWalletReq) → DebitWalletResp` — **扣款/减款**
   - 扣减指定钱包余额 + 写资金流水，余额不足返回错误，幂等键 = refOrderNo
   - 调用方：财务（人工减款）、游戏（投注扣款）、兑换（法币扣减）

3. [必须] `FreezeBalance(FreezeBalanceReq) → FreezeBalanceResp` — **冻结余额**
   - 冻结指定金额（优先中心钱包，不足部分冻结奖励钱包），幂等键 = refOrderNo
   - 调用方：提现流程（创建订单时冻结）、财务（人工冻结）

4. [必须] `UnfreezeBalance(UnfreezeBalanceReq) → UnfreezeBalanceResp` — **解冻余额**
   - 解冻已冻结金额，恢复可用余额，幂等键 = refOrderNo
   - 调用方：财务（提现驳回/出款失败/人工解冻）

5. [必须] `ConfirmDebit(ConfirmDebitReq) → ConfirmDebitResp` — **确认扣款**
   - 将冻结金额正式扣除 + 写流水（冻结终态），幂等键 = refOrderNo
   - 调用方：财务（三方出款成功后确认）

冻结三阶段关系：

```
FreezeBalance ──→ 审核通过 + 出款成功 ──→ ConfirmDebit （余额正式减少）
              └─→ 审核驳回 / 出款失败 ──→ UnfreezeBalance（余额恢复可用）
```

### 3.2 钱包管理（1个）

6. [必须] `InitWallet(InitWalletReq) → InitWalletResp` — **钱包初始化**
   - 为新用户创建全部币种的子钱包记录，幂等键 = userId
   - 调用方：ser-user（用户注册成功后）
   - 依赖：币种配置.GetCurrencyList（获取所有启用币种）

### 3.3 查询能力（6个，只读）

7. [必须] `GetBalance(GetBalanceReq) → GetBalanceResp` — **余额查询**
   - 查询指定用户 + 币种下各子钱包余额（中心/奖励/冻结）
   - 调用方：财务（人工操作前展示）、直播（余额校验）、风控

8. [必须] `GetCurrencyList(GetCurrencyListReq) → GetCurrencyListResp` — **可用币种列表**
   - 返回启用状态的币种配置列表（RPC版，供其他模块使用）
   - 调用方：财务、ser-user、前端网关

9. [必须] `GetExchangeRate(GetExchangeRateReq) → GetExchangeRateResp` — **获取汇率**
   - 返回指定币种对的入款/出款/实时汇率
   - 调用方：财务（报表换算）、充值/提现/兑换（锁定汇率）

10. [应有] `BatchGetBalance(BatchGetBalanceReq) → BatchGetBalanceResp` — **批量余额查询**
    - 批量查询多用户余额，B端报表/列表使用
    - 调用方：财务统计、运营报表

11. [应有] `GetBaseCurrency(GetBaseCurrencyReq) → GetBaseCurrencyResp` — **基准币种信息**
    - 返回平台基准币种代码和配置
    - 调用方：财务报表（统一换算基准）、活动（奖金计算）

12. [商榷] `GetWalletFlow(GetWalletFlowReq) → GetWalletFlowResp` — **资金流水查询**
    - 查询资金流水明细（RPC版，供风控/财务对账用）
    - 商榷：初期可直查DB，非核心链路必备接口

## 四、调用外部 RPC 接口（ser-wallet → 其他模块）

共 12 个：6 必须 + 3 应有 + 3 商榷

### 4.1 已有模块（RPC客户端已就绪，可直接调用）

#### ser-kyc — KYC认证查询（编号012）

1. [必须] `rpc.KycClient().KycQuery(KycQueryReq) → KycQueryResp`
   - 查询用户KYC认证状态，提现前硬阻塞（未认证仅可USDT提现）
   - 使用场景：获取提现方式列表(C-6)、创建提现订单(C-7)

#### ser-blog — 审计日志（编号006）

2. [必须] `rpc.BLogClient().AddActLog(ActAddReq) → oneway void`
   - B端所有写操作的审计日志（新增/编辑/启禁币种、设置基准币、刷新汇率等）
   - 异步 oneway 调用，不阻塞主流程

#### ser-i18n — 国际化翻译（编号004）

3. [商榷] `rpc.I18nClient().GetTrans(TransReq) → TransResp`
   - 错误码/枚举文案翻译
   - 商榷：大多数场景由网关层通用Handler自动完成，仅特殊场景需模块内主动调用

#### ser-user — 用户信息查询（编号008）

4. [应有] `rpc.UserClient().GetUserInfoExpend(GetUserInfoReq) → UserInfoExpendResp`
   - B端记录列表展示用户昵称/账号

5. [应有] `rpc.UserClient().BatchGetUserInfo(BatchGetUserInfoReq) → BatchUserInfoResp`
   - B端批量查询多用户信息

#### ser-buser — 后台用户信息（编号002）

6. [应有] `rpc.BUserClient().QueryUserByIds(RpcSysUserReq) → map<i64, string>`
   - B端展示操作人名称（币种创建人/修改人、汇率日志操作人等）

### 4.2 未建模块（需Mock，编码阶段先用桩接口替代）

#### ser-finance — 财务/支付（最大阻塞风险，模块不存在）

7. [必须] `GetRechargeMethodList` — **获取充值方式列表**
   - 按 currencyCode 返回可用充值方式（银行卡/电子钱包/USDT等）
   - 使用场景：充值方式列表(C-3)

8. [必须] `GetWithdrawMethodList` — **获取提现方式列表**
   - 按 currencyCode 返回可用提现方式
   - 使用场景：提现方式列表(C-6)

9. [必须] `MatchChannelAndCreateOrder` — **匹配通道 + 创建通道订单**
   - 充值时轮询匹配代收通道，获取三方支付URL
   - 使用场景：创建充值订单(C-4)

10. [必须] `GetWithdrawConfig` — **获取提现限额/手续费规则**
    - 提现金额校验、手续费计算
    - 使用场景：创建提现订单(C-7)

11. [商榷] `SubmitWithdrawToChannel` — **提交三方出款**
    - 提现审核通过后发起三方出款
    - 商榷：可能由财务模块自驱（监听订单状态变更），而非钱包主动调用

12. [商榷] `NotifyNewWithdrawOrder` — **新提现订单通知**
    - 通知财务模块有新提现订单待审核
    - 商榷：可改用 MQ/Kafka 消息队列替代 RPC 直调

## 八、关键风险点

1. **ser-finance 未建** — 充值/提现6个C端核心接口强依赖财务模块（4个P0 RPC），是最大阻塞风险
   - 应对：编码阶段用 Mock/桩接口替代，接口层预留对接口，联调时替换
   - 需协调：与财务负责人对齐 IDL 定义 + 联调时间节点

2. **充值回调幂等** — rechargeCallback 必须保证重复回调不重复上账
   - 应对：CreditWallet 以 refOrderNo 做幂等键，DB唯一索引保障

3. **冻结一致性** — FreezeBalance / UnfreezeBalance / ConfirmDebit 三阶段必须状态闭环
