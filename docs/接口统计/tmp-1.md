## 一、币种配置模块

### 1.1 B端 HTTP 接口（7个）

> 路由前缀：`/admin/api/wallet/currency`

| # | 方法 | 路由 | 说明 | 必要性 |
|---|------|------|------|--------|
| B1 | GET | `/admin/api/wallet/currency/list` | 币种配置分页列表 | [必须] |
| B2 | POST | `/admin/api/wallet/currency/create` | 新增币种 | [必须] |
| B3 | PUT | `/admin/api/wallet/currency/update` | 编辑币种配置 | [必须] |
| B4 | PUT | `/admin/api/wallet/currency/status` | 启用/禁用币种 | [必须] |
| B5 | PUT | `/admin/api/wallet/currency/base` | 设置基准币种（一次性） | [必须] |
| B6 | GET | `/admin/api/wallet/currency/rate/log` | 汇率变更日志分页查询 | [必须] |
| B7 | GET | `/admin/api/wallet/currency/detail` | 币种详情查询 | [应有] |

**逐条说明：**

- **B1 — 币种配置分页列表** [必须]
  - 需求原型明确存在此页面，含 20+ 字段列表（币种名称/代码/类型/实时汇率/平台汇率/偏差值/入款汇率/出款汇率/状态等）
  - 是币种管理的入口页面，11 份评估文档全部一致认为必须

- **B2 — 新增币种** [必须]
  - 后台新增币种配置，含 18 字段（名称/代码/类型/符号/小数位/千分位/图标上传 SVG/PNG≤5KB/汇率浮动比例/偏差阈值等）
  - 首次启用平台时必须配置币种，属于基础数据初始化的前置条件

- **B3 — 编辑币种配置** [必须]
  - 弹窗编辑已有币种的汇率浮动比例、偏差阈值、精度、图标等
  - 运营过程中必然需要调整汇率参数

- **B4 — 启用/禁用币种** [必须]
  - 控制币种在前端是否可见可用
  - 业务运营基础能力，不可缺

- **B5 — 设置基准币种** [必须]
  - 基准币种 = USDT，设置后不可修改
  - BSB 锚定比例（10 BSB = 1 USDT）依赖此设置
  - 一次性操作，但系统初始化绕不过

- **B6 — 汇率变更日志分页查询** [必须]
  - 需求原型明确存在「汇率日志」Tab 页
  - 记录每次汇率变更的时间、旧值、新值、触发方式（自动/手动）
  - 审计追溯的基础能力

- **B7 — 币种详情查询** [应有]
  - 查看单个币种的完整配置信息
  - 列表页已展示大部分字段，详情页是锦上添花
  - c-5 和 d-2 建议保留，c-4 未单独列出

### 1.2 C端 HTTP 接口（2个）

> 路由前缀：`/api/wallet/currency`

| # | 方法 | 路由 | 说明 | 必要性 |
|---|------|------|------|--------|
| C-CC1 | GET | `/api/wallet/currency/list` | 获取已启用币种列表 | [必须] |
| C-CC2 | GET | `/api/wallet/currency/rate` | 获取指定币种汇率 | [必须] |

**逐条说明：**

- **C-CC1 — 获取已启用币种列表** [必须]
  - 钱包页面币种切换下拉、充值/提现页面币种选择的数据源

- **C-CC2 — 获取指定币种汇率** [必须]
  - 兑换页面展示汇率、充值/提现金额换算
  - 返回：实时汇率、入款汇率、出款汇率、BSB 锚定比例
  - 前端金额展示和计算的基础依赖

### 1.3 RPC 暴露接口（3个）

> Thrift Service：CurrencyConfigService

| # | 方法 | 说明 | 调用方 | 必要性 |
|---|------|------|--------|--------|
| CC-R1 | GetCurrencyList | 获取可用币种列表 | ser-finance 等所有需要币种下拉的模块 | [必须] |
| CC-R2 | GetExchangeRate | 获取指定币种汇率（实时/平台/入款/出款） | ser-finance 报表换算、结算模块 | [必须] |
| CC-R3 | GetBaseCurrency | 获取基准币种信息 | ser-finance 统计报表、结算基准 | [应有] |

**逐条说明：**

- **CC-R1 — GetCurrencyList** [必须]
  - 所有需要币种信息的模块都依赖此接口
  - 返回：CurrencyInfo 列表（代码/名称/类型/符号/精度/千分位/图标/排序/状态）
  - RPC 暴露分析 tmp-1 确认为 P0

- **CC-R2 — GetExchangeRate** [必须]
  - 财务报表需要将各币种金额统一换算为基准币种
  - 返回：实时汇率、平台汇率、入款汇率、出款汇率、更新时间
  - 跨模块联调的核心依赖

- **CC-R3 — GetBaseCurrency** [应有]
  - 返回基准币种代码（USDT）、名称、精度、BSB 锚定比例
  - 财务统计确定换算基准时使用
  - 使用频率低但信息不可缺，RPC 暴露分析定为 P2

### 1.4 RPC 调用接口（3个）

> ser-wallet（币种配置模块）作为客户端调用外部

| # | 目标模块 | 方法 | 说明 | 必要性 |
|---|---------|------|------|--------|
| CC-D1 | ser-blog | AddActLog (oneway) | B端操作审计日志 | [必须] |
| CC-D2 | ser-s3 | UploadFile | 币种图标上传 | [应有] |
| CC-D3 | ser-buser | QueryUserByIds | B端列表显示操作人名称 | [商榷] |

**逐条说明：**

- **CC-D1 — ser-blog.AddActLog** [必须]
  - 所有 B 端写操作（新增/编辑/启禁用/上传图标/刷新汇率）均需记录审计日志
  - oneway 异步不阻塞主流程，失败仅 warn 不影响业务
  - 工程规范一致要求

- **CC-D2 — ser-s3.UploadFile** [应有]
  - 需求原型明确币种编辑弹窗可上传图标（SVG/PNG≤5KB）
  - 极低频操作，但需求有明确要求
  - 可降级：上传失败不阻塞币种保存

- **CC-D3 — ser-buser.QueryUserByIds** [商榷]
  - 列表页展示「最后操作人」列时需要
  - ser-blog 已返回 createName，可能无需额外查询
  - 多份评估意见不一致：c-4 列入 P1，RPC 调用分析定为 P2（弱依赖可降级）

### 1.5 定时任务（1个）

| # | 任务 | 说明 | 必要性 |
|---|------|------|--------|
| T1 | 汇率定时刷新 | 轮询 3 家三方 API 取均值，偏差超阈值自动更新平台汇率 | [必须] |

**说明：**
- 需求明确要求实时汇率定时自动更新

- 入款/出款汇率任一超阈值即触发更新
- 列表页有实时汇率倒计时展示，依赖此任务

### 2.2 B端 HTTP 接口（0个）

> **重要说明**：多份评估经讨论后达成共识 —— 钱包模块不直接暴露 B 端 HTTP 接口。
>
> 所有 B 端运营操作（查看用户余额、查看交易流水、人工加款/减款/补单）均由 **ser-finance** 模块的 B 端页面发起，通过 **RPC 调用 ser-wallet** 完成。
>
> 原因：
> 1. 需求文档中「人工加款/减款/补单」明确在财务管理后台操作
> 2. 钱包是资金引擎，不直接面向 B 端运营人员
> 3. 避免 B 端接口在两个模块重复建设
>
> **商榷项**：c-2/c-5/d-2/d-3 提到可能需要 2-3 个 B 端查询接口（用户钱包余额查询、用户交易流水查询），供运营后台独立查看。若评审确认需要，可补充为 [应有]，但当前版本不纳入。

### 2.3 RPC 暴露接口（9个）

> Thrift Service：WalletService

#### 2.3.1 核心写操作（5个）

| # | 方法 | 说明 | 主要调用方 | 必要性 |
|---|------|------|-----------|--------|
| W-R1 | CreditWallet | 加款/上分（充值入账、奖金发放、人工加款、补单、兑换入、返奖） | ser-finance、活动系统、结算 | [必须] |
| W-R2 | DebitWallet | 扣款/下分（人工减款、兑换出、消费扣款） | ser-finance、投注、商城 | [必须] |
| W-R3 | FreezeBalance | 冻结余额（提现冻结、风控冻结） | ser-wallet 内部、ser-finance | [必须] |
| W-R4 | UnfreezeBalance | 解冻余额（审核驳回、出款失败） | ser-finance | [必须] |
| W-R5 | ConfirmDebit | 确认扣款冻结金额（出款成功） | ser-finance | [必须] |

**核心设计要点：**

- **CreditWallet (W-R1)**：统一加款接口，通过 bizType 区分业务场景（充值=1/充值奖金=2/补单=3/人工加款=4/活动奖金=5/兑换入=6/兑换赠送=7/主播打赏=8/投注返奖=9）
  - 幂等键：(refOrderNo, bizType) 唯一索引
  - 必传字段：userId, currencyCode, walletType, amount, bizType, refOrderNo

- **DebitWallet (W-R2)**：统一扣款接口，同样通过 bizType 区分
  - 防透支：SQL 保证 `available_balance - frozen_amount >= amount`
  - 消费优先级：中心钱包优先 → 奖励钱包补足

- **FreezeBalance (W-R3)**：冻结 ≠ 扣款，总余额不变，冻结额增加，可用额减少
  - 冻结优先级：中心钱包优先 → 奖励钱包
  - 需记录冻结明细，供后续解冻/确认扣款关联

- **UnfreezeBalance (W-R4)** 与 **ConfirmDebit (W-R5)**：互斥关系
  - 同一笔冻结只能走其中一条路径：解冻退回 OR 确认扣款
  - 通过 refOrderNo 关联原始冻结记录

- **冻结三阶段状态转换**：
  ```
  用户提现 → FreezeBalance(冻结)
                  │
          ┌───────┴───────┐
          │               │
     审核通过+出款成功   审核驳回/出款失败
          │               │
     ConfirmDebit      UnfreezeBalance
     (确认扣款)        (解冻退回)
  ```

- **安全约束（5 个写操作共性）**：
  - 幂等性：(refOrderNo, bizType) 唯一索引防重
  - 原子性：余额变更 + 流水写入同一 DB 事务 + SELECT FOR UPDATE
  - 非负约束：SQL WHERE 条件保证不透支
  - 并发控制：SELECT FOR UPDATE 行级锁（同用户同币种串行）
  - 可审计：wallet_transaction 不可变 + ser-blog 审计日志

#### 2.3.2 管理操作（1个）

| # | 方法 | 说明 | 主要调用方 | 必要性 |
|---|------|------|-----------|--------|
| W-R6 | InitWallet | 用户注册时初始化多币种钱包 | ser-user | [应有] |

- **InitWallet (W-R6)**：
  - 查询所有启用币种 → 为每个币种创建 N 个子钱包（中心/奖励/...）→ 设置默认币种
  - 幂等：已有钱包则返回成功不重复创建
  - 触发时机：用户注册成功后 ser-user 调用

#### 2.3.3 查询操作（3个）

| # | 方法 | 说明 | 主要调用方 | 必要性 |
|---|------|------|-----------|--------|
| W-R7 | GetBalance | 查询单用户余额（各子钱包） | ser-finance（余额展示/校验）、投注、直播 | [必须] |
| W-R8 | BatchGetBalance | 批量查询多用户余额 | ser-finance（B端列表批量展示） | [商榷] |
| W-R9 | GetWalletFlow | 查询交易流水分页 | ser-finance（B端交易详情/对账） | [商榷] |

- **GetBalance (W-R7)** [必须]：
  - 最高频 RPC 查询，ser-finance 在人工加款/减款弹窗中需展示当前余额
  - 投注时需校验余额是否充足
  - 返回：各子钱包的余额、冻结额、可用额

- **BatchGetBalance (W-R8)** [商榷]：
  - B 端列表页批量展示用户余额，避免 N+1 查询
  - 工程规范建议有（参考 ser-user.BatchGetUserInfo 模式）
  - 但首版 B 端操作由 ser-finance 负责，钱包不直接面向 B 端，是否首版实现待定
  - c-1/c-2 列入应有，c-4/RPC 暴露分析定为 P2

- **GetWalletFlow (W-R9)** [商榷]：
  - B 端查看用户交易明细、对账核验时使用
  - 同样是 ser-finance B 端页面发起调用
  - 首版是否需要取决于 ser-finance 的进度和需求
  - RPC 暴露分析定为 P2

### 2.4 RPC 调用接口（4个）

> ser-wallet（钱包模块）作为客户端调用外部

| # | 目标模块 | 方法 | 说明 | 必要性 |
|---|---------|------|------|--------|
| W-D1 | ser-user | GetUserInfoExpend | 查询用户信息（状态/KYC/国家） | [必须] |
| W-D2 | ser-kyc | KycQuery | 查询 KYC 认证状态及实名信息 | [必须] |
| W-D3 | ser-blog | AddActLog (oneway) | 所有写操作的审计日志 | [必须] |
| W-D4 | ser-user | BatchGetUserInfo | B端列表批量查用户信息 | [应有] |

**逐条说明：**

- **W-D1 — ser-user.GetUserInfoExpend** [必须]
  - 提现页面校验用户状态（封号禁止提现）、账户类型（游客限制功能）
  - 充值时根据 countryCode 推断默认币种
  - 强依赖，P0

- **W-D2 — ser-kyc.KycQuery** [必须]
  - 提现流程核心依赖：KYC status=3(已通过) → 全渠道；status≠3 → 仅 USDT
  - 持卡人姓名比对：kycResp.name vs 提现账户持卡人（大小写不敏感、trim）
  - 强依赖，P0

- **W-D3 — ser-blog.AddActLog** [必须]
  - oneway 异步调用，不阻塞主流程
  - 所有金融写操作（加款/扣款/冻结/解冻/确认/初始化）提交事务后异步发送
  - 失败仅 warn 日志，不影响业务
  - 工程规范一致要求

- **W-D4 — ser-user.BatchGetUserInfo** [应有]
  - B 端记录列表批量展示用户昵称/UID
  - 单次限 100 用户，结果按 userId map 化 O(1) 查找
  - 非核心业务链路，可降级
