## 分类标识说明

```
●  必须（Must）    — 产品流程必经之路，绕不开的核心接口
○  应该有（Should） — 体验完整性或运营管理所需，非最小可行路径但缺少影响体验
△  需商榷（Debatable）— 文档间存在分歧、归属争议或可延后决策
```

---

## 路由约定说明

```
C端（gate-font）：POST /api/{domain}/{action}     需登录态（FontAuthMiddleware）
B端（gate-back）：POST /admin/api/{domain}/{action} 需登录态 + URL权限校验（BackAuthMiddleware + CheckUrlAuth）
开放接口：       POST /open/{domain}/{action}      无需登录
RPC（Kitex）：   Thrift 协议，ETCD 服务发现
```

- 全站统一 POST 方法（工程既有约定）
- 路由前缀按业务域划分（wallet / deposit / exchange / withdraw / record / currency）
- 同一 ser-wallet 服务，不同业务域使用不同路由前缀（与 ser-app 的 banner/popup/notice 模式一致）

## 一、C端 HTTP 接口（用户端 · gate-font）

> 消费方：C端用户（H5 / App 前端）
> 17 个必须 + 3 个应该有 + 3 个需商榷 = 23 个

---

### 1.1 钱包核心（2 个）

**C-01 ● `POST /api/wallet/balance`** — 查询钱包余额

    当前选中币种下的钱包余额。返回中心钱包（center）和奖励钱包（reward）
    各自的可用余额与冻结余额，前端加总显示可用金额。
    入参：无（当前币种从用户会话获取）
    出参：centerAvailable, centerFrozen, rewardAvailable, rewardFrozen, currencyCode
    调用链：纯内部查询，无外部依赖

**C-02 ● `POST /api/wallet/overview`** — 多币种余额总览

    返回用户所有已激活币种及其余额概览，供币种切换下拉/面板使用。
    标记当前选中币种。每个币种返回汇总可用余额。
    入参：无
    出参：List<CurrencyBalance>{currencyCode, name, icon, totalAvailable, isSelected}
    调用链：纯内部查询，无外部依赖

---

### 1.2 充值（5 个）

**C-03 ● `POST /api/deposit/methods`** — 获取充值方式列表

    当前币种可用的充值方式。聚合两个数据源：
    (1) 调用 ser-finance.GetPaymentMethods 获取支付通道（名称、图标、限额、排序）
    (2) 内部币种配置获取金额档位列表 + 当前汇率
    入参：currencyCode
    出参：List<Method>{name, icon, minAmount, maxAmount, sortOrder, recommended}
          + amountTiers[] + exchangeRate
    调用链：ser-finance.GetPaymentMethods（同步RPC）

**C-04 ● `POST /api/deposit/create`** — 创建充值订单

    统一充值入口，内部根据支付方式路由到不同处理逻辑（银行/QR/USDT等）。
    核心流程：
    (1) 校验金额 >= 最低档位
    (2) 重复充值检测（同用户+同币种+同金额 pending >= 3 拦截，USDT链上豁免）
    (3) 锁定当前汇率快照（法币→BSB 场景）
    (4) 调用 ser-finance.MatchDepositChannel 匹配最优支付通道
    (5) 创建"待支付"状态订单，返回支付跳转信息
    入参：currencyCode, amount, methodType, bonusActivityId(optional)
    出参：orderId, payUrl/payInfo, expireTime
    调用链：ser-finance.MatchDepositChannel（同步RPC · 关键路径，失败=创建失败）

**C-05 ● `POST /api/deposit/usdt`** — 获取 USDT 充值地址

    USDT 专用流程。为用户分配或获取收款钱包地址。
    返回地址字符串 + 二维码数据 + 网络类型（TRC20/ERC20/BEP20）。
    地址策略：一用户一地址（热钱包），用于链上交易归属识别。
    入参：networkType(TRC20/ERC20/BEP20)
    出参：address, qrCodeData, networkType, minAmount
    调用链：视实现方案，可能调用外部地址分配服务

**C-06 ○ `POST /api/deposit/status`** — 查询充值订单状态

    前端轮询接口。从第三方支付页返回后，前端定时查询订单支付状态。
    状态枚举：待支付 / 处理中 / 成功 / 失败 / 超时。
    入参：orderId
    出参：status, amount, actualReceived(if success)
    说明：部分方案将此标为必须，但实际可通过 WebSocket 推送替代轮询。
          保守起见标为应该有。

**C-07 ○ `POST /api/deposit/bonus`** — 获取充值优惠活动

    当前币种可参与的充值优惠活动列表。
    每个活动包含：最低充值门槛、加赠比例、流水要求倍数。
    每笔充值最多关联一个优惠活动（互斥）。
    入参：currencyCode, amount(optional, for filtering)
    出参：List<Bonus>{activityId, name, minDeposit, bonusPercent, turnoverMultiple}
    注意：11 份方案中 d-1/d-3/tmp-3 标为必须，c-1/c-2/tmp-1/tmp-2 标为应该有。
          若优惠活动由独立活动模块管理，此接口可能仅做代理透传。
          最终归属取决于活动模块边界，暂标应该有。

### 1.3 兑换（2 个）

**C-08 ● `POST /api/exchange/preview`** — 兑换预览

    纯计算接口，不写库。输入法币金额，返回兑换结果预览。
    仅 VND/IDR/THB 等法币钱包有兑换入口，BSB 钱包无此功能。
    入参：fromCurrency, amount
    出参：exchangeRate, bsbAmount, bonusPercent, bonusAmount, actualReceived, rateExpireTime
    调用链：纯内部计算（读取币种配置中的汇率 + 加赠比例）

**C-09 ● `POST /api/exchange/execute`** — 执行兑换

    原子事务操作：
    (1) 扣减源法币中心钱包（available -= amount）
    (2) 增加 BSB 中心钱包（主体金额）
    (3) 增加 BSB 奖励钱包（加赠金额，附带流水要求）
    执行时重新获取并锁定汇率（防止预览后汇率变动）。
    入参：fromCurrency, amount
    出参：orderId, deductedAmount, receivedBsb, bonusBsb, appliedRate
    调用链：纯内部事务（balance_engine），零外部依赖。可最早独立开发测试。

---

### 1.4 提现（6 个）

**C-10 ● `POST /api/withdraw/methods`** — 获取提现方式列表

    当前币种可用的提现方式。根据用户 KYC 状态过滤：
    - 未认证：仅显示 USDT 通道
    - 已认证：显示全部通道（银行卡/电子钱包/USDT）
    入参：currencyCode
    出参：List<Method>{name, type, icon, minAmount, maxAmount, feeRule, arrivalTime}
    调用链：ser-kyc.KycQuery + ser-finance.GetPaymentMethods（两个同步RPC）

**C-11 ● `POST /api/withdraw/create`** — 创建提现订单

    核心提现流程：
    (1) 校验金额：单笔限额 + 当日累计限额
    (2) 冻结余额：available -= amount, frozen += amount（原子操作）
    (3) 创建"待审核"状态订单
    (4) 推送至 ser-finance 进入三级审核流程（风控→财务→出款）
    入参：currencyCode, amount, methodType, withdrawAccountId, tradePwd(optional)
    出参：orderId, frozenAmount, estimatedArrival
    调用链：ser-kyc.KycQuery（校验）+ 内部 balance_engine（冻结）
    说明：钱包只负责冻结和创建订单，后续审核/出款/回调均由 ser-finance 驱动

**C-12 ● `POST /api/withdraw/limit`** — 查询提现限额

    展示当前币种的提现规则，供提现页面展示。
    入参：currencyCode
    出参：perTxMin, perTxMax, dailyLimit, dailyUsed, dailyRemaining,
          feeType, feeRate, feeFixed, arrivalDays
    调用链：纯内部查询（读取币种配置中的提现限额配置）

**C-13 ● `POST /api/withdraw-account/list`** — 提现账户列表

    用户已绑定的提现账户。可能包含多种类型：
    - 银行卡：银行名 + 卡号尾4位 + 户名
    - 电子钱包：钱包类型 + 手机号 + 户名
    - USDT 地址：网络类型 + 地址（脱敏）
    入参：无
    出参：List<Account>{id, type, displayName, maskedInfo, isDefault}
    调用链：纯内部查询

**C-14 ● `POST /api/withdraw-account/save`** — 绑定/保存提现账户

    绑定新的提现账户。核心规则：
    - 调用 ser-kyc.KycQuery 获取实名姓名，自动填充"户名"字段（只读，不可修改）
    - 支持类型：银行卡（bankName + accountNo）/ 电子钱包（walletType + phone）/ USDT（network + address）
    入参：type, bankName, accountNo, walletType, phone, usdtNetwork, usdtAddress
    出参：accountId
    调用链：ser-kyc.KycQuery（同步RPC，获取实名信息）

**C-15 ○ `POST /api/withdraw/check`** — 提现资格预检

    提交提现前的前置检查，返回用户当前提现资格信息：
    KYC 状态 + 可用余额 + 剩余日限额 + 是否满足提现条件。
    入参：currencyCode
    出参：kycStatus, availableBalance, dailyRemaining, eligible, reason
    调用链：ser-kyc.KycQuery + 内部余额查询 + 内部限额配置
    说明：多数方案将此逻辑合并到 C-11 CreateWithdrawOrder 的前置校验中。
          独立接口的价值在于前端可提前展示资格信息，优化用户体验。

### 1.5 账单记录（5 个）

> 核心规则：当前选中币种 = 记录币种，切换币种刷新所有标签页，无跨币种混合记录

**C-16 ● `POST /api/record/deposit/page`** — 充值记录分页

    当前币种的充值历史列表。
    字段：订单号、金额、充值方式、状态、创建时间。
    入参：currencyCode, pageNo, pageSize, startTime(opt), endTime(opt)
    出参：pageList[], total, pageNo, pageSize
    调用链：纯内部查询

**C-17 ● `POST /api/record/withdraw/page`** — 提现记录分页

    当前币种的提现历史列表。含审核进度状态。
    字段：订单号、金额、提现方式、状态（待审核/审核中/打款中/成功/失败/已取消）、时间。
    入参：同 C-16
    出参：同 C-16

**C-18 ● `POST /api/record/exchange/page`** — 兑换记录分页

    当前币种的兑换历史列表。
    字段：兑换金额、加赠比例、实得 BSB、适用汇率、时间。
    入参：同 C-16
    出参：同 C-16

**C-19 ● `POST /api/record/reward/page`** — 奖励记录分页

    当前币种的奖励历史列表。
    字段：奖励金额、类别（充值加赠/活动奖励/返水/邀请奖励等）、时间、流水要求。
    入参：同 C-16
    出参：同 C-16

**C-20 ● `POST /api/record/detail`** — 记录详情

    单条记录的完整详情。传入记录 ID + 记录类型（deposit/withdraw/exchange/reward）。
    字段：订单号、金额、实到/实扣、方式、状态、时间线、失败原因（如有）。
    入参：recordId, recordType
    出参：完整订单详情（字段因类型而异）
    调用链：纯内部查询

### 1.6 C端可选/备选接口（3 个）

**C-21 △ `POST /api/deposit/cancel`** — 取消充值订单

    取消未支付的充值订单。
    分歧：仅 c-2/tmp-2 标为可选，多数方案未提及。
    争议点：是否需要用户主动取消？订单超时机制（T-02）可自动处理过期订单。
    若支持取消，需同步通知 ser-finance 释放通道资源。

**C-22 △ `POST /api/withdraw/cancel`** — 取消提现订单

    用户主动取消待审核的提现订单。取消后需解冻余额。
    分歧：仅 c-2/d-3 提及。
    争议点：一旦进入审核流程，是否允许用户取消？
    若允许，需与 ser-finance 审核状态同步，存在并发竞态风险。
    建议：产品评审时明确决策。

**C-23 △ `POST /api/withdraw-account/delete`** — 删除提现账户

    删除已绑定的提现账户。仅 d-3 提及。
    争议点：删除后是否可重新绑定不同户名的账户？若可以，存在安全风险。
    若户名锁定（只能绑定 KYC 实名），则删除后重新绑定不改变户名，风险可控。
    建议：视安全策略决定是否开放。

## 二、B端 HTTP 接口（管理端 · gate-back）

> 消费方：B端管理员（后台管理系统）
> 7 个必须 + 7 个应该有 = 14 个

---

### 2.1 币种配置管理（6 个）

**B-01 ● `POST /admin/api/currency/page`** — 币种列表分页

    分页查询币种配置。
    筛选项：币种代码、类型（法币/虚拟币）、状态（启用/禁用）。
    展示字段：代码、名称、类型、图标、国家、符号、千分位规则、小数精度、
              实时汇率、平台汇率、浮动比例、充值汇率、提现汇率、偏差阈值、
              排序权重、最后操作人、最后更新时间。
    入参：pageNo, pageSize, currencyCode(opt), type(opt), status(opt)
    出参：pageList[], total, pageNo, pageSize

**B-02 ● `POST /admin/api/currency/create`** — 创建币种

    新增币种配置。默认状态 = 禁用。
    图标通过 ser-s3.UploadFile 上传（SVG/PNG，<= 5KB）。
    入参：code, name, type, country, symbol, precision, thousandSep, icon,
          rateFloat, rateThreshold, sortOrder
    出参：currencyId
    调用链：ser-s3.UploadFile（图标上传）+ ser-blog.AddActLog（审计日志）
    注意：c-1 曾主张币种由开发直接入库无需接口，但 c-2/c-3/c-4/d-1/d-2/d-3/tmp-1
          7 份方案均认为需要 B 端接口。裁定：保留接口，初期可限制高级权限。

**B-03 ● `POST /admin/api/currency/update`** — 编辑币种

    编辑币种属性。可修改项：汇率浮动比例、偏差阈值、图标、排序权重、
    千分位规则、小数精度等。
    不可修改项：币种代码（code）、名称（name）。
    入参：currencyId, rateFloat, rateThreshold, icon, sortOrder, ...
    出参：空（成功即可）
    调用链：ser-s3.UploadFile（如更换图标）+ ser-blog.AddActLog

**B-04 ● `POST /admin/api/currency/status`** — 启用/禁用币种

    切换币种启用状态。独立于编辑接口（工程既有约定：Status 操作不合并到 Update）。
    禁用后 C 端不可使用该币种充值/提现。
    若用户名下该币种有余额，前端应展示提示信息。
    入参：currencyId, status(enable/disable)
    出参：空
    调用链：ser-blog.AddActLog

**B-05 ● `POST /admin/api/currency/detail`** — 币种详情

    单个币种的完整配置信息，用于编辑表单预填。
    入参：currencyId
    出参：完整币种配置字段

**B-06 ● `POST /admin/api/currency/base/set`** — 设置基准币种

    设置平台基准币种（推荐 USDT）。
    关键约束：一旦设置不可更改。前端需实现二次确认弹窗。
    BSB 锚定比例：10 BSB = 1 USDT（配置化，但初始值固定）。
    入参：currencyCode
    出参：空
    调用链：ser-blog.AddActLog
    安全性：需最高管理权限

---

### 2.2 汇率管理（3 个）

**B-07 ● `POST /admin/api/rate/log/page`** — 汇率变更日志分页

    汇率变更记录查询。每次平台汇率更新（自动或手动）生成一条日志。
    字段：日志ID、更新时间、币种代码、更新前汇率、更新后汇率、
          实时汇率、平台汇率、充值汇率、提现汇率、触发方式（自动/手动）。
    入参：pageNo, pageSize, currencyCode(opt), startTime(opt), endTime(opt)
    出参：pageList[], total, pageNo, pageSize

**B-08 ○ `POST /admin/api/rate/refresh`** — 手动刷新汇率

    手动触发第三方 API 汇率刷新。
    机制：拉取 3 个第三方数据源 → 取平均值 → 对比偏差 → 超阈值则更新平台汇率。
    日常由定时任务（T-01）自动执行，此接口用于运营人员即时刷新。
    入参：currencyCode(opt, 空=全部刷新)
    出参：List<RateResult>{currencyCode, oldRate, newRate, updated}
    调用链：第三方汇率 API（HTTP）+ ser-blog.AddActLog

**B-09 ○ `POST /admin/api/rate/log/export`** — 汇率日志导出

    汇率变更日志 CSV 导出。复用 B-07 的查询条件。
    入参：同 B-07（去掉分页参数）
    出参：binary（CSV 文件流）

---

### 2.3 操作日志（1 个）

**B-10 ○ `POST /admin/api/currency/op-log/page`** — 币种操作日志分页

    币种配置变更的操作日志查询。
    记录：谁（operatorId/name）在什么时间对哪个币种做了什么修改。
    数据来源：ser-blog 审计日志（按 model=currency 筛选）。
    入参：pageNo, pageSize, currencyCode(opt), operatorName(opt), startTime(opt), endTime(opt)
    出参：pageList[], total
    说明：若 ser-blog 已提供通用操作日志查询接口，此接口可仅做参数封装代理。

---

### 2.4 用户钱包管理（4 个）

**B-11 ○ `POST /admin/api/wallet/user/page`** — 用户钱包分页查询

    管理员/客服查询用户钱包余额。
    筛选：用户ID、币种、钱包类型（中心/奖励/主播/代理）。
    入参：pageNo, pageSize, userId(opt), currencyCode(opt), walletType(opt)
    出参：pageList[]{userId, currencyCode, walletType, available, frozen, updateTime}, total
    分歧：tmp-3 标为必须，c-1/c-4/tmp-1 认为无需（由 ser-finance B 端处理）。
    裁定：应该有。客服处理用户投诉时需要直接查看钱包状态，不应绕道 ser-finance。

**B-12 ○ `POST /admin/api/wallet/user/detail`** — 用户钱包详情

    单个用户的完整钱包树：每个币种 x 4 钱包类型（中心/奖励/主播/代理），
    每个钱包显示可用余额 + 冻结余额。
    入参：userId
    出参：List<CurrencyWallet>{currencyCode, wallets[]{type, available, frozen}}

**B-13 ○ `POST /admin/api/wallet/user/transaction/page`** — 用户交易流水

    管理员查询指定用户的交易流水。
    筛选：用户ID、币种、交易类型（充值/提现/兑换/冻结/解冻/手动调账）、时间范围。
    入参：userId(required), pageNo, pageSize, currencyCode(opt), txType(opt),
          startTime(opt), endTime(opt)
    出参：pageList[]{txId, type, amount, balanceBefore, balanceAfter, refOrderNo, time}, total

    出参：binary（CSV 文件流）

## 三、RPC 暴露接口（供其他模块调用）

> 协议：Kitex Thrift RPC
> IDL 文件：common/idl/ser-wallet/wallet_rpc.thrift
> 主要消费方：ser-finance

> 6 个必须 + 3 个应该有 = 9 个

### 3.1 写操作（5 个）— 全部经由 balance_engine 统一事务入口

**R-01 ● `CreditBalance`** — 入账

    触发场景：充值到账确认 / 手动加款审核通过 / 充值补单审核通过
    调用方：ser-finance
    余额变动：available_balance += amount
    副作用：写 wallet_transaction 流水日志（同事务）
    入参：userId, currencyCode, walletType, amount, txType, refOrderNo, remark
    幂等：refOrderNo + txType + walletType 唯一索引，重复调用直接返回成功

**R-02 ● `DebitBalance`** — 扣款

    触发场景：提现打款成功 / 手动减款审核通过
    调用方：ser-finance
    余额变动：frozen_balance -= amount（从冻结扣减，非可用余额）
    入参：同 R-01
    说明：提现扣款是扣冻结金额（已在 C-11 CreateWithdrawOrder 时冻结），
          不是扣可用余额

**R-03 ● `FreezeBalance`** — 冻结

    触发场景：用户提交提现时内部调用（也暴露供风控模块冻结）
    调用方：ser-wallet 内部 + ser-finance（风控）
    余额变动：available_balance -= amount, frozen_balance += amount
    入参：同 R-01
    原子性：两个字段在同一 DB 事务内更新，使用 SELECT ... FOR UPDATE 行锁

**R-04 ● `UnfreezeBalance`** — 解冻

    触发场景：提现审核拒绝 / 提现打款失败 / 提现超时
    调用方：ser-finance
    余额变动：frozen_balance -= amount, available_balance += amount（资金返还用户）
    入参：同 R-01

**R-05 ● `ManualAdjust`** — 手动调账

    触发场景：手动调账审核通过后执行
    调用方：ser-finance
    特殊性：支持单次请求对多个钱包类型执行加/减操作，在单个 DB 事务内完成
    入参：userId, currencyCode, refOrderNo, adjustments[]{walletType, direction(credit/debit), amount}
    幂等：refOrderNo 全局唯一

### 3.2 读操作（4 个）

**R-06 ● `GetBalance`** — 查询单用户余额

    出参：available, frozen

**R-07 ○ `BatchGetBalance`** — 批量查询用户余额

    入参：List<Query>{userId, currencyCode, walletType}
    出参：List<Balance>{userId, currencyCode, walletType, available, frozen}

**R-08 ○ `GetActiveCurrencyList`** — 获取已启用币种列表

**R-09 ○ `GetExchangeRate`** — 查询汇率

## 五、定时任务

> 1 个必须 + 1 个应该有 + 1 个需商榷 = 3 个

---

**T-01 ● `SyncRealtimeRate`** — 实时汇率同步

    执行周期：可配置（建议 5-15 分钟）
    机制：
    (1) 拉取 3 个第三方汇率 API
    (2) 取平均值作为实时汇率
    (3) 计算与当前平台汇率的偏差
    (4) 偏差 >= 阈值时更新平台汇率（同时更新充值/提现汇率）
    (5) 生成汇率变更日志（写入 rate_change_log 表）
    数据源：3 个独立第三方 API，互为备份
    容错：单个数据源失败不影响，至少 1 个成功即可计算

**T-02 ○ `DepositOrderTimeout`** — 充值订单超时关闭

    执行周期：建议 1-2 分钟
    机制：扫描"待支付"状态超过有效期（如 30 分钟）的充值订单，自动关闭。
    说明：若 C-21 取消接口不实现，此任务为订单生命周期管理的唯一出口。

**T-03 △ `BalanceReconciliation`** — 余额对账检查

    执行周期：建议每日凌晨
    机制：对比 wallet_balance 表余额与 wallet_transaction 流水汇总是否一致。
    不一致时告警（不自动修复）。
    说明：运营阶段的健壮性工具，初期可不实现。
    11 份方案中仅 c-1（可选）和 tmp-2（应该有）提及，标为商榷。

## 七、关键裁定说明

> 11 份接口评估方案存在若干分歧，以下为整合时的裁定及理由

---

### 裁定 1：充值订单创建 — 统一入口 vs 按类型拆分

- 分歧：c-3 拆分为 3 个独立接口（CreateUSDTDepositOrder / CreateBankDepositOrder / CreateQRDepositOrder），其余 10 份方案使用统一 CreateDepositOrder
- 裁定：**统一入口**（C-04）
- 理由：前端只需对接一个接口，methodType 参数区分内部路由。符合"表面简单，内部分流"原则。减少前端对接成本和路由重复代码。

### 裁定 2：账单记录 — 统一接口 vs 按类型拆分

- 分歧：c-3 拆分为 4 个列表接口（W23/W24/W25/W26），其余方案使用统一记录接口
- 裁定：**按类型拆分**（C-16~C-19 四个列表 + C-20 一个通用详情）
- 理由：4 个 tab 的返回字段差异较大（充值有支付方式、提现有审核状态、兑换有汇率、奖励有流水要求），统一接口会导致大量 nullable 字段。拆分后每个接口的 Response 结构清晰，前端类型安全。与 ser-app 的 banner/popup/notice 分接口模式一致。

### 裁定 3：提现账户 — 通用接口 vs 按账户类型拆分

- 分歧：c-3 拆分为 6 个接口（银行卡 get/save + 电子钱包 get/save + USDT get/save），其余方案使用通用 list + save
- 裁定：**通用接口**（C-13 list + C-14 save）
- 理由：3 种账户类型共享"列表+绑定"逻辑，差异仅在字段组合。通用接口 + 类型字段判断，避免 6 个接口的维护成本。前端一个组件动态渲染不同表单即可。

### 裁定 4：CreateCurrency 是否需要 B 端接口

- 分歧：c-1 认为币种应由开发直接入库无需接口，7 份方案认为需要接口
- 裁定：**保留接口**（B-02）
- 理由：运营团队需要自主管理币种生命周期，不应每次新增币种都依赖开发介入。初期可通过权限控制限制使用范围。

### 裁定 5：重复充值检测 — 独立接口 vs 合并到创建

- 分歧：c-2/d-1/d-2/tmp-2/tmp-3 设为独立接口 CheckDuplicateDeposit，c-3/c-4 合并到创建流程
- 裁定：**合并到 C-04 CreateDepositOrder 内部逻辑**
- 理由：重复检测是创建前的校验步骤，不需要前端单独调用。作为 CreateDepositOrder 的前置校验，命中规则时返回明确错误码，前端展示 warning/block 提示即可。

### 裁定 6：钱包 B 端管理接口 — 是否需要

- 分歧：tmp-3 标为必须（7-9 个），c-1/c-4/tmp-1 认为无需（0 个），其余标为应该有
- 裁定：**应该有**（B-11~B-14 共 4 个）
- 理由：客服处理用户投诉时需要直接查看用户钱包状态和交易流水，不应绕道 ser-finance。但非首批必须，可在核心流程完成后补充。

### 裁定 7：币种配置 C 端接口 — 是否独立暴露

- 分歧：c-3/tmp-1 给币种配置独立 C 端接口（2-3 个），其余方案认为 C 端通过钱包模块聚合获取币种信息
- 裁定：**不独立暴露**
- 理由：C 端用户感知的是"钱包"而非"币种配置"。币种列表通过 C-02 GetWalletOverview 聚合返回，汇率通过 C-08 ExchangePreview 返回。无需让前端直接调用币种配置接口，减少接口暴露面。

### 裁定 8：提现资格预检 — 独立接口 vs 合并到创建

- 分歧：c-2/d-1 设为独立核心接口 CheckWithdrawEligibility，多数方案合并到 CreateWithdrawOrder
- 裁定：**保留独立接口但标为应该有**（C-15）
- 理由：独立预检接口允许前端在用户点击"提现"按钮前即展示资格信息（KYC 状态、可用余额、剩余限额），提升用户体验。但其逻辑在 CreateWithdrawOrder 中会重复执行，因此非必须。

## 八、边界模糊区域 · 归属待定

> 以下接口在 IDL 设计文档中被归入 ser-wallet，但在 rpc暴露/rpc调用 分析中被归入 ser-finance
> 最终归属取决于跨团队评审结论，此处记录以供对齐

---

### 8.1 充值订单 B 端管理（归 ser-finance 的可能性高）

```
- POST /admin/api/deposit/page       — 充值订单列表（B端查看所有充值记录）
- POST /admin/api/deposit/detail     — 充值订单详情
- POST /admin/api/deposit/callback   — 充值回调处理（第三方支付回调）
```

归属分析：充值订单由 C 端在钱包侧创建，但后续状态流转（通道回调→入账）由 ser-finance 驱动。
B 端查看充值记录更接近"财务管理"职责。
当前结论：倾向归 ser-finance。钱包侧通过 C-16 提供 C 端充值记录查询。

### 8.2 提现订单 B 端管理 + 审核（归 ser-finance）

```
- POST /admin/api/withdraw/page      — 提现订单列表
- POST /admin/api/withdraw/detail    — 提现订单详情
- POST /admin/api/withdraw/audit     — 提现审核（风控/财务/出款 三级）
- POST /admin/api/withdraw/lock      — 锁定提现单（防止并发审核）
- POST /admin/api/withdraw/unlock    — 解锁提现单
```

归属分析：三级审核流程完全在 ser-finance 内部，钱包仅提供冻结/解冻/扣款 RPC。
当前结论：确定归 ser-finance。

### 8.3 兑换订单 B 端管理（归属模糊）

```
- POST /admin/api/exchange/page      — 兑换订单列表
- POST /admin/api/exchange/detail    — 兑换订单详情
```

归属分析：兑换是钱包内部完成的操作，不涉及 ser-finance。但 B 端管理兑换记录是否在钱包还是财务？
当前结论：归 ser-wallet 更合理（兑换全链路在钱包内部）。但因 B-13 用户交易流水已覆盖兑换记录查询能力，独立接口优先级低。标为后期按需补充。

### 8.4 手动调账 B 端界面（归 ser-finance）

```
- POST /admin/api/wallet/adjust/create  — 创建手动调账单
- POST /admin/api/wallet/adjust/audit   — 审核手动调账
- POST /admin/api/wallet/adjust/page    — 手动调账记录
```

归属分析：手动调账的创建和审核流程属于财务管理。审核通过后 ser-finance 调用 R-05 ManualAdjust 执行。
当前结论：确定归 ser-finance。钱包仅提供 RPC 执行能力。

## 十一、与 IDL 设计文档的口径对照

差异说明：

