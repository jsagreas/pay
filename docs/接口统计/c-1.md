# ser-wallet 接口统计清单（整合版）

> 整合来源：11 份接口评估方案（c-1~c-5、d-1~d-3、tmp-1~tmp-3）+ IDL 设计 + RPC 暴露/调用分析 + 依赖关系 + 产品原型
> 统计口径：ser-wallet 模块（含币种配置子模块）自身需要实现的全部接口
> 分类标准：必须 = 所有方案一致认定的核心接口；应有 = 多数方案包含、产品原型有对应页面；商榷 = 方案间有分歧或归属待确认

---

## 全局数量概览

```
C 端接口（gate-font）  ·····  25 个
B 端接口（gate-back）  ·····  10 个
RPC 暴露（被其他模块调用）··  19 个
RPC 调用（我们调其他模块）··   7 个
定时任务  ················   2 个
─────────────────────────────
合计  ···················  63 个
```

## 一、C 端接口（25 个）

> 通过 gate-font 网关暴露给 App/H5 前端，走 Hertz HTTP → Kitex RPC 桥接。
> 路径前缀统一为 `/api/wallet/`，HTTP 方法遵循 RESTful 语义。

### 1.1 钱包基础（3 个）

```
GET  /api/wallet/currencies           获取币种列表及余额
GET  /api/wallet/balance              查询当前币种各子钱包余额
POST /api/wallet/currency/switch      切换当前操作币种
```

- **GET /api/wallet/currencies** — 【必须】
  C端首页入口。返回所有已启用币种 + 每种币种的总余额，用于币种切换列表展示。
  无参数，userId 从 token 获取。11/11 方案一致。

- **GET /api/wallet/balance** — 【必须】
  查询当前选中币种下各子钱包（中心/奖励/主播/代理/场馆）的可用余额和冻结余额。
  参数：`currencyCode`。11/11 方案一致。

- **POST /api/wallet/currency/switch** — 【应有】
  切换用户当前操作币种，校验目标币种是否启用。
  部分方案（c-4、d-1）认为可合并到 currencies 接口中由前端自行记录，但多数方案保留为独立接口。8/11 方案保留。

### 1.2 充值（6 个）

```
GET  /api/wallet/deposit/methods           获取充值方式列表
POST /api/wallet/deposit/order             创建充值订单
GET  /api/wallet/deposit/usdt-payment      获取 USDT 支付页信息
POST /api/wallet/deposit/check-duplicate   重复转账检查
GET  /api/wallet/deposit/bonus-list        获取额外奖金活动列表
GET  /api/wallet/deposit/order/status      查询充值订单状态
```

- **GET /api/wallet/deposit/methods** — 【必须】
  返回当前币种可用的充值方式（USDT/银行转账/电子钱包扫码），含图标、金额范围、预设档位。
  数据来源于财务模块 RPC（我们调用 `ser-finance.GetRechargeMethodList`）。
  11/11 方案一致。

- **POST /api/wallet/deposit/order** — 【必须】
  统一创建充值订单入口，通过 `methodType` 区分 USDT/银行/扫码。
  内部逻辑：重复检查 → 锁汇 → 创建本地订单 → 对接通道（或通过财务转发）。
  10/11 方案为统一接口；c-3 拆为 3 个独立接口（按方式类型），但多数方案不采纳拆分。

- **GET /api/wallet/deposit/usdt-payment** — 【必须】
  返回 USDT 充值地址、链网络（TRC20/ERC20/BEP20）、二维码数据、过期时间。
  产品原型明确有独立的 USDT 支付页。11/11 方案一致。

- **POST /api/wallet/deposit/check-duplicate** — 【商榷】
  检查是否有未完成的重复充值订单。规则：1-2 笔软拦截（弹窗警告），≥3 笔硬拦截。
  分歧：c-1 认为可合并到 CreateDepositOrder 内部逻辑（不单独暴露接口）；c-3 认为必须独立。
  建议：作为独立接口保留，前端在用户点击充值前主动调用，提升用户体验。7/11 方案保留。

- **GET /api/wallet/deposit/bonus-list** — 【商榷】
  返回当前可用的充值奖金活动列表（如"获得额外 50% 奖金，需最低充值 BSB 1000"）。
  分歧：数据源归属未确认——运营活动模块管理 or 财务模块管理 or 钱包自身维护？
  产品原型有明确的"选择奖金"弹窗页面，接口本身需要，但数据源待评审确认。9/11 方案保留。

- **GET /api/wallet/deposit/order/status** — 【应有】
  前端轮询充值订单状态（待支付/支付中/成功/失败/已过期）。
  用于用户提交支付后等待页面的状态刷新。10/11 方案保留。

### 1.3 提现（7 个）

```
GET  /api/wallet/withdrawal/methods         获取提现方式列表
GET  /api/wallet/withdrawal/limit           获取提现限额与费率
POST /api/wallet/withdrawal/order           创建提现订单
GET  /api/wallet/withdrawal/account         获取提现账户信息
POST /api/wallet/withdrawal/account         保存提现账户信息
GET  /api/wallet/withdrawal/order/detail    提现订单详情
GET  /api/wallet/withdrawal/records         提现记录列表
```

- **GET /api/wallet/withdrawal/methods** — 【必须】
  返回当前币种可用的提现方式（银行卡/GoPay/OVO/DANA/PromptPay/USDT）。
  需结合 KYC 状态过滤：未认证 → 仅显示 USDT；已认证 → 全部方式。
  数据来源于财务模块 RPC + KYC 查询。11/11 方案一致。

- **GET /api/wallet/withdrawal/limit** — 【应有】
  返回指定提现方式的单笔最低/最高、单日限额、手续费率、T+N 到账时间。
  分歧：c-4 认为可合并到 methods 的响应中；c-3/d-1 认为应独立。
  数据来源于财务模块 RPC（`ser-finance.GetWithdrawConfig`）。8/11 方案保留。

- **POST /api/wallet/withdrawal/order** — 【必须】
  创建提现订单。整个系统中事务性最强的接口。
  内部逻辑：余额校验 → KYC 校验 → 锁汇 → 冻结余额（本地 DB）→ 提交财务审核（RPC）→ 失败则回滚冻结。
  11/11 方案一致。

- **GET /api/wallet/withdrawal/account** — 【必须】
  获取用户已保存的提现账户（银行卡/电子钱包/USDT 地址）。
  分歧：c-3 按账户类型拆为 3 个 GET 接口，但多数方案为统一接口，通过 `accountType` 参数区分。
  10/11 方案为统一接口。

- **POST /api/wallet/withdrawal/account** — 【必须】
  保存/更新提现账户信息。账户持有人姓名自动从 KYC 获取（灰显不可编辑）。
  首次保存后不可自行修改（需联系客服）。
  分歧：同上，c-3 拆为 3 个 POST。10/11 方案为统一接口。

- **GET /api/wallet/withdrawal/order/detail** — 【应有】
  提现订单详情页（金额/方式/账户/状态/时间轴）。
  9/11 方案保留。

- **GET /api/wallet/withdrawal/records** — 【必须】
  提现记录分页列表。参数：`pageNo`、`pageSize`、`status`（筛选）。
  11/11 方案一致。

### 1.4 兑换（3 个）

```
POST /api/wallet/exchange/preview    兑换预览（汇率试算）
POST /api/wallet/exchange/execute    执行兑换
GET  /api/wallet/exchange/records    兑换记录列表
```

- **POST /api/wallet/exchange/preview** — 【应有】
  输入法币金额，返回预计获得的 BSB 数量、赠送 10% 奖金数量、实际到账总额。
  分歧：c-1 认为可由前端本地计算（汇率已通过其他接口获取）；其余 10/11 方案认为应由服务端计算确保精度。
  建议保留，服务端计算更可靠。

- **POST /api/wallet/exchange/execute** — 【必须】
  执行法币 → BSB 兑换。仅支持单向兑换。
  内部逻辑：锁汇 → 扣法币余额（中心钱包）→ 加 BSB 余额（中心钱包）→ 赠送 10% 加 BSB（奖励钱包）。
  零外部 RPC 依赖，完全本地事务。11/11 方案一致。

- **GET /api/wallet/exchange/records** — 【必须】
  兑换记录分页列表。
  10/11 方案一致（c-1 标记为 Should，但产品原型有明确 Tab）。

### 1.5 记录与奖励（6 个）

```
GET  /api/wallet/deposit/records      充值记录列表
GET  /api/wallet/deposit/order/detail 充值订单详情
GET  /api/wallet/exchange/records     兑换记录列表（已列于 1.4）
GET  /api/wallet/withdrawal/records   提现记录列表（已列于 1.3）
GET  /api/wallet/reward/records       奖励记录列表
GET  /api/wallet/record/detail        通用记录详情
```

> 注：充值记录和提现记录已分别列在 1.2 和 1.3 中，此处补充未列出的部分。

- **GET /api/wallet/deposit/records** — 【必须】
  充值记录分页列表。参数：`pageNo`、`pageSize`、`status`。
  11/11 方案一致。

- **GET /api/wallet/deposit/order/detail** — 【应有】
  充值订单详情。
  9/11 方案保留。

- **GET /api/wallet/reward/records** — 【应有】
  奖励记录分页列表（活动奖金、返奖、赠送等）。
  产品原型有"奖励"Tab。10/11 方案保留。

- **GET /api/wallet/record/detail** — 【应有】
  通用记录详情（通过 `type` + `orderId` 查询，统一入口）。
  分歧：c-1/c-3 按类型拆为 4 个详情接口；c-4/d-1/tmp-2 统一为一个。
  建议统一，避免接口膨胀。8/11 方案采用统一入口。

> **关于记录接口是否合并的说明**：
> 部分方案（c-4/tmp-1/tmp-2/tmp-3）提出将充值/提现/兑换/奖励记录合并为一个统一的 `GET /api/wallet/records?type=deposit|withdraw|exchange|reward` 接口。
> 优点是接口数量少；缺点是各类型记录字段差异大，统一响应体会臃肿。
> 本文采用**按类型拆分**方案（与产品原型的 4 个 Tab 对应），但各记录详情合并为统一入口。

## 二、B 端接口（10 个）

> 通过 gate-back 网关暴露给运营后台前端。
> 路径前缀统一为 `/api/wallet/admin/`。

### 2.1 币种配置管理（8 个）

```
GET  /api/wallet/admin/currency/list           币种列表分页查询
GET  /api/wallet/admin/currency/detail         币种详情
POST /api/wallet/admin/currency/update         编辑币种配置
POST /api/wallet/admin/currency/toggle-status  启用/禁用币种
POST /api/wallet/admin/currency/set-base       设置基准币种
GET  /api/wallet/admin/currency/rate-log       汇率日志分页查询
POST /api/wallet/admin/currency/refresh-rate   手动刷新实时汇率
GET  /api/wallet/admin/currency/operation-log  操作日志查询
```

- **GET .../currency/list** — 【必须】
  B 端币种列表页。20 个字段（代码/名称/类型/状态/汇率/精度/图标等）。
  11/11 方案一致。

- **GET .../currency/detail** — 【应有】
  编辑前获取单个币种完整配置，用于表单回填。
  9/11 方案保留。

- **POST .../currency/update** — 【必须】
  编辑币种配置（汇率浮动比例、偏差阈值、图标、精度、千分位等）。
  操作完成后调用 `ser-blog.AddActLog` 写审计日志。
  11/11 方案一致。

- **POST .../currency/toggle-status** — 【必须】
  启用/禁用币种。基准币种不可禁用。
  11/11 方案一致。

- **POST .../currency/set-base** — 【必须】
  设置基准币种（锚定 USDT，10 BSB = 1 USDT）。一次性操作，不可逆，需二次确认。
  11/11 方案一致。

- **GET .../currency/rate-log** — 【必须】
  汇率日志分页列表（每次汇率变更记录：实时汇率/平台汇率/入款汇率/出款汇率/变更时间/操作人）。
  产品原型有独立 Tab。11/11 方案一致。

- **POST .../currency/refresh-rate** — 【应有】
  手动触发从三方 API 拉取实时汇率并更新平台汇率。
  产品原型有"刷新汇率"按钮和倒计时（约 2 分钟间隔）。9/11 方案保留。

- **GET .../currency/operation-log** — 【应有】
  操作日志查询（编辑/启禁用/设置基准等操作的审计记录）。
  数据来源于 ser-blog，此接口可能通过 ser-blog 的 RPC 查询，也可本地 DB 冗余存储。
  分歧：部分方案认为直接走 ser-blog 的 B 端页面即可，不需要钱包独立接口。7/11 方案保留。

### 2.2 B 端辅助（2 个）

```
POST /api/wallet/admin/currency/create         新增币种
GET  /api/wallet/admin/rate-log/export          汇率日志导出
```

- **POST .../currency/create** — 【商榷】
  新增币种（代码/名称/类型/精度等）。
  分歧：c-1 明确说"币种新增是开发直接写入数据库，不走后台接口"；c-2/c-3/c-4 认为应有正式接口。
  币种数量少（4-6 种），变动极低频，直接 DB 写入的方案可行但不规范。
  建议评审确认。6/11 方案保留接口。

- **GET .../rate-log/export** — 【商榷】
  汇率日志导出为 CSV。
  仅 c-3/d-3 提及。功能不影响核心流程，可后置。2/11 方案提及。

## 三、RPC 暴露接口（19 个）

> 被其他模块通过 Kitex RPC 调用，不走网关。
> 所有 RPC 方法定义在 `service.thrift` 中，Req/Resp 结构体在 `wallet_rpc.thrift` 和 `currency_rpc.thrift` 中。

### 3.1 钱包资金操作（6 个）— 被财务模块调用

```
RpcGetUserBalance           查询用户余额
RpcCreditWallet             充值/加款入账
RpcDebitWallet              减款扣款
RpcFreezeBalance            冻结余额
RpcUnfreezeBalance          解冻余额
RpcConfirmFrozenDeduction   确认冻结扣除
```

- **RpcGetUserBalance** — 【必须 · P0】
  财务后台"人工加款/减款"弹窗展示用户当前各子钱包余额。
  入参：`userId` + `currencyCode`。返回各子钱包可用/冻结余额。
  11/11 方案一致。

- **RpcCreditWallet** — 【必须 · P0】
  通用入账接口。通过 `bizType` 区分场景（充值入账/人工加款/补单/返奖）。
  入参：`userId`、`currencyCode`、`walletType`（目标子钱包）、`amount`、`orderId`（幂等键）。
  CAS 乐观锁 + 事务保证（余额表 + 流水表同一事务）。
  11/11 方案一致（部分方案拆为 ManualCredit/SupplementCredit 等独立接口，但核心逻辑相同）。

- **RpcDebitWallet** — 【必须 · P0】
  通用扣款接口。通过 `bizType` 区分场景（人工减款/消费扣款）。
  入参同 CreditWallet。余额不足时：人工减款扣实际余额，消费扣款直接拒绝。
  11/11 方案一致。

- **RpcFreezeBalance** — 【必须 · P0】
  冻结可用余额（available → frozen）。提现创建时触发。
  入参：`userId`、`currencyCode`、`amount`、`orderId`（关联提现订单号，幂等键）。
  11/11 方案一致。

- **RpcUnfreezeBalance** — 【必须 · P0】
  解冻归还余额（frozen → available）。提现审核驳回或出款失败时触发。
  入参：`userId`、`currencyCode`、`orderId`（原冻结订单号，幂等键）。
  11/11 方案一致。

- **RpcConfirmFrozenDeduction** — 【必须 · P0】
  最终确认扣除冻结金额（frozen - amount）。提现出款成功时触发。
  入参：同 UnfreezeBalance。
  11/11 方案一致。

### 3.2 余额查询与扩展操作（4 个）— 被多个模块调用

```
RpcBatchGetBalance       批量查询多用户余额
RpcSupplementCredit      补单入账（主金额+奖金分别入不同子钱包）
RpcConsumeDebit          消费扣款（投注/商城，按优先级扣：中心→奖励）
RpcRewardCredit          返奖/打赏入账（入奖励钱包或主播钱包）
```

- **RpcBatchGetBalance** — 【应有 · P1】
  批量查询多个用户余额，用于财务报表和统计。
  入参：`userIds` 列表 + `currencyCode`。
  8/11 方案保留。

- **RpcSupplementCredit** — 【必须 · P0】
  补单入账，需同时操作两个子钱包：主金额 → 中心钱包，奖金额 → 奖励钱包。
  入参增加 `mainAmount` + `bonusAmount`。
  分歧：部分方案合并到 CreditWallet（调用两次），但多数方案保留为独立接口以保证原子性。
  9/11 方案保留。

- **RpcConsumeDebit** — 【应有 · P1】
  投注下注/商城购买等消费场景的扣款。优先扣中心钱包，不足时扣奖励钱包。
  分歧：c-1 标记"IDL 预留方法定义，实现随投注模块同步"；c-5/d-2 标记为必须。
  当前阶段预留接口定义，实现可后置到投注域开发时。7/11 方案保留。

- **RpcRewardCredit** — 【应有 · P1】
  返奖/打赏收入入账。入中心钱包或奖励钱包或主播钱包，由 `walletType` 指定。
  场景：投注中奖返奖、直播打赏收入。
  与 CreditWallet 的区别：RewardCredit 有按原始投注比例分配中心/奖励的逻辑。
  7/11 方案保留。

### 3.3 币种配置对外（7 个）— 被全平台调用

```
RpcGetPlatformRate            获取平台汇率
RpcConvertToBaseCurrency      转换为基准币种金额
RpcGetCurrencyConfig          获取单个币种配置详情
RpcGetAllEnabledCurrencies    获取所有已启用币种列表
RpcGetTransactionRate         获取并锁定交易汇率
RpcSnapshotBettingRate        快照投注汇率
RpcIsCurrencyEnabled          校验币种是否启用
```

- **RpcGetPlatformRate** — 【应有 · P1】
  获取两个币种之间的平台汇率。财务统计报表的多币种折算依赖此接口。
  入参：`fromCurrency` + `toCurrency`。返回 `rate` + `updateTime`。
  10/11 方案保留。

- **RpcConvertToBaseCurrency** — 【应有 · P1】
  将任意币种金额转换为基准币种（USDT）金额。
  入参：`amount` + `fromCurrency`。返回 `baseAmount` + `rateUsed`。
  8/11 方案保留。

- **RpcGetCurrencyConfig** — 【必须 · P0】
  获取单个币种的精度、符号、小数位、千分位、图标等配置。
  被全平台服务的金额展示逻辑依赖。
  11/11 方案一致。

- **RpcGetAllEnabledCurrencies** — 【必须 · P0】
  获取所有已启用币种列表。用于各服务的下拉选项、路由校验。
  11/11 方案一致。

- **RpcGetTransactionRate** — 【必须 · P0】
  获取并锁定交易汇率（充值/提现/兑换下单时）。
  返回 `lockedRate` + `expireTime` + `rateId`（关联到订单）。
  钱包内部也使用（同进程函数调用），同时通过 RPC 暴露给外部。
  10/11 方案保留。

- **RpcSnapshotBettingRate** — 【商榷 · P2】
  投注下单时快照当前汇率，用于结算时折算。
  调用方为竞彩/投注域，当前该域尚未进入开发。IDL 预留定义即可。
  5/11 方案提及。

- **RpcIsCurrencyEnabled** — 【必须 · P0】
  轻量校验某币种是否启用。各服务的入参前置校验依赖。
  10/11 方案保留。

### 3.4 通用辅助（2 个）

```
RpcGetBaseCurrency     获取基准币种信息
RpcGetCurrencyList     获取币种列表（含启用+禁用）
```

- **RpcGetBaseCurrency** — 【必须 · P0】
  返回当前平台基准币种代码和名称（如 USDT）。
  10/11 方案保留。

- **RpcGetCurrencyList** — 【应有 · P1】
  返回所有币种列表（含禁用的），供后台管理场景使用。
  与 GetAllEnabledCurrencies 的区别：本接口含禁用币种。
  8/11 方案保留。

## 四、RPC 调用接口（7 个）

> 我们主动调用其他模块的 RPC，需要在 ser-wallet 代码中引入对应的 RPC 客户端。

### 4.1 调用 ser-finance（4 个）— IDL 不存在，最大卡点

```
finance.GetRechargeMethodList     获取充值方式列表
finance.GetWithdrawMethodList     获取提现方式列表
finance.GetWithdrawConfig         获取提现限额/费率配置
finance.SubmitWithdrawOrder       提交提现订单到财务审核
```

- **finance.GetRechargeMethodList** — 【必须 · P0】
  C 端充值页渲染所需。入参：`currencyCode`。返回可用充值方式列表。
  **ser-finance IDL 不存在**，需与财务同事协调定义。
  在 IDL 就绪前使用 Mock 先行开发。

- **finance.GetWithdrawMethodList** — 【必须 · P0】
  C 端提现页渲染所需。入参：`currencyCode`。返回可用提现方式列表。
  同上，IDL 不存在。

- **finance.GetWithdrawConfig** — 【必须 · P0】
  提现须知展示（限额/手续费/T+N）+ 金额校验逻辑依赖。
  入参：`currencyCode` + `methodType`。
  同上，IDL 不存在。

- **finance.SubmitWithdrawOrder** — 【必须 · P0】
  提现流程中事务性最强的调用。钱包冻结余额后调用此接口提交审核。
  失败时**必须回滚冻结**（本地 UnfreezeBalance）。
  同上，IDL 不存在。

### 4.2 调用 ser-kyc（1 个）— IDL 已存在

```
kyc.KycQuery     查询 KYC 认证状态 + 真实姓名
```

- **kyc.KycQuery** — 【必须 · P0】
  两个子场景：① 提现方式展示时判断 KYC 状态（未认证仅 USDT）；② 提现账户持有人姓名自动填充。
  IDL 已存在（`common/idl/ser-kyc/service.thrift`），客户端 `rpc.KycClient()` 已注册，直接使用。
  降级策略：KYC 服务不可用时默认按"未认证"处理，仅展示 USDT 提现。

### 4.3 调用 ser-blog（1 个）— IDL 已存在

```
blog.AddActLog     B 端写操作审计日志（oneway）
```

- **blog.AddActLog** — 【应有 · P1】
  所有 B 端写操作（编辑/启禁用/设基准/刷新汇率）完成后触发。
  oneway void，不等待响应，不阻塞业务。工程规范强制（全工程所有 B 端服务都调用）。
  IDL 已存在，客户端 `rpc.BLogClient()` 已注册，直接使用。

### 4.4 调用 ser-buser（1 个）— IDL 已存在

```
buser.QueryUserByIds     B 端列表操作人姓名解析
```

- **buser.QueryUserByIds** — 【应有 · P2】
  B 端币种列表"最后修改人"、汇率日志"操作人"字段的 ID→姓名解析。
  入参：`ids []int64`。返回 `map[int64]string`。
  IDL 已存在，客户端 `rpc.BUserClient()` 已注册，直接使用。
  降级策略：不可用时直接显示操作人 ID。

## 五、定时任务（2 个）

```
SyncRealtimeRate           汇率定时更新
DepositOrderAutoExpire     充值订单超时自动关闭
```

- **SyncRealtimeRate** — 【必须 · P0】
  从 3 个第三方汇率 API 拉取实时汇率 → 取平均值 → 计算偏差 → 超阈值则更新平台汇率 → 记录汇率日志。
  频率：约 2 分钟一次（可配置）。
  11/11 方案一致。

- **DepositOrderAutoExpire** — 【应有 · P1】
  待支付的充值订单超时后自动关闭（设为"已过期"状态）。
  防止订单长期挂起占用系统资源。
  7/11 方案保留。

## 七、各方案间的关键分歧与决策建议

| # | 分歧点 | 方案立场 | 建议 |
|---|-------|---------|------|
| 1 | **充值订单是否按方式拆接口** | c-3 拆为 3 个（USDT/银行/扫码）；其余统一 | **统一**：1 个 CreateDepositOrder，通过 `methodType` 区分 |
| 2 | **提现账户是否按类型拆接口** | c-3 拆为 6 个（3×GET + 3×POST）；其余统一 | **统一**：GET/POST 各 1 个，通过 `accountType` 区分 |
| 3 | **记录查询是否合并为一个接口** | c-4/tmp-1/tmp-2 统一 `?type=`；c-1/c-3 按类型拆 | **按类型拆**：与产品原型 4 个 Tab 对应，返回字段差异大 |
| 4 | **兑换预览是否需要后端接口** | c-1 认为前端可本地计算；其余保留 | **保留**：服务端计算精度更可靠 |
| 5 | **人工加减款是否独立 RPC 接口** | c-1/c-4 独立 ManualCredit/ManualDebit；c-3/d-2/rpc暴露 合并到 CreditWallet/DebitWallet | **合并**：通过 `bizType` 区分，减少接口数量 |
| 6 | **B 端是否需要钱包管理接口** | c-1 认为 0 个（全走财务 RPC）；d-1/tmp-3 有 2-5 个 | **最小化**：当前阶段 0 个 B 端钱包管理接口，通过 RPC 暴露即可 |
| 7 | **币种新增是否需要接口** | c-1 认为开发直接写 DB；c-2/c-3/c-4 保留接口 | **商榷**：评审确认，倾向保留接口更规范 |
| 8 | **充值回调归属** | c-3 认为钱包接收；c-1/c-4 认为财务接收 | **财务接收**：银行/扫码回调归财务，USDT 链上确认可能钱包直接处理 |

## 附录 A：按业务场景的接口调用链

### 充值场景

```
① 用户进入充值页
   C 端 → GET  /deposit/methods              ← 内部调 finance.GetRechargeMethodList
   C 端 → GET  /deposit/bonus-list           ← 数据源待确认

② 用户选择方式+金额，点击充值
   C 端 → POST /deposit/check-duplicate      ← 本地查询
   C 端 → POST /deposit/order                ← 内部调 币种配置.GetTransactionRate（锁汇）
                                              ← 内部调 finance.通道创建（待确认）

③ USDT 充值展示地址
   C 端 → GET  /deposit/usdt-payment

④ 等待支付结果
   C 端 → GET  /deposit/order/status          ← 本地查询

⑤ 财务回调入账
   财务 → RPC  RpcCreditWallet                ← 被动接收
```

### 提现场景（依赖链最长）

```
① 用户进入提现页
   C 端 ← 内部调 kyc.KycQuery（KYC 状态+姓名）
   C 端 → GET  /withdrawal/methods            ← 内部调 finance.GetWithdrawMethodList
   C 端 → GET  /withdrawal/limit              ← 内部调 finance.GetWithdrawConfig

② 用户选择账户
   C 端 → GET  /withdrawal/account
   C 端 → POST /withdrawal/account            ← 内部调 kyc.KycQuery（姓名自动填充）

③ 用户确认提现
   C 端 → POST /withdrawal/order              ← 内部：锁汇 → FreezeBalance → finance.SubmitWithdrawOrder
                                              ← 提交失败必须回滚冻结

④ 审核结果
   财务 → RPC  RpcUnfreezeBalance             ← 驳回/失败
   财务 → RPC  RpcConfirmFrozenDeduction      ← 出款成功
```

### 兑换场景（零外部依赖）

```
① 用户进入兑换页
   C 端 → POST /exchange/preview              ← 内部调 币种配置.GetPlatformRate

② 用户确认兑换
   C 端 → POST /exchange/execute              ← 内部：锁汇 → 扣法币 → 加 BSB → 加奖励
                                              ← 完全本地事务
```
