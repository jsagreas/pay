## 二、币种配置模块 — B端 HTTP 接口（gate-back）

> 路由前缀：`/admin/api/wallet/...`
> 鉴权：BackAuthMiddleware
> 共7个接口（核心6 + 应该有1）
> 所有写操作需调 ser-blog.AddActLog() 记录审计日志

### 币种管理

1. **`POST /admin/api/wallet/currency/page`** — QueryCurrencyPage ｜ **[核心]**
   币种列表分页查询。返回所有已配置币种信息，含币种代码、名称、类型（法币/加密/平台币）、国家、图标、符号、千分位规则、小数位数、实时汇率、平台汇率、汇率浮动%、入款汇率、出款汇率、偏差阈值%、精度、排序、状态等字段。支持按类型和状态筛选。

2. **`POST /admin/api/wallet/currency/create`** — CreateCurrency ｜ **[核心]**
   新增币种。填写币种代码（不可重复）、名称、类型、国家、图标上传（SVG/PNG ≤ 5KB）、汇率浮动百分比、偏差阈值百分比、小数精度等。币种代码和类型创建后不可修改。

3. **`POST /admin/api/wallet/currency/update`** — UpdateCurrency ｜ **[核心]**
   编辑币种信息。可修改项：图标、汇率浮动%、偏差阈值%、千分位符号、小数位数、排序权重等。不可修改项：币种代码、类型。

4. **`POST /admin/api/wallet/currency/status`** — ToggleCurrencyStatus ｜ **[核心]**
   启用/禁用币种。禁用后C端不可见、不可操作。基准币种不可禁用。禁用时需考虑是否校验有余额用户。

5. **`POST /admin/api/wallet/currency/detail`** — GetCurrencyDetail ｜ **[应该有]**
   获取单个币种完整配置详情。用于编辑弹窗数据回显。也可复用列表接口单条数据，但独立接口更符合工程规范（参照 ser-item 的 QueryItemDetail 模式）。

### 基准币种

6. **`POST /admin/api/wallet/baseCurrency/set`** — SetBaseCurrency ｜ **[核心]**
   设置平台基准币种（USDT）。一次性操作，提交后不可修改，需二次确认弹窗。设置后按钮隐藏或置灰。BSB锚定关系：10 BSB = 1 USDT。

### 汇率日志

7. **`POST /admin/api/wallet/rateLog/page`** — QueryRateLogPage ｜ **[核心]**
   汇率日志分页查询。每次平台汇率变更（手动或定时任务触发）自动生成一条日志。字段含：编号、时间、币种、变更前/后实时汇率、变更前/后平台汇率、入款汇率、出款汇率、触发方式（自动/手动）。支持按日期范围和币种筛选。

## 三、币种配置模块 — RPC 暴露接口（供其他服务调用）

> IDL 位置：`common/idl/ser-wallet/currency.thrift`（规划）
> 调用方：钱包模块内部、财务模块、未来投注/礼物/统计等模块
> 共5个接口（核心3 + 应该有2）

1. **`GetActiveCurrencyList`** ｜ **[核心]**
   获取所有已启用币种列表。返回：币种ID、代码、名称、类型、图标URL、符号、千分位规则、小数位数、排序。
   调用方：C端钱包页币种切换、财务模块币种下拉筛选、未来各业务模块。

2. **`GetCurrencyConfig`** ｜ **[核心]**
   获取指定币种的完整配置信息。返回：精度规则、符号、千分位规则、状态、类型、图标等。所有涉及金额处理的模块均需调用此接口获取精度规则。
   调用方：钱包金额格式化、财务订单金额处理、报表展示。

3. **`GetExchangeRate`** ｜ **[核心]**
   获取指定币种的当前汇率。入参：源币种、目标币种、方向（入款/出款/平台）。返回：对应方向的汇率值。充值展示入款汇率，提现展示出款汇率，统计使用平台汇率。
   调用方：钱包充值/提现/兑换页面、财务统计报表。

4. **`CalcExchangeAmount`** ｜ **[应该有]**
   汇率换算计算。入参：源币种、目标币种、金额、方向。返回：换算后金额 + 使用的汇率值。统一换算入口，避免各模块自行计算出现精度差异。
   调用方：钱包兑换、USDT充值换算、报表对账。

5. **`BatchGetExchangeRates`** ｜ **[应该有]**
   批量获取多币种汇率。返回 `map<currencyCode, rateInfo>`。面向内部批量场景（统计报表、对账任务），区别于单次查询的 GetExchangeRate。
   调用方：财务统计模块、对账任务。

## 四、币种配置模块 — 定时任务

> 非接口，但属于 ser-wallet 服务内部必须实现的后台逻辑
> 共2个（核心1 + 应该有1）

1. **汇率偏差检测与自动更新** ｜ **[核心]**
   定时拉取3家三方汇率API获取实时汇率，取平均值计算偏差。当偏差值 ≥ 配置的阈值百分比时，自动更新平台汇率（入款和出款任一方向超阈值都触发更新）。更新后生成汇率日志记录。执行频率：分钟级。

2. **充值订单超时处理** ｜ **[应该有]**
   未支付的充值订单超时自动失效。QR码过期、支付链接超时后，订单状态标记为"支付超时"。避免长期挂起的待支付订单影响重复转账检测等逻辑。

## 五、多币种钱包模块 — C端 HTTP 接口（gate-font）

> 路由前缀：`/api/wallet/...`（需登录） 或 `/open/wallet/...`（免登录）
> 鉴权：FontAuthMiddleware（/api 路由组）
> 共19个接口（核心14 + 应该有3 + 商榷2）

### 钱包核心（2个）

1. **`POST /api/wallet/overview`** — GetWalletOverview ｜ **[核心]**
   钱包余额概览。返回当前币种下各子钱包余额：资金钱包（= 中心 + 奖励合并展示）、主播钱包、代理钱包。个人中心页和钱包首页的核心数据接口。子钱包类型：中心钱包（充值/消费主体）、奖励钱包（活动奖金，需完成流水要求）、主播钱包（直播收入）、代理钱包（预留）。

2. **`POST /api/wallet/currencies`** — ListActiveCurrencies ｜ **[核心]**
   获取用户可用币种列表。返回已启用币种的代码、名称、图标、符号，以及用户在各币种下的合计余额（用于币种切换下拉展示）。内部调用币种配置 RPC 获取启用币种，再查各币种余额。

### 充值相关（5个）

3. **`POST /api/wallet/deposit/methods`** — GetDepositMethods ｜ **[核心]**
   获取当前币种可用充值方式列表。返回各方式的名称、图标、充值档位（固定金额列表）、金额范围（最小/最大）、手续费率、关联的奖金活动摘要。数据来源：调用财务模块 RPC 获取支付方式配置。按当前币种过滤，不同币种支持的充值方式不同（如 VND 有银行转账、USDT 有链上充值）。

4. **`POST /api/wallet/deposit/create`** — CreateDepositOrder ｜ **[核心]**
   创建充值订单。统一入口，按充值方式类型分发处理（USDT链上 / 银行转账跳三方托管页 / 电子钱包跳三方）。流程：校验金额范围 → 重复转账检测 → 匹配支付通道（调财务 RPC） → 生成平台订单号（C + 16位） → 返回支付信息（三方URL / 收款地址 / 二维码）。含奖金活动选择（可选，单笔互斥）。

5. **`POST /open/wallet/deposit/usdt`** — GetUSDTDepositInfo ｜ **[核心]**
   获取 USDT 链上充值信息。返回：可选网络类型（TRC20 / ERC20 / BEP20）、对应收款地址、二维码数据。放在 Open 路由组（游客也可查看充值方式），属于平台承载页场景。

6. **`POST /api/wallet/deposit/status`** — QueryDepositStatus ｜ **[应该有]**
   查询充值订单当前状态。前端在支付页/等待页轮询此接口获取支付结果。返回：订单状态（待支付/支付中/成功/失败/超时）、预计到账时间。部分版本认为可合并到记录详情接口，但充值进行中页面需要轻量级轮询接口，独立更合理。

7. **`POST /api/wallet/deposit/bonus`** — ListAvailableBonus ｜ **[应该有]**
   获取当前充值可选的额外奖金活动列表。返回：活动名称、+N% 标签、最低充值门槛、流水倍数要求。单笔充值只能选一个活动（互斥）。数据来源待确认：可能来自活动模块 RPC 转调，也可能由财务支付配置关联。

### 兑换相关（2个）

8. **`POST /api/wallet/exchange/preview`** — PreviewExchange ｜ **[核心]**
   兑换预览试算。入参：法币金额。返回：兑换获得 BSB 数量 + 赠送 BSB 数量（10% 赠送，入奖励钱包）+ 实际到账 BSB + 使用汇率 + 流水要求说明。统一展示口径：1 BSB = X 法币。用户输入金额后实时计算展示。

9. **`POST /api/wallet/exchange/create`** — CreateExchangeOrder ｜ **[核心]**
   执行法币 → BSB 单向兑换。扣除法币余额（中心钱包），BSB 兑换部分入中心钱包、赠送部分入奖励钱包（标记需完成 X 倍流水后可提现）。生成兑换流水记录。

### 提现相关（5个）

10. **`POST /api/wallet/withdraw/methods`** — GetWithdrawMethods ｜ **[核心]**
    获取当前币种可用提现方式列表。根据用户 KYC 认证状态动态过滤：未认证仅显示 USDT 提现，已认证显示全渠道。同时受充值路由规则约束：法币充值 → 只能法币提现，USDT 充值 → 只能 USDT 提现，混合充值 → 法币提现。返回各方式的名称、图标、单笔限额、单日限额、手续费率、T+N 到账时间。数据来源：调用财务模块 RPC + 内部 KYC 校验。

11. **`POST /api/wallet/withdraw/account/get`** — GetWithdrawAccount ｜ **[核心]**
    获取用户已绑定的提现账户信息。按提现方式类型返回：银行卡（银行名称/账号/持有人/网点）、电子钱包（类型/账号）、USDT 地址（链类型/地址）。首次提现时为空，需先保存账户信息。非首次提现自动填充，用户不可自行修改。

12. **`POST /api/wallet/withdraw/account/save`** — SaveWithdrawAccount ｜ **[核心]**
    首次保存提现账户信息。入参根据提现方式不同：银行卡需填银行名称、账号、持有人姓名、网点；电子钱包需填类型和账号；USDT 需填链类型和地址。持有人姓名须与 KYC 实名一致（忽略大小写和多余空格），通过调用 ser-kyc 获取认证姓名进行校验。保存后不可自行修改（修改需联系客服）。

13. **`POST /api/wallet/withdraw/create`** — CreateWithdrawOrder ｜ **[核心]**
    创建提现订单。流程：校验 KYC 状态 → 校验单笔/单日限额 → 校验账户信息完整性 → 冻结对应金额 → 生成订单号（T + 16位） → 订单进入「待风控审核」状态。后续由财务模块 B 端执行风控审核 → 财务审核 → 出款三级流程。

14. **`POST /api/wallet/withdraw/limits`** — GetWithdrawLimits ｜ **[商榷]**
    获取提现限额信息：单笔最低/最高、单日限额、今日已用额度、手续费率。
    **分歧点**：7/11 份评估将此数据合并在 GetWithdrawMethods 的返回值中而非独立接口。独立接口的好处是前端提现表单可单独调用做校验提示，合并的好处是减少请求数。建议评审确认。

### 记录相关（2个）

15. **`POST /api/wallet/record/page`** — QueryRecordPage ｜ **[核心]**
    记录分页查询。按类型参数区分4个 Tab：充值记录 / 提现记录 / 兑换记录 / 奖励记录。严格绑定当前币种，不支持跨币种混合查询。列表含状态色标（已完成/审核中-橙色/失败-红色）。
    **设计决策**：11份评估中，8份使用统一接口 + type 参数区分 Tab，3份拆为4个独立接口。推荐统一接口方案（参照工程惯例，减少接口数）。

16. **`POST /api/wallet/record/detail`** — GetRecordDetail ｜ **[核心]**
    记录详情。返回：订单号、金额明细（原始金额/手续费/实际到账）、状态、时间线、方式、失败原因（如有）。失败记录含"更换方式"操作链接。

### 辅助接口（3个）

17. **`POST /api/wallet/rate`** — GetCurrentRate ｜ **[核心]**
    获取当前币种的汇率信息。返回入款汇率、出款汇率、BSB 换算比。充值页/提现页/兑换页均需展示汇率参考值。
    **设计决策**：部分版本将汇率数据内嵌在 GetDepositMethods / GetWithdrawMethods 返回值中。独立接口的好处是前端可统一调用，避免重复嵌入；缺点是多一次网络请求。推荐独立接口。

18. **`POST /api/wallet/deposit/duplicate`** — CheckDuplicateDeposit ｜ **[应该有]**
    重复转账检测。校验当前用户是否存在同币种 + 同金额的待处理充值订单。返回：当前待处理订单数 + 弹窗策略（1~2笔软拦截显示警告、≥3笔硬拦截禁止创建）。
    **分歧点**：4/11 份评估将此逻辑合并到 CreateDepositOrder 内部。但产品原型有独立的警告弹窗交互流程（前端需先检测再决定是否创建），因此推荐保留为独立接口。

19. **`POST /api/wallet/deposit/cancel`** — CancelDepositOrder ｜ **[商榷]**
    取消超时未支付的充值订单。
    **分歧点**：仅 3/11 份评估列出此接口。多数版本认为超时订单应由后端定时任务自动处理（见定时任务 T2），无需 C 端主动取消。如产品评审要求用户可手动取消，则需要此接口。

## 六、多币种钱包模块 — B端 HTTP 接口（gate-back）

> 路由前缀：`/admin/api/wallet/...`
> 说明：钱包模块的 B 端管理功能（充值记录/提现记录/人工修正/数据统计）由财务管理模块承担，钱包本身的 B 端接口需求极少
> 共2个（商榷2，归属待确认）

1. **`POST /admin/api/wallet/user/page`** — QueryUserWalletPage ｜ **[商榷]**
   B 端查询用户钱包余额列表。客服/风控场景需查看指定用户的各币种各子钱包余额。
   **归属分歧**：6/11 份评估认为此功能通过财务模块 RPC 调用 GetUserBalance 实现（不需要钱包提供独立 B 端页面），5/11 份认为应有独立 B 端页面。如归钱包模块则需此接口。

2. **`POST /admin/api/wallet/user/detail`** — QueryUserWalletDetail ｜ **[商榷]**
   B 端查询指定用户的钱包详情。展示用户在各币种下的中心/奖励/主播/代理钱包余额明细及冻结金额。
   **同上**：归属取决于产品设计——是否在钱包模块提供独立的 B 端用户钱包管理页面。

## 七、多币种钱包模块 — RPC 暴露接口（供财务等模块调用）

> IDL 位置：`common/idl/ser-wallet/wallet.thrift`（规划）
> 这是 ser-wallet 的核心能力输出，也是与财务模块的关键接口契约
> 共11个接口（核心8 + 应该有2 + 商榷1）

### 资金操作类 — 财务模块调用（7个核心）

1. **`WalletCredit`** ｜ **[核心]**
   充值入账。将充值金额存入用户指定币种的中心钱包，生成入账流水。
   调用方：财务模块 → 充值成功回调、补单审核通过。
   入参：userId、currencyCode、walletType（中心）、amount、bizType（充值/补单）、orderNo（幂等键）。

2. **`WalletCreditReward`** ｜ **[核心]**
   奖金入账。将奖金金额存入用户奖励钱包，标记流水稽核要求（需完成 X 倍流水后可提现）。
   调用方：财务模块 → 充值额外奖金发放；活动模块 → 任务奖励发放。
   入参：userId、currencyCode、amount、auditMultiplier（流水倍数要求）、bizType、orderNo。
   **设计决策**：5/11 份评估将此合并到 WalletCredit（通过 walletType=奖励 + 附加 auditMultiplier 字段区分），6/11 份独立。推荐独立接口，因奖金入账有流水稽核这一额外业务逻辑。

3. **`WalletFreeze`** ｜ **[核心]**
   冻结余额。提现发起时，将提现金额从可用余额转为冻结余额，冻结期间不可消费。
   调用方：财务模块 → 创建提现订单时。
   入参：userId、currencyCode、walletType、amount、orderNo。

4. **`WalletUnfreeze`** ｜ **[核心]**
   解冻余额。提现被驳回（风控/财务任一阶段）或出款失败时，将冻结金额归还为可用余额。
   调用方：财务模块 → 提现驳回、出款失败回调。
   入参：userId、currencyCode、walletType、amount、orderNo。

5. **`WalletDeduct`** ｜ **[核心]**
   扣款确认。提现出款成功后，从冻结余额中最终扣除（冻结 → 扣减，非从可用余额扣）。
   调用方：财务模块 → 代付成功回调。
   入参：userId、currencyCode、walletType、amount、orderNo。

6. **`WalletManualAdd`** ｜ **[核心]**
   人工加款。按指定子钱包类型（中心/奖励/主播/代理）分别加款，支持一次操作指定多个子钱包的加款金额。
   调用方：财务模块 → 人工加款审核通过后（订单号 A + 16位）。
   入参：userId、currencyCode、walletAmounts（map<walletType, amount>）、orderNo、reason。

7. **`WalletManualSub`** ｜ **[核心]**
   人工减款。按指定子钱包类型分别减款，余额不足时实际扣款 = 当前余额（不会出现负数）。
   调用方：财务模块 → 人工减款审核通过后（订单号 M + 16位）。
   入参：同 WalletManualAdd。

### 查询类（1个核心）

8. **`GetUserBalance`** ｜ **[核心]**
   查询用户钱包余额。返回指定用户在指定币种下各子钱包的可用余额和冻结金额。与 C 端 GetWalletOverview 的区别在于：返回更多内部字段（冻结金额、子钱包级别明细），面向服务间调用。
   调用方：财务模块 → 人工加减款弹窗展示余额、补单表单关联用户；其他服务 → 余额校验。

### 消费/返奖类 — 未来业务模块调用（2个应该有）

9. **`WalletConsume`** ｜ **[应该有]**
   消费扣款。按消费优先级自动扣款：先扣中心钱包，中心不足再扣奖励钱包，混合扣款按比例记录（用于后续按比例返奖）。调用方无需关心优先级策略，由钱包内部统一处理。
   调用方：未来投注模块 → 投注扣款；礼物模块 → 打赏扣款。
   **说明**：当前版本无投注/礼物功能，但架构上需预留此接口。11份评估中 8份包含此接口，设计时建议提前定义 IDL 但可暂不实现业务逻辑。

10. **`WalletReward`** ｜ **[应该有]**
    投注返奖入账。按投注扣款时中心/奖励钱包的金额占比，将返奖金额分别归入对应子钱包。
    调用方：未来投注模块 → 返奖。
    **同上**：预留设计。

### 批量查询类（1个商榷）

11. **`BatchGetUserBalances`** ｜ **[商榷]**
    批量查询多用户余额。返回 `map<userId, balanceInfo>`。面向运营报表、数据统计等批量场景。
    **分歧点**：仅 4/11 份评估列出。部分版本认为统计需求应直接查数据库（报表服务读从库）而非 RPC。如后续有跨服务批量查余额的明确需求再添加。

### 8.2 财务模块（待对接，所有接口需 Mock）

财务模块（ser-finance）由其他开发者负责，当前 IDL 和服务名待定。以下是根据业务链路分析推导的必要调用。对接前钱包模块需编写 Mock 实现以支持独立开发。

6. **财务 → `GetDepositMethods`** ｜ **[核心]**
   获取充值方式配置。按币种返回可用充值方式列表（方式名称/图标/档位/金额范围/手续费/关联通道）。C 端 GetDepositMethods 的底层数据来源。
   **P0 优先级**：充值链路绕不过，需最先完成 Mock。

7. **财务 → `MatchDepositChannel`** ｜ **[核心]**
   充值通道匹配。充值下单时调用财务模块，按轮询算法（成功率系数 + 时间系数 + 预警系数）选择最优代收通道，创建通道侧订单，返回三方支付跳转信息。
   **P0 优先级**：同上。

8. **财务 → `GetWithdrawMethods`** ｜ **[核心]**
   获取提现方式配置。按币种返回可用提现方式列表（方式名称/费率/限额/T+N天数/关联通道）。C 端 GetWithdrawMethods 的底层数据来源。

9. **财务 → `SubmitWithdrawRequest`** ｜ **[核心]**
   提交提现出款请求。提现审核通过后，调用财务模块发起代付通道出款。
   **说明**：此调用方向存在架构讨论（见待确认事项 Q-1），可能由财务模块主动监听提现订单状态变化而非钱包主动调用。

10. **财务 → `QueryChannelOrderStatus`** ｜ **[应该有]**
    查询通道订单状态。C 端充值状态轮询时，可能需要调用财务模块查询通道侧订单的实时状态。
    **说明**：如果充值回调由财务模块处理后 RPC 通知钱包，则此接口不需要。

### 8.3 外部 HTTP 调用

11. **三方汇率 API（HTTP）** ｜ **[商榷]**
    定时拉取3家三方汇率源取平均值。非 RPC 调用而是 HTTP 请求外部 API。
    **分歧点**：此调用由定时任务触发而非用户请求触发，部分版本不将其计入接口依赖。实际开发中需要实现 HTTP Client 调用三方汇率 API，工作量客观存在。

### 10.2 数据流向（核心链路）

**充值流程**
```
用户 → GetDepositMethods → 选择方式 → CreateDepositOrder
    → 调财务.MatchDepositChannel(匹配通道) → 返回支付信息/跳转URL
    → 用户支付 → 三方回调 → 财务.CollectNotify
    → 财务调 wallet.WalletCredit(入中心钱包)
    → 如选了奖金活动: 财务调 wallet.WalletCreditReward(入奖励钱包)
```

**提现流程**
```
用户 → GetWithdrawMethods(含KYC过滤) → 选择方式
    → GetWithdrawAccount → SaveWithdrawAccount(首次)
    → CreateWithdrawOrder → wallet.WalletFreeze(冻结)
    → 财务B端: 风控审核 → 财务审核
    → 审核通过: 财务发起代付 → 三方回调
       → 成功: 财务调 wallet.WalletDeduct(扣冻结)
       → 失败: 财务调 wallet.WalletUnfreeze(解冻归还)
    → 审核驳回: 财务调 wallet.WalletUnfreeze(解冻归还)
```

**兑换流程**
```
用户 → PreviewExchange(试算) → 确认 → CreateExchangeOrder
    → 内部调 CalcExchangeAmount(汇率换算)
    → 扣法币(中心钱包) → 加BSB(中心钱包+奖励钱包)
    → 生成兑换流水记录
```

**人工修正流程**
```
B端运营 → 财务.CreateManualAddOrder(创建加款申请)
    → 财务B端弹窗调 wallet.GetUserBalance(展示当前余额)
    → 审核 → 财务.ReviewCorrectionOrder(审核通过)
    → 财务调 wallet.WalletManualAdd(执行加款)
```

## 十二、待评审确认事项

> 以下事项在11份评估中反复被提及为不确定项，需在正式需求评审或跨部门沟通中明确

### Q-1：充值链路中钱包与财务的调用方向

钱包创建充值订单后，是钱包主动调财务的 MatchDepositChannel（钱包 → 财务），还是财务订阅钱包的订单事件？
**当前假设**：钱包 → 财务（9/11 份评估的一致假设）

### Q-2：充值回调接收方

三方代收支付回调由哪个服务接收？
**当前假设**：财务模块接收（财务持有通道密钥进行签名验证），验证通过后 RPC 调钱包 WalletCredit 入账。USDT 链上充值可能例外（平台承载页场景）。

### Q-3：充值订单归属

充值订单表由钱包创建还是财务创建？
**当前假设**：钱包创建平台订单（C + 16位），财务创建通道订单。两表分属两个模块。

### Q-4：奖金活动数据来源

C 端 ListAvailableBonus 的活动数据由谁维护？
**三种可能**：(a) 财务模块支付配置中关联活动规则 (b) 独立活动模块维护 (c) 钱包模块自建简易活动配置。
**当前假设**：暂按财务关联或活动模块 RPC 转调，钱包本身不维护活动数据。

### Q-5：提现账户修改权限

需求明确"用户不可自行修改提现账户"，但是否有 B 端运营修改通道？
**当前假设**：当前无此需求，后续如有可在财务或钱包 B 端增加。

### Q-6：币种配置是独立服务还是钱包子模块

币种配置和钱包是否部署为同一个 ser-wallet 服务？
**当前假设**：同一服务内的两个子模块（同一 IDL 文件组织），类似 ser-item 的 item + item_publish 模式。如需独立部署，IDL 和 rpc_client 需分别创建。

### Q-7：提现出款提交方向

提现审核通过后，是钱包调财务提交出款，还是财务审核通过后自行发起出款？
**当前假设**：财务审核通过后自行发起代付（财务持有通道 API 对接逻辑）。钱包无需调用 SubmitWithdrawRequest。但如果架构设计为钱包主控提现全流程，则需要此调用。

### Q-8：汇率定时任务执行方式

汇率拉取由 ser-wallet 内置 cron 还是调 ser-cron 服务触发？
**当前假设**：参照工程中 ser-cron 服务的定位，可能由 ser-cron 定时触发调用 ser-wallet 的更新汇率 RPC 方法。具体取决于团队对定时任务的管理方式。
