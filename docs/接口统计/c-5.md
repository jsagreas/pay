# ser-wallet 接口统计（整合版）

> 整合来源：接口评估（11 个迭代版本）+ IDL 设计 + 实现表达 + RPC 暴露/调用分析 + 网关路由规范
> 方法论：以 IDL 设计中的 46 个 RPC 方法为骨架，逐个映射 HTTP 路由 + 标注层级 + 说明用途
> 路由规范：参照现有 gate-back / gate-font 实际代码格式，全部 POST，路径 `/{module}/{resource}/{action}`
> 层级说明：🔴 Core = 必须有 · 🟡 Should = 应该有 · ⚪ Optional = 可商榷

---

# 速查数字

```
接口总数     46 个 RPC 方法（1 个 WalletService）
  B端接口    12 个（币种配置 9 + 钱包管理 3）
  C端接口    21 个（首页 2 + 充值 6 + 兑换 2 + 提现 6 + 记录 5）
  RPC接口    12 个（余额操作 8 + 币种查询 4）
  通用        1 个（枚举查询）

层级分布
  🔴 Core       38 个（必须交付）
  🟡 Should      7 个（应该有，工程或业务需要）
  ⚪ Optional    1 个（可商榷，视需求评审）

网关分布
  gate-back    12 个路由（B端后台管理）
  gate-font    21 个路由（C端用户操作）
  仅 RPC       12 个（不经网关，模块间直调）
  通用          1 个（gate-back + gate-font 各 1）
```

# 一、币种配置 · B端管理（9 个）

> 网关：gate-back · 认证：my_router.Api（B端 JWT）
> 对应文件：currency_service.go
> 开发阶段：第二期

```
POST /wallet/currency/page            → QueryCurrencyPage
POST /wallet/currency/create          → CreateCurrency
POST /wallet/currency/update          → UpdateCurrency
POST /wallet/currency/status          → ToggleCurrencyStatus
POST /wallet/currency/detail          → QueryCurrencyDetail
POST /wallet/rate/save                → SaveExchangeRate
POST /wallet/rate/page                → QueryExchangeRatePage
POST /wallet/rateLog/page             → QueryRateLogPage
POST /wallet/operationLog/page        → QueryOperationLogPage
```

### 逐项说明

**1. `POST /wallet/currency/page` → QueryCurrencyPage** 🔴 Core

币种列表分页查询。B 端后台币种配置列表页的数据源。
返回所有币种（含启用/禁用），支持按名称、类型、状态筛选。
→ 原型依据：币种配置列表页

**2. `POST /wallet/currency/create` → CreateCurrency** 🔴 Core

新增币种。填写币种代码、名称、图标、精度、充提限额、手续费、T+N 等完整参数。
精度字段一旦设定发布后**不可修改**（影响全部金额计算），需评审确认。
写操作完成后调 `ser-blog.AddActLog` 记审计日志。
→ 原型依据：新增按钮 + 编辑弹窗

**3. `POST /wallet/currency/update` → UpdateCurrency** 🔴 Core

编辑币种配置。可修改名称、图标、限额、手续费、T+N 等。
**精度(precision)字段不允许修改**——修改会导致所有历史余额和流水数据失真。
写操作完成后调 `ser-blog.AddActLog` 记审计日志。
→ 原型依据：编辑按钮 + 编辑弹窗

**4. `POST /wallet/currency/status` → ToggleCurrencyStatus** 🔴 Core

币种启用/禁用。参数 `status`：1=启用，0=禁用。
禁用后 C 端不再展示该币种，但已有余额不受影响。
写操作完成后调 `ser-blog.AddActLog` 记审计日志。
→ 原型依据：状态切换操作

**5. `POST /wallet/currency/detail` → QueryCurrencyDetail** 🟡 Should

单个币种详情查询。编辑前回显使用，也可合并到列表接口的每行数据中返回。
如列表接口已返回完整字段，此接口可省略。
→ 原型依据：编辑回显需要

**6. `POST /wallet/rate/save` → SaveExchangeRate** 🔴 Core

汇率设置。支持新增和编辑。参数：源币种 → BSB 汇率值。
写入后同时写 `rate_log` 日志表，并调 `ser-blog.AddActLog`。
汇率变更后需刷新 Redis 缓存（C 端实时读取）。
→ 需求文档 4-2：汇率规则说明

**7. `POST /wallet/rate/page` → QueryExchangeRatePage** 🔴 Core

汇率配置列表查询。B 端管理页展示当前各币种汇率。
→ 需求文档 4-1：汇率应用场景

**8. `POST /wallet/rateLog/page` → QueryRateLogPage** 🔴 Core

汇率变更日志查询。展示谁在什么时间把汇率从 X 改成了 Y。
→ 需求文档 4-4：明确要求汇率日志

**9. `POST /wallet/operationLog/page` → QueryOperationLogPage** 🟡 Should

币种配置操作日志查询。展示所有 B 端操作记录（新增/编辑/启禁用/汇率变更）。
数据来源于 ser-blog 的审计日志，此接口做聚合查询。
→ 原型依据：操作日志 Tab 页

# 二、币种配置 · RPC 暴露（4 个）

> 网关：不经网关，Kitex RPC 直调
> 调用方：财务模块、未来的游戏/直播/统计等模块
> 开发阶段：第二期

```
RPC  GetAvailableCurrencyList       → 获取可用币种列表
RPC  GetCurrencyInfo                → 获取币种详情
RPC  GetExchangeRate                → 获取汇率
RPC  GetCurrencyConfig              → 获取币种配置参数
```

### 逐项说明

**10. `RPC GetAvailableCurrencyList`** 🔴 Core

返回平台所有启用状态的币种列表。支持按 `currencyType` 过滤（1=法币，2=数字货币，不传=全部）。
任何需要币种下拉选择的模块都调此接口。
内部走 Redis 缓存（Cache-Aside），命中率极高。
→ 需求文档 3-1：币种类型划分

**11. `RPC GetCurrencyInfo`** 🔴 Core

获取币种元数据（名称、图标、精度、类型）。支持单个查询（传 `currencyCode`）和批量查询（传 `currencyCodes` 列表）。
其他模块展示币种信息时调此接口，避免各模块自己存币种数据。
→ 需求文档 3-3：后台配置字段

**12. `RPC GetExchangeRate`** 🔴 Core

获取指定币种对 BSB 的实时汇率。返回汇率值、精度、最后更新时间。
财务模块做基准币种换算、C 端充值/兑换/提现展示都依赖此接口。
内部走 Redis 缓存。
→ 需求文档 4-1/4-2：汇率应用场景

**13. `RPC GetCurrencyConfig`** 🔴 Core

获取币种的完整配置参数：精度、充值最低/最高、提现最低/最高、每日提现限额、手续费类型和值、T+N 天数。
财务模块做限额校验时需要，钱包内部也用（但走本地方法，不走 RPC）。
→ 需求文档 3-3：配置字段

# 三、钱包 · C 端首页（2 个）

> 网关：gate-font · 认证：my_router.Api（C端 JWT）
> 对应文件：wallet_service.go
> 开发阶段：第三期
> userId 来源：JWT → metainfo → tracer.GetUserId(ctx)，IDL 不暴露

```
POST /wallet/balance/query            → QueryWalletBalance
POST /wallet/currency/list            → QueryUserCurrencyList
```

### 逐项说明

**14. `POST /wallet/balance/query` → QueryWalletBalance** 🔴 Core

钱包余额查询。传入当前选中币种，返回 4 个子钱包（中心/奖励/代理/主播）的可用余额和冻结余额。
这是钱包首页的核心数据接口。余额**不走缓存**，直接查 DB（金融数据必须实时准确）。
→ 需求文档 4-1：钱包总体设计

**15. `POST /wallet/currency/list` → QueryUserCurrencyList** 🔴 Core

可用币种列表 + 各币种余额概览。C 端币种切换弹窗的数据源。
查币种配置（走缓存）+ 查该用户每个币种下的余额合计 → 返回列表。
币种切换是前端行为，后端只是换个参数重新查。
→ 需求文档 4-2：币种切换规则

# 四、钱包 · C 端充值（6 个）

> 网关：gate-font · 认证：my_router.Api
> 对应文件：deposit_service.go
> 开发阶段：第四期（依赖 ser-finance）
> 核心特点：我负责"头和尾"（发起 + 入账），中间的渠道对接交给财务

```
POST /wallet/deposit/methods          → QueryDepositMethodList
POST /wallet/deposit/create           → CreateDepositOrder
POST /wallet/deposit/address          → GetUsdtDepositAddress
POST /wallet/deposit/status           → QueryDepositStatus
POST /wallet/bonus/list               → QueryBonusActivityList
POST /wallet/deposit/checkDuplicate   → CheckDuplicateDeposit
```

### 逐项说明

**16. `POST /wallet/deposit/methods` → QueryDepositMethodList** 🔴 Core

获取当前币种可用的充值方式列表。
内部调 `rpc.FinanceClient().QueryPayConfig()` 获取支付配置，透传给 C 端展示。
不同币种对应不同充值方式（法币=银行转账/电子钱包，USDT=链上转账）。
→ 需求文档 5-1：充值方式总览

**17. `POST /wallet/deposit/create` → CreateDepositOrder** 🔴 Core

创建充值订单。这是充值流程最核心的接口。
步骤：参数校验 → 重复转账拦截 → 生成本地订单（状态=待支付） → 调 `rpc.FinanceClient().CreateChannelOrder()` → 返回支付信息。
此时用户余额**不变**，充值是先支付后入账。
→ 需求文档 5-2/5-3/5-4：各充值方式

**18. `POST /wallet/deposit/address` → GetUsdtDepositAddress** 🔴 Core

获取 USDT 充值地址。用户选择 USDT 充值时，返回链地址 + 二维码数据。
支持不同网络（TRC20/ERC20 等），参数传 `network`。
具体地址由财务模块/钱包地址服务提供，此接口做聚合。
→ 需求文档 5-2：USDT 充值

**19. `POST /wallet/deposit/status` → QueryDepositStatus** 🔴 Core

充值订单状态查询。C 端支付完成后轮询此接口确认到账。
状态：待支付 → 支付中 → 已入账 / 已失败 / 已过期。
→ 原型：通用回调页

**20. `POST /wallet/bonus/list` → QueryBonusActivityList** 🔴 Core

可选奖金活动列表。充值确认前展示当前可参与的奖金活动。
用户选择后，充值入账时额外往奖励钱包写入奖金。
→ 需求文档 5-6：额外奖金选择规则

**21. `POST /wallet/deposit/checkDuplicate` → CheckDuplicateDeposit** 🟡 Should

重复转账预校验。查询同金额同币种是否有"处理中"的订单，有则弹窗警告。
可合并到 CreateDepositOrder 内作为前置校验返回警告字段，不一定需要独立接口。
→ 需求文档 5-5：重复转账拦截规则

# 五、钱包 · C 端兑换（2 个）

> 网关：gate-font · 认证：my_router.Api
> 对应文件：exchange_service.go
> 开发阶段：第三期（完全闭环，无外部依赖，"烟囱测试"）

```
POST /wallet/exchange/preview         → PreviewExchange
POST /wallet/exchange/create          → CreateExchangeOrder
```

### 逐项说明

**22. `POST /wallet/exchange/preview` → PreviewExchange** 🔴 Core

兑换汇率预览。传入源币种和目标币种（固定为 BSB），返回实时汇率、赠送比例和换算结果。
展示："1 BSB = 23000 VND" + "赠送 5%"。
汇率走 Redis 缓存（币种配置内部调用，非 RPC）。
→ 需求文档 6：兑换模块

**23. `POST /wallet/exchange/create` → CreateExchangeOrder** 🔴 Core

创建兑换订单。核心事务操作：
① 重新获取最新汇率（不用页面旧汇率）
② 余额预检
③ 事务内：法币中心钱包 - amount，BSB 中心钱包 + amount，BSB 奖励钱包 + gift，写订单 + 写流水
兑换是即时完成的，不像充值/提现有异步等待。
→ 需求文档 6：法币/USDT → BSB 单向兑换

# 六、钱包 · C 端提现（6 个）

> 网关：gate-font · 认证：my_router.Api
> 对应文件：withdraw_service.go
> 开发阶段：第四期（依赖 ser-finance + ser-kyc）
> 核心特点：跨度最长、模块最多、状态最复杂

```
POST /wallet/withdraw/methods         → QueryWithdrawMethodList
POST /wallet/withdraw/account         → QueryWithdrawAccount
POST /wallet/withdraw/account/save    → SaveWithdrawAccount
POST /wallet/withdraw/create          → CreateWithdrawOrder
POST /wallet/withdraw/limit           → QueryWithdrawLimit
POST /wallet/kyc/status               → QueryKycStatus
```

### 逐项说明

**24. `POST /wallet/withdraw/methods` → QueryWithdrawMethodList** 🔴 Core

获取可用提现方式列表。
内部调 `rpc.FinanceClient().QueryPayConfig(type=提现)` 获取方式，再按**充值来源过滤**。
合规要求"钱从哪进从哪出"：只充过法币 → 只保留法币提现，只充过 USDT → 只保留 USDT。
→ 需求文档 7-1：提现方式由充值来源决定

**25. `POST /wallet/withdraw/account` → QueryWithdrawAccount** 🔴 Core

提现账户信息查询。返回用户已保存的银行卡/电子钱包/USDT 地址信息。
首次提现需填写后保存，后续自动回显。
银行卡号等敏感字段 AES 加密存储，返回时脱敏展示。
→ 需求文档 7-2/7-3：提现账户信息

**26. `POST /wallet/withdraw/account/save` → SaveWithdrawAccount** 🔴 Core

保存/更新提现账户信息。
支持多种账户类型：银行卡（卡号/开户行/户名）、电子钱包（账号/平台）、USDT（链地址/网络）。
银行卡号使用 `EncryptedString` 类型 AES 加密存储。
→ 需求文档 7-2/7-3：首次填写

**27. `POST /wallet/withdraw/create` → CreateWithdrawOrder** 🔴 Core

创建提现订单。这是全模块最复杂的接口。
步骤：并行校验（限额 + KYC 户名比对 + 余额）→ 冻结余额（乐观锁）→ 生成订单（状态=审核中）→ 调 `rpc.FinanceClient().SubmitWithdrawReview()`。
**关键风险**：冻结成功 → 提交审核 RPC 失败 → 必须立即解冻 + 标记失败 + 写回退流水。
→ 需求文档 7-1：提现申请

**28. `POST /wallet/withdraw/limit` → QueryWithdrawLimit** 🔴 Core

提现限额查询。返回当前币种的单笔最低/最高、每日限额、今日已用额度、手续费。
数据来源：币种配置（限额参数）+ 本地查询（今日已提现金额）。
→ 需求文档 7-5：限额展示

**29. `POST /wallet/kyc/status` → QueryKycStatus** 🟡 Should

KYC 状态查询（聚合转发）。
内部调 `rpc.KycClient().KycQuery()` 获取用户认证状态，透传给 C 端。
非 USDT 提现页进入时前置检查：未通过 KYC → 提示"请先完成实名认证"。
标注 Should-have 的原因：C 端可直接调 ser-kyc 的接口，不一定需要钱包转发。
但聚合在钱包里可减少 C 端调用复杂度（提现页一次请求搞定所有前置检查）。
→ 需求文档 7-1：非 USDT 提现需 KYC

# 九、钱包 · RPC 暴露（8 个）

> 网关：不经网关，Kitex RPC 直调
> 调用方：ser-finance（财务模块）
> 对应文件：rpc_service.go
> 开发阶段：第三期实现，第四期联调
> 核心设计：所有写方法必须带 refOrderNo（幂等键），userId 显式传参

```
RPC  DepositCredit                    → 充值入账
RPC  WithdrawFreeze                   → 提现冻结
RPC  WithdrawConfirm                  → 提现确认扣款
RPC  WithdrawUnfreeze                 → 提现失败解冻
RPC  ManualCredit                     → 人工加款
RPC  ManualDebit                      → 人工减款
RPC  ManualPatch                      → 人工补单
RPC  QueryBalanceRpc                  → 余额查询（RPC）
```

### 统一处理模式

8 个接口遵循相同模式：

```
① 幂等校验（refOrderNo 查状态，已完成 → 直接返回成功）
② 参数校验（userId / currency / amount / walletType）
③ 事务内操作（余额变动 + 订单状态更新 + 写流水）
④ 返回成功/失败（失败走 Kitex BizErr）
```

### 逐项说明

**38. `RPC DepositCredit`** 🔴 Core

充值入账。财务确认支付成功后调此接口，增加用户余额。
事务操作：中心钱包 `available + amount` + 更新订单状态=成功 + 写流水。
如有奖金活动：奖励钱包额外 `available + giftAmount` + 写奖金流水。
**幂等**：refOrderNo（充值订单号）+ changeType=充值 → 唯一键。
→ 财务需求 2-3：充值流程 callback → 入账

**39. `RPC WithdrawFreeze`** 🔴 Core

提现冻结（预扣）。创建提现订单时冻结用户余额。
事务操作：中心钱包 `available - amount`，`frozen + amount`（乐观锁，available >= amount）。
虽然 C 端 CreateWithdrawOrder 内部也会做冻结，但此接口作为 RPC 暴露，支持财务模块独立发起冻结。
→ 需求文档 7-1：提现冻结余额

**40. `RPC WithdrawConfirm`** 🔴 Core

提现确认扣款。三级审核通过 + 渠道出款成功后，正式扣除冻结金额。
事务操作：`frozen - amount` + 更新订单状态=已完成 + 写流水。
→ 财务需求 2-4：提现流程

**41. `RPC WithdrawUnfreeze`** 🔴 Core

提现失败解冻。审核拒绝或渠道出款失败时，退回冻结金额。
事务操作：`available + amount`，`frozen - amount` + 更新订单状态=已拒绝/失败 + 写回退流水。
→ 财务需求 2-4：提现失败退回

**42. `RPC ManualCredit`** 🔴 Core

人工加款。后台管理员审核通过后，手动给用户增加余额。
需指定 `walletType`（可加到任意子钱包）和 `operatorId`（审计追踪）。
→ 财务需求 2-5：人工修正-加款

**43. `RPC ManualDebit`** 🔴 Core

人工减款。后台管理员审核通过后，手动扣减用户余额。
需指定 `walletType`，`available >= amount` 才可执行（乐观锁）。
→ 财务需求 2-5：人工修正-减款

**44. `RPC ManualPatch`** 🔴 Core

人工补单入账。充值支付成功但因系统异常未入账的订单，手动补录。
需关联原始充值订单号（`originalOrderNo`），避免重复补单。
→ 财务需求 2-5：补单=漏入账的充值补录

**45. `RPC QueryBalanceRpc`** 🔴 Core

余额查询（RPC 版）。与 C 端 QueryWalletBalance 区别：此接口 userId 显式传参（不从 JWT 取）。
财务模块在人工修正前确认用户余额时调用。
纯读取，无状态，可安全重试。
→ 跨模块通用需求
