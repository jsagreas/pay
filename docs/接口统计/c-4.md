## 一、币种配置模块 — B端接口

> 定位：基础设施层，为钱包和财务模块提供币种、汇率基础数据
> 路由：gate-back → `/admin/api/wallet/...` → RPC → ser-wallet
> 使用者：后台管理员

### 接口列表

```
B-1  POST /admin/api/wallet/currency/page        币种列表分页查询      [必须有]
B-2  POST /admin/api/wallet/currency/edit         编辑币种配置          [必须有]
B-3  POST /admin/api/wallet/currency/status       启用/禁用币种         [必须有]
B-4  POST /admin/api/wallet/currency/setBase      设置基准币种          [必须有]
B-5  POST /admin/api/wallet/rate/log/page         汇率日志分页查询      [必须有]
B-6  POST /admin/api/wallet/currency/detail       币种详情查询          [应该有]
B-7  POST /admin/api/wallet/rate/refresh          手动刷新实时汇率      [待商榷]
```

### 接口说明

**B-1 币种列表分页查询** `[必须有]`
- 返回所有已配置币种，含20个字段（币种代码/名称/类型/国家/图标/符号/千分位/小数位/实时汇率/平台汇率/入款汇率/出款汇率/浮动%/阈值%/偏差值/精度/排序/状态/创建时间/更新时间）
- 支持按币种代码、类型、状态筛选
- 需求依据：币种配置需求文档3-3后台配置页面说明

**B-2 编辑币种配置** `[必须有]`
- 弹窗编辑：汇率浮动%（2位小数）、偏差阈值%（2位小数）、图标上传（SVG/PNG≤5KB）、状态等
- 不可编辑项：币种代码、类型（创建时确定）
- 需求依据：币种配置需求文档3-4操作行为描述

**B-3 启用/禁用币种** `[必须有]`
- 独立于编辑的状态切换操作（遵循项目 ToggleXxxStatus 惯例）
- 禁用后C端不可使用该币种，基准币种不可禁用
- 需求依据：币种配置需求文档3-3状态字段

**B-4 设置基准币种** `[必须有]`
- 一次性设置，提交后不可更改，推荐USDT
- 需二级确认弹窗，设置后按钮隐藏
- BSB锚定关系：10BSB = 1USDT
- 需求依据：币种配置需求文档3-2平台基准币种

**B-5 汇率日志分页查询** `[必须有]`
- 记录每次平台汇率变更的日志，按时间/币种代码筛选
- 字段：编号/更新时间/币种代码/实时汇率/平台汇率/入款汇率/出款汇率
- 需求依据：币种配置需求文档4-4汇率日志

**B-6 币种详情查询** `[应该有]`
- 编辑弹窗数据填充时使用，也可复用列表单条数据
- 工程规范：CRUD标配Get，但可以不独立暴露
- 11份评估中9份包含此接口

**B-7 手动刷新实时汇率** `[待商榷]`
- 手动触发拉取三方API实时汇率并更新偏差值
- 原型中有刷新倒计时展示（00:01:59）
- 争议点：是否需要独立接口，还是前端展示定时任务结果即可
- 11份评估中5份包含此接口

|------|------|
| 必须有 | 5 |
| 应该有 | 1 |
| 待商榷 | 1 |

## 二、多币种钱包模块 — C端接口

> 定位：核心业务层，C端用户操作资金的唯一入口
> 路由：gate-font → `/api/wallet/...` → RPC → ser-wallet
> 使用者：App/H5终端用户
> 鉴权：`/api`（需Token），`/open`（免登录，仅USDT充值地址获取可能用到）

### 接口列表

```
=== 钱包核心 ===
C-1  POST /api/wallet/balance                     钱包余额查询          [必须有]
C-2  POST /api/wallet/currencies                   可用币种列表          [必须有]

=== 充值 ===
C-3  POST /api/wallet/deposit/methods              充值方式列表          [必须有]
C-4  POST /api/wallet/deposit/create               创建充值订单          [必须有]
C-5  POST /api/wallet/deposit/usdt-address         USDT充值地址获取      [必须有]

=== 兑换 ===
C-6  POST /api/wallet/exchange/preview             兑换预览/汇率试算     [必须有]
C-7  POST /api/wallet/exchange/execute             执行兑换              [必须有]

=== 提现 ===
C-8  POST /api/wallet/withdraw/methods             提现方式列表          [必须有]
C-9  POST /api/wallet/withdraw/create              创建提现订单          [必须有]
C-10 POST /api/wallet/withdraw/account/get         获取提现账户信息      [必须有]
C-11 POST /api/wallet/withdraw/account/save        保存提现账户信息      [必须有]

=== 记录 ===
C-12 POST /api/wallet/record/list                  交易记录列表          [必须有]
C-13 POST /api/wallet/record/detail                交易记录详情          [必须有]

=== 辅助 ===
C-14 POST /api/wallet/deposit/bonus-list           可用奖金活动列表      [应该有]
C-15 POST /api/wallet/deposit/status               充值订单状态查询      [应该有]
```

### 接口说明

#### 钱包核心

**C-1 钱包余额查询** `[必须有]`
- 查询当前币种下各子钱包余额：中心钱包 + 奖励钱包 + 资金钱包(合计)
- 个人中心页面核心数据展示
- 主播钱包/代理钱包当前预留，数据模型有但不展示
- 内部调用币种配置获取精度和展示规则（同服务函数调用，非RPC）
- 需求依据：钱包需求文档4-1钱包总体设计

**C-2 可用币种列表** `[必须有]`
- 返回当前用户可用的已启用币种列表（BSB/VND/IDR/THB），含各币种余额概览
- 用于币种切换下拉，含当前选中币种标记
- 内部调用币种配置获取启用币种（同服务函数调用）
- 需求依据：钱包需求文档4-2币种切换规则

#### 充值

**C-3 充值方式列表** `[必须有]`
- 按当前币种返回可用充值方式（USDT/银行转账/电子钱包），含方式名称/图标/金额档位/金额范围/汇率
- 数据来源于财务模块支付配置 → 内部RPC调用ser-finance获取
- 不同币种展示不同的充值方式（如BSB仅USDT，VND有银行+电子钱包）
- 需求依据：钱包需求文档5-1充值方式总览

**C-4 创建充值订单** `[必须有]`
- 选择充值方式+金额(+可选奖金活动) → 创建平台订单(C+16位订单号)
- 内置重复转账拦截逻辑：同UserID+同币种+同金额→1-2笔软拦截警告、≥3笔硬拦截
- 银行转账/电子钱包 → 调用财务模块匹配通道 → 返回三方跳转URL
- USDT → 返回收款地址（平台自承载）
- 需求依据：钱包需求文档5-2~5-5

**C-5 USDT充值地址获取** `[必须有]`
- 返回USDT收款地址、QR码数据、网络类型（TRC20默认/ERC20/BEP20）
- 平台承载页场景，游客也可访问（可能走/open路由）
- 需求依据：钱包需求文档5-2 USDT充值

#### 兑换

**C-6 兑换预览/汇率试算** `[必须有]`
- 输入法币金额 → 返回：兑换获得BSB + 赠送BSB(10%) + 实际到账BSB + 使用汇率
- 统一展示口径：1BSB = X 法币
- 仅法币/USDT → BSB单向兑换
- 需求依据：钱包需求文档6兑换模块

**C-7 执行兑换** `[必须有]`
- 法币余额 → BSB，扣法币中心钱包 + 加BSB中心钱包 + 赠送部分入奖励钱包
- 赠送部分标记：需完成N倍流水后可提现
- 全程在钱包模块内完成，不需要调用财务模块
- 需求依据：钱包需求文档6兑换模块

#### 提现

**C-8 提现方式列表** `[必须有]`
- 按当前币种+KYC状态返回可用提现方式+限额+手续费+T+N到账时间
- KYC过滤规则：未认证仅USDT，认证后全渠道
- 提现币种路由规则：法币充→法币提，USDT充→USDT提，混合→法币提
- 内部调用 ser-kyc 查询认证状态、调用 ser-finance 获取提现方式配置
- 需求依据：钱包需求文档7-1~7-5

**C-9 创建提现订单** `[必须有]`
- 校验KYC→校验限额(单笔/单日)→校验余额→冻结余额→创建提现订单(T+16位)→状态=待审核
- 冻结余额通过内部FreezeBalance操作
- 需求依据：钱包需求文档7-1提现模块

**C-10 获取提现账户信息** `[必须有]`
- 查询用户已保存的提现账户（银行卡/电子钱包/USDT地址）
- 首次为空需填写，已保存则返回只读信息（用户不可自行修改）
- 需求依据：钱包需求文档7-2~7-4

**C-11 保存提现账户信息** `[必须有]`
- 首次提现时填写：银行名称/账号/持有人/网点 或 电子钱包类型/账号 或 USDT链地址
- 持有人字段须与KYC实名一致（自动填充灰显，忽略大小写和空格比对）
- 保存后不可自行修改
- 需求依据：钱包需求文档7-2~7-4

#### 记录

**C-12 交易记录列表** `[必须有]`
- 按类型Tab（充值/提现/兑换/奖励）分页查询，严格绑定当前币种不支持跨币种
- 列表含状态色标：已完成/审核中-橙色/失败-红色
- 需求依据：钱包需求文档8-1~8-3

**C-13 交易记录详情** `[必须有]`
- 订单号/金额明细/状态/时间/方式/失败原因
- 失败记录含"更换方式"链接
- 需求依据：钱包需求文档8-4详情页

#### 辅助

**C-14 可用奖金活动列表** `[应该有]`
- 充值时可选的额外奖金活动（+50%等），含最低充值门槛、奖金比例、N倍流水要求
- 单笔互斥（只能选一个）
- 数据源待评审确认：可能来自运营活动模块（钱包做聚合/转发），也可能钱包自管
- 需求依据：钱包需求文档5-6额外奖金选择规则

**C-15 充值订单状态查询** `[应该有]`
- 充值提交后前端轮询等待支付结果
- 也可合并到前端通过WebSocket接收推送，是否独立接口视前端方案
- 需求依据：原型-充值进行中弹窗

### 关于重复转账检测和兑换预览的归并说明

11份评估中有部分将以下功能列为独立接口，整合后认为应合并：

| 功能 | 归并方案 | 理由 |
|------|---------|------|
| 重复转账检测 | 合并到 C-4 创建充值订单内部逻辑 | 检测+创建是原子操作，拆分会导致竞态 |
| 币种切换 | 合并到 C-2 返回值+前端本地状态 | 切换币种是前端状态变更，无需独立写接口 |
| 提现限额查询 | 合并到 C-8 提现方式列表返回 | 限额信息与方式配置同源，一次返回减少调用 |

## 三、RPC暴露接口 — 供其他模块调用

> 定位：ser-wallet 的核心能力输出，余额只有钱包能改
> 协议：Kitex Thrift（TTHeader），通过ETCD服务发现
> 调用方：ser-finance(财务管理)、未来的投注/直播/活动模块
> IDL位置：common/idl/ser-wallet/service.thrift

### 接口列表

```
=== 币种配置域（纯查询，4个） ===
R-1  GetCurrencyList              获取已启用币种列表        [必须有]
R-2  GetExchangeRate              获取指定币种汇率          [必须有]
R-3  GetBaseCurrency              获取基准币种信息          [必须有]
R-4  GetCurrencyByCode            按币种代码查详情          [应该有]

=== 钱包操作域（写操作为主，8个） ===
R-5  GetUserWalletBalance         查询用户钱包余额          [必须有]
R-6  CreditDeposit                充值入账                  [必须有]
R-7  FreezeForWithdrawal          提现冻结余额              [必须有]
R-8  UnfreezeForWithdrawal        提现解冻退回              [必须有]
R-9  ConfirmWithdrawalDeduct      提现确认扣款              [必须有]
R-10 ManualCredit                 人工加款                  [必须有]
R-11 ManualDebit                  人工减款                  [必须有]
R-12 CreditCorrection            补单入账                  [必须有]
```

### 接口说明

#### 币种配置域（供财务/其他模块查询基础数据）

**R-1 GetCurrencyList** `[必须有]`
- 返回所有启用状态的币种列表（代码/名称/类型/符号/精度/图标/排序/状态）
- 调用方：ser-finance（所有B端页面的币种下拉筛选、通道配置、支付配置）
- 调用频率：中频（B端页面加载）
- 11份评估全部包含，无争议

**R-2 GetExchangeRate** `[必须有]`
- 返回指定币种的平台汇率/入款汇率/出款汇率
- 调用方：ser-finance（金额换算、统计报表基准换算）
- 充值用入款汇率、提现用出款汇率、统计用平台汇率
- 11份评估全部包含，无争议

**R-3 GetBaseCurrency** `[必须有]`
- 返回基准币种信息（USDT）+ BSB锚定比例（10BSB=1USDT）
- 调用方：ser-finance（统计报表换算基准）、未来的结算模块
- 11份评估中8份包含

**R-4 GetCurrencyByCode** `[应该有]`
- 按币种代码查询单个币种完整配置（含精度、千分位规则等）
- 轻量级接口，供其他服务精确查询单个币种信息
- 11份评估中6份包含，部分认为可通过R-1过滤替代

#### 钱包操作域（供财务模块调用，所有涉及余额变更的操作）

**R-5 GetUserWalletBalance** `[必须有]`
- 查询指定用户+币种的各子钱包余额（中心/奖励/主播/代理，含冻结金额）
- 调用方：ser-finance（人工加减款弹窗展示余额）
- 与C端余额查询区别：返回更多内部字段（冻结金额等），面向内部服务
- 11份评估全部包含，无争议

**R-6 CreditDeposit** `[必须有]`
- 充值成功后入账：充值金额→中心钱包，奖金→奖励钱包（分别入账）
- 调用方：ser-finance（充值回调验签通过后）
- 幂等要求：基于订单号去重，同一订单号重复调用不会重复入账
- 必须生成流水记录（金额/钱包类型/操作类型/关联订单号/操作前后余额）
- 11份评估全部包含，无争议

**R-7 FreezeForWithdrawal** `[必须有]`
- 冻结指定金额：可用余额→冻结余额，冻结期间不可使用
- 调用方：提现创建时内部调用 或 ser-finance调用
- 必须事务保障（GORM事务+悲观锁 SELECT FOR UPDATE）
- 11份评估全部包含，无争议

**R-8 UnfreezeForWithdrawal** `[必须有]`
- 解冻退回：冻结余额→可用余额
- 触发场景：风控审核驳回 / 财务审核驳回 / 三方出款失败
- 调用方：ser-finance
- 幂等要求：同一提现订单号重复解冻不重复操作
- 11份评估全部包含，无争议

**R-9 ConfirmWithdrawalDeduct** `[必须有]`
- 终态扣款：从冻结余额中最终扣除（不是从可用余额扣）
- 触发场景：三方出款成功回调后
- 调用方：ser-finance
- 11份评估全部包含，无争议

**R-10 ManualCredit** `[必须有]`
- 人工加款：按指定子钱包类型（中心/奖励/主播/代理4种）分别加款
- 触发场景：财务管理→人工加款审核通过后
- 调用方：ser-finance
- 订单号：A+16位
- 11份评估全部包含，无争议

**R-11 ManualDebit** `[必须有]`
- 人工减款：按指定子钱包类型分别减款
- 业务规则：余额不足时实际扣款=当前余额（不报错，扣到0）
- 触发场景：财务管理→人工减款审核通过后
- 调用方：ser-finance
- 订单号：M+16位
- 11份评估全部包含，无争议

**R-12 CreditCorrection** `[必须有]`
- 补单入账：充值漏入账的补录，补单金额→中心钱包 + 补单赠送→奖励钱包
- 触发场景：财务管理→补单审核通过后
- 调用方：ser-finance
- 订单号：B+16位

## 四、RPC调用依赖 — 我调用其他模块

> 定位：ser-wallet 作为 RPC 客户端，主动调用其他服务获取信息
> 特征：以查询为主，不改别人的数据
> 注册位置：common/rpc/rpc_client.go

### 接口列表

```
EXT-1  ser-finance → GetAvailableDepositMethods       获取充值方式配置    [必须有/阻塞]
EXT-2  ser-finance → GetAvailableWithdrawalMethods     获取提现方式配置    [必须有/阻塞]
EXT-3  ser-kyc    → KycQuery                          查询KYC认证状态     [必须有/可降级]
EXT-4  ser-blog   → AddActLog                         审计日志记录        [必须有/非阻塞]
EXT-5  ser-buser  → QueryUserByIds                    查询操作人姓名      [应该有/非阻塞]
```

### 接口说明

**EXT-1 获取充值方式配置** `[必须有/阻塞]`
- 调用 ser-finance 获取当前币种可用的充值方式+金额档位+费率
- C端充值页面加载时触发（C-3接口的数据来源）
- 此接口不可用时充值页面无法渲染 → **阻塞级依赖**
- RPC客户端状态：**需新建**（当前rpc_client.go中无finance客户端）
- 编码阶段策略：**mock**（ser-finance未开发完，需mock返回值）

**EXT-2 获取提现方式配置** `[必须有/阻塞]`
- 调用 ser-finance 获取当前币种可用的提现方式+限额规则
- C端提现页面加载时触发（C-8接口的数据来源）
- 此接口不可用时提现页面无法渲染 → **阻塞级依赖**
- RPC客户端状态：**需新建**（同EXT-1，共享finance客户端）
- 编码阶段策略：**mock**

**EXT-3 KYC认证状态查询** `[必须有/可降级]`
- 调用 ser-kyc 查询用户KYC认证状态，决定可用提现渠道（未认证仅USDT）
- 提现页面加载时触发（C-8接口内部逻辑）
- IDL方法：`KycQuery(KycQueryReq{optional i64 uid})` → 返回 `UserKycInfo{status, name...}`
- RPC客户端状态：**已存在**（rpc_client.go中已注册KycClient）
- 此接口不可用时可降级为"仅展示USDT提现方式"

**EXT-4 审计日志记录** `[必须有/非阻塞]`
- 所有写操作调用 ser-blog.AddActLog 记录审计日志
- `oneway void` — 发射即忘，不等返回，不影响业务主流程
- 项目强制规范：所有服务的所有B端写操作都必须调用
- 参数：backend/model/page/action/isSuccess
- RPC客户端状态：**已存在**（rpc_client.go中已注册BLogClient）
- 预估调用点：币种编辑、启禁用、基准设置、充值创建、提现创建、兑换执行等 约14个

**EXT-5 查询操作人姓名** `[应该有/非阻塞]`
- 调用 ser-buser.QueryUserByIds 解析B端列表中操作人ID对应的姓名
- `QueryUserByIds(RpcSysUserReq{required list<i64> ids})` → `map<i64, string>`
- RPC客户端状态：**已存在**（rpc_client.go中已注册BUserClient）
- 用于：币种配置列表、汇率日志等B端页面展示操作人名称

| tmp-3 修正 | 49~56 | 34 | tmp-3统计维度最宽（含RPC调用、定时任务、预留） |

### 数量差异核心原因

本整合版比部分方案数量少，因为：

1. **合并了可合并的接口**：重复转账检测→合并到创建充值、提现限额→合并到提现方式列表、币种切换→合并到币种列表
2. **剥离了RPC调用依赖**：EXT-1~5不是"我的新接口"，是"我调别人的接口"，是对接工作量而非接口定义工作量
3. **剥离了定时任务**：T-1是后台任务不是接口
4. **不计入未来预留接口**：投注扣款、直播打赏等当前需求无明确场景，IDL预留签名即可，不计入实现接口数
5. **不计入他人负责的回调**：充值回调归财务模块处理，不计入钱包接口

### 一句话总结

> **我方需定义实现的接口 = 34个**（C端15 + B端7 + RPC暴露12），其中29个核心必须、4个应该有、1个待商榷。配套需对接5个外部RPC调用 + 1个定时任务。规模与工程内 ser-user(35方法) 相当，在合理范围内。
