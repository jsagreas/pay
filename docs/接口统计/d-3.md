# ser-wallet 接口统计清单（整合版）

> **文档定位**：集成 11 份接口评估文档（c-1~c-5、d-1~d-3、tmp-1~tmp-3）+ IDL设计/d-3.md + 链路关系/d-3.md + rpc暴露/d-3.md + rpc调用/d-3.md 的全量分析成果，输出一份**去重、分类、定性**的最终接口清单。
>
> **使用方式**：编码前的"菜单"——打开这份文档，你就知道 ser-wallet 一共要做多少个接口、每个接口做什么、优先级如何、归谁调用。
>
> **分类原则**：
> - 按**业务域**分区（币种配置 / 钱包核心 / 充值 / 提现 / 兑换 / 记录 / 对外RPC / 出站依赖 / 定时任务）
> - 每个接口标注**定性**：🔴 必须 / 🟡 应该有 / 🔵 商榷
> - HTTP 路径遵循工程规范（B端 `/admin/api/wallet/...`，C端 `/api/wallet/...`），实际路由在 gate 层定义
>
> **定性标准**：
> - **🔴 必须**：缺了无法完成核心业务链路，或工程规范强制要求
> - **🟡 应该有**：不影响核心链路但属于完整产品体验所需，或工程规范推荐
> - **🔵 商榷**：需求未明确定型、首期可不做、或存在替代方案需评审确认
>
> **权威基准**：service.thrift 的 55 个 RPC 方法签名（IDL设计/d-3.md）

## 一、币种配置模块 — B端管理接口（9个）

> 调用路径：管理员浏览器 → gate-back HTTP → RPC → ser-wallet
> 对应 IDL 文件：currency.thrift
> 对应 service 层：currency_service.go

### 接口列表

```
POST /admin/api/wallet/currency/page            ← B01 币种列表分页
POST /admin/api/wallet/currency/create          ← B02 新增币种
POST /admin/api/wallet/currency/update          ← B03 编辑币种
POST /admin/api/wallet/currency/status          ← B04 启用/禁用币种
POST /admin/api/wallet/currency/base/get        ← B05 查询基准币种
POST /admin/api/wallet/currency/base/set        ← B06 设置基准币种
POST /admin/api/wallet/currency/rate/list       ← B07 汇率展示列表
POST /admin/api/wallet/currency/rate-log/page   ← B08 汇率日志分页
POST /admin/api/wallet/currency/oper-log/page   ← B09 操作日志分页
```

### 逐接口说明

**B01 — 币种列表分页** 🔴 必须
- RPC 方法：`PageCurrency`
- 作用：B端币种管理首页，分页展示所有币种及其配置
- 筛选项：币种类型（法币/加密/平台币）、状态（启用/禁用）
- 返回字段：币种编码、名称、类型、符号、图标URL、小数位、千分位规则、汇率浮动%、偏差阈值%、状态、创建时间
- 11个评估文档均确认此接口为核心必须

**B02 — 新增币种** 🔴 必须
- RPC 方法：`CreateCurrency`
- 作用：管理员新增一种币种配置
- 核心逻辑：编码唯一性校验 → 图标上传（调 ser-s3，SVG/PNG ≤ 5KB）→ 写入DB → 审计日志（调 ser-blog）
- 关键约束：币种编码创建后不可修改（需求文档 3-3 明确要求）

**B03 — 编辑币种** 🔴 必须
- RPC 方法：`UpdateCurrency`
- 作用：修改币种的名称、图标、汇率浮动%、偏差阈值%、小数精度、排序等
- 不可编辑字段：币种编码、币种类型
- 附带操作：有新图标时调 ser-s3 上传 + 调 ser-blog 审计日志

**B04 — 启用/禁用币种** 🔴 必须
- RPC 方法：`StatusCurrency`
- 作用：切换币种启用/禁用状态
- 业务影响：禁用后C端 `GetEnabledCurrencyList` 不再返回该币种，用户无法选择该币种操作
- 校验项（商榷）：禁用前是否需要检查有用户持有该币种余额

**B05 — 查询基准币种** 🔴 必须
- RPC 方法：`GetBaseCurrency`
- 作用：查看当前平台的基准币种设置（USDT，10 BSB = 1 USDT）
- 返回：基准币种编码、名称、BSB锚定比例；或"未设置"提示

**B06 — 设置基准币种** 🔴 必须
- RPC 方法：`SetBaseCurrency`
- 作用：一次性设置平台基准币种（设置后不可更改）
- 关键约束：已设置 → 拒绝修改；未设置 → 校验币种存在且启用 → 写入 → 审计日志
- 需二级确认弹窗（前端逻辑）

**B07 — 汇率展示列表** 🔴 必须
- RPC 方法：`GetRateList`
- 作用：展示所有启用币种的汇率一览表
- 展示项（每行）：币种编码、实时汇率（三方均值）、平台汇率（当前生效值）、入款汇率、出款汇率、偏差值（|实时-平台|/实时 ×100）
- 计算逻辑：入款汇率 = 实时 × (1+浮动%)，出款汇率 = 实时 × (1-浮动%)

**B08 — 汇率日志分页** 🟡 应该有
- RPC 方法：`PageRateLog`
- 作用：查看汇率自动更新的变更历史
- 数据来源：定时任务检测偏差超阈值时自动写入（非人工操作）
- 筛选项：币种、时间范围
- 说明：数据产出依赖定时任务（T01），但查询接口本身是展示需求

**B09 — 操作日志分页** 🟡 应该有
- RPC 方法：`PageOperationLog`
- 作用：查看币种配置的操作审计记录（谁在什么时间做了什么修改）
- 两种实现方案：① 调用 ser-blog 按模块过滤查询 ② 本地维护操作日志表
- 评估文档 c-1/c-3/c-5 确认为应该有

## 二、币种配置模块 — RPC暴露接口（6个）

> 调用路径：其他服务（财务/游戏/统计） → Kitex RPC → ser-wallet
> 对应 IDL 文件：currency.thrift（RPC 部分）
> 这些接口也被 ser-wallet 自身的钱包模块进程内调用

### 接口列表

```
RPC  GetCurrencyInfo         ← R01 获取单个币种完整信息
RPC  GetEnabledCurrencyList  ← R02 获取所有启用币种列表
RPC  GetExchangeRate         ← R03 获取指定币种汇率
RPC  GetBaseCurrencyInfo     ← R04 获取基准币种信息
RPC  BatchGetExchangeRates   ← R05 批量获取多币种汇率
RPC  ValidateCurrency        ← R06 校验币种有效性
```

### 逐接口说明

**R01 — GetCurrencyInfo** 🔴 必须
- 作用：根据币种编码获取完整配置（编码、名称、类型、图标、小数位、千分位、浮动%、阈值%、状态）
- 调用方：钱包模块（充值/提现/兑换时获取币种属性）、财务（通道配置关联币种）、游戏（场馆币种转换）
- 返回缓存策略：建议 Redis 缓存，币种配置变更时主动失效

**R02 — GetEnabledCurrencyList** 🔴 必须
- 作用：返回所有启用状态的币种列表
- 调用方：钱包C端（币种切换下拉）、财务（支付配置币种选择）
- 返回字段精简：编码、名称、类型、图标URL、排序号

**R03 — GetExchangeRate** 🔴 必须
- 作用：获取指定币种的当前汇率（实时、平台、入款、出款四种）
- 调用方：钱包（兑换预览/充值展示/提现计算）、财务（统计报表换算）
- 关键逻辑：入款汇率 = 实时 × (1+浮动%)，出款汇率 = 实时 × (1-浮动%)

**R04 — GetBaseCurrencyInfo** 🔴 必须
- 作用：获取平台基准币种设置（USDT/BSB锚定信息）
- 调用方：财务（报表以基准币种统一换算）、活动（奖金计算基准）

**R05 — BatchGetExchangeRates** 🟡 应该有
- 作用：一次获取多个币种的汇率，减少多次单查的网络开销
- 调用方：钱包（余额概览需将各币种余额折算为统一展示）、财务（批量统计）
- 返回：`map<currencyCode, RateInfo>`

**R06 — ValidateCurrency** 🟡 应该有
- 作用：轻量级校验——币种编码是否存在且启用
- 调用方：钱包/财务（下单前快速校验，避免全量查询）
- 返回：bool + 错误码
- 说明：部分评估文档认为可合并到 R01 逻辑中，但独立接口更轻量

## 三、钱包核心模块（4个）

> C端路径：gate-font → RPC → ser-wallet
> B端路径：gate-back → RPC → ser-wallet
> 对应 IDL 文件：wallet.thrift

### 接口列表

```
POST /api/wallet/balance              ← F01 查询钱包余额（C端）
POST /api/wallet/currency-list        ← F02 获取用户可用币种列表（C端）
POST /api/wallet/info                 ← F03 获取用户钱包完整信息（C端）
POST /admin/api/wallet/user/info      ← B10 B端查询用户钱包信息（B端）
```

### 逐接口说明

**F01 — 查询钱包余额** 🔴 必须
- RPC 方法：`GetWalletBalance`
- 作用：返回当前币种下用户各子钱包余额
- 返回结构：资金钱包（中心+奖励合并展示）、中心钱包余额、奖励钱包余额、主播钱包余额（主播用户才有）、代理钱包余额（代理用户才有）
- 钱包类型：1=中心 2=奖励 3=主播 4=代理 5=场馆（预留）
- 使用场景：钱包首页、个人中心余额展示、充值/提现/兑换前的余额确认

**F02 — 获取用户可用币种列表** 🔴 必须
- RPC 方法：`GetUserCurrencyList`
- 作用：返回用户维度的可用币种列表，附带每个币种的合并余额（资金钱包余额），用于币种切换下拉
- 与 R02 的区别：R02 是系统级（所有启用币种），F02 是用户级（含余额信息、当前选中标记）

**F03 — 获取用户钱包完整信息** 🟡 应该有
- RPC 方法：`GetUserWalletInfo`
- 作用：聚合展示用户所有币种的钱包概况（适用于个人中心的钱包概览区）
- 说明：多个评估文档建议此接口可与 F01 合并返回；独立存在的好处是个人中心和钱包首页调用不同接口减少数据量
- 商榷点：是否真的需要独立接口，还是通过 F01 + F02 组合实现

**B10 — B端查询用户钱包信息** 🔴 必须
- RPC 方法：`AdminGetWalletInfo`
- 作用：运营后台查看指定用户的钱包详情（所有币种 × 所有子钱包的余额）
- 使用场景：客服查询用户余额、人工加减款前确认余额、风控查看用户资金状况
- 入参：userId（管理员指定查询哪个用户）

## 四、充值模块（4个）

> 调用路径：用户APP → gate-font HTTP → RPC → ser-wallet
> 对应 IDL 文件：deposit.thrift
> 对应 service 层：deposit_service.go

### 接口列表

```
POST /api/wallet/deposit/methods      ← F03 获取充值方式列表
POST /api/wallet/deposit/create       ← F04 发起充值
POST /api/wallet/deposit/status       ← F05 查询充值订单状态
POST /api/wallet/deposit/bonus        ← F06 获取奖金活动列表
```

### 逐接口说明

**F03（充值） — 获取充值方式列表** 🔴 必须
- RPC 方法：`GetDepositMethods`
- 作用：用户打开充值页时，按当前币种展示可用的充值方式
- 核心逻辑：调用 ser-finance 获取支付配置（充值方式 + 金额档位 + 限额 + 手续费说明）→ 排序 → 组装C端数据
- **出站依赖**：ser-finance.GetDepositMethods ← 这是跨模块RPC调用
- 按币种展示不同方式：
  - BSB：USDT、银行转账、电子钱包
  - VND：银行转账（越南银行列表）
  - IDR：电子钱包扫码（推荐）、印尼银行
  - THB：电子钱包扫码（推荐）、泰国银行

**F04（充值） — 发起充值** 🔴 必须
- RPC 方法：`CreateDeposit`
- 作用：用户确认金额后创建充值订单
- 核心链路：参数校验 → 币种校验 → 重复充值检测 → 调 ser-finance 创建订单（含通道匹配）→ 本地记录 → 审计日志 → 返回支付信息
- **出站依赖**：ser-finance.CreateDepositOrder ← 关键路径，失败则充值无法继续
- 返回信息（按支付类型）：URL跳转地址 / USDT收款地址+二维码 / 扫码支付二维码
- 重复充值拦截规则（内嵌逻辑）：同用户+同币种+同金额近期订单 → 1-2笔弹警告 → ≥3笔拒绝
- 说明：部分评估文档将"重复检测"拆为独立接口（CheckDuplicateDeposit），此处合并为 CreateDeposit 的内部逻辑（见商榷区 X01）

**F05（充值） — 查询充值订单状态** 🟡 应该有
- RPC 方法：`GetDepositStatus`
- 作用：用户从三方支付页返回后，前端轮询订单状态（支付中/成功/失败/已失效）
- **出站依赖**：ser-finance.GetOrderStatus
- 轮询建议：3-5秒一次，超时后展示"请稍后查看充值记录"

**F06（充值） — 获取奖金活动列表** 🟡 应该有
- RPC 方法：`GetBonusActivityList`
- 作用：充值页面展示可选的额外奖金活动（如"首充加送50%"）
- **出站依赖**：活动服务.GetAvailableBonusActivities ← 活动模块尚未开发
- 业务规则：单笔充值只能选一个活动（互斥）、有最低充值门槛、赠送部分需满N倍流水才可提现
- 说明：活动系统未开发前可返回空列表，不影响核心充值链路

## 五、提现模块（7个）

> 调用路径：用户APP → gate-font HTTP → RPC → ser-wallet
> 对应 IDL 文件：withdraw.thrift
> 对应 service 层：withdraw_service.go
> 说明：提现是 C端接口数量最多的域，涉及 KYC 校验 + 账户管理 + 限额 + 冻结

### 接口列表

```
POST /api/wallet/withdraw/methods          ← F08 获取提现方式列表
POST /api/wallet/withdraw/limit            ← F12 获取提现限额信息
POST /api/wallet/withdraw/create           ← F11 发起提现
POST /api/wallet/withdraw/account/list     ← F09 获取提现账户列表
POST /api/wallet/withdraw/account/create   ← F10 新增提现账户
POST /api/wallet/withdraw/account/update   ← F10b 编辑提现账户
POST /api/wallet/withdraw/account/delete   ← F10c 删除提现账户
```

### 逐接口说明

**F08 — 获取提现方式列表** 🔴 必须
- RPC 方法：`GetWithdrawMethods`
- 作用：用户打开提现页时，展示当前币种可用的提现方式
- 核心逻辑：
  1. 查询用户充值来源（本地DB）
  2. 调 ser-finance 获取全量提现方式
  3. 按充值来源规则二次过滤：纯法币充→法币提，纯USDT充→USDT提，混合充→法币提
- **出站依赖**：ser-finance.GetWithdrawMethods
- 关键逻辑："充值来源决定提现方式"过滤在钱包侧完成，不由财务负责

**F12 — 获取提现限额信息** 🔴 必须
- RPC 方法：`GetWithdrawLimit`
- 作用：提现页面展示限额提示（单笔最低/最高、单日限额、今日已用、手续费率、T+N到账天数）
- 核心逻辑：查提现限额配置 + 查今日已提现累计 + 查余额和冻结 → 计算剩余可提
- 说明：部分评估文档建议合并到 F08 返回值中，但独立接口更清晰（限额可能动态变化，需独立刷新）

**F11 — 发起提现** 🔴 必须
- RPC 方法：`CreateWithdraw`
- 作用：用户确认提现并提交（所有 C端接口中链路最长、逻辑最复杂的接口）
- 核心链路（7个步骤）：
  1. KYC校验 → 调 ser-kyc.KycQuery（未认证+非USDT方式=拒绝）
  2. 充值来源校验 → 本地DB
  3. 限额校验 → 本地DB（单笔+单日）
  4. 余额校验 → 本地DB（可用余额 ≥ 提现金额+手续费）
  5. 手续费计算 → 本地配置（固定 or 比例）
  6. 冻结余额 → 本地DB事务（可用余额减少，冻结增加）
  7. 调 ser-finance.CreateWithdrawOrder → 创建财务提现订单进入审核
- **出站依赖**：ser-kyc、ser-finance、ser-blog
- 关键约束：步骤6必须在步骤7之前；步骤7失败必须回滚步骤6（解冻）

**F09 — 获取提现账户列表** 🔴 必须
- RPC 方法：`GetWithdrawAccounts`
- 作用：展示用户已绑定的提现账户（银行卡/电子钱包/USDT地址）
- 入参：userId + 账户类型筛选
- 返回：账户ID、类型、银行名称/钱包类型/链类型、账号/地址、持有人姓名

**F10 — 新增提现账户** 🔴 必须
- RPC 方法：`CreateWithdrawAccount`
- 作用：用户首次提现时填写收款信息并保存
- 三种账户类型的必填字段：

## 六、兑换模块（2个）

> 调用路径：用户APP → gate-font HTTP → RPC → ser-wallet
> 对应 IDL 文件：exchange.thrift
> 对应 service 层：exchange_service.go
> 特点：最简单的C端模块，全程不涉及跨模块RPC调用，纯进程内事务

### 接口列表

```
POST /api/wallet/exchange/preview     ← F15 兑换预览
POST /api/wallet/exchange/create      ← F16 执行兑换
```

### 逐接口说明

**F15 — 兑换预览** 🔴 必须
- RPC 方法：`PreviewExchange`
- 作用：用户输入法币金额后，实时展示兑换结果预览
- 展示项：输入金额（法币）→ 使用汇率 → 兑换获得BSB → 赠送部分（如10%）→ 实际到账BSB
- 计算逻辑：BSB到账 = 法币金额 × 入款汇率 × (1+赠送比例)；赠送部分需X倍流水才可提现
- 不涉及跨模块调用：汇率通过进程内调用币种配置获取

**F16 — 执行兑换** 🔴 必须
- RPC 方法：`CreateExchange`
- 作用：确认并执行法币→BSB单向兑换
- 核心逻辑：法币余额校验 → 锁定汇率快照 → DB事务（法币中心钱包扣 + BSB中心钱包入 + 赠送部分进BSB奖励钱包）→ 记录流水 → 审计日志
- 关键约束：双币种操作必须在同一个DB事务中完成（两钱包同时写），防止部分成功
- 不涉及跨模块RPC调用——这是C端所有资金操作中链路最短、最自治的接口

### 逐接口说明

#### 资金核心操作（首期必须，ser-finance 直接调用）

**E01 — WalletCredit 通用入账** 🔴 必须
- 作用：往指定用户的指定币种、指定钱包类型增加余额
- 调用方：ser-finance（充值成功回调、充值补单审核通过）
- 入参：userId, currencyCode, walletType, amount, orderNo, bizType, remark
- 幂等约束：同一 orderNo 只处理一次（通过 DB 唯一索引 or Redis SetNX 防重）

**E02 — WalletDebit 通用扣款** 🔴 必须
- 作用：从指定用户的指定币种、指定钱包类型扣减余额
- 调用方：ser-finance（提现出款成功后确认扣款——从冻结金额中正式扣除）
- 余额校验：扣款金额不能超过可用余额（含冻结场景下扣冻结金额）

**E03 — WalletFreeze 冻结余额** 🔴 必须
- 作用：将用户可用余额的一部分标记为"冻结"，不可用于其他操作
- 调用方：ser-finance（用户提交提现申请时冻结对应金额）
- 效果：可用余额减少，冻结金额增加，总余额不变

**E04 — WalletUnfreeze 解冻余额** 🔴 必须
- 作用：将冻结金额恢复为可用余额
- 调用方：ser-finance（提现被任一级审核驳回 / 出款失败时解冻）
- 关联 E03：冻结和解冻通过 orderNo 关联

**E05 — ManualCredit 人工加款** 🔴 必须
- 作用：管理员审批通过后，往指定用户的指定钱包类型加款
- 调用方：ser-finance（人工加款审核通过后）
- 特殊点：支持指定钱包类型（中心/奖励/主播/代理），不限于中心钱包

**E06 — ManualDebit 人工减款** 🔴 必须
- 作用：管理员审批通过后，从指定用户的指定钱包类型扣款
- 调用方：ser-finance（人工减款审核通过后）
- 特殊逻辑：余额不足时扣实际余额（不拒绝，但记录实际扣减金额）

#### 余额查询

**E07 — RpcGetBalance 查询指定币种余额** 🔴 必须
