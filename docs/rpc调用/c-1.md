## 3. 关键发现：ser-finance IDL 不存在

**这是整个出向依赖分析中最重要的结论。**

扫描 `common/idl/` 目录，当前存在的所有服务 IDL：

```
ser-app  ser-bag  ser-bcom  ser-blog  ser-buser  ser-contentcenter
ser-cron  ser-exp  ser-i18n  ser-ip  ser-item  ser-kyc  ser-live
ser-mall  ser-recommend  ser-rtc  ser-s3  ser-shortvideos  ser-socket  ser-user
```

**没有 `ser-finance/` 目录。** 同时 `common/rpc/rpc_client.go` 中也没有 `FinanceClient()`。

这意味着：
- 我们 4 个 P0 级出向依赖（O1-O4）全部指向一个 **IDL 尚未定义的服务**
- 在财务 IDL 就绪之前，充值和提现的 C 端流程 **无法真正联调**
- 开发阶段必须 **Mock 财务 RPC 返回值**，先把钱包侧逻辑走通
- **最紧迫的跨团队协作任务**：与财务同事尽快对齐这 4 个接口的签名

## 4. 依赖一：ser-finance — 充值方式查询（O1）

### 4.1 基本信息

| 维度 | 内容 |
|------|------|
| 被调用服务 | ser-finance（财务管理模块，他人负责） |
| 方法名（拟定） | `GetRechargeMethodList` |
| IDL 状态 | **不存在**，需与财务团队联合定义 |
| 优先级 | **P0** — 缺失则充值页白屏 |

### 4.2 触发场景

C端用户进入充值页面 → 钱包需要展示当前币种的可用充值方式列表。

### 4.3 数据流

```
发送：
  currencyCode: string    // 当前选中币种（如 "VND"、"IDR"、"THB"）

期望返回：
  list<RechargeMethodItem>
    ├── methodType     i32       // 方式类型 1=USDT 2=银行转账 3=电子钱包
    ├── methodName     string    // 方式名称
    ├── icon           string    // 方式图标 URL
    ├── minAmount      string    // 最小充值金额
    ├── maxAmount      string    // 最大充值金额
    ├── presetAmounts  list<string>  // 预设金额档位
    ├── feeRate        string    // 手续费率（展示用）
    ├── isRecommend    bool      // 是否推荐
    └── sortOrder      i32       // 排序
```

### 4.4 产品原型证据

| 原型截图 | 展示的充值方式 |
|---------|-------------|
| BSB 充值页 | USDT（推荐）、越南银行转账、电子钱包扫码 |
| IDR 充值页 | 电子钱包扫码（推荐）、印尼银行转账 |
| THB 充值页 | PromptPay 扫码、泰国银行转账 |

**不同币种的充值方式列表不同**——这些数据来自财务后台的"支付配置"模块（代收通道+支付方式关联），钱包只负责展示。

## 6. 依赖三：ser-finance — 提现限额费率配置（O3）

### 6.1 基本信息

| 维度 | 内容 |
|------|------|
| 被调用服务 | ser-finance |
| 方法名（拟定） | `GetWithdrawConfig` |
| IDL 状态 | **不存在** |
| 优先级 | **P0** — 缺失则无法展示提现须知、无法做限额校验 |

### 6.2 触发场景

提现页面展示"提现须知"区域 + 用户输入金额时实时校验和计算手续费。

### 6.3 数据流

```
发送：
  currencyCode: string    // 币种
  methodType:   i32       // 提现方式类型

期望返回：
  minAmount       string    // 单笔最低金额
  maxAmount       string    // 单笔最高金额
  dailyLimit      string    // 单日提现限额
  feeRate         string    // 手续费率（百分比，如 "0.01" 表示 1%）
  fixedFee        string    // 固定手续费（如有）
  arrivalTimeDesc string    // 到账时间描述（如 "T+1 工作日"）
```

### 6.4 产品原型证据

| 原型截图 | 展示内容 |
|---------|---------|
| 越南银行卡提现 | 提现须知：每日提现限额 ₫2,500,000 / 单笔最低 ₫2,500,000 / 手续费 1% |
| 未认证 USDT 提现 | 每日限额 USDT 10,000 / 单笔最低 USDT 10 / 手续费 1% |
| 提现金额填写 | 实时显示三行：提现金额 / 手续费 / 到账金额 |

### 6.5 钱包侧使用方式

```
1. 展示提现须知（UI 渲染）：
   "每日提现限额：{dailyLimit} / 单笔最低：{minAmount} / 手续费：{feeRate*100}%"

2. 金额校验（创建提现订单前）：
   if amount < minAmount → 拒绝，提示最低限额
   if amount > maxAmount → 拒绝，提示最高限额
   if todayTotal + amount > dailyLimit → 拒绝，提示已超日限额

3. 手续费计算（实时）：
   fee = amount * feeRate + fixedFee
   actualAmount = amount - fee
   展示：提现金额{amount} / 手续费{fee} / 到账金额{actualAmount}
```

- `z-readme/依赖关系/c-3.md:131` — `查询提现限额费率 → 依赖：财务.GetWithdrawConfig`
- 需求文档 `7-5提现限制` — "后台可配：单笔限额、单日限额、手续费率、T+N到账时间"

## 7. 依赖四：ser-finance — 提现订单提交审核（O4）

### 7.1 基本信息

| 维度 | 内容 |
|------|------|
| 被调用服务 | ser-finance |
| 方法名（拟定） | `SubmitWithdrawOrder` |
| IDL 状态 | **不存在** |
| 优先级 | **P0** — 这是提现流程中**事务性最强**的节点 |

### 7.2 触发场景

用户确认提现 → 钱包先冻结余额（本地 DB） → 再调用此接口将提现单提交给财务进入三级审核流程（风控→财务→出款）。

### 7.3 数据流

```
发送：
  orderId          string    // 提现订单号（T+16位，钱包侧生成）
  userId           i64       // 用户ID
  currencyCode     string    // 提现币种
  amount           string    // 提现金额
  fee              string    // 手续费金额
  actualAmount     string    // 实际到账金额（amount - fee）
  methodType       i32       // 提现方式类型
  accountInfo      string    // 提现账户信息（JSON：银行卡号/电子钱包号/USDT地址）
  lockedRateId     string    // 锁汇快照ID（关联到币种配置的汇率快照）
  lockedRate       string    // 锁定的出款汇率

期望返回：
  success          bool      // 是否提交成功
  financeOrderId   string    // 财务侧订单号（可选，用于后续状态同步）
```

### 7.4 关键：事务补偿

```
钱包侧操作序列（提现创建）：

① 校验余额 ≥ amount                     → 不足则拒绝
② 校验 KYC 状态 (O5)                    → 未认证+非USDT则拒绝
③ 获取锁汇汇率 (内部币种配置)            → 锁定当前出款汇率
④ FreezeBalance(userId, currency, amount) → 本地 DB 冻结
⑤ SubmitWithdrawOrder(O4)               → 提交财务审核
   ├── 成功 → 返回"提现中"状态给用户
   └── 失败 → ⚠️ 必须回滚：UnfreezeBalance(userId, currency, orderId)
```

**如果 ⑤ 失败但 ④ 已成功，不回滚就会导致用户资金被永久冻结。** 这是整个钱包模块事务安全要求最高的节点。

## 8. 依赖五：ser-kyc — KYC 状态查询（O5）

### 8.1 基本信息

| 维度 | 内容 |
|------|------|
| 被调用服务 | ser-kyc（编号 012） |
| 方法名 | `KycQuery` |
| IDL 状态 | **已存在** `common/idl/ser-kyc/service.thrift` |
| 客户端状态 | **已注册** `rpc.KycClient()` |
| 优先级 | **P0** — 提现流程的前置校验 |

### 8.2 触发场景

两个子场景：

**场景 A — 提现方式展示（决定可用渠道）**：
- `kycStatus == 3`（已通过）→ 展示全部提现方式
- 其他状态 → 弹窗提示"为了您的资金安全，请先完成KYC认证，解锁更多提现方式"，仅展示 USDT 提现

**场景 B — 账户持有人姓名自动填充**：
- `kycInfo.name`（真实姓名）→ 写入提现账户的"账户持有人"字段（灰显不可编辑）
- 产品要求："账户持有人姓名=KYC认证姓名，忽略大小写，忽略多余空格"

### 8.3 已有 IDL 签名


## 11. 内部依赖：币种配置子模块（5个函数调用）

币种配置是 ser-wallet 的**内部子模块**，两者部署在同一个进程中，调用方式是 **Go 函数直接调用**（不走 Kitex RPC）。但逻辑上它们是独立的业务域，需要清晰的接口边界。

### 11.1 内部调用清单

| # | 调用的函数 | 触发场景 | 数据流 |
|---|-----------|---------|-------|
| I1 | `GetAllEnabledCurrencies()` | 钱包首页/充值页币种切换列表 | → 所有已启用币种列表 |
| I2 | `GetCurrencyConfig(code)` | 余额展示时获取精度/符号/千分位格式 | → 币种的显示规则 |
| I3 | `IsCurrencyEnabled(code)` | 用户切换币种时校验目标币种是否可用 | → bool |
| I4 | `GetPlatformRate(from, to)` | 充值页汇率展示、兑换预览计算 | → 当前平台汇率 |
| I5 | `GetTransactionRate(code, txType)` | 创建充值/提现/兑换订单时锁汇 | → 锁定汇率 + 过期时间 |

### 11.2 为什么不走 RPC？

因为币种配置和钱包是**同一个 Kitex 服务进程**（共用 `WalletService`），内部调用直接走 Go 函数即可，无需网络序列化开销。但币种配置**同时也通过 RPC 暴露给外部服务**（如财务、投注域调用 `RpcGetPlatformRate`），这些外部调用才走 Kitex RPC。

```
模块内部调用（函数）：
  钱包 wallet_service.go → currency_service.go.GetPlatformRate()

模块外部调用（RPC）：
  ser-finance → rpc.WalletClient().RpcGetPlatformRate()
```

### 11.3 产品原型证据

| 内部调用 | 原型页面 | 展示内容 |
|---------|---------|---------|
| I1 | 个人中心-切换币种 | BSB/VND/IDR/THB 四个币种列表 |
| I2 | 钱包首页 | BSB 1,000,000.00 / ₫ 500,000,000（格式各异） |
| I4 | 充值页底部 | 汇率：1BSB=0.1USDT |
| I4 | 兑换页 | 1BSB=₫2,620，兑换获得/赠送10%/实际到账 |
| I5 | 创建充值/提现/兑换订单 | 锁定汇率快照（订单生命周期内不变） |

## 12. 外部 HTTP 依赖（非 RPC）

### 12.1 三方实时汇率 API

| 维度 | 内容 |
|------|------|
| 调用方 | 币种配置子模块的定时任务 |
| 目标 | 3 个第三方汇率 API（具体供应商待产品确认） |
| 方式 | HTTP GET 定时轮询 |
| 频率 | 需求文档要求"可约定频率，开发测试期间可低频率"，原型显示倒计时约 2 分钟 |
| 逻辑 | 3 家取汇 → 算平均值 → 作为实时汇率 → 计算平台汇率（实时汇率 × 浮动系数） |
| 优先级 | **P0** — 没有实时汇率数据，所有跨币种操作无法计算 |

```
实时汇率 = avg(API_1, API_2, API_3)
平台入款汇率 = 实时汇率 × (1 + 入款浮动%)    // 用于充值/兑换
平台出款汇率 = 实时汇率 × (1 - 出款浮动%)    // 用于提现
```

### 12.2 三方支付通道 API

| 维度 | 内容 |
|------|------|
| 调用方 | 充值流程（创建支付订单）/ 提现流程（出款请求） |
| 目标 | 代收通道（充值）/ 代付通道（提现） |
| 方式 | HTTP（各通道 API 格式不同） |
| 关键疑问 | **由谁对接？** |

**待确认事项**：充值创建订单时，是钱包直接对接三方通道，还是通过财务模块转发？

- `z-readme/依赖关系/c-3.md:211-213` 充值链路图显示：钱包直接请求三方通道
- `z-readme/财务结构/c-1.md:845` 描述：`MatchChannelAndCreatePayment` → 财务侧轮询选通道 → 向三方发起

两种方案的影响：

| 方案 | 钱包侧需要做的 | 说明 |
|------|-------------|------|
| A: 通过财务转发 | 调用财务 RPC（如 `AssignChannelAndCreatePayment`） | 钱包不感知通道细节，简单 |
| B: 钱包直接对接 | 自己实现通道选择 + API 对接 + 签名验签 | 更独立，但复杂度高 |

**当前建议**：倾向方案 A，因为通道配置（商户ID/密钥/公钥/私钥）在财务后台管理，钱包不应感知这些敏感信息。

## 17. 开发阶段影响与风险

### 17.1 开发阶段划分

```
Phase 0 — 零外部依赖（可立即开工）
├── 币种配置模块全部（B端+RPC暴露）
├── 钱包余额 RPC 暴露层（GetBalance/Credit/Debit/Freeze 等）
├── 兑换功能全流程（仅依赖内部币种配置函数调用）
├── 提现账户管理（银行卡/电子钱包/USDT 的增删查，纯本地 DB）
├── 记录查询（充值/提现/兑换/奖励记录，纯本地 DB）
└── B端审计日志（O6 ser-blog 已就绪）+ 操作人姓名（O7 ser-buser 已就绪）

Phase 1 — 需要 ser-kyc（已就绪，可立即用）
└── 提现页面展示逻辑（KYC 状态判断 + 账户持有人姓名自动填充）

Phase 2 — 需要 ser-finance IDL（⚠️ 最大卡点）
├── 充值方式展示（O1）→ 阻塞充值 UI 渲染
├── 提现方式展示（O2）→ 阻塞提现 UI 渲染

    → 在 IDL 就绪前可使用 Mock 先行开发钱包侧逻辑

Phase 3 — 需要三方 API 对接
└── 三方汇率 API 接入（3 家取均值）
    → 可先用固定汇率 Mock
```

### 17.2 风险评估

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| **ser-finance IDL 延迟** | 充值/提现无法联调 | Mock 先行，定期对齐进度 |

| **通道对接方案未确认** | 充值创建订单逻辑不确定 | 评审中明确归属（方案 A 通过财务 vs 方案 B 直接对接） |

## 18. 与 rpc暴露/c-1.md 的双向对照

### 18.1 入向 vs 出向全景对比

| 维度 | 入向（别人调我们） | 出向（我们调别人） |
|------|-----------------|-----------------|
| 接口总数 | 19 个 RPC | 7 个 RPC + 2 类 HTTP |
| 涉及服务数 | 5+（财务/投注/直播/商城/全平台） | 4 个（财务/KYC/日志/B端用户） |
| 最大调用方/被调方 | 财务（10个RPC调我们） | 财务（4个RPC我们调它） |
| IDL 就绪度 | 我们自己定义（全部需创建） | 3/4 已就绪，**财务的不存在** |
| 最高风险 | 幂等/CAS 实现正确性 | **财务 IDL 延迟阻塞联调** |

### 18.2 双向依赖图（钱包 ↔ 财务）

```
              我们调财务（4个出向 RPC）
              ─────────────────────►
┌──────────┐                        ┌──────────────┐
│ ser-wallet│                        │ ser-finance  │
│ (我们的)  │                        │ (他人负责)    │
└──────────┘                        └──────────────┘
              ◄─────────────────────
              财务调我们（10个入向 RPC）
```

| 方向 | 数量 | 内容 |
|------|------|------|
| 钱包 → 财务 | 4 | 充值方式列表、提现方式列表、提现限额配置、提现提交审核 |
| 财务 → 钱包 | 10 | 余额查询、充值入账、加减款、冻结/解冻/确认、补单、汇率查询、金额换算 |

**不对称性**：财务调我们的接口（10个）远多于我们调财务的（4个）。这反映了钱包的定位——**资金操作的执行者**，而非**业务流程的编排者**。编排逻辑（审核流、通道选择、出款调度）在财务侧。
