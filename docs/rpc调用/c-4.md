## 一、调用方向全景

上一篇文档（rpc暴露/c-4.md）梳理的是 **别人调我** 的12个接口。本文档梳理的是 **我调别人** 的全部依赖。

```
我调别人 vs 别人调我：

     我调别人（本文档）                    别人调我（rpc暴露文档）
     ─────────────────                    ─────────────────────
     ser-finance  ← 2~4个查询/通知         ser-finance  → 8个（7写+1查）
     ser-kyc      ← 1~2个查询
     ser-blog     ← 1个审计日志(oneway)
     ser-buser    ← 1个操作人姓名解析
     ser-i18n     ← 0个（网关层处理）
     ser-user     ← 0个（不需要）

特征对比：
  我调别人：以查询为主，不改别人的数据
  别人调我：以写操作为主，改我的余额数据
```

## 二、确定的RPC调用（5个接口，4个服务）

以下调用在需求文档和链路分析中有**明确的业务场景**，是确定性依赖。

### 2.1 调用 ser-finance —— 查询充值方式

```
编号：EXT-1
调用接口：GetAvailableDepositMethods（暂定名，最终以财务模块IDL为准）
被调接口所属：ser-finance（财务管理模块，别人负责）
```

**触发场景**：

```
用户操作                    钱包内部                        财务管理
   │                         │                              │
   │  打开充值页面             │                              │
   │ ───────────────────►    │                              │
   │                         │  W-3 GetDepositMethods       │
   │                         │  需要当前币种可用的充值方式    │
   │                         │                              │
   │                         │  RPC调用 ──────────────────► │
   │                         │  入参：currency_code          │
   │                         │                              │
   │                         │  ◄── 返回：充值方式列表       │
   │                         │      每种方式的金额档位       │
   │                         │      费率信息                 │
   │                         │      关联通道信息              │
   │                         │                              │
   │  ◄── 渲染充值方式列表    │                              │
```

**为什么我不能自己查**：充值方式配置（哪些通道可用、金额档位、费率）归财务模块管理，数据在财务侧的数据库里。钱包不持有这份数据，必须跨服务查询。

**调用频率**：中频——每次用户进入充值页面就调一次。

**失败影响**：**阻塞级**——此接口不可用时，充值页面无法渲染充值方式，用户看不到怎么充钱。

**我期望的返回结构（需与财务对齐）**：

```
出参（期望）：
  methods: []DepositMethod
    - method_type: string          // "USDT" / "bank_transfer" / "e_wallet"
    - method_name: string          // "USDT(TRC20)" / "银行转账" / "GoPay"
    - min_amount: string           // 最低充值金额
    - max_amount: string           // 最高充值金额
    - amount_presets: []string     // 金额档位 ["100","500","1000","5000"]
    - fee_rate: string             // 手续费率（"0" / "0.01"）
    - fee_fixed: string            // 固定手续费
    - is_platform_hosted: bool     // 是否平台自承载（USDT=true）
    - channel_info: optional       // 通道信息（三方跳转URL模板等）
```

> 依据：链路关系文档 3.2——"W-3 → EXT-1 GetAvailableDepositMethods"；需求文档 E2——充值方式配置在财务侧

### 2.3 调用 ser-kyc —— 查询KYC认证状态

```
编号：EXT-3
调用接口：KycQuery
被调接口所属：ser-kyc（KYC认证模块）
IDL位置：/Users/mac/gitlab/common/idl/ser-kyc/service.thrift
客户端函数：rpc.KycClient().KycQuery(ctx, req)
```

**触发场景**：

```
用户操作                    钱包内部                        ser-kyc
   │                         │                              │
   │  打开提现页面             │                              │
   │ ───────────────────►    │                              │
   │                         │  W-10 GetWithdrawalInfo      │
   │                         │                              │
   │                         │  RPC调用 ──────────────────► │
   │                         │  入参：uid (用户ID)           │
   │                         │                              │
   │                         │  ◄── 返回：KYC状态+认证姓名   │
   │                         │                              │
   │                         │  [根据KYC状态过滤提现方式]    │
   │                         │  未认证 → 只保留USDT方式      │
   │                         │  已认证 → 保留全部方式         │
```

**已有IDL定义（直接可用）**：

```thrift
// /Users/mac/gitlab/common/idl/ser-kyc/service.thrift
service KycService {
    user_kyc.KycQueryResp KycQuery(1: user_kyc.KycQueryReq req)
}

// 请求
struct KycQueryReq {
    1: optional i64 uid
}

// 响应（关键字段）
struct KycQueryResp {
    1: optional UserKycInfo kycInfo
    // kycInfo 含：uid, name, country, phone, idType, idNumber, status, rejectReason 等
}
```

**我需要用到的字段**：
- `kycInfo.status` → 判断认证状态（决定可用提现方式）
- `kycInfo.name` → 认证姓名（自动填充提现账户持有人，W-12 SaveWithdrawalAccount 使用）

**调用频率**：中频——每次用户进入提现页面。

**失败影响**：**可降级**——KYC不可用时，可降级为"仅允许USDT提现"（未认证的默认行为），不阻塞整个提现页面。

**调用点汇总**：

| 钱包内接口 | 调用时机 | 用途 |
|-----------|---------|------|
| W-10 GetWithdrawalInfo | 提现页面加载 | 判断KYC状态 → 过滤提现方式 |
| W-12 SaveWithdrawalAccount | 保存提现账户 | 获取认证姓名 → 自动填充账户持有人 |

> 依据：链路关系文档 3.6——"W-10 → EXT-3 ser-kyc.KycQuery"；需求文档 A5——"未认证仅USDT提现"

### 2.4 调用 ser-blog —— 写操作审计日志

```
编号：EXT-4
调用接口：AddActLog（oneway void，火并即忘）
被调接口所属：ser-blog（审计日志模块）
IDL位置：/Users/mac/gitlab/common/idl/ser-blog/service.thrift
客户端函数：rpc.BLogClient().AddActLog(ctx, req)
```

**已有IDL定义（直接可用）**：

```thrift
// /Users/mac/gitlab/common/idl/ser-blog/service.thrift
service BLogService {
    oneway void AddActLog(1: model.ActAddReq req)
}

// 请求结构
struct ActAddReq {
    1: string backend       // 系统标识（如 "wallet_backend"）
    2: string model         // 操作模块（如 "currency_config" / "wallet"）
    3: string page          // 操作页面（如 "currency_list" / "deposit"）
    4: string action        // 操作描述（如 "创建币种: VND, 操作人: xxx"）
    5: i32 isSuccess        // 1=成功, 0=失败
}
```

**已有调用模式（直接复用）**：

```go
// 参考 ser-buser 的调用方式
rpc.BLogClient().AddActLog(ctx, &ser_blog.ActAddReq{
    Backend:   "wallet_backend",
    Model:     "currency_config",
    Page:      "currency_management",
    Action:    fmt.Sprintf("创建币种: %s, 操作人ID: %d", currencyCode, operatorId),
    IsSuccess: 1,
})
// oneway：不等返回，不影响主流程
```

**调用频率**：高频——每个写操作都触发。

**失败影响**：**不阻塞**——oneway调用，日志服务挂了不影响业务。日志会丢失，但交易不受影响。

**触发该调用的全部接口**：

```
币种配置域（B端写操作）：
  CC-1  SetBaseCurrency          → AddActLog("设置基准币种: USDT")
  CC-2  CreateCurrency           → AddActLog("创建币种: VND")
  CC-3  UpdateCurrency           → AddActLog("更新币种: VND, 修改字段: ...")
  CC-4  ToggleCurrencyStatus     → AddActLog("禁用币种: VND")

钱包域（C端写操作）：
  W-4   CreateDepositOrder       → AddActLog("创建充值订单: C0001, 金额: 1000 VND")
  W-9   ExecuteExchange          → AddActLog("执行兑换: 1000 VND → 10 BSB")
  W-11  CreateWithdrawalOrder    → AddActLog("创建提现订单: T0001, 金额: 500 VND")
  W-12  SaveWithdrawalAccount    → AddActLog("保存提现账户: 银行卡 ***1234")

钱包域（RPC写操作，由财务触发但日志由我记录）：
  W-R2  CreditDeposit            → AddActLog("充值上账: C0001, +1000 VND")
  W-R4  UnfreezeForWithdrawal    → AddActLog("提现解冻: T0001, 原因: 审核拒绝")
  W-R5  ConfirmWithdrawalDeduct  → AddActLog("提现扣款: T0001, -500 VND")
  W-R6  ManualCredit             → AddActLog("人工加款: A0001, +2000 VND(中心)")
  W-R7  ManualDebit              → AddActLog("人工减款: M0001, -1000 VND(奖励)")
  W-R8  CreditCorrection         → AddActLog("充值补单: B0001, +1500 VND")
```

**合计约14个调用点**，全部是 oneway，统一封装为一个内部工具函数即可。

> 依据：链路关系文档全文——所有写操作均标注"→ EXT-4 ser-blog.AddActLog（oneway）"


**场景描述**：

```
用户提交充值 → 钱包完成业务校验(重复拦截+奖金关联) → ???

可能的方案：

方案A（钱包调财务）：
  钱包创建预订单 → RPC调财务"分配通道+获取支付信息" → 财务返回三方跳转URL → 钱包返回给用户

  → 不需要额外RPC调用

方案C（前端直接调财务）：
  钱包完成业务校验返回 → 前端拿到结果后直接调网关走财务路由创建支付
```

**当前我的倾向**：方案A，理由见评审反馈文档 P0-3 的建议。但这取决于财务模块开发者的设计。

**影响范围**：如果走方案A，钱包需要新增对 ser-finance 的一次写操作RPC调用。

> 依据：链路关系文档 W-4——"[RPC?] 通知财务分配通道（具体交互方式待确认）"

## 四、明确不需要调用的服务（排除清单）

以下服务经过分析确认 **ser-wallet 不需要主动调用**：

| 服务 | 为什么不需要调 | 说明 |
|------|--------------|------|
| **ser-i18n** | 多语言翻译在网关层处理 | gate-font/gate-back 的 handler 中使用 `rpc.I18nClient()` 做错误信息翻译，ser-wallet 服务内部返回错误码即可，翻译由网关负责 |
| **ser-user** | 钱包不需要用户昵称/头像等信息 | C端用户基本信息由前端自行从 ser-user 获取，钱包只需要 userId（从网关上下文拿到），不需要反向查用户信息 |
| **ser-ip** | IP黑白名单在网关层处理 | gate-font 的中间件 BlackCheck 已处理IP校验，不需要服务层重复校验 |
| **ser-bcom** | 开关配置在网关层处理 | 功能开关等通过网关中间件或前端控制 |
| **ser-app** | 无业务关联 | Banner/首页分类与钱包无关 |
| **ser-item** | 无业务关联 | 道具管理与钱包无关（未来礼物/打赏可能关联，但当前需求不涉及） |
| **ser-s3** | 币种图标上传由网关处理 | B端上传币种图标走 gate-back → ser-s3，钱包服务内部只存图标URL |
| **ser-live / ser-rtc / ser-shortvideos** | 当前需求无关联 | 未来直播打赏可能需要调钱包（方向是别人调我），但钱包不需要调直播 |
| **ser-recommend** | 无业务关联 | 推荐系统与钱包无关 |
| **ser-exp** | 无业务关联 | VIP经验值与钱包无直接调用关系 |

**排除依据**：逐一对照了 common/rpc/rpc_client.go 中注册的22个服务客户端，结合需求文档和链路分析，确认只有4个服务（finance、kyc、blog、buser）是 ser-wallet 的调用目标。

## 七、编码阶段的封装建议

### 7.1 统一审计日志工具函数

14个调用点都调 AddActLog，应封装为内部工具函数避免重复代码：

```go
// internal/pkg/audit/log.go

func LogAction(ctx context.Context, module, page, action string, success bool) {
    isSuccess := int32(0)
    if success {
        isSuccess = 1
    }
    // oneway，不等返回，不处理error
    rpc.BLogClient().AddActLog(ctx, &ser_blog.ActAddReq{
        Backend:   "wallet_backend",
        Model:     module,
        Page:      page,
        Action:    action,
        IsSuccess: isSuccess,
    })
}

// 使用示例
audit.LogAction(ctx, "currency_config", "currency_list",
    fmt.Sprintf("创建币种: %s", req.CurrencyCode), true)
```

### 7.2 统一外部RPC调用封装

### 与上一篇文档的对照

```
我给别人（rpc暴露）          我从别人拿（rpc调用，本文档）
─────────────────          ─────────────────────────
12个接口                    5个确定 + 2~3个待确认
```
