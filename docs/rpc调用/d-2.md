## 一、总览 — ser-wallet 需要调用谁

```
ser-wallet 对外依赖图
│
├── 已有模块（IDL + 实现都已就绪，可直接调用）
│   ├── ser-kyc ─────── KYC认证状态查询（提现前置）
│   ├── ser-blog ────── 操作日志记录（B端操作审计）
│   ├── ser-i18n ────── 多语言翻译（错误码/枚举翻译）
│   ├── ser-user ────── 用户信息查询（流水/记录展示）
│   └── ser-buser ───── B端操作人信息查询（后台操作记录）
│
└── 未建模块（IDL 待定义，需要对方先暴露接口）
    └── ser-finance ─── 充值方式/提现方式/通道匹配/出款提交
```

### 调用量级预估

| 被调用模块 | 接口数量 | 调用频率 | 依赖强度 |
|-----------|---------|---------|---------|
| **ser-finance（财务）** | 5-6个 | 高（每次充值/提现） | **强依赖** — 缺了充值提现跑不通 |
| **ser-kyc** | 1个 | 中（每次提现） | **强依赖** — 缺了提现不可用 |
| **ser-blog** | 1个 | 高（每次B端操作） | **弱依赖** — oneway异步，不阻塞主流程 |
| **ser-i18n** | 1-2个 | 高（每次响应） | **弱依赖** — 缺了只是没翻译，不影响业务 |
| **ser-user** | 1-2个 | 中（展示场景） | **弱依赖** — 缺了只是没用户名，不影响核心业务 |
| **ser-buser** | 1个 | 低（B端场景） | **弱依赖** — 仅后台展示操作人 |

## 二、已有模块 — 可直接调用的 RPC 接口

### 2.1 ser-kyc（KYC认证服务） — 1个接口

#### 接口：KycQuery

```
被调用服务：ser-kyc (KycService)
客户端获取：rpc.KycClient()
方法签名：KycQuery(user_kyc.KycQueryReq) → user_kyc.KycQueryResp
调用方式：同步调用
```

**什么场景调用**：

用户发起提现时，需要检查其 KYC 认证状态：
- **已通过** → 继续提现流程（所有提现方式可用）
- **未认证** → 弹窗提示：只能 USDT 提现 / 去认证
- **审核中** → 等待审核结果

**需求文档依据**：
- 需求 7-1提现模块：提现前必须检查 KYC 状态
- 产品原型 8-1~8-4提现触发认证：不同 KYC 状态的交互流程

**调用链路位置**：

```
用户点击提现
  │
  ├── [→ ser-kyc] KycQuery(userId)
  │     └── 返回 KYC 状态（1:去认证 2:审核中 3:已通过 4:重新认证）
  │
  ├── [判断]
  │   ├── status=3(已通过) → 进入正常提现流程
  │   ├── status=1/4(未认证/重新认证) → 弹窗："仅USDT可提现" / "去认证"
  │   └── status=2(审核中) → 提示等待
  │
  └── 继续或中断
```

**工程验证**：
- IDL定义：`/Users/mac/gitlab/common/idl/ser-kyc/service.thrift` → `KycQuery` 方法已存在
- 客户端工厂：`common/rpc/rpc_client.go` → `rpc.KycClient()` 已注册
- 调用模式参照：gate-font 中 `rpc.KycClient().KycQuery()` 已有使用先例

**我方调用代码位置**：
```
ser-wallet/internal/ser/withdraw_service.go
  → 创建提现订单前调用 rpc.KycClient().KycQuery()
```

### 2.2 ser-blog（操作日志服务） — 1个接口

#### 接口：AddActLog

```
被调用服务：ser-blog (BLogService)
客户端获取：rpc.BLogClient()
方法签名：AddActLog(model.ActAddReq) → oneway void
调用方式：异步oneway（发射即忘，不等响应）
```

**什么场景调用**：

所有 B端管理操作完成后，记录操作日志用于审计追溯：

| 操作场景 | 日志内容 |
|---------|---------|
| 创建币种配置 | 操作人、币种代码、创建时间 |
| 编辑币种配置 | 操作人、修改字段、修改前后值 |
| 启用/禁用币种 | 操作人、币种、状态变更 |
| 设置基准币种 | 操作人、基准币种代码 |
| 手动刷新汇率 | 操作人、刷新的币种 |

**需求文档依据**：
- 币种配置需求：操作日志功能（产品原型有"操作日志"Tab页）
- 工程惯例：所有已有模块（ser-app、ser-buser、ser-bcom、ser-ip、ser-s3）的 B端操作都调用 AddActLog

**工程验证**：
- IDL定义：`/Users/mac/gitlab/common/idl/ser-blog/service.thrift` → `AddActLog` 方法（oneway void）
- 客户端工厂：`rpc.BLogClient()` 已注册
- 调用先例（大量）：
  - `ser-app/internal/ser/banner.go` — Banner增删改后调用
  - `ser-buser/internal/ser/user_service.go` — 用户管理操作后调用
  - `ser-buser/internal/ser/role_service.go` — 角色管理操作后调用
  - `ser-buser/internal/ser/menu_service.go` — 菜单管理操作后调用
  - `ser-ip/internal/ser/c_black_service.go` — 黑名单操作后调用
  - `ser-s3/internal/ser/com_service.go` — S3操作后调用

**关键特性**：
- **oneway** = 异步单向调用，ser-wallet 发出后不等待响应，不阻塞主业务流程
- **失败不影响主流程** = 日志写入失败不应影响币种配置等操作的成功返回
- 这是工程内的通用模式，我方遵循即可

**我方调用代码位置**：
```
ser-wallet/internal/ser/currency_service.go  → 币种配置CRUD后调用
```

### 2.3 ser-i18n（国际化服务） — 1-2个接口

#### 接口1：GetTrans（推荐）

```
被调用服务：ser-i18n (I18nService)
客户端获取：rpc.I18nClient()
方法签名：GetTrans(lang.TransReq) → lang.TransResp
调用方式：同步调用
```

#### 接口2：LangTransList（备选）

```
方法签名：LangTransList(list<i64>) → map<i64, string>
```

**什么场景调用**：

1. **错误码翻译** — RPC 返回错误时，将错误码翻译为用户语言的文案
2. **枚举值翻译** — 状态文案（如充值状态"处理中"/"成功"/"失败"）的多语言展示

**需求文档依据**：
- 平台面向多地区运营（越南、印尼、泰国等），多语言是基础需求
- 需求 1-需求背景.png 明确多地区运营

**工程验证**：
- 已有调用模式：`common/rpc/handler.go` 中的通用 Handler 函数已集成 I18nClient 用于错误翻译
- `ser-bcom/internal/ser/com_service.go` 中 QueryEnum 使用 `I18nClient().LangTransOne()` 翻译枚举标签

**关键说明**：
- 网关层（gate-font/gate-back）的通用 `rpc.Handler()` 已自动处理错误码翻译，ser-wallet 的 handler 层不需要自己调用 I18n
- 仅在 **ser-wallet 内部业务逻辑** 需要翻译文案时（如记录列表的状态文案）才需要直接调用
- 如果状态文案由前端根据状态码自行翻译，则 ser-wallet 可以不直接调用 I18n

**结论**：按工程现有模式，**绝大多数翻译由网关层通用 Handler 自动完成**，ser-wallet 直接调用 I18n 的需求很低。但预留此依赖，如后续有业务层翻译需求可随时接入。

#### 接口2：BatchGetUserInfo

```
方法签名：BatchGetUserInfo(user_rpc.BatchGetUserInfoReq) → user_rpc.BatchUserInfoResp
```

**什么场景调用**：

| 场景 | 用途 | 频率 |
|------|------|------|

| 提现订单列表（B端） | 订单中展示用户基本信息 | 中 |
| 充值订单列表（B端） | 同上 | 中 |

**需求文档依据**：
- 财务需求 2-3充值记录、2-4提现记录 — 记录列表中展示用户信息

**工程验证**：
- IDL定义：`/Users/mac/gitlab/common/idl/ser-user/service.thrift` → `GetUserInfoExpend`、`BatchGetUserInfo` 已存在

**关键说明**：
- ser-wallet 的 **C端接口不需要调用 ser-user** — 因为 C 端请求自带 userId（从 token 中解析），不需要额外查用户信息

- 初期 B 端列表可以只展示 userId，不展示昵称（减少依赖），后续需要时再接入

### 2.5 ser-buser（B端系统用户服务） — 1个接口

#### 接口：QueryUserByIds

```
被调用服务：ser-buser (BUserService)
客户端获取：rpc.BUserClient()
方法签名：QueryUserByIds(sys_user.RpcSysUserReq) → map<i64, string>
调用方式：同步调用
```

**什么场景调用**：

B 端管理页面中展示"操作人"信息（如币种配置的创建人、修改人）。

| 场景 | 用途 |
|------|------|
| 币种配置列表 | 展示"创建人"、"最后修改人"列 |
| 汇率日志 | 展示"操作人"列 |

**工程验证**：
- 调用先例：`ser-item/internal/domain/user_service.go` 中 `rpc.BUserClient().QueryUserByIds()` 用于映射 userId→用户名
- `ser-bcom/internal/ser/backend_white_service.go` 中 `rpc.BUserClient().QueryUserDetail()` 用于获取操作人信息

**关键说明**：
- 这是 **非必要依赖** — 如果表中直接存储操作人名称（而非仅存 ID），则不需要调用
- 工程惯例是表中存 operatorId（i64），展示时反查用户名
- 初期可以直接展示 ID 或在创建/修改时同时存储操作人名称，避免实时反查

#### 接口5：SubmitWithdrawToChannel — 提交提现到通道（出款）

```
调用场景：提现审核全部通过后，财务模块自行调用三方出款
调用频率：中
```

**特别说明**：这个接口存在两种可能的调用方向：

**方案A**：ser-wallet 不直接调用此接口 — 提现审核和出款都在财务模块完成，财务审核通过后自行调用三方，成功后回调 ser-wallet 的 ConfirmDebit

**方案B**：ser-wallet 调用此接口 — 提现创建订单后主动通知财务进入审核流程

```
方案A流程（推荐，解耦更好）：
  ser-wallet: FreezeBalance → 创建提现订单
  ser-finance: 审核通过 → 调三方出款 → 成功 → 调 ser-wallet.ConfirmDebit
                                        → 失败 → 调 ser-wallet.UnfreezeBalance

方案B流程：
  ser-wallet: FreezeBalance → 创建提现订单 → 调 ser-finance.SubmitWithdraw
  ser-finance: 审核通过 → 调三方出款 → 成功 → 调 ser-wallet.ConfirmDebit
```

**当前判断**：倾向方案A。提现审核是财务域的职责，ser-wallet 只负责冻结资金和接收最终结果。ser-wallet 可能只需通知财务"有一笔新的提现订单"（轻量通知），而非指挥出款流程。

这个接口是否需要、接口形式如何，**取决于与财务模块负责人的接口契约协商**。

---

#### 接口6（待确认）：NotifyNewWithdrawOrder — 通知财务有新提现订单

```
调用场景：ser-wallet 创建提现订单后通知财务
调用频率：中
可替代方案：通过 MQ/NATS 消息通知代替 RPC
```

- RPC 调用：ser-wallet → ser-finance.NotifyNewWithdrawOrder
- 消息队列：ser-wallet 发 NATS/Kafka 消息 → ser-finance 消费

**当前判断**：此处留为待定，需与财务模块负责人协商确定。

## 六、依赖分析总结

### 6.1 一句话概括

ser-wallet 对外 RPC 调用依赖分为两层：
- **已有模块**：ser-kyc（1个必须接口） + ser-blog（1个审计接口） + ser-user/ser-buser/ser-i18n（3个辅助接口），全部客户端已就绪，可直接调用
- **未建模块**：ser-finance（4-6个核心接口），充值/提现的核心链路强依赖，需要对方先定义 IDL 并暴露接口

### 6.2 对编码阶段的影响

| 阶段 | 可做的事 | 被阻塞的事 |
|------|---------|-----------|
| **搭骨架阶段** | 定义调用接口的抽象层（interface），在代码中预留调用位置 | 无阻塞 |
| **币种配置模块** | 完整实现（仅依赖 ser-blog） | 无阻塞 |
| **钱包核心模块** | 余额管理、初始化等内部逻辑 | 无阻塞 |
| **兑换模块** | 完整实现（内部闭环） | 无阻塞 |
| **提现模块** | KYC 调用可真实接入；财务相关 Mock | ser-finance 接口待定 |
| **充值模块** | 订单创建逻辑可实现；充值方式和通道 Mock | ser-finance 接口待定 |
| **联调阶段** | 全部接通 | ser-finance 必须就绪 |

### 6.3 Mock 策略

对于 ser-finance 相关的接口，编码阶段采用 Mock/TODO 占位：

```go
// ser-wallet/internal/ser/recharge_service.go

func (s *RechargeService) GetRechargeMethodList(ctx context.Context, currencyCode string) ([]RechargeMethod, error) {
    // TODO: 调用 ser-finance.GetRechargeMethodList RPC
    // 财务模块 IDL 未定义，暂用 Mock 数据
    // 联调时替换为：rpc.FinanceClient().GetRechargeMethodList(ctx, req)
    return mockRechargeMethodList(currencyCode), nil
}
```

这符合之前约定的编码策略：**可以暂时模糊的用 Mock 占位，不能没有的（"火"）主动反馈**。ser-finance 接口就是"调料"（可 Mock 先行），而接口契约协商就是"火"（必须推进）。
