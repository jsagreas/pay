# ser-wallet 外部 RPC 调用依赖分析

> **文档定位**：系统分析 ser-wallet 需要**主动调用**其他模块的哪些 RPC 接口（出站依赖），不含 B端/C端 HTTP 接口，也不含模块内部进程内调用。
>
> **核心问题**：ser-wallet 在完成自身业务逻辑过程中，需要依赖外部哪些服务的哪些能力？哪些已有现成的 RPC 客户端？哪些需要新建？
>
> **分析依据**：
> - 已完成的分析文档：链路关系/d-3.md、依赖关系/d-3.md、rpc暴露/d-3.md、IDL设计/d-3.md、钱包结构/d-3.md
> - 工程事实：`common/rpc/rpc_client.go` 已注册的 22 个 RPC 客户端、各服务 IDL 定义
> - 需求文档：存储会话/c-3.md、result/c-3.md 中的充值/提现/兑换/币种配置全链路分析
>
> **与 rpc暴露/d-3.md 的关系**：暴露分析研究的是"别人调我"（25 个入站 RPC），本文研究的是"我调别人"（出站 RPC）。两者互为补充，共同构成 ser-wallet 的 RPC 全景图。

## 一、为什么需要分析出站 RPC 依赖

### 1.1 入站 vs 出站：方向完全不同

```
┌───────────────────────────────────────────────────────────────┐
│                       ser-wallet 的两类 RPC 交互                │
├───────────────────────────────────────────────────────────────┤
│                                                                │
│  入站 RPC（别人调我）— rpc暴露/d-3.md 已分析                     │
│  · 25 个方法，定义在 wallet_rpc.thrift + currency.thrift         │
│  · 我方是服务端，对方是客户端                                     │
│  · 我方决定接口契约，对方遵守                                     │
│  · 核心关注：幂等、错误码、超时约束                                │
│                                                                │
│  出站 RPC（我调别人）— 本文分析                                   │
│  · 共涉及 5 个目标服务、约 10 个方法                              │
│  · 我方是客户端，对方是服务端                                     │
│  · 对方决定接口契约，我方遵守                                     │
│  · 核心关注：依赖可用性、降级策略、Mock 方案                       │
│                                                                │
└───────────────────────────────────────────────────────────────┘
```

### 1.2 为什么出站依赖更需要提前梳理

| 维度 | 入站 RPC | 出站 RPC |
|------|---------|---------|
| 控制权 | 我方完全控制 | 对方控制，我方适配 |
| 开发阻塞 | 不阻塞（自己定义自己实现） | **可能阻塞**（对方未开发=我方无法联调） |
| Mock 策略 | 不需要（我就是提供方） | **必须有**（对方未就绪时需要 mock 运行） |
| 错误处理 | 定义错误码即可 | 需处理对方的错误码 + 超时 + 不可用 |

**结论**：出站 RPC 是 ser-wallet 开发中的**外部依赖风险点**，必须搞清楚——谁已经有了，谁还没有，谁可以 mock，谁必须真实对接。

## 三、逐服务详细分析

### 3.1 ser-blog — 审计日志服务（006）

#### D01. AddActLog — 操作审计日志

**IDL 定义**（`common/idl/ser-blog/service.thrift`）：
```thrift
service BLogService {
    // 使用 oneway 消息类型语法，客户端发起一个请求后不等待响应
    oneway void AddActLog(1: model.ActAddReq req);
}
```

**请求结构体**（`common/idl/ser-blog/model.thrift`）：
```thrift
struct ActAddReq {
    1: string backend        // 所属后台 → "wallet_backend"
    2: string model          // 操作模块 → "wallet" / "currency"
    3: string page           // 操作页面 → "币种管理" / "充值订单" / "提现审核" 等
    4: string action         // 操作行为 → "新增 币种管理 币种编码:1001_USD"
    5: i32 isSuccess         // 操作状态 → 1=成功 / 0=失败
}
```

**调用特征**：

| 属性 | 值 |
|------|-----|
| 调用类型 | **oneway**（发后即忘，不等响应） |
| 阻塞性 | **不阻塞主流程**（失败不影响业务逻辑） |
| 优先级 | P0（首期必须，所有写操作都需要审计） |
| 调用频率 | 中频（每次 B端写操作 + C端关键资金操作时触发） |
| RPC 客户端 | `rpc.BLogClient()` ✅ 已注册 |

**触发场景全集**：

| 场景编号 | 触发动作 | page 值 | action 示例 |
|---------|---------|---------|------------|
| L01 | B端新增币种 | currency_config | `新增 币种管理 币种编码:1001_USD` |
| L02 | B端编辑币种 | currency_config | `编辑 币种管理 币种编码:1001_USD` |
| L03 | B端切换币种状态 | currency_config | `禁用 币种管理 币种编码:1001_USD` |
| L04 | B端设置基准币种 | currency_config | `设置 基准币种 BSB` |
| L05 | C端发起充值 | deposit_order | `充值 用户:uid 币种:VND 金额:500000` |
| L06 | C端发起提现 | withdraw_order | `提现 用户:uid 币种:VND 金额:300000` |
| L07 | C端执行兑换 | exchange_order | `兑换 用户:uid VND→BSB 金额:200000` |
| L08 | C端绑定提现账户 | withdraw_account | `绑定 提现账户 银行:XXX 用户:uid` |
| L09 | C端删除提现账户 | withdraw_account | `删除 提现账户 ID:xxx 用户:uid` |
| L10 | RPC 入账操作 | wallet_rpc | `入账 用户:uid 币种:VND 金额:100000 订单:xxx` |
| L11 | RPC 扣款操作 | wallet_rpc | `扣款 用户:uid 币种:VND 金额:50000 订单:xxx` |
| L12 | RPC 冻结操作 | wallet_rpc | `冻结 用户:uid 币种:VND 金额:80000 订单:xxx` |
| L13 | RPC 解冻操作 | wallet_rpc | `解冻 用户:uid 币种:VND 金额:80000 订单:xxx` |
| L14 | RPC 人工加款 | wallet_rpc | `人工加款 用户:uid 币种:VND 金额:10000 审批单:xxx` |
| L15 | RPC 人工减款 | wallet_rpc | `人工减款 用户:uid 币种:VND 金额:5000 审批单:xxx` |

**代码模式**（参照 ser-app/ser-buser 现有实现）：
```go
// 推荐封装为统一函数，避免每次手写
func addWalletLog(ctx context.Context, page, action string, isSuccess int32) {
    rpc.BLogClient().AddActLog(ctx, &ser_blog.ActAddReq{
        Backend:   "wallet_backend",
        Model:     "wallet",
        Page:      page,
        Action:    action,
        IsSuccess: isSuccess,
    })
}
```

**注意事项**：
1. oneway 调用**不会返回错误**，但如果网络不通也不会有任何提示
2. 建议在 `defer` 中异步调用，避免影响主流程耗时
3. action 字段应包含足够的审计信息（谁做了什么、涉及什么对象、关键参数值）

### 3.2 ser-kyc — KYC 实名认证服务（012）

#### D02. KycQuery — 查询用户 KYC 状态

**IDL 定义**（`common/idl/ser-kyc/service.thrift`）：
```thrift
service KycService {
    // KYC 认证查询
    user_kyc.KycQueryResp KycQuery(1: user_kyc.KycQueryReq req);
}
```

**请求/响应结构体**（`common/idl/ser-kyc/user_kyc.thrift`）：
```thrift
struct KycQueryReq {
    1: optional i64 uid       // 用户 UID
}

struct KycQueryResp {
    1: optional UserKycInfo kycInfo  // KYC 信息（可能为空=未提交过）
}

struct UserKycInfo {
    1:  required i64 id
    2:  required i64 uid
    3:  required string name           // 真实姓名 ← 钱包用这个
    4:  required string country        // 国家
    5:  required string phone          // 电话
    6:  required i32 idType            // 证件类型：1身份证 2护照 3驾照
    7:  required string idNumber       // 证件号码
    8:  required string idFrontUrl     // 证件正面照
    9:  optional string idBackUrl      // 证件反面照
    10: required string idFrontS3Key
    11: optional string idBackS3Key
    12: required i32 status            // 状态：1去认证 2审核中 3已通过 4重新认证
    13: optional string rejectReason
    14: required i32 submitCount
    15: required i64 createAt
    16: required i64 updateAt
}
```

**调用特征**：

| 属性 | 值 |
|------|-----|
| 调用类型 | **同步阻塞** |
| 阻塞性 | **条件阻塞**（KYC 状态决定是否允许操作） |
| 优先级 | P0（首期必须，提现流程的前置校验） |
| 调用频率 | 低频（仅提现相关操作时调用） |
| RPC 客户端 | `rpc.KycClient()` ✅ 已注册 |

**触发场景**：

| 场景 | 对应接口 | 为什么需要 KYC | 使用的返回字段 |
|------|---------|---------------|-------------|
| **F11 发起提现** | `CreateWithdraw` | 需求文档 7-1：未认证用户只能 USDT 提现，法币提现必须已通过 KYC | `status`（3=已通过才允许法币提现） |
| **F10 绑定提现账户** | `AddWithdrawAccount` | 需求文档 7-2：银行卡/电子钱包的持有人姓名必须与 KYC 认证姓名一致 | `name`（真实姓名用于校验） |

**调用逻辑详解**：

**场景1：提现 KYC 校验**
```
用户发起提现(F11)
  → 从参数获取提现方式类型
  → 如果是法币提现（银行卡/电子钱包）：
    → rpc.KycClient().KycQuery(ctx, &KycQueryReq{Uid: userId})
    → 判断返回：
      → kycInfo == nil → 拒绝："请先完成实名认证"
      → kycInfo.Status != 3（已通过） → 拒绝："实名认证未通过，暂不支持法币提现"
      → kycInfo.Status == 3 → 通过，继续提现流程
  → 如果是 USDT 提现：
    → 跳过 KYC 校验（USDT 提现不要求实名）
```

**场景2：绑定提现账户持有人校验**
```
用户绑定银行卡/电子钱包提现账户(F10)
  → rpc.KycClient().KycQuery(ctx, &KycQueryReq{Uid: userId})
  → 判断返回：
    → kycInfo != nil && kycInfo.Status == 3：
      → 取出 kycInfo.Name（KYC 认证姓名）
      → 前端持有人姓名字段自动填充（只读）
      → 后端校验：请求中的 holderName 必须 == kycInfo.Name
    → 否则：
      → 允许手动输入持有人姓名（后续提现时再校验 KYC）
```

**错误处理策略**：
- KYC 服务不可用 → 返回业务错误 "系统繁忙，请稍后再试"，**不应默认放行**
- KYC 状态非 3（已通过） → 精确错误码区分（未认证 / 审核中 / 重新认证）

### 3.3 ser-user — 用户信息服务（008）

#### D03. GetUserInfoExpend — 获取单个用户信息

**IDL 定义**（`common/idl/ser-user/service.thrift`）：
```thrift
service UserService {
    // 获取单个用户信息
    user_rpc.UserInfoExpendResp GetUserInfoExpend(1: user_rpc.GetUserInfoReq req);
}
```

**请求/响应结构体**（`common/idl/ser-user/user_rpc.thrift`）：
```thrift
struct GetUserInfoReq {
    1: i64 userId    // 用户 ID（主键）
    2: i64 uid       // 用户编号（业务 UID）
}

struct UserInfoExpendResp {
    1:  i64 userId
    2:  i64 uid
    3:  string nickname       // 昵称
    4:  string mobile         // 手机号
    5:  string email          // 邮箱
    6:  i32 status            // 账号状态
    7:  i32 accountType       // 账号类型（1注册 2访客 3主播）
    8:  string avatar         // 头像
    9:  i64 exp               // 经验值
    10: i32 level             // 等级
    11: i32 totalFans         // 粉丝数
    12: string countryCode    // 国家码 ← 钱包关注这个
    13: i64 createAt          // 注册时间
    14: i32 gender            // 性别
}
```

**调用特征**：

| 属性 | 值 |
|------|-----|
| 调用类型 | **同步阻塞** |
| 阻塞性 | **弱阻塞**（失败可降级，但最好有数据） |
| 优先级 | P1（首期需要，但部分场景可降级处理） |
| 调用频率 | 低频 |
| RPC 客户端 | `rpc.UserClient()` ✅ 已注册 |

**触发场景**：

| 场景 | 对应接口 | 为什么需要用户信息 | 使用的返回字段 |
|------|---------|-----------------|-------------|
| **F10 绑定提现账户** | `AddWithdrawAccount` | 获取 KYC 认证姓名自动填充持有人 | `nickname`（辅助显示） |
| **B端查询用户钱包** | `QueryUserWallets` | B端展示用户信息 + 钱包余额 | `nickname`, `uid`, `accountType` |
| **自动开户** | `InitUserWallets` | 用户注册时根据国家码确定默认币种 | `countryCode` |

> **说明**：F10 绑定提现账户场景中，持有人姓名的权威来源是 **ser-kyc**（KycQuery 的 `name` 字段），而非 ser-user。ser-user 提供的是昵称/国家码等辅助信息。两者不要混淆。

**错误处理策略**：
- 用户信息查询失败时，根据场景决定是否阻塞：
  - 自动开户 → 可使用默认币种兜底
  - B端展示 → 返回部分数据（用户信息缺失，仅展示余额）

### 3.4 ser-s3 — 文件存储服务（007）

#### D04. UploadFile — 通用文件上传

**IDL 定义**（`common/idl/ser-s3/service.thrift`）：
```thrift
service S3Service {
    /**
     * 文件上传(用于小文件低频上传)
     */
    model.UploadFileResp UploadFile(1: model.UploadFileReq reqData);
}
```

**请求/响应结构体**（`common/idl/ser-s3/model.thrift`）：
```thrift
struct UploadFileReq {
    1: required i32 bsType       // 业务类型 → 需约定一个 wallet 专用类型
    2: required i64 ownerId      // 所属对象 ID → 币种 ID
    3: required string fileName  // 文件名 → "currency_1001_USD.png"
    4: required binary content   // 文件二进制数据
    6: required i32 duration     // 时长 → 0（图片不需要时长）
}

struct UploadFileResp {
    1: required string fileUrl   // 文件访问 URL ← 存入币种表 icon_url 字段
    2: required string s3Key     // S3 相对路径
}
```

#### D05. uploadIcon — 图标专用上传

**IDL 定义**：
```thrift
service S3Service {
    // 上传图标
    string uploadIcon(1: model.UploadIconReq reqData);
}

struct UploadIconReq {
    1: required i32 bsType         // 业务类型 → 1=静态PNG/JPG 2=动态GIF
    2: required binary content     // 文件二进制数据
}
```

**两个方法的选择**：
- `UploadFile`：通用上传，返回 UploadFileResp（含 fileUrl + s3Key），信息更完整
- `uploadIcon`：图标专用，仅返回 string（URL），更简单

**建议**：使用 `UploadFile`，因为它返回 s3Key，后续如需删除旧图标可以用 `DeleteFile(s3Key)` 清理。

**调用特征**：

| 属性 | 值 |
|------|-----|
| 调用类型 | **同步阻塞** |
| 阻塞性 | **条件阻塞**（仅上传图标时触发，失败不影响币种配置核心逻辑） |
| 优先级 | P2（可延后，初期币种配置可不上传图标） |
| 调用频率 | 极低（仅 B端管理员新增/编辑币种时偶尔触发） |
| RPC 客户端 | `rpc.S3Client()` ✅ 已注册 |

**触发场景**：

| 场景 | 对应接口 | 为什么需要上传 | 文件约束 |
|------|---------|-------------|---------|
| **B2 新增币种** | `CurrencyAdd` | 管理员上传币种图标（SVG/PNG） | ≤ 5KB，SVG/PNG 格式 |
| **B3 编辑币种** | `CurrencyEdit` | 管理员更换币种图标 | 同上 + 需删除旧图标 |

**调用流程**（B2 新增币种为例）：
```
B端管理员上传币种图标
  → [内部] 前置校验：
    → 文件大小 ≤ 5KB？
    → 文件格式是 SVG/PNG？
    → 校验失败 → 直接返回错误，不调 S3
  → rpc.S3Client().UploadFile(ctx, &ser_s3.UploadFileReq{
      BsType:   walletIconBsType,  // 需约定常量
```

```
    → [6] 冻结余额 → 本地 DB 事务
```

### 7.3 为什么要封装而不是直接调用

1. **隔离外部 IDL 变更**：如果 ser-finance 的 thrift 结构体改了字段名，只需改 rpc_finance.go 一处
2. **统一错误处理**：在封装层统一处理 `kerrors.FromBizStatusError`，service 层不需要关心 RPC 错误提取细节
3. **Mock 友好**：封装层可以通过 interface 抽象，单测时替换为 mock 实现
4. **日志统一**：在封装层统一打印 RPC 调用日志（调用方、参数、耗时、结果）

## 八、首期 vs 预留

### 8.1 首期必须对接的出站 RPC

| 编号 | 方法 | 首期必须的原因 | Mock 方案（对方未就绪时） |
|------|------|-------------|---------------------|
| D01 | AddActLog | 所有写操作需审计，合规要求 | 可运行无 mock（oneway 失败不影响） |
| D02 | KycQuery | 提现链路的前置校验 | 返回固定值：`status=3`（默认已认证），开发阶段先放行 |
| D06 | GetDepositMethods | 充值页面必须展示方式列表 | 返回硬编码方式列表（BSB:USDT+银行；VND:银行+电子钱包） |
| D07 | CreateDepositOrder | 充值核心链路 | 返回模拟订单号 + 模拟支付 URL |
| D08 | GetOrderStatus | 前端轮询必须有响应 | 返回固定状态：`status=2`（成功） |
| D09 | GetWithdrawMethods | 提现页面必须展示方式列表 | 同 D06 mock 策略 |
| D10 | CreateWithdrawOrder | 提现核心链路 | 返回模拟订单号 + `status=1`（审核中） |

### 8.2 可延后的出站 RPC

| 编号 | 方法 | 延后原因 | 替代方案 |
|------|------|---------|---------|
| D03 | GetUserInfoExpend | 仅辅助信息，核心流程不强依赖 | 自动开户默认币种暂用配置文件指定 |
| D04/D05 | UploadFile/uploadIcon | 币种图标是视觉需求，不影响功能 | 初期币种配置不上传图标，icon_url 留空 |

### 8.3 Mock 策略与切换机制

**推荐做法**：在 `rpc_finance.go` 中预留 mock 开关：

```go
// 通过配置控制是否使用 mock（ETCD 配置 or 启动参数）
var useFinanceMock = true // 初期默认 true，联调时切为 false

func GetDepositMethods(ctx context.Context, currencyCode string) ([]*DepositMethodItem, error) {
    if useFinanceMock {
        return mockDepositMethods(currencyCode), nil
    }
    // 真实 RPC 调用
    return realGetDepositMethods(ctx, currencyCode)
}
```

**好处**：
1. ser-wallet 可以独立开发、独立测试，不被 ser-finance 开发进度阻塞
2. 联调时只需改一个开关，不需要改业务代码
3. 符合用户"炒菜比喻"——调料可以后放，但火必须有（mock 就是替代调料）

### 9.1 P0 阻塞问题（必须在编码前与财务开发者对齐）

| Q3 | **提现冻结在谁的事务里？** | 两种方案：① 钱包先冻结再调财务建单 ② 财务建单时回调钱包冻结。影响 D10 的调用时机 | 链路关系/d-3.md 采用方案①（钱包先冻结），需与财务确认 |
| Q4 | **充值成功后谁通知钱包入账？** | 是财务主动回调钱包 WalletCredit（入站 RPC），还是钱包轮询 D08 后自行入账？ | 需确认回调机制 vs 轮询机制 |

### 9.3 不对称依赖观察

```
                    出站 RPC：10 方法（我调别人）
                              ↕
                    入站 RPC：25 方法（别人调我）

比例约 1:2.5 — ser-wallet 是典型的"多被调、少外调"服务
```

**这意味着**：
- ser-wallet 的**核心价值是提供资金操作能力**给其他模块
- 出站依赖虽少但**每个都很关键**（特别是 ser-finance 的 5 个方法）
- 开发策略应该是：**先把入站 RPC 骨架搭好**（因为自己能控制），然后**对出站依赖做 mock**，最后联调时替换为真实调用
