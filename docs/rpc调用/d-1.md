## 二、ser-wallet → ser-blog（审计日志）

### 2.1 AddActLog — 写操作审计日志

**目标模块**：ser-blog（006号服务）
**已有 RPC 客户端**：`rpc.BLogClient()`（common/rpc/rpc_client.go 已注册）
**调用方式**：oneway（单向，不等待响应，不阻塞主流程）

**什么时候调用**：

ser-wallet 中**所有写操作**执行完成后，都应记录审计日志。具体包括：

| 操作模块 | 操作页面 | 触发动作 | Action 描述示例 |
|---------|---------|---------|----------------|
| 币种配置 | 币种管理 | 创建币种 | "新增 币种管理 币种编号:1001_USD" |
| 币种配置 | 币种管理 | 编辑币种 | "编辑 币种管理 币种编号:1001_USD" |
| 币种配置 | 币种管理 | 启用/禁用 | "启用 币种管理 币种编号:1001_USD" |
| 币种配置 | 汇率管理 | 更新汇率 | "编辑 汇率管理 币对:USD/BSB" |
| 钱包 | 充值订单 | 创建充值订单 | "新增 充值订单 订单号:D202602280001" |
| 钱包 | 兑换 | 执行兑换 | "新增 兑换记录 订单号:E202602280001" |
| 钱包 | 提现订单 | 创建提现申请 | "新增 提现订单 订单号:W202602280001" |
| 钱包 | 提现账户 | 绑定提现账户 | "新增 提现账户 账户类型:银行卡" |
| 钱包 | 提现账户 | 删除提现账户 | "删除 提现账户 账户ID:12345" |
| 钱包 | 余额变动 | RPC加款入账 | "新增 余额变动 加款:用户123_金额5000_订单D001" |
| 钱包 | 余额变动 | RPC冻结 | "新增 余额变动 冻结:用户123_金额3000_订单W001" |
| 钱包 | 余额变动 | RPC解冻 | "新增 余额变动 解冻:用户123_金额3000_订单W001" |
| 钱包 | 余额变动 | RPC扣款 | "新增 余额变动 扣款:用户123_金额3000_订单W001" |
| 钱包 | 手动调账 | 手动调账 | "新增 手动调账 用户123_加款5000_原因:客诉补偿" |

**接口签名**（已有 IDL 定义）：

```thrift
// common/idl/ser-blog/service.thrift
oneway void AddActLog(1: model.ActAddReq req)

// common/idl/ser-blog/model.thrift
struct ActAddReq {
    1: string backend     // 所属后台（如 "wallet_backend"）
    2: string model       // 操作模块（如 "wallet"）
    3: string page        // 操作页面（如 "currency_config"）
    4: string action      // 操作描述（拼接的完整描述字符串）
    5: i32 isSuccess      // 1=成功 0=失败
}
```

**调用代码模式**（对标现有工程 ser-app/ser-buser 的写法）：

```go
// ser-wallet 内部调用模式
func addWalletLog(ctx context.Context, page string, action string, status int32) error {
    return rpc.BLogClient().AddActLog(ctx, &ser_blog.ActAddReq{
        Backend:   enum.WalletBackend,    // "wallet_backend"
        Model:     enum.WalletModel,      // "wallet"
        Page:      page,                  // 如 enum.CurrencyConfigPage
        Action:    action,                // 拼接的操作描述
        IsSuccess: status,                // enum.Success(1) / enum.Fail(0)
    })
}
```

**代码实证来源**：
- ser-app/internal/ser/banner.go（第714-722行）：`addBannerLog()` 函数
- ser-buser/internal/ser/user_service.go（第67-73行）：用户创建后写日志
- ser-buser/internal/ser/role_service.go（第31-37行）：角色创建后写日志

**为什么 ser-wallet 自己做不了**：合规要求审计日志集中存储在 ser-blog，由统一的日志服务管理，不在各业务服务中分散存储。

**依据文档**：流程思路/d-1.md §9.4、接口评估/d-1.md（D7 AddActLog）

## 三、ser-wallet → ser-kyc（KYC 认证查询）

### 3.1 KycQuery — 提现前 KYC 状态校验

**目标模块**：ser-kyc（012号服务）
**已有 RPC 客户端**：`rpc.KycClient()`（common/rpc/rpc_client.go 已注册）
**调用方式**：同步阻塞

**什么时候调用**：

| 触发场景 | 调用时机 | 业务目的 |
|---------|---------|---------|
| 用户点击"提现"按钮 | C端 CheckWithdrawEligibility 接口 | 判断用户是否有提现资格、可用哪些提现方式 |
| 用户首次绑定提现账户 | C端 BindWithdrawAccount 接口 | 获取认证姓名，自动填充"账户持有人"字段 |

**接口签名**（推断，基于现有 IDL 模式）：

```
RPC: KycQuery（具体方法名以 ser-kyc IDL 为准）

入参:
  userId         i64    required  用户ID

出参:
  kycStatus      i32    认证状态（1=未认证 2=审核中 3=已认证）
  realName       string 认证姓名（已认证时有值）
  certificateType i32   证件类型（如 1=身份证 2=护照）
```

**ser-wallet 内部使用方式**：

```
场景A — 提现资格判断（CheckWithdrawEligibility）：
  1. 调用 KycQuery 获取 kycStatus
  2. kycStatus = 已认证：
     → 返回所有可用提现方式（银行卡/电子钱包/USDT）
  3. kycStatus = 未认证：
     → 返回弹窗提示数据（"去认证" / "继续提现"）
     → 用户选择"继续提现"时仅展示 USDT 提现方式

场景B — 绑定提现账户（BindWithdrawAccount）：
  1. 调用 KycQuery 获取 realName
  2. 如果 realName 非空：
     → 前端"账户持有人"字段自动填充为 realName，设为只读
  3. 如果 realName 为空（未认证）：
     → 前端"账户持有人"字段允许手动输入
```

**为什么 ser-wallet 自己做不了**：KYC（实名认证）是独立的身份验证域，由专门的模块管理，钱包不拥有也不应该拥有用户的认证数据。

**依据文档**：需求理解/d-1.md（提现流程第429-441行、账户绑定第559-561行）、链路关系/d-1.md §6.2 提现链路、流程思路/d-1.md §6.2

## 四、ser-wallet → ser-finance（财务模块）

### 4.1 GetPaymentMethods — 获取可用支付方式

**目标模块**：ser-finance（编号待定，当前未在 rpc_client.go 中注册）
**调用方式**：同步阻塞

**什么时候调用**：

| 触发场景 | 调用时机 | 业务目的 |
|---------|---------|---------|
| 用户进入充值页面 | C端 GetDepositMethods 接口 | 展示该币种下可用的充值方式列表 |
| 用户进入提现页面 | C端 GetWithdrawMethods 接口 | 展示该币种下可用的提现方式列表 |

**接口签名**（基于财务结构/d-1.md 联调地图推导）：

```
RPC: GetPaymentMethods

入参 GetPaymentMethodsReq:
  currencyId     i64    required  币种ID
  methodType     i32    required  方式类型（1=充值方式 2=提现方式）

出参 GetPaymentMethodsResp:
  list           list<PaymentMethodItem>

PaymentMethodItem:
  methodId       i64    方式ID
  methodName     string 方式名称（如 "银行转账"、"USDT"）
  iconUrl        string 方式图标
  minAmount      i64    最低金额（最小单位）
  maxAmount      i64    最高金额（最小单位）
  sortOrder      i32    排序序号
  isRecommended  bool   是否推荐（如 QRIS 推荐标签）
```

**ser-wallet 内部使用方式**：

```
C端充值页面加载流程：
  1. 用户选择币种（或默认币种）
  2. ser-wallet 调用 ser-finance.GetPaymentMethods(currencyId, methodType=充值)
  3. ser-wallet 组装充值页面数据：
     → 币种信息（内部币种配置查询）
     → 支付方式列表（finance RPC 返回）
     → 充值配置（最低金额/快捷金额，内部配置查询）
  4. 返回给前端展示

产品原型展示的充值方式（按币种不同）：
  BSB：USDT、银行转账、电子钱包
  VND：银行转账（越南银行）
  IDR：电子钱包扫码（推荐）、印尼银行
  THB：电子钱包扫码（推荐）、泰国银行
```

**为什么 ser-wallet 自己做不了**：支付方式/通道的配置和管理属于财务模块的职责域。财务模块负责通道的启用/禁用、费率配置、轮询权重等。钱包只消费这些配置数据，不拥有它们。

**依据文档**：需求理解/d-1.md（充值方式总览第282-304行）、财务结构/d-1.md §16 联调地图、链路关系/d-1.md §6.1

### 4.2 MatchDepositChannel — 充值通道匹配

**目标模块**：ser-finance
**调用方式**：同步阻塞（**关键路径，失败则充值下单失败**）

**什么时候调用**：

| 触发场景 | 调用时机 | 业务目的 |
|---------|---------|---------|
| 用户确认充值金额并提交 | C端 CreateDepositOrder 接口内部 | 根据币种+支付方式+金额匹配最优支付通道 |

**接口签名**（基于财务结构/d-1.md 联调地图推导）：

```
RPC: MatchDepositChannel

入参 MatchDepositChannelReq:
  currencyId     i64    required  币种ID
  methodId       i64    required  支付方式ID（用户在前一步选择的）
  amount         i64    required  充值金额（最小单位）
  userId         i64    optional  用户ID（可能影响通道匹配策略）

出参 MatchDepositChannelResp:
  channelId      i64    匹配到的通道ID
  channelName    string 通道名称
  paymentUrl     string 支付跳转地址（如果通道直接返回）
  channelParams  string 通道参数（JSON，供后续回调解析用）
```

**ser-wallet 内部使用方式**：

```
CreateDepositOrder 核心流程：
  1. 校验充值金额（≥ 最低金额）
  2. 调用 ser-finance.MatchDepositChannel(currencyId, methodId, amount)
  3. 如果匹配成功：
     → 创建 deposit_order（记录 channelId）
     → 返回支付信息给前端
  4. 如果匹配失败（无可用通道）：
     → 返回错误："充值通道异常失效，请更换充值方式"
     → 不创建订单
```

**通道匹配算法**（由 ser-finance 内部实现，ser-wallet 不关心细节）：
- 综合评分 = 成功率 × 时间系数 × 告警因子
- 轮询策略：加权轮询（高评分通道优先）
- ser-wallet 只拿结果，不参与算法

**为什么 ser-wallet 自己做不了**：通道匹配涉及通道成功率统计、权重配置、告警状态等数据，全部属于财务模块管理。钱包不拥有这些数据。

**为什么是关键路径**：如果这个 RPC 调用失败，用户无法完成充值下单。这是整个充值链路中唯一的外部阻塞点。

**依据文档**：需求理解/d-1.md（充值流程图第603-628行）、财务结构/d-1.md §16 联调地图、流程思路/d-1.md §4.2

## 五、ser-wallet → ser-s3（文件服务）

### 5.1 UploadFile — 币种图标上传

**目标模块**：ser-s3（007号服务）
**已有 RPC 客户端**：`rpc.S3Client()`（common/rpc/rpc_client.go 已注册）
**调用方式**：同步阻塞

**什么时候调用**：

| 触发场景 | 调用时机 | 业务目的 |
|---------|---------|---------|
| B端管理员创建/编辑币种 | B端 CreateCurrency / UpdateCurrency | 上传币种图标文件 |

**接口签名**（基于现有 ser-app 使用模式推导）：

```
RPC: UploadFile（具体方法名以 ser-s3 IDL 为准）

入参:
  fileData       binary  文件二进制数据
  fileName       string  文件名
  contentType    string  MIME 类型（如 "image/png", "image/svg+xml"）
  bucket         string  存储桶名称

出参:
  fileUrl        string  文件访问 URL
```

**约束**：
- 文件大小限制 ≤ 5KB（产品需求要求）
- 文件格式：SVG / PNG
- 校验在 ser-wallet 侧执行（上传前检查大小和格式）

**为什么 ser-wallet 自己做不了**：文件存储是基础设施能力，由 ser-s3 统一管理（对接 MinIO/AWS S3）。所有模块共享同一套文件服务。

**优先级**：P2（不阻塞核心充提流程，币种配置可先用默认图标占位）

**依据文档**：需求理解/d-1.md（币种图标上传，5K限制约第140-141行）

## 六、不需要调用的模块（排除分析）

以下模块经过分析确认 ser-wallet **不需要主动调用**：

### 6.1 ser-user（C端用户服务）

**是否需要调用**：否

**原因**：
- 用户 ID 和用户名通过 JWT Token 自动注入上下文，使用 `tracer.GetUserId(ctx)` / `tracer.GetUserName(ctx)` 直接获取
- 钱包不需要展示其他用户的信息（不是社交场景）
- 余额查询基于自身数据库，不依赖用户服务

**代码实证**：现有模块（ser-app、ser-buser）也是通过 tracer 获取当前用户信息，不额外调用 ser-user。

### 6.2 ser-buser（B端用户服务）

**是否需要调用**：否

**原因**：
- B端操作人信息同样通过 JWT Token + tracer 获取
- 审计日志的 action 字段使用 `tracer.GetUserName(ctx)` 获取操作人名称
- 不需要批量查询其他 B端用户信息

**代码实证**：ser-buser/internal/ser/user_service.go 中审计日志使用 `req.UserName`（请求参数）而非调用其他服务。

### 6.3 ser-bcom（配置中心）

**是否需要调用**：否

**原因**：
- ser-wallet 的配置通过 ETCD 直接读取（`etcd.DirectGet()`），不经过 ser-bcom
- 币种配置是 ser-wallet 自己管理的数据，不在 ser-bcom 中

### 6.4 ser-app（基础服务）

**是否需要调用**：否

**原因**：
- ser-app 提供 Banner/分类等功能，与钱包无业务关联

### 6.5 投注/游戏模块

**是否需要调用**：否

**原因**：
- 当前需求范围内，钱包不主动调用游戏/投注模块
- 投注扣款是由投注模块调用钱包（方向反了），且当前不在 ser-wallet 开发范围内

### 6.6 ser-finance 的 MatchWithdrawChannel

**是否需要调用**：否（重要区分）

**原因**：
- 提现通道匹配由 **ser-finance 在审核通过后自行执行**
- ser-wallet 创建提现订单后，将订单推送给 ser-finance 进行审核
- 审核通过后 ser-finance 自己匹配出款通道、发起三方出款
- 出款结果通过 DebitBalance/UnfreezeBalance 回调通知 ser-wallet
- 所以：**ser-wallet 不需要主动调用 MatchWithdrawChannel**

**依据**：流程思路/d-1.md 提现流程——钱包负责冻结+建单，财务负责审核+匹配通道+出款

## 七、依赖关系与开发优先级

### 7.1 依赖阻塞分析

```
P0 阻塞关系（没有则核心流程跑不通）：

  充值流程：
    CreateDepositOrder
      ├── 依赖 D03 GetPaymentMethods（展示可用方式）
      └── 依赖 D04 MatchDepositChannel（匹配通道）
    → 充值流程完全依赖 ser-finance

  提现流程：
    CheckWithdrawEligibility
      └── 依赖 D02 KycQuery（KYC 校验）
    → 提现入口依赖 ser-kyc

  所有写操作：
    → 依赖 D01 AddActLog（审计日志）
    → 但 oneway 不阻塞，即使 ser-blog 不可用也不影响主流程

P2 非阻塞：

  币种配置：
    CreateCurrency / UpdateCurrency
      └── 依赖 D05 UploadFile（图标上传）
    → 可先用默认图标，后续补传
```

### 7.2 Mock 策略

基于编码阶段准则（"调料可缺，火不可缺"），各依赖的 Mock 可行性：

| 依赖 | Mock 可行性 | Mock 方式 | 说明 |
|------|-----------|----------|------|
| D01 ser-blog.AddActLog | 可 Mock | 空实现（不写日志） | oneway 本身不阻塞，开发阶段可跳过 |
| D02 ser-kyc.KycQuery | 可 Mock | 固定返回"已认证" | 提现流程可先跑通，KYC 逻辑后补 |
| D03 ser-finance.GetPaymentMethods | 可 Mock | 固定返回测试方式列表 | 充值页面展示可先用假数据 |
| D04 ser-finance.MatchDepositChannel | **需澄清** | 返回固定测试通道 | 充值下单可先跑通，但**实际充值到账需要真实通道** |
| D05 ser-s3.UploadFile | 可 Mock | 返回固定图标 URL | 币种图标非关键路径 |

**"火"级别的依赖**：

D04 MatchDepositChannel 在端到端验证时**不能 Mock**——如果需要验证"用户充值→三方支付→回调到账"全链路，必须有真实的 ser-finance 通道匹配和三方对接。这属于联调阶段必须解决的问题，在单模块开发阶段可以 Mock。

### 7.3 开发顺序建议

```
第一阶段（可独立开发，无外部依赖）：
  ├── 币种配置 CRUD（不含图标上传，用固定URL）
  ├── 汇率管理 CRUD
  ├── 钱包账户管理
  ├── balance_engine 余额引擎
  ├── 暴露的 RPC 接口实现（CreditBalance/DebitBalance/Freeze/Unfreeze 等）
  └── 交易流水查询

第二阶段（需 Mock 外部依赖）：
  ├── 充值流程（Mock GetPaymentMethods + MatchDepositChannel）
  ├── 提现流程（Mock KycQuery）
  ├── 兑换流程（内部币种配置已就绪）
  └── 审计日志接入（调用 ser-blog）

第三阶段（联调阶段，真实外部服务）：
  ├── 与 ser-finance 联调充值全链路
  ├── 与 ser-finance 联调提现全链路
  ├── 与 ser-kyc 联调 KYC 校验
  └── 与 ser-s3 联调图标上传
```

## 九、风险与待确认事项

| 事项 | 当前状态 | 风险级别 | 说明 |
|------|---------|---------|------|
| ser-finance 的 RPC 客户端尚未注册 | rpc_client.go 中无 finance 客户端 | 低 | 编码时按模式新增即可，不影响设计 |
| KycQuery 的具体方法签名未确认 | 基于 KycClient 已存在推断 | 低 | 需查看 ser-kyc IDL 确认 |
| GetPaymentMethods 的返回结构未与财务同事对齐 | 基于需求推导 | 中 | 联调前必须双方确认接口契约 |
| MatchDepositChannel 的返回结构未与财务同事对齐 | 基于需求推导 | 中 | 同上 |
| 充值活动奖金数据源未确定 | 可能来自活动模块或钱包内部配置 | 低 | P1 功能，不阻塞核心流程 |

> 以上待确认事项均为"调料"级别——不影响骨架搭建和核心模块开发，在联调阶段确认即可。
