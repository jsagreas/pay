# ser-wallet 需主动调用的外部 RPC 接口分析

> 分析目标：识别 ser-wallet 模块在完成自身业务逻辑时，需要**主动调用**哪些外部模块的 RPC 接口
> 视角方向：与"rpc暴露"相反 —— 上次分析的是"别人调我"，这次分析"我调别人"
> 依据来源：需求文档/原型（~136 份）+ 链路关系分析 + 依赖关系分析 + 工程代码模式
> 分析方法：从每个业务场景出发，追踪 ser-wallet 的接口在处理过程中需要向外请求什么数据或能力

---

# 一、分析思路

"我调别人"的识别不是凭空猜测，而是从三个维度交叉确认：

1. **链路关系/c-2.md 第8.1节**：已逐接口标注了"调用出去"列，明确列出了哪些接口需要调外部
2. **链路关系/c-2.md 第10.4节**：汇总了所有调出去的接口统计
3. **需求文档**：充值流程图（2-3充值流程）、提现流程图（2-4提现流程）、人工修正（2-5）等，明确标注了跨系统交互节点
4. **工程代码模式**：现有22个服务的 RPC 调用模式（如 BLog 审计日志、S3 文件上传等通用基础设施调用）

上次 rpc暴露 分析的是"被动能力提供"——别人需要我的 Credit/Debit/QueryBalance 等能力。
这次分析的是"主动依赖获取"——我的业务流程走不通时，需要去找谁要什么。

**核心区别**：
- rpc暴露的接口（如 Credit/Debit）→ **自身逻辑完整，不调出去**，纯粹被动响应
- rpc调用的接口（如 GetDepositMethods）→ **自身逻辑不完整，必须向外获取数据/能力才能继续**

# 三、逐个外部调用分析

## 3.1 财务管理模块（ser-finance）— 最核心的外部依赖

> ser-finance 是 ser-wallet 最重要的外部依赖。钱包的充值和提现功能的完整闭环，必须通过财务模块的通道调度和审核能力才能实现。
> **注意**：ser-finance 目前在工程中尚未创建，需要等财务团队开发。

### CALL-01: GetDepositMethods（获取充值方式列表）

**调用场景**（需求文档依据）：

| 场景 | 触发接口 | 需求依据 |
|------|---------|---------|
| 用户打开充值页面 | C端 GetDepositMethods | 充值方式总览（5-1）：页面展示可用充值方式列表（银行转账/QRIS/USDT等） |

**调用链路**（来源：链路关系/c-2.md 4.1节）：
```
用户打开充值页
  │
  ▼
ser-wallet.GetDepositMethods(currencyId)
  │
  ├──→ [内部] CurrencyService.GetUserCurrentCurrency()
  │
  └──→ [RPC 调出] 财务模块.GetDepositMethods(currencyId)
           │
           └── 返回: [{方式名称, 方式类型, icon, 最小金额, 最大金额, 排序值}]
```

**我需要传什么**：`currencyId`（当前币种ID）
**我需要拿到什么**：该币种下已启用的充值方式列表（方式名称、类型、icon、金额范围）
**如果调不通**：充值页面无法展示，用户无法充值（**阻塞性依赖**）

---

### CALL-02: MatchDepositChannel（匹配代收通道）

**调用场景**：

| 场景 | 触发接口 | 需求依据 |
|------|---------|---------|
| 用户提交充值订单 | C端 CreateDepositOrder | 充值流程图（2-3）：通道匹配→按权重轮询选择代收通道 |

**调用链路**（链路关系/c-2.md 4.2节）：
```
用户提交充值
  │
  ▼
ser-wallet.CreateDepositOrder(currencyId, methodId, amount, bonusActivityId?)
  │
  ├──→ [内部] 重复转账拦截检查
  │
  ├──→ [RPC 调出] 财务模块.MatchDepositChannel(currencyId, methodId, amount)
  │         │
  │         ├── 财务内部按轮询规则选通道
  │         └── 返回: {channelId, channelCode, ...} 或 "通道繁忙"错误
  │
  └──→ 接着调 CALL-03（创建通道订单）
```

**我需要传什么**：`currencyId, methodId, amount`
**我需要拿到什么**：匹配到的代收通道信息（channelId、通道编码等）
**如果调不通**：无法创建充值订单，用户充值失败

---

### CALL-03: CreateChannelOrder（创建通道订单）

**调用场景**：

| 场景 | 触发接口 | 需求依据 |
|------|---------|---------|
| 匹配通道成功后创建三方支付订单 | C端 CreateDepositOrder（续） | 充值流程图（2-3）：创建通道订单→获取支付链接/二维码 |

**调用链路**：
```
匹配通道成功
  │
  ▼
[RPC 调出] 财务模块.CreateChannelOrder(channelId, userId, amount, orderNo)
  │
  ├── 财务内部创建支付订单、调三方通道API
  └── 返回: {thirdPartyOrderNo, payUrl/qrCode, expireTime}
```

**我需要传什么**：`channelId, userId, amount, orderNo`
**我需要拿到什么**：三方支付链接或二维码地址、有效期
**如果调不通**：用户无法跳转到支付页面

**CALL-01/02/03 三者的关系**：
```
GetDepositMethods → MatchDepositChannel → CreateChannelOrder
     展示阶段              提交阶段              提交阶段
     可独立调用             必须前一步成功          必须前一步成功
```

这三个调用构成了充值流程的"钱包 → 财务"方向的完整链路。

### CALL-04: GetWithdrawMethods（获取提现方式列表）

**调用场景**：

| 场景 | 触发接口 | 需求依据 |
|------|---------|---------|
| 用户打开提现页面 | C端 GetWithdrawMethods | 提现模块（7-1）：展示可用提现方式列表 |

**调用链路**（链路关系/c-2.md 5.1节）：
```
用户打开提现页
  │
  ▼
ser-wallet.GetWithdrawMethods(currencyId)
  │
  ├──→ [RPC 调出] 财务模块.GetWithdrawMethods(currencyId)
  │         │
  │         └── 返回: [{方式名称, 方式类型, icon, 排序值}]
```

**如果调不通**：提现页面无法展示

**注意**：拿到方式列表后还要结合用户的资金来源规则和KYC状态做二次过滤（内部逻辑），最终展示给用户的是过滤后的子集。

### CALL-05: SubmitWithdrawRequest（提交提现到审核流程）

**调用场景**：

| 场景 | 触发接口 | 需求依据 |
|------|---------|---------|
| 用户提交提现申请（冻结后） | C端 CreateWithdrawOrder | 提现流程图（2-4）：提交到财务三级审核 |

**调用链路**（链路关系/c-2.md 5.2节）：
```
用户提交提现
  │
  ▼
ser-wallet.CreateWithdrawOrder(currencyId, methodId, amount, receiverInfo)
  │
  ├──→ [内部] 余额校验 + 限额校验 + 资金来源规则校验
  ├──→ [内部] WalletService.Freeze(userId, currencyId, amount) — 冻结金额
  ├──→ [内部] OrderService.CreateWithdrawOrder — 创建钱包侧订单
  │
  └──→ [RPC 调出] 财务模块.SubmitWithdrawRequest(
           orderNo, userId, currencyId, amount, methodId, receiverInfo)
           │
           └── 返回: {financeOrderNo, status="风控审核中"}
```

**我需要传什么**：`orderNo, userId, currencyId, amount, methodId, receiverInfo(收款信息)`
**我需要拿到什么**：财务侧订单号、审核状态
**如果调不通**：提现申请无法进入审核流程

**关键时序**：先冻结金额，再提交审核。如果提交审核RPC调用失败，需要回滚冻结（Unfreeze）。

### CALL-06: QueryChannelOrderStatus（查询通道订单状态）

**调用场景**：

| 场景 | 触发接口 | 需求依据 |
|------|---------|---------|
| 用户查看充值订单当前状态 | C端 GetDepositOrderStatus | 充值总览（5-1）：用户查看充值进度 |
| 定时任务检查未完成的充值订单 | 定时任务 | 超时处理：未支付订单过期关闭 |

**调用链路**：
```
ser-wallet.GetDepositOrderStatus(orderNo)
  │
  └──→ [RPC 调出] 财务模块.QueryChannelOrderStatus(orderNo)
           │
           └── 返回: {status, actualAmount?, thirdPartyInfo?}
```

**我需要传什么**：`orderNo`（钱包侧充值订单号）
**我需要拿到什么**：通道侧最新状态（待支付/支付中/成功/失败/超时）
**必要性判断**：这个调用主要用于**主动查询**，而充值成功的核心路径是财务回调钱包 Credit（被动等通知）。所以这个接口是"锦上添花"而非"核心必须"。

### 财务模块调用汇总

```
                    必要性              调用时机              调用方向
                    ─────              ────────              ────────

CALL-01 GetDepositMethods      核心必须    用户打开充值页        钱包 → 财务
CALL-02 MatchDepositChannel    核心必须    用户提交充值          钱包 → 财务
CALL-03 CreateChannelOrder     核心必须    匹配通道成功后        钱包 → 财务
CALL-04 GetWithdrawMethods     核心必须    用户打开提现页        钱包 → 财务
CALL-05 SubmitWithdrawRequest  核心必须    用户提交提现          钱包 → 财务
CALL-06 QueryChannelOrderStatus 应该有     查询充值进度          钱包 → 财务

充值方向: 钱包主动调财务 3次（展示1 + 提交2）
提现方向: 钱包主动调财务 2次（展示1 + 提交1）
状态查询: 1次（非核心路径）
```

## 3.2 KYC 认证服务（ser-kyc）— 提现流程的前置校验

### CALL-07: KycQuery（查询用户KYC认证状态）

**调用场景**（需求文档依据）：

| 场景 | 触发接口 | 需求依据 |
|------|---------|---------|
| 用户打开提现页面 | C端 CheckWithdrawEligibility | 提现模块（7-1）：法币和电子钱包提现需完成KYC认证 |
| 提现页面引导KYC | 前端联动 | 提现触发认证（9-1~9-4）：未认证用户点提现→引导去KYC |

**调用链路**（链路关系/c-2.md 5.1节）：
```
用户打开提现页
  │
  ▼
ser-wallet.CheckWithdrawEligibility(currencyId)
  │
  ├──→ [RPC 调出] ser-kyc.KycQuery(userId)
  │         │
  │         └── 返回: KycQueryResp {
  │                kycInfo: {
  │                    status: i32,        // 认证状态：待审核/已通过/已拒绝
  │                    name: string,       // 实名姓名
  │                    idType: i32,        // 证件类型
  │                    submitCount: i32,   // 提交次数
  │                }
  │            }
  │
  ├──→ [内部] 资金来源规则判断
  │
  └──→ 综合判断：
         · 法币/电子钱包提现 + 未KYC → "需要先完成KYC"
         · 纯USDT充值用户 → 只允许USDT提现（USDT提现可不做KYC）
```

**如果调不通的降级策略**：
- 方案A：KYC服务不可用时，禁止所有法币/电子钱包提现，只允许USDT提现

## 3.3 操作审计日志服务（ser-blog）— B端操作合规记录

### CALL-08: AddActLog（记录B端操作日志）

**调用场景**（工程模式依据）：

| 场景 | 触发接口 | 依据 |
|------|---------|------|
| 管理员创建/编辑/启用/禁用币种 | B端 CreateCurrency/UpdateCurrency/ToggleCurrencyStatus | 工程惯例：所有B端写操作需审计 |
| 管理员修改汇率配置 | B端 UpdateRateConfig | 同上 |
| 管理员查看用户钱包信息 | B端 AdminGetUserWallet | 同上（敏感数据查看也需审计） |
| 其他B端管理操作 | 所有B端写接口 | 同上 |

**工程现有模式**（已确认的通用做法）：

现有服务的 B 端操作都通过 `rpc.BLogClient().AddActLog()` 记录审计日志。这是一个 **oneway** 调用（异步发送，不等响应），不影响主流程性能。

```go
// 现有服务的调用模式（如 ser-bcom/internal/ser/）
rpc.BLogClient().AddActLog(ctx, &ser_blog.ActAddReq{
    Backend: "wallet-admin",     // 所属后台
    Model:   "币种配置",          // 操作模块
    Page:    "币种管理",          // 操作页面
    Action:  "创建币种: VND",     // 操作行为
    IsSuccess: 1,                // 操作状态
})
```

**已有 IDL 定义**：
```thrift
// 位置: /Users/mac/gitlab/common/idl/ser-blog/service.thrift
service BLogService {
    oneway void AddActLog(1: model.ActAddReq req);
}

// 位置: /Users/mac/gitlab/common/idl/ser-blog/model.thrift
struct ActAddReq {
    1: string backend      // 所属后台
    2: string model        // 操作模块
    3: string page         // 操作页面
    4: string action       // 操作行为
    5: i32 isSuccess       // 操作状态：成功=1/失败=0
}
```

**已有 RPC Client**：
```go
// 位置: /Users/mac/gitlab/common/rpc/rpc_client.go 第200行
func BLogClient() blogservice.Client  // 编号 006
```

**特点**：
- **oneway 调用**：客户端发送后不等响应，不阻塞主流程
- **失败不影响业务**：日志记录失败不应导致业务操作失败
- **覆盖范围广**：所有 B 端写操作 + 敏感数据查看操作

## 3.5 C端用户服务（ser-user）— 用户信息查询

### CALL-10: GetUserInfoExpend / BatchGetUserInfo（获取用户信息）

**调用场景**：

| 场景 | 触发接口 | 依据 |
|------|---------|------|
| B端管理员查看用户钱包时需展示用户基本信息 | B端 AdminGetUserWallet | 链路关系8.2：需要展示昵称、手机号、头像等 |
| C端钱包页面展示用户简要信息 | C端 GetWalletInfo（可选） | 钱包总体设计（4-1）：页面可能需要用户昵称/头像 |
| 提现流程中需要用户实名信息做比对 | C端 CreateWithdrawOrder（可选） | 提现限制（7-5）：可能需比对用户姓名与收款人姓名 |

**已有 IDL 定义**：
```thrift
// 位置: /Users/mac/gitlab/common/idl/ser-user/service.thrift
service UserService {
    // 获取单个用户信息（含扩展：头像、等级、粉丝数等）
    user_rpc.UserInfoExpendResp GetUserInfoExpend(1: user_rpc.GetUserInfoReq req);

    // 批量获取用户信息
```

## 3.9 第三方汇率 API — 外部 HTTP 调用（非 RPC）

### CALL-EXT: 三方汇率 API × 3

**调用场景**：

| 场景 | 触发方式 | 需求依据 |
|------|---------|---------|
| 定时拉取实时汇率 | 定时任务 SyncThirdPartyRates | 币种配置（3-汇率配置）：定时从多个数据源获取实时汇率 |

**调用链路**（链路关系/c-2.md 2.3节）：
```
定时触发（分钟级）
  │
  ▼
SyncThirdPartyRates
  │
  ├──→ [外部 HTTP] ThirdPartyAPI_1.GetRate(currency → USDT)
  ├──→ [外部 HTTP] ThirdPartyAPI_2.GetRate(currency → USDT)
  ├──→ [外部 HTTP] ThirdPartyAPI_3.GetRate(currency → USDT)
  │
  ├──→ 计算平均值
  └──→ 更新实时汇率 → 触发阈值检查 → 可能更新平台汇率
```

**注意**：这不是 RPC 调用，是 HTTP/REST 方式调用外部第三方 API。不需要在 rpc_client.go 注册，但需要在代码中实现 HTTP Client。

**降级策略**：
- 单个 API 不可用 → 用其他两个的平均值
- 全部不可用 → 使用最近一次缓存的实时汇率
- 长时间不可用 → 告警通知运维

```
                                                  ← TransferToCenter (稽核转入)

汇总:
  调出: 13个调用点（含待确认2个）
  调入: 14个RPC接口（rpc暴露已分析）
```

**关键发现**：
1. **调出方向集中在财务模块**（6/13），这是最核心也最受制于人的依赖
2. **调入方向更分散**（投注、直播、活动、财务都调我），说明钱包是平台核心基础能力
3. **双向耦合只存在于钱包↔财务**：钱包调财务获取通道能力，财务调钱包操作余额
4. **其他服务的调用都是单向的**：要么我调他（KYC/S3/BLog），要么他调我（投注/直播）
