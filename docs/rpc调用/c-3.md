## 二、分类说明

在分析之前，先明确三种不同性质的"依赖"：

### 2.1 跨模块RPC调用（本文重点）

通过 `rpc.XxxClient().Method(ctx, req)` 发起的Kitex RPC调用，目标是**另一个独立部署的微服务**。

特征：网络调用、有超时风险、需要对方服务就绪、需要在 `init.go` 中初始化RPC客户端。

### 2.2 同进程内部调用（非RPC）

币种配置模块和钱包模块虽然在需求上是两个"模块"，但在工程上是**同一个ser-wallet进程**。币种配置的Service层方法直接通过Go包调用，不走RPC。

特征：函数调用、无网络开销、无超时风险、自己控制。

### 2.3 外部三方服务（非RPC）

调用外部三方API（汇率数据源、支付通道），走HTTP/WebSocket等协议，不走Kitex RPC。

特征：外网调用、需要鉴权、超时/重试策略独立管理。

**本文重点分析第一类（跨模块RPC调用）**，第二、三类简要说明。

### 3.1 F1：GetRechargeMethodList — 获取充值方式列表

**要解决的问题**：用户进入充值页面时，钱包需要知道当前币种支持哪些充值方式（USDT/银行转账/扫码支付等）。这些充值方式由**财务模块**管理配置（关联代收通道），钱包自己没有这份数据。

**需求依据**：
- 需求文档 5-1 充值方式总览：不同币种支持不同充值方式（越南=银行转账，印尼=QRIS扫码+银行等）
- 财务需求文档 2-2 支付配置-1 充值方式：充值方式按币种配置，关联多个代收通道

**触发场景**：

| 用户操作 | 钱包做什么 | 调用什么 |
|----------|-----------|---------|
| C端 → 进入充值页面 | 展示可用充值方式列表 | `rpc.FinanceClient().GetRechargeMethodList(ctx, req)` |

**调用频率**：中等（每次用户进入充值页面触发，可做短时缓存）

**请求/响应结构（预期）**：
```
请求：
  - currencyCode: string    // 当前币种（如"BSB"、"VND"）
  - region: string          // 地区（可选，用于过滤可用方式）

响应：
  - methods: list<RechargeMethod>
    - methodType: i32       // 1=USDT 2=银行转账 3=扫码
    - methodName: string    // 展示名称
    - iconUrl: string       // 图标
    - minAmount: string     // 最低充值金额
    - maxAmount: string     // 最高充值金额
    - channels: list<ChannelBrief>  // 可用通道列表
```

**若财务模块不可用**：用户无法看到充值方式 → 充值功能不可用。但钱包余额查询、兑换、提现等功能**不受影响**。

---

### 3.2 F2：GetWithdrawMethodList — 获取提现方式列表

**要解决的问题**：用户进入提现页面时，钱包需要知道当前币种支持哪些提现方式（银行卡/电子钱包/USDT等）。同样由财务模块管理配置。

**需求依据**：
- 需求文档 7-1 提现模块：不同地区有不同提现方式（越南=银行卡T+1，印尼=GoPay/OVO/DANA/银行T+1等）
- 财务需求文档 2-2 支付配置-2 提现方式：提现方式按币种配置，关联代付通道

**触发场景**：

| 用户操作 | 钱包做什么 | 调用什么 |
|----------|-----------|---------|
| C端 → 进入提现页面 | 展示可用提现方式列表 | `rpc.FinanceClient().GetWithdrawMethodList(ctx, req)` |

**注意**：此接口通常与 KYC 状态联合使用——先查KYC状态，再根据认证级别过滤可用提现方式（未认证仅USDT，已认证全渠道）。

**请求/响应结构（预期）**：
```
请求：
  - currencyCode: string    // 当前币种
  - kycLevel: i32           // KYC级别（钱包先从ser-kyc获取后传入，或由财务自己查）

响应：
  - methods: list<WithdrawMethod>
    - methodType: i32       // 1=银行卡 2=电子钱包 3=USDT
    - methodName: string
    - iconUrl: string
    - walletTypes: list<string>  // 如 ["GoPay","OVO","DANA"]（电子钱包类型）
    - arrivalTime: string        // "T+1"、"即时"
```

**若财务模块不可用**：提现功能不可用。

---

### 3.3 F3：GetWithdrawConfig — 获取提现限额与费率配置

**要解决的问题**：用户在提现页面输入金额时，钱包需要校验是否超出限额，并展示手续费和到账时间。这些配置由财务模块管理。

**需求依据**：
- 需求文档 7-5 提现限制："后台可配：单笔限额、单日限额、手续费率、T+N到账时间"
- 财务需求文档 2-2 支付配置-2 提现方式：每种提现方式关联限额和费率

**触发场景**：

| 用户操作 | 钱包做什么 | 调用什么 |
|----------|-----------|---------|
| C端 → 提现页面展示限额信息 | 获取并展示限额/费率 | `rpc.FinanceClient().GetWithdrawConfig(ctx, req)` |
| C端 → 用户输入提现金额后 | 校验金额是否在限额内 | 使用已缓存的配置数据校验 |
| C端 → 点击确认提现前 | 展示"手续费X，到账Y" | 使用已缓存的配置数据计算 |

**请求/响应结构（预期）**：
```
请求：
  - currencyCode: string
  - withdrawMethodType: i32  // 1银行卡 2电子钱包 3USDT

响应：
  - singleMinAmount: string   // 单笔最低
  - singleMaxAmount: string   // 单笔最高
  - dailyMaxAmount: string    // 单日最高
  - feeRate: string           // 手续费率（如"1.00"表示1%）
  - fixedFee: string          // 固定手续费（如果有）
  - arrivalDays: i32          // T+N 中的N
  - dailyUsed: string         // 当日已用额度（需要钱包传userId？或由财务查）
```

**关键业务逻辑**：
- 手续费计算：提现金额 × feeRate / 100 = 手续费
- 到账金额 = 提现金额 - 手续费
- 限额校验：singleMinAmount ≤ 提现金额 ≤ singleMaxAmount，且当日累计 ≤ dailyMaxAmount

---

### 3.4 F4：SubmitWithdrawOrder — 提交提现订单到财务审核

**要解决的问题**：用户确认提现后，钱包先冻结用户余额（本地操作），然后需要把提现订单提交给财务模块进入审核流程（风控审核→财务审核→通道出款）。

**需求依据**：
- 财务需求文档 2-4 提现记录-1 提现流程：提现订单进入"待风控审核→待财务审核→待出款→出款中→已完成"
- 依赖关系分析 c-3.md 第3.2节：步骤⑩"钱包→财务.提交提现订单"

**触发场景**：

| 用户操作 | 钱包做什么 | 调用什么 |
|----------|-----------|---------|
| C端 → 确认提现 | ①冻结余额(本地) ②提交财务审核 | `rpc.FinanceClient().SubmitWithdrawOrder(ctx, req)` |

**请求/响应结构（预期）**：
```
请求：
  - orderNo: string          // 提现单号 T+16位（钱包生成）
  - userId: i64
  - currencyCode: string
  - amount: string           // 提现金额
  - feeAmount: string        // 手续费
  - actualAmount: string     // 到账金额 = amount - feeAmount
  - withdrawMethod: i32      // 1银行卡 2电子钱包 3USDT
  - accountInfo: WithdrawAccountInfo  // 提现账户信息（银行名/卡号/户名 等）
  - exchangeRate: string     // 锁汇汇率（如果涉及跨币种）

响应：
  - success: bool
  - financeOrderNo: string   // 财务侧订单号（可选）
```

**关键风险**：
- 如果冻结成功但提交财务失败 → **必须回滚解冻**
- 实现代码中必须有 `defer` 或事务补偿逻辑确保一致性

**调用时序**：
```
用户点击确认提现
    │
    ▼
①  钱包.FreezeBalance(本地)  ──成功──┐
    │                               │
    │  失败→ 返回错误，不走下一步     │
    │                               ▼
②  rpc.FinanceClient().SubmitWithdrawOrder(ctx, req)
    │
    ├── 成功 → 返回用户"提现进行中"
    │
    └── 失败 → 钱包.UnfreezeBalance(本地回滚) → 返回用户"提现失败"
```

### 4.1 K1：KycQuery — 查询用户KYC认证状态

**要解决的问题**：用户发起提现时，钱包需要知道用户是否已通过KYC认证。KYC状态直接决定可用提现渠道：
- **未认证（status=1）**：仅允许USDT提现
- **审核中（status=2）**：仅允许USDT提现（等同未认证）
- **已通过（status=3）**：全渠道可用（银行卡+电子钱包+USDT）
- **重新认证（status=4）**：等同未认证

**需求依据**：
- 需求文档 7-1 提现模块："KYC为可选非强制，但作为解锁高阶提现渠道的必要前提。未认证仅USDT提现，认证后全渠道"
- 需求文档 9 非功能性需求："KYC强制校验"
- 需求文档 7-2 银行卡提现 / 7-3 电子钱包提现："账户持有人 = KYC认证姓名（置灰不可编辑）"

**触发场景**：

| 用户操作 | 钱包做什么 | 调用什么 |
|----------|-----------|---------|
| C端 → 点击"提现" | 查询KYC状态，决定展示页面 | `rpc.KycClient().KycQuery(ctx, req)` |
| C端 → 银行卡/电子钱包提现填写 | 自动填充"账户持有人"为KYC认证姓名 | 使用上一步已获取的KYC数据 |

**已有的实际接口签名**（从IDL读取）：

```thrift
// ser-kyc/service.thrift
KycQueryResp KycQuery(1: user_kyc.KycQueryReq req)

// user_kyc.thrift
struct KycQueryReq {
    1: optional i64 uid    // 用户UID
}

struct KycQueryResp {
    1: optional UserKycInfo kycInfo
}

struct UserKycInfo {
    1: required i64 id
    2: required i64 uid
    3: required string name         // ← 这就是"KYC认证姓名"
    4: required string country
    5: required string phone
    6: required i32 idType          // 1身份证 2护照 3驾照
    7: required string idNumber
    8: required string idFrontUrl
    9: optional string idBackUrl
    10: required string idFrontS3Key
    11: optional string idBackS3Key
    12: required i32 status         // 1去认证 2审核中 3已通过 4重新认证
    13: optional string rejectReason
    14: required i32 submitCount
    15: required i64 createAt
    16: required i64 updateAt
}
```

**钱包侧调用代码（示意）**：
```go
// 伪代码
kycReq := &user_kyc.KycQueryReq{Uid: &userId}
kycResp, err := rpc.KycClient().KycQuery(ctx, kycReq)
if err != nil {
    // RPC失败 → 降级为"未认证"状态，仅允许USDT提现
    log.Warn("KYC query failed, degrade to unauthenticated", err)
    kycStatus = 1
    kycName = ""
} else if kycResp.KycInfo != nil {
    kycStatus = kycResp.KycInfo.Status   // 1/2/3/4
    kycName = kycResp.KycInfo.Name       // KYC认证姓名
} else {
    kycStatus = 1  // 无KYC记录 = 未认证
    kycName = ""
}
```

**钱包使用KYC数据的两个用途**：

| 用途 | 使用的字段 | 业务规则 |
|------|----------|---------|
| 决定可用提现渠道 | `status` | status=3 → 全渠道；其他 → 仅USDT |
| 自动填充提现账户持有人 | `name` | 银行卡/电子钱包提现时，"账户持有人"字段预填KYC姓名，置灰不可编辑 |

**降级策略**：KYC服务不可用时，降级为"未认证"状态，用户仅能选择USDT提现。这是安全的降级——不会因为KYC服务故障就允许未认证用户使用银行卡提现。

### 5.1 U1：GetUserInfoExpend — 获取用户扩展信息

**要解决的问题**：在B端管理场景中（如查询用户钱包信息、人工修正弹窗），需要展示用户基本信息（昵称、UID、账号类型等）。C端场景中网关已通过Token鉴权注入用户身份信息，通常不需要额外调用。

**需求依据**：
- 财务需求文档 2-5 人工修正-2 人工加款：管理员输入用户编号后查询回显用户信息
- 钱包B端操作日志中需要记录用户UID/昵称等

**触发场景**：

| 操作 | 钱包做什么 | 调用什么 |
|------|-----------|---------|
| B端 → 查询用户钱包信息（输入UID） | 获取用户基本信息展示 | `rpc.UserClient().GetUserInfoExpend(ctx, req)` |
| C端 → 大部分操作 | **不需要调用** — 网关已注入userId/uid | 无需RPC |

**已有的实际接口签名**（从IDL读取）：

```thrift
// ser-user/service.thrift
UserInfoExpendResp GetUserInfoExpend(1: user_rpc.GetUserInfoReq req)

// user_rpc.thrift
struct GetUserInfoReq {
    1: i64 userId       // 用户ID（二选一）
    2: i64 uid          // 用户UID（二选一）
}

struct UserInfoExpendResp {
    1: i64 userId
    2: i64 uid
    3: string nickname
    4: string mobile
    5: string email
    6: i32 status          // 用户状态
    7: i32 accountType     // 1普通 2游客 3主播
    8: string avatar
    9: i64 exp
    10: i32 level
    11: i32 totalFans
    12: string countryCode
    13: i64 createAt

}
```

**说明**：严格来说，C端操作时用户身份已由网关层处理（Token解析后注入请求上下文），钱包C端接口拿到的是已验证的userId，不需要再调ser-user确认用户存在。此接口主要用于**B端管理场景**的用户信息回显。

**与K1(KycQuery)的区分**：
- `KycQuery` 获取的是**KYC认证信息**（认证姓名、证件号、认证状态）
- `GetUserInfoExpend` 获取的是**用户基本信息**（昵称、手机、头像、等级）
- 两者用途不同，不能互相替代

### 6.1 L1：AddActLog — 写入操作审计日志

**要解决的问题**：B端管理员对钱包模块进行操作（币种配置增删改、汇率刷新等）时，需要记录操作日志用于审计追溯。这是**工程级规范要求**，所有B端模块都遵循此模式。

**需求依据**：
- 币种配置需求文档 4-4 操作日志：汇率变更需要记录操作人、操作时间、变更内容
- 财务需求文档 2-1 通道配置-4 操作日志说明：标准操作日志格式
- 工程规范：现有所有B端服务（ser-app、ser-bcom、ser-buser等）都调用 BLogClient().AddActLog

**触发场景**：

| B端操作 | 日志内容 | 调用什么 |
|---------|---------|---------|
| 创建/编辑/启禁用币种 | backend="wallet", model="currency", action="create/update/toggle" | `rpc.BLogClient().AddActLog(ctx, req)` |
| 设置基准币种 | backend="wallet", model="currency", action="setBase" | 同上 |
| 手动刷新汇率 | backend="wallet", model="rate", action="refresh" | 同上 |

**已有的实际接口签名**（从IDL读取）：

```thrift
// ser-blog/service.thrift
oneway void AddActLog(1: model.ActAddReq req)   // 注意：oneway = 异步发送，不等待响应

// model.thrift
struct ActAddReq {
    1: string backend     // 服务标识，如 "wallet"
    2: string model       // 模块标识，如 "currency"、"rate"
    3: string page        // 页面标识，如 "currencyList"、"rateLog"
    4: string action      // 操作标识，如 "create"、"update"、"refresh"
    5: i32 isSuccess      // 1成功 0失败
}
```

**关键特性**：`oneway` — 这是一个**异步单向调用**，发送后不等待响应。即使ser-blog服务不可用，也不会阻塞业务操作。这意味着：
- **不会影响业务流程**：日志写失败不影响币种配置操作
- **无需降级策略**：oneway调用本身就是最轻量的方式
- 请求中的操作人信息（createBy、createName、ip等）由BLog服务从请求上下文中自动提取，不需要钱包显式传入

## 十、按业务场景的依赖调用链路

### 10.1 用户充值（以银行转账充BSB为例）

```
用户进入充值页面
    │
    ▼
① [钱包] → 内部调用 C2 GetAllEnabledCurrencies → 获取可用币种
② [钱包] → 内部调用 C1 GetCurrencyConfig → 获取币种精度/符号
③ [钱包] → RPC调用 F1 FinanceClient().GetRechargeMethodList → 获取充值方式列表
    │
用户选择银行转账，输入金额1000BSB
    │
    ▼
④ [钱包] → 内部调用 C4 GetPlatformRate → 展示"汇率：1BSB=₫2,620"
⑤ [钱包] → 检查重复转账（本地查询）
    │
用户点击"去充值"
    │
    ▼
⑥ [钱包] → 内部调用 C5 GetTransactionRate → 锁汇
⑦ [钱包] → 本地创建充值订单
⑧ [钱包] → 三方通道API 创建订单（或通过财务中转）
    │
    ▼
返回用户支付页面

依赖总结：2个内部调用(C1,C2) + 2个内部调用(C4,C5) + 1个RPC(F1) + 1个三方API
```

### 10.2 用户提现（以VND银行卡提现为例）

```
用户点击"提现"
    │
    ▼
① [钱包] → RPC调用 K1 KycClient().KycQuery → 获取KYC状态和认证姓名
    │
    ├── status=3(已通过) → 全渠道可用

    │
    ▼
② [钱包] → RPC调用 F2 FinanceClient().GetWithdrawMethodList → 获取提现方式列表

    │  "账户持有人"字段自动填充KYC认证姓名（步骤①获取），置灰不可编辑

### 三个核心认知

1. **核心依赖集中在 ser-finance（4个RPC）**
   - 充值和提现两条核心业务链路都依赖财务模块提供的配置和审核入口
   - 这4个接口的IDL定义是**跨团队协作的第一个卡点**，要尽早与财务团队对齐
   - 开发阶段可Mock，但Mock的请求/响应结构要贴近真实设计

2. **已存在的模块可直接对接（kyc/user/blog/s3）**
   - KYC和User的IDL签名已确定，直接调用即可
   - BLog是oneway异步，零风险
   - 这些模块**不是我们的阻塞点**

3. **兑换是唯一零外部依赖的业务场景**
   - 兑换仅依赖币种配置（同进程内部调用），不调用任何外部RPC
   - 这意味着：实现阶段可以**最先跑通兑换流程**作为验证，不受任何外部模块制约

### 与上一篇（RPC暴露）的关系对照

```
