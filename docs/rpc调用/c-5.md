## 2.1 ser-finance — 财务模块（4 个调用，全部 Core）

### 模块状态：**尚未存在**

ser-finance 是与 ser-wallet 同期开发的新模块，当前 IDL 文件、namesp 常量、rpc_client 均不存在。这是 ser-wallet 最重要的外部依赖。

### 依赖接口清单

| # | 方法名（预期） | 调用场景 | 层级 | 触发点 | 证据 |
|---|---------------|---------|------|--------|------|
| 1 | **QueryPayConfig** | C端进入充值页，获取可用充值方式列表 | Core | DepositService | 实现表达 5.3 line 403 |
| 2 | **CreateChannelOrder** | C端确认充值，创建渠道支付订单 | Core | DepositService | 实现表达 5.3 line 409 |
| 3 | **QueryPayConfig** | C端进入提现页，获取可用提现方式列表 | Core | WithdrawService | 实现表达 5.5 line 457 |
| 4 | **SubmitWithdrawReview** | C端确认提现，提交进入三级审核流 | Core | WithdrawService | 实现表达 5.5 line 465 |

> **说明**：接口评估 2.4 还列了"通道状态/可用性查询"（Should-have），实现表达 8.1 也提到"通道可用性查询"。此接口是否独立出来取决于财务方 IDL 设计，可能合并到 QueryPayConfig 内返回。暂不单独计入确定依赖。

### 调用时序分析

**充值流程（实现表达 5.3）**：

```
[C端] 进入充值页
  │
  ├─ ① wallet.GetDepositMethods()
  │     └─ 内部调 rpc.FinanceClient().QueryPayConfig(ctx, &QueryPayConfigReq{
  │              Currency: "VND",
  │              ConfigType: 1,   // 1=充值
  │         })
  │     └─ 透传返回给C端（方式名/图标/限额/快捷金额）
  │
[C端] 选方式 + 输金额 + 点确认
  │
  ├─ ② wallet.CreateDepositOrder()
  │     └─ 参数校验（币种/金额/限额）
  │     └─ 重复转账拦截
  │     └─ 生成本地充值订单（状态=待支付）
  │     └─ 调 rpc.FinanceClient().CreateChannelOrder(ctx, &CreateChannelOrderReq{
  │              UserId:     userId,
  │              Currency:   "VND",
  │              Amount:     10000000,  // 最小单位整数
  │              PayConfigId: configId,
  │              OrderNo:    "C202602280001",
  │         })
  │     └─ 拿到支付信息（payUrl/payAddress）返回C端
  │
  ⚠ 此时用户余额不变，充值是先支付后入账
  ⚠ CreateChannelOrder 失败：本地订单标记失败即可，无补偿问题
```

**提现流程（实现表达 5.5）**：

```
[C端] 进入提现页（并行发4个请求）
  │
  ├─ ① KYC状态查询（→ ser-kyc，见 2.2）
  ├─ ② rpc.FinanceClient().QueryPayConfig(ctx, &QueryPayConfigReq{
  │        Currency: "VND",
  │        ConfigType: 2,   // 2=提现
  │   })
  │   + 按充值来源过滤（本地查充值记录判断）
  ├─ ③ 提现限额查询（读本地币种配置，内部调用）
  └─ ④ 提现账户信息（读本地表）

[C端] 确认提现
  │
  ├─ 并行校验：限额 + KYC户名比对 + 余额
  ├─ 冻结余额：available - amount, frozen + amount（乐观锁）
  ├─ 生成提现订单（状态=审核中）
  └─ ⑤ rpc.FinanceClient().SubmitWithdrawReview(ctx, &SubmitWithdrawReq{
           UserId:    userId,
           Currency:  "VND",
           Amount:    5000000,
           Fee:       100000,
           Method:    "bank_transfer",
           OrderNo:   "T202602280001",
           AccountInfo: {...},
       })

  ⚠ 关键风险：冻结成功 → SubmitWithdrawReview 失败 → 用户资金卡死
  ⚠ 补偿方案：立即解冻 + 订单标记失败 + 写回退流水
  ⚠ 极端情况：解冻也失败 → 记日志告警 → 人工介入
```

### 代码层对应

```
│       ├── GetPaymentConfig()  # 封装 QueryPayConfig
```

## 2.2 ser-kyc — KYC 认证服务（2 个调用，Should-have → 实际 Core）

### 模块状态：**已存在**

KYC 服务 RPC 客户端已在 `common/rpc/rpc_client.go` 注册（`KycClient()`），IDL 已定义在 `common/idl/ser-kyc/service.thrift`。

### 依赖接口清单

| # | 方法名（IDL 已定义） | 调用场景 | 层级 | 触发点 | 证据 |
|---|---------------------|---------|------|--------|------|
| 1 | **KycQuery** | C端进入提现页，查询用户KYC认证状态 | Core | WithdrawService | 实现表达 5.5 line 456 |
| 2 | **KycDetail** | C端确认提现时，获取KYC详情做户名比对 | Core | WithdrawService | 实现表达 5.5 line 462 |

> **为什么接口评估标 Should-have，这里标 Core**：
> 接口评估是宏观评估阶段的分级，当时"KYC校验"可能被单独服务承担。但实现表达明确写了 ser-wallet 的 WithdrawService 必须在提现前置校验中调用 KYC，且合规要求"非 USDT 提现必须 KYC"是硬约束。所以实际是 Core 级别。

### 调用时序

```
[C端] 进入提现页
  │
  ├─ ① rpc.KycClient().KycQuery(ctx, &KycQueryReq{
  │        // userId 从 ctx 中获取（tracer.GetUserId）
  │   })
  │   → 返回 KycQueryResp { Status: 1 }  // 0=未认证, 1=审核中, 2=已通过, 3=已拒绝
  │   → 非 USDT 提现 && Status != 2 → 前端展示"请先完成KYC认证"
  │
[C端] 确认提现
  │
  ├─ ② rpc.KycClient().KycDetail(ctx, &KycDetailReq{
  │        // 获取 KYC 审核通过的实名信息
  │   })
  │   → 返回 KycAudit { RealName: "张三", ... }
  │   → 比对：提现账户户名 vs KYC实名（忽略大小写+空格）
  │   → 不一致 → 拒绝提现
```

### 现有 IDL 方法签名

```thrift
// common/idl/ser-kyc/service.thrift
service KycService {
    KycQueryResp KycQuery(1: user_kyc.KycQueryReq req)  // 查询KYC状态
    KycAudit KycDetail(1: kyc.KycDetailReq req)          // 获取KYC审核详情
}
```

### 代码层对应

```
ser-wallet/internal/
├── domain/
│   └── user_rpc.go             # 用户/KYC服务RPC封装
│       ├── GetKycStatus()      # 封装 KycClient().KycQuery()
│       └── GetKycDetail()      # 封装 KycClient().KycDetail()
│
├── ser/
│   └── withdraw_service.go     # 调用 user_rpc.GetKycStatus + GetKycDetail
```

### 阻塞分析

| 阶段 | 影响 | 应对 |
|------|------|------|
| 第一~三期 | **不阻塞** | 提现在第四期联调 |
| 第四期 | **不阻塞**（接口已存在） | KycClient 已注册，IDL 已有，直接调用 |

### 条件逻辑

```
if 提现方式 == USDT {
    // USDT 提现不要求 KYC（链上匿名转账）
    跳过 KYC 校验
} else {
    // 法币提现（银行转账/电子钱包）必须 KYC
    status := rpc.KycClient().KycQuery(ctx, req)
    if status != KYC_PASSED {
        return error("请先完成KYC认证")
    }
    kycDetail := rpc.KycClient().KycDetail(ctx, req)
    if kycDetail.RealName != withdrawAccount.HolderName {
        return error("提现账户户名与实名认证不一致")
    }
}
```

```
ser-ip/internal/ser/sys_white_service.go   — 3次调用（白名单）

ser-item/internal/ser/item_service.go      — 1次调用（道具配置）
```

**结论**：所有 B 端写操作必须记审计日志，ser-wallet 的币种配置模块作为 B 端管理功能，必须遵循此规范。

```go

```

# 三、内部调用 — 同服务内非 RPC

## 3.1 币种配置 → 钱包核心（本地方法调用）

接口评估 2.4 列了 7 个"RPC 调用"，其中 **#1 获取币种信息/列表** 和 **#2 获取汇率** 的被调用方是"币种配置模块"。

**关键设计决策**（实现表达 2.2 line 117-118）：

> 币种配置和余额在同一个数据库，余额操作需要实时读取币种参数（精度/限额/手续费），放在同一个服务里走本地方法调用，不走 RPC，减少一次网络开销和一个故障点。

因此这 2 个"调用"是 **Service 层本地方法调用**，不是 RPC：

| # | 调用 | 方式 | 调用路径 |
|---|------|------|---------|
| 1 | 获取币种信息/列表 | **本地方法** | `DepositService → domain.currency_cache.GetCurrencyConfig()` |
| 2 | 获取汇率 | **本地方法** | `ExchangeService → domain.currency_cache.GetExchangeRate()` |

这两个走 Redis 缓存（Cache-Aside），命中率极高，不经过网络。

**不算入外部 RPC 依赖**。但币种配置模块同时通过 RPC **暴露**给外部（4 个 RPC 暴露接口供财务等模块调用），这部分在 [rpc暴露/c-5.md](../rpc暴露/c-5.md) 中已分析。


### 工程模式分析

扫描现有代码，**所有服务的 I18n 调用都在网关层**：

```
gate-back/biz/handler/buser/    → ret.HttpOne(ctx, resp, rpc.I18nClient(), err)
gate-back/biz/handler/bcom/     → rpc.Handler(ctx, c, rpc.BComClient().XXX)
gate-font/biz/handler/user/     → ret.HttpOne(ctx, resp, rpc.I18nClient(), err)
```

**结论**：ser-wallet 服务代码本身**不需要调用 I18nClient**。错误码翻译由网关层统一处理。但配置 gate-back / gate-font 的路由时，需确保使用了正确的 I18n 响应模式。

**不算入 ser-wallet 的 RPC 调用依赖**。

# 七、与 RPC 暴露的对照

上一份文档分析了"别人调我"（12 个暴露接口），本文分析"我调别人"（8-10 个调用接口）。放在一起看清完整的 RPC 双向依赖：

```
                    我调别人                          别人调我
              ─────────────────                ─────────────────
              ser-finance (4个)                 ser-finance (8个)
                QueryPayConfig ──→              ←── DepositCredit
                CreateChannelOrder ──→          ←── WithdrawConfirm
                SubmitWithdrawReview ──→        ←── WithdrawUnfreeze
                QueryChannelStatus ──→          ←── ManualCredit
                                                ←── ManualDebit
              ser-kyc (2个)                     ←── ManualPatch
                KycQuery ──→                    ←── QueryBalanceRpc
                KycDetail ──→                   ←── (余额查询)

              ser-blog (1个)                    ser-wallet 暴露给全体 (4个)
                AddActLog ──→                   ←── AvailableCurrencyList
                                                ←── QueryCurrencyInfo
              ser-user (1-2个)                  ←── QueryExchangeRate
                GetUserInfoExpend ──→           ←── QueryCurrencyConfig
                BatchGetUserInfo ──→

              合计：我调 8-10 个               合计：别人调 12 个
```

**不对称分析**：

- "别人调我"（12 个）> "我调别人"（8-10 个），这符合 ser-wallet 作为**资产管理核心服务**的定位 — 被依赖多于依赖别人
- 最重的外部依赖集中在 **ser-finance**（双向共 12 个接口），是最关键的协同模块
- ser-kyc、ser-blog、ser-user 都是**已存在的成熟服务**，调用风险低

# 九、风险与铁律

## 9.1 架构铁律

```
RPC 在事务之外做，DB 操作在事务之内做。
                            — 实现表达 line 152
```

**所有外部 RPC 调用**必须遵守此规则：

```go
// ✅ 正确：RPC 在事务外
kycStatus, err := domain.GetKycStatus(ctx, userId)  // RPC
if err != nil { return err }

err = rep.WithdrawInTx(ctx, func(tx *gorm.DB) error {  // 事务开始
    // 只做 DB 操作：冻结余额 + 写订单 + 写流水
    return nil
})  // 事务结束

reviewNo, err := domain.SubmitWithdrawReview(ctx, req)  // RPC

// ❌ 错误：事务内含 RPC
err = rep.WithdrawInTx(ctx, func(tx *gorm.DB) error {
    kycStatus, err := rpc.KycClient().KycQuery(ctx, req)  // ❌ 事务内RPC
    // ...
})
```

## 9.2 幂等与重试

| RPC 调用 | 重试安全？ | 幂等保障 |
|---------|-----------|---------|
| QueryPayConfig | 🟢 安全（纯读取） | 无状态 |
| CreateChannelOrder | ⚠️ 需幂等 | 靠 OrderNo 唯一，财务方幂等 |
| SubmitWithdrawReview | ⚠️ 需幂等 | 靠 OrderNo 唯一，财务方幂等 |
| KycQuery | 🟢 安全（纯读取） | 无状态 |
| KycDetail | 🟢 安全（纯读取） | 无状态 |
| AddActLog | 🟢 安全（oneway） | fire-and-forget，丢了不影响业务 |
| GetUserInfoExpend | 🟢 安全（纯读取） | 无状态 |

## 9.3 补偿场景

唯一需要补偿的场景（实现表达 line 477）：

```
冻结余额成功 → SubmitWithdrawReview RPC 失败 → 用户资金卡死

补偿方案：
  ① 立即解冻（available + amount, frozen - amount）
  ② 订单标记失败
  ③ 写回退流水
  ④ 如果解冻也失败 → 记日志告警 → 人工介入
```
