## 五、跨模块联动点（需评审明确）

| 联动场景 | 涉及模块 | 说明 |
|---------|---------|------|
| 充值成功入账 | 财务→钱包 | 通道回调确认后，需更新钱包余额 |

## 六、接口多维度迭代评估（第二版）

> 迭代日期：2026-02-26
> 迭代依据：结合工程架构事实（双网关模式、RPC客户端模式、IDL规范、三层分层架构）与需求文档/原型分析
> 分类维度：**C端接口**（gate-font路由）/ **B端接口**（gate-back路由）/ **RPC对外**（本服务IDL暴露供他人调用）/ **RPC调用**（本服务作为客户端调用他人）
> 优先级：**核心**（Must — 绕不过的，缺失则功能不成立）/ **应该有**（Should — 重要但非首版阻塞）/ **可选**（Optional — 可合并、可延后、可前端承担）

### 工程事实前提

以下评估基于已确认的工程事实，非假设推演：

| 事实项 | 内容 |
|--------|------|
| 服务命名空间 | 下一可用编号 **022**，将注册为 `wallet_service` |
| 网关路由模式 | gate-font: `/open`(免登录) + `/api`(需Token) ；gate-back: `/admin/open` + `/admin/api`(Token+RBAC) |
| RPC转发模式 | `rpc.Handler[REQ,RESP]`、`rpc.HandlerNoParam[RESP]`、`rpc.HandlerWithCSV[REQ,RESP]` |
| IDL命名规范 | `{Action}{Entity}Req` / `{Action}{Entity}Resp`（如 `CreateBannerReq`） |
| 服务内部分层 | handler.go → internal/ser/ → internal/rep/ → internal/gorm_gen/ |
| RPC客户端模式 | `sync.Once` 单例 + ETCD服务发现，在 `common/rpc/rpc_client.go` 注册 |
| 审计日志 | 所有服务统一调用 `rpc.BLogClient().AddActLog()` |
| 已有服务IDL方法数参考 | UserService(46)、LiveService(50)、AppService(20)、BComService(18)、KycService(8)、ItemService(9) |
| 当前状态 | ser-wallet 尚无IDL、无路由、无RPC客户端，仅有空壳 gorm_gen.go |

#### B. B端接口（gate-back 路由）— 预估 0 个

> 原因：钱包模块不直接对B端暴露HTTP接口。所有B端对钱包的操作（查余额、加减款、审核等）由**财务管理模块**承接，财务模块通过RPC调用钱包完成余额变更。这与现有工程模式一致——各服务专注自身端侧，跨端通过RPC协作。

### 6.6 补充说明

**关于充值/提现流程中的模块职责边界**

基于需求分析（财务模块拥有通道配置/密钥/轮询参数，钱包模块拥有用户余额/C端交互），推演最合理的分工：

```
【充值流程】
C端 → gate-font → ser-wallet(C4:创建充值订单)
    ser-wallet → ser-finance(D5:匹配通道创建支付)  ← 财务做通道轮询+三方支付
    ser-finance → 三方通道 → 返回支付页信息
    ser-wallet → C端（展示支付页/跳转三方）
    ...用户支付...
    三方通道 → ser-finance(FR4:充值回调)  ← 财务验签+更新记录
    ser-finance → ser-wallet(R2:入账)  ← 钱包增加余额

【提现流程】
C端 → gate-font → ser-wallet(C7:创建提现订单)
    ser-wallet：校验KYC/限额/余额 → 冻结余额(R3) → 创建订单
    B端财务审核 → ser-finance → ser-wallet(R5:确认扣款 或 R4:驳回解冻)
    ser-finance → 三方通道（发起代付）

【兑换流程】
C端 → gate-font → ser-wallet(C11:执行兑换)
    ser-wallet → 币种配置(D1:查汇率) → 计算 → 扣法币余额 + 加BSB余额
    全程在钱包模块内完成，无需调用财务
```

**关于RPC对外方法的业务安全性**

钱包的RPC对外方法（R1-R12）涉及资金变更，属高风险操作：
- 所有余额变更方法必须做**幂等处理**（基于订单号/请求ID去重）
- 冻结/扣减必须有**事务保障**（GORM事务 + 乐观锁或悲观锁）
- 入账/扣款必须生成**流水记录**（金额、钱包类型、操作类型、关联订单号、操作前后余额）
- 这些是实现层面的要求，不影响接口数量评估，但影响工作量
