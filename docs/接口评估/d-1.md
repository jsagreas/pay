# 接口迭代评估（基于工程规范校准）

> 迭代日期：2026-02-26
> 评估依据：需求文档 + 产品原型 + 工程分析记录（c-2/c-3） + 现有4服务IDL接口粒度对比（ser-app/ser-item/ser-user/ser-kyc）
> 评估原则：宁多不少，但不过度设计；分层级（核心必须/应该有/可选）；分类型（C端/B端/RPC提供/RPC调用）

---

## 一、评估方法论

### 1.1 接口粒度校准标准（来源：现有工程IDL实证）

| 操作类型 | 项目惯例 | 说明 |
|---------|---------|------|
| 新建 | `CreateXxx` | 独立方法，返回 `{id}` |
| 编辑 | `UpdateXxx` | 独立方法，与Create严格分离 |
| 删除 | `DeleteXxx` | 独立方法，软删除 |
| 状态切换 | `StatusXxx` | 独立方法，不合并到Update |
| B端分页 | `PageXxx` / `QueryXxxPage` | 含丰富可选过滤字段 |
| B端详情 | `DetailXxx` / `QueryXxxDetail` | 按ID查 |
| C端列表 | `ListXxx` | 简化结构，状态过滤后全量返回 |
| C端详情 | `GetXxx` | 从Token/参数取 |
| RPC查询 | `GetXxx` / `BatchGetXxx` | 无分页，全量或批量 |
| 单字段切换 | `ToggleXxx` | 每个可切换字段独立方法 |
| 导出 | `QueryXxxCsv` | 独立方法，返回二进制流 |
| 写操作响应 | 空结构体 `XxxResp{}` | 成功即隐含，不返回bool |

### 1.2 接口优先级定义

| 层级 | 含义 | 判断标准 |
|------|------|---------|
| **P0 核心必须** | 没有就无法运转 | 产品原型中明确展示的核心功能、资金链路上的必经节点 |
| **P1 应该有** | 完整产品所需 | 需求文档描述但原型未重点展示，或管理运维必需 |
| **P2 可选** | 锦上添花或可后置 | 优化体验、数据导出、统计报表等，评审可能调整 |

       RPC调用↓   ↓    ↓    ↓   ↑ RPC提供
    ┌─────────┐ ┌──┐ ┌──┐ ┌──┐ │
    │币种配置  │ │KYC│ │日志│ │  │ │

**RPC调用方向总结**：

| 调用方 → 被调用方 | 接口 | 方向含义 |
|------------------|------|---------|
| 钱包 → 币种配置 | GetActiveCurrencyList, GetExchangeRate, GetDepositWithdrawConfig | 钱包读取基础配置 |
| 钱包 → 财务 | GetPaymentMethods, MatchDepositChannel | 钱包获取支付通道信息 |
| 钱包 → ser-kyc | KycQuery | 提现前KYC校验 |
| 钱包 → ser-blog | AddActLog (oneway) | 审计日志 |
| 财务 → 钱包 | CreditBalance, DebitBalance, FreezeBalance, UnfreezeBalance, ManualAdjust | 资金操作（入账/冻结/解冻/扣款/调整） |
| 财务 → 币种配置 | GetActiveCurrencyList, GetExchangeRate | 财务读取币种和汇率 |

### 与初版对比

| 模块 | 初版估算 | 迭代评估 | 差异原因 |
|------|---------|---------|---------|
| 币种配置 | ~10 | 16 | +6：按项目惯例拆分Create/Update/Status/Delete为独立方法；补充BatchGetExchangeRates等RPC |
| 多币种钱包 | ~25 | 35 | +10：拆分"获取/绑定提现账户"为List+Bind两个；补充GetDepositOrderStatus轮询；补充B端流水查询/详情/导出；冻结/解冻拆为独立方法 |
| 财务管理 | ~22 | ~38 | +16：初版将CRUD合并计算，按项目惯例拆分后每个实体需5-7个方法 |

### 关键说明

1. **币种配置 vs 钱包 是否拆为两个服务**：待定。从职责边界看可以是同一个 ser-wallet 内的两个子模块（类似 ser-item 的 item + item_publish），也可以是独立服务。当前按同一服务内两个子模块评估
2. **充值回调 DepositCallback**：归属需明确 — 如果三方支付直接回调钱包服务，则钱包需要暴露HTTP回调端点（不走网关）；如果三方回调财务模块，财务再RPC调钱包，则钱包只需RPC接口。当前按RPC接口评估
3. **活动奖金 GetDepositBonusList**：数据来源可能是活动系统而非钱包本身，钱包可能只是透传。具体取决于活动系统设计，当前列为P1
4. **提现审核流程在财务模块**：钱包只负责冻结余额和最终扣款，三级审核逻辑在财务模块内
5. **P0的36个接口是确定性骨架**：无论评审怎么调整细节，这些都绕不过去
