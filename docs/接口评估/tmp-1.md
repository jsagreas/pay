### 模块间依赖关系

```
┌─────────────┐       ┌─────────────────┐       ┌─────────────┐
│  币种配置    │◄──────│   多币种钱包     │◄──────│  财务管理    │
│  (基础设施)  │ 汇率  │   (核心业务)     │ 入账  │  (运营管理)  │
│             │ 币种  │                 │ 冻结  │             │
│             │ 精度  │                 │ 解冻  │             │
│             │       │                 │ 扣款  │             │
│             │       │                 │ 加减款│             │
└─────────────┘       └─────────────────┘       └─────────────┘
     基础层                 业务层                  管理层
```

- **币种配置 → 钱包**：钱包的所有金额操作依赖币种精度、汇率计算
- **财务 → 钱包**：充值入账、提现冻结/解冻/扣款、人工加减款均通过内部RPC调用钱包
- **钱包 → 财务**：财务弹窗需读取钱包余额
- **钱包 → 币种配置**：兑换/充值/提现均需获取汇率和币种信息

## 五、说明与保留事项

1. **预评审状态**：以上接口清单基于当前产品原型和需求文档，尚未通过正式需求评审，细节可能在跨部门评审后发生变更（新增/删除/调整）
2. **粒度说明**：CRUD类接口（如通道配置）是合并计数还是拆分计数，取决于最终接口设计规范，当前按功能组粗估
3. **定时任务**：汇率偏差检测更新、对账等可能以定时任务形式实现而非暴露为接口，未计入接口数
4. **内部RPC vs HTTP**：C端接口走HTTP（Hertz），模块间联动走RPC（Kitex），上述估算包含两种类型
5. **核心职责边界**：币种配置 + 多币种钱包 = 你负责的 ~20~25 个接口；财务管理模块为他人负责，需清楚依赖关系但不负责实现

## 五、迭代说明与保留事项

1. **数量变化说明**：初版粗估38~47个，迭代后精细评估65~68个。差异原因：(1)初版CRUD合并计数，迭代后按工程规范拆分；(2)初版未区分RPC对外提供，迭代后单独列出13个；(3)财务模块从粗估18~22拆细到28个
2. **你负责的接口总量**：~37~39个（币种12 + 钱包25~27），核心不可少的30个，应该有7个，可选2个
3. **预评审弹性空间**：
   - 充值方式可能增减（如新增本地银行直连）→ 影响CreateDepositOrder内部分支，不增加接口数
   - 提现渠道评审可能调整 → 同上，不影响接口定义层
   - 额外奖金规则可能细化 → ListAvailableBonus字段可能变化
   - 子钱包类型（代理/场馆为预留）→ 如果评审去掉预留，WalletManualAdd/Sub的子钱包枚举减少，接口不变
4. **架构待确认项**：wallet与finance之间充值发起时的调用方向（wallet调finance获取通道？还是finance订阅wallet订单？）需与财务模块负责人对齐
5. **订单号规范遵循**：C+16位(充值) / T+16位(提现) / B+16位(补单) / A+16位(加款) / M+16位(减款)
6. **错误码需申请码段**：按工程规范 `6{xxx}{xxx}` 格式，ser-wallet需向项目分配新的服务编号
