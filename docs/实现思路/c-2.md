# 一、基础设施注册（第一步，绕不过）

新模块要跑起来，首先要在项目的基础设施层面完成注册。这些是框架层面的硬性要求。

## 1.1 服务编号与 ETCD 常量

> 依据：`common/pkg/consts/namesp/namesp.go`，现有编号 001~021，016 空缺

需要在 `namesp.go` 中注册 ser-wallet 的常量：

```
编号选择：022（或 016，看团队约定）

需要新增的常量：
  EtcdWalletService = "wallet_service"           // ETCD 服务注册名
  EtcdWalletPort    = "/slg/serv/wallet/port"    // 端口配置路径
  EtcdWalletDb      = "/slg/serv/wallet/db"      // 数据库名路径
```

同时需要在 ETCD 中配置对应的 key-value（端口号、数据库名），这是服务启动的前提。

## 1.2 go.work 注册

> 依据：项目根目录 `go.work`，当前管理 12 个模块

新增一行 `./ser-wallet` 到 go.work 的 use 列表中，让 workspace 能发现这个模块。

## 1.3 RPC 客户端工厂注册

> 依据：`common/rpc/rpc_client.go`，现有 20 个活跃客户端，统一使用 sync.Once 惰性单例

需要在 `rpc_client.go` 中新增 `WalletClient()` 函数，模式与现有完全一致：

```go
// 模式参考（不是最终代码，只是说明结构）
var walletService *walletservice.Client
var walletOnce    sync.Once

func WalletClient() walletservice.Client {
    walletOnce.Do(func() {
        client, _ := walletservice.NewClient(
            namesp.EtcdWalletService,
            InitRpcClientParams()...,
        )
        walletService = &client
    })
    return *walletService
}
```

这个客户端注册后，两个网关（gate-back、gate-font）和其他服务就能通过 `rpc.WalletClient()` 调用 ser-wallet。

# 二、IDL 设计（接口契约，必须先行）

> 依据：项目采用 IDL-first 开发模式，所有接口定义先写 Thrift IDL，再通过 gen_kitex.sh 生成桩代码
> 参考：`common/idl/ser-app/service.thrift` 及其子文件

## 2.1 IDL 文件规划

ser-wallet 的接口分属两个业务模块（钱包 + 币种配置），但同属一个 Kitex 服务，共用一个 service 定义。

```
common/idl/ser-wallet/
├── service.thrift          ← 服务入口，定义 WalletService
├── wallet.thrift           ← 钱包模块的请求/响应结构
├── currency.thrift         ← 币种配置模块的请求/响应结构
├── exchange.thrift         ← 兑换相关结构（可选，看复杂度是否需要单独拆）
└── record.thrift           ← 记录查询相关结构（可选）
```

## 2.2 service.thrift 的方法规划

## 2.3 IDL 编写的工程约定

> 依据：现有 IDL 文件归纳

| 约定项 | 标准 | 说明 |
|-------|------|------|
| namespace | `namespace go ser_wallet` | 决定生成的 Go 包名 |
| 命名规范 | `ActionEntityReq/Resp` | 如 `CreateDepositOrderReq` |
| ID 类型 | `i64` | 所有 ID 统一 int64 |
| 时间类型 | `i64`（毫秒时间戳） | 存储和传输都是毫秒 |
| 金额类型 | 建议 `string` | 避免浮点精度问题，金额作为字符串传输 |
| 分页入参 | `i32 pageNo`, `i32 pageSize` | 标准分页字段 |
| 字段注解 | `(api.body = "fieldName")` | Hertz 绑定用 |

## 2.4 金额类型的选择（关键决策）

钱包模块涉及大量金额计算，类型选择需要提前确定：

```
方案A: IDL 用 string，Go 内部用 shopspring/decimal
  · 优点：精度完全可控，不存在浮点误差
  · 缺点：每次计算需要转换

方案B: IDL 用 i64（分/厘为单位的整数）
  · 优点：计算简单，无精度问题
  · 缺点：需要统一单位约定（法币的最小单位各不相同）

方案C: IDL 用 double
  · 不推荐：浮点精度问题在金融场景下不可接受

建议方向：string 或 i64 整数，具体取决于团队的币种最小单位约定
```

## 2.5 生成桩代码

IDL 写完后，执行代码生成：

```bash
# 在项目根目录执行
bash common/script/gen_kitex.sh
```

脚本会扫描 `common/idl/ser-wallet/` 目录，生成：
- `common/pkg/kitex_gen/ser_wallet/` — 结构体定义
- `common/pkg/kitex_gen/ser_wallet/walletservice/` — Client/Server 接口

首次执行还会在 `ser-wallet/` 下生成 `handler.go` 和 `main.go` 骨架。

# 三、数据库设计方向（核心表结构）

> 说明：这里只列出必须存在的表和关键字段方向，具体字段等需求定型后细化

## 3.1 需要的核心表

根据需求分析，ser-wallet 至少需要以下表：

```
钱包模块:
  ① user_wallet          — 用户钱包余额表（核心中的核心）
  ② deposit_order        — 充值订单表
  ③ withdraw_order       — 提现订单表
  ④ exchange_order       — 兑换订单表
  ⑤ wallet_flow          — 钱包流水记录表（所有余额变动的审计日志）
  ⑥ reward_record        — 奖励记录表

币种配置模块:
  ⑦ currency_config      — 币种配置表
  ⑧ rate_config          — 汇率配置表（实时/平台/充值/提现汇率）
  ⑨ rate_change_log      — 汇率变更日志表

辅助:
  ⑩ idempotent_record    — 幂等去重表（可选，也可以用订单状态机代替）
```

## 3.2 user_wallet 表设计方向（最关键的表）

```
核心思路：每个用户 × 每个币种 × 每种钱包类型 = 一行记录

关键字段方向：
  · user_id        — 用户ID
  · currency_id    — 币种ID（关联 currency_config）
  · wallet_type    — 钱包类型（中心/奖励/主播/代理/场馆）
  · balance        — 余额（整数/decimal，取决于金额类型决策）
  · frozen_amount  — 冻结金额（提现冻结时使用）
  · status         — 状态

并发安全考虑：
  · 这张表是读写最频繁的表，Credit/Debit/Freeze/Unfreeze 都操作它
  · 必须考虑行锁策略（SELECT FOR UPDATE）或乐观锁（version 字段）
  · 建议加 version 字段做乐观锁：UPDATE ... WHERE id=? AND version=?

唯一索引：
  · (user_id, currency_id, wallet_type) 唯一
```

## 3.3 订单表设计方向

```
三张订单表（deposit/withdraw/exchange）的共同设计方向：

  · order_no       — 订单号（唯一索引，格式：C/T + 16位）
  · user_id        — 用户ID
  · currency_id    — 币种ID
  · amount         — 金额
  · status         — 状态（状态机，不同订单类型的状态流转不同）
  · 审计字段       — create_by, create_at, update_by, update_at（GORM UserPlugin 自动填充）

订单号格式（已在需求中确定）：
  · 充值订单：C + 16位随机数
  · 提现订单：T + 16位随机数
  · 兑换订单：可以用 E + 16位
  · 补单：B + 16位（财务模块生成）
  · 人工加款：A + 16位（财务模块生成）
  · 人工减款：M + 16位（财务模块生成）
```

## 3.4 wallet_flow 表设计方向

```
所有余额变动都写一条流水记录，用于审计和对账：

  · flow_no        — 流水号
  · user_id        — 用户ID
  · currency_id    — 币种ID
  · wallet_type    — 钱包类型
  · flow_type      — 变动类型（充值/提现/兑换/奖励/人工加/人工减/冻结/解冻）
  · amount         — 变动金额（正数=增加，负数=减少）
  · balance_before — 变动前余额
  · balance_after  — 变动后余额
  · order_no       — 关联订单号
  · remark         — 备注

这张表只写不改（append-only），是审计的核心依据。
```

## 3.5 GORM Gen 生成

表建好后，通过 GORM Gen 自动生成 model/query/repo：

## 4.2 main.go 启动流程

> 依据：`ser-app/main.go`，所有服务启动流程一致

```
main() 启动顺序：
  1. lg.InitKLog()                    → 日志初始化
  2. etcd.LoadAndWatch("/slg/")       → 加载 ETCD 配置 + 监听变更
  3. cfg.InitRedis()                  → Redis 连接
  4. cfg.InitMySQL()                  → MySQL 连接 + query.SetDefault(db)
  5. 解析端口（命令行 / ETCD）
  6. 创建 Service 实例（注入依赖）
  7. walletservice.NewServer(impl, rpc.InitRpcServerParams(...))
  8. srv.Run()
```

说明：与 ser-app 相比，ser-wallet 多了一步 `cfg.InitMySQL()`，因为 ser-app 的 MySQL 是懒初始化的（banner service 中 sync.Once），而 ser-wallet 的余额操作从启动就需要数据库连接。

## 4.3 handler.go 委托模式

> 依据：`ser-app/handler.go`，所有方法一行委托

```go
type WalletServiceImpl struct {
    walletSer    *ser.WalletService
    currencySer  *ser.CurrencyService
    rateSer      *ser.RateService
    recordSer    *ser.RecordService
}

// 每个方法一行委托到对应的 Service
func (s *WalletServiceImpl) GetWalletInfo(ctx context.Context, req *ser_wallet.GetWalletInfoReq) (*ser_wallet.GetWalletInfoResp, error) {
    return s.walletSer.GetWalletInfo(ctx, req)
}

func (s *WalletServiceImpl) CreateCurrency(ctx context.Context, req *ser_wallet.CreateCurrencyReq) (*ser_wallet.CreateCurrencyResp, error) {
    return s.currencySer.CreateCurrency(ctx, req)
}
```

handler.go 不写任何业务逻辑，只做方法委托分发。

## 4.4 ser 层的组织思路

ser 层是核心业务代码所在地。按业务子模块拆分文件：

```
wallet_ser.go     — 钱包核心
  · 构造函数接收 rep 和 cache 依赖
  · GetWalletInfo / SwitchCurrency
  · Credit / Debit / Freeze / Unfreeze / QueryBalance（RPC被调方法）
  · CreateExchangeOrder（兑换核心，事务操作）

deposit_ser.go    — 充值业务（如果 wallet_ser.go 太大可以拆出来）
  · GetDepositMethods / CreateDepositOrder
  · 重复转账拦截逻辑

withdraw_ser.go   — 提现业务
  · CheckWithdrawEligibility / CreateWithdrawOrder
  · 资金来源规则判断

currency_ser.go   — 币种配置
  · CRUD + 状态管理
  · 汇率配置

rate_ser.go       — 汇率计算
  · RecalculateRates（核心联动方法）
  · 定时任务：SyncThirdPartyRates / CheckRateThreshold

record_ser.go     — 记录查询
  · 四种记录的分页查询（结构一致，type 参数不同）
```

# 六、网关路由接入

> 依据：gate-back 和 gate-font 的 router/handler 模式

## 6.1 需要接入两个网关

ser-wallet 同时提供 B 端和 C 端接口，因此两个网关都需要接入：

```
gate-back（B端管理网关）:
  · 币种配置 CRUD（8个接口）
  · 管理员查看用户钱包（2个接口）
  → 路由前缀: /admin/api/wallet/...
  → 中间件: SetHeaderInfo + WhiteCheck + BackAuthMiddleware

gate-font（C端用户网关）:
  · 钱包操作（余额/充值/提现/兑换/记录，约20个接口）
  → 路由前缀: /api/wallet/... 或 /open/wallet/...
  → 中间件: SetHeaderInfo + BlackCheck + FontAuthMiddleware
```

## 6.2 具体接入步骤

```
gate-back 接入:
  1. 创建 gate-back/biz/handler/wallet/    ← handler 文件
  2. 创建 gate-back/biz/router/wallet/     ← router 文件
  3. 在 gate-back/biz/router/register.go 添加 wallet.RouterWallet(r)

gate-font 接入:
  1. 创建 gate-font/biz/handler/wallet/    ← handler 文件
  2. 创建 gate-font/biz/router/wallet/     ← router 文件
  3. 在 gate-font/biz/router/register.go 添加 wallet.RouterWallet(r)
```

## 6.3 Handler 和 Router 模式

Handler（一行代码模式）：
```go
func GetWalletInfo(ctx context.Context, c *app.RequestContext) {
    rpc.Handler(ctx, c, rpc.WalletClient().GetWalletInfo)
}
```

Router（POST 统一模式）：
```go
func RouterWallet(r *server.Hertz) {
    // B端 - 币种配置
    my_router.Api.POST("/wallet/currency/list", wallet.GetCurrencyConfigList)
    my_router.Api.POST("/wallet/currency/create", wallet.CreateCurrency)
    // ...
}
```

路由命名规范：`/{module}/{entity}/{action}`，全部 POST。

# 七、核心业务逻辑的实现方向

> 这部分不写具体代码，而是明确每个核心场景"必须怎么做"的方向

## 7.1 余额操作的安全保证（绕不过）

所有涉及余额变更的操作，必须保证三个性质：

```
① 原子性（事务）
   · 余额变更 + 流水记录 必须在同一个 MySQL 事务中
   · 兑换操作的 3 次余额变更（扣法币 + 加BSB + 加奖金）必须在同一事务
   · 使用 GORM 的 query.Q.Transaction(func(tx *query.Query) error { ... })

② 幂等性
   · Credit/Debit/Freeze/Unfreeze 必须基于 orderNo 幂等
   · 财务模块的回调可能重试，不能重复加款/扣款
   · 实现方式：
     方案A：wallet_flow 表对 order_no 建唯一索引，写入失败说明已处理
     方案B：订单状态机，已完成的直接返回成功
     方案C：独立幂等去重表

③ 并发安全
   · 余额更新时使用行锁或乐观锁
   · 行锁方式：SELECT ... FOR UPDATE（在事务中锁定钱包行再操作）
   · 乐观锁方式：UPDATE ... SET balance=balance+?, version=version+1 WHERE version=?
   · 多个余额操作按固定顺序加锁（如按 currency_id+wallet_type 排序），防止死锁
```

## 7.2 充值流程的实现方向

```
钱包这边需要做的事（不含财务模块的部分）:

1. GetDepositMethods
   → 调财务模块 RPC 获取充值方式列表，透传返回
   → 开发阶段可以 Mock 返回固定数据

2. CreateDepositOrder
   → 重复转账拦截（查自己的订单表）
   → 调财务 RPC: MatchDepositChannel + CreateChannelOrder
   → 创建自己的充值订单记录
   → 开发阶段可以跳过财务 RPC，Mock 返回

3. Credit（被财务回调）
   → 幂等校验 → 余额增加 → 写流水 → 更新订单状态
   → 这个方法是纯被动的，自己先开发好等财务联调

关键点：充值的主动流程依赖财务 6 个 RPC，但 Credit 被动接口可以独立开发
```

## 7.3 提现流程的实现方向

```
钱包这边需要做的事:

1. CheckWithdrawEligibility
   → 调 ser-kyc 的 GetKycStatus（外部 RPC）
   → 分析充值记录判断资金来源规则（纯内部查询）
   → 综合判断是否可提现 + 允许的提现方式类型

2. CreateWithdrawOrder
   → 余额校验 + 限额校验 + 资金来源规则校验（内部）
   → Freeze 冻结金额（内部操作，重要）
   → 创建提现订单
   → 调财务 RPC: SubmitWithdrawRequest（送审）

3. Debit / Unfreeze（被财务回调）
   → 审核通过+出款成功：Debit（从冻结金额扣除，不是从可用余额扣）
   → 审核拒绝/出款失败：Unfreeze（冻结金额归还到可用余额）

关键点：冻结和解冻是钱包自己控制的，要保证 Freeze 和 Debit/Unfreeze 的金额一致性
```

## 7.4 兑换流程的实现方向

```
完全自闭环，不依赖任何外部模块:

CreateExchangeOrder
  → 获取平台汇率（内部调 CurrencyService）
  → 计算到账 BSB + 10% 奖金
  → 校验法币余额
  → 事务内执行：扣法币 + 加BSB中心 + 加BSB奖励 + 写记录
  → 返回结果

关键点：
  · 这是整个模块中对事务要求最严格的操作（3次跨币种余额变更）
  · 使用下单时刻的汇率，不是页面展示时的
  · 可以最先开发和测试（零外部依赖）
```

## 7.5 币种配置的实现方向

```
完全独立，标准 CRUD:

  · 8 个 B 端接口：本质是对 currency_config 和 rate_config 表的增删改查
  · 3 个 RPC 接口：读取币种和汇率数据供外部调用
  · 2 个定时任务：拉取第三方汇率 + 检查阈值更新平台汇率

核心逻辑：
  · RecalculateRates 方法被 3 处调用（UpdateCurrency, UpdateRateConfig, CheckRateThreshold）
  · 四层汇率联动计算：实时汇率 → 阈值判断 → 平台汇率 → 充值/提现汇率
  · 汇率变更必须写变更日志

最先开发完成，因为它是整个系统的数据地基
```

# 八、Redis 缓存策略方向

> 依据：`common/pkg/consts/redis_key/redis_key.go`，现有 redis key 前缀规范

## 8.1 Redis Key 规划

需要在 `redis_key.go` 中新增 ser-wallet 相关的 Key 前缀：

```
wallet:balance:       — 用户余额缓存（高频读，缓存减轻数据库压力）
wallet:currency:      — 可用币种列表缓存
wallet:rate:          — 平台汇率缓存（定时更新）
wallet:user_currency: — 用户当前选中的币种
wallet:idempotent:    — 幂等去重缓存（可选）
```

## 8.2 缓存使用方向

```
需要缓存的场景（高频读）:
  · 用户余额 → 每次打开钱包页都会查
  · 可用币种列表 → 不经常变，缓存收益大
  · 平台汇率 → 定时更新，两次更新之间走缓存
  · 用户当前选中币种 → 几乎每个接口都需要

不应该缓存的场景:
  · 订单数据 → 写多读少，缓存容易不一致
  · 冻结金额 → 必须读数据库保证准确

缓存一致性：
  · 余额缓存在 Credit/Debit/Freeze/Unfreeze 后主动失效
  · 币种配置缓存在 CRUD 操作后主动失效
  · 汇率缓存通过定时任务覆盖更新
```

# 九、定时任务接入

> 依据：项目支持 XXL-JOB（分布式调度）和 robfig/cron（进程内调度）

## 9.1 ser-wallet 需要的定时任务

```
任务1: SyncThirdPartyRates（汇率同步）
  · 频率：分钟级（如每 5 分钟）
  · 内容：调 3 个第三方汇率 API，取平均值更新实时汇率
  · 触发后联动 CheckRateThreshold

任务2: CheckRateThreshold（阈值检查）
  · 频率：跟随任务1
  · 内容：判断实时汇率与平台汇率的偏差是否超阈值
  · 超阈值则更新平台汇率 + 联动更新充值/提现汇率 + 写变更日志
```

## 9.2 接入方式选择

```
方案A: 进程内 cron（适合简单场景）
  · 在 main.go 启动时注册 cron 任务
  · 适合单实例部署的定时汇率同步

方案B: XXL-JOB（适合分布式场景）
  · 通过 XXL-JOB 执行器接收调度
  · 适合多实例部署时保证只有一个实例执行

方案C: 通过 ser-cron 服务 RPC 调用
  · 项目已有 ser-cron（编号020）
  · 由 cron 服务定时调用 ser-wallet 的同步方法

建议根据部署方式选择，单实例用 A，多实例用 B 或 C
```

# 十、跨模块 RPC 对接策略

> 核心问题：充值和提现强依赖财务模块，但两边可能不是同步开发

## 10.1 需要调出去的 RPC（依赖他人）

```
调用财务管理模块（6个）:
  · GetDepositMethods       ← 充值页获取可用方式
  · MatchDepositChannel     ← 充值时匹配通道
  · CreateChannelOrder      ← 创建通道订单
  · GetWithdrawMethods      ← 提现页获取可用方式
  · SubmitWithdrawRequest   ← 提交提现审核
  · QueryChannelOrderStatus ← 查通道订单状态

调用其他服务（4个）:
  · ser-kyc.GetKycStatus    ← 提现资格校验
  · ser-buser.GetUserInfo   ← B端查用户信息
  · ser-s3.Upload           ← 币种 icon 上传
  · 第三方汇率 API × 3      ← 定时拉取汇率
```

## 10.2 需要暴露的 RPC（被他人调）

```
被财务管理模块调用（7个）:
  · QueryBalance       ← 人工修正前查余额
  · Credit             ← 充值成功/补单/人工加款
  · Debit              ← 提现完成/人工减款
  · Freeze             ← 提现冻结
  · Unfreeze           ← 提现解冻
  · QueryWalletDetail  ← B端查用户钱包
  · CreditReward       ← 奖金发放

被其他服务调用:
  · GetAvailableCurrencies ← 币种列表
  · GetExchangeRate        ← 汇率查询
  · GetCurrencyInfo        ← 币种信息
```

## 10.3 联调策略

```
核心思路：先把被调接口做好，再逐步对接调出接口

阶段一（独立开发期）:
  · 被调接口（Credit/Debit/Freeze 等）先开发好
    → 这些接口逻辑独立，不依赖任何外部模块
    → 开发好后财务模块可以直接调，不阻塞对方
  · 调出接口先写框架，用 Mock 数据跑通流程
    → 充值方式列表 Mock 固定数据
    → 通道匹配 Mock 固定返回
    → 提现提交 Mock 直接返回成功

阶段二（联调期）:
  · 财务模块接口就绪后，替换 Mock 为真实 RPC
  · 联调优先级：Credit(充值上分) > Debit/Unfreeze(提现结果) > 其他
  · KYC 服务如果不可用，提现资格校验做降级（允许USDT提现，禁止法币提现）
```

## 10.4 与财务模块的接口协商要点

```
开发前需要和财务模块负责人对齐的事项：

1. RPC 方法签名：双方确认请求/响应的字段定义
   · 财务调我的 Credit/Debit：orderNo、userId、currencyId、walletType、amount、reason
   · 我调财务的 MatchDepositChannel：currencyId、methodId、amount
   · 建议先各自定义 IDL 中的 Req/Resp，再对齐确认

2. 回调时机和方式：
   · 充值成功后，财务通过 RPC 同步调用 Credit（不是消息队列）
   · 提现审核完成后，财务通过 RPC 同步调用 Debit 或 Unfreeze
   · 确认：是同步 RPC 还是异步消息？这决定了错误处理策略

3. 幂等约定：
   · 确认 orderNo 的唯一性保证由谁负责
   · 确认重试策略（超时后是否重试、重试间隔）

4. 错误处理约定：
   · Credit 失败（如余额操作异常），财务应该怎么处理？
   · Debit 失败（如余额不足），财务应该怎么处理？
```

# 十二、需要提前确认的技术决策

这些是开发过程中绕不过去的决策点，建议在开始写代码前确认：

## 12.1 金额存储和计算方式

```
决策：金额在数据库中用什么类型存储？IDL 中用什么类型传输？

选项 A: 数据库 DECIMAL(20,8)，IDL string，Go 内部 shopspring/decimal
选项 B: 数据库 BIGINT（以最小单位的整数存储），IDL i64
选项 C: 数据库 BIGINT（统一用"分"为单位），IDL i64

影响范围：所有涉及金额的表和接口，一旦确定不易更改
```

## 12.2 乐观锁 vs 悲观锁

```
决策：余额更新的并发控制用什么方式？

选项 A: 悲观锁（SELECT ... FOR UPDATE）
  · 优点：简单可靠，不会丢失更新
  · 缺点：并发高时锁争用

选项 B: 乐观锁（version 字段）
  · 优点：无锁等待，并发性能好
  · 缺点：冲突时需要重试逻辑

选项 C: 基于 SQL 的原子更新（UPDATE balance = balance + ? WHERE balance >= ?）
  · 优点：单条 SQL 原子操作，无需锁
  · 缺点：无法获取变更前后的余额值（流水记录需要额外查询）

建议方向：选项 A 或 C，视并发量级而定
```

## 12.3 服务编号确认

```
决策：ser-wallet 使用哪个编号？

选项 A: 022（顺延，安全稳妥）
选项 B: 016（填空缺）

需要和团队确认，因为编号影响：
  · 错误码前缀（6022xxx vs 6016xxx）
  · ETCD 路径
  · namesp.go 常量
```

## 12.4 充值回调的通信方式

```
决策：财务模块充值成功后如何通知钱包上分？

选项 A: 同步 RPC 调用（财务直接调钱包 Credit）
  · 简单直接，错误可以立即处理

选项 B: 消息队列（Kafka/NATS）
  · 解耦，异步可靠投递
  · 需要消费确认和补偿机制

需要和财务模块负责人协商确认
```

# 十三、风险点和应对方向

## 13.1 已识别的风险

| 风险 | 影响 | 应对方向 |
|------|------|---------|
| 财务模块进度 | 充值/提现 C 端流程无法完整跑通 | 先开发独立部分(70%)，联调部分用 Mock |
| KYC 服务不可用 | 提现资格校验失败 | 降级策略：允许 USDT 提现，禁止法币提现 |
| 第三方汇率 API 不可用 | 实时汇率无法更新 | 使用最近一次缓存的汇率 |
| 余额操作并发 | 超扣/重复加款 | 幂等 + 行锁/乐观锁 |
| 兑换跨币种事务 | 扣了法币没加 BSB | 同一事务内完成，失败全部回滚 |
| 通道回调丢失 | 用户充值了但没到账 | 需要补单机制（财务模块负责） |

## 13.2 自己可以提前做的防护

```
1. 所有余额操作写流水（wallet_flow 表）
   → 即使出问题也有据可查可对账

2. Credit/Debit 幂等
   → 无论被调多少次，结果一致

3. 关键操作记录 klog 日志
   → 余额变更前后值、orderNo、操作类型

4. 单元测试覆盖核心场景
   → 正常加款、余额不足扣款、重复加款(幂等)、并发加扣款
```

# 总结

```
一句话总结实现思路：

  基础设施注册 → IDL 定义 → 建表生成代码 → 币种配置先行 →
  钱包 RPC 核心能力 → C 端独立功能 → 联调充值提现

绕不过的关键点：
  ① 金额类型决策（影响所有表和接口）
  ② 余额操作的幂等+事务+并发安全（质量底线）
  ③ 币种配置是地基（最先完成）
  ④ 被调 RPC 先就绪（不阻塞财务模块）
  ⑤ 兑换事务的跨币种原子性（最严格的事务场景）
  ⑥ 与财务模块的接口协商（回调方式、字段约定、错误处理）
```
