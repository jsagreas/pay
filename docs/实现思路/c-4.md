## 一、服务定位与模块划分

### 1.1 一个服务，两个业务域

ser-wallet 是一个 Kitex RPC 服务，内含两个业务域：

```
ser-wallet 服务
├── 币种配置域（B端）  → 纯数据管理，不涉及资金操作
└── 钱包域（C端+RPC供他） → 资金操作中枢，所有余额变动最终落这里
```

两个域在同一个服务内，好处是：币种配置的数据（币种列表、汇率）被钱包高频使用，同服务内部函数调用，零网络开销。

### 1.2 内部分层架构

遵循既有模块的分层规范（ser-app/ser-item 已验证的模式）：

```
handler.go                    ← IDL Service 接口实现，零逻辑，纯委托
    ↓
internal/ser/                 ← 业务逻辑层：参数校验、业务规则、编排
    ├── currency_service.go   ← 币种配置相关
    ├── wallet_service.go     ← 钱包 C端功能
    ├── balance_service.go    ← 余额操作底层能力（统一入口）
    └── exchange_service.go   ← 兑换功能
    ↓
internal/rep/                 ← 数据访问层：GORM Gen 查询、事务
    ├── currency_repo.go
    ├── wallet_repo.go
    ├── transaction_repo.go
    └── withdrawal_account_repo.go
    ↓
internal/gorm_gen/            ← 自动生成：model + query + repo
    ↓
internal/domain/              ← 领域层：外部服务封装、复杂校验
    ├── finance_service.go    ← 封装调用财务 RPC
    └── kyc_service.go        ← 封装调用 KYC RPC
```

**为什么这样分**：

- `ser/` 层编排业务流程，调用 `rep/` 做数据操作，调用 `domain/` 做外部交互
- `rep/` 层负责事务边界，ser 层不直接操作数据库
- `domain/` 层隔离外部 RPC 依赖，方便 Mock 和降级
- `balance_service.go` 是关键——所有余额变更操作的统一入口，下面展开说

> 依据：ser-app handler.go 委托模式 + ser-item domain/ 外部服务封装模式

## 二、IDL 设计思路

### 2.1 文件拆分策略

```
common/idl/ser-wallet/
├── service.thrift            ← 入口文件，定义 WalletService，include 所有子文件
├── currency.thrift           ← 币种配置域：所有请求/响应结构
├── wallet.thrift             ← 钱包 C端功能：余额、充值、兑换、提现
├── transaction.thrift        ← 交易流水：记录查询相关结构
└── rpc_for_finance.thrift    ← RPC供财务：被财务调用的接口结构
```

### 2.2 service.thrift 整体骨架

### 2.3 IDL 设计要点

**金额字段统一用 `i64`**：所有金额在传输和存储层面都使用最小单位的整数（如分、聪），避免浮点精度问题。展示层再按币种规则格式化。这是金融系统的标准做法。

**分页请求统一格式**：遵循 ser-app 的 `pageNo/pageSize` 在高编号位的惯例。

**C端接口不传 userId**：C端用户的 ID 从 TTHeader 上下文中获取（`tracer.GetUserId(ctx)`），不作为请求参数。RPC供他接口需要显式传 `userId`。

> 依据：工程认知 c-4 §3.1 IDL命名规范 + §2.1.6 链路追踪上下文

## 三、表结构设计思路

### 3.1 必须的核心表

需求确定不会变的"骨架表"——不管细节怎么调，这些表绕不过：

| 表名 | 职责 | 理由 |
|------|------|------|
| `currency_config` | 币种定义（代码/名称/类型/国家/图标/符号/展示规则/状态/排序） | 币种配置需求 B3，18个配置字段 |
| `exchange_rate` | 汇率数据（币种/实时/平台/入款/出款/浮动%/阈值%） | 汇率规则需求 B4，可以和 currency_config 合表也可独立，看字段量 |
| `rate_change_log` | 汇率变更日志（时间/币种/变更前后值） | 需求 B4 汇率日志页，运营要看 |
| `wallet` | 用户钱包余额（用户/币种/钱包类型/余额/冻结额/状态） | 钱包核心，一条记录 = 一个用户一个币种一个钱包类型 |
| `wallet_transaction` | 交易流水（用户/币种/类型/金额/订单号/状态/时间） | C端记录展示 + 对账 |
| `deposit_order` | 充值订单（订单号/用户/币种/金额/方式/状态/奖金关联） | 充值业务链路核心 |
| `withdrawal_order` | 提现订单（订单号/用户/币种/金额/方式/状态/账户信息） | 提现业务链路核心 |
| `withdrawal_account` | 提现账户（用户/账户类型/银行名/卡号/持有人/币种） | 用户保存的提现账户信息 |

### 3.2 表结构设计原则

**1. 金额字段全部用 `bigint`，存最小单位**

```sql
`balance`        bigint NOT NULL DEFAULT 0 COMMENT '余额（最小单位）',
`frozen_amount`  bigint NOT NULL DEFAULT 0 COMMENT '冻结金额（最小单位）',
```

为什么不用 `decimal`：`bigint` 在 Go 中映射为 `int64`，计算无精度损失，性能更优。展示时按币种规则除以单位因子再格式化。法币 × 100（分），加密货币 × 10^8（聪）。精度由币种配置中的小数位定义决定。

**2. wallet 表的唯一键设计**

```sql
UNIQUE KEY `uk_user_currency_type` (`user_id`, `currency_code`, `wallet_type`, `delete_at`)
```

一个用户 + 一个币种 + 一个钱包类型 = 唯一一条 wallet 记录。这保证了不会出现重复钱包。

**3. 订单表的幂等键设计**

```sql
-- deposit_order
UNIQUE KEY `uk_order_no` (`order_no`, `delete_at`)
-- withdrawal_order
UNIQUE KEY `uk_order_no` (`order_no`, `delete_at`)
```

订单号全局唯一，是幂等校验的锚点。

**4. wallet_transaction 的索引设计**

```sql
-- C端记录查询的核心索引
KEY `idx_user_currency_type` (`user_id`, `currency_code`, `tx_type`)
-- 幂等校验索引
UNIQUE KEY `uk_order_no_wallet_type` (`order_no`, `wallet_type`, `tx_type`, `delete_at`)
```

C端记录页按 `用户+币种+类型` 筛选是最高频查询，索引覆盖。

**5. 审计字段和软删除**

所有表统一：`create_by`, `update_by`, `create_at`, `update_at`, `delete_at`。GORM UserPlugin 自动填充审计字段。

**6. wallet_type 枚举值预留**

```
1=中心钱包, 2=奖励钱包, 3=主播钱包, 4=代理钱包(预留), 5=场馆钱包(预留)
```

代理和场馆虽然首版不实现业务逻辑，但 `wallet_type` 字段本身支持这些值，表结构不需要改。

> 依据：工程认知 c-4 §6.1 建表规范 + 需求理解 c-4 §3.1 五种子钱包 + 需求理解 c-4 §9 订单号规范

### 3.3 currency_config vs exchange_rate 合表 or 拆表

**倾向合表**。理由：

- 币种和汇率是 1:1 关系（每个币种只有一套当前汇率）
- 币种配置总量极少（4-6种），不存在性能问题
- 汇率同步更新时直接更新币种表的汇率字段，一条 SQL 搞定
- 拆表只增加了 JOIN 复杂度，收益不大

汇率历史变更用独立的 `rate_change_log` 表记录。

### 3.4 关于"钱包记录 vs 财务记录"

充值/提现订单在**钱包和财务各自存一份**：

- 钱包存 `deposit_order` / `withdrawal_order` —— 面向C端用户展示、面向余额变更触发
- 财务存自己的订单表 —— 面向B端运营管理、面向通道和审核流程

两侧通过**订单号**关联。钱包不需要知道财务的通道细节，财务不需要知道钱包的余额细节。

> 依据：流程思路 c-4 §10.2 待确认决策点1，当前倾向=各存各的

## 四、核心底层能力：统一余额变更入口

这是整个钱包模块最关键的设计。所有余额变动——不管是充值上账、兑换扣减、提现冻结、人工加减款——**都必须经过统一的底层函数**。

### 4.1 为什么必须统一

1. **对账一致性**：每次余额变更都伴随一条交易流水，统一入口保证不遗漏
2. **并发安全**：统一入口可以加行锁，防止同一个用户的余额被并发操作破坏
3. **审计追溯**：所有变动都有来源（充值/兑换/提现/人工），统一入口保证来源信息完整
4. **代码复用**：W-R2/W-R4/W-R5/W-R6/W-R7/W-R8/W-9/W-11 全部复用同一套底层

### 4.2 五个底层函数的职责

```
balance_service.go 提供的五个核心操作：

addBalance(ctx, userId, currency, walletType, amount, txType, orderNo)
  → 增加余额，写入流水
  → 调用方：W-R2(充值上账), W-R6(人工加款), W-R8(补单), W-9(兑换加BSB)

deductBalance(ctx, userId, currency, walletType, amount, txType, orderNo)
  → 减少余额（需校验余额充足），写入流水
  → 调用方：W-R5(确认扣款), W-R7(人工减款), W-9(兑换扣法币)

freezeBalance(ctx, userId, currency, walletType, amount, orderNo)
  → 可用余额 → 冻结余额，写入流水
  → 调用方：W-11(创建提现时冻结)

unfreezeBalance(ctx, userId, currency, walletType, amount, orderNo)
  → 冻结余额 → 可用余额，写入流水
  → 调用方：W-R4(提现解冻返还)

confirmDeduct(ctx, userId, currency, walletType, amount, orderNo)
  → 冻结余额永久扣除（冻结额减少，资金移出），写入流水
  → 调用方：W-R5(提现出款成功)
```

### 4.3 每个底层函数内部的通用流程

```
任何余额变更函数内部：
  BEGIN TRANSACTION
    SELECT balance, frozen_amount FROM wallet
      WHERE user_id=? AND currency_code=? AND wallet_type=?
      FOR UPDATE;                              ← 行锁，防并发

    -- 前置校验（余额充足、冻结额充足等）
    -- 计算新余额

    UPDATE wallet SET balance=?, frozen_amount=? WHERE id=?;

    INSERT INTO wallet_transaction (...) VALUES (...);  ← 同事务写流水
  COMMIT

为什么用数据库行锁：
  - 锁粒度精确到「用户+币种+钱包类型」，不影响其他用户
  - 事务结束自动释放，不存在锁泄漏
  - 余额和流水在同一事务中，原子一致
```

### 4.4 事务在哪一层管理

**事务在 rep 层管理，ser 层编排调用**。

单操作场景（如人工加款只加一个钱包）：rep 层函数内部开事务。

多操作场景（如兑换：扣法币中心钱包 + 加BSB中心钱包 + 加BSB奖励钱包）：ser 层调用 rep 的事务函数，rep 函数接受 `*query.Query`（事务上下文），在同一个事务里完成多步操作。

```go
// rep 层提供事务包装
func (r *WalletRepo) WithTransaction(fn func(tx *query.Query) error) error {
    return query.Q.Transaction(fn)
}

// ser 层编排：兑换场景
func (s *ExchangeService) Execute(ctx, req) {
    s.walletRepo.WithTransaction(func(tx *query.Query) error {
        // 1. 扣法币
        if err := s.balanceService.DeductInTx(tx, ctx, ...); err != nil { return err }
        // 2. 加BSB中心钱包
        if err := s.balanceService.AddInTx(tx, ctx, ...); err != nil { return err }
        // 3. 加BSB奖励钱包（赠送部分）
        if err := s.balanceService.AddInTx(tx, ctx, ...); err != nil { return err }
        return nil
    })
}
```

> 依据：ser-app/internal/rep 事务模式（query.Q.Transaction） + 流程思路 c-4 §七 共享函数层

## 五、幂等性实现思路

### 5.1 哪些接口必须幂等

所有 RPC 供财务调用的写操作：W-R2/W-R3/W-R4/W-R5/W-R6/W-R7/W-R8。

原因：财务侧可能因网络抖动重试，三方支付回调可能重复推送。同一笔操作不能执行两次。

### 5.2 幂等键 = 订单号

每个 RPC 写操作都携带 `orderNo`。在执行余额变更前：

```
1. 查 wallet_transaction 表：是否已存在该 orderNo 的已完成记录
2. 存在 → 直接返回成功（已处理过）
3. 不存在 → 执行余额变更 + 写入流水
```

`wallet_transaction` 表的 `uk_order_no_wallet_type` 唯一键提供数据库级防重保障——即使应用层并发两个相同请求通过了查询，INSERT 时也会唯一键冲突。

### 5.3 C端接口的幂等考虑

- W-4 CreateDepositOrder：用重复转账拦截逻辑（同用户+同币种+同金额+状态=待支付/进行中，≥3笔拦截）作为业务层防重
- W-9 ExecuteExchange：汇率快照ID作为幂等因子，同一个快照只能执行一次
- W-11 CreateWithdrawalOrder：冻结和创建在同一事务内，冻结的原子性自动保证

> 依据：流程思路 c-4 §4.2 RPC通用处理模式 + 需求理解 c-4 §4.5 非功能性要求

## 六、汇率系统实现思路

### 6.1 汇率数据结构

每个币种维护四个汇率值（都以"1单位该币种 = X 单位基准币种"为基准方向）：

```
realtime_rate    实时汇率：三方API取均值
platform_rate    平台汇率：偏差超阈值时从实时汇率同步
deposit_rate     入款汇率：platform_rate × (1 + float_percent)
withdrawal_rate  出款汇率：platform_rate × (1 - float_percent)
```

### 6.2 定时同步策略

```
SyncRealtimeRate 定时任务（建议 cron 触发或内部 ticker）

每次执行：
  1. 获取所有已启用币种列表
  2. 对每个币种并发请求3家汇率API（HTTP）
  3. 容错：
     - 3家都成功 → 取平均值
     - 2家成功1家失败 → 取2家平均值
     - 1家成功2家失败 → 取该1家的值
     - 全部失败 → 本轮跳过该币种，不更新
  4. 计算偏差 = |新实时 - 当前平台| / 新实时 × 100%
  5. 偏差 < 阈值 → 只更新 realtime_rate
  6. 偏差 ≥ 阈值 → 更新全部四个汇率 + 写 rate_change_log
```

### 6.3 精度规则

```
法定货币（VND/IDR/THB）：保留2位小数，× 100 存整数
加密货币（USDT/BTC/ETH）：保留8位小数，× 10^8 存整数
平台币（BSB）：保留2位小数，× 100 存整数
```

**汇率值本身的存储**：因为汇率是两个币种的比率（可能是小数），建议汇率字段使用 `bigint` 存放放大后的值（如放大 10^8），在应用层做计算和格式化。

### 6.4 汇率快照机制（兑换场景）

用户在 W-8 预览兑换时：

```
1. 读取当前平台汇率
2. 生成 snapshot_id（可以用 "currency_code + timestamp + rate_value" 的哈希）
3. 将 snapshot_id → rate_value 存入 Redis，设置 TTL（如5分钟）
4. 返回给前端

用户在 W-9 执行兑换时：
1. 传入 snapshot_id
2. 从 Redis 读取快照汇率
3. 不存在（过期）→ 拒绝，要求重新预览
4. 存在 → 用快照汇率执行兑换
```

> 依据：需求理解 c-4 §3.3 汇率体系 + 流程思路 c-4 §2.4 汇率同步 + §3.2 兑换

## 七、缓存策略思路

### 7.1 哪些数据需要缓存

| 数据 | 缓存理由 | 缓存位置 | 过期/失效策略 |
|------|---------|---------|-------------|
| 启用的币种列表 | C端每次打开钱包都查，币种数据几乎不变 | Redis | B端修改时主动清除 |
| 当前汇率值 | 兑换/充值/提现页面频繁读取 | Redis | 汇率同步任务更新时覆盖 |
| 基准币种 | 设一次不变 | 内存/Redis | 永不过期 |
| 汇率快照 | 兑换预览→执行之间的临时数据 | Redis | TTL 5分钟自动过期 |

### 7.2 哪些数据不应该缓存

- **余额**：不缓存。每次查余额都走数据库。原因：余额变动频繁且要求强一致，缓存引入的一致性问题远大于性能收益
- **交易流水**：不缓存。分页查询走数据库索引已经够快
- **订单数据**：不缓存。订单状态流转频繁，缓存维护成本高

### 7.3 Redis Key 规范

```
wallet:currency:list                    → 启用币种列表
wallet:currency:rate:{currency_code}    → 当前汇率值
wallet:currency:base                    → 基准币种代码
wallet:exchange:snapshot:{snapshot_id}  → 汇率快照
```

> 依据：ser-app cache 层模式（service层管缓存读写、mutation时清除）

## 八、与财务模块联调思路

### 8.1 联调接口清单

**我调财务（2个查询接口，需要财务先提供）**：

| 接口 | 调用场景 | 失败影响 | 降级策略 |
|------|---------|---------|---------|
| 财务.GetAvailableDepositMethods | W-3 用户打开充值页 | 充值方式列表为空 | 返回空列表 + 提示"暂无可用充值方式" |
| 财务.GetAvailableWithdrawalMethods | W-10 用户打开提现页 | 提现方式列表为空 | 返回空列表 + 提示"暂无可用提现方式" |

**财务调我（8个接口，我需要先提供）**：

| 接口 | 财务触发场景 | 优先级 |
|------|------------|--------|
| W-R1 GetUserWalletBalance | 加减款弹窗查余额 | 高（联调前置） |
| W-R2 CreditDeposit | 充值回调上账 | 高（充值链路终点） |
| W-R3 FreezeForWithdrawal | 提现冻结 | 高（提现链路关键） |
| W-R4 UnfreezeForWithdrawal | 提现解冻 | 高（提现链路关键） |
| W-R5 ConfirmWithdrawalDeduct | 提现扣款 | 高（提现链路关键） |
| W-R6 ManualCredit | 人工加款 | 中 |
| W-R7 ManualDebit | 人工减款 | 中 |
| W-R8 CreditCorrection | 补单 | 中 |

### 8.2 联调策略

**先内后外，先独立后联动**：

```
第一步：先把币种配置做完，自测闭环
  → 不依赖任何外部模块，可独立完成

第二步：兑换做完，自测闭环
  → 不依赖财务，只需要币种配置的汇率数据（内部调用）
  → 此时可以验证：余额变更+流水写入的事务正确性、汇率快照机制

第三步：提供 W-R1~W-R8 给财务
  → 这8个接口不依赖财务的任何接口
  → 可以写单元测试模拟调用验证：幂等性、事务原子性、状态机正确性
  → 财务可以先用这些接口跑通他们的充值回调、提现审核、人工修正流程

第四步：对接财务的 2 个查询接口
  → 充值方式 + 提现方式的查询
  → 此时充值和提现的 C端页面才能渲染完整
```

### 8.3 联调约定（需要和财务方提前对齐的）

| 约定项 | 说明 |
|--------|------|
| 订单号规范 | C+16位（充值）、T+16位（提现）、A/M/B+16位（加/减/补） |
| 金额单位 | 最小单位整数，不传浮点 |
| 幂等约定 | 同一个 orderNo 重复调用返回成功，不重复执行 |
| 错误码约定 | 钱包返回 6022xxx，财务需处理 |
| 状态同步 | 充值订单和提现订单各自管各自的状态，通过订单号关联 |

> 依据：链路关系 c-4 §四 跨模块调用汇总 + 依赖关系 c-4 §六 双向依赖分析

## 九、外部 RPC 调用封装

### 9.1 调用财务的封装


## 十五、绕不过的关键设计总结

把前面内容浓缩为"无论需求怎么变，这些一定要做"的清单：

### 必须做的结构性设计

| 编号 | 设计项 | 原因 |
|------|--------|------|
| S-1 | 统一余额变更入口（balance_service） | 对账一致性的根基，没有这个一切金融操作都不可靠 |
| S-2 | 余额变更 + 流水写入在同一事务 | 金融系统的硬性要求 |
| S-3 | 行级锁（SELECT FOR UPDATE） | 防止并发操作破坏余额一致性 |
| S-4 | 所有 RPC 写操作幂等 | 三方回调重复、网络重试是必然场景 |
| S-5 | 冻结/解冻/扣款状态机 | 提现链路的资金安全保障 |
| S-6 | 金额用整数存储（最小单位） | 避免浮点精度问题，金融系统标准 |
| S-7 | 汇率快照机制 | 用户看到的汇率和执行的汇率必须一致 |
| S-8 | 订单号全局唯一 + 作为幂等键 | 跨模块关联、防重处理的锚点 |

### 必须遵守的工程规范

| 编号 | 规范 | 原因 |
|------|------|------|
| E-1 | handler.go 零逻辑 | 既有模块规范，保持一致 |
| E-2 | 错误码用 6022xxx | 工程错误码体系 |
| E-3 | 日志带 ctx（自动关联 TraceID） | 链路追踪 |
| E-4 | C端 userId 从 ctx 取 | TTHeader 自动传播 |
| E-5 | 所有表有审计字段 + 软删除 | GORM UserPlugin + 工程建表规范 |
| E-6 | 写操作后记审计日志（oneway ser-blog） | 工程审计要求 |
| E-7 | 外部 RPC 调用封装在 domain 层 | 隔离依赖，便于 Mock 和降级 |

### 必须提前想清楚的业务规则

| 编号 | 规则 | 原因 |
|------|------|------|
| B-1 | 先扣中心钱包再扣奖励钱包 | 消费/投注的扣款优先级，影响数据模型设计 |
| B-2 | 奖金/赠送进奖励钱包，有流水要求 | 影响充值上账和兑换的流水记录设计 |
| B-3 | 法币充→法币提，USDT充→USDT提 | 提现规则校验需要追溯充值来源 |
| B-4 | W-R4 和 W-R5 互斥 | 提现订单终态只能走一条路径 |
| B-5 | 减款余额不足扣实际余额 | 人工减款的特殊逻辑，不报错 |
| B-6 | 账户持有人 = KYC认证姓名 | 提现账户管理的合规约束 |
