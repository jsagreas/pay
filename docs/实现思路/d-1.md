## 一、整体实现策略

### 1.1 一个服务，两个子模块

币种配置和钱包业务放在**同一个 ser-wallet 服务内**，通过 internal 包分层隔离。

**为什么不拆两个服务**：
- 币种配置的调用方主要就是钱包自己，内部方法调用比 RPC 更高效
- 两者共享同一个数据库，事务操作更简单
- 参考现有工程：ser-item 内部也包含了 item + item_publish 两个子模块，惯例如此
- 对外暴露的 RPC 仍然是一个 WalletService，财务模块只需要引入一个客户端

**内部分层方式**：
```
ser-wallet/internal/
├── ser/
│   ├── currency_service.go     ← 币种配置业务
│   ├── rate_service.go         ← 汇率管理业务
│   ├── wallet_service.go       ← 余额核心引擎
│   ├── deposit_service.go      ← 充值业务
│   ├── withdraw_service.go     ← 提现业务
│   ├── exchange_service.go     ← 兑换业务
│   └── record_service.go       ← 记录查询
├── rep/
│   ├── currency_repo.go
│   ├── wallet_account_repo.go
│   ├── deposit_order_repo.go
│   └── ...
└── ...
```

币种配置和钱包业务之间的调用就是 service 层的普通方法调用（如 deposit_service 内部调用 currency_service 获取汇率），不走 RPC。

### 1.2 开发主线

```
IDL定义 → 代码生成 → 基础设施注册 → 表结构设计 → 模型生成
→ 币种配置实现 → 余额引擎实现 → 各业务模块实现 → 网关路由注册 → 联调
```

这条主线中，每一步都是下一步的前置依赖，顺序不可跳过。

## 二、IDL 设计思路

### 2.1 绕不过的事

IDL 是整个服务的**契约起点**，所有 RPC 代码都由它生成。必须先定义好 IDL 才能生成 handler 骨架、才能让财务模块引入客户端调用。

### 2.2 文件拆分策略

```
common/idl/ser-wallet/
├── service.thrift          ← 主入口：定义 WalletService，include 其他文件
├── wallet.thrift           ← 余额相关请求/响应（C端查询+RPC提供给财务）
├── deposit.thrift          ← 充值相关
├── withdraw.thrift         ← 提现相关
├── exchange.thrift         ← 兑换相关
├── currency.thrift         ← 币种配置相关（B端CRUD + RPC供给）
└── record.thrift           ← 记录查询
```

**为什么这样拆**：参考 ser-app 的 IDL 组织方式（service.thrift include 多个子文件），按业务域拆分便于后续修改某个域的接口时不影响其他文件。

### 2.3 接口命名粒度

按工程惯例，写操作严格拆分（参考 ser-app/ser-item 的实际 IDL）：

```
Create/Update/Status/Delete 四个独立方法，不合并
B端分页用 Page 前缀，C端列表用 Get/List 前缀
写操作返回空结构体（成功即隐含，不返回 bool）
```

### 2.4 优先约定给财务的 RPC 接口

钱包↔财务是双向依赖，最容易互相阻塞。应该**最早约定**这些接口的 IDL 签名，让财务方可以提前引入客户端：

| 接口 | 方向 | 为什么要先定 |
|------|------|------------|
| CreditBalance | 财务→钱包 | 充值入账，财务回调必用 |
| FreezeBalance | 财务→钱包 | 提现冻结 |
| UnfreezeBalance | 财务→钱包 | 提现驳回/出款失败 |
| DebitBalance | 财务→钱包 | 提现出款成功 |
| ManualAdjust | 财务→钱包 | 人工加款/减款 |
| GetUserBalance | 财务→钱包 | 减款前校验 |

这 6 个接口的 request/response 结构需要和财务方**尽早对齐**，避免后期返工。

同时需要和财务确认的 2 个接口（我调财务）：
- GetPaymentMethods — 入参出参约定
- MatchDepositChannel — 入参出参约定

## 四、数据库表结构设计思路

### 4.1 绕不过的事

表结构是所有业务逻辑的**数据载体**，设计不对后面改动成本极高。但目前需求未完全定型，所以策略是：**核心骨架表必须有，辅助字段可以后加**。

### 4.2 必须存在的核心表

以下表是**无论需求怎么变都绕不过去**的：

**① wallet_account — 钱包账户表（核心中的核心）**

```
每个用户 × 每个币种 × 每种钱包类型 = 一行记录
关键字段（确定性的）：
  - user_id           用户ID
  - currency_id       关联币种
  - wallet_type       钱包类型（中心/奖励/主播/代理）
  - available_balance  可用余额
  - frozen_balance     冻结余额
  - created_at / updated_at / deleted_at  标准时间字段
```

这张表的设计原则：**余额字段用 bigint 存储最小单位**（如越南盾存整数、BSB 存到小数点后2位换算的整数），避免浮点精度问题。具体精度方案需要再定，但"用整数存最小单位"这个方向是确定的。

**② transaction_log — 流水表（账务可追溯的根基）**

```
每次余额变动必须有一条对应的流水
关键字段：
  - user_id
  - currency_id
  - wallet_type
  - tx_type            交易类型（充值/提现/兑换/冻结/解冻/扣款/奖励/人工加/人工减）
  - amount             变动金额（正数增加，负数减少）
  - balance_after      变动后余额（事后对账用）
  - ref_order_id       关联单号（充值单/提现单/兑换单/修正单）
  - created_at
```

这张表只增不改、不删。它是**对账和审计的唯一数据源**。

**③ deposit_order — 充值订单表**

```
关键字段：
  - order_no           订单号（雪花ID）
  - user_id
  - currency_id
  - amount             充值金额
  - actual_amount      实际到账（可能因汇率有差异）
  - payment_method     充值方式
  - channel_id         关联通道
  - status             状态（待支付/进行中/成功/失败/超时）
  - rate_snapshot      汇率快照（BSB充值时）
  - bonus_activity_id  关联奖金活动（可选）
  - bonus_amount       奖金金额（可选）
  - created_at / updated_at
```

**④ withdraw_order — 提现订单表**

```
关键字段：
  - order_no
  - user_id
  - currency_id
  - amount             提现金额
  - fee                手续费
  - withdraw_method    提现方式
  - withdraw_account_id 关联提现账户
  - status             状态（待审核/审核中/出款中/已完成/失败）
  - rate_snapshot      汇率快照（BSB提现时）
  - freeze_record_id   关联冻结记录
  - created_at / updated_at
```

**⑤ withdraw_account — 提现账户表**

```
关键字段：
  - user_id
  - currency_id
  - account_type       账户类型（银行卡/电子钱包/USDT地址）
  - bank_name / wallet_type / network_type  根据类型不同的字段
  - account_no         账号/手机号/地址
  - holder_name        持有人姓名（=KYC姓名）
  - created_at / updated_at / deleted_at
```

**⑥ exchange_record — 兑换记录表**

```
关键字段：
  - user_id
  - source_currency_id    源币种
  - target_currency_id    目标币种（BSB）
  - source_amount         源金额
  - target_amount         兑换获得
  - bonus_ratio           赠送比例
  - bonus_amount          赠送金额
  - rate_snapshot         兑换汇率
  - created_at
```

**⑦ currency — 币种配置表**

```
关键字段：
  - currency_code       币种代码（VND/IDR/THB/USDT/BSB）
  - currency_name       币种名称
  - currency_type       类型（法定/加密/平台）
  - icon_url            图标URL
  - country             国家
  - symbol              货币符号（d / Rp / ฿ / $ 等）
  - thousands_sep       千分位符号
  - decimal_sep         小数位符号
  - decimal_places      小数位数
  - platform_rate       平台汇率
  - deposit_rate        入款汇率
  - withdraw_rate       出款汇率
  - rate_float_pct      汇率浮动%
  - threshold_pct       阈值%
  - sort_order          排序
  - status              启用/禁用
  - operator / created_at / updated_at / deleted_at
```

**⑧ exchange_rate_log — 汇率变更日志表**

```
关键字段：
  - currency_id
  - real_time_rate      拉取时的实时汇率
  - platform_rate       更新后的平台汇率
  - deposit_rate        更新后的入款汇率
  - withdraw_rate       更新后的出款汇率
  - created_at
```

### 4.3 可能需要但暂不确定的表

| 表 | 用途 | 不确定的原因 |
|---|------|------------|
| freeze_record | 冻结/解冻记录 | 可能合并到 transaction_log 中用 tx_type 区分，也可能独立 |
| deposit_config / withdraw_config | 充值档位/提现限额配置 | 可能直接挂在 currency 表上作为 JSON 字段，也可能独立表 |
| bonus_config | 奖金活动配置 | 数据归属（钱包 or 活动系统）未确定 |

**思路**：先不急建这些表，等核心表跑通后再按需补充。

### 4.4 表设计的通用约定

按工程惯例（工程认知文档确认的规范）：
- 时间字段用 `bigint` 毫秒时间戳，不用 `datetime`
- 软删除用 `deleted_at bigint default 0`，0 表示未删除
- 金额字段用 `bigint` 存最小单位，**不用 decimal**（避免 ORM 类型映射问题，精度由应用层控制）
- 主键自增 ID + 业务唯一索引（如 order_no）

## 五、余额引擎实现思路

### 5.1 绕不过的事

余额引擎是钱包的**心脏**。所有业务流程（充值/兑换/提现/人工修正）最终都归结为对余额的操作。这部分的设计质量直接决定资金安全。

### 5.2 核心设计：两个底层方法

整个余额引擎只暴露两个内部方法，所有上层接口通过它们操作余额：

**getBalance**：读取某用户某币种某钱包类型的可用余额和冻结余额。如果记录不存在则懒创建。

**modifyBalance**：在事务内修改余额 + 同时写流水。这是**唯一允许修改余额的入口**。

```
modifyBalance 的核心逻辑伪代码：

func modifyBalance(tx, userId, currencyId, walletType, opType, amount, refOrderId) error {
    1. 加行锁：SELECT ... FOR UPDATE（防止并发修改同一账户）
    2. 根据 opType 执行：
       - credit:       available += amount
       - debit:        available -= amount（前置校验 available >= amount）
       - freeze:       available -= amount, frozen += amount
       - unfreeze:     frozen -= amount, available += amount
       - debit_frozen: frozen -= amount
    3. 更新 wallet_account
    4. 写入 transaction_log（金额、类型、变动后余额、关联单号）
    5. 返回
}
```

**为什么这样设计**：
- 统一入口保证了**任何余额变动都有流水记录**，不可能出现"余额变了但没流水"的情况
- 行锁（FOR UPDATE）防止并发操作同一账户导致超扣
- 事务保证余额和流水的原子性

### 5.3 幂等方案

所有资金操作的 RPC 接口都必须幂等。方案：

```
每个请求带一个 ref_order_id（关联单号），作为幂等键。
处理前先查：该 ref_order_id 是否已处理过？
  → 已处理 → 直接返回成功
  → 未处理 → 执行操作

实现方式：transaction_log 表上对 ref_order_id 建唯一索引，
写入时如果冲突说明已处理过。
```

### 5.4 金额精度方案

```
原则：数据库和内部计算全部用整数（最小单位），仅在展示层换算

示例：
  VND 无小数位 → 数据库存储 1000 即代表 1000 VND
  BSB 2位小数 → 数据库存储 100 即代表 1.00 BSB
  THB 2位小数 → 数据库存储 350 即代表 3.50 THB
  USDT 8位小数 → 数据库存储 100000000 即代表 1.00000000 USDT

汇率计算时用整数乘除+取整规则，避免浮点数误差
```

具体换算倍率（每个币种的 decimal_places 在 currency 表中配置），在应用层读取后做乘除。

## 六、币种配置实现思路

### 6.1 绕不过的事

币种配置是**所有业务的前置依赖**（依赖关系文档确认），没有币种数据就没有后续一切。必须最先实现。

### 6.2 实现要点

**B端 CRUD**：标准模式，参考 ser-app 的 home_category 实现，没有特殊逻辑。注意：
- 币种数据由开发导入，后台可编辑图标/状态/汇率浮动/阈值
- `StatusCurrency` 独立方法，启用/禁用切换
- 基准币种设置是**一次性操作**，提交后不可修改 — 需要一个标记位或独立配置表记录

**汇率自动更新**：需要一个定时任务（可用 cron 或 ticker），流程是：
1. 请求 3 家三方汇率 API → 取平均
2. 比较偏差值是否超阈值
3. 超阈值则更新平台汇率 + 重算入出款汇率 + 写日志

定时任务的实现方式有几种选择：
- 在 ser-wallet 内部起一个 goroutine ticker（最简单，单实例部署可行）
- 通过 ser-cron 调度（工程中已有 cron 服务）
- 通过外部调度触发一个刷新接口

**具体选哪种取决于部署方案，先按最简单的内部 ticker 实现，后续可调整。**

**RPC 供给接口**：5 个 RPC 接口给钱包业务层和财务模块用。因为同服务内的调用走内部方法，实际 RPC 只有财务才会走到。但接口定义和实现都要做完整，保证内部调用和 RPC 调用走同一套逻辑。

## 七、充值实现思路

### 7.1 绕不过的事

充值链路最长，横跨多个模块。但拆开来看，**我这边要实现的核心就三件事**：
1. 展示充值方式（调财务）
2. 创建充值订单（本地）
3. 充值回调入账（被财务调）

### 7.2 实现要点

**充值方式展示**：本质是一个聚合接口 — 调财务拿方式列表 + 调币种配置拿档位 + 拿汇率。组装后返回给前端。需要等财务就绪才能联调，但接口框架可以先实现，财务部分先 mock。

**创建充值订单**：核心在于校验链（重复检测 → 金额校验 → 通道匹配 → 建单）。其中通道匹配调财务，其他全是本地逻辑。**汇率快照要在建单时锁定**，写入订单记录，后续入账按快照汇率计算。

**充值回调入账（CreditBalance / DepositCallback）**：这是**被财务调用的 RPC 接口**。核心就是调用余额引擎的 modifyBalance(credit)，加上幂等校验。实现上：
- 不依赖财务的任何代码，可完全独立开发
- 用 mock 请求自测即可
- 后续联调只需对齐入参出参

**重复转账拦截**：纯本地 DB 查询（同 userId + 同 currencyId + 同 amount 的待支付订单数），逻辑简单。需求文档给了明确的分级规则（1~2笔提示、3笔强拦截），照着实现即可。

### 7.3 与财务的接口边界

```
充值流程中我和财务的分工边界：

我负责的：
├── 充值方式展示（我聚合、我返给前端）
├── 重复拦截（我自己检测）
├── 创建订单（我自己建单）
├── 入账（我执行余额变更）
└── 订单状态查询（我查自己的表）

财务负责的（我只是调它）：
├── GetPaymentMethods — 告诉我哪些方式可用
├── MatchDepositChannel — 告诉我用哪个通道
├── 接收三方回调 — 三方先回调财务
└── 转发给我入账 — 财务调我的 CreditBalance

分工清晰，我只有 2 个 RPC 出口依赖财务
```

## 八、提现实现思路

### 8.1 绕不过的事

提现是**流程最复杂**的链路，但从我的模块视角看，核心就四件事：
1. 前置校验（KYC + 限额）
2. 冻结余额 + 建单
3. 提交给财务审核
4. 响应财务回调（解冻/扣款）

### 8.2 实现要点

**KYC 校验**：调 ser-kyc 的现有接口（KycQuery），返回认证状态。这个不需要我实现，只是调一下。处理逻辑是前端根据返回的 KYC 状态决定弹窗还是直接进入。

**冻结 + 建单必须原子**：
```
一个DB事务内：
  1. modifyBalance(freeze, 提现金额+手续费)
  2. 写入 withdraw_order（状态=待审核）
  3. 两步都成功 → 提交事务
  任一失败 → 回滚（余额不动，订单不建）
```
这是资金安全的硬约束，没有商量余地。

**提交给财务审核**：这个环节的对接方式需要和财务确认。可能是：
- 方案A：我建好提现单后，财务定时查询我的待审核订单（轮询）
- 方案B：我建好单后 RPC 通知财务有新的提现单
- 方案C：我建好单后发消息到 MQ（NATS/Kafka），财务消费

无论哪种方案，我这边要做的核心是**建好单+冻结好余额**。提交方式是对接细节，后续和财务对齐。

**响应财务回调**：UnfreezeBalance 和 DebitBalance 的实现，本质是调用余额引擎的 modifyBalance(unfreeze) 和 modifyBalance(debit_frozen)。逻辑与 CreditBalance 类似（幂等 + 事务），可独立开发，用 mock 自测。

### 8.3 提现账户管理

首次提现需绑定账户。实现简单：一张 withdraw_account 表，CRUD 操作。唯一的联动点是从 KYC 拿认证姓名填充"持有人"字段。后续提现复用已绑定信息。

## 九、兑换实现思路

### 9.1 绕不过的事

兑换是钱包内部**完全自闭环**的功能，不依赖财务，不涉及三方。因此可以**最早做出可验证的端到端功能**。

### 9.2 实现要点

```
两个接口：
  ExchangePreview — 纯计算，不写DB，输入金额→输出预估结果
  ExecuteExchange — 事务内执行，扣减源币种+增加BSB

事务内操作：
  1. 获取并锁定当前汇率
  2. 校验源币种可用余额
  3. modifyBalance(源币种·中心, debit, 输入金额)
  4. modifyBalance(BSB·中心, credit, 兑换获得)
  5. modifyBalance(BSB·奖励, credit, 赠送金额)   ← 如果有赠送
  6. 写入 exchange_record
```

**赠送比例的数据来源待确认**（币种配置管理 or 活动系统），但不影响主流程实现 — 先按配置参数传入，来源后定。

## 十、记录查询实现思路

### 10.1 绕不过的事

四类记录（充值/提现/兑换/奖励）是用户在 C 端看到的历史数据，也是客服和运营在 B 端排查问题的工具。

### 10.2 实现要点

**全部是纯 DB 查询**，无外部依赖，实现最简单：
- C 端：按 userId + currencyId + 分页查询，状态筛选
- B 端：按各种维度筛选 + 导出 CSV

**核心规则**：当前币种 = 记录币种，不支持跨币种混合记录。这决定了查询条件中 currencyId 是必传参数。

**实现优先级高**：可以在表设计好之后立即实现，作为后续测试充值/兑换/提现是否正确的**验证窗口**。

## 十一、网关路由集成思路

### 11.1 绕不过的事

所有接口必须通过网关才能被前端调用。gate-font（C 端）和 gate-back（B 端）各自需要注册路由和 handler。

### 11.2 实现要点

**C 端路由（gate-font）**：

```
路由组：my_router.Api（需要登录态）
路由前缀：/api/wallet/...

钱包核心：
  POST /api/wallet/balance          → GetWalletBalance
  POST /api/wallet/overview         → GetWalletOverview

充值：
  POST /api/wallet/deposit/methods  → GetDepositMethods
  POST /api/wallet/deposit/create   → CreateDepositOrder
  POST /api/wallet/deposit/usdt     → GetUsdtDepositAddress
  POST /api/wallet/deposit/check    → CheckDuplicateDeposit
  POST /api/wallet/deposit/status   → GetDepositOrderStatus

兑换：
  POST /api/wallet/exchange/preview → ExchangePreview
  POST /api/wallet/exchange/execute → ExecuteExchange

提现：
  POST /api/wallet/withdraw/methods      → GetWithdrawMethods
  POST /api/wallet/withdraw/check        → CheckWithdrawEligibility
  POST /api/wallet/withdraw/accounts     → ListWithdrawAccounts
  POST /api/wallet/withdraw/bind         → BindWithdrawAccount
  POST /api/wallet/withdraw/create       → CreateWithdrawOrder
  POST /api/wallet/withdraw/limit        → GetWithdrawLimit

记录：
  POST /api/wallet/record/deposit/page   → PageDepositRecords
  POST /api/wallet/record/deposit/detail → DetailDepositRecord
  ...（提现/兑换/奖励同理）
```

**B 端路由（gate-back）**：

```
路由组：my_router.Api（需要登录态 + URL权限校验）
路由前缀：/admin/api/wallet/...

币种配置：
  POST /admin/api/wallet/currency/page    → PageCurrency
  POST /admin/api/wallet/currency/create  → CreateCurrency
  POST /admin/api/wallet/currency/update  → UpdateCurrency
  POST /admin/api/wallet/currency/status  → StatusCurrency
  POST /admin/api/wallet/currency/detail  → DetailCurrency
  POST /admin/api/wallet/rate/update      → UpdateExchangeRate
  POST /admin/api/wallet/rate/log/page    → PageExchangeRateLog

用户钱包管理：
  POST /admin/api/wallet/user/page             → QueryUserWalletPage
  POST /admin/api/wallet/user/detail           → QueryUserWalletDetail
  POST /admin/api/wallet/user/transaction/page → QueryUserTransactionPage
```

**每个 handler 只有一行代码**（工程惯例）：

```go
func GetWalletBalance(ctx context.Context, c *app.RequestContext) {
    rpc.Handler(ctx, c, rpc.WalletClient().GetWalletBalance)
}
```

## 十二、错误码体系思路

### 12.1 绕不过的事

错误码是前端展示和排查问题的基础，且工程有明确的编号规范。

### 12.2 错误码分配方案

```
前缀：6022（服务编号022）

按子模块分段：
  6022000 ~ 6022099  币种配置错误
  6022100 ~ 6022199  钱包余额错误
  6022200 ~ 6022299  充值错误
  6022300 ~ 6022399  提现错误
  6022400 ~ 6022499  兑换错误
  6022500 ~ 6022599  记录查询错误

举例（确定会用到的）：
  6022001  币种参数错误
  6022002  币种不存在
  6022003  币种已禁用
  6022004  币种代码重复
  6022101  余额不足
  6022102  冻结失败
  6022103  重复操作（幂等拦截）
  6022201  充值金额低于最低限
  6022202  重复充值拦截（3笔上限）
  6022203  通道匹配失败
  6022301  提现未通过KYC
```

## 十三、与财务模块的联调思路

### 13.1 绕不过的事

钱包和财务是整个系统中**唯一的双向依赖**，联调不可避免。但可以通过策略**最小化阻塞时间**。

### 13.2 联调策略

**阶段一：先各自独立开发（最大化并行）**

```
我这边：
  ✓ 币种配置全部功能（独立）
  ✓ 余额引擎（独立）
  ✓ 兑换全流程（独立）
  ✓ 记录查询（独立）
  ✓ 提现账户管理（独立）
  ✓ RPC提供接口（CreditBalance等5个，mock测试）
  ✓ 充值建单的本地逻辑（校验+建单，通道匹配先mock）
  ✓ 提现的冻结+建单逻辑（独立）

财务那边（并行）：
  ✓ 通道配置CRUD
  ✓ 支付配置CRUD
  ✓ 通道轮询匹配逻辑
  ✓ 充值记录管理
  ✓ 提现三级审核流程
  ✓ 人工修正流程
```

**阶段二：接口对齐（IDL先行）**

```
在各自开发的同时，先约定好这 8 个跨模块接口的 IDL 签名：

我提供给财务的（6个）：
  CreditBalance / FreezeBalance / UnfreezeBalance
  DebitBalance / ManualAdjust / GetUserBalance

财务提供给我的（2个）：
  GetPaymentMethods / MatchDepositChannel

签名定好后各自按签名实现，联调时只需要验证数据流通
```

**阶段三：联调打通（阻塞面可控）**

```
联调点1：充值方式展示
  我调财务 GetPaymentMethods → 验证返回数据 → 组装展示

联调点2：充值通道匹配
  我调财务 MatchDepositChannel → 验证通道匹配 → 完成建单

联调点3：充值回调入账
  财务调我 CreditBalance → 验证入账成功 → 余额变更正确

联调点4：提现审核回调
  财务调我 UnfreezeBalance / DebitBalance → 验证冻结/解冻/扣款正确

联调点5：人工修正
  财务调我 ManualAdjust → 验证加款/减款正确
```

**核心认知**：我的 35 个接口中，只有 3 个（C3充值方式、C5充值建单、C11提现方式）直接阻塞于财务接口。其余 32 个可完全独立开发测试。

## 十四、技术要点备忘

### 14.1 确定要用的公共组件

| 需求 | 用什么 | 说明 |
|------|-------|------|
| 分布式ID | `utils.GenerateSnowflakeID()` | 订单号生成 |
| DB连接 | `mysql.Init(opt)` + `query.SetDefault(db)` | 参考 ser-app/internal/cfg/ |
| 分页查询 | `query.FindByPage(offset, limit)` | GORM Gen 自带 |
| 缓存 | `rds.Set/Get/HSet/HGet` | 币种列表/汇率可缓存 |
| 获取用户ID | `tracer.GetUserId(ctx)` | C端接口从token自动注入 |
| 获取操作人 | `tracer.GetUserName(ctx)` | B端接口审计日志 |
| 错误返回 | `ret.BizErr(ctx, errCode, msg)` | 统一错误链路 |
| 审计日志 | `rpc.BLogClient().AddActLog()` | oneway调用 |
| KYC查询 | `rpc.KycClient().KycQuery()` | 提现校验 |

### 14.2 并发安全

```
余额操作：SELECT ... FOR UPDATE 行锁（同一事务内）
幂等控制：ref_order_id 唯一索引（DB层保证）
懒创建：可能需要 INSERT ... ON DUPLICATE KEY UPDATE 或分布式锁
充值重复检测：纯查询统计，不需要锁
```

### 14.3 缓存策略

```
可缓存的数据（变化频率低）：
  ├── 启用币种列表 — 变更时主动失效
  ├── 币种汇率 — TTL 缓存，与汇率更新周期对齐
  └── 充值档位/提现限额配置 — 变更时主动失效

不可缓存的数据（实时性要求高）：
  ├── 用户余额 — 每次必须读DB（资金数据不能有脏读）
  ├── 充值/提现订单状态 — 回调后状态变更，必须实时
  └── 今日已提现额度 — 每次操作都可能变
```

## 十六、实现优先级排序

按"确定性 × 阻塞影响"两个维度排序：

```
最高优先级（确定要做 + 阻塞下游）：
  ■ IDL定义（阻塞所有代码生成）
  ■ 基础设施注册（阻塞服务启动）
  ■ 表结构设计（阻塞所有数据操作）
  ■ 币种配置实现（阻塞所有业务）
  ■ 余额引擎（阻塞所有资金操作）
  ■ 给财务的RPC接口（阻塞联调）

高优先级（确定要做 + 独立可测）：
  ■ 兑换模块（自闭环，最早可验证）
  ■ 记录查询（纯读取，验证数据正确性）
  ■ 提现账户管理（简单CRUD）
  ■ 充值本地逻辑（校验+建单，通道mock）

中优先级（确定要做 + 依赖联调）：
  ■ 充值方式展示（等财务）
  ■ 充值通道匹配（等财务）
  ■ 提现提交审核（等财务对接方式确认）

后置（可后补）：
  ■ 奖金活动展示
  ■ B端管理查询+导出
  ■ 网关路由注册（可以在接口实现完之后统一注册）
```

## 十七、总结

### 确定性骨架（无论需求怎么变都要做的）

```
1. 一个 ser-wallet 服务，内含币种配置+钱包两个子模块
2. IDL 先行定义全部接口签名，尤其优先约定与财务的 8 个跨模块接口
3. 8 张核心表（wallet_account / transaction_log / deposit_order / withdraw_order /
   withdraw_account / exchange_record / currency / exchange_rate_log）
4. 余额引擎（getBalance + modifyBalance）：事务+幂等+流水，是心脏
5. 币种配置先行实现，是所有业务的地基
6. 给财务的 5 个 RPC 接口独立开发，不等财务
7. 兑换模块自闭环优先实现，作为端到端验证
8. 错误码体系 6022 前缀分段
9. 网关路由 C端 /api/wallet/ + B端 /admin/api/wallet/
```

### 弹性空间（细节待评审确认的）

```
- 奖金活动数据归属
- 兑换赠送比例来源
- 提现提交给财务的对接方式
- 充值回调的入口设计
- 冻结记录是独立表还是合并到流水表
- 充值档位/提现限额是独立表还是挂在币种表
```

### 一句话

**先把地基（币种配置）和引擎（余额引擎）做扎实，再按业务域逐个展开，联调最后集中打通。确定的先做，不确定的预留接口但不钉死实现。**
