# 第一部分：实现全局视角

## 1.1 要做什么

从工程维度看，ser-wallet 需要交付的东西：

```
一个 Kitex RPC 服务进程（ser-wallet）
  │
  ├─ 内含两个业务子模块（共享进程，进程内函数调用，零网络开销）
  │   ├─ 币种配置子模块（L0 基础底座）
  │   └─ 钱包核心子模块（L1 业务核心）
  │
  ├─ 对外暴露
  │   ├─ RPC 接口 → 供 gate-font、gate-back、ser-finance、游戏/直播/活动等服务调用
  │   └─ 不直接暴露 HTTP → 所有 HTTP 请求通过 gate-font / gate-back 路由转发
  │
  ├─ 依赖外部
  │   ├─ TiDB（独立数据库 wallet_db）
  │   ├─ Redis（缓存 + 分布式锁）
  │   ├─ ETCD（服务注册 + 配置读取）
  │   ├─ ser-finance（充值/提现业务协作，双向 RPC）
  │   ├─ ser-kyc / ser-user / ser-blog / ser-s3（单向调用）
  │   └─ 三方汇率 API（HTTP，定时拉取）
  │
  └─ 接口规模（评估）
      ├─ C 端接口：21 个（gate-font 路由进入）
      ├─ B 端接口：11 个（gate-back 路由进入）
      ├─ RPC 暴露：24 个（供其他服务调用）
      └─ RPC 调用：13 个（调用其他服务）
```

## 1.2 实现的核心矛盾

这个模块的核心难点不在于单个接口的复杂度，而在于：

| 矛盾点 | 说明 |
|--------|------|
| **钱包↔财务双向强耦合** | 12 个双向 RPC 接口，两边互为调用方和被调用方，不能等对方做完再开始 |
| **资金安全零容忍** | 任何一笔入账/扣款错误都是生产事故，幂等性、事务性、流水完整性缺一不可 |
| **多维度隔离** | 用户×币种×钱包类型 三维隔离，每个组合独立余额和流水 |
| **汇率实时性** | 充值/提现/兑换都依赖汇率，汇率来自外部三方 API，有不可用风险 |

## 1.3 实施阶段总览

```
Phase 0: 工程骨架 + 基础设施
         IDL 定义 → 代码生成 → 服务注册 → DB/Redis/ETCD 初始化
         ↓
Phase 1: 币种配置（L0 底座，无外部依赖）
         B 端 CRUD + RPC 暴露 + 汇率定时任务
         ↓
Phase 2: 钱包核心能力（RPC 暴露层）
         入账/扣款/冻结/解冻/余额查询
         ↓
Phase 3: 兑换模块（最独立，可单独测试）
         法币 → BSB 兑换，纯进程内事务
         ↓
Phase 4: 充值 + 提现（依赖财务服务联调）
         C 端全链路，需与财务开发者对齐 RPC 契约
         ↓
Phase 5: 扩展 RPC + 记录查询 + B 端查询
         投注/礼物/奖励/主播等 RPC + 四类记录
```

# 第二部分：工程骨架搭建思路

> 绕不开的第一步：没有骨架，后续所有业务逻辑无处落地

## 2.1 IDL 先行（Thrift 契约定义）

这是整个工程最不可跳过的一步。按项目规范，所有 RPC 接口先定义 Thrift IDL，再 gen_kitex.sh 生成代码。

**需要创建的 IDL 文件**：

```
common/idl/wallet.thrift        ← ser-wallet 对外暴露的所有 RPC 方法
```

**IDL 设计的粗粒度思路**：

```thrift
// 一个 service 包含所有方法（参照既有服务的做法，如 app.thrift）
service WalletService {

    // ===== 币种配置 B 端 =====
    CurrencyListResp       CurrencyList(1: CurrencyListReq req)
    CurrencyAddResp        CurrencyAdd(1: CurrencyAddReq req)
    CurrencyEditResp       CurrencyEdit(1: CurrencyEditReq req)
    CurrencyStatusResp     CurrencyStatus(1: CurrencyStatusReq req)
    BaseCurrencyGetResp    BaseCurrencyGet(1: BaseCurrencyGetReq req)
    BaseCurrencySetResp    BaseCurrencySet(1: BaseCurrencySetReq req)
    RateListResp           RateList(1: RateListReq req)
    RateLogListResp        RateLogList(1: RateLogListReq req)
    CurrencyOpLogResp      CurrencyOpLog(1: CurrencyOpLogReq req)

    // ===== 币种配置 RPC 暴露 =====
    GetCurrencyInfoResp    GetCurrencyInfo(1: GetCurrencyInfoReq req)
    GetEnabledListResp     GetEnabledCurrencyList(1: GetEnabledListReq req)
    GetExchangeRateResp    GetExchangeRate(1: GetExchangeRateReq req)
    GetBaseCurrencyResp    GetBaseCurrency(1: GetBaseCurrencyReq req)
    // ...

    // ===== 钱包核心 RPC 暴露 =====
    WalletCreditResp       WalletCredit(1: WalletCreditReq req)
    WalletDebitResp        WalletDebit(1: WalletDebitReq req)
    WalletFreezeResp       WalletFreeze(1: WalletFreezeReq req)
    WalletUnfreezeResp     WalletUnfreeze(1: WalletUnfreezeReq req)
    BetDebitResp           BetDebit(1: BetDebitReq req)
    BetCreditResp          BetCredit(1: BetCreditReq req)
    // ...

    // ===== 钱包 C 端 =====
    WalletBalanceResp      WalletBalance(1: WalletBalanceReq req)
    DepositMethodsResp     DepositMethods(1: DepositMethodsReq req)
    DepositCreateResp      DepositCreate(1: DepositCreateReq req)
    WithdrawCreateResp     WithdrawCreate(1: WithdrawCreateReq req)
    ExchangePreviewResp    ExchangePreview(1: ExchangePreviewReq req)
    ExchangeExecuteResp    ExchangeExecute(1: ExchangeExecuteReq req)
    RecordListResp         RecordList(1: RecordListReq req)
    // ...
}
```

**关键设计决策**：

| 决策项 | 方向 | 依据 |
|--------|------|------|
| 一个 service 还是多个 | 一个 `WalletService` | 参照 ser-app 等既有服务，单 service 单进程 |
| 请求/响应结构体 | 每个方法独立的 Req/Resp | 项目既有规范，便于独立扩展 |
| 金额字段类型 | `string` | 避免浮点精度问题，由业务层解析为 decimal 运算 |
| 币种字段 | `string currency_code` | 币种编码（如 "VND"、"BSB"），非 ID |
| 钱包类型字段 | `i32 wallet_type` | 枚举值（1 中心/2 奖励/3 主播/4 代理/5 场馆） |
| 业务类型字段 | `string biz_type` | 标识操作来源（deposit/withdraw/bet/gift/bonus 等） |
| 幂等键字段 | `string order_no` | 每笔资金操作的唯一订单号，幂等校验依据 |

**IDL 定义后的工程动作**：

```
① 编写 common/idl/wallet.thrift
② 执行 gen_kitex.sh wallet → 自动生成到 common/pkg/kitex_gen/wallet/
③ 创建 ser-wallet/main.go 引用生成的 service
④ 创建 ser-wallet/handler.go 实现所有方法（初始全部返回"未实现"）
```

# 第三部分：数据库设计思路

> 绕不开的核心：资金系统的数据模型决定了后续所有业务逻辑的实现路径

## 3.1 需要哪些表

粗粒度划分，至少需要以下核心表（具体字段等需求定型后细化）：

```
币种配置子模块:
  ├─ t_currency              ← 币种主表（编码/名称/类型/图标/浮动%/阈值%/状态）
  ├─ t_currency_rate         ← 汇率表（币种×汇率类型：实时/平台/入款/出款）
  ├─ t_currency_rate_log     ← 汇率变更日志（平台汇率因偏差超阈值自动更新时记录）
  └─ t_base_currency         ← 基准币种配置（全局唯一，设置后不可改）

钱包核心子模块:
  ├─ t_wallet                ← 钱包余额主表（user_id × currency × wallet_type → balance, frozen）
  ├─ t_wallet_transaction    ← 资金流水表（每笔余额变动的完整记录，资金对账的核心依据）
  ├─ t_deposit_order         ← 充值订单表（关联财务平台订单号）
  ├─ t_withdraw_order        ← 提现订单表（关联财务平台订单号）
  ├─ t_withdraw_account      ← 提现账户表（用户绑定的银行卡/电子钱包/USDT 地址）
  ├─ t_exchange_order        ← 兑换记录表（法币→BSB，含汇率快照/赠送金额/稽核要求）
  └─ t_reward_record         ← 奖励记录表（活动/任务/返奖入账记录）
```

## 3.2 核心表设计考量

### t_wallet — 钱包余额主表

这是整个系统最核心的表，每一行代表"一个用户在某个币种下某种钱包类型的余额"。

**唯一索引**：`(user_id, currency_code, wallet_type)` — 三维唯一

**必须有的字段**：
- `balance` — 可用余额（decimal，精度由币种类型决定）
- `frozen` — 冻结金额（提现审核期间冻结的金额）
- `version` — 乐观锁版本号（可选，配合 SELECT FOR UPDATE 使用）

**关键设计决策**：

| 问题 | 选项 | 建议方向 | 理由 |
|------|------|---------|------|
| 金额精度 | int64(分) vs decimal | **decimal(20,8)** | 加密货币需要 8 位小数（USDT），统一用 decimal 避免类型混用 |
| 余额是否可以为负 | 允许 vs 禁止 | **禁止** | 数据库层面加 CHECK 约束 `balance >= 0`，应用层也校验 |
| 冻结是否独立字段 | 独立 frozen vs 扣可用余额 | **独立 frozen 字段** | 可提现余额 = balance - frozen，提现驳回时 frozen 减少而非 balance 增加 |
| 软删除 | delete_at=0 | **遵循项目规范** | 工程认知确认：delete_at=0 为未删除，非零为删除时间戳 |

### t_wallet_transaction — 资金流水表

**核心设计原则**（需求文档 9："任何资金变动都要有流水记录"）：

- 每笔余额变动 **必须** 伴随一条流水记录，且在同一个 DB 事务中
- 流水表是对账的核心依据，永远只增不改不删（逻辑删除也不做）

**必须有的字段**：
- `user_id` + `currency_code` + `wallet_type` — 定位到具体哪个钱包
- `direction` — 收入(1) / 支出(2)
- `amount` — 变动金额（正数）
- `balance_before` / `balance_after` — 变动前后余额（对账关键）
- `biz_type` — 业务类型（deposit/withdraw/exchange/bet/gift/bonus/manual_credit/manual_debit...）
- `order_no` — 关联的业务订单号（幂等键）
- `remark` — 备注

**幂等索引**：`UNIQUE(biz_type, order_no)` — 同一业务类型+订单号只能有一条流水

### t_currency_rate — 汇率表

**设计要点**：
- 按币种存储多种汇率（实时/平台），入款/出款汇率是实时计算的（实时汇率 × 浮动比例），不需要落表
- 或者落表但标记 `rate_type`（实时=1, 平台=2），入款/出款在查询时实时计算
- BSB 的汇率是固定锚定的（10 BSB = 1 USDT），可以在初始化时写入，不需要三方同步

## 3.3 金额精度统一策略

这是资金系统绕不开的问题。需求文档明确了不同币种的展示精度：

| 币种类型 | 展示小数位 | 存储精度建议 |
|---------|----------|------------|
| 法币 VND/IDR | 0（整数） | decimal(20,2) 或 decimal(20,0) |
| 法币 THB | 2 | decimal(20,2) |
| 加密货币 USDT | 8 | decimal(20,8) |
| 平台币 BSB | 2 | decimal(20,2) |

**统一策略建议**：存储层统一用 `decimal(20,8)`，展示层按币种配置中的小数位数进行截断/四舍五入。这样：
- 数据库表结构统一，不因新增币种改表
- 汇率换算中间结果不丢精度
- 最终展示/入参由业务层按币种配置格式化

**Go 层面**：使用 `shopspring/decimal` 库进行所有金额运算，禁止使用 float64。

# 第四部分：资金安全实现思路

> 这是整个钱包系统最绕不开的核心关切。资金系统没有"差不多就行"，必须在设计阶段就把安全机制嵌入

## 4.1 幂等性（防重复操作）

**场景**：财务服务调用 WalletCredit 充值入账时，网络超时重试会导致同一笔入账被调用多次。如果不做幂等，用户余额会多加。

**实现策略**：

```
每个资金操作 RPC 接口（入账/扣款/冻结/解冻/人工加减款/投注/返奖/礼物/奖励）
都必须携带 order_no 参数

处理逻辑：
  ① 查询流水表：是否存在 (biz_type, order_no) 的记录
  ② 存在 → 直接返回成功（幂等，不重复执行）
  ③ 不存在 → 执行业务逻辑 + 插入流水记录（同一事务）
```

**数据库保底**：流水表 `UNIQUE(biz_type, order_no)` 唯一索引，即使应用层并发漏判，数据库层面也会拒绝重复插入。

## 4.2 事务性（原子操作）

**原则**：余额变更 + 流水记录 必须在同一个 DB 事务中，不允许出现"余额变了但流水没记"或"流水记了但余额没变"的情况。

**通用事务模板**：

```
所有资金操作的事务结构：

BEGIN TRANSACTION
  ① SELECT ... FROM t_wallet WHERE user_id=? AND currency=? AND wallet_type=? FOR UPDATE
     → 锁定行，防止并发修改
  ② 业务校验（余额充足？冻结金额有效？）
  ③ UPDATE t_wallet SET balance = balance +/- amount, frozen = frozen +/- amount
  ④ INSERT INTO t_wallet_transaction (...)
     → 记录变动前后余额、业务类型、订单号
COMMIT

任何一步失败 → ROLLBACK → 用户余额不变
```

## 4.3 并发控制（行级锁）

**场景**：用户同时发起提现和投注，两个请求并发修改同一个钱包余额。

**策略**：`SELECT ... FOR UPDATE` 悲观锁锁定目标钱包行。Kitex RPC 是并发处理请求的，必须通过行级锁保证同一个钱包的余额操作串行化。

**注意**：FOR UPDATE 的锁定范围要精确到 `(user_id, currency_code, wallet_type)` 这一行，不要锁整张表。

## 4.4 余额非负保护

**应用层**：扣款前先查余额，不足则拒绝
**数据库层**：建议在 TiDB 层面加 CHECK 约束 `balance >= 0 AND frozen >= 0`，作为最终兜底

**特殊情况 — 投注优先级扣款**：BetDebit 需要同时操作中心+奖励两个钱包，必须在同一事务中锁定两行再计算分配。

## 4.5 流水完整性

**不可跳过的规则**：
- 每一次余额变动，无论来源（RPC/内部/定时任务），都必须写流水
- 流水记录 `balance_before` 和 `balance_after` — 这是对账审计的核心数据
- 流水只增不改不删 — 即使是"修正"也通过新增一条反向流水实现

# 第五部分：币种配置实现思路

> L0 层基础底座，最先实现，不依赖任何其他业务模块

## 5.1 核心实现范围

```
B 端管理接口（9 个）:
  币种 CRUD + 状态切换 + 基准币种查询/设置 + 汇率展示 + 汇率日志 + 操作日志

RPC 暴露接口（6 个）:
  GetCurrencyInfo / GetEnabledCurrencyList / GetExchangeRate
  GetBaseCurrency / BatchGetExchangeRates / ValidateCurrency

内部能力（定时任务 + 进程内调用）:
  三方汇率定时拉取 + 平台汇率偏差更新 + 为钱包核心模块提供进程内查询
```

## 5.2 汇率系统实现思路

汇率是币种配置中最需要考虑的部分。

**汇率层次**：

```
三方 API（3 家）→ 取均值 → 实时汇率（DB 存储）
                                │
                    ┌───────────┼───────────┐
                    │           │           │
             入款汇率       出款汇率      平台汇率
          = 实时×(1+F%)   = 实时×(1-F%)  = 条件触发更新
          (实时计算)       (实时计算)     (DB 存储)
```

**定时任务实现要点**：

| 要点 | 说明 |
|------|------|
| 触发方式 | XXL-Job 定时任务 或 ser-cron 协调，建议 1~5 分钟频率 |
| API 容错 | 3 家 API 并行调用，任一失败降级为其余均值，全失败则跳过本轮并告警 |
| BSB 豁免 | BSB 锚定 USDT（10:1），不需要三方同步，初始化时写入固定值即可 |
| 平台汇率更新判断 | 偏差值 = \|实时 - 平台\| / 实时 × 100，偏差 ≥ 阈值时更新并写日志 |
| 缓存策略 | 实时汇率可 Redis 缓存（TTL = 同步频率），减少 DB 读压力 |

**汇率给谁用**：

| 场景 | 使用的汇率 | 获取方式 |
|------|----------|---------|
| C 端充值（跨币种） | 入款汇率 | 进程内调用 currency_ser.GetExchangeRate(code, "deposit") |
| C 端提现（跨币种） | 出款汇率 | 进程内调用 currency_ser.GetExchangeRate(code, "withdraw") |
| C 端兑换（法币→BSB） | 入款汇率 | 进程内调用 |
| 财务统计/报表 | 平台汇率 | Kitex RPC 调用 GetExchangeRate |
| 游戏场馆进出 | 入款汇率 + 快照 | Kitex RPC 调用 |

## 5.3 币种配置的缓存思路

币种信息变化频率低（管理员手动操作才变），但查询频率高（每个 C 端请求都可能查币种信息）。

**策略**：
- Redis 缓存启用状态的币种列表，TTL 适当（如 5 分钟）
- B 端修改币种信息后主动清除缓存（或发 NATS/Kafka 事件通知）
- 汇率信息缓存 TTL 短一些（= 定时同步频率）

# 第六部分：钱包核心能力实现思路

> 这一层是所有业务（充值/提现/兑换/投注/礼物/奖励）的底层原子能力

## 6.1 核心 Service 方法

`wallet_ser.go` 应提供以下核心方法，供上层所有业务 service 和 RPC handler 调用：

```
Credit(ctx, userId, currency, walletType, amount, bizType, orderNo)   → 入账
Debit(ctx, userId, currency, walletType, amount, bizType, orderNo)    → 扣款
Freeze(ctx, userId, currency, amount, orderNo)                         → 冻结
Unfreeze(ctx, userId, currency, amount, orderNo)                       → 解冻
DebitFrozen(ctx, userId, currency, amount, orderNo)                    → 从冻结中扣除
GetBalance(ctx, userId, currency)                                       → 查询余额
GetWalletInfo(ctx, userId)                                             → 查询全量钱包信息
```

**设计思路 — 通用方法 vs 专用方法**：

底层只有上述 7 个原子方法。上层的各种 RPC 接口（WalletCredit / ManualCredit / BonusCredit / AnchorCredit）本质上都是调用 `Credit()`，只是参数不同（walletType 和 bizType 不同）。

```
RPC R1  WalletCredit   → Credit(walletType=传入, bizType="deposit")
RPC R5  ManualCredit   → Credit(walletType=传入, bizType="manual_credit")
RPC R15 AnchorCredit   → Credit(walletType=主播, bizType="anchor_income")
RPC R16 BonusCredit    → Credit(walletType=奖励, bizType=传入)
```

**例外 — BetDebit（投注扣款）**：不能直接复用 `Debit()`，因为有"优先扣中心钱包、不足再扣奖励钱包"的专用逻辑，需要独立方法。

## 6.2 钱包初始化策略

**问题**：用户首次使用钱包时，DB 中还没有钱包记录。什么时候创建？

**选项对比**：

| 方案 | 优点 | 缺点 |
|------|------|------|
| A. 注册时创建所有币种所有钱包类型 | 逻辑简单 | 可能创建大量用户不会用的空钱包（如 IDR 用户不用 THB） |
| B. 首次操作时懒创建 | 按需创建，无浪费 | 每次操作都要先检查是否存在，代码复杂度稍高 |
| C. 首次查询余额时批量创建当前启用币种 | 折中方案 | 新增币种后老用户需要查询触发创建 |

**建议方向**：方案 B（懒创建），在 `Credit()` / `Debit()` / `GetBalance()` 中，如果目标钱包不存在则自动创建。这样：
- 不浪费存储
- 新增币种后，用户第一次涉及该币种时自动初始化
- 创建逻辑封装在底层，上层无感知

## 6.3 投注扣款的特殊处理

投注扣款是所有 RPC 中逻辑最特殊的一个，必须单独设计：

```
BetDebit(userId, currency, totalAmount, orderNo)

事务内：
  ① 锁定中心钱包 + 奖励钱包（同一事务，两次 FOR UPDATE）
  ② centerDebit = min(centerBalance, totalAmount)
  ③ remaining = totalAmount - centerDebit
  ④ bonusDebit = min(bonusBalance, remaining)
  ⑤ 校验 centerDebit + bonusDebit >= totalAmount
  ⑥ 中心钱包 -= centerDebit
  ⑦ 奖励钱包 -= bonusDebit
  ⑧ 记录两条流水
  ⑨ 计算并返回 {centerRatio, bonusRatio}
     → 调用方必须保存此比例，用于后续 BetCredit 按比例返奖
```

**调用方责任**：游戏服务拿到 ratio 后必须持久化，赛事结算时传回给 BetCredit。这是跨模块的数据契约。

# 第七部分：与财务模块协作实现思路

> 这是实施过程中最需要人与人沟通的部分，技术上用 Mock + 并行开发 + 联调的方式推进

## 7.1 双向 RPC 接口清单

```
钱包暴露 → 财务调用（7 个核心）:
  R1 WalletCredit     ← 充值成功入账 / 补单入账
  R2 WalletDebit      ← 提现出款成功后扣款
  R3 WalletFreeze     ← 提现发起时冻结
  R4 WalletUnfreeze   ← 提现驳回时解冻
  R5 ManualCredit     ← 人工加款
  R6 ManualDebit      ← 人工减款
  R8 GetWalletInfo    ← 人工修正时查询用户余额

财务暴露 → 钱包调用（5 个核心）:
  GetDepositMethods     ← 充值方式列表
  GetWithdrawMethods    ← 提现方式列表
  CreateDepositOrder    ← 创建充值订单
  CreateWithdrawOrder   ← 创建提现订单
  GetOrderStatus        ← 查询订单状态
```

## 7.2 协作开发策略

```
阶段 1: 对齐 IDL 契约
  │  双方约定 12 个 RPC 接口的 Request/Response 结构
  │  重点约定：金额字段类型(string)、订单号格式、错误码范围、钱包类型枚举值
  │
阶段 2: 各自实现 + Mock 对方（并行开发）
  │  钱包侧：
  │    · 先实现 RPC 暴露接口（R1-R8），这些不依赖财务，可独立测试
  │    · C 端接口中调用财务的部分，先写 Mock 函数返回假数据
  │  财务侧：
  │    · 先实现 RPC 暴露接口
  │    · 调用钱包的部分先写 Mock
  │
阶段 3: 联调对接
  │  充值全链路 → 提现全链路 → 人工修正链路
  │  逐个链路替换 Mock 为真实 RPC 调用
  │
阶段 4: 异常场景验证
     RPC 超时重试的幂等验证
     财务调钱包入账失败时的补偿策略
```

## 7.3 必须与财务开发者对齐的关键问题

这些问题在编码前必须有明确答案，否则联调阶段会出现不兼容：

| 编号 | 问题 | 影响 | 建议方向 |
|------|------|------|---------|
| Q1 | **提现冻结时序**：钱包先冻结再调财务建单，还是财务建单时回调钱包冻结？ | 决定 F11 和 R3 的调用顺序 | 建议钱包先冻结（保证余额安全），成功后再调财务建单 |
| Q2 | **幂等键 order_no 由谁生成**？格式约定？ | 所有 RPC 接口的幂等判断依据 | 建议由发起方（调用方）生成，钱包侧只校验不生成 |
| Q3 | **提现驳回通知**：同步 RPC 回调还是异步消息？ | 影响 R4 WalletUnfreeze 的触发时机 | 建议同步 RPC（简单可靠），异步消息作备选 |
| Q4 | **入账钱包类型由谁决定**？ | R1 WalletCredit 的 walletType 是调用方指定还是钱包推断 | 建议调用方显式指定（钱包不了解业务来源语义） |
| Q5 | **冻结金额包含手续费吗**？ | R3 WalletFreeze 的 amount = 提现金额 or 提现金额+手续费 | 建议包含（冻结全部需要扣除的金额） |
| Q6 | **错误码范围**：钱包 6022xxx，财务用哪个段？ | 避免错误码冲突 | 财务应分配独立编号（如 023 → 6023xxx） |
| Q7 | **充值奖金入账是财务触发还是钱包自行处理**？ | 决定奖金入账的调用链路 | 需评审确认，可能由活动服务监听充值事件触发 |
| Q8 | **RPC 调用失败的补偿策略**？ | 三方回调后钱包入账失败怎么办 | 财务侧重试 + 钱包侧幂等 + 补单兜底 |

# 第八部分：充值/提现实现思路

> 这两个模块是与财务模块耦合最深的部分，需要在财务 RPC 就绪后才能完整测试

## 8.1 充值实现思路

**钱包侧在充值链路中的角色**：

```
钱包负责的:
  · F3 获取充值方式列表 → 透传调用财务 RPC，钱包只做币种过滤
  · F4 发起充值 → 参数校验 + 重复转账检测(本地) + 调财务建单(RPC) + 记录本地订单
  · F5 查询充值状态 → 透传调用财务 RPC
  · F6 获取奖金活动 → 调用活动服务 RPC（或后续开发）
  · F7 重复转账检测 → 可内置在 F4 中，纯本地 DB 查询

钱包不负责的:
  · 通道轮询分配 → 财务负责
  · 三方支付对接 → 财务负责
  · 回调接收 → 财务负责，财务收到回调后调钱包 R1 入账

简单说：充值的"入口"在钱包，"执行"在财务，"入账"回到钱包
```

**重复转账检测逻辑**：

```
查询: 同 user_id + 同 currency_code + 同 amount，近 N 分钟内的充值订单数
  0 笔 → 安全，直接通过
  1-2 笔 → 返回警告标记（前端弹窗让用户确认）
  ≥3 笔 → 拦截，拒绝本次充值
```

## 8.2 提现实现思路

提现是整个模块中最复杂的链路，因为涉及多个校验步骤和冻结/解冻生命周期。

**钱包侧在提现链路中的角色**：

```
钱包负责的:
  · KYC 校验 → 调 ser-kyc 查询状态
  · 充值来源规则过滤 → 本地 DB 查用户历史充值方式，决定可用提现方式
  · 提现账户管理（CRUD）→ 本地 DB
  · 限额校验 → 本地配置 + DB 查今日已提现累计
  · 余额校验 + 冻结 → 本地 DB 事务
  · 调财务建单 → RPC
  · 暴露 R3/R4/R2 供财务回调 → 冻结/解冻/扣除

钱包不负责的:
  · 三级审核（风控/财务/出款）→ 财务后台 B 端功能
  · 通道出款 → 财务对接三方
```

**冻结/解冻/扣除的状态机**：

```
[可用余额] ──冻结(R3)──→ [已冻结] ──解冻(R4)──→ [可用余额]   ← 驳回路径
                                   ──扣除(R2)──→ [已扣除]     ← 出款成功路径
```

**"充值来源决定提现方式"规则实现**：

```
查询该用户+该币种的历史充值记录：
  · 只有法币充值 → 只返回法币提现方式
  · 只有 USDT 充值 → 只返回 USDT 提现方式
  · 混合充值（法币+USDT）→ 只返回法币提现方式
```

这个规则需要在 `t_deposit_order` 表中记录充值方式类型，提现时聚合查询。

# 第九部分：兑换模块实现思路

> 最独立的模块 — 不依赖财务、不依赖 KYC、不依赖任何外部 RPC，建议最先实现来验证基础设施

## 9.1 兑换的本质

兑换就是一次进程内的 **双币种原子转移**：法币钱包扣款 + BSB 钱包入账 + BSB 奖励钱包入账（赠送部分），全部在同一个 DB 事务中完成。

## 9.2 实现要点

```
F15 兑换预览:
  ① 校验：源币种必须是法币（VND/IDR/THB）
  ② 获取入款汇率（进程内调用币种配置）
  ③ 计算：兑换获得 BSB = 法币金额 / 入款汇率
  ④ 计算：赠送 BSB = 兑换获得 × 10%
  ⑤ 返回预览结果（不扣款）

F16 发起兑换:
  ① 重新获取入款汇率（防止预览后汇率已变）
  ② 重新计算兑换结果
  ③ 校验法币余额充足
  ④ DB 事务（6 步原子操作）:
     · 法币中心钱包 -= 兑换金额
     · 记录法币扣款流水
     · BSB 中心钱包 += 兑换获得
     · 记录 BSB 入账流水
     · BSB 奖励钱包 += 赠送金额
     · 记录 BSB 赠送流水
     · 创建兑换订单记录（含汇率快照）
     · 记录稽核流水要求（赠送部分需 X 倍流水）
  ⑤ 返回兑换结果
```

**为什么建议最先实现**：
- 不需要任何外部 RPC 依赖（财务、KYC、用户服务都不需要）
- 唯一依赖是币种配置（同进程内调用）
- 涉及双钱包事务操作，可以验证事务机制、流水记录、FOR UPDATE 等核心能力
- 开发完成后可以独立自测，不需要等任何人

# 第十一部分：缓存与性能思路

## 11.1 哪些数据需要缓存

| 数据 | 缓存策略 | TTL | 更新机制 |
|------|---------|-----|---------|
| 启用币种列表 | Redis 缓存完整列表 | 5 分钟 | B 端修改币种状态后主动清除 |
| 单个币种信息 | Redis Hash | 5 分钟 | B 端编辑后主动清除 |
| 实时汇率 | Redis 缓存 | = 同步频率（如 3 分钟） | 定时任务每轮更新 |
| 平台汇率 | Redis 缓存 | 较长（如 30 分钟） | 偏差触发更新时刷新 |
| 基准币种 | Redis 缓存 | 永久（设置后不变） | 设置时写入 |
| 用户钱包余额 | **不建议缓存** | — | 资金数据实时性要求高，读 DB |

**为什么余额不缓存**：资金类数据的一致性要求极高，缓存与 DB 之间的任何不一致都可能导致超扣或展示错误。查询余额的 QPS 在可控范围内（用户级别），DB 行级查询性能足够。

## 11.2 哪些场景需要分布式锁

正常情况下，DB 行级锁（FOR UPDATE）已足够保证并发安全。但以下场景可能需要 Redis 分布式锁作为前置控制：

| 场景 | 原因 | 锁 Key |
|------|------|--------|
| 基准币种设置 | 一次性操作，防并发双击 | `lock:base_currency:set` |
| 兑换执行 | 防用户快速双击产生两笔兑换 | `lock:exchange:{userId}` |
| 提现发起 | 防用户快速双击产生两笔提现 | `lock:withdraw:{userId}` |

锁的 TTL 设置为请求预期耗时的 2~3 倍（如 10 秒），防止死锁。

# 第十四部分：风险点与注意事项

## 14.1 技术风险

| 风险 | 说明 | 应对 |
|------|------|------|
| 三方汇率 API 不稳定 | 某家 API 突然不可用 | 3 家冗余 + 降级策略 + 沿用上次汇率 + 告警 |
| 财务 RPC 联调延迟 | 财务模块进度不确定 | 钱包侧先 Mock，核心能力（Phase 1-3）不受影响 |
| 高并发下余额扣款 | 同一用户大量并发请求 | FOR UPDATE 行级锁 + 幂等校验 |
| 浮点精度问题 | Go float64 运算丢精度 | 统一使用 decimal 库，禁止 float64 |
| 跨币种事务 | 兑换涉及两个币种钱包 | 同一 DB 实例内的事务保证原子性（两个钱包在同一张表不同行） |

## 14.2 业务风险

| 风险 | 说明 | 应对 |
|------|------|------|
| 需求评审后变更 | 当前是预评审，细节可能调整 | 代码层面模块化设计，service 层按业务划分，改动局部化 |
| 充值来源规则复杂化 | "法币充→法币提"规则可能有例外 | 提现来源判断逻辑独立为方法，方便后续调整 |
| 稽核流水（X 倍流水）规则不明确 | 奖励钱包转中心的条件具体值未定 | 设计为可配置参数，不硬编码 |
| 新增币种 | 上线后可能新增新的法币/加密货币 | 币种配置表驱动，新增币种只需 B 端添加记录，无需改代码 |

## 14.3 绝不能做的事

| 禁止项 | 原因 |
|--------|------|
| 余额使用 float64 | 浮点精度丢失，会导致对账差异 |
| 余额变更不写流水 | 无法对账，无法审计 |
| 余额变更和流水分两个事务 | 可能出现数据不一致 |
| 硬编码币种信息 | 新增币种时需要改代码重新部署 |
| 跳过幂等校验 | RPC 重试会导致重复入账/扣款 |
| 扣款不校验余额 | 可能出现负余额 |
| 信任前端传的 userId | userId 必须从 Token 中间件获取，不接受前端参数 |
| 日志打印完整银行卡号 | 违反数据安全规范 |

# 第十五部分：实现思路速查总结

| 维度 | 核心方向 |
|------|---------|
| **架构** | 单 Kitex 进程，内含币种配置+钱包核心两个子模块，进程内函数调用 |
| **开发顺序** | IDL→骨架→币种配置→钱包核心 RPC→兑换→充值/提现→扩展 RPC |
| **数据安全** | 事务+行级锁+幂等+流水完整+余额非负，五重保障 |
| **金额处理** | 统一 decimal(20,8) 存储，shopspring/decimal 运算，按币种格式化展示 |
| **汇率系统** | 3 家 API 均值→实时汇率→派生入款/出款汇率，平台汇率条件更新 |
| **与财务协作** | IDL 先对齐→Mock 并行→联调对接，12 个双向 RPC 是核心契约 |
| **缓存** | 币种信息和汇率 Redis 缓存，余额不缓存（实时性优先） |
| **最先可验证** | 兑换模块 — 零外部依赖，可独立跑通完整事务链路 |
| **最复杂接口** | F11 发起提现（7 步校验+冻结+调财务）、R10 BetDebit（优先级扣款） |
