| 业务 | 绕不过的原因 |
|------|------------|
| 钱包初始化 | 用户注册必须创建钱包，否则所有资金操作无载体 |

## 2. 服务架构决策

### 2.1 一个服务还是两个服务？

**核心问题**：钱包模块和币种配置模块是放在同一个 ser-wallet 中，还是拆成 ser-wallet + ser-currency？

**分析**：

| 维度 | 同一服务 (ser-wallet) | 拆分两个服务 |
|------|---------------------|-------------|
| 调用开销 | 币种查询为进程内函数调用，零网络开销 | 每次查汇率/币种都是 RPC 调用（3s超时） |
| 部署复杂度 | 一个服务，运维简单 | 两个服务，独立部署、独立扩容 |
| 数据库事务 | 兑换操作可在同一事务中操作钱包+读汇率 | 兑换需先 RPC 查汇率再本地事务 |
| 故障隔离 | 汇率定时任务异常可能影响钱包主流程 | 隔离性好，互不影响 |
| 团队协作 | 我一人负责两模块，无团队冲突 | 无必要 |
| 工程惯例 | 参照 ser-app 等服务，一个服务内可有多个子模块 | — |

**建议方向（待评审确认）**：**同一服务 ser-wallet**，币种配置作为内部子模块

- 理由：我一人负责两模块，进程内调用性能最优，兑换事务最简单
- 币种配置的 B 端接口通过 gate-back 暴露，C 端币种数据通过钱包接口间接返回
- 币种配置的 RPC 接口同样通过 ser-wallet 对外暴露（财务调 GetExchangeRate 走 ser-wallet）
- 汇率定时任务在 ser-wallet 内部 goroutine 或由 ser-cron 调度（待确认）

## 4. 数据模型设计方向

| # | 表名(方向) | 归属子模块 | 核心职责 | 关键字段方向 |

### 4.2 关键设计考量

#### 4.2.1 wallet 表的主键设计

**方案方向**：`(user_id, currency_code, wallet_type)` 作为唯一联合索引

```
每个用户 × 每个启用币种 × 每个钱包类型 = 一行记录

示例：用户 10001 的记录：
user_id | currency_code | wallet_type | balance    | frozen_amount
10001   | BSB           | CENTER      | 1500.00    | 200.00
10001   | BSB           | REWARD      | 300.00     | 0.00
10001   | VND           | CENTER      | 5000000    | 0
10001   | VND           | REWARD      | 100000     | 0
10001   | IDR           | CENTER      | 0          | 0
10001   | IDR           | REWARD      | 0          | 0
...
```

**绕不过的考量**：
- **余额字段精度**：用 DECIMAL(20,8) 覆盖法币整数到加密货币8位小数的范围
- **乐观锁或行锁**：余额更新必须防并发超扣（TiDB 场景下 `SELECT ... FOR UPDATE` 或 version 字段）
- **frozen_amount 字段**：提现冻结金额，可用余额 = balance - frozen_amount

#### 4.2.2 wallet_flow 表的设计

**核心原则**：流水表是**只追加（append-only）**，永不更新永不删除

```
每次 CreditWallet/DebitWallet/FreezeBalance 等操作都写一条流水

关键字段：
- direction: IN/OUT/FREEZE/UNFREEZE
- biz_type: 充值/提现/兑换/人工加款/人工减款/活动奖金/...
- before_balance / after_balance: 变动前后余额（审计必须）
- ref_order_no: 关联的业务订单号（充值订单号/提现订单号/...）
- idempotent_key: 幂等键（防重复写入）
```

**依据**：需求 9 非功能性需求 — "账务流水与余额必须强一致"

#### 4.2.3 freeze_detail 表的必要性

**为什么需要独立的冻结明细表**：

提现冻结时，可能涉及两个钱包（先冻中心，不足部分冻奖励）。解冻/确认扣款时需要知道**每个钱包冻了多少**才能精确还原。

```
示例：提现 1500 BSB
  - 中心钱包有 1200 → 冻结 1200
  - 奖励钱包有 500 → 冻结 300

freeze_detail 记录：
order_no | wallet_type | freeze_amount | status
W001     | CENTER      | 1200          | frozen
W001     | REWARD      | 300           | frozen

解冻时按此明细精确解冻各钱包
确认扣款时按此明细精确扣减各钱包
```

#### 4.2.4 金额精度统一策略

**绕不过的问题**：不同币种小数位不同

| 币种类型 | 展示精度 | 存储精度 | 依据 |
|----------|---------|---------|------|
| VND/IDR（法币整数） | 0位小数 | DECIMAL(20,8)，实际存整数 | 需求 4-1 金额展示规则 |
| THB/BSB | 2位小数 | DECIMAL(20,8) | 需求 4-1 金额展示规则 |
| USDT（加密货币） | 8位小数 | DECIMAL(20,8) | 需求 3-3 小数位数 |

**策略方向**：
- 数据库统一用 DECIMAL(20,8) 存储所有金额
- 展示精度由币种配置表的 `decimal_digits` 字段决定
- 所有金额计算使用 `decimal` 库（Go 的 `shopspring/decimal`），避免浮点误差
- **千分位格式化在返回 JSON 时由后端处理**（或前端处理，待与前端确认）
- 依据：需求 4-1 金额展示规则 — VND/IDR `.` 作为千分位，THB/BSB `,` 作为千分位

## 5. 资金安全体系设计

> 这是整个钱包模块**最核心、最不可妥协**的部分

### 5.1 事务一致性

**原则**：余额变更 + 流水写入 + 状态更新 = 必须在同一数据库事务中

```
// 伪代码示意
func CreditWallet(ctx, req) {
    tx := db.Begin()
    defer tx.Rollback()

    // 1. 查钱包（加行锁）
    wallet := tx.SelectForUpdate(userId, currency, walletType)

    // 2. 更新余额
    wallet.Balance += amount
    tx.Save(wallet)

    // 3. 写流水
    flow := WalletFlow{
        BeforeBalance: wallet.Balance - amount,
        AfterBalance:  wallet.Balance,
        ...
    }
    tx.Create(flow)

    tx.Commit()
}
```

**关键要点**：
- 使用 `SELECT ... FOR UPDATE` 加行锁，防止并发更新余额导致数据不一致
- 余额和流水在同一事务中，要么同时成功要么同时失败
- TiDB 兼容 MySQL 事务，支持行级锁

**依据**：需求 9 非功能性需求 — "余额与流水强一致"

### 5.2 幂等性设计

**绕不过的幂等场景**：

| 操作 | 幂等键 | 幂等策略 |
|------|--------|---------|
| CreditWallet（RPC） | ref_order_no + biz_type | 查 wallet_flow 是否已存在该 (ref_order_no, biz_type) 记录，已存在则直接返回成功 |
| DebitWallet（RPC） | ref_order_no + biz_type | 同上 |
| FreezeBalance（RPC） | ref_order_no | 查 freeze_detail 是否已存在该 order_no 的冻结记录 |
| UnfreezeBalance（RPC） | ref_order_no | 查 freeze_detail 状态，已解冻则直接返回成功 |
| ConfirmDebit（RPC） | ref_order_no | 查 freeze_detail 状态，已确认则直接返回成功 |
| 充值回调 | 平台订单号 | 查充值订单状态，非"待支付"则直接返回成功 |

**设计方向**：
- **所有 RPC 资金操作接口必须幂等** — 这是分布式系统的基本要求
- 幂等检查在事务外执行（先查后做），减少锁持有时间
- 幂等键必须在请求参数中由调用方传入，不可由被调用方生成

### 5.3 冻结-确认/解冻 三阶段模型

```
提现流程的资金状态机：

可用余额 ──[创建提现订单]──▶ 冻结状态 ──┬──[出款成功]──▶ 已扣除
                                       │
                                       └──[驳回/失败]──▶ 已解冻（余额恢复）

操作：
FreezeBalance:   balance 不变，frozen_amount += x，可用余额(balance-frozen_amount)减少
ConfirmDebit:    balance -= x，frozen_amount -= x，真正扣减
UnfreezeBalance: frozen_amount -= x，可用余额恢复
```

**为什么不能直接扣款**：
- 提现从创建到出款成功，中间可能经历风控审核、财务审核、三方出款，时间跨度长
- 冻结期间余额"占用但不扣减"，用户可看到冻结金额但不能再消费
- 审核驳回时直接解冻恢复，无需"加回"操作
- 依据：流程思路/d-2.md 第4章提现流程

### 5.4 消费扣款优先级

```
扣款/冻结时的优先级处理：

需要扣/冻 X 金额：
  ① 先查中心钱包可用余额 A
  ② 若 A >= X → 全额扣/冻中心钱包
  ③ 若 A < X → 扣/冻中心钱包 A，剩余 (X-A) 扣/冻奖励钱包
  ④ 若中心+奖励仍不足 → 拒绝操作
```

**影响范围**：提现冻结、游戏投注扣款（未来）、兑换扣款
**依据**：需求 2-1 资金钱包体系 — "优先扣中心钱包"

## 6. RPC 接口契约设计

### 6.1 资金操作原语的接口设计方向

这些接口是**被财务等外部服务调用的**，契约设计需要双方协商。

#### CreditWallet（上账/加款）

```
请求方向：
  - user_id: int64         // 用户ID
  - currency_code: string  // 币种代码（BSB/VND/IDR/THB）
  - wallet_type: i32       // 钱包类型枚举（CENTER/REWARD/...）
  - amount: string         // 金额（用字符串传递避免精度丢失）
  - biz_type: i32          // 业务类型枚举（充值/补单/人工加款/...）
  - ref_order_no: string   // 关联业务订单号（幂等键的一部分）
  - remark: string         // 备注（可选）

响应方向：
  - code: i32              // 0=成功
  - msg: string
  - balance: string        // 操作后余额（可选返回）
```

**要点**：
- 金额用 string 传递，不用 double/float，避免 Thrift 序列化/反序列化精度丢失
- ref_order_no 是幂等的核心依据，调用方必须保证唯一
- 这个接口的调用方：财务（充值成功/补单/人工加款）

#### FreezeBalance / UnfreezeBalance / ConfirmDebit

```
请求方向（三者结构相似）：
  - user_id: int64
  - currency_code: string
  - amount: string
  - ref_order_no: string   // 提现订单号，三个操作用同一个订单号关联

响应方向：
  - code: i32
  - msg: string
```

**要点**：
- 三个操作通过同一个 ref_order_no 关联到同一笔提现
- Freeze 时记录冻结明细，Unfreeze/ConfirmDebit 时按明细操作
- 调用方：财务（提现创建/驳回/出款成功/出款失败）

### 6.2 币种查询接口的设计方向

#### GetExchangeRate

```
请求方向：
  - currency_code: string  // 币种代码
  - rate_type: i32         // 汇率类型（实时/入款/出款）

响应方向：
  - code: i32
  - rate: string           // 汇率值（字符串精度）
  - base_currency: string  // 基准币种代码
```

**要点**：
- 这是高频接口，被钱包充值/提现/兑换 + 财务统计报表调用
- 应优先从 Redis 缓存读取，缓存未命中时读 DB
- 汇率值用 string 传递保精度

### 6.3 与财务的 RPC 契约（我方作为调用方）

> 以下接口需要与财务开发者协商确认

| 我方调用 | 用途 | 关键入参 | 关键出参 |
|----------|------|---------|---------|
| GetRechargeMethodList | 获取充值方式配置 | currency_code | 方式列表(名称/图标/档位/范围) |
| GetWithdrawMethodList | 获取提现方式配置 | currency_code | 方式列表(名称/图标/费率) |
| MatchChannel | 充值匹配代收通道 | currency_code, method, amount | 通道信息 + 三方支付URL |
| SubmitPayout | 提现提交出款 | order_no, amount, account_info | 出款结果 |

**协商要点**：
- 接口命名、字段名需要双方统一
- 错误码范围需要约定（财务的错误码 vs 钱包的错误码）
- 超时和重试策略需要约定
- MatchChannel 失败时的降级策略

## 7. C 端接口实现思路

### 7.1 钱包信息 + 币种切换

**实现思路**：
- 进入钱包页面时一次性返回：当前币种余额 + 可用币种列表
- 币种列表从币种配置模块获取（进程内调用），做本地缓存（TTL 1h）
- 余额查询走 Redis 缓存优先，miss 时查 DB 写缓存
- 切换币种只更新用户偏好（Redis 存 `wallet:user:{uid}:currency` = "BSB"），前端重新请求刷新数据
- **不存在"当前币种"的服务端持久化需求**，可以放 Redis 或依赖前端每次请求时传入

**依据**：流程思路/d-2.md 第7章

### 7.2 充值流程

**实现思路**：
- 分两个接口：获取充值方式列表（展示用）+ 创建充值订单（提交用）
- 充值方式列表调财务 GetRechargeMethodList，按游客/登录状态过滤后返回
- 创建充值订单：重复检查 → 汇率查询(仅BSB) → 匹配通道(调财务) → 创建订单 → 返回支付信息
- **充值回调**是关键节点：验签 → 幂等 → 事务(上账+流水+状态更新) → 审计日志

**关键不确定项**：
- 充值回调的接收方是钱包还是财务？→ **待评审确认**
- 活动奖金的配置来源？活动奖金列表是钱包本地管理还是调活动服务？→ **待评审确认**

**依据**：流程思路/d-2.md 第3章

### 7.3 提现流程

**实现思路**：
- 分阶段职责清晰：我方负责到"创建订单+冻结"为止，后续审核和出款由财务接管
- 展示阶段：并行调 ser-kyc(KYC状态) + 财务(提现方式列表)，按 KYC 过滤后返回
- 提交阶段：多重校验链 → 冻结余额（同一事务） → 创建订单（状态=待风控审核）
- 审核结果由财务通过 RPC 回调我方：ConfirmDebit(成功) / UnfreezeBalance(失败)

**关键设计**：
- 冻结和创建订单必须在同一事务中（避免冻结成功但订单创建失败的不一致）
- 提现限制（日次数/日限额/单笔限额）的配置存放位置待确认（本地还是 ETCD）

**依据**：流程思路/d-2.md 第4章

### 7.4 兑换流程

**实现思路**：
- 完全自主，不依赖财务，唯一外部依赖是汇率查询（进程内调用）
- 事务操作：DebitWallet(法币) + CreditWallet(BSB-中心) + CreditWallet(BSB-奖励,如有赠送)
- 兑换提交时**重新获取汇率**（防止页面停留期间汇率变动）
- 仅法币→BSB单向，BSB不可反向兑换

**关键设计**：
- 跨币种操作在同一事务中（同一个 TiDB 实例的不同行）
- 赠送比例的配置来源待确认（ETCD 配置 / 数据库表 / 活动模块）

**依据**：流程思路/d-2.md 第5章

### 7.5 记录查询

**实现思路**：
- 四种记录类型（充值/提现/兑换/奖励）共用一个列表接口，通过 tab 参数分发查不同的表
- 币种维度过滤是**强制条件**（当前选择的币种 = 记录的币种）
- 纯读取操作，完全独立，无外部依赖
- 列表分页 + 详情两个接口

**关键设计**：
- 查询走读库（如果有读写分离）
- 大量历史数据场景下的分页性能：cursor-based 分页优于 offset 分页（待评估）

**依据**：流程思路/d-2.md 第6章

### 7.6 提现账户管理

**实现思路**：
- 完全独立的 CRUD，最简单的子模块
- 保存/列表/删除三个接口
- 账户信息用 JSON 存储（不同类型的账户字段不同：银行卡有卡号/银行名/持有人，电子钱包有钱包类型/账号）
- **持有人姓名与 KYC 一致性校验**：保存时调 ser-kyc 获取认证姓名比对

**依据**：流程思路/d-2.md 子模块F

## 10. 缓存与性能策略

### 10.1 缓存层次规划

| 缓存对象 | Redis Key 方向 | TTL | 更新时机 | 依据 |
|----------|---------------|-----|---------|------|
| 用户钱包余额 | `wallet:bal:{uid}:{currency}` | 5min | CreditWallet/DebitWallet/Freeze/Unfreeze 时清除 | 高频读，写时失效 |
| 币种列表 | `wallet:currencies` | 1h | 新增/编辑/启用/禁用时清除 | 极少变，读频繁 |
| 汇率数据 | `wallet:rate:{currency}` | 1-5min | 定时任务更新时刷新 | 定时拉取自然更新 |
| 币种详情 | `wallet:currency:{code}` | 1h | 编辑时清除 | 同币种列表 |
| 基准币种 | `wallet:base_currency` | 永久 | 一次性设置 | 不可更改 |
| 用户当前币种 | `wallet:user:{uid}:currency` | 7d | 切换时更新 | 用户偏好 |

**Redis Key 命名规范**：遵循工程认知/d-2.md 第16章 Redis 键规范
**注册位置**：`common/pkg/consts/redis_key/` 增加 wallet 模块的键定义

### 10.2 缓存一致性策略

**原则**：写操作后**删除缓存**（不是更新缓存），下次读时重建

```
CreditWallet:
  1. 事务内更新余额
  2. 事务提交后删除缓存 wallet:bal:{uid}:{currency}
  3. 下次 GetBalance 缓存 miss 时从 DB 重建

为什么删除而不是更新：
  - 避免事务回滚但缓存已更新的不一致
  - 避免并发写时缓存覆盖问题
```

### 10.3 热点数据优化

- **币种列表**：全局共享数据，用单个 Redis key 存储，所有用户共用
- **汇率数据**：定时任务主动写入 Redis，查询时直接读取（写优先于读）
- **余额查询**：高频读，短 TTL 缓存 + 写时失效，在资金安全前提下的最优平衡

## 11. 定时任务设计

### 11.1 汇率定时拉取

**实现方向**：

```
触发方式：
  方案A：ser-wallet 内部 goroutine + time.Ticker（分钟级）
  方案B：ser-cron 定时调用 ser-wallet 的 RPC 接口
  → 待确认采用哪种，建议方案A（简单、无外部依赖）

处理逻辑：
  1. 查所有启用且需要汇率更新的币种
  2. 对每个币种：
     a. 并发调 3 家三方汇率 API（HTTP GET）
     b. 排除异常值，取有效值均值 → 新实时汇率
     c. 计算偏差值 = |新实时汇率 - 当前平台汇率| / 新实时汇率 × 100%
     d. 偏差 < 阈值 → 仅更新实时汇率字段
     e. 偏差 ≥ 阈值 → 更新平台汇率 + 重算入款/出款汇率 + 写汇率日志
  3. 更新 Redis 缓存

容错：
  - 某家 API 超时/失败 → 用剩余的取均值（至少1家可用就更新）
  - 全部失败 → 记录告警日志，保持旧汇率不变
  - 汇率异常波动 → 可选：设置涨跌幅上限，超出则告警不更新
```

**依据**：流程思路/d-2.md 第9章

### 11.2 三方汇率 API 对接

**实现方向**：
- 定义统一的汇率 API 接口（interface），3 家 API 各实现一个 adapter
- 每次拉取并发调用 3 家，设 3-5 秒超时
- 结果取均值时排除明显异常值（如偏离中位数超过 X%）
- 三方 API 的地址/密钥放 ETCD 配置

## 12. 错误码分配规划

### 12.1 错误码格式

根据工程认知/d-2.md 第15章：错误码格式 `6{NNN}{MMM}`
- `6` = 前缀
- `NNN` = 服务编号（ser-wallet = 022）
- `MMM` = 模块内序号

**ser-wallet 错误码范围**：`6022001` ~ `6022999`

### 12.2 错误码分段方向

| 段 | 范围 | 归属子模块 | 典型错误 |
|----|------|-----------|---------|
| 001-099 | 6022001-6022099 | 钱包核心（子模块A） | 余额不足、钱包不存在、幂等重复 |
| 100-199 | 6022100-6022199 | 充值（子模块B） | 重复转账拦截、通道匹配失败、回调验签失败 |

## 13. 分阶段实施路径

### 13.1 总体路径

```
┌─────────────────────────────────────────────────────────┐
│  第零阶段：工程骨架搭建                                    │
│  IDL定义 → 代码生成 → 目录创建 → 表结构创建 → 配置注册       │
│  → 产出：可编译运行的空服务壳子                             │
└────────────────────────┬────────────────────────────────┘
                         ▼
┌─────────────────────────────────────────────────────────┐
│  第一阶段：核心基座（无外部依赖，可独立自测）                  │
│  钱包核心（7个RPC接口）+ 币种配置（11个B端+RPC接口）           │
│  + 提现账户（3个C端接口）                                    │
│  → 产出：资金操作原语可用，币种配置可管理                      │
└────────────────────────┬────────────────────────────────┘
                         ▼
┌─────────────────────────────────────────────────────────┐
│  第二阶段：自主业务（依赖第一阶段，不依赖财务）                │
│  兑换（2个C端接口）+ 记录查询（2个C端接口）+ 汇率定时任务       │
│  → 产出：兑换可用，记录可查，汇率自动更新                      │
└────────────────────────┬────────────────────────────────┘
                         ▼
┌─────────────────────────────────────────────────────────┐
│  第三阶段：跨模块业务（需要与财务联调）                       │
│  充值（5个C端接口）+ 提现（4个C端接口）                       │
│  → 产出：完整的充值提现链路                                  │
└─────────────────────────────────────────────────────────┘
```

### 13.2 第零阶段详细步骤

```
0.1  在 common/pkg/consts/namesp/ 注册 ser-wallet (022, wallet_service)
0.2  编写 Thrift IDL（common/idl/ser-wallet/）
0.3  执行 gen_kitex.sh 生成 RPC 代码
0.4  创建 ser-wallet/ 目录结构
0.5  在 TiDB 中创建数据库和数据表（DDL）
0.6  编写 gorm_gen.go 并执行，生成 model/query/rep
0.7  在 ETCD 中注册服务配置（DB连接、Redis、端口）
0.8  编写 main.go + handler.go 基础骨架
0.9  在 common/rpc/ 增加 WalletClient() 工厂函数
0.10 在 gate-font 和 gate-back 注册路由
0.11 在 go.work 注册 ser-wallet
0.12 编译运行，验证服务启动 + ETCD 注册 + 健康检查
```

### 13.3 第一阶段详细方向

**A. 钱包核心（优先级最高）**：
- InitWallet → CreditWallet → DebitWallet → GetBalance → FreezeBalance → UnfreezeBalance → ConfirmDebit
- 每实现一个接口就编写对应的自测脚本
- 重点验证：事务一致性、幂等性、消费优先级

**B. 币种配置**：
- SetBaseCurrency → AddCurrency → CurrencyPage → EditCurrency → ToggleCurrency → GetCurrencyList(RPC) → GetExchangeRate(RPC) → GetCurrencyDetail(RPC) → RateLogPage → GetBaseCurrency(RPC)
- 先手动插入测试数据（BSB/VND/IDR/THB），验证 CRUD
- 重点验证：汇率浮动计算、偏差检测逻辑

**C. 提现账户**：
- SaveWithdrawAccount → GetWithdrawAccounts → DeleteWithdrawAccount
- 简单 CRUD，最快完成

### 13.4 第二阶段详细方向

**D. 兑换**：
- GetExchangeInfo → DoExchange
- 重点验证：跨币种事务原子性、赠送奖金进奖励钱包
- 自测方案：给测试用户加 VND 余额 → 执行兑换 → 验证 VND 减少 + BSB 增加

**E. 记录查询**：
- GetRecordList → GetRecordDetail
- 前置条件：充值/提现/兑换表有数据（可由前面步骤产生）

**F. 汇率定时任务**：
- 对接三方汇率 API → 偏差检测 → 自动更新
- 需要三方 API 的地址和密钥（待确认）

### 13.5 第三阶段详细方向

**B. 充值**：
- GetRechargeMethods → CreateRechargeOrder → RechargeCallback → CheckDuplicateRecharge → GetUsdtAddress
- 强依赖财务的通道匹配接口，需要联调
- 重复转账拦截和 USDT 地址可先独立实现

**C. 提现**：
- GetWithdrawMethods → CreateWithdrawOrder
- 强依赖财务的提现方式配置 + KYC 认证
- 冻结逻辑已在第一阶段实现，这里主要是展示+校验+创建订单

## 附录：实现思路一句话总结

```

安全第一 → 余额+流水同事务、所有RPC幂等、冻结三阶段、消费优先级
方向确定 → 绕不过的都列清楚了，细节等需求定型后再精确定义
弹性空间 → 汇率参数、限额配置、奖金来源等细节留了待确认项，不把设计过死
```
