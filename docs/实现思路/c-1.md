## 一、实现前的总体认知

### 1.1 我们要做什么

在既有微服务工程体系下，新增 `ser-wallet`（编号 022），承载两个功能域：

| 功能域 | 面向端 | 核心职责 |
|--------|--------|---------|
| **钱包模块** | C端为主 | 用户余额管理、充值、提现、兑换、记录查询；对外暴露余额变更RPC供财务调用 |
| **币种配置** | B端 | 币种参数管理、汇率机制、汇率日志；对外暴露汇率/币种查询RPC供钱包和财务调用 |

两个功能域合并在一个服务内（共享IDL、共享数据库），理由：
- 币种配置是钱包的基础数据层，钱包几乎每个接口都要查汇率/币种
- 合并后汇率查询变成本地方法调用而非跨进程RPC，减少延迟和故障点
- 业务规模合理——预估 IDL 约 43 个方法，参照 UserService(46)、LiveService(50) 均在合理区间
- 评审后如果决定拆分为独立服务，IDL 结构已按功能域分文件组织，拆分成本低

### 1.2 绕不过的工程约束

以下来自对既有工程的一手分析，是必须遵守的规范：

| 约束 | 来源 | 说明 |
|------|------|------|
| IDL 先行 | 工程惯例 | 所有接口先在 Thrift IDL 定义，再 gen_kitex.sh 生成代码 |
| 三层分层 | ser-app 等参照 | handler.go → internal/ser/ → internal/rep/ → internal/gorm_gen/ |
| handler 纯委托 | 工程规范 | handler.go 每个方法一行委托到 ser 层，不写业务逻辑 |
| 网关一行转发 | rpc.Handler 模式 | gate-font/gate-back 的 handler 用 `rpc.Handler(ctx, c, rpc.XxxClient().Method)` |
| ETCD 驱动配置 | 工程规范 | 端口、DB名、Redis 等配置全从 ETCD 获取 |
| 金额用 string | IDL 规范 | 避免浮点精度丢失，Thrift 中金额字段统一 `string` |
| 时间用 i64 | IDL 规范 | 毫秒时间戳，不用字符串日期 |
| 错误码 6022xxx | 错误码规范 | 前缀 `6` + 服务编号 `022` + 3位模块序号 |
| 复用已有轮子 | 工程禁令 | common/pkg 下已有的工具（雪花ID、分页、日志、Redis、S3等）不得重复造 |
| 代码生成 | 工程规范 | Model/Query/Repo 由 gorm/gen 生成，手写 ser 和 rep 层 |

## 二、基础设施搭建（绕不过的第一步）

在写任何业务代码之前，必须先把服务的"骨架"立起来。以下每一项都是后续开发的前置依赖。

### 2.1 ETCD 命名空间注册

**文件**：`common/pkg/consts/namesp/namesp.go`

需要新增的常量（参照已有服务的命名模式）：

```
EtcdWalletService = "wallet_service"           // 服务注册名
EtcdWalletPort    = "/slg/serv/wallet/port"    // 服务端口
EtcdWalletDb      = "/slg/serv/wallet/db"      // 数据库名
```

同时需要在 ETCD 中实际写入这些 Key 的值（端口号、数据库名称），否则服务启动时 `etcd.DirectGet` 拿不到配置。

### 2.2 RPC 客户端注册

**文件**：`common/rpc/rpc_client.go`

新增 `WalletClient()` 函数，遵循既有的 sync.Once 懒加载单例模式。这一步完成后，财务模块和其他服务就可以通过 `rpc.WalletClient().XxxMethod()` 调用钱包的 RPC 接口。

在 use 列表中添加 `./ser-wallet`。否则 workspace 内的其他模块（gate-font、gate-back、common 等）无法 import ser-wallet 的代码。

### 2.4 gorm_gen.go 修正

**文件**：`ser-wallet/internal/gen/gorm_gen.go`

当前错误指向：`namesp.EtcdAppDb` 和 `"ser-app"`

### 2.5 go.mod 初始化

ser-wallet 目录下当前没有 go.mod，需要初始化（module 名参照其他服务的命名风格）。

## 三、IDL 设计思路

### 3.1 文件组织

按功能域拆分 IDL 文件，每个文件职责清晰：

```
common/idl/ser-wallet/
├── service.thrift             ← 服务入口：include 所有子文件，定义 WalletService
├── wallet.thrift              ← 钱包核心：余额查询、钱包概览的 Req/Resp
├── deposit.thrift             ← 充值：充值方式、创建订单、USDT支付页等 Req/Resp
├── withdrawal.thrift          ← 提现：提现方式、创建订单、账户管理等 Req/Resp
├── exchange.thrift            ← 兑换：汇率查询、执行兑换的 Req/Resp
├── record.thrift              ← 记录：充值/提现/兑换/奖励记录的 Req/Resp
├── currency.thrift            ← 币种配置B端：币种列表/编辑/基准设置/汇率日志 Req/Resp
└── wallet_rpc.thrift          ← RPC对外：供财务调用的余额变更方法 Req/Resp
```

### 3.2 service.thrift 方法分组

service.thrift 中的方法按调用方分组，方便理解和维护：

```
service WalletService {

    // ========== 钱包 C端 ==========
    // 余额 & 币种
    GetCurrencyListWithBalance / GetBalance / GetWalletOverview

    // 充值
    GetDepositMethods / CreateDepositOrder / GetUsdtPaymentInfo / CheckDuplicateDeposit

    // 提现
    GetWithdrawalMethods / CreateWithdrawalOrder /
    SaveWithdrawalAccount / GetWithdrawalAccount / GetWithdrawalLimits

    // 兑换
    GetExchangeRateForSwap / ExecuteExchange

    // 记录
    GetDepositRecords / GetWithdrawalRecords / GetExchangeRecords / GetRewardRecords
    GetDepositOrderDetail / GetWithdrawalOrderDetail

    // 奖金活动（数据源待评审）
    GetBonusActivities

    // ========== 币种配置 B端 ==========
    PageCurrencyConfig / EditCurrencyConfig / SetBaseCurrency
    PageRateChangeLog / RefreshRealtimeRate

    // ========== RPC对外（供财务等模块调用）==========
    GetUserBalance / CreditDeposit / FreezeBalance / UnfreezeBalance
    ConfirmWithdrawal / ManualCredit / ManualDebit / SupplementCredit

    // ========== 通用 ==========
    GetEnumOptions
}
```

**预估方法总数**：C端 ~22 + B端 ~5 + RPC对外 ~8 + 通用 1 = **~36 个方法**

### 3.3 IDL 设计的关键约定

| 约定 | 说明 | 依据 |
|------|------|------|
| 金额字段用 `string` | 如 `"100.50"`，避免 double/float 精度丢失 | 工程 IDL 规范，需求中涉及法币+加密货币精度不同 |
| 时间字段用 `i64` | 毫秒时间戳 | 工程 IDL 规范 |
| 枚举值用 `i32` | 如 walletType: 1=中心,2=奖励,3=主播... | 工程 IDL 规范 |
| 分页请求固定字段 | `pageNo`(i32) + `pageSize`(i32) | 工程分页模式 |
| 分页响应固定字段 | `pageList`(list) + `total`(i64) + `pageNo`(i32) + `pageSize`(i32) | 工程分页模式 |
| 命名规范 | `{Action}{Entity}Req` / `{Action}{Entity}Resp` | 工程 IDL 命名规范 |
| RPC对外方法的 orderId | 必须包含 orderId/requestId 字段用于幂等 | 余额变更安全要求 |

### 3.4 RPC对外方法的 Req 设计要点

财务模块调用钱包的余额变更方法时，Req 中必须包含足够信息做幂等和审计：

```
所有余额变更RPC方法的 Req 应至少包含：
├── userId      (i64)    ← 操作对象
├── currency    (string) ← 币种代码
├── walletType  (i32)    ← 钱包类型（部分方法需要）
├── amount      (string) ← 变更金额
├── orderId     (string) ← 唯一业务单号（幂等依据）
└── reason      (string) ← 操作原因/备注（审计用）
```

## 四、数据库表设计思路

### 4.1 需要哪些表

根据需求分析和链路关系梳理，ser-wallet 至少需要以下表：

```
钱包核心
├── wallet                   ← 用户钱包余额表（核心中的核心）
├── transaction_log          ← 余额变更流水表（每笔余额变动必须记录）
└── idempotency_record       ← 幂等去重表（或用 transaction_log 表做去重）

充值
└── deposit_order            ← 充值订单表

提现
├── withdrawal_order         ← 提现订单表
└── withdrawal_account       ← 提现账户表（银行卡/电子钱包信息）

兑换
└── exchange_record          ← 兑换记录表

奖励
└── reward_record            ← 奖励记录表（签到/任务/VIP等奖励入账记录）

币种配置
├── currency_config          ← 币种配置表（币种参数、汇率浮动、阈值、状态）
├── rate_change_log          ← 汇率变更日志表
└── system_config            ← 系统配置表（基准币种等全局设置）
```

### 4.2 wallet 表——设计核心

这是整个系统最关键的表，设计时需要考虑的要点：

**唯一约束**：`user_id + currency_code + wallet_type` 三元组唯一——同一用户同一币种同一钱包类型只有一行记录。

**余额字段拆分**：
- `available_balance`：可用余额（用户可直接使用的）
- `frozen_balance`：冻结余额（提现待审核时冻结的）

不变量：`available + frozen = 该钱包的真实余额`，所有操作不得打破这个等式。

**金额存储精度**：使用 `DECIMAL` 类型而非 `FLOAT/DOUBLE`。法币 2 位小数，加密货币 8 位小数。具体精度需考虑最大值范围（如 VND 金额可能很大）。

**并发安全**：余额扣减/冻结操作使用乐观锁（`WHERE available_balance >= amount`，检查 `affected_rows`），防止超额操作。

### 4.3 deposit_order / withdrawal_order 表——订单核心

**订单号编码**：
- 充值：`C` + 16位（用 `utils.GenerateSnowflakeID()` 或随机数）
- 提现：`T` + 16位
- 补单：`B` + 16位（财务模块生成，钱包只接收）
- 加款：`A` + 16位（同上）
- 减款：`M` + 16位（同上）

**状态字段**：使用 `i32` 枚举，值在 `internal/enum/` 中定义。
- 充值状态：待支付 / 支付成功 / 支付失败
- 提现状态：待风控审核 / 风控驳回 / 待财务审核 / 财务驳回 / 待出款 / 出款失败 / 出款成功

**汇率快照**：订单创建时保存当时的汇率值到订单记录中，不依赖后续汇率变动。这是资金安全的基本要求——入账/出款金额以创建时的汇率为准。

### 4.4 currency_config 表——币种配置核心

关键字段：
- `code`（币种代码，唯一主键或唯一索引）
- `name`、`type`（法币/加密/平台）、`country`、`icon_url`
- `currency_symbol`、`thousand_separator`、`decimal_separator`
- `realtime_rate`（实时汇率，从三方API获取）
- `platform_rate`（平台汇率，阈值触发后更新）
- `rate_floating`（汇率浮动百分比）
- `threshold`（阈值百分比）
- `status`（启用/禁用）
- `sort_order`（排序）

入款汇率和出款汇率是**计算值**而非存储值：
- 入款汇率 = platform_rate × (1 + rate_floating%)
- 出款汇率 = platform_rate × (1 - rate_floating%)

### 4.5 transaction_log 表——流水记录

每笔余额变更必须生成一条流水。关键字段：
- `user_id`、`currency`、`wallet_type`
- `type`（操作类型：充值/提现冻结/提现扣款/提现解冻/兑换/人工加款/人工减款/补单等）
- `amount`（变更金额）
- `balance_before`（变更前余额）
- `balance_after`（变更后余额）
- `order_id`（关联业务单号）
- `remark`（备注）

这张表的数据量会很大（每笔操作至少一条），需要考虑索引策略和未来分表。

### 4.6 表设计的时间安排

表结构不需要一次性全部确定。按实现阶段：
- **Phase 1**：`wallet` + `currency_config` + `system_config` + `transaction_log` + `rate_change_log`
- **Phase 2**：`deposit_order`（做充值时）
- **Phase 3**：`withdrawal_order` + `withdrawal_account`（做提现时）
- **Phase 4**：`exchange_record` + `reward_record`（做兑换和记录时）

## 五、服务内部架构思路

### 5.1 handler.go — 纯委托层

遵循工程规范，handler.go 不写任何业务逻辑，每个方法一行委托到 ser 层。WalletServiceImpl 持有多个 service 实例的引用：

```
WalletServiceImpl {
    walletSer     ← 余额核心操作
    depositSer    ← 充值业务
    withdrawalSer ← 提现业务
    exchangeSer   ← 兑换业务
    recordSer     ← 记录查询
    currencySer   ← 币种配置B端
    rpcSer        ← RPC对外入口
}
```

每个 IDL 方法根据所属功能域，委托给对应的 service。

### 5.2 internal/ser/ — 业务逻辑层（核心代码所在）

按功能域划分文件，各 ser 之间有明确的调用关系：

```
ser/
├── wallet_ser.go       ← 余额操作原子方法集合（最底层，被所有业务线调用）
│                          getBalance / addBalance / deductBalance /
│                          freezeBalance / unfreezeBalance / confirmDeduct /
│                          createTransactionLog / idempotencyCheck
│
├── rate_ser.go         ← 汇率获取封装（封装对 currency_config 的查询，可加缓存）
│                          被 deposit/withdrawal/exchange 共同调用
│
├── deposit_ser.go      ← 充值业务（调用 wallet_ser + rate_ser + 财务RPC）
├── withdrawal_ser.go   ← 提现业务（调用 wallet_ser + rate_ser + 财务RPC + KYC RPC）
├── exchange_ser.go     ← 兑换业务（调用 wallet_ser + rate_ser）
├── record_ser.go       ← 记录查询（纯查DB，不调 wallet_ser）
├── currency_ser.go     ← 币种配置B端（增删改查 currency_config）
└── rpc_ser.go          ← RPC对外入口（统一入口，委托 wallet_ser 执行余额变更）
```

**调用规则**：
- deposit/withdrawal/exchange → 可调 wallet_ser 和 rate_ser
- rpc_ser → 只调 wallet_ser
- record_ser → 直接查 DB，不经过 wallet_ser
- wallet_ser → 不调任何其他 ser（它是最底层）
- rate_ser → 查本地 DB 或 Redis 缓存

### 5.3 internal/rep/ — 数据访问层

rep 层封装数据库查询逻辑，使用 gorm/gen 生成的 query 构建器：

```
rep/
├── wallet_rep.go        ← wallet 表操作
├── deposit_rep.go       ← deposit_order 表操作
├── withdrawal_rep.go    ← withdrawal_order + withdrawal_account 表操作
├── exchange_rep.go      ← exchange_record 表操作
├── reward_rep.go        ← reward_record 表操作
├── transaction_rep.go   ← transaction_log 表操作
├── currency_rep.go      ← currency_config 表操作
├── rate_log_rep.go      ← rate_change_log 表操作
└── system_rep.go        ← system_config 表操作
```

**rep 层的定位**：gorm/gen 自动生成的 repo 提供了基础 CRUD，rep 层用于封装**复杂查询**（如多条件筛选、聚合统计、跨表关联）。简单查询可直接使用 `query.Q.Xxx` 不经过 rep 层。

### 5.4 internal/errs/ — 错误码

错误码格式：`6` + `022`(服务编号) + `XXX`(3位模块序号)

按功能域分配错误码区间：

```
6022 001 ~ 099  ← 钱包通用（余额不足、币种禁用等）
6022 100 ~ 199  ← 充值相关
6022 200 ~ 299  ← 提现相关
6022 300 ~ 399  ← 兑换相关
6022 400 ~ 499  ← 币种配置相关
6022 500 ~ 599  ← RPC对外（幂等冲突、参数异常等）
```

错误码的 msg 只在代码中做标记，实际用户展示走 `ser-i18n` 翻译。

### 5.5 internal/enum/ — 枚举

需要注册的枚举至少包括：

```
钱包类型     1=中心 2=奖励 3=主播 4=代理 5=场馆
币种类型     1=法币 2=加密货币 3=平台币
充值状态     1=待支付 2=支付成功 3=支付失败
提现状态     1=待风控审核 2=风控驳回 3=待财务审核 4=财务驳回 5=待出款 6=出款失败 7=出款成功
流水类型     1=充值 2=提现冻结 3=提现扣款 4=提现解冻 5=兑换扣减 6=兑换入账
             7=人工加款 8=人工减款 9=补单入账 10=奖励入账 ...
币种状态     0=禁用 1=启用
```

遵循工程枚举注册机制——在 `enum_registry.go` 中注册，通过 `GetEnumOptions` 接口暴露给前端。

## 六、网关路由设计思路

### 6.1 gate-font（C端）路由

钱包模块的 C 端接口全部需要用户登录（走 `/api` 路由组，BlackCheck + FontAuth）：

```
/api/wallet/balance/getCurrencyList       ← 获取币种列表及余额
/api/wallet/balance/get                   ← 查询当前币种余额
/api/wallet/balance/overview              ← 钱包概览

/api/wallet/deposit/methods               ← 获取充值方式列表
/api/wallet/deposit/create                ← 创建充值订单
/api/wallet/deposit/usdtPaymentInfo       ← USDT支付页信息
/api/wallet/deposit/checkDuplicate        ← 重复充值检查

/api/wallet/withdrawal/methods            ← 获取提现方式列表
/api/wallet/withdrawal/create             ← 创建提现订单
/api/wallet/withdrawal/saveAccount        ← 保存提现账户
/api/wallet/withdrawal/getAccount         ← 获取提现账户
/api/wallet/withdrawal/limits             ← 提现限额查询

/api/wallet/exchange/rate                 ← 获取兑换汇率
/api/wallet/exchange/execute              ← 执行兑换

/api/wallet/record/deposit                ← 充值记录列表
```

## 七、RPC 接口契约思路

### 7.1 我方对外暴露（别人调我）

这是需要**优先确定**的接口——财务模块开发同事需要基于这些接口契约来编码。

| 方法 | 主要调用方 | 核心语义 |
|------|-----------|---------|
| `GetUserBalance` | 财务 | 查询用户指定币种/钱包类型的余额（纯读取） |
| `CreditDeposit` | 财务 | 充值入账：中心钱包 available +N |
| `FreezeBalance` | 钱包内部/投注 | 冻结余额：available -N, frozen +N |
| `UnfreezeBalance` | 财务 | 解冻退回：frozen -N, available +N（提现驳回） |
| `ConfirmWithdrawal` | 财务 | 确认提现扣款：frozen -N（钱出去了） |
| `ManualCredit` | 财务 | 人工加款：指定钱包 available +N |
| `ManualDebit` | 财务 | 人工减款：指定钱包 available -N（校验余额） |
| `SupplementCredit` | 财务 | 补单入账：中心钱包 +N，奖励钱包 +赠送（如有） |

**统一设计原则**：
- 除 `GetUserBalance` 外，其他 7 个方法都**必须幂等**（基于 orderId 去重）
- 除 `GetUserBalance` 外，其他 7 个方法都**必须在事务内**执行余额变更+写流水
- 所有方法的 Resp 返回操作后的余额信息，方便调用方校验

### 7.2 我方调用外部（我调别人）

| 调用目标 | 方法 | 场景 | 优先级 |
|---------|------|------|--------|
| **财务模块** | 查充值方式列表 | C端充值页展示 | 需要与财务对齐契约 |
| **财务模块** | 查提现方式列表 | C端提现页展示 | 需要与财务对齐契约 |
| **财务模块** | 匹配通道并创建支付 | 用户点击充值时 | 需要与财务对齐契约 |
| **KYC(012)** | 查询认证状态和姓名 | 提现前置判断 | 确认 KYC 已有此接口 |
| **BLog(006)** | AddActLog | 所有关键操作审计 | 通用接口，已有 |
| **S3(007)** | Upload | 币种图标上传 | 通用接口，已有 |
| **BUser(002)** | 查询操作人名称 | B端列表展示"操作人"字段 | 通用接口，已有 |

**关键协作点**：与财务模块需要**双向对齐** RPC 契约——我们暴露 R1-R8 供财务调用，财务暴露 FR1-FR3 供我们调用。建议尽早确定 IDL，双方可各自 mock 数据先独立开发。

### 7.3 与财务模块的联调策略

```
阶段1: 各自独立开发
├── 钱包侧：充值/提现流程中调用财务的地方，先用 mock 数据（返回固定方式列表/模拟通道匹配）
├── 财务侧：审核通过后调钱包的地方，先用 mock 数据（模拟余额变更成功）
└── 前提：双方 IDL 已对齐，接口契约已确定

阶段2: 联调对接
├── 替换 mock 为真实 RPC 调用
├── 重点联调链路：
│   ├── 充值回调 → 入账（财务调钱包 R2）
│   ├── 提现审核通过 → 扣款（财务调钱包 R5）
│   ├── 提现审核驳回 → 解冻（财务调钱包 R4）
│   ├── 充值方式查询（钱包调财务 FR1）
│   └── 通道匹配（钱包调财务 FR3）
└── 关键验证：幂等性（重复调用不重复操作）、事务一致性（余额+流水原子）
```

## 八、余额安全机制思路

这是整个钱包系统最核心的技术要求，绕不过，必须从一开始就设计好。

### 8.1 事务保障

凡是涉及余额变更的操作，必须在数据库事务内同时完成：

```
BEGIN TRANSACTION
├── 余额 UPDATE（可能多条：如兑换涉及扣法币+加BSB+加赠送）
├── 流水日志 INSERT（每个余额变动对应一条）
├── 订单状态 UPDATE（如有关联订单）
└── 幂等记录 INSERT（如适用）
COMMIT
```

使用 gorm/gen 提供的事务支持：
```go
query.Q.Transaction(func(tx *query.Query) error {
    // 使用 tx 操作，不用 query.Q
    return nil
})
```

### 8.2 幂等设计

**场景**：财务模块调用钱包RPC入账时，可能因网络超时重试，导致同一笔入账被调用多次。

**方案**：基于 orderId 去重。每个余额变更RPC方法：
1. 先查 `transaction_log` 中是否已存在该 orderId 的记录
2. 存在 → 直接返回成功（幂等响应）
3. 不存在 → 执行余额变更，在事务内写入 orderId

也可以用独立的 `idempotency_record` 表做去重，视具体实现选择。

### 8.3 乐观锁防超额

余额扣减/冻结操作使用 WHERE 条件保障：

```sql
UPDATE wallet
SET available_balance = available_balance - :amount,
    frozen_balance = frozen_balance + :amount
WHERE user_id = :userId
  AND currency_code = :currency
  AND wallet_type = :walletType
  AND available_balance >= :amount   -- 关键：防止超额
```

检查 `affected_rows`：
- 0 → 余额不足，返回错误
- 1 → 操作成功

### 8.4 数据一致性不变量

整个系统需要维护的核心不变量：

```
对于每个 (user_id, currency_code, wallet_type) 组合：
available_balance + frozen_balance = 该钱包的真实余额

任何操作都不能打破这个等式。
```

可以考虑后期加定时对账任务（`transaction_log` 正向累计 vs `wallet` 当前余额），但首版不是必须。

## 九、汇率机制实现思路

### 9.1 四层汇率结构

```
实时汇率 → 平台汇率 → 入款汇率 / 出款汇率

实时汇率：3家三方API取平均值，定时拉取，存入 currency_config.realtime_rate
平台汇率：只在偏差达到阈值时才更新为实时汇率值，存入 currency_config.platform_rate
入款汇率：= platform_rate × (1 + rate_floating%)，计算值不存储
出款汇率：= platform_rate × (1 - rate_floating%)，计算值不存储
```

### 9.2 阈值更新逻辑

定时任务的核心逻辑：

```
FOR EACH 启用的币种（BSB除外）:
    实时汇率 = avg(API_1, API_2, API_3)
    入款偏差 = |实时汇率 - 当前入款汇率| / 实时汇率 × 100%
    出款偏差 = |实时汇率 - 当前出款汇率| / 实时汇率 × 100%

    IF 入款偏差 >= 阈值% OR 出款偏差 >= 阈值%:
        UPDATE currency_config SET platform_rate = 实时汇率
        INSERT rate_change_log（记录变更）
```

### 9.3 BSB 特殊处理

BSB 锚定 USDT（10 BSB = 1 USDT），这个关系**写死在代码中**，不走配置。BSB 行的 platform_rate 固定为 10（对 USDT），不参与定时更新。

### 9.4 汇率缓存策略

汇率查询是高频接口（充值/提现/兑换都要调），考虑在 rate_ser 中加 Redis 缓存：
- 缓存 Key：如 `wallet:rate:{currency_code}`
- 缓存 TTL：30秒 ~ 1分钟
- 写入时机：查询 DB 后写入
- 失效时机：阈值更新触发后主动删除缓存

但对于**执行操作**（创建订单、执行兑换），应实时查 DB 取最新汇率，不用缓存值。

### 9.5 三方汇率 API 接入

需要接入 3 家外部汇率 API，取平均值。具体 API 选哪 3 家需要评审确认。实现时：
- 用 `common/pkg/utils/http_util.go` 发 HTTP 请求
- 3 个 API 并发请求（goroutine + WaitGroup），取平均
- 某个 API 请求失败时用剩余可用的取平均（至少 1 个可用才更新）
- API 地址/密钥等配置存 ETCD

## 十、定时任务思路

### 10.1 需要哪些定时任务

| 任务 | 触发方式 | 优先级 | 说明 |
|------|---------|--------|------|
| 汇率定时更新 | cron 定时 | 核心 | 拉取三方API → 阈值判断 → 更新平台汇率 → 写日志 |
| 充值订单超时关闭 | cron 定时 | 核心 | 待支付订单超过有效期（如30分钟）自动置为失败 |
| 对账校验 | cron 定时 | 可选（后期） | 流水正向累计 vs 当前余额，发现不一致告警 |

### 10.2 定时任务实现方式

参照工程中已有的 `ser-cron`（编号 020）服务，定时任务通常有两种方式：
- **方案A**：在 ser-wallet 内部用 Go 的 cron 库自行调度
- **方案B**：由 ser-cron 调度，通过 RPC 调用 ser-wallet 的方法执行

具体用哪种方式需看工程中 ser-cron 的使用模式。如果 ser-cron 是统一的任务调度中心，优先用方案B。

## 十三、实现中的注意事项

### 13.1 不能犯的错误

| 错误 | 后果 | 正确做法 |
|------|------|---------|
| 余额用 float/double 存储 | 精度丢失，对账不平 | 用 DECIMAL 存储，IDL 用 string 传输 |
| 冻结和创建订单不在同一事务 | 时间窗口内并发操作可能超额 | 同一个数据库事务 |
| RPC对外余额变更不做幂等 | 重复回调导致重复入账/扣款 | 基于 orderId 去重 |
| 余额变更不记流水 | 无法对账追溯 | 事务内同时写流水 |
| 汇率查询不做缓存 | 高频RPC调用拖慢整体性能 | rate_ser 加 Redis 缓存 |
| 网关handler写业务逻辑 | 违反工程规范 | 一行委托到 ser 层 |
| 重复造轮子 | 违反工程规范 | 用 common/pkg 已有工具 |
| 手写 Model 层 | 与工程规范冲突 | 用 gorm/gen 生成 |

### 13.2 需要复用的已有轮子

| 需求 | 已有工具 | 位置 |
|------|---------|------|
| 分布式唯一ID | `utils.GenerateSnowflakeID()` | common/pkg/utils/snowflake.go |
| 分页参数处理 | `utils.GetOffsetAndLimit()` | common/pkg/utils/request.go |
| 时间处理 | `utils.UnixTimeToDatetimeStr()` 等 | common/pkg/utils/time_util.go |
| JSON序列化 | `utils.MarshalStr()` / `UnmarshalStr()` | common/pkg/utils/json_util.go |
| 审计日志 | `rpc.BLogClient().AddActLog()` | common/rpc/rpc_client.go |
| S3上传 | `rpc.S3Client().Upload()` | common/rpc/rpc_client.go |
| Redis操作 | `rds` 包 | common/pkg/rds/ |
| HTTP请求 | `utils` HTTP工具 | common/pkg/utils/http_util.go |
| 错误返回 | `ret.BizErr()` / `ret.RpcOne()` | common/pkg/ret/ |
| 上下文取值 | `contextx.UserID(ctx)` 等 | common/pkg/contextx/ |

### 13.3 金额计算注意

所有涉及金额计算的地方（汇率换算、手续费、赠送等），Go 代码中应使用精确计算而非浮点运算：
- 推荐使用 `shopspring/decimal` 库做金额运算
- 或者用整数运算（最小单位存储）
- Thrift 中 string 传输 → Go 中 Decimal 计算 → 存储时 DECIMAL 类型

### 14.3 一句话概括实现思路

**从 IDL 开始，先立地基（币种配置 + 余额核心 + RPC对外），再建业务（兑换→充值→提现），全程遵守工程规范，与财务模块尽早对齐接口契约、各自独立开发、最后联调对接。**
