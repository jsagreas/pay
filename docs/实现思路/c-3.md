## 一、先决问题：一个服务还是两个服务

### 1.1 需要决定的事

币种配置和多币种钱包是做成一个 `ser-wallet` 还是拆成 `ser-wallet` + `ser-coincfg`。

### 1.2 建议方向

**建议同一个服务 `ser-wallet`，内部分 package**。理由：

- 工程中现有模式就是一个服务承载多个业务领域（如 ser-app 里有 banner + category + contentStrategy + enum）
- 币种配置和钱包数据关联紧密（钱包几乎每个接口都要查币种/汇率），同服务内部方法调用比 RPC 调用性能更好、事务更简单
- 减少基础设施开销（一套 ETCD 配置、一个端口、一套 DB 连接池）
- IDL 层面可以拆分文件（`currency.thrift` / `wallet.thrift` / `exchange.thrift` 等），service.thrift 定义一个 `WalletService` 包含全部方法

### 1.3 如果选择两个服务

则需要两套独立的基础设施配置（两个服务编号、两个端口、可能两个 DB），币种配置对外暴露 RPC 供钱包调用。代码量和维护成本增加但解耦更彻底。

**评审时确认此点，后续按确认结果执行。**

## 二、IDL 定义（绕不过的第一步）

> 依据：工程认知c-3.md 第五节 IDL先行开发模式

### 2.1 为什么 IDL 是第一步

工程采用 IDL 先行开发模式。所有 RPC 接口必须先在 Thrift IDL 中定义，然后 `gen_kitex.sh` 生成 Kitex 代码（客户端/服务端桩代码），再编写实现。**不定义 IDL 就没有代码可写**。

同时，IDL 也是与财务模块团队对齐的"合同"——双方依赖的 RPC 接口必须先在 IDL 层面锁定。

### 2.2 需要创建的 IDL 文件

```
common/idl/ser-wallet/
├── service.thrift          # RPC 服务接口定义（WalletService 全部方法）
├── currency.thrift         # 币种配置相关的请求/响应结构体
├── wallet.thrift           # 钱包核心+余额相关
├── recharge.thrift         # 充值相关
├── exchange.thrift         # 兑换相关
├── withdraw.thrift         # 提现相关
└── record.thrift           # 记录查询相关
```

### 2.3 IDL 遵循的规范

遵循工程已有规范（参考 ser-app 的 IDL）：

- **命名空间**：`namespace go ser_wallet`
- **请求/响应命名**：`{Action}{Entity}Req` / `{Action}{Entity}Resp`
  - B端分页用 `Page` 前缀：`PageCurrencyReq` / `PageCurrencyResp`
  - C端列表用 `List` 前缀：`ListEnabledCurrencyReq` / `ListEnabledCurrencyResp`
  - 创建/更新/删除：`Create` / `Update` / `Delete` / `Status`
- **分页响应标准结构**：`pageList` + `total` + `pageNo` + `pageSize`
- **字段注解**：`required` 必填 / `optional` 可选，中文注释说明

### 2.5 优先锁定的 IDL 接口

**以下接口必须最先锁定**，因为是跨团队依赖的"合同"：

| 优先级 | 接口 | 原因 |
|--------|------|------|
| P0 | E1~E6（钱包6个RPC） | 财务模块的硬依赖，不锁定财务无法开发 |
| P0 | F1~F4（财务4个RPC） | 钱包充值/提现流程的硬依赖 |
| P1 | R1~R7（币种配置7个RPC） | 钱包自身几乎所有接口都依赖 |

**行动项**：评审后第一时间和财务团队对齐 E1~E6 + F1~F4 的 IDL 定义。

### 3.4 表设计通用规范

遵循工程已有规范（参考工程认知c-3.md 3.4节）：

- **主键**：`id BIGINT AUTO_INCREMENT`
- **软删除**：`delete_at BIGINT NOT NULL DEFAULT 0`（0=未删除，非零=删除时间戳毫秒）
- **审计字段**：`create_by` / `create_at` / `update_by` / `update_at`（GORM 审计插件自动填充）
- **金额字段**：用 `DECIMAL` 类型，法币精度2位、加密精度8位（**不要用浮点数**）
- **所有查询必须加** `.Where(DeleteAt.Eq(0))`

## 六、核心资金操作层（最关键的代码，绕不过）

> 依据：链路关系c-3.md 3.6节RPC暴露层 + 流程思路c-3.md 3.7节

### 6.1 为什么这层最关键

`WalletCoreService`（对应 E1~E6）是整个钱包系统的核心：
- 被财务模块通过 RPC 调用（充值入账、提现冻结/解冻/确认、人工加减款）
- 被钱包内部充值/兑换/提现子模块复用
- 所有资金变动都经过这层，确保逻辑统一

**不管需求细节怎么变，这6个操作的核心逻辑不会变。**

### 6.2 六个核心操作的实现要点

```
E1 GetBalance — 查询余额
   输入：用户ID + 币种代码
   输出：各子钱包(CENTER/REWARD/STREAMER/AGENT)的可用余额和冻结余额
   要点：纯读操作，无特殊逻辑

E2 CreditWallet — 加款 ⚠️ 最高频
   输入：用户ID + 币种代码 + 子钱包类型 + 金额 + 订单号 + 来源标识
   操作：可用余额 += 金额
   必须做：
     ① 幂等：同一订单号不重复入账（先查流水表有无该订单号）
     ② 写流水：操作前余额 → 操作后余额完整记录
     ③ 并发安全：UPDATE ... SET balance = balance + amount WHERE ...
        (数据库层面原子操作，或使用乐观锁/悲观锁)

E3 DebitWallet — 减款
   输入：同E2
   操作：可用余额 -= 金额
   特殊规则：余额不足时实际扣款=当前余额（不报错）
   必须做：同E2的幂等和流水

E4 FreezeBalance — 冻结
   输入：用户ID + 币种代码 + 金额 + 订单号
   操作：可用余额 -= 金额，冻结余额 += 金额
   校验：可用余额 ≥ 冻结金额
   必须做：原子操作（两个字段同时变更）

E5 UnfreezeBalance — 解冻
   输入：同E4
   操作：冻结余额 -= 金额，可用余额 += 金额
   必须做：幂等（同一订单号不重复解冻）

E6 ConfirmFrozenDeduction — 确认扣除
   输入：同E4
   操作：冻结余额 -= 金额（资金真正离开用户钱包）
   必须做：幂等（同一订单号不重复确认）
```

### 6.3 幂等机制的实现思路

**每次资金操作都在流水表中记录，用 `(订单号 + 操作类型)` 作为幂等键：**

```
创建资金流水前，先查询：
  SELECT 1 FROM wallet_transaction
  WHERE order_no = ? AND operation_type = ?
  LIMIT 1

如果已存在 → 说明已处理过 → 直接返回成功（不重复操作）
如果不存在 → 执行余额变更 + 写入流水（在同一个事务内）
```

也可以在流水表上建唯一索引 `(order_no, operation_type)` ，利用数据库唯一约束保证幂等。

### 6.4 并发安全的实现思路

资金操作必须防止并发导致的数据不一致。两种可选方案：

**方案A：数据库原子更新（推荐，简单可靠）**
```sql
UPDATE wallet_balance
SET available_balance = available_balance + ?
WHERE user_id = ? AND currency_code = ? AND wallet_type = ?
  AND available_balance + ? >= 0  -- 扣款时防止负数
```
利用数据库行级锁保证原子性。

**方案B：SELECT FOR UPDATE（悲观锁）**
```sql
BEGIN;
SELECT * FROM wallet_balance WHERE ... FOR UPDATE;
-- 业务逻辑校验
UPDATE wallet_balance SET ...;
INSERT INTO wallet_transaction ...;
COMMIT;
```

方案A更简单，方案B控制力更强。实际选择在编码阶段确定。

### 6.5 事务要求

以下操作必须在同一个数据库事务内完成：

```
CreditWallet：余额变更 + 流水写入 → 同一事务
DebitWallet：余额变更 + 流水写入 → 同一事务
FreezeBalance：可用余额减 + 冻结余额增 + 流水写入 → 同一事务
UnfreezeBalance：冻结余额减 + 可用余额增 + 流水写入 → 同一事务
ConfirmFrozenDeduction：冻结余额减 + 流水写入 → 同一事务
兑换(W13)：扣源币种 + 加BSB + 加赠送 + 写兑换流水 → 同一事务
```

使用工程已有的 GORM Gen 事务模式：
```go
query.Q.Transaction(func(tx *query.Query) error {
    // 所有操作使用 tx 而非 query.Q
    return nil
})
```

## 七、汇率系统实现（绕不过的基础能力）

> 依据：需求理解c-3.md 2.4节汇率体系 + 流程思路c-3.md 2.3节

### 7.1 汇率系统核心逻辑

无论需求细节怎么变，汇率系统的核心公式和阈值机制不会变：

```
实时汇率 = 3家三方API取均值
平台汇率 = 实时汇率（首次）或 达到阈值时更新
入款汇率 = 平台汇率 × (1 + 汇率浮动%)
出款汇率 = 平台汇率 × (1 - 汇率浮动%)
偏差值 = |实时汇率 - 平台汇率| / 实时汇率 × 100%
```

### 7.2 定时任务实现思路

工程已有 XXL-JOB 定时任务基础设施（参考工程认知c-3.md 3.10.3节）。

汇率刷新逻辑：
1. 并发请求3家三方汇率API（需调研选择哪3家）
2. 取成功返回的结果平均值作为实时汇率
3. 遍历每个已启用币种，判断偏差值是否 ≥ 阈值
4. 超阈值的币种更新平台汇率 + 写汇率日志
5. 手动刷新(B7)和定时任务(T1)共用同一个核心方法

### 7.3 锁汇机制

创建订单时调用 `GetTransactionRate` 获取锁定汇率，**将锁定的汇率写入订单记录**。后续计算以订单中的锁定汇率为准，不受平台汇率变动影响。

## 十一、缓存策略（需要但不紧急的优化层）

> 依据：工程认知c-3.md 7.5节缓存失效策略 + 链路关系c-3.md 9.3节调用频次

### 11.1 需要缓存的数据

| 数据 | 缓存理由 | 更新频率 | 建议TTL |
|------|---------|---------|--------|
| 已启用币种列表 | 钱包首页每次加载都查 | B端启禁用时变更（低频） | 5~10分钟 |
| 币种配置（精度/符号） | 余额格式化每次都用 | B端编辑时变更（低频） | 5~10分钟 |
| 平台汇率 | 每次创建订单都查 | 定时任务更新（可配频率） | 等于定时任务间隔 |
| 基准币种信息 | 一次设置后不变 | 永不变 | 长期/永久 |

### 11.2 缓存失效策略

遵循工程已有模式（写时全量清除）：
- B端编辑/启禁用操作后 → `cache.DelAll(ctx)` 清除相关缓存
- 定时任务更新汇率后 → 清除汇率缓存
- C端读取时 → 缓存旁路模式（Redis优先 → 未命中查DB → 写Redis）

### 11.3 钱包余额不缓存

钱包余额是高频写、强一致的数据，不适合用缓存旁路模式。直接读 DB 即可。如果后续有性能瓶颈，再考虑用 Redis 做余额缓存（但需要处理缓存与DB一致性问题）。

## 十二、与财务模块的联调方案（绕不过的协作）

> 依据：依赖关系c-3.md 第七节跨团队协调 + 第五节双向依赖分析

### 12.1 联调的三个阶段

```
阶段一：IDL接口锁定（越早越好）
    双方确认：
    ├── 钱包暴露的6个RPC (E1~E6) 的 Thrift IDL
    └── 财务暴露的4个RPC (F1~F4) 的 Thrift IDL
    产出：双方IDL文件，各自可以开始生成代码

阶段二：Mock独立开发
    ├── 钱包侧：Mock财务的F1~F4 → 充值/提现流程开发
    └── 财务侧：Mock钱包的E1~E6 → 审核/入账流程开发

阶段三：真实联调
    重点验证：
    ├── 充值回调入账（E2 CreditWallet 幂等测试）
    ├── 提现完整流程（冻结→审核→解冻/确认扣除的时序正确性）
    └── 人工修正（加减款余额变更的正确性）
```

### 12.2 联调时的关键测试点

| 测试场景 | 验证内容 |
|---------|---------|
| 充值入账重复回调 | 同一订单号E2被调用多次，余额只增加一次 |
| 提现创建后F4失败 | 冻结的余额能正确回滚解冻 |
| 提现审核驳回 | E5解冻后可用余额恢复，冻结余额减少 |
| 提现出款成功 | E6确认扣除后冻结余额减少，资金真正出账 |
| 人工加款多子钱包 | E2被调用多次(每个子钱包一次)，各子钱包余额分别正确 |
| 人工减款余额不足 | E3减款时余额不足，实际扣款=当前余额不报错 |

### 12.3 充值回调归属

充值回调（银行/扫码类）大概率由财务模块接收（因为通道由财务管理），然后财务调用钱包 E2 入账。

USDT 链上确认回调可能直接到钱包模块（平台自主承载）。

**此归属需评审确认，但不影响钱包侧 E2 的实现——无论回调走哪边，最终都是调用 E2。**

## 十四、需要但暂不紧急的实现要点

以下是重要但可以在开发过程中逐步完善的部分：

### 14.1 金额精度处理

- Go 内部使用 `decimal` 库（如 `shopspring/decimal`）处理金额计算
- 数据库存储用 `DECIMAL` 类型
- 法币精度2位、加密货币精度8位
- 展示时的四舍五入、千分位格式化由 C端或 Service 层处理
- **不要用 float64 做金额计算**

### 14.2 消费优先级规则

资金钱包余额 = 中心钱包 + 奖励钱包。消费时先扣中心再扣奖励。混合投注返奖按占比分别计入。

**这个规则目前由投注/游戏模块触发，钱包模块只需暴露 E2/E3 即可。消费优先级的编排逻辑在调用方实现。但如果后续要求钱包模块内部处理消费优先级，则需要新增一个复合操作接口。**

### 14.3 操作日志和审计

- 每个 B端写操作产生操作日志（参考 ser-app 中 `addBannerLog` 模式）
- 资金流水表本身就是审计日志
- Service 层使用 `defer` + 命名返回值记录失败操作的日志

### 14.4 提现币种规则校验

```
法币充值 → 法币提现
USDT充值 → USDT提现
法币+USDT混合 → 法币提现
法币→BSB兑换 → 法币提现
```

这个规则需要在提现创建时校验。具体实现可能需要追溯用户的充值历史来判断"充值方式"。评审时确认这个校验的精确逻辑。

### 14.5 重复转账拦截

仅适用于银行转账和扫码充值（不适用USDT链上充值）：
- 查询 `同UserID + 同币种 + 同金额` 的待支付订单数
- 1-2笔 → 返回警告（C端弹窗，用户可选继续）
- ≥3笔 → 硬拦截，禁止创建

### 14.6 功能开关

可在 ETCD 中配置功能开关（参考工程认知c-3.md 9.4节）：

```
/slg/switch/wallet/withdrawal/enabled   提现开关
/slg/switch/wallet/exchange/enabled     兑换开关
/slg/switch/wallet/recharge/enabled     充值开关
```

运维可实时关闭某个功能，无需重启服务。使用 `etcd.Get()` 读取，`etcd.Hook()` 监听变更。

## 十五、总结：实现的关键路径

```
第1步  IDL定义 + 跨团队IDL对齐
       ↓ 产出：Thrift IDL文件，gen_kitex.sh 生成代码
       ↓ 关键：E1~E6 和 F1~F4 必须优先锁定

第2步  数据库表设计 + GORM代码生成
       ↓ 产出：model/query/repo 三层生成代码
       ↓ 关键：wallet_balance 和 wallet_transaction 表设计

第3步  服务骨架搭建 + 基础设施注册
       ↓ 产出：main.go + handler.go + cfg/enum/errs + ETCD + 网关路由
       ↓ 关键：遵循现有工程模式，不要创造新模式

第4步  WalletCoreService 实现 (E1~E6)
       ↓ 产出：核心资金操作能力，可被财务联调
       ↓ 关键：幂等 + 事务 + 并发安全

第5步  币种配置全部实现 (B1~B10, C1~C3, R1~R7, T1)
       ↓ 产出：汇率和币种基础数据就绪
       ↓ 关键：汇率阈值更新逻辑，三方API对接

第6步  兑换实现 → 充值实现 → 提现实现
       ↓ 从简到难，逐步集成
       ↓ 关键：提现的冻结-提交一致性

第7步  与财务模块真实联调
       ↓ 产出：全链路跑通
       ↓ 关键：充值回调幂等 + 提现审核全流程
```

### 核心原则

1. **IDL 先行** — 不定义 IDL 就没有代码可写，和财务的 IDL 对齐是第一优先级
2. **遵循现有模式** — 不创造新的架构模式，照着 ser-app 的四层结构复制
3. **核心先行** — WalletCoreService(E1~E6) 是一切的基础，先做好先联调
4. **从简到难** — 兑换 → 充值 → 提现，兑换最先验证核心Service层设计
5. **幂等为王** — 所有资金操作必须幂等，这是不可商量的硬要求
6. **事务保一致** — 余额变更和流水写入必须在同一个事务内
