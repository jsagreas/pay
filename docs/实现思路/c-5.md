# 一、数据模型 — 一切的地基

## 1.1 绕不过的核心模型

不管需求细节怎么调整，以下数据模型是确定的，因为它们直接对应需求中的硬性业务概念：

**余额表（最核心）**

这张表是整个钱包系统的心脏。需求明确了"4子钱包 × 多币种"的余额矩阵（需求文档4-1），所以余额表的唯一键必然是 `user_id + currency + wallet_type` 三元组。

```
每条记录代表：某用户的某币种下某子钱包的余额

核心字段（绕不过）：
  user_id       用户ID
  currency      币种代码（BSB/VND/IDR/THB/USDT）
  wallet_type   子钱包类型（中心/奖励/代理/主播）
  available     可用余额
  frozen        冻结余额（提现冻结用）

唯一约束：user_id + currency + wallet_type
```

为什么余额不能用单行存所有币种：因为币种是可配置的、可新增的，不能硬编码为列。用行来表达币种维度，扩展性好。

为什么要有冻结字段：提现流程是"先冻结→审核→扣或退"，需要区分可用和冻结。

**充值订单表**

```
核心字段：
  order_no        平台订单号（雪花ID）
  user_id         用户ID
  currency        币种
  amount          充值金额
  method          充值方式
  status          状态（待支付/成功/失败/超时）
  channel_order   渠道订单号（财务返回的）
  bonus_id        关联奖金活动ID（可选）
  bonus_amount    奖金金额（可选）
```

**提现订单表**

```
核心字段：
  order_no        平台订单号
  user_id         用户ID
  currency        币种
  amount          提现金额
  fee             手续费
  method          提现方式
  status          状态（审核中/成功/失败）
  account_info    提现账户信息（银行卡/电子钱包，JSON或关联ID）
  fail_reason     失败原因（审核拒绝时记录）
```

**兑换订单表**

```
核心字段：
  order_no        平台订单号
  user_id         用户ID
  from_currency   源币种
  to_currency     目标币种（固定BSB）
  from_amount     源金额
  to_amount       目标金额
  rate            使用的汇率（创建时锁定）
  gift_amount     赠送金额
  status          状态（成功/失败）
```

**余额变动流水表**

```
核心字段：
  user_id         用户ID
  currency        币种
  wallet_type     子钱包类型
  change_type     变动类型（充值/提现冻结/提现扣款/提现退回/兑换扣减/兑换增加/人工加款/人工减款/补单...）
  amount          变动金额（正数=增，负数=减）
  before_balance  变动前余额
  after_balance   变动后余额
  ref_order_no    关联订单号

用途：对账、审计、C端记录查询的底层数据源
```

**币种配置表**

```
核心字段：
  currency_code   币种代码（唯一键）
  currency_name   币种名称
  currency_type   类型（法币/加密/平台）
  icon            图标
  precision       精度（小数位数）
  deposit_min/max 充值限额
  withdraw_min/max/daily 提现限额
  fee_type        手续费类型（固定/百分比）
  fee_value       手续费值
  arrive_days     T+N到账天数
  sort_order      排序
  status          启用/禁用
```

**汇率配置表**

```
核心字段：
  currency_code   币种代码
  rate            汇率值（1 BSB = X 该币种）
  updated_by      最后修改人
  updated_at      最后修改时间
```

**提现账户表**（用户保存的银行卡/电子钱包信息，后续回显用）

```
核心字段：
  user_id         用户ID
  currency        币种
  method          提现方式
  account_info    账户信息（银行名称/卡号/户名 或 电子钱包类型/ID）
```

**日志表**（汇率变更日志、操作日志 — 币种配置模块用）

## 1.2 关键设计决策

**余额精度用什么类型？**

这是绕不过的技术决策。金额字段不能用 float，必须用精确类型。

两个方向：
- **方向A：数据库用 DECIMAL，Go 用 shopspring/decimal 库** — 精确，但性能稍慢，GORM 需要自定义类型映射
- **方向B：数据库用 BIGINT 存最小单位（分/最小精度单位），Go 用 int64** — 性能好，简单，但需要统一单位换算

建议偏向方向B（int64存最小单位），原因：
- 现有工程的ID、时间戳都是 int64，风格一致
- 数据库操作（加减比较）用整数更快且无精度问题
- VND 这种大数法币（1VND = 1最小单位）天然适合整数
- 缺点是展示层需要做精度换算，但这是可控的前端/出参处理

**这个决策需要你在编码前确认。**

**余额表要不要拆？**

当前阶段一张余额表够用（user_id + currency + wallet_type 联合唯一键）。单表结构简单，事务操作方便。

如果后期用户量级到千万以上，可以按 user_id 做分表，但现在不需要。先做简单的，遇到瓶颈再优化。

# 二、IDL 设计 — IDL 先行

## 2.1 为什么 IDL 是绕不过的第一步

工程是 IDL-First 开发模式（工程认知已确认）。所有 RPC 接口必须先在 Thrift IDL 中定义，然后 gen_kitex.sh 生成桩代码，才能开始写业务逻辑。

更关键的是：**你暴露给财务的 RPC 接口，财务那边也需要你的 IDL 来生成客户端桩代码调用你**。所以 IDL 越早定义越好，不用等业务逻辑写完。

## 2.2 IDL 文件组织思路

```
common/idl/ser-wallet/
├── service.thrift         主入口：定义 WalletService 的所有方法签名
├── wallet.thrift          钱包相关：余额/充值/提现/兑换 的请求/响应/枚举
├── currency.thrift        币种配置相关：币种/汇率 的请求/响应
└── common_types.thrift    共享类型（如果需要）：分页、通用枚举
```

遵循现有 IDL 命名规范（namespace go ser_wallet，PascalCase 方法名，Req/Resp 后缀）。

## 2.3 IDL 方法分组思路

粗粒度分三大组，每组里面的方法与接口评估文档对应：

**组1：币种配置（B端管理 + RPC 暴露）**

B端管理方法（gate-back 路由进来）：
- 币种 CRUD（列表/新增/编辑/启禁用）
- 汇率设置/查询
- 日志查询（汇率日志、操作日志）

RPC 暴露方法（被钱包自身和财务调用）：
- 获取可用币种列表
- 获取币种详情/配置参数
- 获取汇率

**组2：钱包 C 端（gate-font 路由进来）**

- 余额查询、可用币种列表
- 充值相关（获取方式、创建订单、状态查询、奖金活动列表）
- 兑换相关（汇率预览、创建兑换）
- 提现相关（获取方式、限额查询、账户信息、创建提现）
- 记录查询（充值/提现/兑换/奖励列表 + 详情）

**组3：钱包 RPC 暴露（供财务模块调用 — 优先级高）**

- 充值入账
- 提现确认扣款
- 提现失败解冻
- 人工加款
- 人工减款
- 人工补单入账
- 余额查询

**组3 是需要最先确定 IDL 的**，因为财务开发者需要这些接口定义来编写调用代码。你可以先把组3的 IDL 定下来给财务，组1组2可以后续补充。

## 2.4 一个 Service 还是拆分多个？

现有工程每个 ser-xxx 模块只有一个 Service（如 ItemService、AppService）。

建议 ser-wallet 也用**一个 WalletService**，不拆。原因：
- 跟工程风格一致
- 一个 Kitex 服务进程对应一个 Service，部署简单
- 方法多不是问题（ser-buser 有 31 个方法，ser-live 有 36 个）
- 内部按 ser/ 目录拆分不同 service 文件就够了（wallet_service.go, currency_service.go 等）

# 四、余额操作 — 最核心的技术关注点

## 4.1 为什么余额操作是绕不过的重点

钱包系统的核心就是**正确地操作余额**。所有业务场景（充值、提现、兑换、人工修正）最终都归结为余额表的增减操作。如果余额操作有任何bug（并发超扣、流水不一致、重复入账），就是资损事故。

## 4.2 余额操作的统一模式

不管哪种业务场景，余额变动操作都应遵循同一个模式：

```
1. RPC/业务校验（在事务之外完成）
2. 开事务
3. 余额变动（带乐观锁条件）
4. 更新订单状态
5. 写余额变动流水
6. 提交事务
```

**绕不过的三条铁律**：

**铁律1：余额变动 + 订单状态 + 流水记录 必须同一个事务**

不允许出现"余额加了但流水没写"或"流水写了但订单状态没更新"的不一致状态。

**铁律2：扣减操作必须用乐观锁**

```sql
-- 正确做法
UPDATE balance SET available = available - amount
WHERE user_id=? AND currency=? AND wallet_type=? AND available >= amount

-- 检查受影响行数，0 表示余额不足
```

绝不能先 SELECT 再应用层判断再 UPDATE — 并发场景必出问题。

**铁律3：RPC 调用不放在 DB 事务内**

```
正确：获取汇率(RPC) → 计算 → [开事务] → 操作余额 → [提交事务]
错误：[开事务] → 获取汇率(RPC) → 操作余额 → [提交事务]
```

RPC 调用可能超时、重试，放在事务内会拉长事务持有时间，影响数据库连接池和锁竞争。

## 4.3 余额操作的复用设计思路

充值入账、兑换、人工加减款……本质都是"给某个子钱包加/减一笔钱 + 写一条流水"。

建议在 balance_service 或 rep 层抽象一个**余额变动原子方法**，内部封装事务+乐观锁+流水写入。上层业务方法（充值/兑换/提现/人工修正）调用这个原子方法，传入不同的参数（变动类型、关联订单号等）。

这样做的好处：
- 余额操作逻辑只写一处，避免各业务场景各写一遍出现不一致
- 事务边界、乐观锁、流水写入这些关键逻辑集中管理
- 测试也只需覆盖一个核心方法

粗粒度的设计方向：

```go
// 伪代码，表达思路，不是最终实现
type BalanceChange struct {
    UserID     int64
    Currency   string
    WalletType int
    Amount     int64          // 正=加，负=减
    ChangeType int            // 枚举：充值/兑换/提现冻结/...
    RefOrderNo string         // 关联订单号
}

// 在同一个事务内执行一组余额变动
func (s *BalanceService) ExecuteChanges(ctx context.Context, tx *gorm.DB, changes []BalanceChange) error {
    for _, ch := range changes {
        // 执行余额变动（带乐观锁）
        // 写余额变动流水（记录变动前后余额）
    }
    return nil
}
```

兑换场景：传入两个 change（源币种扣减 + BSB增加），如有赠送再加一个 change
充值入账：传入一个 change（中心钱包增加），如有奖金再加一个 change
提现冻结：传入一个 change（available减 + frozen加）
...

# 五、幂等性 — 所有写 RPC 的底线

## 5.1 为什么幂等绕不过

你暴露给财务的 RPC 接口（充值入账、提现扣款/解冻、人工加减款），在以下场景会被重复调用：
- 网络超时重试
- 渠道回调重发
- 财务侧补偿重试

如果不做幂等，同一笔充值可能入账两次 = 直接资损。

## 5.2 幂等实现思路

**每个写操作的 RPC 接口，都用一个业务唯一键做幂等**：

```
充值入账      → 用平台订单号 order_no 做幂等键
提现确认/解冻  → 用提现订单号 order_no 做幂等键
人工加款/减款  → 用修正单号做幂等键
人工补单      → 用补单号做幂等键
```

实现方式（两层防护）：

```
第一层（快速返回）：
  进入方法后，先查订单/流水状态，如果已完成 → 直接返回成功，不执行任何操作

第二层（兜底）：
  流水表的关联订单号字段加唯一索引
  即使第一层并发穿透，写入流水时唯一键冲突 → catch 错误 → 返回成功
```

两层结合：正常情况第一层快速返回，极端并发第二层兜底。

# 七、与财务模块的对接 — 双向 RPC 协调

## 7.1 绕不过的协调事项

钱包和财务的 RPC 依赖是双向的，你们必须协商的事项：

**你需要先给财务的（越早越好）：**

1. **IDL 文件中组3（RPC暴露）的方法签名** — 财务需要生成客户端桩代码
2. **请求/响应结构体定义** — 特别是入参（哪些字段必填、类型、含义）
3. **幂等约定** — 用什么做幂等键，重复调用返回什么
4. **错误码约定** — 你返回哪些错误码，财务需要据此做分支处理

**你需要财务先给你的：**

1. **支付配置查询 RPC 的 IDL** — 你的 C 端充值/提现页面依赖这个接口获取方式列表
2. **创建渠道支付订单 RPC 的 IDL** — 你的充值流程依赖这个
3. **提交提现审核 RPC 的 IDL** — 你的提现流程依赖这个
4. **回调通知的时机和参数约定** — 财务什么时候调你的入账/扣款/解冻

## 7.2 协调的优先级

```
最优先（阻塞双方开发）：
  你的 RPC 暴露接口 IDL → 给财务
  财务的支付配置查询 IDL → 给你

次优先（可以先 mock）：
  财务的创建渠道订单 / 提交审核 → 你的充值/提现流程可以先 mock 财务返回
  你的入账/扣款/解冻 → 财务可以先 mock 你的返回

最后联调：
  全链路打通：充值 → 渠道回调 → 入账
  全链路打通：提现 → 审核 → 扣款/解冻
  人工修正 → 余额操作
```

## 7.3 IDL 先行的实操方式

你可以先定义好 IDL 并执行 gen_kitex.sh，生成桩代码，然后把 `common/idl/ser-wallet/` 和 `common/pkg/kitex_gen/ser_wallet/` 提交。财务那边 pull 代码后就可以用 `rpc.WalletClient()` 调用你的接口了（虽然你的服务还没启动，但客户端代码已经能编译）。

这就是 IDL-First 的价值：**接口定义先行，双方并行开发，联调时对接**。

# 八、缓存策略 — 高频读场景

## 8.1 需要缓存的数据

币种配置的 RPC 接口调用频率会很高（C 端每次操作都可能调），但数据变更频率极低（运营偶尔改一次）。这是典型的"读多写极少"场景，适合缓存。

```
数据                  变更频率      建议缓存方式        过期时间
─────────────────────────────────────────────────────────
可用币种列表          极低（天级）   Redis + 本地缓存     5-10分钟
币种配置参数          极低          Redis               5-10分钟
汇率                  低（小时级）   Redis               1-3分钟
```

**缓存更新策略**：B 端修改操作成功后，主动删除对应 Redis 缓存键。下次查询时自动重建缓存。

简单的 Cache-Aside 模式够用，不需要复杂的缓存方案。

## 8.2 不能缓存的数据

```
用户余额              不缓存 — 余额必须实时准确，从DB读
充值/提现订单状态      不缓存 — 状态流转中，必须实时
```

余额是**强一致**场景，任何缓存都可能导致用户看到错误余额。直接读 DB，走索引查询很快。

# 九、枚举定义 — 统一语言

## 9.1 绕不过的枚举

以下枚举在 IDL、Go 代码、数据库中都会用到，需要尽早统一定义：

```
子钱包类型 WalletType:
  1=中心钱包  2=奖励钱包  3=代理钱包  4=主播钱包

  1=法币  2=加密货币  3=平台币

  1=充值入账  2=充值奖金
  3=提现冻结  4=提现扣款  5=提现退回
  6=兑换扣减  7=兑换增加  8=兑换赠送

提现订单状态 WithdrawStatus:
```

# 十、开发分期 — 从独立到依赖

## 10.1 分期思路

按依赖关系由少到多排列，先做独立闭环的，后做需要联调的：

**第一期：基础设施（无外部依赖）**

```
做什么：
  1. 编写 IDL（全部方法签名，重点是 RPC 暴露给财务的那组）
  2. gen_kitex.sh 生成桩代码
  3. 注册服务到 common（namesp, rpc_client, go.work）
  4. 建表、建库、配 ETCD
  5. GORM Gen 生成 model/query/repo
  6. 搭建 ser-wallet 骨架（main.go, handler.go, cfg, errs, enum）
  7. 接入 gate-back 和 gate-font 路由

验收标准：
  服务能启动，能注册到 ETCD，网关能路由到 ser-wallet，空接口能通
```

**第二期：币种配置（自闭环，不依赖任何其他模块）**

```
做什么：
  1. 币种 CRUD（B端）
  2. 汇率设置/查询（B端）
  3. 日志查询（B端）
  4. RPC 暴露接口（获取币种列表/详情/汇率/配置参数）
  5. 缓存（Redis）

验收标准：
  B端能配置币种和汇率，RPC 接口能返回正确数据
  这期完成后，其他依赖币种配置的模块就可以调用了
```

**第三期：钱包核心能力（低外部依赖）**

```
做什么：
  1. 余额查询（C端）— 读本地表
  2. 可用币种列表（C端）— 调自己的币种配置 RPC
  3. 兑换功能（C端）— 只依赖汇率，完全闭环
  4. 记录查询（C端）— 读本地表
  5. RPC 暴露接口全部实现（入账/扣款/解冻/加减款/补单/余额查询）

验收标准：
  兑换能完整跑通（最早的端到端交易验证）
  RPC 暴露接口能被调用并正确操作余额
  提供给财务联调的接口全部就绪
```

**第四期：充值 + 提现（需要财务接口联调）**

```
做什么：
  1. 充值功能（C端）— 依赖财务的支付配置 + 创建渠道订单
  2. 提现功能（C端）— 依赖财务的支付配置 + 提交审核 + 用户服务 KYC
  3. 充值入账回调对接（财务调你）
  4. 提现审核结果回调对接（财务调你）

开发策略：
  先写好你这边的逻辑，用 mock 代替财务和用户服务的 RPC 返回
  财务接口就绪后替换 mock 做联调
```

## 10.2 为什么这个顺序

```
第一期（骨架）→ 不依赖任何人，你自己搞定
第二期（币种）→ 不依赖任何人，且产出被其他模块依赖，越早越好
第三期（核心）→ 低依赖 + 产出给财务联调，兑换可以最早验证余额操作正确性
第四期（充提）→ 强依赖财务，放最后，需要联调
```

兑换是你的**最佳烟囱测试场景** — 它涉及两个币种的余额变动、事务操作、流水记录，但不依赖外部模块。如果兑换跑通了，说明你的余额操作核心逻辑是对的，充值入账和提现扣款只是同一套逻辑的不同参数。

# 十三、非功能性关注点

## 13.1 绕不过的非功能性要求

以下是需求文档明确提出的，不是"有则更好"而是"必须做到"：

**账务一致性**

余额变动必须和流水记录严格一致。上面第四节的事务铁律就是为此服务的。

**幂等性**

所有 RPC 暴露的写接口必须幂等。上面第五节已覆盖。

**KYC 后端强制校验**

非USDT 提现的 KYC 校验必须在后端做，不能只靠前端拦截。即使 C 端被绕过（伪造请求），后端创建提现订单时也必须校验 KYC 状态。

## 13.2 应该关注但不急的

```
审计日志        每个写操作调 rpc.BLogClient().AddActLog()（工程已有范式）
异常检测        短时间大量充值/提现的告警机制（可后续迭代）
对账机制        定时任务核对余额和流水是否一致（可后续迭代）
白名单          提现白名单（特殊用户绕过限制）（需求提了但细节待定）
```

# 十四、关键风险点

## 14.1 技术风险

**并发余额操作**

同一用户同时发起多笔交易（充值入账 + 兑换 + 人工加款同时到达），都要操作同一行余额记录。乐观锁能解决正确性问题，但可能导致部分操作因锁竞争失败需要重试。

应对：乐观锁 + 调用方重试（RPC 暴露接口的调用方如财务，应有重试机制）。

**提现补偿**

冻结余额成功但提交审核 RPC 失败时，必须解冻。如果解冻也失败（DB 异常），用户的钱就卡住了。

应对：本地先写一条补偿记录（状态=待补偿），定时任务扫描补偿。或者简单处理 — 解冻失败时记日志告警，人工介入。初期可用简单方案，规模大了再上定时补偿。

## 14.2 协调风险

**财务接口迟迟不就绪**

你的充值和提现强依赖财务的 RPC 接口。如果财务开发进度滞后，你会被阻塞。

应对：
- 第一二三期完全不依赖财务，先推进
- 充值/提现先用 mock，财务就绪后替换
- IDL 先行 — 接口定义先对齐，实现可以各自推进

**需求变更**

预评审阶段，细节可能变。但以下骨架不会变：
- 4子钱包 × 多币种余额矩阵
- 充值/兑换/提现三大核心流程
- 与财务的双向 RPC 依赖
- BSB 平台基准币种概念
- 提现先冻结后审核的流程

可能变的是：限额规则的具体数值、审核层级数量、某些页面字段、奖金活动的具体规则。这些变化不影响核心架构和数据模型，只影响参数和分支逻辑。

# 十五、总结 — 绕不过清单

按重要性排列，你在编码前/编码中必须确认和执行的事项：

```
1. 数据模型     余额表的三元组设计 + 金额精度类型决策 → 一切的地基
2. IDL 定义     组3（RPC暴露给财务）优先定义 → 解除双方阻塞
3. 服务注册     namesp + rpc_client + go.work → 让其他模块能调你
4. 余额操作     事务 + 乐观锁 + 流水一致性 → 钱包系统的核心正确性
5. 幂等设计     所有写 RPC 的幂等键 → 防止资损
6. 错误码       6022xxx 码段规划 → 跨模块错误沟通的基础
7. 枚举统一     子钱包类型/订单状态/变动类型 → 全系统统一语言
8. 缓存策略     币种配置高频读缓存 → 性能保障
9. 开发分期     骨架→币种→核心→充提 → 降低依赖风险
10. 财务协调    IDL先给、mock先行、联调后置 → 并行开发不等人
```
