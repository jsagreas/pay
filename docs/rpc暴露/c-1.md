### 2.2 工程现有 RPC 模式（我们必须遵循的）

通过扫描全工程 22 个已注册服务，提炼出统一模式：

#### 服务端启动模式

```go
// main.go
func main() {
    lg.InitKLog()
    etcd.InitEtcd()
    port := flag.String("port", etcd.DirectGet(ctx, namesp.EtcdXxxPort), "端口号")
    flag.Parse()

    srv := xxxservice.NewServer(
        &XxxServiceImpl{...},
        rpc.InitRpcServerParams(namesp.EtcdXxxService, utils.S2Int(ctx, *port))...,
    )
    srv.Run()
}
```

#### 客户端注册模式

```go
// common/rpc/rpc_client.go
var (
    xxxService *xxxservice.Client
    xxxOnce    sync.Once
)

func XxxClient() xxxservice.Client {
    xxxOnce.Do(func() {
        c, err := xxxservice.NewClient(namesp.EtcdXxxService, InitRpcClientParams()...)
        xxxService = &c
        if err != nil {
            lg.Log().CtxErrorf(context.Background(), "RPC 创建ser-xxx客户端错误 %s", err)
        }
    })
    return *xxxService
}
```

#### handler.go 委托模式

```go
// ser-xxx/handler.go — 零业务逻辑，纯委托
type XxxServiceImpl struct {
    xxxService *ser.XxxService
}

func (s *XxxServiceImpl) MethodName(ctx context.Context, req *ser_xxx.MethodReq) (resp *ser_xxx.MethodResp, err error) {
    return s.xxxService.MethodName(ctx, req)
}
```

#### IDL 中 RPC 接口的组织方式

通过分析 `ser-user/service.thrift` 和 `ser-live/service.thrift`，RPC 接口遵循以下模式：

- **一个 service 声明包含所有端**：B端、C端、RPC 在同一个 `service XxxService` 中
- **用注释分区块**：`// ========== RPC接口（供其他服务调用）==========`
- **RPC 结构体单独文件**：`xxx_rpc.thrift` 放跨服务结构体定义，`service.thrift` 通过 include 引用

```thrift
// service.thrift 中的 RPC 区块
service UserService {
    // ========== C端接口 ==========
    users.LoginResp Login(1: users.LoginReq req)

    // ========== B端接口 ==========
    users.UserPageResp QueryUserPage(1: users.UserPageReq req)

    // ========== RPC接口（供其他服务调用）==========
    user_rpc.UserInfoExpendResp GetUserInfoExpend(1: user_rpc.GetUserInfoReq req)
    user_rpc.BatchUserInfoResp BatchGetUserInfo(1: user_rpc.BatchGetUserInfoReq req)
}
```

## 4. 暴露接口分类一：钱包资金操作（6个）

这 6 个 RPC 是**财务模块（ser-finance）的硬依赖**，也是钱包模块的核心能力输出。

### 4.1 接口清单

| # | RPC 方法名 | 调用方 | 触发场景 | 操作性质 |
|---|-----------|-------|---------|---------|
| W1 | `RpcGetUserBalance` | 财务 | 人工加/减款弹窗展示余额 | 读 |
| W2 | `RpcCreditWallet` | 财务 | 充值成功回调入账、人工加款审核通过 | 写（加） |
| W3 | `RpcDebitWallet` | 财务 | 人工减款审核通过 | 写（减） |
| W4 | `RpcFreezeBalance` | 财务（或钱包自调用） | 用户提交提现申请，冻结可用余额 | 写（冻结） |
| W5 | `RpcUnfreezeBalance` | 财务 | 提现审核驳回、出款失败 | 写（解冻） |
| W6 | `RpcConfirmFrozenDeduction` | 财务 | 提现出款成功，最终确认扣除 | 写（确认扣除） |

### 4.2 资金操作三阶段模型

```
      可用余额（available）             冻结余额（frozen）
           │                                │
           │  W2 CreditWallet (+)           │
           │  W3 DebitWallet (-)            │
           │                                │
           │         W4 FreezeBalance        │
           │  ─────────────────────────────►│
           │       available - amount       │  frozen + amount
           │                                │
           │        W5 UnfreezeBalance       │
           │  ◄─────────────────────────────│
           │       available + amount       │  frozen - amount
           │                                │
           │                                │  W6 ConfirmFrozenDeduction
           │                                │  frozen - amount（最终扣除）
           ▼                                ▼
```

**设计约束**（来自参考项目 A + B 的借鉴）：

| 约束 | 说明 | 来源 |
|------|------|------|
| CAS 乐观锁 | 每次余额变更带 version 字段做乐观锁 | 参考项目 A（p9-services） |
| 幂等性 | W2/W3/W4/W5/W6 全部以 `orderId` 做幂等去重 | 参考项目 A + 需求文档 |
| balance_before/after | 每笔流水记录变更前后余额，便于审计对账 | 参考项目 A |
| 事务保证 | 余额表 + 流水表在同一个数据库事务内 | 基础要求 |

### 4.3 各接口的业务链路定位

```
充值链路：  三方回调 → 财务确认 → W2 CreditWallet → 钱包余额+
提现链路：  用户申请 → W4 Freeze → 财务审核
            ├── 驳回/失败 → W5 Unfreeze → 钱包余额恢复
            └── 出款成功 → W6 Confirm → 冻结余额清除
补单链路：  财务创建补单 → 审核通过 → W2 CreditWallet → 入账
加款链路：  财务创建加款 → 审核通过 → W2 CreditWallet → 入账
减款链路：  财务查余额(W1) → 创建减款 → 审核通过 → W3 DebitWallet → 扣款
```

## 5. 暴露接口分类二：余额查询扩展（4个）

这 4 个 RPC 服务于更广泛的业务场景，调用方不限于财务模块。

### 5.1 接口清单

| # | RPC 方法名 | 调用方 | 触发场景 | 说明 |
|---|-----------|-------|---------|------|
| E1 | `RpcBatchGetBalance` | 财务/统计 | 批量查询多个用户余额（报表、批量操作） | 批量读 |
| E2 | `RpcConsumeDebit` | 投注域/商城 | 消费扣款（下注、购买等） | 写（扣） |
| E3 | `RpcRewardCredit` | 投注域/直播 | 返奖入账、打赏收入入账 | 写（加） |
| E4 | `RpcSupplementCredit` | 财务 | 补单入账（充值金额 + 额外奖金分别入不同子钱包） | 写（加） |

### 5.2 与分类一的区别

| 维度 | 分类一（资金操作） | 分类二（扩展） |
|------|----------------|-------------|
| 调用方 | 几乎全是财务模块 | 多种模块 |
| 确定性 | 需求文档中明确定义 | 部分为预判性设计 |
| 当前阶段 | P0 必须实现 | E1/E4 为 P0，E2/E3 为 P1（可后置） |

### 5.3 E2/E3 的预判依据

**RpcConsumeDebit（消费扣款）**：
- 竞彩域下注时需要扣除用户余额 → 项目背景明确定位为"合规竞彩平台"
- 商城购买（ser-mall）的 `currencyType` 字段暗示未来可能关联真实钱包
- 与 `RpcDebitWallet` 的区别：ConsumDebit 是 C 端用户主动消费，DebitWallet 是 B 端管理员操作

**RpcRewardCredit（返奖/打赏入账）**：
- 投注域结算返奖需要入账到中心钱包
- 直播打赏收入需要入账到主播钱包（`ser-live/live_rpc.thrift` 中 `totalIncome` 字段印证）
- 与 `RpcCreditWallet` 的区别：RewardCredit 入奖励钱包，CreditWallet 入中心钱包

| 维度 | C5 GetTransactionRate | C6 SnapshotBettingRate |
|------|----------------------|----------------------|
| 用途 | 充值/提现/兑换下单时锁汇 | 投注下单时汇率快照 |
| 锁定时长 | 订单生命周期内有效（分钟级） | 投注-结算周期（可能数天） |
| 调用方 | 钱包内部 | 竞彩/投注域 |
| 当前状态 | P0 必须实现 | P2 预留，调用方尚未确定 |

## 10. RPC 接口详细签名设计

以下签名遵循工程 IDL 规范（参见 `z-readme/IDL设计/c-1.md` 第 2 章）。

### 10.1 wallet_rpc.thrift — 钱包资金操作 + 余额查询

```thrift
namespace go ser_wallet

// ============================================================
// 一、钱包资金操作（供财务模块调用）
// ============================================================

// W1 — 查询用户余额
struct RpcGetUserBalanceReq {
    1: required i64 userId;          // 用户ID
    2: required string currencyCode; // 币种代码
}
struct RpcGetUserBalanceResp {
    1: i64 userId;
    2: string currencyCode;
    3: list<RpcWalletBalanceItem> wallets; // 各子钱包余额
    4: string totalAvailable;              // 总可用余额
    5: string totalFrozen;                 // 总冻结余额
}
struct RpcWalletBalanceItem {
    1: i32 walletType;        // 钱包类型 1=中心 2=奖励 3=主播 4=代理 5=场馆
    2: string available;      // 可用余额
    3: string frozen;         // 冻结余额
}

// W2 — 充值/加款入账
struct RpcCreditWalletReq {
    1: required i64 userId;
    2: required string currencyCode;
    3: required i32 walletType;       // 目标钱包类型
    4: required string amount;        // 入账金额
    5: required string orderId;       // 幂等键（外部订单号）
    6: required i32 bizType;          // 业务类型（1=充值 2=加款 3=返奖 ...）
    7: optional string remark;        // 备注
}
struct RpcCreditWalletResp {
    1: bool success;
    2: string balanceAfter;           // 操作后余额
    3: string transactionId;          // 钱包内部流水号
}

// W3 — 减款扣款
struct RpcDebitWalletReq {
    1: required i64 userId;
    2: required string currencyCode;
    3: required i32 walletType;
    4: required string amount;
    5: required string orderId;       // 幂等键
    6: required i32 bizType;          // 业务类型（1=减款 2=消费 ...）
    7: optional string remark;
}
struct RpcDebitWalletResp {
    1: bool success;
    2: string balanceAfter;
    3: string transactionId;
}

// W4 — 冻结余额
struct RpcFreezeBalanceReq {
    1: required i64 userId;
    2: required string currencyCode;
    3: required string amount;
    4: required string orderId;       // 关联的提现订单号（幂等键）
}
struct RpcFreezeBalanceResp {
    1: bool success;
    2: string availableAfter;         // 冻结后可用余额
    3: string frozenAfter;            // 冻结后冻结余额
}

// W5 — 解冻余额
struct RpcUnfreezeBalanceReq {
    1: required i64 userId;
    2: required string currencyCode;
    3: required string orderId;       // 原冻结订单号（幂等键）
}
struct RpcUnfreezeBalanceResp {
    1: bool success;
    2: string availableAfter;
    3: string frozenAfter;
}

// W6 — 确认冻结扣除
struct RpcConfirmFrozenDeductionReq {
    1: required i64 userId;
    2: required string currencyCode;
    3: required string orderId;       // 原冻结订单号（幂等键）
}
struct RpcConfirmFrozenDeductionResp {
    1: bool success;
    2: string frozenAfter;
}

// ============================================================
// 二、余额查询扩展
// ============================================================

// E1 — 批量查询多用户余额
struct RpcBatchGetBalanceReq {
    1: required list<i64> userIds;    // 用户ID列表
    2: required string currencyCode;
}
struct RpcBatchGetBalanceResp {
    1: map<i64, RpcGetUserBalanceResp> balances; // userId → 余额信息
}

// E2 — 消费扣款（投注/商城等场景）
// 复用 RpcDebitWalletReq/Resp，通过 bizType 区分

// E3 — 返奖/打赏入账
// 复用 RpcCreditWalletReq/Resp，通过 bizType 和 walletType 区分

// E4 — 补单入账（需同时入中心钱包和奖励钱包）
struct RpcSupplementCreditReq {
    1: required i64 userId;
    2: required string currencyCode;
    3: required string mainAmount;    // 主金额 → 中心钱包
    4: required string bonusAmount;   // 奖金额 → 奖励钱包
    5: required string orderId;       // 幂等键
    6: optional string remark;
}
struct RpcSupplementCreditResp {
    1: bool success;
    2: string mainBalanceAfter;       // 中心钱包操作后余额
    3: string bonusBalanceAfter;      // 奖励钱包操作后余额
    4: string transactionId;
}
```

### 10.2 currency_rpc.thrift — 币种配置对外

```thrift
namespace go ser_wallet

// ============================================================
// 三、币种配置对外 RPC
// ============================================================

// C1 — 获取平台汇率
struct RpcGetPlatformRateReq {
    1: required string fromCurrency;  // 源币种
    2: required string toCurrency;    // 目标币种
}
struct RpcGetPlatformRateResp {
    1: string rate;                   // 汇率
    2: i64 updateTime;                // 汇率更新时间（毫秒时间戳）
}

// C2 — 转换为基准币种金额
struct RpcConvertToBaseCurrencyReq {
    1: required string amount;        // 原始金额
    2: required string fromCurrency;  // 源币种
}
struct RpcConvertToBaseCurrencyResp {
    1: string baseAmount;             // 基准币种金额
    2: string baseCurrency;           // 基准币种代码（如 USDT）
    3: string rateUsed;               // 使用的汇率
}

// C3 — 获取单个币种配置
struct RpcGetCurrencyConfigReq {
    1: required string currencyCode;
}
struct RpcCurrencyConfigInfo {
    1: string currencyCode;           // 币种代码
    2: string currencyName;           // 币种名称
    3: i32 currencyType;              // 币种类型 1=法币 2=加密 3=平台
    4: i32 decimalPlaces;             // 小数位数
    5: string symbol;                 // 币种符号（$, ₮, ...）
    6: string icon;                   // 图标URL
    7: string thousandSeparator;      // 千分位分隔符
    8: i32 status;                    // 状态 1=启用 2=禁用
}

// C4 — 获取所有已启用币种
struct RpcGetAllEnabledCurrenciesResp {
    1: list<RpcCurrencyConfigInfo> currencies;
}

// C5 — 获取并锁定交易汇率
struct RpcGetTransactionRateReq {
    1: required string currencyCode;  // 交易币种
    2: required i32 txType;           // 交易类型 1=充值 2=提现 3=兑换
}
struct RpcGetTransactionRateResp {
    1: string lockedRate;             // 锁定汇率
    2: i64 expireTime;                // 汇率过期时间
    3: string rateId;                 // 汇率快照ID（关联到订单）
}

// C6 — 快照投注汇率
struct RpcSnapshotBettingRateReq {
    1: required list<string> currencies; // 需要快照的币种列表
}
struct RpcSnapshotBettingRateResp {
    1: map<string, string> rateSnapshot; // currency → rate
    2: i64 snapshotTime;                 // 快照时间
    3: string snapshotId;                // 快照ID
}

// C7 — 校验币种是否启用
struct RpcIsCurrencyEnabledReq {
    1: required string currencyCode;
}
struct RpcIsCurrencyEnabledResp {
    1: bool enabled;
}

// ============================================================
// 四、通用辅助
// ============================================================

// U1 — 获取基准币种信息
struct RpcGetBaseCurrencyResp {
    1: string currencyCode;           // 基准币种代码
    2: string currencyName;           // 基准币种名称
}

// U2 — 获取币种列表（简化版）
struct RpcGetCurrencyListResp {
    1: list<RpcCurrencyConfigInfo> currencies; // 含启用+禁用
}
```

| 接口 | 内部依赖 | 外部依赖 |
|------|---------|---------|
| RpcCreditWallet | 余额表+流水表+CAS | 无 |
| RpcDebitWallet | 余额表+流水表+CAS | 无 |
| RpcFreezeBalance | 余额表+CAS | 无 |
| RpcUnfreezeBalance | 余额表+CAS | 无 |
| RpcConfirmFrozenDeduction | 余额表+流水表+CAS | 无 |
| RpcSupplementCredit | 余额表+流水表+CAS | 无 |

**特征**：全部写操作，需要 CAS 乐观锁 + 幂等机制 + 事务保证。是整个钱包模块的核心。

### 13.4 Phase 3 — 汇率（交易汇率 RPC）

```
目标：让充值/提现/兑换的锁汇功能可用
接口：C1/C2/C5
阻塞：充值/提现/兑换下单
```

| 接口 | 内部依赖 | 外部依赖 |
|------|---------|---------|
| RpcGetPlatformRate | 汇率表 | 三方汇率 API |
| RpcConvertToBaseCurrency | 汇率表 | 三方汇率 API |
| RpcGetTransactionRate | 汇率表+锁汇缓存 | 三方汇率 API |

**特征**：依赖外部三方汇率源，需要对接第三方 API。

### 13.5 Phase 4 — 扩展（消费/返奖/投注）

```
目标：支撑投注/直播/商城等扩展场景
接口：E2/E3 + C6
可独立：不阻塞主流程
```

| 接口 | 说明 |
|------|------|
| RpcConsumeDebit | 复用 DebitWallet 逻辑，增加 bizType |
| RpcRewardCredit | 复用 CreditWallet 逻辑，增加 walletType |
| RpcSnapshotBettingRate | 汇率快照 + 持久化 |

**特征**：基于 Phase 2/3 的能力扩展，逻辑复用度高。
