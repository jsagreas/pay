# ser-wallet 对外 RPC 接口暴露分析

> **文档定位**：系统分析 ser-wallet 需要暴露哪些 RPC 接口供工程内其他模块调用（不含 B端/C端 HTTP 接口）。
>
> **核心问题**：除了 B端管理接口（gate-back）和 C端用户接口（gate-font）外，ser-wallet 还需要**通过 RPC 直接暴露哪些能力**给 ser-finance、游戏服务、直播服务、活动服务等内部模块？
>
> **分析依据**：
> - 需求原型：`/Users/mac/gitlab/ser-wallet/tmp/` 下约 120 张产品原型和文档
> - 已完成的分析文档：依赖关系/d-3.md、链路关系/d-3.md、架构设计/d-3.md、IDL设计/d-3.md、流程思路/d-3.md、接口评估/d-3.md、评审反馈/d-3.md、模块参考A/d-3.md、模块参考B/d-3.md
> - 工程事实：现有 IDL 模式（ser-item 的 `GetItemsByType`/`BatchGetItemsByIds` 供他服调用的先例）
>
> **核心原则**：RPC 接口是 ser-wallet 对外的"契约"，一旦发布，调用方会依赖它。因此必须**想清楚再暴露**——接口太少会导致调用方绕路，接口太多会增加维护成本。

## 一、为什么需要 RPC 暴露

### 1.1 ser-wallet 的三类对外接口

```
┌─────────────────────────────────────────────────────────┐
│                   ser-wallet 对外接口全景                   │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  ① B端接口（gate-back → RPC → ser-wallet）               │
│     · 币种配置 CRUD、汇率展示、后台查看用户钱包              │
│     · 调用路径：管理员浏览器 → gate-back HTTP → RPC        │
│     · 对应 IDL：currency.thrift + wallet.thrift 中的 B端方法│
│                                                          │
│  ② C端接口（gate-font → RPC → ser-wallet）               │
│     · 余额查询、充值、提现、兑换、记录查询                    │
│     · 调用路径：用户 APP → gate-font HTTP → RPC            │
│     · 对应 IDL：wallet/deposit/withdraw/exchange/record    │
│                                                          │
│  ③ RPC 暴露接口（其他 ser-xxx → RPC → ser-wallet）  ← 本文 │
│     · 入账、扣款、冻结、解冻、投注、礼物、奖励、查询等        │
│     · 调用路径：ser-finance/game/live/activity 直接 RPC 调用│
│     · 对应 IDL：wallet_rpc.thrift                          │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### 1.2 为什么需要独立的 RPC 暴露层

B端和 C端接口是**用户触发**的——人在操作。
RPC 暴露接口是**系统触发**的——程序在调用。

两者的差异：

| 维度 | B端/C端 | RPC 暴露 |
|------|--------|---------|
| 触发者 | 人（浏览器/APP 点击） | 程序（其他服务代码调用） |
| 调用频率 | 低（用户操作频率） | 可能极高（投注高峰每秒数千次） |
| 幂等要求 | 前端防重即可 | **必须服务端幂等**（重试/回调可能重复） |
| 错误处理 | 返回前端展示 | 调用方程序逻辑处理，需精确错误码 |
| 超时容忍 | 用户可等 2-3s | 调用方可能设 1-3s 超时 + 重试 |
| 参数来源 | 用户输入 + Context | 程序构造（userId, amount, orderNo） |

### 2.1 调用方全景

```
                      ┌─────────────────────┐
                      │     ser-wallet      │
                      │   (WalletService)   │
                      └──────────┬──────────┘
                                 │
          ┌──────────────────────┼──────────────────────┐
          │                      │                      │
    ┌─────┴──────┐        ┌─────┴──────┐        ┌─────┴──────┐
    │ser-finance │        │  游戏/投注   │        │  直播/礼物   │
    │ (023)      │        │  (预留)     │        │  (预留)     │
    │ 财务管理    │        │             │        │             │
    └────────────┘        └────────────┘        └────────────┘
          │
    ┌─────┴──────┐        ┌────────────┐
    │  活动/任务   │        │  统计/报表   │
    │  (预留)     │        │  (预留)     │
    └────────────┘        └────────────┘
```

### 2.2 各调用方的调用场景

#### ser-finance（财务管理）— **最核心的调用方**

| 场景 | 触发时机 | 需要钱包做什么 |
|------|---------|-------------|
| 充值入账 | 三方支付回调成功 | 往用户中心钱包加钱 |
| 充值补单 | 人工确认补单 | 往用户中心钱包加钱（同入账，不同 bizType） |
| 提现冻结 | 用户提交提现申请 | 冻结用户中心钱包对应金额 |
| 提现驳回解冻 | 审核驳回（任一级） | 解冻被冻结的金额 |
| 提现出款确认 | 三方代付成功回调 | 从冻结金额中正式扣除 |
| 人工加款 | 管理员审批通过 | 往指定钱包（中心/奖励/主播/代理）加钱 |
| 人工减款 | 管理员审批通过 | 从指定钱包扣钱 |
| 查看用户钱包 | 人工修正页面查询 | 返回用户所有币种所有钱包余额 |

#### 游戏/投注服务 — **高频调用方（预留）**

| 场景 | 触发时机 | 需要钱包做什么 |
|------|---------|-------------|
| 投注前校验 | 用户下注前 | 检查余额是否充足 |
| 投注扣款 | 确认下注 | 从中心+奖励钱包扣款（优先级扣款） |
| 返奖入账 | 赛事/游戏结算 | 按原扣款比例分别入账中心和奖励钱包 |
| 场馆进入 | 进入第三方游戏场馆 | 从中心钱包转入场馆钱包（预留） |
| 场馆退出 | 离开第三方游戏场馆 | 从场馆钱包转回中心钱包（预留） |

#### 直播/礼物服务 — **中频调用方（预留）**

| 场景 | 触发时机 | 需要钱包做什么 |
|------|---------|-------------|
| 送礼扣款 | 用户送礼 | 从中心钱包扣款（仅中心，不扣奖励） |
| 主播收益 | 收到礼物打赏 | 往主播钱包加钱 |

#### 活动/任务服务 — **低频调用方（预留）**

| 场景 | 触发时机 | 需要钱包做什么 |
|------|---------|-------------|
| 活动奖励 | 活动条件达成 | 往奖励钱包加钱（附带稽核倍数要求） |
| 任务奖励 | 任务完成 | 往奖励钱包加钱 |
| 充值奖金 | 充值关联活动 | 往奖励钱包加钱（充值成功后触发） |

### 2.3 首期 vs 预留判定

| 调用方 | 首期状态 | 判定依据 |
|--------|---------|---------|
| **ser-finance** | **必须首期** | 充值/提现是钱包核心链路，无财务联调=钱包无法使用 |
| 游戏/投注 | 预留（接口定义 + TODO 实现） | 投注模块尚未开发，但接口契约需要提前定义 |
| 直播/礼物 | 预留 | 直播模块已有但礼物打赏功能未开发 |
| 活动/任务 | 预留 | 活动模块尚未开发 |
| 统计/报表 | 预留 | 当前无此服务 |

## 三、RPC 接口完整清单

### 3.1 总览

**共 24 个 RPC 方法，分为 7 组：**

| 组别 | 方法数 | 首期/预留 | 调用方 |
|------|--------|---------|--------|
| A. 资金核心操作 | 7 | **首期必须** | ser-finance |
| B. 余额查询 | 3 | **首期必须** | ser-finance + 其他 |
| C. 投注操作 | 2 | 预留 | 游戏/投注 |
| D. 礼物/主播操作 | 2 | 预留 | 直播/礼物 |
| E. 奖励操作 | 1 | 预留（接口首期） | 活动/任务 |
| F. 代理操作 | 1 | 预留 | 代理模块 |
| G. 钱包间转账/场馆 | 3 | 预留 | 多方 |
| **H. 币种配置 RPC** | **6** | **首期必须** | **财务/游戏/统计** |
| **合计** | **25** | | |

> **注意**：H 组（币种配置 RPC）虽然也是 RPC 暴露，但定义在 `currency.thrift` 中（同进程子模块），与 A-G 组的 `wallet_rpc.thrift` 分开。详见[第八章](#八与币种配置-rpc-的区分)。

### 3.2 A 组：资金核心操作（7 个）— 首期必须

这是钱包的"命脉"——所有资金变动都通过这些接口。

#### A1. WalletCredit — 通用入账

```
做什么：往指定用户的指定钱包加钱
谁调用：ser-finance（充值回调成功、补单）
为什么需要：
  · 充值成功后，财务模块收到三方回调，验证通过后调此接口入账
  · 这是"钱从外部进入用户钱包"的唯一入口
  · 补单场景也复用此接口（bizType 区分）

入参：
  · userId        — 必填，入账用户
  · currencyCode  — 必填，币种（如 "VND"）
  · walletType    — 必填，目标钱包类型（充值入账=1中心钱包）
  · amount        — 必填，入账金额（string，如 "5000000.00000000"）
  · bizType       — 必填，业务类型（1充值 2补单）
  · orderNo       — 必填，业务订单号（幂等键）
  · remark        — 选填，备注

出参：
  · balanceAfter  — 变更后余额
  · transactionId — 流水记录 ID

关键约束：
  · 幂等：同一 orderNo 重复调用，第二次直接返回首次结果，不重复入账
  · 事务：余额更新 + 流水记录在同一个 DB 事务中
```

#### A2. WalletDebit — 通用扣款

```
做什么：从指定用户的指定钱包扣钱
谁调用：ser-finance（提现出款确认）
为什么需要：
  · 提现审核通过 + 三方代付成功后，财务调此接口正式扣款
  · 区别于冻结：冻结是"预扣"，此接口是"确认扣"

入参：
  · userId, currencyCode, walletType, amount, bizType, orderNo, remark

出参：
  · balanceAfter, transactionId

关键约束：
  · 幂等：同一 orderNo 不重复扣款
  · 余额校验：可用余额不足时返回明确错误码
```

#### A3. WalletFreeze — 冻结余额

```
做什么：冻结用户中心钱包的指定金额（标记为"不可用但未扣除"）
谁调用：ser-wallet 内部（用户提交提现时内部调用）或 ser-finance
为什么需要：
  · 用户提交提现时，金额不能立即扣除（还要经过审核）
  · 但也不能让用户继续使用这笔钱（可能导致余额不足）
  · 冻结 = "锁定但不扣"，审核通过出款后才真正扣除

入参：
  · userId, currencyCode, amount, orderNo

出参：
  · frozenAfter   — 冻结后的冻结总额
  · transactionId

关键约束：
  · 可用余额 = balance - frozenAmount，冻结时需检查可用余额是否充足
  · 同一 orderNo 不重复冻结
```

#### A4. WalletUnfreeze — 解冻余额

```
做什么：解冻之前冻结的金额（资金回到可用状态）
谁调用：ser-finance（提现审核驳回）
为什么需要：
  · 提现被驳回时（一级/二级/三级任一级），冻结金额需要释放
  · 用户能重新看到并使用这笔钱

入参：
  · userId, currencyCode, amount, orderNo

出参：
  · availableAfter — 解冻后的可用余额
  · transactionId

关键约束：
  · 幂等：同一 orderNo 不重复解冻
  · 校验：解冻金额不能超过当前冻结总额
```

#### A5. DebitFrozen — 从冻结金额中扣除

```
做什么：提现出款成功后，正式从冻结金额中扣除（余额永久减少）
谁调用：ser-finance（三方代付成功回调后）
为什么需要：
  · 冻结 → 审核通过 → 代付出款 → 出款成功 → 调用此接口
  · 和 WalletDebit 的区别：WalletDebit 从可用余额扣，DebitFrozen 从冻结余额扣
  · 操作：balance -= amount 且 frozenAmount -= amount

入参：
  · userId, currencyCode, amount, orderNo

出参：
  · balanceAfter, transactionId

关键约束：
  · 幂等：同一 orderNo 不重复执行
  · 校验：冻结金额必须 >= 扣除金额

说明：
  IDL设计/d-3.md 中最初没有单独定义 DebitFrozen，而是用 WalletDebit 覆盖。
  但提现的扣款操作逻辑上是"从冻结区扣除"而非"从可用余额扣除"，两者的
  数据库操作不同（WalletDebit 操作 balance，DebitFrozen 同时操作 balance 和 frozenAmount）。
  实现上可以在 WalletDebit 内部根据 bizType=提现 走不同分支，也可以独立方法。
  此处独立列出以明确语义，具体实现方式在编码阶段确定。
```

#### A6. ManualCredit — 人工加款

```
做什么：管理员审批通过后，往用户指定类型钱包加钱
谁调用：ser-finance（人工修正流程审批通过后）
为什么需要：
  · 客服处理用户投诉、系统补偿、运营活动手动发放等场景
  · 和 WalletCredit 的区别：walletType 可以是任意类型（1中心/2奖励/3主播/4代理）
  · WalletCredit 的 walletType 固定为中心钱包

入参：
  · userId, currencyCode, walletType（必须指定：1/2/3/4）, amount, orderNo, remark（必填）

出参：
  · balanceAfter, transactionId

关键约束：
  · remark 必填（人工操作必须有理由，审计要求）
  · walletType 由财务端指定，钱包不推断
```

#### A7. ManualDebit — 人工减款

```
做什么：管理员审批通过后，从用户指定类型钱包扣钱
谁调用：ser-finance（人工修正流程审批通过后）
为什么需要：
  · 用户违规扣款、异常数据修正、系统差错调整等场景

入参/出参/约束：同 ManualCredit 对称
```

### 3.3 B 组：余额查询（3 个）— 首期必须

#### B1. RpcGetBalance — 指定币种余额查询

```
做什么：查询用户在指定币种下的各钱包余额
谁调用：ser-finance（人工修正页面展示）、游戏（余额展示）、活动（奖励发放前检查）
为什么需要：
  · 这是最基础的"我有多少钱"查询
  · 各调用方在不同业务环节需要知道用户余额

入参：
  · userId        — 必填
  · currencyCode  — 必填

出参：
  · currencyCode
  · centerBalance   — 中心钱包余额
  · bonusBalance    — 奖励钱包余额
  · anchorBalance   — 主播钱包余额（非主播用户=0）
  · agentBalance    — 代理钱包余额（非代理用户=0）
  · frozenAmount    — 冻结总额
  · availableBalance — 可提现余额（center - frozen）
```

#### B2. RpcGetWalletInfo — 用户钱包完整信息

```
做什么：查询用户所有币种、所有类型钱包的完整信息
谁调用：ser-finance（人工修正页面，需要看到用户全貌）
为什么需要：
  · 管理员做人工加减款前，需要看到用户所有币种所有钱包的余额
  · 比 RpcGetBalance 更全面——不限定单个币种

入参：
  · userId        — 必填
  · currencyCode  — 选填（不传=返回所有币种）

出参：
  · list<WalletInfo> wallets  — 所有钱包详情列表
  · 每个 WalletInfo 含：walletId, userId, currencyCode, walletType, balance, frozenAmount
```

#### B3. CheckBalance — 余额充足性校验

```
做什么：检查用户余额是否满足某个金额要求（不扣款）
谁调用：游戏（下注前校验）、直播（送礼前校验）
为什么需要：
  · 调用方不想直接读余额再自己算，而是让钱包统一判断
  · 避免调用方自行计算时遗漏冻结金额的扣减

入参：
  · userId, currencyCode, amount（需要的金额）
  · walletType — 选填（不传=中心+奖励合计）

出参：
  · sufficient — bool，是否充足
  · available  — 当前可用余额
```

### 3.4 C 组：投注操作（2 个）— 预留

#### C1. BetDebit — 投注扣款（优先级扣款）

```
做什么：投注下单时扣款，先扣中心钱包，不足再扣奖励钱包
谁调用：游戏/投注服务
为什么独立（不用 WalletDebit）：
  · 有独特的"优先级扣款"逻辑——先中心后奖励
  · 需要返回 centerDeducted + bonusDeducted 两个金额（供后续返奖按比例分配）
  · WalletDebit 只操作单个钱包，无法满足

入参：
  · userId, currencyCode, amount（总投注金额）, orderNo

出参：
  · centerDeducted     — 从中心钱包扣除的金额
  · bonusDeducted      — 从奖励钱包扣除的金额
  · centerBalanceAfter — 中心钱包剩余
  · bonusBalanceAfter  — 奖励钱包剩余

业务规则示例：
  用户投注 100，中心钱包 80，奖励钱包 50
  → centerDeducted=80, bonusDeducted=20
  → centerRatio=0.8, bonusRatio=0.2（调用方需保存此比例）

关键约束：
  · 调用方（游戏服务）必须保存返回的扣款比例，用于后续 BetCredit 返奖
  · 两个钱包的扣款在同一个 DB 事务中
```

#### C2. BetCredit — 投注返奖（按比例返还）

```
做什么：赛事/游戏结算时返奖，按原投注扣款比例分别返还到中心和奖励钱包
谁调用：游戏/投注服务
为什么独立：
  · 需要 centerRatio + bonusRatio 参数来决定分配
  · 返奖不是简单加钱，而是"按比例加回原来扣款的两个钱包"

入参：
  · userId, currencyCode, amount（总返奖金额）, orderNo
  · centerRatio（如 "0.8"）, bonusRatio（如 "0.2"）

出参：
  · centerCredited, bonusCredited
  · centerBalanceAfter, bonusBalanceAfter

业务规则示例：
  返奖 150，centerRatio=0.8, bonusRatio=0.2
  → centerCredited=120, bonusCredited=30
  → 保证奖励钱包的钱不会"洗"进中心钱包
```

## 五、关键业务链路中的 RPC 调用

### 5.1 充值全链路

```
用户发起充值
  ↓
[C端] ser-wallet.CreateDeposit()
  │ ·调用 ser-finance.GetDepositMethods() → 获取充值方式
  │ ·调用 ser-finance.CreateDepositOrder() → 获取支付 URL
  │ ·返回支付链接给前端
  ↓
用户在三方支付完成付款
  ↓
三方回调 → ser-finance 回调处理
  │ ·验签、校验金额、更新订单状态
  ↓
[RPC] ser-finance → ser-wallet.WalletCredit() ← ★ 关键节点
  │ ·幂等检查（orderNo）
  │ ·中心钱包 balance += amount
  │ ·写入 wallet_transaction 流水
  ↓
如有奖金活动：
[RPC] ser-finance → ser-wallet.BonusCredit() ← ★ 关键节点
  │ ·奖励钱包 balance += bonusAmount
  │ ·写入 reward_record（含 auditMultiplier）
  ↓
充值完成
```

### 5.2 提现全链路

```
用户发起提现
  ↓
[C端] ser-wallet.CreateWithdraw()
  │ ·调用 ser-kyc.QueryKYCStatus() → 校验 KYC
  │ ·调用 ser-user.GetUserInfo() → 获取真实姓名
  │ ·校验可用余额 >= 提现金额 + 手续费
  ↓
[内部] ser-wallet.WalletFreeze() ← ★ 冻结
  │ ·frozenAmount += amount
  │ ·可用余额减少
  ↓
[RPC] ser-wallet → ser-finance.CreateWithdrawOrder()
  │ ·创建提现订单，进入审核流程
  ↓
审核环节（ser-finance 内部三级审核）
  ↓
┌─ 驳回路径 ─────────────────────────────┐
│ [RPC] ser-finance → ser-wallet.WalletUnfreeze() ← ★ 解冻
│   ·frozenAmount -= amount
│   ·用户余额恢复可用
└────────────────────────────────────────┘
┌─ 通过路径 ─────────────────────────────┐
│ ser-finance → 三方代付通道 → 出款成功回调
│   ↓
│ [RPC] ser-finance → ser-wallet.DebitFrozen() ← ★ 正式扣除
│   ·balance -= amount
│   ·frozenAmount -= amount
│   ·写入扣款流水
└────────────────────────────────────────┘
```

### 5.3 投注全链路（预留）

```
用户下注 100
  ↓
[RPC] 游戏服务 → ser-wallet.CheckBalance()
  ·返回：sufficient=true, available=100
  ↓
[RPC] 游戏服务 → ser-wallet.BetDebit() ← ★ 优先级扣款
  ·中心 80 扣 80，奖励 50 扣 20
  ·返回：centerDeducted=80, bonusDeducted=20
  ·游戏服务保存比例：center=0.8, bonus=0.2
  ↓
赛事/游戏进行 ...
  ↓
结算：用户赢 150
  ↓
[RPC] 游戏服务 → ser-wallet.BetCredit() ← ★ 按比例返奖
  ·传入：amount=150, centerRatio=0.8, bonusRatio=0.2
  ·中心入 120，奖励入 30
```

### 5.4 礼物全链路（预留）

```
用户送礼 50
  ↓
[RPC] 直播服务 → ser-wallet.CheckBalance()
  ↓
[RPC] 直播服务 → ser-wallet.GiftDebit() ← ★ 仅扣中心钱包
  ↓
[RPC] 直播服务 → ser-wallet.AnchorCredit() ← ★ 入主播钱包

```

## 六、接口契约关键约束

### 6.1 幂等性（最重要）

```
所有 A 组资金操作必须幂等。

幂等键 = orderNo（业务订单号）
规则：
  · 每次资金操作前，先查 wallet_transaction 表是否已存在该 orderNo
  · 如果已存在 → 直接返回之前的结果（不重复操作）
  · 如果不存在 → 执行操作 + 写入流水（在同一事务中）

为什么必须幂等：
  · 三方支付回调可能重发（HTTP 超时重试）
  · ser-finance 调用超时后可能重试
  · 网络抖动导致的重复 RPC 调用
  · 如果不幂等 → 用户被重复入账/扣款 → 资金事故
```

### 6.2 金额类型

```
IDL 层面：所有金额字段使用 string 类型
  · 入参示例："5000000.00000000"
  · 出参示例："4999000.00000000"

Service 层面：
  · 接收 string → decimal.NewFromString() 转为精确 decimal
  · 计算完成 → decimal.String() 转为 string 返回
  · 数据库存储：DECIMAL(20,8)

禁止事项：
  · ❌ 使用 float64 / double 进行金额计算
  · ❌ 在 RPC 传输中使用 i64 表示最小单位
```

### 6.3 错误码

```
ser-wallet RPC 错误码段：6022200 ~ 6022299（预留 100 个）

关键错误码（需要调用方程序逻辑判断的）：
  · 6022201 = 余额不足（InsufficientBalance）
  · 6022202 = 冻结金额不足（InsufficientFrozen）
  · 6022203 = 重复操作已处理（DuplicateOrder，幂等命中）
  · 6022204 = 币种不存在或未启用（InvalidCurrency）
  · 6022205 = 钱包不存在（WalletNotFound）
  · 6022206 = 金额非法（InvalidAmount，≤0 或精度超限）
  · 6022207 = 钱包类型不匹配（InvalidWalletType）
  · 6022208 = 转账规则不满足（TransferRuleViolation，稽核未完成等）

约定：
  · 6022203（DuplicateOrder）不算"错误"——调用方收到此码时应视为成功
  · 调用方需区分"业务失败"（余额不足等）和"系统错误"（RPC 超时等）
```

### 6.4 超时与重试

```
调用方配置建议：
  · RPC 超时：3s（资金操作不应超过 3s）
  · 读操作（查询/校验）：可设 1-2s
  · 写操作（入账/扣款/冻结）：3s + 最多 2 次重试（间隔 500ms, 1s）

钱包侧保证：
  · 所有写操作幂等 → 调用方可安全重试
  · 查询操作天然幂等

注意：
  · 现有工程 RPC 超时配置为 1200s（init.go），但这是全局默认值
  · 资金操作的调用方应该用自定义超时覆盖
```

## 八、与币种配置 RPC 的区分

### 8.1 两类 RPC 的关系

```
ser-wallet 对外 RPC 暴露：

├── wallet_rpc.thrift（18 个方法）
│   └── 钱包账户操作：入账/扣款/冻结/投注/礼物/奖励/转账/查询
│   └── 核心特征：涉及余额变动，需要事务和幂等
│
└── currency.thrift 中的 RPC 方法（6 个方法）
    └── 币种配置查询：币种信息/汇率/基准币种/校验
    └── 核心特征：只读查询，无状态变更，可缓存

两者共处一个 WalletService（同一进程），但职责完全不同。
在 service.thrift 的方法签名中通过注释分组区分。
```

### 8.2 为什么币种 RPC 不放在 wallet_rpc.thrift 中

```
1. 职责分离
   · wallet_rpc = 资金操作契约（改变余额状态）
   · currency RPC = 数据查询契约（只读）
   · 放一起会混淆语义

2. 演进独立性
   · 币种配置 RPC 未来可能独立为 ser-currency 服务
   · 保持 IDL 文件独立方便迁移

3. 对标现有工程
   · ser-item 的 GetItemsByType / BatchGetItemsByIds 就是定义在 item_publish.thrift 中
   · 不是所有对外 RPC 都要放在一个 _rpc.thrift 文件里
```

## 总结

ser-wallet 需要暴露 **25 个 RPC 方法**（wallet_rpc 19 个 + currency RPC 6 个），供 5 类调用方使用：

| 调用方 | 接口数 | 首期状态 |
|--------|--------|---------|
| ser-finance | 13 | 全部首期必须 |
| 游戏/投注 | 8 | 预留（首期定义接口） |
| 直播/礼物 | 4 | 预留 |
| 活动/任务 | 2 | 预留 |
| 代理模块 | 1 | 纯预留 |

**首期必须实现 16 个，首期定义+TODO 实现 6 个，纯预留 3 个。**

核心思路：钱包是所有资金操作的**唯一入口**——不管钱从哪来（充值/奖励/投注返奖/礼物打赏），也不管钱往哪去（提现/投注/送礼），都必须经过钱包的 RPC 接口。这保证了**余额一致性**和**流水可追溯**。
