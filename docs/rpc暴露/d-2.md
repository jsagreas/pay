# ser-wallet 对外暴露 RPC 接口分析

> 分析目标：梳理 ser-wallet 除 C端/B端 HTTP 接口外，需要对工程内其他模块暴露哪些 RPC 接口
> 分析依据：需求理解/d-2.md、依赖关系/d-2.md、链路关系/d-2.md、IDL设计/d-2.md、模块参考A/d-2.md、模块参考B/d-2.md、工程实际代码
> 分析范围：ser-wallet 作为被调用方，对外提供的 RPC 能力（不含 C/B 端接口）
> 文档状态：预评审阶段

---

## 一、为什么需要对外 RPC 接口

### 1.1 钱包不是孤岛

ser-wallet 在工程中不是单兵作战模块。从依赖关系分析可知：

```
                    ┌───────────────┐
                    │   ser-user    │
                    │ (用户注册触发  │
                    │  钱包初始化)   │
                    └───────┬───────┘
                            │ InitWallet
                            ▼
┌──────────────┐    ┌───────────────┐    ┌──────────────┐
│   财务管理    │───▶│  ser-wallet   │◀───│  活动/奖金    │
│ (充值入账/    │    │  (资金操作    │    │ (发放活动    │
│  提现冻解/    │    │   原语提供方)  │    │  奖励到账)   │
│  人工修正)    │    │              │    │              │
└──────────────┘    └───────┬───────┘    └──────────────┘
                            │
                    ┌───────▼───────┐
                    │   其他业务     │
                    │ (直播打赏/     │
                    │  商城消费等    │
                    │  未来扩展)     │
                    └───────────────┘
```

**核心定位**：ser-wallet 的 RPC 层是整个平台的**资金操作原语提供方**。任何涉及用户余额变动的业务，最终都必须通过 ser-wallet 的 RPC 接口完成。

### 1.2 调用方分析

| 调用方模块 | 调用场景 | 依赖强度 | 当前状态 |
|-----------|---------|---------|---------|
| **财务管理（ser-finance）** | 充值入账、提现冻结/解冻/确认扣款、人工加减款 | **强依赖** | 他人负责，需我方暴露RPC |
| **用户服务（ser-user）** | 用户注册时初始化钱包 | **强依赖** | 已存在，需我方暴露RPC |
| **活动/奖金系统** | 发放活动奖金到奖励钱包 | **中依赖** | 需求文档提及，具体模块待定 |
| **直播服务（ser-live）** | 打赏送礼扣款、主播收益上账 | **潜在依赖** | 工程已存在ser-live，功能待扩展 |
| **商城服务（ser-mall）** | 商品购买扣款 | **潜在依赖** | 工程已存在ser-mall，功能待扩展 |
| **定时任务（ser-cron）** | 汇率定时拉取、日终对账 | **弱依赖** | 工程已存在ser-cron |
| **任何需要查询币种/汇率的模块** | 获取币种列表、汇率换算 | **通用依赖** | 币种配置作为基础设施层 |

## 二、RPC 接口全景

基于需求文档、依赖关系、业务链路的综合分析，ser-wallet 需要对外暴露的 RPC 接口分为**三大类**：

```
ser-wallet 对外 RPC 能力
│
├── 第一类：资金操作原语（核心，写操作）
│   ├── CreditWallet      上账/加款
│   ├── DebitWallet        扣款/减款
│   ├── FreezeBalance      冻结余额
│   ├── UnfreezeBalance    解冻余额
│   └── ConfirmDebit       确认已冻结金额的扣除
│
├── 第二类：钱包管理能力（管理，写操作）
│   └── InitWallet         用户注册时初始化钱包
│
└── 第三类：查询能力（只读）
    ├── GetBalance          单用户余额查询
    ├── BatchGetBalance     批量用户余额查询
    ├── GetWalletFlow       资金流水查询
    ├── GetCurrencyList     获取可用币种列表
    ├── GetExchangeRate     获取币种汇率
    └── GetBaseCurrency     获取基准币种信息
```

**总计：12 个 RPC 接口**

---

## 三、第一类：资金操作原语（5个）

> 这是 ser-wallet 对外最核心的能力。平台内任何模块需要变动用户余额，都必须通过这5个原语接口。
> 设计原则：原子性、幂等性、可审计

### 3.1 CreditWallet — 上账/加款

```
接口名称：CreditWallet
操作类型：写（余额增加）
幂等要求：必须（refOrderNo 去重）
```

**谁会调用、什么场景**：

| 调用方 | 场景 | bizType | 目标钱包(walletType) |
|--------|------|---------|---------------------|
| 财务管理 | 充值支付成功后入账 | 1（充值） | 1（中心钱包） |
| 财务管理 | 人工加款审核通过 | 5（人工加款） | 1（中心钱包） |
| 财务管理 | 补单操作（充值到账但未入账） | 7（补单） | 1（中心钱包） |
| 活动系统 | 首充奖金/活动奖金发放 | 4（奖金） | 2（奖励钱包） |
| 兑换内部 | 法币→BSB兑换的BSB到账 | 3（兑换入） | 1（中心钱包） |
| 兑换内部 | 兑换赠送部分 | 4（奖金） | 2（奖励钱包） |
| 直播模块 | 主播收到打赏的收益上账 | 8（打赏收入）| 3（主播钱包） |

**需求文档依据**：
- 充值入账：需求5-1~5-7充值模块完整描述
- 人工加款：财务需求2-5人工修正（加款+减款+冻结+解冻四种操作）
- 活动奖金：需求5-6额外奖金选择规则（充值时可选额外奖金）
- 兑换赠送：需求6兑换模块（兑换BSB时赠送部分进奖励钱包）

**接口契约骨架**：

| 字段 | 类型 | 必传 | 说明 |
|------|------|------|------|
| userId | i64 | 是 | 目标用户ID |
| currencyCode | string | 是 | 币种代码（BSB/VND/IDR/THB） |
| walletType | i32 | 是 | 钱包类型 1中心 2奖励 3主播 4代理 5场馆 |
| amount | i64 | 是 | 金额（最小单位整数，必须>0） |
| bizType | i32 | 是 | 业务类型（区分上账原因） |
| refOrderNo | string | 是 | 关联业务订单号（幂等键） |
| remark | string | 否 | 备注信息 |

**返回**：操作后余额（afterBalance）

**关键设计要点**：
1. **幂等性**：相同 refOrderNo 重复调用不重复上账，返回首次操作结果
2. **流水记录**：每次上账必须写入 wallet_flow 表，记录前后余额
3. **并发控制**：同用户同币种操作需要分布式锁或乐观锁保护
4. **参考工程A验证**：p9-live-backend 的 ModifyBalance 同样采用 orderNo 幂等 + Redisson 分布式锁

### 3.2 DebitWallet — 扣款/减款

```
接口名称：DebitWallet
操作类型：写（余额减少）
幂等要求：必须（refOrderNo 去重）
```

**谁会调用、什么场景**：

| 调用方 | 场景 | bizType | 扣款顺序 |
|--------|------|---------|---------|
| 财务管理 | 人工减款审核通过 | 6（人工减款） | 中心钱包优先 |
| 兑换内部 | 法币→BSB兑换扣除法币 | 3（兑换出） | 中心钱包优先 |
| 直播模块 | 用户打赏送礼扣款 | 9（打赏支出） | 中心优先→奖励其次 |
| 商城模块 | 商品购买扣款 | 10（商城消费） | 中心优先→奖励其次 |

**需求文档依据**：
- 人工减款：财务需求2-5人工修正
- 兑换扣款：需求6兑换模块
- 消费扣款顺序：需求4-1钱包总体设计明确"消费优先扣中心钱包"

**接口契约骨架**：与 CreditWallet 结构一致（userId、currencyCode、walletType、amount、bizType、refOrderNo、remark）

**关键设计要点**：
1. **防超扣**：扣款前检查可用余额（balance - frozenAmount >= amount），不足则拒绝
2. **扣款顺序**：按需求规则——中心钱包优先扣，不足部分扣奖励钱包
3. **幂等性**：同 CreditWallet
4. **参考工程A验证**：p9-live-backend 的资金操作也区分中心/奖励的优先级扣款

---

### 3.3 FreezeBalance — 冻结余额

```
接口名称：FreezeBalance
操作类型：写（余额状态变更：可用→冻结）
幂等要求：必须（refOrderNo 去重）
```

**谁会调用、什么场景**：

| 调用方 | 场景 | 说明 |
|--------|------|------|
| 钱包自身 | 用户提交提现订单 | 冻结提现金额，防止提现期间余额被消费 |
| 财务管理 | 人工冻结操作 | 风控场景下冻结用户资金 |

**需求文档依据**：
- 提现流程：需求7-1~7-6提现模块——用户提交提现→冻结金额→进入审核流程
- 人工修正：财务需求2-5人工修正包含冻结操作

**接口契约骨架**：

| 字段 | 类型 | 必传 | 说明 |
|------|------|------|------|
| userId | i64 | 是 | 目标用户ID |
| currencyCode | string | 是 | 币种代码 |
| amount | i64 | 是 | 冻结金额 |
| refOrderNo | string | 是 | 关联提现订单号（幂等键） |

**关键设计要点**：
1. **冻结分配**：冻结金额按消费顺序分配（先冻中心、不足冻奖励），需记录每种钱包冻结了多少
2. **冻结≠扣款**：冻结后 balance 不变，但 frozenAmount 增加，availableBalance 减少
3. **记录冻结明细**：写入 freeze_detail 表，后续解冻/确认扣款需要依据此明细
4. **参考工程A验证**：p9-live-backend 采用冻结三阶段（freeze→unfreeze/confirm），我方设计一致

---

### 3.4 UnfreezeBalance — 解冻余额

```
接口名称：UnfreezeBalance
操作类型：写（余额状态变更：冻结→可用）
幂等要求：必须（refOrderNo 去重）
```

**谁会调用、什么场景**：

| 调用方 | 场景 | 说明 |
|--------|------|------|
| 财务管理 | 提现风控审核驳回 | 解冻全部金额，恢复用户可用余额 |
| 财务管理 | 提现财务审核驳回 | 同上 |
| 财务管理 | 三方出款失败 | 出款通道返回失败，解冻全部金额 |
| 财务管理 | 人工解冻操作 | 风控场景下解冻 |

**需求文档依据**：
- 依赖关系/d-2.md 提现链路步骤9b：审核驳回或出款失败→UnfreezeBalance
- 财务需求2-5人工修正包含解冻操作

**接口契约骨架**：

| 字段 | 类型 | 必传 | 说明 |
|------|------|------|------|
| userId | i64 | 是 | 目标用户ID |
| currencyCode | string | 是 | 币种代码 |
| refOrderNo | string | 是 | 关联提现订单号（通过此查找冻结明细） |

**关键设计要点**：
1. **根据冻结明细还原**：按 refOrderNo 查找 freeze_detail，各钱包各还原对应冻结金额
2. **状态校验**：只有"已冻结"状态的记录才能解冻，防止重复解冻
3. **与 ConfirmDebit 互斥**：同一笔冻结只能走解冻或确认扣款，不能两者都执行

---

### 3.5 ConfirmDebit — 确认已冻结金额的扣除

```
接口名称：ConfirmDebit
操作类型：写（余额状态变更：冻结→实际扣除）
幂等要求：必须（refOrderNo 去重）
```

**谁会调用、什么场景**：

| 调用方 | 场景 | 说明 |
|--------|------|------|
| 财务管理 | 三方出款成功 | 通道回调成功，将冻结金额实际从余额扣除 |

**需求文档依据**：
- 依赖关系/d-2.md 提现链路步骤9a：三方出款成功→ConfirmDebit
- 这是冻结三阶段的最终环节

**接口契约骨架**：与 UnfreezeBalance 相同（userId、currencyCode、refOrderNo）

**关键设计要点**：
1. **根据冻结明细扣款**：按 refOrderNo 查找 freeze_detail，将冻结金额从 balance 中实际扣除
2. **冻结→0**：frozenAmount 减少，balance 同步减少
3. **写流水**：扣款成功后写入 wallet_flow
4. **参考工程A验证**：p9-live-backend 的 ConfirmFrozenDeduction 同样的三阶段终态

---

### 3.6 冻结三阶段完整链路

```
用户提交提现 → FreezeBalance（冻结）
                    │
        ┌───────────┴───────────┐
        │                       │
  审核通过+出款成功          审核驳回/出款失败
        │                       │
  ConfirmDebit（确认扣除）  UnfreezeBalance（解冻）
        │                       │
  balance 减少              balance 不变
  frozenAmount 减少         frozenAmount 减少
  写扣款流水                恢复 availableBalance
```

**为什么采用三阶段而非直接扣款**：
- 提现有多级审核（风控→财务→出款），审核可能长达数小时甚至天
- 冻结期间用户不能用这笔钱消费（防止余额不足），但如果审核不过可以全额返还
- 参考工程A的冻结三阶段已在生产验证，设计成熟可靠

## 四、第二类：钱包管理能力（1个）

### 4.1 InitWallet — 钱包初始化

```
接口名称：InitWallet
操作类型：写（创建钱包记录）
幂等要求：必须（userId 去重，重复调用不报错）
```

**谁会调用、什么场景**：

| 调用方 | 场景 | 说明 |
|--------|------|------|
| ser-user | 用户注册成功后 | 为新用户创建所有启用币种的钱包记录 |

**需求文档依据**：
- 依赖关系/d-2.md Phase3：用户注册→ser-user调用ser-wallet.InitWallet
- 需求4-1钱包总体设计：每个用户每个币种都有独立的钱包

**接口契约骨架**：

| 字段 | 类型 | 必传 | 说明 |
|------|------|------|------|
| userId | i64 | 是 | 新注册用户ID |

**内部逻辑**：
1. 查询当前所有启用状态的币种（调内部币种配置能力）
2. 为该用户的每个币种创建钱包记录（中心钱包、奖励钱包等子钱包，余额初始为0）
3. 设置默认币种偏好（按系统配置的默认币种）

**关键设计要点**：
1. **幂等性**：已有钱包的用户重复调用不报错，不创建重复记录
2. **后续新增币种**：如果系统启用了新币种，对存量用户的钱包初始化是另一个场景（可能走批处理或懒加载），此接口只负责注册时刻
3. **参考工程B验证**：tg-services 的 wallet-server 通过 POST /wallet/create 为用户创建 TRON 地址（1:1映射），我方类似但创建的是多币种钱包记录

### 5.4 GetCurrencyList — 获取可用币种列表

```
接口名称：GetCurrencyList（RPC版）
操作类型：只读
```

**谁会调用、什么场景**：

| 调用方 | 场景 |
|--------|------|
| 财务管理 | 配置充值/提现方式时需要知道有哪些币种 |
| 财务管理 | 统计报表按币种维度展示 |
| ser-user | 用户注册时确定为哪些币种创建钱包 |
| 任何业务模块 | 需要知道平台支持哪些币种 |

**接口契约骨架**：

| 字段 | 类型 | 必传 | 说明 |
|------|------|------|------|
| currencyType | i32 | 否 | 筛选类型（1法币 2加密货币 3平台货币），不传返回全部启用币种 |

**返回**：币种列表（代码、名称、类型、图标、小数位、千分位符、小数点符）

**与 B 端 PageCurrency 的区别**：
- B端接口返回管理视角数据（含禁用的、含汇率详情、分页）
- RPC接口返回业务视角数据（仅启用的、精简字段、全量无分页）

## 七、与业务链路的关系映射

### 7.1 充值链路中涉及的 RPC

```
充值链路：用户提交充值 → ... → 支付成功 → 入账

RPC接口参与点：
  充值回调处理时：
    └── CreditWallet（中心钱包上账，bizType=1）
  如有额外奖金：
    └── CreditWallet（奖励钱包上账，bizType=4）
  页面展示时：
    ├── GetCurrencyList（币种列表）
    └── GetExchangeRate（汇率信息，仅BSB充值时）
```

### 7.2 提现链路中涉及的 RPC

```
提现链路：用户提交提现 → 冻结 → 审核 → 出款 → 确认/驳回

RPC接口参与点：
  用户提交时：
    └── FreezeBalance（冻结提现金额）
  审核驳回/出款失败时：
    └── UnfreezeBalance（解冻返还）
  出款成功时：
    └── ConfirmDebit（确认扣除已冻结金额）
  页面展示时：
    ├── GetCurrencyList
    └── GetExchangeRate（仅BSB提现时）
```

### 7.3 兑换链路中涉及的 RPC

```
兑换链路：用户法币→BSB，钱包内部闭环

RPC接口参与点：
  兑换执行时（内部调用，非跨模块RPC）：
    ├── DebitWallet（扣法币中心钱包）
    ├── CreditWallet（加BSB中心钱包）
    └── CreditWallet（赠送部分→BSB奖励钱包）
  页面展示时：
    ├── GetCurrencyList
    └── GetExchangeRate（兑换汇率）
```

### 7.4 人工修正链路中涉及的 RPC

```
人工修正：B端运营 → 财务管理 → 调用钱包RPC

四种操作：
  加款：CreditWallet
  减款：DebitWallet
  冻结：FreezeBalance
  解冻：UnfreezeBalance
辅助查询：
  展示余额：GetBalance
  汇率换算：GetExchangeRate
```

## 十一、注意事项与风险

### 11.1 资金操作接口的安全要求

| 要求 | 说明 |
|------|------|
| **幂等性** | 所有写接口必须基于 refOrderNo 幂等，防止重复操作 |
| **原子性** | CreditWallet/DebitWallet 的余额变更和流水写入必须在同一事务 |
| **并发控制** | 同用户同币种的余额操作必须串行化（分布式锁或乐观锁） |
| **防超扣** | DebitWallet 和 FreezeBalance 必须检查可用余额 |
| **审计追溯** | 每次操作写 wallet_flow，记录操作前后余额、关联订单号 |
| **参数校验** | amount 必须>0，userId/currencyCode 必须有效 |

### 11.2 接口契约稳定性

RPC 接口一旦被其他模块调用，修改成本很高（需要协调多方）。因此：

- **接口签名要稳定**：Req/Resp 的字段设计要预留扩展空间（optional字段）
- **语义要明确**：每个接口做且只做一件事，不要混合多种操作
- **版本意识**：后续如需变更，通过新增字段而非修改已有字段

### 11.3 当前未确定的事项

| 事项 | 说明 | 建议处理 |
|------|------|---------|
| bizType 枚举值完整列表 | 目前已知1-7，直播/商城等场景的编号待定 | 预留空间，后续按需添加 |
| walletType 是否新增 | 主播/代理/场馆钱包是预留，何时启用待定 | 代码支持，但业务上暂不开放 |
| 活动/奖金系统是否有独立模块 | 还是财务管理内嵌 | 不影响RPC设计，都通过CreditWallet调用 |
| 日终对账接口是否需要 | 目前未列出 | 初期可通过GetWalletFlow查询实现，后续按需补充 |
