## 一、为什么需要暴露RPC接口

### 1.1 钱包不是孤岛

钱包模块（ser-wallet）在整个平台中的定位是**资金操作的执行者**——它不做业务决策（审核、风控、通道匹配），但所有涉及用户资金变动的操作最终都要经过它。

```
                    ┌────────────────────────────────┐
                    │         ser-wallet              │
                    │   "我只管钱的加减冻结解冻"         │
                    │   "谁让我做，得给我订单号"         │
                    │   "同一个订单号来两次？幂等退回"    │
                    └──────┬─────────────┬────────────┘
                           │             │
              ┌────────────┘             └─────────────┐
              ▼                                        ▼
    ┌─────────────────┐                   ┌─────────────────┐
    │   ser-finance    │                   │   其他模块       │
    │  "我做决策和审核"  │                   │  （未来扩展）     │
    │  "决定加还是减"    │                   │  直播/商城/投注   │
    └─────────────────┘                   └─────────────────┘
```

### 1.2 B端和C端走HTTP网关，模块间走RPC

工程架构决定了三种调用路径：

| 调用路径 | 入口 | 协议 | 示例 |
|----------|------|------|------|
| B端管理员操作 | gate-back → ser-wallet | HTTP转RPC | 币种CRUD、汇率日志 |
| C端用户操作 | gate-font → ser-wallet | HTTP转RPC | 余额查询、充值、提现、兑换 |
| **模块间内部调用** | ser-finance → ser-wallet | **纯RPC** | **入账、冻结、解冻、扣款** |

第三种路径就是本文要分析的重点——这些RPC接口**不经过网关、不对外暴露、只供工程内部其他服务调用**。

### 1.3 分析的核心原则

- **以业务链路为驱动**：每个RPC接口的存在必须由一条真实的业务链路"倒逼"出来
- **有需求文档依据**：不凭空设想，每个接口对应产品需求中的具体场景
- **不过度设计**：当前需求没有的不强加，留弹性空间即可

## 二、RPC接口总览

基于需求文档和跨模块依赖分析，ser-wallet 需要对外暴露 **9个RPC方法**：

| 序号 | 方法名 | 一句话功能 | 主要调用方 | 触发场景 |
|------|--------|----------|-----------|---------|
| R1 | **WalletCredit** | 入账到中心钱包 | ser-finance | 充值成功/补单入账 |
| R2 | **WalletCreditReward** | 奖金入账到奖励钱包 | ser-finance | 充值赠送/活动奖金 |
| R3 | **WalletFreeze** | 冻结指定子钱包余额 | ser-finance | 用户发起提现 |
| R4 | **WalletUnfreeze** | 解冻归还冻结余额 | ser-finance | 提现审核驳回/出款失败 |
| R5 | **WalletDeduct** | 最终扣除冻结资金 | ser-finance | 提现出款成功 |
| R6 | **WalletManualAdd** | 人工加款（按子钱包分别加） | ser-finance | B端人工加款审核通过 |
| R7 | **WalletManualSub** | 人工减款（按子钱包分别减） | ser-finance | B端人工减款审核通过 |
| R8 | **GetUserBalance** | 查询用户各子钱包余额 | ser-finance | 人工修正弹窗展示余额 |
| R9 | **GetCurrencyConfig** | 获取币种配置信息 | ser-finance | 统计报表汇率换算/展示 |

**为什么是9个，不多不少？** 下一章逐一给出每个接口的分析依据。

## 三、逐个接口的详细分析与依据

### R1：WalletCredit — 入账到中心钱包

**功能**：将指定金额加入用户指定币种的中心钱包可用余额，同时写入资金流水。

**为什么需要这个RPC**：

钱包自己不做充值回调——充值流程中，用户支付完成后，三方支付通道的回调到达的是**财务模块**（财务负责通道配置和对账）。财务验证金额无误后，必须调用钱包的RPC来执行实际入账。

**需求依据**：
- 财务需求文档 2-3 充值记录-1 充值流程："订单状态改为支付成功，打款到对应用户上分"
- 充值链路分析（依赖关系/c-3.md 第3.1节）：三方通道回调 → 财务确认 → 调用钱包 CreditWallet 入账

**触发场景**（3种）：

| 场景 | orderNo前缀 | 说明 |
|------|------------|------|
| 充值成功入账 | C（充值单号） | 三方通道回调确认支付成功 |
| 充值补单入账 | B（补单号） | B端管理员审核通过充值补单 |
| 人工加款（中心钱包部分） | — | 由R6 ManualAdd覆盖，此处不重复 |

**接口行为**：
- 根据 `userId + currencyCode` 定位中心钱包记录
- 增加可用余额 `available_balance += amount`
- 写入资金流水（flow_type=入账，related_order_no=orderNo）
- **幂等**：同一 `orderNo` 重复调用直接返回成功，不重复入账

---

### R2：WalletCreditReward — 奖金入账到奖励钱包

**功能**：将奖金金额加入用户指定币种的奖励钱包，同时记录流水倍数要求。

**为什么单独一个接口（而非复用R1）**：

1. 中心钱包和奖励钱包是独立的子钱包，余额分开管理
2. 奖金入账附带**流水倍数要求**（turnoverMulti），这是中心钱包入账没有的参数
3. 奖金来源多样（充值赠送、活动发放），需要单独的流水类型标识

**需求依据**：
- 需求文档 2-1 资金钱包定义："奖励钱包：一定模板提现水到提现"
- 需求文档 5-2 充值奖金选择："选择50%奖金，最低BSB 1000"——充值赠送的奖金部分入奖励钱包
- 需求文档 6-1 兑换模块："赠送部分→奖励钱包，需完成X倍流水"

**触发场景**（3种）：

| 场景 | 说明 |
|------|------|
| 充值赠送奖金 | 用户充值时选择了奖金活动，充值金额入中心钱包（R1），赠送金额入奖励钱包（R2） |
| 兑换赠送部分 | 法币兑换BSB时的赠送部分入奖励钱包（此场景钱包本地执行，不走RPC） |
| 活动/任务奖励 | 运营活动发放奖金（如果有独立活动模块，会调用此RPC） |

**注意**：兑换的赠送是钱包内部操作（一个事务里扣法币、加BSB、加奖励），不需要走RPC。此接口主要供充值赠送和外部活动模块调用。

---

### R3：WalletFreeze — 冻结余额

**功能**：将指定子钱包的可用余额转移到冻结余额，表示"这笔钱已被锁定，不可再使用"。

**为什么需要这个RPC**：

提现流程的第一步是**冻结**——用户提交提现后，提现金额从可用余额移入冻结余额，确保在审核期间这笔钱不会被花掉。

**需求依据**：
- 财务需求文档 2-4 提现记录-1 提现流程：用户提交提现 → 冻结相应金额
- 提现链路分析（依赖关系/c-3.md 第3.2节）：步骤⑨"钱包执行FreezeBalance → 冻结₫7,000,000"
- 需求文档 7-1 提现模块：用户确认提现后进入"提现处理中"状态

**触发场景**：

| 场景 | 触发方 | 说明 |
|------|--------|------|
| 用户提交提现 | ser-wallet自身（C端接口内部调用，非RPC） | 钱包的C端CreateWithdrawOrder方法内部直接调用冻结逻辑 |
| 财务发起冻结 | ser-finance（RPC） | 如果提现创建由财务侧发起（某些架构下） |

**重要说明**：在当前设计中，冻结操作发生在钱包的C端`CreateWithdrawOrder`方法内部——用户提交提现 → 钱包先本地冻结 → 再RPC调用财务.SubmitWithdrawOrder提交审核。所以冻结更多是**钱包内部执行**，暴露RPC是为了：
1. 财务模块在特殊场景下也能主动发起冻结（如风控强制冻结）
2. 保持接口完整性——冻结/解冻/扣除三件套必须同时暴露

**接口行为**：
- `available_balance -= amount`
- `frozen_balance += amount`
- 写入冻结记录（freeze_record表，含 orderNo 关联）
- **前置校验**：可用余额 ≥ 冻结金额，不足则拒绝

---

### R4：WalletUnfreeze — 解冻归还

**功能**：将之前冻结的余额归还到可用余额——"提现没成功，钱退给你"。

**为什么需要这个RPC**：

提现审核有多个环节可能驳回，驳回时必须把冻结的钱还给用户。这个"还钱"操作的决策权在财务模块（因为审核在财务端），执行权在钱包模块。

**需求依据**：
- 财务需求文档 2-4 提现记录-1 提现流程："不通过→解冻用户冻结的金额，退回钱包余额"
- 提现链路分析（依赖关系/c-3.md 第3.2节）：步骤⑫~⑬"驳回→UnfreezeBalance"，步骤⑮"出款失败→UnfreezeBalance"

**触发场景**（3种）：

| 场景 | 触发方 | 说明 |
|------|--------|------|
| 风控审核驳回 | ser-finance | 风控判定该笔提现异常 |
| 财务审核驳回 | ser-finance | 财务主管审核不通过 |
| 代付通道出款失败 | ser-finance | 通道出款失败回调 |

**接口行为**：
- 根据 `orderNo` 查找冻结记录
- `frozen_balance -= 冻结记录金额`
- `available_balance += 冻结记录金额`
- 更新冻结记录状态为"已解冻"
- **幂等**：同一 orderNo 已解冻的，重复调用直接返回成功

---

### R5：WalletDeduct — 最终扣除冻结资金

**功能**：将之前冻结的余额真正扣除——"提现出款成功，这笔钱真正离开了用户账户"。

**为什么需要这个RPC**：

提现出款成功后，冻结的资金不再需要还给用户，而是真正从账户中移除。

**需求依据**：
- 财务需求文档 2-4 提现记录-1 提现流程：代付通道出款成功 → 确认扣除冻结资金
- 提现链路分析（依赖关系/c-3.md 第3.2节）：步骤⑮"出款成功→ConfirmFrozenDeduction"

**触发场景**：

| 场景 | 触发方 | 说明 |
|------|--------|------|
| 代付通道出款成功回调 | ser-finance | 通道确认资金已到达用户银行/钱包 |

**接口行为**：
- 根据 `orderNo` 查找冻结记录
- `frozen_balance -= 冻结记录金额`（资金真正离开系统）
- 写入资金流水（flow_type=提现扣除）
- 更新冻结记录状态为"已扣除"
- **幂等**：同一 orderNo 已扣除的，重复调用直接返回成功

### R3/R4/R5 三件套的关系

```
                              ┌─────────────┐
              用户提现         │  冻结余额    │
  可用余额 ─────R3─────────►  │  (锁住不动)   │
              FreezeBalance   └──────┬──────┘
                                     │
                            ┌────────┴────────┐
                            │                 │
                       审核驳回/            出款成功
                       出款失败
                            │                 │
                       R4 Unfreeze       R5 Deduct
                            │                 │
                            ▼                 ▼
                     ┌──────────┐      ┌──────────┐
                     │ 可用余额  │      │  账户外   │
                     │ (钱还回来)│      │ (钱出去了)│
                     └──────────┘      └──────────┘
```

这三个接口**必须同时暴露**——冻结了就必须有解冻或扣除的出口，否则冻结的钱会永远锁死。

---

### R6：WalletManualAdd — 人工加款

**功能**：B端管理员审核通过后，按子钱包类型分别加入指定金额。

**为什么需要这个RPC（而非复用R1+R2组合）**：

1. 人工加款一次操作涉及**最多4个子钱包**同时加款（中心/奖励/主播/代理），需要在一个事务中完成
2. 有独立的修正原因类型（correctionReason）和审计需求
3. 订单号格式独立（A+16位），与充值/奖金的订单号区分

**需求依据**：
- 财务需求文档 2-5 人工修正-2 人工加款：管理员填写各子钱包加款金额 → 审核通过 → 执行加款
- 人工修正链路分析（依赖关系/c-3.md 第3.4节）：财务 → 钱包.CreditWallet（本文合并为ManualAdd）

**触发场景**：

| 场景 | 说明 |
|------|------|
| 运营补偿 | 用户投诉后的补偿性加款 |
| 系统异常修正 | 充值入账异常的手动补录 |
| 活动补发 | 活动奖金未自动发放的手动补发 |

**接口行为**：
- 按传入的各子钱包金额分别执行加款
- 未传或为"0"的子钱包不操作
- 一个事务内完成所有子钱包的加款 + 流水写入
- **幂等**：同一 `orderNo`（A+16位）重复调用不重复加款

---

### R7：WalletManualSub — 人工减款

**功能**：B端管理员审核通过后，按子钱包类型分别扣减指定金额。

**为什么需要这个RPC**：

与人工加款对称。人工减款有一个特殊规则：**余额不足时实际扣款=当前余额**（而非拒绝操作）。

**需求依据**：
- 财务需求文档 2-5 人工修正-3 人工减款：减款金额 > 当前余额时，实际扣款 = 当前余额
- IDL设计方案 wallet_rpc.thrift：WalletManualSubResp 返回 `actualXxxAmount` 字段

**触发场景**：

| 场景 | 说明 |
|------|------|
| 异常资金回收 | 系统错误多入的资金扣回 |
| 违规处罚 | 用户违规行为的资金扣减 |
| 手动调账 | 对账发现差异后的修正 |

**接口行为**：
- 按传入的各子钱包金额分别执行扣减
- **特殊逻辑**：如果某子钱包余额 < 期望扣减金额，则实际扣减 = 当前余额（扣到0）
- 响应中返回每个子钱包的**实际扣减金额**，让财务侧知道实际扣了多少
- **幂等**：同一 `orderNo`（M+16位）重复调用不重复扣减

---

### R8：GetUserBalance — 查询用户余额

**功能**：查询指定用户指定币种下所有子钱包的余额（可用+冻结）。

**为什么需要这个RPC**：

财务模块在执行人工修正时，需要展示用户当前各子钱包的余额——管理员看到余额后才能决定加多少或减多少。

**需求依据**：
- 财务需求文档 2-5 人工修正-2 人工加款：创建弹窗中显示"当前余额"列
- 财务需求文档 2-5 人工修正-3 人工减款：创建弹窗中显示"当前余额"列
- 人工修正链路分析（依赖关系/c-3.md 第3.4节）：步骤②"财务 → 钱包.GetBalance"

**触发场景**：

| 场景 | 调用方 | 说明 |
|------|--------|------|
| 人工加/减款创建弹窗 | ser-finance | 展示用户各子钱包当前余额 |
| 风控审核查看 | ser-finance（或未来风控模块） | 查看用户资金状态 |
| 数据统计/报表 | ser-finance | 用户资金快照 |

**接口行为**：
- 纯查询，不修改任何数据
- 返回：userId、currencyCode、子钱包列表（每个含walletType、availableBalance、frozenBalance）、总可用、总冻结
- 无幂等要求（只读操作）

---

### R9：GetCurrencyConfig — 获取币种配置

**功能**：获取指定币种的完整配置信息（名称、类型、符号、精度、千分位、汇率、状态）。

**为什么需要这个RPC**：

币种配置是ser-wallet负责管理的数据。其他模块（特别是财务模块）在展示金额、做汇率换算时，需要知道币种的精度规则和当前汇率。

**需求依据**：
- 需求文档 3-3 后台币种配置：千分位符号、小数点符号、精度——这些由币种配置模块管理
- 财务需求文档 2-3 充值记录列表：第22-24列"基准币种订单金额/基准币种活动优惠/基准币种手续费"需要汇率换算
- 依赖关系分析（依赖关系/c-3.md 第10.3节）：财务 → 币种配置.GetPlatformRate/ConvertToBaseCurrency

**触发场景**：

| 场景 | 调用方 | 说明 |
|------|--------|------|
| 统计报表汇率换算 | ser-finance | 将各币种金额折算为基准币种USDT |
| 充值/提现记录展示 | ser-finance | "基准币种订单金额"等字段需要币种精度和汇率 |
| 金额格式化 | 任何需要展示金额的模块 | 精度、千分位、符号 |

**接口行为**：
- 纯查询，不修改任何数据
- 返回：币种代码、名称、类型（法币/加密/平台）、符号、精度、千分位符号、小数点符号、平台汇率、入款汇率、出款汇率、状态
- 无幂等要求（只读操作）
- 建议调用方做本地缓存（币种配置不会频繁变动）

## 五、业务链路驱动的RPC推导

以下从每条核心业务链路出发，推导出需要的RPC接口，证明9个接口"不多不少"。

### 5.1 充值链路 → 需要 R1 + R2

```
用户选择充值 → 钱包创建订单(本地) → 三方通道支付 → 通道回调
                                                        │
                                                        ▼
                                              ┌─────────────────┐
                                              │   ser-finance    │
                                              │  验证金额一致性   │
                                              └────────┬────────┘
                                                       │
                                          ┌────────────┴────────────┐
                                          ▼                         ▼
                                   【R1 WalletCredit】       【R2 WalletCreditReward】
                                   充值金额→中心钱包          赠送金额→奖励钱包
                                   （每笔必调）                （用户选了奖金才调）
```

**推导结论**：充值链路需要 R1（必须）+ R2（条件触发）= 2个RPC。

### 5.2 提现链路 → 需要 R3 + R4 + R5

```
用户提交提现 → 钱包冻结余额(本地R3逻辑) → 提交财务审核
                                                  │
                                    ┌──────────────┼──────────────┐
                                    ▼              ▼              ▼
                              风控审核驳回      财务审核驳回    出款处理
                                    │              │              │
                                    ▼              ▼         ┌────┴────┐
                             【R4 Unfreeze】  【R4 Unfreeze】 │         │
                              解冻归还         解冻归还      成功      失败
                                                              │         │
                                                              ▼         ▼
                                                        【R5 Deduct】 【R4 Unfreeze】
                                                        最终扣除       解冻归还
```

**推导结论**：提现链路需要 R3（冻结）+ R4（解冻，3个触发点）+ R5（扣除）= 3个RPC。
其中R3在当前设计中主要是钱包内部执行，但作为RPC暴露是完整性需要。

### 5.3 人工修正链路 → 需要 R6 + R7 + R8（+ R1/R2复用）

```
B端管理员创建修正单
         │
         ├── 充值补单 ── 审核通过 → 【R1 WalletCredit】(补单金额) + 【R2 CreditReward】(补单赠送)
         │
         ├── 人工加款 ── 查余额【R8】→ 填写各子钱包金额 → 审核通过 → 【R6 ManualAdd】
         │
         └── 人工减款 ── 查余额【R8】→ 填写各子钱包金额 → 审核通过 → 【R7 ManualSub】
```

**推导结论**：人工修正链路需要 R6 + R7 + R8 = 3个RPC（加上复用R1+R2）。

### 5.4 统计报表 → 需要 R9

```
B端财务报表页面
         │
         └── 各币种金额折算为基准币种USDT
                    │
                    ▼
             【R9 GetCurrencyConfig】
              获取币种精度、汇率信息
```

**推导结论**：统计报表需要 R9 = 1个RPC。

### 5.5 汇总

| 业务链路 | 涉及的RPC | 数量 |
|----------|----------|------|
| 充值 | R1, R2 | 2 |
| 提现 | R3, R4, R5 | 3 |
| 人工修正 | R6, R7, R8（复用R1, R2） | 3（+2复用） |
| 统计报表 | R9 | 1 |
| **合计（去重）** | **R1~R9** | **9** |

## 六、RPC接口的IDL定义回顾

以下是 wallet_rpc.thrift 中已设计的9个接口的数据结构摘要（完整定义见 IDL设计/tmp-1.md 第十一章）：

### 6.1 资金操作类（R1~R7）

| 接口 | 请求关键字段 | 响应关键字段 |
|------|-------------|-------------|
| R1 WalletCredit | userId, currencyCode, amount, orderNo, remark? | success, newBalance |
| R2 WalletCreditReward | userId, currencyCode, amount, orderNo, turnoverMulti?, remark? | success |
| R3 WalletFreeze | userId, currencyCode, walletType, amount, orderNo | success, freezeRecordId |
| R4 WalletUnfreeze | orderNo, remark? | success |
| R5 WalletDeduct | orderNo, remark? | success |
| R6 WalletManualAdd | userId, currencyCode, orderNo, centerAmount?, rewardAmount?, streamerAmount?, agentAmount?, correctionReason?, remark? | success |
| R7 WalletManualSub | userId, currencyCode, orderNo, centerAmount?, rewardAmount?, streamerAmount?, agentAmount?, correctionReason?, remark? | success, actualCenterAmount?, actualRewardAmount?, actualStreamerAmount?, actualAgentAmount? |

### 6.2 查询类（R8~R9）

| 接口 | 请求关键字段 | 响应关键字段 |
|------|-------------|-------------|
| R8 GetUserBalance | userId, currencyCode | userId, currencyCode, subWallets[](walletType, walletTypeName, availableBalance, frozenBalance), totalAvailable, totalFrozen |
| R9 GetCurrencyConfig | currencyCode | data(CurrencyConfigInfo): currencyCode, currencyName, currencyType, currencySymbol, precision, thousandsSep, decimalSep, platformRate, depositRate, withdrawRate, status |

### 6.3 设计要点

**1. R4/R5 只需 orderNo，不需要 userId/currencyCode/amount**

为什么？因为冻结时（R3）已经创建了冻结记录，解冻/扣除时只需通过 orderNo 找到对应的冻结记录，冻结记录中包含了所有需要的信息。这简化了财务模块的调用逻辑——财务只需要记住"提现订单号"即可。

**2. R7 ManualSub 响应返回实际扣减金额**

需求文档明确："减款金额 > 当前余额时，实际扣款 = 当前余额"。所以财务需要知道实际扣了多少（可能与请求金额不同），响应中的 `actualXxxAmount` 字段解决了这个问题。

**3. 所有金额字段用 string 类型**

遵循 ADR-06 决策：IDL传输用string，内部运算用 shopspring/decimal，DB存储用 DECIMAL(30,8)。支持法币2位到加密币8位的多精度需求。

## 七、幂等性设计

所有资金变动类RPC接口（R1~R7）都**必须幂等**。这是整个设计的安全底线。

### 7.1 幂等key对照表

| RPC | 幂等key | 格式 | 说明 |
|-----|---------|------|------|
| R1 WalletCredit | orderNo | C+16位（充值）/ B+16位（补单） | 同一订单只入账一次 |
| R2 WalletCreditReward | orderNo + flow_type | 与R1同一订单号，但flow_type不同 | 同一订单奖金只入一次 |
| R3 WalletFreeze | orderNo | T+16位（提现单号） | 同一提现只冻结一次 |
| R4 WalletUnfreeze | orderNo | T+16位 | 同一冻结只解冻一次 |
| R5 WalletDeduct | orderNo | T+16位 | 同一冻结只扣款一次 |
| R6 WalletManualAdd | orderNo | A+16位（加款单号） | 同一修正只加一次 |
| R7 WalletManualSub | orderNo | M+16位（减款单号） | 同一修正只减一次 |

### 7.2 幂等实现机制

```
资金流水表（wallet_flow）唯一索引：
  UNIQUE KEY idx_order_flow (related_order_no, flow_type)

执行逻辑：
  1. 先查询 (orderNo, flowType) 是否已存在
  2. 已存在 → 直接返回成功（幂等）
  3. 不存在 → 执行余额变更 + 写入流水（一个事务）
  4. 唯一索引兜底：即使并发，第二个INSERT会失败 → 返回幂等成功
```

### 7.3 为什么幂等这么重要

```
真实场景：三方通道充值回调

  三方通道 ──回调1──► ser-finance ──R1──► ser-wallet（入账100）
       │                                      ✓ 成功
       │  （网络超时，通道认为失败，重试）
       │
       └──回调2──► ser-finance ──R1──► ser-wallet（入账100）
                                              ✓ 幂等返回（不重复入账）

如果没有幂等 → 用户充100，实际到账200 → 资金损失
```

## 八、接口间的依赖关系与调用时序

### 8.1 R3/R4/R5 的互斥关系

冻结→解冻 和 冻结→扣除 是**互斥的终态**——一笔冻结要么被解冻（钱还回去），要么被扣除（钱出去），不可能两个都发生。

```
冻结记录状态机：

   FROZEN（已冻结）
       │
       ├── R4 Unfreeze ──► UNFROZEN（已解冻） —— 终态
       │
       └── R5 Deduct ────► DEDUCTED（已扣除） —— 终态

约束：
  - UNFROZEN 状态下再调 R5 → 拒绝（或幂等返回已解冻错误）
  - DEDUCTED 状态下再调 R4 → 拒绝（或幂等返回已扣除错误）
  - FROZEN 状态下重复调 R3 → 幂等返回已冻结
```

### 8.2 调用时序约束

| 前置 | 后置 | 约束说明 |
|------|------|---------|
| R3 (Freeze) | R4 (Unfreeze) 或 R5 (Deduct) | 必须先有冻结记录，才能解冻或扣除 |
| R8 (GetBalance) | R6 (ManualAdd) 或 R7 (ManualSub) | 非技术约束，但业务流程上是先查后改 |
| R1 (Credit) | — | 独立，无前置约束 |
| R2 (CreditReward) | — | 独立，通常与R1同一业务流程中一起调用 |

### 8.3 典型调用序列

**充值成功入账**：
```
ser-finance:
  1. 验证三方回调金额
  2. 调用 R1 WalletCredit(orderNo=C2026022800001, amount=1000)
  3. 如果用户选了奖金:
     调用 R2 WalletCreditReward(orderNo=C2026022800001, amount=500, turnoverMulti=3)
  4. 更新本地充值记录状态
```

**提现审核驳回**：
```
ser-finance:
  1. B端管理员点击"驳回"
  2. 调用 R4 WalletUnfreeze(orderNo=T2026022800001, remark="风控异常")
  3. 更新本地提现记录状态
```

**人工加款**：
```
ser-finance:
  1. 管理员输入用户ID
  2. 调用 R8 GetUserBalance(userId=12345, currencyCode="VND") → 展示各子钱包余额
  3. 管理员填写加款金额并提交
  4. 另一管理员审核通过
  5. 调用 R6 WalletManualAdd(orderNo=A2026022800001, centerAmount=500000, rewardAmount=100000)
```

## 十一、总结

### 一句话结论

ser-wallet 需要对外暴露 **9个RPC方法**（7个资金操作 + 2个查询），全部由 **ser-finance** 调用，覆盖充值入账、提现冻结/解冻/扣除、人工修正加减款、余额查询、币种配置查询 6大场景。

### 接口速查卡

```
┌─────────────────────────────────────────────────────────────────────┐
│           ser-wallet RPC 暴露接口（9个）                              │
│                                                                      │
│  ┌── 资金入账 ──────────────────────────────────────────────────┐    │
│  │  R1  WalletCredit        充值成功/补单 → 中心钱包入账         │    │
│  │  R2  WalletCreditReward  充值赠送/活动 → 奖励钱包入账         │    │
│  └──────────────────────────────────────────────────────────────┘    │
│                                                                      │
│  ┌── 提现三件套 ────────────────────────────────────────────────┐    │
│  │  R3  WalletFreeze        提现发起 → 冻结可用余额              │    │
│  │  R4  WalletUnfreeze      审核驳回/出款失败 → 解冻归还          │    │
│  │  R5  WalletDeduct        出款成功 → 最终扣除冻结资金           │    │
│  └──────────────────────────────────────────────────────────────┘    │
│                                                                      │
│  ┌── 人工修正 ──────────────────────────────────────────────────┐    │
│  │  R6  WalletManualAdd     B端审核通过 → 按子钱包分别加款       │    │
│  │  R7  WalletManualSub     B端审核通过 → 按子钱包分别减款       │    │
│  └──────────────────────────────────────────────────────────────┘    │
│                                                                      │
│  ┌── 查询 ──────────────────────────────────────────────────────┐    │
│  │  R8  GetUserBalance      查询用户各子钱包余额                  │    │
│  │  R9  GetCurrencyConfig   获取币种配置信息（精度/汇率/符号）     │    │
│  └──────────────────────────────────────────────────────────────┘    │
│                                                                      │
│  调用方：ser-finance（当前唯一确定调用方）                            │
│  幂等：R1~R7 全部基于 orderNo 幂等                                   │
│  金额类型：全部 string（shopspring/decimal 运算）                     │
│  IDL文件：wallet_rpc.thrift                                          │
└─────────────────────────────────────────────────────────────────────┘
```

### 三个核心认知

1. **9个RPC = 6个原子操作 + 3个组合操作**
   - 原子操作：查余额(R8)、入账(R1)、奖金入账(R2)、冻结(R3)、解冻(R4)、扣除(R5)
   - 组合操作：人工加款(R6=多子钱包R1)、人工减款(R7=多子钱包扣减)、查配置(R9)

2. **钱包是执行者，财务是决策者**
   - 钱包不做审核、不做通道匹配、不做风控判定
   - 钱包只管：你给我订单号和金额，我执行加/减/冻结/解冻，保证幂等和事务

3. **这9个接口是财务模块的命脉**
   - 没有这些RPC，财务的充值入账、提现审核、人工修正全部无法完成
   - IDL接口定义要尽早与财务团队对齐，是跨团队协作的第一个卡点
