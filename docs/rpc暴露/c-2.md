# ser-wallet 需暴露的 RPC 接口分析

> 分析目标：识别 ser-wallet 模块需要暴露给工程内其他模块调用的 RPC 接口
> 视角范围：仅分析内部 RPC 接口（不含 C 端 HTTP / B 端 HTTP 接口）
> 依据来源：需求文档/原型（~136 份）+ 工程代码分析 + 之前的链路/依赖分析
> 分析方法：从业务场景出发，追踪每个场景中"谁需要调用钱包的什么能力"

---

# 一、分析思路

RPC 接口的暴露不是拍脑袋定的，而是由**业务场景驱动**的。我从三个角度交叉验证：

1. **需求文档/原型**：哪些业务流程涉及其他模块操作钱包（充值回调、提现审核、投注扣款等）
2. **工程现状**：现有 22 个服务的 RPC 调用模式（ser-live、ser-bag、ser-user 等谁需要钱包）
3. **之前的链路分析**：链路关系/c-2.md 中已识别的跨模块调用点

---

# 三、RPC 接口逐个分析

## 3.1 余额操作类（写操作 — 最核心）

### RPC-01: Credit（加款/上分）

**调用场景**（需求文档依据）：

| 场景 | 调用方 | 需求依据 |
|------|--------|---------|
| 充值成功回调 | 财务管理模块 | 充值流程图（2-3充值流程）：支付成功→放款成功→订单状态为已完成并给用户上分 |
| 充值补单 | 财务管理模块 | 人工修正（2-5）：充值补单审核通过后给用户上分 |
| 人工加款 | 财务管理模块 | 人工修正（2-5）：人工加款审核通过后给用户加款 |
| 投注返奖 | 投注/结算模块 | 币种配置（2-1）：投注返奖计入中心钱包 |
| 主播收入 | ser-live | 币种配置（2-1）：礼物打赏分成、主播活动/任务奖励转入主播钱包 |
| 代理佣金 | 代理模块 | 币种配置（2-1）：代理分成转入代理钱包 |

**接口签名设计**：
```
Credit(userId, currencyId, walletType, amount, orderNo, reason, operatorId?)
```

**关键参数说明**：
- `walletType`：指定加款到哪个钱包（中心/奖励/主播/代理）
- `orderNo`：幂等键，防止同一笔订单重复加款（**必须**）
- `reason`：加款原因枚举（充值/返奖/补单/人工加款/收入等），用于流水记录
- `operatorId`：人工操作时的操作员 ID（可选，自动流程不传）

**内部处理逻辑**：
1. 幂等校验（orderNo 去重）
2. 增加指定钱包余额（乐观锁保证并发安全）
3. 创建账变流水记录

---

### RPC-02: Debit（扣款/下分）

**调用场景**：

| 场景 | 调用方 | 需求依据 |
|------|--------|---------|
| 投注下注 | 投注模块 | 币种配置（2-1）：投注时优先扣中心钱包余额，再扣奖励钱包余额 |
| 送礼消费 | ser-live | 币种配置（2-1）：娱乐场消费（刷礼物、购买虚拟道具等） |
| 购买道具 | ser-item/ser-mall | 币种配置（2-1）：购买虚拟道具（头像框、入场坐骑、靓号等） |
| 人工减款 | 财务管理模块 | 人工修正（2-5）：人工减款审核通过后扣减用户余额 |
| 提现出款成功 | 财务管理模块 | 提现流程图（2-4）：出款成功后扣减冻结金额 |

**接口签名设计**：
```
Debit(userId, currencyId, walletType, amount, orderNo, reason, operatorId?)
```

**特殊逻辑 — 资金钱包双扣**（需求依据：币种配置 2-1 资金钱包体系7）：
> "消费时优先扣除中心钱包余额，再扣奖励钱包余额"
> "单次下注使用到两个钱包余额混合投注时，需根据投注金额占比分开计返奖"

这意味着 Debit 可能需要支持"自动拆分扣款"：先扣中心钱包，不够再扣奖励钱包。或者由调用方自行拆分两次调用。

**设计决策（待确认）**：
- 方案 A：Debit 只操作单个 walletType，调用方负责拆分 → 简单但需要调用方理解规则
- 方案 B：提供 DebitAuto(userId, currencyId, amount) 自动按优先级扣减 → 封装规则但复杂

建议：**先实现方案 A（单钱包扣款），后续按需加 DebitAuto**。理由：保持 RPC 接口的原子性和单一职责，复杂的扣款顺序逻辑由调用方（或提供一个 helper）控制。

---

### RPC-03: Freeze（冻结金额）

**调用场景**：

| 场景 | 调用方 | 需求依据 |
|------|--------|---------|
| 用户提交提现 | ser-wallet 自身（C端接口 CreateWithdrawOrder 调用内部 Service） | 提现流程图（2-4）：提交提现后冻结金额，进入审核 |
| 外部冻结请求 | 财务管理模块/风控模块 | 可能存在管理员冻结用户资金的场景 |

**接口签名设计**：
```
Freeze(userId, currencyId, amount, orderNo, reason)
```

**内部处理**：
1. 检查可用余额 ≥ amount
2. 可用余额 -= amount，冻结金额 += amount（总余额不变）
3. 记录冻结流水

**注意**：虽然当前主场景是提现时内部调用，但仍需暴露为 RPC——因为财务模块可能需要直接冻结用户资金（如异常交易风控冻结）。

---

### RPC-04: Unfreeze（解冻金额）

**调用场景**：

| 场景 | 调用方 | 需求依据 |
|------|--------|---------|
| 提现审核拒绝 | 财务管理模块 | 提现流程图（2-4）：任一级审核拒绝→资金解冻→余额恢复 |
| 提现出款失败 | 财务管理模块 | 提现流程图（2-4）：代付通道出款失败→资金解冻 |

**接口签名设计**：
```
Unfreeze(userId, currencyId, amount, orderNo, reason)
```

**内部处理**：
1. 检查冻结金额 ≥ amount
2. 冻结金额 -= amount，可用余额 += amount
3. 记录解冻流水

---

### RPC-05: DebitFromFrozen（从冻结金额扣款）

**调用场景**：

| 场景 | 调用方 | 需求依据 |
|------|--------|---------|
| 提现出款成功 | 财务管理模块 | 提现流程图（2-4）：出款成功→扣减冻结金额（不是扣可用余额） |

**接口签名设计**：
```
DebitFromFrozen(userId, currencyId, amount, orderNo, reason)
```

**为什么不能用 Debit 代替**：
提现流程中，用户提交提现时已经 Freeze 了资金（从可用余额转入冻结）。出款成功后应该从冻结金额中扣减，而不是从可用余额扣。如果用 Debit 会导致：
- 可用余额被多扣一次
- 冻结金额没有释放
所以需要一个独立的"从冻结扣款"操作。

**内部处理**：
1. 检查冻结金额 ≥ amount
2. 冻结金额 -= amount（总余额 -= amount）
3. 记录扣款流水（标注来源为冻结扣减）

---

### RPC-06: CreditReward（奖励金发放）

**调用场景**：

| 场景 | 调用方 | 需求依据 |
|------|--------|---------|
| 充值奖金活动 | 财务管理模块/活动模块 | 额外奖金选择规则（5-6）：选择奖金活动后，充值金额 + 奖金一起到账，奖金入奖励钱包 |
| 活动/任务奖励 | 活动模块 | 币种配置（2-1）：活动/任务返奖进入奖励钱包 |
| 首充奖金 | 活动模块 | 额外奖金选择规则（5-6）：首充活动奖金 |

**接口签名设计**：
```
CreditReward(userId, currencyId, amount, orderNo, reason, turnoverMultiplier)
```

**特殊参数**：
- `turnoverMultiplier`：稽核倍数（如 x5 流水要求）。需求明确指出"赠送部分需完成 X 倍流水后可提现"（额外奖金规则 5-6 + 兑换模块 6.4）

**与 Credit 的区别**：
- Credit 是通用加款（指定 walletType）
- CreditReward 固定加到**奖励钱包**，且附带稽核倍数信息

**设计决策**：也可以不单独设计此接口，而是用 Credit(walletType=reward) + 额外参数。但考虑到奖励金有特殊的稽核逻辑，独立接口更清晰。

---

### RPC-07: TransferToCenter（转入中心钱包）

**调用场景**：

| 场景 | 调用方 | 需求依据 |
|------|--------|---------|
| 奖励钱包完成稽核流水转入中心 | ser-wallet 自身/活动模块 | 币种配置（2-1）：奖励钱包完成稽核流水后可转入中心钱包 |
| 主播钱包转入中心 | ser-wallet 自身 | 币种配置（2-1）：主播钱包内资金支持提现，可转入中心钱包 |
| 代理钱包转入中心 | ser-wallet 自身 | 币种配置（2-1）：代理钱包可直接提现，可转入中心钱包 |

**接口签名设计**：
```
TransferToCenter(userId, currencyId, fromWalletType, amount, orderNo)
```

**内部处理**：
1. 检查源钱包可用余额 ≥ amount
2. 源钱包余额 -= amount
3. 中心钱包余额 += amount
4. 两步在同一事务中（原子性）
5. 记录转账流水

**是否需要暴露为 RPC**：主场景是用户在 C 端操作触发（内部调用），但其他模块（如活动模块判断稽核完成后自动触发转入）也可能需要调用。**建议暴露**。

## 3.2 查询类（读操作）

### RPC-08: QueryBalance（查询余额）

**调用场景**：

| 场景 | 调用方 | 需求依据 |
|------|--------|---------|
| 投注前检查余额 | 投注模块 | 通用逻辑：下注前需确认余额充足 |
| 送礼前检查余额 | ser-live | 通用逻辑：送礼前需确认余额充足 |
| 购买前检查余额 | ser-item/ser-mall | 通用逻辑：购买前需确认余额充足 |
| 管理后台查看用户余额 | gate-back（B端网关） | 人工修正（2-5）：加款/减款时需展示用户当前余额 |
| 财务模块查看余额 | 财务管理模块 | 充值/提现记录页展示用户余额信息 |

**接口签名设计**：
```
QueryBalance(userId, currencyId)

返回：{
    centerBalance:  int64,   // 中心钱包余额
    rewardBalance:  int64,   // 奖励钱包余额
    frozenAmount:   int64,   // 冻结金额
    availableAmount: int64,  // 可用余额 = center + reward - frozen（资金钱包余额）
}
```

**这是被调用频率最高的 RPC 接口**。几乎所有涉及"用户要花钱"的场景都会先调用此接口。

---

### RPC-09: QueryWalletDetail（查询钱包详情）

**调用场景**：

| 场景 | 调用方 | 需求依据 |
|------|--------|---------|
| B 端后台查看用户钱包全貌 | gate-back | 管理后台需查看用户所有钱包（中心/奖励/主播/代理）的完整状态 |
| 客服查看用户资产 | 客服系统 | 处理用户投诉时需要全景视图 |

**接口签名设计**：
```
QueryWalletDetail(userId, currencyId)

返回：{
    wallets: [{
        walletType:  int,     // 钱包类型
        balance:     int64,   // 余额
        frozenAmount: int64,  // 冻结金额
        status:      int,     // 状态
    }],
    currentCurrency: string,  // 当前选中币种
}
```

**与 QueryBalance 的区别**：QueryBalance 返回资金钱包维度的汇总数据（给消费场景用），QueryWalletDetail 返回所有钱包类型的明细（给管理场景用）。

---

### RPC-10: BatchQueryBalance（批量查询余额）

**调用场景**：

| 场景 | 调用方 | 需求依据 |
|------|--------|---------|
| B 端用户列表展示余额 | gate-back / ser-buser | 管理后台用户列表可能需展示余额列 |
| 批量风控检查 | 风控模块 | 批量检查多个用户的余额状态 |

**接口签名设计**：
```
BatchQueryBalance(userIds[], currencyId)

返回：{
    balances: map<int64, BalanceInfo>  // userId → 余额信息
}
```

**必要性判断**：视管理后台需求而定。如果用户列表页不展示余额，此接口可延后。**优先级：中低**。

## 3.3 初始化类

### RPC-11: InitWallet（初始化用户钱包）

**调用场景**：

| 场景 | 调用方 | 需求依据 |
|------|--------|---------|
| 用户注册完成 | ser-user | 币种配置（2-1）：资金钱包用于承载用户在平台内的真实资金。用户注册后需要有钱包才能操作 |

**接口签名设计**：
```
InitWallet(userId, defaultCurrencyId?)

返回：{ success: bool }
```

**内部处理**：
1. 创建该用户的默认币种中心钱包（余额 = 0）
2. 创建该用户的默认币种奖励钱包（余额 = 0）
3. 其他币种的钱包采用**懒初始化**（用户切换币种时创建）
4. 主播钱包/代理钱包在用户获得对应角色后创建（非注册时）

**关键设计**：
- 注册时只创建一套默认币种钱包，不需要创建所有币种 × 所有类型的笛卡尔积
- 懒初始化策略来自需求分析：用户切换币种时才创建该币种的钱包（SwitchCurrency 内部 EnsureWalletExists）

## 3.4 币种查询类

### RPC-12: GetAvailableCurrencies（获取可用币种列表）

**调用场景**：

| 场景 | 调用方 | 需求依据 |
|------|--------|---------|
| 财务模块创建通道时选择币种 | 财务管理模块 | 通道配置（2-1）：创建代收/代付通道时需关联币种 |
| 支付配置关联币种 | 财务管理模块 | 支付配置（2-2）：充值方式/提现方式需关联币种 |
| 其他需要币种下拉列表的场景 | 各模块 B 端 | 管理后台凡涉及币种选择的地方 |

**接口签名设计**：
```
GetAvailableCurrencies()

返回：{
    currencies: [{
        currencyId:   int64,
        currencyName: string,  // 如 "越南盾"
        currencyCode: string,  // 如 "VND"
        icon:         string,  // 图标 URL
        type:         int,     // 1=法币 2=虚拟币 3=平台币
        status:       int,     // 1=启用 2=禁用
        decimals:     int,     // 小数位数（VND/IDR=0, THB/BSB=2）
    }]
}
```

### RPC-14: GetExchangeRate（获取汇率）

**调用场景**：

| 场景 | 调用方 | 需求依据 |
|------|--------|---------|
| 充值页面展示 BSB 等价金额 | C 端内部使用为主 | 充值总览（5-1）：只有 BSB 充值金额输入框下有对应汇率展示 |
| 财务模块统计需要汇率换算 | 财务管理模块 | 数据统计可能需要统一换算为基准币种 |

**接口签名设计**：
```
GetExchangeRate(fromCurrencyId, toCurrencyId)

返回：{
    rate:         string,  // 汇率（字符串精度安全）
    depositRate:  string,  // 充值汇率（含浮动）
    withdrawRate: string,  // 提现汇率（含浮动）
    updatedAt:    int64,   // 汇率更新时间
}
```

# 四、RPC 接口优先级排序

根据**紧急重要四象限 + 前后依赖关系**排序：

## 第一优先级（P0 — 核心骨架，其他模块无法联调的阻塞项）

| 接口 | 理由 |
|------|------|
| **RPC-08: QueryBalance** | 被调用频率最高，几乎所有消费场景的前置依赖 |
| **RPC-01: Credit** | 充值回调必须，不上线就无法完成充值闭环 |
| **RPC-02: Debit** | 投注/消费/提现必须，不上线就无法扣款 |
| **RPC-11: InitWallet** | ser-user 注册后必须调用，否则用户没有钱包 |

## 第二优先级（P1 — 提现流程依赖）

| 接口 | 理由 |
|------|------|
| **RPC-03: Freeze** | 提现流程必须，提交提现后冻结金额 |
| **RPC-04: Unfreeze** | 提现拒绝/出款失败必须，解冻恢复余额 |
| **RPC-05: DebitFromFrozen** | 提现出款成功必须，从冻结金额中扣减 |

## 第三优先级（P2 — 奖励/转账/币种）

| 接口 | 理由 |
|------|------|
| **RPC-06: CreditReward** | 奖金活动上线后需要，前期可用 Credit(walletType=reward) 临时代替 |
| **RPC-07: TransferToCenter** | 用户端转账功能，可独立开发不阻塞其他模块 |
| **RPC-12: GetAvailableCurrencies** | 财务模块配置通道时需要，但前期可硬编码币种 |
| **RPC-13: GetCurrencyInfo** | 同上，前期可临时 mock |
| **RPC-14: GetExchangeRate** | 兑换功能上线后需要 |

## 第四优先级（P3 — 管理/批量）

| 接口 | 理由 |
|------|------|
| **RPC-09: QueryWalletDetail** | B 端管理需求，非核心链路阻塞项 |
| **RPC-10: BatchQueryBalance** | 批量查询，视 B 端需求决定是否实现 |

# 五、接口参数规范（遵循工程现有约定）

基于对现有 Thrift IDL 的分析（ser-bag/bag.thrift、ser-live/live_rpc.thrift 等），梳理规范：

## 5.1 命名规范

```
方法名：PascalCase，动词+名词（如 QueryBalance, CreditReward）
请求体：XxxReq（如 CreditReq, QueryBalanceReq）
响应体：XxxResp（如 CreditResp, QueryBalanceResp）
字段名：camelCase
```

## 5.2 通用响应结构

现有工程的响应约定（参考 ser-bag）：
```thrift
struct CreditResp {
    1: required i32 code            // 业务状态码（0=成功）
    2: optional string message      // 错误信息
    3: optional CreditResult data   // 业务数据
}
```

## 5.3 金额字段

统一使用 `i64`（对应 Go int64），表示最小货币单位整数：
- VND：1 = 1越南盾（整数币种）
- THB：100 = 1泰铢（2位小数币种，存储扩大100倍）
- BSB：100 = 1BSB（2位小数）
- USDT：1000000 = 1USDT（6位小数，链上精度）

## 5.4 幂等键

所有写操作接口必须包含 `orderNo` 字段，用于幂等控制。格式约定（来自链路分析）：
- 充值订单：`C` + 16位
- 提现订单：`T` + 16位
- 兑换订单：`B` + 16位（BSB 兑换）
- 活动奖金：`A` + 16位
- 人工修正：`M` + 16位
