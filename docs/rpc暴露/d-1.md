# ser-wallet 需暴露的内部 RPC 接口分析

> 文档定位：分析 ser-wallet 模块除 B端/C端 HTTP 接口外，需要**暴露给工程内其他模块调用的 RPC 接口**。这些接口是模块间联调的基础，决定了 ser-wallet 在整个工程体系中的"对外契约"。
>
> 编写依据：IDL设计/d-1.md（wallet_rpc.thrift 设计）+ 链路关系/d-1.md（跨模块调用链）+ 依赖关系/d-1.md（模块依赖分析）+ 财务结构/d-1.md（联调地图）+ 流程思路/d-1.md（业务流程）+ 需求理解/d-1.md（产品需求）+ 接口评估/d-1.md + 工程认知/d-1.md（现有 RPC 模式）
>
> 分析范围：仅内部 RPC 接口（Kitex 协议，service-to-service），不含 B端/C端 HTTP 接口（Hertz 协议）

## 一、分析方法论

### 1.1 "谁需要我"视角

分析 RPC 接口不是从"我有什么"出发，而是从**"谁需要我提供什么"**出发。具体方法：

```
步骤1：遍历所有业务场景，找出"其他模块需要调用 ser-wallet"的触发点
步骤2：对每个触发点，明确调用方、输入、输出、业务含义
步骤3：按调用方归类，检查是否遗漏
步骤4：对照现有工程 RPC 模式，确认接口设计符合规范
```

### 1.2 调用方识别

基于前置文档分析，以下模块存在调用 ser-wallet 的需求：

| 调用方模块 | 调用场景 | 依据文档 |
|-----------|---------|---------|
| **ser-finance（财务）** | 充值到账、提现出款、冻结/解冻、手动调账 | 财务结构/d-1.md 联调地图、链路关系/d-1.md |
| **ser-finance（财务）** | 查询余额、查询币种、查询汇率 | 依赖关系/d-1.md、流程思路/d-1.md |
| **其他潜在调用方** | 见第六章分析 | 需求理解/d-1.md、架构设计/d-1.md |

## 二、ser-finance → ser-wallet（写操作类 — 5个接口）

这是最核心的一组 RPC 接口——财务模块是 ser-wallet 最主要的内部调用方。

### 2.1 CreditBalance — 加款

**谁调用**：ser-finance

**什么时候调用**：

| 触发场景 | 说明 | 依据 |
|---------|------|------|
| 充值到账 | 三方支付回调成功 → 财务确认 → 通知钱包加款 | 流程思路/d-1.md 充值流程、财务结构/d-1.md §16 |
| 充值补单 | 充值订单异常 → 财务人工补单审批通过 → 通知钱包加款 | 需求理解/d-1.md 补单流程 |
| 手动加款 | 运营/客服手动给用户加钱（补偿/活动等） → 审批通过 → 通知钱包加款 | 需求理解/d-1.md 手动调账 |

**接口定义**：

```
RPC: CreditBalance

入参 CreditBalanceReq:
  userId         i64    required  用户ID
  currencyId     i64    required  币种ID
  walletType     i32    required  钱包类型（1中心 2奖励 3主播 4代理）
  amount         i64    required  加款金额（正整数，最小单位）
  refOrderNo     string required  关联订单号（幂等键）
  txType         i32    required  交易类型（充值/返奖/赠送/补偿/活动奖励...）
  remark         string optional  备注

出参 CreditBalanceResp:
  （空，成功则无报错返回）
```

**幂等保障**：`refOrderNo + txType + walletType` 组合唯一索引。重复调用返回成功，不重复执行。

**内部执行逻辑**（wallet 内部）：
```
1. 幂等检查 → 已处理则直接返回成功
2. 开启 DB 事务
3. SELECT FOR UPDATE 锁定 wallet_account 行
4. available_balance += amount
5. 写 wallet_transaction 流水记录
6. 提交事务
7. 返回成功
```

**依据**：IDL设计/d-1.md §14 wallet_rpc.thrift、财务结构/d-1.md §16 联调地图接口①

---

### 2.2 DebitBalance — 扣款确认

**谁调用**：ser-finance

**什么时候调用**：

| 触发场景 | 说明 | 依据 |
|---------|------|------|
| 提现出款成功 | 三方出款回调成功 → 财务确认 → 通知钱包从冻结中扣除 | 流程思路/d-1.md 提现流程 |
| 手动扣款 | 运营手动扣用户钱（风控/惩罚等） → 审批通过 → 通知钱包扣款 | 需求理解/d-1.md 手动调账 |

**接口定义**：

```
RPC: DebitBalance

入参 DebitBalanceReq:
  userId         i64    required  用户ID
  currencyId     i64    required  币种ID
  walletType     i32    required  钱包类型
  amount         i64    required  扣款金额（正整数）
  refOrderNo     string required  关联订单号（幂等键）
  txType         i32    required  交易类型（提现扣款/手动扣款/惩罚...）
  remark         string optional  备注

出参 DebitBalanceResp:
  （空）
```

**幂等保障**：同 CreditBalance，`refOrderNo + txType + walletType` 组合唯一。

**内部执行逻辑**：
```
1. 幂等检查
2. 开启 DB 事务
3. SELECT FOR UPDATE 锁定 wallet_account
4. 校验 frozen_balance >= amount（从冻结中扣）
5. frozen_balance -= amount
6. 写 wallet_transaction 流水
7. 提交事务
```

**关键点**：DebitBalance 操作的是**冻结余额**，不是可用余额。资金在 FreezeBalance 阶段已从可用转入冻结，DebitBalance 是最终确认扣除。

**依据**：IDL设计/d-1.md §14、财务结构/d-1.md §16 联调地图接口②

---

### 2.3 FreezeBalance — 冻结余额

**谁调用**：ser-wallet 内部（CreateWithdrawOrder 流程）；也可能被 ser-finance 在特定场景调用。

**什么时候调用**：

| 触发场景 | 说明 | 依据 |
|---------|------|------|
| 用户发起提现 | C端提交提现申请 → 钱包冻结对应金额 | 流程思路/d-1.md 提现流程步骤① |
| 风控冻结 | 财务/风控要求冻结某用户资金（异常交易排查） | 需求理解/d-1.md 风控场景 |

**接口定义**：

```
RPC: FreezeBalance

入参 FreezeBalanceReq:
  userId         i64    required  用户ID
  currencyId     i64    required  币种ID
  walletType     i32    required  钱包类型
  amount         i64    required  冻结金额（正整数）
  refOrderNo     string required  关联订单号（提现订单号）

出参 FreezeBalanceResp:
  （空）
```

**内部执行逻辑**：
```
1. 幂等检查
2. 开启 DB 事务
3. SELECT FOR UPDATE 锁定 wallet_account
4. 校验 available_balance >= amount
5. available_balance -= amount
6. frozen_balance += amount
7. 写 wallet_transaction 流水
8. 提交事务
```

**是否纯内部？** 主场景是 ser-wallet 内部调用（提现申请时自动冻结）。但作为 RPC 接口暴露是必要的——当财务模块需要执行风控冻结时，也需要此接口。暴露不代表一定被外部调用，而是**保留能力**。

**依据**：IDL设计/d-1.md §14、流程思路/d-1.md 提现链路

### 2.4 UnfreezeBalance — 解冻余额

**谁调用**：ser-finance

**什么时候调用**：

| 触发场景 | 说明 | 依据 |
|---------|------|------|
| 提现审核驳回 | 风控/财务审核不通过 → 通知钱包解冻 → 资金回到可用余额 | 流程思路/d-1.md 提现驳回路径 |
| 提现出款失败 | 三方出款回调失败 → 财务确认 → 通知钱包解冻 | 财务结构/d-1.md §16 联调地图接口③ |
| 提现取消 | 用户取消提现（如果产品支持） | 需求理解/d-1.md |

**接口定义**：

```
RPC: UnfreezeBalance

入参 UnfreezeBalanceReq:
  userId         i64    required  用户ID
  currencyId     i64    required  币种ID
  walletType     i32    required  钱包类型
  amount         i64    required  解冻金额（正整数）
  refOrderNo     string required  关联订单号

出参 UnfreezeBalanceResp:
  （空）
```

**内部执行逻辑**：
```
1. 幂等检查
2. 开启 DB 事务
3. SELECT FOR UPDATE 锁定 wallet_account
4. 校验 frozen_balance >= amount
5. frozen_balance -= amount
6. available_balance += amount（资金归还用户）
7. 写 wallet_transaction 流水
8. 提交事务
```

**依据**：IDL设计/d-1.md §14、财务结构/d-1.md §16 联调地图接口③

### 2.5 ManualAdjust — 手动调账（多钱包）

**谁调用**：ser-finance

**什么时候调用**：

| 触发场景 | 说明 | 依据 |
|---------|------|------|
| 手动补款（多钱包） | 一次调整涉及多个钱包类型（如同时加中心钱包 + 奖励钱包） | 需求理解/d-1.md 手动调账页面 |
| 手动扣款（多钱包） | 一次扣款涉及多个钱包类型 | 需求理解/d-1.md |
| 混合调账 | 一个钱包加、另一个钱包扣（转移场景） | 流程思路/d-1.md |

**为什么不用 CreditBalance/DebitBalance 替代？**

CreditBalance/DebitBalance 是单钱包单方向操作。但产品原型中的"手动调账"页面允许运营人员**一次性调整一个用户的多个钱包类型**。如果用 CreditBalance + DebitBalance 分开调用：
- 两次 RPC 非原子性，中间可能失败导致数据不一致
- 不符合"一次审批 = 一次执行"的业务语义

所以需要一个**批量原子调账**接口。

**接口定义**：

```
RPC: ManualAdjust

入参 ManualAdjustReq:
  userId         i64               required  用户ID
  currencyId     i64               required  币种ID
  adjustments    list<AdjustItem>  required  调整明细列表
  refOrderNo     string            required  关联审批单号（幂等键）
  reason         string            required  调账原因
  operatorId     i64               required  操作人ID

AdjustItem:
  walletType     i32    required  钱包类型
  opType         i32    required  操作类型（1=加款 2=扣款）
  amount         i64    required  金额（正整数）

出参 ManualAdjustResp:
  （空）
```

**内部执行逻辑**：
```
1. 幂等检查（refOrderNo 唯一）
2. 开启 DB 事务
3. 遍历 adjustments 列表：
   对每个 AdjustItem：
     a. SELECT FOR UPDATE 锁定对应 wallet_account
     b. 加款：available_balance += amount
     c. 扣款：校验 available_balance >= amount，然后 -= amount
     d. 写 wallet_transaction 流水
4. 全部成功 → 提交事务
5. 任一失败 → 回滚全部
```

**依据**：需求理解/d-1.md 手动调账需求、流程思路/d-1.md 调账流程、IDL设计/d-1.md（该接口在 IDL 设计中标注为"待定"，现在根据需求分析确认需要）

## 三、ser-finance → ser-wallet（读操作类 — 4个接口）

### 3.1 GetBalance — 查询单用户余额

**谁调用**：ser-finance

**什么时候调用**：

| 触发场景 | 说明 | 依据 |
|---------|------|------|
| 手动扣款前校验 | 财务扣款前先查余额是否充足 | 流程思路/d-1.md 扣款前置校验 |
| 财务报表数据 | 充提统计需要用户当前余额 | 依赖关系/d-1.md |
| 订单审核辅助 | 审核提现订单时查看用户余额状况 | 需求理解/d-1.md |

**接口定义**：

```
RPC: GetBalance

入参 GetBalanceReq:
  userId         i64    required  用户ID
  currencyId     i64    required  币种ID
  walletType     i32    optional  钱包类型（不传返回该币种所有钱包类型）

出参 GetBalanceResp:
  balances       list<WalletBalanceItem>  余额列表

WalletBalanceItem:
  walletType     i32    钱包类型
  available      i64    可用余额
  frozen         i64    冻结金额
  total          i64    总额（available + frozen）
```

**为什么返回 list 而不是单个值？**

一个用户在同一币种下可能有多个钱包类型（中心/奖励/主播/代理）。当 walletType 不传时，应返回所有钱包类型的余额。单独返回某一个不能满足"查看用户全部资产"的场景。

**幂等**：只读接口，无需幂等。

**依据**：IDL设计/d-1.md §14 wallet_rpc.thrift

### 3.2 BatchGetBalance — 批量查询用户余额

**谁调用**：ser-finance

**什么时候调用**：

| 触发场景 | 说明 | 依据 |
|---------|------|------|
| 批量财务报表 | 导出多用户余额快照 | 依赖关系/d-1.md |
| 对账 | 批量比对用户余额与交易流水 | 架构设计/d-1.md |
| 统计报表 | 用户余额分布统计 | 流程思路/d-1.md |

**接口定义**：

```
RPC: BatchGetBalance

入参 BatchGetBalanceReq:
  userIds        list<i64>  required  用户ID列表
  currencyId     i64        required  币种ID
  walletType     i32        optional  钱包类型（不传返回所有类型）

出参 BatchGetBalanceResp:
  list           list<UserBalanceItem>

UserBalanceItem:
  userId         i64
  balances       list<WalletBalanceItem>  该用户的各钱包类型余额
```

**性能考虑**：应限制单次查询的 userIds 数量上限（建议 ≤ 100），防止单次 RPC 查询过大。

**依据**：IDL设计/d-1.md §14

### 3.3 GetActiveCurrencyList — 获取已启用币种列表

**谁调用**：ser-finance

**什么时候调用**：

| 触发场景 | 说明 | 依据 |
|---------|------|------|
| 配置充值通道 | 财务配置充值通道时，需要下拉选择币种 | 财务结构/d-1.md §16 |
| 配置提现通道 | 财务配置提现通道时，需要下拉选择币种 | 财务结构/d-1.md §16 |
| 配置支付方式 | 支付方式绑定币种时，需要币种列表 | 需求理解/d-1.md 财务管理页面 |

**接口定义**：

```
RPC: GetActiveCurrencyList

入参 GetActiveCurrencyListReq:
  （空，或 optional 筛选条件）

出参 GetActiveCurrencyListResp:
  list           list<CurrencyItem>

CurrencyItem:
  id             i64     币种ID
  code           string  币种代码（如 USD, VND, USDT）
  name           string  币种名称
  symbol         string  货币符号（如 $, ₫, ₮）
  currencyType   i32     币种类型（1法币 2虚拟币）
  decimalPlaces  i32     精度（小数位数）
  iconUrl        string  币种图标URL
  country        string  国家/地区代码（法币用）
```

**为什么是 RPC 而不是让财务模块直接查自己的库？**

币种配置是 ser-wallet 的职责域（币种配置子模块）。按照"每服务独立数据库"的架构原则，ser-finance 不应直接访问 ser-wallet 的库。数据所有权在 ser-wallet，其他模块通过 RPC 获取。

**依据**：依赖关系/d-1.md（币种配置职责归属）、架构设计/d-1.md（服务间数据隔离）

### 3.4 GetExchangeRate — 查询汇率

**谁调用**：ser-finance

**什么时候调用**：

| 触发场景 | 说明 | 依据 |
|---------|------|------|
| 充提统计换算 | 财务报表需要把多币种金额换算为基础币种 | 流程思路/d-1.md |
| 通道匹配 | 匹配充值/提现通道时需要参考汇率 | 财务结构/d-1.md |
| 费用计算 | 计算提现手续费时可能涉及汇率换算 | 需求理解/d-1.md |

**接口定义**：

```
RPC: GetExchangeRate

入参 GetExchangeRateReq:
  currencyId     i64    required  源币种ID
  targetCurrencyId i64  optional  目标币种ID（不传则返回对基础币种的汇率）

出参 GetExchangeRateResp:
  platformRate   i64    平台汇率（最小单位表示）
  depositRate    i64    充值汇率
  withdrawRate   i64    提现汇率
  decimalPlaces  i32    汇率精度
  updatedAt      i64    最后更新时间（Unix 时间戳）
```

**为什么区分三种汇率？**

基于需求理解/d-1.md，平台可能对充值和提现使用不同的汇率（买入/卖出价差），平台汇率用于内部统计换算。这三种汇率的值可能不同。

**依据**：需求理解/d-1.md 币种配置需求、IDL设计/d-1.md

## 四、完整接口清单

### 4.1 汇总表

| 编号 | RPC 接口 | 类型 | 主要调用方 | 幂等 | 优先级 | 业务场景 |
|------|---------|------|-----------|------|--------|---------|
| W01 | **CreditBalance** | 写 | ser-finance | 是 | P0 | 充值到账/补单/手动加款 |

| W04 | **UnfreezeBalance** | 写 | ser-finance | 是 | P0 | 提现驳回/出款失败解冻 |
| W05 | **ManualAdjust** | 写 | ser-finance | 是 | P1 | 多钱包批量调账 |
| W06 | **GetBalance** | 读 | ser-finance | — | P0 | 查询单用户余额 |
| W07 | **BatchGetBalance** | 读 | ser-finance | — | P1 | 批量查询用户余额 |
| W08 | **GetActiveCurrencyList** | 读 | ser-finance | — | P1 | 获取已启用币种列表 |
| W09 | **GetExchangeRate** | 读 | ser-finance | — | P1 | 查询汇率 |

### 4.2 按依赖优先级排序

```
第一批（P0 — 没有这些接口，充提流程无法跑通）：
  W01 CreditBalance      ← 充值到账的终点
  W02 DebitBalance        ← 提现出款的终点
  W03 FreezeBalance       ← 提现申请的起点
  W04 UnfreezeBalance     ← 提现失败的恢复
  W06 GetBalance          ← 余额校验的基础

第二批（P1 — 运营和管理必需，但不阻塞核心充提）：
  W05 ManualAdjust        ← 运营调账
  W07 BatchGetBalance     ← 报表/对账
  W08 GetActiveCurrencyList ← 财务配置
  W09 GetExchangeRate     ← 汇率查询
```

## 五、接口设计要点

### 5.1 统一设计原则

**原则1：所有写操作必须幂等**

```
幂等键：refOrderNo + txType + walletType（写操作）
         refOrderNo（ManualAdjust）
实现方式：wallet_transaction 表唯一索引
重复调用行为：检查已存在 → 返回成功，不重复执行
```

依据：表结构设计/d-1.md wallet_transaction 唯一索引设计

**原则2：所有余额变动必须在同一 DB 事务内**

```
一次 RPC 调用 = 一个 DB 事务 = 原子性保证
事务内操作：
  ① SELECT FOR UPDATE 锁行
  ② 余额校验（不允许负数）
  ③ 更新 wallet_account
  ④ 写 wallet_transaction 流水
任一步骤失败 → 全部回滚
```

依据：架构设计/d-1.md balance_engine 统一事务设计

**原则3：余额永不为负**

```
available_balance >= 0（始终）
frozen_balance >= 0（始终）
扣款/冻结前必须校验，不足则返回错误码
```

**原则4：遵循现有工程 RPC 模式**

```
IDL 定义位置：common/idl/ser-wallet/wallet_rpc.thrift
命名空间：namespace go ser_wallet
请求命名：{Action}{Entity}Req（如 CreditBalanceReq）
响应命名：{Action}{Entity}Resp（如 CreditBalanceResp）
客户端注册：common/rpc/rpc_client.go → WalletClient()
服务注册名：namesp.EtcdWalletService = "wallet_service"
```

依据：工程认知/d-1.md、现有模块 IDL 扫描（ser-user/user_rpc.thrift、ser-live/live_rpc.thrift 等）

### 5.2 错误码设计

RPC 接口需要返回业务错误码（区别于框架级错误），遵循工程已有的错误码分段体系：

| 错误场景 | 错误码 | 说明 |
|---------|--------|------|
| 余额不足 | wallet 段位待定 | available_balance < 需扣金额 |
| 冻结余额不足 | wallet 段位待定 | frozen_balance < 需解冻/扣除金额 |
| 幂等命中（已处理） | 无错误，返回成功 | 正常情况，不是错误 |
| 钱包账户不存在 | wallet 段位待定 | 该用户该币种该钱包类型无账户记录 |
| 币种不存在或已禁用 | wallet 段位待定 | currencyId 无效或已下线 |
| 参数校验失败 | wallet 段位待定 | 金额≤0 / 必填字段缺失 |

依据：工程认知/d-1.md 错误码分段规范

### 5.3 接口与 balance_engine 的关系

所有写操作 RPC 接口（W01-W05）都应通过 balance_engine 统一入口执行余额变动，而不是各自直接操作数据库：

```
CreditBalance  → balance_engine.Credit(...)
DebitBalance   → balance_engine.Debit(...)
FreezeBalance  → balance_engine.Freeze(...)
UnfreezeBalance → balance_engine.Unfreeze(...)
ManualAdjust   → balance_engine.BatchAdjust(...)（循环调用 Credit/Debit）
```

这样保证：
- 事务逻辑集中在一处，不分散
- 流水记录写入逻辑统一
- 余额校验逻辑不重复

依据：架构设计/d-1.md balance_engine 核心引擎设计

## 六、潜在调用方分析

除了 ser-finance 这个确定的调用方外，是否还有其他模块需要调用 ser-wallet？

### 6.1 已分析排除的模块

| 模块 | 是否需要调 ser-wallet | 分析结论 |
|------|---------------------|---------|
| ser-blog（日志） | **否**（方向反了） | 是 ser-wallet 调用 ser-blog 的 AddActLog 写审计日志，不是反过来 |
| ser-buser（用户/KYC） | **否**（方向反了） | 是 ser-wallet 调用 ser-buser 查 KYC 状态，不是反过来 |
| ser-app（基础服务） | **否** | ser-app 提供公共能力（S3/Banner等），不需要钱包数据 |
| ser-bcom（配置中心） | **否** | 提供系统配置，不消费钱包数据 |

### 6.2 未来可能的调用方

| 潜在调用方 | 可能场景 | 当前状态 | 建议 |
|-----------|---------|---------|------|
| 投注/游戏模块 | 投注扣款、派彩加款 | **当前需求范围外** | 架构上已兼容——CreditBalance/DebitBalance 接口的 txType 支持扩展 |
| 活动/促销模块 | 活动奖励发放 | **当前通过财务模块中转** | 如果未来活动模块直接调钱包，CreditBalance 接口已可支持 |
| 风控模块 | 紧急冻结/解冻 | **当前通过财务模块中转** | FreezeBalance/UnfreezeBalance 已暴露，风控模块可直接调用 |
| 数据/BI 模块 | 余额统计/分析 | **当前需求范围外** | GetBalance/BatchGetBalance 已可支持 |

**结论**：当前 9 个 RPC 接口的设计已经具备足够的扩展性。如果未来有新的调用方，不需要新增接口，只需要扩展 txType 枚举值即可。接口设计是**调用方无关**的——CreditBalance 不关心是谁调的，只关心加多少钱、记什么类型。

## 七、接口调用时序总览

### 7.1 充值全链路中的 RPC 调用

```
用户发起充值 → C端接口(ser-wallet HTTP)
  → 创建充值订单(ser-wallet 内部)
  → 调用 ser-finance.MatchDepositChannel(wallet→finance RPC)
  → 返回支付链接给用户

用户完成支付 → 三方回调 → ser-finance 接收
  → ser-finance 确认订单
  → ser-finance 调用 ser-wallet.CreditBalance(finance→wallet RPC) ★
  → ser-wallet 加款 + 写流水
  → 完成
```

**ser-wallet 在此链路中暴露的 RPC**：CreditBalance（W01）

### 7.2 提现全链路中的 RPC 调用

```
用户发起提现 → C端接口(ser-wallet HTTP)
  → ser-wallet 内部调用 FreezeBalance(W03) ★
  → 创建提现订单
  → 通知 ser-finance 有新提现待审

ser-finance 三级审核
  → 驳回路径：ser-finance 调用 ser-wallet.UnfreezeBalance(W04) ★
  → 通过路径：ser-finance 发起三方出款

三方出款回调
  → 成功：ser-finance 调用 ser-wallet.DebitBalance(W02) ★
  → 失败：ser-finance 调用 ser-wallet.UnfreezeBalance(W04) ★
```

**ser-wallet 在此链路中暴露的 RPC**：FreezeBalance(W03)、UnfreezeBalance(W04)、DebitBalance(W02)

### 7.3 手动调账链路中的 RPC 调用

```
运营创建调账单 → ser-finance 后台
  → 审批通过
  → ser-finance 调用 ser-wallet.ManualAdjust(W05) ★
  → ser-wallet 批量调整多钱包余额 + 写流水
  → 完成
```

**ser-wallet 在此链路中暴露的 RPC**：ManualAdjust(W05)

### 7.4 财务配置链路中的 RPC 调用

```
财务管理员配置通道/方式
  → ser-finance 后台需要币种下拉
  → ser-finance 调用 ser-wallet.GetActiveCurrencyList(W08) ★

财务生成报表
  → 需要汇率换算
  → ser-finance 调用 ser-wallet.GetExchangeRate(W09) ★

财务审核提现
  → 需要查看用户余额
  → ser-finance 调用 ser-wallet.GetBalance(W06) ★
```

## 八、与现有工程 RPC 模式的对齐

### 8.1 IDL 文件规划

基于现有工程模式（ser-user 有 user_rpc.thrift、ser-live 有 live_rpc.thrift），ser-wallet 的 RPC 接口应定义在：

```
common/idl/ser-wallet/
├── service.thrift          ← 主服务定义（包含所有方法声明）
├── wallet_rpc.thrift       ← RPC 专用的请求/响应结构体
├── wallet.thrift           ← B端/C端 HTTP 请求/响应结构体
├── currency.thrift         ← 币种配置相关结构体
└── ...
```

`service.thrift` 中同时声明 HTTP 方法和 RPC 方法：

```thrift
service WalletService {
    // === B端/C端 HTTP 方法 ===
    wallet.WalletHomeResp    WalletHome(1: wallet.WalletHomeReq req)
    // ... 其他 HTTP 方法

    // === 内部 RPC 方法（供其他模块调用） ===
    wallet_rpc.CreditBalanceResp      CreditBalance(1: wallet_rpc.CreditBalanceReq req)
    wallet_rpc.DebitBalanceResp       DebitBalance(1: wallet_rpc.DebitBalanceReq req)
    wallet_rpc.FreezeBalanceResp      FreezeBalance(1: wallet_rpc.FreezeBalanceReq req)
    wallet_rpc.UnfreezeBalanceResp    UnfreezeBalance(1: wallet_rpc.UnfreezeBalanceReq req)
    wallet_rpc.ManualAdjustResp       ManualAdjust(1: wallet_rpc.ManualAdjustReq req)
    wallet_rpc.GetBalanceResp         GetBalance(1: wallet_rpc.GetBalanceReq req)
    wallet_rpc.BatchGetBalanceResp    BatchGetBalance(1: wallet_rpc.BatchGetBalanceReq req)
    wallet_rpc.GetActiveCurrencyListResp  GetActiveCurrencyList(1: wallet_rpc.GetActiveCurrencyListReq req)
    wallet_rpc.GetExchangeRateResp    GetExchangeRate(1: wallet_rpc.GetExchangeRateReq req)
}
```

### 8.2 客户端注册

其他模块（如 ser-finance）调用 ser-wallet 时，需要在 `common/rpc/rpc_client.go` 中注册：

```go
// 023 wallet
var (
    walletService *walletservice.Client
    walletOnce    sync.Once
)

func WalletClient() walletservice.Client {
    walletOnce.Do(func() {
        wallet, err := walletservice.NewClient(
            namesp.EtcdWalletService,
            InitRpcClientParams()...,
        )
        walletService = &wallet
        if err != nil {
            lg.Log().CtxErrorf(context.Background(), "RPC 创建ser-wallet客户端错误 %s", err)
        }
    })
    return *walletService
}
```

### 8.3 ETCD 命名空间

需在 `common/pkg/consts/namesp/namesp.go` 中新增：

```go
// EtcdWalletService 023
EtcdWalletService = "wallet_service"
EtcdWalletPort    = "/slg/serv/wallet/port"
EtcdWalletDb      = "/slg/serv/wallet/db"
```
