# ser-wallet RPC暴露接口分析

> 分析目的：梳理ser-wallet需要暴露哪些RPC接口给工程内其他模块调用
> 分析依据：①现有工程IDL/代码(22个服务) ②需求文档/原型(163张图) ③已有设计文档(IDL设计+实现表达+财务结构)
> 范围说明：本文聚焦"被调用"（其他模块调我），不包含"主动调用"（我调其他模块）

---

## 一、分析方法

RPC暴露接口不能拍脑袋想，需要**三个维度交叉验证**：

1. **需求文档驱动** — 产品需求中明确涉及跨模块交互的场景
2. **工程代码驱动** — 现有服务已有的经济体系（钻石、消费、收益）与钱包的关联
3. **架构逻辑驱动** — 微服务拆分后，哪些数据本该从钱包获取

以下分析每个RPC接口都标注其**需求依据**和**调用方**。

## 三、RPC接口分组详解

### 第一组：财务模块调用（8个）— 已在IDL设计中定义

> 依据：实现表达/c-5.md 第8.1节"财务调我（8个接口）"
> 调用方：ser-finance
> IDL位置：wallet_rpc.thrift
> 状态：**已设计，已确认**

| # | 方法名 | 签名 | 用途 | 需求依据 |
|---|--------|------|------|---------|
| 1 | `DepositCredit` | `bool DepositCredit(1: wallet_rpc.DepositCreditReq req)` | 充值入账：支付渠道确认后，财务调我增加用户余额 | 需求文档5.2"充值到账后更新余额" |
| 2 | `WithdrawFreeze` | `bool WithdrawFreeze(1: wallet_rpc.WithdrawFreezeReq req)` | 提现冻结：用户发起提现，冻结对应金额(available-,frozen+) | 需求文档7.1"确认提现时冻结余额" |
| 3 | `WithdrawConfirm` | `bool WithdrawConfirm(1: wallet_rpc.WithdrawConfirmReq req)` | 提现确认：三级审核通过+渠道出款成功，正式扣除冻结金额(frozen-) | 需求文档7.5"提现审核通过后扣款" |
| 4 | `WithdrawUnfreeze` | `bool WithdrawUnfreeze(1: wallet_rpc.WithdrawUnfreezeReq req)` | 提现解冻：审核拒绝或出款失败，退回冻结金额(available+,frozen-) | 需求文档7.5"提现失败退回" |
| 5 | `ManualCredit` | `bool ManualCredit(1: wallet_rpc.ManualCreditReq req)` | 人工加款：后台管理员手动增加用户余额 | 财务需求文档2.5"人工加款" |
| 6 | `ManualDebit` | `bool ManualDebit(1: wallet_rpc.ManualDebitReq req)` | 人工减款：后台管理员手动扣减用户余额 | 财务需求文档2.5"人工减款" |
| 7 | `ManualPatch` | `bool ManualPatch(1: wallet_rpc.ManualPatchReq req)` | 人工补单：充值未到账的补录入账 | 财务需求文档2.5"充值补单" |
| 8 | `QueryBalanceRpc` | `wallet_rpc.RpcBalanceResp QueryBalanceRpc(1: wallet_rpc.RpcBalanceReq req)` | 余额查询：财务查看用户各子钱包余额 | 财务需求文档2.5"查看用户余额" |

**核心设计原则：**
- 每个写方法必须带`refOrderNo`（幂等键），通过`balance_flow(ref_order_no, change_type)`唯一键保证幂等
- userId显式传参（RPC调用不经过gateway，没有JWT）
- 返回bool（工程惯例，失败走Kitex BizErr机制）

### 第三组：通用余额操作RPC（新增分析）— 需进一步确认

> 依据：工程代码分析 + 需求文档推理
> 核心问题：除了财务模块，还有哪些模块需要操作用户余额？
> 状态：**分析推导，需评审确认**

#### 3.1 分析：ser-live（直播消费）是否需要调钱包？

**代码依据：**

```
live_consume.thrift（第26行）：
    10: i64 diamondAmount  // 消费钻石数
    11: i64 anchorId       // 收益主播用户id

live_room.thrift（第35行）：
    15: i64 totalRevenue   // 总流水(分)
```

**分析：** ser-live已有独立的"钻石消费"经济体系，当前和钱包模块无交互（工程中没有任何thrift文件引用wallet）。这意味着目前有两种可能：

| 方案 | 说明 | 钱包是否需要暴露RPC |
|------|------|-------------------|
| A | 直播钻石是虚拟货币，和钱包余额是**独立体系** | 不需要 |
| B | 直播消费从钱包余额扣除，钻石只是展示单位 | **需要** |

**结论：** 当前代码表明ser-live的经济体系是独立的。**但如果产品规划中直播消费需要对接钱包余额**，则需要额外暴露以下RPC：

```
// [待确认] 通用消费扣款（游戏/直播/商城通用）
struct ConsumeDeductReq {
    1: required i64 userId
    2: required string currencyCode
    3: required i64 amount
    4: required string refOrderNo       // 幂等键
    5: required i32 walletType          // 从哪个子钱包扣
    6: required i32 consumeType         // 消费类型（直播送礼/商品购买/游戏投注等）
    7: optional string remark
}

// [待确认] 通用收益入账（主播收益/返利/佣金通用）
struct RevenueCreditReq {
    1: required i64 userId
    2: required string currencyCode
    3: required i64 amount
    4: required string refOrderNo
    5: required i32 walletType          // 入哪个子钱包（主播→4=主播钱包）
    6: required i32 revenueType         // 收益类型
    7: optional string remark
}
```

**我的建议：暂不新增。** 理由如下：
1. 现有`DepositCredit`和`ManualDebit`的参数结构已经足以覆盖"加钱/扣钱"操作
2. 如果要区分"充值入账"和"消费扣款"的流水类型，只需在`DepositCreditReq`或统一余额操作中增加一个`changeType`字段即可
3. 过早设计消费/收益专用接口，万一直播经济体系不对接钱包，这些接口就是浪费

**但需要关注一个问题：** 当前已定义的8个财务RPC方法（第一组）都是面向"充值/提现/人工修正"场景设计的。如果后续有"游戏消费扣款""直播打赏扣款""主播收益入账"等场景，这些场景的`changeType`和`walletType`需要在枚举中预留位置。

#### 3.2 分析：ser-mall（商城）是否需要调钱包？

**代码依据：** ser-mall的IDL中有`price i64`字段（商品价格，单位：分）

**分析：** 与ser-live类似，当前ser-mall和钱包无交互。如果商品购买从钱包余额扣款，可以复用`ManualDebit`或扩展为通用消费扣款。

**结论：** 暂不新增专用RPC。后续按需从第一组的8个方法中扩展。

#### 3.3 分析：是否需要"批量余额查询"RPC？

**场景：** 管理后台的"数据统计"页面，可能需要批量查询多个用户的余额做汇总。

**代码依据：** 参考ser-user已有`BatchGetUserInfo(BatchGetUserInfoReq)`，批量查询是工程惯例。

**分析：** 当前`QueryBalanceRpc`是单用户查询。如果后台统计、财务对账等场景需要批量查询余额，逐个调用效率低。

**建议：预留但暂不实现。** 可以在IDL中定义但实现阶段先不做，作为TODO。

```
// [P2] 批量余额查询（管理后台统计/对账）
struct BatchBalanceReq {
    1: required list<i64> userIds
    2: optional string currencyCode
}

struct BatchBalanceResp {
    1: optional map<i64, list<RpcWalletItem>> balances  // key=userId
}
```

## 六、RPC接口的参数设计要点

### 6.1 所有写操作必须带幂等键

```
refOrderNo (required string) — 调用方的订单号/修正单号
```

钱包侧通过`balance_flow(ref_order_no, change_type)`唯一键保证幂等。调用方重复调用同一个refOrderNo不会产生重复余额变动。

### 6.2 RPC接口必须显式传userId

C端接口：userId从gateway JWT → ctx metainfo获取，IDL中不暴露
RPC接口：调用方是其他微服务（不经过gateway），必须显式传`userId i64`

### 6.3 返回值遵循工程惯例

写操作返回`bool`（成功=true，失败走Kitex BizErr），参考：
- `bool MenuCreate(...)` — ser-buser
- `bool UserUpdate(...)` — ser-buser

查询操作返回Resp struct。

### 6.4 walletType字段统一定义

```
walletType i32:
    1 = 中心钱包 (center)     — 充值入这里
    2 = 奖励钱包 (reward)     — 奖金/赠送入这里
    3 = 代理钱包 (agent)      — 代理佣金入这里
    4 = 主播钱包 (anchor)     — 主播收益入这里
```

所有RPC写操作都需要指定walletType，因为余额按子钱包隔离。

## 七、RPC调用流程图

### 7.1 充值全链路

```
用户发起充值（C端）
    ↓ HTTP via gateway
ser-wallet: 创建充值订单 → 调 ser-finance RPC 获取支付配置
    ↓
ser-finance: 创建渠道支付订单 → 返回支付链接/地址
    ↓
用户支付 → 支付渠道回调
    ↓
ser-finance: 确认支付 → 调 ser-wallet RPC
    ↓
ser-wallet: DepositCredit(userId, currency, amount, refOrderNo, walletType=1)
    ↓ 事务内
    ① 幂等校验（refOrderNo）
    ② 余额变动（available + amount）
    ③ 写balance_flow
    ④ 更新充值订单状态
```

### 7.2 提现全链路

```
用户发起提现（C端）
    ↓ HTTP via gateway
ser-wallet: WithdrawFreeze(userId, currency, amount, refOrderNo, walletType)
    ↓ 事务内
    ① available - amount（乐观锁）
    ② frozen + amount
    ③ 写balance_flow
    ④ 创建提现订单（状态=审核中）
    ↓
ser-wallet: 调 ser-finance RPC 提交审核
    ↓
ser-finance: 三级审核（风控→财务→出款）
    ↓ 审核结果回调
    ├── 通过+出款成功 → 调 ser-wallet: WithdrawConfirm(refOrderNo)
    │   → frozen - amount → 写balance_flow → 订单状态=成功
    │
    └── 拒绝/失败 → 调 ser-wallet: WithdrawUnfreeze(refOrderNo)
        → available + amount, frozen - amount → 写balance_flow → 订单状态=失败
```

### 7.3 人工修正全链路

```
管理员操作（B端后台）
    ↓ HTTP via gate-back
ser-finance: 创建修正单 → 审核
    ↓ 审核通过后
ser-finance: 调 ser-wallet RPC
    ├── ManualCredit(userId, currency, amount, refOrderNo, walletType, operatorId)
    ├── ManualDebit(userId, currency, amount, refOrderNo, walletType, operatorId)
    └── ManualPatch(userId, currency, amount, refOrderNo, walletType, operatorId, originalOrderNo)
    ↓
ser-wallet: 统一走幂等校验 + 余额变动 + 流水
```

### 7.4 币种配置RPC调用

```
任何模块需要币种信息时
    ↓ Kitex RPC
rpc.WalletClient().AvailableCurrencyList(req)    → 获取可用币种
rpc.WalletClient().QueryCurrencyInfo(req)        → 获取币种详情
rpc.WalletClient().QueryExchangeRate(req)        → 获取汇率
rpc.WalletClient().QueryCurrencyConfig(req)      → 获取配置参数
```

## 八、开发优先级与分期对应

| 期次 | RPC暴露方法 | 数量 | 说明 |
|------|-----------|------|------|
| 第一期（骨架） | 无（只定义IDL、生成桩代码） | 0 | IDL提交后其他模块可编译WalletClient() |
| 第二期（币种） | AvailableCurrencyList + QueryCurrencyInfo + QueryExchangeRate + QueryCurrencyConfig | 4 | 币种RPC先行，其他模块可开始使用币种数据 |
| 第三期（核心） | DepositCredit + WithdrawFreeze/Confirm/Unfreeze + ManualCredit/Debit/Patch + QueryBalanceRpc | 8 | 财务模块可开始联调 |
| 第四期（充提） | 联调验证，无新增RPC | 0 | 与财务联调全链路 |

**关键节点：**
- 第一期完成后：财务可以`import walletservice`编译通过
- 第二期完成后：全工程可以查询币种和汇率
- 第三期完成后：**通知财务开始联调**（这是协同核心时间点）

## 九、需要主动确认的问题（"火"而非"调料"）

以下问题属于"没有火不能炒菜"级别，在编码RPC接口前必须确认：

| # | 问题 | 影响范围 | 确认方 |
|---|------|---------|--------|
| 1 | **ser-live的消费是否从钱包余额扣？** | 决定是否需要新增ConsumeDeduct RPC | 产品 |
| 2 | **主播收益是入钱包还是独立体系？** | 决定是否需要新增RevenueCredit RPC | 产品 |
| 3 | **游戏投注是否涉及钱包余额？** | 决定是否需要新增BetDeduct/BetSettle RPC | 产品 |
| 4 | **财务的IDL什么时候提供？** | 决定我调财务的4个RPC什么时候能编译通过 | 财务负责人 |
| 5 | **人工加减款是否走财务审核后再调钱包？** | 决定ManualCredit/Debit的调用方是财务还是管理后台直接调 | 产品+财务 |

**说明：** 问题1-3决定的是"是否需要新增RPC方法"。如果答案是"是"，则新增的方法本质上和DepositCredit/ManualDebit结构相同（加钱/扣钱），只是`changeType`枚举值不同。所以即使现在不新增专用方法，后续扩展的成本也很低——**只需扩展changeType枚举 + 复用现有余额操作逻辑**。

## 十、总结

### 10.1 一句话结论

ser-wallet需要暴露**12个确定的RPC接口**（8个余额操作 + 4个币种查询），调用方主要是**ser-finance**和**全工程公共**。直播/游戏/商城的消费扣款暂时不新增专用RPC，后续通过扩展changeType枚举即可覆盖。

### 10.2 数字速查

| 维度 | 数量 |
|------|------|
| 确定暴露的RPC方法 | 12个 |
| 财务调钱包 | 8个（写操作7个 + 查询1个） |
| 全局币种查询 | 4个（列表 + 详情 + 汇率 + 配置） |
| 待确认的潜在RPC | 4个（消费扣款 + 收益入账 + 批量查询 + 历史汇率） |
| 我调财务 | 4个（支付配置 + 创建订单 + 提交审核 + 通道查询） |
| RPC方法的IDL位置 | wallet_rpc.thrift(8个) + currency.thrift(4个) |
| 第二期完成暴露 | 4个（币种） |
| 第三期完成暴露 | 8个（余额操作） |

### 10.3 与IDL设计文档的一致性

本分析结果与IDL设计/c-5.md中已定义的12个RPC方法**完全一致**，无新增、无遗漏。说明之前的IDL设计已经充分考虑了跨模块需求。待确认的4个潜在RPC需要产品层面确认后再决定是否纳入。
