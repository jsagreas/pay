# ser-wallet RPC 暴露接口分析：我需要给谁提供什么能力

> 来源：20份 c-4 分析文档 + 工程代码级分析（RPC注册模式、IDL规范、客户端工厂） + 136张需求/原型文档
> 分析时间：2026-02-28
> 分析范围：ser-wallet 作为 RPC 服务端，需要暴露哪些接口给工程内其他模块调用
> 不包含：B端接口（gate-back路由）、C端接口（gate-font路由）——那些是网关层调用，不是模块间RPC
> 核心原则：余额只有钱包能改，所有涉及余额变更的模块都必须通过我的RPC

---

## 一、为什么要单独梳理RPC暴露接口

B端接口和C端接口走网关（gate-back / gate-font），由用户操作触发，路由已经在之前的链路关系文档中梳理清楚。

**RPC暴露接口是完全不同的东西**——它不是给用户用的，是给工程内部其他微服务用的。当财务模块完成充值回调验签后，它需要通过RPC告诉我"给这个用户上账"；当投注模块要扣用户余额时，它也必须通过RPC告诉我"扣这个用户的钱"。

这些接口的调用方不是网关，而是其他 ser-xxx 服务。调用方式是 Kitex RPC（TTHeader协议），通过ETCD服务发现找到 ser-wallet 实例。

**不梳理清楚这些，会出什么问题**：
- 别人不知道该调我哪个接口，联调时才发现没有 → 返工
- 接口定义不对齐（参数/返回值/语义），联调时才发现对不上 → 返工
- 遗漏关键接口，某条业务链路跑不通 → 阻塞整个项目

## 三、当前需求明确的RPC接口（12个）

这些接口在136张需求文档中有**明确的业务场景和调用链路**，是确定性骨架。

### 3.1 币种配置域 —— 4个RPC供他接口

这4个接口是**纯查询**，不改任何数据。是整个平台的"度量衡"数据源。

#### CC-R1. GetCurrencyList（获取已启用币种列表）

```
调用方：ser-finance（所有B端页面的币种下拉筛选）
触发时机：财务管理任何含币种筛选的页面加载时
调用频率：中频（B端操作频率）

入参：
  - status: optional   // 筛选状态（不传=只返回启用的，传"all"=全部）

出参：
  - list: []CurrencyInfo
      - currency_code: string     // "VND" / "BSB" / "USDT"
      - currency_name: string     // "越南盾"
      - currency_type: int8       // 1=法币 2=加密 3=平台币
      - currency_symbol: string   // "₫" / "$" / "₿"
      - decimal_places: int8      // 小数位数
      - thousand_sep: string      // 千分位符号
      - icon_url: string          // 币种图标
      - sort_order: int32         // 排序权重
      - status: int8              // 1=启用 0=禁用

使用场景：
  1. 财务B端 → 通道配置页 → 币种下拉筛选
  2. 财务B端 → 充值方式配置 → 币种下拉
  3. 财务B端 → 充值记录/提现记录 → 币种筛选
  4. 财务B端 → 统计报表 → 币种维度汇总
  5. 钱包内部 → 函数调用（不走RPC，零网络开销）
```

> 依据：财务管理产品原型 F 系列——所有页面均含币种筛选下拉框

#### CC-R2. GetExchangeRate（获取汇率）

```
调用方：ser-finance（报表换算）、未来可能有结算模块
触发时机：财务统计页面需要基准币种换算时
调用频率：中频

入参：
  - currency_code: string         // 查哪个币种的汇率
  - rate_type: optional int8      // 1=实时 2=平台 3=入款 4=出款（不传=返回全部）

出参：
  - currency_code: string
  - rate_realtime: string         // 实时汇率（decimal字符串）
  - rate_platform: string         // 平台汇率
  - rate_deposit: string          // 入款汇率
  - rate_withdrawal: string       // 出款汇率
  - updated_at: int64             // 最后更新时间戳

使用场景：
  1. 财务B端 → 充值记录表格 → "基准换算"列（金额换算为USDT展示）
  2. 财务B端 → 提现记录表格 → "基准换算"列
  3. 财务B端 → 统计报表 → 各币种金额统一换算为基准货币汇总
```

> 依据：财务管理需求文档 E3/E4——充值/提现记录表格含"基准换算"列

入参：无

#### CC-R4. GetCurrencyByCode（按代码查单个币种详情）

```
调用方：ser-finance（订单详情页展示币种信息）、其他需要币种详情的模块
触发时机：需要某个具体币种的完整信息时
调用频率：中频

入参：
  - currency_code: string     // "VND"

出参：
  - CurrencyInfo（同 CC-R1 的结构）

使用场景：

### 3.2 钱包域 —— 8个RPC供他接口

这8个接口中**7个是写操作**（改变余额），1个是查询。是整个资金系统的"账本操作入口"。

#### W-R1. GetUserWalletBalance（查用户钱包余额）

```
调用方：ser-finance（人工加减款弹窗显示当前余额）
触发时机：财务人员在B端创建加款/减款操作时
调用频率：低频（B端操作频率）

入参：
  - user_id: int64
  - currency_code: string     // 查哪个币种的余额

出参：
  - user_id: int64
  - currency_code: string
  - wallets: []WalletBalance
      - wallet_type: int8       // 1=中心 2=奖励 3=主播 4=代理 5=场馆
      - wallet_type_name: string // "中心钱包"
      - balance: string          // 可用余额（decimal字符串）
      - frozen_amount: string    // 冻结金额
  - total_available: string      // 资金钱包余额（中心+奖励）

使用场景：
  1. 财务B端 → 人工加款页面 → 弹窗展示"当前中心钱包余额: xxx，奖励钱包余额: xxx"
  2. 财务B端 → 人工减款页面 → 展示余额（减款不能超过实际余额）
  3. 财务B端 → 充值补单页面 → 展示用户当前余额信息

幂等要求：无（纯查询）
```

> 依据：依赖关系文档 5.4 人工修正链路——财务创建加减款时先调 GetUserWalletBalance 查余额展示在弹窗中

#### W-R2. CreditDeposit（充值上账）

```
调用方：ser-finance（充值成功回调后通知钱包入账）
触发时机：三方支付回调到达财务 → 财务验签成功 → 调用钱包上账
调用频率：高频（与充值笔数成正比）
重要程度：★★★★★ 直接影响用户余额

入参：
  - order_no: string           // 充值订单号（C+16位）—— 幂等键
  - user_id: int64
  - currency_code: string
  - amount: string             // 充值金额（decimal字符串）
  - bonus_activity_id: optional int64   // 奖金活动ID（无奖金不传）
  - bonus_ratio: optional string        // 奖金比例（如 "0.50" 表示50%）
  - bonus_amount: optional string       // 奖金金额（冗余传入，钱包侧验算）
  - turnover_multiple: optional int32   // 流水倍数要求（奖金对应的X倍流水）

出参：
  - success: bool
  - balance_after: string       // 上账后余额（供财务侧记录）
  - message: string             // 失败原因（如"订单已处理"）

业务逻辑要点：
  1. 幂等校验：同一 order_no 只入账一次（通过唯一索引保证）
  2. 充值金额 → 中心钱包 balance 增加
  3. 如果有奖金 → 奖励钱包 balance 增加（金额=充值×比例）
  4. 如果有奖金 → 记录稽核要求（X倍流水才能提现）
  5. 写入交易流水（tx_type=充值，含 balance_before / balance_after）
  6. 更新订单状态 → 已完成
  7. 全部在同一个数据库事务内

幂等要求：★★★★★ 必须（三方回调可能重复推送多次）
并发要求：同一用户同时充值 → SELECT FOR UPDATE 行级锁保护
```

> 依据：链路关系文档 3.3 充值链路——"财务验签成功 → RPC调钱包 CreditDeposit → 上账"；需求文档 A7——幂等性要求

#### W-R3. FreezeForWithdrawal（提现冻结）

```
调用方：ser-finance（创建提现订单后冻结金额）
        也被 ser-wallet 内部 CreateWithdrawalOrder 逻辑复用（函数调用，非RPC）
触发时机：提现订单创建后立即冻结
调用频率：与提现笔数成正比

入参：
  - order_no: string           // 提现订单号（T+16位）—— 幂等键
  - user_id: int64
  - currency_code: string
  - wallet_type: int8          // 从哪个子钱包冻结
  - amount: string             // 冻结金额

出参：
  - success: bool
  - frozen_balance: string     // 冻结后的冻结总额
  - available_balance: string  // 冻结后的可用余额
  - message: string            // 失败原因（如"余额不足"）

业务逻辑要点：
  1. 幂等校验：同一 order_no 只冻结一次
  2. 校验可用余额 ≥ 冻结金额（SQL级保证 balance >= amount）
  3. 原子操作：balance -= amount, frozen_amount += amount
  4. 写入交易流水（tx_type=提现冻结）

幂等要求：★★★★★ 必须
并发要求：SELECT FOR UPDATE 保护（防止超额冻结）
```

> 依据：链路关系文档 3.6 提现链路——"创建订单(T+16) → FreezeForWithdrawal → 冻结钱包金额"

#### W-R4. UnfreezeForWithdrawal（提现解冻）

```
调用方：ser-finance（审核拒绝 / 出款失败后解冻返还）
触发时机：
  - 风控审核拒绝
  - 财务审核拒绝
  - 通道出款失败回调
调用频率：低于冻结（不是所有提现都拒绝）

入参：
  - order_no: string           // 提现订单号 —— 幂等键
  - reason: string             // 解冻原因："风控拒绝" / "财务拒绝" / "出款失败"

出参：
  - success: bool
  - available_balance: string  // 解冻后可用余额
  - message: string

业务逻辑要点：
  1. 幂等校验：同一 order_no 只解冻一次
  2. 状态校验：该订单必须处于"已冻结"状态（终态不可再操作）
  3. 互斥校验：如果已被 ConfirmWithdrawalDeduct 处理 → 拒绝
  4. 原子操作：frozen_amount -= amount, balance += amount
  5. 写入交易流水（tx_type=提现解冻，记录解冻原因）
  6. 更新订单状态 → 已拒绝/已失败

幂等要求：★★★★★ 必须
互斥关系：与 W-R5 ConfirmWithdrawalDeduct 互斥（同一笔只走一条路）
```

> 依据：依赖关系文档 提现状态机——"审核拒绝/出款失败 → Unfreeze → 已解冻(返还)"

#### W-R5. ConfirmWithdrawalDeduct（提现确认扣款）

```
调用方：ser-finance（通道出款成功后确认扣款）
触发时机：代付通道回调出款成功
调用频率：与成功提现笔数成正比

入参：
  - order_no: string           // 提现订单号 —— 幂等键

出参：
  - success: bool
  - message: string

业务逻辑要点：
  1. 幂等校验：同一 order_no 只确认扣款一次
  2. 状态校验：该订单必须处于"已冻结"状态
  3. 互斥校验：如果已被 UnfreezeForWithdrawal 处理 → 拒绝
  4. 原子操作：frozen_amount -= amount（不恢复到 balance，永久扣除）
  5. 写入交易流水（tx_type=提现完成，含 balance_before / balance_after）
  6. 更新订单状态 → 已完成

幂等要求：★★★★★ 必须
互斥关系：与 W-R4 UnfreezeForWithdrawal 互斥
这是提现链路的终点接口：执行后资金从用户钱包永久移出
```

> 依据：链路关系文档 3.7——"出款成功 → ConfirmWithdrawalDeduct → 扣除冻结金额"

#### W-R6. ManualCredit（人工加款）

```
调用方：ser-finance（加款审核通过后通知钱包加钱）
触发时机：财务B端创建加款 → 另一人审核通过 → 调用钱包
调用频率：低频（运营人工操作）

入参：
  - order_no: string           // 加款订单号（A+16位）—— 幂等键
  - user_id: int64
  - currency_code: string
  - items: []ManualCreditItem
      - wallet_type: int8       // 加到哪个子钱包
      - amount: string          // 加款金额
  - operator_id: int64          // 操作人（审核人）
  - remark: string              // 加款原因

出参：
  - success: bool
  - message: string

业务逻辑要点：
  1. 幂等校验：同一 order_no 只加款一次
  2. 按 items 列表分别为每个子钱包增加余额
  3. 每个子钱包变动各写一条交易流水
  4. 全部在同一事务内（多个子钱包变动是原子的）

幂等要求：★★★★ 必须（审核系统可能重试）

为什么入参是 items 列表：
  需求文档人工加款页面——可以同时向多个子钱包（中心/奖励/代理/主播）分别加不同金额
```

> 依据：需求理解文档 6.3——"人工加款审核通过 → 按钱包类型分别加钱"

#### W-R7. ManualDebit（人工减款）

```
调用方：ser-finance（减款审核通过后通知钱包扣钱）
触发时机：财务B端创建减款 → 另一人审核通过 → 调用钱包
调用频率：低频

入参：
  - order_no: string           // 减款订单号（M+16位）—— 幂等键
  - user_id: int64
  - currency_code: string
  - items: []ManualDebitItem
      - wallet_type: int8       // 从哪个子钱包扣
      - amount: string          // 减款金额
  - operator_id: int64
  - remark: string

出参：
  - success: bool
  - actual_deducted: []DeductResult  // 每个子钱包实际扣了多少
      - wallet_type: int8
      - requested: string        // 请求扣多少
      - actual: string           // 实际扣了多少（余额不足时=实际余额）
  - message: string

业务逻辑要点：
  1. 幂等校验：同一 order_no 只减款一次
  2. 按 items 列表分别扣各子钱包
  3. 安全保护：余额不足时扣实际余额（不允许负数），返回实际扣减金额
  4. SQL级保证：UPDATE SET balance = 0 WHERE balance < amount（不够就清零）
     或者 UPDATE SET balance = balance - LEAST(balance, amount)
  5. 每个子钱包变动各写一条交易流水

幂等要求：★★★★ 必须
特殊处理：不拒绝余额不足——扣能扣的，返回实际值
```

> 依据：需求理解文档 6.3——"人工减款审核通过 → 按钱包类型分别扣钱，余额不足扣实际余额"

#### W-R8. CreditCorrection（充值补单上账）

```
调用方：ser-finance（充值补单审核通过后通知钱包入账）
触发时机：原始充值失败 → 财务创建补单（B+16位）→ 审核通过 → 通知钱包
调用频率：低频（异常流程）

入参：
  - order_no: string           // 补单订单号（B+16位）—— 幂等键
  - original_order_no: string  // 原始充值订单号（C+16位）
  - user_id: int64
  - currency_code: string
  - amount: string             // 补单金额
  - bonus_amount: optional string      // 补单关联的赠送金额
  - turnover_multiple: optional int32  // 流水倍数要求
  - operator_id: int64
  - remark: string

出参：
  - success: bool
  - message: string

业务逻辑要点：
  1. 幂等校验：同一 order_no 只补单一次
  2. 补单金额 → 中心钱包（与 CreditDeposit 类似）
  3. 赠送金额 → 奖励钱包（如果有）
  4. 写入交易流水（tx_type=充值补单）

幂等要求：★★★★ 必须
与 CreditDeposit 的区别：
  - CreditDeposit 是正常充值回调触发，order_no 是 C 开头
  - CreditCorrection 是人工补单触发，order_no 是 B 开头
  - 业务语义不同，但余额操作逻辑相似
```

> 依据：需求理解文档 6.3——"补单审核通过 → 补单金额+赠送入账"

### 5.1 财务 → 钱包（8个，主要是写操作）

```

场景3：提现审核拒绝 / 出款失败
  风控/财务审核拒绝 或 通道出款失败 → UnfreezeForWithdrawal(W-R4) → 解冻返还

场景4：出款成功
  通道出款成功回调 → ConfirmWithdrawalDeduct(W-R5) → 确认扣款

场景5：人工加款审核通过
  财务人员审核通过 → ManualCredit(W-R6) → 按子钱包加钱

场景6：人工减款审核通过
  财务人员审核通过 → ManualDebit(W-R7) → 按子钱包扣钱

场景7：充值补单审核通过
  财务人员审核通过 → CreditCorrection(W-R8) → 补单入账

场景8：加减款前查余额
  财务人员打开加/减款弹窗 → GetUserWalletBalance(W-R1) → 展示当前余额

## 七、12个确定接口的横向对比

### 7.1 按调用触发方式分类

| 触发方式 | 接口 | 说明 |
|---------|------|------|
| **三方回调触发** | CreditDeposit, ConfirmWithdrawalDeduct, UnfreezeForWithdrawal | 外部事件驱动，可能重复 |
| **人工审核触发** | ManualCredit, ManualDebit, CreditCorrection | B端操作，频率低 |
| **用户操作触发** | FreezeForWithdrawal | 用户发起提现时内部调用 |
| **页面加载触发** | GetUserWalletBalance, GetCurrencyList, GetExchangeRate, GetBaseCurrency, GetCurrencyByCode | 查询类，可缓存 |

### 7.2 按幂等要求分级

| 级别 | 接口 | 原因 |
|------|------|------|
| **★★★★★ 绝对幂等** | CreditDeposit, FreezeForWithdrawal, UnfreezeForWithdrawal, ConfirmWithdrawalDeduct | 三方回调可能重复推送 |
| **★★★★ 必须幂等** | ManualCredit, ManualDebit, CreditCorrection | 审核系统可能重试 |
| **★★★ 天然幂等** | 全部查询接口（W-R1, CC-R1~R4） | 查询本身幂等 |

### 7.3 按并发风险分级

| 级别 | 接口 | 原因 | 保护手段 |
|------|------|------|---------|
| **高** | CreditDeposit, FreezeForWithdrawal | 同一用户同时充值/提现 | SELECT FOR UPDATE |
| **中** | UnfreezeForWithdrawal, ConfirmWithdrawalDeduct | 互斥操作（同一订单不会同时解冻和扣款） | 状态机 + 唯一索引 |
| **低** | ManualCredit, ManualDebit, CreditCorrection | 人工操作频率低 | 幂等键即可 |
| **无** | 全部查询接口 | 只读 | 无需 |

### 7.4 按互斥关系分组

```
互斥组1：提现终态
┌─────────────────────────────────────────────────┐
│  FreezeForWithdrawal（冻结）                      │
│        │                                         │
│        ├──→ UnfreezeForWithdrawal（解冻）         │  ← 二选一
│        └──→ ConfirmWithdrawalDeduct（扣款）       │  ← 二选一
│                                                  │
│  同一个 order_no 只能走其中一条路，不可逆          │
└─────────────────────────────────────────────────┘

互斥组2：上账类（不互斥，但业务语义不同）
┌─────────────────────────────────────────────────┐
│  CreditDeposit      → 正常充值回调               │
│  CreditCorrection   → 充值补单（人工）            │
│  ManualCredit       → 人工加款（非充值）           │
│                                                  │
│  三者操作同一个用户余额，但 order_no 前缀不同     │
│  C开头 / B开头 / A开头，天然不冲突                │
└─────────────────────────────────────────────────┘
```

    wallet.GetUserWalletBalanceResp   GetUserWalletBalance(1: wallet.GetUserWalletBalanceReq req)

    wallet.CreditDepositResp          CreditDeposit(1: wallet.CreditDepositReq req)
    wallet.FreezeForWithdrawalResp    FreezeForWithdrawal(1: wallet.FreezeForWithdrawalReq req)
    wallet.UnfreezeForWithdrawalResp  UnfreezeForWithdrawal(1: wallet.UnfreezeForWithdrawalReq req)
    wallet.ConfirmWithdrawalDeductResp ConfirmWithdrawalDeduct(1: wallet.ConfirmWithdrawalDeductReq req)

## 十、关键约束与注意事项

### 10.1 所有写操作的通用约束

```
1. 幂等性：order_no 作为唯一键，重复调用返回"已处理"而非报错
2. 事务性：余额变更 + 流水写入必须在同一个数据库事务内
3. 非负性：SQL级保证 balance >= 0（UPDATE ... WHERE balance >= amount）
4. 流水性：每次余额变动必须写入 wallet_transaction（对账依赖）
5. 审计性：所有写操作都调 ser-blog.AddActLog 记录审计日志（oneway）
```

### 10.2 不存在循环调用

```
充值链路：用户 → 钱包 → 财务(查方式) → 返回    // 链路结束
          后续：财务回调 → 钱包(上账)             // 新的独立链路

提现链路：用户 → 钱包 → 财务(查方式) → 返回    // 链路结束
          后续：财务审核 → 钱包(解冻/扣款)       // 新的独立链路

不存在：A调B的同时B回调A 的情况
```

> 依据：依赖关系文档 六、双向依赖分析——"不存在循环调用"

### 10.3 接口扩展性预留

当前12个RPC接口的设计不需要为未来模块做"提前实现"，但需要在IDL层面预留扩展空间：

```
已经足够通用，未来可直接复用的：
  - GetUserWalletBalance  → 投注模块查余额可以直接用
  - GetCurrencyList       → 任何模块的币种下拉都能用
  - GetExchangeRate       → 任何模块的汇率换算都能用

未来需要新增的（当前不实现，但知道会来）：
  - DeductForBet          → 投注扣款（和 ManualDebit 语义不同）
  - CreditSettlement      → 结算返奖入账（和 CreditDeposit 语义不同）
  - DeductForGift         → 打赏扣款
  - CreditStreamer        → 主播收益入账

为什么不现在就做：
  1. 需求未明确，参数设计会猜测 → 违反"禁止想当然"准则
  2. 不阻塞当前开发进度
  3. 真正需要时再加，成本不高（加一个RPC方法+handler+service）
```
