## 5. 反思：哪些需要提升

### 5.1 历史教训回顾

| 错误 | 场景 | 根因 | 教训 |
|------|------|------|------|
| `make gen` 想当然 | 描述开发流程时 | 没有验证工程中的实际脚本名 | **回答必须基于工程中的真实路径和脚本** |
| RPC超时注释错误 | init.go分析时 | 注释写"3秒"但代码是1200s | **看代码本身，不信注释** |
| 图片敷衍标记"不可读" | 产品原型解析时 | 没有尽最大努力解析 | **先尽最大努力，确实无法才报告** |

### 5.2 本轮需要改进的点

**I1. 大文件输出的截断风险**

参考工程分析中，Agent输出超过65KB被截断，需要手动补救读取源码。

- 改进：对于超大分析任务，应预判输出规模，拆分为多个Agent分工，每个聚焦一个子领域，避免单Agent输出过大

**I2. 重复读取的效率问题**

部分文件被Agent和我分别读取了一次（如WalletTrxBusiness.java），存在重复IO。

- 改进：更好的分工规划——Agent负责探索发现，我负责关键文件精读，避免两者重叠

**I3. 对"最有价值"的敏感度**

你在参考B的需求中明确说了"最有价值的就是虚拟币trc20充值完整实现思路"，我确实将其作为第5章重点，但可以更早在分析规划阶段就围绕这个核心组织探索任务，而不是先全面探索再聚焦。

- 改进：先抓用户明确标注的"最高价值点"，围绕它深度展开，再辐射到周边

**I4. 文档间的交叉引用**

11份文档之间虽然有逻辑递进关系，但缺少显式的交叉引用链接。比如模块参考B中提到"用参考A的CAS安全机制"，可以直接引用`模块参考A/c-1.md 第X章`。

- 改进：在后续文档中增加跨文档引用，形成知识网络而非孤立文档

**I5. 主动总结节奏**

你需要主动提出"复盘总结"才触发了这个回顾。理想状态下，我应该在完成阶段性工作后主动提议复盘。

## 6. 认知升级：从执行中提炼的经验

### 6.1 关于"深度分析"的认知

**之前的理解**：深度 = 读更多文件、产出更多文字

**现在的理解**：深度 = **对关键路径的穿透性理解**

参考B的分析让我理解到，538行的WalletTrxBusiness.java一个文件的价值，可能超过读100个配置文件。深度不是广度的堆叠，而是对核心链路（充值17步、提现流程、归集机制）的完整穿透。

**实践原则**：先找到核心链路 → 沿链路逐节点穿透 → 再横向扩展到关联模块

### 6.2 关于"借鉴参考"的认知

**之前的理解**：借鉴 = 看看别人怎么做的，列个表

**现在的理解**：借鉴 = **提取设计思想 + 评估适配性 + 标注改进点**

两个参考工程的分析让我形成了一套方法论：

```
借鉴三层模型：
├─ 第1层：看到什么（事实描述）
│   "参考B用AES/ECB加密私钥"
├─ 第2层：评估什么（判断）
│   "ECB模式不安全，相同明文产生相同密文"
└─ 第3层：我们怎么做（转化）
    "采用AES/GCM模式，带认证的加密"
```

只有到第3层才算完成了"借鉴"，停在第1层只是"抄写"。

### 6.3 关于"完整性"的认知

**之前的理解**：完整 = 没有遗漏文件

**现在的理解**：完整 = **闭环**

参考B分析中，如果只看wallet-server不看user-server，就永远理解不了TRC20充值的完整链路——因为充值的触发、余额判断、订单管理、超时取消都在user-server中。

**实践原则**：分析一个功能时，必须沿着调用链追到上下游，直到链路闭合

### 6.4 关于"有依据"的认知

**之前的理解**：有依据 = 引用文件名

**现在的理解**：有依据 = **结论←代码←行号←逻辑推理**

```
错误示范：
  "参考B使用了Gas费补充机制"（结论，无依据）

正确示范：
  "参考B在TrxDriver.java第166-178行，singleCollect()方法中，
   对TRC20操作前检查用户地址的TRX余额是否<30，
   如果不足则从gasAddress（通过AwsKmsDriver解密获取）
   向用户地址转入30TRX作为Gas费。"（结论+代码+位置+逻辑）
```

### 6.5 关于"两个参考工程"的互补认知

```
参考A（p9-services）的核心贡献：
├─ 余额安全模型（CAS乐观锁 + SQL WHERE检查）
├─ 冻结/解冻/回滚三步模式
├─ 流水审计字段（coinFrom/coinTo/balanceType）
└─ 适用场景：内部余额管理（我们的"钱包余额"部分）

参考B（tg-services）的核心贡献：
├─ TRC20链上充提完整实现
├─ 四地址体系 + 归集机制
├─ 冷热签名分离安全设计
├─ Gas费自动管理
└─ 适用场景：链上资产操作（我们的"虚拟币充提"部分）

整合：A的余额安全 + B的链上操作 = ser-wallet的完整设计
```

这种**互补式借鉴**比单一参考更有价值，因为没有任何一个参考工程能完美匹配我们的需求。

| 幂等性 | 充值/提现的重复请求 | 唯一索引 + 幂等token |

```
3. 面对用户强调的重点：优先聚焦，围绕核心展开
4. 面对不确定的信息：标注"待确认"，不混淆确定与推测
5. 面对错误：承认、修正、记录教训，不掩盖不辩解
6. 面对下一阶段：延续"信得过、可靠、负责"的定位
```
