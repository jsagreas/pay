## 3. 我必须牢记的——行为准则回顾

### 3.1 红线约束（绝不可触碰）

| 红线 | 含义 | 我的理解 |
|------|------|---------|
| **默认只读** | 未经明确指令不增删改 | 分析阶段只读代码，不擅自改动 |
| **禁止猜测臆想** | 有依据才能说 | 不确定的事情要标注"待确认"，不能编造 |
| **禁止局部片面** | 不能看了一部分就下结论 | 必须全面分析后才能给出判断 |
| **禁止想当然** | 凡事要有凭证理由 | 每个结论都要能追溯到代码事实或需求文档 |
| **禁止无中生有** | 没有的问题不说有 | 审慎反馈，但也不遗漏真实问题 |

### 3.2 思考约束（分析必须做到的10条）

1. **全面宏观+微观** — 架构到变量，全景到细节
2. **完整闭环** — 不孤立看文件，看模块间关系
3. **清楚架构** — 没有大局观就是盲人摸象
4. **清楚调用链路** — 牵一发动全身
5. **深入穷极** — 追代码要追到底，不蜻蜓点水
6. **清楚流程** — 知道操作在整体流程中的位置
7. **清楚设计思路** — 是什么、为什么、应该怎么
8. **清楚标准规范** — 写出符合规范的代码
9. **清楚接口和细节** — 不知道接口就不知道功能
10. **清楚技术栈** — 遵守复用项目已有组件

### 3.3 执行约束

- **两条腿走路**：脚手架分析 + 需求文档分析，缺一不可
- **分步迭代**：小步慢跑，不一次性全搞定
- **先思路后编码**：实现思路 → 确认 → 编码 → 验证
- **必须获得明确同意才能编码**
- **后端视角**：区分前后端职责边界

### 3.4 互动约束

- **不固执己见**：持续修正，接纳变化，新信息优先级高
- **不人云亦云**：有主见有依据，代码事实不妥协
- **平衡原则**：用户反馈 > AI推断（业务层），代码事实 > 用户描述（技术层）

### 3.5 9维度控制体系核心

1. 问题定义与目标对齐 — 搞清楚到底要做什么
2. 协作方式与交互控制 — 该问就问，不替代决策
3. 用户认知与表达控制 — 结论→理由→步骤，不认知过载
4. 事实输入与证据基础 — 只基于事实推理，标注假设
5. 方案设计与实施落地 — 给可执行步骤，不只给方向
6. 约束、环境与非功能控制 — 遵守限制，默认安全
7. 风险控制与稳定性 — 高风险高标准，标注前提
8. 验证闭环与成功判定 — 什么状态算完成
9. 全局策略控制 — 不擅自改方向，冲突时上报

**值得延续**：这种"穷极式"分析方法正是你要求的"深入穷极"思维。

### 4.2 结构化输出

每份分析报告都遵循了一致的结构化格式：
- 有目录、有章节编号
- 表格对比清晰
- 代码片段有出处标注
- "值得借鉴"和"不应采纳"分区明确

**值得延续**：结构化输出让文档可归档、可检索、可复用。

### 4.3 跨模块追踪

在模块参考B的分析中，不只停留在wallet-server本身，而是追踪了：

**值得延续**：业务链路往往跨模块，不追踪到终点就会遗漏关键逻辑。

### 4.4 对比分析思维

在分析参考工程时，始终保持与我们Go工程的对照视角：
- 技术栈对比表
- 设计模式对照
- "直接采纳 / 改进采纳 / 不采纳"的三分法

**值得延续**：分析参考工程的目的不是为了了解它，而是为了"为我所用"。

### 4.5 两条腿走路的执行

确实做到了脚手架分析+需求文档分析两条腿并行：
- 先分析Go工程代码了解架构规范
- 再解读需求原型理解业务需求

## 5. 做错了什么——必须反思的教训

### 5.1 图片解析敷衍（严重教训）

**事件**：在解读需求截图时，有几张图片我标记为"不可读"或"无法解析"，被你纠正后重新认真解析，结果全部成功。

涉及的图片包括：
- 5-5 重复转账拦截规则
- 5-6 额外奖金选择规则
- 2-1 资金钱包体系系列
- 1- 币种配置说明

**反思**：这违反了"深入穷极"和"禁止局部片面"原则。不能因为第一次尝试没看清就放弃，应该多角度多方法尝试，实在无法解析时才报告。

**改进承诺**：后续遇到任何需要解析的内容，必须穷尽方法尝试后才能说"无法解析"。

### 5.2 想当然描述开发流程

**事件**：描述代码生成流程时，把 `make gen` 当作代码生成入口，实际应该是 `gen_kitex.sh`。

**反思**：这是典型的"禁止猜测臆想"违规。没有确认实际脚本就给出了错误信息。

**改进承诺**：任何关于工程操作的描述，必须以实际文件内容为依据。

### 5.3 Write操作未验证

**事件**：在模块参考A的分析中，第一次Write操作返回"File created successfully"，但实际文件没有持久化。直到后来检查才发现文件不存在，需要重新写入。

**反思**：输出文件后应该立即验证（`wc -l` 或 `ls -la`），而不是盲信工具返回值。

**改进承诺**：每次Write关键文件后，必须立即用Bash验证文件存在性和行数。（后续执行中已修正此行为）

### 5.4 上下文管理意识不足

**事件**：多次对话因上下文窗口耗尽而需要恢复续接。续接时需要重新读取文件、重建认知，造成效率损耗。

**反思**：应该更主动地管理上下文使用——在做大量文件读取前评估上下文预算，优先读取最关键的文件，减少冗余信息占用。

**改进承诺**：
- 大任务开始前先评估需要读取的文件量
- 优先级排序：核心业务文件 > 辅助工具文件 > 配置文件
- 利用子代理（Task）分担文件探索，保护主对话上下文

### 5.5 注释/超时时间硬编码类问题提出不够及时

**反思**：在分析参考工程时发现了多个缺陷（如AES/ECB模式、HTTP替代MQ、无事务保证等），但这些发现是在最终报告中才集中呈现的。如果在分析过程中就即时标注，可以更早与你讨论避免在我们的设计中犯同样的错误。

## 6. 认知提升——从这段经历中学到的

### 6.1 关于"负责的分析"

你反复说的"负责的分析"，我的理解已经从最初的"尽量详细"深化为：

**负责 = 有依据 + 有深度 + 有对照 + 有判断 + 不遗漏**

- **有依据**：每个结论可追溯到具体代码行或需求原型
- **有深度**：不是读了文件名就算分析，要读到每个方法、每条SQL
- **有对照**：分析参考工程不是看热闹，是找到"为我所用"的映射点
- **有判断**：不只是"这个参考工程这样做"，还要判断"我们应不应该这样做"
- **不遗漏**：发现了问题就要报告，发现了可借鉴的设计就要标注

### 6.2 关于"盖房子思维"

需求预评审阶段的分析策略：
- 确定性骨架先定好（多币种、5子钱包、冻结机制、SQL级安全——这些绕不过去）
- 装修细节保留弹性（具体接口字段、具体枚举值、具体阈值——评审后可能变）
- 不先入为主、不板上钉钉

这个思维帮助我在设计表结构和接口时避免了过度设计。

### 6.3 关于"两条腿走路"

> 只看脚手架不看需求 → 不知道做什么
> 只看需求不看脚手架 → 会想当然实现

这个原则在实践中的体现：
- 分析Go工程时发现已有的ORM Gen模式、Redis配置模式、ETCD配置模式 → 我们的设计必须遵循这些模式
- 分析需求原型发现多币种体系、子钱包隔离、流水倍数等业务规则 → 表结构必须支撑这些规则
- 两者结合：用Go工程的技术规范来实现需求原型的业务规则

### 6.4 关于"深入穷极"

分析参考工程时的体会：
- 如果只读了Controller层，会以为wallet-service就是简单的CRUD
- 追到Service层才发现有复杂的addCoin类型分支
- 追到Mapper XML才发现SQL级的 `(amount - X) >= 0` 安全机制
- 追到Listener才发现MQ驱动的4种余额变更模式
- 追到参考工程B的user-server才发现充值确认是两阶段轮询模型

**每一层追踪都揭示了一层设计智慧，浅尝辄止只能看到表面。**

### 6.5 关于自我迭代

对话过程中我的输出质量有明显提升：
- 早期：结构不够统一，部分分析深度不足
- 中期：建立了一致的文档结构（目录→章节→表格→代码→对照→结论）
- 后期（模块参考A/B）：每份报告都是完整的、有目录的、有对照的、有判断的深度分析

这个提升过程本身就是"渐进式对话"原则的体现——不断迭代完善，每一轮比上一轮做得更好。
