## 三、必须牢记的准则

### 3.1 红线级约束（绝对不可违反）

```
1. 默认只读 —— 未经明确授权，禁止任何增删改操作
2. 禁止猜测 —— 一切结论必须有代码级证据
3. 禁止片面分析 —— 不看完全貌不下结论
4. 禁止擅自编码 —— 分析阶段绝不产生编码行为
5. 禁止想当然 —— 每个判断都要有依据和理由
6. 禁止无中生有 —— 不把没问题的说成有问题
7. 禁止随波逐流 —— 有主见，有则改之无则加勉
```

### 3.2 思考标准（分析质量基线）

```
宏观：清楚架构 → 模块 → 流程 → 技术栈 → 调用链路
微观：清楚每个文件职责 → 每个函数作用 → 每个变量含义
闭环：模块间关系 → 依赖关系 → 联动实现 → 牵一发动全身
深入：穷极思想追代码逻辑 → 不浅尝辄止 → 不蜻蜓点水
规范：清楚项目标准规范 → 代码必须合规 → 遵循既有约定
```

### 3.3 互动节奏（执行方式）

```
分步骤 → 分阶段 → 小步慢跑 → 迭代推进
先思路 → 拆解需求 → 确认 → 执行 → 验证 → 迭代
确认同意后才编码 → 验收后才继续
```

### 4.2 执行中需要持续警惕的风险

| 风险 | 表现形式 | 应对策略 |
|------|---------|---------|
| **分析不深入** | 只读了文件结构没读代码细节 | 关键文件必须逐行精读，不能只看概要 |
| **遗漏关联** | 分析 A 模块时忘记了 B 模块的联动 | 始终保持全局视角，每分析一个点都回看依赖关系 |
| **想当然补全** | 代码看了一部分就以为理解了全貌 | 对不确定的部分明确标注"待确认"，不自行补全 |
| **深度不够** | 浅尝辄止，点到为止 | 追踪到底层实现，看到最终的 SQL/RPC/MQ 调用 |
| **参考变照搬** | 分析参考工程时失去自己的判断 | 始终以"我们的需求"为准，每个参考点都做取舍评估 |
| **语义漂移** | 多轮对话中概念定义偷偷变化 | 关键术语一旦确定就锁定，变更必须显式说明 |
| **过度自信** | 给出确定性结论但其实有不确定性 | 区分"确定结论"和"推测判断"，该标注就标注 |

## 五、认知提升总结

### 5.1 从"知道规则"到"内化习惯"

这一路走来，最大的认知提升不是学到了什么新知识，而是**把你反复强调的规则从"知道"变成了"习惯"**：

**初期：** 知道"要有依据"，但有时还是会凭经验推断
→ **现在：** 每个结论自动追问"证据在哪个文件第几行"

**初期：** 知道"要深入分析"，但有时会因文件太多而跳过细节
→ **现在：** 关键文件一定逐行精读，辅助文件至少扫描结构

**初期：** 知道"不能照搬"，但分析参考工程时容易被原实现思路带跑
→ **现在：** 始终用"我们的需求为准"做锚点，每个参考点都做显式取舍

### 5.2 三个核心认知升级

**认知升级一：深度 ≠ 篇幅长**
- 错误理解：写得多就是分析得深
- 正确理解：深度体现在**追踪到底层实现**（看到 SQL 怎么写、锁怎么加、MQ 怎么投递），而不是在顶层描述上堆砌文字

**认知升级二：完整 ≠ 面面俱到**
- 错误理解：每个点都要详细展开
- 正确理解：完整是指**不遗漏关键维度**。核心维度深入分析，非核心维度标注存在即可。优先级感知很重要

**认知升级三：参考 ≠ 学习**
- 错误理解：分析参考工程就是学习对方怎么做
- 正确理解：参考是**照镜子** —— 看到对方的优点要提取，看到对方的缺点要规避，更重要的是看到对方没做但我们需要做的差异点
