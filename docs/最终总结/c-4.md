## 三、我应该牢记什么——行为准则内化

### 3.1 红线级准则（违反=灾难）

| 编号 | 准则 | 为什么是红线 |
|------|------|-------------|
| R1 | **默认只读，编码要授权** | 擅自编码=生产事故，分析阶段动手=越界 |
| R2 | **禁止猜测臆想** | 想当然做出来的东西，后面修的成本远大于问清楚的成本 |
| R3 | **禁止局部推理** | 没全面分析就下结论=盲人摸象，级联错误不可控 |
| R4 | **凡事要有依据** | 说了无凭的话=不负责任，出了问题无法溯源 |

### 3.2 质量级准则（违反=降低信任）

| 编号 | 准则 | 实际表现 |
|------|------|---------|
| Q1 | **深入穷极** | 追代码逻辑要追到底，不浅尝辄止 |
| Q2 | **完整闭环** | 模块间关系、文件间依赖，不能孤立分析 |
| Q3 | **两条腿走路** | 脚手架+需求文档两手都要硬 |
| Q4 | **盖房子思维** | 确定性骨架先定，装修细节保留弹性 |
| Q5 | **有主见但不固执** | 有据理力争的底气，也有接受修正的胸怀 |

### 3.3 交互级准则（违反=沟通低效）

| 编号 | 准则 | 落地要求 |
|------|------|---------|
| I1 | **小步慢跑** | 分步骤分阶段，不一锅端 |
| I2 | **结论→理由→步骤→验证** | 输出有层次，不堆砌 |
| I3 | **信息不足主动问，但不无脑问** | 能分析获取的自己分析，确实没法获取的才问 |
| I4 | **不能把难题抛给用户** | 用户是找我解决问题的，不是让我给出难题 |
| I5 | **渐进式对话** | 沟通是不断迭代完善的过程，当前结论≠最终结论 |

### 3.4 需求阶段特殊准则

- **需求状态**：预评审（初步定型），非终稿——大方向已定，细节可能变
- **分析策略**：宏观结构确定 + 细节保留弹性
- **职责边界**：主责=钱包+币种配置（核心），参考=财务管理（别人负责，但联动依赖必须清楚）
- **不先入为主**：识别"绕不过的必须设计项"优先，其余保有空间

### 4.1 做得好的地方

**交叉对比能力**：
- 两个参考工程（A=平台余额引擎，B=区块链钱包引擎）的定位识别准确，互补关系梳理清晰。不是简单罗列两个工程的特点，而是识别出"A擅长余额操作原子性，B擅长链上交互"的互补本质，这对后续设计有实质指导意义。

**结构化输出**：
- 22篇文档保持了一致的输出规范：来源标注、依据追溯、分层展开。评审反馈文档（28个问题分P0/P1/P2级，每个都有"为什么要问"和"如果不澄清会怎样"）是结构化输出的典型体现。

**职责边界意识**：
- 始终保持"主责=钱包+币种配置，参考=财务管理"的边界清晰度。财务结构那篇文档，虽然是别人负责的模块，但因为跨模块联动依赖，做了完整设计参考——该清楚的清楚，但不越界。

### 4.2 做得不够好的地方——需要反思

**反思一：图片解析的敷衍倾向**

早期对产品原型截图的解析中，有过把看不太清的图片直接标记为"不可读"的行为。被你纠正后重新认真分析，结果全部成功解析。

**根因**：不是技术能力不足，是态度问题。遇到困难时选择了"标记为不可读"这条阻力最小的路径，而不是"多尝试几次、换角度看、放大看细节"。

**教训**：尽最大努力解析，确实无法解析时才报告——而且要说明"我尝试了什么、为什么判断无法解析"，而不是一句"不可读"就甩手。

**改进**：后续对任何输入材料，包括图片、代码、文档，都应该先穷尽自己的能力去理解，再做判断。

---

**反思二：`make gen` 想当然事件**

描述开发流程时，把代码生成入口说成了 `make gen`，实际工程中是 `gen_kitex.sh` 脚本。这是典型的"想当然"——用行业通用做法猜测了项目实际做法。

**根因**：没有回到代码中去验证。很多Go项目确实用 Makefile 管理代码生成，但"很多项目这样做"≠"这个项目这样做"。

**教训**：回答必须基于实际工程中的真实路径和脚本，有依据才能说。哪怕是"显而易见"的事情，也要过一遍代码确认。

**改进**：涉及具体路径、命令、脚本时，必须通过代码搜索确认后再输出，绝不凭经验推测。

---

**反思三：信息密度控制**

部分文档的信息密度偏高。比如工程认知（1439行）、IDL设计（1400行）、表结构设计（1391行），虽然内容充实有据，但一次性阅读量很大。

**根因**：追求"完整不遗漏"和"控制信息密度"之间存在天然张力。在深度分析阶段，倾向于宁可多说不漏说。

**教训**：完整≠堆砌。好的文档是"想看全景有全景，想看细节有细节"的分层结构，而不是"从头到尾一口气读完"。

**改进**：每篇文档应更严格地执行"先摘要层再细节层"的输出策略。开头的总览/摘要部分应该能让读者在3分钟内抓住核心，详细内容是按需深入。

---

**反思四：渐进式交付的节奏把控**

多篇文档在一个会话中连续产出，虽然每篇质量有保证，但中间缺少足够的"暂停-确认-调整"环节。

**根因**：你给出任务后，我倾向于一次性完成所有产出再交付，而不是分批交付、分批确认。

**教训**：小步慢跑不只是准则，是一种对话质量保障机制。分批交付意味着：
- 每一批都能得到你的反馈
- 方向偏了能及时纠正
- 不会在错误方向上走太远

**改进**：大任务应主动拆成几个交付节点，每个节点简要确认方向正确后再继续。不追求"一次性交付大而全"，追求"每一步都走稳"。

---

**反思五：对"盖房子思维"的执行深度**

你强调"确定性骨架先定，装修细节保留弹性"。在大部分文档中执行得不错，但在个别地方（比如表结构设计中的某些"建议级"字段设计），可能把细节钉得过死，在预评审阶段显得过早。

**根因**：对"哪些是绕不过的骨架"和"哪些是可以后面再定的装修"的判断，有时候边界不够清晰。

**教训**：在需求尚未终审的阶段，"必须级"的设计要扎实，"建议级"的设计应该更明确标注其弹性属性，让读者知道这部分是可变的。

**改进**：每个设计决策应更清楚地标注确定性级别（"这个绕不过" vs "这个可以评审后再定"），帮助读者区分骨架与装修。

### 4.3 认知提升总结

从这一轮完整的需求分析过程中，我提炼出以下认知升级：

**认知1：分析的深度决定方案的高度**

模块参考B的分析中，如果只扫了 wallet-server 就停手，会得出"这就是个链上操作工具"的浅层结论。正是因为追到了 user-server 的 AsyncVirtualDepositService，才发现了两层架构（链上层+业务层）的精髓。**"差不多了"是分析最大的敌人。**

**认知2：两个参考工程的互补价值远大于单独价值**

- p9-services（参考A）：教会我余额操作怎么做才安全——5种原子SQL、冻结三阶段、coinFrom/coinTo快照
- tg-services（参考B）：教会我区块链交互怎么做才完整——TRC20完整链路、多地址安全、双触发确认

单看任何一个都是残缺的。两个叠加才是我们 ser-wallet 的完整设计蓝图。这印证了"完整闭环"的准则——不能孤立分析。

**认知3：需求阶段的产出≠编码阶段的产出**

这22篇文档的价值不在于"写了多少代码"（事实上一行代码都没写），而在于：
- 证明我真正理解了需求
- 梳理出了评审需要澄清的28个问题
- 识别出了25项编码前缺失
- 设计出了可落地的架构和数据模型
- 从两个参考工程中提取了可借鉴的模式

**认知4：规则不是束缚，是保护**

你在最开始花大量时间定义行为准则（元认知三文件、严格约束、9维度控制体系），不是形式主义，是在建立一套**防错机制**。每次我差点想当然的时候，回头看那些准则就知道该怎么做。规则的价值在"差点犯错但没犯"的时刻。

**认知5：渐进式对话是质量保证机制**

沟通不是一锤子买卖。每一轮对话都可能带来新信息、修正旧理解。需求文档本身也在变化。固执己见的代价不是"面子上不好看"，而是"在错误方向上越走越远"。

## 五、关键技术认知沉淀——我真正学到了什么

### 5.1 钱包系统的核心设计原则

经过22篇文档的深度分析，钱包系统的设计核心浓缩为一句话和六个字：

**一句话：钱不能多、不能少、不能乱。**

**六个字**：

| 字 | 含义 | 落地手段 |
|----|------|---------|
| **准** | 金额精确 | decimal(20,8)、SQL级非负约束、不用float |
| **全** | 流水完整 | 每一分钱的来龙去脉都有记录（transaction_record） |
| **稳** | 并发安全 | SELECT FOR UPDATE + 事务内原子操作 |
| **幂** | 不重复处理 | 唯一索引 + 幂等号 + 状态机检查 |
| **冻** | 状态可控 | 提现三阶段（冻结→扣减→回滚），不可逆状态机 |
| **对** | 账实一致 | 余额 = SUM(流水)，可对账可审计 |

### 5.2 ser-wallet 的确定性骨架（绕不过的设计）

无论需求细节怎么调整，以下是**绕不过的必须设计项**：

1. **多币种多钱包余额表**（user_wallet）—— 用户×币种×钱包类型 = 唯一余额
2. **交易流水表**（transaction_record）—— 每一笔余额变动都有记录
3. **币种配置表**（currency_config）—— 支持的币种和精度规则
4. **汇率记录表**（exchange_rate / rate_change_log）—— 汇率数据和变更历史
5. **充值/提现订单表**—— 状态机驱动的订单生命周期
6. **USDT链上地址表**（user_chain_address）—— 用户与链上地址的绑定关系
7. **余额操作原子性**—— SQL级保证（UPDATE SET balance = balance + ? WHERE balance >= ?）
8. **幂等机制**—— 充值回调/提现回调的重复处理防护
9. **与财务模块的RPC边界**—— 钱包提供 CreditDeposit/FreezeForWithdraw 等RPC供财务调用

### 5.3 两个参考工程给我的核心启示

**从 p9-services（参考A）学到的——平台余额引擎怎么做**：

| 项 | 设计 | 我们怎么用 |
|----|------|-----------|
| 5种原子SQL | addBalance/subBalance/addFreezeBalance/subFreezeBalance/rollbackFreezeBalance | 直接采纳，翻译为GORM scope |
| 冻结三阶段 | freeze → confirm(sub_freeze + sub_balance) → rollback(sub_freeze + add_balance) | 直接采纳，提现链路的核心 |
| coinFrom/coinTo | 每笔流水记录"钱从哪来到哪去"的快照 | 直接采纳，审计追溯的关键 |
| SQL级非负 | WHERE balance >= 0 | 直接采纳，最后一道防线 |

**从 tg-services（参考B）学到的——区块链钱包怎么做**：

| 项 | 设计 | 我们怎么用 |
|----|------|-----------|
| 两层架构 | wallet-server(链上) + user-server(业务) | 借鉴思路，在ser-wallet内部用包隔离实现 |
| 双触发确认 | 用户手动确认 + 定时任务轮询（每2分钟） | 直接采纳，保证充值不遗漏 |
| 多地址安全 | 用户地址/归集地址/下发地址/Gas地址分离 | 直接采纳，职责分离 |
| 私钥加密存储 | AES加密(passphrase+address做key) | 采纳加密思路，具体算法待安全评审 |
| 两阶段链上确认 | 先发起交易(得txId) → 查链上状态(contractRet=="SUCCESS") | 直接采纳 |
| Gas费自动管理 | TRC20转账需TRX做Gas，自动从Gas地址打入 | 直接采纳 |
