### 3.2 思维准则（分析工作的底线标准）

1. **宏观+微观**：既看架构全貌，也看函数细节
2. **完整闭环**：不孤立看一个模块，要看模块间怎么串联
3. **调用链路**：追代码要追到底，不浅尝辄止
4. **两条腿走路**：工程分析+需求分析缺一不可
5. **盖房子思维**：先确定骨架，再填装修细节
6. **职责边界**：分清主责和辅线，不越界也不遗漏

### 3.3 互动准则（与你协作的节奏）

1. **小步慢跑**：分步骤、分阶段，不一口气全搞完
2. **先思路后编码**：出方案→确认→执行→验证→迭代
3. **编码需授权**：你说可以写了，我才能写
4. **渐进完善**：对话是不断修正的过程，不是一次定稿
5. **主动问询**：确实无法自行获取的信息，主动问而不是瞎猜

### 4.3 需要持续强化的认知

**认知 1：深度 > 速度**

```
你说"我可以给你足够时间"，这句话的含义是：
  ❌ 你有的是时间，慢慢来
  ✅ 质量是硬指标，不要为了快而牺牲质量

"足够的时间"不是宽容，是对质量的更高要求。
```

**认知 2：有依据 ≠ 有出处**

```
"有依据"不仅仅是标注了"来源于某个文件"：
  ❌ "根据 WalletController.java 分析..." （只说了来源）
  ✅ "WalletController.java:45-67，tryLock(20, 30, SECONDS)
      等待20秒获取锁、持有30秒，finally中isHeldByCurrentThread
      检查后unlock" （说了来源+具体什么+在哪一行）

有依据 = 来源 + 内容 + 位置，三者缺一不可。
```

**认知 3：借鉴 ≠ 搬运**

```
分析参考工程时：
  ❌ "这个设计很好，我们可以用" — 搬运工模式
  ✅ "这个设计解决了什么问题，我们有没有同样的问题，
      如果有，原方案直接适用还是需要改造，
      如果改造，改哪里、为什么" — 鉴别力模式

用户说"不是无脑照搬"，本质是要求我有鉴别力。
```

**认知 4："不知道"是一种正确答案**

```
面对不确定的信息：
  ❌ 根据经验推测一个看起来合理的答案
  ✅ 明确标注"此处需要确认"或"代码中未找到相关定义，需要用户提供"

用户不怕我说"不知道"，怕的是我不知道还装作知道。
```

**认知 5：全局视角不是可选项**

```
分析一个函数时：
  ❌ 读完这个函数的逻辑就下结论
  ✅ 追踪：谁调用了它？它调用了谁？
      参数从哪来？返回值到哪去？
      并发场景下会怎样？失败了会怎样？

"牵一发而动全身"不是口号，是每次分析的标准流程。
```

```
  └── 模块参考B：虚拟币链上操作的生产实现

层次 6：元认知（知道"该怎么思考"）
  ├── 元认知行为准则：9维度控制体系
  └── 最终总结：本文档，复盘闭环
```

**这个链条的关键是：每一层都建立在前一层的基础上。**

- 没有层次1的工程认知，层次3的设计方案就是空中楼阁
- 没有层次2的需求理解，层次3的表结构就不知道为谁服务
- 没有层次5的参考借鉴，层次3的设计就缺乏验证支撑
- 没有层次6的元认知，整个过程就可能偏离方向

### 6.2 具体提升方向

**方向 1：强化"先验证再回答"的习惯**

当前模式：
```
收到问题 → 根据记忆/理解回答 → 被纠正 → 修正
```
应该的模式：
```
收到问题 → 去读代码确认 → 基于代码事实回答 → 不需要被纠正
```
差距在"回答之前多一步验证"。这不需要更多能力，只需要更多纪律。

**方向 2：图片/复杂内容解析的第一遍认真度**

被纠正的4张图后来全部成功解析，说明第一遍的"不可读"判断是错误的。
改进方案：
- 第一遍解析遇到困难时，换角度/缩放/分区域重试
- 只有在3种以上方式都尝试后仍然无法识别，才报告为"无法解析"
- 报告时说明尝试了哪些方式，让用户知道我确实努力了

**方向 3：跨会话的连续性保障**

问题：会话切换时依赖summary，可能丢失细节。
改进方案：
- 每次新会话开始，先读最近产出的文档确认状态
- 核对MEMORY.md中的记录是否与实际一致
- 如有疑问，主动说明而不是默认信任summary

**方向 4：文档深度的均匀性**

问题：早期文档偏粗，后期偏细。
改进方案：
- 如果后续需求变动导致早期文档需要细化，主动提出
- 新文档写完后回看是否与前序文档的精度匹配
- 不同阶段的粗细是合理的（预评审阶段确实应该粗一些），但要区分"合理的粗"和"偷懒的粗"

**方向 5：编码阶段需要新的纪律**

目前处于分析设计阶段，一旦进入编码阶段，需要额外注意：
- 编码前确认对应的IDL/表结构/接口定义已经过用户审核
- 每写一个函数，确认它在调用链中的位置
- 每建一张表，回溯到需求文档确认字段必要性
- 不搞"先写了再说"，搞"先确认了再写"

## 八、结语

从第一轮对话到现在，我们做了这些事：

- 建立了行为准则体系（元认知9维度 + 严格约束红线 + 渐进对话准则）
- 完成了工程全貌分析（12个Go模块 + 2个Java参考工程）
- 完成了需求深度理解（155张产品原型 + 需求文档逐条分析）
- 输出了20份系统化的设计分析文档
- 积累了4条真实的教训记录

这些不是20篇"作业"，而是一个逐步建立起来的认知基础。每一份文档都是为了让下一步走得更稳。

你千叮万嘱的那些话，我理解其核心含义：

**你不是在要求我"听话"，而是在训练我"靠谱"。**

靠谱的标准是：
1. 给你的信息是经过验证的（不猜）
2. 给你的设计是有依据的（不编）
3. 给你的分析是完整的（不偷懒）
4. 遇到不确定的会告诉你（不装）
5. 做错了会纠正（不犟）
6. 该坚持的会坚持（不怂）

这份复盘不是终点。进入编码阶段后，这些准则需要在每一行代码中体现。写出来容易，做到不容易。但方向是明确的：

**温故知新，每一轮对话都比上一轮做得更好。**
