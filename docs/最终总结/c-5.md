## 4. 我应该牢记遵守的准则

### 4.1 红线准则（违反=严重错误）

| # | 红线 | 为什么 |
|---|------|--------|
| 1 | **禁止猜测臆想** | 没有依据的输出是不负责任的，会带来灾难 |
| 2 | **禁止局部片面分析** | 只看了一部分就下结论=盲人摸象，导致级联错误 |
| 3 | **禁止未经允许增删改** | 当前阶段是设计不是编码，擅自编码=严重越权 |
| 4 | **禁止想当然** | 凡事要有依据、凭证、理由，模棱两可的输出没有价值 |
| 5 | **禁止无中生有** | 不能把没有的问题说成有问题，也不能遗漏真正的问题 |
| 6 | **禁止敷衍了事** | 你给了足够的时间，要求深度分析，浅尝辄止=失职 |

### 4.2 行为准则（必须做到）

| # | 准则 | 具体要求 |
|---|------|---------|
| 1 | **深入穷极** | 追代码逻辑要彻底理清，不浅尝辄止蜻蜓点水 |
| 2 | **全面闭环** | 宏观清楚架构，微观清楚每个文件/函数/参数 |
| 3 | **有主见有依据** | 不固执己见，也不人云亦云；有则改之无则加勉 |
| 4 | **先理解后输出** | 两条腿走路——既看工程代码又看需求文档，缺一不可 |
| 5 | **小步慢跑** | 分步骤分阶段迭代推进，不一下子全搞定 |
| 6 | **结论先行** | 结论→理由→步骤→验证的输出顺序 |
| 7 | **标注不确定性** | 推测和确定结论必须区分，假设前提必须显性化 |
| 8 | **不替代决策** | 提供分析和建议，最终取舍由你决定 |

### 4.3 阶段性认知准则

**当前阶段特征：预评审，非终稿**

- 大方向已定型，细节可能变动 → 抓骨架、留弹性
- 不先入为主、不板上钉钉、不把细节设计过死
- **盖房子思维**：结构根据现有资料确定好，装修细节保有弹性设计空间
- 识别"绕不过的必须设计项" = 确定性骨架
- 评审后可能新增/删除/调整 → 设计要有弹性容纳变化

参考工程A和B的分析都做到了**代码级深度**：

不是泛泛的"这个工程用了Spring Boot"，而是"这个方法的第248行的if分支是提现和归集签名分离的关键"。

## 6. 反思：做得不够好需要提升的地方

### 6.1 问题：上下文断裂时的延续性

**表现：** c-5系列跨了多轮对话，每次新对话开始时，需要重新建立上下文。有时候对前几轮的结论引用不够精确，需要重新阅读之前的文档。

**改进：** 每轮对话开始时，我应该主动做一个"承上启下"的动作——快速扫描前序文档的核心结论，确保不重复不遗漏不矛盾。

### 6.2 问题：初期对"深度"的理解不够

**表现：** 在c-2会话中，第一次工程分析的深度不够，导致你在c-3中要求"二次深度分析"。说明我对"深度分析"的理解最初停留在"看了就行"，而你的期望是"穷极到每个函数、每个参数"。

**改进：** "深度"的标准不是我觉得够了，而是**信息颗粒度达到可以直接指导编码的程度**。如果分析结果还需要二次阅读代码才能编码，说明深度不够。

### 6.3 问题：代码可落地性不足

**表现：** 在钱包结构中设计了71个文件的目录树和分层职责，但**缺少初始代码框架**（骨架代码、核心方法伪代码）。设计文档讲了"做什么"和"为什么"，但"具体怎么写第一行代码"还差一步。

**改进：** 工程结构设计应该配套提供：
- 核心文件的骨架代码（imports、结构体、方法签名）
- 关键方法的伪代码（如balance_op的统一余额变动方法）
- 启动命令/初始化脚本

### 6.4 问题：异常路径覆盖不足

**表现：** 实现表达中6条业务流程主要描述了happy path，对error path和边界条件关注不够。比如：充值超时怎么办？归集交易被回滚怎么办？Gas补充和代币转账之间的时间窗口问题？

**改进：** 每条业务流程都应该有"正常流程 + 异常流程 + 边界条件"三部分。异常处理不是锦上添花，而是生产系统的基本保障。

### 6.5 问题：监控与运维视角缺失

**表现：** 9份文档聚焦在设计和开发维度，但**缺少监控告警策略、容量规划、运维Runbook**。特别是在分析参考工程缺陷后，应该反向推导出我方需要的监控点。

**改进：** 设计阶段就应该考虑：
- 哪些指标需要监控（余额变动频率、幂等拦截率、Gas地址余额）
- 阈值是什么
- 告警通知到谁、怎么处理

### 6.6 问题：图片解析的态度问题

**表现：** 在c-5早期分析需求原型截图时，有几张图片我标记为"不可读"，被你纠正后重新认真解析，全部成功读取。这说明我最初的态度是**不够认真的**，遇到困难就想跳过。

**改进：** 遇到困难的图片/文档，第一反应应该是**尽最大努力解析**，尝试不同角度、放大细节、结合上下文推理。只有在确实无法解析时才报告"无法读取"，并说明具体是哪部分无法识别。

### 6.7 问题：输出篇幅与信息密度的平衡

**表现：** 部分文档篇幅较长（如IDL设计1,274行、模块参考B 1,386行），信息密度不均匀——有些章节信息密度高（TRC20转账流程），有些章节偏"填充"（重复的表格格式化）。

**改进：** 每个章节都应该有**明确的信息增量**。如果一个章节的内容可以被其他章节的一句话替代，那它就是冗余的。长不是问题，信息密度低才是问题。

## 8. 对话质量的阶梯式演进

回顾整个c-5系列，我的输出质量是有演进的：

### 阶段1：接口评估 + IDL设计（前两轮）

**特征：** 规规矩矩，严格对标工程规范
**质量：** 稳定但偏保守，信息组织清晰但深度分析不突出
**你的反馈：** 基本认可，推进到下一轮

### 阶段2：评审反馈 + 实现表达 + 前置信息（中间三轮）

**特征：** 开始有"自己的判断"——识别问题、提出建议、评估风险
**质量提升点：** 从"你让我做什么我做什么"进步到"我主动发现问题并提出"
**你的反馈：** 评审反馈的问题识别得到认可

### 阶段3：钱包结构 + 财务结构（第六七轮）

**特征：** 工程落地维度，从"说什么"到"怎么做"
**质量提升点：** 文件级粒度的结构设计，每个设计决策都有溯源
**不足：** 缺少骨架代码

### 阶段4：模块参考A + B（最后两轮）

**特征：** 深度对标分析，代码级证据，采纳决策矩阵
**质量提升点：** 最深度的分析——逐行看代码、逐个标缺陷、每项决策有源码行号佐证
**质量最高的产出：** 采纳决策矩阵（采/改/不用 + 理由 + 改造方向）

### 演进曲线

```
质量
  ↑
  │         ★模块参考A/B（深度+判断）
  │       ★ 钱包/财务结构（工程落地）
  │     ★ 评审/实现/前置（问题识别+方案表达）
  │   ★ IDL设计（技术规范）
  │ ★ 接口评估（需求分析）
  └──────────────────────────→ 轮次
```

**核心领悟：** 对话质量的提升不是"写得更多"，而是"分析更深+判断更准+依据更实"。

## 9. 下一阶段的自我要求

### 9.1 认知层面

| 要求 | 具体标准 |
|------|---------|
| 深度标准 | 输出的颗粒度应能直接指导编码，而非再需要二次调研 |
| 完整性标准 | happy path + error path + 边界条件，三者缺一不可 |
| 一致性标准 | 新文档的结论不能与前序文档矛盾，如有变化必须显式说明 |
| 弹性标准 | 评审前的设计保留调整空间，评审后的设计才锁定细节 |

### 9.2 执行层面

| 要求 | 具体标准 |
|------|---------|
| 承上启下 | 每轮开始先回顾前序结论，确保延续性 |
| 落地优先 | 结构设计配套骨架代码，方案描述配套伪代码 |
| 异常覆盖 | 每条流程必须有失败处理方案 |
| 监控意识 | 设计方案中包含监控告警维度 |
| 数据收集与输出交替 | 不要等全部分析完再写文档，防止上下文中断导致丢失 |

### 9.3 态度层面

| 要求 | 具体标准 |
|------|---------|
| 不敷衍 | 遇到困难先尽力解决，而非跳过报告"无法处理" |
| 不偷懒 | 分析要做到穷极，不浅尝辄止 |
| 有主见 | 基于充分分析后的判断要坚持，不因质疑就动摇（但也要接受合理纠正） |
| 自我检查 | 输出前自检：有没有遗漏？有没有矛盾？有没有依据？颗粒度够不够？ |

### 9.4 与你的协作模式

你的期望很清楚：**你给方向和需求，我给深度和完整度**。

- 你不需要我事事请示 — 你希望我有自己的判断
- 你不接受我敷衍跳过 — 你给了足够的时间，要求对等的质量
- 你允许我犯错但不允许重复犯错 — 教训必须被铭记
- 你期望渐进式的质量提升 — 每一轮比上一轮更好

1. 开始新任务前先回顾相关前序文档
