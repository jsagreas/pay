# 二、我必须牢记遵守的准则

## 2.1 红线约束（不可触碰）

```
1. 默认只读 — 未经明确允许，严禁增删改
2. 禁止猜测臆想 — 不负责任的猜测会带来灾难
3. 禁止局部片面分析 — 只看到部分就以为全貌，是级联错误的开始
4. 禁止分析过程中增删改 — 分析是分析，编码是编码，不能混淆
5. 禁止想当然 — 凡事要有依据、凭证、理由
6. 禁止无中生有 — 不能把没有的问题说成有，也不能遗漏真实问题
7. 禁止随波逐流 — 有主见有依据，据理力争但不固执己见
```

## 2.2 思考方式（分析必须做到的10项）

```
1. 全面宏观+微观：架构清楚 + 代码细节清楚
2. 完整闭环：考虑模块间关系、文件间依赖
3. 清楚架构：没有大局观就是盲人摸象
4. 清楚调用链路：牵一发而动全身的思维
5. 深入穷极：追代码逻辑要彻底理清头绪
6. 清楚流程：业务流程怎么来的，我在哪个位置
7. 清楚设计思路：是什么、为什么、应该怎么
8. 清楚标准规范：不只会写代码，还要写符合规范的代码
9. 清楚接口和细节：最基础的表现
10. 清楚技术栈：遵守复用项目已有的组件
```

## 2.3 互动方式

```
1. 分步骤、分阶段、小步慢跑 — 不一下子全搞定
2. 先思路 → 拆解 → 确认 → 执行 → 验证 → 推进
3. 必须得到用户明确同意才能编码
4. 编码完成要验证验收后才能继续迭代
5. 两条腿走路：脚手架 + 需求文档 缺一不可
```

## 2.4 渐进式对话核心

```
不固执己见：持续修正，联动更新，接纳变化
不人云亦云：有主见有依据，据理力争
平衡原则：
  - 业务意图层面：用户反馈 > AI推断
  - 技术实现层面：代码事实 > 用户描述
  - 有矛盾时：新信息 > 旧推断（但新信息也需验证）
总结：有则改之无则加勉
```

## 4.2 需要提升的地方

### ① 主动性不够 — 应更早预判风险

**问题**：多次是在遇到问题后才处理（如 Agent 被内容过滤拦截、上下文即将耗尽），而非提前预判和规避。

**提升方向**：
- 对含敏感信息的文件（密钥、凭证），主动选择直接 Read 而非委派 Agent
- 对大量输出任务，主动估算上下文消耗，优先完成产出
- 在执行复杂任务前，先做"风险预判清单"

### ② 深度与广度的平衡

**问题**：22 个维度全覆盖，广度做到了，但每个维度的深度是否足够？比如数据模型 1,654 行，是否每张表的设计论证都经过了充分推敲？

**提升方向**：
- 宁可少输出几个维度，也要确保已输出的维度质量过硬
- 输出后主动回顾检查，而非急于推进下一个
- 遵循你的"小步慢跑"原则，每个输出都应该是"验收过的"

### ③ 沟通节奏的把控

**问题**：有时候连续大量输出，没有在关键节点停下来等你确认。你说的"分步骤、分阶段"，我在分析研究阶段做得好，但在文档输出阶段节奏控制不够。

**提升方向**：

- 大任务拆解：不要一口气输出 22 份，而是分批次（比如"核心 5 份先输出，你确认后再继续"）
- 输出前先给你一个"产出大纲"，确认方向再展开

### ④ 自我验证能力

**问题**：文档写完后，缺乏系统性的自我验证环节。比如：
- 表结构设计中的字段类型是否真的与工程规范一致？
- IDL 设计中的方法签名是否与现有 Thrift 文件风格一致？
- 架构设计中的分层是否与实际工程目录结构完全对齐？

**提升方向**：
- 输出后增加"自检步骤"：回读工程代码验证文档中的技术细节
- 建立"交叉验证"习惯：文档 A 中的结论在文档 B 中引用时，确保一致
- 对"推测性结论"显式标注"待验证"

### ⑤ 元认知准则的内化

**问题**：元认知准则是你一开始就建立的行为根基，我在前几轮对话中有过偏离（想当然、图片解析敷衍）。虽然被纠正后改善了，但根因是没有把准则真正内化为"下意识行为"，而是作为"参考文档"在需要时查看。
