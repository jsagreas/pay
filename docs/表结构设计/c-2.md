# 二、分级标准定义

```
必须（MUST）:
  · 缺了这张表，核心业务流程无法完成
  · 涉及资金安全、数据一致性的刚性需求
  · 用户基本功能不可用（充值/提现/兑换/查余额）
  · 举例: user_wallet — 没有余额表，整个钱包就不存在

应该（SHOULD）:
  · 缺了不影响核心流程，但会导致:
    - 运营管理不便（无法审计、无法追溯）
    - 用户体验下降（无法保存偏好、无法查看稽核进度）
    - 安全/合规有隐患（缺少操作日志）
  · 建议首版就做，但优先级低于"必须"
  · 举例: rate_change_log — 没有日志不影响汇率更新，但运营无法追溯变更历史

可选（OPTIONAL）:
  · 属于优化或增强功能
  · 可以用其他方式替代（如实时计算替代预聚合表）
  · 可以在后续迭代中补充
  · 举例: channel_stat_daily — 统计数据可以从订单表实时聚合，预聚合表只是性能优化
```

# 三、钱包模块表结构（自己负责 — 8张表）

## 3.1 【必须】user_wallet — 用户钱包账户表

### 为什么必须

```
业务依据:
  · 产品原型"个人中心"页: 展示当前币种下各钱包余额
  · 所有余额操作（充值/提现/兑换/消费）的最终落点
  · 没有这张表，用户看不到余额，所有交易无法进行

设计依据:
  · 每个用户 × 每个币种 × 每种钱包类型 = 一行记录
  · 钱包类型: 中心(1)、奖励(2)、主播(3)、代理(4预留)、场馆(5预留)
  · 余额用 BIGINT（最小货币单位整数）— 业界标准，杜绝浮点误差
```

### 完整字段定义

```sql
CREATE TABLE user_wallet (
    id              BIGINT       NOT NULL PRIMARY KEY                COMMENT '主键（雪花ID）',
    user_id         BIGINT       NOT NULL                            COMMENT '用户ID',
    currency_id     BIGINT       NOT NULL                            COMMENT '币种ID（关联 currency_config.id）',
    wallet_type     TINYINT      NOT NULL                            COMMENT '钱包类型: 1=中心 2=奖励 3=主播 4=代理(预留) 5=场馆(预留)',
    balance         BIGINT       NOT NULL DEFAULT 0                  COMMENT '余额（最小货币单位整数）',
    frozen_amount   BIGINT       NOT NULL DEFAULT 0                  COMMENT '冻结金额（提现时冻结）',
    version         BIGINT       NOT NULL DEFAULT 0                  COMMENT '乐观锁版本号（预留，当前主用行锁）',
    status          TINYINT      NOT NULL DEFAULT 1                  COMMENT '状态: 1=正常 2=冻结 3=禁用',
    create_by       BIGINT       NOT NULL DEFAULT 0                  COMMENT '创建人ID',
    create_at       BIGINT       NOT NULL DEFAULT 0                  COMMENT '创建时间（毫秒时间戳）',
    update_by       BIGINT       NOT NULL DEFAULT 0                  COMMENT '更新人ID',
    update_at       BIGINT       NOT NULL DEFAULT 0                  COMMENT '更新时间（毫秒时间戳）',
    delete_at       BIGINT       NOT NULL DEFAULT 0                  COMMENT '软删除（0=未删除）',

    UNIQUE KEY uk_user_currency_type (user_id, currency_id, wallet_type, delete_at),
    INDEX idx_user_id (user_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户钱包账户表';
```

### 字段逐一分析

| 字段 | 类型 | 为什么需要 | 业务来源 |
|------|------|-----------|---------|
| id | BIGINT | 主键，雪花ID | 项目统一规范，TiDB 适配 |
| user_id | BIGINT | 标识钱包所属用户 | 所有页面: 按用户查余额 |
| currency_id | BIGINT | 标识钱包所属币种 | 产品原型: 切换币种=切换整套钱包 |
| wallet_type | TINYINT | 区分钱包性质 | 需求: 中心钱包+奖励钱包+主播钱包 |
| balance | BIGINT | 存储余额 | 核心中的核心，所有交易的结果 |
| frozen_amount | BIGINT | 提现冻结池 | 需求: 提现时冻结金额，审核通过后扣减 |
| version | BIGINT | 乐观锁预留 | 架构设计: 当前主用行锁，预留升级能力 |
| status | TINYINT | 账户状态管理 | 风控: 管理员可冻结/禁用问题账户 |

### 关键索引说明

```
uk_user_currency_type (user_id, currency_id, wallet_type, delete_at):
  WHY: 保证一个用户在一个币种下只有一种类型的钱包
  HOW: 同时覆盖最高频查询 WHERE user_id=? AND currency_id=? AND wallet_type=?
  delete_at 加入原因: 软删除后允许重建（0时唯一约束生效，非0时不冲突）
```

## 3.2 【必须】wallet_flow — 钱包流水记录表

### 为什么必须

```
业务依据:
  · 产品原型"交易记录→奖励记录"页: 需要展示余额变动明细
  · 资金安全: 每笔余额变动必须有迹可查（审计需求）
  · 对账: balance_before + amount = balance_after 可反算验证
  · 幂等: 唯一索引防止重复操作（如财务模块重复调用Credit）

如果没有这张表:
  · 无法审计（谁在什么时候操作了多少金额）
  · 无法对账（余额对不上时无法追溯）
  · 无法幂等（重复回调会导致多次加款）
  · 这三个能力缺任何一个，钱包系统都不可上线
```

### 完整字段定义

```sql
CREATE TABLE wallet_flow (
    id              BIGINT       NOT NULL PRIMARY KEY                COMMENT '主键（雪花ID）',
    user_id         BIGINT       NOT NULL                            COMMENT '用户ID',
    currency_id     BIGINT       NOT NULL                            COMMENT '币种ID',
    wallet_type     TINYINT      NOT NULL                            COMMENT '钱包类型',
    flow_type       TINYINT      NOT NULL                            COMMENT '流水类型: 1=充值 2=提现 3=兑换出 4=兑换入 5=奖励 6=冻结 7=解冻 8=人工加 9=人工减 10=消费 11=稽核转入 12=补单',
    amount          BIGINT       NOT NULL                            COMMENT '变动金额（正=增 负=减）',
    balance_before  BIGINT       NOT NULL DEFAULT 0                  COMMENT '变动前余额',
    balance_after   BIGINT       NOT NULL DEFAULT 0                  COMMENT '变动后余额',
    frozen_before   BIGINT       NOT NULL DEFAULT 0                  COMMENT '变动前冻结金额',
    frozen_after    BIGINT       NOT NULL DEFAULT 0                  COMMENT '变动后冻结金额',
    order_no        VARCHAR(32)  NOT NULL                            COMMENT '关联业务订单号',
    biz_type        VARCHAR(32)  NOT NULL DEFAULT ''                 COMMENT '业务类型标识（扩展预留）',
    remark          VARCHAR(256) NOT NULL DEFAULT ''                 COMMENT '备注',
    create_at       BIGINT       NOT NULL DEFAULT 0                  COMMENT '创建时间（毫秒时间戳）',

    UNIQUE KEY uk_order_flow (order_no, wallet_type, flow_type),
    INDEX idx_user_currency_time (user_id, currency_id, create_at),
    INDEX idx_user_flow_type (user_id, flow_type, create_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='钱包流水记录表';
```

### 字段逐一分析

| 字段 | 类型 | 为什么需要 | 业务来源 |
|------|------|-----------|---------|
| flow_type | TINYINT | 区分12种余额变动类型 | 需求: 充值/提现/兑换/奖励/冻结/解冻/人工加减/消费/稽核/补单 |
| amount | BIGINT | 正负表示增减方向 | 充值+100, 提现-100, 一目了然 |
| balance_before/after | BIGINT | 变动前后余额快照 | 审计: 可验证 before+amount=after |
| frozen_before/after | BIGINT | 冻结金额变动快照 | 仅冻结/解冻时使用，其他类型为0 |
| order_no | VARCHAR(32) | 关联到业务订单 | 充值单(C)、提现单(T)、兑换单(E)等 |

### 为什么是 append-only（只写不改不删）

```
· 没有 update_by / update_at / delete_at 字段
· 流水一旦写入就是永久证据
· 如果需要"冲正": 写一条反向流水（而不是修改原记录）
· 这是金融系统的行业惯例（银行流水也是只增不改）
```

### 幂等索引设计分析

```
uk_order_flow (order_no, wallet_type, flow_type):

为什么不是简单的 UNIQUE(order_no)?
  → 一笔兑换订单(E+16位)会产生3条流水:
    (E123, type=1中心, flow=3兑换出)  扣法币
    (E123, type=1中心, flow=4兑换入)  加BSB
    (E123, type=2奖励, flow=5奖励)    加BSB奖金
  → 需要3个维度才能唯一标识

幂等机制:
  → ledger 层先 INSERT 流水（利用唯一索引）
  → 唯一冲突 = 已处理过 → 直接返回成功
  → 不冲突 = 首次执行 → 继续后续余额操作
  → 这比"先查后写"更安全（避免并发窗口期）
```

## 3.3 【必须】deposit_order — 充值订单表

### 为什么必须

```
业务依据:
  · 产品原型"充值中心": 用户提交充值后需要跟踪订单状态

```

```
idx_duplicate_check (user_id, currency_id, amount, status):

  逻辑:
```


### receiver_info 加密存储结构

```
该字段是 AES 加密后的 JSON，按 method_type 区分结构:

bank（银行转账）:
  {"bank_name":"Vietcombank","bank_code":"VCB","card_number":"123...","account_name":"NGUYEN VAN A"}

e_wallet（电子钱包）:
  {"wallet_type":"GoPay","account_number":"08123456789"}

usdt（USDT提现）:
  {"network":"TRC20","address":"TXyz..."}

加密参考: common/pkg/db/cmd/generate.go 中 user_kyc 表的 encryptedFields 模式
脱敏展示: receiver_summary 字段存储脱敏后的摘要（如 "****1234"），列表页直接用此字段
```

### fund_source_type 字段来源

```
来自产品需求"提现资金来源规则":
  · 纯法币充值用户 → 只能法币提现 (type=1)
  · 纯USDT充值用户 → 只能USDT提现 (type=2)
  · 混合充值用户 → 只能法币提现 (type=3)

创建提现订单时，通过分析该用户的充值历史（查 deposit_order）自动计算:
  SELECT DISTINCT currency_type FROM deposit_order d
  JOIN currency_config c ON d.currency_id = c.id
  WHERE d.user_id = ? AND d.status = 1

记录在订单上用于审计追溯（"为什么这个用户只能法币提现"）
```

## 3.5 【必须】exchange_order — 兑换订单表

### 为什么必须

```
业务依据:

  · 兑换涉及跨币种操作: 一笔订单关联3条流水（扣法币+加BSB+加奖金）
```

### 完整字段定义

```sql
CREATE TABLE exchange_order (

```

### 为什么没有 update 和 delete 字段

```
兑换是同步原子操作:
  · 在一个数据库事务内完成: 扣法币 + 加BSB + 加奖金 + 创建订单
  · 要么全部成功（状态=1），要么全部失败（事务回滚，不写入）
  · 不存在"待处理"→"已完成"的状态变更
  · 因此不需要 update_by/update_at（创建即终态）
  · 不需要 delete_at（兑换记录不可删除）
```

### 精度位数（decimals）与展示转换

```
币种     decimals    数据库存储            展示给用户
────    ────────    ──────────           ──────────

IDR     0           100000               100,000 IDR
THB     2           10050                100.50 THB
USDT    8           100000000            1.00000000 USDT
BSB     8           1000000000           10.00000000 BSB

转换公式: 展示值 = 数据库值 / 10^decimals
```


## 4.2 【必须】currency_rate — 汇率数据表

### 为什么必须

```
业务依据:
  · 四层汇率体系是币种配置的核心能力
  · C端兑换页: 需要读取平台汇率计算预估BSB
  · 充值时: 需要充值汇率计算实际到账金额
  · 提现时: 需要提现汇率计算实际出款金额
  · 定时任务: 需要存储/更新实时汇率

如果没有这张表:
  · 无法获取汇率 → 兑换/充值/提现 的金额计算全部无法进行
```

### 完整字段定义

```sql
CREATE TABLE currency_rate (
    id              BIGINT       NOT NULL PRIMARY KEY                COMMENT '主键（雪花ID）',
    currency_id     BIGINT       NOT NULL                            COMMENT '币种ID（关联 currency_config.id）',
    realtime_rate   VARCHAR(32)  NOT NULL DEFAULT '0'                COMMENT '实时汇率（3API均值, 币种→USDT）',
    platform_rate   VARCHAR(32)  NOT NULL DEFAULT '0'                COMMENT '平台汇率（超阈值时从实时更新）',
    deposit_rate    VARCHAR(32)  NOT NULL DEFAULT '0'                COMMENT '充值汇率 = 实时×(1+浮动%)',
    withdraw_rate   VARCHAR(32)  NOT NULL DEFAULT '0'                COMMENT '提现汇率 = 实时×(1-浮动%)',
    rate_source_1   VARCHAR(32)  NOT NULL DEFAULT '0'                COMMENT 'API源1汇率',
    rate_source_2   VARCHAR(32)  NOT NULL DEFAULT '0'                COMMENT 'API源2汇率',
    rate_source_3   VARCHAR(32)  NOT NULL DEFAULT '0'                COMMENT 'API源3汇率',
    last_sync_at    BIGINT       NOT NULL DEFAULT 0                  COMMENT '最近同步时间',
    update_at       BIGINT       NOT NULL DEFAULT 0                  COMMENT '更新时间（毫秒时间戳）',

    UNIQUE KEY uk_currency (currency_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='汇率数据表';
```

### 为什么汇率用 VARCHAR 而不是 DECIMAL

```
· 汇率精度要求极高（如 1 VND = 0.0000000395 USDT）
· 不同币种精度差异巨大，DECIMAL 精度需预先指定
· VARCHAR 存原始精度字符串，Go 内部用 shopspring/decimal 计算
· 这是避免浮点误差的最安全方案
```

# 八、分级决策矩阵

> 以下矩阵解释每张表为什么被分到当前级别

## 8.1 必须级（14张）— 缺了业务跑不通

| # | 表 | 缺失后果 |
|---|------|---------|
| 1 | user_wallet | 没有余额，整个钱包不存在 |
| 2 | wallet_flow | 无审计、无幂等、无对账能力 |
| 3 | deposit_order | 充值流程无法跟踪，记录页空白 |
| 4 | withdraw_order | 提现流程无法跟踪，冻结金额无法管理 |
| 5 | exchange_order | 兑换操作无记录，记录页空白 |
| 6 | currency_config | 无币种定义，所有引用 currency_id 的表无法工作 |
| 7 | currency_rate | 无汇率数据，兑换/充值/提现金额计算无法进行 |
| 8 | collect_channel | 没有代收通道，充值无法匹配支付渠道 |
| 9 | payout_channel | 没有代付通道，提现无法出款 |
| 10 | polling_rule | 无轮询规则，通道匹配无法执行 |
| 11 | polling_rule_channel | 无权重配置，轮询无法按比例分配 |
| 12 | deposit_method | C端看不到充值方式，无法充值 |
| 13 | withdraw_method | C端看不到提现方式，无法提现 |
| 14 | finance_channel_order | 财务侧无法跟踪通道订单和回调 |

## 8.2 应该级（7张）— 缺了体验或安全有缺陷

| # | 表 | 缺失后果 | 替代方案（代价） |
|---|------|---------|---------------|
| 15 | user_wallet_setting | Redis重启后用户丢失币种偏好 | 只用Redis（重启后需重新选择） |

# 九、建表执行顺序

```
按依赖关系，分4个批次建表:

批次1 — 地基层（无外键依赖）:
  ① currency_config           ← 所有表的根依赖
  ② user_wallet_setting       ← 独立表

批次2 — 配置层（依赖 currency_config）:
  ③ currency_rate             ← 引用 currency_config.id
  ④ rate_change_log           ← 引用 currency_config.id
  ⑤ collect_channel           ← 引用 currency_config.id
  ⑥ payout_channel            ← 引用 currency_config.id
  ⑦ deposit_method            ← 引用 currency_config.id + collect_channel.id
  ⑧ withdraw_method           ← 引用 currency_config.id + payout_channel.id
  ⑨ polling_rule              ← 引用 currency_config.id
  ⑩ polling_rule_channel      ← 引用 polling_rule.id + channel.id
  ⑪ deposit_bonus_config      ← 引用 currency_config.id（可选）

批次3 — 业务层（依赖配置层）:
  ⑫ user_wallet               ← 引用 currency_config.id
  ⑬ deposit_order             ← 引用 currency_config.id
  ⑭ withdraw_order            ← 引用 currency_config.id
  ⑮ exchange_order            ← 引用 currency_config.id
  ⑯ finance_channel_order     ← 引用 channel.id
  ⑰ supplement_order          ← 引用 deposit_order.order_no
  ⑱ manual_adjust_order       ← 引用 user + currency

批次4 — 日志/统计层（依赖业务层）:
  ⑲ wallet_flow               ← 逻辑关联 user_wallet + 订单表
  ⑳ reward_audit              ← 逻辑关联 wallet_flow
  ㉑ withdraw_audit_log        ← 引用 withdraw_order.order_no
  ㉒ channel_stat_daily        ← 聚合 finance_channel_order
  ㉓ transaction_stat_daily    ← 聚合 deposit_order + withdraw_order
  ㉔ user_payment_method       ← 独立表
  ㉕ rate_source_config        ← 独立表

注: 这些都是逻辑引用关系，不建物理外键（微服务标准做法，应用层保证一致性）
```

# 十、总结

```
三大模块完整表结构设计:

  钱包模块（自己负责）:     5必须 + 2应该 + 1可选 = 8张表
  币种配置模块（自己负责）:  2必须 + 1应该 + 1可选 = 4张表
  财务管理模块（他人负责）:  7必须 + 3应该 + 2可选 = 12张表
  跨模块（归属待确认）:      0必须 + 1应该 + 0可选 = 1张表
  ─────────────────────────────────────────────────
  合计:                     14必须 + 7应该 + 4可选 = 25张表

自己需要建的表: 12张（8钱包 + 4币种配置）
需要理解的表:   13张（12财务 + 1跨模块）

核心设计原则:
  ① 金额 BIGINT（最小货币单位整数）— 全部25张表统一
  ② 雪花ID主键 — 全部25张表统一
  ③ 毫秒时间戳 — 全部时间字段统一
  ④ 软删除 BIGINT 0 — 需要软删的表统一
  ⑤ 两模块通过 order_no 关联，通过 RPC 交互 — 不直接读写对方的表

首版优先级: 先建14张必须表 → 再补7张应该表 → 可选表视需求迭代
```
