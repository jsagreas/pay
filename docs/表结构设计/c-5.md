## 1.2 为什么按"必须/应该/可选"分级

```
必须（Must）：
  没有这张表，对应的核心业务流程无法运转
  例：没有 user_balance，余额都存不了；没有 deposit_order，充值流程断裂

应该（Should）：
  缺了不影响核心流程运转，但影响功能完整性、合规性或用户体验
  例：没有 withdraw_account，用户每次提现都要重新填银行卡信息（可用但体验差）
  例：没有 currency_log，无法追溯谁改了币种配置（可运行但不合规）

可选（Optional）：
  当前 MVP 阶段不需要，后续按运营需求和规模增长决定是否引入
  例：没有 balance_snapshot，不影响任何功能，只是后续对账方便
  例：没有 reward_flow_rule，可以先在代码中硬编码流水倍数规则
```

## 1.3 设计规范（全部表统一遵循）

```
字段类型：
  主键          int64 (BIGINT AUTO_INCREMENT)
  金额          int64 (BIGINT，最小单位整数，不用 DECIMAL)
  时间戳        int64 (BIGINT，毫秒级时间戳，UserPlugin自动填充)
  状态/枚举     int32 (INT)
  短字符串      string / VARCHAR(N)
  可选字段      指针类型 (*int64, *string → NULL)
  必填字段      非指针类型 (int64, string → NOT NULL)

审计字段（每张表携带）：
  create_by     *int64    创建人ID（UserPlugin自动填充）
  create_at     int64     创建时间（UserPlugin自动填充）
  update_by     *int64    更新人ID（UserPlugin自动填充）
  update_at     int64     更新时间（UserPlugin自动填充）

软删除（仅配置表需要）：
  delete_at     int64     删除时间（0=未删除）
  delete_by     *int64    删除人ID

外键：不使用。微服务架构用应用层保证引用完整性。
```

# 二、币种配置模块 — 表结构详解

> 模块定位：基础数据层，为钱包和财务提供币种元数据和汇率数据
> 你负责，无外部依赖，应最先开发

---

## 2.1 【必须】currency_config — 币种配置表

### 为什么必须

需求文档明确了"三类币种"（法币/加密/平台币）和"B端币种管理页面"。**整个体系中所有涉及金额的操作，都需要从这张表获取币种的精度、限额、手续费、状态等参数**。没有这张表：
- C端不知道展示哪些币种
- 充值/提现无法校验限额
- 金额展示不知道保留几位小数
- 手续费无法计算

这是整个系统的**配置地基**。

### 业务来源

| 需求点 | 对应字段 |
|--------|---------|
| 产品原型：币种配置列表页 | 整张表 |
| 需求：B端配置币种基本信息（名称/代码/图标/精度） | currency_code, currency_name, icon, precision |
| 需求：充值配置（单笔最低/最高限额、是否启用） | deposit_min, deposit_max, deposit_enable |
| 需求：提现配置（限额/手续费/T+N到账天数） | withdraw_min, withdraw_max, withdraw_daily, fee_type, fee_value, arrive_days |
| 需求：状态控制（启用/禁用后C端不可见） | status |
| 需求：排序（影响C端展示顺序） | sort_order |
| 需求：三类币种（法币/加密/平台币） | currency_type |

### 完整字段

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | int64 | PK, AUTO_INCREMENT | 主键 |
| currency_code | string | NOT NULL, UK(+delete_at) | 币种代码（BSB/VND/IDR/THB/USDT） |
| currency_name | string | NOT NULL | 币种名称（显示用） |
| currency_type | int32 | NOT NULL | 币种类型 1=法币 2=加密货币 3=平台币 |
| icon | *string | | 币种图标URL |
| precision | int32 | NOT NULL, DEFAULT 2 | 小数精度（VND=0, BSB=2, USDT=2），决定 BIGINT↔展示的换算 |
| deposit_min | int64 | NOT NULL, DEFAULT 0 | 充值单笔最低限额（最小单位） |
| deposit_max | int64 | NOT NULL, DEFAULT 0 | 充值单笔最高限额（最小单位） |
| deposit_enable | int32 | NOT NULL, DEFAULT 1 | 充值启用 0=禁用 1=启用 |
| withdraw_min | int64 | NOT NULL, DEFAULT 0 | 提现单笔最低限额 |
| withdraw_max | int64 | NOT NULL, DEFAULT 0 | 提现单笔最高限额 |
| withdraw_daily | int64 | NOT NULL, DEFAULT 0 | 提现每日总额上限 |
| fee_type | int32 | NOT NULL, DEFAULT 0 | 手续费类型 0=无 1=固定 2=百分比（万分比） |
| fee_value | int64 | NOT NULL, DEFAULT 0 | 手续费值 |
| arrive_days | int32 | NOT NULL, DEFAULT 0 | T+N到账天数（0=即时） |
| withdraw_enable | int32 | NOT NULL, DEFAULT 1 | 提现启用 0=禁用 1=启用 |
| sort_order | int32 | NOT NULL, DEFAULT 0 | 排序权重（越大越前） |
| status | int32 | NOT NULL, DEFAULT 1 | 状态 0=禁用 1=启用 |
| create_by | *int64 | | 创建人ID |
| create_at | int64 | NOT NULL | 创建时间 |
| update_by | *int64 | | 更新人ID |
| update_at | int64 | NOT NULL | 更新时间 |
| delete_at | int64 | NOT NULL, DEFAULT 0 | 软删除（0=未删除） |
| delete_by | *int64 | | 删除人ID |

### 索引

```
UNIQUE KEY uk_currency_code (currency_code, delete_at)
  → 币种代码唯一，加 delete_at 支持删除后重建同代码
  → 币种数量极少（<10），无需额外索引
```

### 缓存策略

Redis Cache-Aside，TTL 5分钟。B端修改后主动删缓存，下次查询自动重建。

## 2.2 【必须】exchange_rate — 汇率配置表

### 为什么必须

需求明确了"汇率由平台手动配置，格式统一为1 BSB = X法币"。**兑换、充值展示、提现换算、财务BSB统计列，全部依赖汇率数据**。没有这张表，任何涉及跨币种计算的操作都无法进行。

### 为什么与 currency_config 分表

- 汇率变更频率 > 币种基本信息变更频率（运营可能每天调汇率）
- 汇率日志只记汇率变化，不夹带币种配置变更
- 缓存粒度更细（汇率 TTL 可以更短，1-3分钟）
- 逻辑职责清晰：一张表管"币种是什么"，一张表管"币种值多少"

### 业务来源

| 需求点 | 对应字段 |
|--------|---------|
| 需求：汇率规则"1 BSB = X 法币"，手动配置 | rate, rate_precision |
| 需求：兑换赠送比例（兑换送5%） | gift_ratio |
| 需求：汇率应用于充值展示/兑换计算/提现换算/报表统计 | 整张表被多处消费 |

### 完整字段

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | int64 | PK, AUTO_INCREMENT | 主键 |
| currency_code | string | NOT NULL, UK | 币种代码 |
| rate | int64 | NOT NULL | 汇率值（BIGINT存储，实际值=rate/10^rate_precision） |
| rate_precision | int32 | NOT NULL, DEFAULT 4 | 汇率精度（小数位数） |
| gift_ratio | int32 | NOT NULL, DEFAULT 0 | 兑换赠送比例（万分比，500=5%） |
| status | int32 | NOT NULL, DEFAULT 1 | 状态 0=禁用 1=启用 |
| create_by | *int64 | | 创建人ID |
| create_at | int64 | NOT NULL | 创建时间 |
| update_by | *int64 | | 更新人ID |
| update_at | int64 | NOT NULL | 更新时间 |

### 索引

```
UNIQUE KEY uk_currency_code (currency_code)
  → 每个币种一条汇率记录
```

### 设计说明

**汇率也用 BIGINT 存储**：理由同金额字段。`1 BSB = 23000 VND` → rate=230000000, rate_precision=4 → 实际值=230000000/10000=23000。全链路整数运算，无浮点精度问题。

**不需要软删除**：汇率不删除，只禁用。币种禁用后对应汇率自然不被使用。

## 2.3 【应该】currency_log — 币种操作日志表

### 为什么是"应该"而非"必须"

需求文档明确要求"操作行为全部记录操作日志"，且产品原型有独立的"操作日志"查询页面。从合规角度这是强需求。但从系统运转角度，没有操作日志不影响充值/提现/兑换等核心流程。

**建议第一期就做**：B端CRUD代码量不大，写日志几行代码的事，不应推迟。合规审计找上来时再补就晚了。

### 业务来源

| 需求点 | 说明 |
|--------|------|
| 产品原型：操作日志Tab页 | B端需要查询谁、什么时间、改了什么 |
| 需求：修改前值、修改后值 | 审计追溯需要 |

### 完整字段

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | int64 | PK, AUTO_INCREMENT | 主键 |
| currency_code | string | NOT NULL | 币种代码 |
| action | int32 | NOT NULL | 操作类型 1=新增 2=编辑 3=启用 4=禁用 5=删除 |
| field_name | *string | | 变更字段名（编辑时记录） |
| old_value | *string | | 变更前值 |
| new_value | *string | | 变更后值 |
| remark | *string | | 备注 |
| operator_id | int64 | NOT NULL | 操作人ID |
| operator_name | *string | | 操作人名称（冗余存储，避免关联查询） |
| create_at | int64 | NOT NULL | 操作时间 |

### 特点

- **纯追加（Append-Only）**：只写不改不删，无 update/delete 审计字段
- **不影响核心流程**：写日志失败不应阻塞币种配置的保存操作（可异步/降级）

### 索引

```
INDEX idx_currency_code (currency_code)     → 按币种查日志
INDEX idx_create_at (create_at)             → 按时间范围查
```

# 三、钱包模块 — 表结构详解

> 模块定位：C端核心功能 + RPC服务提供方
> 你负责，是你的核心主责

---

## 3.1 【必须】user_balance — 用户余额表

### 为什么必须

**这是整个钱包系统的心脏**。需求明确了"4子钱包 × 多币种 = 余额矩阵"，这张表是唯一存储用户资金余额的地方。没有它，钱包就不存在。

所有业务流程最终都归结到这张表的读写：
- 充值入账 → available +
- 兑换 → A币种 available - / B币种 available +
- 提现冻结 → available - / frozen +
- 提现扣款 → frozen -
- 提现解冻 → frozen - / available +
- 人工加减款 → available ±

### 业务来源

| 需求点 | 对应字段 |
|--------|---------|
| 需求：4个子钱包（中心/奖励/代理/主播） | wallet_type |
| 需求：每个子钱包下按币种隔离余额 | currency |
| 需求：提现先冻结后扣款 | frozen 字段 |
| 需求：用户查看余额 | available 字段 |
| 需求：余额矩阵 = user_id × currency × wallet_type | 三元组唯一键 |

### 完整字段

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | int64 | PK, AUTO_INCREMENT | 主键 |
| user_id | int64 | NOT NULL | 用户ID |
| currency | string(10) | NOT NULL | 币种代码 |
| wallet_type | int32 | NOT NULL | 子钱包类型 1=中心 2=奖励 3=代理 4=主播 |
| available | int64 | NOT NULL, DEFAULT 0 | 可用余额（最小单位整数） |
| frozen | int64 | NOT NULL, DEFAULT 0 | 冻结余额（提现冻结中的金额） |
| version | int32 | NOT NULL, DEFAULT 0 | 乐观锁版本号（备用） |
| create_by | *int64 | | 创建人ID |
| create_at | int64 | NOT NULL | 创建时间 |
| update_by | *int64 | | 更新人ID |
| update_at | int64 | NOT NULL | 更新时间 |

### 核心设计决策

**金额为什么用 BIGINT 不用 DECIMAL**

```
理由：
  1. 整数运算无精度丢失（0.1+0.2≠0.3 的问题不存在）
  2. 数据库层 BIGINT 比较和加减更快
  3. 现有工程所有 ID 和时间戳都是 int64，风格一致
  4. VND 最小单位就是1（无小数），BIGINT 天然适配

存储换算规则（由 currency_config.precision 决定）：
  BSB  (precision=2): 存储 100050 → 展示 1000.50 BSB
  VND  (precision=0): 存储 1000000 → 展示 1,000,000 VND
  USDT (precision=2): 存储 10050 → 展示 100.50 USDT
```

**available 和 frozen 为什么不分表**

```
提现冻结操作需要原子地：available -= amount, frozen += amount
同一行同一个 UPDATE 语句最简单最安全
分表需要跨表事务，复杂度暴增但收益为零
```

**为什么不需要软删除**

余额记录永不删除。用户注册后余额行始终存在（余额为0也是有效状态）。

### 索引

```
UNIQUE KEY uk_user_currency_wallet (user_id, currency, wallet_type)
  → 所有余额操作的必经之路，覆盖全部查询场景
  → 无需额外索引
```

### 并发安全 SQL 模式

```sql
-- 增加余额（充值入账、人工加款）
UPDATE user_balance SET available = available + ?
WHERE user_id = ? AND currency = ? AND wallet_type = ?

-- 扣减余额（兑换扣减、人工减款）— 乐观锁
UPDATE user_balance SET available = available - ?
WHERE user_id = ? AND currency = ? AND wallet_type = ? AND available >= ?

-- 提现冻结 — 乐观锁
UPDATE user_balance SET available = available - ?, frozen = frozen + ?
WHERE user_id = ? AND currency = ? AND wallet_type = ? AND available >= ?

-- 提现确认扣款
UPDATE user_balance SET frozen = frozen - ?
WHERE user_id = ? AND currency = ? AND wallet_type = ? AND frozen >= ?

-- 提现解冻退回
UPDATE user_balance SET available = available + ?, frozen = frozen - ?
WHERE user_id = ? AND currency = ? AND wallet_type = ? AND frozen >= ?
```

检查受影响行数：0行 = 余额/冻结不足。

## 3.2 【必须】balance_flow — 余额变动流水表

### 为什么必须

需求文档明确要求"账务一致性——余额变动必须和流水记录严格一致"。这是钱包系统的**审计底线和对账基石**。每一笔余额变动必须有对应流水记录，否则出了问题无法追溯。

同时，C端"奖励记录"等查询也依赖流水表数据。

### 业务来源

| 需求点 | 对应字段 |
|--------|---------|
| 需求：账务一致性 | before_balance + after_balance（可验证连续性） |
| 需求：幂等性（防重复入账） | uk_order_type 唯一键兜底 |
| 需求：记录查询（C端充值/提现/兑换/奖励列表） | 按 change_type 过滤 |
| 需求：关联订单号追溯 | ref_order_no |

### 完整字段

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | int64 | PK, AUTO_INCREMENT | 主键 |
| user_id | int64 | NOT NULL | 用户ID |
| currency | string(10) | NOT NULL | 币种代码 |
| wallet_type | int32 | NOT NULL | 子钱包类型 |
| change_type | int32 | NOT NULL | 变动类型枚举（见下方） |
| amount | int64 | NOT NULL | 变动金额（正=增加 负=减少） |
| before_balance | int64 | NOT NULL | 变动前可用余额 |
| after_balance | int64 | NOT NULL | 变动后可用余额 |
| ref_order_no | string(32) | NOT NULL, DEFAULT '' | 关联订单号 |
| remark | *string | | 备注 |
| create_at | int64 | NOT NULL | 创建时间 |

### change_type 枚举定义

```
1  = 充值入账（中心钱包+）
2  = 充值奖金（奖励钱包+）
3  = 提现冻结（available- frozen+）
4  = 提现确认扣款（frozen-）
5  = 提现解冻退回（frozen- available+）
6  = 兑换扣减（源币种中心钱包-）
7  = 兑换增加（BSB中心钱包+）
8  = 兑换赠送（BSB奖励钱包+）
9  = 人工加款
10 = 人工减款
11 = 人工补单入账
```

### 特点

- **不可变记录（Append-Only）**：一旦写入不允许修改和删除。写错只能追加"冲正"流水
- **没有 update/delete 审计字段**：不会被更新，只有 create_at
- **连续性校验能力**：`flow_N.after_balance == flow_N+1.before_balance`，不等则有异常

### 索引

```
INDEX idx_user_currency (user_id, currency)     → C端按用户+币种查记录
INDEX idx_ref_order (ref_order_no)               → 通过订单号反查流水
INDEX idx_create_at (create_at)                  → 按时间范围查询
UNIQUE KEY uk_order_type (ref_order_no, change_type)  → 幂等兜底
```

**uk_order_type 的幂等意义**：同一个订单号+同一种变动类型只能有一条流水。即使应用层幂等判断被并发穿透，数据库唯一键也会阻止重复写入。

### 为什么必须

提现审核流程在财务侧，B端"提现记录"管理页面+多级审核操作都依赖这张表。

### 与钱包 withdraw_order 的区别

```
withdraw_order（钱包侧）：
  管理 = 冻结/扣款/解冻的余额状态
  视角 = 用户和余额视角

finance_withdraw（财务侧）：
  管理 = 风控/财务/出款三级审核流程 + 渠道出款 + 锁定/解锁
  视角 = 运营审核视角

同样通过 order_no 关联
```

### 完整字段（预估）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | int64 | 主键 |
| order_no | string | 平台订单号 |
| user_id | int64 | 用户ID |
| currency | string | 币种 |
| amount | int64 | 提现金额 |
| fee | int64 | 手续费 |
| method | string | 提现方式 |
| channel_id | *int64 | 出款通道ID |
| channel_order_no | *string | 渠道订单号 |
| risk_status | int32 | 风控审核状态 |
| finance_status | int32 | 财务审核状态 |
| payout_status | int32 | 出款审核状态 |
| lock_by | *int64 | 锁定审核人（防并发） |
| bsb_amount | int64 | BSB换算金额 |
| status | int32 | 总状态 |
| fail_reason | *string | 失败原因 |
| 审计字段 | ... | |

## 4.5 【必须】manual_adjustment — 人工修正表

### 为什么必须

需求明确了三种人工修正操作（补单/加款/减款），各有独立的创建+审核流程。审核通过后调用钱包 RPC 执行余额操作。

### 业务来源

| 需求点 | 说明 |
|--------|------|
| 补单（B+16位） | 充值漏入账，人工补录 |
| 加款（A+16位） | 活动补偿、客诉赔付 |
| 减款（M+16位） | 用户违规、误操作退回 |
| 审核机制：创建人 ≠ 审核人 | 二级确认 |

### 完整字段（预估）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | int64 | 主键 |
| adjust_no | string | 修正单号（B/A/M + 16位） |
| adjust_type | int32 | 类型 1=补单 2=加款 3=减款 |
| user_id | int64 | 目标用户ID |
| currency | string | 币种 |
| wallet_type | int32 | 指定子钱包类型 |
| amount | int64 | 金额 |
| ref_order_no | *string | 关联原始订单号（补单时） |
| reason | string | 修正原因 |
| status | int32 | 状态 1=待审核 2=审核通过 3=审核拒绝 4=执行成功 5=执行失败 |
| reviewer_id | *int64 | 审核人ID |
| reviewed_at | *int64 | 审核时间 |
| execute_result | *string | 执行结果描述 |
| 审计字段 | ... | |

### 与钱包的关系

审核通过后，财务调用钱包的 RPC 接口执行余额操作：
- 补单 → `钱包.人工补单入账(adjust_no, user_id, currency, amount, ref_order_no)`
- 加款 → `钱包.人工加款(adjust_no, user_id, currency, wallet_type, amount)`
- 减款 → `钱包.人工减款(adjust_no, user_id, currency, wallet_type, amount)`

钱包侧用 adjust_no 做幂等键。

## 4.6 【应该】withdraw_review — 提现审核流水表

### 为什么是"应该"

需求描述了三级审核（风控→财务→出款），每级审核的操作人、时间、结果需要记录。可以用独立的审核流水表，也可以在 finance_withdraw 中直接记录。

如果用独立表，更清晰（一次审核一条记录，可追溯全流程），推荐这种方式。

### 完整字段（预估）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | int64 | 主键 |
| withdraw_id | int64 | 关联提现记录ID |
| order_no | string | 提现订单号 |
| review_level | int32 | 审核级别 1=风控 2=财务 3=出款 |
| reviewer_id | int64 | 审核人ID |
| result | int32 | 结果 1=通过 2=拒绝 |
| remark | *string | 审核备注 |
| create_at | int64 | 审核时间 |

---

## 4.7 【应该】channel_account — 通道账户配置表

### 为什么是"应该"

支付通道的 API 接入参数（商户号、密钥、回调地址等）可能很复杂且不同通道不同格式。从 payment_channel 中拆出来管理更清晰。

也可以直接用 payment_channel.api_config（JSON字段）替代。取决于财务开发者的设计选择。

---

## 4.8 【可选】channel_stats / finance_stats — 统计表

### 为什么是"可选"

需求有"通道统计""充值统计""提现统计"页面，但统计数据可以实时聚合查询（从订单表 SUM/COUNT），也可以预计算存储到统计表。

MVP 阶段建议直接聚合查询，数据量大了再考虑预计算。

## 5.1 关系图

```
                    ┌─────────────────┐
                    │ currency_config  │ ← 所有表的 currency 字段值来源
                    │ exchange_rate    │ ← 汇率被多处消费
                    │ currency_log     │
                    │ rate_log         │
                    └────────┬────────┘
                             │ 币种/汇率数据
                    ┌────────┴────────┐
                    │                 │
            ┌───────▼──────┐   ┌──────▼───────┐
            │  钱包模块表群  │   │  财务模块表群  │
            └───────┬──────┘   └──────┬───────┘
                    │                 │
                    │  order_no 关联   │
                    ├─────────────────┤
                    │                 │
     ┌──────────────┼──────────────┐  │
     │              │              │  │
┌────▼────┐  ┌──────▼──────┐  ┌───▼──▼──────┐
│  user_  │  │  balance_   │  │  deposit_   │
│ balance │◄─│   flow      │◄─│   order     │
└─────────┘  └─────────────┘  │ withdraw_   │
                              │   order     │  ←→  finance_deposit
                              │ exchange_   │  ←→  finance_withdraw
                              │   order     │
                              └──────┬──────┘
                                     │ account_id
                              ┌──────▼──────┐
                              │  withdraw_  │
                              │   account   │
                              └─────────────┘


  manual_adjustment ──(adjust_no)──→ balance_flow
                    ──(RPC)──→ user_balance
```

## 5.2 关键关联关系

| 关系 | 关联键 | 说明 |
|------|--------|------|
| currency_config ↔ exchange_rate | currency_code | 一个币种一条汇率 |
| user_balance ↔ balance_flow | user_id + currency + wallet_type | 每次余额变动写一条流水 |
| deposit_order → balance_flow | ref_order_no = order_no | 充值入账写 1~2 条流水 |
| withdraw_order → balance_flow | ref_order_no = order_no | 提现操作写 1 条流水 |
| exchange_order → balance_flow | ref_order_no = order_no | 兑换写 2~3 条流水 |
| withdraw_order → withdraw_account | account_id | 提现使用的账户信息 |
| deposit_order ↔ finance_deposit | order_no | 钱包侧和财务侧通过订单号关联 |
| withdraw_order ↔ finance_withdraw | order_no | 同上 |
| manual_adjustment → balance_flow | adjust_no = ref_order_no | 人工修正通过修正单号关联流水 |

# 七、开发落地建议

## 7.1 建表顺序

```
Phase 1（基础设施）：
  1. currency_config    — 币种配置（所有模块的地基）
  2. exchange_rate      — 汇率配置

Phase 2（核心账本）：
  3. user_balance       — 用户余额（心脏）
  4. balance_flow       — 变动流水（审计底线）

Phase 3（订单表群）：
  5. exchange_order     — 兑换订单（最先验证的交易链路）
  6. deposit_order      — 充值订单
  7. withdraw_order     — 提现订单

Phase 4（辅助表）：
  8. withdraw_account   — 提现账户
  9. currency_log       — 币种日志
  10. rate_log          — 汇率日志
```

## 7.2 建表后的下一步

```
建库建表 → GORM Gen 生成 model/query/repo
         → IDL 定义（优先组3：RPC暴露给财务的接口）
         → gen_kitex 生成桩代码
         → 搭建 ser-wallet 骨架
         → 开始编码（按四期开发计划）
```

## 7.3 表设计注意事项

```
1. 金额一律 BIGINT（int64），存最小单位整数
2. 时间一律 BIGINT（int64），毫秒级时间戳
3. 状态/枚举用 INT（int32），预留码段
4. 订单号用 VARCHAR(32)，雪花ID填充
5. 敏感信息（银行卡号等）AES 加密存储
6. 配置表加软删除，订单/流水/余额表不加
7. 不建外键，用应用层保证一致性
8. 索引按查询驱动，不先建再说
```

# 八、决策清单

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 你的表数量 | 必须7 + 应该3 = 10张 | 覆盖币种配置+钱包全部核心功能 |
| 钱包和财务充值/提现是否共表 | 分开（各自独立表） | 微服务各管各的数据，通过 order_no 关联 |
| 汇率是否与币种配置同表 | 分表 | 变更频率不同，缓存粒度不同，日志分开更清晰 |
| 日志表是否独立 | 独立（currency_log + rate_log） | 字段结构不同，查询场景独立 |
| 统计表是否建 | 可选不建 | MVP 先聚合查询，规模大了再预计算 |
| 奖励流水规则是否建表 | 可选不建 | 需求细节未定型，先代码或配置控制 |
| 余额快照是否建 | 可选不建 | balance_flow 已支持基本对账 |
