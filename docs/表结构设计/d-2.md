## 2. 分级标准说明

### 2.1 三级分类定义

| 级别 | 标记 | 定义 | 判断依据 |
|------|------|------|---------|
| **必须 (Must)** | ■ | 缺了这张表业务跑不通，核心功能无法实现 | 需求文档明确提到的核心实体 + 资金安全硬性要求 |
| **应该 (Should)** | □ | 不做也能跑，但会留坑（性能、安全、运营管控） | 业务合理性推导 + 行业惯例 + 非功能性需求 |
| **可选 (Nice to have)** | ○ | 锦上添花，提升可观测性/运营效率，可后期补建 | 运维需要 + 数据分析需要 + 未来扩展预留 |

### 2.2 判断原则

```
"产品需求明确描述了这个业务实体" → 必须
"产品没说但缺了会导致安全/一致性问题" → 必须
"产品提到了但属于配置化管理需求" → 应该
"产品没提但行业实践中都有" → 应该
"对账/统计/监控类的辅助能力" → 可选
```

## 3. 钱包核心表（必须）

> 核心表是整个资金体系的基石，所有资金流转最终都体现在这 3 张表上

### 3.1 ■ wallet — 用户钱包

**为什么必须**：没有这张表，用户余额无处存放，所有资金操作无载体。

**需求依据**：
- [需求 2.1] "钱包按币种维度隔离，每个币种独立余额"
- [需求 2.1] "每个币种下有中心钱包、奖励钱包、主播钱包、代理钱包、场馆钱包"
- [需求 5.1] "提现时需要冻结余额"

**表做什么**：存储每个用户每个币种每个钱包类型的余额和冻结金额。一个用户如果有 5 个币种 × 2 个钱包类型 = 10 行记录。

| 字段 | 类型 | 约束 | 说明 | 需求来源 |
|------|------|------|------|---------|
| id | BIGINT | PK, AUTO_INCREMENT | 主键 | [工程规范] |
| user_id | BIGINT | NOT NULL | 用户ID | [需求 2.1] 每个用户有钱包 |
| currency_code | VARCHAR(10) | NOT NULL | 币种代码 BSB/VND/IDR/THB/USDT | [需求 2.1] 按币种隔离 |
| wallet_type | TINYINT | NOT NULL | 钱包类型 1中心/2奖励/3主播/4代理/5场馆 | [需求 2.1] 多种子钱包 |
| balance | BIGINT | NOT NULL DEFAULT 0 | 余额（最小单位整数） | [需求 2.1] 独立余额 |
| frozen_amount | BIGINT | NOT NULL DEFAULT 0 | 冻结金额（最小单位整数） | [需求 5.1] 提现冻结 |
| status | TINYINT | NOT NULL DEFAULT 1 | 1正常/2冻结/3关闭 | [架构] 状态管理 |
| create_at | BIGINT | NOT NULL | 毫秒时间戳 | [工程规范] |
| update_at | BIGINT | NOT NULL | 毫秒时间戳 | [工程规范] |
| delete_at | BIGINT | NOT NULL DEFAULT 0 | 0=未删除 | [工程规范] |

**核心约束**：
- UNIQUE (user_id, currency_code, wallet_type, delete_at) — 同用户同币种同类型唯一
- 可用余额 = balance - frozen_amount（代码层计算，不建列）
- balance >= 0、frozen_amount >= 0 — 代码层强校验

### 3.2 ■ wallet_flow — 资金流水

**为什么必须**：没有流水表无法审计、无法幂等防重、无法对账。这是资金安全的基石。

**需求依据**：
- [需求 9] "账务流水与余额必须强一致"
- [需求 9] "充值回调必须幂等"
- [需求 6.1] "记录分四大类：充值、提现、兑换、奖励"

**表做什么**：记录每一笔余额变动（只追加不修改不删除），形成完整审计轨迹。通过 idempotent_key 唯一约束保证幂等。

| 字段 | 类型 | 约束 | 说明 | 需求来源 |
|------|------|------|------|---------|
| id | BIGINT | PK, AUTO_INCREMENT | 主键 | [工程规范] |
| user_id | BIGINT | NOT NULL | 用户ID | 关联用户 |
| currency_code | VARCHAR(10) | NOT NULL | 币种代码 | 关联币种 |
| wallet_type | TINYINT | NOT NULL | 钱包类型 | 关联具体钱包 |
| direction | TINYINT | NOT NULL | 1入账/2出账/3冻结/4解冻 | [需求 6.1] 记录类型 |
| biz_type | TINYINT | NOT NULL | 1充值/2提现/3兑换/4奖金/5人工加/6人工减/7补单/8冻结/9解冻/10确认扣 | [需求 6.1] 业务分类 |
| amount | BIGINT | NOT NULL | 变动金额（最小单位，正数） | [需求 6.2] 金额 |
| before_balance | BIGINT | NOT NULL | 变动前余额 | [需求 9] 强一致审计 |
| after_balance | BIGINT | NOT NULL | 变动后余额 | [需求 9] 强一致审计 |
| ref_order_no | VARCHAR(64) | NOT NULL DEFAULT '' | 关联业务订单号 | [需求 8.4] 详情页 |
| idempotent_key | VARCHAR(128) | NOT NULL | 幂等键（格式：{biz_type}:{ref_order_no}:{wallet_type}） | [需求 9] 幂等性 |
| remark | VARCHAR(256) | NOT NULL DEFAULT '' | 备注 | 运营可见说明 |
| create_at | BIGINT | NOT NULL | 毫秒时间戳 | [工程规范] |
| update_at | BIGINT | NOT NULL | 毫秒时间戳 | [工程规范] |
| delete_at | BIGINT | NOT NULL DEFAULT 0 | 0=未删除 | [工程规范] |

**核心约束**：
- UNIQUE (idempotent_key, delete_at) — 幂等最终防线
- INDEX (user_id, currency_code, create_at) — C 端记录分页查询
- INDEX (ref_order_no) — 按订单号查关联流水
- **append-only**：只追加、不更新、不删除

### 3.3 ■ freeze_detail — 冻结明细

**为什么必须**：提现冻结可能跨两个钱包（中心 + 奖励），解冻时必须知道每个钱包冻了多少才能精确还原。没有这张表，解冻无法正确分配。

**需求依据**：
- [架构决策5] 冻结→确认/解冻三阶段
- [需求 2.2] 消费优先级：先中心后奖励
- [需求 5.1] 提现涉及冻结

**表做什么**：记录每笔提现冻结操作中，每个钱包类型具体冻结了多少金额，跟踪状态直到确认扣款或解冻。

| 字段 | 类型 | 约束 | 说明 | 需求来源 |
|------|------|------|------|---------|
| id | BIGINT | PK, AUTO_INCREMENT | 主键 | [工程规范] |
| user_id | BIGINT | NOT NULL | 用户ID | 关联用户 |
| currency_code | VARCHAR(10) | NOT NULL | 币种代码 | 关联币种 |
| wallet_type | TINYINT | NOT NULL | 钱包类型 | [需求 2.2] 按优先级分配 |
| order_no | VARCHAR(64) | NOT NULL | 关联提现订单号 | 关联 withdraw_order |
| freeze_amount | BIGINT | NOT NULL | 该钱包冻结金额（最小单位） | [架构决策5] |
| status | TINYINT | NOT NULL DEFAULT 1 | 1冻结中/2已确认扣款/3已解冻 | [架构决策5] 三阶段 |
| create_at | BIGINT | NOT NULL | 毫秒时间戳 | [工程规范] |
| update_at | BIGINT | NOT NULL | 毫秒时间戳 | [工程规范] |
| delete_at | BIGINT | NOT NULL DEFAULT 0 | 0=未删除 | [工程规范] |

**核心约束**：
- INDEX (order_no) — 按提现订单查冻结明细
- 一笔提现产生 1-2 条记录（中心钱包一条 + 奖励钱包一条，若奖励不参与则只有一条）

### 4.3 ■ exchange_record — 兑换记录

**为什么必须**：兑换是独立的业务操作（法币→BSB），必须有记录表来承载 C 端"兑换记录"tab的数据。

**需求依据**：
- [需求 4] 兑换模块 — 法币→BSB + 赠送
- [需求 6.1] 兑换记录 tab

**表做什么**：记录每笔兑换的源/目标币种、金额、汇率、赠送等信息。

## 13. 分析依据与追溯

### 13.1 "必须"级别的判定依据

| 表名 | 判定理由 |
|------|---------|
| wallet | 没有余额载体，所有资金操作无处落地 |
| wallet_flow | 没有流水无法审计、无法幂等、无法对账，违反[需求 9]强一致要求 |
| freeze_detail | 跨钱包冻结无法正确解冻，资金安全漏洞 |
| recharge_order | 充值是核心业务流程，必须有订单跟踪 |
| withdraw_order | 提现涉及多阶段审核，必须有状态机承载 |
| exchange_record | 兑换是产品明确的功能模块，C端有独立记录tab |
| withdraw_account | 产品明确要求"首次填写后记录，后续复用" |
| currency_config | 没有币种配置无法知道支持哪些币种、汇率、展示规则 |
| rate_log | 产品明确有"汇率日志"查询页面 |
| base_currency_config | 基准币种影响全局换算，且不可更改 |
| deposit/payout_channel | 没有通道配置无法接入三方支付 |
| recharge/withdraw_method | 没有支付配置 C 端无法展示充值/提现方式 |
| manual_patch/credit/debit | 产品明确有补单/加款/减款功能 |

### 13.2 "应该"级别的判定依据

| 表名 | 判定理由 | 不做的后果 |
|------|---------|-----------|
| withdraw_limit_config | 产品明确提到提现限制参数 | 限制参数硬编码，改限制要上线 |
| user_currency_preference | 产品说"切换币种后联动" | 用 Redis 替代可以，但有过期丢失风险 |
| channel_polling_config | 产品明确有轮询规则配置 | 轮询算法参数硬编码 |
| channel_daily_stat | B 端有通道统计页面 | 每次查询实时聚合，数据量大后慢 |

### 13.3 "可选"级别的判定依据

| 表名 | 判定理由 | 不做的后果 |
|------|---------|-----------|
| wallet_daily_snapshot | 行业惯例但非产品明确需求 | 无法自动对账，异常只能人工排查 |
| rate_source_log | 排障辅助工具 | 三方 API 异常时缺少原始数据 |
| recharge_daily_stat | 大数据量优化 | 数据量小时实时聚合即可 |
| withdraw_daily_stat | 大数据量优化 | 同上 |
