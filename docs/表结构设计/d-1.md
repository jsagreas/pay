# 表结构设计（完整 DDL + 必须/应该/可选分级）

> 来源：数据模型(d-1) + 需求理解(d-1) + 架构设计(d-1) + 实现思路(d-1) + 流程思路(d-1) + 工程认知(d-1) + 现有工程 DDL 实证
> 定位：从数据模型推进到可直接执行的表结构设计，每张表都明确"为什么需要、字段干什么、优先级如何"
> 约束：严格遵循现有工程 DDL 规范（utf8mb4、InnoDB、bigint 时间戳、delete_at=0 软删除、审计字段六件套）

## 一、设计原则与工程 DDL 规范

### 1.1 DDL 文件规范

基于现有工程 SQL 文件（ser-buser/sql/、ser-i18n/sql/、ser-item/sql/、ser-s3/sql/）的实证，所有 DDL 文件必须遵循以下格式：

```sql
-- ser-wallet DDL
SET NAMES utf8mb4;
SET FOREIGN_KEY_CHECKS = 0;

-- 表定义...

SET FOREIGN_KEY_CHECKS = 1;
```

### 1.2 表级规范

| 规范项 | 值 | 依据 |
|--------|-----|------|
| 引擎 | `ENGINE=InnoDB` | 全工程统一 |
| 字符集 | `DEFAULT CHARSET=utf8mb4` | 全工程统一 |
| 排序规则 | `COLLATE=utf8mb4_unicode_ci` | ser-buser/ser-i18n 实证 |
| 主键 | `id BIGINT NOT NULL AUTO_INCREMENT` | 全工程统一 |
| 软删除 | `delete_at BIGINT NOT NULL DEFAULT 0` | 0=未删除，非0=删除时间戳 |
| 表注释 | `COMMENT='中文说明'` | 全工程统一 |

### 1.3 字段级规范

| 规范项 | 规范 | 依据 |
|--------|------|------|
| 时间字段 | `BIGINT NOT NULL DEFAULT 0`，存毫秒时间戳 | ser-app Banner 等模型实证 |
| 金额字段 | `BIGINT NOT NULL DEFAULT 0`，存最小单位整数 | 架构设计 ADR-5 |
| 汇率字段 | `BIGINT NOT NULL DEFAULT 0`，×10^8 存储 | 数据模型 currency 定义 |
| 百分比字段 | `INT NOT NULL DEFAULT 0`，×100 存储 | 1.40% → 存 140 |
| 状态/类型 | `TINYINT NOT NULL DEFAULT 0` | 全工程统一 |
| 字符串 | `VARCHAR(N) NOT NULL DEFAULT ''` | 非空字符串统一有默认值 |
| 布尔标记 | `TINYINT NOT NULL DEFAULT 0`（0=否 1=是） | 不用 BIT |

### 1.4 审计字段六件套（通用模板）

有写操作的表必须包含以下审计字段，由 UserPlugin GORM 回调自动填充：

```sql
`create_by`        BIGINT       NOT NULL DEFAULT 0 COMMENT '创建人ID',
`create_by_name`   VARCHAR(64)  NOT NULL DEFAULT '' COMMENT '创建人名称',
`create_at`        BIGINT       NOT NULL DEFAULT 0 COMMENT '创建时间（毫秒时间戳）',
`update_by`        BIGINT       NOT NULL DEFAULT 0 COMMENT '更新人ID',
`update_by_name`   VARCHAR(64)  NOT NULL DEFAULT '' COMMENT '更新人名称',
`update_at`        BIGINT       NOT NULL DEFAULT 0 COMMENT '更新时间（毫秒时间戳）',
```

### 1.5 索引命名规范

| 类型 | 前缀 | 示例 |
|------|------|------|
| 唯一索引 | `uk_` | `uk_order_no` |
| 普通索引 | `idx_` | `idx_user_id` |

### 1.6 不使用的约定

| 不用的 | 原因 |
|--------|------|
| 外键约束 | 全工程无一处使用 DB 层外键，逻辑外键在应用层保证 |
| DECIMAL 类型 | GORM 映射不友好（映射为 string 或 float64），改用 BIGINT 整数 |
| DATETIME 类型 | 全工程统一用 BIGINT 毫秒时间戳 |
| BIT 类型 | 改用 TINYINT 0/1 |
| JSON 列类型 | 存 VARCHAR/TEXT 字符串，应用层序列化/反序列化 |

## 二、表优先级分类标准

按"是否能绕过"和"影响范围"划分三个等级：

| 等级 | 含义 | 判定标准 |
|------|------|---------|
| **必须 (Must)** | 无论需求怎么变都绕不过，缺少则核心业务无法运作 | 产品需求明确描述 + 多个流程依赖 + 架构设计已确认 |
| **应该 (Should)** | 产品需求有描述，但可以暂时用其他方式替代或延后 | 有明确需求来源，但独立性强，不阻塞主链路 |
| **可选 (Optional)** | 当前需求未明确，但工程常识或后续扩展可能需要 | 预留性质，当前阶段可以不建 |

## 三、表清单全景

### 3.1 汇总表

| 序号 | 表名 | 所属域 | 优先级 | 职责一句话 |

| 6 | `withdraw_order` | 订单域 | **必须** | 提现订单全生命周期 |
| 7 | `exchange_record` | 订单域 | **必须** | 兑换操作记录 |
| 8 | `withdraw_account` | 账户域 | **必须** | 用户绑定的提现账户信息 |

### 3.2 优先级分布

```
必须 (Must) — 8张表
  currency / exchange_rate_log / wallet_account / transaction_log
  deposit_order / withdraw_order / exchange_record / withdraw_account
  → 覆盖：币种配置、余额管理、充值、提现、兑换、提现账户全部核心链路

应该 (Should) — 2张表
  deposit_config / withdraw_config
  → 覆盖：充值档位配置、提现限额配置（B端配置页面需求明确，但可初期硬编码）

可选 (Optional) — 2张表
  base_currency_config / freeze_record
  → 覆盖：基准币种设置（可用ETCD替代）、独立冻结记录（可合并到流水表）
```

## 四、必须表 — 详细设计

### 4.1 currency（币种配置表）

#### 优先级：必须

**为什么必须**：币种是整个钱包体系的数据地基。余额管理、充值、提现、兑换、汇率计算、C端展示全部依赖币种配置数据。没有这张表，其他所有表和业务都无法运作。

**产品需求依据**：
- 币种类型划分（法币/加密/平台）→ 币种配置需求文档/3-1
- 后台配置页面字段列表 → 币种配置需求文档/3-3 + 产品原型
- 汇率浮动/阈值/入出款汇率计算 → 币种配置需求文档/4-2
- BSB基准币种 → 币种配置需求文档/3-2
- 金额展示规则（千分位/小数位/货币符号）→ 多币种钱包需求文档/4-1

#### DDL

```sql
-- ----------------------------
-- Table: currency（币种配置表）
-- ----------------------------
DROP TABLE IF EXISTS `currency`;
CREATE TABLE `currency` (
    `id`               BIGINT       NOT NULL AUTO_INCREMENT COMMENT '主键ID',

    -- 币种基本信息
    `currency_code`    VARCHAR(10)  NOT NULL COMMENT '币种代码（VND/IDR/THB/USDT/BSB）',
    `currency_name`    VARCHAR(50)  NOT NULL COMMENT '币种名称（越南盾/印尼盾/泰铢/...）',
    `currency_type`    TINYINT      NOT NULL COMMENT '币种类型：1法定货币 2加密货币 3平台货币',
    `country`          VARCHAR(50)  NOT NULL DEFAULT '' COMMENT '所属国家（越南/印尼/泰国/...）',
    `icon_url`         VARCHAR(512) NOT NULL DEFAULT '' COMMENT '币种图标URL（SVG/PNG，≤5KB）',

    -- 展示规则
    `symbol`           VARCHAR(10)  NOT NULL DEFAULT '' COMMENT '货币符号（d/Rp/฿/$/...）',
    `thousands_sep`    VARCHAR(5)   NOT NULL DEFAULT ',' COMMENT '千分位符号（逗号或点）',
    `decimal_sep`      VARCHAR(5)   NOT NULL DEFAULT '.' COMMENT '小数位符号（点或逗号）',
    `decimal_places`   TINYINT      NOT NULL DEFAULT 0 COMMENT '小数位数（VND=0, THB=2, BSB=2, USDT=8）',
    `multiplier`       BIGINT       NOT NULL DEFAULT 1 COMMENT '最小单位换算倍率（10^decimal_places）',

    -- 汇率相关（对基准币种，×10^8 存储）
    `real_time_rate`   BIGINT       NOT NULL DEFAULT 0 COMMENT '实时汇率（×10^8存储）',
    `platform_rate`    BIGINT       NOT NULL DEFAULT 0 COMMENT '平台汇率（×10^8存储）',
    `deposit_rate`     BIGINT       NOT NULL DEFAULT 0 COMMENT '入款汇率（×10^8存储）',
    `withdraw_rate`    BIGINT       NOT NULL DEFAULT 0 COMMENT '出款汇率（×10^8存储）',
    `rate_float_pct`   INT          NOT NULL DEFAULT 0 COMMENT '汇率浮动百分比（×100存储，如140=1.40%）',
    `threshold_pct`    INT          NOT NULL DEFAULT 0 COMMENT '汇率更新阈值百分比（×100存储）',

    -- 状态与排序
    `status`           TINYINT      NOT NULL DEFAULT 0 COMMENT '状态：0禁用 1启用',
    `sort_order`       INT          NOT NULL DEFAULT 0 COMMENT '排序值（越小越靠前）',

    -- 审计字段（UserPlugin 自动填充）
    `create_by`        BIGINT       NOT NULL DEFAULT 0 COMMENT '创建人ID',
    `create_by_name`   VARCHAR(64)  NOT NULL DEFAULT '' COMMENT '创建人名称',
    `create_at`        BIGINT       NOT NULL DEFAULT 0 COMMENT '创建时间（毫秒时间戳）',
    `update_by`        BIGINT       NOT NULL DEFAULT 0 COMMENT '更新人ID',
    `update_by_name`   VARCHAR(64)  NOT NULL DEFAULT '' COMMENT '更新人名称',
    `update_at`        BIGINT       NOT NULL DEFAULT 0 COMMENT '更新时间（毫秒时间戳）',
    `delete_at`        BIGINT       NOT NULL DEFAULT 0 COMMENT '删除时间（毫秒，0=未删除）',

    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_code` (`currency_code`, `delete_at`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='币种配置表';
```

#### 字段说明

| 字段 | 类型 | 做什么用 | 产品依据 |
|------|------|---------|---------|
| `currency_code` | VARCHAR(10) | 币种唯一代码，如 VND/IDR/THB/USDT/BSB | 需求文档明确的币种代码，列表唯一 |
| `currency_name` | VARCHAR(50) | 币种中文/英文名称，展示用 | 产品原型列表页字段 |
| `currency_type` | TINYINT | 区分法币(1)/加密(2)/平台(3)，影响汇率精度和提现规则 | 需求文档/3-1 币种类型划分 |
| `country` | VARCHAR(50) | 币种所属国家，展示用 | 产品原型列表页字段 |
| `icon_url` | VARCHAR(512) | 币种图标 URL，上传到 ser-s3 后存储 | 产品原型编辑弹窗，SVG/PNG≤5KB |
| `symbol` | VARCHAR(10) | 货币符号（d/Rp/฿/$），C端金额展示拼接用 | 产品原型金额展示规则 |
| `thousands_sep` | VARCHAR(5) | 千分位分隔符，VND/IDR用`.`，THB/BSB用`,` | 需求文档/4-1 金额展示规则表 |
| `decimal_sep` | VARCHAR(5) | 小数点符号 | 需求文档/4-1 金额展示规则表 |
| `decimal_places` | TINYINT | 小数位数，VND=0/THB=2/BSB=2/USDT=8 | 需求文档/4-1 + 汇率展示规则 |
| `multiplier` | BIGINT | 最小单位倍率，VND=1/THB=100/BSB=100/USDT=10^8 | 架构设计金额精度方案 |
| `real_time_rate` | BIGINT | 三方API拉取的实时汇率，×10^8整数存储 | 需求文档/4-2 汇率规则，产品原型列表字段 |
| `platform_rate` | BIGINT | 平台汇率，阈值触发更新 | 需求文档/4-2 偏差值≥阈值时更新 |
| `deposit_rate` | BIGINT | 入款汇率 = platform_rate × (1 + rate_float_pct/10000) | 需求文档/4-2 汇率公式 |
| `withdraw_rate` | BIGINT | 出款汇率 = platform_rate × (1 - rate_float_pct/10000) | 需求文档/4-2 汇率公式 |
| `rate_float_pct` | INT | 汇率浮动%，×100存储，如1.40%→140 | 产品原型编辑弹窗"汇率浮动" |
| `threshold_pct` | INT | 汇率更新阈值%，×100存储 | 产品原型编辑弹窗"阈值" |
| `status` | TINYINT | 启用/禁用，禁用后C端不可充值不可提现 | 产品原型编辑弹窗状态开关 |
| `sort_order` | INT | 列表排序值 | 产品原型列表页"排序"字段 |

#### 索引说明

| 索引 | 字段 | 目的 |
|------|------|------|
| `uk_code` | `currency_code, delete_at` | 币种代码唯一（含软删除支持）；B端列表按code筛选 |

#### 汇率存储精度解释

```
所有汇率字段用 BIGINT ×10^8 存储（与 USDT 8位小数对齐）：

  BSB:VND 实时汇率 = 71.00
  → real_time_rate = 7100000000（71 × 10^8）

  浮动 1.40%（rate_float_pct = 140）
  → 入款汇率 = 71 × (1 + 0.0140) = 71.994
  → deposit_rate = 7199400000

  BSB:USDT 汇率 = 0.10000000
  → 存 10000000（0.1 × 10^8）

计算还原：实际汇率 = rate_value / 100000000
```

### 4.2 exchange_rate_log（汇率变更日志表）

#### 优先级：必须

**为什么必须**：产品原型在操作日志页面有独立的"汇率日志"Tab，明确定义了日志字段。需求文档规定"所有平台汇率更新后需要生成对应汇率日志"。这是运营监控汇率变化的必要工具。

**产品需求依据**：
- 日志字段定义（编号/时间/币种/实时汇率/平台汇率/入款/出款）→ 币种配置需求文档/4-4
- "所有平台汇率更新后需要生成对应汇率日志"→ 币种配置需求文档/4-2
- 产品原型"操作日志-汇率日志"Tab → 币种配置产品原型/3-操作日志.png

#### DDL

```sql
-- ----------------------------
-- Table: exchange_rate_log（汇率变更日志表）
-- ----------------------------
DROP TABLE IF EXISTS `exchange_rate_log`;
CREATE TABLE `exchange_rate_log` (
    `id`               BIGINT       NOT NULL AUTO_INCREMENT COMMENT '主键ID',

    -- 关联信息
    `currency_id`      BIGINT       NOT NULL COMMENT '关联币种ID',
    `currency_code`    VARCHAR(10)  NOT NULL COMMENT '币种代码（冗余，避免join）',

    -- 汇率快照
    `real_time_rate`   BIGINT       NOT NULL COMMENT '更新时的实时汇率（×10^8）',
    `platform_rate`    BIGINT       NOT NULL COMMENT '更新后的平台汇率（×10^8）',
    `deposit_rate`     BIGINT       NOT NULL COMMENT '更新后的入款汇率（×10^8）',
    `withdraw_rate`    BIGINT       NOT NULL COMMENT '更新后的出款汇率（×10^8）',

    -- 触发时的配置快照
    `rate_float_pct`   INT          NOT NULL COMMENT '当时的汇率浮动百分比（×100）',
    `threshold_pct`    INT          NOT NULL COMMENT '当时的阈值百分比（×100）',
    `deviation_pct`    INT          NOT NULL COMMENT '触发更新时的实际偏差值（×100）',

    -- 触发来源
    `trigger_type`     TINYINT      NOT NULL DEFAULT 1 COMMENT '触发类型：1自动（定时任务） 2手动（后台操作）',

    -- 时间（只增不改不删）
    `create_at`        BIGINT       NOT NULL DEFAULT 0 COMMENT '日志生成时间（毫秒时间戳）',

    PRIMARY KEY (`id`),
    INDEX `idx_currency_time` (`currency_id`, `create_at` DESC),
    INDEX `idx_code_time` (`currency_code`, `create_at` DESC)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='汇率变更日志表';
```

#### 字段说明

| 字段 | 做什么用 | 为什么需要 |
|------|---------|-----------|
| `currency_id` | 关联哪个币种 | 和 currency 表逻辑关联 |
| `currency_code` | 冗余币种代码 | B端分页查询时避免 join，产品原型列表直接展示代码 |
| `real_time_rate` | 触发更新时的三方实时汇率 | 产品原型"实时汇率"字段 |
| `platform_rate` | 更新后的平台汇率 | 产品原型"平台汇率"字段 |
| `deposit_rate` | 更新后的入款汇率 | 产品原型"入款汇率"字段 |
| `withdraw_rate` | 更新后的出款汇率 | 产品原型"出款汇率"字段 |
| `rate_float_pct` | 快照当时的浮动% | 事后审计：还原当时的计算参数 |
| `threshold_pct` | 快照当时的阈值% | 事后审计：还原触发条件 |
| `deviation_pct` | 实际偏差值 | 事后分析汇率波动幅度 |
| `trigger_type` | 区分自动/手动触发 | 审计追溯：是定时任务还是运营手动 |

#### 索引说明

| 索引 | 目的 |
|------|------|
| `idx_currency_time` | B端按币种ID+时间倒序分页查询 |
| `idx_code_time` | B端按币种代码+时间倒序分页查询（冗余索引支持按code直接筛选） |

#### 特殊说明

```
1. 只增不改不删：
   日志表不需要 update/delete 审计字段，也不需要 delete_at 软删除。
   每条记录一旦写入，永远保留。

2. 没有 create_by/update_by：
   自动更新由定时任务触发（trigger_type=1），没有操作人。
   手动更新时（trigger_type=2），操作人信息通过 ser-blog 审计日志记录，
   不需要在这张表重复存储。
```

### 4.3 wallet_account（钱包账户表）

#### 优先级：必须

**为什么必须**：这是整个钱包系统的心脏表。每个用户每个币种每种钱包类型的余额都存储在这里。充值入账、提现冻结/扣款、兑换扣减/入账、人工加减款——所有资金操作最终都归结为对这张表的修改。没有这张表，钱包系统不存在。

**产品需求依据**：
- 5类钱包（中心/奖励/主播/代理/场馆）→ 币种配置需求文档/2-1
- 每个币种一套独立余额 → 多币种钱包需求文档/4-1
- 资金钱包余额 = 中心余额 + 奖励余额 → 币种配置需求文档/2-1
- 冻结余额（提现时占用）→ 流程思路/提现流程
- 消费扣款优先级（先中心后奖励）→ 币种配置需求文档/2-1

#### DDL

```sql
-- ----------------------------
-- Table: wallet_account（钱包账户表）
-- ----------------------------
DROP TABLE IF EXISTS `wallet_account`;
CREATE TABLE `wallet_account` (
    `id`                 BIGINT       NOT NULL AUTO_INCREMENT COMMENT '主键ID',

    -- 定位维度：用户 × 币种 × 钱包类型 = 一行
    `user_id`            BIGINT       NOT NULL COMMENT '用户ID',
    `currency_id`        BIGINT       NOT NULL COMMENT '关联币种ID',
    `wallet_type`        TINYINT      NOT NULL COMMENT '钱包类型：1中心 2奖励 3主播 4代理 5场馆',

    -- 余额（bigint 最小单位整数）
    `available_balance`  BIGINT       NOT NULL DEFAULT 0 COMMENT '可用余额（最小单位整数）',
    `frozen_balance`     BIGINT       NOT NULL DEFAULT 0 COMMENT '冻结余额（最小单位整数）',

    -- 版本号（预留乐观锁）
    `version`            INT          NOT NULL DEFAULT 0 COMMENT '版本号（预留）',

    -- 审计字段
    `create_by`          BIGINT       NOT NULL DEFAULT 0 COMMENT '创建人ID',
    `create_by_name`     VARCHAR(64)  NOT NULL DEFAULT '' COMMENT '创建人名称',
    `create_at`          BIGINT       NOT NULL DEFAULT 0 COMMENT '创建时间（毫秒时间戳）',
    `update_by`          BIGINT       NOT NULL DEFAULT 0 COMMENT '更新人ID',
    `update_by_name`     VARCHAR(64)  NOT NULL DEFAULT '' COMMENT '更新人名称',
    `update_at`          BIGINT       NOT NULL DEFAULT 0 COMMENT '更新时间（毫秒时间戳）',
    `delete_at`          BIGINT       NOT NULL DEFAULT 0 COMMENT '删除时间（毫秒，0=未删除）',

    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_user_currency_type` (`user_id`, `currency_id`, `wallet_type`, `delete_at`),
    INDEX `idx_user_id` (`user_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='钱包账户表';
```

#### 字段说明

| 字段 | 做什么用 | 关键程度 |
|------|---------|---------|
| `user_id` | 账户归属用户 | 核心定位维度之一 |
| `currency_id` | 关联哪个币种 | 核心定位维度之二 |
| `wallet_type` | 钱包类型（中心/奖励/主播/代理/场馆） | 核心定位维度之三 |
| `available_balance` | 用户可自由支配的余额 | 所有资金操作的直接目标 |
| `frozen_balance` | 提现申请中被冻结的金额 | 提现流程的安全保障 |
| `version` | 预留乐观锁版本号 | 当前用悲观锁(FOR UPDATE)，version预留 |

#### 索引说明

| 索引 | 目的 |
|------|------|
| `uk_user_currency_type` | 保证一用户一币种一钱包类型只有一行；SELECT FOR UPDATE 行锁精确定位用这个索引 |
| `idx_user_id` | B端按用户ID查询其所有钱包；C端余额概览按用户ID查所有币种所有钱包类型 |

#### 核心设计逻辑

```
1. 一行一账户：
   user_id=123 + currency_id=1(VND) + wallet_type=1(中心) → 一行
   user_id=123 + currency_id=1(VND) + wallet_type=2(奖励) → 另一行
   用户首次操作某币种时懒创建。

2. 余额双字段：
   available_balance = 可花的钱
   frozen_balance = 提现中被锁的钱
   产品的"钱包余额" = available + frozen（用户看到的总额）
   但操作只能用 available

3. 并发安全：
   SELECT ... FOR UPDATE 走 uk_user_currency_type 精确锁单行。
   不同用户/不同币种完全不阻塞。

4. 金额存储示例：
   1000 VND (decimal_places=0) → available_balance = 1000
   3.50 THB (decimal_places=2) → available_balance = 350
   1.00 BSB (decimal_places=2) → available_balance = 100
   1.00000000 USDT (decimal_places=8) → available_balance = 100000000
```

### 4.4 transaction_log（交易流水表）

#### 优先级：必须

**为什么必须**：流水表是账务可追溯的根基。需求文档明确要求"完整、可追溯的资金记录"和"对账一致性"。架构设计已确认"余额变更和流水写入在同一事务中"的硬约束。这张表同时承担幂等键的功能（ref_order_no 唯一索引）。没有这张表，资金安全无法保障。

**产品需求依据**：
- "提供完整、可追溯的资金记录"→ 需求目标第4条
- "对账一致性"→ 非功能性需求
- 记录模块四类Tab（充值/提现/兑换/奖励）→ 多币种钱包需求文档/8-1
- 幂等保障 → 非功能性需求 + 架构设计 ADR-3

#### DDL

```sql
-- ----------------------------
-- Table: transaction_log（交易流水表）
-- ----------------------------
DROP TABLE IF EXISTS `transaction_log`;
CREATE TABLE `transaction_log` (
    `id`                 BIGINT       NOT NULL AUTO_INCREMENT COMMENT '主键ID',

    -- 关联维度
    `user_id`            BIGINT       NOT NULL COMMENT '用户ID',
    `currency_id`        BIGINT       NOT NULL COMMENT '币种ID',
    `wallet_type`        TINYINT      NOT NULL COMMENT '钱包类型：1中心 2奖励 3主播 4代理 5场馆',

    -- 操作信息
    `op_type`            TINYINT      NOT NULL COMMENT '操作类型：1入账 2扣款 3冻结 4解冻 5冻结扣款',
    `tx_type`            TINYINT      NOT NULL COMMENT '交易类型：1充值 2提现 3兑换入 4兑换出 5奖励 6人工加 7人工减 8投注扣 9返奖 10冻结 11解冻 12冻结扣款',
    `amount`             BIGINT       NOT NULL COMMENT '变动金额（最小单位，正数）',
    `direction`          TINYINT      NOT NULL COMMENT '方向：1收入(+) 2支出(-)',

    -- 余额快照（变动后，对账用）
    `balance_after`      BIGINT       NOT NULL COMMENT '变动后可用余额（最小单位）',
    `frozen_after`       BIGINT       NOT NULL COMMENT '变动后冻结余额（最小单位）',

    -- 关联单号（幂等键）
    `ref_order_no`       VARCHAR(64)  NOT NULL COMMENT '关联业务单号（充值/提现/兑换/修正单号）',

    -- 描述
    `remark`             VARCHAR(255) NOT NULL DEFAULT '' COMMENT '备注说明',

    -- 时间（只增不改不删）
    `create_at`          BIGINT       NOT NULL DEFAULT 0 COMMENT '创建时间（毫秒时间戳）',

    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_ref_order_tx` (`ref_order_no`, `tx_type`, `wallet_type`),
    INDEX `idx_user_currency_time` (`user_id`, `currency_id`, `create_at` DESC),
    INDEX `idx_user_tx_type_time` (`user_id`, `tx_type`, `create_at` DESC)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='交易流水表';
```

#### 字段说明

| 字段 | 做什么用 | 为什么需要 |
|------|---------|-----------|
| `user_id` | 归属用户 | 按用户查询流水 |
| `currency_id` | 归属币种 | C端按币种筛选记录 |
| `wallet_type` | 归属钱包类型 | 区分同一笔兑换操作影响的不同钱包 |
| `op_type` | 操作类型（入账/扣款/冻结/解冻/冻结扣款） | 对账时回放余额变动 |
| `tx_type` | 交易类型（充值/提现/兑换/奖励/人工/投注/冻结...） | C端记录Tab筛选 + B端按类型查询 |
| `amount` | 变动金额（正数） | 金额记录 |
| `direction` | 方向（1收入/2支出） | 比存负数更清晰，避免混淆 |
| `balance_after` | 变动后可用余额快照 | 对账：上一笔after ± 本笔amount = 本笔after |
| `frozen_after` | 变动后冻结余额快照 | 同上，冻结余额的对账追溯 |
| `ref_order_no` | 关联的业务单号 | 幂等键（唯一索引）+ 跨表关联查询 |
| `remark` | 备注 | 人工加减款的原因说明 |

#### 索引说明

| 索引 | 目的 |
|------|------|
| `uk_ref_order_tx` | **幂等键**：同一ref_order_no + tx_type + wallet_type 只允许写入一次，DB层防重复操作 |
| `idx_user_currency_time` | C端记录按用户+币种+时间倒序分页查询 |
| `idx_user_tx_type_time` | C端按交易类型筛选（充值Tab/提现Tab/兑换Tab/奖励Tab） |

#### tx_type 与 C端记录Tab 的映射

```
充值记录Tab  → WHERE tx_type = 1
提现记录Tab  → WHERE tx_type IN (2, 10, 11, 12)  -- 提现+冻结+解冻+冻结扣款
兑换记录Tab  → WHERE tx_type IN (3, 4)            -- 兑换入+兑换出
奖励记录Tab  → WHERE tx_type = 5
```

#### 特殊说明

```
1. 只增不改不删：
   没有 update_by/update_by_name/update_at/delete_at 字段。
   流水是审计和对账的唯一数据源，一旦写入永不修改。

2. 幂等机制：
   uk_ref_order_tx 唯一索引冲突 = 重复操作 = 幂等成功（直接返回）。
   与余额变更在同一事务内，原子性天然保证。

3. 为什么幂等键包含 wallet_type：
   兑换操作：一笔订单同时影响源币种中心钱包（兑换出）和 BSB中心+奖励钱包（兑换入）。
   同一 ref_order_no + 同一 tx_type=3(兑换入) 需要 wallet_type 区分中心和奖励两行。
```


| `channel_order_no` | 三方代付通道的订单号 | 与三方对账 |
| `amount` | 提现申请金额 | 提现记录详情"提现金额" |
| `fee` | 手续费 | 提现页面"手续费"展示；提现限额配置决定 |
| `actual_amount` | 实际到账 = amount - fee | 提现页面"到账金额"展示 |
| `withdraw_method` | 提现方式 | 提现记录详情"提现方式" |
| `withdraw_account_id` | 关联用户绑定的提现账户 | 出款时需要账户信息 |
| `channel_id` | 财务分配的代付通道 | 财务审核后分配 |
| `rate_snapshot` | BSB提现时锁定的出款汇率 | 保证提现金额按建单时汇率换算 |
| `frozen_amount` | 冻结的金额 | 解冻/扣款时的依据 |
| `frozen_wallet_type` | 冻结的是哪个钱包 | 解冻时知道恢复到哪个钱包 |
| `status` | 订单状态 | 状态机：待审核→审核中→出款中→已完成/已失败 |
| `audit_status` | 三级审核进度 | 冗余展示：风控→财务→出款各级通过状态 |
| `fail_reason` | 驳回/失败原因 | 提现记录详情"失败原因"（信息有误/系统维护/额度受限） |
| `paid_at` | 实际出款时间 | 提现记录详情"出款时间" |

#### 索引说明

| 索引 | 目的 |
|------|------|
| `uk_order_no` | 订单号唯一 |
| `idx_user_currency_status` | C端按用户+币种+状态筛选提现记录 |
| `idx_status` | 财务按状态批量查待审核订单；定时任务扫描超时订单 |
| `idx_status_create` | 按状态+时间范围筛选 |
| `idx_user_create` | C端按用户+时间倒序分页查提现记录 |

#### 特殊说明

```
1. 不设 delete_at：订单记录不做软删除。

2. frozen_amount 和 frozen_wallet_type：
   提现申请时冻结的金额和来源钱包。
   审核驳回 → UnfreezeBalance 用 frozen_amount 恢复到 frozen_wallet_type 钱包
   出款成功 → DebitBalance 用 frozen_amount 从冻结余额扣减

3. 今日已提现额度计算：
   SELECT SUM(amount) FROM withdraw_order
   WHERE user_id=? AND currency_id=? AND status NOT IN (4,5)
   AND create_at >= 今日零点毫秒时间戳

4. 状态机：
   待审核(0) → 审核中(1) → 出款中(2) → 已完成(3)
   任一环节驳回 → 已失败(4) + UnfreezeBalance
   出款失败 → 已失败(4) + UnfreezeBalance
   用户取消（如支持）→ 已取消(5) + UnfreezeBalance
```

### 4.7 exchange_record（兑换记录表）

#### 优先级：必须

**为什么必须**：兑换（法币→BSB）是产品需求明确定义的核心功能，是钱包内部自闭环的资金操作。兑换记录是C端"兑换"Tab的数据源。没有这张表，兑换功能没有操作记录，也无法满足"可追溯"的需求。

**产品需求依据**：

- 兑换获得 + 赠送奖励 → 多币种钱包需求文档/6
- 赠送部分需完成X倍流水 → 多币种钱包需求文档/6
- 兑换记录Tab字段 → 多币种钱包需求文档/8-1


#### 优先级：必须

**为什么必须**：产品需求明确"首次提现需填写账户信息""平台记录账户信息""后续提现无需重复填写"。提现流程强依赖绑定的提现账户信息（银行账号/电子钱包手机号/USDT地址）。没有这张表，提现功能缺少出款目标。

### 6.1 base_currency_config（基准币种配置表）

#### 优先级：可选

**为什么可选**：基准币种只有一条数据（USDT），设置后不可修改。可以用 ETCD 配置键实现，不一定需要独立数据库表。

**替代方案**：
```
ETCD 键：
  /slg/serv/wallet/base_currency_code = "USDT"
  /slg/serv/wallet/base_currency_set  = "1"

优势：零表维护，LoadAndWatch 热读取
```

### 6.2 freeze_record（冻结记录表）

#### 优先级：可选

**为什么可选**：冻结/解冻操作已通过 transaction_log 的 tx_type=10(冻结)/11(解冻)/12(冻结扣款) 完整记录。withdraw_order 也存储了 frozen_amount 和 frozen_wallet_type。独立冻结表的价值在于更直观地追踪冻结状态（冻结中/已解冻/已扣款），但功能上可以通过 withdraw_order.status 推导。

**什么时候需要建**：
- 如果未来有非提现场景的冻结需求（如投注预冻结、风控临时冻结）
- 如果运营需要独立的"冻结记录"管理页面

## 九、索引策略总结

### 9.1 核心业务查询与对应索引

| 业务查询 | 涉及表 | 使用索引 | SQL 模式 |
|---------|--------|---------|---------|
| 用户余额查询 | wallet_account | `uk_user_currency_type` | WHERE user_id=? AND currency_id=? AND wallet_type=? |
| SELECT FOR UPDATE行锁 | wallet_account | `uk_user_currency_type` | 同上 + FOR UPDATE |
| 幂等检测 | transaction_log | `uk_ref_order_tx` | INSERT ... ON DUPLICATE KEY 或唯一冲突捕获 |
| C端流水分页 | transaction_log | `idx_user_currency_time` | WHERE user_id=? AND currency_id=? ORDER BY create_at DESC LIMIT ? |
| C端按类型筛选 | transaction_log | `idx_user_tx_type_time` | WHERE user_id=? AND tx_type=? ORDER BY create_at DESC |
| 重复充值检测 | deposit_order | `idx_user_currency_amount_status` | WHERE user_id=? AND currency_id=? AND amount=? AND status IN (0,1) |
| 充值回调查单 | deposit_order | `uk_order_no` | WHERE order_no=? |
| 三方订单号查单 | deposit_order | `idx_channel_order` | WHERE channel_order_no=? |
| 超时订单扫描 | deposit_order | `idx_status_create` | WHERE status=0 AND create_at < ? |
| 待审核提现查询 | withdraw_order | `idx_status` | WHERE status=0 |
| 今日已提现统计 | withdraw_order | `idx_user_currency_status` | WHERE user_id=? AND currency_id=? AND status NOT IN (4,5) AND create_at >= ? |
| C端兑换记录 | exchange_record | `idx_user_source_time` | WHERE user_id=? AND source_currency_id=? ORDER BY create_at DESC |
| 汇率日志分页 | exchange_rate_log | `idx_currency_time` | WHERE currency_id=? ORDER BY create_at DESC |
| 用户提现账户 | withdraw_account | `idx_user_currency_type` | WHERE user_id=? AND currency_id=? AND account_type=? AND delete_at=0 |
