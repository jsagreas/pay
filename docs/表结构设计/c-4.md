# ser-wallet 表结构设计：从需求到落地的完整表结构方案

> 来源：需求理解 + 链路关系 + 流程思路 + 实现思路 + 架构设计 + 依赖关系 + 数据模型（7份 c-4 文档）+ 工程现有建表规范（ser-item/ser-app/ser-s3）
> 定位：针对钱包、币种、财务三大模块，逐表论证需要哪些表、每张表的字段和用途，按"必须/应该/可选"三级分类
> 遵循规范：工程现有建表规范（bigint unsigned PK、秒级时间戳、utf8mb4_bin、软删除、审计字段）
> 阶段：需求预评审期，"确定性骨架"定型，字段级细节保留弹性

---

## 一、总览：三大模块需要哪些表

### 1.1 表结构全景（按模块和分级）

| 分级 | 币种配置模块（我负责） | 钱包模块（我负责） | 财务模块（别人负责，参考） |
|------|---------------------|-------------------|------------------------|
| **必须** | currency_config、rate_change_log | wallet、wallet_transaction、deposit_order、withdrawal_order、withdrawal_account、exchange_order | payment_channel、payment_method、finance_order、manual_adjustment |
| **应该** | — | wager_requirement | audit_review |
| **可选** | rate_api_provider | wallet_daily_snapshot | channel_daily_stat |

**我负责的表：10 张（必须 8 + 应该 1 + 可选 1）**
**财务参考表：5 张（了解联动关系即可，不由我建）**

### 1.2 分级评估标准

| 分级 | 判定条件 | 含义 |
|------|---------|------|
| **必须（Must）** | 需求文档明确要求 + 缺失则核心链路断裂 | 系统无法运转，必须首版实现 |
| **应该（Should）** | 需求文档有提及但归属/细节待确认 + 缺失则部分功能降级 | 强烈建议首版实现，至少预留表结构 |
| **可选（Optional）** | 需求文档未明确 + 属于工程最佳实践或未来增强 | 首版可不建，后续按需补充 |

## 二、必须级表结构（8 张）

### 2.1 currency_config —— 币种配置表（含汇率）

#### 为什么必须

- 需求文档 B3 明确列出 18 个配置字段，B4 定义 4 种汇率和偏差阈值规则
- 币种是整个系统的"度量衡"，钱包的充值/兑换/提现、财务的通道配置/统计报表全部依赖币种数据
- 没有这张表，系统第一步（设置基准币种）都无法执行

#### 设计决策

- **合并存储币种定义和汇率数据**：币种与汇率是 1:1 关系（每个币种只有一套当前汇率），总数据量极少（4-6 条），拆表只增加 JOIN 复杂度无收益（架构决策 AD-10）
- **is_base 标记基准币种**：基准币种只是 currency_config 中的一条特殊记录，不需要独立表

#### 完整字段定义

| 字段名 | 类型 | 约束 | 说明 | 需求依据 |
|--------|------|------|------|---------|
| id | bigint unsigned | PK, AUTO_INCREMENT | 主键 | 工程规范 |
| currency_code | varchar(16) | NOT NULL | 币种编码，如 USDT/VND/BSB | B3 币种代码（唯一） |
| currency_name | varchar(64) | NOT NULL | 币种名称，如"越南盾" | B3 币种名称 |
| currency_type | tinyint unsigned | NOT NULL | 1=法定货币 2=加密货币 3=平台货币 | B3 币种类型 |
| country_code | varchar(16) | DEFAULT '' | 国家/地区编码，如 VN/ID/TH | B3 所属国家 |
| icon_url | varchar(255) | DEFAULT '' | 币种图标URL（SVG/PNG, ≤5KB） | B3 币种图标上传 |
| currency_symbol | varchar(8) | DEFAULT '' | 货币符号，如 ₫/Rp/฿ | B3 货币符号 |
| thousands_separator | varchar(4) | DEFAULT '' | 千分位符号：点号或逗号 | B3 千分位规则（VND用.，THB用,） |
| decimal_separator | varchar(4) | DEFAULT '' | 小数点符号 | B3 小数点规则 |
| decimal_digits | tinyint unsigned | NOT NULL DEFAULT 2 | 小数位数：法币2，加密8 | B3 精度规则 |
| is_base | tinyint unsigned | NOT NULL DEFAULT 0 | 是否基准币种 0否 1是（全局仅一个） | B3 基准币种设置（设后不可改） |
| rate_realtime | bigint | NOT NULL DEFAULT 0 | 实时汇率（×10^8 存储） | B4 三方API取均值 |
| rate_platform | bigint | NOT NULL DEFAULT 0 | 平台汇率（×10^8 存储） | B4 偏差超阈值时从实时同步 |
| rate_deposit | bigint | NOT NULL DEFAULT 0 | 入款汇率（×10^8）= 平台×(1+浮动%) | B4 入款汇率规则 |
| rate_withdrawal | bigint | NOT NULL DEFAULT 0 | 出款汇率（×10^8）= 平台×(1-浮动%) | B4 出款汇率规则 |

| deviation_threshold | int unsigned | NOT NULL DEFAULT 0 | 偏差阈值百分比（×100 存储，2位小数） | B4 偏差触发更新规则 |
| sort_order | int unsigned | DEFAULT 1 | 排序值（越小越靠前） | B3 列表排序 |
| status | tinyint unsigned | NOT NULL DEFAULT 1 | 1=启用 2=禁用 | B3 状态切换 |
| create_by | bigint unsigned | DEFAULT '0' | 创建人ID | 工程审计规范 |
| update_by | bigint unsigned | DEFAULT '0' | 更新人ID | 工程审计规范 |
| create_at | bigint unsigned | NOT NULL DEFAULT '0' | 创建时间（秒级时间戳） | 工程审计规范 |
| update_at | bigint unsigned | NOT NULL DEFAULT '0' | 更新时间（秒级时间戳） | 工程审计规范 |
| delete_at | bigint unsigned | NOT NULL DEFAULT '0' | 删除时间（0=未删除） | 工程软删除规范 |

#### CREATE TABLE SQL

```sql
DROP TABLE IF EXISTS `currency_config`;

(
    `id`                  bigint unsigned  NOT NULL AUTO_INCREMENT COMMENT '主键ID',

    `currency_name`       varchar(64)      NOT NULL COMMENT '币种名称',

    `decimal_separator`   varchar(4)                DEFAULT '' COMMENT '小数点符号',
    `decimal_digits`      tinyint unsigned NOT NULL DEFAULT '2' COMMENT '小数位数 法币2 加密8',
    `is_base`             tinyint unsigned NOT NULL DEFAULT '0' COMMENT '是否基准币种 0否 1是',
    `rate_realtime`       bigint           NOT NULL DEFAULT '0' COMMENT '实时汇率（×10^8）',
    `rate_platform`       bigint           NOT NULL DEFAULT '0' COMMENT '平台汇率（×10^8）',
    `rate_deposit`        bigint           NOT NULL DEFAULT '0' COMMENT '入款汇率（×10^8）',
    `rate_withdrawal`     bigint           NOT NULL DEFAULT '0' COMMENT '出款汇率（×10^8）',
    `float_rate_percent`  int unsigned     NOT NULL DEFAULT '0' COMMENT '汇率浮动百分比（×100）',
    `deviation_threshold` int unsigned     NOT NULL DEFAULT '0' COMMENT '偏差阈值百分比（×100）',
    `sort_order`          int unsigned              DEFAULT '1' COMMENT '排序值（越小越靠前）',
    `status`              tinyint unsigned NOT NULL DEFAULT '1' COMMENT '状态 1启用 2禁用',
    `create_by`           bigint unsigned           DEFAULT '0' COMMENT '创建人ID',
    `update_by`           bigint unsigned           DEFAULT '0' COMMENT '更新人ID',
    `create_at`           bigint unsigned  NOT NULL DEFAULT '0' COMMENT '创建时间（秒级时间戳）',
    `update_at`           bigint unsigned  NOT NULL DEFAULT '0' COMMENT '更新时间（秒级时间戳）',
    `delete_at`           bigint unsigned  NOT NULL DEFAULT '0' COMMENT '删除时间（秒级时间戳，0表示未删除）',
    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_currency_code` (`currency_code`, `delete_at`),
    KEY `idx_status` (`status`),
    KEY `idx_is_base` (`is_base`)
) ENGINE = InnoDB
  DEFAULT CHARSET = utf8mb4
  COLLATE = utf8mb4_bin COMMENT ='币种配置表';
```

#### 索引设计说明

| 索引 | 类型 | 用途 | 查询场景 |
|------|------|------|---------|
| uk_currency_code | 唯一 | 币种编码全局唯一 | CreateCurrency 防重、所有按 code 查询 |
| idx_status | 普通 | 按状态筛选 | C端只查启用币种、B端支持按状态筛选 |
| idx_is_base | 普通 | 快速定位基准币种 | GetBaseCurrency（虽然只有一条，但语义明确） |

#### 关键设计要点

1. **汇率×10^8 存 bigint**：避免浮点精度丢失。例 1 USDT = 25432.50 VND → rate_platform = 2543250000000
2. **float_rate_percent×100 存整数**：2位小数的百分比，如 1.50% 存 150
3. **is_base 全局唯一约束**：应用层保证只有一条 is_base=1 的记录，SetBaseCurrency 检查已设置则拒绝
4. **数据量极小**：4-6 条，全量缓存到 Redis，读操作不走数据库

```sql
    PRIMARY KEY (`id`),
    KEY `idx_currency_code` (`currency_code`),
    KEY `idx_create_at` (`create_at`)

```

#### 索引设计说明

| 索引 | 类型 | 用途 | 查询场景 |
|------|------|------|---------|
| idx_currency_code | 普通 | 按币种筛选 | C3 汇率日志页按币种筛选 |
| idx_create_at | 普通 | 按时间范围筛选 | C3 汇率日志页按"更新时间范围"筛选 |

#### 关键设计要点

1. **不设 delete_at**：日志只增不删，没有软删除需求，减少索引开销
2. **记录 before/after**：完整记录变更前后值，便于运营追溯变更幅度
3. **trigger_type 区分来源**：自动同步和手动修改的日志需要区分，审计时用
4. **数据量评估**：假设 4 个币种、每 10 分钟一次同步、阈值命中率 10%→每天约 60 条，每年约 2 万条，不需要分区

### 2.3 wallet —— 用户钱包表

#### 为什么必须

- 这是整个系统的**核心中的核心**——用户的余额存在这里
- 需求文档 B2 定义了 5 种子钱包体系（中心/奖励/主播/代理/场馆）
- 需求文档 A2 定义了币种隔离原则——用户的每个币种余额独立
- 所有资金操作（充值上账、兑换、提现冻结/扣款、人工加减款）的终点都是修改这张表

#### 核心设计原则

**一条记录 = 一个用户 + 一个币种 + 一种钱包类型的余额**

例：用户 10001 使用 VND 币种，会有最多 3 条 wallet 记录（中心/奖励/主播各一条，代理/场馆预留暂不创建）。

#### 完整字段定义

| 字段名 | 类型 | 约束 | 说明 | 需求依据 |
|--------|------|------|------|---------|
| id | bigint unsigned | PK, AUTO_INCREMENT | 主键 | 工程规范 |
| user_id | bigint unsigned | NOT NULL | 用户ID | 所有余额操作的主体 |
| currency_code | varchar(16) | NOT NULL | 币种编码 | A2 币种隔离原则 |
| wallet_type | tinyint unsigned | NOT NULL | 钱包类型 1中心 2奖励 3主播 4代理(预留) 5场馆(预留) | B2 五种子钱包 |
| balance | bigint | NOT NULL DEFAULT '0' | 可用余额（最小单位） | 核心资金字段 |
| frozen_amount | bigint | NOT NULL DEFAULT '0' | 冻结金额（最小单位） | A5 提现冻结机制 |
| fund_source | tinyint unsigned | NOT NULL DEFAULT '0' | 资金来源 0=无 1=法币 2=USDT 3=混合 | A5 "法币充→法币提"合规规则 |
| version | bigint unsigned | NOT NULL DEFAULT '0' | 乐观锁版本号 | 架构设计——并发控制 |
| status | tinyint unsigned | NOT NULL DEFAULT '1' | 1=正常 2=冻结(管理员冻结整个钱包) | 风控需要 |
| create_by | bigint unsigned | DEFAULT '0' | 创建人ID | 工程审计规范 |
| update_by | bigint unsigned | DEFAULT '0' | 更新人ID | 工程审计规范 |
| create_at | bigint unsigned | NOT NULL DEFAULT '0' | 创建时间（秒级时间戳） | 工程审计规范 |
| update_at | bigint unsigned | NOT NULL DEFAULT '0' | 更新时间（秒级时间戳） | 工程审计规范 |
| delete_at | bigint unsigned | NOT NULL DEFAULT '0' | 删除时间（0=未删除） | 工程软删除规范 |

#### CREATE TABLE SQL

```sql
DROP TABLE IF EXISTS `wallet`;
CREATE TABLE `wallet`
(
    `id`            bigint unsigned  NOT NULL AUTO_INCREMENT COMMENT '主键ID',
    `user_id`       bigint unsigned  NOT NULL COMMENT '用户ID',
    `currency_code` varchar(16)      NOT NULL COMMENT '币种编码',
    `wallet_type`   tinyint unsigned NOT NULL COMMENT '钱包类型 1中心 2奖励 3主播 4代理 5场馆',
    `balance`       bigint           NOT NULL DEFAULT '0' COMMENT '可用余额（最小单位）',
    `frozen_amount` bigint           NOT NULL DEFAULT '0' COMMENT '冻结金额（最小单位）',
    `fund_source`   tinyint unsigned NOT NULL DEFAULT '0' COMMENT '资金来源 0无 1法币 2USDT 3混合',
    `version`       bigint unsigned  NOT NULL DEFAULT '0' COMMENT '乐观锁版本号',
    `status`        tinyint unsigned NOT NULL DEFAULT '1' COMMENT '状态 1正常 2冻结',
    `create_by`     bigint unsigned           DEFAULT '0' COMMENT '创建人ID',
    `update_by`     bigint unsigned           DEFAULT '0' COMMENT '更新人ID',
    `create_at`     bigint unsigned  NOT NULL DEFAULT '0' COMMENT '创建时间（秒级时间戳）',
    `update_at`     bigint unsigned  NOT NULL DEFAULT '0' COMMENT '更新时间（秒级时间戳）',
    `delete_at`     bigint unsigned  NOT NULL DEFAULT '0' COMMENT '删除时间（秒级时间戳，0表示未删除）',
    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_user_curr_type` (`user_id`, `currency_code`, `wallet_type`, `delete_at`),
    KEY `idx_user_id` (`user_id`)
) ENGINE = InnoDB
  DEFAULT CHARSET = utf8mb4
  COLLATE = utf8mb4_bin COMMENT ='用户钱包表';
```

#### 索引设计说明

| 索引 | 类型 | 用途 | 查询场景 |
|------|------|------|---------|
| uk_user_curr_type | 唯一 | 保证同一用户同一币种同一钱包类型只有一条记录 | BalanceEngine 每次操作的定位基础 |
| idx_user_id | 普通 | 按用户查所有钱包 | W-R1 GetUserWalletBalance（查某用户所有子钱包余额） |

#### 关键设计要点

1. **balance + frozen_amount 分离**：可用余额和冻结金额独立记录。提现冻结时 balance 减少、frozen_amount 增加；出款成功 frozen_amount 减少。任何时刻 `balance + frozen_amount = 用户实际总资产`
2. **version 乐观锁**：每次余额变更 version+1，UPDATE 时 WHERE version=旧值，保证并发安全（架构三层防线之一）
3. **fund_source 反洗钱追踪**：记录资金来源类型，提现时校验"法币充→法币提"规则。首次法币充值设为 1，首次 USDT 充值设为 2，混合充值设为 3。**冗余字段但避免了提现时回查全部充值历史**
4. **wallet_type 预留**：枚举值 4/5 预留给代理/场馆钱包，首版不创建对应记录但表结构已兼容
5. **首次写入自动创建**：用户首次充值某币种时，BalanceEngine 自动创建对应的 wallet 记录（balance=0），然后执行加款

### 2.4 wallet_transaction —— 交易流水表

#### 为什么必须

- 需求文档 A6 要求 C 端展示四类记录（充值/提现/兑换/奖励）
- 需求文档 A7 要求"对账一致性"——流水总和 = 账户余额
- 架构设计核心约束：每次余额变更必须同事务写一条流水
- 这是**对账生命线**——没有流水，余额出错时无法追溯

#### 核心设计原则

**每一笔余额变动对应一条流水**。flow：BalanceEngine.Add/Deduct/Freeze/Unfreeze/ConfirmDeduct → 同事务 INSERT wallet_transaction。

#### 完整字段定义

| 字段名 | 类型 | 约束 | 说明 | 需求依据 |
|--------|------|------|------|---------|

#### 关键设计要点

1. **balance_before + balance_after**：金融系统标配。对账校验：`balance_after == balance_before + amount`，最后一条流水的 balance_after == wallet.balance
2. **frozen_before + frozen_after**：冻结/解冻/确认扣款操作也需要记录冻结额的前后变化，对账完整性
3. **10 种交易类型**：覆盖所有可能的余额变动场景。兑换拆为兑出(扣法币)+兑入(加BSB)两条流水，因为跨币种跨钱包，每条流水只对应一个钱包的一次余额变动
4. **uk_order_wallet_tx 三字段唯一键**：为什么不只用 order_no？因为同一个兑换订单会产生多条流水（法币中心钱包兑出 + BSB中心钱包兑入 + BSB奖励钱包奖金），需要 wallet_type + tx_type 区分
5. **数据量评估**：核心增长表。假设日活 10 万用户，每人日均 3 笔操作→日增约 30 万条。需要关注分页查询性能，idx_user_curr_type 索引覆盖主要查询

### 2.8 exchange_order —— 兑换订单表

#### 为什么必须

- 需求文档 A4 定义了兑换模块（法币→BSB，单向兑换，赠送机制）
- 产品原型 D7 展示了兑换页面（输入金额、预览、确认）
- 兑换涉及跨币种操作（扣法币、加BSB），需要记录源/目标币种、汇率快照、赠送明细

#### 为什么不只用 wallet_transaction

wallet_transaction 是**单钱包单币种**的余额变动记录。兑换是跨币种操作，一笔兑换会产生 2-3 条 wallet_transaction（法币兑出 + BSB中心兑入 + BSB奖励奖金）。exchange_order 记录的是这一次兑换操作的**业务全貌**——源币种、目标币种、汇率、赠送比例等——这些信息分散在多条流水里是不完整的。

### 4.1 wallet_daily_snapshot —— 钱包日快照表

#### 为什么"可选"

- 需求文档未明确要求此表，但金融系统对账是行业标配
- 架构设计中已规划了"每日对账任务"，如果不建这张表，可以通过遍历流水逆推对账
- 建此表可以**大幅提升对账效率**——直接用快照与当前余额比对，无需扫描全部流水

每天凌晨定时任务对每个有余额的钱包拍快照，记录当日余额。对账时：快照余额 + 当天流水变动量 = 当前实际余额。不等则告警。

## 五、财务模块参考表结构（别人负责，了解联动）

### 5.1 财务模块可能需要的表

| 表名（推测） | 用途 | 与钱包的关联 |
|-------------|------|-------------|
| payment_channel | 代收/代付通道配置 | 无直接关联，财务内部使用 |
| payment_method | 充值/提现方式配置（关联通道） | 钱包 W-3/W-10 通过 RPC 读取此表数据 |

| finance_withdrawal_order | 财务侧提现订单（含三级审核记录、通道出款信息） | 通过 order_no 关联钱包的 withdrawal_order |

| channel_daily_stat | 通道日统计 | 无直接关联 |

### 5.2 钱包与财务的数据关联模式

```
钱包侧数据                   财务侧数据
─────────                   ─────────
deposit_order ←──order_no──→ finance_deposit_order
  钱包存：金额、状态、奖金      财务存：通道、回调详情、审核
  钱包管：余额变更              财务管：通道分配、回调处理

withdrawal_order ←─order_no─→ finance_withdrawal_order
  钱包存：金额、冻结状态         财务存：三级审核、通道出款
  钱包管：冻结/解冻/扣款         财务管：审核流程、出款调度

                               manual_adjustment
                                 ──order_no──→ wallet_transaction
                                 财务创建订单     钱包执行余额变更
```

**核心原则**：钱包和财务各存各的订单数据，通过 order_no 关联。钱包不需要知道通道细节，财务不需要知道余额细节。

### 7.2 关联关系汇总

| 关系 | 关联方式 | 说明 |
|------|---------|------|
| wallet ↔ wallet_transaction | user_id + currency_code + wallet_type | 一个钱包有多条流水 |
| deposit_order → wallet_transaction | order_no | 充值成功后产生一条充值流水 |
| withdrawal_order → wallet_transaction | order_no | 提现产生冻结/解冻/扣款流水 |
| exchange_order → wallet_transaction | order_no | 兑换产生兑出+兑入+奖金流水 |
| withdrawal_order → withdrawal_account | account_id | 提现订单关联使用的账户 |
| deposit_order → wager_requirement | order_no = source_order_no | 充值奖金产生稽核要求 |
| exchange_order → wager_requirement | order_no = source_order_no | 兑换赠送产生稽核要求 |
| currency_config → rate_change_log | currency_code | 汇率变更产生日志 |

**注意：所有关联都是逻辑外键，不建物理外键约束**（TiDB/MySQL 分布式场景下物理外键影响性能和灵活性）。

## 八、各表数据量评估

| 表名 | 首年预估数据量 | 增长模式 | 需要关注的性能点 |
|------|-------------|---------|-----------------|
| currency_config | 4-6 条 | 几乎不增长 | 全量缓存，无性能问题 |
| rate_change_log | ~2 万条 | 线性缓慢增长 | 无性能问题 |
| wallet | ~50 万条（假设10万用户×5币种） | 线性增长 | uk_user_curr_type 索引覆盖 |
| wallet_transaction | ~1000 万条 | **核心增长表** | 注意分页查询性能，走 idx_user_curr_type |
| deposit_order | ~100 万条 | 中等增长 | uk_order_no 覆盖主查询 |
| withdrawal_order | ~50 万条 | 中等增长 | uk_order_no 覆盖主查询 |
| withdrawal_account | ~30 万条 | 用户级增长 | 无性能问题 |
| exchange_order | ~50 万条 | 中等增长 | 无性能问题 |
| wager_requirement | ~50 万条 | 跟随充值/兑换增长 | idx_user_status 覆盖查询 |

**wallet_transaction 是唯一需要中长期关注的表**。如果日增超过 100 万条，需要考虑按 user_id 分区或按时间归档。

## 十、设计决策说明

### 10.1 为什么每张表都有依据

| 表 | 需求来源 | 产品原型 | 不建会怎样 |
|----|---------|---------|-----------|
| currency_config | B3（18字段）+B4（4种汇率） | C1/C2 币种列表+编辑弹窗 | 没有币种数据，系统无法启动 |
| rate_change_log | B4 汇率日志页面 | C3 汇率日志Tab | 运营看不到汇率变更历史 |
| wallet | B2（5子钱包）+A2（币种隔离） | D2 钱包余额展示 | 没有地方存余额 |
| wallet_transaction | A6（4类记录）+A7（对账一致性） | D10 记录页 | 没有流水无法对账、用户看不到记录 |
| deposit_order | A3（充值模块）+重复拦截 | D3/D4/D5 充值页面 | 充值流程断裂，无法跟踪状态 |
| withdrawal_order | A5（提现模块）+冻结状态机 | D8/D9 提现页面 | 提现流程断裂，资金冻结无法管理 |
| withdrawal_account | A5（账户管理）+KYC姓名绑定 | D9 提现账户表单 | 用户每次提现都要重填信息 |
| exchange_order | A4（兑换模块）+赠送机制 | D7 兑换BSB页 | 兑换无法记录详情，对账缺失 |
| wager_requirement | A3/A4 稽核倍数 | 需求文字描述 | 无法追踪奖金流水达标状态 |

### 10.2 为什么不需要更多表

以下是"看起来可能需要但实际不需要独立建表"的分析：

| 看起来需要的 | 为什么不建 | 替代方案 |
|-------------|-----------|---------|
| exchange_rate（独立汇率表） | 币种和汇率 1:1 关系，数据量极小（4-6条） | 合并在 currency_config 中（AD-10） |
| base_currency（基准币种表） | 只有一条数据 | currency_config 用 is_base 标记 |
| wallet_audit_log（钱包操作日志） | 工程已有 ser-blog 审计日志 | 所有写操作 oneway RPC → ser-blog |
| bonus_activity（奖金活动配置） | 需求中奖金活动的数据源待确认，可能由独立活动服务管理 | 首版 bonus_activity_id 引用外部数据，不自建表 |
| usdt_address（USDT收款地址） | 收款地址管理方式待确认（固定配置 or 链上服务动态生成） | 首版可用配置文件/ETCD管理 |
| wallet_freeze_log（冻结明细日志） | wallet_transaction 已包含冻结/解冻/确认扣款类型的流水 | tx_type=4/5/6 + frozen_before/after 已覆盖 |

### 10.3 关键技术决策汇总

| 决策 | 选择 | 理由 | 替代方案 | 否决理由 |
|------|------|------|---------|---------|
| 金额存储 | bigint 最小单位 | 无精度丢失、int64 运算快、业界标准 | decimal | Go decimal 库性能差 |
| 汇率存储 | bigint ×10^8 | 统一整数运算、精度到 8 位小数 | decimal/varchar | 同上 |
| 币种+汇率合表 | currency_config 一张表 | 1:1 关系、数据量极小、查询简单 | 拆 currency + rate 两张表 | 增加 JOIN 无收益 |
| 唯一键含 delete_at | 所有 UK 包含 delete_at | 软删除后允许重建同编码记录 | 不含 delete_at | 软删除后无法重建 |
| 流水 before/after | balance_before + balance_after + frozen_before + frozen_after | 对账必需、审计追溯 | 只记 amount | 无法独立验证余额正确性 |
| 兑换独立订单表 | exchange_order 独立于 wallet_transaction | 业务信息跨币种不适合放流水表 | 只用 wallet_transaction | 源/目标币种、汇率快照等信息无处记录 |
| 时间戳精度 | 秒级（bigint unsigned） | 遵循 ser-item 规范 | 毫秒级（ser-app 模式） | 钱包不需要毫秒精度 |
