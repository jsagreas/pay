## 一、表结构总览与优先级分类

### 1.1 三级分类标准

| 级别 | 定义 | 判断标准 |
|------|------|---------|
| **必须（Must）** | 缺少则系统无法启动或核心功能完全不可用 | 被E1~E6核心资金操作、币种基础数据直接依赖 |
| **应该（Should）** | 缺少则某条完整业务链路跑不通 | 被充值/提现/兑换等端到端业务流程依赖 |
| **可选（Nice-to-Have）** | 缺少不影响核心功能，但影响运营管理或审计追溯 | 日志类、统计类、辅助管理类 |

### 1.2 12张表分类总览

| 序号 | 表名 | 所属领域 | 优先级 | 用途 | 来源依据 |
|------|------|---------|--------|------|---------|
| 1 | `currency_config` | 币种配置 | **必须** | 存储平台支持的所有币种及其配置参数 | 需求理解2.5.2节 列表20个字段 |
| 2 | `base_currency` | 币种配置 | **必须** | 存储基准币种(USDT)和BSB锚定关系 | 需求理解2.3节 基准币种 |
| 3 | `currency_rate` | 币种配置 | **必须** | 存储各币种实时汇率和平台汇率 | 需求理解2.4节 汇率体系 |
| 4 | `wallet_balance` | 钱包核心 | **必须** | 存储用户各子钱包的可用余额和冻结余额 | 需求理解2.1.2节 五种子钱包 |
| 5 | `wallet_transaction` | 钱包核心 | **必须** | 记录每一笔资金变动的不可变流水 | 需求理解3.6节 对账一致性 |

### 1.3 各优先级数量分布

```
必须（Must）    ：5张表 — 缺任何一张系统无法运行
```

### 3.1 currency_config — 币种配置表

**用途**：存储平台支持的所有币种及其配置参数（图标、精度、汇率参数、状态）。

**为什么是「必须」**：
- 钱包几乎每个接口都要查币种配置（W1余额格式化需要精度/千分位符号，W2需要币种列表，W3切换需要校验启用状态）
- R1~R7 所有RPC接口的数据来源
- 币种数据不存在 → 整个系统无法知道支持哪些币种

**来源依据**：需求理解c-3.md 2.5.2节 列表20个字段 + 2.5.3节 编辑弹窗字段

| 字段名 | 类型 | 约束 | 说明 | 产品字段映射 |
|--------|------|------|------|-------------|
| `id` | `BIGINT` | `PK AUTO_INCREMENT` | 主键 | — |
| `currency_code` | `VARCHAR(16)` | `NOT NULL UNIQUE` | 币种代码（如BTC、USDT、VND） | 编号2-币种代码 |
| `currency_name` | `VARCHAR(64)` | `NOT NULL DEFAULT ''` | 币种名称 | 编号3-币种名称 |

| `country` | `VARCHAR(64)` | `NOT NULL DEFAULT ''` | 币种国别（加密货币默认"全球"） | 编号5-币种国别 |
| `icon_url` | `VARCHAR(512)` | `NOT NULL DEFAULT ''` | 币种图标URL（SVG/PNG≤5KB） | 编号6-币种图标 |

### 3.2 base_currency — 基准币种表

**用途**：存储平台基准币种(USDT)和BSB锚定关系，全表最多一条记录。

**为什么是「必须」**：
- 基准币种是汇率计算的参照基准，所有币种金额换算为基准币种统计
- BSB价值(10BSB=1USDT)锚定在此表，兑换/统计都依赖
- 不存在基准币种 → 汇率体系无法运转

**来源依据**：需求理解c-3.md 2.3节 基准币种

| 字段名 | 类型 | 约束 | 说明 | 产品字段映射 |
|--------|------|------|------|-------------|
| `id` | `BIGINT` | `PK AUTO_INCREMENT` | 主键 | — |
| `currency_code` | `VARCHAR(16)` | `NOT NULL` | 基准币种代码（如USDT） | 基准币种下拉选择 |
| `bsb_anchor_rate` | `DECIMAL(24,8)` | `NOT NULL DEFAULT 10.00000000` | BSB锚定比例：10BSB=1USDT | 代码写入，不走配置 |
| `set_by` | `BIGINT` | `NOT NULL DEFAULT 0` | 设置人ID | 操作人 |
| `set_at` | `BIGINT` | `NOT NULL DEFAULT 0` | 设置时间（毫秒时间戳） | 操作时间 |
| `create_by` | `BIGINT` | `NOT NULL DEFAULT 0` | 创建者 | — |
| `create_at` | `BIGINT` | `NOT NULL DEFAULT 0` | 创建时间 | — |
| `update_by` | `BIGINT` | `NOT NULL DEFAULT 0` | 修改者 | — |
| `update_at` | `BIGINT` | `NOT NULL DEFAULT 0` | 修改时间 | — |
| `delete_at` | `BIGINT` | `NOT NULL DEFAULT 0` | 软删除 | — |

**索引**：

| 索引名 | 字段 | 类型 | 用途 |
|--------|------|------|------|
| `idx_delete_at` | `(delete_at)` | NORMAL | 查询唯一有效记录 |

**业务规则**：
- 全表最多一条有效记录（`delete_at = 0`）
- 一次性写入，配置后不可修改（需求理解2.3节）
- BSB价值和锚定汇率由代码写入，不做配置（需求理解2.3节）
- 设置时需二级确认弹窗（需求理解2.3节）

### 3.3 currency_rate — 币种汇率表

**用途**：存储各币种的实时汇率和平台汇率，定时任务和手动刷新时更新。

**为什么是「必须」**：
- 所有涉及跨币种的操作（充值/兑换/提现/统计）都依赖汇率数据
- R1(GetPlatformRate)和R5(GetTransactionRate)的数据来源 — 调用频率最高的RPC接口
- 汇率不存在 → BSB充值无法计算金额、兑换无法换算、跨币种提现无法计算

**来源依据**：需求理解c-3.md 2.4节 汇率体系 + 2.5.2节 编号10~15

| 字段名 | 类型 | 约束 | 说明 | 产品字段映射 |
|--------|------|------|------|-------------|
| `id` | `BIGINT` | `PK AUTO_INCREMENT` | 主键 | — |
| `currency_code` | `VARCHAR(16)` | `NOT NULL` | 币种代码 | 关联currency_config |
| `real_time_rate` | `DECIMAL(24,8)` | `NOT NULL DEFAULT 0.00000000` | 实时汇率（3家API均值） | 编号10-实时汇率 |
| `platform_rate` | `DECIMAL(24,8)` | `NOT NULL DEFAULT 0.00000000` | 平台汇率（超阈值时更新） | 编号12-平台汇率 |
| `deposit_rate` | `DECIMAL(24,8)` | `NOT NULL DEFAULT 0.00000000` | 入款汇率=平台汇率×(1+浮动%) | 编号13-入款汇率 |
| `withdraw_rate` | `DECIMAL(24,8)` | `NOT NULL DEFAULT 0.00000000` | 出款汇率=实时汇率×(1-浮动%) | 编号14-出款汇率 |
| `deviation_percent` | `DECIMAL(10,4)` | `NOT NULL DEFAULT 0.0000` | 当前偏差值百分比 | 编号15-偏差值 |
| `last_refresh_at` | `BIGINT` | `NOT NULL DEFAULT 0` | 最后刷新时间（毫秒时间戳） | 实时汇率刷新时间 |
| `create_by` | `BIGINT` | `NOT NULL DEFAULT 0` | 创建者 | — |
| `create_at` | `BIGINT` | `NOT NULL DEFAULT 0` | 创建时间 | — |
| `update_by` | `BIGINT` | `NOT NULL DEFAULT 0` | 修改者 | — |
| `update_at` | `BIGINT` | `NOT NULL DEFAULT 0` | 修改时间 | — |
| `delete_at` | `BIGINT` | `NOT NULL DEFAULT 0` | 软删除 | — |

**索引**：

| 索引名 | 字段 | 类型 | 用途 |
|--------|------|------|------|
| `uk_currency_code_del` | `(currency_code, delete_at)` | UNIQUE | 每个币种只有一条有效汇率记录 |

**业务规则**：
- 每个已启用币种对应一条汇率记录
- 定时任务(T1)和手动刷新(B7)共用同一套核心逻辑更新（流程思路2.3节）
- 入款汇率公式：平台汇率 × (1 + 汇率浮动%) — 需求理解2.4.2节
- 出款汇率公式：实时汇率 × (1 - 汇率浮动%) — 需求理解2.4.2节
- 偏差值 = |实时汇率 - 平台汇率| / 实时汇率 × 100% — 需求理解2.4.2节
- 入款或出款偏差值任一 ≥ 阈值即更新该币种平台汇率 — 需求理解2.4.2节红字强调

### 3.4 wallet_balance — 钱包余额表

**用途**：存储每个用户在每个币种下每个子钱包的可用余额和冻结余额，是整个系统最核心的表。

**为什么是「必须」**：
- E1~E6 所有核心资金操作的目标表
- 钱包系统的核心数据载体 — 用户的钱在这里
- 余额表不存在 → 查余额、加款、减款、冻结、解冻全部不可用

**来源依据**：需求理解c-3.md 2.1.2节 五种子钱包 + 架构设计c-3.md 六节资金安全

| 字段名 | 类型 | 约束 | 说明 | 产品字段映射 |
|--------|------|------|------|-------------|
| `id` | `BIGINT` | `PK AUTO_INCREMENT` | 主键 | — |
| `user_id` | `BIGINT` | `NOT NULL` | 用户ID | C端Token中的UserID |
| `currency_code` | `VARCHAR(16)` | `NOT NULL` | 币种代码 | 当前操作币种 |
| `wallet_type` | `TINYINT` | `NOT NULL` | 子钱包类型：1中心 2奖励 3主播 4代理 | 需求理解2.1.2节 |
| `available_balance` | `DECIMAL(24,8)` | `NOT NULL DEFAULT 0.00000000` | 可用余额 | C端展示的可操作余额 |
| `frozen_balance` | `DECIMAL(24,8)` | `NOT NULL DEFAULT 0.00000000` | 冻结余额（提现审核中被冻结的金额） | 提现流程冻结 |
| `create_by` | `BIGINT` | `NOT NULL DEFAULT 0` | 创建者 | — |
| `create_at` | `BIGINT` | `NOT NULL DEFAULT 0` | 创建时间 | — |
| `update_by` | `BIGINT` | `NOT NULL DEFAULT 0` | 修改者 | — |
| `update_at` | `BIGINT` | `NOT NULL DEFAULT 0` | 修改时间 | — |
| `delete_at` | `BIGINT` | `NOT NULL DEFAULT 0` | 软删除 | — |

**索引**：

| 索引名 | 字段 | 类型 | 用途 |
|--------|------|------|------|
| `uk_user_currency_wallet` | `(user_id, currency_code, wallet_type, delete_at)` | UNIQUE | 同一用户+同一币种+同一子钱包只有一条记录 |
| `idx_user_currency` | `(user_id, currency_code, delete_at)` | NORMAL | 查询用户某币种下所有子钱包余额(W1) |

**业务规则**：
- 唯一约束 `(user_id, currency_code, wallet_type)` 保证一个用户一个币种一个子钱包只有一条余额记录
- 资金钱包余额 = 中心钱包余额 + 奖励钱包余额（需求理解2.1.3节）
- 消费优先级：先扣中心钱包，再扣奖励钱包（需求理解2.1.3节）
- 余额不可为负：`available_balance >= 0`，`frozen_balance >= 0`
- E4(冻结)：`available_balance -= amount`, `frozen_balance += amount`（同一行原子操作）
- E5(解冻)：`frozen_balance -= amount`, `available_balance += amount`（同一行原子操作）
- E6(确认扣除)：`frozen_balance -= amount`（资金真正离开用户钱包）
- **并发控制**：使用数据库行级锁（`UPDATE ... SET available_balance = available_balance + ? WHERE ...`）— 架构设计ADR-003
- **不缓存余额**：直接读DB — 架构设计ADR-005

**wallet_type 枚举值**：

## 六、三级分类决策依据

### 6.1 为什么这样分类

| 表 | 级别 | 决策依据 |
|----|------|---------|
| `currency_config` | **必须** | 没有币种配置 → R1~R7 全部不可用 → 钱包所有接口无法运行 |
| `base_currency` | **必须** | 没有基准币种 → 汇率计算无参照 → 所有涉及换算的操作失败 |
| `currency_rate` | **必须** | 没有汇率 → 充值/兑换/提现中的汇率锁定(R5)无数据 → 订单无法创建 |
| `wallet_balance` | **必须** | 没有余额表 → E1~E6全部无法执行 → 系统核心功能不存在 |
| `wallet_transaction` | **必须** | 没有流水表 → 幂等机制无法实现 → 资金安全无法保证 |
| `recharge_order` | **应该** | 充值订单不存在 → W4~W11跑不通，但E1~E6/兑换/提现不受影响 |
| `withdraw_order` | **应该** | 提现订单不存在 → W14~W22跑不通，但E1~E6/充值/兑换不受影响 |
| `withdraw_account` | **应该** | 提现账户不存在 → 用户无法保存/复用账户信息，但可临时硬编码 |
| `exchange_record` | **应该** | 兑换流水不存在 → W12~W13跑不通、W25无数据，但其他功能不受影响 |
| `user_currency_preference` | **应该** | 偏好不存在 → 用户币种选择无法持久化，但可用默认值降级 |
| `rate_log` | **可选** | 日志不存在 → B8/B9不可用，但汇率更新逻辑正常工作 |
| `operation_log` | **可选** | 日志不存在 → B10不可用，但配置操作正常执行 |

### 6.2 开发顺序建议

```
第1批（Phase 0）：必须的5张表 — 系统启动的最低要求
  currency_config → base_currency → currency_rate → wallet_balance → wallet_transaction

第2批（Phase 1~2）：应该的5张表 — 业务链路完整化
  user_currency_preference → exchange_record → recharge_order → withdraw_account → withdraw_order

第3批（Phase 3）：可选的2张表 — 运营管理功能
  rate_log → operation_log
```

### 8.2 关键索引设计说明

**最重要的两个索引**：

1. **`wallet_balance.uk_user_currency_wallet`** — 这是余额行级锁的定位索引。`UPDATE wallet_balance SET available_balance = available_balance + ? WHERE user_id = ? AND currency_code = ? AND wallet_type = ? AND delete_at = 0` 必须命中此索引实现行级锁，否则会升级为表级锁导致并发问题。

2. **`wallet_transaction.uk_order_operation`** — 这是幂等机制的保证。同一个订单号+同一种操作类型不可能写入两条流水，数据库层面的最终兜底。

### 11.2 两个最关键的设计决策

1. **`wallet_transaction.uk_order_operation`**（幂等唯一键）
   `(order_no, operation_type, delete_at)` UNIQUE — 数据库层面保证同一订单同一操作不会重复执行。这是资金安全的最后一道防线，无论业务代码如何Bug，数据库都不会允许重复入账。

2. **`wallet_balance.uk_user_currency_wallet`**（余额行级锁）
   `(user_id, currency_code, wallet_type, delete_at)` UNIQUE — UPDATE时通过此索引精确定位到单行，InnoDB自动加行级锁，不会锁表。配合 `UPDATE ... SET available_balance = available_balance + ? WHERE ...` 原子操作，保证并发安全。
