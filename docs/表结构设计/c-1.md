## 一、设计方法论

### 1.1 分析维度说明

本文档从三个维度评估每张表的必要性：

| 维度 | 定义 | 判断标准 |
|------|------|---------|
| **必须（Must）** | 缺少这张表，核心业务流程无法运转 | 需求文档中明确描述的实体，且至少有一个核心接口依赖它做读写 |
| **应该（Should）** | 缺少这张表，业务能跑通但缺少关键支撑能力（审计、日志、配置） | 需求文档中有明确页面或功能依赖，但不直接阻塞核心资金流转 |
| **可选（Optional）** | 增强型表，当前阶段可以不建，未来按需引入 | 需求文档中灰字标注、或属于优化/预留/统计增强需求 |

### 1.2 表的设计来源链路

每张表的设计都可以追溯到一条完整的来源链路：

```
产品原型页面上的实体/操作
    ↓ 需求理解文档抽象
业务概念（用户有钱包、用户发起充值、币种有汇率...）
    ↓ 链路关系文档拆解
接口读写哪些数据（C4 创建充值订单需要写 deposit_order）
    ↓ 架构设计文档约束
安全/性能/一致性要求（余额变更必须记流水、RPC 调用必须幂等）
    ↓ 工程认知文档规范
字段类型/命名/索引/审计字段的工程标准
    ↓
最终表结构
```

### 1.3 工程通用字段模板

根据工程认知文档 11.1 节，**每张表必须包含**以下通用审计字段：

```sql
-- 通用审计字段（以下 DDL 中用 [通用审计字段] 标记代替）
id              BIGINT        NOT NULL AUTO_INCREMENT  COMMENT '主键ID',
create_by       BIGINT        NOT NULL DEFAULT 0       COMMENT '创建人ID',
create_by_name  VARCHAR(50)   NOT NULL DEFAULT ''      COMMENT '创建人名称',
create_at       BIGINT        NOT NULL DEFAULT 0       COMMENT '创建时间（毫秒时间戳）',
update_by       BIGINT        NOT NULL DEFAULT 0       COMMENT '更新人ID',
update_by_name  VARCHAR(50)   NOT NULL DEFAULT ''      COMMENT '更新人名称',
update_at       BIGINT        NOT NULL DEFAULT 0       COMMENT '更新时间（毫秒时间戳）',
delete_at       BIGINT        NOT NULL DEFAULT 0       COMMENT '软删除（0=未删除，非0=删除时间戳）',
PRIMARY KEY (id)
```

## 二、表结构总览——必须 / 应该 / 可选

### 2.1 全景分类

```
┌─── 必须（Must）— 缺一不可，核心资金流转依赖 ──────────────────────────┐
│                                                                      │
│  ① wallet              用户钱包余额表      ← 所有资金操作的核心       │
│  ② transaction_log     余额变更流水表      ← 安全审计+幂等去重       │
│  ③ deposit_order       充值订单表          ← 充值业务的载体          │
│  ④ withdrawal_order    提现订单表          ← 提现业务的载体          │
│  ⑤ withdrawal_account  提现账户信息表      ← 提现流程强制依赖        │
│  ⑥ currency_config     币种配置表          ← 所有币种/汇率的数据源   │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘

┌─── 应该（Should）— 业务支撑，需求明确要求 ─────────────────────────────┐
│                                                                      │
│  ⑦ exchange_record     兑换记录表          ← C端兑换记录页展示        │
│  ⑧ reward_record       奖励记录表          ← C端奖励记录页展示        │
│  ⑨ rate_change_log     汇率变更日志表      ← B端汇率日志页展示        │
│  ⑩ system_config       系统配置表          ← 基准币种等全局配置        │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘

┌─── 可选（Optional）— 增强/预留，当前可不建 ────────────────────────────┐
│                                                                      │
│  ⑪ deposit_address     USDT收款地址池      ← USDT充值地址管理增强     │
│  ⑫ balance_snapshot    余额快照表          ← 定时对账增强             │
│  ⑬ exchange_bonus_config  兑换赠送配置表   ← 赠送比例配置化增强       │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

### 2.2 分类依据详解

#### 为什么是"必须"

| 表 | 为什么必须 | 缺少会怎样 |
|----|----------|-----------|
| wallet | 所有余额查询、充值入账、提现冻结/扣款、兑换、人工加减款都操作这张表 | 系统无法记录任何用户资金状态 |
| transaction_log | 每笔余额变更必须记录流水（需求理解 3.5"对账一致性"）；同时承担幂等去重（架构设计 5.2） | 无法审计追溯任何资金操作，RPC重试会导致重复入账 |
| deposit_order | C端充值记录展示（C12/C17）、充值回调入账更新状态（R2）、重复充值检查（C21）、超时关闭 | 充值流程无法运转 |
| withdrawal_order | C端提现记录展示（C13/C18）、提现审核状态流转（R4/R5）、今日已提现限额查询（C19） | 提现流程无法运转 |
| withdrawal_account | 提现流程强制要求（C7创建订单时必须关联账户信息）、非首次提现自动带出（C9） | 用户无法提现 |
| currency_config | 所有涉及汇率的接口（C3/C4/C6/C7/C10/C11）、B端币种管理页面（B1/B2）、汇率定时更新 | 系统不知道支持哪些币种、不知道汇率是多少 |

#### 为什么是"应该"

| 表 | 为什么是应该而非必须 | 什么场景需要 |
|----|-------------------|------------|
| exchange_record | 兑换操作本身只需要修改 wallet + 写 transaction_log 就能完成。但 C 端"兑换"标签页（需求理解 3.4）明确要求展示兑换记录（金额+赠送+到账+时间），transaction_log 的字段不足以满足展示需求 | C端兑换记录页 |
| reward_record | 奖励入账只需要修改 wallet + 写 transaction_log。但 C 端"奖励"标签页（需求理解 3.4）要求展示奖励类别名称、来源等信息，transaction_log 不够用 | C端奖励记录页 |
| rate_change_log | 汇率更新本身只需要修改 currency_config。但 B 端明确有"汇率日志"Tab 页（需求理解 4.3），要求展示 8 个字段的历史变更记录 | B端汇率日志页 |
| system_config | 基准币种"只可设置一次不可更改"（需求理解 4.1）需要持久化存储。如果不建表，可以用 ETCD 存储，但 ETCD 不适合存业务配置 | 基准币种设置 |

#### 为什么是"可选"

| 表 | 为什么可选 | 什么时候需要 |
|----|----------|------------|
| deposit_address | USDT充值收款地址目前可能是平台统一地址（需求中没有明确说需要地址池）。如果是统一地址则代码配置即可，不需要表 | 未来需要为每个用户/每笔订单分配独立收款地址时 |
| balance_snapshot | 定时对账（transaction_log正向累计 vs wallet当前余额）是"可选后期"任务（实现思路 10.1）。可以直接在对账任务中实时计算，不一定需要落表 | 对账任务发现不一致时需要历史快照排查 |
| exchange_bonus_config | 兑换赠送比例（如10%）的配置来源待评审确认（需求理解 8.5）。可能由活动模块管理，也可能写死代码，也可能独立配置表 | 评审确认赠送比例由钱包模块自管配置时 |

## 四、必须（Must）— 6 张核心表

### 4.1 表 ① wallet — 用户钱包余额表

#### 为什么必须

这是整个钱包系统的核心中的核心。**所有涉及用户资金的操作最终都归结为对这张表的读写**。

被调用的接口（依据：链路关系 5.1 内部共享方法梳理）：

| 操作类型 | 接口 | 说明 |
|---------|------|------|
| **读取** | C1 币种列表余额、C2 查询余额、C7 提现校验余额、C11 兑换校验余额、C14 钱包概览、R1 GetUserBalance、R7 减款前校验余额 | 7 个接口读取 |
| **写入** | C7 提现冻结、C11 兑换扣减+入账(×3)、R2 充值入账、R4 解冻退回、R5 确认扣款、R6 人工加款、R7 人工减款、R8 补单入账 | 8 个接口写入 |

如果没有这张表，用户没有余额概念，所有充值、提现、兑换流程无从谈起。

#### 表做什么

记录每个用户、每个币种、每种钱包类型的可用余额和冻结余额。

一句话概括：**用户有多少钱、多少钱被冻结了**。

#### DDL

```sql
CREATE TABLE `wallet` (
  `id`                  BIGINT        NOT NULL AUTO_INCREMENT  COMMENT '主键ID',
  `user_id`             BIGINT        NOT NULL                 COMMENT '用户ID',
  `currency_code`       VARCHAR(10)   NOT NULL                 COMMENT '币种代码（VND/IDR/THB/BSB/USDT）',
  `wallet_type`         INT           NOT NULL                 COMMENT '钱包类型 1中心 2奖励 3主播 4代理 5场馆',
  `available_balance`   DECIMAL(20,8) NOT NULL DEFAULT 0       COMMENT '可用余额',
  `frozen_balance`      DECIMAL(20,8) NOT NULL DEFAULT 0       COMMENT '冻结余额（提现待审核时冻结）',
  -- [通用审计字段]
  `create_by`           BIGINT        NOT NULL DEFAULT 0       COMMENT '创建人ID',
  `create_by_name`      VARCHAR(50)   NOT NULL DEFAULT ''      COMMENT '创建人名称',
  `create_at`           BIGINT        NOT NULL DEFAULT 0       COMMENT '创建时间（毫秒时间戳）',
  `update_by`           BIGINT        NOT NULL DEFAULT 0       COMMENT '更新人ID',
  `update_by_name`      VARCHAR(50)   NOT NULL DEFAULT ''      COMMENT '更新人名称',
  `update_at`           BIGINT        NOT NULL DEFAULT 0       COMMENT '更新时间（毫秒时间戳）',
  `delete_at`           BIGINT        NOT NULL DEFAULT 0       COMMENT '软删除（0=未删除）',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_user_currency_type` (`user_id`, `currency_code`, `wallet_type`, `delete_at`),
  KEY `idx_user_id` (`user_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户钱包余额表';
```

#### 字段逐一说明

| 字段 | 类型 | 必填 | 说明 | 需求来源 |
|------|------|------|------|---------|
| user_id | BIGINT | 是 | 钱包所属用户 | 需求理解 1.3：用户角色（普通用户/主播） |
| currency_code | VARCHAR(10) | 是 | 币种代码如 "VND"/"BSB" | 需求理解 2.1：多币种隔离，每个币种独立世界 |
| wallet_type | INT | 是 | 钱包类型枚举值 | 需求理解 2.2：5种钱包（中心/奖励/主播/代理/场馆） |
| available_balance | DECIMAL(20,8) | 是 | 可用余额，用户可以直接使用的钱 | 需求理解 2.2："C端展示的余额"、消费/提现从此扣除 |
| frozen_balance | DECIMAL(20,8) | 是 | 冻结余额，提现审核中被锁定的钱 | 需求理解 3.2：提现审核流程中冻结金额 |

**为什么 available_balance 和 frozen_balance 要分两个字段**：

需求理解 3.2 描述的提现流程是"用户创建订单→冻结余额→审核→通过扣款/驳回退回"。这种二阶段机制要求余额有两个状态：
- 冻结时：available -= N, frozen += N（钱还在用户名下，但不能再用）
- 通过时：frozen -= N（钱真正出去）
- 驳回时：frozen -= N, available += N（钱退回可用）

如果只有一个 balance 字段，提现创建时直接扣除，驳回时再加回——中间如果用户看余额会以为钱没了，与产品体验不符。

**核心不变量**（必须在所有操作中维护）：
```
available_balance >= 0
frozen_balance >= 0
(available_balance + frozen_balance) = 该钱包的真实余额
```

#### 索引说明

| 索引名 | 字段 | 类型 | 用途 |
|--------|------|------|------|
| uk_user_currency_type | user_id, currency_code, wallet_type, delete_at | UNIQUE | 保证同用户+同币种+同类型只有一行；所有余额操作的精确定位条件 |
| idx_user_id | user_id | 普通 | C1 查用户所有币种余额（SELECT ... WHERE user_id = ? GROUP BY currency_code） |

#### 数据量预估

每用户最多 5 币种 × 5 钱包类型 = 25 行（实际当前版本 5 币种 × 3 类型 = 15 行，代理和场馆暂不建数据）。10 万用户约 150 万行，属于小表。

### 4.2 表 ② transaction_log — 余额变更流水表

#### 为什么必须

承担两个核心职责，缺一不可：

**职责一：资金安全审计**。需求理解 3.5 明确要求"对账一致性——账务流水与账户余额需要强一致性校验"。每笔余额变动必须留痕，否则无法追溯任何资金操作。

**职责二：幂等去重**。架构设计 5.2 确定了幂等方案——基于 (order_id, log_type) 唯一索引。财务模块 RPC 调用钱包入账/扣款时可能因网络超时重试，没有幂等机制会导致重复入账。

被调用的接口：

| 操作类型 | 接口 | 说明 |
|---------|------|------|
| **读取（幂等检查）** | R2/R4/R5/R6/R7/R8（所有余额变更RPC） | 执行前先检查 order_id 是否已存在 |

#### 表做什么

一句话概括：**钱包的"银行流水"，每一分钱的变动都有记录**。

**幂等机制说明**：

```
同一笔充值订单 C123456 只能产生一条 log_type=1（充值入账）的流水
同一笔提现订单 T789012 会产生两条流水：
  - log_type=2（提现冻结）— 创建订单时
  - log_type=3（提现扣款）— 出款成功时
  或：
  - log_type=2（提现冻结）— 创建订单时
  - log_type=4（提现解冻）— 驳回时

(order_id, log_type) 唯一索引保证每种操作只执行一次
```

| 索引名 | 字段 | 类型 | 用途 |
|--------|------|------|------|
| uk_order_log_type | order_id, log_type | UNIQUE | **幂等去重核心索引**：同一 order_id + 同一 log_type 不允许重复插入 |
| idx_user_currency_time | user_id, currency_code, create_at | 普通 | C端记录查询：按用户+币种+时间排序分页 |
| idx_user_type_time | user_id, log_type, create_at | 普通 | 按流水类型筛选：如只看充值流水、只看提现流水 |

#### 数据量预估

这是**增长最快的表**。每笔余额变动至少一条，兑换操作产生3条。假设日均1万笔操作，半年约180万条。如果用户增长到10万活跃用户，半年可达百万级。超过1000万行时可考虑按 user_id 范围分表，但当前阶段不过度设计。

### 4.3 表 ③ deposit_order — 充值订单表

#### 为什么必须

充值是用户把钱充进平台的核心流程。需要一张订单表承载以下职责：
1. **创建订单**（C4）：记录用户选择的充值金额、方式、汇率快照、奖金活动等信息
2. **重复充值检查**（C21）：查同用户+同币种+同金额的待支付订单数量
3. **充值入账更新**（R2）：财务回调后更新订单状态为"支付成功"
4. **超时关闭**（定时任务）：超过有效期的待支付订单自动标记失败
5. **C端充值记录展示**（C12/C17）：用户查看自己的充值历史
6. **B端充值记录展示**（需求理解 5.3）：24个字段的管理视图

#### 表做什么

记录每一笔充值订单的完整信息：订单号、用户、金额、方式、状态、汇率、奖金活动、通道信息等。

一句话概括：**每一笔充值操作的完整档案**。

### 4.4 表 ④ withdrawal_order — 提现订单表

#### 为什么必须

提现是用户把钱提出平台的核心流程，涉及冻结、多级审核、出款，是**最复杂的业务链路**。需要一张订单表承载：
1. **创建订单+冻结**（C7）：与余额冻结在同一个数据库事务内
2. **审核状态流转**（R4/R5）：7 种状态的多级流转
3. **今日已提现限额查询**（C19）：SUM(amount) WHERE today
4. **C端提现记录展示**（C13/C18）：用户查看提现历史
5. **B端提现记录展示**（需求理解 5.3）：27个字段的管理视图

#### 状态流转图

```
创建订单: status = 1（待风控审核）
    │
    ├── 风控驳回: status → 2  → 钱包解冻(R4)
    ├── 风控通过: status → 3（待财务审核）
    │       ├── 财务驳回: status → 4 → 钱包解冻(R4)
    │       ├── 财务通过: status → 5（待出款）
    │       │       ├── 出款失败: status → 6 → 钱包解冻(R4)
    │       │       └── 出款成功: status → 7 → 钱包确认扣款(R5)
```

|--------|------|------|------|

| idx_user_currency_time | user_id, currency_code, create_at | 普通 | C13 提现记录列表 |
| idx_user_currency_status | user_id, currency_code, status | 普通 | C19 查今日已提现总额（排除驳回/失败状态） |
| idx_status | status | 普通 | B端按状态筛选提现记录 |

### 4.5 表 ⑤ withdrawal_account — 提现账户信息表

### 4.6 表 ⑥ currency_config — 币种配置表

#### 为什么必须

**这是所有币种和汇率数据的唯一源头**。所有涉及币种展示、汇率查询的接口都依赖这张表。

被依赖的接口：

| 操作类型 | 接口 | 说明 |
|---------|------|------|
| **读取** | C1 币种列表、C3/C6 充值/提现方式（获取汇率）、C4/C7 创建订单（锁定汇率）、C10/C11 兑换（获取汇率）、B1 币种列表、CR1/CR2 RPC 对外 | 几乎所有接口都读取 |
| **写入** | B2 币种编辑、B5 手动刷新实时汇率、汇率定时更新任务 | B端管理+定时任务写入 |

如果没有这张表，系统不知道支持哪些币种、每个币种的汇率是多少、币种是否启用，所有后续功能无从运转。

管理平台支持的所有币种的配置参数：基本信息（代码/名称/类型/国家）、展示规则（图标/符号/分隔符/小数位）、汇率体系（实时汇率/平台汇率/浮动/阈值）、状态。

一句话概括：**平台支持哪些币种、每个币种长什么样、汇率是多少**。

| 字段 | 说明 | 来源 |
|------|------|------|
| realtime_rate | 实时汇率（3家三方API取平均） | 需求理解 4.2："三方API获取，取平均值" |

### 5.1 表 ⑦ exchange_record — 兑换记录表

#### 为什么是"应该"而非"必须"

兑换操作本身的资金流转（扣法币+加BSB）通过 wallet 表 + transaction_log 即可完成。但 C 端"兑换"标签页（需求理解 3.4）需要展示：兑换金额、赠送x%、实际到账、时间——这些信息 transaction_log 的字段不足以完整覆盖（比如赠送比例、流水倍数要求等）。

**如果不建这张表**：兑换操作仍然可以执行，但用户在 C 端的"兑换"记录页无法展示完整信息。只能退化为在 transaction_log 中查看，缺少赠送比例、到账金额等聚合展示。

**建议**：首版就建。因为需求文档中明确有"兑换"记录标签页，且产品原型给出了明确的展示字段。

### 5.4 表 ⑩ system_config — 系统配置表

#### 为什么是"应该"而非"必须"

系统级配置（如基准币种）如果不建表，技术上可以存 ETCD 或环境变量。但需求理解 4.1 明确要求"基准币种只可设置一次，提交后按钮隐藏"，这种**业务层的一次性配置**应该有业务层的存储和状态管理，ETCD 更适合存技术配置（端口/DB名/地址）。

**如果不建这张表**：基准币种配置需要存到 ETCD 或代码常量，缺少"是否已设置"的状态管理能力。

**建议**：首版就建。字段少、结构简单，建了后续可扩展存放其他系统级配置。

#### DDL

```sql
CREATE TABLE `system_config` (
  `id`                BIGINT        NOT NULL AUTO_INCREMENT  COMMENT '主键ID',
  `config_key`        VARCHAR(50)   NOT NULL                 COMMENT '配置键',
  `config_value`      VARCHAR(500)  NOT NULL DEFAULT ''      COMMENT '配置值',
  `config_desc`       VARCHAR(200)  NOT NULL DEFAULT ''      COMMENT '配置说明',
  `is_locked`         INT           NOT NULL DEFAULT 0       COMMENT '是否锁定（1=锁定不可修改）',
  -- [通用审计字段]
  `create_by`         BIGINT        NOT NULL DEFAULT 0       COMMENT '创建人ID',
  `create_by_name`    VARCHAR(50)   NOT NULL DEFAULT ''      COMMENT '创建人名称',
  `create_at`         BIGINT        NOT NULL DEFAULT 0       COMMENT '创建时间（毫秒时间戳）',
  `update_by`         BIGINT        NOT NULL DEFAULT 0       COMMENT '更新人ID',
  `update_by_name`    VARCHAR(50)   NOT NULL DEFAULT ''      COMMENT '更新人名称',
  `update_at`         BIGINT        NOT NULL DEFAULT 0       COMMENT '更新时间（毫秒时间戳）',
  `delete_at`         BIGINT        NOT NULL DEFAULT 0       COMMENT '软删除（0=未删除）',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_config_key` (`config_key`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='系统配置表';
```

#### 预置配置项

| config_key | config_value | config_desc | is_locked |
|-----------|-------------|-------------|-----------|
| base_currency | （初始为空） | 基准币种代码（设置后锁定） | 0 → 设置后改为 1 |
| bsb_usdt_rate | 10 | BSB 锚定 USDT 比率（10 BSB = 1 USDT） | 1（写死不可改） |
| deposit_order_expire_minutes | 30 | 充值订单过期时间（分钟） | 0 |

## 十一、设计决策记录

### 11.1 已确定的设计决策

| 编号 | 决策 | 理由 | 来源 |
|------|------|------|------|
| D1 | 金额统一 DECIMAL(20,8) | 法币最大值（VND可达万亿级）需要20位整数部分；加密货币需要8位小数 | 工程认知 11.1 |
| D2 | 时间统一 BIGINT 毫秒时间戳 | 工程全局规范 | 工程认知 11.1 |
| D3 | 软删除用 delete_at（非 is_deleted） | 工程全局规范 | 工程认知 11.1 |
| D4 | 不建物理外键 | 微服务架构标准做法，利于分库分表 | 工程认知 11.1 |
| D5 | wallet 用双字段（available + frozen） | 提现二阶段机制需要区分可用和冻结 | 架构设计 5.1 |
| D6 | transaction_log 用 (order_id, log_type) 唯一索引做幂等 | 比独立幂等表更简单，一举两得 | 架构设计 5.2 |
| D7 | 入款/出款汇率不存储在 currency_config | 计算值，避免冗余和一致性问题 | 数据模型 10.3 |
| D8 | 入款/出款汇率存储在 rate_change_log | 历史快照，后续浮动值可能变化 | 数据模型 11.3 |
| D9 | 订单表冗余用户昵称/通道名称/账户信息 | 避免展示时关联查询，且作为历史快照 | 架构设计 4.6 |
| D10 | 代理钱包/场馆钱包只预留枚举不建数据 | 需求灰字"暂不制作" | 需求理解 2.2 |

### 11.2 待评审确认的设计点

### 12.2 核心检验标准

建表前/评审时，用以下标准检验表结构设计是否正确：

```
✓ 必须级的 6 张表覆盖了所有核心资金流程（充值、提现、兑换、余额管理、币种配置）
✓ 应该级的 4 张表覆盖了所有需求文档明确要求的页面展示（兑换记录、奖励记录、汇率日志、基准配置）
✓ 可选级的 3 张表都有明确的"什么时候需要"判断条件
✓ 每个产品原型页面上的字段，都能在某张表中找到对应
✓ 每个 C 端列表页的展示字段，都能通过单表查询获得（不需要复杂 JOIN）
✓ 每个 B 端列表页的 20+ 字段，都能在表中直接映射
✓ 金额字段统一 DECIMAL(20,8)
✓ 时间字段统一 BIGINT 毫秒时间戳
✓ 每张表都有通用审计字段
✓ wallet 表的双字段设计覆盖冻结/解冻/扣款所有场景
✓ transaction_log 的唯一索引满足幂等要求
✓ 订单表的汇率快照字段确保不依赖后续汇率变动
✓ 不建物理外键，由应用层保证一致性
```
