# 第一部分：表结构总览与分级

## 1.1 分级标准定义

| 级别 | 含义 | 判定标准 |
|------|------|---------|
| **必须 (Must)** | 缺少该表则系统核心功能不可用 | 需求文档明确定义的核心业务实体，或多个模块的数据交互依赖该表 |
| **应该 (Should)** | 缺少该表则功能不完整或缺少关键审计能力 | 需求文档有对应需求但系统不会因此停转，或属于对账/审计/运营管理的必要支撑 |
| **可选 (Optional)** | 增强型功能表，可初期不做但长期需要 | 预留扩展、辅助分析、运营优化等，需求文档中有提及但非首版必做 |

## 1.2 全部表清单（按级别分类）

### 必须 (Must) — 6 张

| 序号 | 表名 | 所属域 | 一句话定位 | 必须的理由 |
|------|------|--------|-----------|-----------|
| 1 | `currency` | 币种配置 | 币种主表，整个资金体系的底座 | 所有涉及资金的操作（钱包/充值/提现/兑换/汇率）都依赖币种编码，无此表则系统无法启动 |
| 2 | `currency_rate` | 币种配置 | 汇率表，记录每个币种的实时和平台汇率 | 充值入款汇率、提现出款汇率、兑换汇率、统计报表全部依赖此表计算，无此表则资金换算全部失效 |
| 3 | `base_currency` | 币种配置 | 基准币种全局唯一配置 | 活动奖金计算、多币种统计报表、对账审计的统一换算基准，设置后不可更改（需求文档3-2） |
| 4 | `wallet` | 账户核心 | 钱包余额主表，全系统最核心 | 承载用户所有币种所有钱包类型的余额和冻结金额，是所有资金操作的读写终点 |
| 5 | `wallet_transaction` | 账户核心 | 资金流水表，对账核心（只增不改不删） | 需求文档9明确要求"任何资金变动都要有流水记录"，是幂等保障的最终兜底和审计合规的必要凭证 |
| 6 | `deposit_order` | 业务订单 | 充值订单表 | 需求文档8-1明确有"充值记录Tab"，C端用户需要查看充值历史；重复转账拦截（需求文档5-5）也依赖此表 |

### 应该 (Should) — 5 张

| 序号 | 表名 | 所属域 | 一句话定位 | 应该的理由 |
|------|------|--------|-----------|-----------|
| 7 | `withdraw_order` | 业务订单 | 提现订单表 | 需求文档8-1明确有"提现记录Tab"，C端用户需要查看提现历史；三级审核流程的订单载体 |
| 8 | `exchange_order` | 业务订单 | 兑换订单表 | 需求文档8-1明确有"兑换记录Tab"，C端用户需要查看兑换历史 |
| 9 | `withdraw_account` | 辅助数据 | 提现账户表（银行卡/电子钱包/USDT地址） | 需求文档7-2/7-3/7-4明确要求保存用户提现账户信息，后续提现可直接选用 |
| 10 | `currency_rate_log` | 币种配置 | 汇率变更日志 | 需求文档4-4明确要求"汇率日志"页面，记录平台汇率自动更新的审计轨迹 |
| 11 | `reward_record` | 辅助数据 | 奖励记录表 | 需求文档8-1明确有"奖励记录Tab"，且稽核流水机制需要跟踪每笔奖励的稽核进度 |

### 可选 (Optional) — 3 张

| 序号 | 表名 | 所属域 | 一句话定位 | 可选的理由 |
|------|------|--------|-----------|-----------|
| 12 | `currency_display` | 币种配置 | 币种多语言展示配置 | 平台多地区运营需要币种名称多语言展示，但首版可通过 i18n 服务 + 固定编码解决 |
| 13 | `wallet_snapshot` | 运营分析 | 钱包余额日快照 | 运营侧需要分析用户余额分布趋势，但首版可通过定时任务从 `wallet` 表导出计算 |
| 14 | `repeat_deposit_log` | 风控辅助 | 重复充值拦截记录 | 需求文档5-5有重复转账拦截规则，但首版可通过内存计数 + `deposit_order` 表查询实现 |

# 第二部分：项目审计字段规范

> 通过分析 ser-buser/sql/01.table.sql 中 `sys_user` 表和 ser-item/sql/1.table.sql 中 `item` 表，提炼项目统一的审计字段规范。

## 2.1 标准审计字段集（全表统一）

```sql
-- 以下字段是项目标准，所有业务表都应包含
create_by       BIGINT        DEFAULT NULL COMMENT '创建者',
create_at       BIGINT        DEFAULT '0' COMMENT '创建时间',
update_by       BIGINT        DEFAULT NULL COMMENT '更新者',
update_at       BIGINT        DEFAULT '0' COMMENT '更新时间',
create_by_name  VARCHAR(64)   DEFAULT '' COMMENT '创建人姓名',
update_by_name  VARCHAR(64)   DEFAULT '' COMMENT '更新人姓名',
delete_by       BIGINT        DEFAULT 0 COMMENT '删除者',
delete_at       BIGINT        NOT NULL DEFAULT 0 COMMENT '软删除(0=未删除,非0=删除时间戳)',
```

## 2.2 简化审计字段集（日志/流水表）

日志表和流水表只增不改不删，不需要 `update_*` 和 `delete_*` 系列字段：

```sql
-- 日志/流水表使用简化版
create_by       BIGINT        DEFAULT NULL COMMENT '创建者',
create_at       BIGINT        DEFAULT '0' COMMENT '创建时间',
create_by_name  VARCHAR(64)   DEFAULT '' COMMENT '创建人姓名',
```

## 2.3 项目命名规范总结

| 规范项 | 项目现有惯例 | 说明 |
|--------|-------------|------|
| 表名前缀 | **无前缀** | `sys_user`、`item`、`langs`，不加 `t_` |
| 字符集 | `utf8mb4` | 全表统一 |
| 排序规则 | `utf8mb4_bin` | 区分大小写（部分表用 `utf8mb4_unicode_ci`） |
| 引擎 | `InnoDB` | 全表统一 |
| 主键 | `bigint NOT NULL AUTO_INCREMENT` | 部分表用 `bigint unsigned` |
| 时间存储 | `bigint` unix 时间戳 | 不使用 `datetime`/`timestamp` |
| 软删除 | `delete_at bigint DEFAULT 0` | 0=未删除，非0=删除时间戳 |
| 外键约束 | **不使用** | 关系通过字段隐含，应用层保证一致性 |
| CHECK约束 | **不使用** | 所有校验在应用层完成 |

# 第三部分：必须(Must)级别表 — 详细设计

## 3.1 currency — 币种主表

### 3.1.1 存在目的

币种主表是整个资金体系的 L0 底座。平台支持多币种运营（VND/IDR/THB/USDT/BSB），每个涉及资金的操作（钱包余额查询、充值、提现、兑换、汇率计算、统计报表）都必须依赖币种编码来定位数据。

**如果没有此表**：系统不知道支持哪些币种，C端无法展示币种切换，充值/提现/兑换全部无法启动。

### 3.1.2 必须性依据

- 币种配置需求文档3-3：后台币种列表页，管理员可新增/编辑/启用/禁用币种
- 币种配置需求文档3-1：三种币种类型（法币/加密/平台），每种类型有不同的展示规则
- 钱包需求文档4-1：金额展示规则（小数位/千分位）由币种决定
- 多个模块依赖：wallet、deposit_order、withdraw_order、exchange_order 等所有表都通过 currency_code 关联

### 3.1.3 DDL

```sql
CREATE TABLE `currency` (
    `id`              bigint        NOT NULL AUTO_INCREMENT COMMENT '主键ID',
    `currency_code`   varchar(10)   NOT NULL COMMENT '币种编码(VND/IDR/THB/USDT/BSB)，创建后不可修改',
    `currency_name`   varchar(64)   NOT NULL COMMENT '币种名称',
    `currency_type`   tinyint       NOT NULL COMMENT '币种类型(1法币/2加密货币/3平台货币)',
    `icon_url`        varchar(256)  NOT NULL DEFAULT '' COMMENT '币种图标URL(SVG/PNG,<=5k)',
    `float_rate`      decimal(10,4) NOT NULL DEFAULT 0.0000 COMMENT '汇率浮动百分比(如5.00表示5%)',
    `threshold`       decimal(10,4) NOT NULL DEFAULT 0.0000 COMMENT '偏差阈值百分比(偏差>=此值时更新平台汇率)',
    `decimal_places`  tinyint       NOT NULL DEFAULT 2 COMMENT '金额展示小数位数(VND/IDR=0,THB/BSB=2,USDT=8)',
    `thousand_sep`    varchar(4)    NOT NULL DEFAULT ',' COMMENT '千分位分隔符(VND/IDR=".",其他=",")',
    `sort_order`      int           NOT NULL DEFAULT 0 COMMENT '排序权重，数值越小越靠前',
    `status`          tinyint       NOT NULL DEFAULT 2 COMMENT '状态(1启用/2禁用)，新增默认禁用',
    `create_by`       bigint        DEFAULT NULL COMMENT '创建者',
    `create_at`       bigint        DEFAULT '0' COMMENT '创建时间',
    `update_by`       bigint        DEFAULT NULL COMMENT '更新者',
    `update_at`       bigint        DEFAULT '0' COMMENT '更新时间',
    `create_by_name`  varchar(64)   DEFAULT '' COMMENT '创建人姓名',
    `update_by_name`  varchar(64)   DEFAULT '' COMMENT '更新人姓名',
    `delete_by`       bigint        DEFAULT 0 COMMENT '删除者',
    `delete_at`       bigint        NOT NULL DEFAULT 0 COMMENT '软删除(0=未删除,非0=删除时间戳)',
    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_currency_code` (`currency_code`, `delete_at`),
    KEY `idx_status` (`status`),
    KEY `idx_currency_type` (`currency_type`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_bin COMMENT='币种主表';
```

### 3.1.4 字段分级

| 字段 | 级别 | 类型 | 用途 | 依据 |
|------|------|------|------|------|
| id | 必须 | bigint PK | 主键 | 项目规范 |
| currency_code | 必须 | varchar(10) UK | 币种唯一标识，全系统关联键 | 需求文档3-3，创建后不可改 |
| currency_name | 必须 | varchar(64) | 展示用名称 | 需求文档3-3"币种名称" |
| currency_type | 必须 | tinyint | 区分法币/加密/平台，影响充提规则 | 需求文档3-1三种类型 |
| icon_url | 应该 | varchar(256) | 币种图标 | 原型2"编辑弹窗：图标上传" |
| float_rate | 必须 | decimal(10,4) | 计算入款/出款汇率 | 需求文档4-1"汇率浮动%" |
| threshold | 必须 | decimal(10,4) | 平台汇率自动更新阈值 | 需求文档4-2"偏差>=阈值时更新" |
| decimal_places | 必须 | tinyint | C端金额展示精度 | 需求文档4-3"VND/IDR=0,THB/BSB=2,USDT=8" |
| thousand_sep | 必须 | varchar(4) | C端千分位展示 | 需求文档4-3"千分位符号" |
| sort_order | 应该 | int | C端币种列表排序 | 原型1"排序" |
| status | 必须 | tinyint | 启用/禁用控制 | 需求文档3-3"启用/禁用" |

## 3.2 currency_rate — 汇率表

### 3.2.1 存在目的

记录每个币种的实时汇率和平台汇率。实时汇率从3家三方API取均值定时更新，平台汇率在偏差超阈值时自动从实时汇率拷贝。入款汇率和出款汇率不落表，由 `实时汇率 × (1±float_rate%)` 实时计算。

**如果没有此表**：充值跨币种换算、提现跨币种换算、兑换（法币→BSB）、统计报表多币种汇总全部失效。

### 3.2.2 必须性依据

- 币种配置需求文档4-1/4-2：实时汇率定义、平台汇率定义、入款/出款汇率计算规则
- 需求理解规则10："充值用入款汇率，提现用出款汇率，统计用平台汇率"
- 需求理解规则16："三方汇率取3家API平均值"
- 钱包/财务/统计等所有涉及汇率的场景都读此表

### 3.2.3 DDL

```sql
CREATE TABLE `currency_rate` (
    `id`              bigint        NOT NULL AUTO_INCREMENT COMMENT '主键ID',
    `currency_code`   varchar(10)   NOT NULL COMMENT '币种编码',
    `rate_type`       tinyint       NOT NULL COMMENT '汇率类型(1实时汇率/2平台汇率)',
    `rate_value`      decimal(20,8) NOT NULL DEFAULT 0.00000000 COMMENT '汇率值(1基准币种=X本币种)',
    `api_a_rate`      decimal(20,8) NOT NULL DEFAULT 0.00000000 COMMENT 'API-A返回汇率(仅实时类型)',
    `api_b_rate`      decimal(20,8) NOT NULL DEFAULT 0.00000000 COMMENT 'API-B返回汇率(仅实时类型)',
    `api_c_rate`      decimal(20,8) NOT NULL DEFAULT 0.00000000 COMMENT 'API-C返回汇率(仅实时类型)',
    `sync_at`         bigint        NOT NULL DEFAULT 0 COMMENT '最近一次同步时间戳',
    `create_by`       bigint        DEFAULT NULL COMMENT '创建者',
    `create_at`       bigint        DEFAULT '0' COMMENT '创建时间',
    `update_by`       bigint        DEFAULT NULL COMMENT '更新者',
    `update_at`       bigint        DEFAULT '0' COMMENT '更新时间',
    `create_by_name`  varchar(64)   DEFAULT '' COMMENT '创建人姓名',
    `update_by_name`  varchar(64)   DEFAULT '' COMMENT '更新人姓名',
    `delete_by`       bigint        DEFAULT 0 COMMENT '删除者',
    `delete_at`       bigint        NOT NULL DEFAULT 0 COMMENT '软删除(0=未删除,非0=删除时间戳)',
    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_code_type` (`currency_code`, `rate_type`, `delete_at`),
    KEY `idx_currency_code` (`currency_code`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_bin COMMENT='汇率表';
```

### 3.2.4 字段分级

| 字段 | 级别 | 类型 | 用途 | 依据 |
|------|------|------|------|------|
| currency_code | 必须 | varchar(10) | 关联币种 | 每个币种有独立汇率 |
| rate_type | 必须 | tinyint | 区分实时/平台汇率 | 需求文档4-1两种汇率 |
| rate_value | 必须 | decimal(20,8) | 汇率值 | 所有汇率计算的基础 |
| api_a/b/c_rate | 应该 | decimal(20,8) | 三方API原始值 | 排查汇率异常时需要追溯每家API的返回值 |
| sync_at | 应该 | bigint | 同步时间 | 判断汇率数据是否过期 |

### 3.2.5 特殊说明

- **每个币种每种类型只有一条记录**（唯一索引 `uk_code_type`），通过 UPDATE 更新最新值
- **BSB 锚定 USDT**（10BSB = 1USDT），BSB 的汇率记录固定写入，不参与三方同步
- **入款/出款汇率不持久化**，实时计算：入款汇率 = rate_value × (1 + float_rate%)，出款汇率 = rate_value × (1 - float_rate%)

## 3.3 base_currency — 基准币种配置

### 3.3.1 存在目的

全局唯一的基准币种设置（推荐USDT），设置后不可更改。所有币种的统计报表和对账审计需要通过基准币种进行统一换算。

**如果没有此表**：多币种报表无法统一汇总，活动奖金计算无统一基准，对账审计缺少统一口径。

### 3.3.2 必须性依据

- 币种配置需求文档3-2："基准币种设置后不可更改"
- 财务需求文档3："所有币种金额在账务层需换算为基准币种进行汇总统计"

### 3.3.3 DDL

```sql
CREATE TABLE `base_currency` (
    `id`              bigint        NOT NULL AUTO_INCREMENT COMMENT '主键ID',
    `currency_code`   varchar(10)   NOT NULL COMMENT '基准币种编码(推荐USDT)',
    `create_by`       bigint        DEFAULT NULL COMMENT '创建者',
    `create_at`       bigint        DEFAULT '0' COMMENT '创建时间',
    `update_by`       bigint        DEFAULT NULL COMMENT '更新者',
    `update_at`       bigint        DEFAULT '0' COMMENT '更新时间',
    `create_by_name`  varchar(64)   DEFAULT '' COMMENT '创建人姓名',
    `update_by_name`  varchar(64)   DEFAULT '' COMMENT '更新人姓名',
    `delete_by`       bigint        DEFAULT 0 COMMENT '删除者',
    `delete_at`       bigint        NOT NULL DEFAULT 0 COMMENT '软删除(0=未删除,非0=删除时间戳)',
    PRIMARY KEY (`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_bin COMMENT='基准币种配置(全局唯一)';
```

### 3.3.4 字段分级

| 字段 | 级别 | 类型 | 用途 | 依据 |
|------|------|------|------|------|
| currency_code | 必须 | varchar(10) | 基准币种编码 | 唯一核心字段 |

### 3.3.5 特殊说明

- 表中最多只有一条有效记录（delete_at=0）

- 为什么单独建表：基准币种是**平台级**配置，不是币种的属性，放在 `currency` 表中不合适

## 3.4 wallet — 钱包余额主表

### 3.4.1 存在目的

承载用户在某个币种下某种类型钱包的余额和冻结金额。这是全系统最核心的表——所有资金操作（充值入账、提现冻结/扣款、兑换、投注扣款/返奖、礼物消费、奖励入账）最终都指向此表的余额修改。

**如果没有此表**：整个钱包系统无法运作，用户没有余额概念。

### 3.4.2 必须性依据

- 钱包需求文档4-1："每个币种独立余额和账本"
- 币种配置需求文档2-1："五种钱包"及各自的操作权限
- 需求文档9："账务流水与账户余额必须强一致"
- 几乎所有外部RPC调用（游戏投注、直播礼物、活动奖励、财务加减款）都以修改此表余额为最终目的

### 3.4.3 DDL


|------|------|------|------|------|
| user_id | 必须 | bigint | 钱包归属用户 | 用户维度隔离 |
| currency_code | 必须 | varchar(10) | 钱包关联币种 | 需求文档4-1"每个币种独立余额" |

### 3.4.5 三维唯一性模型

```
用户 Alice (user_id=1001) 的钱包矩阵：

              VND        IDR        THB        USDT       BSB
中心(1)    [wallet_1]  [wallet_2]  [wallet_3]  [wallet_4]  [wallet_5]
奖励(2)    [wallet_6]  [wallet_7]  [wallet_8]  [wallet_9]  [wallet_10]
主播(3)    [wallet_11] [wallet_12] [wallet_13] [wallet_14] [wallet_15]
代理(4)    (预留)      (预留)      (预留)      (预留)      (预留)
场馆(5)    (预留)      (预留)      (预留)      (预留)      (预留)

每个格子 = 一行 wallet 记录，通过 (user_id, currency_code, wallet_type) 唯一定位
按懒创建策略，用户首次涉及某币种某钱包时才创建
```

### 3.4.6 冻结模型

```
提现全生命周期中 wallet 的字段变化：

[初始]  balance=10000, frozen=0       → 可提现=10000
        ↓ Freeze(5000)
[冻结]  balance=10000, frozen=5000    → 可提现=5000
        ↓
        ├─ Unfreeze(5000) [审核驳回]
        │  balance=10000, frozen=0    → 可提现=10000（恢复原状）
        │
        └─ DebitFrozen(5000) [出款成功]
           balance=5000, frozen=0     → 可提现=5000（余额正式减少）

关键：Freeze 只增 frozen 不扣 balance（用户看到总余额不变）
     DebitFrozen 同时扣 balance 和 frozen（出款确认后正式扣除）
     Unfreeze 只减 frozen（驳回时恢复可提现额度）
```

## 3.5 wallet_transaction — 资金流水表

### 3.5.1 存在目的

记录每一笔余额变动的完整信息，包括变动前后的余额快照。这是对账审计的核心凭证表，也是幂等保障的最终兜底（唯一索引防止重复入账）。**只增不改不删**——任何修正通过新增反向流水实现。

**如果没有此表**：无法对账审计，无法追溯余额变化历史，幂等性在DB层失去最终兜底。

### 3.5.2 必须性依据

- 需求文档9："账务流水与账户余额必须强一致，任何资金变动都要有流水记录"
- 需求文档9："充值回调幂等处理，防止重复入账"——唯一索引 `uk_biz_order_wallet` 是幂等的最终保障
- 架构设计4.3：5层资金安全设计中，唯一索引是第5层兜底

### 3.5.3 DDL

```sql
CREATE TABLE `wallet_transaction` (
    `id`              bigint        NOT NULL AUTO_INCREMENT COMMENT '主键ID',
    `user_id`         bigint        NOT NULL COMMENT '用户ID',

    `amount`          decimal(20,8) NOT NULL COMMENT '变动金额(始终为正数)',
    `balance_before`  decimal(20,8) NOT NULL COMMENT '变动前余额',
    `balance_after`   decimal(20,8) NOT NULL COMMENT '变动后余额',
    `frozen_before`   decimal(20,8) NOT NULL DEFAULT 0.00000000 COMMENT '变动前冻结金额',
    `frozen_after`    decimal(20,8) NOT NULL DEFAULT 0.00000000 COMMENT '变动后冻结金额',

    KEY `idx_user_biz` (`user_id`, `biz_type`, `create_at`)

```

### 3.5.4 字段分级

| 字段 | 级别 | 类型 | 用途 | 依据 |
|------|------|------|------|------|
| user_id + currency_code + wallet_type | 必须 | — | 定位到具体钱包 | 流水归属 |
| direction | 必须 | tinyint | 收入/支出方向 | 方向明确，避免正负歧义 |
| amount | 必须 | decimal(20,8) | 变动金额（正数） | 核心数据 |
| balance_before / balance_after | 必须 | decimal(20,8) | 余额快照 | 对账核心，验证余额链条连续性 |
| frozen_before / frozen_after | 应该 | decimal(20,8) | 冻结快照 | 冻结/解冻操作审计 |
| biz_type | 必须 | varchar(32) | 业务类型 | 幂等唯一索引的一部分 |
| order_no | 必须 | varchar(64) | 业务订单号 | 幂等唯一索引的一部分 |
| remark | 可选 | varchar(256) | 补充说明 | 人工加减款等场景需要备注 |

### 3.5.5 唯一索引 `uk_biz_order_wallet` 设计依据

```
为什么是 UNIQUE(biz_type, order_no, wallet_type) 三字段组合？

场景：BetDebit 投注扣款
  一笔投注(order_no=BET001)需要同时：
    - 从中心钱包扣80 → (biz_type=bet_debit, order_no=BET001, wallet_type=1)
    - 从奖励钱包扣20 → (biz_type=bet_debit, order_no=BET001, wallet_type=2)

  如果唯一索引不含 wallet_type，第二条流水会被拒绝。

同理：返奖、兑换（法币扣+BSB中心加+BSB奖励加）都涉及同一 order_no 多个 wallet_type。
```

### 3.5.6 流水链验证原理

```
对账公式：
  wallet.balance == transaction[最新一条].balance_after
  且对每一条流水 T[n]：
    T[n].balance_before == T[n-1].balance_after
    T[n].balance_after  == T[n].balance_before + T[n].amount  (direction=1)
    T[n].balance_after  == T[n].balance_before - T[n].amount  (direction=2)

如果任何一条不满足，说明数据异常需人工介入。
```

### 3.5.7 为什么没有 update/delete 审计字段

流水表天然只增不改不删，修正通过新增反向流水实现（如充值补单对应一条新的 deposit 流水），因此不需要 `update_by`、`update_at`、`update_by_name`、`delete_by`、`delete_at` 字段。

## 4.1 withdraw_order — 提现订单表

### 4.1.1 存在目的

记录用户发起的提现请求全生命周期信息，包括三级审核状态、冻结金额、出款结果等。提现涉及钱包冻结→审核→出款→确认扣款/解冻的复杂流程，此表是该流程的订单载体。

### 4.1.2 为什么是"应该"而非"必须"

从技术角度，提现订单的**主体管理**在财务模块（三级审核、通道出款）。钱包侧的 `withdraw_order` 是一份**本地副本**，主要服务于：
- C端提现记录Tab查询（避免每次查询都RPC调用财务服务）
- 本地冻结金额核对（frozen_amount 与 wallet.frozen 对照）

如果紧急上线，可以只在财务侧维护提现订单，钱包侧通过RPC查询。但这会导致C端查询链路变长、依赖增加，因此**应该**在钱包侧维护本地副本。

## 5.3 repeat_deposit_log — 重复充值拦截记录

### 5.3.1 场景分析

需求文档5-5要求"同UserID+同币种+同金额"的重复充值拦截：1-2笔弹窗警告，第3笔强制拦截。

### 5.3.2 为什么是"可选"

- 首版可通过查询 `deposit_order` 表实现：`SELECT COUNT(*) FROM deposit_order WHERE user_id=? AND currency_code=? AND amount=? AND create_at > ?`
- 配合 Redis 计数器可进一步提升性能
- 只有当拦截规则变得更复杂（如需要记录拦截详情、拦截历史）时，才需要独立表

### 5.3.3 DDL（预设计）

```sql
CREATE TABLE `repeat_deposit_log` (
    `id`              bigint        NOT NULL AUTO_INCREMENT COMMENT '主键ID',
    `user_id`         bigint        NOT NULL COMMENT '用户ID',
    `currency_code`   varchar(10)   NOT NULL COMMENT '币种编码',
    `amount`          decimal(20,8) NOT NULL COMMENT '充值金额',
    `hit_count`       int           NOT NULL COMMENT '命中次数(1=首次警告/2=二次警告/3=拦截)',
    `action`          tinyint       NOT NULL COMMENT '处理动作(1警告继续/2警告取消/3强制拦截)',
    `deposit_order_no` varchar(64)  NOT NULL DEFAULT '' COMMENT '关联充值订单号(用户选择继续时)',
    `create_at`       bigint        DEFAULT '0' COMMENT '创建时间',
    `create_by_name`  varchar(64)   DEFAULT '' COMMENT '创建人姓名',
    PRIMARY KEY (`id`),
    KEY `idx_user_currency_amount` (`user_id`, `currency_code`, `amount`, `create_at`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_bin COMMENT='重复充值拦截记录(可选)';
```

## 7.2 索引设计原则

| 原则 | 说明 |
|------|------|
| 唯一索引含 delete_at | 软删除后允许重建同编码/同订单号的记录 |
| 复合索引遵循最左前缀 | 查询条件中最常用的字段放在最左 |
| C端查询索引必含 user_id | 所有C端列表查询都按用户维度隔离 |
| 不建冗余索引 | 如已有 (user_id, currency_code, create_at)，不再单独建 (user_id, currency_code) |
| 不建外键索引 | 项目不使用外键约束，关系通过应用层保证 |

# 第十部分：实施建议

## 10.1 开发优先级

```
第一阶段（P0 核心底座）：
  currency → base_currency → currency_rate → wallet → wallet_transaction
  理由：没有这5张表，整个钱包系统无法启动

第二阶段（P1 业务订单）：
  deposit_order → withdraw_order → exchange_order → withdraw_account
  理由：支撑充值、提现、兑换的完整业务流程

第三阶段（P2 审计完善）：
  currency_rate_log → reward_record
  理由：汇率审计和奖励稽核，完善运营管理能力

第四阶段（P3 可选增强）：
  currency_display → wallet_snapshot → repeat_deposit_log
  理由：多语言、运营分析、风控细化，按需上线
```

## 10.3 核心设计决策汇总

| 决策 | 选择 | 依据 |
|------|------|------|
| 金额精度 | DECIMAL(20,8) 统一 | USDT需8位小数，法币在展示层截断（架构设计3.2.4） |
| 余额冻结模型 | balance + frozen 双字段 | 冻结不扣balance，可提现=balance-frozen（架构设计8.1） |
| 并发控制 | FOR UPDATE + version 乐观锁 | 双重保障（架构设计4.1） |
| 幂等兜底 | wallet_transaction 唯一索引 | DB层最终保障同一笔不重复入账（架构设计4.3） |
| 流水表策略 | 只增不改不删 | 修正通过反向流水（架构设计3.2.3） |
| 钱包创建 | 懒创建（首次涉及时创建） | 避免预创建大量空钱包行 |
| 表名规范 | 无前缀 | 匹配项目现有惯例 |
| 软删除 | delete_at=0 为未删除 | 项目统一规范 |
