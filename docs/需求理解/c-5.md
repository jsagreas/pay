# 一、币种配置模块

> 定位：基础配置模块，为钱包和财务提供币种元数据和汇率数据
> 使用者：B端运营/管理人员配置，C端和其他模块读取消费
> 自己负责

## 1. 资金钱包体系（核心基础）

这是整个钱包系统的架构基础，所有模块都建立在这个体系之上。

### 1.1 四个子钱包

每个用户拥有4个独立的子钱包，各自独立核算：

| 子钱包 | 用途 | 资金来源 |
|--------|------|----------|
| 中心钱包 | 用户主钱包，充值/提现/投注/消费的主体 | 用户充值 |
| 奖励钱包 | 存放平台发放的奖励/活动奖金 | 充值奖金、活动奖励 |
| 代理钱包 | 代理角色的佣金收入 | 代理返佣 |
| 主播钱包 | 主播角色的收益（礼物分成等） | 礼物分成、直播收益 |

关键规则：
- 每个子钱包下，按币种隔离余额，即"子钱包 × 币种"构成余额矩阵
- 举例：用户张三的中心钱包下有 BSB余额、VND余额、IDR余额，互不干扰
- 奖励钱包的资金有流转限制：需满足X倍流水才能提现（防薅羊毛）

### 1.2 BSB — 平台基准币种

BSB 是平台内部定义的虚拟基准币种，不是法币也不是加密货币，作用类似于平台内部的"计量单位"。

BSB 的核心用途：
- **统一度量**：不同法币用户（越南盾、印尼盾、泰铢）在平台内用 BSB 做统一价值计量
- **跨币种换算基准**：所有法币和USDT都通过"1 BSB = X 法币"的汇率与BSB挂钩
- **报表统计基准**：后台统计、财务对账统一折算为BSB
- **投注/游戏计价**：平台内的投注、游戏、道具等以BSB计价

简单理解：用户充值法币 → 换成BSB → 在平台内消费/投注 → 提现时BSB → 换回法币。但也支持直接用法币余额提现（如果用户没有兑换过）。

## 2. 币种类型划分

平台支持三类币种：

| 类型 | 具体币种 | 特点 |
|------|----------|------|
| 法币 | VND（越南盾）、IDR（印尼盾）、THB（泰铢） | 各地区用户的本地货币 |
| 加密货币 | USDT（支持TRC20/ERC20/BEP20三条链） | 通用充值手段，跨地区 |
| 平台币 | BSB | 平台内部基准币种 |

关键规则：
- 用户注册时会有一个默认币种（根据地区），但可以切换查看其他币种钱包
- 币种切换是**整体切换**：切换后整个页面（余额、充值方式、记录）全部变为对应币种的数据，不存在跨币种混合展示

## 3. 后台币种配置（B端）

运营人员在后台配置每个币种的参数，这些参数直接影响C端行为。

核心配置字段：
- **基本信息**：币种名称、币种代码、图标、精度（小数位数）
- **充值配置**：单笔最低/最高限额、是否启用
- **提现配置**：单笔最低/最高限额、每日限额、手续费规则、T+N到账天数
- **状态控制**：启用/禁用（禁用后C端不可见该币种）
- **排序**：影响C端币种列表展示顺序

操作行为全部记录操作日志：谁在什么时间做了什么修改，改前值是什么，改后值是什么。

## 4. 汇率体系

### 4.1 汇率规则

- 汇率由平台运营人员**手动配置**，不是实时对接外部市场汇率
- 格式统一为："1 BSB = X 法币"（如 1 BSB = 23000 VND）
- USDT对BSB也有一个配置汇率

为什么手动配置而非实时汇率：平台需要控制汇率差来保证收益和稳定性，避免市场波动直接冲击平台资金。

### 4.2 汇率应用场景

汇率在以下场景被使用：
- **充值**：法币充值时展示对应BSB数量
- **兑换**：法币/USDT兑换BSB时按汇率折算
- **提现**：BSB提现为法币时按汇率折算
- **报表/统计**：财务后台统一折算为BSB统计

### 4.3 汇率日志

每次汇率变更都记录日志：修改人、修改时间、币种、旧汇率、新汇率。供审计追溯。

# 二、多币种钱包模块

> 定位：C端核心功能模块，用户直接操作的充值/兑换/提现/记录
> 使用者：C端用户（App/H5/Web）
> 自己负责

## 1. 钱包总体设计

### 1.1 页面结构

用户进入个人中心/钱包页面，看到的是**当前选中币种**下的余额和功能入口。

BSB钱包和法币钱包的功能按钮有区别：
- **BSB钱包**：显示"充值"和"提现"按钮
- **法币钱包**（VND/IDR/THB）：显示"充值"、"兑换"和"提现"按钮

为什么法币多一个"兑换"：因为法币用户需要将法币兑换成BSB才能在平台消费/投注。BSB本身就是平台币，不需要兑换。

### 1.2 币种切换规则（关键）

这是一条贯穿整个钱包的核心规则：**币种切换 = 整体上下文切换**。

具体含义：
- 用户从VND切换到BSB后，余额变成BSB余额，充值方式变成BSB的充值方式，记录变成BSB的充值/提现记录
- 不存在"VND钱包里看BSB的记录"这种场景
- 切换是即时的，不涉及资金操作，只是视图切换

## 2. 充值模块

### 2.1 充值方式总览（按币种区分）

不同币种支持的充值方式不同，这是由财务后台的"支付配置"决定的：

| 币种 | 支持的充值方式 | 说明 |
|------|---------------|------|
| BSB | USDT（推荐）+ 银行转账 + 电子钱包 | 最全面，USDT是主推方式 |
| VND | 本地银行转账 | 越南地区 |
| IDR | 电子钱包（推荐）+ 银行转账 | 印尼地区，电子钱包主推 |
| THB | 电子钱包（推荐）+ 银行转账 | 泰国地区，电子钱包主推 |

### 2.2 USDT充值流程

1. 用户选择USDT充值 → 选择链网络（TRC20/ERC20/BEP20）
2. 系统返回该网络的收款地址 + 二维码
3. 用户在自己的钱包App扫码/复制地址转账
4. 区块链确认（2-10分钟）→ 系统自动检测到账 → 入账到用户钱包

特点：没有金额输入环节，用户转多少就到多少。

### 2.3 银行转账/电子钱包充值流程

1. 用户选择充值方式 → 输入金额 → 选择奖金活动（可选）
2. 系统创建订单 → 跳转到第三方支付页面
3. 用户在第三方页面完成支付
4. 第三方回调通知系统 → 系统确认入账

扫码充值（如QRIS）有5分钟倒计时，过期订单作废。

### 2.4 重复转账拦截规则（关键业务规则）

防止用户因网络延迟等原因重复充值：

系统检查条件：**同一用户 + 同一币种 + 同一金额 + 存在处理中的订单**

三级递进响应：
| 情况 | 系统行为 |
|------|----------|
| 有1笔处理中的同金额订单 | 弹窗提醒"您有一笔相同金额的订单正在处理"，用户可选择继续或取消 |
| 有2笔处理中的同金额订单 | 弹窗警告，建议修改金额 |
| 有3笔及以上处理中的同金额订单 | 直接拦截，强制用户选择其他金额或充值方式 |

### 2.5 额外奖金选择规则

充值时可以选择参加平台活动获得额外奖金：
- 充值页面有一个"选择奖金"入口 → 弹出半屏弹窗展示可选活动列表
- 每个活动有最低充值门槛（如"充值满100才能选此活动"）
- 快捷金额标签上会显示对应的奖金百分比（如"100 +5%"）
- 奖金不进入中心钱包，而是进入**奖励钱包**
- 奖励钱包的资金需要完成**X倍流水**才能提现（防止直接薅奖金提现）

## 3. 兑换模块

### 3.1 兑换规则

- 方向：**法币/USDT → BSB**（单向兑换，BSB不能换回法币）
- 汇率：使用币种配置模块设定的汇率，"1 BSB = X 法币"
- 兑换时有一个"赠送比例"（如兑换送5%），赠送的部分进入**奖励钱包**，同样需要X倍流水才能提现

为什么只能单向：BSB是平台消费币，鼓励用户充值消费。提现走的是独立的提现通道，不走兑换。

### 3.2 兑换页面

- 展示当前汇率："1 BSB = 23000 VND"
- 用户输入法币金额 → 实时显示可获得的BSB数量 + 赠送数量
- 确认兑换 → 法币余额减少，BSB余额增加（中心钱包），赠送进奖励钱包

## 4. 提现模块

### 4.1 提现方式确定规则（关键业务规则）

用户能用什么方式提现，**不是自由选择的**，而是由用户的充值历史决定：

| 用户的充值来源 | 可用的提现方式 |
|---------------|---------------|
| 只充过法币 | 只能法币提现（银行卡/电子钱包） |
| 只充过USDT | 只能USDT提现 |
| 既充过法币又充过USDT | 只能法币提现（以法币通道为准） |
| 法币充值后兑换成BSB再提现 | 还是法币提现（追溯原始充值来源） |

核心逻辑：**资金从哪个通道进来，就从哪个通道出去**。混合充值场景以法币为准。这是合规和反洗钱的要求。

### 4.2 KYC前置校验

- **USDT提现**：不需要KYC（加密货币匿名特性）
- **法币提现（银行卡/电子钱包）**：必须先完成KYC实名认证

KYC需要填写的信息：
- 真实姓名、国家/地区、手机号
- 证件类型（身份证/护照等）、证件号码
- 证件正面照片、证件反面照片

关键规则：KYC通过后，**姓名不可修改**。后续提现时银行卡/电子钱包的户名必须与KYC姓名一致（比对规则：忽略大小写、忽略首尾空格，不要求完全精确匹配字符串）。

### 4.3 银行卡提现

- 首次提现：用户手动填写银行名称、卡号、开户行、户名（自动填充KYC姓名）
- 后续提现：自动回显上次填写的银行卡信息（只读），用户只需输入金额
- 到账时间：T+1（由币种配置中的T+N参数决定）
- 有单笔限额和每日限额（由币种配置决定）

### 4.4 电子钱包提现

- 支持的电子钱包因地区而异：
  - IDR（印尼）：GoPay、OVO、DANA、ShopeePay
  - THB（泰国）：PromptPay
- 到账时间：即时到账
- 同样有限额和手续费规则

### 4.5 BSB提现

- BSB提现时，支持所有提现方式（因为BSB需要先按汇率折算回法币/USDT再出款）
- 页面上额外展示汇率换算信息（如"提现100 BSB ≈ 2,300,000 VND"）
- 实际出款金额按当时汇率折算

### 4.6 提现限制（后台可配）

- 单笔最低/最高金额
- 每日提现总额上限
- 手续费规则（固定金额或百分比）
- T+N到账天数

### 4.7 提现流程（涉及财务模块联动）

1. 用户提交提现申请 → 钱包冻结对应金额（不扣除，先冻结）
2. 订单进入财务模块的多级审核链：风控审核 → 财务审核 → 出款审核
3. 审核全部通过 → 财务调度渠道出款 → 出款成功 → 通知钱包正式扣除冻结金额
4. 如果任一环节拒绝或出款失败 → 通知钱包解冻金额，资金退回用户余额

## 5. 记录模块

### 5.1 记录分类

4个Tab页，分别展示不同类型的记录：

| Tab | 内容 | 状态 |
|-----|------|------|
| 充值记录 | 所有充值订单 | 处理中/成功/失败 |
| 提现记录 | 所有提现订单 | 处理中/审核中/成功/失败 |
| 兑换记录 | 法币/USDT兑换BSB的记录 | 成功/失败 |
| 奖励记录 | 充值奖金、活动奖励等 | 已领取/已使用/已过期 |

### 5.2 核心规则

- 记录**跟随当前币种**：切换到VND，只看VND的充值/提现/兑换记录；切换到BSB，只看BSB的
- 不同状态用不同颜色标记（如：处理中=黄色，成功=绿色，失败=红色）
- 列表字段：订单号、金额、状态、时间、充值/提现方式
- 详情弹窗：在列表基础上增加失败原因（如有）、渠道信息等

### 5.3 页面交互

- 下拉刷新、上拉加载更多
- 点击记录项弹出详情弹窗
- 不同Tab之间可切换

## 6. 非功能性需求

需求文档明确提出的技术要求：
- **账务一致性**：余额变动必须和流水记录严格一致，不能出现余额对不上流水的情况
- **幂等性**：充值回调必须幂等处理，防止第三方支付渠道重复回调导致重复入账
- **KYC强制校验**：非USDT提现必须通过KYC，后端必须校验（不能只靠前端拦截）
- **白名单机制**：支持提现白名单（特殊用户可绕过某些限制）
- **异常检测**：对异常充值/提现行为（短时间大量操作、异常金额等）需有检测和告警机制

# 三、财务管理模块（他人负责，联动参考）

> 定位：B端管理模块，负责支付通道管理、充提记录管理、人工修正、数据统计
> 使用者：B端运营/财务/风控人员
> 他人负责，但与钱包模块存在双向依赖

## 1. 通道配置

### 1.1 什么是通道

通道就是具体的支付渠道，比如"某某银行代收"、"某某电子钱包代付"。每个通道有自己的API接口、费率、限额。

### 1.2 通道分两类

- **代收通道**（Collection）：用于充值，接收用户的钱
- **代付通道**（Payout）：用于提现，把钱付给用户

### 1.3 通道轮询规则

当有多个可用通道时，系统自动选择最合适的通道，选择依据：
- **成功率系数**：历史成功率高的通道优先
- **时间系数**：最近表现好的通道优先
- **告警系数**：有异常告警的通道降权

这是一个自动化的负载均衡+质量选择机制。

### 1.4 配置字段

- 通道名称、通道编码、类型（代收/代付）
- 支持的币种、单笔限额、每日限额
- 手续费（固定/百分比）
- API地址、密钥等接入参数
- 权重（影响轮询选择概率）
- 状态（启用/禁用）

## 2. 支付配置（关键依赖项）

### 2.1 支付配置的作用

支付配置是**钱包C端和通道之间的桥梁**。它定义了"用户在C端看到的充值/提现方式"。

举例：
- 运营在B端创建一条支付配置："BSB - USDT充值"，绑定了2个代收通道
- 用户在C端BSB钱包看到"USDT充值"这个选项，就是这条配置决定的
- 用户点击充值 → 系统根据这条配置找到绑定的通道 → 轮询选择最优通道 → 创建订单

### 2.2 配置字段

- 支付方式名称、图标
- 适用币种、适用端（App/H5/Web）
- 金额范围（最低/最高）
- 快捷金额选项（如 100/500/1000/5000）
- 绑定的通道列表
- 排序权重（影响C端展示顺序）
- 状态（启用/禁用）

### 2.3 与钱包的依赖关系

**钱包C端的"获取可用充值方式"和"获取可用提现方式"接口，数据来源就是财务的支付配置。** 这是最核心的跨模块依赖之一。

## 3. 充值记录管理

### 3.1 充值流程（全链路）

```
用户（C端）→ 钱包服务（创建订单）→ 财务服务（匹配通道、创建渠道订单）
    → 第三方支付渠道（用户支付）→ 渠道回调 → 财务服务（确认）
    → 钱包服务（入账到用户余额）
```

### 3.2 B端充值记录功能

- 列表查询：支持按用户ID、订单号、币种、状态、时间范围等筛选
- 每条记录有两个订单号：平台订单号 + 渠道订单号
- 展示字段包含基准币种（BSB）换算金额，方便统计
- 支持导出（导出需输入密码 + Google Authenticator验证码，安全要求）

## 4. 提现记录管理

### 4.1 提现审核流程（关键）

提现不是用户提交就直接打款，而是经过多级审核：

```
用户提交提现 → 钱包冻结金额 → 进入审核队列
    → 风控审核（第一级）→ 财务审核（第二级）→ 出款审核（第三级）
    → 调度渠道出款 → 渠道回调结果
    → 成功：通知钱包正式扣款
    → 失败：通知钱包解冻退回
```

任何一级审核拒绝 → 直接通知钱包解冻退回，流程终止。

### 4.2 B端提现记录功能

- 列表查询：同充值记录，多种筛选条件
- 审核操作：不同角色看到不同审核按钮（风控/财务/出款）
- 锁定/解锁：审核人可以锁定某条记录，防止其他人同时审核（防并发）
- 导出：同样需要安全验证

## 5. 人工修正

### 5.1 为什么需要人工修正

三种场景，对应三种操作：

| 操作 | 场景 | 订单号格式 | 说明 |
|------|------|-----------|------|
| 补单 | 用户实际付款了但系统没收到回调，导致没入账 | B + 16位数字 | 人工确认后补录入账 |
| 加款 | 活动补偿、客诉赔付、系统误差补偿等 | A + 16位数字 | 给用户指定子钱包加钱 |
| 减款 | 用户违规、误操作退回、财务调整等 | M + 16位数字 | 从用户指定子钱包扣钱 |

### 5.2 操作对钱包的影响

这三种操作都**直接操作用户的钱包余额**，而且可以指定操作哪个子钱包（中心/奖励/代理/主播）。

这意味着：财务模块的人工修正功能，需要调用钱包模块的RPC接口来执行余额变动。这是重要的跨模块依赖。

### 5.3 审核机制

三种操作都需要审核：创建人提交 → 审核人审核（通过/拒绝）→ 通过后执行余额操作。创建人和审核人不能是同一个人。

## 6. 数据统计

### 6.1 三类统计

- **通道统计**：每个通道的交易笔数、金额、成功率、手续费等
- **充值统计**：按时间段/币种/方式等维度汇总充值数据
- **提现统计**：按时间段/币种/方式等维度汇总提现数据

### 6.2 时间处理规则

- 数据存储统一使用 **UTC+0** 时间
- 前端展示根据站点所在时区转换显示（支持IANA时区格式）
- 统计卡片展示汇总数字，下方明细表展示逐条数据

## 2. 钱包 ↔ 财务（双向依赖）

### 钱包 → 财务

| 钱包需要什么 | 财务提供什么 |
|-------------|-------------|
| C端展示可用充值方式 | 支付配置（已启用的充值方式列表） |

# 五、需求中的关键业务规则汇总

> 这些是"绕不过的"确定性骨架，不管细节怎么调，这些规则大概率不会变

1. **4子钱包 × 多币种 = 余额矩阵**：这是数据模型的基础
2. **币种切换 = 整体上下文切换**：不存在跨币种混合展示
3. **BSB作为平台基准币种**：统一计量、换算、统计
4. **充值方式由B端支付配置决定**：C端是消费方，不能硬编码
5. **提现方式由充值来源决定**：资金通道追溯，合规要求
6. **非USDT提现必须KYC**：后端强制校验
7. **提现户名 = KYC姓名**：一致性校验（宽松比对）
8. **充值重复转账三级拦截**：1笔提醒/2笔警告/3笔拦截
9. **奖金进奖励钱包 + X倍流水限制**：防薅羊毛
10. **兑换单向（法币→BSB）+ 赠送也有流水限制**
11. **提现先冻结后扣款**：多级审核通过才正式扣款
12. **人工修正可指定子钱包操作**：加款/减款/补单都需审核
13. **汇率平台手动配置**：不接外部实时汇率
14. **充值回调幂等**：防止重复入账
15. **数据统计UTC+0存储**：前端按站点时区展示
