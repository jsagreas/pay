## 一、我要做什么（职责定位）

**主责两个模块：**
- **多币种钱包** — 平台资金体系的核心中枢，管理用户的钱
- **币种配置** — 服务于钱包的基础配置，管理币种和汇率

**参考一个模块：**
- **财务管理** — 他人负责，但和钱包有双向依赖，必须清楚边界和联动方式

**一句话概括：** 用户充钱进来、换币、花钱、提钱，这一整套资金进出的后端逻辑是我的活。通道怎么配、订单怎么审、统计怎么出，是财务那边的活。但钱最终进出用户钱包，都要经过我这里。

## 二、钱包体系设计（我理解的核心骨架）

### 2.1 钱包长什么样

用户注册后拥有多个币种的钱包，每个币种互相独立，互不影响。

```
用户张三的钱包
├── BSB钱包（平台币）
│   ├── 中心钱包: BSB 800.00      ← 充值/投注返奖进这里，消费/提现从这里扣
│   ├── 奖励钱包: BSB 200.00      ← 活动/任务奖金进这里，完成流水后可转入中心
│   ├── 主播钱包: BSB 500.00      ← 直播收益进这里，可提现或转入中心
│   ├── 代理钱包: (预留)          ← 当前版本暂不做
│   └── 场馆钱包: (预留)          ← 当前版本暂不做
├── VND钱包（越南盾）
│   ├── 中心钱包: d 50,000,000
│   ├── 奖励钱包: d 5,000,000
│   └── ...
├── IDR钱包（印尼盾）
│   └── ...
└── THB钱包（泰铢）
    └── ...
```

**依据：** 币种配置需求文档第2章"资金钱包体系"、钱包需求文档第4章"钱包总体设计"

### 2.2 几个绕不过的核心规则

| 规则 | 具体内容 | 依据 |
|------|---------|------|
| 币种隔离 | 每个币种一套独立余额和账本，不支持跨币种混合展示和操作 | 需求4.1、4.2 |
| 余额计算 | 资金钱包余额 = 中心钱包 + 奖励钱包（主播/代理/场馆不算在内） | 币种配置2.1余额规则 |
| 消费顺序 | 先扣中心钱包，再扣奖励钱包 | 币种配置2.1余额规则 |
| 混合投注 | 中心+奖励混合花的时候，返奖要按投注金额占比分开算 | 币种配置2.1余额规则 |
| BSB锚定 | 10 BSB = 1 USDT，BSB汇率锚定USDT | 币种配置3.2 |
| 基准币种 | USDT，所有币种在账务层需换算为基准币种统计，设置一次不可改 | 币种配置3.2 |

### 2.3 资金流向（钱从哪来、到哪去）

```
钱进来的路径：
  充值（法币/USDT）       ──→ 中心钱包
  投注返奖                ──→ 中心钱包
  活动/任务返奖           ──→ 奖励钱包
  直播礼物收益            ──→ 主播钱包
  代理佣金                ──→ 代理钱包（预留）

钱出去的路径：
  消费（投注/礼物/道具）  ←── 中心钱包优先，不够再扣奖励钱包
  提现                    ←── 中心钱包（完成稽核流水后）
  提现                    ←── 主播钱包（可直接提现）

钱包之间流转：
  奖励钱包 ──完成稽核流水──→ 中心钱包
  主播钱包 ──可选转入────→ 中心钱包
  代理钱包 ──可选转入────→ 中心钱包（转入后需稽核）
```

**依据：** 币种配置需求文档2.1"资金流向"

## 三、充值（用户把钱充进来）

### 3.1 三种充值方式

| 方式 | 适用场景 | 承载页 | 到账机制 | 后端关注点 |
|------|---------|--------|---------|-----------|
| USDT充值 | 所有币种（有USDT支付方式时） | **平台自己的页面**（展示收款地址+二维码） | 区块链监听到账 | 需要和链上监听服务对接，回调幂等 |
| 银行转账 | 法币钱包（越南/印尼/泰国银行） | **三方页面**（跳转到三方） | 三方支付通道回调 | 创建订单→调三方→等回调→更新状态 |
| 扫码支付 | IDR(QRIS)/THB(PromptPay) | **三方页面**（二维码+倒计时） | 三方支付通道回调 | 同银行转账，多了倒计时过期处理 |

**关键细节：**
- 游客也能用USDT充值（需求5.2），但法币充值需要登录（原型2-3温馨提示）
- 每个币种有默认充值方式和默认最小档位（需求5.1规则1）
- 只有BSB充值时才展示汇率和换算金额（需求5.1规则4）
- 充值档位由**财务模块的支付配置**决定，不是钱包自己配的

**依据：** 钱包需求文档第5章、原型3系列/4/5系列/7系列

### 3.2 重复转账拦截

**场景：** 防止用户短时间内重复提交同样的充值。

**规则：** 同一用户 + 同一币种 + 同一金额，看有几笔"待支付"或"进行中"的订单：
- 1-2笔 → 弹窗提醒但允许继续
- 3笔 → 强制拦截，不让创建新订单

**适用范围：** 银行转账、扫码支付。USDT链上充值**不适用**（因为链上转账机制不同）。

**后端关注点：** 需要查询用户当前待处理订单数量，这个查询需要考虑性能（高频操作）。

**依据：** 钱包需求文档5.5、原型7-3/7-4

### 3.3 额外奖金选择

**场景：** 充值时可以选一个活动奖金，充值金额会有加成，但加成部分要完成流水要求才能提现。

**规则要点：**
- 一笔充值只能关联一个奖金活动（互斥）
- 奖金比例和充值金额联动（如选50%奖金，充1000，额外+500）
- 有最低充值门槛校验
- 奖金活动列表从**活动模块**获取，不是钱包自己管的

**后端关注点：** 钱包需要调用活动模块获取可用奖金列表，创建充值订单时记录关联的活动ID和加成金额。奖金到账进**奖励钱包**，需要记录稽核流水倍数要求。

**依据：** 钱包需求文档5.6、原型6系列

## 四、兑换（法币换成平台币BSB）

### 4.1 兑换规则

- **方向：** 单向，只能法币→BSB，不能反过来
- **入口：** 只有法币钱包（VND/IDR/THB）有兑换入口，BSB钱包没有
- **汇率展示：** 统一口径 1 BSB = X 法币（如 1 BSB = d2,620）
- **赠送：** 兑换有赠送比例（如10%），赠送部分需完成X倍流水

### 4.2 计算逻辑

```
输入：用户输入法币金额（不能超过当前余额）
兑换获得BSB = 法币金额 / 汇率
赠送BSB = 兑换获得BSB × 赠送比例
实际到账BSB = 兑换获得 + 赠送

兑换获得 → 中心钱包
赠送部分 → 奖励钱包（需完成流水要求）
```

**后端关注点：**
- 汇率从币种配置模块获取
- 扣减法币中心钱包余额，增加BSB中心+奖励钱包余额
- 需要在同一事务中完成，保证资金一致
- 赠送比例和流水倍数的配置来源待确认（可能是运营活动配置）

**依据：** 钱包需求文档第6章、原型8

## 五、提现（用户把钱提出去）

### 5.1 KYC与提现的关系

```
用户点击提现
  │
  ├── 已KYC认证 → 直接进入提现方式选择页（全渠道可用）
  │
  └── 未KYC认证 → 弹窗提示
        ├── "去认证" → 跳转KYC页面（认证后解锁全渠道）
        └── "继续提现" → 仅限USDT提现
```

**一句话：** KYC不强制，但不做就只能提USDT。

**依据：** 钱包需求文档7.1第1点、原型9系列

### 5.2 提现规则（充什么提什么）

| 用户充值方式 | 可用提现方式 | 逻辑 |
|-------------|-------------|------|
| 只充过法币 | 法币提现 | 充什么提什么 |
| 只充过USDT | USDT提现 | 充什么提什么 |
| 法币+USDT都充过 | 法币提现 | 混合充值只能提法币 |
| 法币→兑换成BSB | 法币提现 | 源头是法币就提法币 |

**后端关注点：** 需要追踪用户的充值来源记录，判断允许哪种提现方式。这是一个**决策逻辑**，后端必须实现。

**依据：** 钱包需求文档7.1第2点

### 5.3 三种提现方式

| 方式 | 需要填什么（首次） | 后续 | 特殊说明 |
|------|------------------|------|---------|
| 银行卡 | 银行名称、账号、持有人（KYC名/置灰）、开户网点、金额 | 账户信息只读，只填金额 | 越南/印尼/泰国各有对应银行 |
| 电子钱包 | 钱包类型、手机号、持有人（KYC名/置灰）、金额 | 同上 | GoPay/OVO/DANA/PromptPay等 |
| BSB提现 | 选择提现到USDT/银行/电子钱包，填对应信息 | 同上 | BSB提现时需展示汇率换算 |

**铁律：** 账户持有人姓名 = KYC认证姓名，自动填充且不可编辑。姓名校验忽略大小写和多余空格。

**后端关注点：**
- 提现账户信息需要持久化（首次填写后平台记录，后续复用）
- 提现时需冻结对应金额（防止余额不足时还能提现）
- 提现限制校验：单笔限额、单日限额、手续费计算
- 创建提现订单后，后续的审核流程（风控→财务→出款）是**财务模块**的事

**依据：** 钱包需求文档7.2~7.6、原型10系列

## 六、交易记录（用户查看自己的流水）

### 6.1 四种记录类型

| Tab | 列表字段 | 详情内容 | 说明 |
|-----|---------|---------|------|
| 提现 | 金额、方式、时间、状态 | 订单号、金额、状态、时间、方式、失败原因 | 状态：已完成/审核中/失败 |
| 充值 | 金额、方式、时间、状态 | 同上 | 状态：已完成/审核中/失败 |
| 兑换 | 兑换金额、赠送%、实际到账、时间 | — | 无详情弹窗（原型未展示） |
| 奖励 | 奖励金额、奖励类别、时间 | — | 无详情弹窗（原型未展示） |

### 6.2 核心规则

- **币种隔离：** 当前选中什么币种，就只看这个币种的记录，不混合
- **失败原因：** 充值失败（支付超时/支付异常/卡片问题）、提现失败（信息有误/系统维护/额度受限）
- 详情以弹窗形式展示，失败状态提供"更换其他方式"快捷入口

**后端关注点：** 充值/提现记录的数据来源可能在财务模块（因为订单归财务管），钱包可能只是提供查询接口转发，或者钱包自己也存一份流水。兑换/奖励记录是钱包自己的数据。

**依据：** 钱包需求文档第8章、原型11系列

## 七、币种配置（B端后台管理）

### 7.1 币种管理

**三类币种：**
- 法定货币（IDR/VND/THB）— 各国官方货币
- 加密货币（USDT/BTC/ETH）— 区块链数字资产
- 平台货币（BSB）— 平台自己的币

**管理方式：**
- 币种数据由**开发导入**（不是后台新建），后台只能编辑部分属性
- 可编辑项：图标（SVG/PNG≤5k）、汇率浮动%、阈值%、启用/禁用
- 基准币种设置一次后不可更改（推荐USDT），提交时需二级确认
- 禁用币种后C端不可使用，有余额的用户需弹提示

**后台页面位置：** 系统管理 > 币种配置

**后端关注点：**
- 币种数据初始化需要开发写导入脚本
- 每个币种唯一（币种代码不可重复）
- 禁用前需检查是否有用户持有该币种余额
- 操作需记录审计日志（调 ser-blog）

**依据：** 币种配置需求文档第3章、原型-币种配置1/2

### 7.2 汇率体系（核心，必须搞清楚）

**四种汇率的关系：**

```
三方API_1 ─┐
三方API_2 ─┼─→ 取平均 ──→ 实时汇率
三方API_3 ─┘                 │
                              │ 偏差值 = |实时 - 平台| / 实时 × 100%
                              │
                    偏差值 > 阈值？
                   /            \
                 是              否
                 │               │
         更新平台汇率      平台汇率不变
         生成汇率日志
                 │
          ┌──────┴──────┐
          │             │
   入款汇率            出款汇率
= 实时×(1+浮动%)    = 实时×(1-浮动%)
   用于充值            用于提现
```

**举例说明：**
- 实时汇率 1 USDT = 15,800 VND
- 汇率浮动 2%
- 入款汇率 = 15,800 × 1.02 = 16,116 VND（用户充值时用这个，对用户不利，平台多收一点）
- 出款汇率 = 15,800 × 0.98 = 15,484 VND（用户提现时用这个，对用户不利，平台少给一点）
- 平台通过入款/出款之间的汇率差赚取利润

**精度要求：**
- 法定货币：小数点后2位
- 加密货币：小数点后8位
- 不足补0，超过四舍五入

**汇率日志：** 每次平台汇率发生变更时自动生成一条日志，记录更新时间、币种、各种汇率值。B端在操作日志页新增"汇率日志"Tab查看。

**后端关注点：**
- 需要对接3家三方汇率API，定时拉取并取平均
- 偏差值计算和阈值判断是自动化逻辑（定时任务或者Hook触发）
- 汇率更新需要原子操作（平台汇率+入款+出款同时更新）
- 汇率数据需要缓存（C端高频读取）

**依据：** 币种配置需求文档第4章、原型-币种配置3

## 八、财务管理（他人负责，我需要清楚的部分）

### 8.1 和我直接相关的模块

**支付配置（财务配的，但数据是C端钱包要用的）：**
- 充值方式：每个币种下有哪些充值方式、快捷金额档位、金额范围、关联哪个通道 — C端充值页展示用
- 提现方式：每个币种下有哪些提现方式、费率、大额费率、关联通道 — C端提现页展示用

**人工修正（财务操作，但最终要动我的钱包余额）：**
- 充值补单（B+）— 修正充值金额，需要调我的上账接口
- 人工加款（A+）— 运营补偿/奖励/修正/测试，需要调我的加款接口，可以同时给4种钱包加款
- 人工减款（M+）— 异常追回/处罚/冲正/调整，需要调我的减款接口

**加款特殊逻辑：** 如果用户名下没有该币种的钱包，系统需**自动创建**。
**减款特殊逻辑：** 余额不足时，从**最高金额**的钱包开始依次扣减，实际减款可能小于设定金额。
**审核：** 所有人工修正均需**双人审核**，不可自审自批。

### 8.2 充值/提现的完整流程（三方协作）

**充值流程：**
```
C端用户 ──→ ser-wallet（选币种/方式/金额/奖金）
               │
               ├─→ 调财务模块（轮询匹配通道 + 创建平台订单）
               │        │
               │        └─→ 调三方通道（创建通道订单）
               │                │
               │      三方回调 ←─┘
               │        │
               │   财务更新订单状态
               │        │
               └── 财务调 ser-wallet 上账（充值成功 → 中心钱包入账）
```

**提现流程：**
```
C端用户 ──→ ser-wallet（选方式/填账户/输金额）
               │
               ├─→ 校验KYC状态（调 ser-kyc）
               ├─→ 校验提现限额
               ├─→ 冻结钱包余额
               │
               └─→ 调财务模块（创建提现订单，状态=待风控审核）
                        │
                   风控审核 → 财务审核 → 出款
                        │
                   ┌────┴────┐
                 成功       驳回
                   │         │
             财务调wallet  财务调wallet
             确认扣款      解冻退回
```

### 8.3 我需要提供给财务的能力（RPC接口）

| 能力 | 什么时候财务会调 |
|------|----------------|
| 上账（入中心钱包） | 充值成功 |
| 冻结余额 | 提现申请（也可能是钱包自己做，看设计） |
| 解冻退回 | 提现驳回 |
| 确认扣款 | 提现出款成功 |
| 人工加款 | A+审核通过 |
| 人工减款 | M+审核通过 |
| 补单修正 | B+审核通过 |
| 查询余额 | 审核时查看用户钱包 |

**依据：** 财务管理需求文档2.3~2.5、充值/提现流程泳道图

```
      │  │        │  │
      │  │        │  └──→ ser-kyc          KYC认证状态查询
      │  │        └─────→ 财务模块         创建提现订单
      │  └──────────────→ 活动模块         获取可用奖金
      └─────────────────→ 财务模块         获取充值方式、创建充值订单

                          ← 财务模块       充值成功上账
                          ← 财务模块       提现成功/驳回
                          ← 财务模块       人工加减款/补单
                          ← 投注模块       投注扣款、返奖入账
                          ← 直播模块       主播收入入账
                          ← 活动模块       活动/任务返奖
```

## 十、后端 vs 前端职责边界（我的理解）

### 后端必须做的

| 事项 | 说明 |
|------|------|
| 钱包余额管理 | 所有余额增减必须后端控制，前端只展示 |
| 充值订单创建 | 生成订单号、匹配通道、记录订单 |
| 重复转账检测 | 查询待处理订单数量，返回是否拦截 |
| 提现规则判断 | 根据充值历史判断允许的提现方式 |
| 提现限额校验 | 单笔/单日限额检查 |
| 冻结/解冻/扣款 | 提现全流程的余额操作 |
| 兑换计算和执行 | 汇率取值、金额计算、余额变更 |
| KYC状态查询 | 调 ser-kyc 确认认证状态 |
| 汇率体系 | 三方API拉取、偏差计算、平台汇率更新 |
| 币种配置管理 | 币种CRUD、基准币种设置 |
| 提现账户存储 | 银行卡/电子钱包信息的持久化和读取 |
| 对账一致性 | 账务流水与账户余额的强一致校验 |
| 幂等处理 | 充值回调防重复入账 |

### 前端做的（后端不用管）

| 事项 | 说明 |
|------|------|
| 币种切换UI | 切换动画、记住上次选择（或后端存储用户偏好） |
| 金额展示格式 | 千分位/小数位渲染（后端提供规则，前端负责渲染） |
| 快捷金额档位展示 | 后端返回档位列表，前端渲染标签 |
| 奖金角标展示 | 如"+50%"标签，后端返回比例，前端渲染 |
| 弹窗/浮窗交互 | KYC提示弹窗、重复充值弹窗、提现状态弹窗等 |
| 表单校验 | 金额范围、必填项检查（后端也要校验，前端做一层预校验） |
| 页面跳转/导航 | 个人中心→充值→三方页面→回调页面 |

### 需要前后端协商的

| 事项 | 说明 |
|------|------|
| 币种偏好持久化 | 用户上次选中的币种，是前端本地存还是后端存 |
| 充值方式档位切换 | 切换充值方式时档位变化，是前端本地计算还是每次调接口 |
| 提现账户信息变更 | "可更改提现信息"的具体交互方式 |
| 充值订单状态轮询 | 创建订单后等待回调的过程，前端轮询频率和方式 |

## 十一、非功能性要求（后端必须考虑的）

| 要求 | 具体内容 | 依据 |
|------|---------|------|
| 对账一致性 | 账务流水和账户余额必须严格一致，任何资金操作都要记流水 | 需求第9章 |
| 幂等性 | 充值回调可能重复到达，必须防重复入账 | 需求第9章 |
| KYC强制校验 | 法币/电子钱包提现前必须通过KYC | 需求第10章 |
| 姓名强一致 | 提现账户持有人必须和KYC姓名一致 | 需求7.1第6点、原型10-3注释 |
| 大额/异常审核 | 大额或异常频率交易触发人工审核 | 需求第10章 |
| 双人审核 | 人工修正操作不可自审自批 | 财务需求2.5 |
