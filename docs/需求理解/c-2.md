# 第一部分：多币种钱包模块（自己负责，C端为主）

## 1. 做这个东西的背景

平台是一个面向多国市场（越南、印尼、泰国等）的泛娱乐+竞彩产品。用户来自不同国家，使用不同货币。之前可能是单一货币体系，现在需要支持多币种，让每个地区的用户用自己熟悉的货币充值、消费、提现，同时平台内部用统一的平台币（BSB）来结算。

简单说：用户可以用本国货币充钱进来，在平台里花（以BSB为单位），也可以把钱提出去。

## 2. 钱包体系设计

### 2.1 币种体系

平台支持三类货币：

| 类型 | 具体币种 | 用途 |
|------|----------|------|
| 平台币 | BSB | 平台内消费的统一货币，10 BSB = 1 USDT（锚定汇率） |
| 法币 | VND（越南盾）、IDR（印尼卢比）、THB（泰铢） | 各地区用户充值/提现用 |
| 加密货币 | USDT | 跨地区通用充提，支持 TRC20/ERC20/BEP20 三条链 |

### 2.2 钱包结构

每个用户在每个币种下有独立的钱包体系：

| 钱包类型 | 说明 | 当前状态 |
|----------|------|----------|
| 中心钱包 | 主钱包，充值直接进这里，消费优先扣这里 | 核心，必做 |
| 奖励钱包 | 奖金/活动奖励进这里，需要完成稽核（X倍流水）才能转到中心钱包 | 核心，必做 |
| 主播钱包 | 主播收入钱包 | 需做 |
| 代理钱包 | 代理佣金钱包 | 预留，本期不做 |
| 场馆钱包 | 场馆结算钱包 | 预留，本期不做 |

**关键规则：**
- 资金钱包余额 = 中心钱包 + 奖励钱包
- 消费扣款顺序：先扣中心钱包，不够再扣奖励钱包
- 每个币种的钱包是独立的，切换币种 = 切换整套钱包和数据
- 用户前端显示的余额是当前选中币种的资金钱包余额

### 2.3 币种切换逻辑

- 用户在个人中心可以切换当前操作币种
- 切换后，充值、提现、兑换、记录等所有页面的数据都跟随变化
- 当前币种是所有操作的"单一数据源"，全局生效

## 3. 充值模块

### 3.1 充值方式

不同币种对应不同的充值方式：

| 币种 | 充值方式 |
|------|----------|
| USDT | 链上转账（选择 TRC20/ERC20/BEP20 网络，展示充值地址+二维码） |
| VND | 银行转账（展示收款银行信息） |
| IDR | QRIS 扫码支付 |
| THB | PromptPay 扫码支付 |
| BSB | 不支持直接充值，只能通过法币兑换获得 |

### 3.2 充值流程（说人话版）

1. 用户进入充值页，看到当前币种下可用的充值方式列表
2. 选一个充值方式，输入金额
3. 如果有奖金活动可选，用户可以勾选一个（有最低充值门槛）
4. 提交后，后台匹配一个可用的支付通道（轮询匹配，按权重分配）
5. 如果匹配不到 → 提示"通道繁忙，请换一种方式"
6. 匹配到了 → 创建订单（状态：待支付），跳转到支付页面
7. 用户完成支付 → 通道回调通知后台 → 充值成功，钱进入用户的中心钱包
8. 支付失败 → 订单标记为"支付失败"

### 3.3 重复转账拦截

防止用户重复提交相同金额的充值：
- 检查条件：同一个用户 + 同一个币种 + 同一个金额
- 1-2笔待处理订单：弹窗警告，用户可确认继续
- 3笔及以上待处理订单：直接阻断，不允许提交

### 3.4 奖金活动选择

充值时可以选择参与奖金活动：
- 活动列表由运营配置（活动名称、奖金比例、最低充值门槛、稽核倍数）
- 用户充值金额 ≥ 最低门槛才能选择该活动
- 选择后，奖金进入奖励钱包，需要完成对应流水要求后才能转入中心钱包
- 一笔充值只能选一个活动

## 4. 提现模块

### 4.1 提现方式

| 币种 | 提现方式 | KYC要求 |
|------|----------|---------|
| USDT | 提供USDT地址（选择网络） | 可选（不强制） |
| VND/IDR/THB | 银行卡转账 | 必须完成KYC |
| VND/IDR/THB | 电子钱包（GoPay/OVO/DANA/ShopeePay/PromptPay） | 必须完成KYC |

### 4.2 提现规则（资金来源决定提现方式）

这是一个重要的风控规则：
- 纯法币充值的用户 → 只能法币提现
- 纯USDT充值的用户 → 只能USDT提现
- 混合充值（既充过法币又充过USDT）的用户 → 只能法币提现

### 4.3 提现限额

| 维度 | 说明 |
|------|------|
| 单笔限额 | 每笔提现的最小/最大金额限制 |
| 每日限额 | 每天提现总额上限 |
| 手续费 | 平台手续费 + 通道手续费 |
| 到账时间 | T+N（具体N待配置） |

### 4.4 提现流程（说人话版）

1. 用户进入提现页，系统先检查KYC状态
2. 未完成KYC且提现方式要求KYC → 引导去做认证
3. 选择提现方式，输入金额和收款信息（银行卡号/电子钱包账号/USDT地址）
4. 系统校验：余额是否足够、是否超限额、资金来源规则是否满足
5. 提交后：冻结对应金额，创建提现订单
6. 订单进入三级审核（财务管理模块负责）：风控审核 → 财务审核 → 出款审核
7. 全部审核通过 → 通过通道出款 → 成功则扣除用户余额
8. 任一环节拒绝 → 解冻金额，退回用户钱包

## 5. 兑换模块

### 5.1 兑换规则

- 方向：**法币 → BSB**（单向兑换，BSB不能换回法币）
- 支持：VND → BSB、IDR → BSB、THB → BSB
- 汇率：使用平台汇率（由币种配置模块管理）
- 兑换奖金：兑换可获得10%额外BSB奖金，但奖金部分进入奖励钱包，需完成X倍流水

### 5.2 兑换流程

1. 用户选择法币币种，输入兑换金额
2. 页面实时显示汇率和预计到账BSB数量
3. 确认兑换 → 扣法币中心钱包 → 加BSB中心钱包 + 奖金进BSB奖励钱包

## 6. 记录模块

用户可以在记录页查看四类交易历史：

| 记录类型 | 展示内容 |
|----------|----------|
| 充值记录 | 订单号、充值方式、金额、状态（待支付/已完成/支付失败）、时间 |
| 提现记录 | 订单号、提现方式、金额、状态（审核中/已完成/已拒绝）、时间 |
| 兑换记录 | 源币种、目标币种、汇率、兑换金额、到账金额、时间 |
| 奖励记录 | 奖励来源（充值奖金/兑换奖金/活动奖励）、金额、稽核状态、时间 |

## 7. 产品原型关键页面（32张原型图总结）

| 页面 | 核心交互 |
|------|----------|
| 个人中心 | 显示当前币种余额、币种切换入口、游客态提示登录 |

## 8. 非功能要求

| 要求 | 说明 |
|------|------|
| 幂等性 | 同一笔充值/提现/兑换不能重复处理 |
| 数据一致性 | 余额变动必须准确，不能出现多扣少扣 |
| 风控 | 重复转账拦截、提现资金来源规则、KYC校验 |

# 第二部分：币种配置模块（自己负责，B端后台）

## 1. 做这个东西的背景

钱包作为平台的核心资产能力，需要后台有一套管理系统来配置支持哪些币种、汇率怎么算、浮动比例多少。这个模块面向运营/管理员使用，不面向普通用户。

## 2. 钱包系统结构定义

### 2.1 基准币种

- 基准币种为 **USDT**
- 配置一次后不可变更（选定后锁死）
- 所有其他币种的汇率都以USDT为基准换算
- BSB与USDT的锚定关系：10 BSB = 1 USDT

### 2.2 币种分类

| 分类 | 具体币种 | 特征 |
|------|----------|------|
| 法币 | VND、IDR、THB | 各国法定货币，面向地区用户 |
| 加密货币 | USDT、BTC、ETH | 链上资产（本期重点USDT） |
| 平台币 | BSB | 平台内部结算货币 |

## 3. 汇率体系

这是币种配置模块最核心的部分：

### 3.1 四层汇率

| 汇率层级 | 来源/计算方式 | 用途 |
|----------|-------------|------|
| 实时汇率 | 从3个第三方API获取，取平均值 | 作为所有汇率计算的基础 |
| 平台汇率 | 基于实时汇率，但不是每次都更新，只有偏差超过阈值才更新 | 对外展示的参考汇率 |
| 充值汇率 | 实时汇率 × (1 + 浮动百分比) | 用户充值时使用（平台多收一点） |
| 提现汇率 | 实时汇率 × (1 - 浮动百分比) | 用户提现时使用（平台少给一点） |

### 3.2 平台汇率更新机制

不是每次实时汇率变动都更新平台汇率，而是：
- 计算偏差：|实时汇率 - 平台汇率| / 实时汇率 × 100%
- 如果偏差 > 配置的阈值 → 触发更新平台汇率
- 更新时记录变更日志

这样做的目的是避免汇率频繁波动导致用户体验差。

### 3.3 浮动比例

运营可以为每个币种配置充值/提现的浮动百分比，这就是平台在汇率上的利润空间。

## 4. B端后台配置页面

### 4.1 币种配置列表页

- 展示所有已配置的币种（名称、简称、icon、汇率浮动比例、状态）
- 支持新增、编辑、启用/禁用操作

### 4.2 币种编辑弹窗

- 币种名称、简称、icon上传
- 充值浮动比例、提现浮动比例
- 汇率更新阈值
- 状态（启用/禁用）

### 4.3 汇率变更日志页

- 记录每次平台汇率变更的时间、旧值、新值、触发原因
- 用于审计和问题排查

## 5. 定时任务

| 任务 | 频率 | 说明 |
|------|------|------|
| 拉取第三方汇率 | 待配置（分钟级） | 从3个API获取汇率，取平均值更新实时汇率 |
| 检查阈值触发 | 伴随汇率拉取 | 实时汇率更新后检查是否需要更新平台汇率 |

# 第三部分：财务管理模块（他人负责，理解依赖）

> 这部分不是自己做的，但因为和钱包模块有大量交互，必须清楚它的结构和接口

## 1. 模块总览

财务管理分为两大板块、8个子模块：

```
财务管理
├── 通道配置
│   ├── 代收通道（收钱的通道）
│   ├── 代付通道（出钱的通道）
│   └── 通道轮询规则（按权重分配流量）
├── 支付配置
│   ├── 充值方式（用户看到的充值选项）
│   └── 提现方式（用户看到的提现选项）
├── 充值记录（充值订单管理 + 导出）
├── 提现记录（提现订单管理 + 三级审核 + 导出）
├── 人工修正
│   ├── 充值补单（漏单处理）
│   ├── 人工加款（给用户加钱）
│   └── 人工减款（给用户扣钱）
└── 数据统计
    ├── 代收通道统计
    ├── 代付通道统计
    ├── 充值统计
    └── 提现统计
```

## 2. 通道配置（与钱包直接相关）

### 2.1 代收通道（用来收用户充值的钱）

每个代收通道有这些关键属性：
- 通道名称（创建后不可改）、通道标识、币种
- 费率（千分比）、最小/最大金额限制
- 关联的代付通道、日交易上限
- 状态（启用/禁用）

**与钱包的关系：** 用户充值时，钱包调用财务模块的轮询接口，财务模块根据权重分配一个代收通道给这笔订单。

### 2.2 代付通道（用来给用户出款的通道）

与代收类似，额外多了：
- 中间钱包余额（加密货币场景下的中转钱包余额）
- 半自动出款标记（是否需要人工确认出款）

**与钱包的关系：** 用户提现时，审核通过后财务模块通过代付通道向用户出款。

### 2.3 通道轮询规则

- 本期支持**按权重轮询**
- 不同币种有独立的轮询规则
- 权重不需要加起来等于100，只看比例（如A=7, B=3 → 70%:30%）
- 只有启用状态的通道参与轮询
- 轮询规则被禁用 = 没有可用通道，走默认策略

## 3. 支付配置（决定用户看到什么充值/提现方式）

### 3.1 充值方式

运营在后台配置用户端看到的充值选项：
- 每个充值方式关联一个代收通道
- 配置：名称、方式类型、币种、最小/最大金额、icon、排序
- 排序值越小越靠前

**与钱包的关系：** 钱包模块的C端"获取充值方式列表"，数据来源就是这里的配置。

### 3.2 提现方式

类似充值方式：
- 每个提现方式关联一个代付通道
- 配置：名称、方式类型、币种、icon、排序

## 4. 充值记录

B端管理员查看所有用户的充值订单：
- 搜索：订单号、三方订单号、用户UID、订单状态、充值方式
- 展示：18个字段（订单号、三方单号、用户、通道、币种、金额、实际到账、手续费、状态、时间等）
- 支持导出（需密码 + 可选谷歌验证码）

## 6. 人工修正（与钱包RPC强关联）

这三个操作都会调用钱包模块的RPC来操作用户余额：

### 6.1 充值补单
- 场景：用户充值了但系统没收到回调，运营手动补单
- 订单号格式：B + 16位随机数
- 操作：输入原充值订单号查询 → 确认补单金额 → 审核通过 → 调钱包RPC给用户加钱
- 约束：同一订单只能补一次，创建人 ≠ 审核人

### 6.2 人工加款
- 场景：赠送奖金、活动补偿、客诉赔付等
- 订单号格式：A + 16位随机数
- 操作：选类型 → 指定用户 → 输入金额和原因 → 审核通过 → 调钱包RPC加款
- 约束：金额只能正数，必须填备注，创建人 ≠ 审核人

### 6.3 人工减款
- 场景：纠正错误加款、违规处罚等
- 订单号格式：M + 16位随机数
- 操作：选类型 → 指定用户 → 输入金额和原因 → 审核通过 → 调钱包RPC扣款
- 约束：余额不足时提示不能减款，创建人 ≠ 审核人

## 2. 核心业务链路

### 链路一：充值（用户充钱进来）

```
用户选币种 → 钱包获取充值方式(←财务) → 输入金额 → 检查重复转账
→ 选奖金活动(可选) → 钱包请求匹配通道(→财务) → 财务创建通道订单
→ 返回支付页面 → 用户支付 → 通道回调(→财务) → 财务通知钱包上分(→Credit RPC)
→ 钱到中心钱包 + 奖金到奖励钱包
```

### 链路二：提现（用户把钱提出去）

```
用户选币种 → 检查KYC(←KYC服务) → 钱包获取提现方式(←财务)
→ 输入金额+收款信息 → 校验(余额/限额/资金来源规则) → 冻结金额
→ 钱包提交提现(→财务) → 三级审核(风控→财务→出款)
→ 通过 → 财务出款(→通道) → 成功 → 财务通知钱包扣款(→Debit RPC)
→ 拒绝 → 财务通知钱包解冻(→Unfreeze RPC)
```

### 链路三：兑换（法币换BSB）

```
用户选法币 → 获取平台汇率(←币种配置,同服务内部调用)
→ 输入金额 → 计算BSB到账数(含10%奖金) → 确认
→ 扣法币中心钱包 → 加BSB中心钱包 + 奖金进BSB奖励钱包
```

### 链路四：人工修正（管理员操作用户余额）

```
管理员在财务后台操作 → 创建加款/减款/补单 → 另一管理员审核
→ 审核通过 → 财务调用钱包RPC(Credit/Debit) → 用户余额变更
```

## 3. 钱包需要暴露给财务的RPC接口

| 接口 | 谁调用 | 什么时候调 |
|------|--------|-----------|
| QueryBalance | 财务 | 人工加减款前查余额、B端展示 |
| Credit（加款） | 财务 | 充值成功、补单通过、人工加款通过 |
| Debit（扣款） | 财务 | 提现完成、人工减款通过 |
| Freeze（冻结） | 财务 | 提现订单创建时 |
| Unfreeze（解冻） | 财务 | 提现被拒绝/取消时 |

# 第五部分：关键业务规则速查

| 规则 | 内容 |
|------|------|
| BSB锚定 | 10 BSB = 1 USDT，不变 |
| 基准币种 | USDT，配置一次不可改 |
| 消费优先级 | 中心钱包 → 奖励钱包 |
| 币种隔离 | 每个币种独立钱包体系 |
| 重复充值拦截 | 同用户+同币种+同金额，≥3笔待处理则阻断 |
| 提现资金规则 | 法币充→法币提，USDT充→USDT提，混合→法币提 |
| KYC要求 | USDT提现可选，法币/电子钱包提现必须 |
| 奖励钱包稽核 | 奖金需完成X倍流水才能转入中心钱包 |
| 汇率更新阈值 | 偏差超阈值才更新平台汇率，避免频繁波动 |
| 轮询规则 | 按权重分配，不同币种独立，只有启用通道参与 |
| 提现审核 | 三级：风控 → 财务 → 出款 |
| 人工修正 | 创建方 ≠ 审核方，双人复核 |
| 订单号格式 | C(充值)/T(提现)/B(补单)/A(加款)/M(减款) + 16位随机数 |
| 时区 | 存储UTC+0，展示按站点时区转换 |
| 导出安全 | 密码 + 可选谷歌验证码 |
