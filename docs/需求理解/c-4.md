## 一、这个系统在做什么

一句话：**给一个面向东南亚多国的泛娱乐+竞彩平台，做多币种钱包系统**。

用户在这个平台上会做的和钱有关的事：
- **充钱进来**（法币/USDT）
- **花钱**（投注、打赏、消费）
- **把钱拿出去**（提现到银行卡/电子钱包/USDT）
- **换钱**（法币换成平台币BSB）

我的工作就是让这些事情能跑通，钱不能多不能少不能乱。

### 用户是谁

- **普通用户**：充值、投注、消费、提现
- **主播**：除了普通功能外，还有打赏收益、主播钱包
- **后台运营/财务**：配置币种汇率、管理通道、审核提现、人工修正

### 覆盖终端

iOS / Android / H5 三端，C端用户操作体验需要一致。

## 三、核心业务概念（必须搞清楚的前置知识）

### 3.1 钱包体系：用户的钱分装在5个口袋里

不是一个用户一个余额这么简单。用户的钱按用途分成5个子钱包：

| 子钱包 | 通俗理解 | 钱从哪来 | 钱往哪去 | 状态 |
|--------|---------|---------|---------|------|
| **中心钱包** | 主钱包，日常用的 | 充值、投注赢了、其他钱包转进来 | 投注、消费、提现 | 核心 |
| **奖励钱包** | 活动给的钱，有限制 | 活动奖金、VIP奖励、分销奖励 | 完成流水要求后→转入中心钱包 | 核心 |
| **主播钱包** | 主播专用收入 | 用户打赏、主播活动奖励 | 提现或转入中心钱包 | 核心 |
| **代理钱包** | 代理佣金 | 代理分成 | 提现或转中心 | **预留，暂不实现** |
| **场馆钱包** | 进游戏场馆临时钱包 | 从中心+奖励转入 | 退出场馆→回到中心+奖励 | **预留，暂不实现** |

**关键规则：**
- 用户看到的"余额" = 中心钱包 + 奖励钱包（资金钱包余额）
- 花钱的时候，**先扣中心钱包，不够再扣奖励钱包**
- 投注赢了返奖，如果是混合下注（中心+奖励都用了），返奖**按比例分回各自钱包**

> 依据：币种配置需求文档 B2 资金钱包体系（24张中的2-1到2-7系列图）

### 3.2 币种体系：不是人民币，是多国多种货币

| 类型 | 具体币种 | 特点 |
|------|---------|------|
| **法定货币** | VND(越南盾)、IDR(印尼盾)、THB(泰铢) | 各国用各国的，金额展示规则不同 |
| **加密货币** | USDT(主要)、BTC、ETH | 链上转账，精度到8位小数 |
| **平台货币** | BSB | 平台自己的币，10 BSB = 1 USDT，汇率锚定USDT |

**基准币种**：USDT，设一次就不能改了。所有统计、换算、对账都以USDT为基准。

**币种隔离原则**：用户选了哪个币种，就只看那个币种的余额和操作。不存在"我的VND余额和IDR余额混在一起看"的情况。

**金额展示差异**（不同币种展示规则不同）：

| 币种 | 千分位 | 小数 | 举例 |
|------|--------|------|------|
| VND | 点号 | 无 | 1.000.000 |
| IDR | 点号 | 无 | 1.000.000 |
| THB | 逗号 | 2位 | 1,000.00 |
| BSB | 逗号 | 2位 | 1,000.00 |

> 依据：多币种钱包需求文档 A2 + 币种配置需求文档 B3

### 3.3 汇率体系：不是一个汇率，而是一套汇率

这块比较重要，因为涉及到用户充值/提现/兑换时看到的金额：

| 汇率名称 | 是什么 | 怎么来的 |
|---------|--------|---------|
| **实时汇率** | 当前市场真实汇率 | 定时从3家三方API取值，算平均 |
| **平台汇率** | 平台展示和计算用的 | 初始=实时汇率；当实时汇率偏离超过阈值时自动更新 |
| **入款汇率** | 用户充值时用的 | = 实时汇率 × (1 + 浮动%)，平台多收一点 |
| **出款汇率** | 用户提现时用的 | = 实时汇率 × (1 - 浮动%)，平台少给一点 |

**偏差触发机制**：偏差值 = |实时 - 平台| / 实时 × 100%，超过后台配置的阈值就自动更新平台汇率，并生成汇率变更日志。

**精度规则**：法币保留2位小数，加密货币保留8位小数。不足补0，超出四舍五入。

> 依据：币种配置需求文档 B4 汇率规则

## 四、我要做的C端功能（多币种钱包）

### 4.1 充值：用户把钱充进来

**三种充值方式，按币种不同组合：**

| 币种 | 推荐方式 | 其他方式 | 金额档位 |
|------|---------|---------|---------|
| BSB | USDT | 银行/电子钱包 | 100~100,000 |
| VND | 银行转账 | — | 200K~200M |
| IDR | 电子钱包扫码 | 银行 | 150K~150M |
| THB | 电子钱包扫码 | 银行 | 300~350K |

**USDT充值比较特殊**：
- 平台自己承载页面（不跳三方），展示收款地址+二维码
- 支持 TRC20(默认)、ERC20、BEP20 三个网络
- 游客也能充值（不需要登录）
- 不受重复转账拦截（因为链上交易无法控制）

**银行转账/电子钱包充值**：
- 跳到三方承载页
- 用户支付后三方回调通知系统

**重复转账拦截**（防止用户手滑或恶意操作）：
- 判断条件：同一个用户 + 同一个币种 + 同样的金额 + 状态是待支付/进行中
- 1~2笔重复：弹窗警告，但允许继续
- 3笔重复：直接拦截，不让下单
- USDT链上充值不受这个规则限制

**额外奖金选择**（充值可以顺带领活动奖金）：
- 位置在充值页面的余额下方、支付方式上方
- 默认选"无奖金"或自动匹配最优活动
- 一笔充值只能选一个奖金活动
- 奖金 = 充值金额 × 奖金比例（比如充100送50%就是额外50）
- 有最低充值门槛，低于门槛弹提示
- **奖金进奖励钱包，不是中心钱包**，需要完成X倍流水才能提现

> 依据：多币种钱包需求文档 A3 充值模块 + 产品原型 D3/D4/D5/D6 充值相关页面

### 4.2 兑换：法币换成平台币BSB

- **方向**：只能法币(VND/IDR/THB)→BSB，不能反过来
- **入口**：在法币币种下才有兑换入口，BSB币种下没有（因为已经是BSB了）
- **汇率**：展示 1 BSB = X 法币
- **赠送**：比如兑换获得100 BSB，赠送10%就是额外10 BSB
- **赠送部分也需要完成X倍流水才能提现**
- 实际到账 = 兑换获得 + 赠送

> 依据：多币种钱包需求文档 A4 + 产品原型 D7 兑换BSB页

### 4.3 提现：用户把钱拿出去

提现是最复杂的功能，因为涉及KYC认证、多种提现方式、提现规则、审核流程。

**第一步：KYC认证检查**
- 未认证：弹窗两个选择——"去认证"（解锁全部提现渠道）或"继续提现"（只能用USDT提现）
- 已认证：直接进入提现方式选择

**提现方式（按地区不同）：**

| 地区 | 可用方式 | 到账时间 |
|------|---------|---------|
| 越南 | 银行卡 | T+1 |
| 印尼 | GoPay/OVO/DANA/ShopeePay + 银行卡 | 电子钱包即时，银行T+1 |
| 泰国 | PromptPay + 银行卡 | PromptPay即时，银行T+1 |
| BSB | 全渠道都可以，展示汇率换算 | 取决于渠道 |

**提现账户管理**：
- 首次提现需要填写银行卡号/电子钱包账号等信息
- 之后自动填充，不用重复填
- **账户持有人 = KYC认证姓名**，自动填进去，用户不能改

**提现规则（重要，涉及反洗钱逻辑）：**
- 法币充的钱 → 只能法币提
- USDT充的钱 → 只能USDT提
- 法币+USDT混合充的 → 法币提
- 法币→兑换成BSB→再提 → 还是按法币提

核心思路：**钱从哪来就往哪去**，不能通过币种转换来绕过合规要求。

**提现限制**（后台可配置）：
- 单笔最低/最高限额
- 每日提现总限额
- 手续费规则
- T+N到账时间
- 超出限额或低于最低金额都会弹窗提示

> 依据：多币种钱包需求文档 A5 + 产品原型 D8/D9 提现相关页面

### 4.4 记录：用户查看资金流水

**四类记录，四个Tab：**
- 提现记录
- 充值记录
- 兑换记录
- 奖励记录

**基本规则：**
- 只展示当前选中币种的记录，不跨币种
- 详情页包含：订单号、金额明细、状态、时间、方式
- 失败的订单要显示失败原因，并提供"换方式"按钮

**状态颜色：**
- 完成 = 默认色
- 审核中 = 橙色
- 失败 = 红色

> 依据：多币种钱包需求文档 A6 + 产品原型 D10 记录页

### 4.5 非功能性要求（不是功能但必须做到）

| 要求 | 说明 | 为什么重要 |
|------|------|----------|
| **对账一致性** | 账务流水总和 = 账户余额，必须强一致 | 钱不能多不能少 |
| **幂等性** | 充值回调即使收到多次，也只入账一次 | 三方回调可能重复推送 |
| **KYC强制校验** | 法币提现必须完成实名认证 | 合规要求 |
| **提现账户白名单** | 只允许往认证过的账户提现 | 防止洗钱 |
| **姓名强一致校验** | 提现账户姓名 = KYC姓名（忽略大小写和空格） | 防止冒名提现 |
| **大额/异常频率触发人工审核** | 超过阈值的操作需要人工介入 | 风控要求 |

> 依据：多币种钱包需求文档 A7 非功能性需求

后台运营人员在这里管理平台支持的所有币种。

**新增/编辑弹窗**：
- 18个字段完整录入
- 可上传币种图标（SVG/PNG，不超过5KB）
- 汇率浮动配置（2位小数的百分比）
- 偏差阈值配置（2位小数的百分比）
- 启用/禁用状态切换

**基准币种设置**：
- 页面顶部一次性设置
- 推荐USDT
- **设了就不能改了**，因为所有统计、对账、换算都依赖这个基准
- 可选范围：BSB/USDT/INR/USD/JPY/BTC（原型展示）

> 依据：币种配置需求文档 B3 + 产品原型 C1/C2 币种配置列表页+编辑弹窗

### 5.2 汇率管理

不是独立页面，而是币种配置的一部分 + 汇率日志页。

**自动汇率同步**（定时任务，不是页面操作）：
- 定时从3家三方API获取实时汇率
- 取平均值作为"实时汇率"
- 计算偏差值：|实时 - 平台| / 实时 × 100%
- 如果偏差 ≥ 后台配置的阈值 → 自动更新平台汇率 + 入出款汇率 + 生成一条汇率日志

**汇率日志页**（在操作日志页下新增"汇率日志"Tab）：
- 筛选：更新时间范围 / 币种代码
- 表格字段：日志编号、更新时间、币种代码、实时汇率、平台汇率、入款汇率、出款汇率

> 依据：币种配置需求文档 B4 + 产品原型 C3 汇率日志页

## 六、财务管理模块（别人负责，但我必须清楚的部分）

### 6.1 为什么我必须清楚

因为资金链路上，财务管理卡在钱包的上下游：

```
用户充值 → 财务管理（通道收款、回调）→ 通知钱包上账
用户提现 → 钱包冻结金额 → 财务管理（三级审核、通道出款）→ 通知钱包扣款
运营修正 → 财务管理（创建加减款、审核）→ 通知钱包变更余额
```

**钱包不直接对接支付通道，中间隔着财务模块。** 但是余额的变化最终都落在我的钱包上。

### 6.2 财务管理做了什么（粗线条）

| 功能 | 通俗理解 |
|------|---------|
| **通道配置** | 管理代收通道（收钱的渠道）和代付通道（出钱的渠道），配置轮询规则 |
| **支付配置** | 配置每个币种有哪些充值方式、提现方式，关联到具体通道 |
| **充值记录** | 所有充值订单的后台管理视图（22个字段），可搜索/导出 |
| **提现记录** | 所有提现订单的后台管理视图，含三级审核（风控→财务→出款）+锁定/解锁 |
| **人工修正** | 充值补单(B开头)、人工加款(A开头)、人工减款(M开头)，创建和审核必须不同人 |
| **数据统计** | 通道统计、充值统计、提现统计，按日维度汇总 |

> 依据：财务管理需求文档 E1~E6 + 产品原型 F 全套页面

### 6.3 财务对钱包的具体影响（这是我必须实现的RPC接口）

| 财务的操作 | 对我钱包的影响 | 哪个子钱包 |
|-----------|-------------|-----------|
| 充值成功回调 | 充值金额入账 | 中心钱包（奖金部分→奖励钱包） |
| 创建提现订单 | 冻结对应金额 | 按用户提现的币种和钱包 |
| 审核拒绝（风控/财务） | 解冻返还 | 原路返回 |
| 出款成功 | 确认扣除已冻结金额 | 已冻结的钱包 |
| 出款失败 | 解冻返还 | 原路返回 |
| 人工加款审核通过 | 按钱包类型分别加钱 | 运营指定的子钱包（中心/奖励/代理/主播） |
| 人工减款审核通过 | 按钱包类型分别扣钱 | 指定的子钱包（余额不足扣实际余额） |
| 补单审核通过 | 补单金额+赠送入账 | 中心钱包+奖励钱包 |

> 依据：财务管理需求文档 E3充值流程、E4提现流程、E5人工修正

## 七、关键业务流程（把上面的知识串起来）

### 7.1 充值链路（用户充钱 → 钱到账）

```
用户选币种 → 选充值方式 → 选/输金额 → (可选)选奖金活动
   → 提交订单 → 重复转账检查(1-2笔警告/3笔拦截)
   → 财务分配通道 → 创建订单(C+16位)
   → 用户付款(三方页面或USDT链上)
   → 回调/链上确认 → 验签
   → 钱包上账：充值金额→中心钱包，奖金→奖励钱包
```

### 7.2 提现链路（用户提钱 → 钱到手）

```
用户选币种 → KYC检查(未认证→弹窗引导/仅USDT)
   → 选提现方式 → 填/选提现账户
   → 输入金额 → 限额校验(单笔/单日)
   → 提交订单(T+16位) → 钱包冻结金额
   → 风控审核 → (通过)财务审核 → (通过)出款审核
   → 通道出款 → 回调
   → 成功：钱包确认扣除冻结金额
   → 失败/拒绝：钱包解冻返还
```

### 7.3 兑换链路（法币 → BSB）

```
用户在法币币种下点兑换 → 输入金额
   → 查询汇率(1BSB = X法币) → 预览：兑换获得 + 赠送(10%) + 实际到账
   → 确认兑换
   → 扣法币余额(中心钱包) → 加BSB余额(兑换部分→中心钱包，赠送→奖励钱包)
```

### 7.4 人工修正链路（运营操作 → 钱包变化）

```
财务人员创建加款/减款/补单 → 另一个人审核(创建≠审核)
   → 审核通过 → 调用钱包RPC → 按子钱包类型分别加/减
   → 审核拒绝 → 不影响钱包
```

## 八、跨模块依赖关系（三个模块怎么配合）

```
┌──────────┐         ┌──────────────┐         ┌──────────┐
│ 币种配置  │────────→│  多币种钱包   │←────────│ 财务管理  │
│ (B端)    │ 提供    │  (C端)       │ 调用    │ (B端)    │
│          │ 币种列表 │              │ 上账    │          │
│          │ 汇率    │              │ 冻结    │          │
│          │ 基准币种 │              │ 解冻    │          │
│          │         │              │ 扣款    │          │
│          │         │              │ 加减款  │          │
└──────────┘         └──────┬───────┘         └──────────┘
      │                     │                       │
      │                     │ 调用                   │
      │                     ↓                       │
      │              ┌──────────┐                   │
      │              │ ser-kyc  │                   │
      │              │ KYC认证   │                   │
      │              └──────────┘                   │
      │                                             │
      └──────── 财务也调用币种配置获取汇率 ──────────────┘
```

**依赖方向说明：**
- 币种配置 → 钱包：钱包内部调用（同服务），获取币种列表和汇率
- 币种配置 → 财务：财务RPC调用，获取币种筛选数据和基准换算
- 财务 → 钱包：财务RPC调用，通知钱包做余额变更（上账/冻结/解冻/扣款/加减款）
- 钱包 → 财务：钱包RPC调用，获取可用充值/提现方式（支付配置数据在财务侧）
- 钱包 → KYC：钱包RPC调用，查询用户认证状态（判断提现渠道权限）

> 依据：三模块需求文档交叉分析 + 工程分析(c-2/c-3)确认的RPC调用模式

## 十、我的理解总结

### 确定的事（大方向，不会变）

1. **五种子钱包结构**确定了（中心/奖励/主播是核心，代理/场馆预留）
2. **四种币种**确定了（BSB/VND/IDR/THB），基准币种USDT
3. **四大功能**确定了（充值/兑换/提现/记录）
4. **三种充值方式**确定了（USDT/银行/电子钱包）
5. **提现需要KYC**确定了，未认证只能USDT提
6. **提现三级审核**确定了（风控→财务→出款）
7. **钱从哪来往哪去**的合规规则确定了
8. **汇率体系**确定了（实时→平台→入款/出款，阈值触发更新）
9. **三模块联动关系**确定了（钱包管余额，财务管通道审核，币种管汇率）

### 可能变的事（细节层面，保留弹性）

1. 具体金额档位（100~100000 等具体数字可能调整）
2. 汇率浮动百分比和阈值的默认值
3. 重复转账拦截的具体阈值（目前3笔上限）
4. 奖金活动的具体规则和X倍流水的具体倍数
5. 提现手续费的计算方式
6. T+N 到账时间的具体天数
7. 各地区具体支持的电子钱包品牌
8. 人工加减款的类型枚举值

### 需要后续确认的（存疑点）

| 序号 | 疑问 | 为什么存疑 |
|------|------|----------|
