## 一、整体认知

### 1.1 我们在做什么

建设一套**多币种钱包系统**，作为平台资金流转的中枢。

简单说：用户在平台上的钱（充进来、花出去、换币种、提出去）全走这套系统。平台同时面向多个国家（越南、印尼、泰国等），每个国家用不同法币，还有平台自己的虚拟货币 BSB 和加密货币 USDT，所以钱包必须支持多币种、互相隔离、互不干扰。

### 1.2 为什么需要这套系统

平台是泛娱乐 + 合规竞彩的综合体，业务场景覆盖：
- **娱乐消费**：直播打赏、购买虚拟道具、特权购买
- **竞彩投注**：电子游戏、真人视讯、彩票、体育
- **充值提现**：法币充值、USDT充值、法币提现、USDT提现
- **兑换**：法币兑换为平台币 BSB

这些场景都涉及"钱"，没有一个统一的钱包体系来承载，资金流向就会乱、对账就会出问题、风控就做不了。

### 1.3 给谁用

| 角色 | 端 | 用什么 |
|------|---|--------|
| 普通用户 | C端（iOS/Android/H5） | 充值、提现、兑换、查看记录 |
| 主播（签约/非签约） | C端 | 同上 + 主播钱包收益 |
| 运营/财务管理员 | B端后台 | 币种配置、汇率管理、充值/提现记录审核、人工加减款、数据统计 |

### 1.4 我的职责范围

| 模块 | 职责 | 说明 |
|------|------|------|
| **钱包模块** | 主责 | C端核心功能：充值、提现、兑换、记录、余额管理 |
| **币种配置** | 主责 | B端配置：币种管理、汇率规则、基准币种、汇率日志 |
| **财务管理** | 别人负责 | B端运营：通道配置、支付配置、充值/提现记录管理、人工修正、数据统计。但与钱包存在**强依赖关系**，必须清楚全貌 |

## 二、核心概念（绕不过的基础认知）

在深入各模块之前，有几个核心概念必须先理清楚，它们贯穿整个系统。

### 2.1 多币种隔离原则

**一句话：每个币种是一个独立的"世界"，互不干扰。**

- 系统当前支持 4 种币种：**BSB**（平台币）、**VND**（越南盾）、**IDR**（印尼盾）、**THB**（泰铢）
- 每个币种对应一套独立的余额、独立的充值/提现/兑换/记录
- 用户在钱包首页选择了某个币种后，所有操作都**只看这个币种**的数据
- **不存在跨币种混合展示**：不会出现"VND 余额 + THB 余额"合并显示的情况
- 切换币种 = 切换整个视图（余额变了、充值方式变了、记录也变了）

**金额展示规则**也按币种区分（这个前端要关注，后端返数据时要清楚）：

| 币种 | 千分位 | 小数位 | 示例 |
|------|--------|--------|------|
| VND | `.`（点） | 整数（无小数） | 1.000.000 |
| IDR | `.`（点） | 整数（无小数） | 1.000.000 |
| THB | `,`（逗号） | 2位小数 | 1,000.00 |
| BSB | `,`（逗号） | 2位小数 | 1,000.00 |

### 2.2 五种钱包类型

用户名下的钱不是一锅粥，而是按用途分成了 5 个"口袋"：

| 钱包 | 一句话解释 | 钱从哪来 | 钱往哪去 |
|------|----------|---------|---------|
| **中心钱包** | 用户的主钱包，充值的钱默认到这里 | 充值、投注返奖、奖励完成稽核转入 | 消费、投注、提现 |
| **奖励钱包** | 活动奖金的"暂存区"，需要完成流水要求才能变成真金白银 | 签到奖励、任务奖励、VIP奖励、分销奖励 | 完成稽核后 → 转入中心钱包 |
| **主播钱包** | 主播赚的钱 | 礼物打赏收益、主播活动奖励 | 可转入中心钱包、可直接提现 |
| **代理钱包**（预留） | 代理分佣收益 | 代理分成 | 可转入中心钱包（转入后需稽核）、可提现 |
| **场馆钱包**（预留） | 进入 iGaming 场馆时的临时资金 | 中心钱包+奖励钱包转入 | 投注；退出场馆时转回原钱包 |

**关键点**：
- **资金钱包余额 = 中心钱包余额 + 奖励钱包余额**（C端展示的"余额"是这两者之和）
- 代理钱包和场馆钱包是预留设计，当前版本暂不制作（需求文档中灰字标注）

### 2.3 消费优先级规则

当用户花钱（投注、打赏等）时，不是随便从哪个钱包扣的，有严格优先级：

**先扣中心钱包，余额不够再扣奖励钱包。**

这带来一个复杂点：如果一笔消费同时用到了中心钱包和奖励钱包的钱（比如中心余额 80，奖励余额 50，投注 100），那返奖时需要**按投注金额占比分别计算返奖**到各自钱包——中心出了 80 那返奖的 80% 回中心，奖励出了 20 那返奖的 20% 回奖励。

### 2.4 BSB 平台币

BSB 是平台自创的虚拟货币，**锚定 USDT**：

- **10 BSB = 1 USDT**（即 1 BSB = 0.1 USDT）
- 这个锚定关系是**写死在代码里的**，不走配置
- BSB 对各法币的汇率：1 BSB ≈ d2,620（VND）/ Rp1,530（IDR）/ B3.50（THB）
- BSB 存在的意义：作为平台内统一的计价/消费单位，用户在平台里消费的"通用货币"

用户可以用法币余额**兑换**成 BSB（有赠送比例激励），但 BSB 不能直接反向兑换回法币。

### 2.5 基准币种

系统内部需要一个**统一的记账基准**，用于多币种之间的统计、对账、报表：

- 推荐使用 **USDT** 作为基准币种
- **一旦设置不可更改**（B端设置时需二级确认）
- 所有币种金额在账务层面需换算为基准币种进行统计
- 使用场景：活动奖金计算、多币种投注统计、财务对账报表

## 三、钱包模块（C端 — 我的主责）

### 3.1 充值

#### 整体理解

充值是用户把钱充进平台钱包的过程。根据币种不同，支持的充值方式不同：

| 充值方式 | 适用币种 | 支付页面 | 回调方式 | 特点 |
|---------|---------|---------|---------|------|
| **USDT** | BSB（其他币种如支持也可） | **平台承载页**（我们自己的页面） | 区块链监听 | 链上转账，收款地址+二维码 |
| **银行转账** | VND、BSB等 | **三方承载页**（三方支付的页面） | 三方通道回调 | 银行名称/账号/金额/备注 |
| **电子钱包扫码** | IDR（QRIS）、THB（PromptPay）、BSB等 | **三方承载页** | 三方通道回调 | 二维码+倒计时 |

#### 充值通用规则

1. 每个币种进来有**默认充值方式**和**默认最小档位快捷金额**
2. 快捷金额是预设的固定档位（比如 VND 的 200,000 / 500,000 / 1,000,000 ...），用户也可以手动输入（不得低于最低档位）
3. 点击"去充值"后系统生成订单并匹配通道
4. **只有 BSB 充值**时金额输入框下面才显示汇率（因为 BSB 是平台币，需要展示与 USDT/法币的兑换关系）

#### USDT 充值（平台承载页）

- 这是**我们自己做的页面**，不是跳转到三方
- 页面展示：网络选择（默认 TRC20，可选 ERC20/BEP20）、收款地址（可复制）、二维码
- **游客状态也支持 USDT 充值**（其他充值方式需要登录）
- 充值单位是钱包当前币种单位，系统根据汇率自动换算对应的 USDT 金额
- 到账靠区块链确认，通常 2-10 分钟

#### 银行转账充值（三方承载页）

- 用户选金额 → 点充值 → 平台向支付通道创建订单 → 获取并展示转账信息（三方页面）
- 转账信息包括：银行名称、账户名、账号、金额、转账备注（UserID + 时间戳）
- 支付完成后三方通道回调平台更新状态
- 回调后用户可以"查看详情"看状态，或"继续转账程序"返回三方页面

#### QRIS / PromptPay 扫码充值（三方承载页）

- 印尼用 QRIS，泰国用 PromptPay，通常是**默认推荐方式**
- 三方页面展示二维码 + 倒计时（倒计时结束订单失效）
- 支付方式、金额（可复制）、备注（UserID + 时间戳）

#### 重复充值拦截规则

**为什么要拦截**：防止用户因操作失误或网络延迟导致重复付款。

**识别维度**：同一 UserID + 同一币种 + 同一金额，且订单处于"待支付"或"进行中"状态。

**注意**：只对银行转账和电子钱包生效，USDT 链上充值**不适用**此规则（因为链上转账机制不同）。

| 待支付订单数 | 系统行为 | 按钮 |
|------------|---------|------|
| 1-2 笔 | 弹窗提醒，但**允许继续** | "查看详情" + "继续充值"（高亮） |
| 3 笔（上限） | 弹窗拦截，**不允许继续** | "查看详情" + "确认"（关闭弹窗，留在充值页） |

#### 奖金活动选择规则

用户充值时可以选择参与一个奖金活动，获得额外奖金。

**交互位置**：充值页面上，位于"余额展示区"下方、"支付方式"上方。

**核心规则**：
- 默认显示"无奖金"或当前最优匹配活动
- 点击弹出半屏浮窗，列出所有可用奖金活动 + "无奖金"选项
- 选择某个活动后，快捷金额标签右上角带奖金比例角标（如 +50%）
- 输入金额后，输入框右侧显示"额外 + XXX"
- 公式：**额外奖金 = 充值金额 × 奖金比例**
- **门槛校验**：手动输入金额若低于所选活动的最低充值门槛，点击"去充值"时弹气泡提示
- **互斥**：一笔充值订单只能参与一个奖金活动
- **流水要求**：底部强制提示"充值金额与奖金将入账充值钱包，需完成 xx 倍流水要求"

### 3.2 提现

#### 整体理解

提现是用户把钱包里的钱提出来的过程。提现比充值复杂得多，因为涉及**合规**（KYC）和**风控**（限额/审核）。

#### KYC 与提现的关系

这是提现模块最重要的前置条件：

| KYC 状态 | 可用提现方式 | 触发时机 |
|---------|------------|---------|
| **未认证** | **仅 USDT 提现** | 点击提现弹出"温馨提示"弹窗，可选"去认证"或"继续提现"（跳过KYC，只能USDT） |
| **已认证** | 全渠道（法币/电子钱包/USDT） | 点击提现直接进入提现方式选择页，无弹窗 |

KYC 认证页面需填写：姓名、国家、手机号、证件类型、证件号码、上传证件正反面。

#### 提现规则

1. **法币充值 → 法币提现**
2. **USDT 充值 → USDT 提现**
3. **法币 + USDT 混合充值 → 只能法币提现**
4. **法币 → 兑换成 BSB → 仍然是法币提现**

简单说：提现方式取决于充值方式，混合充值时按法币走。

#### 提现账户管理

- **首次提现**：需要填写账户信息（银行名称、账号、持有人姓名、开户网点 等）
- **非首次提现**：自动带出已保存的账户信息（只读），只需输入提现金额
- **账户持有人姓名 = KYC 认证姓名**：自动填充且不可编辑
- **姓名一致性校验**：忽略大小写、忽略多余空格，不做完全等值字符串比对

#### 各国提现方式

| 币种/国家 | 可用方式 | 到账时间 |
|---------|---------|---------|
| VND（越南） | 银行卡提现 | T+1 工作日 |
| IDR（印尼） | GoPay / OVO / DANA / ShopeePay（即时）、银行卡（T+1） |  |
| THB（泰国） | PromptPay（即时，当天到账）、银行卡（T+1） |  |
| BSB | USDT / 各大银行 / GoPay / OVO / DANA / PromptPay | 按方式定 |

**BSB 提现特殊点**：金额输入框下方要显示汇率（因为 BSB 需要换算为法币/USDT 出款）。

#### 提现限制（后台可配置）

- 单笔限额（最低/最高）
- 单日限额
- 手续费规则（如 1%）
- T+N 到账时间

提现时页面展示三行金额：**提现金额 / 手续费 / 到账金额**

确认提现时校验：输入金额超出限额或低于单笔最低时弹窗提示。

#### 提现审核流程（多级）

提现不是用户提了就到账的，需要经过审核：

```
用户创建提现订单 → 待风控审核 → 风控通过/驳回
                              → 待财务审核 → 财务通过/驳回
                                            → 待出款 → 出款成功/失败
```

- **驳回退款**：审核驳回后冻结金额退回用户钱包
- **锁定机制**：防止多个管理员同时操作同一订单（支持普通解锁 + 强制解锁）

### 3.3 兑换（法币 → BSB）

#### 整体理解

兑换是用户把法币余额换成 BSB 的过程。方向是单向的：**只能法币 → BSB**，不能反向。

BSB 充值页面没有"兑换"按钮，只有法币（VND/IDR/THB）充值页面才有。

#### 兑换规则

- 只能用当前币种余额兑换
- 超出余额提示"已达当前可兑换上限"
- 汇率统一展示口径：**1 BSB = X 法币**
  - 示例：1 BSB = d2,620（VND）/ Rp1,530（IDR）/ B3.50（THB）

#### 兑换计算

输入法币金额后，展示：
1. **兑换获得**：BSB xxx.xx
2. **赠送奖励**：如 10%
3. **实际到账 BSB = 兑换获得 + 赠送**

赠送部分需完成 **X 倍流水**后才可提现（页面明确提示）。

#### 兑换交互

- 点击兑换后 3 秒内显示兑换成功提示
- 左上角返回充值中心，右上角跳转记录

### 3.4 记录模块

#### 整体理解

记录是用户查看自己资金流水的地方，分为 4 类标签页：

| 标签 | 展示字段 | 状态标识 |
|------|---------|---------|
| **充值** | 金额、方式、时间、状态、操作 | 已完成(绿) / 充值中(蓝) / 失败(红) |
| **提现** | 金额、方式、时间、状态、操作 | 已完成(绿) / 审核中(黄) / 失败(红) |
| **兑换** | 兑换金额、赠送x%、实际到账、时间 | — |
| **奖励** | 奖励金额、奖励类别、时间 | — |

#### 核心规则

- **记录跟随币种切换**：当前选中什么币种就看什么币种的记录，不存在跨币种混合
- 右上角可切换币种（切换后记录数据整体刷新）
- 详情页所有金额**必须与列表页币种一致**

#### 详情页内容

**充值详情**：订单号、订单金额、入账金额、状态、订单时间、充值方式 icon
- 失败情况展示原因（支付超时/支付异常/卡片问题）+ "更换其他充值方式"链接

**提现详情**：订单号、提现金额、手续费、到账金额、申请时间、状态、提现方式、持卡人姓名、银行卡名称、卡号
- 失败情况展示原因（信息有误/系统维护/额度受限）+ "更换其他提现方式"链接

### 3.5 风控与安全要求

| 要求 | 说明 |
|------|------|
| KYC 强制校验 | 法币/电子钱包提现必须完成 KYC |
| 提现账户白名单 | 账户信息平台记录，不支持用户自行随意修改 |
| 姓名强一致校验 | 提现持有人 = KYC 姓名（忽略大小写和空格） |
| 幂等性 | 充值回调必须幂等处理，防止重复入账 |
| 对账一致性 | 账务流水与账户余额需要强一致性校验 |
| 人工介入 | 异常情况（大额/高频）触发人工审核 |

### 3.6 页面交互要点（后端需要理解的）

虽然页面是前端的事，但有些交互决定了后端接口设计：

**个人中心钱包区域**：
- 显示当前选中币种余额
- 币种切换 icon → 跳转币种选择页（列出所有币种+余额+国旗+选中标识）
- "记录"和"充值"按钮

**游客状态点击充值**：
- 弹窗提示"当前为游客状态，登录后解锁更多充值方式"
- 按钮："继续充值"（仅 USDT）、"去登录"
- 登录后返回原界面再点充值 → 展示所有充值方式

**充值页面**：
- 余额区：当前币种余额 + "兑换"（仅法币有）+ "提现"按钮
- 切换充值方式后，底下的快捷金额档位跟随变化
- 点击"去充值"后跳转三方承载页或 USDT 支付页

**提现页面**：
- 选择提现方式 → 跳转该方式的信息输入页
- 确认提现 → 弹出"提现状态"半屏浮窗

## 四、币种配置（B端 — 我的主责）

### 4.1 币种管理

#### 整体理解

币种配置是运营后台的功能，位于**系统管理 → 币种配置**。管理的是平台支持哪些币种、每个币种的汇率参数、状态开关等。

**重要前提**：币种的**新增**是开发直接写入数据库的，不走后台接口。后台只做**编辑**（修改参数/状态）。

#### 币种类型

| 类型 | 说明 | 示例 |
|------|------|------|
| 法定货币 | 国家/地区官方货币 | VND、IDR、THB |
| 加密货币 | 区块链数字资产 | USDT、BTC、ETH |
| 平台货币 | 平台自创货币 | BSB |

所有类型在配置层统一为"币种"，在业务规则层才区分。

#### 列表页面

**Tab 标签**：币种配置 | 汇率配置

**基准币种配置区**（页面顶部）：
- 下拉选择基准币种 + 提交按钮
- **只可设置一次**，提交时二级确认，提交后按钮隐藏
- 红色警示文字提醒"不可更改"

**筛选条件**：币种代码、币种类型、状态

**列表 20 个字段**：序号、币种代码、币种名称、币种类型、币种国家（加密货币默认"全球"）、币种图标、货币符号、千分位符号、小数点符号、实时汇率、平台汇率、汇率浮动(%)、入款汇率、出款汇率、阈值(%)、排序、状态、操作人、操作时间、操作

**实时汇率刷新**：页面展示倒计时，手动刷新页面才展示新的实时汇率值。

#### 编辑弹窗

- 币种代码（只读）
- 币种名称（只读）
- 汇率浮动（2 位小数，单位 %）
- 币种图标上传（SVG/PNG 格式，不超过 5KB）
- 阈值（2 位小数，单位 %）
- 状态（启用/禁用）

**约束**：
- 每个币种在列表中唯一
- 禁用后 C端不可使用、不可充值、不可提现
- **基准货币无法禁用/隐藏**

### 4.2 汇率机制

这是币种配置模块最核心的部分，理解汇率规则是理解整个资金体系的关键。

#### 四种汇率的关系

```
三方 API（3家取平均）
    ↓ 产出
实时汇率 ← 最新的市场汇率
    ↓ 经过阈值判断后
平台汇率 ← 平台实际采用的基础汇率
    ↓ 加减浮动
入款汇率 = 平台汇率 × (1 + 汇率浮动%)  ← 充值时用
出款汇率 = 平台汇率 × (1 - 汇率浮动%)  ← 提现时用
```

**为什么有浮动**：入款汇率加浮动意味着用户充值时换算偏高（平台少收一点），出款汇率减浮动意味着用户提现时换算偏低（平台多留一点）。这是平台的汇率利差。

#### 汇率使用场景

| 场景 | 用哪个汇率 | 说明 |
|------|-----------|------|
| 充值 | 入款汇率 | 跨币种充值时（如用 VND 充值 BSB） |
| 提现 | 出款汇率 | 跨币种提现时（如 BSB 提现为 VND） |
| 投注 | 入款汇率 | 进入游戏时转换，记录汇率，退出时原汇率转回 |
| 统计 | 平台汇率 | 报表统计，所有币种换算为基准币种 |

#### 阈值更新机制

平台汇率不是实时跟随市场波动的，而是**只有偏差够大时才更新**：

**偏差值 = (实时汇率 - 平台汇率) / 实时汇率 × 100%**

**当偏差值 >= 阈值%时 → 更新平台汇率（及入款/出款汇率）**

**举例**：BSB:VND 平台汇率 70，实时汇率变成 71
- 偏差值 = (71-70)/71 = 1.4%
- 如果阈值设的是 2%，那 1.4% < 2%，**不更新**
- 如果实时汇率变成 72，偏差 = (72-70)/72 = 2.78% > 2%，**触发更新**

**关键细节**：
- 入款和出款的偏差值需要**分别判断**，任一方达到阈值就更新
- 未超过阈值的币种不更新
- 每次更新后生成汇率日志

#### 汇率精度

| 货币类型 | C端展示 | 系统内部 |
|---------|--------|---------|
| 法定货币 | 2 位小数 | 2 位小数，不足补 0，超过四舍五入 |
| 加密货币 | 8 位小数 | 8 位小数，不足补 0，超过四舍五入 |

#### 实时汇率获取

- 通过 **3 家三方 API** 获取，取平均值作为实时汇率
- 获取频率可配置（测试期间约 1 天 1 次，上线后按运营需求调整）
- 平台汇率初始为空时，直接取实时汇率值

#### BSB 特殊处理

- BSB 的价值（10 BSB = 1 USDT）和锚定汇率**写入代码**，不做后台配置
- BSB 行的实时汇率/平台汇率就是 10（对 USDT）

### 4.3 汇率日志

**页面位置**：运营后台 → 系统管理 → 操作日志 → 汇率日志 Tab（与登入日志、业务后台日志等并列）

当有币种的平台汇率发生变更时，自动生成一条日志。

**日志字段**：序号、日志编号（唯一编码）、更新时间、币种代码、实时汇率、平台汇率、入款汇率、出款汇率

**筛选条件**：更新时间（日期范围）、币种代码

## 五、财务管理（B端 — 别人负责，我需理解联动）

> 这部分由其他同事负责开发，但因为和钱包存在大量联动依赖，我必须清楚它的全貌。

### 5.1 通道配置

通道就是连接平台和三方支付公司的"管道"。分为代收通道（收钱，对应充值）和代付通道（付钱，对应提现）。

**代收通道**：
- 列表字段：币种、通道名称、支付类型（唤醒/原生/混合/扫码）、代收费率、单笔限额、代收权重、状态、预警金额
- 编辑可改：费率、限额、权重、预警金额、状态
- 还有一组敏感的**代收配置**：请求地址、商户 ID、密钥、AppKey、支付编码、平台公钥、公钥、私钥

**代付通道**：类似代收，多了"单笔手续费"字段

**通道轮询规则**：当多个通道同时启用时，系统用**加权轮询**方式选通道，权重由三个维度的系数决定：
1. **成功率系数**：单位时间内成功率 → 对应系数
2. **时间系数**：平均完成时间 → 对应系数
3. **预警系数**：累计金额达到预警值时降低权重

单通道启用时直接走那个通道，无需轮询。

### 5.2 支付配置

支付配置管理的是 C端用户看到的"充值方式"和"提现方式"。

**充值方式**：
- 列表字段：币种、充值方式、支付图标、充值特征、充值金额范围、关联通道（多选）、排序、状态
- 新增/编辑时设置：金额档位（快捷金额）或金额范围（最低-最高）、关联哪些代收通道

**提现方式**：
- 类似充值方式，有提现金额范围和关联的代付通道

**与钱包模块的关系**：C端用户看到的充值方式列表、提现方式列表的数据，源头在这里。钱包模块通过 RPC 从财务获取这些配置。

### 5.3 充值记录 & 提现记录

**充值记录**（B端视角，24 个字段）：
- 核心：创建时间、完成时间、平台订单号、通道订单号、用户编号、昵称、通道名称、订单币种、充值方式、订单状态、订单金额
- 活动：活动类型、活动优惠
- 费用：充值费率、手续费、到账总金额
- 汇率：入款汇率、支付币种、支付金额、平台汇率
- 基准币种：基准币种订单金额、活动优惠、手续费
- 订单号格式：**C + 16位随机数**
- 订单状态：待支付 / 支付成功 / 支付失败
- 支持导出（需密码 + 谷歌验证码）

**提现记录**（B端视角，27 个字段）：
- 比充值多了审核相关字段：拒绝原因、审核备注、操作人
- 订单号格式：**T + 16位随机数**
- 订单状态（7 种）：待风控审核 / 风控驳回 / 待财务审核 / 财务驳回 / 待出款 / 出款失败 / 出款成功
- 操作：审核（风控/财务/出款）、锁定、解锁、强制解锁
- 支持导出（需密码 + 谷歌验证码）

### 5.4 人工修正

人工修正是处理异常情况的后门，分三种：

**充值补单**：
- 场景：用户充值了但系统没收到回调（三方掉单），运营人员手动补单
- 订单号格式：**B + 16位随机数**
- 流程：输入原充值订单号查询 → 填写补单金额/赠送金额/稽核倍数 → 提交 → 审核通过/拒绝
- 审核通过后 → RPC 调钱包模块入账

**人工加款**：
- 场景：运营补偿、运营奖励、财务修正、测试加款
- 订单号格式：**A + 16位随机数**
- 特点：**支持多钱包同时加款**（中心钱包、奖励钱包、代理钱包、主播钱包分别指定金额）
- 审核通过后 → RPC 调钱包模块对指定钱包类型加款

**人工减款**：
- 场景：异常追回、风控处罚、财务冲正、费用调整
- 订单号格式：**M + 16位随机数**
- 特点：显示用户各钱包当前余额，减款金额不得超过余额
- 审核通过后 → RPC 调钱包模块对指定钱包类型减款

### 5.5 数据统计

**时区处理规则**（重要）：
- 服务器存储：**UTC+0** 基准时间
- 后台展示：**站点时区**（可配置，默认 UTC+0，支持 IANA Time Zone 标准）
- 查询时需要把站点时区转换为 UTC 时间再查数据库

**统计类型**：

| 统计 | 汇总卡片数 | 核心指标 |
|------|-----------|---------|
| 代收通道统计 | 8个 | 订单总数/金额、成功数/金额/率、手续费、净入账、平均入款时间 |
| 代付通道统计 | 类似 | 增加出款总金额 |
| 充值统计 | 10个 | 充值人数/成功人数、人均充值、待支付/失败订单 |
| 提现统计 | 12个 | 提现人数/成功人数、平台/通道手续费、多级审核状态统计 |

所有统计都支持按日期和币种维度查看，金额同时展示原币种和基准币种两套数据。

## 六、跨模块联动关系

这部分很关键——三个模块不是各自独立的，它们之间有大量数据流和调用关系。

### 6.1 联动全景

```
                    ┌──────────────┐
                    │   币种配置    │
                    │ （汇率/状态） │
                    └──────┬───────┘
                 提供汇率/币种  │  提供汇率/币种
                ┌──────────┘  └──────────┐
                ↓                         ↓
        ┌──────────────┐          ┌──────────────┐
        │   钱包模块    │◄────────►│   财务管理    │
        │ （C端/余额）  │ 双向依赖  │ （B端/通道）  │
        └──────────────┘          └──────────────┘
```

### 6.2 具体联动场景

| 场景 | 数据流向 | 具体动作 |
|------|---------|---------|
| 充值成功入账 | 财务 → 钱包 | 财务接收通道回调验签后，RPC 调钱包增加余额 |
| 提现冻结 | 钱包内部 | 用户创建提现订单时，钱包自行冻结对应金额 |
| 提现审核通过 | 财务 → 钱包 | 财务审核通过后，RPC 调钱包确认扣除冻结金额 |
| 提现审核驳回 | 财务 → 钱包 | 财务审核驳回后，RPC 调钱包解冻退回金额 |
| 补单入账 | 财务 → 钱包 | 补单审核通过后，RPC 调钱包入账 |
| 人工加款 | 财务 → 钱包 | 加款审核通过后，RPC 调钱包按指定钱包类型加款 |
| 人工减款 | 财务 → 钱包 | 减款审核通过后，RPC 调钱包按指定钱包类型减款 |
| 查询余额 | 财务 → 钱包 | 提现审核/加减款时，RPC 调钱包查用户当前余额 |
| 获取充值方式 | 钱包 → 财务 | C端充值时，钱包 RPC 调财务获取当前币种的充值方式配置 |
| 获取提现方式 | 钱包 → 财务 | C端提现时，钱包 RPC 调财务获取当前币种的提现方式配置 |
| 匹配通道支付 | 钱包 → 财务 | 创建充值订单时，钱包 RPC 调财务进行通道轮询匹配+创建支付 |
| 查询汇率 | 钱包 → 币种配置 | 充值/提现/兑换时，钱包 RPC 调币种配置获取当前汇率 |
| 查询币种状态 | 钱包 → 币种配置 | 展示币种列表时检查各币种是否启用 |
| 统计换算 | 财务 → 币种配置 | 财务报表统计时，获取平台汇率换算为基准币种 |
| 币种禁用影响 | 币种配置 → 钱包 | 币种禁用后，C端该币种不可充值/提现/兑换 |

### 6.3 模块职责边界总结

| 职责 | 归属 | 说明 |
|------|------|------|
| 用户余额管理 | 钱包 | 余额的增/减/冻结/解冻/查询 |
| 用户可见订单 | 钱包 | C端充值/提现/兑换订单的创建和状态展示 |
| 通道对接 | 财务 | 通道配置/密钥管理/轮询匹配/回调验签 |
| 充值/提现方式配置 | 财务 | B端配置哪些充值方式/提现方式可用 |
| 充值/提现记录管理 | 财务 | B端查看/审核/导出/统计 |
| 人工修正 | 财务 | 补单/加款/减款的发起和审核（审核通过后调钱包RPC） |
| 币种/汇率管理 | 币种配置 | 币种参数、汇率规则、阈值机制、汇率日志 |

## 七、核心规则速查

把分散在各模块中的关键规则集中在此，方便随时查阅。

### 7.1 关键公式

| 公式 | 含义 |
|------|------|
| 资金钱包余额 = 中心钱包 + 奖励钱包 | C端展示的"余额" |
| 10 BSB = 1 USDT | BSB 锚定（代码写死） |
| 入款汇率 = 平台汇率 × (1 + 汇率浮动%) | 充值时用 |
| 出款汇率 = 平台汇率 × (1 - 汇率浮动%) | 提现时用 |
| 偏差值 = \|(实时汇率 - 平台汇率) / 实时汇率\| × 100% | 判断是否更新 |
| 当偏差值 >= 阈值% → 更新平台汇率 | 阈值触发 |
| 额外奖金 = 充值金额 × 奖金比例 | 充值活动奖金 |

### 7.3 状态枚举

**提现订单**：待风控审核 → 风控驳回 / 待财务审核 → 财务驳回 / 待出款 → 出款失败 / 出款成功
