## 一、全局概述

### 1.1 需求背景

平台支持**直播 + 游戏 + 打赏 + 多国家市场**，资金体系逐步复杂化，需要建立一套清晰、安全、可扩展的多币种钱包系统，支撑以下能力：

- 多币种充值与支付
- 游戏投注与结算
- 礼物打赏与主播收益
- 提现与合规风控（KYC）
- 资金流水与对账追溯

### 1.2 需求目标

1. 支持多币种钱包体系（法币 / 平台币）
2. 明确资产分层，避免资金逻辑混乱
3. 支持不同币种对应不同充值/提现方式
4. 提供完整、可追溯的资金记录
5. 确保提现合规（KYC + 风控）

### 1.3 适用范围

- 客户端：iOS / Android / H5
- 用户角色：普通用户、主播（含签约/非签约）

### 1.4 三模块定位与职责划分

| 模块 | 定位 | 职责归属 | 说明 |
|------|------|---------|------|
| 币种配置 | 基础设施层 | **自己负责** | 为钱包和财务体系提供币种定义、汇率、钱包分层结构 |
| 多币种钱包 | 核心业务层 | **自己负责** | C端充值/提现/兑换/记录，暴露RPC给财务模块 |
| 财务管理 | 运营管理层 | 他人负责 | B端通道/支付配置、充值提现记录审核、人工修正、统计 |

### 1.5 文档颜色规范

- **绿字**：解释说明内容
- **红字**：需要特别注意的内容
- **灰字**：当前版本暂不制作
- **删除线**：已移除的内容

## 二、币种配置模块（自己负责 — 基础设施层）

> 依据来源：币种配置需求文档（24张图）+ 产品原型（3张图）

### 2.1 资金钱包体系

#### 2.1.1 定义

资金钱包用于承载用户在平台内的**真实资金**，用于 igaming 投注、结算、兑换钻石及提现等操作，是风控与财务的核心模块。资金钱包支持平台币、多法币、多虚拟币，C端钱包中各币种金额相互独立。

> 依据：2-1资金钱包体系.png

#### 2.1.2 五种子钱包

| 子钱包 | 定义 | 资金来源 | 资金流向 | 状态 |
|--------|------|---------|---------|------|
| **中心钱包** | 用户主资金钱包，承担充值、投注扣减、投注返奖、提现 | 充值、完成稽核的奖励金、主播/代理钱包转入 | 投注、娱乐消费、提现（需完成稽核） | 核心 |
| **奖励钱包** | 活动/任务奖金暂存钱包 | 领取各活动奖励金、各任务、VIP、分销奖励 | 完成稽核流水后→转入中心钱包 | 核心 |
| **主播钱包** | 承接主播/创作者结算的收益资金 | 直播礼物收益、主播活动/任务奖励 | 可直接提现，可转入中心钱包（转入后需稽核） | 核心 |
| **代理钱包** | 代理专属资金钱包 | 代理分成佣金 | 可直接提现，可转入中心钱包（转入后需稽核） | **预留** |
| **场馆钱包** | 用户进入igaming场馆时的临时资金钱包 | 中心钱包+奖励钱包余额转入 | 退出场馆时归还到中心钱包+奖励钱包 | **预留** |

> 依据：2-1资金钱包体系2.png ~ 2-1资金钱包体系6.png

#### 2.1.3 核心余额规则

- **资金钱包余额 = 中心钱包余额 + 奖励钱包余额**
- 资金钱包余额可在全平台进行任意消费
- **消费优先级**：先扣中心钱包余额，再扣奖励钱包余额
- 当前游戏均为自研游戏，进入游戏后钱包余额 = 资金钱包余额
- 投注时优先扣中心钱包余额，再扣奖励钱包余额
- **混合投注返奖**：中心钱包余额投注返奖计入中心钱包，奖励钱包余额投注返奖计入奖励钱包，单次下注使用到两个钱包余额混合投注时，**需根据投注金额占比分开计返奖**

> 依据：2-1资金钱包体系7.png

#### 2.1.4 使用场景

- 娱乐消费：刷礼物、购买虚拟道具（头像框、入场坐骑、靓号等）、购买特权及付费解锁内容等
- 投注：电子、真人视讯、彩票、体育中投注
- 提现：法币充值可提法币，虚拟币充值可提虚拟币；法币+虚拟币混合充值只可提法币

> 依据：2-1资金钱包体系3.png

#### 2.1.5 获得途径

- 充值：直充法币或虚拟币
- 投注返奖：通过投注场景赚取的资金
- 活动/任务返奖：参与游戏活动/任务获得的奖金

> 依据：2-1资金钱包体系3.png

### 2.3 基准币种

- **基准币种定义**：系统内部统一使用一个基准币种进行账务换算
- 推荐使用：**USDT**，可配置，**配置后无法更改**
- **BSB价值：10 BSB = 1 USDT，BSB汇率锚定USDT**
- 所有币种金额在账务层需换算为基准币种再进行统计

**使用场景**：
- 活动奖金计算
- 多币种投注与返奖统计
- 多币种钱包资产统计
- 财务对账与统计报表

**后台操作**：
- 基准币种只可设置1次，下拉框选项 = 币种配置表中所有启用状态的币种
- 提交时需**二级确认弹窗**："您确认要将平台基准币种设置为 XXXX 吗?"
- 配置并提交后，提交按钮隐藏，上线后不可修改
- BSB价值、锚定汇率直接代码写入，不做配置

### 2.4 汇率体系

#### 2.4.1 汇率应用场景

| 场景 | 使用汇率 | 说明 |
|------|---------|------|
| 充值 | 平台入款汇率 | 跨币种充值时使用，如USDT充BSB |
| 提现 | 平台出款汇率 | 跨币种提现时使用，如BSB提USDT |
| 投注 | 平台入款汇率 | 游戏不支持当前钱包币种时，进入游戏转换为可投注币种，记录该汇率，退出时用同一汇率转回 |
| 统计 | 平台汇率 | 充值、提现、账变等统计需转化为基准货币 |

> 依据：4-1汇率应用场景.png

#### 2.4.2 汇率规则

**实时汇率**：
- 通过**3家三方API取汇率**，算平均值作为实时汇率
- 多家三方也可以避免接口API停用或挂掉的问题
- 后台展示，作为平台汇率的基准来源
- 刷新频率可配（开发测试期间可低频约1天1次，上线根据运营需求设定）

**平台汇率**：
- 初始平台汇率为空，直接取实时汇率值为平台汇率

**汇率公式**：
- **平台入款汇率** = 实时汇率 × (1 + 汇率浮动%)
- **平台出款汇率** = 实时汇率 × (1 - 汇率浮动%)
- **偏差值** = (实时汇率 - 平台汇率) / 实时汇率 × 100%

**阈值机制**：
- 阈值可配置，当偏差值 ≥ 阈值% 时，更新平台汇率为新平台汇率
- **需同时判断入款和出款偏差值，其中一个 ≥ 阈值时就更新平台汇率**（红字强调）
- 某个币种偏差值 ≥ 阈值时，只更新该币种的平台汇率和出入款汇率，未超阈值的币种不更新
- 所有平台汇率更新后需生成对应**汇率日志**

**更新流程**：
1. 实时汇率更新（3家API取均值）
2. 是否存在平台汇率？
   - 不存在 → 直接根据实时汇率计算平台汇率
   - 存在 → 计算偏差值是否 ≥ 阈值%
     - 是 → 根据实时汇率计算新的平台汇率，生成汇率日志
     - 否 → 当前平台汇率保持不变

> 依据：4-2汇率规则说明1.png、4-2汇率规则说明2.png

#### 2.4.3 汇率展示精度

| 货币类型 | C端展示汇率 | 系统侧汇率 |
|---------|------------|-----------|
| 法定货币 | 保留小数点后2位（具体见钱包功能文档） | 保留小数点后2位，不足补0，超过四舍五入 |
| 加密货币 | 保留小数点后8位（具体见钱包功能文档） | 保留小数点后8位，不足补0，超过四舍五入 |

## 三、多币种钱包模块（自己负责 — 核心业务层）

### 3.1 钱包总体设计

#### 3.1.1 币种结构

- 钱包按**币种维度隔离**，每个币种一套独立余额与账本
- 当前支持币种：**BSB**（1BSB=0.1USDT）、**VND**、**IDR**、**THB**

#### 3.1.2 金额展示规则

| 币种 | 千分位 | 小数 | 示例 |
|------|--------|------|------|
| VND | `.` | 整数（无小数） | 9.000.000 |
| IDR | `.` | 整数（无小数） | 8.000.000 |
| THB | `,` | 2位小数 | 1,000.00 |
| BSB | `,` | 2位小数 | 1,000,000.00 |

> 依据：4-1钱包总体设计.png

#### 3.1.3 币种切换规则

- 钱包首页可切换当前币种
- **当前选中币种 = 所有操作的唯一数据源**：充值、提现、兑换、交易记录
- 切换币种即切换：余额、充值/支付方式、记录列表数据
- **不支持跨币种混合展示与操作**

> 依据：4-2币种切换规则.png

#### 3.1.4 页面交互

- 点击切换 → 跳转选择币种界面（列表展示所有币种余额，当前选中有勾选标记）
- 点击充值 → 游客状态弹出温馨提示："当前为游客状态，登录后解锁更多充值方式"
  - "继续充值" → 仅支持USDT充值
  - "去登录" → 跳转登录界面
- 点击记录 → 跳转该币种记录界面

> 依据：4-3页面交互.png、产品原型2-1~2-3

### 3.2 充值模块

#### 3.2.1 充值方式总览

| 类型 | 方式 | 是否平台承载页 | 状态回调 | 备注 |
|------|------|-------------|---------|------|
| 虚拟币 | USDT（TRC20/ERC20/BEP20） | 是 | 区块链监听确认到账 | 链上转账，游客也可用 |
| 银行 | 银行转账 | 否（第三方承载页） | 第三方支付通道回调 | 第三方通道 |
| 电子钱包 | QRIS/PromptPay扫码 | 否（第三方承载页） | 第三方支付通道回调 | 二维码+倒计时 |

**各币种默认充值方式**：
- BSB：USDT（推荐）> 银行 > 电子钱包扫码
- VND：越南银行转账
- IDR：电子钱包扫码（推荐）> 印尼银行
- THB：电子钱包扫码（推荐）> 泰国银行

**充值通用规则**：
- 每个币种进来有默认充值方式及默认最小档位快捷金额
- 充值金额输入框支持手动填写（不得小于最低档位）
- 点击"去充值"后生成订单匹配通道
- 只有BSB充值金额输入框下有对应汇率展示及兑换所得金额

> 依据：5-1充值方式总览.png ~ 5-1充值方式总览3.png、产品原型3-1~3-4、5-1~5-4

#### 3.2.2 USDT充值

**入口规则**：仅在支持USDT支付方式的币种下可见，游客登录状态也支持

**金额规则**：充值单位为钱包币种单位，系统根据汇率自动换算USDT

**支付页**：

#### 4.1.2 代付通道

- 关联代付配置

> 依据：2-1财务管理-通道配置-2代付通道.png

#### 4.1.3 通道轮询规则

**核心排序公式**：
```
轮询优先级 = 成功率系数权重 × 成功率系数 + 时间系数权重 × 时间系数 + 预警系数权重 × 预警系数
```

**权重比例**：三个权重之和必须 = 100%
**初始默认**：成功率 50% : 时间 30% : 预警 20%

**成功率系数**：
- 单位时间（分钟）内计算通道成功率
- 样本最小订单数：单位时间内未达到最小样本量则不计算
- 成功率区间 → 对应系数值（可配多行）

**时间系数**：
- 单位时间内的平均完成时间
- 未达到最小样本量则时间系数默认为0
- 平均完成时间区间(s) → 对应系数值

**预警系数**：
- 通道累计金额 ≥ 预警金额时触发
- 达到预警值时对应的预警系数

代收轮询和代付轮询配置结构完全一致。

> 依据：2-1财务管理-通道配置-3通道轮询规则.png

**列表关键字段**（24列）：

#### 4.3.2 完整充值流程

```
C端用户选择币种 → 选择充值方式 → 选择/输入金额 → 提交订单
    ↓
服务端轮询匹配可用代收通道
    ├─ 无可用通道 → 提示"通道维护中" → 订单状态:支付失败 → 返回充值页选择新方式
    └─ 匹配成功 → 创建平台充值订单号(C+16位) → 订单状态:待支付
        ↓
    通道创建通道充值订单号
        ↓
    服务端返回支付页面/信息给C端
        ↓
    C端拉起支付页面等待用户支付
        ↓
    通道回传支付结果
        ↓
    服务端判断
    ├─ 支付失败 → 更新状态:支付失败 → 返回C端
    └─ 支付结果 → 验证金额是否一致
        ├─ 不一致 → 状态:支付失败
        └─ 一致 → 状态:支付成功 → **打款到对应用户上分（入账到用户钱包）**
```

**与钱包模块交互点**：充值成功 → 调用钱包RPC加款（订单金额入中心钱包，活动赠送入奖励钱包）

#### 4.4.2 完整提现流程

```
C端用户选择币种+提现方式+输入金额 → 提交订单
    ↓
服务端创建平台订单号(T+16位) → 订单状态:待风控审核
    ↓
**冻结用户钱包对应金额**（从可用余额移入冻结余额）
    ↓
风控审核
├─ 驳回 → 状态:风控驳回 → **解冻归还冻结金额**
└─ 通过 → 进入财务审核
    ↓

├─ 驳回 → 状态:财务驳回 → **解冻归还冻结金额**
└─ 通过 → 进入出款环节
    ↓

    ↓
通道回传出款结果
├─ 出款失败 → 状态:出款失败 → **解冻归还冻结金额**
└─ 出款成功 → 状态:出款成功 → **扣减冻结金额（实际扣款）**
```

| 财务驳回 | **UnfreezeBalance** | 解冻归还 |
| 出款失败 | **UnfreezeBalance** | 解冻归还 |

### 4.5 人工修正

人工修正包含三类：充值补单、人工加款、人工减款

**通用审核规则**：
- 提交后产生审核单据，状态为"待审核"
- **审核人与提交人不能为同一人**
- 审核选项：通过/拒绝，拒绝必须填写原因
- 审核通过后执行对应钱包操作，拒绝则不执行

#### 4.5.1 充值补单

**订单号格式**：**B+16位随机数**

**创建补单流程**：
1. 输入充值订单号 → 点击"查询" → 回显原订单信息和用户信息
2. 填写：补单订单金额、补单赠送金额、补单原因、订单稽核倍数、赠金稽核倍数
3. 系统自动计算：补单总金额、订单稽核流水、赠金稽核流水

**列表关键字段**：补单订单号(B+16位)、订单号(平台C+16位)、用户编号、订单币种、原订单金额、补单金额、原赠送金额、补单赠送金额、原订单总金额、补单订单总金额、平台汇率、基准币种补单金额/赠送/总金额

**与钱包模块交互**：审核通过 → **CreditWallet**（补单金额+赠送金额入账用户钱包）

> 依据：2-5财务管理-人工修正-1充值补单.png

#### 4.5.2 人工加款

**订单号格式**：**A+16位随机数**

**加款类型**：运营补偿 / 运营奖励 / 财务修正 / 测试加款

**创建加款流程**：
1. 输入用户编号 → 查询 → 回显用户信息
2. 选择加款币种
3. **按4种钱包类型分别填写加款金额**：

| 钱包类型 | 加款金额 | 加款后余额 | 稽核倍数 | 稽核流水 |
|---------|---------|-----------|---------|---------|
| 中心钱包 | 输入 | 自动计算 | 输入 | 自动计算 |
| 奖励钱包 | 输入 | 自动计算 | 输入 | 自动计算 |
| 代理钱包 | 输入 | 自动计算 | 输入 | 自动计算 |
| 主播钱包 | 输入 | 自动计算 | 输入 | 自动计算 |

**与钱包模块交互**：审核通过 → **CreditWallet**（按各钱包类型分别加款到对应子钱包）

> 依据：2-5财务管理-人工修正-2人工加款.png

#### 4.5.3 人工减款

**订单号格式**：**M+16位随机数**

**减款类型**：异常追回 / 风控处罚 / 财务冲正 / 费用调整

**创建减款流程**：
1. 输入用户编号 → 查询 → 回显用户信息 + **各钱包当前余额**
2. 选择减款币种
3. 按4种钱包类型分别填写减款金额（显示当前余额、减款金额、减款后余额）

**特殊规则**：减款金额不超过当前余额（余额不足时无法全额减款）

**与钱包模块交互**：
- 审核通过 → **DebitWallet**（按各钱包类型分别扣减）
- 创建减款弹窗需要查询用户各子钱包余额 → 调用钱包RPC **GetBalance**

> 依据：2-5财务管理-人工修正-3人工减款.png
