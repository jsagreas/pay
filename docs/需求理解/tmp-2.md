## 一、这个系统到底要做什么（一句话说清）

简单说：**给一个多国运营的泛娱乐+竞彩平台，建一套支持多种货币的电子钱包**。

用户在平台上打赏主播、买道具、玩游戏、投注竞彩，都需要用钱。这个钱包就是管"钱进钱出"的核心模块——充值把钱存进来，消费把钱花出去，提现把钱取出去，兑换把不同货币互相转。

平台目前面向东南亚（越南、印尼、泰国），所以需要支持这些国家的法币，同时还有一种叫 BSB 的平台自己的虚拟币，以及 USDT 加密货币作为跨国通用充值手段。

## 二、涉及三个模块，各自的分工

| 模块 | 谁负责 | 一句话定位 | 面向谁 |
|------|--------|----------|--------|
| **多币种钱包** | 我方 | 用户直接使用的钱包功能（充值/提现/兑换/查记录） | C端用户（手机App/H5） |
| **币种配置** | 我方 | 给钱包提供"基础数据"——哪些币种可用、汇率是多少、精度规则是什么 | B端运营后台 |
| **财务管理** | 他方 | B端运营人员管理支付通道、审核提现、人工补单/加减款、看统计报表 | B端运营后台 |

**三者关系可以这么理解：**
- 币种配置是**地基**——告诉系统"有哪些币种、汇率怎么算"
- 钱包是**房子**——用户在上面充值、消费、提现
- 财务管理是**物业管理**——运营人员在后台审核、调账、看数据

它们不是独立运行的，而是有大量联动：财务模块审核通过一笔提现后，需要调钱包的接口扣钱；钱包展示汇率时，需要读币种配置的数据；财务模块人工给用户加减款，也需要调钱包接口操作余额。

## 三、钱包模块 — 我理解的核心功能

### 3.1 钱包的"容器"结构

一个用户不是只有一个钱包，而是**按币种隔离、按类型分层**：

**按币种：** 用户有 BSB 钱包、VND 钱包、IDR 钱包、THB 钱包，各自独立，余额互不相干。用户在 App 上切换币种时，看到的余额、能用的充值方式、历史记录全部切换。不存在"把越南盾和泰铢混在一起"的情况。

**按类型：** 每个币种下又分几个子钱包：

| 子钱包 | 通俗理解 | 当前状态 |
|--------|---------|---------|
| 中心钱包 | 主钱包，充值进来的钱在这里，消费/提现也从这里扣 | 已实现 |
| 奖励钱包 | 平台给的奖金/赠送在这里，花之前有"流水要求"门槛 | 已实现 |
| 主播钱包 | 主播的收入（打赏/佣金等）在这里，可以提现 | 已实现 |
| 代理钱包 | 代理的佣金收入，预留功能 | 预留 |
| 场馆钱包 | 进游戏场馆时的临时钱包，预留功能 | 预留 |

**关键规则：消费时先扣中心钱包，不够再扣奖励钱包。** 返奖时按投注时各钱包出资比例拆分归还。这意味着钱包服务内部需要处理"优先级扣款"和"按比例返奖"的逻辑。

**用户看到的"资金钱包余额"= 中心钱包 + 奖励钱包的合计**，但后台实际是分开记账的。

### 3.2 充值 — 把钱存进来

用户要充钱，不同币种有不同的充值方式，这是由财务模块在后台配置的：

| 币种 | 可用的充值方式 | 产品设计的推荐方式 |
|------|--------------|------------------|
| BSB | USDT链上转账 + 银行转账 + 电子钱包 | USDT |
| VND | 越南银行转账 | — |
| IDR | 电子钱包扫码 + 印尼银行 | 电子钱包 |
| THB | 电子钱包扫码 + 泰国银行 | 电子钱包 |

**充值流程（我的理解）：**
1. 用户在 App 上选币种 → 选充值方式 → 输入金额（或选快捷档位）
2. 钱包创建平台订单 → 调用财务模块匹配支付通道 → 返回支付信息给用户
3. 用户完成支付（跳转三方页面/扫码/链上转账）
4. 三方通道回调通知财务模块支付结果
5. 财务模块收到"支付成功" → 调钱包 RPC 接口 CreditWallet → 钱入账

**USDT 充值特殊性：**
- 游客（未登录）也能用——这在 App 里是个开放入口
- 支持 3 种网络（TRC20/ERC20/BEP20），默认 TRC20
- 展示收款地址+QR码，用户在链上转账，2~10 分钟到账
- 金额单位是钱包币种，系统自动按汇率换算为 USDT

**充值时还有两个重要的附加机制：**

**① 重复转账拦截：** 同一用户、同一币种、同一金额如果已有 1~2 笔待处理订单，弹窗警告但允许继续；如果已有 3 笔，强制拦截不让创建新订单。目的是防止用户误操作或恶意刷单。仅适用于银行转账和电子钱包，USDT 链上充值不受此限制（因为链上无法拦截）。

**② 奖金/优惠选择：** 充值时用户可以选择一个奖金活动（如"充 100 送 50%"），选择后快捷金额只显示满足活动门槛的档位。一笔充值只能参与一个活动。奖金部分入奖励钱包，需完成 N 倍流水才能提现。

### 3.3 兑换 — 法币换成 BSB

兑换是**单向的**：只能法币 → BSB，不能反过来。BSB 钱包页面不显示兑换入口。

**为什么要有这个功能：** 部分游戏/消费场景可能只支持 BSB 结算，用户充了法币后需要换成 BSB 才能使用。

**兑换逻辑：**
- 输入法币金额 → 按汇率算出 BSB 数量 → 赠送奖金（如 10%）→ 实际到账 = 兑换所得 + 赠送
- 赠送部分需完成 X 倍流水后才能提现（和充值奖金一样的稽核机制）
- 兑换金额不能超过当前法币余额

**汇率显示格式统一为 `1 BSB = X 法币`**，例如 1 BSB = 2,620 VND。

### 3.4 提现 — 把钱取出去

提现是整个钱包里**规则最复杂**的部分，因为涉及合规、风控、KYC。

**KYC 与提现的关系：**
- 没做 KYC 认证的用户：只能 USDT 提现（加密货币不需要身份验证）
- 做了 KYC 的用户：法币/电子钱包/USDT 全部渠道都能用

**提现渠道路由规则（关键规则）：**

这个规则的核心思想是"怎么充的就怎么提"，防洗钱：

| 你是怎么充的 | 允许你怎么提 | 通俗理解 |
|------------|------------|---------|
| 只充了法币 | 只能法币提 | 人民币进，人民币出 |
| 只充了 USDT | 只能 USDT 提 | U 进 U 出 |
| 法币 + USDT 都充过 | 法币提 | 按法币通道走 |
| 法币充值后换成了 BSB | 法币提 | 追溯到最初的充值方式 |

**提现账户管理：**
- 首次提现需要填写银行卡/电子钱包账户信息
- 账户持有人必须和 KYC 认证姓名一致（模糊匹配：忽略大小写和多余空格）
- 填写后**用户不能自己修改**，后续提现自动使用保存的账户
- 这是防止盗号后改提现账户的安全措施

**提现限额和手续费：**
- 每日提现限额、单笔最低提现金额、手续费比例——这些由财务模块在后台配置
- 超限弹窗提示，低于最低额弹窗提示

**提现流程（我的理解）：**
1. 用户选提现方式 → 输入金额 → 提交
2. 钱包创建提现订单（T+16位）→ **立即冻结用户钱包对应金额**
3. 订单进入"待风控审核"状态 → 风控人员在 B 端审核
4. 风控通过 → "待财务审核" → 财务人员审核
5. 财务通过 → "待出款" → 系统发起代付到三方通道
6. 通道返回结果：成功 → 扣冻结金额；失败 → 保持冻结，需人工处理

**驳回时：风控驳回/财务驳回/出款驳回，都是解冻金额归还用户。**
**出款失败时：不自动解冻，保持冻结等人工介入。** 这个设计很谨慎，避免自动解冻后用户重复提现。

### 3.5 账变记录 — 看历史流水

分 4 个 Tab：充值 / 提现 / 兑换 / 奖励。

**强制规则：当前选中的币种 = 记录展示的币种。** 切换币种后记录列表立即刷新为对应币种的数据。不支持跨币种混合查看。

每种记录有不同的列表字段和详情字段。失败订单提供"更换方式重试"的入口。

### 3.6 非功能性要求（后端必须兜住的底线）

这些不是一个具体的功能按钮，但是后端实现时必须做到的：

1. **对账一致性**：账务流水记录和账户余额之间必须能对上，需要有校验机制
2. **幂等性**：充值回调可能重复到达（网络重试），必须保证不会重复入账
3. **KYC 强验证**：法币提现必须通过 KYC，后端不能只靠前端拦截
4. **姓名一致性**：提现账户持有人必须和 KYC 姓名匹配
5. **提现账户不可自改**：只有首次保存，后续后端拒绝修改请求
6. **异常人工处理**：大额或异常频率触发人工审核

## 四、币种配置模块 — 我理解的核心功能

### 4.1 这个模块存在的意义

钱包的所有金额操作都依赖"币种信息"和"汇率"。币种配置就是给整个系统提供这些基础数据的地方。

**打个比方：** 如果钱包是一台售货机，币种配置就是告诉售货机"能收哪些硬币、每种硬币值多少钱"。

### 4.2 币种体系

平台把所有货币统一归为三类：

| 类型 | 说明 | 示例 | 汇率特点 |
|------|------|------|---------|
| 法定货币 | 各国法币 | VND、IDR、THB | 实时浮动，从三方 API 获取 |
| 加密货币 | 区块链资产 | USDT（未来可能 BTC、ETH） | 实时浮动 |
| 平台货币 | 平台自己的虚拟币 | BSB | 固定锚定 USDT（10 BSB = 1 USDT） |

**每个币种对应一套独立的钱包**，配置层统一管理，业务规则层做差异化处理。

### 4.3 基准币种

这是一个核心概念：**系统内部统一用一种币种作为核算基准**，所有币种的金额在做统计、对账、报表时都换算为基准币种。

- 推荐设为 USDT
- **只能设置一次，设完不可改**（后台提交按钮会隐藏）
- 设置时需要二次确认弹窗

**基准币种的用途：** 活动奖金计算、多币种投注统计、钱包资产统计、财务对账报表——这些跨币种的数据都需要有一个统一的"度量衡"。

### 4.4 汇率机制

汇率是这个模块最复杂的部分，我理解的完整链路：

**第一步：获取实时汇率**
- 定时从 **3 个三方汇率 API** 获取数据，取**平均值**
- 为什么用 3 个：避免单一 API 宕机或返回异常数据

**第二步：计算平台汇率**
- 平台汇率不是每次都跟着实时汇率变：只有当偏差达到配置的阈值时才更新
- 公式：`偏差 = |实时汇率 - 平台汇率| / 实时汇率 × 100%`
- 偏差 >= 阈值 → 更新平台汇率，生成汇率日志
- 偏差 < 阈值 → 平台汇率保持不变
- 这个设计的目的：避免汇率微小波动导致频繁更新，给运营一个稳定的汇率环境

**第三步：计算入款/出款汇率**
- `入款汇率 = 实时汇率 × (1 + 浮动%)`  → 用户充值时用这个，平台多收一点
- `出款汇率 = 实时汇率 × (1 - 浮动%)`  → 用户提现时用这个，平台少付一点
- 浮动百分比由运营在后台配置（每个币种独立）
- 这就是平台的汇率差价利润空间

**汇率显示精度：** 法定货币 2 位小数，加密货币 8 位小数。

**BSB 特殊处理：** BSB 的汇率是固定锚定（10 BSB = 1 USDT），不走三方 API，不可配置，不可禁用。

### 4.5 后台配置页面

运营人员在 B 端后台看到的是一个币种列表，每行 20 个字段。可以做的操作：
- 编辑：只能改汇率浮动、图标、阈值、状态（其他字段只读或自动生成）
- 启用/禁用：禁用的币种 C 端不可见、不可充提
- 设置基准币种：一次性操作
- 查看汇率日志：每次平台汇率变更都有记录

### 4.6 汇率日志

每次平台汇率因偏差超阈值而更新时，自动生成一条日志：
- 字段：日志编号、更新时间、币种代码、实时汇率、平台汇率、入款汇率、出款汇率
- 运营可以按时间范围和币种筛选
- 位于操作日志页面的独立 Tab

## 五、财务管理模块 — 虽然不是我做，但必须搞清楚

### 5.1 为什么必须清楚

财务管理模块和钱包模块有**大量的 RPC 调用关系**：
- 充值成功 → 财务调钱包入账
- 提现创建 → 钱包冻结余额
- 提现审核驳回 → 财务调钱包解冻
- 出款成功 → 财务调钱包扣冻结
- 人工加减款 → 财务调钱包加/减余额
- 补单 → 财务调钱包入账

钱包需要暴露的 6 个核心 RPC 接口（CreditWallet、DebitWallet、FreezeBalance、UnfreezeBalance、DeductFrozenAmount、QueryWalletBalance）就是为了对接财务模块。

反过来，钱包也需要调财务模块获取支付方式配置和通道信息。

### 5.2 财务管理的 6 大功能块

**① 通道配置**
- 代收通道（充值用）和代付通道（提现出款用）
- 通道由开发团队集成时录入，运营只能编辑费率/限额/权重/状态
- 有个轮询算法：综合得分 = 成功率系数 × 时间系数 × 预警系数 × 权重，得分最高的通道接单

**② 支付配置**
- 配置每个币种可用的充值方式（名称/图标/档位/金额范围/关联通道）
- 配置每个币种可用的提现方式（比充值简单，无档位和通道关联）

**③ 充值记录**
- 查看所有充值订单，字段丰富（24个字段），包含平台订单号、通道订单号、各种金额换算
- 订单号格式：C + 16 位随机数

**④ 提现记录**
- 提现有 7 个状态的完整流转：待风控审核 → 待财务审核 → 待出款 → 出款成功/失败
- 风控审核、财务审核、锁定/解锁操作都在这里
- 订单号格式：T + 16 位随机数

**⑤ 人工修正**（3 种操作）
- **充值补单（B+16位）**：用户实际付了钱但系统没收到回调，运营手动补单入账
- **人工加款（A+16位）**：给用户加钱（运营补偿/奖励/测试等），可以指定加到哪个子钱包
- **人工减款（M+16位）**：从用户扣钱（异常追回/风控处罚等），可以指定从哪个子钱包扣，不能超过当前余额

加款和减款都支持同时操作 4 种子钱包（中心/奖励/代理/主播），每个钱包独立指定金额和稽核倍数。

**⑥ 数据统计**
- 通道统计（代收/代付成功率、金额、手续费等）
- 充值统计（订单数、人数、人均等）
- 提现统计（各状态分布、手续费、出款金额等）
- 时间基准 UTC+0，运营可配站点时区

## 六、贯穿全局的关键业务规则

### 6.1 钱的进出规则

```
钱进来的途径：充值、投注返奖、活动/任务奖励、人工加款、补单
钱出去的途径：提现、消费（买礼物/道具/投注）、人工减款
```

### 6.2 币种隔离原则

这是**铁律**：每个币种的数据完全隔离。余额不能混、记录不能混、操作不能跨币种。用户切换币种就像切换了一个完全不同的钱包。

### 6.3 BSB 的特殊地位

BSB 是平台自己的虚拟币，特殊在：
- 固定汇率：10 BSB = 1 USDT，不随市场浮动
- 不可配置：汇率/代码/类型/状态都锁死
- 不可禁用：基准币种不可隐藏
- 法币可以单向换成 BSB，但 BSB 不能直接换回法币
- BSB 的充值方式最多（USDT + 银行 + 电子钱包都支持）
- BSB 提现时需要选目标法币/USDT，显示汇率换算

### 6.4 稽核/流水机制

"稽核"简单说就是**防薅羊毛机制**：平台给你的赠送不是白给的，你得在平台上消费够一定金额才能提现。

涉及稽核的场景：
- 充值选择了奖金活动 → 奖金入奖励钱包 → 需完成 N 倍流水
- 法币兑换 BSB 时的赠送部分 → 需完成 N 倍流水
- 人工加款/补单 → 也记录稽核倍数

### 6.5 订单号命名规范

| 类型 | 前缀 | 格式 | 归属模块 |
|------|------|------|---------|
| 充值 | C | C + 16 位随机数 | 钱包/财务 |
| 提现 | T | T + 16 位随机数 | 钱包/财务 |
| 补单 | B | B + 16 位随机数 | 财务 |
| 人工加款 | A | A + 16 位随机数 | 财务 |
| 人工减款 | M | M + 16 位随机数 | 财务 |

### 6.6 状态机

```
充值订单：  待支付 → 支付成功 / 支付失败
           （简单两态，成功就入账，失败可补单）

提现订单：  待风控审核 → 风控驳回（终态，解冻）
                      → 待财务审核 → 财务驳回（终态，解冻）
                                    → 待出款 → 出款失败（冻结不动，等人工）
                                              → 出款成功（终态，扣冻结）
           （三层审核，驳回就解冻，失败不解冻）

人工修正：  待审核 → 审核拒绝 / 操作成功
           （补单/加款/减款都是这个流程）
```

## 七、跨模块数据流向（我的理解）

### 7.1 充值链路

```
用户App → [钱包C端] 创建充值订单
              ↓
         [钱包] 调 [财务] RPC：匹配通道、创建通道订单
              ↓
         [财务] 返回支付信息（三方页面URL / QR码 / 收款地址）
              ↓
         用户完成支付
              ↓
         [三方通道] 回调 → [财务] 接收并验证
              ↓
         [财务] 调 [钱包] RPC CreditWallet：入账
              ↓
         用户钱包余额增加
```

### 7.2 提现链路

```
用户App → [钱包C端] 创建提现订单
              ↓
         [钱包] 冻结用户余额，订单状态=待风控审核
              ↓
         [财务B端] 风控审核 → 通过/驳回
              ↓                    ↓
         [财务B端] 财务审核    [钱包] RPC 解冻（驳回时）
              ↓
         [财务] 调通道发起代付
              ↓
         [三方通道] 回调 → [财务] 接收
              ↓
         成功: [财务] 调 [钱包] RPC DeductFrozenAmount
         失败: 保持冻结，等人工处理
```

### 7.3 人工修正链路

```
运营B端 → [财务B端] 创建补单/加款/减款申请
              ↓
         [财务] 调 [钱包] RPC QueryWalletBalance：查当前余额
              ↓
         审核通过
              ↓
         补单/加款: [财务] 调 [钱包] RPC CreditWallet → 余额增加
         减款:     [财务] 调 [钱包] RPC DebitWallet → 余额减少
```

### 7.4 汇率更新链路

```
定时任务 → 拉取 3 个三方汇率API → 取平均值 = 实时汇率
              ↓
         计算偏差 = |实时 - 平台| / 实时 × 100%
              ↓
         偏差 >= 阈值？
         ├── 是 → 更新平台汇率 + 入款汇率 + 出款汇率 → 写汇率日志
         └── 否 → 不动
```

## 八、我需要特别注意的设计要点

### 8.1 后端必须做的校验（不能只靠前端）

| 校验项 | 原因 |
|--------|------|
| 提现前检查 KYC 状态 | 安全合规，前端可以被绕过 |
| 提现账户持有人 = KYC 姓名 | 防盗号提现 |
| 充值回调幂等处理 | 网络重试可能重复回调 |
| 重复转账检测（同用户+同币种+同金额） | 防误操作/恶意刷单 |
| 减款金额 <= 当前余额 | 余额不能为负 |
| 基准币种只能设一次 | 设完不可改，后端必须校验 |
| 禁用币种不可充提 | 后端必须校验，不能只靠前端隐藏 |

### 8.2 钱包服务暴露给外部的 RPC 接口

这 6 个接口是钱包对外的"合同"，其他模块（不只是财务）都可能调用：

| 接口 | 谁在调 | 什么时候调 |
|------|--------|----------|
| CreditWallet | 财务（充值/补单/加款）、游戏（返奖）、活动（奖励） | 需要给用户加钱的时候 |
| DebitWallet | 财务（减款）、游戏（投注）、商城（购买） | 需要从用户扣钱的时候 |
| FreezeBalance | 提现创建时 | 用户申请提现的时候 |
| UnfreezeBalance | 审核驳回时 | 提现被拒绝的时候 |
| DeductFrozenAmount | 出款成功时 | 钱实际打出去了 |
| QueryWalletBalance | 财务（加减款表单）、其他模块 | 需要看用户余额的时候 |

### 8.3 预评审阶段的不确定性

以下事项可能在正式评审后变动：

1. 充值回调最终由哪个模块接收和处理
2. 充值订单表归属钱包还是财务
3. 奖金活动数据由运营活动模块还是财务模块维护
4. 消费优先级扣款是否需要独立 RPC 接口
5. 提现账户是否未来支持 B 端运营修改
6. 具体的限额数值、手续费率、稽核倍数（运营配置，非固定值）
