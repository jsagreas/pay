# 第一部分：币种配置模块

> 来源：币种配置需求文档（24张）+ 币种配置产品原型（3张）
> 定位：整个资金体系的基础底座，所有涉及资金的场景都依赖它

## 1. 文档说明

币种配置模块为平台提供统一的币种管理和汇率管理能力，服务于钱包、充值、提现、兑换、投注、结算、统计等所有涉及资金的业务场景。

## 2. 资金钱包体系

这是币种配置文档中描述的钱包架构设计（需求文档2-1系列，共7张），是整个资金体系的顶层架构。

### 2.1 五种钱包

| 钱包类型 | 用途 | 资金来源 | 提现规则 | 当前状态 |
|---------|------|---------|---------|---------|
| 中心钱包 | 用户主钱包，日常消费扣款来源 | 充值、稽核完成的奖励金转入、主播/代理钱包转入 | 完成稽核流水后可提现 | 需实现 |
| 奖励钱包 | 活动/任务/投注奖金暂存 | 活动返奖、任务返奖、投注返奖 | 需完成稽核流水后转入中心钱包才可提现 | 需实现 |
| 主播钱包 | 主播收益独立管理 | 礼物打赏收入、主播活动奖励 | 可直接提现，也可转入中心钱包 | 需实现 |
| 代理钱包 | 代理佣金独立管理 | 代理分成 | 可直接提现，也可转入中心钱包 | 预留 |
| 场馆钱包 | 游戏场馆内临时钱包 | 进入场馆时从中心+奖励钱包转入 | 退出场馆时自动转回原钱包 | 预留 |

### 2.2 钱包间关系与规则

- **资金钱包余额** = 中心钱包余额 + 奖励钱包余额（对用户展示的"总余额"）
- **消费优先级**：先扣中心钱包，不足部分再扣奖励钱包
- **混合投注返奖**：按投注时中心/奖励钱包的金额占比，分别返奖到对应钱包
- **稽核流水**：奖励钱包资金需完成指定倍数的投注流水后，才能转入中心钱包提现
- **场馆钱包转入**：进入游戏场馆时，用入款汇率将法币转为场馆币种；退出时用进入时记录的汇率原路转回

### 2.3 每种钱包的操作权限

| 操作 | 中心钱包 | 奖励钱包 | 主播钱包 | 代理钱包 | 场馆钱包 |
|------|---------|---------|---------|---------|---------|
| 充值入账 | ✅ | ❌ | ❌ | ❌ | ❌ |
| 活动/任务奖励 | ❌ | ✅ | ❌ | ❌ | ❌ |
| 投注返奖 | ✅(按比例) | ✅(按比例) | ❌ | ❌ | ❌ |
| 礼物打赏收入 | ❌ | ❌ | ✅ | ❌ | ❌ |
| 代理分成 | ❌ | ❌ | ❌ | ✅ | ❌ |
| 投注扣款 | ✅(优先) | ✅(补充) | ❌ | ❌ | ❌ |
| 提现 | ✅ | ❌(需先转入中心) | ✅ | ✅ | ❌ |
| 转入中心 | — | ✅(完成稽核后) | ✅ | ✅ | ✅(退出场馆时) |

## 3. 币种类型划分

### 3.1 三种币种类型

| 类型 | 代表 | 特点 |
|------|------|------|
| 法定货币（Fiat） | VND（越南盾）、IDR（印尼盾）、THB（泰铢） | 对应真实国家货币，用户本地充值/提现 |
| 加密货币（Crypto） | USDT、BTC、ETH | 区块链数字资产，跨境通用 |
| 平台货币（Platform） | BSB | 平台专属虚拟币，锚定USDT（10BSB = 1USDT），用于统一游戏场馆内的币种 |

### 3.2 币种与地区关系

- VND → 越南市场
- IDR → 印尼市场
- THB → 泰国市场
- USDT → 跨地区通用
- BSB → 平台内部通用（游戏场馆使用）

## 4. 基准币种

- 推荐选择USDT作为基准币种
- **设置后不可更改**（一次性配置）
- 用途：
  - 活动奖金计算的统一基准
  - 多币种财务报表/统计的换算基准
  - 对账审计的统一口径
- 所有币种金额在账务层需换算为基准币种进行汇总统计

## 5. 汇率体系

### 5.1 汇率概念

| 概念 | 定义 | 用途 |
|------|------|------|
| 实时汇率 | 接入3家三方汇率API，取平均值 | 作为计算入款/出款汇率的基础 |
| 平台汇率 | 初始值=实时汇率，后续按阈值规则自动更新 | 统计报表、对账 |
| 汇率浮动% | 后台可配置，每个币种独立设置 | 计算入款/出款汇率的浮动比例 |
| 入款汇率 | 实时汇率 × (1 + 汇率浮动%) | 充值、进入游戏场馆时使用 |
| 出款汇率 | 实时汇率 × (1 - 汇率浮动%) | 提现、退出游戏场馆时使用 |
| 偏差值 | \|实时汇率 - 平台汇率\| / 实时汇率 × 100 | 判断是否需要更新平台汇率 |
| 阈值 | 后台可配置，每个币种独立 | 偏差值 >= 阈值时，平台汇率自动更新为当前实时汇率 |

### 5.2 汇率应用场景

| 场景 | 使用的汇率 | 说明 |
|------|----------|------|
| 充值（跨币种） | 入款汇率 | 例如用VND充值BSB时 |
| 提现（跨币种） | 出款汇率 | 例如BSB提现为VND时 |
| 进入游戏场馆 | 入款汇率 | 法币转为场馆币种，并记录该汇率快照 |
| 退出游戏场馆 | 进入时的汇率快照 | 用进入时记录的汇率原路转回，保证用户不因汇率波动亏损 |
| 统计/报表/对账 | 平台汇率 | 所有币种统一换算为基准币种进行统计 |

### 5.3 金额展示规则

| 币种类型 | 小数位数 | 千分位符号 | 示例 |
|---------|---------|----------|------|
| 法币（VND/IDR） | 整数（0位） | `.` | 1.000.000 |
| 法币（THB） | 2位 | `,` | 1,000.00 |
| 加密货币（USDT） | 8位 | — | 0.00000001 |
| 平台货币（BSB） | 2位 | `,` | 1,000.00 |

# 第二部分：多币种钱包模块

> 来源：多币种钱包需求文档（30张）+ 多币种钱包产品原型（30张）
> 定位：C端用户的核心资金操作入口

## 1. 需求背景与目标

### 1.1 背景

平台面向多个国家/地区运营（越南、印尼、泰国等），用户使用不同法定货币。需要一个统一的多币种钱包系统，让用户可以在不同币种之间切换操作，同时支持USDT等加密货币和平台币BSB。

### 1.2 目标

- 支持多币种并行管理，每个币种独立余额和账本
- 提供充值、提现、兑换、记录查看等完整的资金操作能力
- 保证资金安全和对账一致性

### 1.3 适用范围

当前版本支持：BSB、VND、IDR、THB

## 2. 钱包总体设计

### 2.1 核心原则

- **按币种维度隔离**：每个币种有独立的余额和账本，互不影响
- **不支持跨币种混合展示与操作**：切换币种 = 切换所有操作的数据源（余额、充值方式、提现方式、记录）

### 2.2 金额展示规则

| 币种 | 千分位符号 | 小数位 | 示例 |
|------|----------|--------|------|
| VND | `.` | 0（整数） | 1.000.000 |
| IDR | `.` | 0（整数） | 1.000.000 |
| THB | `,` | 2 | 1,000.00 |
| BSB | `,` | 2 | 1,000.00 |

### 2.3 页面交互（原型2-1~2-3个人中心）

- 个人中心页面顶部展示当前选中币种的钱包余额
- 余额下方展示充值、提现、兑换入口按钮
- 支持币种切换（下拉或弹窗选择）
- 切换币种后，页面所有数据联动刷新

## 3. 充值模块

### 3.1 充值方式总览

| 充值方式 | 承载页面 | 回调机制 | 适用场景 | 原型参考 |
|---------|---------|---------|---------|---------|
| USDT充值（TRC20/ERC20/BEP20） | 平台自有页面展示钱包地址 | 区块链监听到账 | 所有币种通用 | 原型4-1 |
| 银行转账 | 跳转三方支付页面 | 三方回调通知 | VND/IDR/THB对应国家银行 | 原型5-x |
| 电子钱包/扫码（QRIS/PromptPay等） | 跳转三方支付页面 | 三方回调通知 | IDR（QRIS）、THB（PromptPay） | 原型5-x |

### 3.2 充值流程（需求文档5-7）

```
用户选择币种 → 选择充值方式 → 输入金额 → (可选)选择奖金活动 → 提交订单
    ↓
服务端：轮询配置分配通道 → 创建平台订单(状态:待支付)
    ↓                    ↓（通道不可用）
通道创建订单              → 提示"通道维护中"
    ↓
返回支付页面/USDT地址 → 用户完成支付
    ↓
通道回调支付结果 → 服务端验证金额 → 订单状态改为充值成功 → 入账到用户钱包
```

### 3.3 币种切换规则（原型3-1~3-4）

- 切换充值币种时，充值方式列表联动刷新为该币种支持的方式
- 不同币种支持的充值方式不同（如VND支持银行转账，IDR支持QRIS）

### 3.4 USDT充值特殊规则（需求文档5-2）

- 平台展示USDT接收地址和二维码，支持TRC20/ERC20/BEP20三种链
- 用户复制地址后在自己的加密钱包中转账
- 通过区块链监听确认到账
- **游客也可以USDT充值**，登录后解锁全部充值方式

### 3.5 银行充值（需求文档5-3）

- 跳转到三方支付页面完成银行转账
- 三方回调通知充值结果
- 支持金额范围由后台配置（最低/最高）

### 3.6 扫码充值（需求文档5-4）

- 跳转到三方支付页面展示二维码
- 用户扫码支付后三方回调通知

### 3.7 重复转账拦截规则（需求文档5-5）

- 拦截维度：同UserID + 同币种 + 同金额
- 1-2笔：弹窗警告，用户可选择继续或取消
- 第3笔起：强制拦截，不允许继续提交
- 目的：防止用户误操作重复充值

### 3.8 额外奖金选择规则（需求文档5-6）

- 充值时可选择参加一个奖金活动（可选，非强制）
- 每个活动有最低充值门槛（如最低充值100才能参加）
- 选择后充值金额会获得额外奖金（如充100送20%=20）
- 奖金部分进入**奖励钱包**，需完成X倍流水后才能转入中心钱包提现
- 原型6-1~6-2展示了奖金选择界面

### 3.9 通用回调页面（原型7-1~7-4）

- 从三方支付页面返回平台后，展示订单结果页面
- 展示：订单状态（成功/失败/处理中）、金额、时间
- 支付中的订单展示等待提示

## 4. 兑换模块

### 4.1 兑换规则（需求文档6）

- **方向**：仅支持法币 → BSB（单向，不可逆）
  - VND → BSB
  - IDR → BSB
  - THB → BSB
- **汇率口径**：1 BSB = X 法币（展示形式）
- **赠送规则**：兑换金额额外赠送10%
  - 赠送部分进入**奖励钱包**，需完成X倍流水后可提现
  - 兑换本身部分进入**中心钱包**
- **到账计算**：实际到账BSB = 兑换获得 + 赠送（10%）

### 4.2 兑换页面（原型8）

- 展示当前法币余额、BSB汇率
- 用户输入兑换金额
- 实时计算并展示：兑换获得BSB + 赠送BSB + 合计到账BSB

## 5. 提现模块

### 5.1 KYC与提现的关系（需求文档7-1，原型9-1~9-4）

- **未KYC认证**：点击提现时弹窗提示，提供两个选择：
  - "去认证" → 跳转KYC页面
  - "继续提现" → 仅允许USDT提现（法币提现需KYC）
- **已KYC认证**：直接进入提现方式选择页面

### 5.2 提现方式对照表

| 提现方式 | 适用地区 | 到账时效 | 需KYC | 原型参考 |
|---------|---------|---------|-------|---------|
| 银行卡转账 | 越南/印尼/泰国 | T+1 | 是 | 原型10-1~10-3 |
| 电子钱包 | 越南/印尼/泰国 | 即时 | 是 | 原型10-4~10-6 |
| USDT（TRC20/ERC20/BEP20） | 通用 | 即时 | 否（但有限额差异） | 原型10-4~10-6 |

### 5.3 提现流程（需求文档7-1）

```
用户点击提现 → KYC校验
    ↓（未认证）              ↓（已认证）
弹窗选择"去认证/继续"       直接进入
    ↓
选择提现方式 → 选择/填写收款账户 → 输入提现金额 → 提交
    ↓
服务端创建订单(状态:审核中) → 三级审核流程
    ↓
风控审核 → 财务审核 → 出款审核 → 通道出款 → 完成
```

### 5.4 提现规则（需求文档7-1）

- 法币充值 → 只能法币提现
- USDT充值 → 只能USDT提现
- 法币+USDT混合充值 → 只能法币提现
- 法币 → 兑换BSB → 提现时按法币提现

简单说：**提现方式由充值来源决定**，不允许充法币提USDT

### 5.5 银行卡提现（需求文档7-2）

- 首次提现需填写银行卡信息：银行名称、卡号、持有人姓名
- **持有人姓名 = KYC认证姓名**，自动填充且不可编辑
- 填写后的账户信息保存，后续提现可直接选用

### 5.6 电子钱包提现（需求文档7-3）

- 类似银行卡，需填写电子钱包账号信息
- 按国家不同支持不同的电子钱包（如越南的MoMo/ZaloPay等）
- 账户信息同样保存复用

### 5.7 USDT提现（需求文档7-4）

- 填写USDT接收地址，选择链类型（TRC20/ERC20/BEP20）
- BSB提现时展示汇率换算（BSB → USDT → 用户到账金额）

### 5.8 提现限制（需求文档7-5）

- 单笔限额：后台可配置最低/最高金额
- 单日限额：后台可配置每日提现上限
- 手续费：后台可配置（固定金额或比例）
- 可提现余额 = 中心钱包余额 - 冻结金额

### 5.9 各地区提现页面差异（原型10-1~10-6）

- **越南（10-1）**：银行卡+电子钱包+USDT
- **印尼（10-2）**：银行卡+电子钱包+USDT
- **泰国（10-3）**：银行卡+PromptPay+USDT
- 各地区的银行列表、电子钱包类型不同，但交互流程一致

## 6. 记录模块

### 6.1 记录分类（需求文档8-1）

四个Tab页签：

| Tab | 内容 | 数据来源 |
|-----|------|---------|
| 充值 | 充值订单记录 | 用户发起的充值订单 |
| 提现 | 提现订单记录 | 用户发起的提现订单 |

### 6.2 核心规则（需求文档8-2）

- **当前币种 = 记录币种**：只展示当前选中币种的记录，不支持跨币种混合展示
- 切换币种后记录列表联动刷新

### 6.3 列表页面（需求文档8-3，原型11-1~11-2）

- 展示字段：订单号、金额、状态、时间、方式/类型
- 支持按时间范围筛选
- 列表按时间倒序排列

### 6.4 详情页（需求文档8-4）

- 展示完整订单信息：
  - 订单号
  - 金额明细（申请金额、手续费、实际到账）
  - 订单状态（成功/失败/处理中/审核中等）
  - 创建时间、完成时间
  - 充值/提现方式
  - 失败原因（如有）

## 7. 非功能性需求（需求文档9）

| 维度 | 要求 |
|------|------|
| 对账一致性 | 账务流水与账户余额必须强一致，任何资金变动都要有流水记录 |
| 幂等性 | 充值回调幂等处理，防止重复入账（同一订单多次回调只入账一次） |
| KYC校验 | 提现前校验KYC状态；法币提现必须完成KYC |
| 提现账户白名单 | 提现账户持有人姓名必须与KYC认证姓名完全一致 |
| 大额/异常审核 | 大额提现或异常频率提现触发人工审核 |

# 第三部分：财务管理模块（辅助理解，非我主责）

> 来源：财务管理需求文档（25张）+ 财务管理产品原型（25张）
> 理解目的：明确钱包模块与财务模块的依赖关系，确定RPC接口契约

## 1. 文档说明

- 统一的财务管理与数据统计模块
- 覆盖：支付通道配置、充值提现记录管理、资金数据统计分析
- 8个子功能：通道配置、支付配置、充值记录、提现记录、人工修正、通道统计、充值统计、提现统计

## 2. 通道配置

### 2.1 代收通道（充值用）

- 列表字段：通道名称、通道编码、代收渠道、币种、支持金额范围、手续费、代收费率、状态
- 操作：编辑、启用/禁用、轮询配置
- 编辑弹窗：通道编码、名称、代收渠道（下拉选择）、币种、费率、手续费模式、最低/最高金额

### 2.2 代付通道（提现用）

- 与代收通道结构类似
- 额外字段：中间价位调整、代付通道层级（一级/二级...最终走最后一级通道）、单笔限额

### 2.3 通道轮询规则

- 通道之间按**配比**分配流量
  - 例如：A通道配比=1，B通道配比=2 → A获得33.3%流量，B获得66.7%
- 半小时切换一次通道
- 最少2个、最多5个通道参与轮询
- 通道不可用时自动跳到下一个有效通道
- 每个配比值为1~100的正整数

## 3. 支付配置

### 3.1 充值方式配置

- 列表字段：充值方式、币种、关联代收通道、图标、支持金额范围、手续费、大额手续费、排序、状态
- 新增/编辑弹窗：支付编码、名称、币种、代收渠道、代收通道（下拉）、图标上传（PNG≤200kb 1:1）、金额范围、手续费、排序
- 排序：数值越小越靠前
- **与钱包的关系**：钱包C端展示的充值方式列表，数据来源就是这里的配置

### 3.2 提现方式配置

- 与充值方式配置结构类似，关联的是代付通道
- **与钱包的关系**：钱包C端展示的提现方式列表，数据来源就是这里的配置

## 4. 充值记录

### 4.1 记录字段

订单号、充值方式、通道名称、充值金额、手续费、到账金额、用户ID、用户编码、平台订单号、三方订单号、订单时间、完成时间、订单状态、付款信息、上游金额、上游手续费、手续费差额、代收费率、收款渠道、代收通道、回调状态

### 4.2 订单状态

充值成功 / 充值失败 / 支付中 / 10分钟已失效

### 4.3 充值流程（泳道图：客户端→服务端→通道）

这是从财务模块视角看的充值流程，与钱包模块的充值流程对应：
1. 客户端提交充值 → 服务端轮询分配通道 → 创建平台订单
2. 通道创建订单 → 返回支付页面 → 用户支付
3. 通道回调结果 → 服务端验证金额 → 充值成功 → **调用钱包入账接口上分**

### 4.4 导出功能

- 导出需要密码校验
- 如果开启了谷歌验证码，需要密码+谷歌验证码双重验证

## 5. 提现记录

### 5.1 记录字段

订单号、提现方式、通道名称、提现金额、手续费、到账金额、用户ID、用户编码、平台订单号、三方订单号、订单时间、完成时间、订单状态、风控审核、财务审核、出款审核、收款信息、上游金额、上游手续费、手续费差额、代付费率、提现渠道、代付通道

### 5.2 订单状态

审核中 / 待出款 / 出款中 / 已完成 / 已拒绝 / 已锁定 / 已驳回

### 5.3 三级审核流程

```
用户提交提现 → 订单状态:审核中
    ↓
风控审核 → 通过/驳回
    ↓（通过）
财务审核 → 通过/驳回
    ↓（通过）
出款审核 → 通过/驳回
    ↓（通过）
通道出款 → 出款成功 → 订单状态:已完成
```

- 任何一级驳回，订单状态变为已驳回，**钱包需解冻余额**
- 支持锁定/解锁/强制解锁操作

### 5.4 提现流程（泳道图：客户端→服务端→通道）

1. 客户端提交提现 → 服务端创建订单(审核中) → **调用钱包冻结余额**
2. 三级审核全部通过 → 通道发起出款
3. 通道回调出款结果 → 出款成功 → **调用钱包确认扣款**
4. 审核驳回 → **调用钱包解冻余额**

## 6. 人工修正

### 6.1 充值补单

- 场景：用户充值成功但系统未到账（回调丢失等异常）
- 操作：管理员新增补单 → 关联原始充值订单 → 审核通过后 → **调用钱包入账**
- 审核人不可与提交人相同

### 6.2 人工加款

- 场景：补偿用户、活动奖励手动发放等
- 操作：选择用户 → 选择币种 → 输入加款金额 → 审核通过后 → **调用钱包加款接口**
- 需指定目标钱包类型（中心/奖励/主播/代理）
- 审核人不可与提交人相同

### 6.3 人工减款

- 场景：异常资金回收、违规扣款等
- 操作：选择用户 → 选择币种 → 输入减款金额 → 选择减款类型 → 审核通过后 → **调用钱包减款接口**
- 审核人不可与提交人相同

### 6.4 订单号规则

| 操作类型 | 订单号前缀 | 格式 |
|---------|----------|------|
| 充值补单 | B | B + 16位随机数 |
| 人工加款 | A | A + 16位随机数 |
| 人工减款 | M | M + 16位随机数 |

## 7. 数据统计

### 7.1 时区处理（重要）

- 管理后台可设置站点时区（默认UTC+0）
- 前端展示按站点时区，服务端存储和查询按UTC+0
- 查询时需将站点时区时间还原为UTC+0再查数据库
- 时区列表通过IANA Time Zone库获取

### 7.2 代收通道统计

- 按通道维度统计：代收总金额、到账总金额、代收总笔数、代收手续费
- 明细字段：通道名称、币种、代收总金额、代收手续费、到账总金额、成功/失败/超时笔数、总笔数、成功率、平均代收时间

### 7.3 代付通道统计

- 与代收通道统计结构类似，维度换成代付通道

### 7.4 充值统计

- 按**日期+币种**维度统计
- 字段：日期、币种、充值总金额、手续费、到账总金额、成功/失败/超时笔数、总笔数、成功率、人均充值金额、人均充值笔数、充值人数

### 7.5 提现统计

- 按**日期+币种**维度统计
- 字段：日期、币种、提现总金额、手续费、到账总金额、成功/失败/超时笔数、总笔数、成功率、平均出款金额、审核驳回数、提现人数
- 额外指标：实际出款总额、出款成功率、平均出款时间
