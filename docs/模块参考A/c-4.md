## 一、技术栈全景对比

### 1.1 基础设施对比

| 维度 | 参考工程（Java） | 我们的工程（Go） | 差异影响 |
|------|-----------------|-----------------|---------|
| 语言 | Java 8+ / Spring Cloud | Go 1.25 / CloudWeGo | 生态差异大，设计思想可迁移 |
| Web 框架 | Spring Boot + Feign | Kitex（RPC）+ Hertz（HTTP Gateway） | 通信模型不同：REST vs Thrift RPC |
| ORM | MyBatis-Plus（DTO=Entity，XML SQL） | GORM Gen（model/query/repo 三层分离） | 我们分层更清晰 |
| 数据库 | MySQL | TiDB（MySQL 协议兼容） | SQL 层面几乎完全兼容 |
| 缓存 | Redis + Redisson（分布式锁） | Redis（go-redis v9） | 我们用行锁替代 Redisson |
| 消息队列 | RabbitMQ（重度依赖，6+ 队列） | Kafka/NATS JetStream（轻度，主要是通知） | ★ 最大架构差异 |
| 配置中心 | K8S SVC（硬编码 URL + bootstrap.yml） | ETCD（100% 配置驱动，热更新） | 我们更灵活 |
| 服务发现 | K8S DNS（http://${k8s.svc.xxx}） | ETCD 服务注册发现 | 模式不同但功能等价 |
| 定时任务 | XXL-Job | 待定（cron / ETCD 分布式锁） | 参考工程方案可借鉴 |
| 共享库 | p9-core-lib（Maven 依赖） | common/pkg（go.work replace） | 分发机制不同，目标一致 |

### 1.2 架构层次对比

```
参考工程（Java / 3 层）                         我们的工程（Go / 4 层）
┌─────────────────────────┐                   ┌─────────────────────────┐
│ Controller              │                   │ handler.go              │
│ · 参数校验              │                   │ · 零逻辑，纯委托        │
│ · 分布式锁获取/释放      │                   ├─────────────────────────┤
│ · 调用 Service           │                   │ internal/ser/           │
├─────────────────────────┤                   │ · 参数校验              │
│ Service                 │                   │ · 业务编排              │
│ · 业务逻辑              │                   │ · 调用 rep/domain       │
│ · @Transactional 事务    │                   ├─────────────────────────┤
│ · 余额操作 + 流水写入    │                   │ internal/rep/           │
├─────────────────────────┤                   │ · 数据访问 + 事务边界   │
│ Repository（MyBatis）   │                   │ · 行锁 + SQL 条件保障   │
│ · XML SQL 定义           │                   ├─────────────────────────┤
│ · 原子余额 SQL           │                   │ internal/domain/        │
└─────────────────────────┘                   │ · 外部 RPC 封装         │
                                              │ · 复杂校验逻辑          │
      MQ Listener                             └─────────────────────────┘
┌─────────────────────────┐
│ AddCoinListener         │
│ · 4 个队列消费者         │
│ · MQ 消息 → 锁 → Service │
└─────────────────────────┘
```

**核心差异分析**：

1. **参考工程的锁在 Controller 层，事务在 Service 层**——锁包裹事务，方向正确但分散在两层。我们的行锁天然在 rep 层的事务内，锁=事务一体
2. **参考工程有独立的 MQ Listener 层**——相当于另一个"入口"（与 Controller 并列）。我们没有 MQ 入口，所有请求通过 RPC handler 进入
3. **我们多一个 domain 层**——参考工程把外部 RPC 调用（Feign）直接写在 Service 里。我们通过 domain 层隔离，Mock 和降级更方便
4. **参考工程没有独立的 Ledger 层**——余额 SQL 操作分散在 Repository + Service 中，没有统一的余额变更引擎。我们的 balance_service.go 是统一入口

### 2.2 表结构逐项对比

#### 对比 A：钱包主表（user_coin vs wallet）

| 维度 | 参考工程 user_coin | 我们的 wallet | 评估 |
|------|-------------------|-------------|------|
| **币种模型** | 单币种/单行（一个用户一种币种一条记录） | 多币种/多行（user_id + currency_code + wallet_type 复合唯一） | 我们需求更复杂，多币种是硬性要求 |
| **金额类型** | DECIMAL(20,2) / BigDecimal | BIGINT（最小货币单位） | 我们方案无浮点误差，计算更高效 |
| **可用余额** | amount - freeze_amount（计算得出） | 同思路，不单独存字段 | 一致，可借鉴 |
| **钱包类型** | wallet_type 区分转账/免转 | wallet_type 区分主钱包/奖金钱包 | 语义不同，但字段设计可参考 |
| **冗余字段** | user_name、user_state、game_type（与钱包无关的字段） | 无冗余，仅存核心余额数据 | 参考工程耦合了用户游戏状态，不应借鉴 |
| **软删除** | 无 delete_at 字段 | delete_at bigint unsigned（工程规范） | 我们有标准化软删除 |
| **唯一约束** | 无显式唯一索引（仅靠 MyBatis-Plus 查询） | uk_user_currency_type(user_id, currency_code, wallet_type, delete_at) | 我们更安全 |

**结论**：user_coin 的核心设计思路（amount + freeze_amount 双字段）与我们一致，但参考工程的单币种模型不满足我们多币种需求，冗余字段（user_state, game_type）是反模式。

#### 对比 B：流水表（coin_record vs wallet_transaction）

| 维度 | 参考工程 coin_record | 我们的 wallet_transaction | 评估 |
|------|---------------------|------------------------|------|
| **幂等保障** | ⚠️ order_no 无唯一索引 | uk_biz_no_type(biz_no, biz_type, delete_at) | 我们必须有幂等 |
| **冗余策略** | 大量冗余（user_name, merchant_name, agent_name, path 等 13 个冗余字段） | 按需冗余（仅 currency_code 等必要快照） | 参考工程过度冗余 |
| **前后快照** | coin_from / coin_to | balance_before / balance_after | ★ 一致，值得采纳 |
| **金额方向** | balance_type(1=收入,2=支出) + coin_value(正数) | tx_type 隐含方向 + amount(正数) | 参考工程显式方向更清晰 |
| **双层枚举** | coin_type(9种) + coin_detail(8种) | biz_type 单层 + direction | 参考工程双层有冗余 |
| **coin_amount** | ≈ coin_to 重复字段 | 无此冗余 | 参考工程字段冗余 |

**结论**：coin_record 的前后余额快照（coinFrom/coinTo）和显式收支方向（balanceType）值得采纳。但其缺少幂等索引是严重缺陷，大量冗余字段是反模式。

#### 对比 C：币种/汇率（currency + currency_rate vs currency_config）

| 维度 | 参考工程（2 张表分离） | 我们的（1 张表合并） | 评估 |
|------|---------------------|-------------------|------|
| **归属** | merchant-service 持有 | ser-wallet 内部持有 | 我们更内聚，币种不需要跨服务管理 |
| **拆合策略** | currency + currency_rate 分表 | currency_config 合表 | 我们的数据量极小（4-6条），合表更简单（AD-10） |
| **汇率模型** | 单一汇率值 currency_value | 4 种汇率（实时/平台/入款/出款） | 我们需求更丰富 |
| **汇率精度** | DECIMAL(20,4) | BIGINT（×10^8 存储） | 我们精度更高 |
| **汇率浮动** | 无浮动/偏差概念 | float_rate_percent + deviation_threshold | 参考工程汇率管理较简单 |
| **展示配置** | 仅 symbol 一个展示字段 | 8 个展示字段（symbol/千分位/小数位/图标/国家代码等） | 我们需求更细致 |
| **变更日志** | 无汇率变更日志 | rate_change_log 独立表 | 我们有审计追溯 |
| **缓存策略** | 全量 + 单 key 双层 Redis 缓存 | 类似策略 | ★ 缓存方案可借鉴 |
| **币种来源** | CurrencyEnum 硬编码 + DB 双重存在 | 纯 DB 驱动，不硬编码 | 参考工程是反模式 |

**结论**：参考工程的币种/汇率设计过于简单（只有一个汇率值、无浮动规则、无变更日志、硬编码枚举），不满足我们的需求。但其 Redis 双层缓存策略值得借鉴。

### 2.3 表数量对比总结

```
参考工程表结构：5 张
  wallet-service:    user_coin（钱包）+ coin_record（流水）+ add_coin_message（MQ 幂等）

  4. 参考工程的钱包没有多币种兑换能力 → 单币种模型
```

## 三、核心业务逻辑深度分析

### 3.1 addCoin —— 余额变更心脏方法

这是参考工程最核心的方法。所有余额变动最终汇聚于此。

#### 调用入口（2 种并列）

```
入口 A：HTTP API
  WalletController.addCoin(CoinRecordVO)
    → 获取 Redis 分布式锁
    → coinRecordService.addCoin()
    → 成功后 sendUserBalance()
    → 释放锁

入口 B：MQ 消费
  AddCoinListener.NoticeAddCoinMessage(jsonStr)
    → JSON 反序列化 → CoinRecordVO
    → 获取 Redis 分布式锁
    → coinRecordService.addCoin()
    → 成功后 sendUserBalance()
    → 释放锁
```

#### addCoin 完整流程（CoinRecordServiceImpl，约 170 行）

```
addCoin(CoinRecordVO) → @Transactional(rollbackFor = Exception.class)
│
├── 1. 参数校验
│     · coinType > 0
│     · coinValue != null && != 0
│
├── 2. 获取用户信息
│     · commonService.getUserInfoByUserId(userId)
│     · 先查 Redis → miss 则 Feign 调 user-service
│     · 获取：walletType, currency, merchantId, userName 等
│
├── 3. ★★ 按钱包类型分支
│     │
│     ├── walletType = 1（转账钱包 / 平台管余额）
│     │     │
│     │     ├── 查 user_coin（by userId）
│     │     │
│     │     ├── 记录存在：
│     │     │     ├── 余额校验（仅 TRANSFER_OUT/BET/SEND_REWARD）
│     │     │     │     available = amount - freezeAmount
│     │     │     │     if available < coinValue → throw AMOUNT_NOT_ENOUGH
│     │     │     │
│     │     │     ├── 收入（balanceType=1）：
│     │     │     │     userCoinRepository.addBalance(id, amount, updateTime)
│     │     │     │     SQL: amount = amount + #{amount}
│     │     │     │     coinTo = oldAmount + coinValue
│     │     │     │
│     │     │     ├── 支出（balanceType=2）：
│     │     │     │     rows = userCoinRepository.subBalance(id, amount, updateTime, coinType)
│     │     │     │     SQL: amount = amount - #{amount}
│     │     │     │          WHERE id=#{id}
│     │     │     │          [AND (amount-#{amount})>=0  当 coinType!=5 且 !=6]
│     │     │     │     if rows == 0 → throw AMOUNT_NOT_ENOUGH
│     │     │     │     coinTo = oldAmount - coinValue
│     │     │     │
│     │     │     └── 设置快照：coinFrom = oldAmount, coinTo = newAmount
│     │     │
│     │     └── 记录不存在（首次）：
│     │           ├── 校验必须是收入（不能首次就扣款）
│     │           ├── INSERT user_coin(userId, amount=coinValue, freeze=0, currency, walletType)
│     │           └── coinFrom = 0, coinTo = coinValue
│     │
│     └── walletType = 2（免转钱包 / 三方管余额）
│           ├── 查订单信息（Feign → order-service）
│           ├── 查商户详情（Redis 缓存 1 天 → Feign → merchant-service）
│           ├── 构造三方请求（transferNo, gameTypeId, totalAmount, transferType...）
│           ├── Feign 调 api-service.amountTransfer()
│           ├── 成功：保存三方订单记录 + MQ 通知余额变更
│           ├── 失败：保存失败记录 + return false → 事务回滚
│           └── coinFrom/coinTo 设为 null（三方余额不在本地）
│
├── 4. 写入流水记录
│     INSERT coin_record（userId, orderNo, coinType, coinDetail, coinValue,
│                         coinFrom, coinTo, balanceType, merchantInfo, agentInfo...）
│
└── 5. 返回 true/false
```

#### 对标我们的 balance_service.go

参考工程的 addCoin 对应我们 balance_service.go 中的统一余额变更引擎。关键映射：

| 参考工程 addCoin 内的操作 | 我们的 balance_service 对应方法 | 差异 |
|-------------------------|------------------------------|------|
| addBalance（收入） | Credit / CreditDeposit / ManualCredit | 我们按业务场景拆分，不混在一个方法 |
| subBalance（支出） | Debit / ManualDebit | 同上 |
| freezeCoin（冻结） | FreezeForWithdrawal | 我们冻结专用于提现场景 |
| subFreezeBalance（确认扣减） | ConfirmWithdrawalDeduct | 确认提现扣款 |
| rollbackFreezeBalance（回滚） | UnfreezeForWithdrawal | 提现失败退回 |
| 首次 INSERT user_coin | 用户注册时初始化 or 首次充值时创建 | 策略待定 |

**核心差异**：参考工程用一个 addCoin 方法处理所有场景（通过 coinType 区分），我们按业务场景拆分为独立方法。我们的方式更清晰、更安全，每个方法只处理一种场景。

### 3.2 余额操作原子 SQL（5 条）—— 参考工程的核心精华

> 源码：UserCoinMapper.xml

```
┌────────────────────────┬──────────────────────────────────────────┬───────────┐
│  操作                  │  SQL 语义                                │ 产生流水？ │
├────────────────────────┼──────────────────────────────────────────┼───────────┤
│  addBalance            │  amount += X                             │  是       │
│  subBalance            │  amount -= X （条件非负，特殊类型例外）   │  是       │
│  addFreezeBalance      │  freeze_amount += X                      │  否       │
│  subFreezeBalance      │  amount -= X AND freeze_amount -= X      │  是       │
│  rollbackFreezeBalance │  freeze_amount -= X                      │  否       │
└────────────────────────┴──────────────────────────────────────────┴───────────┘
```

#### 完整 SQL 逐条分析

```xml
<!-- 1. addBalance：加钱（无条件，直接加） -->
<update id="addBalance">
    UPDATE user_coin
    SET amount = amount + #{vo.amount}, updated_time = #{vo.updateTime}
    WHERE id = #{vo.id}
</update>

<!-- 2. subBalance：扣钱（★ 含数据库级非负保障） -->
<update id="subBalance">
    UPDATE user_coin
    SET amount = amount - #{vo.amount}, updated_time = #{vo.updateTime}
    WHERE id = #{vo.id}
    <!-- coinType=6（结算回滚）和 coinType=5（取消）允许负余额 -->
    <if test="vo.coinType != 6 and vo.coinType != 5">
        AND (amount - #{vo.amount}) >= 0
    </if>
</update>

<!-- 3. addFreezeBalance：冻结（总额不变，可用减少） -->
<update id="addFreezeBalance">
    UPDATE user_coin
    SET freeze_amount = freeze_amount + #{vo.amount}, updated_time = #{vo.updateTime}
    WHERE id = #{vo.id}
</update>

<!-- 4. subFreezeBalance：确认扣减冻结（总额减少，冻结减少，可用不变） -->
<update id="subFreezeBalance">
    UPDATE user_coin
    SET amount = amount - #{vo.amount},
        freeze_amount = freeze_amount - #{vo.amount},
        updated_time = #{vo.updateTime}
    WHERE id = #{vo.id}
</update>

<!-- 5. rollbackFreezeBalance：回滚冻结（总额不变，冻结减少，可用恢复） -->
<update id="rollbackFreezeBalance">
    UPDATE user_coin
    SET freeze_amount = freeze_amount - #{vo.amount}, updated_time = #{vo.updateTime}
    WHERE id = #{vo.id}
</update>
```

#### ★★★ 核心借鉴：subBalance 的数据库级非负保障

```sql
UPDATE user_coin SET amount = amount - #{amount}
WHERE id = #{id} AND (amount - #{amount}) >= 0
```

这条 SQL 的精妙之处：

1. **原子性**——检查和扣减在一条 SQL 内完成，无竞态窗口
2. **通过 affected_rows 判断**——返回 0 表示余额不足，Java 层检查 `rows == 0` 则抛异常
3. **特殊类型豁免**——结算回滚（coinType=6）和取消（coinType=5）跳过非负条件，允许余额为负
4. **双保险**——Java 层的 `insufficientBalance()` 方法 + SQL 条件互为保障

**平移到我们的 GORM 实现**：

```go
// rep/wallet.go 中的扣减实现（平移方案）
result := tx.Model(&model.Wallet{}).
    Where("id = ?", walletID).
    Where("balance - ? >= 0", amount).  // 数据库级非负保障
    Update("balance", gorm.Expr("balance - ?", amount))

if result.RowsAffected == 0 {
    return errs.ErrInsufficientBalance
}
```

对于特殊场景（类似结算回滚）需要跳过非负检查的：

```go
// 无条件扣减（仅用于系统级操作如结算回滚）
result := tx.Model(&model.Wallet{}).
    Where("id = ?", walletID).
    Update("balance", gorm.Expr("balance - ?", amount))
```

### 3.3 冻结三阶段机制

参考工程的冻结设计是最值得借鉴的模式之一：

```
阶段 1：freezeCoin（预扣/冻结）
  操作：freeze_amount += X
  效果：总额不变，可用减少
  流水：不产生账变记录
  场景：投注时预扣、提现审核时预扣

阶段 2a：subFreezeBalance（确认扣减）
  操作：amount -= X, freeze_amount -= X（同时减）
  效果：总额减少，冻结减少，可用不变
  流水：产生账变记录
  场景：提现成功确认、投注结算确认

阶段 2b：rollbackFreezeBalance（回滚释放）
  操作：freeze_amount -= X
  效果：总额不变，冻结减少，可用恢复
  流水：不产生账变记录
  场景：提现被拒退回、注单取消
```

**数值示例**（初始 amount=1000, freeze=0, 可用=1000）：

```
操作：冻结 200
  → amount=1000, freeze=200, 可用=800     ← addFreezeBalance

路径 A：确认扣减
  → amount=800,  freeze=0,   可用=800     ← subFreezeBalance

路径 B：回滚释放
  → amount=1000, freeze=0,   可用=1000    ← rollbackFreezeBalance
```

#### 对标我们的提现冻结流程

| 阶段 | 参考工程 | 我们的设计 | 一致性 |
|------|---------|-----------|--------|
| 冻结 | freezeCoin（通用） | FreezeForWithdrawal（专用于提现） | 思路一致，我们更专用 |
| 确认扣减 | subFreezeBalance（通用） | ConfirmWithdrawalDeduct（专用于提现） | 思路一致 |
| 回滚 | rollbackFreezeBalance（通用） | UnfreezeForWithdrawal（专用于提现） | 思路一致 |
| 产生流水的时机 | 仅确认扣减时产生 | 同 | 一致 |

**差异点**：参考工程的冻结是通用的（投注、提现都用），我们目前只在提现场景使用冻结。如果后续投注场景需要冻结，可以扩展为通用模式。

### 3.4 余额校验规则

```java
// 参考工程的余额校验（CoinRecordServiceImpl.insufficientBalance）
private void insufficientBalance(CoinRecordVO vo, UserCoinDTO userCoin) {
    if (coinType == TRANSFER_OUT      // 转出
        || coinType == BET            // 下注
        || coinType == SEND_REWARD)   // 打赏
    {
        BigDecimal available = userCoin.amount.subtract(userCoin.freezeAmount);
        if (available.compareTo(vo.coinValue) < 0) {
            throw new MayaDefaultException(AMOUNT_NOT_ENOUGH);
        }
    }
    // 结算(4)、取消(5)、回滚(6)、重算(7)、闪电投注(9) → 不校验
}
```

**设计意图**：只有**用户主动发起的操作**才校验余额，**系统级操作**跳过校验。原因：

- 结算回滚（coinType=6）：原注单已扣款，用户可能已提走余额，回滚时余额可能不足，强制校验会导致结算流程卡死
- 取消（coinType=5）：同理，取消可能在余额已变动后发生
- 重算（coinType=7）：二次结算可能涉及补扣

**平移评估**：这个设计思想值得采纳。在我们的 balance_service.go 中：

| 需要校验余额的操作 | 不需要校验的操作 |
|------------------|----------------|
| 用户发起提现（CreateWithdrawalOrder） | 充值入账（CreditDeposit） |
| 用户发起兑换（ExecuteExchange） | 人工加款（ManualCredit） |
| | 人工减款（ManualDebit）— 运营操作，可以强制 |
| | 结算退款（CreditCorrection） |

### 3.5 首次加钱自动创建钱包（Lazy Init）

```java
// 参考工程：用户首次加钱时，如果 user_coin 记录不存在，自动创建
if (userCoin == null) {
    checkCoinChange(coinRecordVO);  // 校验必须是收入类型
    UserCoinDTO newUserCoin = new UserCoinDTO();
    newUserCoin.setUserId(userId);
    newUserCoin.setAmount(coinValue);
    newUserCoin.setFreezeAmount(BigDecimal.ZERO);
    newUserCoin.setCurrency(userInfoVO.getCurrency());
    newUserCoin.setWalletType(WalletTypeEnum.TRANSFER.getCode());
    userCoinMapper.insert(newUserCoin);
}
```

**平移评估**：Lazy Init 模式可以考虑。两种方案：

| 方案 | 触发时机 | 优点 | 缺点 |
|------|---------|------|------|
| 注册时创建 | 用户注册成功后 | 钱包始终存在，无空指针 | 新用户可能从不充值，浪费记录 |
| 首次充值时创建 | 收到充值回调时 | 按需创建，无浪费 | 需要处理"钱包不存在"分支 |

参考工程用的是方案 B。考虑到我们是多币种模型（一个用户可能有多个币种钱包），方案 B 更合理——用户充值哪个币种就创建哪个钱包。

## 四、并发控制机制

### 4.1 参考工程的 Redisson 分布式锁

```java
// 锁 Key 模式
RedisLockKey.getAddCoinLockKey(userId)  →  "add.coin.lock.key.{userId}"

// 锁参数
RLock rLock = redisUtil.getLock(lockKey);
rLock.tryLock(
    20000,      // 等待时间：20 秒
    30000L,     // 持有时间：30 秒（自动释放）
    TimeUnit.MILLISECONDS
);

// 安全释放模式
finally {
    if (Objects.nonNull(rLock) && rLock.isLocked() && rLock.isHeldByCurrentThread()) {
        rLock.unlock();
    }
}
```

#### 锁的使用位置一览

| 入口 | 锁粒度 | 锁层级 | 备注 |
|------|--------|--------|------|
| WalletController.addCoin | userId 级 | Controller | HTTP 入口 |
| WalletController.addCoinList | userId 级（取首个） | Controller | 批量操作 |
| WalletController.freezeCoin | userId 级 | Controller | |
| WalletController.subFreezeBalance | userId 级 | Controller | |
| WalletController.rollbackFreezeBalance | userId 级 | Controller | |
| AddCoinListener.NoticeAddCoinMessage | userId 级 | Listener | MQ 入口 |
| AddCoinListener.noticeDeductCoinOrderMessage | ⚠️ **无锁** | — | 漏洞 |
| AddCoinListener.addCoinNotNoticeBalance | ⚠️ **无锁** | — | 漏洞 |

**发现的问题**：

1. **部分 MQ 消费者没有加锁**——`noticeDeductCoinOrderMessage` 和 `addCoinNotNoticeBalance` 直接调用 `deductCoinBatch` 而没有获取分布式锁。如果同一用户的多条消息并发到达，存在竞态风险
2. **锁在 Controller/Listener 层，事务在 Service 层**——锁包裹事务（方向正确），但分散在两层增加维护成本
3. **批量操作只锁首个用户**——`addCoinList` 只取第一个 CoinRecordVO 的 userId 加锁，如果批量中包含多个用户，其他用户无锁保护

### 4.2 与我们方案的对比

| 维度 | 参考工程（Redis 分布式锁） | 我们的方案（数据库行锁） |
|------|-------------------------|----------------------|
| **锁实现** | Redisson RLock（Redis SETNX + TTL） | SELECT ... FOR UPDATE（TiDB 行级锁） |
| **锁粒度** | userId（一个用户一把锁） | user_id + currency_code + wallet_type（一个钱包一把锁） |
| **锁与事务关系** | 分离（锁在 Controller，事务在 Service） | 统一（锁在事务内，事务结束自动释放） |
| **死锁风险** | 锁过期机制兜底（30 秒自动释放） | TiDB 死锁检测 + 超时自动回滚 |
| **锁泄漏风险** | 存在（GC 停顿可能导致锁过期但操作未完成） | 不存在（事务回滚自动释放） |
| **额外依赖** | 依赖 Redis 可用性 | 不引入额外依赖 |
| **适用场景** | 高并发场景（DB 连接是瓶颈时） | 中等并发（我们的场景） |
| **非负保障** | 应用层校验 + SQL WHERE 条件双保险 | SQL WHERE 条件 + 行锁串行化 |

**结论**：对于我们的场景（中等并发、单库 TiDB），数据库行锁更简单可靠。参考工程用 Redisson 主要是因为 RabbitMQ 异步消费场景下需要跨线程/跨进程的锁，而我们用同步 RPC 不存在此问题。

### 4.3 参考工程的事务管理

```java
// 不同级别的事务标注
@Transactional(rollbackFor = Exception.class)      // addCoin — 最严格，所有异常回滚
@Transactional(rollbackFor = Throwable.class)       // deductCoin — 更严格，包括 Error
@Transactional                                       // freezeCoin — 默认只回滚 RuntimeException

// 事务覆盖范围

freezeCoin:       更新 freeze_amount（一个事务）
subFreezeBalance: 更新 amount + freeze_amount + 写 coin_record（一个事务）
addCoinBatch:     循环调 addCoin，外层事务包裹全批（一个失败全部回滚）
```

**平移评估**：事务边界的设计思路一致——余额变更 + 流水写入必须在同一个事务内。我们用 GORM 的 `tx.Transaction()` 实现等价效果。

### 7.3 对标我们的服务交互

| 参考工程的交互 | 我们的对应交互 | 差异 |
|--------------|-------------|------|
| wallet → user-service（获取用户信息） | ser-wallet → ser-user（Kitex RPC） | 通信方式不同，功能等价 |
| wallet → merchant-service（获取商户信息） | 不需要（我们不是多商户架构） | 业务模型差异 |
| wallet → order-service（查订单/回滚） | 不需要（订单在 ser-wallet 内管理） | 我们的订单表自持 |
| wallet → api-service（三方余额操作） | 不需要（我们没有免转钱包） | 业务差异 |
| 游戏 → wallet（MQ 投注/结算） | 游戏 → ser-wallet（Kitex RPC） | MQ → RPC |
| wallet → websocket（MQ 余额通知） | ser-wallet → 通知服务（Kafka/NATS） | 仍需异步 |
| ser-finance → ser-wallet（充值/提现/人工操作） | 这是我们的核心交互，参考工程没有等价物 | ★ 新增 |

**关键发现**：参考工程的 wallet-service **没有**与独立的财务模块交互（参考工程没有 finance-service）。充值/提现在参考工程中是通过 api-service 直接操作 wallet 的。而我们的架构中 ser-finance 是独立模块，负责通道管理和审核流程，通过 13 个 RPC 接口与 ser-wallet 交互。**这是最大的架构差异之一**。

### 7.4 参考工程的一个架构缺陷：共享数据库

```
⚠️ order-service 直接读 wallet-service 的 user_coin 表

原因：投注时需要极快的余额校验，避免 RPC 延迟
问题：破坏微服务数据隔离原则
影响：user_coin 表结构变更需要同时通知 order-service
```

**我们必须避免**：ser-wallet 的表只有 ser-wallet 访问。其他服务通过 RPC 查询余额（GetUserWalletBalance 接口已在 IDL 设计中定义）。

### 8.3 汇率换算对比

```
参考工程（BigDecimal）：
  toUSDAmount(currency, amount) = amount / currencyRate
  例：100 CNY → USD, rate=7.2 → 100 / 7.2 = 13.89 USD

我们的方案（BIGINT ×10^8）：
  换算公式需要特殊处理精度：
  假设 100 VND（最小单位：VND，decimal_digits=0，存储值=100）
  rate_platform = 2543250000000（表示 1 USDT = 25432.50 VND）

  换算为 USDT：100 * 10^8 / 2543250000000 = 0.00000393（USDT）
  存储值（×10^8）：393

  关键：中间计算需要先乘后除，避免整数除法截断
```

## 九、VO/DTO 设计模式分析

### 9.1 参考工程的 VO 设计特点

参考工程的核心 VO 定义在 p9-core-lib（共享库）中，被所有服务引用：

| VO 类 | 职能 | 关键字段 | 问题 |
|-------|------|---------|------|
| CoinRecordVO | 账变操作入参 + 查询出参（混用） | userId, orderNo, coinType, coinDetail, coinValue, balanceType, coinFrom, coinTo | 入参出参混用，字段职责不清 |
| UserCoinVO | 钱包余额出参 | userId, amount, walletType, currency, md5Sign | md5Sign 是 API 签名验证字段，不应在 VO 中 |
| CoinFreezeVO | 冻结/回滚入参 | userId, freezeAmount | 简洁，可参考 |
| CoinFreezeRecordVO | 确认扣减入参 | userId, freezeAmount, orderNo, coinType, coinDetail, balanceType | 比 CoinFreezeVO 多了订单和账变信息 |
| CoinRecordPageParamVO | 分页查询入参 | userId, userName, orderNo, coinTypes, startTime, endTime, minAmount, maxAmount, balanceType... | 字段完整，可参考筛选维度 |
| CoinRecordPageVO | 分页查询出参 | 全量字段 + coinTypeName, coinDetailName, balanceTypeName（枚举名称） | 返回枚举名称是好实践 |

### 9.2 平移评估

**参考工程 VO 混用反模式**：CoinRecordVO 同时作为「操作入参」和「查询出参」，导致部分字段只在入参时有意义（如 coinFrom 入参时为空，出参时有值）。

**我们的优势**：Thrift IDL 天然将 Request/Response 分离。每个接口有独立的 XxxReq 和 XxxResp，字段语义明确。

**值得借鉴的设计**：
1. **CoinRecordPageParamVO 的筛选维度**——多维度筛选（用户名/订单号/金额范围/时间范围/账变类型/收支方向）可作为我们 PageTransaction 接口的参考
2. **CoinRecordPageVO 返回枚举名称**——除了返回 coinType 数值，同时返回 coinTypeName 字符串，前端不需要维护枚举映射
3. **CoinFreezeVO 的简洁性**——冻结操作入参只需 userId + freezeAmount，最小化参数

## 十三、关键设计模式提炼与平移评估

### 13.1 直接采纳的模式（A 级：已验证，直接用）

| 编号 | 模式名称 | 参考工程做法 | 平移到我们的具体做法 |
|------|---------|------------|-------------------|
| A1 | 数据库级非负保障 | `UPDATE SET amount = amount - ? WHERE id = ? AND (amount - ?) >= 0` + 检查 affected_rows | GORM: `Where("balance - ? >= 0", amount).Update(...)` + 检查 RowsAffected |
| A2 | 前后余额快照 | coin_record 存 coinFrom / coinTo | wallet_transaction 存 balance_before / balance_after |
| A3 | 冻结三阶段 | freeze → subFreeze/rollbackFreeze | FreezeForWithdrawal → ConfirmWithdrawalDeduct / UnfreezeForWithdrawal |
| A4 | 特殊操作跳过非负校验 | coinType=5/6 时 SQL 不加 `>=0` 条件 | 人工减款/结算回滚场景不加非负条件 |
| A5 | 汇率 Redis 双层缓存 | 全量列表 + 单 key 双层 | cache/currency.go 同策略 |
| A6 | 用户信息缓存 + RPC 兜底 | Redis 查 → miss 则 Feign → 回写（应该回写但参考工程漏了） | Redis 查 → miss 则 Kitex RPC → 回写 Redis |
| A7 | 余额变更主动通知 | 余额变动后 MQ → WebSocket | 余额变动后 Kafka/NATS → WebSocket 推送 |
| A8 | 首次加钱 Lazy Init | 用户首次充值时自动创建 wallet 记录 | 同策略，按币种创建 |

### 13.2 借鉴思路但需改造（B 级：方向对，细节需适配）

| 编号 | 参考工程做法 | 我们的改造方向 | 改造原因 |
|------|------------|--------------|---------|
| B1 | coin_type + coin_detail 双层枚举 | 改为 tx_type 单层 + direction 字段 | 减少冗余，语义更清晰 |
| B2 | CoinRecordVO 入参出参混用 | Thrift IDL 天然 Request/Response 分离 | 工程规范已解决 |
| B3 | addCoin 一个方法处理所有场景 | 按业务场景拆分为 CreditDeposit/ManualCredit/ManualDebit 等 | 单一职责，更安全 |
| B4 | 余额变更 MQ 广播（RabbitMQ） | Kafka/NATS 异步通知 | MQ 产品不同，功能等价 |
| B5 | add_coin_message MQ 幂等表 | 幂等保障在流水表唯一索引（uk_biz_no_type） | 我们不用 MQ，但幂等思想一致 |
| B6 | 试玩用户数据定期清理 | 后期运维阶段加入定时清理 | 当前阶段不急 |
| B7 | 分页查询返回枚举名称 | Response 中同时返回类型值和名称 | 减少前端枚举维护 |

### 13.3 明确不采纳（C 级：不适用或是反模式）

| 编号 | 参考工程做法 | 不采纳原因 |
|------|------------|-----------|
| C1 | Redisson 分布式锁 | 行锁更简单，不引入额外依赖 |
| C2 | order-service 直读 user_coin 表 | 严重违反微服务数据隔离 |

| C4 | 币种枚举硬编码（CurrencyEnum） | 纯 DB 驱动，不硬编码 |

| C11 | 部分 MQ 消费者无锁 | 所有余额操作必须有锁保护 |

```java
// 业务含义：用户充值后需要完成一定投注流水才能提现（反洗钱合规）
```

这个概念在我们的需求中体现为 `wager_requirement`（打码量要求）表。参考工程的做法验证了这个需求的合理性。我们的表结构设计中已经将 wager_requirement 列为"应该级"。

### 15.2 试玩用户与正式用户分离

参考工程在 coin_record 中有 `user_type` 字段（0=正式, 1=试玩），且：
- 所有分页查询自动过滤 `WHERE user_type = 0`
- 定时任务清理试玩数据

这提示我们：如果后续有试玩功能，需要在流水表中区分用户类型，并在查询中默认过滤。

### 15.3 API 签名验证（md5Sign）

参考工程的 `UserCoinVO` 有 `md5Sign` 字段，用于 API 层面的签名验证（非正式用户 balance 强制返回 0）。这是三方对接的安全措施。我们当前不需要，但如果未来开放 API 给三方，需要类似机制。

### 15.4 批量操作的事务策略

参考工程的 `addCoinBatch()` 用循环调用 `addCoin()`，外层 `@Transactional` 包裹。一个失败全部回滚。这是合理的批量事务策略，如果我们有批量操作需求可参考。

### 15.5 余额通知的 3 种粒度

参考工程通过不同 MQ 队列实现了 3 种通知粒度：

| 队列 | 通知策略 | 用途 |
|------|---------|------|
| ADD_COIN_QUEUE | 操作后立即通知 | 普通场景 |
| ADD_COIN_NOT_NOTICE_BALANCE_QUEUE | 操作后不通知 | 防止 C 端余额抖动（批量结算时） |
| ADD_COIN_RECORD_AND_NOTICE_BALANCE_QUEUE | 批量操作完成后统一通知一次 | 多笔操作合并通知 |

这个设计思想值得借鉴——不是每次余额变更都推送前端，批量操作时应该合并通知，避免用户看到余额快速跳动。

## 十七、结论

### 17.1 参考工程的核心价值

参考工程是一个**成熟的生产级钱包系统**，其核心价值在于：

1. **验证了我们的设计方向**——amount + freeze_amount 双字段模型、冻结三阶段、前后余额快照、SQL 级非负保障，这些关键设计在参考工程中已经被生产验证
2. **暴露了常见陷阱**——流水无幂等索引、部分操作无锁、共享 DB、枚举硬编码，这些是我们需要避免的
3. **提供了具体的 SQL 模式**——5 条原子 SQL 是可以直接平移到 GORM 的核心资产
4. **揭示了边缘场景**——结算回滚允许负余额、试玩用户数据隔离、打码量限制等业务细节

### 17.2 参考工程的局限性

1. **单币种模型**——参考工程一个用户只有一种币种，无法覆盖我们的多币种需求
2. **无汇率管理**——汇率只是一个简单的数值，没有自动同步、偏差触发、多汇率类型等
3. **无充值/提现流程编排**——这些由 api-service 和 order-service 处理，wallet-service 只管加减钱
4. **无兑换功能**——单币种模型不需要兑换
5. **无独立财务模块**——没有通道管理、审核流程、手动调账等运营功能
6. **重度 MQ 依赖**——与我们的同步 RPC 架构差异大

### 17.3 一句话总结

> 参考工程给了我们一面镜子：**余额引擎的核心模式（加减钱+冻结+快照+非负保障）是经过生产验证的正确方向，我们的设计与之高度一致；而在币种管理、业务流程编排、幂等保障、数据隔离等方面，我们的设计更完善。借鉴其精华（5 条原子 SQL + 冻结三阶段 + 缓存策略），避免其陷阱（无幂等索引 + 共享 DB + 枚举硬编码），把它的"转账钱包"单引擎模式扩展为我们的"多币种+多业务流程"完整方案。**
