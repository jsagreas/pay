## 1.2 关键差异分析

### 差异1：金额存储 — BigDecimal vs BIGINT

```
参考工程：
  UserCoinDTO.amount     → BigDecimal    → MySQL DECIMAL(19,4)
  CoinRecordDTO.coinValue → BigDecimal   → MySQL DECIMAL(19,4)
  汇率 currencyValue      → BigDecimal   → MySQL DECIMAL

我们的设计：
  user_balance.available  → int64         → MySQL BIGINT
  balance_flow.amount     → int64         → MySQL BIGINT
  exchange_rate.rate      → int64         → MySQL BIGINT

  换算规则：实际值 = 存储值 / 10^precision
  precision 来自 currency_config 表
```

**评估**：我们的 BIGINT 方案更优。

BigDecimal 在 Java 中虽然不丢精度，但：
- SQL 层面 `amount - #{vo.amount} >= 0` 的 DECIMAL 比较存在隐患
- 序列化/反序列化（JSON/RPC）时需要特殊处理
- Go 语言没有原生 BigDecimal，用第三方库（shopspring/decimal）有性能开销
- BIGINT 在 TiDB/MySQL 中索引和比较效率更高

**结论**：不采纳他们的 BigDecimal，坚持我们的 BIGINT 方案。

### 差异2：账变记录字段冗余度

```
参考工程 coin_record（35个字段）：
  冗余了全部用户元信息：userName, nickName, userType, userBelong
  冗余了全部商户信息：merchantId, merchantName, merchantNo
  冗余了全部代理信息：agentId, agentNo, agentName, path

  好处：查询时不用 JOIN，直接输出
  代价：每条账变记录写入35个字段，写放大严重
        用户改名后历史记录的名字不更新（可能是特性也可能是bug）

我们的 balance_flow 设计（~15个字段）：
  只存核心交易信息：user_id, currency, wallet_type, change_type, amount, before, after
  关联信息通过 ref_order_no 追溯

  好处：写入精简，表体积小
  代价：管理后台查询需要 JOIN 或应用层补全
```

**评估**：两种方案各有场景。但参考工程的 coin_record 35字段过于冗余。

**结论**：不采纳全冗余。但可借鉴"部分冗余"思路——在 balance_flow 中增加 `operator_name` 字段（操作人名称冗余），方便管理后台查账变时减少一次 JOIN。其他字段不冗余。

### 差异3：子钱包体系

```
参考工程：
  wallet_type = 2种：1=转账钱包 2=免转钱包
  "转账钱包"：余额在自己DB，走本地SQL
  "免转钱包"：余额在外部平台，走API代理

  这两种是互斥的（一个商户只有一种），不是"用户同时拥有多种钱包"

我们的设计：
  wallet_type = 4种：1=中心 2=奖励 3=代理 4=主播
  每个用户可以同时拥有多个子钱包
  user_balance 的唯一键 = (user_id, currency, wallet_type)

  这是"余额矩阵"：一个用户可以有 4子钱包 × N币种 = 4N 条记录
```

**评估**：完全不同的业务模型，不可参考。

**结论**：不采纳。他们的 wallet_type 是商户级别的分叉（转账vs免转），我们的是用户级别的余额隔离（中心/奖励/代理/主播）。业务语义完全不同。

# 2. 余额操作核心设计

## 2.1 参考工程的 5 条 SQL（全部余额操作）

从 `UserCoinMapper.xml` 精读，这 5 条 SQL 是整个钱包系统余额操作的**全部**：

```sql
-- ① addBalance: 增加可用余额
UPDATE user_coin b
SET b.amount = b.amount + #{vo.amount},
    b.updated_time = #{vo.updateTime}
WHERE b.id = #{vo.id}

-- ② subBalance: 扣减可用余额（带余额保护）
UPDATE user_coin b
SET b.amount = b.amount - #{vo.amount},
    b.updated_time = #{vo.updateTime}
WHERE b.id = #{vo.id}
  -- 结算回滚(6)和取消(5)允许减为负数，其他必须余额充足
  <if test="vo.coinType != 6 and vo.coinType != 5">
    AND (b.amount - #{vo.amount}) >= 0
  </if>

-- ③ addFreezeBalance: 冻结（freeze_amount增加）
UPDATE user_coin b
SET b.freeze_amount = b.freeze_amount + #{vo.amount},
    b.updated_time = #{vo.updateTime}
WHERE b.id = #{vo.id}

-- ④ subFreezeBalance: 扣减冻结（amount和freeze_amount同时减少）
UPDATE user_coin b
SET b.amount = b.amount - #{vo.amount},
    b.freeze_amount = b.freeze_amount - #{vo.amount},
    b.updated_time = #{vo.updateTime}
WHERE b.id = #{vo.id}

-- ⑤ rollbackFreezeBalance: 解冻回滚（只减freeze_amount）
UPDATE user_coin b
SET b.freeze_amount = b.freeze_amount - #{vo.amount},
    b.updated_time = #{vo.updateTime}
WHERE b.id = #{vo.id}
```

## 2.2 余额公式与操作矩阵

```
user_coin 表的余额语义：
  amount       = 总余额（可用 + 已冻结的原始值）
  freeze_amount = 冻结金额
  可用余额      = amount - freeze_amount

操作矩阵：
┌───────────────────┬──────────┬───────────────┬──────────────┬──────────┐
│       操作         │ amount   │ freeze_amount │ 可用余额变化  │ 产生账变 │
├───────────────────┼──────────┼───────────────┼──────────────┼──────────┤
│ addBalance(收入)   │ +delta   │ 不变          │ +delta       │ 是       │
│ subBalance(支出)   │ -delta   │ 不变          │ -delta       │ 是       │
│ addFreeze(冻结)    │ 不变     │ +delta        │ -delta       │ 否       │
│ subFreeze(扣冻结)  │ -delta   │ -delta        │ 不变         │ 是       │
│ rollbackFreeze     │ 不变     │ -delta        │ +delta       │ 否       │
└───────────────────┴──────────┴───────────────┴──────────────┴──────────┘
```

## 2.3 与我们设计的对照

```
我们的 user_balance 余额语义：
  available = 可用余额（直接存可用值）
  frozen    = 冻结余额
  总余额    = available + frozen

  ★ 语义差异：他们 amount = total，我们 available = 净可用

我们的操作矩阵：
┌──────────────────┬───────────┬────────┬──────────────┐
│       操作        │ available │ frozen │    说明       │
├──────────────────┼───────────┼────────┼──────────────┤
│ 充值入账          │ +delta    │ 不变   │ 同他们       │
│ 提现冻结          │ -delta    │ +delta │ ★不同于他们  │
│ 提现扣款(冻结扣)  │ 不变      │ -delta │ ★不同于他们  │
│ 提现退回(解冻)    │ +delta    │ -delta │ ★不同于他们  │
└──────────────────┴───────────┴────────┴──────────────┘
```

**对照分析**：

```
关键差异在于"冻结操作"：

他们的做法：
  冻结时：amount 不变，freeze_amount += delta
  → 可用 = amount - freeze_amount，可用减少
  → 但 amount 字段的值并没有真正变化
  → 查可用余额需要计算

我们的做法：
  冻结时：available -= delta, frozen += delta
  → available 字段直接就是可用值，查询不需要计算
  → 但需要一条 UPDATE 同时改两个字段（和他们的 subFreeze 类似）

实际SQL层面的差异：

  他们冻结：UPDATE SET freeze_amount = freeze_amount + delta (1字段)
  他们扣冻结：UPDATE SET amount = amount - delta, freeze_amount = freeze_amount - delta (2字段)

  我们冻结：UPDATE SET available = available - delta, frozen = frozen + delta WHERE available >= delta (2字段)
  我们扣冻结：UPDATE SET frozen = frozen - delta (1字段)
```

**评估**：两种方案数学等价，但我们的方案更好——`available` 直接可读，无需每次查询都做减法计算。并且在冻结操作时直接把 `WHERE available >= delta` 的余额保护做进去了，比他们在应用层判断 `amount >= freezeAmount` 更安全。

**结论**：不改我们的设计。两种方案本质等价，但我们的直觉性更好。

# 3. 冻结/解冻机制

## 3.1 参考工程的三操作模型

```
操作1: freezeCoin（冻结）
  用途：下注前冻结、提现前冻结
  入参：CoinFreezeVO { userId, freezeAmount }
  流程：
    1. 查 user_coin 记录
    2. 验证 amount >= freezeAmount（注意是 amount 不是 available）
    3. SQL: freeze_amount += delta
    4. 不产生 coin_record

  ★ 问题：步骤2验证的是 amount（总额），没有考虑已冻结部分
           如果 amount=1000, freeze_amount=800, 又冻结 300
           验证 1000 >= 300 通过，但实际可用只有 200
           → 存在超额冻结风险！

操作2: subFreezeBalance（扣减冻结）
  用途：提现成功后扣减冻结金额
  入参：CoinFreezeRecordVO { userId, freezeAmount, orderNo, coinType, balanceType }
  流程：
    1. 查 user_coin
    2. 验证 freeze_amount >= delta
    3. SQL: amount -= delta, freeze_amount -= delta
    4. 产生 coin_record

操作3: rollbackFreezeBalance（回滚冻结）
  用途：提现拒绝后解冻
  入参：CoinFreezeVO { userId, freezeAmount }
  流程：
    1. 查 user_coin
    2. SQL: freeze_amount -= delta
    3. 不产生 coin_record

  ★ 问题：没有验证 freeze_amount >= delta，理论上可以减成负数
```

## 3.2 参考工程冻结机制的缺陷

| 缺陷 | 说明 | 我们如何避免 |
|------|------|-------------|
| **超额冻结** | freezeCoin 验证 amount >= delta，没减去已冻结部分 | 我们在 SQL 层 `WHERE available >= delta` |
| **rollback无校验** | rollbackFreezeBalance 没验证 freeze >= delta | 我们在 SQL 层 `WHERE frozen >= delta` |
| **冻结不产生账变** | freezeCoin 只改余额不记流水，审计链断裂 | 我们的 ChangeType=3(提现冻结) 会记 balance_flow |
| **subFreeze缺事务注解** | 代码中 subFreezeBalance 方法没有 @Transactional | 我们用 GORM 事务显式包裹 |

**结论**：借鉴他们的三操作模型（冻结/扣冻结/回滚），但修正以上所有缺陷。我们的设计已经规避了这些问题。

# 4. 并发控制与一致性

## 4.1 参考工程的并发控制策略

```
两层防护：

第一层：Redisson 分布式锁（应用层）
  锁粒度：per userId
  等待超时：20秒（tryLock wait）
  持有超时：30秒（lease time）
  锁Key格式：RedisLockKey.getAddCoinLockKey(userId)

  代码模板：
    RLock rLock = redisUtil.getLock(key);
    try {
      if (rLock.tryLock(20000, 30000L, TimeUnit.MILLISECONDS)) {
        // 业务逻辑
      } else {
        log.info("锁超时");
      }
    } finally {
      if (rLock != null && rLock.isLocked() && rLock.isHeldByCurrentThread()) {
        rLock.unlock();
      }
    }

第二层：SQL 条件防护（数据库层）
  仅在 subBalance 中：AND (b.amount - #{vo.amount}) >= 0
  通过返回 affected rows = 0 检测余额不足

  其他 4 条 SQL（addBalance, addFreeze, subFreeze, rollback）无任何 WHERE 保护
```

## 4.2 他们 vs 我们的并发控制对照

```
┌──────────────┬─────────────────────────────┬─────────────────────────────┐
│     维度      │        参考工程              │        我们的设计            │
├──────────────┼─────────────────────────────┼─────────────────────────────┤
│ 分布式锁     │ Redisson RLock per userId    │ Redis SETNX per userId      │
│ 锁粒度       │ userId 级                   │ userId 级（可能更细到 userId │
│              │                             │   + wallet_type + currency） │
│ 余额保护     │ 仅扣款SQL有 WHERE 保护       │ 所有减少操作都有 WHERE 保护  │
│ 乐观锁       │ ×（用 updated_time 但非乐观锁）│ version 字段（备用）        │
│ 事务         │ @Transactional 但不一致      │ GORM 事务显式管理            │
│ 幂等性       │ ×（无幂等保护）              │ balance_flow UK(ref_order_no,│
│              │                             │   change_type) 双层幂等      │
│ 防负数       │ 仅 subBalance，且结算回滚除外 │ 所有减少操作 WHERE >= delta  │
└──────────────┴─────────────────────────────┴─────────────────────────────┘
```

## 4.3 参考工程并发控制的缺陷

| 问题 | 具体表现 | 风险等级 |
|------|---------|---------|
| **无幂等保护** | 同一条 MQ 消息重复消费，会重复加钱 | ★★★致命 |
| **批量锁粒度错误** | addCoinList 用第一个 userId 加锁，多用户情况下并发不安全 | ★★严重 |
| **事务注解不一致** | subFreezeBalance/rollbackFreezeBalance 缺少 @Transactional | ★★严重 |
| **冻结无SQL保护** | addFreezeBalance 无 WHERE 条件，理论上可超额冻结 | ★★严重 |
| **30秒锁持有过长** | 正常操作 <100ms，30秒锁持有浪费资源 | ★一般 |

**结论**：

- **借鉴**：分布式锁 + SQL条件保护的双层防护思想
- **不借鉴**：他们的具体实现有多处漏洞
- **我们已规避**：幂等性（UK约束）、全操作WHERE保护、显式事务

## 6.2 前后余额的记录方式（重要借鉴点）

```java
// CoinRecordServiceImpl.addCoin() 中的关键逻辑：

// 收入路径：
if (balanceType == income) {
    userCoinRepository.addBalance(balanceVO);  // 先改余额
    coinTo = userCoinDTO1.getAmount().add(coinRecordVO.getCoinValue());  // 后算结果
}

// 支出路径：
if (balanceType == expenses) {
    int i = userCoinRepository.subBalance(balanceVO);  // 先改余额
    if (i == 0) throw AMOUNT_NOT_ENOUGH;               // 余额不足
    coinTo = userCoinDTO1.getAmount().subtract(coinRecordVO.getCoinValue());
}

coinRecordDTO.setCoinTo(coinTo);              // 变动后余额
coinRecordDTO.setCoinFrom(userCoinDTO1.getAmount());  // 变动前余额（先查后改的值）
```

**问题**：这里有并发风险。`coinFrom` 取的是 SELECT 时的 amount，但 UPDATE 是在 SELECT 之后，两者之间如果有其他事务修改了 amount，`coinFrom/coinTo` 就不准确了。虽然有分布式锁保护，但如果锁失效或跨节点，仍有风险。

**我们的做法**：用 `UPDATE ... RETURNING` 或者在事务中 `SELECT FOR UPDATE` 确保 before/after 值准确。GORM + TiDB 支持在事务中安全读写。

## 6.3 两套枚举的设计（coin_type vs coin_detail）

```
coin_type（账变类型，9种）：
  1=转入  2=转出  3=下注  4=派彩  5=取消  6=结算回滚  7=重算  8=打赏  9=闪电投注

coin_detail（账变来源，8种）：
  1=会员存款  2=会员提款  3=下注  4=结算  5=取消  6=结算取消  7=打赏  8=闪电投注

★ 两套枚举的区别：
  coin_type 描述"发生了什么"
  coin_detail 描述"为什么发生"

  实际使用中这两套枚举高度重合，多数场景 coin_type == coin_detail
  只有少数情况不同（如 type=4结算 + detail=6结算取消）
```

**评估**：双枚举设计的初衷是"类型+来源"二维描述，但实际实现中几乎 1:1 映射，增加了复杂度却没有明显收益。

**结论**：不采纳双枚举。我们的 `change_type` 单一枚举（11+种）已经足够描述所有场景。如果后续需要更细粒度，通过扩展 change_type 的枚举值实现，而非增加第二维度。

## 7.1 参考工程的 MQ 驱动设计

```
参考工程的核心理念：余额变动通过 RabbitMQ 异步驱动

4 个消费队列：

① ADD_COIN_QUEUE（核心——余额加减）
   生产者：order-service（结算/打赏后）

   生产者：order-service（下注时）

   行为：deductCoinBatch() → 失败则回退注单 → 通知余额
   场景：玩家下注后扣减余额

   场景：批量结算中间过程

④ ADD_COIN_RECORD_AND_NOTICE_BALANCE_QUEUE（记录+通知）
   生产者：order-service
   消费者：AddCoinListener.addCoinRecordAndNoticeBalanceQueue()
   行为：changeCoinAndNoticeBalance()
   场景：需要同时写流水和通知的场景

1 个广播交换机：
   EXCHANGE_NOTICE_USER_BALANCE_CHANGE（Fanout）
   → 广播到所有 WebSocket 服务器
   → WebSocket 推送给在线用户（余额实时更新）

消息投递确认：
   WallRabbitTemplateConfig 实现 ConfirmCallback
   ACK → 更新 add_coin_message.status = SUCCESS
   NACK → 更新 add_coin_message.status = FAILURE
```

## 7.2 与我们架构的对照

```
参考工程（MQ驱动）：
  order-service →[MQ]→ wallet-service → 改余额 →[MQ]→ WebSocket → C端

  优点：解耦、削峰、异步
  缺点：链路长、延迟高、幂等难保证、MQ故障=资金异常

我们的设计（RPC同步）：
  调用方 →[Kitex RPC]→ ser-wallet → 改余额 → 返回结果

  优点：简单、同步、强一致、调用方立即知道结果
  缺点：需要处理RPC超时、调用方阻塞等待
```

**评估**：

参考工程用 MQ 是因为他们的钱包和游戏服务是松耦合的（不同团队、不同技术栈）。MQ 作为缓冲层可以理解。但这导致了：
- 下注扣款是异步的，扣款失败需要"回退注单"（compensate），增加了补偿复杂度
- MQ 消息丢失/重复消费 = 资金异常
- 需要额外的 `add_coin_message` 表追踪消息状态

我们的 RPC 同步调用更适合当前场景——调用方立即得到结果，不需要补偿机制。

**结论**：

- **不采纳** MQ 驱动余额变动的架构
- **可借鉴** 余额变化后推送通知到前端的设计（我们也需要通知，但用 RPC 回调或 WebSocket 推送，不用 MQ Fanout）
- **可借鉴** add_coin_message 追踪表的思路——如果后续需要异步操作，可参考其消息追踪机制

- 主动调：ser-wallet 调 ser-finance（查支付配置、提交财务审核）
- 主动调：ser-wallet 调 ser-buser（查用户KYC信息）

**结论**：交互模式本质相同（双向依赖），但通信方式不同（Feign+MQ vs Kitex RPC）。

# 12. 设计亮点 — 值得借鉴

## 12.1 分布式锁的完整模板

```java
// 参考工程的锁模板（值得借鉴的代码结构）
RLock rLock = redisUtil.getLock(lockKey);
try {
    if (rLock.tryLock(waitTime, leaseTime, TimeUnit.MILLISECONDS)) {
        // 业务逻辑
    } else {
        log.info("锁超时：{}", orderId);
        // 返回失败，不是抛异常
    }
} catch (Exception e) {
    log.error(e.getMessage(), e);
} finally {
    // 三重验证后解锁
    if (rLock != null && rLock.isLocked() && rLock.isHeldByCurrentThread()) {
        rLock.unlock();
    }
}
```

**借鉴点**：finally 中的三重检查（非空 + 已锁 + 当前线程持有）是好的防御性编程。我们的 Go 代码中也应该用 defer + 类似检查。

## 12.2 SQL 层面的余额保护

```sql
-- subBalance 中的条件保护
AND (b.amount - #{vo.amount}) >= 0
```

**借鉴点**：在 SQL WHERE 中做余额充足性检查，通过 affected rows == 0 检测失败。这比先 SELECT 再判断再 UPDATE 更安全（避免了 TOCTOU 问题）。我们已经在设计中采用了这个思路。

## 12.3 MQ 消息投递确认追踪

```
add_coin_message 表记录每条消息的投递状态：
  message_id    消息ID
  status        0=待发送 1=成功 2=失败
  try_count     重试次数

ConfirmCallback 回调更新状态
```

**借鉴点**：虽然我们不用 MQ，但对于异步操作（如回调通知），可以参考这种"消息追踪表"设计，记录每次外部调用的状态。

## 12.4 WebSocket 余额推送

```
每次余额变动后：
  1. coinRecordService.sendUserBalance(userId)
  2. RabbitMQ Fanout → 所有 WebSocket 服务器
  3. WebSocket 推送给在线用户

用户在多个设备登录 → 所有设备同时更新余额
```

**借鉴点**：余额变动后主动推送给 C 端的设计。我们也需要在充值/提现/兑换/人工调账成功后，通知前端刷新余额。

## 12.5 前后余额快照

```
每条 coin_record 记录：
  coin_from = 变动前余额
  coin_to   = 变动后余额
  coin_value = 变动金额

满足等式：coin_to = coin_from ± coin_value
```

**借鉴点**：必须借鉴。前后余额快照是对账和排查问题的关键数据。我们的 balance_flow 已经设计了 before/after 字段。

# 13. 设计缺陷 — 引以为戒

## 13.1 致命级缺陷

| # | 缺陷 | 具体表现 | 我们的规避措施 |
|---|------|---------|---------------|
| F1 | **无幂等保护** | MQ消息重复消费会重复加钱，无 UK 约束防重 | balance_flow UK(ref_order_no, change_type) |
| F2 | **超额冻结** | freezeCoin 验证 amount>=delta，遗漏已冻结部分 | SQL: WHERE available >= delta |
| F3 | **事务注解缺失** | subFreezeBalance 无 @Transactional，改余额和写流水非原子 | GORM 显式事务包裹所有余额操作 |

## 13.2 严重级缺陷

| # | 缺陷 | 具体表现 | 我们的规避措施 |
|---|------|---------|---------------|
| S1 | **解冻无校验** | rollbackFreeze 不验证 freeze>=delta，可减成负数 | SQL: WHERE frozen >= delta |
| S2 | **批量锁粒度错** | addCoinList 用第一个userId加锁，多用户不安全 | 不支持跨用户批量，或逐用户加锁 |
| S3 | **前后余额不精确** | SELECT后UPDATE，中间可能有并发修改 | 事务内 SELECT FOR UPDATE 或 RETURNING |
| S4 | **coin_amount冗余** | coin_amount = coin_to，纯粹冗余字段 | 不设计冗余字段 |
| S5 | **硬编码币种** | CurrencyEnum 5种固定，新增需改代码发版 | 纯DB配置 currency_config |

## 13.3 一般级缺陷

| # | 缺陷 | 具体表现 | 我们的规避措施 |
|---|------|---------|---------------|
| G1 | 30秒锁持有过长 | 正常操作<100ms，浪费资源 | 5秒 lease time |
| G2 | 双枚举冗余 | coin_type + coin_detail 高度重合 | 单一 change_type |
| G3 | 35字段流水表 | 冗余太多用户/商户信息 | 精简到15字段 |
| G4 | SQL注入风险 | ORDER BY 直接拼接 ${vo.createdTimeBy} | GORM 参数化查询 |
| G5 | 无速率限制 | 无余额变动频率限制 | 可在分布式锁中实现 |

# 15. 综合结论

## 15.1 参考工程的定位

参考工程是一个**以游戏为核心的钱包系统**——它的主要职责是为"下注→结算→派彩"这条链路提供余额加减能力。充值和提现的核心逻辑并不在这个代码库中，而是在外部服务。

我们的系统是一个**以资金管理为核心的钱包系统**——充值、提现、兑换、人工调账都是核心流程，都在我们的 ser-wallet 中实现。

两者虽然都叫"钱包"，但**业务重心完全不同**。

## 15.2 已确认的设计正确性

通过对照分析，我们之前的设计决策得到了验证：

| 我们的设计 | 对照结论 |
|-----------|---------|
| BIGINT存金额 | 参考工程用BigDecimal有精度隐患，我们的方案更优 |
| 4子钱包矩阵 | 参考工程是2种互斥钱包，我们是4种并存，业务不同设计合理 |
| 幂等UK约束 | 参考工程完全没有幂等保护，我们的UK设计填补了这个致命缺陷 |
| 全操作WHERE保护 | 参考工程只有subBalance有保护，我们的设计更完善 |
| 冻结记流水 | 参考工程冻结不记流水导致审计断裂，我们记录是正确的 |
| 币种纯DB配置 | 参考工程硬编码5种币种是明显缺陷 |
| RPC同步调用 | 参考工程MQ异步导致补偿复杂度高，我们的同步方案更简洁 |

## 15.3 需要补充的设计

从参考工程中发现我们可能遗漏或需要强化的点：

| # | 补充项 | 来源 | 建议 |
|---|--------|------|------|
| 1 | balance_flow 增加 direction 字段 | coin_record.balance_type | 方便后台按收支筛选 |
| 2 | 余额变动后推送通知机制 | sendUserBalance + MQ Fanout | 设计通知链路 |
| 3 | 充值单用户并发限制 | 同时只能一笔进行中 | deposit_order 状态检查 |
| 4 | 异步操作追踪表思路 | add_coin_message | 如果有异步回调，需要追踪 |
