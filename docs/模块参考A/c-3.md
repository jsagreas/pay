## 二、技术栈对照

| 维度 | 参考Java工程 | 我们Go工程 | 对照说明 |
|------|-------------|-----------|---------|
| **语言** | Java 17 | Go 1.23+ | 完全不同，需重写 |
| **框架** | Spring Boot 2.7.4 + Spring Cloud 2021.0.4 | CloudWeGo Kitex + Hertz | 微服务框架不同，但角色一致 |
| **ORM** | MyBatis Plus 3.5.3.1 | GORM Gen | 都是代码生成型ORM |
| **服务发现** | Spring Cloud（Eureka/Nacos） | ETCD | 功能等价 |
| **RPC** | OpenFeign（HTTP调用） | Kitex RPC（Thrift协议） | 我们RPC性能更好 |
| **HTTP网关** | 直接暴露REST | Hertz网关→Kitex RPC | 我们多一层网关转发 |
| **消息队列** | RabbitMQ | 待定（可选Kafka/NATS/Redis Stream） | 需在评审确认 |
| **缓存** | Redis + Redisson | Redis（go-redis） | 功能等价 |
| **分布式锁** | Redisson RLock | go-redis + setnx / redsync | 需要自行实现或选库 |
| **金额精度** | BigDecimal | shopspring/decimal（string传输） | 同样高精度方案 |
| **时间格式** | Date + @JsonFormat (UTC) | int64毫秒时间戳 | 我们用时间戳更简洁 |
| **ID策略** | AUTO_INCREMENT | Snowflake / AUTO_INCREMENT | 我们已有雪花ID基础设施 |
| **软删除** | 无软删除 | delete_at 字段（0=未删除） | 我们设计更完善 |

## 三、数据模型深度分析

### 3.1 参考工程表结构（3张表）

#### 表1：user_coin（用户钱包余额）

```
user_coin
├── id              BIGINT AUTO_INCREMENT PK     ← 主键
├── user_id         VARCHAR                      ← 用户ID
├── user_name       VARCHAR                      ← 用户名
├── amount          DECIMAL (BigDecimal)         ← 可用余额
├── freeze_amount   DECIMAL (BigDecimal)         ← 冻结余额
├── user_state      INT                          ← 用户游戏状态
├── game_type       INT                          ← 当前游戏类型
├── wallet_type     INT                          ← 1=转账钱包, 2=免转钱包
├── currency        VARCHAR                      ← 币种代码(USD/CNY/...)
├── remark          VARCHAR                      ← 备注
├── creator         BIGINT                       ← 创建者
├── updater         BIGINT                       ← 更新者
├── created_time    DATETIME (UTC)               ← 创建时间
└── updated_time    DATETIME (UTC)               ← 更新时间
```

**设计解读：**
- **一用户一行**：user_id 定位唯一钱包（配合 wallet_type 可多行）
- **双余额模型**：amount（总余额）+ freeze_amount（冻结）→ 可用 = amount - freeze_amount
- **钱包类型分流**：wallet_type 区分"转账钱包"和"免转钱包"两种处理路径
- **currency字段**：每个钱包绑定单一币种
- **无软删除**：没有 delete_at，删除即物理删除

#### 表2：coin_record（账变流水记录）

```
coin_record
├── id              BIGINT AUTO_INCREMENT PK     ← 主键
├── user_id         VARCHAR                      ← 用户ID
├── user_name       VARCHAR                      ← 用户名
├── user_type       INT                          ← 0=正式, 1=试玩
├── nick_name       VARCHAR                      ← 昵称
├── user_belong     INT                          ← 用户所属类型
├── merchant_id     BIGINT                       ← 商户ID
├── merchant_name   VARCHAR                      ← 商户名称
├── merchant_no     VARCHAR                      ← 商户编号
├── agent_no        VARCHAR                      ← 代理编号
├── agent_id        BIGINT                       ← 代理ID
├── agent_name      VARCHAR                      ← 代理名称
├── path            VARCHAR                      ← 上级代理路径
├── order_no        VARCHAR                      ← 关联订单号
├── coin_type       INT                          ← 账变类型(CoinChangeEnum 1~9)
├── coin_detail     INT                          ← 账变明细(CoinChangeDetailEnum 1~8)
├── coin_value      DECIMAL                      ← 变动金额（绝对值）
├── coin_from       DECIMAL                      ← 变动前余额
├── coin_to         DECIMAL                      ← 变动后余额
├── coin_amount     DECIMAL                      ← 变动时当前总余额
├── balance_type    INT                          ← 1=收入, 2=支出
├── remark          VARCHAR                      ← 备注
├── creator         BIGINT                       ← 创建者
├── updater         BIGINT                       ← 更新者
├── created_time    DATETIME (UTC)               ← 创建时间
└── updated_time    DATETIME (UTC)               ← 更新时间
```

**设计解读：**
- **宽表设计**：把用户信息、商户信息、代理信息全部冗余到流水表中
- **前后快照**：coin_from（变动前）+ coin_to（变动后），完整审计链
- **双维度分类**：coin_type（大类 1~9）+ coin_detail（明细 1~8），层级分类
- **不可变流水**：只 INSERT 不 UPDATE（仅查询用途），天然审计日志
- **balance_type**：显式标记收支方向（1收入/2支出），冗余但便于查询

#### 表3：add_coin_message（MQ消息追踪）

```
add_coin_message
├── id              BIGINT PK                    ← 主键
├── message_id      VARCHAR                      ← 消息唯一ID
├── order_id_list   VARCHAR                      ← 关联订单列表
├── status          INT                          ← 消息状态
├── try_count       INT DEFAULT 0                ← 重试次数
├── create_time     DATETIME                     ← 创建时间
└── update_time     DATETIME                     ← 更新时间
```

**设计解读：**
- **消息可靠性追踪**：用于保证MQ消息至少处理一次
- **重试计数**：try_count 记录重试次数，防无限重试
- **关联性**：message_id 关联MQ消息，order_id_list 关联业务订单

### 3.2 与我们的表结构对比

| 参考工程表 | 我们对应表 | 覆盖关系 | 差异点 |
|-----------|-----------|---------|--------|
| `user_coin` | `wallet_balance` | 功能等价 | 我们：多子钱包类型(5种)、有 wallet_type_id 关联，更灵活 |
| `coin_record` | `wallet_transaction` | 功能等价 | 我们：不冗余用户信息（JOIN查）、有 snapshot_rate 汇率快照 |
| `add_coin_message` | 无直接对应 | 可参考 | 我们可考虑在MQ方案中增加类似追踪表 |
| 无 | `currency_config` | 我们独有 | 参考工程币种靠枚举硬编码 |
| 无 | `base_currency` | 我们独有 | 参考工程基准币种固定为USD |
| 无 | `currency_rate` | 部分对应 | 参考工程汇率仅在Redis缓存，无持久化表 |
| 无 | `recharge_order` | 我们独有 | 参考工程充值由别的服务处理 |
| 无 | `withdraw_order` | 我们独有 | 同上 |
| 无 | `exchange_record` | 我们独有 | 参考工程无兑换概念 |

**关键差异总结：**
- 参考工程 **3张表** vs 我们 **12张表** — 我们业务域更完整（含币种配置、充值、提现、兑换）
- 参考工程 **5固定币种硬编码** vs 我们 **动态币种配置表** — 我们更灵活
- 参考工程 **宽表冗余用户信息** vs 我们 **规范化设计+JOIN** — 各有优劣
- 参考工程 **无软删除** vs 我们 **统一 delete_at** — 我们更安全

## 四、枚举体系完整拆解

### 4.1 CoinChangeEnum — 账变大类（9种）

```
位置：p9-core-lib → WalletUtilEnum.CoinChangeEnum

  Code  |  枚举名                |  中文          |  收支方向
 -------+------------------------+----------------+----------
    1   | TRANSFER_IN            | 转入           |  收入
    2   | TRANSFER_OUT           | 转出           |  支出
    3   | BET                    | 下注           |  支出
    4   | SETTLEMENT             | 派彩（结算）    |  收入
    5   | ORDER_CANCEL           | 注单取消       |  收入（退回）
    6   | SETTLEMENT_ROLLBACK    | 结算回滚       |  支出（追回）
    7   | SETTLEMENT_SECOND      | 二次结算（重算） |  收入或支出
    8   | SEND_REWARD            | 打赏           |  支出
    9   | LIGHTNING_BET          | 闪电投注       |  支出
```

**特殊业务规则：**
- `coinType=5`（ORDER_CANCEL）和 `coinType=6`（SETTLEMENT_ROLLBACK）在 subBalance SQL 中 **跳过余额充足校验**
  - 原因：取消和回滚是系统强制操作，即使余额不足也必须执行
  - 这意味着这两个操作可能导致负余额

## 五、核心业务逻辑分析

### 5.1 余额操作 — 5个原子SQL

这是参考工程最核心的设计，5个 MyBatis XML 中定义的原子 SQL 操作：

#### ① addBalance — 加余额

```sql
UPDATE user_coin b
SET b.amount = b.amount + #{vo.amount},
    b.updated_time = #{vo.updateTime}
WHERE b.id = #{vo.id}
```

- 无条件加款，用于收入类操作
- 参数：id（钱包主键）、amount（加多少）、updateTime

#### ② subBalance — 减余额

```sql
UPDATE user_coin b
SET b.amount = b.amount - #{vo.amount},
    b.updated_time = #{vo.updateTime}
WHERE b.id = #{vo.id}
  /* 结算回滚(6)和取消(5)跳过余额校验 */
  <if test="vo.coinType != 6 and vo.coinType != 5">
    AND (b.amount - #{vo.amount}) >= 0
  </if>
```

- **条件扣款**：WHERE 中嵌入余额充足校验 `(amount - 扣款额) >= 0`
- **特例**：coinType=5（取消）和 coinType=6（回滚）不校验，允许负余额
- 通过 **UPDATE 影响行数** 判断是否成功（i=0 表示余额不足）

#### ③ addFreezeBalance — 增加冻结

```sql
UPDATE user_coin b
SET b.freeze_amount = b.freeze_amount + #{vo.amount},
    b.updated_time = #{vo.updateTime}
WHERE b.id = #{vo.id}
```

- 只增加冻结金额，不动可用余额
- 业务校验在 Service 层做（检查 amount >= freezeAmount）

#### ④ subFreezeBalance — 扣除冻结（确认消费）

```sql
UPDATE user_coin b
SET b.amount = b.amount - #{vo.amount},
    b.freeze_amount = b.freeze_amount - #{vo.amount},
    b.updated_time = #{vo.updateTime}
WHERE b.id = #{vo.id}
```

- **同时减少** amount 和 freeze_amount → 钱真正扣掉了
- 场景：冻结的钱确认被消费（如投注确认、结算扣款）

#### ⑤ rollbackFreezeBalance — 回滚冻结（取消冻结）

```sql
UPDATE user_coin b
SET b.freeze_amount = b.freeze_amount - #{vo.amount},
    b.updated_time = #{vo.updateTime}
WHERE b.id = #{vo.id}
```

- **只减少** freeze_amount，不动 amount → 钱退回可用
- 场景：冻结后取消（如订单取消）

#### 5个操作的余额效果图

```
操作前余额状态：amount=100, freeze_amount=20, 可用=80

① addBalance(30):
   amount=130, freeze=20, 可用=110

② subBalance(30):
   amount=70,  freeze=20, 可用=50
   (若可用<30且非5/6类型则失败)

③ addFreezeBalance(10):
   amount=100, freeze=30, 可用=70
   (冻结前Service层校验: amount(100) >= 新freeze(30))

④ subFreezeBalance(10):
   amount=90,  freeze=10, 可用=80
   (冻结的10块确认消费)

⑤ rollbackFreezeBalance(10):
   amount=100, freeze=10, 可用=90
   (冻结的10块退回可用)
```

### 5.3 冻结/解冻/回滚三部曲

```
场景：用户下注 100 元

Step 1: freezeCoin (下注时冻结)
  ├── 校验钱包存在
  ├── 校验 amount(500) >= freezeAmount(100)  ✓
  ├── addFreezeBalance(100)
  │   → amount=500, freeze=100, 可用=400
  └── ★ 不生成账变记录

Step 2a: subFreezeBalance (结算时确认扣款)
  ├── 校验钱包存在
  ├── 校验 freeze >= 100  ✓
  ├── subFreezeBalance(100)
  │   → amount=400, freeze=0, 可用=400
  └── ★ 生成账变记录（coinType=BET, balanceType=支出）

Step 2b: rollbackFreezeBalance (取消时退回)
  ├── 校验钱包存在
  ├── rollbackFreezeBalance(100)
  │   → amount=500, freeze=0, 可用=500
  └── ★ 不生成账变记录
```

**关键发现：**
- freeze 和 rollback **不生成账变记录** — 只是"锁定/解锁"，不算真正的资金流动
- subFreeze **生成账变记录** — 这才是真正的资金扣除
- 这与我们的 E4/E5/E6 设计完全一致

### 5.4 余额校验规则详解

```java
// CoinRecordServiceImpl 中的 insufficientBalance 校验逻辑（还原）

if (balanceType == 支出) {
    BigDecimal available = amount;
    if (freezeAmount != null) {
        available = amount - freezeAmount;  // 可用 = 总额 - 冻结
    }
    if (available < coinValue) {
        return true;  // 余额不足
    }
}

// 但在 SQL 层还有第二道防线：
// subBalance 的 WHERE 条件：(b.amount - #{vo.amount}) >= 0
// 注意：这里检查的是 amount（总额），不是 available（可用余额）
// 这意味着冻结校验在 Service 层，总额校验在 SQL 层
```

**双重校验机制：**
1. **Service层**：检查 `(amount - freezeAmount) >= coinValue` → 可用余额够不够
2. **SQL层**：检查 `(amount - coinValue) >= 0` → 总余额够不够（兜底）
3. 两层都通过才扣款成功

## 六、分布式锁设计

### 6.1 锁键定义

```java
// RedisLockKey.java
public static String ADD_COIN_LOCK_KEY = "add.coin.lock.key.%s";

public static String getAddCoinLockKey(String userId) {
    return String.format(ADD_COIN_LOCK_KEY, userId);
}
```

**锁粒度：per-userId**
- 同一用户的余额操作串行化
- 不同用户完全并行
- 格式：`add.coin.lock.key.{userId}`

### 6.2 锁参数

```java
RLock rLock = redisUtil.getRedissonClient().getLock(lockKey);
boolean tryLock = rLock.tryLock(20000, 30000, TimeUnit.MILLISECONDS);
```

| 参数 | 值 | 含义 |
|------|-----|------|
| waitTime | 20000ms (20秒) | 等待获取锁的最大时间 |
| leaseTime | 30000ms (30秒) | 持锁最大时间（自动释放防死锁） |
| 类型 | Redisson RLock | 可重入分布式锁 |

### 6.3 锁释放安全模式

```java
finally {
    if (rLock != null && rLock.isLocked() && rLock.isHeldByCurrentThread()) {
        rLock.unlock();
    }
}
```

**三重检查释放：**
1. `rLock != null` — 锁对象存在
2. `rLock.isLocked()` — 锁确实被持有
3. `rLock.isHeldByCurrentThread()` — 是当前线程持有（防误解锁）

### 6.4 哪些操作加锁

| 操作 | 是否加锁 | 说明 |
|------|---------|------|
| addCoin（HTTP） | ✅ 加锁 | 余额变动必须串行 |
| addCoinList（HTTP） | ✅ 加锁 | 批量也加锁（用第一个用户的锁） |
| freezeCoin（HTTP） | ✅ 加锁 | 冻结操作加锁 |
| subFreezeBalance（HTTP） | ✅ 加锁 | 解冻扣款加锁 |
| rollbackFreezeBalance（HTTP） | ✅ 加锁 | 回滚冻结加锁 |
| NoticeAddCoinMessage（MQ） | ✅ 加锁 | MQ单条消息加锁 |
| noticeDeductCoinOrderMessage（MQ） | ❌ 不加锁 | MQ批量靠事务保证 |
| addCoinNotNoticeBalance（MQ） | ❌ 不加锁 | 同上 |
| 所有查询操作 | ❌ 不加锁 | 读操作无需锁 |

### 6.5 可借鉴的点

- **锁粒度 per-userId** 是正确的设计，我们应该采纳
- **锁参数** 20s等待/30s持有，在高并发游戏场景下合理
- **三重检查释放** 模式值得参考
- **MQ批量不加分布式锁，靠DB事务保证** — 这是性能和一致性的权衡

### 6.6 Go 实现方案

```go
// 对应的 Go 实现思路（使用 go-redis）
lockKey := fmt.Sprintf("wallet.lock.%s", userId)
lock, err := redisLock.Obtain(ctx, lockKey, 30*time.Second, &redisLock.Options{
    RetryStrategy: redisLock.LimitRetry(redisLock.LinearBackoff(500*time.Millisecond), 40),
    // 40次重试 × 500ms = 20秒最大等待时间
})
defer lock.Release(ctx)
```

或使用 `redsync` 库：

### 9.2 汇率换算逻辑

```java
// CurrencyBusiness.toUSDAmount()
public BigDecimal toUSDAmount(String currency, BigDecimal amount) {
    if (currency == "USD") return amount;           // USD直接返回
    BigDecimal rate = getCurrencyRate(currency);     // 从Redis取汇率
    return amount.divide(rate, 2, RoundingMode.HALF_UP); // 金额 ÷ 汇率 = USD
}

// 汇率定义：1 USD = X 目标币种
// 例如：1 USD = 7.0 CNY
// 100 CNY → toUSDAmount = 100 / 7.0 = 14.29 USD
```

**汇率方向**：`1 USD = X 目标币种`（以美元为基准）

## 十一、设计亮点与不足

### 11.1 设计亮点（值得学习）

#### ✅ 亮点1：SQL层余额校验（乐观更新）

```sql
-- subBalance 的 WHERE 条件
AND (b.amount - #{vo.amount}) >= 0
```

- 不用 SELECT→CHECK→UPDATE 的三步模式，直接在 UPDATE 的 WHERE 中校验
- 影响行数=0 即表示余额不足，无需额外查询
- **原子性保证**：避免并发下的 TOCTOU 问题
- **我们可以采纳**：GORM 的 `RowsAffected` 可实现同样效果

#### ✅ 亮点2：冻结/解冻/回滚三操作分离

- freeze: 只加 freeze_amount
- subFreeze: 减 amount + 减 freeze_amount（真正扣钱）
- rollback: 只减 freeze_amount（退回可用）
- 语义清晰，且只有 subFreeze 生成账变记录
- **完全匹配我们 E4/E5/E6 设计** — 验证了我们的方向正确

#### ✅ 亮点3：前后快照审计（coinFrom/coinTo）

- 每笔账变记录 变动前余额(coinFrom) 和 变动后余额(coinTo)
- 形成完整的余额变化链，便于对账审计
- **我们应该采纳**：在 wallet_transaction 表中增加 before_balance / after_balance

#### ✅ 亮点4：per-userId 粒度锁

- 锁键格式 `add.coin.lock.key.{userId}`
- 不同用户完全并行，同一用户串行化
- 锁参数合理（20s等待/30s持有）
- **我们应该采纳**：Go 中用 redis setnx 或 redsync 实现同样粒度

#### ✅ 亮点5：结算回滚允许负余额

```sql
-- coinType != 6(回滚) and coinType != 5(取消) 时才校验
<if test="vo.coinType != 6 and vo.coinType != 5">
    AND (b.amount - #{vo.amount}) >= 0
</if>
```

- 系统级操作（取消/回滚）必须执行，不能被余额阻挡
- 这是**金融系统的正确做法** — 先保证账务正确性，再处理负余额
- **我们应该采纳**：在特定 coinType 下跳过余额校验

### 11.2 设计不足（需要规避）

#### ❌ 不足1：币种硬编码

```java
public enum CurrencyEnum {
    USD("USD", "USD", "$",true),
    CNY("CNY", "CNY", "CN¥",true),
    // ... 只有5种
}
```

- 新增币种需要改代码发版
- **我们已规避**：currency_config 动态表

#### ❌ 不足2：账变记录冗余过多字段

coin_record 冗余了 user_name, nick_name, merchant_id, merchant_name, merchant_no, agent_no, agent_id, agent_name, path, user_belong 等 10+ 字段。

- 优点：查询不需要 JOIN
- 缺点：写入时要查用户/商户/代理信息→增加延迟，数据可能不一致
- **我们的策略**：wallet_transaction 只存 user_id + 关键ID，查询时 JOIN

#### ❌ 不足3：MQ无重试和死信机制

```java
// 消费失败直接吞掉
catch (Exception e) {
    log.error("消费失败", e);
}
```

- 无死信队列、无重试、无补偿机制
- 依赖订单状态回滚 → 但这不保证最终一致性
- **我们应该做得更好**：消息重试 + 死信队列 + 补偿Job

#### ❌ 不足4：SQL注入风险

```xml
ORDER BY s.created_time ${vo.createdTimeBy}
<!-- ${} 是字符串拼接，不是参数绑定 -->
```

- 排序字段通过 `${}` 拼接到 SQL → SQL注入风险
- **我们绝不能犯这个错误**：排序字段必须白名单校验

#### ❌ 不足5：批量加锁只锁第一个用户

```java
// addCoinList 方法
String userId = coinRecordVOList.get(0).getUserId(); // 只取第一个
RLock rLock = redisUtil.getRedissonClient().getLock(lockKey);
```

- 批量操作可能涉及多个用户，但只锁了第一个
- 并发情况下其他用户的余额操作不受保护
- **我们应该避免**：批量操作要么逐用户加锁，要么限制同一用户

#### ❌ 不足6：无软删除

- user_coin 和 coin_record 都没有软删除字段
- 物理删除后无法恢复
- **我们已规避**：统一 delete_at 软删除

#### ❌ 不足7：getWallet 安全黑科技

```java
// 非正式用户通过API查余额时，强制返回0
if (md5Sign != null && userType != FORMAL_PLAY) {
    amount = ZERO;  // 防止把体验会员的钱搞到包网
}
```

- 这种"安全控制"通过返回虚假数据实现 → 不透明、难维护
- **我们不应该这样做**：安全控制应该在鉴权层，不应该在业务数据层伪造返回值

### 13.1 强烈建议采纳（验证了我们设计方向正确）

| # | 采纳项 | 参考来源 | 我们的实现方式 | 优先级 |

### 13.2 建议采纳（提升工程质量）

| # | 采纳项 | 参考来源 | 我们的实现方式 | 优先级 |
|---|--------|---------|--------------|--------|
| B1 | **双重校验** (Service+SQL) | addCoin逻辑 | Service层先查可用余额做业务校验，SQL层做兜底 | 应该 |

### 13.3 锁参数建议值

```
参考值        : waitTime=20s, leaseTime=30s
建议我们采用  : waitTime=15s, leaseTime=25s

理由：
- 我们用 Kitex RPC（高性能）替代 Feign HTTP → 内部调用更快
- 正常一笔余额操作 < 100ms，15s 等待绰绰有余
- 25s 持有可覆盖正常流程 + 网络抖动
- 如果 25s 还没完成，说明有异常，自动释放比死等好
```

## 十五、总结

### 15.1 参考工程定位

这是一个 **生产环境运行的游戏平台钱包系统**，设计务实但不完美：
- 核心余额操作逻辑**设计正确且经过验证**
- 冻结/解冻/回滚模式**成熟可靠**
- 工程细节（MQ、排序、批量锁）**有瑕疵但不影响核心逻辑**
- 币种管理**过于简陋**（硬编码5种），不满足我们多币种动态配置的需求

### 15.2 核心收获

1. **验证了我们的E4/E5/E6设计方向完全正确** — 参考工程用同样的 freeze/sub/rollback 三操作模式
2. **SQL层余额校验是关键设计** — 不能只在Service层校验，SQL层的WHERE条件是最后的安全网
3. **per-userId粒度锁是标准做法** — 参考工程和我们的设计方向一致
4. **前后余额快照是必须的** — coinFrom/coinTo 形成审计链，我们应该在表结构中加上
5. **特殊操作允许负余额** — 结算回滚和订单取消不受余额限制，这是金融系统正确做法

### 15.3 核心差异

我们的设计在以下方面**优于参考工程**：
- 多币种动态配置（vs 硬编码5种）
- 表结构规范化（vs 宽表冗余）
- 软删除（vs 无软删除）
- 更完整的业务域覆盖（12表 vs 3表）
- IDL先行的RPC设计（vs HTTP REST）

参考工程在以下方面**值得我们学习**：
- SQL层原子性余额校验
- 生产验证的锁参数配置
- 前后余额快照机制
- 余额变化实时通知机制
- MQ消息追踪表
