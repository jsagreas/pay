## 3. 核心数据模型（3表）

参考工程钱包核心仅 **3 张表**，对比我方 **10 张表**。

### 3.1 user_coin（用户钱包主表）

```sql
-- 对应 UserCoinDTO.java
user_coin (
    id              BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id         VARCHAR        -- 用户 ID（字符串）
    user_name       VARCHAR        -- 用户名
    amount          DECIMAL        -- 可用余额（BigDecimal）
    freeze_amount   DECIMAL        -- 冻结金额（BigDecimal）
    user_state      INT            -- 用户状态
    game_type       INT            -- 游戏类型
    wallet_type     INT            -- 钱包模式：1=TRANSFER / 2=FREE_TRANSFER
    currency        VARCHAR        -- 币种编码
    remark          VARCHAR        -- 备注
    creator         BIGINT         -- 创建人
    updater         BIGINT         -- 更新人
    created_time    DATETIME       -- 创建时间
    updated_time    DATETIME       -- 更新时间
)
```

**设计特点**：
- **单钱包制**：一个 user_id 只有一行记录（一个钱包 + 一个币种）
- **BigDecimal 存储**：金额直接用 DECIMAL 类型，不做最小单位转换
- **可用余额 = amount - freeze_amount**：可用余额不独立存储，运行时计算
- **无多钱包类型区分**：不区分充值钱包/赠金钱包/佣金钱包，一个用户一个余额
- **wallet_type 含义不同**：这里的 wallet_type 是钱包模式（内部管理 vs 第三方 API），不是我方的多钱包类型

**与我方 wallet 表对比**：

| 维度 | 参考工程 user_coin | 我方 wallet |
|------|-------------------|-------------|
| 主键逻辑 | user_id（1对1） | user_id + wallet_type + currency（1对多） |
| 金额类型 | BigDecimal | int64（最小单位整数） |
| 余额字段 | amount + freeze_amount | balance + frozen_amount |
| 可用余额 | amount - freeze_amount（计算） | available_balance（字段 or 计算，待定） |
| 多钱包 | 不支持 | 支持（充值/赠金/佣金/彩金等） |
| 币种 | 单币种字段 | 独立维度（user_id + wallet_type + currency 三元唯一） |
| 版本控制 | 无 | version（乐观锁） |

### 3.2 coin_record（流水记录表）

```sql
-- 对应 CoinRecordDTO.java
coin_record (
    id              BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id         VARCHAR        -- 用户 ID
    user_name       VARCHAR        -- 用户名
    user_type       INT            -- 用户类型：0=正式 / 1=Demo
    nick_name       VARCHAR        -- 昵称
    user_belong     VARCHAR        -- 归属
    merchant_id     VARCHAR        -- 商户 ID
    merchant_name   VARCHAR        -- 商户名称
    merchant_no     VARCHAR        -- 商户编号
    agent_no        VARCHAR        -- 代理编号
    agent_id        VARCHAR        -- 代理 ID
    agent_name      VARCHAR        -- 代理名称
    path            VARCHAR        -- 代理层级路径
    order_no        VARCHAR        -- 关联订单号
    coin_type       INT            -- 变动类型（CoinChangeEnum：1-9）
    coin_detail     INT            -- 变动明细（CoinChangeDetailEnum：1-8）
    coin_value      DECIMAL        -- 变动金额（BigDecimal）
    coin_from       DECIMAL        -- 变动前余额
    coin_to         DECIMAL        -- 变动后余额
    coin_amount     DECIMAL        -- 变动时总余额
    balance_type    INT            -- 收支方向：1=收入 / 2=支出
    remark          VARCHAR        -- 备注
    creator         BIGINT
    updater         BIGINT
    created_time    DATETIME
    updated_time    DATETIME
)
```

**设计特点**：
- **快照式流水**：每笔流水记录 coin_from（变动前）和 coin_to（变动后）+ coin_amount（总余额）
- **商户/代理冗余**：直接在流水表冗余商户名称、代理名称、代理路径，方便后台查询不需 JOIN
- **coin_type + coin_detail 二级分类**：大类（9种）+ 细分（8种），组合表达完整业务语义
- **无幂等键**：没有 idempotent_key / request_id 字段，幂等依赖 order_no + 分布式锁
- **无 wallet_type 关联**：因为只有单钱包，流水不需要区分来自哪个钱包类型

**与我方 wallet_flow 表对比**：

| 维度 | 参考工程 coin_record | 我方 wallet_flow |
|------|---------------------|-----------------|
| 幂等机制 | 依赖 order_no + 锁 | idempotent_key 唯一索引 |
| 多钱包 | 无 | wallet_type 字段 |
| 快照 | coin_from / coin_to / coin_amount | before_balance / after_balance |
| 业务分类 | coin_type(9) + coin_detail(8) | biz_type + biz_sub_type |
| 商户冗余 | 冗余在流水表 | 不冗余（查询时关联） |
| Demo 隔离 | user_type 字段 | 无（不涉及 Demo 用户） |

### 3.3 add_coin_message（MQ 消息追踪表）

```sql
-- 对应 OrderMessageDTO.java
add_coin_message (
    id              BIGINT AUTO_INCREMENT PRIMARY KEY,
    message_id      VARCHAR        -- 消息唯一 ID
    order_id_list   VARCHAR        -- 关联订单 ID 列表（JSON 或逗号分隔）
    status          INT            -- 投递状态
    try_count       INT DEFAULT 0  -- 重试次数

    update_time     DATETIME
)
```

**设计特点**：
- **MQ 可靠投递保障**：先写 DB 记录 → 发 MQ → ConfirmCallback 更新状态

- **批量关联**：一条消息可关联多个订单

**借鉴价值**：我方如果使用 NATS/Kafka 做异步资金操作，可参考此模式保障消息可靠性。但如果所有资金操作都走同步 RPC，则此表不需要。

### 4.3 汇率模型

**汇率语义**：`1 USD = X 目标币种`

```java
// CurrencyBusiness.java — 汇率换算核心
public BigDecimal toUSDAmount(BigDecimal amount, BigDecimal currencyRate) {
    // USD 金额 = 本地币种金额 / 汇率
    return amount.divide(currencyRate, 2, RoundingMode.HALF_UP);
}
```

**换算方向**：
- 充值时：用户支付 1000 VND → 除以汇率 → 得到 USD 等值金额
- 钱包余额始终以某一基准币种存储（取决于用户/商户配置）

### 4.4 Redis 多层缓存策略

```
层级1：all-service::currency                   → 全量币种列表（JSON）
层级2：all-service::currency_rate              → 全量汇率列表（JSON）
层级3：all-service::currency_rate::{code}      → 单币种汇率值
层级4：all-service::currency::{symbol}::code   → 符号 → 编码映射
```

**刷新链路**：DB 更新 → Redis 全量刷新 → MQ 通知下游（用户层级系统）

### 5.4 钱包模式（WalletTypeEnum）

```java
public enum WalletTypeEnum {
    TRANSFER(1, "转账模式"),         // 内部余额管理
    FREE_TRANSFER(2, "免转模式");    // 第三方商户 API 管理余额
}
```

**TRANSFER vs FREE_TRANSFER 解读**：
- **TRANSFER（转账模式）**：用户余额由平台自身管理，存储在 user_coin.amount 中
- **FREE_TRANSFER（免转模式）**：用户余额由第三方游戏商管理，平台不存储真实余额，查询时返回 0 或调用外部 API

这是 B2B 游戏平台特有的概念。我方项目如果不涉及第三方游戏商余额托管，则只需 TRANSFER 模式。

## 6. 资金操作核心流程

### 6.1 加款流程（addCoin）

```
CoinRecordServiceImpl.addCoin(UserCoinVO vo)
│
├── 1. 判断 wallet_type
│   ├── TRANSFER → 继续内部处理
│   └── FREE_TRANSFER → 调用外部 API 加款 → return
│
├── 2. 查询 user_coin 记录
│   └── 不存在 → 抛异常（用户不存在）
│
├── 3. 余额验证（仅扣款类型）
│   ├── coin_type ∈ {TRANSFER_OUT, BET, SEND_REWARD} → 校验 amount - freeze ≥ 请求金额
│   ├── coin_type ∈ {SETTLEMENT_ROLLBACK, ORDER_CANCEL} → 跳过校验（允许负余额）
│   └── 其他 → 不校验（加款无需校验）
│
├── 4. 执行 SQL 更新 user_coin
│   ├── 加款 → UPDATE user_coin SET amount = amount + #{value}
│   └── 扣款 → UPDATE user_coin SET amount = amount - #{value} WHERE amount - freeze_amount >= #{value}
│
├── 5. 构建 coin_record 流水
│   ├── 拉取商户信息（CommonService → Redis 缓存 → Feign 降级）
│   ├── 填充 coin_from = 变动前 amount
│   ├── 填充 coin_to = 变动后 amount
│   └── INSERT coin_record
│
├── 6. 发送余额变动通知（MQ → WebSocket）
│   └── RabbitSender.send(NOTICE_BALANCE_MQ, balanceMessage)
│
└── 7. return 结果
```

**关键设计决策**：
- 余额校验在 SQL 层面有 WHERE 兜底（`amount - freeze_amount >= #{value}`），即使并发场景也不会扣负
- SETTLEMENT_ROLLBACK 和 ORDER_CANCEL 允许负余额 —— 结算回滚和取消退款属于异常修正，必须执行
- 商户信息查询用 Redis 缓存（TTL 1天），缓存未命中才 Feign 远程调用

### 6.2 扣款流程（deductCoin）

```

├── balance_type 设为 expenses

│        WHERE user_id = #{userId}
│        AND amount - freeze_amount >= #{value}   ← SQL 层余额兜底
```

### 6.4 余额查询（getWallet）

```java
// UserCoinServiceImpl.getWallet()
public UserCoinDTO getWallet(String userId) {
    UserCoinDTO userCoin = userCoinMapper.selectOne(userId);
    if (userCoin == null) return null;

    // TRANSFER 模式：返回 amount - freezeAmount
    if (walletType == TRANSFER) {
        userCoin.setAmount(amount.subtract(freezeAmount));
    }
    // FREE_TRANSFER 模式：返回 0（余额在第三方）
    else {
        userCoin.setAmount(BigDecimal.ZERO);
    }

    // 安全校验：非正式 API 用户强制返回 0
    if (user != null && user.getUserType() != FORMAL_API_USER) {
        userCoin.setAmount(BigDecimal.ZERO);
    }

    return userCoin;
}
```

**关键点**：
- 返回给外部的 amount 是 `amount - freezeAmount`（可用余额），不是数据库原始值
- 非正式用户（如测试用户通过非标准渠道接入）强制返回 0，防止信息泄露

## 7. 冻结模型与三阶段设计

### 7.1 冻结流程（freezeCoin → addFreezeBalance）

```sql
-- UserCoinMapper.xml
<update id="addFreezeBalance">
    UPDATE user_coin
    SET freeze_amount = freeze_amount + #{freezeAmount}
    WHERE user_id = #{userId}
    AND amount - freeze_amount >= #{freezeAmount}   ← 可用余额 ≥ 冻结金额
</update>
```

**语义**：将指定金额从"可用"区域转移到"冻结"区域。`amount` 不变，`freeze_amount` 增加。

### 7.2 冻结确认扣款（subFreezeBalance）

```sql
-- UserCoinMapper.xml
<update id="subFreezeBalance">
    UPDATE user_coin
    SET amount = amount - #{freezeAmount},
        freeze_amount = freeze_amount - #{freezeAmount}
    WHERE user_id = #{userId}
</update>
```

**语义**：冻结的钱真正扣出去。`amount` 减少，`freeze_amount` 回减。两个字段同步减少同一金额。

### 7.3 冻结回滚（rollbackFreezeBalance）

```sql
-- UserCoinMapper.xml
<update id="rollbackFreezeBalance">
    UPDATE user_coin
    SET freeze_amount = freeze_amount - #{freezeAmount}
    WHERE user_id = #{userId}
</update>
```

**语义**：冻结取消，钱返回"可用"区域。只减 `freeze_amount`，`amount` 不变。

```
                        │ freeze = 0  │

                               │
                    freezeCoin(30)
                               │
```

| 维度 | 参考工程 | 我方设计 |
|------|---------|---------|

## 9. 分布式锁与并发控制

### 9.1 参考工程方案：Redisson 分布式锁

```java
// WalletController.java — 所有写操作入口
@PostMapping("/addCoin")
public Result addCoin(@RequestBody UserCoinVO vo) {
    RLock lock = redissonClient.getLock(
        RedisLockKey.ADD_COIN_LOCK_KEY + vo.getUserId()
    );
    try {
        if (lock.tryLock(20, 30, TimeUnit.SECONDS)) {
            // waitTime=20s：等待获取锁最长 20 秒
            // leaseTime=30s：锁自动释放时间 30 秒
            return coinRecordService.addCoin(vo);
        } else {
            return Result.fail("系统繁忙，请稍后再试");
        }
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
        return Result.fail("操作中断");
    } finally {
        if (lock.isHeldByCurrentThread()) {
            lock.unlock();
        }
    }
}
```

**锁 Key 模板**：`add.coin.lock.key.{userId}`

**特点**：
- 锁粒度：**用户级别**（同一用户的所有资金操作串行）
- 锁位置：**Controller 层**（最外层加锁）
- 等待策略：最长等待 20 秒，超时返回失败
- 自动释放：30 秒自动释放，防死锁
- 可重入：Redisson 原生支持

### 9.2 我方方案：SELECT FOR UPDATE 悲观锁

```sql
-- 我方设计
SELECT * FROM wallet
WHERE user_id = ? AND wallet_type = ? AND currency = ?
FOR UPDATE
```

**特点**：
- 锁粒度：**钱包行级别**（比用户级更细）
- 锁位置：**domain 层**（资金原子操作内部加锁）
- 等待策略：数据库行锁等待（TiDB 默认超时配置）
- 自动释放：事务结束自动释放
- 无需额外组件（不需要 Redisson）

### 9.3 锁方案对比

| 维度 | Redisson 分布式锁 | SELECT FOR UPDATE |
|------|------------------|-------------------|
| 锁粒度 | 用户级（粗） | 行级（细） |
| 跨钱包影响 | 同用户所有操作排队 | 仅锁定目标钱包行 |
| 性能 | 锁在 Redis，DB 无锁竞争 | 锁在 DB 行，可能有行锁等待 |
| 依赖 | 依赖 Redis 可用性 | 仅依赖 DB |
| 死锁风险 | 超时自动释放 | 需注意加锁顺序 |
| 实现复杂度 | 需额外库 | 原生 SQL |

**结论**：两种方案各有优劣。我方选择 SELECT FOR UPDATE 是合理的，因为：
1. 多钱包设计下行级锁粒度更合适（不同钱包类型可并发操作）
2. TiDB 对行锁有良好支持
3. 减少基础设施依赖
4. domain 层加锁 + 明确的加锁顺序可避免死锁

| 对比项 | 参考工程 | 我方设计 | 评估 |
|--------|---------|---------|------|
| 冻结模型 | 三阶段（freeze/sub/rollback） | 三阶段 + 冻结明细 + 消费优先级 | 核心思路一致，我方更精细 |

| 允许负余额 | SETTLEMENT_ROLLBACK, ORDER_CANCEL | 待定 | 可借鉴：异常修正场景允许负余额 |

| 余额通知 | MQ → WebSocket | 待定 | 可借鉴模式 |
| Demo 用户 | user_type 隔离 | 不涉及 | 不需要 |

### 13.4 BigDecimal vs int64 深度对比

| 维度 | BigDecimal（参考工程） | int64 最小单位（我方） |
|------|----------------------|---------------------|
| 精度 | 任意精度 | 固定精度（由 scale 决定） |
| 性能 | 对象操作，较慢 | 整数运算，极快 |
| 存储 | DECIMAL 列 | BIGINT 列 |
| 溢出风险 | 无（任意精度） | 需注意（int64 max ≈ 922 亿 × 10^8） |
| 前端交互 | 需要字符串传输 | 需要 前端/后端 各自转换 |
| 汇率换算 | 直接 divide | 需注意整数除法精度 |
| 业界偏好 | Java 项目主流 | Go 项目常见 |

**结论**：我方选择 int64 最小单位存储是 Go 项目的标准做法，与参考工程的差异来自语言生态而非设计缺陷。

## 15. 不宜照搬清单与理由

| # | 不宜照搬项 | 理由 |
|---|-----------|------|
| 1 | **单钱包制** | 我方需要多钱包（充值/赠金/佣金/彩金），参考工程单钱包无法满足 |
| 2 | **BigDecimal 金额** | Go 没有原生 BigDecimal，int64 最小单位是 Go 生态标准做法 |
| 3 | **FREE_TRANSFER 模式** | 第三方游戏商余额托管是参考工程特有场景，我方不涉及 |
| 4 | **商户/代理冗余在流水表** | 冗余太多列影响性能和一致性，我方流水表只存 ID，查询时关联 |
| 5 | **Demo 用户隔离** | 我方不涉及 Demo 用户机制 |
| 6 | **Redisson 分布式锁** | 我方已选择 SELECT FOR UPDATE，行级锁粒度更适合多钱包场景 |
| 7 | **Controller 层手写 REST** | 我方走 Thrift IDL → Kitex 生成代码，不手写 Controller |
| 8 | **MyBatis XML SQL** | 我方使用 gorm_gen 自动生成 + rep 层手写复杂查询 |

### B.1 余额不足判断逻辑

```java
// CoinRecordServiceImpl.java — insufficientBalance()
private boolean insufficientBalance(UserCoinDTO userCoin, BigDecimal value, Integer coinType) {
    // 结算回滚（5）和订单取消（6）不校验余额 — 允许负余额
    if (coinType == 5 || coinType == 6) {
        return false;
    }
    // 只对转出（2）、投注（3）、发奖（7）校验
    if (coinType == 2 || coinType == 3 || coinType == 7) {
        BigDecimal available = userCoin.getAmount().subtract(userCoin.getFreezeAmount());
        return available.compareTo(value) < 0;
    }
    return false;
}
```

### B.2 MQ 消费端加锁模式

```java
// AddCoinListener.java — 消费时加锁
@RabbitListener(queues = MqConstants.ADD_COIN_QUEUE)
public void addCoin(UserCoinVO vo) {
    RLock lock = redissonClient.getLock(
        RedisLockKey.ADD_COIN_LOCK_KEY + vo.getUserId()
    );
    try {
        if (lock.tryLock(20, 30, TimeUnit.SECONDS)) {
            coinRecordService.addCoin(vo);
        } else {
            log.error("获取锁超时, userId={}", vo.getUserId());
            // 消息会被 NACK，进入重试队列
        }
    } finally {
        if (lock.isHeldByCurrentThread()) {
            lock.unlock();
        }
    }
}
```
