**与我们的对比**：参考工程是标准三层（Controller → Service → Repository），**没有独立的 Ledger 层**。所有余额操作（加减钱、冻结、解冻）直接在 Service 里调 Repository 的原子 SQL。我们设计了独立的 Ledger 层（5 个原子操作 + 事务 + 幂等），这是我们的架构优势。

```sql
    updated_time    DATETIME

```

**关键设计观察**：

| 观察点 | 参考工程做法 | 我们的设计方向 | 备注 |
|--------|------------|--------------|------|
| 币种模型 | **单币种/单行**（user_id + currency 一行） | 多币种/多行（user_id + currency_code 复合唯一） | 参考工程一个用户只有一种币种 |
| 金额类型 | BigDecimal(20,2) | BIGINT（最小货币单位） | 我们方案无浮点误差风险 |
| 可用余额 | 计算得出（amount - freeze_amount） | 同思路，不单独存 | 一致 |
| wallet_type | 区分转账/免转两种模式 | 暂无此需求 | 可能用不到 |
| user_state/game_type | 存在钱包表里 | 不应该放钱包表 | 参考工程设计耦合 |

# 四、核心业务逻辑深度分析

## 4.1 addCoin — 加减钱主流程（最核心方法）

> 源码：`CoinRecordServiceImpl.addCoin()` — 这是钱包的**心脏方法**，所有余额变动最终走这里

```
调用入口（两种）：
  1. HTTP API → WalletController.addCoin() → 获取分布式锁 → service.addCoin()
  2. MQ 消费 → AddCoinListener.NoticeAddCoinMessage() → 获取分布式锁 → service.addCoin()

addCoin 内部流程：
  ┌─────────────────────────────────────────────────────────────┐
  │ 1. 参数校验                                                  │
  │    · coinType 必须有效（> 0）                                 │
  │    · coinValue 必须 > 0（在 Controller 层校验）               │
  ├─────────────────────────────────────────────────────────────┤
  │ 2. 获取用户信息                                              │
  │    · commonService.getUserInfoByUserId(userId)               │
  │    · 先查 Redis 缓存，miss 则走 Feign 调 user-service        │
  │    · 获取：userName, walletType, currency, merchantId 等     │
  ├─────────────────────────────────────────────────────────────┤
  │ 3. ★★ 按钱包类型分支                                        │
  │    ┌──────────────────────┬─────────────────────────────┐    │
  │    │  walletType = 1      │  walletType = 2             │    │
  │    │  转账钱包（本地余额） │  免转钱包（三方 API 余额）    │    │
  │    ├──────────────────────┼─────────────────────────────┤    │
  │    │ 查 user_coin 记录    │ 查订单信息                    │    │
  │    │ ↓ 存在:              │ 构造三方请求                  │    │
  │    │  余额校验            │ 调 api-service 加减余额       │    │
  │    │  收入→addBalance     │ 记录三方订单                  │    │
  │    │  支出→subBalance     │ 发送余额变更 MQ 通知          │    │
  │    │  记录 coinFrom/To    │ coinFrom/coinTo 设为 null    │    │
  │    │ ↓ 不存在:            │                              │    │
  │    │  新建 user_coin      │                              │    │
  │    │  （必须是收入）       │                              │    │
  │    ├──────────────────────┼─────────────────────────────┤    │
  │    │ 写入 coin_record     │ 写入 coin_record             │    │
  │    └──────────────────────┴─────────────────────────────┘    │
  └─────────────────────────────────────────────────────────────┘
```

**addCoin 关键代码逻辑解读**：

```java
// 转账钱包分支核心逻辑（简化）
if (walletType == TRANSFER) {
    UserCoinDTO userCoin = findByUserId(userId);
    if (userCoin != null) {
        // ★ 余额校验：转出/下注/打赏 需要检查可用余额
        insufficientBalance(coinRecordVO, userCoin);

        if (balanceType == income) {
            userCoinRepository.addBalance(balanceVO);     // SQL: amount += X
            coinTo = userCoin.amount + coinValue;
        }
        if (balanceType == expenses) {
            int rows = userCoinRepository.subBalance(balanceVO);  // SQL: amount -= X (含非负判断)
            if (rows == 0) throw "余额不足";
            coinTo = userCoin.amount - coinValue;
        }
        // 记录前后余额快照
        coinRecord.coinFrom = userCoin.amount;
        coinRecord.coinTo = coinTo;
    } else {
        // 首次加钱：创建钱包记录
        checkCoinChange(coinRecordVO);  // 不能是支出
        insert(new UserCoinDTO(userId, coinValue, 0, currency, walletType));
    }
    insert(coinRecord);  // 写入流水
}
```

**我们可以借鉴的点**：
1. **前后余额快照**（coinFrom / coinTo）写入流水 — 审计追踪必需
2. **余额校验仅对特定操作**（转出/下注/打赏才检查，结算回滚允许为负）
3. **首次加钱自动创建钱包** — lazy init 模式
4. **免转钱包分支**的概念 — 我们暂时不需要，但知道其存在

**参考工程的问题/不足**：
1. **没有事务注解在 Controller 层**，Service 层有 `@Transactional` 但锁在 Controller — 锁和事务边界不一致
2. **addCoin 方法过长**（170+ 行，两种钱包类型混在一起） — 可读性差
3. **余额校验和 SQL 校验重复** — `insufficientBalance()` 在 Java 层检查，`subBalance` SQL 里还有 `AND (amount - X) >= 0` 条件
4. **没有流水幂等** — order_no 未做唯一索引，相同订单可能写多条流水

## 4.2 余额操作原子 SQL（5 条）

> 源码：`UserCoinMapper.xml` — 这是钱包最底层的 5 个原子操作

```
┌─────────────────────────────────────────────────────────────────────┐
│  操作            │  SQL 语义                          │  产生账变？  │
├─────────────────┼────────────────────────────────────┼────────────┤
│  addBalance     │  amount += X                       │  是         │
│  subBalance     │  amount -= X （条件非负*）          │  是         │
│  addFreezeBalance│  freeze_amount += X               │  否         │
│  subFreezeBalance│  amount -= X, freeze_amount -= X  │  是         │
│  rollbackFreeze │  freeze_amount -= X                │  否         │
└─────────────────┴────────────────────────────────────┴────────────┘

* subBalance 的非负条件：
  - 默认：AND (amount - #{amount}) >= 0  → 防止余额为负
  - 例外：coinType=6(结算回滚) 或 coinType=5(取消) 时跳过此条件 → 允许负余额
```

**具体 SQL（逐条分析）**：

```xml
<!-- 1. 加余额：无条件，直接加 -->
<update id="addBalance">
    UPDATE user_coin SET amount = amount + #{vo.amount}, updated_time = #{vo.updateTime}
    WHERE id = #{vo.id}
</update>

<!-- 2. 减余额：有条件非负检查，结算回滚/取消 除外 -->
<update id="subBalance">
    UPDATE user_coin SET amount = amount - #{vo.amount}, updated_time = #{vo.updateTime}
    WHERE id = #{vo.id}
    <if test="vo.coinType != 6 and vo.coinType != 5">
        AND (amount - #{vo.amount}) >= 0    -- ★ 数据库级非负保障
    </if>
</update>

<!-- 3. 冻结：增加冻结金额，总额不变，可用减少 -->
<update id="addFreezeBalance">
    UPDATE user_coin SET freeze_amount = freeze_amount + #{vo.amount}, updated_time = #{vo.updateTime}
    WHERE id = #{vo.id}
</update>

<!-- 4. 确认扣减冻结：总额减少，冻结减少，可用不变 -->
<update id="subFreezeBalance">
    UPDATE user_coin SET amount = amount - #{vo.amount}, freeze_amount = freeze_amount - #{vo.amount},
    updated_time = #{vo.updateTime}
    WHERE id = #{vo.id}
</update>

<!-- 5. 回滚冻结：冻结减少，总额不变，可用恢复 -->
<update id="rollbackFreezeBalance">
    UPDATE user_coin SET freeze_amount = freeze_amount - #{vo.amount}, updated_time = #{vo.updateTime}
    WHERE id = #{vo.id}
</update>
```

**与我们 Ledger 5 操作的映射**：

| 参考工程 SQL | 对应我们 Ledger 操作 | 差异 |
|-------------|---------------------|------|
| addBalance | Credit（入账） | 我们有事务+幂等+行锁包裹 |
| subBalance | Debit（出账） | 我们有事务+幂等+行锁包裹 |
| addFreezeBalance | Freeze（冻结） | 一致 |
| subFreezeBalance | Unfreeze + Debit（确认扣减） | 我们拆成两步更清晰 |
| rollbackFreezeBalance | Unfreeze（回滚冻结） | 一致 |
| 无 | Exchange（换汇） | 参考工程没有换汇操作 |

**关键借鉴**：参考工程的 `subBalance` 用 SQL 条件 `AND (amount - X) >= 0` 实现**数据库级非负保障**，通过返回影响行数 `rows == 0` 判断失败。这个模式值得我们直接采用——比先查再改更安全（消除竞态窗口）。

## 4.3 冻结三阶段机制

> 这是参考工程设计最值得借鉴的模式之一

```
冻结场景：投注时先冻结金额，游戏结束后根据结果确认扣减或回滚

阶段 1：freezeCoin（下注时）
  freeze_amount += 下注金额
  可用余额减少（amount - freeze_amount 变小）
  总额 amount 不变
  ★ 不产生账变记录

阶段 2a：subFreezeBalance（结算确认扣减时）
  amount -= 下注金额          // 总额减少
  freeze_amount -= 下注金额   // 冻结释放
  可用余额不变（因为两者同时减）
  ★ 产生账变记录

阶段 2b：rollbackFreezeBalance（取消/退还时）
  freeze_amount -= 下注金额   // 冻结释放
  amount 不变                 // 总额不变
  可用余额恢复
  ★ 不产生账变记录
```

```
余额变化示意（假设初始 amount=1000, freeze=0）：

下注 200:    amount=1000, freeze=200,  可用=800   ← freezeCoin
结算确认:    amount=800,  freeze=0,    可用=800   ← subFreezeBalance
  或
取消回滚:    amount=1000, freeze=0,    可用=1000  ← rollbackFreezeBalance
```

**借鉴价值**：
- 三阶段冻结模式是行业标准做法，我们的 Ledger 层已经设计了 Freeze/Unfreeze，方向一致
- 参考工程 subFreezeBalance 把"解冻+扣减"合为一条 SQL，效率高但原子性弱于两步分离
- freezeCoin 和 rollbackFreezeBalance 不产生账变 — 这是合理的，中间状态不需要记录

## 4.4 余额校验规则

```java
// insufficientBalance() — 只有特定操作才校验余额
private void insufficientBalance(CoinRecordVO coinRecordVO, UserCoinDTO userCoin) {
    // 以下类型需要校验余额：
    if (coinType == TRANSFER_OUT     // 转出
        || coinType == BET           // 下注
        || coinType == SEND_REWARD)  // 打赏
    {
        BigDecimal available = userCoin.amount - userCoin.freezeAmount;  // 可用余额
        if (available < coinValue) {
            throw "余额不足";
        }
    }
    // 其他类型（结算、结算回滚、取消等）不校验 → 允许为负
}
```

**设计意图**：结算回滚（coinType=6）和取消（coinType=5）可能在已经扣款后需要回退金额，此时余额可能已经不足，强制校验会导致业务流程卡死。所以**只有用户主动发起的操作才校验余额，系统级操作跳过校验**。

**SQL 层也有对应逻辑**：`subBalance` 的非负条件 `<if test="vo.coinType != 6 and vo.coinType != 5">` 与 Java 层互为双保险。

# 五、并发控制机制

## 5.1 分布式锁（Redisson）

```java
// 锁的 Key 模式
RedisLockKey.getAddCoinLockKey(userId)  →  "add.coin.lock.key.{userId}"

// 锁参数
rLock.tryLock(
    20000,    // 等待时间：20 秒（获取不到锁则等待）
    30000L,   // 持有时间：30 秒（自动续约/过期）
    TimeUnit.MILLISECONDS
)
```

**锁的使用位置**（所有涉及余额变动的入口都加锁）：

| 入口 | 锁粒度 | 锁位置 |
|------|--------|--------|
| WalletController.addCoin | 用户级 | Controller 层 |
| WalletController.addCoinList | 用户级（取第一个用户） | Controller 层 |
| WalletController.freezeCoin | 用户级 | Controller 层 |
| WalletController.subFreezeBalance | 用户级 | Controller 层 |
| WalletController.rollbackFreezeBalance | 用户级 | Controller 层 |
| AddCoinListener.NoticeAddCoinMessage | 用户级 | Listener 层 |
| AddCoinListener.noticeDeductCoinOrderMessage | **无锁** ⚠️ | — |
| AddCoinListener.addCoinNotNoticeBalance | **无锁** ⚠️ | — |

**问题发现**：
1. 部分 MQ 消费者**没有加锁** — `noticeDeductCoinOrderMessage` 和 `addCoinNotNoticeBalance` 直接调用 `deductCoinBatch` 而没有获取分布式锁，存在并发风险
2. 锁在 Controller 层，事务在 Service 层 — **锁的范围 > 事务范围**，这是正确的（锁必须包裹事务）
3. 锁粒度是 userId — 同一用户的所有操作串行化，不同用户完全并行

**与我们的对比**：
- 参考工程用 Redisson 分布式锁（Redis 级别），我们计划用数据库行锁（`SELECT ... FOR UPDATE`）
- 行锁优势：不依赖额外组件、天然与事务绑定、不存在锁过期问题
- 行锁劣势：锁粒度与 DB 绑定、长事务下性能不如 Redis 锁
- 对于我们的场景（单库 TiDB、中等并发），**数据库行锁更简单可靠**

## 5.2 事务管理

```java
@Transactional(rollbackFor = Throwable.class)   // addCoin — 最严格
@Transactional(rollbackFor = Exception.class)    // addCoin 另一处 — 几乎等效
@Transactional                                    // freezeCoin 等 — 默认只回滚 RuntimeException
```

**事务覆盖范围**：
- `addCoin` / `deductCoin`：查余额 → 更新余额 → 写流水，在一个事务内
- `freezeCoin` / `subFreezeBalance` / `rollbackFreezeBalance`：更新冻结金额，在一个事务内
- 但 `addCoinBatch` 用循环调 `addCoin` — 外层 `@Transactional` 包裹整批，一个失败全部回滚

## 8.3 汇率换算逻辑

```java
// CurrencyBusiness.toUSDAmount() — 其他币种转 USD
public BigDecimal toUSDAmount(String currency, BigDecimal amount) {
    if (currency == "USD") return amount;          // 本身就是 USD
    BigDecimal rate = getCurrencyRate(currency);   // 从 Redis 获取汇率
    return amount.divide(rate, 2, RoundingMode.HALF_UP);  // 除法，保留 2 位小数
}

// 例：100 CNY → USD
// rate = 7.2 (1 USD = 7.2 CNY)
// result = 100 / 7.2 = 13.89 USD
```

**我们如果用 BIGINT 的换算方式**：
```
假设 100 CNY（最小单位：分），rate = 7.2
存储值 = 10000（分）
换算 USD：10000 / 720 = 13（分） → $0.13（有精度丢失）

需要更高精度中间计算：
10000 * 100 / 720 = 1388 → 四舍五入 = 1389（分） → $13.89
```

# 十二、关键设计模式提炼

## 12.1 值得借鉴的模式

### 模式 1：数据库级非负余额保障

```sql
-- 通过 SQL 条件保障，而非仅靠应用层校验
UPDATE user_coin SET amount = amount - #{amount}
WHERE id = #{id} AND (amount - #{amount}) >= 0
-- 返回 affected_rows == 0 表示余额不足
```

**价值**：消除应用层"先查后改"的竞态窗口。即使没有锁，这条 SQL 也能保证余额不为负。我们的 GORM 实现可以直接采用此模式。

### 模式 2：前后余额快照

```
每条账变记录保存：
  coin_from = 操作前余额
  coin_to   = 操作后余额
  coin_value = 变动金额
```

**价值**：审计追踪时可以不依赖时序重算，每条记录自描述。出现异常时可以快速定位"从哪里开始不对"。

### 模式 3：冻结三阶段

```
freeze（预扣） → subFreeze（确认） / rollback（回滚）
```

**价值**：经典的预授权模式，适合投注/支付场景。我们已经在 Ledger 层设计了 Freeze/Unfreeze，方向一致。

### 模式 4：用户信息 Redis 缓存 + Feign 兜底

```java
// CommonServiceImpl.getUserInfoByUserId()
1. Redis 查缓存 → 命中直接返回
2. 未命中 → Feign 调 user-service → 返回（但未回写缓存！）
```

**注意**：参考工程获取用户信息后**没有写回 Redis**（只有 getMerchantDetailVO 才写缓存）。这是一个遗漏。我们应该做到：查DB/RPC → 写缓存 → 返回。

### 模式 5：余额变更主动推送

```
余额变动后 → MQ 发通知 → WebSocket 推送给 C 端
```

**价值**：用户在前端能实时看到余额变化。我们虽然不用 MQ，但也需要考虑余额变更后如何通知前端（可能通过网关层 WebSocket 推送）。

## 12.2 不应借鉴的反模式

### 反模式 1：Service 内混合两种钱包类型

`addCoin()` 方法内用 `if-else` 区分转账钱包和免转钱包，170+ 行代码耦合在一起。如果我们有类似场景，应该用策略模式或拆分为不同方法。

### 反模式 2：order-service 直接访问 wallet 数据库

破坏微服务数据隔离。任何其他服务需要余额数据，应该通过 RPC 接口获取。

### 反模式 3：流水表缺少幂等索引

coin_record 的 order_no 没有唯一索引。同一笔订单如果重复投递（MQ 重复消费、RPC 重试等），会产生重复流水。我们必须通过 `UNIQUE(biz_type, biz_no)` 保障幂等。

### 反模式 4：锁和事务边界不一致

分布式锁在 Controller 层获取，事务在 Service 层开启。虽然"锁包事务"的方向是对的，但分散在两层会增加维护成本。我们的行锁方案天然与事务绑定，更简洁。

### 反模式 5：枚举硬编码币种

`CurrencyEnum` 把 5 种币种硬编码在代码中。新增币种需要改代码重新发布。我们应该做到数据库驱动，币种配置在 DB 中管理。

### 反模式 6：DTO 既是 Entity 又是传输对象

`UserCoinDTO` 既有 `@TableName("user_coin")` 做 ORM 映射，又在 Service 间传递数据。这导致数据库字段变更会影响到所有上下游。我们的 GORM Gen 天然分离 model 和 query。

# 十三、全链路对比总表

| 维度 | 参考工程（Java） | 我们的设计（Go） | 结论 |
|------|-----------------|-----------------|------|
| **分层** | Controller → Service → Repository（3 层） | handler → ser → ledger → rep（4 层） | 我们多一个 Ledger 层，职责更清晰 |
| **并发** | Redisson 分布式锁（Redis 级） | 数据库行锁（SELECT FOR UPDATE） | 我们更简单，不依赖额外组件 |
| **幂等** | 无（流水表无唯一索引） | 流水表 UNIQUE(biz_type, biz_no) | 我们更安全 |
| **余额类型** | BigDecimal(20,2) | BIGINT（最小货币单位） | 我们无浮点误差 |
| **冻结** | 三阶段（freeze → sub/rollback） | Ledger 层 Freeze/Unfreeze | 思路一致，我们更规范 |
| **非负保障** | SQL 条件 AND (amount-X)>=0 | 可直接采用同模式 | ★ 值得借鉴 |
| **前后快照** | coinFrom / coinTo 记录 | 同样设计 | ★ 值得借鉴 |
| **币种模型** | 单币种/单行 | 多币种/多行 | 我们需求更复杂 |
| **汇率** | USD 基准，Redis 全量缓存 | 可借鉴缓存策略 | ★ 缓存方案可借鉴 |
| **通信** | RabbitMQ（重度） + Feign | 同步 RPC（Kitex） | 架构差异最大 |
| **账变枚举** | coin_type(9种) + coin_detail(8种) 双层 | biz_type + direction 更清晰 | 参考枚举值，简化层级 |
| **事务** | Spring @Transactional | GORM 事务 + Ledger 封装 | 本质一致 |
| **共享库** | p9-core-lib（枚举/VO） | common/（consts/utils） | 模式一致 |
| **定时清理** | XXL-Job 触发清理试玩数据 | 可借鉴 | 运维策略可参考 |

# 十四、从钱包开发者视角的采纳建议

## 14.1 直接采纳（已验证的好模式）

| 编号 | 模式 | 具体做法 |
|------|------|---------|
| A1 | 数据库级非负保障 | `UPDATE SET amount = amount - ? WHERE id = ? AND (amount - ?) >= 0`，通过 affected rows 判断 |
| A2 | 前后余额快照 | 流水表包含 balance_before / balance_after / change_amount |
| A3 | 冻结三阶段 | Freeze → Confirm(Debit) / Cancel(Unfreeze)，与我们 Ledger 设计吻合 |
| A4 | 特定操作跳过非负校验 | 结算回滚、取消类操作允许余额为负（通过参数控制 SQL 条件） |
| A5 | 汇率 Redis 缓存 | 全量列表 + 单币种 key 双层缓存 |
| A6 | 用户信息缓存 | 频繁调用的用户信息走 Redis 缓存减少 RPC |

## 14.2 借鉴思路但需改造

| 编号 | 参考工程做法 | 我们的改造方向 |
|------|------------|--------------|
| B1 | coin_type + coin_detail 双层枚举 | 改为 biz_type（业务标识）+ direction（收/支），更清晰 |
| B2 | CoinRecordVO 入参出参混用 | Thrift IDL 天然 Request/Response 分离 |
| B3 | addCoin 内 if-else 分支两种钱包 | 如有类似需求，用策略模式分离 |
| B4 | 余额变更 MQ 推送 | 考虑通过 RPC 回调或 WebSocket 层通知 |
| B5 | 试玩用户数据定期清理 | 可在后期运维阶段加入 |

## 14.3 明确不采纳

| 编号 | 参考工程做法 | 不采纳原因 |
|------|------------|-----------|
| C1 | Redisson 分布式锁 | 行锁更简单，不引入额外依赖 |
| C2 | 共享数据库（order 直读 user_coin） | 破坏服务边界 |
| C3 | 流水无幂等索引 | 必须有唯一索引保障幂等 |
| C4 | 币种枚举硬编码 | 数据库驱动，支持动态增删 |
| C5 | DTO 即 Entity | GORM Gen 已天然分离 |
| C6 | 免转钱包分支 | 当前需求无此场景 |
| C7 | BigDecimal 金额存储 | 已确定用 BIGINT |
