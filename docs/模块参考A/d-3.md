## 二、数据模型分析

### 2.1 核心表结构（从 DTO + Mapper XML 推演）

#### 表 1：user_coin（用户钱包主表）

| 字段 | 类型（推演） | 说明 |
|------|-------------|------|
| id | BIGINT PK | 主键 |
| user_id | VARCHAR | 用户 ID（唯一，查询主键） |
| user_name | VARCHAR | 用户名 |
| amount | DECIMAL(18,4) | **总余额**（含冻结部分） |
| freeze_amount | DECIMAL(18,4) | 冻结金额 |
| user_state | INT | 用户状态 |
| game_type | INT | 游戏类型 |
| wallet_type | INT | 钱包类型（1=转账, 2=免转） |
| currency | VARCHAR | 币种代码（USD/CNY/TWD/INR/BRL） |
| remark | VARCHAR | 备注 |
| created_time | DATETIME | 创建时间 |
| updated_time | DATETIME | 更新时间 |
| creator | BIGINT | 创建人 |
| updater | BIGINT | 更新人 |

**关键设计解读**：

```
总余额 = amount
冻结额 = freeze_amount
可用余额 = amount - freeze_amount（不落库，查询时计算）
```

> **⚠ 设计细节**：`amount` 字段的语义是"总余额"而非"可用余额"。这在 `UserCoinServiceImpl.getWallet()` 中验证——返回给前端时 `amount = amount - freezeAmount`。但在 `insufficientBalance()` 校验中也是 `balance = amount - freezeAmount`，验证可用余额 >= 扣减额。

#### 表 2：coin_record（帐变记录表）

| 字段 | 类型（推演） | 说明 |
|------|-------------|------|
| id | BIGINT PK | 主键 |
| user_id | VARCHAR | 用户 ID |
| user_name | VARCHAR | 用户名 |
| user_type | INT | 用户类型（0=正式, 1=体验） |
| nick_name | VARCHAR | 昵称 |
| user_belong | INT | 用户归属 |
| merchant_id | BIGINT | 商户 ID |
| merchant_name | VARCHAR | 商户名 |
| merchant_no | VARCHAR | 商户编号 |
| agent_id | BIGINT | 代理 ID |
| agent_no | VARCHAR | 代理编号 |
| agent_name | VARCHAR | 代理名 |
| path | VARCHAR | 层级路径（代理链） |
| order_no | VARCHAR | 关联订单号（**跨服务串联键**） |
| coin_type | INT | 帐变类型（1~9 枚举） |
| coin_detail | INT | 帐变明细（1~8 枚举） |
| coin_value | DECIMAL(18,4) | **变动金额**（始终 > 0） |
| coin_from | DECIMAL(18,4) | 变动前余额 |
| coin_to | DECIMAL(18,4) | 变动后余额 |
| coin_amount | DECIMAL(18,4) | 当前余额（≈ coin_to） |
| balance_type | INT | 收支方向（1=收入, 2=支出） |
| remark | VARCHAR | 备注 |
| created_time | DATETIME | 创建时间 |
| updated_time | DATETIME | 更新时间 |
| creator | BIGINT | 创建人 |
| updater | BIGINT | 更新人 |

**关键设计解读**：

- `coin_from / coin_to`：**变动前后快照**，形成完整审计链。每条记录 = `从 A 变到 B，变了 C`
- `coin_amount ≈ coin_to`：冗余字段，代码中 `setCoinAmount(coinTo)` 直接赋值
- `balance_type` + `coin_type` + `coin_detail`：三层分类（方向 → 大类 → 小类）
- 商户/代理冗余：大量宽表冗余字段，避免跨表 JOIN——典型的**反范式设计**，适合交易流水表
- `user_type = 0` 是硬编码过滤条件：所有 B 端查询 SQL 都带 `WHERE user_type = 0`（排除体验用户）

#### 表 3：add_coin_message（MQ 消息跟踪表）

| 字段 | 类型（推演） | 说明 |
|------|-------------|------|
| id | BIGINT PK | 主键 |
| message_id | VARCHAR | 消息唯一 ID（CorrelationData） |
| order_id_list | TEXT | 关联订单 ID 列表（JSON 数组） |
| status | INT | 投递状态（0=待发, 1=已发, 2=失败） |
| try_count | INT | 重试次数 |
| create_time | DATETIME | 创建时间 |
| update_time | DATETIME | 更新时间 |

### 2.3 与我们 ser-wallet 表设计的对比

| 维度 | Java 参考工程 | 我们 ser-wallet 规划 |
|------|-------------|---------------------|
| 钱包主表 | 1 张 user_coin（单币种/用户） | wallet_account（多币种，一用户多钱包） |
| 帐变记录 | 1 张 coin_record（宽表） | wallet_transaction + wallet_transaction_detail（主从） |
| 冻结记录 | 无独立表（freeze_amount 只在 user_coin 上） | wallet_freeze_record（独立表，支持多笔冻结） |

> **核心差异**：对方是**单钱包模型**（一个用户一行 user_coin），我们是**多钱包模型**（一用户按币种多行）。这是架构层面的根本差异，由需求决定。

## 三、余额操作五式

对方工程将余额变动拆为 5 个独立的 SQL UPDATE，映射到 `UserCoinMapper.xml`。这是钱包系统的**心脏级代码**：

### 3.1 五式一览

```
┌──────────────────────────────────────────────────────────────────┐
│                    user_coin 五种 UPDATE 操作                     │
├─────────────────────┬────────────┬────────────┬─────────────────┤
│ 操作                │ amount     │ freeze     │ 帐变记录        │
├─────────────────────┼────────────┼────────────┼─────────────────┤
│ 1. addBalance       │ +N         │ 不变       │ ✅ 产生         │
│ 2. subBalance       │ -N         │ 不变       │ ✅ 产生         │
│ 3. addFreezeBalance │ 不变       │ +N         │ ❌ 不产生       │
│ 4. subFreezeBalance │ -N         │ -N         │ ✅ 产生         │
│ 5. rollbackFreeze   │ 不变       │ -N         │ ❌ 不产生       │
└─────────────────────┴────────────┴────────────┴─────────────────┘
```

### 3.2 逐式 SQL 拆解

**① addBalance — 加钱（充值/结算/转入）**
```sql
UPDATE user_coin b
SET b.amount = b.amount + #{amount},
    b.updated_time = #{updateTime}
WHERE b.id = #{id}
```
- 无条件加。收入类操作永远不会失败（不存在"加钱加不上"的场景）

**② subBalance — 扣钱（提现/下注/转出）**
```sql
UPDATE user_coin b
SET b.amount = b.amount - #{amount},
    b.updated_time = #{updateTime}
WHERE b.id = #{id}
  -- 结算回滚(6) 和 取消(5) 不校验余额，允许扣为负
  <if test="coinType != 6 and coinType != 5">
    AND (b.amount - #{amount}) >= 0
  </if>
```
- **SQL 级兜底**：`AND (amount - N) >= 0` 防止超扣
- **特例豁免**：`coinType=5`（取消）和 `coinType=6`（结算回滚）允许余额为负——因为这两种场景是"修正"操作，必须执行
- 返回影响行数：`i == 0` → 余额不足，抛异常

**③ addFreezeBalance — 冻结（提现申请时冻结）**
```sql
UPDATE user_coin b
SET b.freeze_amount = b.freeze_amount + #{amount},
    b.updated_time = #{updateTime}
WHERE b.id = #{id}
```
- 只增加冻结额，总额不变
- 效果：可用余额（amount - freeze_amount）减少，但 amount 不动
- **不产生帐变记录**——仅是状态标记，钱没有真正出去

**④ subFreezeBalance — 确认扣减（审核通过出款）**
```sql
UPDATE user_coin b
SET b.amount = b.amount - #{amount},
    b.freeze_amount = b.freeze_amount - #{amount},
    b.updated_time = #{updateTime}
WHERE b.id = #{id}
```
- 同时减少总额和冻结额——钱真正出去了
- **产生帐变记录**——这是"钱动了"
- 效果：可用余额不变（两项同减），但总资产减少

**⑤ rollbackFreezeBalance — 回滚冻结（审核驳回）**
```sql
UPDATE user_coin b
SET b.freeze_amount = b.freeze_amount - #{amount},
    b.updated_time = #{updateTime}
WHERE b.id = #{id}
```
- 只减少冻结额——冻结撤销，钱回到可用
- **不产生帐变记录**——因为钱并没有真正变过

### 3.3 冻结三部曲（完整提现流程示例）

```
初始状态：amount=1000, freeze=0, 可用=1000

步骤1 → freezeCoin(200)
  amount=1000, freeze=200, 可用=800
  （冻结 200，可用减少，不产生帐变）

路径A → 审核通过 → subFreezeBalance(200)
  amount=800,  freeze=0,   可用=800
  （确认出款，总额和冻结都减少，产生帐变）

路径B → 审核驳回 → rollbackFreezeBalance(200)
  amount=1000, freeze=0,   可用=1000
  （撤销冻结，回到初始状态，不产生帐变）
```

### 3.4 借鉴评估

| 设计点 | 评价 | 借鉴建议 |
|--------|------|---------|
| 5 个独立 SQL 操作 | ✅ 清晰语义，职责单一 | **强烈建议采纳**——映射为 Go 的 5 个 Repository 方法 |
| SQL 级余额保护 | ✅ `AND (amount-N)>=0` 防并发超扣 | **强烈建议采纳**——GORM 的 `Where` 条件 |
| 特定类型豁免负余额 | ⚠ 硬编码 coinType=5,6 | **需改进**——用枚举标记 `AllowNegative` 属性 |
| 冻结无帐变 | ✅ 正确语义——锁定不是变动 | **建议采纳** |
| subFreeze 双减 | ✅ 原子性好——一条 SQL 完成 | **建议采纳** |

## 四、核心业务流程

### 4.1 addCoin 全链路（最核心方法，522 行中的灵魂）

```
addCoin(CoinRecordVO)
│
├─ 1. 校验 coinType > 0
│
├─ 2. 获取用户信息 getUserInfoByUserId(userId)
│     └── 带 Redis 缓存（24h TTL）
│
├─ 3. 判断钱包类型
│     │
│     ├─ TRANSFER（类型1：转账钱包 — 直管资金）
│     │   │
│     │   ├─ 3a. 查询 user_coin（SELECT WHERE userId）
│     │   │
│     │   ├─ [存在] ──→ 3b. 余额校验 insufficientBalance()
│     │   │               │   仅 TRANSFER_OUT(2)/BET(3)/SEND_REWARD(8) 校验
│     │   │               │   校验：amount - freeze_amount >= coinValue
│     │   │               │
│     │   │          ──→ 3c. 构建 BalanceVO(id, amount, coinType)
│     │   │          ──→ 3d. 收入? → addBalance(+N)
│     │   │               支出? → subBalance(-N)
│     │   │                       └── subBalance 返回 0 → 抛异常"余额不足"
│     │   │          ──→ 3e. 记录快照 coinFrom=旧amount, coinTo=新amount
│     │   │
│     │   └─ [不存在] ─→ 3f. checkCoinChange() 验证必须是收入
│     │               ─→ 3g. INSERT user_coin（首次充值自动开户）
│     │               ─→ 3h. coinFrom=0, coinTo=充值金额
│     │
│     │   ──→ 3i. INSERT coin_record（帐变记录）
│     │
│     └─ FREE_TRANSFER（类型2：免转钱包 — 资金在三方）
│         │
│         ├─ 4a. 获取订单信息（Feign → OrderResource）
│         ├─ 4b. 获取商户配置（Feign → MerchantResource）
│         ├─ 4c. 构建 ThirdOrderRequestVO
│         ├─ 4d. 调用三方 API（Feign → ApiResource.amountTransfer）
│         │     │
│         │     ├─ SUCCESS → 记录 ThirdOrderRecord + INSERT coin_record
│         │     │            （coinFrom/coinTo = null，三方钱包无快照）
│         │     │
│         │     └─ FAIL → 记录 ThirdOrderRecord(UNDONE) + return false
│         │
│         └─ 4e. INSERT coin_record
│
└─ return true/false
```

### 4.2 余额校验逻辑细节

```java
// insufficientBalance — 只对特定帐变类型校验
private void insufficientBalance(CoinRecordVO vo, UserCoinDTO userCoin) {
    // 仅 转出(2) / 下注(3) / 打赏(8) 需要校验
    if (coinType == TRANSFER_OUT || coinType == BET || coinType == SEND_REWARD) {
        可用余额 = amount - freezeAmount
        if (可用余额 < 扣减额) {
            throw "余额不足"
        }
    }
    // 结算(4)/取消(5)/回滚(6)/重算(7)/闪电(9) — 不校验（允许负）
}
```

**设计思想**：
- 用户**主动发起的操作**（转出/下注/打赏）必须校验——不能花没有的钱
- 系统**被动触发的操作**（结算/回滚/重算）不校验——这些是"修正"，必须执行
- **双层防护**：Service 层校验 + SQL 层 `AND (amount-N)>=0` 兜底

### 4.3 帐变类型枚举体系

```
CoinChangeEnum（一级分类 — 帐变大类）
├── 1. TRANSFER_IN     转入       收入
├── 2. TRANSFER_OUT    转出       支出
├── 3. BET             下注       支出
├── 4. SETTLEMENT      派彩/结算   收入
├── 5. ORDER_CANCEL    取消       收入（退还）
├── 6. SETTLEMENT_ROLLBACK 结算回滚 支出（扣回）
├── 7. SETTLEMENT_SECOND   重算    收支皆可
├── 8. SEND_REWARD     打赏       支出
└── 9. LIGHTNING_BET   闪电投注    支出

CoinChangeDetailEnum（二级分类 — 帐变来源）
├── 1. TRANSFER_IN     会员存款
├── 2. TRANSFER_OUT    会员提款
├── 3. BET             下注
├── 4. SETTLEMENT      结算
├── 5. ORDER_CANCEL    取消
├── 6. SETTLEMENT_CANCEL 结算取消
├── 7. SEND_REWARD     打赏
└── 8. LIGHTNING_BET   闪电投注

BalanceTypeEnum（收支方向）
├── 1. income      收入
└── 2. expenses    支出
```

**三层分类的好处**：`balance_type`（快速过滤收/支）→ `coin_type`（业务大类）→ `coin_detail`（精确来源）

### 4.4 自动开户机制

```
首次充值时，如果 user_coin 不存在：
├── 必须是收入操作（checkCoinChange：支出操作不允许创建钱包）
├── INSERT user_coin:
│   ├── userId, userName
│   ├── amount = 首笔充值金额
│   ├── freezeAmount = 0
│   ├── currency = 用户注册币种（从 UserInfoVO 获取）
│   └── walletType = 用户的钱包类型
└── 同时写帐变：coinFrom=0, coinTo=充值额
```

> **借鉴点**：我们的需求中用户注册时可能需要预创建钱包（多币种场景，注册时选择主币种即创建对应钱包），但"首次充值自动开户"作为**兜底机制**仍值得保留。

## 五、钱包类型双轨制

### 5.1 两种钱包类型

```
WalletTypeEnum:
├── TRANSFER (1)      转账钱包 — 资金在自己平台
└── FREE_TRANSFER (2) 免转钱包 — 资金在三方游戏平台

### 5.2 架构差异

|------|---------------------|--------------------------|
| 资金存储 | 自己数据库 user_coin.amount | 三方游戏平台 |
| 余额变动 | 直接 SQL UPDATE | 调用三方 API（amountTransfer） |
| 余额查询 | SELECT user_coin | 调用三方 API 或返回 0 |
| 帐变记录 | coinFrom/coinTo 有值 | coinFrom/coinTo = null |
| 冻结操作 | 本地 freeze_amount | 不支持（或委托三方） |
| 复杂度 | 中等（纯 DB 操作） | 高（涉及三方调用、重试、状态跟踪） |
| 可靠性 | 高（本地事务保证） | 依赖三方——需额外补偿机制 |

### 5.3 免转钱包处理流程

```
addCoin (FREE_TRANSFER)
│
├── 1. 确定 transferType（根据 coinType 映射）
│      SEND_REWARD → REWARD
│      SETTLEMENT_SECOND / SETTLEMENT_ROLLBACK → RE_CALCULATE_ROUND
│      SETTLEMENT → NORMAL_SETTLEMENT
│
├── 2. 获取订单信息（gameTypeId, playType）
│      打赏特殊处理：gameTypeId=100, playType="GIFT"
│
├── 3. 获取商户配置（addDeductionUrl = 三方加扣款接口地址）
│
├── 4. 构建请求体 ThirdOrderRequestVO:
│      transferNo（唯一流水号）
│      totalAmount（正=加钱, 负=扣钱）

│
├── 5. 调用 apiResource.amountTransfer()
│
├── 6. 判断结果：
│      SUCCESS → 记录到 third_order_record(COMPLETE) + 写帐变
│      FAIL    → 记录到 third_order_record(UNDONE) + return false
│
└── 7. 通过 MQ 通知余额变更

## 六、并发控制策略

### 6.1 Redisson 分布式锁模式

对方所有写操作都使用相同的锁模式：

```java
// 锁粒度：per userId（用户级别）
RLock rLock = redisUtil.getLock(RedisLockKey.getAddCoinLockKey(userId));
try {
    // 等待获取锁 20 秒，持有锁 30 秒自动释放
    if (rLock.tryLock(20000, 30000L, TimeUnit.MILLISECONDS)) {
        // ===== 业务逻辑 =====
        result = coinRecordService.addCoin(coinRecordVO);
        // =====================
    } else {
        log.info("锁超时：{}", orderNo);
    }
} catch (Exception e) {
    log.error(e.getMessage(), e);
} finally {
    // 三重安全检查释放锁
    if (Objects.nonNull(rLock)
        && rLock.isLocked()
        && rLock.isHeldByCurrentThread()) {
        rLock.unlock();
    }
}
```

### 6.2 锁策略分析

| 维度 | 对方实现 | 分析 |
|------|---------|------|
| 锁粒度 | 用户级（userId） | ✅ 合理——不同用户互不影响 |
| 锁 Key | `addCoin_lock_{userId}` | 所有操作共用同一个 Key |
| 等待时间 | 20 秒 | 偏长——说明预期存在较高并发 |
| 持有时间 | 30 秒自动释放 | 防止死锁的兜底 |
| 释放方式 | finally + 三重检查 | ✅ 安全——防重复释放 |
| 锁位置 | Controller 层 | ⚠ 不够好——应在 Service 层 |

### 6.3 双层防护体系

```
第一层：Redisson 分布式锁（用户粒度）
  → 保证同一用户的操作串行化
  → 防止并发读取余额后同时扣减

第二层：SQL WHERE 条件（数据库行级）
  → subBalance: AND (amount - N) >= 0
  → 即使锁失效，SQL 也能阻止超扣
  → 返回 affected_rows = 0 → Service 抛异常
```

### 6.4 Go 平移方案

| Java 方案 | Go 对应方案 |
|-----------|------------|
| Redisson RLock | Redis `SET NX EX`（或 go-redsync） |
| tryLock(20s, 30s) | Lua 脚本加锁 + context.WithTimeout |
| finally unlock | defer unlock |
| SQL AND (amount-N)>=0 | GORM `.Where("amount - ? >= 0", n).Update(...)` |

> **借鉴建议**：双层防护体系（分布式锁 + SQL 兜底）**强烈建议采纳**。但锁应放在 Service 层（domain 层）而非 Controller 层。

### 7.2 四个队列的设计逻辑

| 队列 | 触发场景 | 是否通知余额 | 失败处理 |
|------|---------|-------------|---------|
| Q1 ADD_COIN | 单笔充值/结算/转入 | ✅ 通知 | 异常日志 |
| Q2 ADD_COIN_ORDER | 批量下注扣款 | ✅ 通知 | **回滚订单为初始状态** |
| Q3 NOT_NOTICE | 批量帐变（如重算） | ❌ 不通知 | **回滚订单为初始状态** |
| Q4 RECORD_AND_NOTICE | 写帐变+通知 | ✅ 通知 | 异常日志 |

**Q3 "不通知" 的设计思考**：批量重算时如果每笔都推送余额变更通知，客户端会"跳动"——先扣再加再扣……用户体验差。所以只写帐变记录，等全部完成后再统一通知。

  WallRabbitTemplateConfig.setConfirmCallback
  ├── ACK → 更新 status=1（已投递）

| RabbitMQ Exchange + Queue | NATS JetStream Stream + Consumer |

## 十一、设计优点与不足评估

### 11.1 值得借鉴的设计（✅ 精华）

| # | 设计点 | 详情 | 我们的采纳方式 |
|---|--------|------|--------------|
| 1 | **余额操作五式** | 5 个独立 SQL 方法语义清晰 | 映射为 5 个 Repository 方法 |
| 2 | **SQL 级余额保护** | `AND (amount-N)>=0` 防超扣 | GORM Where 条件 |
| 3 | **双层防护** | 分布式锁 + SQL 兜底 | Redis 锁 + GORM 条件更新 |
| 4 | **coinFrom/coinTo 快照** | 每笔帐变记录变动前后余额 | 完整审计链 |
| 5 | **冻结三部曲** | freeze/subFreeze/rollback 三态 | 直接采用 |
| 6 | **帐变三层分类** | direction + type + detail | 采纳（可能调整枚举值） |
| 7 | **首次充值自动开户** | 兜底机制防止钱进不来 | 作为备选方案 |
| 8 | **MQ 职责分离** | 4 队列区分通知策略和失败处理 | NATS JetStream 不同 Subject |

public void NoticeAddCoinMessage(String jsonStr) {
    CoinRecordVO vo = parseJSON(jsonStr);
    if (vo == null || vo.getCoinValue() == 0) return;

    RLock rLock = redisUtil.getLock(lockKey(vo.getUserId()));
    try {
        if (rLock.tryLock(20000, 30000L, MILLISECONDS)) {

}
