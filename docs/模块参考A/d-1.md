## 二、数据模型分析

### 2.1 核心表：user_coin（用户钱包余额）

```
表名：user_coin
一人一行（转账钱包用户），免转钱包用户不在此表中
```

| 字段 | 类型 | 说明 | 对标我方 |
|------|------|------|---------|
| id | BIGINT AUTO_INCREMENT | 主键 | wallet_account.id |
| user_id | VARCHAR | 玩家 ID（业务主键） | wallet_account.user_id |
| user_name | VARCHAR | 玩家账号 | — |
| **amount** | DECIMAL | **总余额（含冻结）** | wallet_account.available_balance + frozen_balance |
| **freeze_amount** | DECIMAL | **冻结金额** | wallet_account.frozen_balance |
| user_state | INT | 游戏状态 | — |
| game_type | INT | 当前游戏 | — |
| wallet_type | INT | 1=转账 2=免转 | wallet_account.wallet_type |
| currency | VARCHAR | 币种代码 | wallet_account.currency_id |
| remark | VARCHAR | 备注 | — |
| creator / updater | BIGINT | 操作人 | — |
| created_time / updated_time | DATETIME | 审计时间 | — |

**关键设计差异（余额模型）：**

```
参考工程：
  amount = 总余额（含冻结）
  freeze_amount = 冻结部分
  可用余额 = amount - freeze_amount（应用层计算）

我方设计：
  available_balance = 可用余额
  frozen_balance = 冻结金额
  总余额 = available_balance + frozen_balance（应用层计算）
```

> **评估**：两种方式本质等价，但参考工程的"总额含冻结"模式在 SQL 扣款时需要额外的 `AND (amount - #{amount}) >= 0` 防超扣校验。我方的"可用/冻结分离"模式更直观，扣可用余额时直接操作 `available_balance` 字段即可，更清晰。**建议保持我方设计**。

> **评估**：coin_type + coin_detail 双层分类值得借鉴。我方可以考虑在 `transaction_log` 中增加 `tx_detail`（交易明细）字段，作为 `tx_type` 的细化补充。但如果需求不需要这层细粒度，也不必强行引入。

## 三、核心业务流程分析

### 3.1 余额变更（addCoin）——核心中的核心

参考工程的余额变更全部汇聚到一个方法：`CoinRecordServiceImpl.addCoin()`

```
addCoin(CoinRecordVO) 流程：
  │
  ├── ① 校验 coinType > 0
  ├── ② 通过 Feign 获取用户信息（merchantId、walletType、currency 等）
  │
  ├── 分支 A：walletType == TRANSFER（1）转账钱包
  │     │
  │     ├── 查询 user_coin 记录
  │     ├── 如果记录存在：
  │     │     ├── 余额检查（仅对转出/下注/打赏检查：可用 < 变动金额 → 抛异常）
  │     │     ├── 收入(1)：addBalance → UPDATE amount = amount + ?
  │     │     └── 支出(2)：subBalance → UPDATE amount = amount - ?
  │     │           └── SQL 条件：(amount - ?) >= 0（coinType=5/6 除外，允许为负）
  │     │     └── 记录 coinFrom / coinTo
  │     │
  │     ├── 如果记录不存在（新用户）：
  │     │     ├── 校验必须是收入（不能对空钱包扣款）
  │     │     └── INSERT user_coin（amount = coinValue, freezeAmount = 0）
  │     │
  │     └── INSERT coin_record（流水记录）
  │
  └── 分支 B：walletType == FREE_TRANSFER（2）免转钱包
        ├── 调用三方 API 进行余额操作
        ├── 成功：INSERT coin_record（coinFrom/coinTo = null）
        └── 失败：记录为 UNDONE 状态
```

**与我方设计的对比：**

| 维度 | 参考工程 addCoin | 我方 ModifyBalance |
|------|-----------------|-------------------|
| 入口统一性 | ✅ 所有余额变更走一个方法 | ✅ 所有余额变更走 BalanceEngine.ModifyBalance |
| 事务保障 | `@Transactional` | DB 事务 + SELECT FOR UPDATE |
| 并发控制 | Redisson 分布式锁（per-user） | 数据库行锁（per-account） |
| 幂等保障 | 无内置幂等（依赖调用方去重） | 唯一索引（ref_order_no + tx_type + wallet_type） |
| 流水写入 | 在余额变更同一事务中 | 在余额变更同一事务中 |
| 余额快照 | coinFrom / coinTo | balance_before / balance_after |

> **评估**：
> - **分布式锁 vs 行锁**——参考工程用 Redisson 锁（等待 20s，持有 30s），我方用 DB 行锁。行锁更可靠（不会因锁服务故障导致并发问题），但在极高并发下性能不如分布式锁。对于钱包场景，正确性优先于性能，**建议保持我方行锁方案**。
> - **幂等机制**——参考工程无内置幂等，依赖上游去重，这是一个设计缺陷。我方通过唯一索引保障幂等更可靠。**我方设计更优**。

### 3.2 冻结/解冻/消费冻结——三步走

```
冻结生命周期：

① freezeCoin（冻结）
   UPDATE user_coin SET freeze_amount = freeze_amount + ?
   → amount 不变，freeze_amount 增加
   → 不写 coin_record（仅标记，非账变）
   → 可用余额 = amount - freeze_amount（减少）

② subFreezeBalance（消费冻结 = 真正扣款）
   UPDATE user_coin SET amount = amount - ?, freeze_amount = freeze_amount - ?
   → 同时减少 amount 和 freeze_amount
   → 写 coin_record（产生账变记录）
   → 钱从系统中真正扣除

③ rollbackFreezeBalance（回滚冻结 = 释放资金）
   UPDATE user_coin SET freeze_amount = freeze_amount - ?
   → amount 不变，freeze_amount 减少
   → 不写 coin_record（仅释放标记）
   → 可用余额恢复

业务映射：
  下注 → ① freezeCoin
  结算赢 → ③ rollbackFreezeBalance + addBalance（解冻 + 加奖金）
  结算输 → ② subFreezeBalance（消费冻结 = 扣掉输的钱）
  取消注单 → ③ rollbackFreezeBalance（释放冻结 = 钱归还）
```

**与我方设计的对比：**

我方的提现冻结流程：
```
提现申请 → FreezeBalance（available -= X, frozen += X）
审核驳回 → UnfreezeBalance（frozen -= X, available += X）
出款成功 → DebitBalance（frozen -= X，钱离开系统）
```

> **评估**：核心逻辑一致——都是"冻结 → 确认扣款/释放"两阶段模式。参考工程的命名（freezeCoin/subFreezeBalance/rollbackFreezeBalance）不如我方（FreezeBalance/UnfreezeBalance/DebitBalance）直观。**建议保持我方命名**。

### 3.3 充值流程

```
C端用户发起充值
  │
  ├── ① 获取充值方式列表
  │     WebSocket → systemBusinessResource.getPayWayList(type=1, currency)
  │     → 返回可用的支付通道列表（含 icon、限额等）
  │
  ├── ② 获取充值汇率
  │     WebSocket → systemBusinessResource.rechargeRate(currency)
  │     → 返回当前币种的充值汇率
  │
  ├── ③a USDT 充值：
  │     → 检查无待处理订单
  │     → 订单号 = "D" + 16位（OrderUtil.getOrderNo("D")）
  │     → 状态 = Pending(0)，客户端状态 = Pending(0)
  │     → USDT 金额 = amount / exchangeRate（向上取整，保留4位小数）
  │     → 调用 wallet-service 创建订单
  │     → 返回 USDT 地址（TRC20 协议）
  │
  ├── ③b 线下充值（银行卡/支付宝）：
  │     → 调用 wallet-service 创建订单
  │     → 用户上传支付凭证（uploadVoucher）
  │
  ├── ④ 后台审核（两级审核）
  │     待处理(0) → 待一审(4) → 一审审核(5) → 待入款(7) → 入款审核(8) → 成功(1)
  │                              ↘ 一审拒绝(6)
  │
  └── ⑤ 审核通过后入账
        → wallet-service.addCoin({
            coinType: TRANSFER_IN(1),
            coinDetail: 会员存款(1),
            balanceType: income(1),
            coinValue: 充值金额
          })
        → user_coin.amount += 充值金额
        → INSERT coin_record
        → MQ 推送余额变更通知到 WebSocket
```

**可借鉴的点：**

1. **订单号前缀区分业务**——充值订单前缀 "D"，提现订单前缀可能是 "W"，便于从订单号直接识别业务类型
2. **两级审核（一审 + 入款审核）**——参考工程的充值审核有两级，我方的需求中也有类似设计
3. **USDT 充值的地址分配**——为用户生成专属 TRC20 链上地址，通过 `TrxUserWalletRequest` 请求
4. **客户端状态与内部状态分离**——`UserPayOrderStatusEnum`（内部 8 种状态）vs `UserOrderCustomerStatusEnum`（C端仅 3 种：处理中/成功/已取消）

> **评估**：客户端状态与内部状态分离是非常好的设计，我方可以借鉴。内部审核流程复杂（多级审核），但面向用户只展示简化后的状态。

### 3.4 提现流程

```
C端用户发起提现
  │
  ├── ① 获取提现方式列表
  │     WebSocket → systemBusinessResource.getPayWayList(type=2, currency)
  │
  ├── ② 获取提现汇率
  │     WebSocket → systemBusinessResource.withdrawRate(currency)
  │
  ├── ③ 获取提现配置
  │     WebSocket → wallet-service.userWithdrawConfigInfo(currency)
  │     → 返回 UserWithdrawConfigVO：
  │         singleDayWithdrawCount     每日提现次数上限
  │         singleMaxWithdrawAmount    每日提现总额上限
  │         bankSingleWithdrawMinAmount 银行卡单笔最低
  │         bankSingleWithdrawMaxAmount 银行卡单笔最高
  │         virtualSingleWithdrawMinAmount USDT 单笔最低
  │         virtualSingleWithdrawMaxAmount USDT 单笔最高
  │         bigMoneyFlag               大额标记阈值
  │
  ├── ④ 提交提现申请
  │     → 校验：非试玩用户、USD 不能用银行卡
  │     → 校验：余额 >= 提现金额
  │     → 调用 wallet-service.userWithdrawApply(UserWithDrawParamVo)
  │
  └── ⑤ 后台审核 → 出款 → 完成
```

**提现限制追踪（WithdrawLimitAmountParamVO.type）：**

| type | 含义 | 说明 |
|------|------|------|
| 1 | 后台人工加额 | 人工加款的金额需要 X 倍流水才能提现 |
| 2 | 三方充值 | 三方支付充值的金额需要 Y 倍流水 |
| 3 | USDT 充值 | USDT 充值需要 Z 倍流水 |
| 4 | 注单 | 投注产生的有效流水 |

> **评估**：提现流水限制（打码量）是博彩/竞彩行业的核心风控机制。参考工程按资金来源分类管理流水要求，这是我方需求中也会涉及的。值得借鉴。

### 3.5 实时余额推送

```
余额变更后的通知链路：

① 余额变更成功
   ↓
② coinRecordService.sendUserBalance(userId)
   ↓
③ RabbitMQ Fanout Exchange: exchange_notice_user_balance_change
   ↓
④ 所有 WebSocket 服务实例收到消息
   ↓
⑤ 推送余额更新到对应用户的 WebSocket 连接
```

**有趣的设计细节：**
- 存在一个专门的 MQ 队列 `wallet_add_coin_not_notice_balance_queue`，用于"写账变但不通知余额"——避免多步操作中间态导致客户端余额数字跳动
- 这体现了对用户体验的细腻考量：结算过程中余额的中间变化不推送，只在最终态推送

> **评估**：我方暂无 WebSocket 推送需求，但"控制余额通知时机"的思想值得记住——如果后续有实时余额推送，需要考虑中间态的抑制。

### 4.1 账变类型（CoinChangeEnum → coin_type）

## 五、并发与一致性机制

### 5.1 参考工程的并发控制

```
所有余额变更操作的并发控制：

HTTP 入口（WalletController）：
  → Redisson 分布式锁 add.coin.lock.key.{userId}
  → tryLock(wait=20s, lease=30s)
  → @Transactional

MQ 入口（AddCoinListener）：
  → 同样的 Redisson 分布式锁
  → 调用同一个 addCoin() 方法

锁粒度：per-user（同一用户的所有操作串行化）
```

**SQL 层面的防护（作为兜底）：**
```sql
-- subBalance 中的条件更新
UPDATE user_coin SET amount = amount - ?
WHERE id = ? AND (amount - ?) >= 0
-- 影响行数 = 0 → 余额不足 → 抛异常
```

### 5.2 我方的并发控制

```
所有余额变更操作的并发控制：

domain/balance_engine.go → ModifyBalance()：
  → BEGIN 事务
  → SELECT * FROM wallet_account WHERE ... FOR UPDATE（行锁）
  → 应用层校验余额充足
  → UPDATE wallet_account SET ...
  → INSERT transaction_log
  → COMMIT
```

### 5.3 对比分析

| 维度 | 参考工程（Redisson 锁） | 我方（DB 行锁） |
|------|----------------------|----------------|
| 锁粒度 | per-user（所有钱包共用一把锁） | per-account（每个钱包账户独立锁） |
| 锁范围 | 跨 DB 操作（整个方法持有锁） | 仅 DB 事务内 |
| 外部依赖 | 依赖 Redis 可用性 | 仅依赖 DB |
| 最大等待 | 20s | DB 锁等待超时（通常 50s） |
| 最大持有 | 30s | 事务时长 |
| 并发能力 | 单用户串行，跨用户并行 | 单账户串行，跨账户并行 |
| 故障影响 | Redis 故障 → 锁失效 → 并发风险 | DB 故障 → 整个服务不可用（但无并发风险） |

> **评估**：
> - 参考工程用 Redisson 锁的原因是它的 addCoin 涉及 Feign 调用（获取用户信息），不适合放在 DB 事务内（事务时间过长）。
> - 我方没有事务内的远程调用，DB 行锁更合适、更可靠。
> - 参考工程的 SQL 条件更新（`AND (amount - ?) >= 0`）作为兜底，是好的防御性编程，但不应作为主要并发控制手段。
> - **建议保持我方 DB 行锁方案**，同时可以考虑在 SQL 层面增加 `AND available_balance >= ?` 条件更新作为兜底。

## 六、币种与汇率管理

### 6.1 参考工程的币种管理架构

```
币种管理归属：merchant-service
汇率写入链路：
  管理员更新汇率
    → CurrencyRateServiceImpl.updateCurrency()
    → 写入 DB（currency_rate 表）
    → 全量刷新 Redis（3 个 key）
    → 通知用户等级模块刷新（MQ）

  任何服务需要汇率
    → CurrencyBusiness.getCurrencyRate(code)
    → 读 Redis key: findRateByCurrencyCode::{code}
    → 命中返回，未命中直接抛异常（不查 DB，强依赖缓存）
```

### 6.2 Redis 缓存策略

| Redis Key | 内容 | TTL | 写入时机 |
|-----------|------|-----|---------|
| `merchant-service::merchantCurrency` | 全量汇率列表 JSON | **永久（无 TTL）** | 启动 + 管理员更新时 |
| `findRateByCurrencyCode::{code}` | 单币种汇率值 | **永久** | 同上 |
| `merchantRateCurrencyCode::{code}` | 单币种完整汇率对象 | **永久** | 同上 |
| `all-service::currency` | 启用币种列表 | **永久** | 启动 + 管理员更新时 |

### 6.3 汇率转换逻辑

```java
// CurrencyBusiness.toUSDAmount()
// 任意币种转 USD：usdAmount = amount / currencyRate
// 精度：scale=2, RoundingMode.HALF_UP
```

### 6.4 对比与借鉴

| 维度 | 参考工程 | 我方设计 |
|------|---------|---------|
| 汇率基准 | 固定 USD（1 USD = X 外币） | 灵活币种对（from → to） |
| 汇率表设计 | 一行一个币种 | 一行一对币种 + 生效时间 |
| 缓存策略 | 永久缓存 + 手动刷新 + Miss 报错 | Cache-Aside + TTL + 定时刷新 |
| 精度 | BigDecimal | bigint ×10^8 |
| 历史汇率 | 不保留（覆盖更新） | exchange_rate_log 保留历史 |

> **评估**：
> 1. **"Miss 直接报错"的强缓存依赖**——参考工程在 Redis 缓存不存在时直接抛异常，不降级查 DB。这是"快速失败"设计，适合汇率这种不能容忍不一致的场景。值得借鉴。
> 2. **全量覆盖刷新**——更新汇率时全量从 DB 读出覆盖 Redis，简单粗暴但有效。
> 3. **永久缓存无 TTL**——参考工程的做法比较激进，我方用 TTL + 定时刷新更稳妥。

## 八、已发现的设计缺陷（前车之鉴）

通过深度分析参考工程代码，发现以下设计缺陷，**我方应避免**：

### 8.1 事务一致性问题

**缺陷：** `subFreezeBalance()` 方法没有 `@Transactional` 注解，但内部执行了两步操作（更新 user_coin + 插入 coin_record）。如果第二步失败，第一步不会回滚。

**我方防范：** ModifyBalance 的所有操作在同一个 DB 事务内完成。

### 8.2 幂等性缺失

**缺陷：** `addCoin()` 方法没有内置幂等机制。如果同一请求被重试（网络超时后重发），会导致重复入账。虽然有 Redisson 锁，但锁释放后重试仍会进入。

**我方防范：** 唯一索引（ref_order_no + tx_type + wallet_type）自动拦截重复请求。

### 8.3 余额通知不一致

**缺陷：** `addCoin()` 在 Controller 层调用 `sendUserBalance()`，但 `addCoinList()` 不调用。批量操作后客户端余额不会实时更新。

**我方防范：** 如果引入余额推送，应在 ModifyBalance 内部统一触发，而非在调用方决定。

### 8.4 缓存写入遗漏

**缺陷：** `getUserInfoByUserId()` 从 Redis 读取用户信息，但 cache miss 后通过 Feign 获取到数据后**没有写回 Redis**（代码中缺少 `redisUtil.set()`），导致每次都穿透到 user-service。

**我方防范：** 所有 Cache-Aside 模式必须确保"读 miss → 查源 → 回写缓存"完整闭环。

### 8.5 SQL 注入风险

**缺陷：** `CoinRecordMapper.xml` 中排序方向使用 `${vo.createdTimeBy}`（${}不转义），如果调用方未校验输入，存在 SQL 注入风险。

**我方防范：** GORM Gen 的类型安全查询 DSL 天然避免此问题。

### 8.6 清理任务失效

**缺陷：** `BackWaterDayTask.deleteAddCoinMessage()` 中删除操作被注释掉了（`//addCoinMessageRepository.delete()`），导致 `add_coin_message` 表永远不会被清理。

**我方防范：** 定时任务的功能完整性需要有测试覆盖。

## 九、可借鉴的设计模式总汇

### 9.1 推荐采纳（与我方设计互补的好思路）

| 编号 | 参考设计 | 可借鉴之处 | 采纳建议 |
|------|---------|-----------|---------|
| A1 | **coin_type + coin_detail 双层分类** | 流水的大类+明细分离，查询和统计更灵活 | 可选。如需求需要细粒度分类，在 transaction_log 增加 `tx_detail` 字段 |
| A2 | **内部状态 vs 客户端状态分离** | 充值/提现的内部审核状态复杂，但面向用户只展示简化状态 | 推荐。在充值/提现接口返回时做状态映射 |
| A3 | **订单号前缀区分业务** | "D" = 充值，"W" = 提现，从号码直接识别业务 | 推荐。如 D+时间戳、W+时间戳、E+时间戳（兑换） |
| A4 | **SQL 条件更新作为兜底** | `UPDATE ... WHERE amount - ? >= 0` 防止余额为负 | 推荐。在 rep 层的 UpdateBalance 中增加条件校验作为二级防护 |
| A5 | **流水冗余存储用户/商户信息** | coin_record 冗余了 merchant/agent 信息，避免 JOIN | 可选。看查询频率，如果 B 端账变查询需要展示用户/币种名称，可在流水中冗余 |
| A6 | **汇率缓存全量列表 + 单码分别缓存** | 批量查和单查各有专属缓存 key | 推荐。我方 exchange_rate 缓存可以同时维护列表 key 和单码 key |
| A7 | **提现流水限制分类管理** | 按资金来源（充值/活动/投注）设定不同的流水倍数要求 | 值得参考。如果我方有打码量需求，可参考此分类设计 |
| A8 | **余额通知时机控制** | 多步操作中抑制中间态的余额推送 | 记住。后续如有实时余额推送需求时参考 |

### 9.2 不推荐采纳（参考工程的设计不如我方的）

| 编号 | 参考设计 | 不推荐原因 | 我方更优设计 |
|------|---------|-----------|------------|
| B1 | Redisson 分布式锁 | 依赖外部服务，锁失效有并发风险 | DB 行锁（SELECT FOR UPDATE）更可靠 |
| B2 | BigDecimal 余额精度 | 浮点运算需要处处注意精度 | bigint 最小单位整数，天然避免精度问题 |
| B3 | 无内置幂等 | 重试会重复入账 | 唯一索引幂等更可靠 |
| B4 | 错误码全局混排 | 维护困难 | 分段错误码（6022xxx）更清晰 |
| B5 | 币种硬编码枚举 + DB 双轨 | 新增币种需改代码+上线 | 纯 DB 配置更灵活 |
| B6 | 单库多服务共享 | 耦合度高，扩展性差 | 每服务独立库 |
| B7 | 余额模型（总额含冻结） | 扣款 SQL 需额外条件，不直观 | 可用/冻结分离更清晰 |
| B8 | 永久缓存无 TTL | 如果刷新机制失败，缓存永远是旧值 | TTL + 主动刷新 + 定时刷新三重保障 |

> **评估**：参考工程的 API 更粗粒度（addCoin 同时处理充值/提现/下注/派彩），我方的 API 按业务语义细分（CreditBalance/DebitBalance/FreezeBalance/UnfreezeBalance），更清晰，更容易理解和维护。**我方设计更优**。

## 十二、总结——"照镜子"的核心发现

### 12.1 参考工程做对的事情

1. **余额变更入口统一**——无论充值、提现、下注、派彩，全部走同一个 addCoin 方法，保证逻辑一致性
2. **流水表只写不改**——coin_record 是 append-only 的不可变日志，保证审计可追溯性
3. **冻结三步走设计清晰**——冻结/消费冻结/回滚冻结分离，状态转换明确
4. **余额快照冗余存储**——每条流水都记录 coinFrom/coinTo，可以事后验证链路连续性
5. **SQL 条件更新作为兜底**——即使应用层漏了校验，SQL 层也能防止余额为负

### 12.2 参考工程的不足（我方应避免）

1. **无内置幂等**——依赖调用方去重，而非系统自身保障
2. **分布式锁依赖外部服务**——Redisson 锁如果 Redis 故障会导致并发问题
3. **事务注解遗漏**——subFreezeBalance 缺少 @Transactional
4. **缓存一致性问题**——用户信息缓存读但不写、清理任务被注释
5. **错误码混乱**——无分段规则，难以维护
6. **充值/提现源码缺失**——关键业务代码未提交到仓库

### 12.3 对我方设计的信心确认

通过与生产运行的参考工程对比，确认我方的以下设计决策是正确的：

| 我方设计 | 参考工程对比 | 结论 |
|---------|-----------|------|
| DB 行锁 | Redisson 分布式锁 | ✅ 我方更可靠 |
| bigint 最小单位 | BigDecimal | ✅ 我方更安全（无精度问题） |
| 唯一索引幂等 | 无内置幂等 | ✅ 我方更完善 |
| 分段错误码 | 全局混排 | ✅ 我方更清晰 |
| 可用/冻结分离 | 总额含冻结 | ✅ 我方更直观 |
| 每服务独立库 | 单库共享 | ✅ 我方更解耦 |
| 纯 DB 币种配置 | 硬编码 + DB 双轨 | ✅ 我方更灵活 |
| 币种对汇率 | 单基准 USD | ✅ 我方更通用 |
| balance_engine 统一事务 | addCoin 但有事务遗漏 | ✅ 我方更严谨 |

### 12.4 从参考工程获取的新启发

1. **考虑增加 `tx_detail` 字段**——如果后续需要更细粒度的流水分类
2. **内部状态 vs 客户端状态映射**——充值/提现的状态展示应该做层级转换
3. **订单号业务前缀**——D=充值, W=提现, E=兑换, A=调账
4. **SQL 条件更新兜底**——在 rep 层增加 `AND available_balance >= ?` 防护
5. **提现流水限制按资金来源分类**——不同来源的充值金额有不同的提现门槛
6. **汇率缓存双级设计**——全量列表 key + 单码 key 并存

> 以上发现均已记录，在后续编码阶段可按需采纳。参考工程的核心价值不在于"怎么写代码"，而在于"哪些业务场景容易出问题"——幂等、并发、事务一致性、缓存一致性——这些坑我方通过更严谨的设计已经提前规避。
