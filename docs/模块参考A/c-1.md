**关键设计观察**：

| 观察点 | 参考工程做法 | 评估 |

**与我们设计的对比**：

```
参考工程 user_coin:
    一用户一行，amount + freezeAmount 两列

我们 wallet 表:
    一用户 × 一币种 × 一钱包类型 = 一行
    available_balance + frozen_balance 两列（思路一致，粒度更细）
```

## 4. 核心业务流程拆解

### 4.1 加币流程（addCoin）— 最核心方法

```
addCoin(CoinRecordVO)
    │
    ├── 1. 校验 coinType 合法性（> 0）
    │
    ├── 2. 查用户信息（Feign → user-service）
    │      → 获取 walletType, currency, merchantId 等
    │
    ├── 3. 判断钱包类型
    │
    ├──── [转账钱包] ───────────────────────────────────────
    │      │
    │      ├── 查 user_coin 表获取当前余额
    │      │
    │      ├── 如果用户不存在钱包记录：
    │      │   ├── 校验不能是支出操作
    │      │   └── INSERT user_coin（首次入账创建钱包）
    │      │
    │      ├── 如果用户已有钱包：
    │      │   ├── 支出类操作（转出/下注/打赏）→ 余额充足性校验
    │      │   │   └── (amount - freezeAmount) >= coinValue
    │      │   │
    │      │   ├── 收入 → UPDATE user_coin SET amount = amount + coinValue
    │      │   └── 支出 → UPDATE user_coin SET amount = amount - coinValue
    │      │            └── SQL 层双重校验：(amount - coinValue) >= 0
    │      │               例外：结算回滚(6)和取消(5)允许负数
    │      │
    │      ├── 计算 coinFrom / coinTo
    │      └── INSERT coin_record（账变记录）
    │
    └──── [免转钱包] ───────────────────────────────────────
           │
           ├── 调 order-service 查订单详情
           ├── 调 merchant-service 查商户配置
           ├── 封装三方请求参数
           ├── 调 api-service → 三方平台转账
           ├── 成功 → 保存三方订单记录 + 通知余额变更
           ├── 失败 → 保存失败记录 + 返回 false
           └── INSERT coin_record（账变记录）
```

**我们可以借鉴的核心逻辑**：

| 步骤 | 参考做法 | 我们采纳/改进 |
|------|---------|------------|
| 余额校验 | 应用层 `(amount - freezeAmount) >= coinValue` | **采纳** + 在 SQL 层再加 CAS 条件 |
| SQL 扣款安全 | `WHERE (amount - #{amount}) >= 0`，结算回滚和取消除外 | **采纳思路** — 我们用 `WHERE available_balance >= amount` |
| 首次入账创建钱包 | 用户没有记录时 INSERT | **采纳** — 用户注册时初始化，或首次充值时 upsert |
| coinFrom/coinTo | 每条流水记录前后余额 | **采纳** — 对账追溯的关键 |
| 双重校验 | 应用层 + SQL 层都校验余额 | **采纳** — 防并发场景下应用层校验通过但 SQL 实际不足 |

### 4.2 冻结机制（freeze/sub/rollback）

```
冻结 freezeCoin(userId, freezeAmount)
    ├── 查 user_coin
    ├── 校验 amount >= freezeAmount
    └── UPDATE: freeze_amount = freeze_amount + freezeAmount
        ※ 注意：amount 不变，仅 freeze_amount 增加
        ※ 无账变记录

解冻确认扣款 subFreezeBalance(userId, freezeAmount)
    ├── 查 user_coin
    ├── 校验 freeze_amount >= freezeAmount
    └── UPDATE: amount = amount - freezeAmount
               freeze_amount = freeze_amount - freezeAmount
        ※ amount 和 freeze_amount 同时减少
        ※ 生成账变记录（createOrder 方法）

回滚冻结 rollbackFreezeBalance(userId, freezeAmount)
    ├── 查 user_coin
    └── UPDATE: freeze_amount = freeze_amount - freezeAmount
        ※ 仅 freeze_amount 减少，amount 不变
        ※ 无账变记录
```

**冻结三步骤的数据变化**：

```
初始状态：amount=1000, freeze=0, 可用=1000

step1 冻结 200：amount=1000, freeze=200, 可用=800
    → 应用层可用 = amount - freeze = 800

step2a 确认扣款 200：amount=800, freeze=0, 可用=800
    → amount 和 freeze 同时减 200
    → 产生账变（coinTo = 800）

step2b 回滚冻结 200：amount=1000, freeze=0, 可用=1000
    → 仅 freeze 减 200，amount 不变
    → 不产生账变
```

**与我们设计的对比**：

| 维度 | 参考工程 | 我们的设计 | 评估 |
|------|---------|----------|------|
| 冻结时 | `freeze += x`，amount 不动 | 同思路：`frozen += x, available -= x` | 我们更好：应用层可用余额直接用 available |
| 确认扣款 | `amount -= x, freeze -= x` | `frozen -= x`（available 不变，已在冻结时扣过） | **差异点**：我们冻结时就扣 available |
| 回滚 | `freeze -= x`，amount 不动 | `frozen -= x, available += x` | 对称回滚 |
| 可用余额 | 需计算 `amount - freeze` | 直接读 `available_balance` | 我们更简洁 |

**设计差异本质**：
- 参考工程：amount = 总余额（含冻结），可用 = amount - freeze（需计算）
- 我们：available = 可用余额（已扣冻结），frozen 单独记录（直接读取）
- **我们的设计更优**：读取时无需计算，减少出错可能

## 5. 余额安全机制分析

### 5.1 分布式锁

**参考工程实现**（WalletController.java）：

```java
// 每个加币操作都加 Redis 分布式锁
RLock lock = redissonClient.getLock(RedisLockKey.getAddCoinLockKey(userId));
try {
    lock.tryLock(20, 30, TimeUnit.SECONDS);  // 等待 20s，持有 30s
    // ... 业务操作 ...
} finally {
    lock.unlock();
}
```

**关键参数**：
- 锁粒度：**用户级别**（每个 userId 一把锁）
- 等待时间：20 秒
- 持有时间：30 秒
- 锁实现：Redisson（支持可重入、看门狗续期）

**与我们的对比**：

| 维度 | 参考工程 | 我们的方案 |
|------|---------|----------|
| 锁粒度 | userId | userId + currency + walletType（更细） |
| 锁工具 | Redisson | go-redis SetNX / RedSync |
| 余额安全 | 锁 + SQL WHERE 条件 | CAS 乐观锁 `WHERE available >= amount` |
| 事务 | @Transactional | GORM 事务 |

**评估**：参考工程用**悲观锁**（Redis Lock），我们用**乐观锁**（CAS WHERE 条件）。乐观锁在低冲突场景性能更好，但高冲突时可能重试多次。对于钱包场景（一个用户不太可能同时发起多笔操作），**CAS 乐观锁足够且更高效**。

### 5.2 SQL 层安全

**参考工程 SQL**（UserCoinMapper.xml）：

```sql
-- 扣款（带安全条件）
UPDATE user_coin
SET amount = amount - #{amount}, updated_time = #{updateTime}
WHERE id = #{id}
  AND (amount - #{amount}) >= 0   ← 仅转出/下注/打赏时启用
```

- 结算回滚(coinType=6)和取消(coinType=5)**不校验余额**：因为这两种情况允许余额变为负数
- 这是合理的：取消/回滚是系统行为，不应被余额不足阻挡

**我们可以采纳**：
- 普通扣款：`WHERE available_balance >= amount`（严格校验）
- 系统行为（回滚/取消/人工减款）：不加余额校验条件（允许负数场景由业务决策）

### 5.3 事务边界

**参考工程做法**：
- `addCoin()`：`@Transactional(rollbackFor = Exception.class)` — 余额更新 + 流水记录在同一事务
- `addCoinBatch()`：外层 `@Transactional` 循环调 `addCoin()`，任何一笔失败全部回滚
- `freezeCoin()`：单独事务

**与我们一致**：余额变更和流水记录必须在同一个数据库事务内。

## 6. 跨服务交互模式

### 6.1 同步调用（Feign → 对标我们的 Kitex RPC）

| 调用方向 | 接口 | 用途 | 我们的对应 |
|---------|------|------|----------|
| wallet → user-service | `getUserByUserId(userId)` | 获取用户信息（币种、商户等） | 钱包 → ser-user（RPC 查用户） |
| wallet → order-service | `findOrderByOrderNo(orderNo)` | 查订单详情 | 不需要（我们的订单在钱包内） |
| wallet → order-service | `updateOrdersToInitialStatus(orderNos)` | 扣币失败回滚订单状态 | 不需要 |
| wallet → merchant-service | `getMerchantDetail(merchantId)` | 查商户配置（缓存 1 天） | 不需要（我们无商户体系） |
| wallet → api-service | `amountTransfer(request)` | 三方游戏转账 | 不需要（游戏特有） |

### 6.2 异步消息（RabbitMQ → 对标我们的 NATS/Kafka）

**参考工程的 MQ 架构**：

```
                     ┌── Q1: ADD_COIN_QUEUE
                     │   → 单笔加币 + 推送余额
                     │
ORDER_ADD_COIN   ────┤── Q2: ADD_COIN_ORDER_QUEUE
(Exchange)           │   → 批量扣币（下注）+ 推送余额
                     │   → 失败回滚订单状态
                     │
                     ├── Q3: ADD_COIN_NOT_NOTICE_BALANCE_QUEUE
                     │   → 批量扣币 + 不推送余额
                     │
                     └── Q4: ADD_COIN_RECORD_AND_NOTICE_BALANCE_QUEUE
                         → 仅记录账变 + 推送余额

EXCHANGE_NOTICE_USER_BALANCE_CHANGE
    → 余额变化通知（推送给 WebSocket → 前端实时更新）
```

**设计特点**：
1. **按场景分队列** — 不同场景走不同队列（加币/扣币/仅记录/推余额/不推余额），消费逻辑分离
2. **MQ 消息持久化** — add_coin_message 表记录每条消息的状态和重试次数
3. **ConfirmCallback** — 发送端确认机制，ack/nack 更新消息状态
4. **余额推送** — 余额变化后通过另一个 Exchange 通知 WebSocket 推送给前端

**我们可以借鉴的**：
- **余额变化实时通知**：虽然我们的 C 端可能不需要 WebSocket 推送，但可以考虑充值到账后通过推送告知用户
- **按场景分队列**：不同操作的消费逻辑隔离，避免互相干扰

## 8. 可借鉴的设计思想

### 8.1 高价值借鉴（直接采纳）

| 编号 | 设计思想 | 参考工程实现 | 我们如何采纳 |
|------|---------|-----------|-----------|
| **B1** | **账变前后余额记录** | coinFrom / coinTo 两个字段 | 在 transaction_log 中增加 `balance_before` + `balance_after` 字段 |
| **B2** | **收支类型标记** | balanceType(1收入/2支出) | 在 transaction_log 中增加 `balance_type` 字段，简化查询筛选 |
| **B3** | **SQL 层双重余额校验** | `WHERE (amount - #{amount}) >= 0` | 我们的 CAS 已有 `WHERE available_balance >= amount` |
| **B4** | **特殊类型跳过余额校验** | 结算回滚(6)和取消(5)不检查余额 | 人工减款、系统冲正等特殊类型不加余额条件 |
| **B5** | **分布式锁保护余额操作** | Redis Lock per userId | 关键操作（提现冻结、人工加减款）可加 Redis 锁 |
| **B6** | **首次入账自动创建钱包** | user_coin 不存在时 INSERT | 用户注册/首次充值时自动创建钱包记录 |
| **B7** | **余额变化通知** | MQ 通知 → WebSocket 推送 | 充值到账后通知用户（APP 推送 / WebSocket） |

### 8.2 中等价值借鉴（评估后选择性采纳）

| 编号 | 设计思想 | 参考工程实现 | 评估 |
|------|---------|-----------|------|
| **M1** | 账变二级分类 | coinType(大类) + coinDetail(来源) | 我们的 log_type 已足够，如需求变复杂可以加 |
| **M2** | MQ 消息持久化 | add_coin_message 表 | 取决于消息中间件的可靠性，NATS/Kafka 可能不需要额外表 |
| **M3** | MQ 按场景分队列 | 4 个不同队列处理不同场景 | 如果异步链路复杂可以采纳 |
| **M4** | 用户信息冗余到流水 | 每条 coin_record 冗余 userName/nickName/merchantId... | 过度冗余，但减少 JOIN。我们评估：只冗余 userId 即可 |
| **M5** | 商户信息缓存 | Redis 缓存 1 天 | 我们的币种配置缓存可以参考这个缓存时间策略 |

### 8.3 参考但不直接采纳的思路

| 编号 | 设计思想 | 为什么不采纳 |
|------|---------|-----------|
| N1 | 一用户一行余额 | 我们需要多币种 × 多钱包类型，一用户多行 |
| N2 | 转账/免转钱包分支 | 游戏业务特有（三方钱包），我们全是本地钱包 |
| N3 | MongoDB 存储 | 游戏流水量极大适合 MongoDB，我们用 MySQL/TiDB 即可 |
| N4 | 商户/代理字段 | 游戏代理体系特有，我们无此需求 |

## 9. 需要改进或不采纳的部分

### 9.1 参考工程的设计不足之处

| 问题 | 参考工程现状 | 我们如何避免 |
|------|-----------|-----------|
| **无幂等唯一索引** | coin_record 表的 orderNo 没有唯一索引，重复调用可能插入重复流水 | 我们的 transaction_log 必须有 `(order_id, log_type)` 唯一索引 |
| **可用余额需计算** | 可用 = amount - freezeAmount，每次查询都要算 | 我们直接用 `available_balance` 字段，读取更高效 |
| **冻结无流水** | 冻结操作不记录 coin_record | 我们可以考虑冻结也记录流水（方便审计） |
| **无币种隔离** | 一个用户只有一条 user_coin 记录，currency 字段在行内但不作为查询条件 | 我们按 `(user_id, currency_code, wallet_type)` 三维度拆分 |
| **DTO 命名混淆** | DTO 既当数据库实体又当接口传输对象 | 我们分层清晰：model(DB) → rep(查询) → ser(业务) → IDL(接口) |
| **无事务一致性保障** | 冻结操作(`freezeCoin`)中先查后更新，无悲观锁也无 CAS | 我们的 CAS `WHERE available >= amount` 在更新时原子校验 |
| **无操作审计** | 缺少操作人、操作时间、操作 IP 等审计字段 | 我们每张表有完整的 create_by/update_by 审计字段 |

### 9.2 参考工程与我们业务场景的差异

| 维度 | 参考工程（游戏平台） | 我们（竞彩+泛娱乐） |
|------|-------------------|-------------------|
| 核心场景 | 下注 → 结算 → 派彩 | 充值 → 消费 → 提现 → 兑换 |
| 账变频率 | 极高（每局游戏都有） | 中等（充值/提现/兑换） |
| 余额变动方向 | 双向高频（下注扣、派彩加） | 单向为主（充值进、提现出） |
| 多币种 | 有（5种法币）但用户只用一种 | 有（5种）且用户可能持有多种 |
| 钱包类型 | 2 种（转账/免转） | 5 种（中心/奖励/冻结/法币/虚拟币） |
| 审核流程 | 二级（一审/二审） | 三级（风控/财务/出款） |
| 兑换功能 | 无 | 有（法币 ↔ BSB） |
| 充值通道管理 | 在 pay-service（独立） | 在 ser-finance（同事模块） |

## 11. 对我们 ser-wallet 的具体启发

### 11.1 transaction_log 表改进建议

基于参考工程的 coinFrom/coinTo 和 balanceType 设计，建议在我们的 transaction_log 表中增加：

| 新增字段 | 类型 | 说明 | 参考来源 |
|---------|------|------|---------|
| `balance_before` | DECIMAL(20,4) | 变动前余额 | 参考工程 coinFrom |
| `balance_after` | DECIMAL(20,4) | 变动后余额 | 参考工程 coinTo |
| `balance_type` | TINYINT | 收支类型（1收入/2支出） | 参考工程 balanceType |

**好处**：
- `balance_before` / `balance_after`：出问题时可逐条对账，不需要从头累计
- `balance_type`：C 端"只看收入/只看支出"筛选直接用索引，不需要 `IN (type1, type2, ...)` 组合条件

### 11.2 余额校验策略矩阵

基于参考工程"不同 coinType 不同校验策略"的思路：

| 操作类型 | 是否校验余额充足 | 说明 |
|---------|--------------|------|
| 充值入账 | 否 | 只加不减 |
| 提现冻结 | **是** | `available >= amount` |
| 提现确认扣款 | **是** | `frozen >= amount` |
| 提现回滚 | 否 | 解冻退回，只增不减 |
| 兑换 | **是** | 法币扣减需校验 |
| 奖励入账 | 否 | 只加不减 |
| 人工加款 | 否 | 只加不减 |
| 人工减款 | **可选** | 系统行为，可允许负数（运营决策） |
| 系统冲正 | 否 | 参考工程的 SETTLEMENT_ROLLBACK 思路 |

### 11.3 首次入账自动建钱包

参考工程在 `addCoin()` 中检测到用户无钱包记录时自动 INSERT。我们的策略：

```
方案A（推荐）：用户注册时批量创建所有币种的钱包记录
    → 优点：后续操作不需要判断是否存在
    → 缺点：预创建可能浪费空间

方案B：首次充值时 upsert
    → 优点：按需创建
    → 缺点：每次操作都要判断
```

### 11.4 冻结是否记录流水

参考工程冻结**不记录流水**，仅确认扣款和回滚时记录。评估：

| 方案 | 优点 | 缺点 |
|------|------|------|
| 冻结不记流水（参考工程做法） | 流水少，简单 | 无法追溯"什么时候冻结的" |
| 冻结也记流水 | 完整审计链 | 流水量翻倍 |
| **折中：冻结记日志表，不记交易流水** | 兼顾审计和性能 | 需要额外表 |

建议：提现冻结**记录到 transaction_log**（log_type = freeze），因为涉及资金安全，审计需要完整链路。
