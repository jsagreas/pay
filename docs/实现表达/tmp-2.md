# 一、需求全貌 — 这个模块到底做什么

## 1.1 一句话定位

给一个面向东南亚的泛娱乐+合规竞彩平台，建一套**支持多种货币的电子钱包系统**。用户在平台上打赏、买道具、投注、提现，所有"钱进钱出"都通过这个钱包。

## 1.2 涉及三个模块，我负责两个

| 模块 | 谁负责 | 一句话定位 |
|------|--------|----------|
| **多币种钱包** | **我** | 用户直接使用的钱包（充值/提现/兑换/查记录），C端面向App/H5用户 |
| **币种配置** | **我** | 钱包的"基础数据源"— 币种/汇率/精度规则，B端面向运营后台 |
| 财务管理 | 他方 | 支付通道/审核/人工修正/统计，B端。**我需要理解它以便联调** |

三者关系：
- 币种配置是**地基** — 告诉系统"有哪些币种、汇率怎么算"
- 钱包是**房子** — 用户在上面充值、消费、提现
- 财务管理是**物业** — 运营人员在后台审核、调账

## 1.3 我从需求中读到的核心业务规则

我完整看了 152 张产品原型和需求文档，总结出以下核心规则，后续的技术方案都围绕这些规则展开：

### 钱包结构 — 按币种隔离，按类型分层

```
用户的钱包不是一个扁平的余额，而是一个二维结构：

                 BSB        VND        IDR        THB       USDT
  中心钱包     100.00    500,000    2,000,000   3,000      50.00
  奖励钱包      20.00    100,000      500,000   1,000       0.00
  主播钱包       0.00          0            0       0        0.00
  代理钱包     (预留)     (预留)       (预留)    (预留)     (预留)
  场馆钱包     (预留)     (预留)       (预留)    (预留)     (预留)

说明：
- 每行是一种子钱包类型，每列是一种币种
- 各自独立，互不影响
- 用户看到的"资金余额" = 中心 + 奖励 合计
- 消费时先扣中心，不够再扣奖励
```

### 四大业务流程

| 流程 | 复杂度 | 关键特征 |
|------|--------|---------|
| **充值** | 中 | 跨模块（需财务匹配通道），有USDT链上特殊路径 |
| **提现** | **高** | 跨 KYC/风控/财务三级审核 + 冻结/解冻/扣除三步余额操作 |
| **兑换** | 低 | 纯内部操作（法币→BSB单向），不依赖外部模块 |
| **记录查询** | 低 | 纯读操作，4种Tab分类查询 |

### 提现 — 最复杂的流程

提现之所以复杂，是因为它涉及**多级审核 + 余额冻结机制**：

```
用户发起提现
    ↓
校验（KYC + 限额 + 姓名一致 + 充值路由规则）
    ↓
★ 立即冻结余额（可用余额 → 冻结余额，总额不变）
    ↓
创建提现订单（T+16位） → 状态=待风控审核
    ↓
风控审核 ──┬── 通过 → 待财务审核
           └── 驳回 → ★ 解冻（冻结 → 可用，钱退回）
                       ↓
财务审核 ──┬── 通过 → 待出款
           └── 驳回 → ★ 解冻
                       ↓
通道出款 ──┬── 成功 → ★ 扣除冻结（资金真正离开钱包）
           └── 失败 → 保持冻结，人工处理
```

**关键设计**：提现全程不直接扣余额，走"冻结→审核→扣除/解冻"三步模式。这是资金安全的核心保障 — 在审核期间资金被冻结，用户无法二次使用。

### 充值路由规则 — "怎么充的怎么提"

这是反洗钱合规要求：

| 充值方式 | 允许提现方式 | 逻辑 |
|---------|------------|------|
| 只充法币 | 只能法币提 | 原路返回 |
| 只充USDT | 只能USDT提 | 原路返回 |
| 法币+USDT都充过 | 法币提 | 法币优先 |
| 法币→兑换BSB | 追溯法币提 | 追溯原始 |

### 汇率机制 — 不是固定值，有阈值触发逻辑

```
定时任务拉取 3 个三方 API → 取平均值 → 得到实时汇率
    ↓
计算偏差 = |实时 - 平台| / 实时 × 100%
    ↓
偏差 < 阈值 → 只更新实时汇率，平台汇率不动
偏差 ≥ 阈值 → 更新平台汇率 + 联动入款汇率/出款汇率 + 生成日志

入款汇率 = 平台汇率 × (1 + 浮动%)   ← 充值用
出款汇率 = 平台汇率 × (1 - 浮动%)   ← 提现用
```

这意味着用户充值和提现看到的汇率是不同的（有差价），这是平台的汇率盈利模型。

## 3.3 账户模型选型

业界有两种主流做法：

| 方案 | 代表 | 适用 |
|------|------|------|
| 完整复式记账 | Stripe Ledger、Modern Treasury | 专业支付/会计平台 |
| **物化余额 + 不可变流水** | 多数业务钱包系统 | **我们的场景** |

**选方案二**。理由：
1. 我们是业务钱包，不是会计系统
2. 复式记账需要维护大量"对手方账户"，当前需求没这个维度
3. 物化余额查询 O(1) + 不可变流水可审计 + 定时对账可验证，已够用
4. 后续有需要可向复式记账演进（流水表结构兼容）

**核心约束**：
- 流水表只有 INSERT，永远没有 UPDATE 和 DELETE
- 余额修正通过"反向流水"实现，不直接改余额字段
- 定时对账恒等式：`当前余额 == SUM(所有流水)`

## 3.4 核心数据流

把所有业务流程抽象到最本质，钱包就是做一件事 — **余额的增减冻解**。所有功能最终都归结到 5 个基本操作：

```
                    所有上层业务最终汇聚到这里
                              │
        ┌──────────┬──────────┼──────────┬──────────┐
        │          │          │          │          │
        ▼          ▼          ▼          ▼          ▼
   Credit       Debit      Freeze    Unfreeze   DeductFrozen
   (加款)       (减款)      (冻结)     (解冻)     (扣冻结)
     │          │          │          │          │
     └──────────┴──────────┴──────────┴──────────┘
                          │
                     统一执行流程
                          │
    ┌─────────────────────┼─────────────────────┐
    │  1. 幂等校验（Redis SetNX）                  │
    │  2. 开启事务                                  │
    │  3. 行锁定（SELECT ... FOR UPDATE）           │
    │  4. 读取当前余额                              │
    │  5. 校验（余额/冻结是否充足）                  │
    │  6. 计算新余额                                │
    │  7. 更新余额                                  │
    │  8. 写入账变流水（前余额→变动→后余额）          │
    │  9. 提交事务                                  │
    │ 10. 返回结果                                  │
    └───────────────────────────────────────────────┘
```

无论是充值到账、提现冻结、兑换扣减、人工加款、游戏扣款 — 进到内核都走这同一套流程。**这是整个钱包最关键的代码，大约 200-300 行，但需要保证 100% 正确。**

# 四、技术挑战与解法 — 核心难点怎么攻克

钱包不是普通 CRUD 业务。我总结了 5 个必须在 v1 解决的致命级挑战：

## 4.1 并发安全 — 防超卖

**风险**：同一账户并发扣款，余额扣成负数。

**解法**：悲观锁 SELECT ... FOR UPDATE

```sql
-- 事务内先锁定再读取再更新
BEGIN;
SELECT available_balance, frozen_balance FROM wallet_account
  WHERE user_id = ? AND currency_code = ? AND wallet_type = ?
  FOR UPDATE;
-- 此时其他事务对同一行的操作会被阻塞，直到当前事务提交
-- 安全地进行余额校验和更新
UPDATE wallet_account SET available_balance = available_balance - ? ...;
INSERT INTO wallet_transaction (...) VALUES (...);
COMMIT;
```

**为什么不用乐观锁**：钱包余额操作频率高（消费+充值+提现并发），乐观锁冲突时需重试，重试成本高于悲观锁的等待成本。

**死锁防范**：兑换操作需要锁两个币种的行。解法是**固定锁序** — 按 currency_code 字母序加锁（BSB < IDR < THB < USDT < VND），任何操作都遵循这个顺序，消除死锁可能。

## 4.2 幂等保障 — 防重复入账

**风险**：网络重试、回调重推导致同一笔钱入账两次。

**解法**：双层保险

```
第一层：Redis SetNX（快速拦截）
    key = "wallet:idempotent:{orderNo}:{opType}"
    TTL = 24h
    设置成功 → 首次请求，继续
    设置失败 → 重复请求，直接返回成功

第二层：数据库唯一索引（兜底）
    wallet_transaction 表 UNIQUE INDEX (order_no, op_type)
    即使 Redis 丢失，数据库层也能拦截重复
```

**关键细节**：重复请求不能返回错误，必须返回成功。否则调用方（比如财务模块）以为失败了会继续重试，形成无限循环。

## 4.3 数据一致性 — 余额与流水必须对得上

**约束**：`当前余额 = 初始值 + SUM(所有流水)` 必须恒成立。

**保障机制**：
1. **同一事务**：余额更新和流水写入在同一个数据库事务中，要么都成功要么都回滚
2. **流水不可变**：wallet_transaction 表只追加不修改，代码中没有 UPDATE/DELETE 操作
3. **定时对账**：每日凌晨跑对账任务，逐用户验证 `余额 == SUM(流水)`，不一致的记录差异+告警
4. **流水记录完整**：每条流水必须包含 before_balance 和 after_balance，形成链式验证

## 4.4 资金安全 — 提现三步模式

提现不直接扣余额，而是走三步：

```
余额初态：可用=1000，冻结=0

步骤1 — 冻结（用户发起提现 500）
    可用=500，冻结=500，总额不变=1000

步骤2a — 解冻（审核驳回）
    可用=1000，冻结=0，总额不变=1000（钱退回来了）

步骤2b — 扣除冻结（出款成功）
    可用=500，冻结=0，总额=500（钱真的走了）

步骤2c — 出款失败
    可用=500，冻结=500，总额=1000（保持冻结，不自动解冻）
    必须人工处理，防止自动解冻后用户趁机转走资金
```

**为什么出款失败不自动解冻**：这是产品的安全设计。出款失败的原因可能是银行卡不存在、通道异常、也可能是风险交易。自动解冻会给欺诈留窗口。

## 4.5 多币种精度 — 不同币种精度不同

| 币种 | 最小精度 | 举例 |
|------|---------|------|
| VND | 整数（无小数） | 500,000 |
| IDR | 整数（无小数） | 2,000,000 |
| THB | 2位小数 | 3,000.00 |
| BSB | 2位小数 | 100.00 |
| USDT | 最多8位小数 | 50.12345678 |

**解法**：
- 存储层统一 DECIMAL(20,8)，覆盖最高精度
- 业务层统一用 shopspring/decimal 运算，禁用 float64
- 显示层根据 currency_config 表中的 decimal_digits 字段截断
- RPC 传输用 string 类型，避免序列化精度丢失

## 5.2 各阶段详解

### Phase 1 — 工程搭建

这是开发的绝对前置，一行业务代码都不能写，先把这些做完：

| 动作 | 说明 | 为什么绕不过 |
|------|------|------------|
| 定义 IDL | common/idl/ser-wallet/ 下 7 个 thrift 文件 | IDL 先行是工程规范，gen_kitex.sh 依赖它生成代码 |
| 执行 gen_kitex.sh | 生成 kitex_gen + handler.go + main.go | 这是 RPC 服务的骨架代码 |
| 注册 go.work | 添加 `./ser-wallet` | 不注册则其他模块无法引用 |
| 注册 namesp.go | 添加 ETCD 常量（Service/Port/DB） | ETCD 注册/配置读取的标识 |
| 注册 rpc_client.go | 添加 WalletClient() 工厂方法 | 其他模块调我方 RPC 的入口 |
| 修复 gorm_gen.go | namesp.EtcdAppDb → EtcdWalletDB | 当前代码引用了错误的数据库 |
| 数据库建表 | 至少核心 7 张 Must 级表 | GORM Gen 需要表结构来生成代码 |
| 执行 GORM Gen | 生成 model/query 代码 | rep 层的基础 |
| 初始化 cfg.go | ETCD 配置加载 + Redis 初始化 | 运行时依赖 |
| 定义错误码 + 枚举 | errs/code.go + enum/ | 业务代码依赖 |

**产出**：一个可以编译运行、能注册到 ETCD、有空 handler 实现的 ser-wallet 服务。

### Phase 2 — 币种配置（可独立完成，零外部依赖）

```
优先做这个的原因：所有资金操作都依赖币种/汇率数据，它是地基中的地基。

开发内容：
├── B端接口（6个）
│   ├── 币种列表查询（分页+筛选）
│   ├── 币种编辑（汇率浮动/阈值/图标/状态）

│   ├── 基准币种设置（一次性不可改）
│   ├── 汇率日志查询
│   └── 币种详情
│
```

### Phase 3 — 钱包核心（可独立完成，是联调的前提）

```
这个阶段完成 = 财务模块可以开始联调

开发内容：
├── 5 个核心余额方法
│   ├── CreditBalance（加款）
│   ├── DebitBalance（减款）
│   ├── FreezeBalance（冻结）
```

**这个阶段完成后**，财务模块就可以拿这 6 个 RPC 做联调 — 调 CreditWallet 模拟充值入账，调 FreezeBalance/UnfreezeBalance 模拟提现审核，不需要等后面的阶段。

### Phase 7 — 联调验证

```

├── 充值全链路：用户→选方式→创建订单→三方支付→回调→入账
├── 提现全链路：用户→冻结→风控审核→财务审核→出款→扣冻结/解冻
```

# 六、跨模块协作 — 怎么和财务模块配合

## 6.1 职责边界一句话

**钱包管余额，财务管通道**。钱包不碰支付通道的密钥和回调验签，财务不直接操作用户余额。两边通过 6 个核心 RPC 接口交互。

## 6.2 调用方向图

```
                     ser-wallet（我方）           ser-finance（他方）

我方对外暴露：       CreditWallet ←─────────────── 充值成功/补单/加款
（财务模块来调）     DebitWallet  ←─────────────── 人工减款
                    FreezeBalance ←────────────── (或我方内部冻结)
                    UnfreezeBalance ←──────────── 审核驳回
                    DeductFrozen ←─────────────── 出款成功
                    QueryBalance ←─────────────── 查余额(加减款表单)

我方对外调用：       GetPaymentMethods ──────────→ 获取充值/提现方式
（我方去调他方）     MatchAndCreateChannelOrder ─→ 通道匹配+创建通道订单
                    InitiateChannelPayout ──────→ 发起代付

我方调KYC：          CheckKycStatus ────────────→ ser-kyc（提现校验）
我方调用户：         GetUserInfo ───────────────→ ser-user（按需）
```

## 6.3 联调策略

```
联调前：各自独立开发

我方                          财务方
┌─────────────┐              ┌─────────────┐
│ 定义 IDL     │              │ 定义 IDL     │
│ Mock 财务RPC │              │ Mock 钱包RPC │
│ 内部逻辑跑通  │              │ 内部逻辑跑通  │
└─────────────┘              └─────────────┘

联调时：双方 RPC 对接，Mock 替换为真实调用

┌────────────────────────────────────────┐
│  我方提供：6 个核心 RPC 的真实实现       │
│  财务提供：支付方式/通道匹配/代付 RPC    │
│                                        │
│  按链路跑通：充值→提现→人工修正          │
└────────────────────────────────────────┘
```

**Mock 策略**：在 ser 层写条件分支，配置项 mock_finance=true 时走 Mock 数据，false 走真实 RPC。联调时一键切换。

# 七、风险管理

## 7.1 已识别的风险和应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 财务模块 RPC 延迟交付 | 充值/提现无法联调 | Phase 2-4 先独立推进，Mock 先行 |
| 需求评审后细节变动 | 接口参数可能调整 | IDL 字段用 optional 预留，核心骨架不变 |
| 金额精度问题 | 不同币种精度不同 | 统一 DECIMAL(20,8) + shopspring/decimal |
| 余额并发冲突 | 高并发超卖 | SELECT FOR UPDATE 行锁 + 固定锁序 |
| 汇率 API 不可用 | 汇率无法更新 | 三个API容错 + 保持旧值 + 告警 |
| ser-kyc 接口不满足 | 提现KYC校验受阻 | 已确认 ser-kyc IDL 有 KycQuery 接口 |
| 活动模块不存在 | 充值奖金功能缺数据 | 预留接口结构，先不实现奖金逻辑 |

## 7.2 不可独立解决、需要评审确认的事项

以下 5 项是**阻塞开发的**，必须在跨部门评审中确认：

| 序号 | 事项 | 为什么阻塞 |
|------|------|----------|
| 1 | 金额单位（string/i64/Decimal） | 三方必须统一，否则联调对不上 |
| 2 | 充值订单归属 + 回调链路 | 决定代码写在哪个服务 |
| 3 | 提现冻结由谁发起 | 决定冻结操作的代码位置 |
| 4 | 钱包类型枚举值 | 写入IDL合同，双方必须一致 |
| 5 | 提现状态机完整定义 | 需要产品补充状态流转图 |

这些已整理成评审反馈文档，评审会上会逐项确认。

# 九、总结

## 9.1 确定性骨架 — 绕不过的 8 件事

```
1. IDL 先行 → gen_kitex.sh 生成代码 → 一切的起点
2. 4 文件注册 → go.work / namesp / rpc_client / gorm_gen → 融入工程
3. 9 张表 → 币种配置到提现账户 → 数据基础
4. 5 个核心方法 → Credit/Debit/Freeze/Unfreeze/DeductFrozen → 心脏
5. 6 个核心 RPC → 与财务模块的联调合同
6. 币种配置+汇率 → 所有资金操作的数据基础
7. 幂等+事务+流水 → 资金安全的三道防线
8. 冻结三步模式 → 提现的核心机制
```

## 9.2 实施路径一句话

**先搭工程骨架（IDL+注册+建表），再建核心能力（余额操作+RPC），然后铺业务功能（币种→兑换→充值→提现），最后联调验证（与财务模块对接全链路）。能独立做的先做，需要联调的 Mock 先行。**

## 9.3 我的信心来源

1. **需求理解充分**：152 张原型全部看完，产出 13 份分析文档，对业务规则、流程链路、技术挑战心中有数
2. **技术栈 90% 复用**：不需要引入新框架，在熟悉的技术体系上开发
3. **独立可做的占多数**：币种配置、钱包核心、兑换、记录查询都不依赖外部模块，可以先行推进
4. **核心难点有方案**：并发安全（行锁）、幂等（双层保险）、一致性（同事务+对账）、提现安全（冻结三步）— 都是业界成熟方案
5. **风险已识别**：知道什么会出问题、什么时候会出问题、怎么应对

以上就是我的实现方案。各位有什么问题，我们现在讨论。
