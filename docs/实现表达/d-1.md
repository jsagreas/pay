## 一、我负责什么

我主要负责 **ser-wallet** 服务的设计与实现，也就是钱包多币种模块。这个模块包含三个部分：

| 部分 | 我的角色 | 说明 |
|------|---------|------|
| **钱包模块** | 主责 | 账户余额、充值、提现、兑换、账单记录 |
| **币种配置模块** | 主责 | 币种管理、汇率管理、充提配置 |
| **财务管理模块** | 联调配合 | 别人负责开发，但涉及充值回调、提现审核回调、出款回调等深度联调 |

在整个系统中，钱包模块处于**资金链路的核心位置**——用户的每一笔钱进来（充值）、出去（提现）、在平台内流转（兑换、投注、购买），都要经过钱包的余额变更。所以这个模块对**数据准确性和安全性**的要求是最高的。

## 二、需求全貌——我要实现什么

### 2.1 一句话概括

> 实现一个支持多币种（法币 + USDT + 平台虚拟币 BSB）的钱包系统，涵盖用户资产管理、充值入金、提现出金、币种兑换四大核心能力，同时提供 B 端管理后台和面向其他服务的 RPC 接口。

### 2.2 用户视角——C 端做了什么

站在用户的角度，整个钱包模块的使用路径是这样的：

```
用户进入 APP
    │
    ├── 充值中心 ──→ 选币种 → 选金额 → 选支付方式 → 支付 → 到账
    │
    ├── 我的钱包 ──→ 查看各币种余额 → 查看资产总览
    │
    ├── 兑换 ────→ 选源币种 → 输金额 → 预览汇率 → 确认兑换 → 到账
    │
    ├── 提现 ────→ 选币种 → 选钱包 → 输金额 → 选提现账户 → 提交 → 等审核 → 到账
    │
    └── 账单 ────→ 查看充值/提现/兑换/奖励等所有资金变动记录
```

### 2.3 管理视角——B 端做了什么

```
运营/财务进入管理后台

    ├── 汇率管理 ──→ 录入/更新汇率、查看汇率变更日志
    │
    ├── 充提配置 ──→ 设置各币种的充值限额、提现限额、手续费
    │
    ├── 充值订单 ──→ 分页查询、详情查看、导出
    │
    ├── 提现订单 ──→ 分页查询、审核（通过/驳回）、详情、导出
    │
    ├── 交易流水 ──→ 全量流水查询、详情、导出
    │
    └── 用户钱包 ──→ 查询用户余额、手动调账（加款/减款）
```

### 2.4 系统视角——RPC 供给什么

钱包不只服务于前端，还要被其他后端模块调用：

| 调用方 | 调用场景 | 钱包提供的能力 |
|--------|---------|--------------|
| 财务模块 | 充值到账通知 | `CreditBalance` — 加款 |
| 财务模块 | 提现出款成功 | `DebitBalance` — 从冻结金额扣款确认 |
| 财务模块 | 提现出款失败 | `UnfreezeBalance` — 解冻退回可用 |
| 投注系统 | 查询余额判断是否够下注 | `GetBalance` — 查余额 |
| 结算/返奖 | 用户赢了给钱 | `CreditBalance` — 加奖金 |
| 活动系统 | 发放奖励 | `CreditBalance` — 加赠送金 |
| 风控系统 | 风控审批需冻结 | `FreezeBalance` — 冻结 |

### 3.3 我的核心设计决策

在深入分析需求和技术挑战后，我做了以下关键设计决策：

#### 决策一：金额用整数存储，不用浮点数

```
存储层：bigint（最小单位整数）
    1 CNY = 100（分）
    1 USDT = 100（美分）或 1000000（微单位，取决于精度确认）

计算层：先乘后除，避免精度丢失
    兑换 100 USD → CNY，汇率 7.24（存储为 724000000，即 ×10^8）
    计算：100_00 × 724000000 / 100000000 = 724_00（分）= 724.00 CNY

展示层：前端根据币种的 decimal_places 除以对应倍数展示
```

**为什么这么做：** 这是支付宝、微信支付、Stripe 的统一做法。浮点数在金融计算中会丢失精度——`0.1 + 0.2 ≠ 0.3` 不是笑话，是真实的线上事故。

#### 决策二：余额变更用悲观行锁 + 数据库事务

```
核心操作模式（ModifyBalance）：

BEGIN 事务
    ① SELECT * FROM wallet_account WHERE user_id=? AND currency_id=? AND wallet_type=? FOR UPDATE
       → 锁住这一行，其他并发请求排队等待
    ② 校验余额是否充足（扣款/冻结场景）
    ③ UPDATE wallet_account SET available_balance = available_balance +/- ? WHERE id = ?
    ④ INSERT INTO transaction_log (...)
       → 流水与余额变更在同一个事务中，要么全成功要么全回滚
COMMIT
```

**为什么选悲观锁而不是乐观锁：**
- 钱包余额是高争抢资源（充值、投注、提现可能同时操作同一个账户）
- 乐观锁在高冲突场景下需要反复重试，反而更慢
- 悲观锁配合短事务（控制在毫秒级），锁等待时间极短
- 支付宝和微信支付的余额操作也是悲观行锁

#### 决策三：幂等性用数据库唯一索引保障

```
transaction_log 表上有唯一索引：
    UNIQUE INDEX uk_ref_order (ref_order_no, tx_type, wallet_type)

含义：同一笔关联订单 + 同一种交易类型 + 同一个钱包类型 = 只允许写入一条流水

效果：即使财务模块因网络重试多次调用 CreditBalance，第二次 INSERT 会触发唯一索引冲突，自动去重。
```

**为什么不用 Redis 做幂等：** Redis 有时间窗口问题（key 过期后就不幂等了），而数据库唯一索引是永久的、和事务原子绑定的。

#### 决策四：跨服务不用分布式事务框架

钱包与财务模块是跨服务调用，但我**不引入 Seata/DTM 等分布式事务框架**。原因：

| 方案 | 优点 | 缺点 |
|------|------|------|
| **Seata/DTM** | 自动化 | 引入重、运维复杂、性能开销大、工程中没有先例 |
| **幂等 RPC + 重试 + 补偿（我选的）** | 简单可靠 | 需要写补偿逻辑 |

我采用的方案是"三板斧"：

```
① 幂等 RPC — 每个操作有唯一的 refOrderNo，重复调用不会重复执行
② 超时重试 — 调用失败后按退避策略重试
③ 定时补偿 — 定时任务扫描"卡住"的订单，主动查询/重试
```

这也是支付宝、微信支付处理跨系统一致性的主流做法。

#### 决策五：提现用冻结-确认/取消模式

```
提现流程中余额变化：

用户发起提现 100 元：
    available_balance: 1000 → 900（减 100）
    frozen_balance:       0 → 100（加 100）
    → 总余额不变，只是从"可用"转到"冻结"

审核通过 + 出款成功：
    frozen_balance: 100 → 0（减 100）
    → 冻结金额真正扣除

审核驳回 / 出款失败：
    available_balance: 900 → 1000（加 100）
    frozen_balance:    100 →    0（减 100）
    → 冻结金额退回可用
```

**为什么这么做：** 提现从申请到出款成功中间可能经过审核、出款等多个环节。如果直接扣余额，审核驳回后还要"补回"，逻辑复杂且有时间窗口风险（用户看到余额为 0，以为钱没了）。冻结模式让用户随时能看到"钱还在，只是暂时不能用"。

## 四、核心流程——关键场景怎么走

### 4.1 充值流程

```
┌──────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 用户  │───→│ ser-wallet │───→│ 财务模块  │───→│ 支付通道  │───→│  银行/链  │
│      │    │ 创建订单   │    │ 发起支付  │    │ 代收      │    │  上网络   │
└──────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
                                                                   │
                ┌──────────────────────────────────────────────────┘
                │ 支付成功回调
                ▼
┌──────────┐    ┌──────────┐
│ 财务模块  │───→│ ser-wallet │
│ 通知到账  │    │ CreditBalance（加款）│
└──────────┘    └──────────┘
```

**关键保障：**
1. 充值订单有**超时自动关闭**（定时扫描 pending 订单）
2. 回调有**幂等保障**（唯一索引去重）
3. 回调丢失有**主动查询**（定时扫描 + 人工补单兜底）

### 4.2 提现流程

```
┌──────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 用户  │───→│ ser-wallet │───→│ 财务模块  │───→│ 风控审核  │───→│ 财务审核  │
│ 申请  │    │ 冻结余额   │    │ 接收申请  │    │ 通过/拒绝  │    │ 通过/驳回 │
└──────┘    └─────┬────┘    └──────────┘    └──────────┘    └────┬─────┘
                  │                                              │
                  │         ┌────────────────────────────────────┘
                  │         │ 审核通过，发起出款
                  │         ▼
                  │    ┌──────────┐    ┌──────────┐
                  │    │ 出款通道  │───→│ ser-wallet │
                  │    │ 代付到账  │    │ DebitBalance（确认扣款）│
                  │    └──────────┘    └──────────┘
                  │
                  │    审核驳回/出款失败
                  │         │
                  │         ▼
                  │    ┌──────────┐
                  └───→│ ser-wallet │
                       │ UnfreezeBalance（解冻退回）│
                       └──────────┘
```

**状态机：**
```
待审核 ──→ 审核通过 ──→ 处理中 ──→ 成功
  │                       │
  ├──→ 驳回               └──→ 失败
```

每个状态转换都使用**条件更新**（`WHERE status = 当前状态`），防止并发导致状态跳跃。

### 4.3 兑换流程

```
用户请求预览：
    输入：100 USD → CNY
    后端查汇率：1 USD = 7.24 CNY
    返回预览：100 USD → 724 CNY（有效期 30 秒）

用户确认兑换（在一个数据库事务内完成）：
    ① 校验汇率是否仍在有效期内且偏差在阈值内
    ② SELECT FOR UPDATE 锁定源钱包（USD）和目标钱包（CNY）
       → 有序加锁：按 wallet_type + currency_id 升序锁定，防止死锁
    ③ 源钱包扣款：available_balance -= 10000（100 USD，单位分）
    ④ 目标钱包加款：available_balance += 72400（724 CNY，单位分）
    ⑤ 写入两条 transaction_log（一扣一加）
    ⑥ 写入 exchange_record（含汇率快照）
    COMMIT
```

**关键点：汇率快照。** 兑换记录中存的是成交时刻的汇率，不是实时汇率。这样即使后续汇率变动，历史记录仍然可追溯、可对账。

### 4.4 余额查询（高频读场景）

```
C端查余额 → 先查 Redis 缓存
                │
                ├── 命中 → 直接返回
                │
                └── 未命中 → 查 DB → 写入 Redis（设 TTL）→ 返回

余额变更后 → 删除 Redis 缓存（下次读时自动重建）
```

对于余额这种高频读的数据，用 **Cache-Aside** 模式。但**写操作永远以数据库为准**，Redis 只是加速读取，不参与余额计算。

## 五、数据设计——表怎么建

### 5.1 表的分类和优先级

我一共设计了 **12 张表**，按必要性分三档：

| 优先级 | 数量 | 表名 | 说明 |
|--------|------|------|------|
| **必须（Must）** | 8 | currency, exchange_rate_log, wallet_account, transaction_log, deposit_order, withdraw_order, exchange_record, withdraw_account | 没有这些表，任何核心功能都无法运行 |
| **应该（Should）** | 2 | deposit_config, withdraw_config | 充提限额配置，初期可用默认值但建议一起建 |
| **可选（Optional）** | 2 | base_currency_config, freeze_record | 后续迭代可加 |

### 5.2 核心表关系

```
currency（币种表）
    │
    ├── 1:N ── exchange_rate_log（汇率日志）
    ├── 1:1 ── deposit_config（充值配置）
    ├── 1:1 ── withdraw_config（提现配置）
    │
    └── 1:N ── wallet_account（钱包账户）
                    │
                    ├── 1:N ── transaction_log（交易流水）
                    ├── 1:N ── deposit_order（充值订单）
                    ├── 1:N ── withdraw_order（提现订单）
                    └── 1:N ── exchange_record（兑换记录）

withdraw_account（提现账户）── N:1 ── user
```

### 5.3 最核心的两张表

**wallet_account — 钱包账户（整个模块的"心脏"）：**

```sql
CREATE TABLE wallet_account (
    id                BIGINT       PRIMARY KEY,
    user_id           BIGINT       NOT NULL,
    currency_id       BIGINT       NOT NULL,
    wallet_type       TINYINT      NOT NULL,     -- 1中心 2奖励 3主播 4代理
    available_balance BIGINT       NOT NULL DEFAULT 0,  -- 可用余额
    frozen_balance    BIGINT       NOT NULL DEFAULT 0,  -- 冻结金额
    status            TINYINT      NOT NULL DEFAULT 1,  -- 1正常 0冻结
    ...
    UNIQUE INDEX uk_user_currency_wallet (user_id, currency_id, wallet_type)
) ENGINE=InnoDB;
```

**transaction_log — 交易流水（整个模块的"账本"）：**

```sql
CREATE TABLE transaction_log (
    id               BIGINT       PRIMARY KEY,
    user_id          BIGINT       NOT NULL,
    wallet_id        BIGINT       NOT NULL,
    tx_type          TINYINT      NOT NULL,      -- 1充值 2提现 3兑换出 4兑换入 5返奖 ...
    direction        TINYINT      NOT NULL,      -- 1收入 2支出
    amount           BIGINT       NOT NULL,
    balance_before   BIGINT       NOT NULL,      -- 变更前余额
    balance_after    BIGINT       NOT NULL,      -- 变更后余额
    ref_order_no     VARCHAR(64)  NOT NULL,      -- 关联订单号
    ...
    UNIQUE INDEX uk_ref_order (ref_order_no, tx_type, wallet_type)  -- 幂等键
) ENGINE=InnoDB;
```

**这两张表的设计逻辑：**
- `wallet_account` 存的是"当前快照"——用户此刻有多少钱
- `transaction_log` 存的是"变更历史"——每一笔钱的来龙去脉
- 两者通过 `balance_before` / `balance_after` 关联：任意时刻，流水的最新一条 `balance_after` 应等于 `wallet_account.available_balance`
- 这就是**对账的基础**——如果不等，说明有异常

### 5.4 索引策略

一共设计了 **27 个索引**，每一个都对应具体的查询场景：

| 索引类型 | 数量 | 说明 |
|---------|------|------|
| 主键索引 | 10 | 每张表一个 |
| 唯一业务索引 | 6 | 幂等保障、实体唯一性约束 |
| 普通查询索引 | 11 | B 端分页查询、C 端列表查询 |

没有一个索引是"凑数"的，每个都能说出对应的查询场景和 SQL。

## 六、接口设计——IDL 怎么拆

### 6.1 文件拆分

按业务域拆分为 **9 个 Thrift 文件**：

```
common/idl/ser-wallet/
├── service.thrift              ← 主入口（WalletService，49 个方法）
├── wallet.thrift               ← 钱包余额
├── currency.thrift             ← 币种 + 汇率 + 充提配置
├── deposit.thrift              ← 充值
├── withdraw.thrift             ← 提现
├── withdraw_account.thrift     ← 提现账户
├── exchange.thrift             ← 兑换
├── record.thrift               ← 账单记录
└── wallet_rpc.thrift           ← RPC 供给接口（供其他服务调用）
```

### 6.2 方法统计

### 6.3 为什么 RPC 接口单独一个文件

## 八、技术挑战——难点怎么解决

我识别了 **18 个技术挑战**，每个都有成熟的解决方案。这里挑最关键的 6 个说明：

### 挑战一：并发余额操作的一致性

**问题：** 用户同时充值和提现，两个请求并发修改同一行余额，可能导致数据错乱。

**方案：** `SELECT FOR UPDATE` 悲观行锁。在事务中锁定 `wallet_account` 的行，其他请求排队等待。事务控制在毫秒级，锁等待时间极短。

### 挑战二：重复回调导致重复入账

**问题：** 支付回调可能因网络重试发送多次，如果不做幂等，用户会多收到钱。

**方案：** `transaction_log` 表的唯一索引 `(ref_order_no, tx_type, wallet_type)`。第二次 INSERT 触发唯一冲突，事务回滚，不会重复入账。

### 挑战三：回调丢失导致充值不到账

**问题：** 支付成功了但回调丢了，用户付了钱但余额没变。

**方案：** 三层保障：
1. **回调** — 支付成功后正常回调
2. **主动查询** — 定时扫描 pending 订单，主动向支付服务查询结果
3. **人工补单** — B 端提供补单入口，最终兜底

### 挑战四：提现冻结后出款方一直不回调

**问题：** 用户的钱被冻结了，但出款方迟迟不回复结果。

**方案：** 定时扫描冻结超时的订单。超过阈值后，系统自动主动查询出款结果；如果查不到，标记为异常等人工处理。绝不让用户的钱无限期冻结。

### 挑战五：兑换涉及两个钱包的原子性

**问题：** 兑换 = 扣 A 钱包 + 加 B 钱包，必须同时成功或同时失败。

**方案：** 在**同一个数据库事务**内完成两个钱包的余额变更。因为两个钱包都在 TiDB 同一个库中，本地事务就能保证原子性，不需要分布式事务。

**防死锁：** 同时锁两行时，按 `(wallet_type, currency_id)` 升序加锁，所有请求的加锁顺序一致，不会形成死锁。

### 挑战六：金额精度

**问题：** 100 / 3 = 33.333...，如何处理无限小数？

**方案：**
- 存储用最小单位整数，避免浮点数
- 汇率用 ×10^8 存储（如 7.24 存为 724000000）
- 计算用"先乘后除"（`amount * rate / 10^8`），中间值用 int64（最大 9.2 × 10^18，足够）
- 汇率换算如果有需要，用 shopspring/decimal 做中间运算，最终截断为整数
- 截断规则：向下取整（保护平台利益），展示层四舍五入（保护用户体验）

## 十、与财务模块的联调边界

### 10.1 谁调谁

```
充值场景：
    用户 → ser-wallet（创建订单）→ 财务（发起支付）→ ... → 财务（回调）→ ser-wallet（CreditBalance）

提现场景：
    用户 → ser-wallet（冻结余额 + 创建订单）→ 财务（审核 + 出款）→ 财务（回调）→ ser-wallet（DebitBalance 或 UnfreezeBalance）
```

**核心原则：**
- 钱包负责**余额变更**（加款、扣款、冻结、解冻）
- 财务负责**通道对接**（支付通道、出款通道）和**审批流程**
- 边界清晰：钱包不直接对接支付通道，财务不直接操作余额

### 10.2 联调接口清单

| 接口 | 调用方向 | 触发时机 |
|------|---------|---------|
| `CreditBalance` | 财务 → 钱包 | 充值到账 / 返奖 / 赠送 |
| `DebitBalance` | 财务 → 钱包 | 提现出款成功 |
| `FreezeBalance` | 钱包内部 | 提现申请时自行冻结 |
| `UnfreezeBalance` | 财务 → 钱包 | 提现驳回 / 出款失败 |
| `GetBalance` | 财务 → 钱包 | 审核时查询用户余额 |

### 10.3 联调需要双方确认的事项

1. `ref_order_no` 的构成规则（确保双方一致）
2. 回调的重试策略（最多几次、间隔多久）
3. 异常状态的处理流程（如财务挂了钱包怎么办）
4. 错误码的映射关系

这些需要在编码前与财务模块开发者对齐。

## 十一、交付计划

### 11.1 分阶段实施

| 阶段 | 内容 | 依赖 |
|------|------|------|
| **第一阶段：基础设施** | IDL 定义 + 代码生成 + 服务骨架 + 建表 + ETCD 配置 | 无 |
| **第二阶段：币种管理** | 币种 CRUD + 汇率管理 + 充提配置 + B 端接口 | 第一阶段 |
| **第三阶段：钱包核心** | 钱包账户创建 + 余额查询 + 余额变更引擎 + RPC 接口 | 第二阶段（需要币种数据） |
| **第四阶段：充值** | 充值全流程 + 回调处理 + 超时扫描 + 幂等 | 第三阶段 + 财务联调 |
| **第五阶段：提现** | 提现全流程 + 冻结/解冻 + 审核 + 回调 + 提现账户 | 第三阶段 + 财务联调 |
| **第六阶段：兑换 + 账单** | 兑换全流程 + 账单记录 + 导出 | 第三阶段 |
| **第七阶段：收尾** | 对账 + 定时任务 + 异常补偿 + 联调测试 | 全部 |

### 11.2 为什么这个顺序

```
币种 → 钱包 → 充值 → 提现 → 兑换
  ↓       ↓       ↓       ↓       ↓
 基础    核心   入金    出金    增值
```

- **币种是前提**：没有币种配置，钱包账户无法创建
- **钱包是核心**：没有余额引擎，充值/提现/兑换都无法运作
- **充值优先于提现**：用户先有钱才能提现
- **兑换最后**：依赖充值（有钱可兑）和汇率（已在币种阶段完成）

### 11.3 可并行的工作

- 第一阶段（基础设施）可以在需求评审通过后立即开始
- 第二阶段（币种管理）是纯 B 端 CRUD，不依赖财务联调，可以先行
- IDL 定义确认后，前端可以并行开发页面（Mock 数据联调）

**一句话概括我的实现方案：**

> 基于工程已有的 Kitex + GORM + Redis + ETCD 技术栈，构建一个标准的 Kitex RPC 服务（ser-wallet，编号 022），通过 **悲观行锁 + 数据库事务** 保证资金安全，通过 **唯一索引** 保证幂等性，通过 **冻结-确认/取消模式** 管理提现流程，通过 **定时任务** 兜底异常场景。12 张表、49 个接口、分 7 个阶段交付。不引入额外的重量级框架，用简单可靠的方案解决复杂的资金问题。

- **可靠** — 三层回调保障、定时补偿、对账机制，不丢一分钱
- **务实** — 复用已有组件、不过度设计、分阶段交付、风险可控
