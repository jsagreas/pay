## 一、开场：我要做什么

### 1.1 一句话概括

我要从零实现一个**多币种资金钱包系统**的后端服务（ser-wallet），让用户能用不同货币充值、兑换、提现，同时给运营提供币种配置和汇率管理能力，给财务模块提供资金操作的RPC接口。

### 1.2 为什么这个模块重要

```
这不是一个普通的CRUD模块。普通业务模块（比如Banner管理）：
  写入频率低、数据不涉及金额、错了可以直接改

钱包模块：
  ① 涉及真金白银 → 错一分钱就是资金事故
  ② 多方并发操作同一余额 → 充值+消费+提现可能同时发生
  ③ 跨服务调用 → wallet ↔ finance 通过RPC通信，网络不可靠
  ④ 状态流转复杂 → 提现有8种状态，任何非法跳转都是漏洞
  ⑤ 外部依赖多 → 三方支付回调、区块链确认、汇率API
  ⑥ 合规要求高 → 每笔变动需审计、对账、防洗钱
```

## 二、需求全景：这套系统的本质是什么

### 2.1 系统定位

平台横跨多个国家（越南、印尼、泰国等），同时包含泛娱乐（直播、打赏）和竞彩（投注、结算）两条业务线。资金体系必须支持多种货币并存、资金性质隔离、合规风控。

### 2.2 三个模块，各解决一个问题

从架构本质看，币种配置 + 钱包 + 财务 构成一个**支付账户体系**：

```
币种配置 = 计价引擎（告诉系统"钱怎么算"）
  → 平台支持哪些货币、汇率怎么算、精度怎么处理

钱包     = 账户引擎（管理"用户有多少钱、钱怎么动"）
  → 充值、兑换、提现、余额管理

财务     = 交易引擎（驱动"钱从外部进出系统"）
  → 通道配置、订单审核、代收代付、人工修正
```

### 2.3 支持的货币类型

| 类型 | 举例 | 特点 |
|------|------|------|
| 法定货币 | VND（越南盾）、IDR（印尼盾）、THB（泰铢） | 对应国家，有实时汇率波动，精度2位小数 |
| 加密货币 | USDT、BTC、ETH | 全球通用，链上交易，精度8位小数 |
| 平台货币 | BSB | 平台自定义，锚定USDT（10 BSB = 1 USDT） |

当前支持4个币种：**BSB、VND、IDR、THB**，未来可扩展。

### 2.4 关键业务流程一览

```
用户侧（C端）：
  充值 → 三种方式（USDT链上 / 银行转账 / 电子钱包扫码）
  兑换 → 法币→BSB 单向兑换
  提现 → 三种方式（银行卡 / 电子钱包 / USDT）
  查看 → 充值/提现/兑换/奖励 四种记录

运营侧（B端）：
  配置 → 币种管理（20+字段）、汇率监控、基准币种设置
  审计 → 汇率变更日志

系统内部：
  汇率 → 定时拉取3家API取平均，偏差达阈值自动更新平台汇率
  RPC  → 供财务模块调用的9个资金操作接口
```

## 四、核心业务模型：五种子钱包体系

### 4.1 这是整个系统最核心的业务设计

用户的"资金钱包"不是一个简单的余额数字，而是由**五种子钱包**组成，每种子钱包有不同的资金来源、使用规则和流向限制：

```
┌─────────────────────────────────────────────────────────────┐
│ 用户的钱包（每个币种独立一套）                                    │
│                                                              │
│  ┌───────────┐  ┌───────────┐  ┌───────────┐              │
│  │ 中心钱包   │  │ 奖励钱包   │  │ 主播钱包   │  本期实现     │
│  │ 充值进来   │  │ 活动返奖   │  │ 打赏收入   │              │
│  │ 投注/消费  │  │ 流水达标→  │  │ 可直接提现  │              │
│  │ 可提现     │  │ 转中心钱包  │  │ 或转中心   │              │
│  └───────────┘  └───────────┘  └───────────┘              │
│                                                              │
│  ┌───────────┐  ┌───────────┐                              │
│  │ 代理钱包   │  │ 场馆钱包   │  预留，本期不实现             │
│  └───────────┘  └───────────┘                              │
│                                                              │
│  资金钱包余额 = 中心钱包余额 + 奖励钱包余额                     │
│  消费优先级：先扣中心，再扣奖励                                  │
│  每个币种一套完整结构，币种间完全隔离                              │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 关键业务规则（绕不过的硬规则）

| 规则 | 具体内容 | 为什么绕不过 |
|------|---------|-----------|
| 余额公式 | 资金钱包余额 = 中心 + 奖励 | C端展示的唯一口径 |
| 消费优先级 | 先中心后奖励 | 混合投注返奖按比例分配的前提 |
| 币种隔离 | 每个币种独立一套子钱包，不可跨币种 | 基本会计原则 |
| 提现冻结 | 提现发起→冻结→审核→出款→扣款/解冻 | 防双花的核心机制 |
| 提现规则 | 法币充→法币提，USDT充→USDT提 | "从哪来回哪去"合规要求 |
| 基准币种 | USDT一次性设置不可改 | 跨币种统计的锚定点 |
| BSB锚定 | 10 BSB = 1 USDT，写死不做配置 | 产品硬要求 |
| 兑换方向 | 仅法币→BSB 单向 | 产品硬要求 |

## 五、架构设计：我选择怎么做

### 5.1 系统拓扑

```
                        ┌──────────────┐
                        │   C端用户     │
                        └──────┬───────┘
                               │ HTTPS
                        ┌──────▼───────┐
                        │  gate-font   │  C端网关(Hertz)
                        └──────┬───────┘
                               │ RPC (Kitex)
            ┌──────────────────┼──────────────────┐
            │                  │                  │
     ┌──────▼───────┐  ┌──────▼───────┐  ┌──────▼───────┐
     │  ser-wallet  │  │  ser-kyc     │  │  ser-user    │
     │  钱包+币种    │  │  KYC认证     │  │  用户服务     │
     └──────┬───────┘  └──────────────┘  └──────────────┘
            │ RPC (被财务调用)
     ┌──────▼───────┐
     │ ser-finance  │
     │ 财务(交易引擎)│
     └──────┬───────┘
            │ HTTP回调
     ┌──────▼───────┐
     │  三方支付/链上 │
     └──────────────┘

     ┌──────────────┐        ┌──────────────┐
     │   B端运营     │───────>│  gate-back   │──── RPC ──→ ser-wallet (B端接口)
     └──────────────┘        └──────────────┘
```

### 5.2 为什么是单一服务

**ser-wallet 是一个进程**，币种配置和钱包业务在同一进程内，通过Go包方法调用。

理由（ADR-07）：
- 钱包每次操作都需要调用币种配置的汇率/精度/换算
- 拆成两个服务 → 每次操作多一次网络RPC调用
- 拆开后兑换操作涉及两个服务的DB → 分布式事务问题
- 两个模块共享db_wallet数据库，拆服务没有数据隔离收益

### 5.3 服务内部分层

```
┌─────────────────────────────────────────────────────────────┐
│                     ser-wallet 服务进程                       │
│                                                              │
│  handler.go ──→ 一行委托，不写业务逻辑                        │
│       │                                                      │
│       ▼                                                      │
│  Service 层 ──→ 业务核心：校验→计算→持久化→缓存→日志          │
│  ┌────────────────────────────────────────────────────┐     │
│  │ CurrencyService ──→ 被所有其他Service依赖            │     │
│  │      │                                              │     │
│  │ WalletService ◄── 六种原子操作（余额变动唯一入口）    │     │
│  │      │                                              │     │
│  │      ├── DepositService  (调 credit)                │     │
│  │      ├── WithdrawService (调 freeze)                │     │
│  │      ├── ExchangeService (调 debit + credit)        │     │
│  │      └── RecordService   (纯查询)                   │     │
│  └────────────────────────────────────────────────────┘     │
│       │                                                      │
│       ▼                                                      │
│  Repository 层 ──→ GORM Gen 查询、事务、软删除               │
│       │                                                      │
│       ▼                                                      │
│  TiDB(db_wallet) + Redis(缓存/锁) + NATS(事件)              │
└─────────────────────────────────────────────────────────────┘
```

**核心约束**：所有余额变动只能通过WalletService的六种原子操作，不允许其他Service直接UPDATE余额字段。这保证了并发控制、流水记录、幂等检查不会被绕过。

## 六、数据设计：9张表撑起整个系统

### 6.1 表结构全景

我需要在db_wallet中建**9张必须表**，每一张都有明确的需求来源，没有拍脑袋加的表：

```
币种配置域（2张）：
├── t_currency          ← 平台支持哪些货币（20+字段，地基中的地基）
└── t_rate_log          ← 每次汇率更新的历史记录（审计需要）

钱包核心域（3张）：
├── t_wallet_account    ← 用户×币种×子钱包类型 = 一个账户（心脏）
├── t_wallet_flow       ← 每笔余额变动的不可变审计记录（幂等+对账）
└── t_freeze_record     ← 提现冻结的生命周期追踪（防双花）

业务域（4张）：
├── t_deposit_order     ← 充值订单（含重复转账拦截索引）
├── t_withdraw_order    ← 提现订单（含双审核字段）
├── t_exchange_order    ← 兑换订单（同步原子操作，创建即终态）
└── t_withdraw_account  ← 提现账户（只写一次不可改，风控要求）
```

### 6.2 三张核心表详解

**t_wallet_account — 系统的心脏**

```
联合唯一：(user_id, currency_code, wallet_type)
含义：一个用户一个币种一种子钱包 = 一个独立账户

关键字段：
  available_balance  DECIMAL(30,8) UNSIGNED  ← 可用余额
  frozen_balance     DECIMAL(30,8) UNSIGNED  ← 冻结余额（提现审核中）
  version            BIGINT                  ← 乐观锁版本号

为什么分 available 和 frozen：
  提现1000 → available -= 1000, frozen += 1000
  审核期间用户只能操作 available，frozen不可碰
  驳回 → available += 1000, frozen -= 1000（精确归还）
  成功 → frozen -= 1000（最终扣除）
  不分开的话，审核期间用户消费了，驳回时"恢复"回来的钱从哪扣？
```

**t_wallet_flow — 审计的基石**

```
唯一索引：(related_order_no, flow_type)  ← 幂等核心
含义：同一订单+同一流水类型 只能有一条记录，防重复入账

关键设计：
  只插入，不更新，不删除 ← 不可变审计记录
  amount 是 SIGNED ← 正数=增加，负数=减少
  balance_before/after ← 变动前后余额快照，用于对账
  frozen_before/after  ← 冻结变动快照

对账公式（任何时刻都必须成立）：
  最后一条流水的 balance_after == account.available_balance
```

**t_freeze_record — 防双花的守卫**

```
状态机：
  1(冻结中) → 2(已解冻)  -- WalletUnfreeze
  1(冻结中) → 3(已扣款)  -- WalletDeduct
  终态互斥：同一条记录只能走到其中一个终态

代码防御：
  UPDATE ... WHERE status = 1 AND id = ?
  affected_rows = 0 → 拒绝操作（已处理过或不存在）
```

### 6.3 建表顺序

```
Phase 1  基础层 → t_currency, t_rate_log（所有表都依赖它的currency_code）
Phase 2  核心层 → t_wallet_account, t_wallet_flow
Phase 3  业务层 → 其余5张表
```

建好表后执行`go run gorm_gen.go`生成model/query代码，然后开始编码。

### 6.4 枚举体系

14个枚举，覆盖所有字段的合法取值。举几个关键的：

| 枚举 | 值 | 使用表 |
|------|------|--------|
| WalletType | 1中心/2奖励/3主播/4代理/5场馆 | account/flow/freeze |
| FlowType | 14种流水类型（充值入账/奖金入账/兑换扣减/冻结/解冻/...） | flow |
| DepositStatus | 1待支付/2处理中/3已完成/4已失败 | deposit_order |
| WithdrawStatus | 8种（审核中→风控通过/驳回→财务通过/驳回→出款中→成功/失败） | withdraw_order |
| FreezeStatus | 1冻结中/2已解冻/3已扣款 | freeze_record |

## 七、接口设计：30个方法的IDL蓝图

### 7.1 IDL文件组织

通过分析工程中现有19个服务、71个thrift文件的实际模式，设计了8个IDL文件：

```
common/idl/ser-wallet/
├── service.thrift       ← 主入口：include所有文件 + WalletService定义（30个方法）
├── currency.thrift      ← 币种配置相关struct
├── wallet.thrift        ← 钱包余额相关struct
├── deposit.thrift       ← 充值相关struct
├── withdraw.thrift      ← 提现相关struct
├── exchange.thrift      ← 兑换相关struct
├── record.thrift        ← 记录查询相关struct
└── wallet_rpc.thrift    ← 跨服务RPC相关struct（供finance调用）
```

### 7.2 30个方法清单

按角色分组：

**B端接口（7个）— 运营后台**
```
CreateCurrency      新增币种
UpdateCurrency      编辑币种
ToggleCurrencyStatus  启禁用币种
SetBaseCurrency     设置基准币种
DetailCurrency      币种详情
PageCurrency        币种列表
PageRateLog         汇率日志
```

**C端接口（14个）— 用户操作**
```

## 八、资金安全：四道防线保障

资金系统最重要的是安全。我设计了四道防线，层层兜底：

### 8.1 四道防线体系

```
┌─ 第一道防线：接口层 ──────────────────────────────────────────┐
│ 参数校验 + 权限验证 + 频率限制                                   │
│ 网关已有：黑名单/白名单 + Token鉴权                              │
│ 服务补充：金额正数校验、币种有效性、用户归属校验                   │
└──────────────────────────────────────────────────────────────┘
                              │
┌─ 第二道防线：业务逻辑层 ──────────────────────────────────────┐
│ 余额充足性校验 + 限额校验 + 状态机守卫 + 业务规则校验            │
│ 提现限额（单笔+单日）+ KYC校验 + 重复转账拦截                   │
└──────────────────────────────────────────────────────────────┘
                              │
┌─ 第三道防线：数据层 ─────────────────────────────────────────┐
│ 乐观锁(version CAS) + 唯一索引幂等 + DB事务原子性 + 不可变流水  │
│ 即使业务逻辑有bug，数据层也能兜住                               │
└──────────────────────────────────────────────────────────────┘
                              │
┌─ 第四道防线：事后审计 ───────────────────────────────────────┐
│ 不可变流水记录 + 定期对账 + ser-blog操作日志 + 告警             │
│ 发现任何不一致 → 告警 → 人工介入（WalletManualAdd/Sub）        │
└──────────────────────────────────────────────────────────────┘
```

### 8.2 对账机制

```
一致性公式（任何时刻都必须成立）：
  account.available_balance == 最后一条flow的balance_after

对账实施：
  开发阶段：每次余额变动后实时assert（快速发现bug）
  上线后：定时任务每日凌晨扫描所有账户，比对流水和余额
  不一致 → 告警日志 → 人工排查 → WalletManualAdd/Sub修正
```

## 九、核心技术方案：五个关键决策

### 9.1 决策一：账户模型 — 多子账户模型（ADR-01）

```
为什么不用单一余额：
  业务明确要求五种子钱包，各有独立使用规则和流向限制
  资金隔离是合规硬要求

做法：
  (user_id, currency_code, wallet_type) 三维唯一确定一个账户
  每个账户有 available_balance + frozen_balance 两个余额字段
  用户首次访问某币种时惰性初始化该币种下的3个子账户
```

### 9.2 决策二：并发控制 — 乐观锁（ADR-03）

**这是P0级技术挑战。** 两个请求同时操作同一账户余额，如果不处理会导致"丢失更新"→ 资金凭空蒸发。

```
方案对比：
┌──────────────┬──────────────────┬──────────────────┐
│              │ 乐观锁(version)   │ 悲观锁(FOR UPDATE)│
├──────────────┼──────────────────┼──────────────────┤
│ 锁粒度        │ 无锁，CAS更新     │ 行级锁，独占      │
│ 死锁风险      │ 无               │ 有               │
│ 性能（低冲突）│ ★★★             │ ★★              │
│ TiDB适配     │ 原生支持          │ 额外开销         │
└──────────────┴──────────────────┴──────────────────┘

选择乐观锁：
  同一用户同一币种同一子钱包的并发冲突概率低
  无死锁风险（钱包系统不能接受死锁导致的余额锁定）
  TiDB对乐观事务有原生支持

代码模式：
  withRetry(ctx, 2, func() error {
      return tx.Transaction(func(tx *query.Query) error {
          读取 account + version
          校验余额充足
          UPDATE SET balance=新值, version=version+1 WHERE version=旧版本
          affected_rows == 0 → 重试
          写流水（同一事务中）
      })
  })

  最多重试2次（共3次尝试），不需要sleep。
  3次都失败 → 返回错误码让上游决定。
  监控重试率 → 持续高于5% → 考虑升级为悲观锁。
```

### 9.3 决策三：幂等性 — 流水表唯一索引（ADR-04）

**这也是P0级挑战。** finance调WalletCredit超时后重试，如果没有幂等保护 → 重复入账 → 资金凭空增加。

```
方案选择：
  ❌ Redis SetNX — 和DB事务不在同一原子单元，可能Redis成功但DB失败
  ❌ 独立幂等表 — 多维护一张表，本质和流水表一样
  ✅ 流水表唯一索引 — t_wallet_flow的(related_order_no, flow_type) UNIQUE

工作原理：
  Transaction {
      UPDATE wallet_account SET balance += 1000 WHERE ...   // 更新余额
      INSERT INTO wallet_flow (order_no, flow_type, ...)    // 写流水（唯一索引）
  }

  首次请求 → 插入成功 → 事务提交 → 返回成功
  重复请求 → 唯一索引冲突 → 事务回滚（余额未变） → 返回"已处理"

优势：
  余额更新和幂等检查在同一DB事务中，原子性天然保证
  不依赖额外组件
  流水表一表两用（审计+幂等）
```

### 9.4 决策四：金额精度 — DECIMAL + string + decimal库（ADR-06）

```
为什么不用float64：
  0.1 + 0.2 = 0.30000000000000004（IEEE 754浮点误差）
  一万笔交易累积误差 → 对账不平 → 用户投诉

三层配合：
  数据库：DECIMAL(30,8)          ← 精确十进制，零浮点误差
  传输层：string                  ← 避免Thrift double精度丢失
  计算层：shopspring/decimal 库   ← Go层精确计算

完整链路：
  前端 "100.50" → decimal.NewFromString("100.50")
  → 运算 → result.StringFixed(8)
  → DB DECIMAL(30,8) → 查出 → string回传前端

这是支付行业标准方案：Stripe、支付宝、微信支付均采用。
```

### 9.5 决策五：跨模块通信 — RPC + MQ混合（ADR-05）

```
同步RPC（Kitex，3s超时）：
  finance → wallet 的所有资金操作（WalletCredit/Freeze/Deduct...）
  原因：资金操作必须知道结果（成功/失败），不能"发出去不管"

异步MQ（NATS JetStream）：
  wallet → finance 的事件通知（提现创建→通知审核，充值创建→通知选通道）
  原因：wallet不需要等finance处理结果，发出去就行

同进程方法调用：
  钱包模块 → 币种配置模块（同一进程，不走网络）

为什么不用分布式事务框架（如Seata）：
  引入新依赖和复杂度 → 过度设计
  用"本地事务 + 重试 + 幂等"模式即可（业界称Best-Effort Delivery）
  极端不一致 → 对账发现 → 人工修正
```

## 十、业务实现：四大子模块的实现要点

### 10.1 币种配置模块（Phase 2，最先做的业务模块）

**为什么先做它**：所有金额操作都依赖它的汇率和精度，它是地基。

```
B端接口（7个）：
  CreateCurrency — 币种代码唯一校验，图标经ser-s3上传
  UpdateCurrency — 代码和类型不可修改
  SetBaseCurrency — 整表仅一条is_base=1，一次性不可改
  ToggleCurrencyStatus — 基准币种不允许禁用
  PageCurrency / DetailCurrency / PageRateLog — 标准分页查询

C端接口（2个）：
  ListActiveCurrencies — 查status=1的币种列表（缓存）
  GetExchangeRate — 获取某币种的平台/入款/出款汇率（缓存）

汇率定时任务：
  定时拉取3家三方API → 取平均 → 偏差 ≥ 阈值 → 更新平台汇率
  平台汇率变更 → 重算入款汇率和出款汇率 → 写汇率日志
  分布式锁（Redis SETNX）→ 多实例只有一个执行

缓存策略：
  币种配置列表 → Redis缓存 + 进程内sync.Map（变化频率极低）
  汇率数据 → Redis缓存（定时任务更新后主动刷新）
  CRUD操作后 → 主动清缓存
```

### 10.2 兑换模块（Phase 4，最简单的业务闭环）

**为什么先做兑换**：内部闭环不涉及三方/审核，用来验证余额体系是否正确。

```
两个接口：
  PreviewExchange — 试算（纯计算，不写DB）
  CreateExchangeOrder — 执行兑换

事务内容（一个DB事务，全成功或全失败）：
  1. 重新读取汇率（不用预览时的快照，防汇率变动）
  2. WalletService.debit(法币中心钱包, 源金额)  ← 扣减法币
  3. WalletService.credit(BSB中心钱包, 兑换金额) ← 增加BSB
  4. WalletService.credit(BSB奖励钱包, 赠送金额) ← 赠送入奖励钱包（如有）
  5. 写兑换订单记录

关键校验：
  源币种必须是法币（VND/IDR/THB），目标固定BSB
  金额 ≤ 法币中心钱包可用余额
  汇率展示口径：1 BSB = X 法币
```

### 10.3 充值模块（Phase 5，中等复杂度）

```
三种充值方式，技术实现路径不同：

USDT链上充值 → 平台承载页（自生成地址+二维码）
  技术：需对接区块链监听服务，收到链上确认后入账
  回调路径：区块链 → ser-finance → WalletCredit

银行转账 → 三方托管页（跳转三方支付）
  技术：wallet创建订单 → finance选通道 → 三方回调 → WalletCredit

电子钱包 → 三方托管页（跳转扫码）
  技术：同银行转账

重复转账拦截（仅银行/电子钱包）：
  查 t_deposit_order：同user + 同币种 + 同金额 + 状态=待支付/进行中
  0笔 → 正常
  1-2笔 → 返回软拦截标记（前端弹窗警告，允许继续）
  ≥3笔 → 返回硬拦截标记（前端弹窗禁止）

额外奖金：
  充值金额 → 中心钱包
  奖金金额 → 奖励钱包（标记流水倍数要求）
  互斥：同一笔充值只能参一个活动
```

### 10.4 提现模块（Phase 6，最复杂）

```
提现流程（我负责创建和冻结，finance负责审核和出款）：

  用户发起
  → KYC校验（rpc.KycClient()查状态，未认证仅限USDT）
  → 提现币种规则校验（法币充→法币提，USDT充→USDT提）
  → 限额校验（单笔最低/最高、单日限额）
  → 手续费计算（到账金额 = 提现金额 - 手续费）
  → 事务：创建提现订单(T+16位) + WalletFreeze冻结
  → 通过NATS通知finance开始审核

审核完成后finance回调我：
  通过 → WalletDeduct（最终扣除冻结金额）
  驳回 → WalletUnfreeze（解冻归还）
  出款失败 → WalletUnfreeze（解冻归还）

提现状态机（8种状态，最复杂的状态机）：
  1(审核中) → 2(风控通过)/3(风控驳回*)
  2 → 4(财务通过)/5(财务驳回*)
  4 → 6(出款中) → 7(出款成功*)/8(出款失败*)
  *号=终态，不可再变

提现账户管理：
  首次绑定：银行名称+账号+持有人（必须=KYC实名）
  不可修改（风控要求）
  每种方式+每种币种只能绑一个
```

## 十一、跨模块协作：与财务模块的联调设计

### 11.1 数据流向全景

```
资金入方向（充值）：
  C端 → gate-font → ser-wallet(创建订单)
  → ser-finance(选通道) → 三方支付 → 回调ser-finance
  → ser-finance RPC调 ser-wallet(WalletCredit入账)

资金出方向（提现）：
  C端 → gate-font → ser-wallet(创建订单+冻结)
  → NATS通知 → ser-finance(审核+代付) → 三方支付
  → 回调ser-finance → RPC调 ser-wallet(WalletDeduct或Unfreeze)

资金内转方向（兑换）：
  C端 → gate-font → ser-wallet(内部闭环，不涉及finance)
```

### 11.2 我提供的9个RPC接口

| 接口 | finance调用场景 | wallet执行动作 | 幂等保障 |
|------|---------------|-------------|---------|
| WalletCredit | 充值成功/补单 | 入中心钱包 | 流水唯一索引 |
| WalletCreditReward | 充值奖金/活动 | 入奖励钱包 | 流水唯一索引 |
| WalletFreeze | 提现创建 | 冻结余额 | 冻结记录状态检查 |
| WalletUnfreeze | 驳回/出款失败 | 解冻归还 | 冻结状态=冻结中才执行 |
| WalletDeduct | 出款成功 | 最终扣款 | 冻结状态=冻结中才执行 |
| WalletManualAdd | 人工加款审核通过 | 按子钱包分别加 | 流水唯一索引 |
| WalletManualSub | 人工减款审核通过 | 按子钱包分别减 | 流水唯一索引 |
| GetUserBalance | 加减款弹窗 | 返回各子钱包余额 | 纯查询 |
| GetCurrencyConfig | 金额格式化 | 返回精度和符号 | 纯查询 |

**所有资金变动类RPC都有幂等保障**，finance端超时重试是安全的。

### 11.3 RPC超时处理策略

```
finance 调 wallet.WalletCredit 超时时：
  → finance不知道wallet是否成功
  → 安全做法：直接重试（wallet有幂等保障）
  → wallet如果已处理 → 幂等拦截返回成功
  → wallet如果未处理 → 正常执行

这就是"本地事务 + 重试 + 幂等"模式的价值：
  不需要分布式事务框架
  不需要两阶段提交
  超时就重试，幂等保证不会重复执行
```

### 11.4 联调顺序建议

```
第一轮：GetCurrencyConfig + GetUserBalance
  → 最简单的查询接口，验证RPC链路是否通

第二轮：WalletCredit + WalletCreditReward
  → 充值入账链路，验证资金操作和幂等

第三轮：WalletFreeze + WalletUnfreeze + WalletDeduct
  → 提现冻结链路，验证状态机

第四轮：WalletManualAdd + WalletManualSub
  → 人工修正链路
```

## 十二、实现节奏：七阶段交付计划

### 12.1 七个Phase及产出

```
Phase 1  工程脚手架 + IDL + 数据库 + 代码生成
         ↓ 产出：ser-wallet能启动，RPC能通，表能建

Phase 2  币种配置模块（CRUD + 定时任务 + 对外查询能力）
         ↓ 产出：B端能配币种，汇率自动更新，RPC可查配置

Phase 3  子钱包余额体系（账户初始化 + 六种原子操作）
         ↓ 产出：子钱包能初始化，余额能增减冻结解冻

Phase 4  兑换子模块（最简单的业务闭环）
         ↓ 产出：法币→BSB兑换全流程通

Phase 5  充值子模块 + 入账RPC
         ↓ 产出：充值下单+USDT回调+WalletCredit通

Phase 6  提现子模块 + 冻结/解冻/扣款RPC
         ↓ 产出：提现下单+冻结+WalletUnfreeze/Deduct通

Phase 7  人工加减款RPC + 记录查询 + 联调收尾
         ↓ 产出：与财务模块全链路联调通过
```

### 12.2 为什么是这个顺序

```
Phase 1 → 2 → 3 是自底向上搭建基础设施：
  脚手架 → 币种配置（地基）→ 余额体系（心脏）

Phase 4 → 5 → 6 是按复杂度递增实现业务：
  兑换（内部闭环，验证余额体系）
  → 充值（对接三方，但没有冻结）
  → 提现（最复杂：冻结+审核+出款）

Phase 7 是收尾联调。

每个Phase都有明确的"验证标准"，做完一个就能验证一个，不会积压风险到最后。
```

### 12.3 依赖finance的时间点

```
Phase 1-4：不依赖finance，我可以独立完成
Phase 5：  依赖finance的"支付方式配置"和"创建支付请求"接口
Phase 6：  依赖finance调用我的Freeze/Unfreeze/Deduct
Phase 7：  依赖finance调用ManualAdd/Sub

策略：确定的先做，不确定的预留接口。
     Phase 1-4先做完，同时与finance对齐接口契约。
     Phase 5开始联调。
```

## 十三、风险识别与应对

### 13.1 技术风险矩阵

```
P0 致命（必须100%解决，已有方案）：
├── 并发余额正确性  → 乐观锁 + 有限重试
├── 重复入账         → 流水表唯一索引幂等
├── 金额精度丢失    → DECIMAL + string + decimal库
└── 提现双花         → 冻结机制 + 状态机守卫

P1 严重（已有方案，需要关注）：
├── 跨服务一致性    → 本地事务 + 重试 + 幂等 + 对账
├── 状态机非法跳转  → WHERE前置状态 + affected_rows检查
└── 对账不平         → 每日对账定时任务 + 告警

P2 重要（有预案）：
├── 热点账户并发    → 监控重试率，>5%考虑悲观锁
├── 汇率API不可用  → 降级策略（2家/1家可用时仍算，全挂跳过不更新）
├── 数据量增长      → TiDB水平扩展，索引设计覆盖主要查询
└── RPC超时         → wallet提供幂等接口，finance安全重试

P3 一般（可控）：
├── 定时任务竞争    → Redis分布式锁
└── 敏感数据保护    → 银行账号脱敏展示，操作日志审计
```

### 13.2 跨部门协作风险

```
风险：finance开发进度延迟 → 我的充值/提现功能无法真实联调

### 14.3 核心结论

```
一句话说清楚我打算怎么做：

  用乐观锁保证并发安全，
  用流水表唯一索引保证幂等，
  用DECIMAL+string保证金额精度，
  用冻结机制防止双花，
  用状态机守卫防止非法跳转，
  用四道防线层层兜底资金安全。

  自底向上分7个阶段实施：
  先搭脚手架和地基（币种+余额），
  再按复杂度递增做业务（兑换→充值→提现），
  最后联调收尾。

  确定的先做，不确定的预留接口。
  每一个设计决策都有对比方案和选择依据，不拍脑袋。
```
