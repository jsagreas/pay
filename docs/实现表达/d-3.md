## 一、我负责什么

我主要负责 **ser-wallet** 服务的设计与实现，也就是钱包多币种模块。这个模块包含三个部分：

| 部分 | 我的角色 | 说明 |
|------|---------|------|
| **钱包模块** | 主责 | 账户余额、充值、提现、兑换、账单记录 |
| **币种配置模块** | 主责 | 币种管理、汇率管理、基准币种 |
| **财务管理模块** | 联调配合 | 别人负责开发，但涉及充值回调、提现审核回调、出款回调等深度联调 |

在整个系统中，钱包模块处于**资金链路的核心位置**——用户的每一笔钱进来（充值）、出去（提现）、在平台内流转（兑换、投注、购买），都要经过钱包的余额变更。所以这个模块对**数据准确性和安全性**的要求是最高的。

## 二、需求全貌——我要实现什么

### 2.1 一句话概括

> 实现一个支持多币种（法币 VND/IDR/THB + USDT + 平台虚拟币 BSB）的钱包系统，涵盖用户资产管理、充值入金、提现出金、币种兑换四大核心能力，同时提供 B 端管理后台和面向其他服务的 RPC 接口。

### 2.2 用户视角——C 端做了什么

站在用户的角度，整个钱包模块的使用路径是这样的：

```
用户进入 APP
    │
    ├── 充值中心 ──→ 选币种 → 选金额/输金额 → 选奖金活动(可选) → 选支付方式 → 支付 → 到账
    │
    ├── 我的钱包 ──→ 查看各币种余额 → 查看资产总览 → 切换币种联动刷新
    │
    ├── 兑换 ────→ 选法币币种 → 输金额 → 预览汇率和到账BSB → 确认兑换 → 即时到账
    │
    ├── 提现 ────→ KYC 校验 → 选提现方式 → 选/新建提现账户 → 输金额 → 提交 → 等审核 → 到账
    │
    └── 账单 ────→ 四个Tab（充值/提现/兑换/奖励）→ 列表 → 详情
```

### 2.3 管理视角——B 端做了什么

```
运营/财务进入管理后台
    │
    ├── 币种管理 ──→ 币种的增删改查、启用/禁用
    │
    ├── 汇率管理 ──→ 查看实时/平台汇率、汇率浮动配置、汇率变更日志
    │
    ├── 基准币种 ──→ 设置全局基准币种（设置后不可更改）
    │
    ├── 用户钱包 ──→ 查询用户余额（由财务模块关联展示）
    │
    └── 人工加减款 ──→ 管理员手动调账（走审批流程，由财务模块负责）
```

### 2.4 系统视角——RPC 供给什么

钱包不只服务于前端，还要被其他后端模块调用：

| 调用方 | 调用场景 | 钱包提供的能力 |
|--------|---------|--------------|
| 财务模块 | 充值到账通知 | `WalletCredit` — 入账到中心钱包 |
| 财务模块 | 提现发起 | `WalletFreeze` — 冻结可用余额 |
| 财务模块 | 提现出款成功 | `WalletDebit` — 从冻结金额确认扣除 |
| 财务模块 | 审核驳回/出款失败 | `WalletUnfreeze` — 解冻退回可用 |
| 财务模块 | 人工加减款 | `ManualCredit / ManualDebit` — 管理员手动调账 |
| 投注系统 | 下注扣款 | `BetDebit` — 优先级扣款（中心优先，奖励补充） |
| 结算/返奖 | 用户赢了给钱 | `BetCredit` — 按投注时比例分回中心/奖励钱包 |
| 活动系统 | 发放奖励 | `WalletCredit` — 入账到奖励钱包 |
| 直播模块 | 打赏消费 | `Debit` — 扣中心钱包 |
| 直播模块 | 主播收入 | `WalletCredit` — 入账到主播钱包 |

### 2.5 五种钱包类型

需求文档定义了五种钱包类型，资金在不同类型之间有严格的流转规则：

| 钱包类型 | 用途 | 资金来源 | 提现规则 | 当前状态 |
|---------|------|---------|---------|---------|
| 中心钱包 | 用户主钱包 | 充值、稽核完成的奖励金转入 | 完成稽核后可提现 | 需实现 |
| 奖励钱包 | 活动/投注奖金暂存 | 活动返奖、投注返奖 | 需完成稽核后转入中心钱包 | 需实现 |
| 主播钱包 | 主播收益独立管理 | 礼物打赏收入 | 可直接提现或转入中心钱包 | 需实现 |
| 代理钱包 | 代理佣金管理 | 代理分成 | 可直接提现或转入中心钱包 | 预留 |
| 场馆钱包 | 游戏场馆内临时钱包 | 进入场馆时从中心+奖励转入 | 退出场馆时原路转回 | 预留 |

**关键规则**：
- 消费优先级：先扣中心钱包，不足部分再扣奖励钱包
- 混合投注返奖：按投注时中心/奖励钱包的金额占比，分别返奖到对应钱包
- 奖励钱包提现：需完成指定倍数的稽核流水后，才能转入中心钱包提现

### 2.6 三种币种类型

| 类型 | 代表 | 特点 |
|------|------|------|
| 法定货币 | VND（越南盾）、IDR（印尼盾）、THB（泰铢） | 对应真实国家货币，用户本地充值/提现 |
| 加密货币 | USDT | 区块链数字资产，跨境通用 |
| 平台货币 | BSB | 平台专属虚拟币，锚定 USDT（10BSB = 1USDT），统一游戏场馆币种 |

### 2.7 汇率体系

| 概念 | 定义 | 用途 |
|------|------|------|
| 实时汇率 | 接入 3 家三方汇率 API，取平均值 | 作为计算基础 |
| 平台汇率 | 初始值=实时汇率，按阈值规则自动更新 | 统计报表、对账 |
| 入款汇率 | 实时汇率 ×（1 + 汇率浮动%） | 充值、进入游戏场馆 |
| 出款汇率 | 实时汇率 ×（1 - 汇率浮动%） | 提现、退出游戏场馆 |

**关键架构决策**：

| 决策 | 选择 | 理由 |
|------|------|------|
| 币种配置是否独立服务 | **否，同进程** | 每次资金操作都需查汇率/币种，同进程函数调用 ~0.001ms vs 独立 RPC ~3-10ms，性能差千倍 |
| 是否引入分布式事务框架 | **否** | 本地事务 + 幂等 + 补单已足够，Seata/DTM 引入大量复杂度且工程无先例 |
| 余额是否缓存到 Redis | **否** | 缓存与 DB 不一致会导致超扣或展示错误，资金场景不可接受。TiDB 主键查询 <1ms，性能足够 |
| 资金操作走 RPC 还是消息队列 | **同步 RPC** | 资金操作需要调用方实时知道成功/失败，消息队列需额外确认机制 |
| NATS JetStream 用在哪 | **事件通知** | 余额变动通知（wallet.balance.changed），不用于资金操作指令 |
| 金额内部表示用什么 | **DECIMAL(20,8) + shopspring/decimal** | 多币种最小单位不同，decimal 语义清晰，避免 int64 换算错误 |

**分层职责边界**：

| 层 | 允许做什么 | 禁止做什么 |
|----|----------|----------|
| handler.go | 一行委托 `return s.xxxService.Method(ctx, req)` | 任何业务逻辑、参数处理 |

| cache/ | Redis 读写、缓存策略 | DB 读写、业务逻辑 |
| gorm_gen/ | 标准 CRUD（自动生成，勿手动修改） | — |

### 3.3 为什么 account_ser 是核心

这是整个模块最关键的设计决策。业界资金系统的标准做法是将**账户操作（余额增减）**从具体业务（充值/提现/投注）中抽离出来，形成独立的核心层：

```
                   ┌─────────────────────┐
                   │    account_ser.go   │  ← 账户核心（资金原子操作）
                   │                     │
                   │  Credit()           │  ← 通用入账
                   │  Debit()            │  ← 通用扣款
                   │  Freeze()           │  ← 冻结
                   │  Unfreeze()         │  ← 解冻
                   │  DebitFrozen()      │  ← 从冻结中扣除
                   │  BetDebit()         │  ← 优先级扣款（专用）
                   │  BetCredit()        │  ← 按比例返奖（专用）
                   │  GetBalance()       │  ← 余额查询
                   └──────────┬──────────┘
                              │ 被以下 Service 调用
          ┌───────────────────┼───────────────────┐
          │                   │                   │
    deposit_ser          withdraw_ser        exchange_ser
    充值成功 → Credit    提现冻结 → Freeze     法币扣 → Debit
                         驳回 → Unfreeze      BSB加 → Credit
                         出款成功 → DebitFrozen

    ┌──────────────────────────────────────────────────┐
    │  外部 RPC 调用方也通过 account_ser 操作余额          │
    │                                                   │
    │  财务 → WalletCredit    = Credit(中心钱包)         │
    │  财务 → WalletFreeze    = Freeze(中心钱包)         │
    │  游戏 → BetDebit        = BetDebit(优先级扣款)     │
    │  直播 → GiftDebit       = Debit(中心钱包)          │
    │  活动 → BonusCredit     = Credit(奖励钱包)         │
    └──────────────────────────────────────────────────┘
```

**核心原则**：
- account_ser 是**唯一能修改余额的地方**，任何路径的资金变动都必须经过它
- 它不关心"为什么变" — 由调用方通过 `biz_type` 和 `order_no` 告知
- 它只关心"变多少、从哪个钱包变、是否幂等"

这是业界支付系统的标准分层——支付宝/微信支付/Stripe 都是类似模式。

## 四、资金安全——这个模块最核心的问题

资金系统的架构质量 = 安全机制的严密程度。我设计了**五层安全防线**，每一笔资金操作必须穿过全部五层才能成功：

```
┌─────────────────────────────────────────────────────────────────┐
│                        五层安全模型                                │
│                                                                  │
│  L1  参数层: 入参校验                                            │
│       · 金额 > 0                                                 │
│       · 币种有效 + 已启用                                         │
│       · 钱包类型合法                                              │
│       · biz_type + order_no 非空                                 │
│                                                                  │
│  L2  防重层: Redis SetNX + 幂等查询                              │
│       · SetNX(key=userId:action:orderNo, TTL=10s) 防用户双击      │
│       · 查 wallet_transaction 是否已存在同一业务流水               │
│       · 已存在 → 直接返回成功（幂等），不重复执行                   │
│                                                                  │
│  L3  事务层: FOR UPDATE 悲观行锁                                 │
│       · SELECT ... FOR UPDATE 锁定目标钱包行                      │
│       · 其他并发请求排队等待                                      │
│       · 在锁内校验余额是否充足                                    │
│       · 更新余额 + 写入流水，同一个 DB 事务                       │
│                                                                  │
│  L4  乐观锁: version 字段                                        │
│       · UPDATE ... SET version = version+1 WHERE version = ?     │
│       · 若 version 不匹配（理论上不可能，安全网）→ 回滚重试        │
│                                                                  │
│  L5  数据层: 唯一索引                                            │
│       · uk_biz_order (biz_type, order_no, wallet_type)           │
│       · DB 层面拒绝重复流水，最终兜底                              │
│                                                                  │
│  任何一笔资金操作必须穿过全部5层才能成功                           │
└─────────────────────────────────────────────────────────────────┘
```

### 4.1 标准事务模型（每一次余额变动都走这个流程）

```
account_ser.Credit(ctx, req) / Debit / Freeze / Unfreeze / DebitFrozen
  │
  ▼
Step 1: 幂等校验（查流水表）
  SELECT id FROM wallet_transaction
  WHERE biz_type = ? AND order_no = ? AND wallet_type = ? LIMIT 1
  → 存在: 直接返回成功（幂等，不重复执行）
  → 不存在: 进入事务
  │
  ▼
Step 2: 开启 DB 事务
  │
  ▼
Step 3: 行级锁定目标钱包
  SELECT balance, frozen, version FROM wallet
  WHERE user_id=? AND currency_code=? AND wallet_type=? FOR UPDATE
  → 不存在: 自动创建钱包记录（懒初始化）
  │
  ▼
Step 4: 业务校验
  · Credit: amount > 0 ✓
  · Debit:  balance >= amount ?  ✓ : 返回余额不足
  · Freeze: balance - frozen >= amount ? ✓ : 可用余额不足
  │
  ▼
Step 5: 更新余额 + version
  UPDATE wallet SET balance=?, frozen=?, version=version+1
  WHERE id=? AND version=?
  │
  ▼
Step 6: 写入流水（同一事务内）
  INSERT INTO wallet_transaction (biz_type, order_no, amount, balance_before, balance_after, ...)
  → 唯一索引 uk_biz_order 兜底防重
  │
  ▼
Step 7: COMMIT
```

**为什么用悲观锁而不是纯乐观锁**：
- 钱包余额是高争抢资源（充值、投注、提现可能同时操作同一个账户）
- 乐观锁在高冲突场景下需要反复重试，反而更慢
- 悲观锁配合短事务（毫秒级），锁等待时间极短
- 业界支付系统标准做法

## 五、核心流程——关键场景怎么走

### 5.1 充值流程

```
用户选择充值
    │
    ▼
ser-wallet: 校验币种+金额 → 调 ser-finance RPC 获取充值方式
    │
    ▼
用户选择支付方式 → ser-wallet 调 ser-finance 创建支付订单
    │
    ▼
ser-finance: 对接三方通道 → 返回支付链接 → 用户完成支付
    │
    ▼
三方通道支付成功 → 回调 ser-finance → ser-finance 调 ser-wallet WalletCredit
    │
    ▼
ser-wallet: Credit(中心钱包, biz_type=deposit, order_no=充值订单号)
    → 走标准事务模型 → 余额增加 → 写入流水 → 返回成功
```

**三层保障**：
1. **回调** — 支付成功后正常回调入账
2. **主动查询** — 定时扫描 pending 订单，主动向支付服务查询结果
3. **人工补单** — B 端提供补单入口，最终兜底

### 5.2 提现流程

```
用户发起提现
    │
    ▼
ser-wallet: KYC 校验（调 ser-kyc）→ 校验余额、限额
    │
    ▼
ser-wallet: Freeze(中心钱包, amount)
    available: 1000 → 900
    frozen:       0 → 100
    → 总余额不变，"可用"转为"冻结"
    │
    ▼
ser-wallet: 调 ser-finance 创建提现订单 → 进入审核流程
    │
    ├── 审核通过 + 出款成功 → ser-finance 回调 WalletDebit
    │     frozen: 100 → 0（冻结金额确认扣除）
    │
    └── 审核驳回 / 出款失败 → ser-finance 回调 WalletUnfreeze
          available: 900 → 1000
          frozen:    100 → 0（冻结退回可用）
```

**状态机**：
```
待审核 ──→ 审核通过 ──→ 处理中 ──→ 成功
  │                       │
  ├──→ 驳回               └──→ 失败（解冻退回）
```

每个状态转换使用**条件更新**（`WHERE status = 当前状态`），防止并发导致状态跳跃。

### 5.3 兑换流程

```
用户请求兑换预览：
    输入：100000 VND → BSB
    后端查汇率 + 入款浮动：
    返回预览：100000 VND → XX BSB（有效期内）

用户确认兑换（同一个 DB 事务内完成）：
    ① 校验汇率是否仍在有效期内且偏差在阈值内
    ② SELECT FOR UPDATE 锁定源钱包（VND）和目标钱包（BSB）
       → 有序加锁：按 (wallet_type, currency_code) 升序，防死锁
    ③ 源钱包扣款：balance -= 扣除金额
    ④ 目标钱包加款：balance += 到账金额
    ⑤ 写入两条 wallet_transaction（一扣一加）
    ⑥ 写入 exchange_order（含汇率快照）
    COMMIT
```

**兑换的关键点**：
- 两个钱包在同一个数据库同一个事务内，本地事务保证原子性，不需要分布式事务
- 汇率快照：兑换记录中存成交时刻的汇率，不是实时汇率，保证历史可追溯
- 防死锁：所有请求按固定顺序加锁

### 5.4 投注扣款（优先级扣款）

```
游戏模块调用 BetDebit(userId, currencyCode, amount, orderNo)

ser-wallet 处理：
    ① 查用户的中心钱包 + 奖励钱包余额
    ② 计算扣款分配：
       中心钱包可用 >= amount → 全扣中心
       中心钱包可用 < amount  → 中心扣完，差额从奖励钱包扣
       两个加起来 < amount   → 余额不足，拒绝
    ③ 在同一事务内 FOR UPDATE 锁定两个钱包 → 扣款 → 写流水
    ④ 返回实际扣款比例（中心:奖励），用于后续返奖

返奖时 BetCredit 按原始比例分回对应钱包
```

## 六、数据设计——表怎么建

### 6.1 表的分类和优先级

一共设计了 **14 张表**，按必要性分三档：

| 优先级 | 数量 | 表名 | 说明 |
|--------|------|------|------|
| **必须（Must）** | 6 | currency, currency_rate, base_currency, wallet, wallet_transaction, deposit_order | 缺少任何一张，核心功能不可用 |
| **应该（Should）** | 5 | withdraw_order, exchange_order, withdraw_account, currency_rate_log, reward_record | 缺少则功能不完整或缺少审计能力 |
| **可选（Optional）** | 3 | currency_display, wallet_snapshot, repeat_deposit_log | 增强型，首版可不做 |

### 6.2 最核心的两张表

**wallet — 钱包余额主表（全系统最核心）**：

```sql
CREATE TABLE wallet (
    id            BIGINT       NOT NULL AUTO_INCREMENT,
    user_id       BIGINT       NOT NULL COMMENT '用户ID',
    currency_code VARCHAR(10)  NOT NULL COMMENT '币种编码(VND/IDR/THB/USDT/BSB)',
    wallet_type   TINYINT      NOT NULL COMMENT '钱包类型(1中心/2奖励/3主播/4代理/5场馆)',
    balance       DECIMAL(20,8) NOT NULL DEFAULT 0 COMMENT '可用余额',
    frozen        DECIMAL(20,8) NOT NULL DEFAULT 0 COMMENT '冻结金额',
    version       INT          NOT NULL DEFAULT 0 COMMENT '乐观锁版本号',
    ...
    UNIQUE KEY uk_user_currency_type (user_id, currency_code, wallet_type, delete_at)
) COMMENT '钱包余额主表';
```

**wallet_transaction — 资金流水表（对账核心，只增不改不删）**：

```sql
CREATE TABLE wallet_transaction (
    id              BIGINT        NOT NULL AUTO_INCREMENT,
    user_id         BIGINT        NOT NULL,
    currency_code   VARCHAR(10)   NOT NULL,
    wallet_type     TINYINT       NOT NULL,
    direction       TINYINT       NOT NULL COMMENT '方向(1收入/2支出)',
    amount          DECIMAL(20,8) NOT NULL COMMENT '变动金额(正数)',
    balance_before  DECIMAL(20,8) NOT NULL COMMENT '变动前余额',
    balance_after   DECIMAL(20,8) NOT NULL COMMENT '变动后余额',
    frozen_before   DECIMAL(20,8) NOT NULL DEFAULT 0,
    frozen_after    DECIMAL(20,8) NOT NULL DEFAULT 0,
    biz_type        VARCHAR(32)   NOT NULL COMMENT '业务类型(deposit/withdraw/exchange/bet/...)',
    order_no        VARCHAR(64)   NOT NULL COMMENT '业务订单号(幂等键)',
    ...
    UNIQUE KEY uk_biz_order (biz_type, order_no, wallet_type)
) COMMENT '资金流水表';
```

**两张表的设计逻辑**：
- `wallet` 存的是"当前快照"——用户此刻有多少钱
- `wallet_transaction` 存的是"变更历史"——每一笔钱的来龙去脉
- 通过 `balance_before / balance_after` 关联：流水最新一条的 `balance_after` 应等于 `wallet.balance`
- 如果不等，说明有异常——这就是**对账的基础**

### 6.3 金额精度架构

这是与常规 int64 做法不同的重要决策点：

```
┌──────────────────────────────────────────────────────────────┐
│                      金额精度分层策略                          │
│                                                               │
│  RPC 接口层 (Thrift IDL)                                     │
│    · 金额字段类型: string                                      │
│    · 原因: Thrift 无 decimal 类型，string 无精度损失            │
│    · 示例: "1000000.00", "0.00000001"                         │
│                                                               │
│  Service 层 (Go)                                              │
│    · 金额运算: shopspring/decimal                              │
│    · 所有金额接收后立即 decimal.NewFromString() 转换            │
│    · 所有运算用 decimal.Add/Sub/Mul/Div                        │
│    · 禁止 float64 参与任何金额计算                              │
│                                                               │
│  数据库层 (TiDB)                                               │
│    · 字段类型: DECIMAL(20,8)                                   │
│    · USDT 需 8 位小数，法币整数只是展示层截断，存储层统一        │
│                                                               │
│  展示层 (响应给前端)                                            │
│    · 按币种配置格式化:                                          │
│      VND/IDR → 整数                                            │
│      THB/BSB → 2 位小数                                        │
│      USDT → 8 位小数                                           │
│    · 格式化在 Service 层完成，返回给网关已是格式化后的 string     │
└──────────────────────────────────────────────────────────────┘
```

**为什么不用 int64 最小单位**：
- 多币种系统中，不同币种的最小单位不同（VND 无小数、USDT 8 位小数）
- 如果用 int64，VND 的 1 = 1，USDT 的 1 = 0.00000001，换算极易出错
- decimal 语义清晰：1.00 就是 1，0.00000001 就是 0.00000001
- shopspring/decimal 是 Go 生态中最成熟的 decimal 库

## 七、跨服务通信——钱包与财务怎么协作

### 7.1 通信模型

```
┌────────────────────────────────────────────────────────────────┐
│               钱包 ◄═══════► 财务  双向 RPC 通信                 │
│                                                                 │
│  钱包→财务 (C端用户主动操作):                                    │
│    · GetDepositMethods     获取充值方式列表                       │
│    · CreateDepositOrder    发起充值下单                           │
│    · GetOrderStatus        查询订单状态                           │
│    · GetWithdrawMethods    获取提现方式列表                       │
│    · CreateWithdrawOrder   发起提现下单                           │
│    超时: 3秒 (项目统一 RPC 超时配置)                               │
│                                                                 │
│  财务→钱包 (回调/审核结果):                                      │
│    · WalletCredit          充值入账                               │
│    · WalletFreeze          提现冻结                               │
│    · WalletUnfreeze        审核驳回解冻                           │
│    · WalletDebit           出款成功扣除                           │
│    · ManualCredit/Debit    人工加减款                              │
│    · GetWalletInfo         查询钱包信息                           │
│    超时: 3秒  |  幂等: 全部支持 (order_no)                        │
│                                                                 │
│  NATS JetStream (辅助通信):                                      │
│    · 钱包发布 wallet.balance.changed 事件（余额变动通知）          │
│    · 订阅方: 活动服务、推送服务、统计服务                          │
│    · 注意: 这是辅助通道，不是资金操作的主通道                      │
└────────────────────────────────────────────────────────────────┘
```

**为什么资金操作用同步 RPC，不用消息队列**：
- 资金操作需要实时确认（调用方必须知道入账/扣款是否成功）
- 消息队列的确认机制在资金场景下反而增加复杂度
- 业界标准：同步保安全，异步做扩展

### 7.2 RPC 调用失败处理

```
策略 1: 调用方重试（财务侧）
  · 同一 order_no 重试 → 钱包幂等保障安全
  · 建议: 立即重试 1 次 → 间隔 3 秒重试 1 次 → 放弃，转人工

策略 2: 补单兜底（人工）
  · 财务后台的"充值补单"功能
  · 管理员确认后，调钱包 WalletCredit
  · order_no 保持不变 → 幂等安全

不采用的策略:
  ✗ 分布式事务 (TCC/Saga) → 过度设计，单 DB 事务已足够
  ✗ 消息补偿 → 增加复杂度，补单功能已覆盖
```

## 八、技术挑战——难点怎么解决

我识别了 **20 个技术挑战**，按严重程度分级。这里挑最关键的几个说明：

### 挑战一（P0）：并发余额操作的一致性

**问题**：用户同时充值和提现，两个请求并发修改同一行余额，可能导致数据错乱。

**方案**：`SELECT ... FOR UPDATE` 悲观行锁 + version 乐观锁双重保障。事务控制在毫秒级，锁等待时间极短。行级锁只锁目标行不锁表，不影响其他用户。

### 挑战二（P0）：金额精度

**问题**：`0.1 + 0.2 ≠ 0.3` 不是笑话，是真实的线上事故。多币种精度不同更复杂。

**方案**：三层统一 —— 存储 DECIMAL(20,8) + 运算 shopspring/decimal + 传输 string。禁止 float64 参与任何金额计算。

### 挑战三（P0）：幂等性

**问题**：支付回调可能因网络重试发送多次，如果不做幂等，用户会多收到钱。

**方案**：两层幂等 —— Redis SetNX 快速拦截 + wallet_transaction 表唯一索引 `(biz_type, order_no, wallet_type)` 最终兜底。数据库唯一索引是永久的，不像 Redis key 会过期。

### 挑战四（P0）：防超扣

**问题**：余额 100，两个并发请求各扣 80，如果不加保护可能余额变成 -60。

**方案**：FOR UPDATE 锁住行后再检查余额 —— 第一个请求锁定、校验、扣款成功；第二个请求等到锁释放后，读到的余额已是 20，校验失败返回余额不足。

### 挑战五（P1）：兑换死锁预防

**问题**：用户 A 兑换 VND→BSB，用户 B 兑换 BSB→VND，可能形成交叉死锁。

**方案**：所有兑换操作按 `(wallet_type, currency_code)` 升序加锁，保证全局加锁顺序一致。

### 挑战六（P1）：跨服务数据一致性

**问题**：钱包与财务是两个独立服务，充值回调丢失怎么办？

**方案**：三板斧 —— ① 幂等 RPC（同一 order_no 重复调用安全） ② 超时重试（财务侧按退避策略重试） ③ 补单兜底（人工查确认后补单）。这也是支付宝、微信支付处理跨系统一致性的主流做法。

### 挑战全景对照表

| 级别 | 挑战 | 方案关键词 |
|------|------|----------|
| **P0 致命** | 并发余额 | FOR UPDATE + version 双保障 |
| **P0 致命** | 金额精度 | DECIMAL(20,8) + shopspring/decimal |
| **P0 致命** | 幂等性 | Redis SetNX + DB 唯一索引 |
| **P0 致命** | 防超扣 | FOR UPDATE 内校验 |
| **P0 致命** | 冻结原子性 | 同一事务内 balance→frozen |
| **P1 严重** | 跨服务一致 | 幂等 RPC + 重试 + 补单 |
| **P1 严重** | 缓存一致 | 币种/汇率 Cache-Aside，余额不缓存 |
| **P1 严重** | 死锁预防 | 全局有序加锁 |
| **P1 严重** | 回调可靠性 | 回调 + 主动查询 + 人工补单三层 |
| **P1 严重** | 优先级扣款 | 同事务双钱包顺序扣 |
| **P2 重要** | 热点账户 | 行级锁 + 短事务，初期够用 |
| **P2 重要** | 流水表膨胀 | 索引覆盖 + TiDB 自动分裂 |
| **P2 重要** | 汇率精度 | decimal 全链路 + 快照存储 |
| **P3 一般** | 订单号唯一 | Snowflake（复用工程已有） |
| **P3 一般** | 防重提交 | Redis SetNX 10 秒窗口 |
| **P3 一般** | 敏感数据 | AES-256 EncryptedString（复用已有） |

## 十、开发计划——分几步走

```
Phase 0: 工程骨架 + 基础设施
         IDL 定义 → gen_kitex.sh 代码生成 → 服务注册（ETCD）
         DB 建表 → GORM Gen 生成 → Redis/NATS 初始化
         预计：起步阶段
         ↓
Phase 1: 币种配置（L0 底座，无外部依赖，最独立）
         B 端 CRUD（币种增删改查、汇率管理、基准币种）
         RPC 暴露（供钱包和其他服务使用）
         定时任务：三方汇率同步 + 阈值判断自动更新平台汇率
         ↓
Phase 2: 钱包核心能力（RPC 暴露层）
         account_ser: Credit / Debit / Freeze / Unfreeze / DebitFrozen / GetBalance
         这是整个模块的心脏，做好之后上层业务全部依赖它
         ↓
Phase 3: 兑换模块（最独立，可单独测试）
         法币 → BSB 兑换，纯进程内事务，不依赖外部服务
         预览汇率 → 确认兑换 → 原子扣加
         ↓
Phase 4: 充值 + 提现（依赖 ser-finance 联调）
         C 端全链路：充值方式 → 下单 → 回调入账 / 提现 → 冻结 → 审核 → 出款/驳回
         需与财务开发者先对齐 RPC 契约（IDL）
         ↓
Phase 5: 扩展 RPC + 记录查询 + B 端管理
         投注扣款/返奖 RPC、礼物消费 RPC、活动奖励 RPC
         四类记录查询（充值/提现/兑换/奖励）
         B 端用户钱包查询
         ↓
Phase 6: NATS 事件通知 + 补充
         wallet.balance.changed 事件发布
         监控指标、日志完善
```

**为什么这个顺序**：
1. Phase 0-1 无任何外部依赖，可以完全独立开发和测试
2. Phase 2 是核心，做好后所有上层业务都是"调用 account_ser 的方法"
3. Phase 3 兑换纯进程内操作，可以独立验证事务逻辑
4. Phase 4 充值/提现需要联调，放在后面给财务模块留准备时间
5. Phase 5-6 是扩展和完善

## 十一、跨团队协作——需要什么配合

### 11.1 与财务模块开发者的协作

这是最重要的协作关系。钱包↔财务之间有 **12 个双向 RPC 接口**：

| 方向 | 接口 | 说明 |
|------|------|------|
| 钱包→财务 | GetDepositMethods | 获取充值方式列表 |
| 钱包→财务 | CreateDepositOrder | 创建充值订单 |
| 钱包→财务 | GetOrderStatus | 查询订单状态 |
| 钱包→财务 | GetWithdrawMethods | 获取提现方式列表 |
| 钱包→财务 | CreateWithdrawOrder | 创建提现订单 |
| 财务→钱包 | WalletCredit | 充值/补单入账 |
| 财务→钱包 | WalletFreeze | 提现冻结 |
| 财务→钱包 | WalletUnfreeze | 审核驳回解冻 |
| 财务→钱包 | WalletDebit | 出款成功扣除冻结 |
| 财务→钱包 | ManualCredit | 人工加款 |
| 财务→钱包 | ManualDebit | 人工减款 |
| 财务→钱包 | GetWalletInfo | 查询钱包信息 |

**协作节奏建议**：
1. **Phase 1 期间**：双方先对齐 IDL 契约（接口定义、字段、错误码范围），各自基于 IDL 独立开发
2. **Phase 2 完成后**：钱包侧 RPC 接口可用，财务侧可以开始对接测试
3. **Phase 4 开始**：正式联调充值/提现全链路

### 11.2 错误码范围分配

按工程既有规范（服务编号 × 1000），ser-wallet 错误码范围：**6022000-6022999**。

## 十二、风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 财务模块进度滞后 | 充值/提现无法联调 | Phase 1-3 完全独立，不受影响；Phase 4 可基于 Mock 先开发 |
| 需求评审后细节调整 | 表结构/接口需要改 | 架构骨架已确定（5层安全、account_ser核心层），细节调整影响有限 |
| 三方汇率 API 不稳定 | 汇率无法更新 | 接入 3 家取平均值，2/3 可用即正常；全挂则沿用最近平台汇率 |
| 数据量增长超预期 | 流水表查询变慢 | TiDB 自动分裂 + 按时间分区索引，初期不需要分表 |
| shopspring/decimal 引入风险 | 新依赖 | 百万级 star 的成熟库，Go 生态标准，Stripe 也在用 |

## 十三、总结

### 一句话架构

> **单服务（ser-wallet）内的分层架构，DB 本地事务保证资金安全，同步 RPC + 幂等实现跨服务协作，Redis 做配置缓存和防重，NATS 做事件通知。不引入分布式事务、不缓存余额、不过度拆分服务。**

### 关键数字

| 指标 | 数值 |
|------|------|
| 服务数 | 1（ser-wallet，币种配置同进程） |
| 数据库 | 1（wallet_db，TiDB 独立库） |
| 核心表 | 14 张（Must 6 + Should 5 + Optional 3） |
| 接口 | ~69 个（C端21 + B端11 + RPC暴露24 + RPC调用13） |
| 安全层数 | 5 层（参数校验 → 幂等 → 行锁 → 乐观锁 → 唯一索引） |
| 技术挑战 | 20 个已识别，全部有成熟方案 |
| 新增依赖 | 1 个（shopspring/decimal） |
| 开发阶段 | 6 个 Phase，Phase 0-3 可独立完成 |

### 我的信心来源

1. **需求理解到位**：120 张产品原型和需求文档逐页分析，5 种钱包类型、3 类币种、4 种汇率概念全部理清
2. **设计方案成熟**：核心设计（悲观锁 + 幂等 + 本地事务 + 补单）是支付宝/微信支付/Stripe 的行业标准做法
3. **工程约束明确**：完全复用现有技术栈（Kitex/GORM/Redis/ETCD/NATS），只新增一个经过充分验证的 decimal 库
4. **风险可控**：Phase 0-3 无外部依赖可独立推进，最大风险（联调）放在后半段，留有充足缓冲
5. **关键难点已有方案**：20 个技术挑战逐一分析，每个都有业界成熟解法和代码级实现思路
