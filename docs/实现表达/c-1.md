## 一、开场：我负责什么

各位好，我负责的是 **ser-wallet（022号服务）—— 多币种钱包模块**的后端设计和实现。

这个模块在业务上承担什么角色？一句话概括：

> **钱包是整个平台的"账本"。用户的钱进来、出去、在内部流转，所有余额变动都由钱包来记录和管理。**

具体来说我的职责范围分三块：

| 板块 | 职责 | 角色 |
|------|------|------|
| **钱包模块** | 用户余额管理、充值/提现/兑换订单、交易流水、提现账户 | **主责，独立开发** |
| **币种配置模块** | 币种参数管理、汇率维护、B端配置页 | **主责，钱包子模块** |
| **财务管理模块** | 支付通道、风控/财务审核、出款执行 | **别人负责，我需要理解并联调** |

三者的关系可以这样理解：
- **钱包 = 记账系统**（管账户余额、管流水）
- **财务 = 资金系统**（管支付通道、管钱的进出）
- **币种配置 = 参数系统**（管汇率、管币种规则）

钱包不直接对接支付通道，财务不直接操作用户余额——各管各的，通过 RPC 协作。

## 二、需求全景：这个模块要做什么

### 2.1 核心业务场景

用户打开钱包，能做四件事：

```
┌─────────────────────────────────────────────────┐
│                    钱包首页                        │
│                                                   │
│   VND 余额: 1,000,000    USDT 余额: 50.00       │
│   BSB 余额: 200.00                               │
│                                                   │
│   ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐       │
│   │ 充值  │  │ 提现  │  │ 兑换  │  │ 记录  │       │
│   └──────┘  └──────┘  └──────┘  └──────┘       │
└─────────────────────────────────────────────────┘
```

**充值**：用户把外部的钱（法币/USDT）充到平台账户
**提现**：用户把平台账户的钱提到外部（银行卡/USDT地址）
**兑换**：用户在平台内部把法币换成平台币（BSB）
**记录**：查看所有历史交易明细

### 2.2 多币种体系

平台支持5种币种，每种币种是一个独立的"世界"：

| 币种 | 类型 | 说明 | 精度 |
|------|------|------|------|
| VND | 法币 | 越南盾 | 整数 |
| IDR | 法币 | 印尼盾 | 整数 |
| THB | 法币 | 泰铢 | 2位小数 |
| USDT | 加密货币 | 稳定币 | 8位小数 |
| BSB | 平台币 | 平台专属（10 BSB = 1 USDT） | 2位小数 |

用户切换币种 = 切换视图，每个币种下有自己的余额、充值记录、提现记录。

### 2.3 五种钱包类型

每个用户在每个币种下，最多有5种钱包：

| 钱包类型 | 定位 | 资金来源 | 第一版 |
|---------|------|---------|--------|
| 中心钱包 | 主钱包 | 充值入账、兑换所得 | 做 |
| 奖励钱包 | 奖金暂存 | 活动奖金、兑换赠送 | 做 |
| 主播钱包 | 主播收入 | 打赏分成 | 做 |
| 代理钱包 | 代理佣金 | 分佣 | 预留 |
| 场馆钱包 | 场馆结算 | 结算 | 预留 |

**核心规则**：
- 消费时先扣中心钱包，不够再扣奖励钱包
- 奖励钱包的钱需要完成流水倍数要求后才能转入中心
- 返奖时按消费时中心和奖励的占比分别回补

### 2.4 接口规模

经过完整梳理，ser-wallet 总共需要实现 **42个接口**：

| 分类 | 数量 | 说明 |
|------|------|------|
| C端接口 | 22 | 用户直接使用（充值/提现/兑换/记录/概览） |
| B端接口 | 5 | 运营后台（币种管理、汇率日志） |
| RPC对外接口 | 15 | 供财务/投注等其他服务调用 |
| **合计** | **42** | |

对应 **10张数据库表**、**7个IDL文件**。

## 三、技术选型：我用什么来做

### 3.1 一个核心结论

> **工程基础设施成熟度极高，钱包模块只需要新增1个依赖。**

在分析了 `common/pkg/` 下所有已封装组件后，钱包需要的20个技术组件中，**19个已有**，只需要引入 `shopspring/decimal` 做金额精确计算。

### 3.2 技术栈全景

| 层级 | 技术 | 状态 | 钱包怎么用 |
|------|------|------|-----------|
| HTTP网关 | Hertz | 已有 | gate-font(C端) + gate-back(B端) 路由转发 |
| RPC框架 | Kitex + Thrift | 已有 | 服务间通信（接收请求+调用财务/KYC） |
| ORM | GORM + gen | 已有 | 类型安全的数据库操作，编译期校验 |
| 数据库 | MySQL | 已有 | 全部业务数据存储，DECIMAL精度存金额 |
| 缓存 | Redis | 已有 | 汇率缓存、币种配置缓存、防重复提交 |
| 配置中心 | ETCD | 已有 | 服务配置、注册发现 |
| 消息队列 | Kafka | 已有 | 余额变更通知、异步事件 |
| 定时任务 | Cron + XXL-JOB | 已有 | 汇率刷新、订单超时、日终对账 |
| 分布式ID | 雪花算法 | 已有 | 订单号、流水号生成 |
| 加密 | AES-256 | 已有 | 提现账户（银行卡号/USDT地址）加密存储 |
| 日志+追踪 | Zap + OpenTelemetry | 已有 | 全链路日志、TraceID贯穿 |
| **金额计算** | **shopspring/decimal** | **新增** | **所有金额运算的唯一方式** |

### 3.3 为什么必须引入 decimal

```
Go原生 float64：0.1 + 0.2 = 0.30000000000000004  ← 金融场景不可接受
decimal：       0.1 + 0.2 = 0.3                   ← 精确
```

金额数据的完整链路：
```
前端(string) → Thrift IDL(string) → Go层(decimal运算) → GORM(string) → MySQL(DECIMAL)
```

全程不经过 float64，零精度损失。

### 3.4 不需要引入的技术

| 技术 | 为什么不用 |
|------|-----------|
| 分布式事务(TCC/Saga/Seata) | 扣款+记流水在同一个MySQL事务内，不需要跨库事务 |
| 分布式锁库(Redisson) | 余额并发靠DB层CAS乐观锁，不依赖应用层分布式锁 |
| 本地缓存(BigCache) | Redis缓存已满足性能需求，多一层是过度设计 |
| 新消息队列(RocketMQ) | 已有Kafka+NATS，不再引入第三种MQ |

## 五、数据库设计：数据怎么存

### 5.1 表清单

经过分析，从"必须、应该、可选"三个维度，确定了以下表结构：

**必须建（6张）**：没有这些表，核心业务跑不起来

| 表名 | 作用 | 核心字段 |
|------|------|---------|
| **wallet** | 余额表（核心中的核心） | user_id + currency_code + wallet_type → 唯一 |
| **transaction_log** | 交易流水（审计+幂等） | order_id + log_type → 唯一（幂等键） |
| **deposit_order** | 充值订单 | 订单号、金额、状态(1-3)、支付方式 |
| **withdrawal_order** | 提现订单 | 订单号、金额、状态(1-7)、审核链路 |
| **withdrawal_account** | 提现账户 | 银行卡/USDT地址（AES加密存储） |
| **currency_config** | 币种配置 | 汇率、精度、浮动比例、阈值 |

**应该建（4张）**：业务完整性需要

| 表名 | 作用 |
|------|------|
| **exchange_record** | 兑换记录 |
| **reward_record** | 奖励记录 |
| **rate_change_log** | 汇率变更日志（审计追踪） |
| **system_config** | 系统配置（KV表，存基准币种等） |

### 5.2 余额表设计（最核心的一张表）

```sql
CREATE TABLE wallet (
    id              BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id         BIGINT NOT NULL,
    currency_code   VARCHAR(20) NOT NULL,
    wallet_type     INT NOT NULL,           -- 1=中心 2=奖励 3=主播 4=代理 5=场馆
    available_balance DECIMAL(20,8) DEFAULT 0,  -- 可用余额
    frozen_balance    DECIMAL(20,8) DEFAULT 0,  -- 冻结余额
    create_at       BIGINT,
    update_at       BIGINT,
    UNIQUE KEY uk_user_currency_type (user_id, currency_code, wallet_type)
);
```

**设计要点**：
- 三元组唯一索引 `(user_id, currency_code, wallet_type)` 保证一个用户在一个币种下每种钱包只有一行
- `DECIMAL(20,8)` 满足所有币种精度要求（法币整数、USDT 8位）
- `available_balance` 和 `frozen_balance` 分开，提现冻结时减available加frozen

### 5.3 流水表设计（幂等的关键）

```sql
CREATE TABLE transaction_log (
    id          BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id     BIGINT NOT NULL,
    currency_code VARCHAR(20) NOT NULL,
    wallet_type INT NOT NULL,
    order_id    VARCHAR(64) NOT NULL,     -- 关联的业务订单号
    log_type    INT NOT NULL,             -- 流水类型（1-12）
    amount      DECIMAL(20,8) NOT NULL,   -- 变动金额
    balance_after DECIMAL(20,8) NOT NULL, -- 变动后余额
    remark      VARCHAR(255),
    create_at   BIGINT,
    UNIQUE KEY uk_order_log_type (order_id, log_type)   -- 幂等键！
);
```

**幂等机制**：同一个 `order_id` + `log_type` 只能写入一次。如果重复调用入账接口，数据库唯一索引会拒绝插入，保证不会重复记账。

## 六、核心技术方案：关键问题怎么解决

这是整个讲解的重点，也是领导和同事最关心的部分。钱包模块面临的技术挑战不在于"用什么新技术"，而在于"怎么把已有技术用对"。

### 6.1 余额并发安全 — CAS乐观锁

**场景**：两个请求同时扣同一个用户的余额，怎么防止超扣？

**方案**：数据库层的条件更新（CAS乐观锁）

```sql
UPDATE wallet
SET available_balance = available_balance - 100,
    update_at = 1709020800000
WHERE user_id = 123
  AND currency_code = 'VND'
  AND wallet_type = 1
  AND available_balance >= 100;   -- 关键条件！
```

**原理**：`WHERE available_balance >= 100` 这个条件让数据库在行锁级别保证余额充足才执行。如果两个请求并发：
- 第一个请求成功，`affected_rows = 1`
- 第二个请求因为余额已被扣减，条件不满足，`affected_rows = 0`
- 代码判断 `affected_rows == 0` 就返回"余额不足"

**为什么选这个方案而不是其他**：

| 方案 | 优点 | 缺点 | 适合场景 |
|------|------|------|---------|
| **CAS乐观锁** | 无额外组件，性能好，MySQL原生支持 | 高竞争时有重试开销 | **我们的场景**（同一用户并发低） |
| 悲观锁(FOR UPDATE) | 强一致 | 锁等待，高并发下性能差 | 极端一致性要求 |
| Redis分布式锁 | 灵活 | 引入额外组件，有锁超时风险 | 跨库操作 |

**我们选CAS的理由**：同一用户同一时刻并发操作余额的概率很低（正常用户不会同时发两笔充值），CAS完全够用。

### 6.2 扣款与记流水的原子性 — 本地事务

**场景**：扣余额和写流水必须同时成功或同时失败，不能出现"扣了钱但没记账"。

**方案**：GORM本地事务，在同一个MySQL事务内完成

```
事务开始
├── 步骤1：UPDATE wallet SET available_balance = available_balance - 100 WHERE ...
├── 步骤2：INSERT INTO transaction_log (order_id, log_type, amount, ...) VALUES (...)
├── 步骤3：（如果需要）UPDATE deposit_order SET status = 2 WHERE ...
事务提交
```

三步要么全成功，要么全回滚。不需要分布式事务框架，因为所有操作都在同一个数据库内。

**为什么不用分布式事务**：
- 扣款和记流水操作的是同一个库的表
- 引入 TCC/Saga 会让代码复杂度翻倍，收益为零
- 跨服务的一致性（如钱包↔财务）用"重试+幂等+对账"三板斧解决，不用分布式事务

### 6.3 防止重复记账 — orderId幂等

**场景**：财务回调通知充值成功，因为网络重试发了两次，怎么防止入账两次？

**方案**：`transaction_log` 表的 `(order_id, log_type)` 唯一索引

```
第一次调用：INSERT INTO transaction_log (order_id='D123456', log_type=1, ...) → 成功
第二次调用：INSERT INTO transaction_log (order_id='D123456', log_type=1, ...) → 唯一索引冲突 → 捕获错误 → 返回"已处理"
```

**调用方看到的效果**：无论调几次，结果都一样（幂等性）。第一次真正入账，后续重复调用直接返回成功。

### 6.4 金额精度 — decimal全链路

**方案**：

```
前端传string "100.50"
  → Thrift IDL定义为string
    → Go层：amount, _ := decimal.NewFromString("100.50")
      → 业务计算：amount.Mul(rate) / amount.Sub(fee)
        → 存库：amount.String() → MySQL DECIMAL(20,8)
```

**关键规则**：
- 全链路不出现 float64
- 所有加减乘除都用 decimal 方法
- 前端传入的金额后端必须重新校验（不信任前端计算结果）
- 汇率换算结果的截断规则：法币向下截断（用户不会多得），加密币保留8位

### 6.5 汇率管理 — 三级汇率体系

```
实时汇率（三方API平均值）
    │
    │ 偏差超过阈值时自动更新
    ▼
平台汇率（数据库存储）
    │
    │ 公式计算
    ├──────────────────────────┐
    ▼                          ▼
入款汇率 = 平台汇率 × (1+浮动%)    出款汇率 = 平台汇率 × (1-浮动%)
（用户充值看到的汇率）              （用户提现使用的汇率）
```

**实现方式**：
- 定时任务每分钟调用三方汇率API（3家取平均，容错降级）
- 实时汇率存Redis缓存（TTL 60秒）
- 偏差超过阈值时自动更新平台汇率到数据库
- 入款/出款汇率是计算值，不存储，查询时实时计算

**为什么要三级**：实时汇率波动大，平台汇率相对稳定，给运营一个缓冲空间。浮动比例让平台在充值和提现上有汇差收益。

### 6.6 跨服务可靠性 — 重试+幂等+对账

**场景**：钱包调财务出款，RPC超时了，到底出了没出？

**三板斧**：

```
第一层：即时重试
  └─ RPC调用失败 → 指数退避重试3次（1s, 2s, 4s）
      └─ 对端接口必须幂等，重试不会产生副作用

第二层：定时补偿
  └─ 订单状态停在"待出款"超过N分钟
      └─ 定时任务扫描 → 重新发起查询/出款
          └─ 对端接口必须幂等

第三层：日终对账
  └─ 每天凌晨跑对账任务
      └─ 钱包流水汇总 vs 余额变化
          └─ 对不上就告警，人工介入
```

这三层是递进关系：即时重试解决99%的临时故障，定时补偿解决持续异常，日终对账兜底所有情况。

### 6.7 提现状态机 — 严格单向流转

```
用户提交(1) → 风控审核
                ├─ 驳回(2)  → 解冻退回 → 终态
                └─ 通过(3)  → 财务审核
                              ├─ 驳回(4)  → 解冻退回 → 终态
                              └─ 通过(5)  → 出款
                                            ├─ 失败(6) → 终态
                                            └─ 成功(7) → 扣除冻结 → 终态
```

**技术保障**：状态流转用 `WHERE status = 旧状态` 条件更新

```sql
-- 风控审核通过（1→3）
UPDATE withdrawal_order SET status = 3 WHERE id = ? AND status = 1;

-- 如果status已经不是1（比如已被另一个请求处理了），affected_rows=0，拒绝操作
```

这保证了状态只能按预设路径流转，不可能跳步或倒退。

## 七、IDL设计：接口怎么定义

### 7.1 文件拆分

按业务域拆成7个IDL文件：

```
common/idl/ser-wallet/
├── service.thrift          # 服务入口（42个方法声明）
├── wallet.thrift           # 钱包核心（余额/概览）
├── deposit.thrift          # 充值全流程
├── withdrawal.thrift       # 提现全流程
├── exchange.thrift         # 兑换全流程
├── currency.thrift         # B端币种管理
└── wallet_rpc.thrift       # RPC对外接口
```

**为什么这样拆**：
- 与产品原型的页面模块一一对应
- 每个文件100-200行，好维护
- 多人协作时各改各的文件，不冲突
- 参照工程中 ser-live（9文件39方法），比例合理

### 7.2 接口分组（service.thrift 中的组织方式）

```
WalletService {
    // ========== C端 · 钱包基础（3个） ==========
    // 币种列表及余额、指定币种余额、钱包概览

    // ========== C端 · 充值（6个） ==========
    // 充值方式、创建订单、USDT支付页、订单详情、记录列表、重复检查

    // ========== C端 · 提现（7个） ==========
    // 提现方式、创建订单、限额查询、保存/获取账户、订单详情、记录列表

    // ========== C端 · 兑换（4个） ==========
    // 汇率查询、执行兑换、兑换预览、记录列表

    // ========== C端 · 奖励（2个） ==========
    // 奖励记录、活动列表

    // ========== B端 · 币种管理（5个） ==========
    // 币种列表、编辑、设基准、汇率日志、手动刷新

    // ========== RPC · 对外接口（15个） ==========
    // 余额操作（查/加/减/冻/解/批量）+ 币种查询（汇率/列表/详情/基准）
}
```

### 8.2 提现流程

```
用户操作                     ser-wallet               ser-finance        ser-kyc
  │                            │                          │                │
  │ 1.点提现                   │                          │                │
  ├───────────────────────────>│ 2.RPC查KYC状态            │                │
  │                            ├──────────────────────────────────────────>│
  │                            │<─────────────────────────────────────────┤
  │                            │ 3.RPC查提现方式           │                │
  │                            ├─────────────────────────>│                │
  │                            │<─────────────────────────┤                │
  │<───────────────────────────┤ 4.返回方式+限额+KYC状态   │                │
  │                            │                          │                │
  │ 5.输入金额,确认提现        │                          │                │
  ├───────────────────────────>│ 6.校验金额+KYC+限额       │                │
  │                            │ 7.事务内：                │                │
  │                            │   ├ 冻结余额(CAS)        │                │
  │                            │   ├ 创建提现订单(状态1)   │                │
  │                            │   └ 写入流水(冻结)        │                │
  │<───────────────────────────┤ 8.返回"提现申请已提交"     │                │
  │                            │                          │                │
  │                            │      后续审核流程（B端）    │                │
  │                            │                          │                │
  │                            │ 风控审核通过→状态1→3       │                │
  │                            │ 财务审核通过→状态3→5       │                │
  │                            │ RPC调财务执行出款          │                │
  │                            ├─────────────────────────>│                │
  │                            │                          │ 对接支付通道     │
  │                            │<─────────────────────────┤ 出款结果回调     │
  │                            │ 成功：扣除冻结→状态7      │                │
  │                            │ 失败：解冻退回→状态6      │                │
```

### 8.3 兑换流程

```
用户操作                     ser-wallet（全部内部完成）
  │                            │
  │ 1.查看兑换汇率             │
  ├───────────────────────────>│ 2.从Redis读缓存汇率
  │<───────────────────────────┤ 3.返回汇率+赠送比例
  │                            │
  │ 4.输入金额,确认兑换        │
  ├───────────────────────────>│ 5.校验金额+余额
  │                            │ 6.获取最新汇率
  │                            │ 7.decimal计算：
  │                            │   ├ 目标金额 = 源金额 × 汇率
  │                            │   └ 赠送金额 = 目标金额 × 赠送比例
  │                            │ 8.事务内：
  │                            │   ├ 扣源币种(CAS)     → 流水(兑换扣减)
  │                            │   ├ 加目标币种中心钱包  → 流水(兑换入账)
  │                            │   ├ 加目标币种奖励钱包  → 流水(兑换赠送)
  │                            │   └ 创建兑换记录
  │<───────────────────────────┤ 9.返回兑换结果
```

兑换的特点：**全部在钱包内部完成**，不需要调用外部服务，不涉及真实资金流动，只是内部账本的数字变动。

| 任务 | 具体内容 | 依赖 |

**产出**：一个能启动、能注册到ETCD、能连接MySQL和Redis的空服务。

### Phase 2：核心能力（余额+币种+RPC）

| 任务 | 具体内容 | 优先级 |
|------|---------|--------|
| 2.1 币种配置CRUD | B端5个接口 + Redis缓存 | P0 |
| 2.2 余额操作核心 | wallet表的增/减/冻/解方法（CAS） | P0 |
| 2.3 流水记录核心 | transaction_log的写入（幂等） | P0 |

**产出**：其他服务（财务/投注）可以调用钱包的RPC接口了。这一步完成后就可以跟财务并行联调。

**关键点**：Phase 2 完成 = 钱包对外能力就绪，不阻塞其他团队。

### Phase 3：C端业务线

按复杂度从低到高：

| 任务 | 接口数 | 复杂度 | 说明 |
|------|--------|--------|------|
| 3.1 兑换功能 | 4 | 低 | 纯内部操作，不依赖外部服务 |
| 3.2 钱包查询 | 3+2 | 低 | 余额/概览/奖励记录 |
| 3.3 充值功能 | 6 | 中 | 依赖财务RPC（充值方式+通道匹配） |
| 3.4 提现功能 | 7 | 高 | 依赖财务+KYC，状态机最复杂 |

**为什么这个顺序**：
- 兑换不依赖任何外部服务，可以独立开发和测试
- 充值和提现依赖财务模块的接口，如果财务还没好，先用Mock
- 提现是最复杂的（7种状态、多级审核、冻结/解冻），放最后打磨

## 十、与财务模块的协作方式

### 10.1 接口契约先行

钱包和财务的协作通过 **Thrift IDL契约** 来约束。我会在 Phase 1 就把 `wallet_rpc.thrift` 定义好，发给财务同学：

> "这是我提供的15个RPC方法，入参和出参都在IDL里了，你按这个调就行。"

同时财务同学也需要提供他们的IDL给我：

> "充值方式列表、提现方式列表、通道匹配——这几个接口的Req/Resp是什么？"

### 10.2 Mock并行开发

在财务接口未就绪时，我会Mock财务的返回值，先把钱包自身逻辑跑通：
- 充值方式列表 → Mock返回固定的方式数组
- 通道匹配 → Mock返回支付URL
- 出款请求 → Mock返回成功

等财务接口好了，把Mock换成真实RPC调用即可。

### 10.3 联调要点

| 联调场景 | 关键验证 |
|---------|---------|
| 充值入账 | 财务回调→钱包入账，验证幂等（重复回调不会重复入账） |
| 提现出款 | 钱包冻结→财务出款→回调结果，验证状态流转正确 |
| 人工操作 | 财务发起人工加减款→钱包执行，验证金额和流水 |
| 异常场景 | RPC超时、回调丢失、重复调用，验证兜底机制 |

## 十一、风险与应对

### 11.1 已识别的风险

| 风险 | 级别 | 应对 |

## 十二、总结

### 一句话概括实现方案

> **钱包是一个记账系统。用乐观锁保证余额安全，用本地事务保证一致性，用唯一索引保证幂等，用decimal保证精度，用缓存保证性能。19个已有组件复用，只新增1个decimal依赖。42个接口，10张表，7个IDL文件，4个阶段递进交付。**

### 关键数字

| 指标 | 数值 |
|------|------|
| 接口总数 | 42（C端22 + B端5 + RPC15） |
| IDL文件数 | 7 |
| 数据库表数 | 10（6必须 + 4应该） |
| 技术组件 | 20（19已有 + 1新增） |
| 新增依赖 | 1个（shopspring/decimal） |
| 开发阶段 | 4个（地基→核心→业务→收尾） |
| 币种支持 | 5种（VND/IDR/THB/USDT/BSB） |
| 钱包类型 | 5种（中心/奖励/主播/代理/场馆） |

### 我有信心的部分

- 技术方案成熟，CAS+事务+幂等+缓存是业界标准做法
- 工程基础设施完善，不需要从零搭建中间件
- IDL先行的工作模式可以解耦前后端和服务间依赖
- 分阶段交付确保其他团队不被钱包进度阻塞

### 需要支持的部分

- 财务模块尽早提供IDL契约，方便Mock并行开发
- 跨部门评审尽快拍板7个P0问题，减少后续返工
- 三方汇率API的选型和账号申请需要提前推进
