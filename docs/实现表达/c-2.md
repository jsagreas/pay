# 一、开场定位（我负责什么，边界在哪）

## 1.1 模块定位

ser-wallet 是平台的**资金中枢**——用户充进来的每一分钱、花出去的每一笔消费、兑换的每一次操作，最终都经过这个模块。

```
用户的钱 ──→ 充值 ──→ 钱包（ser-wallet）──→ 消费/兑换/提现
                           ↕
                     币种配置（同模块）
                           ↕
                     财务管理（他人负责，RPC 联动）
```

## 1.2 职责划分

| 模块 | 负责人 | 说明 |
|------|--------|------|
| **钱包模块** | 自己 | C 端核心——余额管理、充值、提现、兑换、记录查询 |
| **币种配置模块** | 自己 | B 端管理——币种 CRUD、四层汇率体系、汇率定时同步 |
| **财务管理模块** | 他人 | 通道配置、充值/提现方式、三级审核、人工修正等——但和钱包有大量 RPC 交互 |

我需要做好前两个，同时**充分理解第三个**，因为充值和提现的完整链路跨越两个模块。

## 1.3 核心数据

- 需求来源：6 个文件夹，共约 133 张产品原型图
  - 多币种钱包需求文档（42 张）+ 产品原型（32 张）
  - 币种配置需求文档（19 张）+ 产品原型（3 张）
  - 财务管理需求文档（25 张）+ 产品原型（26 张）
- 接口规模：约 42 个方法（39 确认 + 约 3 预留）
- 数据表：约 12 张（钱包 8 张 + 币种 4 张）
- 技术栈：18 项复用 + 1 项新增

# 二、需求全景（我们要做的事）

## 2.1 用户视角的核心功能

从用户角度看，钱包模块提供四个核心能力：

```
┌──────────────────────────────────────────────────────────────────┐
│                          用户钱包                                 │
│                                                                  │
│  ① 充值：用本国货币或 USDT 往平台充钱                             │
│     VND/IDR/THB → 银行转账/扫码支付                               │
│     USDT → 链上转账（TRC20/ERC20/BEP20）                         │
│     BSB 不能直接充值，只能通过兑换获得                              │
│                                                                  │
│  ② 提现：把钱从平台提出去                                         │
│     法币 → 银行卡/电子钱包                                        │
│     USDT → 链上提现                                              │
│     关键规则：法币充→法币提，USDT充→USDT提，混合充→法币提          │
│                                                                  │
│  ③ 兑换：法币 → BSB（单向，不可逆）                               │
│     使用平台汇率，额外 10% 奖金进奖励钱包                          │
│                                                                  │
│  ④ 记录：查看充值/提现/兑换/奖励四类交易历史                       │
└──────────────────────────────────────────────────────────────────┘
```

## 2.2 钱包体系结构

平台支持三类货币，每个用户在每个币种下有独立的钱包体系：

```
币种体系:
  平台币: BSB（10 BSB = 1 USDT，锚定汇率）
  法币:   VND（越南盾）、IDR（印尼卢比）、THB（泰铢）
  加密:   USDT（支持 TRC20/ERC20/BEP20 三条链）

钱包类型:
  中心钱包 ← 充值入账，消费优先扣
  奖励钱包 ← 奖金入账，需完成稽核才能转中心
  主播钱包 ← 主播收入
  代理钱包 ← 预留，本期不做
  场馆钱包 ← 预留，本期不做

关键规则:
  · 资金余额 = 中心钱包 + 奖励钱包
  · 消费优先级：先扣中心 → 再扣奖励
  · 币种隔离：切换币种 = 切换整套钱包
```

## 2.3 币种配置（B 端管理）

运营人员在后台管理系统中配置平台支持的币种和汇率：

```
核心能力:
  · 币种 CRUD（名称、简称、icon、精度、状态）
  · 四层汇率体系管理
  · 汇率变更日志审计

四层汇率（从底到顶）:
  实时汇率 ← 3 个第三方 API 取平均值（定时拉取）
  平台汇率 ← 基于实时汇率，偏差超阈值才更新（避免频繁波动）
  充值汇率 = 实时汇率 × (1 + 浮动%)（平台多收一点）
  提现汇率 = 实时汇率 × (1 - 浮动%)（平台少给一点）
```

# 三、架构设计（怎么组织代码）

## 3.1 核心架构决策

在开始编码之前，有 8 个核心决策需要确定，这些决策一旦定了就不容易改变：

| 决策项 | 选择 | 为什么 |
|--------|------|--------|
| 账户模型 | 余额账户制（非复式记账） | 业务复杂度适中，不需要完整的复式记账系统 |
| 余额权威 | 数据库为准，Redis 为辅 | 金融数据以 DB 为权威来源，缓存加速读 |
| 并发控制 | 数据库行锁（悲观锁）为主 | TiDB 行锁成熟，简单可靠，不过度设计 |
| 幂等方案 | 流水表唯一索引 | 一石二鸟——既做审计又做幂等防重 |
| 金额精度 | BIGINT 整数（最小货币单位） | 无浮点误差，性能好，Stripe/支付宝/微信支付都这么做 |
| 事务模型 | 本地事务（单库） | 钱包数据在一个 TiDB 库内，不需要分布式事务 |
| 通信方式 | 同步 RPC 为主 | 现有工程全部是同步 RPC，保持一致，不引入 MQ 额外复杂度 |
| 汇率计算 | shopspring/decimal | 唯一新增依赖，解决汇率浮动比例的高精度计算 |

## 3.2 四层架构

现有服务用的是 `handler → ser → rep/repo` 三层。钱包模块在 `ser` 和 `rep` 之间**新增 ledger 层**，专门处理余额操作：

```
┌──────────────────────────────────────────────────────────────┐
│  handler.go（接口层）                                          │
│  · 纯委托，一行代码转发到对应的 ser 层 Service                   │
│  · ~42 个方法，分发到 5 个 Service                              │
├──────────────────────────────────────────────────────────────┤
│  internal/ser/（业务编排层）                                    │
│  · 业务规则校验（余额够不够、KYC通不通过、资金来源对不对）       │
│  · 编排调用流程（先查→再校验→调外部RPC→调ledger→更新缓存）      │
│  · 包含：WalletSer / DepositSer / WithdrawSer / ExchangeSer  │
│          CurrencySer / RateSer / RecordSer                    │
├──────────────────────────────────────────────────────────────┤
│  internal/ledger/（账务核心层）★ 新增层，钱包特有                │
│  · 余额变更的 5 个原子操作：Credit / Debit / Freeze / Unfreeze / Exchange │
│  · 事务管理（余额变更 + 流水记录同一事务）                       │
│  · 幂等控制（基于 orderNo 防重复）                               │
│  · 并发安全（数据库行锁）                                       │
│  · 这一层是资金安全的最后防线，不允许被绕过                      │
├──────────────────────────────────────────────────────────────┤
│  internal/gorm_gen/ + internal/rep/（数据访问层）               │
│  · gorm_gen：自动生成的 19 个标准 CRUD 方法 / 表                │
│  · rep：手写复杂查询扩展（带锁查询、聚合统计等）                │
└──────────────────────────────────────────────────────────────┘
```

## 3.3 为什么需要 ledger 层

**这是和现有服务最大的区别，也是本次设计的核心亮点。**

普通服务（ser-app 等）不需要 ledger 层，因为它们的写操作是简单的 CRUD——写错了改回来就行。但钱包不同，**每一次余额变更都可能造成资金损失**。

如果不引入 ledger 层，把余额操作散在 ser 层的各个方法里，会出现三个问题：
1. **代码重复**——每个改余额的地方都要写事务 + 幂等 + 行锁
2. **容易遗漏**——某个方法忘了加锁或做幂等检查
3. **难以审计**——安全相关逻辑分散在各处，review 成本高

收敛到 ledger 层后，**所有余额变更只有一个入口**，ser 层不允许直接 UPDATE user_wallet 表的 balance/frozen_amount 字段。这是钱包系统业界的标准做法。

## 3.4 ledger 层的 5 个原子操作

```
Credit(orderNo, userId, currencyId, walletType, amount, flowType)
  → 增加余额（充值上分 / 奖金发放 / 人工加款）

Debit(orderNo, userId, currencyId, walletType, amount, flowType)
  → 减少余额（提现扣款 / 人工减款 / 消费）

Freeze(orderNo, userId, currencyId, amount)
  → 冻结金额（提现时冻结可用余额）

Unfreeze(orderNo, userId, currencyId, amount)
  → 解冻金额（提现被拒绝/失败时归还）

Exchange(orderNo, userId, fromCurrencyId, toCurrencyId, fromAmount, toAmount, bonusAmount)
  → 跨币种原子操作（扣法币 + 加 BSB + 加奖金，同一事务）
```

## 3.5 单笔余额变更的内部流程（以 Credit 为例）

这是整个架构中最核心的代码路径，每一步都有明确的安全考量：

```
Credit(ctx, params) 内部:

  ┌────────────────── 数据库事务开始 ──────────────────┐
  │                                                     │
  │  Step 1: 幂等检查（先写流水）                        │
  │    INSERT INTO wallet_flow (order_no, wallet_type, ...) │
  │    → 唯一索引 uk_order_wallet 冲突 = 已处理过 → 直接返回成功 │
  │    → 成功 = 首次执行 → 继续                          │
  │                                                     │
  │  Step 2: 锁定钱包行                                  │
  │    SELECT * FROM user_wallet                         │
  │    WHERE user_id=? AND currency_id=? AND wallet_type=? │
  │    FOR UPDATE                                        │
  │    → 其他并发请求在此阻塞等待                         │
  │                                                     │
  │  Step 3: 执行余额变更                                │
  │    UPDATE user_wallet SET balance = balance + ?       │
  │                                                     │
  │  Step 4: 补全流水（balance_before / balance_after）   │
  │    UPDATE wallet_flow SET balance_before=?, balance_after=? │
  │                                                     │
  └────────────────── 事务提交 ──────────────────────┘

  Step 5: 删除缓存（事务提交后）
    DEL wallet:bal:{userId}:{currencyId}

  返回成功
```

**为什么"先写流水再更新余额"？**
- 利用唯一索引做幂等比"先查后写"更安全——两个并发请求不可能同时通过 INSERT 的唯一索引检查
- 如果后续步骤失败，整个事务回滚，流水也一起回滚

## 3.6 Exchange（兑换）——最复杂的事务

兑换涉及跨币种的 3 次余额变更，是最严格的事务场景：

```
单个数据库事务内:
  ① INSERT 3 条流水记录（order_no 相同，wallet_type 不同 → 唯一索引不冲突）
  ② 锁定法币中心钱包 → 检查可用余额
  ③ 锁定 BSB 中心钱包
  ④ 锁定 BSB 奖励钱包
  ⑤ 执行 3 次余额变更:
     法币中心 balance -= fromAmount
     BSB 中心 balance += toAmount
     BSB 奖励 balance += bonusAmount
  ⑥ 补全流水 → 事务提交

加锁顺序（防死锁）:
  固定按 (currency_id, wallet_type) 升序加锁
  例: VND(id=2)中心(1) → BSB(id=5)中心(1) → BSB(id=5)奖励(2)
```

# 四、数据模型设计

## 4.1 金额存储策略

**决策：所有金额用 BIGINT 存储，以该币种的最小货币单位为单位。**

```
VND:  decimals=0   → 存 50000    → 展示 50,000 VND
IDR:  decimals=0   → 存 50000    → 展示 50,000 IDR
THB:  decimals=2   → 存 10050    → 展示 100.50 THB
USDT: decimals=8   → 存 100000000 → 展示 1.00000000 USDT
BSB:  decimals=8   → 存 1000000000 → 展示 10.00000000 BSB

转换公式: 展示值 = 数据库值 / 10^decimals
```

为什么不用 DECIMAL 或 double？
- **double 不可接受**：0.1 + 0.2 ≠ 0.3，金融场景的硬性红线
- **BIGINT 优于 DECIMAL**：整数运算更快、存储更紧凑（8 字节 vs 动态长度）、Stripe/支付宝/微信支付都用最小单位整数

IDL 传输中，**C 端接口用 string**（前端友好，避免 JS 精度丢失），**RPC 接口用 i64**（服务间传输高效）。

## 4.2 核心表清单

```
钱包模块（8 张表）:
  ① user_wallet        ← 用户钱包余额（最核心，行锁目标）
  ② wallet_flow        ← 钱包流水（审计+幂等载体，append-only）
  ③ deposit_order      ← 充值订单
  ④ withdraw_order     ← 提现订单
  ⑤ exchange_order     ← 兑换订单
  ⑥ reward_record      ← 奖励记录
  ⑦ user_fund_source   ← 用户资金来源记录（判定提现方式）
  ⑧ user_currency_pref ← 用户币种偏好

币种配置模块（4 张表）:
  ⑨ currency_config    ← 币种配置
  ⑩ currency_rate      ← 汇率数据（实时/平台/充值/提现四层）
  ⑪ rate_change_log    ← 汇率变更日志
  ⑫ rate_source_log    ← 第三方汇率拉取日志
```

## 4.3 两张最关键的表

### user_wallet — 用户钱包余额表

```sql
CREATE TABLE user_wallet (
    id              BIGINT       PRIMARY KEY,          -- 雪花 ID
    user_id         BIGINT       NOT NULL,
    currency_id     BIGINT       NOT NULL,
    wallet_type     TINYINT      NOT NULL,             -- 1=中心 2=奖励 3=主播
    balance         BIGINT       NOT NULL DEFAULT 0,   -- 余额（最小货币单位）
    frozen_amount   BIGINT       NOT NULL DEFAULT 0,   -- 冻结金额
    version         BIGINT       NOT NULL DEFAULT 0,   -- 预留乐观锁
    status          TINYINT      NOT NULL DEFAULT 1,   -- 1=正常 2=冻结 3=禁用
    -- 审计字段由 GORM UserPlugin 自动填充
    create_by/create_at/update_by/update_at/delete_at

    UNIQUE KEY uk_user_currency_type (user_id, currency_id, wallet_type, delete_at)
);

-- 设计要点:
-- · 唯一索引保证一个用户+一个币种+一个钱包类型只有一行记录
-- · 可用余额 = balance - frozen_amount（计算值，不存储）
-- · version 预留给乐观锁（当前主用行锁，未来可升级）
```

### wallet_flow — 流水表（审计核心 + 幂等载体）

```sql
CREATE TABLE wallet_flow (
    id              BIGINT       PRIMARY KEY,
    user_id         BIGINT       NOT NULL,
    currency_id     BIGINT       NOT NULL,
    wallet_type     TINYINT      NOT NULL,
    flow_type       TINYINT      NOT NULL,             -- 1=充值 2=提现 3=兑换出 4=兑换入
                                                       -- 5=奖励 6=冻结 7=解冻 8=人工加 9=人工减
    amount          BIGINT       NOT NULL,             -- 变动金额
    balance_before  BIGINT       NOT NULL,             -- 变动前余额
    balance_after   BIGINT       NOT NULL,             -- 变动后余额
    order_no        VARCHAR(32)  NOT NULL,             -- 关联业务订单号

    UNIQUE KEY uk_order_wallet (order_no, wallet_type) -- 幂等索引
);

-- 设计要点:
-- · append-only，只写不改不删
-- · uk_order_wallet 同时承担"幂等防重"功能
-- · 一笔兑换产生 3 条流水（扣法币 + 加BSB中心 + 加BSB奖励），order_no 相同但 wallet_type 不同
```

# 五、资金安全——四道防线

这是向领导汇报时最需要讲清楚的部分。钱包最怕的不是功能做不出来，而是**资金安全出问题**。

```
第一道: 幂等（防重复操作）
  wallet_flow 表的 uk_order_wallet 唯一索引
  → 数据库层面的硬保证，比应用层判断可靠
  → 财务模块重试调 Credit 10 次，也只会加一次款

第二道: 事务（防部分失败）
  余额变更 + 流水记录在同一个 DB 事务中
  → 不存在"钱加了但没记录"或"记录了但钱没加"

第三道: 行锁（防并发冲突）
  SELECT FOR UPDATE 锁定钱包行后再操作
  → 两个并发的 Credit 不会导致余额丢失更新
  → 固定加锁顺序防死锁

第四道: 对账（防积累误差）
  wallet_flow 记录每次变更前后余额
  → 可以反算：初始余额 + 所有流水 = 当前余额
  → 定期对账脚本校验一致性
```

**讲人话**：幂等保证"不多做"，事务保证"要么全做要么全不做"，行锁保证"一个一个来"，对账保证"做完了能验证"。

# 六、技术栈选择（能复用的坚决复用）

## 6.1 总览

```
复用现有（18 项）:
  Kitex / Hertz / GORM+Gen / Thrift IDL / TiDB / Redis / ETCD
  Snowflake / AES / Sonic / Zap / TTHeader / RPC 工厂
  XXL-JOB / robfig/cron / FontAuth / BackAuth+RBAC / BlackCheck+WhiteCheck

现有扩展（5 项）:
  Redis 频控封装 / Redis Key 前缀 / 错误码段 / 枚举注册 / ETCD 功能开关

新增引入（1 项）:
  shopspring/decimal v1.4.0 — 汇率浮动比例的高精度计算

明确不引入（6 项）:
  × 分布式事务框架   → 单库本地事务够用
  × 消息队列(Kafka)  → 同步 RPC 够用，不增加复杂度
  × Elasticsearch    → 流水查询走 TiDB 索引即可
  × Redlock 分布式锁 → 数据库行锁够用
  × 限流中间件库     → Redis INCR+EXPIRE 够用
  × 复式记账框架     → 余额账户制够用
```

## 6.2 为什么只新增一个依赖

钱包模块看起来复杂，但技术层面**几乎所有能力现有工程都已具备**。唯一欠缺的是汇率浮动比例计算（例如 `实时汇率 × 1.025`）需要高精度小数，原生 float64 有精度风险，所以引入 shopspring/decimal。

其他能力的映射：

| 需求场景 | 技术方案 | 来源 |
|----------|----------|------|
| 并发余额操作 | TiDB SELECT FOR UPDATE | 项目已有 TiDB |
| 幂等防重 | wallet_flow 唯一索引 | GORM Gen + TiDB |
| 缓存余额 | Redis Cache-Aside | 项目已有 Redis 封装 |
| 定时拉取汇率 | robfig/cron 或 XXL-JOB | 项目已有 |
| 敏感数据加密 | AES 对称加密 | common/pkg/utils 已有 |
| 接口限流 | Redis INCR + EXPIRE | 项目已有 Redis |
| 配置热更新 | ETCD Watch | 项目已有 |

## 7.1 文件结构

遵循 IDL-first 开发模式，先定义接口，再生成代码：

## 7.3 关键设计约定

| 约定项 | 规则 | 说明 |
|--------|------|------|
| C 端金额 | string 类型 | 前端友好，避免 JS 精度丢失 |

| 命名 | ActionEntityReq/Resp | 如 CreateDepositOrderReq |

# 八、缓存策略

## 8.1 缓存分层

```
读请求
  │
  ▼
Redis 缓存（热数据，高频读的第一道屏障）
  │ cache miss
  ▼
TiDB 数据库（权威数据源，所有写操作直接写这里）
```

写请求永远直接写 DB，写成功后删除缓存（Cache-Aside 模式）。

## 8.2 各类数据的缓存策略

| 数据类型 | 缓存 | TTL | 失效时机 |
|----------|------|-----|----------|
| 用户余额 | 是 | 5 min | Credit/Debit/Freeze 后删除 |
| 可用币种列表 | 是 | 10 min | 币种 CRUD 后删除 |
| 平台汇率 | 是 | 5 min | 定时任务覆盖更新 |
| 用户当前币种 | 是 | 30 min | SwitchCurrency 后删除 |
| 订单数据 | **否** | — | 写多读少，缓存收益低 |
| 冻结金额 | **否** | — | 涉及资金安全，必须读 DB |
| 流水记录 | **否** | — | append-only，走 DB 索引 |

## 8.3 余额缓存一致性

```
写后失效 + 短 TTL 双重保证:

  Credit/Debit 执行后:
    1. DB 事务提交
    2. 立即 DEL 缓存 key
    3. 下次读请求 cache miss 时从 DB 加载最新值

  如果 DEL 失败（极端情况）:
    TTL=5min 后缓存自动过期
    5 分钟内可能读到旧余额（可接受——充值/提现结果看订单状态，不看余额）

  不用"写后更新缓存"的原因:
    并发写入时可能把旧值覆盖新值，删除比更新更安全
```

# 九、跨模块集成策略（怎么和财务模块协作）

## 9.1 初始化依赖层级

```
L0 (地基)             L1 (通道)           L2 (方式)            L3 (业务可用)
币种配置 ──────────→ 代收/代付通道 ────→ 充值/提现方式 ────→ 用户可以充提兑
  (自己)                (财务)              (财务)               (钱包)

结论: 币种配置是最底层依赖，必须最先开发。
```

## 9.2 Mock 先行策略

核心思路：**先把被调接口做好，再逐步对接调出接口**。

```
阶段一（独立开发期）:
  · 被调接口（Credit/Debit/Freeze 等）先开发好
    → 逻辑独立，零外部依赖
    → 做好了财务模块可以直接调，不阻塞对方

  · 调出接口用 Mock 跑通流程
    → 充值方式列表返回固定 Mock 数据
    → 通道匹配返回固定通道
    → 提现审核直接返回成功

  实现方式（面向接口编程）:
    type FinanceClient interface {
        GetDepositMethods(ctx, req) (resp, error)
        MatchDepositChannel(ctx, req) (resp, error)
        ...
    }
    type realFinanceClient struct{}    // 真实 RPC（联调时用）
    type mockFinanceClient struct{}    // Mock 数据（开发期用）

阶段二（联调期）:
  · 替换 Mock 为真实 RPC
  · 联调优先级: Credit(上分) > Debit/Unfreeze(提现结果) > 其他
```

## 9.3 接口协商要点

联调前需要和财务模块负责人对齐：

1. **RPC 方法签名**：双方确认 Req/Resp 字段定义
2. **回调方式**：充值成功后同步 RPC 还是异步消息？（建议同步 RPC，与现有工程一致）
3. **幂等约定**：orderNo 唯一性由调用方保证，重复调用返回成功
4. **错误处理**：Credit 系统异常时财务该怎么重试、间隔多久

## 10.2 为什么这个顺序

```
1. 币种配置最先做（Step 2）:
   因为它是整个系统的数据地基——没有币种，钱包无法初始化

2. 核心 RPC 其次（Step 3）:
   因为这些接口是被动接口，做好了不阻塞财务模块

3. 兑换优先于充值/提现（Step 4）:
   因为兑换完全自闭环（零外部依赖），可以最先端到端测试

4. 充值/提现最后（Step 5-6）:
   因为依赖财务模块 RPC，需要等对方接口就绪才能联调
   但被调端（Credit/Debit）已提前就绪，不阻塞对方开发
```

# 十一、技术挑战与应对

## 11.1 挑战全景

我从 4 个维度识别了 18 个技术挑战，按优先级分批解决：

```
★★★ 资金安全（5 个）— 出错=资损，最高优先
  ① 并发余额操作 → 数据库行锁 + 固定加锁顺序
  ② 幂等性保障   → 流水表唯一索引
  ③ 金额精度     → BIGINT 整数 + shopspring/decimal
  ④ 兑换原子性   → 本地事务 + 3 次变更同一 TX
  ⑤ 资金对账     → 流水反算 + 定期脚本

★★ 性能可用性（5 个）— 影响用户体验
  ⑥ 热点账户     → 当前不优化，预留 version 字段
  ⑦ 缓存一致性   → Cache-Aside + 短 TTL + 写后删除
  ⑧ 流水表膨胀   → TiDB 自动分片，不需要手动分表
  ⑨ RPC 超时     → 幂等设计允许安全重试
  ⑩ 汇率一致性   → 下单时快照汇率，不用展示时汇率

★ 安全合规（4 个）
  ⑪ 接口防刷     → Redis INCR + EXPIRE 频控
  ⑫ 敏感数据     → AES 加密存储 + 展示脱敏
  ⑬ 资金来源追踪 → fund_source_type 状态机
  ⑭ 防篡改重放   → 签名 + 幂等 + 鉴权

☆ 工程实践（4 个）
  ⑮ ledger 防绕过   → 代码约束 + review + 对账
  ⑯ 跨模块联调      → IDL-first + Mock + 依赖注入
  ⑰ 可观测性        → 结构化日志 + TraceID 贯穿
  ⑱ 连接池管理      → 事务超时控制 + 池配置
```

## 11.2 解决时机

```
编码前必须就绪（Batch 1）:
  金额类型、并发控制策略、幂等方案、事务设计、IDL 契约

测试阶段（Batch 3）:

上线后（Batch 4）:
```

# 十二、明确不做的事（避免过度设计）

| 不做的事 | 为什么不做 |
|----------|-----------|
| 分布式事务（TCC/SAGA） | 钱包数据在同一个 TiDB 库中，本地事务足够 |
| 消息队列驱动余额操作 | 增加复杂度，现有工程全部是同步 RPC |
| 复式记账（双边记账） | 没有平台账户和用户账户之间的资金转移核算需求 |
| 热点账户优化 | 当前并发量不需要，预留了 version 字段可升级 |
| Redis 写余额 | 余额以 DB 为准，Redis 只做读缓存 |
| 独立幂等服务 | wallet_flow 的唯一索引已承担幂等功能 |
| 独立对账服务 | 通过流水表可以脚本对账，不需要独立服务 |
| 分库分表 | TiDB 是分布式数据库，自动处理数据分片 |

**每一项都是"可以但当前不需要"**。如果未来业务规模增长到需要的程度，可以在现有架构上渐进式引入，当前架构已为此预留了扩展空间。

# 十三、风险识别与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 财务模块进度延迟 | 充值/提现 C 端流程无法完整跑通 | 先完成 70% 独立接口 + Mock 联调部分 |
| KYC 服务不可用 | 提现资格校验失败 | 降级策略：允许 USDT 提现，禁止法币提现 |
| 第三方汇率 API 不可用 | 实时汇率无法更新 | 使用最近一次缓存的汇率，平台汇率不受影响 |
| 余额并发操作 | 超扣 / 重复加款 | 幂等 + 行锁（四道防线） |
| 兑换跨币种事务 | 扣了法币但 BSB 没加上 | 同一事务内完成，失败全部回滚 |
| 通道回调丢失 | 用户充值了但没到账 | 补单机制（财务模块负责）+ 对账脚本发现 |
| 需求评审后变动 | 接口/字段调整 | IDL-first + 生成代码，变动成本低 |

# 十四、总结

## 一句话概括

> 以数据库事务为核心，以流水表为载体，以行锁为保障，用最简单的手段解决资金安全问题。

## 关键架构决策（8 项）

```
① 引入 ledger 层 → 资金安全的单一入口，不允许绕过
② 流水表唯一索引 → 一石二鸟（审计 + 幂等防重）
③ 数据库行锁     → 简单可靠，不过度设计
④ BIGINT 整数    → 杜绝浮点精度问题
⑤ 本地事务       → 不引入分布式事务的复杂度
⑥ 缓存只加速读   → 写路径不经过 Redis
⑦ Mock 先行      → 不被财务模块进度阻塞
⑧ 不做复式记账   → 简单方案能解决问题，就是最好的方案
```

## 开发节奏

```
Step 1 基础设施 → Step 2 币种配置（地基）→ Step 3 核心 RPC（安全核心）
→ Step 4 C 端独立功能 → Step 5 充值联调 → Step 6 提现联调 → Step 7 收尾

70% 可独立开发，30% 需要联调
核心代码集中在 Step 3（ledger）和 Step 4（兑换事务）
```

## 技术栈

```
18 项复用 + 1 项新增（shopspring/decimal）+ 6 项明确不引入
复用率 > 95%，工程风险极低
```

*以上就是 ser-wallet 模块的完整实现方案。核心思想是：在资金安全上做到最严格，在工程设计上做到最简洁，能复用的坚决复用，不需要的坚决不做。*
