## 1. 模块定位与职责边界

### 1.1 我负责什么

一句话概括：**我负责平台的"钱袋子"——从用户余额管理到币种配置的完整资金底座**。

具体来说，ser-wallet 是一个独立微服务（编号 022），承担两大职责：

```
ser-wallet (022)
├── 多币种钱包（主业务）
│   ├── 钱包核心 —— 余额管理、资金流水、冻结解冻
│   ├── 充值 —— 创建充值订单、接收上账通知
│   ├── 提现 —— 创建提现申请、冻结余额、确认扣款/解冻
│   ├── 兑换 —— 法币 → BSB 单向兑换
│   ├── 记录 —— 充值/提现/兑换/奖励四类流水查询
│   └── 提现账户 —— 用户绑定的银行卡/电子钱包/USDT 地址管理
│
└── 币种配置（配置支撑）
    ├── 币种管理 —— 后台对币种参数的增删改查
    ├── 汇率引擎 —— 定时拉取三方汇率、偏差检测、平台汇率更新
    └── 基准币种 —— USDT 作为定价锚点，设定后不可变更
```

### 1.2 我不负责什么（但需要联调配合）

**财务模块**（另一位同事负责）：
- 充提通道配置与管理（代收/代付通道）
- 充值/提现方式的 B 端配置（档位、金额范围、图标等）
- 对接三方支付平台（回调接收、验签、轮询）
- 出款执行（调三方代付）
- 风控审核、财务审核
- 人工加减款的操作与审核流程
- 数据统计（通道/充值/提现统计报表）

**关键联动**：财务模块是我最大的上下游依赖方——充值回调后财务调我上账，提现审核后财务调我确认扣款或解冻。我给财务提供 RPC 接口，财务给我提供充提方式配置数据。

### 1.3 为什么是一个服务而不是拆两个

考虑过把钱包和币种配置拆成两个独立服务，但**决定不拆**，原因：

| 维度 | 拆分 | 合并 |
|------|------|------|
| 调用频率 | 钱包每次操作都要查汇率和币种配置，拆开后变成高频 RPC | 内部函数调用，零网络开销 |
| 数据一致性 | 兑换时需要同事务读取汇率 + 操作余额，跨服务做不到 | 同服务同库同事务 |
| 运维成本 | 2 个部署单元、2 套配置、2 套监控 | 1 个部署单元 |
| 团队规模 | 就我一个人开发 | 一个人管一个服务已经足够 |

**结论**：币种配置作为 ser-wallet 的内部子模块，代码分包隔离，部署在同一服务内。这是业务规模和团队规模下的务实选择，未来如果币种配置复杂度上升，可以平滑拆出。

## 2. 我理解的需求全貌

### 2.1 用户侧（C 端）看到什么

```
┌─────────────────── 个人中心 ───────────────────┐
│                                                  │
│  当前币种：VND ▼（可切换 BSB/VND/IDR/THB）       │
│  资金钱包：5,000,000 VND                         │
│  ├── 中心钱包：3,000,000                         │
│  └── 奖励钱包：2,000,000                         │
│                                                  │
│  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐            │
│  │ 充值 │ │ 提现 │ │ 兑换 │ │ 记录 │            │
│  └──────┘ └──────┘ └──────┘ └──────┘            │
└──────────────────────────────────────────────────┘
```

用户在 App/H5/Web 上的操作链路：
- **查看余额**：按币种展示资金钱包（= 中心 + 奖励），可展开看明细
- **充值**：USDT 链上充值（平台承载）/ 银行转账 / 电子钱包扫码（后两者跳三方）
- **提现**：选提现方式 → 选/填账户 → 输金额 → 提交（进入审核流程）
- **兑换**：仅法币 → BSB 单向，按实时汇率换算，有赠送奖励
- **记录**：四个 Tab（充值/提现/兑换/奖励），按当前币种过滤
- **币种切换**：切换后全页面联动刷新数据

### 2.2 管理侧（B 端）看到什么

- **币种配置列表**：币种代码、名称、类型、汇率（实时/入款/出款）、浮动比例、偏值、状态
- **币种编辑**：调整汇率浮动百分比、偏值、图标、启用/禁用
- **基准币种设置**：USDT（一次性设置，锁定后不可更改）
- **汇率日志**：每次平台汇率变动的记录追溯

B 端的充值/提现记录管理、审核操作、人工加减款都在财务模块，不在我这里。

### 2.3 核心业务规则提炼

我从产品文档中提炼出的**绝对不能做错**的规则：

| 规则 | 说明 | 影响 |
|------|------|------|
| 币种隔离 | 每个币种独立余额和账本，绝不跨币种混合操作 | 数据模型基石 |
| 消费优先级 | 先扣中心钱包，不足再扣奖励钱包 | 冻结和扣款逻辑 |
| 余额与流水强一致 | 每次余额变动必须同事务写一条流水 | 审计基石 |
| 幂等性 | 充值回调重试不能重复上账 | 资金安全 |
| 冻结三阶段 | 提现必须"冻结→确认/解冻"，不可直接扣 | 提现安全 |
| 金额全程整数 | 所有金额以最小单位 int64 存储和计算，不用浮点 | 精度安全 |
| 汇率 ×10^8 | 汇率统一乘 10^8 转整数，计算全在整数域 | 精度安全 |

### 2.4 币种体系

| 币种 | 类型 | 精度 | 乘数因子 | 存储示例 |
|------|------|------|---------|---------|
| BSB | 平台币 | 2 位小数 | ×100 | 200.50 BSB → 存 20050 |
| VND | 法币 | 0 位小数 | ×1 | 5,000,000 VND → 存 5000000 |
| IDR | 法币 | 0 位小数 | ×1 | 500,000 IDR → 存 500000 |
| THB | 法币 | 2 位小数 | ×100 | 1,500.50 THB → 存 150050 |
| USDT | 加密货币 | 8 位小数 | ×10^8 | 1.50000000 USDT → 存 150000000 |

BSB 锚定关系：10 BSB = 1 USDT（即 1 BSB = 0.1 USDT）。

## 3. 架构设计与核心决策

### 3.1 整体架构

```
                     客户端（APP / H5 / Web）
                              │
              ┌───────────────┴───────────────┐
              │                               │
       gate-font (C端网关)            gate-back (B端网关)
       /open  /api                    /admin/open  /admin/api
              │ Kitex RPC                     │ Kitex RPC
              └───────────────┬───────────────┘
                              │
                    ser-wallet (022)
         ┌────────────────────┼────────────────────┐
         │   钱包核心层（原子操作）                    │
         │   CreditWallet / DebitWallet             │
         │   FreezeBalance / UnfreezeBalance         │
         │   ConfirmDebit / GetBalance               │
         ├──────────────────────────────────────────┤
         │   业务子模块                               │
         │   充值 │ 提现 │ 兑换 │ 记录 │ 提现账户     │
         ├──────────────────────────────────────────┤
         │   币种配置层                               │
         │   币种 CRUD │ 汇率引擎 │ 基准币种          │
         └──────────────────────────────────────────┘
              │           │           │
         ┌────┘     ┌─────┘     ┌─────┘
    TiDB(db_wallet)  Redis    ETCD
```

### 3.2 分层架构

服务内部我严格按照四层架构来组织代码：

```
Handler 层（接口入口）
  │  接收 RPC/HTTP 请求，参数校验，调用 Service
  ▼
Service 层（业务编排）
  │  编排业务流程，调用多个 Domain/Repository，管理事务
  ▼
Domain 层（领域逻辑）
  │  资金操作原子函数（加钱、扣钱、冻结、解冻、确认扣款）
  │  这一层封装"钱怎么动"的核心逻辑
  ▼
Repository 层（数据访问）
  │  GORM 数据操作，由 gorm_gen 生成标准 CRUD
  │  自定义复杂查询写在这里
  ▼
Infrastructure（基础设施）
  TiDB + Redis + ETCD + 外部 RPC
```

**为什么要单独抽一个 Domain 层？**

因为资金操作（CreditWallet、DebitWallet、FreezeBalance 等）有双重身份：
- 作为 **RPC 接口**：财务模块远程调用（走 Handler → Service → Domain）
- 作为 **内部函数**：充值上账、兑换扣款等内部流程直接调用（Service → Domain）

把这些原子操作收敛到 Domain 层，无论哪条路径进来，核心的"余额变动 + 流水记录 + 冻结处理"逻辑只写一份，保证一致性。

### 3.3 五个核心架构决策

以下是我做的五个最关键的技术决策，每一个都直接影响代码怎么写：

#### 决策一：金额用 int64 最小单位整数，不用 DECIMAL

```
为什么不用 DECIMAL：
  - Go 原生不支持 decimal 类型，需要引入 shopspring/decimal 第三方库
  - 全工程没有任何模块用过 DECIMAL，引入新类型增加团队认知成本
  - int64 最大值 9.2 × 10^18，即使 USDT（×10^8）也能存 92 亿，远超业务需要
  - Stripe、支付宝、微信支付都用整数最小单位方案

做法：
  - VND/IDR：×1（整数币种，1:1 存储）
  - THB/BSB：×100（2 位小数）
  - USDT：×10^8（8 位小数）
  - 汇率统一 ×10^8 转整数
  - 所有金额运算在整数域完成，永远不出现浮点数
```

#### 决策二：并发控制用悲观锁（SELECT FOR UPDATE），不用乐观锁

```
为什么不用乐观锁（version 字段）：
  - 提现冻结场景可能同时操作 2 个钱包（中心 + 奖励），乐观锁重试成本高
  - 钱包操作本身很快（1-2ms SQL），锁持有时间极短
  - 我们是 TiDB，行级锁粒度到用户级别，不同用户互不影响
  - 乐观锁在高冲突场景下退化为忙等待，不如悲观锁直接排队

做法：
  - SELECT ... WHERE user_id=? AND currency_code=? AND wallet_type=? FOR UPDATE
  - 事务内完成：读余额 → 校验 → 更新余额 → 写流水
  - 加锁顺序固定：先按 currency_code 升序，再按 wallet_type 升序（防死锁）
```

#### 决策三：余额 + 流水账本模型，不用复式记账

```
为什么不用复式记账（借贷双向）：
  - 我们的场景是 C 端钱包，不是企业财务系统
  - 用户只关心"我的钱多了还是少了"，不需要看"借方/贷方"
  - 复式记账的复杂度是余额模型的 3-5 倍，但收益在我们场景里很小
  - 业界 C 端钱包标准做法就是 balance + flow（Grab、OVO、ShopeePay 都是）

做法：
  - wallet 表：存当前余额（balance）和冻结金额（frozen_amount）
  - wallet_flow 表：每笔变动一条记录（append-only），记录 before_balance 和 after_balance
  - 通过 flow 表可以重建和审计余额的完整链条
```

#### 决策四：提现采用"冻结三阶段"，不直接扣款

```
为什么不直接扣款：
  - 提现有审核流程（风控 → 财务 → 出款），周期可能几小时甚至几天
  - 如果先扣后退，用户看到余额突然减少会恐慌
  - 如果先不扣后扣，审核期间用户可能把这笔钱花掉导致余额不足

做法（冻结三阶段）：
  Phase 1 — 冻结：提交提现时冻结对应金额
    → balance 不变，frozen_amount 增加
    → 可用余额 = balance - frozen_amount 自然减少
    → 用户看到的余额没变，但不能超额操作

  Phase 2a — 确认扣款：出款成功后
    → balance 减少，frozen_amount 减少（对冲）
    → 钱真正扣走

  Phase 2b — 解冻：审核驳回或出款失败时
    → frozen_amount 减少
    → 可用余额恢复，用户的钱没损失
```

**补充说明**：提现冻结可能跨两个钱包（先冻中心，不够再冻奖励），所以我设计了一张 freeze_detail 表来精确记录每个钱包冻结了多少。解冻和确认扣款时按明细精确操作，不会出错。

#### 决策五：缓存策略用 Cache-Aside + 写后删除

```
为什么不用 Write-Through 或 Write-Behind：
  - 余额数据的一致性要求极高，不能容忍缓存和数据库不一致
  - Write-Through 在事务内写缓存增加了事务复杂度和失败面
  - Write-Behind 延迟写入数据库，资金场景绝对不允许

做法：
  - 读：先查 Redis → miss → 查 TiDB → 回写 Redis（TTL 5 分钟）
  - 写：先写 TiDB（事务内）→ 事务提交后删除 Redis 缓存
  - 下次读自然从 TiDB 加载最新值到 Redis
  - 不主动更新缓存，只做删除，避免双写不一致
```

## 4. 数据模型设计

### 4.1 全局视图

```
db_wallet（数据库）
│
├── 核心表（资金底座）
│   ├── wallet              用户钱包（一行 = 一个用户的一个币种的一种钱包类型）
│   ├── wallet_flow         资金流水（只追加、不修改、不删除 —— 审计基石）
│   └── freeze_detail       冻结明细（跟踪每笔提现的每个钱包冻结了多少）
│
├── 业务表
│   ├── recharge_order      充值订单
│   ├── withdraw_order      提现订单
│   ├── exchange_record     兑换记录
│   └── withdraw_account    提现账户（银行卡/电子钱包/USDT 地址，加密存储）
│
└── 配置表
    ├── currency_config     币种配置（5 行数据：BSB/VND/IDR/THB/USDT）
    ├── rate_log            汇率变动日志
    └── base_currency_config 基准币种配置（1 行数据：USDT）
```

**一共 10 张核心表**。后续根据需要可以补充 withdraw_limit_config（提现限额配置）和 user_currency_preference（用户币种偏好），但核心的 10 张足以支撑全部业务。

### 4.2 核心表的设计要点

#### wallet 表 — 为什么这么设计

```sql
UNIQUE KEY uk_user_currency_type (user_id, currency_code, wallet_type, delete_at)
```

一行 = 一个用户 + 一个币种 + 一个钱包类型（中心/奖励/主播/代理/场馆）。

用 UNIQUE 约束保证物理上不可能出现重复钱包行。这样即使代码有 bug 尝试重复创建，数据库也会拦住。

**数据量预估**：10 万用户 × 5 币种 × 2 类型（中心 + 奖励）= 100 万行。TiDB 轻松承受。

#### wallet_flow 表 — 为什么叫"审计基石"

```
设计核心原则：只追加，不更新，不删除（append-only ledger）
```

每条流水记录 before_balance 和 after_balance，形成链式审计轨迹：

```
Flow #1: before=0,      amount=+100, after=100     (充值 100)
Flow #2: before=100,    amount=+50,  after=150     (充值 50)
Flow #3: before=150,    amount=-30,  after=120     (兑换扣 30)
Flow #4: before=120,    amount=-80,  after=40      (提现 80)
```

任何时刻，当前余额 = 最后一条流水的 after_balance。如果不等，说明系统有 bug。这就是"余额可重建"的审计能力。

**幂等键设计**：每条流水有 `idempotent_key`（UNIQUE 约束），格式为 `{biz_type}:{ref_order_no}:{wallet_type}`。同一笔业务操作即使重试，也只能插入一条流水。

**数据量关注**：这是增长最快的表（每笔操作至少 1 条），日增量预估 50-200 万行。索引设计上优先保证 `(user_id, currency_code, create_at)` 的查询性能。

#### freeze_detail 表 — 为什么需要它

提现 1500 BSB 的场景（中心余额 1200，奖励余额 800）：

```
冻结时：创建 2 条 freeze_detail
  → (wallet_type=1中心, freeze_amount=1200, status=冻结中)
  → (wallet_type=2奖励, freeze_amount=300,  status=冻结中)

出款成功 → 按明细精确扣款
  → 中心: balance -= 1200, frozen_amount -= 1200
  → 奖励: balance -= 300,  frozen_amount -= 300

审核驳回 → 按明细精确解冻
  → 中心: frozen_amount -= 1200（余额恢复可用）
  → 奖励: frozen_amount -= 300（余额恢复可用）
```

没有这张表，解冻时不知道每个钱包冻结了多少，只能猜——这在资金系统里是绝对不允许的。

### 4.3 金额存储规范

全项目统一一套规范，不存在二义性：

| 数据类型 | Go 类型 | DB 类型 | 存储方式 |
|---------|---------|---------|---------|
| 余额/金额 | int64 | BIGINT NOT NULL DEFAULT 0 | 最小单位整数 |
| 汇率 | int64 | BIGINT NOT NULL | ×10^8 |
| 费率/百分比 | int32 | INT NOT NULL DEFAULT 0 | ×10000（如 50 = 0.50%） |
| 赠送比例 | int32 | INT NOT NULL DEFAULT 0 | ×100（如 1000 = 10.00%） |
| 时间戳 | int64 | BIGINT NOT NULL | 毫秒时间戳 |
| 敏感字段 | utils.EncryptedString | VARCHAR(256) | AES-256-CBC 加密 |

## 5. 核心技术难点与应对方案

我梳理了 18 个技术挑战，这里讲 6 个最关键的：

### 5.1 并发余额操作的一致性（P0 致命级）

**风险**：两笔操作同时读到余额 1000，各扣 800，最终余额 200（正确应该第二笔失败）。

**方案**：

```go
// 所有余额变动都在事务内完成
err := db.Transaction(func(tx *gorm.DB) error {
    // 1. 行级锁定钱包行
    var w model.Wallet
    tx.Where("user_id=? AND currency_code=? AND wallet_type=?",
        userId, currencyCode, walletType).
        Set("gorm:query_option", "FOR UPDATE").
        First(&w)

    // 2. 校验余额充足
    available := w.Balance - w.FrozenAmount
    if available < amount {
        return errs.InsufficientBalance
    }

    // 3. 更新余额
    tx.Model(&w).Update("balance", w.Balance - amount)

    // 4. 同事务写流水（before/after 快照）
    flow := model.WalletFlow{
        BeforeBalance: w.Balance,
        AfterBalance:  w.Balance - amount,
        Amount:        amount,
        // ...
    }
    tx.Create(&flow)

    return nil
})
```

**关键保证**：
- 步骤 1-4 在同一个数据库事务内，要么全成功要么全失败
- FOR UPDATE 保证串行化，不会出现"幻读余额"
- before/after 快照保证事后可审计

### 5.2 幂等性保障（P0 致命级）

**风险**：充值回调重试，同一笔充值上账两次，用户凭空多了一倍的钱。

**方案 — 两层防御**：

```
第一层 —— 业务语义检查（快速挡住）：
  查询充值订单状态，如果已经是"支付成功"，直接返回成功，不再操作余额

第二层 —— 数据库 UNIQUE 约束（兜底防线）：
  wallet_flow 表的 idempotent_key 字段有 UNIQUE 索引
  即使第一层漏了，重复插入流水会触发唯一约束冲突，数据库拒绝
  idempotent_key 格式："{biz_type}:{order_no}:{wallet_type}"
  例如："1:R20260226143052123456:1"
```

为什么要两层？第一层性能好（一次 SELECT），第二层绝对安全（数据库硬约束）。单靠任何一层都不够。

### 5.3 冻结操作的消费优先级（P0 致命级）

**规则**：先冻中心钱包，不够的部分再冻奖励钱包。

**实现**：

```go
func freezeWithPriority(tx *gorm.DB, userId int64, currency string, totalAmount int64) ([]FreezeDetail, error) {
    // 1. 按优先级加锁（固定顺序：先中心后奖励，防死锁）
    centerWallet := lockWallet(tx, userId, currency, WalletTypeCenter)
    rewardWallet := lockWallet(tx, userId, currency, WalletTypeReward)

    centerAvail := centerWallet.Balance - centerWallet.FrozenAmount
    rewardAvail := rewardWallet.Balance - rewardWallet.FrozenAmount

    // 2. 校验总可用余额
    if centerAvail + rewardAvail < totalAmount {
        return nil, errs.InsufficientBalance
    }

    // 3. 分配冻结金额
    var details []FreezeDetail
    centerFreeze := min(centerAvail, totalAmount)      // 中心能冻多少就冻多少
    rewardFreeze := totalAmount - centerFreeze          // 剩余的从奖励冻

    // 4. 执行冻结 + 写明细
    if centerFreeze > 0 {
        centerWallet.FrozenAmount += centerFreeze
        details = append(details, FreezeDetail{WalletType: 1, FreezeAmount: centerFreeze})
    }
    if rewardFreeze > 0 {
        rewardWallet.FrozenAmount += rewardFreeze
        details = append(details, FreezeDetail{WalletType: 2, FreezeAmount: rewardFreeze})
    }

    return details, nil
}
```

**关键**：加锁顺序固定（center → reward），永远不会反过来，从根源上杜绝死锁。

### 5.4 汇率精度与计算（P2 重要级）

**做法**：所有汇率 ×10^8 转整数，计算全在整数域完成。

以兑换为例：用户拿 5,000,000 VND 兑换 BSB（汇率 1 BSB = 2,300 VND）

```
存储：exchange_rate = 2300 × 10^8 = 230,000,000,000

计算（全整数）：
  from_amount = 5,000,000 （VND, precision_factor=1）
  to_amount   = from_amount × 10^8 / exchange_rate × to_precision_factor
              = 5,000,000 × 100,000,000 / 230,000,000,000 × 100
              = 217,391 （即 2,173.91 BSB）

赠送 10%：
  bonus_amount = 217,391 × 1,000 / 10,000 = 21,739（即 217.39 BSB）

结果：
  中心钱包 +217,391（2,173.91 BSB）
  奖励钱包 +21,739（217.39 BSB）
```

全程没有浮点数出现。尾差处理策略：**截断（向下取整）**，不是四舍五入——资金系统宁可少给用户 0.01，也不能多给。

### 5.5 防超扣（P0 致命级）

四层防御体系：

```
第一层 — 代码校验：
  if available < amount { return InsufficientBalance }

第二层 — 流水断言：
  assert(before_balance + amount == after_balance)  // 入账
  assert(before_balance - amount == after_balance)  // 出账

第三层 — 数据库约束（防御性兜底）：
  业务层确保 balance 永远 >= 0
  UPDATE wallet SET balance = balance - ? WHERE balance >= ?

第四层 — 定时对账：
  定时任务检查 wallet.balance == SUM(wallet_flow) 重算值
  不一致则告警，人工介入
```

### 5.6 死锁预防（P1 严重级）

**做法**：全服务统一加锁顺序——

```
规则：先按 currency_code 字母升序，再按 wallet_type 数字升序
```

兑换 VND → BSB 场景（需要锁 VND 中心钱包 + BSB 中心钱包 + BSB 奖励钱包）：

```
固定顺序：
  1. BSB center  （B < V，先锁 BSB）
  2. BSB reward  （同 BSB，先 center 再 reward）
  3. VND center  （最后锁 VND）
```

无论哪条代码路径，只要遵守这个规则，不可能产生 A 等 B、B 等 A 的死锁环。

## 6. IDL 接口设计

### 6.1 文件组织

遵循现有工程 IDL 规范，设计 **8 个 Thrift 文件**：

```
common/idl/ser_wallet/
├── service.thrift          服务入口（include 所有 struct 文件，定义全部方法）
├── wallet.thrift           钱包核心结构体（余额查询、币种切换）
├── recharge.thrift         充值相关结构体
├── withdraw.thrift         提现 + 提现账户相关结构体
├── exchange.thrift         兑换相关结构体
├── record.thrift           记录查询相关结构体
├── currency.thrift         币种配置 B 端相关结构体
└── wallet_rpc.thrift       对外提供的 RPC 接口结构体（财务等服务调用）
```

**为什么是 8 个文件？**

- 参考了现有服务的做法：ser-live 用了 9 个文件，ser-user 用了 11 个，ser-app 用了 5 个
- 钱包模块有 7 个子模块，每个子模块的结构体放一个文件，逻辑清晰
- 特别把 `wallet_rpc.thrift` 独立出来（参照 ser-live 的 `live_rpc.thrift`），因为 RPC 接口是给外部服务调的，和 C 端接口的请求响应结构完全不同

### 6.2 接口数量总览

| 分类 | 数量 | 说明 |
|------|------|------|
| C 端接口 | ~17 | 钱包信息、充值、提现、兑换、记录、提现账户 |
| B 端接口 | ~8 | 币种配置 CRUD、汇率日志、基准币种 |
| RPC 接口 | ~12 | 资金操作原语 + 币种信息查询 |
| **合计** | **~37** | |

### 6.3 核心 RPC 接口一览

这是钱包对外提供的核心能力，也是财务模块调我们的接口：

| 接口 | 用途 | 调用方 |
|------|------|--------|
| CreditWallet | 上账（加钱到指定钱包） | 财务（充值成功、补单、人工加款） |
| DebitWallet | 扣款（从指定钱包扣钱） | 财务（人工减款） |
| FreezeBalance | 冻结余额（按消费优先级） | 内部（提现申请时） |

| InitWallet | 初始化用户钱包 | ser-user（注册成功后） |

每个 RPC 接口都满足幂等性——调用 N 次和调用 1 次效果相同。

### 7.3 兑换流程

兑换是最"干净"的流程——**完全在 ser-wallet 内部完成，不依赖任何外部服务**：

```
用户（C端）                    ser-wallet
    │                              │
    │── 获取兑换信息 ─────────────→│
    │                              │ 读取币种配置+当前汇率+赠送比例
    │←── 汇率、赠送规则、余额 ───│
    │                              │
    │── 执行兑换（VND 500万→BSB）──→│
    │                              │ 1. 重新获取最新汇率（防止页面停留期间汇率变化）
    │                              │ 2. 整数域计算兑换金额和赠送金额
    │                              │ 3. 事务（同一个事务，原子操作）：
    │                              │    a. 锁定 VND 中心钱包 → 扣法币
    │                              │    b. 锁定 BSB 中心钱包 → 加 BSB
    │                              │    c. 锁定 BSB 奖励钱包 → 加赠送 BSB
    │                              │    d. 写 3 条资金流水
    │                              │    e. 创建兑换记录
    │                              │ 4. 删除相关余额缓存
    │←── 兑换成功 ────────────────│
```

**关键**：一个事务内操作 3 个钱包，加锁顺序固定（BSB center → BSB reward → VND center），保证原子性且不死锁。

### 7.4 汇率引擎

```
定时任务（每 N 分钟）
    │
    │── 并行请求 3 家三方汇率 API（超时 3-5 秒）
    │
    │── 聚合策略：
    │   3 家都返回 → 取中位数（抗异常值）
    │   2 家返回   → 取均值
    │   1 家返回   → 使用该值但标记"降级"
    │   0 家返回   → 跳过本轮，沿用上次汇率，记录告警日志
    │
    │── 计算偏差：|新汇率 - 当前平台汇率| / 当前平台汇率 × 100%
    │
    │── 偏差 ≥ 阈值？
    │   是 → 更新平台入款/出款汇率 → 写 rate_log → 删除 Redis 汇率缓存
    │   否 → 跳过更新（减少无效变更）
```

入款汇率 = 实时汇率 × (1 + 浮动%)，对用户"贵一点"（平台利润）。
出款汇率 = 实时汇率 × (1 - 浮动%)，对用户"便宜一点"（平台利润）。

## 10. 分阶段实现路径

### 10.1 总体规划

```
Phase 0 ─── 工程骨架搭建 ──────────────────────────────
            注册服务、写IDL、生成代码、建库建表、搭项目骨架
            交付物：服务能启动，健康检查通过

Phase 1 ─── 核心底座（可独立开发）─────────────────────
            钱包核心层 + 币种配置 + 提现账户管理
            交付物：7个RPC接口可调通、B端币种管理可用

Phase 2 ─── 自治业务（可独立开发）─────────────────────
            兑换 + 记录查询 + 汇率定时任务
            交付物：兑换全链路可用、记录页面有数据

Phase 3 ─── 联调业务 ────────────────────────────────
            充值 + 提现（依赖财务模块接口）
            交付物：充提全链路打通
```

### 10.2 各阶段详细说明

#### Phase 0 — 工程骨架

| 步骤 | 内容 | 产出 |
|------|------|------|
| 0-1 | 注册服务编号 022 到 namesp.go | 常量可用 |
| 0-2 | 编写 8 个 IDL 文件 | 接口定义完成 |
| 0-3 | gen_kitex.sh 生成桩代码 | kitex_gen 代码 |
| 0-4 | 创建项目目录结构 + main.go | 服务可启动 |
| 0-5 | 创建 db_wallet + 10 张表 DDL | 数据库可用 |
| 0-6 | gorm_gen 生成模型和查询代码 | CRUD 可用 |
| 0-7 | ETCD 配置写入 | 配置可读 |
| 0-8 | 注册 RPC 客户端 | 其他服务可发现 |
| 0-9 | 注册 gateway 路由 | 网关可转发 |
| 0-10 | go.work 注册 | 编译通过 |

#### Phase 1 — 核心底座

这是最重要的阶段，完成后整个服务的"地基"就稳了：

**1. Domain 层 — 资金操作原子函数（5 个函数）**

| 函数 | 做什么 | 关键保证 |
|------|--------|---------|
| CreditWallet | 加钱到指定钱包 | 事务内：锁行→更新 balance→写 flow |
| DebitWallet | 从指定钱包扣钱 | 事务内：锁行→校验余额→更新 balance→写 flow |
| FreezeBalance | 按优先级冻结 | 事务内：锁行→分配冻结→更新 frozen→写 detail→写 flow |
| UnfreezeBalance | 按明细解冻 | 事务内：查 detail→锁行→减 frozen→更新 detail→写 flow |
| ConfirmDebit | 冻结转扣款 | 事务内：查 detail→锁行→减 balance+frozen→更新 detail→写 flow |

这 5 个函数写好了，整个钱包的资金安全就有保障了。后续所有业务（充值、提现、兑换、人工操作）都是在这 5 个原子函数之上编排。

**2. RPC 接口层（7 个对外接口）**

基于 Domain 层的原子函数，封装成 RPC 接口对外提供：
- CreditWallet / DebitWallet / FreezeBalance / UnfreezeBalance / ConfirmDebit / GetBalance / InitWallet

**3. 币种配置管理（B 端 8 个接口）**

标准的 CRUD + 汇率日志查询 + 基准币种设置。遵循现有工程的 handler → service → repository 分层模式。

**4. 提现账户管理（C 端 3 个接口）**

新增/删除/列表查询。银行卡号和持有人姓名使用 `utils.EncryptedString` 加密存储。

#### Phase 2 — 自治业务

**1. 兑换功能（C 端 2 个接口）**

- GetExchangeInfo：查当前法币余额、BSB 汇率、赠送比例
- DoExchange：执行兑换（一个事务操作 3 个钱包）

兑换完全在 ser-wallet 内部完成，不依赖外部，可以独立测试。

**2. 记录查询（C 端 2 个接口）**

- RecordList：统一入口，type 参数区分四类 Tab
- RecordDetail：根据记录类型查不同的订单详情表

**3. 汇率定时任务**

用内部 goroutine + time.Ticker 实现（不依赖 ser-cron），每 N 分钟拉三方汇率、偏差检测、自动更新。

#### Phase 3 — 联调业务

**1. 充值（C 端 ~5 个接口）**

- 获取充值方式列表（调财务 RPC）
- 创建充值订单 + 请求支付通道（调财务 RPC）
- USDT 充值地址查询
- 重复转账检测
- 奖金列表查询

CreditWallet 在 Phase 1 已完成，财务只需要在回调后调这个 RPC 即可上账。

**2. 提现（C 端 ~2 个接口）**

- 获取提现方式列表（调财务 RPC）
- 创建提现申请（冻结余额 + 创建订单）

FreezeBalance / UnfreezeBalance / ConfirmDebit 在 Phase 1 已完成，财务审核和出款后调这些 RPC 即可。

### 10.3 阶段依赖关系

```
Phase 0 ──→ Phase 1 ──→ Phase 2
                │              │
                └──── Phase 3 ←┘
                      (需要财务IDL就绪)
```

Phase 1 和 Phase 2 之间是线性依赖（2 依赖 1 的 Domain 层），但 Phase 2 和 Phase 3 可以部分并行——Phase 2 的兑换/记录和 Phase 3 的充提可以同时开发，只要财务那边的 IDL 就绪。

## 11. 质量保障与风险控制

### 11.1 资金安全保障体系

```
┌─────────────────────────── 资金安全五重保障 ───────────────────────────┐
│                                                                        │
│  第一重：事务一致性                                                      │
│  → 余额变动 + 流水记录在同一个数据库事务内                                 │
│                                                                        │
│  第二重：幂等双层防御                                                     │
│  → 业务状态检查 + 数据库 UNIQUE 约束                                      │
│                                                                        │
│  第三重：悲观锁串行化                                                     │
│  → SELECT FOR UPDATE 消除并发读写冲突                                     │
│                                                                        │
│  第四重：防超扣四层校验                                                   │
│  → 代码校验 + 流水断言 + SQL条件保护 + 定时对账                            │
│                                                                        │
│  第五重：审计可追溯                                                       │
│  → append-only 流水表 + before/after 快照 + 操作日志                      │
│                                                                        │
└────────────────────────────────────────────────────────────────────────┘
```

### 11.2 测试策略

| 层级 | 覆盖内容 | 方式 |
|------|---------|------|
| 单元测试 | Domain 层 5 个原子函数的正确性 | 构造不同金额/余额组合 |
| 并发测试 | 同一用户同时操作余额 | goroutine 模拟并发请求 |
| 幂等测试 | 同一笔操作重复调用 | 验证余额只变动一次 |
| 边界测试 | 余额为 0、冻结全部余额、刚好扣完等 | 边界值覆盖 |
| 全链路测试 | 充值→余额变动→兑换→提现→余额归零 | 端到端验证 |
| 对账测试 | wallet.balance 是否等于 SUM(flow) 重算 | 定时任务验证 |

### 11.3 监控与告警

| 监控项 | 告警条件 | 说明 |
|--------|---------|------|
| 余额与流水不一致 | 定时对账发现差异 | P0 告警 |
| 汇率拉取全部失败 | 3 家 API 都超时 | P1 告警 |
| 充值上账延迟 | 回调到上账超过 10s | P1 告警 |
| 冻结明细状态异常 | 冻结超过 72 小时未处理 | P2 告警 |
| 提现订单堆积 | 待审核超过 100 笔 | P2 告警（通知财务） |

### 11.4 风险清单与应对

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| 财务 IDL 延迟交付 | 中 | 阻塞 Phase 3 | Phase 1+2 先行，Mock 财务接口 |
| 汇率 API 不稳定 | 中 | 兑换/充提汇率获取失败 | 降级策略（沿用上次汇率） |
| 高并发下数据库锁竞争 | 低 | 接口延迟升高 | 用户级行锁，不同用户互不影响 |
| 需求评审后大幅调整 | 中 | 返工 | 核心模型稳定（余额+流水），变化的多是展示和规则 |

## 总结

用一张图概括我的实现方案：

```
需求理解 ────────────────────────────────────────────────────────
  多币种钱包：充值/提现/兑换/记录/提现账户
  币种配置：币种管理/汇率引擎/基准币种
  财务联动：充值上账/提现冻结确认/人工加减

架构设计 ────────────────────────────────────────────────────────
  单服务（022） │ 四层架构 │ Domain层收敛资金操作

核心决策 ────────────────────────────────────────────────────────
  int64整数 │ 悲观锁 │ 余额+流水模型 │ 冻结三阶段 │ Cache-Aside

数据模型 ────────────────────────────────────────────────────────
  10张核心表 │ wallet/flow/freeze 三表联动

安全保障 ────────────────────────────────────────────────────────
  事务一致 │ 幂等双防 │ 锁串行化 │ 防超扣四层 │ 审计追溯

落地路径 ────────────────────────────────────────────────────────
  Phase 0 骨架 → Phase 1 核心底座 → Phase 2 自治业务 → Phase 3 联调
  60%工作量可独立推进，不被外部依赖阻塞
```

以上是我对 ser-wallet 模块的完整实现方案。核心思路是：**用最简单可靠的技术方案解决资金安全问题，分阶段交付降低风险，先独立后联调保证进度可控**。
