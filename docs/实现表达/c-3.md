## 一、开场：我负责什么，整体是什么

### 1.1 一句话说清楚

我负责的是**多币种钱包系统**的后端实现——让平台用户能用多种货币（法币 VND/IDR/THB、加密货币 USDT、平台币 BSB）进行充值、兑换、提现，并保证每一笔钱的进出都安全、准确、可追溯。

### 1.2 我的职责范围

| 模块 | 定位 | 我的角色 | 接口量 |
|------|------|---------|--------|
| **币种配置** | 基础设施层 | **全部负责** | 21个接口 + 1个定时任务 |
| **多币种钱包** | 核心业务层 | **全部负责** | 43~48个接口 |
| 财务管理 | 运营管理层 | 配合联调 | 对方负责，我暴露6个RPC供其调用 |

**合计需要开发 64~69 个接口**，其中核心（绕不过必须有的）约 58 个，占 85%。

### 1.3 我做的事在整个系统中的位置

```
C端用户（充值/提现/兑换/查看余额）
        │
        ▼
  gate-font 网关（HTTP → RPC转发）
        │
        ▼
┌────────────────────────────────────────────┐
│           ser-wallet（我负责的服务）          │
│                                            │
│  币种配置：提供币种列表、汇率、精度给所有人用  │
│  钱包核心：管理用户余额，执行加减冻结操作      │
│  充值/兑换/提现：完整业务流程                  │
└──────────┬───────────────┬─────────────────┘
           │               │
     查汇率/币种      操作余额/提交审核
           │               │
    ┌──────▼──────┐  ┌─────▼──────┐
    │ 币种配置RPC  │  │ 财务模块    │（他人负责）
    │ （同服务内部）│  │ ser-finance │
    └─────────────┘  └────────────┘
```

**一句话总结我的定位**：币种配置是地基，钱包核心是承重墙，充值/兑换/提现是上面的房间。财务模块是隔壁的房间，我们共用承重墙（E1~E6 资金操作 RPC）。

## 二、需求全貌：三模块定位与职责边界

### 2.1 我对需求的理解

基于 ~90 张需求文档和 ~54 张产品原型的完整分析，这套系统要解决的核心问题是：

1. **多币种充值与支付** — 用户可以用多种货币充值，系统按币种独立管理余额
2. **资产分层** — 5种子钱包（中心/奖励/主播/代理/场馆），避免不同来源的资金混乱
3. **兑换** — 法币→BSB单向兑换，有赠送机制
4. **提现** — 需要 KYC、风控审核、财务审核、通道出款，流程最复杂
5. **可追溯** — 每一笔资金变动都有流水记录，支持对账

### 2.2 五种子钱包体系

这是整个钱包系统最基础的设计，后续所有操作都围绕这个结构展开：

| 子钱包 | 资金来源 | 资金流向 | 当前状态 |
|--------|---------|---------|---------|
| **中心钱包** | 充值、稽核完成的奖励金 | 投注、消费、提现 | 核心 |
| **奖励钱包** | 活动/任务奖励 | 完成稽核→转入中心钱包 | 核心 |
| **主播钱包** | 礼物收益、活动奖励 | 可直接提现或转入中心钱包 | 核心 |
| **代理钱包** | 代理佣金 | 可直接提现或转入中心钱包 | 预留 |
| **场馆钱包** | 中心+奖励钱包转入 | 退出场馆归还 | 预留 |

**关键规则**：
- 资金钱包余额 = 中心钱包 + 奖励钱包
- 消费扣款优先级：先扣中心，再扣奖励
- 混合投注返奖：按投注金额占比分别计入对应钱包

> 依据：需求文档 2-1资金钱包体系.png 系列

### 2.3 币种与汇率体系

| 币种类型 | 示例 | 精度 | 千分位 |
|---------|------|------|-------|
| 法定货币 | VND、IDR、THB | 0~2位小数 | `.` 或 `,` |
| 加密货币 | USDT | 8位小数 | `,` |
| 平台币 | BSB | 2位小数 | `,` |

**汇率三层体系**（这是一个很重要的设计）：
1. **实时汇率** = 3家三方API取均值（参考用，定时刷新）
2. **平台汇率** = 偏差值 ≥ 阈值时才更新（稳定，不频繁跳动）
3. **订单锁定汇率** = 下单时锁定，后续结算以此为准

入款汇率 = 平台汇率 × (1 + 浮动%)，出款汇率 = 平台汇率 × (1 - 浮动%)。

> 依据：需求文档 4-2汇率规则说明.png

### 2.4 与财务模块的职责边界

这一点非常重要，因为钱包和财务是双向依赖关系：

```
钱包 → 财务（同步，用户操作时触发）：
  • 获取充值方式列表
  • 获取提现方式/限额/费率
  • 提交提现订单到财务审核

财务 → 钱包（异步，后台事件触发）：
  • 充值成功 → 调用 CreditWallet 入账
  • 审核驳回/出款失败 → 调用 UnfreezeBalance 解冻
  • 出款成功 → 调用 ConfirmFrozenDeduction 确认扣除
  • 人工加/减款 → 调用 CreditWallet / DebitWallet
```

**形象比喻**：钱包是"执行者"——负责真正操作钱；财务是"决策者"——负责审核、配置通道、决定放款。两边在时间上是分离的，不存在同一个请求里 A→B→A 的循环调用。

## 三、核心业务：我需要实现的五大功能线

### 3.1 功能线总览

```
功能线1：币种配置（21接口 + 1定时任务）
         B端后台管理 + C端展示 + RPC暴露给其他模块
         → 所有人的地基，零依赖，最先做

功能线2：钱包核心 E1~E6（6个RPC + 3个C端接口）
         余额查询 / 加款 / 减款 / 冻结 / 解冻 / 确认扣除
         → 一切资金操作的原子单元，财务模块的硬依赖

功能线3：充值（8个C端接口 + 回调处理）
         USDT/银行/扫码三种充值方式
         → 依赖财务模块提供充值方式列表

功能线4：兑换（2个C端接口）
         法币→BSB单向兑换 + 赠送
         → 最简洁的链路，仅依赖币种配置

功能线5：提现（9个C端接口 + 6个账户管理接口）
         KYC校验 + 冻结 + 财务审核 + 通道出款
         → 最复杂的链路，依赖4个模块
```

### 3.2 接口总量分解

| 类别 | 币种配置 | 钱包模块 | 合计 |
|------|---------|---------|------|
| B端 HTTP | 10 | 0~3 | 10~13 |
| C端 HTTP | 3 | 27 | 30 |
| RPC 暴露 | 7 | 6 | 13 |
| RPC 调用(Client) | 0 | 10 | 10 |
| 回调 | 0 | 0~2 | 0~2 |
| 定时任务 | 1 | 0 | 1 |
| **合计** | **21** | **43~48** | **64~69** |

## 四、技术选型：用什么来做

### 4.1 核心原则：复用现有工程技术栈

我分析了整个工程的 go.mod 依赖和 common/pkg/ 基础设施代码后，结论是：**92% 的技术组件都已经存在，直接复用。只需要新增 1 个 Go 依赖。**

### 4.2 复用的核心技术栈

| 技术 | 版本 | 用途 | 工程中的位置 |
|------|------|------|------------|
| **Kitex** | v0.16.0 | RPC 框架，服务间通信 | common/rpc/ |
| **Hertz** | v0.10.3 | HTTP 网关（gate-font / gate-back） | gate-font/gate-back |
| **GORM** | v1.31 | ORM 框架 | common/pkg/db/ |
| **GORM Gen** | v0.3.27 | 类型安全代码生成 | 各 ser-*/internal/gen/ |
| **TiDB** | — | MySQL 兼容分布式数据库 | ETCD 配置数据库连接 |
| **Redis** (go-redis/v9) | v9 | 缓存、防重、分布式锁 | common/pkg/rds/ |
| **ETCD** | v3.6.7 | 服务注册、配置中心、功能开关 | common/pkg/etcd/ |
| **NATS JetStream** | v1.48 | 异步消息、延迟消息 | common/pkg/nats_cli/ |
| **XXL-JOB** + robfig/cron | — | 分布式 + 本地定时任务 | common/pkg/xxl_client/ |
| **Snowflake** | — | 分布式 ID 生成 | common/pkg/utils/snowflake.go |
| **AES-256-CBC** | — | 敏感字段加密 | common/pkg/utils/security/ |
| **Zap** + Lumberjack | — | 结构化日志 | common/pkg/lg/ |

### 4.3 唯一需要新增的依赖

| 依赖 | 版本 | 用途 | 为什么需要 |
|------|------|------|-----------|
| **shopspring/decimal** | v1.4.0 | 高精度十进制金额计算 | Go 的 float64 有精度丢失问题（0.1+0.2≠0.3），金融系统必须用定点数计算 |

这是钱包系统特有的需求——涉及大量汇率换算、手续费计算、赠送比例计算，必须全链路精确。工程中现有模块不涉及金额计算，所以之前没有这个依赖。

### 4.4 明确不引入的技术

| 不引入 | 原因 |
|--------|------|
| 分布式事务框架（Seata/DTM） | 补偿模式足够覆盖提现场景，引入框架成本远大于收益 |
| 事件溯源 | 流水表已能满足审计追溯需求，事件溯源过度设计 |
| Elasticsearch | TiDB 分布式特性 + 合理索引已够用，后续有瓶颈再引 |
| MongoDB | 统一用 TiDB（MySQL 兼容），不增加存储类型 |

## 五、架构设计：怎么组织代码

### 5.1 整体分层

遵循工程已有模式（参考 ser-app），不创造新的架构范式：

```
┌──────────────────────────────────────────────────────────┐
│ C端(gate-font) / B端(gate-back) — HTTP → RPC 转发         │
└────────────────────────┬─────────────────────────────────┘
                         │ Kitex RPC
┌────────────────────────▼─────────────────────────────────┐
│ handler.go — 纯委派，零业务逻辑                            │
│   每个方法一行代码：return s.xxxService.Method(ctx, req)    │
├──────────────────────────────────────────────────────────┤
│ internal/ser/ — 业务逻辑层（7个Service文件）               │
│                                                          │
│   CurrencyService     — 币种配置B端+C端逻辑              │
│   RateService         — 汇率管理+定时任务                 │
│   WalletCoreService   — E1~E6 核心资金操作 ⚠️最关键       │
│   RechargeService     — 充值业务                          │
│   ExchangeService     — 兑换业务                          │
│   WithdrawService     — 提现业务                          │
│   RecordService       — 记录查询                          │
├──────────────────────────────────────────────────────────┤
│ internal/rep/ — 数据访问层（GORM Gen 生成 + 自定义仓储）   │
├──────────────────────────────────────────────────────────┤
│ internal/cfg/     — MySQL/Redis sync.Once 初始化          │
│ internal/enum/    — 枚举常量（子钱包类型、订单状态等）      │
│ internal/errs/    — 错误码 6022XXX                        │
│ internal/cache/   — 缓存封装（币种/汇率缓存）             │
└──────────────────────────────────────────────────────────┘
```

### 5.2 六条架构原则

| 原则 | 在本系统中的体现 |
|------|-----------------|
| **遵循现有工程模式** | 照着 ser-app 的四层结构复制，不发明新模式 |
| **资金安全第一** | 幂等、事务、防重、审计流水，不可妥协 |
| **不过度设计** | 不引入分布式事务框架、不引入事件溯源 |
| **数据库是最后防线** | 余额变更 + 流水写入必须在同一个 DB 事务内 |
| **最小依赖原则** | 核心资金操作 E1~E6 零外部依赖，纯本地 DB 操作 |
| **渐进式复杂度** | 先用最简单方案跑通，有瓶颈再升级 |

### 5.3 单服务内部分 Package

币种配置和钱包做成**一个 ser-wallet 服务**，内部分 package，不拆成两个服务。理由：

1. 工程已有模式就是一个服务承载多个领域（如 ser-app 含 banner+category 等）
2. 币种配置和钱包数据关联紧密，同服务内方法调用比 RPC 性能更好
3. 减少基础设施开销（一套 ETCD、一个端口、一套 DB 连接池）

> 依据：工程中 ser-app 的目录组织模式

### 5.4 需要注册的基础设施

| 注册项 | 位置 | 内容 |
|--------|------|------|
| 命名空间常量 | `common/pkg/consts/namesp/` | `EtcdWalletService`、`EtcdWalletPort`、`EtcdWalletDb` |
| RPC Client | `common/rpc/rpc_client.go` | `WalletClient()` 单例 |
| ETCD配置 | ETCD服务端 | `/slg/serv/wallet/port`、`/slg/serv/wallet/db` |
| C端网关路由 | `gate-font/biz/router/` | `/api/wallet/...` 系列路由 |
| B端网关路由 | `gate-back/biz/router/` | `/admin/api/wallet/...` 系列路由 |

## 六、数据设计：12张表怎么来的

### 6.1 表结构总览

我把需求文档中的每一个业务实体都映射成了数据库表，共 12 张，按优先级分三级：

```
必须（Must）— 缺了系统跑不起来：
  ① currency_config     币种配置表        ← 需求 2.5.2节 列表20字段
  ② base_currency       基准币种表        ← 需求 2.3节（一次性写入，全表≤1条）
  ③ currency_rate       币种汇率表        ← 需求 2.4节 汇率体系
  ④ wallet_balance      钱包余额表 ⚠️     ← 需求 2.1.2节 五种子钱包
  ⑤ wallet_transaction  资金流水表 ⚠️     ← 需求 3.6节 对账一致性

应该（Should）— 缺了某条业务链路跑不通：
  ⑥ recharge_order      充值订单表        ← 需求 3.2节 + 4.3节
  ⑦ withdraw_order      提现订单表        ← 需求 3.4节 + 4.4节
  ⑧ withdraw_account    提现账户表        ← 需求 3.4.3节
  ⑨ exchange_record     兑换流水表        ← 需求 3.3节
  ⑩ user_currency_pref  用户币种偏好表    ← 需求 3.1.3节

可选（Nice-to-Have）— 缺了不影响核心功能：
  ⑪ rate_log            汇率日志表        ← 需求 2.6节
  ⑫ operation_log       操作日志表        ← 需求 2.7节
```

### 6.2 最核心的两张表

**wallet_balance（钱包余额表）**——整个系统的心脏：

```sql
唯一约束：(user_id, currency_code, wallet_type, delete_at)
核心字段：available_balance DECIMAL(24,8)  — 可用余额
         frozen_balance    DECIMAL(24,8)  — 冻结余额
```

每个用户 × 每种币种 × 每种子钱包 = 一行记录。E1~E6 所有操作都在这张表上执行。

**wallet_transaction（资金流水表）**——审计的命脉：

```sql
幂等唯一索引：(order_no, operation_type, delete_at)
核心字段：operation_type — 操作类型（充值/提现/兑换/冻结/解冻/确认扣除/加款/减款）
         amount         — 变动金额
         before_balance — 操作前余额
         after_balance  — 操作后余额
特征：INSERT ONLY，只写不改，不可变
```

每一次资金变动都在这里留下一条不可篡改的记录。对账、审计、纠纷排查全靠它。

### 6.3 金额字段规范

这是一个全局的硬性规范，贯穿所有表：

| 层级 | 规范 | 原因 |
|------|------|------|
| DB层 | `DECIMAL(24,8)` | 覆盖法币2位和加密8位精度 |
| Go层 | `shopspring/decimal` | 避免 float64 精度丢失 |
| IDL层 | `string` 传输 | 避免 JSON 序列化精度丢失 |
| 禁止 | `FLOAT` / `DOUBLE` / `float64` 做金额运算 | 金融系统铁律 |

> 依据：架构设计 ADR-006 + 业界金融系统标准做法（Stripe、支付宝）

### 6.4 通用字段规范

遵循工程已有规范，每张表都必须包含：
- `id BIGINT AUTO_INCREMENT` — 主键
- `create_by/create_at/update_by/update_at` — GORM 审计插件自动填充
- `delete_at BIGINT DEFAULT 0` — 软删除，所有查询加 `.Where(DeleteAt.Eq(0))`

## 七、IDL设计：接口怎么定义

### 7.1 为什么 IDL 是第一步

工程采用 **IDL 先行开发模式**——先定义 Thrift IDL → `gen_kitex.sh` 生成代码 → 再写实现。不定义 IDL 就没有代码可写。

同时 IDL 是与财务团队的"合同"，双方依赖的 RPC 接口必须先在 IDL 层面锁定。

### 7.2 推荐 8 文件方案

分析了工程中 72 个现有 .thrift 文件和 20 个服务的 IDL 组织方式后，推荐 8 文件方案：

```
common/idl/ser-wallet/
├── service.thrift          # 入口文件，54个方法签名
├── currency.thrift         # 币种配置B端+C端（13方法，~30结构体）
├── currency_rpc.thrift     # 币种RPC R1~R7（7方法，~16结构体）⚠️跨团队合同
├── wallet.thrift           # 钱包核心+E1~E6（9方法，~20结构体）⚠️跨团队合同
├── recharge.thrift         # 充值（8方法，~19结构体）
├── exchange.thrift         # 兑换（2方法，~5结构体）
├── withdraw.thrift         # 提现+账户管理（9方法，~23结构体）
└── record.thrift           # 记录4个Tab（5方法，~15结构体）
```

**为什么是8个而不是更多或更少：**

- 参考 ser-live（60+方法，9个文件）和 ser-user（50+方法，12个文件），我们54个方法用8个文件合理
- `currency_rpc.thrift` 独立是因为它是跨团队合同，财务/竞彩域都要引用
- `exchange.thrift` 虽然只有2个方法但独立是因为兑换是独立业务域
- E1~E6 合并在 `wallet.thrift` 而不独立，因为共享 WalletBalance 等实体

### 7.3 优先锁定的接口

| 优先级 | 接口 | 原因 |
|--------|------|------|
| **P0** | E1~E6（钱包6个RPC） | 财务模块的硬依赖，不锁定他们无法开发 |
| **P0** | F1~F4（财务4个RPC） | 我们充值/提现流程的硬依赖 |
| **P1** | R1~R7（币种配置7个RPC） | 我们自身几乎所有接口都依赖 |

**行动项：评审后第一时间和财务团队对齐 E1~E6 + F1~F4 的 IDL 定义。**

## 八、核心实现：六大资金操作怎么写

### 8.1 为什么 E1~E6 是最关键的代码

`WalletCoreService` 对应的 E1~E6 是整个系统的核心：

- 被财务模块通过 RPC 调用（充值入账、提现冻结/解冻/确认、人工加减款）
- 被钱包内部充值/兑换/提现子模块复用（方法调用，非RPC）
- **所有资金变动都经过这层**，逻辑统一

**不管需求细节怎么变，这6个操作的核心逻辑不会变。**

### 8.2 六个操作的核心设计

```
E1 GetBalance — 查询余额
   输入：用户ID + 币种代码
   输出：各子钱包的可用余额和冻结余额
   要点：纯读操作，查本地表

E2 CreditWallet — 加款 ⚠️最高频
   操作：可用余额 += 金额
   保障：幂等（同一订单号不重复入账）+ 流水记录
   并发：原子SQL → UPDATE balance = balance + amount
   调用方：充值入账、补单、人工加款、兑换入BSB

E3 DebitWallet — 减款
   操作：可用余额 -= 金额
   特殊：余额不足时实际扣款=当前余额（不报错）
   保障：幂等 + 流水
   调用方：人工减款

E4 FreezeBalance — 冻结
   操作：可用余额 -= 金额，冻结余额 += 金额（同行两字段原子变更）
   校验：可用余额 ≥ 冻结金额
   调用方：提现创建时

E5 UnfreezeBalance — 解冻
   操作：冻结余额 -= 金额，可用余额 += 金额
   保障：幂等（同一订单号不重复解冻）
   调用方：审核驳回、出款失败

E6 ConfirmFrozenDeduction — 确认扣除
   操作：冻结余额 -= 金额（资金真正离开用户钱包）
   保障：幂等（同一订单号不重复确认）
   调用方：出款成功
```

### 8.3 三个硬性要求

**（1）幂等性 — 不重复入账不重复扣款**

```
双层保障：
  第一层（应用层）：操作前查流水表
    SELECT 1 FROM wallet_transaction
    WHERE order_no = ? AND operation_type = ? LIMIT 1;
    → 已存在 → 直接返回成功（幂等响应）

  第二层（数据库层）：唯一索引兜底
    UNIQUE KEY (order_no, operation_type, delete_at)
    → 即使并发窗口穿透第一层，INSERT也会唯一索引冲突 → 事务回滚 → 返回成功
```

这是支付宝 `(biz_no, biz_type)` 唯一索引的标准做法，零额外组件依赖。

**（2）事务性 — 余额和流水必须同写同失败**

```
所有操作遵循同一模式：
  query.Q.Transaction(func(tx *query.Query) error {
      // ① 余额变更 (UPDATE wallet_balance)
      // ② 流水写入 (INSERT wallet_transaction)
      return nil  // 两步都成功才提交
  })
```

如果余额改了但流水没写，或者流水写了但余额没改——对账就过不了。所以必须同事务。

**（3）并发安全 — 不丢失更新**

```
E2/E5/E6（简单加减）→ 原子SQL更新
  UPDATE wallet_balance
  SET available_balance = available_balance + ?
  WHERE user_id = ? AND currency_code = ? AND wallet_type = ?;
  → InnoDB UPDATE 自带行级排他锁，同一行并发自动串行

E3（减款需判断余额不足）→ SELECT FOR UPDATE
  BEGIN;
  SELECT available_balance FROM wallet_balance WHERE ... FOR UPDATE;
  → 锁定后做余额校验
  UPDATE ... SET available_balance = ...;
  COMMIT;

E4/W13（多字段/多行变更）→ SELECT FOR UPDATE
  冻结需要同时修改 available_balance 和 frozen_balance
  兑换需要操作两种币种的余额行
```

## 九、关键流程：四条业务链路走读

### 9.1 链路一：兑换（最简洁，最先跑通）

```
用户选择THB→BSB兑换，输入฿1,000
    │
    ├─ 币种配置.R1(GetPlatformRate) → "1 BSB = ฿3.50"
    ├─ 预览：兑换获得 BSB 285.71，赠送10% BSB 28.57，实际到账 BSB 314.28
    │
    ├─ 用户确认兑换
    ├─ 币种配置.R5(GetTransactionRate) → 锁定汇率
    │
    ├─ 一个事务中完成：
    │   ① 扣THB中心钱包 ฿1,000（复用E3逻辑）
    │   ② 加BSB中心钱包 285.71（复用E2逻辑）
    │   ③ 加BSB奖励钱包 28.57（赠送，需完成X倍流水）
    │   ④ 写入兑换流水记录
    │
    └─ 返回"兑换成功，获得BSB 314.28"
```

**为什么最先做兑换**：它只依赖币种配置，不涉及财务模块和三方通道，是第一个能端到端跑通的完整业务链路，可以最早验证 E2/E3 核心逻辑。

### 9.2 链路二：充值（核心流程）

以用户用越南银行转账充 BSB 为例：

```
① 获取充值方式 → 钱包调财务.F1 → 返回可用方式列表
② 用户选银行转账、输入金额、选额外奖金
③ 重复转账检查（本地查订单表）→ 1-2笔软拦截/≥3笔硬拦截
④ 锁定汇率 → 币种配置.R5
⑤ 生成订单(C+16位) → 写入本地DB → 对接三方通道
⑥ 返回三方支付页面 → 用户完成支付

  --- 时间间隔 ---

⑦ 三方通道回调 → 财务模块接收 → 验证金额
⑧ 财务调用钱包.E2(CreditWallet) → 入账
   → 订单金额入中心钱包 + 奖金入奖励钱包
   → 幂等保证：同一订单号不重复入账
```

**关键点**：回调大概率由财务模块接收（通道由财务管理），然后财务调我的 E2 入账。USDT 链上确认可能直接到我这边。无论回调走哪条路，E2 的实现不变。

### 9.3 链路三：提现（最复杂，涉及4个模块）

```
① 查KYC状态 → 用户模块.D9 → 未认证仅USDT提现，认证后全渠道
② 获取提现方式 → 财务.F2 → 按币种+KYC状态过滤
③ 获取限额费率 → 财务.F3 → 单笔限额/单日限额/手续费率/T+N
④ 用户填写账户信息 → 保存到本地 withdraw_account 表
⑤ 输入金额 → 校验限额 → 计算手续费

⑥ 用户确认提现 → 锁汇 → ⚠️冻结余额(E4) → 生成订单(T+16位)
⑦ 提交财务审核 → 财务.F4(SubmitWithdrawOrder)
   → 成功 → 进入审核
   → 失败 → ⚠️必须回滚解冻

  --- 异步等待 ---

⑧ 风控审核 →
   驳回 → 财务调E5(解冻归还) / 通过 → 进入财务审核
⑨ 财务审核 →
   驳回 → 财务调E5(解冻归还) / 通过 → 轮询代付通道出款
⑩ 通道出款 →
   失败 → 财务调E5(解冻归还)
   成功 → 财务调E6(确认扣除) → 资金真正出账
```

**最大风险点**：第⑥步冻结和第⑦步提交之间的一致性——如果冻结成功但提交失败，用户的钱被锁住了。解决方案见下一节。

### 9.4 链路四：人工修正（财务→钱包，单向）

```
B端管理员创建加款/减款单
  ↓
财务调钱包.E1(GetBalance) → 展示各子钱包当前余额
  ↓
管理员按4种子钱包分别填写金额
  ↓
另一管理员审核（审核人≠提交人）
  ↓
审核通过 → 财务调钱包.E2/E3 按子钱包分别操作
```

对我来说只需要保证 E1~E3 的正确性，人工修正的业务流程由财务团队实现。

## 十、技术挑战：四个P0问题怎么解决

### 10.1 挑战全景

我识别了 14 个技术挑战，按风险等级分为：

| 等级 | 数量 | 说明 |
|------|------|------|
| **P0 致命**（处理不好直接资金损失） | 4个 | 并发一致性、幂等性、跨服务一致性、金额精度 |
| **P1 严重**（处理不好业务不可用） | 5个 | 热点行、汇率一致性、回调可靠性、防刷、流水膨胀 |
| **P2 一般**（影响开发效率或运维） | 5个 | 多币种抽象、跨团队联调、可观测性、配置热更新、敏感数据 |

下面重点讲4个P0级别的解决方案。

### 10.2 P0-① 并发场景下的余额一致性

**场景**：充值回调入账(E2) 和 用户提现冻结(E4) 同时到达同一用户。

**方案**：
- E2/E5/E6（简单加减）→ 原子SQL更新，InnoDB 行级锁自动保证
- E3/E4/W13（需要校验或多行操作）→ SELECT FOR UPDATE 悲观锁

**选择原子SQL而不用Redis分布式锁的原因**：
- 数据库行锁已经足够，不增加额外组件依赖
- 同一用户同一时刻的并发操作通常 ≤ 3，这个量级的行锁等待在毫秒级
- 架构设计原则"渐进式复杂度"——先用简单方案，有瓶颈再升级

### 10.3 P0-② 幂等性保证

**场景**：三方充值通道重复回调、财务RPC调用超时重试、前端用户双击。

**方案**：三层防护
1. **前端防抖**：按钮置灰3秒 + debounce
2. **应用层查询**：操作前查流水表是否已处理
3. **数据库兜底**：流水表 `(order_no, operation_type)` 唯一索引

**核心原则**：即使前两层都穿透，数据库唯一索引也能兜住。100%不重复入账。

### 10.4 P0-③ 跨服务一致性（提现冻结-提交问题）

**场景**：冻结余额成功 → 提交财务审核失败/超时。

**方案**：补偿模式（Saga 简化版）

```
正常路径：
  冻结余额(本地) → 提交财务(RPC) → 成功 → 返回

失败路径（明确失败）：
  冻结余额(本地) → 提交财务(RPC) → 失败 → 本地解冻 → 返回失败

超时路径（最复杂）：
  冻结余额(本地) → 提交财务(RPC) → 超时
  → 标记订单 status=SUBMIT_UNCERTAIN
  → 启动异步查询（NATS延迟消息或XXL-JOB定时扫描）
  → 查询财务侧订单是否已接收
    → 已接收 → 更新 status=SUBMITTED
    → 未接收 → 执行解冻补偿

兜底：
  UNCERTAIN 状态超过30分钟 → 触发告警 → 人工介入
```

**不用TCC框架的原因**：只有两方参与（钱包+财务），补偿模式足够覆盖。引入Seata/DTM的运维成本远大于收益。

### 10.5 P0-④ 金额精度安全

**场景**：汇率换算、手续费计算中的精度丢失。

```
Go float64 的问题：
  0.1 + 0.2 = 0.30000000000000004  ← 这就是不能用float的原因
```

**方案**：全链路精确

```
DB     → DECIMAL(24,8)
Go     → shopspring/decimal
IDL    → string
四舍五入 → 法币2位，加密8位，统一用 decimal.RoundHalfUp
```

**代码规范**：代码审查中任何出现 `float64` 做金额运算的代码一律不过。

## 十一、跨团队协作：与财务模块的联调策略

### 11.1 联调三阶段

```
阶段一：IDL接口锁定（评审后第一时间）
  ├── 我暴露的6个RPC (E1~E6) 的 Thrift IDL
  └── 财务暴露的4个RPC (F1~F4) 的 Thrift IDL
  产出：双方IDL文件提交到 common/idl/，各自生成代码

阶段二：Mock独立开发
  ├── 我这边：Mock财务的F1~F4 → 充值/提现流程开发不被阻塞
  └── 财务那边：Mock我的E1~E6 → 审核/入账流程开发不被阻塞

阶段三：真实联调
  重点验证：
  ├── 充值回调入账 → E2被重复调用时余额只增加一次
  ├── 提现完整流程 → 冻结→审核→解冻/确认扣除时序正确
  └── 人工修正 → 加减款余额变更正确
```

### 11.2 关键联调测试点

| 场景 | 验证内容 |
|------|---------|
| 充值入账重复回调 | 同一订单号E2调用多次，余额只增加一次 |
| 提现创建后F4失败 | 冻结的余额能正确回滚解冻 |
| 提现审核驳回 | E5解冻后可用余额恢复 |
| 提现出款成功 | E6确认扣除后冻结余额减少 |
| 人工加款多子钱包 | E2被调用多次，各子钱包余额分别正确 |
| 人工减款余额不足 | E3减款时实际扣款=当前余额不报错 |

### 11.3 Mock策略

在 Service 层通过接口抽象财务调用：

```go
type FinanceRPC interface {
    GetRechargeMethodList(ctx, req) (resp, error)
    GetWithdrawMethodList(ctx, req) (resp, error)
    GetWithdrawConfig(ctx, req) (resp, error)
    SubmitWithdrawOrder(ctx, req) (resp, error)
}

// 开发阶段
type MockFinanceRPC struct{}

// 联调阶段
type RealFinanceRPC struct{ client financeservice.Client }
```

这样切换 Mock/真实只需要换一个实现，不改业务代码。

## 十三、总结

### 13.1 五个核心认知

| # | 认知 | 说明 |
|---|------|------|
| 1 | **币种配置是地基** | 没有币种和汇率，其他模块寸步难行，必须最先做完 |
| 2 | **E1~E6 是承重墙** | 所有资金变动都经过这6个操作，是财务模块的硬依赖 |
| 3 | **提现是最复杂的链路** | 涉及4个模块的双向异步调用，是系统联调的最大风险点 |
| 4 | **幂等是不可商量的硬要求** | 所有资金操作必须幂等，保证重复调用不会重复入账/扣款 |
| 5 | **复用现有技术栈** | 92%的组件已有，只需新增 shopspring/decimal，不造轮子 |

### 13.2 六条设计原则

1. **IDL 先行** — 不定义 IDL 就没有代码可写
2. **遵循现有模式** — 照着 ser-app 复制，不发明新架构
3. **核心先行** — E1~E6 先做好先联调
4. **从简到难** — 兑换 → 充值 → 提现
5. **幂等为王** — 流水表唯一索引是最后防线
6. **事务保一致** — 余额变更和流水写入同一个事务

### 13.3 数字总结
