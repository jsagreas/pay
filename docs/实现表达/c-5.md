# 开场：一句话说清楚我要做什么

我负责 **ser-wallet**（钱包服务）的从 0 到 1 实现，核心是**多币种电子钱包 + 币种配置管理**两大模块，同时需要与财务模块完成双向 RPC 联调。

简单说：**管用户的钱——钱怎么进来（充值）、怎么流转（兑换）、怎么出去（提现）、怎么查（记录）、怎么配（币种/汇率）**。

# 一、我理解的业务全貌

> 这一节说明：我对需求的理解是什么，我的模块在整个系统中的位置

## 1.1 业务定位

ser-wallet 在整个平台中的角色类似**支付宝的"余额"模块**——不直接对接支付渠道（那是财务模块的事），但管理用户的资金账本。

从用户视角看，用了我模块的地方是：

```
打开钱包 → 看余额
充值     → 法币/USDT 进钱包
兑换     → 法币/USDT 换成 BSB（平台币）
提现     → BSB/法币 出钱包
查记录   → 充值/提现/兑换/奖励的流水
```

从后台运营视角看：

```
配币种   → 新增/编辑/启禁用币种（VND/IDR/THB/USDT/BSB）
配汇率   → 设置 "1 BSB = X 法币"
看日志   → 操作日志、汇率变更日志
查用户钱包 → 客服/风控场景查看用户余额和流水
```

## 1.2 核心业务概念

在讲实现之前，先对齐几个核心概念：

**五种币种**：VND（越南盾）、IDR（印尼盾）、THB（泰铢）、USDT（加密货币）、BSB（平台基准币种）。BSB 是平台内部的统一结算单位，用户充值法币/USDT 后兑换成 BSB 用于平台内消费。

**四个子钱包**：每个用户每个币种下有四个子钱包——中心钱包（主钱包）、奖励钱包（活动奖金）、代理钱包（代理佣金）、主播钱包（打赏收入）。不同来源的资金进不同钱包，提现规则也可能不同。

**余额矩阵**：一个用户的余额是 **5 币种 × 4 子钱包 = 最多 20 个余额记录**。这是整个数据模型的基础。

## 1.3 我的职责边界

```
┌──────────────────────────────────────────────┐
│  我负责（主责，完整实现）                       │
│  ① 币种配置模块（B端）：9个B端接口 + 4个RPC接口 │
│  ② 钱包模块（C端+RPC）：21个C端 + 3个B端 + 8个RPC│
│  合计：45个接口                                │
├──────────────────────────────────────────────┤
│  他人负责，我配合联调                           │
│  ③ 财务模块（B端）：36个接口                    │
│  我需要调财务3个RPC，财务需要调我8个RPC          │
│  双向依赖，必须协同                             │
└──────────────────────────────────────────────┘
```

## 1.4 与外部的交互关系

```
          ┌─────────┐
          │ gate-font│ ← C端用户请求（充值/兑换/提现/查余额/查记录）
          └────┬────┘
               │
          ┌────▼────┐      ┌──────────┐
          │         │─RPC→ │ 财务服务   │  获取支付配置 / 创建渠道订单 / 提交审核
          │ser-wallet│      │          │
          │         │←RPC─ │          │  充值入账 / 提现确认 / 人工加减款
          └────┬────┘      └──────────┘

我主动调别人：财务（4个场景）、用户服务（2个场景）、币种配置（内部调用）

```

# 二、技术架构方案

> 这一节说明：我打算用什么技术架构来实现，以及为什么

## 2.1 架构决策总览

先把关键决策亮出来，后面逐一解释：

| 决策项 | 选择 | 一句话理由 |
|--------|------|-----------|
| 服务形态 | **单服务** ser-wallet | 工程惯例，ser-buser 31 方法、ser-live 36 方法都是单服务 |
| 分层架构 | **四层**：Handler → Service → Rep → Model | 照搬 ser-item、ser-buser 的成熟范式 |
| 金额存储 | **BIGINT 最小单位整数** | 避免浮点精度问题，整数运算无丢失 |
| 并发控制 | **乐观锁**（CAS 条件更新） | 无死锁、并发度高、单条 SQL 原子完成 |
| 幂等保障 | **两层防护**（应用层状态检查 + DB 唯一键） | 防重复入账/扣款，金融级底线 |
| 缓存策略 | 币种配置 **Cache-Aside**，余额**不缓存** | 配置读多写少适合缓存，余额必须实时准确 |
| 分布式事务 | **不用框架**，本地事务 + 补偿 | 当前规模不需要 Seata/DTM，补偿机制够用 |
| 新组件引入 | **零** | 现有 common/pkg 完全覆盖所有需求 |

## 2.2 为什么是单服务不拆分

有人可能会问：币种配置和钱包功能不同，要不要拆成两个服务？

**不拆。三个理由：**

第一，**数据内聚**。币种配置和余额在同一个数据库，余额操作需要实时读取币种参数（精度/限额/手续费），放在同一个服务里走本地方法调用，不走 RPC，减少一次网络开销和一个故障点。

第二，**事务需要**。兑换操作涉及两个币种的余额变动（扣法币 + 加 BSB），必须在同一个数据库事务里。如果拆了，就变成分布式事务，复杂度暴增。

第三，**工程一致**。ser-buser（31个方法，管菜单+角色+用户+认证）、ser-app（19个方法，管banner+策略+分类+S3+枚举）都是单服务多职责。45 个方法，在同一量级。

## 2.3 四层架构

```
┌───────────────────────────────────────────────────────┐
│  Handler 层（handler.go）                               │
│  一行委托，不写任何业务逻辑                               │
│  参照 ser-item/handler.go                               │
├───────────────────────────────────────────────────────┤
│  Service 层（internal/ser/*.go）                        │
│  业务编排中心 — 校验、调RPC、计算、发起事务               │
│  8个文件：currency / rate / balance / deposit /         │
│          withdraw / exchange / record / manual          │
├───────────────────────────────────────────────────────┤
│  Repository 层（internal/rep/*.go）                     │
│  数据访问 — 复杂SQL、事务内多表操作                       │
│  基于 GORM Gen 的 query 扩展自定义方法                    │
├───────────────────────────────────────────────────────┤
│  Model 层（internal/gorm_gen/model/*.go）               │
│  自动生成，不手写                                        │
└───────────────────────────────────────────────────────┘
```

**层间调用规则**：
- Handler 只调 Service
- Service 可以调 Service（同模块互调，如 DepositService 调 BalanceService）
- Service 调 Rep（数据访问）、调外部 RPC
- **Rep 不调 RPC** — 数据层不能有远程调用，保证事务内不包含 RPC

这条规则特别重要：**RPC 在事务之外做，DB 操作在事务之内做**。

## 2.4 技术栈 — 全部复用，零引入

我扫了一遍 common/pkg 全量代码，确认 ser-wallet 不需要引入任何新组件：

| 需求 | 用什么 | 来源 |
|------|--------|------|
| RPC 框架 | Kitex + Thrift IDL + gen_kitex.sh | 已有 |
| HTTP 网关 | Hertz（gate-back + gate-font） | 已有 |
| 数据库 | TiDB + GORM + Gen + UserPlugin | 已有 |
| 缓存 | Redis（common/pkg/rds/） | 已有 |
| 配置/注册 | ETCD | 已有 |
| 订单号生成 | Snowflake（utils.GenerateSnowflakeID） | 已有 |
| 敏感数据加密 | AES EncryptedString（utils/crypto_util） | 已有，用于银行卡号等 |
| 错误码 | ret + errs + i18n | 已有 |
| 审计日志 | ser-blog RPC（AddActLog） | 已有 |
| 链路透传 | tracer（metainfo 8 个 Header） | 已有 |

**不需要 Kafka**（没有异步消息场景）、**不需要 ES**（没有全文搜索）、**不需要 Seata/DTM**（本地补偿够用）。

# 三、数据模型设计

> 这一节说明：我设计了哪些表、为什么这么设计、核心表长什么样

## 3.1 全局表设计

三个模块共需 21 张表，其中我负责 11 张：

```
币种配置模块（4张，我负责）          钱包模块（7张，我负责）
───────────────────────         ───────────────────────
[必须] currency_config 币种配置   [必须] user_balance    余额表（心脏）
[必须] exchange_rate   汇率配置   [必须] balance_flow    流水表（账本）
[应该] currency_log    配置日志   [必须] deposit_order   充值订单
[应该] rate_log        汇率日志   [必须] withdraw_order  提现订单
                                 [必须] exchange_order  兑换订单
                                 [应该] withdraw_account 提现账户
                                 [可选] reward_flow_rule 奖励规则

财务模块（10张，他人负责）
───────────────────────
payment_channel / payment_config / finance_deposit
finance_withdraw / manual_adjustment / ...
```

## 3.2 核心表：user_balance（余额表）

这是整个系统的心脏，所有业务最终都操作这张表。

```
user_balance
──────────────────────────────
user_id     BIGINT NOT NULL          ┐
currency    VARCHAR(10) NOT NULL     ├─ 三元组唯一键
wallet_type TINYINT NOT NULL         ┘
available   BIGINT NOT NULL DEFAULT 0    可用余额（最小单位整数）
frozen      BIGINT NOT NULL DEFAULT 0    冻结余额

UNIQUE KEY (user_id, currency, wallet_type)
```

**三个关键设计决策**：

**决策一：金额用 BIGINT，不用 DECIMAL**

```
所有金额存储为最小单位整数：
  "10.50 BSB"（精度2位） → 数据库存 1050
  "100000 VND"（精度0位）→ 数据库存 100000
  "50.00 USDT"（精度2位）→ 数据库存 5000

理由：
  ① 整数加减无精度丢失（0.1 + 0.2 ≠ 0.3 的问题不存在）
  ② 比 DECIMAL 比较和运算更快
  ③ 展示层做精度换算：available ÷ 10^precision = 展示金额
  ④ 现有工程所有 ID/时间戳都是 int64，风格一致
```

**决策二：余额行设计为 user_id + currency + wallet_type 三元组**

不是一行存一个用户的所有余额（那样币种扩展要加列），而是每个用户每个币种每个子钱包一行。好处是币种可动态新增，不影响表结构。

**决策三：available 和 frozen 两个字段在同一行**

提现冻结时需要原子地 "available 减 + frozen 加"，在同一行更新最简单。如果拆成两张表就需要跨表事务。

## 3.3 核心表：balance_flow（流水表）

```
balance_flow
──────────────────────────────
user_id        BIGINT NOT NULL
currency       VARCHAR(10)
wallet_type    TINYINT
change_type    TINYINT NOT NULL     变动类型（充值/提现/兑换/人工...枚举）
amount         BIGINT NOT NULL      变动金额（正=增 负=减）
before_balance BIGINT NOT NULL      变动前余额
after_balance  BIGINT NOT NULL      变动后余额
ref_order_no   VARCHAR(32)          关联订单号

UNIQUE KEY (ref_order_no, change_type)    ← 幂等兜底
INDEX (user_id, currency)                 ← 记录查询
```

这张表的核心价值有三个：
1. **审计追溯**：每笔余额变动都有完整记录，可追溯到具体操作
2. **对账校验**：相邻两条流水的 after_balance(上一条) = before_balance(下一条)，不等就说明有异常
3. **幂等兜底**：ref_order_no + change_type 的唯一键，数据库层面阻止重复写入

## 3.4 订单表简述

```
deposit_order  充值订单 — 状态机：待支付 → 成功/失败/超时/补单成功
withdraw_order 提现订单 — 状态机：审核中 → 成功/失败
exchange_order 兑换订单 — 状态机：成功/失败（即时完成）
```

三张订单表结构类似：平台订单号（雪花 ID）+ 用户 + 币种 + 金额 + 状态 + 方式 + 关联信息。不展开细节，表结构设计文档里有完整字段。

# 四、核心技术方案

> 这一节说明：钱包系统最核心的技术难点我怎么解决

钱包系统的技术核心就三个字：**不出错**。余额不能多也不能少，流水不能漏也不能重。下面说我怎么保证。

## 4.1 并发安全 — 乐观锁

**问题**：同一用户同时发起兑换和提现，两个请求并发操作同一行余额记录。

**方案**：在 UPDATE 语句中加条件判断，让数据库保证原子性。

```sql
-- 扣减余额：一条SQL完成"检查 + 扣减"
UPDATE user_balance
SET available = available - ?
WHERE user_id = ? AND currency = ? AND wallet_type = ?
  AND available >= ?     -- 关键：数据库层保证不超扣

-- 检查受影响行数：0 = 余额不足（被并发请求先扣了），1 = 成功
```

**为什么不用 SELECT FOR UPDATE（悲观锁）**：
- 钱包的余额操作全是简单加减，不需要"先读再算再写"
- 乐观锁无死锁风险、锁持有时间极短、并发度高
- 工程代码中 ser-item 也没用悲观锁，风格一致

## 4.2 幂等保障 — 两层防护

**问题**：财务调我的"充值入账"接口，网络超时重试了一次，如果不做幂等，同一笔充值入账两次 = 直接资损。

**方案：两层防护，正常情况快速返回，极端并发数据库兜底。**

```
第一层（应用层，快速返回）：
  进入方法 → 用订单号查状态 → 已入账 → 直接返回"成功"

第二层（数据库层，兜底）：
  balance_flow 表的 (ref_order_no, change_type) 唯一键
  即使第一层被并发穿透，写流水时唯一键冲突 → 捕获异常 → 返回"成功"
```

所有 8 个 RPC 暴露的写接口全部遵循这个模式：

| 接口 | 幂等键 |
|------|--------|
| 充值入账 | 充值订单号 |
| 提现确认 / 解冻 | 提现订单号 |
| 人工加款 / 减款 / 补单 | 修正单号 |

## 4.3 事务边界 — 铁律三条

**铁律一：余额变动 + 订单状态 + 流水记录，必须同一个数据库事务**

不允许出现"余额加了但流水没写"或"流水写了但订单状态没改"。

**铁律二：扣减操作必须用乐观锁**

绝不能先 SELECT 余额 → 应用层判断够不够 → 再 UPDATE。并发场景必出超扣。

**铁律三：RPC 调用在事务之外**

```
正确：获取汇率(RPC) → 计算 → [开事务] → 操作余额 → 写流水 → [提交事务]
错误：[开事务] → 获取汇率(RPC) → 操作余额 → [提交事务]
```

RPC 可能超时、可能重试，放在事务内会拉长事务持有时间，导致数据库连接池耗尽、锁等待升高。这是现有工程的统一范式——ser-item 的事务内只有 GORM 操作，没有 RPC。

## 4.4 余额操作的统一抽象

充值入账、兑换扣减、兑换增加、提现冻结、人工加款……本质都是"给某个子钱包的余额做加减 + 写一条流水"。

我的方案是在 balance_service 层抽象一个**统一余额变动方法**：

```
输入：用户ID + 币种 + 子钱包类型 + 变动金额 + 变动类型 + 关联订单号
操作：
  1. 更新余额（带乐观锁）
  2. 写流水（记录变动前后余额）
保证：步骤 1 和 2 在同一事务内
```

上层业务场景只需组装参数调用：

```
充值入账 → [{中心钱包 +100, 类型=充值}]
兑换     → [{VND中心 -230000, 类型=兑换扣减}, {BSB中心 +10, 类型=兑换增加}]
提现冻结 → [{中心钱包 available-500 frozen+500, 类型=提现冻结}]
```

**好处**：余额操作逻辑只写一处，所有场景复用。事务边界、乐观锁、流水写入集中管理，测试也只需覆盖一个核心方法。

# 五、六条业务流程的实现方案

> 这一节说明：每条核心流程我怎么实现，重点讲关键节点和难点

## 5.1 流程线一：币种配置管理（B端，最基础，最先做）

**特点**：纯 CRUD，无外部依赖，整个系统的配置地基。

**实现要点**：
- 币种 CRUD：标准分页查询 + 新增 + 编辑 + 启禁用，参照 ser-buser 的 RBAC 管理风格
- 汇率设置：修改汇率时写变更日志（旧值/新值/操作人），格式统一为 "1 BSB = X 法币"
- 缓存：Redis Cache-Aside，币种列表 TTL 5 分钟，汇率 TTL 1-3 分钟。B 端修改后主动删缓存，下次查询自动重建
- RPC 暴露：4 个只读接口（获取可用币种列表/币种详情/汇率/配置参数），供钱包模块和财务模块调用

**没有难点，就是标准的 B 端管理模块。**

## 5.2 流程线二：钱包首页 + 余额查询（C端，纯读取）

**特点**：纯读操作，无事务，无外部依赖。

**实现要点**：
- 余额查询：用户传当前币种 → 查 user_balance 表 → 返回 4 个子钱包的可用/冻结余额
- 可用币种列表：查币种配置（走缓存）+ 查该用户每个币种下的余额合计 → 返回列表
- 币种切换：前端行为，后端只是换个参数重新查
- user_id 从 ctx 获取（gateway JWT → metainfo → tracer.GetUserId），IDL 不暴露

## 5.3 流程线三：充值（C端，跨模块，五阶段）

**特点**：我负责"头和尾"（发起 + 入账），中间的渠道对接交给财务。

```
[阶段一] 进入充值页
  → 调财务RPC获取可用充值方式 → 透传给C端

[阶段二] 用户选方式、输金额、点确认
  → 参数校验（币种/金额/限额）
  → 重复转账拦截（同金额同币种处理中订单数）
  → 生成本地充值订单（状态=待支付）
  → 调财务RPC创建渠道支付订单 → 拿到支付信息返回C端
  ⚠ 注意：此时用户余额不变，充值是先支付后入账

[阶段三] 等待支付 → 渠道回调 → 财务确认 → 调我的RPC

[阶段四] 充值入账（我暴露的RPC接口，财务调我）
  → 幂等校验（订单号查状态）
  → 事务内：更新订单状态=成功 + 中心钱包余额增加 + 写流水
  → 如有奖金活动：奖励钱包余额增加 + 写奖金流水

[阶段五] C端轮询/推送确认到账
```

**关键难点**：阶段四的入账操作必须幂等（财务可能因超时重试），必须在事务内保证余额+订单+流水三者一致。

## 5.4 流程线四：兑换（C端，完全闭环，最简单的交易）

**特点**：只依赖汇率，不跨模块，全在本服务内完成。是我**最先验证的交易链路**。

```
[步骤一] 获取汇率预览
  → 查汇率（走缓存或DB）
  → 返回 "1 BSB = 23000 VND" + 赠送比例

[步骤二] 用户输金额，确认兑换
  → 重新获取最新汇率（不用页面上的旧汇率）
  → 金额计算：230000 VND ÷ 23000 = 10 BSB，赠送 5% = 0.5 BSB
  → 余额预检：VND 可用余额 ≥ 230000？

[步骤三] 事务操作（核心）
  ┌─ 事务 ──────────────────────────┐
  │ ① VND 中心钱包 available - 230000  │
  │ ② BSB 中心钱包 available + 10     │
  │ ③ BSB 奖励钱包 available + 0.5    │
  │ ④ 写兑换订单记录                   │
  │ ⑤ 写余额变动流水（3条）             │
  └──────────────────────────────────┘
```

**为什么兑换最先做**：它涉及两个币种余额变动、事务操作、流水记录、乐观锁——覆盖了余额操作的所有核心逻辑，但不依赖任何外部模块。如果兑换跑通了，充值入账和提现扣款只是同一套逻辑的不同参数。**兑换是我的烟囱测试。**

## 5.5 流程线五：提现（C端，跨模块最复杂）

**特点**：跨度最长、模块最多、状态流转最复杂。我的代码负责头（发起+冻结）和尾（确认/解冻）。

```
[阶段一] 进入提现页（并行发4个请求）
  → KYC状态查询（调用户服务RPC，非USDT提现必须KYC）
  → 可用提现方式（调财务RPC + 按充值来源过滤）
  → 提现限额查询（读币种配置）
  → 提现账户信息（读本地表，首次填写后回显）

[阶段二] 确认提现（最复杂的订单创建）
  → 并行校验：限额 + KYC户名比对 + 余额
  → 冻结余额：available 减 + frozen 加（乐观锁）
  → 生成提现订单（状态=审核中）
  → 调财务RPC提交审核
  ⚠ 如果提交审核RPC失败：必须解冻余额（补偿）

[阶段三] 财务三级审核 + 渠道出款（我不参与）

[阶段四] 财务回调我（两种结果）
  → 审核通过+出款成功 → 调我的"提现确认" → 冻结转正式扣除
  → 审核拒绝/出款失败 → 调我的"提现解冻" → 冻结退回可用
```

**关键难点**：

**冻结→提交审核的补偿问题**：冻结余额成功后，如果调财务提交审核 RPC 失败，用户的钱就"卡住了"。我的方案：立即解冻 + 订单标记失败 + 写回退流水。如果解冻也失败（极端情况），记日志告警，人工介入。

**提现方式过滤**：合规要求"钱从哪进从哪出"。用户只充过法币 → 只保留法币提现方式。只充过 USDT → 只保留 USDT。混合充值 → 以法币为准。我通过查本地充值记录来判断用户的充值来源。

## 5.6 流程线六：RPC 被调用（财务调我的 8 个接口）

**特点**：被动接收调用，执行余额操作。所有接口遵循统一模式。

```
8个接口的统一处理模式：

  ① 幂等校验（订单号/修正单号查状态，已完成→直接返回成功）
  ② 参数校验（用户/币种/金额/子钱包类型）
  ③ 事务操作（余额变动 + 订单状态更新 + 写流水）
  ④ 返回成功/失败

不同接口只是参数不同：
  充值入账    → available + amount
  提现确认    → frozen - amount
  提现解冻    → available + amount, frozen - amount
  人工加款    → available + amount（指定子钱包）
  人工减款    → available - amount（指定子钱包，带乐观锁）
  人工补单    → available + amount（关联原充值订单）
  余额查询    → 纯读取
```

## 7.2 为什么这个顺序

```
第一期（骨架）→ 不依赖任何人，产出 IDL 给财务
第二期（币种）→ 不依赖任何人，产出被其他模块依赖的基础数据
第三期（核心）→ 低依赖，兑换验证核心逻辑，RPC接口给财务联调
第四期（充提）→ 强依赖放最后，可用mock先行，财务就绪后联调
```

核心思路：**先把不依赖别人的做完，再把别人依赖我的做好，最后做我依赖别人的**。

# 八、与财务模块的协同方案

> 这一节说明：我和财务怎么配合，什么时间点对什么

## 8.1 双向 RPC 依赖全景

```
我调财务（4个场景）                财务调我（8个接口）
─────────────────                ─────────────────
支付配置查询(充值/提现方式)        充值入账
创建渠道支付订单                  提现确认扣款
提交提现审核                      提现失败解冻
通道可用性查询                    人工加款 / 减款 / 补单
                                 余额查询
```

## 8.2 协同时间线

```
[评审通过后立即]
  我：提交 IDL + 生成桩代码 → 财务 pull 代码后可编译 rpc.WalletClient()
  财务：提供支付配置查询 IDL → 我 pull 后可编译 rpc.FinanceClient()

[第一二期]
  各自独立开发，不互相阻塞

[第三期]
  我：完成全部8个RPC接口实现 → 通知财务可以调了
  财务：可以开始对接我的入账/扣款/解冻接口

[第四期]
  联调：充值全链路（用户支付 → 渠道回调 → 财务确认 → 我入账）
  联调：提现全链路（我冻结 → 提交审核 → 三级审核 → 我确认/解冻）
  联调：人工修正（财务创建 → 审核 → 调我加款/减款/补单）
```

## 8.3 IDL 先行策略

这是 CloudWeGo 工程体系的核心优势：**接口定义先行，实现可以后做**。

我第一期就把全部 46 个方法的 IDL 定义好，gen_kitex.sh 生成桩代码，提交到 common 仓库。财务那边 pull 后就有了 `rpc.WalletClient()` 的类型定义，可以开始写调用代码了——虽然我的服务还没跑起来。

同理，财务提交他们的 IDL 后，我也可以基于 `rpc.FinanceClient()` 的类型定义先写调用代码，用 mock 返回值开发。联调时只需要替换 mock。

# 九、风险识别与应对

> 这一节说明：我预见到什么风险，打算怎么应对

## 9.1 技术风险

| 风险 | 影响 | 应对 |
|------|------|------|
| **并发余额操作超扣** | 资损 | 乐观锁（数据库层条件更新），100% 防超扣 |
| **重复入账/扣款** | 资损 | 两层幂等（应用层 + DB唯一键），100% 防重复 |
| **提现冻结后审核RPC失败** | 用户资金卡死 | 即时补偿解冻 + 日志告警 |
| **事务内含RPC导致连接池耗尽** | 系统不可用 | 铁律：RPC 在事务外，事务只做DB操作 |
| **金额精度丢失** | 对账差异 | BIGINT 整数存储，全链路无浮点运算 |

## 9.2 协调风险

| 风险 | 影响 | 应对 |
|------|------|------|
| **财务接口迟迟不出** | 充值/提现无法联调 | 一二三期不依赖财务先推进，第四期用mock先行 |
| **需求评审后细节变更** | 部分返工 | 骨架（表结构/IDL方法签名）大概率不变，字段级调整成本低 |
| **双方接口理解不一致** | 联调时发现对不上 | IDL先行 + 评审后立即对齐接口协议 |

## 9.3 我需要评审时确认的关键问题

有几个问题如果理解偏了会直接导致返工，我已经整理在评审反馈文档中，这里列最核心的三个：

1. **四个子钱包的资金流转规则**（充值入哪个、提现从哪个、钱包间能不能互转）
2. **每个币种的精度定义**（上线后不可修改，影响所有金额计算）
3. **提现手续费从余额扣还是从提现金额扣**（影响全部提现相关逻辑）

                    ┌─────────────────────────────────────┐

    │ Redis缓存    │          │ 兑换=烟囱测试 │          │ mock先行    │
    │ 零外部依赖   │          │ 低外部依赖   │          │ 强外部依赖   │
