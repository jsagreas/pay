## 一、我要做什么

### 1.1 一句话定位

我负责的是：**给平台做一套多币种钱包系统**——让用户能充钱、换钱、提钱、查记录，让运营能配置币种和汇率。

这是一个资金系统，核心要求就一句话：**钱不能多、不能少、不能乱**。

### 1.2 我的职责范围

| 模块 | 端 | 做什么 | 谁用 |
|------|-----|--------|------|
| **多币种钱包** | C端 | 余额管理、充值、兑换、提现、交易记录 | 终端用户 |
| **币种配置** | B端 | 币种 CRUD、汇率同步、汇率日志 | 运营人员 |

这两个模块在同一个服务 `ser-wallet` 里实现，代码层面按包隔离。

### 1.3 和其他模块的关系

```
我做的                                    别人做的
┌──────────────┐                        ┌──────────────┐
│  币种配置     │───提供币种和汇率──────→ │              │
│  (B端)       │                        │   财务管理    │
│              │                        │   (B端)      │
├──────────────┤                        │              │
│  多币种钱包   │←──充值上账/冻结/扣款──│              │
│  (C端)       │──→查充值方式/提现方式─→│              │
└──────────────┘                        └──────────────┘
```

**一句话概括分工**：币种配置提供"尺子"（汇率和币种规则），财务管理负责"搬运"（通道收付款和审核），我的钱包负责"记账"（余额增减和流水）。

**为什么放一个服务**：币种数据被钱包高频使用（每次余额展示、每次兑换都要读汇率），放同一个服务内部函数调用零网络开销。工程内 ser-app 也是多域合一的先例。

## 二、系统怎么运转

### 2.1 资金只有三个流向

不管业务多复杂，所有操作归结为三种资金流动：

```
① 外部 → 用户钱包    充值、奖励、补单、人工加款
② 用户钱包 → 外部    提现、人工减款
③ 钱包内部流转        法币兑换BSB、子钱包间调拨
```

### 2.2 钱包体系：一个用户有多个"口袋"

用户的钱不是一个总余额，按用途分成子钱包：

| 子钱包 | 用途 | 首版状态 |
|--------|------|---------|
| 中心钱包 | 主钱包，充值/投注/消费/提现 | 实现 |
| 奖励钱包 | 活动奖金，有流水要求才能提现 | 实现 |
| 主播钱包 | 打赏收益 | 实现 |
| 代理钱包 | 代理佣金 | 预留 |
| 场馆钱包 | 游戏场馆临时钱包 | 预留 |

**关键规则**：用户看到的"余额" = 中心 + 奖励。花钱时先扣中心，不够再扣奖励。

### 2.3 币种体系

| 类型 | 币种 | 特点 |
|------|------|------|
| 法币 | VND(越南盾) / IDR(印尼盾) / THB(泰铢) | 各国货币，展示规则不同 |
| 加密货币 | USDT | 链上转账，精度 8 位 |
| 平台币 | BSB | 10 BSB = 1 USDT，平台内流通 |

基准币种 USDT，设置后不可更改。所有统计和对账都以 USDT 为基准。

### 2.4 汇率体系（四种汇率）

```
实时汇率 ← 定时从3家三方API取均值
    │
    │ 偏差超阈值时自动同步
    ▼
平台汇率 ← 对外展示和计算的基准
    │
    ├──→ 入款汇率 = 平台 × (1 + 浮动%)    ← 用户充值时用
    └──→ 出款汇率 = 平台 × (1 - 浮动%)    ← 用户提现时用
```

不是每次轮询都更新，只有偏差超过后台配置的阈值才触发更新，避免频繁写入。

### 2.5 三大业务链路总览

```
充值：用户付款 → 三方收款 → 财务验签 → RPC调钱包上账 → 余额增加
提现：用户申请 → 钱包冻结 → 财务三级审核 → 通道出款 → 成功:扣款 / 失败:解冻
兑换：用户确认 → 钱包内部扣法币 + 加BSB（同一事务，不过外部服务）
```

## 三、我怎么设计

### 3.1 架构分层

遵循工程已有的分层规范（ser-app/ser-item 模式）：

```
handler.go                       ← IDL 接口实现，零逻辑，纯委托
    ↓
internal/ser/                    ← 业务逻辑层
    ├── currency_service.go      ← 币种配置
    ├── wallet_service.go        ← 钱包C端功能
    ├── balance_engine.go        ← 余额变更统一入口（核心中的核心）
    └── exchange_service.go      ← 兑换
    ↓
internal/rep/                    ← 数据访问层（GORM Gen 查询 + 事务）
    ↓
internal/domain/                 ← 外部服务封装
    ├── finance_rpc.go           ← 调财务RPC
    └── kyc_rpc.go               ← 调KYC RPC
```

**BalanceEngine 是核心设计**——所有余额变更操作（充值上账、冻结、解冻、扣款、加款、减款、兑换）都经过这一个入口。不允许任何地方绕过它直接改余额。好处是：安全逻辑（行锁、乐观锁、幂等校验、流水写入）只写一遍，不重复。

### 3.2 数据模型

**10 张表，160+ 字段，27 个索引**：

```
币种模块（2张）：
  currency_config     ← 币种定义+汇率（合一张表，总共4-6条数据，全量Redis缓存）
  rate_change_log     ← 汇率变更日志（只增不改，B端汇率日志页的数据源）

钱包模块（6张必须 + 1张应该 + 1张可选）：
  wallet              ← 核心：一条记录 = 一个用户 + 一个币种 + 一个钱包类型
  wallet_transaction  ← 交易流水：每笔余额变动的完整记录（含before/after）
  deposit_order       ← 充值订单（5种状态：待支付→进行中→成功/失败/过期）
  withdrawal_order    ← 提现订单（6种状态：处理中→审核中→出款中→成功/失败/拒绝）
  withdrawal_account  ← 提现账户（银行卡/电子钱包，敏感字段AES加密）
  exchange_order      ← 兑换订单（含汇率快照和赠送金额）
  wager_requirement   ← 流水稽核（应该级，奖励钱包的流水要求追踪）
  wallet_daily_snapshot ← 日快照（可选级，用于历史余额回溯）
```

**关键设计决策**：

| 决策 | 方案 | 理由 |
|------|------|------|
| 金额存储 | bigint 最小单位（分/聪） | 整数运算无精度丢失，业界标准（支付宝/Stripe） |
| 汇率存储 | bigint × 10^8 | 避免浮点，保留 8 位小数精度 |
| 币种+汇率 | 合一张表 | 1:1 关系，数据量极少（4-6 条），拆表无收益 |
| 流水记录 | before/after 快照 | 同事务写入，对账的安全网 |
| 敏感数据 | EncryptedString(AES-256) | GORM 自动加解密，业务层无感知 |

### 3.3 接口设计

**IDL 先行，6 个 thrift 文件，35 个接口**：

```
common/idl/ser-wallet/
├── service.thrift         ← 入口：WalletService，35个方法
├── model.thrift           ← 公共子结构体
├── currency.thrift        ← 币种配置域（10个接口）
├── wallet.thrift          ← 钱包核心域（14个接口）
├── transaction.thrift     ← 交易记录（2个接口）
└── rpc.thrift             ← RPC供财务调用（8个接口）
```

**接口分布**：

| 分类 | 数量 | 说明 |
|------|------|------|
| B端接口（币种配置） | 6 | CRUD + 设基准 + 汇率日志 |
| C端接口（钱包功能） | 14 | 余额2 + 充值5 + 兑换2 + 提现5 |
| C端接口（交易记录） | 2 | 列表 + 详情 |
| RPC供财务调用 | 8 | 上账/冻结/解冻/扣款/加减款/补单/查余额 |
| 公共枚举 | 1 | GetEnumOptions |
| 定时任务 | 1 | 汇率同步（XXL-JOB） |

**关键接口设计原则**：
- C端接口不传 userId，从 Context 获取（TTHeader 透传）
- RPC供财务接口必传 userId + idempotencyKey（幂等键 required）
- 金额字段全部用 i64（最小单位整数），前端负责按币种格式化展示

### 3.4 技术栈

**全部复用现有工程基础设施，零新引入**：

```
框架层：  Kitex(RPC) + Hertz(HTTP网关) + Thrift(IDL)
存储层：  TiDB(主存储) + GORM(ORM) + GORM Gen(代码生成)
缓存层：  Redis(缓存+锁) + ETCD(配置+服务发现)
消息层：  Kafka(资金事件流) + NATS(实时通知)
安全层：  AES-256(字段加密) + EncryptedString(GORM自动加解密)
调度层：  XXL-JOB(分布式定时任务) + robfig/cron(本地任务)
工具层：  Snowflake(ID生成) + Sonic(JSON) + Zap(日志)
```

唯一需要小幅增强的：Redis 分布式锁封装（现有 SetNX 缺少安全解锁，新增约 50 行 Lua 脚本代码）。

## 四、核心链路怎么走

### 4.1 充值链路

```
用户选币种 → 选充值方式 → 选/输金额 → 选奖金活动（可选）→ 提交

                    钱包处理：
                    ├→ 重复转账拦截（同用户+同币种+同金额，≥3笔拦截）
                    ├→ 奖金门槛校验
                    ├→ 生成订单号 C+Snowflake
                    ├→ 创建充值订单（状态=待支付，此时不改余额）
                    └→ 调财务分配通道 → 返回支付链接/USDT收款地址

用户在三方页面/链上完成支付

                    财务接收回调 → 验签 → RPC调钱包 CreditDeposit

                    钱包上账（CreditDeposit）：
                    ├→ 幂等校验（同订单只入一次账）
                    ├→ 充值金额 → 中心钱包 +
                    ├→ 奖金金额 → 奖励钱包 +
                    ├→ 写交易流水（同一事务，带 before/after）
                    └→ 发 Kafka 事件 + NATS 实时通知

用户看到余额更新 ✓
```

**核心要点**：
- 创建订单时**不改余额**——钱还没到，不能先加
- 财务回调上账时**才改余额**——钱到了才加
- 余额变更和流水写入在**同一事务**——对账的安全保障
- 幂等键防重复入账——三方回调可能推多次

### 4.2 兑换链路

```
用户在法币币种下点"兑换BSB" → 输入金额

                    钱包预览（GetExchangePreview）：
                    ├→ 读平台汇率（内部函数调用，零网络开销）
                    ├→ 计算：兑换获得 + 赠送(10%) + 实际到账
                    └→ 生成汇率快照ID（绑定当前汇率）

用户确认兑换

                    钱包执行（ExecuteExchange）：
                    ├→ 校验汇率快照未过期（超5分钟要求重新预览）
                    ├→ 校验法币余额充足
                    ├→ 同一事务中：
                    │   ① 扣法币中心钱包余额
                    │   ② 加BSB中心钱包余额（兑换部分）
                    │   ③ 加BSB奖励钱包余额（赠送部分）
                    │   ④ 写交易流水
                    └→ 返回成功

用户看到法币减少、BSB增加 ✓
```

**核心要点**：
- **零跨模块依赖**——兑换是钱包内部闭环，不调任何外部 RPC
- **汇率快照机制**——用户看到的汇率和执行的汇率必须一致
- 赠送进奖励钱包——不进中心钱包，需完成流水要求才能提现

### 4.3 提现链路

```
用户点"提现"

                    钱包加载信息（GetWithdrawalInfo）：
                    ├→ 调 ser-kyc 查KYC状态
                    │   未认证 → 仅返回USDT提现方式
                    │   已认证 → 返回全部提现方式
                    ├→ 调财务查可用提现方式
                    └→ 返回：KYC状态 + 方式列表 + 限额 + 可提余额

用户选方式 → 选/填账户 → 输金额 → 提交

                    钱包创建提现（CreateWithdrawalOrder）：
                    ├→ 限额校验（单笔+单日）
                    ├→ 余额校验
                    ├→ 合规校验（法币充→法币提，USDT充→USDT提）
                    ├→ 生成订单号 T+Snowflake
                    ├→ 立即冻结余额（frozen +，available -）
                    ├→ 创建提现订单（状态=处理中）
                    └→ 通知财务开始审核流程

                    财务三级审核：
                    风控审核 → 财务审核 → 出款审核

                    审核拒绝/出款失败：
                    └→ 财务 RPC 调钱包 UnfreezeForWithdrawal → 解冻返还

                    出款成功：
                    └→ 财务 RPC 调钱包 ConfirmWithdrawalDeduct → 确认扣款
                       （frozen - 且 balance -，状态→已完成）

用户收到钱 ✓
```

**核心要点**：
- **提交即冻结**——保护资金，审核期间这笔钱不能被花掉
- **冻结/解冻/扣款是严格的状态机**——不可逆，防止资金异常
- 提现合规规则（钱从哪来往哪去）需要追踪资金来源

## 五、难点怎么解决

这个模块涉及资金操作，存在九大技术挑战，我逐一说明解决方案：

### 5.1 余额并发安全（难度最高）

**问题**：同一用户同时充值回调和提现冻结，余额不能算错。

**方案：三层防线**

```
第一层：数据库行级锁（SELECT FOR UPDATE）
  → 同一用户同一钱包，同一时间只有一个事务能操作
  → 锁和数据在同一事务，天然原子
  → 事务结束自动释放，无锁泄漏

第二层：乐观锁（version 字段）
  → UPDATE wallet SET balance=x, version=version+1 WHERE version=旧值
  → 更新0行说明并发冲突，重试

第三层：分布式锁（Redis SetNX + Lua 解锁）
  → 行锁之上的二级保护，快速失败返回"操作中请稍后"
  → 用户体验优化，避免长时间等待数据库锁

实际选择：首版行锁+乐观锁足够（用户级并发不高），分布式锁作为可选加强。
```

### 5.2 幂等性保障

**问题**：三方回调可能重复推送，ManualCredit 可能因超时重试。

**方案：三层幂等**

```
第一层：应用层检查 → 先查 idempotencyKey 是否已处理
第二层：数据库唯一键 → wallet_transaction 的 UK(order_no+wallet_type+tx_type)
第三层：业务状态机 → 已完成的订单不再重复处理
```

### 5.3 对账一致性

**问题**：余额和流水必须永远对得上。

**方案**：

```
写入时保证：余额更新 + 流水写入在同一数据库事务中，带 balance_before/balance_after
校验时保证：每日凌晨对账任务，校验 当前余额 = 初始余额 + SUM(所有流水变动)
异常处理：不一致时自动告警，人工介入排查
```

### 5.4 跨服务数据一致性

**问题**：钱包和财务各管各的数据库，网络调用可能失败。

**方案**：本地事务 + 幂等重试（不用 TCC/Saga）

```
以充值上账为例：
  财务侧：验签成功 → 更新财务订单状态 → RPC调钱包CreditDeposit
  如果RPC超时：财务侧重试（因为钱包有幂等键，重试安全）
  如果钱包返回失败：财务记录异常，人工介入

为什么不用TCC/Saga：
  ① 复杂度远大于收益
  ② 我们的场景不需要跨库事务回滚
  ③ 幂等重试 + 人工补偿已经足够
  ④ 业界大部分支付系统也是这个方案（微信支付、Stripe）
```

### 5.5 其他挑战（简要说明）

| 挑战 | 方案 |
|------|------|
| 金额精度 | bigint 最小单位存储，整数计算无损 |
| 提现状态机 | balance + frozen 双字段分离，状态严格不可逆 |
| 汇率一致性 | 快照机制 + TTL，用户看到的和执行的一定一致 |
| 敏感数据 | AES-256 + EncryptedString，GORM 自动加解密 |
| 性能扩展 | 合理索引首版够用，流水表增长后按需冷热分离 |

## 六、怎么一步步落地

### 6.1 分阶段实施

```
阶段一：基础骨架（约1周）
  ├→ 编写 Thrift IDL（6 个文件）
  ├→ 执行建表 SQL（10 张表）
  ├→ 运行 GORM Gen 生成 Model + Query
  ├→ 搭建 ser-wallet 服务骨架
  ├→ 注册 ETCD 服务 + Redis Key 常量
  └→ 封装 redis_lock.go（分布式锁增强）

阶段二：币种配置（约1周）
  ├→ B端 CRUD 6 个接口
  ├→ RPC 供他 4 个接口
  ├→ 汇率同步定时任务
  └→ Redis 缓存策略（币种+汇率全量缓存）

阶段三：钱包核心（约2周）
  ├→ BalanceEngine 余额变更引擎（核心中的核心）
  ├→ 余额查询 2 个接口
  ├→ 兑换 2 个接口（内部闭环，可先做）
  ├→ RPC 供财务 8 个接口
  └→ 交易记录 2 个接口

阶段四：充值+提现（约1.5周）
  ├→ 充值 5 个接口
  ├→ 提现 5 个接口
  └→ 提现账户管理

阶段五：联调+收尾（约1周）
  ├→ 与财务模块联调（10 个 RPC 接口）
  ├→ 与前端联调（C端 + B端）
  ├→ 对账任务 + 异常处理
  └→ 代码审查 + 修复
```

### 6.2 为什么这个顺序

```
阶段一（骨架）→ 所有后续工作的基础，不搭好无法开发
阶段二（币种）→ 币种是数据基座，钱包的所有操作都依赖币种数据
阶段三（钱包核心）→ BalanceEngine 是中枢，先把核心跑通
  ├→ 兑换放这里做是因为零外部依赖，可以独立验证 BalanceEngine
  └→ RPC供财务也在这里，财务开发可以提前对接
阶段四（充值+提现）→ 依赖财务模块的通道和回调，放后面给财务留开发时间
阶段五（联调）→ 双方各自开发完成后集中联调
```

## 附：听众可能会问的问题（提前准备）

### Q1：为什么不用 decimal 存金额？

bigint 最小单位存储是金融系统的业界标准。支付宝、微信支付、Stripe 都是这样做的。原因三点：① int64 计算无精度丢失（0.1+0.2≠0.3 的问题不存在）；② 性能优于 decimal（索引更高效）；③ 与 Go 的 int64 天然映射，不需要额外的 decimal 库。

### Q2：如果用户并发操作余额怎么办？

三层防线：数据库行级锁保证同一时间只有一个事务操作同一行；乐观锁 version 字段防并发更新丢失；Redis 分布式锁提供快速失败的用户体验。实际上，用户级并发不高（一个用户不会同时在两个设备充值+提现），行锁足够。

### Q3：和财务模块之间的调用，如果 RPC 超时怎么办？

幂等重试。所有供财务调用的 RPC 接口都有 idempotencyKey 参数（required），同一个 key 多次调用只生效一次。所以超时后重试是安全的。如果重试多次仍失败，进入人工补偿流程。

### Q4：流水表数据量会不会很大？

会。wallet_transaction 是只增不删的流水表，长期运营后会是数据量最大的表。应对策略：首版通过合理索引保证查询性能（user_id + currency_code + tx_type 的联合索引覆盖 90% 查询）；后续如果单表超千万，按月分区或冷热分离。TiDB 的分布式存储比单机 MySQL 更能承受大表。

### Q5：为什么币种配置和钱包放一个服务？

两个理由：① 钱包每个操作都要读汇率和币种信息，放同一个服务内部函数调用零网络开销，拆成两个服务每次多 1-3ms RPC 开销；② 就我一个人开发，拆两个服务运维复杂度翻倍但收益为零。代码层面按包隔离，未来要拆改 import 即可。工程内 ser-app 也是多域合一的先例。

### Q6：汇率如果三方API全部挂了怎么办？

本轮跳过不更新，继续使用上一次的平台汇率。同时发告警通知运维。汇率不会因为 API 故障变成 0 或异常值，因为更新是"偏差超阈值才触发"，API 挂了就不触发更新。

### Q7：提现审核期间用户能不能取消提现？

这取决于产品需求（评审反馈文档中未覆盖此问题，可以补充提问）。从技术上，如果允许取消，就是执行解冻操作（和审核拒绝一样的逻辑）。但需要产品确认：① 用户提交后能否主动取消？② 审核中状态能否取消？③ 出款中状态能否取消？建议只允许"处理中"状态可取消，进入审核流程后不允许取消。

### Q8：这套方案首版预计多久能完成？

我规划了 5 个阶段，总体上依赖和财务模块的联调配合。具体排期按项目管理流程确定，我负责的部分会按阶段里程碑推进。
