## 开场：我负责的是什么

各位好，我负责的是 ser-wallet 服务——**多币种钱包 + 币种配置**两个模块。

一句话定位：**用户充钱进来、换币、花钱、提钱，这一整套资金进出的后端逻辑是我的活。** 通道怎么配、订单怎么审、统计怎么出，是财务模块那边的活。但钱最终进出用户钱包，都要经过我这里。

在整个资金体系中，我这个服务扮演的是**账户层**的角色：

```
【我负责】账户层 — ser-wallet
  管用户有多少钱，每一分钱怎么来的怎么走的
  类比：银行的核心账务系统

【我负责】配置层 — ser-wallet 内的币种配置子模块
  管平台支持什么币种，各币种之间怎么换算
  类比：银行的外汇牌价管理

【他人负责】通道层 — 财务模块
  管钱从哪个通道进来、从哪个通道出去
  类比：银行的支付清算系统
```

关键边界：**ser-wallet 只管「账」不管「通道」**。我不直接和三方支付打交道，财务模块调我给用户加钱减钱，我保证加准减对。

## 第一部分：需求全貌——我要实现什么

### 1.1 业务全景

钱包服务同时服务三个端：

```
┌─────────────────────────────────────────────────────────┐
│  C端（App用户）                                          │
│  → 查余额、充值、兑换、提现、查记录                        │
│  → 16 个接口，全部需要登录                                │
├─────────────────────────────────────────────────────────┤
│  B端（管理后台）                                          │
│  → 币种配置、汇率管理、用户钱包查询                        │
│  → 7 个接口，需要管理员权限                                │
├─────────────────────────────────────────────────────────┤
│  RPC（其他服务调用）                                      │
│  → 上账、扣款、冻结、解冻、确认扣除、人工加减、查余额、返奖  │
│  → 10 个接口，供财务/投注/直播/活动模块调用                  │
└─────────────────────────────────────────────────────────┘
合计：33 个接口方法
```

### 1.2 钱包体系——用户的钱长什么样

每个用户按币种维度持有独立钱包，每个币种下分为多个钱包类型：

```
用户张三的资金体系：
├── BSB钱包（平台币）
│   ├── 中心钱包: BSB 800.00    ← 充值/投注返奖进这里，消费/提现从这里扣
│   └── 奖励钱包: BSB 200.00    ← 活动奖金进这里，完成流水要求后可转入中心
├── VND钱包（越南盾）
│   ├── 中心钱包: d 50,000,000
│   └── 奖励钱包: d 5,000,000
├── IDR钱包（印尼盾）
│   └── ...
└── THB钱包（泰铢）
    └── ...

资金钱包余额 = 中心 + 奖励（主播/代理/场馆预留不参与）
消费优先扣中心钱包，不够再扣奖励钱包
```

### 1.3 四大业务场景

| 场景 | 用户操作 | 钱包职责 | 外部依赖 |
|------|---------|---------|---------|
| **充值** | 选币种→选方式→输金额→支付 | 创建订单入口、重复检测、成功后上账到中心钱包 | 财务模块（充值方式/通道/回调） |
| **兑换** | 法币→BSB 单向兑换 | 法币余额减少、BSB余额增加、赠送进奖励钱包 | 无（内部闭环） |
| **提现** | KYC校验→选方式→填账户→输金额 | 冻结余额、创建提现申请、等审核结果 | ser-kyc（认证）+ 财务模块（审核/出款） |
| **记录** | 查看充值/提现/兑换/奖励记录 | 提供查询接口 | 部分数据可能来自财务模块 |

**加上 B 端的币种配置管理和 RPC 对外的余额操作能力，这就是我要做的全部东西。**

## 第三部分：架构设计——我打算怎么搭

### 3.1 整体分层

在现有工程 handler → ser → rep → gorm_gen 四层基础上，我在 Service 层内部增加一个**账户核心层**，把所有余额操作集中管理：

```
┌───────────────────────────────────────────────────────────────┐
│  接入层    gate-font (C端)     gate-back (B端)     RPC 直调   │
└─────────────────┬───────────────────┬────────────────┬────────┘
                  │                   │                │
┌─────────────────▼───────────────────▼────────────────▼────────┐
│  Handler 层    handler.go  — 薄封装，IDL 方法 → Service 方法   │
└────────────────────────────────┬──────────────────────────────┘
                                 │
┌────────────────────────────────▼──────────────────────────────┐
│  业务编排层    deposit_service / withdraw_service /             │
│  (Service)    exchange_service / record_service                │
│               组合调用账户核心层 + 对接外部模块                    │
└────────────────────────────────┬──────────────────────────────┘
                                 │
┌────────────────────────────────▼──────────────────────────────┐
│  账户核心层    wallet_core.go                                   │
│  (Core)       creditCore / debitCore / freezeCore /            │
│               unfreezeCore / confirmDebitCore                  │
│               幂等、事务、并发控制全在这一层                       │
│               C端调和 RPC 调最终走的是同一段代码                   │
├───────────────────────────────────────────────────────────────┤
│  配置核心层    currency_core.go                                  │
│  (Config)     币种 CRUD / 汇率计算 / 汇率刷新                    │
└────────────────────────────────┬──────────────────────────────┘
                                 │
┌────────────────────────────────▼──────────────────────────────┐
│  数据访问层    rep/  (Repository，基于 GORM Gen 扩展)            │
└────────────────────────────────┬──────────────────────────────┘
                                 │
┌────────────────────────────────▼──────────────────────────────┐
│  生成代码层    gorm_gen/  (model + query + repo，自动生成)       │
└───────────────────────────────────────────────────────────────┘
```

**为什么要拆出"账户核心层"？这是我这个设计中最关键的决策。**

原因：充值成功要上账调 creditCore、活动返奖也调 creditCore、人工加款还是调 creditCore——所有"给用户加钱"的操作最终走同一段代码。把幂等校验、并发控制、事务管理集中在一处，**不管请求从哪来，安全保障都一样**。这是业界标准做法，Stripe 和蚂蚁金服都是这个思路。

### 3.2 五个业务域

```
ser-wallet 内部分为 5 个域：

① 币种配置域（最底层，被所有域依赖，自身无依赖）
   职责：币种管理、汇率管理、汇率日志

② 账户域（核心层，依赖①，被③和 RPC 调用方依赖）
   职责：钱包账户、余额操作（6 个原子操作）、冻结管理、流水记录

③ 交易域（最上层，依赖①②和外部模块）
   职责：充值业务、提现业务、兑换业务

④ 记录查询域（只读，依赖各域数据）
   职责：交易记录查询

⑤ 定时任务域（依赖①）
   职责：汇率定时刷新

域间规则：严禁循环依赖。②不能依赖③，①不能依赖②。
```

### 3.3 数据库表设计

我设计了 **14 张表**，按优先级分为三级：

```
【必须】8 张 — 没有这些表，模块核心功能无法运行
  ① currency_config        币种配置表
  ② user_wallet            用户钱包账户表（核心中的核心）
  ③ wallet_flow            资金流水表（对账/审计/幂等凭证）
  ④ wallet_freeze_record   冻结记录表（提现冻结状态机载体）
  ⑤ deposit_order          充值订单表
  ⑥ withdraw_order         提现订单表
  ⑦ withdraw_account       提现账户表（银行卡/电子钱包信息）
  ⑧ exchange_record        兑换记录表

【应该】3 张 — 功能完整性需要
  ⑨ exchange_rate_log      汇率变更日志表
  ⑩ manual_adjustment_order 人工修正订单表
  ⑪ wallet_config          钱包业务配置表

【可选】3 张 — 可观测性和运维支撑
  ⑫ deposit_bonus_snapshot 充值奖金快照
  ⑬ wallet_daily_snapshot  每日钱包快照
  ⑭ rate_provider_log      三方汇率源日志
```

**核心表的关键设计决策：**

**① 账户模型**：一个用户 × 一个币种 × 一个钱包类型 = 一条记录

```
user_wallet 表唯一索引：(user_id, currency_code, wallet_type)

用户 10001 在系统中可能有这些记录：
  (10001, VND, center)   → 可用余额 5000万, 冻结 0
  (10001, VND, reward)   → 可用余额 500万,  冻结 0
  (10001, BSB, center)   → 可用余额 1000,   冻结 200
  (10001, BSB, reward)   → 可用余额 100,    冻结 0

好处：
  ✓ 币种隔离（需求要求）
  ✓ 钱包类型独立（中心/奖励各管各的）
  ✓ 按三个维度精准定位一行，并发操作不同币种/类型互不阻塞
```

**② 金额存储**：全部用 BIGINT 整数，存最小货币单位

```
不用浮点、不用 DECIMAL，用整数。

原因：0.1 + 0.2 ≠ 0.3（浮点精度问题），资金系统绝不允许。
做法：
  VND/IDR（整数币种）→ 1 VND = 1
  THB/BSB（2位小数） → 1.50 THB 存 150
  USDT（8位小数）    → 1 USDT 存 100000000

Stripe、PayPal、蚂蚁金服都这么做。现有工程 ser-item 的价格字段也是整数。
```

**③ 流水表的幂等保障**：wallet_flow 表的 (order_no, flow_type) 做唯一索引

```
每次余额变更写一条流水。
order_no 唯一索引保证：
  同一笔订单不会被重复处理（回调重发、RPC 重试都安全）。
```

## 第四部分：核心技术方案——关键挑战怎么解决

### 4.1 并发余额操作（P0 级挑战）

**问题**：两个请求同时扣同一个用户的余额，可能超扣。

**我的方案**：条件更新（`UPDATE ... WHERE balance >= amount`）

```go
// 扣款时：
UPDATE user_wallet
SET available_balance = available_balance - ?
WHERE user_id = ? AND currency_code = ? AND wallet_type = ?
  AND available_balance >= ?    // 关键：数据库层面保证不超扣

// 如果 affected_rows == 0 → 余额不足，拒绝
// 如果 affected_rows == 1 → 扣款成功

// 上账时（加钱不会失败，但要处理钱包不存在的情况）：
UPDATE user_wallet
SET available_balance = available_balance + ?
WHERE user_id = ? AND currency_code = ? AND wallet_type = ?
// 如果 affected_rows == 0 → 钱包不存在，自动创建后重试
```

**为什么选这个方案而不是悲观锁 SELECT FOR UPDATE**：
1. 现有工程 ser-item 已经在用这个模式（`gorm.Expr + Add/Sub`）
2. 不需要额外的 version 字段
3. 单条 UPDATE 语句在数据库层面天然原子
4. 性能优于 SELECT FOR UPDATE（减少锁持有时间）
5. 当前并发量级足够应对

### 4.2 幂等性（P0 级挑战）

**问题**：财务回调可能重发、RPC 可能超时重试，不做幂等就会重复入账。

**我的方案**：用 order_no 做唯一键，在流水表中控制

```
每个余额变更操作的第一步：
  1. 拿到 order_no
  2. 在事务内尝试 INSERT wallet_flow (order_no, flow_type, ...)
     → 插入成功 = 首次请求，继续执行余额变更
     → 唯一键冲突 = 重复请求，查出上次结果直接返回
  3. 余额变更和流水写入在同一事务内提交

不需要额外的幂等表，流水记录本身就是幂等证据。
```

### 4.3 跨服务一致性（P0 级挑战）

**问题**：充值成功后要上账（钱包）+ 更新订单（财务），两个服务的数据怎么一致？

**我的方案**：不用分布式事务，用本地事务 + 幂等 + 补偿

```
核心思路：
  每个服务只管好自己的事务。
  调用方通过 order_no 保证可安全重试。
  最终一致性而非强一致性。

举例——充值上账：
  财务回调 → ser-wallet.CreditWallet(orderNo=xxx, amount=1000)
    → ser-wallet 事务内：余额 +1000、写流水
    → 成功 → 返回 OK
    → 失败 → 返回 Error → 财务重试
    → 重试时 orderNo 命中幂等 → 直接返回上次结果

不引入 Seata 等分布式事务框架。原因：
  1. 现有工程没有用
  2. 引入成本高、运维复杂
  3. 本地事务 + 幂等重试已经能解决问题
```

### 4.4 金额精度（P0 级挑战）

**问题**：涉及汇率换算时，浮点计算会丢精度。

**我的方案**：全链路整数

```
存储层  →  BIGINT（最小单位整数）
传输层  →  i64（Thrift IDL）
计算层  →  Go int64 运算
展示层  →  按币种的 decimal_digits 配置，除回去格式化

汇率也用整数：× 10^8 存储，保留 8 位精度。
计算时先乘后除，避免精度丢失。
```

### 4.5 缓存策略

| 缓存对象 | Key | 有效期 | 失效方式 |
|---------|-----|--------|---------|
| 用户余额 | `wallet:bal:{uid}:{curr}` | 30-60s | 余额变更时主动 DEL |
| 币种列表 | `wallet:currencies` | 5-10min | B端编辑后主动 DEL |
| 汇率数据 | `wallet:rate:{curr}` | 由定时任务覆盖 | 定时任务写入新值 |

缓存失效用 DEL 而非更新值——下次请求自动回源数据库，避免缓存和数据库不一致的窗口。

## 第五部分：六个原子操作——钱包的能力根基

这是 ser-wallet 对外输出能力的核心。不管充值、提现、投注、返奖、人工修正，最终都要调这些原子操作。

```
┌───────────────────────────────────────────────────────────┐
│                    六个原子操作                              │
├──────────────┬──────────────┬─────────────────────────────┤
│ 操作          │ 作用          │ 余额变化                     │
├──────────────┼──────────────┼─────────────────────────────┤
│ creditCore   │ 上账（加钱）   │ 指定钱包.可用 += amount      │
│ debitCore    │ 扣款（减钱）   │ 中心 -= X, 奖励 -= Y        │
│ freezeCore   │ 冻结          │ 可用 -= amount, 冻结 += amount│
│ unfreezeCore │ 解冻退回       │ 冻结 -= amount, 可用 += amount│
│ confirmDebit │ 确认扣除       │ 冻结 -= amount（永久减少）     │
│ writeFlow    │ 写流水         │ 无余额变化                   │
└──────────────┴──────────────┴─────────────────────────────┘

每个操作遵循相同模式：
  ① 幂等校验（order_no 查重）
  ② 前置校验（余额充足？状态正确？）
  ③ 事务执行（余额变更 + 流水写入在同一事务）
  ④ 缓存失效（DEL 该用户该币种的余额缓存）
  ⑤ 返回结果
```

**debitCore 的特殊逻辑**：优先扣中心钱包，不够再扣奖励钱包

```
用户中心 800，奖励 200，要扣 1000：
  centerDebit = min(800, 1000) = 800  → 中心钱包 -= 800
  rewardDebit = 1000 - 800 = 200     → 奖励钱包 -= 200

返回值带上各钱包实际扣了多少，投注模块后续算返奖比例要用。
```

## 第六部分：主要业务流程——具体怎么跑

### 6.1 充值流程

```
用户操作                 ser-wallet                           财务模块

  │ 进入充值页              │                                    │
  ├───────────────────────→│ GetDepositMethods                  │
  │                        ├──────────────────────────────────→│ 获取充值方式列表
  │                        │←─────────────────────────────────│ 返回方式+档位
  │←───────────────────────│ 返回充值页数据                      │
  │                        │                                    │
  │ 点击"去充值"            │                                    │
  ├───────────────────────→│ CreateDepositOrder                 │
  │                        │ 1.参数校验                          │
  │                        │ 2.重复转账检测（查自己的订单表）       │
  │                        │ 3.BSB充值→锁定汇率快照               │
  │                        │ 4.创建本地充值订单                   │
  │                        ├──────────────────────────────────→│ 创建通道订单
  │                        │←─────────────────────────────────│ 返回支付信息
  │←───────────────────────│ 返回支付跳转信息                    │
  │                        │                                    │
  │  ......用户去支付......  │                                    │
  │                        │                                    │
  │                        │         三方支付回调→财务更新状态      │
  │                        │←─────────────────────────────────│ CreditWallet(orderNo, amount)
  │                        │ creditCore:                        │
  │                        │   幂等→事务→余额+流水→失效缓存       │
  │                        ├──────────────────────────────────→│ 返回上账结果
  │                        │                                    │
```

### 6.2 兑换流程（完全内部闭环，零外部依赖）

```
用户操作                 ser-wallet
  │                        │
  │ 查看汇率               │
  ├───────────────────────→│ GetExchangeRate
  │                        │  读币种配置缓存→返回 1BSB = X 法币
  │←───────────────────────│
  │                        │
  │ 输入金额，点击兑换      │
  ├───────────────────────→│ DoExchange
  │                        │  1. 校验法币余额 >= 输入金额
  │                        │  2. 计算：获得BSB = 输入 / 汇率
  │                        │  3. 计算：赠送BSB = 获得 × 赠送比例
  │                        │  4. 事务内：
  │                        │     法币中心 -= 输入金额
  │                        │     BSB中心 += 获得BSB
  │                        │     BSB奖励 += 赠送BSB（需记录流水倍数要求）
  │                        │     写 3 条流水
  │                        │  5. 失效双币种缓存
  │←───────────────────────│ 返回兑换结果
```

  │                        │←─────────────────────────────────────│

## 第七部分：IDL 设计——接口契约

我设计了 **8 个 IDL 文件**，完全对齐现有 20 个服务的 Thrift IDL 规范：

```
common/idl/ser-wallet/
├── service.thrift          ← 入口：WalletService，33 个方法声明
├── wallet.thrift           ← 钱包核心类型
├── currency.thrift         ← 币种配置类型
├── deposit.thrift          ← 充值类型
├── withdraw.thrift         ← 提现类型
├── exchange.thrift         ← 兑换类型
├── record.thrift           ← 记录查询类型
└── wallet_rpc.thrift       ← RPC 对外类型（供其他服务调用）
```

**service.thrift 方法按调用端分组**：C端（16个）→ B端（7个）→ RPC（10个），用注释分隔。

**wallet_rpc.thrift 是最先需要交付的**——财务模块需要看到 CreditWallet/DebitWallet 等接口的类型定义才能编写集成代码。越早给出，跨团队阻塞越少。

## 第八部分：状态机设计

### 8.1 冻结状态机（提现核心）

```
   freeze                    confirmDebit（出款成功）
   ───────→  冻结中(Frozen)  ─────────────────────→  已确认(Confirmed)
                    │                                  冻结永久扣除
                    │
                    │ unfreeze（驳回/失败）
                    └─────────────────────────────→  已解冻(Unfrozen)
                                                      余额退回可用

约束：Confirm 和 Unfreeze 互斥，终态不可逆
实现：数据库 WHERE status = 'Frozen' 条件更新，affected_rows=0 说明状态不对
```

### 8.2 充值订单状态机

```
创建 → 待支付(Pending) → 支付成功(Success) / 超时失败(Failed)
```

### 8.3 提现订单状态机

```
创建 → 待风控审核 → 待财务审核 → 待出款 → 出款成功
                ↘ 风控驳回   ↘ 财务驳回   ↘ 出款失败
```

所有状态变更通过 `WHERE status = 期望值` 条件更新实现，不引入额外的状态机框架。

### 10.2 为什么是这个顺序

```
阶段一（基建）    → 没有这个，后面都写不了代码
阶段二（核心能力） → 钱包的根基，RPC 对外能力就绪 → 解除财务模块的集成阻塞
阶段三（兑换优先） → 零外部依赖、覆盖全链路，是最佳的冒烟验证场景
阶段四（充值）    → 依赖财务但可 Mock 先行
阶段五（提现）    → 最复杂的流程放在能力已验证之后
阶段六（收尾联调） → Mock 替换真实调用 + 查询类接口 + 对账
```

**核心理念：先把钱包自己的能力做扎实（阶段二），再一层层往上搭业务流程。外部依赖用 Mock 隔离，联调时只需要把 Mock 替换成真实调用。**

## 第十一部分：与财务模块的协作计划

### 11.1 依赖关系

```
我调财务：
  GetPaymentConfig   → 获取充值/提现方式列表
  CreatePayOrder     → 创建充值通道订单
  CreateWithdrawOrder → 创建提现审核订单

财务调我：
  CreditWallet      → 充值成功上账
  ConfirmDebit      → 出款成功确认扣除
  UnfreezeBalance   → 驳回/失败解冻退回
  ManualCredit      → 人工加款
  ManualDebit       → 人工减款
  SupplementCredit  → 充值补单
  GetBalance        → 查询用户余额
```

### 11.2 协作策略

```
第一步：我先出 wallet_rpc.thrift（RPC 对外的类型定义）→ 给财务看
第二步：财务出他们的 IDL → 我看
第三步：双方确认调用时序和字段含义
第四步：各自 Mock 对方先开发
第五步：联调替换 Mock
```

**关键时间节点**：我的 RPC 接口定义（阶段二就绑定了类型），需要在财务模块开始编码前交付。越早越好。

### 11.3 Mock 策略

```go

    // 返回硬编码的充值方式列表
    return &PaymentConfig{Methods: [...]}, nil

```

这样做的好处：充值/提现流程可以在财务模块还没开发时就独立跑通和自测。

## 第十二部分：风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| 财务模块 IDL 延迟 | 充值/提现联调推后 | Mock 先行，不等不靠 |
| 需求评审后变更 | 部分设计需要调整 | IDL 字段用 optional，表结构可追加 |
| 汇率 API 选型未定 | 定时任务无法实现 | 抽象汇率获取接口，先用免费 API 开发 |
| 提现规则细节不清 | "充什么提什么"判定有歧义 | 评审会上确认，先按保守规则实现 |
| 并发压力超预期 | 条件更新可能性能不足 | 预留分布式锁扩展点，但不提前引入 |

## 总结

```
一句话：
  ser-wallet 是一个从零开始但工程基础完备的新服务。
  8 个 IDL 文件、33 个接口方法、14 张数据库表、6 个原子操作。
  全部复用现有技术栈，零新增组件。
  分 6 个阶段推进，兑换流程最先跑通验证全链路，
  外部依赖用 Mock 隔离，保证可以独立开发不被阻塞。

三个关键词：
  安全 — 条件更新防超扣 + 幂等防重复 + 事务保一致
  复用 — 26 个组件全复用、6 个原子操作全复用
  渐进 — 先核心再业务、先独立再联调、先 Mock 再真实
```

以上就是我对 ser-wallet 多币种钱包模块的完整实现方案。
