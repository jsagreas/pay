### 1.1 一句话定位

### 1.4 钱包体系的核心规则（18条金律）

这些规则贯穿整个模块，任何接口的设计和实现都不能违背：

1. **币种隔离**：每个币种独立余额、独立账本，不混合
2. **消费优先级**：先扣中心钱包，不足部分扣奖励钱包
3. **奖励稽核**：奖励钱包余额需完成 X 倍投注流水才能转入中心钱包提现
4. **充值来源决定提现方式**：法币充→法币提，USDT充→USDT提，混合充→法币提
5. **KYC与提现**：法币提现必须 KYC 已通过；USDT提现可不 KYC 但有限额
6. **持有人锁定**：提现账户持有人姓名必须等于 KYC 认证姓名，不可编辑
7. **重复转账拦截**：同用户+同币种+同金额，1-2笔警告，≥3笔拒绝
8. **兑换单向**：只支持法币→BSB，不可反向，赠送10%进奖励钱包
9. **基准币种不可变**：设置后永不可更改（推荐USDT）
10. **三种汇率**：充值用入款汇率，提现用出款汇率，统计用平台汇率
11. **金额精度**：VND/IDR整数0位，THB/BSB两位，USDT八位
12. **幂等保障**：所有资金操作通过 orderNo 幂等，防重复执行
13. **流水完整**：每笔余额变动必须有对应的 wallet_transaction 记录
14. **提现三级审核**：风控→财务→出款，任一级驳回则解冻
15. **人工加减款**：审核人不能等于提交人
16. **汇率三家均值**：三方API取平均值，偏差超阈值自动更新平台汇率
17. **余额不缓存**：余额数据直接查TiDB（<1ms），不放Redis，防一致性问题
18. **返奖按比例**：投注扣款记录各钱包比例，返奖按原比例分回

### 2.2 架构决策记录（ADR）

在设计过程中做了 8 个关键决策，每个都有明确的依据和对标：

| ADR | 决策 | 否决的方案 | 核心依据 |
|-----|------|----------|---------|
| ADR-01 | TiDB 存储 | PostgreSQL/CockroachDB | 项目统一标准，自动分片，兼容MySQL |
| ADR-02 | 余额不缓存到Redis | Cache-Aside模式 | 金融场景缓存不一致=超扣风险，TiDB PK查询<1ms够快 |
| ADR-03 | DECIMAL(20,8) + shopspring/decimal | float64 / int64最小单位 | float精度灾难；int跨币种换算易出错；decimal是Stripe/PayPal标准做法 |
| ADR-04 | 币种配置与钱包同进程 | 拆为独立微服务 | 每笔交易都查汇率，进程内调用~0.001ms vs RPC~3-10ms |
| ADR-05 | 悲观锁（FOR UPDATE）+ version字段 | 纯乐观锁CAS | 资金场景零容忍，高争用时乐观锁频繁失败 |
| ADR-06 | 同步RPC + 幂等重试 | Saga/TCC分布式事务 | Saga/TCC增加3倍复杂度，收益有限；Stripe/支付宝验证了同步+幂等足够 |
| ADR-07 | NATS JetStream（Phase 5+） | Kafka | NATS更轻量、已部署；Kafka留作后备 |
| ADR-08 | 单service定义（WalletService） | 拆CurrencyService + WalletService | 全工程无多service先例，gen_kitex.sh只识别单入口 |

## 三、表结构设计：11张表支撑全部业务

### 3.1 总览

```
┌─ 币种配置域（4张）─────────────────────────────┐
│  currency          币种主表（编码/名称/类型/图标/精度/浮动%/阈值%）  │
│  currency_rate     汇率表（实时/平台两种，per币种）                 │
│  currency_rate_log 汇率变更日志（偏差超阈值时自动写入）              │
│  base_currency     基准币种配置（全局唯一，设置后不可改）            │
└────────────────────────────────────────────────┘

┌─ 钱包核心域（2张）─────────────────────────────┐
│  wallet             用户钱包（user_id × currency × wallet_type）  │
│  wallet_transaction 资金流水（只增不改，对账核心）                   │
└────────────────────────────────────────────────┘

┌─ 订单域（3张）─────────────────────────────────┐
│  deposit_order     充值订单（待支付→支付中→成功/失败/失效）        │
│  withdraw_order    提现订单（审核中→待出款→出款中→完成/驳回）     │
│  exchange_order    兑换订单（法币→BSB，含赠送和稽核倍数）         │
└────────────────────────────────────────────────┘

┌─ 辅助域（2张）─────────────────────────────────┐
│  withdraw_account  提现账户（银行卡/电子钱包/USDT地址，三合一）    │
│  reward_record     奖励记录（含稽核进度追踪）                     │
└────────────────────────────────────────────────┘
```

### 3.2 核心表详解

#### wallet — 钱包余额主表（最重要的表）

```sql
CREATE TABLE wallet (
  id              BIGINT PRIMARY KEY AUTO_INCREMENT,
  user_id         BIGINT       NOT NULL,                -- 用户ID
  currency_code   VARCHAR(10)  NOT NULL,                -- 币种编码
  wallet_type     TINYINT      NOT NULL,                -- 1中心 2奖励 3主播 4代理 5场馆
  balance         DECIMAL(20,8) NOT NULL DEFAULT 0,     -- 可用余额
  frozen          DECIMAL(20,8) NOT NULL DEFAULT 0,     -- 冻结金额
  version         INT          NOT NULL DEFAULT 0,      -- 乐观锁版本号
  -- 审计字段省略 --

  UNIQUE KEY uk_user_currency_type (user_id, currency_code, wallet_type, delete_at),
  INDEX idx_user_currency (user_id, currency_code)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_bin;
```

**设计要点**：
- 三维唯一约束 `(user_id, currency_code, wallet_type)` 确保一个用户在一个币种下只有一个特定类型的钱包
- `frozen` 字段：提现申请时冻结，审核通过正式扣除，驳回则解冻恢复
- `version` 字段：悲观锁主力 + 乐观锁兜底的双重保护
- **冻结模型**：`可提现余额 = balance - frozen`，用户看到的总余额不变

#### wallet_transaction — 资金流水表（对账核心）

```sql
CREATE TABLE wallet_transaction (
  id              BIGINT PRIMARY KEY AUTO_INCREMENT,
  user_id         BIGINT       NOT NULL,
  currency_code   VARCHAR(10)  NOT NULL,
  wallet_type     TINYINT      NOT NULL,
  direction       TINYINT      NOT NULL,                -- 1收入 2支出
  amount          DECIMAL(20,8) NOT NULL,               -- 变动金额（始终正数）
  balance_before  DECIMAL(20,8) NOT NULL,               -- 变动前余额
  balance_after   DECIMAL(20,8) NOT NULL,               -- 变动后余额
  frozen_before   DECIMAL(20,8) NOT NULL DEFAULT 0,     -- 变动前冻结
  frozen_after    DECIMAL(20,8) NOT NULL DEFAULT 0,     -- 变动后冻结
  biz_type        VARCHAR(32)  NOT NULL,                -- 业务类型
  order_no        VARCHAR(64)  NOT NULL,                -- 业务订单号（幂等键）
  remark          VARCHAR(256) NOT NULL DEFAULT '',
  create_at       BIGINT       NOT NULL,

  UNIQUE KEY uk_biz_order_wallet (biz_type, order_no, wallet_type),
  INDEX idx_user_currency_time (user_id, currency_code, create_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_bin;
```

**设计要点**：
- **只 INSERT 不 UPDATE/DELETE**——不可变账本，任何修正通过反向流水
- `(biz_type, order_no, wallet_type)` 唯一索引实现 DB 层幂等兜底
- `balance_before/after` 链式校验：`T[n].balance_before == T[n-1].balance_after`
- 包含 `wallet_type` 是因为一笔投注可能同时影响中心和奖励两个钱包

### 3.3 金额存储统一规范

```
所有金额字段：DECIMAL(20,8)
  → 20位总宽度，8位小数
  → 支持最大值：999,999,999,999.99999999（近万亿级）
  → 最小精度：0.00000001（满足USDT的8位精度）

IDL 传输：string 类型（"1000.50000000"）
  → Thrift 无 decimal 类型，string 无精度损失

Go 运算：shopspring/decimal 库
  → decimal.NewFromString("1000.50") → 精确计算 → .String() 输出
```

## 五、技术栈：选了什么、为什么选、拒绝了什么

### 5.1 核心技术栈

| 层次 | 技术 | 为什么选 | 拒绝了什么 |
|------|------|---------|-----------|
| HTTP网关 | CloudWeGo Hertz | 字节优化，与Kitex同生态 | Gin（非CloudWeGo） |
| RPC框架 | CloudWeGo Kitex + Thrift | 高性能，原生TTHeader传播 | gRPC（更重） |
| 数据库 | TiDB | 项目标准，MySQL兼容，自动分片 | PostgreSQL（无自动分片） |
| ORM | GORM + Gen | 类型安全查询，代码生成 | sqlx（手写SQL易出错） |
| 缓存 | Redis（go-redis v9） | 配置/汇率缓存 + 防重锁 | 本地缓存（多实例不一致） |
| 注册中心 | ETCD v3 | 配置中心+服务发现 | Consul/ZK（非项目标准） |
| 精度计算 | shopspring/decimal | 任意精度decimal运算，5.7k star | float64（0.1+0.2≠0.3） |
| 消息队列 | NATS JetStream（Phase 5+） | 轻量持久化，已部署 | Kafka（目前用不到） |
| 定时任务 | robfig/cron v3 | 进程内定时，已有实践 | XXL-Job（过重） |

### 5.2 唯一需要新增的依赖

**shopspring/decimal** — 这是本模块唯一的新依赖：
- 用途：所有金额的加减乘除运算
- 为什么必须：Go 没有原生 decimal 类型，float64 在金融场景是灾难
- 风险评估：5.7k GitHub star，活跃维护，Stripe/PayPal 使用同类方案
- 影响范围：加入 `common/pkg/go.mod`

## 六、20个技术挑战与解决方案

### P0 级 — 资金安全（数据丢失风险）

**挑战1：并发余额操作一致性**
- 问题：两个请求同时扣同一用户余额，可能超扣
- 方案：`SELECT ... FOR UPDATE` 悲观锁（TiDB默认悲观模式）+ version 字段双保险
- 锁持有时间：~10-20ms，用户级频率完全可接受

**挑战2：金额精度与浮点陷阱**
- 问题：float64 下 `0.1 + 0.2 ≠ 0.3`，累积误差导致对账失败
- 方案：DB 统一 `DECIMAL(20,8)` + Go 层 `shopspring/decimal` + IDL 用 string 传输
- GORM 映射：`gen.FieldType("balance", "decimal.Decimal")` 避免默认 float64

**挑战3：分布式环境幂等性**
- 问题：网络超时/重试导致同一操作执行多次
- 方案：双层幂等——L1 流水表快速查重 + L2 DB UNIQUE(biz_type, order_no, wallet_type) 兜底
- 效果：即使代码bug也不会重复记账

**挑战4：防止超扣（余额变负）**
- 问题：并发操作 + 代码缺陷可能导致余额 < 0
- 方案：五层防线
  ```
  L1: 参数校验（amount > 0，currency 有效）
  L2: 流水表查重（幂等快速拒绝）
  L3: FOR UPDATE + 余额 >= amount 校验（事务内）
  L4: version 自增验证（乐观锁双保险）
  L5: DB UNIQUE 约束（最终兜底）
  ```

**挑战5：冻结-确认/解冻原子性**
- 问题：提现冻结后，审核驳回解冻或审核通过扣款必须原子操作
- 方案：frozen 金额记录在 wallet 行 + withdraw_order 行，同一事务操作
- 状态机：冻结 → (驳回)解冻 / (通过)从冻结中扣除

### P1 级 — 业务可用性

**挑战6：跨服务数据一致性**
- 问题：钱包RPC成功但财务记录失败
- 方案：同步RPC + 幂等接口 + 财务侧负责重试/补单
- 不用 Saga/TCC：复杂度增加3倍，收益有限（Stripe/支付宝验证了此方案）

**挑战7：死锁预防**
- 问题：兑换操作涉及两个币种钱包，可能循环等待
- 方案：固定加锁顺序——始终按 `(currency_code ASC, wallet_type ASC)` 加锁
- 兜底：捕获 MySQL 错误码 1213 重试一次

**挑战8：多钱包优先级扣款**
- 问题：投注1500 BSB，中心1000+奖励800，怎么扣？
- 方案：原子事务内先扣中心(1000)再扣奖励(500)，记录比例 `{center:0.667, bonus:0.333}`
- 返奖：按记录的比例分回

**挑战9：支付回调可靠性**
- 问题：回调丢失/延迟/重复/伪造
- 方案：① 验签防伪造 ② 金额校验 ③ 幂等处理 ④ 主动轮询补偿
- 职责划分：回调处理归财务模块，钱包只需保证 WalletCredit 幂等

**挑战10：汇率实时性与准确性**
- 问题：汇率波动快，3家API可能返回异常值
- 方案：每3分钟轮询 → 3家取中位数 → 异常检测 → 偏差超阈值自动更新平台汇率
- 降级：API全挂时使用上次成功率 + 告警

### P2 级 — 性能与运维

**挑战11-15**：流水表增长（TiDB自动分片，免应用层分表）、订单号全局唯一（前缀+时间戳+随机数+UNIQUE兜底）、操作防重（Redis SetNX 10s TTL）、服务降级（财务不可用→禁止充提但保留查询）、对账审计（流水链式校验 + 定期批量比对）

### P3 级 — 体验与规范

**挑战16-20**：多币种展示格式化（服务端按currency配置格式化返回原始值+显示值）、GORM与decimal集成（gen.FieldType自定义映射）、缓存与DB一致性（币种配置Cache-Aside+写后删）、错误码体系（6022000-6022999段）、日志可追溯（trace_id + 关键业务参数结构化日志）

## 十、我准备好了被问的20个刁钻问题

### Q1：为什么余额不放 Redis 缓存？每次查 DB 不慢吗？

**答**：TiDB 主键查询延迟 < 1ms，完全不慢。而如果缓存余额，多实例并发场景下缓存和 DB 不一致会导致**超扣**——用户余额100，缓存说100，两个请求各扣80都通过了。在金融场景，数据一致性绝对高于读性能。Stripe、支付宝余额也不走缓存。

### Q2：为什么不用分布式事务（Saga/TCC）？

**答**：钱包和财务的交互模式是"同步 RPC + 幂等重试"，足够简单且可靠。Saga/TCC 会增加 3 倍代码复杂度（补偿逻辑、协调器、状态机），而收益有限——我们的场景是两个服务协作，不是多服务编排。Stripe、PayPal、支付宝的核心支付链路也不用 TCC。

### Q3：一个用户大量并发请求会不会锁表？

**答**：不会。`SELECT FOR UPDATE` 是**行级锁**，锁的是 `(user_id, currency_code, wallet_type)` 对应的那一行。不同用户之间完全不影响。同一用户的并发请求会**串行化**，这正是我们期望的——防止同一用户并发操作导致数据不一致。锁持有时间 ~10-20ms，单用户不可能高频到出问题。

### Q4：流水表数据量会不会爆炸？

**答**：即使 10M 用户每天 10 笔交易 = 1亿条/月 = 12亿条/年。TiDB 的分布式架构天然支持 TB 级数据量，不需要应用层分表。查询走索引 `(user_id, currency_code, create_at)` 游标分页，不用 OFFSET。如果未来超过 5 亿条，可按月做分区。

### Q5：shopspring/decimal 库万一有 bug 怎么办？

**答**：这个库 5.7k star、7年历史、持续维护。但更重要的是，我们有**多层验证**：① 流水表的 `balance_before + amount = balance_after` 链式校验 ② 定期对账：`wallet.balance == 最新一条流水.balance_after` ③ 单测覆盖关键金额计算。即使库有bug，对账会立即发现。

### Q6：兑换涉及两个币种两个钱包，怎么保证原子性？

**答**：同一个 DB 事务内操作。先按固定顺序加锁（`currency_code ASC` 先锁法币再锁 BSB），然后法币中心钱包扣 → BSB中心钱包入 → BSB奖励钱包入（赠送部分）→ 三条流水 → 一条兑换订单，全部在同一个 `BEGIN...COMMIT` 中。失败自动回滚，不存在部分成功。

### Q7：提现冻结后系统崩溃了怎么办？

**答**：冻结和创建财务订单在同一个事务中完成。如果在调财务 RPC 前崩溃，事务回滚，冻结不生效。如果 RPC 调出去但没收到响应——我们有 `orderNo` 幂等，重启后重试同一个 orderNo，财务侧返回已存在，钱包侧补记状态。最坏情况：冻结了但财务单没建成——定时任务扫描"超时未关联财务订单的冻结记录"，自动解冻。

### Q8：为什么不把币种配置拆成独立服务？

**答**：每笔交易都需要查汇率和币种信息。如果是独立服务，一笔充值就多一次 RPC 调用（3-10ms）；同进程内调用只需 ~0.001ms。另外，工程上所有现有服务（ser-live/ser-user/ser-app）都是"子模块在同一进程"的模式，拆分反而破坏一致性。

### Q9：如果三家汇率 API 全挂了怎么办？

**答**：降级策略是使用**上次成功获取的汇率**继续服务，同时触发告警。因为汇率有"偏差阈值"机制（比如5%），即使用几分钟前的汇率，偏差也在可接受范围内。绝不能因为汇率 API 挂了就停止所有充值提现——这是"火"（不能没有），汇率略有延迟是"调料"（可以暂时没有）。

### Q10：提现来源规则"混合充→法币提"合理吗？用户 USDT 部分永远提不回来？

**答**：这确实是一个需要产品确认的问题（P0-04）。从合规角度这是为了防止洗钱（USDT→法币→USDT 可能被利用）。但对用户体验有影响。我的建议是在产品上明确告知用户"混合充值后统一走法币提现"，或提供"按充值来源拆分提现"的选项——后者实现复杂度更高但更公平。

### Q11：55个接口怎么排期？会不会做不完？

**答**：不是55个同时做。6阶段渐进交付，Phase 0-3 无外部依赖可独立完成（骨架+币种配置+账户核心+兑换 = 24个接口），能在2-3周内交付可验证版本。Phase 4-5 依赖财务联调，但可通过 Mock 策略提前开发。55个里有14个是"预留"（TODO实现+接口定义），首期实际实现约41个。

### Q12：USDT 充值和法币充值有什么本质区别？

**答**：法币走传统支付通道（银行/电子钱包），有三方网关回调；USDT走区块链，需要链上监听确认。两者在钱包侧的终点相同——都是调 `WalletCredit` 入账。区别在 ser-finance 侧：法币通道有回调 URL，USDT 有链上确认数（比如 TRC20 需要 19 个确认）。

### Q13：同一个用户能同时有 VND 和 THB 的钱包吗？

**答**：可以。wallet 表的唯一键是 `(user_id, currency_code, wallet_type)`。一个用户理论上可以有 5币种 × 5钱包类型 = 25个钱包行。首期用户注册时按国家默认分配一个法币+BSB，后续可手动开通其他币种。

### Q14：人工加减款有什么风控措施？

**答**：人工加减款不是钱包直接操作，而是通过 ser-finance 的审核流程——操作员提交 → 另一个审核员审批（提交人≠审核人）→ 审批通过后财务调钱包 `ManualCredit/ManualDebit` RPC。钱包侧只负责执行，通过 `orderNo` 幂等 + 审计日志确保可追溯。

### Q15：如果产品说要加一个新币种 BRL（巴西雷亚尔），改动大吗？

**答**：改动极小。B端管理员在"币种配置"页面新增即可——填入编码 BRL、名称、类型=法币、小数位=2、千分位=.、浮动%、阈值%。不需要改代码、不需要发版。这就是"币种配置"模块的设计价值——所有币种行为由配置驱动，不硬编码。

### Q16：为什么用 Thrift string 传金额而不是 double？

**答**：`double` 在 IEEE 754 标准下有精度损失——`0.1` 存储为 `0.100000000000000005551115...`。金融场景多次运算后误差累积会导致"钱对不上"。`string "100.50000000"` 零精度损失，Service 层用 `decimal.NewFromString()` 转换后精确运算。这是 Stripe、Square 等一线支付公司 API 的标准做法。

### Q17：版本号（version字段）和 FOR UPDATE 都用，是不是重复了？

**答**：不重复，是"双保险"。FOR UPDATE 是主力防线，确保同时只有一个事务修改余额。version 是辅助防线，防止极端场景（比如代码重构时不小心去掉了 FOR UPDATE）。两个同时存在的代价是多一个自增操作（几乎为零），但多一层安全保障。金融系统宁可"过度防御"也不能"欠防御"。

### Q18：钱包模块怎么保证不被 DDoS 打垮？

**答**：① gate-font/gate-back 层有限流（已有中间件）② Redis SetNX 防同一用户短时间重复请求（10s窗口）③ 核心资金操作有 FOR UPDATE 行锁，天然串行化不会被打爆（只是慢，不会错）④ 查询类接口走缓存/索引，TiDB 自动水平扩展。

### Q19：11 张表会不会太多了？能不能合并？

**答**：其实 11 张是偏少的。每张表对应一个独立的业务实体，职责清晰不交叉。如果硬合并（比如三种订单合一张表），会导致大量 NULL 字段和复杂条件查询。参考 Java 法币钱包模块有 15 张表、TRC20 虚拟币模块有 12 张表——11 张是合理且克制的。

### Q20：你最担心的风险是什么？

**答**：**ser-finance 的 IDL 和 RPC 契约未定义**。钱包的5个出站RPC依赖全部指向 ser-finance，但它的 IDL 目录甚至还没创建。这意味着 Phase 4（充值/提现）的联调存在时间风险。我的对策是：① 首期通过 Mock 独立开发 ② 尽快与财务开发者对齐 5 个出站 + 7 个入站共 12 个 RPC 的签名 ③ Phase 0-3 不依赖财务，先交付可验证版本。
