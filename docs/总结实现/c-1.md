## 第一章 设计思路与整体认知

### 1.1 我们在做什么

ser-wallet 是平台资金流转的"心脏"。用户的每一分钱——充值进来、提现出去、兑换币种、赢取奖励——都要经过这个模块。它不是一个简单的CRUD服务，而是一个**金融级别的资金安全系统**。

我们的核心使命是三个字：**不差钱**。

- 每一笔余额变更都有据可查（审计性）
- 同一笔操作绝不会被执行两次（幂等性）
- 并发场景下余额永远不会出现负数或超扣（一致性）
- 提现冻结的钱，审核驳回后100%回到用户可用余额（可逆性）

### 1.2 设计思路的六个关键决策

**决策一：钱包 + 币种配置合为一个服务**

币种配置是钱包所有操作的前置依赖（汇率、精度、展示规则），将它们放在同一个服务内，避免了高频RPC调用的网络损耗，同时简化了部署。IDL层面已做好拆分（wallet_rpc.thrift + currency.thrift），未来若需独立部署可平滑拆出。

**决策二：available_balance + frozen_balance 双字段设计**

一个余额字段无法表达"提现审核中"的资金状态。双字段让我们实现了提现的三步生命周期：冻结 → 确认扣除/解冻归还，用户随时能看清"可用"和"冻结中"各多少。

**决策三：CAS 乐观锁而非悲观锁**

金融系统的天然直觉是"加锁"，但我们选择了更轻量的方案——在 UPDATE SQL 的 WHERE 条件中直接做余额充足检查 `available_balance >= amount`。单条 SQL 的原子性由 MySQL InnoDB 保证，无需额外的 SELECT FOR UPDATE 或 Redis 分布式锁。

**决策四：(order_id, log_type) 联合唯一索引实现幂等**

不依赖 Redis Token、不依赖前端防重，而是在数据库层面用唯一索引做最终防线。同一笔业务单号+同一类型的操作只允许插入一条流水，重复调用直接返回已有结果。这个设计一举两得：既是幂等机制，又是审计流水。

**决策五：string 类型传递金额，DECIMAL(20,8) 存储**

从 Thrift IDL 到 Go 服务到数据库，金额全程以 string/Decimal 形式流转，杜绝了 float64 的精度丢失。DECIMAL(20,8) 能容纳 VND 的千亿级数字和 USDT 的8位小数。

**决策六：5种子钱包 × 5种币种 = 矩阵式隔离**

每个用户最多 25 个钱包记录（5类型 × 5币种），彼此完全隔离。这个设计天然支持未来新增币种（只需 INSERT currency_config 一行数据）和新增钱包类型（只需新增枚举值）。

### 1.3 一句话概括我们的设计哲学

> **用最简单的机制解决最关键的问题**：单表事务代替分布式事务，唯一索引代替 Redis 幂等令牌，SQL WHERE 代替悲观锁。不引入不必要的复杂度，但在资金安全上绝不妥协。

## 第二章 模块边界与职责划分

### 2.1 我负责什么（主责）

| 子模块 | 职责范围 | 核心能力 |
|--------|---------|---------|
| 钱包核心 | 余额管理、资金冻结/解冻、流水记录 | CAS余额更新、幂等流水、审计追踪 |
| 充值流程 | 创建充值订单、USDT支付页、重复拦截、订单超时 | 订单状态机、汇率锁定、回调入账 |
| 提现流程 | 创建提现订单、冻结余额、账户管理 | 三级审核状态机、冻结/解冻/确认 |
| 兑换功能 | 法币→BSB单向兑换、赠送入奖励钱包 | 汇率计算、三笔流水原子写入 |
| 记录查询 | 充值/提现/兑换/奖励四类记录分页查询 | 多维度索引、币种隔离 |
| 币种配置 | 币种CRUD、汇率管理、基准币种设置 | 三源汇率聚合、阈值触发更新 |
| RPC暴露 | 向财务/投注/直播等模块提供余额操作能力 | 19个RPC接口、P0-P2分级 |

### 2.2 我不负责什么（参考/联动）

| 模块 | 负责人 | 与我的关系 |
|------|--------|-----------|
| ser-finance（财务） | 财务同事 | 最紧密：充值/提现方式、审核流程、通道对接都在财务 |
| 充值通道对接 | 财务/支付 | 三方回调、通道选择不在我这里 |
| 提现审核 | 财务/风控 | 风控审核 + 财务审核 + 出款执行都在财务 |
| 奖金活动管理 | 运营/活动 | 活动规则配置待确认归属 |
| KYC认证 | 用户模块 | 我只调用查询接口，认证流程不归我 |

### 2.3 五种子钱包类型

| 编号 | 类型 | 资金来源 | 用途 | 状态 |
|------|------|---------|------|------|
| 1 | 中心钱包 | 充值、返奖、奖励完成转入 | 消费（投注、购买）、提现 | 核心-必做 |
| 2 | 奖励钱包 | 活动奖金、签到、任务、VIP | 需完成N倍流水后自动转入中心钱包 | 核心-必做 |
| 3 | 主播钱包 | 直播打赏礼物收入 | 可提现或转入中心钱包（需流水审计） | 核心-必做 |
| 4 | 代理钱包 | 分销佣金 | 可提现或转入中心钱包 | 预留-枚举占位 |
| 5 | 场馆钱包 | 中心+奖励临时转入 | 游戏场馆内使用，退出时原路返还 | 预留-枚举占位 |

### 3.2 表① wallet — 用户钱包余额表

```sql
CREATE TABLE `wallet` (
  `id`                BIGINT        NOT NULL AUTO_INCREMENT  COMMENT '主键ID',
  `user_id`           BIGINT        NOT NULL                 COMMENT '用户ID',
  `currency_code`     VARCHAR(10)   NOT NULL                 COMMENT '币种代码（VND/IDR/THB/BSB/USDT）',
  `wallet_type`       INT           NOT NULL                 COMMENT '钱包类型 1中心 2奖励 3主播 4代理 5场馆',
  `available_balance` DECIMAL(20,8) NOT NULL DEFAULT 0       COMMENT '可用余额',
  `frozen_balance`    DECIMAL(20,8) NOT NULL DEFAULT 0       COMMENT '冻结余额（提现待审核时冻结）',
  `create_by`         BIGINT        NOT NULL DEFAULT 0,
  `create_by_name`    VARCHAR(50)   NOT NULL DEFAULT '',
  `create_at`         BIGINT        NOT NULL DEFAULT 0       COMMENT '创建时间（毫秒时间戳）',
  `update_by`         BIGINT        NOT NULL DEFAULT 0,
  `update_by_name`    VARCHAR(50)   NOT NULL DEFAULT '',
  `update_at`         BIGINT        NOT NULL DEFAULT 0,
  `delete_at`         BIGINT        NOT NULL DEFAULT 0       COMMENT '软删除（0=未删除）',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_user_currency_type` (`user_id`, `currency_code`, `wallet_type`, `delete_at`),
  KEY `idx_user_id` (`user_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户钱包余额表';
```

**设计要点**：
- `available_balance + frozen_balance = 用户真实总余额`，任何操作都不能破坏这个恒等式
- UNIQUE KEY 保证一个用户一个币种一种钱包类型只有一条记录
- 所有金额字段 DECIMAL(20,8)，兼容 VND 千亿级和 USDT 8位小数

### 3.3 表② transaction_log — 余额变更流水表

```sql
CREATE TABLE `transaction_log` (
  `id`              BIGINT        NOT NULL AUTO_INCREMENT,
  `user_id`         BIGINT        NOT NULL,
  `currency_code`   VARCHAR(10)   NOT NULL,
  `wallet_type`     INT           NOT NULL,
  `log_type`        INT           NOT NULL                 COMMENT '流水类型（见枚举4.4）',
  `amount`          DECIMAL(20,8) NOT NULL                 COMMENT '变更金额（正数）',
  `direction`       INT           NOT NULL                 COMMENT '方向 1=收入 2=支出',
  `balance_before`  DECIMAL(20,8) NOT NULL                 COMMENT '变更前可用余额',
  `balance_after`   DECIMAL(20,8) NOT NULL                 COMMENT '变更后可用余额',
  `order_id`        VARCHAR(50)   NOT NULL                 COMMENT '关联业务单号（幂等依据）',
  `remark`          VARCHAR(500)  NOT NULL DEFAULT '',
  `create_by`       BIGINT        NOT NULL DEFAULT 0,
  `create_by_name`  VARCHAR(50)   NOT NULL DEFAULT '',
  `create_at`       BIGINT        NOT NULL DEFAULT 0,
  `update_by`       BIGINT        NOT NULL DEFAULT 0,
  `update_by_name`  VARCHAR(50)   NOT NULL DEFAULT '',
  `update_at`       BIGINT        NOT NULL DEFAULT 0,
  `delete_at`       BIGINT        NOT NULL DEFAULT 0,
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_order_log_type` (`order_id`, `log_type`),
  KEY `idx_user_currency_time` (`user_id`, `currency_code`, `create_at`),
  KEY `idx_user_type_time` (`user_id`, `log_type`, `create_at`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='余额变更流水表';
```

**设计要点**：
- `UNIQUE KEY (order_id, log_type)` 是整个系统幂等性的基石。同一笔订单的同类型操作只能插入一次
- `balance_before / balance_after` 是审计对账的核心依据
- 同一笔提现订单会产生两条流水：log_type=2（冻结）+ log_type=3（确认扣除）或 log_type=4（解冻）
- `amount` 始终为正数，由 `direction` 字段标识收支方向

### 3.4 表③ deposit_order — 充值订单表

```sql
CREATE TABLE `deposit_order` (
  `id`                    BIGINT          NOT NULL AUTO_INCREMENT,
  `order_no`              VARCHAR(50)     NOT NULL                 COMMENT '平台订单号（C+SnowflakeID）',
  `user_id`               BIGINT          NOT NULL,
  `user_nickname`         VARCHAR(100)    NOT NULL DEFAULT ''      COMMENT '用户昵称（冗余，方便B端展示）',
  `currency_code`         VARCHAR(10)     NOT NULL,
  `method_type`           INT             NOT NULL                 COMMENT '充值方式 1=USDT 2=银行 3=电子钱包',
  `method_name`           VARCHAR(50)     NOT NULL DEFAULT '',
  `amount`                DECIMAL(20,8)   NOT NULL                 COMMENT '订单金额',
  `fee_rate`              DECIMAL(10,4)   NOT NULL DEFAULT 0       COMMENT '手续费率（0.01=1%）',
  `fee_amount`            DECIMAL(20,8)   NOT NULL DEFAULT 0,
  `arrival_amount`        DECIMAL(20,8)   NOT NULL DEFAULT 0       COMMENT '到账金额 = amount - fee_amount',
  `rate_snapshot`         DECIMAL(20,8)   NOT NULL DEFAULT 0       COMMENT '下单时锁定的汇率',
  `payment_currency`      VARCHAR(10)     NOT NULL DEFAULT '',
  `payment_amount`        DECIMAL(20,8)   NOT NULL DEFAULT 0,
  `platform_rate`         DECIMAL(20,8)   NOT NULL DEFAULT 0       COMMENT '平台汇率快照（B端统计用）',
  `bonus_activity_id`     BIGINT          NOT NULL DEFAULT 0,
  `bonus_activity_name`   VARCHAR(100)    NOT NULL DEFAULT '',
  `bonus_ratio`           DECIMAL(10,4)   NOT NULL DEFAULT 0       COMMENT '奖金比例（0.50=50%）',
  `bonus_amount`          DECIMAL(20,8)   NOT NULL DEFAULT 0       COMMENT '奖金 = amount × bonus_ratio',
  `channel_order_no`      VARCHAR(100)    NOT NULL DEFAULT ''      COMMENT '三方通道订单号',
  `channel_name`          VARCHAR(100)    NOT NULL DEFAULT '',
  `payment_url`           VARCHAR(500)    NOT NULL DEFAULT ''      COMMENT '三方支付页URL',
  `status`                INT             NOT NULL DEFAULT 1       COMMENT '1待支付 2成功 3失败',
  `fail_reason`           VARCHAR(200)    NOT NULL DEFAULT '',
  `complete_at`           BIGINT          NOT NULL DEFAULT 0,
  `expire_at`             BIGINT          NOT NULL DEFAULT 0       COMMENT '订单过期时间',
  `base_currency_amount`  DECIMAL(20,8)   NOT NULL DEFAULT 0       COMMENT '基准币种金额（B端统计）',
  `base_currency_bonus`   DECIMAL(20,8)   NOT NULL DEFAULT 0,
  `base_currency_fee`     DECIMAL(20,8)   NOT NULL DEFAULT 0,
  `create_by`       BIGINT    NOT NULL DEFAULT 0,
  `create_by_name`  VARCHAR(50) NOT NULL DEFAULT '',
  `create_at`       BIGINT    NOT NULL DEFAULT 0,
  `update_by`       BIGINT    NOT NULL DEFAULT 0,
  `update_by_name`  VARCHAR(50) NOT NULL DEFAULT '',
  `update_at`       BIGINT    NOT NULL DEFAULT 0,
  `delete_at`       BIGINT    NOT NULL DEFAULT 0,
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_order_no` (`order_no`),
  KEY `idx_user_currency_status` (`user_id`, `currency_code`, `status`),
  KEY `idx_user_currency_time` (`user_id`, `currency_code`, `create_at`),
  KEY `idx_status_expire` (`status`, `expire_at`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='充值订单表';
```

**设计要点**：
- `idx_user_currency_status` 用于重复转账检查（同用户同币种待支付订单数）
- `idx_status_expire` 用于定时任务扫描超时订单
- `rate_snapshot` 在创建订单时锁定，防止支付过程中汇率波动
- 奖金相关字段（bonus_*）在创建时快照，与活动模块解耦

**汇率计算规则**（不存储，动态计算）：

## 第四章 枚举定义与状态机

### 4.1 wallet_type 钱包类型

| 值 | 名称 | 必做 | 说明 |
|----|------|------|------|
| 1 | 中心钱包 | 是 | 充值入账、消费优先扣减、可提现 |
| 2 | 奖励钱包 | 是 | 活动奖金临时存放，完成流水后转中心 |
| 3 | 主播钱包 | 是 | 打赏收入，可提现或转中心 |
| 4 | 代理钱包 | 预留 | 分销佣金，枚举占位 |
| 5 | 场馆钱包 | 预留 | 游戏场馆临时资金，枚举占位 |

### 4.2 currency_type 币种类型

| 值 | 名称 | 币种 |
|----|------|------|
| 1 | 法币 | VND, IDR, THB |
| 2 | 加密货币 | USDT |
| 3 | 平台币 | BSB |

### 4.3 提现订单状态机

```
用户提交提现 → [1] 待风控审核
                  ├── 风控驳回 → [2] 风控已驳回 → 解冻余额（终态）
                  └── 风控通过 → [3] 待财务审核
                                    ├── 财务驳回 → [4] 财务已驳回 → 解冻余额（终态）
                                    └── 财务通过 → [5] 待出款
                                                      ├── 出款失败 → [6] 出款失败 → 解冻余额（终态）
                                                      └── 出款成功 → [7] 出款成功 → 确认扣除冻结（终态）
```

### 4.4 log_type 流水类型（12种）

| 值 | 名称 | 方向 | 触发接口 |
|----|------|------|---------|
| 1 | 充值入账 | 收入 | RpcCreditWallet |
| 2 | 提现冻结 | 支出 | C端创建提现 |
| 3 | 提现确认扣除 | 支出 | RpcConfirmFrozenDeduction |
| 4 | 提现解冻 | 收入 | RpcUnfreezeBalance |
| 5 | 兑换扣除法币 | 支出 | C端执行兑换 |
| 6 | 兑换入账BSB | 收入 | C端执行兑换 |
| 7 | 兑换赠送奖励 | 收入 | C端执行兑换 |
| 8 | 人工加款 | 收入 | RpcCreditWallet(ManualCredit) |
| 9 | 人工减款 | 支出 | RpcDebitWallet |
| 10 | 补单入账 | 收入 | RpcSupplementCredit |
| 11 | 补单奖金 | 收入 | RpcSupplementCredit |
| 12 | 奖励入账 | 收入 | 签到/任务/VIP/分销 |

### 4.5 订单号前缀约定

| 业务 | 前缀 | 生成方 | 格式 |

## 第六章 核心业务流程

### 6.1 充值流程

```
用户选币种 → 选充值方式(调财务RPC) → 输入金额
    → 重复转账检查(同用户+同币种+同金额，1-2笔警告/≥3笔阻断)
    → 选择奖金活动(可选) → 门槛校验(金额≥活动最低)
    → 点击"去充值"
    → 锁定汇率快照 → 创建充值订单(本地DB，状态=待支付)
    → 请求三方通道 → 返回支付页面/链接
    → 用户支付 → 三方回调
    → 验签+金额一致性校验
    → CreditWallet入账: 订单金额→中心钱包 + 奖金→奖励钱包
    → 订单状态→已完成
```

**三种充值方式的差异**：

| 方式 | 承载页 | 重复拦截 | 特殊处理 |
|------|--------|---------|---------|
| USDT链上 | 平台自有页 | 不适用 | TRC20/ERC20/BEP20选择，展示收款地址+QR码 |
| 银行转账 | 三方承载页 | 适用 | 展示汇款信息（银行+账号+金额+备注） |
| QR扫码 | 三方承载页 | 适用 | QRIS(印尼)/PromptPay(泰国)，倒计时过期 |

### 6.2 提现流程

```
用户点击提现 → 查询KYC状态(调ser-kyc)
    → 未认证: 弹窗"去认证" / "继续提现(仅USDT)"
    → 已认证: 进入提现页
    → 选提现方式(调财务RPC) → 获取限额费率
    → 首次: 填写账户信息 / 非首次: 自动带出(只读)
    → 账户持有人 = KYC姓名(灰显不可改)
    → 输入金额 → 校验(≥最低 ≤单笔最高 ≤日限额)
    → 展示: 提现金额 / 手续费 / 到账金额
    → 确认提现
    → 获取出款汇率(BSB提现时)
    → ① 冻结余额(本地DB: available -= N, frozen += N)
    → ② 提交提现订单到财务(调财务RPC O4)
         ├── 成功 → 返回"提现中"弹窗
         └── 失败 → **回滚**: 解冻余额(frozen -= N, available += N)
    → 财务端三级审核:
         ├── 风控驳回/财务驳回/出款失败 → 调 RpcUnfreezeBalance → 解冻
         └── 出款成功 → 调 RpcConfirmFrozenDeduction → 确认扣除
```

**冻结/解冻/确认 三步生命周期**：

| 步骤 | 触发 | available_balance | frozen_balance | 说明 |
|------|------|------------------|----------------|------|
| 冻结 | 创建提现 | -N | +N | 钱还是用户的，但不能再花 |
| 解冻 | 审核驳回/出款失败 | +N | -N | 钱回到可用 |
| 确认扣除 | 出款成功 | 不变 | -N | 钱真正离开系统 |

### 6.3 兑换流程

```
用户选择兑换 → 获取平台汇率(本地DB) → 展示: 1 BSB = X 法币
    → 输入法币金额(不超过当前余额)
    → 实时计算:
         兑换获得 = 法币金额 / 平台汇率
         赠送奖金 = 兑换获得 × 10%(可配置)
         实际到账 = 兑换获得 + 赠送奖金
    → 确认兑换
    → 获取锁汇汇率(本地DB)
    → 单事务内执行3笔操作:
         ① 扣减法币钱包(log_type=5)
         ② 加BSB中心钱包(log_type=6)
         ③ 加BSB奖励钱包-赠送部分(log_type=7)
    → 创建exchange_record
    → 返回"兑换成功"
```

**零外部依赖**：兑换是纯本地操作，只查一次币种配置的汇率，不调任何外部RPC。

### 6.4 汇率更新流程

```
定时任务(~2分钟一次)
    → 并发请求3个三方汇率API
    → 取可用结果的平均值(≥1个可用即有效)
    → 计算偏差: |实时汇率 - 平台汇率| / 实时汇率 × 100%
    → 偏差 < 阈值 → 仅更新realtime_rate字段，不动platform_rate
    → 偏差 ≥ 阈值 → 更新platform_rate + 记录rate_change_log
    → 清除Redis汇率缓存(下次查询时重建)
    → 全部API失败 → 保持当前平台汇率不变(降级)
```

### 6.5 CAS余额更新的核心事务

```sql
-- 所有余额变更都在一个数据库事务内完成：
BEGIN;
  -- 步骤1：幂等检查（插入流水，利用唯一索引防重复）
  INSERT INTO transaction_log (order_id, log_type, user_id, currency_code, wallet_type,
    amount, direction, balance_before, balance_after, ...)
  VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?);
  -- 如果 (order_id, log_type) 已存在 → 触发 Duplicate Key Error → 回滚返回已有结果

  -- 步骤2：更新余额（WHERE条件保证余额充足）
  UPDATE wallet
  SET available_balance = available_balance + ?,
      update_at = ?, update_by = ?
  WHERE user_id = ? AND currency_code = ? AND wallet_type = ?
    AND available_balance >= ?;  -- 扣款时此条件生效，入账时设为0
  -- affected_rows = 0 → 余额不足 → 回滚

  -- 步骤3：读取最终状态
  SELECT available_balance, frozen_balance FROM wallet WHERE ...;
COMMIT;
```

## 第八章 技术挑战与解决方案

### 8.1 P0 级挑战（必须解决）

#### 挑战1：余额并发安全

**问题**：同一用户同时发生充值入账 + 提现冻结 + 投注扣款，余额会不会乱？

**方案**：SQL层面的原子性保证

```sql
-- 扣款场景：WHERE条件确保余额充足，单条UPDATE原子执行
UPDATE wallet
SET available_balance = available_balance - ?
WHERE user_id = ? AND currency_code = ? AND wallet_type = ?
  AND available_balance >= ?;
-- affected_rows = 0 表示余额不足，1 表示成功
```

**为什么不用分布式锁**：
- 我们操作的是单一数据库中的单一表，MySQL InnoDB 的行级锁已经保证了并发安全
- 加分布式锁反而引入了额外的复杂度（锁超时、死锁、锁续期）
- CAS 方案简单、可靠、性能好

#### 挑战2：操作幂等性

**问题**：财务模块重试 RPC 调用（网络超时、回调重复），余额会不会被加两次？

**方案**：(order_id, log_type) 联合唯一索引

```go
func CreditWallet(orderId string, amount decimal.Decimal) {
    // 1. 尝试插入流水记录
    err := insertTransactionLog(orderId, LogTypeDeposit, amount)
    if isDuplicateKeyError(err) {
        // 幂等：已处理过，直接返回已有结果
        return getExistingResult(orderId, LogTypeDeposit)
    }
    // 2. 插入成功，继续更新余额（同一事务内）
    updateBalance(...)
}
```

**为什么不用 Redis 幂等令牌**：
- Redis 有丢失风险（宕机、重启），数据库唯一索引不会丢
- 不需要额外的 Token 申请/消费流程
- 流水记录本身就是审计需求，一举两得

#### 挑战3：分布式数据一致性

**问题**：充值入账跨钱包和财务两个服务，如何保证一致？

**方案**：重试 + 幂等 + 定时对账（而非分布式事务）

```
调用方（财务）── 重试策略: 指数退避 3次重试
被调方（钱包）── 幂等机制: orderId去重
兜底            ── 定时对账任务: 每日核对未完成记录
```

**为什么不用 TCC/Saga**：
- 钱包模块不需要跨数据库事务，所有10张表在同一个MySQL实例
- TCC 的 Try/Confirm/Cancel 三步引入大量代码和状态，金融系统的复杂性反而增加
- "重试+幂等+对账"是互联网支付行业的实战标准

#### 挑战4：金额精度

**问题**：VND 面额到千亿级（如 100,000,000,000），USDT 需要8位小数，float64 无法兼顾

**方案**：全链路 string/Decimal

```
前端 → Thrift IDL(string) → Go(decimal.Decimal) → DB(DECIMAL(20,8))
                                                      ↓
                         Go(decimal.StringFixed) ← DB(string读取)
                                ↓
                     Thrift IDL(string) → 前端
```

```go
// 兑换计算示例
rate := decimal.NewFromString("26200")       // 1 BSB = 26200 VND
vnd := decimal.NewFromString("1000000")      // 用户输入 1,000,000 VND
bsb := vnd.Div(rate).RoundFloor(2)           // 38.16 BSB（向下取整保护平台）
bonus := bsb.Mul(decimal.NewFromFloat(0.10)) // 3.81 BSB 赠送
total := bsb.Add(bonus)                      // 41.97 BSB 实际到账
```

#### 挑战5：提现事务补偿

**问题**：本地冻结余额成功，但调财务 SubmitWithdrawOrder(O4) 失败，用户资金被永久冻结

**方案**：两阶段补偿 + 超时兜底

```go
func CreateWithdrawalOrder(req) {
    // 阶段1：本地冻结
    freezeResult := FreezeBalance(req.UserId, req.Amount)
    if freezeResult.Error != nil {
        return err // 余额不足
    }

    // 阶段2：提交财务
    financeResult := finance.SubmitWithdrawOrder(req)
    if financeResult.Error != nil {
        // 补偿：立即解冻
        UnfreezeBalance(req.UserId, req.OrderId)
        return err
    }

    return success
}
```

兜底措施：定时任务扫描 "冻结超过X小时但财务侧无对应订单" 的异常记录，触发告警。

### 8.2 P1 级挑战（应当解决）

#### 挑战6：汇率系统稳定性

**问题**：3个三方汇率API可能超时/返回异常/数据差异大

**方案**：3源聚合 + 阈值触发 + 降级保护

```
1. 并发请求3个API（超时3秒）
2. 过滤异常响应（负数、零值、偏差>50%的剔除）
3. 取有效值的平均
4. 偏差 < 阈值 → 不更新平台汇率（减少不必要波动）
5. 偏差 ≥ 阈值 → 更新平台汇率 + 记录变更日志
6. 全部失败 → 保持当前平台汇率（降级而非清零）
```

**缓存策略**：
- 展示类查询：Redis缓存，TTL 30秒
- 执行类查询（创建订单）：直接读DB，保证实时性
- 用户余额：**不缓存**（变更频繁，一致性优先）

#### 挑战7：提现状态机完整性

**问题**：7个状态、3个审核阶段，状态跳转逻辑复杂，并发操作可能导致非法状态

**方案**：应用层FSM定义 + SQL WHERE 状态守卫

```go
// 应用层：定义合法状态转换
var withdrawalFSM = map[int32][]int32{
    1: {2, 3},    // 待风控 → 驳回 | 通过
    3: {4, 5},    // 待财务 → 驳回 | 通过
    5: {6, 7},    // 待出款 → 失败 | 成功
}

// 数据库层：WHERE条件防止非法跳转
func UpdateStatus(orderNo string, fromStatus, toStatus int32) error {
    result := db.Model(&WithdrawalOrder{}).
        Where("order_no = ? AND status = ?", orderNo, fromStatus).
        Update("status", toStatus)
    if result.RowsAffected == 0 {
        return errors.New("状态已变更，请刷新")
    }
    return nil
}
```

#### 挑战8：防重复提交

**问题**：用户快速连点"充值""提现"按钮，创建了重复订单

**方案**：四层防御

```
第1层（前端）：按钮点击后禁用+loading
第2层（网关）：用户维度限流（可选）
第3层（服务）：Redis SETNX
    Key: wallet:submit:{userId}:{action}
    TTL: 5秒
    冲突返回"操作过于频繁"
第4层（业务）：充值重复检查(C7)
    同用户+同币种+同金额的待支付订单数
    1-2笔 → 软拦截（警告+可继续）
    ≥3笔 → 硬拦截（阻断创建）
```

#### 挑战9：充值订单超时

**问题**：用户创建订单后不支付，订单永久挂在"待支付"状态

**方案**：定时任务扫描 + 状态转为"已过期"

```go
// 每5分钟执行
func DepositOrderAutoExpire() {
    expireMinutes := systemConfig.Get("deposit_order_expire_minutes") // 30
    threshold := time.Now().Add(-time.Duration(expireMinutes) * time.Minute).UnixMilli()

    db.Model(&DepositOrder{}).
        Where("status = ? AND create_at < ?", StatusPending, threshold).
        Update("status", StatusFailed).
        Update("fail_reason", "订单超时未支付")
}
```

### 8.3 P2 级挑战（可后置）

#### 挑战10：每日对账机制

```
对于每个(user_id, currency_code, wallet_type)组合：
  ∑(transaction_log direction=收入) - ∑(transaction_log direction=支出)
  = wallet.available_balance + wallet.frozen_balance

  不匹配 → 告警 → 人工调查
```

#### 挑战11：敏感数据加密

- 银行卡号：存储加密（AES-256），展示脱敏（仅显示后4位）
- 加密地址：部分脱敏（前6位+...+后4位）
- 工程已有 `EncryptedString` GORM类型，复用即可

#### 挑战12：多币种天然隔离

所有查询和操作都以 `(user_id, currency_code, wallet_type)` 三元组为键，币种之间天然不会串数据。新增币种只需 INSERT 一条 currency_config 记录，零代码修改。

## 第十二章 需求评审高频Q&A预案

### Q1：为什么不用分布式事务框架？

**A**：钱包模块的10张表都在同一个MySQL实例内，所有余额操作都是本地事务，不需要跨库的分布式事务。跨服务的一致性（如财务调钱包入账）通过"调用方重试 + 被调方幂等 + 定时对账"三板斧解决。这是互联网支付行业的标准实践，比引入TCC/Saga框架简单可靠得多。

### Q2：CAS乐观锁在高并发下会不会大量失败？

**A**：钱包余额操作的并发冲突概率其实很低——同一个用户同一个钱包同时执行多次余额变更的场景不常见。即使冲突了，我们的CAS不需要业务层重试循环，因为 `UPDATE wallet SET available_balance = available_balance - ? WHERE available_balance >= ?` 本身就是原子操作。`affected_rows=0` 表示余额不足，直接返回业务错误即可，不是CAS冲突。

### Q3：DECIMAL(20,8) 会不会浪费存储空间？

**A**：DECIMAL(20,8) 在MySQL中占用约9-10字节。相比于业务安全性（VND的千亿级金额 + USDT的8位小数），这点存储代价完全可以接受。统一精度避免了不同币种用不同字段类型的复杂度。展示层面根据 currency_config.decimal_places 动态控制显示位数。

### Q4：汇率波动期间用户充值/提现怎么保证不亏？

**A**：三层保护机制：
1. **汇率浮动（rate_floating）**：入款汇率在平台汇率基础上上浮、出款汇率下浮，形成买卖价差（类似银行的买入/卖出价）
2. **阈值触发更新**：只有偏差超过阈值才更新平台汇率，避免微小波动频繁改价
3. **锁汇机制**：用户创建订单时锁定当时的汇率快照，支付过程中汇率变动不影响已创建的订单

### Q5：如果三方汇率API全挂了怎么办？

**A**：降级策略——保持当前平台汇率不变。定时任务每2分钟执行，3个API并发请求，只要1个可用就能计算均值。全挂的概率极低。即使全挂，平台汇率不更新不等于不可用，充值/提现/兑换仍然可以正常操作，只是汇率不是最新的。持续全挂超过阈值时间应触发运维告警。

### Q6：提现审核驳回后用户的钱去哪了？

**A**：100%返回用户可用余额。提现创建时我们做的是 `available -= N, frozen += N`（钱还是用户的，只是冻结了）。驳回时调用 RpcUnfreezeBalance 做 `frozen -= N, available += N`，钱从冻结回到可用。每一步都有 transaction_log 记录（log_type=2冻结、log_type=4解冻），balance_before/after 完整可审计。

### Q7：兑换赠送的10%进了奖励钱包，用户能直接提现吗？

**A**：不能。奖励钱包的资金需要完成N倍流水要求后才能转入中心钱包，只有中心钱包的余额才能提现。这个设计防止用户利用赠送机制套现（充值→兑换拿赠送→提现→再充值循环）。流水倍数由 exchange_record.turnover_multiplier 记录。

### Q8：transaction_log表增长最快，到千万级怎么处理？

**A**：初期不做分片，依靠索引保证查询性能。索引 `(user_id, currency_code, create_at)` 覆盖了C端记录查询的主要场景。到千万级时，可考虑两个方向：
1. **按时间归档**：3个月前的数据迁移到历史表
2. **按user_id分片**：水平拆分，同一用户的数据落在同一个分片上
具体方案视实际数据增长速度和查询性能指标决定。

### Q9：新增一个币种需要改代码吗？

**A**：不需要。只需在B端管理页面或数据库中INSERT一条 currency_config 记录，设置好展示规则（符号、千分位、小数位）。所有查询和计算都通过 currency_code 动态适配，不需要硬编码币种逻辑。

### Q10：如果财务模块还没开发完，钱包怎么先行开发？

**A**：分阶段独立推进：
- Phase 0（币种配置）零依赖财务，可立即开工
- Phase 1（余额查询）零依赖财务，可立即开工
- Phase 2（资金操作）中需要调财务的4个RPC，先用Mock函数模拟
- 联调时替换Mock为真实RPC客户端

Mock 策略已设计好，不阻塞钱包核心逻辑的开发和单元测试。

### Q11：消费扣款时中心钱包不够怎么办？

**A**：消费优先扣中心钱包，不足时从奖励钱包补足。RpcConsumeDebit 接口返回 `deductedFromCenter`（中心扣了多少）和 `deductedFromReward`（奖励扣了多少）的比例。返奖时按此比例分别入账，确保资金来源追踪不混乱。

### Q12：BSB是平台币，为什么不像积分一样直接发放？

**A**：BSB锚定USDT（10 BSB = 1 USDT），具有真实价值，所以需要通过兑换获取而非直接发放。兑换行为需要扣减对应法币余额，保证资金平衡。BSB的定位是"跨币种结算媒介"而非"无成本积分"。
