## 一、设计思路总纲

### 1.1 一句话概括

**钱包模块是整个平台的"账本+记账员"——所有资金变动最终都在这里完成余额增减和流水记录。**

设计核心六个字：**准、全、稳、幂、冻、对**

| 字 | 含义 | 落地手段 |
|----|------|---------|
| **准** | 金额精确不差分毫 | bigint存最小单位、不用float/decimal、统一精度规则 |
| **全** | 每一分钱的来龙去脉有记录 | wallet_transaction流水表、balance_before/after |
| **稳** | 并发操作不丢不错 | SELECT FOR UPDATE行锁 + version乐观锁 + 同事务写入 |
| **幂** | 重复请求只生效一次 | order_no唯一键 + 应用层检查 + 状态机 |
| **冻** | 提现资金状态可控 | 冻结三阶段（freeze → confirm/rollback）不可逆状态机 |
| **对** | 账实一致可审计 | 余额 = SUM(流水)、异步对账任务、balance_before/after链式校验 |

### 1.2 设计原则

**三个"只有"——职责分权：**
- 余额只有钱包能改。财务模块不直接操作余额，必须通过钱包RPC
- 通道只有财务能管。钱包不直接对接三方支付，必须通过财务的通道
- 汇率只有币种配置能出。钱包和财务都从币种配置获取汇率，不各自维护

**三个"统一"——收口管控：**
- 统一余额变更入口：所有资金操作必须经过 BalanceEngine，不允许绕过
- 统一订单号规范：C+16位（充值）、T+16位（提现）、B+16位（补单）、A+16位（加款）、M+16位（减款）
- 统一金额存储：bigint存最小单位，传输层i64，展示层按币种decimal_digits格式化

**一个"不"——不过度设计：**
- 不用分布式事务框架（TCC/Saga），本地事务+幂等重试够用
- 不用Redis分布式锁做资金操作主锁，数据库行锁就够
- 不提前做分库分表，流水表增长评估后按需处理
- 不把币种配置拆成独立服务，模块化单体（Modular Monolith）

### 1.3 盖房子思维——确定性骨架

不管需求细节怎么调整，以下设计绕不过：

```
绕不过的9项必须设计：
  1. 多币种多钱包余额表（wallet）—— 用户×币种×钱包类型=唯一余额
  2. 交易流水表（wallet_transaction）—— 每笔余额变动有记录
  3. 币种配置表（currency_config）—— 支持的币种和精度规则
  4. 汇率记录表（rate_change_log）—— 汇率数据和变更历史
  5. 充值/提现订单表 —— 状态机驱动的订单生命周期
  6. 余额操作原子性 —— SQL级保证
  7. 幂等机制 —— 充值回调/提现回调的重复处理防护
  8. 与财务模块的RPC边界 —— CreditDeposit/FreezeForWithdrawal等
  9. 定时汇率同步 —— 三方API取均值、偏差触发更新
```

## 二、模块定位与职责边界

### 2.1 三个模块的角色定位

```
┌─────────────────────────────────────────────────────────────┐
│                   资金系统的三个切面                           │
│                                                             │
│   币种配置 = 基础数据层    提供"度量衡"（币种定义+汇率规则）   │
│   钱  包  = 账务核心层    提供"账本"（余额管理+交易流水）      │
│   财务管理 = 运营管控层    提供"管道"（支付通道+审核流程）      │
│                                                             │
│   一句话：币种配置提供"尺子"，财务负责"搬运"，钱包负责"记账"  │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 我的职责范围

```
主责（深度掌握，由我实现）：
  ✦ 钱包模块（C端）—— 15个HTTP接口
  ✦ 币种配置（B端）—— 7个HTTP接口
  ✦ RPC暴露（供他人调用）—— 12个RPC接口
  ✦ 定时任务 —— 1个（汇率同步）
  ✦ 配套工作 —— 10张表DDL + 34个IDL方法 + 5个外部RPC对接

参考（清楚全貌，别人实现）：
  ○ 财务管理（B端）—— 约35-45个接口，需清楚双向RPC依赖
```

### 2.3 为什么币种配置和钱包是一个服务

| 考量 | 合并（选用） | 拆分 |
|------|-------------|------|
| 调用频率 | 钱包每个操作都读汇率/币种，内部函数调用0延迟 | 拆成两个服务每次增加1-3ms网络开销 |
| 数据耦合 | 钱包余额展示强依赖币种展示规则 | 拆开后两个服务还是紧密耦合 |
| 运维复杂度 | 1个服务部署 | 2个服务+2倍配置+2倍监控 |
| 团队边界 | 就我一个人做 | 没有拆分的组织动力 |

**结论：不拆。代码按包隔离，物理上一个服务——"模块化单体"（Modular Monolith）。工程中ser-app已有先例（Banner+HomeCategory+ContentStrategy三个域在一个服务内）。**

### 3.2 核心表字段设计要点

#### 3.2.1 currency_config（币种配置表）

```sql
CREATE TABLE `currency_config` (
    `id`                  bigint unsigned NOT NULL AUTO_INCREMENT,
    `currency_code`       varchar(16) NOT NULL COMMENT '币种编码 USDT/VND/IDR/THB/BSB',
    `currency_name`       varchar(64) NOT NULL COMMENT '币种名称',
    `currency_type`       tinyint unsigned NOT NULL COMMENT '1法定货币 2加密货币 3平台货币',
    `country_code`        varchar(16) DEFAULT '' COMMENT '国家/地区编码',
    `icon_url`            varchar(255) DEFAULT '' COMMENT '币种图标URL',
    `currency_symbol`     varchar(8) DEFAULT '' COMMENT '货币符号 ₫/Rp/฿/$',
    `thousands_separator` varchar(4) DEFAULT '' COMMENT '千分位符号',
    `decimal_separator`   varchar(4) DEFAULT '' COMMENT '小数点符号',
    `decimal_digits`      tinyint unsigned NOT NULL DEFAULT '2' COMMENT '小数位数',
    `is_base`             tinyint unsigned NOT NULL DEFAULT '0' COMMENT '是否基准币种 0否 1是',
    `rate_realtime`       bigint NOT NULL DEFAULT '0' COMMENT '实时汇率（×10^8）',
    `rate_platform`       bigint NOT NULL DEFAULT '0' COMMENT '平台汇率（×10^8）',
    `rate_deposit`        bigint NOT NULL DEFAULT '0' COMMENT '入款汇率（×10^8）',
    `rate_withdrawal`     bigint NOT NULL DEFAULT '0' COMMENT '出款汇率（×10^8）',
    `float_rate_percent`  int unsigned NOT NULL DEFAULT '0' COMMENT '汇率浮动百分比（×100）',
    `deviation_threshold` int unsigned NOT NULL DEFAULT '0' COMMENT '偏差阈值百分比（×100）',
    `sort_order`          int unsigned DEFAULT '1',
    `status`              tinyint unsigned NOT NULL DEFAULT '1' COMMENT '1启用 2禁用',
    -- 审计字段省略（create_by/update_by/create_at/update_at/delete_at）
    UNIQUE KEY `uk_currency_code` (`currency_code`, `delete_at`),
    KEY `idx_status` (`status`)
) COMMENT='币种配置表';
```

**设计决策**：币种定义和汇率合一张表。理由：1:1关系、总量4-6条、拆表只增JOIN复杂度无收益。

**汇率存储**：统一×10^8存bigint。例：1 USDT = 25432.50 VND → rate_platform = 2543250000000

#### 3.2.2 wallet（用户钱包表）——核心中的核心

```sql
CREATE TABLE `wallet` (
    `id`            bigint unsigned NOT NULL AUTO_INCREMENT,
    `user_id`       bigint unsigned NOT NULL COMMENT '用户ID',
    `currency_code` varchar(16) NOT NULL COMMENT '币种编码',
    `wallet_type`   tinyint unsigned NOT NULL COMMENT '1中心 2奖励 3主播 4代理(预留) 5场馆(预留)',
    `balance`       bigint NOT NULL DEFAULT '0' COMMENT '可用余额（最小单位）',
    `frozen_amount` bigint NOT NULL DEFAULT '0' COMMENT '冻结金额（最小单位）',
    `fund_source`   tinyint unsigned NOT NULL DEFAULT '0' COMMENT '资金来源 0无 1法币 2USDT 3混合',
    `version`       bigint unsigned NOT NULL DEFAULT '0' COMMENT '乐观锁版本号',
    `status`        tinyint unsigned NOT NULL DEFAULT '1' COMMENT '1正常 2冻结',
    UNIQUE KEY `uk_user_curr_type` (`user_id`, `currency_code`, `wallet_type`, `delete_at`),
    KEY `idx_user_id` (`user_id`)
) COMMENT='用户钱包表';
```

**关键设计5要点**：

1. **balance + frozen_amount 分离**：任何时刻 `balance + frozen_amount = 用户实际总资产`
2. **version 乐观锁**：每次余额变更 version+1，UPDATE时WHERE version=旧值
3. **fund_source 反洗钱追踪**：提现时校验"法币充→法币提"规则，避免回查全部充值历史
4. **wallet_type 预留**：枚举值4/5预留给代理/场馆，表结构已兼容
5. **首次写入自动创建**：用户首次充值某币种时BalanceEngine自动创建wallet记录

#### 3.2.3 wallet_transaction（交易流水表）——对账生命线

```sql
CREATE TABLE `wallet_transaction` (
    `id`             bigint unsigned NOT NULL AUTO_INCREMENT,
    `user_id`        bigint unsigned NOT NULL,
    `currency_code`  varchar(16) NOT NULL,
    `wallet_type`    tinyint unsigned NOT NULL,
    `tx_type`        tinyint unsigned NOT NULL COMMENT '1充值 2兑出 3兑入 4提现冻结 5提现解冻
                                                        6提现扣款 7人工加款 8人工减款 9补单 10奖金',
    `amount`         bigint NOT NULL COMMENT '变动金额（正=增加 负=减少）',
    `balance_before` bigint NOT NULL COMMENT '变更前可用余额',
    `balance_after`  bigint NOT NULL COMMENT '变更后可用余额',
    `frozen_before`  bigint NOT NULL DEFAULT '0' COMMENT '变更前冻结金额',
    `frozen_after`   bigint NOT NULL DEFAULT '0' COMMENT '变更后冻结金额',
    `order_no`       varchar(32) NOT NULL COMMENT '关联订单号',
    `status`         tinyint unsigned NOT NULL DEFAULT '1' COMMENT '1已完成 2处理中 3已失败 4已拒绝',
    UNIQUE KEY `uk_order_wallet_tx` (`order_no`, `wallet_type`, `tx_type`, `delete_at`),
    KEY `idx_user_curr_type` (`user_id`, `currency_code`, `tx_type`),
    KEY `idx_order_no` (`order_no`),
    KEY `idx_user_create_at` (`user_id`, `create_at`)
) COMMENT='钱包交易流水表';
```

**关键设计**：
- **10种交易类型**覆盖所有余额变动场景
- **uk_order_wallet_tx三字段唯一键**：同一订单号可能产生多条流水（如兑换：法币扣出+BSB加入+奖金加入），用wallet_type+tx_type区分
- **balance_before/after + frozen_before/after**：对账校验 `balance_after == balance_before + amount`

#### 3.2.4 其他表概要

| 表 | 核心字段 | 状态机 | 数据量评估 |
|----|---------|--------|-----------|
| deposit_order | order_no(C+16), amount, deposit_method, bonus_*, rate_snapshot | 5态（待支付→进行中→已完成/失败/关闭） | 日增数千 |
| withdrawal_order | order_no(T+16), amount, fee_amount, withdrawal_method, account_id | 6态（处理中→审核→出款→完成/失败/拒绝） | 日增数千 |
| withdrawal_account | user_id, account_type, bank_name, account_no(加密), holder_name | — | 用户级 |
| exchange_order | order_no, from_currency, to_currency, amount, rate_snapshot | 3态（处理中→完成/失败） | 日增数千 |
| rate_change_log | currency_code, rate_*_before/after, deviation_percent, trigger_type | — | 年增约2万 |

### 3.3 金额存储统一规则

```
存储层：bigint（最小单位）
  法币（VND/IDR/THB）：× 100（分），精度2位小数
  加密货币（USDT）：    × 10^8（聪），精度8位小数
  平台币（BSB）：       × 100，精度2位小数

传输层：i64（Thrift IDL），和存储层一致
展示层：按 currency_config.decimal_digits 格式化

为什么不用 decimal：
  1. TiDB的bigint性能优于decimal，索引更高效
  2. Go的int64运算比decimal库快一个数量级
  3. 整数不存在精度丢失
  4. 支付宝、微信支付、Stripe都是整数存储（单位：分）
```

## 六、核心技术方案

### 6.1 余额变更引擎（BalanceEngine）——最关键的设计

**定位：所有资金变动都必须通过这个引擎，不允许绕过。**

```
BalanceEngine 提供5个核心操作：

  Add()           增加可用余额     → 充值入账/人工加款/补单/兑换加BSB
  Deduct()        扣减可用余额     → 人工减款/兑换扣法币
  Freeze()        冻结             → 可用余额→冻结余额（提现创建时）
  Unfreeze()      解冻             → 冻结余额→可用余额（审核驳回时）
  ConfirmDeduct() 确认扣除冻结额   → 冻结余额永久移出（出款成功时）
```

**每个操作内部的统一流程**：

```go
func (e *BalanceEngine) Add(ctx context.Context, op *BalanceOp) error {
    return e.walletRepo.WithTransaction(func(tx *query.Query) error {
        // 第1步：幂等检查（应用层快速拒绝）
        exists := tx.WalletTransaction.Where(orderNo, walletType, txType).Exist()
        if exists { return nil } // 已处理，直接成功

        // 第2步：行锁定位
        wallet := tx.Wallet.Where(userId, currencyCode, walletType).ForUpdate().First()
        if wallet == nil { wallet = autoCreateWallet(tx, op) } // 首次自动创建

        // 第3步：计算新余额
        newBalance := wallet.Balance + op.Amount

        // 第4步：乐观锁更新
        affected := tx.Wallet.Where(id=wallet.ID, version=wallet.Version).
            UpdateSimple(balance=newBalance, version=wallet.Version+1)
        if affected == 0 { return ErrConcurrentConflict } // 冲突重试

        // 第5步：同事务写流水（balance_before/after）
        tx.WalletTransaction.Create(&WalletTransaction{
            BalanceBefore: wallet.Balance,
            BalanceAfter:  newBalance,
            Amount:        op.Amount,
            OrderNo:       op.OrderNo,
            // ...
        })
        return nil
    })
}
```

### 6.2 三层并发防线

```
第1层：数据库行锁（事务内强一致）
  SELECT ... FOR UPDATE WHERE user_id=? AND currency_code=? AND wallet_type=?
  锁粒度：用户+币种+钱包类型（不影响其他用户）
  自动释放：事务COMMIT/ROLLBACK

第2层：乐观锁（跨事务冲突检测）
  UPDATE wallet SET balance=?, version=version+1 WHERE id=? AND version=?
  更新0行→冲突→上层重试

第3层：幂等键（重复请求过滤）
  wallet_transaction唯一键 uk_order_wallet_tx(order_no, wallet_type, tx_type)
  并发INSERT→只有一个成功

为什么不用Redis分布式锁？
  ① 锁和数据操作不在同一事务中，需额外保证一致性
  ② Redis故障影响核心资金操作
  ③ 当前规模单用户并发不高，数据库行锁够用
  ④ 未来需要时可在行锁之上加Redis前置过滤——不过度设计
```

### 6.3 三层幂等架构

```
Layer 1: 应用层检查（快速拒绝）
  查wallet_transaction是否已存在该order_no → 存在则直接返回成功

Layer 2: 数据库唯一键（最终防线）
  uk_order_wallet_tx(order_no, wallet_type, tx_type)
  并发INSERT时只有一个成功

Layer 3: 业务状态机（逻辑防线）
  提现：只有"已冻结"才允许扣款或解冻
  充值：只有"待支付/进行中"才允许上账
  已终态的订单直接拒绝
```

### 6.4 对账一致性保障

```
实时保障：
  BalanceEngine每个操作 = 同一事务内 UPDATE wallet + INSERT wallet_transaction
  流水记录 balance_before/after → balance_after == balance_before + amount

定期校验（每日凌晨对账任务）：
  ① wallet.balance == 最新流水的balance_after？
  ② 每条流水的 balance_after == balance_before + amount？
  ③ 第N条的 balance_before == 第N-1条的 balance_after？
  不一致 → 告警
```

### 6.5 汇率系统

```
四个汇率值（per币种）：
  realtime_rate    实时汇率：三方API取均值
  platform_rate    平台汇率：偏差超阈值时从实时汇率同步
  deposit_rate     入款汇率 = platform × (1 + float_percent)
  withdrawal_rate  出款汇率 = platform × (1 - float_percent)

定时同步策略（每5-10分钟）：
  1. 并发请求3家三方汇率API
  2. 容错：3→均值 / 2→均值 / 1→取值 / 全失败→跳过
  3. 偏差 = |实时-平台| / 实时 × 100%
  4. 偏差 < 阈值 → 只更新realtime
  5. 偏差 ≥ 阈值 → 更新全部四汇率 + 写rate_change_log

兑换汇率快照机制：
  预览时：读当前汇率→生成snapshot_id→存Redis(TTL 5min)
  执行时：传入snapshot_id→从Redis读快照汇率→过期则拒绝要求重新预览
```

### 6.6 缓存策略

| 数据 | 缓存 | 策略 | 理由 |
|------|------|------|------|
| 启用币种列表 | Redis | B端修改时主动清除 | C端高频读、极少变 |
| 当前汇率值 | Redis | 同步任务更新时覆盖 | 高频读 |
| 基准币种 | Redis/内存 | 永不过期 | 设一次不变 |
| 汇率快照 | Redis | TTL 5min自动过期 | 临时数据 |
| **余额** | **不缓存** | — | 变动频繁、要求强一致 |
| **交易流水** | **不缓存** | — | 索引查询已够快 |

## 七、九大技术挑战与对策

### 挑战1：余额并发安全 ★★★★★

**问题**：充值回调和提现冻结同时到达，可能丢失更新或超额扣款
**方案**：行锁+乐观锁+幂等键三层防线（见6.2节）
**业界对标**：支付宝（行锁+乐观锁）、Stripe（行锁+幂等键）
**如果被问到**："我们用的是行锁+乐观锁+幂等键三重防线，和支付宝的方案思路一致，但不用Redis分布式锁——因为锁和事务分离会增加一致性风险，当前规模数据库行锁足够。"

### 挑战2：幂等性保障 ★★★★

**问题**：三方回调重复推送、网络重试可能导致重复入账
**方案**：应用层检查+数据库唯一键+状态机三层（见6.3节）
**如果被问到**："每个RPC写操作都携带唯一订单号，wallet_transaction表的唯一键是数据库级防线——即使应用层并发穿透，INSERT也只有一个成功。"

### 挑战3：对账一致性 ★★★★

**问题**：余额和流水必须永远对得上
**方案**：同事务写入+balance_before/after+异步对账（见6.4节）
**如果被问到**："余额变更和流水写入在同一个数据库事务中，任何一步失败全部回滚。每日凌晨还有异步对账任务校验。"

### 挑战4：跨服务数据一致性 ★★★★

**问题**：钱包和财务各管各的数据，RPC调用可能一边成功一边失败
**方案**：本地事务+幂等重试（不用TCC/Saga）
**核心逻辑**：
  - 充值：财务先提交→RPC调钱包CreditDeposit（幂等）→失败则定时重试
  - 提现：钱包先冻结→通知财务审核→审核结果回调钱包（幂等）
**如果被问到**："不引入分布式事务框架，用本地事务+幂等重试保证最终一致。理由：两个服务不存在需要同时提交的场景，幂等重试足够安全。团队一个人引入TCC是过度设计。"

### 挑战5：金额精度 ★★★

**问题**：多币种多精度不能出现分/厘级误差
**方案**：bigint存最小单位、×10^8存汇率、Go int64运算（见3.3节）
**如果被问到**："全链路整数运算，展示层才按币种精度格式化。支付宝微信Stripe都是这个方案。"

### 挑战6：提现资金状态管理 ★★★★

**问题**：冻结/解冻/扣款状态机必须严格不可逆
**方案**：提现三阶段（Freeze → ConfirmDeduct/Unfreeze），状态机互斥
```
提交提现 → 立即冻结（balance减少, frozen增加）
  ├→ 审核拒绝 → Unfreeze（frozen减少, balance增加）
  └→ 出款成功 → ConfirmDeduct（frozen减少, 资金移出）
  Unfreeze和ConfirmDeduct互斥（状态机保证同一笔订单只走一条路）
```
**如果被问到**："提交即冻结，保证审核通过时一定有足够的钱。参考了p9-services的冻结三阶段模型。"

### 挑战7：汇率实时性与一致性 ★★★

**问题**：用户看到的汇率和执行的汇率必须一致
**方案**：汇率快照机制（见6.5节），预览时锁定汇率快照存Redis 5min
**如果被问到**："用户预览兑换时生成汇率快照ID，执行时用快照汇率——保证所见即所得。"

### 挑战8：敏感数据安全 ★★★

**问题**：银行卡号、USDT地址等不能泄露
**方案**：工程已有AES-256-CBC加密 + GORM EncryptedString类型
**如果被问到**："利用GORM已有的EncryptedString类型做字段级透明加解密，写入自动加密、读取自动解密，应用层无感知。"

### 挑战9：性能与可扩展性 ★★★

**问题**：流水表增长快（日增30万+），查询不能变慢
**方案**：
  - 索引覆盖：idx_user_curr_type覆盖C端记录主查询
  - TiDB天然水平扩展
  - 首版不分表，监控数据量增长趋势，按需处理
**如果被问到**："流水表有4个索引覆盖主要查询场景，TiDB分布式存储本身就比单机MySQL更适合高增长表。"

## 八、核心业务流程

### 8.1 充值流程

```
用户选择充值方式+金额(+可选奖金活动) → 提交
  │
  ▼
C-4 CreateDepositOrder（钱包服务）
  │
  ├→ 重复转账拦截：查同用户+同币种+同金额待支付/进行中订单
  │    ≤2笔→软拦截(警告) / ≥3笔→硬拦截(拒绝)
  │
  ├→ 创建钱包侧充值订单（C+16位, status=待支付）
  │
  ├→ 如果是银行/电子钱包：RPC调财务分配通道 → 返回三方跳转URL
  │   如果是USDT：返回收款地址+QR码+网络类型
  │
  └→ 返回给前端

用户完成支付 → 三方回调 → 财务服务验签
  │
  ▼
财务 RPC 调钱包 R-6 CreditDeposit（幂等）
  │
  ├→ 幂等检查：该订单已入账？→ 直接返回成功
  ├→ BalanceEngine.Add()：充值金额 → 中心钱包
  ├→ BalanceEngine.Add()：奖金金额 → 奖励钱包（如有奖金活动）
  ├→ 更新充值订单状态=已完成
  └→ 返回成功
```

### 8.2 兑换流程

```
用户输入法币金额 → 预览
  │
  ▼
C-6 GetExchangePreview
  ├→ 读取当前平台汇率
  ├→ 计算：兑换BSB = 法币 × 汇率
  ├→ 计算：赠送BSB = 兑换BSB × 10%
  ├→ 生成汇率快照存Redis(TTL 5min)
  └→ 返回预览结果 + snapshot_id

用户确认 → 执行
  │
  ▼
C-7 ExecuteExchange（同一事务内完成三步）
  ├→ 验证snapshot_id（过期→拒绝）
  ├→ BalanceEngine.Deduct()：扣法币中心钱包
  ├→ BalanceEngine.Add()：加BSB中心钱包
  ├→ BalanceEngine.Add()：加BSB奖励钱包（赠送部分，标记X倍流水要求）
  ├→ 创建兑换订单记录
  └→ 返回成功

关键：三步余额操作在同一个数据库事务中，任何一步失败全部回滚。
不需要调用财务模块，全部在钱包内部完成。
```

### 8.3 提现流程

```
用户打开提现页面
  │
  ▼
C-8 GetWithdrawalMethods
  ├→ 调ser-kyc查询KYC状态
  ├→ 未认证→仅返回USDT提现方式
  │   已认证→返回全部提现方式
  ├→ 调ser-finance获取各方式的限额+手续费
  └→ 返回方式列表+限额信息

用户选择方式+输入金额 → 提交
  │
  ▼
C-9 CreateWithdrawalOrder（钱包服务）
  │
  ├→ 校验KYC状态
  ├→ 校验限额（单笔/单日）
  ├→ 校验余额充足
  ├→ 校验"法币充→法币提"合规规则（fund_source字段）
  ├→ BalanceEngine.Freeze()：冻结提现金额（提交即冻结）
  ├→ 创建提现订单（T+16位, status=处理中）
  ├→ 通知财务进入审核流程
  └→ 返回成功

财务三级审核（风控→财务→出款）
  │
  ├→ 审核拒绝 → 财务RPC调 R-8 UnfreezeForWithdrawal（幂等）
  │              钱包解冻余额，订单状态→已拒绝
  │
  └→ 出款成功 → 财务RPC调 R-9 ConfirmWithdrawalDeduct（幂等）
                 钱包确认扣除冻结额，订单状态→已完成
```

### 8.4 人工加减款流程

```
运营在财务B端创建加款/减款申请 → 财务模块审核通过
  │
  ▼
财务RPC调钱包 R-10 ManualCredit / R-11 ManualDebit（幂等）
  │
  ├→ ManualCredit：BalanceEngine.Add()加款到指定子钱包（4种可选）
  ├→ ManualDebit：BalanceEngine.Deduct()减款（余额不足→扣实际余额到0）
  └→ 返回成功

订单号：A+16位（加款）/ M+16位（减款）
```

  → 此时充值和提现的C端页面才能渲染完整
  → 初期用mock返回值

### 9.3 联调约定（需提前对齐）

| 维度 | 约定 |
|------|------|
| 幂等键 | 所有RPC写操作携带order_no，钱包侧基于此做幂等 |
| 金额单位 | i64最小单位，不传小数，双方统一 |
| 币种编码 | 统一使用currency_config.currency_code |
| 超时策略 | RPC调用超时3s，超时后财务侧定时重试 |
| 错误码 | 钱包6022xxx段，财务6023xxx段（预分配） |
| 调用方向 | 余额操作=财务调钱包；充值/提现方式查询=钱包调财务 |

## 十二、潜在质疑与应答预案

### Q1："为什么不用分布式事务框架（TCC/Saga）？"

**A**：钱包和财务之间不存在需要同时原子提交的场景。充值链路是"财务先提交→RPC调钱包（幂等）→失败重试"，提现链路是"钱包先冻结→通知财务→审核结果回调（幂等）"。本地事务+幂等重试完全够用。引入TCC需要协调器（单点风险）、每个接口实现Try/Confirm/Cancel三套逻辑，团队一个人做是过度设计。

### Q2："余额为什么用bigint不用decimal？"

**A**：bigint存最小单位（分/聪），Go的int64运算零精度损失、性能比decimal库快一个数量级、TiDB索引更高效。支付宝、微信支付、Stripe都是整数存储方案。decimal在数据库层看起来直观，但应用层Go没有原生decimal类型，引入第三方库反而增加复杂度。

### Q3："为什么不把币种配置拆成独立服务？"

**A**：钱包每个操作都要读汇率和币种信息，如果拆成两个服务就是RPC调用（每次1-3ms网络开销），但数据耦合度极高（余额展示依赖币种展示规则）。就一个人做，运维两个服务增加复杂度。代码层按包隔离，未来需要拆时改import即可——模块化单体是微服务过度拆分反思后的主流实践。

### Q4："34个接口会不会太多/太少？"

**A**：这是11份评估方案集思广益后的整合结果。各方案评估范围从26到69不等，差异主要在：是否计入RPC调用依赖、是否计入未来预留接口、是否拆分可合并接口。34个是"我需要定义实现的接口"——C端15+B端7+RPC暴露12。规模与工程内ser-user(35方法)相当，在合理范围内。

### Q5："流水表增长快怎么办？"

**A**：日增约30万条（日活10万×3笔/人），有4个索引覆盖主查询。TiDB天然分布式存储，写入性能不是瓶颈。首版不做分表，监控增长趋势，如果年增1亿条以上可以按user_id做Range分区——但这是未来的事，不提前做。

### Q6："如果Redis挂了怎么办？"

**A**：Redis在钱包中只用于缓存（币种列表、汇率）和汇率快照。核心资金操作（余额变更、幂等）完全依赖数据库，Redis故障不影响。币种缓存失效后降级为直接查数据库（4-6条数据，零压力）。汇率快照失效→用户需重新预览兑换（体验略降但资金安全）。

### Q7："如何保证充值不会重复入账？"

**A**：三层幂等防线——应用层查order_no、数据库唯一键、状态机检查。即使三方支付重复推送10次回调，也只会入账1次。具体：第1次成功写入流水+更新余额；第2次起，应用层查到已有记录直接返回成功，即使应用层穿透也会被唯一键拦截。

### Q8："两个参考工程给了什么启发？"

**A**：
- **p9-services（法币钱包）**：采纳了5种原子SQL模式、冻结三阶段、coinFrom/coinTo流水快照、SQL级非负约束
- **tg-services（TRC20充值）**：采纳了两层架构（链上+业务）、双触发确认（手动+定时）、多地址分离、Gas费自动管理

两个互补——A教会余额怎么做安全，B教会链上怎么做完整。

### Q9："你对这个模块的整体评估是什么？"

**A**：
- **复杂度**：中高（金融系统对一致性和安全的要求远高于普通CRUD）
- **风险点**：跨服务一致性（与财务联调）、USDT方案待定、奖金活动归属未明
- **信心度**：高（核心设计方案成熟，有业界成熟实践和两个参考工程佐证，技术栈全部复用已有）
- **工期评估**：4期约7-8周（纯后端，不含联调等待时间）
- **最大阻塞**：P0-1~P0-4这4个问题如果评审不澄清，充值和提现链路无法启动
