## 一句话定位

ser-wallet 是平台资金体系的核心枢纽 —— 管余额、管流水、管币种、管汇率。充值/提现/兑换/人工调账，所有资金变动最终都落到这里。它不直接对接三方支付通道（那是财务模块的活），但任何一笔钱进出平台，都必须经过 ser-wallet 的 CreditWallet/DebitWallet 才能改余额。

```
              ┌─ ser-finance（充值/提现/补单/加减款 → 审核通过后调我）
              │
用户 → 网关 → ser-wallet ← ser-user（注册后调 InitWallet）
              │
              └─ ser-live / ser-game / ser-activity（消费/奖励 → 调我加减款）
```

## 第一章 设计思路：我们怎么想的

### 1.1 整体架构定位

钱包不是一个孤立模块，它是整个资金体系的"账本层"。我们的设计原则：

**原则一：钱包只管余额，不管通道**
- 三方支付通道的接入、轮询、密钥管理 → 财务模块
- 钱包只做：收到"充值成功"通知后执行 CreditWallet 上账
- 好处：职责清晰，通道出问题不影响余额安全

**原则二：所有余额变动必须有流水**
- wallet_flow 表 append-only，每一笔余额变动都有 before/after 快照
- 幂等键（idempotent_key）唯一索引兜底，重复请求不会重复变动余额
- 好处：可审计、可对账、可追溯

**原则三：冻结是三阶段，不是两阶段**
- 提现不是直接扣余额，而是：冻结 → 审核出款 → 确认扣/解冻
- FreezeBalance → ConfirmDebit（成功）或 UnfreezeBalance（失败/驳回）
- 好处：提现审核期间用户看到"余额减少+冻结金额增加"，任何异常都能回滚

**原则四：币种维度隔离**
- 每个用户 × 每个币种 × 每个钱包类型 = 一行 wallet 记录
- 10万用户 × 5币种 × 2钱包类型 = 100万行（完全可控）
- 不存在跨币种混合余额，兑换是独立的"扣A加B"原子操作

**原则五：遵循工程一致性，不发明新模式**
- 单 WalletService（与 ser-live、ser-app 一致），不拆多 service
- GORM Gen 自动生成 model/query（与现有所有模块一致）
- handler → service → repo 三层分离（与现有模块一致）
- 错误码前缀 6013xxx（按编号 013 分配）

### 1.2 钱包结构全景

```
用户张三的钱包世界：

├── VND（越南盾）
│   ├── 中心钱包  balance=5,000,000  frozen=0        ← 充值到这里，消费优先扣这里
│   └── 奖励钱包  balance=1,000,000  frozen=0        ← 活动赠送到这里，消费其次扣这里
│
├── IDR（印尼盾）
│   ├── 中心钱包  balance=200,000    frozen=50,000   ← 有一笔提现在审核中
│   └── 奖励钱包  balance=30,000     frozen=0
│
├── THB（泰铢）
│   ├── 中心钱包  balance=0          frozen=0
│   └── 奖励钱包  balance=0          frozen=0
│
├── BSB（平台币，10 BSB = 1 USDT）
│   ├── 中心钱包  balance=500        frozen=0        ← 兑换法币→BSB，到这里
│   └── 奖励钱包  balance=100        frozen=0        ← 兑换赠送到这里
│
└── USDT
    ├── 中心钱包  balance=0          frozen=0
    └── 奖励钱包  balance=0          frozen=0
```

**关键概念**：
- **资金钱包** = 中心余额 + 奖励余额（对C端展示的"总余额"）
- **可用余额** = balance - frozen_amount（代码计算，不建列）
- **消费优先级**：先扣中心，中心不够再扣奖励
- **冻结分配**：冻结也遵循先中心后奖励（freeze_detail 表记录每个钱包冻了多少）
- **主播钱包、代理钱包、场馆钱包**：预留类型，Phase 1 不实现

### 1.3 核心业务链路

#### 链路1：充值（资金进入平台）
```
用户选充值方式 → 选金额 → 选奖金
    ↓
ser-wallet: createRechargeOrder
    ├── RPC → ser-finance.MatchChannelAndCreateOrder（匹配通道，获取支付URL）
    ├── 写 recharge_order（status=待支付）
    └── 返回支付URL给前端
    ↓
用户在三方页面支付
    ↓
三方回调 → ser-wallet: rechargeCallback（open接口，无需登录）
    ├── 验签 + 校验订单
    ├── 幂等检查（idempotent_key 唯一约束）
    ├── CreditWallet → 中心钱包 +金额 + 写 wallet_flow
    ├── 奖金 > 0 → CreditWallet → 奖励钱包 +奖金 + 写 wallet_flow
    └── 更新 recharge_order（status=成功）
```

#### 链路2：提现（资金离开平台）
```
用户选提现方式 → 输金额 → 选提现账户
    ↓
ser-wallet: createWithdrawOrder
    ├── RPC → ser-kyc.KycQuery（KYC校验，未认证仅可USDT）
    ├── RPC → ser-finance.GetWithdrawConfig（限额/费率校验）
    ├── FreezeBalance → 冻结（先中心后奖励，写 freeze_detail）
    ├── 写 withdraw_order（status=待风控）
    └── 通知财务模块有新提现单
    ↓
财务审核（风控→财务→出款）
    ├── 驳回/失败 → RPC ser-wallet.UnfreezeBalance → 解冻 + 更新 freeze_detail
    └── 出款成功 → RPC ser-wallet.ConfirmDebit → 正式扣余额 + 更新 freeze_detail
```

#### 链路3：兑换（平台内部币种转换）
```
用户查看兑换信息（汇率、可兑余额、赠送比例）
    ↓
ser-wallet: doExchange（事务内原子操作）
    ├── DebitWallet → 法币中心钱包 -X VND + 写 wallet_flow
    ├── CreditWallet → BSB中心钱包 +Y BSB + 写 wallet_flow
    ├── bonus > 0 → CreditWallet → BSB奖励钱包 +Z BSB + 写 wallet_flow
    └── 写 exchange_record
```

#### 链路4：钱包初始化（用户注册触发）
```
用户注册成功 → ser-user RPC调用 → ser-wallet.InitWallet
    ├── 查询所有启用币种（GetCurrencyList）
    └── 为每个币种创建 2 条 wallet 记录（中心+奖励），balance=0
```

### 1.4 与财务模块的边界

这个边界必须讲清楚，因为很多人会混淆：

| 职责 | ser-wallet（我们） | ser-finance（他们） |
|------|-------------------|-------------------|
| 余额存储 | wallet 表 | 不存余额 |
| 余额变动 | CreditWallet/DebitWallet | 审核通过后调我们的 RPC |
| 充值方式展示 | C端接口入口 | 提供方式列表数据 |
| 通道匹配 | 不涉及 | 根据权重/成功率匹配最优通道 |
| 三方密钥 | 不涉及 | 存储管理 |
| 充值回调 | 接收回调+上账 | 不涉及（或回调直接到我们） |
| 提现审核 | 创建订单+冻结 | 风控审核+财务审核+发起出款 |
| 人工补单/加减款 | 提供 CreditWallet/DebitWallet RPC | 审核流程+调我们上/下账 |
| 汇率管理 | 币种配置+汇率拉取+汇率日志 | 使用汇率（RPC查我们） |
| 币种配置 | B端CRUD全生命周期 | 使用配置（RPC查我们） |

**一句话总结**：财务管"钱从哪来到哪去"，钱包管"账上有多少钱"。

## 第二章 表结构：数据怎么存

### 2.1 全景总览

```
我方负责：14张表（必须10 + 应该2 + 可选2）
他方负责：11张表（必须7 + 应该2 + 可选2）
总计：25张表
共用数据库：db_wallet（我方）、db_finance（他方）
```

| 分级 | 钱包模块 | 币种配置 | 财务模块 | 合计 |
|------|:-------:|:-------:|:-------:|:----:|
| 必须 | 7 | 3 | 7 | 17 |
| 应该 | 2 | 0 | 2 | 4 |
| 可选 | 1 | 1 | 2 | 4 |
| 合计 | 10 | 4 | 11 | 25 |

### 2.2 钱包核心表（3张，必须）

#### wallet — 用户钱包（余额载体）

一个用户一个币种一个钱包类型 = 一行。所有资金操作的最终落点。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK | 自增主键 |
| user_id | BIGINT NOT NULL | 用户ID |
| currency_code | VARCHAR(10) NOT NULL | 币种代码 BSB/VND/IDR/THB/USDT |
| wallet_type | TINYINT NOT NULL | 1中心/2奖励/3主播/4代理/5场馆 |
| balance | BIGINT NOT NULL DEFAULT 0 | 余额（最小单位整数） |
| frozen_amount | BIGINT NOT NULL DEFAULT 0 | 冻结金额 |
| status | TINYINT NOT NULL DEFAULT 1 | 1正常/2冻结/3关闭 |
| create_at | BIGINT NOT NULL | 毫秒时间戳 |
| update_at | BIGINT NOT NULL | 毫秒时间戳 |
| delete_at | BIGINT NOT NULL DEFAULT 0 | 0=未删除 |

核心约束：
- UNIQUE (user_id, currency_code, wallet_type, delete_at)
- balance >= 0, frozen_amount >= 0（代码层强校验）
- 可用余额 = balance - frozen_amount（代码计算，不建列）

#### wallet_flow — 资金流水（审计基石）

append-only，每笔余额变动都记录。是幂等性、对账、审计的最终防线。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK | 自增主键 |
| user_id | BIGINT NOT NULL | 用户ID |
| currency_code | VARCHAR(10) NOT NULL | 币种 |
| wallet_type | TINYINT NOT NULL | 钱包类型 |
| direction | TINYINT NOT NULL | 1入账/2出账/3冻结/4解冻 |
| biz_type | TINYINT NOT NULL | 1充值/2提现/3兑换/4奖金/5人工加/6人工减/7补单/8冻结/9解冻/10确认扣 |
| amount | BIGINT NOT NULL | 变动金额（正数） |
| before_balance | BIGINT NOT NULL | 变动前余额 |
| after_balance | BIGINT NOT NULL | 变动后余额 |
| ref_order_no | VARCHAR(64) NOT NULL DEFAULT '' | 关联业务订单号 |
| idempotent_key | VARCHAR(128) NOT NULL | 幂等键 格式：{biz_type}:{ref_order_no}:{wallet_type} |
| remark | VARCHAR(256) NOT NULL DEFAULT '' | 备注 |
| create_at | BIGINT NOT NULL | 毫秒时间戳 |
| update_at | BIGINT NOT NULL | 毫秒时间戳 |
| delete_at | BIGINT NOT NULL DEFAULT 0 | 0=未删除 |

核心约束：
- UNIQUE (idempotent_key, delete_at) — 幂等最终防线
- INDEX (user_id, currency_code, create_at) — C端记录查询
- INDEX (ref_order_no) — 按订单号查流水
- **绝对规则：只追加、不更新、不删除**

#### freeze_detail — 冻结明细

跨钱包冻结的精确记录。提现冻结可能横跨中心+奖励两个钱包，解冻时必须精确还原。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK | 自增主键 |
| user_id | BIGINT NOT NULL | 用户ID |
| currency_code | VARCHAR(10) NOT NULL | 币种 |
| wallet_type | TINYINT NOT NULL | 钱包类型 |
| order_no | VARCHAR(64) NOT NULL | 关联提现订单号 |
| freeze_amount | BIGINT NOT NULL | 该钱包冻结金额 |
| status | TINYINT NOT NULL DEFAULT 1 | 1冻结中/2已确认扣款/3已解冻 |
| create_at / update_at / delete_at | BIGINT | 标准字段 |

一笔提现产生 1-2 条记录（中心一条 + 奖励一条，若金额全在中心则只有一条）。

### 2.3 钱包业务表（4张，必须）

#### recharge_order — 充值订单

19个字段，覆盖充值全生命周期：订单金额、通道信息、奖金信息、状态流转。

关键字段（完整字段见表结构设计/d-2.md）：
- order_no: 唯一订单号（R+时间戳+随机数）
- order_amount / actual_amount / fee_amount: 金额三件套
- exchange_rate: 入款汇率（×10^8，仅USDT→BSB有值）
- bonus_type / bonus_amount / bonus_turnover: 奖金三件套
- status: 1待支付/2成功/3失败/4取消

#### withdraw_order — 提现订单

24个字段，7种状态的完整状态机：

```
1.待风控 ──→ 2.风控驳回（终态，解冻）
         ──→ 3.待财务 ──→ 4.财务驳回（终态，解冻）
                      ──→ 5.待出款 ──→ 6.出款失败（终态，解冻）
                                   ──→ 7.出款成功（终态，确认扣）
```

关键字段：account_info（JSON快照，防账户修改后信息丢失）、risk_reviewer_id/fin_reviewer_id（双重审核人）

#### exchange_record — 兑换记录

记录法币→BSB的每笔兑换。关键字段：from_currency/to_currency、exchange_rate、bonus_rate/bonus_amount（兑换赠送）

#### withdraw_account — 提现账户

用户绑定的银行卡/电子钱包/USDT地址。**account_no 和 account_name 加密存储**（AES，与现有 user_kyc 表一致）。

### 2.4 币种配置表（3张，必须）

#### currency_config — 币种配置

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK | 主键 |
| currency_code | VARCHAR(10) UNIQUE | 币种代码 |
| currency_name | VARCHAR(50) | 币种名称 |
| currency_type | TINYINT | 1法币/2加密/3平台 |
| country | VARCHAR(50) | 国籍/地区 |
| icon_url | VARCHAR(256) | 图标URL |
| thousands_sep | CHAR(1) DEFAULT ',' | 千分位符号 |
| decimal_sep | CHAR(1) DEFAULT '.' | 小数点符号 |
| decimal_digits | TINYINT DEFAULT 2 | 小数位数 |
| precision_factor | BIGINT DEFAULT 100 | 精度系数 10^decimal_digits |
| real_rate | BIGINT DEFAULT 0 | 实时汇率（×10^8） |
| platform_rate | BIGINT DEFAULT 0 | 平台汇率（×10^8） |
| deposit_rate | BIGINT DEFAULT 0 | 入款汇率 = 实时×(1+浮动%) |
| withdraw_rate | BIGINT DEFAULT 0 | 出款汇率 = 实时×(1-浮动%) |
| rate_float_pct | INT DEFAULT 0 | 汇率浮动%（×100，500=5%） |
| deviation_pct | INT DEFAULT 0 | 偏差阈值%（×100） |
| sort_order | INT DEFAULT 0 | 排序 |
| status | TINYINT DEFAULT 0 | 0禁用/1启用 |
| create_at / update_at / delete_at | BIGINT | 标准字段 |

汇率四元组解释：
- **real_rate**：从三方API拉取的真实汇率
- **platform_rate**：平台给用户看的汇率（当偏差 >= 阈值时自动同步到 real_rate）
- **deposit_rate**：充值用的汇率 = real_rate × (1 + rate_float_pct/10000)（平台赚差价）
- **withdraw_rate**：提现用的汇率 = real_rate × (1 - rate_float_pct/10000)（平台赚差价）

#### rate_log — 汇率变动日志

记录每次汇率变动的 before/after 快照 + 触发原因（1定时/2手动/3偏差触发/4编辑浮动）+ 操作人。

#### base_currency_config — 基准币种配置

全局唯一（永远只有1行数据）。设置后不可更改。推荐 USDT。BSB 锚定基准币种（10 BSB = 1 USDT）。

### 2.5 钱包补充表（2张，应该有）

| 表名 | 用途 | 不做的后果 |
|------|------|-----------|
| withdraw_limit_config | 按币种配置提现限制（日次数/日限额/单笔范围/大额阈值） | 限制参数硬编码，改限制要发版 |
| user_currency_preference | 记录用户当前选择的币种 | 用 Redis 替代，有过期丢失风险 |

### 2.6 可选增强表（2张）

| 表名 | 用途 | 建议 |
|------|------|------|
| wallet_daily_snapshot | 每日余额快照，对账校验用 | Phase 4 补建，初期靠 wallet_flow 审计 |
| rate_source_log | 三方汇率API原始返回数据 | 遇到汇率异常时再建不迟 |

### 2.7 字段设计规范（全局统一标准）

```
金额：BIGINT，最小单位整数（VND/IDR 1:1，THB/BSB ×100，USDT ×10^8）
汇率：BIGINT，×10^8 转整数（1 BSB = 2300 VND → 存 230000000000）
百分比：INT，×100（5.00% → 存 500）
费率：INT，×10000（0.50% → 存 50）
时间：BIGINT，毫秒时间戳 UTC
主键：BIGINT 自增
软删除：delete_at BIGINT DEFAULT 0（0=未删除）
```

订单号生成规则：
| 业务 | 前缀 | 格式 |
|------|------|------|
| 充值 | R | R + 时间戳 + 随机数 |
| 提现 | W | W + 时间戳 + 随机数 |
| 兑换 | E | E + 时间戳 + 随机数 |

## 第三章 接口全景：对外暴露什么能力

### 3.1 总量概览

| 分类 | 必须 | 应有 | 商榷 | 小计 |
|------|:----:|:----:|:----:|:----:|
| C端 HTTP 接口 | 13 | 5 | 2 | **20** |
| B端 HTTP 接口 | 7 | 2 | 2 | **11** |
| RPC 暴露接口 | 9 | 2 | 1 | **12** |
| RPC 调用（已有模块） | 2 | 3 | 1 | **6** |
| RPC 调用（未建模块） | 4 | 0 | 2 | **6** |
| 定时任务 | 1 | 0 | 0 | **1** |
| **合计** | **36** | **12** | **8** | **56** |

**核心数字**：我方开发 44 个接口，调用外部 12 个接口，其中 36 个必须交付。

路由规范（遵循现有工程约定）：
```
C端：POST /api/wallet/xxx（登录态） | POST /open/wallet/xxx（无需登录）
B端：POST /admin/api/wallet/xxx（登录态）
```
全部 POST 方法，camelCase 命名，请求体传参。

### 3.2 C端 HTTP 接口（20个）

**钱包核心（2个）**

### 4.2 关键技术选型说明

**为什么用 BIGINT 存金额不用 DECIMAL？**
- TiDB 的 DECIMAL 在高并发 UPDATE 场景有锁竞争问题
- BIGINT 运算比 DECIMAL 快 5-10 倍
- 参考工程 A（Java 法币钱包）和工程 B（TRC20 钱包）都用整数存金额
- 精度零损失：VND/IDR 1:1，THB/BSB ×100，USDT ×10^8
- 代码层提供 amountToDisplay(amount, precision_factor) 工具函数做展示转换

**为什么汇率用 ×10^8？**
- 覆盖所有精度场景（USDT/BTC 对各法币都可能有 8 位小数）
- 10^8 = 1亿，BIGINT 最大约 9.2×10^18，汇率值最大约 9.2×10^10 = 920亿，够用
- 计算时：result = amount × rate / 10^8（整数除法，最后根据目标精度取整）

**为什么用悲观锁而不是乐观锁？**
- 钱包余额更新是 读→计算→写 三步操作，高并发下乐观锁重试代价高
- 悲观锁方案：SELECT ... FOR UPDATE（锁单行 wallet 记录）→ 校验余额 → UPDATE balance
- TiDB 对行锁支持好，单用户并发操作不多（不是秒杀场景）
- 参考工程 A 用 Redisson 分布式锁，参考工程 B 用数据库悲观锁 — 我们选数据库悲观锁，简单直接

**为什么不用分布式事务框架（Seata/DTM）？**
- 钱包的跨模块交互都是"调RPC通知，各管各的事务"
- 充值：回调幂等 + 订单状态兜底，不需要分布式事务
- 提现：冻结三阶段已经解耦了事务边界
- 兑换：纯本地事务（扣A加B在同一个数据库内）
- 引入 Seata 的运维成本远大于收益

## 第五章 技术挑战与攻坚方案

### 5.1 挑战一：资金操作幂等性（最高优先级）

**问题本质**：充值回调、RPC调用可能因网络抖动重复到达。重复执行 CreditWallet 会导致用户余额凭空增加。

**攻坚方案**：
```
幂等三重保障：
1. 业务层：idempotent_key = {biz_type}:{ref_order_no}:{wallet_type}
2. 数据库层：wallet_flow 表 UNIQUE(idempotent_key, delete_at)
3. 操作层：在事务内先 INSERT wallet_flow → 再 UPDATE wallet.balance
   如果 INSERT 因唯一约束失败 → 返回"已处理"，不执行 UPDATE
```

**代码层伪逻辑**：
```go
func CreditWallet(ctx, req) (resp, error) {
    key := fmt.Sprintf("%d:%s:%d", req.BizType, req.RefOrderNo, req.WalletType)

    return db.Transaction(func(tx *gorm.DB) error {
        // 1. 悲观锁读钱包
        wallet := tx.Where("user_id=? AND currency_code=? AND wallet_type=?", ...).
            Set("gorm:query_option", "FOR UPDATE").First()

        // 2. 尝试插入流水（幂等检查）
        flow := &WalletFlow{
            IdempotentKey: key,
            BeforeBalance: wallet.Balance,
            AfterBalance:  wallet.Balance + req.Amount,
            ...
        }
        err := tx.Create(flow).Error
        if isDuplicateKeyError(err) {
            return nil // 已处理，幂等返回
        }

        // 3. 更新余额
        return tx.Model(wallet).Update("balance", wallet.Balance + req.Amount).Error
    })
}
```

### 5.2 挑战二：冻结跨钱包一致性

**问题本质**：提现冻结100元，中心钱包只有80元，需要从奖励钱包冻结20元。解冻时必须精确还原。

**攻坚方案**：
```
冻结操作（FreezeBalance）：
1. FOR UPDATE 锁中心钱包 → 可用余额 = balance - frozen_amount
2. 如果中心可用 >= 冻结金额 → 全部冻中心
3. 如果中心可用 < 冻结金额 → 冻中心可用部分 + 剩余冻奖励
4. 写 freeze_detail（1-2条，记录每个钱包冻了多少）
5. 更新 wallet.frozen_amount（两个钱包分别 +冻结金额）
全在一个事务内。

解冻操作（UnfreezeBalance）：
1. 查 freeze_detail（按 order_no），获取每个钱包冻了多少
2. FOR UPDATE 锁对应钱包
3. 各自 frozen_amount -= freeze_amount
4. 更新 freeze_detail.status = 3（已解冻）

确认扣款（ConfirmDebit）：
1. 查 freeze_detail（按 order_no）
2. FOR UPDATE 锁对应钱包
3. 各自 balance -= freeze_amount, frozen_amount -= freeze_amount
4. 更新 freeze_detail.status = 2（已确认扣款）
5. 写 wallet_flow（记录正式扣款）
```

### 5.3 挑战三：兑换原子性

**问题本质**：法币→BSB 兑换涉及两个钱包（扣法币 + 加BSB），必须要么都成功要么都失败。

**攻坚方案**：
```
关键：两个钱包在同一个 db_wallet 数据库中，用本地事务即可保证原子性。

事务内操作：
1. 锁定汇率（从缓存或DB读取当前兑换汇率）
2. FOR UPDATE 锁法币中心钱包 → 校验可用余额
3. 扣法币：balance -= fromAmount
4. 计算BSB金额：toAmount = fromAmount × rate / 10^8
5. FOR UPDATE 锁BSB中心钱包 → balance += toAmount
6. 如果有赠送：FOR UPDATE 锁BSB奖励钱包 → balance += bonusAmount
7. 写 3 条 wallet_flow（扣法币 + 加BSB中心 + 加BSB奖励）
8. 写 1 条 exchange_record
全在一个事务内，要么全成功要么全回滚。
```

不需要分布式事务框架 — 这是纯本地事务。

### 5.4 挑战四：汇率实时性与容错

**问题本质**：三方汇率API可能超时、返回异常数据、或完全不可用。

**攻坚方案**：
```
拉取策略（每3分钟执行一次）：
1. 并发调3个三方API（goroutine + context.WithTimeout(5s)）
2. 收集结果，排除失败的和异常值（偏离均值>30%的）
3. 剩余≥2个 → 取平均值 = new_real_rate
4. 剩余=1个 → 保守策略：不更新平台汇率，仅更新 real_rate
5. 全部失败 → 不做任何更新，记日志告警
6. abs(new_real_rate - platform_rate) / platform_rate >= deviation_pct
   → 自动更新 platform_rate + deposit_rate + withdraw_rate
   → 写 rate_log（trigger_type=1定时或3偏差触发）
7. 否则 → 只更新 real_rate，platform_rate 不变

容错设计：
- 3个API独立超时（5秒），互不阻塞
- 至少2个有效结果才更新（防止单个API被篡改）
- 偏差阈值可配置（B端 deviation_pct 字段）
- 手动刷新接口（B端 refreshRate）作为兜底
```

### 5.5 挑战五：并发余额更新

**问题本质**：同一个用户可能同时有充值回调、兑换、游戏扣款等操作并发修改余额。

**攻坚方案**：
```
策略：数据库行级悲观锁（SELECT ... FOR UPDATE）

为什么不用 Redis 分布式锁？
1. 钱包余额操作必须和 wallet_flow 写入在同一个事务内
2. 用 Redis 锁+DB事务 = 两个锁源，增加死锁风险和复杂度
3. DB FOR UPDATE 已经解决并发问题，加 Redis 锁是多此一举

为什么不用乐观锁（version 字段）？
1. 充值高峰期同一用户可能有多笔充值同时回调
2. 乐观锁冲突后需要重试，重试又冲突 → 雪崩
3. 悲观锁排队等待，延迟稳定可预期

性能评估：
- 单用户并发操作不高（不是秒杀），锁等待时间 < 10ms
- 不同用户操作互不影响（锁的是不同行）
- TiDB 行锁实现成熟，无需担心表锁升级
```

### 5.6 挑战六：提现账户敏感信息

**问题本质**：银行卡号、持有人姓名是敏感信息，明文存储违规。

**攻坚方案**：
```
方案：AES 加密存储（复用现有 utils.EncryptedString 方案）

加密位置：account_no、account_name 两个字段
加密密钥：ETCD 配置 /slg/conf/security/sensitive_data_key + IV
加解密时机：
  - 写入DB前：自动加密（GORM Hook BeforeCreate/BeforeUpdate）
  - 读取DB后：自动解密（GORM Hook AfterFind）
  - 或：手动在 service 层调用 encrypt/decrypt

参照：现有 ser-kyc 的 user_kyc 表已经用同样方案加密身份证号等字段
```

### 5.7 挑战七：充值回调安全性

**问题本质**：充值回调是 open 接口（/open/wallet/rechargeCallback），任何人都可以调用。必须防止伪造回调。

**攻坚方案**：
```
多重校验：
1. 验签：根据通道配置的公钥/密钥验证回调签名（由 ser-finance 提供验签规则）
2. 订单校验：回调中的订单号必须在 recharge_order 表中存在且 status=待支付
3. 金额校验：回调金额与订单金额比对（允许小范围误差）
4. IP白名单：仅允许三方通道的出口IP（在网关层配置，或在 ser-wallet 层校验）
5. 幂等兜底：即使前4步都被绕过，idempotent_key 唯一约束也能防止重复上账
```

## 第九章 高频质疑预判与应答

### Q1: "为什么钱包和币种配置放在一个服务里，不拆两个服务？"

**答**：
1. **工程一致性**：ser-live 包含房间/观众/消费/审核/设置等 7 个子模块，全在一个 LiveService 中。ser-wallet 的 7 个子模块（钱包核心/充值/提现/兑换/记录/账户/币种配置）规模相当。
2. **数据耦合**：币种配置是钱包每个操作的前置依赖（查汇率、查精度、查展示规则），拆成两个服务意味着每次操作多一跳 RPC，增加延迟和故障点。
3. **运维成本**：多一个服务 = 多一套部署、监控、日志、ETCD注册，当前团队规模不支撑。
4. **弹性预留**：如果未来币种配置的读取频率极高需要独立扩容，可以通过 Redis 缓存解决，不需要拆服务。

### Q2: "金额为什么不用 DECIMAL，用 BIGINT 不怕精度问题？"

**答**：
1. **不存在精度问题**：所有金额以最小单位整数存储。VND 1元 = 存1，THB 1.50元 = 存150，USDT 1.00000001 = 存100000001。加减乘都是整数运算，精度零损失。
2. **性能优势**：BIGINT 的加减运算比 DECIMAL 快 5-10 倍，在高频余额更新场景下有实际收益。
3. **行业实践**：参考工程 A（Java法币钱包）和参考工程 B（TRC20钱包）都用整数存金额。Stripe、支付宝的最小单位方案也是如此。
4. **展示转换**：代码层提供 amountToDisplay(amount, precision_factor) 统一转换，在 API 响应和前端展示时才做格式化。

### Q3: "提现为什么不直接扣余额，要搞冻结三阶段？"

**答**：
1. **用户体验**：提现提交后用户立刻看到"余额减少 + 冻结增加"，知道钱在处理中。如果直接扣余额，出款失败后退款的交互体验差。
2. **资金安全**：冻结期间余额不可用于其他消费，防止超支。
3. **审核流程**：提现需要风控+财务双重审核，审核期间资金状态必须明确（不是"扣了"也不是"没扣"，而是"冻着"）。
4. **回滚简单**：任何环节失败，UnfreezeBalance 直接解冻。比"退款"逻辑简单得多。
5. **审计清晰**：freeze_detail 表完整记录了冻结→解冻/确认的全过程。

### Q4: "汇率更新频率3分钟会不会太频繁/太慢？"

**答**：
- 3分钟是初始值，可配置（在定时任务配置中调整）
- 太频繁的顾虑：3分钟 × 5币种 × 3个API = 每3分钟15次HTTP请求，量级很小
- 太慢的顾虑：偏差阈值机制保证了重大波动时实时响应（偏差 >= 阈值就更新），3分钟只是轮询兜底
- B端还有手动刷新接口（refreshRate）作为最终兜底

### Q5: "Phase 1 做完能跑什么业务？"

**答**：
Phase 1 做完后，虽然C端用户还不能充值/提现，但：
1. **用户注册**可以正常创建钱包（ser-user 调 InitWallet）
2. **运营可以**在B端配置币种、设置基准币种、查看汇率日志
3. **财务模块可以**调用 CreditWallet/DebitWallet 做人工加减款
4. **活动模块可以**调用 CreditWallet 发奖金
5. **前端可以**调用 GetBalance 展示余额（余额=0，但链路通了）

也就是说，Phase 1 解锁了其他模块对钱包的**依赖链**，让其他模块可以并行开发。

### Q6: "如果评审时产品说要加新币种怎么办？"

**答**：
币种是配置化的，不是硬编码的。加新币种只需要：
1. B端 createCurrency → 填写新币种配置（代码/名称/精度/汇率API等）
2. 定时任务自动开始拉取新币种汇率
3. 所有已注册用户通过"懒初始化"或"批量补建"创建新币种的子钱包
无需改代码，无需发版。

### Q7: "56个接口，你一个人写得完吗？"

**答**：
56 个是总量，但实际开发量没那么大：
- RPC调用的12个是**调别人的**，我们只是写客户端调用代码
- B端9个币种配置接口是标准CRUD，模式化开发
- C端记录查询2个是纯DB分页查询
- RPC暴露的12个虽然是核心，但模式统一（悲观锁→校验→写流水→更新余额）
- 真正复杂的是充值/提现的6个C端接口（跨模块交互多）

按 Phase 分批交付，每个 Phase 聚焦一类接口，不是一次全写完。

### Q8: "钱包模块的数据量会不会很大？TiDB 撑得住吗？"

**答**：
- wallet 表：10万用户 × 5币种 × 2类型 = 100万行，稳定
- wallet_flow 表：增长最快，按月增 100-500万行估算，年增 5000万行
- TiDB 单表亿级无压力（分布式存储，自动分片）
- wallet_flow 按 create_at 做范围分区（或按月归档），保持热数据在最近3个月
- 实在不放心，Phase 4 补建 wallet_daily_snapshot 做每日对账

### Q9: "如果财务模块迟迟不开发，我们怎么办？"

**答**：
1. Phase 1 和 Phase 2 完全不依赖财务模块，可以先做完
2. Phase 3 用 Mock 桩接口替代，先把我方逻辑写完并自测
3. Mock 的数据结构提前与财务对齐，确保联调时无缝替换
4. 最坏情况：Phase 3 的充值/提现功能推迟上线，但钱包信息、兑换、记录查询、B端币种配置、RPC资金操作 —— 全部可用

### Q10: "你怎么保证余额不会出现负数？"

**答**：
三层保障：
1. **代码层**：DebitWallet/FreezeBalance 操作前先读可用余额，不足则返回错误拒绝操作
2. **SQL层**：UPDATE wallet SET balance = balance - ? WHERE id = ? AND balance >= ?（WHERE 条件兜底）
3. **事务层**：FOR UPDATE 悲观锁防止并发读到过期余额后同时扣款
三层任何一层拦住，余额就不会为负。
