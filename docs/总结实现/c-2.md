## 一、一句话概括

**ser-wallet 是一个多币种多子钱包的资金账户引擎**，核心职责是"管住钱、记清账"——通过 5 种原子操作（加款、扣款、冻结、解冻、兑换）对用户余额做精确变更，每一笔变更都产生不可篡改的流水记录，确保资金安全、数据一致、全程可审计。

模块覆盖 3 大业务域：
- **币种配置**（B端）— 管理币种元数据 + 四层汇率体系
- **多币种钱包**（C端 + RPC）— 充值 / 提现 / 兑换 / 余额管理
- **与财务模块的接口契约**（RPC暴露）— 作为资金操作的唯一入口供外部调用

## 二、设计思路与整体架构

### 2.1 设计哲学

| 原则 | 具体体现 |
|------|---------|
| **余额账户制** | 每个用户 × 每种币种 × 每种子钱包 = 一行账户记录，不搞流水汇总算余额 |
| **DB 即真相** | 所有余额以数据库为准，Redis 仅做读缓存，不参与余额计算 |
| **悲观锁保资金** | 涉及余额变更一律 SELECT FOR UPDATE 行锁，不用乐观锁（资金场景不允许重试失败） |
| **流水只增不删** | wallet_flow 表是 append-only 的审计账本，任何余额变更都对应一条流水 |
| **幂等靠 DB** | 流水表建 (order_no, wallet_type, flow_type) 唯一索引，重复请求直接被 DB 拦截 |
| **本地事务够用** | 钱包内部操作全部在单库事务内完成，跨模块靠幂等 RPC + 重试 + 日对账，不引入分布式事务框架 |
| **金额用整数** | BIGINT 存储最小货币单位（如 VND 的"盾"），计算用 shopspring/decimal，杜绝浮点精度问题 |

### 2.2 四层代码架构

```
请求方向:  gate-font / gate-back  →  ser-wallet
                                       │
                                  ┌─────┴─────┐
                                  │ handler.go │  ← 薄委托层，只做参数透传
                                  └─────┬─────┘
                                        │
                                  ┌─────┴─────┐
                                  │ ser 层     │  ← 业务编排：校验、组装、调外部 RPC、调 ledger
                                  │ 7个文件    │     (wallet_ser / deposit_ser / withdraw_ser /
                                  └─────┬─────┘      exchange_ser / record_ser / currency_ser / rate_ser)
                                        │
                              ┌─────────┴─────────┐
                              │ ledger 层（★新增） │  ← 钱包独有，余额变更唯一入口
                              │ 5 个原子操作       │     Credit / Debit / Freeze / Unfreeze / Exchange
                              └─────────┬─────────┘     每个操作 = 行锁 + 余额变更 + 写流水（单事务）
                                        │
                              ┌─────────┴─────────┐
                              │ repo + gorm_gen   │  ← 数据访问层
                              │ 手写 rep + 自动生成 │     手写 rep 覆盖行锁查询、聚合统计等 gorm_gen 不覆盖的场景
                              └───────────────────┘
```

**ledger 层**是与其他服务的关键区别：
- ser 层**禁止**直接 UPDATE user_wallet 表
- 所有余额变更必须经过 ledger 层的 5 个原子函数
- 每个原子函数内部：开事务 → SELECT FOR UPDATE 锁行 → 校验余额 → UPDATE 余额 → INSERT 流水 → 提交
- 保证"锁、改、记"三步原子完成

### 2.3 五种子钱包

| 子钱包 | wallet_type | 用途 | 资金来源 |
|--------|-------------|------|---------|
| 中心钱包 | 1 | 充值/消费的主账户 | 充值入款、兑换获得 BSB |
| 奖励钱包 | 2 | 活动奖金、兑换赠送 | 充值奖金、兑换 10% 赠送 |
| 主播钱包 | 3 | 直播收入 | 直播打赏分成 |
| 代理钱包 | 4 | 代理佣金（预留） | 代理返佣 |
| 场馆钱包 | 5 | 真人视讯（预留） | — |

**C端展示口径**：资金钱包 = 中心 + 奖励（合并展示），主播钱包独立展示，代理钱包独立展示。

### 2.4 币种体系

| 币种代码 | 类型 | 说明 |
|---------|------|------|
| BSB | 平台币 | 投注/消费的唯一计价单位，10 BSB = 1 USDT |
| USDT | 加密货币 | 基准币种，链上充值（TRC20/ERC20/BEP20） |
| VND | 法币 | 越南盾 |
| IDR | 法币 | 印尼盾 |
| THB | 法币 | 泰铢 |

**四层汇率体系**：

```
第1层：实时汇率   ← 每分钟从3家三方API取均值
第2层：平台汇率   ← 实时汇率偏差 ≥ 阈值时自动更新（记日志）
第3层：入款汇率   = 平台汇率 × (1 + 入款浮动%) ← 用户充值时看到的汇率
第4层：出款汇率   = 平台汇率 × (1 - 出款浮动%) ← 用户提现时看到的汇率
```

浮动百分比和偏差阈值均在 B 端币种配置中设置，汇差即平台利润来源之一。

## 三、核心表结构设计

### 3.1 表清单

共 9 张核心表，全部位于 db_wallet 库：

| # | 表名 | 职责 | 量级预估 |
|---|------|------|---------|
| 1 | currency_config | 币种元数据（代码/名称/类型/图标/精度/限额） | 极小（5~20 行） |
| 2 | currency_rate | 每种币种的四层汇率 + 3个 API 源数据 | 极小（同币种数） |
| 3 | rate_change_log | 汇率变更审计日志 | 中等（日增数十~数百） |
| 4 | user_wallet | 用户钱包账户（余额 + 冻结 + 版本号） | 大（用户数 × 币种数 × 子钱包数） |
| 5 | wallet_flow | 流水账本（只增不删，审计核心） | 超大（每笔交易 1~3 条） |
| 6 | deposit_order | 充值订单 | 大 |
| 7 | withdraw_order | 提现订单 | 大 |
| 8 | exchange_order | 兑换订单（法币→BSB） | 大 |
| 9 | freeze_record | 冻结记录（提现冻结/解冻/扣款的生命周期追踪） | 中~大 |

### 3.2 核心表字段设计

#### (1) user_wallet — 用户钱包账户

```sql
CREATE TABLE user_wallet (
  id            BIGINT       PRIMARY KEY COMMENT '雪花ID',
  user_id       BIGINT       NOT NULL COMMENT '用户ID',
  currency_id   BIGINT       NOT NULL COMMENT '币种ID → currency_config.id',
  wallet_type   TINYINT      NOT NULL COMMENT '子钱包类型: 1中心 2奖励 3主播 4代理 5场馆',
  balance       BIGINT       NOT NULL DEFAULT 0 COMMENT '可用余额（最小货币单位）',
  frozen_amount BIGINT       NOT NULL DEFAULT 0 COMMENT '冻结金额',
  version       INT          NOT NULL DEFAULT 0 COMMENT '乐观锁版本号（配合悲观锁双保险）',
  status        TINYINT      NOT NULL DEFAULT 1 COMMENT '1正常 2冻结 3禁用',
  create_at     BIGINT       NOT NULL COMMENT '创建时间(ms)',
  update_at     BIGINT       NOT NULL COMMENT '更新时间(ms)',
  delete_at     BIGINT       NOT NULL DEFAULT 0 COMMENT '软删除',
  UNIQUE KEY uk_user_currency_type (user_id, currency_id, wallet_type)
) COMMENT '用户钱包账户';
```

**设计要点**：
- 唯一索引 `(user_id, currency_id, wallet_type)` 保证一个用户在一个币种下一个类型只有一个账户
- `balance` 和 `frozen_amount` 均用 BIGINT 存储最小货币单位整数值
- `version` 作为乐观锁辅助字段，但核心并发控制靠 `SELECT FOR UPDATE` 悲观锁
- 账户按需创建（用户首次使用某币种时自动建账户），非预建

#### (2) wallet_flow — 流水账本

```sql
CREATE TABLE wallet_flow (
  id             BIGINT       PRIMARY KEY COMMENT '雪花ID',
  user_id        BIGINT       NOT NULL COMMENT '用户ID',
  currency_id    BIGINT       NOT NULL COMMENT '币种ID',
  wallet_type    TINYINT      NOT NULL COMMENT '子钱包类型',
  flow_type      TINYINT      NOT NULL COMMENT '流水类型: 1充值 2提现 3兑换入 4兑换出 5冻结 6解冻 7扣款确认 8人工加款 9人工减款 10消费 11返奖 12流水稽核转移',
  amount         BIGINT       NOT NULL COMMENT '变动金额（正=增加,负=减少）',
  balance_before BIGINT       NOT NULL COMMENT '变动前可用余额',
  balance_after  BIGINT       NOT NULL COMMENT '变动后可用余额',
  frozen_before  BIGINT       NOT NULL COMMENT '变动前冻结金额',
  frozen_after   BIGINT       NOT NULL COMMENT '变动后冻结金额',
  order_no       VARCHAR(32)  NOT NULL COMMENT '关联订单号（幂等键）',
  biz_type       TINYINT      NOT NULL DEFAULT 0 COMMENT '业务子类型',
  remark         VARCHAR(256) DEFAULT '' COMMENT '备注',
  create_at      BIGINT       NOT NULL COMMENT '创建时间(ms)',
  UNIQUE KEY uk_order_wallet_flow (order_no, wallet_type, flow_type)
) COMMENT '钱包流水账本（只增不删）';
```

**设计要点**：
- **只增不删**：任何情况下不 UPDATE、不 DELETE，这是审计的命脉
- 唯一索引 `(order_no, wallet_type, flow_type)` 是**幂等的核心保障**——同一订单 + 同一子钱包 + 同一操作类型只能写入一条流水，重复 RPC 调用直接被 DB 拒绝
- `balance_before / balance_after / frozen_before / frozen_after` 四个快照字段，任意一条流水都能独立验证余额变化链是否连续
- `amount` 带符号：正数表示增加可用余额（充值、入账、解冻），负数表示减少（扣款、冻结）

```sql
CREATE TABLE withdraw_order (
  id               BIGINT       PRIMARY KEY COMMENT '雪花ID',
  order_no         VARCHAR(32)  NOT NULL COMMENT '平台订单号: T + 16位',
  user_id          BIGINT       NOT NULL COMMENT '用户ID',
  currency_id      BIGINT       NOT NULL COMMENT '币种ID',
  wallet_type      TINYINT      NOT NULL COMMENT '扣款子钱包类型',
  amount           BIGINT       NOT NULL COMMENT '提现金额',
  actual_amount    BIGINT       NOT NULL DEFAULT 0 COMMENT '实际到账金额',

  status           TINYINT      NOT NULL DEFAULT 1 COMMENT '1待风控审核 2风控通过 3待财务审核 4财务通过 5出款中 6成功 7失败 8驳回',
```

- `frozen_flow_id` 关联到冻结流水记录，提供完整的冻结→出款/驳回生命周期追踪

### 5.2 为什么不引入这 6 项？

| 不引入 | 原因 |
|--------|------|
| **分布式事务框架** | 钱包模块所有余额操作在单库内完成，不需要跨库事务。跨模块（钱包↔财务）靠幂等 RPC + 重试 + 日对账即可保证最终一致。引入 Seata/DTM 反而增加运维复杂度和单点风险。 |
| **消息队列** | 当前链路中无需异步解耦场景——充值回调是财务 RPC 调钱包（同步入账），提现审核是财务 B 端操作后 RPC 调钱包（同步冻结/解冻）。如未来有高吞吐异步需求（如打赏流水），可后加。 |
| **Elasticsearch** | C端记录查询按 (user_id, currency_id) 查，走数据库索引足够。B端统计报表由财务模块负责。钱包模块无全文搜索需求。 |
| **Redlock 分布式锁** | 余额并发控制用 DB 行锁（SELECT FOR UPDATE）——数据库是资金的唯一真相源，用 Redis 分布式锁反而引入 Redis 故障时的一致性风险。 |
| **限流中间件** | 网关层已有限流（gate-font / gate-back 中间件），服务层不需要重复限流。 |
| **复式记账** | 平台钱包不是银行，不需要借贷平衡验证。单式流水表 + 余额快照 + 对账任务已满足审计要求。 |

### 5.3 shopspring/decimal 使用场景

```go
import "github.com/shopspring/decimal"

// 汇率计算示例：法币金额 → BSB 数量
func CalcExchange(fiatAmount int64, rate string) (bsbAmount int64, err error) {
    amt := decimal.NewFromInt(fiatAmount)         // 100000 (VND, 最小单位)
    r, err := decimal.NewFromString(rate)          // "0.00003929" (1 VND = ? USDT)
    if err != nil {
        return 0, err
    }
    bsbRate := decimal.NewFromInt(10)              // 1 USDT = 10 BSB
    result := amt.Mul(r).Mul(bsbRate)              // 100000 × 0.00003929 × 10
    return result.IntPart(), nil                   // 截断为整数 BSB
}
```

## 六、技术挑战与解决方案

### 6.1 P0 级 — 资金安全红线

#### 挑战 1：并发余额操作的一致性

**问题**：两个请求同时扣同一用户余额（如同时充值回调 + 提现冻结），可能出现覆盖写（Lost Update）或超扣。

**方案**：**悲观锁 SELECT FOR UPDATE**

```go
// ledger 层内部，事务内操作
func (l *Ledger) Debit(ctx context.Context, tx *gorm.DB, userId, currencyId int64, walletType int8, amount int64, orderNo string) error {
    // 1. 行锁查询
    wallet, err := tx.Raw("SELECT * FROM user_wallet WHERE user_id=? AND currency_id=? AND wallet_type=? FOR UPDATE",
        userId, currencyId, walletType).First(&wallet)

    // 2. 余额校验
    if wallet.Balance < amount {
        return errs.BalanceInsufficient
    }

    // 3. 扣减余额
    tx.Model(&wallet).Update("balance", wallet.Balance - amount)

    // 4. 写流水（幂等：唯一索引保护）
    tx.Create(&WalletFlow{
        OrderNo: orderNo, WalletType: walletType, FlowType: FlowTypeDebit,
        Amount: -amount, BalanceBefore: wallet.Balance, BalanceAfter: wallet.Balance - amount,
    })

    return tx.Commit().Error
}
```

**为什么选悲观锁而不是乐观锁？**
- 乐观锁（version 字段比较）在冲突时需重试，资金场景重试可能导致用户体验差且逻辑复杂
- 悲观锁虽然有等待开销，但钱包场景的并发粒度是"单用户单币种单子钱包"（极少冲突），等待时间很短
- 资金操作不允许"乐观失败后重试"这种语义——必须确保要么成功要么明确失败

**防死锁**：多行锁场景（如兑换同时扣法币 + 加 BSB）统一按 `(user_id, currency_id, wallet_type)` 升序锁定。

#### 挑战 2：幂等性保证

**问题**：财务模块 RPC 调 WalletCredit 充值入账，网络超时后重试，可能导致重复入账。

**方案**：**DB 唯一索引 + 错误码区分**

```
wallet_flow 唯一索引: (order_no, wallet_type, flow_type)
```

- 重复请求 → INSERT 触发 Duplicate Key → 捕获此 error → 返回"已处理"而非"失败"
- 调用方收到"已处理"后不再重试，也不会误认为失败
- 不依赖 Redis 做幂等（Redis 故障 = 幂等失效 = 资金风险）

#### 挑战 3：金额精度

**问题**：浮点数计算 0.1 + 0.2 ≠ 0.3，汇率可能有 8 位以上小数。

**方案**：**BIGINT 存储 + shopspring/decimal 计算**

- 数据库存储：所有金额字段用 BIGINT，存最小货币单位整数（如 VND 存"盾"、USDT 存 10^-8 单位）
- 业务计算：所有涉及乘除运算（汇率换算、手续费计算）使用 shopspring/decimal
- 汇率存储：用 VARCHAR(32) 存储汇率原始值，避免 DECIMAL 类型的精度限制
- 前端展示：C端接口返回 string 类型金额（JavaScript 的 Number 有精度限制）

#### 挑战 4：数据一致性

**问题**：一次操作涉及多步（如兑换：扣法币 + 加 BSB + 加赠送 BSB + 写 3 条流水 + 写订单），部分成功部分失败怎么办？

**方案**：**单库本地事务**

所有 ledger 层的原子操作都在同一个数据库事务内完成：
- 锁行 → 改余额 → 写流水 → 写订单 —— 全在一个 `tx.Transaction()` 内
- 任何一步失败，整个事务回滚，不会出现中间状态
- 钱包模块的所有表都在同一个 db_wallet 库内，无跨库问题

### 6.2 P1 级 — 跨模块协作

#### 挑战 5：跨服务事务（钱包↔财务）

**问题**：充值成功后，财务模块回调钱包入账。如果入账 RPC 超时，财务不知道钱包是否已入账。

**方案**：**幂等 RPC + 有限重试 + 日对账**

```
财务 → RPC WalletCredit(orderNo=C202602280001)
    ├─ 成功 → 完成
    ├─ 超时 → 重试（钱包幂等，重复入账被 DB 拦截，返回"已处理"）
    └─ 明确失败 → 告警 + 人工介入

每日对账任务：
  财务成功订单列表 ↔ 钱包入账流水列表 → 差异 = 补单候选
```

不用分布式事务框架的理由：
1. 钱包作为"被调方"只需保证幂等即可，不需要参与两阶段提交
2. 对账是金融系统的标配兜底机制，比任何分布式事务框架都可靠
3. 引入 Seata/DTM 会使部署架构复杂度增加一个量级，且本身也可能成为单点故障

#### 挑战 6：提现的冻结-确认模型

**问题**：提现从发起到出款成功可能经历数小时（风控审核→财务审核→通道出款），期间需要保证资金不被消费。

**方案**：**freeze → deduct/unfreeze 三步模型**

```
提现发起:  WalletFreeze(amount)  → available -= amount, frozen += amount
出款成功:  WalletDeduct(amount)  → frozen -= amount（实际扣除）
出款失败:  WalletUnfreeze(amount) → frozen -= amount, available += amount（归还）
```

每一步都是独立的幂等 RPC 调用，每一步都写流水。即使任何一步超时重试，幂等机制保证不会重复冻结/解冻。

### 6.3 P2 级 — 性能与运维

#### 挑战 7：热点账户

**问题**：头部主播可能在短时间内收到大量打赏（高并发写同一行），行锁排队导致吞吐下降。

**方案（预留，当前阶段不实现）**：
- 方案 A：主播钱包做"分桶"——将一个逻辑账户拆为 N 个物理行，写操作随机落到不同桶，读操作汇总
- 方案 B：批量合并写——将短时间内的多笔打赏合并为一次余额变更
- 当前阶段用户量不足以触发此问题，先用基础方案（单行锁），监控 P99 延迟，超阈值再升级

#### 挑战 8：wallet_flow 表膨胀

**问题**：流水表只增不删，随业务增长将达到亿级。

**方案**：
- TiDB 天然支持水平扩展，亿级数据量在合理分区下性能可控
- 查询走 (user_id, currency_id) 索引，不会全表扫描
- 冷数据归档策略（预留）：超过 N 个月的流水迁移到历史表，C端记录查询只查近 N 个月

#### 挑战 9：汇率缓存一致性

**问题**：汇率数据被频繁读取（每次充值/提现/兑换页面都要展示汇率），但更新频率低（分钟级）。

**方案**：**Cache-Aside + 短 TTL**
- 读：先查 Redis → 未命中查 DB → 写 Redis（TTL 60s）
- 写（定时任务更新汇率时）：先写 DB → 删 Redis 缓存 → 下次读重新加载
- 最大不一致窗口 = TTL 时间（60s），对汇率展示场景可接受
- 实际下单时的汇率以 DB 为准（不走缓存），保证交易精确

### 6.4 P3 级 — 安全与合规

#### 挑战 10：敏感数据保护

**方案**：
- 提现收款信息（银行卡号、USDT 地址）在 withdraw_order 表中 AES 加密存储（receiver_info 字段）
- 列表展示使用脱敏摘要（receiver_summary），仅详情页解密完整信息
- 加密密钥从 ETCD 配置中心获取，不硬编码

#### 挑战 11：接口安全

**方案**：
- C端：FontAuthMiddleware 验证 JWT Token，用户只能操作自己的钱包
- B端：BackAuthMiddleware + RBAC 权限控制，IP 白名单
- RPC：服务间通信走内网，TTHeader 传递 trace-id 全链路追踪
- 金额类接口统一校验：amount > 0，amount ≤ 限额，币种有效等

## 十、潜在问题与应对预案

> 以下是在需求大会或技术评审中可能被问到的问题，提前准备应答。

### Q: 为什么不用分布式事务框架？

**答**：钱包模块的核心特征是"余额操作发生在单库内"——user_wallet 和 wallet_flow 在同一个 db_wallet 库，一个事务搞定。跨模块（钱包↔财务）不需要强一致性，用"幂等 RPC + 重试 + 对账"就够了。Seata/DTM 引入后会增加部署复杂度（独立 TC 节点）、增加单点风险、增加 RPC 延迟（多了 2 轮通信），收益不抵成本。业界（支付宝、微信支付）在类似场景中也优先采用幂等+对账模式。

### Q: 乐观锁和悲观锁怎么选？

**答**：核心余额操作选悲观锁。原因：
1. 资金扣减不允许"失败重试"语义——用户点了提现，系统告诉他"抢锁失败请重试"是不可接受的
2. 冲突粒度是"单用户单币种单子钱包"，真实并发极低（一个用户不太可能在同一毫秒做两笔操作），行锁等待时间极短
3. 悲观锁 + 短事务（毫秒级）= 吞吐完全够用

### Q: wallet_flow 表数据量越来越大怎么办？

**答**：
1. 近期靠 TiDB 的分布式能力扛住亿级数据（已在其他大表验证过）
2. 查询全部走索引 (user_id, currency_id)，不会全表扫描
3. 中期做时间分区（按月），超过 6 个月的数据归档到历史表
4. 远期如有全文检索需求（如运营搜索流水备注），可增加 ES 做查询入口

### Q: Redis 缓存和 DB 不一致怎么办？

**答**：钱包模块的缓存策略是 Cache-Aside，只缓存读频高写频低的数据（币种配置、汇率信息），不缓存余额。余额查询直接走 DB（行锁场景已经在 DB 了，多一次读开销不大）。币种/汇率缓存即使短暂不一致（最多 60s TTL），不影响资金安全——实际交易时以 DB 为准。

### Q: 如果财务模块延期怎么办？

**答**：钱包模块约 70% 的工作量不依赖财务模块（Phase 1~5），可独立推进：
- 币种配置：完全自闭环
- Ledger 核心 + RPC 暴露：只需定义好接口契约，不需要财务先实现
- 兑换流程：完全在钱包内部完成
- 余额/记录查询：不依赖外部

仅充值和提现的"创建订单→通道匹配→回调入账"链路需要财务对接。对接前，我们编写 Mock 实现（模拟通道匹配返回假的支付 URL、模拟回调触发入账），可以独立完成全流程的功能验证。

### Q: 安全方面做了哪些考虑？

**答**：
1. **接口层**：C端 JWT 鉴权 + 用户只能操作自己的钱包（userId 从 token 取），B端 RBAC + IP 白名单
2. **数据层**：提现收款信息 AES 加密存储，列表仅展示脱敏摘要
3. **资金层**：悲观锁防并发超扣，幂等索引防重复入账，流水只增不删防篡改
4. **审计层**：B 端所有写操作记审计日志（ser-blog），流水表记录变更前后快照，日对账发现差异
5. **业务层**：重复转账检测（1~2 笔软拦截、≥3 笔硬拦截），提现三级审核（风控→财务→出款），提现账户绑定后不可自改

### Q: 如何保证汇率的准确性？

**答**：四层汇率体系 + 三方交叉验证：
1. 从 3 家独立 API 源拉取实时汇率，取平均值（排除异常值）
2. 当实时汇率与平台汇率偏差 ≥ 阈值时自动更新（分钟级检测）
3. 每次更新自动生成审计日志（变更前后值、触发方式）
4. 入款/出款汇率在平台汇率基础上加浮动百分比，B 端可配置
5. 下单瞬间锁定汇率（写入订单表），不受后续汇率波动影响

### Q: 消费（投注/打赏）时的扣款优先级怎么设计？

**答**：预留设计，未来实现 WalletConsume 接口时的扣款优先级：
1. 先扣中心钱包，中心不足再扣奖励钱包
2. 混合扣款时按比例记录（如一笔 100 BSB 的投注，中心扣 70、奖励扣 30）
3. 返奖时按原始扣款比例归还（赢了 200 BSB，按 7:3 比例返回中心 140、奖励 60）
4. 调用方（投注模块）不需要关心优先级策略，由钱包内部统一处理

### Q: 兑换赠送的 10% BSB 进奖励钱包，流水要求怎么实现？

**答**：奖励钱包的资金有"流水稽核"约束——需完成 X 倍流水后才能提现。具体实现：
1. CreditReward 入账时，在流水记录中标记 `audit_multiplier`（如 3 倍）
2. 用户消费（投注等）时，消费金额计入流水贡献
3. 当累计流水 ≥ 奖金金额 × 倍数时，奖励钱包余额"解锁"可提现
4. 提现时校验：奖励钱包是否有未达标的流水要求，未达标部分不可提现
