## 一、一句话定位

ser-wallet 是一个**多币种、多子钱包、跨境支付**的资金管理模块——支持法币(VND/IDR/THB)、加密货币(USDT)、平台币(BSB)三类币种，覆盖充值、兑换、提现、记录四大用户场景，对内暴露9个RPC给财务模块执行资金操作，对外依赖5个模块8个RPC完成业务闭环。

**一句话关系**：钱包是资金的**执行者**（管钱的加减冻结），财务是资金的**决策者**（管审核、通道、风控）。

## 二、设计思路总览

### 2.1 设计四原则

| 原则 | 含义 | 体现 |
|------|------|------|
| **安全第一** | 资金零容错，宁可操作失败也不能多入/少扣 | DB层UNSIGNED约束 + 乐观锁 + 幂等 + 对账 |
| **不过度设计** | 当前需求没有的不强加 | 代理/场馆钱包预留字段但不实现 |
| **复用工程模式** | 融入已有19服务的开发规范 | 同一个WalletService、同一种IDL规范、同一套GORM Gen |
| **演进式扩展** | 关键位置留接口，不提前实现 | 9个RPC原子操作覆盖未来直播/商城/投注场景 |

### 2.3 数据流方向

| 资金流向 | 链路 | 特点 |
|---------|------|------|
| **充值（资金入）** | 用户 → 三方支付 → 财务确认 → 钱包入账 | 回调驱动，幂等核心 |
| **提现（资金出）** | 用户 → 钱包冻结 → 财务审核 → 代付通道 → 钱包扣款/解冻 | 链路最长，冻结三件套 |
| **兑换（内部流转）** | 用户 → 钱包扣法币 + 加BSB | 零外部依赖，最简闭环 |

## 三、五种子钱包体系

### 3.1 子钱包结构

| 子钱包 | wallet_type | 资金来源 | 用途 | 本期状态 |
|--------|-------------|---------|------|---------|
| **中心钱包** | 1 | 充值、人工加款 | 投注、消费、提现（需完成稽核流水） | **必须实现** |
| **奖励钱包** | 2 | 活动奖金、充值赠送、兑换赠送 | 完成N倍流水后可转中心 | **必须实现** |
| **主播钱包** | 3 | 礼物打赏、短视频收益 | 可直接提现或转中心 | **必须实现** |
| **代理钱包** | 4 | 代理佣金（下级分佣） | 可提现或转中心 | **预留不实现** |
| **场馆钱包** | 5 | 中心+奖励转入 | iGaming场馆临时使用 | **预留不实现** |

### 3.2 核心规则（绕不过的）

- **资金钱包余额** = 中心钱包.available + 奖励钱包.available（不含主播/代理）
- **消费优先级**：先扣中心，不足再扣奖励（硬规则）
- **混合投注返奖**：投注100（中心70+奖励30），返奖200 → 中心140 + 奖励60（按原始比例）
- **币种隔离**：每个币种独立一套完整子钱包体系，互不干扰，不支持跨币种混合
- **账户唯一约束**：`(user_id, currency_code, wallet_type)` 三元组全局唯一

### 3.3 账户生命周期

```
用户首次访问某币种
    │
    ▼
惰性初始化 3 个子钱包（中心/奖励/主播），余额均为 0
    │  INSERT ON DUPLICATE KEY IGNORE（并发安全）
    │
    ▼
只增不删，balance 可为 0 但账户永不删除
```

## 四、三类币种与汇率体系

### 4.1 三类币种

| 类型 | currency_type | 代表 | 精度 | 千分位 | 小数点 |
|------|--------------|------|------|--------|--------|
| 法定货币 | 1 | VND/IDR/THB | 2位（VND/IDR实际显示整数） | `.` | 无/`.` |
| 加密货币 | 2 | USDT | 8位 | `,` | `.` |
| 平台货币 | 3 | BSB | 2位 | `,` | `.` |

### 4.2 基准币种

- **USDT** 是全局基准币种，**只能设置一次，不可修改**
- BSB 锚定 USDT：**10 BSB = 1 USDT**（写死）
- 所有跨币种统计、报表、对账最终折算为 USDT

### 4.3 四层汇率体系

```
         三方API-1 ─┐
         三方API-2 ──┤──取平均──► 实时汇率（real_rate）
         三方API-3 ─┘                  │
                                       │ 偏差 = |实时 - 平台| / 实时 × 100%
                                       │
                               偏差 ≥ 阈值% ？
                                   │        │
                                  是        否
                                   │        │
                                   ▼        └──► 不更新
                            更新平台汇率（platform_rate）
                                   │
                        ┌──────────┴──────────┐
                        ▼                     ▼
                入款汇率                 出款汇率
          platform × (1+浮动%)     platform × (1-浮动%)
          用户少拿，平台赚差价       用户少拿，平台赚差价
```

**汇率使用场景**：
| 场景 | 使用哪个汇率 | 说明 |
|------|-------------|------|
| C端展示（充值/兑换页面） | 平台汇率 | 参考展示 |
| 创建充值订单 | 入款汇率 → 锁汇快照 | 用户整个订单生命周期内固定 |
| 创建提现订单 | 出款汇率 → 锁汇快照 | 同上 |
| 创建兑换订单 | 入款汇率 → 锁汇快照 | 执行时重新获取，不用预览时的值 |
| 统计报表折算 | 平台汇率 | 折算为基准币种USDT |

### 4.4 精度规则

| 场景 | 法币 | 加密币 | 说明 |
|------|------|--------|------|
| 显示 | 2位小数（VND/IDR显示整数） | 8位小数 | 不足补0，超过四舍五入 |
| 传输（IDL） | string | string | ADR-06：string传输，避免浮点精度丢失 |
| 运算（Service层） | shopspring/decimal | shopspring/decimal | 高精度十进制 |
| 存储（DB） | DECIMAL(30,8) | DECIMAL(30,8) | 统一30位总长8位小数 |

## 五、核心业务链路

### 5.1 充值链路（三种方式）

**USDT充值（平台自建页面）**：
```
用户选USDT → 选网络(TRC20/ERC20/BEP20) → 输入金额
    → 钱包换算USDT金额(用入款汇率)
    → 展示收款地址 + 二维码
    → 用户链上转账
    → 区块链监听服务确认(2~10分钟)
    → 财务确认金额 → 调用 WalletCredit 入账
```

**银行转账/电子钱包扫码（三方托管页面）**：
```
用户选银行/扫码 → 输入金额 → 选奖金活动(可选)
    → 重复转账拦截检查
        1~2笔同类待处理 → 软拦截（弹窗警告，允许继续）
        ≥3笔 → 硬拦截（禁止创建）
    → 钱包创建本地订单 → 三方通道创建订单
    → 跳转三方支付页面
    → 三方回调(成功/失败)
    → 财务确认 → WalletCredit(中心) + WalletCreditReward(奖励,如有)
```

**额外奖金规则**：
- 每笔充值只能选一个奖金活动（互斥）
- 有最低充值门槛（如"需最低充值BSB 1000"）
- 充值金额 → 中心钱包，奖金金额 → 奖励钱包
- 奖金部分需完成N倍流水后可提现

### 5.2 兑换链路（法币→BSB，单向）

```
用户进入兑换页面
    → 展示汇率 "1 BSB = ₫2,620"（平台汇率）
    → 输入法币金额 ₫1,000
    → 预览：兑换获得 ≈ 0.38 BSB + 赠送10% ≈ 0.038 BSB = 实际到账 0.418 BSB
    → 点击兑换
    → 重新获取入款汇率（不用预览时的快照）
    → 一个事务内完成：
        ① 扣法币(中心钱包) available -= ₫1,000
        ② 加BSB(中心钱包) available += 0.38 BSB
        ③ 加BSB(奖励钱包) available += 0.038 BSB（赠送部分）
        ④ 写入兑换订单 + 三条资金流水
    → 返回成功
```

**兑换特点**：
- **唯一零外部RPC依赖**的业务场景
- 仅 法币→BSB 单向（BSB不能换回法币）
- 赠送部分入奖励钱包，需完成流水倍数要求
- 适合最先跑通验证

### 5.3 提现链路（最复杂）

```
用户点击提现
    │
    ▼
① KYC校验 → rpc.KycClient().KycQuery()
    ├── status=3(已通过) → 全渠道可用（银行卡+电子钱包+USDT）
    └── 其他 → 仅USDT提现
    │
    ▼
② 获取提现配置 → rpc.FinanceClient().GetWithdrawMethodList() + GetWithdrawConfig()
    │  展示：可用提现方式、限额、费率、到账时间
    │
    ▼
③ 用户填写/复用提现账户
    │  首次：填写银行名/账号/户名（自动填KYC姓名，灰显不可编辑）
    │  非首次：自动带出已有信息（只读）
    │  校验：账户持有人 = KYC认证姓名（忽略大小写和多余空格）
    │
    ▼
④ 用户输入金额，确认提现
    │  校验：单笔限额、单日限额、余额充足
    │
    ▼
⑤ 钱包本地操作（一个事务内）
    │  - 创建提现订单(T+16位)
    │  - 冻结余额：available -= amount, frozen += amount
    │  - 创建冻结记录
    │
    ▼
⑥ 提交财务审核 → rpc.FinanceClient().SubmitWithdrawOrder()
    │  如果提交失败 → 必须回滚解冻（defer补偿）
    │
    ▼
⑦ 财务侧异步流程
    │
    ├── 风控审核驳回 → 财务调用 WalletUnfreeze → 解冻归还
    ├── 财务审核驳回 → 财务调用 WalletUnfreeze → 解冻归还
    ├── 代付通道出款失败 → 财务调用 WalletUnfreeze → 解冻归还
    └── 代付通道出款成功 → 财务调用 WalletDeduct → 最终扣除冻结资金
```

**提现币种规则**（硬规则）：
| 充值方式 | 允许的提现方式 |
|---------|-------------|
| 法币充值 | 法币提现 |
| USDT充值 | USDT提现 |
| 法币+USDT混合 | 法币提现 |
| 法币→兑换BSB | 法币提现 |

### 5.4 人工修正链路

```
B端管理员创建修正单
    │
    ├── 补单(B+16位)
    │   管理员输入原始充值订单号 → 自动回填
    │   审核通过 → WalletCredit(中心) + WalletCreditReward(奖励,如有)
    │
    ├── 人工加款(A+16位)
    │   财务先调 GetUserBalance → 展示各子钱包当前余额
    │   管理员填写各子钱包加款金额 + 原因
    │   审核通过 → WalletManualAdd（一个事务内最多4子钱包同时加）
    │
    └── 人工减款(M+16位)
        财务先调 GetUserBalance → 展示各子钱包当前余额
        管理员填写各子钱包减款金额 + 原因
        审核通过 → WalletManualSub
        特殊规则：余额不足时，实际扣减=当前余额（扣到0，不会变负）
        响应返回实际扣减金额（可能与请求金额不同）

创建人 ≠ 审核人（双人复核规则）
```

## 六、表结构设计

### 6.1 总览（9张必须表，db_wallet库）

```
币种配置域
  ├── t_currency          币种配置（20+字段，全局唯一currency_code）
  └── t_rate_log          汇率变更日志（只INSERT不UPDATE，审计不可变）

钱包核心域
  ├── t_wallet_account    子钱包账户（user×currency×walletType 三维唯一）
  ├── t_wallet_flow       资金流水（不可变审计记录，14种flow_type）
  └── t_freeze_record     冻结记录（冻结→解冻/扣除 状态机）

充值域
  └── t_deposit_order     充值订单（C+16位订单号）

提现域
  ├── t_withdraw_order    提现订单（T+16位，8种状态）
  └── t_withdraw_account  提现账户（用户绑定的银行卡/电子钱包/USDT地址）

兑换域
  └── t_exchange_order    兑换订单（E+16位，同步操作，创建即终态）
```

### 6.2 逐表详解

#### t_currency — 币种配置表

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK | 主键 |
| currency_code | VARCHAR(10) UNIQUE | 币种代码，如 VND/USDT/BSB |
| currency_name | VARCHAR(50) | 币种名称 |
| currency_type | TINYINT | 1法币 2加密 3平台 |
| country | VARCHAR(50) | 所属国家 |
| icon_url | VARCHAR(255) | 图标URL |
| currency_symbol | VARCHAR(10) | 货币符号，如 ₫/Rp/฿/$ |
| thousands_sep | VARCHAR(5) | 千分位符号 |
| decimal_sep | VARCHAR(5) | 小数点符号 |
| precision | TINYINT | 精度位数（法币2/加密8） |
| real_rate | DECIMAL(30,8) | 实时汇率（三方API） |
| platform_rate | DECIMAL(30,8) | 平台汇率 |
| rate_fluctuation | DECIMAL(10,4) | 汇率浮动百分比 |
| rate_threshold | DECIMAL(10,4) | 偏差阈值百分比 |
| deposit_rate | DECIMAL(30,8) | 入款汇率 = 平台 × (1+浮动%) |
| withdraw_rate | DECIMAL(30,8) | 出款汇率 = 平台 × (1-浮动%) |
| is_base | TINYINT DEFAULT 0 | 是否基准币种（整表仅一条为1） |
| sort_value | INT | 排序 |
| status | TINYINT DEFAULT 1 | 1启用 2禁用 |
| create_by/update_by | BIGINT | 操作人 |
| create_at/update_at | BIGINT | 时间戳 |
| delete_at | BIGINT DEFAULT 0 | 软删除 |

**关键约束**：
- `currency_code` 创建后不可修改
- `currency_type` 创建后不可修改
- `is_base` 只能设置一次，对应币种不可禁用
- 索引：`uk_currency_code`, `idx_status_sort`

#### t_wallet_account — 子钱包账户表

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK | 主键 |
| user_id | BIGINT | 用户ID |
| currency_code | VARCHAR(10) | 币种代码 |
| wallet_type | TINYINT | 1中心 2奖励 3主播 4代理 5场馆 |
| available_balance | DECIMAL(30,8) UNSIGNED | 可用余额 |
| frozen_balance | DECIMAL(30,8) UNSIGNED | 冻结余额 |
| version | BIGINT DEFAULT 0 | 乐观锁版本号 |
| create_at/update_at | BIGINT | 时间戳 |
| delete_at | BIGINT DEFAULT 0 | 软删除 |

**关键约束**：
- **唯一索引** `(user_id, currency_code, wallet_type)` — 核心约束
- `UNSIGNED` 确保余额不能变负（DB层兜底）
- `version` 每次余额变更 +1（乐观锁）
- 索引：`idx_user_currency`

#### t_wallet_flow — 资金流水表（不可变）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK | 主键 |
| user_id | BIGINT | 用户ID |
| currency_code | VARCHAR(10) | 币种 |
| wallet_type | TINYINT | 子钱包类型 |
| flow_type | TINYINT | 14种流水类型（见下表） |
| amount | DECIMAL(30,8) **SIGNED** | 变动金额（正=增/负=减） |
| balance_before | DECIMAL(30,8) | 变动前可用余额 |
| balance_after | DECIMAL(30,8) | 变动后可用余额 |

## 八、技术栈与选型理由

### 8.1 技术栈全景

| 层面 | 技术 | 版本 | 选型理由 |
|------|------|------|---------|
| 语言 | Go | 1.25.5 | 团队统一，go.work多模块管理 |
| HTTP网关 | Hertz | v0.10.3 | CloudWeGo生态，gate-back/font已用 |
| RPC框架 | Kitex + Thrift | v0.16.0 | 19个服务统一用，ETCD服务发现 |
| ORM | GORM + GORM Gen | — | 类型安全查询，自动生成model/query/repo |
| 数据库 | TiDB | MySQL兼容 | 已有基建，支持分布式事务 |
| 缓存 | Redis | — | 汇率缓存、分布式锁 |
| 配置/注册中心 | ETCD | — | 服务发现+动态配置，无本地文件 |
| 消息队列 | NATS JetStream | — | 事件通知（充值/提现创建后通知财务） |
| 对象存储 | AWS S3 | SDK v2 | 币种图标上传 |
| 任务调度 | robfig/cron | — | 进程内定时任务（汇率同步） |
| 高精度计算 | shopspring/decimal | — | 金额运算，避免浮点精度丢失 |
| ID生成 | 雪花算法 | — | 订单号生成（C/T/E+16位） |

### 8.2 选型要点说明

**为什么金额用 string 传输（而非工程现有的 i64）**：
- 现有工程用 i64（最小单位，如分），适用于单精度场景（直播打赏）
- 钱包是多精度场景：法币2位、加密币8位，i64需要不同的乘数，前后端易出错
- string传输 + shopspring/decimal运算 + DECIMAL(30,8)存储 = 全链路精度安全

**为什么用乐观锁（而非悲观锁）**：
- 同一账户并发写概率低（用户级别隔离）
- 无死锁风险（钱包系统不能死锁）
- TiDB原生支持乐观事务
- 重试逻辑简单（2~3次，无sleep）

**为什么汇率定时任务用进程内cron（而非ser-cron集中）**：
- 汇率逻辑与币种配置数据紧耦合
- 进程内调用零网络开销
- Redis SETNX分布式锁防止多实例重复执行

## 九、七大技术挑战与解决方案

### 挑战1：资金一致性 — 如何保证钱不会多/少

**问题**：充值回调可能重复、并发操作可能导致余额不一致

**解决方案（四层防御）**：

| 防御层 | 机制 | 作用 |
|--------|------|------|
| DB层 | `UNSIGNED` 约束 | 余额不能变负，DB层兜底 |
| 索引层 | `(related_order_no, flow_type)` 唯一索引 | 幂等性，同一订单不能重复入账 |
| 应用层 | 6个原子操作，禁止直接UPDATE余额 | 所有余额变动走统一入口 |
| 审计层 | t_wallet_flow不可变记录 | 余额 = 流水SUM，可对账校验 |

**对账公式（恒等式）**：
```
account.available_balance == last_flow.balance_after
account.frozen_balance == last_flow.frozen_after
```

### 挑战2：幂等性 — 如何防止重复入账

**问题**：三方充值回调可能重试，网络超时可能导致重复调用

**解决方案**：

```
每次资金操作：
  1. 先查 t_wallet_flow 是否已存在 (order_no, flow_type)
  2. 已存在 → 直接返回成功（幂等）
  3. 不存在 → 执行余额变更 + 写入流水（一个事务）
  4. 唯一索引兜底：即使并发，第二个INSERT会失败 → 返回幂等成功

示例：三方充值回调重试
  回调1 → WalletCredit(orderNo=C001, amount=100) → 入账成功
  回调2 → WalletCredit(orderNo=C001, amount=100) → 检测到已存在 → 幂等返回成功
```

### 挑战3：并发控制 — 多个操作同时修改同一余额

**问题**：用户同时发起提现和兑换，两个操作同时修改同一子钱包

**解决方案（乐观锁 + 重试）**：

```go
func withRetry(ctx context.Context, maxRetries int, fn func() error) error {
    for i := 0; i <= maxRetries; i++ {
        err := fn()
        if err == ErrConcurrencyConflict && i < maxRetries {
            continue  // 不sleep，冲突是瞬时的
        }
        return err
    }
}

// 事务内
tx.WalletAccount.
    Where(ID == account.ID, Version == account.Version).  // 版本号匹配
    UpdateSimple(AvailableBalance.Add(-amount), Version.Add(1))

if RowsAffected == 0 → return ErrConcurrencyConflict  // 触发重试
```

### 挑战4：提现冻结事务一致性 — 冻结成功但提交财务失败

**问题**：CreateWithdrawOrder中，先冻结余额，再RPC提交财务审核。如果提交失败，冻结的钱卡住了

**解决方案（补偿回滚）**：

```go
func CreateWithdrawOrder(ctx, req) error {
    // ① 本地事务：创建订单 + 冻结余额
    err := tx(func() {
        createOrder(...)
        freezeBalance(...)
    })
    if err != nil { return err }

    // ② RPC提交财务（事务外）
    err = rpc.FinanceClient().SubmitWithdrawOrder(ctx, req)
    if err != nil {
        // ③ 提交失败 → 必须补偿回滚
        unfreezeBalance(orderNo)  // 解冻归还
        updateOrderStatus(orderNo, StatusFailed)
        return err
    }
    return nil
}
```

### 挑战5：多币种精度 — 法币2位 vs 加密币8位

**问题**：VND精度2位，USDT精度8位，汇率换算中间结果可能需要更高精度

**解决方案**：

```
全链路统一：
  IDL传输：string（"100.50"、"0.00012345"）
  Service运算：shopspring/decimal（任意精度）
  DB存储：DECIMAL(30,8)（统一30位总长，8位小数）

展示格式化：
  根据 t_currency.precision 字段决定显示几位小数
  根据 thousands_sep/decimal_sep 决定千分位和小数点符号

汇率换算示例：
  法币充值：₫2,620,000 × 入款汇率(0.0003817) = 1000.054 USDT
  按USDT精度8位保留：1000.05400000
  按法币精度2位保留：1000.05
```

### 挑战6：汇率实时性 vs 订单锁汇

**问题**：用户看到汇率A，点击充值时汇率可能已变成B

**解决方案（锁汇快照）**：

```
预览阶段：展示平台汇率（实时）
创建订单阶段：重新获取当前入款/出款汇率 → 快照写入订单
整个订单生命周期内：使用快照汇率，不再变动

t_deposit_order.exchange_rate  = 创建时锁定的汇率
t_withdraw_order.exchange_rate = 创建时锁定的汇率
t_exchange_order.exchange_rate = 创建时锁定的汇率
```

### 挑战7：KYC状态降级 — KYC服务不可用时提现功能怎么办

**问题**：ser-kyc宕机，用户无法提现

**解决方案（安全降级）**：

```go
kycResp, err := rpc.KycClient().KycQuery(ctx, req)
if err != nil {
    // KYC服务不可用 → 降级为"未认证"
    // 安全的：不会因故障允许未认证用户使用银行卡提现
    kycStatus = 1  // 未认证
    kycName = ""
    // 用户仅能选择USDT提现
}
```

## 十一、跨模块依赖与协作

### 11.1 依赖矩阵

```
调用方 ＼ 被调方 │ 币种配置  │ 钱包模块  │ 财务模块  │ 用户/KYC
────────────────┼─────────┼─────────┼─────────┼─────────
币种配置         │   —     │   ✗     │   ✗     │   ✗
钱包模块         │  ★★★   │   —     │  ★★    │   ★
财务模块         │   ★     │  ★★★   │   —     │   ✗

★★★ = 强依赖（多接口，核心链路）  ★★ = 中依赖  ★ = 弱依赖  ✗ = 无
```

### 11.2 双向依赖核心风险

钱包 ↔ 财务是**双向依赖**（互相调用RPC），处理不当会导致循环阻塞。

| 方向 | 性质 | 处理方式 |
|------|------|---------|
| 钱包→财务 | 同步查配置、同步提交订单 | 正常RPC，失败需补偿回滚 |
| 财务→钱包 | 异步回调触发（充值确认/审核结果） | 必须幂等，重复调用不重复执行 |

### 11.3 跨团队协作节点

```
时间线 ─────────────────────────────────────────────────────►

    ❶ IDL定义对齐           ❷ Mock联调          ❸ 真实联调
    │                      │                   │
    │  双方同时确认：        │  各自Mock：         │  真实环境：
    │  钱包暴露的9个RPC     │  钱包Mock财务RPC   │  充值回调
    │  财务暴露的4个RPC     │  财务Mock钱包RPC   │  提现审核全流程
    │                      │                   │  人工修正
    │                      │                   │
    ▼                      ▼                   ▼
  越早越好               各自开发阶段          集成测试阶段
```

## 十五、高频提问预案（FAQ）

### Q1：为什么选择一个WalletService而不是拆成多个服务？

**A**：遵循现有工程19个服务的统一模式——每个服务一个Service定义。30个方法的量级与 ser-live（37方法）和 ser-app（25方法）相当。拆多个Service会增加运维复杂度且无明显收益。B端/C端/RPC通过网关路由和调用方式自然区分。

### Q2：钱包余额的一致性怎么保证？不会出错吗？

**A**：四层防御——DB层UNSIGNED兜底（余额不可能变负）、唯一索引幂等（同一订单不重复操作）、乐观锁版本号（并发安全）、不可变流水记录（余额=流水SUM，可对账）。这不是靠某一层保证，而是每一层都有独立的防线。

### Q3：财务模块还没创建，你怎么开发？

**A**：按"调料vs火"分类——币种配置是"火"（同进程，必须先实现），财务RPC是"调料"（可Mock继续开发）。开发阶段Mock财务的4个RPC，硬编码返回充值方式列表、提现配置等。IDL定义要尽早与财务团队对齐，这是并行开发的前提。

### Q4：为什么金额不用float64/double？

**A**：浮点精度问题。`0.1 + 0.2 = 0.30000000000000004` 这在金融场景是不可接受的。我们用 string传输 + shopspring/decimal运算 + DECIMAL(30,8)存储，全链路无精度丢失。这也是与现有工程不同的地方——现有服务用 i64 最小单位方案，但那适用于单精度场景（直播打赏固定2位），我们是多精度（法币2位 + 加密8位）。

### Q5：提现的冻结/解冻/扣除为什么要暴露RPC？钱包内部不能直接完成吗？

**A**：因为提现审核流程在**财务模块**——冻结后进入风控审核→财务审核→代付出款，每一步的结果决定是解冻还是扣除。钱包只是执行者（你让我冻/解/扣，给我订单号，我执行），财务是决策者（判断该冻该解还是该扣）。这是职责分离原则。

### Q6：汇率是实时的吗？用户看到的价格和实际成交价格一样吗？

**A**：展示用平台汇率（近实时，定时更新），成交用锁汇机制——创建订单时获取当前入款/出款汇率并快照写入订单，整个订单生命周期内固定。这意味着用户看到的和实际成交之间**可能有微小差异**（展示时和点击之间汇率可能更新），但订单一旦创建就是确定的价格。

### Q7：KYC不通过就不能提现吗？

**A**：不是完全不能提现。未认证用户可以走USDT提现（因为USDT不需要实名银行账户）。KYC认证是解锁银行卡/电子钱包等法币提现渠道的前提。这是"可选但有激励"的设计，不是强制屏障。

### Q8：九张表够吗？不需要更多表吗？

**A**：9张是"必须表"——没有它任何一条业务链路都跑不通。还有一些"应该有"和"可选"的表（如 t_bonus_activity 奖金活动配置、t_daily_statistics 日统计、t_deposit_address USDT地址池），等需求评审确认后按需补充。先把骨架搭好，细节按需扩展。

### Q9：兑换为什么只能法币→BSB单向？

**A**：产品设计如此。BSB是平台币，平台希望用户充值后用BSB消费（投注/打赏），而不是直接将BSB换回法币提走。想要取出资金，需要走正式的提现流程（有审核、有手续费）。这是商业模式层面的设计，不是技术限制。

### Q10：这个模块什么时候能做完？

**A**：（不给时间估算，只说阶段和依赖）分7个Phase，Phase 1~4（脚手架→币种→余额→兑换）零外部依赖可独立推进；Phase 5~6（充值→提现）依赖财务模块RPC就绪或Mock；Phase 7 是全链路联调。每个Phase有明确的产出和验证标准，按阶段推进，卡点在跨团队IDL对齐。
