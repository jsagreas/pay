# 第一章 · 模块定位与全局认知

## 1.1 我负责什么

ser-wallet 是平台资金链路的核心枢纽。一句话概括：**平台上每一分钱的流动，都必须经过钱包模块的余额变更**。

```
充值到账 ──→ 钱包入账（CreditBalance）
提现冻结 ──→ 钱包冻结（FreezeBalance）
提现成功 ──→ 钱包扣款（DebitBalance）
提现失败 ──→ 钱包解冻（UnfreezeBalance）
手动调账 ──→ 钱包调整（ManualAdjust）
兑换执行 ──→ 钱包内部扣减+增加（原子事务）
投注扣款 ──→ 钱包扣减（预留）
中奖返还 ──→ 钱包入账（预留）
```

主要职责范围（两个子模块合为一个服务）：

- **子模块 A：币种配置**（基础设施层）— 币种 CRUD、汇率管理、充提配置、RPC 供给
- **子模块 B：钱包业务**（核心业务层）— 余额管理、充值、兑换、提现、账单记录、提现账户、RPC 供给

不属于我但必须了解的（同事负责，有联动依赖）：

- **ser-finance 财务管理** — 支付通道对接、三级提现审核、手动调账发起、数据统计

# 第二章 · 设计思路

## 2.1 核心设计哲学

**六个字：安全、正确、简单。**

- **安全**：资金操作零容错。宁可慢一点（悲观锁），不能错一笔（乐观锁高冲突重试）
- **正确**：余额 = 流水汇总，任何时刻可验证。每一次余额变动必须伴随一条流水记录
- **简单**：不引入分布式事务框架、不引入消息队列、不引入事件溯源，用最朴素的"DB 事务 + 幂等重试 + 定时补偿"三板斧覆盖所有场景

## 2.2 六大架构决策（ADR）

| 编号 | 决策 | 选择 | 放弃方案 | 核心理由 |
|------|------|------|----------|----------|
| ADR-1 | 并发控制 | SELECT FOR UPDATE 悲观行锁 | Redisson 分布式锁 / 乐观锁 | 钱包余额高争用，乐观锁重试率高；行锁粒度=一用户一币种一钱包类型=一行，锁持有 <10ms；支付宝/微信支付同方案 |
| ADR-2 | 金额精度 | BIGINT 最小单位整数 | DECIMAL / BigDecimal | 无浮点误差；Go 原生 int64；Stripe/支付宝/微信行业标准 |
| ADR-3 | 幂等保障 | DB 唯一索引（ref_order_no + tx_type + wallet_type） | Redis SetNX | Redis key 有 TTL 会过期丢失幂等性；DB 唯一索引与事务原子绑定 |
| ADR-4 | 跨服务一致性 | 幂等 RPC + 超时重试 + 定时补偿 | Seata/DTM 分布式事务 | 工程内无 Seata 先例；资金路径需同步确认；微信支付同方案 |
| ADR-5 | 余额缓存 | 不缓存，每次查 DB | Redis 缓存余额 | 行锁使缓存无意义（锁内操作 <10ms）；缓存一致性成本 > 收益；"资金数据宁慢勿错" |
| ADR-6 | 服务拆分 | 币种配置 + 钱包业务合为一个服务 | 拆成两个独立服务 | 币种配置的主要调用方就是钱包（内部方法调用 > RPC）；共享 DB 事务更简单；一个人负责 |

## 2.3 钱包结构设计

### 五种钱包类型

```
用户
 ├── 中心钱包（Center）     ← 充值到账、投注返奖、其他钱包转入 → 投注消费、提现、兑换扣减
 ├── 奖励钱包（Reward）     ← 充值加赠、活动奖励、签到任务     → 满足流水要求后转入中心钱包
 ├── 主播钱包（Streamer）   ← 礼物打赏收入、主播活动奖励       → 可直接提现或转入中心钱包
 ├── 代理钱包（Agent）      ← 代理佣金分成                    → 可提现或转入中心钱包（预留）
 └── 场馆钱包（Venue）      ← 进入iGaming场馆时从中心+奖励转入 → 退出场馆时原路返还（预留）
```

**关键公式**：用户可见余额（资金钱包余额）= 中心钱包可用 + 奖励钱包可用

**消费优先级**：先扣中心钱包，不足则继续扣奖励钱包。混合投注时按比例记录，返奖按比例返还。

**当前实现范围**：中心钱包 + 奖励钱包 + 主播钱包（必须），代理和场馆预留（表结构设计但不实现业务逻辑）。

### 余额模型

每个钱包账户有两个独立余额字段：

```
wallet_account:
  available_balance  BIGINT  — 可用余额（用户可操作的）
  frozen_balance     BIGINT  — 冻结余额（提现等待审核中的）
```

为什么不用"总额 - 冻结 = 可用"模型？
- 参考项目 A 用 `amount - freeze_amount = available`，扣减 SQL 需要 `WHERE (amount - freeze_amount) >= deduct_amount`，容易写错
- 我们的模型直接 `WHERE available_balance >= deduct_amount`，简单明确，两个字段在同一事务内独立更新

### 币种体系

| 类型 | 币种 | 小数位 | 存储倍率 | 千分位 | 示例 |
|------|------|--------|----------|--------|------|
| 法币 | VND | 0 | x1 | `.` | 1.000.000 |
| 法币 | IDR | 0 | x1 | `.` | 1.000.000 |
| 法币 | THB | 2 | x100 | `,` | 1,000.00 |
| 虚拟币 | USDT | 8 | x10^8 | `,` | 1,000.00000000 |
| 平台币 | BSB | 2 | x100 | `,` | 1,000.00 |

**BSB 锚定**：10 BSB = 1 USDT，硬编码逻辑。

**基准币种**：推荐 USDT，一旦设置不可更改。所有统计报表转换为基准币种。

**币种隔离原则**：切换币种 = 切换整个世界。余额、充值方式、账单记录全部按当前选中币种过滤，不存在跨币种混合展示。

### 汇率体系（四层）

```
第三方API（3个数据源取平均）
    → 实时汇率（市场价）
        → 偏差检测：|实时 - 平台| / 实时 × 100%
            → 偏差 ≥ 阈值 → 更新平台汇率
                → 充值汇率 = 平台汇率 × (1 + 浮动比例%)
                → 提现汇率 = 平台汇率 × (1 - 浮动比例%)
```

汇率存储：BIGINT × 10^8（如 1 USDT = 25,620 VND 存为 2,562,000,000,000）。

汇率精度：法币 2 位小数，虚拟币 8 位小数（四舍五入，补零）。

## 3.2 核心表关键字段

### ③ wallet_account（心脏表）

```sql
CREATE TABLE wallet_account (
    id                BIGINT NOT NULL AUTO_INCREMENT,
    user_id           BIGINT NOT NULL,
    currency_id       BIGINT NOT NULL,
    wallet_type       TINYINT NOT NULL,    -- 1中心 2奖励 3主播 4代理
    available_balance BIGINT NOT NULL DEFAULT 0,  -- 可用余额（最小单位整数）
    frozen_balance    BIGINT NOT NULL DEFAULT 0,  -- 冻结余额
    version           INT NOT NULL DEFAULT 0,     -- 预留乐观锁（当前用悲观锁）
    -- 审计六件套 + 软删除
    UNIQUE KEY uk_user_currency_type (user_id, currency_id, wallet_type, delete_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**并发控制**：`SELECT * FROM wallet_account WHERE user_id=? AND currency_id=? AND wallet_type=? FOR UPDATE`
- 锁粒度 = 精确一行
- 锁持有时间 < 10ms
- 单用户单账户 TPS ~100（足够）
- 死锁预防：多钱包操作时按 currency_id 升序加锁

**懒创建**：首次操作时自动创建 4 种钱包类型（center/reward/streamer/agent），非注册时批量创建。

### ④ transaction_log（流水表 · 对账根基）

```sql
CREATE TABLE transaction_log (
    id            BIGINT NOT NULL AUTO_INCREMENT,
    user_id       BIGINT NOT NULL,
    currency_id   BIGINT NOT NULL,
    wallet_type   TINYINT NOT NULL,
    op_type       TINYINT NOT NULL,  -- 1入账 2扣减 3冻结 4解冻 5扣减冻结
    tx_type       TINYINT NOT NULL,  -- 1充值 2提现 3兑换出 4兑换入 5奖励 6手动加 7手动减 ...
    amount        BIGINT NOT NULL,   -- 正数
    direction     TINYINT NOT NULL,  -- 1收入 2支出（从op_type推导）
    balance_after BIGINT NOT NULL,   -- 操作后可用余额快照
    frozen_after  BIGINT NOT NULL,   -- 操作后冻结余额快照
    ref_order_no  VARCHAR(64) NOT NULL, -- 关联订单号（幂等键）
    remark        VARCHAR(256) DEFAULT '',
    -- 只有 create_by/create_at，无 update/delete
    UNIQUE KEY uk_ref_order_tx (ref_order_no, tx_type, wallet_type)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**幂等唯一索引**为什么是三个字段？
- `ref_order_no`：同一笔业务订单号
- `tx_type`：同一笔兑换订单会产生"兑换出"和"兑换入"两条流水
- `wallet_type`：兑换入会同时写中心钱包（主体）和奖励钱包（加赠），同一 ref_order_no + 同一 tx_type

**对账公式**：`最新一条 transaction_log.balance_after == wallet_account.available_balance`（任何时刻成立）

### ⑤ deposit_order（充值订单）

```
关键字段：order_no, channel_order_no, user_id, currency_id, amount, actual_amount,
         payment_method(1银行/2电子钱包/3USDT), payment_network(TRC20/ERC20/BEP20),
         channel_id, rate_snapshot(x10^8), bonus_activity_id, bonus_amount,
         status(0待支付/1支付中/2待入账/3已完成/4失败/5超时), pay_url, pay_expired_at

状态机：严格单向，UPDATE SET status=new WHERE status=old，affected_rows=0 表示已处理（幂等）
重复充值检测索引：idx_user_currency_amount_status（同用户+同币种+同金额+status IN(0,1) ≥3 拦截）
超时扫描索引：idx_status_create（status, create_at）
```

### ⑥ withdraw_order（提现订单）

```
关键字段：order_no, user_id, currency_id, amount, fee, actual_amount(=amount-fee),
         withdraw_method(1银行/2电子钱包/3USDT), withdraw_account_id,
         rate_snapshot, frozen_amount, frozen_wallet_type,
         status(0待审核/1审核中/2打款中/3已完成/4失败/5已取消),
         audit_status(0未审/1风控过/2财务过/3出款过), fail_reason, paid_at

日提现额度查询：SELECT SUM(amount) WHERE user_id=? AND currency_id=? AND status NOT IN(4,5) AND create_at >= today_start
```

### ⑧ withdraw_account（提现账户）

```
关键字段：user_id, currency_id, account_type(1银行卡/2电子钱包/3USDT),
         bank_name, bank_branch, ewallet_type(GoPay/OVO/DANA/ShopeePay/PromptPay),
         network_type(TRC20/ERC20/BEP20),
         account_no(VARCHAR 512, AES-256加密), holder_name(VARCHAR 512, AES-256加密),
         is_default

三种账户类型共用一张表，未使用字段默认空字符串。
holder_name 从 KYC 实名自动填充，用户不可修改。
```

## 3.3 工程约定

| 约定 | 规则 |

# 第五章 · 核心业务流程

## 5.1 充值全链路

```
用户点"充值"
    │
    ▼
C-03 获取充值方式 ← 聚合：ser-finance.GetPaymentMethods + 内部币种配置（金额档位+汇率）
    │
    ▼
用户选择方式、输入金额、选择优惠活动
    │
    ▼
C-04 创建充值订单
    ├── 校验金额 ≥ 最低档位
    ├── 重复充值检测（同用户+同币种+同金额 pending ≥3 拦截，USDT链上豁免）
    ├── 锁定汇率快照（法币→BSB 场景）
    ├── 调用 ser-finance.MatchDepositChannel（⚠️关键路径）
    ├── 创建"待支付"订单，返回支付跳转信息
    │
    ▼
用户在第三方页面完成支付
    │
    ▼
第三方回调 → ser-finance 接收
    │
    ▼
ser-finance 调用 ser-wallet.CreditBalance（R-01）
    ├── 幂等检查（唯一索引命中直接返回成功）
    ├── BEGIN TX → SELECT FOR UPDATE → available += amount → 写 transaction_log → COMMIT
    ├── 充值金额 → 中心钱包
    ├── 加赠金额 → 奖励钱包（附带流水要求）
    │
    ▼
充值完成
```

**USDT 充值特殊流程**：无需通道匹配，展示平台地址+二维码，链上转账确认后回调入账。

**三层回调保障**：
1. 第三方主动回调（覆盖 95%+ 订单）
2. 定时任务扫描 pending 订单 → 主动查询第三方状态
3. B 端手动补单（最后兜底）

## 5.2 提现全链路

```
用户点"提现"
    │
    ▼
C-10 获取提现方式 ← ser-kyc.KycQuery 判断KYC状态
    ├── 未认证：仅显示USDT
    ├── 已认证：显示全部通道
    │
    ▼
C-14 绑定提现账户（首次）
    ├── 调用 ser-kyc.KycQuery 获取实名
    ├── 自动填充"户名"（只读不可改）
    ├── 保存银行卡/电子钱包/USDT地址
    │
    ▼
C-11 创建提现订单
    ├── 校验金额（单笔限额 + 日限额）
    ├── 原子操作：冻结余额 + 创建订单（同一DB事务）
    │   available -= amount, frozen += amount
    ├── 推送至 ser-finance 进入三级审核
    │
    ▼
ser-finance 三级审核（风控→财务→出款）
    │
    ├── 拒绝 → ser-finance 调用 UnfreezeBalance（R-04）
    │          frozen -= amount, available += amount（资金返还）
    │
    ├── 通过 + 打款成功 → ser-finance 调用 DebitBalance（R-02）
    │          frozen -= amount（从冻结扣减，非可用）
    │

                          frozen -= amount, available += amount（资金返还）
```

**冻结-确认/取消模式**（简化版 TCC）：
- Try（冻结）：available → frozen
- Confirm（确认）：frozen 扣减
- Cancel（取消）：frozen → available

## 5.3 兑换全链路

```
用户点"兑换BSB"（仅法币钱包有此入口，BSB钱包无）
    │
    ▼
C-08 兑换预览（纯计算，不写库）
    ├── 输入法币金额
    ├── 获取当前汇率
    ├── 计算：可得BSB = 金额 × 汇率
    ├── 计算：加赠BSB = 可得BSB × 加赠比例
    ├── 返回：汇率、可得BSB、加赠比例、加赠金额、实际到账
    │
    ▼
C-09 执行兑换（单DB事务，零外部依赖）
    ├── 重新获取汇率（不信任预览时的汇率）
    ├── 死锁预防：按 currency_id 升序加锁
    ├── BEGIN TX
    │   SELECT FOR UPDATE 源法币钱包（中心）
    │   SELECT FOR UPDATE BSB钱包（中心 + 奖励）

    │   增加BSB中心钱包 available += bsb_amount

    │   写 transaction_log（2~3条：兑换出、兑换入、奖励入）
    ├── COMMIT
    │
    ▼
兑换完成（3秒内，成功Toast提示）
```

**兑换是最纯粹的自闭环模块**：零外部RPC依赖，可最早独立开发和测试。

# 第七章 · 技术挑战与解决方案

## 7.1 资金安全类（最高优先级）

### 挑战 1：并发余额操作（超扣/重复入账）

**场景**：用户同时发起提现和投注，两个请求同时读到可用余额 1000，各扣 800，结果余额变成 -600。

**方案**：SELECT FOR UPDATE 悲观行锁。
- 锁粒度 = user_id + currency_id + wallet_type = 精确一行
- 第二个请求在锁等待队列中排队，等第一个事务提交后获取锁时看到的余额已是 200
- 锁持有 < 10ms，单用户单账户 TPS ~100

**为什么不用乐观锁？** 钱包余额是高争用资源，乐观锁在并发场景下频繁重试，反而更慢。

**为什么不用 Redisson 分布式锁？** Redis 宕机则锁失效；且参考项目 A 用 Redisson 已被评估为反模式。

### 挑战 2：重复回调导致重复入账

**场景**：支付通道因超时重试发送两次成功回调，用户被入账两次。

**方案**：DB 唯一索引 `uk_ref_order_tx (ref_order_no, tx_type, wallet_type)` 在 transaction_log 表上。
- 第一次写入成功
- 第二次 INSERT 触发唯一索引冲突（Duplicate Key），捕获错误后返回"已处理成功"
- 与业务事务原子绑定，不存在 Redis 过期后丢失幂等性的风险

### 挑战 3：金额精度（浮点误差）

**场景**：`0.1 + 0.2 != 0.3`（经典浮点问题），金融场景零容忍。

**方案**：BIGINT 最小单位整数存储。
- VND/IDR 存原始整数（无小数位）
- THB/BSB 存分（×100）
- USDT 存最小单位（×10^8）
- 汇率统一 ×10^8 存储
- 计算规则："先乘后除"防溢出；用户收取金额向下截断，用户支付金额向上取整
- 中间计算用 shopspring/decimal 库

## 7.2 分布式协调类

### 挑战 4：跨服务事务（钱包 ↔ 财务）

**场景**：财务模块通知充值成功，钱包入账后财务状态更新失败，数据不一致。

**方案**：幂等 RPC + 超时重试 + 定时补偿（三板斧）。
- 每个写 RPC 天然幂等（DB 唯一索引保障）
- 财务可以安全重试 CreditBalance（重复调用返回成功）
- 定时任务扫描异常状态订单，主动查询第三方状态进行补偿

### 挑战 5：提现冻结-确认/取消一致性

**场景**：提现审核通过后打款失败，冻结资金悬挂。

**方案**：显式双字段（available_balance + frozen_balance）+ 三个终态。
- 冻结：available -= X, frozen += X
- 确认（打款成功）：frozen -= X
- 取消（拒绝/失败）：frozen -= X, available += X
- 超时兜底：定时任务扫描冻结超 N 小时的订单，主动查询 finance 状态

### 挑战 6：回调丢失

**场景**：第三方支付成功但回调未到达我方系统。

**方案**：三层保障。
1. 回调处理（覆盖 95%+ 订单）
2. 定时任务扫描 pending 订单，主动查询第三方（覆盖 4%）
3. B 端手动补单（最后 1% 兜底）

## 7.3 性能类

### 挑战 7：大表查询性能（transaction_log 增长最快）

**预估**：10万活跃用户 × 日均5笔 = 50万条/天 = 1.8亿条/年

**方案**：分阶段应对。
- Phase 1（<1000万行）：索引优化 + 标准分页
- Phase 2（1000万~1亿行）：TiDB 分区表（按 create_at 按月分区）
- Phase 3（>1亿行）：冷热分离，>90天数据归档到历史表

### 挑战 8：汇率缓存一致性

**方案**：Cache-Aside + 主动写入。
- 汇率更新时主动写 Redis（而非删除后等下次读回填），防止高并发下的缓存击穿
- TTL 与拉取频率对齐（如拉取间隔 2 分钟，TTL 设 3 分钟）
- 创建订单时锁定汇率快照（写入订单的 rate_snapshot 字段），后续汇率变动不影响已创建订单

## 7.4 安全类

### 挑战 9：敏感数据保护

**方案**：应用层 GORM 自动加解密（AES-256-CBC），已有现成设施。
- `withdraw_account.account_no` 和 `holder_name` 配置为 `utils.EncryptedString`
- 数据库存储密文，应用读取时自动解密
- 密钥从 ETCD 获取

### 挑战 10：API 防刷

**方案**：五层防护。
1. 网关认证（FontAuth/BackAuth 已有）
2. 参数校验（Validator 已有）
3. 频率限制（Redis INCR + TTL，需新增）
4. 幂等控制（DB 唯一索引，已设计）
5. 审计日志（ser-blog，已有）

## 7.5 业务复杂度类

### 挑战 11：订单状态机

**方案**：DB 条件更新。
```sql
UPDATE deposit_order SET status = 3 WHERE order_no = ? AND status = 2
```
- `affected_rows = 0` 表示当前状态不对（已被其他进程处理），等价于幂等
- 状态严格单向推进，无回退

### 挑战 12：多钱包消费优先级

**场景**：用户投注 100，中心钱包 80，奖励钱包 50。

**方案**：
- 先扣中心 80（全部可用余额）
- 再扣奖励 20（差额部分）
- 记录比例：中心80%，奖励20%
- 返奖时按比例返回：如赢 200，中心得 160，奖励得 40
- 多钱包操作按 wallet_type 升序加锁，防死锁

### 挑战 13：兑换原子性（跨两个币种钱包）

**场景**：扣减 VND 成功但增加 BSB 失败（应用中途崩溃）。

**方案**：单 DB 事务包裹全部操作。
- BEGIN TX
- FOR UPDATE 源币种钱包（VND center）
- FOR UPDATE 目标币种钱包（BSB center + BSB reward）
- 加锁顺序：按 currency_id 升序，防死锁
- 全部操作在同一事务内，要么全成功要么全回滚
- COMMIT

# 第八章 · Balance Engine（余额引擎 · 核心中的核心）

## 8.1 设计理念

全模块唯一的余额变更入口。所有上层业务（充值、提现、兑换、手动调账、投注、返奖……）要修改余额，**必须**经过 balance_engine，不允许任何绕过。

```
deposit_service  ─┐
withdraw_service ─┤
exchange_service ─┤──→ balance_engine.ModifyBalance() ──→ wallet_account + transaction_log
manual_adjust    ─┤
betting(future)  ─┤
reward(future)   ─┘
```

## 8.2 仅暴露两个方法

**GetBalance**：读操作，返回 available + frozen。支持懒创建（首次访问自动创建 4 种钱包）。

**ModifyBalance**：写操作，参数如下：

```go
type ModifyBalanceParams struct {
    UserID      int64   // 用户ID
    CurrencyID  int64   // 币种ID
    WalletType  int8    // 钱包类型（1中心/2奖励/3主播/4代理）
    OpType      int8    // 操作类型（1入账/2扣减/3冻结/4解冻/5扣减冻结）
    Amount      int64   // 金额（正数）
    RefOrderNo  string  // 幂等键（关联订单号）
    TxType      int8    // 交易类型（1~12）
    Remark      string  // 备注
}
```

## 8.3 内部流程

```
ModifyBalance(params)
    │
    ├── 1. 幂等检查：查 transaction_log 是否已存在 ref_order_no + tx_type + wallet_type
    │      → 已存在则直接返回成功（重复调用安全）
    │
    ├── 2. BEGIN TX
    │
    ├── 3. SELECT * FROM wallet_account WHERE ... FOR UPDATE（行锁）
    │
    ├── 4. 前置校验
    │      · 扣减/冻结：available >= amount（否则余额不足）
    │      · 扣减冻结：frozen >= amount（否则冻结不足）
    │      · 入账/解冻：无条件限制
    │
    ├── 5. UPDATE wallet_account SET available = ?, frozen = ?
    │      · 入账：available += amount
    │      · 扣减：available -= amount
    │      · 冻结：available -= amount, frozen += amount
    │      · 解冻：frozen -= amount, available += amount
    │      · 扣减冻结：frozen -= amount
    │
    ├── 6. INSERT INTO transaction_log（balance_after, frozen_after 从操作后状态推导）
    │
    ├── 7. COMMIT
    │
    └── 返回操作结果
```

**关键约束**：
- 步骤 5 和步骤 6 在同一事务内，要么都成功要么都回滚
- **任何余额变动必须伴随一条流水记录**——这是铁律，无例外
- 对账验证：balance_after == wallet_account.available_balance（任何时刻）

## 8.4 五种操作类型与余额字段映射

```
OpType=1 入账(Credit)      : available += amount, frozen 不变
OpType=2 扣减(Debit)       : available -= amount, frozen 不变
OpType=3 冻结(Freeze)      : available -= amount, frozen += amount
OpType=4 解冻(Unfreeze)    : frozen -= amount, available += amount
OpType=5 扣减冻结(DebitFrozen): frozen -= amount, available 不变
```

## 8.5 位于工程中的位置

```
ser-wallet/internal/domain/balance_engine.go
```

这是 ser-wallet 独有的 domain 层（参考 ser-app 等模块没有 domain 层）。之所以抽出来：
- 被 6 个以上 service 文件调用
- 包含跨表事务（wallet_account + transaction_log）
- 包含行锁逻辑
- 避免任何 service 跳过统一入口直接操作数据

# 第十一章 · 潜在问答准备

> 预判需求大会上可能被问到的问题及应答要点

---

### Q1：为什么不用分布式事务框架（Seata/DTM）？

**答**：三个原因。
1. 工程内没有 Seata/DTM 的先例，引入新基础设施的风险 > 收益
2. 我们的跨服务场景只有"钱包↔财务"，通过幂等 RPC + 超时重试 + 定时补偿完全覆盖
3. 微信支付的核心资金系统用的也是这套方案（幂等+补偿），不是分布式事务框架

### Q2：余额为什么不缓存到 Redis？

**答**：四个原因。
1. 资金数据宁慢勿错——缓存一致性出问题就是资损
2. 行锁操作 < 10ms，缓存带来的性能提升可忽略
3. 每次写操作都要更新缓存，增加维护成本但收益极低
4. 唯一索引点查（uk_user_currency_type）< 1ms，和 Redis 查询延迟相当

### Q3：为什么充值订单和提现订单要分两张表？

**答**：参考项目 B 用合表（SsOrder），实际效果是大量 nullable 字段（充值没有 fee 字段、提现没有 pay_url 字段），查询需要 WHERE type=xxx 过滤。分表后：
- 每张表字段语义清晰
- 索引设计针对性更强
- 状态机不同（充值 5 个状态 vs 提现 6 个状态 + 3 级审核状态）
- 不同的增长速度和归档策略

### Q4：BIGINT 存金额，前端怎么展示？

**答**：约定三层分工。
1. **存储层**：BIGINT 最小单位整数（分/聪）
2. **API 层**：返回 BIGINT 原值 + currency_code + decimal_places
3. **前端层**：根据 decimal_places 和 currency 的千分位/小数位规则格式化展示
- 示例：BIGINT 1000000 + VND(decimal_places=0) → 显示 "1.000.000"
- 示例：BIGINT 100000000 + USDT(decimal_places=8) → 显示 "1.00000000"

### Q5：如果提现审核通过、打款却失败了怎么办？

**答**：财务模块收到第三方打款失败回调后，调用 UnfreezeBalance，冻结资金自动返还到用户可用余额。如果回调也丢失，定时任务扫描"打款中"状态超过 N 小时的订单，主动查询第三方状态，发现失败则触发解冻。最后兜底：B 端手动处理。

### Q6：怎么保证一笔充值不会入账两次？

**答**：三层保障。
1. **唯一索引**（核心保障）：transaction_log 上 `(ref_order_no, tx_type, wallet_type)` 唯一索引，重复 INSERT 触发 Duplicate Key 错误
2. **订单状态机**：`UPDATE deposit_order SET status=3 WHERE status=2`，第二次调用 affected_rows=0
3. **RPC 幂等**：调用方（finance）可以安全重试，钱包侧保证重复执行等价于执行一次

### Q7：兑换和充值有什么本质区别？

**答**：
- **充值**需要跨服务协调（钱包↔财务↔第三方），有异步回调，有状态机
- **兑换**完全在钱包内部完成，单 DB 事务，零外部依赖，创建即完成（无状态机）
- 兑换是最纯粹的自闭环模块，可以最早独立开发和测试

### Q8：多币种之间怎么隔离？

**答**：完全物理隔离。
- wallet_account 表的唯一索引包含 currency_id（一用户一币种一钱包类型=一行）
- 所有 C 端接口必传 currencyCode 参数
- 切换币种 = 切换整个数据视图（余额、充值方式、记录）
- 没有跨币种汇总展示（除了币种切换面板显示各币种余额概览）

### Q9：为什么币种配置和钱包放在同一个服务？

**答**：三个实际理由。
1. 币种配置的主要调用方就是钱包（内部方法调用比 RPC 快且简单）
2. 兑换操作需要读取汇率 + 修改余额，放在同一服务可以共享 DB 事务
3. 一个人负责两个子模块，合为一个服务降低运维和部署复杂度
4. 参考工程已有先例（ser-item 包含多个子模块）

### Q10：如果用户在提现审核中又发起投注，余额怎么算？

**答**：冻结余额用户不可操作。用户可用余额 = available_balance，冻结部分（frozen_balance）已经从 available 扣除，投注只能使用 available 部分。不存在冲突。

### Q11：开发周期怎么安排？外部依赖怎么解耦？

**答**：四阶段推进。
- **Phase 1（零外部依赖）**：币种 CRUD、汇率管理、balance_engine、兑换、账单记录（~14 个接口）
- **Phase 2（Mock 外部依赖）**：充值流程（Mock finance）、提现流程（Mock kyc）、RPC 暴露接口（~20 个接口）
- **Phase 3（集成联调）**：打通 finance RPC、打通 kyc、B 端管理（~20 个接口）
- 77% 的接口可以不依赖任何外部模块独立开发

### Q12：参考了哪些项目？学到了什么？

**答**：深度分析了两个 Java 生产项目。
- **参考项目 A**（p9-live-backend）：法币钱包系统，借鉴了双层交易分类（coin_type + coin_detail）、内外状态分离、订单号业务前缀等；规避了 Redisson 分布式锁、无幂等保障、共享数据库等反模式
- **参考项目 B**（tg-services）：TRC20 虚拟币充值系统，借鉴了四层地址架构（热钱包/归集/出款/Gas）、外部签名服务、Gas 自动补充等；规避了 AES/ECB 弱加密、缺少链上状态持久化等缺陷

我们的设计方案在并发控制（DB行锁 > Redisson）、精度（BIGINT > BigDecimal）、幂等（DB唯一索引 > 无/Redis）、表设计（分表 > 合表）四个维度上都优于两个参考项目。

### Q13：如果 Redis 挂了怎么办？如果 TiDB 挂了怎么办？

**答**：
- **Redis 挂了**：缓存 miss 直接降级到 DB 查询（慢但正确）；分布式锁降级到 DB 行锁；核心资金操作不依赖 Redis，不受影响
- **TiDB 挂了**：所有资金操作失败返回错误（无降级方案——资金操作不能降级）；待 DB 恢复后自动恢复；幂等设计保证恢复后可以安全重试所有 pending 操作

### Q14：团队只有一个人做这个模块，怎么保证质量？

**答**：
1. **设计阶段**：24 份分析文档覆盖所有维度，两个参考项目交叉验证
2. **编码阶段**：balance_engine 统一入口（核心逻辑集中在一个文件），GORM Gen 代码生成（减少手写代码），工程约定一致
3. **测试阶段**：对账机制（balance_after = available_balance）随时可验证数据正确性
4. **防错设计**：幂等索引、状态机条件更新、行锁——即使代码有 bug，DB 层面也会兜底拦截

### Q15：重复充值检测的具体规则是什么？

**答**：
- 检测条件：同一用户 + 同一币种 + 同一金额 + 状态为"待支付"或"支付中"的 pending 订单
- 1~2 笔 pending：弹窗警告，用户可选择继续（"查看详情" + "继续充值"）
- ≥3 笔 pending：硬拦截，禁止创建新订单（"查看详情" + "确定"关闭）
- **USDT 链上充值豁免**：链上交易有去重机制（txHash 唯一），不受此规则限制
- 实现：DB 索引查询 `idx_user_currency_amount_status`，不需要额外表
